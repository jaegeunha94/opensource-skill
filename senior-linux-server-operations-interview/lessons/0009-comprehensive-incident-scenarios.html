❓ 오늘의 면접 질문

시나리오 질문 1 — CPU 폭식

"새벽 3시, 모니터링에서 API 서버의 응답 지연 알림이 왔고 load average가 평소보다 5배 높습니다. 원인이 CPU 연산 폭증인지, I/O 대기인지, 아니면 스레드가 서로 락을 기다리며 멈춰 있는 것인지 어떻게 구분하고, 각 경우 어떤 조치를 취하시겠습니까?"

시나리오 질문 2 — 디스크 풀(Disk Full)

"결제 서비스가 갑자기 500 에러를 내기 시작했고, 로그에는 'No space left on device'가 찍혀 있습니다. df -h로는 여유 공간이 30% 남아 있는 것으로 보이는데도 같은 에러가 계속 발생합니다. 무엇을 의심하고, 어떤 순서로 확인하시겠습니까?"

시나리오 질문 3 — 서비스 재시작 반복(Crash Loop)

"배포 직후부터 특정 서비스가 몇 초 간격으로 계속 재시작되고 있습니다. systemctl status로는 'activating (auto-restart)' 상태만 반복해서 보일 뿐 원인을 알기 어렵습니다. 이 상황을 어떻게 진단하고, 반복 재시작이 시스템에 주는 2차 피해는 무엇이라고 보십니까?"

💡

Day 1~8은 CPU, 메모리, 디스크, 로그, systemd, 네트워크, 권한을 각각 따로 진단하는 도구였습니다. 하지만 실제 장애는 이 영역들이 서로 얽혀서 발생합니다 — 디스크가 가득 차면 로그 쓰기가 실패하고, 로그 쓰기 실패가 예외를 던지면 서비스가 재시작을 반복하고, 재시작 반복이 CPU를 잡아먹어 load average가 튑니다. 면접관이 Day 9에서 보는 것은 "어느 도구를 아느냐"가 아니라 "여러 증상이 뒤섞여 있을 때 어떤 순서로 가설을 좁혀가느냐"입니다.

📚 전제 개념 — 통합 진단이란 무엇인가

1. 왜 "각각 잘하는 것"과 "통합해서 잘하는 것"은 다른 능력인가

쉽게 말하면: 요리사가 칼질, 불 조절, 간 맞추기를 각각 완벽하게 할 줄 알아도, 실제 주문이 몰리는 저녁 시간에 "무엇을 먼저 볶고, 무엇을 삶는 동안 다른 걸 손질할지" 순서를 잘못 짜면 요리가 다 타버립니다. 서버 운영도 같습니다. top, iostat, journalctl을 각각 쓸 줄 아는 것과, 새벽에 알림이 울렸을 때 이 중 무엇을 몇 초 안에 먼저 확인해야 하는지를 판단하는 것은 완전히 다른 층위의 숙련도입니다.

  • 주니어는 "이 상황에서 어떤 명령어를 쓰나요?"에 답할 수 있습니다.
  • 미드레벨은 여러 명령어를 순서대로 실행할 수 있습니다.
  • 시니어는 첫 번째 증상만 보고 다음에 나올 증상을 예측하며, 확인 순서 자체를 비용(장애 지속 시간)과 정보량(가설을 얼마나 좁혀주는가)의 trade-off로 설계합니다.
✅

면접 팁: Day 9 질문에 대한 답변은 반드시 "먼저 A를 본다 → A의 결과에 따라 B 또는 C로 분기한다"는 조건부 구조로 말해야 합니다. "top 보고, iostat 보고, journalctl 봅니다"처럼 도구를 나열만 하면 순서를 설계하지 못한다는 신호로 읽힙니다.

2. 증상 지도(symptom map) — 하나의 신호가 가리킬 수 있는 여러 원인

쉽게 말하면: 의사가 "복통"이라는 하나의 증상만으로 맹장염인지 장염인지 판단하지 않고, 통증 위치·발열 여부·혈액 검사로 좁혀가듯이, 서버 장애도 하나의 표면 증상(응답 지연, 500 에러, 재시작 반복) 뒤에 여러 원인 후보가 있다는 것을 아는 것이 출발점입니다.

표면 증상가능한 원인 후보 (Day 1~8 연결)
load average 급증CPU 연산 폭증(Day 1,2) / iowait 급증(Day 1,4) / 락 대기로 인한 D 상태 프로세스 누적(Day 2)
응답 지연CPU 포화 / 메모리 스와핑(Day 3) / 디스크 I/O 병목(Day 4) / TCP 백로그 포화(Day 7)
500 에러 + 로그에 I/O 에러디스크 공간 부족(Day 4) / inode 고갈(Day 4) / 권한 문제(Day 8) / 읽기 전용 파일시스템 전환
서비스 반복 재시작설정 파일 오류 / 의존 서비스 미기동(Day 6 After= vs Requires=) / 리소스 고갈로 인한 즉시 크래시(Day 2,3,4) / OOM Killer(Day 3)
💡

실무 연결: 이 지도를 머릿속에 갖고 있으면, 표면 증상 하나를 보는 순간 "다음에 무엇을 확인해야 후보를 절반으로 줄일 수 있는가"를 계산할 수 있습니다. 이것이 바로 "경험 많은 엔지니어는 감이 좋다"고 표현되는 것의 실체 — 사실은 감이 아니라 이 증상-원인 지도가 머릿속에 구조화되어 있는 것입니다.

3. 연쇄 장애(cascading failure) — 원인과 증상이 뒤바뀌어 보이는 이유

쉽게 말하면: 도미노처럼, 하나의 작은 문제(디스크 공간 부족)가 다른 문제(로그 쓰기 실패)를 일으키고, 그것이 또 다른 문제(예외 발생 → 프로세스 크래시)를 일으키고, 그것이 또 다른 증상(CPU 급증 — 재시작을 반복하며 초기화 연산을 계속 수행)으로 이어지는 연쇄를 연쇄 장애라고 합니다. 문제는 엔지니어가 가장 마지막에 나타난, 가장 눈에 띄는 증상(CPU 급증)부터 보게 된다는 점 — 진짜 최초 원인(디스크 공간 부족)은 시간상 더 먼저 일어났지만 화면에는 가장 늦게, 가장 크게 나타납니다. 

# 연쇄의 실제 순서(시간순)
1. 디스크 공간 부족 발생 (근본 원인, 하지만 조용함)
2. 애플리케이션 로그 쓰기 실패 → 예외 발생
3. 예외를 처리 못한 프로세스 크래시
4. systemd Restart=always 로 즉시 재기동
5. 재기동 시 초기화 로직(캐시 warm-up 등)이 CPU를 다시 사용
6. 재기동 실패 → 다시 크래시 → 반복
7. 결과적으로 화면에 가장 크게 보이는 것: CPU 급증 + load average 급등
  • 시니어와 주니어의 차이는 "가장 늦게 나타나 가장 시끄러운 증상"에 낚이지 않고, "무엇이 가장 먼저 일어났을까"를 역순으로 추적하는 습관에 있습니다.
  • 이 역추적의 실무적 도구가 바로 journalctl --since로 각 서비스의 실패 최초 시각을 비교하는 것 — 가장 이른 타임스탬프를 가진 실패가 근본 원인일 확률이 높습니다.
⚠️

흔한 실수: CPU 급증만 보고 "CPU 코어를 늘리자"거나 "무거운 프로세스를 kill하자"는 조치를 취하면, 근본 원인(디스크 공간)은 그대로 남아 잠시 후 같은 장애가 재발합니다. 눈에 보이는 증상을 없애는 것과 원인을 없애는 것은 다릅니다.

4. "즉각 조치"와 "근본 원인 분석"을 병렬로 진행하는 사고

쉽게 말하면: 화재 현장에서 소방관은 불을 끄는 팀과 화재 원인을 조사하는 팀을 동시에 움직입니다 — 원인 조사가 끝날 때까지 불을 그냥 두지 않습니다. 서버 장애도 마찬가지로, "서비스를 다시 살리는 조치"(고객 영향 최소화)와 "왜 이런 일이 일어났는지 분석"(재발 방지)은 순차적이 아니라 병렬적으로, 그러나 서로 다른 우선순위 기준으로 진행되어야 합니다.

  • 즉각 조치 기준: "지금 당장 고객 영향을 줄이는 가장 빠른 방법이 무엇인가" — 근본 원인을 몰라도 실행 가능해야 합니다(예: 오래된 로그 파일 삭제로 디스크 공간 확보, 문제 인스턴스를 로드밸런서에서 제외).
  • 근본 원인 분석 기준: "같은 장애가 다시 발생하지 않으려면 무엇을 바꿔야 하는가" — 즉각 조치보다 시간이 걸려도 됩니다(예: 로그 로테이션 정책 변경, 디스크 사용량 알림 임계치 조정).
  • 이 둘을 혼동하면 "일단 서비스가 살아났으니 끝"이라는 함정에 빠지거나, 반대로 "정확한 원인을 알기 전까지는 아무 조치도 하지 않는다"는 또 다른 함정에 빠집니다.
✅

면접 팁: 면접관은 종종 "지금 당장 뭘 하시겠습니까?"와 "재발 방지책은 무엇입니까?"를 같은 질문 안에 섞어서 묻습니다. 이 두 축을 분리해서 답변하는 것만으로도 "이 사람은 화재 진압과 화재 조사를 동시에 운영할 줄 안다"는 신호를 줍니다.

🎯 왜 면접에서 묻는가

종합 시나리오 질문은 개별 명령어 지식이 아니라, 실전 장애에서 살아남는 사고 체계 전체를 평가합니다. 10년차 이상에게 회사가 정말로 맡기고 싶은 것은 "명령어를 아는 사람"이 아니라 "새벽 3시에 혼자 깨어나 판단해야 하는 순간에 신뢰할 수 있는 사람"이기 때문입니다.

면접관이 실제로 평가하는 것

  1. 도구를 순서 있게 조합하는가?
    Day 1~8의 개별 지식을 "이 상황엔 이거, 저 상황엔 저거"로 흩어서 아는 것이 아니라, 하나의 의사결정 트리로 엮어서 실행할 수 있는지를 봅니다.
  2. 연쇄 장애에서 최초 원인을 역추적하는가?
    가장 시끄러운 증상(CPU 급증)에 낚이지 않고, 타임라인을 재구성해 가장 먼저 일어난 사건을 찾아내는 사고를 하는지를 봅니다.
  3. 즉각 조치와 재발 방지를 구분해서 병렬로 사고하는가?
    "일단 됐다"에서 멈추지 않고, 같은 사고가 재발하지 않도록 프로세스나 설정을 바꾸는 데까지 사고가 이어지는지를 봅니다.
  4. 불확실한 상황에서 커뮤니케이션할 수 있는가?
    원인을 100% 확신하지 못하는 순간에도 "현재까지 확인된 사실 / 남은 가설 / 다음 확인 계획"을 구조적으로 전달할 수 있는지를 봅니다. 실전 장애는 결코 깔끔하게 답이 딱 떨어지지 않기 때문입니다.
💡

본질은 "지식의 양"이 아니라 "지식을 압박 속에서 조립하는 속도와 순서"입니다. 면접관은 여러분이 Day 1~8의 모든 명령어를 외웠는지보다, 모르는 상황에서도 무너지지 않는 사고의 골격이 있는지를 확인하려고 합니다.

🏭 실무 연결 — 3대 종합 장애 시나리오 전체 진단 흐름

시나리오 A: CPU 폭식 — 연산 과부하인가, I/O 대기인가, 락 경합인가

새벽 알림: API 응답 지연 + load average 5배 급증. 세 가지 근본적으로 다른 원인이 같은 증상(load average 급증)으로 나타날 수 있어, 첫 번째 확인이 이후 전체 대응 방향을 가릅니다. 

# 1단계 — Day 1의 첫 스냅샷: us/sy/wa 분리로 큰 범주 파악
$ top -bn1 | head -5
%Cpu(s): 78.2 us,  3.1 sy,  0.4 ni, 15.1 id,  2.8 wa,  0.0 hi,  0.4 si,  0.0 st
# us(user)가 압도적으로 높음 → "I/O 대기"가 아니라 "연산 자체가 많다"는 1차 결론

# 2단계 — Day 2의 프로세스별 분해: 어떤 프로세스가 CPU를 먹는가
$ top -bn1 -o %CPU | head -8
  PID  USER  %CPU  %MEM  COMMAND
 8823  app    412%  3.2  java  (멀티스레드로 4코어 넘게 사용 중)

# 3단계 — 스레드 단위로 더 쪼개기: 특정 스레드 하나가 폭주하는가, 전체가 골고루 바쁜가
$ top -bn1 -H -p 8823 | head -10
# 특정 스레드 ID(TID) 하나가 CPU를 독점 → GC 스레드 폭주 또는 무한루프 버그 의심
# 반대로 여러 스레드가 골고루 바쁘면 → 순수 트래픽 증가로 인한 정상적 연산 증가일 가능성

# 4단계 — "락 경합으로 멈춰 있는 것"과 구분: D 상태와 우선순위 역전 확인
$ ps -eo pid,stat,wchan:20,cmd | awk '$2 ~ /D/'
# D 상태 프로세스가 다수라면 CPU 문제가 아니라 I/O 대기 — 처음 결론을 뒤집어야 함

이 스레드가 폭주하는 원인이 정말 무한루프인지 확인하려면 strace -p <PID> -c로 요약된 syscall 통계를 잠깐 뜬 뒤(Day 2), syscall이 거의 없는데 CPU만 계속 도는 패턴이면 순수 연산 루프(버그 또는 알고리즘 비효율)로 확진합니다. 이 사례는 실제로 캐시 무효화 로직에서 조건 분기 실수로 무한 재시도가 발생한 것이 원인이었고, 즉각 조치로 해당 스레드를 가진 인스턴스를 로드밸런서에서 빼고 재시작했으며, 근본 원인은 코드 리뷰로 재시도 조건을 수정해 다음 배포에 반영했습니다.

시나리오 B: 디스크 풀 — df는 여유 있는데 계속 "No space left on device"

Day 4에서 배운 df -h와 df -i는 별개 자원이라는 지식과, rm이 즉시 공간을 회수하지 않는다는 지식을 실제 순서로 조합해야 하는 사례입니다. 

# 1단계 — 용량이 아니라 inode 고갈 의심 (Day 4 핵심 개념)
$ df -i /var
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda2      1310720 1310720      0  100% /var
# 용량(df -h)은 30% 여유였지만 inode는 100% 소진 — 진짜 원인 확정

# 2단계 — 어느 디렉터리가 소용량 파일을 대량 생성했는가
$ for d in /var/*/; do echo "$d $(find "$d" -xdev | wc -l)"; done | sort -k2 -nr | head -5
/var/spool/myapp-queue/ 842103   # 실패한 메시지가 재시도 로직 오류로 계속 파일 생성됨

# 3단계 — 만약 df -h 자체가 100%였다면? '삭제된 열린 파일' 가능성도 함께 확인
$ df -h /var        # 100%인데 du -sh로 합산하면 훨씬 작음 → 삭제됐지만 열려있는 fd 의심
$ lsof +L1 | grep deleted
myapp    8823  app   15w  REG  8,2  4123456789    0 /var/log/myapp/app.log (deleted)
# 로그 로테이션 후에도 프로세스가 예전 fd를 계속 붙들고 쓰기 때문에 공간이 회수되지 않음

이 시나리오의 진짜 함정은 "df -h로 여유가 있으니 디스크 문제가 아니다"라고 1차 판단을 성급하게 내리는 것입니다. Senior는 "No space left on device"라는 메시지를 보는 즉시 df -h와 df -i를 둘 다 확인하는 반사적 습관이 있어야 합니다. 이번 사례에서는 실패한 큐 메시지가 재시도할 때마다 임시 파일을 새로 만들고 지우지 않는 버그가 원인이었고, 즉각 조치로 오래된 임시 파일을 일괄 삭제해 inode를 회수했으며, 재발 방지로 해당 큐 디렉터리에 대한 inode 사용률 알림을 새로 추가했습니다.

시나리오 C: 서비스 재시작 반복(Crash Loop) — 원인 규명과 2차 피해 통제

Day 6의 systemd 지식과 Day 5의 journalctl 필터링을 조합해, "무한 재시작 자체가 추가 장애를 만드는" 상황을 통제하는 사례입니다. 

# 1단계 — 반복 재시작의 정확한 패턴 확인 (초 단위인가, 분 단위인가)
$ systemctl status myapp
Active: activating (auto-restart) (Result: exit-code) since ... 4s ago
Process: exit status 1

# 2단계 — 가장 처음 실패가 시작된 시각을 특정 (연쇄의 시작점 추적, Day 5)
$ journalctl -u myapp --since "1 hour ago" | grep -m1 "Main process exited"
Jul 08 02:14:07 ... myapp[8801]: Main process exited, code=exited, status=1/FAILURE
# 이 시각을 기준으로 다른 서비스/시스템 로그와 대조

$ journalctl --since "02:13:00" --until "02:15:00" -p err
# 같은 시각대에 kernel: EXT4-fs error 또는 disk full 로그가 있다면 -> 근본 원인이 다른 계층

# 3단계 — 반복 재시작 자체가 만드는 2차 피해 진단
$ journalctl -u myapp --since "1 hour ago" | grep -c "Started myapp"
347   # 1시간에 347번 재기동 -> 매번 커넥션 풀 재생성/캐시 워밍업으로 DB/CPU에 부하 가중

실제 원인은 배포된 설정 파일에 오타가 있어 애플리케이션이 시작 즉시 예외를 던지는 것이었지만, Restart=always와 짧은 RestartSec= 설정 때문에 1시간에 347번 재기동을 시도하면서 매번 DB 커넥션 풀을 새로 만들어 DB 서버 자체에도 연쇄적으로 부하를 주고 있었습니다. 즉각 조치로 systemctl stop으로 재시작 루프를 멈춰 2차 피해(DB 부하)부터 차단한 뒤, 설정 파일을 롤백해 정상 기동을 확인했고, 재발 방지로 StartLimitBurst를 낮춰 짧은 시간 내 과도한 재시도 자체를 systemd 레벨에서 차단하도록 조정했습니다.

3대 시나리오에서 공통으로 배운 핵심 교훈

  • 표면 증상 하나(load average, 500 에러, 재시작 반복)는 여러 근본 원인의 입구일 뿐 — 첫 번째 확인이 다음 확인의 방향을 결정하므로, "가장 정보량이 많은 확인"부터 시작해야 한다.
  • 가장 눈에 띄는 증상이 가장 나중에 일어난 결과일 수 있다 — 타임스탬프를 비교해 역방향으로 최초 원인을 추적해야 한다.
  • 즉각 조치(피해 통제)와 근본 원인 분석(재발 방지)은 분리된 두 개의 트랙이며, 즉각 조치는 원인을 몰라도 실행할 수 있어야 한다.
  • 반복되는 실패(crash loop, 재시도 폭주) 자체가 2차 피해를 만드는 독립적인 위험 요소이므로, 근본 원인을 찾기 전에 반복 자체를 먼저 멈추는 판단이 필요할 때가 있다.

🧠 답변 사고 순서 — 통합 트리아지 메타 프레임워크

  • 1. 영향 범위 확정 + 피해 통제 여부 즉시 판단

    "지금 고객에게 영향이 계속 커지고 있는가"를 가장 먼저 판단합니다. 반복 재시작처럼 방치할수록 2차 피해가 누적되는 패턴이면, 원인을 찾기 전이라도 그 반복 자체를 멈추는 조치(서비스 임시 중지, 로드밸런서에서 제외)를 우선 고려합니다.

  • 2. 가장 정보량이 큰 스냅샷부터 확보

    top의 us/sy/wa 분리, df -h & df -i 동시 확인, journalctl의 최초 실패 시각처럼, 한 번의 확인으로 여러 원인 후보를 동시에 배제할 수 있는 지표부터 봅니다.

  • 3. 타임라인을 재구성해 연쇄의 시작점 역추적

    여러 서비스/시스템 로그의 타임스탬프를 나란히 놓고, 가장 이른 실패가 무엇인지 확인합니다. 가장 시끄러운 증상이 아니라 가장 먼저 일어난 사건이 근본 원인일 확률이 높다는 것을 전제로 접근합니다.

  • 4. 후보를 좁힌 도구로 확진(confirm)

    1~2차 스냅샷으로 후보가 2~3개로 좁혀지면, strace, lsof, ausearch처럼 더 구체적이지만 비용이 큰 도구를 그제서야 투입해 정확히 하나의 원인으로 확진합니다.

  • 5. 즉각 조치와 재발 방지를 분리해서 실행 및 커뮤니케이션

    서비스 복구는 원인 확진과 별개로 최대한 빨리 진행하고, 근본 원인은 재발 방지책(설정 변경, 알림 임계치 조정, 코드 수정)으로 이어 붙입니다. 두 트랙의 진행 상황을 이해관계자에게 각각 구분해서 전달합니다.

💬 핵심 답변 예시

세 시나리오 각각에 대해 면접에서 자연스럽게 말할 수 있는 구조입니다. 외우지 말고 논리의 흐름을 이해하세요.

💬 면접관: "load average가 5배 급증했습니다. CPU 연산 폭증인지, I/O 대기인지, 락 경합인지 어떻게 구분하시겠습니까?"

"가장 먼저 top의 us/sy/wa 비율을 봅니다. us가 높으면 순수 연산, wa가 높으면 I/O 대기, 그리고 ps로 D 상태 프로세스 수를 함께 확인해서 락 대기로 인한 정체인지도 같이 판단합니다. us가 높다면 프로세스별 CPU를 top으로 분해하고, 특정 프로세스가 지목되면 스레드 단위(top -H)로 한 번 더 쪼개서 폭주하는 스레드 하나인지 전체가 골고루 바쁜지 구분합니다. 폭주하는 스레드가 하나라면 strace -c로 syscall 패턴을 잠깐 확인해서 순수 연산 루프(버그)인지 정상적인 트래픽 증가인지 확진합니다. 이 판단에 따라 조치가 완전히 달라지기 때문에 순서를 건너뛰지 않는 것이 중요합니다."

💬 면접관: "df -h는 여유가 있는데 계속 'No space left on device'가 발생합니다."

"df -h만으로 '디스크 문제가 아니다'라고 단정하지 않고, 곧바로 df -i로 inode 사용률을 함께 확인합니다. 용량과 inode는 완전히 별개의 자원이기 때문에 소용량 파일이 대량으로 생성되는 패턴에서는 용량은 남아 있어도 inode가 먼저 고갈될 수 있습니다. inode도 정상이라면, 삭제됐지만 프로세스가 여전히 붙들고 있는 파일 디스크립터를 lsof +L1로 확인합니다 — rm은 디렉터리 엔트리만 지울 뿐, 그 파일을 열고 있는 프로세스가 남아 있으면 실제 공간은 회수되지 않기 때문입니다. 이 두 가지를 df -h 하나만 보고 넘어가지 않는 것이 이 문제의 핵심입니다."

💬 면접관: "서비스가 몇 초 간격으로 계속 재시작되고 있습니다. 어떻게 접근하시겠습니까?"

"먼저 이 반복 재시작 자체가 2차 피해(예: 매번 DB 커넥션 풀을 새로 만들어 DB에 부하를 주는 것)를 만들고 있는지부터 판단합니다. 만약 그렇다면 원인을 찾기 전에 systemctl stop으로 재시작 루프 자체를 먼저 멈춰서 피해 확산을 차단합니다. 그 다음 journalctl -u <service>로 가장 처음 실패가 시작된 시각을 특정하고, 그 시각대의 시스템 전체 로그(journalctl --since ... -p err)를 함께 봐서 이 서비스만의 문제인지 디스크나 커널 레벨의 더 근본적인 문제가 있는지 확인합니다. 원인을 확정한 뒤에는 정상 기동을 복구하는 것과 별개로, StartLimitBurst 같은 systemd 설정으로 향후 유사한 크래시가 발생해도 무한 재시도로 이어지지 않도록 재발 방지를 함께 적용합니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "여러 원인 후보가 동시에 그럴듯해 보일 때, 확인 순서를 어떻게 정하나요?"

"저는 두 가지 기준으로 순서를 정합니다. 첫째는 확인 비용이 낮은 것부터 — top이나 df 한 번 실행하는 것은 몇 초면 되지만 strace로 장시간 추적하는 것은 프로덕션에 부하를 줄 수 있으므로 비용이 큽니다. 둘째는 정보량이 큰 것부터 — 하나의 확인으로 여러 후보를 동시에 배제할 수 있는 지표를 우선합니다. 예를 들어 us/sy/wa 비율 하나만 봐도 'CPU 연산인가 I/O 대기인가'라는 큰 갈래를 즉시 나눌 수 있으므로, 이런 지표를 먼저 확인한 뒤 남은 후보에 대해서만 비용이 큰 도구를 투입합니다."

평가 포인트: 확인 순서를 비용과 정보량이라는 두 축의 trade-off로 설명하는가

Q. "근본 원인을 아직 모르는데 팀에게 어떻게 상황을 전달하나요?"

"저는 항상 세 가지를 구분해서 전달합니다 — 지금까지 확인된 사실(fact), 아직 검증되지 않은 가설(hypothesis), 그리고 다음에 확인할 계획(next step)입니다. 예를 들어 'inode가 100% 소진된 것은 확인된 사실이고, 특정 큐 디렉터리가 원인일 것이라는 것은 아직 가설이며, 지금 find로 어느 디렉터리인지 좁히는 중입니다'처럼 말합니다. 이렇게 하면 아직 원인을 모르는 상태에서도 팀이 불필요하게 불안해하지 않고, 각자 무엇을 도와줄 수 있는지 판단할 수 있습니다."

평가 포인트: 불확실성을 숨기지 않고 fact/hypothesis/next step으로 구조화해서 전달하는가

Q. "즉각 조치가 근본 원인 분석을 방해할 수도 있지 않나요? (예: kill이 증거를 없앰)"

"맞습니다. 그래서 저는 되돌릴 수 없는 조치(프로세스 kill, 서버 재부팅) 전에 가능한 한 증거를 먼저 확보합니다. 예를 들어 CPU를 독점하는 프로세스를 죽이기 전에 jstack이나 gcore로 스택 트레이스나 코어 덤프를 먼저 남기고, crash loop를 멈추기 전에 마지막 실패 로그를 저장해둡니다. 다만 고객 영향이 매우 크고 시간이 없다면, 증거 확보보다 서비스 복구를 우선하고 대신 비슷한 증상이 재현될 때를 대비해 모니터링을 강화하는 방식으로 타협합니다. 즉 '항상 증거부터'가 아니라, 영향의 크기에 따라 이 우선순위 자체를 조정하는 것이 핵심입니다."

평가 포인트: 증거 확보와 피해 통제 사이의 긴장을 인지하고, 상황에 따라 우선순위를 조정하는가

Q. "이런 연쇄 장애가 다시 발생하지 않도록 시스템 차원에서 무엇을 바꾸시겠습니까?"

"세 가지 층위에서 접근합니다. 첫째, 개별 자원(디스크, inode, CPU, 메모리)에 대한 경보 임계치를 '가득 참'이 아니라 '가득 차기 전 여유 구간'(예: 80%)에서 미리 울리도록 조정합니다. 둘째, systemd의 StartLimitBurst나 재시도 로직에 지수 백오프(exponential backoff)를 넣어서 한 번의 실패가 무한 반복으로 증폭되지 않도록 시스템 자체에 제동장치를 넣습니다. 셋째, 이번 장애의 연쇄 경로(디스크 → 로그 실패 → 크래시 → 재시작 폭주 → DB 부하)를 문서화해서, 다음에 비슷한 표면 증상이 보였을 때 팀 전체가 이 경로를 참고할 수 있는 런북(runbook)으로 남깁니다."

평가 포인트: 재발 방지를 알림/제동장치/지식 자산화라는 여러 층위로 나눠 설계하는가

Q. "포스트모템(postmortem)을 작성할 때 가장 중요하게 다루는 부분은 무엇인가요?"

"저는 '무엇이 잘못됐는가'보다 '왜 우리가 이것을 더 빨리 발견하지 못했는가'에 더 많은 비중을 둡니다. 예를 들어 디스크 풀 사례라면, 버그 자체를 고치는 것은 당연하지만, '왜 inode 사용률에 대한 알림이 없었는가'라는 관측성의 공백이 더 근본적인 재발 방지 포인트입니다. 또한 사람을 탓하는 어조를 피하고 (blameless postmortem), 시스템과 프로세스의 개선에 초점을 맞춰서 팀원들이 다음에 유사한 상황을 숨기지 않고 투명하게 공유하도록 만드는 것을 중요하게 생각합니다."

평가 포인트: 원인 수정을 넘어 탐지 공백을 지적하고, blameless 문화를 이해하는가

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"top 켜보고, df 봐보고, journalctl 봐서 로그 확인하고, 문제 있으면 재시작하겠습니다."
왜 약한 답변인가:
  • 도구를 순서 없이 나열만 할 뿐, 각 도구의 결과가 다음 확인을 어떻게 좁혀주는지 연결하지 못합니다.
  • "재시작하겠습니다"는 근본 원인과 무관한 만능 해결책으로, 연쇄 장애의 근본 원인이 그대로 남아 재발 가능성이 높습니다.
  • 즉각 조치와 재발 방지를 구분하지 않고 하나로 뭉뚱그립니다.
좋은 답변 (Senior 수준)
"가장 정보량이 큰 지표(us/sy/wa 또는 df -h와 df -i)부터 확인해서 큰 갈래를 나누고, 타임라인을 비교해서 가장 먼저 일어난 사건을 역추적한 뒤, 좁혀진 후보에만 비용이 큰 도구를 투입해 확진합니다. 서비스 복구는 원인 확정과 별개로 먼저 진행하고, 재발 방지는 알림 임계치와 시스템 제동장치로 별도 트랙에서 처리합니다."
왜 좋은 답변인가:
  • 확인 순서를 정보량/비용의 trade-off로 설계하고, 조건부 분기 구조로 설명합니다.
  • 연쇄 장애에서 표면 증상이 아니라 최초 원인을 역추적하는 사고를 보여줍니다.
  • 즉각 조치(피해 통제)와 근본 원인 분석(재발 방지)을 명확히 분리된 두 트랙으로 병렬 처리합니다.

🚫 흔한 오해와 정확한 이해

오해 1: "가장 눈에 띄는 증상이 곧 근본 원인이다"

연쇄 장애에서는 가장 나중에 발생한 증상이 가장 크고 시끄럽게 보이는 경우가 많습니다. 타임스탬프를 비교해 가장 먼저 일어난 사건을 찾는 역추적이 필요합니다.

오해 2: "df -h로 여유가 있으면 디스크는 원인이 아니다"

용량(df -h)과 inode(df -i)는 별개의 자원이고, 삭제됐지만 열려 있는 파일도 용량을 계속 점유합니다. 디스크 관련 증상에서는 이 둘을 항상 함께 확인해야 합니다.

오해 3: "원인을 확실히 알기 전까지는 아무 조치도 하면 안 된다"

고객 영향이 계속 커지는 상황(반복 재시작으로 인한 DB 부하 등)에서는 원인을 모르더라도 피해 확산을 먼저 차단하는 조치가 필요합니다. 즉각 조치와 원인 규명은 병렬로 진행하는 것이 원칙입니다.

오해 4: "서비스가 다시 살아나면 장애는 끝난 것이다"

서비스 복구는 즉각 조치의 완료일 뿐, 재발 방지책(알림 임계치, 제동장치, 코드 수정)이 적용되지 않으면 같은 연쇄가 다시 발생할 수 있습니다. 복구와 재발 방지는 별개의 완료 기준을 가져야 합니다.

오해 5: "모든 도구를 다 써봐야 정확한 진단이다"

비용이 큰 도구(strace 장시간 추적 등)를 무분별하게 모두 사용하면 오히려 프로덕션에 부하를 주고 진단 시간을 늘립니다. 저비용·고정보량 지표로 후보를 먼저 좁힌 뒤, 남은 후보에만 정밀 도구를 투입하는 것이 효율적입니다.

Day 9 핵심 포인트 요약

  • 표면 증상 하나는 여러 원인 후보의 입구일 뿐이며, 저비용·고정보량 지표부터 확인해 후보를 좁혀야 한다.
  • 가장 눈에 띄는 증상이 가장 나중에 일어난 결과일 수 있으므로, 타임라인 비교로 최초 원인을 역추적해야 한다.
  • 즉각 조치(피해 통제)와 근본 원인 분석(재발 방지)은 분리된 병렬 트랙이며, 반복/재시도 자체가 만드는 2차 피해는 원인 규명 전에도 먼저 통제해야 할 때가 있다.
  • 원인을 모르는 순간에도 fact / hypothesis / next step으로 구조화해서 커뮤니케이션하는 것이 senior의 신뢰를 만든다.
  • 재발 방지는 코드 수정 하나로 끝나지 않고, 알림 임계치, 시스템 제동장치, 런북 자산화까지 다층적으로 설계해야 한다.
