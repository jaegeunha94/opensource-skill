📚 전제 개념 — 이것을 먼저 이해해야 한다

1. TCP 연결 상태 머신 — "전화 통화"에 비유해서 이해하기

쉽게 말하면: TCP 연결은 전화 통화와 비슷합니다. 전화를 걸고 (SYN), 상대가 받고(SYN-ACK), 내가 "여보세요"라고 확인하면(ACK) 통화가 시작됩니다(ESTABLISHED). 통화를 마칠 때도 한쪽이 먼저 "끊을게요"(FIN)라고 말하고 상대가 "네, 알겠습니다"(ACK)로 응답하는 절차를 거칩니다. 이 절차 하나하나가 소켓의 상태로 기록되고, ss/netstat으로 그 상태를 눈으로 확인할 수 있습니다. 

클라이언트                    서버
   SYN        ─────────────▶   (LISTEN 상태로 대기 중이던 소켓이 요청을 받음)
              ◀─────────────   SYN-ACK
   ACK        ─────────────▶   ESTABLISHED (양쪽 모두 통신 준비 완료)

   ... 데이터 송수신 (ESTABLISHED) ...

   FIN        ─────────────▶   (먼저 끊는 쪽: FIN_WAIT_1 → FIN_WAIT_2)
              ◀─────────────   ACK           (받는 쪽: CLOSE_WAIT)
              ◀─────────────   FIN           (받는 쪽이 자기 할 일 마치고 끊을 때)
   ACK        ─────────────▶   (먼저 끊은 쪽: TIME_WAIT → 일정 시간 후 소멸)
  • LISTEN: 서버 프로세스가 특정 포트에서 연결 요청을 기다리는 상태 — "이 포트가 열려있다"는 것은 바로 이 상태의 소켓이 존재한다는 의미.
  • ESTABLISHED: 3-way handshake가 끝나고 실제 데이터를 주고받는 정상 연결 상태.
  • TIME_WAIT: 연결을 먼저 끊은(active close) 쪽에 남는 상태 — 뒤늦게 도착하는 패킷을 처리하기 위한 안전장치로, 일정 시간(보통 60초 전후) 후 자동 소멸.
  • CLOSE_WAIT: 상대가 먼저 끊었다는 FIN을 받았지만, 내 애플리케이션이 아직 close()를 호출하지 않은 상태 — 정상이라면 순식간에 지나가야 하며, 오래 쌓이면 애플리케이션 버그(소켓 leak)를 의심.
✅

면접 팁: TIME_WAIT은 "내가 먼저 끊었을 때" 생기고, CLOSE_WAIT은 "상대가 먼저 끊었는데 내가 아직 안 끊었을 때" 생긴다는 방향성을 정확히 설명할 수 있으면, 단순히 "연결 종료 상태"라고 뭉뚱그려 아는 사람과 구분됩니다.

2. ss / netstat — 포트와 연결 상태를 숫자로 확인한다

쉽게 말하면: ss는 커널의 소켓 테이블을 직접 읽는 최신 도구이고, netstat은 오래된 도구지만 여전히 많은 서버에 남아있습니다. 최신 배포판(특히 컨테이너 이미지)은 기본적으로 net-tools (netstat 포함 패키지)를 빼고 iproute2(ss 포함)만 넣는 추세라서, ss에 익숙한 것이 실무적으로 더 유리합니다. 

# 현재 LISTEN 중인 TCP/UDP 포트 전체 목록 (n=이름 조회 생략, 빠름)
$ ss -tulnp
Netid State  Local Address:Port   Peer Address:Port  Process
tcp   LISTEN 0.0.0.0:8080         0.0.0.0:*          users:(("myapp",pid=1234,fd=6))
tcp   LISTEN 127.0.0.1:5432       0.0.0.0:*          users:(("postgres",pid=980,fd=3))
# 모든 TCP 연결과 상태 (t=tcp, a=all, n=numeric)
$ ss -tan
# 상태별 연결 수 집계 — 폭증 여부를 한눈에 확인
$ ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn
  15234 ESTAB
  48213 TIME-WAIT
     312 CLOSE-WAIT
       2 LISTEN
  • -t: TCP만, -u: UDP만, -l: LISTEN 상태만, -n: 이름 조회(reverse DNS) 생략 — 대량 연결 환경에서 -n 없으면 명령어 자체가 매우 느려짐.
  • -p: 해당 소켓을 소유한 프로세스 정보 표시 — "8080 포트를 어떤 프로세스가 쓰고 있는가"를 바로 확인 가능 (root 권한 필요할 수 있음).
  • 127.0.0.1:5432 vs 0.0.0.0:8080: 바인딩 주소 차이가 매우 중요 — 127.0.0.1은 로컬호스트에서만 접속 가능(외부에서 절대 접속 불가), 0.0.0.0은 모든 인터페이스에서 접속 허용. "포트가 열려있는데 외부에서 안 된다"는 신고의 절반은 이 바인딩 주소 차이가 원인.
⚠️

실무 연결: DB를 127.0.0.1에만 바인딩하는 것은 보안상 의도적인 설정인 경우가 많습니다(외부 노출 방지). 이걸 모르고 무작정 0.0.0.0으로 바꾸면 포트는 열리지만 보안 경계가 뚫립니다 — "왜 이렇게 바인딩되어 있는지" 먼저 이해하고 바꾸는 것이 senior의 태도입니다.

3. TIME_WAIT 폭증 vs CLOSE_WAIT 폭증 — 원인이 완전히 다르다

쉽게 말하면: 둘 다 "연결이 안 없어지고 쌓인다"는 증상은 비슷해 보이지만, 원인과 대응은 정반대입니다. TIME_WAIT 폭증은 "정상적으로 빨리 끊었는데 뒷정리가 너무 많이 쌓이는" 규모의 문제이고, CLOSE_WAIT 폭증은 "애플리케이션이 끊었어야 할 걸 안 끊고 있는" 버그의 문제입니다.

구분 TIME_WAIT 폭증 CLOSE_WAIT 폭증
원인 짧은 연결을 매우 자주 새로 맺고 끊는 트래픽 패턴 (예: keep-alive 미사용 HTTP 호출) 애플리케이션이 소켓을 받은 뒤 close()를 호출하지 않는 코드 버그 (소켓 leak)
실질적 위험 ephemeral port(임시 포트) 고갈로 새 아웃바운드 연결을 못 맺음 파일 디스크립터 고갈(Day 2에서 다룬 ulimit -n) → 결국 프로세스가 새 연결을 못 받음
1차 대응 Connection keep-alive/재사용, 커넥션 풀 도입으로 연결 생성 자체를 줄임 애플리케이션 코드에서 소켓/커넥션을 닫는 경로(특히 예외 처리 경로)를 점검
임시방편(위험 동반) net.ipv4.tcp_tw_reuse=1 — 뒤에서 다룰 trade-off 주의 임시방편 없음 — 근본적으로 코드를 고쳐야 함(재시작은 증상만 잠시 없앰)
⚠️

trade-off: tcp_tw_reuse는 TIME_WAIT 소켓을 재사용해 포트 고갈을 완화하지만, NAT 뒤에 여러 클라이언트가 있는 환경에서는 타임스탬프 기반 판별이 꼬여 드물게 데이터 혼선이 발생할 수 있다는 것이 알려진 위험입니다. 근본 해결(연결 재사용 설계)보다 먼저 이 커널 파라미터부터 만지는 것은 "증상 억제"이지 "원인 해결"이 아니라는 점을 답변에서 짚어야 합니다.

4. tcpdump 기초 — "정말 패킷이 오고 가는지" 눈으로 확인한다

쉽게 말하면: ss가 "지금 연결 상태가 어떤지"를 커널 테이블에서 요약해서 보여준다면, tcpdump는 실제로 선을 타고 오가는 패킷 하나하나를 그대로 보여줍니다. "방화벽 문제인지, 애플리케이션이 응답을 안 하는지"를 구분할 때 결정적인 증거가 됩니다. 

# 8080 포트로 오가는 패킷만 필터링해서 확인 (n=이름 조회 생략, i=인터페이스)
$ sudo tcpdump -i eth0 -n port 8080
12:03:01.123 IP 10.0.1.5.51234 > 10.0.1.10.8080: Flags [S], seq 100, win 64240
# SYN만 보이고 SYN-ACK 응답이 없다면 → 방화벽이 막았거나, 그 포트에 리스닝 프로세스가 없음
$ sudo tcpdump -i eth0 -n port 8080 -c 20
  • SYN만 보이고 SYN-ACK이 없음: 클라이언트의 패킷이 서버에 도달했지만 응답이 없다 — 방화벽이 조용히 드롭(DROP)했거나, 그 포트에 아무도 LISTEN하고 있지 않은 상태(REJECT라면 보통 즉시 RST가 옴).
  • 서버 쪽에서 아예 SYN조차 안 보임: 패킷이 서버까지 도달하지 못했다 — 중간 방화벽/보안그룹, 라우팅 문제를 의심.
  • RST(Reset) 응답: 명시적 거부(방화벽 REJECT 정책) 또는 해당 포트에 아무도 듣고 있지 않을 때 커널이 자동으로 보내는 응답.
✅

면접 팁: "클라이언트에서는 SYN을 보냈다고 하는데, 서버의 tcpdump에서 그 SYN 자체가 안 보인다면 이건 서버 애플리케이션 문제가 아니라 그 전 단계(네트워크 경로, 보안그룹, 방화벽)의 문제"라는 식으로 캡처 위치를 근거로 어느 계층 문제인지 좁혀나가는 설명이 핵심입니다.

5. ip route — 패킷이 "어느 문으로 나가는지"의 지도

쉽게 말하면: 서버에 네트워크 인터페이스가 여러 개 있거나(사설망 + 공인망), VPN/터널이 얽혀 있으면 "이 목적지로 가는 패킷은 어느 인터페이스, 어느 게이트웨이로 나가야 하는지"를 커널이 라우팅 테이블을 보고 결정합니다. 이 테이블이 잘못되어 있으면 방화벽도 애플리케이션도 문제가 없는데 연결이 안 되는 상황이 생깁니다. 

# 현재 라우팅 테이블 확인
$ ip route
default via 10.0.1.1 dev eth0
10.0.1.0/24 dev eth0 proto kernel scope link src 10.0.1.10
172.31.0.0/16 dev eth1 proto kernel scope link src 172.31.5.20
# 특정 목적지로 나갈 때 커널이 실제로 어느 경로를 고르는지 시뮬레이션
$ ip route get 172.31.5.100
172.31.5.100 dev eth1 src 172.31.5.20
  • default via ... dev ...: 라우팅 테이블에 명시되지 않은 모든 목적지가 나가는 기본 게이트웨이 — 이게 잘못되면 외부 전체와 통신 불가.
  • ip route get <대상IP>: "내가 이 IP로 보내면 실제로 어느 인터페이스/소스 IP로 나가는지"를 즉시 확인 — 멀티 홈(여러 NIC) 서버 장애 진단의 핵심 명령어.
  • 보안그룹/방화벽은 정상인데 연결이 안 될 때, 의외로 비대칭 라우팅(요청은 A 경로로 나갔는데 응답은 B 경로로 들어와 상태 추적 방화벽이 응답을 매치 못 함)이 원인인 경우가 있습니다.
💡

실무 연결: 클라우드 환경에서 서버에 NIC를 하나 추가하거나 VPC peering/VPN을 새로 연결한 직후에 "일부 목적지만 연결이 안 된다"는 신고가 생기면, 십중팔구 라우팅 테이블에 새 경로가 누락되었거나 우선순위가 꼬인 경우입니다.

6. 방화벽 — iptables/nftables/firewalld/ufw는 결국 같은 커널 기능의 다른 얼굴

쉽게 말하면: 리눅스 방화벽은 커널의 netfilter 프레임워크 위에서 동작합니다. iptables(구세대), nftables(신세대 저수준 도구), firewalld(RHEL/CentOS 계열의 zone 기반 관리 도구), ufw(Ubuntu의 단순화된 프론트엔드)는 모두 이 하나의 커널 기능을 서로 다른 방식으로 설정하는 도구일 뿐입니다. 여러 도구가 동시에 규칙을 넣으면 서로 충돌하거나 덮어써서 예측 불가능한 상태가 될 수 있습니다. 

# 현재 활성화된 iptables 규칙 확인 (가장 저수준, 어떤 도구를 쓰든 최종 결과는 여기 반영됨)
$ sudo iptables -L -n -v --line-numbers
# firewalld 사용 시: 특정 포트가 열려있는지 zone 기준으로 확인
$ sudo firewall-cmd --list-all
# ufw 사용 시
$ sudo ufw status verbose
  • DROP vs REJECT: DROP은 패킷을 조용히 버림(클라이언트는 타임아웃까지 기다림 — 위 tcpdump에서 SYN만 있고 응답이 없는 패턴), REJECT는 명시적으로 거부 응답(RST/ICMP unreachable)을 보냄(클라이언트가 즉시 연결 거부를 인지).
  • 규칙은 순서(위에서 아래로 첫 매치)가 중요 — 뒤에 아무리 좋은 ALLOW 규칙을 추가해도, 앞쪽에 더 넓은 범위의 DROP 규칙이 있으면 무의미.
  • 클라우드 환경(AWS Security Group, GCP Firewall Rule 등)은 인스턴스 밖의 방화벽이므로, OS 내부 방화벽(iptables 등)이 다 열려있어도 클라우드 보안그룹이 막고 있으면 여전히 연결 실패 — 두 계층을 모두 확인해야 함.
⚠️

흔한 실수: firewalld가 설치된 서버에 별도로 raw iptables 명령어로 규칙을 추가하면, firewalld가 재시작되거나 reload될 때 그 수동 규칙이 조용히 사라질 수 있습니다. 방화벽을 관리하는 도구는 하나로 통일하고, "지금 이 서버는 어떤 도구가 방화벽을 관리하는가"를 먼저 확인하는 것이 원인 모를 규칙 소실을 막습니다.

🎯 왜 면접에서 묻는가

네트워크 진단 질문은 명령어 옵션 암기가 아니라, "연결 실패라는 하나의 증상 뒤에 있는 여러 계층(애플리케이션, OS 소켓, 방화벽, 라우팅, 물리적 경로) 중 어디가 문제인지 증거를 근거로 좁혀나갈 수 있는가"를 평가합니다.

면접관이 실제로 평가하는 것

  1. 계층을 구분해서 원인을 좁혀나가는가?
    "안 된다"는 증상 하나를 애플리케이션/소켓 상태/방화벽/라우팅 중 어디서부터 확인해야 가장 빠르게 원인을 좁힐 수 있는지 순서를 아는지를 봅니다.
  2. 연결 상태를 근거로 판단하는가?
    TIME_WAIT과 CLOSE_WAIT처럼 겉보기엔 비슷한 증상을 정확히 구분해서 서로 다른 원인과 대응을 제시할 수 있는지를 봅니다.
  3. 패킷 레벨 증거를 활용할 줄 아는가?
    tcpdump로 "패킷이 실제로 어디까지 도달했는지"를 확인해서 애플리케이션 문제와 인프라 문제를 명확히 구분할 수 있는지를 봅니다.
  4. 인프라 계층의 경계를 이해하는가?
    OS 내부 방화벽과 클라우드 보안그룹처럼, 눈에 보이지 않는 계층이 여러 겹 존재한다는 것을 인지하고 있는지를 봅니다.
💡

본질은 "네트워크는 여러 계층이 겹쳐진 시스템이며, 증상만으로 성급히 결론 내리지 않고 계층별로 증거를 모아 원인을 특정할 수 있는가"입니다. 이 능력이 부족하면 방화벽 팀, 네트워크 팀, 애플리케이션 팀 사이에서 책임 소재만 오가며 장애 시간이 길어집니다.

🏭 실무 연결 — 이런 상황에서 실제로 쓴다

시나리오 A: 배포 직후 특정 포트로 접속이 안 된다는 신고

새 버전을 배포한 직후 헬스체크가 실패한다는 알림을 받았습니다. ss -tulnp로 확인하니 애플리케이션이 127.0.0.1:8080에만 바인딩되어 있었습니다(이전 버전은 0.0.0.0:8080). 설정 파일의 바인딩 주소가 배포 스크립트에서 환경변수 기본값으로 잘못 덮어써진 것이 원인이었고, 설정을 되돌린 뒤 배포 파이프라인에 바인딩 주소 검증 단계를 추가했습니다.

시나리오 B: 외부 API 호출이 간헐적으로 실패하는 서버

결제 API를 호출하는 서버에서 간헐적으로 연결 실패가 발생한다는 신고를 받고 ss -tan | awk '{print $1}' | sort | uniq -c로 확인하니 TIME_WAIT 소켓이 4만 개 넘게 쌓여 있었습니다. 매 요청마다 새 연결을 맺고 끊는 구조가 원인이었고, HTTP 클라이언트에 커넥션 풀을 도입해 연결을 재사용하도록 바꾼 뒤 TIME_WAIT 수가 정상 범위로 줄었습니다. tcp_tw_reuse는 NAT 환경 위험 때문에 최후 수단으로만 검토 대상에 남겨두었습니다.

시나리오 C: 새 NIC 추가 후 특정 대역만 통신 불가

사설망용 NIC를 하나 추가한 뒤 일부 내부 서버와만 통신이 안 된다는 신고를 받았습니다. ip route get <대상IP>로 확인하니 그 대역으로 가는 경로가 라우팅 테이블에 없어 기본 게이트웨이(공인망 NIC)로 나가고 있었습니다. 정적 라우트를 추가해 문제를 해결했고, tcpdump로 해당 인터페이스에서 실제 응답 패킷이 돌아오는 것을 확인해 조치를 검증했습니다.

실무에서 배운 핵심 교훈

  • "포트가 열려있다"는 말은 바인딩 주소까지 확인해야 완전한 확인이다 — 127.0.0.1과 0.0.0.0은 완전히 다른 이야기다.
  • TIME_WAIT 폭증은 트래픽 패턴(연결 재사용 부재) 문제, CLOSE_WAIT 폭증은 애플리케이션 버그 — 같은 "연결이 쌓인다"는 증상도 원인은 정반대일 수 있다.
  • tcpdump로 "패킷이 실제로 어디까지 도달했는지" 확인하면, 여러 팀이 서로 책임을 미루는 상황에서 객관적인 증거로 원인을 좁힐 수 있다.
  • OS 방화벽이 다 열려있어도 클라우드 보안그룹이라는 또 다른 계층이 막고 있을 수 있다 — 항상 "밖에서부터 안으로" 계층을 하나씩 확인해야 한다.

🧠 답변 사고 순서 — 네트워크 연결 실패 진단 흐름

  • 서버 안에서 포트가 실제로 열려있는지 확인

    ss -tulnp | grep <port>로 해당 포트가 LISTEN 상태인지, 어떤 주소(0.0.0.0인지 127.0.0.1인지)에 바인딩됐는지, 어떤 프로세스가 소유하고 있는지부터 확인합니다.

  • 연결 상태 분포로 규모 문제인지 확인

    ss -tan | awk '{print $1}' | sort | uniq -c로 TIME_WAIT/CLOSE_WAIT이 비정상적으로 많이 쌓여있지 않은지 먼저 확인해, 포트 고갈이나 fd 고갈 같은 리소스 문제가 이미 진행 중인지 배제합니다. 

    $ ss -tulnp | grep 8080
$ ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn

  • 패킷 캡처로 실제 도달 여부를 증거로 확인

    서버 쪽에서 tcpdump로 필터를 걸어, 클라이언트가 보낸 SYN이 실제로 서버까지 도달하는지, 도달했다면 응답(SYN-ACK/RST)이 나가는지를 확인합니다. SYN 자체가 안 보이면 문제는 서버 이전 구간(방화벽/보안그룹/라우팅)에 있다는 강한 증거입니다.

  • 라우팅과 방화벽 계층을 순서대로 점검

    ip route get으로 실제 나가는 경로를 확인하고, OS 방화벽 (iptables/firewalld/ufw)을 점검한 뒤, 클라우드 환경이라면 보안그룹/네트워크 ACL 같은 인스턴스 밖의 계층까지 확인 범위를 넓힙니다.

  • 원인 계층에 맞는 조치와 재발 방지

    원인이 애플리케이션 설정(바인딩 주소)이면 배포 검증 단계를, 연결 관리 문제(TIME_WAIT/CLOSE_WAIT)면 커넥션 풀이나 코드 수정을, 인프라 문제(라우팅/방화벽)면 해당 팀과 협업해 변경 이력을 남기고 유사 변경 시 체크리스트에 반영합니다.

💬 핵심 답변 예시

면접에서 자연스럽게 말할 수 있는 구조입니다. 외우지 말고 논리의 흐름을 이해하세요.

💬 면접관: "8080 포트로 연결이 안 된다는 신고를 받았습니다. 어떻게 접근하시겠습니까?"

"가장 먼저 서버 안에서 ss -tulnp | grep 8080으로 그 포트가 실제로 LISTEN 상태인지, 그리고 어떤 주소에 바인딩되어 있는지부터 확인합니다. 127.0.0.1에만 바인딩되어 있다면 외부 접속 자체가 애초에 불가능한 설정 문제이므로 여기서 원인이 바로 나올 수 있습니다.

포트도 정상적으로 0.0.0.0에 열려있다면, tcpdump로 실제 클라이언트의 SYN 패킷이 서버까지 도달하는지 확인합니다. SYN 자체가 안 보인다면 문제는 서버 애플리케이션이 아니라 그 앞 단계, 즉 방화벽이나 클라우드 보안그룹, 라우팅에 있다는 강한 증거가 됩니다.

SYN은 도달하는데 응답이 없다면 OS 방화벽(iptables 또는 firewalld/ufw) 규칙을 확인하고, 여기서도 문제가 없다면 클라우드 보안그룹처럼 인스턴스 밖의 계층까지 확인 범위를 넓힙니다. 이렇게 안에서 바깥으로 계층별 증거를 쌓아가면, 어느 팀이 처리해야 할 문제인지 빠르게 명확해집니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "TIME_WAIT 소켓이 많은 것 자체가 항상 문제인가요?"

"아닙니다. TIME_WAIT은 TCP 프로토콜이 뒤늦게 도착하는 패킷을 안전하게 처리하기 위해 의도적으로 설계된 정상 동작입니다. 트래픽이 많은 서버라면 수천 개의 TIME_WAIT은 자연스러운 현상입니다. 문제가 되는 건 ephemeral port 범위(기본적으로 약 28,000~32,000개)를 소진할 만큼 쌓여서 새 아웃바운드 연결을 못 맺는 지경에 이를 때입니다. 그래서 저는 '몇 개가 있는가'보다 '연결을 새로 맺는 데 실패하는 에러가 실제로 발생하는가'를 기준으로 문제 여부를 판단합니다."

평가 포인트: 지표 자체가 아니라 실제 영향(포트 고갈로 인한 연결 실패) 기준으로 문제를 정의하는가

Q. "DROP과 REJECT 중 어떤 방화벽 정책을 선호하시나요? 왜인가요?"

"상황에 따라 다릅니다. 외부에 노출된 경계 방화벽이라면 DROP을 선호합니다 — 공격자에게 '포트가 존재하지만 막혀있다'는 정보조차 주지 않아 스캐닝 시간을 늘리는 효과가 있기 때문입니다. 반면 내부망에서 서비스 간 통신을 의도적으로 차단할 때는 REJECT를 선호합니다 — 클라이언트가 DROP으로 인한 타임아웃(보통 수십 초)을 그대로 기다리게 하면 장애 전파가 느려지고 사용자 체감 지연이 커지기 때문입니다. 결국 '보안 은닉의 이득'과 '실패를 빠르게 알려서 시스템이 빨리 복구 로직을 타게 하는 이득' 사이의 trade-off입니다."

평가 포인트: 보안과 장애 전파 속도라는 서로 다른 축의 trade-off를 이해하는가

Q. "CLOSE_WAIT이 계속 쌓이는 걸 발견했는데, 재시작 말고 근본 원인을 어떻게 찾으시겠습니까?"

"재시작은 증상만 없앨 뿐 같은 코드 경로를 다시 타면 또 쌓입니다. 먼저 ss -tan state close-wait로 어느 원격 주소와의 연결이 CLOSE_WAIT에 몰려있는지 확인하고, lsof -p <pid>로 그 소켓 fd를 애플리케이션 코드의 어느 부분과 연결할 수 있는지 유추합니다. 대부분은 예외가 발생했을 때 finally나 defer 블록에서 소켓을 닫는 처리가 누락된 코드 경로가 원인이므로, 최근 배포 이력에서 예외 처리 로직이 바뀐 부분을 우선 의심합니다."

평가 포인트: 재시작을 임시방편으로 인지하고 코드 레벨 원인 추적까지 연결하는가

Q. "온프레미스와 클라우드 환경에서 네트워크 장애 진단 절차에 어떤 차이가 있나요?"

"OS 내부 진단(ss, tcpdump, ip route, iptables/firewalld)은 동일하게 적용됩니다. 다만 클라우드 환경에서는 인스턴스 밖에 있는 계층이 하나 더 있습니다 — 보안그룹, 네트워크 ACL, 그리고 관리형 로드밸런서의 헬스체크 설정 같은 것들입니다. 이런 계층은 tcpdump로도 보이지 않기 때문에(패킷이 인스턴스에 도달하기 전에 이미 걸러짐), OS 안에서 아무 문제가 없다고 확인됐다면 곧바로 클라우드 콘솔이나 IaC 설정으로 시야를 넓혀야 한다는 점이 다릅니다."

평가 포인트: OS 레벨 도구의 관측 범위 한계를 알고 클라우드 계층까지 시야를 넓히는가

Q. "네트워크 대역폭이나 패킷 손실이 의심될 때는 어떤 도구를 추가로 쓰시나요?"

"연결 자체는 되는데 느리거나 간헐적으로 실패한다면 ip -s link로 인터페이스의 에러/드롭 카운터를 먼저 확인합니다. 여기서 드롭이 늘고 있다면 NIC나 스위치 포트 레벨 문제를 의심하고, mtr(traceroute와 ping을 결합한 도구)로 구간별 손실률과 지연을 확인해 어느 홉에서 문제가 생기는지 좁힙니다. 이건 Day 1의 USE 방법론과 같은 사고입니다 — 네트워크 인터페이스도 결국 자원이므로 사용률/포화/에러 세 가지를 체계적으로 확인하는 것입니다."

평가 포인트: 연결 상태 진단을 넘어 물리적 경로 성능 진단까지 확장하고, 이전에 배운 방법론과 연결하는가

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"방화벽을 확인해보고, 안 되면 서버를 재시작해보겠습니다."
왜 약한 답변인가:
  • 포트가 실제로 LISTEN 상태인지, 어느 주소에 바인딩됐는지 확인하는 단계가 없습니다.
  • "방화벽 확인"이 구체적으로 어떤 명령어와 어떤 근거로 확인하는지가 없습니다.
  • 여러 계층(애플리케이션/소켓/방화벽/라우팅) 중 어디부터 봐야 하는지 우선순위가 없습니다.
좋은 답변 (Senior 수준)
"먼저 ss로 포트의 LISTEN 상태와 바인딩 주소를 확인하고, tcpdump로 패킷이 실제로 서버까지 도달하는지 증거를 확보한 뒤, 그 결과에 따라 방화벽·라우팅·클라우드 보안그룹 순으로 확인 범위를 넓힙니다."
왜 좋은 답변인가:
  • 안(애플리케이션)에서 밖(네트워크 경계)으로 확인 순서가 명확합니다.
  • 패킷 캡처를 "증거"로 활용해 추측이 아니라 근거 기반으로 판단합니다.
  • OS 내부 방화벽과 클라우드 보안그룹처럼 서로 다른 계층을 구분해서 확인합니다.
  • 바인딩 주소(0.0.0.0 vs 127.0.0.1) 같은 구체적인 확인 포인트를 짚습니다.

🚫 흔한 오해와 정확한 이해

오해 1: "프로세스가 그 포트로 떠 있으면 외부에서 무조건 접속할 수 있다"

바인딩 주소가 127.0.0.1이면 로컬호스트에서만 접속 가능하며 외부에서는 절대 접속할 수 없습니다. 0.0.0.0(또는 특정 외부 인터페이스 IP)에 바인딩되어 있어야 외부 접속이 가능합니다.

오해 2: "TIME_WAIT이 많이 보이는 것은 그 자체로 버그다"

TIME_WAIT은 TCP 프로토콜이 의도적으로 설계한 정상 동작입니다. 트래픽이 많은 서버에서는 자연스럽게 많이 관찰되며, 문제는 개수 자체가 아니라 ephemeral port 고갈로 실제 연결 실패가 발생하는지 여부입니다.

오해 3: "CLOSE_WAIT과 TIME_WAIT은 비슷한 문제니까 대응도 비슷하다"

TIME_WAIT은 트래픽 패턴(연결을 자주 새로 맺음)의 문제로 커넥션 재사용으로 완화하지만, CLOSE_WAIT은 애플리케이션이 소켓을 닫지 않는 코드 버그로 코드 수정 없이는 근본적으로 해결되지 않습니다. 재시작은 증상만 잠시 없앨 뿐입니다.

오해 4: "OS 방화벽만 확인하면 연결 문제의 전부를 알 수 있다"

클라우드 환경에서는 인스턴스 밖의 보안그룹, 네트워크 ACL 같은 계층이 별도로 존재합니다. tcpdump로도 이 계층에서 걸러진 패킷은 보이지 않으므로, OS 내부가 모두 정상이라면 반드시 그 바깥 계층까지 확인해야 합니다.

오해 5: "tcpdump는 위험하니 함부로 쓰면 안 된다"

적절한 필터(특정 포트/호스트)를 걸어 사용하면 프로덕션에서도 안전하게 쓸 수 있는 표준 진단 도구입니다. 다만 필터 없이 전체 트래픽을 캡처하면 디스크를 빠르게 채우고 성능에 영향을 줄 수 있으므로, 반드시 포트/호스트 필터와 캡처 개수 제한 (-c)을 함께 사용하는 것이 실무 관행입니다.

Day 7 핵심 포인트 요약

  • "포트가 열려있다"는 바인딩 주소까지 확인해야 완전한 확인이다.
  • TIME_WAIT은 연결을 먼저 끊은 쪽에 생기는 정상 동작, CLOSE_WAIT은 상대가 끊었는데 내가 안 끊은 버그 신호 — 방향성으로 구분한다.
  • tcpdump로 패킷의 실제 도달 여부를 증거로 확보해서 계층을 좁혀나간다.
  • OS 내부 방화벽과 클라우드 보안그룹은 서로 다른 계층이며 둘 다 확인해야 한다.
  • 임시방편(tcp_tw_reuse 등)은 trade-off를 이해하고 최후 수단으로만 검토한다.
