"배포한 서비스가 몇 분 간격으로 계속 재시작되고 있습니다(crash loop). systemctl status만으로 어디까지 파악할 수 있고, 원인을 어떻게 좁혀나가시겠습니까?"

관련 질문

"새로 작성한 myapp.service unit 파일이 있는데, 서버가 재부팅되면 이 서비스가 DB 연결에 실패한 채로 시작됩니다. unit 파일의 어떤 설정이 원인일 수 있을까요?"

관련 질문

"Restart=always로 설정했더니 애플리케이션 버그로 무한 재시작 루프에 빠져서 CPU를 100% 소모하고 있습니다. 이 설정을 어떻게 개선하시겠습니까?"

💡

Day 1~5가 "이미 떠 있는 프로세스"를 진단하는 능력이었다면, Day 6은 "그 프로세스가 어떻게 태어나고, 죽고, 다시 태어나는지"를 관리하는 계층을 다룹니다. 면접관이 보는 것은 systemctl restart 같은 명령어 암기가 아니라, unit 파일의 선언이 실제 장애 상황(재시작 폭주, 시작 순서 실패, 자원 제한)에서 어떻게 동작하는지 예측할 수 있는가입니다.

📚 전제 개념 — 이것을 먼저 이해해야 한다

1. systemd란 무엇인가 — PID 1과 "관리자"의 역할

쉽게 말하면: 리눅스 커널이 부팅되면 가장 먼저 실행되는 프로세스가 하나 있고, 이 프로세스가 다른 모든 프로세스의 "조상"이 됩니다(PID 1). 현대 리눅스 배포판 대부분(Ubuntu, RHEL/CentOS, Debian 등)은 이 역할을 systemd가 맡습니다. 회사에 비유하면 systemd는 "신입 사원을 언제 채용하고, 문제가 생기면 재고용하고, 퇴사 처리를 하는 인사팀"입니다. 여기서 각 "직원"에 해당하는 것이 unit이고, 그중 프로세스를 관리하는 unit이 service unit입니다. 

# 현재 시스템의 init 시스템이 systemd인지 확인
$ ps -p 1 -o comm=
systemd
# 모든 서비스 unit의 상태 목록
$ systemctl list-units --type=service --state=running
# 특정 서비스의 상세 상태 (활성 여부, 최근 로그 일부, PID)
$ systemctl status myapp.service
✅

면접 팁: "systemd는 단순 프로세스 실행기가 아니라 의존성 순서 관리, 실패 시 재시작, 자원 제한(cgroup), 소켓 활성화까지 포함하는 서비스 관리 프레임워크"라는 한 문장으로 정의할 수 있으면, systemctl 명령어만 아는 사람과 구분됩니다.

2. systemctl 핵심 명령 — 상태 전이를 정확히 구분한다

쉽게 말하면: start/stop은 "지금 당장" 서비스를 켜고 끄는 것이고, enable/disable은 "다음 부팅부터" 자동으로 켜질지 여부를 등록하는 것입니다. 이 둘은 완전히 독립적인 축입니다 — 지금 실행 중이라고 해서 다음 부팅에도 실행된다는 보장이 없고, enable 되어있다고 지금 실행 중이라는 보장도 없습니다. 

# 지금 당장 시작 / 중지 / 재시작
$ sudo systemctl start myapp.service
$ sudo systemctl stop myapp.service
$ sudo systemctl restart myapp.service
# 설정 파일만 다시 읽히면 되는 애플리케이션이면 무중단으로 시도
$ sudo systemctl reload myapp.service
# 다음 부팅부터 자동 시작되도록 등록 / 해제 (지금 상태와는 무관)
$ sudo systemctl enable myapp.service
$ sudo systemctl disable myapp.service
# unit 파일을 고쳤다면 커널/systemd가 새 정의를 읽도록 반드시 필요
$ sudo systemctl daemon-reload
조합 의미
running + enabled 정상 — 지금 실행 중이고 재부팅 후에도 자동 시작
running + disabled 누군가 수동으로 start만 했다 — 재부팅하면 서비스가 사라짐(운영 사고의 흔한 원인)
stopped + enabled 부팅 시 시작 시도는 하지만 실패했거나 수동으로 stop한 상태 — 원인 확인 필요
⚠️

실무 연결: "긴급 배포라서 systemctl start만 하고 enable을 깜빡했다"는 실수는 실제로 흔합니다. 다음 서버 재부팅(패치 적용, 인스턴스 교체 등) 시 서비스가 조용히 사라져서 장애로 이어집니다. 배포 체크리스트에 systemctl is-enabled myapp.service 확인을 넣는 것이 이런 사고를 예방하는 senior의 습관입니다.

3. unit 파일 구조 — 선언적으로 "무엇을, 언제, 어떻게" 정의한다

쉽게 말하면: unit 파일은 "이 서비스를 실행하는 방법"에 대한 설명서입니다. 셸 스크립트처럼 절차를 나열하는 것이 아니라, "이런 조건에서 이렇게 시작하고, 이렇게 실패를 처리하라"는 선언적 설정입니다. 크게 [Unit](메타정보와 의존성), [Service](실행 방법과 재시작 정책), [Install](enable 시 어디에 연결할지) 세 섹션으로 나뉩니다. 

# /etc/systemd/system/myapp.service
[Unit]
Description=My Application Server
After=network-online.target postgresql.service
Wants=network-online.target
Requires=postgresql.service

[Service]
Type=simple
User=myapp
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/bin/server --config /etc/myapp/config.yaml
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5
StartLimitIntervalSec=200
StartLimitBurst=5
TimeoutStopSec=30
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
  • [Unit] After=: 시작 순서만 보장 (postgresql이 이 unit보다 먼저 시작되도록 순서를 정할 뿐, postgresql이 반드시 존재해야 한다는 의미는 아님).
  • [Unit] Requires=: 의존성을 강제 — 대상이 실패하면 이 unit도 함께 실패 처리됨. After=와 반드시 짝을 이뤄야 순서+강제 의존이 모두 보장됨.
  • [Service] Type=simple: ExecStart로 실행한 프로세스 자체가 메인 프로세스 (가장 흔한 경우). Type=forking은 데몬화(자기 자신을 fork해서 부모를 종료)하는 전통적인 프로그램에 사용.
  • [Service] ExecStart: 반드시 절대 경로여야 하며, 실패 시 재시작 정책(Restart=)이 적용되는 대상.
  • [Install] WantedBy=multi-user.target: enable 시 이 target이 활성화될 때 함께 시작되도록 심볼릭 링크가 생성됨 — 이 섹션이 없으면 enable 자체가 동작하지 않음.
✅

면접 팁: After=와 Requires=/Wants=의 차이를 "순서 vs 강제성"으로 명확히 구분해서 설명하면, unit 파일을 복사-붙여넣기만 해본 사람과 실제로 설계해본 사람의 차이가 드러납니다.

4. Restart 정책 — "다시 켜는 것"과 "무한 루프"는 한 끗 차이

쉽게 말하면: Restart=는 "프로세스가 죽으면 자동으로 다시 켜라"는 지시이고, 이것만 켜두면 애플리케이션 자체에 버그가 있을 때 "죽고 → 켜지고 → 또 죽고"를 초당 수십 번 반복하며 CPU와 로그를 낭비할 수 있습니다(crash loop). systemd는 이를 막기 위해 속도 제한(rate limit) 장치를 함께 제공합니다. 

[Service]
Restart=on-failure       # 0이 아닌 종료 코드/시그널로 죽었을 때만 재시작
RestartSec=5             # 재시작 전 5초 대기 (즉시 재시작으로 인한 폭주 완화)
StartLimitIntervalSec=200  # 200초 동안
StartLimitBurst=5          # 5번 넘게 재시작을 시도하면 → 그 이후는 포기하고 failed 상태로 전환
# 재시작 폭주로 failed 상태에 갇혔을 때 초기화 후 재시도
$ sudo systemctl reset-failed myapp.service
$ sudo systemctl start myapp.service
  • Restart=no(기본값): 자동 재시작 안 함 — 배치 작업성 unit에 적합.
  • Restart=on-failure: 비정상 종료(0이 아닌 exit code, 시그널에 의한 종료)일 때만 재시작 — 대부분의 서버 프로세스에 권장되는 균형점.
  • Restart=always: 정상 종료를 포함해 어떤 이유로든 죽으면 무조건 재시작 — 의도적인 systemctl stop은 예외로 처리되지만, 애플리케이션 버그로 인한 즉시 재크래시에는 특히 주의가 필요.
  • StartLimitBurst/StartLimitIntervalSec 없이 Restart=always만 쓰면, 버그가 있는 배포판이 CPU를 100% 소모하며 무한 재시작할 위험이 있습니다.
⚠️

trade-off: RestartSec을 너무 짧게 하면 재시작이 문제 해결의 시간을 벌어주지 못하고 폭주하고, 너무 길게 하면 실제 장애(예: 일시적 네트워크 문제로 죽은 경우) 복구가 느려집니다. 보통 5~10초 정도로 시작해 애플리케이션의 평균 복구 시간에 맞춰 조정합니다.

5. systemctl status 해석 — 세 줄 요약 뒤에 숨은 정보

쉽게 말하면: systemctl status는 짧은 요약을 보여주지만, 각 필드가 정확히 무엇을 의미하는지 알아야 다음 조사 방향을 정할 수 있습니다. 

$ systemctl status myapp.service
● myapp.service - My Application Server
     Loaded: loaded (/etc/systemd/system/myapp.service; enabled; vendor preset: enabled)
     Active: activating (auto-restart) (Result: exit-code) since Sat 2026-07-04 03:12:41 UTC; 4s ago
    Process: 18422 ExecStart=/opt/myapp/bin/server --config /etc/myapp/config.yaml (code=exited, status=1/FAILURE)
   Main PID: 18422 (code=exited, status=1/FAILURE)
        CPU: 89ms
  • Loaded: unit 파일 자체를 systemd가 정상적으로 읽었는지 (경로, enabled 여부). not-found면 파일 경로나 이름 오타부터 의심.
  • Active: 실제 실행 상태. active (running)은 정상, activating (auto-restart)는 방금 죽어서 재시작을 시도 중(위 예시), failed는 재시작 한도(StartLimitBurst)까지 소진하고 포기한 상태.
  • Process/code=exited, status=1: 프로세스가 어떤 종료 코드로 끝났는지 — status=1/FAILURE은 애플리케이션이 명시적으로 비정상 종료했다는 의미로, 애플리케이션 자체 로그를 봐야 진짜 원인을 알 수 있음.
  • 화면에 보이는 로그 몇 줄은 요약일 뿐입니다 — 전체 맥락은 Day 5에서 다룬 journalctl -u myapp.service로 확인해야 합니다.
✅

면접 팁: "status의 Active 필드가 activating (auto-restart)라면 이미 crash loop 중이라는 신호이므로, 재시작 카운트가 한도를 넘기 전에 서둘러 journalctl -u myapp -n 50으로 마지막 실패 직전 로그를 확인해야 한다"는 판단을 보여주면 상태 필드를 진짜로 읽을 줄 안다는 신호가 됩니다.

🎯 왜 면접에서 묻는가

systemd 질문은 명령어 암기가 아니라, "프로세스의 생명주기를 선언적으로 설계하고 실패 상황을 예측할 수 있는가"를 평가합니다. 이는 배포 안정성과 직결되는 운영 역량입니다.

면접관이 실제로 평가하는 것

  1. 선언과 실제 동작을 연결할 수 있는가?
    unit 파일의 각 설정(Restart=, After=, Type=)이 장애 상황에서 실제로 어떻게 동작하는지 예측할 수 있는지를 봅니다.
  2. 순서 보장과 강제 의존을 구분하는가?
    After=와 Requires=/Wants=를 혼동하지 않고, 의도한 의존성 그래프를 정확히 설계할 수 있는지를 봅니다.
  3. 재시작 정책의 이면을 이해하는가?
    Restart=always가 만능이 아니라 crash loop 위험이 있다는 것을 알고, 속도 제한 설정과 함께 써야 한다는 균형 감각이 있는지를 봅니다.
  4. status 출력을 진단 도구로 쓸 수 있는가?
    요약된 상태 필드 하나하나(Loaded/Active/Process)를 근거로 다음 조사 단계를 정할 수 있는지를 봅니다.
💡

본질은 "배포 자동화와 장애 복원력을 시스템 수준에서 선언할 수 있는가"입니다. 이 능력이 부족하면 매번 사람이 수동으로 서비스를 지켜보고 재시작해줘야 합니다.

🏭 실무 연결 — 이런 상황에서 실제로 쓴다

시나리오 A: 배포 직후 서비스가 crash loop에 빠진 경우

새 버전 배포 직후 모니터링 알림이 폭주해서 systemctl status myapp을 확인하니 Active: activating (auto-restart)였습니다. 즉시 journalctl -u myapp -n 50 --no-pager로 마지막 실패 직전 로그를 확인해 새 설정 파일의 필수 환경변수 누락이 원인임을 확인했고, StartLimitBurst가 소진되어 failed로 전환되기 전에 롤백을 완료했습니다.

시나리오 B: 재부팅 후 애플리케이션이 DB 연결 실패로 죽어있던 경우

서버 패치 후 재부팅했더니 myapp.service가 실행되지 않았습니다. unit 파일에 After=postgresql.service만 있고 Requires=가 없어서, PostgreSQL이 아직 초기화 중일 때 myapp이 먼저 연결을 시도하다 실패했습니다. Requires=postgresql.service를 추가하고, 애플리케이션 쪽에도 연결 재시도 로직을 보강해 이중으로 방어했습니다.

시나리오 C: enable을 깜빡해서 재부팅 후 서비스가 사라진 경우

긴급 핫픽스를 systemctl start로만 적용하고 enable을 깜빡한 채 다음 정기 점검에서 서버가 재부팅되어 서비스가 조용히 내려가 있었습니다. 이후 배포 체크리스트에 systemctl is-enabled myapp.service 확인 단계를 추가해 같은 실수를 재발 방지했습니다.

실무에서 배운 핵심 교훈

  • Active 필드가 activating (auto-restart)로 보이면, 이미 재시작을 반복 중이라는 신호이므로 한도가 소진되기 전에 원인을 확인해야 한다.
  • After=만으로는 순서만 보장될 뿐 대상이 준비됐다는 보장은 없다 — 진짜 강한 의존이 필요하면 Requires=와 애플리케이션 레벨 재시도를 함께 쓴다.
  • start와 enable은 서로 다른 축이므로, 배포 체크리스트에 둘 다 확인하는 항목을 넣어야 한다.
  • unit 파일을 수정한 뒤 daemon-reload를 잊으면 이전 정의로 계속 동작하므로, 변경 → reload → restart 순서를 습관화해야 한다.

🧠 답변 사고 순서 — systemd 서비스 장애 진단 흐름

  • 현재 상태부터 정확히 읽는다

    systemctl status의 Active 필드로 지금이 정상 실행인지, 재시작 반복 중인지, 이미 포기한(failed) 상태인지부터 구분합니다.

  • enabled/실행 상태를 별개로 확인

    systemctl is-enabled와 is-active를 함께 확인해서, "지금 떠 있는가"와 "재부팅해도 다시 뜨는가"를 혼동하지 않습니다.

  • 실패 직전 로그로 근본 원인 확인

    Day 5에서 다룬 journalctl -u myapp -n 50으로 애플리케이션이 남긴 마지막 로그를 확인해, unit 설정 문제인지 애플리케이션 버그인지 구분합니다. 

    $ systemctl status myapp.service
$ systemctl is-enabled myapp.service
$ journalctl -u myapp.service -n 50 --no-pager

  • unit 파일 설정을 근본 원인과 연결

    의존성 순서 문제라면 After=/Requires=를, 재시작 정책 문제라면 Restart=/StartLimitBurst=를 점검합니다.

  • 수정 후 재적용과 재발 방지

    unit 파일을 고쳤다면 daemon-reload 후 restart하고, 재시작 한도가 소진돼 failed 상태라면 reset-failed 후 재시작합니다. 마지막으로 배포 체크리스트나 알림 임계치를 재발 방지 관점에서 보완합니다.

💬 핵심 답변 예시

면접에서 자연스럽게 말할 수 있는 구조입니다. 외우지 말고 논리의 흐름을 이해하세요.

💬 면접관: "배포한 서비스가 몇 분 간격으로 계속 재시작되고 있습니다. 어떻게 접근하시겠습니까?"

"먼저 systemctl status myapp로 Active 필드를 확인해서 지금이 activating (auto-restart)인지, 이미 재시작 한도를 소진해 failed로 넘어갔는지부터 구분합니다. 이 구분에 따라 얼마나 급하게 움직여야 하는지가 달라지기 때문입니다.

아직 재시작을 반복 중이라면, journalctl -u myapp -n 50으로 마지막으로 죽기 직전의 로그를 확인해서, 이게 unit 설정 문제(예: 의존 서비스가 아직 준비되지 않음)인지 애플리케이션 자체 버그인지 구분합니다.

원인이 애플리케이션 버그로 확인되면, 무한 재시작으로 인한 CPU 낭비를 막기 위해 일단 Restart= 정책과 StartLimitBurst가 적절히 설정돼 있는지 점검하고, 필요하면 롤백을 우선 진행합니다. 근본 원인이 unit 파일의 의존성 누락(Requires= 부재)이었다면 이를 수정한 뒤 daemon-reload로 반영하고, 배포 체크리스트에 재발 방지 항목을 추가합니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "After=를 설정했는데도 DB가 준비되기 전에 애플리케이션이 시작해서 실패합니다. 왜 그런가요?"

"After=는 systemd가 유닛을 시작하는 순서만 보장할 뿐, 대상 서비스가 '연결을 받을 준비가 완료됐다'는 것까지 보장하지 않습니다. 예를 들어 PostgreSQL의 systemd unit은 프로세스가 실행되기 시작한 시점에 이미 active로 표시될 수 있지만, 실제 초기화(WAL 복구 등)에는 몇 초가 더 걸릴 수 있습니다. 이 간극을 메우려면 Requires=로 강한 의존을 추가하는 것 외에도, 애플리케이션 레벨에서 연결 재시도(retry with backoff) 로직을 두는 것이 근본적인 해결책입니다. systemd의 순서 보장만으로는 애플리케이션 레벨의 '준비 완료' 신호를 대체할 수 없습니다."

평가 포인트: systemd 의존성의 한계를 알고 애플리케이션 레벨 방어까지 설계하는가

Q. "Restart=always와 StartLimitBurst를 함께 설정했는데도 왜 위험할 수 있나요?"

"StartLimitBurst는 재시작 폭주를 막아 failed 상태로 전환시켜줄 뿐, 그 한도(예: 200초에 5번)에 도달하기 전까지는 여전히 빠르게 반복 재시작하며 CPU와 로그, 그리고 다운스트림 시스템(DB 커넥션 시도 등)에 부하를 줄 수 있습니다. 특히 RestartSec이 너무 짧으면 한도에 도달하기까지 걸리는 시간 동안 이미 상당한 부하가 발생합니다. 그래서 RestartSec을 적절히(수 초 이상) 설정하고, 애플리케이션 자체에도 '연속 실패 시 자체적으로 지연 후 재시도'하는 방어 로직을 두는 것이 더 안전합니다."

평가 포인트: 속도 제한이 있어도 한도 도달 전까지의 부하를 인지하는가

Q. "Type=simple과 Type=forking의 차이가 실무에서 왜 중요한가요?"

"Type=simple은 ExecStart로 실행한 프로세스 자체를 메인 프로세스로 간주하고, 그 프로세스가 시작되는 즉시(실제로 서비스 준비가 되었는지와 무관하게) systemd가 'active'로 표시합니다. 반면 Type=forking은 전통적인 데몬처럼 부모 프로세스가 fork 후 종료되는 방식을 다루며, PID 파일을 통해 실제 메인 프로세스를 추적합니다. Type을 잘못 지정하면 systemd가 프로세스 상태를 잘못 추적해서, 실제로는 아직 초기화 중인데 active로 잘못 보고하거나, 반대로 정상 동작 중인데 실패로 오판할 수 있습니다. 최신 애플리케이션 대부분은 Type=simple이나 준비 완료 신호를 명시적으로 보내는 Type=notify를 사용하는 것이 더 안전합니다."

평가 포인트: Type 설정이 상태 추적 정확성에 미치는 영향을 이해하는가

Q. "systemd unit의 자원 제한(cgroup 기반)을 활용해본 적이 있나요? 왜 필요한가요?"

"MemoryMax=, CPUQuota=, LimitNOFILE= 같은 설정을 unit 파일에 선언하면, systemd가 cgroup을 통해 해당 서비스가 시스템 전체 자원을 독점하지 못하도록 제한할 수 있습니다. 예를 들어 메모리 누수가 있는 서비스라도 MemoryMax를 설정해두면 시스템 전체 OOM(Day 3에서 다룬 OOM Killer가 무관한 다른 프로세스까지 죽이는 상황)으로 번지기 전에 해당 서비스만 먼저 제한을 받게 할 수 있습니다. 이는 '장애의 영향 범위를 격리한다'는 운영 원칙을 systemd 레벨에서 구현하는 방법입니다."

평가 포인트: unit 파일을 프로세스 실행 그 이상의 자원 격리 도구로 활용할 줄 아는가

Q. "여러 서비스가 서로 의존하는 복잡한 시스템에서 systemd만으로 배포 순서를 관리하는 것의 한계는 무엇인가요?"

"단일 서버 안에서의 시작 순서는 After=/Requires=로 충분히 표현할 수 있지만, 여러 서버·컨테이너에 걸친 서비스 의존성(예: 서비스 A가 다른 호스트의 서비스 B를 필요로 하는 경우)은 systemd의 범위를 벗어납니다. 이런 경우에는 헬스체크 기반의 오케스트레이션(Kubernetes의 readiness probe 등)이나 서비스 디스커버리, 그리고 애플리케이션 레벨의 재시도/서킷 브레이커 패턴이 필요합니다. systemd는 '한 호스트 안에서 프로세스를 안정적으로 띄우고 유지하는 것'까지가 역할이고, 그 이상의 분산 시스템 조정은 상위 계층의 책임입니다."

평가 포인트: systemd의 책임 범위와 상위 오케스트레이션 계층의 경계를 구분하는가

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"systemctl restart myapp으로 재시작해보고, 안 되면 다시 재시작해보겠습니다."
왜 약한 답변인가:
  • 재시작이 반복되는 근본 원인을 확인하지 않고 증상만 반복 처리합니다.
  • Active 필드나 로그를 통한 상태 진단 과정이 전혀 없습니다.
  • unit 파일의 설정(의존성, 재시작 정책)이 원인일 가능성을 전혀 고려하지 않습니다.
좋은 답변 (Senior 수준)
"status의 Active 필드로 재시작 반복 여부부터 확인하고, journalctl로 실패 직전 로그를 확인해 unit 설정 문제인지 애플리케이션 버그인지 구분한 뒤, 근본 원인에 맞춰 Requires=나 Restart= 설정을 조정하고 재발 방지책까지 마련합니다."
왜 좋은 답변인가:
  • 상태 필드를 근거로 상황의 긴급도를 먼저 판단합니다.
  • 로그를 통해 unit 설정 문제와 애플리케이션 버그를 구분합니다.
  • 순서 보장(After=)과 강제 의존(Requires=)의 차이를 알고 정확한 수정을 제안합니다.
  • 재시작 정책의 trade-off(폭주 방지 vs 복구 속도)를 함께 고려합니다.

🚫 흔한 오해와 정확한 이해

오해 1: "systemctl start만 하면 재부팅 후에도 계속 서비스가 실행된다"

start는 지금 당장의 실행 여부만 결정합니다. 재부팅 후에도 자동으로 시작되려면 별도로 enable을 실행해서 [Install] 섹션에 정의된 target에 연결해야 합니다.

오해 2: "After=만 설정하면 대상 서비스가 완전히 준비된 뒤에 시작된다"

After=는 systemd가 유닛을 시작하는 순서만 보장합니다. 대상 프로세스가 이미 떠 있어도 내부적으로 아직 요청을 받을 준비가 안 됐을 수 있으므로, 진짜 필요한 경우 애플리케이션 레벨의 재시도 로직으로 이 간극을 메워야 합니다.

오해 3: "Restart=always를 설정하면 서비스가 절대 죽지 않는다"

Restart=always는 프로세스가 죽었을 때 다시 실행을 시도할 뿐이며, 애플리케이션 자체의 버그가 원인이라면 같은 이유로 다시 죽는 crash loop이 반복될 뿐입니다. 속도 제한(StartLimitBurst) 없이는 오히려 CPU와 로그를 낭비하며 문제를 키울 수 있습니다.

오해 4: "unit 파일을 수정하면 바로 적용된다"

systemd는 unit 파일을 처음 읽을 때 메모리에 캐시해두므로, 파일을 직접 수정한 뒤에는 반드시 systemctl daemon-reload로 새 정의를 다시 읽혀야 합니다. 이를 생략하면 restart를 해도 이전 설정으로 동작합니다.

오해 5: "systemctl status에 보이는 로그 몇 줄이 로그의 전부다"

status는 최근 로그를 요약해서 보여줄 뿐입니다. 전체 맥락과 더 과거의 기록은 Day 5에서 다룬 journalctl -u <service>로 별도 확인해야 합니다.

Day 6 핵심 포인트 요약

  • start/stop과 enable/disable은 서로 독립적인 축이다.
  • After=는 순서만 보장하고, 강한 의존이 필요하면 Requires=와 애플리케이션 재시도가 함께 필요하다.
  • Restart=always는 속도 제한(StartLimitBurst) 없이는 crash loop 위험이 있다.
  • status의 Active 필드로 지금이 정상/재시작 반복/포기 상태인지 즉시 구분해야 한다.
  • unit 파일 수정 후에는 반드시 daemon-reload가 필요하다.
