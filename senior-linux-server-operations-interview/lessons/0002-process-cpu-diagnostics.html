📚 전제 개념 — 이것을 먼저 이해해야 한다

1. ps aux 필드 해석 — VSZ와 RSS는 다른 것을 말한다

쉽게 말하면: VSZ는 "이 프로세스가 예약해 둔 방(주소 공간)의 크기"이고, RSS는 "지금 실제로 그 방에서 사람이 앉아 있는 자리 수(실제 물리 메모리)"입니다. 방을 크게 예약해도 실제로 다 쓰지 않을 수 있습니다. 

$ ps aux | head -5
USER   PID %CPU %MEM    VSZ   RSS TTY  STAT START   TIME COMMAND
app   4821 87.3  2.1 2145920 345612 ?   R    09:12  12:44 java -jar app.jar
app   4822  0.3  0.5  812400  81200 ?   S    09:12   0:02 java -jar worker.jar
  • VSZ (Virtual Size): 프로세스에 할당된 가상 주소 공간 전체. 실제 사용량이 아니라 "예약된 최대치"에 가깝습니다. JVM처럼 힙을 미리 크게 예약하는 프로세스는 VSZ가 매우 커도 정상입니다.
  • RSS (Resident Set Size): 실제 물리 메모리(RAM)에 올라와 있는 크기. 이것이 실제 메모리 압박과 직결됩니다.
  • %CPU: 해당 프로세스가 최근 사용한 CPU 시간의 비율. 멀티코어 서버에서는 100%를 넘을 수 있습니다(코어 여러 개를 동시에 쓰면 200%, 300%도 가능).
  • STAT: 프로세스 상태 코드(Day 1의 R/D/S/Z) + 부가 플래그. <는 높은 우선순위, N은 낮은 우선순위(nice), s는 세션 리더, l은 멀티스레드, +는 포그라운드 프로세스 그룹.
  • TIME: 프로세스가 실제로 CPU를 사용한 누적 시간(경과 시간이 아님). START 이후 오래 지났는데 TIME이 매우 크다면 지속적으로 CPU를 소모하고 있다는 뜻입니다.
⚠️

중요: VSZ만 보고 "메모리 문제"라고 판단하면 안 됩니다. VSZ는 예약량이고, 실제 압박 여부는 RSS와 시스템 전체의 available(Day 1), 그리고 swap 활동으로 판단해야 합니다.

2. top의 상세 필드 — PR, NI, VIRT, RES, SHR

top 화면의 프로세스 목록 필드는 ps와 이름은 다르지만 개념은 같습니다. 면접에서 실수하기 쉬운 것은 PR과 NI의 관계입니다. 

  PID USER   PR  NI    VIRT    RES    SHR S  %CPU %MEM  TIME+ COMMAND
 4821 app    20   0  2096m  337m   45m R  87.3  2.1  12:44.02 java
 4822 app    30  10  793m   79m   12m S   0.3  0.5   0:02.10 java
  • NI (nice 값): 사용자가 지정하는 우선순위 힌트. -20(가장 높은 우선순위) ~ 19(가장 낮은 우선순위) 범위이며, 기본값은 0입니다.
  • PR (priority): 커널이 실제 스케줄링에 사용하는 값으로, 일반 프로세스는 PR = 20 + NI로 계산됩니다. PR 값이 낮을수록 먼저 스케줄됩니다.
  • SHR (Shared): 다른 프로세스와 공유하는 메모리(공유 라이브러리, mmap된 파일). RES에서 SHR을 뺀 값이 이 프로세스만의 순수 메모리에 가깝습니다.
✅

면접 팁: "nice 값을 낮추면(음수) 우선순위가 높아진다"는 말을 정확히 하려면 "낮은 nice 값 = 낮은 PR 값 = 스케줄러가 더 자주/먼저 선택"이라는 인과관계까지 설명해야 단순 암기가 아니라 이해로 보입니다.

멀티스레드 프로세스를 진단할 때는 top -H(스레드 단위로 표시) 또는 ps -eLf를 사용합니다. 하나의 PID 안에서 특정 스레드(LWP, Light Weight Process)만 CPU를 독점하는 경우가 실무에서 매우 흔합니다. 

# 스레드별 CPU 사용량 확인 (top -H는 갱신마다 스레드를 표시)
$ top -H -p 4821

# 특정 프로세스의 모든 스레드를 LWP 단위로 표시
$ ps -eLf | grep 4821

3. /proc/PID — 커널이 공개하는 프로세스의 속살

쉽게 말하면: /proc/PID는 실행 중인 프로세스의 "진단 창구"입니다. 파일처럼 보이지만 실제로는 커널이 그 순간의 프로세스 상태를 실시간으로 보여주는 가상 파일시스템입니다.

경로 확인할 수 있는 것 언제 쓰는가
/proc/PID/statusVmRSS, VmSwap, Threads 수, 상태메모리 누수 추적, 스레드 수 급증 확인
/proc/PID/statutime/stime(누적 CPU 시간), 상태 코드일정 간격으로 두 번 읽어 CPU 사용 delta 계산
/proc/PID/cmdline실행된 정확한 명령어와 인자PID만 보고 무슨 프로세스인지 모를 때
/proc/PID/fd/열려 있는 파일 디스크립터 목록파일/소켓 fd 고갈, "Too many open files" 원인 추적
/proc/PID/maps메모리 매핑 영역(코드, 힙, 스택, 공유 라이브러리)메모리 세그먼트별 크기 분석
/proc/PID/limits이 프로세스에 적용된 ulimit 값fd 개수 제한, 프로세스 수 제한 확인
 
# 프로세스가 지금 열고 있는 파일/소켓 개수 세기
$ ls /proc/4821/fd | wc -l

# 이 프로세스에 허용된 최대 fd 수와 비교
$ cat /proc/4821/limits | grep "open files"
Max open files            1024                 4096                 files
# 열린 fd 개수가 soft limit(1024)에 근접하면 곧 오류가 난다
✅

면접 팁: lsof -p PID도 같은 정보를 보여주지만 /proc/PID/fd를 직접 읽는 것이 더 가볍고, 프로덕션에서 lsof가 느리거나 설치되지 않은 상황에서 대안이 됩니다. "/proc을 직접 활용할 수 있다"는 답변은 도구 의존도가 낮다는 신호로 좋은 인상을 줍니다.

4. strace — 프로세스가 커널에 무엇을 요청하는지 엿보기

쉽게 말하면: 프로세스와 커널 사이의 모든 "대화"(시스템 콜)를 도청하는 도구입니다. 프로세스가 파일을 읽으려 하는지, 네트워크 응답을 기다리는지, 락을 기다리는지가 시스템 콜 이름으로 드러납니다. 

# 이미 실행 중인 프로세스에 붙어서 syscall 실시간 추적
$ strace -p 4821
futex(0x7f2a3c001d40, FUTEX_WAIT_PRIVATE, 2, NULL) = 0
futex(0x7f2a3c001d40, FUTEX_WAIT_PRIVATE, 2, NULL) = 0
futex(0x7f2a3c001d40, FUTEX_WAIT_PRIVATE, 2, NULL) = 0

# 짧은 시간 동안 요약 통계만 (프로덕션에서 더 안전한 방식)
$ strace -c -p 4821
% time     seconds  usecs/call     calls    syscall
------ ----------- ----------- --------- ----------------
 78.40    0.004521         452        10 read
 15.20    0.000876         438         2 futex
  6.40    0.000368         368         1 epoll_wait
  • syscall이 거의 없거나 드물게 나온다 → 유저 공간에서 순수 연산 중 → CPU 바운드 가능성.
  • read/recvfrom/epoll_wait가 반복된다 → 네트워크/파일 I/O 대기 → I/O 바운드.
  • futex가 반복된다 → 락 경합(다른 스레드가 락을 쥐고 있어서 대기) → 동시성 문제.
⚠️

프로덕션 위험: strace -p는 ptrace 시스템 콜을 사용해서 대상 프로세스의 모든 syscall을 가로챕니다. syscall이 매우 빈번한 프로세스에서는 수 배에서 수십 배까지 느려질 수 있습니다. 트래픽이 많은 프로덕션 서버에서는 strace -c로 짧게 요약만 뜨거나, 오버헤드가 훨씬 낮은 perf trace, bpftrace를 우선 고려해야 합니다.

5. CPU 바운드 vs I/O 바운드 — 같은 "CPU 100%"도 원인이 다르다

쉽게 말하면: CPU 바운드는 "머리를 계속 굴리느라 바쁜 것"이고, I/O 바운드는 "택배(디스크/네트워크 응답)가 오기를 기다리느라 다른 일을 못 하는 것"입니다. 그런데 기다리는 방식(busy-wait, polling)에 따라 I/O 바운드도 top에서는 CPU를 많이 쓰는 것처럼 보일 수 있습니다.

판단 신호 CPU 바운드 I/O 바운드
프로세스 상태 (ps STAT) 지속적으로 R D 또는 S를 오가며 반복
strace syscall 패턴 거의 없음 (연산만 수행) read/recv/futex/poll 반복
top의 us/sy 비율 us(user) 압도적으로 높음 wa(iowait) 또는 sy(system) 비중 큼
%CPU가 100%여도 응답이 느린 이유 연산 자체가 오래 걸림 (알고리즘/데이터량) busy-wait/polling 루프로 CPU를 태우며 응답을 기다림
⚠️

함정: "CPU 100% = CPU 바운드"라고 단정하면 안 됩니다. 일부 라이브러리나 락 구현은 잠들지 않고 반복 확인(busy-wait, spinlock)하는 방식으로 대기하는데, 이 경우 ps STAT는 R이고 %CPU도 높지만 실제로는 다른 무언가(락 해제, 응답)를 "기다리는" I/O적 성격의 문제입니다. strace로 syscall 유무를 확인하거나, 코드/스택을 직접 봐야 진짜 원인이 드러납니다.

🎯 왜 면접에서 묻는가

"CPU 100% 프로세스를 어떻게 진단하나요?"는 Day 1의 "어느 리소스가 병목인가"보다 한 단계 더 깊이 들어가서, "그 프로세스 내부에서 무슨 일이 벌어지고 있는지 코드를 못 보고도 추론할 수 있는가"를 평가합니다.

면접관이 실제로 평가하는 것

  1. "top만 보고 끝"이 아니라 한 단계 더 파고드는가?
    top에서 범인 프로세스를 찾는 것은 시작일 뿐입니다. Senior는 "그 다음엔 무엇을 봅니까?"라는 후속 질문에 막히지 않고 strace, /proc, top -H로 이어갑니다.
  2. 도구를 프로덕션 안전성과 함께 고려하는가?
    strace가 프로세스를 느리게 만들 수 있다는 걸 모르고 프로덕션에 바로 붙이면 실무 경험이 없다는 신호입니다. 오버헤드를 언급하며 대안(요약 모드, perf, bpftrace)을 아는지가 차별점입니다.
  3. CPU 사용률이라는 숫자 뒤의 "왜"를 추론하는가?
    %CPU가 높다는 사실 자체보다, syscall 패턴과 프로세스 상태를 조합해서 "연산 때문인지, 대기 때문인지"를 구분하는 능력을 봅니다.
  4. 멀티스레드 환경에서 문제를 좁힐 수 있는가?
    실무 서비스는 대부분 멀티스레드/멀티프로세스입니다. "프로세스 전체가 아니라 스레드 단위로 좁힐 수 있다"는 답변이 나오면 실제 운영 경험이 있다는 신호입니다.
💡

이 질문은 "당신이 낯선 서버에서, 소스 코드를 열어보지 못하는 상황에서도 무엇이 잘못됐는지 추론할 수 있는가"를 시험합니다. 블랙박스를 다루는 능력이 핵심 평가 포인트입니다.

🏭 실무 연결 — 이런 상황에서 실제로 쓴다

시나리오 A: Java 서비스의 스레드 하나만 CPU 100%

모니터링에서 특정 Java 프로세스의 CPU가 100%를 계속 유지합니다. 전체 애플리케이션이 멈춘 것은 아니고 응답 지연만 발생합니다. top -H -p PID로 확인해보니 특정 LWP(스레드) 하나만 CPU를 독점하고 있었습니다. 이 LWP의 PID를 16진수로 변환해서 jstack 출력에서 같은 nid(native thread id)를 가진 스레드의 스택 트레이스를 확인하면, 어떤 메서드에서 무한 루프나 비효율적인 정규식이 실행 중인지 알 수 있습니다.

시나리오 B: 응답 지연인데 strace가 계속 futex만 찍힌다

API가 느려졌다는 신고에 문제 프로세스를 strace -c -p PID로 짧게 관찰했더니 대부분의 시간이 futex에 몰려 있었습니다. 이것은 CPU 연산 문제가 아니라 락 경합(다른 스레드가 락을 오래 쥐고 있음) 문제였고, 실제 원인은 커넥션 풀 크기가 작아서 스레드들이 커넥션을 기다리며 락을 두고 경쟁하는 상황이었습니다.

시나리오 C: "Too many open files" 에러

특정 시간대에 서비스가 갑자기 연결을 거부하기 시작했습니다. /proc/PID/fd 개수를 세어보니 /proc/PID/limits의 soft limit에 근접해 있었고, 원인은 HTTP 커넥션을 정상적으로 닫지 않는 코드 버그로 소켓 fd가 계속 누적된 것이었습니다.

실무에서 배운 핵심 교훈

  • 멀티스레드 서비스의 CPU 문제는 프로세스 단위가 아니라 스레드 단위로 좁혀야 정확한 원인이 나옵니다.
  • strace는 강력하지만 프로덕션에서는 반드시 오버헤드를 의식하고 짧게, 요약 모드로 써야 합니다.
  • fd 고갈, 메모리 누수처럼 "서서히 악화되는" 문제는 한 시점의 스냅샷이 아니라 추세(시간에 따른 변화)를 봐야 발견됩니다.
  • 애플리케이션 레벨 도구(jstack, pprof)와 OS 레벨 도구(strace, /proc)를 함께 쓸 줄 알아야 완전한 그림이 나옵니다.

🧠 답변 사고 순서 — 프로세스 내부 진단 흐름

Day 1에서 "CPU가 병목"이라고 범주를 좁혔다는 전제 하에, 특정 프로세스의 내부를 진단하는 순서입니다.

  • 범인 프로세스 확정 + 기본 필드 확인

    ps aux --sort=-%cpu 또는 top에서 상위 프로세스를 확인하고, VSZ/RSS, STAT, TIME(누적 CPU 시간)을 함께 봅니다. TIME이 START 시각 대비 비정상적으로 크다면 지속적인 문제입니다.

  • 단일 스레드인가 멀티스레드인가 확인

    ps -eLf | grep PID 또는 cat /proc/PID/status | grep Threads로 스레드 수를 확인합니다. 스레드가 여러 개라면 top -H -p PID로 어느 스레드(LWP)가 CPU를 독점하는지 좁힙니다. 

    $ cat /proc/4821/status | grep -E "Threads|VmRSS|VmSwap"
Threads:        24
VmRSS:      345612 kB
VmSwap:          0 kB

  • 프로세스 상태의 시간적 패턴 관찰

    한 번의 스냅샷이 아니라 watch -n1 'ps -o pid,stat,%cpu,time -p PID'처럼 반복 관찰해서 상태가 계속 R인지, D/S를 오가는지 확인합니다.

  • strace로 syscall 패턴 짧게 샘플링

    프로덕션이라면 strace -c -p PID로 몇 초만 관찰하고 종료합니다(Ctrl+C). syscall 비중을 보고 "연산 중심(CPU 바운드)"인지 "대기 중심(I/O 바운드/락 경합)"인지 가설을 세웁니다. 

    # 2~3초만 관찰하고 Ctrl+C, 오버헤드 최소화
$ strace -c -p 4821
# futex 비중이 높다 → 락 경합 의심
# read/recvfrom 비중이 높다 → I/O 대기 의심
# syscall이 거의 없다 → 순수 연산, CPU 바운드 의심

  • 애플리케이션 레벨 도구로 교차 검증

    OS 레벨 관찰로 가설을 세웠다면, 언어/런타임 도구로 확정합니다. Java는 jstack으로 특정 스레드의 스택 트레이스를, Python은 py-spy dump를, Go는 pprof를 사용합니다. OS 레벨 신호(어떤 스레드, 어떤 syscall)와 애플리케이션 레벨 신호(어떤 코드 라인)를 연결하는 것이 마무리 단계입니다.

    ✅

    면접에서 "OS 도구 → 애플리케이션 도구로 좁혀간다"는 순서를 언급하면 단일 도구만 아는 사람과 뚜렷하게 구분됩니다.

💬 핵심 답변 예시

면접에서 자연스럽게 말할 수 있는 구조입니다. 외우지 말고 논리의 흐름을 이해하세요.

💬 면접관: "특정 프로세스가 CPU를 100% 쓰고 있습니다. 무엇을 하느라 그런지 어떻게 확인하시겠습니까?"

"먼저 ps aux --sort=-%cpu나 top으로 해당 프로세스의 기본 정보(STAT, TIME, 시작 시각)를 확인해서 일시적인 스파이크인지 지속적인 문제인지 파악합니다.

그 다음 이 프로세스가 멀티스레드인지 확인합니다. /proc/PID/status의 Threads 수를 보거나 top -H -p PID로 스레드별 CPU 사용량을 봅니다. 스레드가 여러 개라면 프로세스 전체가 아니라 특정 스레드 하나가 범인인 경우가 많습니다.

원인 후보를 좁힌 다음 strace -c -p PID로 몇 초만 짧게 관찰합니다. syscall이 거의 없이 CPU만 계속 쓰고 있다면 순수 연산(CPU 바운드) 가능성이 높고, futex가 반복된다면 락 경합, read/recvfrom이 반복된다면 I/O 대기를 의심합니다. strace는 ptrace 오버헤드가 있어서 프로덕션에서는 짧게, 요약 모드로만 사용하는 것이 중요합니다.

마지막으로 애플리케이션 레벨 도구로 교차 검증합니다. Java라면 해당 스레드의 native thread id(LWP)를 16진수로 변환해서 jstack 출력에서 같은 nid를 가진 스레드의 스택 트레이스를 확인하면 정확히 어떤 코드가 실행 중인지 알 수 있습니다. 이렇게 OS 레벨 신호와 애플리케이션 레벨 신호를 연결해야 코드 수정까지 이어지는 근본 원인 분석이 가능합니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "strace를 프로덕션 서버에 바로 붙여도 되나요?"

"신중해야 합니다. strace -p는 ptrace 시스템 콜로 대상 프로세스의 모든 syscall을 가로채기 때문에, syscall이 빈번한 프로세스에서는 체감될 정도로 느려질 수 있습니다. 트래픽이 많은 서비스라면 이 지연이 추가 장애로 이어질 수 있습니다. 그래서 저는 strace -c로 짧게(몇 초) 요약만 뜨는 방식을 우선 사용하고, 오버헤드가 더 낮은 perf trace나 bpftrace를 대안으로 고려합니다."

평가 포인트: 도구의 부작용을 알고 프로덕션 안전성을 함께 고려하는가

Q. "top에서 VIRT는 2GB인데 RES는 300MB인 프로세스, 문제인가요?"

"그 자체로는 문제가 아닙니다. VIRT(가상 메모리)는 프로세스가 예약해 둔 주소 공간 전체이고, 실제로 물리 메모리에 올라온 양은 RES입니다. JVM처럼 힙을 크게 예약해두고 실제로는 필요한 만큼만 쓰는 런타임에서는 이 차이가 매우 흔합니다. 제가 실제로 주목하는 지표는 RES의 증가 추세입니다. 시간이 지날수록 RES가 계속 증가한다면 메모리 누수를 의심합니다."

평가 포인트: VIRT/RES를 혼동하지 않고, 절대값보다 추세를 보는 사고

Q. "멀티스레드 프로세스에서 스레드 하나만 CPU 100%입니다. 어떻게 좁히시겠습니까?"

"top -H -p PID로 스레드(LWP) 단위 CPU 사용량을 보고 범인 스레드의 ID를 확인합니다. 이후 언어/런타임에 맞는 도구로 그 스레드의 스택을 확인합니다. 예를 들어 Java라면 해당 LWP ID를 16진수로 변환한 뒤 jstack 출력에서 nid=0x... 값이 일치하는 스레드를 찾아 어떤 메서드가 실행 중인지 확인합니다. 프로세스 전체를 보고 추측하는 것보다 훨씬 정확하게 원인 코드에 도달할 수 있습니다."

평가 포인트: 프로세스 단위가 아니라 스레드 단위로 좁히는 실전 경험

Q. "CPU 사용률이 100%인데 strace를 보니 syscall이 거의 없습니다. 어떻게 해석하시나요?"

"syscall이 거의 없다는 것은 커널에 요청할 일 없이 유저 공간에서 순수 연산만 반복하고 있다는 뜻이므로 CPU 바운드로 판단합니다. 다만 주의할 점은, 락 구현에 따라 잠들지 않고 반복 확인하는 busy-wait/spinlock 방식일 수도 있다는 것입니다. 이 경우도 syscall은 거의 없지만 실제로는 다른 자원(락 해제)을 기다리는 것이므로, 애플리케이션 스택 트레이스를 확인해서 정말 연산 중인지 스핀 대기 중인지 구분해야 합니다."

평가 포인트: syscall 부재를 CPU 바운드로 단정하지 않고 busy-wait 가능성까지 고려

Q. "renice로 nice 값을 낮추면 무조건 빨라지나요?"

"무조건은 아닙니다. nice 값은 CPU가 병목일 때 스케줄러가 우선순위를 정하는 힌트일 뿐입니다. 만약 병목이 CPU가 아니라 디스크 I/O나 락 경합이라면 nice를 낮춰도 효과가 없습니다. 또한 CPU가 여유 있는 상황(코어 대비 프로세스 수가 적을 때)에서는 이미 스케줄러가 충분히 CPU 시간을 주고 있으므로 renice의 효과가 거의 없습니다. renice는 'CPU 경쟁이 있고, 이 프로세스를 상대적으로 우대하고 싶을 때'에만 의미가 있습니다."

평가 포인트: nice/renice를 만능 해결책이 아니라 조건부 도구로 이해

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"top으로 확인해서 CPU 높은 프로세스를 찾고, 그게 원인이니 재시작하거나 nice 값을 낮추겠습니다."
왜 약한 답변인가:
  • "CPU가 높다"는 사실만 확인하고 왜 높은지 파고들지 않습니다.
  • 멀티스레드 여부를 고려하지 않아서, 실제로는 스레드 하나의 문제인데 프로세스 전체를 재시작합니다.
  • strace, /proc 같은 심화 진단 도구를 언급하지 않습니다.
  • nice 값 조정이 실제 병목 종류(CPU vs I/O)와 무관하게 만병통치약처럼 제시됩니다.
  • 재시작은 원인을 지우고 재발 방지로 이어지지 않습니다.
좋은 답변 (Senior 수준)
"ps/top으로 프로세스를 특정한 뒤, 멀티스레드인지 확인하고 top -H로 스레드 단위로 좁힙니다. strace -c로 짧게 syscall 패턴을 보고 CPU 바운드인지 락 경합인지 I/O 대기인지 가설을 세운 뒤, jstack 같은 애플리케이션 도구로 정확한 코드 위치까지 확인합니다. strace는 오버헤드가 있으니 프로덕션에서는 짧고 안전하게 사용합니다."
왜 좋은 답변인가:
  • 프로세스 단위에서 스레드 단위로, OS 레벨에서 애플리케이션 레벨로 단계적으로 좁혀갑니다.
  • strace의 부작용을 알고 안전하게 쓰는 방법까지 설명합니다.
  • syscall 패턴으로 CPU 바운드/I/O 바운드/락 경합을 구분하는 구체적 근거를 제시합니다.
  • 재시작이 아니라 코드 레벨 원인까지 도달하는 경로를 보여줍니다.

🚫 흔한 오해와 정확한 이해

오해 1: "VSZ(VIRT)가 크면 메모리를 많이 쓰는 프로세스다"

VSZ/VIRT는 예약된 가상 주소 공간일 뿐, 실제 사용량이 아닙니다. JVM, Go 런타임처럼 메모리를 미리 크게 예약하는 프로그램은 VSZ가 커도 정상입니다. 실제 물리 메모리 사용량은 RSS로, 그리고 시간에 따른 RSS의 증가 추세로 판단해야 합니다.

오해 2: "strace는 안전한 진단 도구이니 언제든 붙여도 된다"

strace는 ptrace 기반이라 대상 프로세스를 syscall마다 잠깐씩 멈추게 합니다. syscall이 빈번한 프로세스에서는 눈에 띄게 느려질 수 있어, 트래픽이 많은 프로덕션 환경에서는 추가 장애를 유발할 위험이 있습니다. 요약 모드(-c)로 짧게 쓰거나 perf/bpftrace 같은 저오버헤드 대안을 우선 고려해야 합니다.

오해 3: "CPU 사용률 100%는 항상 CPU 바운드(연산 문제)다"

busy-wait나 spinlock 기반 대기는 syscall 없이 CPU를 계속 소모하면서도 실제로는 다른 자원(락 해제, 조건 충족)을 기다리는 것입니다. ps STAT은 R이고 %CPU도 높지만 본질은 "대기"입니다. strace의 syscall 유무만으로 단정하지 말고, 애플리케이션 스택 트레이스로 실제 코드를 확인해야 정확합니다.

오해 4: "top의 %CPU가 100%를 넘으면 잘못된 값이다"

top의 %CPU는 코어 하나 기준 100%가 아니라, 사용 중인 코어 수만큼 합산될 수 있습니다. 8코어 서버에서 프로세스가 4개 코어를 동시에 활용하면 %CPU가 400%까지 표시될 수 있고, 이는 오류가 아니라 정상적인 표기입니다. 코어 수 대비 몇 %인지로 다시 환산해서 판단해야 합니다.

오해 5: "renice로 nice 값을 낮추면 어떤 성능 문제든 개선된다"

nice/renice는 CPU 스케줄링 우선순위에만 영향을 줍니다. 병목이 디스크 I/O, 네트워크, 락 경합이라면 nice 값을 낮춰도 개선되지 않습니다. 또한 CPU 자원에 여유가 있다면 이미 스케줄러가 충분한 시간을 배정하고 있어 renice의 효과 자체가 미미합니다.

Day 2 핵심 포인트 요약

  • VSZ(VIRT)는 예약된 가상 메모리, RSS(RES)가 실제 물리 메모리 사용량이다.
  • 멀티스레드 프로세스는 top -H로 스레드 단위까지 좁혀야 정확하다.
  • strace -c로 syscall 패턴을 보면 CPU 바운드/I/O 바운드/락 경합을 구분할 수 있다.
  • strace는 ptrace 오버헤드가 있으므로 프로덕션에서는 짧고 신중하게 사용한다.
  • CPU 100% ≠ 항상 CPU 바운드. busy-wait일 수 있으니 애플리케이션 스택으로 교차 검증한다.
  • OS 레벨 진단(strace, /proc)과 애플리케이션 레벨 진단(jstack 등)을 연결해야 코드 원인까지 도달한다.
