💡

Day 1~4가 "지금 이 순간의 상태"(top, free, iostat)를 읽는 능력이었다면, Day 5는 "과거에 무슨 일이 있었는가"를 시간순으로 재구성하는 능력입니다. 로그 질문에서 면접관이 보는 것은 명령어 옵션 암기가 아니라, 로그가 여러 시스템(journald, rsyslog, 애플리케이션 자체 로그)에 흩어져 있다는 것을 알고 그것들을 하나의 타임라인으로 엮어낼 수 있는지입니다.

📚 전제 개념 — 이것을 먼저 이해해야 한다

1. journalctl 기본 — "누가, 언제, 얼마나"를 좁혀가는 필터

쉽게 말하면: journalctl은 텍스트 파일을 grep하는 것이 아니라, systemd-journald가 구조화된 형태(바이너리, 인덱싱됨)로 보관해둔 로그를 조회하는 것입니다. 도서관 사서에게 "어제 오후에 A라는 사람이 대출한 책 목록"을 물어보는 것과 비슷합니다 — 책을 한 권씩 뒤지는 게 아니라 이미 색인된 조건으로 바로 찾아줍니다. 그래서 grep보다 시간 범위, 서비스 단위 필터링이 훨씬 빠릅니다. 

# 특정 서비스의 로그만, 특정 시간 범위로 좁혀서 확인
$ journalctl -u myapp.service --since "2026-07-04 03:00" --until "2026-07-04 03:30"
# 우선순위(priority) 기준으로 에러 이상만 필터
$ journalctl -u myapp.service -p err
# 실시간 tail (systemd 로그의 tail -f)
$ journalctl -u myapp.service -f
# 가장 최근 부팅 이후 로그만 (재부팅 전후 구분에 필수)
$ journalctl -b -u myapp.service
# 스크립트에서 파싱하기 위한 구조화 출력
$ journalctl -u myapp.service -o json --since "-10min"
  • -u: unit(서비스) 단위로 좁힘 — 여러 서비스가 뒤섞인 전체 로그를 볼 필요가 없어집니다.
  • -p: emerg부터 debug까지 syslog 우선순위 레벨로 필터. -p err은 err 이상(crit, alert, emerg 포함)만 보여줍니다.
  • --since/--until: 절대 시각("2026-07-04 03:00")뿐 아니라 상대 시각("-1h", "-10min")도 지원 — 장애 신고 시각 기준으로 빠르게 좁힐 때 유용합니다.
  • -b: 특정 부팅(boot) 세션의 로그만. 서비스가 재부팅 때문에 죽었는지 아닌지 구분할 때 필수입니다.
✅

면접 팁: 필터를 조합해서 쓴다는 것을 보여주세요. 예를 들어 journalctl -u myapp -p err --since "-30min"처럼 서비스+우선순위+시간을 동시에 좁히는 습관은 "명령어를 안다"가 아니라 "빠르게 원인을 좁히는 사고방식이 있다"는 신호를 줍니다.

2. journald vs rsyslog vs /var/log — 로그는 한 곳에만 있지 않다

쉽게 말하면: 회사에 비유하면, systemd-journald는 "모든 부서의 업무 기록을 실시간으로 받아 적는 속기사"이고, rsyslog는 "그 기록을 부서별 서류철(/var/log/syslog, /var/log/auth.log 등)로 정리해서 캐비닛에 보관하는 문서 담당자"입니다. 최신 배포판은 둘 다 존재하는 경우가 많아서, 같은 이벤트가 journalctl로도, 전통적인 텍스트 로그 파일로도 보일 수 있습니다.

구분 systemd-journald rsyslog / /var/log
저장 형식 바이너리, 구조화된 메타데이터(PID, unit, boot ID 등) 포함 평문 텍스트, 라인 단위
보존 정책 기본은 /run/log/journal에 휘발성(재부팅 시 소실) — /etc/systemd/journald.conf에서 Storage=persistent로 영구 보존 전환 가능 logrotate가 파일 크기/기간 기준으로 회전 및 보관
조회 도구 journalctl (필터링에 최적) tail, grep, awk 등 텍스트 도구
⚠️

실무 연결: /etc/systemd/journald.conf에서 Storage=volatile(기본값에 가까운 설정)이면 서버가 재부팅되는 순간 이전 장애의 journald 로그가 사라집니다. 그래서 장애 조사 중 서버를 재부팅해야 하는 상황이라면, 재부팅 전에 반드시 journalctl 결과를 파일로 백업 (journalctl -u myapp --since "-1h" > incident.log)해야 한다는 판단이 senior 레벨에서 중요합니다.

3. /var/log 구조 — 어디를 봐야 하는지 아는 것 자체가 실력이다

쉽게 말하면: 병원에 비유하면, 환자의 전체 상태(/var/log/syslog, /var/log/messages)를 보는 종합 차트가 있고, 특정 검사 결과만 모아둔 전문 차트(/var/log/auth.log — 로그인/권한, /var/log/kern.log, dmesg — 커널/하드웨어, /var/log/nginx/ — 특정 애플리케이션)가 따로 있습니다. 증상에 맞는 차트부터 봐야 시간을 낭비하지 않습니다. 

# 전체 시스템 로그(배포판에 따라 syslog 또는 messages)
$ tail -f /var/log/syslog
# 로그인 시도, sudo 사용, SSH 인증 실패 등 인증 관련
$ sudo tail -100 /var/log/auth.log
# 커널 메시지 — 디스크 I/O 에러, OOM Killer 동작, 하드웨어 이벤트
$ dmesg -T | tail -50
# 부팅 시점 커널 메시지 (dmesg는 버퍼가 순환되어 유실될 수 있음)
$ journalctl -k -b
  • "로그인 실패가 반복된다"는 신고라면 syslog가 아니라 곧장 auth.log부터 봅니다.
  • "디스크 에러/하드웨어 이슈가 의심된다"면 dmesg나 journalctl -k(커널 로그)를 먼저 봅니다.
  • 애플리케이션(nginx, PostgreSQL 등)은 보통 자체 로그 디렉터리(/var/log/nginx/error.log 등)를 따로 가지므로, systemd unit으로 실행 중이어도 journalctl만으로는 애플리케이션 레벨의 상세 에러(요청 파라미터, 쿼리 등)가 보이지 않을 수 있습니다.
✅

면접 팁: "증상의 종류에 따라 어느 로그부터 볼지 즉시 판단한다"는 것을 보여주면, 단순히 journalctl만 아는 사람과 차별화됩니다. 인증 문제 → auth.log, 하드웨어/커널 문제 → dmesg, 애플리케이션 로직 에러 → 애플리케이션 자체 로그, 서비스 생명주기(시작/중단/재시작) → journalctl 순으로 답하면 구조적으로 들립니다.

4. grep/awk/sed 실전 조합 — 대용량 로그에서 신호를 뽑아낸다

쉽게 말하면: grep은 "이 줄에 이 단어가 있는가"만 판단하는 도구이고, awk는 "각 줄을 필드로 쪼개서 계산·집계"하는 도구, sed는 "패턴에 맞춰 줄을 변형"하는 도구입니다. 각각 잘하는 일이 다르기 때문에, 실무에서는 파이프로 이어서 조합해 씁니다. 

# 특정 시간대 범위 내에서 에러 라인만 추출 (시간 문자열 범위로 grep)
$ grep -E "^Jul 04 03:(0[0-9]|1[0-9]|2[0-9])" /var/log/syslog | grep -i error
# 에러 메시지를 종류별로 집계해서 가장 빈번한 패턴부터 정렬
$ grep -i error app.log | awk -F': ' '{print $2}' | sort | uniq -c | sort -rn | head -10
   842 connection timeout to db-primary
   213 upstream request failed
    31 disk quota exceeded
# 응답 시간(response_time) 필드가 특정 임계치를 넘는 요청만 추출
$ awk -F'"' '{split($0,a," "); if (a[NF] > 3000) print}' access.log
# 민감정보(카드번호 패턴 등)를 로그에서 마스킹해서 다른 사람에게 공유
$ sed -E 's/[0-9]{4}-[0-9]{4}-[0-9]{4}-[0-9]{4}/****-****-****-****/g' app.log > app.masked.log
  • grep -E + 시간 패턴: journalctl의 --since가 없는 순수 텍스트 로그에서 시간 범위를 흉내낼 때 사용합니다.
  • awk ... | sort | uniq -c | sort -rn: "어떤 에러가 가장 자주 발생하는가"를 눈대중이 아니라 숫자로 보여주는 표준 패턴입니다. 면접에서 이 조합을 자연스럽게 말하면 인상적입니다.
  • sed로 민감정보를 마스킹한 뒤 로그를 공유하는 습관은 보안 감각으로도 평가됩니다.
⚠️

주의: 수십 GB 로그 파일에 grep만 반복 실행하면 디스크 I/O 부하가 추가로 발생해 이미 느려진 서버를 더 압박할 수 있습니다. 가능하면 --since/-p처럼 인덱싱된 필터가 있는 journalctl을 우선 쓰고, 순수 텍스트 로그라면 zgrep(압축 로그용), 필요한 만큼만 범위를 좁힌 뒤 grep을 실행하는 것이 안전합니다.

5. 로그 로테이션(logrotate) — 로그가 디스크를 채우는 것과 유실되는 것 사이의 균형

쉽게 말하면: 로그는 계속 쌓이기만 하면 디스크를 채워버리고(Day 4에서 본 디스크 고갈 문제), 그렇다고 마음대로 지우면 장애 조사에 필요한 과거 기록이 사라집니다. logrotate는 "일정 크기/기간마다 파일을 교체하고, 오래된 것은 압축해서 보관하다가, 더 오래되면 삭제하는" 규칙을 관리해서 이 균형을 맞추는 도구입니다. 

# /etc/logrotate.d/myapp 예시
/var/log/myapp/*.log {
    daily
    rotate 14
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
  • rotate 14: 회전된 로그를 14세대까지 보관 — 최소 2주 전 기록까지는 되돌아볼 수 있습니다.
  • compress / delaycompress: 즉시 압축하면 바로 직전 로그를 읽는 도구(일부 프로세스)가 압축 파일을 다루지 못할 수 있어, 한 세대는 압축을 늦춥니다.
  • copytruncate: Day 4에서 다룬 "삭제됐지만 열려 있는 파일" 문제를 근본적으로 회피하는 옵션 — 파일을 옮기는 대신 내용만 비워서 애플리케이션이 이미 열어둔 파일 디스크립터를 그대로 재사용하게 합니다. 대신 truncate와 마지막 쓰기 사이의 극히 짧은 순간에 로그 몇 줄이 유실될 수 있다는 trade-off가 있습니다.
✅

면접 팁: "copytruncate는 애플리케이션이 SIGHUP 등으로 파일을 재오픈하도록 신호를 보낼 수 없는 경우의 대안이며, 그 대가로 아주 짧은 로그 유실 가능성을 감수한다"는 trade-off까지 설명하면, 옵션 이름을 아는 것을 넘어 언제 어떤 옵션을 선택해야 하는지 판단할 수 있다는 신호를 줍니다.

🎯 왜 면접에서 묻는가

로그 분석 질문은 단일 명령어 지식이 아니라, 흩어진 여러 로그 소스를 시간 순서로 엮어 하나의 "사건 재구성" 스토리를 만들 수 있는가를 평가합니다. 이것이 실제 포스트모템 작성 능력과 직결됩니다.

면접관이 실제로 평가하는 것

  1. 어디를 봐야 하는지 아는가?
    journald, rsyslog/var/log, 애플리케이션 자체 로그 중 증상에 맞는 소스를 즉시 선택할 수 있는지를 봅니다. 무조건 journalctl -xe만 반복하는지, 아니면 증상별로 경로를 다르게 타는지가 갈립니다.
  2. 필터링으로 좁혀나가는가, 전체를 눈으로 훑는가?
    수십 GB 로그를 cat이나 전체 스크롤로 보는 것이 아니라, 시간·서비스· 우선순위로 좁혀서 효율적으로 찾아내는 습관이 있는지를 봅니다.
  3. 로그의 휘발성과 보존 정책을 이해하는가?
    journald의 기본 보존 정책, 재부팅 시 로그 소실 가능성, logrotate의 보관 기간을 인지하고 사고 발생 시 "증거 보전"을 먼저 챙기는지를 봅니다.
  4. 정량적으로 원인을 좁힐 수 있는가?
    "에러가 많이 발생한 것 같다"가 아니라 awk | sort | uniq -c로 실제 빈도를 집계해서 근거를 제시할 수 있는지를 봅니다.
💡

이 질문의 본질은 "장애 발생 시점 전후의 사실관계를 재구성할 수 있는가"입니다. 재구성이 정확해야 원인 분석도, 이후의 포스트모템 보고도 신뢰를 얻습니다.

🏭 실무 연결 — 이런 상황에서 실제로 쓴다

시나리오 A: 새벽에 서비스가 재시작된 원인을 추적한 경우

모니터링 알림으로 새벽 3시 12분에 서비스가 재시작된 것을 발견했습니다. journalctl -u myapp -b로 가장 최근 부팅 세션의 로그를 먼저 확인해 재부팅 때문인지 서비스 자체의 크래시인지 구분했고, 재부팅이 아니었기에 journalctl -u myapp --since "03:05" --until "03:15"로 좁혀 OOM Killer가 해당 프로세스를 종료한 커널 메시지(journalctl -k)를 발견했습니다. 이후 dmesg -T로 같은 시각의 메모리 상태를 교차 확인해 메모리 누수로 결론지었습니다.

시나리오 B: 재부팅 후 장애 조사 로그가 사라진 경우

장애 대응 도중 시스템이 응답하지 않아 부득이하게 재부팅했는데, journald.conf가 Storage=volatile로 설정되어 있어 재부팅 전 장애 순간의 journald 로그가 전부 소실됐습니다. 이후 재발 방지책으로 Storage=persistent로 전환하고, 장애 대응 체크리스트에 "재부팅 전 반드시 journalctl ... > incident.log로 백업" 항목을 추가했습니다.

시나리오 C: 대용량 로그에서 간헐적 에러의 패턴을 찾은 경우

"가끔 요청이 실패한다"는 모호한 신고를 받고, grep -i error access.log | awk -F': ' '{print $2}' | sort | uniq -c | sort -rn으로 에러 메시지를 종류별로 집계했더니 "connection timeout to db-primary"가 압도적으로 많다는 것을 확인했습니다. 감으로 짐작하는 대신 숫자로 근거를 확보해 DB 커넥션 풀 설정 문제로 범위를 좁혔습니다.

실무에서 배운 핵심 교훈

  • 장애 조사 중 재부팅이 필요하다면, journald 로그부터 파일로 백업한 뒤 재부팅한다.
  • 증상의 종류(인증/커널/애플리케이션/서비스 생명주기)에 따라 첫 번째로 열어볼 로그가 다르다는 것을 습관화해야 시간을 낭비하지 않는다.
  • "에러가 많다"는 느낌이 아니라 awk/uniq -c로 집계한 숫자가 원인 분석의 근거가 되어야 한다.
  • 운영 환경의 journald.conf, logrotate.d 설정은 장애가 나기 전에 미리 확인해둬야 할 항목이다.

🧠 답변 사고 순서 — 로그 분석 진단 흐름

  • 증상에 맞는 로그 소스부터 선택

    서비스 생명주기(시작/중단/재시작) 문제인지, 인증 문제인지, 커널/하드웨어 문제인지, 애플리케이션 로직 문제인지에 따라 journalctl, auth.log, dmesg, 애플리케이션 자체 로그 중 첫 번째로 열어볼 대상을 정합니다.

  • 시간 범위로 즉시 좁힘

    신고나 알림에 찍힌 시각을 기준으로 --since/--until(또는 grep의 시간 패턴)로 좁혀서, 전체 로그를 스크롤하며 눈으로 찾는 낭비를 없앱니다.

  • 부팅 경계와 로그 유실 가능성 확인

    journalctl -b로 재부팅 전후를 구분하고, journald가 휘발성 보존 정책이라면 조사에 필요한 구간을 먼저 파일로 백업합니다.

  • 정량적으로 패턴을 집계

    에러 메시지가 여러 종류라면 awk | sort | uniq -c | sort -rn으로 빈도순 정렬을 해서 가장 지배적인 원인부터 검증합니다. 

    $ journalctl -u myapp -b --since "-1h" -p err
$ journalctl -k --since "-1h"
$ grep -i error app.log | awk -F': ' '{print $2}' | sort | uniq -c | sort -rn | head

  • 원인 확정과 재발 방지

    타임라인이 재구성되면 근본 원인을 확정하고, 로그 보존 정책(journald persistent 전환, logrotate 보관 기간)과 알림 규칙을 재발 방지 관점에서 함께 점검합니다.

💬 핵심 답변 예시

면접에서 자연스럽게 말할 수 있는 구조입니다. 외우지 말고 논리의 흐름을 이해하세요.

💬 면접관: "새벽에 서비스가 재시작된 흔적이 있습니다. 어떻게 원인을 찾으시겠습니까?"

"먼저 journalctl -u myapp -b로 가장 최근 부팅 세션의 로그를 확인해서, 서버 자체가 재부팅된 것인지 서비스만 재시작된 것인지부터 구분합니다. 이 구분이 이후 조사 방향을 완전히 다르게 만들기 때문입니다.

서버 재부팅이 아니라면, 알림에 찍힌 시각을 기준으로 journalctl -u myapp --since "03:05" --until "03:15"처럼 시간 범위를 좁혀서 서비스가 종료된 직전 로그와 systemd의 재시작 사유를 확인합니다.

서비스 자체 로그에 단서가 부족하면 journalctl -k로 커널 로그도 같은 시간대로 확인합니다. OOM Killer가 프로세스를 강제 종료했다면 커널 로그에만 기록이 남기 때문입니다.

조사 도중 서버를 재부팅해야 하는 상황이 생기면, journald가 휘발성 저장이라면 재부팅 전에 반드시 관련 구간을 파일로 백업합니다. 원인이 확정되면 재발 방지책(메모리 누수라면 자원 제한 조정과 알림 임계치, 로그 보존 정책 강화)까지 포함해서 마무리합니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "journalctl로는 아무 단서가 없는데 서비스가 분명히 죽어있었습니다. 어디를 더 봐야 하나요?"

"먼저 journalctl -k로 커널 로그를 같은 시간대로 확인합니다. OOM Killer에 의한 강제 종료나 디스크 I/O 에러처럼 애플리케이션이 스스로 로그를 남기지 못하고 강제로 종료된 경우, 흔적은 애플리케이션 unit 로그가 아니라 커널 로그에만 남습니다. 그래도 없다면 dmesg -T로 부팅 이후 순환 버퍼에 남은 하드웨어 이벤트, 그리고 애플리케이션이 별도 로그 디렉터리를 쓴다면 그쪽도 함께 확인합니다."

평가 포인트: journald 하나로 끝내지 않고 커널/애플리케이션 레벨까지 시야를 넓히는가

Q. "journald.conf의 Storage=volatile과 persistent 차이가 실무에서 왜 중요한가요?"

"volatile은 로그를 /run(tmpfs, 메모리 기반)에만 보관해서 재부팅하면 모두 사라지고, persistent는 /var/log/journal에 디스크 기반으로 저장해 재부팅 이후에도 남습니다. 장애 조사 중에 서버를 재기동해야 하는 경우가 실무에서 드물지 않은데, volatile 상태에서 재부팅하면 정작 필요한 장애 순간의 로그가 통째로 사라질 수 있습니다. 그래서 프로덕션 서버는 기본적으로 persistent로 전환해두는 것을 권장합니다."

평가 포인트: 설정값의 차이를 실제 장애 대응 시나리오와 연결해서 설명하는가

Q. "여러 대의 서버에 로그가 분산되어 있다면 journalctl/grep 조합으로 충분한가요?"

"단일 서버 조사에는 충분하지만, 여러 서버에 걸친 요청을 추적하려면 한계가 있습니다. 각 서버에 SSH로 접속해서 개별 조회하는 것은 서버 수가 늘어날수록 비효율적이고, 요청이 여러 서비스를 거치는 경우 요청 단위로 상관관계를 지을 방법이 없습니다. 이런 경우에는 중앙 집중형 로그 수집(ELK, Loki 등)과 분산 트레이싱 ID를 도입해서 한 요청의 전체 흐름을 하나의 쿼리로 재구성할 수 있어야 합니다. 다만 그 이전 단계로, 단일 서버에서 journalctl/grep으로 빠르게 1차 원인을 좁히는 능력은 여전히 필요합니다."

평가 포인트: 단일 서버 도구의 한계를 알고 더 큰 규모의 관측성 체계로 확장할 수 있는가

Q. "logrotate가 설정되어 있는데도 로그 파티션이 계속 꽉 찹니다. 무엇을 의심하시겠습니까?"

"몇 가지를 순서대로 확인합니다. 첫째, logrotate가 실제로 크론/타이머로 정상 실행되고 있는지(systemctl status logrotate.timer, 또는 dry-run으로 logrotate -d). 둘째, copytruncate가 아닌 방식인데 애플리케이션이 SIGHUP 등으로 새 로그 파일을 재오픈하지 않아서 이전 파일이 삭제되지 않고 계속 쓰이고 있는지(Day 4에서 다룬 삭제된 열린 파일 문제와 동일한 원인). 셋째, 로그 레벨이 과도하게 verbose해서 회전 주기 안에 처리할 수 없을 만큼 로그량 자체가 급증했는지를 봅니다."

평가 포인트: logrotate 설정 존재 여부만 확인하지 않고 실제 동작 여부까지 검증하는가

Q. "간헐적으로만 발생하는 문제인데 로그에 재현 시점의 기록이 부족합니다. 어떻게 대응하시겠습니까?"

"우선 현재 로그 레벨이 너무 낮게(예: warning 이상만) 설정되어 있어서 문제 재현에 필요한 세부 정보가 애초에 기록되지 않았을 가능성을 확인합니다. 그렇다면 임시로 해당 컴포넌트의 로그 레벨을 debug로 높이고 문제가 재현될 때까지 모니터링합니다. 다만 로그 레벨을 과도하게 높이면 로그량 폭증으로 디스크나 성능에 영향을 줄 수 있으므로, 영향 범위를 좁힌 특정 모듈/서비스에만 한시적으로 적용하고, 재현 후에는 원복합니다."

평가 포인트: 로그 자체가 부족한 상황을 인지하고 관측성을 능동적으로 개선하는 판단을 하는가

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"journalctl -xe로 확인해서 에러 메시지를 찾아보겠습니다."
왜 약한 답변인가:
  • 시간 범위나 서비스 단위로 좁히지 않고 최신 로그를 전체 스크롤하는 방식에 머물러 있습니다.
  • journald 이외의 로그 소스(커널, 애플리케이션 자체 로그)는 전혀 고려하지 않습니다.
  • 재부팅 시 로그가 소실될 수 있다는 것을 인지하지 못해, 조사 중 무심코 재부팅하면 증거를 잃을 수 있습니다.
좋은 답변 (Senior 수준)
"먼저 부팅 경계를 -b로 확인하고, 알림 시각 기준으로 --since/ --until로 좁힌 뒤, 서비스 로그에 단서가 부족하면 커널 로그(-k)까지 확인합니다. 재부팅이 필요한 상황이면 그 전에 관련 로그를 먼저 백업합니다."
왜 좋은 답변인가:
  • 부팅 경계, 시간 범위, 우선순위 필터를 조합해서 효율적으로 좁혀 들어갑니다.
  • journald뿐 아니라 커널/애플리케이션 로그까지 층위를 넓혀서 봅니다.
  • 로그의 휘발성을 인지하고 증거 보전을 먼저 챙기는 판단력을 보여줍니다.
  • 정량적 집계(awk/uniq -c)로 근거 있는 결론을 낼 수 있습니다.

🚫 흔한 오해와 정확한 이해

오해 1: "journalctl만 확인하면 서버에서 일어난 모든 일을 알 수 있다"

journald는 systemd가 관리하는 유닛과 커널 메시지를 주로 다룹니다. nginx, 데이터베이스처럼 자체 로그 파일 체계를 가진 애플리케이션의 상세 로그(요청 파라미터, 쿼리 등)는 journald가 아니라 해당 애플리케이션의 로그 디렉터리에 별도로 기록됩니다.

오해 2: "journald 로그는 rsyslog나 /var/log와 항상 완전히 동일한 내용이다"

journald는 구조화된 메타데이터(PID, 유닛, 부팅 ID 등)까지 포함해서 더 풍부하게 저장하는 반면, rsyslog는 설정된 규칙에 따라 특정 우선순위/시설(facility)만 특정 파일로 걸러서 씁니다. 두 시스템의 필터링 규칙이 다르면 내용도 달라질 수 있습니다.

오해 3: "로그는 최대한 오래, 많이 남겨두는 것이 항상 좋다"

보존 기간과 로그 레벨은 디스크 용량(Day 4에서 다룬 디스크 고갈)과 직결됩니다. 무제한 보존은 오히려 디스크 부족으로 인한 새로운 장애를 유발할 수 있으므로, 보존 기간과 레벨은 "장애 조사에 필요한 최소 기간"과 "디스크 여유"의 trade-off로 결정해야 합니다.

오해 4: "grep으로 못 찾으면 로그에 문제가 기록되지 않은 것이다"

로그 레벨 설정 때문에 애초에 기록되지 않았거나, 문제가 커널 레벨에서 발생해 애플리케이션 로그에는 흔적이 없을 수 있습니다. grep 결과가 없다는 것은 "그 소스에 없다"는 뜻이지 "어디에도 기록이 없다"는 뜻이 아닙니다.

오해 5: "copytruncate 옵션을 쓰면 로그 로테이션 시 데이터 유실이 전혀 없다"

copytruncate는 파일 이동/삭제 문제를 피하지만, 파일을 복사한 시점과 실제로 원본을 truncate하는 시점 사이의 극히 짧은 간격에 쓰인 로그는 유실될 수 있습니다. 완벽한 무손실이 필요하다면 애플리케이션이 SIGHUP 등으로 로그 파일을 재오픈하도록 설계하는 방식이 더 안전합니다.

확장 시야 단일 서버 도구의 한계를 알고 분산 로그/트레이싱 필요성까지 연결 단일 서버 진단에는 능숙하지만 확장 논의는 약함 단일 서버 명령어 수준에서 답변 종료

점수 해석: 6개 항목 × 3점 = 최대 18점. 15점 이상 → Senior 수준 답변 완성. 11~14점 → 구조는 있지만 일부 키워드/관점 보완 필요. 10점 이하 → 이 레슨을 다시 읽고, 모범 답변을 소리 내어 말해보세요.

Day 5 핵심 포인트 요약

  • journalctl은 -u/-p/--since/--until/-b를 조합해서 좁혀야 효율적이다.
  • journald는 기본적으로 휘발성일 수 있으므로, 재부팅 전 증거 보전이 필요하다.
  • 증상 종류에 따라 journalctl/auth.log/dmesg/애플리케이션 로그 중 첫 확인 대상이 달라야 한다.
  • awk | sort | uniq -c로 정량적으로 집계해서 원인을 좁히는 습관이 중요하다.
  • logrotate의 copytruncate는 삭제된 열린 파일 문제를 피하지만 미세한 유실 가능성이라는 trade-off가 있다.
