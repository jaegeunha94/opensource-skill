❓ 오늘의 면접 질문

핵심 질문

"배포 스크립트가 애플리케이션 로그 파일에 쓰기를 시도하다가 'Permission denied' 에러로 실패했습니다. ls -la로 확인한 rwx 권한은 문제가 없어 보이는데도 계속 실패합니다. 어떤 순서로 원인을 좁혀나가시겠습니까?"

관련 질문

"운영팀 신규 입사자에게 서버 운영 권한을 줘야 합니다. root 비밀번호를 공유하는 방식과 sudo로 세분화된 권한을 주는 방식 중 무엇을 선택하고, 왜 그렇게 판단하시겠습니까?"

관련 질문

"80번 포트로 애플리케이션을 띄우려는데 일반 사용자 권한으로는 바인딩이 안 됩니다. root로 실행하는 것 말고 더 안전한 대안이 있을까요?"

💡

Day 1~7이 "무엇이 병목이고 무엇이 막혀 있는가"를 진단하는 능력이었다면, Day 8은 "누가 무엇을 할 수 있고 없는지"를 진단하고 설계하는 능력입니다. 면접관이 보는 것은 chmod 755 같은 숫자 암기가 아니라, 왜 이 권한 구조가 이렇게 설계되어야 하는지, 그리고 최소 권한 원칙(least privilege)을 실제 운영 판단에 적용할 수 있는지입니다.

📚 전제 개념 — 이것을 먼저 이해해야 한다

1. 리눅스 권한 모델 — "누구에게 어떤 열쇠를 줄 것인가"

쉽게 말하면: 리눅스의 모든 파일과 디렉터리는 "이 파일을 소유한 사람(owner)", "같은 그룹에 속한 사람들(group)", "그 외 모든 사람(other)" 세 부류로 나눠서 읽기(r)/쓰기(w)/실행(x) 권한을 각각 다르게 줄 수 있습니다. 마치 건물에 "본인 열쇠", "부서 공용 열쇠", "방문객 출입증"을 따로 발급하는 것과 같습니다. 

$ ls -la /var/log/myapp.log
-rw-r----- 1 appuser appgroup 15234 Jul  7 10:00 /var/log/myapp.log
# 앞 10자리 해부: -(파일종류) rw-(owner) r--(group) ---(other)
  • 숫자 표기(numeric): r=4, w=2, x=1을 더해서 3자리 숫자로 표현. rw-r----- = owner(rw-=6) group(r--=4) other(---=0) → chmod 640.
  • 문자 표기(symbolic): chmod u+x,g-w file처럼 기존 권한에 상대적으로 더하거나 빼는 방식 — 현재 권한을 모르는 상태에서 "실행 권한만 추가"하고 싶을 때 숫자 표기보다 안전.
  • 디렉터리의 x(실행) 권한은 "그 안으로 들어갈 수 있는가(cd 가능 여부)"를 의미 — 디렉터리에 r만 있고 x가 없으면 파일 목록은 보여도 안으로 들어가거나 그 안의 파일을 열 수 없는, 초보자가 자주 헷갈리는 조합.
  • chown user:group file로 소유자/그룹을 바꾸고, chmod로 권한을 바꾸는 것은 별개의 축 — "누가 소유하는가"와 "그 소유자가 무엇을 할 수 있는가"를 혼동하지 않는 것이 중요.
✅

면접 팁: 위 질문("rwx는 문제없어 보이는데 Permission denied")의 정답 후보 중 하나가 바로 이 디렉터리 x 권한입니다. 파일 자체의 권한만 보고 "문제없다"고 판단하지 않고, 그 파일이 위치한 모든 상위 디렉터리의 실행 권한까지 확인하는 습관이 senior와 junior를 가르는 지점입니다.

2. umask — "새로 만들어지는 파일의 기본 권한은 왜 이렇게 정해지나"

쉽게 말하면: 파일을 만들 때마다 매번 권한을 지정하지 않아도, 시스템이 자동으로 적용하는 "기본 차감값"이 umask입니다. 파일의 이론적 최대 권한(666, 실행 비트는 프로그램이 명시할 때만 부여)에서 umask 값을 뺀 것이 실제 생성되는 파일의 권한입니다. 

$ umask
0022
# 파일 기본 666 - umask 022 = 644 (rw-r--r--)
# 디렉터리 기본 777 - umask 022 = 755 (rwxr-xr-x)
$ umask 027   # 그룹 쓰기, other 전체 차단 — 더 보수적인 기본값
  • umask는 "권한을 부여하는 것"이 아니라 "기본값에서 빼는 것" — 이 방향을 거꾸로 이해하면 계산이 꼬입니다.
  • 애플리케이션 서비스 계정에 umask 027이나 077을 적용하면, 그 프로세스가 새로 만드는 로그/설정 파일이 처음부터 other에게 노출되지 않아 사후에 chmod로 조이는 것보다 근본적으로 안전합니다.
  • systemd 서비스 unit에서는 UMask= 지시자로 서비스별 umask를 개별 지정할 수 있어, 전역 셸 umask와 무관하게 일관된 권한을 강제할 수 있습니다.
💡

실무 연결: 여러 개발자가 같은 그룹으로 배포 서버에 접근하는 환경에서 umask가 022로 되어 있으면, 누군가 실수로 만든 설정 파일이 other에게 읽기 권한까지 열려버릴 수 있습니다. 공유 배포 계정은 umask를 027 이상으로 맞추는 것이 실무 관행입니다.

3. sudo / sudoers — "root 비밀번호를 나눠주지 않고 권한을 위임하는 방법"

쉽게 말하면: root 계정 비밀번호를 여러 사람과 공유하면 "누가 무엇을 했는지" 추적이 불가능하고, 권한을 회수하려면 비밀번호 자체를 바꿔야 합니다. sudo는 각자의 개인 계정으로 로그인한 상태에서, 정해진 범위의 명령어만 root 권한으로 실행하도록 위임하는 구조입니다 — 누가(who) 언제(when) 무엇을(what) 실행했는지 로그(/var/log/auth.log 또는 journalctl _COMM=sudo)에 그대로 남습니다. 

# /etc/sudoers (반드시 visudo로 편집 — 문법 오류 시 저장 전 검증)
$ sudo visudo
deploy ALL=(root) NOPASSWD: /usr/bin/systemctl restart myapp
%ops    ALL=(ALL) ALL
# deploy 계정은 딱 하나의 명령어만, 비밀번호 없이 실행 가능
# ops 그룹은 모든 명령어를 (비밀번호 입력 후) 실행 가능
  • visudo를 쓰는 이유: 직접 /etc/sudoers를 편집하다 문법 오류가 나면 어떤 sudo 명령어도 실행할 수 없게 잠기는 치명적 상황이 발생 — visudo는 저장 전 문법을 검증해 이를 방지.
  • NOPASSWD는 자동화 스크립트(배포 파이프라인 등)에서 사람의 개입 없이 특정 명령어를 실행해야 할 때 필요하지만, 범위를 넓게 주면(NOPASSWD: ALL) 사실상 root 비밀번호 공유와 다를 바 없어지는 trade-off가 있습니다.
  • 명령어를 와일드카드로 열어두면(/usr/bin/systemctl *) 의도한 것보다 넓은 권한이 열릴 수 있음 — 예를 들어 systemctl restart myapp만 허용하려 했는데 와일드카드 때문에 systemctl stop sshd까지 가능해지는 식.
⚠️

흔한 실수: 특정 프로그램에 sudo 권한을 줬는데, 그 프로그램 안에서 셸을 실행할 수 있는 기능(예: 일부 에디터의 :!sh, less의 !)이 있다면 결과적으로 root 셸을 얻는 것과 같습니다(GTFOBins로 잘 알려진 패턴). "이 명령어만 허용했으니 안전하다"는 판단은 그 명령어의 내부 기능까지 확인한 뒤에야 유효합니다.

4. setuid / setgid / sticky bit — "실행하는 순간 다른 사람의 권한을 빌린다"

쉽게 말하면: 보통 프로그램을 실행하면 그 프로그램을 실행한 사람의 권한으로 동작합니다. 그런데 setuid 비트가 설정된 프로그램은 "누가 실행했든 상관없이, 이 프로그램의 소유자 권한으로 동작"합니다. 대표적 예가 비밀번호를 바꾸는 passwd 명령어 — 일반 사용자가 실행해도 내부적으로는 /etc/shadow(root만 쓰기 가능)를 수정해야 하므로 setuid로 root 소유의 프로그램이 root 권한을 잠깐 빌려줍니다. 

$ ls -la /usr/bin/passwd
-rwsr-xr-x 1 root root 68208 ... /usr/bin/passwd
# owner의 x 자리가 's' — setuid 비트가 설정됨을 의미
# 숫자 표기로는 4자리: chmod 4755 file (맨 앞 4가 setuid)
$ find / -perm -4000 -type f 2>/dev/null   # setuid 걸린 파일 전수 조사(보안 점검 시 필수)
  • setgid(숫자 2xxx): 디렉터리에 걸면 그 안에서 새로 만들어지는 파일/디렉터리가 부모 디렉터리의 그룹을 자동 상속 — 여러 사용자가 협업하는 공유 디렉터리에서 그룹 권한 일관성을 유지하는 실무적으로 매우 유용한 설정.
  • sticky bit(숫자 1xxx, /tmp가 대표 예): 디렉터리에 걸면 그 디렉터리 안의 파일은 소유자 본인만 삭제/이름변경 가능 — 여러 사용자가 쓰기 권한을 공유하는 /tmp에서 남의 파일을 함부로 지우지 못하게 막는 안전장치.
  • setuid 프로그램은 강력한 만큼 위험도 큽니다 — 그 프로그램에 버그(예: 버퍼 오버플로)가 있으면 공격자가 root 권한을 그대로 획득할 수 있어, 보안 점검 시 find / -perm -4000으로 불필요한 setuid 바이너리를 주기적으로 점검하는 것이 실무 관행.
✅

면접 팁: "80번 포트 바인딩" 질문에서 setuid로 root 권한을 통째로 빌리는 방식(과거 관행)은 대안이 있다는 걸 아는 것이 핵심입니다 — 다음에 나올 Linux capability(CAP_NET_BIND_SERVICE)가 정확히 이 문제를 위한 더 안전한 해법입니다.

5. SELinux — "rwx가 다 맞아도 막힐 수 있다는 두 번째 잠금장치"

쉽게 말하면: 지금까지의 rwx 권한 모델을 DAC(Discretionary Access Control, 임의적 접근 제어)라고 부릅니다 — 파일 소유자가 "내 마음대로" 권한을 설정할 수 있다는 뜻입니다. 그런데 만약 애플리케이션이 해킹당해 공격자가 그 애플리케이션 계정 권한을 통째로 얻으면, DAC만으로는 그 계정이 접근 가능한 모든 파일이 뚫립니다. SELinux(MAC, Mandatory Access Control)는 여기에 "이 프로세스는 설령 소유자 권한이 있어도, 정책에 명시된 라벨 (context)이 일치하는 자원에만 접근 가능"이라는 두 번째 독립적인 잠금을 더합니다. 

$ ls -Z /var/www/html/index.html
system_u:object_r:httpd_sys_content_t:s0 /var/www/html/index.html
# httpd 프로세스는 httpd_sys_content_t 라벨이 붙은 파일만 읽도록 정책에 정의됨
$ getenforce
Enforcing
$ sudo ausearch -m avc -ts recent   # SELinux가 막은 시도 로그 확인 (거부 이유 파악)
$ sudo semanage fcontext -a -t httpd_sys_content_t "/data/web(/.*)?" && restorecon -Rv /data/web
  • rwx 권한(DAC)이 완벽해도, SELinux 컨텍스트(MAC)가 정책과 맞지 않으면 접근이 거부됩니다 — 이 질문의 정답 후보 중 가장 중요한 것이 바로 이 지점: "권한은 맞는데 안 된다"는 SELinux/AppArmor를 먼저 의심해야 하는 강한 신호.
  • 웹 콘텐츠를 표준 경로(/var/www) 밖으로 옮기면 새 위치는 기본적으로 올바른 SELinux 컨텍스트가 없어 접근이 거부됩니다 — mv가 아니라 semanage fcontext + restorecon으로 컨텍스트를 명시적으로 지정해야 함.
  • AppArmor(Ubuntu/Debian 계열)는 SELinux와 목적은 같지만(MAC), 파일 라벨 대신 경로 기반 프로파일로 프로세스별 허용 범위를 정의 — 접근 방식은 다르지만 "DAC를 통과해도 두 번째 정책 계층이 막을 수 있다"는 개념은 동일.
⚠️

trade-off: setenforce 0으로 SELinux를 끄면 문제가 "해결된 것처럼" 보이지만, 이는 원인을 찾은 게 아니라 보안 계층 전체를 제거한 것입니다. 정답은 ausearch -m avc로 실제 거부된 컨텍스트를 확인하고, semanage/restorecon으로 올바른 라벨을 지정하거나 필요한 정책만 audit2allow로 추가하는 것 — "끄기"와 "고치기"는 완전히 다른 조치입니다.

6. Linux Capability — "root냐 아니냐"의 이분법을 잘게 쪼갠다

쉽게 말하면: 전통적으로 리눅스 권한은 "root(전지전능) vs 일반 사용자(제한적)"라는 두 단계뿐이었습니다. 하지만 실제로는 "1024번 이하 포트 바인딩", "시스템 시간 변경", "네트워크 인터페이스 설정 변경"처럼 root 권한 중에서도 아주 좁은 기능 하나만 필요한 경우가 많습니다. capability는 root의 거대한 권한 덩어리를 세분화된 조각(약 40개 이상)으로 나눠서, 필요한 조각 하나만 프로세스에 부여할 수 있게 한 메커니즘입니다. 

# 일반 사용자로도 80번(1024 이하) 포트를 바인딩할 수 있도록, 그 기능만 부여
$ sudo setcap 'cap_net_bind_service=+ep' /usr/bin/myapp
$ getcap /usr/bin/myapp
/usr/bin/myapp cap_net_bind_service=ep
# 컨테이너에서 불필요한 capability를 제거해 공격 표면을 줄이는 것도 같은 원리
$ docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myapp
  • setuid root와 비교하면, capability는 "필요한 기능 하나만" 부여하므로 그 프로그램에 취약점이 있어도 공격자가 얻는 권한의 범위가 훨씬 좁습니다 — 최소 권한 원칙(least privilege)의 정석적인 구현.
  • systemd 서비스 unit의 AmbientCapabilities=CAP_NET_BIND_SERVICE로도 동일한 효과를 낼 수 있어, 바이너리 자체를 건드리지 않고 서비스 실행 시점에 capability를 부여하는 방법도 실무에서 많이 사용됩니다.
  • 컨테이너 환경(Docker/Kubernetes)에서는 기본적으로 상당수 capability가 이미 제한되어 있는데, 여기에 추가로 --cap-drop=ALL 후 필요한 것만 --cap-add하는 것이 컨테이너 보안 강화의 표준 패턴.
💡

실무 연결: "왜 root로 실행하면 간단한데 굳이 capability를 쓰는가"라는 질문에는, "root로 실행된 프로세스가 원격 코드 실행 취약점에 걸리면 시스템 전체가 뚫리지만, cap_net_bind_service만 가진 프로세스가 뚫려도 공격자가 추가로 얻는 것은 포트 바인딩 권한뿐"이라는 피해 반경(blast radius) 축소 논리로 답하는 것이 senior다운 답변입니다.

🎯 왜 면접에서 묻는가

권한/보안 질문은 chmod 숫자 암기가 아니라, "이 시스템에 누가 무엇을 할 수 있어야 하는가"를 최소 권한 원칙에 따라 설계하고, 권한 문제가 발생했을 때 DAC과 MAC 두 계층을 구분해서 원인을 좁힐 수 있는지를 평가합니다.

면접관이 실제로 평가하는 것

  1. 권한 문제를 계층별로 진단하는가?
    "Permission denied"라는 하나의 증상 뒤에 파일 rwx, 상위 디렉터리 실행 권한, SELinux/AppArmor 컨텍스트라는 여러 계층이 있다는 것을 알고 순서대로 확인할 수 있는지를 봅니다.
  2. 최소 권한 원칙을 실제로 설계에 적용하는가?
    root 비밀번호 공유 대신 sudo로 세분화하고, setuid 전체 권한 대신 capability로 필요한 기능만 부여하는 식으로 "필요한 만큼만" 권한을 설계하는 습관이 있는지를 봅니다.
  3. 보안 계층을 끄는 것과 고치는 것을 구분하는가?
    SELinux를 끄거나 sudo 권한을 무한정 넓히는 임시방편의 위험을 인지하고, 근본 원인(정책/컨텍스트/범위)을 고치는 방향으로 판단하는지를 봅니다.
  4. 추적 가능성(accountability)을 이해하는가?
    root 직접 로그인 대신 개인 계정 + sudo를 쓰는 이유가 "누가 언제 무엇을 했는지" 감사 로그로 추적하기 위함이라는 것을 설명할 수 있는지를 봅니다.
💡

본질은 "권한은 편의를 위해 넓히는 것이 아니라, 사고가 났을 때의 피해 반경을 줄이기 위해 최소한으로 설계하는 것"이라는 사고방식입니다. 이 감각이 없으면 운영 편의를 위해 권한을 넓히다가, 사고 한 번에 시스템 전체가 뚫리는 결과로 이어집니다.

🏭 실무 연결 — 이런 상황에서 실제로 쓴다

시나리오 A: rwx는 정상인데 계속 Permission denied

배포 스크립트가 /data/app/logs/app.log에 쓰기를 시도하다 실패했습니다. 파일 자체의 권한(rw-rw-r--)과 소유자는 문제가 없었지만, 상위 디렉터리인 /data/app이 최근 보안 점검 과정에서 chmod 700으로 바뀌어 있었습니다. 배포 계정이 그 그룹에 속하지 않아 디렉터리 진입 자체가 막힌 것이 원인이었고, 그룹 소유권을 조정한 뒤 chmod 750으로 되돌려 문제를 해결했습니다.

시나리오 B: 신규 웹 서버에서 SELinux가 정적 파일 접근을 막음

새 스토리지 볼륨을 /mnt/newdisk/www에 마운트하고 심볼릭 링크로 웹 루트를 연결했더니 502 에러가 발생했습니다. ausearch -m avc -ts recent로 확인한 결과 SELinux가 httpd 프로세스의 접근을 거부한 로그가 쌓여 있었고, 새 경로에 httpd_sys_content_t 컨텍스트가 없는 것이 원인이었습니다. semanage fcontext로 컨텍스트를 등록하고 restorecon으로 적용한 뒤 정상화되었으며, 이 경험을 계기로 신규 마운트 볼륨 체크리스트에 SELinux 컨텍스트 확인 단계를 추가했습니다.

시나리오 C: 운영팀 권한 위임 재설계

기존에는 운영팀 전원이 root 비밀번호를 공유해 서버에 로그인했습니다. 감사 추적이 불가능하고 퇴사자 발생 시 전체 서버 root 비밀번호를 바꿔야 하는 부담이 있어, 개인 계정 + sudo 구조로 전환했습니다. 일상적인 서비스 재시작/로그 확인은 NOPASSWD로 세분화된 명령어만 허용하고, 시스템 설정 변경처럼 위험도 높은 작업은 비밀번호 재확인을 요구하도록 sudoers를 계층화했습니다.

실무에서 배운 핵심 교훈

  • "Permission denied"는 파일 권한 → 상위 디렉터리 권한 → SELinux/AppArmor 순서로, 최소 두 개의 독립적인 계층을 확인해야 완전한 진단이다.
  • root 권한은 공유하는 것이 아니라 위임하는 것 — sudo와 capability 모두 "필요한 만큼만, 추적 가능하게" 권한을 나눠주는 같은 철학의 다른 구현이다.
  • 보안 계층(SELinux, 방화벽 등)을 끄는 것과 원인을 찾아 고치는 것은 전혀 다른 조치이며, 끄는 순간 감춰졌던 위험이 그대로 열린다.
  • 새로운 인프라(볼륨 마운트, 신규 서비스 계정)를 추가할 때는 rwx뿐 아니라 SELinux 컨텍스트, capability, sudoers 범위까지 체크리스트에 포함해야 재발을 막을 수 있다.

🧠 답변 사고 순서 — 권한 문제 진단 흐름

  • 대상 파일/프로세스의 정확한 정체 확인

    어떤 프로세스(어떤 사용자 계정)가, 어떤 파일/포트/자원에 접근하려다 실패했는지부터 명확히 합니다. ps aux | grep과 id로 실제 실행 계정을 확인하는 것이 출발점입니다.

  • DAC(rwx) 계층 확인 — 파일과 모든 상위 디렉터리

    ls -la로 파일 자체의 owner/group/other 권한을 확인하고, 그 파일이 위치한 모든 상위 디렉터리의 실행(x) 권한까지 순서대로 확인합니다. 하나라도 막혀 있으면 그 지점이 원인입니다. 

    $ namei -l /data/app/logs/app.log   # 경로 전체의 권한을 한 번에 확인

  • MAC(SELinux/AppArmor) 계층 확인

    DAC이 모두 정상인데도 거부된다면 getenforce로 SELinux 활성 여부를 확인하고, ausearch -m avc -ts recent 또는 journalctl에서 최근 거부(denied) 로그를 찾아 정확한 원인 컨텍스트를 확인합니다.

  • 권한 위임 구조 점검(sudo/capability)

    특정 계정이 특정 작업을 못 하는 상황이라면, 그 계정에 필요한 권한이 sudoers나 capability로 충분히, 그러나 과하지 않게 부여되어 있는지 확인합니다.

  • 최소 권한 원칙에 맞는 조치와 재발 방지

    문제를 "끄는" 방식(SELinux 비활성화, 777 권한 부여, root 실행)이 아니라 정확한 원인 위치에 필요한 만큼만 권한을 부여하는 방식으로 해결하고, 향후 유사한 인프라 변경 시 체크리스트에 반영합니다.

💬 핵심 답변 예시

면접에서 자연스럽게 말할 수 있는 구조입니다. 외우지 말고 논리의 흐름을 이해하세요.

💬 면접관: "rwx 권한은 문제없어 보이는데 계속 Permission denied가 발생합니다. 어떻게 접근하시겠습니까?"

"가장 먼저 실제로 접근을 시도하는 프로세스가 어떤 계정으로 실행되는지 ps aux와 id로 확인합니다. 그 다음 대상 파일뿐 아니라 namei -l로 경로 전체의 상위 디렉터리 실행 권한까지 확인합니다 — 파일 자체는 문제없어도 상위 디렉터리 하나가 막혀 있으면 같은 증상이 나타나기 때문입니다.

여기까지 DAC(전통적 rwx) 관점에서 문제가 없다면, getenforce로 SELinux가 Enforcing 상태인지 확인하고, ausearch -m avc -ts recent로 최근 거부 로그를 찾습니다. rwx가 통과해도 SELinux 컨텍스트가 정책과 맞지 않으면 똑같이 Permission denied가 발생할 수 있기 때문입니다.

원인을 찾은 뒤에는 SELinux를 끄거나 권한을 777로 여는 방식이 아니라, semanage fcontext로 올바른 컨텍스트를 지정하거나 필요한 그룹 권한만 정확히 부여하는 방식으로 해결합니다. 이렇게 하면 문제를 감추지 않고 정확한 지점만 열어주기 때문에 다른 보안 경계는 그대로 유지됩니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "SELinux가 원인인지 아닌지 가장 빠르게 확인하는 방법은 무엇인가요?"

"임시로 setenforce 0으로 Permissive 모드로 바꿔서 같은 동작을 재현해보는 방법이 있습니다. Permissive 모드에서는 SELinux가 정책 위반을 '차단'하지 않고 '로그만' 남기기 때문에, 이 상태에서 문제가 사라진다면 SELinux가 원인이라는 강한 증거가 됩니다. 다만 이건 어디까지나 진단 목적의 임시 조치이고, 확인 즉시 다시 setenforce 1로 되돌리고 ausearch 로그를 근거로 정확한 컨텍스트를 고치는 것이 다음 단계입니다. Permissive 상태로 방치하는 것은 보안 계층을 사실상 꺼둔 것과 같습니다."

평가 포인트: 진단용 임시 조치와 영구적 해결책을 구분하고, Permissive를 방치하지 않는가

Q. "sudoers에 NOPASSWD를 광범위하게 설정하면 어떤 위험이 있나요?"

"가장 큰 위험은 그 계정이 해킹당했을 때, 공격자가 비밀번호 확인 없이 곧바로 root 권한의 모든 명령을 실행할 수 있다는 점입니다. 즉 그 계정의 인증 자체가 시스템 전체의 보안 경계가 되어버립니다. 자동화 파이프라인처럼 사람이 개입할 수 없는 경우에는 불가피하게 NOPASSWD가 필요하지만, 그럴 때는 범위를 '서비스 재시작' 같은 구체적인 명령어 하나로 최대한 좁히고, 와일드카드나 셸 접근이 가능한 프로그램은 제외해야 합니다. 즉 '필요성'과 '범위'를 분리해서 판단해야 합니다."

평가 포인트: NOPASSWD의 존재 이유와 범위를 분리해서 판단하고, 공격 시 피해 반경을 설명하는가

Q. "setuid root 프로그램 대신 capability를 쓰는 것이 항상 더 나은가요?"

"대부분의 경우 그렇지만 항상은 아닙니다. capability는 커널이 지원하는 특정 기능 단위로만 세분화되어 있기 때문에, 프로그램이 필요한 동작이 딱 맞는 capability 하나로 떨어지지 않고 여러 개를 조합해야 한다면 오히려 setuid보다 관리가 복잡해질 수 있습니다. 또한 파일 시스템이 capability를 지원하지 않는 환경(일부 네트워크 파일시스템)에서는 아예 적용이 안 되는 제약도 있습니다. 그래서 저는 '이 프로그램이 필요한 권한이 하나의 capability로 명확히 떨어지는가'를 먼저 확인하고, 그렇다면 capability를, 그렇지 않고 범위가 넓다면 차라리 setuid 대신 별도의 권한 상승 없는 헬퍼 프로세스로 분리하는 구조를 고려합니다."

평가 포인트: capability를 만능 해법으로 보지 않고 적용 한계와 대안까지 고려하는가

Q. "umask를 서비스별로 다르게 관리해야 하는 이유가 있을까요?"

"서비스마다 다루는 데이터의 민감도가 다르기 때문입니다. 예를 들어 결제 정보를 다루는 서비스가 만드는 로그 파일은 처음부터 other 권한을 완전히 차단(umask 077)하는 것이 안전하지만, 여러 팀이 함께 보는 배치 작업 산출물은 그룹 읽기 권한 정도는 열어두는 것(umask 027)이 협업에 유리합니다. 전역 umask 하나로 통일하면 한쪽은 과도하게 열리고 다른 쪽은 과도하게 막히는 문제가 생기므로, systemd unit의 UMask=로 서비스별 기본값을 분리하는 것이 실무적으로 더 정확합니다."

평가 포인트: umask를 전역 설정이 아니라 데이터 민감도에 따른 서비스별 설계 요소로 보는가

Q. "컨테이너 환경에서는 이 권한 모델이 어떻게 달라지나요?"

"기본 rwx, sudo, capability 개념은 그대로 적용되지만, 컨테이너는 기본적으로 호스트 root와 컨테이너 root가 같은 UID를 공유하는 경우가 많아서 컨테이너 탈출 시 호스트 root 권한으로 이어질 위험이 있습니다. 그래서 저는 --cap-drop=ALL로 불필요한 capability를 모두 제거한 뒤 필요한 것만 --cap-add하고, 가능하면 non-root 사용자로 프로세스를 실행하도록 Dockerfile에 USER를 명시하며, User Namespace remapping으로 컨테이너 내부 root를 호스트의 일반 사용자로 매핑하는 방식까지 고려합니다. 결국 같은 최소 권한 원칙을 컨테이너 경계에도 동일하게 적용하는 것입니다."

평가 포인트: 전통적 권한 모델의 원칙을 컨테이너라는 새 경계에도 일관되게 확장하는가

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"권한 문제니까 chmod 777로 바꿔서 일단 되게 하겠습니다."
왜 약한 답변인가:
  • 원인을 찾지 않고 모든 권한을 열어버려 보안 경계를 완전히 무너뜨립니다.
  • DAC과 MAC(SELinux) 두 계층을 구분하지 못해 실제 원인이 SELinux라면 아예 해결되지 않습니다.
  • "왜 안 되는지"에 대한 진단 과정 없이 결과만 맞추려는 접근입니다.
좋은 답변 (Senior 수준)
"먼저 파일과 상위 디렉터리 전체의 rwx를 확인하고, 그래도 안 되면 ausearch로 SELinux 거부 로그를 확인해서 정확한 컨텍스트 문제를 찾은 뒤, 그 지점에만 필요한 권한을 정확히 부여하겠습니다."
왜 좋은 답변인가:
  • DAC(rwx) → MAC(SELinux) 순서로 계층을 구분해서 진단합니다.
  • namei -l, ausearch 같은 구체적인 진단 도구와 근거를 제시합니다.
  • "필요한 만큼만" 권한을 부여해 최소 권한 원칙을 지킵니다.
  • 보안 계층을 끄는 대신 정확한 원인을 고치는 방향을 선택합니다.

🚫 흔한 오해와 정확한 이해

오해 1: "파일의 rwx 권한만 맞으면 접근이 가능하다"

rwx가 완벽해도 상위 디렉터리의 실행(x) 권한이 없거나, SELinux/AppArmor 같은 MAC 정책이 별도로 막고 있으면 접근이 거부됩니다. rwx는 여러 계층 중 하나일 뿐입니다.

오해 2: "SELinux는 문제만 일으키니 끄는 게 정석이다"

SELinux를 끄는 것은 원인을 고치는 것이 아니라 보안 계층 전체를 제거하는 것입니다. 정석은 ausearch로 정확한 거부 원인을 찾아 컨텍스트를 고치거나 필요한 정책만 추가하는 것입니다.

오해 3: "sudo 권한을 주는 것과 root 비밀번호를 공유하는 것은 결국 같다"

sudo는 개인 계정으로 인증하고 실행 범위를 좁힐 수 있으며, 모든 실행이 로그로 남아 추적 가능합니다. root 비밀번호 공유는 누가 무엇을 했는지 추적이 불가능하고 권한 회수도 어렵습니다. 통제 가능성 자체가 다릅니다.

오해 4: "80번 포트를 쓰려면 반드시 root로 실행해야 한다"

setcap cap_net_bind_service=+ep로 그 기능 하나만 부여하면 일반 사용자 권한으로도 1024번 이하 포트를 바인딩할 수 있습니다. root 전체 권한을 줄 필요가 없습니다.

오해 5: "umask는 한 번 설정하면 시스템 전체에 항상 똑같이 적용해야 한다"

서비스마다 다루는 데이터의 민감도가 다르므로 umask도 서비스별로 다르게 관리하는 것이 정확합니다. systemd unit의 UMask=로 서비스 단위 기본값을 개별 지정할 수 있습니다.

Day 8 핵심 포인트 요약

  • "Permission denied"는 파일 rwx → 상위 디렉터리 rwx → SELinux/AppArmor 순서로 최소 두 개 이상의 독립적인 계층을 확인해야 완전한 진단이다.
  • root 권한은 공유가 아니라 위임 — sudo와 capability 모두 "필요한 만큼만, 추적 가능하게"라는 같은 원칙의 다른 구현이다.
  • SELinux를 끄는 것과 ausearch/semanage로 고치는 것은 전혀 다른 조치다.
  • setuid root 대신 capability로 필요한 기능만 부여하면 사고 시 피해 반경이 줄어든다.
  • umask는 전역 설정이 아니라 서비스별 데이터 민감도에 맞춰 설계하는 요소다.
