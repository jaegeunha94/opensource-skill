💡

📚 전제 개념 — 이것을 먼저 이해해야 한다

1. Load Average — "서버가 얼마나 바쁜가"의 지표

쉽게 말하면: 슈퍼마켓 계산대에 줄 서 있는 사람 수의 평균입니다. 계산대(CPU)가 1개인데 10명이 줄 서 있다면 매우 혼잡한 상태이고, 2개인데 1명이 줄 서 있다면 여유 있는 상태입니다.

기술적 정의: 리눅스에서 load average는 실행 대기 중이거나(R 상태) I/O 대기 중인(D 상태) 프로세스 수의 이동 평균입니다. uptime이나 top으로 확인할 수 있는 세 숫자(1분, 5분, 15분 평균)로 표시됩니다. 

$ uptime
 14:32:01 up 42 days,  3:17,  2 users,  load average: 2.45, 1.89, 1.23
                                                        ────  ────  ────
                                                        1분   5분   15분

해석 방법: load average 숫자를 CPU 코어 수로 나눠서 판단합니다.

  • 코어 4개, load 2.0 → 50% 사용 (여유 있음)
  • 코어 4개, load 4.0 → 100% 포화 (처리 한계)
  • 코어 4개, load 8.0 → 200% 초과 (심각한 병목)
⚠️

중요: load average는 CPU만의 지표가 아닙니다. D 상태(Uninterruptible Sleep = I/O 대기) 프로세스도 포함되기 때문에, load가 높다고 반드시 CPU 문제는 아닙니다. 디스크 I/O 병목이 원인일 수 있습니다.

트렌드 해석: 1분 > 15분이면 상황이 악화 중, 1분 < 15분이면 회복 중입니다.

2. 프로세스 상태 — R, D, S, Z의 차이

top이나 ps에서 보이는 S 열의 의미입니다. 장애 원인을 좁히는 핵심 단서입니다.

상태 코드 의미 load average에 포함? 장애 신호?
RRunning / Runnable (실행 중 또는 실행 대기)✅ 포함R 프로세스가 많으면 CPU 포화
DUninterruptible Sleep (I/O 대기, 깨울 수 없음)✅ 포함D 프로세스가 많으면 I/O 병목
SInterruptible Sleep (이벤트 대기, 정상)❌ 미포함정상 상태
ZZombie (종료했지만 부모가 wait 안 함)❌ 미포함누적되면 PID 고갈 위험
✅

면접 팁: "CPU 사용률은 낮은데 서버가 느리다"는 상황이 나오면 D 상태 프로세스 개수를 확인하는 것이 첫 번째 답변 포인트입니다. ps aux | awk '$8 == "D"' | wc -l로 확인합니다.

3. CPU 사용률의 분류 — us, sy, wa, id의 차이

top의 CPU 줄에서 보이는 항목들입니다. 각 항목이 어떤 의미인지 알아야 "CPU 바쁘다"를 정확히 해석할 수 있습니다. 

%Cpu(s): 45.2 us,  3.1 sy,  0.0 ni, 46.5 id, 4.8 wa,  0.0 hi,  0.4 si
         ──────    ────                 ────       ────
         사용자     시스템               유휴        I/O 대기
  • us (user): 사용자 프로세스가 CPU를 사용 중. 높으면 애플리케이션 로직 과부하.
  • sy (system): 커널이 CPU를 사용 중. 높으면 시스템 콜, I/O 처리가 많음.
  • wa (iowait): CPU가 I/O 완료를 기다리며 유휴 상태. 높으면 디스크 I/O 병목.
  • id (idle): 아무것도 안 하는 상태. 낮을수록 CPU가 바쁨.
  • si (softirq): 소프트웨어 인터럽트. 높으면 네트워크 패킷 처리 과부하 의심.
⚠️

흔한 착각: wa가 높을 때 "CPU가 바쁘다"고 말하는 건 틀렸습니다. wa는 CPU가 I/O를 기다리며 멈춰 있는 시간입니다. 원인은 디스크 I/O입니다.

4. USE 방법론 — 체계적 성능 진단의 표준

Brendan Gregg (Netflix 성능 엔지니어)가 제안한 방법론으로, 어떤 리소스든 세 가지 관점에서 확인합니다. 면접에서 "어떻게 접근하시겠습니까?"라는 질문에 이 틀을 사용하면 체계적이고 신뢰감 있는 답변이 됩니다.

관점 질문 CPU 예시 명령어
Utilization (사용률) 이 리소스가 얼마나 사용되고 있나? top의 %Cpu us+sy
Saturation (포화) 처리 못 해서 대기가 쌓이나? vmstat의 r 열 (run queue)
Errors (오류) 오류가 발생하고 있나? dmesg, journalctl

이 방법론을 CPU, 메모리, 디스크, 네트워크 각 리소스에 반복 적용합니다. "한 리소스가 괜찮으면 다음 리소스로 넘어가는" 체계적 점검이 가능해집니다.

🎯 왜 면접에서 묻는가

"서버 응답이 느려졌을 때 어떻게 하시겠습니까?"는 표면적으로 단순해 보이지만, 면접관이 평가하는 것은 명령어 암기가 아닙니다.

면접관이 실제로 평가하는 것

  1. 당황하지 않고 체계적으로 접근하는가?
    10년차 개발자라면 "일단 top 켜보자"가 아니라 "먼저 영향 범위를 파악하고, 데이터를 수집하고, 가설을 세워서 검증한다"는 순서가 있어야 합니다.
  2. 가설 없이 데이터를 먼저 보는가?
    경험 없는 개발자는 증상을 보자마자 "메모리 부족이겠지", "DB가 느리겠지"처럼 직감으로 행동합니다. Senior는 먼저 데이터를 보고 범위를 좁힙니다.
  3. 영향 범위와 비즈니스 임팩트를 먼저 파악하는가?
    기술적 원인 파악보다 "지금 사용자 몇 명이 영향받고 있는가", "전체 서비스 장애인가 일부인가"를 먼저 파악해야 우선순위 결정이 가능합니다.
  4. 명령어를 아는 것과 해석을 아는 것이 다르다는 걸 보여주는가?
    "top 칩니다"가 아니라 "top으로 D 상태 프로세스 수를 확인해서 I/O 병목 여부를 판단합니다"가 다릅니다.
  5. 즉각 조치와 장기 대책을 구분하는가?
    프로세스를 kill하는 것은 즉각 조치지만, 왜 그 프로세스가 CPU를 폭식했는지 모르면 재발합니다. Senior는 "이번 조치"와 "재발 방지"를 함께 생각합니다.
💡

이 질문은 사실 "당신은 서버 장애 상황에서 리더로서 팀을 이끌 수 있는가?"를 묻는 질문입니다. 기술 지식뿐 아니라 소통, 우선순위 판단, 재발 방지 사고까지 함께 보여주어야 합니다.

🏭 실무 연결 — 이런 상황에서 실제로 쓴다

시나리오 A: API 응답 지연 신고

오후 2시, 고객 CS에서 "결제 API가 30초 넘게 안 된다"는 신고가 들어옵니다. 모니터링 대시보드를 봤더니 API 서버의 CPU 사용률은 15%로 낮습니다. 하지만 load average는 평소의 5배입니다.

이 상황에서 CPU 사용률만 보고 "CPU 문제 아님"이라고 결론 내리면 틀립니다. load average가 높은데 CPU가 낮다는 것은 D 상태 프로세스, 즉 I/O 대기를 의심해야 합니다. 실제 원인은 데이터베이스 디스크 풀이었을 수 있습니다.

시나리오 B: 새벽 배포 후 메모리 경보

새벽 3시, 배포 후 메모리 사용량이 급증해서 OOM Killer가 프로세스를 종료했습니다. 애플리케이션이 재시작했지만 사용자 요청은 일부 실패했습니다. "메모리가 충분했는데 왜?"라는 질문에 답하려면 dmesg | grep -i oom과 journalctl -xe로 OOM Killer가 어떤 프로세스를 왜 종료했는지 확인해야 합니다.

실무에서 배운 핵심 교훈

  • 장애 시 가장 먼저 확인할 것: 최근 배포/변경 이력. 대부분의 장애는 변경에서 시작됩니다.
  • 빠른 조치(kill, restart)와 근본 원인 분석(RCA)을 분리해서 생각해야 합니다.
  • 장애 중에 내린 결정과 그 근거를 실시간으로 기록하면 포스트모템이 쉬워집니다.
  • "나 혼자 해결"이 아니라 관련 팀(DBA, 인프라팀, 개발팀)에 동시에 연락하는 것이 Senior의 역할입니다.

🧠 답변 사고 순서 — 5단계 Triage 프레임워크

아래 순서를 외우는 것이 목표가 아닙니다. 이 순서 뒤에 있는 "왜"를 이해하면 응용 상황에서도 같은 논리로 접근할 수 있습니다.

  • 상황 파악 & 영향 범위 확인 (30~60초)

    기술적 해결보다 먼저 해야 할 일입니다. 전체 사용자가 영향받는지, 일부인지에 따라 긴급도가 달라집니다. 최근 배포나 설정 변경이 있었는지 확인합니다.

    • 어떤 서비스가 느린가? (API 전체 vs 특정 엔드포인트)
    • 언제부터? (특정 시간 이후 → 배포/트래픽 급증 의심)
    • 얼마나 많은 사용자? (전체 장애 vs 일부 지역 vs 특정 기능)
    • 최근 30분 내 배포, 설정 변경, 유지보수가 있었나?
    ✅

    면접에서 이 단계를 먼저 언급하는 것만으로도 Senior로서의 사고방식을 보여줍니다. 많은 지원자가 영향 범위 파악을 생략하고 바로 명령어로 넘어갑니다.

  • 시스템 전반 상태 스냅샷 (1~2분)

    빠르게 전체 리소스 상태를 보고 어느 범주에서 문제인지 파악합니다. 아직 원인을 결론짓지 않고, 가설의 범위를 좁히는 단계입니다. 

    # 1. load average + CPU/메모리 상위 프로세스 한 번에 보기
$ top -bn1 | head -25

# 2. 메모리 상태 (실제 사용 가능 메모리 = available 열)
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           31G         24G        512M        1.2G        6.5G        5.8G

# 3. 디스크 여유 공간 (/ 또는 /data 가 가득 차면 서버 전체 마비)
$ df -h

# 4. vmstat로 CPU, 메모리, I/O, 스왑 흐름을 한 번에
$ vmstat 1 5
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 2  0      0 524288 102400 6553600    0    0     0    12  892 1234 45  3 48  4  0
 3  0      0 520000 102400 6553600    0    0     0     0  910 1251 48  4 43  5  0
# r: 실행 대기 프로세스, b: I/O 대기 프로세스, wa: iowait

  • 병목 범주 결정 — 어느 리소스인가?

    스냅샷에서 본 데이터로 CPU, 메모리, Disk I/O, 네트워크 중 어느 쪽인지 좁힙니다. 이 판단이 다음 단계 명령어를 결정합니다.

    신호 판단 다음 단계
    us 높음, R 프로세스 많음 CPU 병목 (애플리케이션) ps aux --sort=-%cpu, profiling
    sy 높음 시스템 콜 / I/O 처리 과부하 strace, perf
    wa 높음, D 프로세스 많음 Disk I/O 병목 iostat -xz 1, iotop
    swap 사용 증가, si/so 0이 아님 메모리 부족 (OOM 직전) free -h, dmesg | grep oom
    리소스 이상 없음, si 높음 네트워크 패킷 과부하 ss -s, ip -s link

  • 원인 특정 — 어떤 프로세스/서비스인가?

    범주가 결정되면 구체적인 원인 프로세스나 서비스를 특정합니다. 명령어를 실행하기 전에 "나는 지금 X를 확인하려고 이 명령어를 쓴다"고 의도를 먼저 설명하는 것이 면접에서 좋은 인상을 줍니다. 

    # CPU 병목: CPU 사용률 상위 프로세스
$ ps aux --sort=-%cpu | head -10

# Disk I/O 병목: 어떤 디스크에 얼마나 쓰이는가
$ iostat -xz 1 5
Device:    r/s   w/s  rkB/s  wkB/s  await  r_await  w_await  util
sda       10.0  250.0   80.0 4000.0  45.23    2.10   47.30   92.5%
# util 90% 이상 = 디스크가 거의 포화 상태

# I/O를 가장 많이 쓰는 프로세스 찾기
$ iotop -o -b -n 3

# 메모리: OOM Killer가 무엇을 죽였는가
$ dmesg -T | grep -i "oom\|out of memory"

# 특정 프로세스가 어떤 파일을 열고 있는가
$ lsof -p <PID>

# 서비스 로그 확인
$ journalctl -u myapp.service -n 100 --no-pager

  • 조치 결정 + 재발 방지 계획

    원인이 특정되었어도 즉시 kill하거나 재시작하기 전에 판단이 필요합니다. 이 판단 과정을 면접에서 설명할 수 있어야 합니다.

    • 즉각 조치 기준: 전체 서비스 장애 또는 데이터 손상 위험이 있으면 원인 파악보다 서비스 복구를 먼저 합니다. (프로세스 재시작, 배포 롤백)
    • 데이터 수집 후 조치: 서비스가 느리지만 동작 중이라면 스택 트레이스, 로그, strace 출력을 먼저 수집하고 조치합니다. kill하면 원인 데이터가 사라집니다.
    • 재발 방지: 조치 후 "왜 이 상황이 발생했는가"의 근본 원인을 찾고, 모니터링 알람 임계값 조정, 코드/설정 수정, 포스트모템 작성이 필요합니다.
    ⚠️

    kill -9는 마지막 수단입니다. 정상 종료 신호(SIGTERM)를 보내고 grace period를 준 다음, 그래도 안 되면 SIGKILL을 사용합니다. 데이터 손상이나 파일 잠금 문제가 생길 수 있기 때문입니다.

💬 면접관: "운영 서버 응답이 느려졌다는 신고가 들어왔습니다. 어떻게 접근하시겠습니까?"

"먼저 기술적 원인 파악보다 영향 범위를 빠르게 파악하겠습니다. 전체 사용자가 영향받는지, 특정 기능만인지, 그리고 최근 배포나 설정 변경이 있었는지 확인합니다. 이 정보에 따라 즉각 롤백이 필요한지, 아니면 원인 분석에 시간을 쓸 수 있는지 판단합니다.

그 다음 시스템 전반 상태 스냅샷을 빠르게 찍습니다. uptime으로 load average를 보고, top으로 CPU와 메모리 상위 프로세스를, df -h로 디스크 여유 공간을 확인합니다. 특히 vmstat 1 5를 보면 CPU iowait, 스왑 사용, run queue 크기를 한 번에 볼 수 있어서 병목이 CPU인지 I/O인지 메모리인지 방향을 잡는 데 유용합니다.

예를 들어 iowait가 높고 D 상태 프로세스가 많다면 Disk I/O 병목으로 보고, iostat -xz 1로 디스크 utilization을 확인하고 iotop으로 어떤 프로세스가 I/O를 점유하는지 특정합니다. 반대로 us(user CPU)가 높고 R 상태 프로세스가 많다면 CPU 병목으로 보고, ps aux --sort=-%cpu로 범인을 찾습니다.

원인이 특정되면 조치 전에 데이터를 먼저 확보합니다. 프로세스를 바로 kill하면 스택 트레이스나 코어 덤프가 사라져서 왜 이 상황이 발생했는지 알 수 없게 됩니다. 서비스가 동작 중이라면 로그와 스택 트레이스를 먼저 수집하고 조치를 결정합니다.

마지막으로 서비스 복구 후에도 근본 원인 분석을 이어갑니다. 왜 이 상황이 발생했는지, 어떻게 재발을 방지할 수 있는지, 알람 임계값 조정이나 코드 수정이 필요한지 확인하고 포스트모템을 작성합니다."

🔍 Senior-level Follow-up 질문과 모범 답변

Q. "load average 1분 값이 15분 값보다 낮을 때, 어떻게 해석하시나요?"

"1분 < 15분이면 상황이 개선되고 있다는 신호입니다. 즉, 과거에 부하가 높았지만 지금은 줄어들고 있는 상태입니다. 반대로 1분 > 15분이면 부하가 점점 증가하고 있어서, 개입이 필요한 시점이라는 신호입니다. 단순히 숫자의 절대값보다 트렌드를 보는 것이 더 중요합니다."

평가 포인트: load average를 단순 수치가 아닌 트렌드로 읽는 능력

Q. "CPU 사용률이 5%인데 서버가 느립니다. 원인 후보를 어떻게 좁히시겠습니까?"

"CPU가 낮은데 느리다면 CPU가 원인이 아닐 가능성이 높습니다. 먼저 D 상태(Uninterruptible Sleep) 프로세스가 많은지 확인합니다. 이는 디스크 I/O 또는 NFS/원격 파일시스템 대기가 원인인 경우가 많습니다. ps aux | awk '$8 == "D"'로 개수를 확인하고, iostat -xz 1으로 디스크 utilization을 봅니다. 디스크도 정상이라면, 네트워크 병목(TCP 연결 대기, DNS 응답 지연, 외부 API 응답 지연)이나 애플리케이션 내부 락 경합을 의심합니다."

평가 포인트: CPU 사용률 ≠ 전체 성능 지표. 리소스 계층별 점검 능력

Q. "장애 원인 프로세스를 찾았습니다. 바로 kill하시겠습니까?"

"아닙니다. 상황에 따라 다릅니다. 서비스 전체가 마비된 상태라면 데이터 수집보다 복구가 먼저이므로 즉각 재시작이나 롤백을 선택합니다. 하지만 서비스가 느리지만 동작하고 있다면, 먼저 해당 프로세스의 스택 트레이스(gdb, jstack)와 strace -p 출력을 확보합니다. kill하면 그 시점의 상태가 사라지기 때문입니다. 또한 kill 전에 SIGTERM으로 정상 종료를 먼저 시도하고, 응답이 없을 때만 SIGKILL을 씁니다."

평가 포인트: 즉각 조치 vs 원인 데이터 보존의 trade-off 판단

Q. "메모리가 남아 있는데 swap이 사용 중입니다. 문제인가요?"

"상황에 따라 다릅니다. 리눅스 커널은 성능 최적화를 위해 잘 안 쓰는 메모리 영역을 미리 swap으로 이동시키는 경향이 있어서, swap이 사용 중이라고 무조건 문제는 아닙니다. 하지만 vmstat에서 si(swap in)와 so(swap out)가 지속적으로 0이 아니라면, 실시간으로 swap에서 메모리를 불러오고 있다는 뜻이므로 성능에 직접 영향을 줍니다. SSD에서도 swap은 RAM보다 수십 배 느립니다. 이 경우 어떤 프로세스가 메모리를 많이 쓰는지 smem이나 /proc/PID/status의 VmRSS를 확인합니다."

평가 포인트: swap 사용 자체 vs 실시간 swap I/O 발생의 구분

Q. "장애를 해결한 뒤 무엇을 합니까?"

"서비스 복구 확인 후, 장애 중에 수집한 데이터를 정리해서 포스트모템을 작성합니다. 타임라인(장애 발생 → 탐지 → 조치 → 복구), 근본 원인, 영향 범위, 즉각 조치와 장기 대책을 구분해서 기록합니다. 알람이 적절한 시점에 발화했는지, 더 일찍 탐지할 수 있었는지도 검토합니다. 재발 방지를 위한 액션 아이템(모니터링 개선, 코드 수정, 운영 가이드 업데이트)에 담당자와 기한을 명확히 지정합니다."

평가 포인트: 장애 대응이 복구로 끝나지 않고 재발 방지까지 이어지는 사고방식

⚖️ 좋은 답변 vs 약한 답변 분석

약한 답변 (Junior 수준)
"top 실행해서 CPU 높은 프로세스 찾고, 범인 프로세스를 kill -9 하겠습니다. 그래도 안 되면 서버 재시작하겠습니다."
왜 약한 답변인가:
  • 영향 범위 파악이 없습니다. 전체 서비스 장애인지, 일부 기능인지 모른 채 행동합니다.
  • CPU가 문제라고 가정하고 시작합니다. I/O 병목, 메모리 문제일 수 있는데 가설 없이 행동합니다.
  • kill -9는 데이터 수집 없이 원인을 지웁니다. 재발을 막을 수 없습니다.
  • 서버 재시작은 임시방편이고, 원인을 찾지 않으면 반드시 재발합니다.
  • 재발 방지 계획이 없습니다.
좋은 답변 (Senior 수준)
"먼저 영향 범위를 파악합니다. 전체 서비스인지, 최근 배포 이후인지 확인하고, 즉각 롤백이 필요한지 판단합니다. 그 다음 vmstat, top, df로 전체 리소스 상태를 보고 병목 범주를 CPU/I/O/메모리 중 특정합니다. iowait가 높으면 I/O 병목이므로 iostat와 iotop으로 원인 프로세스를 찾습니다. kill하기 전에 스택 트레이스와 로그를 먼저 확보합니다. 복구 후에는 포스트모템을 작성하고 재발 방지 액션을 챙깁니다."
왜 좋은 답변인가:
  • 영향 범위 → 데이터 수집 → 가설 → 검증 → 조치 → 재발 방지의 순서가 있습니다.
  • 가정 없이 데이터를 먼저 봅니다.
  • iowait를 언급함으로써 CPU 수치만이 아니라 I/O 개념을 알고 있음을 보여줍니다.
  • kill 전 데이터 확보로 조치와 원인 분석을 분리합니다.
  • 포스트모템 언급으로 운영 문화를 이해하고 있음을 보여줍니다.

🚫 흔한 오해와 정확한 이해

오해 1: "load average가 1.0 이하면 안전하다"

load average는 CPU 코어 수에 비례해서 해석해야 합니다. 16코어 서버에서 load average 8.0이면 아직 여유가 있습니다. 반면 2코어 서버에서 load average 3.0이면 포화 상태입니다. 절대 수치가 아니라 코어 수 대비 비율로 판단하세요.

오해 2: "CPU 사용률 100%면 반드시 CPU 병목이다"

CPU 100%의 원인이 CPU 연산이 아닐 수 있습니다. wa(iowait)가 높다면 CPU가 디스크 I/O 완료를 기다리느라 100%입니다. us + sy가 높아야 진짜 CPU 연산 병목입니다. top에서 CPU 줄의 각 항목을 반드시 구분해서 봐야 합니다.

오해 3: "메모리 사용량이 높으면 무조건 더 늘려야 한다"

리눅스는 가용 메모리를 최대한 파일 캐시로 활용합니다. free -h의 used가 높아도 available이 충분하면 문제 없습니다. 중요한 수치는 available입니다. 실시간 swap I/O가 발생하거나 OOM Killer가 동작하면 그때 메모리 부족을 의심합니다.

오해 4: "서버 재시작이 가장 빠른 해결책이다"

서버 재시작은 증상을 없애지만 원인을 남깁니다. 재시작 후 같은 조건이 충족되면 다시 장애가 납니다. 더 심각한 문제는, 재시작하면 장애 시점의 로그와 상태가 사라져서 원인 분석이 불가능해질 수 있습니다. 재시작은 복구 방법으로 사용하되, 원인 분석은 반드시 병행해야 합니다.

오해 5: "장애 대응은 혼자서 빠르게 해결하는 것이 능력이다"

Senior 개발자의 장애 대응에서 가장 중요한 것 중 하나는 소통입니다. 관련 팀에 조기에 알리고, 현재 상태를 공유하며, 필요한 지원을 요청하는 것이 혼자 끙끙대는 것보다 훨씬 빠른 복구로 이어집니다. 면접에서 소통과 에스컬레이션을 언급하면 확실한 차별점이 됩니다.

  • 장애 대응의 첫 번째는 영향 범위 파악이다. 명령어보다 먼저.
