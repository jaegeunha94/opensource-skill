커리큘럼 최신성 안내 (2026-07-06 기준 확인) — 이 레슨은 Cloudflare 공식 문서(Cache Rules settings, Edge/Browser Cache TTL, Cache Key customization, default cache behavior, Tiered Cache)와 공식 changelog/블로그를 다시 확인해 작성했다. Day 1~2 이후 새로 확인한 핵심 변화 두 가지: ① Tiered Cache(Argo Smart Routing, Regional Tiered Cache, Cache Reserve, Connection Reuse)는 2026년 기준 "Smart Shield"라는 하나의 origin 보호 상품 우산 아래로 재편됐다 — 예전에는 각각 별도 토글이었다. ② stale-while-revalidate가 2026-02부터 비동기(asynchronous) 방식으로 전환됐다 — 과거에는 캐시 만료 후 첫 요청이 origin 응답을 기다려야 했지만, 지금은 첫 요청도 즉시 stale 콘텐츠를 받고 백그라운드에서 갱신이 일어난다(UPDATING 상태). 대용량/스트리밍 파일에는 "Concurrent Streaming Acceleration"으로 캐시 락 자체를 제거해 origin에서 받아오는 도중에도 여러 클라이언트가 동시에 읽을 수 있다. 이 변화는 "캐시 스탬피드" 답변의 핵심 최신 근거이므로 이 레슨에서 비중 있게 다룬다.

2. 전제 개념 — 쉬운 한국어로

CDN 캐시의 본질 — "origin에게 같은 질문을 두 번 안 하기"

CDN 캐시는 결국 "이미 답을 알고 있는 질문에 origin이 매번 다시 대답하지 않게 하는" 장치다. 방문자가 이미지, CSS, HTML 같은 응답을 요청하면, Cloudflare는 그 응답을 자기 엣지 서버(전 세계 데이터센터, PoP)에 저장해두고 다음 방문자에게는 origin을 거치지 않고 바로 돌려준다. 이게 잘 되면 origin 서버 부하와 지연시간이 동시에 줄어든다. 문제는 "얼마나 오래 저장할지(TTL)", "무엇을 기준으로 같은 응답인지 구분할지(Cache Key)", "저장소를 어떻게 계층화할지(Tiered Cache)", "만료 순간 몰리는 요청을 어떻게 처리할지(스탬피드 방지)"를 잘못 설계하면 캐시가 오히려 장애의 원인이 된다는 점이다.

기본 캐시 동작 — 무엇이 기본으로 캐싱되는가

Cloudflare는 기본적으로 확장자 기반으로 캐시 여부를 정한다. 이미지, CSS, JS, 폰트 같은 정적 파일 확장자는 기본적으로 캐싱되지만, .html이나 API 응답(JSON)은 기본적으로 캐싱되지 않는다 — 로그인 상태, 개인화된 콘텐츠가 섞여 있을 가능성이 높기 때문이다. 또한 origin 응답에 Set-Cookie 헤더가 있으면 그 응답은 기본적으로 캐싱에서 제외된다(개인 세션 정보가 다른 사용자에게 새어나가는 것을 막기 위한 안전장치).

Edge TTL vs Browser TTL — "누구의 캐시를 얼마나 신뢰할 것인가"

이 둘을 혼동하는 게 실무에서 제일 흔한 실수다. Edge TTL은 Cloudflare의 엣지 서버(우리가 아직 통제할 수 있는 인프라)에 응답을 얼마나 오래 저장할지를 정한다. Browser TTL은 방문자의 브라우저(우리가 전혀 통제할 수 없는 클라이언트 로컬 캐시)에 얼마나 오래 저장할지를 정한다.

왜 둘을 다르게 설정해야 하는가

Edge TTL이 만료되면 Cloudflare가 다시 origin에 물어봐서 최신 콘텐츠로 갱신할 수 있다. 하지만 Browser TTL이 만료되기 전에는, Cloudflare가 아무리 최신 콘텐츠를 갖고 있어도 그 사용자의 브라우저는 자기 로컬 캐시에 있는 옛날 응답을 계속 쓴다 — 강제로 갱신시킬 방법이 없다(캐시 무효화를 브라우저에 원격으로 지시할 수 없다). 그래서 자주 바뀔 수 있는 리소스는 Browser TTL을 짧게(또는 0에 가깝게), Edge TTL은 상대적으로 길게 잡아서 "origin 부하는 줄이되, 사용자에게 옛날 콘텐츠가 오래 고정되는 리스크는 낮추는" 조합을 쓴다. 반대로 버전 해시가 파일명에 박혀 있는 정적 자산(예: app.a1b2c3.js)은 내용이 바뀌면 파일명 자체가 바뀌므로 Browser TTL을 아주 길게 잡아도 안전하다.

Cache Rules — Edge TTL을 결정하는 3가지 모드

Cache Rules에서 Edge TTL을 설정하는 방식은 세 갈래로 나뉜다. 이 선택이 곧 "누구를 신뢰의 기준으로 삼을지"를 정하는 결정이다.

1. Cache-Control 있으면 따르고, 없으면 캐시하지 않음(bypass)
origin이 명시적으로 캐시 지침을 준 경우에만 캐싱한다. 가장 보수적 — origin 개발자가 캐시 정책의 최종 결정권을 갖는다.
2. Cache-Control 있으면 따르고, 없으면 Cloudflare 기본 동작 사용
origin이 지침을 안 주면 Cloudflare의 기본 캐시 규칙(확장자 기반)으로 대체한다. 실무에서 가장 널리 쓰이는 절충안.
3. Cache-Control을 무시하고 지정한 TTL을 강제
origin이 뭐라고 하든 Cloudflare가 정한 시간만큼 캐싱한다. origin 코드를 못 건드리는 legacy 시스템이나, "이 경로는 무조건 이 정책"을 강제해야 하는 상황에 쓴다. 잘못 쓰면 origin이 의도한 캐시 무효화 신호를 무시하게 되는 위험이 있다.

Cache Key 커스터마이징 — "무엇을 기준으로 같은 응답이라고 볼 것인가"

기본 Cache Key는 host + path + query string이다. 즉 쿼리 스트링이 한 글자만 달라도 Cloudflare는 다른 응답으로 취급해서 별도로 캐싱한다. 이게 문제가 되는 대표 사례가 ?utm_source=, ?v= 같은 캐시 버스터/트래킹 파라미터다 — 실제 콘텐츠는 완전히 동일한데, 파라미터 조합 수만큼 캐시 항목이 쪼개져서 히트율이 급락한다.

커스터마이징 방향 두 가지

① 쿼리 스트링을 무시하거나 특정 파라미터만 포함: 정적 자산처럼 쿼리 스트링이 콘텐츠에 영향을 주지 않으면 아예 무시하고, id나 lang처럼 실제로 다른 콘텐츠를 만드는 파라미터만 선택적으로 포함한다.

② 특정 헤더/쿠키를 Cache Key에 포함: Accept-Language를 포함하면 언어별로 다른 HTML을 올바르게 구분해서 캐싱할 수 있고, CF-Device-Type을 포함하면 모바일/데스크톱 버전을 구분할 수 있다. 쿠키를 Cache Key에 포함하는 건 드물게 쓰이는데, 대부분의 경우 "쿠키 기반으로 캐시를 나눈다"보다는 "그 요청 자체를 캐시에서 제외(bypass)한다"는 신호로 쓰는 게 더 안전하다.

Cache Key 설계 실수가 만드는 두 가지 사고

① 히트율 붕괴: 트래킹 파라미터를 Cache Key에서 빼지 않으면 사실상 캐싱이 거의 작동하지 않는 것과 같은 효과가 난다. ② 캐시 오염(cache poisoning)/개인정보 노출: 반대로 사용자를 구분해야 하는 신호(예: 로그인 세션을 나타내는 헤더나 쿠키)를 Cache Key에서 빠뜨리면, A라는 사용자에게 만들어진 개인화 응답이 캐싱된 뒤 B라는 사용자에게 그대로 재사용되는 심각한 사고로 이어진다. Cache Key 설계는 "히트율을 올리는 작업"이자 동시에 "누구의 응답과 누구의 응답을 절대 섞으면 안 되는지를 정의하는 보안 경계 작업"이다.

Tiered Cache — 엣지를 계층화해서 origin 요청 자체를 줄이기

Cloudflare PoP은 전 세계 수백 곳에 있다. Tiered Cache가 없으면, 캐시가 만료될 때마다 각 PoP이 각자 독립적으로 origin에 재검증 요청을 보낸다 — PoP 수만큼 origin이 두들겨 맞는 구조다. Tiered Cache는 PoP을 lower-tier(방문자와 가까운 엣지)와 upper-tier(origin과 가까운 상위 계층)로 나눠서, upper-tier만 origin에 직접 요청하도록 강제한다. lower-tier는 캐시가 없으면 origin이 아니라 upper-tier에게 먼저 물어본다.

Smart Tiered Cache — 그리고 2026년의 "Smart Shield" 재편

Smart Tiered Cache는 지연시간 데이터를 기반으로 origin마다 가장 가까운 upper-tier를 자동으로 선택한다(수동 토폴로지 설정 불필요). 클라우드에 호스팅된 origin처럼 anycast/리전 네트워킹 때문에 지연시간만으로 위치 추정이 어려운 경우, cloud region hint를 지정해 정확도를 높일 수 있다. 2026년 기준 공식 문서 구조가 바뀌어, Smart Tiered Cache·Regional Tiered Cache·Cache Reserve·Connection Reuse·Argo Smart Routing이 "Smart Shield"라는 하나의 origin 보호 상품군으로 재편됐다. 개념 자체(계층화로 origin 요청을 줄인다)는 그대로지만, 인터뷰에서 "Tiered Cache가 정확히 어떤 상품 이름 아래 있느냐"를 물으면 이 재편을 아는지로 최신성을 가늠할 수 있다.

캐시 스탬피드(cache stampede) — 그리고 2026년의 핵심 변화

캐시 스탬피드(thundering herd)는 인기 있는 캐시 항목이 만료되는 바로 그 순간, 동시에 도착한 수많은 요청이 전부 "캐시 미스"로 판정돼 origin으로 그대로 몰리는 현상이다. 평소엔 캐시가 origin을 보호하다가, 만료 순간에는 오히려 "동시에 재검증하려는 요청들"이 origin에 순간적인 과부하를 만든다.

Cloudflare의 기본 방어 — request collapsing(cache lock)

Cloudflare 엣지는 기본적으로 같은 자산에 대해 동시에 여러 요청이 캐시 미스/만료를 겪으면, 그중 딱 하나만 origin으로 실제 요청을 보내고 나머지는 그 응답을 기다렸다가 함께 받는다(요청 병합, per-asset cache lock). origin 입장에서는 수천 개의 동시 요청이 아니라 요청 1건만 보이므로, 스탬피드가 origin까지 전파되지 않는다.

2026-02 변화 — stale-while-revalidate의 비동기 전환

기존에는 캐시가 만료된 뒤 "가장 먼저 도착한 요청"이 origin 응답이 돌아올 때까지 기다려야 했다(그 방문자는 REVALIDATED/EXPIRED 상태를 최종적으로 받았다). 즉 request collapsing이 origin 부하는 막아줬지만, 그 대표 요청을 보낸 방문자 한 명은 여전히 origin 지연시간을 그대로 체감했다. 2026-02부터 Cloudflare는 이 과정을 완전히 비동기로 바꿨다. 만료 직후 도착한 요청은 origin을 기다리지 않고 즉시 기존 stale 콘텐츠를 UPDATING 상태로 받고, 갱신은 백그라운드에서 별도로 진행된다. 결과적으로 모든 방문자가 항상 캐시에서(즉시) 응답을 받고, origin 지연이나 일시적 오류가 방문자 체감 지연으로 전혀 전파되지 않는다.

대용량/스트리밍 파일 특수 케이스 — Concurrent Streaming Acceleration

라이브 스트리밍이나 대용량 파일처럼 "응답을 origin에서 아직 받는 중"인 상태가 오래 지속되는 경우, 과거의 cache lock 방식은 그 응답이 완전히 끝날 때까지 다른 클라이언트를 대기시켰다(실측 사례: OTT 고객의 캐시 락 대기시간이 P99 기준 1.5초). Cloudflare는 이 케이스를 위해 캐시 락 자체를 제거하고, origin에서 파일을 받아오는 도중에도 여러 클라이언트가 이미 받은 부분부터 동시에 읽을 수 있게 만들었다(단일 origin fetch + 다중 동시 read). 스탬피드 방지의 목표(origin에는 요청 1건만 가도록)는 유지하면서, 방문자 체감 대기시간을 사실상 제거한 것이다.

Cache Rules와 레거시 Page Rules의 경계 (Day 1/2 복습 + 캐시 세부 매핑)

Day 1~2에서 다뤘듯 Page Rules는 공식적으로 deprecated고 신규 생성이 전 플랜에서 차단됐다. 캐시 관점에서 구체적으로 매핑하면: 과거 Page Rules의 Cache Level, Edge Cache TTL, Browser Cache TTL, Bypass Cache on Cookie 설정은 각각 Cache Rules의 Cache Eligibility, Edge TTL, Browser TTL, Cache Key(쿠키 조건)/Bypass 액션으로 대체된다. 가장 중요한 실무 차이는 평가 방식이다 — Page Rules는 URL 패턴 매치 시 "가장 먼저 매치된 규칙"이 우선했지만, Cache Rules(다른 Rules 제품군과 동일)는 목록 순서대로 평가하되 나중에 실행된 규칙이 같은 필드를 덮어쓸 수 있는 방식이라, 규칙 순서를 Page Rules 시절의 직관대로 짜면 의도와 다르게 동작할 수 있다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 캐시를 "히트율 지표" 하나로만 보지 않는가: 히트율을 올리는 선택(쿼리 스트링 무시, 긴 TTL)이 신선도·개인정보 격리와 어떻게 충돌하는지 균형을 잡을 수 있는가
  • 캐시 오염/개인정보 노출을 보안 경계 문제로 인식하는가: Cache Key 설계 실수가 곧 "다른 사용자의 데이터가 섞이는 사고"로 이어질 수 있음을 아는가
  • 대규모 트래픽 하의 실패 모드를 예측하는가: 캐시가 만료되는 "그 순간"에 무슨 일이 벌어지는지(스탬피드) 구조적으로 설명할 수 있는가
  • 최신 운영 표준을 따라가고 있는가: Cloudflare가 스탬피드 대응을 어떻게 발전시켜왔는지(request collapsing → 비동기 SWR → 캐시 락 제거) 아는 사람은 실제로 최신 문서/changelog를 챙겨보는 습관이 있다는 신호다

이 질문은 "TTL을 설정할 줄 아는가"를 넘어, "캐시가 성능 최적화 도구인 동시에 보안 경계이자 장애 원인이 될 수 있다는 걸 아는가"를 검증한다. 대규모 트래픽 서비스를 실제로 운영해본 사람만 스탬피드와 캐시 오염을 자연스럽게 같이 언급한다.

4. 실무 연결

시나리오 1 — 개인화 응답이 다른 사용자에게 노출된 캐시 오염 사고

마이페이지 API가 사용자별로 다른 데이터를 응답하는데, 이 응답 헤더에 Set-Cookie가 없어서 기본 캐시 예외 대상이 아니었고, 팀이 성능 개선을 위해 이 경로에 Cache Rules로 강제 캐싱을 걸었다. 문제는 사용자 식별이 URL이나 쿼리 스트링이 아니라 인증 헤더(Authorization)로만 이뤄졌는데, Cache Key에는 이 헤더를 포함하지 않았다는 점이다. 결과적으로 첫 번째 사용자의 응답이 캐싱된 뒤, 같은 URL로 요청한 다른 사용자들이 첫 번째 사용자의 개인 데이터를 그대로 받는 사고가 발생했다.

이 사례가 중요한 이유: "캐싱해도 되는 경로인가"를 URL 패턴만으로 판단하면 안 되고, 그 응답이 요청자에 따라 달라지는 근거(헤더/쿠키/세션)가 Cache Key에 반영돼 있는지까지 확인해야 한다.

시나리오 2 — 이벤트 시작 순간 캐시 스탬피드로 origin 다운

선착순 이벤트 페이지의 캐시 TTL을 60초로 짧게 잡아 "최신성"을 유지하려 했는데, 이벤트 시작 정각에 수만 명이 동시 접속했다. 마침 그 순간이 60초 TTL의 만료 시점과 겹치면서, request collapsing이 없었다면(또는 이 기능이 적용되기 전 구성이었다면) 수만 건의 요청이 동시에 캐시 미스로 판정돼 origin에 그대로 몰릴 뻔한 상황이었다. 사후 분석에서 팀은 "TTL을 짧게 잡을수록 신선하다"는 생각만 했지, 다수 요청이 동시에 만료 순간과 겹칠 확률은 고려하지 않았다는 걸 확인했다.

이 사례가 중요한 이유: TTL은 "신선도"만의 문제가 아니라 "만료 시점에 트래픽이 몰릴 가능성"까지 함께 설계해야 하는 값이다. 트래픽 급증이 예정된 이벤트라면 TTL을 이벤트 시작 시점과 정렬되지 않도록 지터(jitter)를 주거나, 사전 워밍(pre-warming)으로 만료 자체를 피하는 전략이 필요하다.

시나리오 3 — 쿼리 스트링 트래킹 파라미터로 인한 히트율 붕괴

마케팅팀이 모든 랜딩 페이지 링크에 utm_source, utm_campaign 등 트래킹 파라미터를 붙이기 시작한 뒤, 같은 콘텐츠임에도 캐시 히트율이 급격히 떨어지고 origin 트래픽이 눈에 띄게 늘었다. 원인은 기본 Cache Key가 쿼리 스트링 전체를 포함해서, 파라미터 조합 수만큼 캐시 항목이 쪼개진 것이었다. Cache Key에서 트래킹 파라미터를 제외하도록 Cache Rules를 조정한 뒤 히트율이 정상으로 돌아왔다.

이 사례가 중요한 이유: 캐시 성능 저하의 원인이 "인프라 문제"가 아니라 "다른 팀이 붙인 쿼리 파라미터"인 경우가 실무에서 매우 흔하다. Cache Key 설계는 마케팅/프론트엔드 팀의 URL 관행까지 감안해야 한다.

시나리오 4 — Page Rules → Cache Rules 마이그레이션 중 규칙 순서 오해

팀이 기존 Page Rules 여러 개를 Cache Rules로 전환하면서, "먼저 매치되는 규칙이 이긴다"는 Page Rules 시절의 습관대로 규칙을 나열했다. 하지만 Cache Rules는 목록을 순서대로 평가하며 이후 규칙이 같은 필드를 덮어쓸 수 있는 방식이라, 의도했던 것과 반대로 마지막 규칙의 설정이 최종 반영되면서 특정 경로의 캐시 정책이 예상과 다르게 동작했다.

이 사례가 중요한 이유: 마이그레이션은 "같은 설정을 새 UI에 옮기는 작업"이 아니라 "평가 모델 자체가 다르다는 것을 이해하고 순서를 재설계하는 작업"이다.

5. 답변 사고 순서

"대규모 트래픽 서비스의 캐시를 어떻게 설계하시겠습니까"라는 질문에는 아래 순서로 답한다.

  1. 캐시 대상과 신선도 요구사항 정의

    어떤 응답이 사용자별로 달라지는지(개인화 여부)부터 구분한다. 정적 자산/공용 콘텐츠와 개인화 응답을 먼저 분리하지 않으면 이후 모든 설계가 흔들린다.

  2. Edge TTL / Browser TTL을 분리해서 설계

    origin 부하 감소는 Edge TTL로, 클라이언트 체감 최신성은 Browser TTL로 각각 목표를 정한다. 버전 해시가 있는 정적 자산과 그렇지 않은 자산을 다르게 취급한다.

  3. Cache Key를 "히트율"과 "격리 경계" 두 기준으로 설계

    트래킹 파라미터는 제외해 히트율을 지키고, 언어/디바이스처럼 실제 콘텐츠를 가르는 신호는 포함한다. 동시에 사용자를 구분해야 하는 인증 신호가 캐시 대상 응답에 존재하는지 반드시 확인한다.

  4. Tiered Cache로 origin 요청 자체를 줄인다

    Smart Tiered Cache(현재 Smart Shield 산하)를 활성화해 PoP별 개별 origin 요청을 상위 계층으로 집약한다. origin이 특정 클라우드 리전에 있다면 cloud region hint로 정확도를 높인다.

  5. 만료 순간의 동시성(스탬피드)까지 설계에 포함

    request collapsing과 비동기 stale-while-revalidate가 기본으로 동작한다는 걸 인지하고, 트래픽 급증이 예정된 이벤트성 콘텐츠는 TTL 만료 시점에 지터를 주거나 사전 워밍으로 "동시 만료"를 피한다.

  6. 레거시 Page Rules와의 경계 확인(마이그레이션 상황이라면)

    Cache Level/Edge TTL/Browser TTL/Bypass on Cookie가 Cache Rules의 어떤 필드로 매핑되는지 확인하고, 평가 순서 모델이 다르다는 것을 감안해 규칙 순서를 재설계한다.

핵심 팁

이 순서에서 "개인화 여부를 먼저 나눴는가"와 "만료 순간의 동시성을 고려했는가" 두 가지를 스스로 먼저 언급하는 것이 시니어 답변의 핵심이다. 둘 다 놓치면 각각 개인정보 사고와 장애로 직결된다.

6. 핵심 답변 예시

면접 답변 예시 — 대규모 트래픽 캐시 설계

"가장 먼저 하는 건 응답을 개인화 여부로 나누는 겁니다. 정적 자산과 공용 콘텐츠는 적극적으로 캐싱 대상으로 삼고, 사용자별로 달라지는 응답은 원칙적으로 캐싱하지 않거나, 캐싱하더라도 Cache Key에 사용자를 구분하는 헤더나 쿠키를 반드시 포함시킵니다. Cache Key 설계 실수는 단순한 성능 문제가 아니라 다른 사용자의 데이터가 섞여 노출되는 보안 사고로 이어질 수 있다고 생각하기 때문입니다.

TTL은 Edge TTL과 Browser TTL을 분리해서 설계합니다. origin 부하를 줄이는 건 Edge TTL의 역할이고, 클라이언트가 옛 콘텐츠를 얼마나 오래 들고 있을지는 Browser TTL의 역할이라, 버전 해시가 없는 자원은 Browser TTL을 짧게 잡아 강제 갱신이 안 된다는 리스크를 관리합니다.

Cache Key는 쿼리 스트링 전체를 그대로 두지 않고, 트래킹 파라미터처럼 콘텐츠에 영향을 주지 않는 값은 제외해서 히트율을 지킵니다. 지리적으로 흩어진 origin이라면 Tiered Cache(현재는 Smart Shield 산하의 Smart Tiered Cache)로 각 PoP의 개별 origin 요청을 상위 계층으로 모아 origin 요청 수 자체를 줄입니다.

마지막으로 캐시 만료 순간의 동시성을 별도로 고려합니다. Cloudflare는 기본적으로 요청 병합(request collapsing)으로 동시 요청을 origin 요청 1건으로 줄이고, 2026년부터는 stale-while-revalidate가 비동기로 동작해서 만료 직후 도착한 요청도 origin을 기다리지 않고 즉시 기존 콘텐츠를 받습니다. 다만 이벤트처럼 트래픽 급증 시점이 예측 가능한 경우에는, TTL 만료 시점에 지터를 주거나 사전 워밍을 걸어서 애초에 대량 동시 만료 자체를 피하는 전략을 추가로 씁니다."

이 답변이 좋은 이유: ① 개인화 여부 구분을 캐시 설계의 첫 단계로 제시 ② Edge/Browser TTL을 목적에 따라 분리 ③ Cache Key를 히트율과 보안 경계 양쪽 관점에서 설명 ④ 스탬피드 대응의 최신 변화(비동기 SWR)까지 인지 ⑤ "기본 방어 + 예측 가능한 상황에 대한 추가 전략"을 구분해서 설명

7. Senior-level Follow-up Q&A

Q1. "Cache-Control 헤더를 origin이 이미 잘 내려주고 있다면, Cache Rules로 TTL을 따로 강제할 필요가 있나요?"

origin 코드를 직접 통제할 수 있고 Cache-Control이 신뢰할 만하다면, 첫 번째 모드("Cache-Control 있으면 따르고 없으면 bypass")로 두고 origin에 결정권을 주는 게 원칙적으로 더 안전하다. Cache Rules로 TTL을 강제하는 건 origin을 못 바꾸는 legacy 시스템, 서드파티 오리진, 혹은 "이 경로만큼은 origin 설정과 무관하게 반드시 이 정책"이라는 조직적 요구가 있을 때로 한정하는 게 맞다. 강제 TTL은 origin이 나중에 내리는 캐시 무효화 신호(예: 콘텐츠 삭제 시 즉시 반영)를 무시하게 만들 수 있다는 트레이드오프를 감안해야 한다.

Q2. "비동기 stale-while-revalidate 덕분에 request collapsing이 이제 필요 없어진 건가요?"

아니다. 둘은 다른 문제를 푼다. request collapsing은 "origin에 몇 건의 요청이 가는가"를 줄이는 메커니즘이고, 비동기 SWR은 "그 대표 요청을 만든 방문자가 응답을 얼마나 기다리는가"를 없애는 메커니즘이다. 두 개가 같이 있어야 "origin 부하도 낮고, 방문자 체감 지연도 없는" 상태가 된다. 비동기 SWR이 없던 시절에는 request collapsing이 origin은 지켰지만, 대표 요청을 만든 방문자 한 명은 여전히 origin 지연을 그대로 느꼈다는 점을 구분해서 설명할 수 있어야 한다.

Q3. "Cache Key에 언어 헤더(Accept-Language)를 포함시켰더니 캐시 항목 수가 너무 많아졌습니다. 어떻게 대응하나요?"

Accept-Language 헤더는 값의 조합이 매우 다양해서(브라우저마다 미묘하게 다른 우선순위 문자열 등) 원본 그대로 Cache Key에 넣으면 쿼리 스트링 문제와 똑같은 히트율 붕괴가 난다. 실무 대응은 origin/Workers 단에서 Accept-Language를 지원하는 언어 목록(예: ko, en, ja)으로 정규화한 뒤, 그 정규화된 값을 별도 헤더(예: CF-Normalized-Lang)로 만들어 Cache Key에는 그 정규화 헤더만 포함시키는 방식이다. "원본 신호 그대로 Cache Key에 넣지 않고, 콘텐츠를 실제로 가르는 최소 단위로 정규화한 뒤 넣는다"는 원칙은 언어 외의 다른 헤더에도 동일하게 적용된다.

Q4. "Tiered Cache를 켜면 무조건 좋은 건가요? 단점은 없나요?"

대부분의 상황에서 origin 요청 감소 효과가 크지만, upper-tier를 한 단계 거치는 구조라 캐시 미스가 발생하는 첫 요청의 지연시간이 이론적으로 소폭 늘어날 수 있다(lower-tier가 origin에 직접 가는 대신 upper-tier를 거쳐야 하므로). 대부분의 서비스는 이 트레이드오프가 origin 부하 감소 이득보다 훨씬 작아서 문제가 되지 않지만, "왜 무조건 켜는 게 이득인지"를 origin 요청 수 감소와 왕복 지연 증가라는 두 축으로 설명할 수 있어야 시니어 답변이다.

Q5. "이벤트 시작 시점에 트래픽이 몰릴 걸 미리 아는 상황이라면, TTL을 어떻게 설계해야 하나요?"

두 가지를 함께 쓴다. 첫째, 이벤트 시작 몇 분 전에 캐시를 미리 워밍(prefetch/조회를 미리 발생시켜 캐시를 채워둠)해서, 실제 몰리는 트래픽이 도착했을 때는 이미 캐시가 채워진 상태로 맞이한다. 둘째, TTL을 이벤트 시작 시각과 정확히 겹치지 않도록 설계한다(예: 정각이 아니라 이벤트 시작 몇 분 전에 이미 만료되도록 역산). request collapsing과 비동기 SWR이 있다고 해서 "아무 대비 없이 TTL만 짧게 잡아도 된다"고 생각하는 건 안전장치를 과신하는 것이다 — 예측 가능한 트래픽 급증은 애초에 스탬피드 상황 자체를 피하도록 설계하는 게 우선이다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 캐시 설계를 시작하기 전에 응답의 개인화 여부부터 구분한다
  • Edge TTL과 Browser TTL을 서로 다른 목적(origin 부하 vs 클라이언트 신선도)으로 분리해 설명한다
  • Cache Key를 히트율과 보안 격리 경계 양쪽 관점에서 함께 설계한다
  • 캐시 스탬피드의 기본 방어(request collapsing)와 2026년 비동기 SWR 변화를 구분해서 설명한다
  • 예측 가능한 트래픽 급증에는 사전 워밍/TTL 지터 같은 능동적 대비를 추가로 언급한다
약한 답변
  • "TTL을 늘리면 캐시 효율이 좋아진다" 정도로만 답한다
  • Cache Key를 히트율 관점으로만 보고 개인정보 격리 위험은 언급하지 않는다
  • 캐시 스탬피드를 아예 모르거나, "TTL을 짧게 하면 된다"고 반대로 답한다
  • Tiered Cache를 "그냥 켜는 옵션" 정도로만 알고 원리를 설명하지 못한다
  • Page Rules와 Cache Rules를 "이름만 바뀐 같은 기능"이라고 단순화한다
왜 약한 답변이 약한가

캐시는 "빠르게 만드는 기능"이 아니라 "무엇을 얼마나 오래, 누구의 기준으로, 누구와 격리해서 저장할 것인가를 정하는 설계"다. 이 관점이 없으면 히트율은 오르지만 개인정보가 새거나, 평소엔 멀쩡하다가 트래픽이 몰리는 정확히 그 순간에만 장애가 나는 패턴을 만든다. 시니어 면접에서는 "캐시를 켤 줄 아는가"가 아니라 "캐시가 깨지는 정확한 순간과 이유를 예측할 수 있는가"를 본다.

9. 흔한 오해

오해 1

"TTL은 길게 잡을수록 무조건 성능에 좋다."

실제

Edge TTL을 길게 잡으면 origin 부하는 줄지만 콘텐츠 최신성이 떨어지고, Browser TTL을 길게 잡으면 origin/Edge가 아무리 최신 콘텐츠를 가져도 클라이언트에 강제로 갱신시킬 방법이 없어 사용자 체감 최신성이 훨씬 나빠질 수 있다. TTL은 "길수록 좋다"가 아니라 콘텐츠의 변경 빈도와 원하는 최신성 수준에 맞춰 정하는 값이다.

오해 2

"캐시 스탬피드는 TTL을 짧게 잡아서 자주 갱신하면 생기지 않는다."

실제

정반대다. TTL을 짧게 잡을수록 만료가 자주 일어나고, 그 만료 순간에 다수 요청이 겹칠 기회 자체가 늘어난다. 스탬피드는 "얼마나 자주 만료되는가"가 아니라 "만료되는 순간에 동시 요청이 몰리는가"의 문제이며, 방어책은 TTL 조정이 아니라 request collapsing·비동기 SWR·사전 워밍 같은 별도 메커니즘이다.

오해 3

"Cache Key는 히트율을 높이기 위한 성능 설정일 뿐이다."

실제

Cache Key는 동시에 "어떤 응답과 어떤 응답을 절대 섞으면 안 되는가"를 정의하는 격리 경계다. 사용자를 구분하는 신호(인증 헤더, 세션 쿠키)가 응답 내용에 영향을 주는데 그 신호가 Cache Key에 빠져 있으면, 최적화를 시도하다가 개인정보가 다른 사용자에게 노출되는 사고로 이어질 수 있다.

오해 4

"stale-while-revalidate는 오래된 콘텐츠를 계속 보여주는 위험한 기능이다."

실제

stale-while-revalidate는 "잠깐 오래된 콘텐츠를 보여주는 대신, 모든 방문자가 즉시 응답을 받고 origin 부하와 지연 전파를 막는" 의도적인 트레이드오프다. 특히 2026년 비동기 전환 이후로는 이 트레이드오프가 훨씬 유리해졌다 — 방문자는 지연 없이 응답을 받고, 갱신은 보이지 않는 곳에서 짧은 시간 안에 끝난다. 콘텐츠가 초 단위로 정확해야 하는 극히 일부 사례(예: 실시간 재고 수량)를 제외하면 대부분의 서비스에 이득이 더 크다.

오해 5

"Page Rules에서 쓰던 캐시 설정을 그대로 Cache Rules로 옮기기만 하면 마이그레이션이 끝난다."

실제

설정값을 그대로 옮겨도 평가 모델 자체가 다르다 — Page Rules는 첫 매치 우선, Cache Rules는 순서대로 평가하며 나중 규칙이 같은 필드를 덮어쓸 수 있다. 값만 옮기고 순서를 재검토하지 않으면 의도와 다른 캐시 정책이 조용히 적용될 수 있다.
