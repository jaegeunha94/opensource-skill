WAF와 Rate Limiting 설계

커리큘럼 최신성 안내 (2026-07-08 기준 확인) — 이 레슨은 Cloudflare 공식 문서(WAF 개요, Custom Rules, Rate Limiting Rules/parameters/best-practices, Managed Rules/troubleshooting, DDoS override 문서)와 2026년 WAF changelog(2026-03-23, 2026-04-27, 2026-05-04/11/20, 2026-06-23)를 재확인해 작성했다. Day 4까지의 커리큘럼과 충돌하는 내용은 없었고, 다음 다섯 가지를 이 레슨의 핵심 최신 근거로 반영했다: ① 보안 규칙 평가 순서는 HTTP DDoS 공격 방어(ddos_l7) → Custom Rules(http_request_firewall_custom) → Rate Limiting Rules(http_ratelimit) → Managed Rules(http_request_firewall_managed) 순이며, Managed Rules가 가장 나중에 평가된다. ② Cloudflare 자신도 Managed Rules 신규/변경 규칙을 7일 릴리스 사이클(1주차 Log 전용 → 2주차 기본 액션 전환)로 점진 배포하며, 긴급 CVE 대응 시에만 이 사이클을 건너뛴다. ③ 오탐 대응은 규칙 전체 비활성화가 아니라 Overrides(액션/민감도 조정)와 Exceptions(특정 조건의 요청을 룰셋 평가 자체에서 제외, Overrides보다 우선순위 높음)로 좁게 처리하는 것이 공식 권장이다. ④ Rate Limiting은 IP 외에도 IP with NAT support(_cfuvid 쿠키 기반), JA3/JA4 TLS 핑거프린트, 헤더/쿠키 값을 카운팅 기준(characteristic)으로 조합할 수 있고, 로그인/결제처럼 민감한 엔드포인트는 IP 단독 대신 이런 조합을 쓰라고 공식 문서가 권고한다. ⑤ OWASP Core Ruleset은 폐기되지 않았지만, 공식 문서가 "Cloudflare Managed Ruleset과 WAF Attack Score 위에 추가했을 때 오탐이 잦고 이득은 제한적"이라고 명시해, Cloudflare Managed Ruleset을 1차 방어선으로 삼는 게 현재 공식 권장이다.

2. 전제 개념 — 쉬운 한국어로

WAF와 Rate Limiting은 서로 다른 질문에 답한다

WAF(Web Application Firewall)는 요청의 "내용"을 본다 — URL, 쿼리스트링, 바디, 헤더에 SQL Injection이나 XSS 같은 공격 패턴이 들어있는지 시그니처/규칙으로 검사한다. Rate Limiting은 요청의 "빈도"를 본다 — 내용이 완전히 정상인 요청이라도, 같은 클라이언트가 짧은 시간에 너무 많이 보내면 막는다. 로그인 브루트포스나 API 크롤링처럼 "내용은 멀쩡한데 양이 문제"인 공격은 WAF 시그니처로는 절대 못 잡고, 오직 Rate Limiting만 잡을 수 있다. 이 둘을 "보안 강화 옵션 두 개" 정도로 뭉뚱그리면 안 되는 이유가 여기 있다.

Custom Rules와 Managed Rules — "내가 정의한 규칙"과 "Cloudflare가 유지관리하는 규칙"

Custom Rules는 내가 직접 조건(expression)과 액션을 정의하는 IF-THEN 규칙이다. Managed Rules(Cloudflare Managed Ruleset, OWASP Core Ruleset 등)는 Cloudflare가 전 세계 트래픽에서 관찰한 공격 패턴을 바탕으로 유지·갱신하는 시그니처 세트다. 공식 문서는 OWASP Core Ruleset이 "Cloudflare Managed Ruleset과 WAF Attack Score 위에 추가했을 때 오탐이 잦고 이득은 제한적"이라고 명시한다 — 즉 Cloudflare Managed Ruleset을 1차 방어선으로 삼고, OWASP CRS는 규제·계약상 요구가 있을 때만 보조적으로 검토하는 것이 현재 공식 권장에 가깝다.

평가 순서 — 왜 이 순서를 모르면 규칙을 설계할 수 없는가

Cloudflare는 하나의 요청을 고정된 순서의 phase로 통과시킨다. 보안 관련 phase만 추리면 다음과 같다.

① HTTP DDoS 공격 방어 (ddos_l7)
상시 활성화된 L7 DDoS 방어가 가장 먼저 평가된다. 여기서 걸리면 아래 단계는 아예 실행되지 않는다.
② Custom Rules (http_request_firewall_custom)
내가 만든 Custom Rules가 위에서 아래로 평가된다. Block/Challenge 같은 terminal 액션을 만나면 그 즉시 요청 평가가 끝난다.
③ Rate Limiting Rules (http_ratelimit)
Custom Rules를 통과한 요청만 Rate Limiting 카운팅/판정 대상이 된다.
④ Managed Rules (http_request_firewall_managed)
Cloudflare Managed Ruleset/OWASP CRS는 가장 마지막에 평가된다. Custom Rules나 Rate Limiting에서 이미 terminal 액션이 실행됐다면 Managed Rules는 아예 그 요청을 보지 않는다.
이 순서가 왜 실무적으로 중요한가

Custom Rules가 Managed Rules보다 먼저 평가되기 때문에, Custom Rules의 Skip 액션으로 "이 트래픽은 Rate Limiting과 Managed Rules를 건너뛰어라"라고 지정할 수 있다 — 즉 알려진 정상 트래픽(예: 사내 모니터링 봇, 파트너 웹훅 발신 IP)을 Managed Rules 오탐으로부터 미리 보호하는 exemption 레이어를 Custom Rules 단계에서 만들 수 있다는 뜻이다. 이 순서를 모르면 "Managed Rules가 자꾸 우리 API를 차단하는데 왜 Custom Rule로 화이트리스트를 만들어도 소용없지?"라는 질문에 스스로 답하지 못한다(액션을 Block이 아니라 Skip으로 잘못 만들었거나, 반대로 순서를 착각해 해결책의 방향 자체를 잘못 잡는 경우가 흔하다).

7일 릴리스 사이클 — Cloudflare도 자기 규칙을 "일단 Log로" 배포한다

Managed Rules의 신규/변경 규칙은 보통 7일 주기(월/화 배포)로 나간다. 1주차에는 Log 액션으로만 배포돼 실제 트래픽에 매칭은 되지만 차단은 하지 않는다 — 이 기간 동안 Security Events에서 오탐 여부를 관찰할 수 있다. 2주차가 되면 원래 의도한 액션(대개 Block)으로 전환된다. 단, 새로운 CVE처럼 긴급한 경우는 이 사이클을 건너뛰고 즉시 배포한다.

인터뷰에서 중요한 이유

이 사실을 아는 것 자체보다 "세계에서 가장 많은 트래픽을 검사하는 회사조차 자기 규칙을 Log 모드로 먼저 관찰한 뒤 차단으로 전환한다"는 원칙을 내 팀의 Custom Rule/Rate Limiting 배포에도 그대로 적용해야 한다는 결론을 낼 수 있는지가 핵심이다. "우리 팀은 바빠서 바로 Block으로 배포했다"는 답은 이 원칙을 거스르는 것으로 들린다.

오탐 대응 — Overrides와 Exceptions는 다른 도구다

Managed Rules가 오탐을 내면 "규칙을 꺼버린다"가 첫 반응이 되기 쉽지만, 이는 그 규칙이 막던 진짜 공격까지 통과시킨다. 공식 문서는 두 가지 정밀 도구를 제시한다.

Overrides — 규칙(또는 태그)의 액션/민감도를 바꾼다

특정 규칙 하나 또는 태그로 묶인 규칙 그룹의 액션을 Block → Log로 낮추거나, DDoS 관련 규칙이라면 민감도(Sensitivity Level)를 Medium → Low → Essentially Off 순으로 단계적으로 낮출 수 있다. 전체 규칙을 끄는 게 아니라 "이 규칙군의 공격적인 정도"를 조절하는 방식이다.

Exceptions — 특정 조건의 요청을 룰셋 평가 자체에서 제외한다

"이 URL 경로 + 이 헤더 조건에 해당하는 요청은 이 룰셋(또는 특정 규칙)의 평가를 아예 건너뛴다"고 지정한다. Exceptions는 Overrides보다 우선순위가 높다 — 즉 Override로 규칙 액션을 Block으로 설정해 놓았어도, 그 요청이 Exception 조건에 해당하면 애초에 그 규칙의 평가 대상에서 빠진다.

실무 판단 기준

오탐이 특정 엔드포인트/요청 패턴에 국한된다면 Exception으로 좁게 예외 처리하고, 오탐이 이 규칙 자체의 공격적인 임계값 문제(예: DDoS 규칙이 트래픽 급증 이벤트를 너무 민감하게 잡음)라면 Override로 민감도를 낮춘다. 두 도구 모두 "규칙을 완전히 끈다"보다 훨씬 좁은 범위로 문제를 해결한다는 공통점이 있다.

Rate Limiting characteristic — "누구를 하나로 셀 것인가"가 규칙의 성패를 가른다

Rate Limiting Rule은 요청을 카운팅할 기준(characteristic)을 지정해야 한다. 가장 단순한 기준은 IP 주소지만, 이것만으로는 위험하다. 공식 문서가 제시하는 대안은 다음과 같다.

IP with NAT support — _cfuvid 쿠키 기반 방문자 식별

회사 VPN, 학교/사내 네트워크, 모바일 통신사(CGNAT) 뒤의 수많은 사용자는 같은 공인 IP를 공유한다. IP만 기준으로 삼으면 이들 전체가 "한 명의 클라이언트"로 카운팅돼, 정상적인 동시 사용만으로도 임계값을 넘겨 집단 차단될 수 있다. "IP with NAT support"는 _cfuvid 쿠키로 같은 IP 뒤의 개별 방문자를 구분해 카운팅을 분리한다. 다만 쿠키를 거부하거나 매번 지우는 클라이언트는 여전히 하나의 버킷으로 묶인다는 한계가 남는다.

JA3 / JA4 TLS 핑거프린트

클라이언트가 TLS 핸드셰이크를 맺는 방식(암호 스위트 조합 등)에서 파생되는 지문이다. 공격자가 IP를 계속 바꿔가며(봇넷, 프록시 로테이션) 공격해도 같은 TLS 스택/클라이언트 구현을 재사용하는 한 JA4 값은 유지되므로, IP 로테이션 회피 공격에 특히 유효하다.

실무 판단 기준

로그인, 결제, 회원가입처럼 남용 시 피해가 큰 엔드포인트는 IP 단독 대신 IP+NAT support, 또는 헤더/쿠키/JA4를 조합한 characteristic을 쓰고, 액션도 처음에는 Block보다 Managed Challenge로 시작해 "봇은 걸러내되 사람은 재시도로 통과"하게 만드는 편이 안전하다. 이건 "IP 기반 리미팅은 항상 정답"이라는 가정이 왜 위험한지, 그리고 그 대안이 무엇인지를 구체적으로 설명할 수 있는지를 보는 지점이다.

레거시 Firewall Rules / 구 Rate Limiting — API 지원은 이미 끝났다

Firewall Rules(Filters API)와 구버전 Rate Limiting 상품은 2025-06-15부로 API 지원이 종료됐고, 대시보드 경로도 WAF Custom Rules / Rate Limiting Rules로 완전히 통합됐다. 기존에 이 레거시 상품으로 IaC(Terraform 등)를 짜둔 팀이 있다면, API가 더 이상 그 리소스를 관리할 수 없으므로 새 리소스 스키마로 재작성하는 마이그레이션이 필요하다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 보안과 가용성의 trade-off를 다룰 수 있는가: "규칙을 켰다"가 끝이 아니라, 오탐이 실제 고객을 막았을 때 이를 좁게(Exception/Override) 되돌릴 수 있는가
  • 규칙 간 상호작용을 예측하는가: Custom Rules → Rate Limiting → Managed Rules 순서를 알고, Skip 액션이 왜 그 순서 때문에 의미가 있는지 설명할 수 있는가
  • "완전 자동 방어"에 안주하지 않는가: Managed Rules를 켜는 것만으로 끝내지 않고, 자신의 배포에도 단계적 검증(Log → Block) 원칙을 스스로 적용하는가
  • IP 기반 사고의 한계를 인식하는가: 공유 IP 환경(NAT, 모바일 캐리어)에서 리미팅이 정상 사용자를 집단으로 오탐할 수 있다는 걸 구체적으로 설명하는가
  • 레거시 자산을 관리할 수 있는가: Firewall Rules/구 Rate Limiting처럼 API 지원이 끝난 리소스가 남아있을 가능성을 스스로 점검하는 습관이 있는가

이 질문은 "WAF 켜봤어요, Rate Limiting 걸어봤어요"라는 경험 유무가 아니라, "오탐이 났을 때 무엇을 어떤 범위로 되돌릴지, 그리고 애초에 오탐이 나지 않도록 어떤 순서로 배포할지"를 검증한다. 규칙을 바로 Block으로 배포했다가 실서비스를 차단한 경험, 혹은 그 반대로 규칙을 통째로 꺼서 실제 공격을 통과시킨 경험 중 하나라도 겪어봤어야 나올 수 있는 구체성이 답변에 드러난다.

4. 실무 연결

시나리오 1 — 로그인 Rate Limiting이 사내 VPN 사용자 전체를 차단

로그인 엔드포인트에 "IP당 분당 10회 초과 시 Block" 규칙을 걸었다. 그런데 다른 지역 사무소 직원들이 회사 VPN(공인 IP 하나를 공유)으로 동시에 로그인을 시도하자, 이 트래픽이 전부 "IP 하나가 짧은 시간에 대량 요청"으로 잡혀 정상 로그인 대부분이 차단됐다. 원인은 characteristic을 순수 IP로만 잡았기 때문이었다. IP with NAT support(_cfuvid 쿠키 기반)로 전환하고, 액션도 Block 대신 Managed Challenge로 낮춰 사람은 재시도로 통과하되 자동화된 브루트포스만 실질적으로 걸러지도록 수정했다.

이 사례가 중요한 이유: Rate Limiting characteristic 선택은 "얼마나 자주"만큼이나 "누구를 하나로 셀 것인가"가 결과를 좌우한다.

시나리오 2 — 신규 Custom Rule을 바로 Block으로 배포해 정상 API 클라이언트 차단

새 파트너 API에 "특정 헤더 패턴이 없으면 Block"하는 Custom Rule을 작성해 곧바로 Block 액션으로 배포했다. 그런데 실제 파트너 시스템 일부가 그 헤더를 대소문자만 다르게 보내고 있었고, 이 차이를 사전에 Log 모드로 관찰하지 않았기 때문에 배포 직후 해당 파트너 트래픽 전체가 차단됐다. 이후 모든 신규 Custom Rule은 반드시 며칠간 Log 액션으로 먼저 배포하고 Security Events에서 매칭 패턴을 확인한 뒤 Block으로 전환하는 절차를 팀 표준으로 도입했다.

이 사례가 중요한 이유: Cloudflare 자신도 Managed Rules를 7일 사이클(1주차 Log)로 배포한다는 사실이, 우리 팀 규칙 배포 절차에도 그대로 적용돼야 하는 근거다.

시나리오 3 — Managed Ruleset이 정상 JSON payload를 SQLi로 오탐

내부 분석 API가 JSON 바디에 SQL 유사 키워드가 포함된 필터 조건(예: 사용자가 입력한 검색어에 OR, SELECT 같은 단어가 우연히 포함)을 전달하는 구조였는데, Cloudflare Managed Ruleset의 SQLi 관련 규칙이 이를 지속적으로 오탐 차단했다. 전체 SQLi 규칙군을 끄는 대신, 해당 API 경로 + 특정 헤더 조건에서만 그 규칙을 건너뛰는 Exception을 만들어 나머지 모든 엔드포인트에서는 SQLi 방어를 그대로 유지했다.

이 사례가 중요한 이유: 오탐 대응의 기본 원칙은 "얼마나 좁게 예외를 만들 수 있는가"이지, "규칙을 끌 것인가 말 것인가"라는 이분법이 아니다.

시나리오 4 — Managed Rule 자체의 결함이 전세계 장애로 이어진 사례(2024-10-10)

Cloudflare가 배포한 WAF 관리형 규칙 중 하나가 특정 입력값에 대해 정규식 처리 시간이 비정상적으로 폭증하는(ReDoS 유사) 문제를 가지고 있었고, 이 규칙이 전 세계 트래픽에 적용되며 대규모 502 오류로 이어졌다. 이 사례는 "Cloudflare가 관리하는 규칙조차 완벽하지 않다"는 걸 보여주며, 우리 팀이 작성하는 Custom Rule의 정규식/조건식도 동일한 방식으로 검토(복잡한 정규식 회피, 성능 테스트)해야 한다는 교훈으로 이어진다.

이 사례가 중요한 이유: 규칙 배포 자체가 장애 원인이 될 수 있다는 걸 인지하고 있으면, "왜 Log 모드로 먼저 관찰하는가"라는 질문에 훨씬 설득력 있게 답할 수 있다.

5. 답변 사고 순서

"WAF/Rate Limiting을 어떻게 설계하고 배포하시겠습니까"라는 질문에는 아래 순서로 답한다.

  1. 위협을 "내용 기반"과 "빈도 기반"으로 먼저 분류

    공격 패턴을 시그니처로 잡아야 하는 문제인지(WAF), 정상 내용이지만 양이 문제인지(Rate Limiting)를 구분해 어떤 도구를 쓸지 결정한다.

  2. 평가 순서를 고려해 규칙 간 상호작용을 설계

    DDoS → Custom Rules → Rate Limiting → Managed Rules 순서를 전제로, 알려진 정상 트래픽에 대한 예외는 Custom Rules 단계의 Skip으로 앞단에서 처리한다.

  3. 모든 신규/변경 규칙은 Log 모드로 먼저 배포

    Cloudflare 자신의 7일 사이클(1주차 Log → 2주차 기본 액션)과 같은 원칙으로, Security Events에서 매칭 패턴을 확인한 뒤 Block/Challenge로 전환한다.

  4. Rate Limiting characteristic을 엔드포인트 민감도에 맞게 선택

    IP 단독의 한계(NAT/모바일 캐리어 공유 IP)를 고려해 IP+NAT support, JA4, 헤더/쿠키 조합을 검토하고, 민감한 엔드포인트는 Block보다 Managed Challenge로 시작한다.

  5. 오탐 발생 시 Exception/Override로 좁게 대응

    규칙 전체 비활성화 대신, 조건이 좁혀지는 문제면 Exception, 임계값/민감도 문제면 Override로 대응 범위를 최소화한다.

  6. 레거시 자산 점검

    Firewall Rules/구 Rate Limiting으로 관리되던 리소스가 남아있는지 확인하고, API 지원 종료(2025-06-15) 이후에도 IaC가 그 리소스를 참조하고 있지 않은지 점검한다.

핵심 팁

이 순서에서 "평가 순서를 알고 있다는 것"과 "그 순서를 이용해 오탐 없는 배포 절차를 설계한다는 것"을 연결해서 설명하는 것이 시니어 답변의 핵심이다. 순서를 아는 것 자체는 암기지만, "그래서 Skip을 어디에 쓰고 Exception을 언제 쓰는가"까지 이어져야 실무 판단으로 인정된다.

6. 핵심 답변 예시

면접 답변 예시 — WAF/Rate Limiting 설계와 배포

"먼저 이 API가 막아야 하는 위협이 '내용 기반'인지 '빈도 기반'인지부터 나눕니다. SQL Injection이나 페이로드 조작처럼 요청 내용 자체가 문제라면 WAF Custom Rule과 Managed Rules로, 로그인 브루트포스나 과도한 API 호출처럼 내용은 정상인데 양이 문제라면 Rate Limiting Rule로 접근합니다.

규칙을 설계할 때는 평가 순서를 전제로 둡니다 — Custom Rules가 Rate Limiting과 Managed Rules보다 먼저 평가되므로, 사내 모니터링 봇이나 파트너 웹훅처럼 알려진 정상 트래픽은 Custom Rules 단계에서 Skip 액션으로 뒤 단계 전체를 건너뛰게 만들어 둡니다. 이렇게 하면 나중에 Managed Rules가 이 트래픽을 오탐으로 막는 상황 자체를 미리 방지할 수 있습니다.

배포는 절대 처음부터 Block으로 하지 않습니다. Cloudflare 자신도 Managed Rules 신규 규칙을 1주차는 Log로만 배포하고 2주차에 기본 액션으로 전환하는데, 저희 팀 규칙도 동일하게 며칠간 Log로 두고 Security Events에서 실제 매칭 패턴을 확인한 뒤 Block이나 Managed Challenge로 전환합니다. Rate Limiting은 characteristic을 IP 하나로만 잡지 않습니다 — 로그인처럼 민감한 엔드포인트는 회사 VPN이나 모바일 통신사처럼 여러 사용자가 IP를 공유하는 환경에서 집단 오탐이 날 수 있어서, IP with NAT support나 JA4 핑거프린트를 조합하고 액션도 Block보다는 Managed Challenge로 시작해 정상 사용자는 재시도로 통과하게 둡니다.

오탐이 실제로 발생하면 규칙을 끄는 대신, 그 문제가 특정 요청 패턴에 국한되면 Exception으로 좁게 예외 처리하고, 규칙 자체의 민감도 문제라면 Override로 액션이나 민감도를 단계적으로 낮춥니다. 마지막으로 레거시 Firewall Rules나 구 Rate Limiting으로 관리되던 리소스가 남아있는지도 확인합니다 — 이건 이미 API 지원이 끝나서 IaC로 더는 관리할 수 없는 상태일 수 있기 때문입니다."

이 답변이 좋은 이유: ① 위협을 내용/빈도로 먼저 분류 ② 평가 순서를 근거로 Skip의 용도를 구체적으로 설명 ③ 자신의 배포 절차에 Cloudflare의 7일 사이클 원칙을 그대로 적용 ④ IP 단독 리미팅의 한계를 구체적 환경(VPN, 모바일 캐리어)으로 설명 ⑤ Exception과 Override를 문제 유형에 따라 구분해서 사용 ⑥ 레거시 자산 점검까지 포함

7. Senior-level Follow-up Q&A

Q1. "Custom Rule에 Skip 액션을 쓰면 정확히 무엇을 건너뛰나요?"

Skip은 지정한 범위만큼 이후 평가를 건너뛴다. 범위는 "이후의 모든 Custom Rules", 또는 특정 phase(Rate Limiting, Managed Rules, Super Bot Fight Mode 등) 단위로 선택할 수 있다. 중요한 건 Skip이 Custom Rules phase 안에서 실행되기 때문에, Custom Rules보다 먼저 평가되는 DDoS 방어는 건너뛸 수 없다는 점이다 — 평가 순서상 이미 지나간 phase는 Skip으로 되돌릴 수 없다.

Q2. "Rate Limiting을 IP 기준으로만 설정하면 왜 위험한가요?"

같은 공인 IP를 공유하는 환경(회사 VPN, 학교/사내 네트워크, 모바일 통신사의 CGNAT)에서는 서로 무관한 수십~수백 명의 사용자가 "하나의 클라이언트"로 카운팅된다. 정상적인 동시 사용만으로도 임계값을 넘겨 전원이 차단될 수 있다. IP with NAT support(쿠키 기반 식별)나 JA4 같은 보조 characteristic을 조합해야 이 위험을 줄일 수 있다.

Q3. "Managed Rule이 새로 배포된 다음 날 갑자기 특정 엔드포인트가 막히기 시작했다면 어떻게 대응하나요?"

먼저 WAF changelog에서 최근 릴리스(대개 7일 주기, 월/화 배포) 내역을 확인해 그 시점에 어떤 규칙이 Log에서 Block으로 전환됐는지 찾는다. Security Events에서 차단된 요청의 매칭 규칙 ID를 확인한 뒤, 그 요청이 진짜 공격인지 정상 트래픽인지 판단한다. 정상 트래픽이면 규칙 전체를 끄지 않고, 해당 엔드포인트/조건에만 적용되는 Exception을 만들거나, 그 규칙만 Override로 액션을 Log로 되돌려 상황을 안정화한 뒤 근본 조건을 조정한다.

Q4. "Exception과 Override 둘 다 설정했는데 충돌하면 어떤 게 우선인가요?"

Exception이 Override보다 우선한다. Override로 특정 규칙의 액션을 Block으로 설정해 두었더라도, 그 요청이 Exception 조건에 해당하면 애초에 그 규칙(또는 룰셋)의 평가 대상에서 제외되므로 Override 자체가 적용될 기회가 없다. 두 설정을 함께 쓸 때는 "이 Exception이 의도한 범위보다 넓게 걸려 있어서, 원래 Override로 막아야 할 트래픽까지 새어나가고 있지 않은가"를 주기적으로 점검해야 한다.

Q5. "레거시 Firewall Rules를 쓰던 계정을 그대로 둬도 되나요?"

기존에 만들어진 규칙 자체는 계속 동작하지만, Firewall Rules(Filters API)와 구 Rate Limiting 상품은 2025-06-15부로 API 지원이 종료됐다. Terraform 같은 IaC로 이 리소스를 관리하고 있었다면 더 이상 API로 갱신/삭제가 불가능해질 수 있으므로, WAF Custom Rules / Rate Limiting Rules 스키마로 재작성하는 마이그레이션이 필요하다. "화면에 규칙이 여전히 보인다"는 것과 "API로 계속 관리할 수 있다"는 것은 다른 문제다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 내용 기반 위협(WAF)과 빈도 기반 위협(Rate Limiting)을 구분해서 접근 방식을 다르게 설계한다
  • 평가 순서(Custom Rules → Rate Limiting → Managed Rules)를 근거로 Skip/예외 설계를 구체적으로 설명한다
  • 신규 규칙을 Log 모드로 먼저 배포하고 검증한 뒤 전환하는 절차를 스스로의 원칙으로 제시한다
  • Rate Limiting characteristic을 엔드포인트 민감도와 클라이언트 환경(NAT, 모바일)에 맞게 선택한다
  • 오탐 대응을 Exception/Override로 구분해서 좁은 범위로 처리한다
약한 답변
  • "WAF랑 Rate Limiting 둘 다 켜두면 된다"고만 답하고 언제 무엇을 쓰는지 구분하지 못한다
  • 평가 순서를 모르거나, Managed Rules가 먼저 평가된다고 착각한다
  • 새 규칙을 곧바로 Block으로 배포해도 된다고 답한다
  • Rate Limiting을 항상 IP 기준으로만 설계하면 된다고 답한다
  • 오탐이 나면 "그 규칙을 끈다"고만 답한다
왜 약한 답변이 약한가

WAF와 Rate Limiting 설계는 "보안 기능을 켜는 것"이 아니라 "가용성을 해치지 않으면서 위협을 좁게 걸러내는 것"이다. 이 관점이 없으면 오탐이 났을 때 규칙을 끄는 극단적 선택으로 이어지거나(진짜 공격이 통과), 반대로 규칙을 그대로 둔 채 고객 불만만 쌓이게(정상 트래픽 차단) 된다. 시니어 면접에서는 "기능을 아는가"가 아니라 "가용성과 보안 사이에서 얼마나 좁고 정확하게 조준할 수 있는가"를 본다.

9. 흔한 오해

오해 1

"WAF만 잘 설정하면 로그인 브루트포스 같은 남용도 막을 수 있다."

실제

WAF는 요청 "내용"에서 공격 시그니처를 찾는 도구다. 브루트포스는 각 요청 내용이 완전히 정상적인 로그인 시도이므로 WAF 시그니처로는 잡히지 않는다. 이건 오직 "빈도"를 보는 Rate Limiting Rule의 영역이다.

오해 2

"Custom Rule에서 화이트리스트를 만들어 Managed Rules 오탐을 막으려면 Custom Rule의 액션을 Log나 Allow로 설정하면 된다."

실제

Managed Rules를 실제로 건너뛰게 하려면 액션을 Skip(대상 phase에 Managed Rules 포함)으로 설정해야 한다. Log는 비-terminal 액션이라 이후 phase(Rate Limiting, Managed Rules) 평가를 막지 못하고 계속 진행시킨다.

오해 3

"오탐이 발생한 Managed Rule은 즉시 꺼서 문제를 해결하는 게 맞다."

실제

규칙 전체를 끄면 그 규칙이 막던 진짜 공격까지 통과한다. 오탐 범위가 좁으면 Exception으로, 규칙 자체가 너무 공격적이면 Override로 액션/민감도를 단계적으로 조정하는 게 공식 권장 방식이다.

오해 4

"IP 기준 Rate Limiting이면 왠만한 상황에서 충분하다."

실제

회사 VPN, 사내망, 모바일 통신사(CGNAT) 뒤에서는 다수의 무관한 사용자가 같은 IP를 공유한다. 민감한 엔드포인트일수록 IP with NAT support, JA4 같은 보조 characteristic을 조합하지 않으면 정상 사용자 집단이 오탐으로 차단될 위험이 크다.

오해 5

"우리가 만든 규칙은 확실히 맞다고 확신하면 처음부터 Block으로 배포해도 된다."

실제

Cloudflare 자신도 전 세계 트래픽을 검사하는 Managed Rules를 신규 배포할 때 1주차는 반드시 Log로만 운영하고 실제 매칭을 관찰한 뒤 2주차에 Block으로 전환한다. "확실하다고 느끼는 것"과 "실제 트래픽에서 검증된 것"은 다르며, Log 단계를 생략하는 건 이 원칙을 스스로 거스르는 것이다.

10. 자기 점검 Rubric

아래 기준으로 자신의 답변 수준을 점검해라.

기준 5 — 시니어 3 — 미드레벨 1 — 개선 필요
WAF vs Rate Limiting 구분 내용 기반/빈도 기반 위협을 구분해 각 도구의 역할을 명확히 설명한다 "둘 다 보안 강화 기능"이라는 수준으로만 안다 두 개념을 혼동한다
평가 순서 이해 Custom Rules → Rate Limiting → Managed Rules 순서와 Skip의 실무적 용도를 설명한다 순서가 있다는 건 알지만 정확한 순서나 Skip 활용은 모른다 평가 순서 개념 자체가 없다
단계적 배포 원칙 Log→Block 전환을 자신의 배포 절차로 체화해 설명한다(Cloudflare 7일 사이클을 근거로 든다) "테스트는 해야 한다" 정도로만 안다 바로 Block으로 배포해도 된다고 답한다
Rate Limiting characteristic 설계 IP 단독의 한계(NAT/모바일)를 알고 NAT support/JA4 등 대안을 제시한다 IP 기준 리미팅만 안다 characteristic 개념 자체가 없다
오탐 대응 정밀도 Exception과 Override를 문제 유형에 따라 구분해서 좁게 대응한다 "규칙을 조정한다" 정도로만 뭉뚱그린다 규칙을 끄는 것만 대안으로 안다
