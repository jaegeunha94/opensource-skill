Firewall Rules와 구 Rate Limiting 상품은 2025-06-15부로 API 지원이 종료되고 WAF의 Custom Rules / Rate Limiting Rules로 통합됐다. 2. 전제 개념 — 쉬운 한국어로

Cloudflare는 근본적으로 무엇을 하는가

Cloudflare는 방문자와 여러분의 서버(origin) 사이에 끼어드는 중간자다. 원래는 방문자가 DNS로 여러분의 서버 IP를 직접 알아내서 바로 연결했다. Cloudflare를 쓰면 방문자는 Cloudflare의 엣지 서버에 먼저 연결하고, Cloudflare가 그 요청을 검사·캐싱·가공한 뒤 필요할 때만 origin에 전달한다.

이 "중간에 끼어드는" 능력이 CDN(캐시로 속도 향상), WAF(악성 요청 차단), DDoS 방어(트래픽 폭주 흡수), Bot 관리(자동화 트래픽 구분)의 공통 기반이다. 모든 게 이 하나의 구조에서 나온다: 방문자가 origin의 진짜 주소를 몰라야, Cloudflare가 그 사이에서 무언가를 할 수 있다.

Proxied(orange cloud) vs DNS-only(grey cloud)

Cloudflare DNS 대시보드에서 레코드마다 구름 아이콘이 있다. 주황색(Proxied)과 회색(DNS-only)의 차이는 다음과 같다.

Proxied (주황 구름)

DNS 조회 결과로 Cloudflare의 IP가 나간다. 방문자는 origin의 진짜 IP를 알 수 없다.

트래픽이 Cloudflare 엣지를 거치므로 캐싱, WAF, DDoS 방어, Bot 관리, Workers, 리다이렉트/변환 규칙이 모두 적용된다.

웹 트래픽(A/AAAA/CNAME)에 기본으로 권장

DNS-only (회색 구름)

Cloudflare는 DNS 응답만 해준다. 방문자는 origin의 진짜 IP로 직접 연결한다.

Cloudflare의 캐싱/WAF/DDoS 방어/Workers가 전혀 적용되지 않는다. 순수 DNS 서비스로만 동작한다.

HTTP/HTTPS가 아닌 프로토콜, 또는 프록시가 프로토콜을 깨뜨리는 레코드에 필수

언제 반드시 DNS-only여야 하는가

메일 관련 레코드(MX가 가리키는 A/CNAME 레코드), SSH처럼 HTTP가 아닌 프로토콜, Cloudflare가 지원하지 않는 포트를 쓰는 서비스가 대표적이다. 메일 서버 A 레코드를 실수로 proxied로 바꾸면 Cloudflare가 SMTP 트래픽을 이해하지 못해 수신·발신 메일이 모두 끊긴다. 실무에서 흔히 발생하는 사고 유형이다.

CDN 캐시 — 왜 origin 보호와 연결되는가

정적 리소스(이미지, CSS, JS, 캐시 가능한 API 응답)를 Cloudflare 엣지에 저장해두면, 같은 요청이 다시 오면 origin까지 가지 않고 엣지에서 바로 응답한다. 이건 속도 문제만이 아니다. origin에 도달하는 요청 수 자체를 줄여준다 — 트래픽 폭주나 소규모 DDoS성 트래픽도 캐시가 상당 부분 흡수한다.

Page Rules는 레거시, 지금은 Rules 제품군

과거엔 URL 패턴 하나에 캐시/리다이렉트/보안 설정을 묶어서 적용하는 Page Rules가 유일한 방법이었다. Page Rules는 "첫 번째로 매칭된 규칙만 적용"되는 구조였고, 계정당 규칙 개수 제한이 빡빡했다.

지금은 목적별로 분리된 Rules 제품군을 쓴다. Cache Rules(캐시 동작), Configuration Rules(보안/성능 기능 on/off), Origin Rules(트래픽이 어디로 나갈지), Redirect Rules(리다이렉트)로 나뉘고, 이들은 Page Rules와 달리 여러 규칙이 함께 누적 적용(stackable)된다. Cloudflare는 2025년부터 기존 Page Rules를 Cache Rules로 자동 전환했지만, "Disable Security"/"Disable Performance" 같은 일부 옵션은 자동 전환되지 않았다. 신규 구성이라면 처음부터 Rules 제품군으로 설계해야 한다.

SSL/TLS 모드 — 암호화 구간이 두 개라는 것을 이해하라

Cloudflare를 쓰면 통신 구간이 두 개로 나뉜다: ① 방문자 ↔ Cloudflare, ② Cloudflare ↔ origin. SSL/TLS 모드는 이 두 구간을 각각 암호화할지, 어떻게 검증할지를 결정한다.

Off
두 구간 모두 평문 HTTP. 2026년 기준 정당한 이유 없이 선택하면 안 되는 설정.
Flexible
① 구간만 HTTPS, ② 구간은 평문 HTTP. origin에 인증서가 없어도 방문자 브라우저엔 자물쇠가 뜬다 — 하지만 Cloudflare-origin 구간은 도청 가능하고, origin이 자체적으로 HTTPS 강제 리다이렉트를 하면 무한 리다이렉트 루프가 발생한다.
Full
두 구간 모두 HTTPS. 하지만 origin 인증서를 검증하지 않는다 — 자체 서명 인증서도 통과된다. 중간자가 origin인 척 자체 서명 인증서를 내밀면 이론상 여전히 취약하다.
Full (strict)
두 구간 모두 HTTPS + origin 인증서를 공인 CA 또는 Cloudflare Origin CA로 검증. 현재 공식 권장 모드.
Origin CA (Origin Certificate)란

Cloudflare가 무료로 발급하는 인증서로, Cloudflare만 신뢰한다 (일반 브라우저는 신뢰하지 않는다). origin 서버에 설치하면 Full (strict) 모드를 쓸 수 있다. 공인 CA(Let's Encrypt 등) 인증서를 origin에 설치해도 Full (strict)를 쓸 수 있지만, Origin CA는 15년짜리 장기 인증서를 발급할 수 있어 갱신 부담이 적다는 실무적 장점이 있다.

계층형 보안 경계 — 하나의 스위치가 아니다

Cloudflare의 보안은 순서대로 쌓이는 계층이다. 시니어 면접에서는 이 계층 전체를 설명할 수 있어야 한다.

1. DNS 경계 — Proxied
origin IP를 감춘다. 모든 하위 계층의 전제 조건이다.
2. DDoS 방어 (L3/L4/L7)
모든 플랜에서 상시 작동. 트래픽 프로파일을 자동 학습해 이상 트래픽을 완화한다.
3. WAF (Security > Security rules)
Managed Rules(OWASP 등 상용 룰셋) + Custom Rules(우리 서비스 특화 규칙). 과거의 "Firewall Rules"는 현재 Custom Rules로 통합됐다.
4. Rate Limiting Rules
로그인 brute-force, API 남용처럼 "정상 요청이지만 너무 자주"인 패턴을 막는다. 과거 별도 상품이었던 Rate Limiting도 지금은 WAF 안에 통합됐다.
5. Bot Management / Turnstile
자동화 트래픽을 구분해 차단하거나 챌린지한다. Turnstile은 사용자 마찰이 적은 CAPTCHA 대체재다.
6. Origin 보호 — 여기가 진짜 마지막 방어선
Authenticated Origin Pulls(Cloudflare가 보낸 요청인지 mTLS로 검증), origin 방화벽에서 Cloudflare IP 대역만 허용, 또는 Cloudflare Tunnel로 origin에 inbound 포트 자체를 열지 않는 구조.
핵심 포인트 — 1번 계층이 무너지면 나머지가 의미 없다

WAF, DDoS 방어, Bot 관리를 아무리 정교하게 설정해도, 공격자가 origin의 진짜 IP를 알아내서 직접 공격하면 이 모든 계층을 그냥 우회한다. 그래서 시니어 레벨 설계에서는 "WAF 규칙을 얼마나 잘 짰는가"보다 "origin IP가 새어나갈 경로가 있는가"를 먼저 점검한다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 보안 경계에 대한 입체적 사고: Cloudflare를 "체크박스 몇 개 켜는 도구"가 아니라 여러 계층이 쌓인 방어 구조로 이해하는가
  • 프로토콜별 판단력: "무조건 proxied가 좋다"가 아니라, 레코드 타입/트래픽 종류에 따라 proxied와 DNS-only를 구분할 수 있는가
  • Origin 노출 리스크 인식: Cloudflare 설정이 아무리 완벽해도 origin IP가 새어나가면 무의미하다는 것을 스스로 짚어내는가
  • 최신성 관리 능력: Cloudflare는 제품이 빠르게 바뀐다(Page Rules → Rules 제품군, Firewall Rules → WAF Custom Rules). 오래된 지식을 그대로 말하는지, 아니면 변화를 인지하고 있는지를 본다
  • trade-off 감각: 캐시를 공격적으로 걸면 성능은 좋아지지만 개인화 콘텐츠가 잘못 캐싱될 위험이 있다. 이런 균형을 스스로 언급하는가

이 질문은 "Cloudflare 대시보드를 써봤는가"를 묻지 않는다. "경계를 어떻게 설계하고, 그 경계가 무너지는 지점을 스스로 찾을 수 있는가"를 묻는다.

4. 실무 연결

실제 사례 1 — DNS-only로 방치된 apex 레코드로 인한 우회

보안 연구자 Smit Gharat는 대규모 도메인 스캔 중, 어떤 서비스의 서브도메인은 proxied로 잘 보호돼 있지만 apex 도메인(루트 도메인) A 레코드가 DNS-only로 origin IP를 그대로 노출하고 있는 걸 발견했다. curl --resolve로 그 IP에 직접 접속하니 Cloudflare 헤더 없이 origin이 그대로 응답했다. 이는 서브도메인 단위로만 보안 설정을 점검하고, 전체 존(zone)의 레코드를 빠짐없이 검토하지 않을 때 생기는 전형적인 사고다.

실제 사례 2 — 메일 서버 IP 노출을 통한 직접 공격

웹 서비스 앞단은 Cloudflare로 완전히 보호돼 있었지만, 같은 서버에서 동작하던 메일 서버의 IP(MX 레코드가 가리키는 A 레코드, 당연히 DNS-only)가 공개돼 있었다. 공격자는 이 IP로 직접 DDoS 트래픽을 쏟아부어 대역폭을 소진시켰다. 웹과 메일이 같은 origin을 공유한다면, "웹은 proxied로 안전하다"는 판단이 전체 인프라의 안전을 보장하지 않는다.

실제 사례 3 — Cross-tenant 우회 리서치 (2023)

보안 연구팀은 공격자가 무료 Cloudflare 계정을 만들어 피해자의 origin IP를 가리키는 커스텀 도메인을 등록하고, 자신의 테넌트에서는 보호 기능을 모두 꺼버린 뒤 그 경로로 공격 트래픽을 흘려보내는 방식이 가능함을 보였다. 이는 "Cloudflare를 쓰고 있다"는 사실 자체가 안전을 보장하지 않으며, origin이 Cloudflare의 IP 대역에서만 오는 트래픽을 받도록(Authenticated Origin Pulls) 스스로 강제해야 함을 보여주는 사례다.

실제 사례 4 — Cloudflare 자체 WAF 규칙이 만든 전세계 502 장애 (2024-10-10)

Cloudflare가 공식 post-mortem에서 밝힌 사례다. 인라인 JavaScript 공격 탐지를 개선하는 새 WAF Managed Rule을 "simulate(로그만 남기고 차단은 안 함)" 모드로 배포했는데, 그 규칙의 정규식이 특정 트래픽에서 재앙적 백트래킹(ReDoS)을 일으켜 전세계 엣지 서버 CPU가 100%까지 치솟았다. 결과적으로 정상 트래픽까지 502 에러를 받으며 전체 트래픽이 한때 약 82% 급감했다. 20~30분 만에 WAF Managed Rulesets를 전역 강제 종료해 복구했다.

이 사례가 중요한 이유: "Log 모드는 안전하다"는 가정이 항상 맞지는 않는다 — 차단 여부와 무관하게 규칙을 평가하는 것 자체가 비용이 될 수 있다. origin을 보호하려고 만든 계층(WAF)이 오히려 전체 장애의 원인이 될 수 있다는 것도 시니어가 짚어야 할 포인트다.

실제 사례 5 — 내부 설정 파일 전파로 인한 대규모 장애 (2025-11-18)

데이터베이스 권한 변경으로, Bot Management가 5분마다 재생성해 전 세계 엣지에 배포하는 "feature file"에 중복 행이 섞여 파일 크기가 예상보다 커졌다. 이 파일에는 200개 features라는 하드코딩된 상한이 있었는데, 상한을 넘는 파일이 들어오자 소프트웨어가 우아하게 처리하지 못하고 그대로 죽어버렸다(panic). 권한 변경이 DB 클러스터 일부에만 먼저 적용돼, 5분 주기마다 정상 파일과 비정상 파일이 번갈아 배포되며 간헐적으로 복구됐다가 다시 죽는 패턴을 보여 원인 파악이 더 어려웠다.

이 사례가 중요한 이유: "설정 파일 배포"도 "코드 배포"와 똑같이 canary·크기 검증·circuit breaker가 필요하다는 것을 보여준다. Cloudflare는 이 사건 이후 blast radius를 줄이는 "Fail Small" 회복탄력성 계획을 공개했다.

시나리오 — Flexible SSL이 만든 무한 리다이렉트

origin 애플리케이션 프레임워크가 기본적으로 HTTP 요청을 HTTPS로 강제 리다이렉트하도록 설정돼 있었다. Cloudflare SSL/TLS 모드가 Flexible이면, Cloudflare는 origin에 HTTP로 요청을 보낸다. origin은 이를 HTTPS로 리다이렉트하고, 방문자는 다시 Cloudflare의 HTTPS 엔드포인트로 오고, Cloudflare는 다시 origin에 HTTP로 요청한다 — 무한 루프. 이 문제는 SSL/TLS 모드를 Full 이상으로 바꾸면 해결된다.

5. 답변 사고 순서

면접에서 "DNS부터 보안 경계까지 어떻게 구성하겠습니까"라는 질문을 받으면, 아래 순서로 사고하고 답변하라.

  1. 트래픽 특성 파악

    정적 콘텐츠 비중, 동적/개인화 콘텐츠 비중, API 트래픽 여부, 메일 등 비-HTTP 서비스가 같은 도메인에 있는지 확인한다.

  2. 레코드별 proxied/DNS-only 결정

    웹 트래픽(A/AAAA/CNAME)은 proxied 기본, 메일(MX가 가리키는 레코드)이나 비-HTTP 서비스는 DNS-only로 명확히 분리한다.

  3. SSL/TLS 모드와 origin 인증서 결정

    Full (strict)를 기본으로 하고, origin에 Origin CA 또는 공인 CA 인증서를 설치한다.

  4. 캐시 전략 설계 (Cache Rules)

    무엇을 캐싱할지(정적 자산), 무엇을 캐싱하면 안 되는지(로그인 세션이 섞인 응답)를 Cache Key와 함께 설계한다. Page Rules가 아니라 Cache Rules로 설계한다.

  5. 보안 계층 설계 (WAF → Rate Limiting → Bot Management)

    Managed Rules로 기본 방어를 깔고, 서비스 특화 로직은 Custom Rules로, 로그인/API 남용은 Rate Limiting Rules로, 자동화 트래픽은 Bot Management/Turnstile로 처리한다.

  6. Origin 은닉 확정

    Authenticated Origin Pulls(mTLS) 또는 origin 방화벽에서 Cloudflare IP 대역만 허용, 필요하면 Cloudflare Tunnel로 inbound 포트 자체를 열지 않는다.

  7. 관측성과 롤백 계획

    Logpush/Analytics로 캐시 히트율, WAF 차단율, 오탐률을 모니터링하고, 신규 규칙은 Log 모드로 먼저 배포한 뒤 Block으로 전환한다.

핵심 팁

순서를 정확히 맞추는 것보다, "origin 은닉"을 스스로 언급하는지가 시니어와 미드레벨을 가르는 핵심 포인트다. 이 부분은 물어보지 않아도 먼저 말해야 한다.

6. 핵심 답변 예시

면접 답변 예시 — 신규 서비스 도메인 구성

"먼저 이 도메인에 어떤 트래픽이 붙는지 확인하겠습니다. 웹 서비스 외에 메일이나 다른 비-HTTP 서비스가 같은 도메인을 쓴다면, 그 레코드는 반드시 DNS-only로 분리합니다. 메일 레코드를 proxied로 두면 수신·발신이 즉시 끊기는 사고로 이어지기 때문입니다.

웹 트래픽용 A/CNAME 레코드는 proxied로 설정해서 origin IP를 감춥니다. SSL/TLS 모드는 Full (strict)로 설정하고, origin에는 Cloudflare Origin CA 인증서를 설치합니다. Flexible은 Cloudflare-origin 구간이 평문이라 도청 위험과 리다이렉트 루프 위험이 있어 프로덕션에서는 쓰지 않습니다.

캐시는 Cache Rules로 설계합니다. 정적 자산은 공격적으로 캐싱하고, 로그인 세션이 섞인 API 응답은 캐싱 대상에서 명확히 제외합니다. 과거 Page Rules 방식은 규칙이 순서에 의존하고 개수 제한이 있어 신규 구성에는 쓰지 않습니다.

보안은 WAF Managed Rules로 기본 방어를 깔고, 로그인/결제 엔드포인트에는 Rate Limiting Rules를 별도로 적용합니다. 봇 트래픽이 문제가 되는 서비스라면 Bot Management나 Turnstile을 도입합니다.

가장 중요한 마지막 단계는 origin 은닉입니다. Authenticated Origin Pulls를 켜서 origin이 Cloudflare가 보낸 요청만 mTLS로 검증하게 하고, origin 방화벽에서도 Cloudflare IP 대역 외의 접근을 차단합니다. 이 단계가 없으면 앞의 모든 보안 설정은 origin IP가 새어나가는 순간 무력화됩니다.

마지막으로 신규 WAF 규칙은 항상 Log 모드로 먼저 배포해 오탐률을 확인한 뒤 Block으로 전환하고, Logpush로 캐시 히트율과 차단 로그를 지속적으로 모니터링하겠습니다."

이 답변이 좋은 이유: ① 레코드 타입별 판단 근거 제시 ② 최신 용어(Cache Rules, WAF Custom/Rate Limiting Rules) 사용 ③ origin 은닉을 스스로 마지막 방어선으로 언급 ④ 단계적 배포(Log → Block)로 운영 리스크 관리를 보여줌

7. Senior-level Follow-up Q&A

Q1. "Flexible SSL이 왜 위험한지 좀 더 구체적으로 설명해 주세요."

Flexible은 방문자-Cloudflare 구간만 HTTPS이고 Cloudflare-origin 구간은 평문 HTTP다. 이 구간이 인터넷을 통과한다면(같은 데이터센터 내부망이 아니라면) 도청이나 변조에 노출된다.

더 실무적인 문제는 리다이렉트 루프다. origin이 HTTP 요청을 HTTPS로 강제 리다이렉트하도록 설정돼 있으면, Cloudflare(HTTP로 요청) → origin(HTTPS로 리다이렉트) → 방문자는 다시 Cloudflare HTTPS로 → Cloudflare는 다시 origin에 HTTP로 요청, 이 루프가 반복돼 사이트 전체가 리다이렉트 에러로 다운된다.

따라서 origin이 HTTPS를 지원한다면 최소 Full, 인증서를 검증할 수 있다면 Full (strict)를 써야 한다.

Q2. "Page Rules가 아직 동작한다고 들었는데, 왜 새로 안 만드나요?"

기존 zone에 있는 Page Rules는 여전히 동작한다. 하지만 Cloudflare는 이를 레거시로 전환하고 있고, 실제로 2025년부터 기존 Page Rules를 Cache Rules로 자동 마이그레이션하는 작업을 진행했다. 다만 "Disable Security", "Disable Performance" 같은 일부 옵션은 자동 전환 대상이 아니었다.

Page Rules는 "첫 번째 매칭 규칙만 적용"이라 규칙이 늘어날수록 순서 관리가 어렵고, 계정당 규칙 개수 제한도 빡빡하다. 반면 Cache Rules/Configuration Rules/Origin Rules/Redirect Rules는 목적별로 분리돼 있고 여러 규칙이 함께 누적 적용된다. 신규 구성이라면 처음부터 이 제품군으로 설계하는 게 유지보수 측면에서 명백히 유리하다.

Q3. "origin IP를 완전히 숨기는 방법에는 뭐가 있고, 뭐가 더 강력한가요?"

두 단계가 있다. ① Authenticated Origin Pulls: origin이 mTLS로 "이 요청이 진짜 Cloudflare에서 왔는가"를 검증한다. origin에 여전히 열린 inbound 포트가 있지만, Cloudflare 클라이언트 인증서 없이는 응답하지 않는다. ② origin 방화벽에서 Cloudflare IP 대역만 허용: 네트워크 레벨에서 아예 다른 곳의 접속을 차단한다.

더 강력한 방법은 Cloudflare Tunnel이다. origin 서버에 아웃바운드로만 연결하는 커넥터를 설치하면, origin은 inbound 포트를 아예 열지 않아도 된다. 공격자가 IP를 알아내도 접속할 포트 자체가 없으니, 위 두 방법보다 근본적으로 노출 표면을 줄인다. 다만 아키텍처 변경 비용이 있으므로 신규 구축이나 Zero Trust 전환 프로젝트에서 우선 검토한다.

Q4. "DNS-only로 둬야 하는 레코드의 구체적인 예시를 들어주세요."

① MX 레코드가 가리키는 메일 서버 A/CNAME 레코드 — proxied로 두면 SMTP 트래픽이 깨진다. ② SSH나 커스텀 TCP 포트로 직접 접근해야 하는 관리용 서버 — Cloudflare 프록시는 HTTP(S) 위주라 임의 포트/프로토콜을 지원하지 않는다(Spectrum 같은 별도 상품이 있지만 기본 프록시와는 다르다). ③ SPF/TXT 레코드처럼애초에 proxied 개념이 적용되지 않는 레코드 타입.

핵심은 "이 레코드가 HTTP(S) 트래픽을 받는가, Cloudflare가 이해하는 프로토콜인가"를 레코드마다 확인하는 것이다.

Q5. "캐시가 왜 성능 주제를 넘어서 보안 주제로도 다뤄지나요?"

캐시 설계를 잘못하면 그 자체가 취약점이 된다. 예를 들어 로그인한 사용자의 개인화된 응답이 실수로 캐싱되면, 다른 사용자에게 그 응답이 그대로 노출되는 캐시 오염(cache poisoning)/캐시 기만(cache deception) 문제가 생길 수 있다.

그래서 Cache Rules를 설계할 때는 "어떤 헤더/쿠키/쿼리 파라미터를 Cache Key에 포함시킬지"를 신중히 정해야 한다. 캐시는 성능 최적화 도구이자, 잘못 쓰면 정보 유출 통로가 되는 양날의 검이다.

Q6. "Cloudflare 자체가 장애를 일으킨 적이 있나요? 그런 경우 어떻게 대비합니까?"

있다. 2024년 10월 Cloudflare는 새 WAF Managed Rule의 정규식이 "로그만 남기는 simulate 모드"였음에도 ReDoS로 전세계 엣지 CPU를 100%까지 소진시켜 502 장애를 냈다고 공식 post-mortem에서 밝혔다. 2025년 11월에는 Bot Management용 내부 설정 파일이 DB 권한 변경으로 예상보다 커지면서 하드코딩된 상한을 넘어 소프트웨어가 죽는 대규모 장애도 있었다.

이런 사례는 "Cloudflare를 쓰면 무조건 더 안전/안정적"이라는 가정이 항상 맞지는 않음을 보여준다. 실무적으로는 ① 신규 보안 규칙을 일부 트래픽/리전에 먼저 적용하는 점진적 롤아웃을 요청하거나 자체 트래픽에 대해 모니터링을 강화하고, ② Cloudflare 상태 페이지·RSS를 장애 대응 온콜 알림에 연동하고, ③ 단일 CDN/보안 벤더 의존이 비즈니스 리스크로 얼마나 큰지 판단해 필요 시 멀티 벤더 전략(예: DNS 페일오버)을 검토하는 것으로 대비한다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 레코드 타입별로 proxied/DNS-only를 구분해서 설명한다
  • SSL/TLS 모드를 두 구간(방문자-Cloudflare, Cloudflare-origin)으로 나눠 설명한다
  • origin 은닉(Authenticated Origin Pulls, IP 허용목록, Tunnel)을 스스로 언급한다
  • Page Rules가 레거시임을 알고 Cache Rules/Configuration Rules로 설명한다
  • 신규 WAF 규칙을 Log 모드로 먼저 검증하는 단계적 배포를 언급한다
약한 답변
  • "Cloudflare 켜면 다 proxied로 하면 된다"고 단순화한다
  • SSL/TLS 모드를 "HTTPS 켜는 옵션" 정도로만 이해한다
  • WAF/DDoS 방어만 이야기하고 origin 보호는 언급하지 않는다
  • Page Rules를 여전히 현재 표준 도구처럼 설명한다
  • 캐시를 "빠르게 해주는 기능"으로만 설명하고 잘못된 캐싱의 위험은 모른다
왜 약한 답변이 약한가

"Cloudflare 켜면 안전하다"는 도구를 써봤다는 것만 보여준다. 시니어 면접에서는 그 도구가 어디까지 보호하고, 어디서부터는 스스로 origin을 지켜야 하는지를 구분할 수 있는지를 본다. Cloudflare가 아무리 강력해도 origin IP가 새어나가면 방어선이 통째로 무력화된다는 것을 모르면, 실제 장애나 침해 사고 앞에서 원인을 못 찾는다.

9. 흔한 오해

오해 1

"Proxied만 켜면 origin이 완전히 안전하다."

실제

Proxied는 DNS 응답이 Cloudflare IP로 나가게 할 뿐이다. origin IP가 다른 경로(같은 서버의 DNS-only 레코드, 과거 인증서 발급 기록의 Certificate Transparency 로그, 메일 헤더 등)로 노출되면 그 경로로 직접 공격이 가능하다. Authenticated Origin Pulls나 IP 허용목록으로 "Cloudflare가 아닌 곳에서 온 요청은 아예 응답하지 않는다"까지 해야 진짜 보호가 된다.

오해 2

"Flexible SSL이면 브라우저에 자물쇠가 뜨니까 안전한 것이다."

실제

자물쇠 아이콘은 방문자-Cloudflare 구간이 암호화됐다는 뜻일 뿐, Cloudflare-origin 구간은 평문 HTTP다. 이 구간이 도청당하면 데이터가 그대로 노출된다. 게다가 origin이 자체적으로 HTTPS 리다이렉트를 강제하면 무한 리다이렉트 루프까지 발생할 수 있다.

오해 3

"Page Rules가 아직 대시보드에 보이니 지금도 새로 만들 수 있다."

실제

기존 zone에 이미 있는 Page Rules는 계속 동작하지만, 2025-01-06부터는 어떤 플랜의 어떤 계정도 신규 Page Rule을 생성할 수 없다(Free/Pro/Business는 2024-07-01부터, Enterprise 신규 계정은 2024-08-01부터 이미 막혔다). Cloudflare는 기존 규칙을 Cache Rules 등으로 자동 전환하는 작업을 진행 중이며, "Disable Security"/"Disable Performance"/Railgun 같은 일부 옵션은 자동 전환 대상이 아니므로 마이그레이션 시 수동 확인이 필요하다.

오해 4

"WAF Managed Rules만 켜면 애플리케이션 보안은 다 해결된다."

실제

Managed Rules는 OWASP 계열의 일반적인 공격 패턴을 막아준다. 우리 서비스만의 로그인 brute-force, 특정 API 엔드포인트 남용 같은 것은 Custom Rules와 Rate Limiting Rules로 별도 설계해야 한다. 또한 Managed Rules는 오탐(false positive)이 발생할 수 있어, 배포 전 Log 모드로 트래픽을 관찰하는 단계가 필요하다.

오해 5

"DNS-only로 두는 것 자체가 보안 취약점이다."

실제

DNS-only 자체는 문제가 아니다. 메일 서버처럼 애초에 Cloudflare 프록시가 지원하지 않는 트래픽이라면 DNS-only가 올바른 선택이다. 진짜 문제는 "이 레코드가 가리키는 서버가 웹 서비스와 같은 origin을 공유하는데도 보호 없이 노출돼 있는가"이다. 레코드 하나하나를 트래픽 종류에 맞게 판단하는 것이 핵심이지, "무조건 proxied가 정답"은 아니다.

아래 기준으로 자신의 답변 수준을 점검해라.

SSL/TLS 모드 이해 두 구간을 나눠 설명하고 Full (strict)를 이유와 함께 권장한다 모드 이름은 알지만 위험을 구체적으로 설명 못한다 "HTTPS 켜는 옵션" 정도로만 이해한다
Origin 은닉 인식 Authenticated Origin Pulls/IP 허용목록/Tunnel을 스스로 언급한다 물어보면 한두 가지는 말한다 origin 보호를 전혀 언급하지 않는다
캐시/Rules 최신성 Cache Rules/Configuration Rules로 설명하고 Page Rules의 한계를 안다 Cache Rules는 알지만 Page Rules와의 차이는 모른다 Page Rules를 현재 표준처럼 설명한다
계층형 보안 설계 DDoS → WAF → Rate Limiting → Bot → Origin까지 전 계층을 순서 있게 설명한다 2~3개 계층만 언급한다 "WAF 켠다" 정도로 뭉뚱그린다
trade-off 언급 캐시 공격성, WAF 오탐, 운영 비용 등 trade-off를 자발적으로 말한다 물어보면 말할 수 있다 trade-off 개념이 없다
장애/실무 사례 연결 구체적 사고 유형(메일 단절, 리다이렉트 루프, IP 노출 우회)을 사례로 든다 이론적으로 가능하다고만 말한다 실무 연결이 없다
