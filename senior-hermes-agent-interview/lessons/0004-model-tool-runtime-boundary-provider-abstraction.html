전제 개념

"경계(boundary)"라는 말이 실제로 가리키는 것

왜 "provider 추상화"가 필요한가

먼저 문제부터 짚는다. Anthropic, OpenAI, Google 같은 회사마다 API 요청 형식, 인증 방식(OAuth vs API 키), tool calling을 표현하는 JSON 스키마가 전부 다르다. 만약 에이전트 코드 안에 "Anthropic이면 이렇게, OpenAI면 저렇게" 식의 분기가 흩어져 있다면, provider를 하나 추가할 때마다 메모리·스킬·tool dispatch 로직까지 전부 건드려야 한다. provider 추상화 계층은 이 차이를 한 곳에 몰아넣어서, 나머지 코드는 "어떤 provider인지 몰라도 되게" 만드는 설계다. 쉬운 비유로, 전원 콘센트 규격이 나라마다 달라도 어댑터 하나만 바꾸면 노트북 자체는 그대로 쓸 수 있는 것과 같다 — provider 추상화가 이 "어댑터" 역할이다.

Provider 계층 구조 — 4개 티어

공식 문서 기준으로 Hermes가 지원하는 provider는 성격이 다른 4개 티어로 나뉜다.

티어 대표 예시 인증/특징
통합 게이트웨이 Nous Portal OAuth 로그인 1번으로 300개 이상 모델 + Tool Gateway(웹검색/이미지생성/TTS/브라우저)까지 하나의 구독으로 처리 — "여러 계정 관리"를 없애는 게 목적
멀티 provider 라우터 OpenRouter 비용 기준 정렬, 처리량 기준 라우팅, 코딩 품질 임계값을 만족하는 가장 싼 모델을 자동 선택하는 실험적 Pareto Code 라우터
Direct 1st-class provider Anthropic(native), OpenAI/Codex, Google Vertex AI, AWS Bedrock, Azure Foundry, GitHub Copilot, MiniMax, xAI Grok, DeepSeek 각자의 인증 방식을 직접 구현(예: Copilot은 gho_*/github_pat_* OAuth 토큰, Vertex AI는 서비스 계정 기반 OAuth2)
Custom OpenAI-호환 엔드포인트 로컬 Ollama/vLLM/llama.cpp, LiteLLM/ClawRouter 같은 상용 프록시, Together AI/Groq/Perplexity "OpenAI 호환 API"라는 최소 계약만 지키면 무엇이든 붙을 수 있음 — provider 추상화의 최종 탈출구(escape hatch)

이 4개 티어가 시사하는 설계 원칙은, provider 추상화가 "모든 provider를 하나의 API로 강제 통일"하는 게 아니라 "1st-class 통합 + 범용 호환 레이어를 함께 제공"하는 절충이라는 점이다. 새 provider가 나오면 1st-class로 깊이 통합하거나, 최소한 OpenAI 호환 엔드포인트로 즉시 붙일 수 있게 이중 경로를 열어둔 것이다.

설정 계층 — 어디에 뭘 적는가

Provider/모델 선택은 hermes model 인터랙티브 커맨드로 하고, 그 결과는 다음 우선순위 계층에 저장된다(높은 순위가 낮은 순위를 덮어씀).

  1. CLI 인자 — 실행 시점 1회성 오버라이드
  2. ~/.hermes/config.yaml — 모델, terminal backend, 압축 설정, 메모리 한도, toolset 등 "비밀이 아닌" 설정 전부
  3. ~/.hermes/.env — API 키, 봇 토큰, 비밀번호 등 "비밀"만
  4. 내장 기본값 — 하드코딩된 최종 fallback
"secret은 .env, 나머지는 config.yaml"이라는 분리 원칙이 핵심이다. 이렇게 나누는 이유는 config.yaml은 팀원과 공유하거나 git에 커밋해도 되지만, .env는 절대 커밋하면 안 되는 파일이기 때문이다 — 설정 파일 하나에 다 몰아넣으면 "공유해도 되는 설정"과 "공유하면 사고 나는 비밀"을 실수로 같이 커밋할 위험이 커진다.

보조 모델(auxiliary model) 라우팅과 provider: "auto"

비전 분석, 웹 요약, 압축, 제목 생성, Mixture-of-Agents 같은 부가 작업은 메인 채팅 모델과 다른(보통 더 싸거나 특화된) provider로 보낼 수 있다. 설정은 다음과 같은 공통 패턴을 따른다. 

auxiliary:
  vision:
    provider: "auto"  # 메인 모델을 그대로 재사용
    model: ""
    base_url: ""

provider: "auto"가 기본값인 이유가 중요하다 — 만약 보조 작업이 항상 "가장 싼 모델"로 자동 다운그레이드된다면, Nous Portal 같은 구독형 aggregator를 쓰는 사용자 입장에서는 "내가 설정하지 않은 낮은 품질의 응답이 예고 없이 섞여 나오는" 놀라운(surprising) 동작이 된다. auto는 이 예측 불가능성을 없애고, 필요할 때만 명시적으로 다른 provider를 지정하게 만든다.

자격증명 풀(credential pool)과 fallback 체인

같은 provider에 API 키가 여러 개 있을 때는 로테이션 전략을 고를 수 있다 — fill_first(기본, 첫 키를 소진할 때까지 사용), round_robin(순서대로 순환), least_used(가장 적게 쓰인 키 우선), random. 그리고 provider 자체가 실패하면 fallback_providers 체인으로 자동 failover 하며, 이 체인은 보조 작업별로도 독립적으로 설정할 수 있다. 즉 "메인 모델이 죽으면 대신 무엇을 쓸지"와 "보조 작업이 죽으면 대신 무엇을 쓸지"를 따로 설계할 수 있는 것이다.

v0.18.0의 두 가지 최신 변화: MoA와 Vertex AI

이번 릴리스에서 Mixture-of-Agents(MoA)가 "이름 붙은 앙상블(named ensemble)"로 1st-class 승격되어, 모델 선택 화면에서 일반 모델과 똑같이 고를 수 있게 됐다. 여러 참조 모델의 추론 과정이 모두 보이고, 최종 aggregator의 답변이 실시간 스트리밍된다. 이게 아키텍처적으로 중요한 이유는, "모델을 고른다"는 인터페이스가 "provider + model ID 튜플"에서 "응답을 만들어내는 그 무엇이든 가리키는 추상 참조(resolvable model reference)"로 확장되었다는 뜻이기 때문이다 — 단일 모델이든, 여러 모델의 앙상블이든 호출하는 쪽 코드 입장에서는 같은 인터페이스로 취급된다.

또한 Google Vertex AI가 Gemini용 1st-class provider로 추가됐다. 정적 API 키 대신 서비스 계정 JSON이나 Application Default Credentials(ADC)에서 단기 OAuth2 토큰(약 1시간 유효)을 직접 발급하고, 세션 중간에 401이 나면 자동으로 재발급한다. 이는 "provider 추상화가 인증 방식의 차이까지 감싸야 한다"는 원칙을 보여주는 구체적 사례다 — API 키 하나로 통일하는 게 아니라, provider별 인증 프로토콜(OAuth 코드 흐름, 서비스 계정 토큰 발급, 정적 키)을 각각 구현하고 위에서는 동일하게 "그냥 provider 하나 선택한 것"처럼 보이게 만든다.

Tool Dispatch 파이프라인 — tool_call이 실행되기까지

모델이 tool_call을 반환한 순간부터의 경로는 다음과 같다.

  1. 모델 응답 — 모델이 tool_call을 반환
  2. Agent loop — run_agent.py가 이를 수신
  3. 1차 핸들러 — model_tools.handle_function_call(name, args, task_id, user_task) 호출
  4. 분기 — agent-loop 툴이면 직접 처리, 아니면 registry로
  5. pre-tool hook — 플러그인이 실행 전 가로챌 기회
  6. registry.dispatch(name, args, **kwargs) — 이름으로 ToolEntry 조회
  7. 핸들러 실행(동기/비동기)
  8. 에러 래핑 — 항상 JSON 형태로 결과 반환
  9. post-tool hook — 실행 후 후처리

여기서 핵심은 todo/memory/session_search/ delegate_task가 registry로 가지 않고 handle_function_call 단계에서 바로 처리된다는 점이다. 이유는 이 툴들이 agent-loop 자체의 상태(현재 task 목록, 세션 컨텍스트 등)에 직접 접근해야 하는데, registry에 등록된 일반 툴은 "독립적으로 실행 가능한 캡슐화된 기능"을 전제로 설계되어 있기 때문이다. 이 툴들도 모델에게 보이는 스키마는 registry에 함께 등록되어 있지만(그래야 모델이 존재를 알 수 있으니까), 실제 dispatch는 절대 registry까지 내려가지 않는다 — 문서에는 "혹시라도 registry까지 도달하면 stub 에러를 낸다"고 명시되어 있을 만큼 의도적으로 분리된 경로다.

에러 처리는 이중이다 — registry.dispatch()가 핸들러 예외를 잡아 {"error": "Tool execution failed: ..."}로 감싸고, handle_function_call()이 dispatch 전체를 한 번 더 감싼다. 이렇게 하는 이유는 모델이 절대 처리되지 않은 예외를 그대로 받아서는 안 되기 때문이다 — 예외가 그대로 새어나가면 모델이 다음 턴에 무엇을 해야 할지 판단할 근거(잘 정의된 JSON 에러 메시지)를 잃는다.

Native tool calling vs text-based fallback

provider/모델에 따라 tool calling을 표현하는 방식이 다르다. Llama 3.x, Qwen 2.5, Mistral, Hermes 2/3, DeepSeek 등은 네이티브 tool calling을 지원하지만, 그렇지 않은 서버는 텍스트 기반 fallback으로 처리된다. 셀프호스팅 서버는 명시적 플래그가 필요하다 — vLLM은 --enable-auto-tool-choice --tool-call-parser hermes, llama.cpp는 --jinja. 이건 provider 추상화와 tool dispatch가 만나는 지점이다 — 같은 tool_call 개념이라도, 어떤 provider/모델을 쓰느냐에 따라 실제 파싱 로직이 달라진다는 뜻이다.

execute_code — Programmatic Tool Calling

execute_code는 에이전트가 Python 스크립트를 작성해서 Hermes 툴을 프로그래밍적으로 호출하게 하는 기능이다. 여러 단계 파이프라인을 LLM 턴 1번으로 압축한다. 동작 방식은 다음과 같다.

  1. 에이전트가 from hermes_tools import ... 형태로 스크립트 작성
  2. Hermes가 RPC 함수를 담은 hermes_tools.py stub 모듈을 자동 생성
  3. Unix domain socket을 열고 RPC 리스너 스레드 시작
  4. 스크립트는 별도 자식 프로세스에서 실행되며, 소켓을 통해 Hermes와 통신
  5. print() 출력만 LLM에게 반환 — 중간 tool 결과는 컨텍스트에 절대 들어가지 않음

스크립트에서 쓸 수 있는 툴은 web_search, web_extract, read_file, write_file, search_files, patch, terminal(foreground only)로 제한된다.

단순히 "여러 tool call을 한 번에 몰아 보낸다"는 것과 다른 지점이 여기다 — 순차 tool calling은 매 tool 결과가 전부 컨텍스트에 쌓이지만, execute_code는 중간 결과(예: 검색 결과 50개 중 필요한 3줄만) 스크립트 내부에서 걸러내고 최종 print() 결과만 돌려준다. 그래서 "3개 이상의 tool call + 그 사이의 처리 로직"이 필요한 작업에서 토큰 비용이 극적으로 줄어든다.

보안 관점에서는 자식 프로세스가 최소 환경(minimal environment)에서 돈다 — KEY, TOKEN, SECRET, PASSWORD, CREDENTIAL, PASSWD, AUTH가 포함된 환경변수는 기본적으로 전부 제거된다. 필요하면 config.yaml에서 명시적으로 allowlist에 넣거나 스킬 frontmatter의 required_environment_variables로 선언해야 통과된다. 리소스 제약도 있다 — 최대 5분 타임아웃, stdout 50KB, stderr 10KB, tool call 50회 상한. 그리고 Unix domain socket에 의존하므로 Linux/macOS에서만 동작하고, Windows에서는 자동으로 비활성화되어 일반 순차 tool calling으로 폴백한다.

세 계층의 경계 정리

계층 책임 모르는 것
Model 계층 tool 스키마를 받고, 결과를 해석해 다음 행동 결정 tool이 실제로 어떤 코드로 실행되는지, 어떤 환경에서 도는지
Tool 계층 기능 등록(ToolEntry), 요청 처리, JSON 반환 실행 환경이 로컬인지 Docker인지, 위험 명령 승인이 어떻게 되는지
Runtime 계층 실행 환경 관리(local/Docker/SSH 등), 터미널 실행, 위험 명령 승인 흐름 어떤 모델이 이 tool을 호출했는지, 그 tool이 무슨 의미인지
이 3계층 분리가 실무적으로 의미하는 것: provider를 바꿔도 tool 계층은 안 바뀌고, tool을 추가해도 runtime 계층(격리·승인 정책)은 안 바뀌고, runtime backend를 바꿔도(로컬 → Docker) tool 계층 코드는 안 바뀐다. 세 계층이 서로 독립적으로 교체 가능하다는 것 자체가 이 아키텍처의 핵심 가치다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • provider lock-in을 피하는 설계 감각이 있는가: "그냥 API 키를 바꾸면 된다"는 피상적 답이 아니라, 인증 방식·tool calling 파싱 방식·context length 해석까지 provider마다 다르다는 것을 아는가
  • 계층 분리 설계력: Model/Tool/Runtime을 독립적으로 교체 가능한 계층으로 나눠서 설명할 수 있는가, 아니면 "다 그냥 하나의 시스템"으로 뭉뚱그리는가
  • 구조적 예외를 구조적으로 설명하는가: todo/memory가 왜 일반 registry 경로를 안 타는지를 "agent-loop 상태 접근이 필요해서"라는 원리로 설명하는가, 아니면 암기한 사실로만 나열하는가
  • 토큰 비용을 아키텍처 문제로 다루는가: execute_code가 왜 존재하는지, 컨텍스트 윈도우 절약이라는 구체적 메커니즘으로 설명하는가
  • 보안 감각: programmatic tool calling이 만드는 새로운 실행 표면(자식 프로세스, 자격증명 노출 가능성)을 인지하고 완화책(env stripping, 리소스 제한, 플랫폼 제약)을 아는가
실무에서 이 판단을 잘못하면, 특정 provider의 API에 강하게 커플링된 코드를 작성해서 나중에 provider 장애나 가격 변동 시 전체 재작성이 필요해지는 사고가 난다. 면접관은 후보가 "지금 당장 동작하는 코드"와 "다음 provider가 추가돼도 버티는 코드"의 차이를 설계 단계에서 이미 고려하는지를 본다.
실무 연결

실제 상황과 연결하기

상황 1: 특정 provider가 rate limit에 걸려 작업이 멈췄다

야간 배치 작업 중 메인 provider가 429를 반환하기 시작했는데, 재시도만으로는 해결이 안 됐다. → fallback_providers 체인을 미리 설정해뒀다면 자동으로 다음 provider로 failover되어 작업이 중단 없이 이어진다. 이걸 사전에 설계하지 않으면 "장애 발생 시점에 사람이 수동으로 provider를 바꿔야 하는" 운영 부담이 생긴다.

상황 2: 팀에서 API 키 여러 개를 돌려쓰는데 특정 키만 계속 소진된다

fill_first(기본 전략)를 그대로 쓰면 첫 번째 키만 계속 소진되고 나머지는 거의 안 쓰이는 불균형이 생긴다. → least_used 전략으로 바꾸면 여러 키의 사용량이 고르게 분산되어, 개별 키의 rate limit에 먼저 걸리는 상황을 줄일 수 있다.

상황 3: 순차 tool calling만 쓰다가 컨텍스트가 금방 찬다

"파일 50개를 검색해서 특정 패턴이 있는 파일만 골라 요약하라"는 작업을 순차 tool call로 처리하면 검색 결과 50개가 모두 컨텍스트에 쌓인다. → execute_code로 전환하면 스크립트 내부에서 필터링·요약 로직을 실행하고 최종 결과만 print()로 반환하므로, 같은 작업을 훨씬 적은 토큰으로 처리할 수 있다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
provider 추상화 계층부터 짚는다 통합 게이트웨이(Nous Portal), 멀티 provider 라우터(OpenRouter), direct 1st-class provider, custom OpenAI-호환 엔드포인트라는 4개 티어로 나눠 설명한다.
2
설정 계층과 secret 분리를 설명한다 CLI 인자 > config.yaml > .env > 기본값 순서, 그리고 왜 비밀과 비밀 아닌 설정을 분리했는지를 연결한다.
3
tool dispatch 파이프라인을 구체적으로 설명한다 run_agent.py → handle_function_call → agent-loop 툴 분기 vs registry.dispatch → ToolEntry까지 이름을 대며 설명한다.
4
execute_code로 토큰 비용 최적화를 연결한다 중간 결과가 컨텍스트에 안 들어가는 구조와, 그로 인한 보안 완화책(env stripping, 리소스 제한)까지 언급한다.
5
Model/Tool/Runtime 3계층 분리로 마무리한다 각 계층이 서로 독립적으로 교체 가능하다는 것이 왜 유지보수와 보안에 중요한지 결론짓는다.
핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 먼저 provider 추상화를 4개 티어로 나눠서 설명하겠습니다. Nous Portal 같은 통합 게이트웨이는 OAuth 로그인 한 번으로 여러 모델과 부가 툴을 하나의 구독으로 묶고, OpenRouter는 비용·성능 기준으로 최적 모델을 골라주는 라우터 역할을 합니다. Anthropic 네이티브나 Vertex AI 같은 direct provider는 각자의 인증 방식(OAuth, 서비스 계정 토큰 발급 등)을 직접 구현하고, 마지막으로 OpenAI 호환 엔드포인트라는 범용 탈출구를 열어둬서 로컬 모델이든 서드파티 프록시든 최소한의 계약만 지키면 붙을 수 있게 합니다.

설정은 CLI 인자, config.yaml, .env, 내장 기본값 순서의 계층으로 관리되고, 비밀(API 키)과 비밀 아닌 설정(모델 선택, 압축 옵션 등)을 파일 단위로 분리해서 실수로 비밀을 커밋하는 사고를 구조적으로 줄입니다.

tool calling 쪽은, 모델이 tool_call을 반환하면 run_agent.py의 agent loop가 이를 받아 handle_function_call로 넘기고, 여기서 todo나 memory처럼 agent-loop 자체의 상태에 직접 접근해야 하는 툴은 registry로 가지 않고 바로 처리됩니다. 그 외 일반 툴은 registry.dispatch가 이름으로 ToolEntry를 찾아 핸들러를 실행하고, 어떤 예외가 나든 항상 잘 정의된 JSON 에러로 감싸서 모델에게 돌려줍니다.

여기에 더해 execute_code라는 programmatic tool calling 기능이 있는데, 이건 에이전트가 Python 스크립트를 작성해서 여러 tool을 한 스크립트 안에서 호출하고, 중간 결과는 컨텍스트에 전혀 들어가지 않고 최종 print() 출력만 돌아오는 구조입니다. 그래서 검색 결과 수십 개 중 필요한 부분만 걸러내는 작업에서 토큰 비용을 크게 줄일 수 있습니다. 다만 스크립트가 별도 자식 프로세스에서 돌기 때문에, API 키나 토큰이 담긴 환경변수는 기본적으로 전부 제거하고, 타임아웃과 호출 횟수도 제한해서 새로운 실행 표면을 통제합니다.

결론적으로 이 모든 설계는 Model 계층(스키마와 결과 해석), Tool 계층(등록과 실행), Runtime 계층 (실행 환경과 승인 정책)을 서로 독립적으로 교체 가능하게 분리한 것이라고 생각합니다. provider를 바꿔도 tool 코드는 그대로고, 실행 환경을 로컬에서 Docker로 바꿔도 tool 계층은 그대로인 게 이 아키텍처의 핵심 가치라고 봅니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

Follow-up 1: "Mixture-of-Agents를 '그냥 모델처럼' 고를 수 있게 만든 게 아키텍처적으로 어떤 의미인가요?"

모범적인 사고 방향: "모델을 선택한다"는 인터페이스가 단순한 "provider + model ID" 튜플에서, 응답을 만들어내는 그 무엇이든(단일 모델이든 여러 모델의 앙상블이든) 동일하게 취급하는 추상 참조(resolvable reference)로 확장됐다는 점을 짚는다. 호출하는 쪽 코드는 "이게 단일 모델인지 앙상블인지" 몰라도 되게 만드는 것이 provider 추상화의 다음 단계이고, 이는 나중에 라우팅 전략이나 새로운 앙상블 기법이 추가돼도 호출부 코드를 안 건드릴 수 있게 해준다고 설명하면 좋다.

Follow-up 2: "execute_code가 왜 Windows에서 비활성화되나요? 이게 프로덕션 운영에 어떤 영향을 주나요?"

모범적인 사고 방향: Unix domain socket이라는 OS 레벨 IPC 메커니즘에 의존하기 때문에 Windows에서는 동작하지 않고, 자동으로 순차 tool calling 폴백으로 전환된다는 사실을 정확히 짚는다. 프로덕션 관점에서는 크로스 플랫폼 fleet을 운영할 때 "일부 노드에서는 토큰 효율적인 경로를 쓰고, 일부에서는 더 비싼 순차 경로를 쓰는" 비일관성이 생길 수 있음을 지적하고, 이를 감안한 비용 추정과 모니터링이 필요하다고 답하면 좋다.

Follow-up 3: "credential pool 로테이션 전략 중 least_used가 유리한 상황과 round_robin이 유리한 상황을 구분해줄 수 있나요?"

모범적인 사고 방향: round_robin은 각 키의 처리량이 비슷하고 "순서대로 고르게 쓰는 것"만으로 충분할 때 적합하지만, 키마다 rate limit이나 이미 소진된 quota가 다르다면 단순 순환은 특정 키가 계속 실패하는 상황을 만들 수 있다. least_used는 실제 사용량을 추적해서 "가장 여유 있는 키"를 우선 선택하므로, 키별 한도가 불균등한 팀 환경에서 더 안정적이라고 답하는 것이 좋다.

Follow-up 4: "tool dispatch에 pre-tool/post-tool hook이라는 플러그인 확장점이 왜 따로 필요한가요? 코어 dispatch 로직에 직접 넣으면 안 되나요?"

모범적인 사고 방향: 감사 로깅, 위험 명령 승인 게이트, 레이트 리밋 같은 횡단 관심사 (cross-cutting concern)를 코어 dispatch 코드에 직접 하드코딩하면, 새로운 정책이 추가될 때마다 dispatch 함수 자체를 계속 수정해야 하고 회귀 위험이 커진다. hook 시스템은 코어 로직을 건드리지 않고 "실행 전/후에 무엇을 검사하거나 가로챌지"를 플러그인 형태로 조립할 수 있게 해준다 — 이는 Day 10에서 다룰 승인 흐름(approval gate)이 실제로 어느 지점에 꽂히는지와도 연결된다고 답하면 좋다.

좋은 답변 vs 약한 답변

차이를 명확히 보기

좋은 답변

  • provider를 4개 티어(게이트웨이/라우터/direct/custom)로 구분해서 설명한다
  • 설정 계층(CLI > yaml > env > 기본값)과 secret 분리 원칙을 함께 언급한다
  • handle_function_call, registry.dispatch, ToolEntry 등 실제 구현 이름을 대며 dispatch 경로를 설명한다
  • todo/memory가 registry를 우회하는 이유(agent-loop 상태 접근)를 구조적으로 설명한다
  • execute_code의 토큰 절약 메커니즘과 보안 완화책을 함께 짚는다
  • Model/Tool/Runtime 3계층이 독립적으로 교체 가능하다는 결론으로 마무리한다

약한 답변

  • "provider를 여러 개 지원한다"는 사실만 나열하고 계층 구조를 설명하지 못한다
  • API 키만 바꾸면 provider 전환이 끝난다고 생각한다(인증 방식·tool calling 파싱 차이를 간과)
  • tool dispatch를 "그냥 모델이 함수를 부른다" 수준으로만 설명한다
  • execute_code를 "여러 tool을 한 번에 부르는 기능" 정도로만 이해하고 컨텍스트 절약 메커니즘을 설명하지 못한다
  • 보안 측면(env stripping, 리소스 제한, 플랫폼 제약)을 전혀 언급하지 않는다
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "provider를 바꾸는 건 그냥 API 키(엔드포인트)만 바꾸면 된다."
실제로는 인증 프로토콜(OAuth vs 정적 키 vs 서비스 계정 토큰), tool calling 표현 방식(네이티브 vs 텍스트 fallback), context length 해석 방식까지 provider마다 다르다. 이 차이를 흡수하는 계층이 없으면 provider 전환 시 예상치 못한 곳에서 tool calling이 깨지거나 context 계산이 틀어진다.
오해 2: "execute_code는 그냥 여러 tool call을 한 번에 몰아 보내는 기능이다."
단순 batch 호출이라면 각 tool의 결과가 여전히 전부 컨텍스트에 쌓인다. execute_code의 진짜 가치는 중간 결과가 컨텍스트에 전혀 들어가지 않고, 스크립트 내부에서 처리한 최종 print() 출력만 반환된다는 점이다 — "한 번에 여러 개"가 아니라 "중간 과정을 LLM 시야 밖에서 처리"하는 것이 핵심이다.
오해 3: "todo/memory도 그냥 하나의 일반 tool일 뿐이다."
registry에 등록된 일반 tool은 "독립적으로 실행 가능한 캡슐화된 기능"을 전제로 한다. 하지만 todo/memory/session_search/delegate_task는 agent-loop 자체의 실행 중 상태에 직접 접근해야 하므로, 애초에 registry를 거치지 않고 handle_function_call 단계에서 가로채는 구조적으로 다른 경로를 가진다.
자기 점검 Rubric

내 답변 수준 확인하기

Weak "여러 provider를 지원한다"는 사실만 말하고, 어떻게 그게 가능한지 구조를 전혀 설명하지 못한다.
Junior provider 티어 구분 정도는 알지만, tool dispatch 파이프라인이나 execute_code의 구체적 메커니즘까지는 연결하지 못한다.
Mid provider 추상화와 tool dispatch를 각각 설명하지만, 이 둘을 "Model/Tool/Runtime 계층 분리"라는 하나의 원칙으로 통합해서 설명하지 못한다.
Senior 4개 provider 티어 → 설정 계층/secret 분리 → 구체적 dispatch 파이프라인(agent-loop 툴 vs registry) → execute_code의 토큰/보안 트레이드오프 → 3계층 독립 교체 가능성이라는 결론까지 하나의 논리로 연결한다.
핵심 요약

오늘 반드시 기억할 것

  • Provider는 통합 게이트웨이(Nous Portal)/멀티 provider 라우터(OpenRouter)/direct 1st-class provider/custom OpenAI-호환 엔드포인트라는 4개 티어로 나뉜다
  • 설정은 CLI 인자 > config.yaml > .env > 기본값 순서의 계층이며, secret과 비-secret 설정을 파일 단위로 분리한다
  • tool_call은 run_agent.py → handle_function_call → (agent-loop 툴 직접 처리 또는 registry.dispatch → ToolEntry) 경로를 거친다
  • todo/memory류가 registry를 우회하는 이유는 agent-loop 상태에 직접 접근해야 하기 때문이다
  • execute_code는 중간 tool 결과를 컨텍스트에서 배제하고 최종 print()만 반환해 토큰을 절약하며, 자식 프로세스 credential stripping과 리소스 제한으로 보안 표면을 통제한다
  • Model/Tool/Runtime 3계층이 서로 독립적으로 교체 가능한 것이 이 아키텍처의 핵심 가치다
