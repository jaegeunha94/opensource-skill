Day 2 · Hermes Agent Interview

Local-first 실행과 배포 토폴로지

6개 실행 backend 사이에서 "항상 켜져 있어야 한다"는 요구와 "쓸 때만 돈 낸다"는 요구를 어떻게 저울질하는가
버전/출처 기준: 이 레슨은 2026-07 기준 Nous Research의 Hermes Agent (github.com/NousResearch/hermes-agent, 최신 태그 v0.18.0 "The Judgment Release", 공식 문서 hermes-agent.nousresearch.com/docs/)를 기준으로 작성됐다. 레슨 작성 시점에 GitHub Releases와 공식 문서를 재확인한 결과 v0.18.0 이후 새 릴리스는 없었고, 6개 실행 backend(local/Docker/SSH/Daytona/Singularity/Modal) 구성도 그대로 유지되고 있다. 일부 3rd-party 블로그는 "7개 sandbox backend"라고 서술하지만, 공식 GitHub README와 공식 문서, 다수의 독립 튜토리얼이 공통적으로 6개를 명시하므로 이 레슨은 6개를 기준으로 한다 — 면접 전에는 반드시 공식 문서로 최신 목록을 재확인해야 한다.
면접 질문
Q1 — 주력 질문 "Hermes Agent는 local, Docker, SSH, Daytona, Singularity, Modal이라는 6개 실행 backend를 제공합니다. always-on 개인 비서를 운영한다고 할 때, 이 중 무엇을 고르고 왜 그렇게 판단하시겠습니까?"
Q2 — 변형 "Daytona나 Modal 같은 서버리스 backend는 유휴 시 hibernate(절전)된다고 들었습니다. 이게 always-on 요구사항과 충돌하지 않나요? 실제로 충돌한다면, 그런데도 왜 쓰는 팀이 있을까요?"
Q3 — 변형 "'$5짜리 VPS 하나로 24시간 돌린다'는 배포 패턴을 들어보셨을 텐데, 이게 왜 always-on 에이전트의 기본 배포 토폴로지로 자리 잡았는지, 그리고 이게 무한정 확장 가능한 해법인지 설명해주세요."
전제 개념

실행 backend란 무엇을 결정하는 선택인가

"어디서 실행되는가"는 "무엇이 실행되는가"와 다른 질문이다

Day 1에서 "local-first는 개인 노트북을 뜻하지 않는다"는 걸 다뤘다. 이번에는 그 판단을 더 구체적으로 쪼갠다. Hermes Agent에서 "terminal backend" 설정은 에이전트의 추론(LLM 호출)이 어디서 일어나는지가 아니라, 에이전트가 실행하는 셸 명령·코드가 물리적으로 어느 환경에서 돌아가는지를 결정한다. LLM 추론은 항상 provider API(Day 4)로 나가지만, "파일을 읽고, 스크립트를 돌리고, 패키지를 설치하는" 실제 작업은 로컬 머신일 수도, Docker 컨테이너일 수도, 원격 SSH 서버일 수도, 서버리스 샌드박스일 수도 있다.

쉬운 비유로, "에이전트의 두뇌(LLM)"는 항상 클라우드에 있는 상담사와 통화하는 것과 같고, "에이전트의 손발(terminal backend)"은 그 상담사의 지시를 실제로 수행하는 현장 작업자가 어디 서 있느냐의 문제다. 상담사가 어디 있든, 작업자가 내 사무실에 있는지, 격리된 창고(컨테이너)에 있는지, 먼 지사(SSH 원격 서버)에 있는지, 필요할 때만 부르는 파견 인력(서버리스)인지에 따라 비용·속도·안전성이 완전히 달라진다.

6개 backend를 목적별로 분류하기

공식 문서와 GitHub README를 종합하면, 6개 backend는 각각 뚜렷한 목적을 가진다. 이걸 그냥 "6가지 옵션이 있다"로 암기하면 얕은 답이 되고, "격리 수준", "상시성", "비용 모델"이라는 3개 축으로 분류해야 시니어답게 설명할 수 있다.

Backend 주 용도 격리 수준 상시성(always-on 적합도) 비용 모델
local 개인 기기에서 직접 실행 없음(호스트와 동일 권한) 낮음 기기 소유 비용만
docker 컨테이너 격리 실행 중간 호스트에 달림 호스트 인프라 비용
ssh 원격 서버에 실행 위임 원격 서버 정책에 종속 높음(원격이 상시 가동이면) 원격 서버 유지비
daytona 일회성/임시 개발 환경 높음 hibernate-on-idle 활성 시간만 과금(유휴시 거의 0)
singularity HPC(고성능 컴퓨팅) 워크로드 연구용 클러스터 정책 종속 클러스터 스케줄러에 종속 클러스터 과금 정책
modal 서버리스 GPU/CPU 샌드박스 높음 cold start 후 실행 실행 시간 단위 과금
3개 축으로 나눠서 기억하면 편하다 — "내 손 안(local)" vs "울타리 안(docker)" vs "남의 집(ssh)" vs "필요할 때만 빌리는 창고(daytona/modal)" vs "연구소 슈퍼컴(singularity)". 면접에서 6개를 나열만 하지 말고 이 3축(격리/상시성/비용)으로 분류해서 설명하면 구조적으로 이해했다는 신호를 준다.

Hibernate-on-idle과 always-on의 근본적 충돌

Daytona와 Modal의 핵심 매력은 "유휴 시 절전, 필요할 때 깨어남"이다. 이건 비용 관점에서는 훌륭하다 — 아무 일도 안 하는 시간에는 거의 과금되지 않는다. 그런데 always-on 에이전트가 요구하는 것은 정반대다: 사용자가 아무 말도 안 해도 예약된 작업(cron)을 스스로 깨워 실행하고, 메신저에서 들어오는 요청에 지연 없이 반응해야 한다.

이 둘은 구조적으로 긴장 관계에 있다. "아무도 안 쓰면 재워서 돈을 아낀다"는 서버리스의 존재 이유와 "아무도 안 써도 깨어서 스스로 뭔가 해야 한다"는 always-on의 요구사항은 거의 정반대 목표다. 실측 자료를 보면 Daytona는 장시간 유휴 후 첫 요청에서 약 300ms 안팎의 깨어남 지연이 있고(이후 요청은 컨테이너가 이미 떠 있으므로 빠르다), Modal은 워크로드에 따라 약 200ms에서 길게는 2~4초의 콜드 스타트가 발생할 수 있다. 사람이 채팅하듯 기다릴 수 있는 수준이라면 문제가 안 되지만, "새벽 3시에 스스로 깨어나 예약된 배치 작업을 실행"하는 시나리오에서는 애초에 "깨울 트리거가 없다"는 게 문제다 — 서버리스는 외부 요청이 와야 깨어나는데, cron처럼 에이전트 스스로 시간이 되어 깨어나야 하는 경우 깨울 주체가 모호해진다.

공식 문서에 세부 스펙이 없는 부분 — 아키텍처 추론: Hermes Agent 공식 문서는 "cron이 서버리스 backend를 어떻게 깨우는지"에 대한 세부 메커니즘을 명시하지 않는다. 구조상 합리적으로 추론하면, cron 스케줄러 자체는 게이트웨이/코어 프로세스(Day 8) 쪽에 상시로 떠 있어야 하고, 그 상시 프로세스가 예정된 시각에 서버리스 backend를 능동적으로 호출해서 깨우는 구조일 것이다. 즉 "서버리스 backend가 스스로 시간을 보고 깨어나는" 게 아니라 "상시 실행되는 다른 컴포넌트가 필요한 순간에 서버리스를 불러 쓰는" 구조로 보는 것이 타당하다 — 면접에서는 이걸 사실처럼 단정하지 말고 추론이라고 밝히며 설명한다.

"$5 VPS" 패턴이 기본값이 된 이유

바로 위의 충돌 때문에, always-on 목적에는 결국 "유휴 시에도 계속 떠 있는 저비용 VPS"가 가장 무난한 기본값이 된다. 실측/실무 가이드를 종합하면 다음과 같은 특징이 있다.

  • DigitalOcean Droplet, Hetzner CX22, Vultr 등 월 5~7달러 수준의 저가 VPS로도 충분히 돌아간다 — 로컬 LLM을 직접 얹지 않는 한 에이전트 자체의 메모리 사용량은 500MB 미만으로 가볍다.
  • 공식적으로 Docker 배포가 권장 방식이며, 공식 이미지(nousresearch/hermes-agent)를 VPS 위에서 컨테이너로 띄우는 패턴이 일반적이다.
  • 프로세스가 죽거나 서버가 재부팅돼도 자동으로 재기동되도록 PM2 같은 프로세스 매니저(또는 systemd 유닛)로 감싸는 것이 always-on 운영의 사실상 표준이다.
  • 총 운영비는 VPS 5~7달러 + LLM API 호출 비용(모델 선택에 따라 월 2~15달러 수준)으로, "24시간 상주 비서"치고는 매우 저렴한 편이다(Day 14에서 provider 라우팅 비용까지 더 깊게 다룬다).

여기서 시니어급 통찰은 "$5 VPS면 다 해결된다"가 아니라, 이 패턴이 유효한 스케일과 무효한 스케일을 구분하는 것이다. 개인 1인 비서, 소규모 팀 자동화라면 단일 VPS로 충분하다. 하지만 여러 사용자, 여러 세션, GPU가 필요한 로컬 모델 추론, 엄격한 네트워크 격리가 필요한 조직이라면 단일 VPS는 단일 장애점 (SPOF)이 되고, 이때는 Docker/Kubernetes 기반의 다중 인스턴스, 혹은 SSH backend로 여러 원격 실행 환경을 묶는 편이 맞는 판단이 된다.

상황 적합한 backend/토폴로지 이유
개인 1인 always-on 비서 저비용 VPS + Docker + PM2 가볍고 저렴하며, 상시 가동을 자연스럽게 보장
일회성 코드 실행/실험, 짧은 세션 Daytona hibernate로 유휴 비용을 없애고, 실험 격리도 보장
GPU 필요한 배치성 작업(로컬 파인튜닝 검증 등) Modal 필요한 순간만 GPU를 빌려 쓰는 비용 구조
이미 있는 사내 서버 인프라 활용 SSH 기존 신뢰 경계·네트워크 정책을 그대로 재사용
연구용 클러스터에서 대규모 계산 Singularity HPC 환경의 표준 컨테이너 런타임과 호환
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "6개 backend 이름을 외웠는가"를 보는 게 아니다. 면접관이 실제로 확인하려는 건 비용·상시성·격리라는 서로 충돌하는 요구사항 사이에서 상황에 맞는 판단을 내릴 수 있는가다.

  • "서버리스 = 항상 좋다"는 유행어에 휘둘리지 않는가: hibernate-on-idle이 always-on과 구조적으로 충돌한다는 것을 스스로 짚어낼 수 있는가
  • 비용 최적화와 요구사항 충족을 동시에 저울질하는가: "$5 VPS"가 만능 해법이 아니라 특정 스케일(1인/소규모)에서 유효한 패턴이라는 걸 아는가
  • 격리 수준이라는 축을 가지고 있는가: local과 Docker와 SSH가 왜 다른 보안 함의를 갖는지 설명할 수 있는가(이건 Day 10 샌드박싱과 직결된다)
  • 단일 장애점(SPOF) 개념을 아키텍처 판단에 연결하는가: 단일 VPS의 한계를 알고 스케일이 커질 때 무엇으로 바꿔야 하는지 아는가
  • 추론과 사실을 구분해서 말하는가: cron이 서버리스 backend를 깨우는 정확한 내부 메커니즘처럼 공식 문서에 없는 부분을 정직하게 "추론"이라고 표시하는가
실무에서 이 판단을 잘못하면 실제로 발생하는 사고가 있다 — 서버리스가 저렴하다는 이유만으로 always-on 비서를 Daytona/Modal 위에 올렸다가, 예약된 새벽 리포트가 몇 주간 조용히 실행되지 않다가 뒤늦게 발견되는 식이다. 면접관은 이런 실제 장애 시나리오를 후보가 사전에 예측할 수 있는지를 본다.
실무 연결

실제 상황과 연결하기

상황 1: "비용 아끼려고 서버리스로 옮겼는데 예약 작업이 멈췄어요"

매달 인프라 비용 절감이 목표라서 always-on 비서를 Modal 서버리스로 옮겼더니, 매일 아침 보내던 리포트가 조용히 실행되지 않는 사고가 발생한다. 원인은 "깨울 트리거가 없다"는 구조적 문제다. → 비용 최적화 이전에 "이 워크로드가 스스로 깨어나야 하는가, 아니면 외부 요청에만 반응하면 되는가"를 먼저 구분해야 한다.

상황 2: 단일 VPS가 다운되어 팀 전체 자동화가 멈췄을 때

개인용으로 시작한 "$5 VPS" 패턴을 그대로 팀 규모로 확장했더니, VPS 하나가 다운되는 순간 여러 팀원의 자동화가 동시에 멈춘다. → 스케일이 커지면 단일 인스턴스의 SPOF 위험을 인지하고 다중화(Docker 오케스트레이션, 여러 SSH 대상 등)로 전환하는 시점을 판단해야 한다.

상황 3: 보안 감사에서 "왜 로컬 backend를 쓰냐"는 지적을 받았을 때

개발 초기에 빠르게 데모하려고 local backend를 그대로 프로덕션에 남겨뒀는데, 감사에서 "에이전트가 호스트 권한으로 임의 명령을 실행할 수 있다"는 지적을 받는다. → 격리 수준이 다른 backend(Docker/SSH 등)로 전환하는 것이 데모 단계와 프로덕션 단계를 구분하는 실무 판단이다 (Day 10에서 심화).

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
"실행 backend"가 무엇을 결정하는지부터 정의한다 LLM 추론 위치가 아니라 셸 명령이 실제로 실행되는 물리적 환경을 결정한다는 점을 짚는다.
2
6개를 3개 축(격리/상시성/비용)으로 분류해서 설명한다 단순 나열이 아니라 구조적 기준으로 묶어서 말한다.
3
hibernate-on-idle과 always-on의 구조적 충돌을 짚는다 서버리스가 저렴한 이유(유휴 시 절전)가 always-on이 요구하는 것(스스로 깨어남)과 정반대라는 점을 명확히 설명한다.
4
"$5 VPS" 패턴을 근거와 함께 제시하고, 유효 범위를 한정한다 저비용 VPS + Docker + 프로세스 매니저가 개인/소규모에는 적합하지만 무한정 확장되는 해법은 아니라는 것까지 말한다.
5
스케일이 바뀌면 판단도 바뀐다는 것으로 마무리한다 규모·보안 요구·워크로드 특성에 따라 backend 선택이 달라지는 trade-off 감각을 보여준다.
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Hermes Agent의 실행 backend 설정은 LLM 추론이 어디서 일어나는지가 아니라, 에이전트가 실행하는 셸 명령이나 코드가 실제로 어느 환경에서 도는지를 결정하는 설정이라고 이해하고 있습니다. local, Docker, SSH, Daytona, Singularity, Modal 6개가 있는데, 이걸 그냥 나열하기보다는 격리 수준, 상시성, 비용 모델이라는 세 축으로 나눠서 봅니다.

always-on 개인 비서를 운영한다면 저는 우선 저비용 VPS 위에 Docker로 배포하고 PM2 같은 프로세스 매니저로 감싸는 조합을 고를 것 같습니다. 이유는, Daytona나 Modal 같은 서버리스 backend는 유휴 시 hibernate돼서 비용은 거의 안 들지만, always-on이 요구하는 건 정반대이기 때문입니다 — 사용자가 아무 말도 안 해도 cron으로 예약된 작업을 스스로 깨워서 실행해야 하는데, 서버리스는 외부 요청이 와야 깨어나는 구조라 '깨울 트리거가 없다'는 문제가 생깁니다. 실제로 비용을 아끼려고 서버리스로 옮겼다가 새벽에 도는 예약 리포트가 조용히 멈추는 사고가 날 수 있다고 봅니다.

그래서 개인이나 소규모 팀 수준에서는 월 5~7달러짜리 VPS 하나로도 충분하다는 게 실무 가이드에서 반복적으로 나오는 패턴입니다. 에이전트 자체 메모리 사용량이 로컬 LLM 없이는 500MB 미만으로 가볍기 때문입니다. 다만 이게 만능은 아니고, 여러 사용자·여러 세션을 다루는 규모가 되면 단일 VPS는 단일 장애점이 되기 때문에, 그때는 Docker 오케스트레이션이나 SSH로 여러 실행 환경을 묶는 쪽으로 판단을 바꿔야 한다고 생각합니다. 반대로 일회성 실험이나 GPU가 필요한 배치 작업이라면 Daytona나 Modal의 hibernate 특성이 오히려 비용 이점이 되는 상황도 있고요."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

Follow-up 1: "그럼 cron 예약 작업과 서버리스 backend를 같이 쓰고 싶다면 어떻게 설계하겠습니까?"

모범적인 사고 방향: cron 스케줄러 자체는 상시 가동되는 별도 컴포넌트(게이트웨이/코어 프로세스, Day 8)에 두고, 그 상시 컴포넌트가 예정된 시각에 서버리스 backend를 능동적으로 호출해서 실제 작업을 위임하는 구조로 분리한다. "항상 떠 있어야 하는 것(트리거)"과 "필요할 때만 뜨면 되는 것(실행 환경)"을 분리해서 생각하는 게 핵심이다. 다만 이건 공식 문서가 세부 명시한 내용이 아니라 구조적 추론임을 밝힌다.

Follow-up 2: "단일 VPS가 다운되면 세션/메모리는 어떻게 되나요? 복구 가능한가요?"

모범적인 사고 방향: VPS가 재부팅되어도 프로세스만 재시작되는 것으로는 부족하고, 세션과 메모리가 VPS의 영속 스토리지(디스크)에 저장되어 있어야 재개할 수 있다. 이건 Day 7(메모리/세션 설계)과 직접 연결되며, 단일 VPS 패턴을 쓸 때는 최소한 디스크 백업이나 스냅샷 전략이 필요하다고 답하는 것이 좋다.

Follow-up 3: "Docker backend와 SSH backend는 둘 다 '격리'라는 단어로 설명되는데, 실제로 어떻게 다른가요?"

모범적인 사고 방향: Docker는 하나의 호스트 안에서 프로세스/파일시스템 수준의 격리를 제공하고, SSH는 아예 물리적으로 다른 머신에 실행을 위임하는 것이므로 격리 단위 자체가 다르다. Docker는 "같은 서버 안에서 사고를 제한", SSH는 "신뢰할 수 있는 다른 서버의 정책을 그대로 물려받는" 방식이라는 점을 구분해서 설명하면 좋다. 즉 SSH backend의 보안 수준은 원격 서버 자체의 보안 수준에 전적으로 종속된다.

Follow-up 4: "비용만 보면 서버리스가 항상 더 싸 보이는데, 왜 always-on에는 VPS를 권장할까요?"

모범적인 사고 방향: 서버리스의 비용 이점은 "유휴 시간이 실제로 존재할 때"만 성립한다. always-on 에이전트는 정의상 "유휴 시간에도 뭔가 해야 하는" 워크로드이므로, 서버리스의 전제 조건 자체가 깨진다. 오히려 always-on 워크로드를 서버리스에 억지로 올리면 콜드 스타트 지연과 트리거 부재 문제로 숨은 비용(장애 대응, 신뢰성 저하)이 더 커질 수 있다고 답하는 것이 trade-off를 제대로 짚는 답이다.

좋은 답변 vs 약한 답변

차이를 명확히 보기

좋은 답변

  • 6개 backend를 격리/상시성/비용 축으로 분류해서 구조적으로 설명한다
  • hibernate-on-idle과 always-on이 정반대 목표라는 구조적 충돌을 명확히 짚는다
  • "$5 VPS" 패턴의 근거(가벼운 메모리, Docker+프로세스 매니저)와 유효 범위를 함께 말한다
  • 규모가 커지면 단일 VPS의 SPOF 위험까지 넘어가서 판단이 달라진다고 설명한다
  • 공식 문서에 없는 세부 메커니즘은 추론이라고 밝히고 넘어간다

약한 답변

  • "6개 backend가 있다"고 이름만 나열하고 선택 기준을 제시하지 못한다
  • "서버리스가 저렴하니까 무조건 서버리스가 좋다"고 단정한다
  • "$5 VPS면 항상 충분하다"며 스케일에 따른 한계를 언급하지 않는다
  • Docker와 SSH의 격리 차이를 "둘 다 안전하다" 수준으로 뭉뚱그린다
  • 모르는 내부 메커니즘을 공식 스펙인 것처럼 단정적으로 말한다
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "서버리스는 무조건 최신이고 좋은 기술이니 always-on에도 서버리스를 써야 한다."
서버리스의 비용 이점은 워크로드에 진짜 유휴 시간이 있을 때만 성립한다. always-on 에이전트처럼 유휴 시간에도 예약 작업을 수행해야 하는 워크로드에는 hibernate-on-idle이 오히려 구조적 리스크가 된다.
오해 2: "$5 VPS 패턴은 장난감 수준이라 실무에서는 안 쓴다."
실제로는 개인·소규모 팀 수준의 always-on 배포에서 가장 널리 쓰이는 검증된 기본 패턴이다. 가볍고 저렴하고 always-on 요구사항과 자연스럽게 맞아떨어진다. "장난감"이 아니라 "특정 스케일에 맞는 합리적 기본값"으로 이해해야 한다.
오해 3: "local backend와 로컬 노트북 배포는 같은 이야기다."
local backend는 "에이전트가 셸 명령을 실행하는 호스트가 (그 명령을 실행하는) 현재 머신"이라는 실행 격리 설정이고, 그 머신이 VPS일 수도 있다. 반대로 Day 1에서 다룬 "개인 노트북 배포는 always-on에 부적합하다"는 논의는 상시 가동 여부에 관한 것이라 별개의 축이다. 두 개념을 섞어서 "local backend = 노트북에서만 쓰는 것"이라고 오해하지 않아야 한다.
자기 점검 Rubric

내 답변 수준 확인하기

Weak "local, Docker, SSH, Daytona, Singularity, Modal이 있다"고 이름만 나열한다.
Junior backend별 용도는 대략 구분하지만, hibernate-on-idle과 always-on의 충돌을 스스로 짚어내지 못한다.
Mid 서버리스와 always-on의 충돌은 설명하지만, "$5 VPS" 패턴의 유효 범위(스케일 한계)까지는 연결하지 못한다.
Senior backend를 격리/상시성/비용 축으로 분류 → hibernate-on-idle과 always-on의 구조적 충돌 → "$5 VPS" 패턴의 근거와 한계 → 스케일이 바뀌면 판단도 바뀐다는 결론까지 하나의 논리로 연결하고, 추론과 사실을 구분해서 말한다.
핵심 요약

오늘 반드시 기억할 것

  • 실행 backend는 LLM 추론 위치가 아니라 셸 명령이 실행되는 물리적 환경을 결정한다
  • 6개 backend(local/Docker/SSH/Daytona/Singularity/Modal)는 격리·상시성·비용이라는 3축으로 분류해서 설명해야 구조적인 답이 된다
  • hibernate-on-idle(서버리스의 핵심 이점)과 always-on(항상 켜져 있어야 함)은 구조적으로 충돌하는 목표다
  • "$5 VPS + Docker + 프로세스 매니저"는 개인/소규모 always-on 배포의 검증된 기본 패턴이지만, 스케일이 커지면 단일 장애점 문제로 다중화 판단이 필요해진다
  • 일회성 실험·GPU 배치 작업처럼 진짜 유휴 시간이 있는 워크로드에는 오히려 서버리스(Daytona/Modal)가 합리적 선택이 된다 — "항상 이게 낫다"는 이분법이 아니라 워크로드 특성에 따른 trade-off다
