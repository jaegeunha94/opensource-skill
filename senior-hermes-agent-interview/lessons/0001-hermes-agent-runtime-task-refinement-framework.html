Day 1 · Hermes Agent Interview

Hermes Agent Runtime과 Task Refinement 운영 프레임워크

"항상 켜져 있는 에이전트"를 하나의 시스템으로 설계·운영하는 사고방식
버전/출처 기준: 이 레슨은 2026-07 기준 Nous Research의 Hermes Agent (github.com/NousResearch/hermes-agent, 최신 태그 v0.18.0 "The Judgment Release", 공식 문서 hermes-agent.nousresearch.com/docs/)를 기준으로 작성됐다. Nous Research의 이전 LLM 파인튜닝 체크포인트 시리즈("Hermes" 모델들)와는 다른 별개의 제품이니 혼동하지 않는다. 이 분야는 인터뷰 큐레이션 자료가 아직 없는 신생 영역이라, 공식 문서에 명시된 사실과 아키텍처상 합리적으로 추론한 내용을 구분해서 설명한다 — 실제 면접 전 공식 changelog로 재확인이 필요하다.
면접 질문
Q1 — 주력 질문 "Hermes Agent 같은 always-on 로컬 에이전트를, 그냥 반복문 안에서 LLM을 계속 호출하는 스크립트와 어떻게 다르게 설계해야 하나요? 런타임 아키텍처를 설명해주세요."
Q2 — 변형 "사용자가 '매일 아침 리포트 보내줘' 처럼 애매한 요청을 던졌을 때, 이걸 실행 가능한 작업으로 다듬는 과정(task refinement)을 어디서, 어떻게 설계하시겠습니까?"
Q3 — 변형 "로컬 우선(local-first) 에이전트를 운영한다고 할 때, 왜 개인 노트북이 아니라 별도 인스턴스에 배포해야 하나요? 그 판단 기준을 설명해주세요."
전제 개념

Always-on 에이전트란 무엇이고, 왜 "그냥 반복 호출"과 다른가

요청-응답(request-response)과 상시 실행(always-on)의 차이

일반적인 챗봇 API는 사용자가 메시지를 보내면 응답하고, 그 뒤로는 아무 일도 하지 않는다. 프로세스 입장에서는 "요청이 오면 깨어나서 답하고 다시 잠든다"는 구조다. 반면 always-on 에이전트는 사용자가 아무 말도 하지 않는 순간에도 프로세스가 계속 살아있다. 예약된 작업을 스스로 깨워서 실행하고 (Day 8에서 다룰 cron 스케줄링), 여러 채널(Telegram, Slack 등)에서 동시에 들어오는 메시지를 받아야 하고, 이전 세션의 기억을 이어받아야 한다. 즉 "대화 한 번"이 아니라 "계속 떠 있는 프로세스 하나"를 설계하는 문제로 바뀐다.

쉬운 비유로 설명하면, 요청-응답 챗봇은 "벨을 누르면 나오는 편의점 직원"이고, always-on 에이전트는 "24시간 상주하며 알아서 재고를 채우고 문제가 생기면 먼저 연락하는 매니저"에 가깝다. 매니저는 벨이 울리지 않아도 스스로 판단해서 움직여야 하므로, "언제 무엇을 할지"를 결정하는 별도의 상시 실행 구조가 필요하다.

단일 core loop, 다중 entry point — Hermes Agent의 AIAgent 구조

Hermes Agent 공식 아키텍처 문서에 따르면, CLI, 메신저 Gateway, IDE용 Agent Control Protocol(ACP), 대량 실행용 Batch 모드, API 서버까지 모든 진입점이 하나의 AIAgent 클래스를 공유한다. 이 클래스는 내부적으로 Prompt Builder → Provider Resolution → Tool Dispatch라는 파이프라인으로 구성된다.

  • Prompt Builder: 어느 채널에서 요청이 왔든(터미널 입력이든, Telegram 메시지든, cron이 깨운 예약 작업이든) 메모리, 이전 세션 요약, 사용자 모델 정보를 모아 모델에게 줄 컨텍스트를 조립한다.
  • Provider Resolution: 지금 이 요청을 어떤 LLM provider(Nous Portal, OpenRouter, OpenAI 등)로 보낼지 결정한다(Day 4에서 상세히 다룬다).
  • Tool Dispatch: 모델이 호출하려는 tool을 권한 검사 후 실제로 실행한다(Day 10의 7-layer defense와 연결된다).

이 구조의 핵심은 "진입점은 여러 개지만 판단 로직은 하나"라는 점이다. Telegram으로 들어온 요청과 CLI로 들어온 요청이 서로 다른 코드 경로를 타면, 똑같은 버그를 두 번 고쳐야 하고 권한 검사를 한쪽에서 빠뜨릴 위험이 생긴다. 채널마다 다른 코드가 아니라 채널은 얇은 어댑터, 판단은 코어 하나로 몰아넣는 것이 always-on 시스템에서 유지보수성과 보안을 동시에 지키는 방법이다.

여기서부터는 공식 문서에 세부 스펙이 없는 부분 — 아키텍처 추론: Hermes Agent 문서는 "task refinement"이라는 이름의 별도 컴포넌트를 명시하지 않는다. 다만 단일 AIAgent loop 구조상, "애매한 요청을 구체적인 작업으로 다듬는 일"은 Prompt Builder 단계에서 맥락을 채워 넣는 과정과 모델이 스스로 되묻거나(clarifying turn) 다음 tool 호출을 결정하는 반복(loop) 안에서 암묵적으로 일어난다고 보는 것이 합리적이다. 별도의 "planner 프로세스"가 존재한다고 단정하지 말고, 면접에서는 "공식적으로 이렇게 분리되어 있다"가 아니라 "이 구조라면 이렇게 동작할 것"이라고 추론 근거를 밝히며 답하는 것이 정직하고 시니어다운 태도다.

Task refinement을 아키텍처 관점에서 다시 정의하기

"Task refinement"을 그냥 "prompt를 잘 쓰는 것"으로 뭉뚱그리면 얕은 답이 된다. 시니어 수준에서는 이를 파이프라인의 어느 지점에서, 어떤 정보를 근거로, 무엇을 결정하는가로 쪼개서 설명해야 한다.

  1. 입력 정규화: 어떤 채널에서 왔든 동일한 내부 표현(메시지 + 메타데이터)으로 변환한다. 이게 안 되어 있으면 Telegram에서 온 요청과 cron이 깨운 요청을 같은 로직으로 다룰 수 없다.
  2. 맥락 주입: Prompt Builder가 이전 세션 요약, 사용자 모델(Day 7의 Honcho 기반 dialectic user modeling), 관련 메모리를 끌어와 "지금 이 요청이 어떤 맥락에서 나왔는지"를 채운다. 맥락이 없으면 모델은 "매일 아침 리포트"가 무슨 리포트를 말하는지 알 수 없다.
  3. 모호성 해소 지점 결정: 맥락을 넣어도 여전히 애매하면, 모델이 즉시 실행하지 않고 되묻는 턴을 만들지, 아니면 안전한 기본값으로 진행 후 결과를 보여주고 확인받을지를 tool의 위험도에 따라 다르게 판단해야 한다(읽기 전용 조회 vs 실제 메시지 발송은 위험도가 다르다).
  4. 실행 경계 설정: Tool Dispatch로 넘어가기 전에, 이번 작업이 손댈 수 있는 범위(어떤 tool, 어떤 채널, 어떤 파일/네트워크 권한)를 좁혀서 넘긴다 — 이건 Day 10의 권한 모델과 직결된다.

Local-first 실행 — "로컬에서 돈다"가 "24시간 켜져 있다"를 보장하지 않는다

Hermes Agent는 공식적으로 local, Docker, SSH, Daytona, Singularity, Modal이라는 6개 실행 backend를 제공한다(세부 trade-off는 Day 2에서 깊게 다룬다). 여기서 Day 1 수준에서 반드시 이해해야 할 핵심은 이것이다 — "local-first"는 "내 개인 노트북에서 실행"을 의미하지 않는다. 로컬 우선이라는 말은 클라우드 벤더의 관리형 서비스(SaaS 챗봇 플랫폼 등)에 종속되지 않고, 내가 통제하는 머신에서 실행된다는 뜻이지, 그 머신이 반드시 사용자의 개인 노트북이어야 한다는 뜻은 아니다.

노트북은 잠들거나(sleep), 재부팅되거나, 네트워크가 바뀐다. always-on을 노트북 위에 올리면 노트북이 꺼지는 순간 "24시간 상주 매니저"가 사라진다. 그래서 공식 문서는 저비용 VPS(문서상 예시로 "$5 VPS") 한 대를 전용으로 띄워, 개인 개발 머신과 분리하는 배포 패턴을 권장한다. 이건 "로컬 우선"이라는 철학과 "상시 가동"이라는 요구사항 사이의 실제 운영 판단이다 — 철학적으로는 로컬이지만, 운영상으로는 "내가 관리하는 전용 서버"가 곧 로컬의 의미가 된다.

배포 위치 always-on 적합성 비고
개인 노트북 부적합 절전/재부팅/네트워크 전환으로 프로세스가 끊긴다. 데모·개발 단계에는 적합.
전용 저비용 VPS (문서 권장 패턴) 적합 개인 개발 머신과 분리, 저렴한 비용으로 상시 가동 확보
Daytona / Modal (서버리스) 조건부 hibernate-on-idle(유휴 시 절전)이 기본이라 always-on 목표와 충돌 — cold start 지연 발생(Day 2)
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "Hermes Agent 설치해봤어요?"가 아니라 "상시 실행되는 시스템을 하나의 아키텍처로 사고할 수 있느냐"를 본다. 신생 프로젝트라서 후보가 실제 사용 경험이 없을 가능성이 높다는 걸 면접관도 안다 — 그래서 진짜로 보는 것은 개념을 처음 접했을 때 구조적으로 분해해서 이해하는 속도와 방식이다.

  • 패러다임 전환을 이해하는가: 요청-응답과 상시 실행 프로세스가 근본적으로 다른 운영 문제라는 것을 아는가
  • 여러 진입점 vs 단일 코어의 설계 감각이 있는가: "채널마다 다른 코드"의 위험성을 직관적으로 아는가
  • task refinement을 프롬프트 엔지니어링으로 뭉개지 않는가: 어느 파이프라인 단계에서 무슨 정보로 무엇을 결정하는지 구체적으로 쪼갤 수 있는가
  • "로컬"이라는 단어의 실무적 함의를 아는가: local-first가 개인 기기 실행을 뜻하지 않는다는 것, 상시 가동을 위해 배포 위치를 별도로 판단해야 한다는 것을 아는가
  • 모르는 것을 정직하게 구분하는가: 공식 문서에 없는 세부사항을 "추론"이라고 밝히며 답하는지, 아니면 마치 아는 것처럼 지어내는지 — 신생 기술 앞에서 시니어와 주니어의 차이가 여기서 드러난다
2026년 초에 공개된 신생 프로젝트이기 때문에, 면접관은 후보가 "Hermes Agent를 잘 안다"보다 "처음 보는 always-on 에이전트 아키텍처를 문서만 보고도 원리 수준으로 파악해서, 우리 회사의 운영 환경에 맞게 판단할 수 있는가"를 더 중요하게 본다. 빠르게 바뀌는 생태계에서 시니어의 가치는 특정 도구의 숙련도가 아니라 이 판단력에 있다.
실무 연결

실제 상황과 연결하기

상황 1: 개인 자동화 데모를 팀 도구로 승격시킬 때

개발자 개인 노트북에서 Hermes Agent를 돌려보며 데모가 잘 되었다고, 그대로 팀 Slack 채널에 연결해 "상시 비서"로 쓰겠다고 하면 문제가 생긴다. 노트북을 덮는 순간 팀 전체의 자동화가 멈춘다. → always-on 요구사항이 생기는 순간 배포 위치를 전용 인스턴스로 옮기는 결정을 아키텍처 리뷰에서 먼저 짚어낼 수 있어야 한다.

상황 2: 여러 메신저 채널에서 같은 에이전트를 쓰고 싶을 때

Telegram으로는 되는데 Slack으로는 권한 검사가 빠져있어 위험한 명령이 그대로 실행되는 사고가 난다면, 이는 channel-specific 코드가 core loop과 분리되지 않고 로직이 중복됐다는 신호다. → "채널은 얇은 어댑터, 판단은 단일 core" 원칙을 지켰는지 코드 리뷰에서 확인해야 한다.

상황 3: 애매한 반복 요청("매일 아침 요약 보내줘")을 처리할 때

"요약"이 무엇을 의미하는지(어제 하루 커밋 로그? 캘린더 일정? 특정 채널의 미확인 메시지?) 확인 없이 바로 cron에 등록해버리면, 매일 아침 엉뚱한 결과를 보내는 사고가 반복된다. → task refinement 단계에서 최초 1회는 반드시 되묻고, 이후 반복 실행에는 확정된 스코프를 재사용하는 설계가 필요하다(이 부분은 Day 8 스케줄링에서 더 깊게 다룬다).

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
Always-on의 정의부터 명확히 한다 요청-응답 모델과 달리, 사용자가 아무 말도 하지 않아도 프로세스가 계속 살아있고 스스로 트리거되는 시스템이라는 점을 먼저 짚는다.
2
단일 core loop + 다중 entry point 구조를 설명한다 Hermes Agent의 AIAgent 클래스가 CLI/Gateway/ACP/Batch/API를 하나의 Prompt Builder → Provider Resolution → Tool Dispatch 파이프라인으로 통일한다는 사실을 근거로 든다.
3
Task refinement을 파이프라인 단계에 위치시킨다 입력 정규화 → 맥락 주입 → 모호성 해소 지점 결정 → 실행 경계 설정, 이렇게 구체적 단계로 쪼개서 설명한다. 공식 문서에 명시된 것과 아키텍처상 추론한 것을 구분해서 말한다.
4
Local-first와 배포 토폴로지 판단으로 연결한다 "로컬"이 개인 기기를 뜻하지 않는다는 것, always-on을 위해서는 전용 인스턴스가 필요하다는 실무 판단을 근거를 들어 설명한다.
5
실패 시나리오로 마무리한다 채널별 코드 중복으로 인한 권한 검사 누락, 노트북 절전으로 인한 서비스 중단처럼 구체적 실패 사례를 들어 왜 이 설계 원칙이 필요한지 입증한다.
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Hermes Agent 같은 always-on 에이전트는 일반 챗봇과 달리 사용자가 아무 요청도 하지 않는 순간에도 프로세스가 살아있어야 합니다. 그래서 '요청이 오면 응답하고 끝'이 아니라 '계속 떠 있으면서 스스로 깨어나 판단하는 시스템'을 설계해야 합니다.

Hermes Agent는 이걸 하나의 AIAgent 코어 클래스로 풀어냅니다. CLI, 메신저 Gateway, IDE용 ACP, 배치 실행, API 서버까지 진입점은 다양하지만, 내부적으로는 전부 같은 Prompt Builder → Provider Resolution → Tool Dispatch 파이프라인을 탄다고 알고 있습니다. 이게 중요한 이유는, 채널마다 판단 로직이 갈라지면 한쪽에서는 권한 검사를 하고 다른 쪽에서는 빠뜨리는 식의 사고가 나기 쉽기 때문입니다.

Task refinement, 즉 애매한 요청을 실행 가능한 작업으로 다듬는 과정은 이 파이프라인 앞단에서 일어난다고 봅니다. 구체적으로는 입력을 정규화하고, Prompt Builder가 이전 세션과 사용자 모델에서 맥락을 끌어오고, 그래도 애매하면 tool의 위험도에 따라 되물을지 진행할지를 결정하고, 마지막으로 Tool Dispatch에 넘기기 전에 이번 작업이 건드릴 수 있는 범위를 좁혀서 넘기는 흐름일 것으로 추론합니다 — 다만 이건 공식 문서가 명시한 컴포넌트 이름이 아니라 아키텍처 구조에서 추론한 겁니다.

배포 관점에서는, '로컬 우선'이라는 말이 개인 노트북에서 돌리라는 뜻은 아니라고 생각합니다. 노트북은 절전되고 재부팅되기 때문에 always-on 목표와 맞지 않고, 실제로 공식 문서도 저비용 전용 VPS에 올리는 패턴을 권장합니다. '로컬'의 의미는 '클라우드 SaaS에 종속되지 않고 내가 통제하는 머신에서 돈다'는 것이지, '내 개인 기기여야 한다'는 게 아니라고 이해하고 있습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

Follow-up 1: "task refinement 단계에서 사용자에게 되물어야 할 시점을 어떻게 판단하나요?"

모범적인 사고 방향: 무조건 되묻거나 무조건 진행하는 이분법이 아니라, tool의 위험도(가역성) 기준으로 판단한다. 읽기 전용 조회는 애매해도 일단 실행 후 결과로 확인받는 편이 낫고, 메시지 발송이나 결제처럼 되돌리기 어려운 행동은 실행 전에 반드시 확인받아야 한다. 이건 Day 10의 권한 모델(위험 명령 승인 계층)과 직접 연결된다.

Follow-up 2: "always-on 에이전트가 VPS에서 도는데 네트워크가 끊기면 어떻게 복구되어야 하나요?"

모범적인 사고 방향: 프로세스 재시작(systemd 등)만으로는 부족하고, 끊긴 시점 이후의 예약 작업이나 진행 중이던 작업의 상태가 세션/메모리에 영속화되어 있어야 재개할 수 있다. 이건 Day 7(메모리/세션 설계)과 Day 8(스케줄링)이 왜 always-on 아키텍처의 핵심인지를 보여주는 지점이다.

Follow-up 3: "6개 실행 backend 중 Modal/Daytona 같은 서버리스를 쓸 때, hibernate-on-idle이 always-on 목표와 충돌하지 않나요?"

모범적인 사고 방향: 충돌한다는 걸 인정하고, 그럼에도 서버리스를 쓰는 경우가 있다면 그 이유(사용량이 적은 팀의 비용 절감, 완전한 상시성보다 온디맨드 응답이면 충분한 유스케이스)를 들어 trade-off로 설명한다. "무조건 이게 낫다"는 식의 이분법적 답은 약하다. 세부 수치와 비교는 Day 2에서 다룬다.

Follow-up 4: "여러 채널에서 동시에 들어온 요청을 always-on 에이전트가 어떻게 순서를 관리해야 하나요?"

모범적인 사고 방향: 단일 AIAgent 인스턴스가 모든 채널의 요청을 순차 처리한다면 한 채널의 긴 작업이 다른 채널을 막아버리는 head-of-line blocking이 생길 수 있다. 세션/대화 단위로 격리해서 동시성을 어떻게 확보할지, 아니면 의도적으로 순차 처리해서 일관성을 지킬지는 요구사항에 따라 달라지는 trade-off라고 답하는 것이 좋다.

좋은 답변 vs 약한 답변

차이를 명확히 보기

좋은 답변

  • always-on을 요청-응답과 대비시켜 "왜 다른 문제인지" 먼저 정의한다
  • 단일 core loop + 다중 entry point라는 구체적 구조를 근거로 든다
  • task refinement을 파이프라인 단계로 쪼개서 설명한다
  • 공식 문서에 있는 사실과 자신의 추론을 구분해서 말한다
  • "로컬"의 실무적 의미(전용 인스턴스 vs 개인 기기)를 정확히 짚는다

약한 답변

  • "Hermes Agent는 로컬에서 도는 AI 에이전트다"라는 표면적 설명에 그친다
  • task refinement을 "프롬프트를 잘 쓰면 된다"로 뭉뚱그린다
  • always-on을 "그냥 while True 루프에 sleep을 넣은 것"으로 단순화한다
  • 모르는 세부사항을 마치 공식 스펙인 것처럼 단정적으로 말한다
  • "로컬 우선 = 무조건 내 노트북에서 실행"이라고 오해한 채 답한다
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "로컬에서 실행되면 무조건 안전하고 always-on도 저절로 된다."
로컬 실행은 "데이터가 외부로 나가지 않는다"는 보안적 이점은 있지만, 상시 가동을 자동으로 보장하지 않는다. 개인 노트북은 절전·재부팅·네트워크 전환으로 언제든 끊긴다. always-on은 배포 위치(전용 VPS 등)를 별도로 설계해야 하는 문제다.
오해 2: "always-on 에이전트 = while True 루프에 sleep 걸어놓은 것."
단순 폴링 루프는 상태를 유지하지 않고, 여러 채널의 동시 요청을 다루지 못하며, 재시작 시 이전 맥락을 잃는다. 진짜 always-on 런타임은 세션 영속성, 다중 entry point 통합, 스케줄링, 장애 복구까지 포함하는 시스템이다.
오해 3: "task refinement은 결국 prompt engineering이다."
좋은 프롬프트 하나로 해결되는 문제가 아니다. 어떤 맥락을 주입할지(메모리 설계), 언제 되물을지(위험도 기반 판단), 실행 범위를 어떻게 제한할지(권한 설계)까지 포함하는 아키텍처 문제다. 프롬프트는 그 중 한 조각일 뿐이다.

Weak "Hermes Agent는 로컬에서 도는 자가학습 에이전트다"라고만 답하고 구조를 설명하지 못한다.
Junior always-on과 요청-응답의 차이는 알지만, task refinement을 프롬프트 문제로만 다룬다.
Mid 단일 core loop + 다중 entry point 구조는 설명하지만, local-first의 실무적 배포 판단까지 연결하지 못한다.
Senior always-on 정의 → 단일 core loop 구조 → task refinement의 파이프라인 위치 → local-first 배포 판단까지 하나의 논리로 연결하고, 공식 사실과 자신의 추론을 구분해서 설명하며, follow-up 질문에 trade-off로 답한다.
핵심 요약

오늘 반드시 기억할 것

  • Always-on 에이전트는 요청-응답 챗봇과 근본적으로 다른 운영 문제 — 프로세스가 상시 살아있어야 한다
  • Hermes Agent는 CLI/Gateway/ACP/Batch/API 진입점을 하나의 AIAgent 코어(Prompt Builder → Provider Resolution → Tool Dispatch)로 통일한다
  • Task refinement은 프롬프트 한 줄이 아니라, 입력 정규화 → 맥락 주입 → 모호성 해소 판단 → 실행 경계 설정으로 이어지는 파이프라인 문제다
  • "로컬 우선"은 개인 기기 실행이 아니라 "내가 통제하는 머신에서 실행"을 뜻하며, always-on을 위해서는 전용 인스턴스 배포가 실무적으로 필요하다
  • 신생 기술 앞에서는 공식 문서의 사실과 자신의 아키텍처적 추론을 구분해서 말하는 것 자체가 시니어의 신호다
