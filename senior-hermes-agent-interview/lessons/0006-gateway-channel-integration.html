Gateway와 채널 통합

Telegram·Discord·Slack·WhatsApp·Signal·WeCom… 20개가 넘는 채널을 하나의 프로세스가 어떻게 안전하게 감당하는가 — 팬아웃, 신뢰 경계, 세션 격리, webhook/polling 판단, 어댑터 보안

전제 개념

"하나의 게이트웨이, 스무 개가 넘는 문"부터 정리한다

지금까지 CLI(Day 1), ACP(Day 5) 같은 진입점을 다뤘다면, Gateway는 사람이 메신저 앱으로 Hermes와 대화하는 진입점이다. 공식 문서는 20개가 넘는 메시징 플랫폼을 나열한다 — Telegram, Discord, Slack, Google Chat, WhatsApp(개인용/Business Cloud API), Signal, SMS, Email, Home Assistant, Mattermost, Matrix, DingTalk, Feishu/Lark, WeCom, WeCom Callback, Weixin(개인 WeChat), BlueBubbles(iMessage), QQ, Yuanbao, Microsoft Teams, LINE, ntfy, Raft, IRC, 그리고 GitHub/GitLab 같은 외부 서비스를 받는 범용 Webhooks 어댑터까지. 공식 문서의 표현을 그대로 옮기면 "게이트웨이는 설정된 모든 플랫폼에 연결하고, 세션을 처리하고, cron 작업을 돌리고, 음성 메시지를 전달하는 단일 백그라운드 프로세스"다.

아키텍처 — 어댑터는 문일 뿐, 엔진은 하나다

공식 아키텍처 다이어그램을 요약하면 이렇다 — 각 플랫폼 어댑터(Telegram/Discord/Slack/…)가 원시 이벤트를 받아 세션 스토어(session store, 채팅방 단위)를 거쳐 AIAgent(run_agent.py)로 넘긴다. 게이트웨이는 동시에 cron 스케줄러도 돌리는데, 60초마다 tick하며 예약된 작업을 실행한다(Day 8에서 깊이 다룬다). 즉 Gateway는 새로운 에이전트 런타임이 아니라, Day 1에서 배운 "단일 core loop + 다중 entry point" 원칙의 또 하나의 진입점이다 — ACP가 에디터 안에 core loop를 얇게 감싼 것처럼, Gateway는 메신저 플랫폼들 앞에 core loop를 감싼 라우팅/세션 레이어다.

Gateway 개발자 문서(gateway-internals.md)는 메시지 처리 흐름을 4단계로 명시한다:

  1. 플랫폼 어댑터가 원시 이벤트를 받아 공통 포맷인 MessageEvent로 정규화한다 — Telegram의 Update 객체든 Discord의 Message 객체든, 어댑터를 통과하면 동일한 내부 구조가 된다. 이게 "채널별 페이로드 → 내부 공통 메시지 포맷" 변환의 실체다.
  2. Base adapter가 활성 세션 가드를 확인한다 — 이미 이 세션에서 에이전트가 실행 중이면 메시지를 큐에 넣고 interrupt 이벤트를 세팅한다. /approve, /deny, /stop 같은 명령은 이 가드를 우회해 즉시 처리된다.
  3. GatewayRunner._handle_message()가 세션 키를 계산하고, 권한을 확인하고, 슬래시 명령인지 판단한 뒤, 아니면 AIAgent 인스턴스를 생성해 대화를 실행한다.
  4. 응답이 다시 같은 플랫폼 어댑터를 통해 전달된다.

어댑터는 connect()/disconnect()/send()/on_message()라는 공통 인터페이스만 구현하면 된다 — on_message()가 원시 이벤트를 MessageEvent로 만들어 self.handle_message(event)를 호출하면, 그 다음부터는 어떤 플랫폼이었는지와 무관하게 동일한 게이트웨이 파이프라인을 탄다. 실제로 공식 문서는 신규 어댑터 추가를 "플러그인 디렉터리 하나 떨어뜨리기"로 설명할 정도로 어댑터 계층과 코어 로직이 명확히 분리돼 있다.

플랫폼마다 다른 툴셋 — "연결된다"가 "같은 권한을 준다"는 뜻은 아니다, 그런데 대부분은 맞다

각 플랫폼은 자기 전용 toolset을 갖는다(hermes-telegram, hermes-discord, hermes-slack…). 여기서 반드시 짚어야 할 점은, 문서에 명시된 표를 보면 Telegram/Discord/Slack/WhatsApp/Signal/SMS/Email 등 거의 모든 채팅형 플랫폼이 "터미널을 포함한 전체 툴"이라고 돼 있다는 것이다. 즉 일단 어떤 사용자가 Telegram DM으로 이 봇에 말을 걸 수 있는 상태가 되면, 그 사람은 사실상 CLI에서 Hermes를 쓰는 것과 거의 동급의 권한(터미널 실행 포함)을 갖게 된다. 예외는 있다 — Home Assistant는 ha_call_service 같은 전용 디바이스 제어 툴이 추가되고, API Server 진입점은 대화형 사용자가 없다는 이유로 clarify/text_to_speech가 빠진다. 하지만 원칙적으로 "채널에 연결된다"는 건 "강력한 권한을 받는다"와 거의 동의어라는 걸 먼저 인지해야, 다음 절의 신뢰 경계 논의가 왜 중요한지 납득이 간다.

신뢰 경계와 사용자 매핑 — 기본값은 "전부 거부"

공식 문서는 이 문장을 굵게 강조한다 — "기본적으로 게이트웨이는 allowlist에 없거나 DM으로 페어링되지 않은 모든 사용자를 거부한다. 이것이 터미널 접근 권한을 가진 봇의 안전한 기본값이다." 개발자 문서는 권한 판단을 5단계 순서로 명시한다:

  1. 플랫폼별 allow-all 플래그(예: TELEGRAM_ALLOW_ALL_USERS) — 설정돼 있으면 그 플랫폼의 모든 사용자를 허용
  2. 플랫폼 allowlist(예: TELEGRAM_ALLOWED_USERS=123456789,987654321) — 콤마로 구분된 사용자 ID
  3. DM 페어링 — 인증된 사용자가 페어링 코드로 새 사용자를 승인
  4. 전역 allow-all(GATEWAY_ALLOW_ALL_USERS) — 모든 플랫폼의 모든 사용자를 허용(터미널 접근이 있는 봇에는 권장하지 않는다고 문서가 명시)
  5. 기본값: 거부

DM 페어링은 allowlist를 수동으로 채우는 대신, 낯선 사용자가 봇에 DM을 보내면 일회성 페어링 코드(예: XKGH5N7P)를 받고, 관리자가 hermes pairing approve telegram XKGH5N7P로 승인하는 흐름이다. 코드는 1시간 후 만료되고, rate-limit이 걸려 있으며, 암호학적 난수를 쓴다. 이메일 채널만 예외인데, 명시적으로 활성화하지 않는 한 낯선 발신자의 이메일은 그냥 무시된다(이메일은 "누구나 보낼 수 있는" 채널 특성상 페어링 코드를 스스로 회신하는 흐름이 신뢰 모델에 안 맞기 때문으로 추론된다).

승인된 사용자 안에서도 다시 admin/일반 사용자 2단계가 있다. Allowlist가 "이 사람이 봇에 도달할 수 있는가"를 답한다면, admin/user 구분은 "일단 도달한 다음 무엇을 할 수 있는가"를 답한다. admin은 등록된 모든 슬래시 명령(내장 + 플러그인)을 실행할 수 있고, 일반 사용자는 명시적으로 허용한 명령만 실행할 수 있다(항상 열려 있는 최소 바닥은 /help, /whoami). 이 티어는 스코프(DM vs 그룹/채널)마다 별도로 설정된다 — DM에서 admin이라고 그룹에서도 admin인 건 아니다. allow_admin_from이 설정 안 돼 있으면 이 구분 자체가 꺼져서(하위 호환) 모든 허용된 사용자가 풀 액세스를 갖는다는 점도 기억해둘 만하다 — 즉 admin/user 분리는 opt-in 기능이다.

다른 레슨의 "allowlist"와 헷갈리지 말 것. 이 트랙에는 최소 세 가지 서로 다른 allowlist가 등장한다 — (1) 여기서 다루는 게이트웨이 사용자 allowlist는 "누가 이 에이전트에 말을 걸 수 있는가"를 결정하는 진입 경계다. (2) Day 5의 MCP 카탈로그는 "어떤 외부 tool 서버를 신뢰해서 연결할 것인가"를 결정하는 공급망 경계다. (3) Day 10에서 다룰 allow_once/allow_session/allow_always 승인 게이트는 "이미 연결된 사용자가 위험한 명령을 실제로 실행하게 둘 것인가"를 결정하는 실행 경계다. 셋 다 "화이트리스트"라는 단어를 쓰지만 막는 대상(사람/서버/명령)과 위치(진입/공급망/실행)가 다르다 — 면접에서 이 세 개를 뭉뚱그리면 구조를 모른다는 신호로 읽힌다.

세션 키 — "채널 간 대화 연속성"은 기본적으로 존재하지 않는다

Gateway 개발자 문서는 세션 키 포맷을 명시적으로 공개한다: 

agent:main:{platform}:{chat_type}:{chat_id}
# 예: agent:main:telegram:private:123456789

즉 세션은 플랫폼 + 채팅 타입 + 채팅 ID의 조합으로 식별된다. 이 키는 build_session_key()로만 생성해야 하고 수동 조립은 금지된다고 문서가 강조한다. 여기서 나오는 결론은 명확하다 — Telegram에서 시작한 대화는 기본적으로 Discord로 이어지지 않는다. 플랫폼이 다르면 세션 키가 다르고, 세션 키가 다르면 완전히 다른 대화다. 심지어 같은 플랫폼 안에서도 더 세분화된다 — Discord 문서는 기본 설정(group_sessions_per_user: true)에서 같은 채널 안의 Alice와 Bob이 서로 다른 세션을 갖는다고 명시한다("두 사람이 같은 `#research` 채널에서 말을 걸어도, Hermes는 기본적으로 완전히 다른 대화로 취급한다"). 이걸 끄면(false) 방 전체가 하나의 공유 세션이 되지만, 그 경우 한 사람의 툴 호출이 다른 사람의 컨텍스트를 부풀리거나, 한 사람의 진행 중인 작업이 다른 사람의 후속 메시지를 인터럽트할 수 있다는 트레이드오프가 있다.

이건 세션/메모리 설계(Day 7의 주제)와는 다른 층위의 이야기라는 걸 정확히 구분해야 한다 — Day 7에서 다룰 FTS5 전문 검색이나 dialectic user modeling은 "하나의 세션 안에서 과거 맥락을 어떻게 압축·검색·요약하는가"의 문제이고, 여기서 다루는 세션 키는 "애초에 어떤 메시지들이 같은 세션으로 묶이는가"를 결정하는 라우팅 경계다. 채널 간 연속성이 없다는 건 메모리 시스템의 한계가 아니라, 세션 키 설계상의 의도된 경계다.

Webhook vs Polling — 로컬-first 배포에서 이게 왜 실전 판단인가

채널마다 인바운드 연결 방식이 다르고, 이건 Day 2에서 배운 "$5 VPS" 패턴이 실제로 왜 동작하는지와 직결된다. 실제로 확인한 채널별 기본 동작은 다음과 같다:

채널기본 연결 방식인바운드 개방 필요?
TelegramLong polling(게이트웨이가 Telegram 서버로 아웃바운드 요청) 기본값은 불필요 — TELEGRAM_WEBHOOK_URL을 설정하면 선택적으로 webhook 모드 전환 가능
SlackSocket Mode(WebSocket, 아웃바운드 연결)불필요 — 공식 문서가 "Socket Mode는 공개 엔드포인트를 노출하지 않는다, 공격 표면이 하나 줄어든다"고 명시
Discorddiscord.py 게이트웨이(지속 WebSocket, 아웃바운드)불필요
Weixin(개인 WeChat)iLink Bot API long-polling(35초 타임아웃 롱폴) 불필요 — 문서가 "공개 엔드포인트, webhook, WebSocket 전혀 필요 없음"이라고 명시
WhatsApp(비공식 브리지)Baileys 브리지(WebSocket)불필요, 다만 계정 정지 위험을 안고 감

WeCom Callback인바운드 HTTP 서버(콜백)필요, 5초 응답 데드라인
범용 Webhooks 어댑터(GitHub/GitLab 등)인바운드 HTTP 서버(:8644) 필요

즉 대부분의 채팅형 메신저는 폴링이나 소켓 방식으로 아웃바운드 연결만 쓰기 때문에, $5 VPS가 방화벽 인바운드를 하나도 열지 않고도 always-on으로 동작할 수 있다 — 이게 Day 2에서 배운 패턴이 실제로 성립하는 이유의 절반이다. 문제는 webhook 방식이 필수인 소수의 채널이다 — WeCom Callback, WhatsApp Business Cloud API, MS Graph webhook, 그리고 GitHub/GitLab 이벤트를 받는 범용 Webhooks 어댑터. 이런 채널을 공인 IP도 포트포워딩도 없는 VPS/홈 서버에서 쓰려면 실무적으로 Cloudflare Tunnel/ngrok/Tailscale 같은 터널링을 앞단에 세우거나, 아예 공인 IP가 있는 VPS에 게이트웨이를 직접 바인딩해야 한다. 다만 이 부분은 공식 문서가 터널링을 표준 워크플로로 명시하고 있지는 않다 — GitHub 이슈 트래커에는 "Cloudflare Tunnel 컨트롤러 추가"(#8127), "SSH 기반 Pinggy 스킬로 제로 설치 터널"(#361) 같은 기능 제안이 아직 열려 있는 채로 남아 있어, 현재는 운영자가 직접 터널 솔루션을 구성해야 하는 수동 단계로 보인다 — 이건 추론이 섞인 판단이니 실제 도입 전 최신 이슈 상태를 재확인해야 한다.

Telegram의 경우는 이 트레이드오프를 문서가 직접 설명한다 — 기본값은 폴링이지만, Fly.io/Railway/Render 같은 서버리스형 배포(Day 2에서 다룬 hibernate-on-idle 백엔드와 결이 비슷하다)에서는 webhook 모드가 더 비용 효율적이다. 이런 플랫폼은 인바운드 HTTP 트래픽에는 자동으로 깨어나지만 아웃바운드 연결로는 깨어나지 않기 때문에, 폴링 봇은 계속 깨어 있어야 해서 "잠들 수 없는" 상태가 된다. webhook 모드는 방향을 뒤집어 Telegram이 봇의 HTTPS URL로 직접 push하게 만들어 idle 시 잠들 수 있게 해준다.

webhook을 켤 때 무엇이 잘못될 수 있는가 — 실제 보안 어드바이저리 사례

Telegram 문서는 TELEGRAM_WEBHOOK_URL을 설정할 때 TELEGRAM_WEBHOOK_SECRET가 필수이며 "게이트웨이는 이게 없으면 시작을 거부한다"고 명시하고, 그 근거로 실제 GitHub 보안 어드바이저리 GHSA-3vpc-7q5r-276h를 직접 인용한다. 이 secret은 Telegram이 매 webhook 요청마다 되돌려 보내는 검증용 토큰으로, 이게 강제되지 않으면 webhook URL을 추측하거나 알아낸 제3자가 Telegram인 척 위조된 업데이트를 게이트웨이에 밀어넣을 수 있는 구조적 위험이 있다고 추론할 수 있다(어드바이저리 원문 페이지 자체는 이번 리서치에서 직접 열람하지 못했고, 문서가 이 어드바이저리를 근거로 secret을 강제한다는 사실만 1차 확인했다). 범용 Webhooks 어댑터도 같은 원칙으로 방어한다 — 모든 라우트에 HMAC 시크릿이 필수이고(테스트용으로만 "INSECURE_NO_AUTH" 허용, 단 게이트웨이가 loopback에 바인딩된 경우에만 — 0.0.0.0 같은 비-loopback 바인딩과 조합하면 아예 기동을 거부한다), GitHub은 X-Hub-Signature-256, GitLab은 X-Gitlab-Token 평문 비교, 범용 V2는 HMAC-SHA256(타임스탬프+본문) 방식으로 재생 공격(replay)까지 막는다(레거시 V1은 재생 방지가 없어 사용 시 deprecation 경고가 뜬다).

Webhook 문서가 명시적으로 강조하는 원칙이 하나 더 있다 — "인증됐다는 것과 신뢰할 수 있다는 것은 다르다". HMAC 서명은 요청을 보낸 주체(예: GitHub)를 인증할 뿐, 그 안에 담긴 PR 제목·커밋 메시지·이슈 본문 같은 내용을 쓴 사람까지 보장하지 않는다 — 그건 임의의 제3자가 쓴 텍스트이고 신뢰할 수 없는 입력으로 다뤄야 한다. 문서는 이걸 블록리스트로 막을 수 없다고 못박고, "신뢰 경계는 입력 채널이 아니라 에이전트의 capability 표면에 있다"는 원칙으로 대응책을 제시한다 — 런타임 샌드박싱(Docker/SSH 백엔드), webhook 트리거 세션의 툴셋 축소(조회·요약만 필요하면 terminal/file 비활성화), 파괴적 작업에 대한 승인 유지, {__raw__} 대신 좁은 named-field 템플릿 사용. 이건 Day 11에서 다룰 prompt injection 방어와 정확히 같은 결의 원칙이다.

채널 어댑터 보안 실패 사례로 배우는 위협 모델링 — CVE-2026-7396

MISSION.md와 RESOURCES.md에 이미 등장한 CVE-2026-7396은 독립 취약점 데이터베이스 다수(OffSeq, SentinelOne, Tenable, VulDB 등)가 공통으로 지목하는 대상이 gateway/platforms/wecom.py(WeCom, 즉 기업용 WeChat Work 어댑터)라는 점에서 특히 시사점이 크다. 여러 2차 출처가 일관되게 서술하는 내용은 — 인증되지 않은 원격 공격자가 파일 경로 파라미터를 조작해 의도한 디렉터리 바깥의 파일에 접근할 수 있는 path traversal이고, 권한이나 사용자 상호작용이 전혀 필요 없으며 공격 복잡도가 낮다는 것이다. WeCom Callback은 위에서 본 대로 인바운드 HTTP 서버로 동작하고 AES 암호화된 미디어를 다루는 어댑터라서( adding-platform-adapters.md가 "AES crypto, multi-app" 패턴의 참고 구현으로 직접 지목한다), 미디어 다운로드/업로드 흐름에서 사용자가 통제 가능한 파일명/경로 조각을 검증 없이 파일시스템 연산에 그대로 흘렸을 가능성이 높다고 추론할 수 있다 — 다만 이건 공개 취약점 DB들의 일관된 서술을 근거로 한 추론이고, 패치 커밋의 정확한 diff까지 확인한 건 아니다. 오늘 다시 읽은 현재 weixin.md 문서(같은 계열의 개인 WeChat 어댑터)에는 "SSRF protection — 아웃바운드 미디어 URL은 다운로드 전 검증된다"는 항목이 명시돼 있어, v0.18.0 시점 문서에는 미디어 처리 계층 전반에 강화된 검증이 반영돼 있는 것으로 보이지만, "path traversal이 구체적으로 이렇게 패치됐다"는 changelog 문장은 이번 리서치에서 직접 찾지 못했다 — 이 지점은 명시적으로 추론임을 밝힌다.

이 사례에서 시니어가 새 채널 어댑터(내장이든 ~/.hermes/plugins/ 경로의 커뮤니티 플러그인이든)를 리뷰할 때 뽑아낼 수 있는 체크리스트는 셋이다:

  1. 입력 검증 — 사용자가 통제하는 파일명·경로·URL 조각이 파일시스템 연산이나 아웃바운드 요청에 그대로 쓰이는 지점이 있는가(path traversal, SSRF). Weixin 문서가 명시한 "아웃바운드 미디어 URL을 다운로드 전 검증한다"는 정확히 이 방어의 예시다.
  2. Rate limiting — Webhooks 어댑터가 라우트당 분당 30회 기본 제한을 걸고, Weixin이 메시지 ID 기준 5분 슬라이딩 윈도우로 중복 처리를 막는 것처럼, 콜백/webhook형 어댑터는 반드시 자체 rate limit과 idempotency(중복 배달 방지)를 가져야 한다 — 안 그러면 외부 서비스의 재시도 폭주가 곧 에이전트 실행 폭주로 이어진다.
  3. 자격증명 격리 — 토큰이 있는 어댑터는 스코프드 락(acquire_scoped_lock())으로 같은 토큰을 여러 프로필/게이트웨이 인스턴스가 동시에 쓰는 사고를 막아야 하고, 콜백형 어댑터는 HMAC/서명 시크릿 없이는 아예 기동을 거부해야 한다(웹훅 어댑터의 INSECURE_NO_AUTH + loopback 전용 제약이 이 원칙을 코드로 강제하는 예다).

채널별 장애 격리와 재시작 복구

운영 관점에서 중요한 질문은 "Discord 어댑터가 죽으면 Telegram도 같이 죽는가"다. 답은 아니다 — 각 어댑터는 circuit breaker로 감싸져 있다. 네트워크 순단, rate-limit 응답, 5xx, WebSocket 연결 끊김 같은 재시도 가능한 실패가 반복되면 breaker가 트립되어 해당 어댑터만 자동으로 일시정지(paused-by-breaker)되고, 다른 살아있는 플랫폼의 home channel로 운영자 알림이 가며, 구조화된 로그 라인이 남는다. 중요한 설계 판단은 breaker가 자동으로 재개하지 않는다는 점이다 — 지속적인 장애 상황에서 게이트웨이가 재연결을 계속 시도하며 스래싱(thrashing)하는 걸 피하려는 의도적 선택이고, 운영자가 /platform resume <name>으로 수동 재개해야 한다. /platform list로 각 어댑터가 running/ paused(수동)/paused-by-breaker(자동) 중 어떤 상태인지 확인할 수 있고, 일시정지는 어댑터를 언로드하지 않고 백그라운드 루프와 연결을 살려둔 채 인바운드만 버리기 때문에 재개가 즉시 이뤄진다.

게이트웨이 프로세스 자체가 재시작되는 경우도 별도로 방어돼 있다. 진행 중이던 tool 호출이나 생성이 있는 상태로 종료되면 해당 세션은 restart_interrupted로 표시되고, 다음 기동 시 자동 재개가 예약된다 — 사용자는 짧은 안내 메시지를 받고, 다음에 아무 메시지나 보내면 마지막으로 커밋된 turn부터 이어진다. 이 동작은 기본으로 켜져 있고 별도 설정이 필요 없다. 또한 각 플랫폼의 home channel로 "에이전트가 돌아왔다/중단됐었다" 알림을 한 번씩 보낼지는 gateway_restart_notification(플랫폼별, 기본 true)로 제어한다. 참고로 2026-07-01 v0.18.0 릴리스는 게이트웨이가 유휴 시 dormant 상태로 들어가고, 재시작/마이그레이션/ 자동 업데이트 전에는 진행 중인 대화를 끊지 않고 quiesce(정지 유예)한 뒤 scale-to-zero까지 가능하게 됐다고 서드파티 릴리스 요약이 전한다 — 이 부분은 릴리스 노트 원문을 직접 인용한 게 아니라 검색 스니펫 기준이라 실제 도입 전 재확인이 필요하다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • 진입점을 새 런타임으로 오해하지 않는가: Gateway를 별도 제품처럼 설명하지 않고, Day 1의 core AIAgent loop 위에 얹힌 라우팅/세션 레이어로 정확히 위치시키는가
  • "연결됨"과 "권한이 있음"을 구분하는가: 채널에 인증만 되면 대부분 터미널 포함 전체 툴셋을 받는다는 걸 인지하고, 그래서 기본값이 왜 "전부 거부"인지 설명할 수 있는가
  • 여러 층위의 allowlist를 뒤섞지 않는가: 게이트웨이 사용자 allowlist, MCP 카탈로그 allowlist, 명령 승인 게이트를 서로 다른 경계로 구분하는가
  • 세션 경계와 메모리 설계를 분리해서 이해하는가: "채널 간 대화가 안 이어진다"를 메모리 시스템의 한계가 아니라 세션 키 설계의 의도된 경계로 설명하는가
  • 인프라 판단력: polling과 webhook의 방향성 차이가 로컬-first 배포의 방화벽/공인 IP 제약과 어떻게 맞물리는지, 인바운드가 꼭 필요한 소수 채널을 어떻게 다뤄야 하는지 아는가
  • 공개된 실패 사례로 위협 모델링을 일반화하는가: 특정 CVE 하나를 암기하는 게 아니라, 거기서 "새 어댑터를 리뷰할 때 무엇을 봐야 하는가"라는 재사용 가능한 체크리스트를 뽑아내는가
실무에서 이 판단을 잘못하면, allowlist 없이 GATEWAY_ALLOW_ALL_USERS를 켠 채 터미널 권한이 있는 봇을 공개 채널에 노출하거나, webhook 시크릿을 생략한 채 인바운드 엔드포인트를 열어 임의의 제3자가 에이전트를 원격으로 트리거하게 만들거나, 어댑터 하나의 장애 대응을 자동 무한 재연결로 설계해 지속 장애 상황에서 게이트웨이 전체를 스래싱시키는 사고로 이어진다.
실무 연결

실제 상황과 연결하기

상황 1: 팀 Slack 채널에 봇을 붙였더니 아무나 터미널 명령을 실행할 수 있었다

데모용으로 GATEWAY_ALLOW_ALL_USERS=true로 급하게 붙인 봇을 그대로 운영 워크스페이스에 남겨뒀다가, 워크스페이스의 아무 멤버나 봇에게 셸 명령을 시켜 실행할 수 있는 상태가 됐다. → allowlist(SLACK_ALLOWED_USERS 등) 또는 DM 페어링으로 전환하고, admin/일반 사용자 티어를 켜서 /status, /model 같은 조회성 명령만 일반 사용자에게 열어준다. "전체 워크스페이스가 봇을 볼 수 있다"와 "전체 워크스페이스가 봇을 조종할 수 있다"는 완전히 다른 얘기라는 걸 설계에 반영해야 한다.

상황 2: 사용자가 "아까 Telegram에서 물어본 거 Discord에서 이어서 할게요"라고 한다

팀원이 모바일에서 Telegram으로 질문하다가 데스크톱 Discord로 넘어가면 대화가 끊긴다는 문의가 들어온다. → 이건 버그가 아니라 세션 키가 {platform}:{chat_type}:{chat_id}로 채널을 포함하기 때문에 발생하는 설계상 동작임을 설명한다. 진짜로 채널 간 연속성이 필요하면, 세션 자체를 공유하는 게 아니라 hermes mcp serve(Day 5)로 노출한 conversations_list/ messages_read 같은 조회 tool을 이용해 다른 에이전트나 대시보드가 여러 채널의 기록을 한 곳에서 참고하게 만드는 우회로를 검토해야 한다 — 세션 자체를 병합하는 기능이 있다는 근거는 찾지 못했다.

상황 3: 공인 IP 없는 홈서버에서 GitHub PR 리뷰 자동화를 붙이려 한다

Day 2의 "$5 VPS" 패턴대로 로컬 홈서버에 게이트웨이를 always-on으로 띄우고 Telegram으로 대화하는 건 문제없이 됐는데, GitHub PR이 열릴 때마다 자동 리뷰를 걸고 싶어서 범용 Webhooks 어댑터를 켰더니 GitHub이 로컬 서버에 도달하지 못한다. → Telegram/Slack/Discord는 전부 아웃바운드 폴링·소켓 연결이라 방화벽을 열 필요가 없었지만, GitHub webhook은 구조적으로 인바운드가 필요하다는 걸 구분해서 설명하고, Cloudflare Tunnel 같은 터널링 솔루션을 앞단에 세우거나 공인 IP가 있는 VPS로 옮기는 두 가지 선택지를 제시한다.

상황 4: 새로 붙인 사내 메신저 플러그인 어댑터가 콜백 방식인데 리뷰에서 뭘 봐야 할지 모르겠다

팀이 사내 메신저를 위해 커스텀 플러그인 어댑터를 만들었는데, WeCom Callback처럼 HTTP 서버를 직접 띄우는 콜백 방식이다. → CVE-2026-7396(WeCom path traversal) 사례를 근거로, 사용자가 넘긴 파일명/경로/URL이 파일시스템 연산이나 아웃바운드 요청에 검증 없이 쓰이는 지점이 있는지, HMAC/서명 시크릿 없이 기동 가능한지, rate limit과 idempotency가 있는지를 리뷰 체크리스트로 요구한다.

상황 5: Discord 어댑터가 장애로 계속 재연결을 시도하며 로그를 도배한다

Discord 쪽 이슈로 연결이 반복적으로 끊기는데, 게이트웨이가 계속 자동 재연결을 시도하면서 로그가 쌓이고 Telegram으로도 재시작 알림이 쏟아진다. → circuit breaker가 트립돼 자동 재개하지 않는 게 의도된 동작임을 설명하고, /platform list로 상태를 확인한 뒤 Discord 상태 페이지로 실제 장애가 끝났는지 확인, 끝났을 때만 /platform resume discord로 수동 재개하는 운영 절차를 세운다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
Gateway를 core loop의 진입점으로 위치시킨다 여러 플랫폼 어댑터가 원시 이벤트를 공통 MessageEvent로 정규화하고, 세션 스토어를 거쳐 AIAgent로 넘어가는 한 방향 파이프라인이라는 큰 그림을 먼저 그린다.
2
신뢰 경계를 "기본값은 거부"부터 설명한다 allowlist/allow-all/DM 페어링 5단계 판단 순서와 admin/일반 사용자 티어를 짚고, "연결됨 = 대부분 터미널 포함 전체 권한"이라는 전제를 먼저 깔아 왜 기본값이 엄격한지 정당화한다.
3
세션 키로 채널 간 격리를 설명한다 agent:main:{platform}:{chat_type}:{chat_id} 포맷을 근거로 채널 간 연속성이 기본적으로 없다는 것, 그리고 이게 메모리 설계(Day 7)와는 다른 층위의 라우팅 경계라는 것을 구분한다.
4
webhook vs polling을 로컬-first 배포와 연결한다 대부분의 채팅형 플랫폼이 아웃바운드 폴링/소켓이라 $5 VPS 패턴이 방화벽 없이 성립하지만, 소수의 webhook 필수 채널(WeCom Callback, WhatsApp Cloud API, 범용 Webhooks)은 인바운드가 필요해 터널링 같은 별도 조치가 필요하다는 점을 짚는다.
5
실제 CVE 사례로 위협 모델링을 일반화한다 CVE-2026-7396(WeCom path traversal)을 예로 들어 입력 검증/rate limiting/자격증명 격리라는 세 축의 재사용 가능한 체크리스트로 정리한다.
6
장애 격리와 재시작 복구로 마무리한다 어댑터별 circuit breaker(자동 정지, 수동 재개)와 게이트웨이 재시작 시 세션 자동 재개 (restart_interrupted)를 운영 관점에서 짧게 덧붙인다.
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Gateway는 Hermes가 메신저 앱으로 사람과 대화하는 진입점입니다. 20개가 넘는 플랫폼 어댑터가 각자의 원시 이벤트를 공통된 MessageEvent 포맷으로 정규화하고, 채팅방 단위의 세션 스토어를 거쳐 같은 core AIAgent로 넘깁니다. 즉 Gateway 자체는 새로운 에이전트 런타임이 아니라, ACP가 에디터를 감싼 것처럼 메신저 플랫폼들을 감싼 얇은 라우팅 레이어입니다.

신뢰 경계 쪽에서 중요한 전제는, 채널에 연결되는 순간 Telegram·Discord·Slack 같은 대부분의 채팅형 플랫폼이 터미널을 포함한 전체 툴셋을 받는다는 겁니다. 그래서 기본값이 "allowlist에 없거나 DM 페어링되지 않은 모든 사용자는 거부"로 엄격하게 잡혀 있고, 판단 순서는 플랫폼별 allow-all → 플랫폼 allowlist → DM 페어링 → 전역 allow-all → 기본 거부 순입니다. 승인된 사용자 안에서도 admin과 일반 사용자를 스코프(DM vs 그룹)별로 나눠서, 일반 사용자는 명시적으로 허용한 슬래시 명령만 실행하게 제한할 수 있습니다.

채널 간 대화 연속성은 기본적으로 없습니다. 세션 키가 agent:main:{platform}:{chat_type}:{chat_id} 형태로 플랫폼과 채팅 ID를 포함하기 때문에, Telegram에서 시작한 대화는 Discord로 자동으로 이어지지 않습니다. Discord 안에서도 기본 설정은 같은 채널 안 사용자별로 세션을 분리합니다. 이건 메모리 시스템의 한계가 아니라 세션 키 설계상 의도된 라우팅 경계라고 이해하는 게 맞다고 생각합니다.

인바운드 방식은 채널마다 다릅니다. Telegram 기본값·Slack Socket Mode·Discord 게이트웨이·Weixin long-polling은 전부 게이트웨이가 아웃바운드로 연결하는 방식이라, 방화벽 인바운드를 하나도 안 열어도 됩니다 — 이게 Day 2에서 다룬 $5 VPS always-on 패턴이 실제로 성립하는 이유의 절반입니다. 반면 WeCom Callback, WhatsApp Business Cloud API, GitHub/GitLab을 받는 범용 Webhooks 어댑터는 구조적으로 공개 HTTPS 엔드포인트가 필요해서, 공인 IP가 없는 로컬-first 배포라면 Cloudflare Tunnel 같은 터널링을 앞단에 세우거나 공인 IP가 있는 VPS를 따로 써야 합니다.

실제 보안 사례로는 CVE-2026-7396이 있습니다. WeCom(기업용 WeChat) 어댑터의 path traversal 취약점으로, 여러 취약점 데이터베이스가 wecom.py를 원인 파일로 지목하고 있고 인증 없이 원격에서 파일 경로를 조작할 수 있었다고 알려져 있습니다. 이 어댑터는 콜백형 HTTP 서버로 AES 암호화된 미디어를 다루는 구조라, 사용자 제어 가능한 경로 조각을 검증 없이 파일 연산에 흘렸을 가능성이 있다고 추론합니다. 여기서 시니어로서 뽑아낼 수 있는 원칙은 새 채널 어댑터를 리뷰할 때 입력 검증(경로/URL 조작 여부), rate limiting/idempotency, 자격증명 격리(스코프드 락, 시크릿 필수) 세 가지를 항상 확인해야 한다는 겁니다.

마지막으로 운영 관점에서는, 각 어댑터가 circuit breaker로 감싸져 있어서 한 채널의 장애가 다른 채널을 끌고 내려가지 않고, breaker는 자동으로 재개되지 않고 운영자가 수동으로 재개하도록 설계돼 있습니다 — 지속 장애 상황에서 게이트웨이가 재연결을 반복하며 스래싱하는 걸 막기 위한 의도적 선택 입니다. 게이트웨이 프로세스 자체가 재시작될 때는 진행 중이던 세션이 restart_interrupted로 표시되고 다음 기동 시 자동 재개가 예약되기 때문에, 재시작이 곧 대화 유실을 의미하지는 않습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

Follow-up 1: "게이트웨이 사용자 allowlist, MCP 카탈로그 승인, 위험 명령 승인 게이트 — 이 셋을 하나의 '보안 계층'으로 뭉뚱그려서 관리해도 되지 않나요?"

모범적인 사고 방향: 세 경계가 막는 대상과 발동 시점이 다르다는 걸 먼저 짚는다 — 게이트웨이 allowlist는 메시지가 도착하기 전에 사람을 걸러내고, MCP 카탈로그는 설치 시점에 외부 코드/네트워크 접근을 걸러내고, 명령 승인 게이트는 실행 직전에 이미 인증된 사용자의 특정 행동을 걸러낸다. 셋을 하나로 합치면, 예를 들어 "이미 allowlist를 통과한 신뢰된 팀원이라도 위험한 셸 명령 전에는 여전히 승인이 필요하다"는 계층적 방어(defense in depth)를 표현할 수 없게 된다는 점을 지적하면 좋다.

Follow-up 2: "Discord의 group_sessions_per_user를 false로 꺼서 방 전체가 하나의 세션을 공유하게 하면 어떤 새로운 리스크가 생기나요?"

모범적인 사고 방향: 편의성과 트레이드오프를 함께 짚는다 — 협업형 채널에서는 팀 전체가 같은 맥락을 보는 게 유용할 수 있지만, 문서가 명시한 대로 (1) 모든 사용자가 토큰 비용/컨텍스트 성장을 공유하고, (2) 한 사람의 툴 호출이 무거우면 다른 사람의 응답 품질까지 깎이고, (3) 한 사람의 진행 중인 요청이 다른 사람의 후속 메시지를 인터럽트할 수 있다. 여기에 더해 신뢰 관점에서, 공유 세션은 대화 히스토리 안에 여러 사람의 발화가 섞이기 때문에 한 사람이 세션 히스토리를 통해 다른 사람이 이전에 나눈 맥락을 간접적으로 들여다볼 수 있다는 프라이버시 함의도 있다고 짚으면 좋다.

Follow-up 3: "webhook 시크릿이 강제되지 않았다면 구체적으로 어떤 공격이 가능했을 것 같습니까?"

모범적인 사고 방향: webhook URL은 흔히 반영구적인 문자열이고(경로만 알면 됨), 시크릿 검증이 없으면 그 URL을 알아낸(로그 유출, 리버스 프록시 설정 실수, 소스코드 유출 등 경로는 다양하다) 제3자가 Telegram/GitHub인 척 위조 페이로드를 게이트웨이에 직접 POST할 수 있다는 시나리오를 제시한다. 게이트웨이 입장에서는 이게 실제 플랫폼에서 온 이벤트인지 구분할 수 없으므로, 위조된 "사용자 메시지"가 곧바로 에이전트 실행으로 이어질 수 있다 — 이게 바로 webhook 어댑터들이 시크릿 없이는 아예 기동을 거부하도록 설계된 이유라고 연결하면 좋다.

Follow-up 4: "circuit breaker가 자동으로 재개되지 않는 설계가 오히려 운영 부담을 늘리는 것 아닌가요? 왜 자동 재시도로 설계하지 않았을까요?"

모범적인 사고 방향: 자동 재시도의 함정을 구체적으로 짚는다 — 플랫폼이 지속적으로 불안정한 상태 (예: Discord 쪽의 장시간 장애)에서 게이트웨이가 계속 재연결을 시도하면, 매 시도가 새로운 실패로 이어지며 로그를 도배하고, 연결 시도 자체가 rate-limit을 더 악화시킬 수 있고, 다른 정상 채널의 운영자 알림 채널까지 노이즈로 채워버릴 수 있다. 수동 재개는 "사람이 원인을 확인하고 실제로 복구됐다는 걸 검증한 뒤에만 재시도한다"는 안전장치이고, 이건 무제한 자동 재시도보다 운영 부담이 크지만 스래싱으로 인한 2차 장애를 막는 트레이드오프라고 설명하면 좋다.

Follow-up 5: "CVE-2026-7396 같은 어댑터 취약점을 사전에 잡으려면, 코드 리뷰 체크리스트 말고 아키텍처 수준에서 할 수 있는 방어는 뭐가 있을까요?"

모범적인 사고 방향: 코드 리뷰는 사람이 놓칠 수 있다는 전제에서 출발해 계층적 방어를 제시한다 — Webhooks 문서가 강조한 원칙처럼 "신뢰 경계를 입력 채널이 아니라 capability 표면에 둔다"는 관점에서, 콜백/webhook형 어댑터가 처리하는 세션은 애초에 terminal/file 같은 고위험 툴을 기본적으로 제외하고 필요한 조회 툴만 열어주는 식으로 툴셋을 좁히거나, 컨테이너/VM 샌드박스 안에서만 그런 어댑터를 운영하는 걸 아키텍처 수준 방어로 들 수 있다. 즉 "이 어댑터 코드가 완벽하다"에 기대는 대신, "이 어댑터가 뚫려도 블라스트 반경이 작다"를 설계 목표로 삼는 게 시니어다운 답변이라고 정리하면 좋다.

좋은 답변 vs 약한 답변

차이를 명확히 보기

좋은 답변

  • Gateway를 새 런타임이 아니라 core AIAgent loop의 진입점으로 정확히 위치시킨다
  • "연결됨 = 대부분 전체 툴셋 권한"이라는 전제부터 짚고 기본값이 왜 엄격한지 설명한다
  • allowlist 5단계 판단 순서, DM 페어링, admin/user 티어를 구체적으로 구분한다
  • 세션 키 포맷을 근거로 채널 간 연속성이 없다는 걸 설명하고, 이걸 메모리 설계와 다른 층위로 구분한다
  • polling/webhook의 방향성 차이를 로컬-first 배포의 방화벽 제약과 실무적으로 연결한다
  • CVE 사례를 암기가 아니라 재사용 가능한 위협 모델링 체크리스트로 일반화한다
  • 사실(문서에 명시된 것)과 추론(2차 출처로 교차검증한 것)을 구분해서 말한다

약한 답변

  • "게이트웨이가 여러 메신저를 연결해준다"는 한 줄로 끝내고 core loop와의 관계를 설명 못한다
  • allowlist를 언급하지 않고 "봇 토큰만 있으면 다 된다"는 식으로 신뢰 경계를 간과한다
  • 채널 간 대화가 왜 안 이어지는지 몰라서 "버그인 것 같다"고 답한다
  • 모든 채널이 다 webhook 방식이라고 오해하거나, 반대로 다 polling이라고 단정한다
  • CVE를 "취약점이 있었다더라" 수준으로만 알고 있고 거기서 뽑아낼 원칙을 제시 못한다
  • circuit breaker나 세션 재개 같은 운영 디테일을 전혀 모른다
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "봇을 특정 Telegram/Discord 채널에만 초대하면 그 자체로 접근이 제한된다."
채널 초대는 "누가 이 봇을 볼 수 있는가"를 정할 뿐, 게이트웨이의 사용자 allowlist가 별도로 없으면 그 채널에 있는 아무나 봇에게 말을 걸 수 있고, 기본 툴셋이 터미널을 포함한다는 걸 감안하면 이건 곧 임의 사용자에게 터미널 접근을 열어주는 것과 같다. 채널 멤버십과 게이트웨이 인가는 서로 다른 경계다.
오해 2: "메모리/세션 시스템이 좋아지면 언젠가 채널 간 대화도 자동으로 이어질 것이다."
세션 키가 플랫폼을 포함하는 건 메모리 압축·검색 기술의 한계가 아니라 라우팅 설계상의 의도된 경계다. Day 7에서 아무리 정교한 dialectic user modeling이 붙어도, 그건 "한 세션 안에서 과거를 어떻게 요약·검색하는가"의 문제이지 "서로 다른 채널의 메시지를 하나의 세션으로 합칠 것인가"의 문제가 아니다. 후자가 필요하면 세션 병합이 아니라 MCP server의 조회 tool 같은 별도 우회 경로를 검토해야 한다.
오해 3: "webhook 방식이 polling보다 항상 더 낫다(더 실시간이고 효율적이다)."
로컬-first/always-on 배포에서는 오히려 반대다. Telegram 문서가 명시하듯 webhook은 서버리스형 플랫폼(Fly.io 등)에서 idle 시 잠들 수 있게 해주는 게 핵심 장점이고, always-on VPS에서는 굳이 공개 HTTPS 엔드포인트와 시크릿 관리, 터널링 부담을 새로 지는 것보다 기본 polling이 더 단순하고 공격 표면도 작다(Slack Socket Mode 문서가 직접 "공개 엔드포인트가 없다 = 공격 표면이 하나 줄어든다"고 명시한다). "무엇이 낫다"가 아니라 "배포 토폴로지에 어떤 방식이 맞는가"로 판단해야 한다.
자기 점검 Rubric

내 답변 수준 확인하기

Weak Gateway를 "메신저 연동 기능" 정도로만 설명하고, core loop와의 관계, 신뢰 경계, 세션 격리를 전혀 구분하지 못한다.
Junior 여러 플랫폼을 연결할 수 있다는 건 알지만, allowlist/DM 페어링 같은 인가 메커니즘이나 webhook/polling의 방향성 차이는 설명하지 못한다.
Mid 인가 메커니즘과 세션 키 격리는 설명할 수 있지만, 이걸 로컬-first 배포 토폴로지(Day 2)나 실제 CVE 사례 같은 운영 판단으로 연결하지 못하고, 여러 층위의 allowlist를 뒤섞어 말한다.
Senior Gateway(core loop의 진입점) → 신뢰 경계(allowlist 5단계, admin/user 티어, "연결=권한" 전제) → 세션 격리(채널별 세션 키, 메모리 설계와의 구분) → webhook/polling과 배포 토폴로지 → 실제 CVE 기반 위협 모델링 체크리스트 → 어댑터별 장애 격리·재시작 복구까지 하나의 논리로 연결하고, 사실과 추론을 구분해서 말한다.
핵심 요약

오늘 반드시 기억할 것

  • Gateway는 20개가 넘는 플랫폼 어댑터가 원시 이벤트를 공통 MessageEvent로 정규화해 같은 core AIAgent loop로 넘기는, 단일 백그라운드 프로세스이자 새 진입점이다
  • 기본값은 "allowlist에 없거나 DM 페어링 안 된 모든 사용자 거부"이며, 채널에 연결된다는 건 대부분 터미널 포함 전체 툴셋 권한을 받는다는 뜻이다
  • 세션 키는 agent:main:{platform}:{chat_type}:{chat_id} 형태로 플랫폼을 포함하므로, 채널 간 대화 연속성은 기본적으로 없다 — 이는 메모리 설계와는 다른 층위의 라우팅 경계다
  • 대부분의 채팅형 플랫폼(Telegram 기본값, Slack Socket Mode, Discord, Weixin)은 아웃바운드 polling/소켓 방식이라 방화벽 인바운드가 필요 없지만, WeCom Callback·WhatsApp Cloud API·범용 Webhooks는 인바운드 webhook이 필수라 로컬-first 배포에서는 터널링 같은 별도 조치가 필요하다
  • CVE-2026-7396(WeCom 어댑터 path traversal)은 입력 검증·rate limiting·자격증명 격리라는 세 축의 어댑터 리뷰 체크리스트로 일반화할 수 있다
  • 각 어댑터는 circuit breaker로 격리돼 있어 한 채널의 장애가 다른 채널을 끌고 내려가지 않고, breaker는 의도적으로 자동 재개하지 않는다. 게이트웨이 재시작 시 진행 중 세션은 restart_interrupted로 표시돼 자동 재개가 예약된다
  • Gateway 사용자 allowlist(진입 경계), MCP 카탈로그(공급망 경계), 명령 승인 게이트(실행 경계)는 서로 다른 층위의 신뢰 경계이며 뭉뚱그리면 안 된다
