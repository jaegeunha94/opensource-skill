1. 핵심 면접 질문

Primary Question

"운영 중인 API 응답 속도가 갑자기 느려졌다는 보고를 받았습니다. 어떻게 접근하시겠어요?"

Variant 1

"성능 최적화를 요청받았을 때 가장 먼저 하는 일이 무엇인가요?"

Variant 2

"특정 API의 p99 latency가 5초를 넘고 있습니다. DB, 캐시, 앱 서버 중 어디를 먼저 보시겠어요? 그 이유는?"

2. 왜 면접에서 묻는가

이 질문은 지식 테스트가 아니라 사고 과정 테스트입니다. 면접관이 보고 싶은 것은 "인덱스 추가하면 됩니다"가 아니라, 데이터 없이 결론 내리지 않는 엔지니어의 체계적 사고입니다.

면접관이 이 질문으로 평가하는 것:

  • 추측 vs 측정 — "아마 DB일 것 같아서 인덱스를 추가했어요"라고 말하는 사람은 운이 좋으면 맞고 나쁘면 틀린다. 시니어는 데이터로 병목을 특정한다.
  • 병목 위치 추론 능력 — 문제가 DB인지, 앱 코드인지, 외부 API인지, 네트워크인지를 체계적으로 좁혀가는 능력.
  • 전체 시스템 시야 — 단일 함수만 보는 것이 아니라 request 경로 전체를 그릴 수 있는가.
  • 우선순위 판단 — 여러 병목이 동시에 있을 때 어디서 최대 효과를 낼 수 있는지 판단하는 능력.
  • 개선 검증 방법 — "개선했는데 어떻게 빨라졌다는 걸 알았나요?"에 답할 수 있는가.

10년차 이하 답변의 전형적인 패턴: "캐시를 추가하거나 인덱스를 보겠습니다." 이 답변은 방향이 옳을 수도 있지만 근거가 없어서 면접관 입장에서는 운에 맡기는 엔지니어로 보입니다.

3. 전제 개념 — 쉬운 한국어로

병목(Bottleneck)이란

병목은 물리적인 병 목처럼, 전체 흐름 중에서 가장 느린 구간입니다. 아무리 나머지를 빠르게 해도 병목이 해소되지 않으면 전체 처리 속도는 올라가지 않습니다.

예: API가 총 2,000ms 걸리는데 DB 쿼리가 1,800ms라면, 앱 코드를 10배 최적화해도 200ms → 20ms로 줄어들 뿐, 전체는 1,820ms가 됩니다. 병목(DB)을 먼저 잡아야 합니다.

Latency와 Throughput의 차이

Latency(지연 시간): 요청 하나가 처리되어 응답이 오기까지 걸린 시간. "이 API 응답이 몇 ms야?"

Throughput(처리량): 단위 시간당 처리할 수 있는 요청 수. "1초에 몇 건 처리할 수 있어?"

둘은 독립적으로 나빠질 수 있습니다. Latency는 낮지만 Throughput도 낮은 경우 (직렬 처리), 또는 Throughput은 높지만 개별 Latency가 높은 경우 (배치 처리)가 있습니다.

p99 latency란

100건의 요청을 응답시간 기준으로 정렬했을 때 99번째 요청의 응답시간. 즉 "상위 1% 느린 요청의 속도"입니다.

평균(mean)은 아웃라이어에 왜곡됩니다. p99는 실제 느린 사용자 경험을 더 잘 나타냅니다. 면접에서 "평균 응답시간"을 말하면 p95/p99 차이를 이해하는지 질문이 이어집니다.

Profiling이란

코드를 실행하면서 어떤 함수/구간이 시간을 얼마나 쓰는지 기록하는 도구. 의사가 청진기로 어디가 아픈지 짚어내는 것과 같습니다. Profiling 없이 추측으로 코드를 수정하는 것은 위치도 모르고 수술하는 것과 같습니다.

Amdahl의 법칙 (Amdahl's Law)

전체 작업 중 최적화 가능한 부분의 비율이 병렬화/최적화의 실제 효과 상한을 결정합니다. 예: 전체 시간의 20%만 병렬화 가능하다면, 아무리 많은 코어를 써도 전체 속도는 25% 이상 빨라지지 않습니다.

실무 함의: "여기를 10배 빠르게 만들었어요!"라고 할 때, 그 구간이 전체의 5%라면 전체 성능 향상은 5% 미만입니다. 병목이 아닌 곳을 최적화한 것입니다.

4. 5단계 병목 분석 프레임워크

시니어가 성능 문제를 받았을 때 실제로 따르는 순서입니다. 면접에서 이 흐름대로 답변하면 "체계가 있는 엔지니어"로 평가받습니다.

  • 1
    증상 구체화 (Symptom Clarification)

    "느리다"는 주관적입니다. 먼저 수치로 만듭니다. "언제부터, 어떤 API에서, 어느 percentile이, 몇 ms인가?" 배포 시점과 교차 확인합니다. 특정 사용자/지역에서만 느리다면 CDN이나 네트워크 문제일 수 있습니다.

    질문 예시: "slow 기준이 뭔가요? p99 5초인가요, 평균 500ms인가요? 언제부터 느려졌나요?"

  • 2
    메트릭 수집 (Metric Collection)

    Grafana, Datadog, CloudWatch 등 모니터링 도구에서 문제 시점의 다음 메트릭을 확인합니다:

    • API 응답시간 (p50 / p95 / p99)
    • Error rate (5xx 증가 여부)
    • Throughput (RPS 변화)
    • 앱 서버 CPU / 메모리 / Thread pool 사용률
    • DB: 슬로우 쿼리 발생, Connection pool 사용률, DB CPU
    • 캐시: Hit rate 변화
    • 외부 API: 응답시간 변화
  • 3
    병목 위치 특정 (Bottleneck Localization)

    Distributed tracing (Jaeger, Zipkin, Datadog APM)이 있다면 Span 기준으로 어느 레이어가 시간을 먹는지 바로 보입니다. 없다면 로그의 타임스탬프를 기준으로 단계별 소요 시간을 계산합니다.

    판단 기준: "전체 2,000ms 중 DB Span이 1,800ms면 DB 병목. 앱 코드 200ms 중 특정 함수가 150ms면 코드 병목."

  • 4
    원인 가설 수립 및 검증 (Hypothesis and Verification)

    병목 위치를 확인했으면 원인 가설을 세우고 각각을 검증합니다.

    • DB 병목이라면: EXPLAIN ANALYZE로 실행 계획 확인 → 인덱스 미사용, N+1, 풀 테이블 스캔 여부
    • 앱 코드 병목이라면: Profiler로 핫 함수 식별
    • I/O 병목이라면: iostat, iotop으로 디스크 포화도 확인
    • 동시성 병목이라면: Thread pool 크기, lock contention 확인
  • 5
    개선 후 수치 검증 (Post-fix Validation)

    "개선했으니 빠르겠지"가 아닙니다. 같은 메트릭으로 before/after를 비교합니다. 부하 테스트 환경에서 재현 가능한지 확인합니다. 개선이 다른 지표(Error rate, DB 부하)를 악화시키지 않았는지 확인합니다.

5. 병목 유형 분류

병목 위치를 특정하는 데 도움이 되는 분류 기준입니다.

유형 주요 신호 확인 방법 흔한 원인
DB I/O DB CPU 높음, 슬로우 쿼리 증가, DB connection 대기 slow query log, EXPLAIN ANALYZE, pg_stat_statements 풀 테이블 스캔, N+1, 인덱스 미사용, 잠금 경합
앱 CPU 앱 서버 CPU 100%, GC 빈번, Profiler에서 핫 함수 py-spy, cProfile, flame graph 비효율 알고리즘, 직렬화 오버헤드, 정규식 남용
네트워크 / 외부 API 외부 API Span 시간 증가, timeout 발생, DNS latency Distributed tracing, 외부 API 헬스체크, curl 벤치마크 외부 서비스 장애, 커넥션 미재사용, DNS TTL 짧음
동시성 / 잠금 CPU는 낮은데 응답이 느림, thread pool 포화, DB deadlock thread dump, DB lock 쿼리, async profiler thread pool 부족, DB row lock, GIL 병목 (Python)
메모리 / GC JVM/Python GC pause 빈번, OOM, 메모리 증가 추세 GC log, memory profiler, heap dump 메모리 누수, 대량 객체 생성, 캐시 미한도 설정
I/O 포화 디스크 I/O wait 높음, 로그/파일 쓰기 지연 iostat, iotop, vmstat 로그 동기 쓰기, 동기 파일 I/O를 메인 스레드에서 실행

6. USE Method & RED Method

면접에서 "어떤 방법론으로 성능을 모니터링하나요?"라고 물으면 이 두 가지를 언급하면 됩니다.

USE Method (Brendan Gregg, Netflix)

시스템 리소스 관점에서 병목을 찾는 방법론입니다. CPU, 메모리, 디스크, 네트워크 등 모든 리소스에 대해 세 가지를 확인합니다.

지표설명예시 질문
Utilization 리소스가 얼마나 바쁜가 (사용률 %) CPU 사용률이 95% 이상인가?
Saturation 리소스가 처리 못 하고 쌓아두는 작업이 있는가 (큐 길이) DB connection queue에 대기 중인 요청이 있는가?
Errors 오류가 발생하고 있는가 5xx 에러, connection reset, timeout 발생 중인가?

언제 씁니까? 인프라/리소스 관점에서 병목을 좁힐 때. "CPU, 메모리, 디스크, 네트워크 중 어디가 포화(Saturation) 상태인가?"를 체계적으로 확인합니다.

RED Method (Tom Wilkie, Grafana)

서비스/API 관점에서 사용자 영향을 모니터링하는 방법론입니다.

지표설명예시 메트릭
Rate 초당 요청 수 (RPS) requests_total / 초
Errors 실패 비율 5xx 응답 비율
Duration 요청 처리 시간 분포 request_duration_seconds의 p50 / p95 / p99

언제 씁니까? 대시보드를 설계하거나 SLO를 정의할 때. "이 서비스 정상 여부를 세 가지 숫자로 본다면 무엇을 볼 것인가?"에 RED를 씁니다.

USE vs RED 조합 방법

RED로 "뭔가 이상하다"를 탐지하고 (Duration 급증, Error 증가), USE로 "왜 이상한가"를 추적합니다 (CPU 포화? DB connection saturation?).

7. 답변 사고 순서

면접에서 이 순서대로 말하면 됩니다. 30초 정도의 시간을 줘도 됩니다.

  1. 먼저 증상을 수치로 정의합니다. "어느 API에서 언제부터 어느 정도 느려졌는지" 확인합니다. 배포 이력, 트래픽 증가 여부도 함께 확인합니다.
  2. 메트릭을 봅니다. 모니터링 도구에서 latency, error rate, throughput 추이를 봅니다. 앱 서버, DB, 외부 API 각각의 CPU / 연결 수 / 응답시간을 확인합니다.
  3. Distributed tracing으로 Span을 분해합니다. 2,000ms 중 어느 레이어가 어느 시간을 쓰는지 Span 단위로 특정합니다.
  4. 병목 위치에 맞는 도구로 원인을 좁힙니다. DB 병목이면 EXPLAIN, 코드 병목이면 Profiler, 외부 API 병목이면 dependency health check.
  5. 개선 후 before/after 수치로 검증합니다. 같은 부하 테스트 시나리오로 재실행해서 개선이 실제로 수치에 반영되었는지 확인합니다.

8. 핵심 답변 예시

면접관: "운영 API 응답 속도가 갑자기 느려졌다는 보고를 받았습니다. 어떻게 접근하시겠어요?"

STRONG ANSWER

먼저 "느리다"를 수치로 만듭니다. 어느 API에서, 어느 percentile이, 언제부터 몇 ms가 됐는지 모니터링 대시보드에서 확인합니다. 배포 이력이 있는지도 함께 봅니다.


메트릭을 확인했으면 distributed tracing으로 request 경로를 분해합니다. 전체 응답 시간 중 DB Span, 외부 API Span, 앱 코드 Span이 각각 몇 %인지 보면 병목 위치가 바로 보입니다.


예를 들어 2,000ms 응답 중 DB Span이 1,800ms라면 DB 병목입니다. 이때는 슬로우 쿼리 로그와 EXPLAIN ANALYZE를 확인해서 풀 테이블 스캔인지, N+1 문제인지, 락 경합인지 좁힙니다.


반대로 DB는 정상인데 앱 서버 CPU가 100%라면 Profiler를 돌려서 핫 함수를 찾습니다. 코드 수정 후에는 동일한 부하 테스트 시나리오로 p99 latency가 실제로 개선됐는지 before/after로 검증합니다.


제가 중요하게 생각하는 것은 '추측하지 않는다'는 겁니다. 데이터 없이 캐시나 인덱스를 먼저 추가하면, 운이 좋으면 맞고 나쁘면 잘못된 방향에 시간을 씁니다.

이 답변이 좋은 이유:

  • 5단계 프레임워크를 자연스럽게 녹임
  • Distributed tracing을 언급 → 현장 도구 이해를 보여줌
  • 구체적인 수치(2,000ms / 1,800ms)를 예로 들어 설명
  • "추측하지 않는다"는 원칙을 명시적으로 말함
  • 검증 단계(before/after)까지 포함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"일단 캐시를 추가해보고, DB에 인덱스가 있는지 확인하고, 코드에 불필요한 로직이 있는지 보겠습니다."

이 답변이 약한 이유

  • 데이터 수집이 없습니다. 어디가 병목인지 모르는 상태에서 해결책을 나열했습니다.
  • "캐시 추가"는 DB가 병목이 아닐 때 효과가 없습니다. 잘못된 병목에 투자하는 것입니다.
  • 측정 → 특정 → 개선 → 검증 순서가 없습니다.
  • p99, distributed tracing 등 시니어 수준의 도구 언급이 없습니다.
  • 면접관이 "왜 캐시를 먼저 보나요?"라고 물으면 방어할 근거가 없습니다.
WEAK ANSWER — 또 다른 패턴

"EXPLAIN을 돌려보고 인덱스가 없으면 추가하겠습니다. 그리고 슬로우 쿼리 로그도 확인하겠습니다."

이 답변이 약한 이유

  • 병목이 DB라는 가정을 먼저 했습니다. 실제 병목이 외부 API나 앱 코드라면 완전히 방향이 틀립니다.
  • "인덱스가 없으면 추가"는 단순화입니다. 인덱스가 있어도 selectivity가 낮거나, 인덱스가 너무 많아서 오히려 느린 경우도 있습니다.
  • 시스템 전체를 보기 전에 DB로 뛰어드는 패턴입니다.
STRONG ANSWER 요소 체크리스트
  • "먼저 어디가 병목인지 데이터로 확인합니다"라는 원칙을 먼저 말한다
  • 메트릭 수집 → tracing으로 위치 특정 → 원인별 도구 사용 순서를 말한다
  • 병목 위치에 따라 다르게 접근함을 분기해서 설명한다
  • 개선 검증 방법(before/after 수치)을 포함한다
  • Amdahl's Law를 암묵적으로 이해하고 있음을 드러낸다 ("병목이 아닌 곳을 최적화해봤자 전체 개선 효과가 없습니다")

10. Senior-level Follow-up Q&A

Q: Distributed tracing이 없는 환경에서는 어떻게 병목을 찾으시겠어요?

모범 답변: 로그의 타임스탬프를 활용합니다. 요청 시작 → DB 쿼리 직전 → DB 쿼리 직후 → 응답 직전에 타임스탬프를 찍어두면 각 단계 소요시간을 계산할 수 있습니다. 또한 middleware 레벨에서 요청별 시간을 기록하는 구조로 개선하는 것을 함께 제안합니다. 모니터링 없이 운영하는 것 자체가 기술 부채임을 팀에 인식시키는 것도 시니어의 역할입니다.

함정: "로그 보면 됩니다"만 말하면 약한 답변. 구체적인 방법(타임스탬프 포인트 설정)과 개선 방향(tracing 도입)을 함께 말해야 합니다.

Q: 병목을 찾았는데 원인이 두 가지 이상이에요. 어떤 것을 먼저 고치시겠어요?

모범 답변: Amdahl's Law를 기준으로 판단합니다. 전체 응답 시간 중 가장 큰 비중을 차지하는 병목을 먼저 제거합니다. 그리고 각 개선 작업의 예상 효과(전체 시간 대비 %)와 공수를 비교해서 우선순위를 결정합니다. 예: 1,800ms DB + 200ms 코드라면, 코드를 0으로 만들어도 전체는 1,800ms입니다. DB를 먼저 해결합니다.

Q: 개선했는데 p99는 줄었는데 p50이 높아졌어요. 무슨 일이 일어난 걸까요?

모범 답변: 일반적인 요청은 느려졌고 특정 패턴(아웃라이어)만 빨라진 것입니다. 예를 들어 캐시를 추가했는데 Cold start 요청이 많이 발생한다면, 캐시 히트 요청은 아주 빠르고 캐시 미스 요청은 오히려 DB 왕복 + 캐시 쓰기 오버헤드로 더 느려질 수 있습니다. 이 경우 Percentile 분포 전체 (히스토그램)를 봐서 어느 구간이 어떻게 변했는지 확인해야 합니다.

Q: 개발 환경에서는 빠른데 운영에서만 느려요. 왜일까요?

모범 답변: 여러 원인이 있습니다. ① 데이터 볼륨 차이 — 운영 DB에는 수천만 건이 있어서 인덱스가 없으면 풀 스캔이 느려집니다. ② 동시 접속 수 차이 — 개발에서는 혼자 쓰지만 운영에서는 수백 명이 동시 접속해서 DB connection pool 고갈이나 lock contention이 발생합니다. ③ 리소스 차이 — 개발 서버는 느리지만 운영 서버는 더 많은 코어가 있어서 오히려 다른 패턴의 문제가 발생합니다. ④ 캐시 warming 차이 — 개발에서는 캐시가 이미 warm up 되어있는 상태에서 테스트합니다.

Q: 성능 최적화 작업을 팀원들에게 어떻게 설득하셨나요?

모범 답변: 사용자 영향을 수치로 만들었습니다. "p99 latency 5초"보다 "매일 5,000명의 사용자가 5초 이상 기다리고 있으며, 이 중 30%가 이탈합니다"가 더 설득력 있습니다. 비즈니스 지표와 성능 지표를 연결하는 것이 핵심입니다. 또한 개선 후 예상 수치를 제시하고, 작업 범위와 위험도를 명확히 해서 의사결정자가 판단할 수 있는 형태로 제안합니다.

이 질문은 기술적 역량이 아니라 커뮤니케이션 역량을 평가합니다. 시니어는 고치는 것뿐만 아니라 설득하는 것도 합니다.

11. 흔한 오해

오해 1

"인덱스를 추가하면 항상 빨라진다"

인덱스가 많으면 INSERT/UPDATE/DELETE가 느려지고, selectivity가 낮은 인덱스는 Full Table Scan보다 느릴 수 있습니다. 인덱스는 도구이지 마법이 아닙니다.

오해 2

"캐시를 추가하면 성능 문제가 해결된다"

캐시는 병목이 DB 읽기인 경우에만 효과적입니다. 병목이 쓰기, CPU 연산, 외부 API라면 캐시는 도움이 안 됩니다. 또한 캐시는 무효화 문제, 일관성 문제, 운영 복잡도를 추가합니다.

오해 3

"평균 응답시간이 빠르면 문제없다"

평균은 아웃라이어에 왜곡됩니다. 99%의 요청이 10ms, 1%가 10,000ms이면 평균은 100ms로 좋아 보입니다. 하지만 1%의 사용자는 10초를 기다립니다. p99 latency를 항상 함께 봐야 합니다.

오해 4

"성능 최적화는 배포 전에 한 번 하면 된다"

데이터가 늘어나거나 트래픽 패턴이 바뀌면 성능 특성이 달라집니다. 성능은 한 번 해결하고 끝이 아니라, 지속적인 모니터링과 회귀 탐지가 필요합니다.

오해 5

"느린 API = DB 문제"

DB가 병목인 경우가 많긴 하지만 전제로 삼으면 위험합니다. 외부 결제 API가 느리거나, 앱 코드 내 비효율 루프가 원인이거나, 네트워크 패킷 손실이 원인일 수 있습니다. 항상 측정 후 결론 내려야 합니다.
