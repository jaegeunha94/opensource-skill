Reranking 전략

1차 검색이 "대충 관련 있는 후보"를 넓게 모으는 단계라면, reranking은 그 후보를 실제 질의 의도에 맞게 정밀하게 재정렬하는 단계다 — 어떤 모델을, 몇 개의 후보에, 어떤 예산으로 적용할지 판단하는 능력을 확인한다.

Q1 — 주력 질문 "Hybrid search로 이미 BM25와 dense를 결합했는데, 왜 reranking 단계가 추가로 필요한가요? Bi-encoder와 cross-encoder의 구조적 차이로 설명해 주세요."
Q2 — 변형 (비용형) "Reranking은 항상 품질을 올려주는데, 왜 모든 요청에 무조건 적용하지 않나요? 비용과 지연 관점에서 설계 기준을 말해 주세요."
Q3 — 변형 (모델 선택형) "Cross-encoder reranker, ColBERT 같은 late-interaction reranker, LLM을 reranker로 쓰는 방식 중 어떤 상황에 어떤 것을 고르시겠습니까?"
Q4 — 변형 (장애형) "Reranker API가 타임아웃되거나 장애가 났습니다. 서비스는 계속 응답해야 합니다. 어떻게 설계하시겠습니까?"
전제 개념

왜 1차 검색만으로는 부족한가

Recall을 넓히는 단계와 Precision을 좁히는 단계는 역할이 다르다

Day 4~7에서 다룬 dense embedding, BM25, hybrid fusion, metadata filter는 모두 1차 검색(first-stage retrieval)이다. 이 단계의 목표는 "정답이 후보 안에 반드시 들어있게 하는 것", 즉 recall을 넓히는 것이다. 그래서 속도를 위해 쿼리와 문서를 각각 독립적으로 벡터로 인코딩한 뒤 내적/코사인 유사도로 비교하는 bi-encoder 구조를 쓴다 — 문서 임베딩은 미리 계산해 인덱스에 저장해두고, 쿼리가 오면 쿼리만 인코딩해서 비교하면 되므로 수백만 문서에서도 빠르다.

문제는 bi-encoder가 쿼리와 문서를 서로 한 번도 같이 보지 않고 각자 인코딩한다는 점이다. "이 문서가 이 쿼리에 얼마나 관련 있는가"를 정밀하게 판단하려면 쿼리의 각 단어와 문서의 각 단어가 서로 어떻게 상호작용하는지를 봐야 하는데, bi-encoder는 이 상호작용 정보를 압축된 벡터 하나에 욱여넣기 때문에 정밀도에 한계가 있다. 그래서 1차 검색으로 뽑은 top-30~100 후보 안에는 진짜 정답이 들어있지만, 그 순서(ranking)는 정확하지 않을 수 있다 — 이게 reranking이 필요한 이유다.

Cross-encoder — 쿼리와 문서를 함께 보고 관련도를 직접 계산

Cross-encoder는 쿼리와 문서 쌍을 하나의 입력으로 합쳐서 트랜스포머에 통째로 넣고, 두 텍스트 사이의 토큰 단위 상호작용(attention)을 모두 계산한 뒤 "이 쌍의 관련도 점수"를 하나의 숫자로 직접 출력한다. Bi-encoder가 "각자 좌표를 찍고 거리를 재는" 방식이라면, cross-encoder는 "둘을 나란히 놓고 직접 비교해서 점수를 매기는" 방식이다. 그만큼 정확하지만, 문서 임베딩을 미리 계산해둘 수 없다 — 쿼리가 올 때마다 후보 문서 각각과 쌍을 이뤄 매번 새로 추론해야 하므로, 후보가 많아질수록 비용과 지연이 선형으로 증가한다. 그래서 cross-encoder는 전체 데이터셋이 아니라 1차 검색이 좁혀준 top-N(보통 수십~100개) 후보에만 적용하는 2단계(two-stage) 구조가 표준이다.

Late-interaction(ColBERT) — 그 중간 지점

ColBERT류의 late-interaction 모델은 절충안이다. 문서를 토큰 단위 임베딩 여러 개로 미리 인코딩해서 저장해두고(그래서 "late"), 쿼리가 오면 쿼리 토큰 임베딩과 문서 토큰 임베딩 사이의 MaxSim(각 쿼리 토큰이 문서 토큰 중 가장 가까운 것과 매칭되는 최댓값의 합) 연산만 쿼리 시점에 수행한다. Cross-encoder처럼 쌍을 다시 통째로 인코딩하지 않아도 되므로 지연이 훨씬 짧다(벤치마크에 따라 p50 수십 ms 수준). 대신 문서마다 벡터 하나가 아니라 토큰 수만큼의 벡터를 저장해야 하므로 인덱스 크기가 단일 벡터 방식 대비 1~2자릿수 커진다는 저장 비용 트레이드오프가 있다.

LLM을 reranker로 쓰는 방식(listwise reranking)

RankGPT류의 접근은 LLM에게 "이 쿼리에 대해 다음 문서들을 관련도 순으로 정렬해줘"라고 통째로 묻는다 (listwise). LLM은 지시사항이나 다단계 추론이 필요한 복잡한 관련도 판단(예: "이 정책 문서가 최신 버전인지까지 고려해서 정렬해줘")에서 cross-encoder보다 더 정교한 판단을 할 수 있지만, 지연이 수 초 단위로 커지고 비용도 훨씬 높다. 그래서 실시간 사용자 응답 경로보다는 배치 평가, 오프라인 데이터셋 큐레이션, 지연에 덜 민감한 워크플로우에 주로 쓰인다.

방식 쿼리-문서 상호작용 시점 지연 정확도 경향 주 용도
Bi-encoder(1차 검색) 없음 — 각자 독립 인코딩 매우 낮음(사전 계산) 낮음~중간 전체 데이터셋에서 recall 확보
Late-interaction(ColBERT) 토큰 벡터는 사전 계산, MaxSim만 쿼리 시점 낮음(p50 수십 ms대 사례) 중간~높음 지연에 민감하지만 cross-encoder급 품질이 필요한 경우
Cross-encoder 쌍 전체를 매번 통째로 인코딩 중간(후보 수에 선형 비례, 보통 +100~200ms) 높음(+5~15 NDCG@10 보고 사례) top-N 재정렬의 표준
LLM listwise(RankGPT류) 후보 리스트 전체를 프롬프트로 한 번에 판단 매우 높음(초 단위) 지시/추론이 필요한 케이스에서 최고 수준 보고 사례 배치·오프라인, 실시간 경로 밖

2026년 기준 모델 지형 — 최소 기준 목록을 넘어서는 최신 근거

Reranker 시장은 embedding 모델만큼 빠르게 바뀐다. 2026년 7월 기준 공개 자료를 종합하면, API 기반으로는 Cohere가 2025년 12월 Rerank v4.0(Pro/Fast 두 변형, 단일 다국어 모델)을 출시해 기존 Rerank 3.5를 대체하는 최신 라인업으로 자리잡았고, Voyage AI는 도메인 특화(code/finance/legal) 변형을 제공하는 rerank-2.5와, 지연에 민감한 경로를 위한 더 가벼운 rerank-2.5-lite를 토큰 기반 과금으로 제공한다. 오픈소스 진영에서는 BGE-reranker-v2-m3, mixedbread mxbai-rerank-large-v2, Jina Reranker v2, Qwen3-Reranker가 단일 GPU에 셀프호스팅하기 좋은 강력한 오픈 웨이트 옵션으로 꼽힌다. Zerank류의 신규 진입자도 벤치마크 상위권에 보고되고 있어, "어느 모델이 최고"라고 고정하기보다 자체 도메인 평가셋으로 주기적으로 재검증하는 태도가 중요하다.

과금 단위 차이에 주의: 같은 "reranking API"라도 과금 방식이 다르다 — Cohere는 "쿼리 1개 + 문서 최대 100개"를 1 search unit으로 묶어 1,000 유닛당 과금하고, Voyage는 처리한 토큰 수 기준으로 과금한다. 후보 문서 수, 문서당 평균 토큰 길이에 따라 실제 비용 구조가 크게 달라지므로, 벤더 비교 시 단순 "가격표 숫자"가 아니라 우리 워크로드의 후보 수·문서 길이 분포를 대입한 실제 예상 비용을 계산해야 한다.
참고: bi-encoder/cross-encoder/late-interaction 구조 차이, 지연·NDCG@10 개선 폭, 2026년 7월 기준 모델 지형(Cohere Rerank v4.0, Voyage rerank-2.5/2.5-lite, BGE-reranker-v2-m3 등 오픈 웨이트 옵션)과 과금 방식은 레슨 작성 시점 공개 자료를 바탕으로 했다. 모델 순위·가격·지연 수치는 벤더가 자주 갱신하므로 도입 전 공식 문서와 자체 벤치마크로 재검증해야 한다. 세부 출처는 RESOURCES.md 참고.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • Bi-encoder와 cross-encoder의 구조적 차이를 원리로 아는가: "reranking 모델을 하나 더 붙이면 좋아진다"는 결과론적 답이 아니라, 왜 1차 검색은 정밀도에 구조적 한계가 있고 reranking이 그 한계를 어떻게 보완하는지 설명할 수 있는지 본다.
  • 비용/지연을 정량적으로 판단하는가: "reranking을 쓰면 좋다"가 아니라, 후보 수·모델 선택에 따른 지연과 비용 증가를 구체적 수치 감각으로 설명하고, 언제 reranking을 생략해도 되는지 판단 기준을 갖고 있는지 본다.
  • 모델 선택을 상황에 맞게 하는가: cross-encoder, late-interaction, LLM reranker를 무조건 "제일 좋은 것"으로 고르지 않고, 지연 요구사항·후보 수·정확도 요구사항에 따라 선택 기준을 다르게 세우는지 확인한다.
  • 장애 내성을 설계에 포함하는가: reranker를 단일 장애점(SPOF)으로 만들지 않고, 타임아웃·장애 시에도 서비스가 저하된 품질로나마 계속 응답하도록 설계하는 습관이 있는지 본다.
Senior 엔지니어는 "reranker를 붙였더니 품질이 좋아졌습니다"에서 멈추지 않는다. 어떤 후보 수에 어떤 모델을 왜 선택했는지, 그 선택이 비용·지연 예산과 어떻게 맞물리는지, reranker가 실패했을 때 서비스가 어떻게 계속 동작하는지까지 하나의 설계 판단으로 엮어서 설명한다.
실무 연결

실제 상황과 연결하기

상황 1: Hybrid search만으로 top-1 정답률이 목표에 못 미침

Day 6의 hybrid search(BM25 + dense + RRF)를 도입했는데도 사용자가 원하는 답이 top-3 밖에 있다는 피드백이 계속 들어온다. 원인을 분석해보면 정답 chunk가 이미 top-30 안에는 들어있지만 순위가 낮다 — recall 문제가 아니라 precision(순서) 문제다. → RRF로 얻은 top-30~50을 cross-encoder reranker에 넣어 재정렬하고, 그 결과를 LLM context로 넘긴다.

상황 2: QPS가 급증하면서 reranking 비용이 인프라 비용의 상당 부분을 차지

트래픽이 늘면서 API 기반 reranker 비용이 vector DB 비용을 넘어서기 시작했다. → 모든 요청에 무조건 가장 비싼 모델을 쓰는 대신, 지연 SLA가 엄격한 경로에는 lite/fast 계열을, 배치· 비동기 경로에는 더 무거운 모델을 쓰는 계층화 전략을 도입한다. 트래픽이 충분히 크면 오픈 웨이트 reranker를 자체 GPU에 셀프호스팅해 호출당 과금 구조를 상각 비용 구조로 전환하는 것도 검토한다.

상황 3: Reranker API가 타임아웃되면서 검색 응답 자체가 실패

Reranking을 동기 호출 경로에 넣었는데, 벤더 API 지연이 튀는 순간 전체 요청이 타임아웃으로 실패했다. → reranking을 선택적 향상(optional enhancement)으로 설계해, 타임아웃이 나면 1차 검색(hybrid fusion) 순위를 그대로 fallback으로 사용하도록 회로 차단기(circuit breaker)와 타임아웃 예산을 둔다. 품질은 살짝 떨어지지만 서비스는 계속 응답한다.

상황 4: 특정 도메인(법률/의료 계약서)에서 범용 reranker가 오히려 순위를 망침

범용 벤치마크 상위 reranker를 그대로 붙였더니, 도메인 특화 용어가 많은 계약서 검색에서 오히려 관련 없는 조항이 위로 올라오는 사례가 보고됐다. → 도메인 특화 reranker 변형(예: 법률/금융 특화 모델)을 평가하거나, Day 11에서 다룰 자체 도메인 평가셋으로 여러 reranker 후보를 재검증한 뒤 배포한다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"먼저 1차 검색과 reranking은 역할이 다르다는 것부터 말씀드리겠습니다. Hybrid search까지는 bi-encoder 기반이라 쿼리와 문서를 각자 독립적으로 인코딩해서 빠르게 비교하는데, 그만큼 둘 사이의 세밀한 상호작용은 놓칩니다. 그래서 top-30~50 안에는 정답이 들어있지만 순위가 정확하지 않을 수 있고, 이걸 바로잡는 게 reranking입니다.

Reranking에는 cross-encoder, ColBERT 같은 late-interaction, LLM 기반 listwise reranking이 있는데 각각 트레이드오프가 다릅니다. Cross-encoder는 쿼리-문서 쌍을 매번 통째로 인코딩하기 때문에 가장 정확하지만 후보 수에 비례해서 비용과 지연이 늘어나므로, 전체 데이터셋이 아니라 1차 검색이 좁혀준 top-N에만 적용하는 2단계 구조를 씁니다. 후보 30개 기준으로 보통 100~200ms 정도 지연이 추가되는 대신 NDCG@10이 5~15점 정도 개선되는 걸 목표 수치로 잡습니다. Late-interaction 방식은 문서 토큰 임베딩을 미리 계산해두기 때문에 cross-encoder보다 지연이 짧지만 인덱스 크기가 훨씬 커진다는 저장 비용이 있고, LLM을 reranker로 쓰는 방식은 지시사항이 복잡한 판단에서는 더 정교하지만 지연이 초 단위라 실시간 사용자 응답 경로보다는 배치·오프라인 평가에 씁니다.

그래서 저는 지연 SLA가 엄격한 실시간 경로에는 cross-encoder의 가벼운 변형이나 late-interaction을, 비동기·배치 경로에는 더 무거운 모델이나 LLM reranker를 쓰는 식으로 계층화합니다. 그리고 트래픽이 커지면 API 호출당 과금 구조의 비용이 vector DB 비용을 넘어설 수 있어서, 그 시점에는 오픈 웨이트 reranker를 자체 GPU에 셀프호스팅해서 상각 비용 구조로 전환하는 것도 검토합니다.

마지막으로 reranker를 동기 호출 경로의 단일 장애점으로 두지 않습니다. Reranker API가 타임아웃되거나 장애가 나도 서비스 전체가 실패하면 안 되기 때문에, 타임아웃 예산을 두고 실패 시에는 hybrid fusion의 원래 순위를 그대로 fallback으로 써서 품질은 살짝 낮아지더라도 서비스는 계속 응답하도록 설계합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "그냥 embedding 모델을 더 좋은 걸로 바꾸면 reranking이 필요 없지 않나요?"

약한 답변: "더 좋은 embedding 모델을 쓰면 reranking 없이도 순위가 정확해집니다."

강한 답변: "Embedding 모델이 아무리 좋아져도 bi-encoder 구조인 이상 쿼리와 문서를 독립적으로 인코딩한다는 근본적인 한계는 남습니다. Cross-encoder처럼 쿼리와 문서를 함께 보고 토큰 단위 상호작용을 직접 계산하는 구조와는 표현력 자체가 다르기 때문에, embedding 모델 품질 개선과 reranking 도입은 서로 대체재가 아니라 보완재입니다."

F2. "Reranking에 넣는 후보 수(top-N)는 어떻게 정하나요? 무조건 많이 넣으면 좋은가요?"

약한 답변: "가능한 한 많이 넣어야 정답을 놓치지 않습니다."

강한 답변: "Cross-encoder는 후보 수에 비례해서 비용과 지연이 늘어나기 때문에 무조건 많이 넣는 게 아니라, 1차 검색의 recall@k 곡선을 보고 '이 정도 후보면 정답이 거의 다 포함된다'는 지점을 찾아 top-N을 정합니다. 보통 수십에서 100 사이에서 품질과 비용의 손익분기점을 실험적으로 찾는 경우가 많습니다."

F3. "ColBERT 같은 late-interaction 모델을 도입하면 인덱스 크기가 커진다고 했는데, 그 비용을 어떻게 관리하나요?"

약한 답변: "저장 공간을 늘리면 됩니다."

강한 답변: "토큰 단위 임베딩은 단일 벡터 방식보다 1~2자릿수 큰 저장 공간을 요구하므로, 전체 컬렉션에 무조건 적용하기보다 지연에 특히 민감하면서 cross-encoder급 품질이 필요한 경로에 선택적으로 적용하거나, quantization·압축을 함께 적용해 저장 비용을 관리하는 방식을 검토합니다."

F4. "LLM을 reranker로 쓰면 정확도가 더 좋다고 하는데, 왜 기본값으로 쓰지 않나요?"

약한 답변: "LLM이 제일 똑똑하니까 항상 쓰면 좋습니다."

강한 답변: "LLM listwise reranking은 지시나 다단계 추론이 필요한 복잡한 관련도 판단에서는 강점이 있지만, 지연이 초 단위로 커지고 비용도 훨씬 높습니다. 실시간 사용자 응답에서는 사용자가 몇 초 안에 이탈하는 경향이 있어서 적합하지 않고, 배치 평가나 오프라인 큐레이션처럼 지연에 덜 민감한 워크플로우에 우선 적용합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Bi-encoder와 cross-encoder의 구조적 차이(상호작용 시점)로 설명한다
  • 지연·비용·품질 개선폭을 구체적 수치 감각으로 제시한다
  • Cross-encoder/late-interaction/LLM reranker를 상황에 맞게 구분해서 고른다
  • Reranker 장애 시 fallback(graceful degradation) 설계를 포함한다
  • 비용 구조(호출당 vs 토큰당 vs 셀프호스팅 상각)를 워크로드에 대입해 판단한다

약한 답변의 특징

  • "reranking을 붙이면 품질이 좋아진다"는 결과만 안다
  • Bi-encoder/cross-encoder 구조 차이를 설명하지 못한다
  • 모든 상황에 같은 reranker를 쓰면 된다고 답한다
  • Reranker 장애 시 서비스가 통째로 실패해도 된다고 가정한다
  • 비용을 고려하지 않고 "제일 좋은 모델"만 언급한다
핵심 차이: 강한 답변은 reranking을 "품질을 올려주는 추가 모델 호출"이 아니라 "1차 검색과 역할을 나눈 2단계 구조 안에서, 후보 수·모델·장애 대응까지 정량적으로 설계해야 하는 문제"로 다룬다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "reranker는 항상 붙일수록 좋다"

사실: reranking은 후보 수에 비례해 지연과 비용이 늘어난다. 지연 SLA가 매우 엄격하거나 트래픽이 매우 크면 무조건 도입하기보다, 지연 예산·트래픽 규모·품질 요구 수준을 함께 놓고 도입 여부와 후보 수를 결정해야 한다.

오해 2: "reranking을 쓰면 1차 검색(hybrid search) 품질은 대충 해도 된다"

사실: Cross-encoder reranker는 1차 검색이 가져온 후보 안에서만 재정렬한다. 애초에 정답이 top-N 후보에 들어있지 않으면 reranker가 아무리 좋아도 찾을 수 없다. 1차 검색의 recall과 reranking의 precision은 서로 대체할 수 없는 별개의 책임이다.

오해 3: "가장 벤치마크 점수가 높은 reranker가 우리 서비스에도 최선이다"

사실: 범용 벤치마크 상위 모델이 특정 도메인(법률, 의료, 사내 전문 용어)에서는 오히려 순위를 악화시킬 수 있다. 도입 전 자체 도메인 평가셋으로 재검증해야 하며, 도메인 특화 변형이 있다면 함께 비교해야 한다.

오해 4: "LLM으로 reranking하면 무조건 더 정확하니 cross-encoder는 구식이다"

사실: LLM listwise reranking은 지시·다단계 추론이 필요한 케이스에서 강점이 있지만 지연이 초 단위로 크고 비용도 훨씬 높다. 실시간 사용자 응답 경로에는 여전히 cross-encoder나 late-interaction이 표준이고, LLM reranking은 배치·오프라인 워크플로우의 도구다.

핵심 요약

오늘 반드시 기억할 것

  • 1차 검색은 recall을 넓히고, reranking은 precision(순서)을 정밀하게 만드는 서로 다른 책임을 가진 단계다.
  • Cross-encoder는 쿼리-문서 쌍을 매번 통째로 인코딩하므로 전체 데이터셋이 아니라 1차 검색이 좁혀준 top-N에만 2단계로 적용하는 것이 표준이다.
  • Late-interaction(ColBERT)은 지연은 짧지만 인덱스 크기가 커지고, LLM listwise reranking은 정확하지만 지연이 초 단위라 실시간 경로보다 배치에 적합하다.
  • 모델·후보 수는 지연 SLA와 비용 구조에 맞춰 계층화해야 하며, 벤더 과금 방식(호출당 vs 토큰당)을 실제 워크로드에 대입해 비용을 계산해야 한다.
  • Reranker를 단일 장애점으로 두지 말고, 장애·타임아웃 시 1차 검색 순위로 graceful degradation하는 fallback을 반드시 설계에 포함해야 한다.
