Day 2 · RAG Interview

문서 수집과 정제 파이프라인

여러 소스에 흩어진 문서를 검색 가능한 형태로 안전하고 신선하게 유지하는 데이터 파이프라인을 설계하는 능력
면접 질문
Q1 — 주력 질문 "Confluence, PDF, Slack, 사내 DB에 흩어진 문서를 RAG에 반영해야 합니다. 수집(ingestion) 파이프라인을 어떻게 설계하시겠습니까?"
Q2 — 변형 "문서 정제 단계에서 무엇을 걸러내야 하나요? 정제를 대충 하면 뒤 단계에서 어떤 문제가 생기나요?"
Q3 — 변형 (운영형) "문서가 매일 수천 건씩 갱신됩니다. 매번 전체 재색인을 하기엔 비용이 부담스러운데 어떻게 설계하시겠습니까?"
전제 개념

Ingestion이 RAG 품질의 상한선을 정한다

왜 Ingestion부터 시작해야 하는가

Day 1에서 RAG 실패의 대다수가 검색(Retrieval) 단계에서 비롯된다고 배웠다. 그런데 검색이 아무리 좋은 임베딩 모델과 reranker를 써도, 애초에 인덱스에 들어간 문서가 지저분하거나, 오래됐거나, 중복돼 있으면 검색 품질은 절대 그 이상을 넘을 수 없다. 이것이 "Garbage In, Garbage Out"이며, Ingestion은 RAG 전체 품질의 상한선(ceiling)을 결정한다.

핵심 전제: Ingestion 파이프라인은 "문서를 벡터DB에 넣는 일"이 아니라 수집 → 파싱 → 정제 → 메타데이터 부착 → (Day 3의 chunking으로 이어지는) 데이터 엔지니어링 파이프라인이다. AI 문제가 아니라 데이터 파이프라인 문제로 접근해야 한다.

수집(Collection) — 소스별 커넥터

실무에서 문서는 한 곳에 있지 않다. Confluence/Notion(구조화된 위키), PDF/Word(반구조화 문서), 스캔본 이미지(비구조화), Slack/이메일(대화형 텍스트), 사내 DB(정형 데이터)처럼 형태가 전부 다르다. 각 소스는 API 인증, 페이지네이션, rate limit, 증분 조회(최근 변경분만 가져오기) 방식이 다르므로 소스별 커넥터를 분리해서 설계해야 한다.

파싱(Parsing) — 구조를 보존하며 텍스트를 뽑아내기

PDF나 스캔본에서 텍스트만 뽑으면 제목/본문/표/코드블록 구분이 사라진다. 예를 들어 표(table)를 일반 텍스트로 풀어버리면 "행과 열의 관계"가 사라져서 숫자만 나열된 의미 없는 텍스트가 된다. 스캔된 이미지 문서는 OCR(광학 문자 인식)을 거쳐야 하는데, OCR 오류(글자 깨짐, 표 인식 실패)가 그대로 검색 품질 저하로 이어진다.

중요: 파싱 단계에서 문서의 구조(제목 계층, 표, 코드블록)를 최대한 보존해야 한다. 이 구조 정보는 Day 3의 chunking 전략(구조 기반 분할)의 입력이 된다. 파싱에서 구조를 뭉개버리면, 아무리 좋은 chunking 전략을 써도 복구할 수 없다.

정제(Cleaning) — 노이즈와 중복 제거

실무에서 반드시 걸러야 하는 노이즈 유형:

노이즈 유형 예시 방치 시 문제
보일러플레이트 페이지 헤더/푸터, 네비게이션 메뉴, "이 문서가 도움이 됐나요?" 같은 반복 문구 모든 chunk에 같은 문구가 섞여 임베딩이 흐려짐
정확 중복(exact duplicate) 동일 문서가 여러 시스템에 복사되어 존재 검색 결과 상위권이 같은 내용으로 도배됨, 임베딩 비용 낭비
근접 중복(near-duplicate) 초안과 최종본, 버전만 다른 정책 문서 서로 다른(때로 상충하는) 버전이 동시에 검색되어 답변이 모순됨
인코딩/포맷 깨짐 깨진 한글 인코딩, HTML 태그 잔존, 이상한 공백/개행 임베딩 품질 저하, chunk 경계가 이상하게 잘림

정확 중복 vs 근접 중복 탐지

정확 중복은 문서 내용의 해시값(예: SHA-256)이 같은지로 즉시 탐지할 수 있다. 근접 중복은 내용이 조금씩 다르므로(초안/최종본, 오탈자 수정본) 해시로는 못 잡는다. SimHash/MinHash 같은 근사 유사도 기법이나, 문서 임베딩 간 코사인 유사도가 임계값 이상이면 근접 중복으로 판단하는 방식을 쓴다. 근접 중복을 발견하면 최신본만 남기고 나머지는 인덱스에서 제외하거나 "구버전" 메타데이터를 붙여 낮은 우선순위로 검색되게 한다.

메타데이터 부착 — 검색 이후를 위한 준비

정제된 문서에는 출처(source), 작성/수정일(freshness 판단용), 문서 버전, 그리고 접근 권한(ACL) 정보를 메타데이터로 함께 저장해야 한다. 이 메타데이터는 Day 6(Metadata Filtering)과 Day 11(권한/보안)에서 검색 시점에 필터링하는 데 쓰인다. Ingestion 단계에서 권한 정보를 놓치면, 나중에 아무리 검색 로직을 잘 짜도 권한 없는 사용자에게 문서가 노출되는 사고로 이어질 수 있다.

신선도(Freshness)와 재색인 전략

문서는 계속 바뀐다. Ingestion 파이프라인은 두 가지 방식 중 선택하거나 조합해서 설계한다.

방식 동작 장점 단점
배치(Batch) 재색인 주기적으로(예: 매일 새벽) 전체 또는 변경분을 스캔해 재색인 구현 단순, 소스 시스템 부하 예측 가능 배치 주기 동안은 최신 정보 미반영 (최대 지연 발생)
이벤트 기반(Event-driven) 증분 색인 소스 시스템의 webhook/CDC(Change Data Capture)로 변경 즉시 색인 큐에 반영 거의 실시간 반영, 변경분만 처리해 비용 절감 구현 복잡도 증가, 멱등성(idempotency) 보장 필요
삭제 반영을 잊지 말 것: 문서가 원본 시스템에서 삭제되거나 접근 권한이 회수됐는데 벡터 인덱스에는 그대로 남아있는 경우가 실무에서 매우 흔한 사고 원인이다. Ingestion 파이프라인은 "추가/수정"뿐 아니라 "삭제 이벤트"도 반드시 처리해야 한다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "임베딩 모델 이름을 아느냐"가 아니라 "RAG를 데이터 파이프라인 엔지니어링 문제로 다룰 수 있느냐"를 본다.

  • Garbage-in-garbage-out을 이해하는가: 검색·생성 품질 개선 전에 Ingestion 품질부터 점검하는 순서로 사고하는가
  • 운영 관점을 갖췄는가: 신선도, 삭제 반영, 멱등성, 증분 처리처럼 "한 번 만들고 끝"이 아니라 지속 운영되는 파이프라인으로 설계하는가
  • 권한/보안을 앞단에서 설계하는가: 접근 권한 메타데이터를 Ingestion 시점부터 부착해야 한다는 것을 아는가, 아니면 나중에 붙이면 된다고 생각하는가
  • 비용/운영 부담을 트레이드오프로 설명하는가: 전체 재색인과 증분 색인의 비용·복잡도·최신성 트레이드오프를 구체적으로 설명할 수 있는가
Senior 엔지니어는 Ingestion을 "문서를 벡터DB에 넣는 전처리 스크립트"로 보지 않는다. 지속적으로 운영되는, 실패를 감지하고 복구할 수 있는 데이터 파이프라인으로 설계한다.
실무 연결

실제 상황과 연결하기

상황 1: Confluence 문서를 수집했는데 권한 없는 사용자가 기밀 문서 답변을 받음

Ingestion 시점에 문서 내용만 가져오고 "누가 볼 수 있는지" 정보를 저장하지 않았다. → 접근 권한 메타데이터를 Ingestion 단계에서부터 부착하고, 검색 시점에 사용자 권한과 대조하는 필터링이 필요하다 (Day 6, Day 11에서 이어짐).

상황 2: 스캔된 계약서 PDF의 검색 정확도가 낮다

OCR 품질이 낮아 숫자와 문자가 뒤섞여 인식됐다. → OCR 신뢰도 점수(confidence score)가 낮은 영역은 별도로 표시하거나 사람이 검수하는 워크플로우를 추가하고, 표는 별도 파서로 구조를 보존해야 한다.

상황 3: 초안과 최종 승인본이 둘 다 검색되어 답변이 오락가락함

같은 정책의 v1(초안)과 v2(최종본)이 모두 인덱스에 있어 질문마다 다른 버전이 검색된다. → 근접 중복 탐지로 구버전을 식별하고, 최신 버전만 활성 인덱스에 남기거나 버전 메타데이터로 최신본에 가중치를 준다.

상황 4: 매일 수천 건의 문서 변경을 전체 재색인하다 임베딩 비용이 폭증

변경된 문서 몇십 건 때문에 전체 수백만 건을 매번 다시 임베딩했다. → 변경 이벤트(webhook/CDC)를 받아 변경분만 증분 재처리하는 구조로 바꾸고, 대량 초기 적재는 배치로, 이후 운영은 이벤트 기반으로 전환한다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
소스를 구조 유형별로 분류하기 "먼저 소스를 구조화(DB, API), 반구조화(Confluence, PDF), 비구조화(스캔 이미지, 대화 로그)로 나누고, 소스별 커넥터와 증분 조회 방식을 설계합니다."
2
파싱에서 구조 보존을 최우선으로 두기 "제목 계층, 표, 코드블록 같은 구조를 최대한 보존해서 파싱합니다. 이 구조는 이후 chunking 품질에 직접 영향을 주기 때문입니다."
3
정제 규칙과 중복 제거 적용하기 "보일러플레이트를 제거하고, 정확 중복은 해시로, 근접 중복은 유사도 기반으로 탐지해 최신 버전만 남깁니다."
4
메타데이터를 Ingestion 시점에 부착하기 "출처, 작성일, 버전, 접근 권한 정보를 이 시점에 함께 저장합니다. 나중에 붙이려면 원본 시스템을 다시 조회해야 해서 비효율적이고, 권한 누락 리스크도 커집니다."
5
신선도 전략을 트레이드오프로 설계하기 "초기 대량 적재는 배치로, 이후 지속 갱신은 이벤트 기반 증분 색인으로 전환합니다. 삭제/권한 회수 이벤트도 반드시 함께 처리합니다."
6
운영 관측 포인트 언급하기 (Senior 포인트) "파싱 실패율, OCR 신뢰도, 색인 지연시간, 중복 탐지율 같은 지표를 모니터링해서 Ingestion 자체의 건강 상태를 추적합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 Ingestion을 RAG 품질의 상한선을 정하는 단계로 봅니다. 아무리 검색과 reranking을 잘 튜닝해도, 애초에 인덱스에 들어간 문서가 지저분하거나 오래됐으면 그 이상 좋아질 수 없습니다.

먼저 소스를 구조화/반구조화/비구조화로 나누고 소스별 커넥터를 설계합니다. 파싱 단계에서는 제목 계층이나 표 같은 구조를 최대한 보존하는 걸 최우선으로 둡니다. 구조가 파싱 단계에서 사라지면 이후 chunking으로도 복구할 수 없기 때문입니다.

정제 단계에서는 보일러플레이트를 제거하고, 정확 중복은 해시로, 초안/최종본 같은 근접 중복은 유사도 기반으로 탐지해서 최신 버전만 남깁니다. 그리고 이 시점에 출처, 버전, 접근 권한 같은 메타데이터를 함께 저장합니다. 권한 정보를 나중에 붙이려고 하면 원본 시스템을 또 조회해야 하고, 누락되면 보안 사고로 이어질 수 있습니다.

신선도는 초기 대량 적재는 배치로 처리하고, 이후 지속 갱신은 webhook이나 CDC 기반의 이벤트 드리븐 증분 색인으로 전환합니다. 이때 삭제나 권한 회수 이벤트도 반드시 반영해야 하고, 재처리가 중복 색인을 만들지 않도록 멱등성을 보장합니다. 마지막으로 파싱 실패율, OCR 신뢰도, 색인 지연 같은 지표를 관측해서 Ingestion 파이프라인 자체의 건강 상태를 지속적으로 추적합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "전체 재색인과 증분 색인, 언제 무엇을 쓰나요?"

약한 답변: "증분 색인이 더 빠르니까 항상 증분으로 합니다."

강한 답변: "초기 대량 적재나, 임베딩 모델을 교체해서 전체 벡터를 다시 만들어야 하는 경우는 전체 재색인이 불가피합니다. 반면 운영 중 일상적인 문서 수정·추가는 증분 색인이 맞습니다. 다만 증분 색인은 멱등성이 핵심입니다 — 같은 변경 이벤트가 중복 전달돼도 중복 벡터가 쌓이지 않도록 문서 ID 기준으로 upsert(있으면 갱신, 없으면 추가)해야 합니다. 또한 증분 색인만으로는 '삭제'나 '정합성 어긋남'이 누적될 수 있어서, 저는 주기적으로 원본과 인덱스 상태를 대조하는 정합성 점검(reconciliation) 배치를 함께 돌리는 것을 권장합니다."

F2. "문서 접근 권한이 바뀌면 어떻게 반영하나요?"

약한 답변: "검색 결과를 화면에 보여줄 때 권한 체크해서 걸러줍니다."

강한 답변: "표시 단계에서만 필터링하면, 검색된 chunk가 이미 LLM 프롬프트에 들어가버려서 권한 없는 사용자가 답변 내용을 통해 정보를 유추할 수 있는 leak 위험이 있습니다. 그래서 권한 필터링은 검색(retrieval) 단계 자체에서 이뤄져야 합니다 (Day 6, 11에서 자세히 다룹니다). Ingestion 관점에서는 원본 시스템의 권한 변경 이벤트를 구독해서, 권한이 바뀌면 해당 문서의 메타데이터를 즉시 갱신하고, 주기적으로 원본과 권한 정보를 대조하는 배치도 함께 운영합니다."

F3. "정확 중복과 근접 중복을 어떻게 구분해서 탐지하나요?"

강한 답변: "정확 중복은 문서 내용의 해시값(SHA-256 등)이 완전히 같은지로 즉시 판단할 수 있어서 비용이 거의 들지 않습니다. 근접 중복은 내용이 조금씩 달라서 해시로는 못 잡기 때문에, SimHash/MinHash 같은 근사 유사도 기법이나 문서 임베딩 간 코사인 유사도로 판단합니다. 임계값을 너무 낮게 잡으면 실제로 다른 문서를 중복으로 오판할 수 있어서, 저는 보통 높은 임계값(예: 0.95 이상)으로 시작해서 실제 샘플을 사람이 검수하며 조정합니다."

F4. "OCR 품질이 나쁜 스캔 문서는 어떻게 처리하시겠어요?"

강한 답변: "OCR 엔진이 제공하는 신뢰도(confidence) 점수를 저장해서, 신뢰도가 낮은 영역은 메타데이터로 표시해두거나 검색 우선순위를 낮춥니다. 표나 서명란처럼 OCR이 특히 약한 영역은 레이아웃 인식(layout-aware) OCR이나 별도 파서를 쓰는 걸 고려하고, 법률 계약서처럼 정확도가 절대적으로 중요한 문서는 사람 검수 워크플로우를 Ingestion 파이프라인에 포함시킵니다. 품질이 검증되지 않은 OCR 결과를 그대로 임베딩에 태우는 건 위험하다고 판단합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Ingestion을 RAG 품질의 상한선으로 명확히 인식한다
  • 파싱에서 구조 보존이 왜 중요한지 chunking과 연결해서 설명한다
  • 정확 중복과 근접 중복을 다른 탐지 기법으로 구분해서 말한다
  • 권한 메타데이터를 Ingestion 시점에 부착해야 한다고 말한다
  • 배치/이벤트 기반 색인을 트레이드오프로 설명하고 삭제 반영·멱등성을 챙긴다
  • Ingestion 자체의 운영 지표(파싱 실패율 등)를 언급한다

약한 답변의 특징

  • "문서를 긁어서 벡터DB에 넣습니다" — 정제/메타데이터 개념 없음
  • 구조(표, 제목)를 보존해야 하는 이유를 설명하지 못한다
  • 중복 제거를 "같은 문서면 하나만 남긴다" 수준으로만 설명한다
  • 권한 필터링을 "화면에 보여줄 때 걸러주면 된다"고 답한다 (leak 위험 간과)
  • 재색인을 "그냥 매번 다시 돌리면 된다"고 답한다 (비용/삭제 반영 간과)
  • Ingestion을 "한 번 세팅하고 끝나는 스크립트"로 취급한다
핵심 차이: 강한 답변은 Ingestion을 "AI 붙이기 전 전처리"가 아니라 "지속 운영되는 데이터 파이프라인이자 보안 경계의 첫 지점"으로 다룬다. 면접관은 데이터 엔지니어링 기본기와 보안 감각을 함께 본다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "정제는 나중에 chunking할 때 대충 해도 된다"

사실: chunking은 이미 정제된 텍스트를 전제로 동작한다. 보일러플레이트나 깨진 인코딩이 남아 있으면 chunk 경계가 이상하게 잘리고, 모든 chunk에 같은 노이즈가 섞여 임베딩 유사도 자체가 흐려진다. 정제는 chunking 이전에 끝나 있어야 한다 (Day 3).

오해 2: "임베딩 모델이 좋으면 노이즈가 있어도 알아서 잘 찾는다"

사실: 좋은 임베딩 모델도 입력이 노이즈로 오염되면 그 노이즈까지 벡터에 반영한다. 모델 성능은 정제된 입력을 전제로 한 상한선이지, 정제를 대신해주지 않는다.

오해 3: "접근 권한은 검색 결과를 보여줄 때 필터링하면 충분하다"

사실: 표시 단계 필터링은 이미 LLM 프롬프트에 권한 없는 내용이 들어간 뒤의 조치라 늦다. 검색 단계에서 권한 기반으로 후보 자체를 제외해야 하며, 이를 위한 권한 메타데이터는 Ingestion 시점부터 정확히 부착돼야 한다.

오해 4: "한 번 색인하면 끝난다"

사실: 문서는 계속 바뀌고 삭제된다. 신선도 전략(배치/이벤트 기반), 삭제 반영, 정합성 점검이 없으면 시간이 지날수록 인덱스와 실제 문서 상태가 어긋나는 "drift"가 누적된다.

4. 배치 재색인과 이벤트 기반 증분 색인의 트레이드오프를 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • Ingestion은 RAG 품질의 상한선을 정한다. 검색·생성을 고치기 전에 수집 품질부터 의심한다.
  • 파싱에서 구조(제목/표/코드블록)를 보존해야 이후 chunking 품질이 확보된다.
  • 정확 중복은 해시로, 근접 중복은 유사도 기반으로 각각 다르게 탐지한다.
  • 권한 메타데이터는 Ingestion 시점에 부착해야 하고, 필터링은 검색 단계에서 이뤄져야 한다.
  • 신선도는 배치와 이벤트 기반을 조합한다. 삭제/권한 회수 이벤트 반영과 멱등성을 반드시 챙긴다.
  • Ingestion 자체도 파싱 실패율, 색인 지연 같은 운영 지표로 관측해야 한다.
