자료구조 10년차 이상 개발자 Interview

Day 5 — 트리와 균형 트리

정렬된 데이터를 다룰 때 배열, BST, balanced tree 중 무엇을 선택하는지, 그리고 균형이 깨진 트리가 왜 조용히 성능 장애로 이어지는지 설명한다.

면접 질문

핵심 질문 1
정렬된 데이터를 다룰 때 배열, BST, balanced tree를 어떻게 비교합니까?
핵심 질문 2
BST 기반 인덱스를 실무에 도입했는데 특정 데이터셋에서만 조회 성능이 O(log n)이 아니라 O(n)에 가깝게 나온다는 리포트를 받았다면, 무엇을 의심하고 어떻게 확인하겠습니까?
핵심 질문 3
AVL 트리와 Red-Black 트리 중 하나를 선택해야 한다면, 어떤 기준으로 결정하겠습니까?

전제 개념 — 여기서부터 읽으면 됩니다

이진 탐색 트리(BST)란 무엇인가 — "정렬을 유지하며 빠르게 찾는" 구조

이진 탐색 트리(Binary Search Tree, BST)는 각 노드가 왼쪽 자식보다 크고 오른쪽 자식보다 작은 값을 가지는 규칙을 지키는 트리입니다. 이 규칙 덕분에 특정 값을 찾을 때 루트에서 시작해 "더 크면 오른쪽, 더 작으면 왼쪽"으로 한 단계씩 내려가며 탐색 범위를 절반씩 줄일 수 있습니다. 이것이 O(log n) 탐색의 원리입니다. 정렬된 배열의 이분 탐색과 개념적으로 비슷하지만, 배열과 달리 삽입/삭제 시 전체 데이터를 밀어내지 않아도 된다는 점이 핵심 차이입니다.

여기서 "log n"이라는 시간복잡도는 트리의 높이(height)에 비례합니다. 탐색 한 번의 비용은 정확히는 "트리의 높이만큼의 비교 횟수"이고, 높이가 log n이 되려면 트리가 균형(balanced) 잡혀 있어야 한다는 전제가 숨어 있습니다. 시니어 면접은 바로 이 숨은 전제를 파고듭니다.

편향 트리(degenerate tree) — BST가 "몰래" 연결 리스트가 되는 순간

BST는 삽입 순서에 따라 모양이 완전히 달라집니다. 만약 데이터를 이미 정렬된 순서로 하나씩 삽입하면(예: 1, 2, 3, 4, 5... 순서로), 매번 새 값이 기존 루트보다 크기 때문에 항상 오른쪽 자식으로만 내려가게 됩니다. 그 결과 트리는 왼쪽 자식이 하나도 없는, 오른쪽으로만 길게 늘어진 모양이 됩니다. 이것을 편향 트리(degenerate tree)라고 부르며, 사실상 배열이 아닌 연결 리스트와 동일한 구조입니다.

이 상태에서 트리의 높이는 log n이 아니라 n이 되고, 탐색/삽입/삭제 시간복잡도는 O(log n)이 아니라 O(n)으로 나빠집니다. 문제는 이 상황이 코드 버그가 아니라 입력 데이터의 순서 때문에 발생한다는 점입니다. 로그 데이터, 타임스탬프, 이미 정렬된 CSV를 그대로 BST에 넣는 것처럼 흔한 실무 패턴이 편향 트리를 만드는 전형적인 원인입니다.

균형(balance)이란 무엇이고, 회전(rotation)은 왜 필요한가

균형이란 트리의 왼쪽 서브트리와 오른쪽 서브트리의 높이 차이가 일정 범위를 넘지 않도록 유지하는 성질입니다. 균형이 유지되면 트리의 높이는 항상 O(log n)으로 보장됩니다. 자가 균형 트리(self-balancing tree)는 삽입/삭제가 일어날 때마다 이 균형 조건이 깨졌는지 확인하고, 깨졌다면 회전(rotation)이라는 연산으로 트리의 모양을 재조정합니다.

회전은 노드 몇 개의 부모-자식 관계를 재배치하는 국소적인 연산으로, 비용은 O(1)이지만 삽입/삭제 경로를 따라 여러 번 연쇄적으로 발생할 수 있어 전체적으로는 삽입/삭제 시간복잡도에 상수 배 정도의 오버헤드를 추가합니다. 즉, 균형을 유지하는 대가로 "삽입/삭제가 항상 O(log n)이라는 보장"을 얻지만, 대신 매번 더 많은 부가 작업(회전, 균형 정보 갱신)을 치릅니다.

AVL 트리 vs Red-Black 트리 — 무엇이 다른가

AVL 트리는 모든 노드에서 왼쪽/오른쪽 서브트리의 높이 차이가 최대 1을 넘지 않도록 엄격하게 균형을 강제합니다. 이 덕분에 트리가 항상 거의 완벽하게 균형 잡혀 있어 조회(lookup)가 이론적으로 가장 빠른 축에 속합니다. 하지만 균형 조건이 엄격한 만큼 삽입/삭제 시 균형이 깨지는 빈도가 높고, 그때마다 회전을 더 자주 수행해야 합니다.

Red-Black 트리는 노드에 빨강/검정 색을 부여하고 몇 가지 규칙(루트는 검정, 빨강 노드의 자식은 검정, 모든 리프까지의 검정 노드 개수는 동일 등)을 지키는 방식으로 느슨하게 균형을 유지합니다. AVL만큼 완벽하게 균형 잡히지는 않지만("최악의 경우도 log n을 넘지 않는다" 정도로 보장), 그 대신 삽입/삭제 시 회전과 재조정 횟수가 AVL보다 적습니다.

정리하면: AVL은 조회 성능에 더 유리하고, Red-Black은 삽입/삭제 성능에 더 유리합니다. 이것이 "둘 다 O(log n)"이라는 Big-O 뒤에 숨은 실무적 차이입니다.

왜 면접에서 이 질문을 묻는가

면접관이 보는 신호

"BST는 O(log n)이다"는 결론은 코딩 테스트 준비생도 다 압니다. 시니어에게 이 질문을 던지는 이유는 다음을 확인하기 위해서입니다.

  • O(log n)이라는 숫자 뒤의 전제 조건을 아는가: 그 전제(균형)가 언제, 왜 깨지는지 설명할 수 있는가.
  • 입력 데이터의 특성이 자료구조 성능을 좌우한다는 것을 아는가: 코드가 아니라 데이터 순서가 문제일 수 있다는 진단 능력.
  • "균형 트리"를 하나의 뭉뚱그린 개념으로만 아는가, 세부 구현체별 trade-off까지 아는가: AVL과 Red-Black을 구분 없이 "그냥 균형 트리"로만 알면 실무 선택 근거를 설명하지 못합니다.
  • 배열/해시/트리 사이에서 "왜 트리인가"를 정당화할 수 있는가: 정렬 유지와 범위 검색이 필요할 때만 트리가 정답이라는 것을 아는가.

주니어는 "BST는 O(log n) 탐색이 가능합니다"라고 답합니다. 시니어는 "그 log n은 트리가 균형 잡혀 있다는 전제 위에 있고, 그 전제가 깨지면 어떻게 되는지, 그리고 그것을 막기 위한 구현체마다 어떤 trade-off가 있는지"까지 설명합니다.

실무 연결

시나리오 1 — 정렬된 로그 데이터를 BST에 넣었더니 조회가 느려짐

타임스탬프 순으로 계속 쌓이는 이벤트 로그를 일반 BST(자가 균형 없는 구현)에 넣고 특정 구간을 조회하는 기능을 만들었는데, 데이터가 쌓일수록 조회 시간이 선형으로 늘어난다.

진단 방향: 삽입 순서가 이미 정렬되어 있는지 확인한다. 정렬된 순서로 삽입되고 있다면 트리가 편향되어 사실상 연결 리스트가 되어 있을 가능성이 높다. 해결책: 일반 BST 대신 처음부터 자가 균형 트리(Red-Black, AVL 기반의 표준 라이브러리 구조, 예: TreeMap/std::map)를 사용하거나, 삽입 전에 데이터를 무작위로 섞어 트리 모양을 개선하는 방법을 고려한다.

시나리오 2 — 정렬된 순서 순회와 범위 검색이 필요한 인메모리 인덱스

가격 범위나 시간 범위로 데이터를 조회해야 하는 인메모리 캐시/인덱스를 설계해야 한다.

선택: 균형 트리 기반 정렬 맵(TreeMap/std::map 계열) — 해시 테이블은 순서 정보가 없어 범위 검색이나 최솟값/최댓값 조회에 적합하지 않다. 균형 트리는 O(log n) 조회/삽입과 함께 정렬된 순서 순회, 범위 검색(subMap/lowerBound 류 연산)을 지원하므로 "정렬이 필요한가"가 해시 대비 트리를 선택하는 결정적 기준이 된다.

시나리오 3 — 쓰기가 잦은 실시간 랭킹 시스템 vs 조회가 압도적인 참조 테이블

실시간으로 순위가 계속 갱신되는 랭킹 시스템과, 한 번 구축된 후 조회만 계속 발생하는 참조용 정렬 인덱스는 요구사항이 다르다.

선택 기준: 삽입/삭제가 잦은 랭킹 시스템은 회전 비용이 적은 Red-Black 계열 구조(대부분의 언어 표준 라이브러리의 정렬 맵이 Red-Black 기반)가 유리하고, 조회가 압도적으로 많고 구축 이후 변경이 드문 참조 테이블은 AVL처럼 더 엄격하게 균형 잡힌 구조나 아예 정적 배열 + 이분 탐색으로 단순화하는 편이 나을 수 있다.

시나리오 4 — 디스크 기반 DB 인덱스는 왜 이진 트리가 아니라 B-Tree/B+Tree인가

인메모리 균형 트리를 그대로 디스크 기반 DB 인덱스에 적용하면 성능이 기대만큼 나오지 않는다.

이유: 이진 트리는 노드당 자식이 최대 2개뿐이라 데이터가 많아지면 트리 높이(디스크 접근 횟수)가 커진다. 디스크 I/O는 메모리 접근보다 수천 배 느리기 때문에, 트리 높이를 최소화하는 것이 곧 디스크 접근 횟수를 줄이는 것과 직결된다. B-Tree/B+Tree는 노드 하나에 훨씬 많은 자식(수십~수백)을 담아 트리 높이를 극단적으로 낮춰, 적은 디스크 접근으로 원하는 데이터에 도달하도록 설계된다. 즉 "메모리에서의 최적"과 "디스크에서의 최적"은 서로 다른 자료구조를 요구한다는 것이 이 시나리오의 핵심 교훈이다.

답변 사고 순서

  1. 정렬/범위 검색이 필요한지부터 확인 — 필요 없다면 해시 테이블이 더 단순하고 빠를 수 있으므로 트리를 선택할 이유부터 정당화한다.
  2. 데이터가 정적인지 동적인지 확인 — 삽입이 드물고 조회가 압도적이면 정렬 배열 + 이분 탐색도 후보가 된다.
  3. 일반 BST의 함정(편향 트리) 언급 — 삽입 순서에 따라 O(log n)이 O(n)으로 무너질 수 있다는 것을 먼저 짚는다.
  4. 자가 균형 트리로 전환하는 이유 설명 — 회전 연산이 이 문제를 어떻게 해결하는지, 그 대가(부가 비용)가 무엇인지 설명한다.
  5. AVL vs Red-Black trade-off로 구체화 — 조회 중심이냐 삽입/삭제 중심이냐로 선택 기준을 좁힌다.
  6. 필요하다면 디스크 기반 구조(B-Tree류)까지 확장 — 메모리와 디스크의 성능 특성 차이를 알고 있다는 것을 보여준다.

핵심 답변 예시

질문: "정렬된 데이터를 다룰 때 배열, BST, balanced tree를 어떻게 비교합니까?"

짧은 버전 (30초)

"데이터가 거의 바뀌지 않고 조회만 많다면 정렬된 배열과 이분 탐색이 가장 단순하고 빠릅니다. 삽입/삭제가 자주 일어나면 배열은 재배치 비용이 크기 때문에 트리가 유리한데, 일반 BST는 삽입 순서에 따라 편향되어 O(n)까지 나빠질 수 있어 실무에서는 대부분 자가 균형 트리(AVL, Red-Black)를 씁니다. 조회가 더 중요하면 AVL, 삽입/삭제가 잦으면 Red-Black 쪽이 유리하다는 정도까지 구분해서 선택합니다."

깊이 있는 버전 (2분)

"세 가지를 비교할 때 저는 먼저 데이터의 변경 빈도를 봅니다. 삽입/삭제가 거의 없고 조회만 반복된다면 정렬된 배열에 이분 탐색을 쓰는 것이 메모리 locality도 좋고 구현도 단순해서 가장 좋은 선택입니다. 하지만 삽입/삭제가 자주 발생하면 배열은 중간에 값을 넣거나 뺄 때마다 뒤의 원소를 전부 밀어야 해서 O(n) 비용이 들기 때문에 트리 구조가 필요해집니다.

여기서 일반 BST를 그대로 쓰면 함정이 있습니다. BST의 O(log n)은 트리가 균형 잡혀 있다는 전제 위에 있는데, 만약 이미 정렬된 순서로 데이터를 삽입하면 트리가 한쪽으로만 길게 늘어진 편향 트리가 되어 사실상 연결 리스트와 같아지고 O(log n)이 O(n)으로 무너집니다. 그래서 실무에서는 삽입 순서와 무관하게 균형을 항상 보장하는 자가 균형 트리를 씁니다.

자가 균형 트리 중에서도 AVL과 Red-Black은 성격이 다릅니다. AVL은 균형 조건이 더 엄격해서 트리가 항상 거의 완벽하게 균형 잡혀 있어 조회 성능이 더 좋지만, 그만큼 삽입/삭제 때 회전이 더 자주 일어납니다. Red-Black은 균형 조건을 조금 느슨하게 잡아 회전 횟수를 줄이는 대신 조회 성능은 AVL보다 약간 떨어질 수 있습니다. 그래서 저는 조회가 압도적으로 많은 워크로드에는 AVL 계열을, 삽입/삭제가 잦은 워크로드에는 Red-Black 계열을 우선 고려하고, 실무에서는 대부분 언어 표준 라이브러리의 정렬 맵(대개 Red-Black 기반)을 신뢰해서 직접 구현하지는 않습니다."

배열 · BST · 균형 트리 상세 비교

기준 정렬된 배열 일반 BST (균형 없음) 자가 균형 트리 (AVL/Red-Black)
조회(탐색) O(log n) — 이분 탐색 평균 O(log n), 최악 O(n) 항상 O(log n) 보장
삽입/삭제 O(n) — 뒤 원소 재배치 평균 O(log n), 최악 O(n) O(log n) + 회전 오버헤드
정렬 순서 유지 항상 유지 중위 순회로 항상 유지 중위 순회로 항상 유지

Senior-level Follow-up 질문

Q1. "사용자 입력을 그대로 BST 키로 받는 시스템이 있다면, 편향 트리로 인한 성능 저하가 악의적으로 유발될 수 있습니까?"
가능합니다. 공격자가 삽입 순서를 제어할 수 있다면 일부러 정렬된 순서(또는 트리를 한쪽으로 늘어뜨리는 순서)로 값을 삽입해 조회를 O(n)까지 저하시키는 알고리즘 복잡도 공격(algorithmic complexity attack)을 시도할 수 있습니다. 해시 DoS와 개념적으로 같은 부류의 위협입니다. 방어책은 애초에 균형을 삽입 순서와 무관하게 보장하는 자가 균형 트리를 쓰는 것이며, 이는 성능 문제이자 동시에 가용성 보안 문제로 이어질 수 있다는 점을 인식하는 것이 중요합니다.
Q2. "언어 표준 라이브러리가 이미 균형 트리를 제공하는데, 왜 직접 구현하는 방법까지 알아야 합니까?"
평소에는 표준 구현을 신뢰해서 쓰는 것이 맞습니다. 하지만 회전 연산의 원리를 이해하고 있어야 "왜 이 자료구조가 이 성능 특성을 갖는지", "어떤 입력 패턴에서 예상과 다르게 동작할 수 있는지"를 근본 원리로 설명할 수 있습니다. 장애 상황에서 표준 라이브러리 자체의 한계(예: 특정 버전의 알려진 성능 이슈)를 의심해야 할 때, 내부 동작을 모르면 "라이브러리가 원래 그렇다"는 결론 이상으로 진단을 진행하지 못합니다.
Q3. "균형 트리 대신 해시 테이블을 쓰면 항상 더 빠르지 않습니까? 왜 트리를 쓰는 경우가 있습니까?"
단순 키-값 조회만 필요하다면 해시 테이블의 평균 O(1)이 트리의 O(log n)보다 빠릅니다. 하지만 해시 테이블은 키를 해시값 기준으로 흩어 저장하기 때문에 순서 정보를 잃습니다. 범위 검색("A 이상 B 이하인 값 모두"), 최솟값/최댓값, 정렬된 순서로 순회하는 기능이 필요하면 해시 테이블로는 불가능하거나 매우 비효율적입니다. 이때는 "더 빠른 자료구조"가 아니라 "요구하는 연산을 지원하는 자료구조"라는 기준으로 트리를 선택해야 합니다.
Q4. "멀티스레드 환경에서 균형 트리를 동시에 갱신하면 어떤 문제가 생깁니까?"
회전 연산은 여러 노드의 포인터를 함께 바꾸는 작업이라, 한 스레드가 회전을 진행하는 도중 다른 스레드가 같은 영역을 읽거나 쓰면 트리 구조가 깨지거나 무한 루프, 잘못된 조회 결과로 이어질 수 있습니다. 실무에서는 트리 전체에 락을 걸어 단순하게 보호하거나, 동시성이 중요한 경우 락 없이 동작하는 스킵 리스트(skip list) 기반 정렬 맵(예: ConcurrentSkipListMap류)처럼 트리와 다른 확률적 자료구조로 대체하는 선택지도 고려합니다.

좋은 답변 vs 약한 답변

강한 답변 (senior level)
  • O(log n)이 트리의 균형을 전제로 한다는 것을 먼저 짚음
  • 편향 트리가 발생하는 조건(정렬된 순서 삽입)을 구체적으로 설명
  • AVL과 Red-Black을 조회/삽입 trade-off로 구분해서 설명
  • 해시 테이블과의 선택 기준을 "속도"가 아니라 "지원 연산"으로 설명
  • 필요하면 디스크 기반 B-Tree까지 확장해 메모리/디스크 차이를 인식
약한 답변 (주니어 패턴)
  • "BST는 O(log n)입니다"라고만 답하고 전제 조건을 언급하지 않음
  • 편향 트리 개념을 모르거나 왜 발생하는지 설명하지 못함
  • AVL과 Red-Black을 "둘 다 균형 트리"로만 뭉뚱그려 구분하지 못함
  • 해시 테이블이 항상 트리보다 빠르다고 오해하고 언제 트리가 필요한지 설명 못함
  • 인메모리 구조와 디스크 기반 인덱스 구조를 구분하지 못함

약한 답변이 약한 이유: "O(log n)"이라는 결론을 외웠지만 그 결론이 성립하는 조건(균형)과 그 조건이 깨지는 실무 상황(정렬된 입력, 편향)을 분해하지 못합니다. 시니어 답변은 이론적 복잡도와 실제 데이터 패턴이 만났을 때 벌어지는 구체적 결과까지 설명합니다.

흔한 오해와 바로잡기

오해 1: "BST는 항상 O(log n)이다"
사실: O(log n)은 트리가 균형 잡혀 있다는 전제 위에서 성립합니다. 삽입 순서에 따라 트리가 한쪽으로 늘어진 편향 트리가 되면 높이가 n에 가까워져 O(n)까지 나빠질 수 있습니다. "BST"와 "균형 잡힌 BST"는 다른 개념이며, 실무에서 필요한 것은 대개 후자입니다.
오해 2: "균형 트리는 항상 최선의 선택이다"
사실: 순서 정보나 범위 검색이 필요 없는 단순 키-값 조회라면 해시 테이블의 평균 O(1)이 균형 트리의 O(log n)보다 빠르고 구현도 단순합니다. 균형 트리는 "정렬 유지 + 범위 검색"이 필요할 때 그 대가(포인터 오버헤드, 회전 비용, 낮은 메모리 locality)를 지불할 가치가 있는 선택지이지, 무조건적인 정답이 아닙니다.
오해 3: "AVL이 항상 Red-Black보다 좋은 트리다 (더 엄격하게 균형 잡히니까)"
사실: AVL은 조회 성능에서 유리하지만 균형 조건이 엄격한 만큼 삽입/삭제 시 회전이 더 자주 발생해 쓰기 비용이 커집니다. Red-Black은 균형을 조금 느슨하게 유지하는 대신 삽입/삭제 시 회전 횟수가 적어 쓰기 중심 워크로드에 유리합니다. "더 엄격한 균형 = 항상 더 좋은 트리"가 아니라 "워크로드에 따라 유리한 트리가 다르다"는 것이 정확한 이해입니다.
오해 4: "트리 성능 문제는 항상 자료구조 구현이 잘못된 것이다"
사실: 구현이 정확한 일반 BST라도 입력 데이터의 삽입 순서 때문에 편향이 생겨 성능이 저하될 수 있습니다. 이는 코드 버그가 아니라 자료구조 선택(자가 균형 여부)과 입력 데이터 패턴의 상호작용에서 오는 문제입니다. 이런 문제는 단위 테스트만으로는 잘 드러나지 않고, 실제 운영 데이터의 삽입 순서 패턴을 분석하거나 부하 테스트로 재현해야 발견되는 경우가 많습니다.

자기 점검 Rubric

아래 항목에 스스로 답할 수 있으면 Day 5를 완료한 것입니다.

  • BST의 O(log n)이 트리 높이, 즉 균형을 전제로 한다는 것을 설명할 수 있다.
  • 편향 트리가 어떤 삽입 패턴에서 발생하고 왜 O(n)까지 나빠지는지 설명할 수 있다.
  • 회전(rotation) 연산이 균형을 어떻게 복원하는지, 그 대가가 무엇인지 설명할 수 있다.
  • AVL과 Red-Black의 trade-off를 조회 중심/쓰기 중심 워크로드로 구분해 설명할 수 있다.
  • 해시 테이블 대신 균형 트리를 선택해야 하는 조건(정렬, 범위 검색)을 설명할 수 있다.
  • 인메모리 균형 트리와 디스크 기반 B-Tree/B+Tree가 왜 다른 설계인지 설명할 수 있다.
  • 약한 답변과 강한 답변의 차이를 본인 말로 설명할 수 있다.
