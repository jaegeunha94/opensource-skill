자료구조 10년차 이상 개발자 Interview

Day 1 — 자료구조 선택 프레임워크

구현을 암기하는 것이 아니라, 요구사항에서 자료구조 선택까지 사고하는 방법을 훈련한다.

면접 질문

핵심 질문 1
요구사항을 받았을 때 어떤 기준으로 자료구조를 선택합니까?
핵심 질문 2
O(1) 검색이 필요한 경우라면 무조건 해시 테이블을 쓰면 됩니까? 그렇지 않다면 어떤 경우에 선택이 달라집니까?
핵심 질문 3
Big-O 복잡도가 같은 두 자료구조가 있을 때, 실무에서 성능 차이가 나는 이유는 무엇입니까?

전제 개념 — 여기서부터 읽으면 됩니다

Big-O 복잡도가 전부가 아닌 이유

Big-O는 입력 크기 n이 매우 클 때 연산 횟수가 어떻게 증가하는지를 나타내는 기호입니다. O(1), O(log n), O(n) 등으로 표현합니다.

그런데 면접에서 "해시 테이블은 O(1) 검색이라 최고 아닌가요?"라고 답하면 탈락합니다. 이유는 세 가지입니다.

  • 상수 인자: O(1)이라도 해시 함수 계산 비용이 크면 O(log n) BST보다 느릴 수 있습니다.
  • 최악 케이스: 해시 충돌이 집중되면 O(n)으로 degradation이 발생합니다.
  • 메모리 locality: 해시 테이블은 메모리 여기저기 흩어져 있어서 CPU 캐시를 잘 쓰지 못합니다. 반면 배열은 연속 메모리라 캐시 효율이 높습니다.

그래서 10년차 이상 면접에서는 Big-O에 더해 상수 인자, 최악 케이스, 메모리 특성, 구현 복잡도까지 묶어서 판단하는지를 봅니다.

메모리 locality란

CPU 캐시는 자주 쓰이는 데이터를 메모리에서 미리 가져와 L1/L2 캐시에 올려두는 구조입니다. 연속된 메모리 주소를 읽을 때는 한 번에 캐시 라인(보통 64바이트) 단위로 가져오므로, 인접한 데이터를 반복해서 읽으면 캐시 hit rate이 높아집니다.

배열은 원소가 메모리에 연속으로 붙어 있어서 순차 탐색 시 캐시 효율이 매우 좋습니다. 연결 리스트는 노드마다 포인터로 연결되어 메모리가 흩어지므로, 탐색마다 캐시 miss가 발생해 실측 성능이 배열보다 훨씬 느릴 수 있습니다.

amortized cost (상각 비용)

동적 배열(Python list, Java ArrayList)은 크기를 초과하면 2배 크기의 새 배열을 할당하고 모든 원소를 복사합니다. 이 복사는 O(n)이지만 자주 일어나지 않습니다.

예를 들어 n번 push 중 복사는 log₂n 번만 발생하므로, 총 복사 비용 O(n)을 n번으로 나누면 append 하나당 상각 비용은 O(1)입니다. 이를 amortized O(1)이라 합니다.

면접에서 "append가 항상 O(1)이냐"고 물으면 "amortized O(1)"이라고 답하고, 실시간 응답이 중요한 시스템(레이턴시 spike 금지)에서는 이 가끔 발생하는 O(n) 복사가 문제가 될 수 있음을 언급해야 합니다.

왜 면접에서 이 질문을 묻는가

면접관이 보는 신호

10년차 이상 개발자에게 기대하는 것은 "어떤 자료구조를 언제 쓰는지 외웠는가"가 아닙니다. 면접관은 다음을 확인합니다.

  • 요구사항을 연산 패턴으로 번역할 수 있는가: "검색이 많다"를 "key lookup 위주, 순서 무관"과 "range scan 위주, 정렬 필요"로 구분하는가.
  • 공간-시간 trade-off를 생각하는가: 속도를 얻기 위해 메모리를 얼마나 쓸 수 있는지, 그 한계를 인식하는가.
  • 시스템 운영 조건을 고려하는가: 단순 알고리즘 성능이 아니라 메모리 한계, GC 부담, 동시성, 캐시 성능까지 묶어서 판단하는가.
  • 답변이 구체적인가: "상황에 따라 다릅니다"로 끝나지 않고, 어떤 조건에서 어떻게 달라지는지 설명하는가.

주니어는 Big-O 테이블을 외워서 답합니다. 시니어는 입력 특성과 운영 조건을 먼저 묻거나 가정하고, 그 조건 아래서 선택을 근거로 설명합니다.

실무 연결

시나리오 1 — 사용자 세션 저장

수백만 사용자의 세션 토큰을 메모리에 저장하고, token → user_id를 O(1)로 조회해야 한다.

선택: HashMap(해시 테이블) — 정렬 순서 불필요, key lookup만 필요. 단, 충돌 관리와 load factor 모니터링이 필요. Redis의 경우 hash 자료형 사용, memory fragmentation 모니터링 추가.

시나리오 2 — 리더보드 Top-K

실시간으로 점수가 업데이트되는 게임 리더보드에서 상위 100명을 빠르게 조회해야 한다.

선택: min-Heap 크기 100 + HashMap — Heap으로 상위 K 유지, HashMap으로 점수 업데이트 위치 추적. 단순 정렬이나 전체 정렬 배열은 업데이트마다 O(n log n) 발생.

시나리오 3 — 검색 자동완성

사용자 타이핑에 따라 접두사(prefix)가 같은 단어 목록을 빠르게 반환해야 한다.

선택: Trie — prefix 기반 검색에 최적. 단, 메모리 사용량이 크므로 compressed trie(radix tree)나 Redis의 Sorted Set 기반 대안과 비교가 필요.

시나리오 4 — 최근 N개 이벤트 유지

최근 1000개의 API 요청 로그만 메모리에 유지하면서 O(1)로 삽입/삭제해야 한다.

선택: 링 버퍼(circular array) — 배열 고정 크기 + head/tail 포인터. 연결 리스트 deque보다 메모리 효율 좋고 locality 우수. append와 pop이 O(1) amortized가 아니라 진짜 O(1).

답변 사고 순서

면접에서 자료구조 선택 질문을 받으면 아래 순서로 생각을 전개합니다. 이 순서 자체를 말해주는 것이 strong answer의 시작입니다.

  1. 연산 패턴 확인 — 주요 연산은 무엇인가? 검색, 삽입, 삭제, 정렬 유지, 범위 쿼리, prefix 매칭 중 어떤 것이 dominant한가?
  2. 접근 패턴 확인 — key 기반 랜덤 접근인가, 순서 기반 순차 접근인가, index 기반 접근인가?
  3. 입력 규모 확인 — 원소 수가 수백인가, 수백만인가? 데이터가 정적인가, 동적으로 변화하는가?
  4. 시간/공간 복잡도 비교 — 후보 자료구조의 Big-O를 비교하되, 상수 인자, 최악 케이스, amortized cost를 함께 언급.
  5. 메모리와 locality 고려 — 메모리 사용량이 중요한가? CPU 캐시 효율이 성능에 미치는 영향은?
  6. 운영/구현 비용 고려 — 구현 복잡도, 라이브러리 지원, 동시성 안전성, 메모리 단편화, GC 부담을 고려.
  7. 최종 선택과 trade-off 명시 — "이 조건 아래서는 X를 선택한다. 단, Y 조건이 달라지면 Z로 전환을 검토한다."

핵심 답변 예시

질문: "요구사항을 받았을 때 어떤 기준으로 자료구조를 선택합니까?"

짧은 버전 (30초)

"먼저 주요 연산이 무엇인지를 봅니다. 검색, 삽입, 삭제, 범위 쿼리, 순서 유지 중 어떤 것이 dominant한지에 따라 후보군이 달라집니다. 그 다음 입력 규모, 메모리 제약, 동시성 요건을 보고 최악 케이스와 상수 인자까지 고려해서 선택합니다. Big-O만 보지 않고, locality나 amortized 비용도 함께 판단합니다."

깊이 있는 버전 (2분)

"자료구조 선택의 첫 번째 기준은 연산 패턴입니다. key로 바로 조회하는 것이 dominant하면 해시 테이블을 먼저 고려하고, 정렬된 순서로 순회하거나 범위 검색이 필요하면 BST 계열을 고려합니다.

두 번째는 입력 규모와 변화 패턴입니다. 데이터가 고정이면 정렬된 배열과 이진 탐색도 경쟁력이 있고, 동적으로 자주 바뀌면 삽입/삭제가 효율적인 자료구조가 필요합니다.

세 번째는 메모리와 CPU 캐시 특성입니다. 예를 들어 연결 리스트는 이론적으로 O(1) 삽입이지만, 포인터 chasing으로 인한 캐시 miss 때문에 실측 순회 성능은 배열보다 훨씬 느릴 수 있습니다.

마지막으로 운영 조건을 봅니다. 실시간 응답이 중요한 시스템에서는 amortized O(1)이 아니라 worst-case O(1)이 필요합니다. 동시성 요건이 있으면 lock-free 자료구조나 분산 환경에서의 일관성 비용도 고려합니다.

이 네 가지 축을 묶어서 '이 조건이라면 X, 단 Y가 달라지면 Z로 전환'이라고 말할 수 있어야 한다고 생각합니다."

Senior-level Follow-up 질문

Q1. "실시간 p99 레이턴시가 10ms 이하여야 한다면, 자료구조 선택이 어떻게 달라집니까?"
amortized가 아닌 worst-case O(1)이 필요합니다. 동적 배열의 resize spike나 해시 테이블의 rehashing은 레이턴시 spike를 일으킵니다. 이런 경우 ring buffer처럼 크기가 고정된 자료구조, 또는 사전에 resize를 예약하거나 별도 스레드에서 처리하는 설계를 고려합니다. Java의 ConcurrentHashMap은 세그먼트 단위 rehashing으로 이 문제를 완화합니다.
Q2. "멀티스레드 환경에서 자료구조를 선택할 때 추가로 고려하는 것은 무엇입니까?"
thread-safe 여부, lock granularity, lock-free 대안 여부를 봅니다. Java의 HashMap은 thread-safe하지 않아서 ConcurrentHashMap이 필요합니다. 그런데 ConcurrentHashMap은 segment lock 방식으로 fine-grained locking을 쓰므로, 전체 map lock(Collections.synchronizedMap)보다 처리량이 높습니다. 더 나아가 read가 dominant하면 ReadWriteLock이나 lock-free 자료구조(CopyOnWriteArrayList 등)를 고려합니다.
Q3. "데이터가 메모리 한계를 초과할 수 있을 때는 어떻게 판단합니까?"
in-memory 자료구조를 disk-backed 자료구조나 외부 스토리지로 대체하거나, 분산 데이터 구조(Redis Cluster, LSM-tree 기반 DB 등)를 고려합니다. 예를 들어 대용량 정렬이 필요한 경우 B+Tree를 쓰는 DB 인덱스가 heap보다 현실적입니다. 이 경우 I/O cost 모델로 O(disk access)를 기준으로 판단합니다.
Q4. "같은 Big-O를 가진 두 자료구조인데 실측 성능이 다른 경험이 있습니까?"
연결 리스트와 배열의 순회 성능 차이가 대표적입니다. 연결 리스트는 O(n) 순회이지만 포인터 chasing으로 인해 캐시 miss가 발생해서, 같은 O(n) 순회인 배열에 비해 수 배 느릴 수 있습니다. 100만 원소 기준 실측 벤치마크에서 5~10배 차이가 나는 케이스도 있습니다. 이는 CPU 캐시 라인 크기(보통 64바이트)와 데이터 크기의 관계에서 비롯됩니다.

좋은 답변 vs 약한 답변

강한 답변 (senior level)
  • 연산 패턴을 먼저 확인하고, 그 패턴에 맞는 자료구조를 선택
  • Big-O에 더해 최악 케이스, 상수 인자, amortized cost, 메모리 locality를 언급
  • 선택한 자료구조의 약점과 그 약점이 언제 문제가 되는지 설명
  • "이 조건이 달라지면 다른 자료구조를 검토할 것"이라고 조건부 답변
  • 실무 예시(캐시, 스케줄러, 리더보드 등)와 연결해서 설명
약한 답변 (주니어 패턴)
  • "빠른 검색이 필요하면 해시, 순서가 필요하면 트리를 씁니다"로 끝냄
  • Big-O 테이블만 읊고 연산 패턴 분석 없이 선택
  • "상황에 따라 다릅니다"라고 하고 어떤 상황에서 달라지는지 설명 안 함
  • 메모리, 동시성, 운영 비용은 언급하지 않음
  • 선택한 자료구조의 단점을 모르거나 "단점이 없습니다"라고 답함

약한 답변이 약한 이유: 주니어는 "언제 무엇을 쓰는지"를 외웁니다. 시니어는 "왜 이것이고, 언제 이 선택이 위험하며, 무엇이 달라지면 재검토하는지"를 설명합니다. 면접관은 이 차이를 첫 30초 안에 감지합니다.

흔한 오해와 바로잡기

오해 1: "해시 테이블은 항상 O(1)이다"
사실: 평균 O(1)이지만 해시 충돌이 집중되면 O(n)으로 degradation됩니다. 특히 악의적인 입력(hash collision attack)으로 의도적으로 최악 케이스를 만들 수 있어서 Python은 hash seed를 랜덤화합니다. 또한 rehashing 시 O(n) 복사가 발생합니다.
오해 2: "Big-O가 같으면 실무 성능도 같다"
사실: 상수 인자와 메모리 locality가 다릅니다. 배열과 연결 리스트는 둘 다 O(n) 순회지만, 실측 성능은 캐시 효율 차이로 수 배 벌어질 수 있습니다.
오해 3: "균형 트리는 항상 O(log n)이 보장된다"
사실: Red-Black Tree, AVL Tree 등 균형 트리는 삽입/삭제 시 rebalancing 비용이 추가됩니다. log n 보장은 맞지만 상수 인자가 단순 BST보다 큽니다. 또한 구현이 복잡해서 운영 시 버그 위험이 있습니다.
오해 4: "연결 리스트는 삽입이 O(1)이라 배열보다 빠르다"
사실: 이미 삽입 위치를 알고 있을 때만 O(1)입니다. 위치를 찾는 탐색은 O(n)이며, 포인터 chasing으로 인한 캐시 miss로 인해 실제 삽입 포함 작업은 배열의 O(n) 이동보다 느린 경우가 많습니다.

  • 자료구조를 선택할 때 먼저 확인하는 4가지 축(연산 패턴, 입력 규모, 메모리/locality, 운영 비용)을 설명할 수 있다.
  • Big-O 복잡도 외에 실무 성능에 영향을 주는 두 가지 요소를 구체적으로 설명할 수 있다.
  • 해시 테이블의 "평균 O(1)"이 성립하는 전제와 그 전제가 깨지는 상황을 말할 수 있다.
  • amortized O(1) append가 실시간 응답 시스템에서 문제가 되는 이유를 설명할 수 있다.
  • 메모리 locality가 배열과 연결 리스트의 실측 성능 차이를 만드는 과정을 설명할 수 있다.

  • 약한 답변과 강한 답변의 차이를 본인 말로 설명할 수 있다.
