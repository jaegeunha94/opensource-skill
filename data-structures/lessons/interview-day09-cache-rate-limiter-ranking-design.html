자료구조 10년차 이상 개발자 Interview

Day 9 — 실무 자료구조 설계: 캐시·Rate Limiter·랭킹

LRU 캐시, Rate Limiter, 실시간 리더보드는 단일 자료구조로는 요구되는 모든 연산을 O(1)~O(log n)으로 만족시킬 수 없다. 왜 두 개 이상의 자료구조를 조합해야 하는지, 어떤 조합이 왜 성립하는지 설명한다.

면접 질문

핵심 질문 1
용량이 제한된 LRU(Least Recently Used) 캐시를 get과 put 모두 O(1)에 동작하도록 설계하려면 어떤 자료구조를 어떻게 조합하겠습니까?
핵심 질문 2
API Gateway에서 "사용자당 초당 100건"으로 요청을 제한하는 Rate Limiter를 설계한다면 어떤 자료구조를 쓰겠습니까? Sliding window와 token bucket 방식은 자료구조 관점에서 무엇이 다릅니까?
핵심 질문 3
실시간 게임 리더보드에서 "내 순위 조회"와 "점수 갱신"이 초당 수만 건씩 발생합니다. 어떤 자료구조 조합을 쓰겠습니까? 정렬된 배열이나 단순 힙으로는 왜 부족합니까?

전제 개념 — 여기서부터 읽으면 됩니다

왜 단일 자료구조로는 부족한가 — "연산 하나만 빠른" 자료구조의 한계

지금까지 배운 자료구조는 저마다 한 가지 접근 패턴에서만 강점을 가집니다. 해시 테이블은 키로 값을 찾는 것은 O(1)이지만 "가장 오래전에 쓰인 항목이 무엇인지"는 전혀 모릅니다. 배열이나 연결 리스트는 순서를 유지할 수 있지만 특정 원소를 찾으려면 O(n)이 걸립니다. 힙은 최솟값/최댓값을 O(1)에 알 수 있지만 "임의의 원소 하나의 값을 바꾸는 것"은 그 원소를 O(n) 탐색으로 찾아야 합니다(Day 6에서 다룬 decrease-key 문제).

실무 시스템은 대개 "조회도 빠르고, 순서/우선순위도 빠르게 유지되고, 갱신도 빨라야" 합니다. LRU 캐시는 "키로 즉시 조회"와 "가장 오래된 항목을 즉시 제거"를 동시에 요구하고, Rate Limiter는 "최근 N초 안의 요청 수를 즉시 집계"해야 하고, 리더보드는 "임의 유저의 순위를 즉시 조회"와 "점수 변경 시 순위를 즉시 재정렬"을 동시에 요구합니다. 이런 요구는 자료구조 하나로는 만족되지 않고, 서로 다른 강점을 가진 자료구조 두 개를 같은 데이터를 가리키도록 묶어서 각 연산을 각자 잘하는 자료구조에 위임하는 방식으로 풀립니다.

LRU 캐시 = 해시 맵 + 이중 연결 리스트

LRU 캐시가 필요한 연산은 세 가지입니다. (1) 키로 값을 즉시 찾는다, (2) 사용된 항목을 "가장 최근"으로 옮긴다, (3) 용량 초과 시 "가장 오래된" 항목을 즉시 제거한다. 해시 맵만 쓰면 (1)은 O(1)이지만 (2), (3)을 위한 "사용 순서"를 표현할 방법이 없습니다. 이중 연결 리스트만 쓰면 (2), (3)은 양 끝 노드 포인터로 O(1)이지만 (1)을 위해 리스트를 처음부터 훑어야 해 O(n)입니다.

해법은 해시 맵의 값으로 "값" 자체가 아니라 이중 연결 리스트 노드에 대한 참조(포인터)를 저장하는 것입니다. get(key)는 해시 맵으로 노드를 O(1)에 찾고, 그 노드를 리스트의 맨 앞(가장 최근)으로 O(1)에 옮깁니다(이중 연결 리스트는 앞뒤 포인터를 알고 있어 임의 노드를 O(1)에 떼어내 다시 붙일 수 있습니다 — 단일 연결 리스트로는 이전 노드를 몰라 불가능합니다). put(key, value)는 새 노드를 만들어 리스트 맨 앞에 넣고 해시 맵에 등록하며, 용량을 넘으면 리스트 맨 뒤(가장 오래된) 노드를 O(1)에 제거하고 해시 맵에서도 지웁니다. 두 자료구조가 "같은 노드 객체"를 공유하기 때문에 한쪽에서 위치를 찾고 다른 쪽에서 순서를 조작하는 것이 모두 O(1)로 끝납니다.

Rate Limiter = 시간 윈도우를 표현하는 자료구조 (카운터, 덱, 원형 버퍼)

Rate Limiter는 "최근 시간 구간 안의 요청 수를 세고, 한도를 넘으면 거절"하는 문제입니다. 가장 단순한 고정 윈도우(fixed window) 카운터는 정수 하나와 타임스탬프만 있으면 되지만, 윈도우 경계에서 순간적으로 한도의 2배 요청이 몰릴 수 있는 결함이 있습니다(예: 0:59에 100건, 1:00에 다시 100건 — 1초 사이에 200건이 통과).

슬라이딩 윈도우 로그(sliding window log)는 각 요청의 타임스탬프를 덱(deque)에 기록합니다. 요청이 오면 덱 앞쪽에서 "윈도우를 벗어난(너무 오래된) 타임스탬프"를 O(1)씩 제거하고(양쪽 끝에서 O(1) 삽입/삭제가 되는 덱이라 가능), 남은 개수가 한도 이하이면 새 타임스탬프를 덱 뒤에 추가해 허용합니다. 정확하지만 요청마다 타임스탬프를 저장해 트래픽이 많으면 메모리가 요청 수에 비례해 늘어난다는 단점이 있습니다.

슬라이딩 윈도우 카운터(sliding window counter)는 윈도우를 여러 개의 작은 버킷(예: 1초 단위)으로 쪼개 원형 버퍼(circular buffer)에 각 버킷의 요청 수만 저장합니다. 개별 타임스탬프 대신 버킷별 합계만 유지하므로 메모리가 버킷 개수(고정)로 제한되고, 정확도는 로그 방식보다 약간 떨어지지만 실무에서 충분한 근사치를 제공합니다. 토큰 버킷(token bucket)은 아예 다른 접근으로, "버킷에 토큰이 일정 속도로 채워지고 요청마다 토큰을 소비"하는 모델이라 카운터 하나와 마지막 충전 시각만 있으면 되어 자료구조가 가장 단순하지만, 순간적인 버스트(burst)를 허용하는 정책이 필요한지에 따라 선택이 갈립니다.

실시간 랭킹 = 균형 트리/스킵 리스트 + 해시 맵

리더보드는 두 종류의 질의를 동시에 빠르게 처리해야 합니다. (1) "유저 X의 현재 순위는?" (2) "유저 X의 점수를 Y로 갱신하라." 정렬된 배열은 순위 조회는 O(1)(인덱스 그대로)이지만 갱신 시 삽입 위치를 옮기는 데 O(n)이 걸립니다. 단순 힙(Day 6)은 최고 점수 조회는 빠르지만 "임의 유저의 순위(몇 등인지)"를 알아내려면 힙 구조상 O(n) 탐색이 필요합니다 — 힙은 오직 최솟값/최댓값 접근에만 최적화된 구조이기 때문입니다.

실무에서는 균형 이진 탐색 트리(order-statistics tree, 각 노드에 "왼쪽 서브트리 크기"를 함께 저장하는 확장된 BST/Red-Black tree)나 스킵 리스트(skip list)를 씁니다. 스킵 리스트는 정렬된 연결 리스트 위에 "여러 층의 지름길(express lane)" 포인터를 랜덤하게 추가한 구조로, 평균 O(log n)에 삽입/삭제/순위 조회가 모두 가능하면서도 균형 트리보다 구현과 동시성 제어(락 범위)가 단순하다는 실무적 장점이 있어 Redis의 Sorted Set이 내부적으로 스킵 리스트 + 해시 맵 조합을 씁니다. 해시 맵은 "유저 ID → 현재 점수"를 O(1)에 조회하는 역할을 맡고, 스킵 리스트(또는 order-statistics tree)는 "점수 순서 유지"와 "순위 계산"을 O(log n)에 맡아, 두 자료구조가 각자 잘하는 연산을 나눠 갖습니다.

왜 면접에서 이 질문을 묻는가

면접관이 보는 신호

단일 자료구조 문제는 "이름을 알면 답할 수 있는" 성격이 강하지만, 조합 설계 문제는 요구사항을 연산 단위로 쪼개고 각 연산을 담당할 자료구조를 스스로 찾아야 하므로 훨씬 더 실무에 가까운 신호를 줍니다.

  • 요구사항을 연산 목록으로 분해할 수 있는가: "LRU 캐시를 만들어라"를 듣고 곧바로 "get, put, evict가 각각 O(1)이어야 한다"로 번역하는가.
  • 단일 자료구조의 한계를 정확히 짚는가: "해시맵만 쓰면 왜 안 되는가", "힙만 쓰면 왜 순위 조회가 느린가"를 구체적으로 설명하는가.
  • 두 자료구조가 "같은 데이터"를 참조하도록 묶는 방법을 아는가: 값의 복사본이 아니라 포인터/참조를 공유해야 두 구조 사이의 갱신이 항상 일치한다는 것을 아는가.
  • 정확도와 자원(메모리/락 경합) 사이의 트레이드오프를 다루는가: 슬라이딩 윈도우 로그(정확하지만 메모리 큼)와 카운터(근사치지만 고정 메모리) 중 상황에 맞게 고르는가.

주니어는 "LRU는 해시맵이랑 링크드리스트 같이 쓰면 돼요"까지는 외워서 답합니다. 시니어는 왜 단일 연결 리스트가 아니라 이중 연결 리스트여야 하는지, 왜 해시맵의 값이 노드 참조여야 하는지, Rate Limiter에서 정확도와 메모리를 어떻게 저울질하는지, 분산 환경에서 이 조합들이 어떻게 깨지는지까지 설명합니다.

실무 연결

시나리오 1 — 애플리케이션 레벨 인메모리 캐시

DB 조회 결과를 서버 프로세스 메모리에 캐싱해 응답 속도를 높이되, 메모리가 한정돼 있어 오래 안 쓴 항목은 자동으로 밀어내야 한다.

선택: 해시 맵 + 이중 연결 리스트(LRU) — 대부분의 언어 표준 라이브러리(Java LinkedHashMap의 접근 순서 모드, Python OrderedDict/functools.lru_cache)가 내부적으로 이 조합을 구현해 제공한다. 다만 멀티스레드 환경에서는 리스트 재배치(get 시 노드를 맨 앞으로 옮기는 연산)에도 쓰기 락이 필요해, 읽기가 압도적으로 많은 워크로드에서는 이 락 경합 자체가 병목이 될 수 있다는 것을 실무에서는 고려해야 한다. 이 때문에 근사 LRU(예: Clock 알고리즘, 샘플링 기반 근사)를 쓰는 시스템도 많다.

시나리오 2 — API Gateway의 사용자별 요청 제한

공용 API를 외부에 노출하면서 한 사용자가 과도한 요청으로 서버를 독점하지 못하도록 "사용자당 분당 1000건"을 제한해야 한다.

선택: Redis + 슬라이딩 윈도우 카운터(원형 버퍼) 또는 토큰 버킷 — 단일 서버가 아니라 여러 API Gateway 인스턴스가 요청을 분산 처리하므로, Rate Limiter 상태를 각 서버 로컬 메모리에 두면 인스턴스마다 따로 카운트해 전체 한도가 무력화된다. 그래서 Redis 같은 공유 저장소에 카운터/버킷을 두고 원자적 증가 연산(INCR)으로 여러 인스턴스가 동시에 접근해도 정확히 세도록 만든다. 정확한 슬라이딩 윈도우 로그는 사용자당 요청 이력을 다 저장해야 해 대규모 트래픽에서 Redis 메모리 비용이 커지므로, 실무에서는 버킷 단위로 근사하는 카운터 방식이 흔히 쓰인다.

시나리오 3 — 실시간 게임/이커머스 랭킹

수백만 유저의 점수를 실시간으로 갱신하면서, "내 순위"와 "상위 100명" 조회가 동시에 초당 수만 건 발생한다.

선택: Redis Sorted Set (스킵 리스트 + 해시 맵) — Redis Sorted Set은 정확히 이 조합을 내장 자료구조로 제공한다. ZADD로 점수를 갱신하면 O(log n), ZRANK로 순위를 조회하면 O(log n), ZSCORE로 특정 유저 점수를 조회하면 해시 맵 경유로 O(1)이다. 직접 구현하는 대신 검증된 자료구조를 제공하는 인프라를 재사용하는 것 자체가 "10년차 이상"의 판단 — 모든 문제를 처음부터 직접 구현하지 않고, 이미 이 트레이드오프를 잘 푼 시스템을 아는 것도 실무 역량이다.

시나리오 4 — 자동완성 + 인기순 정렬

검색창 자동완성에서 접두사가 일치하는 후보들을 인기 점수 순으로 상위 K개만 보여줘야 한다.

선택: 트라이(Day 7) + 각 노드/서브트리에 미리 계산된 상위 K 힙 — 트라이만으로는 접두사 일치까지는 빠르지만(O(문자열 길이)), 그 아래 매달린 후보 수천 개를 매번 정렬하면 느리다. 각 트라이 노드에 "이 서브트리에서 가장 인기 있는 K개"를 미리 캐싱해두면(정적 데이터면 오프라인 배치, 동적이면 소형 힙을 노드마다 유지) 자동완성 응답이 O(문자열 길이 + K)로 끝난다. 이는 트라이(문자열 검색에 특화)와 힙(순위 유지에 특화)을 한 번 더 조합한 예시다.

답변 사고 순서

  1. 요구되는 연산을 모두 나열 — "조회", "삽입/갱신", "삭제", "순서/순위 조회"처럼 필요한 연산을 빠짐없이 적는다.
  2. 각 연산의 목표 시간복잡도를 정한다 — 어떤 연산이 초당 수만 건 이상 발생해 반드시 O(1)/O(log n)이어야 하는지 확인한다.
  3. 단일 자료구조로 시도해보고 실패 지점을 특정 — "해시맵만 쓰면 순서를 모른다", "힙만 쓰면 임의 원소 접근이 O(n)이다" 처럼 구체적으로 막히는 연산을 짚는다.
  4. 부족한 연산을 보완할 두 번째 자료구조를 선택 — 순서가 필요하면 연결 리스트/스킵 리스트/균형 트리, 시간 윈도우가 필요하면 덱/원형 버퍼를 고려한다.
  5. 두 자료구조가 같은 데이터를 참조하도록 연결 — 값의 복사가 아니라 포인터/참조 공유로 갱신 일관성을 보장하는 방법을 설명한다.
  6. 정확도-메모리-동시성 트레이드오프를 점검 — 정확한 방식(로그 기반)과 근사 방식(카운터/버킷 기반) 중 트래픽 규모에 맞는 것을 고른다.
  7. 분산/동시성 환경으로 확장했을 때 무엇이 깨지는지 언급 — 단일 프로세스 가정이 여러 서버로 나뉘면 왜 실패하고 무엇으로 대체하는지(Redis 등 공유 저장소, 원자적 연산) 설명한다.

핵심 답변 예시

질문: "LRU 캐시를 get/put 모두 O(1)에 동작하도록 설계하려면 어떻게 하겠습니까?"

짧은 버전 (30초)

"해시 맵과 이중 연결 리스트를 조합합니다. 해시 맵은 키에서 이중 연결 리스트 노드로의 참조를 저장해 O(1) 조회를 담당하고, 이중 연결 리스트는 사용 순서(최근 사용 순)를 유지합니다. get 시 해당 노드를 O(1)에 찾아 리스트 맨 앞으로 옮기고, put 시 용량을 넘으면 리스트 맨 뒤(가장 오래된) 노드를 O(1)에 제거합니다. 단일 연결 리스트가 아니라 이중 연결 리스트를 쓰는 이유는, 임의 노드를 제거하려면 그 이전 노드를 알아야 하는데 단일 연결로는 그걸 알 수 없기 때문입니다."

깊이 있는 버전 (2분)

"먼저 필요한 연산을 나눠봅니다. get(key)은 값을 O(1)에 반환해야 하고, 동시에 그 항목을 '가장 최근 사용'으로 표시해야 합니다. put(key, value)은 값을 O(1)에 갱신하거나 새로 넣어야 하고, 용량 초과 시 '가장 오래 사용 안 된' 항목을 O(1)에 제거해야 합니다. 해시 맵 하나만 쓰면 조회는 O(1)이지만 사용 순서를 표현할 방법이 없어 eviction 대상을 찾으려면 전체를 훑어야 합니다. 배열이나 단일 연결 리스트만 쓰면 순서는 유지되지만 키로 위치를 찾으려면 O(n) 탐색이 필요합니다.

그래서 두 구조를 묶습니다 — 이중 연결 리스트로 사용 순서를 유지하고(맨 앞 = 가장 최근, 맨 뒤 = 가장 오래됨), 해시 맵은 값 대신 그 리스트 노드에 대한 참조를 저장합니다. get이 호출되면 해시 맵으로 노드를 O(1)에 찾고, 그 노드를 리스트에서 떼어내(이중 연결이라 앞뒤 포인터를 바로 알아 O(1)) 맨 앞에 다시 붙입니다. put이 호출되면 새 노드를 만들어 맨 앞에 넣고 해시 맵에 등록하며, 용량을 넘으면 리스트 꼬리 노드를 제거하고 그 키를 해시 맵에서도 지웁니다. 핵심은 두 구조가 '같은 노드 객체'를 참조로 공유한다는 점이라, 한쪽에서 위치를 찾고 다른 쪽에서 순서를 조작해도 항상 일관됩니다.

실무로 확장하면, 멀티스레드 환경에서는 get조차 리스트를 갱신하는 쓰기 연산이라 락이 필요해 읽기 위주 워크로드에서 병목이 될 수 있고, 여러 서버에 캐시를 나눠야 하는 규모에서는 이 자료구조 조합 자체를 각 노드가 독립적으로 운영하되 일관성은 포기하는 방향(로컬 캐시 + 짧은 TTL, 또는 분산 캐시 서버에 위임)으로 타협합니다."

시스템별 자료구조 조합 비교

시스템 필요 연산 단일 구조의 실패 지점 조합 복잡도
LRU 캐시 조회, 최근 사용 갱신, 오래된 항목 제거 해시맵: 순서 없음 / 리스트: 조회 O(n) 해시 맵 + 이중 연결 리스트 get/put 모두 O(1)
Rate Limiter (정확) 최근 N초 요청 수 집계 단일 카운터: 윈도우 경계 버스트 허용 타임스탬프 덱(sliding window log) 요청당 O(1) 상각, 메모리 O(요청 수)
Rate Limiter (근사) 최근 N초 요청 수 집계 고정 윈도우: 경계 결함 / 로그: 메모리 부담 버킷 단위 원형 버퍼 O(1), 메모리 O(버킷 수, 고정)
실시간 랭킹 순위 조회, 점수 갱신 정렬 배열: 갱신 O(n) / 힙: 순위 조회 O(n) 스킵 리스트(또는 order-statistics tree) + 해시 맵 갱신/순위조회 O(log n), 점수조회 O(1)
자동완성 인기순 접두사 검색, 상위 K 정렬 트라이만: 서브트리 정렬 매번 O(n log n) 트라이 + 노드별 소형 힙(top-K 캐시) O(문자열 길이 + K)

Senior-level Follow-up 질문

Q1. "이 LRU 캐시를 여러 서버(수평 확장된 API 서버)에 걸쳐 일관되게 유지하려면 어떻게 합니까?"
단일 프로세스 메모리 안의 해시맵+이중연결리스트 조합은 애초에 분산 환경을 가정하지 않습니다. 확장 방법은 크게 두 갈래입니다. 첫째, 각 서버가 독립적인 로컬 LRU 캐시를 유지하되 일관성을 포기하고 짧은 TTL로 오차를 제한하는 방식(단순하지만 캐시 적중률이 서버마다 갈림). 둘째, Redis 같은 외부 캐시 서버로 캐시 자체를 중앙화해 모든 API 서버가 같은 저장소를 바라보게 하는 방식(일관성은 확보되지만 네트워크 왕복 비용과 그 서버의 병목/장애가 전체에 영향)입니다. 트래픽 패턴과 일관성 요구 수준에 따라 로컬 캐시(L1) + 분산 캐시(L2)를 계층화하는 것이 실무에서 흔한 절충입니다.
Q2. "슬라이딩 윈도우 로그 방식은 정확하지만 왜 대규모 트래픽에서 잘 안 씁니까?"
사용자마다, 요청마다 타임스탬프를 저장해야 하므로 메모리 사용량이 (사용자 수) × (윈도우 시간 동안의 요청 수)에 비례해 커집니다. 초당 수만 건의 요청을 처리하는 서비스에서 이 방식을 그대로 쓰면 Redis 같은 공유 저장소의 메모리 비용과 각 요청마다 오래된 항목을 정리(trim)하는 연산 비용이 함께 커져 Rate Limiter 자체가 병목이 됩니다. 그래서 실무에서는 정확도를 약간 희생하는 대신 메모리를 버킷 수로 고정하는 슬라이딩 윈도우 카운터나, 아예 개별 요청 기록을 남기지 않는 토큰 버킷을 선호합니다.
Q3. "Redis Sorted Set은 왜 균형 이진 트리 대신 스킵 리스트로 구현되어 있습니까?"
평균적인 시간복잡도는 둘 다 O(log n)으로 같지만, 스킵 리스트는 구현이 훨씬 단순합니다(회전 연산이 없고, 각 노드가 몇 개의 레벨을 가질지 무작위로 정해 삽입/삭제 로직이 직관적입니다). 또한 범위 질의(특정 순위 구간, 특정 점수 구간 조회)를 자연스럽게 순차 접근으로 처리할 수 있어 ZRANGE 같은 연산에 유리하고, 락이 필요한 구간을 좁게 유지하기가 균형 트리의 회전 연산보다 쉬워 동시성 제어 측면에서도 실무적으로 선호됩니다. Redis 개발진이 공개적으로 "구현 단순성과 성능의 균형" 때문에 스킵 리스트를 택했다고 밝힌 바 있습니다.
Q4. "이런 조합 자료구조에서 두 구조 사이의 상태가 어긋나면(예: 해시맵엔 있는데 리스트엔 없음) 어떻게 방지합니까?"
근본 원칙은 두 구조가 '같은 데이터의 복사본'이 아니라 '같은 객체에 대한 서로 다른 접근 경로'가 되도록 설계하는 것입니다. LRU 예시에서 해시맵의 값은 데이터 복사본이 아니라 연결 리스트 노드 자체에 대한 참조이므로, 노드를 지우면 두 구조에서 동시에(또는 한 트랜잭션 안에서) 사라집니다. 만약 언어/시스템 제약으로 값을 복사해야 한다면, 모든 갱신 연산을 하나의 함수/트랜잭션으로 캡슐화해 "삽입은 항상 두 구조에 함께, 삭제도 항상 두 구조에서 함께" 실행되도록 강제하고, 멀티스레드 환경이라면 두 구조에 대한 갱신을 하나의 락으로 묶어 원자성을 보장해야 합니다.

좋은 답변 vs 약한 답변

강한 답변 (senior level)
  • 요구사항을 먼저 연산 단위로 쪼개고, 각 연산의 목표 시간복잡도를 명시
  • 단일 자료구조가 실패하는 지점을 구체적인 연산으로 짚어 설명
  • 두 구조가 값 복사가 아니라 참조를 공유해야 하는 이유를 설명
  • 정확도-메모리-동시성 트레이드오프를 실제 수치/시나리오로 비교
  • 분산/멀티스레드 환경으로 확장했을 때 조합이 어떻게 깨지고 무엇으로 대체하는지까지 언급
약한 답변 (주니어 패턴)
  • "해시맵이랑 링크드리스트 같이 쓰면 됩니다"라고 결론만 말하고 왜 그런지 설명 못 함
  • 단일 연결 리스트와 이중 연결 리스트의 차이를 구분하지 못함
  • Rate Limiter를 그냥 "카운터 하나면 된다"고 답하고 윈도우 경계 문제를 모름
  • 랭킹 시스템에 정렬된 배열을 제안하고 갱신 비용을 고려하지 않음
  • 단일 서버 가정을 벗어나면 이 설계가 왜 깨지는지 전혀 다루지 못함

약한 답변이 약한 이유: 자료구조 이름을 나열하는 데서 멈추고, 그 조합이 "왜" 필요한지(단일 구조가 정확히 어떤 연산에서 실패하는지)와 "어떻게" 두 구조가 일관성을 유지하는지(참조 공유), 그리고 단일 프로세스를 벗어나면 무엇이 깨지는지를 설명하지 못합니다. 시니어 답변은 요구사항 → 연산 분해 → 실패 지점 → 조합 → 트레이드오프 → 확장 시 한계까지 하나의 사고 흐름으로 연결합니다.

흔한 오해와 바로잡기

오해 1: "자료구조를 여러 개 조합하면 복잡도만 늘어나고 손해다"
사실: 조합의 목적은 각 자료구조가 잘하는 연산만 맡겨 전체 시스템의 각 연산을 모두 빠르게 만드는 것입니다. 구현 복잡도는 늘지만, 단일 구조로는 애초에 불가능했던 "모든 핵심 연산이 O(1)~O(log n)"이라는 성능 목표를 달성하기 위한 필수적인 트레이드오프입니다.
오해 2: "LRU 캐시는 단일 연결 리스트로도 충분하다"
사실: get 연산이 임의의 노드를 리스트 중간에서 맨 앞으로 옮기려면, 그 노드를 리스트에서 떼어내기 위해 '이전 노드'를 알아야 합니다. 단일 연결 리스트는 다음 노드만 가리키므로 이전 노드를 알려면 처음부터 순회해야 해 O(n)이 됩니다. 이중 연결 리스트여야 임의 노드를 O(1)에 떼어낼 수 있습니다.
오해 3: "Rate Limiter는 카운터 하나로 충분하다"
사실: 카운터를 일정 주기로 초기화하는 고정 윈도우 방식은 윈도우 경계에서 순간적으로 한도의 최대 2배까지 요청이 통과할 수 있는 구조적 결함이 있습니다. 정확한 제한이 필요하면 시간 정보를 함께 담는 자료구조(덱, 원형 버퍼)가 필요합니다.
오해 4: "힙이 있으면 순위 조회도 당연히 빠르다"
사실: 힙은 최솟값/최댓값 접근에만 O(1)로 최적화된 구조입니다. "임의 원소가 몇 번째로 큰가"라는 순위 질문에 답하려면 힙 내부를 O(n) 탐색해야 하므로, 순위 조회가 잦은 워크로드에는 스킵 리스트나 order-statistics tree처럼 순서 정보를 유지하는 구조가 필요합니다.

자기 점검 Rubric

아래 항목에 스스로 답할 수 있으면 Day 9을 완료한 것입니다.

  • LRU 캐시가 해시 맵과 이중 연결 리스트를 조합해야 하는 이유를, 각 연산이 실패하는 지점부터 설명할 수 있다.
  • 왜 단일 연결 리스트가 아니라 이중 연결 리스트여야 하는지 설명할 수 있다.
  • 고정 윈도우, 슬라이딩 윈도우 로그, 슬라이딩 윈도우 카운터, 토큰 버킷의 자료구조 차이와 정확도-메모리 트레이드오프를 비교할 수 있다.
  • 실시간 랭킹에서 정렬 배열과 단순 힙이 각각 어떤 연산에서 실패하는지 설명할 수 있다.
  • 스킵 리스트 + 해시 맵 조합이 순위 조회와 점수 갱신을 모두 O(log n)/O(1)로 만드는 원리를 설명할 수 있다.
  • 조합 자료구조에서 값 복사가 아니라 참조 공유가 왜 일관성 유지에 필수적인지 설명할 수 있다.
  • 단일 프로세스용 설계가 분산/멀티스레드 환경으로 확장될 때 무엇이 깨지고 어떻게 보완하는지 설명할 수 있다.
  • 약한 답변과 강한 답변의 차이를 본인 말로 설명할 수 있다.
