자료구조 10년차 이상 개발자 Interview

Day 7 — 트라이와 문자열 검색

자동완성, IP 라우팅, 맞춤법 검사에서 왜 해시 테이블이 아니라 트라이를 쓰는지, 그리고 "문자마다 노드 하나"라는 구조가 어떤 메모리 비용을 치르는지 설명한다.

면접 질문

핵심 질문 1
자동완성을 구현할 때 trie를 쓰면 어떤 비용을 치릅니까? 해시 테이블로는 왜 자동완성을 구현하기 어렵습니까?
핵심 질문 2
영어 소문자만 다루는 사전이라면 트라이 노드마다 26개의 자식 포인터를 배열로 들고 있을 수 있습니다. 그런데 실제 사전 데이터는 대부분의 자식 슬롯이 비어 있을 텐데, 이 메모리 낭비를 어떻게 해결합니까?
핵심 질문 3
수백만 개의 라우팅 규칙 중 가장 긴 접두사가 일치하는 규칙을 찾아야 하는 IP 라우팅 테이블(longest prefix match)을 설계한다면, 트라이를 어떤 형태로 변형해서 쓰겠습니까?

전제 개념 — 여기서부터 읽으면 됩니다

트라이(trie)란 무엇인가 — "문자열 하나가 노드 하나"가 아니라 "문자 하나가 노드 하나"

트라이(trie, prefix tree)는 문자열들을 저장할 때 문자열 전체를 하나의 노드에 담는 것이 아니라, 문자 하나하나를 트리의 간선(edge)으로 표현하는 자료구조입니다. 예를 들어 "car", "care", "cat"이라는 세 단어를 저장하면, 루트에서 시작해 c → a까지는 세 단어가 경로를 공유하고, 그 다음 r과 t에서 갈라지며, "car"와 "care"는 r까지 공유하다가 "care"만 e로 한 번 더 이어집니다. 이렇게 공통 접두사(prefix)를 가진 문자열들이 트리의 앞부분 경로를 함께 쓴다는 것이 트라이의 핵심 아이디어입니다.

각 노드는 자신이 "완성된 단어의 끝"인지를 표시하는 종료 플래그(word-end flag)를 가집니다. "car"와 "care"를 모두 저장하면 r 노드와 e 노드 둘 다 종료 플래그가 켜져 있어야, "car"만 검색했을 때와 "care"까지 검색했을 때를 구분할 수 있습니다. 이 플래그가 없으면 트라이는 그냥 접두사 집합만 표현할 뿐 "어디까지가 실제로 저장된 단어인지"를 알 수 없습니다.

탐색과 삽입이 O(문자열 길이)인 이유 — 해시 테이블의 O(1)과는 다른 종류의 빠름

트라이에서 문자열을 검색하거나 삽입하는 비용은 문자열의 길이(m)에 비례합니다. 한 글자씩 루트부터 따라 내려가면서 해당 문자로 이어지는 자식이 있는지 확인하고, 없으면 새로 만들면 되기 때문입니다. 저장된 문자열의 개수(n)나 트라이 전체 크기와는 무관하게, 딱 그 문자열 길이만큼의 포인터 이동만 필요합니다.

여기서 중요한 구분이 있습니다. 해시 테이블의 "평균 O(1)"은 문자열 하나를 통째로 해시해서 버킷을 계산하는 것이므로, 문자열이 아무리 길어도 검색 자체는 "그 해시값의 버킷을 본다"는 한 번의 접근입니다(해시 계산 자체는 여전히 O(m)이 필요하지만, 데이터 규모 n과는 무관합니다). 반면 트라이는 "존재 여부"를 확인하는 데에도 문자를 한 글자씩 따라가야 합니다. 즉 정확히 일치하는 단어 하나를 찾는 것만 목적이라면 해시 테이블이 상수 인자 면에서 더 유리한 경우가 많습니다. 트라이가 진짜 강점을 보이는 지점은 "정확히 일치"가 아니라 "이 접두사로 시작하는 모든 단어"를 찾는 연산입니다.

Prefix search가 해시 테이블로는 왜 근본적으로 어려운가

해시 함수는 입력을 완전히 다른 값으로 흩어버리는(scrambling) 것이 설계 목표입니다. "car"의 해시값과 "care"의 해시값은 서로 아무 규칙 없이 멀리 떨어진 버킷에 저장될 수 있습니다. 그래서 해시 테이블은 "car로 시작하는 모든 단어를 찾아라"라는 질문에 답하려면 저장된 모든 키를 순회하며 접두사가 일치하는지 하나씩 확인하는 것 외에 방법이 없습니다. 이는 사실상 O(n)이며, 해시 테이블이 자랑하는 O(1) 조회의 이점이 완전히 사라집니다.

반면 트라이는 애초에 공통 접두사를 공유하는 문자열들이 같은 경로에 모여 있도록 설계된 구조입니다. "car"로 시작하는 모든 단어를 찾으려면 c → a → r 경로를 따라간 뒤, 그 노드를 루트로 하는 서브트리를 순회하기만 하면 됩니다. 즉 접두사 검색 비용이 "접두사 길이 + 결과로 나오는 단어 수"에 비례하지, 저장된 전체 단어 수에 비례하지 않습니다. 이것이 자동완성처럼 "접두사로 시작하는 후보들을 빠르게 찾아야 하는" 워크로드에서 트라이가 선택되는 근본 이유입니다.

메모리 비용의 실체 — "문자마다 노드"의 대가

트라이의 가장 큰 실무 비용은 메모리입니다. 알파벳이 26개인 영어 소문자만 다룬다고 해도, 노드마다 26개의 자식 포인터를 배열로 들고 있는 구현이라면 노드 하나당 26개의 포인터 슬롯이 필요합니다. 실제 사전 데이터에서는 한 노드가 26개 중 2~3개 자식만 갖는 경우가 대부분이라, 나머지 20개 이상의 슬롯은 null로 낭비됩니다. 여기에 유니코드 문자(한글, 한자, 이모지 등)까지 다뤄야 하면 가능한 다음 글자의 종류가 수만 개로 늘어나므로, 배열 기반 자식 표현은 사실상 불가능해지고 해시맵이나 정렬된 리스트로 자식을 표현해야 합니다. 이 경우 각 자식 접근에 해시 계산이나 이진 탐색 비용이 추가로 붙어, 트라이의 "O(길이)" 이론적 이점이 상수 인자로 많이 깎여 나갑니다.

또한 "car"와 "cart"처럼 긴 공유되지 않는 꼬리(suffix)가 이어지는 단어가 많으면, 노드 하나가 자식을 하나만 가지는 일렬로 늘어선 체인(chain)이 길게 생깁니다. 이런 체인은 압축 없이는 노드 하나당 포인터 오버헤드(자식 배열/맵 + 종료 플래그 + 부모 포인터 등)를 그대로 지불하게 되어, 원본 문자열을 그냥 배열에 저장하는 것보다 오히려 메모리를 더 많이 쓸 수 있습니다.

압축 트라이(Radix Tree / Patricia Trie) — 체인을 접어서 메모리를 아낀다

Radix tree(또는 Patricia trie, compressed trie)는 자식이 하나뿐인 노드들의 체인을 하나의 간선으로 합쳐서, 간선에 문자 하나가 아니라 문자열 조각(substring)을 저장하는 방식입니다. 예를 들어 "international"과 "internet"만 저장한다면, 일반 트라이는 i-n-t-e-r-n까지 한 글자씩 노드를 만들지만, radix tree는 공유되는 intern까지를 간선 하나로 압축하고, 그 다음 ational과 et로 분기하는 두 개의 간선만 만듭니다. 이렇게 하면 노드 수가 크게 줄어 메모리 오버헤드가 감소하지만, 대신 삽입/삭제 시 기존 간선을 쪼개거나 합치는 로직이 필요해 구현이 더 복잡해집니다. Linux 커널의 라우팅 테이블(FIB)이나 많은 IP 라우터의 longest prefix match 구현이 radix tree/Patricia trie를 쓰는 이유가 바로 이 메모리 효율 때문입니다.

왜 면접에서 이 질문을 묻는가

면접관이 보는 신호

"트라이는 문자열을 문자 단위로 저장하는 트리다"는 코딩 테스트 준비생도 압니다. 시니어에게 이 질문을 던지는 이유는 다음을 확인하기 위해서입니다.

  • 정확히 일치 검색과 접두사 검색을 구분해 자료구조를 고르는가: "문자열을 저장하고 찾는다"는 요구사항을 뭉뚱그리지 않고, 접두사 질의가 실제로 필요한지 먼저 확인하는가.
  • 트라이가 항상 메모리를 아낀다는 오해에 빠지지 않는가: 공유되는 접두사가 적은 데이터셋에서는 트라이가 오히려 원본보다 메모리를 더 쓸 수 있다는 것을 아는가.
  • 알파벳 크기와 데이터 특성에 따라 구현 방식을 조정하는가: 26개 소문자면 배열, 유니코드라면 해시맵/정렬 리스트로 자식을 표현해야 한다는 것을 아는가.
  • 실무에서 쓰이는 압축 기법(radix tree, DAWG)까지 알고 있는가: "트라이"라는 이름만 아는 것과 실제 프로덕션에서 쓰이는 변형을 아는 것의 차이.

주니어는 "트라이는 문자열 저장에 씁니다"라고 답합니다. 시니어는 "정확히 일치하는 검색만 필요하면 해시 테이블이 상수 인자 면에서 유리하고, 접두사 검색이나 정렬된 순회가 필요할 때 트라이를 씁니다. 다만 트라이는 노드당 포인터 오버헤드가 있어 공유되는 접두사가 적으면 메모리 이점이 사라지므로, 실무에서는 체인을 압축하는 radix tree를 더 많이 씁니다"까지 설명합니다.

실무 연결

시나리오 1 — 검색창 자동완성(autocomplete)

사용자가 "sea"까지 입력하면 "search", "seattle", "seasonal" 같은 후보 단어들을 빠르게 보여줘야 한다.

선택: 트라이 — "sea"라는 접두사 경로까지 내려간 뒤, 그 지점을 루트로 하는 서브트리를 순회(DFS)하면 후보 단어들을 얻을 수 있다. 실무에서는 여기서 한 걸음 더 나아가, 각 노드에 "이 서브트리에서 가장 인기 있는 top-k 단어"를 미리 캐시해두는 방식(예: 각 노드가 인기도 상위 5개를 들고 있는 heap을 유지)으로 순회 비용 자체를 줄인다. 순수 트라이 순회는 서브트리가 크면 느려질 수 있어, "얼마나 많은 후보 중 상위 몇 개만 보여줄 것인가"라는 실무 제약이 구현을 좌우한다.

시나리오 2 — IP 라우팅에서 longest prefix match

라우터는 목적지 IP 주소에 대해 여러 라우팅 규칙(예: 10.0.0.0/8, 10.1.0.0/16) 중 가장 구체적인(가장 긴 접두사) 규칙을 골라야 한다.

선택: 비트 단위 트라이(binary trie) 또는 radix tree — IP 주소를 비트 문자열로 보고 트라이를 구성하면, 주소를 따라 내려가면서 만나는 모든 종료 플래그(등록된 규칙) 중 가장 깊이 내려간(가장 긴 접두사) 것을 채택하면 된다. 규칙 수가 수백만 개에 달하는 백본 라우터에서는 일반 트라이의 노드 오버헤드가 감당하기 어려워, 체인을 압축한 radix tree(또는 더 나아가 multibit trie)를 사용해 메모리와 조회 속도를 동시에 관리한다. "정확한 매치가 아니라 가장 긴 접두사 매치"라는 요구사항 자체가 해시 테이블로는 표현할 수 없는 질의이기 때문에 트리 계열 구조가 필수적이다.

시나리오 3 — 맞춤법 검사기의 사전 조회

사용자가 입력한 단어가 사전에 있는지, 없다면 편집 거리 1~2 이내의 유사 단어를 제안해야 한다.

선택: 트라이 + 편집 거리 가지치기(pruning) — 단순히 사전에 있는지 없는지만 확인한다면 해시 테이블로 충분하지만, "유사 단어 제안"은 트라이를 DFS로 순회하면서 각 경로마다 현재까지의 편집 거리를 계산해, 이미 허용 범위를 넘은 경로는 더 내려가지 않고 가지를 쳐내는 방식(예: DP 기반 편집 거리 + 트라이 결합)으로 구현한다. 이는 사전 전체를 순회하며 각 단어와 편집 거리를 계산하는 것보다 훨씬 적은 연산으로 후보를 찾을 수 있다 — 공유된 접두사에 대한 편집 거리 계산 결과를 재사용할 수 있기 때문이다.

시나리오 4 — 파일 시스템 경로나 URL 라우팅 테이블

웹 프레임워크의 라우터가 /users/:id, /users/:id/posts, /users/settings 같은 경로 패턴 중 들어온 요청 경로와 가장 잘 맞는 것을 찾아야 한다.

선택: 경로 세그먼트 단위 트라이(radix tree 변형) — 많은 고성능 웹 프레임워크(예: 여러 Go/Node 라우터 구현체)가 URL 경로를 /로 나눈 세그먼트를 트리의 간선으로 쓰는 radix tree 기반 라우터를 사용한다. 이는 공통 경로 접두사(/users)를 공유하는 라우트가 많을 때 등록된 라우트 수에 비례해 선형 탐색하는 것보다 훨씬 빠르게 매칭할 수 있게 해주며, 동적 세그먼트(:id)와 와일드카드까지 자연스럽게 트리 분기로 표현할 수 있다.

답변 사고 순서

  1. 정확히 일치 검색인지, 접두사/범위 검색인지 구분 — 정확히 일치만 필요하면 해시 테이블이 상수 인자 면에서 더 유리할 수 있다는 것을 먼저 짚는다.
  2. 알파벳/문자 집합의 크기 확인 — 소문자 26개처럼 작으면 배열 기반 자식, 유니코드처럼 크면 해시맵/정렬 리스트 기반 자식으로 구현이 달라진다.
  3. 저장할 문자열들이 접두사를 얼마나 공유하는지 판단 — 공유되는 접두사가 적으면 트라이의 메모리 이점이 사라지므로 압축(radix tree)이나 다른 구조를 고려한다.
  4. 메모리 오버헤드를 구체적으로 언급 — 노드당 포인터/플래그 비용, 긴 체인이 압축 없이 생길 때의 낭비를 설명한다.
  5. 실무 변형 제시 — radix tree/Patricia trie로 체인을 압축하거나, DAWG로 공통 접미사까지 공유하는 방법을 제안한다.
  6. 대안과의 비교로 마무리 — 해시 테이블(정확 일치), 정렬된 배열/BST(범위 질의), 트라이(접두사 질의)를 워크로드에 따라 구분해 정리한다.

핵심 답변 예시

질문: "자동완성을 구현할 때 trie를 쓰면 어떤 비용을 치릅니까?"

짧은 버전 (30초)

"트라이는 문자를 한 글자씩 노드로 표현해서 공통 접두사를 공유하기 때문에, '이 접두사로 시작하는 모든 단어'를 빠르게 찾을 수 있습니다. 다만 노드마다 자식 포인터(알파벳이 작으면 배열, 유니코드처럼 크면 해시맵)와 종료 플래그를 저장해야 해서, 공유되는 접두사가 적은 데이터셋에서는 원본 문자열을 그냥 저장하는 것보다 메모리를 더 쓸 수 있습니다. 그래서 실무에서는 자식이 하나뿐인 체인을 압축한 radix tree를 더 많이 씁니다."

깊이 있는 버전 (2분)

"자동완성의 핵심 요구사항은 '정확히 일치하는 단어 찾기'가 아니라 '이 접두사로 시작하는 모든 후보 찾기'입니다. 해시 테이블은 해시 함수가 입력을 의도적으로 흩어버리기 때문에, 'sea'로 시작하는 단어들이 해시 공간 어디에 흩어져 있는지 알 방법이 없어 전체를 순회해야 합니다. 반면 트라이는 공통 접두사를 가진 문자열들이 애초에 같은 경로를 공유하도록 구성되어 있어서, 접두사 경로까지 내려간 뒤 그 서브트리만 순회하면 됩니다. 그래서 접두사 검색에서는 트라이가 구조적으로 맞는 선택입니다.

하지만 여기에는 분명한 비용이 있습니다. 노드마다 다음 글자로 이어지는 자식을 표현해야 하는데, 영어 소문자 26개처럼 알파벳이 작으면 배열로 표현할 수 있지만 대부분의 슬롯이 비어 낭비가 생기고, 유니코드처럼 알파벳이 크면 배열 자체가 불가능해 해시맵이나 정렬 리스트로 자식을 표현해야 해서 자식 접근마다 추가 비용이 붙습니다. 또한 공유되는 접두사가 적은 단어들(예: 서로 다른 언어의 고유명사)이 많으면 자식이 하나뿐인 노드가 길게 이어지는 체인이 생기는데, 압축 없이는 노드 하나하나의 포인터 오버헤드를 그대로 지불하게 되어 원본 문자열 배열보다 메모리를 더 씁니다.

그래서 저는 실무에서 순수 트라이보다 radix tree(Patricia trie)를 더 신뢰합니다. 자식이 하나뿐인 체인을 하나의 간선에 문자열 조각으로 압축해서 저장하기 때문에 노드 수가 크게 줄어들고, 이것이 IP 라우팅 테이블이나 많은 URL 라우터가 실제로 쓰는 방식입니다. 또한 자동완성에 순위(랭킹)까지 필요하다면, 각 노드에 서브트리 내 인기 상위 몇 개를 미리 캐시해서 순회 자체를 줄이는 최적화도 함께 고려합니다."

Senior-level Follow-up 질문

Q1. "트라이에서 여러 단어가 공통 접미사(suffix)를 공유하는 경우(예: 'reading', 'heading'이 모두 'ing'으로 끝남)는 어떻게 최적화합니까?"
일반 트라이는 접두사만 공유하고 접미사는 각 경로마다 따로 저장하기 때문에, 접미사가 겹쳐도 메모리를 아끼지 못합니다. 이 경우 DAWG(Directed Acyclic Word Graph, 방향 비순환 단어 그래프)를 씁니다. DAWG는 트라이의 동일한 서브트리(같은 접미사로 이어지는 구조)를 하나로 합쳐서 그래프로 만들기 때문에, 접두사와 접미사를 모두 공유해 트라이보다 훨씬 적은 노드로 같은 단어 집합을 표현할 수 있습니다. 다만 노드를 공유하기 때문에 특정 경로가 '진짜 단어의 끝'인지 판단하는 로직이 트라이보다 복잡해지고, 구성 알고리즘도 더 까다롭습니다. 사전 크기가 매우 크고 메모리가 제약된 임베디드 환경(예: 오프라인 사전 검색기)에서 고려할 선택지입니다.
Q2. "여러 스레드가 동시에 트라이에 단어를 삽입하면서 다른 스레드가 접두사 검색을 수행하면 어떤 문제가 생깁니까?"
삽입 중에는 새 노드를 만들고 부모의 자식 포인터/맵 엔트리를 갱신하는 여러 단계가 필요한데, 이 중간 상태를 다른 스레드가 읽으면 '아직 완성되지 않은 노드'를 만나 잘못된 결과를 낼 수 있습니다(예: 자식은 연결됐지만 종료 플래그는 아직 안 켜진 상태). 가장 단순한 방어는 트라이 전체 또는 서브트리 단위로 읽기-쓰기 락을 거는 것이지만, 삽입이 잦은 워크로드에서는 병목이 됩니다. 실무에서는 삽입을 배치로 모아 백그라운드에서 새 트라이(또는 트라이의 일부)를 구성한 뒤, 완성된 시점에 원자적으로 포인터를 교체하는 copy-on-write / 스냅샷 교체 방식을 쓰거나, 읽기가 압도적으로 많은 워크로드라면 락-프리 읽기와 드문 쓰기를 분리하는 구조(RCU 유사 패턴)를 고려합니다.
Q3. "한글처럼 초성/중성/종성으로 분해되는 문자를 트라이에 넣을 때 무엇을 고려해야 합니까?"
한글 완성형 문자를 그대로 유니코드 코드포인트 단위로 트라이에 넣으면 자식 후보가 11,000개가 넘는 완성형 음절 전체가 되어 자식 표현이 해시맵일 수밖에 없고, 자모 단위의 공통 접두사(예: 초성이 같은 단어들)를 활용하지 못합니다. 자동완성처럼 자모 단위 입력(타이핑 중간 상태)까지 지원해야 하는 서비스에서는 완성형 대신 초성/중성/종성으로 분해한 자모 시퀀스를 트라이의 단위로 쓰는 경우가 많습니다. 이렇게 하면 자식 알파벳 크기가 훨씬 줄어들고, "학교"를 입력 중 "ㅎ", "하", "학"처럼 조합 중간 단계에서도 접두사 검색이 자연스럽게 동작합니다. 다만 이는 도메인(검색어 자동완성 vs 사전 검색)에 따라 완성형이 더 적합할 수도 있어, "어떤 입력 단위로 접두사 매칭이 필요한가"를 먼저 확인해야 합니다.
Q4. "저장된 단어 수는 적은데 각 단어가 매우 길다면(예: DNA 서열, 긴 URL) 트라이가 여전히 좋은 선택입니까?"
단어 수가 적고 길이가 길면서 접두사 공유가 적다면, 트라이는 긴 체인을 그대로 노드로 펼쳐놓게 되어 메모리 이점 없이 오버헤드만 커집니다. 이런 경우 순수 트라이보다 (1) radix tree로 체인을 압축하거나, (2) 접두사 검색이 실제로 자주 필요한 게 아니라면 애초에 해시 테이블이나 정렬된 배열 + 이진 탐색으로 되돌아가는 것이 더 합리적입니다. "문자열이니까 트라이"가 아니라, 데이터 특성(길이, 개수, 접두사 공유 정도)과 실제 필요한 질의 종류(정확 일치 vs 접두사 vs 범위)를 먼저 따져야 합니다.

좋은 답변 vs 약한 답변

강한 답변 (senior level)
  • 정확히 일치 검색과 접두사 검색을 구분하고, 해시 테이블이 접두사 검색에 근본적으로 부적합한 이유(해시의 scrambling 특성)를 설명
  • 트라이의 메모리 비용을 알파벳 크기, 접두사 공유 정도와 연결해 구체적으로 설명
  • 공유되는 접두사가 적으면 트라이가 오히려 손해일 수 있다는 것을 인식
  • radix tree/Patricia trie, DAWG 같은 실무 압축 기법과 각각의 트레이드오프를 제시
  • 동시성, 유니코드/언어별 특성 같은 실무 확장 질문에도 구조적으로 답변
약한 답변 (주니어 패턴)
  • "트라이는 문자열 저장에 쓰는 트리입니다"라고만 답하고 왜 그런지 설명하지 못함
  • 트라이가 항상 해시 테이블보다 메모리를 아낀다고 오해
  • 알파벳 크기에 따라 자식 표현 방식이 달라져야 한다는 것을 모름
  • 접두사 검색이 필요 없는 상황에서도 무조건 트라이를 먼저 제안
  • 압축 트라이나 DAWG 같은 실무 변형을 전혀 모름

약한 답변이 약한 이유: "트라이 = 문자열 저장"이라는 이름과 용도만 외우고, 그 대가로 어떤 메모리 비용을 치르는지, 그리고 그 비용이 데이터 특성(접두사 공유 정도, 알파벳 크기)에 따라 어떻게 달라지는지를 설명하지 못합니다. 시니어 답변은 "언제 트라이가 이기고 언제 지는지"를 데이터 특성으로 설명하고, 실무에서 그 비용을 줄이는 구체적인 기법(압축, 캐싱)까지 제시합니다.

흔한 오해와 바로잡기

오해 1: "트라이는 항상 해시 테이블보다 메모리를 아낀다"
사실: 트라이가 메모리를 아끼는 것은 저장된 문자열들이 접두사를 많이 공유할 때뿐입니다. 서로 접두사를 거의 공유하지 않는 문자열들을 저장하면, 노드마다 붙는 포인터/플래그 오버헤드 때문에 원본 문자열을 그냥 배열에 저장하는 것보다 메모리를 더 쓸 수 있습니다.
오해 2: "트라이 노드는 항상 알파벳 크기만큼의 배열로 자식을 표현한다"
사실: 그것은 알파벳이 작을 때(예: 영어 소문자 26개)의 구현 선택일 뿐입니다. 유니코드처럼 가능한 다음 문자의 종류가 매우 많으면 배열 방식은 메모리를 감당할 수 없어, 해시맵이나 정렬된 리스트로 자식을 표현하는 것이 일반적입니다. 이 경우 자식 접근 비용이 O(1)이 아니라 해시 계산이나 이진 탐색 비용이 붙습니다.
오해 3: "문자열을 다루는 문제면 무조건 트라이가 정답이다"
사실: 정확히 일치하는 검색만 필요하면 해시 테이블이 구현도 단순하고 상수 인자도 작아 더 나은 선택인 경우가 많습니다. 트라이는 접두사 검색, 사전순 순회, longest prefix match처럼 "문자열의 구조(순서, 접두사 관계)"를 활용해야 하는 질의에서만 확실한 이점이 있습니다.
오해 4: "압축 트라이(radix tree)는 구현이 복잡하니 실무에서는 잘 안 쓴다"
사실: 오히려 반대입니다. 순수 트라이는 교과서적 설명에는 적합하지만, 노드 오버헤드가 커서 대규모 실무 시스템(Linux 커널 라우팅 테이블, 많은 고성능 웹 프레임워크의 URL 라우터, 일부 데이터베이스의 인덱스)은 대부분 radix tree/Patricia trie 계열을 사용합니다. 구현 복잡도가 있지만, 메모리와 조회 성능의 실질적 이득이 그 비용을 상회하기 때문에 프로덕션 표준으로 자리잡았습니다.

자기 점검 Rubric

아래 항목에 스스로 답할 수 있으면 Day 7을 완료한 것입니다.

  • 트라이가 문자열을 문자 단위 경로로 표현해 공통 접두사를 공유한다는 것을 설명할 수 있다.
  • 해시 테이블이 접두사 검색을 근본적으로 지원하기 어려운 이유(해시의 scrambling 특성)를 설명할 수 있다.
  • 트라이의 메모리 비용이 알파벳 크기와 접두사 공유 정도에 따라 달라진다는 것을 구체적으로 설명할 수 있다.
  • 공유되는 접두사가 적을 때 트라이가 오히려 손해일 수 있다는 것을 설명할 수 있다.
  • radix tree(Patricia trie)가 체인을 압축해 메모리를 절약하는 방식과, DAWG가 접미사까지 공유하는 방식의 차이를 설명할 수 있다.
  • 자동완성, IP 라우팅(longest prefix match), URL 라우팅 같은 실무 사례에서 트라이 계열 구조가 선택되는 이유를 설명할 수 있다.
  • 약한 답변과 강한 답변의 차이를 본인 말로 설명할 수 있다.
