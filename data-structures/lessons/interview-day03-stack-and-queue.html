전제 개념 — 여기서부터 읽으면 됩니다

스택(Stack)과 큐(Queue)의 기본 동작

스택은 마지막에 넣은 것이 가장 먼저 나오는 LIFO(Last-In-First-Out) 구조입니다. push로 넣고 pop으로 꺼내며, 항상 한쪽 끝(top)에서만 삽입/삭제가 일어납니다.

큐는 먼저 넣은 것이 먼저 나오는 FIFO(First-In-First-Out) 구조입니다. enqueue로 뒤(rear)에 넣고 dequeue로 앞(front)에서 꺼냅니다.

여기까지는 CS 개론 수준입니다. 시니어 면접이 실제로 보는 것은 "이 단순한 규칙을 실제 시스템의 메모리 위에 어떻게 구현하고, 용량과 동시성 문제를 어떻게 다루는가"입니다.

호출 스택(call stack)과 재귀의 관계

함수를 호출하면 런타임은 그 함수의 지역 변수, 매개변수, "돌아갈 주소(return address)"를 스택 프레임이라는 단위로 call stack에 push합니다. 함수가 끝나면 그 프레임을 pop하고 돌아갑니다. 함수 A가 B를 호출하고 B가 C를 호출하면, C가 끝나야 B로, B가 끝나야 A로 돌아갈 수 있는 LIFO 순서와 정확히 일치하기 때문에 스택으로 구현됩니다.

재귀 함수는 자기 자신을 호출할 때마다 새 스택 프레임을 계속 쌓습니다. 재귀 깊이가 너무 깊어지면(예: 정렬 안 된 100만 건 리스트를 재귀로 순회) 스택 공간이 바닥나 StackOverflowError / RecursionError가 발생합니다. call stack 크기는 보통 스레드당 수백 KB~수 MB로 제한되어 있어서, 힙 메모리는 넉넉해도 재귀는 이 제한에 먼저 걸립니다.

배열 기반 큐를 만들 때 생기는 함정 — 그리고 원형 큐(circular buffer)

배열로 큐를 순진하게 구현하면, dequeue할 때마다 앞의 원소를 지우고 나머지를 한 칸씩 당기거나(O(n) 이동), front 인덱스만 증가시키면서 원소를 그대로 두는 방식을 씁니다. 후자는 이동 비용은 없지만 계속 enqueue/dequeue를 반복하면 배열 앞쪽에 "죽은 공간"이 쌓여 결국 배열 끝에 도달해 더 넣을 자리가 없어 보이는 문제가 생깁니다.

이를 해결하는 방법이 원형 큐(circular buffer/ring buffer)입니다. front와 rear 인덱스를 배열 크기로 나눈 나머지(modulo)로 관리해서, 배열 끝에 도달하면 다시 처음(index 0)으로 돌아가게 만듭니다. 이렇게 하면 고정 크기 배열 하나로 앞쪽 빈 공간을 재사용하면서 enqueue/dequeue 모두 진짜 O(1)을 유지합니다.

원형 큐에서 실무적으로 까다로운 부분은 "큐가 꽉 찬 상태"와 "큐가 빈 상태"를 어떻게 구분하는가입니다. front == rear라는 조건만으로는 두 상태를 구분할 수 없어서, 별도의 size 카운터를 두거나, 한 칸을 항상 비워두는(capacity - 1개까지만 채우는) 방식을 씁니다. 이 경계 조건을 놓치면 큐가 가득 찼는데 비어 있다고 오판하거나 그 반대의 버그가 생깁니다.

bounded queue와 backpressure

bounded queue는 최대 용량이 정해진 큐이고, unbounded queue는 이론상 메모리가 허용하는 한 계속 커지는 큐입니다.

backpressure(배압)란 "소비자(consumer)가 처리하는 속도보다 생산자(producer)가 넣는 속도가 빠를 때, 시스템이 생산자 쪽 속도를 억제하거나 신호를 보내는 메커니즘"을 말합니다. bounded queue가 가득 차면 추가 enqueue를 막거나(블로킹), 생산자에게 실패를 알리거나(에러 반환), 가장 오래된/새 항목을 버리는(drop) 정책 중 하나를 선택해야 합니다. 이 선택 자체가 시스템 설계 결정입니다.

unbounded queue를 쓰면 이 문제를 "일단 피할" 수 있지만, 사라지는 게 아니라 메모리 사용량이 무한정 증가하는 문제로 형태만 바뀝니다. 소비자가 느려지거나 멈추면 큐에 쌓인 항목이 계속 늘어나 결국 OutOfMemoryError나 프로세스 강제 종료(OOM killer)로 이어집니다. 즉 "큐가 가득 차서 거부당하는 에러"를 "서비스 전체가 죽는 장애"로 미룬 것뿐입니다.

왜 면접에서 이 질문을 묻는가

면접관이 보는 신호

"스택과 큐가 뭔가요?"는 신입도 답할 수 있습니다. 10년차 이상에게 이 주제를 묻는 진짜 목적은 다음을 확인하는 것입니다.

  • 경계 조건을 스스로 설계할 수 있는가: 큐가 가득 찼을 때/비었을 때 무엇을 해야 하는지 코드를 짜기 전에 정책으로 먼저 결정하는가.
  • 용량 문제를 메모리·장애 관점으로 연결하는가: "unbounded면 안전하지 않나요?"라는 함정에서, 무한정 큐가 실제로는 지연된 폭탄(OOM)이라는 것을 아는가.
  • 동시성 환경에서의 안전성을 아는가: 여러 스레드가 동시에 enqueue/dequeue하면 무슨 문제가 생기고, 이를 막는 실무 도구(락, lock-free queue, 언어 표준 라이브러리의 blocking queue)를 아는가.
  • 재귀와 스택 메모리의 관계를 실제 장애 경험으로 설명할 수 있는가: 이론이 아니라 "왜 재귀 깊이 제한에 걸렸고 어떻게 고쳤는지"를 말할 수 있는가.

주니어는 "스택은 LIFO, 큐는 FIFO"라고 정의를 말합니다. 시니어는 "그 자료구조를 실제 서비스에 넣었을 때 용량이 넘치면 무슨 일이 생기고, 그걸 어떻게 막을지"를 말합니다.

실무 연결

시나리오 1 — 메시지 브로커 앞단의 요청 큐

API 서버가 급증한 트래픽을 처리하지 못해 요청을 내부 큐에 쌓아 뒤에서 순차 처리하려 한다.

선택: bounded queue + 명시적 거부 정책(load shedding) — 큐 크기를 무한으로 두면 처리 지연이 계속 누적되어(latency가 계속 증가) 결국 모든 요청이 타임아웃되는 "queueing이 오히려 전체 장애를 만드는" 상황이 됩니다. 큐 크기를 제한하고, 초과분은 빠르게 429/503으로 거부하는 편이 일부 요청이라도 정상 지연 시간 안에 처리되게 만듭니다.

시나리오 2 — 웹 브라우저의 undo/redo 기능

사용자의 편집 동작을 순서대로 되돌리거나 다시 실행해야 한다.

선택: 스택(두 개) — undo 스택에서 pop한 동작을 redo 스택에 push하는 방식은 "가장 최근 동작을 먼저 되돌린다"는 LIFO 요구사항과 정확히 맞습니다. 큐로 구현하면 가장 오래된 동작부터 되돌리게 되어 사용자가 기대하는 동작과 어긋납니다.

시나리오 3 — 백그라운드 작업 큐(이메일 발송, 이미지 리사이즈)

다수의 워커(consumer)가 큐에서 작업을 꺼내 처리하고, 작업 생성 속도가 순간적으로 몰릴 수 있다.

선택: 영속(persistent) 메시지 큐 + bounded 처리 + 재시도/DLQ — 인메모리 큐만 쓰면 프로세스가 죽을 때 큐에 남은 작업이 유실됩니다. Kafka/RabbitMQ/SQS 같은 영속 큐를 쓰고, 워커 수와 prefetch 개수로 사실상 bounded consumption을 만들어 워커가 감당 못 할 양을 한 번에 끌어오지 않게 제한합니다.

시나리오 4 — 그래프 탐색 API (최단 경로, 추천 확산)

깊이 우선 탐색(DFS)과 너비 우선 탐색(BFS)을 사용자 요청 처리 중 호출한다.

선택: DFS는 명시적 스택, BFS는 큐 — DFS를 재귀로 구현하면 그래프가 깊거나 사이클이 있을 때 재귀 깊이 제한에 걸릴 위험이 있습니다. 대규모 그래프를 다루는 실무 코드는 재귀 대신 명시적 스택(배열/deque)으로 DFS를 구현해 재귀 한계를 우회합니다. BFS는 애초에 큐 없이는 구현할 수 없는 구조입니다.

답변 사고 순서

  1. 순서 요구사항 확인 — LIFO가 맞는가(가장 최근 것부터), FIFO가 맞는가(먼저 온 것부터), 아니면 우선순위 기반인가?
  2. 용량 정책 확인 — 무한정 쌓여도 되는가, 아니면 최대 크기를 반드시 정해야 하는가? (거의 항상 후자가 정답에 가깝습니다.)
  3. 초과 시 동작 정의 — 가득 찼을 때 블로킹할 것인가, 거부할 것인가, 오래된 것을 버릴 것인가? 이것이 backpressure 정책입니다.
  4. 동시성 요구 확인 — 단일 스레드에서만 쓰는가, 여러 producer/consumer가 동시에 접근하는가? 후자면 thread-safe 구현이나 언어 표준 blocking queue를 검토한다.
  5. 영속성 요구 확인 — 프로세스가 죽어도 큐 내용이 남아야 하는가? 그렇다면 인메모리 구조가 아니라 메시지 브로커가 필요하다.
  6. 최종 선택과 실패 시나리오 명시 — 선택한 구조가 어떤 상황에서 문제를 일으킬 수 있는지까지 포함해서 답한다.

핵심 답변 예시

질문: "큐를 직접 구현할 때 어떤 경계 조건과 운영 이슈를 고려합니까?"

짧은 버전 (30초)

"먼저 큐가 가득 찼을 때와 비었을 때를 구분하는 경계 조건을 명확히 해야 합니다. 배열 기반이면 원형 큐로 구현해서 인덱스 재사용과 O(1) enqueue/dequeue를 보장하고요. 그다음은 용량을 무한으로 둘지 제한할지인데, 저는 기본적으로 bounded queue로 설계하고 가득 찼을 때 어떻게 할지(블로킹, 거부, drop)를 명시적으로 정합니다. 마지막으로 여러 스레드가 동시에 접근한다면 락이나 thread-safe 큐 구현을 써야 race condition을 막을 수 있습니다."

깊이 있는 버전 (2분)

"큐를 배열로 직접 구현할 때 가장 먼저 부딪히는 문제는 dequeue 후 앞쪽에 생기는 빈 공간을 어떻게 재사용하느냐입니다. 이걸 원형 큐(circular buffer)로 풀면, front/rear 인덱스를 modulo 연산으로 순환시켜서 고정 크기 배열 하나로 O(1) enqueue/dequeue를 유지할 수 있습니다. 다만 이때 front == rear라는 조건만으로는 '꽉 참'과 '빔'을 구분할 수 없다는 함정이 있어서, 별도 size 카운터를 두거나 한 칸을 비워두는 방식으로 경계 조건을 명시적으로 처리해야 합니다.

그다음 더 중요한 결정은 용량 정책입니다. unbounded 큐는 구현이 간단해 보이지만, 소비자가 생산자보다 느려지는 순간 큐 크기가 무한히 늘어나고 결국 메모리 부족으로 프로세스가 죽습니다. 이건 '에러가 안 난다'가 아니라 '에러가 훨씬 늦게, 훨씬 크게 난다'는 뜻입니다. 그래서 저는 bounded queue를 기본으로 하고, 가득 찼을 때의 정책 — 생산자를 블로킹할지, 즉시 실패를 반환할지(backpressure), 오래된 항목을 버릴지 —을 요구사항에 맞춰 명시적으로 결정합니다. 예를 들어 실시간 시세 데이터라면 오래된 값은 의미가 없으니 최신 값으로 덮어쓰는 정책이 맞고, 결제 요청이라면 하나도 유실되면 안 되니 블로킹이나 영속 큐가 맞습니다.

마지막으로 동시성입니다. 여러 스레드가 동시에 enqueue/dequeue하면 인덱스 갱신 자체가 race condition에 노출됩니다. 직접 락을 걸어 구현할 수도 있지만, 실무에서는 언어가 제공하는 검증된 구현(Java BlockingQueue, Python queue.Queue, Go 채널)을 우선 사용하고, 정말 초고성능이 필요할 때만 lock-free 큐 구현을 검토합니다."

스택 vs 큐 vs 데크(Deque) 상세 비교

기준 스택 큐 (원형 큐) 데크(Deque)
삽입/삭제 순서 LIFO (한쪽 끝만) FIFO (양 끝 각각 1방향) 양쪽 끝 모두 삽입/삭제 가능
주요 연산 복잡도 push/pop O(1) enqueue/dequeue O(1) (원형 큐 기준) 양끝 push/pop O(1)
대표 구현 동적 배열, 연결 리스트 원형 버퍼(배열), 연결 리스트 이중 연결 리스트, 원형 버퍼
전형적 실무 사례 call stack, undo/redo, 괄호 검사, DFS(명시적 스택) 작업 큐, BFS, 메시지 브로커, 프린터 스풀 슬라이딩 윈도우, 브라우저 히스토리, monotonic queue
경계 조건 위험 재귀/push 과다로 인한 stack overflow 가득 참/빔 구분, backpressure 정책 부재 양방향 인덱스 관리 복잡도 증가
동시성 고려 단일 스레드 사용이 흔함(call stack은 스레드별) producer-consumer에서 필수 (blocking/lock-free) 양쪽에서 동시 접근 시 락 범위 설계가 더 까다로움
메모리 성장 패턴 깊이에 비례, 재귀는 스레드 스택 한도에 걸림 unbounded 시 무한 증가 위험 (backpressure 필요) unbounded 시 큐와 동일한 위험

Senior-level Follow-up 질문

Q1. "재귀로 짠 DFS가 프로덕션에서 StackOverflowError를 냈습니다. 어떻게 고치겠습니까?"
먼저 실제 입력 규모에서 재귀 깊이가 얼마나 커질 수 있는지 확인합니다. 트리/그래프가 예상보다 깊거나 사이클로 인해 방문 처리가 잘못돼 무한 재귀에 가깝게 깊어졌을 가능성도 점검합니다. 근본적인 해결은 재귀를 명시적 스택(배열이나 Deque)을 사용하는 반복문 형태로 바꾸는 것입니다. 이러면 스레드 콜 스택 한도가 아니라 힙 메모리 한도까지 사용할 수 있어 훨씬 깊은 구조도 처리할 수 있습니다. 언어에 따라 스레드 스택 크기를 늘리는 임시방편도 있지만, 이는 근본 해결이 아니라 한계를 조금 뒤로 미루는 것뿐입니다.
Q2. "생산자가 소비자보다 훨씬 빠른 상황에서 bounded queue가 가득 찼습니다. 어떤 정책을 선택하겠습니까?"
데이터의 성격에 따라 다릅니다. 결제·주문처럼 유실이 치명적인 데이터는 생산자를 블로킹하거나(backpressure로 생산 속도를 낮춤) 영속 큐로 넘겨 재시도 가능하게 만듭니다. 실시간 센서 값이나 시세처럼 최신 값만 의미 있는 데이터는 오래된 항목을 버리고 최신 값으로 덮어쓰는 정책(drop-oldest)이 낫습니다. 중요한 것은 이 선택을 코드 구현 세부사항이 아니라 비즈니스 요구사항에서 먼저 결정해야 한다는 점입니다.
Q3. "lock 기반 큐와 lock-free 큐의 차이와 각각을 언제 선택합니까?"
lock 기반 큐는 구현이 단순하고 검증하기 쉽지만, 경쟁이 심한 고동시성 환경에서는 락 대기가 병목이 됩니다. lock-free 큐는 CAS(compare-and-swap) 같은 원자적 연산으로 락 없이 동시 접근을 처리해 처리량이 높지만, 구현이 훨씬 복잡하고 ABA 문제 같은 미묘한 버그에 취약합니다. 저는 기본적으로 언어 표준 라이브러리의 검증된 blocking queue를 쓰고, 프로파일링으로 락 경합이 실제 병목으로 확인된 경우에만 lock-free 구현이나 검증된 라이브러리(Disruptor 등) 도입을 검토합니다.
Q4. "모놀리식 서비스 내부의 인메모리 큐와 Kafka 같은 외부 메시지 큐를 선택하는 기준은 무엇입니까?"
인메모리 큐는 지연이 매우 낮고 구현이 단순하지만, 프로세스가 죽으면 큐 내용이 사라지고 단일 프로세스 용량을 넘어설 수 없습니다. 외부 메시지 큐는 영속성, 여러 프로세스 간 분산 처리, 재시도/DLQ, 순서 보장 같은 기능을 제공하지만 네트워크 왕복 지연과 운영 복잡도가 추가됩니다. 같은 프로세스 안에서 짧게 끝나는 작업이면 인메모리로 충분하고, 실패해도 재시도가 필요하거나 여러 워커가 나눠 처리해야 하거나 유실이 치명적이면 외부 큐를 선택합니다.

좋은 답변 vs 약한 답변

강한 답변 (senior level)
  • 원형 큐의 "꽉 참 vs 빔" 경계 조건을 구체적으로 설명
  • bounded/unbounded를 명시적으로 비교하고 backpressure 정책까지 제시
  • 재귀-스택 관계를 실제 장애 시나리오(StackOverflowError)로 설명
  • 동시성 환경에서 thread-safe 구현의 필요성과 대안(표준 라이브러리 우선)을 언급
  • 데이터 성격(유실 허용 여부)에 따라 정책이 달라진다는 것을 명시
약한 답변 (주니어 패턴)
  • "스택은 LIFO, 큐는 FIFO"라는 정의만 말하고 끝냄
  • 배열로 큐를 구현할 때 생기는 빈 공간 문제를 인지하지 못함
  • unbounded 큐가 "안전한 선택"이라고 잘못 판단
  • 동시성 문제를 전혀 고려하지 않고 단일 스레드 가정만 함
  • 모든 상황에 동일한 큐 구현을 적용하려 함(데이터 성격 구분 없음)

약한 답변이 약한 이유: 정의를 아는 것과 그 정의를 실제 메모리 위에 안전하게 구현하는 것은 다른 레벨의 지식입니다. 시니어 답변은 "가득 찼을 때 무엇을 할 것인가"라는 결정을 회피하지 않고, 그 결정이 시스템의 실패 모드를 어떻게 바꾸는지까지 설명합니다.

흔한 오해와 바로잡기

오해 1: "큐 용량을 무한으로 두면 요청을 놓치지 않으니 더 안전하다"
사실: unbounded 큐는 유실 대신 지연 누적과 메모리 무한 증가로 문제 형태가 바뀔 뿐입니다. 소비자가 못 따라가면 큐가 계속 커지다 결국 메모리 부족으로 프로세스 전체가 죽고, 그 순간 큐에 쌓여 있던 모든 항목이 한꺼번에 유실됩니다. bounded queue + 명시적 거부/drop 정책이 오히려 일부 요청이라도 안전하게 처리하고 장애 범위를 좁히는 설계입니다.
오해 2: "재귀는 항상 반복문보다 느리고 위험하니 쓰면 안 된다"
사실: 재귀 자체가 문제가 아니라 재귀 깊이가 입력 규모에 비례해서 예측 불가능하게 커질 수 있는 상황이 문제입니다. 트리 높이가 log n으로 제한되는 균형 트리 순회 같은 경우는 재귀가 안전하고 가독성도 좋습니다. 반대로 정렬 안 된 연결 리스트나 편향된 트리, 큰 그래프의 DFS처럼 깊이가 n에 비례할 수 있는 경우에만 명시적 스택으로 바꾸는 것이 맞는 판단입니다.
오해 3: "원형 큐는 front == rear면 항상 빈 상태다"
사실: front == rear는 "빈 상태"와 "가득 찬 상태" 모두에서 나타날 수 있는 조건입니다(포인터 값만으로는 구분 불가). 그래서 실무 구현은 별도의 size/count 변수를 유지하거나, 배열 한 칸을 항상 비워두어 "가득 참"과 "빔"이 서로 다른 인덱스 조합으로 나타나게 설계합니다. 이 경계 조건을 놓치면 큐가 실제로는 가득 찼는데 빈 것으로 착각해 데이터를 덮어쓰는 조용한 버그가 생깁니다.
오해 4: "큐는 항상 FIFO이므로 우선순위가 필요하면 못 쓴다"
사실: 일반 큐는 순서만 보장하지만, 우선순위가 필요한 경우 힙 기반의 우선순위 큐(priority queue)를 쓰면 됩니다. "큐"라는 이름과 FIFO 큐의 내부 구현(배열/연결 리스트)을 동일시하지 않는 것이 중요합니다. 우선순위 큐는 인터페이스만 큐이고 내부 구현은 힙이라는 점을 구분해서 설명할 수 있어야 합니다. (힙과 우선순위 큐는 Day 6에서 자세히 다룹니다.)

자기 점검 Rubric

아래 항목에 스스로 답할 수 있으면 Day 3를 완료한 것입니다.

  • 원형 큐가 왜 필요한지, 그리고 "가득 참"과 "빔"을 어떻게 구분하는지 설명할 수 있다.
  • call stack과 재귀 깊이의 관계, 그리고 StackOverflowError가 발생하는 조건을 설명할 수 있다.
  • bounded queue와 unbounded queue의 차이를 backpressure 개념으로 설명할 수 있다.
  • unbounded 큐가 "안전한 선택"이 아니라 문제를 지연·확대시킨다는 것을 근거와 함께 말할 수 있다.
  • 큐가 가득 찼을 때 취할 수 있는 정책(블로킹, 거부, drop)을 데이터 성격에 맞게 선택할 수 있다.
  • 동시성 환경에서 큐를 안전하게 쓰는 방법(표준 라이브러리 blocking queue, lock-free 큐)을 설명할 수 있다.
  • 약한 답변과 강한 답변의 차이를 본인 말로 설명할 수 있다.
