# 자료구조 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 개발자 기술 면접에서 자료구조 지식을 **문제 해결, 시스템 설계, 성능 최적화, 메모리 사용량, 운영 trade-off**와 연결해 설명할 수 있도록 훈련한다.

경력자 자료구조 면접은 구현 암기보다 "왜 이 자료구조를 선택했는가", "입력 규모와 접근 패턴이 바뀌면 선택이 어떻게 달라지는가", "실무 시스템에서 어떤 위험과 비용이 생기는가"를 본다.

## 평가 대상 역량

| 역량 | 면접에서 확인하려는 것 |
|------|------------------------|
| 선택 기준 | 배열, 리스트, 해시, 트리, 힙, 그래프를 요구사항에 맞게 고르는가 |
| 복잡도 분석 | 평균/최악 시간복잡도와 공간복잡도를 상황에 맞게 설명하는가 |
| 메모리 관점 | locality, pointer overhead, allocation, GC 부담을 고려하는가 |
| 동적 데이터 | 삽입/삭제/검색/정렬/range query 패턴 변화에 대응하는가 |
| 실무 연결 | 캐시, ranking, scheduler, routing, autocomplete, index 설계와 연결하는가 |
| 경계 조건 | 충돌, 편향 트리, resize, recursion depth, concurrency 문제를 예상하는가 |
| 커뮤니케이션 | 약한 답변과 강한 답변의 차이를 예시로 설명하는가 |

## Interview 레슨 작성 규칙

각 Day 레슨은 반드시 **면접 질문 중심**으로 작성한다.

1. **면접 질문**
   - 자료구조 선택 이유와 trade-off를 요구하는 질문으로 시작한다.
   - 예: "해시 테이블이 평균 O(1)인데도 정렬된 배열이나 트리를 선택해야 하는 경우는 언제입니까?"
2. **전제 개념**
   - 어려운 개념은 먼저 쉬운 한국어로 설명한다.
   - Big-O, amortized cost, locality, balance, collision 같은 개념을 필요한 배경부터 풀어쓴다.
3. **왜 면접에서 묻는가**
   - 문제 풀이 암기가 아니라 요구사항을 자료구조 선택으로 번역하는 능력을 보는 이유를 적는다.
4. **실무 연결**
   - 캐시, 큐, 우선순위 작업 처리, 검색 자동완성, DB 인덱스, 그래프 탐색, API 응답 최적화와 연결한다.
5. **답변 사고 순서**
   - 연산 패턴 확인 -> 입력 규모 확인 -> 시간/공간 복잡도 비교 -> 구현/운영 비용 비교 -> 최종 선택 순서로 답변하도록 안내한다.
6. **핵심 답변 예시**
   - 면접에서 바로 말할 수 있는 답변을 짧은 버전과 깊이 있는 버전으로 제공한다.
7. **Trade-off**
   - 조회, 삽입, 삭제, 정렬 유지, 범위 검색, 메모리, 구현 복잡도, 동시성 안전성을 비교한다.
8. **흔한 오해**
   - "해시는 항상 O(1)", "Big-O만 같으면 실무 성능도 같다", "트리는 균형이라고 가정하면 된다" 같은 오해를 바로잡는다.
9. **Follow-up 질문**
   - 면접관이 더 깊게 물을 질문과 답변 방향을 포함한다.
10. **10년차 이상 답변 기준**
   - 알고리즘 풀이가 아니라 입력 특성, 운영 조건, 유지보수 비용까지 고려하는지 평가한다.

## 완료 기준

- 자료구조를 이름이 아니라 연산 패턴과 제약 조건으로 선택할 수 있다.
- Big-O, 메모리, locality, 구현 복잡도를 함께 설명할 수 있다.
- 코딩 테스트 답변을 실무 설계 답변으로 확장할 수 있다.
- 약한 답변과 강한 답변의 차이를 스스로 설명할 수 있다.
