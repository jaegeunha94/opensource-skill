# Observability / 장애 대응 — 10년차 이상 개발자 인터뷰 대비 학습 미션

## 학습 목표

10년차 이상 백엔드/플랫폼 개발자로서 Observability와 장애 대응에 관한 시니어 수준 인터뷰를 통과한다.

단순히 "Prometheus는 이런 도구입니다"라고 도구를 나열하는 수준이 아니라, 다음을 설명할 수 있는 수준을 목표로 한다:

- **왜** 특정 관측 전략을 선택했는지 — 설계 판단과 trade-off
- **무엇을** 보고 장애를 탐지하고 분류했는지 — 신호 해석 능력
- **어떻게** 원인을 추적하고 격리했는지 — 디버깅 사고 흐름
- **누구에게** 무엇을 언제 전달했는지 — 장애 중 커뮤니케이션
- **왜 다시 안 일어나도록** 무엇을 바꿨는지 — 재발 방지 설계

## 전제 지식

이 학습 과정은 다음을 이미 알고 있다고 가정한다:

- REST API 개발 경험 (백엔드 서비스 운영 경험)
- 기본적인 Linux 명령어 (로그 확인, 프로세스 조회)
- 데이터베이스, 캐시 등 백엔드 컴포넌트에 대한 기본 이해
- HTTP 요청 흐름 이해

기본 Python 문법이나 주니어 수준 CS 지식은 다루지 않는다.

## 커리큘럼 범위

| 영역 | 핵심 주제 |
|------|-----------|
| 관측성 기초 | Observability vs Monitoring, Three Pillars (Logs/Metrics/Traces) |
| 로그 설계 | 구조화 로깅, 로그 레벨 전략, 컨텍스트 식별자 |
| 메트릭과 알림 | RED/USE 방법론, 알림 설계, false positive 방지 |
| 분산 트레이싱 | Trace ID 전파, span 설계, 병목 분석 |
| 장애 원인 분석 | RCA 프레임워크, 5 Whys, 데이터 기반 추론 |
| 대시보드 설계 | 역할별 대시보드, 지표 선택 원칙 |
| SLO / SLI / SLA | Error Budget, 소진율 알림 |
| Postmortem | 장애 보고서 작성, 재발 방지 액션 아이템 |
| 장애 커뮤니케이션 | 이해관계자 보고, 장애 브리핑 |
| 대규모 아키텍처 | 관측성 파이프라인, 비용과 신뢰성 trade-off |

## 인터뷰에서 평가하는 것

시니어 인터뷰어는 다음을 평가한다:

1. **경험 기반 답변** — "실제로 이런 장애가 있었는데, 이렇게 탐지/분석/해결했습니다"
2. **설계 판단력** — "이 방식보다 저 방식이 더 나은 이유는 이 trade-off 때문입니다"
3. **커뮤니케이션 능력** — "장애 중에 어떤 채널로 누구에게 어떤 정보를 전달했는지"
4. **시스템 사고** — "한 컴포넌트 문제가 어떻게 전파되는지, 왜 알람이 안 울렸는지"
5. **재발 방지 설계** — "임시 fix와 근본 fix를 구분하고 후자를 선택한 이유"
