왜 인터뷰에서 묻는가

메트릭과 알림은 "대시보드에 그래프 몇 개 띄워놓는 것"처럼 보이지만, 시니어 인터뷰에서는 이 주제로 지원자의 운영 판단력을 검증한다. 면접관이 실제로 보는 것은 다음 세 가지다.

1. 무엇을 측정해야 하는지 스스로 결정할 수 있는가

주니어는 "CPU, 메모리 사용률을 본다"는 식으로 눈에 보이는 리소스 지표를 나열한다. 시니어는 이 서비스가 실패하는 방식이 무엇인지 먼저 정의하고, 그것을 드러내는 지표를 역으로 설계한다. RED/USE 같은 방법론을 아는 것 자체보다, "왜 이 서비스에는 이 지표가 맞는가"를 설명할 수 있는지가 핵심이다.

2. 통계를 다루는 방식 — 평균의 함정을 아는가

"평균 응답시간이 200ms라서 문제 없다"는 흔한 착각이다. 실제 사용자 100명 중 5명이 5초씩 기다리고 있어도 평균은 낮게 나올 수 있다. 시니어는 percentile(p50/p95/p99) 개념으로 분포를 이해하고, "평균이 정상이어도 꼬리(tail)가 나쁠 수 있다"는 것을 안다.

3. 알림 설계가 조직의 신뢰와 직결된다는 것을 이해하는가

알림을 너무 민감하게 만들면(threshold를 낮게 잡으면) 사소한 변동에도 알림이 울려서 온콜 담당자가 무시하게 되고(Alert Fatigue, Day 2에서 다룬 개념과 직결), 반대로 너무 둔감하게 만들면(threshold를 높게 잡으면) 진짜 장애를 놓친다. 이 둘 사이의 균형을 의도적으로 설계했는지가 실무 경험의 증거다.

인터뷰어가 듣고 싶어하는 신호
"저는 메트릭을 고를 때 '이 지표가 이상해지면 실제로 사용자가 영향을 받는가'를 먼저 묻습니다. 리소스 사용률 자체보다 사용자가 체감하는 증상(요청 실패율, 응답 지연)을 우선 알림으로 걸고, 리소스 지표는 원인 분석용 대시보드로 분리해서 봅니다."

전제 개념 — 쉬운 언어로 먼저 이해하기

메트릭(Metric)이란?

특정 시점에 측정한 숫자를 시간 순서대로 모아놓은 데이터다. 로그는 "무슨 일이 있었는지" 이야기를 담고, 메트릭은 "지금 수치가 얼마인지"를 담는다. 로그 한 줄 한 줄을 읽는 대신, 메트릭은 "지난 5분간 에러율이 몇 %였는가"처럼 집계된 숫자로 상태를 빠르게 파악하게 해준다.

메트릭에는 크게 세 가지 형태가 있다.

종류의미예시
Counter 계속 증가만 하는 누적 값 (재시작 시 리셋) 총 요청 수, 총 에러 수
Gauge 오르내리는 현재 값 현재 메모리 사용량, 현재 대기 중인 큐 길이
Histogram 값의 분포를 구간(bucket)별로 기록 응답 시간 분포 (0-100ms, 100-500ms, 500ms+ 구간별 개수)
흔한 실수
응답 시간을 Gauge(마지막 값만)로 기록하면 순간의 값만 남고 분포를 알 수 없다. 응답 시간처럼 "분포가 중요한 값"은 반드시 Histogram으로 기록해야 나중에 percentile을 계산할 수 있다.

RED 방법론이란?

요청을 처리하는 서비스(API, 마이크로서비스)를 모니터링할 때 쓰는 3가지 핵심 지표다. Weaveworks의 Tom Wilkie가 정리해서 널리 퍼졌다.

📈
Rate
초당 처리하는 요청 수 (Throughput)
🚨
Errors
실패한 요청의 비율/개수
⏱️
Duration
요청 처리에 걸린 시간 (Latency)

이 세 가지만 있어도 "지금 이 API가 정상인가?"라는 질문에 대부분 답할 수 있다. 요청이 갑자기 줄었다면(Rate 급감) 클라이언트 쪽 문제일 수 있고, 에러가 늘었다면(Errors 급증) 서버 로직이나 의존성 문제이며, 느려졌다면(Duration 증가) 병목이 생긴 것이다.

USE 방법론이란?

CPU, 메모리, 디스크, 네트워크 같은 리소스(Resource)를 모니터링할 때 쓰는 3가지 지표다. Brendan Gregg가 정리했다.

🔧
Utilization
리소스가 바쁜 시간의 비율 (예: CPU 사용률 80%)
📦
Saturation
처리하지 못해 대기 중인 작업의 양 (큐 길이, Run Queue)
❌
Errors
리소스 레벨의 에러 (디스크 I/O 에러 등)
Utilization과 Saturation을 헷갈리지 말 것
Utilization은 "얼마나 바쁜가"이고, Saturation은 "얼마나 밀려있는가"다. CPU 사용률이 70%(Utilization)여도 대기 큐(Saturation)가 계속 쌓이고 있다면 이미 처리 능력의 한계에 다다른 것이다. 톨게이트에 비유하면, Utilization은 "게이트가 몇 % 가동 중인가"이고 Saturation은 "게이트 앞에 차가 얼마나 밀려 있는가"다. 게이트 가동률이 70%여도 밀려있는 차가 계속 늘어나고 있으면 곧 정체가 심해진다는 뜻이다.

RED vs USE, 언제 무엇을 쓰는가

둘은 경쟁 관계가 아니라 측정 대상이 다르다. 서비스(요청을 처리하는 것)는 RED로, 리소스(요청을 처리하기 위해 쓰는 것)는 USE로 본다. 실무에서는 둘을 함께 쓴다 — RED로 "사용자가 겪는 증상"을 보고, USE로 "그 증상의 리소스 레벨 원인"을 찾는다.

Golden Signals란?

Google SRE Book에서 제시한 4가지 핵심 지표로, RED와 USE를 아우르는 상위 개념에 가깝다: Latency(지연), Traffic(트래픽), Errors(에러), Saturation(포화도). RED의 Duration/Rate/Errors에 USE의 Saturation 개념을 더한 것으로 이해하면 된다. 인터뷰에서는 이 세 방법론이 서로 겹치는 개념이라는 것, 그리고 회사마다 부르는 이름이 다를 뿐 핵심 사고는 같다는 것을 설명할 수 있으면 좋은 신호다.

Percentile(p50, p95, p99)이란?

전체 값을 크기 순으로 줄 세웠을 때, 특정 위치에 있는 값을 뜻한다. p95 = 500ms라는 것은 "요청의 95%는 500ms 이내에 끝났고, 나머지 5%는 500ms보다 오래 걸렸다"는 뜻이다.

평균(Average)의 함정
요청 100개 중 95개가 50ms, 5개가 5000ms 걸렸다고 하자. 평균은 (95×50 + 5×5000) / 100 ≈ 297ms로 "그럭저럭 괜찮아 보이는" 숫자가 나온다. 하지만 p95를 보면 5000ms 근처가 나오고, 실제로 5%의 사용자는 5초를 기다린 것이다. 평균은 소수의 극단값을 다수의 정상값이 희석시켜 문제를 숨긴다.

그래서 실무에서는 응답 시간, 지연 시간처럼 "느린 소수가 중요한" 지표는 반드시 percentile(주로 p50, p95, p99)로 본다. p50은 일반적인 사용자 경험, p99는 최악에 가까운 사용자 경험을 보여준다.

알림 임계값(Alert Threshold), False Positive/Negative란?

알림은 "특정 지표가 어떤 기준(threshold)을 넘으면 사람에게 알린다"는 규칙이다. 이 기준을 어디에 두느냐에 따라 두 가지 실패가 생긴다.

구분의미결과
False Positive (오탐) 실제로는 문제가 아닌데 알림이 울림 threshold가 너무 낮음 → 알림 남발 → Alert Fatigue
False Negative (미탐) 실제로 문제인데 알림이 안 울림 threshold가 너무 높음 → 장애를 늦게 알아챔 → MTTD 증가
둘은 반비례 관계다
threshold를 조여서(민감하게) False Negative를 줄이면 False Positive가 늘고, threshold를 풀어서(둔감하게) False Positive를 줄이면 False Negative가 늘어난다. "정답"인 threshold는 없고, 서비스의 중요도와 조직이 감당할 수 있는 알림 피로도 사이에서 의도적으로 선택하는 것이다.

메트릭과 알림 설계 원칙

면접에서 "메트릭/알림을 어떻게 설계하나요?"라는 질문에 이 원칙들로 답하면 구조가 잡힌다.

1. 증상 기반 알림(Symptom-based)을 우선한다

알림은 원칙적으로 사용자가 실제로 영향을 받는 증상에 대해서만 울려야 한다 (예: 에러율 5% 초과, p99 지연 2초 초과). CPU 사용률처럼 "원인이 될 수 있는" 지표는 알림보다는 대시보드로 두고, 증상이 발생했을 때 원인 분석용으로 들여다본다.

2. 원인 기반 알림(Cause-based)은 최소화하고 예방적으로만 쓴다

디스크 사용률 90% 같은 지표는 아직 사용자에게 증상이 나타나지 않았지만 곧 문제가 될 것을 예고한다. 이런 알림은 "지금 당장 깨워야 할 알림"이 아니라 "업무 시간에 확인해야 할 예방적 알림"으로 심각도를 다르게 분리한다.

3. 모든 알림은 actionable해야 한다

알림을 받은 사람이 "그래서 지금 뭘 해야 하지?"라고 되묻게 만들면 나쁜 알림이다. 알림에는 runbook(대응 절차 문서) 링크를 함께 붙이고, "무엇을 확인하고, 무엇을 실행해야 하는지"가 명확해야 한다.

4. 평균이 아니라 percentile로 알림을 건다

지연 시간 알림은 평균이 아니라 p95 또는 p99 기준으로 건다. 평균 기준 알림은 소수의 사용자가 겪는 심각한 지연을 놓치기 때문이다.

5. 라벨(카디널리티)을 신중하게 설계한다

메트릭에 user_id처럼 값의 종류가 무한히 늘어나는 라벨을 붙이면, 시계열 데이터베이스가 각 조합마다 별도의 시계열을 만들어야 해서(카디널리티 폭발) 메트릭 시스템 자체가 느려지거나 다운될 수 있다. 라벨은 service, endpoint, status_code처럼 값의 종류가 유한한것만 쓴다. 사용자 단위 추적은 메트릭이 아니라 로그/트레이스의 역할이다.

6. 단일 window 알림의 한계를 인식한다 (SLO 예고)

"5분간 에러율 5% 초과"처럼 짧은 시간 창(window)만 보면 순간적인 스파이크에도 알림이 울리기 쉽다. 반대로 긴 시간 창만 보면 알림이 늦게 온다. 성숙한 조직은 여러 시간 창을 조합한 Multi-window, Multi-burn-rate 알림을 쓴다 (Day 7 SLO/Error Budget에서 자세히 다룬다).

시니어 답변에서 나와야 하는 내용
"저희는 알림을 심각도(Severity)로 나눠서, 사용자 영향이 확실한 증상 기반 알림만 온콜 페이지로 보내고, 리소스 예방 지표는 업무 시간 슬랙 채널로 분리했습니다. 그리고 모든 페이지 알림에는 runbook 링크를 필수로 붙이도록 템플릿을 강제했습니다. 이후 온콜 담당자의 알림 피로도가 눈에 띄게 줄었습니다."

답변 사고 순서 — 면접장에서 어떻게 풀어나가는가

질문 1: "RED와 USE, 어떤 메트릭을 어떤 기준으로 수집하나요?"

  1. 측정 대상을 먼저 구분 (10초) "저는 먼저 측정 대상이 '요청을 처리하는 서비스'인지 '요청을 처리하기 위한 리소스'인지 구분합니다."
  2. RED 설명 (15초) "서비스는 RED로 봅니다. Rate(초당 요청 수), Errors(에러율), Duration(응답 시간)만 있어도 '이 API가 정상인가'를 대부분 판단할 수 있습니다."
  3. USE 설명 (15초) "리소스는 USE로 봅니다. Utilization(사용률), Saturation(대기량), Errors(리소스 에러)를 보고, 특히 Saturation을 중요하게 봅니다. Utilization은 낮아도 Saturation이 쌓이면 이미 한계입니다."
  4. 연결 관계 설명 (15초) "RED로 사용자가 느끼는 증상을 먼저 보고, 이상이 감지되면 USE로 내려가서 리소스 레벨 원인을 찾는 순서로 씁니다."
  5. 실무 경험 연결 (20초) 본인이 특정 지표(예: DB connection pool saturation)로 장애를 사전에 예측했거나, RED 지표만으로 장애를 빠르게 특정한 경험을 구체적으로 말한다.

질문 2: "알림 임계값을 어떻게 정하고, False Positive/Negative는 어떻게 다루나요?"

  1. 트레이드오프 인정 (10초) "False Positive와 False Negative는 반비례 관계이기 때문에, 완벽한 threshold는 없다고 생각합니다."
  2. 기준 제시 (15초) "그래서 저는 서비스의 중요도에 따라 다르게 접근합니다. 결제처럼 중요한 경로는 민감하게(threshold를 낮게), 덜 중요한 배치 작업은 둔감하게 설정합니다."
  3. 구체적 설계 방법 (20초) "정적 threshold보다는 과거 트래픽 패턴 대비 이상치(anomaly)를 보거나, 여러 시간 창을 조합한 multi-window 알림을 써서 순간 스파이크와 진짜 추세를 구분합니다."
  4. 운영 프로세스로 보완 (15초) "그리고 알림이 울렸는데 조치가 필요 없었던 경우를 주기적으로 리뷰해서 threshold를 계속 튜닝합니다. 알림 설계는 한 번 정하고 끝나는 것이 아니라 지속적으로 조정하는 과정이라고 생각합니다."

핵심 답변 예시

질문 1 모범 답변: "RED와 USE"

강한 답변

"저는 측정 대상을 서비스와 리소스로 나눠서 접근합니다. API 서버 같은 서비스는 RED(Rate, Errors, Duration)로 보고, 이 세 지표만으로 '지금 이 서비스가 정상인가'를 대부분 판단할 수 있습니다. 반면 DB, 큐, 서버 리소스 같은 것은 USE(Utilization, Saturation, Errors)로 봅니다.

실제 경험으로는, 한 번 결제 서비스의 응답 시간이 늘어났을 때 RED의 Duration이 먼저 튀는 걸 알림으로 받았고, 바로 이어서 DB 커넥션 풀의 Saturation(대기 중인 요청 수)이 함께 올라가는 것을 보고 커넥션 풀 고갈이 원인이라는 것을 5분 안에 특정할 수 있었습니다. RED로 증상을 먼저 잡고 USE로 원인을 찾아 내려가는 흐름이 실제로 유효했던 사례입니다."

질문 2 모범 답변: "알림 임계값 설계"

강한 답변

"완벽한 threshold는 없다고 생각합니다. False Positive를 줄이려고 threshold를 높이면 False Negative가 늘어나는 트레이드오프가 항상 있기 때문입니다. 그래서 저는 서비스 중요도에 따라 다르게 설계합니다.

결제, 로그인처럼 핵심 경로는 에러율 1% 초과, p99 지연 2초 초과처럼 민감하게 걸고 온콜 페이지로 바로 연결합니다. 반면 배치 작업이나 내부 관리 도구는 임계값을 넓게 잡고 슬랙 알림 정도로만 둡니다.

또한 5분 단기 window만 보면 순간 스파이크에도 알림이 울리기 때문에, 5분과 1시간 두 window를 함께 만족할 때만 페이지를 보내도록 해서 노이즈를 줄였습니다. 그리고 매주 발생한 알림을 리뷰해서, 조치가 필요 없었던 알림의 비율을 추적하며 threshold를 계속 조정하고 있습니다."

⚖️

좋은 답변 vs 약한 답변의 차이

강한 답변 — 메트릭 선택
"서비스는 RED, 리소스는 USE로 구분해서 보고, RED로 증상을 먼저 감지한 뒤 USE로 내려가 원인을 찾습니다."

측정 대상에 따라 방법론을 구분하고, 둘을 연결하는 사고 흐름이 있다.

약한 답변 — 메트릭 선택
"CPU, 메모리, 디스크 사용률을 그라파나 대시보드에 띄워놓고 봅니다."

왜 그 지표를 보는지, 사용자 영향과 어떻게 연결되는지에 대한 설명이 없다. 도구 나열에 그친다.

강한 답변 — 알림 설계
"핵심 경로는 민감하게, 비핵심 경로는 둔감하게 차등을 두고, multi-window로 순간 스파이크를 걸러내며, 정기적으로 알림을 리뷰해서 threshold를 조정합니다."

트레이드오프 인식, 차등 설계, 지속적 개선 프로세스까지 담겨있다.

약한 답변 — 알림 설계
"에러율이 5%를 넘으면 알림이 오도록 설정했습니다."

왜 5%인지, 서비스별 차등이 있는지, false positive는 어떻게 관리하는지 근거가 없다.

강한 답변 — 평균 vs Percentile
"평균은 극단값을 다수의 정상값이 희석시키기 때문에, 지연 시간은 항상 p95/p99로 봅니다. 평균이 정상이어도 p99가 나쁘면 소수의 사용자가 심각한 지연을 겪고 있다는 뜻입니다."

평균의 한계와 percentile의 역할을 정확히 설명한다.

약한 답변 — 평균 vs Percentile
"평균 응답 시간이 괜찮으면 대체로 서비스는 정상이라고 봅니다."

평균의 함정을 모르는 답변이다. 소수 사용자의 심각한 지연을 놓칠 수 있다는 인식이 없다.

Senior-level Follow-up 질문과 답변 방향

Q. "정적 임계값(static threshold) 대신 이상 탐지(anomaly detection)를 써 본 경험이 있나요?"
핵심 논리: 트래픽 패턴이 요일/시간대에 따라 크게 변하는 서비스(예: 커머스의 새벽 vs 저녁)는 고정된 threshold가 항상 맞지 않는다. 과거 같은 요일/시간대 대비 편차를 기준으로 삼는 것이 더 정확할 수 있다.

실무 답변 방향: "트래픽이 요일별로 크게 다른 서비스에서는 전날 같은 시간대 대비 편차(예: 평소보다 3배 표준편차 이상 벗어남)를 기준으로 알림을 걸었습니다. 다만 이상 탐지는 설명 가능성이 떨어져서, 왜 알림이 울렸는지 온콜 담당자가 바로 이해하기 어려운 단점도 있어 핵심 경로에는 여전히 명시적 threshold를 함께 씁니다."
Q. "메트릭 시스템의 카디널리티가 폭발해서 장애가 난 적 있나요?"
핵심 논리: user_id, request_id처럼 무한히 늘어나는 값을 라벨로 쓰면 시계열 데이터베이스가 감당할 수 없는 수의 시계열을 생성한다. 이는 메트릭 시스템 자체의 장애로 이어진다.

실무 답변 방향: "한 번 개발자가 실수로 user_id를 메트릭 라벨에 추가했다가, Prometheus의 메모리 사용량이 폭증해서 메트릭 수집 자체가 멈춘 적이 있습니다. 이후 라벨에 사용할 수 있는 값을 화이트리스트로 관리하고, CI에서 카디널리티가 높은 라벨 추가를 자동으로 감지해서 리뷰를 거치도록 했습니다."
Q. "알림이 너무 많이 와서 팀이 무시하기 시작한 경험이 있나요? 어떻게 개선했나요?"
핵심 논리: Alert Fatigue는 한 번 생기면 진짜 장애 알림도 무시하게 만드는 심각한 문제다. 개선은 알림 감소가 아니라 '조치 가능한 알림만 남기는 것'이 목표여야 한다.

실무 답변 방향: "한 달간 울린 알림을 전수 조사해서, 실제로 조치가 필요했던 알림의 비율을 계산했더니 20%도 안 됐습니다. 원인 기반(리소스) 알림을 대부분 페이지에서 제거하고 증상 기반 알림만 남겼더니, 알림 수는 70% 줄었지만 놓친 장애는 없었습니다."
Q. "Symptom-based 알림과 Cause-based 알림을 어떻게 구분해서 운영하나요?"
핵심 논리: 증상 기반은 "사용자가 지금 영향을 받고 있다"는 신호이므로 즉시 대응(페이지), 원인 기반은 "곧 문제가 될 수 있다"는 예고이므로 예방적 대응(업무시간 확인)으로 심각도를 분리한다.

실무 답변 방향: "에러율, 지연 시간 같은 증상 기반 알림만 온콜 페이지로 보내고, 디스크 사용률, 커넥션 풀 사용률 같은 원인 기반 지표는 대시보드와 낮은 심각도의 슬랙 알림으로만 남겼습니다. 이렇게 하니 새벽에 깨어나야 하는 알림이 실제로 조치가 필요한 것들로만 좁혀졌습니다."
Q. "Burn rate 알림이 무엇이고, 일반적인 threshold 알림과 어떻게 다른가요?"
핵심 논리: Burn rate는 Error Budget(허용 가능한 실패 예산)을 얼마나 빠르게 소진하고 있는지를 보는 알림 방식이다. 단순 threshold보다 "이 속도로 계속 가면 이번 달 SLO를 못 지킨다"는 미래 예측이 가능하다.

실무 답변 방향: "이 부분은 Day 7에서 SLO와 함께 더 깊게 다룰 계획이지만, 핵심만 말씀드리면 단순 threshold는 순간의 상태만 보고, burn rate는 '이 소진 속도가 얼마나 위험한가'를 여러 시간 창으로 함께 봐서 급한 장애와 서서히 나빠지는 문제를 구분해서 알림 심각도를 다르게 줍니다."
⚠️

흔한 오해와 Trade-off

오해 1: "CPU 사용률이 높으면 무조건 장애다"

현실: CPU 사용률 90%는 오히려 리소스를 잘 활용하고 있다는 신호일 수 있다. 오토스케일링 환경에서는 일부러 높은 사용률까지 활용하도록 설계하기도 한다. 진짜 위험 신호는 사용률이 아니라 Saturation(대기량)이 계속 쌓이는 것이다.

올바른 접근: Utilization은 참고 지표로, Saturation과 실제 사용자 영향(RED의 Duration/Errors)을 함께 봐야 진짜 문제인지 판단할 수 있다.

오해 2: "알림은 최대한 많이 걸어두는 것이 안전하다"

현실: Day 2에서 다룬 Alert Fatigue와 정확히 같은 문제다. 알림이 많아질수록 팀은 알림을 확인하는 대신 무시하는 습관이 생기고, 이는 오히려 진짜 장애를 놓치는 결과로 이어진다.

올바른 접근: 알림의 목표는 "많이"가 아니라 "정확하게, 조치 가능하게"이다. 정기적으로 알림 발생 이력을 리뷰해서 불필요한 알림을 제거하는 것이 알림을 추가하는 것보다 중요하다.

오해 3: "메트릭이 있으면 로그나 트레이싱은 덜 중요하다"

현실: 메트릭은 "무엇이 이상한가(what)"는 빠르게 보여주지만 "왜 이상한가(why)"는 알려주지 않는다. 에러율이 올랐다는 것은 메트릭으로 알 수 있어도, 어떤 요청이 어떤 이유로 실패했는지는 로그나 트레이스를 봐야 한다.

올바른 접근: 메트릭으로 탐지(Detect) → 로그/트레이스로 진단(Diagnose)하는 역할 분담으로 이해한다.

Trade-off: 알림 민감도(Sensitivity) vs 알림 신뢰도(Precision)

threshold를 낮추면(민감하게) 실제 장애를 놓칠 확률(False Negative)은 줄지만, 정상 상황에서도 알림이 울릴 확률(False Positive)이 늘어난다. 반대로 threshold를 높이면 알림 신뢰도는 올라가지만 장애 탐지가 늦어질 수 있다.

핵심 Trade-off: "빠르게 알아채는 것 vs 잘못된 알림에 시달리지 않는 것". 실무에서는 서비스 중요도별 차등, multi-window 알림, 정기적 threshold 리뷰로 이 균형을 지속적으로 관리한다. 완벽한 균형점은 없고, 조직이 감당할 수 있는 알림량과 감당할 수 없는 장애 지연 리스크 사이의 의도적 선택이라는 것이 시니어급 이해다.

자기 점검 Rubric

이 답변들을 자신있게 말할 수 있으면 Day 3를 완료한 것이다.

RED와 USE 방법론의 구성 요소와, 각각 언제 사용하는지 설명할 수 있다 필수

서비스는 RED, 리소스는 USE라는 구분과 그 이유(측정 대상이 다름)를 말할 수 있어야 한다.

Utilization과 Saturation의 차이를 구체적 예시로 설명할 수 있다 필수

"사용률이 낮아도 대기량이 쌓이면 위험하다"는 논리를 설명할 수 있어야 한다.

평균과 percentile(p50/p95/p99)의 차이와, 왜 지연 시간은 percentile로 봐야 하는지 설명할 수 있다 필수

평균이 극단값을 희석시켜 소수의 나쁜 경험을 숨긴다는 논리가 포함되어야 한다.

False Positive와 False Negative의 트레이드오프 관계를 설명할 수 있다 강함

threshold를 조정하면 둘이 반비례한다는 것과, 정답이 없고 의도적 선택이라는 점을 말할 수 있어야 한다.

증상 기반(symptom-based)과 원인 기반(cause-based) 알림을 구분해서 설계할 수 있다 강함

증상 기반은 즉시 대응, 원인 기반은 예방적 대응으로 심각도를 분리한다는 설계 판단이 있어야 한다.

메트릭 카디널리티 문제와 라벨 설계 원칙을 설명할 수 있다 보너스

user_id 같은 무한 카디널리티 라벨을 피해야 하는 이유를 설명할 수 있으면 시니어급 답변이 된다.

알림 피로도를 줄이기 위한 실제 개선 경험(리뷰 프로세스, multi-window 등)을 말할 수 있다 보너스

일회성 설정이 아니라 지속적으로 튜닝한 운영 경험이면 더 좋다.

실무 연결 — 이 개념이 실제 어디에 쓰이는가

스타트업 (소규모 팀)

API 서버 몇 개, DB 한두 대인 초기에는 RED 지표(요청 수, 에러율, 응답 시간) 몇 개와 기본 리소스 알림만으로도 충분하다. 알림 채널도 슬랙 하나로 단순하게 유지하되, 결제처럼 핵심 경로만이라도 증상 기반 알림을 먼저 갖추는 것이 우선순위다.

성장기 회사 (수백만 사용자)

서비스 수가 늘어나면서 알림이 급격히 늘어나기 시작하는 시기다. 이때 심각도 체계(Severity)와 온콜 로테이션, runbook 연결이 필수가 된다. 알림 피로도 관리(정기 리뷰, 불필요한 알림 제거)가 본격적으로 필요해진다.

대형 플랫폼 (수천만 사용자)

수백 개 서비스, 수만 개 메트릭 시계열을 다루면서 카디널리티 관리가 인프라 전담 조직의 과제가 된다. Multi-window, multi-burn-rate 알림, 그리고 SLO 기반 알림 체계(Day 7)가 표준으로 자리잡는다. 이 규모에서는 메트릭 저장/쿼리 비용 자체도 큰 인프라 예산 항목이 된다.
