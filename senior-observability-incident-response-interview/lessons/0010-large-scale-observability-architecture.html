DAY 10 · Observability / 장애 대응 인터뷰

대규모 관측성 아키텍처

Day 1~9에서 배운 로그·메트릭·트레이스·알림·SLO·postmortem은 "한 서비스"를 전제로 한 이야기였다. 서비스가 수백 개, 서버가 수천 대가 되는 순간 관측성은 그 자체로 하나의 분산 시스템이 되고, "무엇을 수집할지"보다 "무엇을 버릴지"가 더 중요한 설계 질문이 된다

🎯

오늘의 핵심 면접 질문

질문 1 — 거의 항상 출제 (아키텍처 이해)
"수백 개의 마이크로서비스에서 나오는 로그, 메트릭, 트레이스를 어떻게 수집·저장·질의하는 파이프라인으로 설계하시겠습니까?"
질문 2 — 자주 출제 (실무 장애 경험)
"메트릭 시스템(Prometheus/TSDB)에 카디널리티 폭발이 일어나 전체 관측성 시스템이 멈춘 경험이 있나요? 어떻게 진단하고 해결했나요?"
질문 3 — 시니어 추가 질문 (비용·조직 판단)
"관측성 데이터 저장 비용이 인프라 비용의 상당 부분을 차지한다면, 어떤 기준으로 무엇을 줄이겠습니까?"
🔍

왜 인터뷰에서 묻는가

Day 1~9는 "이미 잘 구축된 관측성 시스템을 어떻게 활용해서 장애를 다루는가"를 다뤘다. 이번 주제는 반대 방향이다. "그 관측성 시스템 자체를 누가, 어떤 판단으로 만들고 유지하는가"를 확인한다. 10년차 이상에게 이 질문을 하는 이유는 다음과 같다.

1. 관측성 도구도 하나의 프로덕션 시스템이라는 것을 이해하는가

주니어는 "Prometheus를 설치하면 메트릭이 자동으로 잘 쌓인다"고 생각하기 쉽다. 하지만 규모가 커지면 관측성 파이프라인 자체가 장애의 원인이 될 수 있는 분산 시스템이 된다. 카디널리티 폭발로 TSDB가 OOM 나서 "정작 진짜 장애가 났을 때 대시보드조차 안 열리는" 상황은 실무에서 드물지 않다. 이 재귀적 위험을 설계 단계에서 인지하고 있는지가 핵심 신호다.

2. "다 수집하고 다 저장한다"가 답이 아니라는 것을 아는가

이론적으로는 모든 로그, 모든 span, 모든 메트릭 시계열을 무한정 보관하면 어떤 장애도 완벽히 재구성할 수 있다. 하지만 실제로는 비용, 질의 성능, 카디널리티, 데이터 거버넌스(PII)가 전부 이를 막는다. 시니어는 "무엇을 얼마나 오래, 어떤 해상도로 남길지"를 비즈니스 가치와 비용의 균형으로 설계할 수 있어야 한다.

3. 조직이 커질 때 관측성이 어떻게 진화하는지 아는가

한 팀, 한 서비스에서 통했던 방식(예: 모든 팀이 같은 Grafana 인스턴스에 대시보드를 직접 만든다)은 팀이 50개가 되면 무너진다. 멀티테넌시, 셀프서비스 온보딩, 표준화된 계측(instrumentation) 라이브러리 같은 조직 스케일링 문제를 관측성 관점에서 풀어본 경험이 있는지를 본다.

인터뷰어가 듣고 싶어하는 신호
"저는 관측성 파이프라인을 Agent(수집) → Collector/Gateway(전처리·라우팅) → Buffer(Kafka 등) → Storage(핫/웜/콜드 계층) → Query 계층으로 나눠서 설계했습니다. 카디널리티가 높은 레이블은 애초에 계측 단계에서 걸러지도록 린팅(linting)을 넣었고, 로그는 심각도와 서비스 중요도에 따라 보존 기간을 차등화해서 저장 비용을 관리했습니다. 이 파이프라인 자체의 헬스도 별도로 모니터링해서, '관측성이 관측성을 감시'하도록 만들었습니다."
📚

전제 개념 — 쉬운 언어로 먼저 이해하기

카디널리티(Cardinality)란 무엇인가 — "값의 조합 개수"

카디널리티는 어떤 레이블(label/tag)이 가질 수 있는 서로 다른 값의 개수, 그리고 그 레이블들을 조합했을 때 생기는 고유한 시계열(time series)의 개수다. Day 3에서 다룬 메트릭에 service, endpoint, status_code 레이블을 붙이면 각각의 조합이 하나의 독립된 시계열이 된다.

문제는 여기에 user_id나 request_id처럼 사실상 무한한 값을 가지는 레이블을 실수로 붙이는 순간 발생한다. 사용자가 100만 명이면 시계열도 최대 100만 개가 만들어지고, TSDB(Time Series Database)는 메모리에 각 시계열의 인덱스를 올려야 하므로 순식간에 메모리를 소진하고 죽는다. 이것이 카디널리티 폭발(cardinality explosion)이다.

🟢 낮은 카디널리티
service(수십 개), env(3~4개), status_class(2xx/4xx/5xx). 메트릭 레이블로 안전하다.
🟡 주의가 필요한 카디널리티
endpoint(수백 개), instance_id(오토스케일링 시 수천 개까지 증가 가능). 조합 폭증을 감시해야 한다.
🔴 높은 카디널리티(금지)
user_id, request_id, 원시 email. 메트릭 레이블로 쓰면 안 되고, 로그/트레이스로 보내야 한다.
흔한 오해 — "레이블을 많이 붙일수록 나중에 분석하기 좋다"
메트릭은 로그가 아니다. 메트릭은 "빠른 집계 질의"를 위해 미리 인덱싱된 시계열 구조이고, 이 구조는 레이블 조합 수에 비례해서 비용이 커진다. 세밀한 개별 요청 단위 분석이 필요하면 메트릭이 아니라 Day 4의 트레이스/로그로 해결해야 한다 — "모든 것을 메트릭 하나로 풀려는 것"이 흔한 설계 실수다.

관측성 파이프라인의 구성 요소 — 데이터가 흐르는 길

대규모 환경에서 로그/메트릭/트레이스는 애플리케이션에서 바로 저장소로 가지 않는다. 중간에 여러 단계를 거친다.

📡
Agent/SDK
앱 옆(사이드카) 또는 라이브러리로 계측 데이터 생성
🔀
Collector
전처리·필터링·라우팅 (예: OTel Collector)
🗂️
Buffer
Kafka 등으로 스파이크 흡수, 저장소 보호
💾
Storage
핫/웜/콜드 계층별 시계열·로그·트레이스 저장소
🔎
Query
대시보드(Day 6)·알림(Day 3)이 질의하는 계층

Collector(수집기)가 왜 필요한가? 앱이 직접 저장소로 데이터를 보내면, 저장소 벤더를 바꿀 때마다 모든 서비스의 코드를 고쳐야 하고, 저장소 장애가 앱까지 전파될 위험이 있다. Collector는 계측(어떻게 만드는가)과 백엔드(어디에 저장하는가)를 분리하는 계층이다. 여기서 PII 마스킹, 샘플링, 카디널리티 제한 같은 정책을 중앙에서 강제할 수 있다.

Buffer(Kafka 등)가 왜 필요한가? 장애 상황에서는 평소보다 훨씬 많은 로그와 메트릭이 한꺼번에 쏟아진다("장애 폭풍"). 이때 저장소가 이 트래픽을 바로 받으면 저장소 자체가 과부하로 무너질 수 있다. Buffer는 수집 속도와 저장 속도 사이의 완충 지대로, 저장소가 자기 속도로 안전하게 처리하도록 보호한다.

핫/웜/콜드 저장 계층 — 왜 하나의 저장소로 안 되는가

최근 데이터는 빠르게 조회해야 하지만(대시보드, 알림), 오래된 데이터는 드물게, 감사/트렌드 분석 목적으로만 조회된다. 이 두 요구를 하나의 저장소로 만족시키려 하면 비용과 성능 중 하나를 반드시 희생하게 된다.

🔥 Hot (예: 최근 7일)
SSD/메모리 기반, 원본 해상도 유지, 실시간 알림·대시보드가 여기를 질의.
🌤️ Warm (예: 8~90일)
저렴한 스토리지로 이동, 다운샘플링(down-sampling)해서 해상도를 낮춤(1분 → 1시간 평균).
🧊 Cold (예: 90일~수년)
오브젝트 스토리지(S3 등)에 압축 보관, 규정 준수·연간 트렌드 분석용, 조회는 느리고 드묾.
🧩

실무 프레임워크 — 대규모 관측성 시스템을 설계하는 절차

1. 계측 표준을 먼저 정한다 (수집 이전 단계)

파이프라인을 아무리 잘 만들어도, 각 팀이 제멋대로 레이블을 붙이면 카디널리티 통제가 불가능하다. OpenTelemetry 같은 벤더 중립 표준으로 공통 계측 라이브러리를 배포하고, 여기에 카디널리티 위험이 큰 레이블(원시 사용자 ID 등)을 애초에 금지하는 린팅/정책을 내장한다.

2. Collector 계층에서 정책을 중앙 집행한다

PII 마스킹, 샘플링 비율(Day 4의 tail-based sampling), 저비용 서비스의 로그 레벨 강등 같은 정책을 각 앱이 아니라 Collector에서 일괄 적용한다. 이렇게 하면 정책이 바뀔 때 수백 개 서비스를 재배포할 필요 없이 Collector 설정만 바꾸면 된다.

3. 데이터 종류별로 보존 정책을 차등화한다

모든 로그를 1년간 원본 그대로 보관하는 것은 대부분 낭비다. 심각도(ERROR/WARN vs INFO/DEBUG)와 서비스 중요도(결제 vs 배치 작업)를 기준으로 보존 기간과 해상도를 다르게 정한다. 예: ERROR 로그는 1년, DEBUG 로그는 3일, 결제 서비스 메트릭은 원본 해상도로 1년 보관.

4. 관측성 파이프라인 자체를 관측한다 ("메타 모니터링")

수집기 큐 적체, Collector CPU/메모리, 저장소 write 실패율 같은 파이프라인 자체의 헬스 지표를 별도의(가능하면 물리적으로 분리된) 최소 관측 시스템으로 감시한다. 그렇지 않으면 "관측성 시스템이 죽어서, 관측성 시스템이 죽은 것도 모르는" 상황이 생긴다.

5. 멀티테넌시와 셀프서비스로 조직 확장에 대응한다

팀이 늘어나면 관측성 팀이 모든 대시보드/알림을 대신 만들어줄 수 없다. Dashboard as Code(Day 6), 표준 알림 템플릿, 팀별 네임스페이스 격리를 통해 각 팀이 스스로 온보딩하면서도 전체 시스템의 카디널리티/비용 예산을 넘지 않도록 셀프서비스 구조를 만든다.

시니어 답변에서 나와야 하는 내용
"저는 계측 표준을 OpenTelemetry로 통일하고, 카디널리티 위험이 큰 레이블은 CI 단계의 린트로 사전에 차단했습니다. Collector에서 PII 마스킹과 tail-based sampling을 중앙 집행했고, 로그는 심각도·서비스 중요도별로 보존 기간을 차등화해서 저장 비용을 통제했습니다. 파이프라인 자체의 헬스도 별도의 경량 모니터링으로 감시해서, 관측성 시스템 장애가 진짜 장애 대응을 막지 않도록 설계했습니다."

답변 사고 순서 — 면접장에서 어떻게 풀어나가는가

질문 2: "카디널리티 폭발로 관측성 시스템이 멈춘 경험을 어떻게 진단하고 해결했나요?"

  1. 증상부터 명확히 진술 (10초) "TSDB의 메모리 사용량이 급증하고 write 지연이 늘면서, 결국 새로운 메트릭 수집(scrape/ingest)이 실패하기 시작했습니다. 대시보드도 함께 응답이 느려졌습니다."
  2. 원인 후보를 시계열 개수 증가 추이로 좁힘 (15초) "TSDB가 자체적으로 노출하는 '활성 시계열 수' 메트릭을 먼저 확인해서, 특정 메트릭 이름에서 시계열 수가 급증한 시점과 배포 시점이 일치하는지 상관관계를 봤습니다. 이건 Day 5의 RCA 절차와 동일합니다."
  3. 레이블 단위로 원인 특정 (15초) "해당 메트릭의 레이블 조합을 살펴보니, 최근 배포에서 실수로 request_id를 레이블로 추가한 것을 발견했습니다. 이 레이블 하나가 시계열 수를 수백 배로 늘렸습니다."
  4. 즉시 완화 후 근본 수정 (10초) "우선 Collector 단계에서 해당 레이블을 drop하는 relabel 규칙을 긴급 적용해서 즉시 완화했고, 이후 해당 서비스의 계측 코드를 수정해서 원천적으로 재발을 막았습니다. 그리고 CI에 카디널리티 위험 레이블을 자동 탐지하는 린트를 추가했습니다."

질문 3: "관측성 데이터 저장 비용을 줄여야 한다면 어떤 기준으로 판단하나요?"

  1. "전체를 균등하게 줄인다"는 접근을 배제 (10초) "모든 서비스의 보존 기간을 일괄로 줄이면 결제처럼 감사·규정 준수가 중요한 서비스의 위험이 커집니다. 균등 삭감이 아니라 차등화가 먼저입니다."
  2. 데이터 유형별 실제 사용 패턴 확인 (15초) "먼저 어떤 데이터가 실제로 조회되는지 access 로그를 봅니다. 대부분의 조직에서 7일 이상 지난 DEBUG 로그는 거의 조회되지 않는다는 것을 데이터로 확인했습니다."
  3. 다운샘플링과 계층 이동을 우선 적용 (15초) "삭제보다 먼저 다운샘플링(원본 해상도 → 집계된 해상도)과 콜드 스토리지 이전을 적용합니다. 이는 완전 삭제보다 리스크가 적으면서 대부분의 비용 절감 효과를 얻습니다."
  4. 남은 절감분은 비즈니스 중요도 기준으로 결정 (10초) "그래도 부족하면 서비스 소유팀과 함께 중요도를 재확인하고, 낮은 중요도 서비스의 원본 해상도 보존 기간을 줄이는 방식으로 진행하되, 결정 과정과 근거를 문서화해서 이후 감사 요청 시 설명할 수 있게 합니다."

핵심 답변 예시

질문 1 모범 답변: "대규모 관측성 파이프라인 설계"

강한 답변

"저는 파이프라인을 Agent → Collector → Buffer → Storage → Query 다섯 단계로 나눠서 설계합니다. Agent/SDK는 OpenTelemetry로 표준화해서 계측 방식이 팀마다 흩어지지 않게 하고, Collector에서 PII 마스킹, 샘플링, 카디널리티 위험 레이블 차단 같은 정책을 중앙에서 집행합니다.

Buffer로 Kafka 같은 메시지 큐를 둬서 장애로 로그·메트릭이 폭증해도 저장소가 직접 그 부하를 받지 않게 보호하고, Storage는 핫(최근·원본 해상도)/웜(다운샘플링)/콜드(장기 압축 보관) 계층으로 나눠서 비용과 조회 성능의 균형을 맞춥니다. 마지막으로 이 파이프라인 자체의 헬스(큐 적체, write 실패율)를 별도로 감시해서, 관측성 시스템 자체의 장애가 실제 인시던트 대응을 막지 않도록 설계했습니다."

좋은 답변 vs 약한 답변의 차이

강한 답변 — 카디널리티 폭발 대응
"활성 시계열 수 추이로 원인 메트릭을 좁히고, 문제 레이블을 특정한 뒤 Collector에서 긴급 drop, 이후 계측 코드 수정과 CI 린트로 재발을 막았습니다."

진단 → 즉시 완화 → 근본 수정 → 재발 방지까지 이어지는 완결된 절차가 있다.

약한 답변 — 카디널리티 폭발 대응
"TSDB 서버의 메모리를 늘려서 해결했습니다."

증상만 완화했을 뿐 근본 원인(잘못된 레이블 설계)을 찾지 않았고, 같은 문제가 다시 발생할 것이다.

강한 답변 — 저장 비용 절감 판단
"실제 조회 패턴을 데이터로 확인하고, 삭제보다 다운샘플링/계층 이동을 우선 적용하며, 서비스 중요도별로 차등화했습니다."

데이터 기반 판단과 리스크가 낮은 방법부터 시도하는 순서가 있다.

약한 답변 — 저장 비용 절감 판단
"전체 로그 보존 기간을 30일에서 7일로 일괄 줄였습니다."

서비스 중요도나 실제 사용 패턴을 고려하지 않은 일괄 조치로, 규정 준수나 장애 분석에 필요한 데이터를 잃을 위험이 있다.

Senior-level Follow-up 질문과 답변 방향

Q. "관측성 파이프라인 자체가 장애 났을 때 어떻게 관측하나요? '관측성을 관측하는 관측성'은 무한 재귀 아닌가요?"
핵심 논리: 무한 재귀를 피하려면 메인 관측성 스택과 완전히 독립된, 훨씬 단순하고 가벼운 별도의 "메타 모니터링" 계층이 필요하다. 여기서는 상세한 대시보드가 아니라 "살아있는지"만 확인하는 최소한의 헬스체크와 알림이면 충분하다.

실무 답변 방향: "저는 메인 관측성 스택과 별도의 인프라(다른 리전이나 별도 클러스터)에 최소한의 heartbeat/synthetic 체크를 뒀습니다. 이 체크는 Collector와 Storage에 실제로 데이터를 보내고 조회까지 되는지 end-to-end로 확인하고, 실패하면 온콜 엔지니어에게 별도 채널(예: SMS)로 알립니다. 이 계층은 의도적으로 단순하게 유지해서, 그 자체가 또 다른 장애 지점이 되지 않도록 합니다."
Q. "여러 리전/클라우드에 걸친 시스템에서 관측성 데이터를 중앙 집중할지, 리전별로 분리할지 어떻게 결정하나요?"
핵심 논리: 중앙 집중은 전역적인 상관관계 분석(리전 간 장애 전파)을 쉽게 하지만, 리전 간 네트워크 장애 시 관측성 자체가 끊길 위험과 데이터 주권(data residency) 규제 문제가 있다. 완전 분리는 안정성은 높지만 전역 장애 시야를 잃는다.

실무 답변 방향: "저는 리전별로 로컬 저장소와 알림 계층을 독립적으로 두어, 리전 간 네트워크 장애가 나도 각 리전이 스스로를 관측할 수 있게 했습니다. 그 위에 리전 경계를 넘는 사용자 여정(Day 7의 user journey SLO)만 별도의 페더레이션 계층으로 집계해서, 전역 시야와 리전별 복원력을 동시에 확보했습니다."
Q. "PII가 로그나 트레이스에 실수로 들어가는 것을 구조적으로 어떻게 막나요?"
핵심 논리: 각 개발자에게 "PII를 로깅하지 마세요"라고 안내하는 것은 사람의 실수를 막지 못한다. 구조적으로 막으려면 계측 라이브러리와 Collector 두 지점에서 자동 탐지·마스킹이 필요하다.

실무 답변 방향: "저는 공통 로깅 라이브러리에 알려진 PII 패턴(이메일, 카드번호 형식)을 자동 마스킹하는 필터를 기본 내장했고, Collector 단계에서도 동일한 패턴을 이중으로 검사해서 라이브러리를 안 쓴 레거시 서비스도 보호했습니다. 이렇게 계측 단계와 파이프라인 단계 이중 방어로 사람의 실수에 의존하지 않도록 만들었습니다."
⚠️

흔한 오해와 Trade-off

오해 1: "저장 공간만 늘리면 카디널리티 문제는 해결된다"

현실: 카디널리티 문제의 병목은 대부분 디스크 용량이 아니라 메모리(인덱스)와 질의 성능이다. 시계열이 늘어날수록 TSDB는 각 시계열의 메타데이터를 메모리에 유지해야 하므로, 디스크를 아무리 늘려도 메모리 부족으로 죽는다.

올바른 접근: 카디널리티는 저장 공간이 아니라 레이블 설계로 해결해야 하는 문제다.

오해 2: "모든 데이터를 최대한 오래 보관하는 것이 가장 안전하다"

현실: 무제한 보관은 비용뿐 아니라 규정 준수 리스크(불필요한 개인정보 장기 보관)와 질의 성능 저하라는 대가를 치른다. "장애 분석을 위해 다 남겨야 한다"는 생각은 실제 조회 패턴과 맞지 않는 경우가 많다.

올바른 접근: 데이터 유형·중요도별로 보존 정책을 명시적으로 설계하고, 그 근거를 문서화한다.

오해 3: "관측성 파이프라인은 부가 시스템이라 프로덕션만큼 신경 쓸 필요 없다"

현실: 진짜 장애가 났을 때 대시보드와 알림이 살아있어야 대응이 가능하다. 관측성 파이프라인이 죽으면 "장애를 장애로 인식하지도 못하는" 최악의 상황이 된다.

올바른 접근: 관측성 파이프라인도 SLO(Day 7)를 갖는 하나의 프로덕션 서비스로 취급하고, 별도의 최소 메타 모니터링으로 감시한다.

Trade-off: 데이터 해상도(정확도) vs 비용·성능

원본 해상도를 오래 유지할수록 세밀한 장애 분석이 가능하지만, 저장·질의 비용이 커진다. 다운샘플링은 비용을 줄이지만 미세한 이상 신호(짧은 스파이크)를 뭉개버릴 수 있다. 이 trade-off는 없앨 수 없고, 데이터의 나이와 중요도에 따라 계층화(핫/웜/콜드)로 관리하는 것이 최선이라는 점을 명확히 해야 한다.

실무 연결 — 이 개념이 실제 어디에 쓰이는가

스타트업 (소규모 팀)

매니지드 SaaS(예: Datadog, Grafana Cloud)를 그대로 쓰며 파이프라인을 직접 운영하지 않는다. 이 단계의 핵심 과제는 계측 표준을 정하고, 요금이 청구량(로그 볼륨, 커스텀 메트릭 수) 기준이라는 것을 이해해 카디널리티를 초기부터 통제하는 것이다.

성장기 회사 (수백만 사용자)

SaaS 비용이 인프라 비용의 상당 부분을 차지하기 시작하며, 일부를 자체 호스팅(OSS 스택)으로 전환하거나 하이브리드로 구성한다. 이때 Collector 계층을 자체적으로 두어 카디널리티/PII 정책을 중앙에서 통제하기 시작한다.

대형 플랫폼 (수천만 사용자)

전용 관측성 플랫폼 팀이 존재하며, 수백 개 팀에 셀프서비스로 계측 SDK, 대시보드 템플릿, 알림 정책을 제공한다. 멀티리전 페더레이션, 계층화된 저장소, 자체 개발한 카디널리티 가드레일(계측 코드 배포 전 자동 검사)이 표준 인프라로 자리 잡는다. 관측성 파이프라인 자체가 자체 SLO와 온콜 로테이션을 가진 하나의 프로덕션 서비스로 운영된다.
