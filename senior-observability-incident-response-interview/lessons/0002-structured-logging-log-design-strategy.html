구조화 로그, 로그 레벨 전략, Correlation ID, 컨텍스트 전파 — "좋은 로그"를 설계하는 사고방식

오늘의 핵심 면접 질문

질문 1 — 자주 출제 (개념 + 설계)
"좋은 로그(good log)란 어떤 로그인가요? 로그를 설계할 때 어떤 기준으로 접근하시나요?"

"마이크로서비스 환경에서 여러 서비스를 거치는 요청 하나를 로그로 어떻게 추적하시나요?"

"로그 레벨을 어떻게 설계하고 운영하시나요? 프로덕션에서 DEBUG 로그는 어떻게 다루시나요?"

왜 인터뷰에서 묻는가

로그는 개발자 대부분이 매일 다루는 도구라서 "당연히 안다"고 생각하기 쉽다. 하지만 시니어 인터뷰에서 로그 질문은 의외로 변별력이 크다. 면접관이 실제로 보는 것은 다음 세 가지다.

1. 로그를 "기록"이 아니라 "설계 대상"으로 다루는가

주니어는 print나 console.log를 디버깅 도구로만 쓴다. 시니어는 로그를 나중에 누군가(미래의 나, 다른 팀원, 온콜 담당자)가 장애 상황에서 검색할 자료로 미리 설계한다. "이 로그 한 줄만 보고 무슨 일이 있었는지 알 수 있는가"를 기준으로 로그를 쓰는지가 핵심이다.

2. 분산 시스템에서 요청을 추적하는 능력

단일 서버에서는 로그 파일 하나만 보면 됐다. 서비스가 5개, 10개로 늘어나면 하나의 사용자 요청이 여러 서비스를 거치면서 로그가 흩어진다. 이걸 다시 하나로 묶어서 볼 수 있는 능력(Correlation ID, 컨텍스트 전파)이 있는지가 마이크로서비스 경험의 진짜 증거다.

3. 비용과 신호 대 잡음비를 이해하는가

"로그는 많을수록 좋다"는 흔한 오해다. 시니어는 로그 저장/검색 비용, 그리고 너무 많은 로그가 오히려 진짜 신호를 찾기 어렵게 만든다는 것(Signal-to-Noise)을 이해하고 있어야 한다.

인터뷰어가 듣고 싶어하는 신호
"저는 로그를 쓸 때 '이 로그 한 줄이 검색 결과에 나왔을 때, 추가 질문 없이 상황을 이해할 수 있는가'를 기준으로 씁니다. 그래서 사람이 읽는 문장형 로그가 아니라, 기계가 파싱할 수 있는 구조화된 필드로 로그를 남기고, 모든 요청에 추적 가능한 ID를 포함시킵니다."
📚

전제 개념 — 쉬운 언어로 먼저 이해하기

비구조화 로그(Unstructured Log)란?

많은 개발자가 처음 배우는 로그 방식이다. 그냥 사람이 읽기 편하게 문장으로 찍는다.

2026-06-30 14:22:01 ERROR 주문 처리 실패 user=12345 order=98765 사유: 결제 타임아웃

사람이 눈으로 읽으면 이해된다. 문제는 기계가 이걸 검색하고 집계하기 어렵다는 것이다. "user=12345 형태로 쓴 로그"와 "userId: 12345 형태로 쓴 로그"와 "고객번호 12345 형태로 쓴 로그"가 섞여 있으면, 하나의 쿼리로 전부 찾을 수 없다.

비구조화 로그의 실제 문제
장애가 났을 때 "user=12345와 관련된 에러를 모두 찾아줘"라는 요청을 받으면, 비구조화 로그에서는 정규식을 여러 개 짜야 하고, 로그를 쓴 사람마다 표현 방식이 달라서 일부는 반드시 놓친다. 100개 서비스가 있다면 100가지 로그 포맷이 존재할 수도 있다.

구조화 로그(Structured Log)란?

로그를 문장이 아니라 정해진 필드를 가진 데이터(주로 JSON)로 남기는 방식이다.

{
  "timestamp": "2026-06-30T14:22:01.482Z",
  "level": "ERROR",
  "service": "order-service",
  "trace_id": "8f3a2c91-...",
  "user_id": "12345",
  "order_id": "98765",
  "event": "payment_timeout",
  "message": "결제 타임아웃으로 주문 처리 실패",
  "duration_ms": 5023,
  "payment_gateway": "card_company_a"
}

이렇게 남기면 user_id: "12345"로 정확히 검색할 수 있고, event: "payment_timeout"로 같은 유형의 장애를 전부 모아볼 수 있고, duration_ms > 3000처럼 조건 검색과 집계가 가능해진다.

핵심 차이 한 문장
비구조화 로그는 사람이 읽으라고 쓰는 것이고, 구조화 로그는 기계가 검색하고, 사람은 결과를 읽으라고 쓰는 것이다.

Correlation ID(trace_id)란?

하나의 사용자 요청이 여러 서비스를 거칠 때, 그 요청 전체를 하나로 묶어주는 식별자다. 실무에서는 흔히 trace_id, request_id, correlation_id 등으로 부른다.

동작 원리는 간단하다:

  1. 요청 진입 시점에 ID 생성 API Gateway나 최초 진입 서비스에서 trace_id를 하나 생성한다. (예: UUID)
  2. 모든 로그에 포함 이 서비스 안에서 남기는 모든 로그에 이 trace_id를 필드로 포함시킨다.
  3. 다음 서비스로 호출할 때 전파 다른 서비스를 호출할 때 HTTP 헤더(예: X-Trace-Id)나 메시지 큐 메타데이터에 같은 값을 실어 보낸다.
  4. 받은 서비스도 동일하게 로깅 + 전파 다음 서비스는 헤더에서 trace_id를 읽어서 자신의 로그에도 똑같이 남기고, 또 다음 서비스로 전파한다.

결과적으로 trace_id 하나로 검색하면, 그 요청이 거쳐간 모든 서비스의 로그를 시간 순서대로 볼 수 있다.

비유로 이해하기
택배 송장 번호와 같다. 상품이 창고, 물류센터, 배송기사를 거치는 동안 각 단계에서 같은 송장 번호로 상태를 기록한다. 송장 번호 하나만 있으면 상품이 지금 어디 있는지, 어디서 지연됐는지 전체 이력을 추적할 수 있다. trace_id가 없다면 각 물류센터에 따로 전화해서 "이 시간에 이 무게의 상품 못 보셨어요?"라고 묻는 것과 같다.

로그 레벨(Log Level)이란?

로그의 심각도/중요도를 구분하는 태그다. 일반적으로 다음 순서로 심각해진다.

레벨의미사용 시점
DEBUG 개발 중 상세 디버깅 정보 변수 값, 함수 진입/종료 등. 프로덕션 상시 비활성화가 보통
INFO 정상적인 주요 이벤트 "주문이 생성됨", "결제가 성공함" 같은 비즈니스 이벤트
WARN 비정상이지만 서비스는 계속 동작 재시도 발생, deprecated API 호출, 임계치 근접
ERROR 요청 처리 실패, 즉각 조치 필요할 수 있음 결제 실패, DB 연결 실패, 예외 발생
FATAL/CRITICAL 서비스 전체가 동작 불가능 프로세스 종료 직전, 필수 설정 로딩 실패
흔한 실수
많은 개발자가 모든 것을 ERROR로 찍거나, 반대로 모든 것을 INFO로 찍는다. 재시도로 정상 복구된 상황을 ERROR로 남기면, 온콜 담당자는 실제로 조치가 필요 없는 알림에 익숙해져서(Alert Fatigue) 진짜 ERROR도 무시하게 된다.
🧩

로그 설계 원칙 — 무엇을, 왜 포함해야 하는가

면접에서 "좋은 로그를 어떻게 설계하나요?"라는 질문에 이 원칙들로 답하면 구조가 잡힌다.

1. 컨텍스트를 항상 포함한다

에러 메시지만 있고 "누구의, 어떤 요청에서" 발생했는지 없는 로그는 쓸모가 없다. 최소한 다음 필드는 모든 로그에 공통으로 포함하는 것이 좋다.

  • trace_id — 요청 전체 추적용
  • service — 어떤 서비스에서 남긴 로그인지
  • user_id / tenant_id — 어떤 사용자/조직 관련인지 (개인정보는 마스킹 또는 별도 관리)
  • timestamp — ISO 8601 형식, 타임존 명시(UTC 권장)
  • level — 심각도

2. 이벤트 이름을 명시적으로 구분한다

message 필드의 자연어 문장만으로 검색하지 말고, event: "payment_timeout"처럼 코드로 검색 가능한 이벤트 이름을 별도 필드로 둔다. 문장은 사람마다 조금씩 다르게 쓰지만, 이벤트 이름은 팀 내 표준으로 고정할 수 있다.

3. 민감 정보는 남기지 않거나 마스킹한다

비밀번호, 카드 번호, 주민등록번호, 인증 토큰 등을 로그에 그대로 남기면 보안 사고와 컴플라이언스 위반이 된다. 마스킹(예: 카드번호 앞 6자리+뒤 4자리만) 또는 아예 로그에서 제외한다.

4. 로그 레벨을 환경과 상황에 맞게 동적으로 조절한다

프로덕션은 기본 INFO 이상으로 운영하되, 특정 장애 조사 중에는 재배포 없이 특정 서비스나 특정 요청 범위만 DEBUG로 올릴 수 있는 기능(Dynamic Log Level)을 갖추는 것이 시니어다운 설계다.

5. 로그는 "질문에 답할 수 있는가"로 검증한다

로그를 작성한 뒤 스스로에게 물어본다: "이 로그만 보고 어떤 사용자의 어떤 요청이 왜 실패했는지 알 수 있는가?" 답이 "아니오"라면 컨텍스트가 부족한 것이다.

시니어 답변에서 나와야 하는 내용
"저희 팀은 로그 스키마를 공통 라이브러리로 표준화해서, 모든 서비스가 같은 필드 이름(trace_id, service, event 등)을 강제로 사용하도록 했습니다. 개발자가 로그를 남길 때 필드 이름을 고민하지 않게 만든 것이 핵심이었고, 이후로 Kibana에서 서비스 전체를 가로지르는 검색이 가능해졌습니다."
🧠

답변 사고 순서 — 면접장에서 어떻게 풀어나가는가

질문 1: "좋은 로그란 어떤 로그인가요?"

  1. 기준 제시 (10초) "저는 좋은 로그를 '한 줄만 보고 추가 질문 없이 상황을 이해할 수 있는 로그'로 정의합니다."
  2. 구조화가 왜 필요한지 (15초) "이를 위해서는 문장형 로그가 아니라 구조화된(JSON) 로그가 필요합니다. 기계가 검색·집계할 수 있어야 하기 때문입니다."
  3. 필수 컨텍스트 필드 (20초) "최소한 trace_id, service, user_id, event, timestamp는 모든 로그에 공통으로 포함시킵니다."
  4. 실무 경험 연결 (30초) 본인이 로그 스키마를 표준화했거나, 컨텍스트 부족으로 장애 분석이 늦어졌던 경험을 구체적으로 말한다.
  5. 마무리 — 비용 인식 (10초) "다만 로그가 많을수록 좋은 것은 아니라서, 레벨과 샘플링으로 비용과 신호 대 잡음비를 관리합니다."

질문 2: "여러 서비스를 거치는 요청을 로그로 어떻게 추적하나요?"

  1. 문제 정의 (10초) "단일 서비스 로그만으로는 요청이 어느 서비스에서 얼마나 걸렸는지, 어디서 실패했는지 알 수 없습니다."
  2. 해결책 제시 (15초) "그래서 trace_id를 요청 진입 시점에 생성하고, 서비스 간 호출마다 HTTP 헤더로 전파해서 모든 로그에 같은 trace_id가 남도록 합니다."
  3. 구현 디테일로 신뢰도 확보 (20초) "이걸 애플리케이션 코드마다 수동으로 하면 누락이 생기기 때문에, 공통 미들웨어나 인터셉터 레벨에서 자동으로 처리하도록 인프라화했습니다."
  4. 실제 효과로 마무리 (15초) "이 표준화 이후 여러 서비스에 걸친 장애 분석 시간이 크게 줄었습니다."

핵심 답변 예시

질문 1 모범 답변: "좋은 로그란?"

강한 답변

"저는 좋은 로그를 '이 로그 한 줄만 보고, 추가로 코드를 찾아보거나 다른 사람에게 물어보지 않아도 상황을 이해할 수 있는 로그'라고 정의합니다. 이를 위해 문장형이 아니라 JSON 형태의 구조화 로그를 씁니다.

모든 로그에 trace_id, service, event, user_id, timestamp를 공통 필드로 강제했습니다. 특히 event 필드는 자유 텍스트가 아니라 payment_timeout, order_created처럼 팀 내에서 미리 합의한 이름을 쓰도록 코드 리뷰 체크리스트에 넣었습니다.

실제로 이전에는 로그 메시지가 개발자마다 제각각이라 '결제 실패'와 '결제 오류'와 'payment failed'가 섞여 있어서 검색이 안 됐던 경험이 있습니다. 이후 공통 로깅 라이브러리를 만들어서 필드와 이벤트 이름을 강제하니, 장애 시 검색 시간이 크게 줄었습니다."

질문 2 모범 답변: "분산 요청 추적"

강한 답변

"저희는 API Gateway 진입 시점에 trace_id를 생성하고, 이후 모든 서비스 간 호출에 X-Trace-Id 헤더로 전파하도록 표준화했습니다. 각 서비스는 로깅 미들웨어에서 이 헤더를 자동으로 읽어 모든 로그에 포함시키기 때문에, 개발자가 매번 신경 쓸 필요가 없습니다.

비동기 처리(메시지 큐)에서도 동일한 원칙을 적용해서, 메시지 헤더에 trace_id를 실어 보냈습니다. 이렇게 하지 않으면 큐를 거치는 순간 추적이 끊기는 경우가 많은데, 실제로 저희도 초기에는 큐 이후 로그가 끊겨서 30분 걸릴 분석이 3시간 넘게 걸린 적이 있었습니다.

이후에는 trace_id 하나로 Kibana에서 검색하면 요청이 거친 모든 서비스의 로그를 시간순으로 볼 수 있게 됐고, 여기에 분산 트레이싱 도구(Jaeger)의 span 정보도 같은 trace_id로 연결해서 로그와 트레이스를 함께 봅니다."

좋은 답변 vs 약한 답변의 차이

강한 답변 — 좋은 로그
"모든 로그에 trace_id, event, user_id를 강제하는 공통 로깅 라이브러리를 만들었고, event 이름은 팀 컨벤션으로 고정했습니다. 이후 검색 시간이 눈에 띄게 줄었습니다."

표준화 방법과 실제 효과가 함께 있다. 팀 차원의 설계 판단이 담겨있다.

약한 답변 — 좋은 로그
"로그는 최대한 자세하게 남기고, try-catch에서 에러가 나면 로그를 찍습니다."

구조화, 컨텍스트, 검색 가능성에 대한 언급이 없다. "자세하게"라는 말은 오히려 비용/노이즈 문제를 모른다는 신호다.

강한 답변 — 분산 추적
"trace_id를 진입점에서 생성해 미들웨어 레벨에서 자동 전파하도록 인프라화했습니다. 큐를 거칠 때도 메시지 헤더에 실어서 끊기지 않게 했습니다."

비동기 구간까지 고려한 완결성. 자동화(미들웨어)로 실수를 구조적으로 방지한 판단이 있다.

약한 답변 — 분산 추적
"각 서비스 로그를 시간 순서로 비교해서 어떤 요청인지 유추합니다."

ID 기반 추적이 아니라 타임스탬프 유추다. 여러 요청이 동시에 오면 구분이 불가능하다. 실무 규모를 겪어보지 못한 답변으로 들린다.

강한 답변 — 로그 레벨
"재시도로 정상 복구된 경우는 WARN으로, 최종 실패만 ERROR로 구분했습니다. ERROR는 온콜 알림과 연결되어 있어서, 이 구분이 없으면 Alert Fatigue가 생깁니다."

레벨 구분이 알림 체계와 어떻게 연결되는지까지 설명한다.

약한 답변 — 로그 레벨
"에러가 나면 ERROR, 그 외에는 INFO로 찍습니다."

WARN의 역할, 알림 연동, Alert Fatigue에 대한 이해가 전혀 없다.

Senior-level Follow-up 질문과 답변 방향

Q. "trace_id가 서비스 경계를 넘어갈 때(예: 다른 회사 API 호출) 끊기는 경우는 어떻게 처리하나요?"
핵심 논리: 외부 시스템은 우리의 trace_id 규약을 모른다. 이 경계에서는 우리 쪽에서 "외부 호출 시작/종료" span을 별도로 남기고, 외부 시스템이 반환하는 자체 요청 ID(있다면)를 함께 로깅해서 연결 고리를 남긴다.

실무 답변 방향: "외부 API 호출 직전/직후에 우리 trace_id와 외부 응답의 request-id를 함께 로그로 남겨서, 외부사에 문의할 때 그 request-id로 연결할 수 있게 했습니다. 완전한 자동 연결은 불가능하지만 최소한의 추적 고리는 만들어 둡니다."
Q. "로그 저장 비용이 너무 커졌습니다. 어떻게 줄이시겠습니까?"
핵심 논리: 모든 로그를 동일한 가치로 취급하지 않는다. 접근 빈도와 중요도에 따라 계층화(hot/warm/cold storage)하고, 샘플링과 보존 기간을 조정한다.

실무 답변 방향: "ERROR/WARN은 90일 보존, INFO는 14일, DEBUG는 프로덕션에서 기본 비활성화하되 필요 시 동적으로 특정 서비스만 활성화하는 구조로 바꿨습니다. 또한 고빈도 INFO 로그(예: 헬스체크)는 샘플링해서 일부만 저장했습니다. 이렇게 해서 로그 비용을 유지하면서도 장애 대응에 필요한 로그는 그대로 남겼습니다."
Q. "구조화 로그로 전환할 때 기존 팀의 저항이 있었을 것 같은데, 어떻게 설득했나요?"
핵심 논리: 강제보다 자동화로 마찰을 줄인다. 개발자가 직접 필드 이름을 신경 쓰지 않아도 공통 라이브러리가 처리하게 만들면 저항이 줄어든다.

실무 답변 방향: "처음엔 '로그 포맷 바꾸라'는 지시로 접근했다가 반발이 컸습니다. 이후 공통 로깅 라이브러리를 만들어서 개발자는 logger.info('event_name', {필드}) 형태로만 쓰면 나머지(trace_id 주입, JSON 직렬화)는 라이브러리가 알아서 처리하도록 바꿨습니다. 실제 장애에서 검색이 빨라진 사례를 공유하니 자발적으로 전환이 이루어졌습니다."
Q. "개인정보가 포함된 로그를 실수로 남긴 적이 있나요? 어떻게 방지하시나요?"
핵심 논리: 사람의 주의력에 의존하는 것은 확장 가능한 해결책이 아니다. 코드 레벨의 자동 마스킹과 로깅 라이브러리 차원의 필드 화이트리스트가 필요하다.

실무 답변 방향: "공통 로깅 라이브러리에 민감 필드 이름 패턴(password, card_number, ssn 등)을 자동 마스킹하는 필터를 넣었습니다. 그리고 로그로 남길 수 있는 필드를 화이트리스트로 관리해서, 새 필드를 로그에 추가하려면 리뷰를 거치도록 했습니다. 실수로 한 번 카드 번호 앞자리가 로그에 남았던 사고 이후 도입한 조치입니다."
Q. "로그와 트레이싱(Distributed Tracing)의 역할이 겹치는 것 같은데, 둘 다 필요한 이유가 뭔가요?"
핵심 논리: 로그는 "특정 시점에 무슨 일이 있었는지"에 강하고(풍부한 컨텍스트), 트레이싱은 "요청이 어디서 시간을 썼는지"에 강하다(정확한 시간/구조). 둘은 같은 trace_id로 연결되어야 진가를 발휘한다.

실무 답변 방향: "트레이스는 서비스 간 호출 구조와 소요 시간을 시각적으로 보여주지만, 왜 실패했는지 구체적인 이유(에러 메시지, 파라미터 값)는 로그에 있습니다. 저희는 트레이스의 span_id를 로그 필드에도 포함시켜서, 느린 span을 발견하면 바로 그 구간의 로그로 드릴다운할 수 있게 연결했습니다."
⚠️

흔한 오해와 Trade-off

오해 1: "구조화 로그는 사람이 읽기 불편하니 개발 환경에서는 안 써도 된다"

현실: 개발 환경과 프로덕션 환경의 로그 포맷이 다르면, 로컬에서 테스트한 로깅 코드가 프로덕션에서 다르게 동작하는 것을 놓치기 쉽다.

올바른 접근: 로그는 항상 구조화(JSON)로 남기고, 로컬 개발 시에는 CLI 도구(예: pino-pretty, jq)로 JSON을 보기 좋게 포맷팅해서 출력한다. 데이터 자체는 동일하게 유지한다.

오해 2: "trace_id만 있으면 로그의 컨텍스트 필드는 최소화해도 된다"

현실: trace_id는 "어떤 요청인지"를 식별할 뿐, "왜 실패했는지"는 알려주지 않는다. trace_id로 요청을 찾은 뒤에도 결국 각 로그 줄에 있는 구체적 필드(에러 코드, 파라미터 값)를 봐야 원인을 알 수 있다.

올바른 접근: trace_id는 "묶어주는" 역할, 나머지 필드는 "설명하는" 역할이다. 둘 다 필요하다.

오해 3: "로그 레벨은 개발자 재량에 맡겨도 된다"

현실: 팀마다 기준이 다르면 어떤 서비스는 모든 것이 ERROR로 시끄럽고, 어떤 서비스는 진짜 장애도 INFO로 조용히 묻힌다. 온콜 알림과 연동된 레벨(보통 ERROR 이상)의 기준이 팀마다 다르면 알림 신뢰도가 무너진다.

올바른 접근: 레벨별 기준을 팀 문서로 명문화하고, 코드 리뷰에서 레벨 오남용을 체크한다.

Trade-off: 로그 상세도 vs 저장/검색 비용

모든 요청, 모든 필드를 다 로깅하면 이상적이지만, 트래픽이 커지면 로그 저장소 비용이 인프라 비용을 초과하는 경우도 생긴다. 특히 고빈도 엔드포인트(헬스체크, 폴링 API)의 로그는 대부분 가치가 낮다.

핵심 Trade-off: "모든 것을 기록하는 안전함 vs 필요한 것만 남기는 비용 효율". 실무에서는 레벨/샘플링/보존 기간을 조합해 이 균형을 맞춘다. 중요한 것은 ERROR와 트랜잭션 경계 로그는 절대 샘플링하지 않고, 고빈도 저가치 INFO만 샘플링 대상으로 삼는 것이다.

자기 점검 Rubric

이 답변들을 자신있게 말할 수 있으면 Day 2를 완료한 것이다.

구조화 로그와 비구조화 로그의 차이와, 구조화가 필요한 이유를 설명할 수 있다 필수

"기계가 검색/집계할 수 있어야 한다"는 핵심 이유가 포함되어야 한다.

Correlation ID(trace_id)의 생성부터 전파까지 전체 흐름을 설명할 수 있다 필수

진입점 생성 → 로그 포함 → 헤더 전파 → 다음 서비스에서 재사용, 이 흐름을 순서대로 말할 수 있어야 한다.

로그 레벨(DEBUG~FATAL)을 상황별로 구분해서 설명할 수 있다 필수

재시도 후 정상 복구는 WARN, 최종 실패는 ERROR처럼 실무 기준 예시가 있어야 한다.

로그에 반드시 포함해야 할 공통 컨텍스트 필드를 나열할 수 있다 강함

trace_id, service, user_id, event, timestamp 등. 왜 각 필드가 필요한지도 설명할 수 있어야 한다.

비동기(메시지 큐) 구간에서도 추적이 끊기지 않게 하는 방법을 설명할 수 있다 강함

메시지 헤더/메타데이터에 trace_id를 실어 보내는 방식을 언급할 수 있어야 한다.

민감 정보 마스킹과 로그 비용 관리(샘플링, 보존 기간)를 설명할 수 있다 보너스

보안/컴플라이언스 인식과 비용 트레이드오프까지 있으면 시니어급 답변이 된다.

공통 로깅 라이브러리로 표준화를 이끈 경험을 말할 수 있다 보너스

강제가 아니라 자동화로 팀 저항을 줄인 리더십 경험이면 더 좋다.

실무 연결 — 이 개념이 실제 어디에 쓰이는가

스타트업 (소규모 팀)

단일 서비스 또는 소수 서비스라도 구조화 로그(JSON)는 처음부터 적용하는 것이 좋다. 나중에 서비스가 늘어난 뒤 바꾸려면 비용이 훨씬 크다. trace_id는 API Gateway나 로드밸런서 레벨에서 헤더로 생성해주는 경우가 많아 도입 비용이 낮다.

성장기 회사 (수백만 사용자)

여러 팀이 서로 다른 언어/프레임워크로 서비스를 만들기 시작하면서 로그 포맷이 갈라지기 쉬운 시기다. 이때 공통 로깅 라이브러리 또는 사이드카(예: Fluent Bit) 표준화가 필요해진다. Correlation ID 전파가 인프라 레벨(서비스 메시)에서 자동화되기 시작한다.

대형 플랫폼 (수천만 사용자)

로그 볼륨 자체가 관측성 파이프라인의 병목이 된다. 카디널리티(고유 필드 값의 개수) 관리, 로그 계층화 저장(hot/warm/cold), 실시간 스트리밍 로그 처리(Kafka 기반 로그 파이프라인)가 전담 팀의 과제가 된다.
