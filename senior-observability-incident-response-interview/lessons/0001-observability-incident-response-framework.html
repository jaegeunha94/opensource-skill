왜 인터뷰에서 묻는가

이 질문은 표면적으로는 "개념을 아는지" 묻는 것처럼 보이지만, 실제로 면접관이 평가하는 것은 다음 세 가지다:

1. 장애 경험과 대응 능력

10년차 개발자라면 수십 번 이상의 프로덕션 장애를 겪었을 것이다. 면접관은 "관측성이 뭔지 정의를 외웠는가"가 아니라, "실제 장애 상황에서 무엇을 보고, 어떻게 판단했는가"를 알고 싶다.

2. 시스템 사고 (Systems Thinking)

Observability를 설명할 때 단순히 "로그, 메트릭, 트레이스"라고 나열하는 개발자와, "이 세 가지를 통해 시스템 내부 상태를 외부 신호만으로 추론할 수 있다"고 설명하는 개발자의 차이는 크다. 후자는 설계 판단력이 있다고 평가된다.

3. 재발 방지 설계 능력

장애 하나를 처리하고 끝내는 주니어, 장애 후 시스템 전체의 관측성 수준을 높이는 시니어 — 이 차이를 이 질문에서 드러낼 수 있다.

인터뷰어가 듣고 싶어하는 신호
"모니터링은 알고 있는 장애를 탐지하는 것이고, 관측성은 알지 못했던 장애까지 질문할 수 있는 능력입니다. 저는 새 서비스를 배포할 때 '내가 어떤 질문을 나중에 해야 할지 모른다'는 가정에서 출발합니다."

전제 개념 — 쉬운 언어로 먼저 이해하기

모니터링(Monitoring)이란?

모니터링은 미리 알고 있는 문제를 감시하는 것이다. 예를 들어:

  • CPU 사용률이 80%를 넘으면 알림을 보낸다
  • 응답 시간이 2초를 넘으면 알림을 보낸다
  • 에러 응답 비율이 5%를 넘으면 알림을 보낸다

이것들은 전부 미리 정한 임계값(threshold)과 비교하는 것이다. 즉, "이런 문제가 생길 수 있다"고 이미 예상한 것들만 감시한다.

모니터링의 한계
2019년에 한 유명 결제 서비스가 장애를 겪었는데, CPU, 메모리, 응답시간 모두 정상이었다. 하지만 특정 카드사로 향하는 결제 요청만 내부적으로 타임아웃이 발생하고 있었다. 기존 모니터링으로는 탐지 불가능한 장애였다.

관측성(Observability)이란?

관측성은 제어 이론(Control Theory)에서 온 개념이다. 원래 의미는 "시스템의 외부 출력만 보고 내부 상태를 추론할 수 있는 능력"이다.

소프트웨어에 적용하면: 미리 예상하지 못한 문제가 생겼을 때, 새로운 질문을 던지고 답을 찾을 수 있는 능력이다.

관측성이 높은 시스템의 특징:

  • 어떤 사용자가, 어떤 API를, 언제 호출했을 때, 어떤 서비스를 거쳐서, 어디서 실패했는지 추적 가능
  • 같은 에러라도 컨텍스트에 따라 왜 다르게 발생하는지 비교 가능
  • 장애 후에 "그때 어떤 상태였는지" 재구성 가능
핵심 차이 한 문장
모니터링은 알고 있는 것을 감시한다. 관측성은 모르는 것을 발견할 수 있게 한다.

Logs

이벤트 기록. "무엇이 일어났는지" 상세 맥락 제공. 검색 가능.

Metrics
얼마나 나빠지고 있는가

숫자로 된 집계 데이터. 트렌드 파악, 알림 기준.

Traces

요청이 여러 서비스를 거치는 경로와 시간. 병목 위치 파악.

왜 세 가지가 모두 필요한가?

실제 장애 시나리오로 설명하면 이해가 쉽다:

상황어떤 신호가 필요한가왜
"주문 API가 간헐적으로 500 에러 반환" Metrics → Logs 메트릭으로 에러 비율 추이 확인 → 로그로 구체적 에러 메시지와 파라미터 확인
"응답 시간이 평소보다 5배 느림" Traces 어떤 서비스 구간에서 시간이 걸리는지 Trace로 파악. 로그만으로는 분산 시스템 병목 위치 알기 어려움
"특정 사용자만 결제 실패" Logs + Traces 로그에서 해당 user_id를 필터링 → Trace로 요청 경로 추적. 메트릭으로는 사용자별 구분 불가
"언제부터 이상했는지 파악 필요" Metrics 시계열 데이터로 정확히 언제 지표가 변했는지 타임라인 재구성

장애 대응 6단계 프레임워크

면접에서 "장애 발생 시 어떻게 대응하나요?"를 물으면 이 구조로 답변하면 된다. 이 순서는 실제 장애 대응에서 보편적으로 사용되는 구조다.

  1. 탐지 (Detection) 알림 수신 또는 고객 신고. 목표: 장애가 있다는 사실을 빠르게 인지

    핵심: MTTD(Mean Time to Detect)를 줄이는 것이 목표다. 고객 신고로 장애를 처음 알게 된다면 관측성이 부족한 것이다.
  2. 분류 (Triage) 영향 범위와 심각도 파악. 목표: 얼마나 급한가, 무엇이 영향받는가

    핵심 질문: 몇 명의 사용자가 영향받는가? 어떤 기능이 동작하지 않는가? 비즈니스 영향이 있는가?
    이 단계에서 "P1(전사 긴급) vs P3(일부 기능 저하)" 등 우선순위를 정한다.
  3. 격리 (Containment) 피해 확산 방지. 목표: 지금보다 더 나빠지지 않게

    예시: Circuit Breaker 활성화, 트래픽 롤백, 특정 기능 일시 비활성화, 영향받는 서버 격리.
    중요: 원인 분석보다 격리가 먼저다. 원인을 모르더라도 피해 확산을 막는 것이 우선이다.
  4. 원인 분석 (Root Cause Analysis) 왜 발생했는지 파악. 목표: 재발 방지를 위한 근본 원인 규명

    로그, 메트릭, 트레이스를 결합해 타임라인을 재구성한다. 증상(symptom)과 원인(cause)을 구분하는 것이 핵심이다.
    예: "DB 응답 느림"은 증상이고, "인덱스 없는 컬럼에 대량 쿼리"가 원인이다.
  5. 해결 (Resolution) 서비스 정상화. 목표: 사용자에게 서비스를 복구

    임시 fix(hotfix)와 근본 fix를 구분한다. 임시 fix를 적용한 후 서비스를 복구하고, 근본 fix는 별도 작업으로 계획한다.
    MTTR(Mean Time to Recover)을 줄이는 것이 목표다.
  6. 재발 방지 (Prevention) Postmortem 작성 + 개선 액션. 목표: 같은 원인으로 다시 장애 나지 않도록

    비난 없는(blameless) postmortem 작성, 액션 아이템 도출 및 담당자·기한 지정, 후속 작업 추적.
    가장 중요한 질문: "이번 장애를 더 빨리 탐지했어야 하는가? 탐지하지 못한 이유는 무엇인가?"
MTTD / MTTR / MTBF 개념
  • MTTD (Mean Time to Detect): 장애 발생 → 탐지까지 평균 시간. 낮을수록 좋음
  • MTTR (Mean Time to Recover): 탐지 → 복구까지 평균 시간. 낮을수록 좋음
  • MTBF (Mean Time Between Failures): 장애 간 평균 간격. 높을수록 좋음
  • 인터뷰에서 "관측성을 개선해서 MTTD를 X분에서 Y분으로 줄였다"고 수치로 말할 수 있으면 강력한 답변이 된다

SLO / SLI / SLA 기초 — 왜 중요한가

관측성 인터뷰에서 빠지지 않는 주제다. 단순히 정의를 외우는 수준이 아니라, "실제 서비스에서 어떻게 정의하고 운영했는가"를 물어본다.

용어의미예시
SLI
(Service Level Indicator)		 서비스 품질을 측정하는 실제 지표 값
"지금 어떤 상태인가"		 지난 5분간 성공한 요청 비율: 99.7%
SLO
(Service Level Objective)		 SLI가 목표해야 하는 값
"어느 수준을 유지해야 하는가"		 월간 가용성 99.9% 이상 유지
SLA
(Service Level Agreement)		 SLO가 달성되지 않을 때 계약적 결과가 있는 약속
"못 지키면 어떻게 되는가"		 가용성 99.9% 미달 시 크레딧 제공
Error Budget SLO 달성을 위해 허용되는 총 장애 시간 또는 에러 수
"이 정도까지는 장애가 나도 된다"		 99.9% SLO 기준 → 월 43.8분 다운타임 허용

Error Budget을 왜 도입하는가?

Error Budget의 핵심 통찰은 이것이다: "100% 가용성은 목표가 아니다. 새 기능을 배포하려면 장애 위험을 감수해야 한다."

Error Budget이 남아있을 때: 새 기능 배포를 과감하게 한다.
Error Budget이 소진됐을 때: 배포를 멈추고 안정화에 집중한다.

이것은 개발팀과 운영팀 사이의 긴장 관계를 데이터로 해소하는 도구다. "배포를 멈춰라"는 결정이 사람의 감정이 아니라 Error Budget 소진이라는 객관적 지표로 이루어진다.

시니어 답변에서 나와야 하는 내용
"저희 팀은 99.9% SLO를 설정했고, 이는 월 43분의 Error Budget을 의미합니다. 큰 배포 전에는 Error Budget 잔량을 확인하고, 잔량이 50% 이하면 배포를 다음 스프린트로 미뤘습니다. 실제로 한 달에 Error Budget의 70%를 써버린 적이 있었는데, 그 달은 신규 기능 배포를 멈추고 안정화 스프린트를 진행했습니다."

답변 사고 순서 — 면접장에서 어떻게 풀어나가는가

질문 1: "Observability란 무엇이며, Monitoring과 어떻게 다른가요?"

  1. 개념 정의 (15초) "관측성은 시스템 외부 신호만으로 내부 상태를 추론할 수 있는 능력입니다."
  2. 모니터링과의 차이 대비 (15초) "모니터링은 미리 예상한 문제를 감시하는 것이고, 관측성은 예상하지 못한 문제도 질문할 수 있는 능력입니다."
  3. 실무 경험 연결 (30초) "실제 사례를 들면, CPU와 메모리가 모두 정상인데 특정 사용자 그룹만 결제가 실패하는 장애가 있었습니다. 기존 모니터링 알림은 아무것도 울리지 않았고, 구조화 로그의 user_segment 필드를 기반으로 검색했을 때 비로소 패턴을 발견했습니다."
  4. Three Pillars 연결 (20초) "이를 가능하게 하는 것이 로그·메트릭·트레이스 세 가지입니다. 각각은 다른 질문에 답합니다."
  5. 마무리 — 설계 관점으로 끌어올리기 (10초) "그래서 저는 새 서비스를 설계할 때 '나중에 어떤 질문을 해야 할지 모른다'는 가정에서 관측성을 먼저 설계합니다."

질문 2: "장애 발생 시 어떤 순서로 대응하나요?"

  1. 구조 먼저 제시 (5초) "크게 6단계로 나눌 수 있습니다: 탐지 → 분류 → 격리 → 원인 분석 → 해결 → 재발 방지입니다."
  2. 핵심 포인트 강조 (20초) "특히 격리를 원인 분석 앞에 두는 것이 중요합니다. 원인을 모르더라도 피해 확산을 막는 것이 우선이기 때문입니다."
  3. 실제 경험으로 구체화 (60초) 본인 경험에서 각 단계를 어떻게 했는지 한 줄씩 말한다.
  4. 재발 방지로 마무리 (20초) "그리고 장애 종료 후 48시간 이내에 postmortem을 작성하고, 액션 아이템을 다음 스프린트에 반영합니다."

핵심 답변 예시

질문 1 모범 답변: "Observability vs Monitoring"

강한 답변

"모니터링은 미리 정해둔 지표와 임계값을 기반으로 알려진 문제를 탐지합니다. 예를 들어 'CPU 80% 초과', '에러율 5% 초과' 같은 룰이죠. 반면 관측성은 예상하지 못한 문제가 발생했을 때 새로운 질문을 던지고 답을 찾을 수 있는 시스템의 능력입니다.

실제로 경험한 사례로는, 특정 지역 사용자만 결제가 느려지는 장애가 있었는데 CPU, 에러율, 응답시간 모두 정상 범위였습니다. 기존 모니터링으로는 탐지가 불가능했지만, 요청마다 region 필드를 구조화 로그에 포함시켜 두었기 때문에 10분 내에 원인 지점을 특정할 수 있었습니다.

이런 경험 때문에 저는 새 서비스를 만들 때 '나중에 어떤 질문을 해야 할지 지금은 모른다'는 전제에서 출발합니다. 모든 요청에 trace_id, user_id, region 같은 컨텍스트를 포함하고, 구조화 로그로 나중에 어떤 각도에서든 검색 가능하게 설계합니다."

질문 2 모범 답변: "장애 대응 순서"

강한 답변

"저는 탐지 → 분류 → 격리 → 원인 분석 → 해결 → 재발 방지 6단계로 접근합니다.

실제 사례로, 결제 서비스에서 P1 알림이 울렸을 때를 말씀드리겠습니다. 먼저 메트릭 대시보드에서 영향 범위를 확인했습니다 — 전체 결제의 약 12%가 실패 중이었고, 특정 카드사 연동 쪽만 에러가 집중됐습니다. 이것이 분류 단계입니다.

원인을 아직 모르는 상태에서, 먼저 해당 카드사로 향하는 요청에 Circuit Breaker를 수동으로 활성화해 나머지 88%에 영향이 가지 않도록 격리했습니다. 그다음 로그와 트레이스를 분석해 카드사 외부 API가 타임아웃을 반환하고 있다는 것을 발견했습니다.

임시 조치로 타임아웃 값을 낮춰서 빠른 실패 처리를 했고, 근본 해결책으로 카드사 API 타임아웃 모니터링과 외부 API 연동 자체에 대한 SLO를 별도로 정의하는 작업을 postmortem 액션 아이템으로 만들었습니다."

⚖️

좋은 답변 vs 약한 답변의 차이

강한 답변
"저희는 모든 API 요청에 X-Trace-ID를 전파하고, 로그에 user_id, session_id, region을 포함합니다. 장애 시 Kibana에서 trace_id로 모든 서비스 로그를 한 번에 검색할 수 있어서 MTTD를 기존 30분에서 5분으로 줄였습니다."

구체적인 기술 결정과 그 결과(수치)가 있다. 왜 그 결정을 했는지 맥락이 담겨있다.

약한 답변
"Prometheus로 메트릭을 수집하고, Grafana로 대시보드를 만들었습니다. 로그는 ELK 스택을 사용합니다."

도구 목록 나열. 왜 그 도구를 선택했는지, 실제로 어떤 장애를 어떻게 해결했는지가 없다. 주니어도 할 수 있는 답변이다.

강한 답변 — 장애 대응
"가장 먼저 한 것은 원인 분석이 아니라 격리였습니다. 원인을 모르는 상태에서 영향 범위가 계속 커지고 있었기 때문입니다. 격리 후에 로그를 분석했습니다."

우선순위 판단의 이유가 있다. 원인 분석보다 격리가 먼저라는 실무 인사이트가 담겨있다.

약한 답변 — 장애 대응
"로그를 확인하고 원인을 파악한 다음 수정했습니다. 그다음 재발 방지를 위해 모니터링을 추가했습니다."

순서가 추상적이고, 실제 장애 경험이 없는 사람도 할 수 있는 답변이다. 구체적인 판단 근거가 없다.

강한 답변 — SLO 관련
"SLO를 99.9%로 설정하면 Error Budget이 월 43분입니다. 이 예산을 다 써버린 달은 신규 배포를 멈추는 것이 저희 팀의 약속이었습니다. 배포 빈도가 리스크와 연결되어 있다는 것을 팀원 모두가 공유하는 언어가 생긴 것이죠."

Error Budget을 숫자로 이해하고, 팀 운영과 연결했다.

약한 답변 — SLO 관련
"SLO는 Service Level Objective의 약자로 목표 가용성입니다. SLA와 다르게 법적 구속력이 없습니다."

정의만 말했다. 실무에서 SLO를 어떻게 활용했는지, Error Budget 개념은 없다.

Senior-level Follow-up 질문과 답변 방향

Q. "Observability 구현 비용이 상당한데, 어느 수준까지 해야 하는지 어떻게 결정하나요?"
핵심 논리: 관측성의 가치는 장애 탐지 시간 단축 + 복구 시간 단축 = 비즈니스 손실 감소다. "관측성 도구 월 비용 vs 장애 MTTD 1시간 단축이 가져다주는 비용 절감"을 비교한다.

실무 답변 방향: "저는 최소한 세 가지는 반드시 해야 한다고 봅니다. 1) 모든 요청에 trace_id 전파, 2) 에러 로그에 충분한 컨텍스트, 3) Golden Signal 기반 알림. 이 이상은 서비스 중요도와 장애 비용을 기준으로 투자를 결정합니다."
Q. "알림이 너무 많이 와서 팀이 알림을 무시하게 되는 Alert Fatigue 문제를 어떻게 해결하셨나요?"
핵심 논리: Alert Fatigue는 actionable하지 않은 알림이 많을 때 발생한다. 해결책은 "모든 알림은 행동을 요구해야 한다"는 원칙이다.

실무 답변 방향: 알림을 Paging(즉시 대응 필요)과 Ticketing(다음 근무일 처리)으로 분리. Paging 알림 기준을 SLO 소진율 기반으로 재정의. 알림 노이즈 분석 주기적 실시 (지난 2주 알림 중 실제 행동이 없었던 알림 목록 검토 후 삭제 또는 수준 낮추기).
Q. "마이크로서비스 환경에서 장애 원인 추적이 모놀리스보다 어렵습니다. 어떻게 접근하셨나요?"
핵심 논리: 마이크로서비스에서는 단일 요청이 10개 이상 서비스를 거친다. 로그만으로는 연결이 안 된다. 분산 트레이싱(Distributed Tracing)이 핵심 해결책이다.

실무 답변 방향: "trace_id를 모든 서비스 호출에 HTTP 헤더(X-Trace-ID)로 전파하도록 인프라 레이어에서 표준화했습니다. 이렇게 하면 trace_id 하나로 모든 서비스의 로그와 스팬을 연결할 수 있습니다. 이것이 없으면 10개 서비스 로그를 타임스탬프 기준으로 수동으로 맞춰야 하는데, 실제로 30분 짜리 장애 분석이 3시간이 걸리는 경험을 한 후 도입했습니다."
Q. "Postmortem을 작성할 때 팀원들이 방어적이 되는 문제가 있었나요? 어떻게 문화를 바꿨나요?"
핵심 논리: Postmortem의 목적은 개인 책임 추궁이 아니라 시스템 개선이다. "Blameless Postmortem" 문화가 핵심이다.

실무 답변 방향: "Postmortem 작성 원칙을 '사람이 아니라 시스템에서 실패 지점을 찾는다'로 명시했습니다. 또한 Postmortem 결과를 인사 평가와 완전히 분리했습니다. 장애를 솔직하게 공유한 팀원을 인정하는 문화를 만들고, 'Incident of the Month' 공유 세션을 운영해 배운 점을 조직 전체가 공유하도록 했습니다."
Q. "SLO를 처음 도입할 때 어떤 숫자로 시작했나요? 99.99%로 해야 하는 것 아닌가요?"
핵심 논리: SLO는 고객이 실제로 요구하는 수준과 현실적으로 달성 가능한 수준 사이에서 정의해야 한다. 너무 높은 SLO는 Error Budget이 거의 없어 개발 속도를 극도로 늦춘다.

실무 답변 방향: "내부 B2B 도구는 99.5% (월 3.6시간), 핵심 결제 서비스는 99.95% (월 21분)로 다르게 설정했습니다. 99.99%는 물리적으로 달성하기 위한 아키텍처 비용이 매우 크고, Error Budget이 월 4분밖에 안 돼 배포 한 번 실수하면 남은 달 전체를 동결해야 합니다. 현실적인 숫자로 시작하고, 달성하면 올리는 방향이 낫습니다."
⚠️

흔한 오해와 Trade-off

오해 1: "로그를 많이 남길수록 좋다"

현실: 로그가 너무 많으면 저장 비용이 폭증하고, 정작 필요한 로그를 찾기 어려워진다. 특히 DEBUG 레벨 로그를 프로덕션에서 항상 켜두는 것은 비용 대비 효과가 낮다.

올바른 접근: 로그 레벨을 환경에 맞게 설정한다. 프로덕션에서는 INFO 이상. 장애 발생 시에만 DEBUG를 동적으로 활성화하는 기능(Dynamic Log Level)을 고려한다.

오해 2: "메트릭이 정상이면 서비스가 정상이다"

현실: 집계 메트릭은 개별 이상 상황을 숨긴다. 전체 응답 시간 평균이 200ms여도, 특정 사용자 그룹의 p99는 5000ms일 수 있다. 평균 에러율이 0.1%여도, 특정 기능은 100% 실패 중일 수 있다.

올바른 접근: 평균보다 백분위수(p95, p99, p999)를 사용하고, 전체 집계와 세그먼트별 지표를 모두 확인한다.

오해 3: "SLA 99.99%를 목표로 해야 한다"

현실: 99.99%는 월 4분의 Error Budget이다. 이 수준을 달성하려면 다운타임 없는 배포, 즉각적인 자동 롤백, 다중 가용성 존 설계가 필수다. 대부분의 서비스에는 과도하다.

올바른 접근: 서비스 성격에 맞는 SLO를 정의한다. 내부 도구는 99.5%, B2C 핵심 서비스는 99.9%, 결제 같은 중요 경로는 99.95% 등 차별화한다.

Trade-off: Observability 비용 vs 장애 탐지 속도

관측성 도구는 비싸다. 대량의 로그를 저장하고 검색하는 비용, 메트릭 수집 인프라, 트레이싱 에이전트 오버헤드. 모든 것을 최고 수준으로 유지하면 월 수백만 원이 나올 수 있다.

핵심 Trade-off: "관측성 비용 vs 장애 발생 시 비즈니스 손실". MTTD가 1시간인 서비스에서 결제 실패 시 분당 손실이 100만원이라면, 탐지를 5분으로 줄이는 관측성 투자 ROI는 쉽게 계산된다.

Three Pillars 각각이 어떤 질문에 답하는지 설명할 수 있다 필수

로그=무슨 일이 일어났나, 메트릭=얼마나 나쁜가, 트레이스=어디가 느린가. 단순 나열이 아니라 각자의 역할 구분이 되어야 한다.

장애 대응 6단계를 순서대로, 각 단계의 이유를 포함해서 설명할 수 있다 필수

"격리가 원인 분석 앞에 오는 이유"를 설명할 수 있어야 한다.

SLO, SLI, SLA, Error Budget을 혼동 없이 설명할 수 있다 필수

정의 외에 Error Budget이 팀 운영(배포 의사결정)에 어떻게 활용되는지 설명할 수 있어야 한다.

MTTD / MTTR 수치를 언급하며 관측성 개선 경험을 말할 수 있다 강함

"X분에서 Y분으로 줄었다"는 식의 수치가 있으면 강력한 답변이 된다.

Alert Fatigue 문제와 해결 방향을 말할 수 있다 강함

Paging vs Ticketing 알림 분류, actionable 알림 원칙을 이해하고 있어야 한다.

Blameless Postmortem 문화를 어떻게 만들었는지 설명할 수 있다 보너스

기술 답변 외에 조직 문화 변화를 이끈 경험까지 있으면 리더십 평가에 좋다.

분산 트레이싱을 도입하거나 설계한 경험을 말할 수 있다 보너스

trace_id 전파 방식, 도입 전후 차이, 어떤 장애를 더 빨리 해결할 수 있었는지 포함하면 좋다.

실무 연결 — 이 개념이 실제 어디에 쓰이는가

스타트업 (소규모 팀)

최소 투자로 최대 관측성을 얻어야 한다. 우선순위: 구조화 로그(JSON) + 에러 트래킹(Sentry) + 응답시간 p95 알림. 이 세 가지만 있어도 대부분의 장애는 30분 내에 원인을 찾을 수 있다.

성장기 회사 (수백만 사용자)

분산 트레이싱이 필수가 된다. 마이크로서비스가 늘어남에 따라 로그만으로는 요청 경로 추적이 불가능해진다. SLO를 도입하고 Error Budget 기반 배포 의사결정이 시작된다.

대형 플랫폼 (수천만 사용자)

관측성 자체가 하나의 팀이 된다. 로그 카디널리티 관리, 샘플링 전략(모든 트레이스를 저장하면 비용이 폭발), 관측성 파이프라인 안정성이 핵심 과제가 된다. SRE 팀이 별도로 존재하며 Error Budget 리뷰를 정기적으로 진행한다.
