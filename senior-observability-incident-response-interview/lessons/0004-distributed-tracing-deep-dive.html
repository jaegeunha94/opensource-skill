왜 인터뷰에서 묻는가

Day 3에서 다룬 메트릭은 "무엇이 이상한가"를 초 단위로 빠르게 보여준다. 하지만 마이크로서비스 10개를 거쳐가는 요청 하나가 왜 느린지는 메트릭만으로 알 수 없다. 이 지점에서 분산 트레이싱이 필요해지고, 인터뷰어는 이 주제로 지원자가 분산 시스템을 하나의 요청 흐름 단위로 사고할 수 있는지를 검증한다.

1. 단일 서버 사고에서 벗어났는가

모놀리식 환경에서는 로그 하나만 보면 요청의 전체 흐름을 알 수 있다. 하지만 서비스가 여러 개로 나뉘면 요청 하나가 5~10개 서비스를 거치면서 로그도 그만큼 분산된다. 시니어는 "이 요청이 어떤 순서로 어떤 서비스를 거쳤는가"를 로그 대신 트레이스라는 별도의 구조로 재구성해야 한다는 것을 안다.

2. 지연의 위치를 정량적으로 특정할 수 있는가

"느린 것 같다"가 아니라 "이 요청 중 320ms는 결제 서비스, 40ms는 네트워크, 15ms는 DB 쿼리에서 소요됐다"처럼 구간별로 시간을 쪼개서 설명할 수 있는지가 핵심이다. 이는 트레이스의 span 구조를 실제로 읽고 해석해본 경험이 있어야 나오는 답변이다.

3. 관측성의 비용을 이해하는가

모든 요청을 100% 기록하면 데이터 양이 폭발적으로 늘어나 저장/처리 비용이 감당하기 어려워진다. 시니어는 "왜 다 안 모으는가", "그러면 중요한 트레이스를 놓치지 않는가"에 대한 답, 즉 Sampling 설계를 가지고 있어야 한다.

인터뷰어가 듣고 싶어하는 신호
"저는 메트릭으로 '어떤 서비스가 이상한가'를 먼저 좁히고, 그 서비스와 관련된 실제 느린 요청의 트레이스를 열어서 어느 span이 전체 시간의 대부분을 차지하는지 봅니다. 그리고 그 span의 태그(attribute)를 보고 어떤 쿼리, 어떤 외부 호출이 원인인지까지 좁혀갑니다."

전제 개념 — 쉬운 언어로 먼저 이해하기

분산 트레이싱(Distributed Tracing)이란?

하나의 요청이 여러 서비스를 거쳐 처리될 때, 그 요청 전체의 여정을 하나로 이어 붙여서 보여주는 기록이다. 로그는 서비스마다 따로 남고, 메트릭은 숫자로 집계되지만, 트레이스는 "요청 A가 API 게이트웨이 → 주문 서비스 → 재고 서비스 → DB" 순서로, 각 구간에 몇 ms가 걸렸는지까지 시간 순서대로 재구성한다.

비유하자면, 택배를 보낼 때 로그는 각 물류센터에서 찍는 개별 스캔 기록이고, 트레이스는 "발송 → 허브1(2시간) → 허브2(5시간) → 배송(1시간)"처럼 전체 배송 과정을 하나의 타임라인으로 이어 붙인 운송장 조회 화면과 같다.

Trace, Span, Trace ID, Span ID란?

트레이싱을 이해하려면 이 네 가지 용어부터 정확히 구분해야 한다.

용어의미
Trace 요청 하나의 전체 여정. 여러 Span의 모음
Span 그 여정 안의 개별 작업 단위 (예: "주문 서비스가 재고 서비스를 호출한 구간")
Trace ID 요청 하나 전체에 부여되는 고유 식별자. 모든 Span이 이 ID를 공유
Span ID 개별 Span마다 부여되는 고유 식별자

Day 2에서 다룬 correlation ID와 개념적으로 같은 역할이다. 다만 correlation ID는 로그를 엮기 위한 단순 문자열이었다면, Trace ID/Span ID는 부모-자식 관계와 시간 정보까지 포함한 구조화된 트리를 만든다는 점이 다르다.

API Gateway
380ms
└ 주문 서비스
340ms
  └ 재고 서비스 호출
70ms
  └ 결제 서비스 호출
220ms ⚠
    └ 카드사 PG 호출
200ms

이런 형태를 waterfall(워터폴) 뷰라고 부른다. 전체 380ms 중 결제 서비스 span이 220ms를 차지하고, 그 안에서도 카드사 PG 호출이 200ms를 차지한다는 것이 한눈에 보인다. 이 요청의 병목은 "우리 서비스"가 아니라 "외부 카드사 응답 지연"이라는 것을 트레이스 하나로 특정할 수 있다.

Span에는 무엇이 담기는가?

  • 이름: 이 작업이 무엇인지 (예: POST /orders, SELECT inventory)
  • 시작 시각 / 종료 시각(duration): 이 작업이 얼마나 걸렸는지
  • 부모 Span ID: 어떤 작업의 하위 작업인지 (트리 구조를 만든다)
  • 속성(Attribute/Tag): http.status_code, db.statement, user.tier처럼 이 작업의 맥락 정보
  • 이벤트(Event)/로그: Span 실행 중 발생한 특정 시점의 사건 (예: 재시도 발생, 캐시 미스)
  • 상태(Status): 성공/실패 여부와 에러 정보

Context Propagation(컨텍스트 전파)이란?

서비스 A가 서비스 B를 HTTP나 메시지 큐로 호출할 때, "이 요청은 Trace ID XYZ의 일부이고 부모 Span은 이것이다"라는 정보를 요청과 함께 실어 보내는 것을 말한다. 이게 없으면 서비스 B는 자신이 어떤 트레이스에 속해 있는지 전혀 알 수 없고, 트레이스가 서비스 경계에서 끊겨버린다.

실무에서는 HTTP 헤더에 Trace 정보를 담아 전파한다. 업계 표준은 W3C Trace Context 스펙의 traceparent 헤더다.

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │  └─ trace-id (32자리 hex)         └─ parent-span-id (16자리 hex)
             └─ version
흔한 실수 — 컨텍스트 전파가 끊기는 지점
메시지 큐(Kafka, SQS)를 통한 비동기 호출, 배치 작업, 새로 만든 스레드나 코루틴으로 작업을 넘기는 지점에서는 컨텍스트가 자동으로 전파되지 않는 경우가 많다. 이런 경계에서 개발자가 명시적으로 trace context를 메시지 속성에 실어 보내지 않으면, 트레이스가 거기서 끊기고 "일부만 보이는 반쪽 트레이스"가 된다. 실무에서 트레이스가 갑자기 끊기는 대부분의 원인이 여기에 있다.

OpenTelemetry(OTel)란?

트레이스, 메트릭, 로그를 수집하는 방식을 표준화한 벤더 중립적인 오픈소스 프로젝트다. 예전에는 Jaeger, Zipkin, DataDog 등이 각자 다른 방식으로 계측(instrumentation) 라이브러리를 요구했지만, OpenTelemetry는 "계측은 한 번만 하고, 어디로 보낼지(백엔드)는 설정만 바꾸면 되도록" 만들었다. 인터뷰에서는 이 표준화가 왜 중요한지(벤더 락인 방지, 계측 코드의 재사용성)를 설명할 수 있으면 좋은 신호다.

Span 설계와 병목 분석 원칙

1. Span의 경계는 "의미 있는 작업 단위"로 나눈다

모든 함수 호출마다 span을 만들면 트레이스가 너무 세밀해져서 오히려 읽기 어려워지고 오버헤드도 커진다. 반대로 너무 큰 단위(예: 전체 요청 하나만 span 하나)로 만들면 어디가 느린지 알 수 없다. 실무 기준은 "네트워크를 넘는 경계"(서비스 호출, DB 쿼리, 외부 API 호출, 큐 발행/소비)마다 span을 만드는 것이다.

2. 병목 분석은 "가장 긴 span"이 아니라 "critical path"를 본다

병렬로 실행되는 span이 있으면, 단순히 가장 오래 걸린 span이 전체 지연의 원인이 아닐 수 있다. 예를 들어 두 개의 외부 호출이 병렬로 실행됐다면, 전체 시간은 그중 더 오래 걸린 쪽에 의해 결정된다(critical path). 반대로 순차로 실행된 짧은 span들이 쌓여서 전체 시간을 늘리는 경우도 있다. 트레이스를 읽을 때는 병렬/순차 구조를 먼저 파악해야 진짜 병목을 찾을 수 있다.

3. 트레이스와 메트릭/로그를 연결한다 (Exemplar)

메트릭에서 "p99 지연이 튀었다"는 것을 확인한 뒤, 그 시점의 실제 느렸던 요청 하나의 트레이스로 바로 이동할 수 있어야 진단이 빨라진다. 이 연결 고리를 Exemplar라고 부른다. 즉, 히스토그램 메트릭의 특정 버킷에 "이 버킷에 해당하는 실제 trace ID는 이것이다"라는 정보를 붙여두는 것이다. 메트릭(탐지) → 트레이스(진단) → 로그(세부 확인)로 이어지는 흐름이 관측성 3요소를 실제로 엮어서 쓰는 방식이다.

4. Span에는 저-카디널리티/고-카디널리티 정보를 구분해서 담는다

Day 3에서 메트릭 라벨은 카디널리티가 낮아야 한다고 배웠다. 트레이스는 반대로 고-카디널리티 정보(user_id, request_id, 특정 주문 번호 등)를 담아도 되는 곳이다. 트레이스는 요청 단위로 개별 저장되기 때문에, 이런 상세 정보를 span attribute에 담아두면 나중에 "이 특정 사용자의 이 요청이 왜 느렸는가"를 정확히 추적할 수 있다. 메트릭과 트레이스의 역할 분담을 이렇게 이해하는 것이 시니어 답변의 핵심이다.

5. Sampling은 "무엇을 버릴지"를 설계하는 문제다

트래픽이 많은 서비스에서 모든 요청을 100% 트레이싱하면 저장 비용과 처리 오버헤드가 감당하기 어려워진다. 그래서 일부만 샘플링해서 저장하는데, 무엇을 남기고 무엇을 버릴지가 설계의 핵심이다.

전략방식장점단점
Head-based Sampling 요청이 시작되는 시점에 미리 정한 확률로 샘플링 여부를 결정 (예: 1%) 구현이 단순하고 오버헤드가 낮음 에러/느린 요청도 99% 확률로 놓칠 수 있음
Tail-based Sampling 요청이 끝난 뒤, 결과(에러 여부, 지연 시간)를 보고 샘플링 여부를 결정 에러/느린 요청을 놓치지 않고 우선 보존 가능 모든 span을 일단 버퍼링해야 해서 구현/인프라 복잡도가 높음
Priority/Rate-based Sampling 서비스별, 엔드포인트별로 샘플링 비율을 다르게 설정 중요 경로는 높은 비율로, 노이즈성 트래픽(헬스체크 등)은 낮은 비율로 세밀하게 제어 설정 관리가 늘어남
시니어 답변에서 나와야 하는 내용
"저희는 head-based sampling으로 기본 1%를 수집하되, 에러가 발생했거나 지연 시간이 특정 threshold를 넘은 요청은 tail-based sampling으로 100% 보존하도록 이중 구조를 썼습니다. 덕분에 저장 비용은 낮추면서도 정작 봐야 할 문제 요청은 놓치지 않을 수 있었습니다."

답변 사고 순서 — 면접장에서 어떻게 풀어나가는가

질문 1: "Trace ID가 서비스 간에 어떻게 전파되나요?"

  1. Trace/Span 정의부터 (10초) "요청 하나 전체를 Trace, 그 안의 개별 작업 단위를 Span이라고 부릅니다. 모든 Span은 같은 Trace ID를 공유하고, 서로 부모-자식 관계를 가집니다."
  2. 전파 메커니즘 설명 (15초) "서비스 A가 서비스 B를 호출할 때, 현재 Trace ID와 자신의 Span ID를 HTTP 헤더(예: W3C 표준의 traceparent 헤더)에 실어서 보냅니다. 서비스 B는 이 헤더를 읽어서 자신의 Span을 그 아래 자식으로 이어 붙입니다."
  3. 비동기 경계의 한계 언급 (15초) "다만 메시지 큐나 배치 작업처럼 HTTP 요청-응답이 아닌 비동기 경계에서는 이 전파가 자동으로 되지 않는 경우가 많아서, 메시지 속성에 명시적으로 trace context를 실어 보내야 합니다."
  4. 실무 경험 연결 (20초) 본인이 트레이스가 특정 지점(큐, 새 스레드)에서 끊겼던 경험과, 이를 어떻게 고쳤는지 구체적으로 말한다.

질문 2: "느린 요청의 병목을 트레이스로 어떻게 찾나요?"

  1. 진입점 설명 (10초) "먼저 메트릭에서 어떤 서비스, 어떤 엔드포인트의 p99가 튀었는지 좁히고, 그 시점의 실제 느린 요청 트레이스를 엽니다."
  2. waterfall 분석 (15초) "waterfall 뷰에서 전체 시간 중 어느 span이 가장 큰 비중을 차지하는지 보고, 병렬로 실행된 span인지 순차로 실행된 span인지부터 구분합니다."
  3. Span 속성으로 원인 좁히기 (15초) "가장 큰 비중을 차지하는 span의 attribute(예: db.statement, http.url)를 보고 구체적으로 어떤 쿼리, 어떤 외부 호출이 원인인지 특정합니다."
  4. 실무 경험 연결 (20초) 본인이 트레이스로 예상 밖의 병목(예: 캐시 미스로 인한 반복 쿼리, 외부 PG사 지연)을 찾아낸 구체적 사례를 말한다.

질문 1 모범 답변: "Trace ID 전파"

강한 답변

"Trace ID는 요청이 처음 시스템에 들어올 때(보통 API 게이트웨이) 생성되고, 이후 모든 서비스 호출에 HTTP 헤더로 실려서 전파됩니다. 저희는 W3C Trace Context 표준의 traceparent 헤더를 썼는데, 여기에 trace-id와 parent-span-id가 함께 담깁니다. 각 서비스는 이 헤더를 읽어서 자신의 새 span을 만들고, 부모 span ID를 이전 값으로 연결한 뒤, 다음 서비스를 호출할 때는 자신의 span ID를 새 parent로 갱신해서 전달합니다.

실제로 저희는 Kafka를 통한 비동기 처리 구간에서 트레이스가 끊기는 문제를 겪었는데, 메시지 헤더에 trace context를 담지 않고 있었던 것이 원인이었습니다. Producer에서 메시지 헤더에 traceparent를 명시적으로 넣고, Consumer에서 이를 읽어 컨텍스트를 복원하도록 고친 뒤로는 큐를 거치는 요청도 하나의 트레이스로 끊김 없이 이어졌습니다."

질문 2 모범 답변: "트레이스로 병목 찾기"

강한 답변

"저는 항상 메트릭에서 먼저 범위를 좁힙니다. 어떤 서비스의 p99 지연이 튀었는지 확인한 다음, 그 시간대의 실제 느린 요청 트레이스를 exemplar 링크로 바로 열어봅니다. waterfall 뷰에서 각 span의 길이를 보고, 특히 병렬 호출인지 순차 호출인지부터 구분합니다. 순차로 쌓인 짧은 span들이 누적된 경우와, 하나의 span이 압도적으로 큰 경우는 원인이 다르기 때문입니다.

실제 사례로, 결제 API가 평소보다 3배 느려진 적이 있었는데 트레이스를 열어보니 재고 서비스 호출 span 하나가 전체의 60%를 차지하고 있었습니다. 그 span의 attribute를 보니 캐시 미스가 발생해서 매 요청마다 DB에 직접 쿼리를 던지고 있었고, 원인은 배포 직후 캐시 워밍업이 안 된 상태였습니다. 로그나 메트릭만으로는 '재고 서비스가 느리다'까지만 알 수 있었을 텐데, 트레이스 덕분에 '캐시 미스로 인한 DB 직접 조회'라는 구체적인 원인까지 5분 만에 특정할 수 있었습니다."

좋은 답변 vs 약한 답변의 차이

강한 답변 — 트레이싱 활용
"메트릭으로 범위를 좁히고, 트레이스의 waterfall에서 critical path를 찾은 뒤, span attribute로 구체적 원인을 특정합니다."

탐지-진단-확인의 흐름과 도구 간 연결 관계가 명확하다.

약한 답변 — 트레이싱 활용
"Jaeger 대시보드에 들어가서 느린 요청을 찾아봅니다."

무엇을 어떤 순서로, 어떤 기준으로 보는지에 대한 사고 과정이 없다. 도구 이름만 언급한다.

강한 답변 — Sampling
"기본은 낮은 비율의 head-based sampling을 쓰되, 에러/느린 요청은 tail-based로 100% 보존하는 이중 구조를 씁니다."

비용과 정확도 사이의 트레이드오프를 이해하고 구체적 설계로 답한다.

약한 답변 — Sampling
"트레이스는 용량이 커서 일부만 저장하도록 설정했습니다."

왜 일부만 저장하는지, 어떤 기준으로 무엇을 남기는지에 대한 설계 근거가 없다.

Senior-level Follow-up 질문과 답변 방향

Q. "서비스 메시(Service Mesh)를 쓰면 트레이싱이 왜 더 쉬워지나요?"
핵심 논리: Istio 같은 서비스 메시는 사이드카 프록시가 모든 서비스 간 통신을 가로채기 때문에, 애플리케이션 코드를 수정하지 않아도 요청/응답 시간, 상태 코드 같은 span 정보를 자동으로 생성할 수 있다. 다만 애플리케이션 내부의 세부 로직(예: 특정 함수 안에서 무엇이 느린지)까지는 메시가 알 수 없어서, 애플리케이션 레벨 계측은 별도로 필요하다.

실무 답변 방향: "서비스 메시는 서비스와 서비스 '사이'의 span은 자동으로 만들어주지만, 서비스 '내부'에서 무엇이 느린지는 여전히 애플리케이션에 OpenTelemetry SDK로 계측해야 합니다. 저희는 인프라 레벨은 메시로, 비즈니스 로직 레벨은 수동 계측으로 이중화했습니다."
Q. "트레이싱 도입 자체가 서비스 성능에 영향을 주지 않나요?"
핵심 논리: 계측 자체(span 생성, context 전파)는 약간의 CPU/메모리 오버헤드가 있고, span을 백엔드로 내보내는 과정(export)이 동기적으로 이뤄지면 지연에 직접 영향을 줄 수 있다.

실무 답변 방향: "저희는 span 데이터를 애플리케이션 스레드에서 바로 백엔드로 보내지 않고, 배치로 모아서 별도 백그라운드 프로세스(OpenTelemetry Collector)가 비동기로 내보내도록 구성해서 요청 경로에 미치는 영향을 최소화했습니다. Sampling 비율도 그 자체가 오버헤드를 줄이는 수단입니다."
Q. "Tail-based sampling은 왜 구현이 복잡한가요?"
핵심 논리: 요청이 끝나기 전까지는 그 요청이 에러가 날지, 느릴지 알 수 없기 때문에, 일단 모든 span을 어딘가에 버퍼링해뒀다가 요청이 끝난 뒤 판단해야 한다. 분산 환경에서는 한 트레이스의 span들이 여러 서비스, 여러 노드에 흩어져 생성되기 때문에, 이걸 한 곳에 모아 완성된 트레이스인지 판단하는 별도의 수집/집계 인프라(Collector)가 필요하다.

실무 답변 방향: "Tail-based sampling을 하려면 모든 span을 우선 일정 시간 버퍼링하는 Collector 계층이 필요하고, 이 계층 자체의 메모리/지연 비용이 추가로 듭니다. 그래서 저희는 트래픽이 아주 큰 서비스에는 head-based를 기본으로 쓰고, 결제처럼 중요도가 높은 경로에만 tail-based를 선택적으로 적용했습니다."
Q. "트레이스와 로그를 어떻게 연결해서 보나요?"
핵심 논리: 트레이스는 '언제, 어느 구간이 느렸는지'를 보여주지만, 그 구간에서 정확히 무슨 값으로 실패했는지 같은 세부 사항은 로그가 더 잘 담는다. Trace ID를 로그 필드에 함께 남기면 트레이스에서 로그로, 로그에서 트레이스로 서로 이동할 수 있다.

실무 답변 방향: "저희는 모든 로그 라인에 trace_id, span_id를 구조화 필드로 남기도록 Day 2에서 다룬 로깅 표준에 포함시켰습니다. 그래서 트레이스에서 이상한 span을 발견하면 그 span_id로 바로 관련 로그를 검색해서 구체적인 에러 메시지나 파라미터 값을 확인할 수 있습니다."
Q. "트레이스만으로는 알 수 없는 것은 무엇인가요?"
핵심 논리: 트레이스는 샘플링된 개별 요청만 보여주기 때문에, "전체 트래픽 중 몇 %가 이런 패턴을 겪고 있는가" 같은 전체 통계적 질문에는 메트릭이 필요하다. 트레이스는 진단(diagnose)에, 메트릭은 탐지(detect)와 규모 파악에 강하다.

실무 답변 방향: "트레이스 하나로 '이 요청이 왜 느렸는지'는 알 수 있지만, '이런 느린 요청이 얼마나 자주 발생하는지'는 알 수 없습니다. 그래서 트레이스에서 패턴을 발견하면, 그 패턴에 해당하는 메트릭(예: 캐시 미스율)을 별도로 만들어서 전체 규모를 파악하는 식으로 둘을 함께 씁니다."

흔한 오해와 Trade-off

오해 1: "트레이싱을 도입하면 모든 지연 문제가 바로 보인다"

현실: Span 경계를 잘못 설계하면(너무 크게 나누면) 트레이스는 있어도 정작 어느 구간이 문제인지 보이지 않는다. 또한 계측이 빠진 구간(라이브러리 내부, 서드파티 SDK)은 트레이스에 "빈 구간(블랙박스)"으로 남아 원인을 알 수 없다.

올바른 접근: 트레이싱은 계측한 만큼만 보여준다. 새로운 병목 패턴을 발견할 때마다 계측 범위를 넓혀가는 지속적인 작업이 필요하다.

오해 2: "Sampling 비율을 낮추면 트레이싱 데이터의 가치도 그만큼 낮아진다"

현실: 단순 확률 기반(head-based) sampling만 쓰면 맞는 말이지만, tail-based sampling으로 에러/느린 요청을 우선 보존하면 전체 저장량은 줄여도 "봐야 할 문제 트레이스"의 보존율은 오히려 높일 수 있다.

올바른 접근: Sampling 비율(quantity)과 Sampling 전략(quality)은 별개의 문제다. 비율을 낮추더라도 전략을 잘 설계하면 실질적 가치는 유지된다.

오해 3: "트레이싱이 있으면 로그는 덜 남겨도 된다"

현실: 트레이스는 샘플링되기 때문에 특정 사용자의 특정 요청이 표본에서 빠질 수 있다. 반면 에러 로그는 보통 100% 남긴다. 또한 트레이스의 span attribute는 용량 제약으로 세부 정보를 다 담기 어렵다.

올바른 접근: 트레이스는 "어디가 느렸는가"의 큰 그림을, 로그는 "정확히 어떤 값으로 무엇이 실패했는가"의 세부 사실을 담당한다. 서로 대체재가 아니라 trace_id로 연결되는 보완재다.

Trade-off: 트레이싱 커버리지(Coverage) vs 비용/오버헤드

더 많은 요청을, 더 세밀한 span 단위로, 더 오래 저장할수록 문제를 놓칠 확률은 줄어들지만 저장 비용, 네트워크 오버헤드, 애플리케이션 오버헤드가 함께 늘어난다. 완벽한 커버리지(100% sampling, 무한 보관)는 비용상 불가능하기 때문에, 어떤 요청을(에러/느림 우선), 어떤 단위로(네트워크 경계), 얼마나 오래(중요도별 차등 보관) 남길지를 의도적으로 설계하는 것이 시니어급 이해다.

계측이 서비스 성능에 미치는 영향과 실무적 완화 방법까지 알면 시니어급 답변이 된다.

실무 연결 — 이 개념이 실제 어디에 쓰이는가

스타트업 (소규모 팀)

서비스가 2~3개뿐이면 로그 correlation ID만으로도 요청 흐름을 대부분 추적할 수 있다. 서비스 수가 늘어나기 시작하는 시점(대략 4~5개 이상)부터 OpenTelemetry SDK를 붙이고, 무료 오픈소스 백엔드(Jaeger, Grafana Tempo)로 트레이싱을 시작하는 것이 현실적인 우선순위다.

성장기 회사 (수백만 사용자)

서비스 수가 두 자릿수를 넘어가면 트레이싱 없이는 장애의 원인 서비스를 특정하는 것 자체가 어려워진다. 이 시기에는 sampling 전략(비용 관리)과 exemplar 연결(진단 속도)을 본격적으로 설계해야 하고, 계측 표준(모든 서비스가 같은 방식으로 span을 생성하도록 강제하는 라이브러리/미들웨어)을 조직 차원에서 갖추는 것이 중요해진다.

대형 플랫폼 (수천만 사용자)

초당 수만~수십만 요청을 처리하는 규모에서는 100% 트레이싱은 물리적으로 불가능하고, tail-based sampling을 위한 대규모 Collector 파이프라인 자체가 별도의 인프라 프로젝트가 된다. 트레이스 저장 비용, 보관 기간별 계층화(hot/cold storage), 그리고 트레이스를 통한 자동 이상 탐지(예: 특정 span 패턴이 급증하면 자동 알림)까지 확장되는 것이 일반적이다.
