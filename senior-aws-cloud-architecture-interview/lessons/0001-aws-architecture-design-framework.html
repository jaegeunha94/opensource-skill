2. 전제 개념 — 쉬운 한국어로

아키텍처 설계란 무엇인가

건물을 짓기 전에 설계도를 그리는 것처럼, 서비스를 만들기 전에 "어떤 컴퓨터에서 실행하고, 데이터는 어디에 저장하고, 트래픽이 몰리면 어떻게 대응하고, 장애가 나면 어떻게 복구하는가"를 미리 결정하는 것이 아키텍처 설계다.

AWS에서의 아키텍처 설계는 EC2, RDS, S3, VPC 같은 AWS 서비스들을 조합해서 이 질문들에 답을 만드는 작업이다. 중요한 것은 "어떤 서비스를 썼냐"가 아니라 "왜 그 조합을 선택했냐"다.

AWS Well-Architected Framework란

AWS가 수천 개의 실제 고객 아키텍처를 검토한 경험을 바탕으로 만든 설계 기준이다. "좋은 클라우드 아키텍처란 무엇인가"에 대한 AWS의 공식 답변이다.

5개의 기둥(Pillar)으로 구성된다. 면접에서 아키텍처를 설명할 때 이 5개 기둥을 기준으로 설명하면 체계적으로 보인다.

운영 우수성
Operational Excellence
변경을 안전하게, 장애를 빠르게 감지, 운영 절차를 코드로
보안
Security
최소 권한, 암호화, 네트워크 경계, 감사 로그
안정성
Reliability
장애를 예상하고 설계, 자동 복구, 수요 변화 대응
성능 효율성
Performance Efficiency
요구사항에 맞는 자원 선택, 병목 제거, 기술 진화 반영
비용 최적화
Cost Optimization
필요한 만큼만 쓰고, 낭비 없이, 비즈니스 가치 대비 비용
2024년 추가된 6번째 기둥

지속가능성 (Sustainability) — 탄소 배출, 에너지 효율, 환경 영향. 아직 면접에서 자주 나오진 않지만 알아두면 차별화된다.

고가용성 vs 내결함성 — 헷갈리는 두 개념

면접에서 자주 혼용되는 두 개념이다. 정확히 구분할 수 있어야 한다.

고가용성 (High Availability)

장애가 나더라도 짧은 다운타임 후 서비스가 다시 살아나는 것. 일부 요청이 실패하거나 몇 초~몇 분의 중단이 있지만, 자동으로 복구된다.

예시: RDS Multi-AZ — 주 DB가 다운되면 수십 초 안에 예비 DB로 전환된다. 그 수십 초 동안은 연결 실패가 발생한다.

목표: 99.9% ~ 99.99% 가용성

내결함성 (Fault Tolerance)

장애가 나도 서비스가 중단 없이 계속 동작하는 것. 사용자가 장애를 전혀 느끼지 못한다. 구현 비용이 훨씬 높다.

예시: 항공기 제어 시스템, 금융 결제 코어 — 단 1건의 트랜잭션도 잃으면 안 된다.

목표: 99.999% (Five 9s) 이상

핵심 포인트

대부분의 웹 서비스는 내결함성(완전무결한 무중단)이 아니라 고가용성(장애 시 빠른 자동 복구)을 목표로 한다. 내결함성은 훨씬 비싸기 때문이다. 면접에서 "고가용성 설계"라고 했을 때 내결함성 수준까지 설계하면 오히려 비용 감각이 없어 보인다.

Region, Availability Zone, Edge Location

AWS 물리 인프라의 3단 구조다. 이것을 모르면 "Multi-AZ"가 왜 중요한지 설명하기 어렵다.

  • Region (리전): 지리적으로 분리된 데이터센터 클러스터. 예: ap-northeast-2 (서울). 법적 데이터 거주 요건, 지연시간, 서비스 제공 여부를 기준으로 선택한다.
  • Availability Zone (가용 영역, AZ): 리전 내의 물리적으로 분리된 데이터센터. 서울 리전에는 4개의 AZ(ap-northeast-2a/b/c/d)가 있다. 하나의 AZ가 정전이나 재해로 다운돼도 다른 AZ는 살아있다.
  • Edge Location: CloudFront CDN 캐시 서버. 전 세계 400개 이상. 리전/AZ와 달리 아키텍처 고가용성 설계에 직접 쓰이진 않는다.
왜 Multi-AZ가 중요한가

같은 리전의 서로 다른 AZ에 서버를 분산하면, 하나의 AZ 전체가 다운돼도 나머지 AZ에서 서비스를 이어간다. 실제로 2011년 AWS us-east-1 리전에서 단일 AZ 장애로 대규모 서비스 중단이 발생했다. 이후 Multi-AZ 설계가 표준이 됐다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 체계적 사고: AWS 서비스를 나열하는 게 아니라, 요구사항을 분석하고 그에 맞는 구조를 선택하는 과정을 볼 수 있는가
  • trade-off 인식: 비용, 가용성, 보안, 성능이 서로 충돌할 때 어떻게 결정하는가. "비용은 생각 안 하고 최대 고가용성"이라고 하면 실무 감각이 없어 보인다.
  • 실무 경험: 장애를 실제로 겪어본 사람은 "이 구조에서 이 부분이 Single Point of Failure가 될 수 있다"고 스스로 짚어낸다.
  • 커뮤니케이션: 개발자가 아닌 사람에게도 아키텍처 결정을 설명할 수 있는가. 시니어는 기술적 의사결정을 비즈니스 언어로 번역해야 한다.

이 질문은 "AWS를 아는가"를 묻는 게 아니다. "아키텍처 설계를 어떻게 사고하는가"를 묻는다. Well-Architected Framework를 암기했느냐가 아니라, 그 사고 방식이 자신의 것이 되어 있느냐를 본다.

4. 실무 연결

실제로 아키텍처 결정이 중요한 순간들

시나리오 1 — 신규 서비스 런칭

"빠르게 출시해야 하는데, 예산이 제한돼 있고, 6개월 후 사용자가 10배 늘 수 있다."

이런 상황에서 처음부터 Multi-AZ + 리드 복제본 + 글로벌 CDN을 모두 구성하면 월 수천 달러다. 초기엔 단순하게 가고(단일 AZ, 소형 인스턴스), 트래픽에 맞춰 확장하는 전략이 현실적이다. 단, 나중에 확장할 수 있도록 설계의 여지를 남겨둬야 한다. 예: Auto Scaling Group으로 감싸두기, RDS를 처음부터 Multi-AZ 전환이 쉬운 구조로.

시나리오 2 — 장애 post-mortem 후 개선

서비스가 단일 AZ EC2 하나에서 실행 중이었는데, 인스턴스가 다운되면서 30분간 서비스 중단.

이후 ALB + Auto Scaling Group + Multi-AZ 배포로 변경. 이제 하나의 EC2가 다운돼도 ALB가 자동으로 트래픽을 살아있는 인스턴스로 보낸다. 이런 경험을 면접에서 구체적으로 말할 수 있으면 강력한 답변이 된다.

시나리오 3 — 비용 스파이크 대응

잘못된 Auto Scaling 설정으로 인해 갑자기 인스턴스가 100개가 뜨면서 월 비용이 3배가 된 사례. CloudWatch 알람과 Max Capacity 설정이 왜 중요한지 실무에서 체감하게 된다.

5. 답변 사고 순서

면접에서 "AWS 아키텍처를 어떻게 설계하시겠습니까"라는 질문을 받으면, 아래 순서로 생각하고 답변하라.

  1. 요구사항 확인

    먼저 질문을 한다. "어떤 서비스입니까? 예상 트래픽 규모는? 데이터 보존 요건은? 목표 가용성은 몇 %입니까? 예산 범위가 있습니까?"
    면접관이 정보를 주면 그걸 쓰고, 안 주면 "그러면 제가 가정을 하겠습니다"라고 한 뒤 시작한다. 요구사항 없이 설계하는 것은 시니어답지 않다.

  2. 보안 경계 설정 (VPC, IAM)

    어떤 구성 요소가 퍼블릭에 노출되고, 어떤 것은 내부에만 있어야 하는가. 최소 권한 원칙으로 IAM을 설계한다.

  3. 컴퓨팅 레이어 설계 (EC2, ECS, Lambda)

    어떤 런타임이 필요한가. 상태가 있는가 없는가. 급격한 트래픽 변동이 있는가. 이에 따라 EC2 + ASG, 컨테이너, 서버리스 중 선택한다.

  4. 데이터 레이어 설계 (RDS, S3, ElastiCache)

    관계형 데이터가 필요한가. 읽기 부하가 높은가(읽기 복제본). 세션/캐시가 필요한가. 정적 파일은 어디에.

  5. 가용성 설계 (Multi-AZ, ELB, Auto Scaling)

    단일 장애 지점(Single Point of Failure)이 어디인가. 각 레이어를 Multi-AZ로 분산하고 있는가.

  6. 운영 가시성 (CloudWatch, 알람)

    CPU, 메모리, 응답시간, 오류율을 어떻게 모니터링하는가. 이상 시 알람이 울리는가. Auto Scaling 트리거 기준은.

  7. 비용 trade-off 언급

    이 설계의 예상 비용 수준과, 비용을 줄이려면 어떤 부분을 타협할 수 있는지 함께 말한다.

핵심 팁

순서가 틀려도 괜찮다. 중요한 것은 모든 레이어를 빠뜨리지 않고 다루는 것, 그리고 각 선택에 이유가 있는 것이다.

6. 핵심 답변 예시

면접 답변 예시 — 웹 서비스 신규 구축

"먼저 요구사항을 확인하겠습니다. 월 평균 트래픽 규모, 목표 가용성, 데이터 법적 거주 요건, 예산 수준을 확인한 뒤 설계를 시작합니다.

일반적인 웹 서비스라면 보안 경계부터 잡겠습니다. VPC를 구성하고, 퍼블릭 서브넷에는 로드밸런서(ALB)만, 프라이빗 서브넷에 EC2 애플리케이션 서버와 RDS를 배치합니다. 인터넷에서 직접 EC2에 접근하는 경로를 차단하고, RDS는 애플리케이션 서버에서만 접근 가능하도록 보안 그룹을 설정합니다.

컴퓨팅은 EC2 + Auto Scaling Group 조합을 사용하고 ALB와 연결합니다. 최소 2개 AZ에 걸쳐 인스턴스를 배포해서 단일 AZ 장애에 대응합니다.

데이터는 RDS Multi-AZ로 구성합니다. 주 DB가 다운되면 수십 초 내에 예비 DB로 자동 전환됩니다. 읽기 부하가 높다면 읽기 복제본을 추가합니다.

정적 파일은 S3 + CloudFront로 분리해서 EC2 부하를 줄이고 응답속도를 높입니다.

CloudWatch로 CPU, 메모리, 응답시간, 오류율을 모니터링하고, 임계값 초과 시 SNS 알람과 Auto Scaling을 연동합니다.

비용 측면에서는 초기에 예약 인스턴스 대신 온디맨드로 시작해 사용 패턴을 파악한 뒤, 안정적인 기준 트래픽은 예약 인스턴스로 전환해 비용을 절감하겠습니다."

이 답변이 좋은 이유: ① 요구사항 확인으로 시작 ② 보안 → 컴퓨팅 → 데이터 → 가용성 → 운영 → 비용 순서로 전 레이어를 다룸 ③ 각 선택에 이유가 있음 ④ 비용 trade-off까지 언급함

7. Senior-level Follow-up Q&A

Q1. "가용성이 99.9%와 99.99%의 차이가 실제로 얼마나 됩니까?"

99.9% → 연간 약 8.7시간의 다운타임 허용. 99.99% → 연간 약 52분. 0.01% 차이가 연간 8시간 차이다.

중요한 것은 비용이다. 99.99%를 달성하려면 모든 레이어의 이중화, 자동 장애 감지, 50ms 이내의 자동 전환이 필요하다. 이 비용이 다운타임으로 인한 매출 손실보다 큰가를 계산해야 한다.

예를 들어 e-커머스에서 시간당 매출이 1억 원이라면 연간 8시간 다운타임 비용은 8억 원이다. 99.99%를 달성하는 인프라 추가 비용이 이보다 작으면 투자할 가치가 있다.

Q2. "Single Point of Failure를 어떻게 찾습니까?"

각 구성 요소를 하나씩 제거했을 때 서비스 전체가 다운되는지 확인하는 mental simulation을 한다.

EC2 하나가 다운되면? → ALB + ASG가 있으면 괜찮다. ALB가 다운되면? → AWS managed 서비스라 실제로는 거의 없지만, ALB 자체도 내부적으로 Multi-AZ다. NAT Gateway가 하나뿐이라면? → 단일 AZ NAT Gateway는 SPOF가 된다. AZ당 하나씩 배치해야 한다.

Q3. "이 설계에서 비용을 50% 줄이라면 어떻게 하겠습니까?"

먼저 현재 비용의 breakdown을 본다. EC2가 크면 예약 인스턴스 전환(최대 70% 절감). RDS Multi-AZ를 제거하면 비용이 절반이 되지만 가용성이 낮아진다 — 이 trade-off를 명시하고 의사결정자에게 판단을 요청한다.

피크 타임이 있는 서비스라면 Spot 인스턴스로 일부 대체. 야간에 개발/스테이징 환경을 자동으로 중단. S3 스토리지를 자주 접근 안 하는 데이터는 S3 Glacier로 이동.

중요한 것은 "무조건 50% 줄이겠다"가 아니라 "어떤 기능/가용성을 포기하면 50%를 줄일 수 있고, 어떤 것은 포기할 수 없는지" 명확히 하는 것이다.

Q4. "Well-Architected Review를 실무에서 어떻게 활용하셨나요?"

AWS Well-Architected Tool은 AWS 콘솔에서 무료로 사용할 수 있다. 5개 기둥별로 수백 개의 체크리스트 질문에 답하면 현재 아키텍처의 위험 요소와 개선 포인트를 리포트로 준다.

실무에서는 신규 서비스 런칭 전이나 연간 인프라 점검 때 팀이 함께 리뷰한다. "보안 그룹이 0.0.0.0/0으로 열려있는 곳은 없는가", "모든 데이터가 암호화됐는가", "CloudTrail이 활성화됐는가" 같은 기준점을 잡는 데 쓴다.

Q5. "리전 선택 기준이 무엇입니까?"

① 법적 데이터 거주 요건: 한국 사용자 개인정보는 국내 서버에 저장해야 한다는 규정이 있다면 ap-northeast-2(서울) 또는 한국 내 대안을 써야 한다.
② 지연시간: 사용자가 주로 어느 지역에 있는가. 서울 사용자에게 us-east-1 리전은 100ms 이상 지연이 발생한다.
③ 서비스 제공 여부: 리전마다 제공하는 AWS 서비스가 다르다. 일부 최신 서비스는 특정 리전에만 있다.
④ 비용: 리전마다 EC2, 데이터 전송 비용이 다르다. us-east-1이 대체로 가장 저렴하다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 요구사항을 먼저 확인하거나 가정을 명시한다
  • 보안, 가용성, 비용 레이어를 모두 다룬다
  • 각 선택에 이유가 있다 ("Multi-AZ를 선택한 이유는 단일 AZ 장애 시 자동 복구를 위해서입니다")
  • trade-off를 자발적으로 말한다 ("이 구성은 비용이 높지만, 가용성을 위해 필요합니다. 만약 비용을 줄여야 한다면 읽기 복제본을 제거하는 방법이 있습니다")
  • 장애 시나리오를 스스로 언급한다 ("이 설계에서 NAT Gateway가 단일 AZ라면 SPOF가 됩니다")
약한 답변
  • 서비스 이름만 나열한다 ("EC2에 올리고, RDS 쓰고, S3로 저장합니다")
  • 왜 그 서비스를 선택했는지 이유가 없다
  • 보안이나 비용 레이어를 빠뜨린다
  • 모든 서비스를 최고 등급으로 설정한다 (비용 감각 없음)
  • 요구사항 없이 설계를 시작한다
  • "음..." 후에 서비스 이름을 연속으로 나열하다가 멈춘다
왜 약한 답변이 약한가

"EC2, RDS, S3를 씁니다"는 AWS를 처음 배운 사람도 말할 수 있다. 시니어 면접에서는 어떤 상황에서 어떤 구성을 선택했고, 그 결과 어떤 trade-off가 생기며, 어떻게 모니터링하고 장애에 대응하는가를 본다. 서비스 이름 나열은 "암기했다"를 보여줄 뿐, "설계할 수 있다"를 보여주지 못한다.

9. 흔한 오해

오해 1

"Multi-AZ RDS는 읽기 성능도 향상된다."

실제

Multi-AZ는 고가용성 목적이다. 예비 인스턴스는 읽기 요청을 처리하지 않고, 오직 페일오버를 위해 대기한다. 읽기 성능을 높이려면 읽기 복제본(Read Replica)을 별도로 추가해야 한다.

오해 2

"Auto Scaling이 있으면 트래픽이 몰릴 때 즉시 대응된다."

실제

EC2 인스턴스가 뜨는 데 2~3분이 걸린다. 갑작스러운 트래픽 스파이크에는 이미 늦다. 이를 보완하려면 ① 항상 여유 인스턴스를 최소 유지하거나, ② 예측 가능한 패턴이면 스케줄 기반 Scaling을 쓰거나, ③ CloudFront 캐싱으로 Origin 부하를 줄여야 한다.

오해 3

"보안 그룹(Security Group)과 NACL은 같은 것이다."

실제

보안 그룹은 Stateful — 인바운드를 허용하면 응답 트래픽은 자동으로 통과된다. NACL은 Stateless — 인바운드와 아웃바운드를 독립적으로 설정해야 한다. 보안 그룹은 인스턴스 레벨, NACL은 서브넷 레벨에서 적용된다. 실무에서는 보안 그룹을 1차 방어선으로 쓰고, NACL은 IP 대역 차단 같은 광범위한 차단에 쓴다.

오해 4

"IAM 사용자에게 AdministratorAccess를 주면 편리하다."

실제

자격증명이 유출되면 계정 전체가 탈취된다. 실무에서는 ① 사람은 최소 권한 IAM 사용자 또는 SSO 기반으로, ② 서비스/EC2는 IAM Role을 통해 필요한 권한만, ③ AdministratorAccess는 긴급 상황에만 쓰는 전용 역할로 제한한다.

오해 5

"같은 리전 내 AZ 간 데이터 전송은 무료다."

실제

같은 리전 내라도 AZ 간 데이터 전송에는 비용이 발생한다 (GB당 약 $0.01). Multi-AZ RDS의 동기 복제, ALB의 AZ 간 라우팅 등이 모두 이 비용에 해당한다. 대용량 데이터를 자주 AZ 간에 전송한다면 이 비용이 의외로 클 수 있다.
