2. 전제 개념 — 쉬운 한국어로

VPC란 무엇인가

VPC(Virtual Private Cloud)는 AWS 안에 내가 마음대로 주소 체계와 경계를 정할 수 있는 나만의 가상 사설 네트워크다. 아파트 단지 전체를 빌린다고 생각하면 쉽다. 단지 안에 몇 개의 동(서브넷)을 만들지, 어느 동은 정문 경비(인터넷 접근)를 두고 어느 동은 외부인 출입을 아예 막을지를 내가 설계한다.

VPC를 만들 때 가장 먼저 정하는 것이 CIDR 블록이다. 예를 들어 10.0.0.0/16은 10.0.0.0부터 10.0.255.255까지 약 6만 5천 개의 IP 주소를 쓸 수 있다는 뜻이다. 이 큰 대역을 잘게 쪼갠 것이 서브넷이다.

서브넷은 "네트워크 위치"이지 "보안 설정"이 아니다

가장 먼저 바로잡아야 할 개념이 있다. 서브넷 자체에는 "퍼블릭"이나 "프라이빗"이라는 속성이 원래 없다. AWS 콘솔에 그런 표시가 나오지만, 그건 그 서브넷의 라우팅 테이블이 인터넷 게이트웨이(IGW)로 가는 경로를 갖고 있는지로 판단한 결과일 뿐이다.

  • 퍼블릭 서브넷: 라우팅 테이블에 0.0.0.0/0 → Internet Gateway 경로가 있다. 이 서브넷의 리소스가 퍼블릭 IP를 가지면 인터넷과 직접 통신할 수 있다.
  • 프라이빗 서브넷: 라우팅 테이블에 IGW로 가는 경로가 없다. 인터넷에서 직접 들어올 수 없다. 대신 0.0.0.0/0 → NAT Gateway 경로를 둬서 아웃바운드(나가는) 트래픽만 허용하는 경우가 많다.
핵심 포인트

"프라이빗 서브넷 = 인터넷 완전 차단"이 아니다. 정확히는 인바운드(외부 → 내부)가 불가능한 것이고, NAT Gateway를 통한 아웃바운드(내부 → 외부)는 가능하게 만드는 것이 일반적인 구성이다. 이 구분을 못 하면 "왜 프라이빗 EC2가 yum update는 되는데 외부에서 SSH는 안 되나요"라는 질문에 답을 못 한다.

라우팅 테이블 — 트래픽이 어디로 갈지 정하는 표

각 서브넷은 반드시 하나의 라우팅 테이블과 연결된다. 라우팅 테이블은 "이 목적지로 가는 트래픽은 이 경로로 보내라"는 규칙의 목록이다. 기본적으로 VPC 내부(10.0.0.0/16) 트래픽은 local로 자동 라우팅되고, 그 외 목적지(예: 0.0.0.0/0, 인터넷 전체)를 어디로 보낼지가 서브넷의 성격을 결정한다.

보안 그룹 vs NACL — Stateful과 Stateless의 차이

둘 다 트래픽을 허용/차단하는 방화벽이지만, 적용 레벨과 동작 방식이 완전히 다르다.

보안 그룹 (Security Group)

인스턴스(ENI) 레벨에서 적용된다.

Stateful: 인바운드를 허용하면, 그에 대한 응답 트래픽은 아웃바운드 규칙과 무관하게 자동으로 통과된다.

규칙은 허용(Allow)만 설정할 수 있다. 명시하지 않은 트래픽은 자동 차단된다.

다른 보안 그룹을 소스로 지정할 수 있어서 "이 보안 그룹을 가진 EC2에서 오는 트래픽만 허용" 같은 유연한 설계가 가능하다.

NACL (Network ACL)

서브넷 레벨에서 적용된다. 그 서브넷의 모든 리소스에 일괄 적용.

Stateless: 인바운드를 허용해도 응답 트래픽(아웃바운드)을 별도로 허용해야 한다. 잊으면 통신이 반쪽만 된다.

규칙에 허용과 차단을 모두 명시적으로 설정할 수 있다. 특정 악성 IP 대역을 명시적으로 차단할 때 유용하다.

규칙 번호 순서대로 평가되고, 먼저 매칭된 규칙이 적용된다.

실무에서는 이렇게 쓴다

대부분의 팀은 보안 그룹을 1차 방어선으로 세밀하게 관리하고, NACL은 건드리지 않거나 "특정 악성 IP 대역 전체 차단" 같은 굵직한 용도로만 쓴다. NACL을 서비스 로직 수준으로 세밀하게 관리하면 Stateless 특성 때문에 실수가 잦고 디버깅이 어려워진다.

NAT Gateway — 프라이빗 서브넷의 아웃바운드 통로

NAT(Network Address Translation) Gateway는 프라이빗 서브넷의 리소스가 사설 IP로는 나갈 수 없는 인터넷에, AWS가 관리하는 퍼블릭 IP로 "대신" 나갈 수 있게 해주는 관리형 서비스다. OS 패치, 외부 API 호출, 패키지 다운로드 등에 필요하다. 대신 인터넷에서 NAT Gateway를 거쳐 프라이빗 서브넷으로 들어올 수는 없다 — 이것이 NAT의 핵심 보안 가치다.

  • NAT Gateway는 반드시 퍼블릭 서브넷에 배치한다 (IGW로 나가야 하므로).
  • NAT Gateway는 단일 AZ 리소스다. 하나만 만들면 그 AZ가 장애 나면 다른 AZ의 프라이빗 서브넷도 아웃바운드가 끊긴다.
  • NAT Instance(직접 관리하는 EC2 기반 NAT)는 옛날 방식이다. 요즘은 관리형인 NAT Gateway를 쓰는 것이 표준이다. 대역폭 제한이 없고 AWS가 가용성을 관리해준다는 것이 장점이지만, 시간당 요금 + 데이터 처리 요금이 함께 청구돼 비용이 상당하다.

Bastion Host와 그 대안

Bastion Host(또는 Jump Box)는 프라이빗 서브넷의 서버에 접속하기 위해 거쳐가는 관문 역할의 퍼블릭 EC2다. 과거에는 표준 패턴이었지만, 지금은 문제가 많은 접근으로 여겨진다.

Bastion Host의 문제
  • 퍼블릭 서브넷에 SSH 포트(22)가 열린 EC2가 항상 떠 있다 — 공격 표면이다.
  • SSH 키 관리, 로테이션, 접근 로그를 별도로 구축해야 한다.
  • Bastion 자체가 뚫리면 내부 전체가 뚫리는 단일 실패 지점이 된다.
실무 대안

AWS Systems Manager Session Manager를 쓰면 인바운드 포트를 전혀 열지 않고도 IAM 권한 기반으로 EC2에 셸 접속을 할 수 있다. 접속 기록이 CloudTrail/CloudWatch Logs에 자동으로 남고, SSH 키 관리가 필요 없다. 최근 시니어 인터뷰에서는 "아직도 Bastion을 쓰시나요?"라는 질문 자체가 최신 실무 감각을 테스트하는 트랩인 경우가 많다.

VPC Endpoint — NAT 없이 AWS 서비스에 접근하기

프라이빗 서브넷의 EC2가 S3나 DynamoDB 같은 AWS 서비스에만 접근하면 되는데 NAT Gateway를 거치면, 실제로는 인터넷을 나갔다 들어오는 것도 아닌데 NAT 처리 비용이 발생한다. VPC Endpoint는 AWS 서비스로 가는 트래픽이 인터넷을 거치지 않고 AWS 내부 네트워크로만 이동하게 해준다. S3/DynamoDB는 무료인 Gateway Endpoint를, 그 외 대부분의 서비스는 Interface Endpoint(PrivateLink 기반, 시간당 요금 있음)를 쓴다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 보안 경계 감각: "무엇을 인터넷에 노출시켜야 하고, 무엇은 절대 노출시키면 안 되는가"를 스스로 판단할 수 있는가. 이건 서비스 나열로 안 되고 원리를 알아야 한다.
  • Blast Radius(피해 범위) 사고: 하나의 서버가 뚫렸을 때 피해가 어디까지 번지는지 예측할 수 있는가. 네트워크 경계를 잘 나누면 침해 확산을 막는다.
  • 실수 경험: NACL Stateless 특성을 몰라서 장애를 겪어봤는가, NAT Gateway 단일 AZ 구성으로 장애를 겪어봤는가 같은 실전 경험을 본다.
  • 최신 실무 감각: Bastion Host 같은 구식 패턴을 여전히 기본값으로 쓰는지, Session Manager 같은 최신 대안을 알고 있는지도 시니어와 미드레벨을 가르는 지점이다.

이 질문은 VPC 설정 화면을 조작할 수 있는지를 묻는 게 아니다. "신뢰할 수 없는 네트워크(인터넷)와 신뢰할 수 있는 네트워크(내부)의 경계를 어디에 어떻게 그을 것인가"라는 보안 설계 사고를 묻는다.

4. 실무 연결

시나리오 1 — RDS가 퍼블릭 서브넷에 있었던 사고

신규 팀원이 빠르게 개발하려고 RDS를 "Publicly Accessible: Yes"로 생성하고 보안 그룹도 임시로 0.0.0.0/0을 열어뒀다. 몇 주 뒤 데이터베이스 스캐너 봇에 의해 무차별 로그인 시도가 관측됐다. 다행히 비밀번호가 강력해서 뚫리진 않았지만, 이후 팀은 "RDS는 무조건 프라이빗 서브넷 + 보안 그룹으로 애플리케이션 서버에서만 접근 가능"이라는 규칙을 코드 리뷰 체크리스트에 넣었다.

시나리오 2 — NAT Gateway 단일 AZ로 인한 부분 장애

비용을 아끼려고 NAT Gateway를 하나만 만들고 모든 AZ의 프라이빗 서브넷이 이를 공유하도록 라우팅했다. 그 NAT Gateway가 있는 AZ에 장애가 나자, 다른 AZ의 EC2들은 살아있는데도 외부 API 호출과 패키지 업데이트가 전부 실패했다. 이후 AZ마다 NAT Gateway를 하나씩 두고, 각 AZ의 프라이빗 서브넷은 같은 AZ의 NAT Gateway만 쓰도록 라우팅을 수정했다. 비용은 늘었지만 AZ 간 장애 격리가 확보됐다.

시나리오 3 — NACL 설정 실수로 인한 반쪽 통신 장애

특정 악성 IP 대역을 NACL에서 인바운드만 차단하고 아웃바운드 규칙은 그대로 뒀다. Stateless 특성을 모르고 "인바운드만 차단하면 되겠지"라고 판단한 것이다. 나중에 다른 목적으로 아웃바운드 규칙까지 수정하다가 정상 트래픽의 응답 패킷이 막혀 일부 API 호출이 타임아웃되는 장애가 발생했다. NACL 변경은 Stateless라는 것을 항상 염두에 두고, 규칙 하나를 바꿀 때 인바운드/아웃바운드 양쪽을 함께 검토해야 한다는 교훈을 얻었다.

시나리오 4 — Bastion Host 키 유출

퇴사한 계약직 개발자의 SSH 키가 Bastion Host에서 제때 회수되지 않아, 몇 달 뒤 감사에서 뒤늦게 발견된 사례가 있다. 이 사건 이후 팀은 SSH 키 기반 Bastion을 전면 폐기하고 Session Manager로 전환했다. IAM 사용자 비활성화 즉시 접근이 차단되고, 모든 세션이 로그로 남아 감사가 쉬워졌다.

5. 답변 사고 순서

면접에서 "VPC를 어떻게 설계하시겠습니까"라는 질문을 받으면, 아래 순서로 생각하고 답변하라.

  1. CIDR 블록 설계

    VPC 전체 대역을 정하고(예: 10.0.0.0/16), 향후 다른 VPC와 Peering/Transit Gateway로 연결될 가능성을 고려해 겹치지 않는 대역을 선택한다. 서브넷 대역도 여유 있게 나눠(예: /20) 나중에 IP 고갈을 피한다.

  2. AZ별 서브넷 계층 나누기

    최소 2개 AZ에 각각 퍼블릭/프라이빗(/필요하면 격리된 데이터 전용) 서브넷을 만든다. 로드밸런서는 퍼블릭, 애플리케이션 서버는 프라이빗, DB는 더 격리된 프라이빗(데이터 전용) 서브넷에 둔다.

  3. 라우팅 테이블 설계

    퍼블릭 서브넷은 IGW로, 프라이빗 서브넷은 같은 AZ의 NAT Gateway로 라우팅한다. DB 전용 서브넷은 아예 인터넷으로 나가는 경로가 필요 없다면 라우팅 자체를 열지 않는다.

  4. 보안 그룹 설계 (최소 권한)

    계층별로 보안 그룹을 나눈다. ALB 보안 그룹은 인터넷에서 443을 허용, 애플리케이션 보안 그룹은 ALB 보안 그룹에서만 트래픽을 허용, DB 보안 그룹은 애플리케이션 보안 그룹에서만 허용. IP 대역이 아니라 보안 그룹을 소스로 지정하면 EC2가 스케일 아웃/인 되어도 규칙을 다시 안 만들어도 된다.

  5. NACL은 보조 수단으로만

    기본 NACL(모두 허용)을 유지하거나, 특정 악성 대역을 명시적으로 차단하는 용도로만 최소한으로 쓴다.

  6. NAT Gateway 고가용성 설계

    비용이 허용된다면 AZ마다 NAT Gateway를 하나씩 둔다. 비용을 줄여야 한다면 단일 NAT를 감수하되, 그것이 SPOF임을 명시하고 trade-off로 설명한다.

  7. 운영 접근 경로 설계

    SSH 포트를 여는 Bastion 대신 Session Manager 또는 VPN(Client VPN, Site-to-Site VPN)을 사용한다. AWS 서비스 접근이 필요하면 VPC Endpoint로 NAT를 거치지 않게 최적화한다.

핵심 팁

"어떤 서비스를 퍼블릭에 두고 어떤 서비스를 프라이빗에 두는가"보다 "왜 그렇게 나눴는가, 하나가 뚫렸을 때 피해가 어디까지 번지는가"를 항상 함께 설명하라.

6. 핵심 답변 예시

면접 답변 예시 — 3-tier 웹 서비스의 VPC 설계

"VPC는 10.0.0.0/16으로 만들고, 최소 2개 AZ에 걸쳐 퍼블릭 서브넷, 애플리케이션용 프라이빗 서브넷, 데이터베이스용 프라이빗 서브넷 이렇게 3계층으로 나누겠습니다.

퍼블릭 서브넷에는 ALB와 NAT Gateway만 두고, 인터넷 게이트웨이로 가는 라우팅을 설정합니다. 애플리케이션 프라이빗 서브넷의 EC2들은 각 AZ에 있는 NAT Gateway로 아웃바운드 라우팅을 하고, 인바운드는 ALB 보안 그룹에서 오는 트래픽만 허용합니다.

데이터베이스 서브넷은 더 격리해서, 애플리케이션 보안 그룹에서만 DB 포트로 접근을 허용하고, 인터넷으로 나가는 라우팅 자체를 열지 않습니다.

NACL은 기본값을 유지하고, 보안 그룹을 1차 방어선으로 세밀하게 관리합니다. 보안 그룹 소스는 IP 대역이 아니라 다른 보안 그룹을 지정해서, Auto Scaling으로 인스턴스가 늘어나거나 줄어도 규칙을 다시 만들 필요가 없게 합니다.

운영 서버 접속은 Bastion Host 대신 Systems Manager Session Manager를 씁니다. 인바운드 포트를 전혀 열지 않고 IAM 권한 기반으로 접속하고, 모든 세션이 CloudTrail에 기록됩니다.

비용을 고려해야 한다면 NAT Gateway를 AZ당 하나씩(고가용성) 둘지, 하나만 공유(비용 절감이지만 SPOF)할지 trade-off를 설명하고 팀과 논의하겠습니다."

이 답변이 좋은 이유: ① 계층별로 서브넷을 명확히 나눔 ② 보안 그룹을 IP가 아닌 보안 그룹 참조로 설계해 확장성을 고려함 ③ NACL과 보안 그룹의 역할을 구분해서 씀 ④ Bastion Host 대신 최신 대안(Session Manager)을 제시함 ⑤ NAT Gateway 비용/가용성 trade-off를 자발적으로 언급함

7. Senior-level Follow-up Q&A

Q1. "NAT Gateway를 AZ마다 두라고 하셨는데, 왜 하나를 여러 AZ가 공유하면 안 됩니까?"

NAT Gateway는 물리적으로 하나의 AZ에만 존재하는 리소스다. 다른 AZ의 프라이빗 서브넷이 그 NAT Gateway를 쓰도록 라우팅하면, 그 NAT Gateway가 있는 AZ가 장애 나거나 유지보수로 내려가는 순간 다른 AZ의 서버들까지 아웃바운드 통신이 끊긴다.

애초에 Multi-AZ로 설계한 목적이 "하나의 AZ 장애가 전체에 영향을 주지 않게"인데, NAT Gateway를 공유하면 그 목적이 깨진다. AZ마다 하나씩 두고 같은 AZ의 서브넷만 쓰도록 라우팅해야 AZ 간 장애 격리가 완성된다.

Q2. "VPC Peering과 Transit Gateway는 언제 각각 선택하시나요?"

VPC Peering은 두 VPC를 1:1로 직접 연결한다. 무료이고 단순하지만, VPC가 5개, 10개로 늘어나면 N(N-1)/2 개의 연결을 각각 관리해야 하고 전이적(transitive) 라우팅이 안 된다 — A-B, B-C가 연결돼 있어도 A에서 C로 직접 갈 수 없다.

Transit Gateway는 허브 역할을 하는 중앙 라우터다. 모든 VPC가 Transit Gateway 하나에만 연결하면 되고, 전이적 라우팅도 지원한다. VPC가 3~4개를 넘어가거나, 온프레미스와의 하이브리드 연결까지 고려한다면 Transit Gateway가 관리 복잡도를 크게 줄인다. 다만 시간당 요금 + 데이터 처리 요금이 있어 소규모에는 과할 수 있다.

Q3. "보안 그룹 규칙이 수백 개로 늘어나서 관리가 안 될 때 어떻게 하시겠습니까?"

먼저 IP 대역 기반 규칙을 보안 그룹 참조 기반으로 바꾼다. "이 보안 그룹에서 오는 트래픽 허용" 방식으로 바꾸면 실제 리소스 구조가 규칙에 그대로 반영돼 가독성이 올라간다.

다음으로 역할별 보안 그룹(ALB용, App용, DB용, 관리용)로 표준화하고, Infrastructure as Code(Terraform/CDK)로 관리해서 콘솔에서 임의로 규칙이 추가되는 것을 막는다. 정기적으로 미사용 규칙과 0.0.0.0/0 인바운드가 남아있는지 감사한다(AWS Trusted Advisor, Security Hub 활용 가능).

Q4. "프라이빗 서브넷의 EC2가 OS 패치를 받으려면 인터넷 접근이 필요한데, 이게 보안상 문제 아닌가요?"

NAT Gateway를 통한 아웃바운드 접근은 "내부에서 외부로 나가는 연결"만 허용하고 "외부에서 내부로 들어오는 연결"은 차단한다는 점에서 인바운드 노출과는 근본적으로 다르다. 공격자가 그 EC2의 IP를 알아도 직접 접속할 방법이 없다.

더 엄격하게 하고 싶다면, 패키지 미러를 VPC 내부(S3 + VPC Endpoint)에 두거나, Systems Manager Patch Manager로 아웃바운드 트래픽 자체를 최소화하는 방법도 있다. 완전히 인터넷 접근을 끊고 싶다면 VPC Endpoint로 필요한 AWS 서비스만 화이트리스트 방식으로 열어주는 완전 격리 설계도 가능하지만, 운영 복잡도가 올라간다는 trade-off가 있다.

Q5. "CIDR 블록을 처음 설계할 때 가장 많이 하는 실수가 무엇입니까?"

가장 흔한 실수는 서브넷 대역을 필요한 만큼만 딱 맞춰(예: /28) 만드는 것이다. Auto Scaling으로 인스턴스가 늘어나거나 새로운 서브넷 종류(예: EKS 전용 서브넷)가 필요해지면 IP가 금방 고갈된다. 여유 있게(예: /20~/22) 잡는 것이 안전하다.

두 번째 실수는 다른 VPC나 온프레미스 네트워크와 CIDR이 겹치는 것이다. 나중에 VPC Peering이나 Site-to-Site VPN으로 연결하려 할 때 겹치는 대역은 라우팅이 불가능해서 VPC를 통째로 다시 만들어야 하는 경우도 있다. 처음부터 회사 전체의 IP 대역 계획(IP addressing plan)을 세워두는 것이 좋다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 서브넷을 "퍼블릭/프라이빗"으로 나누는 기준이 라우팅 테이블(IGW 경로 유무)이라는 것을 정확히 설명한다
  • 보안 그룹과 NACL을 Stateful/Stateless로 명확히 구분하고, 실무에서 각각 어떤 용도로 쓰는지 말한다
  • NAT Gateway의 AZ 단위 특성과 고가용성 trade-off를 스스로 짚는다
  • Bastion Host의 한계를 알고 Session Manager 같은 최신 대안을 제시한다
  • 보안 그룹 소스를 IP가 아니라 보안 그룹 참조로 설계해 확장성을 고려한다
약한 답변
  • "퍼블릭 서브넷과 프라이빗 서브넷을 나눕니다"라고만 말하고 판단 기준을 설명 못 한다
  • 보안 그룹과 NACL을 "둘 다 방화벽이에요"로 뭉뚱그린다
  • NAT Gateway를 하나만 만들어도 되는지 아닌지 판단하지 못한다
  • 여전히 Bastion Host가 유일한 방법이라고 생각한다
  • 모든 보안 그룹 규칙에 0.0.0.0/0을 넣어도 문제라고 인식하지 못한다
왜 약한 답변이 약한가

"퍼블릭/프라이빗을 나눕니다"는 AWS 콘솔에서 체크박스를 본 사람이면 누구나 말할 수 있다. 시니어 면접에서는 그 구분이 실제로 어떤 메커니즘(라우팅 테이블)으로 동작하는지, 각 계층의 방화벽 규칙을 어떻게 최소 권한으로 설계하는지, 장애나 침해가 발생했을 때 피해 범위가 어디까지인지를 스스로 추론할 수 있어야 한다.

9. 흔한 오해

오해 1

"프라이빗 서브넷은 인터넷과 완전히 단절되어 있다."

실제

정확히는 인바운드(외부→내부)가 불가능한 것이다. NAT Gateway를 통해 아웃바운드(내부→외부)는 가능한 것이 일반적인 구성이다. 완전히 인터넷과 단절시키려면 라우팅 테이블에 0.0.0.0/0 경로 자체를 두지 않아야 한다(격리 서브넷).

오해 2

"보안 그룹만 잘 설정하면 NACL은 신경 쓸 필요가 없다."

실제

기본 NACL(모두 허용)을 그대로 두면 실무에서 문제가 되진 않는다. 하지만 NACL을 완전히 몰라도 된다는 뜻은 아니다. 특정 악성 IP 대역을 서브넷 전체 레벨에서 차단하고 싶을 때, 또는 규제 요건상 다층 방어(defense in depth)가 필요할 때는 NACL이 여전히 유효한 도구다.

오해 3

"NAT Gateway와 NAT Instance는 이름만 다르고 기능은 같다."

실제

NAT Gateway는 AWS 관리형 서비스로 가용성과 확장을 AWS가 책임진다. NAT Instance는 사용자가 직접 관리하는 EC2로, 패치, 스케일링, 장애 복구를 직접 해야 한다. 대역폭 한계도 인스턴스 타입에 종속된다. 오늘날 신규 구축에서는 특별한 이유(예: NAT Gateway가 지원하지 않는 특정 프로토콜 사용)가 없다면 NAT Gateway가 표준이다.

오해 4

"Bastion Host를 쓰는 것은 여전히 업계 표준이고 문제없다."

실제

Bastion Host는 여전히 쓰이지만, 인바운드 포트를 상시 열어둬야 한다는 근본적인 공격 표면 문제가 있다. AWS Systems Manager Session Manager는 인바운드 포트 없이 IAM 기반으로 접속하고 세션 기록이 자동으로 남아, 최신 아키텍처에서는 기본 선택지로 자리 잡았다. 완전한 폐쇄망 등 특수한 이유가 없다면 Session Manager를 우선 고려하는 것이 현재 실무 감각이다.

오해 5

"VPC의 CIDR을 /16으로 크게 잡으면 충분하고 더 고민할 게 없다."

실제

대역이 커도 여러 VPC/온프레미스와 연결할 계획이 있다면 CIDR이 겹치지 않게 회사 전체의 IP 주소 계획을 미리 세워야 한다. 겹치는 대역은 나중에 Peering이나 VPN 연결 시 라우팅이 불가능해 VPC를 재구축해야 하는 심각한 문제로 이어진다.
