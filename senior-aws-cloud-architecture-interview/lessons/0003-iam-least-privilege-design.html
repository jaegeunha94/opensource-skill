2. 전제 개념 — 쉬운 한국어로

IAM의 세 가지 질문: 누가, 무엇을, 어디에

IAM(Identity and Access Management)은 결국 세 가지 질문에 답하는 시스템이다. Principal(누가 요청하는가), Action(무엇을 하려는가), Resource(어떤 대상에 대해서인가). 이 세 가지를 조합한 규칙이 Policy(정책)이고, Policy는 JSON 문서로 작성된다.

Policy 문서의 최소 구성

Effect(Allow/Deny) · Action(예: s3:GetObject) · Resource(예: 특정 S3 버킷 ARN) · Condition(예: 특정 IP 대역에서만, MFA 필수 등 선택적 제약)

User, Group, Role — 신원의 세 가지 형태

셋 다 "권한을 가질 수 있는 신원"이라는 점은 같지만, 존재 목적이 다르다. 회사 조직에 비유하면 이해가 쉽다.

  • IAM User: 특정 사람이나 애플리케이션에 부여되는 장기 신원이다. 고정된 로그인 정보(비밀번호)와 장기 Access Key를 가질 수 있다. 회사의 정규직 사원증과 비슷하다 — 한 번 발급되면 계속 유효하다.
  • IAM Group: User를 묶어서 정책을 한 번에 관리하는 용도다. Group 자체는 로그인하거나 API를 호출할 수 있는 신원이 아니다. "개발팀 사원증에 공통으로 붙는 출입 권한 스티커" 같은 것이다.
  • IAM Role: 고정된 자격증명이 없는 임시 신원이다. 누군가(사람, EC2, Lambda, 다른 AWS 계정)가 Role을 AssumeRole하면, STS(Security Token Service)가 유효 시간이 정해진 임시 Access Key/Secret Key/Session Token을 발급한다. "방문증" 같은 것이다 — 필요할 때 빌려 쓰고, 시간이 지나면 자동으로 만료된다.
핵심 포인트

Role이 User보다 안전한 이유는 단순히 "AWS가 권장해서"가 아니다. 자격증명이 코드나 서버에 영구히 저장되지 않고, 짧은 시간(기본 1시간, 최대 12시간) 후 자동으로 무효화되기 때문이다. EC2 인스턴스에 Access Key를 하드코딩하거나 환경변수로 박아두면, 그 서버가 뚫리는 순간 그 키는 로테이션되기 전까지 영구히 유효한 만능 열쇠가 된다. Instance Profile을 통해 Role을 붙이면, 애플리케이션 코드는 자격증명을 전혀 몰라도 되고, 탈취되어도 짧은 유효 시간 안에서만 위험이 제한된다.

정책의 두 종류: Identity-based와 Resource-based

권한은 "권한을 받는 쪽(신원)"에 붙일 수도 있고, "권한이 걸리는 대상(자원)"에 붙일 수도 있다.

  • Identity-based Policy: User, Group, Role에 직접 붙는 정책. "이 Role은 S3 버킷 A를 읽을 수 있다"처럼 신원 쪽에서 정의한다.
  • Resource-based Policy: S3 버킷 정책, KMS 키 정책처럼 자원 쪽에 붙는 정책. "이 버킷은 계정 X의 Role Y가 읽을 수 있다"처럼 자원 쪽에서 정의한다. 특히 다른 AWS 계정에서의 접근을 허용할 때는 Resource-based Policy가 필요한 경우가 많다.

정책 평가 로직 — 왜 순서가 중요한가

한 요청에 여러 정책이 동시에 적용될 수 있다(Identity Policy, Resource Policy, Permission Boundary, SCP 등). AWS는 이걸 정해진 순서로 평가한다.

  1. 기본은 암묵적 거부(Implicit Deny): 아무 정책도 명시적으로 허용하지 않으면 요청은 자동으로 거부된다.
  2. 명시적 Deny가 있는지 확인: 관련된 어느 정책에서든 Effect: Deny가 하나라도 매치되면, 다른 모든 Allow와 상관없이 즉시 거부된다. Deny는 항상 최우선이다.
  3. 명시적 Allow가 있는지 확인: Deny가 없다면, 관련된 정책 중 하나라도 Effect: Allow로 매치되면 허용된다.
실무에서 자주 놓치는 지점

"정책을 하나 더 추가했는데 왜 갑자기 접근이 막히지?"라는 질문의 답은 대부분 어딘가에 명시적 Deny가 걸려 있다는 것이다. Deny는 SCP, Permission Boundary, Identity Policy, Resource Policy, VPC Endpoint Policy 어디에든 있을 수 있어서, 디버깅할 때는 이 다섯 곳을 모두 확인해야 한다.

SCP와 Permission Boundary — "권한을 주는 것"이 아니라 "천장을 긋는 것"

시니어 인터뷰에서 가장 자주 틀리는 개념이다. SCP와 Permission Boundary는 권한을 부여하지 않는다. 이미 부여된 권한의 최대 한도(guardrail)를 설정할 뿐이다.

SCP (Service Control Policy)

AWS Organizations 수준에서 계정 전체에 적용되는 상한선이다.

예: "이 OU(조직 단위)에 속한 모든 계정은 특정 리전 밖에서는 아무것도 할 수 없다"처럼 조직 전체 거버넌스에 쓴다.

SCP가 허용하지 않으면, 그 계정 안에서 아무리 강력한 IAM 정책(심지어 AdministratorAccess)을 줘도 소용없다.

Permission Boundary

특정 User나 Role 하나에 적용되는 상한선이다.

예: "이 Role은 Permission Boundary 안에서, 그리고 실제로 부여받은 Identity Policy 안에서, 두 개의 교집합만큼만 할 수 있다"

주로 위임 관리 시나리오에 쓴다. "주니어 관리자가 새로운 Role을 만들 수는 있지만, 자기 자신보다 강한 권한의 Role은 절대 못 만들게" 제한할 때 유용하다.

Confused Deputy Problem과 ExternalId

다른 AWS 계정(예: 서드파티 SaaS 벤더)에게 내 계정의 Role을 맡길 때(Cross-account Role), "Confused Deputy(혼동된 대리인)" 문제가 생길 수 있다. 벤더가 여러 고객의 리소스를 관리하는 동일한 Role ARN을 재사용하면, 악의적인 제3자가 그 벤더를 속여서 자신의 계정이 아닌 다른 고객의 계정에 접근하도록 유도할 수 있다. 이를 막기 위해 Trust Policy에 ExternalId 조건을 넣는다 — 벤더가 나에게 발급해준 고유 문자열을 AssumeRole 요청에 반드시 포함하도록 강제해서, 엉뚱한 계정으로의 요청이 성공하지 못하게 막는다.

자격증명이 유출됐을 때 무엇이 위험한가

IAM User의 Access Key가 GitHub 같은 공개 저장소에 노출되면, 자동화된 스캐너 봇이 수 분 안에 이를 탐지해서 악용한다. 흔한 패턴은 저렴한 리전에 대규모 GPU/컴퓨팅 인스턴스를 띄워 암호화폐 채굴에 쓰는 것이다. Access Key는 로테이션하기 전까지 영구히 유효하다는 점이 핵심 위험이다. 이것이 바로 Role(임시 자격증명)을 우선하고, User의 장기 Access Key 발급 자체를 최소화해야 하는 이유다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 최소 권한 원칙을 체화했는가: "일단 되게 하려고 AdministratorAccess를 붙였다가 나중에 줄이겠다"는 사고방식은 시니어 답변으로 인정받지 못한다. 처음부터 필요한 Action/Resource만 명시할 수 있는가를 본다.
  • 정책 평가 로직의 정확한 이해: Deny 우선, SCP와 Permission Boundary가 권한을 "주는 것"이 아니라 "제한하는 것"이라는 구분을 정확히 아는가. 이 개념을 헷갈리면 실무에서 "왜 관리자 권한을 줬는데도 안 되지?" 같은 장애를 디버깅하지 못한다.
  • Blast Radius(피해 범위) 사고: 하나의 자격증명이 유출됐을 때 공격자가 할 수 있는 최대 범위를 스스로 예측하고, 그 범위를 최소화하는 설계를 하는가.
  • 사고 대응 경험: Access Key 유출 같은 실제 인시던트를 겪어봤는지, 탐지-차단-복구의 순서를 체계적으로 말할 수 있는지를 본다.

이 질문은 "IAM 콘솔에서 정책을 어떻게 붙이는지"를 묻는 게 아니다. "신뢰를 어떻게 최소 단위로 쪼개서 부여하고, 그 신뢰가 깨졌을 때 피해를 어떻게 제한할 것인가"라는 권한 설계 사고를 묻는다. VPC의 네트워크 경계 설계(Day 2)와 정확히 짝을 이루는 주제다 — 하나는 "누가 들어올 수 있는가"이고, 다른 하나는 "들어온 뒤 무엇을 할 수 있는가"이다.

4. 실무 연결

시나리오 1 — GitHub에 노출된 Access Key로 인한 크립토마이닝 사고

신입 개발자가 로컬 테스트 편의를 위해 .env 파일에 AWS Access Key를 저장했는데, .gitignore에 등록하지 않고 실수로 공개 저장소에 커밋했다. 20분도 안 되어 자동화된 봇이 이를 탐지했고, 여러 리전에 걸쳐 대형 GPU 인스턴스가 대량으로 생성되어 청구서에 수천 달러가 찍혔다. 사고 후 팀은 IAM User의 장기 Access Key 발급을 원칙적으로 금지하고, 로컬 개발조차 aws sso 기반 임시 자격증명으로 전환했다. 또한 GitHub 저장소에 pre-commit 훅으로 시크릿 스캐너(git-secrets류)를 강제 적용했다.

시나리오 2 — 과도하게 권한을 준 EC2 Role이 SSRF로 탈취된 사고

웹 애플리케이션에 SSRF(Server-Side Request Forgery) 취약점이 있었고, 공격자가 이를 이용해 EC2 인스턴스 메타데이터 서비스(IMDS)에 접근해 그 인스턴스에 붙어 있던 Role의 임시 자격증명을 탈취했다. 문제는 그 Role에 AdministratorAccess에 가까운 광범위한 권한이 붙어 있었다는 점이었다 — "귀찮으니 일단 다 열어두고 나중에 줄이자"는 관행 때문이었다. 사고 이후 모든 EC2/Lambda Role에 필요한 Action과 Resource ARN을 구체적으로 명시하는 최소 권한 정책으로 전면 재작업했고, IMDSv2(토큰 기반 메타데이터 접근, SSRF로 단순 GET 요청만으로는 탈취 불가능하게 하는 방식)를 강제했다.

시나리오 3 — SCP 설정 실수로 조직 전체 계정이 잠긴 사고

보안 강화를 위해 루트 OU에 "특정 리전 밖의 모든 서비스 호출을 차단"하는 SCP를 적용했는데, 예외로 둬야 했던 IAM(글로벌 서비스)까지 실수로 포함시켰다. 배포 직후 여러 계정에서 IAM 관련 콘솔 작업 자체가 막혔고, 심지어 SCP를 되돌리는 작업(Organizations API 호출)까지 일부 영향을 받아 복구에 시간이 걸렸다. 이후 팀은 SCP 변경 시 반드시 소수의 테스트 계정에 먼저 적용하고, IAM/Organizations/STS 같은 글로벌·거버넌스 서비스는 기본적으로 SCP 리전 제한에서 예외 처리하는 표준을 만들었다.

시나리오 4 — ExternalId 없이 벤더에게 Cross-account Role을 내준 사고

모니터링 SaaS 벤더에게 리소스 조회용 Cross-account Role을 만들어줬는데, 급하게 처리하느라 Trust Policy에 ExternalId 조건을 넣지 않았다. 보안 감사에서 이 설정이 Confused Deputy 공격에 취약하다는 지적을 받았다 — 벤더가 동일한 Role ARN을 여러 고객에게 재사용하고 있었기 때문에, 이론적으로 악의적인 제3자가 벤더를 속여 이 계정에 접근할 경로가 존재했다. 모든 Cross-account Role Trust Policy에 벤더가 발급한 고유 ExternalId를 필수 조건으로 넣는 것을 표준 체크리스트에 추가했다.

5. 답변 사고 순서

면접에서 "권한을 어떻게 설계하시겠습니까"라는 질문을 받으면, 아래 순서로 생각하고 답변하라.

  1. Principal 파악

    요청 주체가 사람인지, EC2/Lambda 같은 AWS 서비스인지, 다른 AWS 계정인지 먼저 구분한다. 사람이면 User(가급적 SSO 연동 Federated 접근)를, 서비스나 애플리케이션이면 무조건 Role을 먼저 고려한다.

  2. 필요한 Action과 Resource를 구체적으로 나열

    "S3 접근 필요" 수준이 아니라 "이 버킷의 이 prefix에 대해 GetObject, PutObject만" 수준으로 구체화한다. 와일드카드(s3:*, Resource: *)는 최후의 수단이다.

  3. Identity Policy로 최소 권한 작성

    Role에 Identity-based Policy를 붙여 앞서 나열한 Action/Resource만 Allow한다. 필요하면 Condition으로 시간대, IP, MFA 여부 등을 추가 제약한다.

  4. Cross-account가 필요하면 Resource Policy + Trust Policy + ExternalId

    다른 계정이나 서드파티가 접근해야 한다면 Resource-based Policy(예: S3 버킷 정책) 또는 Role의 Trust Policy를 사용하고, Confused Deputy 방지를 위해 ExternalId를 반드시 포함한다.

  5. 조직 차원의 상한선 고려

    여러 계정/팀을 운영한다면 SCP로 조직 전체의 절대적 한도(예: 특정 리전만 허용)를, 위임 관리 시나리오라면 Permission Boundary로 개별 신원의 한도를 설정한다.

  6. 자격증명 유출 대비책을 함께 설계

    CloudTrail로 모든 API 호출을 기록하고, GuardDuty로 비정상 행위(평소 안 쓰던 리전에서의 활동 등)를 탐지한다. Access Key를 부득이하게 발급해야 한다면 정기 로테이션과 미사용 키 자동 비활성화 정책을 함께 둔다.

핵심 팁

"어떤 정책을 붙였는가"보다 "왜 그만큼만 허용했는가, 이 자격증명이 탈취되면 피해가 어디까지 번지는가"를 항상 함께 설명하라.

6. 핵심 답변 예시

면접 답변 예시 — EC2 애플리케이션의 S3/DynamoDB 접근 설계

"먼저 IAM User와 Access Key는 쓰지 않겠습니다. EC2 인스턴스에 IAM Role을 Instance Profile로 붙여서, 애플리케이션이 자격증명을 코드나 환경변수로 다루지 않고 EC2 메타데이터를 통해 자동으로 임시 자격증명을 받도록 하겠습니다. 이때 IMDSv2를 강제해서 SSRF로 메타데이터가 탈취되는 경로를 막습니다.

Role의 Identity Policy는 필요한 만큼만 씁니다. S3는 s3:GetObject, s3:PutObject를 특정 버킷의 특정 prefix ARN으로 한정하고, DynamoDB는 dynamodb:GetItem, dynamodb:PutItem, dynamodb:Query를 해당 테이블 ARN으로 한정합니다. Resource: "*"나 서비스 전체 와일드카드는 쓰지 않습니다.

만약 이 애플리케이션이 여러 팀의 리소스를 공유해서 접근해야 한다면, S3 버킷 정책(Resource-based Policy)으로 이 Role의 ARN만 명시적으로 허용하는 것도 함께 검토하겠습니다.

조직 차원에서는, 이 계정이 특정 리전 밖에서 리소스를 만들 수 없도록 SCP로 제한하는 것을 조직 관리자와 논의하겠습니다.

마지막으로, CloudTrail을 활성화해 이 Role이 실제로 어떤 API를 호출하는지 로그를 남기고, 나중에 실사용 패턴을 보고 처음에 부여한 권한 중 실제로 안 쓰는 것이 있으면(IAM Access Analyzer의 정책 생성 기능으로 확인 가능) 정책을 더 좁히는 과정을 반복하겠습니다."

이 답변이 좋은 이유: ① User/Access Key 대신 Role을 기본값으로 선택함 ② IMDSv2까지 언급해 SSRF 방어를 함께 고려함 ③ Action/Resource를 구체적으로 한정해 최소 권한을 실제로 적용함 ④ Cross-team 접근 시 Resource Policy 옵션까지 고려함 ⑤ SCP로 조직 차원의 상한선을 함께 언급함 ⑥ CloudTrail/Access Analyzer로 사후 권한 축소(권한을 지속적으로 좁혀가는) 프로세스까지 제시함

7. Senior-level Follow-up Q&A

Q1. "SCP와 Permission Boundary는 둘 다 '제한'인데, 실무에서 어떤 기준으로 구분해서 쓰시나요?"

SCP는 AWS Organizations 안에서 계정 단위 또는 OU 단위로 적용되는 조직 차원의 거버넌스 도구다. "이 사업부의 모든 계정은 특정 리전만 쓸 수 있다"처럼, 그 계정 안의 어떤 IAM 정책으로도 뚫을 수 없는 절대적 상한선을 정할 때 쓴다.

Permission Boundary는 계정 안의 특정 User나 Role 하나에 붙는 개별 상한선이다. 주로 위임 관리에 쓴다. 예를 들어 플랫폼팀이 각 서비스팀에게 "너희 팀 안에서 IAM Role을 자유롭게 만들어도 되지만, Permission Boundary 안에서만"이라고 위임하면, 서비스팀이 실수로 또는 의도적으로 자기 자신보다 강한 권한의 Role을 만드는 권한 상승(Privilege Escalation)을 막을 수 있다.

Q2. "Explicit Deny가 항상 Allow보다 우선한다고 하셨는데, 이 설계가 왜 실무에서 위험한 실수로 이어지나요?"

가장 흔한 실수는 "안전하게 하려고" 넓은 범위에 Deny를 걸었다가, 의도치 않게 필요한 접근까지 막는 것이다. 예를 들어 특정 팀의 실수를 막으려고 iam:*에 대한 Deny를 특정 조건 없이 걸었는데, 그 Deny가 SCP 레벨에 있어서 나중에 그 SCP를 만든 관리자 본인의 IAM 작업까지 막혀버리는 경우다.

Deny 정책을 쓸 때는 반드시 Condition으로 예외 범위를 명확히 하고(예: 특정 Role ARN은 예외), 배포 전에 영향 범위를 시뮬레이션(IAM Policy Simulator)으로 검증해야 한다. "일단 막고 보자" 식의 넓은 Deny는 SCP 실수 장애의 가장 흔한 원인이다.

Q3. "Access Key가 GitHub 공개 저장소에 노출된 것을 방금 발견했습니다. 어떤 순서로 대응하시겠습니까?"

순서가 중요하다. 첫째, 해당 Access Key를 즉시 비활성화(deactivate) 또는 삭제한다 — 조사보다 차단이 먼저다. 둘째, CloudTrail에서 그 Access Key로 수행된 모든 API 호출을 시간순으로 조회해서 어떤 리소스가 생성/변경/삭제됐는지 파악한다. 특히 새로 생성된 EC2 인스턴스, IAM User/Role, 비활성화된 로깅 설정이 있는지 우선 확인한다.

셋째, 발견된 악의적 리소스(크립토마이닝 인스턴스 등)를 정리하고, 공격자가 추가로 만들었을 수 있는 백도어성 IAM User나 Role이 없는지 전체 계정을 감사한다. 넷째, GitHub 커밋 히스토리에서 해당 키를 완전히 제거(git filter-repo 등)하고, 재발 방지를 위해 pre-commit 훅 기반 시크릿 스캐너와 GuardDuty를 상시 활성화한다. 마지막으로 사고 원인과 대응을 문서화해 팀에 공유한다.

Q4. "왜 EC2/Lambda에는 User 대신 Role을 쓰라고 하는데, 그런데도 여전히 Access Key를 발급해야 하는 경우가 있나요?"

Role은 AWS 내부(EC2, Lambda, ECS 등)에서 실행되는 워크로드나, AssumeRole을 지원하는 주체(다른 AWS 계정, SSO 연동 사용자)에게만 유효하다. 문제는 AWS 밖에서 실행되는 프로그램이 AWS API를 호출해야 하는 경우다 — 예를 들어 온프레미스 서버의 배치 스크립트나, AWS를 지원하지 않는 레거시 CI 도구.

이런 경우에도 가능하면 IAM Roles Anywhere(X.509 인증서 기반으로 온프레미스에서도 임시 자격증명을 받는 기능)나 OIDC 연동(GitHub Actions의 경우 OIDC로 Role을 assume하는 방식)을 우선 검토한다. 정말 불가피하게 장기 Access Key를 써야 한다면, 최소 권한 + 정기 로테이션(예: 90일) + 미사용 키 자동 감지(IAM Access Analyzer, Credential Report)를 반드시 함께 운영해야 한다.

Q5. "Cross-account Role에 ExternalId가 왜 필요한가요? 구체적으로 어떤 공격을 막나요?"

Confused Deputy Problem을 막기 위해서다. 예를 들어 모니터링 SaaS 벤더가 자기 계정에서 여러 고객의 계정으로 Role을 assume하는 구조를 생각해보자. 벤더가 고객마다 별도의 Role ARN을 만들지 않고 동일한 중개 Role을 재사용한다면, 벤더의 시스템 자체를 신뢰하는 것과 별개로, 벤더에게 "고객 A의 Role을 assume해줘"라고 요청만 하면 그 요청이 실제로 고객 A가 보낸 것인지 벤더 입장에서는 구분하기 어려운 상황이 생길 수 있다.

ExternalId는 내가 벤더에게만 알려준 고유한 비밀 문자열을 Trust Policy의 Condition에 넣어서, AssumeRole 요청에 그 값이 정확히 포함되지 않으면 거부하게 만드는 장치다. 이렇게 하면 다른 고객의 정보를 안다고 해서 내 계정의 Role을 assume할 수 없다 — 벤더가 실수하거나 속더라도 ExternalId를 모르면 내 계정은 안전하다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 서비스/애플리케이션에는 기본적으로 Role을 쓰고, User와 장기 Access Key는 최후의 수단으로 취급한다
  • Action과 Resource를 구체적인 ARN 단위로 한정하고, 와일드카드 사용을 스스로 지양한다
  • SCP와 Permission Boundary가 "권한을 주는 것"이 아니라 "상한선을 긋는 것"이라는 점을 정확히 구분한다
  • Cross-account 접근에서 ExternalId, Confused Deputy 같은 구체적 위험을 스스로 짚는다
  • 자격증명 유출 시 탐지-차단-조사-복구의 순서를 체계적으로 설명한다
약한 답변
  • "일단 필요한 권한을 다 주고 나중에 줄이면 됩니다"라고 말한다
  • User, Group, Role을 "다 비슷한 거 아닌가요"로 뭉뚱그린다
  • SCP를 "IAM 정책이랑 비슷한 건데 조직 레벨인 것" 정도로만 설명하고 권한을 주는지 제한하는지 구분 못 한다
  • Access Key 유출 대응을 "키를 바꾼다" 한 문장으로 끝낸다
  • Cross-account 접근을 그냥 "Role을 assume하게 하면 됩니다"로만 답하고 Confused Deputy를 모른다
왜 약한 답변이 약한가

"일단 필요한 권한을 다 준다"는 접근은 개발 속도만 본 판단이고, 이 신원이 탈취됐을 때의 피해 범위를 전혀 고려하지 않은 것이다. 시니어 면접에서는 처음부터 최소한으로 설계하고, 실사용 패턴을 관찰하며 점진적으로 조정하는 프로세스 자체를 설계할 수 있어야 한다. 권한 설계는 "한 번 설정하고 끝"이 아니라 지속적으로 좁혀가는 운영 프로세스라는 것을 이해하고 있는지가 핵심이다.

9. 흔한 오해

오해 1

"Role은 User보다 그냥 조금 더 편리한 기능일 뿐이다."

실제

Role의 핵심 가치는 편의성이 아니라 임시성이다. STS가 발급하는 자격증명은 짧은 시간(기본 1시간) 후 자동으로 만료되므로, 탈취되더라도 피해가 시간적으로 제한된다. User의 장기 Access Key는 로테이션하지 않는 한 영구히 유효하다는 근본적인 차이가 있다.

오해 2

"SCP를 계정에 붙이면 그 계정에 해당 권한이 생긴다."

실제

SCP는 권한을 주지 않는다. 그 계정 안에서 IAM 정책으로 실제 부여된 권한의 상한선일 뿐이다. SCP로 s3:*를 허용해도, 그 계정 안의 어떤 User/Role에도 S3 Identity Policy가 없다면 아무도 S3에 접근할 수 없다. 반대로 SCP가 특정 서비스를 막으면, 계정 안에 아무리 강력한 IAM 정책이 있어도 그 서비스는 쓸 수 없다.

오해 3

"Deny 정책이 없으면 기본적으로 다 막혀있으니 안전하다, 굳이 Deny를 쓸 필요가 없다."

실제

암묵적 거부(Implicit Deny)가 기본값인 것은 맞지만, 명시적 Deny가 필요한 상황이 분명히 있다. 예를 들어 "특정 Role은 Allow 정책이 광범위하게 걸려 있지만, MFA 없이는 절대 삭제 작업을 못 하게" 만들고 싶을 때는 Condition이 붙은 명시적 Deny로 예외를 만들어야 한다. Deny는 넓은 Allow 위에 정교한 예외를 두는 도구로도 쓰인다.

오해 4

"Access Key를 로테이션(재발급)하기만 하면 유출 사고 대응은 끝난 것이다."

실제

키를 바꾸는 것은 재발을 막는 조치일 뿐, 이미 유출된 동안 무엇을 당했는지는 별개로 조사해야 한다. 공격자가 그 키로 새로운 IAM User나 Role, 백도어성 Lambda를 만들어뒀다면 키만 바꿔서는 그 백도어가 그대로 남는다. CloudTrail 로그로 유출 기간 동안의 모든 활동을 반드시 추적해야 한다.

오해 5

"루트 계정은 평소에 로그인하지 않으니 보안 설정을 신경 쓰지 않아도 된다."

실제

루트 계정은 어떤 IAM 정책이나 SCP로도 제한할 수 없는 계정 안의 최상위 권한을 가진다. 평소에 안 쓰더라도 MFA를 반드시 활성화하고, 루트 계정의 Access Key는 애초에 발급하지 않거나 즉시 삭제해야 한다. 루트 계정이 뚫리면 계정 전체가 뚫린 것과 같다 — "잘 안 쓰니까 안전하다"는 정반대로, "잘 안 쓰는데 뚫리면 늦게 발견된다"는 위험으로 봐야 한다.
