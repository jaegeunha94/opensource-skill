2. 전제 개념 — 쉬운 한국어로

S3의 기본 모델 — "폴더처럼 보이지만 사실은 키-값 저장소"

S3는 버킷(Bucket)이라는 이름공간 안에 객체(Object)를 저장하는 서비스다. 각 객체는 images/2026/profile.png 같은 키(Key) 하나로 식별된다. 콘솔에서는 이게 폴더 구조처럼 보이지만, 실제로는 슬래시(/)를 포함한 문자열 키를 가진 평면적인 키-값 저장소다. "폴더 권한"이라는 개념이 파일시스템처럼 따로 존재하지 않고, 정책에서 images/2026/* 같은 키 프리픽스(prefix) 패턴으로 범위를 지정하는 것뿐이다. 이 차이를 모르면 "특정 폴더만 공개하고 싶다"는 요구를 정책으로 어떻게 표현해야 하는지 헤매게 된다.

기본 원칙 — S3는 기본적으로 완전히 비공개다

새로 만든 S3 버킷과 그 안의 객체는 버킷 소유자를 제외한 누구에게도 접근 권한이 없다. 접근을 허용하려면 명시적으로 권한을 부여하는 정책을 작성해야 한다. 이 "기본값이 비공개"라는 전제를 먼저 말하고 시작하는 것이 시니어 답변의 출발점이다 — 실무에서 벌어지는 사고 대부분은 이 기본값을 누군가 의도적으로 또는 실수로 풀어버렸을 때 발생한다.

권한을 결정하는 네 가지 레이어

S3 접근 제어는 서로 다른 성격의 네 가지 레이어가 겹쳐서 최종 결과를 만든다. 이 레이어들을 구분하지 못하면 "정책을 분명히 맞게 썼는데 왜 접근이 안 되지"라는 상황을 진단할 수 없다.

IAM 정책 (Identity-based)
IAM 사용자/역할에 붙는 정책. "이 사람/역할이 어떤 S3 액션을 할 수 있는가"를 정의한다. 같은 계정 내부 사용자 권한 관리의 기본 도구.
버킷 정책 (Resource-based)
버킷 자체에 붙는 정책. "이 버킷에 누가(다른 AWS 계정 포함) 접근할 수 있는가"를 정의한다. 계정 경계를 넘는 접근(cross-account)을 표현할 수 있는 유일한 방법 중 하나.
ACL (Access Control List)
객체/버킷 단위의 레거시 접근 제어 방식. AWS는 신규 버킷에서 "Bucket owner enforced" 설정으로 ACL을 사실상 비활성화할 것을 권장한다.
Block Public Access
계정/버킷 레벨의 안전장치. 다른 정책이 퍼블릭 접근을 허용하더라도, 이 설정이 켜져 있으면 실제 퍼블릭 접근을 강제로 차단한다.
권한 평가 원칙 — 명시적 Deny가 항상 이긴다

여러 정책(IAM 정책, 버킷 정책, Block Public Access, SCP 등)이 동시에 존재할 때, AWS는 기본적으로 명시적 거부(Explicit Deny)가 있으면 다른 모든 허용(Allow)보다 우선해서 최종 접근을 차단한다. 반대로 어느 정책에도 허용이 없으면 기본값은 거부다. Block Public Access는 이 원칙을 이용해 "정책 실수로 퍼블릭이 허용되더라도 마지막 방어선에서 강제로 막는" 안전장치로 동작한다 — 그래서 버킷 정책만 정확히 쓰는 것과 별개로 Block Public Access를 켜두는 것이 심층 방어(defense in depth)다.

ACL은 왜 레거시가 되었나

ACL은 객체 하나하나에 개별적으로 권한을 붙일 수 있어 유연해 보이지만, 버킷 안 수많은 객체의 ACL이 제각각이면 전체 접근 범위를 한눈에 파악하기 어렵다는 근본적 단점이 있다. 반면 버킷 정책은 버킷 단위로 한 곳에서 관리되므로 감사와 추론이 훨씬 쉽다. 그래서 AWS는 새 버킷의 기본값을 Object Ownership: Bucket owner enforced로 두어 ACL을 사실상 끄고, cross-account 접근도 버킷 정책이나 IAM 정책만으로 표현하도록 권장한다. 인터뷰에서 "ACL을 아직도 쓰나요?"라는 질문의 기대 답변은 "레거시 호환을 위해 남아 있지만, 신규 설계에서는 버킷 정책/IAM 정책으로 대체하고 ACL은 비활성화한다"는 것이다.

정적 웹사이트 호스팅 — "버킷을 공개하지 않고도 웹사이트를 서비스할 수 있다"

S3는 정적 웹사이트 호스팅(Static Website Hosting) 기능으로 HTML/CSS/JS를 직접 서빙할 수 있지만, 이 기능을 켜서 만드는 웹사이트 엔드포인트는 HTTP만 지원하고, 사용하려면 버킷 자체(또는 해당 객체)가 퍼블릭 read 권한을 가져야 한다. 이는 곧 버킷을 공개해야 한다는 뜻이라 보안 요구가 있는 서비스에는 부적합하다.

더 나은 패턴 — CloudFront + OAC(Origin Access Control)

실무 기본형은 CloudFront를 S3 앞에 두고, OAC(Origin Access Control)를 이용해 CloudFront만 버킷에 접근하도록 제한하는 것이다. 이렇게 하면 (1) 버킷은 계속 완전히 비공개로 유지되고, (2) CloudFront가 HTTPS 종단, 캐싱, 글로벌 배포를 담당하며, (3) 버킷 정책에는 "이 CloudFront 배포에서만 오는 요청을 허용"하는 조건만 존재하면 된다. 사용자는 절대 S3 엔드포인트에 직접 접근하지 않고 항상 CloudFront를 거친다. (참고: OAC는 레거시 OAI(Origin Access Identity)를 대체하는 최신 방식이며, 신규 설계에서는 OAC를 사용하는 것이 표준이다.)

버전 관리(Versioning) — "삭제는 지우는 게 아니라 표시를 남기는 것"

버킷에서 버전 관리를 켜면, 같은 키에 새 객체를 업로드해도 이전 버전이 사라지지 않고 별도 버전으로 남는다. 더 중요한 것은 삭제 동작의 의미가 바뀐다는 점이다 — 일반 삭제 요청은 실제로 데이터를 지우는 것이 아니라 삭제 마커(Delete Marker)라는 특별한 버전을 새로 추가할 뿐이다. 삭제 마커가 최신 버전이 되면 일반 조회에서는 "파일이 없는 것처럼" 보이지만, 실제 데이터는 이전 버전으로 여전히 존재하고 삭제 마커를 지우면 복구된다.

더 강력한 보호가 필요하면 MFA Delete를 함께 활성화해, 버전을 영구히 삭제하거나 버전 관리 자체를 끄는 작업에 루트 계정의 MFA 인증을 요구할 수 있다. 이는 공격자나 실수한 운영자가 "삭제 마커까지 지워서 완전 삭제"하는 것을 어렵게 만든다.

수명 주기 정책(Lifecycle Policy) — "시간이 지난 데이터를 자동으로 값싼 계층으로 옮긴다"

대부분의 객체는 시간이 지날수록 접근 빈도가 떨어진다(최근 업로드 이미지는 자주 조회되지만, 1년 지난 로그는 거의 조회되지 않는다). 수명 주기 정책은 이 접근 패턴 변화에 맞춰 객체를 자동으로 저렴한 스토리지 클래스로 전환(Transition)하거나, 일정 기간 후 만료(Expiration)시켜 삭제하는 규칙이다. 버전 관리를 켠 버킷에서는 이전 버전(Noncurrent Version)에도 별도의 수명 주기 규칙을 걸 수 있어, "버전 관리는 켜두되 오래된 버전은 자동으로 정리"하는 설계가 가능하다. 또한 미완료 멀티파트 업로드(Incomplete Multipart Upload)가 정리되지 않고 남아 과금되는 것을 막는 규칙도 수명 주기 정책의 일부다.

암호화 — SSE-S3 vs SSE-KMS

S3는 기본적으로 서버 측 암호화(SSE)를 적용한다. SSE-S3는 AWS가 완전히 관리하는 키로 암호화해 설정이 간단하지만, 키 사용 이력을 세밀하게 감사하거나 키 접근 권한을 별도로 통제하기 어렵다. SSE-KMS는 KMS(Key Management Service)의 고객 관리형 키를 사용해, 키 자체에 대한 IAM 권한을 별도로 걸고, 모든 키 사용 이력을 CloudTrail로 감사할 수 있다. 규제 산업이나 "누가 언제 이 데이터를 복호화했는지 증명해야 하는" 요구가 있으면 SSE-KMS가 기본 선택이다. 다만 KMS API 호출량이 많아지면 요청당 비용이 추가된다는 trade-off가 있다.

Presigned URL — "임시로 열어주는 뒷문"

버킷을 비공개로 유지하면서 특정 사용자에게 특정 객체만 일시적으로 접근시키고 싶을 때 쓰는 것이 Presigned URL이다. 서버가 자신의 IAM 자격 증명으로 서명을 생성해 "이 URL은 N분 동안만 이 객체에 접근 가능"이라는 임시 URL을 발급한다. 중요한 점은 이 URL을 가진 사람은 누구든(추가 인증 없이) 만료 전까지 접근할 수 있다는 것이다 — URL이 로그, 브라우저 히스토리, 채팅으로 유출되면 그 URL 자체가 접근 키가 된다. 그래서 만료 시간을 실제 필요한 최소 시간으로 좁히는 것이 설계의 핵심이다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 권한 모델을 계층으로 이해하는가: IAM 정책(누가), 버킷 정책(어느 리소스에, 계정 경계 포함), ACL(레거시), Block Public Access(안전장치)를 각각의 역할로 구분해서 설명할 수 있는가.
  • Private-by-default와 심층 방어 감각: "정책만 정확히 쓰면 된다"가 아니라, 실수를 가정하고 Block Public Access 같은 마지막 방어선을 함께 설계하는가.
  • 퍼블릭 접근 요구를 안전하게 해결하는 능력: 정적 자산을 서비스해야 할 때 "버킷을 공개"가 아니라 CloudFront+OAC로 우회하는 패턴을 아는가.
  • 데이터 보호와 복구 가능성: 애플리케이션/운영 실수로 인한 덮어쓰기·삭제를 버전 관리로 어떻게 되돌릴 수 있는지 설명할 수 있는가.
  • 비용 감각: 스토리지 클래스와 수명 주기 정책을 접근 패턴에 근거해 선택하고, "무조건 싼 클래스"가 답이 아님을 아는가.

EC2/Auto Scaling(Day 4)과 RDS(Day 5)가 각각 컴퓨트와 정형 데이터 계층을 다뤘다면, S3(Day 6)는 대규모 비정형 객체 데이터를 다루는 계층이다. 이 계층의 사고는 대부분 "인프라가 다운돼서"가 아니라 "권한 설정 실수로 데이터가 노출되어서" 발생한다는 점에서 앞선 주제들과 평가 축이 다르다. 인터뷰어는 가용성 설계 능력뿐 아니라 접근 제어를 계층적으로 사고하는 보안 판단력을 이 주제에서 별도로 평가한다.

4. 실무 연결

시나리오 1 — 버킷 정책 실수로 인한 개인정보 노출 사고

한 팀이 파트너사와의 데이터 공유를 위해 버킷 정책에 Principal: "*"로 읽기 권한을 부여했다가, 실제로는 해당 프리픽스에 개인정보가 포함된 로그 파일이 섞여 있었고 검색 엔진에 크롤링되어 노출됐다. 사고 이후 팀은 계정 전체에 Block Public Access를 기본 활성화하고, 예외가 필요한 특정 버킷만 명시적으로 검토·승인하는 프로세스를 도입했다. 파트너사 접근은 Principal을 특정 AWS 계정 ID로 한정하는 cross-account 버킷 정책으로 재설계했다.

시나리오 2 — S3 정적 호스팅에서 CloudFront + OAC로 전환

초기에는 마케팅 랜딩 페이지를 S3 정적 웹사이트 호스팅 엔드포인트로 직접 서비스했다. 이후 HTTPS가 필수 요구사항이 되고, 트래픽 급증 시 S3 요청 비용이 예측 어려운 문제가 겹치자 CloudFront를 앞에 두고 OAC로 전환했다. 전환 후 버킷은 완전히 비공개로 유지되고, CloudFront 캐싱 덕분에 원본 S3 요청 수 자체가 크게 줄어 비용도 함께 낮아졌다.

시나리오 3 — 버전 관리가 꺼져 있어 발생한 복구 불가 사고

배포 스크립트의 버그로 사용자 프로필 이미지 버킷의 파일들이 잘못된 빈 파일로 일괄 덮어써졌다. 버전 관리가 꺼져 있던 탓에 이전 파일을 복구할 방법이 없어, 결국 별도로 보관 중이던 오래된 백업에서 일부만 복구할 수 있었다. 사고 이후 모든 사용자 데이터 버킷에 버전 관리를 기본 활성화하고, 오래된 버전은 수명 주기 정책으로 90일 후 Glacier로 이동, 1년 후 만료되도록 비용을 관리했다.

시나리오 4 — 수명 주기 정책 없이 쌓인 로그 스토리지 비용

애플리케이션 로그를 S3 Standard에 무기한 적재하고 있었고, 2년이 지나자 스토리지 비용이 팀 예산에서 눈에 띌 만큼 커졌다. 대부분의 로그는 생성 후 한 달만 지나면 조회가 거의 없었다. 분석 후 30일이 지난 로그는 Standard-IA로, 180일이 지나면 Glacier Deep Archive로 전환하고, 3년이 지나면 만료시키는 수명 주기 정책을 도입해 스토리지 비용을 크게 절감했다.

시나리오 5 — 만료 시간이 긴 Presigned URL의 유출

사용자 다운로드 기능에서 Presigned URL의 만료 시간을 편의상 7일로 길게 설정했는데, 해당 URL이 공유 링크로 복사되어 의도치 않은 사람들에게 퍼지면서 만료 전까지 누구나 파일에 접근할 수 있었다. 이후 만료 시간을 실제 다운로드에 필요한 수 분 단위로 줄이고, 다운로드 페이지에 접근할 때마다 새 URL을 발급하는 방식으로 바꿨다.

5. 답변 사고 순서

면접에서 "S3 접근 제어와 비용을 어떻게 설계하시겠습니까"라는 질문을 받으면, 아래 순서로 생각하고 답변하라.

  1. 기본 원칙 선언: private-by-default와 Block Public Access

    버킷은 기본적으로 비공개이며, 계정 레벨에서 Block Public Access를 활성화하는 것을 출발점으로 삼는다고 먼저 말한다.

  2. 누가 접근해야 하는지에 따라 정책 종류를 선택

    같은 계정 내부 사용자/역할이면 IAM 정책, 다른 AWS 계정이나 특정 조건부 접근(예: 특정 VPC 엔드포인트에서만)이 필요하면 버킷 정책을 사용한다. ACL은 사용하지 않고 Bucket owner enforced로 비활성화한다.

  3. 퍼블릭 접근이 필요한 경우 CloudFront + OAC로 우회

    정적 자산을 대중에게 서비스해야 한다면 버킷을 공개하지 않고 CloudFront + OAC로 앞단을 감싼다. 사용자 개인 파일처럼 특정 사용자에게만 임시로 열어줘야 하면 Presigned URL을 짧은 만료 시간으로 발급한다.

  4. 데이터 보호: 버전 관리와 삭제 방지

    실수로 인한 덮어쓰기·삭제에 대비해 버전 관리를 켜고, 중요 데이터에는 MFA Delete를 추가로 검토한다.

  5. 비용 최적화: 접근 패턴 기반 수명 주기 정책

    접근 빈도가 떨어지는 데이터를 Standard-IA/Glacier 계열로 자동 전환하고, 이전 버전과 미완료 멀티파트 업로드도 정리 규칙에 포함한다. 접근 패턴이 불규칙하면 Intelligent-Tiering을 검토한다.

  6. 암호화와 감사

    규제·감사 요구가 있으면 SSE-KMS로 키 사용 이력을 CloudTrail에 남기고, 접근 로그(S3 Server Access Logging 또는 CloudTrail data event)로 누가 언제 접근했는지 추적 가능하게 만든다.

핵심 팁

"버킷 정책을 이렇게 쓰겠습니다"로 끝내지 말고, "정책 실수가 있더라도 어떤 안전장치가 마지막으로 막아주는지"를 함께 설명하라. 그것이 심층 방어를 실제로 설계에 반영하고 있다는 신호다.

6. 핵심 답변 예시

면접 답변 예시 — 사용자 업로드 파일과 정적 자산을 다루는 S3 설계

"먼저 이 버킷은 기본적으로 완전히 비공개라는 전제에서 시작하겠습니다. 계정 레벨에서 Block Public Access를 활성화해두고, 이후 어떤 정책 실수가 있어도 실제 퍼블릭 노출로 이어지지 않도록 마지막 방어선을 만들어두겠습니다.

접근 권한은 성격별로 다르게 설계하겠습니다. 내부 애플리케이션 서버가 이 버킷에 쓰기/읽기를 하는 경우는 해당 서버의 IAM 역할에 최소 권한 IAM 정책을 붙이고, 파트너사처럼 다른 AWS 계정에서 특정 프리픽스만 읽어야 하는 경우는 버킷 정책에 그 계정 ID를 명시한 cross-account 정책으로 한정하겠습니다. ACL은 사용하지 않고 Object Ownership을 Bucket owner enforced로 설정해 아예 비활성화하겠습니다.

정적 웹 자산처럼 불특정 다수에게 공개해야 하는 콘텐츠는 버킷 자체를 공개하지 않고, CloudFront를 앞에 두고 OAC로 CloudFront만 버킷에 접근하도록 제한하겠습니다. 반면 로그인한 사용자에게 자신의 파일만 다운로드하게 해야 하는 경우는 요청 시점에 서버가 짧은 만료 시간(예: 5분)의 Presigned URL을 발급하는 방식을 쓰겠습니다.

데이터 보호를 위해 버전 관리를 활성화해서 애플리케이션 버그나 운영 실수로 인한 덮어쓰기·삭제가 발생해도 이전 버전으로 복구할 수 있게 하겠습니다. 다만 버전이 무한정 쌓이면 비용이 늘어나므로, 이전 버전은 90일 후 Glacier로 전환하고 1년 후 만료되도록 수명 주기 정책을 함께 설정하겠습니다. 미완료 멀티파트 업로드도 자동 정리 규칙에 포함하겠습니다.

비용 최적화 관점에서는 접근 패턴을 기준으로 스토리지 클래스를 계층화하겠습니다. 최근 업로드된 활성 데이터는 Standard에 두고, 30일 이상 조회가 없는 데이터는 Standard-IA로, 180일 이상 지난 데이터는 Glacier로 전환하겠습니다. 접근 패턴을 예측하기 어려운 데이터셋은 Intelligent-Tiering을 사용해 자동으로 계층을 맡기겠습니다.

마지막으로 규제 요구가 있는 데이터는 SSE-KMS로 암호화해서 키 사용 이력을 CloudTrail로 감사 가능하게 만들고, 그렇지 않은 일반 데이터는 SSE-S3로 충분하다고 판단하겠습니다."

이 답변이 좋은 이유: ① private-by-default와 Block Public Access를 심층 방어로 먼저 선언함 ② IAM 정책과 버킷 정책의 쓰임을 접근 주체(내부 vs cross-account)에 따라 명확히 구분함 ③ 퍼블릭 요구를 버킷 공개가 아닌 CloudFront+OAC와 Presigned URL로 우회함 ④ 버전 관리와 수명 주기 정책을 "보호"와 "비용"이라는 두 목표로 함께 설계함 ⑤ 암호화 방식도 데이터 성격(규제 여부)에 따라 차등 적용함

7. Senior-level Follow-up Q&A

Q1. "버킷 정책만 정확히 작성하면 되는데, 왜 Block Public Access까지 켜야 하나요? 이중 설정 아닌가요?"

정책이 항상 "정확히" 작성된다는 전제가 사고의 시작점이다. 실무에서는 CI/CD 자동화 스크립트의 버그, 급하게 붙인 예외 조건, 신입 엔지니어의 실수 등으로 정책이 의도치 않게 넓어지는 일이 반복적으로 발생한다.

Block Public Access는 "정책이 틀렸을 가능성"을 가정하고 마지막에 한 번 더 막아주는 심층 방어 계층이다. 이중 설정처럼 보이지만, 보안 설계에서는 단일 지점의 실수가 곧바로 사고로 이어지지 않도록 여러 계층을 겹치는 것이 표준적인 접근이다.

Q2. "정적 웹사이트를 CloudFront 없이 S3 단독으로 운영하면 안 되나요?"

가능은 하지만 세 가지 제약이 있다. 첫째, S3 웹사이트 엔드포인트는 HTTPS를 지원하지 않아 별도 방법 없이는 평문 HTTP로만 서비스된다. 둘째, 버킷 자체가 퍼블릭 read 권한을 가져야 하므로 비공개 유지가 불가능하다. 셋째, 캐싱 계층이 없어 트래픽이 몰릴 때 S3 요청 비용과 지연이 그대로 사용자에게 전달된다.

CloudFront + OAC는 이 세 가지를 모두 해결하면서 글로벌 엣지 캐싱까지 얻을 수 있어, HTTPS나 트래픽 규모가 조금이라도 고려되는 서비스라면 기본 선택지로 두는 것이 맞다.

Q3. "Presigned URL의 만료 시간은 어떤 기준으로 정하나요?"

기준은 "이 URL이 유출됐을 때 노출되는 시간 창을 최소화"하는 것이다. 사용자가 버튼을 눌러 즉시 다운로드하는 용도라면 수 분(예: 5분) 이내로 충분하고, 배치 작업이 파일을 처리하는 데 시간이 걸린다면 그 작업 소요 시간에 맞춰 조금 더 길게 잡되 필요 이상으로 늘리지 않는다.

또한 URL을 발급한 IAM 자격 증명의 유효 기간보다 길게 설정할 수 없다는 제약도 있으므로, 임시 자격 증명(STS)으로 발급하는 경우 그 세션 만료 시간도 함께 고려해야 한다.

Q4. "버전 관리를 켜면 스토리지 비용이 끝없이 늘어나지 않나요?"

맞는 우려다. 버전 관리 자체는 이전 버전을 삭제하지 않으므로 관리하지 않으면 비용이 누적된다. 하지만 이는 버전 관리를 끄는 이유가 아니라, 수명 주기 정책을 함께 설계해야 하는 이유다.

수명 주기 규칙에서 "이전 버전(Noncurrent Version)"을 대상으로 별도 전환/만료 규칙을 걸 수 있으므로, 예를 들어 "이전 버전은 30일 후 Glacier로, 180일 후 완전 삭제"처럼 설정하면 데이터 보호와 비용 관리를 동시에 달성할 수 있다.

Q5. "Intelligent-Tiering과 수동으로 수명 주기 정책을 짜는 것 중 어느 쪽을 선택하나요?"

접근 패턴을 예측할 수 있는지가 기준이다. 로그처럼 "생성 후 시간이 지날수록 조회가 줄어든다"는 패턴이 명확하면, 수동 수명 주기 정책이 모니터링 비용 없이 더 저렴하게 동작한다.

반대로 사용자별 콘텐츠처럼 어떤 객체가 언제 다시 조회될지 예측하기 어려운 데이터셋에는 Intelligent-Tiering이 적합하다. 객체당 소액의 모니터링 비용이 들지만, 실제 접근 여부를 관찰해 자동으로 계층을 옮겨주므로 수동 규칙보다 총비용이 낮아지는 경우가 많다. 두 방식을 데이터셋 성격별로 혼용하는 것이 현실적인 답이다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • IAM 정책(내부 주체)과 버킷 정책(cross-account/리소스 기반)의 쓰임을 구분해서 설명한다
  • Block Public Access를 심층 방어로 설명하고, 정책만으로 충분하다고 말하지 않는다
  • 퍼블릭 자산도 버킷을 공개하지 않고 CloudFront+OAC로 우회하는 패턴을 안다
  • 버전 관리와 수명 주기 정책을 "보호"와 "비용"이라는 두 목표로 함께 설계한다
  • 스토리지 클래스 선택을 접근 빈도/지연 허용치 근거로 설명한다
약한 답변
  • "버킷 정책을 잘 쓰면 된다"로 끝내고 Block Public Access를 언급하지 않는다
  • ACL, 버킷 정책, IAM 정책을 구분 없이 "권한 설정"으로 뭉뚱그린다
  • 정적 자산을 공개하려면 버킷을 퍼블릭으로 만들어야 한다고 단정한다
  • 버전 관리를 켜라고만 하고 비용 관리(수명 주기)를 언급하지 않는다
  • 모든 데이터를 무조건 가장 저렴한 스토리지 클래스로 옮기자고 제안한다
왜 약한 답변이 약한가

"정책만 잘 쓰면 된다"는 답은 실무에서 정책은 언젠가 실수로 잘못 작성된다는 현실을 반영하지 못한 것이다. 실제 S3 노출 사고의 상당수는 "정책 문법이 틀려서"가 아니라 "의도보다 넓은 범위를 허용해서" 발생한다. 시니어는 정책을 정확히 쓰는 능력과, 정책이 틀렸을 때도 사고로 이어지지 않게 만드는 방어 계층 설계를 함께 갖춰야 한다.

"버킷 정책에서 퍼블릭 접근을 허용하지 않았으니 이 버킷은 절대 노출되지 않는다."

실제

ACL 설정, IAM 정책의 넓은 권한, 잘못된 cross-account 설정 등 여러 경로로 의도치 않은 노출이 발생할 수 있다. Block Public Access를 계정 레벨에서 켜두는 것이 마지막 안전장치다.

오해 2

"정적 웹사이트를 서비스하려면 버킷을 반드시 퍼블릭으로 만들어야 한다."

실제

S3 웹사이트 호스팅 엔드포인트를 직접 쓰는 경우에만 해당한다. CloudFront + OAC를 쓰면 버킷을 완전히 비공개로 유지하면서도 대중에게 콘텐츠를 서비스할 수 있다.

오해 3

"버전 관리를 켜두면 실수로 삭제해도 파일이 그 자리에 그대로 남아 있다."

실제

삭제 요청은 실제로는 삭제 마커를 추가하는 것이라, 일반 조회에서는 파일이 "없는 것처럼" 보인다. 데이터 자체는 이전 버전으로 남아 있지만, 삭제 마커를 제거하거나 특정 버전을 지정해 조회해야 복구할 수 있다.

오해 4

"Presigned URL은 발급 후 누가 접근하든 서버가 요청마다 권한을 다시 검증하므로 안전하다."

실제

Presigned URL은 발급 시점에 서명이 고정되며, 만료 전까지는 URL을 가진 누구든 추가 인증 없이 접근할 수 있다. URL 자체가 곧 접근 권한이 되므로 유출 경로(로그, 브라우저 히스토리, 공유 링크)에 유의해 만료 시간을 최소화해야 한다.

오해 5

"비용을 줄이려면 오래된 데이터를 무조건 가장 저렴한 Glacier Deep Archive로 옮기는 것이 정답이다."

실제

Glacier Deep Archive는 저장 비용은 최저지만 조회에 수 시간이 걸리고 조회 자체에도 비용이 든다. 실제로 다시 조회될 가능성이 있는 데이터를 이곳으로 옮기면 총비용과 대응 속도 모두 악화될 수 있다. 접근 패턴을 먼저 분석하고 계층을 선택해야 한다.
