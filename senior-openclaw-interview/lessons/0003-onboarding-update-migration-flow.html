Day 3 · OpenClaw Interview

온보딩과 업데이트/마이그레이션 흐름

"migrate가 dry-run과 backup을 지원하니 업그레이드도 항상 안전하다" — 이 문장이 왜 절반만 맞는지 구조적으로 설명하는 능력
버전/시점 기준: 이 레슨은 2026-07 기준 공식 문서(docs.openclaw.ai/cli/onboard, docs.openclaw.ai/cli/migrate, docs.openclaw.ai/plugins/sdk-migration, docs.openclaw.ai/gateway/doctor)와 GitHub 저장소(openclaw/openclaw)의 이슈·체인지로그를 교차 검증해 작성했다. 플래그 이름, 버전 번호, 이슈 번호는 리서치 시점 기준이며 실제 면접 전 최신 문서/저장소로 재확인한다. 이 레슨의 핵심 주장 — "`migrate`는 강력한 안전망을 제공하지만 그 범위는 다른 도구에서의 import에 한정되고, 버전 간 config 스키마 변경에는 같은 수준의 안전망이 없다" — 은 GitHub 이슈(#35957, #38249)로 직접 확인된 내용이다.
면접 질문
Q1 — 주력 질문 "OpenClaw를 새로 설치할 때의 온보딩 흐름과, 이미 운영 중인 Gateway를 새 버전으로 업그레이드할 때의 마이그레이션 흐름을 설명해주세요. 이 두 흐름이 실제로 '안전망'을 제공하는 지점과, 제공하지 않는 지점은 어디입니까?"
Q2 — 변형 "`openclaw migrate`가 dry-run과 자동 백업을 지원한다고 들었습니다. 그러면 프로덕션 Gateway를 마이너 버전 업그레이드할 때도 이 도구가 위험을 막아줍니까?"
Q3 — 변형 "새 버전에서 config 스키마가 바뀌어서 Gateway가 아예 기동하지 않는 장애가 발생했다면, 어떻게 대응하고 재발을 방지하시겠습니까?"
전제 개념

온보딩·마이그레이션·복구, 세 가지 서로 다른 안전망

왜 이 셋을 구분해야 하는가 — "업그레이드는 다 똑같이 위험하다"는 생각부터 깨기

많은 개발자가 "설치 마법사(onboard) → 업그레이드 도구(migrate) → 문제 생기면 진단 도구(doctor)"를 하나의 연속된 안전망처럼 생각한다. 하지만 이 셋은 서로 완전히 다른 문제를 풀도록 설계된 도구다. 이 차이를 구분하지 못하면, "migrate가 있으니 업그레이드는 안전하다"는 잘못된 안도감을 갖게 되고, 실제로는 아무 안전망이 없는 지점에서 사고가 난다.

도구실제로 푸는 문제안전망 범위
openclaw onboard신규 설치 시 최초 설정(모델 provider 인증, Gateway 모드, 채널, 스킬)최초 1회, 백지 상태에서 시작
openclaw migrate다른 도구(Claude Code, Codex CLI, Hermes 등)에서 OpenClaw로 이전plan/dry-run/backup 있음 — 단, "타 도구 → OpenClaw" 방향에 한정
openclaw doctor [--fix]이미 망가진 상태를 진단하고 복구사후 대응(reactive) — 사전에 "무엇이 깨질지" 알려주지 않음
가장 중요한 구분: 같은 버전 내에서 "다른 도구를 OpenClaw로 가져오는 것"과, "OpenClaw 자체를 구버전에서 신버전으로 올리는 것"은 완전히 다른 작업이다. 전자는 migrate가 정식으로 지원하지만, 후자(버전 간 config 스키마 변경)에는 같은 수준의 dry-run/diff 도구가 없다.

1) 온보딩(`openclaw onboard`) — 세 갈래 흐름

최초 설치 시 실행하는 가이드 마법사로, 모델 provider 인증(예: --auth-choice openai-api-key|anthropic|custom-api-key), Gateway 실행 모드(로컬/원격, --install-daemon), 워크스페이스/에이전트 정체성 초기화, 채널 연결, 스킬 설치, 마지막 헬스체크까지 이어진다. 크게 세 갈래로 나뉜다.

  • --flow quickstart: 토큰 자동 생성 등 최소한의 질문만으로 빠르게 시작
  • --flow manual/advanced: 세부 옵션을 직접 선택
  • --flow import: 기존에 쓰던 도구(Claude Code/Desktop, Codex CLI, Hermes 등)를 자동으로 감지해서, 무엇을 가져올지 미리보기(plan)를 보여주고 확인을 받은 뒤 적용한다. instructions/CLAUDE.md, MCP 서버 설정, 스킬, 모델 설정, (opt-in 시) API 키/자격 증명까지 옮길 수 있다.

--non-interactive 모드로 무인 설치를 할 때는 --accept-risk 같은 명시적 플래그를 요구한다 — 이는 "확인 없이 진행하는 리스크는 운영자가 명시적으로 인지하고 동의해야 한다"는 원칙을 CLI 수준에서 강제하는 설계다.

2) 마이그레이션(`openclaw migrate`) — plan/dry-run/backup이 실제로 존재하는 지점

migrate는 "타 에이전트 도구 → OpenClaw"로 넘어올 때 쓰는 도구다. 흐름은 다음과 같다. 

# 1) 무엇을 가져올 수 있는지 확인
openclaw migrate list

# 2) dry-run — 실제로 아무것도 바꾸지 않고 "적용하면 무엇이 바뀌는지" 계획만 생성
openclaw migrate claude --dry-run

# 3) 검토 후 실제 적용 — 적용 직전 OpenClaw 자체 백업을 생성/검증한 뒤 진행
openclaw migrate apply claude --yes

# 참고 플래그
# --json                 : 계획을 기계가 읽을 수 있는 형태로 출력
# --from <path>           : 기본 위치가 아닌 다른 소스 경로 지정
# --overwrite             : 기존 값이 있어도 덮어씀
# --include-secrets       : 자격 증명까지 포함(기본은 redact)
# --no-backup --force     : 백업을 건너뜀(위험 — 명시적 --force 필요)
왜 이렇게 설계했는가: apply는 적용 전 OpenClaw 자체 백업을 생성하고 검증하며, plan 리포트에는 비밀 값이 기본적으로 마스킹(redact)된다. 그리고 충돌(conflict)이 있으면 apply가 계속 진행하지 않고 멈춘다. 이건 "가져오는 행위 자체가 기존 설정을 되돌릴 수 없이 덮어쓸 수 있는 위험한 작업"이라는 걸 도구 설계 단계에서 인정하고, 되돌릴 수 있는 경로(백업)와 확인 단계 (dry-run)를 기본값으로 강제한 것이다. 이 안전장치는 문서 기준 v2026.4.26에서 정식 도구로 도입되었다.

3) `openclaw doctor` / `doctor --fix` — 사후 진단과 복구

doctor는 흔한 고장 패턴을 점검한다: 오래된 per-agent OAuth 잔여 정보, config 손상(마지막 정상 백업으로 복원하거나 JSON이 아닌 접두 문자열을 제거), .clobbered.*/.rejected.* 서비스 파일, 포트/서비스 충돌, launchd/systemd/schtasks 중복 등록, 플러그인/채널 로드 실패, 스킬 준비 상태. --deep은 실제 TCP 클라이언트/프로세스 상태까지 살아있는 상태에서 점검하고, --fix/--yes는 발견한 문제를 자동으로 복구한다.

실제 사례(단일 출처 취합, 확정도 중간): v2026.4.2에서 xAI x_search 설정 경로가 플러그인 전용 경로로 옮겨지면서, 공식 안내가 "업그레이드 후 반드시 doctor --fix를 실행하라"였다 — 즉 doctor는 업그레이드가 이미 끝난 뒤 뒷정리를 담당하는 도구지, 업그레이드 전에 "무엇이 깨질지" 미리 알려주는 도구가 아니다.

4) 진짜 공백 — 버전 간 config 스키마 변경에는 "diff 도구"가 없다

플러그인 SDK 쪽은 규율이 잡혀 있다. 공식 마이그레이션 가이드(docs/plugins/sdk-migration.md)는 "새 계약 도입 → 런타임 경고와 함께 호환 어댑터 유지 → 마이그레이션 경로 문서화 → 다음 메이저 릴리스에서만 제거"라는 절차를 명시한다. 예를 들어 deactivate → gateway_stop 별칭은 2026-08-16까지 경고와 함께 유지된다.

그러나 Gateway 코어의 config 스키마는 같은 수준의 규율을 항상 지키지 못했다. 대표적으로 GitHub 이슈 #35957은, v2026.3.2 → v2026.3.3 업그레이드에서 gateway.controlUi.autoApproveDevices/requirePairing 키가 새 버전에서 거부(reject)되면서 Gateway가 기동에 실패해 18시간 40분 이상의 다운타임이 발생한 실제 production 장애다. 이 이슈에는 마이그레이션 도구가 전혀 없었고, "not planned"로 종료됐다. 이후 커뮤니티가 요청한 doctor --config-diff(업그레이드 시 어떤 키가 조용히 무효화되는지 미리 보여주는 기능, 이슈 #38249)도 "not planned"로 종료됐다.

정리: migrate = "타 도구 → OpenClaw" 안전한 import 도구. doctor --fix = 이미 벌어진 문제의 반응적(reactive) 복구 도구. "이번 버전을 올리면 정확히 무엇이 깨지는가"를 사전에 알려주는 proactive diff 도구는 2026-07 시점에 존재하지 않는다 — 이게 실제 production 갭이고, 이 레슨에서 가장 중요한 시니어 포인트다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "CLI 명령어를 외웠는가"가 아니라, "도구가 광고하는 안전장치의 실제 적용 범위를 스스로 검증하는 습관이 있는가"를 본다. 시니어 엔지니어와 그렇지 않은 엔지니어를 가르는 지점은 정확히 여기다 — 문서에 "dry-run과 backup을 지원한다"고 쓰여 있으면 그걸 모든 상황에 확대 적용해 믿는 사람과, "이 안전장치가 정확히 어느 작업에 적용되는지" 경계를 확인하는 사람.

  • 도구의 보장 범위를 정확히 읽는가 — "migrate가 안전하다"와 "업그레이드가 안전하다"를 혼동하지 않는가
  • 사전 예방과 사후 복구를 구분하는가 — doctor --fix는 사후 대응 도구라는 것을 아는가
  • 공백을 스스로 메우는 운영 설계 능력 — 도구가 제공하지 않는 안전망(스테이징 카나리, 버전 고정, 변경 로그 리뷰, 백업 절차)을 직접 설계할 수 있는가
  • 실제 장애 사례에서 배우는가 — "구조적으로 이런 사고가 왜 반복되는지"를 설명할 수 있는가
Senior 개발자는 "onboard/migrate/doctor가 있다"를 나열하지 않는다. "이 셋이 커버하지 못하는 정확한 공백이 어디인지, 그리고 그 공백을 팀 프로세스로 어떻게 메울지"까지 설명해야 한다.
실무 연결

실제 상황과 연결하기

상황 1: 팀이 Claude Code에서 쓰던 스킬/설정을 OpenClaw로 옮기려 한다

이건 openclaw migrate claude의 정석적인 사용 사례다. 먼저 --dry-run으로 무엇이 옮겨지는지(instructions, MCP 서버, 스킬, 모델 설정) 계획을 검토하고, 자격 증명까지 옮길지 (--include-secrets)를 팀 보안 정책에 따라 명시적으로 결정한 뒤, apply --yes로 적용한다. 적용 전 자동 백업이 생기므로, 잘못 옮겨졌다면 되돌릴 수 있다.

상황 2: 마이너 버전 업그레이드 후 Gateway가 기동하지 않는다 (실제 장애 패턴)

GitHub #35957 사례처럼, 이전 버전에서 정상이던 config 키가 새 버전에서 거부되어 부팅에 실패하는 경우가 실제로 있었다. 이때 "migrate가 있으니 자동으로 고쳐지겠지"라는 기대는 틀렸다 — migrate는 이 상황을 다루는 도구가 아니다. 실무에서는 (1) 이전 버전의 백업된 config로 즉시 롤백해 서비스를 복구하고, (2) 새 버전의 변경 로그(changelog)에서 어떤 키가 바뀌었는지 확인하고, (3) doctor --fix로 자동 복구가 가능한지 시도하고, (4) 안 되면 config를 수동으로 새 스키마에 맞게 고치는 순서로 대응한다.

상황 3: 업그레이드 후 특정 채널/도구가 조용히 동작하지 않는다

v2026.4.2에서 xAI 검색 설정 경로가 플러그인 전용 경로로 옮겨진 사례처럼, config 키가 "거부"되지 않고 조용히 무시되는 경우가 더 위험하다 — Gateway는 정상 기동하지만 특정 기능만 죽어 있어서 발견이 늦어진다. 이런 사고를 줄이려면 업그레이드 직후 doctor --deep --yes를 표준 절차로 실행해, 로드 실패한 플러그인/채널이 있는지 확인하는 습관이 필요하다.

상황 4: "이번 업그레이드에 breaking change가 있는지" 사전에 알고 싶다

2026-07 시점 OpenClaw에는 이걸 자동으로 알려주는 doctor --config-diff 같은 도구가 없다 (커뮤니티가 요청했지만 #38249로 종료됨). 그래서 팀 차원의 보완 절차가 필요하다: 프로덕션에 앞서 스테이징/카나리 Gateway에서 먼저 업그레이드하고, 릴리스 노트/CHANGELOG의 "breaking" 또는 "migration" 태그를 사람이 직접 읽고, config를 버전 관리(git)에 두어 변경 diff를 눈으로 확인하고, 업그레이드 직전 수동으로 백업을 한 번 더 뜬다. 이건 도구가 아니라 운영 프로세스로 메워야 하는 공백이다.

공백을 메우는 팀 프로세스를 제시하기 "그래서 저는 도구가 안전을 보장한다고 가정하지 않고, 스테이징 카나리 업그레이드, 변경 로그 리뷰, config의 git 버전 관리, 업그레이드 직전 수동 백업을 표준 런북에 넣습니다."
5
장애 발생 시 대응 순서로 마무리하기 "실제로 기동 실패가 발생하면, 백업으로 즉시 롤백해 서비스를 복구한 뒤, changelog로 원인을 확인하고, doctor --fix로 자동 복구를 시도하고, 안 되면 수동으로 config를 새 스키마에 맞춥니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"OpenClaw의 온보딩과 마이그레이션은 서로 다른 세 도구로 나뉩니다. openclaw onboard는 최초 설치 시 모델 provider 인증, Gateway 모드, 채널, 스킬을 설정하는 마법사이고, quickstart/manual 외에 Claude Code나 Codex CLI 같은 기존 도구에서 설정을 가져오는 import 흐름도 지원합니다. openclaw migrate는 이 import 작업을 더 명시적으로 다루는 도구로, dry-run으로 무엇이 바뀔지 미리 계획을 보여주고, 실제 적용 시 OpenClaw 자체 백업을 생성·검증한 뒤 진행하며, 충돌이 있으면 멈춥니다. 이건 실제로 신뢰할 수 있는 안전장치입니다.

다만 여기서 중요한 건, 이 안전장치의 범위가 '타 도구에서 OpenClaw로 가져오는 작업'에 한정 된다는 겁니다. OpenClaw 자체를 구버전에서 신버전으로 올리는 일반적인 업그레이드에는 같은 수준의 dry-run/diff 도구가 없습니다. 실제로 GitHub에 보고된 사례를 보면, 한 마이너 버전 업그레이드에서 이전에 유효했던 config 키가 새 버전에서 거부되면서 Gateway가 기동조차 못 하는 장애가 있었고, 다운타임이 18시간을 넘겼습니다. 이걸 예방할 수 있는 'config-diff' 기능이 커뮤니티에서 요청됐지만 아직 채택되지 않았습니다.

그래서 저는 이 공백을 팀 운영 프로세스로 메웁니다. 프로덕션에 바로 올리지 않고 스테이징이나 카나리 Gateway에서 먼저 업그레이드해보고, 릴리스 노트의 breaking change 안내를 사람이 직접 읽고, config 파일을 git으로 버전 관리해서 변경 전후 diff를 눈으로 확인하고, 업그레이드 직전에 수동으로 한 번 더 백업을 뜹니다. 그리고 업그레이드 직후에는 항상 doctor --deep --yes를 실행해서, 기동은 됐지만 조용히 로드에 실패한 플러그인이나 채널이 없는지 확인합니다.

만약 실제로 기동 실패 장애가 발생한다면, 먼저 백업으로 즉시 롤백해서 서비스를 복구하는 걸 최우선으로 하고, 그 다음에 changelog로 어떤 config 키가 바뀌었는지 확인하고, doctor --fix로 자동 복구가 되는지 시도한 뒤, 안 되면 수동으로 새 스키마에 맞게 config를 고칩니다. 핵심은 'migrate가 있으니 업그레이드도 안전하다'는 확대 해석을 하지 않고, 도구가 실제로 보장하는 범위와 그렇지 않은 범위를 구분해서 운영 설계를 하는 겁니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "플러그인 SDK는 deprecation alias로 하위 호환을 지키는데, 왜 core config 스키마는 같은 방식을 쓰지 않을까요?"

약한 답변: "그냥 개발팀이 놓친 것 같습니다."

강한 답변: "플러그인 SDK는 '함수/이벤트 계약'이라 호환 어댑터를 만들기 쉽습니다 — 옛 이름으로 호출되면 새 이름으로 매핑하고 경고만 띄우면 됩니다. 반면 core config 스키마는 종종 '이 값의 의미 자체'가 바뀝니다. 예를 들어 boolean 플래그 하나가 새 버전에서 다른 필드 3개로 쪼개지는 식이라면, 단순 별칭 매핑으로는 옛 값을 새 구조로 안전하게 변환할 수 없는 경우가 많습니다. 그래서 스키마 마이그레이션은 SDK 별칭보다 구조적으로 더 어렵고, 그만큼 자동화 투자 우선순위에서 밀렸을 가능성이 높다고 봅니다 — 다만 이건 '어렵다'는 것이지 '불필요하다'는 뜻은 아니므로, 팀 입장에서는 이 공백을 전제로 운영 설계를 해야 합니다."

F2. "스테이징 환경을 따로 둘 여유가 없는 소규모 팀이라면 어떻게 하시겠어요?"

강한 답변: "스테이징 Gateway 전체가 부담스럽다면, 최소한 세 가지는 비용 대비 효과가 큽니다. 첫째, config를 git으로 버전 관리해서 업그레이드 전후 diff를 sanity check로 본다. 둘째, 마이너 버전이 나와도 즉시 올리지 않고 며칠~1~2주 정도 지켜본 뒤(다른 사용자들의 이슈 리포트를 확인한 뒹) 올리는 '지연 업그레이드' 정책을 둔다. 셋째, 업그레이드 직전 수동 백업 + 업그레이드 직후 doctor --deep --yes를 절대 생략하지 않는 체크리스트화. 이건 도구가 없어도 프로세스만으로 리스크를 크게 줄입니다."

F3. "`openclaw migrate`의 dry-run이 실제로는 모든 경우를 다 잡아내지 못할 수도 있지 않을까요? 이걸 어떻게 검증하시겠어요?"

강한 답변: "맞습니다, dry-run은 '계획'이지 '실행 후 결과의 완전한 보증'은 아닙니다. 그래서 저는 dry-run 결과를 그대로 신뢰하지 않고, 중요한 마이그레이션이라면 (1) --json 출력으로 계획을 리뷰하고, (2) 실제 apply 전에 자동 백업이 정말 생성/검증됐는지 확인하고, (3) apply 직후 핵심 기능(모델 provider 호출, 채널 응답)을 스모크 테스트로 직접 확인합니다. 도구의 dry-run은 신뢰의 '출발점'이지 '종착점'이 아니라고 생각합니다."

F4. "이런 이슈들이 'not planned'로 닫혔다는 건 회사가 이 문제를 중요하게 여기지 않는다는 뜻 아닐까요?"

강한 답변: "그렇게 단정하기보다는, 오픈소스 프로젝트가 빠르게 성장하는 국면에서 신규 기능(멀티채널, 마켓플레이스, provider 확장)에 리소스가 먼저 몰리고, 스키마 마이그레이션 같은 '보이지 않는 인프라'는 후순위로 밀리는 흔한 패턴이라고 봅니다. 면접에서 중요한 건 회사를 평가하는 게 아니라, '이 공백이 지금 존재한다는 사실을 인지하고, 그걸 전제로 우리 팀의 운영 프로세스를 설계하는 것'입니다. 도구가 언젠가 이 공백을 메울 수도 있지만, 지금 프로덕션을 운영하는 입장에서는 그걸 기다릴 수 없습니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 온보딩/마이그레이션/doctor를 서로 다른 문제를 푸는 도구로 명확히 구분한다
  • migrate의 안전장치가 "타 도구 → OpenClaw" 방향에 한정된다는 경계를 정확히 짚는다
  • 버전 간 config 스키마 변경에 실제 장애 사례(다운타임 시간 등)를 근거로 공백을 설명한다
  • 공백을 팀 운영 프로세스(스테이징, changelog 리뷰, git 버전관리, 체크리스트)로 메우는 구체적 대안을 제시한다
  • doctor --fix가 사후 복구 도구라는 것을 알고, 장애 대응 순서에서 올바른 위치에 배치한다

약한 답변의 특징

  • "migrate가 dry-run/backup을 지원한다"까지만 말하고 적용 범위를 확대 해석한다
  • 온보딩/마이그레이션/doctor를 뭉뚱그려 "업그레이드 도구들"로만 설명한다
  • 버전 업그레이드도 항상 안전하다고 전제하고, 실제 장애 가능성을 고려하지 않는다
  • 공백이 있다는 것 자체를 인지하지 못하거나, 인지해도 대안 없이 "조심해야 한다"로 끝낸다
  • 장애 발생 시 "doctor를 실행해본다"고만 말하고 롤백·백업 우선순위를 언급하지 못한다
핵심 차이: 강한 답변은 "이런 도구들이 있다"를 설명하지 않는다. "이 도구들이 정확히 어디까지 보장하고, 어디서부터는 보장하지 않는지, 그리고 그 공백을 어떻게 메울지"를 설명한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "migrate가 dry-run/backup을 지원하니 모든 업그레이드가 안전하다"

사실: migrate는 타 도구(Claude Code, Codex, Hermes 등)에서 OpenClaw로 가져오는 작업을 위한 도구다. OpenClaw 자체의 버전 간 업그레이드(config 스키마 변경 포함)에는 이 수준의 dry-run/diff 도구가 존재하지 않는다.

오해 2: "doctor --fix를 실행하면 업그레이드로 인한 문제는 다 해결된다"

사실: doctor --fix는 이미 알려진 고장 패턴(OAuth 잔여 정보, 손상된 config, 포트 충돌 등)을 복구하는 도구다. 새 버전에서 완전히 거부되어 아예 기동에 실패하는 상황이라면, 롤백이 먼저고 doctor는 그 다음이다.

오해 3: "config 스키마가 바뀌면 항상 명확한 에러로 알려준다"

사실: 어떤 경우는 명확히 거부(기동 실패)되지만, 어떤 경우는 조용히 무시 되어 Gateway는 정상 기동하는데 특정 기능만 죽어 있는 경우도 실제로 있었다(예: v2026.4.2의 설정 경로 이동 사례). 후자가 발견하기 훨씬 어렵고 위험하다.

오해 4: "플러그인 SDK처럼 core config도 당연히 deprecation 경고를 거쳐 제거될 것이다"

사실: 플러그인 SDK의 규율 있는 deprecation 절차(경고 → 문서화된 마이그레이션 경로 → 다음 메이저에서 제거)는 플러그인 계약에는 확인되지만, core config 스키마 변경에 항상 같은 절차가 적용된다는 근거는 없다. GitHub 이슈로 확인된 사례에서는 사전 경고 없이 키가 거부된 적이 있다.

실무 체크리스트

OpenClaw 버전 업그레이드 전/후 확인할 것

단계 확인 사항 생략 시 위험
업그레이드 전 릴리스 노트/CHANGELOG에서 breaking change 문구를 직접 확인 알려진 config 키 거부/이동을 모르고 그대로 진행
업그레이드 전 config를 git으로 버전 관리해 변경 diff 확보 문제 발생 시 무엇이 바뀌었는지 추적 불가
업그레이드 전 가능하면 스테이징/카나리 Gateway에서 먼저 적용 프로덕션에서 최초로 문제를 발견
업그레이드 전 수동 백업(자동 백업과 별개로 한 번 더) 롤백 시점에 최신 백업이 없어 복구 지연
업그레이드 후 doctor --deep --yes로 조용한 실패 여부 확인 특정 채널/플러그인이 죽은 채로 방치
장애 발생 시 백업 롤백 → changelog 확인 → doctor --fix → 수동 수정 순서 준수 원인 파악 없이 임기응변 수정으로 상태 악화

주니어 migrate가 dry-run/backup을 지원한다는 것은 알지만, 이게 "타 도구 import"에 한정된다는 것은 모름
중급 migrate의 적용 범위를 정확히 구분하고, doctor --fix가 사후 복구 도구라는 것을 설명할 수 있음
시니어 세 도구의 경계를 정확히 구분하고, 버전 간 config 스키마 변경에는 proactive diff 도구가 없다는 실제 공백을 근거(장애 사례)와 함께 짚으며, 이 공백을 스테이징 카나리·changelog 리뷰·git 버전관리· 체크리스트 같은 팀 프로세스로 메우는 구체적 운영 설계를 즉시 제시함. 장애 발생 시 롤백 우선 대응 순서도 명확함
자가 점검 질문:
1. 온보딩, migrate, doctor가 각각 어떤 문제를 푸는 도구인지 한 문장씩 구분해 말할 수 있는가?
2. migrate의 dry-run/backup 안전장치가 정확히 어느 방향(타 도구 → OpenClaw)에 적용되는지 설명할 수 있는가?
3. 버전 간 config 스키마 변경에는 왜 같은 수준의 안전장치가 없는지, 실제 장애 사례를 근거로 설명할 수 있는가?
4. 이 공백을 메우는 팀 운영 프로세스(스테이징, changelog 리뷰, 백업, 체크리스트)를 스스로 설계해 말할 수 있는가?
5. 실제 기동 실패 장애가 발생했을 때 롤백 → 원인 확인 → 자동 복구 시도 → 수동 수정의 순서를 즉시 말할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • 온보딩(onboard) / 마이그레이션(migrate) / 진단·복구(doctor)는 서로 다른 문제를 푸는 도구다. 최초 설치, 타 도구 import, 사후 복구로 명확히 구분한다.
  • migrate의 dry-run/plan/자동 백업은 실제로 신뢰할 수 있는 안전장치지만, 이 범위는 "타 도구 → OpenClaw" import에 한정된다.
  • OpenClaw 버전 간 config 스키마 변경에는 proactive diff 도구가 없다 — 실제로 이전 버전에서 유효했던 키가 새 버전에서 거부되어 장시간 다운타임이 발생한 사례(GitHub #35957)가 있다.
  • 플러그인 SDK는 규율 있는 deprecation 절차(경고 → 문서화 → 다음 메이저에서 제거)를 갖지만, 이 규율이 core config 스키마에도 항상 적용된다고 가정하면 안 된다.
  • 공백은 도구가 아니라 팀 프로세스로 메운다 — 스테이징 카나리, changelog 리뷰, config git 버전관리, 업그레이드 전후 체크리스트가 실제 대안이다.
  • 장애 대응 우선순위는 "롤백 먼저, 원인 분석은 그 다음"이다 — doctor --fix는 서비스 복구 이후의 정리 단계에 쓴다.
