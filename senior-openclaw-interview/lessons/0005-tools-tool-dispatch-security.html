Day 5 · OpenClaw Interview

Tools와 Tool Dispatch 보안

"tool을 몇 개 껐다"가 아니라, tool 권한이 서로 독립적인 세 개의 통제축(무엇을 줄지 / 어디서 실행할지 / exec만의 예외)으로 이루어져 있고, 그 축들이 서로를 대신해주지 않는다는 것을 구조적으로 설명하는 능력
버전/시점 기준: 이 레슨은 2026-07 기준 OpenClaw 공식 문서(docs/tools/exec-approvals.md, docs/tools/exec.md, docs/tools 개요, docs/gateway/config-tools, docs/gateway/sandbox-vs-tool-policy-vs-elevated.md, 브라우저 tool 문서)와 GitHub 이슈/보안 권고 (#16323, GHSA-m3mh-3mpg-37hw, #12173, GHSA-cv7m-c9jx-vg7q/CVE-2026-26329, #12202, #5948, #32637)를 직접 조사해 작성했다. docs.openclaw.ai 도메인은 자동화 접근에 403을 반환하는 경우가 있어, raw.githubusercontent.com 경로의 문서 원문과 검색 스니펫, GitHub 이슈/보안 권고 원문으로 교차 검증했다. 정책 축 구분(sandbox/tool policy/elevated)과 승인 바인딩 로직, 실제 취약점 사례는 문서·이슈 원문으로 직접 확인한 높은 확정도이며, 세부 설정 키 이름은 릴리스마다 바뀔 수 있어 면접 전 최신 문서로 재확인이 필요하다.
면접 질문
Q1 — 주력 질문 "OpenClaw 에이전트에게 파일을 읽고, 셸 명령을 실행하고, 브라우저를 조작할 수 있는 tool을 준다고 할 때, 이 tool 실행 권한을 어떻게 설계하시겠습니까? 실제 OpenClaw의 tool dispatch 구조를 기준으로 설명해주세요."
Q2 — 변형 "write/edit tool은 막아두고 exec tool만 열어둔 에이전트가 있습니다. 이 구성이 안전하다고 볼 수 있을까요?"
Q3 — 변형 "샌드박스(Docker) 안에서 실행되는 에이전트라면 tool 허용 범위를 느슨하게 잡아도 괜찮을까요?"
전제 개념

Tool, Tool Policy, 그리고 "세 가지 독립된 통제축"

1) Tool이란 무엇인가 — 그리고 정책은 "모델이 부르기 전"에 적용된다

tool은 에이전트가 호출할 수 있는 타입이 있는 함수다. 파일을 읽거나, 셸 명령을 실행하거나, 메시지를 보내거나, provider를 호출하거나, 다른 시스템을 조작할 때 쓰인다. OpenClaw의 tool은 크게 아래처럼 나뉜다.

카테고리대표 tool할 수 있는 것
Runtimeexec, process, code_execution셸 명령/코드 실행
Filesread, write, edit, apply_patch워크스페이스 파일 읽기/쓰기/패치
Webweb_search, x_search, web_fetch외부 정보 조회
Browserbrowser탭 열기/클릭/입력/스크린샷/JS 실행
Messagingmessage채널로 메시지 발신
Session/Agentsubagents, session_status, goal하위 에이전트 기동, 세션/목표 조회
Automationcron, heartbeat_respond예약 실행, 헬스체크 응답
Mediaimage, image_generate, music_generate, tts미디어 생성
핵심 설계 포인트: 문서는 이렇게 명시한다 — "정책이 tool을 제거하면, 모델은 그 턴에 그 tool의 스키마 자체를 받지 않는다(If policy removes a tool, the model does not receive that tool's schema for the turn)." 즉 tool 정책은 "모델이 쓰려고 하면 그때 막는" 런타임 차단이 아니라, 모델이 그 tool의 존재 자체를 모르게 만드는 더 강한 방어다. 이건 프롬프트 인젝션 방어에서 특히 중요하다 — 공격자가 아무리 정교하게 "exec로 이걸 실행해"라고 프롬프트를 주입해도, 모델에게 애초에 exec 스키마가 없으면 그 tool call 자체가 생성될 수 없다.

2) Tool Policy — allow/deny, 프로파일, provider별 예외

OpenClaw는 전역·에이전트별로 tools.allow/tools.deny를 설정한다. 같은 tool이 allow와 deny 양쪽에 있으면 항상 deny가 이긴다. 매칭은 대소문자를 구분하지 않고 * 와일드카드와 group:* 같은 그룹 축약도 지원한다. 실제 적용 순서는 base profile → provider profile → allow/deny다 — 즉 프로파일이 먼저 기본 tool 세트를 정하고, provider별 예외(tools.byProvider, 예를 들어 특정 저가/고속 모델에는 exec를 주지 않는 식)가 그 위에 얹히고, 마지막으로 allow/deny가 최종 확정한다. tools.profile: "coding"은 기본적으로 browser tool을 제외하며, 필요하면 tools.alsoAllow: ["browser"]로 다시 켜야 한다.

3) Exec 승인 구조 — 위험한 tool은 별도의 승인 레이어를 가진다

exec는 가장 위험한 tool이라 tools.exec.security(또는 최신 tools.exec.mode) 로 별도의 보안 모드를 갖는다.

모드동작
deny호스트 명령 실행 요청을 전부 차단
allowlistallowlist에 매칭되는 명령만 프롬프트 없이 허용, 나머지는 ask 모드가 완화하지 않는 한 거부
full승인 게이트 없이 무제한 실행(권한 상승 모드에 준함)

여기에 ask 파라미터(off/on-miss/always)가 프롬프트 여부를 조절한다. allowlist는 glob 패턴을 쓰는데, rg처럼 맨 이름만 쓰면 PATH로 해석되는 바이너리만 매칭되고(./rg 같은 상대경로는 매칭 안 됨), ~/.local/bin/* 같은 경로 glob은 특정 위치를 신뢰하는 식이다. argPattern 정규식으로 인자까지 제한할 수 있다 — 예를 들어 "pattern": "python3", "argPattern": "^safe\.py$"는 python3 safe.py만 허용한다. 셸 체인(a && b 등)은 모든 세그먼트가 allowlist를 통과해야 한다. 프롬프트 UI에 접근할 수 없을 때(macOS 앱이 없거나 타임아웃)는 askFallback이 결정하는데, 생략 시 기본값은 deny다.

기본값의 비대칭이 인터뷰에서 자주 놓치는 함정이다: 문서 기준으로 샌드박스 안에서는 기본 security=deny, ask=off이지만, Gateway/Node(호스트)에서는 기본이 security=full, ask=off다. 즉 샌드박스를 켜지 않으면, 게이트웨이는 기본적으로 승인 없이 셸 명령을 실행하는 상태라는 뜻이다. "샌드박스를 안 쓰기로 했다"는 결정이 곧 "exec를 사실상 무제한으로 열어두기로 했다"는 결정과 같아진다 — 이 연결을 설명할 수 있는지가 시니어 신호다. 운영 시 최소권한 권장은 샌드박스를 켜고, 게이트웨이/노드 실행에도 security=allowlist를 유지하는 것이다. 전부를 완전히 열어젖히는 프리셋도 존재한다 — tools.exec.security=full + ask=off + host askFallback=full을 한 번에 맞추는 YOLO 모드 (openclaw exec-policy preset yolo)이며, 이건 명시적으로 이름 붙여진 "위험을 알고 켜는" 프리셋이다.

4) 승인 바인딩 — "승인 후 몰래 바뀌는 것"을 막는 설계

명령이 한 번 승인되면, OpenClaw는 그 시점의 실행 컨텍스트(작업 디렉터리, 정확한 인자, 환경변수 바인딩, 고정된 실행 파일 경로)를 통째로 바인딩한다. 스크립트/인터프리터 호출(python script.py 등)은 하나의 구체적인 파일 오퍼랜드에 best-effort로 바인딩되고, 승인 이후 그 내용이 바뀐 것이 감지되면 재승인을 요청하는 게 아니라 그냥 실행을 거부한다. 이건 "승인은 받았지만 그 사이 스크립트 내용이 바꿔치기 당했다" 같은 TOCTOU(Time-Of-Check to Time-Of-Use)류 공격을 구조적으로 막기 위한 설계다.

5) 세 가지 통제축 — Sandbox / Tool Policy / Elevated는 서로 다른 질문에 답한다

OpenClaw 공식 문서(sandbox-vs-tool-policy-vs-elevated)는 tool 권한을 하나의 스위치가 아니라 서로 독립된 세 개의 축으로 명시적으로 분리한다.

축답하는 질문설정
Sandbox어디서 실행되는가agents.defaults.sandbox.mode: off/non-main/all
Tool Policy어떤 tool을 쓸 수 있는가전역/에이전트별/provider별 allow-deny
Elevatedexec만의 탈출구 — 샌드박스 세션이 host에서 명령을 실행하도록 승인exec 전용 오버라이드
가장 흔한 오판정 두 가지가 문서에 명시적으로 경고돼 있다:

1) "tool policy는 이름으로 거른다. exec 안에서 벌어지는 부작용(side effect)까지는 보지 않는다." write/edit tool을 막아도 exec가 열려 있으면, 모델은 셸 명령 (echo ... > file, rm, curl | sh 등)으로 파일을 읽고 쓰고 지울 수 있다 — "tool policy is the hard stop: /exec는 거부된 exec tool을 우회할 수 없다"지만, 반대로 exec 자체가 허용돼 있다면 다른 파일 tool을 다 막아도 무의미하다.

2) "elevated는 추가 tool을 주는 게 아니라, exec가 어디서 도는지만 바꾼다." elevated를 켜도 새로운 tool이 생기지 않는다 — 다만 exec의 실행 위치가 sandbox에서 host로 넘어갈 뿐이다. 그리고 샌드박스 자체도 /var/run/docker.sock 같은 걸 마운트하면 "사실상 host 제어권 전체를 샌드박스에 넘겨주는 것"과 같아져, 격리라는 전제 자체가 무너진다.

6) 파일 tool 스코핑 — read/write/edit/apply_patch의 경로 통제

tools.fs.workspaceOnly: true는 read/write/edit/apply_patch 경로(및 프롬프트 이미지 자동 로드 경로)를 워크스페이스 디렉터리로 제한한다. 중요한 건 이게 옵트인이지 기본값이 아니라는 점이다.

실제 취약점 — apply_patch 경로 트래버설(#12173, CVSS 8.5): src/agents/apply-patch.ts의 resolvePatchPath()는 샌드박스가 있을 때는 assertSandboxPath()로 경로 컨테인먼트를 검증하지만, 샌드박스가 없을 때(sandboxRoot가 undefined인 게이트웨이/베어메탈 모드)는 resolvePathFromCwd()로 빠지는데, 이 경로는 절대경로와 ../ 시퀀스를 컨테인먼트 체크 없이 그대로 path.normalize()/path.resolve()한다. 프롬프트 인젝션이나 손상된 플러그인이 apply_patch를 호출할 수 있는 상태에서, /root/.ssh/authorized_keys나 ../../../etc/cron.d/malicious-job 같은 경로를 주면 임의 파일 쓰기가 가능하다. 이 이슈는 "closed as not planned"로 종료됐다 — 즉 전용 코드 수정 없이, "샌드박스를 안 쓰면 감수해야 할 리스크"로 취급됐다는 뜻이다.

더 흥미로운 신호는 커뮤니티가 반복적으로 요청했지만 아직 없는 기능이다. 이슈 #12202는 에이전트별 파일 경로 allow/deny를 tool 실행 레이어에서 강제해달라는 요청인데, 제출자는 신뢰 등급이 다른 세 에이전트를 같은 OS 계정으로 돌리면서 SOUL.md 프롬프트 지시("이 파일은 읽지 마라")로 막으려 했지만 "Gemini 3 Flash 모델이 그 지시를 무시하고 config 파일을 읽었다"고 보고했다 — 프롬프트 기반 가드레일은 모델에 따라 신뢰할 수 없다는 것을 보여주는 실증 사례다. 이슈 #5948(Docker 없이 파일시스템 경로 제한)과 #32637(에이전트별 cwd 분리)도 비슷한 맥락에서 열려 있다. 즉 "tool 이름 단위"의 정책은 성숙해 있지만, "경로/인자 단위"의 세밀한 스코핑은 exec의 argPattern 정도를 빼면 아직 구조적으로 약한 영역이라는 걸 알고 있으면 좋은 신호가 된다.

7) 브라우저 tool — 프로파일 격리, SSRF 가드, 임의 JS 실행

browser tool은 탭을 열고, 페이지를 읽고, 클릭/입력하고, 스크린샷/PDF를 뽑고, 임의 JS를 평가(act:evaluate)할 수 있다. 세 가지 프로파일이 있다 — openclaw(전용 격리 프로필, 개인 브라우저를 전혀 건드리지 않음), user/chrome(실제 로그인된 세션을 그대로 사용 — 문서가 명시적으로 "고위험(higher-risk)"이라 표기). 네비게이션 전에는 SSRF 가드가 강제 적용되며(browser.ssrfPolicy.hostnameAllowlist), 사설망 접근은 dangerouslyAllowPrivateNetwork: true로만 옵트인할 수 있다. 임의 JS 실행 (act:evaluate)은 기본값이 활성화 상태이며, 끄려면 명시적으로 evaluateEnabled: false를 설정해야 한다.

실제 취약점 — 브라우저 업로드 경로 트래버설(GHSA-cv7m-c9jx-vg7q / CVE-2026-26329, High, CWE-22): 브라우저 업로드 기능이 파일 경로를 검증하지 않고 Playwright의 setInputFiles()에 그대로 전달해, 유효한 게이트웨이 인증 토큰을 가진 공격자가 ../../../etc/passwd 같은 경로를 "업로드"라는 명목으로 넘겨 임의 파일을 읽고, 페이지 JS(FileReader)나 에이전트/브라우저 스냅샷으로 그 내용을 유출할 수 있었다. 2026.2.14에서 업로드 경로를 임시 디렉터리로 제한하고 트래버설 시도를 거부하도록 수정됐다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "tool을 껐다 켰다 할 수 있다"는 사실을 아는지가 아니라, tool 권한을 하나의 온/오프 스위치가 아니라 서로 독립된 여러 축의 조합으로 이해하고, 그 축들이 서로를 대신해주지 않는다는 것을 아는지를 본다. OpenClaw 공식 보안 문서가 제시하는 계층적 접근은 명확하다 — 먼저 신원(누가 봇과 대화할 수 있는가), 다음으로 범위(봇이 어디까지 행동할 수 있는가), 마지막으로 모델(모델은 조작될 수 있다고 가정하고 blast radius를 제한한다) 순서로 방어를 쌓는다. 시니어는 "tool policy가 있으니 안전하다"에서 멈추지 않고, "정책이 이름으로만 거른다"는 한계(#16323, exec-vs-write/edit 함정), "기본값 자체가 비대칭적으로 위험하다"는 사실(샌드박스 없을 때 gateway exec 기본이 full), "프롬프트 지시는 아키텍처 강제를 대체할 수 없다"는 실증(#12202)까지 구조적으로 설명할 수 있어야 한다.

실무 연결

실제 상황과 연결하기

Slack/Discord 봇으로 OpenClaw를 배포하는 팀을 상상해보자. 명시적인 tool 정책을 설정하지 않고 배포했다면, 과거 실제로 보고된 문제(#16323)와 같은 상황에 놓인다 — 그 이슈는 "정책이 명시적으로 정의되지 않으면 exec를 포함한 모든 tool이 기본적으로 허용된다"는 것과, 세션 디렉티브 (/exec host=gateway 같은)가 권한 검증 없이 세션 상태에 그대로 저장되는 문제, Windows에서 runCommandWithTimeout이 비실행파일에 대해 암묵적으로 shell: true를 써서 셸 메타문자(&, |, ^ 등) 인젝션이 가능했던 문제를 함께 지적했다. 메인테이너는 PR #16320으로 기본값을 deny-first로 바꾸고, 권한 없는 디렉티브를 파싱 단계에서 명시적으로 제거하고, Windows 배치 파일 실행 시 cmd.exe /c를 verbatim 인자로 명시 호출하도록 고쳤다 — "기본값이 안전할 것"이라는 가정 자체가 실제 사고 원인이었다는 사례다.

또 다른 예로, 로컬 플러그인/훅을 openclaw plugins install <path>로 설치하는 것도 최소권한이 적용돼야 할 표면이다. GHSA-m3mh-3mpg-37hw(High, CVSS 8.6, 2026.3.24에서 수정)는 설치 단계에서 npm install --ignore-scripts를 쓰더라도, 프로젝트의 .npmrc가 git=<악성 프로그램>으로 git 실행 파일 자체를 하이재킹하고, package.json의 git 의존성이 그 악성 프로그램을 트리거해 신뢰 확립 이전에 임의 코드 실행이 벌어질 수 있었다. 즉 "런타임 tool 호출"만 최소권한 대상이 아니라, 신뢰하지 않는 플러그인/스킬을 설치하는 행위 자체가 이미 공급망 공격 표면이라는 것 — 이는 Day 13(마켓플레이스 보안)에서 더 깊이 다룬다.

범위 밖 참고: 2026년 초에 보고된 CVE-2026-25253(Control UI가 쿼리스트링의 gatewayUrl을 검증 없이 신뢰해 게이트웨이 토큰을 자동 전송, WebSocket origin 미검증으로 1-클릭 RCE)과 ClawJacked(게이트웨이가 로컬 origin 연결을 과도하게 신뢰해 웹사이트 JS가 몰래 기기 페어링을 승인시킴)는 tool dispatch 정책 자체의 결함이 아니라 인증/원격 노출 문제다 (Day 10에서 다룬다). 하지만 이 두 사고가 보여주는 함의는 이 레슨과 직결된다 — 인증 경계가 뚫리면 공격자는 그 세션에 이미 허용된 tool을 그대로 쓸 수 있으므로, tool policy는 인증/노출 경계와 반드시 함께 설계돼야 한다는 것이다.
답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
이 에이전트에게 진짜로 필요한 최소 tool 집합이 무엇인지 먼저 정의한다 프로파일(예: coding은 browser 제외)과 allow/deny로 "무엇을 줄지"를 확정한다.
2
그 tool이 어디서 실행되는지를 별도로 결정한다 sandbox(off/non-main/all)로 격리 여부를 정한다 — 이건 tool policy와 완전히 다른 축이라는 것을 짚는다.
3
exec처럼 위험한 tool은 세 번째 레이어(승인 정책)로 한 번 더 좁힌다 security mode(deny/allowlist/full), allowlist+argPattern, ask/askFallback, 그리고 host별 기본값 비대칭 (샌드박스 없으면 gateway 기본이 full이라는 것)을 명시적으로 재설정한다.
4
tool policy가 "이름"만 거른다는 한계를 인지하고 조합을 설계한다 write/edit을 막아도 exec가 열려 있으면 동일한 side effect가 가능하다는 것, elevated는 tool을 더 주는 게 아니라 exec의 실행 위치만 바꾼다는 것을 답변에 명시한다.
5
설치 시점(플러그인/스킬)도 별도의 최소권한 검토 대상으로 본다 런타임 tool 호출뿐 아니라 신뢰하지 않는 로컬 패키지 설치 자체도 공격 표면이라는 것을 짚는다.
6
이 모든 설계가 "모델은 조작될 수 있다"는 전제 위에 있다는 것으로 마무리한다 프롬프트 지시(SOUL.md 등)는 방어가 아니라 힌트일 뿐이며, 실제 방어는 아키텍처 레벨 강제(스키마 제거, workspaceOnly, allow/deny, 샌드박스)에서 나온다는 것을 강조한다.
핵심 답변 예시

면접 현장 답변 예시 (전체)

"OpenClaw에서 tool 권한은 하나의 스위치가 아니라 세 개의 독립된 축으로 설계돼 있습니다. Tool Policy는 '이 모델에게 어떤 tool이 존재하는지'를 결정하고, 정책으로 제거된 tool은 모델에게 스키마 자체가 전달되지 않습니다 — 즉 모델이 프롬프트 인젝션을 당해도 애초에 그 tool을 호출할 방법이 없습니다. Sandbox는 '그 tool이 어디서 실행되는지'를 결정하는 완전히 다른 축이고, Elevated는 exec에 한정된 탈출구로, 추가 tool을 주는 게 아니라 실행 위치만 host로 바꿉니다.

여기서 실무적으로 가장 중요한 건 이 세 축이 서로를 대신해주지 않는다는 겁니다. write/edit tool을 막아도 exec가 열려 있으면 셸 명령으로 똑같이 파일을 조작할 수 있고, 샌드박스를 켰어도 도커 소켓을 마운트해두면 격리 자체가 무의미해집니다. 그리고 기본값 자체에 비대칭이 있습니다 — 샌드박스가 없는 상태에서 게이트웨이의 exec 기본 보안 모드는 'full'이라, 샌드박스를 안 쓰기로 한 결정이 곧 exec를 사실상 무제한으로 열어두는 결정과 같아집니다. 실제로 이 기본값 문제로 인해 정책이 명시되지 않으면 모든 tool이 기본 허용되던 이슈가 보고된 적이 있고, 지금은 deny-first로 고쳐졌습니다.

그래서 저는 (1) 프로파일/allow-deny로 최소 tool 집합을 정의하고, (2) 샌드박스 모드를 명시적으로 켜고, (3) exec는 allowlist + argPattern으로 한 번 더 좁히고, ask/askFallback을 deny 성향으로 유지하고, (4) 파일 tool은 workspaceOnly를 켜서 경로를 워크스페이스로 제한하고, (5) 브라우저 tool은 격리 프로파일을 기본으로 쓰고 evaluateEnabled와 SSRF allowlist를 검토하고, (6) 신뢰하지 않는 스킬/플러그인 설치 자체도 별도 검토 대상으로 삼습니다. 마지막으로, 이 모든 설계는 '모델이 프롬프트 지시를 항상 지킬 것'이라는 가정에 기대지 않습니다 — 실제로 특정 모델이 '이 파일은 읽지 마라'는 지시를 무시한 사례가 보고돼 있어서, 방어는 항상 아키텍처 레벨 강제로 걸어야 한다고 생각합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

Q. 샌드박스를 켰는데 도커 소켓(/var/run/docker.sock)을 마운트해뒀다면 안전합니까?

아니요. 그 마운트 자체가 샌드박스 밖의 도커 데몬을 조작할 수 있는 통로가 되어, 사실상 host 전체 제어권을 넘겨주는 것과 같습니다. 격리는 "컨테이너 경계"만으로 성립하지 않고, 그 경계 안에 무엇이 bind되는지까지 함께 봐야 합니다.

Q. write/edit tool을 다 막았는데 exec는 열어뒀습니다. 안전한가요?

아니요. tool policy는 tool 이름 기준으로 필터링할 뿐, exec 안에서 실행되는 셸 명령의 부작용까지는 보지 않습니다. echo ... > file이나 rm, curl | sh 같은 명령으로 write/edit이 하던 일을 그대로 할 수 있습니다.

Q. system prompt에 "민감한 설정 파일은 읽지 마라"라고 써두면 충분한가요?

충분하지 않습니다. 실제로 커뮤니티에서 보고된 사례로, 특정 모델이 그 지시를 무시하고 config 파일을 읽은 경우가 있었습니다. 프롬프트 지시는 모델별로 신뢰도가 다르고 강제력이 없기 때문에, 실제 방어는 workspaceOnly나 allow/deny처럼 tool 실행 레이어에서 강제되는 통제여야 합니다.

Q. YOLO 모드(exec 완전 개방)를 실무에서 써도 되는 상황이 있을까요?

blast radius가 이미 0에 가까운 환경 — 예를 들어 일회성 CI 컨테이너나 매번 폐기되는 격리된 데모 환경 — 정도로 한정해야 합니다. 영구적인 워크스페이스나 실제 자격 증명에 접근 가능한 세션에서는 YOLO를 상시 프리셋으로 두면 안 됩니다.

Q. 승인된 exec 명령의 인자가 승인 후에 바뀌면 어떻게 됩니까?

OpenClaw는 승인 시점의 작업 디렉터리·인자·환경·실행 파일 경로를 바인딩해두고, 실행 직전에 그 컨텍스트가 바뀐 걸 감지하면 재승인을 요청하는 대신 그냥 거부합니다. "승인된 스크립트의 내용을 그 사이에 바꿔치기하는" 류의 TOCTOU 공격을 구조적으로 막기 위한 설계입니다.

좋은 답변 vs 약한 답변

차이를 명확히 보기

좋은 답변

  • sandbox / tool policy / elevated를 서로 다른 축으로 명확히 구분해서 설명한다
  • "tool policy는 이름 기준이라 exec 안의 부작용까지 못 본다"는 함정을 구체적 명령 예시로 짚는다
  • 샌드박스 없을 때 gateway exec 기본값이 full이라는 비대칭을 언급한다
  • 프롬프트 지시가 아키텍처 강제를 대체할 수 없다는 것을 실제 사례로 든다
  • 실행 시점뿐 아니라 설치 시점(플러그인/스킬)도 최소권한 대상이라는 것을 안다

약한 답변

  • "tool을 꺼두면 안전하다"는 단일 스위치 관점에서 멈춘다
  • 샌드박스와 tool policy를 같은 것처럼 뭉뚱그려 말한다
  • system prompt 지시("이건 하지 마라")를 실제 보안 통제처럼 제시한다
  • exec와 write/edit이 사실상 겹치는 권한이라는 걸 인지하지 못한다
  • "tool 정책 기본값이 안전할 것"이라고 별도 확인 없이 가정한다
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1 — "위험해 보이는 tool 몇 개만 꺼두면 안전하다." tool policy는 이름 기준 필터일 뿐이다. exec가 열려 있으면 write/edit/read가 하던 일을 셸 명령으로 그대로 재현할 수 있다. 위험한 조합을 함께 봐야 한다.

오해 2 — "system prompt로 지시하면 지켜진다." 실제로 특정 모델이 "이 파일은 읽지 마라"는 지시를 무시한 사례(#12202)가 보고됐다. 프롬프트는 힌트이지 강제가 아니다 — 강제는 tool 실행 레이어(스키마 제거, workspaceOnly, allow/deny)에서 걸어야 한다.

오해 3 — "샌드박스만 켜면 tool 권한은 느슨해도 된다." elevated 오남용이나 docker.sock 마운트 같은 설정 실수로 샌드박스 격리 자체가 무의미해질 수 있다. sandbox와 tool policy는 독립적으로 둘 다 최소화해야 한다.

오해 4 — "기본 설정이 항상 안전한 쪽으로 잡혀 있을 것이다." 실제로 정책이 명시되지 않으면 exec를 포함해 모든 tool이 기본 허용되던 사고(#16323)가 있었고, 샌드박스가 없을 때 gateway exec 기본값은 지금도 full이다. 기본값은 항상 직접 확인해야 할 대상이지, 신뢰의 근거가 아니다.

실무 체크리스트

Tool Dispatch 운영 시 확인할 것

  • 에이전트별로 tool 프로파일/allow-deny가 "필요한 최소 집합"으로 명시돼 있는가(기본값에 기대지 않았는가)
  • tools.exec.security/mode가 명시적으로 설정돼 있고, 샌드박스 유무에 따른 기본값 비대칭을 알고 있는가
  • 고위험 명령에 argPattern까지 걸어 인자 단위로 좁혔는가
  • askFallback이 deny 성향으로 유지되는가(프롬프트 UI 부재 시 자동 허용되지 않는가)
  • tools.fs.workspaceOnly가 켜져 있는가, 샌드박스가 없는 배포에서 apply_patch류 경로 트래버설 리스크를 인지하고 있는가
  • 브라우저 tool의 프로파일이 격리형(openclaw)인지, evaluateEnabled와 SSRF allowlist가 검토됐는지
  • elevated 오버라이드나 sandbox bind mount(도커 소켓 등)로 격리가 무력화될 여지가 없는지
  • 로컬 플러그인/훅/스킬 설치도 검토 없이 자동 신뢰하지 않는지(설치 시점 공급망 리스크)
  • 최신 GitHub 보안 권고(GHSA)와 이슈를 정기적으로 확인해 기본값 변경 이력을 추적하는가

내 답변 수준 확인하기

미흡 tool 권한을 "켜고 끄는" 단일 스위치로만 설명하고, sandbox/tool policy/elevated를 구분하지 못함
주니어 세 축이 있다는 건 알지만, exec와 write/edit이 사실상 겹치는 권한이라는 함정이나 기본값 비대칭은 설명 못함

시니어 세 통제축의 독립성과 그 축들이 서로를 대신해주지 않는 이유, 샌드박스 유무에 따른 exec 기본값 비대칭, 프롬프트 기반 가드레일의 실증적 한계(#12202), 실제 사고 사례(#16323, GHSA-m3mh, #12173, GHSA-cv7m)를 근거로 들며, 이를 프로파일/allowlist/workspaceOnly/샌드박스 조합으로 방어하는 구체적 운영 설계까지 즉시 제시함

4. system prompt 지시가 왜 실제 보안 통제가 될 수 없는지, 실증 사례를 근거로 들 수 있는가?
5. 실행 시점뿐 아니라 설치 시점(플러그인/스킬)도 최소권한 대상이라는 것을 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • tool 권한은 세 개의 독립된 축이다 — sandbox(어디서), tool policy(무엇을), elevated(exec만의 탈출구). 하나를 조여도 다른 축이 열려 있으면 방어가 성립하지 않는다.
  • tool policy는 이름 기준 필터다 — exec 안의 부작용(side effect)까지는 보지 않으므로, write/edit을 막아도 exec가 열려 있으면 동일한 피해가 가능하다.
  • 기본값 자체가 비대칭적으로 위험하다 — 샌드박스가 없으면 gateway/node의 exec 기본 보안 모드는 여전히 full이다. "샌드박스 안 쓰기"는 곧 "exec 사실상 무제한"과 같다.
  • 승인은 시점 바인딩된다 — cwd/args/env/실행 파일 경로를 승인 시점에 고정하고, 이후 드리프트가 감지되면 재승인이 아니라 거부한다(TOCTOU 방지).
  • 파일 경로 스코핑은 옵트인이다 — workspaceOnly를 켜지 않으면, 특히 샌드박스가 없는 배포에서 apply_patch 같은 tool이 경로 트래버설에 취약할 수 있다(#12173).
  • 프롬프트 지시는 보안 통제가 아니다 — 모델에 따라 "이 파일은 읽지 마라" 같은 지시를 무시한 실증 사례가 있다(#12202). 강제는 항상 tool 실행 레이어에서 걸어야 한다.
  • 최소권한은 런타임 호출뿐 아니라 설치 시점에도 적용된다 — 신뢰하지 않는 로컬 플러그인/훅 설치가 --ignore-scripts에도 불구하고 .npmrc git 하이재킹으로 임의 코드 실행에 이른 사례(GHSA-m3mh-3mpg-37hw)가 있다.
