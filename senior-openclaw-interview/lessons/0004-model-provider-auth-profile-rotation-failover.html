Day 4 · OpenClaw Interview

Model Provider 설정과 Auth Profile Rotation/Failover

"failover가 있으니 안전하다"가 왜 절반만 맞는 말인지 — 라운드로빈 설계의 의도와, 그 설계가 실제로 만들어내는 비용/신뢰성 사고를 구조적으로 설명하는 능력
버전/시점 기준: 이 레슨은 2026-07 기준 OpenClaw 공식 문서(docs/concepts/model-providers.md, docs/concepts/model-failover.md, docs/concepts/models.md)와 소스 코드 (src/agents/auth-profiles/order.ts 등), 그리고 GitHub 이슈(#100067, #48623, #99809, #99993, #92864, #74395, #73182, #92674, #88371)를 직접 조사해 작성했다. docs.openclaw.ai 도메인은 자동화 접근에 403을 반환해, 이 레슨은 raw.githubusercontent.com 경로의 문서 원문과 GitHub 코드/이슈 검색으로 교차 검증했다. 정확한 쿨다운 시간 수치(1분/5분/25분/1시간, 5시간/24시간)는 문서 요약 기반이라 중간 확정도로 표시하며, 나머지(2단계 failover 구조, 프로필 정렬 로직, SQLite 마이그레이션 이슈, 실제 사고 사례)는 소스 코드/이슈 원문으로 직접 확인한 높은 확정도다.
면접 질문
Q1 — 주력 질문 "OpenClaw에서 사용 중인 모델 provider나 API 키에 장애(rate limit, 인증 실패, 서버 오류)가 발생하면, 실제로 어떤 순서로 복구를 시도합니까? 이 구조를 설계 관점에서 설명해주세요."
Q2 — 변형 "같은 provider에 API 키(auth profile)를 여러 개 등록해두면 어떤 순서로 사용됩니까? 방금 성공적으로 응답한 프로필을 우선적으로 계속 쓰지 않는 이유가 있을까요?"
Q3 — 변형 "모델 failover 체인이 있으면 비용이 자동으로 안전하게 관리된다고 볼 수 있을까요? 실제로 어떤 함정이 있을 수 있습니까?"
전제 개념

Provider 설정, Auth Profile, 그리고 "2단계 failover"

1) 모델 참조 문법과 config 계층 — "어떤 모델을 쓸지"는 세 겹으로 정해진다

OpenClaw는 모델을 provider/model 형태로 참조한다 — 예를 들어 anthropic/claude-opus-4-6, openai/gpt-5.5, google/gemini-3.1-pro-preview, xai/grok-4.3. 이 참조는 세 계층의 config로 조립된다.

계층설정 키역할
기본 모델agents.defaults.model.primary평소에 실제로 쓰는 모델
모델별 오버라이드agents.defaults.models["provider/model"]특정 모델에만 적용할 메타데이터(예: reasoning 강도, context 제한)
장애 대응 체인agents.defaults.model.fallbacks기본 모델이 계속 실패할 때 순서대로 시도할 다음 모델 목록
커스텀/프록시 providermodels.providers.<id> (또는 models.json)OpenAI/Anthropic 호환 커스텀 엔드포인트(baseUrl, 모델 목록) 등록

API 키는 환경변수로도 주입할 수 있는데, 같은 provider에 여러 방식이 동시에 있으면 우선순위가 있다: OPENCLAW_LIVE_<PROVIDER>_KEY > <PROVIDER>_API_KEYS(목록) > <PROVIDER>_API_KEY > <PROVIDER>_API_KEY_*(번호가 붙은 여러 개). 이 우선순위를 모르면, "분명 새 키를 export했는데 왜 예전 키가 계속 쓰이지?" 같은 디버깅에서 헤맨다.

2) auth-profiles — "같은 provider 안에 여러 자격 증명을 어떻게 관리하는가"

한 provider(예: Anthropic)에 API 키가 여러 개, 또는 API 키와 OAuth 로그인이 함께 등록될 수 있다. 이걸 관리하는 게 auth profile이다. 각 프로필은 id, type (oauth / token / api-key), provider, 실제 자격 증명, 그리고 usageStats(프로필별 사용 통계)를 갖는다. usageStats는 provider:profileId 단위로 lastUsed, cooldownUntil, cooldownReason, cooldownModel, blockedUntil, blockedReason, blockedModel, disabledUntil, errorCount, failureCounts를 기록한다 — 즉 "이 프로필이 최근에 무엇 때문에 얼마나 실패했는가"를 프로필 단위로 추적한다.

저장 위치가 2026년 중 바뀌는 중이다: 전통적으로 이 정보는 ~/.openclaw/agents/<agentId>/agent/auth-profiles.json에 저장됐다. 그런데 2026.6.x 릴리스부터 이 저장소가 SQLite(openclaw-agent.sqlite)로 옮겨가는 중이며, 이 마이그레이션 자체가 아직 불안정하다(아래 GitHub #100067 참고). 이 트랙의 이전 레슨/자료가 "auth-profiles.json"만 언급했다면, 그건 2026.5.x 이하 기준이고 2026.6.x 이후는 SQLite 전환 리스크까지 함께 알아야 한다 — OpenClaw는 이렇게 저장 계층 자체가 마이그레이션 도중인 소프트웨어라는 전제를 인터뷰에서 보여줄 수 있으면 좋은 신호가 된다.

3) 프로필 정렬 로직 — "왜 방금 성공한 프로필을 계속 쓰지 않는가"

여러 프로필 중 어떤 걸 다음에 쓸지 정하는 순서(order.ts 기준)는 다음과 같다.

  1. 명시적 오버라이드가 최우선 — auth.order[provider] config, 또는 사용자가 채팅에서 직접 /model openai/gpt-5.4@<profileId>처럼 프로필을 지정한 경우, 무조건 그 선택을 따른다.
  2. 그 외에는 타입 우선순위 — oauth > token > api-key 순으로 먼저 그룹을 나눈다.
  3. 같은 타입 안에서는 라운드로빈 — lastUsed가 가장 오래된 프로필부터 순서대로 쓴다.
  4. 쿨다운 중인 프로필은 뒤로 밀림 — 제외되는 게 아니라, cooldownUntil이 가장 빨리 끝나는 순서로 뒤쪽에 재배치된다.
설계 의도가 명시적으로 코드 주석에 남아있다: "lastGood(마지막으로 성공한 프로필)는 의도적으로 무시한다 — 이걸 우선하면 다른 정상 프로필들이 굶주리기(starve) 때문이다." 실제로 한 사용자가 "가장 최근에 성공한 프로필을 우선해달라"고 요청한 이슈(#48623, 2026-03 제출)는 "not_planned"로 종료됐다. 즉 이건 버그가 아니라, "공정성(fairness)"을 "직전 성공 이력"보다 우선하기로 한 의도적인 트레이드오프다.

4) 2단계 failover — "프로필 로테이션"과 "모델 폴백"은 다른 층위다

모델 호출이 실패했을 때 OpenClaw의 대응은 두 단계로 나뉜다.

1
같은 provider, 같은 모델 안에서 auth profile 로테이션 예를 들어 Anthropic 키 3개가 등록돼 있는데 지금 쓰던 키가 rate limit(429)에 걸렸다면, 먼저 같은 anthropic/claude-opus-4-6 모델을 유지한 채 다음 순번의 auth profile로 재시도한다.
2
그래도 실패하면 모델 자체를 전환 같은 provider의 모든 프로필이 소진되거나(쿨다운/차단),애초에 대체 프로필이 없다면 agents.defaults.model.fallbacks에 정의된 다음 모델(다른 provider일 수도 있음)로 전환한다.

여기에 중요한 비대칭이 하나 더 있다 — 사용자가 채팅에서 직접 /model로 모델을 명시적으로 지정한 경우, 실패해도 자동으로 fallback 체인을 타지 않고 그 모델로 그대로 실패한다. 자동 폴백은 "설정된 기본 모델" 경로에서만 전체 체인을 순회하며, 그 과정에서 주기적으로(문서 기준 약 5분 간격) 원래 모델이 복구됐는지 재시도(re-probe)한다.

5) 세션 단위 pinning — 왜 매 요청마다 로테이션하지 않는가

한번 선택된 auth profile은 세션 단위로 고정(pin)된다. 매 요청마다 프로필을 바꾸면 provider 쪽의 prompt caching이 매번 깨져서 비용과 지연이 늘어나기 때문에, 실패가 없는 한 같은 프로필을 계속 쓴다. 이 pin은 세션 리셋, 컨텍스트 compaction, 또는 pin된 프로필이 쿨다운에 들어갈 때 풀린다.

6) 에러 분류 — retryable vs terminal

분류예시대응
재시도 가능(retryable) 인증 실패, rate limit(429), overloaded 신호, 타임아웃, billing 비활성화, 후보가 남아있는 미분류 오류 프로필 쿨다운 처리 후 다음 후보로 로테이션/폴백
종결(terminal) 명시적 abort, context length 초과(request_too_large), 모든 후보 소진 더 이상 재시도하지 않고 사용자에게 실패로 반환

쿨다운 백오프는(문서 요약 기준, 중간 확정도) rate limit류는 1분 → 5분 → 25분 → 1시간으로 상한까지 점점 길어지고, billing 관련 비활성화는 5시간부터 시작해 2배씩 증가, 최대 24시간까지 늘어난다.

이 분류 자체가 실무에서 틀릴 수 있다: GitHub #99993(closed) 사례는, DeepSeek의 401 인증 오류(원래는 종결에 가까운 오류)를 시스템이 "일시적 실패"로 잘못 분류해 cooldownUntil/disabledUntil을 걸어버린 버그였다 — 그 결과, 사용자가 키를 이미 고쳤는데도 오래된 쿨다운 상태 때문에 정상 키가 계속 뒤로 밀렸다. 그리고 #99809(open, 2026-07-04 제출)는, subscription_limit 차단이 원래 모델 단위(blockedModel 필드가 존재)로 스코프돼야 하는데 실제로는 프로필 전체에 적용되어, 그 프로필로 쓸 수 있는 다른 정상 모델까지 함께 막히는 문제다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "failover 기능이 있다"를 아는지가 아니라, "자동화된 안전장치가 새로운 실패 모드를 만들어낼 수 있다는 것을 이해하는가"를 본다. 시니어는 "failover가 있으니 안전하다"에서 멈추지 않고, "failover의 설계 선택(라운드로빈, 세션 pin, 에러 분류)이 어떤 상황에서 역효과를 내는지"까지 설명할 수 있어야 한다.

  • 설계 트레이드오프를 이해하는가 — lastGood을 의도적으로 무시하는 게 왜 "버그"가 아니라 "선택"인지 설명할 수 있는가
  • 로테이션과 폴백을 다른 층위로 구분하는가 — "프로필 교체"와 "모델 교체"를 뭉뚱그리지 않는가
  • 숨겨진 비용 리스크를 인지하는가 — 폴백/오버라이드가 조용히 더 비싼 모델로 전환되거나 유지될 수 있다는 것을 아는가
  • 에러 분류 실패의 실무 영향을 설명할 수 있는가 — 잘못된 재시도 가능성 판단이 정상 자격 증명을 오래 막아버릴 수 있다는 것을 아는가
Senior 개발자는 "auth profile rotation과 model fallback이 있다"를 나열하지 않는다. "이 자동화가 어떤 가정 위에서 동작하고, 그 가정이 깨지면 어떤 사고로 이어지는지, 그리고 그걸 어떻게 모니터링/제어할지" 까지 설명한다.
실무 연결

실제 상황과 연결하기

상황 1: 팀에서 같은 provider 키를 여러 개 등록해 rate limit을 분산한다

이건 auth profile 라운드로빈의 정석적인 활용이다. 다만 "직전에 성공한 키를 계속 쓰겠지"라고 기대하면 안 된다 — lastUsed 기준으로 오래 쉰 프로필부터 균등하게 순환하도록 설계돼 있다. 이건 의도적으로 "한 키에 몰빵되어 그 키만 rate limit에 몰리는 상황"을 막기 위한 설계다.

상황 2: 세션 중 비싼 모델로 잠깐 바꿨는데, 하루 만에 예상보다 훨씬 큰 비용이 청구됐다

실제 GitHub 사례(#92864, closed)에서, 사용자가 /model로 더 비싼 모델 (claude-opus-4-8)을 잠깐 시도했는데, 그날 밤 컨텍스트 compaction이 일어나면서 새 세션이 "원래 기본 모델로 돌아가지 않고" 그 비싼 모델을 그대로 물려받아 하루 만에 $300가 청구된 사고가 있었다. 이건 failover/오버라이드 로직 자체의 버그라기보다, "명시적 오버라이드의 생명주기가 세션 경계를 넘어 조용히 이어질 수 있다"는 걸 몰랐던 운영 공백이다.

상황 3: 업그레이드/온보딩 이후 비용이 조용히 늘었다

비슷한 패턴의 사고가 여러 건 보고돼 있다. 업그레이드 시 자동 마이그레이션이 기본 모델을 유료 -preview 변형으로 조용히 바꾼 사례(#74395, open), Claude 계열 모델의 reasoning(thinking) 기본값이 조용히 "on"으로 바뀌어 Anthropic 비용이 두 배가 되고 thinking 블록까지 채팅에 노출된 사례(#73182, open), thinking 레벨이 "adaptive"로 폴백되면서 토큰 사용량이 4~5배로 뛴 사례(#92674, open), Windows 온보딩에서 첫 채팅이 크레딧 경고 없이 유료 Anthropic 모델로 시작되는 사례(#88371, open). 공통 패턴은, 모델/설정 전환이 "조용히(silently)" 일어난다는 것이다 — 즉 이 트랙 Day 3에서 다룬 "조용한 config 변경" 문제가 비용 영역에서도 똑같이 반복된다.

상황 4: 여러 provider가 걸린 failover 체인에서, 첫 시도가 타임아웃 예산을 다 써버린다

GitHub #48623에는 lastGood 요청과 별개로 두 번째 버그가 함께 보고돼 있다 — 경량 실행 경로에서는 모든 failover 시도가 하나의 타임아웃 예산을 공유해서, 첫 시도가 rate limit으로 지연되면 그 시간을 다 써버려 정작 정상적으로 동작하는 백업 프로필까지 시도하지 못하고 전체가 실패하는 경우가 있었다. "failover 체인에 백업이 있다"는 것과 "그 백업이 실제로 시도될 시간이 남아있다"는 것은 별개의 문제다.

상황 5: 버전 업그레이드 중 auth 저장소 자체가 마이그레이션된다

#100067(open, 2026-07-04)은 2026.6.x의 JSON→SQLite 저장소 전환에서, 기존 auth-profiles.json의 프로필들이 새 저장소로 옮겨지지 않고 경고도 없이 "버려지는" 사고를 보고한다 — 그 결과 OAuth/토큰 기반 provider의 인증이 전부 끊기고, 폴백 체인도 조용히 무력화된다. 이건 Day 3에서 다룬 "버전 간 스키마 변경에 proactive diff 도구가 없다"는 공백이, 이번엔 auth 계층에서도 반복되고 있다는 실제 사례다.

정렬 로직과 그 설계 의도를 설명하기 "타입(oauth>token>api-key) 우선 후 lastUsed 기준 라운드로빈이고, lastGood은 의도적으로 무시합니다 — 특정 프로필에 트래픽이 몰려 다른 정상 프로필이 굶주리는 걸 막기 위한 설계입니다."
3
세션 pin과 그 이유 짚기 "매 요청 로테이션은 provider caching을 깨서 비용/지연을 늘리므로, 실패가 없으면 세션 동안 같은 프로필을 고정합니다."
4
실제 실패 모드를 근거와 함께 짚기 (Senior 포인트) "다만 이 자동화가 항상 안전한 건 아닙니다. 모델 오버라이드가 세션 경계를 넘어 조용히 유지돼 하루 만에 $300가 청구된 사례, 업그레이드 후 기본 모델이 조용히 유료 preview로 바뀐 사례, 에러 분류가 틀려서 정상 키가 오래 막힌 사례가 실제로 보고돼 있습니다."
5
운영 대응책 제시로 마무리하기 "그래서 저는 failover를 '있으니 안전한 기능'이 아니라 '모니터링이 필요한 자동화'로 취급합니다 — 모델별 비용 알림, 세션 오버라이드의 만료/원복 정책, 업그레이드 전후 auth profile 상태 점검을 팀 프로세스에 넣습니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"OpenClaw의 장애 대응은 두 단계로 구성됩니다. 첫 단계는 같은 provider, 같은 모델 안에서의 auth profile 로테이션입니다 — 예를 들어 Anthropic 키가 여러 개 등록돼 있고 지금 쓰던 키가 rate limit에 걸리면, 같은 모델을 유지한 채 다음 순번의 키로 재시도합니다. 이때 순서는 타입(oauth, token, api-key 순)으로 먼저 나뉘고, 같은 타입 안에서는 가장 오래 쉰 프로필부터 쓰는 라운드로빈입니다. 흥미로운 점은, 방금 성공한 프로필을 우선하는 'lastGood' 방식이 아니라는 겁니다 — 이건 실수가 아니라, 특정 프로필에 트래픽이 몰려서 다른 정상 프로필들이 굶주리는 걸 막기 위한 의도적인 설계 선택입니다.

두 번째 단계는, 같은 provider의 모든 프로필이 소진되거나 대체 프로필 자체가 없을 때, 설정된 fallback 체인을 따라 다른 모델(다른 provider일 수도 있습니다)로 전환하는 겁니다. 여기서 한 가지 비대칭이 있는데, 사용자가 채팅에서 직접 모델을 지정한 경우는 실패해도 자동으로 이 체인을 타지 않고 그 모델로 그대로 실패합니다 — 자동 폴백은 '설정된 기본 모델' 경로에서만 동작합니다. 그리고 한번 선택된 프로필은 provider 쪽 캐싱을 따뜻하게 유지하기 위해 세션 단위로 고정되고, 실패가 없으면 계속 그 프로필을 씁니다.

다만 저는 이 자동화가 있다고 해서 안전이 자동으로 보장된다고 생각하지 않습니다. 실제로 GitHub에 보고된 사례들을 보면, 세션 중 잠깐 비싼 모델로 바꿨는데 그 설정이 세션 경계를 넘어 조용히 유지돼 하루 만에 300달러가 청구된 경우, 업그레이드 후 기본 모델이 경고 없이 유료 preview 모델로 바뀐 경우, reasoning 기본값이 조용히 켜져서 비용이 두 배가 된 경우가 있습니다. 그리고 에러 분류 자체가 틀리는 버그도 있었습니다 — 인증 오류를 일시적 오류로 잘못 판단해서, 이미 고친 키가 오래된 쿨다운 상태 때문에 계속 뒤로 밀리는 경우입니다.

그래서 저는 이 failover 체계를 '설정만 해두면 끝'이 아니라 '지속적인 모니터링이 필요한 자동화'로 취급합니다. 모델별/프로필별 비용에 알림을 걸고, 세션에서의 명시적 모델 오버라이드는 유효기간이나 원복 정책을 팀 규칙으로 명시하고, 버전 업그레이드 전후에는 auth profile 상태와 실제 사용 중인 모델이 기대한 것과 같은지 확인하는 걸 표준 절차에 넣습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "lastGood 대신 라운드로빈을 쓰는 게 정말 더 나은 설계일까요? 단점은 없나요?"

약한 답변: "공정하니까 더 좋은 설계입니다."

강한 답변: "공정성 관점에서는 맞지만 공짜는 아닙니다. 예를 들어 프로필 A는 방금 성공했고 프로필 B는 사실 이미 문제가 있는 키인데 단순히 오래 쉬었다는 이유로 다음 순번이 되면, B가 다시 실패할 때까지 한 번의 왕복(요청)을 낭비하게 됩니다. 즉 '트래픽 분산의 공정성'과 '최근 성공 이력을 활용한 효율성' 사이의 트레이드오프이고, OpenClaw는 전자를 선택한 겁니다. 이걸 보완하려면, 애플리케이션 레이어에서 명백히 죽은 프로필은 아예 auth.order에서 제외하거나 비활성화하는 운영 개입이 필요합니다."

F2. "세션 pin이 있는데, failover가 어떻게 그 pin을 깨고 다른 프로필로 넘어가나요?"

강한 답변: "pin은 '실패가 없는 한' 유지되는 기본값이지, 절대적인 고정이 아닙니다. pin된 프로필이 실제로 쿨다운/차단 상태에 들어가면 그 시점에 pin이 풀리고 다음 순번 프로필이 새로 pin됩니다. 즉 pin의 목적은 '평상시 캐싱 효율'이고, failover의 목적은 '장애 시 가용성'이라 서로 다른 층위의 관심사 이며, 장애가 실제로 감지된 순간에만 pin이 재계산됩니다."

F3. "이 실패 사례들(#92864, #73182, #92674 등)이 공통적으로 보여주는 패턴은 무엇인가요?"

강한 답변: "공통점은 '모델/설정 전환이 사용자에게 보이지 않는 곳에서 일어난다'는 겁니다. 세션 오버라이드가 세션 경계를 넘어 이어지거나, 업그레이드 마이그레이션이 기본값을 바꾸거나, thinking 레벨이 폴백되는 것 모두 시스템이 '더 나은 판단'을 자동으로 내린 결과인데, 그 판단이 비용에 미치는 영향은 로그를 보지 않는 한 알기 어렵습니다. 그래서 이런 종류의 자동화에는 항상 '무엇이 바뀌었는지 사용자가 확인할 수 있는 가시성(observability)'이 같이 설계돼야 한다고 생각합니다."

F4. "여러분 팀이라면 이런 실패 모드를 어떻게 모니터링/알림으로 잡겠어요?"

강한 답변: "세 가지를 두겠습니다. 첫째, 모델/프로필별 호출 수와 비용을 대시보드로 분리해서, 기대하지 않은 모델(예: preview, 고가 모델)로 트래픽이 몰리면 알림이 오게 합니다. 둘째, 세션에서 명시적으로 모델을 오버라이드하면 그 사실과 예상 비용 배수를 로그에 명확히 남기고, 일정 시간/세션 이후 자동으로 기본값에 원복하는 정책을 둡니다. 셋째, 업그레이드 직후에는 openclaw models status와 auth profile 상태를 스모크 테스트 체크리스트에 포함해서, 기본 모델이나 인증 상태가 조용히 바뀌지 않았는지 확인합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • auth profile 로테이션과 model fallback을 서로 다른 층위로 명확히 구분한다
  • 라운드로빈이 lastGood을 의도적으로 무시한다는 설계 배경(공정성 vs 효율성 트레이드오프)을 설명한다
  • 사용자 명시적 오버라이드와 자동 폴백의 비대칭(오버라이드는 fallback 없이 그대로 실패)을 안다
  • 실제 비용/신뢰성 사고 사례를 근거로 "failover = 안전"이 아니라는 걸 설명한다
  • 에러 분류(retryable vs terminal) 자체가 틀릴 수 있다는 것과, 그게 실무에 미치는 영향(정상 키가 오래 막힘)을 안다
  • 모니터링/알림/원복 정책 같은 구체적인 운영 대응책을 제시한다

약한 답변의 특징

  • "auth profile이 여러 개면 로테이션되고, 실패하면 fallback 모델로 넘어간다"까지만 말하고 끝낸다
  • 라운드로빈이 왜 lastGood을 안 쓰는지 모르거나 "버그 같다"고 단정한다
  • 세션 오버라이드나 마이그레이션이 조용히 비용을 늘릴 수 있다는 리스크를 전혀 언급하지 않는다
  • 에러 분류가 절대 틀리지 않는다고 전제한다
  • "failover가 있으니 안전하다"로 결론짓고 운영 대응책을 제시하지 못한다
핵심 차이: 강한 답변은 "로테이션과 폴백이 있다"에서 멈추지 않는다. "이 자동화가 어떤 트레이드오프 위에 설계됐고, 그 트레이드오프가 실제로 어떤 사고를 만들어냈는지, 그리고 그걸 어떻게 모니터링할지"까지 설명한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "여러 키를 등록하면 가장 최근에 성공한 키를 계속 쓸 것이다"

사실: lastGood 기준이 아니라, 타입 우선순위 후 lastUsed 기준 라운드로빈이다. 이건 특정 프로필에 트래픽이 몰려 다른 정상 프로필이 굶주리는 걸 막기 위한 의도적 선택이다 (관련 기능 요청 이슈는 not_planned로 종료됨).

오해 2: "model fallback 체인이 있으면 비용은 자동으로 안전하게 관리된다"

사실: 실제로는 세션 오버라이드가 조용히 세션 경계를 넘어 유지되거나, 업그레이드가 기본 모델을 조용히 유료 변형으로 바꾸거나, reasoning/thinking 설정이 조용히 바뀌어 비용이 몇 배로 뛴 사례가 보고돼 있다. 폴백/오버라이드 시스템은 가용성을 위한 설계이지 비용 통제를 위한 설계가 아니다.

오해 3: "인증 실패든 rate limit이든 서버 오류든, 다 같은 재시도 로직을 탄다"

사실: retryable(인증 실패, rate limit, 타임아웃 등)과 terminal(context length 초과, 명시적 abort)이 구분돼 있고, 이 분류 자체가 항상 정확한 것도 아니다 — 실제로 인증 오류를 일시적 오류로 잘못 분류해 정상 키를 오래 막아버린 버그 사례가 있다.

오해 4: "auth-profiles.json이 유일한 저장 방식이고 앞으로도 그럴 것이다"

사실: 2026.6.x부터 SQLite 저장소로의 전환이 진행 중이며, 이 전환 자체가 기존 프로필을 경고 없이 유실시키는 버그를 아직 안고 있다. OpenClaw는 저장 계층까지도 빠르게 바뀌는 프로젝트라는 걸 전제해야 한다.

핵심 요약

오늘 반드시 기억할 것

  • 장애 대응은 2단계다 — 같은 provider·같은 모델 안의 auth profile 로테이션이 먼저, 그래도 실패하면 agents.defaults.model.fallbacks의 다음 모델로 전환한다.
  • 프로필 정렬은 타입 우선순위 + lastUsed 라운드로빈이고, lastGood은 의도적으로 무시한다 — 공정성을 최근 성공 이력보다 우선한 설계 선택이다.
  • 세션 pin은 provider 캐싱 효율을 위한 것이고, 실제 실패가 감지될 때만 풀린다.
  • 사용자 명시적 오버라이드는 자동 폴백을 타지 않는다 — 자동 폴백은 설정된 기본 모델 경로에서만 전체 체인을 순회한다.
  • failover는 비용 안전망이 아니다 — 세션 오버라이드 잔존, 조용한 기본 모델 전환, reasoning 기본값 변경으로 인한 실제 과금 사고가 GitHub 이슈로 보고돼 있다.
  • 에러 분류(retryable/terminal) 자체가 틀릴 수 있다 — 인증 오류를 일시적 오류로 오분류해 정상 키가 오래 막힌 버그 사례가 있다.
  • auth 저장소 자체도 2026.6.x부터 SQLite로 전환 중이며, 이 전환이 기존 프로필을 경고 없이 유실시킬 수 있다는 미해결 이슈가 있다.
