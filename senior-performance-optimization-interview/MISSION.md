# 성능 최적화 — 시니어 개발자 인터뷰 대비 학습 미션

## 목표

10년차 이상 백엔드 개발자로서 성능 최적화 인터뷰에서 요구되는 수준의 사고력과 언어를 익힌다.

단순 튜닝 팁 암기나 "인덱스 추가했더니 빨라졌어요" 수준이 아니라, 다음을 명확히 설명하고 방어할 수 있어야 한다:

- **측정 먼저 (Measure First)**: 추측이 아닌 데이터로 병목을 특정하는 절차
- **병목의 위치 판단**: CPU, DB I/O, 네트워크, 동시성, 메모리 중 어디인지 추론하는 프레임워크
- **개선 전략 선택과 trade-off**: 캐시, 쿼리 튜닝, 비동기화, 스케일아웃 각각의 비용과 한계
- **개선 효과 검증**: before/after 수치로 결과를 증명하는 습관
- **부하 테스트 설계**: 실제 운영 트래픽 패턴을 재현한 테스트로 병목을 조기 발견

## 대상 역할

- 백엔드 시니어 엔지니어 (10년차 이상)
- 아키텍처 결정에 참여하거나 팀의 성능 문제를 진단·해결하는 역할

## 전제 지식

이 워크스페이스는 다음을 이미 안다고 가정한다:

- HTTP API 개발 경험 (FastAPI, Django, Spring 등 어느 프레임워크든)
- SQL 기본 쿼리 작성 능력
- 리눅스 터미널 기본 사용
- 기본적인 비동기 개념 (async/await 또는 스레드 개념)

기본 Python 문법이나 SQL SELECT 문법 설명은 하지 않는다.

## 커리큘럼 개요 (20 Days)

| Day | 주제 |
|-----|------|
| 1 | 성능 병목 분석 프레임워크 — 측정 없이 최적화하지 않는다 |
| 2 | DB 쿼리 튜닝 — EXPLAIN 읽기와 N+1 문제 |
| 3 | 인덱스 최적화 — 언제 인덱스가 오히려 느려지나 |
| 4 | 느린 API 분석 — latency 분해와 병목 특정 |
| 5 | 캐시 전략 설계 — 히트율, 무효화, 일관성 trade-off |
| 6 | 비동기 처리와 동시성 — 언제 async가 해결책이고 언제 함정인가 |
| 7 | 부하 테스트 설계 — 현실적인 트래픽 시뮬레이션 |
| 8 | Profiling 실전 — CPU/메모리/I/O 프로파일링 해석 |
| 9 | 성능 지표 해석 — p50/p95/p99, throughput, error rate |
| 10 | DB Connection Pool 튜닝 — pool 고갈과 deadlock |
| 11 | 페이지네이션과 대용량 조회 최적화 |
| 12 | 외부 API 의존성과 timeout/circuit breaker 설계 |
| 13 | 메모리 최적화 — 누수 탐지와 GC 이해 |
| 14 | CDN과 정적 자원 최적화 |
| 15 | 읽기 복제본과 CQRS 적용 판단 |
| 16 | 파티셔닝과 샤딩 — 성능 vs 운영 복잡도 |
| 17 | 배치 처리 설계 — 대량 데이터 처리 전략 |
| 18 | Observability와 성능 회귀 탐지 |
| 19 | 성능 리뷰와 기술 리더십 커뮤니케이션 |
| 20 | 종합 실전 인터뷰 시뮬레이션 |

## 성공 기준

- 면접관이 "느린 API가 있는데 어떻게 접근하시겠어요?"라고 물었을 때 5단계 사고 순서를 즉시 말할 수 있다
- 쿼리 EXPLAIN 결과를 보고 문제 유형을 분류하고 개선 방향을 제시할 수 있다
- 캐시 적용 여부를 trade-off 포함해 설득력 있게 주장할 수 있다
- 부하 테스트 결과에서 병목이 DB인지 앱 서버인지 판별하는 근거를 설명할 수 있다
