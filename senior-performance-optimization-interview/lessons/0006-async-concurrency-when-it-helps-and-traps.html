1. 핵심 면접 질문

Primary Question

"동기 API를 비동기(async)로 바꾸면 항상 성능이 좋아지나요? 아니라면 어떤 경우에 그런가요?"

Variant 1

"이벤트 루프 기반 서버에서 특정 요청 하나의 처리가 느려지자, 그 요청과 전혀 관련 없는 다른 요청들까지 함께 느려졌습니다. 원인이 무엇일까요?"

Variant 2

"Python에는 GIL이 있어서 한 번에 하나의 스레드만 바이트코드를 실행한다고 알고 있는데, 그런데도 asyncio로 동시에 수천 개의 요청을 처리한다는 건 어떻게 가능한가요? 만약 CPU를 많이 쓰는 작업이라면 어떻게 하시겠어요?"

2. 왜 면접에서 묻는가

Day 1~5가 "병목을 찾고 DB/캐시 계층에서 줄이는 능력"이었다면, Day 6은 그다음 단계 — 같은 자원으로 더 많은 요청을 동시에 처리하기 위해 실행 모델 자체를 바꾸는 결정을 다룹니다. "비동기로 바꾸면 빨라진다"는 말은 실무에서 가장 많이 오용되는 성능 신화 중 하나입니다. 면접관은 지원자가 async를 "무조건 좋은 최신 기술"이 아니라 "기다리는 시간(I/O wait)을 낭비하지 않기 위한 특정 조건의 도구"로 이해하는지 확인합니다.

면접관이 이 질문으로 평가하는 것:

  • 동시성(Concurrency)과 병렬성(Parallelism)을 구분하는가 — 여러 작업을 "번갈아 진행"하는 것과 "동시에 실제로 실행"하는 것이 다르다는 것을 알고, 각각이 해결하는 문제가 다르다는 것을 아는가.
  • I/O bound와 CPU bound를 구분해 도구를 선택하는가 — 병목이 "기다리는 시간"인지 "계산하는 시간"인지에 따라 async, 멀티스레드, 멀티프로세스 중 무엇을 써야 하는지 판단하는가.
  • 이벤트 루프가 단일 스레드로 동작한다는 것의 함의를 아는가 — 이벤트 루프 안에서 단 하나의 blocking 호출이 그 루프 전체를 멈춘다는 것, 즉 async 코드 한 줄의 실수가 전체 서버의 모든 요청에 영향을 줄 수 있다는 것을 이해하는가.
  • 비동기화가 늘리는 복잡도를 인지하는가 — 에러 전파, 취소(cancellation), 디버깅 난이도, 커넥션 풀과의 상호작용처럼 async가 공짜로 얻는 성능이 아니라는 것을 아는가.

10년차 이하의 전형적인 약한 패턴: "async/await를 쓰면 논블로킹이라 빠릅니다"에서 답이 끝남. 시니어는 "이 작업이 기다리는 작업인지, 계산하는 작업인지부터 구분하고, 기다리는 작업이라도 이벤트 루프를 막는 코드가 하나라도 섞이면 오히려 전체 처리량이 무너진다"는 것까지 설명합니다.

3. 전제 개념 — 쉬운 한국어로

동시성(Concurrency) vs 병렬성(Parallelism) — 식당 종업원 비유

종업원 한 명이 여러 테이블을 맡고 있다고 생각해봅시다. 종업원은 한 테이블에 주문을 넣어두고(주방에 전달), 음식이 나오길 기다리는 동안 다른 테이블로 가서 주문을 받거나 물을 채워줍니다. 종업원은 항상 한 번에 한 가지 일만 하지만, 여러 테이블의 일이 "번갈아 가며" 동시에 진행되는 것처럼 보입니다. 이것이 동시성(Concurrency)입니다 — 실제로는 한 명이지만 기다리는 시간을 낭비하지 않고 여러 작업을 겹쳐서 진행하는 것.

반면 병렬성(Parallelism)은 종업원을 여러 명 두어 각자 다른 테이블을 동시에 물리적으로 응대하는 것입니다. 진짜로 동시에 여러 일이 벌어집니다. 종업원(CPU 코어)이 여러 명 있어야만 가능합니다.

핵심: 비동기(async)는 기본적으로 동시성을 얻는 기법입니다. 종업원(스레드/프로세스) 수를 늘리지 않고도, "기다리는 동안 노는 시간"을 없애 처리량을 늘립니다. CPU 코어를 더 써서 진짜로 동시에 계산하고 싶다면 그것은 병렬성의 문제이고, async만으로는 해결되지 않습니다.

Blocking I/O vs Non-blocking I/O

DB 쿼리를 보내고 결과를 기다리는 상황을 생각해봅시다. Blocking I/O는 결과가 올 때까지 그 스레드가 아무것도 못 하고 그냥 멈춰서 기다리는 방식입니다. 스레드는 살아있지만(메모리도 차지하고 있지만) CPU는 그 스레드에서 할 일이 없어 다른 일을 할 수도 있는데 못 하고 있는 상태입니다.

Non-blocking I/O는 "결과를 요청만 해두고, 결과가 준비되면 알려달라"고 한 뒤 그 스레드는 즉시 다른 일을 하러 가는 방식입니다. 결과가 준비되면(운영체제가 알려주면) 그 작업을 이어서 처리합니다. 이 방식 덕분에 하나의 스레드가 "기다리는 동안" 다른 수백 개의 요청을 처리할 수 있습니다.

이벤트 루프(Event Loop) — 단일 종업원의 작업 순번표

이벤트 루프는 "지금 당장 실행 가능한 작업이 무엇인지"를 계속 확인하며 순서대로 실행해주는 단일 스레드 스케줄러입니다. 어떤 작업이 I/O를 기다려야 하면(예: DB 응답 대기) 이벤트 루프는 그 작업을 잠시 옆으로 치워두고, 이미 준비된 다른 작업을 실행합니다. DB 응답이 도착하면 그 작업을 다시 실행 가능한 목록에 넣습니다. 이 모든 스케줄링이 단 하나의 스레드 안에서 벌어집니다 — 이것이 왜 "이벤트 루프를 막는 코드 한 줄"이 치명적인지의 근거입니다(4번 섹션에서 다룸).

async/await는 무엇을 대신 해주는 것인가

async def로 선언된 함수는 "이 함수는 중간에 기다릴 일이 생기면 실행을 잠시 양보(yield)할 수 있다"는 의미입니다. await는 "여기서 결과를 기다려야 하니, 그동안 이벤트 루프가 다른 작업을 실행해도 된다"는 명시적인 양보 지점입니다. 콜백(callback) 지옥 없이 마치 동기 코드처럼 순차적으로 읽히게 해주는 문법이지만, 내부 동작은 여전히 "기다리는 동안 다른 작업으로 전환하는" 협력적 멀티태스킹(Cooperative Multitasking)입니다.

스레드 풀(Thread Pool)이 필요한 이유

모든 작업이 async 친화적인 것은 아닙니다. 예를 들어 오래된 DB 드라이버나 파일 압축 라이브러리처럼 동기적으로만 동작하는(blocking) 코드를 이벤트 루프 안에서 그대로 호출하면 이벤트 루프 전체가 그 작업이 끝날 때까지 멈춥니다. 이런 경우 그 작업을 별도의 스레드 풀에 위임(offload)해서 실행하고, 완료되면 결과만 이벤트 루프로 돌려받는 방식을 씁니다. 즉, 스레드 풀은 "async로 만들 수 없는 blocking 코드를 이벤트 루프 밖으로 격리"하는 안전판 역할을 합니다.

4. 언제 async가 진짜 해결책인가 — I/O bound vs CPU bound

async 도입 여부를 결정하는 가장 근본적인 질문은 하나입니다: "이 작업의 시간은 대부분 '기다리는 시간'인가, '계산하는 시간'인가?"

구분정의예시병목 완화 도구
I/O BoundCPU는 거의 안 쓰고 외부 응답(네트워크, 디스크, DB)을 기다리는 시간이 대부분DB 쿼리, 외부 API 호출, 파일 읽기/쓰기, 사용자 입력 대기async I/O, non-blocking 소켓 — 기다리는 동안 다른 요청을 처리해 처리량(throughput)을 늘림
CPU Bound기다리는 시간 없이 CPU가 계속 계산을 수행이미지 리사이징, 암호화, 대용량 정렬, 복잡한 수치 연산멀티프로세스(Multiprocessing) — 여러 CPU 코어에 실제로 작업을 분산해 병렬로 계산
실무 감각: API 서버가 느린 이유의 대부분은 DB 쿼리, 외부 API 호출 같은 I/O 대기입니다. 이런 경우 async로 바꾸면 같은 서버 자원으로 훨씬 많은 동시 요청을 처리할 수 있어 처리량이 크게 늘어납니다. 하지만 요청 하나하나의 응답 시간(latency) 자체가 줄어드는 것은 아닙니다 — async는 "한 명의 요청을 더 빠르게 처리"하는 게 아니라 "동시에 더 많은 요청을 처리"하는 기법입니다. 만약 병목이 이미지 처리나 복잡한 계산 같은 CPU 작업이라면, async로 바꿔봐야 이벤트 루프가 그 계산이 끝날 때까지 다른 일을 못 하므로 오히려 문제가 그대로거나 더 나빠질 수 있습니다.

동시성 수준(Concurrency Level)과 처리량의 관계

Little's Law(L = λ × W, 시스템 내 평균 요청 수 = 요청 도착률 × 평균 처리 시간)로 설명하면, I/O 대기 시간이 긴 작업일수록 동시에 처리할 수 있는 요청 수를 늘리는 것(동시성)이 전체 처리량에 훨씬 큰 영향을 줍니다. 동기 blocking 방식은 스레드 하나당 요청 하나를 처리하므로, 요청이 각각 100ms씩 I/O를 기다리는 상황에서 스레드 수만큼만 동시에 처리할 수 있습니다. async는 스레드 하나로도 대기 중인 수백~수천 개의 요청을 동시에 들고 있을 수 있어 훨씬 적은 자원으로 높은 처리량을 얻습니다.

5. Event Loop Blocking — 가장 흔한 함정

왜 blocking 코드 한 줄이 전체 서버를 멈추는가

이벤트 루프는 단일 스레드에서 동작합니다. 이 말은 이벤트 루프가 "지금 실행 중인 코드"를 다른 작업으로 넘겨줄 수 있는 시점은 오직 await처럼 명시적으로 양보하는 지점뿐이라는 뜻입니다. 만약 async 함수 안에서 동기(blocking) 함수를 실수로 직접 호출하면 — 예를 들어 동기 방식의 DB 드라이버, time.sleep(), 무거운 CPU 계산, 동기 파일 I/O — 이벤트 루프는 그 코드가 끝날 때까지 다른 어떤 요청도 처리하지 못하고 완전히 멈춥니다.

정상 (모두 non-blocking)
1000개 요청이 이벤트 루프에서 번갈아 처리됨
blocking 호출 1건 발생
그 요청 처리 중 이벤트 루프 전체 정지
그 사이 대기 중인 다른 요청들
999개 요청 모두 응답 지연 (자신과 무관한 요청인데도)

이것이 면접에서 나온 Variant 1 질문 — "관련 없는 다른 요청까지 함께 느려졌다"의 정답입니다. 동기 멀티스레드 서버라면 스레드 하나가 느려져도 다른 스레드는 각자의 요청을 계속 처리합니다. 하지만 단일 이벤트 루프 서버에서는 blocking 코드 하나가 그 순간 모든 요청의 진행을 완전히 멈춥니다. 동시성 모델을 잘못 이해하면 "비동기라서 더 안정적일 것"이라는 기대와 정반대의 결과를 만듭니다.

흔한 blocking 실수 패턴

실수 패턴왜 문제인가해결
async 함수 안에서 동기 DB 드라이버 호출드라이버 내부가 blocking 소켓을 쓰면 응답 올 때까지 이벤트 루프 정지async 지원 드라이버로 교체(예: asyncpg, aiomysql) 또는 스레드 풀로 offload
무거운 JSON 파싱/직렬화, 이미지 처리를 async 핸들러 안에서 직접 실행CPU 연산이라 await할 지점이 없어 그 시간만큼 이벤트 루프가 통째로 멈춤스레드 풀 executor로 위임하거나, CPU가 큰 작업은 별도 프로세스/워커 큐로 분리
time.sleep()을 async 코드에 그대로 사용비동기 버전인 asyncio.sleep()과 달리 스레드 자체를 멈춤await asyncio.sleep()으로 교체
동기 로깅/모니터링 라이브러리가 네트워크로 로그를 동기 전송평소엔 안 보이다가 로그 수집 서버가 느려지는 순간 전체 서비스가 함께 느려짐비동기 전송 지원 라이브러리 사용 또는 로컬 버퍼링 후 별도 워커에서 전송
실무 감각: 이런 blocking 실수는 평상시 부하가 낮을 때는 거의 드러나지 않습니다. 트래픽이 몰릴 때, 또는 blocking 대상(느린 DB, 느린 로그 서버)이 실제로 느려지는 순간에야 "왜 전체 서비스가 한꺼번에 느려졌지?"라는 형태로 터집니다. 그래서 이벤트 루프 지연(Event Loop Lag/Delay)을 별도 지표로 모니터링하는 것이 중요합니다 — 이벤트 루프가 예정된 콜백을 실제로 실행하기까지 걸린 지연 시간을 측정하면, blocking 코드가 섞여 있는지 조기에 발견할 수 있습니다.

6. GIL과 진짜 병렬 처리가 필요한 경우

GIL(Global Interpreter Lock)이란 무엇인가 — 쉬운 비유

CPython(표준 Python 구현체)에는 "한 번에 하나의 스레드만 Python 바이트코드를 실행할 수 있다"는 전역 락이 있습니다. 이것을 GIL이라고 부릅니다. 마치 하나의 마이크를 여러 사람이 돌려가며 쓰는 회의실과 같습니다 — 여러 사람(스레드)이 있어도 마이크(GIL)를 쥔 사람만 말할(코드를 실행할) 수 있고, 나머지는 마이크가 넘어오길 기다립니다.

그런데 왜 asyncio로 수천 개를 동시에 처리할 수 있는가

GIL은 "동시에 계산하는 것"을 막을 뿐, "기다리는 것"까지 막지는 않습니다. I/O 대기 중에는 스레드가 어차피 계산할 게 없으므로, 그 순간 GIL을 놓아줍니다(release). asyncio의 이벤트 루프는 대부분의 시간을 "I/O 응답을 기다리는" 상태로 보내기 때문에, GIL이 있어도 수천 개의 요청을 "번갈아" 효율적으로 처리할 수 있습니다. 즉, GIL은 CPU 계산의 병렬 처리를 막는 것이지, I/O 대기의 동시 처리를 막는 것이 아닙니다.

CPU bound 작업에는 GIL이 그대로 발목을 잡는다

이미지 리사이징처럼 순수 계산 작업을 여러 스레드로 나눠 돌려도, GIL 때문에 결국 한 번에 하나의 스레드만 실제로 계산하므로 스레드를 늘려도 처리 속도가 거의 개선되지 않습니다(오히려 컨텍스트 스위칭 비용으로 더 느려질 수 있음). 진짜 병렬 계산이 필요하다면 GIL의 영향을 받지 않는 별도의 프로세스(Multiprocessing)로 작업을 분산해야 합니다. 각 프로세스는 자신만의 인터프리터와 GIL을 가지므로 여러 CPU 코어를 실제로 동시에 활용할 수 있습니다.

작업 유형적합한 동시성 모델이유
I/O bound (DB, 외부 API, 파일 I/O)asyncio (이벤트 루프)대기 시간 동안 GIL을 놓아주므로 단일 스레드로도 높은 동시성 확보
CPU bound (이미지 처리, 암호화, 수치 연산)Multiprocessing / 별도 워커 프로세스GIL의 영향을 받지 않는 독립 프로세스로 실제 병렬 계산 확보
blocking-only 라이브러리를 async 서버에서 써야 할 때Thread Pool Executor로 위임이벤트 루프를 막지 않게 격리하되, GIL 때문에 계산 자체가 빨라지진 않음(격리 목적)
실무 감각: 이 구분을 못 하면 흔히 "CPU 작업이 느려서 스레드를 늘렸는데 왜 그대로지?"라는 문제에 빠집니다. 답은 GIL 때문에 스레드를 늘려도 실제 계산은 여전히 한 번에 하나씩 처리되기 때문입니다. 이 경우 정답은 스레드가 아니라 프로세스를 늘리는 것입니다. 다른 언어(Go, Java, Node.js의 Worker Thread 등)는 GIL과 유사한 제약이 없거나 다른 방식으로 동시성을 처리하므로, 이 논리는 Python/CPython에 특화된 것임을 인터뷰에서 명확히 구분해 말하는 것이 좋습니다.

7. 실무 연결

상황 1: 동기 API를 async로 마이그레이션했는데 처리량이 오히려 줄었다

원인은 ORM이 여전히 동기 드라이버를 쓰고 있었던 것이었습니다. async 핸들러 안에서 동기 DB 호출을 그대로 하고 있었고, 이 blocking 호출이 매 요청마다 이벤트 루프 전체를 멈추면서, 오히려 기존 멀티스레드 방식(스레드별로 독립적으로 blocking되던 방식)보다 못한 성능이 나왔습니다. async 드라이버로 교체한 뒤에야 기대한 처리량 개선이 나타났습니다.

상황 2: 특정 엔드포인트 하나가 느려지자 헬스체크(health check)까지 타임아웃이 났다

느려진 엔드포인트 안에 동기 방식의 대용량 CSV 파싱 코드가 섞여 있었습니다. 이 CPU 작업이 이벤트 루프를 점유하는 동안, 완전히 무관한 /health 엔드포인트조차 응답할 수 없었습니다. 이 작업을 별도 워커 큐(백그라운드 작업)로 옮겨 API 서버의 이벤트 루프에서 완전히 분리했습니다.

상황 3: 스레드 풀 크기를 기본값 그대로 뒀더니 특정 시간대에 요청이 대기열에 쌓였다

blocking 작업을 스레드 풀로 offload하고 있었는데, 스레드 풀 크기가 작아 동시에 들어오는 blocking 작업 수를 감당하지 못했습니다. 스레드 풀 자체가 새로운 병목이 된 경우입니다. 스레드 풀 크기를 다운스트림(DB 커넥션 풀 등)의 실제 동시 처리 한계에 맞춰 조정하고, 그 이상은 대기열에서 명시적으로 거부(backpressure)하도록 바꿨습니다.

상황 4: 비동기 작업 하나가 실패했는데 에러가 조용히 사라졌다

asyncio.create_task()로 fire-and-forget 형태로 실행한 작업에서 예외가 발생했지만, 아무도 그 태스크의 결과를 await하지 않아 예외가 로그에도 남지 않고 그냥 사라졌습니다. 비동기화는 실행 흐름을 분리하는 만큼 예외 전파 경로도 함께 설계해야 한다는 것을 깨닫고, 모든 백그라운드 태스크에 예외 핸들러를 명시적으로 붙이도록 규칙을 정했습니다.

8. 답변 사고 순서

  • 1
    병목이 I/O 대기인지 CPU 계산인지부터 구분한다

    "비동기로 바꾸면 빨라지나요"라는 질문에 바로 예/아니오로 답하지 않고, 병목의 성격을 먼저 확인합니다. I/O bound라면 async가 처리량을 늘리는 좋은 도구지만, CPU bound라면 async만으로는 해결되지 않습니다.

  • 2
    동시성과 병렬성 중 무엇이 필요한지 판단한다

    "더 많은 요청을 동시에 들고 있어야 하는가(동시성 → async)"인지 "계산 자체를 여러 코어에 나눠야 하는가(병렬성 → 멀티프로세스)"인지 구분합니다.

  • 3
    이벤트 루프를 막을 수 있는 코드가 있는지 점검한다

    동기 드라이버, 무거운 계산, time.sleep() 같은 blocking 코드가 async 경로 안에 섞여 있지 않은지 확인하고, 있다면 async 전용 라이브러리로 교체하거나 스레드 풀/별도 프로세스로 격리합니다.

  • 4
    비동기화가 늘리는 복잡도(에러 전파, 취소, 디버깅)를 함께 설계한다

    fire-and-forget 태스크의 예외 처리, 타임아웃/취소 전파, 다운스트림(DB 커넥션 풀, 스레드 풀) 용량과의 정합성을 함께 검토합니다.

  • 5
    처리량과 지연 시간을 구분해 실측으로 검증한다

    async 도입 후 응답 시간(latency)이 아니라 동시 처리 가능한 요청 수(throughput)가 실제로 늘었는지, 이벤트 루프 지연 지표가 정상 범위인지 측정합니다.

9. 핵심 답변 예시

면접관: "동기 API를 비동기(async)로 바꾸면 항상 성능이 좋아지나요? 아니라면 어떤 경우에 그런가요?"

STRONG ANSWER

항상 좋아지는 것은 아니고, 병목이 무엇인지에 따라 다릅니다. 이 API가 느린 이유가 DB 쿼리나 외부 API 호출처럼 응답을 기다리는 시간 때문이라면, 비동기로 바꾸는 것이 효과적입니다. 동기 방식에서는 요청 하나당 스레드 하나가 그 대기 시간 동안 아무것도 못 하고 묶여 있지만, 비동기 방식에서는 그 대기 시간 동안 같은 스레드가 다른 요청을 처리할 수 있어 같은 자원으로 훨씬 많은 동시 요청을 처리할 수 있습니다. 다만 이건 처리량(throughput)이 늘어나는 것이지, 요청 하나의 응답 시간(latency) 자체가 짧아지는 건 아닙니다.


반대로 병목이 이미지 처리나 복잡한 계산 같은 CPU 작업이라면 비동기로 바꿔도 개선되지 않습니다. 이벤트 루프는 단일 스레드로 동작하기 때문에, CPU 계산이 진행되는 동안은 await할 지점이 없어 그 계산이 끝날 때까지 이벤트 루프 전체가 멈추고, 오히려 그 순간 다른 모든 요청까지 지연시키는 위험이 있습니다. 이 경우는 async가 아니라 별도 프로세스로 계산을 분산하는 병렬 처리가 필요합니다.


그리고 실제로 마이그레이션할 때는, 코드 안에 동기 방식의 라이브러리 호출(예: 동기 DB 드라이버, time.sleep)이 섞여 있지 않은지가 핵심입니다. 이런 blocking 코드 하나가 async 경로에 남아 있으면 그 순간 이벤트 루프 전체가 멈춰서, 오히려 마이그레이션 전보다 안 좋아지는 경우를 실제로 겪은 적이 있습니다. 그래서 저는 async 도입을 결정하기 전에 병목의 성격(I/O vs CPU)을 먼저 확인하고, 도입 후에는 이벤트 루프 지연 지표와 실제 처리량 변화를 측정해서 효과를 검증합니다.

이 답변이 좋은 이유:

  • "항상 좋아지지 않는다"로 시작해 조건부 판단임을 명확히 함
  • I/O bound와 CPU bound를 구분해 각각 다른 도구(async vs 멀티프로세스)가 필요함을 설명
  • 처리량(throughput)과 응답 시간(latency)을 구분하는 정확한 언어를 사용
  • 이벤트 루프가 단일 스레드라는 것의 실무적 함의(blocking 코드의 위험)까지 연결
  • 실측 검증(이벤트 루프 지연, 처리량 변화)으로 답변을 마무리

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"네, async/await는 논블로킹 방식이라 항상 동기 방식보다 빠릅니다. 최신 프레임워크는 다 비동기를 지원하니까 무조건 바꾸는 게 좋습니다."

이 답변이 약한 이유

  • "항상 빠르다"는 전제가 틀렸음 — CPU bound 작업에는 해당되지 않음
  • 처리량과 응답 시간을 구분하지 못함 — 무엇이 어떻게 개선되는지에 대한 설명이 없음
  • 비동기화가 만드는 위험(blocking 코드 하나가 전체 서버를 멈추는 것)을 전혀 언급하지 않음
WEAK ANSWER — 또 다른 패턴

"GIL 때문에 Python은 멀티스레드가 의미 없으니, 성능이 필요하면 무조건 asyncio를 써야 합니다."

이 답변이 약한 이유

  • GIL은 CPU 병렬 계산을 막을 뿐, I/O 대기 동시 처리와는 무관하다는 것을 혼동함
  • CPU bound 작업에는 asyncio가 아니라 멀티프로세스가 필요하다는 것을 놓침
  • "무조건"이라는 단정으로 병목 유형에 따른 판단 과정을 생략함
STRONG ANSWER 요소 체크리스트
  • 동시성(Concurrency)과 병렬성(Parallelism)을 구분해 설명한다
  • I/O bound와 CPU bound에 따라 async와 멀티프로세스 중 적합한 도구를 다르게 제시한다
  • 처리량(throughput)과 응답 시간(latency)을 구분해서 async의 효과를 정확히 설명한다
  • 이벤트 루프가 단일 스레드라는 것과 blocking 코드가 만드는 위험을 연결해 설명한다
  • GIL이 I/O 대기가 아니라 CPU 계산의 병렬성만 제한한다는 것을 정확히 구분한다
  • 도입 후 이벤트 루프 지연, 처리량 변화를 실측해 효과를 검증하는 단계를 포함한다

11. Senior-level Follow-up Q&A

Q: 이벤트 루프 기반 서버에서 특정 요청 하나의 처리가 느려지자, 그 요청과 전혀 관련 없는 다른 요청들까지 함께 느려졌습니다. 원인이 무엇일까요?

모범 답변: 이벤트 루프는 단일 스레드에서 동작하기 때문에, 해당 요청 처리 경로 안에 동기(blocking) 코드 — 동기 DB 드라이버, 무거운 CPU 연산, time.sleep() 등 — 가 섞여 있었을 가능성이 큽니다. await할 지점이 없는 blocking 코드가 실행되는 동안 이벤트 루프 전체가 멈추고, 그 순간 대기 중이던 다른 모든 요청의 처리도 함께 멈춥니다. 해결책은 해당 코드를 async 전용 라이브러리로 교체하거나, 스레드 풀 executor로 위임해 이벤트 루프를 막지 않도록 격리하는 것입니다.

Q: Python에는 GIL이 있는데 왜 asyncio로 수천 개의 동시 요청을 처리할 수 있나요? CPU를 많이 쓰는 작업이라면 어떻게 하시겠어요?

모범 답변: GIL은 한 번에 하나의 스레드만 Python 바이트코드를 실행하게 막지만, I/O 대기 중에는 스레드가 어차피 계산할 게 없으므로 GIL을 놓아줍니다. asyncio는 대부분의 시간을 I/O 응답 대기로 보내기 때문에, GIL이 있어도 단일 스레드로 매우 많은 요청을 효율적으로 번갈아 처리할 수 있습니다. 반대로 CPU를 많이 쓰는 작업은 GIL 때문에 스레드를 늘려도 실제 계산은 한 번에 하나씩만 진행되므로 개선되지 않습니다. 이 경우는 별도의 프로세스(Multiprocessing)로 작업을 분산해 여러 CPU 코어를 실제로 병렬 활용하거나, 무거운 계산을 별도 워커 큐/서비스로 분리합니다.

Q: async로 마이그레이션한 뒤 스레드 풀 executor로 blocking 작업을 위임하고 있는데, 특정 시간대에 요청이 대기열에 계속 쌓입니다. 무엇을 점검하시겠어요?

모범 답변: 스레드 풀 자체가 새로운 병목이 됐을 가능성을 먼저 봅니다. 스레드 풀 크기가 다운스트림(DB 커넥션 풀 등)이 실제로 감당할 수 있는 동시 처리량보다 작다면, 아무리 스레드 풀에 작업을 위임해도 대기열에 계속 쌓입니다. 스레드 풀 크기, 큐 대기 시간, 다운스트림 자원의 동시 처리 한계를 함께 점검하고, 한계를 넘는 요청은 조용히 쌓이게 두지 않고 명시적으로 거부하거나 지연시키는 backpressure를 적용합니다.

Q: 비동기 작업(fire-and-forget)에서 예외가 발생했는데 아무 데도 로그가 남지 않았습니다. 왜 이런 일이 생기고, 어떻게 막으시겠어요?

모범 답변: 태스크를 생성만 하고 아무도 그 결과를 await하거나 콜백을 붙이지 않으면, 그 태스크 안에서 발생한 예외는 어디로도 전파되지 않고 조용히 사라질 수 있습니다. 비동기화는 실행 흐름을 호출자와 분리하는 만큼, 그 분리된 흐름의 예외 처리 경로도 명시적으로 설계해야 합니다. 모든 백그라운드 태스크에 예외 핸들러(또는 완료 콜백)를 명시적으로 붙이고, 예외 발생 시 로깅과 알림이 나가도록 표준화합니다.

Q: 비동기 처리를 도입하면 지연 시간(latency)도 개선되나요, 아니면 처리량(throughput)만 개선되나요?

모범 답변: 기본적으로는 처리량이 개선되는 것입니다. 요청 하나의 실제 작업 시간(DB 응답 시간 등)이 줄어드는 것이 아니라, 그 대기 시간 동안 다른 요청도 함께 처리할 수 있게 되어 같은 자원으로 더 많은 요청을 동시에 처리하는 것입니다. 다만 서버가 이미 스레드/커넥션 부족으로 큐잉 지연(queueing delay)을 겪고 있었다면, 동시성을 늘려 대기 없이 바로 처리되면서 체감 지연 시간도 개선될 수 있습니다. 이는 병목이 처리 자체가 아니라 "자원 대기"였던 경우에 한정됩니다.

12. 흔한 오해

오해 1

"비동기로 바꾸면 항상 더 빠르다"

비동기는 I/O 대기 시간을 낭비하지 않아 처리량을 늘리는 기법입니다. CPU bound 작업에는 효과가 없고, blocking 코드가 섞여 있으면 오히려 전체 서버 처리량이 악화될 수 있습니다.

오해 2

"GIL이 있으니 Python 멀티스레드/멀티프로세스는 항상 의미 없다"

GIL은 CPU 계산의 병렬 처리만 제한합니다. I/O bound 작업에서는 멀티스레드도 여전히 유효하고(GIL이 I/O 대기 중 풀리므로), CPU bound 작업은 멀티프로세스로 GIL을 우회해 실제 병렬 처리가 가능합니다.

오해 3

"async 함수 안에서는 모든 코드가 자동으로 non-blocking이 된다"

async def는 그 함수가 중간에 양보할 수 있다는 문법일 뿐, 내부에서 동기 라이브러리를 호출하면 그 부분은 여전히 blocking입니다. async 생태계 전체(라이브러리, 드라이버)가 non-blocking으로 맞춰져 있어야 효과가 있습니다.

오해 4

"동시성을 무한히 늘리면 처리량도 계속 늘어난다"

다운스트림(DB 커넥션 풀, 외부 API의 rate limit)에는 실제 처리 한계가 있습니다. 애플리케이션 서버의 동시성만 늘리면 다운스트림에 과부하를 주거나 커넥션 풀 고갈로 이어질 뿐, 전체 시스템의 처리량은 결국 가장 느린 다운스트림 자원에 의해 결정됩니다.

오해 5

"비동기화는 코드만 바꾸면 되는 문제이지 아키텍처 결정이 아니다"

비동기화는 에러 전파 경로, 취소(cancellation) 처리, 디버깅 난이도(스택 트레이스가 여러 태스크에 걸쳐 흩어짐), 다운스트림 용량 계획까지 함께 바꾸는 결정입니다. 문법만 바꾸고 이 책임들을 놓치면 운영 중 추적하기 어려운 버그로 이어집니다.
