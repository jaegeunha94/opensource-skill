1. 핵심 면접 질문

Primary Question

"Composite Index(복합 인덱스)를 만들 때 컬럼 순서가 왜 중요한가요? 순서를 잘못 정하면 어떤 일이 벌어지나요?"

Variant 1

"쓰기(INSERT/UPDATE)가 많은 테이블에 인덱스를 무분별하게 추가하면 왜 문제가 될 수 있나요?"

Variant 2

"Covering Index가 무엇이고, 언제 성능에 실질적인 도움이 되나요?"

2. 왜 면접에서 묻는가

이 질문은 "인덱스를 걸 줄 아느냐"가 아니라 "인덱스가 내부적으로 어떻게 동작하고, 어떤 비용을 대가로 얻는 이득인지 아느냐"를 봅니다. Day 2에서 "인덱스는 만능이 아니다"를 EXPLAIN 결과로 확인했다면, Day 3은 그 이유를 인덱스 내부 구조 수준까지 설명할 수 있는지를 요구합니다.

면접관이 이 질문으로 평가하는 것:

  • B-Tree 구조를 이해하고 있는가 — 인덱스가 정렬된 트리 구조라는 것을 알고, 그로부터 "왜 컬럼 순서가 중요한지", "왜 범위 조건 뒤 컬럼은 인덱스를 못 타는지"를 스스로 유추할 수 있는가.
  • 인덱스를 공짜라고 생각하지 않는가 — 인덱스는 읽기를 빠르게 하는 대신 쓰기(INSERT/UPDATE/DELETE)마다 인덱스 구조도 함께 갱신해야 하는 비용이 있다는 것을 아는가.
  • 설계 단계에서 트레이드오프를 말할 수 있는가 — "이 컬럼에 인덱스를 걸지 말지"를 읽기 빈도, 쓰기 빈도, 선택도, 테이블 크기를 근거로 판단하는가.
  • 실측 없이 인덱스를 추가하지 않는가 — "일단 인덱스부터 걸어보자"가 아니라 실행 계획과 쿼리 패턴을 먼저 확인하는 습관이 있는가.

10년차 이하의 전형적인 약한 패턴: "WHERE 절에 쓰는 컬럼마다 인덱스를 걸겠습니다." 이 답변은 인덱스의 이득만 보고 비용은 고려하지 않은 것으로, 면접관에게는 "실무에서 쓰기 성능 저하를 겪어본 적이 없다"는 신호로 읽힙니다.

3. 전제 개념 — 쉬운 한국어로

인덱스는 왜 필요한가 — 책의 "찾아보기"에 비유

인덱스가 없는 테이블에서 특정 행을 찾는 것은 목차도 찾아보기도 없는 두꺼운 책에서 특정 단어가 나온 페이지를 처음부터 한 장씩 넘기며 찾는 것과 같습니다(Seq Scan). 인덱스는 이 책 뒤에 "단어 — 페이지 번호"를 정렬해서 모아 둔 찾아보기(색인)입니다. 찾아보기가 있으면 원하는 단어를 훨씬 빨리 찾지만, 책 내용이 바뀔 때마다(행이 추가/수정/삭제될 때마다) 찾아보기도 함께 고쳐 써야 한다는 대가가 있습니다.

B-Tree 구조 감 잡기

대부분의 RDB(PostgreSQL, MySQL InnoDB 등)는 인덱스를 B-Tree(균형 트리) 구조로 저장합니다. 핵심 특성 두 가지만 기억하면 됩니다.

  • 정렬되어 있다 — B-Tree의 리프 노드(맨 아래 단계)는 인덱스를 건 컬럼 값 순서대로 정렬되어 있습니다. 그래서 WHERE age > 30처럼 범위 조건이나 ORDER BY age 정렬에도 인덱스를 그대로 활용할 수 있습니다.
  • 트리를 타고 내려가며 찾는다(로그 시간) — 루트에서 시작해 조건에 맞는 값이 있을 만한 가지로만 내려가므로, 테이블 크기가 100배가 되어도 탐색 깊이는 조금만 늘어납니다. 이것이 Seq Scan(테이블 크기에 비례해서 느려짐)과의 근본적인 차이입니다.
실무 감각: "인덱스가 정렬되어 있다"는 사실 하나가 Composite Index의 컬럼 순서 문제, 범위 조건 뒤 컬럼이 인덱스를 못 타는 이유, ORDER BY 최적화까지 모두 설명해줍니다. 이 레슨의 나머지는 사실 이 한 문장을 여러 상황에 적용하는 연습입니다.

인덱스가 걸리는 대가 — 쓰기 비용

인덱스는 읽기 전용 데이터 구조가 아닙니다. 테이블에 행을 INSERT하면, DB는 테이블에 행을 쓰는 것과 별개로 그 테이블에 걸린 모든 인덱스에도 새 항목을 정렬된 위치에 끼워 넣어야 합니다. UPDATE로 인덱스 걸린 컬럼 값이 바뀌면 기존 인덱스 항목을 지우고 새 위치에 다시 써야 합니다. 인덱스가 5개 걸린 테이블에 행 하나를 쓰면, 실제로는 테이블 쓰기 1번 + 인덱스 쓰기 5번이 발생하는 셈입니다.

4. Composite Index와 컬럼 순서

Composite Index(복합 인덱스)는 여러 컬럼을 묶어서 하나의 인덱스로 만든 것입니다. 예를 들어 (status, created_at) 순서로 인덱스를 만들면, B-Tree는 먼저 status 값으로 정렬되고, 같은 status 안에서 created_at으로 정렬됩니다. 전화번호부가 "성 → 이름" 순으로 정렬된 것과 같은 원리입니다 — 이름만으로는 전화번호부를 빠르게 뒤질 수 없지만, 성을 먼저 알면 그 안에서 이름 순으로 빠르게 찾을 수 있습니다.

왼쪽 우선 원칙 (Leftmost Prefix Rule)

(status, created_at) 인덱스가 있을 때:

  • WHERE status = 'active' → 인덱스 사용 가능 (왼쪽 컬럼만 사용)
  • WHERE status = 'active' AND created_at > '2026-01-01' → 인덱스 완전히 사용 가능
  • WHERE created_at > '2026-01-01' → 인덱스 사용 불가 (전화번호부에서 성 없이 이름만으로 찾으려는 것과 같음)
-- 인덱스: (status, created_at)

-- 인덱스 활용: status로 좁힌 뒤 그 안에서 created_at 범위 탐색
SELECT * FROM orders
WHERE status = 'pending' AND created_at > '2026-06-01';

-- 인덱스 활용 불가: 왼쪽 컬럼(status) 조건이 없음
SELECT * FROM orders
WHERE created_at > '2026-06-01';

범위 조건 뒤의 컬럼은 인덱스를 못 탄다

(a, b, c) 순서의 인덱스에서 a는 등호(=) 조건, b는 범위(>, BETWEEN) 조건으로 쓰이면, c는 인덱스로 좁혀지지 않습니다. B-Tree가 정렬된 것은 "a 안에서 b 순"까지이지, b가 범위로 흩어지는 순간 그 안에서의 c 정렬은 더 이상 연속적이지 않기 때문입니다. 그래서 실무에서는 등호 조건 컬럼을 앞에, 범위 조건 컬럼을 뒤에 두는 것이 원칙입니다.

Covering Index

Day 2에서 다룬 Index Only Scan을 만드는 조건이 바로 Covering Index입니다. SELECT하는 모든 컬럼이 인덱스 안에 이미 포함되어 있으면, DB는 인덱스만 읽고 실제 테이블 데이터 페이지에 접근할 필요가 없습니다.

-- 인덱스: (status, created_at) INCLUDE (id, total_amount)
-- 아래 쿼리는 테이블 데이터 페이지에 접근하지 않고 인덱스만으로 완결됨
SELECT id, total_amount FROM orders
WHERE status = 'pending' AND created_at > '2026-06-01';
주의: Covering Index는 SELECT *처럼 모든 컬럼을 가져오는 쿼리에는 적용하기 어렵습니다. 컬럼을 인덱스에 포함시킬수록 인덱스 자체의 크기와 쓰기 비용도 커지므로, "자주 실행되는 특정 쿼리"를 위한 좁고 명확한 최적화로 접근해야 합니다.

5. 인덱스의 숨은 비용

쓰기 성능 저하

인덱스가 많을수록 INSERT/UPDATE/DELETE는 느려집니다. 읽기 위주(주문 조회 페이지)와 쓰기 위주(로그 적재, 이벤트 스트림 저장) 테이블은 인덱스 전략이 달라야 합니다 — 후자는 인덱스를 최소화하는 것이 원칙입니다.

저장 공간과 캐시 오염

인덱스는 그 자체로 디스크 공간을 차지합니다. 테이블보다 인덱스가 더 큰 경우도 흔합니다. 더 중요한 문제는 메모리(버퍼 캐시)입니다. DB는 자주 쓰는 데이터/인덱스 페이지를 메모리에 캐싱하는데, 불필요한 인덱스가 많으면 정작 자주 쓰이는 인덱스가 캐시에서 밀려나 디스크 I/O가 늘어날 수 있습니다.

플래너 혼란

비슷한 조건을 커버하는 인덱스가 여러 개 있으면, 쿼리 플래너가 항상 최선의 인덱스를 고르지 못하고 통계 오차로 인해 차선의 인덱스를 선택하는 경우가 생깁니다. 인덱스가 많다고 무조건 유리한 것이 아닙니다.

락(Lock)과 인덱스 생성 비용

운영 중인 대형 테이블에 인덱스를 새로 만드는 작업 자체가 부하입니다. PostgreSQL의 CREATE INDEX CONCURRENTLY나 MySQL의 온라인 DDL처럼 락을 최소화하는 방법을 쓰지 않으면, 인덱스 생성 도중 테이블 쓰기가 막혀 장애로 이어질 수 있습니다.

6. 실무 연결

상황 1: 대시보드 쿼리 3종에 맞춰 인덱스 3개를 각각 추가했더니 쓰기가 느려짐

쿼리마다 인덱스를 하나씩 대응시키는 대신, 공통 접두어(왼쪽 우선 원칙)를 공유하는 Composite Index로 통합할 수 있는지 먼저 검토합니다. (status), (status, created_at), (status, created_at, user_id) 세 인덱스는 사실상 마지막 하나로 앞의 두 쿼리 패턴도 커버할 수 있습니다.

상황 2: 로그/이벤트 테이블에 인덱스를 걸었더니 INSERT 처리량이 급감함

초당 수천 건씩 쓰는 로그성 테이블은 인덱스를 최소화하고, 조회는 별도의 조회 최적화 저장소(예: 검색엔진, 집계 테이블, 컬럼 스토어)로 분리하는 것이 정석입니다. "쓰기 테이블"과 "조회 테이블"의 인덱스 전략을 같은 기준으로 가져가면 안 됩니다.

상황 3: 대형 테이블에 인덱스를 하나 추가했더니 서비스 응답이 잠깐 멎음

기본 CREATE INDEX는 테이블에 쓰기 락을 걸 수 있습니다. 운영 중인 테이블이라면 CREATE INDEX CONCURRENTLY(PostgreSQL) 같은 온라인 생성 방식을 쓰고, 트래픽이 적은 시간대에 진행하며, 생성 진행률과 실패 가능성(concurrently 생성은 실패 시 invalid 인덱스가 남을 수 있음)까지 고려합니다.

상황 4: 쿼리 조건 순서를 SQL에서 바꿨더니 실행 계획도 바뀔 거라 기대했지만 아무 변화 없음

흔한 오해입니다. WHERE b = 1 AND a = 2처럼 SQL 문장에서 조건 순서를 바꿔도 플래너는 인덱스 (a, b)를 그대로 활용합니다. 중요한 것은 SQL에 조건을 나열한 순서가 아니라 인덱스 자체에 컬럼을 정의한 순서입니다.

7. 답변 사고 순서

  • 1
    실제 쿼리 패턴을 먼저 확인한다

    어떤 컬럼이 WHERE/JOIN/ORDER BY에 자주 쓰이는지, 얼마나 자주 실행되는 쿼리인지를 슬로우 쿼리 로그나 pg_stat_statements 같은 도구로 근거를 확보합니다. "느릴 것 같아서"가 아니라 실제로 자주 실행되고 느린 쿼리를 대상으로 합니다.

  • 2
    선택도를 확인해 인덱스 효과가 있을지 판단한다

    조건 컬럼의 카디널리티(값의 종류 수)와 선택도를 봅니다. 선택도가 낮으면(예: boolean) 인덱스를 걸어도 플래너가 안 쓸 수 있으므로 Composite Index로 다른 컬럼과 조합하는 것을 고려합니다.

  • 3
    Composite Index라면 컬럼 순서를 정한다

    등호 조건 컬럼을 앞에, 범위 조건 컬럼을 뒤에 둡니다. 여러 쿼리 패턴이 있다면 왼쪽 우선 원칙으로 하나의 인덱스가 여러 쿼리를 커버할 수 있는지 검토합니다.

  • 4
    쓰기 비용과 저장 비용을 함께 고려한다

    이 테이블의 쓰기 빈도가 얼마나 되는지, 이미 인덱스가 몇 개 있는지 확인합니다. 쓰기가 잦은 테이블이라면 인덱스 추가가 정말 필요한 최소한인지 재검토합니다.

  • 5
    운영 환경에 안전하게 적용하고 효과를 검증한다

    대형 테이블이면 온라인 인덱스 생성 방식을 쓰고, 적용 후 EXPLAIN ANALYZE로 실행 계획이 실제로 바뀌었는지, 목표 쿼리의 실행 시간이 개선됐는지 확인합니다.

8. 핵심 답변 예시

면접관: "Composite Index를 만들 때 컬럼 순서가 왜 중요한가요? 순서를 잘못 정하면 어떤 일이 벌어지나요?"

STRONG ANSWER

대부분의 RDB 인덱스는 B-Tree로 저장되고, B-Tree는 정렬된 구조입니다. Composite Index (a, b)는 먼저 a 값으로 정렬되고, 같은 a 값 안에서 b가 정렬됩니다. 그래서 a 조건 없이 b만으로 조회하면, 이 정렬 구조를 활용할 수 없어 인덱스를 타지 못합니다 — 전화번호부에서 성 없이 이름만으로 빠르게 찾을 수 없는 것과 같은 이유입니다. 이를 왼쪽 우선 원칙이라고 부릅니다.


순서를 정할 때는 두 가지를 봅니다. 첫째, 등호 조건으로 쓰이는 컬럼을 앞에, 범위 조건(>, BETWEEN)으로 쓰이는 컬럼을 뒤에 둡니다. 범위 조건이 걸리는 순간 그 뒤 컬럼의 정렬 연속성이 깨지기 때문입니다. 둘째, 여러 쿼리 패턴이 있다면 왼쪽 우선 원칙을 활용해 하나의 인덱스로 여러 쿼리를 커버할 수 있는 순서를 찾습니다. (status, created_at) 인덱스는 status만 쓰는 쿼리와 status + created_at을 함께 쓰는 쿼리를 모두 커버합니다.


순서를 잘못 정하면 인덱스는 존재하지만 플래너가 활용하지 못해 Seq Scan으로 빠지고, 인덱스 유지 비용(쓰기 비용, 저장 공간)만 부담하고 이득은 못 얻는 상황이 됩니다. 그래서 실제 쿼리 패턴을 먼저 확인한 다음 순서를 정하고, EXPLAIN ANALYZE로 의도한 인덱스가 실제로 사용되는지 검증합니다.

이 답변이 좋은 이유:

  • B-Tree의 정렬 특성이라는 근본 원리에서 컬럼 순서 규칙을 유도함(암기가 아니라 이해)
  • 등호 vs 범위 조건 배치 원칙을 명시
  • 왼쪽 우선 원칙으로 인덱스 개수를 줄이는 실무적 사고까지 포함
  • 순서를 잘못 정했을 때의 구체적인 손해(인덱스 유지 비용은 지불하되 이득은 없음)를 언급
  • 검증 단계(EXPLAIN ANALYZE)를 포함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"자주 조회되는 컬럼 순서대로 인덱스에 넣으면 됩니다."

이 답변이 약한 이유

  • B-Tree 정렬 구조를 근거로 들지 않아 "왜"에 대한 답이 없음
  • 등호 조건과 범위 조건을 구분하지 않음 — 순서 규칙의 핵심을 놓침
  • "자주 조회된다"는 것과 "왼쪽 우선 원칙을 만족한다"는 것은 다른 기준인데 이를 혼동함
WEAK ANSWER — 또 다른 패턴

"WHERE 절에 쓰는 컬럼은 모두 인덱스를 걸어두는 게 안전합니다."

이 답변이 약한 이유

  • 인덱스의 쓰기 비용, 저장 비용을 전혀 고려하지 않음
  • 선택도가 낮은 컬럼에는 인덱스가 무용할 수 있다는 것을 모름
  • "안전하다"는 표현 자체가 트레이드오프 사고 없이 리스크 회피로만 접근했다는 신호
STRONG ANSWER 요소 체크리스트
  • B-Tree의 정렬 구조로 컬럼 순서 규칙을 설명한다
  • 등호 조건 컬럼을 앞, 범위 조건 컬럼을 뒤에 두는 이유를 근거와 함께 말한다
  • 인덱스의 읽기 이득과 쓰기/저장 비용을 함께 언급한다
  • 왼쪽 우선 원칙으로 인덱스 개수를 통합하는 실무적 판단을 보여준다
  • 운영 환경 적용 시 락/온라인 생성 이슈까지 고려한다

10. Senior-level Follow-up Q&A

Q: 인덱스를 5개 걸어둔 테이블에 대량 INSERT를 해야 합니다. 성능을 어떻게 지킬까요?

모범 답변: 대량 적재 시에는 가능하면 인덱스를 임시로 제거하고 데이터를 적재한 뒤 인덱스를 재생성하는 방법이 각 행마다 인덱스를 갱신하는 것보다 훨씬 빠른 경우가 많습니다(배치 인덱스 빌드가 증분 갱신보다 효율적). 다만 이 테이블이 서비스 중이라 인덱스 제거 구간에 조회 성능 저하를 감수할 수 없다면, 트래픽이 적은 시간대에 배치로 나눠 적재하거나 파티셔닝된 신규 테이블에 적재 후 스왑하는 방식을 검토합니다.

Q: Covering Index를 만들면 항상 더 빠른가요?

모범 답변: 항상은 아닙니다. Covering Index는 자주 실행되는 특정 쿼리의 Index Only Scan을 위해 컬럼을 인덱스에 추가로 포함시키는 것인데, 포함되는 컬럼이 많아질수록 인덱스 크기와 쓰기 비용이 커집니다. 게다가 포함된 컬럼이 자주 갱신되는 컬럼이라면, 그 갱신마다 인덱스도 함께 갱신해야 해서 오히려 손해가 될 수 있습니다. 실행 빈도가 매우 높고 갱신이 적은 컬럼 조합에 한해 적용하는 것이 원칙입니다.

Q: (a, b) 인덱스가 있는데 WHERE a = 1 OR b = 2 조건은 인덱스를 잘 활용하나요?

모범 답변: 아닙니다. Composite Index는 AND로 연결된 조건에서 왼쪽 우선 원칙을 따라 활용되지만, OR 조건은 각 컬럼을 독립적으로 만족하는 행을 모두 찾아야 하므로 이 정렬 구조를 그대로 쓸 수 없습니다. 이런 경우 플래너는 Seq Scan을 택하거나, DB에 따라 각 컬럼에 대해 별도로 인덱스를 스캔한 뒤 결과를 합치는 Bitmap Index Scan/Index Merge 같은 전략을 쓰기도 합니다. OR가 잦은 쿼리라면 컬럼별 단일 인덱스를 따로 두는 것이 더 예측 가능할 수 있습니다.

Q: 운영 중인 대형 테이블에 인덱스를 새로 만들어야 합니다. 어떻게 접근하시겠어요?

모범 답변: 기본 CREATE INDEX는 테이블에 쓰기를 막는 락을 걸 수 있으므로, PostgreSQL이라면 CREATE INDEX CONCURRENTLY 같은 온라인 생성 방식을 씁니다. 이 방식은 락을 최소화하는 대신 시간이 더 걸리고, 중간에 실패하면 invalid 상태의 인덱스가 남을 수 있어 실패 시 정리 절차도 함께 준비합니다. 가능하면 트래픽이 적은 시간대에 진행하고, 생성 중 테이블 상태와 진행률을 모니터링합니다.

Q: 인덱스가 있는데도 정렬(ORDER BY) 쿼리가 여전히 느립니다. 왜일까요?

모범 답변: 인덱스의 정렬 순서와 ORDER BY의 정렬 순서(컬럼, 오름차순/내림차순)가 일치하지 않으면 인덱스를 정렬에 활용하지 못하고 별도의 정렬 작업(Sort 단계)이 추가로 발생합니다. 또한 WHERE 조건에서 이미 인덱스의 왼쪽 컬럼들을 다 소진했는지도 확인해야 합니다 — 예를 들어 (a, b) 인덱스에서 WHERE a IN (1,2,3) ORDER BY b는 a가 여러 값이라 b 기준 전역 정렬을 인덱스만으로 보장하지 못할 수 있습니다. EXPLAIN에서 Sort 노드가 보이는지 직접 확인하는 것이 가장 확실합니다.

11. 흔한 오해

오해 1

"인덱스는 많을수록 좋다"

인덱스는 읽기를 빠르게 하는 대신 모든 쓰기(INSERT/UPDATE/DELETE)마다 갱신 비용이 붙습니다. 쓰기가 잦은 테이블에서는 꼭 필요한 인덱스만 최소한으로 유지해야 합니다.

오해 2

"SQL 문장에서 WHERE 조건을 쓴 순서가 인덱스 활용 순서를 결정한다"

플래너가 참고하는 것은 SQL 문장의 조건 나열 순서가 아니라 인덱스 자체에 정의된 컬럼 순서입니다. WHERE b = 1 AND a = 2로 써도 인덱스 (a, b)는 그대로 활용됩니다.

오해 3

"Composite Index (a, b)가 있으면 b만으로 조회할 때도 어느 정도는 빨라진다"

왼쪽 우선 원칙에 따라 왼쪽 컬럼(a) 조건 없이 b만으로 조회하면 이 인덱스는 사실상 활용되지 않습니다(일부 DB의 Index Skip Scan 같은 예외적 최적화를 제외하면). b만 자주 단독으로 조회된다면 별도 인덱스가 필요합니다.

오해 4

"Primary Key는 인덱스와 별개다"

대부분의 RDB에서 Primary Key는 자동으로 (Unique) 인덱스가 생성됩니다. 이미 커버되는 조회 패턴에 중복 인덱스를 또 만드는 실수를 피해야 합니다.

오해 5

"인덱스를 걸면 즉시 효과가 나타난다"

인덱스를 새로 만든 직후에는 통계 정보가 갱신되지 않아 플래너가 여전히 옛 계획을 쓸 수 있습니다. DB에 따라 ANALYZE를 명시적으로 실행해 통계를 갱신해야 플래너가 새 인덱스를 제대로 고려합니다.
