1. 핵심 면접 질문

Primary Question

"새로운 기능을 배포하기 전에 부하 테스트를 어떻게 설계하시겠어요? 실제 트래픽 패턴과 다르게 테스트하면 어떤 문제가 생기나요?"

Variant 1

"부하 테스트에서는 초당 5,000 요청까지 문제없이 처리했는데, 실제 운영에서는 초당 1,000 요청 수준에서 이미 에러가 발생했습니다. 무엇이 잘못됐을까요?"

Variant 2

"부하 테스트 결과 응답 시간이 급격히 나빠지는 시점을 확인했습니다. 그 시점에서 병목이 애플리케이션 서버인지 DB인지 어떻게 구분하시겠어요?"

2. 왜 면접에서 묻는가

Day 1~6이 "이미 발생한 병목을 찾아 원인을 특정하고 완화하는 능력"이었다면, Day 7은 그 반대 방향 — 병목이 운영 환경에서 터지기 전에, 실험실 조건에서 미리 재현해 조기에 찾아내는 능력을 다룹니다. "부하 테스트를 돌려봤습니다"라는 말은 아무 의미가 없습니다. 면접관은 지원자가 부하 테스트 설계 자체가 실제 트래픽을 얼마나 정직하게 반영하는지를 검증하는 습관이 있는지, 그리고 테스트 결과에서 병목의 위치를 논리적으로 좁혀가는 절차가 있는지를 확인합니다.

면접관이 이 질문으로 평가하는 것:

  • 부하 패턴을 현실적으로 설계하는가 — 요청을 균일하게 쏟아붓는 것이 아니라, 실제 사용자 행동(버스트, 리트라이, 세션 지속 시간)을 반영해 트래픽을 만드는가.
  • 테스트 환경과 운영 환경의 차이를 인지하는가 — 캐시 웜업 상태, 데이터 볼륨, 네트워크 토폴로지, 커넥션 풀 크기 같은 조건이 다르면 결과도 무의미해진다는 것을 아는가.
  • 부하 테스트 결과에서 병목을 구조적으로 좁혀가는가 — "느려졌다"에서 멈추지 않고, 앱 서버/DB/네트워크/외부 의존성 중 어디가 한계에 도달했는지 지표로 증명하는가.
  • 목표(SLO)를 먼저 정의하고 테스트를 설계하는가 — "얼마나 빨라야 하는가"를 먼저 정하지 않은 채 막연히 부하를 올리기만 하는 것은 방향 없는 테스트라는 것을 아는가.

10년차 이하의 전형적인 약한 패턴: "JMeter로 동시 사용자 1,000명 돌려봤는데 괜찮았습니다"에서 답이 끝남. 시니어는 "그 1,000명이 실제 사용자 행동 패턴(로그인 후 여러 페이지 이동, 일부는 장바구니만 보고 이탈)을 반영했는지, 테스트 데이터 볼륨이 운영 DB와 비슷했는지, 그리고 결과가 나빠진 지점에서 CPU/DB 커넥션/네트워크 중 무엇이 포화 상태였는지"까지 설명합니다.

3. 전제 개념 — 쉬운 한국어로

부하 테스트(Load Test)란 — 다리의 하중 실험 비유

새로 지은 다리를 개통하기 전, 실제 차량이 다니기 전에 미리 무거운 화물차를 여러 대 세워두고 다리가 버티는지 확인합니다. 소프트웨어의 부하 테스트도 똑같습니다. 실제 사용자가 몰리기 전에, 가상의 요청을 인위적으로 만들어 시스템에 흘려보내고, 시스템이 목표한 트래픽 수준에서 정상적으로 동작하는지 확인하는 것입니다. 다리 실험에서 실제 차량 무게 분포와 다른 하중(예: 한쪽에만 집중된 무게)을 쓰면 결과가 의미 없듯, 부하 테스트도 실제 트래픽 패턴과 얼마나 비슷한가가 결과의 신뢰도를 결정합니다.

가상 사용자(Virtual User, VU)란 무엇인가

부하 테스트 도구는 실제 브라우저나 사용자 대신, 요청을 자동으로 반복해서 보내는 스크립트를 여러 개 동시에 실행합니다. 이 스크립트 하나하나를 가상 사용자(VU)라고 부릅니다. "동시 사용자 1,000명 테스트"란 실제로 1,000개의 스크립트가 동시에 각자의 시나리오(로그인 → 목록 조회 → 상세 조회 → 주문)를 반복 실행하는 것을 의미합니다. 중요한 것은 VU 수 자체가 아니라, 각 VU가 실제 사용자처럼 행동하는가입니다 — 실제 사용자는 페이지를 본 뒤 몇 초간 머물다가 다음 행동을 하는데, 이 "생각하는 시간(think time)"을 넣지 않으면 실제보다 훨씬 공격적인 트래픽이 만들어집니다.

초당 요청 수(RPS/Throughput)와 동시 사용자 수의 차이

동시 사용자 수(VU)와 초당 요청 수(RPS)는 다른 개념입니다. VU 1,000명이 각자 10초에 한 번씩 요청을 보낸다면 RPS는 대략 100 정도입니다. 반대로 VU 100명이 1초에 한 번씩 요청을 보내면 RPS는 100으로 동일하지만 서버가 느껴지는 부하 패턴은 다를 수 있습니다(예: 커넥션 수, 세션 상태 유지 비용). 면접에서 "동시 사용자 수"만 말하고 "초당 요청 수"를 말하지 않으면, 실제로 서버에 가해지는 부하를 정량화하지 못한 것으로 보입니다.

Ramp-up이란 — 왜 요청을 한 번에 쏟아붓지 않는가

Ramp-up은 목표 부하까지 트래픽을 점진적으로 늘려가는 구간입니다. 처음부터 목표 부하(예: 5,000 RPS)를 한 번에 쏟아부으면, 시스템이 어느 지점부터 무너지기 시작하는지 정확히 알 수 없습니다 — 이미 임계점을 훌쩍 넘겨서 테스트가 시작된 셈이기 때문입니다. 트래픽을 0에서 목표치까지 서서히 올리면서 각 구간의 응답 시간과 에러율을 관찰하면, 시스템이 감당할 수 있는 한계선(breaking point)을 훨씬 정확하게 찾을 수 있습니다. 이는 실제 트래픽 증가 양상(마케팅 이벤트로 트래픽이 서서히 몰리는 경우, 갑자기 스파이크가 오는 경우)에 따라서도 다르게 설계해야 합니다.

포화(Saturation)와 병목(Bottleneck) — 가장 먼저 한계에 도달하는 자원

시스템은 CPU, 메모리, DB 커넥션, 네트워크 대역폭, 스레드/이벤트 루프 등 여러 자원으로 구성됩니다. 부하를 올리다 보면 이 중 하나의 자원이 가장 먼저 100%에 가깝게 사용되며 한계에 도달합니다(포화). 이 자원이 전체 시스템 처리량을 결정하는 병목이 됩니다. 부하 테스트의 핵심 목적은 "언제 느려지는가"를 확인하는 것이 아니라 "무엇이 가장 먼저 한계에 도달하는가"를 찾는 것입니다. Day 1에서 다룬 USE Method(Utilization, Saturation, Errors)가 바로 이 단계에서 다시 등장합니다.

4. 부하 패턴 설계 — ramp-up, spike, soak, stress

목적에 따라 서로 다른 부하 패턴을 설계해야 합니다. "부하 테스트 한 번 돌렸다"가 아니라, 확인하려는 질문에 맞는 패턴을 선택했는지가 시니어와 주니어를 가르는 지점입니다.

패턴목적트래픽 모양확인하는 것
Smoke Test기본 동작 확인매우 낮은 고정 부하 (VU 1~5)테스트 스크립트와 환경 자체가 정상인지
Load Test기대 트래픽 수준에서의 정상 동작 확인목표 RPS까지 ramp-up 후 일정 시간 유지SLO(목표 latency/error rate)를 만족하는지
Stress Test시스템의 한계선(breaking point) 확인목표치를 넘어 계속 증가시킴어느 지점에서 에러율이 급증하고 무엇이 포화되는지
Spike Test급격한 트래픽 폭증 대응력 확인짧은 시간에 급격히 증가 후 급격히 감소Auto Scaling 반응 속도, 큐잉/거절 동작
Soak Test (Endurance)장시간 운영 시 누적 문제 확인중간 수준 부하를 수 시간~하루 유지메모리 누수, 커넥션 누수, 디스크/로그 누적 문제
실무 감각: 블랙프라이데이 같은 이벤트 트래픽 대비라면 Spike Test가 핵심이고, 배치 처리 서버처럼 오래 켜져 있는 서비스라면 Soak Test에서 메모리 누수나 커넥션 누수가 드러납니다. 신규 기능 배포 전에는 보통 Load Test로 SLO 충족을 확인하고, 용량 계획(capacity planning)을 위해서는 Stress Test로 한계선을 찾습니다. "부하 테스트"라는 한 단어로 뭉뚱그리지 않고 목적에 맞는 패턴을 고르는 것 자체가 시니어의 신호입니다.

현실적인 트래픽 패턴 재현 — 균일한 부하의 함정

가장 흔한 실수는 모든 VU가 동일한 간격으로 동일한 요청을 반복하는 것입니다. 실제 트래픽은 균일하지 않습니다: 특정 엔드포인트에 요청이 몰리고(예: 상품 상세 조회가 검색보다 10배 많음), 사용자 행동에 think time이 있고, 리트라이와 실패한 요청의 재시도가 섞여 있습니다. 현실적인 부하 테스트는 다음을 반영해야 합니다.

  • 엔드포인트별 트래픽 비율: 실제 접근 로그(access log)를 분석해 어떤 엔드포인트가 얼마나 호출되는지 비율을 반영
  • Think Time: 실제 사용자가 화면을 보고 다음 행동을 하기까지의 지연을 시나리오에 포함
  • 데이터 분포: 인기 상품 몇 개에 조회가 몰리는 것처럼, 요청 대상 데이터의 편향(hot key)을 재현 — 이는 캐시 히트율과 DB 부하 분산 양상에 직접 영향을 줌
  • 인증/세션 상태: 매 요청마다 새로 로그인하는 것이 아니라, 실제 세션 유지 시간과 유사하게 토큰을 재사용
실무 감각: "부하 테스트에서는 잘 버텼는데 운영에서 무너졌다"는 사례의 상당수는 테스트 트래픽이 균일했기 때문입니다. 균일한 트래픽은 캐시가 항상 웜업된 상태를 유지하게 만들어 실제보다 훨씬 좋은 결과를 냅니다. 반면 실제 트래픽은 특정 데이터에 쏠리는 구간이 있고, 그 구간에서 캐시 미스가 몰리면서 DB에 갑자기 부하가 집중되는 상황이 재현되지 않습니다.

테스트 환경과 운영 환경의 격차(Environment Parity)

부하 테스트 결과가 신뢰할 만하려면 테스트 환경이 운영 환경과 최대한 유사해야 합니다. 다음 항목이 다르면 결과가 왜곡됩니다.

격차 요소왜 결과를 왜곡하는가
데이터 볼륨테스트 DB에 데이터가 적으면 인덱스 효과, 쿼리 플랜이 운영과 달라져 실제보다 빠르게 나옴
인스턴스 스펙/개수운영보다 작은 스펙에서 테스트하면 병목 위치 자체가 다르게 나타남
캐시 웜업 상태테스트 직전에 캐시를 미리 채워두면 실제 콜드 스타트 상황의 부하가 재현되지 않음
네트워크 토폴로지같은 데이터센터 내부에서 테스트하면 실제 사용자의 지리적 지연(RTT)이 빠짐
다운스트림 의존성외부 API를 mock으로 대체하면 실제 외부 의존성의 latency/장애가 반영되지 않음

5. 도구 선택 — k6, Locust, JMeter, wrk

도구 선택 자체는 지엽적인 지식처럼 보이지만, 면접관은 도구의 동작 모델 차이를 아는지를 통해 부하 테스트의 정확도에 대한 이해도를 확인합니다.

도구실행 모델장점주의할 점
k6Go 기반, 각 VU가 경량 goroutine으로 실행적은 리소스로 높은 VU 수 생성 가능, 스크립트를 JS로 작성, Grafana/Prometheus 연동 용이복잡한 브라우저 렌더링 시나리오는 별도 확장(k6 browser) 필요
LocustPython 기반, 이벤트 기반 또는 gevent 코루틴Python으로 복잡한 사용자 행동 로직을 자유롭게 작성 가능, 분산 실행 지원Python GIL 특성상 부하 생성기 자체가 CPU bound가 되면 부하 생성기가 병목이 될 수 있음 — 분산 워커로 확장 필요
Apache JMeterJVM 기반, 스레드당 VU(기본 모델)GUI로 시나리오 작성, 폭넓은 프로토콜 지원, 기업 환경에서 표준적으로 사용스레드 기반이라 VU 수가 커지면 JMeter 자체의 메모리/CPU 사용량이 급증 — 대규모 테스트는 분산 실행 필요
wrk / heyC 기반, 이벤트 기반 non-blocking매우 높은 처리량의 순수 HTTP 벤치마크에 적합, 가벼움복잡한 다단계 시나리오(로그인 → 여러 페이지 이동) 작성에는 부적합, 단일 엔드포인트 벤치마크용
실무 감각: 여기서 가장 중요한 개념은 "부하 생성기 자체가 병목이 되지 않아야 한다"는 것입니다. 부하 생성기를 한 대의 서버에서만 돌리면, 목표 부하에 도달하기 전에 부하 생성기의 CPU나 네트워크 대역폭이 먼저 한계에 도달할 수 있습니다. 이 경우 테스트 대상 서버가 아니라 부하 생성기가 병목이 되어 "우리 서버는 5,000 RPS 이상은 못 버틴다"는 잘못된 결론을 내릴 수 있습니다. 대규모 테스트에서는 여러 대의 부하 생성기(분산 실행)로 나눠 이 문제를 피해야 합니다.

7. 실무 연결

상황 1: 테스트에서는 5,000 RPS를 버텼는데 운영에서 1,000 RPS에 무너졌다

원인은 두 가지였습니다. 첫째, 테스트 데이터가 운영 DB보다 훨씬 적어 인덱스가 메모리에 전부 올라가 있었던 반면, 운영 DB는 데이터가 많아 디스크 I/O가 추가로 발생했습니다. 둘째, 테스트에서는 모든 VU가 동일한 인기 상품 몇 개만 조회해 캐시 히트율이 비정상적으로 높게 나왔지만, 실제 트래픽은 훨씬 다양한 상품을 조회해 캐시 미스가 잦았습니다. 테스트 데이터 볼륨을 운영 수준으로 맞추고, 조회 대상 분포를 실제 접근 로그 기반으로 재설계한 뒤에야 신뢰할 수 있는 결과가 나왔습니다.

상황 2: Stress Test로 breaking point를 찾았더니 DB 커넥션 풀이 원인이었다

부하를 계속 올리자 앱 서버 CPU는 40% 수준에서 안정적인데 응답 시간만 계속 나빠졌습니다. 커넥션 풀 대기 시간 지표를 확인하니, 풀 크기가 DB가 실제로 감당 가능한 동시 커넥션 수보다 훨씬 작게 설정되어 있었고, 요청들이 풀에서 커넥션을 얻기 위해 대기하고 있었습니다. 앱 서버 자체는 여유로운데 전체 응답 시간이 나빠지는 전형적인 신호였습니다. 풀 크기를 DB 용량에 맞춰 조정한 뒤 처리량이 개선됐습니다.

상황 3: Soak Test에서만 발견된 메모리 누수

일반 Load Test(짧은 시간 실행)에서는 문제가 없었지만, 8시간 동안 중간 수준 부하를 유지하는 Soak Test에서 메모리 사용량이 선형으로 계속 증가하는 것이 발견됐습니다. 원인은 캐시 라이브러리의 특정 캐시 항목에 만료 시간이 누락되어 무한정 쌓이고 있었던 것이었습니다. 짧은 부하 테스트만 했다면 배포 후 며칠이 지나서야 운영 중에 발견됐을 문제였습니다.

상황 4: 배포 전 Spike Test로 Auto Scaling 반응 지연을 발견

마케팅 이벤트를 앞두고 급격한 트래픽 증가를 시뮬레이션하는 Spike Test를 돌렸더니, Auto Scaling이 새 인스턴스를 띄우는 데 몇 분이 걸리는 동안 기존 인스턴스들이 과부하로 에러를 반환하는 것을 확인했습니다. 이 결과를 바탕으로 이벤트 시작 전 수동으로 최소 인스턴스 수를 미리 늘려두는 운영 절차를 추가했습니다.

8. 답변 사고 순서

  • 1
    목표(SLO)를 먼저 정의한다

    "얼마나 빨라야 하는가", "어느 수준의 트래픽까지 버텨야 하는가"를 먼저 숫자로 정의합니다. 목표 없이 부하만 올리는 테스트는 결과를 판단할 기준이 없습니다.

  • 2
    실제 트래픽 패턴을 분석해 시나리오에 반영한다

    접근 로그 기반 엔드포인트 비율, think time, 데이터 분포(hot key), 세션 유지 방식을 시나리오에 녹여 균일한 부하가 아닌 현실적인 부하를 설계합니다.

  • 3
    테스트 환경과 운영 환경의 격차를 최소화한다

    데이터 볼륨, 인스턴스 스펙, 캐시 웜업 상태, 네트워크 조건이 운영과 다르면 결과가 왜곡된다는 것을 인지하고 가능한 한 맞춥니다.

  • 4
    목적에 맞는 부하 패턴을 선택한다

    SLO 충족 확인이면 Load Test, 한계선 파악이면 Stress Test, 이벤트 대비면 Spike Test, 장시간 안정성이면 Soak Test를 선택합니다.

  • 5
    결과에서 병목을 지표로 좁혀간다

    "느려졌다"에서 멈추지 않고, 부하 증가 시점과 앱 서버/DB/네트워크 지표를 시간축으로 겹쳐 어느 자원이 먼저 포화되는지 증명합니다.

9. 핵심 답변 예시

면접관: "새로운 기능을 배포하기 전에 부하 테스트를 어떻게 설계하시겠어요? 실제 트래픽 패턴과 다르게 테스트하면 어떤 문제가 생기나요?"

STRONG ANSWER

먼저 목표를 숫자로 정의합니다. 이 기능이 목표하는 트래픽 수준(예: 초당 2,000 요청)에서 p95 응답 시간과 에러율의 허용 기준을 먼저 정하고, 그 기준을 만족하는지 확인하는 것이 테스트의 목적입니다.


그다음 실제 트래픽 패턴을 최대한 재현하려고 합니다. 모든 가상 사용자가 동일한 간격으로 동일한 요청을 반복하면 실제보다 훨씬 낙관적인 결과가 나옵니다. 실제 접근 로그를 분석해서 엔드포인트별 호출 비율을 반영하고, 실제 사용자가 화면을 보고 다음 행동을 하기까지의 think time을 시나리오에 넣고, 인기 상품 몇 개에 조회가 몰리는 것 같은 데이터 편향도 재현합니다. 이걸 생략하면 캐시가 항상 웜업된 상태로 테스트되어 실제 운영에서 캐시 미스가 몰리는 순간의 부하가 전혀 재현되지 않습니다.


테스트 환경도 운영과 최대한 맞춥니다. 특히 DB 데이터 볼륨이 적으면 인덱스 효과가 실제와 달라지고, 캐시를 테스트 직전에 미리 채워두면 콜드 스타트 상황이 반영되지 않습니다. 목적에 따라 패턴도 다르게 씁니다. 배포 전 SLO 충족 확인은 Load Test로, 시스템의 한계선을 알아야 할 때는 Stress Test로 트래픽을 계속 올리면서 breaking point를 찾고, 장시간 운영 시 메모리 누수 같은 문제는 Soak Test로만 드러나기 때문에 별도로 돌립니다.


마지막으로 결과가 나빠지는 지점에서는 앱 서버 CPU, DB 커넥션 풀 대기 시간, 네트워크 처리량을 동시에 시간축으로 겹쳐서 어느 자원이 가장 먼저 포화되는지 확인합니다. "느려졌다"로 끝내지 않고 병목의 위치를 지표로 증명하는 것이 실제로 조치할 수 있는 결론을 만듭니다.

이 답변이 좋은 이유:

  • SLO를 먼저 정의해 테스트에 목적성을 부여함
  • 현실적인 트래픽 재현(think time, 데이터 편향, 엔드포인트 비율)의 구체적 요소를 설명
  • 테스트 환경과 운영 환경의 격차가 결과를 왜곡하는 구체적 메커니즘까지 설명
  • 목적별 부하 패턴(Load/Stress/Soak/Spike)을 구분해 적용
  • 결과 해석을 지표 상관관계 기반 병목 판별로 마무리

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"JMeter나 k6로 동시 사용자 1,000명을 설정해서 몇 분 동안 돌려보고, 에러 없이 통과하면 배포합니다."

이 답변이 약한 이유

  • 목표(SLO)가 정의되지 않아 "통과"의 기준이 불명확함
  • 동시 사용자 수만 언급하고 실제 트래픽 패턴(엔드포인트 비율, think time, 데이터 분포)에 대한 고려가 없음
  • 테스트 환경과 운영 환경의 차이를 전혀 언급하지 않음
  • 결과가 나빠졌을 때 병목을 어떻게 찾을지에 대한 절차가 없음
WEAK ANSWER — 또 다른 패턴

"부하를 최대한 많이 줘서 서버가 죽는지 안 죽는지 확인합니다."

이 답변이 약한 이유

  • 목적(SLO 충족 확인인지, 한계선 파악인지, 장시간 안정성 확인인지)을 구분하지 않고 뭉뚱그림
  • "죽는지 안 죽는지"라는 이진적 기준만 있고, 어느 자원이 먼저 포화되는지 판별하는 절차가 없음
  • 부하 생성기 자체가 병목이 될 수 있다는 것을 고려하지 않음
STRONG ANSWER 요소 체크리스트
  • 부하 테스트 전에 SLO(목표 latency/error rate)를 숫자로 정의한다
  • 실제 접근 로그 기반 트래픽 패턴(엔드포인트 비율, think time, 데이터 편향)을 시나리오에 반영한다
  • 테스트 환경과 운영 환경의 격차(데이터 볼륨, 캐시 웜업, 네트워크)를 인지하고 최소화한다
  • 목적에 맞는 부하 패턴(Load/Stress/Spike/Soak)을 구분해 선택한다
  • 부하 생성기 자체가 병목이 되지 않도록 분산 실행을 고려한다
  • 결과 해석 시 앱 서버/DB/네트워크 지표를 겹쳐 병목을 지표로 증명한다

11. Senior-level Follow-up Q&A

Q: 부하 테스트에서는 초당 5,000 요청까지 문제없이 처리했는데, 실제 운영에서는 초당 1,000 요청 수준에서 이미 에러가 발생했습니다. 무엇이 잘못됐을까요?

모범 답변: 테스트 환경과 운영 환경의 격차를 먼저 의심합니다. 가장 흔한 원인은 테스트 DB의 데이터 볼륨이 운영보다 훨씬 적어 인덱스 효과나 쿼리 플랜이 실제와 다르게 나온 경우, 그리고 테스트 트래픽이 균일해서 캐시가 항상 웜업된 상태로 테스트된 경우입니다. 실제 트래픽은 데이터 조회가 특정 항목에 쏠리는 구간에서 캐시 미스가 몰려 DB 부하가 급증하는데, 균일한 테스트 트래픽은 이 상황을 재현하지 못합니다. 테스트 데이터 볼륨을 운영 수준으로 맞추고, 실제 접근 로그 기반의 데이터 분포로 시나리오를 재설계해 재현을 시도합니다.

Q: 부하 테스트 결과 응답 시간이 급격히 나빠지는 시점을 확인했습니다. 그 시점에서 병목이 애플리케이션 서버인지 DB인지 어떻게 구분하시겠어요?

모범 답변: 그 시점의 앱 서버 CPU/메모리 사용률, DB 커넥션 풀 대기 시간, 쿼리 latency, 네트워크 처리량을 동시에 시간축으로 겹쳐 봅니다. 앱 서버 CPU가 포화 상태이고 DB는 여유롭다면 애플리케이션 로직(직렬화, 비효율적 알고리즘)이 병목이고, 앱 서버는 여유로운데 DB 커넥션 풀 대기 시간이 늘어난다면 커넥션 풀 크기나 느린 쿼리가 병목입니다. USE Method(Utilization, Saturation, Errors)를 각 자원에 적용해 어느 자원이 가장 먼저 한계에 도달하는지로 구분합니다.

Q: 부하 테스트 도구로 목표 RPS를 만들어냈는데도 서버 응답이 느립니다. 부하 생성기 자체가 문제일 가능성은 어떻게 배제하시겠어요?

모범 답변: 부하 생성기를 실행하는 머신의 CPU, 메모리, 네트워크 대역폭 사용률을 함께 확인합니다. 만약 부하 생성기 자체가 CPU나 네트워크 한계에 도달해 있다면, 목표 RPS를 실제로 만들어내지 못하고 있을 수 있습니다. 이 경우 부하 생성기를 여러 대로 분산 실행해서 생성기 자체가 병목이 아님을 확인한 뒤에야 테스트 대상 서버의 결과를 신뢰합니다.

Q: Load Test는 통과했는데 배포 며칠 뒤 메모리 사용량이 계속 증가하다 서버가 재시작되는 문제가 발생했습니다. 왜 사전에 발견하지 못했을까요?

모범 답변: 일반적인 Load Test는 몇 분에서 길어야 한두 시간 정도만 실행하기 때문에, 느리게 누적되는 메모리 누수나 커넥션 누수는 그 시간 안에 드러나지 않습니다. 이런 문제는 중간 수준의 부하를 수 시간에서 하루 이상 유지하는 Soak Test에서만 재현됩니다. 앞으로는 배포 전 체크리스트에 Soak Test를 포함해, 짧은 시간에 통과했다고 해서 장시간 안정성까지 보장된 것은 아니라는 점을 팀에 공유하겠습니다.

Q: 부하를 계속 올렸는데 어느 순간부터 처리량(throughput)이 더 이상 늘지 않고 평평해집니다. 이 현상을 어떻게 설명하시겠어요?

모범 답변: 이는 시스템의 특정 자원이 포화 상태에 도달해 그 이상의 동시 처리를 받아들이지 못하는 것입니다(처리량 정체, plateau). Little's Law(L = λ × W) 관점에서 보면, 처리 시간(W)이 고정된 자원(스레드 풀, DB 커넥션 풀, 이벤트 루프)에 막혀 있다면 도착률(λ)을 아무리 늘려도 시스템 내부에 머무는 요청 수(L)만 늘어나고 실제 처리량은 늘지 않습니다. 이 지점부터는 요청이 대기열에 쌓이며 응답 시간만 계속 나빠지므로, 어느 자원이 고정 한계인지 찾아 확장하거나(커넥션 풀 크기, 인스턴스 수) 그 자원 앞에서 명시적으로 요청을 거절하는 backpressure를 적용해야 합니다.

12. 흔한 오해

오해 1

"동시 사용자 수(VU)가 많을수록 더 강한 부하 테스트다"

VU 수 자체보다 각 VU가 실제 사용자 행동(think time, 엔드포인트 비율, 데이터 분포)을 얼마나 정직하게 반영하는지가 결과의 신뢰도를 결정합니다. VU 1만 명이라도 균일하고 비현실적인 트래픽이면 결과는 무의미합니다.

오해 2

"한 번의 부하 테스트로 모든 것을 확인할 수 있다"

Load, Stress, Spike, Soak은 각각 다른 질문에 답합니다. SLO 충족(Load)과 한계선 파악(Stress)은 다른 테스트이고, 메모리 누수 같은 시간 누적 문제는 Soak Test에서만 드러납니다. 목적에 맞는 패턴을 별도로 설계해야 합니다.

오해 3

"테스트 환경이 운영보다 작아도 비율만 맞으면 결과를 그대로 신뢰할 수 있다"

데이터 볼륨이 작으면 인덱스가 메모리에 전부 올라가 실제보다 빠르게 나오는 등, 규모 차이는 단순 비율로 환산되지 않는 비선형적 효과를 만듭니다. 특히 DB 성능과 캐시 히트율은 규모에 따라 質적으로 달라질 수 있습니다.

오해 4

"부하 테스트에서 에러가 없으면 성능에 문제가 없다는 뜻이다"

에러율이 0%여도 응답 시간(특히 p95/p99)이 SLO를 넘었다면 사용자 경험은 이미 나빠진 것입니다. 에러 발생 여부만으로 판단하지 않고 latency 분포를 함께 확인해야 합니다.

오해 5

"부하 테스트 결과가 나쁘면 무조건 서버를 늘리면 해결된다"

병목이 DB 커넥션 풀이나 외부 API rate limit처럼 인스턴스를 늘려도 해결되지 않는 자원이라면, 앱 서버를 늘려봐야 그 다운스트림 자원에 더 많은 부하가 몰릴 뿐입니다. 병목의 실제 위치를 지표로 확인한 뒤 그에 맞는 해법(커넥션 풀 확장, 쿼리 튜닝, 캐시 도입)을 선택해야 합니다.
