핵심 질문

2. 전제 개념 — 쉬운 한국어로

Service는 "떠 있는 서버"가 아니라 "규칙의 묶음"이다

Pod IP는 Pod가 재시작되거나 재스케줄되면 바뀐다. 애플리케이션이 매번 바뀌는 Pod IP를 직접 알아야 한다면 운영이 불가능하다. Service는 이 문제를 "라벨 셀렉터에 맞는 Pod 그룹을 가리키는 안정적인 가상 IP(ClusterIP)와 DNS 이름"으로 추상화한 것이다.

중요한 건, Service 자체는 데이터 플레인에 실체가 있는 프로세스나 서버가 아니라는 점이다. Service는 API 서버에 저장된 하나의 오브젝트(의도)일 뿐이고, 실제로 트래픽을 그 가상 IP에서 진짜 Pod로 옮기는 일은 각 노드에서 도는 kube-proxy(또는 그 대체제)가 커널 네트워킹 규칙(iptables/IPVS/nftables 등)을 만들어서 수행한다. "Service가 로드밸런싱을 한다"는 말은 정확히는 "Service라는 선언을 보고 kube-proxy가 각 노드에 규칙을 심어 놓아서, 그 규칙이 로드밸런싱을 한다"는 뜻이다.

EndpointSlice — Service 뒤에 실제로 붙어 있는 Pod IP 목록

Service의 라벨 셀렉터에 맞는 Pod들의 실제 IP:포트 목록은 EndpointSlice 오브젝트에 저장된다. 과거에는 이 정보가 Service당 하나의 Endpoints 오브젝트에 전부 들어갔는데, Pod가 수천 개인 대형 서비스에서는 이 하나의 오브젝트가 너무 커져서 API 서버·컨트롤러의 부하가 심해지는 문제가 있었다. EndpointSlice는 이걸 여러 개의 작은 조각(기본 100개 엔드포인트 단위)으로 나눠, Pod 하나가 추가/제거될 때 전체 목록을 통째로 다시 쓰지 않고 해당 조각만 갱신하도록 만들었다.

레거시 Endpoints API는 v1.33부터 공식적으로 폐기(deprecated) 경로에 들어갔고, EndpointSlice가 서비스 디스커버리의 사실상 표준(source of truth)이다. 다만 하위 호환을 위해 셀렉터가 있는 Service는 지금도 여전히 대응하는 Endpoints 오브젝트를 함께 만든다 — 이 오브젝트를 직접 사용하는 옛날 컨트롤러/도구가 아직 남아 있기 때문이다. kube-proxy와 CoreDNS의 kubernetes 플러그인은 이미 EndpointSlice를 watch한다. https://kubernetes.io/docs/concepts/services-networking/endpoint-slices/

여기서 나오는 운영상 중요한 성질이 전파 지연(propagation delay)이다. Pod가 새로 뜨거나 사라지면 ① kubelet이 상태를 알리고 → ② EndpointSlice 컨트롤러가 EndpointSlice를 갱신하고 → ③ kube-proxy/CoreDNS가 그 변경을 watch로 받아 각자의 규칙/캐시를 갱신하는 단계를 거친다. 이 체인 전체가 즉시 일어나지 않고, 클러스터 규모가 크면 수 초 단위로 걸릴 수 있다 — "Pod가 Ready인데 왜 아직 트래픽을 못 받는가", "Pod를 지웠는데 왜 잠깐 그쪽으로 요청이 계속 가는가"의 근본 원인이 대개 이 지연이다.

Service 타입 — ClusterIP, NodePort, LoadBalancer, Headless

ClusterIP (기본값)
클러스터 내부에서만 유효한 가상 IP. 서비스 간 내부 통신의 기본형. 외부에서 직접 접근 불가.
NodePort
모든(또는 대상) 노드의 동일한 포트(기본 30000~32767)를 열어 외부에서 노드IP:포트로 접근. ClusterIP를 내부적으로 포함.
LoadBalancer
클라우드 프로바이더의 외부 로드밸런서를 프로비저닝(외부 CCM 필요). NodePort+ClusterIP를 내부적으로 포함하는 상위 계층.
Headless (ClusterIP: None)
가상 IP를 만들지 않고, DNS 조회 시 각 Pod의 IP를 직접 반환. StatefulSet의 안정적 개별 접근이나 클라이언트 사이드 로드밸런싱(gRPC 등)에 사용.

네 타입은 서로 대체재가 아니라 포함 관계다: LoadBalancer는 NodePort를, NodePort는 ClusterIP를 내부적으로 포함한다. "외부 공개가 필요 없다"면 ClusterIP, "클라우드 LB 없이 노드 레벨에서 외부 접근이 필요하다"면 NodePort(주로 온프레미스나 디버깅용), "표준적인 외부 서비스 공개"라면 LoadBalancer, "VIP 로드밸런싱이 아니라 개별 Pod 정체성 자체가 필요하다"면 Headless다.

kube-proxy 모드 — iptables, IPVS, nftables, 그리고 kube-proxy 없는 클러스터

Service VIP로 온 패킷을 실제 Pod로 바꿔주는(DNAT) 작업을 커널 레벨에서 어떻게 구현하느냐가 kube-proxy의 모드다.

  • iptables(기본값): 규칙을 순차 매칭하는 방식이라 Service/엔드포인트 수가 많아지면 규칙 체인이 선형으로 길어져 패킷 처리 지연과 규칙 갱신(sync) 비용이 함께 증가한다. 대규모 클러스터에서 흔한 성능 병목 지점.
  • IPVS: 커널의 해시 테이블 기반 로드밸런서를 사용해 대규모에서 iptables보다 유리하지만, 운영 복잡도(추가 커널 모듈, 디버깅 도구 차이)가 있다.
  • nftables: iptables의 후속 넷필터 프레임워크를 사용하는 모드로 v1.29 alpha → v1.31 beta → v1.33 GA로 안정화됐다. 대규모 클러스터·최신 커널(5.13+) 환경에 권장되지만, 2026-07 시점에도 여전히 기본값은 iptables다 — 아직 상대적으로 신생 모드라 일부 CNI 플러그인과의 호환성을 확인해야 하고, 실사용에서 대규모 스케일에서의 성능 이슈가 새로 보고되는 사례도 있어 "무조건 nftables가 정답"은 아니다. https://kubernetes.io/blog/2025/02/28/nftables-kube-proxy/

한 걸음 더 나아가면, kube-proxy 자체를 쓰지 않는 클러스터도 있다. Cilium 같은 eBPF 기반 CNI는 kube-proxy가 하던 Service 로드밸런싱을 커널의 eBPF 훅으로 직접 구현해 iptables/nftables 규칙 순회 자체를 없앤다("kube-proxy replacement" 또는 "kube-proxy-free" 모드). Service라는 API 오브젝트와 그 의미(안정적 VIP, 엔드포인트 그룹)는 그대로지만, 그걸 실제로 구현하는 데이터 플레인 구현체가 kube-proxy에서 CNI의 eBPF로 바뀐 것뿐이다. "kube-proxy가 없으면 Service가 안 된다"는 생각은 틀렸다 — Service 개념과 kube-proxy라는 구현체는 분리해서 이해해야 한다.

externalTrafficPolicy / internalTrafficPolicy — client IP 보존과 트래픽 분산의 trade-off

NodePort/LoadBalancer Service로 들어온 외부 트래픽은 기본(externalTrafficPolicy: Cluster)적으로 어느 노드로 들어오든 클러스터의 모든 Pod에 균등하게 분산될 수 있다. 하지만 이 과정에서 커널이 SNAT(Source NAT)를 하기 때문에, 실제 백엔드 Pod 입장에서는 원래 클라이언트의 IP가 아니라 트래픽을 중계한 노드의 IP가 보인다. 클라이언트 IP 기반 접근 제어나 로깅이 필요하면 이건 치명적이다.

externalTrafficPolicy: Local로 바꾸면 SNAT 없이 client IP가 그대로 보존되지만, 대신 그 노드에 해당 Service의 Pod가 로컬로 떠 있지 않으면 트래픽이 다른 노드로 넘어가지 않고 그냥 드롭된다. 그래서 Local을 쓰려면 로드밸런서/DNS 레벨에서 "Pod가 실제로 떠 있는 노드에만" 트래픽을 보내도록 헬스체크가 맞물려야 하고, Pod 분포가 노드마다 고르지 않으면 특정 노드에 트래픽이 쏠리는 불균형이 생길 수 있다. internalTrafficPolicy: Local도 같은 원리를 클러스터 내부 트래픽에 적용한다 — 같은 노드의 로컬 엔드포인트만 쓰게 강제해 노드 간 홉을 줄이지만, 로컬 엔드포인트가 없으면 마찬가지로 실패한다.

trafficDistribution — "완전히 로컬만" 대신 "가능하면 가까운 곳" (GA, v1.35)

Local 정책의 "로컬 엔드포인트 없으면 드롭"이라는 극단성을 완화하기 위해, v1.35에서 spec.trafficDistribution이 안정화됐다. PreferSameZone(과거 이름 PreferClose, 하위 호환으로 계속 동작)은 같은 가용 영역(zone)의 엔드포인트를 우선 사용하되, 없으면 다른 zone으로 안전하게 폴백한다. PreferSameNode는 같은 노드를 우선한다. 이는 zone 간 데이터 전송 비용(cross-AZ 트래픽 비용)을 줄이고 지연을 낮추면서도, Local 정책처럼 트래픽을 완전히 끊어버리지 않는 "안전한 선호"라는 점에서 비용/가용성 trade-off를 더 유연하게 만든다. CoreDNS — 클러스터 내부 이름 해석과 그 실패 패턴

Pod가 my-svc.my-namespace.svc.cluster.local 같은 이름으로 다른 서비스를 찾을 때 이를 실제 ClusterIP(또는 Headless의 경우 Pod IP 목록)로 바꿔주는 것이 CoreDNS다. CoreDNS도 결국 Deployment로 떠 있는 일반 Pod이고, 그 안의 kubernetes 플러그인이 Service/EndpointSlice를 watch해서 DNS 레코드를 만든다. 즉 CoreDNS 자체도 위에서 설명한 "EndpointSlice 전파 지연"의 영향을 그대로 받는다.

실무에서 자주 마주치는 CoreDNS 관련 문제 세 가지가 있다.

  • ndots:5 증폭 문제: 대부분의 Pod의 /etc/resolv.conf에는 ndots:5가 기본 설정돼 있다. 점(.)이 5개 미만인 이름(예: api.stripe.com, 점 2개)을 조회할 때, 리졸버는 이걸 바로 절대 이름으로 취급하지 않고 클러스터 내부 search domain들을 먼저 하나씩 붙여가며 조회한다. 그 결과 외부 API 하나를 부를 때마다 실패하는 내부 도메인 쿼리가 여러 번 CoreDNS를 거친 다음에야 진짜 외부 조회가 나간다 — 지연 증가와 CoreDNS 부하 증가를 동시에 일으킨다.
  • CoreDNS Pod 재시작 시 순간 실패: 롤링 업데이트나 노드 드레인으로 CoreDNS Pod가 내려갈 때, 그 Pod를 가리키던 kube-proxy 규칙이 아직 갱신되기 전에 이미 종료돼 버리면 그 규칙으로 들어온 요청이 실패한다. CoreDNS는 이를 완화하기 위해 lameduck 구간(종료 신호를 받아도 일정 시간 동안 진행 중인 요청은 계속 처리)을 지원한다 — 이건 애플리케이션의 graceful shutdown/preStop 패턴을 인프라 컴포넌트 자신에게도 적용한 것이다.
  • 대규모 클러스터에서의 CoreDNS 과부하: 노드/Pod 수가 늘면 DNS 쿼리량도 같이 늘어난다. CoreDNS는 보통 오토스케일러(노드 수와 코어 수 기준 수식으로 replica 조정)로 스케일하지만, 그래도 매 Pod가 CoreDNS를 직접 거치는 구조 자체가 대규모에서는 한계에 부딪힌다. 이를 노드 단위로 분산시키는 것이 NodeLocal DNSCache다 — 각 노드에 로컬 DNS 캐시 데몬(DaemonSet)을 두어, 캐시 히트는 CoreDNS까지 가지 않고 노드 안에서 즉시 응답하게 만든다.
2026-07 기준 확인 사항 — Service/kube-proxy/CoreDNS 관련 최신 사실

v1.36("Haru") 기준으로도 kube-proxy의 기본 모드는 여전히 iptables다. nftables는 v1.33에 GA됐지만 "권장 방향"이지 기본값 전환은 아직 없다. 레거시 Endpoints API는 v1.33부터 공식 폐기(deprecated) 경로에 들어갔으며, EndpointSlice가 서비스 디스커버리의 사실상 표준이다(하위 호환을 위한 Endpoints 오브젝트는 계속 병행 생성). trafficDistribution의 PreferSameNode/PreferSameZone(과거 이름 PreferClose)은 v1.35에 stable이 됐다. CoreDNS는 v1.11 계열이 v1.36과 함께 검증돼 있다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 추상화 뒤의 실체 이해: Service가 "떠 있는 로드밸런서"가 아니라 kube-proxy(또는 eBPF)가 만드는 규칙이라는 걸 알고, 문제가 났을 때 "규칙이 왜 이렇게 만들어졌는가"로 파고들 수 있는가.
  • trade-off 판단: client IP 보존(Local) vs 균등 분산(Cluster), zone 우선(trafficDistribution) vs 완전 로컬 강제라는 상충 관계를 실제 장애 상황에서 골라낼 수 있는가.
  • 전파 지연/eventual consistency 감각: Kubernetes의 선언적 모델이 "즉시 반영"이 아니라 "watch 기반으로 결국 반영되는" 구조라는 것을, Service/EndpointSlice/CoreDNS 체인에서 구체적으로 설명할 수 있는가.
  • DNS를 인프라로 다루는 감각: CoreDNS를 "그냥 잘 도는 부속품"이 아니라, ndots·lameduck·NodeLocal DNSCache 같은 구체적 운영 지식으로 다룰 수 있는가.

Service/CoreDNS는 겉보기에 "그냥 이름으로 접근하게 해주는 것"처럼 단순해 보이지만, 실제로는 클러스터의 모든 서비스 간 통신이 거쳐 가는 공통 경로다. 이 계층의 동작을 얕게 아는 사람은 "Pod는 문제없어 보이는데 왜 연결이 안 되지"에서 멈추지만, 시니어는 규칙 전파 지연, 트래픽 정책, DNS 캐싱까지 이어서 원인을 좁혀 나갈 수 있어야 한다.

4. 실무 연결

시나리오 1 — externalTrafficPolicy: Local 전환 후 특정 노드에서 연결 실패

client IP 기반 rate limiting을 붙이기 위해 externalTrafficPolicy를 Cluster에서 Local로 바꿨다. 그 순간부터 해당 서비스 Pod가 배치되지 않은 노드로 외부 로드밸런서가 트래픽을 보낼 때마다 연결이 그냥 드롭됐다. 원인은 로드밸런서의 헬스체크가 Local 정책의 의미(그 노드에 로컬 엔드포인트가 없으면 트래픽을 받으면 안 된다)를 반영하지 못하고 단순 TCP 헬스체크만 하고 있었기 때문이었다. 해결은 로드밸런서의 헬스체크를 kube-proxy가 노출하는 서비스별 헬스체크 포트(healthCheckNodePort)로 바꿔, 로컬 엔드포인트가 없는 노드는 헬스체크 자체에서 걸러지도록 한 것이었다.

여기서 얻은 교훈은, client IP 보존 같은 기능 하나를 켜는 것이 로드밸런서 설정까지 함께 바꿔야 하는 연쇄 변경이라는 점이다. Kubernetes Service 스펙만 보고 판단하면 안 되고, 그 앞단의 외부 로드밸런서/CCM이 이 정책을 어떻게 인지하는지까지 함께 검토해야 한다.

시나리오 2 — 스케일 아웃 직후 몇 초간 새 Pod로 트래픽이 가지 않음

트래픽 급증에 대응해 HPA가 Pod를 순간적으로 3배 늘렸는데, 새로 뜬 Pod들이 Ready가 된 이후에도 몇 초간 실제로는 요청을 거의 받지 못했다. 원인을 추적해 보니 대규모 클러스터에서 EndpointSlice 컨트롤러가 새 Pod들을 반영하고, 그 변경이 모든 노드의 kube-proxy로 전파돼 iptables 규칙이 갱신되기까지의 시간이 누적돼 있었다. 이 지연 자체는 없앨 수 없지만, HPA의 스케일업을 더 이르게 트리거하도록 지표 기준을 앞당기고, 급격한 단일 스케일업 대신 단계적 스케일업으로 바꿔 "새 Pod가 트래픽을 받기 시작하는 시점"과 "실제 부하가 몰리는 시점" 사이의 간격을 벌려 완화했다.

이 사례가 보여주는 것은, Kubernetes의 선언적 리소스 상태(Ready)와 데이터 플레인의 실제 반영 상태 사이에는 항상 eventual consistency 간격이 있다는 것이다. "Pod가 Ready니까 바로 트래픽을 받을 것"이라는 가정으로 오토스케일링이나 배포 타이밍을 설계하면 이런 간극에서 실패가 생긴다.

시나리오 3 — 외부 API 호출 지연의 원인이 CoreDNS ndots였던 사례

결제 API(외부 도메인) 호출의 p99 지연이 이상하게 높다는 신고가 있었다. 애플리케이션 코드, 네트워크 대역폭 모두 문제가 없었는데, 패킷 캡처를 떠 보니 도메인 하나를 조회할 때마다 내부 search domain이 붙은 실패 쿼리 여러 개가 먼저 나가고 있었다(ndots:5 기본값 때문). 트래픽량이 많은 서비스일수록 이 증폭된 쿼리량이 CoreDNS 자체의 부하까지 늘려 다른 서비스의 내부 DNS 조회 지연에도 영향을 주고 있었다.

완화책으로는 해당 워크로드의 Pod spec에 dnsConfig로 ndots를 낮추거나 완전한 도메인 이름(끝에 .을 붙인 FQDN)을 코드에서 사용하도록 바꿨고, 클러스터 전체 차원에서는 NodeLocal DNSCache를 도입해 반복 조회가 노드 안에서 캐시되도록 해 CoreDNS로 가는 쿼리량 자체를 줄였다.

5. 답변 사고 순서

"Service/DNS 관련 트래픽 문제"를 받으면 아래 순서로 답하라. 문제 범위 구분(라우팅 vs 이름 해석) → Service/Endpoint 상태 확인 → 트래픽 정책 확인 → kube-proxy 규칙 반영 확인 → DNS 계층 확인 순서다.

  1. "연결 자체가 안 되는 것"인지 "이름 해석이 느리거나 실패하는 것"인지부터 나눈다

    ClusterIP로 직접 연결은 되는데 서비스 이름으로는 안 된다면 DNS(CoreDNS) 쪽 문제로 좁히고, ClusterIP로도 안 된다면 Service/Endpoint/kube-proxy 쪽으로 좁힌다.

  2. Service의 EndpointSlice에 실제로 대상 Pod가 들어있는지 확인한다

    kubectl get endpointslices로 라벨 셀렉터가 의도한 Pod와 실제로 매칭되는지, Pod의 readiness 상태가 반영돼 있는지 확인한다. 여기가 비어 있거나 잘못됐다면 애초에 kube-proxy/CoreDNS까지 갈 필요가 없다.

  3. 트래픽 정책(externalTrafficPolicy/internalTrafficPolicy/trafficDistribution)이 상황에 맞게 설정됐는지 확인한다

    특정 노드/zone에서만 실패한다면 Local 정책과 실제 Pod 분포의 불일치를 의심한다. 로드밸런서 헬스체크가 이 정책을 인지하고 있는지도 함께 본다.

  4. 변경 시점과 증상 발생 시점의 간격을 본다

    배포/스케일 이벤트 직후 짧게(수 초) 나타났다 사라지는 증상이라면 EndpointSlice/kube-proxy 규칙 전파 지연을 우선 의심하고, 지속적으로 재현된다면 설정 자체의 오류로 좁힌다.

  5. DNS 계층까지 내려가 ndots/캐싱/CoreDNS 가용성을 확인한다

    지연이 외부 도메인 조회에 집중된다면 ndots 증폭을 의심하고, CoreDNS 자체의 재시작/스케일 이벤트와 증상 시점이 겹치는지 확인한다.

6. 핵심 답변 예시

면접 답변 예시 — "externalTrafficPolicy: Local 전환 후 일부 노드 연결 실패, 스케일 아웃 직후 트래픽 누락"

"이 문제는 두 가지 서로 다른 원인이 겹쳐 있다고 보고 나눠서 접근하겠습니다. 첫 번째, 노드별 연결 실패는 externalTrafficPolicy: Local의 본질적인 trade-off입니다. Local은 SNAT을 생략해 client IP를 보존하는 대신, 해당 노드에 로컬 엔드포인트가 없으면 트래픽을 다른 노드로 넘기지 않고 그냥 드롭합니다. 이건 버그가 아니라 설계된 동작이고, 진짜 문제는 외부 로드밸런서의 헬스체크가 이 정책을 모르고 있다는 겁니다. kube-proxy가 노출하는 healthCheckNodePort를 로드밸런서 헬스체크에 연결해, 로컬 엔드포인트가 없는 노드는 애초에 트래픽 대상에서 빠지도록 해야 합니다.

두 번째, 스케일 아웃 직후 몇 초간 트래픽이 안 가는 현상은 EndpointSlice와 kube-proxy 규칙 갱신의 전파 지연으로 봅니다. Pod가 Ready 상태가 됐다고 해서 모든 노드의 kube-proxy 규칙이 즉시 갱신되는 건 아니고, watch 기반으로 결국 반영되는 구조라 클러스터 규모에 따라 수 초의 간격이 생길 수 있습니다. 이건 완전히 없앨 수는 없으니, 스케일업 트리거를 더 앞당기거나 단계적으로 늘려서 '새 Pod가 실제로 트래픽을 받기 시작하는 시점'과 '부하가 몰리는 시점' 사이의 여유를 만드는 방향으로 완화하겠습니다.

두 문제 모두 근본적으로는 '내가 설정을 바꾼 부분'과 '그 설정을 실제로 구현하는 데이터 플레인(kube-proxy/로드밸런서)의 반영 방식' 사이의 괴리에서 나온다는 공통점이 있습니다."

이 답변이 좋은 이유: ① 두 증상을 뭉뚱그리지 않고 서로 다른 원인으로 명확히 분리 ② Local 정책을 버그가 아니라 의도된 trade-off로 정확히 설명 ③ 로드밸런서 헬스체크라는, Kubernetes 밖의 연쇄 설정까지 고려 ④ 전파 지연을 "없앨 수 없는 구조적 특성"으로 인정하고 완화 전략을 제시 ⑤ 마지막에 두 문제를 관통하는 공통 원리로 요약

7. Senior-level Follow-up Q&A

Q1. "Headless Service는 언제 씁니까? ClusterIP를 안 쓰는 이유가 뭡니까?"

클라이언트가 로드밸런싱된 하나의 가상 IP가 아니라 개별 Pod의 정체성(IP)을 직접 알아야 하는 경우에 씁니다. 대표적으로 StatefulSet의 각 순번(ordinal) Pod에 안정적인 DNS 이름(pod-0.svc-name...)으로 개별 접근해야 할 때, 또는 gRPC나 클라이언트 사이드 로드밸런싱을 쓰는 클라이언트가 DNS 조회로 받은 여러 IP 중 자신의 로직으로 직접 커넥션 분산을 하고 싶을 때입니다. ClusterIP를 거치면 커널 레벨 로드밸런싱이 이미 한 번 개입해버려서, 클라이언트가 개별 백엔드를 구분하거나 자신만의 로드밸런싱 전략을 적용할 방법이 없어집니다.

Q2. "IPVS나 nftables로 바꾸면 무조건 성능이 좋아집니까?"

아닙니다. iptables는 Service/엔드포인트 수가 매우 많은 대규모 클러스터에서 규칙 순회 비용이 선형으로 늘어나는 게 문제이지, 소규모 클러스터에서는 체감 차이가 크지 않습니다. IPVS는 해시 테이블 기반이라 대규모에서 유리하지만 추가 커널 모듈 의존성과 디버깅 도구 차이 같은 운영 복잡도가 늘어납니다. nftables는 v1.33에 GA됐고 방향성 자체는 권장되지만, 아직 상대적으로 신생 모드라 CNI 플러그인 호환성을 개별적으로 확인해야 하고, 대규모 스케일에서 예상치 못한 성능 이슈가 보고되는 사례도 있습니다. '무조건 최신 모드가 낫다'가 아니라 클러스터 규모, CNI 호환성, 운영 경험을 종합해 판단해야 합니다.

Q3. "kube-proxy 없이 Cilium 같은 eBPF CNI를 쓰면 Service 개념 자체가 바뀝니까?"

Service라는 API 오브젝트와 그 의미(안정적 VIP + 엔드포인트 그룹)는 그대로입니다. 바뀌는 건 그걸 데이터 플레인에서 구현하는 방식입니다. kube-proxy는 iptables/IPVS/nftables 같은 넷필터 규칙으로 구현하고, eBPF 기반 CNI는 커널 훅에 프로그램을 붙여 규칙 순회 없이 훨씬 이른 단계에서 패킷을 처리합니다. 후자가 보통 지연과 처리량 면에서 유리하지만, eBPF 프로그램 자체의 디버깅 도구(예: iptables-save로 규칙을 보는 익숙한 방법)가 다르기 때문에 팀의 트러블슈팅 역량이 그 CNI에 맞춰 갖춰져 있는지가 도입 판단의 중요한 기준입니다.

Q4. "CoreDNS도 결국 Pod인데, CoreDNS 자체가 죽으면 클러스터 전체가 마비됩니까?"

DNS로 이름을 찾는 모든 통신에 영향을 줍니다 — 즉 서비스 이름 기반으로 서로를 호출하는 대부분의 마이크로서비스 통신이 새로 연결을 맺을 때마다 실패하게 됩니다. 다만 이미 맺어져 있던 커넥션이나 클라이언트가 IP를 캐시해 둔 경우는 즉시 영향받지 않을 수 있습니다. 그래서 CoreDNS는 기본적으로 Deployment로 여러 replica가 뜨고 PodDisruptionBudget/PriorityClass(system-cluster-critical)로 보호되며, 오토스케일러가 노드/코어 수에 맞춰 replica 수를 조정합니다. '단일 장애점처럼 보이지만 실제로는 다중화·우선순위·오토스케일로 방어된 컴포넌트'라는 것을 함께 짚어야 합니다.

Q5. "NodeLocal DNSCache를 도입하면 정말 좋기만 합니까? 트레이드오프가 있습니까?"

캐시 히트 시 CoreDNS까지 가지 않아 지연과 CoreDNS 부하를 크게 줄이지만, 모든 노드에 캐시 데몬(DaemonSet)이 추가로 뜨는 만큼 노드당 리소스 오버헤드가 생기고(Day 5에서 다룬 DaemonSet 배수 효과와 동일한 원리), 캐시 자체의 TTL 관리·무효화 로직이 새로운 실패 지점이 됩니다. 예를 들어 캐시가 오래된 레코드를 들고 있으면 실제로는 죽은 Pod IP로 계속 연결을 시도하게 될 수 있습니다. 쿼리량이 많고 지연에 민감한 대규모 클러스터에서는 이득이 비용을 상회하지만, 소규모 클러스터에서 기본으로 켤 이유는 크지 않습니다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • Service를 "실체 있는 로드밸런서"가 아니라 kube-proxy/eBPF가 구현하는 규칙 추상화로 설명한다
  • externalTrafficPolicy Local의 트래픽 드롭을 버그가 아닌 설계된 trade-off로 이해하고, 로드밸런서 헬스체크 연계까지 고려한다
  • EndpointSlice/kube-proxy 규칙 갱신에 전파 지연이 있다는 eventual consistency 감각을 갖고 있다
  • kube-proxy 모드 선택을 클러스터 규모·CNI 호환성 기준으로 판단하고, "최신이 무조건 낫다"고 단정하지 않는다
  • CoreDNS를 ndots/lameduck/NodeLocal DNSCache 같은 구체적 운영 지식으로 다룬다
약한 답변
  • "Service는 로드밸런서다"라고만 말하고 그 뒤에 무엇이 실제로 라우팅을 수행하는지 설명하지 못한다
  • ClusterIP/NodePort/LoadBalancer/Headless를 단순 기능 나열로만 답하고 포함 관계나 선택 기준을 설명하지 못한다
  • Local 정책의 트래픽 드롭을 버그로 오해하거나, 반대로 왜 드롭되는지 원리를 설명하지 못한다
  • 스케일 이벤트 직후 트래픽 누락을 "이상 현상"으로만 보고 전파 지연이라는 구조적 원인을 짚지 못한다
  • DNS 문제를 "CoreDNS 재시작하면 해결"식으로만 접근하고 근본 원인(ndots, 규칙 전파)을 보지 못한다
왜 약한 답변이 약한가

Service/DNS 계층은 "잘 동작하는 게 당연해 보이는" 인프라이기 때문에, 얕은 이해로도 평소에는 티가 안 난다. 하지만 장애가 나는 순간 이 계층의 실제 구현(규칙 기반, watch 기반 전파, DNS 증폭)을 모르면 "재시작해보자", "롤백해보자" 같은 근거 없는 시도만 반복하게 된다. 시니어는 문제를 겪기 전에 이미 이 계층의 동작 원리를 알고 있어서, 증상만 보고도 원인 후보를 좁혀 나갈 수 있어야 한다.

9. 흔한 오해

오해 1

"Service는 그 자체로 트래픽을 처리하는 실행 중인 프로세스다."

실제

Service는 API 서버에 저장된 선언(오브젝트)일 뿐이다. 실제 트래픽 라우팅은 각 노드의 kube-proxy(또는 eBPF 기반 CNI)가 그 선언을 보고 만든 커널 네트워킹 규칙이 수행한다.

오해 2

"kube-proxy가 없는 클러스터에서는 Service를 쓸 수 없다."

실제

Service라는 개념과 API는 그대로 유지된다. 다만 그걸 구현하는 데이터 플레인이 kube-proxy의 iptables/IPVS/nftables에서 Cilium 같은 CNI의 eBPF로 바뀌었을 뿐이다.

오해 3

"externalTrafficPolicy: Local로 트래픽이 드롭되는 건 설정이 잘못됐거나 버그다."

실제

Local은 client IP 보존을 위해 SNAT을 생략하는 대신, 로컬 엔드포인트가 없는 노드로 트래픽이 오면 의도적으로 드롭하도록 설계돼 있다. 이 정책을 쓰려면 로드밸런서 헬스체크가 이 동작을 인지하도록 별도로 맞춰야 한다.

오해 4

"Pod가 Ready 상태가 되는 순간 클러스터의 모든 곳에서 즉시 트래픽을 받기 시작한다."

실제

Ready 상태 반영과 EndpointSlice 갱신, 그리고 그 갱신이 모든 노드의 kube-proxy 규칙(또는 CoreDNS 캐시)에 전파되는 것 사이에는 watch 기반의 전파 지연이 있다. 대규모 클러스터일수록, 대량 스케일 이벤트일수록 이 간격이 눈에 띄게 커질 수 있다.

오해 5

"nftables가 GA됐으니 이제 기본값이고, 안 쓰면 뒤처진 것이다."

실제

nftables는 v1.33에 stable이 됐지만 2026-07 시점에도 기본값은 여전히 iptables다. 방향성으로는 권장되지만, CNI 호환성과 대규모 스케일에서의 실사용 검증이 아직 iptables/IPVS만큼 축적되지 않았기 때문에 무조건 전환이 정답은 아니다.
