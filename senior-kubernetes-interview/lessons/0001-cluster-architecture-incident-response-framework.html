2. 전제 개념 — 쉬운 한국어로

Kubernetes는 "선언한 상태를 계속 맞춰주는 시스템"이다

Kubernetes에 "Pod 3개를 이 이미지로 띄워줘"라고 선언(desired state)하면, Kubernetes는 그 상태가 실제로 유지되도록 끊임없이 확인하고 고친다. 이걸 reconciliation loop(조정 루프)라고 한다. "지금 실행해라"라는 명령이 아니라 "이 상태를 유지해라"라는 목표를 계속 감시하는 구조이기 때문에, 사람이 개입하지 않아도 컨테이너가 죽으면 다시 띄우고, 노드가 사라지면 다른 노드로 옮긴다.

이 목표(desired state)와 실제 상태(actual state)는 모두 etcd라는 저장소에 기록된다. Kubernetes의 거의 모든 컴포넌트는 이 상태를 감시(watch)하다가, 목표와 실제가 다르면 실제를 목표에 맞추는 동작을 한다.

컨트롤 플레인 — "무엇을 해야 하는지 결정하는 곳"

컨트롤 플레인은 클러스터의 두뇌다. 실제로 컨테이너를 실행하지 않고, "어떤 노드에 무엇을 실행해야 하는가"를 결정만 한다.

kube-apiserver
모든 요청의 관문. kubectl, 컨트롤러, kubelet 모두 apiserver를 거쳐야만 상태를 읽고 쓸 수 있다. 상태 없이(stateless) 동작해서 여러 개를 두고 로드밸런서로 앞단을 묶어 수평 확장할 수 있다.
etcd
클러스터의 유일한 진실 소스(Single Source of Truth). 분산 key-value 저장소이며 Raft 합의 알고리즘으로 여러 노드 간 데이터 일관성을 보장한다. etcd가 없으면 클러스터의 "기억"이 없다.
kube-scheduler
아직 노드가 배정되지 않은 Pod를 감시하다가, 리소스/제약조건에 맞는 노드를 골라 배정한다. 배정만 하고 실제로 실행시키지는 않는다.
kube-controller-manager
Node Controller, ReplicaSet Controller, Endpoint Controller 등 수십 개의 컨트롤러를 하나의 프로세스로 실행한다. "실제 상태를 목표 상태에 맞추는" 조정 루프의 핵심 집합체.
cloud-controller-manager
클라우드 벤더 종속적인 로직(LoadBalancer 프로비저닝, 노드 생명주기, 라우트)을 분리한 컴포넌트. in-tree 클라우드 프로바이더는 완전히 제거됐고, 외부 CCM이 필수가 됐다.
2026-07 기준 확인 사항

최신 안정 버전은 v1.36(2026-04 릴리스)이며, 프로젝트는 최근 3개 마이너 버전(1.34~1.36)을 지원한다. in-tree 클라우드 프로바이더 지원은 완전히 제거되어 외부 Cloud Controller Manager가 필수다. 몇 달 뒤에 이 레슨을 다시 본다면 kubernetes.io/releases에서 최신 버전을 다시 확인하자.

노드(데이터 플레인) — "실제로 컨테이너를 실행하는 곳"

  • kubelet: 각 노드에서 실행되는 에이전트. apiserver를 지속적으로 watch하면서 "이 노드에 배정된 Pod 목록"을 받아, 컨테이너 런타임에게 실제 실행을 지시한다. 컨테이너 상태를 감시하고 죽으면 재시작하는 것도 kubelet의 로컬 책임이다.
  • kube-proxy: Service의 가상 IP를 실제 Pod IP로 연결하는 네트워크 규칙(iptables 또는 nftables)을 각 노드에 구성한다. v1.33부터 nftables 모드가 stable이지만 기본값은 여전히 iptables다.
  • 컨테이너 런타임 (containerd 등): CRI(Container Runtime Interface)를 통해 kubelet의 지시를 받아 실제로 컨테이너를 생성/삭제한다.
핵심 포인트 — 컨트롤 플레인과 데이터 플레인은 분리돼 있다

컨트롤 플레인은 "결정"을 하고, 데이터 플레인(노드)은 "실행과 서빙"을 한다. 이 분리를 이해하면 장애 대응의 첫 단추를 정확히 끼울 수 있다: 컨트롤 플레인이 죽어도 이미 배정되고 실행 중인 워크로드는 당장 멈추지 않는다. kubelet은 마지막으로 받은 Pod 목록을 로컬에 캐시해두고, apiserver 없이도 그 목록 기준으로 컨테이너를 감시·재시작할 수 있기 때문이다. 다만 새로운 배포, 스케일링, Service 엔드포인트 갱신처럼 "새로운 결정"이 필요한 동작은 즉시 멈춘다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 구조적 사고: 장애가 났을 때 "일단 재시작해본다"가 아니라, 컨트롤 플레인/노드/워크로드/네트워크 중 어느 계층 문제인지 좁혀나가는 체계적 진단 능력을 보는가.
  • blast radius 판단력: 컴포넌트 하나가 죽었을 때 실제로 무엇이 멈추고 무엇이 계속 동작하는지 정확히 아는가. "apiserver가 죽으면 서비스도 다 죽는다"처럼 뭉뚱그려 답하면 실무 경험이 얕아 보인다.
  • 실무 경험: etcd 쿼럼, 인증서 만료, 노드 리소스 압박 같은 실제로 겪어본 사람만 구체적으로 말할 수 있는 장애 패턴을 아는가.
  • 침착한 커뮤니케이션: 장애 상황에서 "무엇을 먼저 확인하고, 무엇을 나중에 확인하는지" 우선순위를 명확히 언어화할 수 있는가. 이는 실제 온콜 상황에서 팀을 이끄는 능력과 직결된다.

이 질문은 "Kubernetes 컴포넌트 이름을 아는가"를 묻지 않는다. "장애 상황에서 침착하게 범위를 좁혀가는 사고 과정을 가지고 있는가"를 묻는다. 리소스 정의를 외운 사람과, 실제로 새벽 3시에 노드가 무더기로 NotReady가 됐을 때 대응해본 사람은 답변의 구체성에서 확연히 갈린다.

4. 실무 연결

실제로 이 지식이 필요한 순간들

시나리오 1 — etcd 디스크 풀(disk full)

etcd가 저장하는 리비전 히스토리가 압축(compaction)되지 않고 계속 쌓여 디스크가 가득 찬 사례. etcd가 쓰기 요청을 거부하기 시작하면서 apiserver를 통한 모든 쓰기(create/update/delete)가 실패한다.

중요한 것은, 이미 실행 중이던 Pod들은 계속 트래픽을 처리하고 있었다는 점이다. 대시보드의 5xx 그래프는 평온했지만, kubectl apply가 전부 타임아웃되면서 배포가 전면 중단됐다. etcd 정기 압축(compaction)과 디스크 용량 알람의 필요성을 체감하게 된다.

시나리오 2 — 컨트롤 플레인 인증서 만료

kubeadm으로 구축한 클러스터에서 컨트롤 플레인 인증서(기본 1년 유효)가 만료되면서 kubectl, kubelet-apiserver 통신이 전부 인증 실패로 끊긴 사례.

이때도 노드에서 이미 실행 중인 Pod는 서빙을 계속했다. 하지만 새 노드 heartbeat이 갱신되지 않으면서 일정 시간 후 노드들이 apiserver 관점에서 NotReady로 표시됐고, 이는 실제 노드 장애가 아니라 "제어 채널 단절"이었다. 인증서 만료일을 사전에 추적하는 것이 왜 운영 체크리스트에 반드시 들어가야 하는지 알게 되는 사례다.

시나리오 3 — 노드 리소스 압박으로 인한 연쇄 Eviction

일부 노드에서 디스크 사용량이 임계치를 넘어 kubelet이 DiskPressure 상태로 전환, 해당 노드의 Pod들을 대량 축출(evict)한 사례. 축출된 Pod들이 남은 노드로 재스케줄되면서 남은 노드도 연쇄적으로 리소스 압박을 받는 cascading failure(연쇄 장애)로 번졌다.

노드 하나의 장애가 클러스터 전체 용량 부족으로 번질 수 있다는 것, 그리고 이런 상황에서는 즉시 신규 노드를 늘리거나 저우선순위 워크로드를 먼저 내리는 판단이 필요하다는 것을 배우게 된다.

5. 답변 사고 순서

면접에서 "장애가 발생했을 때 어떻게 접근하시겠습니까"라는 질문을 받으면, 아래 순서로 생각하고 답변하라. 실제 온콜에서도 동일한 순서를 쓴다.

  1. 범위(blast radius) 파악

    전체 클러스터 문제인가, 특정 노드 그룹인가, 특정 네임스페이스/워크로드만인가? kubectl get nodes, kubectl get pods -A 로 큰 그림부터 본다. 범위가 넓을수록 컨트롤 플레인/인프라 문제일 확률이 높고, 좁을수록 특정 워크로드나 배포 문제일 확률이 높다.

  2. 컨트롤 플레인 상태 확인

    apiserver가 응답하는가(지연시간 포함), etcd가 쿼럼을 유지하고 있는가, scheduler/controller-manager가 leader election을 정상적으로 유지하고 있는가. 매니지드 서비스라면 벤더 상태 페이지와 이벤트 로그를 함께 본다.

  3. 노드 상태 확인

    NotReady 노드가 있다면 원인이 kubelet 자체 문제인지, 네트워크 단절(제어 채널만 끊긴 것인지)인지, 실제 리소스 압박(disk/memory pressure)인지 구분한다. kubectl describe node의 Conditions와 Events가 첫 단서다.

  4. 워크로드 레벨 확인

    Pod 이벤트, 최근 배포 이력(rollout history), 리소스 사용량을 본다. 장애 시작 시각과 최근 배포/설정 변경 시각이 겹치는지 항상 먼저 확인한다 — 실무 장애의 상당수는 최근 변경이 원인이다.

  5. 네트워크 계층 확인

    CNI, CoreDNS, kube-proxy 규칙 등 네트워크 계층에서 문제가 없는지 본다. 애플리케이션은 정상인데 통신만 안 되는 경우가 여기 해당한다.

  6. 완화(mitigation) 조치와 근본 원인 파악의 우선순위 분리

    서비스 영향이 있다면 근본 원인을 다 밝히기 전에 먼저 완화한다 — 최근 배포 롤백, 문제 노드 격리(cordon/drain), 트래픽 우회, 스케일 조정. "원인을 다 알아낸 다음 고치겠다"는 태도는 장애 지속 시간을 늘린다.

  7. 재발 방지

    postmortem을 작성하고, 이번 장애를 더 빨리 감지/대응할 수 있는 알람이나 런북이 있었는지, 근본 원인을 구조적으로 막을 방법(예: PodDisruptionBudget, 리소스 quota, 인증서 자동 갱신)이 있는지 정리한다.

핵심 팁

이 순서를 그대로 외워서 말하는 것보다, "왜 이 순서인가"를 설명할 수 있는 게 더 중요하다. 범위가 넓은 것부터 좁혀 들어가는 이유는, 좁은 범위(개별 Pod)부터 파고들면 컨트롤 플레인 전체 장애일 때 시간을 낭비하기 때문이다.

6. 핵심 답변 예시

면접 답변 예시 — "여러 노드가 갑자기 NotReady가 됐습니다"

"먼저 범위를 확인하겠습니다. kubectl get nodes로 몇 개 노드가 영향을 받았는지, 특정 AZ나 특정 노드 그룹에 몰려있는지 봅니다. 만약 노드 전체 또는 대부분이 동시에 NotReady라면, 개별 노드 문제보다 컨트롤 플레인과의 통신 문제이거나 공통 인프라(네트워크, DNS, 인증서) 문제일 가능성을 먼저 의심합니다.

다음으로 컨트롤 플레인 상태를 확인합니다. apiserver 응답 지연이 있는지, etcd 쿼럼이 유지되고 있는지 봅니다. 매니지드 클러스터라면 벤더의 상태 대시보드도 함께 확인합니다. 이 시점에 전체 노드가 동시에 NotReady로 바뀐 게 실제 노드 장애가 아니라 '컨트롤 플레인이 노드 heartbeat을 받지 못해서 그렇게 보이는 것'일 수 있다는 가능성을 열어둡니다.

컨트롤 플레인이 정상이라면 개별 노드로 좁힙니다. kubectl describe node로 Conditions(DiskPressure, MemoryPressure, NetworkUnavailable)와 최근 Events를 봅니다. kubelet 로그도 함께 확인해서 kubelet 프로세스 자체가 크래시한 것인지, 네트워크 단절인지 구분합니다.

이 시점에서 서비스 영향이 이미 발생하고 있다면, 원인을 다 규명하기 전에 완화 조치부터 취합니다. 예를 들어 남은 정상 노드로 트래픽이 충분히 처리되는지 용량을 확인하고, 부족하면 즉시 노드를 추가합니다. 문제 노드는 cordon으로 격리해서 추가 스케줄링을 막습니다.

상황이 안정된 후에는 postmortem을 작성해서, 이번 장애를 더 빨리 감지할 수 있었던 알람이 있었는지, 구조적으로 재발을 막을 방법(예: 노드 heartbeat 임계값 조정, 리소스 압박 사전 경고)이 있는지 정리합니다."

이 답변이 좋은 이유: ① 범위 파악으로 시작해 넓은 것에서 좁은 것 순으로 진행 ② "노드 문제처럼 보이지만 실제로는 컨트롤 플레인 문제일 수 있다"는 blast radius 판단이 들어감 ③ 원인 규명과 완화 조치를 분리해서 우선순위를 명확히 함 ④ 재발 방지까지 마무리함

7. Senior-level Follow-up Q&A

Q1. "kube-apiserver가 완전히 다운되면 이미 실행 중인 Pod들은 어떻게 됩니까?"

계속 실행된다. kubelet은 마지막으로 apiserver로부터 받은 Pod 스펙을 로컬에 캐시해두고, 그 캐시를 기준으로 컨테이너가 죽으면 재시작하는 자가치유(self-healing)를 apiserver 없이도 수행할 수 있다. kube-proxy가 구성해둔 네트워크 규칙도 그대로 유지되므로 Service를 통한 트래픽 라우팅도 계속된다.

다만 새로운 결정이 필요한 모든 동작은 멈춘다: 새 배포/스케일링 반영, HPA에 의한 오토스케일링(메트릭 조회와 결정이 apiserver를 거침), Service 엔드포인트 갱신(새로 뜬 Pod가 있어도 반영되지 않음), 노드 장애 시 재스케줄링. 즉 "현재 상태 유지"는 되지만 "변화에 대응"은 멈춘다.

Q2. "etcd 쿼럼을 잃으면 어떻게 됩니까? 왜 etcd는 홀수 개의 노드로 구성합니까?"

etcd는 Raft 합의 알고리즘을 쓰는데, 쓰기 작업이 성사되려면 과반수(quorum) 노드의 동의가 필요하다. 5개 노드 클러스터라면 3개가 동의해야 쓰기가 확정된다. 쿼럼을 잃으면(과반수 노드가 죽으면) etcd는 새로운 쓰기를 확정할 수 없게 되어 사실상 읽기 전용 또는 완전 불가 상태가 되고, apiserver를 통한 모든 변경 작업이 실패한다.

홀수 개를 쓰는 이유는 짝수여도 내결함성이 늘지 않기 때문이다. 4개 노드는 2개 장애까지만 견디는데(과반수 3 필요), 5개 노드도 동일하게 2개 장애까지 견딘다. 즉 짝수는 노드 하나를 더 쓰면서 얻는 이득이 없고, 오히려 네트워크 분할 시 표(vote)가 정확히 반으로 갈리는 상황(무승부)이 생길 수 있어 홀수가 항상 유리하다.

Q3. "매니지드 Kubernetes(EKS/GKE/AKS)를 쓰면 이런 지식이 필요 없지 않나요?"

컨트롤 플레인 운영 부담(etcd 백업, apiserver 이중화, 인증서 갱신)은 벤더가 대신 지지만, 장애가 아예 안 나는 것은 아니다. 벤더 쪽 컨트롤 플레인 장애도 실제로 발생하며, 이때는 벤더의 상태 페이지와 감사 로그(CloudTrail, Cloud Audit Logs 등)로 간접적으로 추론해야 하기 때문에 오히려 원리를 더 잘 알아야 한다.

또한 노드/워크로드 레벨 장애(리소스 압박, CrashLoopBackOff, 네트워크 문제)는 매니지드 여부와 무관하게 동일하게 발생한다. "컨트롤 플레인은 벤더 책임, 그 위의 워크로드는 내 책임"이라는 경계를 정확히 아는 것 자체가 시니어의 신호다.

Q4. "static pod가 무엇이고, 왜 컨트롤 플레인 컴포넌트를 static pod로 실행하는 클러스터가 많습니까?"

static pod는 apiserver가 아니라 특정 노드의 kubelet이 로컬 매니페스트 파일(예: /etc/kubernetes/manifests/)을 직접 읽어서 띄우는 Pod다. apiserver를 거치지 않기 때문에, kubeadm 기반 클러스터에서는 kube-apiserver 자신, etcd, kube-scheduler, kube-controller-manager를 static pod로 실행한다.

이렇게 하는 이유는 "닭이 먼저냐 달걀이 먼저냐" 문제를 피하기 위해서다 — apiserver를 일반 Pod로 apiserver 위에서 관리하려면 apiserver가 이미 떠 있어야 하는 모순이 생긴다. kubelet이 apiserver 없이도 로컬 파일 기준으로 이 컴포넌트들을 계속 살려두기 때문에, 컨트롤 플레인 컴포넌트가 죽어도 해당 노드의 kubelet이 스스로 재시작한다.

Q5. "노드가 NotReady가 된 뒤 Pod가 다른 노드로 옮겨지기까지 얼마나 걸립니까? 왜 즉시 옮기지 않습니까?"

기본적으로 node-monitor-grace-period(기본 40초) 동안 heartbeat이 없으면 노드가 NotReady로 표시되고, 이후 pod-eviction-timeout(기본 5분, 최신 버전은 taint 기반 eviction으로 노드별 tolerationSeconds로 조정 가능)이 지나야 그 노드의 Pod들이 축출되고 재스케줄링된다.

즉시 옮기지 않는 이유는, 네트워크 일시 단절처럼 짧은 순간의 통신 문제일 수도 있는데 즉시 축출하면 오히려 불필요한 재스케줄링 폭증(thundering herd)과 서비스 흔들림을 유발하기 때문이다. 이 유예 시간은 "빠른 복구"와 "불필요한 흔들림 방지" 사이의 trade-off이며, 클러스터 특성에 따라 조정할 수 있다는 것을 아는 게 시니어 답변의 신호다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 범위(blast radius)부터 파악하고, 넓은 범위에서 좁은 범위 순으로 진단한다
  • 컨트롤 플레인 장애와 노드 장애를 구분해서 말한다 ("노드가 NotReady로 보여도 실제로는 컨트롤 플레인과의 통신 문제일 수 있습니다")
  • "이미 실행 중인 워크로드는 계속 서빙된다"처럼 구체적인 blast radius를 안다
  • 원인 규명과 완화 조치를 분리해서 우선순위를 명확히 한다
  • 재발 방지까지 답변에 포함한다
약한 답변
  • 컴포넌트 이름만 나열한다 ("apiserver, etcd, scheduler, controller-manager가 있습니다")
  • "apiserver가 죽으면 서비스도 다 죽는다"처럼 컨트롤 플레인과 데이터 플레인을 구분하지 못한다
  • 바로 "노드를 재부팅하겠습니다"처럼 원인 파악 없이 조치부터 말한다
  • 범위 확인 없이 개별 Pod 로그부터 파고든다
  • 재발 방지나 postmortem을 언급하지 않는다
왜 약한 답변이 약한가

컴포넌트 이름 나열은 문서를 읽은 사람이면 누구나 할 수 있다. 시니어 면접에서는 각 컴포넌트가 죽었을 때 실제로 무엇이 멈추고 무엇이 계속 동작하는지, 그리고 그 판단을 바탕으로 어떤 순서로 진단하고 조치하는지를 본다. "재부팅해보겠습니다"는 근본 원인을 우연히 가릴 뿐, 구조적 이해를 보여주지 못한다.

9. 흔한 오해

오해 1

"컨트롤 플레인이 다운되면 서비스도 즉시 다운된다."

실제

이미 실행 중인 Pod는 kubelet이 로컬 캐시 기준으로 계속 관리하고, kube-proxy 네트워크 규칙도 유지되므로 기존 트래픽 서빙은 계속된다. 멈추는 것은 "새로운 결정"이 필요한 동작(배포, 오토스케일링, 신규 Pod의 엔드포인트 반영, 장애 노드의 재스케줄링)이다. 다만 컨트롤 플레인 장애가 길어질수록 이 누적된 "결정 지연"이 결국 서비스에도 영향을 준다.

오해 2

"kube-apiserver를 여러 개 두면 자동으로 로드밸런싱되고 고가용성이 확보된다."

실제

apiserver 여러 인스턴스는 stateless라서 수평 확장이 "가능"할 뿐, 자동으로 앞단에 로드밸런서가 생기지 않는다. 별도의 로드밸런서(클라우드 LB 또는 kube-vip 같은 솔루션)로 여러 apiserver 앞단을 묶어야 실제 고가용성이 된다. kubelet, 컨트롤러, kubectl 모두 이 로드밸런서 엔드포인트를 바라보도록 구성해야 한다.

오해 3

"kubectl 명령이 안 되면 클러스터 전체가 죽은 것이다."

실제

kubectl이 안 되는 것은 "내 로컬 환경에서 apiserver에 접근하지 못한다"는 뜻일 뿐이다. apiserver 자체는 정상이고 네트워크/인증서/kubeconfig 문제일 수도 있고, apiserver가 정말 다운됐어도 데이터 플레인(노드, 워크로드)은 앞서 설명한 대로 상당 부분 계속 동작한다. "kubectl 불통 = 전체 장애"로 성급히 결론 내리면 잘못된 대응(예: 불필요한 노드 재부팅)으로 이어질 수 있다.

오해 4

"etcd는 그냥 데이터베이스라서, 백업만 있으면 아무 때나 복구하면 된다."

실제

etcd 스냅샷 복구는 그 시점의 클러스터 상태로 "되돌리는" 것이다. 복구 시점 이후 생성/삭제된 리소스나, 클라우드 LoadBalancer처럼 클러스터 밖의 실제 인프라와 상태가 어긋날 수 있다. 예를 들어 복구 후 etcd에는 없지만 실제 클라우드에는 남아있는 LoadBalancer가 고아(orphan) 리소스로 남을 수 있다. 스냅샷 백업 주기, RPO(Recovery Point Objective), 복구 후 정합성 점검 절차까지 함께 설계해야 한다.

오해 5

"노드가 NotReady가 되면 그 노드의 Pod들은 즉시 다른 노드로 옮겨진다."

실제

즉시 옮겨지지 않는다. node-monitor-grace-period와 taint 기반 eviction 유예 시간(기본 5분)이 지나야 축출과 재스케줄링이 시작된다. 이 유예 시간 동안은 해당 노드의 Pod가 "떠 있지만 응답하지 않는" 애매한 상태로 남을 수 있고, 이는 의도된 설계다(짧은 네트워크 단절에 과민 반응하지 않기 위해).
