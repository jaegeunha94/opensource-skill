2. 전제 개념 — 쉬운 한국어로

Deployment는 직접 Pod를 관리하지 않는다 — ReplicaSet을 통해서 관리한다

Deployment를 수정하면(이미지 태그 변경 등) Kubernetes는 기존 Pod를 바로 바꾸지 않는다. 대신 새로운 ReplicaSet을 하나 더 만들고, 새 ReplicaSet의 replica 수를 서서히 늘리면서 기존 ReplicaSet의 replica 수를 서서히 줄인다. 즉 하나의 Deployment 아래에는 배포 이력마다 ReplicaSet이 하나씩 쌓이고, 이 ReplicaSet 목록 자체가 롤백을 가능하게 하는 "세대(revision) 역사"다.

revisionHistoryLimit은 이 옛 ReplicaSet을 몇 개까지 보관할지 정하는 값이다(기본 10). 이 값을 0으로 설정하면 롤백 자체가 불가능해진다 — 이전 세대의 ReplicaSet과 Pod 템플릿 정보가 아예 사라지기 때문이다.

maxSurge와 maxUnavailable — "얼마나 넘치게" vs "얼마나 부족하게"를 허용할지

maxSurge — 목표치보다 얼마나 더 띄울 수 있는가
새 버전 Pod를 먼저 추가로 띄운 뒤 구버전을 줄이는 방식이다. 예를 들어 replica 10에 maxSurge 25%면, 롤아웃 중 최대 12~13개까지 동시에 떠 있을 수 있다. 클러스터에 여유 자원이 있어야 하고, 순간적으로 자원 사용량이 늘어난다.
maxUnavailable — 목표치보다 얼마나 덜 떠 있어도 되는가
구버전 Pod를 먼저 줄인 뒤 새 버전을 채우는 정도를 허용하는 값이다. replica 10에 maxUnavailable 25%면, 롤아웃 중 최소 7~8개까지만 있어도 허용한다. 자원은 절약되지만, 그만큼 처리 가능한 트래픽 용량이 순간적으로 줄어든다.

두 값 모두 percentage 또는 절대값으로 지정할 수 있고, 기본값은 각각 25%다. 지연시간/가용성이 중요한 서비스는 maxUnavailable을 0에 가깝게, maxSurge로 여유를 두는 방향이 일반적이다. 반대로 자원이 빠듯한 클러스터에서는 maxSurge를 낮추고 maxUnavailable을 어느 정도 허용하는 트레이드오프를 택한다.

롤아웃이 "멈췄다"는 것을 Kubernetes가 어떻게 판단하는가 — progressDeadlineSeconds

새 ReplicaSet의 Pod가 계속 Ready 상태가 되지 못하면(CrashLoopBackOff, readiness probe 실패 등), Deployment는 무한정 기다리지 않는다. progressDeadlineSeconds(기본 600초) 동안 진전이 없으면 Deployment의 Progressing Condition이 False로 바뀌고 사유가 ProgressDeadlineExceeded로 표시된다. 이것이 "이 롤아웃은 사실상 실패했다"는 Kubernetes 자체의 신호다. 다만 이 신호가 뜬다고 Kubernetes가 자동으로 롤백하지는 않는다 — 여전히 사람이나 자동화 파이프라인이 kubectl rollout undo를 실행해야 한다.

Proportional Scaling — 롤아웃 도중 replica 수가 바뀌면 무슨 일이 일어나는가

롤아웃이 진행 중인데 HPA나 수동 조작으로 replica 수를 늘리면, Deployment 컨트롤러는 이 추가 replica를 신버전/구버전 ReplicaSet에 비율대로 분배한다(더 많은 replica를 가진 쪽에 더 많이 배분). 이 메커니즘 덕분에 배포 도중 트래픽이 몰려도 전체 capacity가 급격히 무너지지 않는다.

2026-07 기준 확인 사항 — PodDisruptionBudget은 Deployment 롤아웃을 제한하지 않는다

많은 사람이 "PDB의 minAvailable/maxUnavailable이 배포 중에도 적용되어 안전망 역할을 한다"고 오해한다. 하지만 Kubernetes 공식 문서는 명확히 구분한다 — Deployment/StatefulSet 같은 워크로드 컨트롤러가 스스로 실행하는 rolling update는 PDB의 제한을 받지 않는다. 롤아웃 중 동시에 내려가는 Pod 수를 조절하는 것은 오직 Deployment 자체의 maxUnavailable/maxSurge다. PDB는 노드 drain, cluster-autoscaler 축소, Karpenter consolidation 같은 "자발적 축출(voluntary eviction)"이 발생할 때만 개입한다.

추가로 v1.27부터 unhealthyPodEvictionPolicy가 안정화되어 실무에서 쓸 수 있다. 기본값(IfHealthyBudget)은 PDB의 healthy pod 수가 기준을 충족해야만 unhealthy Pod의 축출도 허용하는데, 이 때문에 이미 망가진 Pod가 오히려 노드 drain을 막아버리는 사례가 있었다. AlwaysAllow로 설정하면 unhealthy Pod는 PDB 조건과 무관하게 항상 축출을 허용해 이런 교착을 막을 수 있다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 배포 안정성 설계 능력: maxSurge/maxUnavailable을 서비스 특성(지연시간 민감도, 자원 여유)에 맞게 조정할 수 있는가, 아니면 기본값을 그냥 쓰는가.
  • 롤백 판단 기준: "일단 롤백"이 항상 정답이 아니다 — 데이터 마이그레이션을 동반한 배포처럼 롤백이 오히려 더 위험한 상황을 구분할 수 있는가.
  • 보호 메커니즘 간의 경계 이해: PDB와 Deployment rollout이 서로 다른 메커니즘이라는 것을 알고 있는가. 이 경계를 모르면 "PDB를 걸어뒀으니 배포는 안전하다"는 잘못된 안도감을 갖게 된다.
  • 운영 신호를 읽는 능력: ProgressDeadlineExceeded 같은 Kubernetes 자체 신호를 알아보고, 이것이 자동 롤백을 의미하지 않는다는 것까지 이해하는가.

이 주제가 반복해서 나오는 이유는, 배포는 시니어 엔지니어가 가장 자주, 그리고 가장 위험한 순간에 개입하는 작업이기 때문이다. 배포 중 장애는 "코드가 잘못됐다"만큼이나 "배포 전략과 안전장치 설계가 잘못됐다"에서 비롯되는 경우가 많고, 이 둘을 구분하는 능력이 시니어와 미드레벨을 가른다.

4. 실무 연결

시나리오 1 — "PDB를 걸어뒀으니 배포는 안전하다"고 믿었던 사례

팀에서 minAvailable: 80%인 PDB를 설정해두고 "이 정도면 배포 중에도 최소 80%는 살아있겠지"라고 안심했다. 그런데 새 버전에 심각한 버그가 있어 신버전 Pod가 뜨는 족족 CrashLoopBackOff에 빠졌고, maxUnavailable: 50%로 설정된 Deployment는 구버전 Pod를 절반이나 내려버린 뒤 새 버전이 못 올라오는 상태로 멈췄다. 가용 Pod 비율이 PDB 기준선 아래로 내려갔는데도 PDB는 이를 막지 못했다 — PDB는애초에 롤아웃을 감시하는 장치가 아니었기 때문이다.

이 사례로 배우는 것은, 배포 안전은 PDB가 아니라 Deployment 자체의 maxUnavailable/maxSurge 설계와 readiness probe의 정확성에 달려 있다는 것이다. PDB는 별개 계층(노드 유지보수, 오토스케일러 축소)의 안전장치다.

시나리오 2 — 롤백이 오히려 더 위험했던 배포

새 버전이 DB 스키마 마이그레이션(컬럼 추가 + 데이터 백필)을 동반한 배포였는데, 애플리케이션 레벨에서 사소한 버그가 발견돼 팀이 반사적으로 kubectl rollout undo를 실행했다. 문제는 구버전 코드가 이미 마이그레이션된 스키마를 모르는 상태였고, 구버전이 다시 트래픽을 받으면서 새 컬럼을 채우지 않은 채 쓰기 작업을 계속해 데이터 정합성이 깨졌다.

이후 이 팀은 "배포와 스키마 마이그레이션을 같이 묶지 않는다(expand/contract 패턴)"는 원칙을 세웠고, 애플리케이션 버그만 있는 롤아웃과 데이터 마이그레이션이 얽힌 롤아웃을 구분해서 롤백 여부를 판단하는 체크리스트를 만들었다.

시나리오 3 — maxSurge 기본값이 클러스터 자원을 고갈시킨 사례

대형 서비스(replica 200)를 기본값(maxSurge: 25%)으로 배포했더니, 롤아웃 도중 순간적으로 250개까지 Pod가 뜨려 하면서 노드 오토스케일러가 따라가지 못해 신규 Pod 다수가 Pending 상태로 남았다. 클러스터 CPU/메모리가 빠듯한 시간대와 겹쳐 롤아웃이 지연되고, 다른 서비스의 스케줄링에도 영향을 줬다.

이후 대형 서비스는 maxSurge를 절대값(예: 10)으로 낮추고, 배포 시간대를 트래픽이 낮은 시간으로 조정하는 정책을 도입했다. "퍼센트 기본값이 모든 규모의 서비스에 안전하지는 않다"는 것이 교훈이었다.

5. 답변 사고 순서

"롤아웃이 멈췄습니다"라는 질문을 받으면 아래 순서로 답하라. 상태 확인 → 원인 분류 → 영향 범위 판단 → 롤백/전진 결정 → 실행 → 재발 방지 순서다.

  1. 롤아웃 상태를 정확히 확인한다

    kubectl rollout status deployment/<name>와 kubectl describe deployment의 Conditions를 본다. Progressing=False, Reason=ProgressDeadlineExceeded인지, 아니면 아직 진행 중(느릴 뿐)인지 구분한다.

  2. 새 ReplicaSet의 Pod 상태와 이벤트를 본다

    kubectl describe pod와 kubectl logs --previous로 CrashLoopBackOff/ImagePullBackOff/readiness 실패 중 무엇인지 원인을 좁힌다. 애플리케이션 버그, 설정값 오류, 이미지 문제, 의존 서비스 문제를 구분한다.

  3. 현재 가용 capacity와 영향 범위를 판단한다

    구버전 Pod가 아직 얼마나 남아있는지, 실제 사용자 트래픽에 영향이 있는지, 아니면 신버전만 못 뜨고 서비스는 정상인지 구분한다. maxUnavailable 설정이 컸다면 이미 실사용자 영향이 있을 수 있다.

  4. 롤백이 안전한지, 오히려 위험한지 판단한다

    이번 배포에 스키마 마이그레이션이나 비가역적 데이터 변경이 포함됐는지 확인한다. 순수 애플리케이션 버그라면 즉시 롤백이 정답에 가깝지만, 데이터 마이그레이션이 얽혀 있다면 롤백보다 hotfix 전진이 더 안전할 수 있다.

  5. 결정한 방향을 실행한다

    롤백이면 kubectl rollout undo deployment/<name> [--to-revision=N]을 실행하고 다시 rollout status로 안정화를 확인한다. 전진이면 롤아웃을 kubectl rollout pause로 멈춰 추가 피해를 막고 hotfix를 준비한다.

  6. 재발 방지책을 제시한다

    readiness probe가 실제 버그를 제때 잡아냈는지 점검하고, canary/blue-green 같은 점진적 배포 전략, 스키마 변경은 expand/contract 패턴으로 배포와 분리하는 정책, maxSurge/maxUnavailable을 서비스 규모에 맞게 재조정하는 것을 제안한다.

6. 핵심 답변 예시

면접 답변 예시 — "롤아웃이 중간에 멈춰 있고 새 Pod가 재시작을 반복합니다"

"먼저 kubectl rollout status와 kubectl describe deployment의 Conditions를 확인합니다. Progressing이 False이고 사유가 ProgressDeadlineExceeded라면, Kubernetes 스스로가 '이 롤아웃은 정상적으로 진행되지 않고 있다'고 판단한 겁니다. 다만 이건 자동 롤백을 의미하지 않으므로, 이 시점부터는 제가 직접 판단해야 합니다.

다음으로 새 ReplicaSet의 Pod를 describe하고 이전 로그(--previous)를 확인해서, CrashLoopBackOff인지 readiness 실패인지, 애플리케이션 버그인지 설정 오류인지를 좁힙니다.

그다음 현재 얼마나 많은 구버전 Pod가 남아 있고 실사용자 영향이 있는지를 봅니다. 그리고 이번 배포에 스키마 마이그레이션 같은 비가역적 변경이 포함됐는지를 반드시 확인합니다. 포함이 안 된 순수 애플리케이션 버그라면 지체 없이 kubectl rollout undo로 롤백합니다. 만약 마이그레이션이 얽혀 있다면 무작정 롤백하는 대신 kubectl rollout pause로 멈추고 hotfix로 전진하는 방향을 우선 검토합니다 — 구버전 코드가 이미 변경된 스키마를 모른 채 쓰기를 계속하면 데이터 정합성이 깨질 수 있기 때문입니다.

마지막으로 재발 방지를 위해 readiness probe가 이 버그를 배포 초기에 더 빨리 잡아낼 수 있었는지 점검하고, 중요 서비스에는 canary 배포를 도입해 전체 트래픽에 영향을 주기 전에 소수 Pod로 문제를 조기에 발견하는 절차를 제안합니다."

이 답변이 좋은 이유: ① Kubernetes 자체 신호(Progressing Condition)를 정확히 해석 ② 자동 롤백과 사람의 판단을 명확히 구분 ③ 롤백이 항상 정답이 아니라는 것을 스키마 마이그레이션 사례로 근거 있게 설명 ④ pause를 활용한 중간 대응 옵션까지 제시 ⑤ 재발 방지를 canary 같은 구조적 개선으로 연결

7. Senior-level Follow-up Q&A

Q1. "PodDisruptionBudget이 설정돼 있으면 이번 같은 롤아웃 장애를 막아줬을까요?"

아닙니다. PDB는 Deployment/StatefulSet이 스스로 실행하는 rolling update에는 적용되지 않습니다. Kubernetes 공식 문서에도 워크로드 컨트롤러의 rolling update는 PDB 제한을 받지 않는다고 명시돼 있습니다. 롤아웃 중 동시에 내려가는 Pod 수를 통제하는 것은 오직 Deployment 자체의 maxUnavailable입니다. PDB는 노드 drain, cluster-autoscaler 축소, Karpenter consolidation처럼 외부에서 발생하는 자발적 축출을 막는 장치입니다.

이 둘을 혼동하면 "PDB를 걸어뒀으니 배포는 안전하다"는 잘못된 안도감을 갖게 되고, 실제로는 Deployment의 maxUnavailable/maxSurge와 readiness probe 품질에 배포 안전을 전혀 의존하지 못하는 상태가 됩니다.

Q2. "maxUnavailable을 0으로 설정하면 항상 안전한가요?"

가용성 측면에서는 안전하지만 대가가 있습니다. maxUnavailable: 0이면 구버전 Pod를 하나도 안 줄이고 새 버전을 다 띄운 뒤에야 전환하므로, 그만큼 maxSurge를 통한 추가 자원이 필요합니다. 클러스터에 여유 자원이 없으면 신규 Pod가 Pending으로 남아 롤아웃 자체가 진행되지 않을 수 있습니다.

또한 replica 수가 적은 서비스(예: replica 1~2)는 maxUnavailable: 0과 정수 maxSurge를 함께 세심하게 계산해야 합니다. 퍼센트 기본값을 그대로 쓰면 반올림 때문에 의도한 것과 다른 동작이 나올 수 있습니다.

Q3. "롤백하면 정확히 이전과 100% 동일한 상태로 돌아가나요?"

Pod 템플릿(이미지, 환경변수, 리소스 설정 등) 기준으로는 이전 ReplicaSet의 스펙으로 복원됩니다. 하지만 애플리케이션 외부 상태(DB 스키마, 캐시에 쌓인 데이터, 외부 API의 상태, 발행된 메시지 큐 이벤트)는 롤백 대상이 아닙니다. 그래서 "롤백 = 완전한 시간 되돌리기"라고 생각하면 위험합니다. 특히 데이터 마이그레이션이나 비동기 이벤트 발행이 얽힌 배포는 롤백 후에도 후속 정리 작업이 필요할 수 있습니다.

Q4. "revisionHistoryLimit을 낮게(예: 2) 설정하면 어떤 위험이 있나요?"

오래된 ReplicaSet이 빨리 삭제되어, 문제가 뒤늦게 발견됐을 때 kubectl rollout undo --to-revision=N으로 여러 단계 이전 버전으로 돌아가고 싶어도 그 ReplicaSet 자체가 이미 사라져 불가능할 수 있습니다. 반대로 너무 높게 두면 클러스터에 불필요한 ReplicaSet 오브젝트가 쌓여 etcd 부담이 됩니다. 배포 빈도와 "문제 발견까지 걸리는 평균 시간"을 고려해서 값을 정해야 합니다.

Q5. "canary 배포는 Deployment의 RollingUpdate와 뭐가 다른가요? 왜 별도로 필요한가요?"

RollingUpdate는 결국 "전체 replica를 순차적으로 교체"하는 것이고, 문제가 있으면 이미 상당수 Pod가 신버전으로 바뀐 뒤에야 감지됩니다. Canary는 신버전을 소수(예: 5%) 트래픽에만 먼저 노출시켜, 실사용자 영향을 최소화한 채로 신버전의 실제 동작을 검증합니다. Kubernetes 기본 오브젝트만으로는 트래픽 비율 기반 canary를 정교하게 구현하기 어렵기 때문에, 실무에서는 Argo Rollouts/Flagger 같은 progressive delivery 컨트롤러나 서비스 메시(Istio 등)의 트래픽 분할 기능을 함께 쓰는 경우가 많습니다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • Progressing Condition과 ProgressDeadlineExceeded를 근거로 롤아웃 실패를 판단한다
  • PDB와 Deployment rollout이 서로 다른 메커니즘이라는 것을 명확히 구분한다
  • 데이터 마이그레이션이 얽힌 배포에서는 롤백이 오히려 위험할 수 있다는 것을 안다
  • maxSurge/maxUnavailable을 서비스 특성(지연시간 민감도, 자원 여유, replica 규모)에 맞게 조정하는 기준을 제시한다
  • 롤백 외에 pause, canary 같은 중간 대응 옵션을 제시한다
약한 답변
  • "일단 롤백하겠습니다"로 바로 결론 낸다 (배포 성격을 구분하지 않음)
  • PDB가 배포 중에도 안전망 역할을 한다고 잘못 설명한다
  • maxSurge/maxUnavailable을 기본값 그대로 쓰는 것을 당연하게 여긴다
  • 롤백을 "완전한 시간 되돌리기"로 생각하고 외부 상태(DB, 캐시)를 고려하지 않는다
  • revisionHistoryLimit이나 rollout history 개념 자체를 모른다
왜 약한 답변이 약한가

"일단 롤백"은 순수 애플리케이션 버그에는 맞는 대응이지만, 데이터 마이그레이션이 얽힌 배포에는 오히려 정합성을 깨는 결정이 될 수 있다. 이 구분을 못 하면 배포 판단이 아니라 반사 행동에 불과하다. 또한 PDB와 rollout 메커니즘을 혼동하면, 실제로는 아무 보호도 받지 못하는 배포를 안전하다고 착각한 채 운영하게 된다.

9. 흔한 오해

오해 1

"PodDisruptionBudget을 설정해두면 Deployment 롤아웃 중에도 최소 가용 Pod 수가 보장된다."

실제

PDB는 Deployment/StatefulSet 같은 워크로드 컨트롤러가 스스로 실행하는 rolling update에는 적용되지 않는다. 롤아웃 중 동시 축소 규모는 오직 Deployment의 maxUnavailable이 결정한다. PDB는 노드 drain, 오토스케일러 축소 같은 외부 자발적 축출만 제한한다.

오해 2

"롤백은 항상 가장 안전한 선택이다."

실제

Pod 템플릿은 이전 상태로 돌아가지만, DB 스키마나 외부 상태는 롤백되지 않는다. 새 버전이 스키마 마이그레이션을 동반했다면, 구버전 코드가 변경된 스키마를 모른 채 다시 쓰기 작업을 하면서 데이터 정합성이 깨질 수 있다. 이런 경우는 롤백보다 hotfix 전진이 더 안전할 수 있다.

오해 3

"maxSurge와 maxUnavailable 기본값(25%)은 어떤 서비스에나 적당하다."

실제

기본값은 중간 규모 서비스를 가정한 값이다. replica가 매우 많은 대형 서비스는 25% surge가 노드 오토스케일러가 따라가지 못할 만큼 큰 자원 요구를 순간적으로 만들 수 있고, replica가 매우 적은 서비스는 퍼센트 반올림 때문에 의도와 다르게 동작할 수 있다. 서비스 규모와 자원 여유에 맞게 값을 조정해야 한다.

오해 4

"ProgressDeadlineExceeded가 뜨면 Kubernetes가 알아서 롤백한다."

실제

ProgressDeadlineExceeded는 Deployment의 Progressing Condition을 False로 바꾸는 신호일 뿐, 자동 롤백을 트리거하지 않는다. 이 상태에서도 새 ReplicaSet은 여전히 존재하고 시도를 계속할 수 있다. 실제 롤백은 사람이나 CI/CD 파이프라인이 kubectl rollout undo를 명시적으로 실행해야 일어난다.

오해 5

"RollingUpdate만 잘 설정하면 canary 배포는 필요 없다."

실제

RollingUpdate는 결국 전체 replica를 순차 교체하는 전부 아니면 전무(all-eventually) 방식이라, 문제가 발견될 즈음이면 이미 상당수 Pod가 신버전으로 바뀐 뒤다. Canary는 실사용자 트래픽의 일부만으로 신버전을 검증해 영향 범위를 의도적으로 줄인다. 배포 빈도가 높고 실패 비용이 큰 서비스일수록 RollingUpdate만으로는 부족하다.
