Day 1 · LLM Wiki / File Format RAG Interview

LLM Wiki와 파일 형식 RAG 시스템 설계/운영 프레임워크

이기종 파일 형식 지식 베이스의 전체 아키텍처를 설계하고, "매번 다시 검색하는" RAG와 "지식을 누적하는" LLM Wiki 중 무엇을 선택할지 판단하며, 장애가 발생했을 때 어느 계층(수집/파싱/큐레이션/검색/생성/권한)의 문제인지 구조적으로 좁혀나가는 능력
면접 질문
Q1 — 주력 질문 "사내에는 PDF, Word, PPT, Excel, Confluence 페이지, 소스 코드, 설정 파일(YAML/JSON)이 뒤섞여 있습니다. 이걸 하나의 질의 응답 가능한 지식 베이스로 만들어야 합니다. 전체 아키텍처를 설계해주세요."
Q2 — 변형 "질문할 때마다 문서를 처음부터 다시 검색하는 방식(RAG) 대신, 지식을 미리 정리해서 누적해 두는 위키 같은 구조(LLM Wiki)로 바꾸고 싶다고 팀장이 요청합니다. 어떻게 설계하고, 기존 RAG와는 어떤 관계를 가져야 할까요?"
Q3 — 변형 "이 시스템에서 장애가 발생했습니다 — 어떤 문서는 검색이 안 되고, 어떤 답변은 오래된 정보를 말하고, 어떤 사용자는 권한이 없는 문서 내용을 답변으로 받았습니다. 어느 계층부터 점검하시겠습니까?"
전제 개념

왜 "파일 형식 RAG"와 "LLM Wiki"가 별도로 어려운 문제인가

일반 RAG 인터뷰가 가정하는 이상화된 상황

일반적인 RAG(Retrieval-Augmented Generation) 논의는 흔히 "텍스트 문서를 청킹해서 벡터 DB에 넣고 검색한다"는 단순한 그림을 가정한다. 하지만 실무의 지식 베이스는 그렇게 깔끔하지 않다. 같은 "문서 하나"라도 포맷에 따라 구조가 완전히 다르다.

포맷 구조적 특징 파싱 시 흔한 문제
PDF 레이아웃 기반(텍스트 위치 좌표), 다단 구성 가능 다단 레이아웃에서 문장이 섞여 읽힘, 스캔 이미지면 텍스트 자체가 없음
DOCX/PPTX XML 기반 구조(제목, 목록, 슬라이드 노트 등 명시적 태그) 구조 태그를 무시하고 텍스트만 뽑으면 제목/본문 위계 정보 소실
XLSX 셀 좌표 기반 표, 병합 셀, 다중 시트, 수식 단순 텍스트 변환 시 행/열 관계가 깨져 표 의미 소실
HTML/Markdown 시맨틱 태그(heading, list, table) 명시적 포함 불필요한 네비게이션/광고 영역까지 본문으로 수집
코드/CSV/JSON/YAML 엄격한 구문 구조(AST, 스키마) 자연어처럼 임의로 청킹하면 함수/레코드 경계가 잘림
핵심 전제: 파일 형식 RAG는 "청킹 전략을 어떻게 짤까"보다 먼저 "이 포맷에서 구조 정보를 어떻게 보존한 채로 텍스트를 추출할까"라는 문제를 푼다. 파싱 단계에서 구조를 잃으면, 그 뒤의 청킹·임베딩·검색을 아무리 정교하게 만들어도 이미 손실된 정보를 복구할 수 없다.

Query-time RAG vs LLM Wiki — 두 가지 지식 활용 패러다임

전통적인 RAG는 질의 시점(query-time)에 원본 문서를 검색한다. 질문이 들어올 때마다 "관련 있어 보이는 chunk"를 다시 찾는다. 이 방식은 원본 데이터가 자주 바뀌어도 최신성을 유지하기 쉽지만, 매 질문마다 파편화된 chunk에서 답을 재구성해야 하므로 같은 지식을 반복해서 다시 조립하는 비효율이 있다.

2026년 초 Andrej Karpathy가 공개한 LLM Wiki 패턴은 이와 다른 접근을 제안한다. LLM이 원본 자료(raw/)를 읽고, 이를 구조화된 마크다운 위키 페이지(wiki/)로 미리 정리해서 누적하는 것이다. 질문이 들어오면 원본을 다시 뒤지는 대신, 이미 정제된 위키 페이지를 먼저 참고한다. 스키마(schema — 어떤 종류의 페이지를 어떤 구조로 쓸지 정의한 규칙)를 두고, ingest(새 자료 반영) → query(질의 응답) → lint(위키 정합성 점검) 세 가지 연산으로 위키를 운영한다. 

# Query-time RAG: 질문마다 원본을 다시 검색
사용자 질문 → 벡터 검색(원본 chunk top-k) → LLM이 그때그때 답 조립

# LLM Wiki: 지식을 미리 정제해서 누적
원본 자료(raw/) → LLM이 구조화된 위키 페이지(wiki/) 작성·갱신
사용자 질문 → 정제된 위키 페이지 우선 참고 → 필요시 raw/ 로 근거 보강(RAG)
둘은 경쟁 관계가 아니다. LLM Wiki는 "자주 참조되고 안정적인 지식"을 미리 정리해 응답 속도와 일관성을 높이고, RAG는 "위키에 아직 반영되지 않았거나 드물게 조회되는 최신·세부 정보"를 그때그때 채운다. 시니어 면접에서는 이 둘을 계층으로 조합할 줄 아는지를 본다 — "위키가 이 정도 커버리지에 도달하면 RAG 호출량이 줄고 비용도 줄어든다"는 식의 정량적 판단까지 포함해서.

전체 아키텍처 — 이기종 문서에서 근거 있는 답변까지

1. 수집(Ingestion)
   커넥터(Confluence/Drive/S3/GitHub 등) → 포맷 감지 → 포맷별 파서 라우팅

2. 정제(Extraction)
   구조 보존 파싱(PDF/DOCX/PPTX/XLSX) + OCR/표/이미지 추출
   → 메타데이터/출처(provenance) 부착

3. 큐레이션(Curation) — 선택적, LLM Wiki 계층
   반복 조회되는 지식을 구조화된 위키 페이지로 정리·누적, lint로 정합성 점검

4. 색인(Indexing)
   문서 구조 기반 semantic chunking → embedding → incremental indexing
   → 중복 제거/버전 관리 → 권한(ACL) 메타데이터 결합

5. 검색과 생성(Retrieval & Generation)
   권한 필터링 → hybrid search → reranking
   → context 구성(prompt injection 방어) → 생성 → citation 근거 표시

6. 평가와 운영(Evaluation & Ops)
   포맷별 파싱 정확도, retrieval/generation 지표, 관측성, 장애 대응, 비용 추적

이 6단계는 Day 2~15에서 각각 깊이 다룬다. Day 1에서 기억해야 할 것은, 답변 품질 문제가 이 6단계 중 정확히 어느 단계에서 발생했는지 좁혀나가는 진단 능력이 이 트랙 전체를 관통하는 핵심 역량이라는 점이다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "벡터 DB에 파일을 넣어봤느냐"가 아니라 "이기종 데이터와 지식 운영이라는, 현실에서 가장 지저분한 부분을 구조적으로 설계하고 디버깅할 수 있느냐"를 본다.

  • 포맷 다양성을 실제로 다뤄봤는가: "PDF는 text extractor로 뽑으면 됩니다" 수준인지, 아니면 다단 레이아웃·병합 셀·스캔 이미지처럼 포맷별로 발생하는 구체적인 실패 양상을 아는지
  • 지식 운영 패러다임의 선택 기준이 있는가: query-time RAG와 LLM Wiki 같은 사전 큐레이션 방식 중 언제 무엇을 쓰는지, 왜 그런지 근거를 댈 수 있는지
  • 증상을 계층으로 좁힐 수 있는가: "검색이 안 됨" → "수집 문제인지 파싱 문제인지 청킹 문제인지" → "권한 문제인지 신선도 문제인지"로 가설을 좁혀나가는 사고를 하는가
  • 보안과 운영을 처음부터 설계에 포함하는가: 권한(ACL)과 provenance를 "나중에 추가"하는 것이 아니라 처음부터 데이터 모델의 1급 요소로 설계하는가
Senior 엔지니어는 "RAG 파이프라인을 만들었다"가 아니라, "이기종 데이터를 다루는 지식 시스템은 결국 데이터 엔지니어링 + 검색 시스템 + 보안 + 평가 체계"라는 관점에서 전체를 설계하고, 장애 시 어느 계층의 문제인지 재현 가능하게 좁혀낸다는 것을 보여줘야 한다.
실무 연결

실제 상황과 연결하기

상황 1: 스캔된 계약서 PDF에서 정보를 못 찾음

사용자가 특정 계약 조항을 물었는데 챗봇이 "찾을 수 없다"고 답한다. 확인해보니 해당 PDF는 스캔 이미지였고, 파서가 이미지에서 텍스트를 추출하지 않아 애초에 인덱스에 텍스트가 없었다. → 수집/정제 단계 실패(OCR 누락)이다. 검색 알고리즘을 아무리 튜닝해도 존재하지 않는 텍스트는 찾을 수 없다.

상황 2: 매번 같은 질문에 다른 요약이 나옴

"우리 회사 휴가 정책이 뭐야?"라는, 자주 반복되는 질문에 매번 조금씩 다른 답이 나온다. 원본 문서 여러 개에서 매번 다른 조합의 chunk가 검색되기 때문이다. → query-time RAG의 구조적 한계다. 이런 "자주 묻고 안정적인 지식"은 LLM Wiki 방식으로 하나의 정제된 페이지로 미리 정리해두면 일관성과 응답 속도가 함께 개선된다.

상황 3: 퇴사자가 접근 권한이 없어야 할 연봉 스프레드시트 내용을 답변으로 받음

벡터 검색은 "의미적으로 관련 있는 chunk"만 찾을 뿐, 그 chunk를 이 사용자가 봐도 되는지는 모른다. 애플리케이션 레이어에서만 권한을 체크하면, 검색 자체가 이미 권한 없는 chunk를 후보로 올려버려서 프롬프트 인젝션 같은 우회 경로로 유출될 수 있다. → 권한이 벡터/인덱스 계층에 강제되지 않은 설계 결함이다.

상황 4: 어제 업데이트된 정책 문서가 반영되지 않음

Confluence 문서를 수정했는데 하루가 지나도 챗봇 답변에 반영되지 않는다. → RAG 알고리즘 문제가 아니라 incremental indexing 주기와 변경 감지 문제다. 배치 재인덱싱 주기가 길거나, 변경 감지(webhook/polling)가 해당 커넥터에 구현되어 있지 않은 경우다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
데이터 다양성과 규모부터 명확히 한다 "먼저 어떤 포맷이 얼마나 있는지, 얼마나 자주 바뀌는지, 스캔 문서 비중이 얼마나 되는지부터 파악합니다. 포맷 다양성이 아키텍처의 복잡도를 결정합니다."
2
수집→파싱→메타데이터를 포맷 독립적인 파이프라인으로 분리 "커넥터가 원본을 가져오고, 포맷 감지 후 포맷별 파서로 라우팅하고, 파싱 결과는 공통 스키마(텍스트+구조+메타데이터)로 정규화합니다. 이렇게 하면 새 포맷이 추가돼도 하위 파이프라인은 그대로 재사용됩니다."
3
query-time RAG와 LLM Wiki 큐레이션을 계층으로 조합 "자주 조회되고 안정적인 지식은 위키 형태로 미리 정리해 응답 속도와 일관성을 높이고, 드물게 조회되거나 최신성이 중요한 정보는 RAG로 그때그때 원본에서 가져옵니다. 어느 쪽 비중을 늘릴지는 조회 빈도 로그로 판단합니다."
4
권한과 provenance를 처음부터 데이터 모델에 포함 "각 chunk에는 출처 문서, 원본 시스템의 접근 권한, 버전/타임스탬프를 메타데이터로 부착합니다. 권한은 애플리케이션 레이어가 아니라 검색 쿼리 자체에 필터로 강제합니다."
5
장애 진단은 계층별로 좁혀나간다 "증상이 들어오면 수집(문서가 인덱스에 있는가) → 파싱(구조/텍스트가 온전한가) → 큐레이션(위키가 최신인가) → 검색(권한 필터링 후 정답 chunk가 top-k에 있는가) → 생성(context를 제대로 근거로 삼았는가) 순서로 좁혀갑니다."
6
평가/관측성/비용을 설계에 포함 (Senior 포인트) "포맷별 파싱 정확도, retrieval/generation 지표를 분리해서 관측하고, 파싱·OCR·임베딩·재인덱싱 비용을 정량적으로 추적해서 어디에 투자할지 데이터로 판단합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 이런 시스템을 6단계로 나눠서 설계합니다. 먼저 커넥터로 이기종 소스를 수집하고, 포맷을 감지해 포맷별 파서로 라우팅한 뒤, 결과를 텍스트+구조+ 메타데이터로 정규화합니다. PDF는 레이아웃 분석과 필요시 OCR을, DOCX/PPTX는 구조 태그를, XLSX는 표 구조를 보존하는 방식으로 각각 다르게 다룹니다.

그 다음 이 데이터를 어떻게 활용할지는 조회 패턴에 따라 두 계층으로 나눕니다. 자주 반복되고 안정적으로 묻는 지식은 LLM이 구조화된 위키 페이지로 미리 정리해서 누적하고, 드물게 조회되거나 최신성이 중요한 정보는 기존 RAG처럼 질의 시점에 원본에서 검색합니다. 이렇게 하면 반복 질문에 대한 응답 일관성과 속도를 높이면서도, 위키가 아직 못 따라간 최신 정보는 RAG가 보완합니다.

권한은 처음부터 데이터 모델에 넣습니다. 모든 chunk에 출처 문서의 접근 권한을 메타데이터로 부착하고, 검색 쿼리 자체에 권한 필터를 강제해서 애플리케이션 레이어의 실수나 프롬프트 인젝션으로 우회되지 않게 합니다.

장애가 나면 저는 수집(문서가 인덱스에 존재하는지) → 파싱(구조와 텍스트가 온전한지) → 큐레이션(위키가 최신인지) → 검색(권한 필터링 후에도 정답 chunk가 top-k에 있는지) → 생성(context를 실제로 근거로 삼았는지) 순서로 가설을 좁혀갑니다. 그리고 이 모든 단계의 지표를 분리해서 관측하고, 파싱· OCR·임베딩·재인덱싱 비용을 추적해서 다음에 어디에 투자할지 데이터로 판단합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "LLM Wiki 방식이 항상 RAG보다 좋은가요?"

약한 답변: "네, 미리 정리해두니까 더 빠르고 정확합니다."

강한 답변: "아니요. LLM Wiki는 큐레이션 비용(LLM이 위키를 쓰고 갱신하는 비용)과 최신성 지연(원본이 바뀐 뒤 위키에 반영되기까지의 시차)이라는 대가가 있습니다. 조회 빈도가 낮은 롱테일 지식까지 모두 위키로 정리하면 큐레이션 비용 대비 효용이 떨어집니다. 그래서 저는 조회 로그를 기준으로 상위 빈도 지식만 위키화하고, 나머지는 RAG로 커버하는 하이브리드 구조를 기본값으로 둡니다."

F2. "이기종 파서를 운영할 때 가장 큰 운영 리스크는 무엇인가요?"

약한 답변: "파서가 에러 안 나게 잘 짜면 됩니다."

강한 답변: "가장 큰 리스크는 조용한 실패(silent failure)입니다. 파서가 에러 없이 실행됐지만 표를 깨뜨리거나, 다단 레이아웃에서 문장 순서를 뒤섞거나, OCR 신뢰도가 낮은데도 그대로 인덱싱하는 경우입니다. 에러 로그에는 안 잡히지만 검색 품질에는 치명적입니다. 그래서 저는 포맷별로 파싱 결과의 품질을 정량 지표(표 셀 수 일치율, OCR confidence score, 문단 순서 일관성 등)로 측정하고, 임계값 미만이면 인덱싱 전에 격리해서 사람이 검토하도록 합니다."

F3. "권한 모델이 서로 다른 여러 출처 시스템을 어떻게 하나로 합치나요?"

강한 답변: "각 출처 시스템(Confluence 그룹 권한, Google Drive 공유 설정, GitHub 리포지토리 권한 등)의 권한 모델을 그대로 유지하지 않고, 인덱스 계층에서 공통 권한 스키마 (예: 사용자/그룹 ID 목록)로 정규화합니다. 그리고 이 권한 메타데이터를 소스 권한이 바뀔 때마다 동기화하는 잡을 별도로 둡니다 — 동기화 지연이 있으면 '권한이 회수됐는데도 계속 검색되는' 유출 사고로 이어지기 때문에, 동기화 지연 자체를 SLO로 관리합니다."

F4. "이 시스템이 프로덕션에서 가장 먼저 병목이 생기는 곳은 어디입니까?"

강한 답변: "초기에는 포맷별 파싱 품질에서 병목이 생깁니다. 특히 스캔 PDF의 OCR과 병합 셀이 많은 Excel 표가 가장 많이 깨집니다. 데이터가 늘어나면 재인덱싱 비용과 권한 동기화 지연이 병목이 되고, 트래픽이 커지면 검색 지연시간과 생성 비용이 병목이 됩니다. 그래서 단계별 지표를 분리해서 보고, '지금 이 순간 어느 단계가 SLO를 위협하는지'를 대시보드로 바로 볼 수 있게 만듭니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 포맷별로 구체적인 파싱 실패 양상(다단 레이아웃, 병합 셀, 스캔 이미지 등)을 안다
  • query-time RAG와 LLM Wiki를 대립이 아니라 계층으로 조합하며, 선택 기준을 조회 빈도 같은 정량 근거로 설명한다
  • 권한과 provenance를 설계 초기부터 1급 데이터로 다룬다
  • 장애 진단을 수집→파싱→큐레이션→검색→생성 계층으로 좁혀나간다
  • 파싱 실패를 에러가 아니라 품질 지표로 측정해야 한다는 점을 안다(조용한 실패 개념)

약한 답변의 특징

  • "PDF도 텍스트로 뽑아서 청킹하면 됩니다" — 포맷별 차이 이해 부족
  • "위키가 RAG보다 항상 좋습니다"처럼 트레이드오프 없이 단정한다
  • 권한을 "나중에 애플리케이션에서 체크하면 된다"고 가볍게 넘긴다
  • 장애 시 "다시 인덱싱해보겠습니다" 같은 재현 불가능한 대응으로 끝난다
  • 파싱 성공 여부를 에러 발생 유무로만 판단한다
핵심 차이: 강한 답변은 "여러 파일 형식을 지원한다"가 아니라 "이기종 데이터, 권한, 지식 신선도라는 서로 다른 축의 실패를 독립적으로 진단하고 정량적으로 관리한다"는 것을 보여준다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "PDF는 다 텍스트 추출 라이브러리 하나로 처리하면 된다"

사실: 텍스트 레이어가 있는 PDF와 스캔 이미지 PDF는 완전히 다른 처리가 필요하고, 다단 레이아웃·표·이미지가 섞인 문서는 레이아웃 분석 모델 없이는 읽기 순서 자체가 깨진다. 포맷 안에서도 문서 유형별 분기가 필요하다 (Day 4, 6에서 자세히 다룬다).

오해 2: "LLM Wiki는 RAG를 완전히 대체하는 차세대 기술이다"

사실: LLM Wiki는 사전 큐레이션이 유리한 지식에 대한 보완 계층이지, 모든 지식을 대체하는 구조가 아니다. 희귀하게 조회되거나 실시간성이 중요한 정보까지 위키로 미리 정리하면 큐레이션 비용만 늘어난다 (Day 2에서 자세히 다룬다).

오해 3: "메타데이터/권한 정보는 검색이 잘 되고 나서 나중에 붙여도 된다"

사실: 권한을 인덱싱 이후에 붙이려면 전체 재인덱싱이 필요한 경우가 많고, 그 사이 기간 동안 권한 없는 사용자에게 노출되는 창(window)이 생긴다. 메타데이터/provenance/권한은 파싱 직후, 인덱싱 이전에 확정해야 한다 (Day 7, 13에서 자세히 다룬다).

오해 4: "OCR은 스캔 문서가 드물면 신경 안 써도 된다"

사실: 스캔 문서 비율이 낮아도, 놓친 문서가 하필 법무·계약·재무처럼 리스크가 큰 문서일 수 있다. OCR 커버리지는 문서 수가 아니라 업무 중요도 기준으로 판단해야 한다(Day 6에서 자세히 다룬다).

실무 체크리스트

장애 발생 시 계층별 진단 체크리스트

확인 항목 정상 신호 이상 신호 → 의심 계층
해당 문서가 인덱스에 존재하는가 있음 없음 → 수집(Ingestion) 실패
추출된 텍스트/표가 원본과 일치하는가 일치 깨짐/누락 → 파싱/OCR 실패
최근 수정된 원본이 답변에 반영되는가 반영됨 지연 → Incremental Indexing 주기 문제
권한 없는 사용자에게 필터링이 적용되는가 차단됨 노출됨 → 벡터 계층 권한 강제 미흡
정답 근거 chunk가 top-k 안에 있는가 있음 없음 → 검색/청킹/임베딩 계층 확인
답변이 인용한 근거와 실제로 일치하는가 일치 불일치 → 생성/context 구성 계층 확인
자기 점검 Rubric

내 답변 수준 확인하기

미흡 "파일들을 텍스트로 바꿔서 벡터 DB에 넣고 검색하면 됩니다" 정도로만 설명하고, 포맷별 차이나 권한/신선도 문제를 언급하지 못함
주니어 수집→파싱→청킹→임베딩→검색→생성의 흐름을 순서대로 나열할 수 있음. 포맷마다 다른 파서가 필요하다는 것은 알지만 구체적인 실패 양상은 막연함
중급 포맷별 파싱 차이와 권한 메타데이터의 필요성을 설명할 수 있고, query-time RAG와 사전 큐레이션(위키) 방식이 다르다는 개념을 알고 있음
시니어 증상을 수집/파싱/큐레이션/검색/생성/권한 계층으로 즉시 분류하고, 각 계층 안에서 원인을 정량 지표로 좁혀나가며, query-time RAG와 LLM Wiki를 조회 빈도 기준으로 조합하는 하이브리드 설계를 근거와 함께 제시한다. "조용한 파싱 실패"와 "권한 동기화 지연"처럼 눈에 잘 안 띄는 리스크까지 선제적으로 짚어낸다
자가 점검 질문:
1. PDF/DOCX/PPTX/XLSX/HTML/코드 각각에서 구조 정보를 잃지 않으려면 무엇이 필요한지 설명할 수 있는가?
2. Query-time RAG와 LLM Wiki를 언제, 어떤 기준으로 조합할지 설명할 수 있는가?
3. 답변 품질/보안 문제가 발생했을 때 어느 계층부터 점검할지 순서를 댈 수 있는가?
4. "메타데이터/권한은 나중에 붙여도 된다"는 말이 왜 위험한지 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • 파일 형식 RAG의 본질은 "포맷별로 구조 정보를 잃지 않고 텍스트를 추출하는 것"이다. 파싱 단계에서 잃은 정보는 이후 단계에서 복구할 수 없다.
  • Query-time RAG와 LLM Wiki는 경쟁이 아니라 계층이다. 자주 조회되는 안정적 지식은 위키로 미리 정리하고, 드물거나 최신성이 중요한 정보는 RAG로 그때그때 가져온다.
  • 전체 아키텍처는 수집→정제→(선택)큐레이션→색인→검색/생성→평가/운영 6단계로 나뉘며, 장애 진단은 이 계층 순서로 좁혀나간다.
  • 권한과 provenance는 처음부터 데이터 모델의 1급 요소다. 나중에 붙이면 노출 창(window)이 생긴다.
  • 파싱 실패는 대부분 "조용한 실패"다. 에러 로그가 아니라 품질 지표(구조 일치율, OCR confidence 등)로 측정해야 한다.
  • 진단과 설계 모두 감이 아니라 계층별로 분리된 정량 지표를 기반으로 재현 가능하게 만들어야 한다.
