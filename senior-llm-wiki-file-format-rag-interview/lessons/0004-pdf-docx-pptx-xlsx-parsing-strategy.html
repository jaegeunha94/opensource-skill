PDF/DOCX/PPTX/XLSX 파싱 전략

"구조가 없는" PDF와 "구조가 이미 있는" DOCX/PPTX/XLSX를 다르게 다뤄야 하는 이유, 레이아웃 분석·VLM 직접 파싱·하이브리드 파이프라인의 2026년 현재 트레이드오프, 그리고 벤더 벤치마크를 그대로 믿으면 안 되는 이유
면접 질문
Q1 — 주력 질문 "PDF, DOCX, PPTX, XLSX처럼 내부 구조가 완전히 다른 포맷을 RAG 파이프라인에 넣어야 합니다. 각 포맷을 어떻게 파싱하시겠습니까? 도구는 어떤 기준으로 선택하시겠습니까?"
Q2 — 변형 "요즘 PDF 페이지를 통째로 이미지로 만들어서 멀티모달 LLM(비전 모델)에 바로 넣는 방식이 유행인데, 그냥 이 방식만 쓰면 안 되나요? 전통적인 레이아웃 분석 파서가 아직도 필요한 이유가 있습니까?"
Q3 — 변형 "특정 파싱 벤더가 '우리가 벤치마크에서 1위'라고 주장하는 자료를 보여준다면, 그 숫자를 어느 정도까지 신뢰하시겠습니까?"
전제 개념

왜 PDF와 DOCX는 "같은 문서 파싱"이 아닌가

많은 지원자가 "문서 파싱"을 하나의 문제로 뭉뚱그려 생각한다. 하지만 PDF와 DOCX/PPTX/XLSX는 근본적으로 다른 종류의 문제다. 이 차이를 먼저 이해하지 못하면 이후의 도구 선택도, VLM 트렌드에 대한 판단도 표면적인 암기에 그친다.

PDF: "문서"가 아니라 "인쇄 결과물"이다

PDF는 원래 화면이나 프린터에 어떻게 그릴지를 기록하는 포맷이다. "이 글자를 (120, 340) 좌표에 이 폰트로 그려라"는 명령의 나열에 가깝고, "이것은 제목이다", "이것은 표의 3번째 열이다" 같은 의미 구조(semantic structure)는 원래 PDF 안에 존재하지 않는다. 그래서 PDF 파싱은 "이미 있는 구조를 읽어내는" 문제가 아니라, 흩어진 텍스트 조각과 좌표로부터 문단·제목·표·목록 같은 구조를 역으로 추론(reconstruct)하는 문제다. 같은 표라도 셀 경계선이 그림으로만 그려져 있거나, 병합된 셀이 실제로는 여러 개의 겹친 텍스트 상자일 수 있다.

흔한 착각: "PDF에서 텍스트를 뽑는 건 pip install pypdf 한 줄이면 되는 거 아닌가요?" — 텍스트를 추출하는 것과, 그 텍스트가 어떤 구조(제목/본문/표/각주)에 속하는지 복원하는 것은 완전히 다른 난이도의 문제다. RAG 품질에 영향을 주는 것은 후자다.

DOCX/PPTX/XLSX: 구조가 "이미" 파일 안에 있다

반면 DOCX/PPTX/XLSX는 내부적으로 ZIP 컨테이너 안에 XML로 구조를 명시적으로 기록한다(Day 3에서 다룬 OOXML 컨테이너 포맷). "이 문단은 Heading 1 스타일이다", "이 셀은 (2,3) 위치이고 병합 범위는 여기까지다" 같은 정보가 이미 태그로 존재한다. 그래서 DOCX/PPTX/XLSX 파싱의 핵심 난이도는 "구조를 추론하는 것"이 아니라 "이미 있는 구조를 놓치지 않고 충실히 읽어내는 것"이다 — 병합 셀, 트랙 변경(추적된 수정 내용), 중첩된 표, 수식(formula) vs 계산된 값(cached value)의 차이, 임베디드 객체(차트, 다른 문서) 같은 엣지 케이스가 문제가 된다.

구분 PDF DOCX / PPTX / XLSX
구조의 존재 여부 없음 — 좌표/폰트 나열, 구조는 추론해야 함 있음 — XML 태그로 명시
핵심 난이도 레이아웃 분석(layout analysis), reading order 복원 엣지 케이스 충실히 반영(병합 셀, 수식, 임베디드 객체)
스캔본 문제 있음 — 이미지로만 존재하면 OCR 필요 거의 없음 — 텍스트 레이어가 항상 존재
대표 실패 유형 다단 레이아웃에서 읽는 순서가 뒤섞임, 표 셀 경계 오인식 병합 셀을 단일 셀로 오인, 수식 대신 캐시된 값만 읽음

레이아웃 분석의 3가지 접근: Rule-based, ML 레이아웃 모델, VLM 직접 파싱

1
Rule-based (규칙 기반) — pdfminer, PyMuPDF 같은 라이브러리가 폰트 크기·좌표·간격 같은 휴리스틱 규칙으로 문단/제목을 추정한다. 빠르고 저렴하지만 복잡한 다단 레이아웃이나 표에서 쉽게 무너진다.
2
ML 기반 레이아웃 탐지 모델 — Docling, Unstructured의 hi_res 전략처럼, 페이지 이미지에 대해 객체 탐지(object detection) 모델을 돌려 "이 영역은 제목", "이 영역은 표"라고 레이블링한 다음 그 영역만 텍스트 추출한다. Rule-based보다 정확하지만 GPU/모델 서빙 비용이 든다.
3
VLM 직접 파싱(Vision-Language Model) — 페이지 전체를 이미지로 바꿔 멀티모달 LLM에 "이 페이지를 Markdown으로 옮겨줘"라고 직접 시킨다. 레이아웃 탐지·OCR·표 인식을 모델 하나가 한 번에 처리한다는 점이 매력적이지만, 모델이 실제로 없는 내용을 "그럴듯하게 지어내는" hallucination 위험이 구조가 복잡한 표에서 특히 크다는 것이 2026년 현재까지도 반복적으로 보고되는 문제다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • "파싱은 다 똑같다"는 착각이 있는지 — PDF의 구조 추론 문제와 DOCX/PPTX/XLSX의 구조 보존 문제를 구분하지 못하면, 포맷별로 다른 전략이 필요하다는 것 자체를 이해하지 못한 것으로 본다.
  • 최신 트렌드를 맹신하지 않고 트레이드오프로 판단하는지 — "요즘은 VLM에 이미지만 넣으면 다 된다"는 마케팅 문구를 그대로 반복하는지, 아니면 표/수치 데이터에서 hallucination이 실제로 보고되고 있다는 것까지 알고 하이브리드 전략을 제시하는지를 본다.
  • 벤더 벤치마크를 비판적으로 읽을 수 있는지 — 파싱 도구 시장은 벤더마다 자사에 유리하게 설계한 벤치마크를 내놓는다. 숫자를 그대로 인용만 하는지, 아니면 "누가 측정했는가", "내 도메인 문서로도 같은 결과가 나올까"를 따지는지가 시니어 판단력의 핵심 신호다.
  • 비용/정확도/지연/자체호스팅 여부를 함께 고려하는 실무 감각이 있는지 — "정확도가 제일 높은 도구"만 고르는 게 아니라, 문서 규모·비용 구조·데이터 residency 요구사항까지 함께 판단하는지를 본다.
실무 연결

2026년 현재 도구 지형과 벤치마크

도구 지형: 레이어별로 나눠서 이해하기

2026년 기준 PDF/오피스 문서 파싱 도구는 크게 세 계층으로 나뉜다.

계층 대표 도구 특징
오픈소스 · 자체호스팅 Docling(IBM, MIT 라이선스), Marker(Datalab), MinerU(OpenDataLab) 무료·데이터 residency 확보 가능하지만 GPU 인프라와 운영 부담을 스스로 짊어져야 함
관리형 문서 파싱 API LlamaParse(LlamaIndex), Reducto, Unstructured(Serverless/Ingest) 페이지당 과금, 표/차트 특화 기능(Agentic 모드 등) 제공, 운영 부담을 벤더에 위임
범용 클라우드 OCR AWS Textract, Google Document AI, Azure Document Intelligence 클라우드 생태계 통합이 쉽지만, 표 추출 정확도는 문서 벤치마크 전용 도구보다 낮게 나오는 경우가 많음

벤치마크 숫자 — 그대로 믿지 말고 "누가 쟀는가"부터 봐라

2026년 문서 파싱 벤치마크 생태계에는 참고할 만한 공개 벤치마크가 여럿 있지만, 숫자를 그대로 인용하면 안 되는 이유가 명확하다.

OmniDocBench (CVPR 2025, arXiv:2412.07626) — 학술 표준 벤치마크. 2026년 4월 v1.6 기준 상위권은 MinerU2.5-Pro(1.2B) 95.75, GLM-OCR(0.9B) 95.22, PaddleOCR-VL-1.5(0.9B) 94.93다. 흥미로운 점은 GPT-4o·Gemini 3 Pro 같은 범용 프론티어 모델(86~93점대)이 오히려 훨씬 작은 문서 전용 특화 모델보다 낮은 점수를 받는다는 것이다 — "제일 큰 모델이 제일 잘한다"는 직관이 문서 파싱에서는 틀릴 수 있다.
ParseBench (LlamaIndex, 2026년 6월 공개) — 엔터프라이즈 문서 2,078페이지로 표 정확도·차트 추출·근거 충실도(faithfulness)를 측정. LlamaParse Agentic가 84.88점으로 1위를 차지했지만, 이 벤치마크 자체를 LlamaIndex가 설계하고 자사 제품을 채점했다는 점을 반드시 함께 언급해야 한다. 데이터셋은 공개돼 있어 재현 가능하지만, 2026년 7월 기준 독립된 제3자가 재검증한 사례는 아직 없다.
실제로 발생한 벤더 간 숫자 왜곡 사례: Mistral이 공개한 olmOCR-Bench 비교 차트에서 경쟁 모델(Chandra)의 점수가 실제보다 낮게 잘못 표기된 사실이 해당 모델 개발자에 의해 지적된 바 있다. 또한 Mistral OCR API는 olmOCR-Bench의 표(table) 세부 항목에서 29.3점에 그쳐, 경쟁 모델(Chandra 50.4점)과 큰 격차를 보였다 — 벤더가 강조하는 "종합 1위" 숫자 뒤에 표처럼 구체적인 세부 항목에서는 취약점이 숨어 있을 수 있다.

독립적으로 수행된 비교 실험(문서 800건 이상, 프론티어 모델 7종 테스트)에서는 도메인에 따라 파서 정확도가 55점 이상 차이 났고, 텍스트 추출 정확도와 "구조 복원" 정확도는 서로 다른 지표라서 어떤 파서는 텍스트는 75% 맞히면서도 문서 구조(표/목록/계층)는 13%만 복원하기도 했다. 즉, 벤치마크 하나의 종합 점수만으로 도구를 고르면 안 되고, 내 도메인의 실제 문서 샘플로 표/차트/스캔본별 정확도를 따로 측정해야 한다는 것이 2026년 실무의 공통된 결론이다.

VLM 직접 파싱 — 유행하지만 hallucination은 여전히 미해결 문제다

Anthropic 공식 문서 자체도 흥미로운 신호를 준다. Claude의 PDF 처리는 "페이지 이미지만 보고 파싱"하는 순수 vision 방식이 아니라, 각 페이지를 이미지로 변환하는 동시에 텍스트 레이어도 함께 추출해 나란히 모델에 제공하는 하이브리드 구조다. 문서를 실제로 만드는 모델 제공사조차 "이미지만 믿고 텍스트 레이어를 버리지 않는" 이유는, 순수 VLM 파싱의 hallucination 위험이 실제로 존재하기 때문이다.

사례 내용
독립 벤치마크(Reducto RD-FormsBench) 밀도 높은 재무 표·손글씨 의료 양식에서 특정 VLM-OCR이 약 45% 정확도에 그쳐, 같은 세트에서 약 80%를 기록한 모델과 큰 격차 발생
LlamaIndex 자체 분석("OmniDocBench is Saturated") 2026년 최상위 파싱 방법도 근거 충실도(faithfulness)가 약 90% 수준 — 바꿔 말하면 SOTA 방법조차 약 10페이지 중 1페이지꼴로 의미 있는 누락/hallucination이 발생
Reducto 자체 공개 수치 자사의 "Agentic OCR"(CV+VLM+다중 패스 결합) 파이프라인도 토큰 recall은 1위였지만, "추가된(존재하지 않는) 토큰" 비율에서는 3번째로 높아 hallucination이 하이브리드 방식에서도 완전히 사라지지 않았음을 스스로 인정
2026년 실무 컨센서스: 순수 VLM 단일 패스보다 구조 기반 1차 파싱(레이아웃 분석 또는 텍스트 레이어 추출) + VLM을 보정/폴백 용도로 결합하는 하이브리드 파이프라인이 프로덕션 표준이다. VLM 단독 파싱은 저위험·탐색적 용도이거나, 레이아웃이 극도로 불규칙해 규칙 기반 접근이 아예 통하지 않는 예외적 케이스에 한정해서 쓰는 경우가 많다.

DOCX/PPTX/XLSX 전용 라이브러리의 유지보수 리스크

의외의 실무 함정이 있다. 오랫동안 표준으로 쓰여온 python-docx (PyPI 최신 1.2.0, 2025-06 릴리스 이후 정체), python-pptx (PyPI 최신 1.0.2, 2024-08 릴리스 이후 정체)는 2026년 기준 1년 이상 새 릴리스가 없어 여러 의존성 위험 추적 도구에서 "사실상 유지보수 중단"으로 지적되고 있다. openpyxl도 PyPI 기준으로는 비슷하게 오래됐지만, 실제 원본 저장소는 GitHub가 아니라 Heptapod (Mercurial 기반)이라 GitHub 활동만으로 유지보수 상태를 판단하면 잘못된 결론에 이를 수 있다 — "저장소를 어디서 확인했는가"까지 검증하는 습관이 시니어 수준의 조사력이다. 실무에서는 이 개별 라이브러리를 직접 쓰기보다, Docling·Marker·MinerU처럼 DOCX/PPTX/XLSX를 함께 지원하는 통합 파서로 옮겨가는 흐름이 뚜렷하다.

실제 GitHub 이슈로 확인되는 구체적 함정

도구 증상 상태
openpyxl (XLSX) data_only=True로 열었는데 수식 셀 값이 None으로 나옴 — 프로그램으로 만들거나 수정한 파일을 Excel이 아직 재계산하지 않았기 때문. 병합 셀의 좌상단이 아닌 셀은 MergedCell 객체가 되어 값을 쓰려 하면 AttributeError 발생 라이브러리의 설계상 한계 — openpyxl은 수식을 직접 계산하지 않음 (공식 튜토리얼에 명시)
python-docx (DOCX) 추적된 변경 내용(track changes, <w:ins>/<w:del>)의 승인/거부 로직을 지원하지 않아, 삭제 예정 텍스트까지 그대로 추출될 수 있음 2016년부터 열려 있는 미해결 이슈 (GitHub #340)
Unstructured (DOCX) 병합 셀이 있는 표에서 셀 텍스트가 병합된 셀 개수만큼 중복 출력되는 것으로 확인됨 공개 이슈로 리포트됨 (GitHub #2106), 부분 수정 PR 존재
Unstructured (XLSX) 캐시된 값이 없는 수식 셀이 있으면 해당 영역이 통째로 텍스트 추출에서 누락될 수 있음 — 우회책은 Excel로 다시 저장해 캐시 값을 강제로 채우는 것 공개 이슈로 리포트되고 종료됨 (GitHub #2485)
공통 패턴: 오피스 포맷의 "구조가 이미 있다"는 장점은, 그 구조를 라이브러리가 실제로 얼마나 충실하게 반영하는가에 의해 실제 가치가 결정된다. 수식-값 관계, 병합 셀, 추적된 변경 내용처럼 "사람이 보기엔 당연한" 정보도 파서 구현에 따라 유실되거나 왜곡될 수 있다는 것을 실제 이슈 사례로 알고 있으면 답변의 신뢰도가 올라간다.

최근 deprecation/migration — 알아두면 감점을 피할 수 있는 것들

  • PyPDF2 → pypdf: PyPDF2는 더 이상 권장되지 않고 pypdf 네임스페이스로 통합됐다. PdfFileReader →PdfReader 같은 API 이름도 스네이크케이스로 정리됐다.
  • langchain-community 저장소가 2026-05-26 아카이브(읽기 전용) 처리됐다 — LangChain은 문서 로더 등 커뮤니티 통합을 개별 파트너 패키지(langchain-openai 등) 구조로 이전 중이다. 오래된 from langchain.document_loaders import ... import 경로를 그대로 쓰고 있다면 마이그레이션이 필요하다.
  • LlamaParse V2 tier 개편: 기존 fast/balanced/premium 명명 체계가 Fast/Cost-Effective/Agentic/Agentic Plus 4단계로 재편됐다. 오래된 모드 이름을 그대로 언급하면 최신성 부족으로 비칠 수 있다.
  • Docling은 v2.70.0부터 Python 3.9 지원을 중단했고, 2026년 7월 기준 v2.110.0까지 거의 매일 단위로 릴리스되며 OCR/VLM 백엔드(NVIDIA Nemotron OCR, vLLM 기반 모델 등)를 계속 추가하고 있다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 먼저 PDF와 DOCX/PPTX/XLSX를 다른 문제로 봅니다. PDF는 좌표와 폰트 나열일 뿐 문단·표·제목 같은 구조가 원래 없어서, 파싱은 그 구조를 역으로 추론하는 문제입니다. 반면 DOCX/PPTX/XLSX는 ZIP 안에 XML로 구조가 이미 있어서, 병합 셀이나 수식처럼 이미 있는 정보를 놓치지 않고 충실히 읽어내는 게 핵심입니다.

PDF 레이아웃 분석은 크게 세 갈래로 나눠서 생각합니다. 규칙 기반 파서는 빠르고 저렴하지만 다단 레이아웃이나 복잡한 표에서 쉽게 무너집니다. ML 기반 레이아웃 탐지 모델은 이미지에 대해 객체 탐지를 돌려 제목·표 영역을 인식하는 방식으로, 정확도는 더 높지만 GPU 서빙 비용이 듭니다. 최근에는 페이지 이미지를 통째로 멀티모달 LLM에 넣는 방식도 유행하지만, 이건 완전한 해결책이 아니라고 봅니다. LlamaIndex 자체 분석에서도 2026년 SOTA 방법조차 근거 충실도가 약 90% 수준이라, 여전히 페이지 10개 중 1개꼴로 의미 있는 hallucination이나 누락이 발생한다고 나와 있습니다. 특히 표처럼 구조가 복잡한 데이터에서 이 위험이 두드러집니다. 그래서 저는 순수 VLM 단일 패스보다, 구조 기반 1차 파싱과 VLM 보정을 결합한 하이브리드 파이프라인을 기본으로 둡니다. 실제로 Claude의 공식 PDF 처리 방식도 이미지만 쓰는 게 아니라 텍스트 레이어를 함께 추출해서 나란히 제공하는 하이브리드 구조입니다.

도구 선택에서는 공개 벤치마크를 참고하되 맹신하지 않습니다. OmniDocBench 같은 학술 벤치마크로 전체적인 경향은 파악하지만, 벤더가 자체 공개한 벤치마크는 누가 설계하고 채점했는지부터 확인합니다. 실제로 한 벤더의 비교 차트에서 경쟁 모델 점수가 잘못 표기된 사례도 있었고, 종합 점수가 높아도 표 항목에서만 유독 낮은 점수를 받는 경우도 있었습니다. 그래서 최종 결정 전에는 반드시 저희 도메인의 실제 문서 샘플로 표·차트·스캔본별 정확도를 직접 측정합니다.

마지막으로 도구 선택은 정확도 하나만 보지 않습니다. 대량 배치라면 자체호스팅 오픈소스(Docling, Marker 등)로 비용을 낮추고, 소량이지만 정확도가 중요한 고가치 문서라면 관리형 API의 프리미엄/Agentic 모드를 씁니다. DOCX/PPTX 같은 오피스 포맷은 python-docx/python-pptx 같은 개별 라이브러리가 유지보수가 정체된 상태라, 최근에는 Docling처럼 여러 포맷을 함께 지원하는 통합 파서로 옮겨가는 추세를 반영합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "VLM이 표에서 hallucination을 일으킨다는 걸 어떻게 검증하고, 프로덕션에서는 어떻게 막으시겠습니까?"

약한 답변: "최신 모델은 성능이 좋아서 괜찮을 것 같습니다."

강한 답변: "먼저 파싱 결과에 대해 자체 검증 규칙을 겁니다 — 예를 들어 표라면 각 행의 합계가 표시된 총계와 일치하는지 같은 정합성 체크를 자동화합니다. 다음으로 파싱 신뢰도 점수(confidence)가 낮거나 검증 규칙을 통과하지 못한 페이지는 사람이 검토하는 큐로 보냅니다. 그리고 애초에 표처럼 hallucination 위험이 높은 영역은 순수 VLM 대신 표 전용 구조 인식 모델(table structure recognition)을 우선 적용하고, VLM은 셀 내용을 보정하는 역할로만 씁니다."

F2. "벤더가 '올해 벤치마크 1위'라고 홍보하는 파싱 도구를 도입하자는 요청을 받으면 어떻게 대응하시겠습니까?"

약한 답변: "1위라니까 검증 없이 바로 도입합니다."

강한 답변: "먼저 그 벤치마크를 누가 설계하고 채점했는지 확인합니다 — 벤더 자체 벤치마크라면 자사에 유리하게 설계됐을 가능성을 염두에 둡니다. 다음으로 종합 점수 뒤에 숨은 세부 항목(특히 표/차트처럼 어려운 항목)의 점수를 따로 확인합니다. 마지막으로 반드시 저희 도메인 문서 100~200건 정도의 골든셋을 만들어 실제로 비교 테스트를 돌린 뒤에 결정합니다. 공개 벤치마크 순위와 저희 도메인에서의 실제 순위가 다를 수 있다는 것을 전제로 접근합니다."

F3. "DOCX 파일에 수식(예: Excel의 =SUM(A1:A10))이 있을 때, 어떤 값을 인덱싱하시겠습니까?"

강한 답변: "수식 자체가 아니라 캐시된 계산 값을 우선 인덱싱합니다. 사용자가 질의할 때 필요한 건 '수식이 어떻게 생겼는가'가 아니라 '실제 값이 무엇인가'이기 때문입니다. 다만 수식 자체를 물어보는 질의(예: '이 시트의 계산 로직을 설명해줘')를 지원해야 한다면, 수식 문자열도 별도 메타데이터로 함께 보존해서 필요할 때 참조할 수 있게 합니다."

F4. "PPTX 슬라이드처럼 텍스트가 거의 없고 다이어그램·이미지 위주인 문서는 어떻게 다루시겠습니까?"

강한 답변: "슬라이드는 텍스트 레이어(제목, 불릿)만 추출하면 다이어그램에 담긴 핵심 정보를 놓치기 쉽습니다. 그래서 슬라이드 이미지 자체를 멀티모달 모델에 넣어 다이어그램 내용을 설명하는 캡션을 생성하고, 이 캡션을 슬라이드의 텍스트 레이어와 함께 하나의 청크로 묶습니다. 이렇게 하면 '이 슬라이드가 뭘 말하는지'에 대한 검색이 가능해집니다."

F5. "XLSX 파일을 파싱했는데 수식이 들어간 셀 값이 전부 비어 있습니다. 원인이 뭘까요?"

약한 답변: "파싱 라이브러리 버그인 것 같습니다."

강한 답변: "openpyxl 같은 라이브러리는 수식을 직접 계산하지 않고, Excel이 마지막으로 저장할 때 캐싱해둔 계산 결과만 읽습니다. 만약 파일이 프로그램(예: pandas, 다른 스크립트)으로 생성되거나 수정된 뒤 Excel에서 한 번도 열려서 재계산되지 않았다면, 캐시된 값 자체가 없어서 None이 반환됩니다. 그래서 데이터 파이프라인에서 XLSX를 생성할 때는 반드시 Excel이나 자동화 도구(xlwings 등)로 한 번 열어 재계산 값을 채워 저장하는 단계를 넣거나, 애초에 계산 결과를 별도 값 필드로 함께 기록하도록 요구합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • PDF(구조 추론)와 DOCX/PPTX/XLSX(구조 보존)의 문제 본질 차이를 먼저 짚는다
  • VLM 직접 파싱 트렌드를 알면서도, 표/수치 데이터 hallucination 위험을 구체적 수치나 사례로 근거를 든다
  • 벤더 벤치마크를 설계·채점 주체 기준으로 비판적으로 해석하고, 자체 도메인 검증을 강조한다
  • 도구 선택을 정확도 하나가 아니라 비용/자체호스팅/ 지연을 함께 고려한 다차원 판단으로 설명한다
  • 수식 vs 캐시된 값, 병합 셀 같은 구체적 엣지 케이스를 언급한다

약한 답변의 특징

  • "파싱 라이브러리로 텍스트만 뽑으면 됩니다"처럼 구조 복원 문제를 인식하지 못한다
  • "요즘은 VLM에 이미지 넣으면 다 됩니다"라고 트렌드를 검증 없이 그대로 반복한다
  • 벤더 벤치마크 숫자를 출처 검증 없이 그대로 인용한다
  • "정확도가 제일 높은 도구를 씁니다"처럼 비용/운영 측면을 고려하지 않는다
  • 포맷별 엣지 케이스(병합 셀, 수식 등)를 구체적으로 말하지 못한다
핵심 차이: 강한 답변은 "파싱 도구"를 유행이나 마케팅 문구로 고르지 않고, 문제의 본질(구조 추론 vs 구조 보존)과 검증 가능한 트레이드오프로 판단한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "PDF 텍스트 추출과 구조 복원은 같은 문제다"

사실: 텍스트 추출은 pypdf 한 줄로도 되지만, 그 텍스트가 어떤 제목/표/문단에 속하는지 구조를 복원하는 것은 완전히 다른 난이도다. 텍스트 추출 정확도가 높아도 구조 복원 정확도는 크게 낮을 수 있다(한 비교 실험에서 텍스트 75% vs 구조 13% 정확도 격차가 보고됨).

오해 2: "가장 큰(똑똑한) 모델이 문서 파싱도 제일 잘한다"

사실: OmniDocBench 2026년 상위권은 오히려 10억 개 파라미터 안팎의 문서 전용 특화 모델이 차지했고, 훨씬 큰 범용 프론티어 모델은 그보다 낮은 점수를 받았다. 범용 능력과 문서 구조 인식 능력은 비례하지 않는다.

오해 3: "VLM에 페이지 이미지를 넣으면 레이아웃 분석·OCR·표 인식이 다 해결된다"

사실: 2026년 SOTA 파싱 방법조차 근거 충실도가 약 90% 수준으로 보고되며, 표처럼 구조가 복잡한 데이터에서 hallucination 위험이 특히 크다. 그래서 문서를 실제로 만드는 모델 제공사조차 순수 이미지 방식이 아니라 텍스트 레이어를 함께 제공하는 하이브리드 구조를 쓴다.

오해 4: "벤더가 공개한 벤치마크 1위 숫자는 그대로 믿어도 된다"

사실: 벤치마크를 설계하고 채점하는 주체가 벤더 자신인 경우가 흔하고, 실제로 경쟁 모델 점수가 잘못 표기된 사례도 보고됐다. 종합 점수가 높아도 특정 세부 항목(표 등)에서는 크게 낮을 수 있으므로 항상 세부 항목과 측정 주체를 확인해야 한다.

문서 파싱 도구 선택 전 확인 체크리스트

핵심 요약

오늘 반드시 기억할 것

  • PDF는 구조를 추론해야 하고, DOCX/PPTX/XLSX는 이미 있는 구조를 충실히 보존해야 한다 — 근본적으로 다른 문제다.
  • 레이아웃 분석은 Rule-based → ML 레이아웃 모델 → VLM 직접 파싱 순으로 정확도와 비용이 함께 올라가는 스펙트럼이다.
  • VLM 직접 파싱은 유행하지만 2026년 현재까지도 표/수치 데이터에서 hallucination이 반복적으로 보고되는 미해결 문제다. 프로덕션 컨센서스는 구조 기반 파싱 + VLM 보정의 하이브리드다.
  • 벤더 벤치마크는 설계·채점 주체를 확인하고 도메인 골든셋으로 자체 검증해야 한다. 종합 순위와 세부 항목(특히 표) 순위가 다를 수 있다.
  • 도구 선택은 정확도만이 아니라 비용, 자체호스팅 여부(데이터 residency), 지연, 라이브러리 유지보수 상태까지 함께 고려하는 다차원 판단이다.
  • 오늘 다룬 표/hallucination 문제는 Day 6(OCR과 Table/Image Extraction)에서 더 깊이 다루고, 벤치마크 회의주의는 Day 14(평가 지표)의 기반이 된다.
