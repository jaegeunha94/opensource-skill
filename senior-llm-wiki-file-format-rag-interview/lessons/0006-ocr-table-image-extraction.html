OCR과 Table/Image Extraction

Day 4에서 "레이아웃 분석"을 큰 틀에서 다뤘다면, 오늘은 그 안에서 가장 실패가 잦은 세 가지 하위 문제 — 스캔 문서의 문자 인식(OCR), 병합 셀이 뒤섞인 표의 구조 복원(TSR), 차트/이미지를 검색 가능하게 만드는 방법 — 을 파고들고, "OCR 정확도가 높으면 RAG 품질도 높다"는 흔한 가정이 왜 틀렸는지를 2026년 최신 벤치마크로 검증한다
면접 질문
Q1 — 주력 질문 "스캔된 계약서나 팩스로 받은 문서를 RAG에 넣어야 합니다. OCR 정확도가 99%라고 벤더가 광고하는데, 이 숫자만 보고 채택 여부를 결정해도 됩니까? OCR 정확도와 RAG 답변 품질은 어떤 관계입니까?"
Q2 — 변형 "재무제표처럼 병합 셀이 많고 여러 페이지에 걸쳐 이어지는 복잡한 표를 어떻게 파싱하시겠습니까? 표를 그냥 텍스트로 뽑으면 어떤 문제가 생깁니까?"
Q3 — 변형 "문서 안에 있는 차트나 다이어그램, 스크린샷은 어떻게 검색 가능하게 만드시겠습니까? 이미지를 텍스트로 설명해서 임베딩하는 것과, 이미지 자체를 임베딩하는 것 중 어떤 걸 선택하시겠습니까?"
전제 개념

OCR·레이아웃 인식·표 구조 인식은 서로 다른 세 가지 문제다

"스캔 문서 파싱"이라는 말을 들으면 대부분 OCR 하나를 떠올리지만, 실제로는 세 개의 독립적인 문제가 순서대로(또는 병렬로) 풀려야 한다. 이 구분을 못 하면, "OCR이 좋으니까 표도 잘 나오겠지" 같은 잘못된 기대를 하게 된다.

1
레이아웃 분석(Layout Analysis) — 페이지 이미지 안에서 "여기는 제목, 여기는 본문 단락, 여기는 표, 여기는 그림"처럼 영역을 구분하고 읽는 순서(reading order)를 결정하는 문제다. 2단 레이아웃 논문에서 왼쪽 컬럼을 다 읽고 오른쪽으로 넘어가야 하는데, 순서를 잘못 판단하면 문장 중간에 엉뚱한 단이 끼어든다.
2
문자 인식(OCR, Optical Character Recognition) — 레이아웃이 잡힌 영역 안에서 실제 글자를 읽어내는 문제다. 이미지 픽셀을 유니코드 문자열로 바꾸는 작업 자체다.
3
표 구조 인식(TSR, Table Structure Recognition) — 표 영역이라는 것을 안 다음, 몇 행 몇 열이고 어느 셀이 병합되어 있는지를 grid(격자) 구조로 복원하는 문제다. OCR로 글자를 다 읽어도 이 grid 구조를 모르면 "숫자들이 흩어져 있을 뿐" 표가 아니다.
흔한 착각: "OCR 정확도가 99%면 문서 파싱은 다 끝난 거 아닌가요?" — 글자 하나하나를 정확히 읽어도(OCR), 그 글자가 표의 몇 행 몇 열에 있었는지를 잘못 복원하면(TSR 실패) RAG는 "매출이 120억"과 "매출이 12억"을 구분하지 못하는 답을 낼 수 있다. OCR 정확도는 세 문제 중 하나의 지표일 뿐이다.

왜 OCR 정확도가 높아도 RAG 품질은 낮을 수 있는가

직관적으로는 "글자를 99% 정확히 읽으면 문서 내용의 99%를 이해한 것"이라 생각하기 쉽지만, 이는 두 가지 이유로 틀렸다.

첫째, 오류는 무작위로 분포하지 않는다. OCR 오류는 특히 숫자, 단위, 고유명사, 표 안의 값처럼 정보 밀도가 가장 높은 위치에 몰리는 경향이 있다. 일반 문장의 조사 하나를 틀리는 것과, 표 안의 "12,000"을 "72,000"으로 잘못 읽는 것은 텍스트 유사도로는 둘 다 "1글자 차이"지만 실제 피해는 전혀 다르다.

둘째, 임베딩과 LLM은 서로 다른 방식으로 오류에 반응한다. 임베딩 모델은 문맥 전체의 의미를 벡터화하기 때문에, 문서 대부분이 깨끗하면 몇 글자가 틀려도 검색(retrieval)까지는 어찌어찌 성공할 수 있다. 하지만 생성(generation) 단계의 LLM은 검색된 청크의 정확한 숫자·단위를 그대로 인용하려 하기 때문에, 바로 그 틀린 글자가 답변에 그대로 반영된다. 즉 OCR 오류가 검색 단계에서는 숨어 있다가 생성 단계에서 터지는 경우가 많다.

병합 셀 표가 어려운 이유 — "읽는 것"과 "구조를 복원하는 것"의 차이

재무제표를 예로 들어보자. "2025년 매출"이라는 헤더 셀이 1월~12월 12개 컬럼에 걸쳐 가로로 병합되어 있고, "아시아 법인"이라는 셀이 4개 분기 행에 걸쳐 세로로 병합되어 있다고 하자. 사람은 표를 보는 순간 이 구조를 즉시 이해하지만, 모델 입장에서는 픽셀만 봐서는 "이 셀이 몇 개의 원래 셀이 합쳐진 것인지"를 추론해야 하는 기하학적 구조 복원 문제다. 이걸 틀리면, 표를 그대로 문자열로 나열했을 때 "1월 매출"이라는 값이 실제로는 "1분기 매출"이었던 것처럼 컬럼이 밀려서 나올 수 있다 — Day 4에서 다룬 DOCX 병합 셀 텍스트 중복 문제(Unstructured Issue #2106)와 본질이 같은 문제가, 스캔 이미지에서는 훨씬 어려운 형태로 다시 나타나는 셈이다.

이미지·차트는 "본문"이 아니라 "별도의 정보 채널"이다

텍스트로 서술된 정보와 차트로 시각화된 정보는 같은 내용을 담고 있어도 표현 방식이 근본적으로 다르다. "3분기 매출이 전분기 대비 12% 하락했다"는 문장은 텍스트 그대로 임베딩할 수 있지만, 막대그래프로만 표현된 같은 정보는 텍스트가 전혀 없다. 이미지를 무시하면 지식이 유실되고, 그렇다고 모든 이미지를 텍스트로 옮기려 하면 "이미지를 텍스트로 바꾸는 과정에서 무엇을 잃는가"라는 새로운 트레이드오프가 생긴다. 이게 오늘 다룰 세 번째 축이다.

계층 푸는 문제 대표 모델/도구 실패 시 증상
레이아웃 분석 영역 구분 + 읽는 순서 DocLayout-YOLO, Heron(Docling), dots.ocr 2단 논문이 뒤섞여 읽힘, 헤더/푸터가 본문에 섞임
OCR 픽셀 → 문자열 Tesseract, PaddleOCR, olmOCR-2, Chandra, Mistral OCR 숫자/단위/고유명사 오인식
표 구조 인식(TSR) grid·병합 셀 복원 TableFormer(Docling), Table Transformer(TATR) 컬럼/행 밀림, 병합 헤더 매핑 오류
이미지/차트 시각 정보 → 검색 가능한 표현 VLM 캡셔닝, 멀티모달 임베딩, ColPali류 차트에만 있던 수치·추세 정보 완전 유실
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • 벤치마크 숫자를 곧이곧대로 믿지 않는지 — "OCR 정확도 99%"라는 벤더 문구를 그대로 채택 근거로 쓰는지, 아니면 그 정확도가 다운스트림 RAG 정확도로 어떻게 이어지는지 검증하는지를 본다. 이건 Day 4에서 다룬 "벤더 벤치마크를 비판적으로 해석하는 능력"의 연장선이다.
  • OCR·레이아웃·TSR을 하나로 뭉뚱그리지 않는지 — 세 문제가 독립적으로 실패할 수 있다는 것을 알아야, 장애가 났을 때 "어느 단계가 문제인지" 정확히 좁혀 진단할 수 있다.
  • 병합 셀 같은 실제 사고 유형을 구체적 메커니즘으로 설명할 수 있는지 — "표를 잘 파싱합니다" 수준이 아니라, grid 복원이 왜 어려운지, 어떻게 검증하는지를 설명하는지를 본다.
  • 이미지 인덱싱 전략의 트레이드오프를 판단할 수 있는지 — 캡션 변환, 네이티브 임베딩, 재조회(2-pass) 중 무엇을 언제 쓸지, 비용·정확도·근거 추적성 사이에서 균형을 잡는 판단력을 본다.
  • 품질 게이트(confidence score, human-in-the-loop)를 설계할 수 있는지 — "OCR 결과를 무조건 신뢰" vs "저신뢰 구간만 선별적으로 검증"을 구분해 비용 대비 품질을 관리하는 실무 감각을 본다.
실무 연결

2026년 현재 OCR/TSR/이미지 추출 지형과 근거

OCR — "정확도가 높으면 RAG도 좋다"는 가정이 무너지는 4가지 국면

2026년 발표된 연구 "When Good OCR Is Not Enough: Benchmarking OCR Robustness for Retrieval-Augmented Generation" (arXiv 2605.00911)은 OCR 정확도와 최종 RAG 정확도 사이의 관계를 체계적으로 측정했다. 이 연구가 제시하는 4가지 국면(regime)은 실무에서 장애를 진단할 때 그대로 쓸 수 있는 프레임이다.

국면 OCR 품질 RAG 품질 의미
OCR Reliable 높음 높음 OCR 오류가 적어 LLM이 그대로 정확한 답을 생성
LLM Compensates 낮음 높음 LLM이 문맥으로 오탈자를 스스로 보정 — 텍스트 산문에서 흔함
Both Weak 낮음 낮음 OCR 오류가 너무 많아 LLM도 보정 한계를 넘어섬
OCR Blind Spot 높음(전체 평균) 낮음 전체 정확도는 높지만 숫자·표처럼 보정 불가능한 핵심 정보에서만 틀려 답변이 망가짐
가장 위험한 국면은 "OCR Blind Spot"이다. 이 연구에서는 문서 단위 OCR 정확도가 82.9%였는데, 그 문서를 기반으로 한 RAG 답변 정확도는 53.0%까지 떨어진 사례가 보고됐다. 전체 정확도만 보면 "그럭저럭 쓸만하다"고 오판하기 쉽지만, 틀린 17%가 하필 숫자·표· 단위처럼 LLM이 문맥으로 보정할 수 없는 부분에 집중되면 답변 품질은 훨씬 큰 폭으로 무너진다. 이게 바로 "OCR 정확도 하나의 숫자만으로 채택을 결정하면 안 되는" 핵심 이유다.

OCR 엔진 지형 — 소형 특화 VLM이 범용 프론티어 모델을 앞선다는 패턴의 반복

Day 4에서 확인한 "가장 크고 똑똑한 범용 모델이 항상 최고의 파서는 아니다"라는 패턴은 OCR 벤치마크(olmOCR-Bench, OmniDocBench)에서도 2026년 현재 그대로 반복된다.

  • Chandra(9B)가 olmOCR-Bench 평균 83.1점으로 선두, olmOCR-2(8B)가 82.3점으로 근소하게 뒤를 따른다 — 둘 다 표(table) 항목을 포함한 전 영역에서 고르게 강하다.
  • Mistral OCR은 olmOCR-Bench 종합 72.0점이지만, Day 4에서 확인했듯 표(Tables) 세부 항목만 떼어 보면 29.3점으로 Chandra의 50.4점과 큰 격차를 보인다 — 종합 점수만 보고 표 위주 문서(재무제표, 명세서)에 채택하면 실패한다.
  • dots.ocr(3B)는 OmniDocBench 88.41점을 기록하며, 레이아웃 분석과 OCR을 하나의 비전-언어 모델(단일 VLM)로 통합해 100개 이상 언어를 지원한다 — 별도의 레이아웃 검출기와 OCR 엔진을 파이프라인으로 잇던 기존 방식과 달리 단일 모델로 두 단계를 끝낸다는 점이 2026년의 아키텍처 변화다.
  • Granite-Docling-258M은 2억 5천만 파라미터의 경량 모델로, 다국어 강건성과 가벼운 배포(온디바이스·저지연 환경)에 특화되어 있으며 문서를 HTML로 직접 변환한다 — 정확도 최상위권은 아니지만 비용·지연시간이 중요한 대량 처리 파이프라인에 적합하다.
Mistral OCR 4(2026-06)의 방향 전환 — Mistral은 2026년 6월 OCR 4를 출시하며 단어·페이지 단위 confidence score를 함께 반환하는 구조로 전환했다. 이는 "OCR 결과를 전량 신뢰하거나 전량 검증하거나"의 이분법 대신, 신뢰도가 낮은 영역만 선별적으로 사람이 검토하도록 라우팅하는 human-in-the-loop 설계를 표준 기능으로 끌어올렸다는 의미다. 2026년 실무 관점에서는 "필드 단위 confidence score를 제공하지 않는 OCR 벤더는 프로덕션에 준비되지 않은 것"이라는 평가가 나올 정도로, confidence score는 선택 사항이 아니라 필수 요구사항으로 자리잡고 있다.

표 구조 인식(TSR) — TableFormer와 Table Transformer(TATR)

Day 4에서 언급한 TableFormer(Docling 내장)는 표 이미지로 대량 학습된 전용 트랜스포머 모델로, 셀 경계와 병합 구조를 grid로 복원하고 수식(수학 표현)까지 함께 식별한다. 지속가능성 보고서 같은 복잡한 표 벤치마크에서 97.9% 정확도를 기록한 사례가 보고됐다.

Table Transformer(TATR)는 Microsoft가 공개한 DETR(객체 탐지 트랜스포머) 기반 표 추출 모델로, 57만 5천 페이지 이상의 과학 논문으로 구성된 PubTables-1M 데이터셋으로 학습됐다. 문서에서 직접 추출한 텍스트를 함께 활용하는 TATR-v1.2-Pub는 표 구조 평가 지표인 GriTS(Grid Table Similarity) 점수 0.980을 기록해, 이전 버전(v1.1) 대비 오류율을 절반으로 줄였다. 여러 페이지에 걸쳐 이어지는 표까지 다루기 위한 후속 데이터셋 PubTables-v2도 공개되어, "표 한 장"이 아니라 "표 하나가 여러 페이지에 걸쳐 있는" 실무 시나리오까지 벤치마크 범위가 넓어지고 있다.

재조립 가능성으로 TSR 결과를 검증하기 — Day 5에서 다룬 AST 청킹의 "재조립하면 원본이 복원돼야 한다"는 검증 원칙은 표 구조 인식에도 그대로 적용된다. 복원된 grid의 각 셀 값을 순서대로 다시 배열했을 때 원본 표 이미지와 대조해 셀 개수·병합 패턴이 일치하는지 자동 검증하는 것이, TSR 파이프라인의 품질을 지속적으로 확인하는 실무 표준이다.

이미지·차트 — "텍스트로 바꿀 것인가, 그대로 임베딩할 것인가, 둘 다 할 것인가"

2026년 실무에서 이미지/차트를 검색 가능하게 만드는 방법은 크게 두 갈래로 수렴한다.

① 텍스트 변환(캡셔닝) 방식

  • VLM이 이미지를 보고 자연어 설명(캡션)을 생성 → 표준 텍스트 임베딩 모델로 인덱싱
  • 기존 텍스트 RAG 파이프라인을 그대로 재사용 가능
  • 단점: 캡션 생성 과정에서 세부 수치·시각적 디테일이 손실될 수 있음 — 차트의 정확한 눈금 값까지 캡션에 담기는 어렵다

② 네이티브 멀티모달 임베딩 방식

  • 이미지를 텍스트로 바꾸지 않고, 텍스트와 이미지를 같은 벡터 공간에 매핑하는 멀티모달 임베딩 모델로 직접 인덱싱
  • 시각적 디테일 손실이 적음
  • 단점: 별도의 멀티모달 인덱스·임베딩 인프라가 필요하고, "왜 이 이미지가 검색됐는지"를 텍스트로 설명하기 어려워 근거 인용(citation grounding)이 까다로워짐
2026년 실무 표준 — "이미지를 두 번 본다"는 2단계 패턴 — 인덱싱 시점에는 가벼운 VLM으로 빠르고 저렴하게 캡션을 생성해 검색 가능한 텍스트 인덱스를 만들고, 실제로 그 이미지가 검색되어 답변을 생성해야 하는 시점에는 더 강력한 VLM이 원본 이미지와 사용자 질문을 함께 다시 읽어 질문에 필요한 정확한 정보(정확한 수치, 특정 범례 값 등)를 추출한다. 인덱싱 단계에서는 "이 이미지가 대략 무엇에 관한 것인가"만 알면 되고, 실제 정밀한 해석은 질문이 확정된 이후 필요한 이미지에만 비용을 들이는 방식이다. 이렇게 하면 대량의 이미지를 저비용으로 인덱싱하면서도, 실제 답변 품질이 필요한 순간에는 정밀도를 확보할 수 있다.

OCR 자체를 생략하는 접근 — ColPali/ColQwen2 계열 시각 문서 검색

Day 5에서 다룬 PixelRAG처럼, 2026년에는 OCR·레이아웃 분석·텍스트 추출을 아예 생략하고 페이지 이미지 자체를 검색 대상으로 삼는 접근이 성숙 단계에 접어들었다. ColPali와 후속 모델(ColQwen2.5 등)은 SigLIP 계열 비전 인코더로 페이지를 패치(작은 격자 조각) 단위 임베딩으로 변환하고, 질의 토큰과 문서 패치 사이의 최대 유사도를 합산하는 MaxSim(late interaction, 지연 상호작용) 방식으로 관련도를 계산한다. 스캔 품질이 나쁘거나 레이아웃이 복잡한 문서에서 OCR 파이프라인 전체를 우회하기 때문에 OCR 오류 자체가 발생하지 않는다는 장점이 있다.

트레이드오프 — 근거 추적성. Day 5의 PixelRAG 논의와 동일한 문제가 여기서도 반복된다. ColPali류 접근은 "이 페이지가 관련 있다"까지는 잘 찾아내지만, "페이지의 어느 부분, 몇 번째 줄이 근거인지"를 텍스트로 정확히 인용하기가 어렵다. Day 12에서 다룰 citation grounding(근거 인용) 요구사항이 엄격한 시스템(법률·의료·감사)이라면, OCR 기반 텍스트 추출 경로를 포기하기 어렵다. 반대로 레이아웃이 극도로 복잡하고 근거 인용보다 "관련 페이지를 찾아내는 재현율"이 더 중요한 상황(예: 방대한 스캔 아카이브에서 관련 문서 찾기)이라면 ColPali류가 더 나은 선택일 수 있다. "어느 쪽이 항상 옳다"가 아니라 요구사항에 따른 트레이드오프로 판단해야 한다.

OCR 정확도가 문서 상태에 따라 극단적으로 갈린다는 사실

문자 인식 정확도는 문서 상태에 크게 좌우된다. 깨끗한 워드프로세서 출력물에서는 95~99%에 이르지만, 반복 복사된 팩스본·손글씨 주석이 섞인 문서·혼합 폰트와 워터마크가 있는 페이지에서는 60~80%까지 떨어질 수 있다. 정확도 80%인 상태에서 500단어짜리 페이지 하나를 OCR하면 산술적으로 약 100개의 토큰이 깨진 상태로 나온다는 뜻이고, 이 정도 규모의 오류가 섞인 임베딩은 원문이 실제로 말하는 내용을 더 이상 제대로 대표하지 못한다. 수집 파이프라인 설계 시점에 "우리 문서군의 상태가 어느 구간에 속하는지"를 먼저 파악해야, OCR 엔진 선택과 confidence score 임계값을 현실적으로 정할 수 있다.

최근 deprecation/migration — 알아두면 감점을 피할 수 있는 것들

  • Mistral OCR 4(2026-06): 이전에는 OCR → 레이아웃 재구성 → 구조화라는 별도 단계로 나뉘어 있던 파이프라인을 단일 호출로 압축하고, 필드 단위 confidence score를 표준으로 추가했다. 이전 버전 기준으로 설계된 후처리(별도 표 구조 재구성 로직)가 있다면 중복 작업이 되지 않는지 재검토가 필요하다.
  • Table Transformer(TATR) v1.2: 문서에서 직접 추출한 텍스트를 함께 활용하도록 개선되며 GriTS 점수가 크게 상승했다 — 순수 이미지 기반 v1.1 파이프라인을 쓰고 있다면 텍스트 레이어 활용 여부를 확인해야 한다.
  • PubTables-v2: 표 하나가 여러 페이지에 걸치는 시나리오까지 다루는 데이터셋/벤치마크가 새로 공개되면서, "표 벤치마크 1위"라는 주장이 단일 페이지 표만 대상으로 한 것인지 여러 페이지 표까지 포함한 것인지 구분해서 봐야 한다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 OCR 정확도 하나의 숫자만으로 벤더나 파이프라인을 채택하지 않습니다. 스캔 문서 파싱은 사실 레이아웃 분석, 문자 인식(OCR), 표 구조 인식이라는 세 개의 독립적인 문제로 나뉘고, 각각 다른 지점에서 실패할 수 있기 때문입니다.

특히 최근 연구에서 확인된 것처럼, OCR 정확도가 전체적으로는 높아도 그 오류가 숫자나 표처럼 LLM이 문맥으로 보정할 수 없는 핵심 정보에 몰리면, 문서 단위 OCR 정확도가 80%대인데 최종 RAG 답변 정확도는 50%대까지 떨어지는 경우가 보고됐습니다. 그래서 저는 OCR 벤더의 종합 점수만 보지 않고, 저희 도메인 골든셋으로 최종 답변 정확도까지 반드시 함께 측정합니다. 특히 표 항목만 따로 떼서 봤을 때 종합 점수만큼 나오는지도 확인하는데, 실제로 어떤 OCR 모델은 종합 점수는 준수해도 표 항목에서는 절반에도 못 미치는 경우가 있었습니다.

병합 셀이 많은 표는 일반 OCR로 글자만 읽어서는 부족하다고 생각합니다. TableFormer나 Table Transformer 같은 전용 표 구조 인식 모델로 grid와 병합 패턴을 복원하고, 복원한 결과를 순서대로 재배열했을 때 원본과 셀 개수·구조가 일치하는지 검증하는 과정을 파이프라인에 넣습니다.

이미지나 차트는 텍스트로 완전히 대체할 수 없는 정보 채널이라고 봅니다. 그래서 인덱싱 시점에는 가벼운 VLM으로 대략적인 캡션을 만들어 저비용으로 검색 가능하게 하고, 실제로 그 이미지가 검색되어 답변에 쓰여야 하는 시점에는 더 강력한 VLM이 원본 이미지와 사용자의 실제 질문을 함께 다시 읽어서 정확한 수치나 디테일을 뽑아내는 2단계 구조를 씁니다. 이렇게 하면 대량의 이미지를 전부 정밀 분석하는 비용을 들이지 않으면서도, 실제로 필요한 순간에는 정밀도를 확보할 수 있습니다.

마지막으로, OCR 결과를 전량 신뢰하거나 전량 사람이 검증하는 방식은 비용이나 품질 어느 한쪽에서 손해를 봅니다. 그래서 필드 단위 confidence score를 활용해서, 신뢰도가 낮은 영역만 선별적으로 사람 검토나 재처리 큐로 보내는 human-in-the-loop 게이트를 둡니다. 그리고 근거 인용이 엄격하게 요구되는 도메인이라면, OCR 자체를 생략하고 페이지 이미지를 통째로 검색하는 접근보다는 텍스트 추출 기반 경로를 유지하는 게 맞다고 판단합니다 — 어느 문장, 몇 페이지가 근거인지 명확히 인용할 수 있어야 하기 때문입니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "OCR 정확도가 99%라는 리포트를 받았는데 사용자들이 RAG 답변이 이상하다고 계속 불만을 제기합니다. 어떻게 진단하시겠습니까?"

약한 답변: "OCR 엔진을 더 좋은 것으로 교체합니다."

강한 답변: "먼저 그 99%가 전체 문자 단위 정확도인지, 아니면 숫자·표처럼 정보 밀도가 높은 영역만 따로 측정한 것인지 확인합니다. 'OCR Blind Spot' 국면처럼 전체 정확도는 높아도 오류가 핵심 정보에 집중되는 경우가 실제로 있기 때문입니다. 실제 불만이 들어온 답변들을 역추적해서, 오답의 원인이 검색 실패인지 OCR 단계의 특정 영역(표, 숫자, 단위) 오류인지 먼저 좁힙니다. 그 다음 해당 영역만 골든셋으로 별도 측정해서 진짜 원인을 확정한 뒤 엔진 교체 여부를 결정합니다."

F2. "표가 4페이지에 걸쳐 이어지는 재무제표가 있습니다. 페이지 경계에서 표가 끊긴 것처럼 파싱됩니다. 어떻게 해결하시겠습니까?"

약한 답변: "페이지마다 독립적으로 파싱하고 나중에 텍스트로 합칩니다."

강한 답변: "페이지 경계를 넘나드는 표는 단일 페이지 기준으로 학습된 TSR 모델이 놓치기 쉬운 케이스라, 페이지 간 헤더 반복 패턴(같은 컬럼 헤더가 다음 페이지 상단에 다시 등장하는 것)을 감지해서 같은 논리적 표로 병합하는 후처리를 둡니다. 최근에는 여러 페이지 표까지 다루는 벤치마크와 데이터셋이 나오고 있어서, 채택한 TSR 모델이 이 케이스를 학습 데이터에 포함했는지 먼저 확인하고, 포함하지 않았다면 자체 도메인 데이터로 병합 로직을 검증합니다."

F3. "OCR confidence score 임계값을 어떻게 정하시겠습니까? 너무 낮게 잡으면 사람 검토 비용이 폭발하고, 너무 높게 잡으면 오류가 그대로 인덱스에 들어갑니다."

강한 답변: "임의의 숫자로 정하지 않고, 라벨링된 도메인 골든셋에서 confidence score별로 실제 오류율을 측정한 캘리브레이션 곡선을 먼저 그립니다. 그 다음 '이 문서군에서 잘못된 정보가 들어갔을 때의 비즈니스 리스크'(예: 법률·재무 문서라면 리스크가 크고, 사내 위키 잡담이라면 작음)에 따라 허용 가능한 오류율을 정하고, 그 오류율에 대응하는 confidence 임계값을 역산합니다. 임계값은 고정값이 아니라 문서 카테고리별로 다르게 설정하고, 정기적으로 재캘리브레이션합니다."

F4. "이미지 100만 장을 인덱싱해야 하는데 VLM 캡셔닝 비용이 예산을 초과합니다. 어떻게 줄이시겠습니까?"

약한 답변: "이미지 수를 줄이거나 캡션을 생략합니다."

강한 답변: "먼저 모든 이미지가 동등하게 중요한지 판단합니다. 순수 장식용 이미지(로고, 구분선)와 정보성 이미지(차트, 다이어그램, 스크린샷)를 먼저 저비용 분류기로 나누고, 정보성 이미지만 VLM 캡셔닝 대상으로 좁힙니다. 그 다음 인덱싱 단계에는 경량 VLM을 쓰고, 실제 답변 생성이 필요한 극소수의 이미지에만 고성능 VLM으로 재조회하는 2단계 구조를 적용해서, 전체 비용을 이미지 수가 아니라 실제 질의 빈도에 비례하게 만듭니다."

F5. "OCR 파이프라인 전체를 생략하고 ColPali 같은 시각 문서 검색으로 갈아타자는 제안이 팀에서 나왔습니다. 어떻게 판단하시겠습니까?"

강한 답변: "먼저 시스템의 근거 인용 요구 수준을 확인합니다. 법률·의료·감사처럼 '몇 페이지 몇 번째 줄이 근거인지' 정확히 답해야 하는 도메인이라면, 페이지 단위로만 관련도를 계산하는 시각 검색 방식은 근거 추적성이 부족해 적합하지 않을 수 있습니다. 반대로 스캔 품질이 매우 나쁘거나 레이아웃이 극도로 복잡해서 기존 OCR 파이프라인의 오류율 자체가 감당이 안 되는 아카이브라면, 관련 페이지를 찾아내는 재현율이 더 중요하므로 시각 검색이 나을 수 있습니다. 전면 교체보다는, 두 경로를 병행하면서 문서군별로 어느 쪽이 실제 지표에서 더 나은지 A/B로 검증한 뒤 도입 범위를 정하는 것을 제안합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • OCR·레이아웃·TSR을 별개 실패 지점으로 구분해서 진단한다
  • OCR 정확도와 다운스트림 RAG 정확도를 분리해서 측정하고, 오류가 어디에 집중되는지까지 확인한다
  • 병합 셀 표를 전용 TSR 모델 + 재조립 검증으로 구체적으로 설명한다
  • 이미지/차트를 비용·정확도·근거 추적성의 트레이드오프로 설명하고, 2단계 인덱싱 같은 구체적 아키텍처를 제시한다
  • confidence score 기반 품질 게이트를 도메인 리스크에 맞춰 캘리브레이션한다는 것을 안다

약한 답변의 특징

  • "OCR 정확도가 높으면 다 잘 됩니다"라고 단순 가정한다
  • 병합 셀 표를 "OCR로 읽으면 된다" 수준으로만 설명한다
  • 이미지는 "캡션 달면 된다"로 끝내고 트레이드오프를 설명하지 못한다
  • confidence score를 모르거나 "그냥 다 사람이 검토합니다"처럼 비용을 고려하지 않은 극단적 답을 한다
  • OCR-free 접근을 무조건 좋다/나쁘다로 단정한다
핵심 차이: 강한 답변은 "정확도라는 숫자 하나"에 의존하지 않고, 그 숫자가 어디서 왜 틀렸는지, 그 오류가 다운스트림에서 어떤 피해로 이어지는지까지 연결해서 설명한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "OCR 정확도가 높으면 RAG 답변도 자동으로 좋아진다"

사실: OCR 오류는 숫자·표처럼 LLM이 보정할 수 없는 핵심 정보에 몰리는 경향이 있어, 전체 OCR 정확도가 높아도 최종 RAG 답변 정확도는 훨씬 크게 떨어질 수 있다("OCR Blind Spot" 국면). 반드시 다운스트림 답변 정확도까지 함께 측정해야 한다.

오해 2: "표는 결국 OCR로 글자만 읽으면 된다"

사실: 표는 글자를 읽는 문제(OCR)와 grid 구조를 복원하는 문제(TSR)가 별개다. 병합 셀이 있는 표는 TableFormer나 Table Transformer 같은 전용 구조 인식 모델 없이는 컬럼·행이 밀리는 오류가 흔하게 발생한다.

오해 3: "이미지는 캡션만 달면 텍스트 RAG에 통합할 수 있다"

사실: 캡션 변환은 정보 손실을 동반한다. 인덱싱 단계의 저비용 캡션과, 실제 질의 시점의 정밀 재조회를 분리하는 2단계 구조가 비용과 정확도를 함께 만족시키는 2026년 실무 표준이다.

오해 4: "OCR-free 시각 문서 검색(ColPali류)이 항상 더 나은 최신 방법이다"

사실: OCR-free 접근은 OCR 오류를 원천 차단하지만, "어느 부분이 근거인지"를 정확히 인용하기 어려운 트레이드오프가 있다. 근거 인용이 엄격한 도메인에서는 텍스트 추출 기반 경로가 여전히 필요하다.

실무 체크리스트

OCR/Table/Image Extraction 파이프라인 점검 체크리스트

확인 항목 정상 신호 이상 신호 → 의심 지점
OCR 품질을 다운스트림 RAG 정확도로도 측정하는가 문자 단위 정확도 + 답변 정확도 이중 측정 벤더의 종합 OCR 정확도만 신뢰 → OCR Blind Spot 위험
표 항목을 세부 지표로 별도 확인했는가 표/숫자 전용 세부 벤치마크 확인 종합 점수만 확인 → 표 취약 모델을 표 중심 문서에 채택
병합 셀 표를 전용 TSR 모델로 처리하는가 TableFormer/TATR + 재조립 검증 OCR 텍스트만 순서대로 나열 → 컬럼/행 밀림
이미지/차트 인덱싱 전략이 명시적인가 경량 캡션(인덱싱) + 고성능 재조회(생성) 2단계 이미지 무시 또는 전량 고비용 VLM 처리
Confidence score 기반 게이트가 있는가 필드 단위 confidence + 도메인별 캘리브레이션 전량 자동 신뢰 또는 전량 사람 검토
근거 추적성 요구사항에 맞는 방식을 선택했는가 엄격한 인용 필요 시 텍스트 추출 경로 유지 근거 인용 요구를 무시하고 OCR-free로 전면 전환
자기 점검 Rubric

내 답변 수준 확인하기

미흡 "OCR 정확도가 높으면 다 잘 됩니다" 수준으로만 답하고, 레이아웃/OCR/TSR의 구분이나 표·이미지의 별도 문제를 전혀 언급하지 못함
주니어 OCR과 표 파싱이 다르다는 것 정도는 알지만, 왜 OCR 정확도가 RAG 품질과 비례하지 않는지, TSR이 구체적으로 무엇을 복원하는지 설명하지 못함
중급 OCR/레이아웃/TSR을 구분해 설명할 수 있고, 병합 셀 표에 전용 모델이 필요하다는 것과 이미지 캡셔닝의 정보 손실 가능성을 인식함
시니어 OCR 정확도와 다운스트림 RAG 정확도의 비선형 관계를 구체적 국면(OCR Blind Spot 등)으로 설명하고, TSR 모델과 재조립 검증 메커니즘을 구체적으로 제시하며, 이미지 인덱싱을 비용·정확도·근거 추적성의 트레이드오프로 설계하고, confidence score 캘리브레이션과 OCR-free 접근의 트레이드오프까지 follow-up에서 구체적 메커니즘으로 대응한다
자가 점검 질문:
1. 레이아웃 분석, OCR, 표 구조 인식(TSR)이 왜 서로 다른 문제인지, 각각 실패했을 때 증상이 어떻게 다른지 설명할 수 있는가?
2. OCR 정확도가 높은데도 RAG 답변 품질이 낮을 수 있는 이유를 구체적 국면으로 설명할 수 있는가?
3. 병합 셀이 있는 표를 어떤 모델로, 어떻게 검증하며 처리하는지 설명할 수 있는가?
4. 이미지/차트를 검색 가능하게 만드는 전략의 트레이드오프(비용, 정확도, 근거 추적성)를 구체적 아키텍처로 설명할 수 있는가?
5. confidence score 임계값을 어떤 근거로 정하고, OCR-free 접근을 언제 선택해야 하는지 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • 스캔 문서 파싱은 레이아웃 분석·OCR·표 구조 인식(TSR)이라는 세 개의 독립적인 문제다 — 하나를 잘한다고 나머지가 자동으로 해결되지 않는다.
  • OCR 정확도와 RAG 답변 정확도는 비선형 관계다 — 오류가 숫자·표 같은 핵심 정보에 집중되면("OCR Blind Spot"), 전체 OCR 정확도가 높아도 답변 정확도는 크게 떨어질 수 있다. 실측된 사례로 82.9% OCR 정확도가 53.0% RAG 정확도로 이어졌다.
  • 2026년 OCR 지형에서도 소형 특화 모델(Chandra, olmOCR-2, dots.ocr)이 범용 프론티어 모델보다 우수한 경우가 많고, 종합 점수와 표 세부 점수가 크게 다를 수 있다.
  • 병합 셀 표는 TableFormer/Table Transformer(TATR) 같은 전용 구조 인식 모델로 grid를 복원하고, 재조립 검증으로 셀 유실 여부를 확인해야 한다.
  • 이미지/차트는 경량 캡션(인덱싱) + 고성능 재조회(생성) 2단계 구조가 비용과 정확도를 함께 만족시키는 실무 표준이며, ColPali류 OCR-free 접근은 근거 추적성과의 트레이드오프로 판단해야 한다.
  • Confidence score 기반 human-in-the-loop 게이트는 2026년 기준 선택이 아니라 프로덕션 OCR 파이프라인의 필수 요구사항으로 자리잡았다.
  • 오늘 다룬 metadata(confidence score, provenance)는 Day 7(Metadata와 Provenance)로, 표/이미지 청킹 전략은 Day 8(Semantic Chunking)으로, 근거 추적성 트레이드오프는 Day 12(Citation Grounding)로 이어진다.
