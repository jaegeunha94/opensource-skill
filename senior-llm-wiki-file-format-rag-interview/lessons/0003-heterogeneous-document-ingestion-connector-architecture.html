Q3 — 변형 "특정 소스 커넥터(예: Confluence)가 장애가 나거나 API rate limit에 걸렸을 때, 전체 인덱싱 파이프라인이 함께 멈추지 않게 하려면 어떻게 설계해야 합니까?"
전제 개념

"커넥터"는 단순한 API 호출 스크립트가 아니다

Day 1~2에서 이기종 RAG의 전체 아키텍처와 지식 큐레이션을 다뤘다. 오늘부터는 파이프라인의 맨 앞단, 즉 원본 데이터가 시스템으로 들어오는 관문을 집중적으로 본다. 많은 지원자가 "커넥터"를 "API를 호출해서 파일을 가져오는 for-loop 스크립트" 정도로 생각하지만, 프로덕션 커넥터는 그보다 훨씬 많은 책임을 진다 — 인증/권한 동기화, 증분 갱신, rate limit 준수, 스키마 변화 대응, 부분 실패 복구까지 모두 커넥터의 몫이다.

Push vs Pull vs 초기 전체 동기화(Full Sync)

방식 동작 장점 한계
Full Sync (초기 적재) 소스의 전체 데이터를 한 번에 크롤링 구현이 단순하고 최초 색인 완결성이 높음 대규모 소스에서는 시간·비용이 크고, 소스 API에 부하를 줌
Pull(폴링) 증분 크롤 주기적으로 "마지막 동기화 이후 변경분"만 조회 (cursor/timestamp 기반) API 부하가 낮고 예측 가능하며, 웹훅이 없는 소스에도 적용 가능 폴링 주기만큼 지연이 생기고, 삭제 이벤트를 놓치기 쉬움
Push(웹훅/이벤트) 소스 시스템이 변경 발생 시 즉시 알림을 보냄 거의 실시간 최신성 확보, API 폴링 부하 최소화 웹훅 유실 가능성이 있고, 모든 소스가 웹훅을 지원하지 않음
실무에서는 하나만 쓰지 않는다: 프로덕션 커넥터는 보통 초기 Full Sync + 웹훅(있으면) + 안전망으로서의 주기적 폴링을 함께 쓴다. 웹훅은 유실될 수 있으므로, 웹훅만 믿고 폴링을 끄면 "분명히 바뀐 문서인데 검색 결과가 옛날 버전"이라는 장애가 조용히 누적된다.

Extract-Stage-Transform: 왜 "가져오기"와 "파싱하기"를 분리해야 하는가

흔한 설계 실수는 커넥터가 파일을 가져오자마자 곧바로 파싱·청킹까지 한 번에 처리하는 것이다. 이렇게 하면 파싱 실패가 곧 수집 실패가 되어, 나중에 파싱 로직을 개선해도 이미 실패 처리된 원본을 다시 가져오려면 커넥터를 또 실행해야 한다. 그래서 실무에서는 단계를 분리한다.

1
Extract — 원본을 있는 그대로 가져온다 커넥터는 원본 바이트와 소스 메타데이터(작성자, 수정 시각, 권한 정보, 소스 고유 ID)만 가져와 객체 스토리지(S3 등)에 가공하지 않은 원본 그대로 저장한다. 이 단계가 성공하면 "우리는 이 시점의 원본을 안전하게 보관하고 있다"는 것이 보장된다.
2
Stage — 큐에 적재하고 상태를 추적한다 가져온 원본은 즉시 파싱하지 않고 처리 대기열(큐)에 적재된다. 이 단계에서 문서별 처리 상태(대기/처리중/성공/실패)를 추적할 수 있어, 실패한 문서만 골라 재처리할 수 있다.
3
Transform — 파싱·청킹·임베딩은 별도 단계에서 Day 4~10에서 다룰 파싱/OCR/청킹/임베딩은 이 단계에서 일어난다. 이 단계가 실패해도 원본(raw)은 이미 안전하게 보관돼 있으므로, 파서 버그를 고친 뒤 재수집 없이 재처리(reprocess)만 하면 된다.
이것이 Day 1~2의 LLM Wiki raw/ 불변성 원칙과 같은 이유다: 원본을 먼저 안전하게 확보해두면, 다운스트림 로직이 아무리 바뀌어도 "처음부터 다시 가져와야 하나?"라는 값비싼 질문에서 자유로워진다.

포맷 감지: 확장자와 클라이언트가 보낸 MIME 타입은 신뢰할 수 없다

업로드된 파일이 정말 그 확장자가 말하는 포맷인지는 내용을 직접 확인하기 전까지 알 수 없다. 확장자는 사용자가 임의로 바꿀 수 있고, 브라우저나 클라이언트가 보내는 Content-Type 헤더도 클라이언트의 자기 신고에 불과하다. 그래서 신뢰할 수 있는 포맷 감지는 파일의 매직 바이트(magic bytes, 파일 시그니처), 즉 파일 앞부분의 고유 바이트 패턴을 직접 읽는 content-based 방식이어야 한다. 예를 들어 PNG는 89 50 4E 47, PDF는 25 50 44 46로 시작한다.

이기종 RAG에서 특히 까다로운 함정 — ZIP 기반 컨테이너 포맷: .docx, .pptx, .xlsx, .epub, .jar는 전부 같은 매직 바이트(50 4B 03 04, 즉 ZIP 시그니처)로 시작한다. 매직 바이트만으로는 이 다섯 포맷을 구분할 수 없다는 뜻이다. 실제 포맷을 알려면 ZIP 컨테이너 내부를 열어 [Content_Types].xml (OOXML 계열: docx/pptx/xlsx), META-INF/MANIFEST.MF(JAR), mimetype 항목(EPUB) 같은 애플리케이션 레벨 마커까지 확인해야 한다. 이 사실을 모르는 채로 "매직 바이트만 보면 다 해결된다"고 답하면, 시니어 면접에서 바로 감점 포인트가 된다.
단계 확인 대상 신뢰도
1차 파일 확장자 낮음 — 임의로 변경 가능
2차 클라이언트가 보낸 MIME 타입(Content-Type) 낮음 — 클라이언트 자기 신고
3차 매직 바이트(파일 시그니처) 중간 — ZIP 계열 포맷은 추가 확인 필요
4차 컨테이너 내부 구조 검사([Content_Types].xml 등) 높음 — 애플리케이션 레벨까지 확정

왜 커넥터 격리(bulkhead)가 필요한가

선박의 방수 격벽(bulkhead)은 한 구획에 물이 차도 다른 구획으로 번지지 않게 막는다. 이기종 문서 수집 파이프라인에서도 같은 원칙이 필요하다 — Confluence 커넥터가 인증 만료로 실패했다고 해서 Slack·GitHub·Salesforce 커넥터까지 함께 멈추면 안 된다. 이를 위해 커넥터마다 독립적인 실행 단위(별도 워커/큐), 독립적인 재시도 정책, 독립적인 서킷 브레이커(circuit breaker)를 둔다. 하나의 소스가 반복적으로 실패하면 그 소스만 자동으로 "차단(open)" 상태로 전환해 불필요한 재시도로 자원을 낭비하거나 소스 API를 더 압박하지 않게 한다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • 데모 제작자인지 운영 경험자인지 — "API 있으면 for-loop 돌리면 되지 않나요?" 수준인지, 부분 실패·rate limit·스키마 변화·재처리 비용 같은 실제 운영 이슈를 미리 설계에 반영하는지를 본다.
  • 신뢰 경계(trust boundary)를 이해하는지 — 확장자와 클라이언트 자기 신고를 신뢰하지 않고, 내용 기반으로 검증하는 보안 감각이 있는지 확인한다. 이는 파일 업로드 보안 전반과 직결되는 판단력이다.
  • 장애 격리 설계 경험이 있는지 — 하나의 소스 실패가 전체 파이프라인을 멈추지 않도록 격리하는 구조(bulkhead, 서킷 브레이커, DLQ)를 실제로 설계해본 적이 있는지를 본다.
  • 확장 가능한 아키텍처로 사고하는지 — 커넥터가 3개일 때와 30개, 300개일 때 무엇이 달라지는지, 반복되는 보일러플레이트를 어떻게 줄이는지에 대한 감각을 평가한다.
실무 연결

실제 상황과 연결하기

Glean의 커넥터 운영 방식 — 초기 동기화 이후의 "steady state"

엔터프라이즈 검색/RAG 플랫폼 Glean의 커넥터는 초기 전체 동기화를 마치면 "steady state"로 전환되어, 이후에는 웹훅(가능한 경우)과 주기적 증분 크롤을 함께 사용해 색인을 최신 상태로 유지한다. 예를 들어 일부 커넥터는 1시간 단위로 그 사이에 발생한 변경분을 모아 한 번에 반영해 API 호출 횟수를 줄인다. 그리고 커넥터가 콘텐츠를 가져올 때 소스의 권한 정보(permission map)도 함께 가져와 색인에 동봉한다 — 이 권한 동기화는 Day 13(Permission-aware Retrieval)에서 다시 깊이 다룰 핵심 개념이지만, 이미 커넥터 설계 단계에서부터 시작되는 책임이라는 점을 기억해야 한다.

범용 데이터 통합 커넥터(Airbyte류) vs 문서 특화 파서(Unstructured류)

실무에서는 두 계층의 도구를 함께 쓰는 경우가 많다. Airbyte 같은 데이터 통합 플랫폼은 CRM, ERP, SaaS 앱, 클라우드 스토리지 등 구조화된 소스에서 데이터를 가져오는 커넥터(증분 동기화, CDC 지원)에 강하고, Unstructured 같은 도구는 PDF·이메일·스캔 문서 같은 비정형 콘텐츠를 파싱해 시맨틱 엘리먼트로 변환하는 데 특화돼 있다. 이기종 RAG 파이프라인은 보통 "커넥터 계층(어디서 가져올 것인가)"과 "파싱 계층(어떻게 구조화할 것인가)"을 분리해서 설계하고, 이 둘 사이를 Extract-Stage-Transform 경계로 나눈다.

오케스트레이션 계층 — 왜 커넥터 스크립트를 cron으로만 돌리면 안 되는가

2026년 기준 RAG 파이프라인 오케스트레이션은 Airflow, Prefect, Dagster, Kestra 같은 워크플로우 오케스트레이터가 담당하는 경우가 많다. 이들이 제공하는 핵심 가치는 단순 스케줄링이 아니라 트리거, 재시도 정책, 실패 관측성, 알림을 하나의 그래프로 관리하는 것이다. cron으로 스크립트를 돌리면 "몇 시에 실패했는지"는 로그를 뒤져야 알 수 있지만, 오케스트레이터를 쓰면 어느 태스크가 실패했고 무엇을 재시도했는지 그래프 단위로 즉시 볼 수 있다.

부분 실패는 예외가 아니라 일상이다 — Dead Letter Queue

수천~수백만 건의 문서를 처리하다 보면 일부는 반드시 실패한다 — 손상된 파일, 예상치 못한 인코딩, 파서가 지원하지 않는 하위 버전 포맷 등. 이런 실패 건을 죽은 편지함(Dead Letter Queue, DLQ)으로 격리해 전체 배치의 성공을 막지 않게 하고, 나중에 실패 사유별로 모아 재처리하거나 사람이 검토할 수 있게 한다. DLQ 없이 "실패하면 전체 배치를 재시도"하는 방식은 배치 규모가 커질수록 재시도 비용이 기하급수적으로 커진다.

Rate Limit과 백프레셔 — 소스를 존중하지 않으면 커넥터 자체가 차단된다

소스 시스템(Confluence, Slack, GitHub API 등)은 저마다 다른 rate limit 정책을 가진다. 커넥터가 이를 무시하고 최대 속도로 요청하면 소스 측에서 일시적으로 또는 영구적으로 접근을 차단할 수 있다. 그래서 프로덕션 커넥터는 소스별로 토큰 버킷(token bucket) 또는 leaky bucket 기반 속도 제한과 지수 백오프(exponential backoff)를 구현하고, 시스템 전체의 동시성 상한(concurrency cap)으로 여러 커넥터가 동시에 자원을 과점유하지 않도록 제어한다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
Extract-Stage-Transform으로 책임을 분리 "저는 먼저 '가져오기'와 '파싱/청킹'을 분리합니다. 커넥터는 원본 바이트와 소스 메타데이터만 그대로 객체 스토리지에 저장하고, 파싱은 별도 단계에서 큐를 통해 비동기로 처리합니다. 이렇게 하면 파서 버그를 고쳐도 재수집 없이 재처리만 하면 됩니다."
2
포맷은 내용 기반으로 검증 "확장자와 클라이언트가 보낸 MIME 타입은 신뢰하지 않고, 매직 바이트로 1차 검증합니다. 다만 DOCX/PPTX/XLSX처럼 ZIP 기반 포맷은 매직 바이트만으로 구분이 안 되므로, 컨테이너 내부의 [Content_Types].xml 같은 애플리케이션 레벨 마커까지 확인합니다."
3
초기 Full Sync + Push/Pull 하이브리드로 최신성 유지 "최초에는 전체 동기화로 색인을 채우고, 이후에는 웹훅이 있으면 웹훅으로, 없거나 유실 가능성에 대비해 주기적 증분 폴링을 안전망으로 함께 둡니다."
4
커넥터 단위 격리와 실패 처리 "커넥터마다 독립된 실행 단위와 서킷 브레이커를 둬서 한 소스의 장애가 다른 소스로 번지지 않게 합니다. 개별 문서 처리 실패는 DLQ로 격리해 전체 배치를 막지 않고, 실패 사유별로 재처리·검토합니다."
5
소스를 존중하는 속도 제어와 확장성 (Senior 포인트) "소스별 rate limit에 맞춘 토큰 버킷과 백오프를 구현하고, 커넥터 수가 늘어나는 걸 감안해 선언적 설정 기반 커넥터 프레임워크로 반복 로직을 줄입니다. 그리고 커넥터별 성공률·지연·큐 적체 지표를 관측성 대시보드로 분리해서 봅니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 커넥터 아키텍처를 크게 세 단계로 나눕니다 — Extract, Stage, Transform입니다. 커넥터는 소스 API나 웹훅에서 원본 바이트와 메타데이터 (작성자, 수정 시각, 소스 고유 ID, 권한 정보)만 가져와 객체 스토리지에 가공 없이 저장합니다. 이 원본은 파싱 로직이 나중에 바뀌어도 절대 다시 가져올 필요가 없는 진실 공급원이 됩니다. 파싱·청킹·임베딩 같은 무거운 작업은 큐를 통해 완전히 분리된 단계에서 처리해서, 파서가 실패해도 수집 자체가 막히지 않게 합니다.

포맷 판별은 확장자나 클라이언트가 보낸 MIME 타입을 신뢰하지 않고 매직 바이트로 내용을 직접 확인합니다. 다만 DOCX, PPTX, XLSX, EPUB, JAR는 전부 같은 ZIP 시그니처로 시작하기 때문에, 매직 바이트만으로는 구분이 안 됩니다. 그래서 ZIP 계열로 판별되면 컨테이너를 열어 [Content_Types].xml이나 META-INF/MANIFEST.MF 같은 애플리케이션 레벨 마커까지 확인해서 실제 포맷을 확정합니다.

최신성은 초기 전체 동기화 이후 웹훅과 주기적 증분 폴링을 함께 사용해서 유지합니다. 웹훅만 믿으면 유실 시 색인이 조용히 stale해질 수 있어서, 폴링을 안전망으로 반드시 함께 둡니다.

운영 안정성을 위해서는 커넥터마다 독립적인 실행 단위와 서킷 브레이커를 둬서 한 소스의 장애나 rate limit이 다른 소스로 번지지 않게 격리합니다. 개별 문서 처리 실패는 전체 배치를 막지 않도록 Dead Letter Queue로 격리하고, 실패 사유별로 자동 재시도하거나 사람이 검토합니다. 그리고 소스별 rate limit에 맞춰 토큰 버킷 기반 속도 제어와 지수 백오프를 적용해서, 저희 쪽의 과도한 요청이 소스 API 차단으로 이어지지 않게 합니다. 마지막으로 커넥터가 수십 개로 늘어나는 걸 감안해서 반복 로직은 선언적 설정으로 뽑아내고, 커넥터별 성공률·지연·큐 적체를 독립된 지표로 관측합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "커넥터가 100개가 넘어가면 무엇이 깨지나요?"

약한 답변: "커넥터를 계속 추가로 만들면 됩니다."

강한 답변: "먼저 커넥터마다 인증/권한 동기화/증분 갱신 로직을 개별 코드로 복붙하면 유지보수가 불가능해집니다. 그래서 공통 로직(재시도, 서킷 브레이커, 속도 제어, 상태 추적)은 커넥터 SDK/프레임워크로 뽑아내고, 소스별 차이(API 엔드포인트, 인증 방식, 페이지네이션 규칙)만 선언적 설정이나 어댑터로 주입합니다. 다음으로 커넥터별 관측성이 하나의 대시보드에 뭉개지면 장애 지점을 찾기 어려워지므로, 커넥터별 성공률·지연·큐 적체를 독립 지표로 분리합니다. 마지막으로 오래된 커넥터의 API 버전이 deprecated될 때 전체 파이프라인에 영향 없이 개별 커넥터만 마이그레이션할 수 있는 버전 관리 체계가 필요합니다."

F2. "소스 시스템이 rate limit을 걸면 어떻게 대응하시겠습니까?"

약한 답변: "실패하면 나중에 다시 시도합니다."

강한 답변: "먼저 소스별로 알려진 rate limit에 맞춰 토큰 버킷으로 사전에 요청 속도를 제한합니다 — 한도에 걸리고 나서 대응하는 게 아니라 애초에 걸리지 않게 설계합니다. 429 응답이 오면 지수 백오프와 지터(jitter)를 적용해 재시도하고, 반복적으로 실패하면 서킷 브레이커로 해당 커넥터를 일시 차단해 소스를 더 압박하지 않습니다. 그리고 초기 전체 동기화처럼 큰 트래픽이 필요한 작업과, 실시간성이 중요한 증분 업데이트를 우선순위 큐로 분리해서, rate limit budget이 부족할 때 증분 업데이트가 먼저 처리되도록 합니다."

F3. "손상되거나 악성일 수 있는 파일이 들어오면 어떻게 하시겠습니까?"

강한 답변: "포맷 판별과 파싱 자체를 신뢰 경계로 취급합니다. 파일 파서는 zip bomb (매우 작은 압축 파일이 압축 해제 시 메모리를 고갈시키는 공격)이나 malformed 파일로 인한 크래시에 취약할 수 있으므로, 파싱은 리소스 제한(메모리/CPU/시간)이 걸린 샌드박스 환경에서 실행합니다. 포맷 감지 단계에서 선언된 포맷과 실제 내용이 일치하지 않으면 정상 파이프라인으로 보내지 않고 격리(quarantine) 버킷으로 분리해 사람이 검토하게 하고, 절대 파서에 그대로 먹이지 않습니다."

F4. "웹훅을 지원하지 않는 레거시 소스는 어떻게 최신성을 유지하나요?"

강한 답변: "커서(cursor) 또는 타임스탬프 기반 증분 폴링으로 대응합니다. 매번 전체 재크롤 대신 '마지막 동기화 시각 이후 수정된 항목'만 조회하는 API가 있으면 그것을 쓰고, 없으면 소스의 변경 로그나 버전 필드를 커서로 활용합니다. 다만 이 방식은 삭제 이벤트를 놓치기 쉬우므로, 주기적으로 (예: 하루~일주일 단위) 전체 목록과 색인 상태를 대조하는 정합성 검사 (reconciliation)를 별도로 돌려서 '삭제됐는데 색인에는 남아있는' 상태를 찾아 정리합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Extract-Stage-Transform을 명시적으로 분리해서 파싱 실패가 수집 실패로 번지지 않게 설계한다
  • 확장자/클라이언트 MIME을 신뢰하지 않고, ZIP 기반 컨테이너 포맷의 모호성까지 알고 있다
  • Push/Pull을 이분법이 아니라 하이브리드로 설계해 웹훅 유실에 대비한다
  • 커넥터 격리(bulkhead), 서킷 브레이커, DLQ를 구체적 메커니즘으로 설명한다
  • rate limit을 사후 대응이 아니라 사전 설계 문제로 다룬다

약한 답변의 특징

  • "API로 파일을 가져와서 바로 파싱하면 됩니다"처럼 수집과 파싱을 분리하지 않는다
  • "확장자만 보면 포맷을 알 수 있다"고 답하거나, ZIP 계열 포맷 문제를 모른다
  • 실패 시 "나중에 다시 시도합니다" 수준으로만 답하고 격리 메커니즘을 제시하지 못한다
  • rate limit을 "걸리면 그때 대응"으로만 다룬다
  • 커넥터 개수가 늘어나는 상황을 고려하지 않고 1개 커넥터 기준으로만 설계를 설명한다
핵심 차이: 강한 답변은 커넥터를 "API 호출 스크립트"가 아니라 "부분 실패가 상시 발생하는 분산 시스템의 한 구성 요소"로 취급한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "확장자만 보면 포맷을 알 수 있다"

사실: 확장자는 사용자가 임의로 바꿀 수 있고, 클라이언트가 보낸 MIME 타입도 자기 신고에 불과하다. 게다가 DOCX/PPTX/XLSX/EPUB/JAR는 매직 바이트조차 동일(ZIP 시그니처)해서, 진짜 포맷을 확정하려면 컨테이너 내부 구조까지 확인해야 한다.

오해 2: "커넥터 하나 만들면 나머지는 복사-수정하면 된다"

사실: 소스마다 인증 방식, rate limit 정책, 권한 모델, 증분 갱신 지원 여부(웹훅/커서/없음)가 전부 다르다. 공통 로직만 프레임워크로 뽑아내지 않으면, 커넥터 수가 늘어날수록 유지보수 비용이 선형이 아니라 더 가파르게 늘어난다.

오해 3: "웹훅이 있으면 폴링은 필요 없다"

사실: 웹훅은 네트워크 문제, 소스 측 재시도 정책 미비, 배포 중 다운타임 등으로 유실될 수 있다. 폴링을 안전망으로 함께 두지 않으면, "웹훅이 왔어야 하는데 오지 않은" 변경 사항이 색인에서 조용히 빠진 채로 남는다.

오해 4: "한 문서 파싱 실패는 전체 배치 실패로 이어져도 괜찮다"

사실: 대규모 배치에서는 반드시 일부 문서가 실패한다. DLQ로 격리하지 않으면 전체 배치를 통째로 재시도해야 하고, 이는 배치 규모가 커질수록 비용이 기하급수적으로 커지며, 이미 성공한 문서까지 불필요하게 재처리하게 만든다.

실무 체크리스트

커넥터/수집 파이프라인 운영 전 확인 체크리스트

확인 항목 정상 신호 이상 신호 → 의심 지점
원본(raw)과 파싱 단계가 분리돼 있는가 분리됨 한 단계 → 파서 버그마다 재수집 필요
포맷 감지가 내용(매직 바이트+컨테이너) 기반인가 내용 기반 확장자/MIME만 신뢰 → 스푸핑 취약
커넥터별 서킷 브레이커/독립 실행 단위가 있는가 격리됨 공유 실행 → 한 소스 장애가 전체 전파
개별 문서 실패가 DLQ로 격리되는가 격리됨 전체 배치 재시도 → 비용 급증
웹훅과 폴링이 함께 동작하는가 하이브리드 웹훅만 의존 → 유실 시 stale 색인
소스별 rate limit에 맞춘 속도 제어가 있는가 사전 제한 사후 대응만 → 소스 차단 위험

내 답변 수준 확인하기

미흡 "API로 파일을 가져와서 파싱하면 됩니다" 정도로만 설명하고, 실패 처리나 포맷 검증을 전혀 언급하지 못함
주니어 매직 바이트로 포맷을 검증해야 한다는 것은 알지만, ZIP 기반 컨테이너 포맷의 모호성이나 커넥터 격리 필요성은 설명하지 못함
중급 Extract-Stage-Transform 분리와 DLQ 개념을 설명할 수 있고, 웹훅/ 폴링 하이브리드가 필요하다는 것을 인식함
시니어 포맷 감지를 신뢰 경계 문제로 이해해 ZIP 계열 포맷의 애플리케이션 레벨 검증까지 설명하고, 커넥터별 서킷 브레이커/rate limit 사전 설계/ 선언적 설정 기반 확장성을 구체적으로 제시하며, 커넥터 수 증가·rate limit·손상 파일 같은 follow-up에 원리 기반으로 대응한다
자가 점검 질문:
1. Extract-Stage-Transform을 왜 분리해야 하는지, 분리하지 않으면 어떤 비용이 발생하는지 설명할 수 있는가?
2. DOCX/PPTX/XLSX/EPUB/JAR가 매직 바이트만으로 구분되지 않는 이유와, 실제로 어떻게 구분하는지 설명할 수 있는가?
3. 웹훅과 폴링을 왜 함께 써야 하는지, 웹훅만 쓰면 어떤 장애가 생기는지 설명할 수 있는가?
4. 커넥터 격리(bulkhead)가 없을 때 발생하는 구체적인 장애 시나리오를 말할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • Extract-Stage-Transform 분리가 이기종 수집 아키텍처의 기본 골격이다. 원본을 먼저 안전하게 확보해야 다운스트림 파싱 로직이 바뀌어도 재수집 없이 재처리할 수 있다.
  • 포맷 감지는 신뢰 경계 문제다. 확장자와 클라이언트 MIME은 신뢰할 수 없고, 매직 바이트도 ZIP 기반 컨테이너 포맷(DOCX/PPTX/ XLSX/EPUB/JAR)에서는 컨테이너 내부까지 확인해야 확정된다.
  • Push(웹훅)와 Pull(폴링)은 이분법이 아니라 하이브리드다. 웹훅 유실에 대비한 폴링 안전망 없이는 색인이 조용히 stale해진다.
  • 커넥터 격리(bulkhead), 서킷 브레이커, DLQ가 없으면 하나의 소스 장애나 문서 실패가 전체 파이프라인을 멈추게 한다.
  • Rate limit은 사전에 설계해야 한다. 사후 대응만으로는 소스 API 차단으로 이어질 수 있다.
  • 커넥터 단계에서부터 권한 정보(permission map)를 함께 가져와야 하며, 이는 Day 13(Permission-aware Retrieval)에서 직접 이어진다. 또한 오늘 다룬 실패 격리와 재처리 가능성은 Day 9(Incremental Indexing)의 기반이 된다.
