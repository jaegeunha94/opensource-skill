Q3 — 변형 "CSV나 JSON으로 된 표 형태 데이터를 그대로 임베딩해서 벡터 검색에 넣었는데 결과가 이상합니다. 왜 이런 일이 생기고, 어떻게 해결하시겠습니까?"
전제 개념

"이미 구조가 있는" 포맷을 다시 뭉개면 안 되는 이유

Day 4에서 PDF는 구조가 없어서 추론해야 하고, DOCX/PPTX/XLSX는 구조가 XML로 있어서 보존해야 한다고 정리했다. 오늘 다루는 HTML, Markdown, 코드, CSV, JSON, YAML은 전부 후자에 더 가깝다 — 저자나 기계가 이미 명시적인 구조(태그, 헤더, 문법, 스키마)를 만들어 두었다. 그런데 실무에서 가장 흔한 실수는, 이 구조를 활용하지 않고 모든 포맷을 "그냥 텍스트"로 취급해 고정 길이로 잘라버리는 것이다. 이 실수가 포맷마다 어떻게 다른 방식으로 품질을 깎아먹는지 이해하는 것이 오늘의 핵심이다.

HTML: "문서"가 아니라 "신호와 잡음이 뒤섞인 화면"이다

HTML 페이지를 열어서 눈에 보이는 텍스트를 전부 뽑으면, 실제 본문 옆에 내비게이션 메뉴, 광고, 쿠키 동의 배너, 관련 글 목록, 푸터 저작권 문구 같은 본문이 아닌 텍스트("boilerplate", 상용구/잡음)가 함께 딸려 온다. 이걸 그대로 청킹해서 인덱싱하면, 사용자의 질문과 무관한 메뉴 텍스트가 검색 결과에 섞여 나오는 문제가 생긴다. HTML5는 이 문제를 줄이기 위해 <article>(본문), <nav>(내비게이션), <main>(주 콘텐츠), <aside>(부가 콘텐츠) 같은 시맨틱 태그를 제공하지만, 실제로 이 태그를 정확히 쓰는 사이트는 일부일 뿐이다. 그래서 시맨틱 태그를 1차 신호로 쓰고, 태그가 없거나 부정확한 페이지를 위한 휴리스틱 본문 추출기를 함께 두는 것이 실무 표준이다.

흔한 착각: "HTML도 결국 텍스트니까 BeautifulSoup으로 get_text()만 하면 되지 않나요?" — 그렇게 하면 메뉴·광고· 쿠키 배너 텍스트까지 전부 본문과 동일한 취급을 받는다. HTML 파싱의 핵심 난이도는 "텍스트를 뽑는 것"이 아니라 "본문과 잡음을 구분하는 것"이다.

Markdown: 저자가 이미 구조를 선언해 두었다

Markdown은 #, ## 같은 헤더 기호로 저자가 의도한 문서 계층 구조를 명시적으로 남긴다. 이 계층을 무시하고 문자 수 기준으로만 자르면, "3번 섹션의 하위 항목"이라는 문맥이 청크에서 사라진다. 또한 많은 Markdown 문서(특히 위키/Skill 문서, Day 2에서 다룬 SKILL.md 스타일)는 파일 맨 위에 frontmatter라는 YAML 블록(---로 감싸인 부분)을 두어 제목·작성자·태그· 최신화 날짜 같은 메타데이터를 담는다. 이 frontmatter를 본문과 분리해서 제대로 추출하지 않으면, Day 7에서 다룰 provenance(출처 추적) 정보를 통째로 잃어버리게 된다.

코드: 소스 코드는 "자연어 텍스트"가 아니라 "실행 가능한 구조체"다

소스 코드를 산문처럼 500자 단위로 자르면, 함수 정의 중간에서 끊기거나 import 문과 실제 사용 코드가 서로 다른 청크로 분리되는 일이 자주 생긴다. 산문은 문장 중간에서 끊겨도 의미가 일부라도 남지만, 코드는 문법이 깨지는 순간 그 조각 자체가 무의미해지는 경우가 많다 — 여는 괄호만 있고 닫는 괄호가 없는 코드 조각은 임베딩 모델에게도, 사람에게도 온전한 의미 단위가 아니다. 이 문제를 해결하는 도구가 AST(Abstract Syntax Tree, 추상 구문 트리)다. AST는 컴파일러/인터프리터가 코드를 이해하기 위해 만드는 트리 구조로, "이 부분은 함수 정의, 그 안에 이 부분은 for 루프, 그 안에 이 부분은 조건문"처럼 코드의 문법 구조를 계층적으로 표현한다. AST 기반 청킹은 이 트리를 따라가면서 함수·클래스·블록 같은 문법적으로 완결된 단위를 경계로 청크를 자른다.

Tree-sitter는 다양한 프로그래밍 언어에 대해 통일된 방식으로 AST를 얻을 수 있게 해주는 파서 생성 도구로, Neovim·Helix· Zed 같은 실제 에디터의 문법 강조(syntax highlighting) 기능을 구동할 만큼 검증된 도구다. 언어별로 파서를 새로 만들 필요 없이 거의 모든 주요 언어에 대해 같은 방식으로 AST를 얻을 수 있어, 코드 RAG 파이프라인의 사실상 표준 파서로 쓰인다.

CSV/JSON/YAML: "구조화 데이터"는 임베딩이 원래 잘하는 일이 아니다

임베딩 모델은 자연어 문장의 의미적 유사도를 포착하도록 학습된다. 그런데 CSV의 숫자 컬럼, JSON의 ID 값, YAML의 설정 키 같은 데이터는 "의미가 비슷한 것"이 아니라 "정확히 일치하는가"가 중요한 경우가 대부분이다. 예를 들어 상품 코드 "A-1234"와 "A-1235"는 임베딩 공간에서 가까울 수도 있고 안 가까울 수도 있는데, 사용자가 원하는 건 "비슷한 코드"가 아니라 "정확히 그 코드"인 경우가 많다. 그래서 구조화 데이터를 무작정 임베딩부터 하기 전에, "이 데이터가 애초에 임베딩 검색에 맞는 데이터인가"부터 판단해야 한다.

포맷 구조의 출처 핵심 원칙 무시했을 때 생기는 문제
HTML DOM 트리 + 시맨틱 태그(있을 수도, 없을 수도) 본문과 잡음(boilerplate)을 분리 메뉴/광고/푸터 텍스트가 검색 결과 오염
Markdown 헤더 계층 + frontmatter 헤더 경로를 메타데이터로 보존 섹션 문맥 소실, frontmatter의 provenance 유실
코드 프로그래밍 언어 문법(AST) 함수/클래스 단위로 문법적으로 완결된 청크 문법이 깨진 무의미한 조각, 검색·생성 품질 저하
CSV / JSON / YAML 스키마 / 키-값 구조 임베딩 대상과 필터/조회 대상을 분리 숫자·ID 검색 실패, "의미상 비슷하지만 틀린" 결과 반환
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • "텍스트는 다 똑같다"는 착각이 있는지 — HTML의 잡음 문제, 코드의 문법 무결성 문제, 구조화 데이터의 임베딩 부적합 문제를 하나로 뭉뚱그려 생각하면, 포맷별로 완전히 다른 도구와 전략이 필요하다는 것 자체를 이해하지 못한 것으로 본다.
  • 임베딩이 만능이 아니라는 것을 아는지 — 구조화 데이터나 코드에는 임베딩 검색만으로는 부족하고, 구조적 검색(AST 그래프 순회, SQL, 정확 일치 필터)을 결합해야 한다는 판단력을 본다.
  • 이미 존재하는 구조(시맨틱 태그, 헤더, 스키마, AST)를 "활용"하려는 관성이 있는지 — 모든 걸 처음부터 규칙으로 재발명하려 하지 않고, 저자/기계가 이미 만들어둔 신호를 우선 활용하는 실무 감각을 본다.
  • 보안 감각이 있는지 — YAML처럼 겉보기엔 단순한 설정 포맷도 파싱 방식에 따라 원격 코드 실행(RCE) 위험이 있다는 것을 아는지가 시니어와 중급의 갈림길이다.
  • 최신 트렌드를 비판적으로 판단하는지 — 스크린샷 기반 RAG 같은 새로운 접근을 무조건 최신이라고 채택하지 않고, 근거 추적성(citation grounding) 같은 다른 요구사항과의 트레이드오프로 판단하는지를 본다.
실무 연결

2026년 현재 포맷별 도구 지형과 근거

HTML — Trafilatura가 "휴리스틱이 ML을 이긴다"는 반직관적 사례

2026년 기준 HTML 본문 추출의 사실상 표준 오픈소스 도구는 Trafilatura다. ScrapingHub 벤치마크에서 F1 0.958을 기록했고, SIGIR 2023 평가에서도 8개 데이터셋 평균 F1이 가장 높았다. 흥미로운 점은, Trafilatura의 핵심 추출 로직은 머신러닝을 쓰지 않는 규칙 기반(heuristic) 방식이라는 것이다 — 같은 SIGIR 2023 논문은 "휴리스틱 추출기가 가장 우수하고 강건하며, 대형 신경망 모델은 오히려 의외로 성능이 낮다"는 결론을 냈다. Day 4에서 본 "제일 큰 모델이 문서 파싱도 제일 잘하는 건 아니다"라는 패턴이 HTML 본문 추출에서도 반복되는 셈이다.

시맨틱 태그 + JSON-LD를 provenance에 활용하기 — <article>/<main> 같은 시맨틱 태그가 있으면 본문 경계를 잡는 강한 신호로 쓸 수 있고, 페이지의 <script type="application/ld+json"> 안에 있는 JSON-LD(Schema.org 구조화 메타데이터)는 저자, 발행일, 문서 종류 같은 정보를 이미 기계가 읽기 좋은 형태로 제공한다. 이걸 활용하면 Day 7에서 다룰 메타데이터/출처 정보를 별도로 추론하지 않고도 상당 부분 확보할 수 있다.
새로운 트렌드 — 스크린샷 기반 RAG: 최근에는 HTML 파싱 자체를 생략하고 페이지를 스크린샷으로 렌더링한 뒤 비전 모델로 임베딩하는 접근(예: PixelRAG)도 등장했다. 텍스트 추출이 어려운 레이아웃에서 정확도가 더 높게 나온 사례가 보고되지만, 이 방식은 어느 텍스트 조각을 근거로 답했는지 정확히 인용(citation grounding)하기가 훨씬 어렵다는 트레이드오프가 있다. Day 12에서 다룰 근거 인용 요구사항이 강한 시스템이라면 이 트레이드오프를 먼저 따져야 한다.

Markdown — 헤더 계층을 메타데이터로 보존하는 구조 인식 분할

LangChain의 MarkdownHeaderTextSplitter는 #, ##, ### 같은 헤더를 기준으로 문서를 나누면서, 각 청크에 "이 청크가 어떤 헤더 경로 아래에 있었는가"를 메타데이터로 함께 저장한다. 이렇게 하면 나중에 청크만 봐도 "이건 '설치 방법 > Docker 설정' 섹션의 내용"이라는 문맥을 알 수 있어, 근거 인용이나 재검색(re-retrieval) 시에도 유용하다. frontmatter(YAML 메타데이터 블록)는 본문 분할 이전에 별도로 파싱해서 문서 단위 메타데이터로 분리 보관하는 것이 원칙이다 — frontmatter를 본문과 함께 청킹해버리면 모든 청크에 같은 메타데이터 텍스트가 반복돼 정보 밀도만 떨어진다.

코드 — AST 기반 청킹의 설계 원칙과 실증 성과

2026년 발표된 연구 cAST(arXiv:2506.15655)는 AST 기반 청킹을 코드 RAG에 적용해, 검색 성능(RepoEval Recall@5 +4.3점)과 생성 성능(SWE-bench Pass@1 +2.67점)을 모두 개선했고, 특정 언어에 과적합되지 않고 여러 언어에 걸쳐 일관되게 효과가 있음(CrossCodeEval +4.3점)을 보였다. 이 연구가 제시하는 청킹 설계 원칙은 다음과 같다.

1
문법적 완결성(syntactic integrity) — 청크 경계는 항상 완결된 문법 단위(함수, 클래스, 블록)와 일치해야 한다.
2
정보 밀도(information density) — 각 청크는 정해진 크기 예산 안에서 최대한 채워야 한다. 너무 잘게 쪼개면 문맥이 부족해지고, 너무 크면 검색 정밀도가 떨어진다.
3
언어 불변성(language invariance) — 특정 언어에만 통하는 휴리스틱이 아니라, tree-sitter처럼 언어 공통의 AST 순회 방식으로 구현해야 여러 저장소·언어에 일관되게 적용할 수 있다.
4
재조립 가능성(plug-and-play) — 모든 청크를 순서대로 이어붙이면 원본 파일이 그대로 복원돼야 한다. 이는 청킹 과정에서 코드 내용이 유실되거나 중복되지 않았음을 검증하는 기준이 된다.

실무적으로 중요한 디테일 하나는 청크 크기를 줄 수(line count)가 아니라 공백을 제외한 문자 수로 측정해야 한다는 것이다. 같은 줄 수라도 언어나 코딩 스타일에 따라 실제 정보량이 크게 다르기 때문에, 줄 수 기준으로는 언어 간·파일 간 일관된 청크 크기를 보장할 수 없다.

구조적 검색으로 임베딩을 보완하기 — 코드 검색은 임베딩 기반 의미 유사도만으로는 부족하다. "이 함수를 호출하는 곳이 어디인가", "이 클래스가 어떤 모듈을 import하는가" 같은 관계형 질의는 임베딩보다 tree-sitter로 만든 의존성 그래프(호출 그래프, import 그래프)를 순회하는 구조적 검색이 훨씬 정확하다. 시니어 수준의 코드 RAG 설계는 임베딩 검색과 구조적 검색을 하이브리드로 결합한다.

CSV/JSON/YAML — 임베딩 대상과 필터/조회 대상을 나누는 하이브리드 전략

구조화 데이터를 다루는 2026년 실무의 공통된 결론은, 표 형태 데이터를 "의미적으로 서술 가능한 부분"과 "정확히 일치해야 하는 사실적 속성"으로 나누는 것이다. 예를 들어 상품 데이터라면 상품 설명·리뷰 요약 같은 자연어 텍스트는 임베딩해서 의미 검색에 쓰고, 가격·재고 수량·카테고리 코드 같은 값은 벡터가 아니라 필터링 가능한 메타데이터로 별도 보관해 정확 일치·범위 조건 검색에 쓴다. 이렇게 분리하지 않고 표 전체를 그대로 문장으로 이어붙여 임베딩하면, 숫자·ID 같은 정보가 "의미상 비슷하지만 틀린" 결과로 섞여 나오는 의미 희석(semantic dilution) 문제가 생긴다.

사용자가 "지난달 매출 상위 10개 상품"처럼 집계·정렬이 필요한 질의를 하는 경우는 애초에 임베딩 검색으로 풀 문제가 아니다. 이런 경우 자연어 질의를 SQL로 변환하는 Text-to-SQL (NL2SQL)을 별도 경로로 두고, RAG는 스키마 설명이나 비정형 텍스트 질의에, Text-to-SQL은 정확한 집계·필터 질의에 쓰는 하이브리드 아키텍처가 2026년 엔터프라이즈 RAG 플랫폼의 공통 패턴이다.

JSON처럼 중첩 구조가 있는 데이터는 두 가지 접근이 함께 쓰인다. LangChain의 RecursiveJsonSplitter처럼 중첩 구조를 유지하면서 크기 제한 안에서 재귀적으로 나누는 구조 보존형 분할과, 각 JSON 객체를 LLM에게 넘겨 "이 객체가 말하는 바를 자연어로 서술해줘"라고 요청해 임베딩 친화적인 서사(narrative)로 변환하는 접근이다. 후자는 임베딩 검색 품질을 높이지만 변환 비용(LLM 호출)이 들기 때문에, 데이터 규모와 질의 패턴(자연어 질문 위주인지, 정확 조회 위주인지)에 따라 선택해야 한다.

실제로 발생하는 CSV/JSON 파싱 함정

함정 내용
pandas의 청크 단위 dtype 추론 대용량 CSV를 나눠서 읽을 때 pandas는 청크마다 독립적으로 타입을 추론하기 때문에, 같은 컬럼의 같은 값이 어떤 청크에서는 정수로, 다른 청크에서는 문자열로 추론될 수 있다. 이 경우 합쳐진 컬럼은 object dtype에 두 표현이 섞여 들어간다 — low_memory=False 또는 명시적 dtype 지정으로 방지해야 한다.
구분자·인코딩 자동 추론(sep=None) 프로덕션 파이프라인에서 구분자를 자동 추론에 맡기면 pandas가 매번 데이터를 미리 훑어야 해서 오버헤드가 생기고, 불규칙한 행이 섞인 파일에서는 일관성 없는 결과가 나올 수 있다. 인코딩도 UTF-8이 아닌 latin-1/cp1252로 저장된 레거시 파일이 실무에서 흔하다.
pandas 3.0(2026-01-21 릴리스)의 기본 문자열 dtype 변경 pandas 3.0부터 read_csv 등에서 문자열 컬럼의 기본 타입이 기존 NumPy object dtype에서 PyArrow 기반의 전용 문자열 dtype으로 바뀌었다. 새 dtype은 문자열이 아닌 값을 저장하려 하면 에러를 낸다 — object dtype이 "아무 파이썬 객체나 담아도 조용히 통과되던" 관행에 의존한 기존 CSV 수집 코드는 pandas 3.0 업그레이드 시 깨질 수 있다.
YAML 파싱은 원격 코드 실행(RCE) 공격 표면이 될 수 있다. PyYAML의 yaml.load()나 yaml.unsafe_load()는 YAML 문서 안의 !python/object/apply 같은 태그를 만나면 임의의 파이썬 객체를 생성하거나 임의의 함수를 실제로 실행할 수 있다. Day 2에서 다룬 위키/Skill 문서의 YAML frontmatter처럼, 신뢰할 수 없는 외부 기여자가 만든 문서를 파싱하는 수집 파이프라인이라면 이건 이론적 위험이 아니라 실제 공격 표면이다. 반드시 yaml.safe_load()(또는 Loader=yaml.SafeLoader)만 사용해야 하며, "그냥 설정 파일이니까 안전하다"는 가정은 틀렸다.

최근 deprecation/migration — 알아두면 감점을 피할 수 있는 것들

  • pandas 3.0(2026-01-21): 문자열 컬럼 기본 dtype이 object에서 PyArrow 기반 전용 string dtype으로 변경. 기존 파이프라인이 object dtype의 느슨한 동작에 암묵적으로 의존했다면 마이그레이션 가이드를 먼저 확인해야 한다.
  • PyYAML: yaml.load()를 Loader 인자 없이 호출하는 것과 yaml.unsafe_load()는 둘 다 안전하지 않다 — 새 코드에서는 반드시 yaml.safe_load()를 명시적으로 사용해야 한다.
  • langchain-community 저장소가 2026-05-26 아카이브(읽기 전용) 처리됐다(Day 4에서도 확인한 사실) — Markdown/JSON 관련 로더나 스플리터도 langchain_text_splitters, langchain-unstructured 같은 개별 패키지 구조로 옮겨가는 추세를 반영해야 한다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 이 포맷들의 공통점부터 짚습니다. HTML, Markdown, 코드, CSV/JSON/YAML은 전부 이미 나름의 구조를 갖고 있어서, 이 구조를 무시하고 고정 길이로 잘라버리는 게 가장 흔하면서도 치명적인 실수라고 생각합니다. 그래서 포맷마다 '이미 있는 구조를 어떻게 보존할 것인가'를 먼저 봅니다.

HTML은 본문과 광고·메뉴 같은 잡음이 섞여 있는 게 핵심 문제입니다. 시맨틱 태그(article, main 같은)가 있으면 우선 신호로 쓰지만, 실제로는 이런 태그를 정확히 안 쓰는 사이트가 많아서 Trafilatura 같은 휴리스틱 본문 추출기를 함께 둡니다. 흥미롭게도 이 분야는 머신러닝 모델보다 규칙 기반 추출기가 벤치마크에서 더 강건하다는 연구 결과도 있어서, 최신 트렌드라고 무조건 신경망 기반 추출을 고르지는 않습니다.

Markdown은 헤더 계층이 저자가 이미 명시한 구조이기 때문에, 헤더 경로를 메타데이터로 함께 저장하는 구조 인식 분할을 씁니다. 코드는 더 엄격합니다 — 문법이 깨지면 청크 자체가 무의미해지기 때문에, AST를 따라 함수나 클래스처럼 문법적으로 완결된 단위로 청킹합니다. 최근 연구에서도 AST 기반 청킹이 검색과 생성 성능을 모두 유의미하게 올린다는 결과가 나왔고, 언어에 상관없이 일관되게 효과가 있다는 점도 확인됐습니다. 다만 임베딩만으로는 부족해서, '이 함수를 어디서 호출하는가' 같은 관계형 질의는 tree-sitter로 만든 호출 그래프를 순회하는 구조적 검색으로 보완합니다.

CSV나 JSON 같은 구조화 데이터는 접근 자체를 다르게 합니다. 임베딩 모델은 자연어 의미 유사도를 위해 만들어진 도구라서, 숫자나 ID처럼 정확 일치가 중요한 값에는 원래 잘 맞지 않습니다. 그래서 저는 표를 통째로 문장으로 이어붙여 임베딩하기보다, 서술적인 텍스트(설명, 리뷰 요약)는 임베딩하고 가격·수량·코드 같은 값은 필터링 가능한 메타데이터로 분리합니다. '지난달 매출 상위 10개' 같은 집계·정렬 질의는 애초에 임베딩 검색의 문제가 아니라서 Text-to-SQL 경로로 따로 처리합니다.

마지막으로 YAML은 보안 측면에서 특히 조심합니다. YAML 파싱 라이브러리에 따라 load()를 잘못 쓰면 파일 안의 태그로 임의 코드가 실행될 수 있어서, 특히 커뮤니티가 기여하는 위키나 Skill 문서처럼 외부에서 들어온 YAML frontmatter를 다루는 파이프라인에서는 반드시 안전한 로더만 쓰도록 강제합니다."

F1. "AST 기반으로 청킹해도, 함수 하나가 임베딩 모델의 청크 크기 제한보다 훨씬 큰 경우도 있습니다. 그럴 땐 어떻게 하시겠습니까?"

약한 답변: "그냥 그 함수도 문자 수로 강제로 자릅니다."

강한 답변: "먼저 AST를 하향식으로 순회하면서, 큰 노드는 다시 그 하위 노드(내부 블록·중첩 함수)로 나눌 수 있는지 봅니다. 그래도 안 나눠지는 최소 단위가 여전히 크기 제한을 넘으면, 이번엔 반대로 인접한 형제 노드끼리 최대한 채워 넣는 방식으로 조각들을 그리디하게 합쳐 정보 밀도를 높입니다. 이렇게 하면 어느 경우든 문법적으로 완결된 단위를 유지하면서 크기 제한 안에 최대한 맞출 수 있고, 모든 청크를 순서대로 이어붙이면 원본 파일이 그대로 복원되는지로 유실 여부를 검증합니다."

F2. "사용자가 CSV 데이터에 대해 '지난달 매출 상위 10개 상품이 뭐야?'라고 물어보면 어떻게 처리하시겠습니까?"

약한 답변: "행 단위로 임베딩해서 벡터 검색으로 상위 결과를 찾습니다."

강한 답변: "이건 애초에 벡터 검색으로 풀 문제가 아니라고 판단합니다. '지난달', '상위 10개'는 집계·정렬 연산이지 의미 유사도 문제가 아니기 때문입니다. 질의 라우터가 이런 패턴을 감지하면 Text-to-SQL 경로로 보내서 실제 집계 쿼리를 실행하고, 정확한 숫자로 답합니다. RAG/임베딩 검색은 스키마가 뭘 의미하는지 설명하거나, 정형화되지 않은 텍스트 질의를 처리하는 데 남겨둡니다."

F3. "커뮤니티가 기여하는 위키 문서의 YAML frontmatter를 파싱하는 코드를 보안 감사해야 합니다. 무엇부터 확인하시겠습니까?"

강한 답변: "가장 먼저 YAML을 로드하는 모든 코드 경로에서 yaml.safe_load()가 아닌 yaml.load()나 yaml.unsafe_load()가 쓰이는 곳이 있는지 grep으로 전수 조사합니다. 두 번째로, 신뢰할 수 없는 외부 소스(외부 기여자 PR, 외부 URL에서 가져온 문서)에서 온 YAML이 내부 신뢰 소스와 같은 코드 경로를 타는지 확인합니다. 마지막으로 CI에 정적 분석 규칙(예: Semgrep의 YAML unsafe load 규칙)을 추가해서 앞으로 같은 실수가 재발하지 않도록 합니다."

F4. "시맨틱 태그가 하나도 없는(전부 div/span인) 레거시 사이트는 어떻게 다루시겠습니까?"

강한 답변: "시맨틱 태그가 없으면 Trafilatura 같은 휴리스틱 본문 추출기로 폴백합니다. 이런 도구는 텍스트 밀도(링크 텍스트 대비 일반 텍스트 비율), DOM 트리에서의 위치, 반복되는 블록 패턴(메뉴는 여러 페이지에 반복적으로 나타난다는 특징) 같은 신호를 종합해 본문 영역을 추정합니다. 추가로 같은 사이트의 여러 페이지를 함께 분석해서 공통으로 등장하는 블록(내비게이션, 푸터)을 잡음으로 분류하는 사이트 단위 학습도 도움이 됩니다."

F5. "JSON 로그 파일처럼 극도로 반복적인 구조를 청킹할 때 뭘 조심하시겠습니까?"

약한 답변: "로그 한 줄 한 줄을 각각 청크로 만들어 임베딩합니다."

강한 답변: "먼저 이게 애초에 임베딩 검색의 대상인지 다시 판단합니다. 반복적인 로그 레코드는 대부분 정확 일치·범위·시간순 조회가 필요한 데이터라서, 전문 검색 엔진이나 시계열 저장소의 필터/집계 기능이 임베딩보다 훨씬 적합한 경우가 많습니다. 정말 의미 검색이 필요한 필드(에러 메시지 본문 등)만 선별해서 임베딩하고, 나머지 구조화 필드(타임스탬프, 상태 코드, 서비스명)는 메타데이터 필터로 분리합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 포맷마다 "이미 있는 구조"의 종류(태그/헤더/문법/스키마)가 다르다는 것을 먼저 구분한다
  • AST 기반 코드 청킹을 구체적 설계 원칙(문법 완결성, 정보 밀도, 재조립 가능성)으로 설명한다
  • 임베딩이 구조화 데이터에 원래 잘 맞지 않는 이유를 설명하고, 필터링/Text-to-SQL과의 하이브리드를 제시한다
  • YAML의 RCE 위험처럼 보안 측면을 스스로 짚는다
  • 새로운 트렌드(스크린샷 기반 RAG 등)를 근거 추적성 같은 다른 요구사항과의 트레이드오프로 판단한다

약한 답변의 특징

  • "결국 다 텍스트니까 같은 방식으로 자르면 됩니다"라고 답한다
  • AST 기반 청킹을 "AST를 쓴다" 정도로만 언급하고 구체적 메커니즘을 설명하지 못한다
  • CSV/JSON도 그냥 임베딩하면 된다고 답한다
  • YAML을 "설정 파일이라 안전하다"고 가정한다
  • 최신 트렌드를 검증 없이 무조건 채택한다고 답한다
핵심 차이: 강한 답변은 "이미 존재하는 구조를 최대한 활용"하는 원칙을 모든 포맷에 일관되게 적용하면서도, 포맷마다 그 구조의 형태와 위험(보안 포함)이 다르다는 것을 구체적으로 짚는다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "결국 다 텍스트니까 청킹 전략도 하나면 충분하다"

사실: HTML은 잡음 제거, Markdown은 헤더 계층 보존, 코드는 문법 무결성 보존, 구조화 데이터는 임베딩 적합성 판단이라는 서로 다른 문제다. 하나의 고정 길이 분할기로 전부 처리하면 포맷마다 다른 방식으로 품질이 깎인다.

오해 2: "제일 정교한 모델에게 파싱을 맡기면 다 해결된다"

사실: HTML 본문 추출에서는 규칙 기반 도구(Trafilatura)가 신경망 기반 접근보다 벤치마크에서 더 강건하다는 연구 결과가 있다. Day 4에서 본 문서 파싱의 패턴과 마찬가지로, "더 큰/똑똑한 모델"이 항상 더 나은 파서는 아니다.

오해 3: "YAML은 그냥 설정 파일이라 안전하다"

사실: yaml.load()나 yaml.unsafe_load()로 신뢰할 수 없는 YAML을 파싱하면 !python/object/apply 같은 태그를 통해 임의 코드가 실행될 수 있다. 반드시 yaml.safe_load()를 명시적으로 써야 한다.

오해 4: "임베딩 검색이 모든 데이터 형태에 통한다"

사실: 임베딩은 자연어 의미 유사도를 위한 도구다. 숫자·ID·집계·정렬처럼 정확성이 중요한 질의는 임베딩보다 필터링, Text-to-SQL, 구조적 검색(그래프 순회)이 더 적합한 경우가 많다.

실무 체크리스트

이기종 텍스트/구조화 포맷 파이프라인 점검 체크리스트

확인 항목 정상 신호 이상 신호 → 의심 지점

Markdown frontmatter가 본문과 분리되는가 메타데이터로 별도 보관 본문과 함께 청킹 → 정보 밀도 저하, provenance 유실
코드 청킹이 문법 단위를 지키는가 AST 기반, 재조립 시 원본 복원 고정 길이 분할 → 문법 깨진 무의미 청크

자기 점검 Rubric

내 답변 수준 확인하기

미흡 "다 텍스트니까 같은 방식으로 자르면 됩니다" 수준으로만 설명하고, 포맷별 구조 차이를 전혀 언급하지 못함
주니어 HTML에 잡음이 섞인다는 것, 코드를 함부로 자르면 안 된다는 것 정도는 알지만, AST의 구체적 설계 원칙이나 구조화 데이터와 임베딩의 근본적 불일치는 설명하지 못함
중급 포맷별로 다른 전략(시맨틱 태그, 헤더 계층, AST 청킹)을 설명할 수 있고, 구조화 데이터를 무작정 임베딩하면 안 된다는 것을 인식함
시니어 포맷별 구조의 본질(잡음 분리 vs 문맥 보존 vs 문법 무결성 vs 임베딩 부적합)을 구분해 설명하고, AST 기반 청킹을 구체적 설계 원칙과 실증 근거로 뒷받침하며, 구조적 검색·Text-to-SQL과의 하이브리드 전략을 제시하고, YAML RCE 같은 보안 위험까지 스스로 짚으며, follow-up 질문에 구체적 메커니즘으로 대응한다
자가 점검 질문:
1. HTML/Markdown/코드/CSV/JSON/YAML 각각에서 "이미 존재하는 구조"가 무엇이고, 그걸 어떻게 활용해야 하는지 포맷별로 설명할 수 있는가?
2. AST 기반 코드 청킹이 문자 수 기준 분할과 무엇이 다른지, 구체적 설계 원칙과 함께 설명할 수 있는가?
3. 구조화 데이터를 임베딩하면 안 되는 경우와, 그 대안(필터링, Text-to-SQL, 구조적 검색)을 설명할 수 있는가?
4. YAML 파싱이 왜 보안 위험이 될 수 있는지, 어떻게 방지하는지 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • HTML, Markdown, 코드, CSV/JSON/YAML은 전부 이미 나름의 구조를 갖고 있다 — 이 구조를 무시하고 고정 길이로 자르는 것이 가장 흔한 실수다.
  • HTML은 본문과 잡음(boilerplate)을 분리하는 것이 핵심이며, 규칙 기반 추출기(Trafilatura)가 신경망 기반보다 벤치마크에서 더 강건하다는 반직관적 사실이 있다.
  • 코드는 AST를 따라 문법적으로 완결된 단위로 청킹해야 하며, cAST 같은 최신 연구가 이 접근의 검색·생성 성능 개선을 실증했다. 임베딩은 관계형 질의(호출 그래프 등)를 대체하지 못해 구조적 검색과 하이브리드로 결합해야 한다.
  • CSV/JSON/YAML 같은 구조화 데이터는 임베딩이 원래 잘 맞는 도구가 아니다 — 서술형 텍스트만 임베딩하고, 정확 일치가 필요한 값은 필터/메타데이터로, 집계·정렬 질의는 Text-to-SQL로 분리해야 한다.
  • YAML 파싱은 보안 이슈다 — yaml.load()/ unsafe_load()는 RCE로 이어질 수 있어 반드시 safe_load()를 써야 하며, 이는 외부 기여자의 문서를 다루는 파이프라인에서 특히 중요하다.
  • 오늘 다룬 시맨틱 태그/메타데이터는 Day 7(Provenance)의 기반이 되고, AST/구조 인식 청킹은 Day 8(Semantic Chunking)에서 더 깊이 다루며, YAML 보안 이슈는 Day 13(Permission-aware Retrieval과 보안)과 Day 12(Prompt Injection 방어)로 이어진다.
