1. 왜 "종합 케이스 스터디"가 이 트랙의 마지막 인터뷰 주제인가

Day 1~9는 인증/인가, 세션/JWT, SQL Injection, XSS/CSRF, 파일 업로드/SSRF, API 보안, 비밀번호/Secret 관리, Secure Coding 원칙을 하나씩 분리해서 다뤘다. 하지만 실제 침해 사고는 이 카테고리 중 하나만 뚫려서 발생하지 않는다. 실무에서 큰 사고는 거의 항상 여러 개의 방어가 동시에, 순차적으로 실패했을 때 일어난다. 시니어 면접에서 "지금까지 배운 걸 종합해서 설계해보라"거나 "이런 사고가 나면 어떻게 하겠냐"는 오픈형 질문을 던지는 이유는, 개별 지식을 나열하는 사람과 여러 취약점이 어떻게 사슬처럼 연결되는지(attack chain)를 꿰뚫어 보는 사람을 구분하기 위해서다.

이 주제가 평가하는 것
  • 단일 취약점이 아닌 공격 체인(attack chain)으로 사고하는가 — "SSRF가 있었다"로 끝내지 않고, "SSRF로 무엇을 얻었고, 그걸로 다음 단계에서 무엇을 했는가"까지 추적하는가.
  • 사고를 개인의 실수가 아니라 구조적 원인으로 재구성하는가 — "누가 설정을 잘못했다"가 아니라 "왜 그 실수가 프로덕션까지 아무 제지 없이 흘러갔는가"를 묻는가.
  • 새로운 상황에서도 Day 1~9의 원칙을 스스로 조합해 적용할 수 있는가 — 암기한 답을 재생하는 게 아니라, 처음 보는 시스템 설계 앞에서 위협 모델링 → 방어 계층 → trade-off 순서로 사고를 재구성하는가.
흔한 오해 — "실제 침해 사례는 CVE 번호와 회사 이름을 외우는 문제다"

면접관이 궁금한 건 당신이 Capital One 사고의 CVE 번호를 외웠는지가 아니다. 회사 이름과 연도를 몰라도, "공개 클라우드에 배포된 웹 애플리케이션 방화벽에 SSRF 취약점이 있었고, 그 뒤에 있던 인스턴스의 IAM 역할이 과도한 권한을 갖고 있었다면 무슨 일이 벌어질까?"라는 질문에 스스로 공격 체인을 재구성할 수 있으면 충분하다. 이 레슨의 목적은 사례 암기가 아니라, 사례를 통해 "방어 계층이 어떻게 도미노처럼 무너지는지"를 보는 눈을 훈련하는 것이다.

2. 통합 분석 프레임워크 — 위협 모델링 → 근본 원인 → 재발 방지

Day 1에서 다룬 STRIDE 위협 모델링을 실제 침해 사고 "사후 분석"에도 그대로 적용할 수 있다. 세 케이스 모두 아래의 3단계 프레임워크로 분석한다. 면접에서 오픈형 질문을 받았을 때도 이 순서를 그대로 말로 풀어내면, 즉흥적으로 떠오르는 대로 답하는 사람과 확연히 다른 인상을 준다.

3단계 분석 프레임워크
  • 1단계 — 공격 체인 재구성(Attack Chain Reconstruction) — 공격자가 최초 진입점에서 시작해 어떤 단계를 거쳐 최종 목표(데이터 유출, 시스템 장악)에 도달했는지 순서대로 나열한다. "SSRF → 메타데이터 탈취 → S3 접근 → 대량 데이터 유출"처럼 화살표로 연결한다.
  • 2단계 — 각 단계별 실패한 방어 계층 식별 — 공격 체인의 각 단계마다 "어떤 방어가 있었어야 막혔을까"를 짚는다. 하나의 방어만 있었다면 뚫렸을 때 즉시 전체 침해로 이어진다 — 이것이 Day 1의 Defense in Depth가 실무에서 왜 중요한지를 보여주는 지점이다.
  • 3단계 — 근본 원인을 "기술적 결함"과 "조직/프로세스 결함"으로 분리 — 취약점 자체(기술)와, 그 취약점이 왜 발견되지 못하고 방치됐는지(프로세스: 모니터링 부재, 리뷰 프로세스 부재, 권한 관리 정책 부재)를 나눠서 본다. 재발 방지 설계는 반드시 두 축을 모두 다뤄야 한다.
왜 이 순서가 중요한가

기술적 결함만 고치면("이 파라미터에 검증 로직 추가") 같은 유형의 새로운 취약점이 다른 곳에서 또 나온다. 프로세스 결함만 고치면("리뷰 절차 강화") 이미 존재하는 유사 취약점을 못 찾는다. 시니어 면접에서 좋은 답변은 "이 구멍을 막았습니다"가 아니라 "이 구멍이 왜 생겼고, 같은 유형의 구멍이 다른 곳에도 있을 가능성을 어떻게 체계적으로 찾아낼 것인가"까지 다룬다.

3. 케이스 1 — Capital One (2019) — SSRF + IAM 과다 권한

2019년 Capital One은 약 1억 6백만 명의 고객 데이터(이름, 주소, 신용점수, 은행 계좌 일부, 사회보장번호 일부)가 유출되는 사고를 겪었다. 이 사고는 Day 5(SSRF)와 Day 1(최소 권한)이 어떻게 연결되는지를 보여주는 가장 대표적인 실무 사례다.

공격 체인 재구성

  • 1단계 — SSRF 취약점 악용: 클라우드에 배포된 웹 애플리케이션 방화벽(WAF)에 오설정된 SSRF 취약점이 있었다. 공격자는 이 WAF를 거쳐, WAF가 실행 중인 클라우드 인스턴스 자신에게 요청을 보내도록 유도했다.
  • 2단계 — 클라우드 메타데이터 엔드포인트 탈취: Day 5에서 다룬 클라우드 메타데이터 엔드포인트(169.254.169.254)로 SSRF 요청을 보내, 그 인스턴스에 연결된 IAM 역할의 임시 자격증명(access key, secret key, session token)을 그대로 응답으로 받아냈다.
  • 3단계 — 과도한 권한을 가진 IAM 역할 악용: 탈취한 자격증명이 속한 IAM 역할은 필요 이상으로 수백 개의 S3 버킷을 나열하고 읽을 수 있는 권한을 갖고 있었다. 공격자는 이 자격증명으로 직접 S3 API를 호출해 대량의 고객 데이터를 다운로드했다.
  • 4단계 — 장기간 미탐지: 비정상적인 대량 S3 접근/다운로드 패턴이 수개월간 로그에 남아 있었지만, 이를 탐지하고 경고하는 모니터링 체계가 실질적으로 작동하지 않아 사고가 외부 제보로 뒤늦게 드러났다.

각 단계별 실패한 방어 계층

단계별 무너진 방어
  • 1단계 방어 실패 — SSRF 방어 부재: Day 5에서 다룬 대로, 서버가 사용자 제어 URL로 요청을 보낼 때 목적지를 검증하지 않았다. 아웃바운드 요청을 애플리케이션 서브넷 내부로 제한하는 네트워크 세그멘테이션도 없었다.
  • 2단계 방어 실패 — 메타데이터 엔드포인트 보호 부재: 사고 당시 클라우드 제공자는 IMDSv1을 기본값으로 사용했다. IMDSv1은 단순 HTTP GET 요청만으로 메타데이터를 반환하기 때문에 SSRF에 그대로 노출된다. IMDSv2(토큰 기반, hop limit 제한)였다면 단순 SSRF만으로는 토큰을 얻을 수 없었을 가능성이 높다.
  • 3단계 방어 실패 — 최소 권한 원칙 위반: 이것이 이 사고의 가장 근본적인 원인이다. WAF를 실행하는 인스턴스가 왜 수백 개의 S3 버킷을 나열하고 읽을 수 있어야 하는가? 이 역할은 자신의 업무(WAF 로그 기록 등)에 필요한 범위로 좁혀졌어야 했다. SSRF 자체를 완전히 막는 건 현실적으로 어렵지만(Day 5, Day 9에서 다룬 "뚫릴 수 있다는 전제"), IAM 권한이 좁았다면 탈취된 자격증명으로 할 수 있는 일이 극히 제한적이었을 것이다.
  • 4단계 방어 실패 — 이상 탐지/로깅 미작동: CloudTrail 로그 자체는 존재했지만, 이를 실시간으로 분석해 "평소와 다른 대량 S3 접근"을 경고하는 체계가 실질적으로 작동하지 않았다.

근본 원인 분리

기술적 결함
  • WAF의 SSRF 취약점(패치 가능한 소프트웨어 결함)
  • IMDSv1 사용(설정 가능한 보안 기본값 문제)
  • IAM 정책이 리소스 단위로 세분화되지 않음
조직/프로세스 결함
  • IAM 역할 권한을 정기적으로 검토·축소하는 프로세스 부재
  • 클라우드 설정(WAF, 메타데이터 버전)에 대한 보안 기준선(baseline) 감사 부재
  • 비정상 데이터 접근 패턴에 대한 실시간 알림 체계 미흡
흔한 오해 — "SSRF만 막았으면 이 사고는 없었을 것이다"

기술적으로는 맞지만 시니어 레벨 사고방식은 아니다. Day 5, Day 9에서 반복적으로 강조했듯, "이 취약점 하나를 완벽히 막을 수 있다"는 전제로 설계하면 안 된다. 진짜 시니어급 인사이트는 "SSRF는 언젠가 어디선가 뚫릴 수 있다는 전제 아래, IAM 권한을 최소화해서 뚫려도 피해가 S3 전체가 아니라 특정 몇 개 버킷으로 제한되도록 설계했어야 한다"는 것이다. 이게 바로 Day 1의 Defense in Depth와 최소 권한 원칙이 만나는 지점이다.

4. 케이스 2 — Uber (2016) — 유출된 자격증명과 은폐

2016년 Uber는 5,700만 명의 승객/운전자 데이터가 유출된 사고를 겪었다. 이 사고는 Day 8(Secret 관리)이 실무에서 왜 그렇게 중요한지, 그리고 보안 사고 대응이 기술 문제를 넘어 조직의 정직성/거버넌스 문제로 확장된다는 걸 보여준다.

공격 체인 재구성

  • 1단계 — 사설 GitHub 저장소에 하드코딩된 AWS 자격증명: Uber 엔지니어가 사설(비공개로 알려졌던) GitHub 저장소에 AWS access key를 코드에 하드코딩해서 커밋했다. Day 8에서 다룬 "환경 변수/Secret이 git 커밋으로 노출되는 경로"의 실제 사례다.
  • 2단계 — 저장소 접근 권한 확보: 공격자가 어떤 경로로든(유출된 접근 권한, 협업자 계정 등) 이 저장소에 접근할 수 있었고, 하드코딩된 키를 발견했다.
  • 3단계 — AWS 자격증명으로 S3 접근: 탈취한 access key로 AWS API를 직접 호출해, 사용자 데이터가 저장된 S3 버킷에 접근하고 대량의 개인정보를 다운로드했다.
  • 4단계 — 사고 은폐와 부적절한 대응: Uber는 이 사고를 즉시 공개하지 않고, 공격자에게 10만 달러를 지급하며 침해 사실을 숨기고 데이터를 삭제했다는 확인서를 요구하는 방식으로 대응했다. 이는 사실상 버그 바운티가 아니라 은폐성 지불로 간주됐고, 이후 규제 기관 조사와 법적 제재로 이어졌다.

각 단계별 실패한 방어 계층

단계별 무너진 방어
  • 1단계 방어 실패 — Secret이 코드/버전관리 시스템에 존재: Day 8에서 다룬 원칙 — "Secret은 코드에 절대 하드코딩하지 않고 Vault/Secrets Manager 같은 전용 저장소에서 런타임에 주입받아야 한다"는 기본 원칙이 지켜지지 않았다. 프리커밋 훅으로 Secret 패턴을 스캔하는 자동화 장치도 없었다.
  • 2단계 방어 실패 — 저장소 접근 통제 미흡: "사설 저장소니까 안전하다"는 가정 자체가 문제다. 사설 저장소도 협업자 계정 침해, 실수로 인한 권한 부여 확대 등으로 노출될 수 있다는 전제가 없었다.
  • 3단계 방어 실패 — 자격증명 노출 시 자동 무효화(rotation) 부재: Day 8에서 다룬 "무중단 로테이션" 체계나, 노출이 의심되는 즉시 키를 자동으로 폐기하는 이상 탐지 체계가 없었다. 노출된 키가 장기간 유효한 상태로 남아 있었다.
  • 4단계 방어 실패 — 사고 대응 거버넌스 부재: 이건 기술적 방어가 아니라 조직 프로세스의 실패다. 침해 사고 발생 시 규정에 따라 투명하게 보고하고 대응하는 표준 절차(incident response policy)가 없었거나 지켜지지 않았다.

근본 원인 분리

기술적 결함
  • Secret 하드코딩(코드 레벨 관행 문제)
  • Secret 노출 탐지/자동 폐기 자동화 부재
  • 버전관리 시스템에 대한 Secret 스캐닝 도구 미도입
조직/프로세스 결함
  • Secret 관리 정책이 실질적으로 강제되지 않음(정책은 있었어도 자동화된 강제 장치 부재)
  • 사고 대응/공개 절차가 규정보다 개인 판단에 의존
  • 보안 사고를 "숨기면 해결된다"고 보는 조직 문화
이 케이스가 주는 시니어 레벨 시사점

기술 지식만으로는 이 두 번째 실패(은폐)를 막을 수 없다. 시니어 개발자/엔지니어링 리더는 "기술적으로 사고를 막는 것"뿐 아니라 "사고가 나면 투명하게 보고되고 신속히 대응되는 조직 프로세스를 설계하는 것"까지 책임 범위로 인식해야 한다. 면접에서 이 사례를 언급할 때, 기술적 원인(하드코딩된 키)뿐 아니라 대응 거버넌스의 실패까지 짚으면 훨씬 성숙한 답변으로 평가받는다.

5. 케이스 3 — 대형 통신사 API 침해 — BOLA와 무제한 스크래핑

2021년 전후 여러 대형 통신사/소셜미디어 서비스에서 반복적으로 발생한 패턴이다 — 공개 API 또는 인증된 API의 엔드포인트가 객체 단위 인가(BOLA/IDOR) 없이 순차적인 ID나 전화번호로 사용자 데이터를 조회할 수 있게 열려 있었고, 공격자가 이를 자동화된 스크립트로 대량 순회(scraping)해 수천만 건의 개인정보를 수집한 사고 유형이다. 이는 Day 6(API 보안)에서 다룬 BOLA가 실제로 어떻게 대규모 침해로 이어지는지 보여준다.

공격 체인 재구성

  • 1단계 — 예측 가능한/순차적인 리소스 식별자: 사용자 계정이나 조회 대상이 순차 증가하는 숫자 ID이거나, 전화번호처럼 유한하고 예측 가능한 값으로 식별됐다.
  • 2단계 — 엔드포인트 단위 인가는 있지만 객체 단위 인가가 없음: "로그인한 사용자만 이 API를 호출할 수 있다"는 인증/엔드포인트 접근 통제는 있었지만, "이 사용자가 조회하려는 그 객체(다른 사용자의 데이터)에 대한 소유권/권한이 있는가"는 검증하지 않았다. Day 6에서 다룬 BOLA의 정의 그대로다.
  • 3단계 — Rate Limiting 부재 또는 우회 가능: 정상적인 단건 조회 API였다면 피해가 제한적이었겠지만, 같은 IP/계정에서 초당 수백~수천 건을 순회 조회해도 이를 막는 속도 제한이 없거나, IP 로테이션으로 쉽게 우회할 수 있었다.
  • 4단계 — 대량 자동화 스크래핑 완료까지 미탐지: 수백만 건이 순차적으로 조회되는 이상 트래픽 패턴이 수 주에서 수개월간 이어졌지만 임계치 기반 이상 탐지가 작동하지 않았다.

각 단계별 실패한 방어 계층

단계별 무너진 방어
  • 1단계 방어 실패 — 식별자 설계: Day 6에서 다룬 대로 "UUID로 바꾸면 근본 해결"이 아니지만, 최소한 예측 불가능한 식별자는 무차별 순회(enumeration) 난이도를 높이는 보조 방어는 된다.
  • 2단계 방어 실패 — 객체 단위 인가 로직 부재(근본 원인): 이것이 가장 핵심적인 실패다. API가 "요청자가 이 리소스의 소유자/권한 보유자인가"를 서버 측에서 매 요청마다 검증해야 하는데, 이 검증 자체가 코드에 존재하지 않았다.
  • 3단계 방어 실패 — Rate Limiting/이상 트래픽 방어 부재: Day 6에서 다룬 IP+계정 복합 키 기반 Rate Limiting과, 짧은 시간에 다수의 서로 다른 리소스를 조회하는 패턴(스크래핑 시그니처)에 대한 탐지가 없었다.
  • 4단계 방어 실패 — 데이터 접근 패턴 모니터링 부재: "평소 사용자 한 명이 하루에 조회하는 건수"의 기준선(baseline)이 없어서, 이상치를 탐지할 기준 자체가 없었다.

근본 원인 분리

기술적 결함
  • 객체 단위 인가 로직 미구현(코드 결함)
  • Rate Limiting 정책 부재/우회 가능한 설계
  • 이상 트래픽 탐지 룰 부재
조직/프로세스 결함
  • 새 API 엔드포인트 추가 시 인가 검증을 강제하는 리뷰/테스트 프로세스 부재 (Day 9에서 다룬 표준 테스트 템플릿의 부재)
  • API 인벤토리 관리 미흡 — 어떤 엔드포인트가 존재하는지 조직이 전체 파악을 못 함(shadow API)
  • 대량 데이터 접근에 대한 비즈니스 임계치 정책 부재
흔한 오해 — "속도 제한만 강하게 걸면 스크래핑을 막을 수 있다"

Rate Limiting은 공격을 느리게 만드는 방어이지, 근본 방어가 아니다. 공격자는 IP를 로테이션하거나, 시간을 충분히 들여 임계치 아래로 요청을 분산시키면 결국 같은 데이터를 다 수집할 수 있다. 근본 방어는 객체 단위 인가 — "이 요청자가 이 데이터를 볼 권한이 있는가"를 애초에 확인하는 것이다. Rate Limiting은 이 근본 방어와 함께 배치되는 보조 계층으로 이해해야 하며, Rate Limiting 하나만으로 인가 부재를 대체할 수 있다고 답하면 시니어 면접에서는 감점 요인이 된다.

6. 면접 질문 1 — "신규 마이크로서비스에 대해 위협 모델링을 해보세요"

면접 질문
"사용자가 프로필 이미지를 업로드하면, 백엔드가 이를 리사이징해서 CDN에 올리고, 다른 마이크로서비스가 이 이미지 URL을 참조해 알림을 보내는 새 기능을 설계 중입니다. 이 시스템에 대해 위협 모델링을 해보세요."

왜 면접에서 묻는가?

실제 시스템 설계 앞에서 Day 1의 STRIDE 프레임워크와 Day 1~9에서 배운 구체적 지식을 스스로 연결할 수 있는지를 본다. 이 질문에는 "정답"이 하나가 아니다 — 면접관은 답의 정확성보다 사고 체계의 완결성을 평가한다.

답변 사고 순서

  1. 데이터/신뢰 경계를 먼저 그린다 — 사용자 → 업로드 API → 리사이징 처리 → CDN → 알림 서비스, 각 구간에서 신뢰 수준이 바뀌는 지점을 표시한다.
  2. 각 구간에 STRIDE를 적용한다 — Day 1에서 배운 Spoofing/Tampering/Repudiation/Information Disclosure/DoS/Elevation of Privilege를 구간별로 적용한다.
  3. 관련 세부 지식을 연결한다 — 파일 업로드(Day 5), 서비스 간 통신 인가(Day 6), 이미지 처리 라이브러리 취약점(Day 5) 등 구체 지식을 매핑한다.
강한 답변 예시

"먼저 데이터 흐름을 구간별로 나누겠습니다. 사용자 → 업로드 API가 첫 신뢰 경계입니다. 여기서는 Day 5에서 다룬 대로 확장자가 아니라 매직 바이트로 실제 파일 형식을 검증하고, 업로드 파일을 실행 가능한 경로가 아닌 별도 스토리지에 서버가 생성한 파일명으로 저장해야 합니다. 다음으로 리사이징 처리 단계 — 이미지 처리 라이브러리 자체의 알려진 취약점(예: 특정 이미지 포맷 파싱 시 버퍼 오버플로우)이 공격 벡터가 될 수 있어서, 라이브러리를 최신 버전으로 유지하고 리사이징 프로세스를 별도의 격리된 샌드박스나 컨테이너에서 실행해서 만약 뚫리더라도 피해 반경을 제한하겠습니다. 그다음 CDN 업로드 단계는 Information Disclosure 관점에서, 비공개여야 하는 이미지가 공개 CDN URL로 추측 가능한 경로에 올라가지 않도록 URL을 예측 불가능하게 만들고 필요하면 서명된 URL(signed URL)을 씁니다. 마지막으로 알림 서비스가 이미지 URL을 참조하는 부분은 서비스 간 통신 문제로, Day 6에서 다룬 대로 알림 서비스가 임의의 외부 URL을 그대로 받아 처리하면 SSRF로 이어질 수 있으니, 이미지 서비스가 발급한 내부 신뢰 토큰이나 화이트리스트된 도메인만 허용하는 방식으로 제한하겠습니다."

약한 답변 예시

"업로드할 때 파일 크기 제한하고, XSS 막고, 인증 확인하면 될 것 같습니다."

왜 약한가? 나열식 답변이다. 데이터가 시스템을 통과하는 흐름을 따라가지 않고 떠오르는 보안 키워드를 그냥 붙였다. 구간별로 어떤 위협이 있고 왜 그 방어가 그 지점에 필요한지 연결하지 못했고, 이 시스템에 특화된 위협(이미지 처리 라이브러리 취약점, 서비스 간 URL 신뢰 문제)은 전혀 언급하지 못했다.

Senior-level Follow-up 질문들

Follow-up: "이 중에서 시간이 부족해 하나만 먼저 처리해야 한다면 무엇을 우선하시겠습니까?"

모범 답변 방향: "업로드 파일 검증을 최우선으로 하겠습니다. 이유는 이 지점이 외부 공격자가 임의의 바이트를 시스템에 주입할 수 있는 최초 진입점이기 때문입니다. 여기서 막히면 뒤의 리사이징, CDN, 알림 단계로 공격이 전파될 가능성 자체가 크게 줄어듭니다. 반대로 뒤쪽 단계부터 방어하면, 앞 단계에서 이미 악성 파일이 시스템 내부로 들어온 상태이기 때문에 위험이 남아있는 채로 여러 겹의 임시방편을 쌓는 셈이 됩니다. 공격 체인에서 가장 이른 단계를 막는 것이 투자 대비 효과가 가장 큽니다."

7. 면접 질문 2 — "Capital One과 유사한 사고가 우리 시스템에서 발생했습니다. 대응하세요"

면접 질문
"새벽에 알림이 왔습니다. 한 서비스 인스턴스의 IAM 자격증명으로 평소보다 수백 배 많은 S3 다운로드 요청이 발생했습니다. 지금부터 무엇을 어떤 순서로 하시겠습니까?"

왜 면접에서 묻는가?

사고 발생 직후의 실무적 판단력을 본다. 침해를 사전에 막는 설계 능력과, 침해가 이미 발생한 상황에서 피해를 최소화하며 침착하게 대응하는 능력은 다른 역량이다. "원인을 분석하겠습니다"로 시작하는 답은 순서가 틀렸다는 신호를 준다.

답변 사고 순서

  1. 즉시 억제(Containment) — 원인 분석보다 먼저, 피해가 더 커지는 것을 막는다.
  2. 증거 보존과 범위 파악 — 무엇이, 언제부터, 얼마나 유출됐는지 확인한다.
  3. 근본 원인 분석과 재발 방지 — 이 레슨의 프레임워크(기술적/조직적 원인 분리)를 적용한다.
  4. 보고와 커뮤니케이션 — 내부 이해관계자, 필요시 규제 기관/고객에게 투명하게 알린다(Uber 사례의 교훈).
강한 답변 예시

"가장 먼저 원인 분석이 아니라 억제부터 하겠습니다. 해당 IAM 자격증명(access key)을 즉시 폐기(revoke)하고, 해당 역할이 접근 가능한 S3 버킷에 대한 접근을 임시로 제한하거나 버킷 정책을 더 좁혀서 추가 유출을 막습니다. 이 단계에서 서비스가 일시적으로 영향을 받더라도, 유출이 계속되는 것보다는 낫다고 판단합니다. 그다음 CloudTrail 로그를 보존하고(추후 분석과 법적 대응을 위해 로그를 건드리지 않는 게 중요합니다), 이 자격증명으로 실제로 어떤 객체가, 언제부터, 얼마나 다운로드됐는지 범위를 파악합니다. 범위가 확인되면 Day 10에서 다룬 프레임워크대로 근본 원인을 기술적 원인(왜 이 역할이 이런 권한을 가지고 있었는가)과 프로세스 원인(왜 이 권한 설정이 사전에 감지되지 않았는가)으로 나눠 분석합니다. 마지막으로, 영향받은 고객 수와 데이터 종류가 확인되는 대로 법무/컴플라이언스 팀과 함께 규제 요건에 따른 통지 절차를 진행합니다. Uber 사례처럼 사고를 숨기거나 축소해서 대응하는 것은 단기적으로도 위험하고 장기적으로는 신뢰와 법적 책임 모두에서 훨씬 큰 손실로 돌아온다는 걸 이번 레슨에서 확인했습니다."

약한 답변 예시

"먼저 로그를 분석해서 어떻게 이런 일이 발생했는지 원인을 찾겠습니다."

왜 약한가? 순서가 틀렸다. 원인 분석에 들어가는 동안에도 유출은 계속될 수 있다. 사고 대응에서는 "왜 이런 일이 생겼는가"보다 "지금 당장 더 커지는 걸 어떻게 막는가"가 항상 먼저다. 이 답변은 침해 대응 경험이 없거나, 사고 대응을 이론으로만 알고 있다는 인상을 준다.

Senior-level Follow-up 질문들

Follow-up: "자격증명을 즉시 폐기했더니 그 자격증명을 쓰던 정상 서비스도 같이 장애가 났습니다. 이건 괜찮은 트레이드오프인가요?"

모범 답변 방향: "이 시점에서는 괜찮은 트레이드오프라고 봅니다. 진행 중인 대량 데이터 유출을 막는 것이 일시적 서비스 장애보다 우선순위가 높습니다. 다만 이게 '괜찮은 설계'라는 뜻은 아닙니다 — 이번 일을 계기로, 하나의 자격증명이 여러 목적에 광범위하게 쓰이는 구조 자체가 문제라는 걸 재발 방지 설계에 반영해야 합니다. 서비스별로 자격증명을 세분화해서, 앞으로는 자격증명 하나를 폐기해도 다른 서비스에 영향을 주지 않도록 만드는 것이 이번 사고에서 얻어야 할 구조적 교훈입니다. 이게 바로 Day 1의 최소 권한 원칙과 서비스 경계 설계가 사고 대응 능력에도 직접 영향을 준다는 증거입니다."

8. 면접 질문 3 — "신규 결제 API를 처음부터 설계한다면, 보안을 어떻게 반영하시겠습니까?"

면접 질문
"새로운 결제 승인 API를 처음부터 설계하고 있습니다. 인증, 인가, 데이터 보호, 입력 검증, 장애 대응까지 전체를 아우르는 보안 설계를 간략히 제시해보세요."

왜 면접에서 묻는가?

이 질문은 이 트랙 전체의 최종 통합 시험이다. 인터뷰어는 특정 취약점 하나를 깊게 캐는 대신, Day 1~9의 지식을 하나의 완결된 설계로 조립할 수 있는지를 짧은 시간 안에 확인하려 한다. 답변의 길이보다 다루는 레이어의 폭과, 그 안에서 우선순위를 명확히 제시하는지가 평가 기준이다.

답변 사고 순서

  1. 레이어를 먼저 선언한다 — "인증/인가 → 입력/데이터 → 트래픽/가용성 → 운영/거버넌스" 같은 구조를 먼저 말해서 나열식이 아님을 보여준다.
  2. 각 레이어에서 이 시스템(결제)에 특화된 판단을 제시한다 — 일반론이 아니라 "결제"라는 도메인의 민감도를 반영한다.
  3. 가장 중요한 trade-off 하나를 짚는다 — 결제는 보안과 가용성이 특히 강하게 충돌하는 도메인이므로, 이 균형을 어떻게 잡을지 스스로 언급한다.
강한 답변 예시

"네 개의 레이어로 나눠 말씀드리겠습니다. 첫째, 인증/인가 레이어 — 결제 API는 세션이 아니라 짧은 만료시간의 Access Token(Day 2)을 쓰고, 모든 결제 요청은 요청자가 그 주문/계좌의 소유자인지 객체 단위로 검증합니다(Day 6의 BOLA 방지). 둘째, 입력/데이터 레이어 — 결제 금액, 계좌 정보는 화이트리스트 기반으로 엄격하게 검증하고(Day 9), 저장 시 카드 정보처럼 민감한 데이터는 필드 단위 암호화를 적용하며, DB 계정은 결제 서비스 전용으로 최소 권한만 부여합니다(Day 1, Day 9). 셋째, 트래픽/가용성 레이어 — Rate Limiting을 IP와 계정 복합 키로 적용해 무차별 결제 시도를 막고(Day 6), 외부 사기 탐지 서비스처럼 결제 승인에 관여하는 외부 의존성이 장애일 때는 원칙적으로 Fail-Closed로 두되, 거래 금액과 사용자 신뢰도 기반으로 리스크를 세분화한 절충안을 둡니다(Day 9). 넷째, 운영/거버넌스 레이어 — 결제 관련 코드 변경은 CODEOWNERS로 시니어 리뷰를 강제하고, Secret은 Vault에서 동적으로 발급하며 정기 로테이션하고(Day 8), 모든 결제 API 호출과 실패를 감사 로그로 남겨 이상 패턴을 탐지합니다. 이 중에서 가장 중요한 trade-off는 가용성과 보안의 충돌이라고 생각합니다 — 결제는 매출과 직결되기 때문에 모든 곳에 Fail-Closed를 걸면 사소한 장애가 매출 중단으로 번지므로, 리스크가 높은 경로(신규 사용자, 고액 거래)만 엄격하게, 리스크가 낮은 경로는 조건부로 유연하게 설계하는 게 실무적으로 맞다고 봅니다."

약한 답변 예시

"HTTPS 쓰고, 비밀번호는 해시로 저장하고, SQL Injection 막으면 될 것 같습니다."

왜 약한가? 결제라는 도메인의 특수성이 전혀 반영되지 않은 일반론이다. HTTPS나 해시 저장은 어느 시스템에나 해당하는 기본값일 뿐, 결제 API가 다른 API보다 왜 더 엄격한 인가·모니터링·장애 정책이 필요한지에 대한 판단이 없다. 또한 레이어 구조 없이 떠오르는 대로 나열해서, 우선순위나 서로의 연결 관계를 전혀 보여주지 못했다.

Senior-level Follow-up 질문들

Follow-up: "이 설계를 팀에 관철시키려는데, '일정이 촉박하니 보안은 나중에 추가하자'는 PM의 반대에 부딪혔습니다. 어떻게 설득하시겠습니까?"

모범 답변 방향: "'보안을 나중에 추가한다'는 접근 자체가 실무에서 왜 더 비싸지는지를 구체적 비용으로 설명하겠습니다. 인증/인가 구조나 데이터 암호화 방식은 시스템 설계의 근간이라서, 나중에 추가하려면 이미 만들어진 API 계약과 데이터 스키마를 다시 바꿔야 하는 경우가 많습니다. 반면 결제 시스템에서 사고가 나면 Capital One이나 Uber 사례처럼 벌금, 소송, 고객 이탈이라는 훨씬 큰 비용이 발생합니다. 다만 무조건 '전부 지금 다 하자'가 아니라, 이번 레슨에서 다룬 것처럼 객체 단위 인가, 입력 검증, 최소 권한 같은 '설계에 내장돼야 하는 항목'과, 이상 탐지 대시보드 고도화처럼 '출시 후에도 점진적으로 강화 가능한 항목'을 구분해서, 전자만 최소 범위로 먼저 확보하고 후자는 후속 스프린트로 미루는 절충안을 제시하겠습니다."

9. 자기 점검 Rubric — 트랙 전체 종합

아래 기준으로 Day 1~10 전체를 점검한다. 각 항목을 다른 사람에게 설명할 수 있어야 진짜로 습득한 것이다.

점검 항목미달 (주니어 수준)달성 (시니어 수준)
침해 사고 분석 방식 "이 취약점 하나가 원인이었다" 공격 체인 전체를 재구성하고, 각 단계에서 실패한 방어 계층을 구분하며, 기술적 원인과 조직적 원인을 분리해서 설명
새로운 시스템 설계 앞에서의 사고 보안 키워드를 떠오르는 대로 나열 데이터 흐름/신뢰 경계를 먼저 그리고, STRIDE를 구간별로 적용한 뒤, Day 1~9의 구체 지식을 해당 구간에 정확히 매핑
단일 방어에 대한 태도 "이 방어 하나면 완전히 안전하다" 어떤 단일 방어도 뚫릴 수 있다는 전제 아래, 뚫렸을 때 피해 반경을 최소화하는 다음 계층(최소 권한, 세그멘테이션)을 함께 설계
사고 대응 순서 원인 분석부터 시작 억제(Containment) → 증거 보존/범위 파악 → 근본 원인 분석 → 투명한 보고 순서로, 피해 확산 방지를 최우선으로 판단
보안과 비즈니스 목표의 충돌 "보안이 항상 최우선이다" 또는 "일정이 급하면 보안은 미룬다" 중 하나로만 답함 설계에 내장돼야 하는 항목과 출시 후 점진적으로 강화 가능한 항목을 구분하고, 리스크 기반으로 절충안을 구체적으로 제시
트랙 완료 — 다음 단계

Day 1부터 Day 10까지, 인증/인가의 위협 모델링 기초부터 실제 침해 사례를 통합 분석하는 수준까지 다뤘다. 이 트랙에서 반복적으로 강조한 핵심은 하나다 — "이 취약점이 무엇인지 아는 것"과 "이 취약점이 실제로 뚫렸을 때 시스템이 어떻게 무너지는지, 그리고 뚫려도 피해를 최소화하는 구조를 설계하는 것"은 완전히 다른 역량이라는 것이다. 면접에서 어떤 보안 질문을 받든, 위협 식별 → 공격 체인 추적 → 계층적 방어 → trade-off 판단 이 네 단계 사고 순서를 그대로 적용하면 이 트랙에서 다루지 않은 새로운 상황에서도 스스로 답을 구성할 수 있다.
