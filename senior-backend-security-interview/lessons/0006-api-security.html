1. 왜 "API 보안"이 SQLi·XSS와 별개로 다뤄지는가

Day 3~5에서 다룬 SQL Injection, XSS/CSRF, 파일 업로드/SSRF는 "입력값이 오염되어 발생하는 취약점"이었다. 반면 Day 6의 API 보안은 조금 다른 층위의 문제를 다룬다 — 입력값 자체는 멀쩡한데, "이 요청을 이 사용자가 이 자원에 대해 수행해도 되는가"를 서버가 제대로 확인하지 않아서 생기는 문제다. OWASP가 별도로 API Security Top 10을 만든 이유도 여기에 있다 — REST/GraphQL API가 보편화되면서 "리소스를 식별자로 직접 조회·수정하는" 패턴 자체가 새로운 공격 표면을 만들었기 때문이다.

실무 상황

한 핀테크 서비스가 GET /api/invoices/{id} API로 청구서를 조회하는 기능을 만들었다. 로그인 토큰 검증(인증, Authentication)은 완벽하게 구현되어 있었다 — 유효한 JWT 없이는 API를 호출할 수 없었다. 하지만 {id}가 내 청구서인지 남의 청구서인지는 확인하지 않고 DB에서 그냥 조회해 반환했다. 공격자는 자신의 정상 계정으로 로그인한 뒤 id값만 1씩 증가시키며 순회해 수만 건의 타인 청구서(이름, 주소, 결제 내역)를 그대로 열람했다. "인증은 완벽했지만 인가가 없었던" 전형적인 사고다. 이런 유형의 취약점은 스캐너가 잡아내기 어렵다 — 요청 자체는 문법적으로 완전히 정상적인 API 호출이기 때문이다.

흔한 오해 — "JWT 검증 미들웨어를 API 앞단에 걸어뒀으니 인가도 처리된 것이다"

인증(Authentication)과 인가(Authorization)는 Day 1에서 다룬 것처럼 완전히 다른 질문이다. JWT 미들웨어가 확인하는 것은 "이 요청을 보낸 사람이 누구인가"뿐이다. "그 사람이 이 특정 자원에 접근해도 되는가"는 엔드포인트/비즈니스 로직 레벨에서 매번 별도로 확인해야 하는 질문이다. 이 둘을 하나로 착각하는 순간, 시스템 전체에 인증은 있지만 인가는 없는 API가 대량으로 생긴다.

2. BOLA(Broken Object Level Authorization) — API 취약점 1위의 정체

전제 개념: IDOR과 BOLA는 같은 문제를 부르는 다른 이름이다

예전에는 IDOR(Insecure Direct Object Reference)라는 이름으로 불리던 문제가, API가 REST 형태로 표준화되면서 BOLA라는 이름으로 OWASP API Security Top 10의 1위 자리에 올라와 있다. 핵심은 동일하다 — URL이나 요청 바디에 들어있는 객체 식별자(id)를 근거로 자원에 접근할 때, "이 id가 요청자의 소유/권한 범위 안에 있는가"를 서버가 검증하지 않는 것이다. 

# 취약한 코드 (의사코드)
@app.get("/api/invoices/{invoice_id}")
def get_invoice(invoice_id: int, current_user):
    invoice = db.query(Invoice).filter(Invoice.id == invoice_id).first()
    return invoice   # invoice.owner_id가 current_user인지 확인하지 않음

# 공격: 내 계정으로 로그인 후 invoice_id를 1001, 1002, 1003...으로 순회
# 인증 미들웨어는 매 요청마다 통과시킨다 — "유효한 로그인 사용자"이기 때문이다
# 하지만 그 청구서가 "내 것"인지는 아무도 확인하지 않았다
핵심 원리

BOLA가 위험한 이유는 탐지가 매우 어렵다는 데 있다. 공격 요청이 문법적으로 완벽하게 정상이고, 인증 토큰도 유효하며, 응답도 200 OK로 정상 반환된다. 방화벽이나 WAF는 "패턴이 이상한 요청"을 찾도록 설계되어 있는데, BOLA 공격은 패턴상 지극히 정상적인 API 호출을 반복할 뿐이다. 그래서 방어는 네트워크 레벨이 아니라 "이 자원을 조회/수정하는 모든 코드 경로에서 소유권·권한을 확인하는 습관", 즉 애플리케이션 레벨의 규율에 의존할 수밖에 없다.

BOLA 방어 — 모든 객체 접근 지점에 적용해야 하는 원칙
  • 1. 소유권/권한 검증을 쿼리 자체에 포함: WHERE id = ? AND owner_id = ?처럼 조회 조건 자체에 소유권을 포함시켜, "존재하지만 내 것이 아닌" 자원은 애초에 조회 결과에 나타나지 않게 한다.
  • 2. 프레임워크의 중앙화된 인가 계층 사용: 매 엔드포인트마다 개발자가 수동으로 if문을 써서 확인하게 두면 실수로 하나쯤 빠뜨리기 마련이다. 정책 기반 인가(policy-based authorization, 예: Day 1에서 다룬 RBAC/ABAC 프레임워크)를 도입해 "이 리소스 타입은 항상 소유자 검증을 거친다"를 코드 리뷰가 아니라 구조로 강제한다.
  • 3. 순차적/예측 가능한 ID 대신 UUID 사용을 병행: 이건 근본 방어가 아니라 공격 난이도를 높이는 보조 수단이다. UUID를 쓰면 순회 스캔은 어려워지지만, 다른 경로(예: 목록 API 응답, 로그, URL 공유)로 다른 사용자의 UUID를 알게 되면 여전히 BOLA는 그대로 뚫린다. "ID를 추측하기 어렵게 만드는 것"과 "권한을 검증하는 것"을 혼동하면 안 된다.
  • 4. 자동화 테스트에 인가 시나리오 포함: "사용자 A의 토큰으로 사용자 B의 자원 id를 요청했을 때 403/404가 반환되는가"를 모든 리소스 엔드포인트에 대해 회귀 테스트로 고정한다.
흔한 오해 — "ID를 UUID로 바꾸면 BOLA는 해결된다"

UUID는 추측을 어렵게 만들 뿐 권한 검증을 대신하지 않는다. 다른 API 응답(예: 게시글 목록에 작성자의 프로필 id가 노출)이나 공유 링크, 브라우저 히스토리, 프록시 로그를 통해 UUID가 노출되는 경로는 여전히 많다. "추측 불가능"과 "권한 없음"은 서로 다른 방어선이고, BOLA의 근본 해법은 항상 후자다.

3. Mass Assignment — 클라이언트가 보낸 필드를 그대로 믿을 때

전제 개념: Mass Assignment(대량 할당)란 무엇인가?

많은 웹 프레임워크는 편의를 위해 요청 바디(JSON)를 객체/모델에 자동으로 매핑해주는 기능을 제공한다. 예를 들어 {"name": "Alice", "email": "a@x.com"}이라는 요청 바디를 받으면, 프레임워크가 알아서 User 모델의 name, email 필드에 값을 채워 넣는다. 문제는 이 자동 매핑이 "개발자가 의도한 필드"뿐 아니라 "모델에 존재하는 모든 필드"에 대해 무분별하게 일어날 수 있다는 점이다. 

# 취약한 코드 (의사코드) — 회원가입 API
@app.post("/api/users")
def create_user(body: dict):
    user = User(**body)   # 요청 바디의 모든 키를 그대로 User 모델 생성자에 전달
    db.save(user)
    return user

# User 모델에 is_admin 필드가 존재한다면?
# 정상 요청: {"name": "Alice", "email": "a@x.com"}
# 공격 요청: {"name": "Alice", "email": "a@x.com", "is_admin": true}
# → 프론트엔드 화면에는 is_admin을 입력하는 필드가 없었지만,
#   API를 직접 호출하면 그 필드까지 그대로 모델에 반영된다
핵심 원리

Mass Assignment의 근본 원인은 "프론트엔드 화면에 없으니 사용자가 보낼 리가 없다"는 가정이다. 하지만 공격자는 브라우저 화면이 아니라 API를 직접 호출한다. 클라이언트가 무엇을 "보낼 수 있는가"는 화면 UI가 아니라 서버가 무엇을 받아들이는가로 결정되어야 한다. 이건 결국 인가(Authorization) 문제의 변형이다 — "이 필드를 이 사용자가 수정해도 되는가"를 서버가 확인하지 않은 것이기 때문이다.

Mass Assignment 방어 — 화이트리스트 기반 필드 매핑
  • 1. 입력 전용 DTO(Data Transfer Object) 분리: 요청 바디를 DB 모델에 직접 매핑하지 않고, "이 API가 받을 수 있는 필드만 명시적으로 선언한 별도 타입"(예: CreateUserRequest{name, email})을 거친다. 그 DTO에 없는 필드는 요청에 포함돼도 애초에 파싱 단계에서 무시된다.
  • 2. 민감 필드는 별도 API/권한으로 분리: is_admin, role, balance 같은 민감 필드는 일반 사용자용 수정 API에서 원천적으로 다루지 않고, 관리자 전용 별도 엔드포인트에서 별도 인가 체크와 함께 처리한다.
  • 3. ORM/프레임워크의 "허용 필드 명시" 기능 사용: 대부분의 최신 프레임워크는 화이트리스트 방식의 필드 매핑 기능(예: 시리얼라이저의 fields 옵션)을 제공한다. "위험 필드를 블랙리스트로 제외"하는 방식은 새 필드가 추가될 때마다 놓치기 쉬우므로, 기본값을 "허용된 것만 통과"로 두는 화이트리스트 방식이 근본적으로 안전하다.
흔한 오해 — "프론트엔드 폼에 그 필드를 안 넣었으니 안전하다"

프론트엔드는 사용자 경험을 위한 계층이지 보안 경계가 아니다. 브라우저 개발자 도구나 Postman, curl로 누구나 임의의 JSON을 서버에 직접 보낼 수 있다. "UI에 없다"는 "API가 막는다"와 전혀 다른 말이다.

4. 과도한 데이터 노출(Excessive Data Exposure) — "프론트가 안 쓰니 괜찮다"는 착각

전제 개념: API 응답과 화면 표시는 서로 다른 계층이다

많은 백엔드 개발자가 편의상 DB 모델 전체를 그대로 직렬화해서 응답하고, "프론트엔드가 필요한 필드만 골라서 화면에 그리면 되겠지"라고 생각한다. 문제는 응답 바디 전체가 네트워크를 통해 클라이언트에 도달하는 순간, 그 데이터는 이미 노출된 것이라는 점이다. 프론트엔드가 화면에 안 그린다고 해서 브라우저 개발자 도구의 Network 탭에서 그 데이터가 안 보이는 게 아니다. 

# 과도한 데이터 노출 예시 (의사코드)
@app.get("/api/users/{id}")
def get_user(id: int):
    user = db.query(User).get(id)
    return user.to_dict()   # 모델의 모든 필드를 그대로 반환

# User 모델 필드: id, name, email, password_hash, ssn, internal_risk_score, is_admin
# 프론트엔드는 name과 email만 화면에 표시하지만,
# 응답 JSON에는 password_hash, ssn, internal_risk_score까지 전부 포함되어 있다
# → 브라우저 Network 탭이나 API를 직접 호출하는 누구나 이 값을 그대로 열람 가능
실무에서 자주 나타나는 두 가지 유형

(1) 응답 스키마 미설계 — DB 모델을 그대로 직렬화해서 반환. 필드가 늘어날 때마다 의도치 않은 민감 정보가 API 응답에 자동으로 섞여 들어간다.
(2) GraphQL의 과다 조회(Over-fetching) — GraphQL은 클라이언트가 원하는 필드를 스스로 선택할 수 있는데, 필드 단위 인가(field-level authorization)가 없으면 클라이언트가 스키마에 존재하는 민감 필드를 그냥 쿼리에 포함시켜 요청하는 것만으로 응답받을 수 있다.

과도한 데이터 노출 방어 — 응답 전용 스키마 설계
  • 1. 응답 전용 DTO/시리얼라이저 명시: "이 API가 반환할 필드"를 화이트리스트로 명시적으로 선언한다. DB 모델에 필드가 추가돼도 API 응답 스키마를 별도로 수정하지 않으면 새 필드는 노출되지 않는 구조가 안전하다.
  • 2. 민감도별 필드 분류와 역할 기반 응답 차등화: 같은 /api/users/{id}라도 요청자가 본인/관리자/타 사용자인지에 따라 반환 필드 집합을 다르게 구성한다(예: 본인 조회 시 email 노출, 타인 조회 시 이름만 노출).
  • 3. GraphQL 필드 레벨 인가: 리졸버(resolver) 단위로 "이 필드를 요청자가 조회해도 되는가"를 확인하고, 스키마에 정의되어 있다는 것과 접근 가능하다는 것을 분리해서 다룬다.
  • 4. 로그·에러 메시지의 데이터 노출도 함께 점검: 응답 바디뿐 아니라 스택 트레이스, 디버그 모드 에러 페이지, 상세 로그에도 동일한 원칙(민감 정보 최소 노출)을 적용해야 한다.
흔한 오해 — "프론트엔드에서 필드를 안 쓰니 API가 반환해도 상관없다"

API 응답은 브라우저의 네트워크 계층에 전부 노출된다. 프론트엔드 코드가 특정 필드를 사용하지 않는 것과, 그 필드가 실제로 클라이언트에 전송되지 않는 것은 완전히 다른 문제다. "필요한 것만 반환"은 프론트엔드 편의가 아니라 서버가 책임져야 할 보안 설계다.

5. Rate Limiting — 어디에, 무엇을 기준으로 제한할 것인가

전제 개념: Rate Limiting은 왜 필요한가?

API에 요청 횟수 제한이 없으면, 정상적인 인증/인가 로직을 모두 통과한 요청이라도 같은 요청을 무한히 반복하는 것 자체가 공격이 된다. 대표적으로 로그인 API에 대한 크리덴셜 스터핑(credential stuffing, 유출된 아이디/비밀번호 조합을 대량으로 시도), 특정 엔드포인트에 대한 무차별 대입(brute force), 그리고 단순히 서버 자원을 고갈시키는 DoS성 남용이 있다.

핵심 원리 — Rate Limiting은 "인증 실패"가 아니라 "속도"를 막는다

SQLi 방어나 인가 검증은 "이 요청이 유효한가/권한이 있는가"를 판단하지만, Rate Limiting은 요청 자체가 유효하더라도 "같은 주체가 짧은 시간에 얼마나 반복하는가"를 별도로 통제한다. 예를 들어 로그인 시도는 비밀번호가 틀려도 API 자체는 "정상적으로 동작"한 것이다 — 다만 그 정상 동작을 수만 번 반복하도록 내버려두면 안 되는 것뿐이다.

Rate Limiting 설계 — 레이어와 기준
  • 1. 어느 레이어에 둘 것인가: CDN/API Gateway(예: Cloudflare, AWS API Gateway, Nginx) 레벨에서 1차로 대략적인 트래픽량을 제한하고, 애플리케이션 레벨에서 "로그인 시도", "비밀번호 재설정 요청"처럼 비즈니스 의미가 있는 특정 엔드포인트에 대해 더 촘촘한 제한을 추가로 둔다. 게이트웨이 레벨만으로는 "이 엔드포인트가 왜 민감한지"를 모르기 때문에 두 레이어가 함께 필요하다.
  • 2. 무엇을 기준(key)으로 제한할 것인가: IP 기준만으로는 NAT 뒤의 다수 정상 사용자를 함께 제한하거나, 공격자가 프록시/봇넷으로 IP를 바꿔가며 우회하는 것을 막지 못한다. IP + 계정(시도한 아이디) 조합, 인증된 사용자라면 사용자 ID 기준을 함께 고려해 제한 기준을 다층화한다.
  • 3. 실패 시 응답 설계: 단순 차단(429 Too Many Requests)뿐 아니라, 로그인 실패가 반복되는 계정에는 CAPTCHA 요구, 점진적 지연(exponential backoff), 혹은 알림 발송 같은 단계적 대응을 함께 설계한다.
  • 4. 정상 사용자 경험과의 trade-off: 제한을 너무 촘촘하게 걸면 정상적인 재시도(비밀번호를 두 번 정도 잘못 입력하는 일반적 상황)까지 차단해 사용성이 나빠진다. 임계값은 실제 정상 사용자 행동 데이터를 근거로 설정해야 하며, "보안이니 무조건 엄격하게"는 답이 아니다.
흔한 오해 — "IP당 분당 요청 수만 제한하면 무차별 대입은 막힌다"

공격자는 프록시 풀이나 봇넷을 이용해 수천 개의 서로 다른 IP에서 계정 하나에 대해 소수의 시도씩 분산시킬 수 있다(이른바 low-and-slow 공격). IP 기준 제한만으로는 이런 분산 공격을 탐지하지 못하므로, "특정 계정에 대한 실패 시도 누적"이라는 계정 기준 지표를 반드시 함께 둬야 한다.

6. API 보안 계층 방어 종합 — Gateway부터 리소스 레벨까지

지금까지 다룬 네 가지(BOLA, Mass Assignment, 과도한 데이터 노출, Rate Limiting)는 각각 다른 문제지만, 시니어 면접에서는 "이걸 하나의 API 보안 아키텍처로 어떻게 계층화하는가"를 종합적으로 설명할 수 있는지를 본다.

API 요청이 통과하는 계층별 책임
  • 1. Edge/Gateway 레벨: TLS 종료, 기본 Rate Limiting, 봇 트래픽 필터링, 스키마 유효성(요청 형식이 API 명세와 맞는가) 1차 검증.
  • 2. 인증(Authentication) 레벨: "이 요청을 보낸 주체가 누구인가"를 토큰/세션 검증으로 확정한다.
  • 3. 인가(Authorization) 레벨 — 엔드포인트 단위: "이 역할(role)이 이 엔드포인트를 호출할 수 있는가"를 확인한다(예: 일반 사용자는 관리자 API 호출 불가).
  • 4. 인가 레벨 — 객체 단위(Object-level): BOLA 방어의 핵심. "이 사용자가 이 특정 자원(id)에 접근해도 되는가"를 쿼리/비즈니스 로직에서 매번 확인한다.
  • 5. 입력 처리 레벨: Mass Assignment 방어. 요청 바디에서 "이 API가 받도록 정의한 필드"만 화이트리스트로 추출한다.
  • 6. 출력 처리 레벨: 과도한 데이터 노출 방어. "이 API가 반환하도록 정의한 필드"만 화이트리스트로 직렬화한다.
면접에서 이 구조를 설명할 때의 핵심

"인증만 확인하면 끝"이라는 답은 2~4번 계층을 하나로 뭉갠 것이다. 시니어 수준 답변은 인증(누구인가) → 엔드포인트 인가(이 역할이 이 API를 쓸 수 있는가) → 객체 인가(이 자원이 이 사람 것인가)를 명확히 구분된 세 개의 질문으로 순서대로 제시하고, 각 단계가 뚫렸을 때 어떤 사고로 이어지는지(예: 객체 인가 누락 → BOLA)를 연결 지어 설명하는 것이다.

7. 면접 질문 1 — "로그인만 되면 인가는 끝난 거 아닌가요?"

면접 질문
"우리 API는 모든 엔드포인트에 JWT 인증을 걸어뒀습니다. 로그인 안 한 사용자는 아예 API를 호출할 수 없어요. 이 정도면 인가 문제는 없다고 봐도 되나요?"

왜 면접에서 묻는가?

인증(Authentication)과 인가(Authorization)를 같은 것으로 착각하는지 확인하는 가장 직접적인 질문이다. 특히 인증은 되지만 인가가 빠진 API(BOLA)가 실무에서 가장 흔한 API 취약점이라는 것을 알고 있는지, 그리고 그것을 구체적인 시나리오로 설명할 수 있는지를 본다.

답변 사고 순서

  1. 인증과 인가의 질문 구분 — "누구인가"와 "이 자원에 접근해도 되는가"가 다른 질문임을 짚는다.
  2. 구체적 반례 제시 — 로그인은 됐지만 객체 소유권 검증이 없어 타인의 자원에 접근 가능한 BOLA 시나리오를 든다.
  3. 결론 — 인증 미들웨어와 별개로 객체 단위 인가 검증을 모든 리소스 접근 지점에 강제해야 함을 제시한다.
강한 답변 예시

"JWT 인증은 '이 요청을 보낸 사람이 누구인가'를 확인하는 것이고, 인가는 '그 사람이 지금 요청한 이 특정 자원에 접근해도 되는가'를 확인하는 것으로, 완전히 다른 질문입니다. 예를 들어 GET /api/invoices/{id}에 JWT 인증이 걸려 있어도, 서버가 조회 시 invoice.owner_id가 요청자와 같은지 확인하지 않으면 로그인한 어떤 사용자든 id 값만 바꿔가며 다른 사람의 청구서를 그대로 열람할 수 있습니다. 이게 BOLA라고 불리는, OWASP API Security Top 10에서 1위로 꼽히는 취약점입니다. 그래서 저희는 인증 미들웨어와는 별개로, 자원을 조회·수정하는 모든 쿼리에 소유권 조건을 포함시키거나, 중앙화된 인가 정책 계층을 둬서 개발자가 실수로 이 검증을 빠뜨리지 않도록 구조적으로 강제하는 방식을 씁니다."

약한 답변 예시

"로그인 안 하면 API를 아예 못 부르니까 인가는 자동으로 처리된 거라고 봅니다."

왜 약한가? 인증과 인가를 동일시하고 있으며, "로그인된 아무 사용자나 임의의 자원에 접근할 수 있는" 시나리오를 전혀 고려하지 못하고 있다.

Senior-level Follow-up 질문들

Follow-up: "리소스가 100개가 넘는 대규모 API 서버에서, 모든 엔드포인트에 객체 단위 인가 검증이 빠짐없이 들어가 있는지 어떻게 확인하고 관리하시겠습니까?"

모범 답변 방향: "개발자 각자의 기억에 의존하는 방식은 규모가 커지면 반드시 구멍이 생깁니다. 우선 리소스 접근 로직을 공통 레포지토리/서비스 레이어로 강제해서 '소유권 조건을 포함한 조회'가 기본값이 되도록 구조를 만듭니다. 그 위에 코드 리뷰 체크리스트에 '이 엔드포인트가 새로운 리소스 타입을 다루는가? 소유권 검증이 포함됐는가'를 명시적 항목으로 넣고, 자동화 테스트로 '사용자 A의 토큰으로 사용자 B의 리소스 id를 요청하면 403/404가 나오는가'를 모든 리소스 엔드포인트에 대해 회귀 테스트로 고정합니다. 정기적으로는 API 명세(OpenAPI 등)를 기반으로 전체 엔드포인트 목록을 뽑아 인가 검증 여부를 감사하는 프로세스도 둡니다."

8. 면접 질문 2 — Rate Limiting을 어느 레이어에 둘 것인가

면접 질문
"로그인 API에 무차별 대입 공격을 막기 위해 Rate Limiting을 도입하려고 합니다. API Gateway에 걸까요, 애플리케이션 코드에 걸까요?"

왜 면접에서 묻는가?

Rate Limiting을 "걸면 끝"인 단순 기능으로 여기는지, 아니면 레이어마다 감지할 수 있는 정보와 한계가 다르다는 것을 이해하고 두 레이어를 목적에 맞게 조합할 수 있는지를 본다. 또한 IP 기준만으로는 부족한 이유(분산 공격)를 구체적으로 설명할 수 있는지도 확인한다.

답변 사고 순서

  1. 두 레이어의 역할 구분 — Gateway는 대략적/일반적 트래픽 제어, 애플리케이션은 비즈니스 의미가 있는 세밀한 제어를 담당함을 제시한다.
  2. 기준(key) 설계의 한계 지적 — IP 단독 기준의 우회 가능성(분산 공격)을 짚는다.
  3. 결론 — 두 레이어를 함께 두고, 계정 기준 지표를 반드시 포함시킨다는 조합형 설계로 마무리한다.
강한 답변 예시

"둘 다 필요하다고 봅니다. API Gateway나 CDN 레벨의 Rate Limiting은 로그인 API뿐 아니라 전체 트래픽에 대해 일반적인 과부하나 봇 트래픽을 걸러내는 1차 방어선 역할을 합니다. 다만 Gateway는 '이 엔드포인트가 로그인처럼 민감한 의미를 가진 API'라는 비즈니스 맥락을 모르기 때문에, 로그인 실패가 반복되는 것 자체를 탐지하려면 애플리케이션 레벨에서 별도로 제한을 걸어야 합니다. 그리고 기준을 IP 하나로만 두면, 공격자가 봇넷이나 프록시 풀로 IP를 수천 개 분산시켜 각 IP에서는 소수의 시도만 하는 low-and-slow 공격에 뚫립니다. 그래서 IP 기준과 별개로 '같은 계정에 대한 실패 시도 누적'이라는 계정 기준 지표를 반드시 함께 두고, 임계값을 넘으면 CAPTCHA 요구나 점진적 지연을 적용하는 식으로 설계합니다."

약한 답변 예시

"API Gateway에 IP당 분당 요청 수 제한만 걸어두면 충분하다고 생각합니다."

왜 약한가? Gateway 레벨 방어가 비즈니스 맥락을 모른다는 한계와, IP 단독 기준이 분산 공격에 취약하다는 점을 전혀 고려하지 못하고 있다.

Senior-level Follow-up 질문들

Follow-up: "Rate Limiting을 너무 엄격하게 설정했더니 정상 사용자들이 '이미 로그인했는데 또 막힌다'는 불만을 제기합니다. 어떻게 균형을 맞추시겠습니까?"

모범 답변 방향: "보안 임계값은 실제 정상 사용자의 행동 데이터를 먼저 분석해서 설정해야 합니다. 예를 들어 정상 사용자의 비밀번호 오타 재시도가 보통 2~3회 안에 끝난다는 데이터가 있다면, 임계값을 그보다 여유 있게(예: 5회) 잡고, 그 이상부터는 즉시 차단이 아니라 CAPTCHA 같은 완충 단계를 먼저 넣습니다. 또한 동일 계정에서 동일 기기/브라우저로 반복되는 정상적인 재시도와, 짧은 시간에 다른 IP·기기에서 몰리는 비정상 패턴을 구분해서, 후자에 대해서만 더 엄격한 제한을 적용하는 식으로 세분화합니다. 결국 '보안이니 무조건 엄격하게'가 아니라, 실제 공격 패턴과 정상 사용 패턴의 차이를 근거로 임계값을 데이터 기반으로 조정하는 게 핵심입니다."

9. 면접 질문 3 — Mass Assignment를 프레임워크가 알아서 막아주지 않나요?

면접 질문
"저희는 최신 웹 프레임워크를 쓰고 있어서 요청 바디를 모델에 자동으로 매핑해줍니다. 이런 편의 기능 덕분에 개발 속도가 빠른데, Mass Assignment 같은 문제가 실제로 발생할 수 있나요?"

왜 면접에서 묻는가?

"프레임워크가 제공하는 편의 기능"과 "그 기능의 기본 설정이 안전한가"를 구분해서 사고하는지 확인하는 질문이다. 편의 기능 자체를 비판하는 게 아니라, 기본 동작이 화이트리스트인지 블랙리스트인지, 그리고 그것을 개발자가 명시적으로 인지하고 있는지가 핵심이다.

답변 사고 순서

  1. 편의 기능 자체는 문제가 아님을 인정 — 자동 매핑은 생산성 도구일 뿐, 위험은 "무엇까지 매핑되는가"에 있음을 짚는다.
  2. 구체적 공격 시나리오 제시 — 화면에 없는 민감 필드(is_admin 등)를 API 직접 호출로 주입하는 예를 든다.
  3. 결론 — 입력 전용 DTO와 화이트리스트 매핑으로 "받을 수 있는 필드"를 명시적으로 좁힌다는 설계로 마무리한다.
강한 답변 예시

"자동 매핑 자체는 좋은 기능이고 문제가 아닙니다. 문제는 그 매핑이 '모델에 존재하는 모든 필드'를 대상으로 무분별하게 일어날 때 발생합니다. 예를 들어 User 모델에 is_admin 필드가 있는데 회원가입 API가 요청 바디를 그대로 모델 생성자에 넘긴다면, 프론트엔드 화면에는 그 필드를 입력하는 곳이 없어도 API를 직접 호출하는 공격자는 {'is_admin': true}를 추가로 보내는 것만으로 자기 계정을 관리자로 만들 수 있습니다. UI에 필드가 없다는 것과 서버가 그 필드를 받지 않는다는 것은 다른 이야기입니다. 그래서 저희는 요청 바디를 DB 모델에 직접 매핑하지 않고, 'name, email처럼 이 API가 명시적으로 받기로 한 필드만 정의한' 입력 전용 DTO를 거치게 해서, 정의되지 않은 필드는 요청에 포함돼도 파싱 단계에서 자동으로 무시되도록 만듭니다. is_admin이나 role 같은 민감 필드는 아예 일반 사용자용 API에서 다루지 않고 관리자 전용 엔드포인트로 분리합니다."

약한 답변 예시

"프론트엔드에서 그런 값을 보내는 폼 자체가 없으니까 실제로 그런 공격이 들어올 가능성은 낮다고 봅니다."

왜 약한가? 공격자가 프론트엔드 UI를 거치지 않고 API를 직접 호출할 수 있다는 사실을 고려하지 않고 있다.

Senior-level Follow-up 질문들

Follow-up: "입력 DTO를 화이트리스트로 잘 분리해뒀는데, 응답에서는 여전히 DB 모델을 그대로 반환하고 있다면 어떤 위험이 남아있을까요?"

모범 답변 방향: "입력을 좁혀도 출력을 좁히지 않으면 과도한 데이터 노출(Excessive Data Exposure) 문제가 그대로 남습니다. password_hash, 내부 리스크 점수, 다른 사용자와의 관계 정보처럼 DB 모델에는 있지만 API 응답으로 나가면 안 되는 필드가 그대로 직렬화되어 클라이언트에 전송될 수 있습니다. 입력과 출력은 별개의 화이트리스트로 다뤄야 하고, 응답도 '이 API가 반환하기로 정의한 필드만 명시적으로 선언한' 응답 전용 DTO/시리얼라이저를 거치도록 구조를 만들어야 이 문제를 함께 막을 수 있습니다."
