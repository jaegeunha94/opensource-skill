1. 왜 "비밀번호 저장"이 별도의 인터뷰 주제인가

Day 1~6에서는 요청이 오가는 흐름(인증/인가, 세션, SQLi, XSS/CSRF, 파일 업로드, API 설계)에서 발생하는 취약점을 다뤘다. Day 7은 시야를 좁혀 "DB에 이미 저장된 비밀 정보 자체가 유출됐을 때 얼마나 큰 피해로 이어지는가"를 다룬다. 이 주제가 시니어 면접에서 반복적으로 나오는 이유는, 다른 모든 방어가 실패해서 공격자가 DB를 통째로 탈취한 최악의 시나리오에서 비밀번호 저장 방식이 "2차 피해의 규모를 결정하는 마지막 방어선"이기 때문이다.

실무 상황

2012년 링크드인(LinkedIn)은 SQL Injection으로 추정되는 침해를 통해 약 6,500만 개의 비밀번호 해시가 유출됐다. 문제는 해시 방식이 솔트(salt) 없는 SHA-1이었다는 점이다. 공격자는 미리 계산해둔 해시 테이블 (레인보우 테이블)과 GPU 병렬 크래킹을 이용해 유출된 해시 대부분을 수일 내로 원문 비밀번호로 복원했다. 더 심각한 문제는, 많은 사용자가 여러 서비스에 같은 비밀번호를 재사용한다는 점이었다 — 복원된 비밀번호는 링크드인 계정뿐 아니라 다른 서비스의 계정까지 크리덴셜 스터핑으로 뚫는 데 그대로 쓰였다. "해시로 저장했다"는 사실 자체가 안전을 보장하지 않는다는 것을 보여준 대표 사례다.

흔한 오해 — "해시로 저장하면 안전하다"

"해시"라는 단어 하나로 뭉뚱그리면 안 된다. 어떤 해시 함수를 썼는가, 솔트를 붙였는가, 그 함수가 의도적으로 느린가에 따라 안전도가 완전히 다르다. 시니어 면접에서 "비밀번호를 해시로 저장합니다"라는 답변만으로는 부족하고, 왜 그 알고리즘이어야 하는지까지 설명할 수 있어야 한다.

2. 해시(Hash) vs 암호화(Encryption) — 근본적으로 다른 도구

전제 개념: 암호화는 "되돌릴 수 있다", 해시는 "되돌릴 수 없다"

암호화(Encryption)는 키(key)를 가진 사람이 원문을 복원할 수 있도록 설계된 양방향 변환이다. 신용카드 번호, 주민등록번호처럼 서비스가 나중에 원문이 필요한 데이터(예: 결제 시 다시 조회해야 함)에 사용한다. 반면 해시(Hash)는 원문을 입력받아 고정 길이의 값을 출력하지만, 그 출력값에서 원문을 수학적으로 복원할 방법이 없도록(one-way) 설계된 단방향 함수다.

핵심 원리 — 비밀번호는 "서비스가 원문을 알 필요가 없는" 유일한 데이터

서비스는 로그인 시 사용자가 입력한 비밀번호를 원문 그대로 알 필요가 없다. "사용자가 입력한 값을 같은 해시 함수로 다시 돌렸을 때, 저장된 해시값과 일치하는가"만 확인하면 충분하다. 그래서 비밀번호는 암호화가 아니라 해시로 저장하는 것이 원칙이다 — 애초에 "복원 가능성"이라는 속성 자체가 비밀번호 저장에는 필요 없고, 오히려 위험 요인이다. 암호화는 키가 유출되면 저장된 모든 원문이 즉시 복원되지만, 제대로 설계된 해시는 키가 없어도 원문 자체가 애초에 존재하지 않으므로 복원 경로가 없다. 

# 암호화 (양방향) — 신용카드 번호 등에 적합, 비밀번호에는 부적합
encrypted = AES_encrypt(card_number, key)
decrypted = AES_decrypt(encrypted, key)   # key만 있으면 원문 복원 가능

# 해시 (단방향) — 비밀번호 저장에 적합
hashed = bcrypt_hash(password)
# hashed에서 password를 역산하는 수학적 방법은 존재하지 않는다
# 로그인 검증: bcrypt_verify(입력한_password, hashed) → true/false
흔한 오해 — "암호화가 해시보다 더 강력한 보안이니 암호화를 쓰는 게 낫다"

"더 강력하다/약하다"의 문제가 아니라 용도가 다르다. 암호화는 "나중에 원문이 필요한 데이터"용 도구이고, 해시는 "검증만 하면 되는 데이터"용 도구다. 비밀번호를 암호화로 저장하면, 키 하나가 유출되는 순간 저장된 모든 사용자의 비밀번호 원문이 한 번에 복원되는 훨씬 더 나쁜 결과로 이어진다. "암호화 = 고급 보안"이라는 인식 자체가 이 주제에서 가장 흔한 오해다.

3. 솔팅(Salting)과 레인보우 테이블 공격

전제 개념: 레인보우 테이블이란 무엇인가?

레인보우 테이블(Rainbow Table)은 자주 쓰이는 비밀번호들(예: "123456", "password", 유출된 비밀번호 목록 전체)을 미리 해시 함수로 계산해 "해시값 → 원문" 매핑을 대량으로 만들어둔 사전 계산 테이블이다. 공격자는 유출된 해시값을 이 테이블에서 조회만 하면 원문을 즉시 알아낼 수 있다 — 매번 새로 계산할 필요가 없으므로 크래킹 속도가 극단적으로 빨라진다.

솔트가 이 공격을 어떻게 무력화하는가

솔트(Salt)는 각 사용자의 비밀번호를 해시하기 전에 붙이는 사용자마다 다른 무작위 값이다. hash(password) 대신 hash(password + salt)를 저장하면, 같은 비밀번호("123456")를 쓰는 두 사용자라도 해시값이 완전히 달라진다. 공격자는 더 이상 "하나의 해시값 → 원문" 매핑을 미리 계산해둘 수 없다 — 솔트값마다 처음부터 다시 계산해야 하므로, 사전 계산된 레인보우 테이블 자체가 무의미해진다. 

# 솔트 없는 해시 — 레인보우 테이블에 취약
hash("123456")  # 항상 동일한 값 → 미리 계산된 테이블에서 즉시 조회 가능

# 솔트를 적용한 해시 — 사용자마다 다른 값
hash("123456" + salt_userA)  # 예: 3f2a91...
hash("123456" + salt_userB)  # 예: 8b7c04...  (같은 비밀번호라도 완전히 다른 해시)
# 공격자는 salt_userA, salt_userB 각각에 대해 별도로 레인보우 테이블을 다시 만들어야 한다
# → 사실상 사전 계산 공격이 무력화된다

# 실무에서는 솔트를 별도 컬럼으로 관리할 필요가 없다 —
# bcrypt/Argon2는 해시 결과 문자열 자체에 salt와 파라미터가 함께 인코딩되어 저장된다
# 예: $2b$12$KIXQ4z9vN...  (알고리즘 버전 $ cost factor $ salt+hash)
흔한 오해 — "솔트를 비밀로 유지해야 안전하다"

솔트는 비밀이 아니어도 된다. 솔트의 목적은 "숨기는 것"이 아니라 "사용자마다 해시값을 다르게 만들어 사전 계산 공격을 무력화하는 것"이다. 그래서 실무에서는 솔트를 해시값과 함께 그대로 저장해도 무방하며(bcrypt/Argon2는 실제로 그렇게 한다), 솔트를 별도로 암호화하거나 숨기려는 시도는 불필요한 복잡도만 늘린다. 솔트와 혼동하기 쉬운 개념으로 pepper가 있는데, 이는 DB가 아니라 애플리케이션 설정/HSM에 별도로 보관하는 전역 비밀값으로, DB만 유출되고 애플리케이션 서버는 유출되지 않은 시나리오에서 추가 방어선 역할을 한다. pepper는 솔트를 대체하는 게 아니라 선택적으로 추가하는 계층이다.

4. 느린 해시 함수가 필요한 이유 — bcrypt, scrypt, Argon2

전제 개념: 왜 SHA-256에 솔트를 붙여도 여전히 부족한가?

솔트를 붙였다고 안심하기엔 이르다. SHA-256, SHA-1 같은 범용 해시 함수는 "빠르게 계산되도록" 설계됐다 — 원래 목적이 파일 무결성 검증처럼 속도가 미덕인 용도였기 때문이다. 문제는 이 "빠름"이 공격자에게도 그대로 이득이 된다는 점이다. 최신 GPU는 SHA-256을 초당 수십억 번 계산할 수 있다. 솔트가 있어 사전 계산은 막았어도, 공격자는 유출된 해시 하나하나에 대해 실시간으로 무차별 대입(brute force)을 시도할 수 있고, 짧거나 흔한 비밀번호는 순식간에 뚫린다.

핵심 원리 — "계산 비용을 의도적으로 비싸게 만든다"

bcrypt, scrypt, Argon2는 "느리게 계산되도록" 의도적으로 설계된 해시 함수다. 내부적으로 해시 연산을 수천~수만 번 반복하거나(cost factor), 대량의 메모리를 사용하도록 강제해서 해시 1회 계산에 사람은 체감하지 못할 정도(수십~수백 ms)의 시간이 걸리지만, 공격자가 수십억 개의 후보 비밀번호를 시도하려면 그 시간이 누적되어 현실적으로 불가능한 수준이 되게 만든다. 정상 로그인은 사용자당 1회 계산이라 지연이 체감되지 않지만, 무차별 대입 공격자는 이 지연을 후보 개수만큼 반복해야 하므로 공격 비용이 기하급수적으로 커진다.

세 알고리즘의 차이 — 무엇을 "비싸게" 만드는가
  • bcrypt: CPU 연산 반복 횟수(cost factor)를 비싸게 만든다. 오래 검증된 알고리즘이고 대부분의 언어/프레임워크에 기본 지원되어 실무에서 가장 널리 쓰인다. 다만 메모리 사용량은 고정적으로 작아서, GPU/ASIC처럼 병렬 연산에 특화된 하드웨어로 어느 정도 가속이 가능하다는 한계가 있다.
  • scrypt: CPU 연산뿐 아니라 메모리 사용량 자체를 크게 요구하도록 설계됐다. GPU는 연산 코어는 많지만 코어당 메모리는 제한적이므로, 메모리를 많이 요구하는 연산은 병렬화 이점이 줄어든다. bcrypt보다 GPU 가속 저항성이 높다.
  • Argon2: 2015년 Password Hashing Competition 우승작으로, 현재 시점에서 가장 권장되는 알고리즘이다. CPU 시간, 메모리 사용량, 병렬도(parallelism)를 각각 독립적으로 조정할 수 있어 GPU/ASIC 가속에 대한 저항성이 가장 뛰어나다. Argon2id variant가 사이드채널 공격과 GPU 가속 저항을 모두 고려한 실무 권장 기본값이다.
흔한 오해 — "MD5나 SHA-256에 솔트만 붙이면 bcrypt와 다를 바 없다"

솔트는 사전 계산 공격(레인보우 테이블)만 막을 뿐, 실시간 무차별 대입 속도는 전혀 늦추지 못한다. MD5/SHA-256은 설계 목적 자체가 "빠른 계산"이기 때문에, 솔트를 아무리 잘 붙여도 유출된 해시 각각을 GPU로 빠르게 brute-force할 수 있다는 문제는 그대로 남는다. bcrypt/scrypt/Argon2가 해결하는 문제는 솔트와는 별개의 축(계산 비용 자체를 늘리는 것)이라는 점을 구분해야 한다.

5. 알고리즘 선택 기준과 파라미터 튜닝

시니어 면접에서는 "Argon2를 씁니다"라는 답변에서 그치지 않고, 왜 그 파라미터인지, trade-off가 무엇인지를 설명할 수 있어야 한다.

파라미터 튜닝의 핵심 trade-off

cost factor(bcrypt)나 메모리/반복 파라미터(Argon2)를 높일수록 공격자의 brute-force 비용은 커지지만, 정상 사용자의 로그인/회원가입 응답 시간과 서버 CPU/메모리 부하도 함께 늘어난다. 목표는 "서버가 감당 가능한 선에서, 로그인 요청 하나당 대략 100~500ms 내외가 걸리도록" 파라미터를 맞추는 것이다. 이 값은 하드웨어가 발전할수록 주기적으로 재조정해야 한다 — 서버 CPU가 빨라지면 같은 cost factor라도 공격자 쪽 계산도 함께 빨라지기 때문이다.

알고리즘 선택 실무 기준
  • 1. 신규 프로젝트: 특별한 제약이 없다면 Argon2id를 우선 고려한다. GPU/ASIC 저항성이 가장 뛰어나고 현재 업계 권장 표준(OWASP Password Storage Cheat Sheet 기준)이다.
  • 2. 언어/프레임워크 생태계 제약: Argon2 라이브러리 지원이 불안정하거나, 기존 시스템과의 호환성이 중요한 경우 bcrypt도 여전히 충분히 안전한 선택이다 — 수십 년간 검증되었고 실무에서 가장 널리 쓰인다.
  • 3. 파라미터는 벤치마크로 결정: "권장값"을 그대로 베끼지 않고, 실제 운영 서버 하드웨어에서 로그인 요청 1건에 걸리는 시간을 측정해 목표 응답 시간(예: 200ms)에 맞춰 cost factor/메모리 파라미터를 조정한다.
  • 4. 절대 쓰면 안 되는 것: MD5, SHA-1, 솔트 없는 SHA-256 등 범용 빠른 해시는 비밀번호 저장에 절대 사용하지 않는다. 이들은 무결성 검증·체크섬 용도로 설계된 것이지 크리덴셜 보호용이 아니다.

6. 유출 대응과 크리덴셜 스터핑 방어 — 저장 방식만으로는 안 끝나는 이유

전제 개념: 크리덴셜 스터핑(Credential Stuffing)이란?

다른 서비스에서 유출된 아이디/비밀번호 조합을 그대로 우리 서비스 로그인에 대량으로 시도하는 공격이다. 사용자들이 여러 서비스에 같은 비밀번호를 재사용하는 습관 때문에 성공률이 예상보다 훨씬 높다. 이 공격은 우리 서비스의 해시 알고리즘이 아무리 강력해도 막을 수 없다 — 공격자는 해시를 크래킹하는 게 아니라, 이미 평문으로 알고 있는 비밀번호를 그대로 로그인 폼에 입력하기 때문이다.

비밀번호 저장 방식과 크리덴셜 스터핑은 서로 다른 방어선이다

Day 7에서 다룬 bcrypt/Argon2는 "우리 DB가 유출됐을 때 원문이 복원되는 것"을 막는 방어선이고, 크리덴셜 스터핑 방어는 "다른 곳에서 이미 유출된 원문 비밀번호로 로그인 시도가 들어오는 것"을 막는 별개의 방어선이다. 시니어 면접에서는 이 둘을 하나로 뭉뚱그리지 않고, "저장 방식을 아무리 강화해도 막을 수 없는 위협이 남아있다"는 것을 인지하고 있는지를 확인한다.

크리덴셜 스터핑 및 유출 대응 실무 조치
  • 1. 유출된 비밀번호 목록 대조: 회원가입/비밀번호 변경 시 Have I Been Pwned API 같은 서비스로 "이미 유출 이력이 있는 비밀번호"인지 확인하고 사용을 거부한다. 이는 NIST SP 800-63B가 권장하는 방식이다.
  • 2. Day 6에서 다룬 Rate Limiting과 결합: 계정 기준 로그인 실패 누적 감지, CAPTCHA, MFA(다중 인증)를 함께 적용해 "원문 비밀번호를 알고 있어도" 추가 인증 요소 없이는 로그인이 완료되지 않도록 만든다.
  • 3. 이상 로그인 탐지: 평소와 다른 위치/기기에서의 로그인 시도를 탐지해 추가 인증을 요구하거나 알림을 발송한다.
  • 4. 실제 유출 발생 시 대응 절차: 침해가 확인되면 전체 사용자 비밀번호를 강제 재설정하고, 유출 사실을 사용자에게 고지하며(법적 의무인 경우도 많음), 재사용 위험을 알린다. 저장 알고리즘이 강력했다면 "즉시 원문이 복원되지는 않았다"는 시간을 벌 수 있다는 것 자체가 강한 해시 알고리즘의 실질적 가치다.

7. 면접 질문 1 — "AES로 암호화해서 저장하면 되지 않나요?"

면접 질문
"비밀번호를 AES로 암호화해서 저장하면 되지 않나요? 필요할 때 복호화해서 비교하면 간단할 것 같은데, 왜 굳이 해시를 써야 하나요?"

왜 면접에서 묻는가?

암호화와 해시의 근본적 차이(양방향 vs 단방향)를 이해하는지, 그리고 "서비스가 원문을 알 필요가 없다"는 비밀번호 저장의 본질적 요구사항을 파악하고 있는지를 확인한다.

답변 사고 순서

  1. 비밀번호 검증에 원문 복원이 필요한지 자문 — 로그인은 "입력값을 재해시해서 비교"만 하면 되므로 원문 복원이 필요 없음을 짚는다.
  2. 암호화의 위험 지적 — 키 하나가 유출되면 저장된 모든 비밀번호 원문이 즉시 복원되는 구조적 취약점을 제시한다.
  3. 결론 — 해시는 애초에 복원 경로가 없어 키 유출이라는 단일 실패 지점 자체를 없앤다는 것으로 마무리한다.
강한 답변 예시

"비밀번호 검증은 사용자가 로그인 시 입력한 값을 같은 함수로 다시 계산해서 저장된 값과 일치하는지 비교하는 것만으로 충분합니다. 서비스가 원문 비밀번호 자체를 알아야 할 이유가 없어요. 그런데 AES 같은 암호화는 키만 있으면 원문을 복원할 수 있는 양방향 구조이기 때문에, 만약 암호화 키가 애플리케이션 설정 파일이나 메모리 덤프를 통해 유출되면 그 순간 DB에 저장된 모든 사용자의 비밀번호 원문이 한꺼번에 복원됩니다. 이건 키 하나가 단일 실패 지점(single point of failure)이 되는 구조입니다. 반면 bcrypt나 Argon2 같은 해시 함수는 설계 자체가 단방향이라서, 알고리즘의 키나 소스코드를 안다고 해도 저장된 해시값에서 원문을 역산할 수학적 방법이 없습니다. 그래서 저희는 비밀번호는 반드시 단방향 해시로, 나중에 원문이 필요한 카드번호 같은 데이터만 암호화로 저장합니다."

약한 답변 예시

"AES가 워낙 강력한 알고리즘이라 안전하다고 생각합니다."

왜 약한가? AES 자체의 강도 문제가 아니라 "양방향이라는 구조 자체"가 비밀번호 저장에 부적합하다는 핵심을 짚지 못했다. 키 유출 시 전체 사용자 데이터가 한 번에 복원된다는 결정적 위험을 설명하지 못하고 있다.

Senior-level Follow-up 질문들

Follow-up: "그럼 신용카드 번호는 왜 해시가 아니라 암호화로 저장하나요?"

모범 답변 방향: "신용카드 번호는 결제 처리 시점에 PG사에 다시 전달하거나, 사용자에게 마지막 4자리를 보여주는 등 서비스가 나중에 원문이 필요한 데이터이기 때문입니다. 해시로 저장하면 원문을 되돌릴 방법이 없어서 애초에 이런 용도 자체가 불가능합니다. 그래서 '원문이 나중에 필요한가'라는 질문이 암호화와 해시를 가르는 기준이 되고, 카드번호는 PCI-DSS 표준에 따라 토큰화(tokenization)나 강력한 암호화(키 관리는 별도 HSM/KMS로 분리)로 보호하는 것이 일반적입니다."

8. 면접 질문 2 — "SHA-256으로 해시하고 솔팅도 했는데 왜 안전하지 않나요?"

면접 질문
"비밀번호를 SHA-256으로 해시하고, 사용자마다 다른 솔트도 붙였습니다. 레인보우 테이블 공격도 막았는데, 이 정도면 충분한 보안 아닌가요?"

왜 면접에서 묻는가?

솔팅과 "계산 비용을 늘리는 것"이 서로 다른 문제를 해결하는 별개의 방어라는 것을 이해하는지 확인한다. 많은 지원자가 솔트만 붙이면 안전하다고 착각하는데, 범용 해시 함수의 "빠른 계산"이라는 특성 자체가 남아있는 문제를 짚을 수 있는지가 핵심이다.

답변 사고 순서

  1. 솔트가 막은 공격과 막지 못한 공격 구분 — 사전 계산(레인보우 테이블)은 막았지만 실시간 brute-force는 막지 못했음을 짚는다.
  2. SHA-256의 설계 목적 지적 — 원래 "빠른 계산"을 위해 설계된 함수라서 GPU로 초당 수십억 번 계산 가능함을 제시한다.
  3. 결론 — bcrypt/Argon2처럼 의도적으로 느린 함수로 교체해야 brute-force 비용 자체를 높일 수 있다는 것으로 마무리한다.
강한 답변 예시

"솔트는 레인보우 테이블 같은 사전 계산 공격을 막는 데는 확실히 효과적입니다. 하지만 SHA-256은 원래 파일 무결성 검증처럼 '빠르게 계산되는 것'이 목적인 함수라서, 최신 GPU로는 초당 수십억 번씩 계산할 수 있습니다. 즉 DB가 유출되면 공격자는 사전 계산된 테이블은 못 쓰더라도, 유출된 해시 하나하나에 대해 실시간으로 무차별 대입을 시도할 수 있고, 짧거나 흔한 비밀번호는 그 자리에서 뚫립니다. 솔트와 '계산 비용을 비싸게 만드는 것'은 서로 다른 축의 방어입니다. 그래서 저희는 SHA-256 대신 bcrypt나 Argon2처럼 의도적으로 느리게 설계된 해시 함수를 씁니다. 이런 함수들은 정상 로그인 한 번에는 100~200ms 정도의 체감되지 않는 지연만 발생시키지만, 공격자가 수십억 개 후보를 시도하려면 이 지연이 그대로 누적되어 현실적으로 불가능한 시간이 걸리게 만듭니다."

약한 답변 예시

"솔트를 붙였으면 레인보우 테이블 공격이 안 통하니까 그걸로 충분하다고 생각합니다."

왜 약한가? 사전 계산 공격(레인보우 테이블)과 실시간 무차별 대입이 서로 다른 공격이라는 것을 구분하지 못했고, SHA-256 자체의 "빠른 계산"이라는 설계 목적이 여전히 위험 요인으로 남아있다는 점을 놓쳤다.

Senior-level Follow-up 질문들

Follow-up: "bcrypt의 cost factor를 얼마로 설정해야 하는지 어떻게 결정하시겠습니까?"

모범 답변 방향: "고정된 '권장값'을 그대로 쓰기보다, 실제 운영 서버 하드웨어에서 벤치마크를 돌려 로그인 요청 1건에 걸리는 시간을 측정합니다. 목표는 사용자가 체감하지 못하면서도 서버 부하가 감당 가능한 수준, 예를 들어 200ms 내외가 되도록 cost factor를 조정하는 것입니다. 그리고 이 값은 한 번 정하고 끝나는 게 아니라, 하드웨어 성능이 발전할수록 공격자 쪽 계산 속도도 함께 빨라지므로 주기적으로 재검토하고, 필요하면 기존 해시를 재해시(re-hash)하는 마이그레이션 절차도 함께 준비해둡니다."

9. 면접 질문 3 — "이미 서비스가 SHA-1로 비밀번호를 저장 중인데 어떻게 마이그레이션하나요?"

면접 질문
"레거시 서비스가 이미 솔트 없는 SHA-1로 수백만 개의 비밀번호를 저장하고 있습니다. 전체 사용자에게 비밀번호를 강제로 재설정시키지 않고 bcrypt로 안전하게 전환할 수 있는 방법이 있을까요?"

왜 면접에서 묻는가?

이론적으로 "올바른 알고리즘"을 아는 것과, 기존 시스템을 서비스 중단 없이 안전하게 전환하는 실무 마이그레이션 설계를 할 수 있는지는 다른 능력이다. "전체 재설정 이메일 발송" 같은 단순 답변을 넘어, 점진적 전환(lazy migration) 패턴을 설계할 수 있는지를 본다.

답변 사고 순서

  1. 왜 전체 강제 재설정이 최선이 아닌지 짚기 — 사용자 경험 저하와 대량 이탈 위험을 지적한다.
  2. 로그인 시점을 활용한 점진적 전환 제시 — 사용자가 로그인할 때만 원문 비밀번호에 접근 가능하다는 점을 활용한다.
  3. 전환 기간 동안의 이중 방어 설계 — 아직 전환되지 않은 계정에 대한 별도 위험 완화 조치를 함께 제시한다.
강한 답변 예시

"전체 사용자에게 강제로 비밀번호를 재설정시키는 건 사용자 경험을 크게 해치고, 실제로 상당수는 재설정 이메일을 무시하거나 서비스 자체를 이탈할 위험이 있습니다. 대신 로그인 시점을 이용한 점진적 마이그레이션 (lazy migration)을 씁니다. 사용자가 로그인에 성공하는 순간, 서버는 이미 입력된 원문 비밀번호를 알고 있으니 그 값을 즉시 bcrypt로 재해시해서 DB의 해시값을 교체합니다. 즉 사용자는 평소처럼 로그인만 하면 되고, 백그라운드에서 알고리즘이 조용히 SHA-1에서 bcrypt로 전환되는 겁니다. 이 방식의 한계는 '로그인을 하지 않는 휴면 계정은 영원히 전환되지 않는다'는 점인데, 이건 일정 기간(예: 6개월) 로그인이 없는 계정에는 별도로 재설정을 유도하는 정책과 결합해서 처리합니다. 또한 전환이 끝나기 전까지는 SHA-1 해시가 여전히 DB에 남아있으니, 그 사이 DB가 유출될 경우를 대비해 Rate Limiting과 이상 로그인 탐지 같은 보완 방어도 함께 강화해둡니다."

약한 답변 예시

"전체 사용자에게 비밀번호 재설정 이메일을 보내서 새로 설정하게 하면 됩니다."

왜 약한가? 실행 가능한 방법이긴 하지만, 대규모 서비스에서의 사용자 이탈·불편 비용을 전혀 고려하지 않았고, 로그인 시점을 활용해 무중단으로 전환할 수 있는 더 나은 실무 패턴을 제시하지 못했다.

Senior-level Follow-up 질문들

Follow-up: "점진적 마이그레이션 도중 실제로 DB가 유출된다면, SHA-1로 아직 남아있는 계정들은 어떻게 보호하시겠습니까?"

모범 답변 방향: "저장 방식만으로는 이미 유출된 SHA-1 해시를 되돌릴 수 없으므로, 유출이 확인되면 아직 전환되지 않은 계정 전체를 대상으로 강제 비밀번호 재설정을 트리거합니다. 동시에 Day 6에서 다룬 계정 기준 Rate Limiting과 이상 로그인 탐지를 일시적으로 더 엄격하게 조정해서, 유출된 해시가 크래킹되어 크리덴셜 스터핑으로 악용되는 것에 대비합니다. 이 시나리오 자체가 '왜 강한 해시 알고리즘으로의 전환이 급하게 필요한 작업인지'를 보여주는 근거이기도 합니다 — 전환이 늦어질수록 유출 시 피해 범위가 넓어지기 때문입니다."

점검 항목미달 (주니어 수준)달성 (시니어 수준)
해시 vs 암호화 "둘 다 보안 강화 기술"로 뭉뚱그림 양방향/단방향 구조 차이와, 비밀번호는 원문 복원이 애초에 필요 없다는 이유를 근거로 설명
솔팅 "솔트를 붙이면 안전하다" 솔트가 막는 것(사전 계산)과 못 막는 것(실시간 brute-force)을 구분
느린 해시 함수 bcrypt/Argon2를 이름만 나열 계산 비용을 의도적으로 높여 brute-force 비용을 기하급수적으로 늘리는 원리와, 파라미터를 하드웨어 기준으로 벤치마크해 결정하는 이유를 설명
알고리즘 선택 "Argon2가 최신이니 무조건 좋다" bcrypt/scrypt/Argon2 각각이 무엇을(CPU/메모리/병렬도) 비싸게 만드는지 구분하고 상황별 선택 근거 제시
크리덴셜 스터핑 "강한 해시를 쓰면 크리덴셜 스터핑도 막힌다" 저장 방식(2차 피해 완화)과 크리덴셜 스터핑 방어(1차 침입 차단)가 서로 다른 방어선임을 구분하고 MFA/유출 비밀번호 대조 같은 보완책 제시
