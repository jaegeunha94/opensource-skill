1. 왜 "Secret 관리"가 별도의 인터뷰 주제인가

Day 7까지는 사용자의 비밀 정보(비밀번호)를 어떻게 저장할지를 다뤘다. Day 8은 시야를 넓혀 애플리케이션 자체가 사용하는 비밀 정보 — DB 접속 정보, 외부 API 키, 암호화 키, 서명 키, 클라우드 자격증명 — 을 어떻게 관리할지를 다룬다. 이 주제가 시니어 면접에서 자주 나오는 이유는, 실제 침해 사고의 상당수가 정교한 취약점 공격이 아니라 "Secret이 있어서는 안 될 곳에 그대로 노출되어 있었다"는 단순한 운영 실수에서 시작되기 때문이다.

실무 상황

2016년 우버(Uber)는 개발자가 사설이라고 착각한 GitHub 저장소에 AWS 자격증명을 하드코딩해 커밋했고, 공격자가 이를 발견해 S3에 저장된 5,700만 명의 사용자·운전자 데이터에 접근했다. 우버는 이 사실을 1년간 은폐하고 공격자에게 10만 달러를 지불한 사실까지 드러나 규제 기관으로부터 대규모 제재를 받았다. 더 흔한 패턴은 CI 빌드 로그에 환경 변수가 그대로 출력되거나, .env 파일이 실수로 git에 커밋되는 경우다. 코드 자체에는 취약점이 전혀 없어도, Secret 하나가 잘못된 곳에 노출되는 순간 그 뒤의 모든 방어(인증, 인가, 암호화)가 한꺼번에 무력화된다.

이 주제가 평가하는 것
  • Secret과 일반 설정값을 구분하는가 — "설정"과 "비밀"을 같은 방식으로 다루면 안 되는 이유를 아는가.
  • 노출 경로를 다각도로 열거할 수 있는가 — 소스코드, 로그, 프로세스 메모리, CI 아티팩트, 서브프로세스 상속 등 Secret이 새어나갈 수 있는 모든 지점을 떠올릴 수 있는가.
  • 정적 방어와 동적 방어를 함께 설계하는가 — "안 새어나가게 막는 것"뿐 아니라 "새어나갔을 때 피해를 최소화하는 것"(로테이션, 최소 권한, 감사 로그)까지 고려하는가.

2. 환경 변수(.env)의 한계 — 어디까지 새어나갈 수 있는가

전제 개념: 환경 변수는 왜 널리 쓰이는가?

환경 변수는 애플리케이션 설정값을 소스코드와 분리해 저장하는 가장 기본적인 방법이다. .env 파일에 DB_PASSWORD=xxxx 같은 값을 적어두고, 실행 시점에 프로세스 환경으로 불러온다. "코드에 하드코딩하지 않는다"는 원칙은 지키지만, 그렇다고 안전하다는 뜻은 아니다 — 환경 변수는 애초에 "비밀을 보호"하기 위해서가 아니라 "설정을 코드에서 분리"하기 위해 설계된 메커니즘이기 때문이다.

핵심 원리 — 환경 변수는 "낮은 권한 경계"를 가진 값이다

환경 변수는 다음과 같은 여러 경로로 노출될 수 있다.

  • 실수로 git 커밋 — .env가 .gitignore에서 누락되거나, 개발자가 급하게 git add .를 실행하는 순간 저장소 히스토리에 영구히 남는다. 한 번 커밋되면 rm으로 지워도 git history에는 그대로 남아있어, 전체 이력을 재작성(rebase/filter-branch)하지 않는 한 계속 조회 가능하다.
  • 프로세스 메모리·환경 덤프 — /proc/[pid]/environ, 코어 덤프, APM/에러 트래킹 도구(예: Sentry)가 예외 발생 시 프로세스 환경 전체를 함께 캡처해 외부 SaaS로 전송하는 경우가 흔하다.
  • 로그 출력 — 디버깅 목적으로 console.log(process.env)나 설정 객체 전체를 로깅하면 중앙 로그 수집 시스템에 평문 Secret이 영구히 저장된다.
  • 서브프로세스로의 상속 — 부모 프로세스의 환경 변수는 기본적으로 자식 프로세스에 그대로 상속된다. 신뢰할 수 없는 서드파티 스크립트나 빌드 도구를 실행하면 의도치 않게 Secret이 그쪽으로 전달될 수 있다.
  • 컨테이너 이미지에 굳어짐 — Dockerfile의 ENV 지시어로 Secret을 넣으면 이미지 레이어에 영구히 남아, 이미지를 받은 누구나 docker history로 값을 꺼낼 수 있다.
흔한 오해 — ".env를 .gitignore에 넣었으니 안전하다"

.gitignore는 미래의 실수를 막는 장치일 뿐, 이미 벌어진 노출을 막지 못한다. 과거에 한 번이라도 커밋된 적이 있다면 히스토리에 남아있고, 다른 개발자가 로컬에서 실수로 강제 추가 (git add -f)할 수도 있다. 또한 환경 변수의 근본적인 문제 — 접근 제어나 감사 로그가 없고, 값이 누구에게나 평문으로 보인다는 것 — 는 .gitignore와 무관하게 그대로 남아있다.

3. 전용 Secret 관리 시스템 — Vault, AWS Secrets Manager/KMS

전제 개념: 환경 변수와 전용 Secret 관리 시스템의 근본적 차이는 무엇인가?

환경 변수는 "값을 어딘가에 넣어두는 것"이고, 전용 Secret 관리 시스템(HashiCorp Vault, AWS Secrets Manager, AWS KMS, GCP Secret Manager 등)은 "누가, 언제, 어떤 값을 조회했는지"까지 통제하는 Secret 전용 인프라다. 핵심 차이는 세 가지다.

전용 Secret 관리 시스템이 추가로 제공하는 것
  • 1. 세밀한 접근 제어(Access Control) — 환경 변수는 프로세스를 실행할 수 있는 사람 누구나 값을 볼 수 있지만, Vault/Secrets Manager는 "이 서비스 계정만 이 경로의 Secret을 읽을 수 있다"는 정책을 IAM 수준에서 강제한다.
  • 2. 감사 로그(Audit Log) — 누가 어떤 Secret을 언제 조회했는지 전부 기록된다. 침해 사고 발생 시 "어떤 Secret이 노출됐는지" 역추적할 수 있다는 것 자체가 환경 변수와의 결정적 차이다.
  • 3. 동적 Secret(Dynamic Secret) — Vault의 대표 기능으로, DB 접속 정보 같은 것을 미리 고정된 값이 아니라 요청 시점에 짧은 TTL을 가진 임시 자격증명으로 발급한다. 이 자격증명은 일정 시간 후 자동으로 만료되므로, 유출되더라도 공격자가 악용할 수 있는 시간이 제한된다.
Vault vs AWS Secrets Manager — 무엇이 다른가

HashiCorp Vault는 클라우드 벤더에 종속되지 않는 범용 Secret 관리 솔루션으로, 동적 Secret, 암호화 as a Service(Transit Engine), 세밀한 정책 언어(HCL)를 제공한다. 다만 직접 운영(self-hosted)해야 하는 인프라 부담이 있다(HA 클러스터, Unseal 절차, 백업). AWS Secrets Manager는 AWS 관리형 서비스로 운영 부담이 거의 없고 IAM과 자연스럽게 통합되지만, AWS 생태계에 종속된다. 멀티 클라우드나 온프레미스가 섞인 환경, 혹은 벤더 종속을 피하고 싶은 조직은 Vault를, AWS 중심 인프라이고 운영 인력을 아끼고 싶다면 Secrets Manager를 선택하는 것이 일반적인 실무 기준이다. 

# 환경 변수 방식 — 정적 값, 접근 제어/감사 없음
DB_PASSWORD=hardcoded-value-forever

# Vault 동적 Secret 방식 — 요청마다 새 자격증명 발급, TTL 이후 자동 만료
$ vault read database/creds/readonly-role
Key                Value
---                -----
lease_id           database/creds/readonly-role/2f6a614...
lease_duration     1h
username           v-readonly-a1b2c3d4
password           A1a-temporary-random-password
흔한 오해 — "Vault를 도입하면 Secret 문제가 전부 해결된다"

Vault/Secrets Manager는 Secret을 안전하게 저장하고 배포하는 인프라일 뿐, 애플리케이션 코드가 그 Secret을 받은 이후 로그에 찍거나, 다른 서비스로 그대로 전달하거나, 메모리에 오래 들고 있는 것까지 막아주지는 않는다. 도구 도입과 별개로 "Secret을 다루는 코드의 습관"을 함께 바꿔야 한다.

4. Secret 로테이션 전략 — 정적 vs 동적 Secret

전제 개념: 왜 "안 새어나가게 막는 것"만으로는 부족한가?

아무리 접근 제어를 잘해도, 장기적으로 보면 Secret은 언젠가 노출될 확률이 0이 아니다 — 퇴사한 직원이 접근 권한을 그대로 갖고 있거나, 로그에 실수로 찍히거나, 서드파티 라이브러리의 취약점을 통해 유출될 수 있다. 로테이션(Rotation)은 "Secret이 결국 노출될 수 있다"는 전제 아래, 노출된 Secret의 유효 기간 자체를 짧게 만들어 피해 범위를 제한하는 전략이다.

정적 Secret vs 동적 Secret

정적 Secret은 한 번 발급되면 사람이 수동으로 바꾸기 전까지 영구히 유효한 값이다(예: DB 계정 비밀번호를 만들어두고 몇 년간 그대로 사용). 동적 Secret은 Section 3에서 다룬 것처럼 요청 시점에 짧은 TTL로 발급되어 자동 만료된다. 정적 Secret이 유출되면 누군가 발견하고 수동으로 폐기하기 전까지 무기한 악용될 수 있지만, 동적 Secret은 유출되어도 TTL이 지나면 자동으로 무효화되므로 "발견까지 걸리는 시간"이 곧 피해 규모를 결정하지 않는다.

무중단 로테이션 설계의 핵심 — "이중 유효 기간(Overlap Window)"
  • 1. 새 Secret 발급 — 기존 Secret은 그대로 둔 채 새 Secret을 추가로 발급한다.
  • 2. 두 Secret이 동시에 유효한 전환 기간을 둔다 — 이 기간 동안 모든 서비스 인스턴스가 새 Secret으로 순차적으로 전환하도록 배포한다. 한 번에 모든 인스턴스를 동시에 전환하면, 일부 인스턴스가 아직 재시작 전이라 옛 Secret을 쓰는 상태에서 옛 Secret을 바로 폐기했을 때 서비스 장애로 이어진다.
  • 3. 모든 인스턴스의 전환 확인 후 이전 Secret 폐기 — 배포/헬스체크로 전체 전환이 끝난 것을 확인한 뒤에야 옛 Secret을 무효화한다.
흔한 오해 — "로테이션은 유출 사고가 났을 때만 하면 된다"

사고 대응용 긴급 로테이션과, 평상시 정기 로테이션은 목적이 다르다. 정기 로테이션은 "혹시 아무도 모르게 이미 유출되어 있을 수도 있는 Secret"의 유효 기간을 근본적으로 줄여, 탐지되지 않은 유출의 악용 가능 시간 자체를 구조적으로 제한하는 예방적 조치다. 사고가 난 뒤에만 로테이션하는 건 "이미 알려진 유출"에만 대응하는 것이고, 시니어 면접에서는 이 차이를 구분해서 설명할 수 있는지를 본다.

5. CI/CD 파이프라인에서의 Secret 노출 위험

전제 개념: CI/CD가 왜 Secret 노출의 새로운 공격 표면인가?

CI/CD 파이프라인은 배포를 위해 클라우드 자격증명, 패키지 레지스트리 토큰, 서명 키 같은 강력한 권한의 Secret을 다룬다. 문제는 CI 파이프라인이 사람이 직접 작성한 코드뿐 아니라, 서드파티 액션/플러그인, 그리고 때로는 외부 기여자의 PR 코드까지 실행한다는 점이다 — Secret을 다루는 위치치고는 신뢰 경계가 상당히 넓다.

핵심 위험 — 빌드 로그와 fork PR
  • 빌드 로그 노출 — 스크립트가 echo $SECRET_TOKEN이나 디버그용 env 출력을 실행하면, CI 로그는 팀 전체(때로는 공개 저장소라면 누구나)에게 보이는 곳에 평문으로 영구히 남는다. 대부분의 CI 시스템은 Secret로 등록된 값이 로그에 그대로 찍히면 자동으로 마스킹하지만, 값을 base64 인코딩하거나 변형해서 출력하면 이 마스킹을 우회할 수 있다.
  • Fork PR을 통한 Secret 접근 — GitHub Actions 등에서 외부 기여자가 올린 fork 저장소의 PR이 pull_request_target처럼 원본 저장소의 Secret에 접근 가능한 이벤트로 워크플로를 트리거하면, PR 코드(악의적으로 조작 가능)가 그대로 실행되며 조직의 Secret에 접근할 수 있다.
  • 서드파티 GitHub Action 공급망 위험 — 2024년 초 발생한 tj-actions/changed-files 사고처럼, 널리 쓰이는 서드파티 액션 자체가 침해되어 악성 코드가 삽입되면, 그 액션을 사용하는 모든 파이프라인의 Secret이 한꺼번에 유출될 수 있다.
실무 방어
  • 1. Secret 최소 노출 범위 — 워크플로 단위가 아니라 필요한 job/step에만 Secret을 스코프하고, fork PR 트리거 이벤트에는 기본적으로 Secret이 주입되지 않는 GitHub Actions 기본 동작을 그대로 유지한다(pull_request_target 사용을 최소화).
  • 2. 서드파티 액션 버전 고정(pinning) — 태그(@v4)가 아니라 커밋 해시로 액션 버전을 고정해, 업스트림이 침해되어 태그가 재발행되어도 우리 파이프라인은 영향을 받지 않게 한다.
  • 3. 짧은 TTL의 CI 전용 자격증명 — 영구 Access Key 대신 OIDC 연동을 통해 CI 실행마다 단기 자격증명을 발급받는 방식(AWS의 GitHub OIDC 연동 등)을 사용해, 자격증명이 CI 시스템에 영구히 저장되지 않게 한다.
  • 4. 로그 마스킹 신뢰하지 않기 — 자동 마스킹에만 의존하지 말고, 코드 리뷰 단계에서 Secret을 출력하는 디버그 코드가 남아있지 않은지 별도로 점검한다.

6. 암호화 키 관리(KMS)와 Envelope Encryption

전제 개념: 암호화 키 자체는 어디에 두어야 하는가?

Day 7에서 "암호화가 필요한 데이터(카드번호 등)"를 언급했는데, 그 암호화에 쓰는 키 자체를 어디에 보관할 것인가는 또 다른 Secret 관리 문제다. 이 문제를 실무에서 표준적으로 해결하는 패턴이 Envelope Encryption(봉투 암호화)다.

핵심 원리 — 키를 계층화해서 "루트 키는 절대 밖으로 나가지 않게" 만든다

데이터를 암호화할 때마다 KMS(Key Management Service)에 매번 원문을 보내 암호화를 요청하면 성능 부담이 크고, 네트워크 경로에 원문이 노출될 위험도 생긴다. Envelope Encryption은 이를 2단계로 분리한다.

  • 1. DEK(Data Encryption Key) 생성 — 실제 데이터를 암호화하는 키(DEK)는 애플리케이션 서버에서 매번 새로 생성해 로컬에서 빠르게 데이터를 암호화한다.
  • 2. KEK(Key Encryption Key)로 DEK를 암호화 — DEK 자체는 KMS에 안전하게 보관된 상위 키(KEK, 루트 키)로 다시 암호화해서, "암호화된 DEK"만 데이터와 함께 저장한다. KEK는 KMS/HSM 밖으로 절대 평문으로 나오지 않는다 — 이 키는 하드웨어 보안 모듈(HSM) 안에서만 연산에 사용된다.
  • 3. 복호화 시 역순으로 진행 — 저장된 "암호화된 DEK"를 KMS에 보내 복호화를 요청하면 KMS가 평문 DEK를 반환하고, 그 DEK로 실제 데이터를 복호화한다.
# Envelope Encryption 흐름
plaintext_data = "1234-5678-9012-3456"

# 1) 매번 새 DEK 생성, 이 DEK로 데이터를 로컬에서 암호화 (빠름)
dek = generate_random_key()
encrypted_data = AES_encrypt(plaintext_data, dek)

# 2) DEK 자체는 KMS의 루트 키(KEK)로 암호화 — KEK는 HSM 밖으로 나오지 않음
encrypted_dek = kms.encrypt(dek, key_id="root-kek")

# 3) 저장은 "암호화된 데이터 + 암호화된 DEK" 세트로
store(encrypted_data, encrypted_dek)

# 복호화 시 KMS에 encrypted_dek만 보내 평문 DEK를 받아온다
dek = kms.decrypt(encrypted_dek, key_id="root-kek")
plaintext_data = AES_decrypt(encrypted_data, dek)
이 구조가 주는 실무적 이점

DB가 통째로 유출돼도 공격자는 "암호화된 데이터 + 암호화된 DEK"만 얻는다. KEK는 애초에 DB나 애플리케이션 서버 어디에도 평문으로 존재한 적이 없으므로, KMS/HSM 자체가 별도로 침해되지 않는 한 데이터를 복호화할 방법이 없다. 또한 키 로테이션 시 모든 데이터를 새 키로 다시 암호화할 필요 없이, KEK만 교체하고 DEK를 새 KEK로 다시 감싸는 것만으로 로테이션이 끝나 대규모 재암호화 비용을 피할 수 있다.

7. 면접 질문 1 — ".env 파일에 다 넣으면 되지 않나요?"

면접 질문
"저희 팀은 DB 비밀번호, API 키 전부 .env 파일에 넣고 .gitignore로 관리하고 있는데, 이 정도면 충분한 것 아닌가요? 왜 Vault 같은 걸 따로 도입해야 하나요?"

왜 면접에서 묻는가?

"커밋만 안 되면 안전하다"는 좁은 시각을 갖고 있는지, 아니면 Secret 노출 경로를 다각도로 열거하고 조직 규모에 따라 필요한 통제 수준이 달라진다는 것을 이해하는지 확인한다.

답변 사고 순서

  1. .env가 막는 것과 못 막는 것 구분 — git 커밋 실수는 어느 정도 예방하지만, 로그·프로세스 덤프·서브프로세스 상속 등 다른 경로는 그대로 남아있음을 짚는다.
  2. 조직 규모에 따른 리스크 변화 제시 — 팀이 커질수록 "누가 이 Secret에 접근할 수 있었는지" 답할 수 없다는 것 자체가 감사·컴플라이언스 리스크가 됨을 짚는다.
  3. 결론 — 전용 Secret 관리 도구가 주는 추가 가치(접근 제어, 감사 로그, 동적 Secret)로 마무리한다.
강한 답변 예시

".env와 .gitignore 조합은 '실수로 git에 커밋되는 것'은 어느 정도 막아주지만, Secret 노출 경로는 그것 말고도 많습니다. 로그에 환경 변수 전체가 찍히거나, APM 도구가 예외 발생 시 프로세스 환경을 통째로 캡처해 외부로 보내거나, 신뢰할 수 없는 서브프로세스로 환경 변수가 그대로 상속되는 경우도 있습니다. 더 근본적인 문제는 .env 방식에는 '누가 언제 이 값을 조회했는지'에 대한 감사 로그가 전혀 없다는 겁니다. 팀 규모가 작을 때는 이게 크게 문제되지 않지만, 팀이 커지고 퇴사자가 생기고 컴플라이언스 요구사항이 생기면 '이 API 키에 누가 접근할 수 있었는가'에 답할 수 있어야 하는데 .env로는 그게 불가능합니다. Vault나 AWS Secrets Manager는 접근 정책과 감사 로그를 기본 제공하고, 특히 Vault는 DB 자격증명 같은 걸 요청 시점마다 짧은 TTL로 발급하는 동적 Secret 기능이 있어서, 설령 유출돼도 피해가 시간적으로 제한됩니다."

약한 답변 예시

"Vault가 더 최신 기술이고 대기업들이 많이 쓰니까 저희도 도입해야 한다고 생각합니다."

왜 약한가? "최신이라서", "대기업이 써서"라는 근거는 실무 판단이 아니라 유행을 따르는 것에 불과하다. .env가 실제로 막지 못하는 구체적인 노출 경로와, 전용 도구가 그 경로들을 어떻게 해결하는지를 설명하지 못했다.

Senior-level Follow-up 질문들

Follow-up: "스타트업처럼 작은 팀에서도 처음부터 Vault를 도입해야 할까요?"

모범 답변 방향: "무조건 그렇지는 않습니다. 팀이 작고 인프라가 단순한 초기 단계에는 Vault를 직접 운영하는 것 자체가 오버엔지니어링일 수 있습니다. 이 단계에서는 클라우드 관리형 Secret 서비스(AWS Secrets Manager, GCP Secret Manager)처럼 운영 부담이 적은 선택지가 더 실용적입니다. 중요한 건 도구 이름이 아니라 '접근 제어와 감사 로그가 있는 중앙화된 저장소를 쓰는가'라는 원칙이고, 이 원칙은 팀 규모와 무관하게 처음부터 지키는 게 나중에 마이그레이션하는 비용보다 쌉니다."

8. 면접 질문 2 — "Vault와 AWS Secrets Manager 중 어떤 걸 선택하시겠습니까?"

면접 질문
"우리 회사는 AWS를 주력으로 쓰고 있고, 앞으로 멀티 클라우드 전환 계획도 있습니다. Secret 관리 시스템으로 HashiCorp Vault와 AWS Secrets Manager 중 무엇을 선택하시겠습니까?"

왜 면접에서 묻는가?

정답이 하나로 고정된 질문이 아니다. 운영 부담, 벤더 종속성, 조직의 인프라 로드맵이라는 여러 축의 trade-off를 놓고 근거 있게 판단할 수 있는지, 그리고 "회사의 향후 계획"이라는 맥락 정보를 답변에 실제로 반영하는지를 본다.

답변 사고 순서

  1. 현재 상태 기준 판단 — AWS 중심이라는 현재 조건에서는 Secrets Manager가 통합 비용이 낮다는 점을 짚는다.
  2. 미래 계획 반영 — 멀티 클라우드 전환이 예정되어 있다면 AWS 종속 도구는 나중에 재작업 비용을 유발할 수 있음을 짚는다.
  3. 결론 — 전환 시점과 비용을 고려한 단계적 접근(지금은 Secrets Manager, 전환 시점에 Vault로 이전) 또는 처음부터 Vault를 선택하는 근거를 제시한다.
강한 답변 예시

"지금 당장은 AWS 중심 인프라이니 Secrets Manager가 IAM과 자연스럽게 통합되고 운영 부담도 적어서 더 실용적인 선택입니다. 하지만 질문에서 언급하신 것처럼 멀티 클라우드 전환 계획이 있다면 얘기가 달라집니다. Secrets Manager로 이미 구축한 뒤 나중에 멀티 클라우드로 가면, Secret을 다루는 애플리케이션 코드와 배포 파이프라인을 전부 다시 손봐야 하는 비용이 발생합니다. 그래서 저라면 전환 계획의 시점을 먼저 확인하겠습니다. 1~2년 내 확정된 계획이라면 처음부터 Vault를 self-hosted 또는 HCP Vault(관리형)로 도입해서 클라우드 벤더에 종속되지 않는 추상화 계층을 만들어두는 게 장기적으로 이득입니다. 반대로 전환이 막연한 로드맵 수준이라면, 지금은 Secrets Manager로 빠르게 가고 실제 전환이 확정되는 시점에 마이그레이션 계획을 별도로 세우는 게 더 합리적입니다."

약한 답변 예시

"Vault가 기능이 더 많으니까 Vault를 선택하겠습니다."

왜 약한가? "기능이 많다"는 것은 선택 기준이 될 수 없다 — 그 기능이 실제로 필요한지, 그리고 self-hosted 운영 부담이라는 비용을 감당할 수 있는지를 따지지 않았다. 질문에 명시된 "AWS 중심 현재 상태"와 "멀티 클라우드 전환 계획"이라는 맥락 정보를 답변에 전혀 반영하지 못했다.

Senior-level Follow-up 질문들

Follow-up: "Secrets Manager에서 Vault로 나중에 마이그레이션한다면, 어떤 점을 가장 조심해야 할까요?"

모범 답변 방향: "가장 위험한 구간은 전환 기간 동안 두 시스템이 동시에 존재하는 시점입니다. 일부 서비스는 아직 Secrets Manager를 참조하고 일부는 Vault로 넘어간 상태에서, 어느 한쪽의 Secret이 로테이션되면 다른 쪽이 옛 값을 참조하는 불일치가 생길 수 있습니다. 그래서 서비스별로 점진적으로 전환하되, 전환 기간 동안은 두 시스템의 Secret 값을 동기화하거나, 애플리케이션이 '어느 Secret 저장소를 쓸지'를 설정으로 추상화해 한 서비스씩 안전하게 전환 검증한 뒤 다음 서비스로 넘어가는 방식을 씁니다."

9. 면접 질문 3 — "GitHub Actions에서 fork PR을 통한 Secret 유출, 어떻게 막으시겠습니까?"

면접 질문
"오픈소스 프로젝트를 운영 중인데, 외부 기여자가 fork에서 PR을 보냈을 때 CI가 실행되면서 우리 조직의 Secret에 접근할 수 있는 구조라는 걸 알게 됐습니다. 어떻게 설계를 바꾸시겠습니까?"

왜 면접에서 묻는가?

CI/CD를 "코드를 실행하는 파이프라인"으로만 보는 게 아니라 신뢰 경계(trust boundary)가 걸린 인프라로 인식하는지를 확인한다. fork PR이라는 시나리오는 "누구나 실행시킬 수 있는 코드"와 "조직의 강력한 권한을 가진 Secret"이 만나는 지점이라는 걸 짚어낼 수 있는지가 핵심이다.

답변 사고 순서

  1. 왜 이 구조가 위험한지 원인 규명 — 어떤 트리거 이벤트(예: pull_request_target)가 fork PR 코드에 Secret 접근 권한까지 부여하는지 짚는다.
  2. 신뢰 경계 재설계 — Secret이 필요한 작업과 외부 코드를 실행하는 작업을 분리한다.
  3. 단계적 승인 절차 추가 — 외부 기여자의 PR은 유지관리자의 명시적 승인 후에만 Secret이 필요한 단계가 실행되도록 게이트를 둔다.
강한 답변 예시

"먼저 원인을 짚으면, GitHub Actions의 pull_request_target 이벤트는 fork PR이 트리거해도 원본 저장소의 Secret과 컨텍스트에 접근할 수 있게 설계되어 있어서, 외부 기여자가 악의적인 코드를 PR에 넣으면 그 코드가 우리 Secret에 접근한 채로 실행될 수 있습니다. 이걸 막기 위해 우선 워크플로를 두 단계로 분리합니다. 빌드/테스트처럼 Secret이 필요 없는 단계는 일반 pull_request 이벤트로 fork 코드를 그대로 실행하되 이때는 Secret이 애초에 주입되지 않는 GitHub 기본 동작을 그대로 둡니다. 배포나 코멘트 작성처럼 Secret이 필요한 단계는 별도 워크플로로 분리해서, 유지관리자가 PR 내용을 검토하고 명시적으로 승인한 뒤에만(예: workflow_run 트리거로 신뢰된 base 브랜치 컨텍스트에서 실행하거나, environment protection rule로 수동 승인 게이트를 걸어) 실행되도록 만듭니다. 이렇게 하면 외부 코드가 Secret에 접근하려면 반드시 사람의 검토를 거치게 됩니다."

약한 답변 예시

"외부 기여자의 PR은 아예 CI를 안 돌리면 되지 않을까요?"

왜 약한가? 오픈소스 프로젝트에서 외부 기여 PR에 CI(빌드/테스트)를 돌리지 않으면 코드 품질 검증 자체가 불가능해진다. "위험을 없앤다"는 명목으로 프로젝트의 핵심 워크플로 자체를 포기하는 답변으로, Secret이 필요한 단계와 필요 없는 단계를 분리한다는 더 정교한 설계를 제시하지 못했다.

Follow-up: "서드파티 GitHub Action 자체가 침해되는 공급망 공격은 이 설계로 막을 수 있나요?"

모범 답변 방향: "아니요, fork PR 격리와는 별개의 위험입니다. 서드파티 액션이 침해되면 그 액션을 신뢰하고 실행하는 우리 워크플로 자체가 오염되므로, Secret 노출 단계를 분리해도 그 단계에서 침해된 액션을 쓰고 있다면 그대로 노출됩니다. 이건 액션 버전을 태그가 아니라 커밋 해시로 고정하고, 의존하는 서드파티 액션의 수를 최소화하며, 가능하면 조직에서 직접 관리하는 fork나 내부 검증을 거친 액션만 사용하는 별도의 공급망 방어로 대응해야 합니다. fork PR 격리와 서드파티 액션 신뢰는 서로 다른 신뢰 경계 문제라서, 하나를 막았다고 다른 하나가 해결되지 않습니다."
