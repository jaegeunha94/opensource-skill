1. 왜 "프로필 사진 업로드" 같은 사소한 기능이 심각한 침해로 이어지는가

Day 4까지는 브라우저에서 벌어지는 공격(XSS, CSRF)을 다뤘다. 이번 레슨은 방향이 반대다 — 서버가 "사용자가 준 데이터"를 직접 저장하거나, 그 데이터를 근거로 스스로 요청을 보내는 지점의 위협이다. 파일 업로드와 SSRF(Server-Side Request Forgery)는 서로 다른 취약점이지만, 면접에서 묶어서 묻는 이유가 있다. 둘 다 "이 정도는 편의 기능이지 위험할 게 있나?"라는 방심에서 출발하기 때문이다.

실무 상황

한 SaaS 서비스가 사용자 프로필 사진 업로드 기능을 만들면서 "이미지 파일만 허용하면 되겠지"라는 생각으로 확장자가 .jpg, .png인지만 검사했다. 공격자는 내용은 PHP 웹쉘 코드이고 이름만 profile.php.jpg인 파일을 업로드했고, 서버 설정 실수로 업로드 디렉터리가 실행 가능한 경로였던 탓에 원격 코드 실행으로 이어졌다. 같은 서비스는 또 "외부 이미지 URL을 넣으면 미리보기를 만들어주는" 기능도 제공했는데, 공격자가 그 URL에 http://169.254.169.254/latest/meta-data/iam/security-credentials/를 넣어 클라우드 자격 증명을 그대로 탈취했다. 두 사고 모두 "이 기능이 서버 내부/외부 자원에 접근하는 다리가 될 수 있다"는 인식이 없었던 것이 근본 원인이다.

시니어에게 기대하는 것은 "파일 확장자를 검사하세요", "내부 IP는 막으세요" 같은 단편적 체크리스트가 아니라, 이 기능이 서버에게 무엇을 하도록 시키는지(저장 후 나중에 실행될 수 있는가? 서버가 임의의 목적지로 요청을 보내는가?)를 먼저 식별하고, 그 신뢰 경계를 어디에 세울지 설계하는 능력이다.

흔한 오해 — "이건 이미지/파일 처리 기능이니 SQL Injection이나 XSS 같은 '진짜 보안 문제'와는 다르다"

파일 업로드와 SSRF는 OWASP Top 10에서도 매년 순위가 오르는 항목이다. 이유는 명확하다 — 클라우드 네이티브 아키텍처에서 "파일을 받는다", "외부 리소스를 가져온다"는 기능이 폭발적으로 늘었고, 이 기능들이 결국 서버가 스스로의 신뢰 경계 안에서 임의의 코드를 실행하거나, 임의의 내부 자원에 접근하게 만드는 통로가 되기 때문이다. "파일/URL 처리"는 부가 기능이 아니라 공격 표면(attack surface) 그 자체다.

2. 파일 업로드의 위협 모델 — 확장자 검증이 왜 충분하지 않은가

전제 개념: 파일의 "확장자"와 "실제 내용물"은 전혀 다른 정보다

확장자는 파일명 끝에 붙은 문자열일 뿐이고, 운영체제나 서버가 "이 파일이 무슨 형식인지 참고용으로" 쓰는 힌트에 불과하다. 실제로 파일이 이미지인지 실행 코드인지는 파일 내용(바이트 스트림)의 구조로 결정된다. 공격자는 파일명을 profile.php.jpg, profile.jpg(내용은 PHP), 심지어 profile.jpg%00.php(널 바이트 삽입, 오래된 파서에서 문자열이 조기 종료되는 버그 악용)처럼 자유롭게 만들 수 있다. 

# 확장자만 검사하는 취약한 코드 (의사코드)
if filename.endswith(('.jpg', '.png', '.gif')):
    save(file, upload_dir + filename)

# 공격 파일명: shell.php.jpg
# → endswith('.jpg') 통과
# → 파일 내용은 아래와 같은 PHP 웹쉘 코드
# 
# 서버 설정에 따라 .php.jpg도 PHP로 실행되거나(Apache 다중 확장자 처리 버그),
# 혹은 업로드 폴더 경로 자체가 실행 가능하게 설정되어 있으면 직접 접근으로 실행됨
핵심 원리

파일 업로드 취약점이 실제 침해로 이어지려면 보통 두 조건이 함께 필요하다. (1) 검증되지 않은 내용이 서버에 저장되고, (2) 그 저장 위치가 웹 서버에 의해 실행 가능한 경로여야 한다. 전자만 있으면 "위험한 파일이 서버에 있다" 수준이고, 후자까지 겹치면 원격 코드 실행(RCE)으로 격상된다. 이 두 조건을 분리해서 생각하는 것이 방어 설계의 출발점이다.

흔한 오해 — "MIME 타입(Content-Type 헤더)을 검사하면 안전하다"

업로드 요청의 Content-Type 헤더는 클라이언트(브라우저나 API 호출자)가 자기 마음대로 적어 보내는 값이다. 공격자는 PHP 웹쉘 파일을 올리면서 Content-Type: image/jpeg라고 그냥 선언하면 그만이다. 서버가 이 헤더를 그대로 신뢰하는 것은, 사용자가 "저는 정상적인 사람입니다"라고 자기소개한 것을 신원 확인으로 인정하는 것과 같다. 신뢰할 수 있는 판단은 서버가 직접 파일의 실제 바이트를 열어서 확인(매직 바이트 검증)했을 때만 가능하다.

3. 파일 업로드 방어 — 계층적 검증과 저장 구조 분리

전제 개념: 매직 바이트(Magic Bytes)란 무엇인가?

대부분의 파일 형식은 파일 맨 앞 몇 바이트에 "이 파일은 이런 형식이다"를 나타내는 고정된 시그니처를 갖는다. 예를 들어 PNG는 항상 89 50 4E 47로 시작하고, JPEG는 FF D8 FF로 시작한다. 이건 파일명이나 헤더처럼 요청자가 임의로 조작하기 어려운, 실제 파일 구조에 새겨진 값이다. 서버가 업로드된 파일을 열어 이 시그니처를 직접 확인하면, "이름과 헤더가 뭐라고 주장하든 실제로 이미지 포맷 구조를 따르는가"를 검증할 수 있다. 

# 매직 바이트 검증 예시 (의사코드)
def is_valid_png(file_bytes):
    return file_bytes[:8] == b'\x89PNG\r\n\x1a\n'

# 파일명이 profile.png이고 Content-Type이 image/png라고 주장해도,
# 실제 첫 8바이트가 PNG 시그니처와 다르면 거부한다.
# 완벽하지는 않다 — 폴리글랏(polyglot) 파일(유효한 PNG이면서 동시에 유효한 PHP로도 해석되는 파일)
# 같은 정교한 우회가 존재하므로, 이것도 단일 방어선으로 삼으면 안 된다.
계층적 방어(Defense in Depth) — 파일 업로드
  • 1차 방어 — 매직 바이트/실제 디코딩 검증: 확장자·Content-Type이 아니라 파일 내용을 실제로 열어 형식을 검증. 이미지라면 실제로 디코딩(리사이즈 등 재인코딩)까지 성공해야 통과시켜, 폴리글랏 우회 가능성을 줄인다.
  • 2차 방어 — 저장 위치를 웹 서버 실행 경로 밖으로 분리: 업로드 파일은 애플리케이션 서버가 아닌 별도 스토리지(S3, GCS 같은 오브젝트 스토리지)에 저장하고, 애초에 실행 권한이 없는 환경에 둔다.
  • 3차 방어 — 파일명을 서버가 재생성: 사용자가 지정한 원본 파일명을 그대로 쓰지 않고, 서버에서 생성한 무작위 ID + 검증된 확장자로 저장해 경로 조작(path traversal, ../../etc/passwd 같은)을 원천 차단한다.
  • 4차 방어 — 별도 도메인/서브도메인에서 서빙: 업로드 파일을 서비스 본 도메인과 다른 도메인(예: usercontent.example.com)에서 제공해, 설령 악성 HTML/SVG 파일이 저장돼도 same-origin policy에 의해 본 서비스의 쿠키·세션에 접근하지 못하게 격리한다.
  • 5차 방어 — 업로드 파일에 대한 스캔/크기 제한: 안티바이러스 스캔, 파일 크기 상한, 업로드 빈도 제한(DoS 방지)을 추가한다.
실무에서 자주 놓치는 지점 — SVG와 이미지 리사이징 라이브러리

SVG는 XML 기반이라 이미지처럼 보이지만 그 안에 <script> 태그를 담을 수 있는 사실상 HTML 문서다. "이미지 확장자니까 안전하다"고 SVG를 그대로 허용하고 브라우저에서 직접 열람하게 하면 Stored XSS로 직결된다. 또한 이미지를 안전하게 만들려고 쓰는 리사이징 라이브러리(ImageMagick 등) 자체에 파서 취약점이 있었던 사례(예: ImageTragick)도 있어서, "라이브러리를 거치니 안전해졌다"는 가정도 라이브러리 버전 관리와 함께 다시 점검해야 한다.

4. SSRF의 원리 — "서버가 대신 요청해주는 기능"이 내부망 침투 통로가 되는 이유

전제 개념: 서버는 왜 "다른 곳에 요청을 대신 보내주는" 기능을 갖게 되는가?

URL 미리보기, 웹훅 등록, 이미지를 외부 URL에서 가져와 리사이징, 다른 마이크로서비스 호출 등 — 현대 백엔드는 사용자가 준 URL이나 식별자를 근거로 서버 자신이 HTTP 요청을 만들어 보내는 기능을 흔히 가진다. 문제는 이 요청을 보내는 주체가 사용자의 브라우저가 아니라 서버 자신이라는 점이다. 서버는 보통 사설 IP 대역(내부망), localhost, 클라우드 메타데이터 엔드포인트에 접근 권한을 갖고 있는데, 사용자가 이 URL을 자유롭게 지정할 수 있다면 공격자가 "서버의 신뢰받은 위치"에서 임의의 목적지로 요청을 대신 보내게 만들 수 있다. 

# 취약한 URL 미리보기 기능 (의사코드)
@app.post("/preview")
def preview(url: str):
    response = requests.get(url)   # 사용자가 준 URL을 서버가 그대로 요청
    return extract_thumbnail(response.content)

# 정상 사용: url = "https://news.example.com/article/123"
# 공격 사용: url = "http://169.254.169.254/latest/meta-data/iam/security-credentials/my-role"
# 서버는 클라우드 내부 메타데이터 서비스에 "자신의 권한으로" 접근해서 그 응답을 그대로 반환한다
핵심 원리

SSRF는 XSS·CSRF처럼 브라우저의 신뢰 모델을 악용하는 것이 아니라, 서버 자신의 네트워크 위치와 권한을 악용한다. 공격자는 외부에서는 절대 도달할 수 없는 내부망 IP, localhost, 클라우드 메타데이터 서비스에 "서버를 대리인 삼아서" 접근한다. 서버가 이런 요청을 만들 수 있는 기능을 가진 순간, 그 기능은 곧 외부 공격자에게 내부망 진입 통로를 열어주는 프록시가 될 수 있다.

흔한 오해 — "URL 유효성 검사(정상적인 http/https URL인지)만 하면 SSRF는 안전하다"

http://169.254.169.254/도 완벽하게 유효한 형식의 http URL이다. "URL 문법이 올바른가"와 "이 URL이 가리키는 목적지가 접근을 허용해도 되는 곳인가"는 완전히 다른 검증이다. SSRF 방어는 목적지의 신뢰 여부를 판단하는 것이지, URL의 형식을 판단하는 것이 아니다.

5. 클라우드 메타데이터 엔드포인트 공격 — SSRF의 대표 실전 사례

전제 개념: 클라우드 메타데이터 서비스란 무엇인가?

AWS, GCP, Azure 같은 클라우드는 각 인스턴스(서버) 내부에서만 접근 가능한 특수한 링크-로컬 IP 주소 (169.254.169.254)로 "이 인스턴스에 대한 메타데이터"를 제공한다. 여기에는 인스턴스에 부여된 IAM 역할의 임시 자격 증명(access key, secret key, session token)이 포함될 수 있다. 이건 인스턴스 자신이 다른 AWS 서비스(S3, DynamoDB 등)를 호출할 때 쓰라고 만든 정상 기능이다 — 문제는 이 엔드포인트가 "인스턴스 내부에서 요청하면 인증 없이 응답한다"는 점이다. 

# AWS EC2 메타데이터 엔드포인트로 IAM 자격 증명 탈취 (SSRF를 통해)
GET http://169.254.169.254/latest/meta-data/iam/security-credentials/
→ 응답: "my-app-role"

GET http://169.254.169.254/latest/meta-data/iam/security-credentials/my-app-role
→ 응답: {
    "AccessKeyId": "AKIA...",
    "SecretAccessKey": "...",
    "Token": "...",
    "Expiration": "..."
  }
# 공격자는 이 자격 증명으로 해당 인스턴스가 접근 가능한 모든 AWS 리소스(S3 버킷, RDS 등)에
# 서버를 거치지 않고 직접 접근할 수 있게 된다
실제 사고 사례가 보여주는 심각성

2019년 Capital One 침해 사고는 잘못 설정된 WAF를 통한 SSRF로 EC2 인스턴스의 메타데이터 엔드포인트에 접근해 IAM 자격 증명을 탈취하고, 그 자격 증명으로 S3에 저장된 1억 건 이상의 고객 데이터를 유출한 사례로 널리 알려져 있다. SSRF 하나가 "웹 애플리케이션 취약점"에서 "클라우드 인프라 전체의 데이터 유출"로 격상될 수 있음을 보여주는 대표 사례로, 시니어 면접에서 SSRF의 심각성을 설명할 때 근거로 들 수 있다.

방어의 핵심 방향 — IMDSv2

AWS는 이 문제에 대응해 IMDSv2를 도입했다. 기존 방식(IMDSv1)은 단순 GET 요청만으로 메타데이터에 접근할 수 있었지만, IMDSv2는 먼저 PUT 요청으로 세션 토큰을 발급받고, 그 토큰을 헤더에 담아야만 메타데이터에 접근할 수 있게 만들었다. 일반적인 SSRF(서버가 사용자 URL을 그대로 GET 요청으로 전달하는 형태)는 커스텀 헤더와 PUT 메서드를 함께 조작하기 어렵기 때문에, 이 변경만으로 상당수의 단순 SSRF 공격이 막힌다. 다만 이건 "메타데이터 탈취"라는 결과를 어렵게 만드는 것이지, SSRF 자체(내부망 다른 자원 접근)를 막는 것은 아니라는 점을 구분해야 한다.

6. SSRF 방어 — 네트워크 경계와 애플리케이션 경계를 함께 설계

SSRF 방어는 애플리케이션 레벨(입력 검증)만으로는 불완전하다. 네트워크 구조 자체가 "설령 애플리케이션 검증이 뚫려도 도달할 수 없는 곳"을 만들어야 근본적으로 안전하다.

계층적 방어(Defense in Depth) — SSRF
  • 1차 방어 — 목적지 allowlist(허용 목록): "이 기능이 실제로 필요로 하는 목적지"를 도메인/IP 단위로 명시적으로 화이트리스트에 등록하고, 그 외는 전부 거부한다. 블랙리스트(내부 IP 대역만 차단)보다 화이트리스트가 근본적으로 안전하다.
  • 2차 방어 — DNS resolution 이후 IP 재검증: 도메인 이름만 검사하고 실제 요청은 별도로 DNS를 다시 조회해서 보내면, 그 사이에 DNS 응답이 바뀌는 "DNS Rebinding" 공격에 노출된다. 검증에 사용한 IP와 실제 요청에 사용하는 IP가 같은지 확인해야 한다.
  • 3차 방어 — 네트워크 세그멘테이션: 외부 요청을 대신 만드는 서버(예: URL 미리보기 서비스)는 별도의 네트워크 구간에 배치하고, 그 구간에서는 내부망 리소스(DB, 관리자 API, 메타데이터 엔드포인트)에 대한 라우팅/방화벽 규칙 자체를 차단한다. 애플리케이션 코드의 실수와 무관하게 네트워크 레벨에서 막는 것이 핵심이다.
  • 4차 방어 — 리다이렉트 추적 제한: 서버가 만드는 HTTP 클라이언트가 리다이렉트를 자동으로 따라가면, 화이트리스트를 통과한 URL이 응답으로 내부 주소를 가리키는 redirect를 반환해 우회할 수 있다. 리다이렉트를 허용하지 않거나, 리다이렉트된 목적지도 동일하게 재검증한다.
  • 5차 방어 — 클라우드 메타데이터 접근 강화: IMDSv2 강제, 메타데이터 접근에 필요한 최소 권한만 IAM 역할에 부여(최소 권한 원칙, Day 1 참고).
흔한 오해 — "사설 IP 대역(10.x, 172.16-31.x, 192.168.x, 127.x)만 차단하면 SSRF는 끝난다"

우회 경로가 많다. (1) DNS Rebinding — 처음 조회 시엔 공인 IP를 응답하다가, 실제 요청 시점엔 내부 IP로 응답을 바꾸는 공격. (2) IP 인코딩 우회 — 127.0.0.1을 8진수(0177.0.0.1), 정수 표현(2130706433), IPv6 매핑 주소(::ffff:127.0.0.1) 등으로 표기해 단순 문자열 blacklist를 우회. (3) 리다이렉트 우회 — 화이트리스트에 있는 도메인이 302로 내부 주소로 리다이렉트. 블랙리스트 기반 문자열 매칭은 결국 "생각하지 못한 표기법 하나"에 뚫리므로, 근본 방어는 애플리케이션 레벨 검증이 아니라 네트워크 세그멘테이션이라는 결론으로 이어져야 한다.

7. 면접 질문 1 — 확장자 검증만으로 왜 부족한가

면접 질문
"파일 업로드 기능에서 허용된 확장자(.jpg, .png 등)만 받도록 이미 검증하고 있습니다. 그런데도 추가로 무엇을 더 해야 하나요?"

왜 면접에서 묻는가?

확장자 검사를 "이미 다 했다"고 여기는 사람과, 파일명 문자열과 파일 실제 내용이 별개의 정보라는 것을 구분해서 사고하는 사람을 가르는 질문이다. 또한 "검증"과 "실행 가능 여부"를 별도의 조건으로 이해하는지도 함께 확인한다.

답변 사고 순서

  1. 확장자와 실제 내용의 괴리 지적 — 파일명은 요청자가 자유롭게 정하는 메타데이터일 뿐임을 짚는다.
  2. 침해로 이어지는 두 번째 조건 제시 — 저장 위치가 실행 가능한 경로인지가 RCE 여부를 가른다는 것을 설명한다.
  3. 계층적 방어로 결론 — 매직 바이트 검증, 저장소 분리, 파일명 재생성, 별도 도메인 서빙을 순서대로 제시한다.
강한 답변 예시

"확장자는 파일명 끝에 붙은 문자열일 뿐이라 공격자가 원하는 대로 지정할 수 있습니다. 예를 들어 PHP 웹쉘 코드를 profile.php.jpg로 이름 붙여 올리면 확장자 검사는 통과합니다. 그래서 서버가 실제로 파일을 열어서 매직 바이트로 진짜 이미지 포맷인지 확인하고, 이미지라면 한 번 디코딩·재인코딩까지 시도해서 성공할 때만 통과시키는 방식이 필요합니다. 하지만 이것만으로도 완전하지 않은데, 진짜 위험은 검증을 통과한 파일이 '실행 가능한 경로'에 저장되는지에 달려 있습니다. 그래서 업로드 파일은 애플리케이션 서버가 아니라 S3 같은 오브젝트 스토리지에, 애초에 실행 권한이 없는 형태로 저장하고, 파일명도 사용자가 준 원본이 아니라 서버가 생성한 무작위 ID로 바꿔서 경로 조작 가능성 자체를 없앱니다. 마지막으로 업로드 파일은 본 서비스 도메인이 아니라 별도 서브도메인에서 서빙해서, 혹시 악성 SVG나 HTML이 통과되더라도 same-origin policy로 본 서비스의 쿠키에 접근하지 못하게 격리합니다."

약한 답변 예시

"확장자 화이트리스트에 이미지 확장자만 넣어뒀으니 그걸로 충분하다고 생각합니다."

왜 약한가? 파일명과 파일 내용을 같은 정보로 취급하고 있고, 저장 위치가 실행 가능한지에 대한 고려가 전혀 없다.

Senior-level Follow-up 질문들

Follow-up: "이미지 리사이징까지 거쳐서 저장하는데도 여전히 위험이 남아있을까요?"

모범 답변 방향: "리사이징(재인코딩) 자체는 원본 파일 구조를 파괴하고 새로 만드는 과정이라 웹쉘 같은 임베디드 코드를 상당수 제거하는 효과는 있습니다. 다만 두 가지가 남습니다. 첫째, 리사이징 라이브러리 자체의 파서 취약점 (예: 과거 ImageMagick의 ImageTragick 사고)이 있다면 그 라이브러리를 통해 서버가 공격당할 수 있어, 라이브러리 버전과 CVE를 지속적으로 관리해야 합니다. 둘째, SVG처럼 XML 기반 포맷은 래스터화 리사이징 경로를 거치지 않고 그대로 렌더링되는 경우가 많아, SVG는 별도로 script 태그나 외부 리소스 참조를 제거하는 sanitize 과정을 거치거나 아예 허용하지 않는 편이 안전합니다."

8. 면접 질문 2 — 이미지 리사이징 라이브러리를 쓰는데도 위험이 남는 이유

면접 질문
"업로드된 이미지를 검증된 오픈소스 라이브러리로 리사이징한 뒤에 저장하고 있습니다. 이 정도면 파일 업로드 보안은 충분한 걸까요?"

왜 면접에서 묻는가?

"검증된 라이브러리를 쓴다"는 답을 최종 결론으로 삼는지, 아니면 라이브러리 자체도 신뢰 경계의 일부이며 지속적으로 관리해야 할 대상이라는 것을 이해하는지를 확인한다. 시니어는 "이 정도면 끝"이라는 판단을 내릴 때 근거와 잔여 위험을 함께 말할 수 있어야 한다.

답변 사고 순서

  1. 리사이징의 방어 효과 인정 — 재인코딩이 임베디드 코드 상당수를 무력화한다는 점을 짚는다.
  2. 라이브러리 자체의 공급망 리스크 제시 — 파서 취약점, 버전 관리, CVE 대응 프로세스가 필요함을 설명한다.
  3. 포맷별 예외(SVG 등) 언급 — 모든 포맷이 동일한 리사이징 경로를 거치지 않는다는 점으로 마무리한다.
강한 답변 예시

"이미지 재인코딩은 확실히 도움이 됩니다 — 원본 바이트 구조를 그대로 두지 않고 픽셀 데이터를 읽어 새 파일로 만들기 때문에, 원본에 숨겨진 대부분의 임베디드 코드나 폴리글랏 트릭이 제거됩니다. 하지만 두 가지를 놓치면 안 됩니다. 하나는 리사이징 라이브러리 자체가 신뢰 경계 안으로 들어온다는 겁니다. 라이브러리가 파일 파싱 과정에서 취약점이 있으면 그 라이브러리를 통해 서버가 공격당할 수 있고, 실제로 이런 사고가 여러 차례 있었습니다. 그래서 라이브러리 버전을 최신으로 유지하고 CVE 공지를 모니터링하는 프로세스가 필요합니다. 다른 하나는 모든 업로드 포맷이 같은 리사이징 경로를 거치지 않는다는 점입니다. SVG는 XML 마크업이라 픽셀 기반 재인코딩과는 다른 처리 경로를 타는 경우가 많아서, 별도로 스크립트 태그나 외부 참조를 제거하는 처리가 필요하거나 아예 허용하지 않는 게 안전합니다. 결론적으로 '검증된 라이브러리를 쓴다'는 1차 방어이지 최종 방어가 아니고, 저장 위치 분리나 별도 도메인 서빙 같은 다른 계층과 함께 가야 합니다."

약한 답변 예시

"유명한 오픈소스 라이브러리를 쓰고 있으니 이미 검증된 거라 추가로 신경 쓸 부분은 없다고 봅니다."

왜 약한가? "유명하다"와 "안전하다"를 동일시하고 있고, 라이브러리도 지속적 취약점 관리 대상이라는 인식이 없다.

Senior-level Follow-up 질문들

Follow-up: "이미지 리사이징 서버가 갑자기 CPU를 100% 사용하며 응답하지 않는 장애가 발생했다면, 파일 업로드와 관련해 어떤 원인을 의심하시겠습니까?"

모범 답변 방향: "이미지 폭탄(image bomb, decompression bomb) 공격을 우선 의심합니다. 파일 크기 자체는 작지만 압축 해제 시 수만 x 수만 픽셀로 확장되는 이미지를 업로드하면, 리사이징 라이브러리가 압축을 풀면서 메모리와 CPU를 순식간에 소진시킬 수 있습니다. 방어로는 압축 해제 전에 이미지 헤더에 선언된 실제 픽셀 크기를 먼저 확인해 상한을 두고, 리사이징 작업에 타임아웃과 메모리 제한을 걸어서 개별 요청이 서버 전체 자원을 독점하지 못하게 격리해야 합니다."

9. 면접 질문 3 — 내부 IP만 차단하면 SSRF는 막히는가

면접 질문
"외부 URL을 받아 미리보기를 생성하는 기능에서, 사설 IP 대역(10.x, 192.168.x, 127.x)으로 시작하는 URL은 이미 차단하고 있습니다. 이 정도면 SSRF는 방어된 건가요?"

왜 면접에서 묻는가?

블랙리스트 방식의 근본적 한계(우회 표기법의 무한한 다양성)를 이해하는지, 그리고 SSRF의 근본 방어는 애플리케이션 레벨 문자열 검사가 아니라 네트워크 구조에 있다는 것을 아는지 확인하는 질문이다. 실무 침해 사례(Capital One 등)를 근거로 심각성을 설명할 수 있는지도 함께 본다.

답변 사고 순서

  1. 블랙리스트의 구조적 한계 지적 — DNS Rebinding, IP 인코딩 우회, 리다이렉트 우회를 구체적으로 제시한다.
  2. 근본 방어는 네트워크 세그멘테이션임을 제시 — 애플리케이션 검증이 뚫려도 도달 자체가 불가능한 구조를 설명한다.
  3. 클라우드 메타데이터라는 구체적 고위험 목적지로 연결 — IMDSv2, 최소 권한 IAM 역할까지 언급한다.
강한 답변 예시

"사설 IP 대역 차단은 좋은 출발점이지만 그것만으로는 충분하지 않습니다. 세 가지 우회가 남아있습니다. 첫째는 DNS Rebinding인데, 처음 도메인을 검증할 때는 공인 IP를 응답하다가 실제 요청이 나갈 때는 DNS 응답을 내부 IP로 바꿔치기하는 공격입니다. 둘째는 IP 표기법 우회로, 127.0.0.1을 8진수나 정수 표현, IPv6 매핑 주소로 바꾸면 단순 문자열 매칭 필터를 피해갈 수 있습니다. 셋째는 화이트리스트에 있는 정상 도메인이 302 리다이렉트로 내부 주소를 가리키는 경우인데, HTTP 클라이언트가 리다이렉트를 자동으로 따라가면 결국 내부로 요청이 나갑니다. 그래서 근본적인 방어는 애플리케이션 코드의 문자열 검사가 아니라, 이 기능을 처리하는 서버 자체를 별도 네트워크 구간에 두고 그 구간에서 내부망 라우팅과 클라우드 메타데이터 엔드포인트로의 접근을 방화벽 레벨에서 차단하는 것입니다. 애플리케이션 검증은 1차 방어로 유지하되, '설령 뚫려도 실제로 도달할 수 없는 구조'를 네트워크 레벨에서 만드는 게 최종 방어선입니다."

약한 답변 예시

"사설 IP 대역을 다 차단해뒀으니 서버 안에서 처리하는 이상 SSRF는 걱정 안 해도 됩니다."

왜 약한가? 블랙리스트가 문자열 매칭이라는 것과, 우회 기법이 다양하게 존재한다는 것을 인지하지 못하고 있다. 네트워크 레벨 방어라는 개념 자체가 빠져 있다.

Senior-level Follow-up 질문들

Follow-up: "이 미리보기 서비스가 AWS EC2에서 실행 중이라면, 애플리케이션 방어와 별개로 인프라 레벨에서 추가로 무엇을 설정하시겠습니까?"

모범 답변 방향: "이 인스턴스에 부여된 IAM 역할을 실제로 필요한 최소 권한만 갖도록 좁혀서, 설령 메타데이터 엔드포인트가 뚫려 자격 증명이 탈취되더라도 피해 범위를 최소화합니다. 또한 IMDSv2를 강제해서 세션 토큰 기반 접근만 허용하고, 단순 GET 요청을 그대로 전달하는 형태의 SSRF로는 메타데이터에 접근하기 어렵게 만듭니다. 마지막으로 이 서비스가 속한 서브넷의 보안 그룹/네트워크 ACL을 구성해서, 이 서비스가 내부망의 DB나 관리자 API 등 민감한 리소스로는 애초에 라우팅되지 않도록 네트워크 경계를 분리합니다."
