1. 왜 "세션 vs JWT"가 시니어 질문인가

주니어에게 이 질문을 하면 "JWT가 더 최신이니까 씁니다"라는 답이 돌아온다. 시니어에게 원하는 답은 다르다 — "우리 시스템의 확장 구조와 취소 요구사항을 보고 선택했다"는 근거다.

실무 상황

단일 서버로 운영하던 서비스가 트래픽이 늘어 서버를 여러 대로 늘려야 한다. 기존 세션 방식이면 "어느 서버가 이 세션을 들고 있는가"가 문제가 된다. 이때 팀은 JWT로 전환할지, 세션을 중앙 저장소(Redis)로 옮길지 결정해야 한다. 이 결정에는 정답이 없고, 트래픽 패턴, 팀 인프라 숙련도, 보안 요구사항에 따라 달라진다.

이 주제가 면접에서 반복되는 이유는 "정답이 없는 trade-off를 근거 있게 설명할 수 있는가"를 가장 명확하게 드러내는 주제이기 때문이다. 인증 방식 하나로 확장성, 보안, 운영 복잡도가 모두 얽힌다.

흔한 오해 — "JWT는 세션보다 항상 우월하다"

JWT는 "서버가 상태를 안 들고 있어도 된다"는 Stateless 특성 때문에 인기가 많아졌지만, 이 특성 자체가 취소(로그아웃, 권한 회수)를 어렵게 만드는 대가다. 우월한 게 아니라 다른 trade-off를 가진 선택지다.

2. 세션 기반 인증 — Stateful 모델

전제 개념: 세션이란 무엇인가?

로그인에 성공하면 서버는 세션 ID라는 임의의 문자열을 생성하고, 그 ID에 대응하는 사용자 정보(세션 데이터)를 서버 쪽 저장소(메모리, Redis, DB)에 저장한다. 클라이언트에는 세션 ID만 쿠키로 내려준다. 이후 요청마다 쿠키의 세션 ID로 서버 저장소를 조회해 사용자를 식별한다.

동작 흐름
  1. 클라이언트가 로그인 요청 → 서버가 자격증명 검증
  2. 서버가 세션 ID 생성 후 세션 저장소(Redis 등)에 {session_id: user_data} 저장
  3. 서버가 Set-Cookie: session_id=...로 클라이언트에 전달
  4. 이후 요청마다 클라이언트가 쿠키 자동 전송 → 서버가 저장소 조회로 사용자 식별
  5. 로그아웃 시 서버가 저장소에서 해당 세션 삭제 → 즉시 무효화
항목세션 기반 특징
상태 위치서버(Stateful) — 세션 저장소가 진실의 원천
즉시 취소가능 — 저장소에서 삭제하면 그 즉시 무효
확장성여러 서버 운영 시 세션 저장소를 공유해야 함(Redis 등 중앙화 필요)
마이크로서비스서비스마다 세션 저장소를 조회해야 하므로 결합도가 높아짐
클라이언트 부담쿠키만 들고 있으면 되므로 가볍다
실무 연결

사내 관리자 도구, 서버 대수가 적은 모놀리식 서비스, "로그아웃 즉시 반영"이 중요한 금융/의료 서비스는 세션 기반이 여전히 합리적인 선택이다. Redis 하나로 여러 서버가 세션을 공유하면 확장성 문제도 크지 않다.

3. JWT 기반 인증 — Stateless 모델

전제 개념: JWT 구조

JWT(JSON Web Token)는 Header.Payload.Signature 세 부분을 점(.)으로 이어붙인 문자열이다. Header와 Payload는 Base64URL로 인코딩(암호화 아님!)되어 있을 뿐이라 누구나 디코딩해서 내용을 읽을 수 있다. Signature만이 "이 토큰이 서버가 발급한 게 맞다"는 것을 보장한다. 

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0Iiwicm9sZSI6InVzZXIifQ.SflKxwRJSMeKKF2QT4fw...
└────── Header ──────┘└──────── Payload ────────┘└──── Signature ────┘

Header:  {"alg": "HS256", "typ": "JWT"}
Payload: {"sub": "1234", "role": "user", "exp": 1735689600}
Signature: HMACSHA256(base64(header) + "." + base64(payload), SECRET_KEY)
흔한 오해 — "JWT는 암호화되어 있어서 안전하다"

Payload는 암호화가 아니라 인코딩이다. 누구나 jwt.io에 붙여넣으면 내용을 읽을 수 있다. JWT에 비밀번호나 카드번호 같은 민감 정보를 넣으면 안 된다. Signature는 "위조 방지"만 하지 "내용 은닉"은 하지 않는다.

서명 알고리즘 선택 — HS256 vs RS256

HS256 (대칭키, HMAC)

같은 비밀키로 서명과 검증을 모두 한다.

적합: 단일 서비스, 발급자와 검증자가 같은 신뢰 영역

위험: 검증하는 모든 서버가 비밀키를 알아야 하므로 키 유출 범위가 넓어짐

RS256 (비대칭키, RSA)

Private Key로 서명, Public Key로 검증한다.

적합: 여러 마이크로서비스가 토큰을 검증만 해야 할 때 — 검증 서버는 Public Key만 가지면 됨

trade-off: 계산 비용이 HMAC보다 높고 키 관리(로테이션, 배포)가 더 복잡

치명적 구현 취약점 — Algorithm Confusion / alg:none

JWT 라이브러리 중 일부는 Header의 alg 필드를 신뢰하고 그 알고리즘으로 검증을 시도한다. 공격자가 alg를 none으로 바꾸면 서명 검증 자체를 건너뛰는 라이브러리가 있었다(과거 다수의 CVE 사례). 또한 RS256으로 발급된 토큰의 alg를 HS256으로 바꾸고, 공개적으로 알려진 Public Key를 HMAC 비밀키처럼 사용해 서명을 위조하는 공격도 가능했다.

방어: 서버 쪽에서 허용할 알고리즘을 화이트리스트로 명시적으로 고정한다 (예: jwt.decode(token, key, algorithms=["RS256"])). 토큰의 alg 헤더를 그대로 신뢰하지 않는다.

항목JWT 기반 특징
상태 위치클라이언트(Stateless) — 토큰 자체에 필요한 정보가 담김
즉시 취소어려움 — 서버가 "이 토큰은 무효"라는 상태를 별도로 관리해야 함(모순적으로 다시 Stateful)
확장성좋음 — 어느 서버든 Public Key/비밀키만 있으면 검증 가능, 중앙 저장소 조회 불필요
마이크로서비스서비스 간 토큰 전파가 쉬움(SSO, API Gateway 연동에 유리)
클라이언트 부담토큰 크기가 커서(Payload 포함) 매 요청마다 더 많은 데이터 전송

4. 토큰 저장 위치 — localStorage vs httpOnly Cookie

전제 개념: 왜 저장 위치가 보안 문제인가?

JWT를 어디에 저장하느냐에 따라 어떤 공격에 노출되는가가 완전히 달라진다. 이건 "취향"이 아니라 "어떤 위협을 더 감당할 수 있는가"를 고르는 문제다.

localStorage 저장

장점: JS에서 자유롭게 읽고 헤더에 담아 전송 가능, CSRF 걱정 없음(쿠키 자동 전송이 아니므로)

치명적 단점: XSS 공격에 완전히 노출된다. 페이지에 악성 스크립트가 하나라도 실행되면 localStorage.getItem("token")으로 토큰을 그대로 탈취당한다.

httpOnly Cookie 저장

장점: JavaScript로 document.cookie를 통해 접근 불가 → XSS로 토큰을 직접 훔칠 수 없음

단점: 쿠키는 브라우저가 자동으로 전송하므로 CSRF 공격에 노출된다 → SameSite 속성과 CSRF 토큰으로 별도 방어 필요

실무에서 흔히 쓰는 조합

Access Token은 httpOnly + Secure + SameSite=Strict/Lax 쿠키에 저장하고, CSRF는 SameSite 속성 + Double Submit Cookie 패턴이나 커스텀 헤더(X-Requested-With) 검증으로 방어하는 조합이 "XSS와 CSRF 둘 다 어느 정도 대응 가능한" 실무 표준에 가깝다.

쿠키 속성역할
httpOnlyJavaScript에서 쿠키 접근 차단 → XSS로 인한 토큰 탈취 방지
SecureHTTPS 연결에서만 쿠키 전송 → 평문 네트워크 도청 방지
SameSite=Strict다른 사이트에서의 요청에는 쿠키를 아예 안 보냄 → CSRF 강력 방어, 단 외부 링크 클릭 후 로그인 유지 등 UX 저하 가능
SameSite=Lax최상위 탐색(링크 클릭)에는 쿠키 전송, 하지만 폼 자동 제출/이미지 태그 등에는 차단 → 실무 기본값으로 많이 사용
흔한 오해 — "httpOnly 쿠키면 CSRF도 막힌다"

httpOnly는 XSS로부터 토큰을 보호할 뿐이다. 브라우저는 여전히 쿠키를 자동으로 요청에 실어 보내므로, 공격자 사이트가 사용자 브라우저를 통해 우리 서버로 요청을 보내면(CSRF) 쿠키가 그대로 실려 나간다. httpOnly와 CSRF 방어는 서로 다른 위협에 대한 서로 다른 방어다.

5. 토큰 갱신 전략 — Access/Refresh Token

전제 개념: 왜 토큰을 두 개로 나누는가?

JWT의 근본 문제는 "만료 전에는 취소할 수 없다"는 것이다. 만료 시간을 길게 잡으면(예: 7일) 탈취당했을 때 피해가 오래간다. 만료 시간을 짧게 잡으면(예: 15분) 사용자가 15분마다 재로그인해야 하는 UX 문제가 생긴다. 이 딜레마를 푸는 실무 패턴이 Access Token + Refresh Token 분리다.

Access/Refresh Token 패턴
  • Access Token: 짧은 수명(5~15분), API 요청마다 사용, 탈취돼도 피해 시간이 짧음
  • Refresh Token: 긴 수명(며칠~몇 주), Access Token이 만료되면 이걸로 재발급 요청, httpOnly 쿠키에만 저장하고 API 요청에는 절대 사용하지 않음

Refresh Token Rotation (회전)

Refresh Token을 한 번 쓸 때마다 새 Refresh Token으로 교체하고 이전 것은 무효화한다. 같은 Refresh Token으로 두 번째 재발급 요청이 오면 "탈취당한 정황"으로 간주하고 해당 사용자의 모든 토큰을 강제 폐기한다. 

# Refresh Token Rotation 흐름
1. 클라이언트가 Refresh Token A로 재발급 요청
2. 서버: A가 유효한지 확인 → 새 Access Token + 새 Refresh Token B 발급, A는 폐기(사용됨으로 표시)
3. 클라이언트는 이제 B를 사용

# 탈취 탐지
4. 만약 이미 폐기된 A로 재요청이 들어오면
   → 정상 사용자가 아니라 탈취범이 예전 토큰으로 시도한 것으로 간주
   → 해당 사용자 계정의 모든 Refresh Token 계열을 즉시 폐기 + 재인증 요구
실무 연결

은행 앱이 "이 기기에서 이상 활동이 감지되어 로그아웃되었습니다"라고 알리는 경우가 이 패턴이다. 토큰 재사용 탐지로 탈취를 사후에라도 잡아낼 수 있다.

6. JWT 취소 문제 — 블랙리스트와 회전

전제 개념: Stateless의 대가

JWT가 자랑하는 "서버가 상태를 안 들고 있어도 된다"는 특성은, 뒤집어 말하면 "서버가 이 토큰이 아직 유효한지 판단할 방법이 토큰 자체의 만료 시간뿐"이라는 뜻이다. 사용자가 로그아웃하거나, 관리자가 계정을 정지하거나, 비밀번호를 바꿔도 — 만료되지 않은 JWT는 여전히 유효하다. 이것이 시니어 인터뷰에서 가장 자주 파고드는 지점이다.

방법동작 방식trade-off
짧은 TTL + Refresh Token Access Token 수명을 짧게(5~15분) 두어 자연 만료로 위험을 제한 완전한 즉시 취소는 아니고 "최대 15분의 위험 창"이 남음
블랙리스트 (Redis) 로그아웃/정지된 토큰의 jti(토큰 ID)를 Redis에 만료 시간까지만 저장, 검증 시마다 조회 결국 Stateless의 장점을 일부 포기하고 저장소 조회가 다시 생김(다만 로그아웃 등 예외 케이스만 조회하므로 세션 방식보다는 부하가 적음)
토큰 버전 관리 (Token Versioning) 사용자 레코드에 token_version 필드를 두고 JWT Payload에 포함, 비밀번호 변경/강제 로그아웃 시 버전 증가 → 이전 버전 토큰은 전부 무효 사용자당 하나의 카운터 조회만 필요해 블랙리스트보다 가볍지만, "특정 토큰 하나만" 취소하기는 어려움(전체 무효화 단위)
Refresh Token Rotation Refresh Token 재사용 탐지로 탈취 시나리오에서 사후 대응 실시간 차단이 아니라 재사용 시점에 탐지 — Access Token 유효 기간만큼은 여전히 노출
실무에서 조합하는 방식

대부분의 실무 시스템은 하나만 쓰지 않는다. 짧은 TTL(15분) + 토큰 버전 관리(비밀번호 변경/강제 로그아웃 시 즉시 무효화) + Refresh Token Rotation(탈취 탐지)을 조합해서, "완전한 즉시 취소는 아니지만 위험을 충분히 제한하는" 현실적인 균형을 잡는다.

흔한 오해 — "JWT는 Stateless니까 서버 부담이 전혀 없다"

실무에서 안전하게 쓰려면 블랙리스트든 토큰 버전이든 결국 서버 쪽에 최소한의 상태가 필요하다. "완전한 Stateless"는 이론에 가깝고, 실무 JWT는 "세션보다 가벼운 상태를 유지하는 하이브리드"에 가깝다는 것을 이해해야 한다.

7. 면접 질문 1 — 세션 vs JWT 선택 기준

면접 질문
"세션 기반 인증과 JWT 기반 인증 중 무엇을 선택하시겠습니까? 어떤 기준으로 결정하나요?"

왜 면접에서 묻는가?

이 질문은 정답 맞히기가 아니다. "트래픽 구조, 조직의 인프라 역량, 보안 요구사항을 종합해서 근거 있게 판단하는가"를 본다. "JWT가 대세니까"라고 답하면 유행을 따라간 것이지 설계한 것이 아니다.

답변 사고 순서

  1. 서버 구성 파악 — 단일 서버인가, 여러 대인가, 이미 Redis 같은 공유 저장소가 있는가?
  2. 취소 요구사항 파악 — 로그아웃/권한 회수가 즉시 반영되어야 하는 도메인인가(금융, 의료)?
  3. 서비스 경계 파악 — 마이크로서비스 간 토큰을 전달해야 하는가, 아니면 단일 서비스인가?
  4. trade-off 명시 — 어느 쪽을 선택해도 포기하는 게 있다는 것을 설명한다.
강한 답변 예시

"저는 먼저 서버 확장 계획과 즉시 취소 요구사항을 봅니다. 만약 서버가 소수이고 이미 Redis를 세션 저장소로 쓰고 있으며, 로그아웃/정지가 즉시 반영되어야 하는 도메인이라면 세션 기반을 유지합니다. 반대로 여러 마이크로서비스가 API Gateway 뒤에서 토큰을 검증해야 하고, 서비스 간 결합도를 낮추는 게 중요하다면 JWT를 선택합니다. 다만 JWT를 선택하더라도 즉시 취소가 필요한 부분(비밀번호 변경, 강제 로그아웃)은 토큰 버전 관리로 보완합니다. 결국 '무조건 JWT'가 아니라 시스템의 확장 패턴과 취소 요구사항의 균형점을 찾는 결정입니다."

약한 답변 예시

"JWT가 요즘 표준이고 Stateless라서 확장성이 좋으니 JWT를 씁니다."

왜 약한가? "요즘 표준"은 근거가 아니다. JWT의 취소 문제, 저장 위치 위험을 전혀 언급하지 않아 trade-off를 이해하지 못한 것으로 보인다.

Senior-level Follow-up 질문들

Follow-up: "세션 기반으로 시작한 서비스를 트래픽 급증 때문에 JWT로 마이그레이션해야 한다면 어떻게 하겠습니까?"

모범 답변 방향: "먼저 트래픽 증가가 정말 '세션 저장소 병목' 때문인지 확인합니다. 대부분의 경우 세션 저장소를 Redis 클러스터로 확장하는 것만으로 충분히 해결됩니다. JWT 전환은 취소 로직 재설계, 클라이언트 저장 방식 변경, 기존 세션 사용자 마이그레이션까지 필요한 큰 작업이므로, 단순 트래픽 증가만으로는 전환 비용을 정당화하기 어렵습니다. 마이크로서비스 분리처럼 아키텍처 자체가 바뀌는 시점에 함께 검토하는 게 합리적입니다."

8. 면접 질문 2 — 토큰 저장 위치와 XSS/CSRF trade-off

면접 질문
"JWT를 클라이언트 어디에 저장하시겠습니까? 그 선택이 어떤 공격에 노출되나요?"

왜 면접에서 묻는가?

이 질문은 "정답이 하나가 아닌 보안 결정"을 다루는 능력을 본다. localStorage와 httpOnly Cookie 모두 완벽하지 않다 — 어떤 공격을 감수할 것인지 명시적으로 선택하고 나머지를 보완할 수 있는가가 핵심이다.

답변 사고 순서

  1. 두 옵션의 공격 표면 나열 — localStorage는 XSS, Cookie는 CSRF에 취약함을 먼저 짚는다.
  2. 선택과 이유 — 보통 httpOnly Cookie를 선택하는 이유(XSS가 더 흔하고 피해가 큼)를 설명한다.
  3. 남은 위협 보완책 제시 — CSRF를 SameSite + 토큰으로 어떻게 막는지 설명한다.
  4. 추가 방어층 언급 — CSP(Content Security Policy)로 XSS 자체 발생 확률을 낮추는 것도 언급하면 더 좋다.
강한 답변 예시

"Access Token은 httpOnly, Secure, SameSite=Lax 쿠키에 저장합니다. localStorage에 저장하면 XSS 한 번으로 토큰이 그대로 탈취되는데, XSS는 CSRF보다 발생 빈도와 피해 범위가 큽니다. 대신 쿠키 저장은 CSRF에 노출되므로, 상태 변경 요청에는 CSRF 토큰을 별도 헤더로 검증하고 SameSite=Lax로 크로스 사이트 자동 전송을 1차로 막습니다. 추가로 CSP를 적용해 인라인 스크립트 실행 자체를 제한해서 XSS 발생 확률도 같이 낮춥니다."

약한 답변 예시

"localStorage에 저장하고 헤더에 담아 보냅니다. 구현이 간단해서 그렇게 했습니다."

왜 약한가? "구현이 간단하다"는 보안 근거가 아니다. XSS 노출 위험을 인지하지 못하고 있다는 신호로 읽힌다.

Senior-level Follow-up 질문들

Follow-up: "모바일 앱(네이티브)에서는 어떻게 저장하나요? 브라우저와 다른가요?"

모범 답변 방향: "네이티브 앱에는 DOM이나 쿠키 자동 전송 개념이 없으므로 위협 모델이 다릅니다. iOS는 Keychain, Android는 Keystore/EncryptedSharedPreferences처럼 OS가 제공하는 보안 저장소를 사용합니다. 브라우저의 XSS/CSRF 고민을 그대로 적용하는 게 아니라, 플랫폼별 안전한 저장소가 있는지부터 확인해야 합니다."

9. 면접 질문 3 — 즉시 로그아웃/권한 회수

면접 질문
"JWT를 쓰는 시스템에서 관리자가 사용자를 즉시 정지시켜야 합니다. 어떻게 구현하시겠습니까?"

왜 면접에서 묻는가?

이 질문은 JWT의 근본 약점(취소 불가)을 실제 운영 요구사항과 충돌시켜서, 이론을 실무 제약과 연결할 수 있는가를 시험한다. "JWT는 원래 그렇습니다"로 끝나면 탈락에 가깝다.

답변 사고 순서

  1. 문제 인정 — 순수 JWT 검증만으로는 만료 전까지 막을 수 없음을 먼저 인정한다.
  2. 보완 메커니즘 선택 — 토큰 버전 관리 또는 블랙리스트 중 상황에 맞는 것을 고른다.
  3. 검증 경로에 삽입 — 매 요청마다 어디서 이 확인을 추가하는지 설명한다(미들웨어).
  4. 비용 설명 — 이 보완책이 Stateless의 일부를 포기하는 trade-off임을 명시한다.
# 토큰 버전 관리 구현 예시
class User:
    id: int
    token_version: int  # 기본값 0, 강제 로그아웃/정지 시 +1

def issue_token(user: User) -> str:
    return jwt.encode(
        {"sub": user.id, "tv": user.token_version, "exp": ...},
        SECRET_KEY, algorithm="HS256"
    )

async def get_current_user(token: str) -> User:
    payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
    user = await db.get_user(payload["sub"])
    if user.token_version != payload["tv"]:
        raise HTTPException(401, "토큰이 더 이상 유효하지 않습니다")  # 정지/강제 로그아웃 반영
    return user

def suspend_user(user: User):
    user.token_version += 1  # 이 순간부터 기존 토큰 전부 무효
    db.save(user)
강한 답변 예시

"순수 JWT는 만료 전까지 취소가 안 되므로, User 테이블에 token_version 필드를 두고 발급 시 Payload에 포함시킵니다. 관리자가 정지시키면 token_version을 증가시키고, 인증 미들웨어에서 매 요청마다 DB의 현재 버전과 토큰의 버전을 비교합니다. 이렇게 하면 정지 조치가 사실상 즉시(다음 요청부터) 반영됩니다. 비용은 매 요청마다 사용자 레코드 조회가 필요하다는 것인데, 이건 캐시(Redis)에 token_version만 캐싱해서 완화할 수 있습니다."

약한 답변 예시

"JWT는 만료될 때까지 유효하니 만료 시간을 짧게 설정합니다."

왜 약한가? "즉시" 정지라는 요구사항에 정면으로 답하지 못했다. 짧은 TTL은 위험 창을 줄일 뿐 즉시 취소가 아니다.

Senior-level Follow-up 질문들

Follow-up: "token_version 비교를 위해 매번 DB를 조회하면 성능에 영향이 크지 않나요?"

모범 답변 방향: "맞습니다, 그래서 실무에서는 token_version만 Redis에 캐싱해서 조회 비용을 낮춥니다. 또는 정지 이벤트 발생 빈도가 낮다는 점을 이용해, 평소엔 캐시된 값으로 빠르게 통과시키고 정지/권한 변경 이벤트가 발생했을 때만 캐시를 무효화하는 방식으로 대부분의 요청은 추가 DB 조회 없이 처리합니다."
