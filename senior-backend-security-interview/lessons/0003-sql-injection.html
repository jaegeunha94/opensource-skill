1. 왜 20년 된 취약점을 아직도 시니어에게 묻는가

SQL Injection은 1998년에 처음 공개적으로 문서화됐다. 그런데도 OWASP Top 10에서 여전히 상위권(Injection 카테고리)에 남아 있고, 실무 침해 사고 리포트에도 매년 등장한다. 면접관이 이 주제를 묻는 이유는 "정의를 아는가"가 아니라 "왜 해결된 문제가 계속 재발하는지, 그리고 그 재발 지점을 구조적으로 어떻게 차단하는가"를 보기 위해서다.

실무 상황

신입 개발자가 관리자 페이지에 "매출 검색" 기능을 추가하면서 검색어를 문자열로 이어붙여 쿼리를 만들었다. 코드 리뷰에서 아무도 지적하지 않고 배포됐다. 6개월 뒤 자동화된 스캐너가 이 엔드포인트를 찾아내 전체 고객 테이블을 덤프해갔다. 이 사고의 원인은 "몰라서"가 아니라 "팀 전체가 안전한 패턴을 기본값으로 강제하지 않아서"였다.

즉 시니어에게 기대하는 것은 개인이 안전한 코드를 짜는 능력을 넘어, "팀이 실수해도 사고로 이어지지 않는 구조"(코드 리뷰 체크리스트, 린터 규칙, ORM 기본 설정, WAF)를 설계할 수 있는가다.

흔한 오해 — "요즘은 다 ORM 쓰니까 SQL Injection은 옛날 얘기다"

ORM은 기본 CRUD를 안전하게 만들어줄 뿐, raw query, 동적 정렬/컬럼명, 복잡한 검색 조건을 다룰 때 개발자가 여전히 문자열을 직접 이어붙이는 경로를 열어둔다. 최신 침해 사고에도 ORM을 쓰는 서비스가 다수 포함된다.

2. SQL Injection의 원리 — 코드와 데이터의 경계 붕괴

전제 개념: SQL 엔진은 "문자열"과 "명령어"를 어떻게 구분하는가?

데이터베이스는 문자열로 받은 쿼리를 그대로 파싱해서 실행한다. 문제는 사용자가 입력한 "데이터"가 쿼리 문자열 안에 그대로 섞여 들어가면, DB 엔진 입장에서는 그것이 원래 "데이터"였는지 "명령어의 일부"였는지 구분할 방법이 없다는 점이다. SQL Injection은 이 경계를 공격자가 의도적으로 무너뜨리는 것이다. 

# 취약한 코드 — 문자열 결합
query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"

# 정상 입력: username=admin, password=1234
SELECT * FROM users WHERE username = 'admin' AND password = '1234'

# 공격 입력: username=admin' -- , password=(아무거나)
SELECT * FROM users WHERE username = 'admin' -- ' AND password = '아무거나'
#                                              └─ 여기부터 주석 처리되어 비밀번호 검사 자체가 사라짐
핵심 원리

공격자가 넣은 '(작은따옴표)가 문자열 리터럴을 조기 종료시키고, 그 뒤에 새로운 SQL 문법이 이어진다. --(주석)로 원래 쿼리의 나머지 부분을 무력화하면 인증 로직 전체를 우회할 수 있다. 이것이 "데이터인 줄 알았던 입력이 명령어로 실행되는" 것 — Injection의 본질이다.

3. 공격 유형 — Union-based, Error-based, Blind

면접에서 "SQL Injection이 뭔가요"에서 끝나지 않고 "그럼 에러 메시지를 다 막아놨는데도 뚫을 수 있나요?"까지 이어지는 경우가 많다. 공격자가 화면에 아무 정보도 노출되지 않는 상황에서도 데이터를 뽑아낼 수 있다는 것을 이해해야 이 follow-up에 답할 수 있다.

Union-based

UNION SELECT로 공격자가 원하는 테이블의 데이터를 원래 쿼리 결과에 이어붙여 화면에 그대로 노출시킨다.

전제 조건: 쿼리 결과가 응답 화면에 그대로 출력되어야 함, 컬럼 개수/타입이 맞아야 함

Error-based

일부러 잘못된 쿼리를 주입해 DB 에러 메시지에 데이터를 실어 나오게 만든다 (예: CAST 타입 에러에 서브쿼리 결과 포함).

전제 조건: 상세 에러 메시지가 클라이언트에 그대로 노출되어야 함

Blind SQL Injection — 화면에 아무것도 안 보여도 뚫리는 이유

화면에 결과도, 에러 메시지도 전혀 노출되지 않아도 공격은 가능하다. 응답이 "참/거짓" 또는 "응답 시간"만 다르면 그 차이 자체가 정보가 되기 때문이다.

  • Boolean-based Blind: AND 1=1(정상 화면)과 AND 1=2(빈 화면/다른 화면)의 응답 차이로 참/거짓을 판별한다. 이걸 반복해서 AND SUBSTRING(password,1,1)='a'처럼 한 글자씩 맞춰나가며 데이터를 통째로 복원한다.
  • Time-based Blind: 화면 차이조차 없을 때, AND IF(SUBSTRING(password,1,1)='a', SLEEP(5), 0)처럼 조건이 참일 때만 응답을 5초 지연시켜, 응답 시간의 차이만으로 데이터를 한 글자씩 추출한다.
흔한 오해 — "화면에 쿼리 결과나 에러가 안 보이면 안전하다"

Boolean/Time-based Blind는 참/거짓 신호 하나만 있으면 성립한다. "결과를 안 보여준다"는 완화책일 뿐, 근본 방어(Parameterized Query)를 대체하지 못한다. 느리지만(수천 번의 요청) 자동화 도구(sqlmap 등)로는 몇 분 안에 데이터베이스 전체를 추출할 수 있다.

4. 근본 방어 — Parameterized Query가 escaping과 다른 이유

전제 개념: "이스케이프 처리"와 "파라미터화"는 다른 층위의 방어다

많은 사람이 "특수문자를 이스케이프하면 안전하다"고 생각하지만, 이스케이프는 "어떤 문자를 이스케이프해야 하는가"를 개발자가 매번 정확히 알아야 하는 블랙리스트 방식이라 DB 엔진, 인코딩, 컨텍스트에 따라 우회 사례가 계속 나온다. Parameterized Query(Prepared Statement)는 접근 자체가 다르다 — 쿼리 구조(명령어)와 값(데이터)을 애초에 분리된 채널로 DB에 전달한다. 

# 취약 — 문자열 결합 (이스케이프를 직접 해도 근본적으로 같은 층위)
query = f"SELECT * FROM users WHERE username = '{username}'"

# 안전 — Parameterized Query
cursor.execute(
    "SELECT * FROM users WHERE username = %s",
    (username,)
)
# DB 드라이버가 쿼리 구조를 먼저 컴파일하고, username 값은 "데이터"로만 바인딩된다.
# username에 '  OR '1'='1가 들어와도 SQL 문법으로 해석되지 않고 그냥 통째로 "문자열 값"으로 취급된다.
왜 이게 "근본" 방어인가

Parameterized Query는 DB 엔진 레벨에서 쿼리 구문 분석(파싱)이 값 바인딩보다 먼저 끝난다. 즉 공격자의 입력이 아무리 SQL 문법처럼 생겼어도, 이미 "구조가 확정된 쿼리"에 값으로만 끼워 넣어지므로 추가 SQL 문법으로 해석될 여지 자체가 없다. 이스케이프처럼 "어떤 패턴을 막을까"를 고민할 필요가 없어지는 게 핵심 차이다.

계층적 방어(Defense in Depth)
  • 1차 방어 — Parameterized Query: 모든 쿼리의 기본값으로 강제(코드 리뷰/린터 규칙)
  • 2차 방어 — 입력 검증: 타입/길이/형식 검증으로 명백히 이상한 입력을 조기 차단
  • 3차 방어 — 최소 권한 DB 계정: 애플리케이션 DB 계정에 DROP/불필요한 스키마 접근 권한을 주지 않음
  • 4차 방어 — WAF: 알려진 Injection 패턴을 네트워크 레벨에서 1차 필터링(우회 가능하므로 근본 방어는 아님)
  • 5차 방어 — 에러 메시지 최소화 + 로깅/알림: 공격 시도를 조기에 탐지

5. ORM을 쓰면 안전하다는 착각

ORM(SQLAlchemy, Django ORM, JPA/Hibernate 등)은 기본 CRUD 메서드(filter, save, find)를 쓰면 내부적으로 Parameterized Query를 생성하기 때문에 안전하다. 문제는 ORM이 표현하기 어려운 요구사항에서 개발자가 다시 문자열 결합으로 돌아가는 지점이다.

위험한 ORM 사용 패턴
# Django — raw()에 문자열 결합
User.objects.raw(
    f"SELECT * FROM users WHERE name = '{name}'"
)

# SQLAlchemy — text()에 문자열 결합
db.execute(
    text(f"SELECT * FROM users WHERE name = '{name}'")
)

# 동적 정렬 컬럼을 그대로 삽입
User.objects.raw(
    f"SELECT * FROM users ORDER BY {sort_column}"
)
안전한 ORM 사용 패턴
# Django — ORM 필터 사용
User.objects.filter(name=name)

# raw()를 써야 한다면 파라미터 바인딩
User.objects.raw(
    "SELECT * FROM users WHERE name = %s",
    [name]
)

# SQLAlchemy — text()에도 바인딩 파라미터
db.execute(
    text("SELECT * FROM users WHERE name = :name"),
    {"name": name}
)
실무 연결

복잡한 리포트 쿼리, 성능 때문에 ORM을 우회해 직접 SQL을 짜야 하는 경우, "정렬 기준을 사용자가 고르게" 하는 기능처럼 ORM의 표준 API로 표현이 안 되는 요구사항에서 raw query가 등장한다. 코드 리뷰에서 "raw() / text() / execute()가 보이면 바인딩 파라미터를 쓰는지부터 확인"하는 체크리스트가 실무에서 실제로 사고를 막는 지점이다.

6. 파라미터화할 수 없는 경우 — 동적 정렬/컬럼명

전제 개념: Parameterized Query는 "값"만 바인딩할 수 있다

Prepared Statement의 파라미터 바인딩은 WHERE column = ?처럼 값(리터럴) 자리에만 쓸 수 있다. 테이블명, 컬럼명, ORDER BY 뒤에 오는 정렬 기준 컬럼, ASC/DESC 같은 식별자(identifier)나 키워드는 파라미터 바인딩 문법 자체가 지원하지 않는다. 이 지점에서 많은 개발자가 다시 문자열 결합으로 돌아가 취약점을 만든다. 

# 안전 — 화이트리스트 매핑으로 식별자를 검증
ALLOWED_SORT_COLUMNS = {"created_at", "name", "price"}
ALLOWED_ORDER = {"asc", "desc"}

def build_query(sort_column: str, order: str):
    if sort_column not in ALLOWED_SORT_COLUMNS:
        raise ValueError("허용되지 않은 정렬 컬럼")
    if order.lower() not in ALLOWED_ORDER:
        raise ValueError("허용되지 않은 정렬 방향")
    # 화이트리스트를 통과한 값만 문자열에 삽입 — 사용자 입력이 그대로 들어가지 않음
    return f"SELECT * FROM products ORDER BY {sort_column} {order}"
핵심 원칙

식별자는 바인딩할 수 없으니 사용자 입력을 직접 쿼리에 넣지 않고, 사전에 정의한 허용 목록(화이트리스트)에서 "선택"만 하게 한다. 사용자가 문자열을 자유롭게 입력하는 게 아니라 서버가 이미 아는 안전한 값 중 하나를 고르는 구조이므로, Injection이 성립할 문자열 결합 지점 자체가 사라진다.

7. 발견 시 실무 대응 순서

SQL Injection 발견 시 대응 흐름
  1. 범위 파악 — 취약한 엔드포인트가 어떤 테이블/권한에 접근 가능한지, 실제 악용 흔적(비정상 쿼리 로그, 대량 SELECT)이 있는지 확인
  2. 긴급 완화 — 패치 배포 전까지 WAF 룰로 해당 패턴 차단 또는 해당 엔드포인트 일시 비활성화
  3. 근본 패치 — 문자열 결합을 Parameterized Query로 교체, 동일 패턴이 다른 곳에도 없는지 코드베이스 전체 검색
  4. 침해 여부 조사 — 실제로 데이터가 유출됐다면 유출 범위 확인, 비밀번호 해시가 노출됐다면 전체 사용자 비밀번호 재설정 강제
  5. 재발 방지 — 린터/CI에 raw query 탐지 규칙 추가, 코드 리뷰 체크리스트 갱신, 팀 공유
흔한 오해 — "패치만 하면 끝이다"

취약점을 막는 것과 이미 뚫렸을 가능성을 조사하는 것은 별개의 작업이다. 시니어 면접에서는 종종 "패치 이후 뭘 더 하시겠어요?"로 이 부분을 따로 확인한다. 쿼리 로그, 접근 로그, 비정상 트래픽 패턴을 함께 점검하지 않으면 이미 유출된 데이터를 놓칠 수 있다.

8. 면접 질문 1 — ORM은 SQL Injection에서 안전한가

면접 질문
"우리 서비스는 ORM을 쓰고 있으니 SQL Injection 걱정은 안 해도 되지 않나요?"

왜 면접에서 묻는가?

이 질문은 "ORM = 안전"이라는 단순화된 믿음을 얼마나 정확히 반박할 수 있는지를 본다. 면접관은 실무에서 raw query가 등장하는 구체적인 상황을 아는지, 그리고 그 상황에서도 안전 패턴을 유지할 수 있는지를 확인한다.

답변 사고 순서

  1. 부분적으로 맞다는 것 인정 — 표준 ORM API는 기본적으로 파라미터화된 쿼리를 생성함을 짚는다.
  2. 예외 지점 제시 — raw query, 동적 정렬/컬럼명, 복잡한 서브쿼리에서 우회 경로가 생긴다는 것을 설명한다.
  3. 팀 차원의 안전장치 제시 — 개인의 주의력에 의존하지 않고 코드 리뷰/린터로 구조적으로 막는 방법을 제시한다.
강한 답변 예시

"표준 ORM 메서드(filter, save 등)를 쓰는 한 파라미터화된 쿼리가 생성되므로 그 범위 안에서는 안전합니다. 하지만 raw()나 text() 같은 원시 쿼리 인터페이스를 쓸 때, 특히 복잡한 리포트 쿼리나 동적 정렬 기능을 구현할 때 문자열 결합으로 돌아가는 경우가 실무에서 자주 생깁니다. 그래서 저희 팀은 raw query에도 반드시 바인딩 파라미터를 쓰도록 코드 리뷰 체크리스트에 명시하고, 정적 분석 도구로 문자열 포매팅이 SQL 실행 함수에 직접 들어가는 패턴을 CI에서 잡아냅니다."

약한 답변 예시

"네, ORM을 쓰면 자동으로 안전하게 처리해줘서 걱정 없습니다."

왜 약한가? raw query 사용 가능성 자체를 인지하지 못하고 있다는 신호다. 실무 경험이 있다면 이 예외 지점을 반드시 경험했을 것이다.

Senior-level Follow-up 질문들

Follow-up: "팀원이 실수로 raw query에 문자열 결합을 쓰는 걸 코드 리뷰에서 매번 사람이 잡아야 하나요?"

모범 답변 방향: "사람이 매번 잡는 건 확장 가능한 방법이 아닙니다. 정적 분석 도구(예: Bandit, Semgrep)로 문자열 포매팅이 SQL 실행 함수의 인자로 바로 들어가는 패턴을 CI 파이프라인에서 자동 탐지하도록 규칙을 넣고, 해당 패턴이 발견되면 빌드를 실패시켜서 리뷰어의 주의력에 의존하지 않는 구조를 만드는 게 맞다고 생각합니다."

9. 면접 질문 2 — Blind SQL Injection 탐지와 방어

면접 질문
"에러 메시지도 안 보여주고 쿼리 결과도 화면에 노출하지 않는데, 그래도 SQL Injection이 가능한가요? 어떻게 막나요?"

왜 면접에서 묻는가?

"결과를 숨기면 안전하다"는 흔한 오해를 넘어서는 이해를 확인하는 질문이다. Blind Injection의 원리를 설명할 수 있는지, 그리고 이것이 완화책과 근본 방어의 차이를 이해하는지를 함께 본다.

답변 사고 순서

  1. 가능하다고 명확히 답변 — 참/거짓 신호나 응답 시간 차이만으로도 데이터 추출이 가능함을 설명한다.
  2. 동작 원리 설명 — Boolean-based와 Time-based를 구체적으로 짚는다.
  3. 결론 — 에러 숨김/결과 숨김은 "완화책"일 뿐 근본 방어(Parameterized Query)를 대체할 수 없음을 명시한다.
강한 답변 예시

"가능합니다. 화면에 정보가 전혀 없어도 응답이 '참일 때'와 '거짓일 때' 다르게 동작하기만 하면 그 차이 자체가 정보가 됩니다. 예를 들어 조건이 참일 때만 결과가 있는 화면이 뜨거나, Time-based라면 조건이 참일 때만 SLEEP으로 응답을 지연시켜서 한 글자씩 데이터를 추출할 수 있습니다. 자동화 도구를 쓰면 수천 번의 요청으로 짧은 시간 안에 데이터베이스 전체를 복원할 수 있어요. 그래서 에러 메시지를 숨기거나 결과를 노출하지 않는 건 공격을 조금 더 어렵고 느리게 만드는 완화책일 뿐이고, 근본 방어는 여전히 Parameterized Query로 애초에 Injection 자체가 성립하지 않게 만드는 것입니다."

약한 답변 예시

"에러 메시지도 안 보여주고 결과도 안 나오면 공격자가 아무 정보도 못 얻으니 안전합니다."

왜 약한가? Blind Injection의 존재 자체를 모르고 있다는 신호다. "정보 노출이 없다 = 안전"이라는 결론은 틀렸다.

Senior-level Follow-up 질문들

Follow-up: "Blind SQL Injection 시도를 로그에서 어떻게 탐지하시겠습니까?"

모범 답변 방향: "같은 IP/세션에서 짧은 시간 안에 유사한 쿼리 파라미터가 반복적으로, 그리고 조금씩 변형되며 대량으로 들어오는 패턴을 탐지 규칙으로 만듭니다. 특히 SLEEP, WAITFOR, BENCHMARK 같은 시간 지연 함수 키워드가 파라미터에 포함되거나, 응답 시간이 비정상적으로 튀는 요청이 반복되면 알림을 겁니다. WAF나 APM에서 이런 패턴 기반 탐지 규칙을 설정해두는 게 실무 대응입니다."

10. 면접 질문 3 — 동적 정렬 컬럼을 안전하게 받는 API

면접 질문
"사용자가 목록 API에서 정렬 기준 컬럼을 쿼리 파라미터로 직접 지정할 수 있어야 합니다. Parameterized Query로 컬럼명을 바인딩할 수 없는데, 어떻게 안전하게 구현하시겠습니까?"

왜 면접에서 묻는가?

"Parameterized Query를 쓰면 된다"는 표면적 답변만으로는 풀리지 않는 케이스를 던져서, 근본 원리(값과 식별자의 차이)를 이해하고 다른 방어 전략(화이트리스트)으로 전환할 수 있는지를 확인하는 질문이다.

답변 사고 순서

  1. 제약 인정 — 컬럼명/식별자는 바인딩 파라미터로 처리할 수 없다는 것을 먼저 짚는다.
  2. 대안 원칙 제시 — 사용자 입력을 쿼리에 직접 넣지 않고, 화이트리스트에서 매핑해 선택하게 한다.
  3. 구현 방식 설명 — 허용 목록 밖의 값은 명시적으로 거부(400 에러)하고 기본값으로 대체하지 않는다.
강한 답변 예시

"컬럼명 같은 식별자는 Prepared Statement의 값 바인딩 문법이 지원하지 않기 때문에 다른 방식으로 접근합니다. 서버에서 허용할 정렬 컬럼과 방향을 화이트리스트(예: created_at, name, price / asc, desc)로 미리 정의해두고, 사용자 입력값이 이 목록에 있는지만 검증합니다. 목록에 없으면 400 에러로 명확히 거부하고, 임의로 기본값으로 대체하지 않습니다 — 조용히 대체하면 공격자가 어떤 입력이 통과됐는지 추론하기 어려워지는 대신 정당한 사용자에게는 예상치 못한 동작으로 보일 수 있기 때문입니다. 검증을 통과한 값만 쿼리 문자열에 삽입하므로, 사용자가 통제할 수 있는 임의 문자열이 쿼리 구조에 들어가는 지점 자체가 없습니다."

약한 답변 예시

"컬럼명에 있는 특수문자만 걸러내고(예: 세미콜론 제거) 그대로 씁니다."

왜 약한가? 블랙리스트 방식은 예상 못 한 우회 패턴에 취약하다. 화이트리스트처럼 "허용된 것만 통과"가 아니라 "위험해 보이는 것만 차단"하는 접근은 근본적으로 취약하다.

Senior-level Follow-up 질문들

Follow-up: "정렬 가능한 컬럼이 수십 개라서 화이트리스트 관리가 부담스럽다면 어떻게 하겠습니까?"

모범 답변 방향: "컬럼 개수가 많아도 원칙은 같습니다. API에서 노출하는 '정렬 키' 이름(예: sort=latest)과 실제 DB 컬럼명(created_at)을 분리한 매핑 테이블을 코드에 상수로 관리하면, 사용자는 애초에 실제 컬럼명을 알 필요도 없고 서버는 매핑에 있는 키만 처리합니다. 이렇게 하면 화이트리스트가 커져도 관리는 매핑 딕셔너리 하나로 집중되고, 동시에 실제 스키마 구조(컬럼명)를 API 응답에서 숨기는 부가 효과도 있습니다."
