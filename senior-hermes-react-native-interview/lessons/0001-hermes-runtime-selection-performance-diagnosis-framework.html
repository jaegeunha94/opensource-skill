Day 1 — Hermes runtime 선택과 React Native 성능 진단 프레임워크

JS 엔진 선택의 근거를 세우고, 프로덕션 성능 문제를 좁혀가는 사고법 · 2026-07-03

최신성 참고 (2026-07-03 기준 리서치)

Hermes/React Native는 최근 구조적으로 크게 변했습니다: JavaScriptCore(JSC)는 RN 0.81(2025-08)에서 코어에서 제거되어 커뮤니티 패키지로만 남았고, New Architecture(Fabric/TurboModules)는 RN 0.82(2025-10)부터 끌 수 없는 유일한 아키텍처가 되었으며, Hermes V1(재작성된 컴파일러/VM)이 RN 0.84(2026-02)부터 기본 엔진이 되었습니다. 이 레슨은 이 시점 기준 최신 상태를 반영합니다. 면접에서 "Hermes vs JSC"를 물으면 이제 "JSC가 대안으로 존재한다"가 아니라 "JSC는 사실상 레거시이고 별도 패키지로만 켤 수 있다"는 전제로 답해야 정확합니다.

1. 핵심 면접 질문

Primary Question

"React Native 앱이 왜 브라우저용 JS 엔진을 그대로 쓰지 않고 Hermes라는 별도 엔진을 쓰는지 설명해 주시겠어요? 그리고 실제로 이 선택이 성능에 어떤 차이를 만드나요?"

Variant 1

"사용자가 '앱이 느려요'라고 리포트했습니다. 코드를 보기 전에, 어떤 순서로 원인을 좁혀가시겠어요?"

Variant 2

"지금 담당하시는 앱이 아직 JSC를 쓰고 있다면 Hermes로 옮기시겠습니까? 어떤 기준으로 판단하시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 "Hermes가 뭔지 아는가"가 아니라 "모바일 런타임의 제약을 이해하고, 그 제약에서 출발해 성능 문제를 체계적으로 좁혀갈 수 있는가"를 테스트합니다. "Hermes는 Meta가 만든 JS 엔진이고 성능이 좋습니다" 수준의 답변은 표면적 사실 나열일 뿐, 왜 좋은지 · 무엇과 비교해서 좋은지 · 언제는 오히려 불리한지를 설명하지 못합니다. 시니어는 이 판단을 순서 있게 설명할 수 있어야 합니다.

면접관이 이 질문으로 평가하는 것:

  • 런타임에 대한 정확한 모델 — JS 엔진이 실제로 무엇을 하는지(파싱, 컴파일, 실행, 메모리 관리), 모바일 환경이 브라우저 환경과 어떤 제약(콜드 스타트, 배터리, 앱 크기, 네이티브 코드와의 경계)에서 다른지 이해하고 있는가.
  • 진단 절차의 체계성 — "느리다"는 모호한 리포트를 받았을 때, 감으로 코드를 뒤지는 게 아니라 startup/interaction/memory/native bridge 같은 축으로 문제를 좁혀가는 절차를 갖고 있는가.
  • 최신 상태에 대한 정확한 인지 — 이 분야는 최근 1~2년 사이 JSC 제거, New Architecture 강제화, Hermes V1 출시처럼 구조적으로 바뀌었습니다. 오래된 지식("Hermes와 JSC 중에 골라 쓴다")만 갖고 있으면 실무 판단이 틀어집니다.
  • trade-off를 다루는 태도 — "Hermes가 무조건 빠르다"가 아니라 "무엇을 얻고 무엇을 내주는지"를 설명할 수 있는가(예: 매우 짧고 CPU-heavy한 스크립트에서는 JIT 없는 인터프리터가 V8류 엔진보다 느릴 수 있다는 것도 압니다).

10년차 이하에서 자주 나오는 답변 패턴: "Hermes를 켜면 앱이 빨라진다고 들었습니다. `gradle.properties`에서 `hermesEnabled=true`로 설정하면 됩니다." 이 답변은 설정 방법만 알 뿐, 왜 빨라지는지 · 무엇을 희생하는지 · 지금(2026년) 시점에 이 선택이 사실상 선택이 아니라 기본값이라는 것을 모릅니다. 시니어는 "무엇을 켜는가"가 아니라 "이 런타임이 어떤 제약 아래에서 어떤 설계를 택했고, 그것이 우리 앱의 어떤 지표에 영향을 주는가"로 답합니다.

3. 전제 개념 — 쉬운 한국어로

JS 엔진(런타임)이란 무엇인가

JavaScript는 그 자체로는 실행될 수 없는 "글자로 적힌 규칙"일 뿐입니다. 이 글자를 실제로 실행 가능한 동작으로 바꿔주는 프로그램이 JS 엔진입니다. 브라우저에는 이미 JS 엔진이 내장돼 있습니다(Chrome은 V8, Safari는 JavaScriptCore/JSC). 그런데 React Native 앱은 브라우저가 아니라 순수 네이티브 앱이기 때문에, JS를 실행할 엔진을 앱 안에 직접 포함시켜야 합니다. 이것이 Hermes가 존재하는 이유입니다 — "브라우저에 내장된 것과 같은 역할을 하는 JS 엔진을, 모바일 앱 배포판(APK/IPA) 안에 직접 넣어서 가져가는 것"입니다.

인터프리터, JIT, AOT — 실행 방식의 차이

JS 엔진이 코드를 실행하는 방식에는 여러 전략이 있습니다. 이 워크스페이스 전체에서 반복해서 등장하는 세 용어를 여기서 정리합니다.

  • 인터프리터(Interpreter) — 코드(또는 그것을 변환한 bytecode)를 한 줄씩 읽으면서 즉시 실행. 시작은 빠르지만 반복되는 코드도 매번 다시 해석하므로 장시간 실행되는 무거운 연산에는 느립니다.
  • JIT(Just-In-Time) 컴파일 — 실행하면서 "자주 실행되는 코드"를 감지해 그 부분만 실시간으로 기계어로 컴파일해 이후 실행을 빠르게 만드는 방식. V8, JSC가 강력한 JIT를 갖고 있습니다. 다만 JIT 자체가 컴파일 작업을 하므로 CPU와 메모리를 씁니다.
  • AOT(Ahead-Of-Time) 컴파일 — 앱을 실행하기 전, 빌드 시점에 미리 컴파일을 끝내두는 방식. 사용자의 기기에서는 이미 컴파일된 결과물만 읽어서 실행하면 되므로, 기기에서 파싱·컴파일에 쓰는 시간과 메모리를 줄일 수 있습니다. Hermes의 핵심 전략이 이것입니다.

택배에 비유하면, 인터프리터는 "주문할 때마다 그 자리에서 조립해서 주는 것", JIT은 "자주 나가는 상품은 미리 조립해 두고 나머지는 그때그때 조립하는 것", AOT는 "공장에서 미리 완제품으로 포장해서 창고에 쌓아두는 것"에 가깝습니다. 완제품을 미리 만들어두면 주문 즉시 내줄 수 있지만(빠른 시작), 상품 다양성에 따라 유연하게 대응하기는 어렵습니다(런타임에 새로 최적화하기 어려움).

Bytecode란 무엇인가

Bytecode는 사람이 읽는 JS 코드와 기계어(CPU가 직접 실행하는 이진 명령) 사이에 있는 중간 표현입니다. Hermes는 빌드 시점에 JS를 Hermes Bytecode(HBC)라는 자체 포맷으로 미리 변환해 둡니다. 이 bytecode는 기기에서 파일을 통째로 파싱하지 않고 메모리에 그대로 매핑(mmap)해서 필요한 부분만 읽어 들일 수 있도록 설계되어 있어, 이것이 Hermes가 앱 시작 시간을 줄이는 핵심 메커니즘입니다. 자세한 컴파일 파이프라인은 Day 3에서 다룹니다.

Native bridge 경계란 무엇인가

React Native 앱은 JS로 작성한 로직(비즈니스 로직, UI 상태)과, 실제 화면을 그리고 카메라·파일시스템 같은 기기 기능을 다루는 네이티브 코드(Java/Kotlin, Objective-C/Swift)가 함께 동작합니다. 이 둘은 서로 다른 실행 환경이기 때문에 "JS가 네이티브에게 뭔가를 요청하고 결과를 받는" 통로가 필요합니다. 이 통로를 예전에는 "bridge"라고 불렀고(JSON으로 직렬화해서 비동기로 주고받음), 지금은 JSI라는 더 직접적인 방식으로 대체되었습니다(Day 7에서 심화). 오늘은 "JS 코드와 네이티브 코드 사이에는 항상 건너야 할 경계가 있고, 이 경계를 넘는 비용과 그 경계에서 나는 장애가 있다"는 사실만 기억하면 충분합니다.

왜 "런타임 선택"이 사소한 설정값이 아닌가

"JS 엔진을 무엇으로 쓸지"는 언뜻 `package.json`의 옵션 하나처럼 보이지만, 실제로는 앱의 콜드 스타트 시간, 메모리 사용량, 크래시 시 디버깅 난이도, 배포 파이프라인 전체(빌드 시간, 소스맵, 바이너리 크기)에 영향을 주는 아키텍처 결정입니다. 이 결정을 잘못 이해하면 "왜 이 크래시 스택은 읽을 수 없는 문자로 나오는가", "왜 OTA 업데이트를 배포했더니 앱이 즉시 죽는가" 같은 실제 장애를 진단할 기초 모델 자체가 없는 상태가 됩니다.

4. Hermes/React Native 런타임 히스토리 지도

면접관이 최신 상태를 정확히 아는지 확인하려고 던지는 질문이 많기 때문에, 이 타임라인을 먼저 정확히 잡아두는 것이 이후 모든 Day의 기준점이 됩니다.

시점 변화 실무적 의미
RN 0.70
(2022-09)		 Hermes가 새 프로젝트의 기본 엔진이 됨 이때까지는 "Hermes를 쓸지 JSC를 쓸지" 선택이 실질적인 의미가 있었음
RN 0.74
(2024-04)		 Bridgeless 모드가 New Architecture 사용 시 기본값이 됨 JSON 직렬화 기반 구 bridge를 거치지 않고 JSI로 직접 통신하는 것이 기본 경로가 되기 시작
RN 0.76
(2024-10)		 New Architecture(Fabric+TurboModules)가 새 프로젝트 기본값이 됨. React Native DevTools가 표준 디버거로 자리잡기 시작 "New Architecture로 옮길지 말지"가 아니라 "언제 옮길지"의 문제가 됨
RN 0.81
(2025-08)		 JavaScriptCore(JSC)의 1st-party 지원이 코어에서 완전히 제거됨 JSC를 쓰려면 @react-native-community/javascriptcore 커뮤니티 패키지를 별도로 설치하고 New Architecture, 특정 빌드 플래그가 필요함 — 더 이상 "기본 옵션 중 하나"가 아님
RN 0.82
(2025-10)		 New Architecture opt-out이 완전히 제거됨(끌 수 없음). Hermes V1이 실험적 opt-in으로 도입됨 레거시 아키텍처는 기술적으로 실행 자체가 불가능해짐. New Architecture 이해가 이제 "심화 주제"가 아니라 "기본 전제"
RN 0.84
(2026-02)		 Hermes V1(재작성된 컴파일러·VM·bytecode 포맷)이 기본 엔진이 됨. 레거시 아키텍처 코드가 대거 제거됨 "Hermes vs JSC" 비교는 사실상 역사적 비교가 됨. Hermes V1과, 아직 실험 단계인 Static Hermes(네이티브 AOT 컴파일)를 혼동하지 않는 것이 중요해짐

왜 이 타임라인을 외우는 것보다 "구조"를 이해하는 게 중요한가

날짜와 버전 번호 자체를 암기하는 것은 이 워크스페이스의 목표가 아닙니다(버전은 계속 올라갑니다). 중요한 것은 이 흐름이 보여주는 일관된 방향성입니다 — React Native는 지속적으로 "선택지를 줄이고 하나의 최적화된 경로로 수렴"시키는 방향으로 움직이고 있습니다(엔진: JSC/Hermes 선택 → Hermes 사실상 고정, 아키텍처: 구/신 선택 → New Architecture 고정). 면접에서 최신 버전 번호를 정확히 모르더라도, "지금 이 생태계가 어느 방향으로 수렴하고 있는가"를 설명할 수 있으면 시니어다운 답변이 됩니다.

5. React Native 성능의 4개 축

"앱이 느려요"는 사실 서로 다른 4가지 문제 중 하나(또는 여럿)를 뭉뚱그린 표현입니다. 이 4개 축을 구분하지 못하면 엉뚱한 곳을 최적화하게 됩니다.

축 무엇을 측정하는가 주로 관련된 컴포넌트
① Startup 앱을 켜서 처음 상호작용 가능해지기까지(TTI, Time-to-Interactive) JS bundle 크기, bytecode 로딩(mmap), 초기 렌더링 비용
② Interaction(런타임) 이미 실행 중인 앱에서 사용자 조작에 반응하는 속도, 프레임 드랍(jank) JS 실행 속도, GC pause, 리렌더링 비용
③ Memory 앱이 사용하는 메모리 총량과 그 증가 추세(OOM 크래시 위험) heap 크기, 메모리 누수, GC가 회수하는 속도
④ Native bridge 경계 JS와 네이티브 코드가 서로를 호출할 때의 비용과 안정성 JSI 호출, TurboModule 예외 전파, 동기/비동기 호출 설계

이 4개 축이 실무에서 중요한 이유

예를 들어 "앱이 켜질 때 흰 화면이 오래 떠 있다"는 리포트는 ①번 축의 문제일 가능성이 높고, "스크롤할 때 자꾸 끊긴다"는 ②번, "오래 쓰면 앱이 강제 종료된다"는 ③번, "특정 네이티브 기능(카메라, 결제)을 쓸 때만 크래시가 난다"는 ④번일 가능성이 높습니다. 각 축은 원인도, 해결 도구도, 담당해야 할 지식(Day 3~6은 ①②③, Day 7~10은 ④)도 다릅니다. 이 구분 없이 "일단 Hermes를 켜본다" 같은 처방은 문제를 우연히 해결할 수는 있어도 재현 가능한 진단이 아닙니다.

6. 4단계 성능 진단 프레임워크

면접에서 "앱이 느려요"라는 모호한 문제를 받았을 때, 혹은 실제 프로덕션 장애 대응에서 시니어가 실제로 쓰는 사고 순서입니다.

  • 1
    언제 느린가를 특정한다 (Timing Localization)

    "앱을 켤 때"인지, "이미 켜져 있는데 특정 화면/조작에서"인지, "오래 쓰다 보면"인지, "특정 네이티브 기능을 쓸 때"인지 먼저 구분합니다. 이 대답만으로 위 4개 축 중 어디를 볼지가 대부분 결정됩니다.

  • 2
    재현 가능한 지표로 바꾼다 (Quantify)

    "느리다"는 주관적 표현을 TTI(ms), 프레임 드랍률, 메모리 사용량(MB) 같은 숫자로 바꿉니다. React Native DevTools의 Performance 패널, Xcode Instruments, Android Studio Profiler 같은 도구로 실제 수치를 확인하기 전에는 어떤 가설도 확정하지 않습니다.

  • 3
    경계를 좁힌다 (JS 로직 vs 엔진/GC vs 네이티브 브릿지)

    문제가 JS 코드 자체(비효율적인 알고리즘, 불필요한 리렌더링)에 있는지, 엔진 레벨(GC pause, bytecode 로딩)에 있는지, 아니면 native bridge 경계(TurboModule 호출 비용, 동기 호출이 메인 스레드를 막는 것)에 있는지를 프로파일러로 좁힙니다. 이 단계에서 "Hermes 자체의 문제"와 "우리가 Hermes 위에서 짠 코드의 문제"를 혼동하지 않는 것이 중요합니다.

  • 4
    배포/환경 변수를 마지막에 점검한다 (Deployment Variable Check)

    같은 코드인데 특정 버전 업그레이드 이후, 혹은 OTA 업데이트 배포 이후에만 느려지거나 크래시가 난다면, 코드 로직이 아니라 Hermes 버전 호환성이나 bytecode 미스매치 같은 배포 계층 문제일 가능성을 점검합니다(Day 16, 18에서 심화). 이 단계를 1~3번보다 먼저 의심하면 잘못된 곳에서 시간을 씁니다 — 항상 "코드/런타임 레벨에서 설명이 안 될 때" 마지막으로 확인하는 순서를 지킵니다.

7. 런타임/아키텍처 선택 판단표

2026년 현재 시점에서 "JS 엔진/아키텍처를 어떻게 선택할 것인가"라는 질문에 대한 실무적 답은 대부분의 신규·현행 프로젝트에서 이미 결정되어 있습니다. 그럼에도 면접에서는 "왜 그렇게 결정되었는가"를 설명해야 합니다.

상황 현실적 선택 근거
신규 React Native 프로젝트 Hermes(V1) + New Architecture, 사실상 기본값이자 유일한 지원 경로 RN 0.82+부터 구 아키텍처는 켤 수조차 없고, JSC는 별도 커뮤니티 패키지·빌드 플래그가 필요해 유지보수 부담이 큼
오래된 앱이 아직 JSC + 구 아키텍처 업그레이드 로드맵 수립이 필요 (Day 10에서 심화) 구 아키텍처는 2025년 6월부터 기능 동결(freeze)되었고, 서드파티 라이브러리들도 점점 신 아키텍처 전용으로 이동 중이라 방치할수록 마이그레이션 비용이 커짐
매우 CPU-heavy한 계산(이미지 처리, 암호화 등)을 JS로 자주 돌려야 하는 경우 Hermes 인터프리터에만 의존하지 않고 네이티브 모듈(JSI 기반)로 오프로드하는 것을 고려 Hermes는 startup/memory에 최적화된 설계이지 강력한 JIT을 가진 V8류 엔진처럼 장시간 연산에 최적화된 것은 아님 — "무조건 빠르다"가 아니라 "무엇에 최적화됐는가"의 문제
"Static Hermes를 프로덕션에 쓸 수 있는가?"라는 질문 2026년 중반 기준 아직 아니다 — 여전히 `static_h` 브랜치 기반 실험 단계 Hermes V1은 Static Hermes 연구 트랙에서 파생된 프로덕션용 엔진이지만, 네이티브 코드 생성이나 JIT 같은 Static Hermes의 핵심 기능은 아직 포함하지 않음 — 이 구분을 못 하면 흔한 오해로 이어짐(12절 참고)

8. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 됩니다.

  1. 질문의 종류를 구분합니다. "왜 Hermes를 쓰는가"류 질문인지, "앱이 느리다"류 진단 질문인지에 따라 3~4절(모바일 런타임 제약) 또는 6절(4단계 프레임워크)로 들어갑니다.
  2. 모바일 런타임의 근본 제약을 먼저 짚습니다. "브라우저와 달리 모바일 앱은 콜드 스타트마다 JS 파싱·컴파일 비용을 지불하고, 앱 크기와 메모리가 사용자 이탈에 직결된다"는 전제를 먼저 세웁니다.
  3. Hermes의 설계 선택을 그 제약과 연결합니다. AOT bytecode 컴파일이 왜 그 제약을 완화하는지 설명합니다.
  4. 진단이 필요한 질문이면 4단계 프레임워크를 순서대로 적용합니다. 시점 특정 → 정량화 → 경계 좁히기 → 배포 변수 점검.
  5. trade-off와 최신 상태를 덧붙입니다. "다만 Hermes가 모든 경우에 우월한 건 아니고, 현재는 사실상 기본값이 되어 선택의 여지가 크지 않다"처럼 균형 잡힌 결론으로 마무리합니다.

9. 핵심 답변 예시

면접관: "사용자가 '앱이 느려요'라고 리포트했습니다. 코드를 보기 전에, 어떤 순서로 원인을 좁혀가시겠어요?"

STRONG ANSWER

가장 먼저 "언제 느린가"를 사용자에게 구체적으로 확인합니다. 앱을 처음 켤 때인지, 특정 화면에서 스크롤/애니메이션할 때인지, 오래 쓰다 보면 그런 건지, 아니면 특정 기능(예: 카메라, 결제)을 쓸 때만인지에 따라 완전히 다른 문제입니다. 이 네 가지는 각각 startup, interaction, memory, native bridge 경계라는 서로 다른 축의 문제이기 때문입니다.


다음으로 이걸 숫자로 바꿉니다. "느리다"는 감상이 아니라 React Native DevTools의 Performance 패널이나 Instruments/Profiler로 실제 TTI, 프레임 드랍률, 메모리 사용량을 측정합니다. 이 단계 없이 가설을 세우면 나중에 되돌리기 어려운 최적화를 잘못된 곳에 쏟게 됩니다.


측정 후에는 문제를 JS 로직 문제인지, 엔진/GC 레벨 문제인지, native bridge 경계 문제인지로 좁힙니다. 예를 들어 프레임 드랍이 GC pause와 시간이 겹치면 GC 문제(Day 6에서 다루는 Hades 같은 GC 설계와 관련), 특정 네이티브 기능 호출 시에만 멈추면 TurboModule의 동기 호출이 메인 스레드를 막고 있는 것일 수 있습니다.


마지막으로, 위 세 단계로 설명이 안 되고 특정 릴리스나 OTA 배포 이후에만 문제가 재현된다면 Hermes 버전 호환성이나 bytecode 미스매치 같은 배포 계층 문제를 의심합니다. 이 순서를 지키는 이유는, 배포 문제를 먼저 의심하면 실제로는 코드 문제인데 배포 파이프라인을 뒤지느라 시간을 낭비하기 쉽기 때문입니다.

이 답변이 좋은 이유:

  • "느리다"는 모호한 리포트를 곧바로 코드 추측으로 넘어가지 않고, 축을 나누는 질문부터 시작함
  • 정량화 단계를 명시적으로 넣어서 "감이 아니라 측정"이라는 시니어급 태도를 드러냄
  • 4개 축(startup/interaction/memory/native bridge)과 구체적 도구(DevTools Performance 패널, Instruments/Profiler)를 연결해서 추상적이지 않게 답함
  • 배포 변수 점검을 의도적으로 마지막에 배치한 이유까지 설명해서, 단순 체크리스트 암기가 아니라 사고 순서라는 것을 보여줌

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"일단 Hermes가 켜져 있는지 확인하고, 안 켜져 있으면 켭니다. Hermes를 쓰면 대부분 빨라진다고 알고 있습니다."

이 답변이 약한 이유

  • 2026년 현재 대부분의 React Native 프로젝트는 이미 Hermes가 기본값이므로, 이 답변은 현재 생태계 상태에 대한 이해가 없다는 인상을 줍니다.
  • "느리다"를 4개 축으로 나누지 않고 곧바로 하나의 처방(Hermes 켜기)으로 건너뛰어, 문제를 진단하는 절차 자체가 없다는 것을 드러냅니다.
  • "대부분 빨라진다"는 근거 없는 일반화이며, Hermes가 무엇에 최적화되어 있고 무엇에는 그렇지 않은지에 대한 이해가 없습니다.
WEAK ANSWER — 또 다른 패턴

"Hermes는 V8보다 빠른 엔진입니다. Meta가 만들어서 신뢰할 수 있습니다."

이 답변이 약한 이유

  • "빠르다/느리다"를 무엇을 기준으로 비교하는지(startup? 장시간 연산? 메모리?) 명시하지 않은 채 단정합니다. 실제로는 트레이드오프가 있는 설계 선택입니다(7절 판단표 참고).
  • "Meta가 만들어서 신뢰할 수 있다"는 출처의 권위에 기댄 답변으로, 기술적 근거가 없습니다.
STRONG ANSWER 요소 체크리스트
  • "느리다"는 리포트를 곧바로 4개 축(startup/interaction/memory/native bridge) 중 무엇인지 좁히는 질문부터 시작한다
  • 가설을 세우기 전에 실제 도구로 정량화하는 단계를 명시한다
  • Hermes의 성능 이점을 "무엇과 비교해서, 무엇을 기준으로"인지 구체적으로 설명한다(예: 콜드 스타트 시 AOT bytecode 로딩)
  • Hermes가 모든 상황에서 우월하지 않다는 trade-off를 알고 있다
  • 배포/버전 계층 문제를 코드 문제와 구분해서 마지막에 점검하는 이유를 설명한다

11. Senior-level Follow-up Q&A

Q: Hermes가 항상 JSC/V8보다 빠른가요?

모범 답변: 아닙니다. Hermes는 콜드 스타트 시간과 메모리 footprint를 줄이는 데 최적화된 설계이고, 이를 위해 AOT bytecode 컴파일과 상대적으로 가벼운 인터프리터를 택했습니다. 반면 V8이나 (구)JSC 같은 엔진은 강력한 JIT을 통해 오래 실행되며 반복이 많은 무거운 연산에서 유리할 수 있습니다. 즉 "짧게, 자주 시작하고 끝나는 모바일 앱의 UI 로직"에는 Hermes의 설계가 유리하지만, 앱 내에서 매우 CPU-heavy한 연산을 오래 돌려야 한다면 JIT이 따라잡을 시간이 있는 엔진이나 아예 네이티브 모듈로 오프로드하는 설계가 더 나을 수 있습니다. "빠르다"는 항상 무엇을 기준으로 하는지와 함께 말해야 정확한 답이 됩니다.

Q: 지금(2026년) 새 프로젝트에서 JSC를 선택할 이유가 있을까요?

모범 답변: 실질적으로 거의 없습니다. JSC는 RN 0.81에서 코어 지원이 제거되어 `@react-native-community/javascriptcore` 같은 별도 커뮤니티 패키지를 설치하고, New Architecture를 전제로 한 특정 빌드 플래그와 네이티브 코드 수정까지 필요합니다. 유지보수 부담이 Hermes를 그대로 쓰는 것보다 훨씬 큽니다. 다만 "이미 JSC 기반으로 깊게 커스터마이징된 레거시 앱을 급하게 옮길 수 없는 경우"처럼 마이그레이션 리스크가 더 큰 특수한 상황에서는 일시적으로 유지할 수는 있습니다 — 이 경우에도 로드맵 상 최종적으로는 Hermes로 옮기는 것이 방향입니다.

Q: Hermes V1과 Static Hermes는 같은 건가요?

모범 답변: 아닙니다. 둘 다 같은 `static_h`라는 실험 브랜치에서 파생됐다는 계보는 같지만, Hermes V1은 재작성된 컴파일러·VM·bytecode 포맷을 프로덕션에 내놓은 것이고, Static Hermes는 타입 정보(TypeScript/Flow)를 활용해 JS를 아예 네이티브 코드(C를 거쳐 기계어)로 AOT 컴파일하는, 훨씬 더 급진적인 실험 트랙입니다. Hermes V1은 아직 네이티브 코드 생성이나 JIT 같은 Static Hermes의 핵심 기능을 포함하지 않습니다. 즉 Hermes V1은 "Static Hermes로 가는 길 위에 있는, 이미 출시된 중간 단계"로 이해하는 것이 정확합니다.

Q: "앱이 느리다"는 리포트에서 native bridge 경계 문제라고 의심할 수 있는 구체적인 신호는 무엇인가요?

모범 답변: 몇 가지 패턴이 있습니다. 첫째, 특정 네이티브 기능(카메라, 결제, 파일 업로드 등)을 호출할 때만 UI가 멈추면 해당 호출이 동기적으로 메인 스레드를 막고 있을 가능성이 큽니다. 둘째, 순수 JS 로직만 도는 화면은 멀쩡한데 네이티브 모듈을 많이 쓰는 화면에서만 프레임 드랍이 있다면 TurboModule 호출 비용이나 직렬화 오버헤드를 의심합니다. 셋째, 크래시 로그에 `EXC_BAD_ACCESS`나 `SIGABRT` 같은 네이티브 레벨 예외가 JS 코드 근처에서 발생한다면, 네이티브 모듈에서 던진 예외가 JS로 제대로 전파되지 않고 프로세스 자체를 종료시키는 경우일 수 있습니다(Day 14에서 구체적인 크래시 패턴을 다룹니다).

Q: 왜 배포/버전 계층 점검을 진단 프레임워크의 마지막에 두나요? 먼저 확인하면 안 되나요?

모범 답변: 먼저 확인해도 틀린 것은 아니지만, 실무에서는 "최근에 배포했으니 그것 때문일 것"이라는 확증편향에 빠지기 쉽습니다. 실제로는 배포와 무관하게 원래 있던 코드/엔진 레벨 문제가 우연히 그 시점에 드러난 경우도 많습니다. 축(4~7절)과 코드/엔진 레벨(1~3단계)로 먼저 좁혀서 "설명 가능한 원인이 없다"는 것을 확인한 뒤에 배포 변수를 보는 것이, 원인을 잘못 지목해서 엉뚱한 롤백이나 재배포를 반복하는 것을 막습니다. 다만 예외적으로 "배포 직후 크래시율이 급등"처럼 신호가 매우 명확한 경우에는 순서를 바꿔 배포 변수부터 즉시 점검하는 것이 맞습니다 — 이 프레임워크는 고정된 규칙이 아니라 기본 순서라는 점이 중요합니다.

12. 흔한 오해

오해 1

"Hermes를 쓰면 JS 코드가 네이티브 코드로 컴파일되어 실행된다"

아닙니다. Hermes는 JS를 bytecode로 AOT 컴파일할 뿐이며, 이 bytecode는 여전히 Hermes VM이라는 인터프리터가 실행합니다. "JS를 아예 네이티브 기계어로 미리 컴파일해서 인터프리터 없이 직접 실행"하는 것은 Static Hermes라는 별도의, 아직 실험 단계인 프로젝트가 지향하는 목표입니다. Hermes V1을 포함한 현재 프로덕션 Hermes는 여전히 bytecode 인터프리터 기반입니다.

오해 2

"JS 엔진 선택은 언제든 쉽게 되돌릴 수 있는 설정값이다"

엔진 전환은 겉보기엔 빌드 설정 플래그 하나처럼 보이지만, 실제로는 네이티브 빌드 구성, 디버깅/크래시 리포팅 도구(source map, symbolication), 서드파티 라이브러리 호환성, 배포 파이프라인 전체에 영향을 줍니다. "설정값을 바꾸면 끝"이 아니라 "런타임을 교체하는 마이그레이션"으로 취급해야 합니다.

오해 3

"Hermes는 무조건 메모리를 적게 쓴다"

Hermes가 콜드 스타트 메모리 사용량을 줄이는 데는 강점이 있지만, GC 설계(GenGC vs Hades) 선택에 따라 특정 워크로드에서는 오히려 메모리 사용량이 늘어나는 trade-off가 보고된 적도 있습니다(Day 6에서 다루는 압축(compaction) 빈도 차이). "Hermes = 저메모리"라는 단순 등식보다, "무엇을 위해 어떤 trade-off를 택했는가"로 이해해야 정확합니다.

오해 4

"New Architecture와 Hermes는 같은 것이다" / "Hermes를 쓰면 자동으로 New Architecture다"

둘은 별개의 축입니다. Hermes는 JS 엔진 선택이고, New Architecture(Fabric+TurboModules+Bridgeless)는 JS와 네이티브가 통신하는 구조 선택입니다. 역사적으로 한동안 Hermes(구 아키텍처) 조합도, JSC(신 아키텍처) 조합도 존재했습니다. 다만 2026년 현재는 신규 프로젝트 기준 사실상 "Hermes V1 + New Architecture"로 수렴되어 있어 구분이 흐려지기 쉬운 것뿐입니다.

오해 5

"앱이 느리면 일단 최신 RN/Hermes 버전으로 올리면 해결된다"

버전 업그레이드가 성능을 개선하는 경우가 많은 것은 사실이지만, "일단 올린다"는 3~4절의 축 구분과 6절의 진단 절차를 건너뛴 처방입니다. 오히려 버전을 올리는 과정에서 Hermes bytecode 버전 미스매치, 서드파티 라이브러리 비호환 같은 새로운 문제가 생길 수 있습니다(Day 16에서 심화). 진단 없는 업그레이드는 "느리다"는 증상을 우연히 없앨 수는 있어도, 다음에 비슷한 문제가 생겼을 때 재사용할 수 있는 지식을 남기지 않습니다.
