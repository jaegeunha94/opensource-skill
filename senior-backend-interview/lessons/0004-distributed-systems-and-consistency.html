1. 오늘 레슨 개요

단일 서버, 단일 DB로 서비스를 운영할 때는 "트랜잭션이면 다 원자적으로 처리된다"는 가정이 유효합니다. 하지만 서비스가 커지면서 데이터가 여러 노드, 여러 서비스로 나뉘는 순간 이 가정은 깨집니다. 네트워크는 끊기고, 메시지는 순서가 뒤바뀌며, 노드는 살아있는지 죽었는지조차 확실하지 않은 상태(unknown)로 관측됩니다. 10년차 이상 면접에서 분산 시스템을 묻는 이유는 "완벽한 일관성은 불가능하다는 전제 위에서, 어떤 손실을 감수하고 어떤 것을 지킬지 설계할 수 있는가"를 확인하기 위해서입니다.

면접관이 실제로 보는 신호
  • CAP를 암기된 정리로 답하는 것 ↔ 파티션 발생 순간에만 강제되는 trade-off라는 것을 이해
  • "결과적 일관성" 단어만 아는 것 ↔ 그로 인해 사용자에게 보이는 구체적 증상과 완화책을 설계
  • 2PC/Saga를 이름만 아는 것 ↔ 실패 시나리오에서 보상(compensation)과 멱등성을 구체적으로 설계

이 레슨은 분산 시스템의 5개 핵심 축을 다루며, 각 영역마다 배경 개념 → 면접 질문 → 답변 구조 → senior follow-up → 좋은/약한 답변 비교 → 자기 점검 순서로 진행합니다.

2. CAP 정리 — 실무에서의 의미와 흔한 오해

배경 개념: CAP는 "3개 중 2개 선택"이 아니다

CAP 정리는 분산 시스템이 Consistency(일관성), Availability(가용성), Partition Tolerance(파티션 내성) 세 가지를 동시에 완벽히 만족할 수 없다는 정리입니다. 그런데 실무에서 흔한 오해는 "이 세 가지 중 두 개를 골라서 시스템을 설계한다"는 것입니다. 실제로는 네트워크 파티션(P)은 분산 시스템에서 선택 사항이 아니라 필연적으로 발생하는 사건입니다. 여러 대의 서버가 네트워크로 연결되어 있는 한, 스위치 장애·케이블 단선·GC 정지로 인한 타임아웃 등으로 노드 간 통신이 끊기는 순간은 반드시 옵니다. 그래서 진짜 질문은 "P를 포기할 것인가"가 아니라, "파티션이 실제로 발생한 그 순간에, C를 포기할 것인가 A를 포기할 것인가"입니다.

면접 질문

"CAP 정리에서 왜 P를 포기하는 선택지는 사실상 없는지 설명해주시고, 여러분이 설계한 서비스에서 C와 A 중 어느 쪽을 포기했는지 구체적 예시로 말씀해주세요."

면접관이 보는 신호

"CA를 선택했다"는 답은 분산 환경에서는 성립하지 않는 개념적 오류입니다. 이 함정을 스스로 짚어내는지, 그리고 도메인마다 C/A 선택이 달라야 한다는 것을 아는지를 봅니다.

파티션이 발생했을 때의 선택

1
파티션이 없는 평상시에는 C와 A 둘 다 얻을 수 있습니다. CAP는 "항상 둘 중 하나만 골라야 한다"는 뜻이 아니라, 파티션이라는 예외 상황에서만 강제되는 trade-off입니다.
2
파티션 발생 시 A를 선택(AP)한다는 것은, 노드가 서로 통신하지 못해도 각자 요청에 응답을 계속한다는 뜻입니다. 이 경우 노드 간 데이터가 서로 다른 값을 가진 상태(inconsistent)로 응답할 수 있습니다. 예: 좋아요 수, 팔로워 수, 상품 조회수 — 잠깐 다른 값을 보여줘도 비즈니스에 큰 손해가 없는 데이터.
3
파티션 발생 시 C를 선택(CP)한다는 것은, 최신 데이터를 보장할 수 없는 노드는 요청을 거부하거나 에러를 반환한다는 뜻입니다. 예: 결제, 재고 차감, 계좌 잔액 — 틀린 값을 보여주느니 잠깐 서비스가 안 되는 게 나은 데이터.
4
한 서비스 전체가 CP나 AP로 고정되지 않습니다. 대부분의 실무 시스템은 도메인별로 다르게 선택합니다. 같은 커머스 서비스라도 "재고/결제"는 CP 성격의 저장소(RDB, 강한 일관성 설정)를, "상품 조회수/추천"은 AP 성격의 저장소(Cassandra, DynamoDB eventual read)를 씁니다.

대표 시스템의 CAP 성향

분류 대표 시스템 파티션 시 동작 적합 도메인
CP 성향 ZooKeeper, etcd, HBase, RDBMS(동기 복제) 과반수(quorum)와 통신 안 되면 쓰기/읽기 거부 설정 저장, 리더 선출, 재고/결제, 잔액
AP 성향 Cassandra, DynamoDB, Riak, CouchDB 파티션 중에도 로컬 노드가 응답, 이후 충돌 해소 세션, 카운터류, 추천, 활동 피드
PACELC로 확장해서 이해하기
CAP는 "파티션이 발생했을 때"만 다룹니다. 하지만 평상시(파티션이 없을 때)에도 지연시간(Latency)과 일관성(Consistency) 사이의 trade-off가 존재합니다. PACELC: Partition 발생 시 Availability와 Consistency 중 선택(PAC), 아니면(Else) Latency와 Consistency 중 선택(LC). 예를 들어 동기 복제로 강한 일관성을 확보하면 평상시에도 쓰기 지연이 늘어납니다. 시니어는 CAP뿐 아니라 이 "평상시 트레이드오프"까지 함께 설명할 수 있어야 합니다.
강한 답변

"네트워크 파티션은 분산 환경에서 발생 여부가 아니라 발생 시점의 문제라서, CA를 선택한다는 것은 성립하지 않습니다. 저희 서비스는 재고 차감은 RDB에서 동기 복제로 처리해 파티션 시 쓰기를 막더라도 정합성을 지키는 CP를 선택했고, 상품 조회수 카운터는 Redis에 AP 성격으로 두어 파티션 중에도 응답이 끊기지 않게 했습니다. 조회수는 잠깐 부정확해도 비즈니스 손실이 없지만, 재고가 틀리면 오버셀링이 발생하기 때문입니다."

약한 답변

"저희는 일관성이 중요해서 CA를 선택했습니다. 모든 데이터가 항상 최신이고 항상 응답 가능하도록 만들었습니다."

약한 답변이 약한 이유: 분산 시스템에서 "CA"는 개념적으로 성립하지 않습니다(파티션이 아예 없다고 가정하는 것과 같음). 이 대답은 CAP 정리 자체를 이해하지 못했다는 신호입니다.

Senior Follow-up

"재고 시스템에 CP를 선택했는데, 파티션이 발생해 재고 확인 API가 계속 실패합니다. 사용자는 '주문하기' 버튼을 눌러도 에러만 봅니다. 어떻게 사용자 경험을 보완하시겠어요?"

모범 추론: 정합성을 지키는 것과 사용자에게 완전히 백지 화면을 보여주는 것은 다른 문제입니다. 1) 재고의 대략적인 상태(예: "재고 있음" 배지)는 약간 지연된 캐시로 보여주되, 2) 실제 결제 확정 시점에만 CP 저장소로 최종 검증합니다. 3) 파티션 중에는 "일시적으로 주문이 지연될 수 있습니다"라는 명시적 안내와 함께 요청을 큐에 적재해 재시도하는 graceful degradation을 적용합니다. 4) Circuit Breaker로 반복 실패를 감지해 완전히 다운되기 전에 대체 경로(웨이팅 리스트, 사전 예약)로 전환합니다. 핵심은 "정합성이 중요한 부분"과 "가용성으로 버텨도 되는 부분"을 같은 화면 안에서도 잘게 나누어 판단하는 것입니다.

3. 일관성 모델 — Strong, Eventual, Causal

배경 개념: "일관성"은 하나의 개념이 아니다

"결과적 일관성(Eventual Consistency)을 씁니다"라는 답은 시작일 뿐입니다. 실무에서 중요한 것은 사용자가 실제로 어떤 이상한 경험을 하게 되는지, 그리고 그것을 완화하는 구체적 기법입니다. 일관성 모델은 "얼마나 빨리, 얼마나 정확하게 모든 노드가 같은 값을 보게 되는가"를 규정하는 계약(contract)입니다.

면접 질문

"Eventual Consistency를 쓰는 저장소로 사용자 프로필을 관리합니다. 사용자가 프로필 사진을 바꾸고 바로 새로고침했는데 이전 사진이 보인다는 CS 문의가 들어왔습니다. 원인과 완화 방법을 설명해주세요."

면접관이 보는 신호

"결과적 일관성이니까 어쩔 수 없다"로 끝나는지, 아니면 일관성 수준을 세분화해서 문제를 좁히는 사고를 하는지를 봅니다.

일관성 모델 비교

모델 보장 내용 비용 적합 사례
Strong Consistency 쓰기 완료 즉시 모든 후속 읽기가 최신 값을 봄 쓰기/읽기 지연 증가, 가용성 저하 가능 결제, 잔액, 재고, 좌석 예약
Eventual Consistency 업데이트가 전파되는 데 시간이 걸리지만 결국 수렴 일시적으로 stale 값 노출 좋아요 수, 조회수, 추천 데이터
Read-your-writes Consistency 본인이 쓴 데이터는 본인이 항상 최신으로 읽음(타인은 지연 허용) 세션/사용자별 라우팅 로직 필요 프로필 수정, 게시글 작성 직후 화면
Monotonic Reads 한 번 최신 값을 읽었으면, 이후 요청에서 더 오래된 값을 보지 않음 같은 replica로 세션 고정 필요 새로고침 시 데이터가 "되돌아가는" 현상 방지
Causal Consistency 인과관계가 있는 이벤트는 그 순서대로 모든 노드에 보임(무관한 이벤트는 순서 보장 안 함) 벡터 시계 등 메타데이터 추적 필요 댓글-답글, 채팅 메시지 순서

단계별 원인 분석과 완화

1
어떤 종류의 불일치인지 먼저 특정합니다. 위 사례는 "본인이 방금 쓴 데이터를 본인이 못 읽는" 문제이므로 Read-your-writes가 깨진 것입니다. 타인이 늦게 보는 것과는 다른 문제입니다.
2
전체 시스템을 Strong Consistency로 바꾸는 것은 대개 과잉 대응입니다. 대신 "본인의 최신 쓰기를 본인에게 보장"하는 좁은 범위의 강한 일관성만 적용합니다. 세션 기반으로 쓰기 직후 일정 시간은 최신 데이터를 보장하는 노드/캐시로 라우팅합니다.
3
클라이언트 측 낙관적 업데이트도 함께 적용합니다. 서버 재조회 없이 UI에서 즉시 변경된 사진을 반영하면, 사용자는 지연을 체감하지 않습니다.
4
모니터링으로 지연의 실제 크기를 측정합니다. "결과적 일관성"이라고만 말하고 넘어가지 않고, p99 전파 지연이 몇 초인지 계측해 완화책의 효과를 검증합니다.
강한 답변

"이건 Read-your-writes Consistency가 깨진 사례입니다. 저희 프로필 저장소는 여러 리전에 결과적 일관성으로 복제되는데, 사용자 본인의 요청은 쓰기 이후 일정 시간(TTL) 동안 원본 리전 또는 캐시된 최신 값으로 라우팅하도록 세션 기반 sticky 정책을 넣었습니다. 동시에 클라이언트에서 업로드 성공 즉시 낙관적으로 사진을 교체해 서버 재조회 자체를 건너뛰게 해서 체감 지연을 없앴습니다. 다른 사용자가 이 프로필을 보는 경우는 결과적 일관성을 그대로 허용해도 문제가 없다고 판단했습니다."

약한 답변

"결과적 일관성이라서 어쩔 수 없습니다. 시간이 지나면 반영됩니다."

약한 답변이 약한 이유: 문제를 "본인이 자신의 쓰기를 못 읽는" 좁은 범위로 좁히지 못했고, 부분적으로 강한 일관성을 적용할 수 있다는 것을 고려하지 않았습니다.

Senior Follow-up

"채팅 앱에서 사용자 A가 메시지를 보내고, B가 그 메시지에 답장을 보냈는데, 다른 사용자 C의 화면에는 답장이 원본 메시지보다 먼저 보입니다. 어떤 문제이고 어떻게 해결하시겠어요?"

모범 추론: 이것은 Causal Consistency가 깨진 사례입니다. 인과관계가 있는 두 이벤트(원본 메시지 → 답장)의 순서가 보장되지 않았습니다. 해결: 1) 각 메시지에 벡터 시계(Vector Clock) 또는 논리적 타임스탬프(Lamport Timestamp)를 부여해 인과관계를 추적합니다. 2) 클라이언트는 원본 메시지를 아직 받지 못한 답장은 버퍼에 잠시 보류했다가, 원본 도착 후 순서대로 렌더링합니다. 3) 실무에서는 완전한 벡터 시계 대신, 채팅방 단위로 단일 시퀀스 번호를 발급하는 중앙 시퀀서(예: 채팅 서버가 방별로 순번을 매김)를 두는 방식이 구현이 단순해 더 많이 쓰입니다. 전역 Causal Consistency는 구현 비용이 크므로, 실무에서는 "같은 대화방/같은 엔티티" 범위로 좁혀 순서를 보장하는 것이 현실적입니다.

4. 분산 합의 — 리더 선출과 Quorum

배경 개념: 누가 "진실"을 결정하는가

여러 노드가 있을 때 "지금 리더가 누구인지", "이 설정 값이 무엇인지"를 모든 노드가 동일하게 알아야 하는 문제를 분산 합의(Consensus) 문제라고 합니다. 네트워크 지연과 노드 장애가 섞인 환경에서 "과반수가 동의한 값만 확정한다"는 원칙(Quorum)이 대부분의 합의 알고리즘의 핵심입니다.

면접 질문

"MySQL 마스터-레플리카 구성에서 마스터가 다운되면 레플리카 중 하나를 새 마스터로 승격합니다. 이 과정을 사람이 수동으로 하지 않고 자동화하려면 어떤 것이 보장되어야 하나요?"

면접관이 보는 신호

단순히 "Raft를 쓴다"는 이름만 아는지, 아니면 왜 과반수(quorum) 기반이어야 스플릿 브레인을 막을 수 있는지 원리를 이해하는지를 봅니다.

Raft 합의의 핵심 개념

1
Term(임기) — 시간은 연속된 임기로 나뉘고, 각 임기마다 최대 한 명의 리더만 존재할 수 있습니다. 리더가 응답하지 않으면 새 임기의 선거가 시작됩니다.
2
과반수 투표(Majority Vote) — 리더가 되려면 전체 노드의 과반수 표를 얻어야 합니다. 5대 중 3대의 동의가 필요하다면, 네트워크 파티션으로 클러스터가 2대와 3대로 쪼개져도 3대 쪽만 과반수를 만족해 리더를 세울 수 있고, 2대 쪽은 리더를 세우지 못합니다. 이것이 스플릿 브레인을 원천 차단하는 원리입니다.
3
로그 복제(Log Replication) — 리더가 받은 쓰기 요청은 과반수 노드에 복제되어야 커밋된 것으로 간주합니다. 아직 과반수에 복제되지 않은 데이터는 "확정되지 않음"으로 취급되어, 리더가 바뀌어도 유실 없이 재구성 가능합니다.
4
하트비트와 타임아웃 — 리더는 주기적으로 하트비트를 보내 자신이 살아있음을 알립니다. 팔로워가 일정 시간 하트비트를 못 받으면 스스로 후보(candidate)가 되어 선거를 시작합니다.

Paxos vs Raft

항목 Paxos Raft
목표 수학적으로 증명된 합의 알고리즘 Paxos와 동등한 안전성, 이해와 구현 용이성 목표
리더 개념 명시적 리더 없이도 동작(Multi-Paxos에서 사실상 리더 도입) 명시적 단일 리더 중심 설계
실무 채택 Chubby(Google), Paxos Made Live 사례 etcd, Consul, CockroachDB, TiKV 등 최근 시스템 다수 채택
이해 난이도 높음(논문 자체가 악명 높게 어려움) 상대적으로 낮음(Term, 로그 복제로 직관적 설명 가능)
실무 팁 — 직접 Raft를 구현할 필요는 거의 없다
대부분의 백엔드 개발자는 Raft/Paxos를 직접 구현하지 않고, 이를 구현한 ZooKeeper, etcd, Consul 같은 조정(coordination) 서비스를 활용해 리더 선출, 분산 락, 설정 저장 문제를 해결합니다. 면접에서 중요한 것은 구현 세부사항 암기가 아니라, "왜 과반수 기반이어야 하는지", "어떤 상황에 이런 조정 서비스가 필요한지" 판단력입니다.
강한 답변

"자동 failover를 안전하게 하려면 과반수 기반 합의가 필요합니다. 저희는 직접 합의 알고리즘을 구현하지 않고 etcd(Raft 기반)를 조정 서비스로 두고, 각 DB 노드의 상태를 관찰하는 Orchestrator가 etcd의 리더 선출 결과를 참조해 승격을 결정하도록 했습니다. 과반수 노드가 기존 마스터와 통신이 끊겼다고 동의해야만 승격이 진행되므로, 네트워크가 일시적으로 불안정한 상황에서 두 대가 동시에 마스터라고 믿는 상황을 방지합니다."

약한 답변

"레플리카 중 아무거나 하나를 새 마스터로 바꿔주는 스크립트를 만들면 됩니다."

약한 답변이 약한 이유: 과반수 합의 없이 단순 스크립트로 승격하면, 네트워크 파티션 상황에서 여러 노드가 동시에 마스터가 되는 스플릿 브레인 위험을 전혀 방지하지 못합니다.

Senior Follow-up

"5대 노드 클러스터에서 네트워크가 3대와 2대로 쪼개졌습니다. 3대 쪽은 새 리더를 세웠는데, 원래 리더가 2대 쪽에 있었고 자신이 여전히 리더라고 믿고 쓰기를 계속 받습니다. 이 상황을 막으려면 무엇이 필요한가요?"

모범 추론: 이것이 바로 스플릿 브레인(Split-Brain)입니다. 방지 원리는 Term(임기) 비교입니다. Raft에서 모든 쓰기 요청과 응답에는 현재 term 번호가 포함됩니다. 2대 쪽에 고립된 옛 리더는 새 term의 존재를 모르지만, 만약 클라이언트나 다른 노드가 새 term 번호를 알고 있다면 옛 리더의 요청을 거부합니다. 또한 옛 리더는 과반수(3대)에 복제 확인을 받아야 커밋을 완료할 수 있는데, 2대만으로는 과반수가 안 되므로 쓰기 자체가 커밋되지 못하고 멈춥니다("blocked", 유실이 아니라 진행 불가 상태). 핵심은 "리더는 스스로 리더라고 믿는 것"이 아니라 매 쓰기마다 과반수의 동의를 실제로 재확인해야 한다는 것이며, 이 메커니즘이 없는 자체 구현 failover 스크립트는 스플릿 브레인에 취약합니다.

5. 분산 트랜잭션 — 2PC의 한계

배경 개념: 여러 서비스에 걸친 "원자성"은 비싸다

단일 DB 트랜잭션은 ACID로 원자성을 보장하지만, MSA 환경에서 주문 서비스·결제 서비스·재고 서비스처럼 서로 다른 DB를 가진 여러 서비스에 걸친 작업을 하나의 트랜잭션처럼 묶으려면 별도의 프로토콜이 필요합니다. 가장 고전적인 방법이 2단계 커밋(2PC, Two-Phase Commit)입니다.

면접 질문

"주문 생성 시 재고 차감과 결제 승인을 원자적으로 처리하고 싶어서 2PC를 검토했습니다. 실제로 MSA에서 2PC를 잘 쓰지 않는 이유는 무엇인가요?"

면접관이 보는 신호

2PC라는 용어를 아는지가 아니라, 블로킹 문제와 코디네이터 SPOF라는 구조적 한계를 이해하고 대안을 제시할 수 있는지를 봅니다.

2PC 동작 순서

1
Prepare(준비) 단계 — 코디네이터가 모든 참가자(재고 서비스, 결제 서비스)에게 "커밋할 준비가 됐는지" 묻습니다. 각 참가자는 실제로 커밋 가능한 상태까지 작업을 수행하고 잠금을 건 뒤 "예/아니오"로 응답합니다.
2
Commit(커밋) 단계 — 모든 참가자가 "예"라고 응답하면 코디네이터가 전원에게 커밋 명령을 보냅니다. 하나라도 "아니오"면 전원에게 롤백 명령을 보냅니다.
3
문제 1 — 블로킹(Blocking Problem): Prepare 단계에서 "예"라고 응답한 참가자는 잠금을 건 채로 코디네이터의 다음 지시를 기다립니다. 이 시점에 코디네이터가 죽으면, 참가자는 커밋해야 할지 롤백해야 할지 알 수 없어 무기한 잠금 상태로 대기합니다. 다른 트랜잭션까지 연쇄적으로 막힙니다.
4
문제 2 — 코디네이터가 SPOF: 코디네이터 자체가 단일 장애점이 되며, 코디네이터를 이중화하려면 다시 별도의 합의 메커니즘이 필요해집니다.
5
문제 3 — 확장성 저하: 참가자가 늘어날수록 모든 참가자가 응답할 때까지 대기해야 하므로, 지연시간이 가장 느린 참가자에 종속되고 처리량이 떨어집니다.
왜 MSA에서는 2PC 대신 Saga를 선호하는가
2PC는 "모든 참가자가 동시에 살아있고 즉시 응답 가능하다"는 강한 가정 위에서 동작합니다. MSA 환경은 각 서비스가 독립적으로 배포·확장되며 네트워크 지연이 크므로 이 가정이 자주 깨집니다. 그래서 실무에서는 즉각적 원자성 대신, 각 단계를 독립적으로 커밋하고 실패 시 보상 작업으로 되돌리는 Saga 패턴을 더 널리 사용합니다.
강한 답변

"2PC는 코디네이터가 죽으면 참가자들이 잠금을 건 채 무기한 대기하는 블로킹 문제가 있고, 코디네이터 자체가 SPOF입니다. 서비스가 독립적으로 배포되는 MSA 환경에서는 한 서비스의 지연이 전체 트랜잭션을 묶어버리는 위험이 너무 큽니다. 그래서 저희는 재고 차감과 결제 승인을 각각 독립적인 로컬 트랜잭션으로 커밋하고, 실패 시 이미 완료된 단계를 역순으로 취소하는 Saga 패턴으로 전환했습니다. 즉각적 원자성 대신 최종적 일관성을 택한 것입니다."

약한 답변

"2PC는 예전 기술이라 요즘은 안 씁니다. 그냥 Saga를 쓰면 됩니다."

약한 답변이 약한 이유: "예전 기술"이라는 근거 없는 이유만 대고, 왜 구조적으로 MSA에 부적합한지(블로킹, SPOF, 확장성) 설명하지 못했습니다.

Senior Follow-up

"그렇다면 2PC는 실무에서 완전히 쓸모없는 기술인가요? 여전히 2PC가 적합한 상황이 있을까요?"

모범 추론: 아닙니다. 2PC는 참가자 수가 적고, 신뢰할 수 있는 네트워크(같은 데이터센터), 강한 즉시 원자성이 반드시 필요한 상황에서는 여전히 유효합니다. 예: 단일 조직이 운영하는 DB 두 대 사이의 분산 트랜잭션, 일부 메시지 브로커의 XA 트랜잭션 지원. 또한 실무 시스템은 완전한 2PC 대신, TCC(Try-Confirm/Cancel) 패턴처럼 2PC의 아이디어를 변형해 타임아웃과 자동 취소를 추가한 절충안을 쓰기도 합니다. 핵심은 "2PC가 항상 나쁘다"가 아니라, 참가자 수·네트워크 신뢰도·서비스 독립 배포 여부에 따라 적합성이 달라진다는 판단입니다.

6. Saga 패턴 — Choreography vs Orchestration

배경 개념: "되돌리기"로 원자성을 흉내낸다

Saga는 하나의 비즈니스 트랜잭션을 여러 개의 로컬 트랜잭션으로 쪼개고, 각 로컬 트랜잭션이 성공하면 다음 단계로 진행하며, 중간에 실패하면 이미 완료된 이전 단계들을 보상 트랜잭션(Compensating Transaction)으로 역순 취소하는 패턴입니다. 2PC처럼 모든 것을 잠그고 기다리는 대신, "일단 진행하고 문제가 생기면 되돌린다"는 접근입니다.

면접 질문

"주문 Saga가 [재고 차감 → 결제 승인 → 배송 요청] 3단계로 구성되어 있습니다. 배송 요청 단계에서 실패했다면 어떻게 되돌리시겠어요? 그리고 이 설계를 Choreography와 Orchestration 중 무엇으로 구현하시겠어요?"

면접관이 보는 신호

보상 트랜잭션을 설계할 수 있는지, 그리고 두 구현 방식의 trade-off를 팀 규모·복잡도 관점에서 판단할 수 있는지를 봅니다.

Choreography vs Orchestration

항목 Choreography(안무형) Orchestration(오케스트레이션형)
구조 각 서비스가 이벤트를 발행/구독하며 스스로 다음 행동 결정 중앙 오케스트레이터가 각 서비스에 명령을 내리고 상태를 관리
결합도 서비스 간 직접 의존 없음, 느슨한 결합 오케스트레이터가 각 서비스 API를 알아야 함
가시성 전체 흐름이 여러 서비스에 분산되어 추적 어려움 전체 흐름이 한 곳에 명시적으로 코드화되어 추적 쉬움
확장성 서비스 수가 적을 때 단순하고 적합 단계가 많고 복잡할수록 유리(중앙에서 타임아웃·재시도 관리)
단점 단계가 늘어나면 이벤트 흐름 파악이 어려워짐("누가 이 이벤트를 구독하는지" 추적 곤란) 오케스트레이터가 SPOF가 될 수 있음(이중화 필요)

보상 트랜잭션 설계 — 단계별 사고

1
각 단계에 대응하는 보상 작업을 미리 정의합니다. 재고 차감 → 보상: 재고 복구. 결제 승인 → 보상: 결제 취소/환불. 배송 요청은 마지막 단계라 보상이 필요 없습니다.
2
실패 지점부터 역순으로 보상을 실행합니다. 배송 요청이 실패했다면: 결제 취소 → 재고 복구 순서로 되돌립니다.
3
보상 트랜잭션은 반드시 멱등(idempotent)해야 합니다. 네트워크 재시도로 같은 보상 요청이 중복 도착해도 재고가 두 번 복구되거나 환불이 두 번 되면 안 됩니다. 요청에 고유한 Saga ID/단계 ID를 포함시켜 이미 처리된 보상은 무시하도록 구현합니다.
4
보상 자체가 실패할 경우를 대비합니다. 재시도(exponential backoff)로도 실패하면 Dead Letter Queue에 적재하고, 알림을 발생시켜 운영자가 수동 개입하도록 합니다. "보상은 반드시 성공한다"는 가정은 위험합니다.
5
중간 상태를 사용자에게 숨기지 않습니다. Saga가 진행 중인 동안 주문 상태를 "결제 확인 중", "배송 준비 중"처럼 명시적으로 노출해, 최종 일관성에 도달하기 전 구간을 사용자가 이해할 수 있게 합니다.

보상 트랜잭션 의사코드

# Orchestration 기반 Saga 예시
def order_saga(order):
    steps = []
    try:
        reserve_inventory(order)
        steps.append("inventory")

        charge_payment(order)
        steps.append("payment")

        request_shipping(order)
        steps.append("shipping")

    except SagaStepFailed as e:
        # 완료된 단계를 역순으로 보상
        for step in reversed(steps):
            compensate(step, order, saga_id=order.saga_id)  # 멱등 처리
        mark_order_failed(order, reason=e)
강한 답변

"3단계 모두 별도 서비스이고 흐름이 명확히 정의되어 있어 Orchestration 방식을 선택했습니다. 오케스트레이터가 배송 요청 실패를 감지하면 결제 취소, 재고 복구 순으로 역순 보상을 실행합니다. 각 보상 API는 Saga ID와 단계 ID를 받아 이미 처리된 요청이면 그대로 성공 응답만 반환하도록 멱등하게 구현했습니다. 보상 자체가 3회 재시도 후에도 실패하면 DLQ에 적재하고 Slack으로 On-call에게 알림을 보내 수동 개입하도록 했습니다. 진행 중에는 주문 상태를 '결제 확인 중'처럼 명시적으로 노출해 사용자가 중간 상태를 이해할 수 있게 했습니다."

약한 답변

"실패하면 이전 단계들을 취소하는 API를 각각 호출하면 됩니다."

약한 답변이 약한 이유: 멱등성, 보상 자체의 실패 처리, Choreography/Orchestration 선택 근거, 사용자에게 중간 상태를 어떻게 보여줄지에 대한 고려가 전혀 없습니다.

Senior Follow-up

"결제 취소(보상) API 자체가 결제 대행사 장애로 계속 실패합니다. 재고는 이미 복구했는데 결제는 취소되지 않은 어중간한 상태가 몇 시간째 지속되고 있습니다. 어떻게 대응하시겠어요?"

모범 추론: 이것은 보상 트랜잭션이 실패하는 최악의 시나리오이며, Saga 설계에서 반드시 다뤄야 하는 부분입니다. 1) 즉시 감지: 보상 실패는 일반 에러보다 높은 심각도(SEV)로 알림을 올려야 합니다. 고객 돈이 묶여 있는 상태이기 때문입니다. 2) 자동 재시도는 exponential backoff로 계속하되, 일정 횟수/시간 초과 시 DLQ로 넘겨 자동 재시도를 멈추고 수동 처리 큐로 전환합니다. 3) 고객 커뮤니케이션: 결제 취소가 지연되고 있음을 고객에게 선제적으로 안내(자동 이메일/알림)해 CS 문의를 줄입니다. 4) 근본 대응: 이런 사고가 재발하지 않도록 결제 대행사에 대한 Circuit Breaker를 두어, 장애가 감지되면 애초에 새로운 결제 승인 자체를 잠시 멈추는 것도 고려합니다. 핵심은 "보상은 항상 성공한다"고 설계하지 않고, 보상마저 실패했을 때의 운영 절차(알림, 수동 처리, 고객 커뮤니케이션)까지 Saga 설계의 일부로 포함시키는 것입니다.

7. 종합 시나리오 — MSA 주문 흐름 설계

면접 질문

"쿠폰 적용, 재고 차감, 결제, 포인트 적립까지 4개의 독립된 서비스에 걸친 주문 흐름을 설계해야 합니다. 데이터 정합성과 시스템 가용성을 어떻게 함께 확보하시겠어요?"

이 질문은 오늘 배운 CAP, 일관성 모델, 2PC의 한계, Saga를 하나의 설계로 종합하는 능력을 봅니다.

1
각 단계의 정합성 요구 수준을 먼저 구분합니다. 재고·결제는 절대 오차가 없어야 하지만(CP 성향), 포인트 적립은 몇 초 늦게 반영돼도 괜찮은 편(AP 성향에 가까움)입니다.
2
2PC 대신 Orchestration 기반 Saga를 채택합니다. 4단계나 되고 각 서비스가 독립 배포되므로, 하나의 오케스트레이터(주문 서비스)가 각 단계를 순서대로 호출하고 실패 시 보상을 트리거합니다.
3
각 단계와 보상 API를 멱등하게 설계합니다. 쿠폰 적용 취소, 재고 복구, 결제 취소, 포인트 회수 각각에 Saga ID를 포함시켜 재시도에 안전하게 만듭니다.
4
실패 가능성이 낮고 되돌리기 어려운 단계는 뒤쪽에 배치합니다. 쿠폰 적용(되돌리기 쉬움) → 재고 차감(되돌리기 쉬움) → 결제(되돌리기 어렵고 외부 의존적) → 포인트 적립(마지막, 실패해도 사용자 영향 적음) 순서로 배치해, 되돌리기 어려운 단계일수록 앞선 단계들이 이미 검증을 마친 상태에서 실행되게 합니다.
5
주문 상태 머신을 명시적으로 노출합니다. "쿠폰 확인 중 → 재고 확인 중 → 결제 중 → 적립 중 → 완료" 각 상태를 DB에 기록하고 사용자에게 실시간으로 보여줘, 최종 일관성에 도달하기까지의 구간을 투명하게 만듭니다.
시니어가 반드시 언급해야 하는 것: "모든 단계를 완벽하게 원자적으로 묶겠다"는 목표 자체가 MSA와 상충한다는 것을 인지해야 합니다. 대신 "어떤 단계는 강한 정합성이 필요하고, 어떤 단계는 최종 일관성으로 충분한지"를 구분하고, 실패 시 되돌리는 보상 로직과 운영 알림 체계까지 설계에 포함하는 것이 시니어의 신호입니다. "이렇게 하면 완벽합니다"가 아니라 "이 설계에서도 결제 취소가 실패하는 등의 엣지 케이스는 남아있고, 그건 알림과 수동 개입으로 보완합니다"라고 한계까지 솔직히 말하는 것이 더 신뢰를 얻습니다.
