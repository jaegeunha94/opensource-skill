1. 오늘 레슨 개요

동시성 문제는 코드 리뷰에서 눈에 보이지 않다가, 트래픽이 몰리는 순간에만 터지는 특징이 있습니다. 로컬 테스트에서는 요청이 하나씩 순서대로 들어오니 재고가 음수가 되거나 중복 결제가 발생하는 버그를 절대 재현할 수 없습니다. 그래서 10년차 이상 면접에서 동시성을 묻는 이유는 "코드가 정상 동작하는가"가 아니라 "동시에 여러 요청이 같은 자원을 건드릴 때도 정상 동작하는가"를 설계 단계에서 예측할 수 있는가를 확인하기 위해서입니다.

면접관이 실제로 보는 신호
  • "락을 걸면 된다"는 선언 ↔ 어떤 락(비관적/낙관적, DB 락/분산 락)을 왜 골랐는지 트래픽 패턴 기준으로 설명
  • 데드락을 "운이 나쁘면 생기는 것"으로 이해 ↔ 락 획득 순서·타임아웃으로 원천 차단하는 설계
  • "멱등성 = 같은 요청 두 번 보내도 똑같은 결과"라는 정의만 암기 ↔ 네트워크 재시도가 실제로 왜 발생하고 멱등키를 어디에 저장해야 하는지 설명
  • 메시지 큐를 "비동기로 빠르게 처리하는 도구"로만 이해 ↔ at-least-once 전달이 유발하는 중복 처리를 컨슈머가 어떻게 흡수하는지 설계

이 레슨은 동시성 제어의 4개 핵심 축을 다루며, 각 영역마다 배경 개념 → 면접 질문 → 답변 구조 → senior follow-up → 좋은/약한 답변 비교 → 자기 점검 순서로 진행합니다.

2. 낙관적 vs 비관적 동시성 제어 — 충돌을 언제 발견할 것인가

배경 개념: 두 트랜잭션이 같은 행을 동시에 건드리면 무슨 일이 생기는가

계좌 잔액이 10,000원인 상태에서, 두 개의 출금 요청(각 7,000원)이 거의 동시에 도착했다고 가정합니다. 두 요청 모두 "현재 잔액 10,000원 → 충분하다"고 읽은 뒤 각자 차감을 시도하면, 두 요청 모두 성공 처리되어 잔액이 -4,000원이 되는 Lost Update(갱신 유실)가 발생할 수 있습니다. 이 문제를 막는 방법은 크게 두 가지 철학으로 나뉩니다. 비관적 동시성 제어(Pessimistic Concurrency Control, PCC)는 "충돌이 자주 일어날 것"이라 가정하고 자원에 접근하는 순간 락을 걸어 다른 트랜잭션의 접근 자체를 차단합니다. 낙관적 동시성 제어(Optimistic Concurrency Control, OCC)는 "충돌은 드물 것"이라 가정하고 일단 각자 작업을 진행한 뒤, 커밋 시점에 "내가 읽은 이후 데이터가 바뀌었는가"만 검사해 바뀌었으면 실패시키고 재시도합니다.

면접 질문

"동시에 여러 요청이 같은 계좌 잔액을 차감하려고 합니다. 낙관적 락과 비관적 락 중 어떤 것을 선택하시겠습니까? 선택 기준은 무엇인가요?"

면접관이 보는 신호

"락을 쓰면 안전하다"는 사실만 아는지, 아니면 충돌 빈도와 트랜잭션 길이에 따라 처리량(throughput)과 안전성이 어떻게 trade-off되는지를 정량적으로 설명하는지를 봅니다.

OCC vs PCC 비교

항목 비관적 락 (PCC) 낙관적 락 (OCC)
동작 방식 SELECT ... FOR UPDATE 등으로 읽는 순간 락을 걸어 다른 트랜잭션의 읽기/쓰기를 대기시킴 버전 컬럼(version)이나 타임스탬프를 두고, 커밋 시 WHERE version = 읽은시점버전으로 갱신 시도. 영향받은 행이 0개면 충돌로 간주하고 재시도
충돌이 드문 경우 불필요하게 락을 잡아 대기 시간이 늘고 처리량이 낮아짐 재시도가 거의 발생하지 않아 락 대기 없이 높은 처리량 확보
충돌이 잦은 경우(핫스팟) 락 대기가 있어도 순서대로 처리되어 결과가 예측 가능 재시도가 반복적으로 발생해 오히려 처리량이 급격히 떨어지고 클라이언트 체감 지연이 커짐
구현 복잡도 DB가 락을 관리해줘서 애플리케이션 코드는 단순 애플리케이션이 재시도 로직(retry loop)을 직접 구현해야 함
데드락 위험 여러 락을 서로 다른 순서로 잡으면 데드락 발생 가능 락 자체가 없어 데드락이 원천적으로 발생하지 않음
적합 상황 같은 행에 대한 동시 쓰기 경쟁이 잦은 경우(인기 상품 재고, 선착순 이벤트) 읽기가 많고 쓰기 충돌이 드문 경우(사용자 프로필 수정, 게시글 편집)

결정 순서

1
충돌 빈도를 추정합니다. "이 데이터를 동시에 수정하려는 요청이 실제로 자주 겹치는가?"를 먼저 봅니다. 인기 상품 재고나 선착순 쿠폰처럼 특정 행에 요청이 몰리는 핫스팟이면 충돌 빈도가 매우 높습니다.
2
충돌 빈도가 낮으면 OCC를 기본으로 검토합니다. 락 대기 없이 처리량을 최대화할 수 있고, 재시도 비용이 거의 발생하지 않습니다.
3
충돌 빈도가 높으면 PCC를 검토하되, 락 보유 시간을 최소화합니다. 트랜잭션 안에서 외부 API 호출이나 무거운 연산을 하지 않고, 락을 잡는 즉시 필요한 갱신만 하고 커밋합니다.
4
OCC를 선택했다면 재시도 한계와 백오프 전략을 정합니다. 무한 재시도는 클라이언트를 무한 대기시키므로, 최대 재시도 횟수와 지수 백오프(exponential backoff)를 함께 설계합니다.
강한 답변

"일반 계좌 잔액 차감은 대부분 서로 다른 계좌를 대상으로 하므로 같은 행에 대한 충돌 빈도가 낮습니다. 그래서 accounts 테이블에 version 컬럼을 두고 낙관적 락(UPDATE accounts SET balance = ?, version = version + 1 WHERE id = ? AND version = ?)을 사용합니다. 영향받은 행이 0이면 충돌로 판단해 최대 3회, 50ms~200ms 지수 백오프로 재시도합니다. 다만 이벤트성 인기 상품 하나의 재고 차감처럼 수천 명이 같은 행을 동시에 건드리는 핫스팟이라면, 낙관적 락은 재시도 폭주로 오히려 처리량이 급락하기 때문에 이 경우는 비관적 락이나 Redis 기반 분산 락으로 접근을 순서대로 직렬화하는 편을 선택합니다."

약한 답변

"동시성 문제가 생기지 않게 그냥 락을 걸겠습니다."

약한 답변이 약한 이유: "락"이라는 단어만 언급했을 뿐 어떤 종류의 락인지, 왜 그 락이 이 상황에 맞는지 근거가 없습니다. 충돌 빈도라는 핵심 변수를 전혀 고려하지 않았습니다.

Senior Follow-up

"낙관적 락을 도입했는데, 특정 시간대(할인 행사 시작 직후)에만 재시도가 폭증하며 응답 지연이 심해집니다. 원인과 대응 방안은 무엇인가요?"

모범 추론: 평소에는 충돌이 드물어 OCC가 잘 맞았지만, 할인 행사 시작 직후에는 특정 상품 행 하나에 트래픽이 집중되는 핫스팟이 생겨 OCC의 전제(충돌이 드물다)가 깨진 상황입니다. 1) 근본 대응: 이 시간대의 이 자원만 일시적으로 비관적 락 또는 분산 락 기반의 순차 처리로 전환하거나, 재고 차감 자체를 Redis의 원자적 연산(DECR)으로 옮겨 DB 경합을 아예 없앱니다. 2) 구조적 대응: 인기 상품의 재고 수량을 여러 개의 파티션(예: 재고 100개를 10개씩 10파티션으로 분할)으로 나눠 한 행에 몰리는 경합 자체를 분산시키는 샤딩 카운터 패턴을 검토합니다. 3) 재시도 정책 조정: 재시도 폭증이 서로를 다시 충돌시키는 thundering herd 현상을 막기 위해, 고정 간격이 아닌 지터(jitter)를 포함한 지수 백오프로 재시도 타이밍을 분산시킵니다. 핵심은 "낙관적 락이 틀렸다"가 아니라 트래픽 패턴이 바뀌면 전제도 재검증해야 한다는 것을 아는 것입니다.

3. 락 전략 — DB 락, 분산 락, 데드락 방지

배경 개념: 락은 "잠그는 범위"와 "잠그는 곳"이 다르면 완전히 다른 도구가 된다

단일 DB 트랜잭션 안에서의 락(SELECT ... FOR UPDATE)은 그 DB 커넥션이 살아있는 동안만 유효하고, 트랜잭션이 끝나면 자동으로 풀립니다. 하지만 여러 서버 인스턴스에 걸쳐 "이 작업은 한 번에 하나의 프로세스만 수행해야 한다"는 것을 보장하려면 DB 락만으로는 부족한 경우가 많습니다 — 예를 들어 배치 작업이 여러 서버에서 중복 실행되지 않게 막거나, 여러 마이크로서비스가 같은 외부 자원에 순차적으로만 접근하게 해야 할 때입니다. 이럴 때 쓰는 것이 분산 락(Distributed Lock)이며, Redis의 SET key value NX PX ttl이나 ZooKeeper의 임시 순차 노드(ephemeral sequential znode)로 구현합니다.

면접 질문

"결제 정산 배치가 여러 서버 인스턴스에서 동시에 실행되어 같은 정산 건을 두 번 처리했습니다. 어떻게 이 문제를 막으시겠습니까?"

면접관이 보는 신호

"락을 걸면 된다"가 아니라, 분산 락이 DB 락과 근본적으로 다른 실패 모드(락 보유자가 죽었을 때 락이 영원히 안 풀리는 문제)를 가진다는 것을 알고 있는지를 봅니다.

DB 락 vs 분산 락

항목 DB 락 (트랜잭션 내) 분산 락 (Redis/ZooKeeper)
보호 대상 DB 행/테이블 단위 데이터 정합성 여러 프로세스/서버에 걸친 "동시에 하나만 실행" 같은 애플리케이션 로직
락 해제 보장 트랜잭션 커밋/롤백 시 DB가 자동 해제 락 보유 프로세스가 크래시하면 수동으로 안 풀림 — 반드시 TTL(만료 시간)을 함께 설정해야 함
대표 구현 SELECT ... FOR UPDATE, SELECT ... FOR SHARE Redis SET key val NX PX ttl, ZooKeeper ephemeral znode, etcd lease
주요 리스크 트랜잭션을 오래 열어두면 락 보유 시간이 늘어나 다른 트랜잭션이 대기 TTL이 너무 짧으면 작업이 끝나기 전에 락이 풀려 다른 프로세스가 동시에 진입(safety 위반), 너무 길면 장애 시 복구가 느려짐

분산 락 설계에서 반드시 짚어야 할 것

1
TTL(만료 시간)을 반드시 설정합니다. 락을 잡은 프로세스가 크래시해도 TTL이 지나면 자동으로 풀려 다른 프로세스가 진입할 수 있게 합니다. TTL이 없는 락은 장애 시 시스템 전체를 영구히 멈추게 하는 SPOF가 됩니다.
2
락 소유자 식별값(고유 토큰)을 value로 저장합니다. 락 해제 시 "내가 잡은 락이 맞는지" 토큰을 비교하고 지웁니다. 이렇게 하지 않으면, 내 작업이 TTL보다 오래 걸려 락이 이미 다른 프로세스에게 넘어간 뒤 내가 무심코 그 락을 삭제해버리는 사고가 생깁니다.
3
TTL보다 작업이 오래 걸릴 가능성을 대비해 락 갱신(watchdog/lease renewal)을 고려합니다. 작업이 진행 중임을 주기적으로 알려 TTL을 연장하는 방식(Redisson의 watchdog 등)으로, 짧은 TTL의 안전함과 긴 작업의 완주를 동시에 만족시킵니다.
4
단일 Redis 노드 기반 락의 한계를 인지합니다. Redis가 마스터-레플리카 구조에서 마스터 장애 시 아직 레플리케이션되지 않은 락 정보가 유실되어, failover 직후 짧은 순간 두 프로세스가 동시에 락을 획득하는 것처럼 보일 수 있습니다. 엄격한 상호 배제가 꼭 필요하면 Redlock 알고리즘이나 ZooKeeper/etcd처럼 합의 기반 시스템을 검토합니다.

데드락 — 락 자체가 아니라 "락을 잡는 순서"의 문제

데드락은 트랜잭션 A가 자원 X를 잠그고 Y를 기다리는 동안, 트랜잭션 B가 자원 Y를 잠그고 X를 기다리는 상황에서 발생합니다. 전형적인 예는 계좌 이체입니다 — A 계좌에서 B 계좌로 이체하는 트랜잭션과 B 계좌에서 A 계좌로 이체하는 트랜잭션이 동시에 실행되면, 하나는 A를 먼저 잠그고 다른 하나는 B를 먼저 잠가 서로를 무한히 기다리게 됩니다.

강한 답변

"정산 배치를 실행하기 전에 Redis에 SET lock:settlement:{batchId} {workerToken} NX PX 30000으로 분산 락을 시도하고, 실패하면 다른 워커가 이미 처리 중이라고 보고 바로 종료합니다. 배치가 30초 안에 끝나지 않을 수도 있어서, 별도 스레드가 10초마다 TTL을 연장하는 watchdog을 두었습니다. 락 해제 시에는 value에 저장한 workerToken이 내 것과 같은지 Lua 스크립트로 원자적으로 확인 후 삭제해, TTL 만료로 넘어간 락을 실수로 삭제하지 않게 했습니다. 계좌 이체처럼 두 자원을 동시에 잠가야 하는 경우는 항상 계좌 ID의 사전순(lexicographic order)으로 락을 획득해, 순환 대기 자체가 발생하지 않도록 설계합니다."

약한 답변

"분산 락을 걸어서 한 서버만 실행하게 하면 됩니다."

약한 답변이 약한 이유: TTL, 소유자 검증, 워커 크래시 시나리오를 전혀 고려하지 않았습니다. "락을 건다"는 결론은 맞지만, 그 락이 안전하게 동작하는 조건을 설명하지 못하면 면접관은 실무 경험이 아니라 개념만 안다고 판단합니다.

Senior Follow-up

"분산 락을 잡은 워커가 GC pause나 네트워크 지연으로 30초 넘게 멈췄다가, TTL이 지나 락이 풀린 후 다시 살아나서 작업을 계속 진행했습니다. 이때 이미 다른 워커가 같은 작업을 시작했다면 어떤 문제가 생기고, 어떻게 막으시겠습니까?"

모범 추론: 이것은 분산 락의 근본적인 한계인 "멈췄다 살아난 프로세스(zombie process)" 문제입니다. TTL이 지나 락이 풀렸다는 것은 "이 프로세스가 죽었다"는 뜻이 아니라 "락 시스템 관점에서 시간이 다 됐다"는 뜻일 뿐이므로, 원래 워커가 실제로는 여전히 살아서 작업을 이어갈 수 있습니다. 1) 근본 원칙: 분산 락은 "동시에 두 프로세스가 실행되지 않는다"를 100% 보장하지 못합니다. 락은 "대부분의 경우 중복 실행을 줄여주는 최적화"로 취급하고, 작업 자체를 멱등하게 설계하는 것이 진짜 안전장치입니다(4절 참고). 2) 펜싱 토큰(fencing token): 락을 획득할 때마다 단조 증가하는 토큰(버전 번호)을 함께 받고, 실제 자원(DB, 파일 등)에 쓰기 작업을 할 때 "내 토큰이 현재 저장된 토큰보다 최신인지"를 확인한 뒤에만 쓰기를 허용합니다. 좀비 워커가 뒤늦게 쓰기를 시도해도 이미 더 최신 토큰을 가진 워커가 갱신했다면 거부됩니다. 3) 작업 자체의 멱등성 확보: 정산 배치라면 "이미 정산 완료된 건은 재실행해도 상태를 다시 바꾸지 않는다"는 멱등 체크를 배치 로직 안에 넣어, 락이 실패해도 최종 결과가 안전하게 유지되도록 이중 방어합니다. 핵심은 락만으로 100% 안전을 보장할 수 없다는 것을 인지하고, 그 아래에 멱등성이라는 두 번째 방어선을 두는 것입니다.

4. 멱등성 — 재시도해도 안전한 시스템 만들기

배경 개념: 네트워크는 "성공/실패"만 있는 게 아니라 "모른다"도 있다

클라이언트가 결제 요청을 보냈는데 서버는 결제를 정상 처리하고 응답을 보내는 도중 네트워크가 끊겼다고 가정합니다. 서버 입장에서는 결제가 성공했지만, 클라이언트는 응답을 받지 못했으니 "실패했을 수도 있다"고 판단해 재시도를 보냅니다. 멱등성(Idempotency)이 없으면 이 재시도가 이중 결제로 이어집니다. 멱등성은 "같은 요청을 여러 번 보내도 실제 부수 효과(side effect)는 한 번만 일어난다"는 성질이며, 재시도가 필연적으로 발생하는 분산 시스템에서는 선택이 아니라 필수 설계 요소입니다.

면접 질문

"결제 API 클라이언트가 타임아웃으로 응답을 못 받으면 자동으로 같은 요청을 재시도합니다. 이 재시도가 이중 결제로 이어지지 않게 하려면 어떻게 설계하시겠습니까?"

면접관이 보는 신호

"멱등키를 쓰면 된다"는 결론만 아는지, 아니면 멱등키를 어디에 저장하고, 동시에 같은 키로 두 요청이 들어왔을 때 무슨 일이 벌어지는지까지 설계할 수 있는지를 봅니다.

멱등키(Idempotency Key) 설계

1
클라이언트가 요청마다 고유한 멱등키(UUID)를 생성해 헤더(Idempotency-Key)로 전달합니다. 이 키는 "논리적으로 같은 요청"을 식별하는 용도이므로, 재시도 시에는 반드시 처음 생성한 키를 그대로 재사용해야 합니다(재시도할 때마다 새 키를 만들면 멱등성이 깨집니다).
2
서버는 요청을 처리하기 전에, 이 키로 이미 처리된 기록이 있는지 원자적으로 확인·기록합니다. DB에 idempotency_key를 유니크 제약(unique constraint)으로 둔 테이블에 먼저 삽입을 시도하고, 삽입이 성공해야만 실제 결제 로직을 진행합니다. 유니크 제약 위반이 발생하면 이미 처리 중이거나 처리된 요청이라는 뜻입니다.
3
처리 결과(응답)를 멱등키와 함께 저장해 두었다가, 같은 키로 재요청이 오면 실제 로직을 재실행하지 않고 저장된 응답을 그대로 반환합니다. 이렇게 하면 클라이언트 입장에서는 "몇 번을 재시도하든 항상 같은 결과"를 받게 됩니다.
4
같은 키로 "아직 처리 중인" 요청이 동시에 들어오는 경쟁 상태도 고려합니다. 유니크 제약을 이용한 삽입 자체가 원자적 락 역할을 하므로, 나중에 도착한 동시 요청은 삽입 실패로 대기하거나 "처리 중" 상태를 반환하도록 설계합니다.

HTTP 메서드별 멱등성 특성

메서드 자연적으로 멱등한가 실무에서 주의할 점
GET 예 — 조회는 여러 번 해도 부수 효과 없음 조회 로그를 남기는 부가 로직이 있다면 그 로직만큼은 멱등하지 않을 수 있음
PUT 예 — 전체 리소스를 같은 값으로 덮어쓰므로 여러 번 실행해도 결과는 동일 서버가 updatedAt 같은 필드를 자동 갱신하면 엄밀히는 응답이 매번 달라짐 — 핵심 상태 값 기준으로 판단
DELETE 예 — 이미 삭제된 리소스를 다시 삭제해도 최종 상태는 "없음"으로 동일 두 번째 삭제 요청에 404를 줄지 204를 줄지는 API 정책으로 정해야 함(멱등성 자체는 유지됨)
POST 아니오 — 기본적으로 "생성"은 호출할 때마다 새 리소스를 만듦 결제, 주문 생성처럼 부수 효과가 큰 POST는 반드시 애플리케이션 레벨의 멱등키를 별도로 구현해야 함
강한 답변

"클라이언트 SDK가 결제 요청 생성 시 UUID를 Idempotency-Key 헤더로 발급하고, 타임아웃 재시도 시에도 같은 키를 재사용하도록 강제합니다. 서버는 idempotency_key에 유니크 인덱스를 건 테이블에 먼저 INSERT를 시도하고, 성공한 요청만 실제 결제 로직을 실행합니다. 처리 완료 후 결과(성공/실패, 응답 바디)를 같은 행에 업데이트해두고, 이후 같은 키로 오는 요청은 결제 로직을 다시 태우지 않고 저장된 응답을 그대로 반환합니다. 동시에 같은 키로 두 요청이 도착하는 경우도 INSERT의 유니크 제약이 DB 레벨에서 원자적으로 하나만 통과시켜주므로 별도 락 없이 안전합니다."

약한 답변

"결제 전에 이미 결제했는지 확인하는 로직을 추가하면 됩니다."

약한 답변이 약한 이유: "확인 후 처리(check-then-act)" 패턴 자체가 두 요청이 동시에 "확인" 단계를 통과해버리는 경쟁 상태(TOCTOU, Time-Of-Check to Time-Of-Use)에 취약합니다. 확인과 기록이 하나의 원자적 연산으로 묶이지 않으면 멱등성이 보장되지 않습니다.

Senior Follow-up

"멱등키 저장 테이블에 유니크 제약을 걸었는데도, 매우 드물게 이중 결제가 발생했습니다. 로그를 보니 첫 번째 요청이 아직 결제 처리 중(결제사 API 응답 대기)인 상태에서 두 번째 재시도 요청이 왔고, 첫 번째 요청의 INSERT가 아직 커밋되지 않아 두 번째 요청도 INSERT에 성공해버렸습니다. 원인과 해결책은 무엇인가요?"

모범 추론: 이것은 멱등키 기록(INSERT)과 실제 결제 처리를 같은 트랜잭션 안에 넣고, 그 트랜잭션이 커밋되기 전까지는 "처리 시작됨" 상태가 다른 요청에게 보이지 않았기 때문에 생긴 문제입니다(트랜잭션 격리 수준에 따라 커밋 전 상태가 다른 요청에 보이지 않아 유니크 제약 검사를 통과해버릴 수 있는 경우, 혹은 INSERT를 별도의 짧은 트랜잭션으로 먼저 커밋하지 않고 전체 결제 트랜잭션과 묶어버린 설계 문제). 해결책: 1) 멱등키 INSERT를 결제 처리와 분리된, 즉시 커밋되는 짧은 트랜잭션으로 먼저 실행합니다. INSERT가 성공적으로 커밋된 것을 확인한 뒤에야 실제 결제 로직(결제사 API 호출)을 시작합니다. 2) 멱등키 레코드에 status 컬럼(PENDING/COMPLETED/FAILED)을 두어, 두 번째 요청이 PENDING 상태를 발견하면 "처리 중"이라는 응답(예: 409 Conflict 또는 대기 후 폴링 안내)을 즉시 반환하고 별도의 결제 시도를 하지 않게 합니다. 3) 결제사 API 자체도 멱등키를 지원한다면(대부분의 PG사가 지원), 우리 시스템의 멱등키를 결제사 API 호출 시에도 그대로 전달해 이중 방어선을 만듭니다. 핵심은 "멱등키를 기록하는 시점"과 "실제 부수 효과가 발생하는 시점" 사이의 시간 간격 동안에도 다른 요청이 안전하게 처리되도록 상태 머신을 명시적으로 설계하는 것입니다.

5. 비동기 처리와 이벤트 큐 — 순서, 중복, 처리 보장

배경 개념: 메시지 큐는 "한 번만, 순서대로" 전달해주지 않는다

많은 개발자가 메시지 큐를 도입하면 "메시지가 정확히 한 번(exactly-once), 보낸 순서대로" 처리될 것이라 기대하지만, 실제로 대부분의 메시지 큐(Kafka, SQS, RabbitMQ)는 기본적으로 at-least-once(최소 한 번) 전달을 보장합니다. 컨슈머가 메시지를 처리한 후 "처리 완료" 확인(ack)을 보내기 전에 크래시하면, 큐는 그 메시지가 처리됐는지 알 수 없어 다시 전달합니다. 즉 컨슈머 입장에서는 같은 메시지를 두 번 받는 상황이 일상적으로 발생하며, 이를 안전하게 처리하는 책임은 큐가 아니라 컨슈머 애플리케이션에 있습니다.

면접 질문

"주문 완료 이벤트를 Kafka로 발행하고, 컨슈머가 이를 받아 사용자에게 알림을 보냅니다. 그런데 같은 주문에 대해 알림이 두 번 발송되는 사례가 보고되었습니다. 원인은 무엇이고 어떻게 해결하시겠습니까?"

면접관이 보는 신호

"큐가 중복을 막아줄 것"이라는 오해 없이, at-least-once 전달을 전제로 컨슈머 쪽에서 중복 제거(dedup)를 설계하는지를 봅니다.

전달 보장 수준 비교

보장 수준 의미 실무에서의 현실
At-most-once 메시지가 한 번만 전달되거나 아예 전달 안 될 수 있음(유실 가능) ack를 먼저 보내고 처리하는 방식. 처리 중 크래시하면 메시지 유실 — 중요한 이벤트에는 부적합
At-least-once 메시지가 최소 한 번은 전달되지만 중복될 수 있음 처리를 먼저 하고 ack를 나중에 보내는 방식(대부분의 큐 기본값). 유실은 없지만 컨슈머가 중복을 직접 처리해야 함
Exactly-once 메시지가 정확히 한 번만 처리됨 큐 자체적으로 완벽히 보장하기 매우 어려움(Kafka의 트랜잭션 프로듀서+컨슈머 조합 등 일부 조건에서만 근사 가능). 일반적으로는 "at-least-once 전달 + 컨슈머의 멱등 처리"로 실질적인 exactly-once 효과를 흉내냄

컨슈머 측 중복 제거 설계

1
메시지에 고유 식별자(이벤트 ID)를 반드시 포함시킵니다. 프로듀서가 메시지를 생성할 때 UUID나 (주문ID + 이벤트타입) 조합의 고유 키를 발급합니다.
2
컨슈머는 처리 전에 이 이벤트 ID로 "이미 처리했는가"를 확인·기록하는 테이블(또는 Redis SETNX)을 둡니다. 멱등키 패턴과 동일한 원리 — 유니크 제약으로 원자적 삽입을 시도하고, 실패하면 이미 처리된 메시지로 간주해 스킵합니다.
3
"부수 효과 실행"과 "처리 완료 기록"을 하나의 트랜잭션으로 묶습니다. 예를 들어 알림 발송 자체는 외부 API 호출이라 트랜잭션에 묶을 수 없다면, "발송 시도 기록을 DB 트랜잭션으로 먼저 커밋 → 성공한 경우에만 실제 발송 API 호출"처럼 원자적으로 처리 가능한 부분과 그렇지 않은 부분을 분리해서 설계합니다.
4
순서가 중요한 이벤트는 파티셔닝 키를 명시적으로 설계합니다. Kafka에서 같은 파티션 내의 메시지는 순서가 보장되지만, 파티션이 다르면 순서가 보장되지 않습니다. "같은 주문의 이벤트는 항상 순서대로 처리되어야 한다"는 요구가 있다면 주문 ID를 파티션 키로 사용해 같은 주문의 이벤트가 항상 같은 파티션으로 가도록 합니다.
강한 답변

"Kafka 컨슈머가 크래시 후 재시작되면서, ack(offset commit) 되지 않은 메시지를 다시 읽어 알림을 중복 발송한 것이 원인입니다. Kafka는 at-least-once가 기본값이므로 이건 설정 실수가 아니라 예상된 동작입니다. 해결을 위해 각 이벤트에 이벤트 ID(주문ID + 'ORDER_COMPLETED')를 포함시키고, 컨슈머가 알림 발송 전 processed_events 테이블에 이벤트 ID로 INSERT를 먼저 시도합니다. 유니크 제약 위반이 나면 이미 처리된 이벤트이므로 스킵하고 즉시 ack만 커밋합니다. 또한 같은 주문의 여러 이벤트(주문생성→결제완료→배송시작)가 순서대로 처리되어야 해서, 주문 ID를 파티션 키로 지정해 같은 주문의 이벤트가 항상 같은 파티션, 같은 컨슈머 스레드에서 순서대로 처리되도록 했습니다."

약한 답변

"Kafka는 메시지를 한 번만 전달해주는 줄 알았는데 버그인 것 같습니다. 큐 설정을 다시 확인해보겠습니다."

약한 답변이 약한 이유: at-least-once가 메시지 큐의 정상적인 기본 동작이라는 것을 모르고 있으며, 문제의 책임 소재를 컨슈머의 멱등 처리 부재가 아니라 큐의 "버그"로 잘못 짚었습니다. 이런 이해로는 원인 분석 자체가 시작되지 않습니다.

Senior Follow-up

"중복 제거 테이블을 도입한 후에도, 트래픽이 급증하는 시간대에 이 테이블 자체가 병목이 되어 컨슈머 처리량이 떨어졌습니다. 어떻게 개선하시겠습니까?"

모범 추론: 모든 이벤트마다 DB에 INSERT 시도를 하는 것 자체가 DB에 부하를 주는 구조이므로, 이 병목을 완화하는 몇 가지 접근이 있습니다. 1) Redis 같은 인메모리 저장소로 중복 체크를 옮깁니다. SET eventId 1 NX EX ttl 형태로 처리하면 DB보다 훨씬 빠르고, 오래된 이벤트 ID는 TTL로 자동 정리되어 무한히 쌓이지 않습니다(단, 중복 판정에 필요한 최소 기간만큼만 TTL을 잡아야 하며, 그 기간이 지나 재전달되면 중복 방지가 안 될 수 있다는 trade-off를 인지해야 함). 2) 컨슈머 자체를 파티션 단위로 수평 확장하여 병목이 특정 컨슈머 인스턴스에 몰리지 않게 분산합니다. 3) 근본적으로 배치 처리로 전환하는 것도 고려합니다 — 이벤트를 하나씩 처리하지 않고 짧은 시간 윈도우로 모아 한 번에 중복 체크·처리하면 저장소 호출 횟수 자체를 줄일 수 있습니다. 4) 비즈니스적으로 "완벽한 중복 제거"가 필요한지 재검토합니다 — 알림 발송처럼 아주 드문 중복이 치명적이지 않은 도메인이라면, 엄격한 dedup 대신 사용자에게 보이는 알림 문구에 최근 발송 시각을 남겨 중복 인지 시 자연스럽게 무시되도록 하는 완화책도 실무적으로 쓰입니다. 반대로 결제처럼 중복이 치명적인 도메인은 성능을 희생하더라도 엄격한 dedup을 유지해야 합니다.

6. 종합 시나리오 — 선착순 쿠폰 발급 시스템 설계

면접 질문

"쿠폰 1,000장을 선착순으로 발급하는 이벤트를 진행합니다. 예상 동시 접속자는 10만 명입니다. 재고를 초과해서 발급되지 않으면서, 특정 사용자가 중복 발급받지 않고, 응답 지연도 최소화하려면 어떻게 설계하시겠습니까?"

이 질문은 오늘 배운 낙관적/비관적 동시성, 분산 락, 멱등성, 비동기 큐를 하나의 실전 설계로 종합하는 능력을 봅니다.

1
DB를 직접 두드리는 방식은 즉시 배제합니다. 10만 명이 동시에 같은 행(쿠폰 재고)을 두고 경쟁하는 극단적 핫스팟이므로, 낙관적 락은 재시도 폭증으로, 비관적 락은 대기열 폭증으로 DB 자체가 죽을 위험이 큽니다.
2
재고 차감을 Redis의 원자적 연산으로 옮깁니다. DECR coupon:stock은 원자적이므로 별도 락 없이도 동시 요청이 순서대로 처리됩니다. 결과값이 0 미만이면 재고 소진으로 즉시 거부하고, 이 판단 자체가 밀리초 단위로 끝나 DB에 부하를 주지 않습니다.
3
중복 발급 방지는 Redis SET user:{userId}:coupon:{eventId} 1 NX로 처리합니다. 같은 사용자가 두 번 요청해도 두 번째는 NX 조건에 걸려 실패하므로, 재고 차감 로직보다 먼저 이 체크를 통과해야만 재고 차감을 시도하도록 순서를 정합니다.
4
Redis에서 재고 확보에 성공한 요청만 실제 쿠폰 지급(DB insert, 사용자 알림)을 비동기 큐에 발행합니다. 응답은 "재고 확보 성공/실패"만 즉시 반환해 사용자 체감 지연을 최소화하고, 실제 DB 반영과 알림 발송은 큐 컨슈머가 뒤에서 처리합니다.
5
큐 컨슈머는 멱등성을 전제로 설계합니다. 이벤트에 (userId, eventId)를 고유 키로 포함시켜, 컨슈머가 크래시 후 재시도로 같은 이벤트를 두 번 받아도 DB에는 한 번만 반영되도록 유니크 제약 기반 INSERT로 처리합니다.
6
Redis와 DB 사이의 정합성 어긋남 가능성을 최종 배치로 검증합니다. Redis 재고 차감은 성공했는데 이후 큐 처리나 DB 반영이 실패해 상태가 어긋나는 극히 드문 경우를 대비해, 이벤트 종료 후 Redis 발급 로그와 DB 지급 내역을 대사(reconciliation)하는 배치를 별도로 둡니다.
시니어가 반드시 언급해야 하는 것: "DB 락으로 순서를 지키겠다"가 아니라 핫스팟 자체를 DB 밖(Redis의 원자적 연산)으로 옮겨 경합을 회피하는 설계 감각, 그리고 "동기 처리로 정합성을 100% 보장"하는 대신 즉시 응답은 가볍게, 실제 반영은 비동기+멱등으로 나중에 처리하는 성능/정합성 trade-off 판단이 시니어 신호입니다. 또한 이 설계 어디에도 "100% 완벽하다"는 주장이 없고, 마지막에 대사 배치로 어긋남 가능성을 인정하고 검증 장치를 둔다는 것이 실무 경험이 있는 답변의 특징입니다.
