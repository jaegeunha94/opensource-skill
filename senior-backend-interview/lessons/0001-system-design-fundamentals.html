DAY 1

시스템 설계 기초 — URL Shortener 전체 설계

시니어 백엔드 인터뷰의 출발점. 요구사항 분석부터 데이터 모델, 확장성, 가용성까지 구조적으로 풀어낸다.

시스템 설계 확장성 데이터 모델링 캐싱 가용성 해시 전략

왜 URL Shortener인가

URL Shortener는 시스템 설계 인터뷰의 "Hello World"다. 그러나 시니어 레벨에서는 단순 구현이 아닌 운영 규모에서의 판단력을 본다.

  • 하루 1억 건 단축 요청 처리 시 DB와 캐시를 어떻게 구성할 것인가
  • 해시 충돌을 피하면서도 짧은 URL을 유지하는 전략
  • 읽기 편향 트래픽(write:read ≈ 1:100)에서의 최적화 방향
  • 다운타임 없이 스키마를 진화시키는 방법
  • 악용 방지와 TTL 관리 정책

핵심 면접 질문

Primary Question

"URL Shortener 서비스를 설계해주세요. TinyURL이나 bit.ly 같은 서비스입니다."

Scoped Question (더 자주 나오는 방식)

"우리 회사가 사내 링크 단축 서비스를 만들려 합니다. 하루 약 1억 건 단축 요청, 10억 건 리다이렉트 요청을 처리해야 합니다. 어떻게 설계하시겠어요?"

시니어 답변 구조 — 4단계 프레임워크

인터뷰어는 바로 DB 스키마를 꺼내는 사람을 싫어한다. 먼저 요구사항을 명확히 하고 규모를 추정한 후 설계한다.

1

요구사항 명확화 (3~5분)

설계를 시작하기 전에 다음을 확인한다:

  • 기능: 단축 URL 생성, 리다이렉트, 커스텀 alias, TTL 설정, 통계(클릭 수)?
  • 규모: DAU, 쓰기 QPS, 읽기 QPS, 데이터 보존 기간
  • 비기능: 가용성 목표(99.9%? 99.99%?), 리다이렉트 지연 SLA, 일관성 수준
  • 제약: 예산, 팀 규모, 기존 인프라

예시 가정 (인터뷰에서 직접 발화):

  • 쓰기: 1억/일 → 약 1,200 QPS
  • 읽기: 10억/일 → 약 12,000 QPS (읽기 편향)
  • URL 길이: 최대 2KB, short key: 7자
  • 데이터 보존: 5년 → 약 1,800억 레코드 (5년 × 365일 × 1억)
  • 가용성: 99.9% (다운타임 약 8.7시간/년)
2

고수준 설계 (아키텍처 스케치)

Client │ ▼ [Load Balancer] ←── 글로벌 anycast, L4/L7 │ ├──► [Short URL Service] (Write Path) │ │ │ ├──► [ID Generator] ← Snowflake / DB auto-increment │ ├──► [Hash / Encode] ← Base62 encoding │ └──► [Primary DB] ← 쓰기 전용 (PostgreSQL / Cassandra) │ └──► [Redirect Service] (Read Path) ← 99% 트래픽 │ ├──► [Cache Layer] ← Redis, TTL=24h, LRU eviction │ │ │ └── MISS ──► [Read Replica DB] │ └──► [Analytics Queue] ← Kafka → ClickHouse (비동기)

핵심 포인트: 쓰기와 읽기 경로를 분리한다. 읽기가 99%이므로 캐시 히트율이 비용과 지연을 결정한다.

3

핵심 설계 결정 — Deep Dive

해시 전략 선택

방식 장점 단점 시니어 선택 시 고려사항
MD5/SHA256 + 앞 7자 구현 단순 충돌 가능, URL 입력이 같으면 결과도 같음 충돌 시 재시도 필요 → 쓰기 지연 증가
전역 카운터 + Base62 충돌 없음, 예측 가능 카운터 단일 장애점, 순차 예측 가능(보안) 카운터를 분산(Snowflake)하거나 DB sequence 활용
Snowflake ID + Base62 분산, 충돌 없음, 시간 순서 구현 복잡도, 8~10자 길어짐 실무 추천. Twitter/Instagram 방식. 7자를 위해 epoch 조정 가능

시니어 답변 포인트: "저는 Snowflake ID 기반을 선택합니다. 분산 환경에서 충돌 없이 ID를 생성하고, Base62로 인코딩하면 약 8자가 됩니다. 7자로 줄이고 싶다면 epoch를 서비스 시작일로 설정해 비트를 절약합니다."

DB 선택

옵션 적합한 이유 부적합한 이유
PostgreSQL + Read Replica ACID, 팀 숙련도, 운영 편의 5년치 1,800억 레코드 → 샤딩 필요
Cassandra 쓰기 처리량 우수, 수평 확장 용이 집계 쿼리 어려움, 운영 복잡도
DynamoDB 관리형, 자동 확장 벤더 종속, 비용 예측 어려움

시니어 답변 포인트: "초기 트래픽이라면 PostgreSQL + 읽기 전용 복제본으로 충분합니다. 18개월 후 데이터가 수천억 건을 넘을 시점에 short_key의 첫 2자를 기준으로 샤딩을 준비합니다. 지금 Cassandra로 시작하는 건 과도한 복잡도입니다."

캐시 전략

# Redis 설정 예시
# 핫 URL은 24시간 TTL, LRU eviction
SET url:{short_key} {original_url} EX 86400

# 통계는 별도 키 (원본 URL과 분리)
INCR stats:{short_key}:clicks
  • 캐시 크기: 상위 20% URL이 80% 트래픽 담당 (파레토). 일일 쓰기 1억 × 0.2 × 2KB ≈ 40GB/일 → Redis 수백 GB면 수일치 캐시 가능
  • Cache-Aside 패턴: 조회 시 캐시 미스 → DB 읽기 → 캐시 저장
  • 캐시 만료 시 DB 부하 spike 방지: Mutex Lock / Probabilistic Early Expiration
4

엣지 케이스 & 운영 고려사항

  • 악용 방지: Rate Limiting (IP당 분 100건), 악성 URL 블랙리스트 (Safe Browsing API 연동)
  • TTL 만료 처리: Soft Delete → 배치로 실제 삭제 (즉시 삭제 시 DB 부하)
  • 커스텀 alias 충돌: 낙관적 락 + Unique 제약조건 → 충돌 시 사용자에게 다른 alias 제안
  • 리다이렉트 방식: 301(영구) vs 302(임시) → 통계가 필요하면 302, 서버 부하 줄이려면 301
  • 지역별 지연: 글로벌 서비스라면 CDN 엣지에서 리다이렉트 처리 (CloudFront Lambda@Edge)

Senior-Level Follow-up 질문 & 예상 답변

Follow-up 1

"캐시 히트율이 갑자기 60%에서 10%로 떨어졌습니다. 어떻게 진단하겠어요?"

시니어 답변 방향:

  1. 먼저 메트릭 확인: Redis 히트율, 미스율 추이 / 언제부터 시작?
  2. 원인 가설: (a) 캐시 eviction 급증 (메모리 부족?), (b) TTL 일괄 만료 (thundering herd), (c) 트래픽 패턴 변화 (새 컨텐츠 폭증), (d) Redis 클러스터 노드 장애로 재샤딩
  3. 확인 방법: Redis INFO stats → evicted_keys, expired_keys 확인 / 분산 트레이싱으로 DB 부하 증가 여부 확인
  4. 즉시 조치: 메모리 부족이라면 eviction policy 조정 or 노드 추가 / thundering herd면 Mutex Lock 적용
  5. 근본 해결: 핫키 패턴 분석 후 캐시 프리워밍 전략 수립
Follow-up 2

"단축 URL을 삭제할 때 이미 캐시된 데이터를 어떻게 처리합니까?"

시니어 답변 방향:

  • DB에서 삭제 시 캐시 즉시 무효화 (Cache Invalidation) → DEL url:{short_key}
  • 문제: 삭제 성공 → 캐시 무효화 실패 시 불일치. 해결: Write-Through 패턴 또는 Saga 패턴으로 두 작업을 원자적으로 처리
  • 실용적 접근: "Deleted At" 필드로 Soft Delete → 리다이렉트 서비스가 DB 조회 시 삭제 여부 확인 → 410 Gone 반환
  • 캐시 TTL이 짧다면(예: 1분) 불일치 윈도우가 작아 허용 가능
Follow-up 3

"5년이 지나 데이터가 1,800억 건이 됐습니다. 성능이 떨어지고 있어요. 어떻게 대응합니까?"

시니어 답변 방향:

  1. 파티셔닝 (샤딩): short_key 첫 2자로 샤드 키 → 균등 분산. 핫스팟 주의
  2. 아카이빙: 1년 이상 미사용 URL → Cold Storage (S3 + Athena). 조회 시 warm-up latency 허용
  3. 인덱스 최적화: short_key는 PK, original_url은 별도 인덱스 필요 여부 재검토 (역방향 조회 있다면)
  4. 읽기 전용 복제본 추가: 샤드별 1-2개 read replica
  5. 선제 대응: "이런 문제가 발생하기 전에 데이터 규모 알람을 설정하고 6개월 전에 마이그레이션을 계획했을 것입니다"
Follow-up 4

"이 설계에서 가장 위험한 단일 장애점(SPOF)은 어디입니까?"

시니어 답변 방향:

  • ID Generator: Snowflake 노드가 죽으면 URL 생성 불가 → 다중 노드 구성, 장애 감지 후 다른 노드로 failover
  • Primary DB: 쓰기 경로 SPOF → Multi-AZ 배포, 자동 장애 조치 (RDS Multi-AZ or Patroni)
  • Redis: 캐시 장애 시 DB로 폴백 → Redis Sentinel/Cluster 구성. 단, DB가 12,000 QPS 처리 가능한지 검증 필요
  • 우선순위: "읽기 트래픽 99%이므로 Redis 장애가 비즈니스 영향이 가장 큼. 이를 먼저 HA 구성합니다"
Follow-up 5 (비기술적)

"팀원이 'NoSQL로 처음부터 설계하자'고 강하게 주장합니다. 어떻게 하겠어요?"

시니어 답변 방향:

  • 먼저 팀원의 근거를 경청: "어떤 구체적인 문제를 해결하려는 건가요? 현재 PostgreSQL로 해결 못 하는 부분이 무엇인가요?"
  • 데이터로 판단: 현재 규모에서 PostgreSQL이 실제로 병목인지 확인. 미래 우려라면 PoC로 검증
  • 비용 명시: NoSQL 전환 시 팀 러닝 커브, 운영 도구 부재, 쿼리 유연성 손실을 수치로 제시
  • 단계적 접근 제안: "지금은 PostgreSQL로 시작하고, 샤딩이 필요한 시점에 NoSQL 마이그레이션을 검토하는 계획을 세우면 어떨까요?"
  • 합의 도출: 결정 기준과 마이그레이션 트리거를 문서화하여 팀 동의 확보

좋은 답변 vs 약한 답변

강한 답변 (시니어 수준)
  • 요구사항을 먼저 질문하고 수치로 규모를 추정한다
  • "읽기 편향이므로 캐시 히트율이 핵심 지표"라고 설계 근거를 명시한다
  • DB 선택 시 "초기엔 PostgreSQL, X 규모가 되면 샤딩"처럼 단계적 접근을 제시한다
  • 301 vs 302의 trade-off(서버 부하 vs 통계 정확성)를 스스로 언급한다
  • 캐시 일관성 문제를 먼저 인식하고 해결책을 제안한다
  • 악용 방지, TTL 정책 등 운영 관점을 포함한다
  • 자신의 경험을 연결: "이전 직장에서 비슷한 문제를 겪었을 때 ..."
약한 답변 (주의)
  • 질문을 받자마자 바로 DB 스키마를 그린다
  • 규모 추정 없이 "그냥 Redis 쓰면 됩니다"라고 답한다
  • MD5 해시를 제안하고 충돌 처리를 고려하지 않는다
  • 읽기/쓰기 비율을 고려하지 않고 단일 DB로 설계한다
  • 장애 상황을 물어보면 "그냥 재시작하면 됩니다"라고 답한다
  • 기술 선택 이유를 물으면 "많이 쓰니까요" 라고 답한다
  • 인터뷰어의 힌트를 무시하고 기존 설계를 고집한다

핵심 차이

시니어는 "무엇을"이 아닌 "왜"와 "언제"를 설명한다. 기술 선택에는 항상 trade-off가 있고, 시니어는 그 trade-off를 비즈니스 맥락(비용, 팀 역량, 운영 복잡도)에서 정당화한다.
