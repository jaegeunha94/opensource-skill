1. 오늘 레슨 개요

배포(Deployment)를 주니어와 시니어가 가장 다르게 이해하는 지점은 "배포 명령어를 실행하는 방법"이 아니라 "배포가 실패했을 때 사용자 피해를 얼마나 좁은 범위로, 얼마나 빨리 되돌릴 수 있는가"입니다. 배포는 본질적으로 확률적 실험입니다 — 스테이징에서 통과한 코드도 프로덕션 트래픽 패턴, 실제 데이터 분포, 다른 서비스와의 상호작용에서는 다르게 동작할 수 있습니다. 10년차 이상 면접에서 배포/DevOps를 묻는 이유는, "배포는 위험할 수 있다"는 전제를 받아들이고 그 위험을 설계로 통제하는 사고방식을 가졌는지 확인하기 위해서입니다.

면접관이 실제로 보는 신호
  • "CI에서 테스트를 돌리고 배포합니다"라는 선언 ↔ 어떤 단계에서 무엇을 검증하고, 실패 시 어디서 멈추는지까지 구조화한 파이프라인 설계
  • "카나리 배포를 씁니다"라는 용어 나열 ↔ 왜 이 상황에는 블루-그린이 아니라 카나리가 맞는지, 트래픽 비율을 어떻게 늘려가는지 근거 있는 선택
  • "문제 생기면 이전 버전으로 롤백합니다" ↔ DB 스키마 변경이 껴 있을 때도 롤백이 안전한지까지 고려한 설계
  • 배포를 "인프라 팀의 일"로 여기는 태도 ↔ 배포 실행 버튼을 누른 사람이 그 배포의 신뢰성을 끝까지 책임지는 소유권 문화

이 레슨은 CI/CD 파이프라인 설계 → 배포 전략(블루-그린/롤링/카나리) → 피처 플래그 → 롤백과 DB 호환성 순서로 진행하며, 각 영역마다 배경 개념 → 면접 질문 → 답변 구조 → senior follow-up → 좋은/약한 답변 비교 → 자기 점검을 다룹니다.

2. CI/CD 파이프라인 — 배포를 "사건"에서 "일상"으로

배경 개념: 왜 "배포"가 무서운 일이 되면 안 되는가

배포가 한 달에 한 번, 새벽 시간에, 여러 팀의 변경 사항을 한꺼번에 묶어서 이뤄진다면 그 자체가 위험 요소입니다. 변경 사항이 많이 쌓일수록 무엇이 문제를 일으켰는지 원인을 좁히기 어렵고, 실패 시 되돌릴 범위도 커집니다. 지속적 통합(Continuous Integration, CI)은 개발자가 코드를 메인 브랜치에 자주(하루에도 여러 번) 병합하고, 그때마다 자동으로 빌드·테스트해 문제를 병합 직후 아주 작은 변경 단위에서 발견하는 관행입니다. 지속적 배포/전달(Continuous Deployment/Delivery, CD)은 그렇게 검증된 코드를 사람이 수동으로 서버에 올리는 대신 자동화된 파이프라인이 프로덕션까지 내보내는 것입니다(Delivery는 마지막 승인 버튼만 사람이 누르고, Deployment는 그마저 자동입니다). 핵심은 배포를 "크고 위험한 이벤트"에서 "작고 자주 일어나는, 실패해도 영향이 작은 일상적 활동"으로 바꾸는 것입니다.

면접 질문

"현재 팀은 2주에 한 번, 여러 기능을 모아서 배포하고 있고, 배포 때마다 몇 시간씩 수동 QA를 거칩니다. 이 배포 프로세스를 개선해달라는 요청을 받았습니다. 어떻게 접근하시겠습니까?"

면접관이 보는 신호

"배포 자동화 스크립트를 짜겠다"는 도구 중심 답변에서 멈추는지, 아니면 파이프라인의 각 단계가 무엇을 검증하고 왜 그 순서인지를 구조적으로 설계하는지를 봅니다.

CI/CD 파이프라인의 단계별 검증 목적

단계 검증 대상 실무 판단 기준
정적 분석 / Lint 코드 스타일, 명백한 버그 패턴 가장 빠르고 저렴하므로 맨 앞에 배치 — 여기서 걸러지면 이후 단계 자원을 아낌
단위 테스트 함수/모듈 단위 로직 정확성 빠르게 실행되어야 함(수 분 이내). 느려지면 개발자가 스킵하려는 유혹이 생김
통합/계약 테스트 서비스 간 인터페이스, DB 연동 단위 테스트보다 느리지만 "서비스 A가 B의 API를 깨뜨렸는가"를 잡아냄(Day 5 계약 테스트 연결)
보안 스캔(SAST/의존성 스캔) 알려진 취약점, 시크릿 하드코딩 Day 8에서 다룬 보안 취약점을 배포 전에 자동으로 차단하는 게이트
스테이징 배포 + E2E 테스트 실제와 유사한 환경에서 전체 흐름 동작 프로덕션과 최대한 유사한 데이터/설정을 써야 신뢰도가 있음
프로덕션 점진 배포 실제 트래픽에서의 동작 3장에서 다룰 카나리/블루-그린으로 위험을 통제하며 진행

결정 순서

1
배포 단위를 작게 쪼갤 수 있는 구조로 바꿉니다. 여러 기능을 모아 2주에 한 번 배포하는 대신, 기능 개발이 끝나는 즉시 병합·배포할 수 있도록 트렁크 기반 개발(trunk-based development)과 짧은 수명의 브랜치를 도입합니다. 아직 사용자에게 노출하면 안 되는 미완성 기능은 배포와 릴리스를 분리하는 피처 플래그(4장)로 감춥니다.
2
수동 QA가 반복적으로 확인하는 항목을 자동화된 테스트로 전환합니다. 매번 사람이 같은 시나리오를 클릭해서 확인한다면 그것은 자동화 가능한 회귀 테스트 후보입니다. 사람은 탐색적 테스트(자동화가 어려운 새로운 시나리오 발굴)에 집중하도록 역할을 재배치합니다.
3
파이프라인 단계를 빠르고 저렴한 검증부터 느리고 비싼 검증 순서로 배치합니다. Lint → 단위 테스트 → 통합 테스트 → 보안 스캔 → 스테이징 E2E 순으로 배치해, 문제가 있는 커밋을 가장 빠른 단계에서 조기에 실패(fail fast)시켜 개발자 피드백 시간을 최소화합니다.
4
프로덕션 배포 자체를 3장에서 다룰 점진적 전략(카나리 등)으로 자동화하고, 이상 감지 시 자동 롤백을 연결합니다. 이렇게 하면 "배포"라는 행위 자체가 사람의 수작업이 아니라 검증된 파이프라인이 통과시킨 결과물을 안전하게 내보내는 절차가 됩니다.
강한 답변

"먼저 배포를 2주 단위의 큰 묶음에서 병합될 때마다 자동으로 흘러가는 파이프라인으로 바꿉니다. 트렁크 기반 개발로 전환해 짧은 수명의 브랜치만 쓰고, 아직 완성되지 않은 기능은 피처 플래그로 꺼둔 채 메인에 병합합니다. 파이프라인은 Lint → 단위 테스트 → 통합 테스트 → 의존성 취약점 스캔 → 스테이징 E2E 순으로 배치해 저렴한 검증에서 먼저 실패하게 합니다. 반복되는 수동 QA 체크리스트 중 결정론적으로 검증 가능한 항목은 회귀 테스트로 자동화하고, 사람은 새로운 엣지 케이스를 탐색하는 데 집중하도록 역할을 바꿉니다. 마지막으로 프로덕션 배포는 카나리 방식으로 자동화해 이상 지표 감지 시 자동 롤백되도록 연결해, 배포 빈도는 늘리면서 실패 시 노출 범위는 오히려 줄어들게 만듭니다."

약한 답변

"Jenkins나 GitHub Actions로 빌드와 배포를 자동화하는 스크립트를 작성하겠습니다."

약한 답변이 약한 이유: 어떤 도구를 쓸지는 부수적인 문제입니다. 진짜 질문은 "왜 2주에 한 번, 큰 묶음으로 배포하는 것이 위험한가"와 "그 위험을 줄이려면 배포 단위, 테스트 자동화 범위, 검증 순서를 어떻게 재설계해야 하는가"입니다. 도구 이름만 나열하면 문제의 근본 원인(배포 배치 크기와 수동 검증 의존)을 건드리지 못합니다.

Senior Follow-up

"CI 파이프라인의 통합 테스트 단계가 평균 25분이 걸려서, 개발자들이 병합 전 로컬에서 기다리다 지쳐 리뷰 승인만 받고 병합 후 CI 결과를 확인하지 않는 습관이 생겼습니다. 이 상황을 어떻게 개선하시겠습니까?"

모범 추론: 이 상황은 파이프라인 속도가 개발자 행동을 왜곡시켜, CI가 존재하는 이유(문제를 조기에 발견)를 무력화시키고 있다는 신호입니다. 1) 먼저 25분이 걸리는 원인을 분해합니다 — 테스트가 실제로 25분이 필요한 양인지, 아니면 병렬화되지 않고 순차 실행되고 있는지, 매번 불필요하게 전체 스위트를 재실행하고 있는지 확인합니다. 2) 변경된 파일과 연관된 테스트만 우선 실행하는 선택적 테스트 실행(test impact analysis)이나, 독립적인 테스트를 여러 워커에 병렬 분산하는 것으로 상당 부분을 줄일 수 있는 경우가 많습니다. 3) 근본적으로는 "병합 전에 반드시 통과해야 하는 필수 게이트"와 "병합 후 비동기로 확인해도 되는 검증"을 구분합니다. 빠른 단위 테스트와 핵심 통합 테스트는 병합을 막는 필수 게이트로 유지하되, 느리고 덜 치명적인 일부 검증은 병합 후 비동기로 돌리고 실패 시 알림을 보내는 방식으로 조정할 수 있습니다. 4) 다만 이 조정이 "느린 테스트는 아무도 안 본다"는 문화로 굳어지지 않도록, 병합 후 실패 알림에 대한 소유권(누가 확인하고 되돌릴 책임을 지는가)을 명확히 합니다. 핵심은 파이프라인 속도 문제를 "더 기다리라"고 요구하는 대신, 테스트 실행 자체의 효율화와 게이트의 재설계로 풀어야 한다는 것이고, 그러지 않으면 개발자들은 항상 안전장치를 우회하는 방향으로 적응한다는 점입니다.

3. 배포 전략 — 블루-그린 / 롤링 / 카나리

배경 개념: "한 번에 전체 교체"는 왜 위험한가

가장 단순한 배포 방식은 기존 서버를 멈추고 새 버전으로 한 번에 전체 교체(in-place)하는 것입니다. 문제는 새 버전에 버그가 있으면 100%의 사용자가 즉시 영향을 받고, 되돌리려면 다시 이전 버전을 배포하는 시간만큼 장애가 지속됩니다. 이를 완화하는 세 가지 대표 전략이 블루-그린(Blue-Green), 롤링(Rolling), 카나리(Canary)입니다. 셋 다 "한 번에 전체를 바꾸지 않는다"는 공통 철학을 갖지만, 무엇을 최적화하는지(전환 속도 vs 자원 비용 vs 위험 노출 범위)가 다릅니다.

면접 질문

"핵심 결제 서비스를 배포하는데 블루-그린, 롤링, 카나리 중 하나를 선택해야 합니다. 어떤 기준으로 선택하고, 왜 그 선택이 결제 서비스에 적합한지 설명해주세요."

면접관이 보는 신호

세 방식의 정의를 나열하는 데서 멈추는지, 아니면 서비스의 위험 민감도, 상태 유무, 롤백 속도 요구사항에 따라 근거 있는 선택을 하는지를 봅니다.

세 전략 비교

전략 동작 방식 장점 단점
블루-그린 기존 환경(Blue)과 동일한 새 환경(Green)을 통째로 준비해 트래픽을 한 번에(또는 매우 빠르게) 전환 롤백이 즉시(트래픽을 다시 Blue로 돌리기만 하면 됨), 새 버전이 완전히 검증된 후 전환 두 배의 인프라 자원이 순간적으로 필요, 전환 순간 여전히 100%가 한 번에 바뀜(트래픽 전환 자체의 리스크는 남음)
롤링 서버 인스턴스를 몇 대씩 순차적으로 새 버전으로 교체 추가 인프라 비용이 적음(기존 자원 재사용), 점진적 배포 중 신-구 버전이 동시에 떠 있어 API 하위 호환성이 반드시 보장되어야 함, 롤백도 다시 순차적으로 되돌려야 해 시간이 걸림
카나리 새 버전을 아주 적은 비율(예: 1%)의 실제 트래픽에만 노출하고, 지표를 보며 점진적으로 비율을 늘림 실제 프로덕션 트래픽으로 검증하되 위험 노출을 최소화, 이상 감지 시 극소수 사용자만 영향받고 즉시 중단 가능 트래픽 분할·지표 비교를 위한 추가 인프라(트래픽 라우팅, 자동 비교 분석)가 필요, 전체 전환까지 시간이 오래 걸릴 수 있음

결정 순서

1
서비스의 위험 민감도와 "새 버전이 실제 트래픽에서 검증되지 않은 채 100%로 갈 때의 피해 규모"를 먼저 평가합니다. 결제처럼 실패 시 직접적인 금전 피해와 신뢰 손상이 큰 서비스는, 소수의 사용자로 실제 검증을 거치는 카나리를 기본으로 고려합니다.
2
API 하위 호환성이 보장되는지 확인합니다. 카나리와 롤링 모두 신-구 버전이 동시에 살아있는 기간이 존재하므로, 이 기간 동안 클라이언트나 다른 서비스가 두 버전 중 어느 쪽에 요청이 가더라도 정상 동작해야 합니다(Day 5의 breaking/non-breaking 판단과 연결).
3
카나리의 트래픽 비율을 늘리는 기준을 사전에 정의합니다. "1% → 10% → 50% → 100%"처럼 단계별 비율과, 각 단계에서 몇 분 이상 관찰하고 어떤 지표(에러율, p99, 결제 성공률)가 신버전과 구버전 간 통계적으로 유의미한 차이가 없어야 다음 단계로 진행하는지를 명시합니다.
4
카나리 배포가 실질적으로 안전하려면 자동화된 지표 비교와 자동 롤백(5장)이 함께 있어야 합니다. 사람이 대시보드를 계속 지켜보며 수동으로 판단하는 방식은 야간 배포나 빠른 확산 상황에서 대응이 늦어질 수 있습니다.
강한 답변

"결제 서비스는 실패 시 금전적 피해와 신뢰 손상이 직접적이므로, 실제 트래픽으로 검증하되 노출 범위를 최소화할 수 있는 카나리를 기본으로 선택합니다. 먼저 API가 신-구 버전 동시 운영 기간 동안 하위 호환되는지 계약 테스트로 확인합니다. 배포 시 신버전에 1%의 실제 트래픽만 보내고, 결제 성공률과 p99 응답 시간을 구버전과 자동으로 비교해 5분간 통계적으로 유의미한 차이가 없으면 10% → 50% → 100%로 단계적으로 늘립니다. 각 단계에서 지표가 임계치를 벗어나면 자동으로 트래픽을 구버전으로 되돌리는 자동 롤백을 연결해, 사람이 밤에 대시보드를 계속 지켜보지 않아도 안전하게 동작하도록 설계합니다. 블루-그린은 결제처럼 소수 트래픽으로 미리 검증하는 단계 없이 전체 전환으로 넘어가는 방식이라, 이 서비스에는 카나리보다 위험 노출이 더 크다고 판단했습니다."

약한 답변

"카나리 배포가 요즘 많이 쓰이는 방식이니 카나리로 하겠습니다."

약한 답변이 약한 이유: "요즘 많이 쓰인다"는 유행을 따르는 이유일 뿐, 결제 서비스라는 구체적 맥락(위험 민감도, 하위 호환성 요구, 트래픽 비율 조정 기준, 자동 롤백 필요성)에 근거한 선택이 아닙니다. 인터뷰어는 왜 그 전략이 이 상황에 맞는지를 듣고 싶어합니다.

Senior Follow-up

"카나리 배포 중 신버전이 결제 실패율은 정상인데, 응답 시간이 구버전보다 눈에 띄게 느려졌습니다(에러는 없음). 트래픽 비율을 계속 늘려도 될까요, 멈춰야 할까요? 어떻게 판단하시겠습니까?"

모범 추론: 에러율만 보는 카나리 분석은 "실패하지는 않지만 느려지는 회귀"를 놓칠 수 있는 함정입니다. 응답 시간 저하는 당장 에러로 이어지지 않아도 사용자 경험 저하, 커넥션 풀 점유 증가로 인한 연쇄 지연(Day 7의 Bulkhead 논의와 연결), SLO 위반(Day 9) 위험으로 이어질 수 있습니다. 1) 먼저 이 지연이 일시적(콜드 스타트, JIT 워밍업 등)인지 지속적인 회귀인지 몇 분 더 관찰해 구분합니다. 2) 지속적이라면 비율을 늘리지 않고 현재 단계에서 멈추거나, 오히려 비율을 낮춰 영향받는 사용자 수를 줄인 채로 트레이싱(Day 9)을 통해 어느 구간에서 지연이 발생했는지 조사합니다. 3) 카나리 자동화 규칙 자체에 에러율뿐 아니라 응답 시간 percentile 회귀도 자동 중단 조건으로 포함되어 있어야 하며, 이번 상황을 계기로 그 규칙이 없었다면 추가합니다. 4) 결제처럼 민감한 서비스는 "에러가 없으니 괜찮다"는 판단 대신 보수적으로 먼저 멈추고 원인을 확인한 뒤 재개하는 태도가 맞습니다. 비율을 계속 늘리는 판단은 실패 시 되돌리기 어려운(더 많은 사용자가 이미 느린 경험을 한) 비가역적 방향이기 때문입니다. 핵심은 카나리 판단 기준을 "에러가 있는가"라는 단일 축이 아니라 사용자 체감과 직결된 여러 지표(에러율 + 지연 percentile)로 다축 설계해야 한다는 것과, 애매하면 확산이 아니라 보수적 정지를 택하는 판단력입니다.

4. 피처 플래그 — 배포와 릴리스의 분리

배경 개념: "배포"와 "릴리스"는 같은 것이 아니다

많은 팀이 "코드가 프로덕션 서버에 올라가는 것(배포)"과 "사용자가 그 기능을 실제로 쓸 수 있게 되는 것(릴리스)"을 같은 순간이라고 여깁니다. 이 둘을 하나로 묶으면, 아직 완성되지 않은 큰 기능은 배포 자체를 미룰 수밖에 없어 앞서 CI/CD에서 말한 "작고 자주 배포"가 어려워집니다. 피처 플래그(Feature Flag / Feature Toggle)는 코드에 조건문을 두어, 배포는 이미 끝났지만 플래그가 꺼져 있는 동안은 사용자에게 보이지 않는 상태를 만듭니다. 플래그를 켜는 것은 코드를 다시 배포하지 않고 설정값만 바꾸는 순간적인 작업이라, 릴리스 시점을 배포 시점과 독립적으로 통제할 수 있습니다.

면접 질문

"3개월짜리 대형 기능을 개발 중인데, 완성될 때까지 메인 브랜치에 병합하지 않고 별도 브랜치에서 작업하다 보니 병합 시점에 충돌이 많고 통합 테스트도 뒤늦게야 이뤄집니다. 이 문제를 어떻게 개선하시겠습니까?"

면접관이 보는 신호

"브랜치 전략을 바꾸겠다"는 표면적 답변에서 멈추는지, 아니면 배포와 릴리스를 분리하는 피처 플래그로 미완성 코드도 안전하게 자주 병합할 수 있게 만드는 근본 해법을 제시하는지를 봅니다.

피처 플래그의 대표 용도

용도 설명
릴리스 토글(Release Toggle) 미완성 기능을 배포는 하되 감춤. 트렁크 기반 개발을 가능하게 하는 핵심 장치
실험 토글(Experiment Toggle) A/B 테스트처럼 사용자군을 나눠 서로 다른 동작을 노출하고 지표 비교
운영 토글(Ops Toggle) 부하가 몰릴 때 특정 무거운 기능(추천, 부가 배너)을 즉시 끄는 킬 스위치(kill switch)
권한 토글(Permission Toggle) 특정 요금제/베타 사용자에게만 기능 노출

결정 순서

1
대형 기능을 작은 단위로 쪼개 각 단위를 플래그 뒤에 두고 자주 메인에 병합하도록 개발 방식을 바꿉니다. 예를 들어 새로운 결제 흐름 전체를 한 번에 완성해서 병합하는 대신, "결제 화면 UI 변경", "새 PG 연동", "기존 흐름과의 전환 로직"을 각각 독립된 플래그로 나눠 완성되는 대로 병합합니다.
2
플래그가 꺼진 상태에서도 기존 코드 경로가 정상 동작하는지 CI에서 검증합니다. 플래그를 도입한다고 테스트가 줄어드는 게 아니라, "켜짐"과 "꺼짐" 두 경로 모두 검증 대상이 된다는 점을 팀에 명확히 합니다.
3
내부 검증(사내 사용자) → 소수 실사용자 → 점진 확대 순으로 플래그를 릴리스 토글로 활용해 카나리처럼 통제된 노출을 합니다. 이 단계에서 배포(3장)의 트래픽 비율 조정과 피처 플래그의 사용자 비율 조정은 서로 다른 층위의 도구지만 같은 철학(점진적 노출)을 공유합니다.
4
기능이 완전히 안정화되고 100% 릴리스된 후에는 플래그와 옛 코드 경로를 제거하는 정리 작업을 반드시 계획합니다. 플래그를 치우지 않고 방치하면 코드베이스에 죽은 분기와 조합 폭발(플래그 여러 개의 켜짐/꺼짐 조합)이 쌓여 오히려 유지보수성을 해치는 "플래그 부채(flag debt)"가 됩니다.
강한 답변

"근본 원인은 '배포'와 '완성된 기능을 사용자에게 노출하는 것'이 같은 시점으로 묶여 있다는 점입니다. 대형 기능을 결제 화면 UI, 새 PG 연동, 전환 로직처럼 작은 단위로 쪼개고 각각을 피처 플래그 뒤에 두어, 완성되는 대로 즉시 메인 브랜치에 병합합니다. CI에서는 플래그가 꺼진 경로와 켜진 경로 모두를 테스트해, 플래그 도입이 테스트 커버리지를 줄이지 않도록 합니다. 릴리스 시점에는 사내 사용자 → 1% 실사용자 → 점진 확대 순으로 플래그를 열어 카나리처럼 통제된 검증을 거칩니다. 그리고 기능이 100% 안정화되면 플래그와 옛 코드 경로를 제거하는 정리 작업을 스프린트 계획에 명시적으로 포함해, 플래그가 계속 쌓여 조합 폭발이 생기는 것을 방지합니다."

약한 답변

"브랜치를 더 자주 리베이스하고 병합 전에 코드 리뷰를 꼼꼼히 하겠습니다."

약한 답변이 약한 이유: 리베이스와 리뷰는 병합 충돌의 증상을 완화할 뿐, "왜 3개월짜리 브랜치가 존재하게 됐는가"라는 근본 원인(배포와 릴리스가 묶여 있어 미완성 기능을 병합할 수 없다는 제약)을 해결하지 않습니다. 브랜치 수명이 길면 결국 같은 문제가 반복됩니다.

Senior Follow-up

"피처 플래그를 도입한 지 1년이 지났는데, 코드베이스에 200개가 넘는 플래그가 쌓여 있고 어떤 플래그가 아직 필요한지 아무도 모릅니다. 어떻게 관리 체계를 세우시겠습니까?"

모범 추론: 이것은 전형적인 플래그 부채 상황으로, 플래그가 "임시 장치"라는 본질을 잊고 영구적인 설정값처럼 방치된 결과입니다. 1) 모든 플래그에 소유자, 생성일, 목적(릴리스/실험/운영/권한), 예상 제거 시점을 메타데이터로 강제합니다. 이 메타데이터가 없는 플래그는 생성 자체를 막는 정책을 CI 게이트로 둘 수 있습니다. 2) 릴리스 토글은 100% 릴리스 후 일정 기간(예: 2주) 지나면 자동으로 "제거 대상" 알림이 담당 팀에 가도록 합니다. 실험 토글은 실험 종료와 동시에 제거를 스프린트 작업으로 티켓화합니다. 3) 이미 쌓인 200개는 일괄 감사(audit)를 통해 "항상 켜짐" 또는 "항상 꺼짐" 상태로 몇 개월째 고정된 플래그부터 우선 제거 대상으로 분류합니다. 이런 플래그는 사실상 실험이나 점진 릴리스 목적을 다했는데 코드만 남아있는 경우가 많습니다. 4) 장기적으로는 운영 토글(킬 스위치)만 영구적으로 유지하고, 릴리스/실험 토글은 "임시"라는 원칙을 조직 문화로 명문화해, 플래그 생성 자체에 만료 계획이 없으면 승인되지 않게 합니다. 핵심은 피처 플래그를 도입할 때부터 "제거는 선택이 아니라 라이프사이클의 필수 단계"로 설계해야 하며, 그러지 않으면 배포 유연성을 얻는 대신 장기적인 코드 복잡도 부채를 떠안게 된다는 트레이드오프입니다.

5. 롤백 전략과 DB 마이그레이션 호환성

배경 개념: "이전 버전으로 되돌리면 된다"는 DB가 껴 있으면 틀린 말이 된다

애플리케이션 코드만 배포한다면 롤백은 단순히 이전 이미지/바이너리를 다시 배포하면 됩니다. 그러나 실무 배포의 상당수는 DB 스키마 변경(마이그레이션)을 동반하고, 이 경우 "코드만 되돌리기"는 위험합니다. 예를 들어 새 버전이 컬럼 이름을 바꾸는 마이그레이션과 함께 배포됐는데 문제가 생겨 애플리케이션 코드만 이전 버전으로 롤백하면, 구버전 코드는 이미 이름이 바뀐 컬럼을 찾지 못해 더 큰 장애가 납니다. 안전한 롤백을 보장하려면 스키마 변경과 코드 변경을 같은 배포에 원자적으로 묶지 않고, 여러 단계로 쪼개 항상 하위 호환을 유지해야 합니다. 이를 흔히 확장-수축(Expand-Contract) 패턴이라 부릅니다.

면접 질문

"사용자 테이블의 name 컬럼을 first_name과 last_name으로 분리해야 합니다. 이 마이그레이션을 배포하는 도중이나 직후에 문제가 생겨도 안전하게 롤백할 수 있으려면 어떻게 단계를 나누시겠습니까?"

면접관이 보는 신호

"마이그레이션 스크립트를 짜서 한 번에 배포하겠다"는 답에서 멈추는지, 아니면 신-구 코드가 공존하는 기간 동안 양쪽 모두가 정상 동작하도록 여러 단계로 쪼개는 설계를 하는지를 봅니다.

Expand-Contract 패턴 단계

1
Expand(확장) — 기존 컬럼은 그대로 두고 새 컬럼을 추가만 합니다. first_name, last_name 컬럼을 추가하되 name은 삭제하지 않습니다. 이 시점에는 구버전 코드(name만 읽고 씀)와 신버전 코드가 각자 자신이 아는 컬럼만 쓰므로 어느 버전이 떠 있어도 안전합니다.
2
Migrate(이행) — 애플리케이션이 두 컬럼 모두를 쓰도록(dual write) 배포하고, 기존 데이터를 배치로 백필(backfill)합니다. 새 코드는 쓰기 시 name과 first_name/last_name을 모두 갱신해, 이 시점에 롤백해도 name 컬럼이 여전히 최신 상태로 유지됩니다.
3
Cutover(전환) — 읽기 경로를 새 컬럼으로 전환한 코드를 배포합니다. 이 시점에도 name 컬럼은 dual write로 계속 갱신되고 있으므로, 문제가 생기면 애플리케이션 코드만 이전 단계로 롤백해도 안전합니다(구버전은 여전히 name을 읽고 그 값은 최신이기 때문).
4
Contract(수축) — 충분한 안정화 기간(예: 몇 주)이 지나 롤백 가능성이 사실상 사라졌다고 판단되면, dual write를 제거하고 마지막으로 name 컬럼을 삭제합니다. 이 마지막 단계는 가장 되돌리기 어려운 단계이므로 앞의 세 단계보다 훨씬 보수적으로, 모니터링 기간을 충분히 두고 진행합니다.
강한 답변

"컬럼 분리를 한 번의 배포로 묶지 않고 Expand-Contract 4단계로 나눕니다. 1단계에서 first_name, last_name 컬럼을 추가만 하고 name은 그대로 둬 어느 버전이 떠 있어도 안전하게 합니다. 2단계에서 애플리케이션이 두 컬럼 모두에 쓰도록(dual write) 배포하고 기존 데이터를 배치로 백필합니다. 3단계에서 읽기를 새 컬럼으로 전환하되, dual write는 유지해 이 단계에서 문제가 생기면 애플리케이션 코드만 이전 단계로 되돌려도 name이 최신 상태라 안전합니다. 마지막 4단계인 name 컬럼 삭제는 몇 주간 안정화를 확인한 뒤, 가장 되돌리기 어려운 단계이므로 별도로 보수적으로 진행합니다. 이렇게 하면 매 단계마다 코드만 롤백해도 데이터 정합성이 깨지지 않습니다."

약한 답변

"마이그레이션 스크립트로 컬럼을 분리하고 데이터를 옮긴 뒤, name 컬럼을 삭제하는 배포를 한 번에 진행하겠습니다."

약한 답변이 약한 이유: 컬럼 삭제까지 한 배포에 묶으면, 배포 직후 문제가 발견돼도 애플리케이션 코드만 롤백하는 것이 불가능합니다(구버전 코드가 이미 삭제된 name 컬럼을 찾기 때문). 결국 롤백을 위해 삭제된 데이터를 백업에서 복구해야 하는, 훨씬 느리고 위험한 복구 절차로 이어집니다.

Senior Follow-up

"자동 롤백 시스템을 구축하려 합니다. '에러율이 5분간 평소보다 3배 이상'이면 자동으로 이전 버전으로 되돌리는 규칙을 넣으려는데, 이 자동화에서 어떤 위험을 고려해야 할까요?"

모범 추론: 자동 롤백은 강력하지만 잘못 설계하면 오히려 상황을 악화시키는 새로운 실패 모드를 만듭니다. 1) DB 마이그레이션이 이미 Contract 단계까지 진행된 배포라면, 코드만 되돌리는 자동 롤백은 데이터 정합성을 깨뜨립니다. 자동 롤백 시스템은 "이 배포가 코드만 롤백해도 안전한 단계인지"를 배포 메타데이터로 알고 있어야 하며, 안전하지 않다면 자동 롤백 대신 사람에게 에스컬레이션해야 합니다. 2) 지표 자체가 배포와 무관한 이유(외부 결제사 장애, 트래픽 급증)로 나빠졌을 때 잘못 롤백하면, 실제 원인은 해결되지 않은 채 정상 배포만 되돌리는 낭비가 생깁니다. 카나리(3장)처럼 신버전과 구버전을 같은 시간대에 동시 비교하는 방식이면 이 문제를 줄일 수 있습니다(동일 시점의 외부 요인은 양쪽에 공평하게 영향을 주므로). 3) 롤백 자체도 실패할 수 있다는 것을 감안해, 롤백이 반복적으로 트리거되어 배포-롤백을 오가는 진동(flapping)이 생기지 않도록 롤백 후 일정 시간 재배포를 잠그는 쿨다운을 둡니다. 4) 자동 롤백이 실행됐다는 사실 자체가 사람에게 즉시 알림으로 전달되어야 하며, "자동으로 처리됐으니 괜찮다"고 방치하지 않고 반드시 사후 분석(Day 7의 포스트모템)으로 이어져야 합니다. 핵심은 자동 롤백을 "코드 되돌리기"라는 단순 동작이 아니라, 데이터 안전성·오판 가능성·반복 트리거 위험까지 고려한 시스템으로 설계해야 한다는 것입니다.

6. 종합 시나리오 — 금요일 오후 배포 후 에러율 급증

면접 질문

"금요일 오후 4시에 배포한 지 10분 후, 결제 성공률이 평소보다 크게 떨어졌다는 알림을 받았습니다. 이 배포에는 애플리케이션 코드 변경과 함께 DB 마이그레이션(새 컬럼 추가, dual write 단계)이 포함되어 있습니다. 지금부터 어떻게 대응하시겠습니까?"

이 질문은 오늘 배운 배포 전략·롤백·DB 호환성·조직 소통을 시간 압박이 있는 실제 장애 상황에서 순서대로 조합하는 능력을 봅니다.

1
이 배포가 카나리 방식이었다면 먼저 카나리 트래픽 비율부터 확인합니다. 아직 일부(예: 10%)만 신버전이라면 영향받는 사용자 범위가 이미 제한되어 있다는 것을 확인하고, 즉시 카나리를 0%로 되돌려 신버전 트래픽을 차단합니다.
2
이번 배포의 DB 마이그레이션 단계가 Expand-Contract 중 어느 단계인지 확인합니다. 문제 설명에 "dual write 단계"라고 명시되어 있으므로, 구버전 코드가 여전히 읽는 원래 컬럼도 최신 상태로 유지되고 있어 애플리케이션 코드만 이전 버전으로 롤백해도 데이터 정합성이 깨지지 않는다는 것을 확인한 뒤 롤백을 실행합니다.
3
롤백 직후 결제 성공률이 정상으로 돌아오는지 확인해, 이번 배포가 실제 원인이었는지(우연히 같은 시각 외부 PG 장애와 겹친 것이 아닌지) 검증합니다. 만약 롤백 후에도 지표가 회복되지 않는다면, 원인은 이번 배포가 아니라 다른 곳(외부 의존성)에 있다는 뜻이므로 조사 방향을 전환합니다.
4
영향받은 결제 건에 대한 데이터 후처리를 확인합니다. 실패로 기록된 결제 중 실제로는 PG사에서 처리됐지만 응답을 못 받은 건(멱등키로 재시도 처리, Day 6 연결)이 있는지 확인해, 이중 청구나 누락이 없는지 점검합니다.
5
"금요일 오후 배포"라는 시점 자체를 조직적으로 되돌아봅니다. 오늘 배운 점진 배포와 안전한 롤백 설계가 갖춰져 있었기에 빠르게 복구할 수 있었지만, 만약 이런 안전장치가 없었다면 금요일 오후·주말 온콜 인력이 적은 시간대의 배포는 더 큰 위험을 안고 있었을 것입니다. 배포 시점 정책(위험도가 높은 변경은 금요일 오후 배포를 지양하는 등)을 팀 차원에서 재논의합니다.
6
블레임리스 포스트모템(Day 7)을 진행해, 이번 회귀가 스테이징에서 발견되지 않은 이유(테스트 커버리지 공백, 실제 프로덕션 트래픽 패턴과의 차이)를 분석하고 재발 방지책을 CI/CD 파이프라인에 반영합니다.
시니어가 반드시 언급해야 하는 것: 배포와 DB 마이그레이션이 얽힌 장애에서 "코드만 롤백해도 안전한 단계인가"를 먼저 확인하는 판단, 카나리처럼 이미 제한된 노출 범위를 활용하는 신속한 대응, 롤백 후에도 원인이 정말 이 배포였는지 검증하는 태도, 그리고 데이터 정합성(이중 청구/누락) 후처리와 배포 시점·문화적 재발 방지까지 연결하는 넓은 시야입니다. 이 네 가지가 없으면 "일단 롤백 버튼을 눌렀다"는 즉흥 대응으로만 보입니다.

7. 자기 점검 Rubric

각 항목을 솔직하게 채점하세요. ●●●=잘 설명할 수 있음, ●●○=부분적으로 앎, ●○○=어렴풋이 앎, ○○○=모름

CI/CD 파이프라인의 각 단계가 무엇을 검증하고 왜 그 순서(빠르고 저렴한 검증부터)로 배치되는지 설명할 수 있다
트렁크 기반 개발과 배포 배치 크기를 줄이는 것이 왜 배포 위험을 낮추는지 설명할 수 있다
블루-그린/롤링/카나리의 차이와, 서비스 위험 민감도에 따라 어떤 전략을 선택할지 근거를 들어 판단할 수 있다
카나리 배포의 자동 중단 기준을 에러율뿐 아니라 지연 percentile 등 다축으로 설계해야 하는 이유를 설명할 수 있다
피처 플래그로 배포와 릴리스를 분리하는 이유와, 플래그 부채를 방지하는 라이프사이클 관리 방법을 설명할 수 있다
Expand-Contract 패턴의 4단계와, 각 단계에서 코드만 롤백해도 데이터 정합성이 유지되는 이유를 설명할 수 있다
자동 롤백 시스템이 데이터 정합성, 외부 요인 오판, 반복 트리거(flapping)를 어떻게 고려해야 하는지 설명할 수 있다
배포 관련 장애 발생 시 롤백 안전성 판단 → 원인 검증 → 데이터 후처리 → 조직적 재발 방지까지 순서대로 대응할 수 있다
