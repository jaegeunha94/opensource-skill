1. 오늘 레슨 개요

데이터베이스 면접 질문은 "ACID가 뭔가요?"처럼 단순 정의를 묻는 것이 아닙니다. 10년차 이상 면접에서는 "격리 수준을 어떻게 선택했고, 그 결과 어떤 이상 현상이 생겼으며, 어떻게 진단했나"처럼 실제 장애 경험과 연결한 판단력을 봅니다.

면접관이 실제로 보는 신호
  • 개념을 안다 ↔ 개념을 실무에서 적용한 결과를 설명할 수 있다
  • 최선의 답 ↔ 상황에 따라 달라지는 trade-off를 명확히 말할 수 있다
  • 단일 DB 관점 ↔ 분산/서비스 경계를 함께 고려한다

이 레슨에서는 6개의 핵심 질문 영역을 다루며, 각 영역마다 면접 질문 → 답변 구조 → senior follow-up → 좋은/약한 답변 비교 → 자기 점검 순서로 진행합니다.

2. ACID — 왜 여전히 면접의 핵심인가

배경 개념: 왜 트랜잭션이 필요한가

은행 이체를 예로 듭니다. A 계좌에서 100만 원을 빼고 B 계좌에 100만 원을 더해야 할 때, 중간에 서버가 죽으면 A에서만 돈이 빠져나가는 문제가 생깁니다. 트랜잭션은 "여러 연산을 하나의 논리 단위로 묶어, 모두 성공하거나 모두 실패하게" 보장합니다. ACID는 그 보장의 4가지 속성입니다.

면접 질문

"ACID를 설명해 주시고, 실무에서 각 속성이 위배됐을 때 어떤 문제가 생기는지 예를 들어 주세요."

면접관이 보는 신호

단순 암기가 아니라 "위반 시 증상 → 진단 방법 → 해결책"까지 연결하는 사고력을 봅니다.

단계별 답변 구조

1
Atomicity (원자성) — 트랜잭션 안의 모든 연산은 전부 반영되거나 전부 롤백됩니다. 위반 시 증상: 부분 업데이트 후 실패 → 데이터 불일치. 진단: 이체 후 총액이 맞지 않거나, order_item 레코드는 생겼는데 order 레코드가 없는 상황.
2
Consistency (일관성) — 트랜잭션 실행 전후, DB는 정의된 제약(FK, 유니크, 체크 제약 등)을 만족해야 합니다. 위반 시 증상: FK가 없는 고아 레코드, 마이너스 잔액. 현대적 관점: 애플리케이션 레벨 비즈니스 규칙까지 포함. DB만으로 보장 불가능한 경우가 많아 서비스 레이어 검증이 중요합니다.
3
Isolation (격리성) — 동시에 실행되는 트랜잭션들이 서로의 중간 상태를 볼 수 없어야 합니다. 위반 시 증상: Dirty Read, Non-Repeatable Read, Phantom Read. 실무 포인트: 완전한 격리는 직렬화(Serializable)이지만 성능 비용이 크므로, 적절한 격리 수준을 선택합니다.
4
Durability (지속성) — 커밋된 트랜잭션은 장애 후에도 유지됩니다. 구현: WAL(Write-Ahead Log) / Redo Log. 서버 크래시 후 재시작 시 redo log로 커밋된 데이터를 복구합니다. 위반 시 증상: 커밋 성공 응답 후 재시작 시 데이터 소실 → 가장 치명적인 버그.
강한 답변

"ACID 각각을 위반했을 때 증상을 먼저 말씀드리겠습니다. Atomicity 위반은 재고 차감만 되고 주문이 생성되지 않는 반쪽짜리 업데이트로 나타납니다. Isolation 위반 중 Non-Repeatable Read는 보고서를 생성하는 트랜잭션 안에서 같은 row를 두 번 읽었을 때 다른 값이 나오는 형태로 나타났고, 이를 해결하기 위해 Repeatable Read 격리 수준을 적용했습니다. Durability는 fsync 설정과 WAL 위치에 따라 실제 내구성이 달라지는데, MySQL innodb_flush_log_at_trx_commit=1이 아닌 설정에서는 OS 버퍼에만 쓰고 커밋 응답을 반환해 크래시 시 데이터를 잃을 수 있습니다."

약한 답변

"ACID는 Atomicity, Consistency, Isolation, Durability의 약자입니다. Atomicity는 전부 성공하거나 전부 실패하는 것이고, Consistency는 일관성이고, Isolation은 격리성이고, Durability는 지속성입니다."

약한 답변이 약한 이유: 정의만 반복하고 위반 시 증상, 실무 경험, trade-off가 없습니다. 시니어 면접에서 이 답변은 주니어 수준으로 평가됩니다.

Senior Follow-up

"NoSQL(예: MongoDB, Cassandra)은 ACID를 포기했다고 하는데, 정말 그런가요? 그리고 만약 이체 로직을 MongoDB로 구현해야 한다면 어떻게 하시겠어요?"

모범 추론: MongoDB 4.0+ 이후 multi-document 트랜잭션을 지원하므로 "NoSQL = no ACID"는 틀린 일반화입니다. 다만 단일 document 연산 수준에서만 원자성이 보장되고, 트랜잭션은 성능 비용이 있습니다. 이체 로직은 document 설계를 바꿔서 account document 하나 안에 balance를 담고 atomic update ($inc) 하거나, 2PC(Two-Phase Commit) 패턴을 직접 구현하거나, saga 패턴으로 보상 트랜잭션을 설계합니다. 핵심: "NoSQL이라 ACID를 못 쓴다"가 아니라 "어떤 ACID 보장이 필요한가, 비용은 얼마인가"를 먼저 정의합니다.

3. 트랜잭션 격리 수준 — 가장 많이 틀리는 영역

배경 개념: 왜 격리 수준이 여러 개인가

완전한 격리(Serializable)는 직렬 실행과 동일한 결과를 보장하지만, 동시성을 거의 허용하지 않아 TPS가 급락합니다. 반대로 격리를 풀면 이상 현상이 생깁니다. SQL 표준(ANSI SQL-92)은 이 균형을 4단계로 정의했습니다.

면접 질문

"Repeatable Read와 Serializable의 차이를 설명하고, MySQL InnoDB의 기본 격리 수준인 Repeatable Read가 Phantom Read를 어떻게 방지하는지 설명해 주세요."

면접관이 보는 신호

격리 수준 표를 외운 게 아니라, 구현 레벨(MVCC, Next-Key Lock)까지 이해하는지 봅니다. 10년차라면 실무 장애와 연결할 수 있어야 합니다.

이상 현상 3가지 — 먼저 완전히 이해하기

이상 현상 발생 조건 실무 증상 예시
Dirty Read 다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽음 TX1이 환불 처리 중, TX2가 롤백될 환불 금액을 읽어 이중 지급
Non-Repeatable Read 같은 row를 같은 트랜잭션 내에서 두 번 읽었을 때 값이 다름 TX1이 재고를 두 번 읽는 동안, TX2가 재고를 변경 → 잘못된 재고 검증
Phantom Read 같은 조건으로 범위 조회 시, 새로운 row가 나타나거나 사라짐 TX1이 "잔액 > 0인 계좌 목록"을 두 번 조회, 중간에 TX2가 새 계좌 추가

격리 수준별 방지 여부

격리 수준 Dirty Read Non-Repeatable Read Phantom Read 성능 영향
Read Uncommitted 허용 허용 허용 가장 빠름
Read Committed 방지 허용 허용 빠름 (PostgreSQL 기본값)
Repeatable Read 방지 방지 허용 (표준상) 중간 (MySQL InnoDB 기본값)
Serializable 방지 방지 방지 가장 느림

핵심 포인트: MySQL InnoDB의 Repeatable Read는 Phantom Read를 방지한다

ANSI 표준상 Repeatable Read는 Phantom Read를 허용하지만, MySQL InnoDB는 Next-Key Lock으로 범위에 갭(Gap) 락을 걸어 새 row 삽입을 막습니다. 즉 MySQL에서는 Repeatable Read가 사실상 Serializable에 가까운 보장을 제공합니다.

Next-Key Lock이란?
레코드 락 + 갭(gap) 락의 조합입니다. 예를 들어 WHERE id BETWEEN 10 AND 20 조회 시, id=10~20 레코드뿐 아니라 그 사이 갭에도 락을 걸어 새 insert를 막습니다. 이 덕분에 Phantom Read가 방지됩니다. 단, 갭 락은 데드락의 원인이 될 수 있습니다.
강한 답변

"표준 SQL에서 Repeatable Read는 Phantom Read를 허용하지만, MySQL InnoDB는 Next-Key Lock을 사용해 실질적으로 방지합니다. MVCC로 스냅샷 읽기를 제공하고, 범위 쿼리 실행 시 갭 락을 추가로 걸어 해당 범위에 새 row가 삽입되지 못하게 합니다. Serializable은 모든 읽기에 공유 락을 걸어 더 강하게 보장하지만, 동시성이 크게 줄어 OLTP에서는 잘 사용하지 않습니다."

약한 답변

"Repeatable Read는 Phantom Read가 발생할 수 있고 Serializable은 모든 이상 현상을 막습니다. Serializable이 더 안전하므로 중요한 곳에 쓰면 됩니다."

약한 답변이 약한 이유: MySQL의 실제 구현을 모르고, Serializable의 성능 비용을 무시합니다. "중요한 곳에 쓰면 된다"는 판단 기준이 없습니다.

Senior Follow-up

"트랜잭션 격리 수준을 Read Committed로 운영하다가, 특정 배치 잡이 같은 데이터를 두 번 처리하는 버그가 발생했습니다. 어떻게 진단하고 해결하시겠어요?"

모범 추론:
1) Read Committed에서 Non-Repeatable Read가 발생하면, 배치 잡이 "처리 전 상태" row를 두 번 읽을 수 있습니다.
2) 진단: slow query log, binlog, 배치 잡 실행 로그를 대조해 중복 처리된 row id를 찾습니다.
3) 해결 방법은 상황에 따라 다릅니다:

  • 격리 수준을 Repeatable Read로 올리거나,
  • 배치 쿼리에 SELECT FOR UPDATE로 명시적 락을 걸거나,
  • 처리 상태 컬럼(processed_at)을 두고 atomic CAS(UPDATE ... WHERE processed_at IS NULL) 패턴으로 멱등성을 보장.
마지막 방법이 가장 견고하며 격리 수준에 의존하지 않습니다.

4. 락 전략 — 낙관적 vs 비관적 vs MVCC

배경 개념: 동시 접근 문제

여러 사용자가 동시에 같은 row를 수정하려 할 때 "Lost Update" 문제가 발생합니다. TX1이 잔액 1000원을 읽고, TX2도 1000원을 읽은 뒤 각각 100원을 빼면 최종 잔액이 900원이 됩니다 (정답 800원). 락은 이를 방지하는 메커니즘입니다.

면접 질문

"낙관적 락과 비관적 락의 차이를 설명하고, 각각 어떤 상황에서 선택하시겠어요? 낙관적 락 구현 시 ABA 문제는 어떻게 해결하나요?"

핵심 개념 비교

항목 비관적 락 (Pessimistic) 낙관적 락 (Optimistic)
가정 충돌이 자주 발생할 것 충돌이 드물 것
구현 SELECT FOR UPDATE, 뮤텍스 version 컬럼, ETag, CAS
충돌 처리 락 해제 대기 (block) 버전 불일치 시 오류 반환 → 재시도
장점 충돌 보장, 단순 로직 락 없이 읽기 성능 우수, 데드락 없음
단점 데드락 위험, TPS 제한, 장기 트랜잭션 위험 재시도 로직 복잡, 충돌 많으면 오히려 느림
적합 상황 충돌 빈번, 금융 이체, 재고 선점 충돌 드묾, 프로필 편집, 장바구니 수정

낙관적 락 구현 예시

-- 1. 조회 시 version 포함
SELECT id, balance, version FROM accounts WHERE id = 1;
-- 결과: id=1, balance=1000, version=5

-- 2. 업데이트 시 version 조건 추가
UPDATE accounts
SET balance = 900, version = version + 1
WHERE id = 1 AND version = 5;
-- 영향받은 row 수가 0이면 충돌 → 재시도

ABA 문제란?

version 대신 단순 값 비교를 쓸 때 발생합니다. 잔액이 1000 → 900 → 1000으로 돌아와도 "1000으로 같으니 변경 없음"으로 잘못 판단합니다. 해결: 단조 증가 version 숫자나 타임스탬프를 사용하면 이전 상태로 돌아가도 version이 다르므로 탐지됩니다.

데드락과 진단

데드락 발생 패턴: TX1이 row A를 락 → TX2가 row B를 락 → TX1이 B를 요청 (대기) → TX2가 A를 요청 (대기) → 교착 상태.

진단: SHOW ENGINE INNODB STATUS에서 LATEST DETECTED DEADLOCK 섹션 확인.

예방: 모든 트랜잭션이 동일한 순서로 row를 잠그도록 강제합니다 (e.g., 항상 id 오름차순). 또는 트랜잭션을 짧게 유지해 락 보유 시간을 줄입니다.
강한 답변

"이커머스 재고 차감에는 비관적 락을 씁니다. 플래시 세일처럼 동시 요청이 몰릴 때 낙관적 락은 재시도 폭발로 오히려 느려지고, 재고 음수(-) 현상도 막아야 하기 때문입니다. SELECT FOR UPDATE로 선점 후 재고를 확인하고 차감합니다. 반면 사용자 프로필 수정은 충돌이 드물어 version 컬럼 기반 낙관적 락으로 처리합니다. 데드락은 항상 fk_id 오름차순으로 잠그고, 트랜잭션 길이를 최소화하는 규칙을 팀에 강제합니다."

약한 답변

"낙관적 락은 락을 안 걸고 버전을 체크하는 거고, 비관적 락은 SELECT FOR UPDATE입니다. 낙관적 락이 더 성능이 좋으니 되도록 낙관적 락을 쓰면 됩니다."

약한 답변이 약한 이유: "되도록 낙관적 락"은 상황 판단 없는 일반론입니다. 충돌이 잦을 때는 낙관적 락이 더 느립니다.

Senior Follow-up

"MVCC(Multi-Version Concurrency Control)는 락 없이 격리를 제공한다고 하는데, SELECT FOR UPDATE와 어떻게 함께 동작하나요?"

모범 추론: MVCC는 읽기 전용 스냅샷에 적용됩니다. 일반 SELECT는 트랜잭션 시작 시점의 스냅샷을 읽어 락이 불필요합니다. 그러나 SELECT FOR UPDATE는 "현재 커밋된 최신 값"을 읽고 배타 락을 겁니다. 즉 MVCC는 읽기 성능을, SELECT FOR UPDATE는 쓰기 충돌 방지를 각각 담당하며 공존합니다. 주의: SELECT FOR UPDATE로 읽은 값은 스냅샷이 아니라 최신 값이므로, MVCC 스냅샷과 다를 수 있습니다.

5. 인덱싱 전략 — B-Tree, 복합 인덱스, 카버링 인덱스

배경 개념: 인덱스는 왜 빠른가

테이블에 1억 개의 row가 있을 때 full scan은 모든 row를 읽습니다. B-Tree 인덱스는 키를 정렬된 트리 구조로 유지해 O(log N)에 찾습니다. 단, 인덱스는 별도 자료구조이므로 쓰기(INSERT/UPDATE/DELETE) 시 인덱스도 갱신해 쓰기 비용이 증가합니다.

면접 질문

"복합 인덱스를 설계할 때 컬럼 순서를 어떻게 결정하시나요? 카버링 인덱스란 무엇이며 언제 유용한가요?"

복합 인덱스 컬럼 순서 결정 원칙

1
Cardinality (선택도) — 고유값이 많은 컬럼을 앞에 둡니다. status(3가지 값) vs user_id(10만 가지 값)이면 user_id가 앞. 선택도가 낮으면 인덱스를 탔어도 full scan에 가까운 row를 읽습니다.
2
쿼리 패턴 우선 — 가장 자주 사용하는 WHERE 조건의 equality 컬럼을 앞에 둡니다. range 조건(>, BETWEEN, LIKE 'a%')은 뒤에 놓아야 뒤 컬럼의 인덱스도 활용됩니다.
3
Leftmost Prefix Rule — INDEX(a, b, c)는 WHERE a=?, WHERE a=? AND b=?, WHERE a=? AND b=? AND c=?에 사용됩니다. WHERE b=?만 있으면 이 인덱스를 쓸 수 없습니다.

예시 시나리오

-- 주문 테이블 (orders)
-- 자주 실행되는 쿼리:
SELECT * FROM orders
WHERE user_id = ? AND status = 'PAID' AND created_at > '2024-01-01';

-- 좋은 인덱스: (user_id, status, created_at)
-- user_id: equality, 선택도 높음 → 1순위
-- status: equality → 2순위
-- created_at: range → 마지막
-- 나쁜 인덱스: (created_at, status, user_id) → range가 앞에 있어 뒤 컬럼 활용 불가

카버링 인덱스(Covering Index)

쿼리에서 필요한 컬럼이 모두 인덱스 안에 포함되어 테이블을 전혀 읽지 않아도 되는 인덱스입니다. I/O가 0에 수렴하므로 대용량 테이블에서 극적인 성능 향상을 가져옵니다. 

-- user_id, status만 필요한 쿼리
SELECT user_id, status FROM orders WHERE user_id = 123;

-- INDEX(user_id, status) 가 있으면:
-- 1. 인덱스에서 user_id=123인 leaf node를 찾음
-- 2. 인덱스 leaf에 status가 이미 있으므로 테이블 접근(random I/O) 없음
-- EXPLAIN에서 Extra: "Using index" → 카버링 인덱스 사용 확인
자주 나오는 함정 질문

"인덱스가 있는데 왜 full scan이 나오나요?"

옵티마이저가 인덱스 대신 full scan을 선택하는 주요 원인:

  • 선택도가 낮아 인덱스를 타도 대부분의 row를 읽어야 할 때 (행의 20~30% 이상)
  • 인덱스 컬럼에 함수/형변환 적용: WHERE YEAR(created_at) = 2024 → created_at 인덱스 무효화
  • 묵시적 형변환: WHERE user_id = '123' (user_id가 int인데 문자열 비교)
  • 통계 정보(statistics)가 오래돼 옵티마이저가 잘못 판단
진단: EXPLAIN / EXPLAIN ANALYZE로 실행 계획 확인.

강한 답변

"복합 인덱스 순서는 쿼리 패턴과 선택도를 함께 봅니다. 항상 equality가 앞에 오고 range가 뒤에 와야 뒤 컬럼까지 인덱스를 활용합니다. Leftmost Prefix Rule 때문에, user_id가 모든 쿼리에 공통이라면 반드시 첫 번째에 둡니다. 카버링 인덱스는 SELECT 컬럼까지 인덱스에 포함해 테이블 접근 자체를 없애는데, 특히 집계 쿼리나 대용량 리스트 조회에서 EXPLAIN에 'Using index'가 보이면 극적인 성능 개선이 가능합니다."

약한 답변

"선택도가 높은 컬럼을 앞에 두면 됩니다. 인덱스가 많을수록 읽기가 빠르니 웬만하면 컬럼마다 인덱스를 만드는 게 좋습니다."

약한 답변이 약한 이유: "인덱스가 많을수록 좋다"는 틀립니다. 인덱스는 쓰기 비용을 올리고, 스토리지를 소모하며, 옵티마이저가 잘못 선택하면 오히려 느려집니다. range보다 선택도가 먼저라는 것도 단순화입니다.

6. 쿼리 최적화 — EXPLAIN 읽기, N+1, Slow Query

면접 질문

"서비스 지연 장애가 발생했고, 원인이 DB 쿼리임을 의심합니다. 어떤 순서로 진단하고 최적화하시겠어요?"

진단 단계별 접근

1
Slow Query Log 확인 — long_query_time 임계값(예: 1초) 초과 쿼리를 수집합니다. mysqldumpslow 또는 pt-query-digest로 집계해 실행 횟수 × 평균 시간이 큰 쿼리를 식별합니다. "가장 오래 걸린 쿼리"가 아니라 "총 소요 시간이 가장 큰 쿼리"가 실제 병목입니다.
2
EXPLAIN / EXPLAIN ANALYZE 분석 — type 컬럼에서 ALL(full scan), index(full index scan)는 위험 신호. ref, eq_ref, const가 목표. rows 컬럼이 예상 탐색 row 수이며, 실제 row 수와 크게 다르면 통계 갱신(ANALYZE TABLE)이 필요합니다.
3
N+1 문제 탐지 — APM(DataDog, New Relic, Sentry)에서 단일 API 호출에 쿼리가 수백 개 발생하면 N+1입니다. 해결: JOIN 또는 IN절로 배치 조회하거나, ORM의 eager loading(include, joinedload)을 사용합니다.
4
인덱스 추가 또는 쿼리 재작성 — EXPLAIN 분석 후 필요 인덱스 추가. 단, 운영 DB에 인덱스 추가 시 pt-online-schema-change 또는 gh-ost로 무중단 변경합니다.

N+1 문제 예시

# Python (Django) 예시 — N+1 발생
orders = Order.objects.filter(user_id=1)  # 쿼리 1번
for order in orders:
    print(order.user.email)               # 각 order마다 user 쿼리 → N번 추가

# 해결: select_related로 JOIN
orders = Order.objects.filter(user_id=1).select_related('user')
# 쿼리 1번으로 user도 함께 로드
Senior Follow-up

"EXPLAIN 결과 rows가 예상보다 100배 많게 나왔습니다. 왜 이런 일이 생기고 어떻게 해결하나요?"

모범 추론: 옵티마이저는 테이블 통계(히스토그램, 인덱스 선택도)를 기반으로 실행 계획을 수립합니다. 대량 데이터 INSERT/DELETE 후 통계가 오래되면 실제 분포와 달라집니다. 해결: ANALYZE TABLE orders로 통계 갱신. MySQL 8.0+는 히스토그램(ANALYZE TABLE ... UPDATE HISTOGRAM)으로 더 세밀한 통계를 수집합니다. 통계 갱신 후에도 개선이 없으면 USE INDEX 힌트로 강제할 수 있지만, 이는 임시방편이므로 쿼리 재설계나 인덱스 변경이 근본 해결책입니다.

7. 정규화 trade-off — 비정규화를 언제 허용하는가

배경 개념: 정규화의 목적

정규화(Normalization)는 데이터 중복을 줄이고 갱신 이상(Update Anomaly)을 방지하기 위해 테이블을 분리하는 과정입니다. 3NF(제3정규형)까지 적용하면 대부분의 데이터 중복이 제거됩니다.

면접 질문

"정규화를 충실히 적용한 DB가 있는데, 특정 조회 성능이 너무 느립니다. 비정규화를 고려하시겠나요? 어떤 기준으로 결정하시겠어요?"

정규화 vs 비정규화 trade-off

항목 정규화 (3NF) 비정규화 (Denormalization)
데이터 일관성 한 곳만 수정하면 됨 (높음) 여러 곳 수정 필요, 불일치 위험
읽기 성능 JOIN이 많아 복잡한 쿼리 필요 단일 테이블 조회 가능 (빠름)
쓰기 성능 한 테이블만 수정 (빠름) 여러 컬럼 동기화 필요 (느림)
스토리지 최소화 중복으로 증가
적합 케이스 쓰기 많은 OLTP, 소규모 읽기 많은 OLAP, 대시보드, 검색

비정규화 결정 기준 (단계적 접근)

1
먼저 인덱스와 쿼리 최적화를 시도 — 비정규화는 마지막 수단입니다. 인덱스 추가, 쿼리 재작성, 카버링 인덱스로 해결되면 비정규화는 불필요합니다.
2
읽기 vs 쓰기 비율 확인 — 읽기가 압도적으로 많고(예: 99:1), 읽기 쿼리가 JOIN을 5개 이상 포함하면 비정규화를 검토합니다.
3
일관성 비용 계산 — 비정규화한 컬럼이 변경될 빈도는? 변경 시 동기화 누락의 영향은? 주문 테이블에 상품명을 비정규화하면 "주문 당시 상품명"이 맞지만, "현재 상품명"을 보여줘야 한다면 JOIN이 옳습니다.
4
대안 검토 — 캐싱(Redis), Materialized View, Read Replica, 검색 엔진(Elasticsearch)이 비정규화 없이 읽기 성능을 해결할 수 있는지 먼저 확인합니다.
시니어가 알아야 할 현실

"이커머스 주문 테이블에 상품명, 가격 스냅샷을 저장하는 것은 비정규화이지만 올바른 설계입니다. 주문 당시의 가격이 현재 가격과 달라도 주문 내역은 원래 금액을 보여줘야 하기 때문입니다. 이처럼 비정규화가 비즈니스 요구사항과 일치하는 경우가 있습니다."

강한 답변

"비정규화는 인덱스·쿼리 최적화·캐싱으로 해결 안 될 때 마지막으로 검토합니다. 결정 전에 읽기/쓰기 비율, 해당 컬럼의 변경 빈도, 불일치 시 비즈니스 영향을 분석합니다. 주문 테이블에 상품명 스냅샷 저장처럼 '불변 과거 데이터'는 자연스러운 비정규화입니다. 그러나 사용자 이메일을 여러 테이블에 복사하면 변경 시 모두 업데이트해야 하므로 절대 피합니다. Materialized View나 Read Replica를 통해 OLTP 스키마를 유지하면서 읽기 성능을 분리하는 것이 더 나은 경우가 많습니다."

약한 답변

"정규화를 유지하면서 인덱스를 더 추가하거나, 아니면 비정규화해서 JOIN을 줄이면 됩니다. 성능이 안 나오면 비정규화가 답입니다."

약한 답변이 약한 이유: 어떤 상황에서 어떤 선택을 하는지 기준이 없습니다. 일관성 비용, 대안 검토, 비즈니스 요구사항이 빠졌습니다.
