1. 오늘 레슨 개요

API는 팀 내부에서만 쓰이는 것 같아도, 일단 다른 서비스나 외부 클라이언트가 호출하기 시작하는 순간 가장 바꾸기 어려운 계약(contract)이 됩니다. DB 스키마는 마이그레이션으로 바꿀 수 있고 내부 로직은 리팩터링할 수 있지만, API 응답 필드 하나를 지우거나 이름을 바꾸는 순간 그 API를 호출하는 모든 클라이언트가 동시에 깨질 수 있습니다. 10년차 이상 면접에서 API 설계를 묻는 이유는 "지금 당장 편한 설계"가 아니라 "몇 년 뒤에도 안전하게 진화시킬 수 있는 설계"를 할 수 있는가를 확인하기 위해서입니다.

면접관이 실제로 보는 신호
  • REST를 "HTTP + JSON"으로만 이해하는 것 ↔ 리소스 지향 설계와 HATEOAS까지의 스펙트럼을 이해하고 실무 절충점을 설명
  • "우리는 REST를 씁니다"라는 선언 ↔ GraphQL/gRPC 대비 언제 REST가 손해인지 구체적 시나리오로 판단
  • 버전을 URL에 박아넣는 것만 아는 것 ↔ 버전 관리 전략별 클라이언트 마이그레이션 비용까지 고려
  • "필드를 추가하는 건 안전하다"는 규칙만 암기 ↔ 실제로 클라이언트 파싱 방식에 따라 무엇이 breaking인지 사례로 구분

이 레슨은 API 설계의 4개 핵심 축을 다루며, 각 영역마다 배경 개념 → 면접 질문 → 답변 구조 → senior follow-up → 좋은/약한 답변 비교 → 자기 점검 순서로 진행합니다.

2. REST 성숙도 모델 — 진짜 REST와 RPC-over-HTTP의 차이

배경 개념: "REST API"라고 부르는 것 대부분은 REST가 아니다

많은 팀이 "REST API"라고 부르지만 실제로는 POST /getUserOrders 같은 동사형 엔드포인트를 쓰고, 상태 코드는 항상 200이며, 응답 바디의 status 필드로 성공/실패를 표현합니다. 이것은 사실 HTTP를 전송 계층으로만 쓰는 RPC(Remote Procedure Call)이지 REST가 아닙니다. REST의 핵심은 "리소스(명사)를 URI로 식별하고, HTTP 메서드(동사)로 행위를 표현하며, 응답 자체에 다음 행동 정보를 포함시킨다"는 리소스 지향 설계 철학입니다. 레너드 리처드슨(Leonard Richardson)은 이 정도를 0~3단계로 나눈 Richardson Maturity Model(RMM)을 제시했습니다.

면접 질문

"저희 API는 모든 요청을 POST로 받고 URL은 /api/action 하나만 씁니다. 이게 REST API가 맞나요? 아니라면 무엇이 문제인가요?"

면접관이 보는 신호

"REST가 아니어도 동작만 하면 된다"고 단순화하는지, 아니면 어느 단계까지 성숙도를 올릴 가치가 있는지 비용 대비로 판단하는지를 봅니다.

Richardson Maturity Model 단계

0
Level 0 — The Swamp of POX(Plain Old XML/JSON): 단일 엔드포인트, 단일 HTTP 메서드(대개 POST)로 모든 요청을 받고, 실제 동작은 요청 바디 안의 필드로 결정합니다. HTTP는 그저 데이터를 실어 나르는 봉투 역할만 합니다. 예: POST /api {"action": "getOrder", "id": 123}.
1
Level 1 — Resources: 개별 리소스마다 별도의 URI를 부여합니다. /orders/123, /users/45처럼 명사 중심 URI가 생기지만, 여전히 메서드는 대부분 POST 하나로 통일되어 있을 수 있습니다.
2
Level 2 — HTTP Verbs: GET/POST/PUT/PATCH/DELETE와 표준 상태 코드(200, 201, 404, 409 등)를 의미에 맞게 사용합니다. 실무에서 "REST API"라고 부르는 것 대부분은 이 레벨입니다. 대부분의 실무 요구사항을 충분히 만족시키는 실용적인 지점입니다.
3
Level 3 — HATEOAS(Hypermedia as the Engine of Application State): 응답 바디 안에 "다음에 가능한 행동"에 대한 링크를 포함시켜, 클라이언트가 URL을 하드코딩하지 않고 서버가 제공하는 링크를 따라가며 상태를 전이합니다. 예: 주문 조회 응답에 "actions": [{"rel": "cancel", "href": "/orders/123/cancel"}]를 포함시켜, 주문 상태가 "배송 완료"면 취소 링크 자체를 응답에서 빼버리는 방식입니다.

Level 3(HATEOAS)를 실무에서 잘 안 쓰는 이유

이론적 장점 실무에서의 현실
클라이언트가 URL을 하드코딩하지 않아도 됨 대부분의 프론트엔드/모바일 클라이언트는 어차피 특정 API 형태를 가정하고 하드코딩되어 있어, 하이퍼미디어를 실제로 "따라가며" 동작하도록 구현하는 경우가 드묾
서버가 상태 전이 가능 여부를 알려줌 이 정보를 프론트엔드가 UI 조건 분기(버튼 활성화 여부)에 활용하려면 별도의 파싱 로직이 필요해 오히려 복잡도 증가
API 문서와 실제 동작이 항상 일치 표준화된 하이퍼미디어 포맷(HAL, JSON:API, Siren)이 여러 개 난립해 생태계 합의가 약함
강한 답변

"엄밀히 말하면 저희 API는 Richardson Maturity Model 기준 Level 0에 가깝습니다. 단일 엔드포인트에 action 필드로 분기하는 구조라, 리소스별 URI가 없어 캐싱이나 로깅에서 요청 종류를 구분하기 어렵고, HTTP 상태 코드를 활용하지 못해 클라이언트가 항상 바디를 파싱해서 성공/실패를 판단해야 합니다. 다만 Level 3(HATEOAS)까지 갈 필요는 없다고 봅니다. 저희 클라이언트는 사내 모바일 앱 한 종류뿐이라 하이퍼미디어로 얻는 유연성보다 구현 복잡도가 더 크기 때문입니다. 목표는 Level 2까지 — 리소스별 URI와 표준 HTTP 메서드/상태 코드를 쓰는 정도가 비용 대비 가장 실용적이라고 판단했습니다."

약한 답변

"네 저희는 REST API 맞습니다. HTTP랑 JSON을 쓰고 있으니까요."

약한 답변이 약한 이유: "HTTP+JSON을 쓴다"는 것과 "REST 원칙을 따른다"는 것은 다른 개념입니다. 현재 상태를 성숙도 모델 기준으로 스스로 진단하지 못했고, 목표 레벨을 왜 그렇게 정했는지 근거도 없습니다.

Senior Follow-up

"Level 2(리소스 + HTTP verb)까지만 가면 충분하다고 하셨는데, 그렇다면 HATEOAS(Level 3)가 실제로 가치 있는 상황은 어떤 경우일까요?"

모범 추론: HATEOAS는 클라이언트와 서버의 배포 주기가 크게 다르고, 클라이언트 종류가 매우 다양하며 서버가 자주 바뀌는 상황에서 가치가 커집니다. 예를 들어 결제 상태 머신처럼 "지금 상태에서 어떤 액션이 가능한지"가 자주 바뀌는 도메인이라면, 이 로직을 모든 클라이언트에 중복 구현하는 대신 서버가 응답에 담아 내려주는 것이 유지보수 비용을 크게 줄입니다. 또한 여러 서드파티 개발사가 하나의 플랫폼 API를 호출하는 오픈 플랫폼(예: 결제 게이트웨이, 클라우드 API)에서는, 클라이언트가 URL 스킴 변경에 영향받지 않도록 하이퍼미디어 링크를 제공하는 것이 실질적인 이점이 됩니다. 핵심은 "이론적으로 더 순수한 REST"가 아니라 "클라이언트 다양성과 상태 전이 복잡도가 이 비용을 정당화하는가"로 판단하는 것입니다.

3. REST vs GraphQL vs gRPC — 프로토콜 선택 기준

배경 개념: "더 좋은 프로토콜"은 없고 "적합한 통신 패턴"만 있다

REST, GraphQL, gRPC는 우열 관계가 아니라 서로 다른 문제를 풀도록 설계된 도구입니다. REST는 리소스 단위 CRUD와 캐싱에 강하고, GraphQL은 클라이언트가 필요한 데이터 모양을 스스로 정의할 수 있게 해 오버페칭/언더페칭 문제를 풀며, gRPC는 서비스 간(내부) 고성능 통신에 최적화되어 있습니다. 면접관은 "무엇이 최신 기술인가"가 아니라 "이 상황에 어떤 통신 패턴의 trade-off가 맞는가"를 봅니다.

면접 질문

"모바일 앱 화면 하나를 그리려고 REST API를 5번 호출해야 하는 상황입니다(오버페칭/언더페칭 문제). GraphQL로 전환을 제안받았습니다. 실제로 전환하시겠어요? 어떤 조건에서 전환이 타당한가요?"

면접관이 보는 신호

GraphQL이 오버페칭 문제를 푸는 것은 알지만, 캐싱 난이도·N+1 쿼리·복잡도 증가라는 대가까지 균형 있게 아는지를 봅니다.

프로토콜별 비교

항목 REST GraphQL gRPC
데이터 형태 결정권 서버가 고정된 응답 스키마를 정의 클라이언트가 쿼리로 필요한 필드만 선택 서버가 Protobuf 스키마로 고정 정의
오버/언더페칭 화면이 필요한 데이터와 API 응답이 안 맞으면 여러 번 호출하거나 불필요한 필드까지 받음 한 번의 요청으로 여러 리소스를 원하는 필드만 조합해 받음 해당 문제 자체가 설계 대상이 아님(서비스 간 통신 목적)
캐싱 HTTP 캐싱(CDN, 브라우저, 리버스 프록시)을 자연스럽게 활용 쿼리마다 응답 모양이 달라 URL 기반 캐싱이 어려움, 별도 캐싱 계층 필요 내부 통신이라 HTTP 캐싱 대상이 아닌 경우가 많음
성능 특성 텍스트 기반(JSON), 사람이 읽기 쉬움 단일 엔드포인트로 유연하지만 리졸버 설계에 따라 N+1 쿼리 위험 Protobuf 바이너리 직렬화 + HTTP/2 멀티플렉싱으로 REST 대비 낮은 지연/높은 처리량
적합 상황 외부 공개 API, 리소스 단위 CRUD가 명확한 도메인 다양한 클라이언트(웹/모바일/워치)가 각기 다른 데이터 모양을 원하는 프론트엔드 대면 API MSA 내부 서비스 간 고빈도·저지연 통신, 스트리밍
주요 리스크 화면 다양화에 따라 엔드포인트가 계속 늘어나는 유지보수 부담 리졸버 하나가 내부적으로 N번 DB를 호출하는 N+1 문제, 악의적 쿼리로 인한 과도한 부하(query complexity 공격) 사람이 직접 브라우저로 호출/디버깅하기 어려움, 브라우저 네이티브 지원 부족

결정 순서

1
통신 상대가 누구인지 먼저 봅니다. 서비스 내부 통신(서버-서버)이면 gRPC가 유력 후보입니다. 사람이 사용하는 프론트엔드/서드파티가 상대라면 REST 또는 GraphQL을 검토합니다.
2
클라이언트 화면의 다양성을 봅니다. 웹/iOS/Android/워치처럼 화면마다 필요한 데이터 조합이 크게 다르고, 이로 인해 REST 엔드포인트가 계속 늘어나는(BFF 패턴 없이는 감당 안 되는) 상황이면 GraphQL이 유력합니다.
3
캐싱 요구를 확인합니다. CDN/HTTP 캐싱이 성능에 중요한 공개 API라면 REST의 캐싱 친화성이 큰 이점입니다. GraphQL은 캐싱을 포기하는 대신 유연성을 얻는 trade-off입니다.
4
GraphQL 채택 시 N+1 문제 대응 계획이 있는지 확인합니다. DataLoader 패턴(배치 로딩)이나 쿼리 복잡도 제한(depth/cost limiting) 없이 GraphQL을 도입하면 오히려 REST보다 DB 부하가 커질 수 있습니다.
강한 답변

"저희 앱은 홈 화면 하나에 주문, 추천, 배너, 사용자 정보 4개 리소스를 조합해서 보여줘야 하고, 태블릿/모바일/워치 세 클라이언트가 각각 다른 조합을 필요로 합니다. REST로는 화면별 BFF를 계속 늘리거나 클라이언트가 4번 호출해야 하는데, 이 부담이 GraphQL 도입 비용(캐싱 재설계, N+1 대응)보다 크다고 판단해 전환했습니다. 전환 시 DataLoader로 리졸버의 N+1 쿼리를 배치 처리하고, 쿼리 depth를 5로 제한해 악의적으로 깊은 중첩 쿼리가 서버를 마비시키지 못하게 했습니다. 반면 외부 파트너에게 공개하는 결제 조회 API는 캐싱과 표준 HTTP 시맨틱이 중요해서 REST를 유지했습니다."

약한 답변

"GraphQL이 더 유연하고 요즘 트렌드니까 전체 API를 GraphQL로 바꾸겠습니다."

약한 답변이 약한 이유: "유연함"과 "트렌드"만 근거로 들었고, 캐싱을 포기하는 대가, N+1 문제 대응 계획, 그리고 API 전체를 하나의 프로토콜로 통일해야 한다는 암묵적 가정 자체를 검증하지 않았습니다.

Senior Follow-up

"GraphQL을 도입한 지 6개월 후, 특정 쿼리 하나가 DB에 순간적으로 수백 개의 쿼리를 발생시켜 장애가 났습니다. 원인과 재발 방지책은 무엇인가요?"

모범 추론: 전형적인 N+1 쿼리 문제입니다. 예를 들어 주문 목록 100개를 조회하는 쿼리에서 각 주문마다 사용자 정보를 리졸빙하면, 리졸버가 순진하게 구현되어 있으면 사용자 조회 쿼리가 100번 발생합니다. 1) 즉시 조치: 문제가 된 쿼리 패턴을 식별해 DataLoader 같은 배치 로딩(같은 틱 안의 요청을 모아 한 번의 IN 쿼리로 처리)으로 리졸버를 수정합니다. 2) 재발 방지: 쿼리 복잡도 분석기(query cost analysis)를 도입해, 예상 리졸버 호출 횟수가 임계치를 넘는 쿼리는 실행 전에 거부합니다. 3) 관측성 보강: 리졸버 단위로 실행 시간과 DB 호출 횟수를 트레이싱에 남겨, 새로운 쿼리 패턴이 배포될 때 이상 패턴을 사전에 감지합니다. 4) 조직적 대응: 프론트엔드가 임의의 쿼리를 자유롭게 작성할 수 있게 두는 대신, persisted query(사전에 등록된 쿼리만 허용)로 전환해 예측 불가능한 쿼리 자체를 원천 차단하는 것도 대규모 서비스에서는 일반적인 절충안입니다.

4. API 버전 관리 전략

배경 개념: 버전 관리는 "언젠가 반드시 필요한 것"이 아니라 "가능한 한 피해야 하는 것"

버전 관리 전략을 묻는 질문의 함정은, 많은 지원자가 "URL에 /v2를 넣으면 된다"로 끝낸다는 점입니다. 하지만 새 버전을 만드는 순간 두 버전을 동시에 운영·테스트·모니터링해야 하는 비용이 발생하고, 클라이언트는 새 버전으로 마이그레이션할 유인이 없으면 옛 버전에 영원히 머무릅니다. 시니어에게 기대되는 답은 "버전을 어떻게 표기할까"가 아니라 "애초에 버전을 새로 만들 필요를 어떻게 최소화하고, 만들어야 한다면 옛 버전을 어떻게 은퇴시킬까"입니다.

면접 질문

"API 버전 관리 방식에는 URL 경로(/v2/orders), 헤더(Accept: application/vnd.api.v2+json), 쿼리 파라미터 방식이 있습니다. 어떤 방식을 선택하시겠고, 그 이유는 무엇인가요?"

면접관이 보는 신호

방식의 이름을 나열하는지, 아니면 캐싱·라우팅·클라이언트 구현 난이도 관점에서 실질적 차이를 설명하는지를 봅니다.

버전 관리 방식 비교

방식 예시 장점 단점
URL 경로 GET /v2/orders/123 가장 직관적, 브라우저에서 바로 확인 가능, 라우팅 규칙(리버스 프록시/게이트웨이)으로 버전별 트래픽 분리가 쉬움 엄밀히는 "같은 리소스의 다른 표현"이 아니라 "다른 리소스"가 되어 REST 원칙과는 약간 어긋남
커스텀 헤더 / Accept 헤더 Accept: application/vnd.myapi.v2+json URL(리소스 식별자)은 그대로 유지되어 REST 철학에 더 부합, 캐시 키에 버전을 자연스럽게 포함 가능(Vary 헤더) 클라이언트/문서화 도구가 헤더 기반 라우팅을 직관적으로 다루기 어려움, 브라우저 주소창으로 테스트 불가
쿼리 파라미터 GET /orders/123?version=2 구현이 간단 캐싱 키에 쿼리 파라미터까지 고려해야 해 복잡, 파라미터 생략 시 기본 버전이 암묵적으로 결정되는 문제

버전을 새로 만들기 전에 먼저 시도할 것들

1
가산적(additive) 변경으로 해결되는지 먼저 확인합니다. 필드 추가, 새 엔드포인트 추가는 대부분 버전을 올리지 않고도 가능합니다. 아래 5절의 하위 호환성 규칙을 먼저 점검합니다.
2
정말 breaking change가 불가피한지 확인합니다. 필드 삭제, 타입 변경, 필수값 추가처럼 기존 클라이언트를 반드시 깨뜨리는 변경인지 재확인합니다.
3
새 버전을 만든다면 은퇴 계획을 함께 세웁니다. "v1을 언제까지 지원할지"에 대한 Deprecation 정책과 sunset 날짜를 API 응답 헤더(Deprecation, Sunset)와 문서에 명시합니다.
4
버전별 트래픽을 계측합니다. v1 호출량이 0에 가까워질 때까지 모니터링하고, 특정 클라이언트가 마이그레이션을 안 하고 있다면 직접 연락해 마이그레이션을 독려합니다. "언젠가 지워도 되겠지"라는 추측이 아니라 실측 데이터로 은퇴 시점을 결정합니다.
강한 답변

"저희는 URL 경로 버전(/v2)을 씁니다. API 게이트웨이 레벨에서 버전별로 트래픽을 라우팅하고 별도 배포 단위로 분리하기가 가장 쉽고, 사내 API 문서 도구도 경로 기반을 잘 지원하기 때문입니다. 다만 버전을 실제로 올리기 전에 항상 '가산적 변경으로 해결 가능한가'를 먼저 검토합니다. v2를 만들 때는 v1의 Deprecation 헤더에 sunset 날짜를 명시하고, v1 호출량을 대시보드로 추적해서 특정 파트너사가 마이그레이션을 안 하고 있으면 직접 연락했습니다. v1 트래픽이 3개월간 0에 수렴한 뒤에야 실제로 v1 코드를 제거했습니다."

약한 답변

"URL에 v1, v2 이렇게 붙이면 됩니다. 새 버전 배포하면 옛날 버전은 나중에 지우면 되죠."

약한 답변이 약한 이유: 버전 표기 방법만 답했을 뿐, 버전을 왜 최소화해야 하는지, 옛 버전을 언제·어떻게 안전하게 은퇴시킬지에 대한 계획이 전혀 없습니다. "나중에 지우면 된다"는 클라이언트 마이그레이션을 강제할 수단이 없다는 현실을 무시한 답입니다.

Senior Follow-up

"v1 sunset 날짜가 다가왔는데, 여전히 트래픽의 5%가 v1을 호출하고 있고 그 클라이언트가 누구인지도 파악이 안 됩니다. 어떻게 하시겠어요?"

모범 추론: 강제로 sunset 날짜에 차단하면 알 수 없는 클라이언트가 갑자기 장애를 겪게 되므로, 단계적 접근이 필요합니다. 1) 먼저 User-Agent, API Key, 호출 IP 등으로 남은 v1 호출자를 식별합니다. API Key 없이 익명 호출을 허용했다면 이번 기회에 식별 가능한 인증 체계로 전환합니다. 2) 즉시 차단 대신 점진적 제한(rate limiting을 강화하거나 일부 트래픽만 의도적으로 느리게 응답)을 걸어 클라이언트가 문제를 자연스럽게 알아채고 팀에 문의하도록 유도합니다. 3) 응답에 Warning 헤더나 로그 레벨 알림을 추가해 "이 API는 곧 종료됩니다"라는 신호를 명시적으로 보냅니다. 4) 최종적으로도 남아있다면, 비즈니스 임팩트와 보안/유지보수 부담을 비교해 언제까지 실제로 sunset을 강행할지 이해관계자와 협의합니다. 기술적으로 "맞다/틀리다"의 문제가 아니라 비즈니스 리스크 판단이 개입되는 지점입니다.

5. 하위 호환성 — Breaking Change를 만들지 않고 진화하기

배경 개념: "필드 추가는 항상 안전하다"는 규칙에는 함정이 있다

흔히 "필드를 추가하는 것은 안전하고, 필드를 지우거나 타입을 바꾸는 것은 위험하다"고 배웁니다. 이것은 출발점으로는 맞지만, 클라이언트가 실제로 파싱하는 방식에 따라 예외가 생깁니다. 예를 들어 클라이언트가 strict schema validation(알 수 없는 필드가 오면 에러)을 쓰고 있다면 필드 추가조차 breaking change가 됩니다. 시니어는 "일반 규칙"과 "우리 클라이언트의 실제 파싱 방식"을 함께 고려해 안전성을 판단해야 합니다.

면접 질문

"주문 응답에 discountAmount 필드를 새로 추가하려고 합니다. 이건 안전한 변경인가요? 안전하지 않을 수 있는 경우가 있다면 어떤 경우인가요?"

면접관이 보는 신호

"필드 추가는 무조건 안전하다"고 규칙만 외우는지, 아니면 클라이언트 구현 방식에 따라 달라진다는 것을 예외 케이스로 설명할 수 있는지를 봅니다.

Breaking / Non-breaking 변경 판단표

변경 유형 일반적으로 예외가 생기는 조건
새 필드 추가 Non-breaking 클라이언트가 strict schema validation(엄격 스키마 검증)을 쓰면 알 수 없는 필드로 에러 발생 가능
필드 삭제 Breaking 없음 — 항상 breaking (해당 필드를 참조하는 클라이언트가 하나라도 있으면 깨짐)
필드 이름 변경 Breaking 없음 — 실질적으로 "삭제 + 추가"와 동일
필드 타입 변경(문자열 → 숫자 등) Breaking 느슨한 타입의 언어(JS)는 우연히 동작할 수 있으나, 강타입 클라이언트(모바일 네이티브)는 파싱 자체가 실패
선택적(optional) 필드를 필수(required)로 변경 Breaking 없음 — 기존 클라이언트가 그 필드 없이 보내던 요청이 전부 실패
Enum 값 추가 Breaking 가능성 있음 클라이언트가 switch문으로 모든 enum 값을 처리하고 default 분기가 없으면, 새 값이 왔을 때 처리 누락 또는 크래시
에러 응답의 상태 코드 변경 Breaking 클라이언트가 특정 상태 코드로 재시도/폴백 로직을 분기하고 있다면 그 로직이 깨짐

안전하게 진화시키는 실무 원칙

1
필드를 지우지 말고 Deprecated로 표시한 채 당분간 유지합니다. 문서에 "이 필드는 v2에서 제거 예정" 명시하고, 새 필드를 병행 제공해 클라이언트가 자기 속도로 전환하게 합니다.
2
Enum에는 항상 UNKNOWN 폴백 값을 두도록 클라이언트 가이드에 명시합니다. 서버가 새 enum 값을 추가해도, 클라이언트가 모르는 값은 안전하게 "미지의 상태"로 처리하도록 계약합니다.
3
계약 테스트(Contract Test, 예: Pact)를 CI에 넣습니다. API를 변경하는 PR이 기존에 합의된 클라이언트 계약을 깨는지 배포 전에 자동으로 검증합니다. "리뷰어가 눈으로 확인"하는 방식은 반드시 놓치는 경우가 생깁니다.
4
Consumer-Driven Contract 방식으로 클라이언트의 실제 파싱 방식을 파악합니다. "필드 추가는 안전하다"는 일반론에 기대지 말고, 실제 소비자(모바일 앱, 파트너사)가 strict validation을 쓰는지 사전에 확인합니다.
강한 답변

"일반적으로 필드 추가는 non-breaking이지만, 저희 파트너사 중 하나는 JSON Schema strict validation을 클라이언트에서 쓰고 있어서 알 수 없는 필드가 오면 요청 자체를 실패시킵니다. 그래서 필드를 추가하기 전에 계약 테스트(Pact)로 각 소비자별 계약을 확인하고, 문제가 될 소비자에게는 사전 공지 후 그들의 스키마를 업데이트하고 나서 필드를 배포했습니다. 또한 discountAmount가 없을 때의 기본값(0)도 문서에 명시해 옛 클라이언트가 이 필드를 못 봐도 로직에 문제가 없게 했습니다."

약한 답변

"필드 추가는 항상 안전하니까 그냥 배포하면 됩니다."

약한 답변이 약한 이유: 일반 규칙을 예외 없이 적용했습니다. 실제 소비자의 파싱 방식(strict validation 여부)을 확인하지 않은 채 "항상 안전하다"고 단정하는 것은 실무에서 그대로 장애로 이어질 수 있습니다.

Senior Follow-up

"결제 상태 enum에 REFUND_PENDING이라는 새 값을 추가했더니, 오래된 버전의 모바일 앱에서 크래시가 발생했습니다. 왜 이런 일이 생겼고, 앞으로 이런 사고를 어떻게 막으시겠어요?"

모범 추론: 원인은 모바일 클라이언트가 결제 상태 enum을 switch문으로 처리하면서 default 분기(알 수 없는 값에 대한 폴백)를 두지 않았기 때문입니다. 서버 입장에서는 "값 추가"였지만 클라이언트 입장에서는 처리되지 않은 케이스가 발생한 breaking change였습니다. 재발 방지책: 1) API 계약 문서에 "이 enum은 향후 값이 추가될 수 있으며, 클라이언트는 반드시 알 수 없는 값에 대한 기본 처리(fallback UI)를 구현해야 한다"는 open enum 정책을 명시합니다. 2) 모바일 SDK 레벨에서 enum 파싱 시 알 수 없는 값을 자동으로 UNKNOWN으로 매핑하는 공통 유틸을 제공해, 개별 개발자의 실수에 의존하지 않게 합니다. 3) 신규 enum 값을 실제 사용자에게 노출하기 전에, 오래된 클라이언트 버전 비율이 낮아질 때까지는 서버에서 새 값 대신 기존 값으로 매핑해 내려주는 점진적 롤아웃을 고려합니다. 핵심은 "서버가 무엇을 추가했는가"가 아니라 "오래된 클라이언트가 그 추가에 어떻게 반응하는가"까지 설계 범위에 포함시키는 것입니다.

6. 종합 시나리오 — 사내 결제 API의 3년 후 진화

면접 질문

"3년 전 만든 결제 API를 현재 20개 이상의 내부 서비스와 5개 파트너사가 호출하고 있습니다. 새로운 결제 수단(간편결제)을 추가하고, 응답 구조도 더 명확하게 재설계하고 싶습니다. 어떻게 접근하시겠어요?"

이 질문은 오늘 배운 REST 성숙도, 프로토콜 선택, 버전 관리, 하위 호환성을 하나의 실전 설계로 종합하는 능력을 봅니다.

1
새 결제 수단 추가는 가산적 변경으로 먼저 시도합니다. 기존 paymentMethod enum에 "EASY_PAY" 값을 추가하되, 클라이언트가 open enum 정책을 따르는지 계약 테스트로 확인한 뒤 배포합니다. 이 변경만으로는 새 버전이 필요 없습니다.
2
응답 구조 재설계는 진짜 breaking change이므로 새 버전이 불가피합니다. 20개 내부 서비스와 5개 파트너사가 동시에 마이그레이션할 수 없으므로, v1과 v2를 병행 운영할 기간을 산정합니다.
3
내부 서비스와 외부 파트너사를 다르게 취급합니다. 내부 서비스는 팀 간 조율로 마이그레이션을 강제할 수 있어 기한을 짧게 잡고, 외부 파트너사는 계약과 공지 기간이 필요해 훨씬 긴 sunset 기간(예: 6개월~1년)을 둡니다.
4
버전 방식은 URL 경로(/v2/payments)로 통일해, API 게이트웨이에서 버전별 트래픽을 라우팅하고 v1/v2 각각의 에러율·지연시간을 독립적으로 모니터링합니다.
5
v1 호출 트래픽을 대시보드로 계측하고, 특정 임계치 이하로 떨어지기 전까지는 v1 제거를 시도하지 않습니다. 남은 호출자를 API Key 기준으로 식별해 개별적으로 마이그레이션을 독려합니다.
시니어가 반드시 언급해야 하는 것: "재설계"라는 단어가 나왔다고 무조건 새 버전부터 만드는 것이 아니라, 어떤 부분이 정말 breaking이고 어떤 부분은 가산적으로 해결되는지 먼저 분리하는 사고가 신호입니다. 또한 "우리 서비스"와 "외부 파트너사"에 같은 마이그레이션 기한을 적용하지 않는다는 것 — 즉 API 소비자군에 따라 버전 은퇴 전략을 다르게 설계한다는 것이 실무 경험이 있는 답변의 특징입니다.

7. 자기 점검 Rubric

각 항목을 솔직하게 채점하세요. ●●●=잘 설명할 수 있음, ●●○=부분적으로 앎, ●○○=어렴풋이 앎, ○○○=모름

Richardson Maturity Model의 각 단계를 설명하고, 우리 API가 어느 단계인지 스스로 진단할 수 있다
HATEOAS(Level 3)가 실무에서 잘 쓰이지 않는 이유와, 그럼에도 가치 있는 상황을 함께 설명할 수 있다
REST/GraphQL/gRPC를 캐싱, 클라이언트 다양성, 통신 상대(내부/외부) 기준으로 구분해 선택할 수 있다
GraphQL의 N+1 문제와 DataLoader/쿼리 복잡도 제한 같은 대응책을 설명할 수 있다
URL/헤더/쿼리 파라미터 버전 관리 방식의 실질적 차이와, 버전을 최소화해야 하는 이유를 설명할 수 있다
"필드 추가는 항상 안전하다"는 규칙의 예외(strict validation, enum switch 등)를 구체적으로 설명할 수 있다
Deprecation 정책, sunset 날짜, 계약 테스트를 활용한 안전한 API 은퇴 절차를 설계할 수 있다
내부 소비자와 외부 파트너사를 다르게 취급하는 버전 마이그레이션 전략을 설계할 수 있다
