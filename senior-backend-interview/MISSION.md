# 백엔드 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 시니어 백엔드 개발자로서 기술 인터뷰에서 **실무 판단력**을 입증한다.

단순 CS 개념 암기나 교과서적 답변이 아닌, 실제 운영 경험과 설계 판단력을 바탕으로 다음을 시연할 수 있어야 한다.

- 복잡한 시스템을 **구조적으로 설계**하고 trade-off를 명확히 설명
- 규모·비용·팀 역량을 고려한 **현실적 아키텍처 선택**
- 장애, 데이터 불일치, 성능 저하 등 **실전 문제 대응 경험** 공유
- 기술적 결정의 영향을 비즈니스 언어로 **번역하여 설명**
- 주니어를 이끌고 기술 문화를 만드는 **리더십 경험** 제시

## 평가 대상 역량

| 역량 | 세부 항목 |
|------|----------|
| 시스템 설계 | 요구사항 분석, 용량 추정, 컴포넌트 설계, 데이터 모델 |
| 아키텍처 | MSA vs Monolith, 이벤트 드리븐, CQRS, Saga |
| 확장성 | 수평 확장, 샤딩, 캐싱 전략, 로드밸런싱 |
| 신뢰성 | CAP, 서킷 브레이커, Rate Limiting, Bulkhead |
| 데이터 | 트랜잭션, 정규화/역정규화, 인덱싱, 쿼리 최적화 |
| 동시성 | 락 전략, 낙관적/비관적 동시성, 멱등성, 분산 락 |
| 분산 시스템 | 일관성 모델, 분산 트랜잭션, 메시지 큐 |
| API 설계 | REST, GraphQL, gRPC, 버전 관리, 하위 호환성 |
| 보안 | AuthN/AuthZ, OWASP, 암호화, 취약점 |
| 관측성 | 로깅, 메트릭, 트레이싱, SLO/SLA |
| 장애 대응 | 장애 탐지, RCA, 포스트모텀, 복구 전략 |
| 배포 | CI/CD, 블루-그린, 카나리, 롤백 |
| 기술 리딩 | 설계 리뷰, 코드 리뷰, 기술 부채 관리 |
| 커뮤니케이션 | 기술-비즈니스 번역, 갈등 해결, 의사결정 |

## 성공 기준

- 인터뷰어가 "follow-up"을 계속 던져도 당황하지 않고 깊이 있게 답변
- 주어진 시나리오에서 "왜 그 선택인가"를 비용·운영·팀 관점에서 정당화
- 자신의 실패 경험을 솔직하게 공유하며 배운 점을 구체적으로 설명
- "당신이라면 지금 당장 어떻게 할 것인가"에 즉답 가능
