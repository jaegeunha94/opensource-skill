Typing 심화
Protocol, Generic, TypeVar, Literal, overload

파이썬 경력자 Interview · 2026-07-08 · 약 70–90분 소요

왜 면접에서 타입 시스템 심화를 묻는가

def f(x: int) -> str: 정도의 타입 힌트는 대부분의 경력자가 이미 쓰고 있다. 면접관이 Protocol, Generic, TypeVar, overload를 묻는 이유는 단순 문법 암기가 아니라, 타입 힌트가 런타임에 아무 영향도 주지 않는다는 사실을 이해한 상태에서, 그럼에도 왜 정적 타입 검사기가 대규모 코드베이스에서 버그를 미리 잡아내는지 설명할 수 있는가를 보기 위해서다. 이건 "타입을 쓸 줄 아는가"가 아니라 "타입 시스템을 설계 도구로 쓸 줄 아는가"의 문제다.

실무에서 이 감각이 갈리는 지점은 명확하다. 팀 규모가 커지고 코드베이스가 수만 줄을 넘어가면, "이 함수에 어떤 타입을 넣어야 하는지"를 문서나 구현 코드를 직접 읽어야만 알 수 있는 코드와, IDE 자동완성과 mypy가 바로 알려주는 코드의 유지보수 비용은 완전히 다르다. 동시에 타입을 과도하게 엄격하게 쓰면 오히려 유연성을 해치는 상황도 있다 — 이 trade-off를 아는가가 경력자와 초급자를 가른다.

실무 연결
"덕 타이핑(duck typing)을 즐겨 쓰는 파이썬 팀에서, 외부에서 들어오는 객체가 특정 인터페이스를 만족하는지 정적으로 검증하고 싶습니다. 상속 구조를 강제하지 않고 어떻게 하시겠습니까?" — 이 질문에 Protocol을 구조적 타이핑(structural typing) 개념과 함께 설명할 수 있는가가 이 레슨의 핵심이다.

전제 개념 ① — 타입 힌트는 "주석"이지 "강제"가 아니다

가장 먼저 짚어야 할 전제는, 파이썬 인터프리터는 타입 힌트를 실행 중에 검사하지 않는다는 사실이다. def f(x: int) -> str: return x처럼 반환 타입이 틀려도 인터프리터는 아무런 에러 없이 그대로 실행한다. 타입 힌트는 CPython 입장에서 그냥 함수 객체의 __annotations__ 딕셔너리에 저장되는 메타데이터일 뿐이다. 

def add(a: int, b: int) -> int:
    return str(a) + str(b)  # 타입 어긴 반환값이지만 런타임 에러 없음

print(add(1, 2))       # "12" — int를 기대했지만 str이 그대로 반환됨
print(add.__annotations__)  # {'a': <class 'int'>, 'b': <class 'int'>, 'return': <class 'int'>}

그렇다면 타입 힌트는 왜 쓸모가 있는가? 답은 별도의 정적 분석 도구(mypy, pyright, pyre 등)가 소스코드를 실행하지 않고 읽기만 해서 타입 불일치를 찾아낸다는 데 있다. 즉 타입 검사는 런타임이 아니라 "코드를 작성하는 시점"에 CI 파이프라인이나 IDE에서 이루어지는 별도의 단계다. 이 구분을 못 하면 "타입 힌트를 썼는데 왜 프로덕션에서 타입 에러가 났나요"라는 질문에 제대로 답하지 못한다.

런타임 강제가 필요하면 다른 도구가 필요하다
입력값을 실제로 런타임에 검증하고 싶다면 pydantic이나 수동 isinstance 체크가 필요하다. typing 모듈의 타입 힌트 자체는 "문서화 + 정적 분석"을 위한 것이지 "런타임 유효성 검사"를 위한 것이 아니라는 걸 명확히 구분해서 답하는 게 중요하다.

면접 질문 ① — 상속 없이 "이 객체가 특정 인터페이스를 만족한다"를 어떻게 타입으로 표현하나요?

close() 메서드만 있으면 되는 함수를 작성하려고 합니다. 파일 객체든, 커스텀 리소스 클래스든 상관없이 close()가 있는 어떤 객체든 받고 싶습니다. 공통 부모 클래스를 강제로 만들지 않고 이걸 타입으로 어떻게 표현하나요?

인터뷰어가 평가하는 신호

  • 덕 타이핑(duck typing)과 구조적 타이핑(structural typing)의 관계를 아는가
  • Protocol이 명목적 타이핑(nominal typing, 상속 기반)과 어떻게 다른지 설명할 수 있는가
  • 이걸 언제 ABC(추상 베이스 클래스) 대신 써야 하는지 판단할 수 있는가

답변 구조 — 생각의 순서

  1. 문제 정의: 상속 강제 없이 "이 메서드가 있는가"만으로 타입을 표현하고 싶다
  2. 파이썬의 기존 방식: 런타임에서는 원래 덕 타이핑으로 이걸 자연스럽게 해왔음
  3. 정적 타입 세계의 대응: typing.Protocol이 이 덕 타이핑을 정적으로 표현하는 도구
  4. 구현: Protocol을 상속한 클래스에 메서드 시그니처만 선언
  5. 핵심 성질: 이 프로토콜을 명시적으로 상속하지 않아도, 같은 메서드를 가진 어떤 클래스든 타입 검사를 통과함(구조적 타이핑)
from typing import Protocol

class Closeable(Protocol):
    def close(self) -> None: ...

def cleanup(resource: Closeable) -> None:
    resource.close()

class DatabaseConnection:  # Closeable을 상속하지 않았다!
    def close(self) -> None:
        print("연결 종료")

cleanup(DatabaseConnection())  # mypy 통과 — close()만 있으면 됨(구조적 타이핑)
강한 답변

"typing.Protocol을 씁니다. 파이썬은 원래 런타임에서 덕 타이핑으로 동작합니다 — '오리처럼 걷고 오리처럼 운다면 오리로 취급한다'는 원칙이죠. 어떤 객체가 close() 메서드를 갖고 있기만 하면, 그 객체가 어떤 클래스를 상속했는지는 신경 쓰지 않고 그냥 호출합니다.

Protocol은 이 덕 타이핑을 정적 타입 검사 세계로 옮긴 것입니다. Protocol을 상속한 클래스에 close(self) -> None이라는 시그니처만 선언해두면, 이 프로토콜을 명시적으로 상속하지 않은 어떤 클래스든 같은 시그니처의 메서드만 있으면 타입 검사를 통과합니다. 이걸 구조적 타이핑(structural typing)이라고 부르고, 클래스 상속 관계로 타입 호환성을 판단하는 명목적 타이핑(nominal typing)과 대비됩니다.

이게 실무적으로 중요한 이유는, 외부 라이브러리의 클래스나 이미 존재하는 코드를 수정할 수 없는 상황에서도 타입 안전성을 확보할 수 있기 때문입니다. ABC(추상 베이스 클래스)로 강제하려면 그 클래스가 ABC를 상속하도록 소스를 고쳐야 하지만, Protocol은 소스를 건드리지 않고도 '이 인터페이스를 만족하는가'만 검사할 수 있습니다."

약한 답변

"인터페이스 같은 겁니다. Protocol을 상속받게 하면 됩니다."

왜 약한가: "인터페이스 같은 것"이라는 비유만 있고, 정작 Protocol의 핵심 차별점인 "상속하지 않아도 통과한다"는 구조적 타이핑 성질을 언급하지 않았다. 오히려 "Protocol을 상속받게 하면 된다"는 표현은 ABC와 혼동하고 있다는 신호로 읽힌다.

전제 개념 ② — Generic은 "타입을 매개변수로 받는 함수"라고 생각하면 쉽다

일반 함수는 값을 매개변수로 받는다. def add(a, b)에서 a, b는 호출 시점에 결정되는 값이다. 제네릭(Generic)은 같은 아이디어를 타입 수준으로 옮긴 것이다 — "이 클래스나 함수가 다루는 타입은 지금 정하지 않고, 사용하는 시점에 정한다"는 뜻이다. list[int], list[str]이 서로 다른 타입으로 취급되는 것도 list가 제네릭이기 때문이다.

이 "타입 매개변수"를 표현하는 게 TypeVar다. 중요한 건 TypeVar 하나가 함수 안에서 여러 곳에 쓰이면, 그 자리들은 모두 같은 타입이어야 한다는 제약을 건다는 점이다. 

from typing import TypeVar

T = TypeVar("T")

def first(items: list[T]) -> T:
    return items[0]

first([1, 2, 3])       # T=int로 추론 → 반환 타입 int
first(["a", "b"])       # T=str로 추론 → 반환 타입 str
# first([1, "a"])  # mypy 오류: list의 원소 타입이 섞여 T를 하나로 확정할 수 없음

만약 T 없이 def first(items: list) -> object:로 썼다면, mypy는 반환값을 그냥 object로만 알기 때문에 first([1,2,3])[0] + 1같은 코드에서 "int 연산인지 확신할 수 없다"는 경고를 낸다. TypeVar는 입력 타입과 출력 타입 사이의 관계를 정적 검사기에게 알려주는 역할을 한다.

TypeVar 제약 방식 3가지
# 1. 무제약 — 어떤 타입이든 가능
T = TypeVar("T")

# 2. bound — 특정 타입의 하위 타입만 허용
NumT = TypeVar("NumT", bound=int | float)

# 3. constrained — 나열한 타입 중 정확히 하나만 허용(하위 타입 불가)
StrOrBytes = TypeVar("StrOrBytes", str, bytes)
Python 3.12+에서는 더 간결한 문법이 생겼다
Python 3.12부터 PEP 695로 def first[T](items: list[T]) -> T:처럼 TypeVar를 별도로 선언하지 않고 함수/클래스 정의에 바로 타입 매개변수를 쓸 수 있다. 다만 아직 많은 프로덕션 코드베이스가 3.12 미만을 지원해야 하므로, 두 문법을 모두 알고 있어야 한다고 답하면 실무 경험이 있다는 신호가 된다.

면접 질문 ② — 제네릭 클래스는 언제 설계하고, 어떤 문제를 막아주나요?

캐시 클래스를 만드는데, 문자열 키를 정수로 매핑하는 캐시도 있고, 정수 키를 객체로 매핑하는 캐시도 있습니다. 매번 다른 클래스를 만들지 않고 하나의 Cache 클래스로 이걸 타입 안전하게 표현하려면 어떻게 하시겠습니까?

인터뷰어가 평가하는 신호

  • Generic[K, V]처럼 여러 타입 매개변수를 쓰는 클래스를 설계할 수 있는가
  • 제네릭이 없을 때 Any를 쓰면 어떤 안전성을 잃는지 설명할 수 있는가
  • "타입 안전성"과 "코드 재사용"이라는 두 가지 이점을 구분해서 말할 수 있는가
from typing import Generic, TypeVar

K = TypeVar("K")
V = TypeVar("V")

class Cache(Generic[K, V]):
    def __init__(self) -> None:
        self._store: dict[K, V] = {}

    def get(self, key: K) -> V | None:
        return self._store.get(key)

    def put(self, key: K, value: V) -> None:
        self._store[key] = value

user_cache: Cache[int, str] = Cache()
user_cache.put(1, "alice")
# user_cache.put("x", "alice")  # mypy 오류: key는 int여야 함
강한 답변

"Generic[K, V]를 상속받아 키 타입과 값 타입을 각각 타입 매개변수로 받는 Cache 클래스를 만듭니다. 사용하는 쪽에서 Cache[int, str]처럼 구체 타입을 지정하면, 그 인스턴스에 대해서는 put/get의 키·값 타입이 고정되어 정적 검사기가 잘못된 타입의 키나 값을 넣는 걸 컴파일 타임에 잡아줍니다.

만약 이걸 Any로 대충 처리하면(dict[Any, Any]), 코드는 똑같이 동작하지만 타입 검사기가 아무것도 검증하지 않는 상태가 됩니다. Any는 '타입을 모른다'가 아니라 '타입 검사를 이 지점에서 끄겠다'는 의미에 가깝기 때문에, 잘못된 타입의 값을 넣어도 mypy가 조용히 통과시켜버립니다. 결과적으로 런타임까지 가서야 TypeError나 논리적 버그로 발견되는 문제를 정적 검사 단계에서 잡을 기회를 스스로 포기하는 셈입니다.

추가로, 제네릭 클래스는 코드 재사용도 제공합니다. IntStrCache, StrObjectCache처럼 조합마다 클래스를 따로 만들지 않고, 하나의 구현으로 모든 키·값 조합을 타입 안전하게 커버할 수 있습니다."

약한 답변

"그냥 dict를 쓰면 됩니다. 파이썬은 동적 타입 언어라 딱히 타입을 안 정해도 상관없습니다."

왜 약한가: 런타임 동작만 보면 틀린 말은 아니지만, 질문의 핵심인 "타입 안전하게"를 무시했다. 이 답변은 대규모 코드베이스에서 정적 타입 검사가 왜 가치가 있는지에 대한 이해가 없다는 신호로 읽힌다.

면접 질문 ③ — 입력 타입에 따라 반환 타입이 달라지는 함수를 어떻게 타입으로 표현하나요?

process(x) 함수가 있는데, x가 str이면 str을 반환하고, x가 bytes면 bytes를 반환합니다. Union[str, bytes]로 반환 타입을 표기하면 정확한가요? 더 정확하게 표현할 방법이 있나요?

인터뷰어가 평가하는 신호

  • Union으로 반환 타입을 뭉뚱그리면 어떤 정보 손실이 생기는지 아는가
  • @overload가 런타임 동작이 아니라 정적 타입 검사기를 위한 시그니처 목록일 뿐이라는 걸 아는가
  • 실제 구현체는 @overload 아래에 한 번 더 작성해야 한다는 걸 아는가

답변 구조 — 생각의 순서

  1. 문제 진단: Union[str, bytes] 반환은 "둘 중 하나"라는 정보만 주고, "입력이 str이면 출력도 str"이라는 대응 관계는 표현하지 못함
  2. 결과: 호출부에서 process("a")의 결과를 str 메서드에 바로 쓰면 mypy가 "bytes일 수도 있다"고 오탐(false positive) 경고를 냄
  3. 해결책: @overload로 입력·출력 타입 조합별 시그니처를 나열
  4. 실제 구현: @overload 시그니처들은 타입 검사기만 보는 것이고, 실제 실행되는 구현은 그 아래에 데코레이터 없이 한 번 더 작성
from typing import overload

@overload
def process(x: str) -> str: ...
@overload
def process(x: bytes) -> bytes: ...

def process(x: str | bytes) -> str | bytes:  # 실제 구현 — 런타임에 호출되는 건 이것뿐
    if isinstance(x, str):
        return x.upper()
    return x.upper()  # bytes도 upper() 지원

reveal_type(process("a"))   # mypy: str로 정확히 추론
reveal_type(process(b"a"))  # mypy: bytes로 정확히 추론
강한 답변

"Union[str, bytes]는 부정확합니다. 그건 '반환값이 str이거나 bytes 둘 중 하나'라는 정보만 줄 뿐, '입력이 str이면 반드시 반환도 str'이라는 입력-출력 대응 관계는 표현하지 못합니다. 그 결과 result = process("a") 다음에 result.decode() 같은 str에 없는 메서드를 쓰려고 하면 mypy가 잘못된 경고를 내거나, 반대로 result가 진짜 str이라는 걸 알면서도 매번 isinstance 체크나 cast를 강제로 넣어야 하는 번거로움이 생깁니다.

정확하게 표현하려면 typing.overload를 씁니다. @overload가 붙은 시그니처들은 실제로 실행되지 않고 정적 검사기만 읽는 '타입 선언 목록'입니다. 진짜 실행되는 구현체는 @overload 없이 그 아래에 한 번 더 작성해야 하며, 이 구현체의 시그니처는 앞서 나열한 모든 오버로드를 포괄할 수 있을 만큼 넓어야 합니다.

이렇게 하면 호출부에서 process("a")를 호출했을 때 mypy가 정확히 str로 추론하고, process(b"a")는 정확히 bytes로 추론합니다. 실무에서는 표준 라이브러리도 이 패턴을 많이 씁니다 — 예를 들어 open()이 mode 인자에 따라 str 핸들을 반환할지 bytes 핸들을 반환할지가 오버로드로 표현되어 있습니다."

약한 답변

"Union[str, bytes]로 충분합니다. 어차피 실행되는 건 똑같으니까요."

왜 약한가: 런타임 동작이 똑같다는 건 맞지만, 질문의 핵심은 "정적 검사기가 얼마나 정확히 타입을 추론하는가"다. 이 답변은 타입 힌트의 목적이 런타임이 아니라 정적 분석 정확도에 있다는 걸 놓치고 있다.

전제 개념 ③ — Literal은 "값 자체"를 타입으로 좁힌다

지금까지 본 타입들은 모두 "이 종류의 어떤 값이든"을 표현했다(int는 모든 정수, str은 모든 문자열). Literal은 반대로 정확히 이 값이어야 한다는 제약을 건다. 문자열 상수로 동작을 분기하는 API에서 오타나 잘못된 값을 정적으로 잡아낼 때 특히 유용하다. 

from typing import Literal

def set_log_level(level: Literal["debug", "info", "warning", "error"]) -> None:
    ...

set_log_level("info")     # OK
# set_log_level("infoo")  # mypy 오류: 오타를 정적으로 잡아냄

실무 활용에서 Literal의 진짜 힘은 다른 매개변수의 타입이 이 리터럴 값에 따라 달라지는 오버로드와 결합할 때 나온다. 예를 들어 요청 응답 형식이 "json"이면 dict를, "text"면 str을 반환하는 함수를 @overload + Literal 조합으로 정확히 표현할 수 있다.

Senior-level Follow-up 질문

Follow-up 1

제네릭 클래스에서 무공변(invariant), 공변(covariant), 반공변(contravariant)의 차이는 무엇이고 왜 신경 써야 하나요?

list[Cat]이 list[Animal]의 하위 타입으로 취급될 수 있는가의 문제다. 만약 그렇게 취급되면(공변), list[Animal] 자리에 list[Cat]을 넣은 뒤 Dog를 append할 수 있게 되어 타입 안전성이 깨진다 — 이게 리스트 같은 가변 컨테이너를 기본적으로 무공변(invariant)으로 설계하는 이유다. 반면 읽기 전용 컨테이너 (예: Sequence[Cat])는 값을 밖으로 꺼내기만 하므로 공변 (covariant)이어도 안전하다. 함수 매개변수처럼 "값을 받기만 하는" 위치는 반대 방향인 반공변(contravariant)이 안전할 수 있다. TypeVar에 covariant=True/contravariant=True를 지정해 이 성질을 명시할 수 있으며, 이걸 잘못 지정하면 mypy가 실제로는 안전하지 않은 대입을 통과시켜버리는 함정이 생긴다.

Follow-up 2

런타임에 타입 힌트가 필요한 경우(예: pydantic, ORM)는 어떻게 동작하나요? from __future__ import annotations은 여기에 어떤 영향을 주나요?

pydantic이나 dataclasses 같은 라이브러리는 typing.get_type_hints()나 __annotations__를 런타임에 직접 읽어서 검증·직렬화 로직을 만든다. 이때 from __future__ import annotations(PEP 563)를 켜면 모든 타입 힌트가 평가되지 않은 문자열로 저장되므로(지연 평가), get_type_hints()가 이 문자열을 실제 타입 객체로 해석하려면 해당 이름들이 그 시점에 임포트되어 있어야 한다. 순환 import를 피하려고 타입 전용 임포트를 if TYPE_CHECKING: 블록 안에 넣었는데 런타임에 get_type_hints()를 호출하면 NameError가 나는 게 이 함정의 전형적인 사례다 — 정적 검사와 런타임 요구사항이 충돌하는 지점을 이해하고 있는가를 시니어 레벨에서 특히 자주 묻는다.

Follow-up 3

대규모 레거시 코드베이스에 타입 힌트를 점진적으로 도입한다면 어떤 전략을 쓰시겠습니까?

한 번에 전체 코드베이스에 엄격한 타입 검사를 걸면 수천 개의 에러가 쏟아져 팀이 압도된다. 실무적 전략은 mypy의 점진적 타이핑(gradual typing) 성질을 활용하는 것이다: (1) mypy.ini에서 모듈 단위로 strict 여부를 다르게 설정, (2) 새로 작성하는 모듈은 처음부터 strict=True로 강제, (3) 기존 모듈은 disallow_untyped_defs 같은 개별 플래그를 하나씩 켜가며 단계적으로 좁혀가기, (4) 타입을 아직 못 붙인 서드파티 의존성은 Any로 격리하되 그 경계를 TODO로 추적. "전부 한 번에"가 아니라 "가장 자주 바뀌고 버그가 많이 나는 모듈부터 우선순위를 매겨 점진적으로"라는 답이 실무 경험을 보여준다.

Follow-up 4

Any와 object는 둘 다 "무엇이든 될 수 있다"처럼 보이는데 타입 검사기 입장에서 어떻게 다른가요?

object는 진짜 타입 계층의 최상위이며, 타입 검사가 여전히 작동한다 — x: object에는 어떤 값이든 대입할 수 있지만, 그 값에서 x.upper()처럼 object가 갖지 않은 메서드를 호출하려 하면 mypy가 정확히 에러를 낸다(써먹으려면 isinstance로 좁혀야 함). 반면 Any는 타입 검사기에게 "이 지점부터는 검사를 하지 마라"고 지시하는 탈출구다. x: Any는 어떤 메서드를 호출해도, 어떤 타입에 대입해도 전부 통과된다. 실무에서 Any를 남발하면 그 지점부터 타입 안전성의 사슬이 끊기고, Any가 전파되며 코드베이스 전체의 타입 검사 효과가 조용히 약해진다 — 이를 "Any 오염(Any contagion)"이라 부르기도 한다.

Trade-off와 흔한 오해

오해 1 — "타입 힌트를 쓰면 런타임이 더 안전해진다"

타입 힌트 자체는 런타임에 아무 검증도 하지 않는다. 안전성은 CI에서 mypy/pyright를 돌리는 프로세스가 만들어주는 것이지, 타입 힌트를 쓰는 행위 자체가 만들어주는 게 아니다. 타입 검사기를 CI에 연결하지 않으면 타입 힌트는 사실상 "실행되지 않는 문서"에 불과하다.

오해 2 — "Protocol과 ABC는 그냥 같은 기능을 문법만 다르게 쓴 것이다"

ABC는 명목적 타이핑(명시적으로 상속해야 인정)이고, Protocol은 구조적 타이핑(상속 없이 시그니처만 맞으면 인정)이다. 게다가 ABC는 @abstractmethod로 런타임에 인스턴스화를 막을 수 있지만, Protocol은 기본적으로 런타임 강제력이 없다(단, @runtime_checkable을 붙이면 isinstance 체크는 가능해지되 메서드 시그니처까지는 검사하지 않는다는 제약이 남는다).

오해 3 — "제네릭을 쓰면 코드가 항상 더 안전하고 좋다"

과도한 제네릭화는 오히려 가독성을 해친다. 타입 매개변수가 3개, 4개로 늘어나는 클래스는 호출부에서 무엇을 넣어야 하는지 파악하기 어려워진다. "이 타입이 실제로 여러 구체 타입으로 재사용될 가능성이 있는가"를 먼저 판단하고, 한 번만 쓰이는 곳까지 제네릭으로 미리 설계하는 건 과설계(over-engineering)에 가깝다.

오해 4 — "mypy 에러가 없으면 타입이 완벽히 안전하다"

Any가 섞여 있거나, # type: ignore 주석으로 특정 줄을 억제했거나, cast()로 강제 형변환한 지점은 mypy가 통과시키더라도 실제로는 런타임에 타입이 안 맞을 수 있다. "mypy 그린" 상태를 "타입이 100% 안전하다"로 오해하면 안 되고, 이런 억제 지점이 코드베이스에 얼마나 있는지를 관리하는 것도 실무 책임의 일부다.

핵심 요약 — 1분 복습 카드

오늘 반드시 기억할 것
  1. 타입 힌트는 런타임에 검증되지 않으며, mypy/pyright 같은 별도 정적 분석 도구가 CI/IDE에서 검사할 때만 의미가 있음
  2. Protocol은 상속 없이 메서드 시그니처만으로 타입 호환성을 판단하는 구조적 타이핑을 제공 — ABC의 명목적 타이핑과 대비됨
  3. TypeVar는 함수/클래스의 입력·출력 타입 사이의 대응 관계를 정적 검사기에게 알려주는 도구
  4. @overload는 정적 검사 전용 시그니처 목록이며, 실제 실행되는 구현체는 별도로 데코레이터 없이 작성해야 함
  5. Literal은 값 자체를 타입으로 좁혀 오타나 잘못된 상수를 정적으로 잡아냄
  6. 가변 컨테이너는 타입 안전성을 위해 기본적으로 무공변이며, 공변/반공변은 컨테이너가 값을 읽기만/쓰기만 할 때 안전해짐
  7. Any는 타입 검사를 끄는 탈출구이므로 남발하면 "Any 오염"으로 코드베이스 전체의 타입 안전성이 조용히 약해짐
