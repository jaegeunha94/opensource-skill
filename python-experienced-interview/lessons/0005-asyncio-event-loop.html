Day 5

asyncio와 이벤트 루프
협력적 멀티태스킹과 await의 실행 모델

파이썬 경력자 Interview · 2026-07-03 · 약 70–90분 소요

왜 면접에서 asyncio를 묻는가

async def와 await를 쓸 줄 아는 개발자는 많다. 하지만 "이벤트 루프가 정확히 무엇을 하는지, 왜 await 지점에서만 제어권이 넘어가는지, 왜 동기 코드 한 줄이 전체 서버를 멈추게 하는지"를 설명할 수 있는 사람은 훨씬 적다. 면접관은 asyncio 문법을 아는지가 아니라, 단일 스레드 협력적 멀티태스킹 모델의 제약을 실무 설계에 반영할 수 있는가를 본다.

Day 4에서 다룬 제너레이터의 "멈추고 재개하는" 능력이 코루틴의 본질이라고 했다. asyncio는 바로 그 능력 위에 스케줄러(이벤트 루프)를 얹어, 수천 개의 코루틴을 하나의 스레드에서 번갈아 실행시키는 시스템이다. 이 관계를 이해하지 못하면 "왜 requests.get()을 비동기 함수 안에서 쓰면 안 되는지" 같은, 실무에서 가장 흔히 발생하는 장애를 설명할 수 없다.

실무 연결
"FastAPI로 만든 API 서버가 트래픽이 조금만 늘어도 전체적으로 느려집니다. 핸들러 하나는 DB 쿼리 하나만 날리는데도요." — 이런 질문에 "동기 라이브러리를 async 핸들러 안에서 호출해서 이벤트 루프가 블로킹됐을 가능성"을 바로 떠올릴 수 있는가가 이 레슨의 핵심이다.

전제 개념 ① — 이벤트 루프는 "하나의 스레드가 여러 작업을 번갈아 처리하는 스케줄러"다

Day 3에서 GIL 때문에 파이썬 스레드는 CPU 코어를 동시에 쓸 수 없다고 배웠다. asyncio는 아예 여러 스레드를 쓰지 않고, 단 하나의 스레드 안에서 작업을 번갈아 실행하는 방식을 택한다. 이것을 협력적 멀티태스킹(cooperative multitasking)이라 부른다 — OS가 강제로 실행을 뺏어가는 스레드의 선점형(preemptive) 스케줄링과 반대되는 개념이다.

이벤트 루프가 하는 일

이벤트 루프는 실행 준비가 된 코루틴들의 큐를 갖고 있다가, 하나를 골라 실행을 재개시킨다. 그 코루틴이 await를 만나면 "나 지금 결과를 기다려야 하니, 그동안 다른 애를 실행해"라며 제어권을 스스로 반납한다. 이벤트 루프는 그 반납받은 시점에 다른 준비된 코루틴을 고른다. OS가 강제로 실행을 끊는 게 아니라, 코루틴이 협력적으로 양보한다는 점이 핵심이다. 

import asyncio

async def say(name, delay):
    print(f"{name} 시작")
    await asyncio.sleep(delay)  # 여기서 제어권을 이벤트 루프에 반납
    print(f"{name} 종료")

async def main():
    await asyncio.gather(say("A", 2), say("B", 1))
    # 출력: A 시작 → B 시작 → B 종료(1초 후) → A 종료(2초 후)
    # 총 소요 시간은 3초(2+1)가 아니라 2초(max) — 단일 스레드에서 동시에 '기다리는 중'이기 때문

asyncio.run(main())
단일 스레드라는 사실이 만드는 절대 규칙
코루틴이 await로 제어권을 넘기지 않으면, 이벤트 루프는 그동안 아무것도 못 한다. 단 하나의 스레드이기 때문이다. 그래서 코루틴 안에서 CPU를 오래 쓰는 계산이나 블로킹 I/O(동기 requests.get(), 동기 파일 읽기, time.sleep())를 호출하면 그 순간 이벤트 루프 전체가 멈춘다 — 다른 모든 요청이 응답을 못 받게 된다.

면접 질문 ① — async def로 정의한 함수를 그냥 호출하면 어떻게 되나요?

async def foo(): ...를 정의하고 foo()라고 그냥 호출했더니 아무 일도 안 일어나는 것처럼 보입니다. 왜 그런가요?

인터뷰어가 평가하는 신호

  • coroutine 함수 호출이 실행이 아니라 coroutine 객체 생성이라는 걸 아는가
  • 이 객체가 실제로 실행되려면 이벤트 루프에 스케줄돼야 한다는 걸 아는가
  • await, asyncio.run(), create_task()의 역할 차이를 구분하는가

답변 구조 — 생각의 순서

  1. 사실 확인: foo() 호출은 함수 본문을 실행하지 않고 coroutine 객체를 반환
  2. 비유: Day 4의 제너레이터와 동일 — 제너레이터 함수 호출도 즉시 실행 안 됨
  3. 실행 트리거: await, asyncio.run(), loop.create_task() 중 하나가 있어야 실제 실행
  4. 경고: "coroutine was never awaited"라는 흔한 RuntimeWarning의 원인
강한 답변

"async def로 정의한 함수를 호출하면, 그 순간 함수 본문이 실행되는 게 아니라 coroutine 객체가 생성됩니다. 이건 Day 4에서 본 제너레이터 함수 호출과 정확히 같은 구조인데, async def가 내부적으로 제너레이터 기반 코루틴을 문법적으로 정리한 것이기 때문입니다.

이 coroutine 객체는 '실행 예약서'일 뿐이고, 실제로 실행되려면 세 가지 방법 중 하나가 필요합니다. await coroutine으로 현재 코루틴 안에서 실행하거나, asyncio.run(coroutine)으로 새 이벤트 루프를 만들어 최상위에서 실행하거나, asyncio.create_task(coroutine)으로 이벤트 루프에 스케줄만 걸어두고 나중에 실행되게 할 수 있습니다.

만약 await도 안 하고 create_task도 안 한 채 coroutine 객체를 그냥 버려두면, 가비지 컬렉션 시점에 RuntimeWarning: coroutine 'foo' was never awaited가 뜹니다. 이건 실무에서 '비동기 함수를 호출했다고 착각하고 await를 빼먹은' 버그를 잡는 신호이기도 합니다."

약한 답변

"async 함수는 await를 붙여야 실행돼요. 안 붙이면 실행이 안 됩니다."

왜 약한가: 결과는 맞지만 왜 그런지(coroutine 객체 생성과 스케줄링의 분리)를 설명하지 못한다. 제너레이터와의 구조적 연결, RuntimeWarning이 뜬다는 실무적 디테일이 빠져 있어 원리를 이해했는지 확인할 수 없다.

전제 개념 ② — coroutine, Task, Future는 서로 다른 층위의 객체다

이 세 용어를 섞어 쓰는 경력자가 매우 많다. 계층 구조로 정확히 구분하는 것 자체가 강한 신호다.

Coroutine 객체

async def 함수를 호출해서 생긴 "실행 예약서". 아직 이벤트 루프에 등록되지 않았다. await로 직접 실행하거나 Task로 감싸야 스케줄링된다.

Task

coroutine을 이벤트 루프에 즉시 스케줄링해서 "지금부터 동시에 진행"시키는 객체. asyncio.create_task(coro)로 만든다. Task는 Future의 서브클래스이며, 취소(cancel()), 완료 콜백 등록, 예외 확인 같은 제어 기능을 제공한다.

Future

"미래에 결과가 채워질 자리"를 나타내는 더 저수준의 객체. 코루틴이 아니라 콜백 기반 비동기 API(저수준 소켓 이벤트 등)를 await 가능하게 감싸는 데 쓰인다. 실무에서 Future를 직접 만들 일은 드물지만, Task가 Future를 상속한다는 관계는 asyncio 내부 구조를 이해하는 데 필요하다. 

import asyncio

async def fetch(n):
    await asyncio.sleep(1)
    return n * 2

async def main():
    coro = fetch(1)              # ① 아직 실행 안 됨 — 예약서일 뿐
    task = asyncio.create_task(fetch(2))  # ② 즉시 이벤트 루프에 스케줄 → 지금부터 동시 진행

    print("다른 일을 하는 척...")
    result1 = await coro          # ③ 여기서야 coro 실행 시작
    result2 = await task          # ④ task는 이미 ②에서부터 진행 중이었음
    print(result1, result2)

asyncio.run(main())
가장 흔한 실무 실수 — coroutine 리스트를 그냥 순차 await
for coro in coros: await coro는 동시 실행이 아니라 순차 실행이다. 각 await가 끝날 때까지 다음 줄로 안 넘어가기 때문이다. 진짜 동시성을 얻으려면 먼저 create_task()로 전부 스케줄한 뒤 모아서 기다리거나(asyncio.gather), 애초에 gather에 coroutine을 바로 넘겨야 한다.

면접 질문 ② — API 서버가 트래픽이 조금만 늘어도 전체적으로 느려집니다

FastAPI로 만든 async def 핸들러가 있는데, 트래픽이 조금만 늘어나면 이 핸들러와 전혀 무관한 다른 API 요청까지 전부 느려집니다. 왜 이런 일이 생기고, 어떻게 진단하고 고치시겠습니까?

인터뷰어가 평가하는 신호

  • 단일 스레드 이벤트 루프 모델에서 블로킹 호출 하나가 전체를 멈춘다는 걸 즉시 연결하는가
  • 흔한 원인(동기 HTTP 클라이언트, 동기 DB 드라이버, CPU 집약적 연산, 동기 파일 I/O)을 구체적으로 짚는가
  • 진단 방법(느린 요청 프로파일링, 이벤트 루프 지연 측정)을 아는가
  • 해결책이 "비동기 라이브러리로 교체" 하나만이 아니라 run_in_executor 같은 대안까지 아는가

답변 구조 — 생각의 순서

  1. 모델 재확인: 이벤트 루프는 단일 스레드, 협력적 스케줄링 → 양보하지 않으면 전체가 멈춤
  2. 범인 후보: async 핸들러 안에서 동기(블로킹) 코드를 호출하고 있는지 확인
  3. 진단: 어떤 핸들러가 이벤트 루프를 오래 점유하는지 측정
  4. 수정: 비동기 라이브러리로 교체하거나, 불가피하면 스레드/프로세스 풀로 위임
강한 답변

"이 증상은 이벤트 루프가 블로킹됐다는 전형적인 신호입니다. asyncio는 단일 스레드에서 협력적으로 코루틴을 번갈아 실행하는데, 어떤 코루틴이 await로 제어권을 넘기지 않고 오랫동안 스레드를 붙잡고 있으면, 그 사이 다른 요청은 전부 대기해야 합니다. '전혀 무관한 API까지 느려진다'는 게 핵심 단서예요 — 특정 핸들러만 느린 게 아니라 전체가 같이 느려진다는 건 리소스(DB 커넥션 풀 등) 경합이 아니라 이벤트 루프 자체가 멈췄다는 뜻입니다.

가장 흔한 원인은 async def 핸들러 안에서 동기(블로킹) 라이브러리를 그대로 호출하는 겁니다. 예를 들어 await 없이 requests.get()을 부르거나, 동기 드라이버로 DB 쿼리를 날리거나, 이미지 리사이징 같은 CPU 집약적 작업을 코루틴 안에서 직접 계산하는 경우입니다. 진단은 각 핸들러의 실행 시간을 프로파일링하거나, 이벤트 루프 지연(loop lag)을 측정하는 도구로 '언제 루프가 오래 멈췄는지'를 확인해서 범인 핸들러를 찾습니다.

해결책은 두 갈래입니다. 첫째, httpx의 AsyncClient, asyncpg처럼 진짜 비동기 라이브러리로 교체하는 게 최우선입니다. 둘째, 어쩔 수 없이 동기 라이브러리를 써야 하거나 CPU 집약적 연산이라면 loop.run_in_executor()로 스레드 풀(I/O bound) 또는 프로세스 풀(CPU bound)에 위임해서 이벤트 루프 자체는 블로킹되지 않게 만듭니다. CPU bound라면 Day 3에서 다룬 GIL 문제도 같이 걸리므로 프로세스 풀을 써야 한다는 점까지 연결해서 답하겠습니다."

약한 답변

"동기 코드를 async 함수 안에서 쓰면 안 됩니다. 비동기 라이브러리로 바꾸면 됩니다."

왜 약한가: 결론은 맞지만 "전혀 무관한 API까지 왜 같이 느려지는지"를 이벤트 루프의 단일 스레드 모델과 연결해서 설명하지 못한다. 진단 방법이나, 라이브러리 교체가 불가능할 때의 대안(run_in_executor)도 빠져 있어 실전 트러블슈팅 경험이 있는지 의심스럽다.

전제 개념 ③ — 여러 Task를 동시에 돌리고, 취소하고, 예외를 다루는 법

실무 asyncio 코드의 대부분은 "여러 개를 동시에 시작하고 결과를 모으는" 패턴이다. 이때 gather, create_task, (3.11+) TaskGroup의 차이, 그리고 하나가 실패했을 때 나머지는 어떻게 되는지를 정확히 아는지가 시니어 시그널이다. 

import asyncio

async def fetch(name, delay, fail=False):
    await asyncio.sleep(delay)
    if fail:
        raise ValueError(f"{name} 실패")
    return f"{name} 완료"

async def main():
    # return_exceptions=False(기본값): 하나라도 예외를 던지면 gather 자체가 즉시 그 예외로 실패
    # 단, 이미 실행 중인 다른 Task들은 '취소되지 않고 백그라운드에서 계속 실행'된다 — 흔한 오해 포인트
    try:
        await asyncio.gather(
            fetch("A", 1),
            fetch("B", 0.5, fail=True),
        )
    except ValueError as e:
        print(f"gather 실패: {e}")

    # return_exceptions=True: 예외도 정상 결과처럼 리스트에 담겨 반환된다 (예외가 전파되지 않음)
    results = await asyncio.gather(
        fetch("C", 0.1),
        fetch("D", 0.1, fail=True),
        return_exceptions=True,
    )
    print(results)  # ['C 완료', ValueError('D 실패')]

asyncio.run(main())
gather의 함정 — 예외가 나도 다른 Task는 계속 돈다
return_exceptions=False일 때 gather는 첫 예외를 만나면 즉시 그 예외를 던지지만, 이미 스케줄된 다른 Task들을 자동으로 취소하지 않는다. 이걸 모르면 "예외 처리를 했는데도 백그라운드에서 알 수 없는 부작용(중복 요청, 리소스 누수)이 생긴다"는 버그를 이해할 수 없다. Python 3.11부터는 asyncio.TaskGroup이 이 문제를 구조적으로 해결한다 — 그룹 내 하나가 실패하면 나머지 전부를 자동으로 취소하고 예외들을 ExceptionGroup으로 묶어 전파한다. 
# Python 3.11+ 권장 패턴 — 하나 실패 시 나머지 자동 취소
async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(fetch("A", 1))
            tg.create_task(fetch("B", 0.5, fail=True))
    except* ValueError as eg:
        print(f"실패한 것들: {eg.exceptions}")

면접 질문 ③ — asyncio.gather와 create_task를 각각 언제 쓰나요?

여러 비동기 작업을 동시에 실행하고 싶을 때 asyncio.gather와 asyncio.create_task는 각각 어떤 상황에 쓰나요? 둘을 같이 써야 하는 경우도 있나요?

인터뷰어가 평가하는 신호

  • create_task는 "지금 즉시 스케줄", gather는 "여러 awaitable을 모아서 기다림"이라는 역할 차이를 아는가
  • gather에 coroutine을 직접 넘겨도 동시 실행되는 이유(내부적으로 Task로 감쌈)를 아는가
  • 결과 순서 보장, 예외 처리 방식(위 전제 개념 ③)까지 연결하는가
강한 답변

"asyncio.gather(*aws)는 여러 awaitable(코루틴이나 Task)을 한 번에 넘겨서 모두 동시에 진행시키고, 전부 끝날 때까지 기다렸다가 결과를 입력 순서 그대로 리스트로 돌려주는 헬퍼입니다. 코루틴을 직접 넘겨도 내부적으로 각각 Task로 감싸서 스케줄하기 때문에 동시 실행이 보장됩니다.

create_task는 그보다 저수준으로, coroutine 하나를 지금 이 시점에 이벤트 루프에 등록해서 바로 진행시키는 역할만 합니다. 결과를 기다리는 로직은 별개입니다. 그래서 '조건부로 몇 개만 기다리고 싶다', '취소 시점을 개별적으로 제어하고 싶다', 'fire-and-forget으로 결과를 기다리지 않고 던져두고 싶다' 같은 세밀한 제어가 필요할 때는 create_task로 Task 객체를 직접 다룹니다.

둘을 같이 쓰는 전형적인 패턴은, 각 작업을 create_task로 먼저 스케줄해서 지금 즉시 동시 진행을 시작시켜 놓고, 뒤에 다른 로직을 실행한 다음, 마지막에 await asyncio.gather(*tasks)로 결과를 한꺼번에 모으는 겁니다. 이렇게 하면 'gather를 호출하기 전까지의 다른 코드 실행 시간'까지 동시성에 포함시킬 수 있어서, 단순히 gather(fetch(...), fetch(...))만 쓰는 것보다 유연합니다."

약한 답변

"둘 다 여러 작업을 동시에 실행할 때 씁니다. 비슷해서 아무거나 써도 됩니다."

왜 약한가: "비슷해서 아무거나"라는 답은 역할 차이를 전혀 구분하지 못한다는 신호다. create_task의 '즉시 스케줄'이라는 성질과, gather의 '결과 취합' 역할을 구분하지 못하면 취소 제어나 fire-and-forget 패턴을 설계할 수 없다.

Senior-level Follow-up 질문

Follow-up 1

"asyncio를 쓰면 race condition에서 완전히 자유로운가요? 락(Lock)이 필요한 경우가 있나요?"

아니다. 단일 스레드라서 "동시에 같은 줄이 실행되는" 일은 없지만, await 지점에서 다른 코루틴이 끼어들어 공유 상태를 바꿀 수 있다. 예를 들어 "잔액을 읽고 → await로 외부 API 호출 → 잔액을 갱신"하는 코드가 있으면, 그 await 사이에 다른 코루틴이 같은 잔액을 먼저 바꿔버릴 수 있다. 이런 await 지점 사이의 경쟁 상태를 막으려면 asyncio.Lock이나 asyncio.Semaphore가 필요하다 — GIL이 스레드 간 바이트코드 원자성을 보장하는 것과는 완전히 다른 층위의 문제라는 걸 구분해서 답해야 한다.

Follow-up 2

"task.cancel()을 호출하면 정확히 무슨 일이 일어나나요?"

Day 4에서 다룬 제너레이터의 throw() 메커니즘과 직결된다. cancel()은 Task가 대기 중인 지점(다음 await 재개 시점)에 asyncio.CancelledError를 주입한다. 코루틴이 이를 try/except로 잡지 않으면 그대로 전파돼 Task가 취소 상태로 끝난다. 만약 코루틴이 except CancelledError로 잡아서 정리 작업만 하고 다시 raise하지 않으면, 취소가 무시된 것처럼 보이는 버그가 생긴다 — 관례상 정리 후 반드시 raise로 다시 던져야 한다.

Follow-up 3

"동기 라이브러리를 async 코드에서 어쩔 수 없이 써야 할 때는 어떻게 하나요?"

loop.run_in_executor(None, blocking_func, *args)로 기본 스레드 풀에 위임하면 이벤트 루프는 블로킹되지 않고 다른 코루틴을 계속 처리할 수 있다. I/O bound 블로킹 코드에는 스레드 풀로 충분하지만(GIL이 I/O 대기 중엔 풀리므로), CPU 집약적 연산이라면 Day 3에서 다룬 GIL 문제가 그대로 적용되어 스레드 풀로는 해결이 안 되고 ProcessPoolExecutor로 넘겨야 한다는 점을 구분해서 답해야 한다.

Follow-up 4

"이벤트 루프 구현체를 직접 골라본 적 있나요? uvloop은 무엇이 다른가요?"

표준 asyncio의 기본 이벤트 루프는 순수 파이썬으로 구현되어 있다. uvloop은 libuv(Node.js가 쓰는 것과 같은 C 라이브러리) 기반으로 이벤트 루프를 재구현해서, 스케줄링과 I/O 폴링 오버헤드를 크게 줄인다. asyncio.set_event_loop_policy()로 교체하며, 코드 변경 없이 순수 I/O bound 워크로드에서 상당한 처리량 개선을 얻을 수 있다. 다만 이건 "이벤트 루프 자체의 효율"을 높이는 것이지, 애초에 블로킹 코드가 섞여 있는 구조적 문제는 해결해주지 않는다는 것도 같이 짚어야 한다.

Trade-off와 흔한 오해

오해 1 — "asyncio를 쓰면 코드가 무조건 더 빨라진다"

틀렸다. asyncio는 I/O 대기 시간을 겹쳐서 처리량을 늘리는 것이지, CPU 연산 자체를 빠르게 만들지 않는다. CPU bound 작업을 async def 안에 그대로 넣으면 오히려 이벤트 루프를 막아서 전체 시스템이 더 나빠질 수 있다. "I/O bound 대기 시간이 많고, 동시에 처리해야 할 요청 수가 많을 때"라는 조건을 명시해야 한다.

오해 2 — "await하면 별도 스레드에서 실행된다"

아니다. await는 같은 스레드 안에서 제어권을 이벤트 루프에 넘기는 것이지 새 스레드를 만들지 않는다. 새 스레드나 프로세스가 관여하는 건 명시적으로 run_in_executor를 쓸 때뿐이다. 이 오해가 있으면 "asyncio 코드는 스레드 안전(thread-safe)할 필요가 없다"는 잘못된 결론(오해 3)으로 이어지기 쉽다.

오해 3 — "단일 스레드니까 asyncio 코드에는 락이 필요 없다"

부분적으로만 맞다. Follow-up 1에서 다뤘듯, await 지점 사이에는 다른 코루틴이 끼어들어 공유 상태를 바꿀 수 있다. "한 줄 안에서 동시에 실행되는 일은 없다"는 것과 "여러 await에 걸친 로직 전체가 원자적이다"는 것은 다른 이야기다.

오해 4 — "asyncio.gather에 넘긴 coroutine들은 항상 병렬로 CPU를 쓴다"

"동시(concurrent)"와 "병렬(parallel)"을 혼동하는 대표적인 경우다. GIL 때문에 여전히 한 순간에 실제로 실행되는 파이썬 바이트코드는 하나뿐이다(Day 3 참고). gather가 만드는 건 "I/O 대기 시간을 겹치는 동시성"이지, 여러 CPU 코어에서 진짜로 동시에 계산이 일어나는 병렬성이 아니다.
