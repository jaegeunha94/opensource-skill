Day 4

Iterator / Generator 프로토콜
lazy evaluation과 yield의 실행 모델

파이썬 경력자 Interview · 2026-07-02 · 약 60–80분 소요

왜 면접에서 Iterator/Generator를 묻는가

for x in obj:를 쓸 줄 아는 것과, 왜 그게 동작하는지, 언제 제너레이터를 써야 메모리와 응답 시간이 실제로 개선되는지를 아는 것은 완전히 다른 수준이다. 면접관은 "제너레이터가 뭐예요?"라는 정의 질문 뒤에, 반드시 "그래서 언제 씁니까? 리스트로 하면 안 되나요?"라는 실무 판단 질문을 이어 붙인다.

Iterator/Generator를 깊이 아는지 확인하는 진짜 목적은, 메모리와 시간을 맞바꾸는 lazy evaluation 사고를 코드 설계에 반영할 수 있는가를 보기 위해서다. 대용량 로그 파일을 한 줄씩 처리하는 코드, DB 커서로 수백만 건을 순회하는 배치 작업, API 페이지네이션을 감싸는 래퍼 — 이런 실무 코드는 전부 이 프로토콜 위에서 동작한다.

실무 연결
"10GB짜리 로그 파일을 분석하는데 서버 메모리가 2GB밖에 없어요. 어떻게 처리하시겠습니까?" 혹은 "제너레이터로 만든 파이프라인을 두 번 순회했더니 두 번째부터 아무것도 안 나와요, 왜죠?"라는 질문에 즉시 원인을 짚어낼 수 있는가가 이 레슨의 핵심이다.

전제 개념 ① — Iterable과 Iterator는 다른 것이다

많은 경력자도 이 둘을 섞어서 말한다. 정확히 구분하는 것 자체가 첫 번째 신호다.

Iterable (반복 가능한 객체)

__iter__() 메서드를 가진 객체. __iter__()를 호출하면 Iterator를 반환한다. 리스트, 튜플, 딕셔너리, 문자열, 파일 객체 등이 여기 해당한다. Iterable 자체는 "순회할 수 있다"는 능력만 있을 뿐, 현재 어디까지 순회했는지 상태를 갖지 않는다.

Iterator (실제로 순회 중인 객체)

__iter__()와 __next__()를 모두 가진 객체. __next__()를 호출할 때마다 다음 값을 하나 반환하고, 더 이상 값이 없으면 StopIteration 예외를 던진다. Iterator는 "지금까지 어디까지 진행했는지"를 내부 상태로 기억한다. Iterator의 __iter__()는 관례적으로 자기 자신(self)을 반환한다 — 이것이 "모든 Iterator는 Iterable이다"라는 말의 근거다. 

class CountUp:
    """리스트를 만들지 않고 0부터 n-1까지 순회하는 이터레이터"""
    def __init__(self, n):
        self.n = n
        self.i = 0

    def __iter__(self):
        # Iterator는 자기 자신을 반환한다
        return self

    def __next__(self):
        if self.i >= self.n:
            raise StopIteration
        value = self.i
        self.i += 1
        return value

it = CountUp(3)
print(next(it))  # 0
print(next(it))  # 1
print(next(it))  # 2
print(next(it))  # StopIteration 발생

for 문이 실제로 하는 일

for x in obj:는 문법 설탕(syntactic sugar)이다. 인터프리터는 내부적으로 다음과 동일하게 동작한다. 

# for x in obj: ... 는 실제로 이렇게 풀린다
_iterator = iter(obj)       # obj.__iter__() 호출
while True:
    try:
        x = next(_iterator)   # _iterator.__next__() 호출
    except StopIteration:
        break
    # 루프 본문 실행
왜 이 구분이 중요한가
Iterable(예: 리스트)은 몇 번이든 새로 순회를 시작할 수 있다 — iter(list)를 호출할 때마다 새 Iterator가 생기기 때문이다. 반면 Iterator는 한 번 소진되면 다시 처음으로 되돌릴 수 없다. 이 차이를 모르면 "제너레이터를 두 번 순회했는데 두 번째는 빈 결과가 나온다"는 버그를 이해하지 못한다.

면접 질문 ① — Iterable과 Iterator의 차이를 설명해보세요

Iterable과 Iterator는 어떻게 다른가요? 리스트는 왜 둘 다로 취급될 수 있죠?

인터뷰어가 평가하는 신호

  • __iter__/__next__ 프로토콜을 정확히 구분해서 설명하는가
  • "리스트는 Iterable이지 Iterator가 아니다"라는 정확한 위치를 아는가
  • Iterator가 상태(현재 위치)를 가지며 소진되면 재사용 불가하다는 점을 아는가

답변 구조 — 생각의 순서

  1. 정의: Iterable = __iter__만, Iterator = __iter__ + __next__
  2. 관계: Iterable의 __iter__()가 Iterator를 만들어 반환
  3. 상태: Iterator만 "지금 어디까지 순회했는지" 상태를 가짐
  4. 재사용성: Iterable은 반복해서 새 순회 가능, Iterator는 1회용
강한 답변

"Iterable은 __iter__() 메서드를 구현해서 '순회를 시작할 수 있다'는 능력만 제공하는 객체입니다. 리스트가 대표적인 예죠. 리스트 자체는 지금 어디까지 순회했는지 상태를 갖지 않습니다. 반면 Iterator는 __iter__()와 __next__()를 모두 구현하고, 다음 값과 현재 진행 상태를 함께 관리합니다. __next__()를 호출할 때마다 값을 하나 내주고, 더 없으면 StopIteration을 던집니다.

리스트가 둘 다처럼 보이는 이유는, list는 Iterable이지만 list_iterator라는 별도의 Iterator 객체가 존재하기 때문입니다. iter(my_list)를 호출할 때마다 새 Iterator가 만들어지므로 리스트는 몇 번이든 다시 순회할 수 있는 겁니다. 반면 제너레이터 객체 자체는 Iterator라서, 한 번 다 순회하면 다시 처음부터 돌릴 수 없고 새로 만들어야 합니다."

약한 답변

"Iterable은 반복 가능한 거고 Iterator는 그걸 실행하는 거예요. for문에서 쓰는 거요."

왜 약한가: "실행하는 거"라는 표현이 모호하다. __iter__/__next__ 프로토콜에 대한 언급이 전혀 없고, 리스트가 왜 재사용 가능한지(새 Iterator 생성)와 제너레이터가 왜 1회용인지(자기 자신이 Iterator)의 차이를 설명하지 못한다.

전제 개념 ② — Generator는 함수를 "일시정지 가능한 상태 기계"로 바꾼다

제너레이터 함수(yield가 있는 함수)를 호출하면 함수 본문이 즉시 실행되지 않는다. 대신 제너레이터 객체가 반환된다. 이 객체가 __iter__와 __next__를 자동으로 구현한 Iterator다. next()가 호출될 때마다 직전에 멈췄던 지점부터 실행을 재개하고, 다음 yield를 만나면 값을 내주고 다시 멈춘다. 

def read_large_file(path):
    # 함수를 호출해도 아래 코드는 즉시 실행되지 않는다
    with open(path) as f:
        for line in f:
            yield line.strip()  # 여기서 값을 내주고 멈춘다

gen = read_large_file("access.log")  # 아직 파일을 열지도 않음!
print(next(gen))  # 이제서야 실행 시작 → 첫 줄까지 읽고 멈춤
print(next(gen))  # 멈췄던 지점(for 루프 내부)부터 재개 → 두 번째 줄

왜 "lazy(지연)"인가 — 메모리 곡선이 완전히 다르다

리스트 컴프리헨션은 모든 결과를 한꺼번에 메모리에 만든다. 제너레이터는 한 번에 값 하나만 만들고, 그 값을 소비한 뒤에야 다음 값을 계산한다. 10GB 로그 파일을 읽는다면, 리스트로 읽으면 10GB(+α)가 메모리에 그대로 올라가지만 제너레이터로 읽으면 한 줄 분량의 메모리만 사용한다. 

# 리스트 컴프리헨션: 즉시 전체 결과를 메모리에 생성 (eager)
lines = [line.strip() for line in open("huge.log")]
# → 파일 전체가 메모리에 리스트로 올라감 (10GB 파일이면 10GB+)

# 제너레이터 표현식: 값이 필요할 때만 계산 (lazy)
lines = (line.strip() for line in open("huge.log"))
# → 이 시점엔 아무 계산도 안 됨. 순회할 때 한 줄씩만 메모리에 존재

for line in lines:
    if "ERROR" in line:
        process(line)
# 이 시점에도 메모리에는 "현재 처리 중인 한 줄"만 존재한다
제너레이터는 "한 번 쓰고 버리는" 스트림이다
제너레이터 객체는 Iterator이므로 len()을 쓸 수 없고, 인덱싱(gen[0])도 안 되고, 한 번 다 순회하면 재순회가 불가능하다. 다시 처음부터 순회하려면 제너레이터 함수를 다시 호출해서 새 객체를 만들어야 한다. 이것이 "제너레이터를 두 번 순회했더니 두 번째는 비어있다"는 흔한 버그의 원인이다. 
def squares(n):
    for i in range(n):
        yield i * i

gen = squares(3)
print(list(gen))  # [0, 1, 4] — 여기서 이미 소진됨
print(list(gen))  # [] — 다시 순회해도 아무것도 안 나옴!

면접 질문 ② — 10GB 로그 파일을 2GB 메모리 서버에서 분석해야 합니다

10GB짜리 로그 파일에서 특정 패턴을 가진 줄만 추출하고 집계해야 하는데, 서버 메모리는 2GB밖에 없습니다. 어떻게 처리하시겠습니까?

인터뷰어가 평가하는 신호

  • 파일 전체를 read()나 readlines()로 읽지 않는다는 것을 즉시 인지하는가
  • 제너레이터 파이프라인(여러 단계를 지연 평가로 연결)을 설계할 수 있는가
  • 집계 결과(최종 카운트 등)는 작으므로 메모리에 남는다는 걸 구분하는가
  • itertools를 활용한 실전 최적화까지 아는가

답변 구조 — 생각의 순서

  1. 문제 재정의: "10GB를 메모리에 올리는 게 아니라 스트리밍으로 한 줄씩 처리해야 한다"
  2. 파일 순회 자체가 이미 lazy: 파일 객체는 그 자체로 Iterator, for line in f가 한 줄씩만 읽음
  3. 파이프라인 설계: 필터링 → 파싱 → 집계를 제너레이터로 체이닝
  4. 집계 결과와 원본 데이터의 크기 차이 구분: 최종 카운터/합계는 작아서 메모리에 남아도 무방
강한 답변

"핵심은 파일 전체를 메모리에 올리지 않는 것입니다. 파이썬 파일 객체는 그 자체로 Iterator라서 for line in f:로 순회하면 한 줄씩만 읽고, 이전 줄은 참조가 없으면 바로 GC 대상이 됩니다. f.read()나 f.readlines()를 쓰면 전체가 메모리에 올라가므로 절대 쓰면 안 됩니다.

그 다음 제너레이터 파이프라인으로 필터링과 파싱을 단계별로 연결합니다. 

def filtered_lines(path, pattern):
    with open(path) as f:
        for line in f:
            if pattern in line:
                yield line

def parsed_records(lines):
    for line in lines:
        yield parse_log_line(line)

# 각 단계는 '연결'만 될 뿐 아직 아무 것도 실행 안 됨
records = parsed_records(filtered_lines("huge.log", "ERROR"))

# 실제 순회 시점에 파일 한 줄 → 필터 → 파싱이 한 번에 하나씩 흘러감
counter = {}
for rec in records:
    counter[rec.code] = counter.get(rec.code, 0) + 1

이 구조에서 어느 순간이든 메모리에 존재하는 건 현재 처리 중인 한 줄과 집계용 딕셔너리뿐입니다. 집계 딕셔너리는 고유 코드 개수만큼만 커지므로 원본 10GB와 무관하게 작게 유지됩니다. 더 정교하게는 itertools.islice로 청크 단위 처리를 하거나, 멀티프로세스로 파일을 나눠 읽는 방식도 검토할 수 있지만, 기본은 '전체를 절대 메모리에 올리지 않는 스트리밍 파이프라인'입니다."

약한 답변

"제너레이터를 쓰면 메모리를 적게 씁니다. yield로 한 줄씩 처리하면 됩니다."

왜 약한가: 맞는 방향이지만 왜 적게 쓰는지(각 줄이 처리 후 GC됨), 파이프라인을 어떻게 구성하는지, 집계 결과와 원본 데이터 크기의 차이를 구분하는 설명이 없어서 실제로 설계해본 적이 있는지 의심스러운 인상을 준다.

전제 개념 ③ — 제너레이터는 값을 "받을" 수도 있다 (send, throw, yield from)

대부분은 제너레이터를 "값을 내보내기만 하는 것"으로 안다. 하지만 yield는 사실 표현식(expression)이고, send()로 값을 되돌려 넣을 수 있다. 이것이 asyncio가 등장하기 전(Python 3.4 이전) 코루틴을 흉내내던 방식이며, 지금도 async def/await의 내부 구현 원리와 직결된다. 

def running_average():
    total = 0
    count = 0
    average = None
    while True:
        # yield는 값을 '내보내는' 동시에, send()로 들어온 값을 '받는' 표현식이다
        value = yield average
        total += value
        count += 1
        average = total / count

avg = running_average()
next(avg)             # 첫 next()로 첫 yield까지 실행 (초기 구동, priming)
print(avg.send(10))  # 10.0 — value=10으로 재개, 다음 yield까지 실행
print(avg.send(20))  # 15.0
print(avg.send(30))  # 20.0
왜 "priming"(첫 next() 호출)이 필요한가
갓 생성된 제너레이터는 함수 본문의 맨 처음에 멈춰있다. 아직 첫 번째 yield에도 도달하지 못한 상태이므로, 이 상태에서 바로 send(값)을 호출하면 TypeError: can't send non-None value to a just-started generator가 난다. 반드시 next()(또는 send(None))를 한 번 호출해서 첫 yield까지 실행시켜야 그 이후로 send()가 의미를 가진다.

throw()와 close() — 제너레이터 내부에 예외를 주입한다

def transaction_worker():
    try:
        while True:
            item = yield
            print(f"처리 중: {item}")
    except GeneratorExit:
        # close() 호출 시 이 지점에서 GeneratorExit이 발생 → 정리 로직
        print("롤백하고 리소스 정리")
        raise  # 관례상 다시 raise (억제하면 RuntimeError)

w = transaction_worker()
next(w)
w.send("item-1")
w.close()  # 내부에서 GeneratorExit 발생 → 정리 코드 실행

throw(exc)는 제너레이터가 멈춰있는 yield 지점에 예외를 강제로 발생시킨다. 제너레이터 내부의 try/except로 잡아서 정리 작업을 할 수 있고, 잡지 않으면 그대로 호출자에게 전파된다. 이 메커니즘이 나중에 async def 코루틴이 취소(cancel)될 때 내부적으로 동작하는 방식과 정확히 같다 — asyncio의 Task.cancel()도 결국 코루틴 내부에 CancelledError를 throw하는 방식으로 구현되어 있다.

yield from — 제너레이터 위임(delegation)

def read_section(lines):
    for line in lines:
        yield line

def read_report():
    yield from read_section(["header-1", "header-2"])
    yield from read_section(["body-1", "body-2"])
    yield "footer"

print(list(read_report()))
# ['header-1', 'header-2', 'body-1', 'body-2', 'footer']

yield from sub_gen은 단순히 for x in sub_gen: yield x의 축약이 아니다. send(), throw(), 반환값(return)까지 서브 제너레이터로 투명하게 전달한다. 제너레이터를 부품처럼 조립하는 실무 패턴(예: 여러 파서를 조합해서 하나의 스트림처럼 만들기)의 기반이 된다.

면접 질문 ③ — 제너레이터 기반 코루틴과 async def는 어떤 관계인가요?

Python 3.4 이전에는 코루틴을 yield로 구현했다고 들었습니다. 지금의 async def/await와는 어떤 관계가 있나요?

인터뷰어가 평가하는 신호

  • 제너레이터의 "실행을 멈추고 재개하는" 능력이 코루틴의 본질이라는 걸 이해하는가
  • 역사적 흐름(PEP 342 → PEP 380 → PEP 492)을 대략이라도 아는가
  • async def가 문법적으로 명확해진 것이지, 근본 메커니즘은 연속선상이라는 걸 아는가
강한 답변

"제너레이터의 본질은 함수 실행을 임의의 지점에서 멈추고, 나중에 그 지점부터 다시 재개할 수 있다는 것입니다. 이 능력이 바로 코루틴에 필요한 능력과 같습니다. 그래서 asyncio가 표준화되기 전에는 yield로 멈추고 send()로 값을 넣어주는 방식(@asyncio.coroutine + yield from, PEP 342/380)으로 코루틴을 구현했습니다.

Python 3.5부터는 PEP 492로 async def/await라는 전용 문법이 생겼는데, 이건 완전히 새로운 메커니즘이 아니라 제너레이터 기반 코루틴을 명확하게 구분되는 문법으로 정리한 것입니다. 내부적으로 코루틴 객체도 여전히 '멈추고 재개하는' 동일한 프레임 저장 방식을 쓰고, await는 yield from과 유사하게 동작합니다. 실제로 Task.cancel()이 코루틴 내부에 CancelledError를 throw하는 방식으로 구현된 것도 제너레이터의 throw() 메커니즘을 그대로 물려받은 것입니다. 즉 async def는 '제너레이터를 코루틴처럼 오용하지 않도록 문법적으로 명확히 분리'한 것이지, 근본 실행 모델은 연속선상에 있습니다."

약한 답변

"예전에는 yield로 비동기를 흉내냈는데 지금은 async/await가 생겨서 안 씁니다."

왜 약한가: "흉내냈다"는 표현이 두 메커니즘을 완전히 별개로 취급한다. 제너레이터의 '멈추고 재개' 능력이 코루틴의 본질이라는 연결 고리를 짚지 못했고, Task.cancel()이 throw() 메커니즘 위에서 동작한다는 실무적으로 유용한 통찰이 빠져 있다.

Senior-level Follow-up 질문

Follow-up 1

"제너레이터 함수 안에서 return value를 쓰면 어떻게 되나요?"

일반 함수와 달리 제너레이터에서 return value는 함수를 끝내면서 StopIteration(value)를 발생시킨다. 이 값은 for 루프로는 접근할 수 없고 (StopIteration이 조용히 루프를 끝냄), yield from으로 위임했을 때 result = yield from sub_gen()처럼 표현식의 결과값으로 받을 수 있다. 이것이 서브 제너레이터의 최종 결과를 상위로 전달하는 공식 통로다.

Follow-up 2

"제너레이터가 GC에 의해 수거되기 전에 close()가 자동으로 호출되나요?"

그렇다 — CPython은 제너레이터 객체가 레퍼런스 카운트 0이 되어 소멸될 때 __del__에서 자동으로 close()를 호출해 GeneratorExit을 주입한다. 하지만 순환 참조로 묶여 cyclic GC를 기다려야 하는 경우(Day 2에서 다룬 내용과 연결), 정리 시점이 예측 불가능하게 늦어질 수 있다. with open(...)처럼 컨텍스트 매니저에 의존하는 정리 로직이 있다면, 명시적으로 close()를 호출하거나 contextlib.closing을 쓰는 게 안전하다는 걸 짚으면 좋은 신호다.

Follow-up 3

"제너레이터 표현식과 리스트 컴프리헨션의 성능 차이를 실제로 측정한다면 어떻게 하시겠습니까?"

단순 순회 총 시간은 큰 차이가 없을 수 있다(제너레이터는 오히려 yield 오버헤드로 순수 순회가 살짝 느릴 수도 있음). 진짜 차이는 메모리다. tracemalloc이나 sys.getsizeof()로 피크 메모리를 비교하고, 조기 종료(early exit) 시나리오 (예: 조건에 맞는 첫 값만 필요한 경우 next(filter(...)))에서는 제너레이터가 전체를 계산하지 않아도 되므로 시간도 절약된다는 점까지 구분해서 답하면 좋다.

Follow-up 4

"제너레이터는 스레드 안전(thread-safe)한가요?"

아니다. 제너레이터는 내부 상태(현재 실행 위치, 로컬 변수)를 하나만 가지므로, 여러 스레드가 동시에 같은 제너레이터 객체의 next()를 호출하면 ValueError: generator already executing이 나거나 상태가 꼬인다. 여러 스레드/태스크가 동시에 접근해야 한다면 스레드마다 별도 제너레이터 인스턴스를 만들거나, 락으로 접근을 직렬화해야 한다. GIL이 있어도 "동시 접근으로부터 안전하다"는 것과는 다른 문제라는 점을 구분해서 답하는 게 핵심이다.

Trade-off와 흔한 오해

오해 1 — "제너레이터는 항상 리스트보다 빠르다"

틀렸다. 제너레이터의 이점은 메모리와 조기 종료 시 계산량 절감이지, 무조건적인 속도 우위가 아니다. 전체 데이터를 어차피 다 순회해야 하고 여러 번 접근해야 한다면, 리스트로 한 번 계산해두는 게 반복 접근 시 더 빠르다. "메모리가 부족하거나 데이터가 매우 크거나 조기 종료 가능성이 있을 때" 제너레이터를 선택한다고 조건을 명시해야 한다.

오해 2 — "제너레이터도 여러 번 순회할 수 있다"

제너레이터 객체는 Iterator이므로 1회용이다. 재사용하려면 제너레이터 함수를 다시 호출해서 새 객체를 만들거나, 애초에 클래스 기반으로 __iter__에서 매번 새 제너레이터를 만들어 반환하는 Iterable 래퍼를 만들어야 한다. 

class Lines:
    """매번 새 제너레이터를 만들어 재순회 가능한 Iterable"""
    def __init__(self, path):
        self.path = path

    def __iter__(self):
        with open(self.path) as f:
            for line in f:
                yield line.strip()

lines = Lines("a.log")
list(lines); list(lines)  # 둘 다 정상 동작 — 매번 __iter__가 새 제너레이터 생성
오해 3 — "제너레이터는 예외를 잡으면 자동으로 재개된다"

StopIteration이나 일반 예외가 제너레이터 내부에서 발생하면 제너레이터는 영구적으로 종료된다. 다시 next()를 호출해도 계속 StopIteration만 나온다. 일시정지는 오직 yield에서만 일어나며, 예외로 함수가 끝나면 그 실행 프레임 자체가 소멸한다.

오해 4 — "함수 안에 yield가 있어도 일반 함수처럼 호출하면 바로 실행된다"

가장 흔한 실수다. yield가 함수 본문 어디에든 있으면, 그 함수는 호출 즉시 본문을 실행하지 않고 제너레이터 객체만 반환한다. 함수 안에 유효성 검증 코드를 앞부분에 넣어놓고 "호출하면 바로 검증되겠지"라고 기대했다가, 실제로는 next()를 처음 호출하는 시점까지 검증이 미뤄져서 에러가 예상치 못한 곳에서 터지는 버그가 실무에서 자주 생긴다.

핵심 요약 — 1분 복습 카드

오늘 반드시 기억할 것
  1. Iterable = __iter__만, Iterator = __iter__ + __next__ (+ 진행 상태)
  2. 리스트는 매번 새 Iterator를 만들어 재순회 가능, 제너레이터 객체 자체는 1회용
  3. 제너레이터는 yield에서 실행을 멈추고, next()로 그 지점부터 재개 — lazy evaluation
  4. 대용량 데이터 처리 = 파일 순회 자체가 이미 lazy → 제너레이터 파이프라인으로 스트리밍 처리
  5. send()로 값 주입(priming 필수), throw()로 예외 주입, yield from으로 위임
  6. 제너레이터의 '멈추고 재개' 능력이 async def/await 코루틴의 근본 메커니즘
