Context Manager
리소스 관리 패턴과 with의 실행 모델

파이썬 경력자 Interview · 2026-07-04 · 약 60–80분 소요

왜 면접에서 Context Manager를 묻는가

with open(...) as f: 정도는 누구나 쓴다. 하지만 "왜 finally로도 되는 걸 굳이 with로 하는지, 예외가 나면 __exit__에 뭐가 전달되는지, __exit__가 True를 반환하면 무슨 일이 생기는지"를 설명할 수 있는 사람은 드물다. 면접관은 with 문법을 아는지가 아니라, "리소스 해제가 예외 상황에서도 반드시 실행되도록 보장하는 책임을, 호출부가 아니라 리소스 자체에 위임하는 설계 사고"를 갖고 있는지를 본다.

이 감각은 Day 2의 메모리 관리(레퍼런스 카운팅이 즉시 해제를 보장하지 않는다는 점), Day 3의 GIL과 락(lock) 관리, Day 5의 asyncio 리소스 정리와 전부 연결된다. 커넥션 풀, 파일 핸들, 락, 트랜잭션, 임시 리소스를 다루는 실무 코드는 거의 전부 context manager 위에서 돌아간다. 이걸 직접 구현할 수 있는지가 "라이브러리를 쓸 줄 아는 사람"과 "라이브러리를 설계할 수 있는 사람"을 가른다.

실무 연결
"DB 커넥션 풀에서 커넥션을 빌려 쓰는 유틸리티 함수를 만들어야 하는데, 쿼리 도중 예외가 나도 커넥션이 반드시 풀에 반납되도록 하고 싶습니다. 어떻게 설계하시겠습니까?" — 이런 질문에 "커넥션 획득/반납을 context manager로 캡슐화해서 __exit__에서 예외 여부와 무관하게 반납을 강제한다"고 바로 연결할 수 있는가가 이 레슨의 핵심이다.

전제 개념 ① — with는 "예외가 나도 정리 코드가 반드시 실행되도록" 프로토콜로 강제하는 것이다

"try/finally로도 똑같이 되는데 왜 with를 쓰나요?"라는 질문에 많은 사람이 "코드가 짧아져서"라고만 답한다. 절반만 맞다. 진짜 이유는 정리 로직을 리소스를 만드는 쪽(클래스 작성자)이 한 번 구현해두면, 사용하는 쪽(호출자)이 매번 try/finally를 직접 쓸 필요 없이, 실수로 빠뜨릴 걱정 없이 재사용할 수 있다는 데 있다.

with 문의 실제 동작 순서

with EXPR as VAR:는 대략 다음과 동일하다. 

# with EXPR as VAR: BLOCK  는 대략 이렇게 동작한다
mgr = EXPR
VAR = mgr.__enter__()
try:
    BLOCK
except BaseException:
    if not mgr.__exit__(*sys.exc_info()):
        raise  # __exit__가 True를 반환하지 않으면 예외를 다시 던진다
else:
    mgr.__exit__(None, None, None)
class ManagedFile:
    def __init__(self, path, mode):
        self.path, self.mode = path, mode

    def __enter__(self):
        self.file = open(self.path, self.mode)
        return self.file  # as로 바인딩되는 값

    def __exit__(self, exc_type, exc_value, traceback):
        self.file.close()          # 예외가 나도 반드시 실행됨
        return False              # 예외를 삼키지 않고 다시 던짐

with ManagedFile("data.txt", "w") as f:
    f.write("hello")
    raise ValueError("중간에 실패")
    # ValueError가 던져져도 __exit__는 반드시 호출되어 file.close()가 실행된다
__exit__의 반환값이 예외를 삼킬 수 있다는 함정
__exit__가 True를 반환하면, 발생한 예외는 조용히 삼켜지고 with 블록 바깥으로 전파되지 않는다. 대부분의 경우 이건 버그다 — 리소스 정리를 하면서 실수로 return True를 쓰거나, 정리 로직 자체가 예외를 던지고 그걸 잡아 True를 반환하면, 원래 예외가 사라지고 아무 일도 없었던 것처럼 코드가 계속 실행된다. 특정 예외 타입만 의도적으로 억제하고 싶은 게 아니라면 __exit__는 항상 False(또는 None)를 반환해야 한다.

면접 질문 ① — try/finally 대신 with를 쓰는 진짜 이유가 뭔가요?

리소스 정리는 try/finally로도 완벽하게 할 수 있는데, 왜 굳이 context manager와 with 문을 쓰나요? 단순히 코드가 짧아지는 것 외에 다른 이유가 있나요?

인터뷰어가 평가하는 신호

  • "코드가 짧아진다"를 넘어 책임의 위치가 바뀐다는 설계 관점을 아는가
  • 재사용성 — 같은 정리 로직을 여러 호출부에서 try/finally 없이 재사용 가능
  • 실수 방지 — 호출자가 finally를 빠뜨리는 휴먼 에러를 구조적으로 제거
  • 여러 리소스를 동시에 관리할 때(with a, b:)의 정리 순서까지 아는가

답변 구조 — 생각의 순서

  1. 공통점 확인: 둘 다 "예외가 나도 정리 코드는 실행된다"는 보장은 동일
  2. 차이 — 책임의 위치: try/finally는 호출부마다 반복 작성, with는 리소스 작성자가 한 번만 구현
  3. 재사용과 실수 방지: 호출자가 정리 로직을 잊을 수 없는 구조
  4. 부가 기능: as 바인딩, 여러 리소스 동시 관리, 예외 정보 접근(exc_type 등)
강한 답변

"둘 다 '예외가 발생해도 정리 코드는 반드시 실행된다'는 보장 자체는 동일합니다. 차이는 그 정리 책임이 어디에 있는가입니다. try/finally는 리소스를 쓰는 쪽이 매번 finally 블록을 직접 작성해야 하고, 이걸 빠뜨리면 바로 리소스 누수로 이어집니다. with는 그 정리 로직을 리소스를 만드는 쪽이 __exit__에 한 번만 구현해두면, 호출자는 with resource() as r:만 쓰면 되고 정리를 빠뜨릴 방법 자체가 없습니다.

즉 try/finally는 '올바르게 쓰기를 호출자의 규율에 의존'하는 반면, with는 '올바르게 쓰는 것을 프로토콜로 강제'합니다. 커넥션 풀이나 락처럼 여러 곳에서 반복적으로 빌리고 반납해야 하는 리소스일수록 이 차이가 커집니다.

부가적으로 with는 as로 값을 바로 바인딩하고, with a() as x, b() as y:처럼 여러 리소스를 선언한 순서대로 진입하고 역순으로 정리하는 걸 언어 차원에서 지원합니다. 또 __exit__(exc_type, exc_value, tb)로 예외 정보에 접근할 수 있어서, '실패했을 때만 롤백, 성공하면 커밋' 같은 트랜잭션 로직도 자연스럽게 표현할 수 있습니다."

약한 답변

"with가 코드를 더 짧고 깔끔하게 만들어줘서 씁니다."

왜 약한가: 가독성 이점만 언급하고, 정리 책임의 위치가 바뀐다는 핵심 설계 의도를 짚지 못한다. 재사용성, 실수 방지, 여러 리소스 관리 같은 실무적 이점이 전혀 연결되지 않아 "그냥 문법을 쓸 줄 아는" 수준으로 보인다.

전제 개념 ② — contextlib.contextmanager는 제너레이터로 __enter__/__exit__를 대신 생성한다

클래스로 __enter__/__exit__를 매번 만드는 건 간단한 경우엔 과하다. Day 4에서 배운 제너레이터의 "멈추고 재개하는" 능력을 그대로 이용해서, 함수 하나로 context manager를 만들 수 있다. yield 이전이 __enter__, yield 이후가 __exit__에 대응한다. 

from contextlib import contextmanager

@contextmanager
def managed_file(path, mode):
    f = open(path, mode)
    try:
        yield f              # 여기까지가 __enter__, yield된 값이 as로 바인딩
    finally:
        f.close()             # yield 이후가 __exit__ — 예외가 나도 반드시 실행

with managed_file("data.txt", "w") as f:
    f.write("hello")
제너레이터 기반 context manager의 함정 — try/finally를 빠뜨리면?
yield 안에서 예외가 나면, 그 예외는 yield 지점에서 다시 던져진다 (Day 4의 throw()와 동일한 메커니즘). try/finally 없이 yield만 쓰면, 블록 안에서 예외가 났을 때 f.close()가 실행되지 않는다. 클래스 기반과 달리 이건 컴파일러가 강제해주지 않으므로, @contextmanager를 쓸 때는 try/finally(또는 try/except)를 직접 챙기는 게 관례이자 필수다.
언제 클래스, 언제 @contextmanager인가

상태가 복잡하거나, __exit__에서 예외 타입별로 다른 분기가 많거나, 여러 진입점에서 재사용될 클래스라면 클래스 기반이 명확하다. 로직이 단순하고 한 번의 진입/정리 흐름으로 표현되면 @contextmanager가 훨씬 짧고 읽기 쉽다. 실무에서는 후자가 압도적으로 많다 — DB 세션, 임시 설정 변경, 락 획득 같은 대부분의 케이스가 여기 해당한다.

면접 질문 ② — DB 커넥션 풀에서 커넥션을 빌리고 반드시 반납하는 유틸리티를 설계해보세요

커넥션 풀에서 커넥션을 빌려 쿼리를 실행하는 함수를 여러 곳에서 씁니다. 쿼리 도중 예외가 나도 커넥션이 반드시 풀에 반납되어야 합니다. 어떻게 설계하고, 왜 그렇게 설계했는지 설명해보세요.

인터뷰어가 평가하는 신호

  • "반납"이라는 정리 로직을 호출자가 아니라 유틸리티 자체가 보장하도록 설계하는가
  • 커밋/롤백처럼 예외 여부에 따라 다른 정리 로직이 필요한 경우까지 확장할 수 있는가
  • @contextmanager와 클래스 기반 중 상황에 맞는 선택을 하는가
  • 중첩된 리소스(커넥션 + 트랜잭션)의 정리 순서를 생각하는가

답변 구조 — 생각의 순서

  1. 문제 재정의: "커넥션 획득 → 사용 → 반납"에서 반납이 항상 실행되어야 함
  2. 설계 선택: 이 흐름을 context manager로 캡슐화 — 호출자는 with만 쓰면 됨
  3. 구현: @contextmanager + try/finally로 획득과 반납을 짝짓기
  4. 확장: 예외 발생 시 롤백, 성공 시 커밋까지 같은 구조로 표현
강한 답변

"핵심은 '커넥션을 빌리고 반납하는 흐름'을 호출자의 규율에 맡기지 않고, 하나의 context manager로 캡슐화하는 겁니다. 이렇게 하면 이 유틸리티를 쓰는 모든 곳에서 반납을 빠뜨릴 방법이 없어집니다. 

from contextlib import contextmanager

@contextmanager
def get_connection(pool):
    conn = pool.acquire()
    try:
        yield conn
    finally:
        pool.release(conn)  # 예외 여부와 무관하게 반드시 반납

with get_connection(pool) as conn:
    conn.execute("SELECT ...")
    # 쿼리 도중 예외가 나도 release(conn)은 실행된다

여기서 한 단계 더 나아가, '성공하면 커밋, 실패하면 롤백'까지 요구된다면 __exit__ (또는 yield 이후)에서 예외 발생 여부를 분기합니다. 

@contextmanager
def transaction(pool):
    conn = pool.acquire()
    try:
        yield conn
        conn.commit()       # 예외 없이 블록이 끝났을 때만 실행
    except Exception:
        conn.rollback()     # 예외가 나면 롤백 후 다시 던짐
        raise
    finally:
        pool.release(conn)  # 커밋/롤백과 무관하게 항상 반납

이 구조는 획득-사용-정리라는 대칭을 코드 상에서 명시적으로 드러내면서도, 호출자는 정리 로직을 전혀 신경 쓰지 않아도 되게 만듭니다. 만약 커넥션에 더해 트랜잭션 객체까지 중첩해서 관리해야 한다면, with get_connection(pool) as conn, transaction(conn) as tx:처럼 중첩해서 선언 역순으로 정리되도록 조합할 수도 있습니다."

약한 답변

"함수 안에서 커넥션을 받고, 쿼리 실행하고, 마지막에 pool.release(conn)을 호출하면 됩니다."

왜 약한가: 예외가 쿼리 실행 중에 발생하면 마지막 줄의 release가 실행되지 않고 커넥션이 누수된다. "반드시 반납되어야 한다"는 요구사항 자체를 만족하지 못하는 설계이며, context manager나 try/finally 같은 보장 메커니즘을 전혀 고려하지 않았다는 신호다.

전제 개념 ③ — 여러 context manager를 조합하고, 스택으로 동적으로 관리하는 법

실무 코드는 리소스 하나만 관리하는 경우보다, 여러 개를 동시에 또는 동적인 개수만큼 관리해야 하는 경우가 많다. 이때 with의 다중 리소스 문법과 ExitStack을 구분해서 아는 것이 시니어 시그널이다. 

# 여러 리소스를 한 with 문으로 — 선언 순서대로 진입, 역순으로 정리
with open("a.txt") as fa, open("b.txt") as fb:
    ...
    # 블록 종료 시: fb.__exit__() 먼저, 그다음 fa.__exit__() — 마지막에 연 것부터 닫힘
개수를 미리 알 수 없을 때 — with로는 표현이 안 된다
"요청마다 파일 개수가 다르다"처럼 런타임에 개수가 정해지는 리소스는 정적인 with a, b, c: 문법으로 쓸 수 없다. 이럴 때 contextlib.ExitStack을 쓴다. 
from contextlib import ExitStack

def process_files(paths):
    with ExitStack() as stack:
        files = [stack.enter_context(open(p)) for p in paths]
        # files 개수는 런타임에 결정됨 — 각각 enter_context로 스택에 등록
        # 블록이 끝나면 등록 역순으로 전부 정리됨(하나가 실패해도 나머지는 계속 정리 시도)
        for f in files:
            process(f)
ExitStack이 실무에서 특히 유용한 경우

조건부로 리소스를 여는 경우("설정에 따라 파일도 열고, 락도 걸 수도 있다")나, 리소스 개수가 입력에 따라 달라지는 경우 유용하다. 또한 stack.callback(cleanup_fn)으로 context manager가 아닌 일반 정리 함수도 같은 스택에 등록해서 정리 순서를 통일된 방식으로 관리할 수 있다.

면접 질문 ③ — with 블록 안에서 return이나 break가 나오면 __exit__는 실행되나요?

함수 안에서 with 블록을 쓰다가, 블록 중간에 return으로 함수를 빠져나가면 해당 context manager의 __exit__는 실행되나요? 반복문 안의 with에서 break가 나오면요?

인터뷰어가 평가하는 신호

  • with가 "정상 종료 / 예외 / return / break / continue" 등 블록을 빠져나가는 모든 경로에서 정리를 보장한다는 걸 아는가
  • 이 보장이 try/finally와 동일한 근거(스택 언와인딩 시점의 finally 실행)에서 나온다는 것을 아는가
  • 단, 프로세스가 강제 종료(os._exit(), kill -9)되는 경우는 예외라는 한계를 아는가
강한 답변

"네, 실행됩니다. with는 내부적으로 try/finally와 동등한 구조로 컴파일되기 때문에, 블록을 빠져나가는 방식이 무엇이든(정상 종료, 예외, return, break, continue) __exit__는 반드시 호출됩니다. return이 실행되면 파이썬 인터프리터는 스택을 '언와인딩'하면서 그 경로에 걸쳐 있는 모든 finally 블록(그리고 with의 암묵적 __exit__ 호출)을 먼저 실행한 뒤에 실제로 함수를 빠져나갑니다.

예외적인 건 프로세스 자체가 정상적인 파이썬 실행 흐름 밖에서 강제 종료되는 경우입니다. os._exit()나 kill -9, 세그멘테이션 폴트처럼 인터프리터가 정리할 기회 자체를 얻지 못하면 __exit__는 실행되지 않습니다. 그래서 '정말 중요한 외부 리소스(락 파일, 프로세스 간 세마포어 등)'는 파이썬 프로세스가 죽는 경우까지 고려한 타임아웃이나 워치독을 별도로 둬야 한다는 한계도 같이 언급하겠습니다."

약한 답변

"with 블록이 끝나면 항상 __exit__가 실행됩니다."

왜 약한가: 결론은 맞지만 왜 그런지(스택 언와인딩과 try/finally의 동등성)를 설명하지 못하고, 강제 종료처럼 이 보장이 깨지는 한계 상황을 전혀 언급하지 않아 피상적인 암기 답변으로 보인다.

Senior-level Follow-up 질문

Follow-up 1

비동기 코드에서는 context manager를 어떻게 다루나요? async with는 무엇이 다른가요?

__aenter__/__aexit__를 구현한 객체는 async with로 쓴다. 차이는 진입/종료 로직 자체가 비동기 작업(await 필요)일 수 있다는 점이다 — 예를 들어 커넥션 풀에서 커넥션을 빌리는 동작이 네트워크 핸드셰이크를 포함해 await가 필요하면 동기 __enter__로는 표현할 수 없다. Day 5에서 다룬 이벤트 루프 모델과 연결하면, __aexit__ 안에서 동기 블로킹 정리 작업을 하면 여전히 이벤트 루프를 멈춘다는 함정도 같이 짚어야 한다. contextlib.asynccontextmanager가 제너레이터 기반 @contextmanager의 비동기 버전이다.

Follow-up 2

__exit__가 예외를 의도적으로 삼켜야 하는 경우가 있을까요?

드물지만 있다. 예를 들어 "특정 재시도 가능한 예외는 여기서 로깅만 하고 흐름을 계속 진행시키고 싶다"는 요구가 있으면 __exit__에서 exc_type이 그 예외 타입에 해당할 때만 True를 반환하도록 설계할 수 있다. 하지만 이건 명시적으로 문서화하지 않으면 호출자가 예외가 사라진 이유를 알 수 없는 매우 위험한 패턴이므로, "특정 타입만, 명확한 이유로, 반드시 문서화"라는 조건을 붙여 답해야 한다.

Follow-up 3

Context manager와 Day 7에서 다룰 descriptor는 어떤 관계가 있나요?

직접적인 상속 관계는 없지만, 둘 다 "특정 dunder 메서드 쌍을 구현하면 파이썬 문법이 자동으로 호출해주는" 프로토콜 기반 설계라는 공통점이 있다. context manager는 __enter__/__exit__가 with 문에 의해, descriptor는 __get__/__set__이 속성 접근에 의해 자동 호출된다. 이 패턴을 인식하면 "파이썬이 왜 이렇게 많은 특수 메서드를 두는가"(Day 1의 data model과 직결)에 대한 답변이 훨씬 체계적으로 정리된다.

Follow-up 4

같은 context manager 인스턴스를 with에 두 번 중첩해서 재사용해도 되나요?

일반적으로 안 된다. 대부분의 context manager는 1회성 상태(파일 핸들, 획득한 락 등)를 __enter__에서 만들고 __exit__에서 정리하도록 설계되기 때문에, 재진입하면 이미 정리된 리소스를 다시 쓰려 하거나 상태가 꼬인다. @contextmanager로 만든 제너레이터 기반 context manager는 특히 재진입이 명시적으로 금지되어 있어 (`RuntimeError: generator didn't stop / already started`), 재사용이 필요하면 매번 새 인스턴스를 만들어야 한다는 걸 실무에서 자주 놓친다.

Trade-off와 흔한 오해

오해 1 — "with를 쓰면 가비지 컬렉션이 즉시 일어난다"

틀렸다. with가 보장하는 건 __exit__ 안에 작성된 정리 로직의 실행이지, 객체 자체의 메모리 해제가 아니다. Day 2에서 다뤘듯 CPython은 레퍼런스 카운트가 0이 되면 즉시 메모리를 회수하지만, 그건 with와 별개의 메커니즘이다. with는 "파일 디스크립터를 닫는다", "락을 푼다" 같은 명시적 정리 코드를 보장하는 것이지 객체 소멸 시점을 결정하는 게 아니다.

오해 2 — "__exit__의 반환값은 신경 안 써도 된다"

위험하다. 기본값처럼 아무 return도 안 쓰면 None이 반환되어 예외가 정상적으로 전파되지만, 실수로 return문 하나가 True를 반환하는 조건에 걸리면 예외가 조용히 삼켜진다. "정리 로직만 챙기면 된다"고 생각하고 반환값을 무심코 다루다가 프로덕션에서 예외가 사라지는 버그로 이어지는 경우가 실제로 많다.

오해 3 — "@contextmanager는 클래스 기반보다 항상 더 나은 선택이다"

상황에 따라 다르다. @contextmanager는 간결하지만, __enter__와 __exit__가 서로 다른 시점에 독립적으로 호출될 수 있어야 하는 경우(예: 객체를 미리 만들어두고 나중에 여러 번 with로 재사용하려는 설계 — 다만 앞서 봤듯 일반적으로 권장되지 않는다)나, __exit__에서 예외 타입별로 복잡한 분기가 필요한 경우엔 클래스 기반이 더 명확하다.

오해 4 — "여러 리소스를 with로 묶으면 정리 순서는 신경 안 써도 된다"

순서가 중요한 경우가 실무에 많다. with lock, transaction:처럼 쓰면 선언 역순으로 정리되므로 transaction이 먼저 정리되고 lock이 나중에 풀린다. 만약 반대로 "트랜잭션을 커밋하기 전에 락을 먼저 풀면 안 되는" 요구가 있다면 선언 순서를 신중하게 설계해야 하고, 이걸 놓치면 동시성 버그(Day 3 GIL과는 다른 층위의 레이스 컨디션)로 이어질 수 있다.

오늘 반드시 기억할 것
  1. with는 정리 책임을 호출자에서 리소스 작성자로 옮겨, 실수로 빠뜨릴 수 없게 만드는 프로토콜

  3. __exit__가 True를 반환하면 예외가 삼켜진다 — 의도치 않으면 항상 False/None
  4. @contextmanager는 제너레이터로 구현 — yield 전이 진입, 후가 정리, try/finally를 직접 챙겨야 함

  6. 비동기 리소스는 __aenter__/__aexit__ + async with — 정리 로직 안의 블로킹 호출은 여전히 이벤트 루프를 멈춤
