왜 면접에서 Python 메모리 관리를 묻는가

경력자 면접에서 메모리 관리를 묻는 이유는 두 가지다. 첫째, 메모리 누수를 직접 디버깅한 경험이 있는가를 확인하기 위해서다. 둘째, CPython의 동작 방식을 언어 수준에서 이해하는가를 테스트하기 위해서다.

Python은 GC가 있으니 메모리를 신경 안 써도 된다고 생각하는 개발자와, 순환 참조·__del__·대용량 객체·캐시 설계에서 메모리 문제를 실제로 다룰 수 있는 개발자를 구분하기 위한 핵심 주제다.

실무 연결
장기 실행 서비스(API 서버, 데이터 파이프라인, ML 훈련 루프)에서 메모리가 계속 늘어난다면 대부분 순환 참조 또는 캐시 설계 문제다. gc, tracemalloc, objgraph를 사용해 진단하고 수정한 경험이 있다면 이 레슨의 내용을 자연스럽게 말할 수 있어야 한다.

전제 개념 ① — 레퍼런스 카운팅(Reference Counting)

CPython은 모든 객체 안에 참조 횟수(reference count)를 숫자로 저장한다. 어떤 변수·컨테이너·스택 프레임이 이 객체를 가리킬 때마다 카운트가 1씩 오른다. 반대로 변수가 재할당되거나 스코프를 벗어나거나 del로 이름이 지워지면 카운트가 1씩 내려간다. 카운트가 0이 되는 순간 CPython은 즉시 그 객체의 메모리를 해제한다.

이것이 Python 메모리 관리의 1차 방어선이다. GC보다 훨씬 자주, 실시간으로 작동한다.

sys.getrefcount로 직접 확인하기

import sys

a = []
print(sys.getrefcount(a))  # 2 ← a 변수 + getrefcount 인수로 임시 참조

b = a
print(sys.getrefcount(a))  # 3 ← a, b, getrefcount 인수

del b
print(sys.getrefcount(a))  # 2 ← b 삭제 후 다시 줄어듦
주의: getrefcount는 항상 1 더 많다
sys.getrefcount(obj)를 호출하면 함수 인자로 넘기는 순간 임시 참조가 1 추가된다. 실제 참조 수는 항상 반환값 − 1이다.

레퍼런스 카운트를 높이는 상황들

  • 변수에 할당: x = obj
  • 리스트·딕셔너리·튜플 등 컨테이너에 담길 때
  • 함수 인자로 전달될 때 (스택 프레임이 참조)
  • 클로저가 변수를 캡처할 때
  • 클래스 인스턴스의 속성으로 설정될 때

레퍼런스 카운팅의 한계 — 순환 참조

레퍼런스 카운팅의 결정적 한계는 순환 참조(circular reference)를 해결할 수 없다는 것이다. 두 객체가 서로를 참조하면 외부에서 두 변수를 삭제해도 내부적으로 카운트가 1씩 남아 0이 되지 않는다. 

class Node:
    def __init__(self, name):
        self.name = name
        self.other = None

a = Node("A")
b = Node("B")
a.other = b   # a → b
b.other = a   # b → a  (순환 참조 발생)

del a
del b
# 외부 참조(a, b 변수)는 사라졌지만
# 두 객체가 서로를 가리키므로 refcount = 1씩 남아 있음
# → 레퍼런스 카운팅만으로는 메모리 해제 불가
이것이 GC가 필요한 이유
순환 참조로 인한 메모리 누수를 잡기 위해 CPython은 레퍼런스 카운팅 외에 가비지 컬렉터(cyclic garbage collector)를 별도로 운영한다.

면접 질문 ① — Python의 메모리 관리 방식

Python은 메모리를 어떻게 관리하나요? GC가 있는데 왜 메모리 누수가 발생할 수 있는지 설명해보세요.

인터뷰어가 평가하는 신호

  • 레퍼런스 카운팅과 GC를 둘 다 언급하는가 (둘은 협력 관계)
  • 순환 참조가 왜 레퍼런스 카운팅으로 처리 안 되는지 구조적으로 설명하는가
  • 실무에서 메모리 누수를 직접 경험했거나 디버깅한 사례를 언급하는가

답변 구조 — 생각의 순서

  1. 1차 메커니즘: 레퍼런스 카운팅 — 실시간, 카운트=0이면 즉시 해제
  2. 한계: 순환 참조는 카운트가 0이 안 됨
  3. 2차 메커니즘: cyclic GC — 순환 참조만 전담, 주기적 실행
  4. 그래도 누수가 생기는 경우: 전역 컨테이너, 클로저, __del__이 있는 순환 참조(옛날 Python)
강한 답변

"CPython의 메모리 관리는 두 층으로 작동합니다. 1차는 레퍼런스 카운팅으로, 모든 객체가 자신을 가리키는 참조 수를 내부에 저장합니다. 카운트가 0이 되면 즉시 메모리가 해제됩니다. 이 방식은 deterministic하고 빠르지만, 순환 참조를 처리하지 못합니다. 두 객체가 서로를 참조하면 외부 변수를 모두 삭제해도 내부 카운트가 1씩 남아 있어 해제되지 않습니다.

이를 해결하기 위해 CPython은 cyclic garbage collector를 별도로 운영합니다. GC는 generation-based로 동작하며, 순환 참조 그래프를 탐색해서 도달 불가능한 그룹을 해제합니다.

GC가 있어도 메모리 누수가 발생하는 대표적인 상황은 전역 딕셔너리나 클래스 속성에 객체를 쌓아두는 경우입니다. GC 입장에서 이 객체들은 전역 루트에서 도달 가능하기 때문에 살아있는 객체로 간주합니다. 장기 실행 서버에서 request별 데이터를 전역 캐시에 계속 추가하면서 만료 처리를 하지 않는 패턴이 대표적입니다."

약한 답변

"Python은 가비지 컬렉터가 있어서 메모리를 자동으로 관리합니다. 메모리 누수는 GC가 처리 못 하는 순환 참조 때문입니다."

왜 약한가: 레퍼런스 카운팅을 전혀 언급하지 않았다. Python 메모리 관리의 주역을 빠뜨린 것이다. "전역 컨테이너" 같은 실무 연결도 없고, GC가 실제로 어떻게 동작하는지도 설명하지 못했다.

전제 개념 ② — cyclic Garbage Collector

Python의 gc 모듈이 제공하는 GC는 순환 참조 전용이다. 일반 객체의 해제는 레퍼런스 카운팅이 담당하고, GC는 레퍼런스 카운팅이 잡지 못하는 "순환 그래프"만 처리한다.

세대(generation) 기반 수집

CPython GC는 3개의 세대(generation 0, 1, 2)로 객체를 관리한다. 새로 생성된 객체는 generation 0에 들어간다. 수집(collection)에서 살아남으면 더 오래된 세대로 이동한다. 오래된 세대는 덜 자주 수집되므로 장수하는 객체에 대한 GC 비용이 줄어든다. 

import gc

# 각 세대의 현재 객체 수 확인
print(gc.get_count())    # e.g. (700, 10, 1)

# 수집 임계값 (각 세대가 이 수를 넘으면 수집 트리거)
print(gc.get_threshold())  # (700, 10, 10) 기본값

# 수동으로 전체 수집 (프로파일링/디버깅용)
gc.collect()

# 특정 세대만 수집
gc.collect(0)  # generation 0만

GC 동작 원리 — 도달 가능성 분석

GC는 "외부에서 도달 가능한가(reachable)"를 기준으로 살아있는 객체를 판별한다. 전역 변수, 스택 프레임, 모듈 등 루트(root)에서 참조 체인을 따라 닿을 수 없는 객체가 수집 대상이 된다.

CPython GC가 추적하는 객체 타입
GC는 컨테이너 타입(list, dict, set, tuple, 사용자 정의 클래스 등)만 추적한다. int, str, float 같은 단순 스칼라는 다른 객체를 참조할 수 없으므로 GC 추적 대상에서 제외한다. 이를 통해 GC 오버헤드를 최소화한다.

gc 모듈로 순환 참조 감지하기

import gc

class Node:
    pass

a = Node(); b = Node()
a.ref = b; b.ref = a   # 순환 참조

del a, b
print(gc.collect())   # 2 — 2개 객체 수집됨

# 누수 진단: 수집 불가능한 객체 목록
gc.set_debug(gc.DEBUG_SAVEALL)
gc.collect()
print(gc.garbage)   # __del__이 있는 순환 참조 객체 등이 여기에 쌓임

면접 질문 ② — 순환 참조와 weakref

순환 참조가 왜 문제인지, 그리고 이를 해결하기 위한 방법에 대해 설명해보세요. weakref는 언제 사용하나요?

인터뷰어가 평가하는 신호

  • 순환 참조가 레퍼런스 카운팅을 무력화하는 메커니즘을 정확히 설명하는가
  • weakref의 동작 원리(참조 카운트를 증가시키지 않음)를 아는가
  • 패턴별 실무 사례(캐시, 부모-자식 트리, 이벤트 리스너)를 언급하는가
  • weakref의 한계(참조 대상이 GC되면 None이 됨)를 아는가

weakref 개념 — "연약한 참조"

일반 참조는 레퍼런스 카운트를 올린다. 약한 참조(weak reference)는 객체를 "가리키되 카운트는 올리지 않는다". 즉, weak ref만 남아 있으면 GC가 그 객체를 수집할 수 있다. 

import weakref

class Node:
    def __init__(self, name):
        self.name = name
        self.parent = None    # 부모를 weak ref로 참조
        self.children = []

class TreeNode(Node):
    def add_child(self, child):
        child.parent = weakref.ref(self)  # 카운트 증가 없이 참조
        self.children.append(child)

root = TreeNode("root")
child = TreeNode("child")
root.add_child(child)

# weak ref 사용: 항상 callable — None 체크 필요
parent_ref = child.parent
if parent_ref is not None:
    parent = parent_ref()   # () 호출로 실제 객체 접근
    if parent is not None:  # 이미 GC됐을 수 있음
        print(parent.name)

weakref 사용 패턴

패턴 1 — 캐시 (WeakValueDictionary)

캐시가 객체의 유일한 참조가 되면 객체를 영원히 살려두게 된다. weakref.WeakValueDictionary를 쓰면 다른 강한 참조가 없는 객체를 자동으로 캐시에서 제거한다. 

import weakref

cache = weakref.WeakValueDictionary()

def get_resource(key):
    resource = cache.get(key)
    if resource is None:
        resource = load_expensive_resource(key)
        cache[key] = resource
    return resource
# 호출자가 resource를 버리면 캐시에서도 자동 제거됨
패턴 2 — 부모-자식 트리 / 이벤트 핸들러

자식이 부모를 강하게 참조하면 순환 참조가 생긴다(부모 → 자식 리스트, 자식 → 부모). 이벤트 시스템에서 리스너가 이벤트 소스를 강하게 참조하는 것도 같은 문제다. 이 방향에 weakref.ref를 쓰면 순환을 끊을 수 있다.

강한 답변

"순환 참조는 두 개 이상의 객체가 서로를 참조할 때 발생합니다. 외부의 모든 변수를 삭제해도 서로 카운트를 1씩 유지하기 때문에 레퍼런스 카운팅으로는 해제되지 않습니다. CPython GC가 주기적으로 이 사이클을 감지해서 해제하지만, GC 실행 타이밍이 비결정적이고 오버헤드가 있습니다.

근본적인 해결책은 참조 방향 중 하나를 weakref로 교체하는 것입니다. weakref.ref는 참조 카운트를 올리지 않아서 순환이 끊깁니다. 실무에서는 부모-자식 트리에서 자식의 부모 참조, 이벤트 리스너가 소스를 역참조하는 경우, WeakValueDictionary를 이용한 캐시 설계에 주로 씁니다.

주의할 점은 weak ref의 대상이 GC되면 ref()가 None을 반환한다는 것입니다. 그래서 weak ref를 역참조할 때는 항상 None 체크가 필요합니다."

약한 답변

"순환 참조를 막으려면 weakref를 쓰면 됩니다. weakref는 참조는 하지만 GC가 해제할 수 있게 해줍니다."

왜 약한가: 왜 레퍼런스 카운팅이 순환을 처리하지 못하는지 설명하지 못했다. weakref가 구체적으로 어떤 패턴에서 쓰이는지, None 체크 같은 주의사항도 빠졌다.

전제 개념 ③ — __del__의 실제 동작과 위험

__del__은 객체가 소멸될 때 호출되는 finalizer다. C++의 소멸자처럼 쓰려는 시도가 많지만, Python에서는 매우 주의가 필요하다.

__del__이 호출되는 시점

  • 레퍼런스 카운트가 0이 되는 즉시 (CPython — deterministic)
  • GC가 순환 참조 그룹을 수집할 때 (non-deterministic, 별도 스레드에서 일어날 수 있음)
  • 인터프리터 종료 시 (모듈 전역이 이미 None으로 바뀐 상태일 수 있음)
__del__의 위험 — 순환 참조와 gc.garbage

Python 3.4 이전에는 __del__이 정의된 객체가 순환 참조에 포함되면 GC가 수집 순서를 결정할 수 없어 영원히 해제하지 못하고 gc.garbage에 쌓았다. Python 3.4+에서는 PEP 442로 이 문제가 완화되었지만, __del__의 실행 순서는 여전히 불확실하다. 

class Resource:
    def __init__(self, name):
        self.name = name
        print(f"open: {self.name}")

    def __del__(self):
        # 위험: 여기서 모듈 수준 이름이 이미 None일 수 있음
        # 예: 인터프리터 종료 시 open, print가 None으로 됨
        print(f"close: {self.name}")  # TypeError 가능성

# 올바른 방법: __del__ 대신 context manager 사용
class Resource:
    def __enter__(self): return self
    def __exit__(self, *args):
        self.close()
    def close(self):
        print("cleaned up")
실무 권장 패턴
리소스 정리는 __del__ 대신 __enter__/__exit__(context manager)로, 또는 try/finally로 명시적으로 처리하라. __del__은 "최후의 안전망"으로는 쓸 수 있지만 주된 정리 경로로 믿으면 안 된다.

면접 질문 ③ — __del__을 리소스 정리에 쓰면 안 되는 이유

__del__을 파일이나 네트워크 연결 정리에 쓰면 어떤 문제가 생기나요? 대신 어떤 패턴을 권장하나요?

인터뷰어가 평가하는 신호

  • __del__ 호출 시점이 보장되지 않는 이유(GC vs 레퍼런스 카운팅)를 구분해서 설명하는가
  • 인터프리터 종료 시 전역이 None이 되는 문제를 아는가
  • context manager를 대안으로 정확히 제시하는가
강한 답변

"__del__은 세 가지 이유로 리소스 정리에 적합하지 않습니다.

첫째, 호출 시점이 보장되지 않습니다. 레퍼런스 카운팅으로 즉시 해제되면 괜찮지만, 순환 참조에 포함된 경우 GC가 실행될 때까지 지연됩니다. 파일 핸들이나 DB 연결이 늦게 닫히면 리소스 고갈로 이어집니다.

둘째, 인터프리터 종료 시 전역 이름이 None이 될 수 있습니다. 모듈이 먼저 정리되면 __del__ 안에서 참조하는 전역 함수나 모듈이 None으로 바뀌어 TypeError가 발생할 수 있습니다.

셋째, 예외가 묵살됩니다. __del__ 안에서 발생한 예외는 무시되고 stderr에만 경고가 찍혀 디버깅이 매우 어렵습니다. 올바른 패턴은 with 문(context manager)으로 명시적 정리 경계를 만드는 것입니다. __del__은 '최후의 안전망' 수준으로만 쓰고, 주된 정리 경로로는 절대 신뢰하지 말아야 합니다."

약한 답변

"파이썬에서 __del__은 GC가 호출해서 타이밍이 불확실하기 때문에 with문을 쓰는 게 낫습니다."

왜 약한가: 인터프리터 종료 문제, 예외 묵살, 레퍼런스 카운팅과 GC의 차이 같은 구체적인 이유를 하나도 언급하지 않았다. "타이밍이 불확실하다"는 결론만 있고 근거가 없다.

메모리 최적화 — 실무에서 쓰는 기법

__slots__으로 인스턴스 메모리 절감

Python 클래스는 기본적으로 인스턴스마다 __dict__를 생성한다. 수백만 개의 인스턴스를 만들면 각 __dict__의 오버헤드가 크다. __slots__을 쓰면 __dict__ 없이 고정된 속성만 저장해서 메모리를 절약한다. 

import sys

class Point:
    def __init__(self, x, y):
        self.x = x; self.y = y

class PointSlot:
    __slots__ = ('x', 'y')
    def __init__(self, x, y):
        self.x = x; self.y = y

p1 = Point(1, 2)
p2 = PointSlot(1, 2)

print(sys.getsizeof(p1))  # ~56 bytes (+ __dict__ ~232 bytes)
print(sys.getsizeof(p2))  # ~48 bytes (__dict__ 없음)
# 수백만 개 인스턴스에서 수십~수백 MB 차이 가능
__slots__ 주의사항
  • __slots__이 있으면 __dict__가 없으므로 동적으로 속성을 추가할 수 없다.
  • 상속 시 부모의 __slots__을 자식도 명시해야 효과가 있다 (부모에 __dict__가 있으면 자식도 가짐).
  • pickle, copy, 일부 ORM 프레임워크와 충돌할 수 있다.

tracemalloc으로 누수 위치 추적

import tracemalloc

tracemalloc.start()

# 의심 코드 실행
data = [{'key': i} for i in range(100_000)]

snapshot = tracemalloc.take_snapshot()
top_stats = snapshot.statistics('lineno')

for stat in top_stats[:5]:
    print(stat)
# example.py:6: size=7.6 MiB, count=100003, average=79 B

Senior-level Follow-up 질문

Follow-up 1

"Python 3.12에서 GIL을 선택적으로 끌 수 있게 되었는데, 레퍼런스 카운팅은 어떻게 바뀌나요?"

GIL이 없으면 멀티스레드 환경에서 레퍼런스 카운트 증감이 race condition이 된다. No-GIL Python(PEP 703)은 biased reference counting + per-object lock을 써서 이를 해결한다. 레퍼런스 카운팅이 GIL과 얼마나 깊이 연결된 개념인지를 답하는 질문이다.

Follow-up 2

"CPython의 small int cache와 string interning은 메모리 관리와 어떤 관계가 있나요?"

CPython은 -5~256 정수와 식별자 형태의 문자열을 캐싱(intern)한다. 즉 같은 값이면 동일 객체를 재사용한다. is와 ==를 혼동하는 원인이 여기서 나온다. 이를 아는 후보는 Python 최적화 구현 세부 사항까지 이해하는 사람으로 평가된다.

Follow-up 3

"장기 실행 Python 서버에서 메모리가 계속 증가할 때 어떤 순서로 디버깅하나요?"

1) tracemalloc으로 할당 위치 파악 → 2) objgraph.show_growth()로 늘어나는 타입 확인 → 3) gc.collect() 후에도 남으면 순환 참조 또는 전역 캐시 점검 → 4) weakref·WeakValueDictionary 또는 LRU 캐시(functools.lru_cache)로 교체. 이 흐름을 체계적으로 말하면 실무 경험을 증명한다.

Trade-off와 흔한 오해

오해 1 — "GC를 끄면 성능이 좋아진다"

gc.disable()을 쓰는 패턴을 가끔 본다. 단기 배치 작업처럼 순환 참조를 만들지 않는 코드에서는 효과가 있을 수 있지만, 일반 서버 애플리케이션에서는 순환 참조가 축적되어 메모리가 무제한 증가한다. Instagram은 GC를 끄되, 각 요청 처리 후 gc.freeze()를 써서 안정적인 객체를 고정시키는 방식을 사용했다. ("Dismissing Python Garbage Collection at Instagram" 참고)

오해 2 — "del x는 메모리를 즉시 해제한다"

del x는 이름 바인딩을 제거하는 것이지, 메모리를 직접 해제하는 것이 아니다. 해제는 레퍼런스 카운트가 0이 될 때 일어난다. 다른 변수나 컨테이너에 동일 객체가 살아있으면 del x는 메모리에 아무 영향이 없다.

오해 3 — "PyPy, Jython에도 레퍼런스 카운팅이 있다"

레퍼런스 카운팅은 CPython의 구현 방식이다. PyPy는 tracing GC를, Jython은 JVM GC를 쓴다. "Python의 메모리 관리"가 아니라 "CPython의 메모리 관리"라는 점을 인지하는 후보가 더 높은 평가를 받는다.

오늘 반드시 기억할 것
  1. CPython 메모리 관리 = 레퍼런스 카운팅(1차) + cyclic GC(2차)
  2. 레퍼런스 카운팅은 빠르고 deterministic하지만 순환 참조를 처리 못한다

  5. __del__은 리소스 정리에 쓰지 말고 context manager를 쓸 것
  6. 메모리 누수 진단: tracemalloc → objgraph → gc.collect → weakref/LRU 교체
