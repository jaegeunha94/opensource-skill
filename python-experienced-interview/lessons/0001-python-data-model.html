Day 1

Python Data Model
__dunder__ 메서드와 객체 프로토콜

파이썬 경력자 Interview · 2026-06-29 · 약 45–60분 소요

왜 면접에서 Python Data Model을 묻는가

Python data model은 Python의 "철학이 언어에 어떻게 구현되었는가"를 드러낸다. 인터뷰어가 data model을 묻는 이유는 단순히 API 암기 여부를 테스트하는 것이 아니라, 지원자가 Python이 어떻게 동작하는지를 내부부터 이해하는가를 평가하기 위해서다.

경력자라면 커스텀 클래스를 설계할 때 Python의 내장 기능(연산자, len(), in, for, with)을 자연스럽게 통합해야 한다. 이 통합의 핵심 메커니즘이 data model이다.

실무 연결
커스텀 컨테이너, ORMs, API 응답 래퍼, 설정 객체, 수학적 타입(벡터, 행렬)을 직접 만들 때 data model을 모르면 Python답지 않은 코드가 된다. 예: len(my_obj) 대신 my_obj.size()를 써서 언어 관용구를 깨는 코드.

전제 개념부터 — Python Data Model이란 무엇인가

Python의 모든 것은 객체다. 그리고 Python 인터프리터는 특정 상황에서 객체의 특별한 이름을 가진 메서드를 자동으로 호출한다. 이 메서드들의 이름은 앞뒤에 이중 언더스코어(__)가 붙어 있어서 "dunder(double underscore) 메서드" 또는 "magic method"라고 부른다.

Python Data Model = Python 인터프리터와 사용자 클래스 사이의 계약(contract)이다. 이 계약을 구현하면 내장 함수·연산자·문법이 커스텀 클래스에서도 동일하게 동작한다.

핵심 정의

Python Data Model은 Python 언어가 제공하는 객체 프로토콜 명세다. __init__, __repr__, __len__, __getitem__ 등의 dunder 메서드를 구현함으로써 객체가 Python 내장 연산과 통합된다.

공식 참조: Python Language Reference → Data Model (docs.python.org)

인터프리터가 dunder를 호출하는 방식

중요한 점: Python은 항상 클래스에서 dunder를 찾는다, 인스턴스에서 찾지 않는다. 이는 메타클래스와 descriptor 프로토콜과 연결되는 핵심 구현 세부 사항이다. 

# 사용자가 쓰는 코드
len(my_list)

# 인터프리터가 실제로 하는 일 (단순화)
type(my_list).__len__(my_list)  # 인스턴스 아닌 클래스에서 찾는다
흔한 오해
"dunder를 인스턴스에 monkey-patch해도 동작한다"고 생각하는 경우가 있다. 특수 메서드(special method)에 한해서는 클래스 레벨에서 정의된 것만 인터프리터가 인식한다. 인스턴스 __dict__에 직접 넣어도 무시된다.

면접 질문 ① — __repr__과 __str__의 차이

Python에서 __repr__과 __str__의 차이를 설명하고, 커스텀 클래스를 만들 때 어떻게 구현해야 하는지 말해보세요.

인터뷰어가 평가하는 신호

  • str()과 repr()이 언제 각각 호출되는지 아는가
  • __repr__의 목적(디버깅/재현)과 __str__의 목적(사용자 표시)을 구분하는가
  • Fall-back 관계를 아는가 (__str__이 없으면 __repr__ 사용)

전제 개념 — 언제 무엇이 호출되는가

호출 방식사용 메서드기본값 (미구현 시)
str(obj), print(obj), f-string f"{obj}"__str____repr__로 fall-back
repr(obj), REPL 출력, f"{obj!r}"__repr__object.__repr__ (메모리 주소 포함 기본값)
컨테이너 안 요소 출력 (list, dict 등)__repr__—
왜 컨테이너는 항상 __repr__인가
[1, "hello", my_obj]를 출력할 때, Python은 각 요소에 repr()을 쓴다. 이는 컨테이너 내 출력이 항상 "재현 가능한 표현"이어야 한다는 Python 철학 때문이다. __str__만 정의하면 리스트 안에서 사람이 읽기 불편한 기본 repr이 나온다.

모범 답변 구조

강한 답변 예시

"__repr__은 개발자를 위한 표현으로, 가능하면 해당 객체를 재현할 수 있는 Python 표현식처럼 작성합니다. REPL에서 표시되거나 디버깅에 쓰입니다. __str__은 최종 사용자를 위한 표현으로, 읽기 편한 문자열을 반환합니다. __str__이 정의되지 않으면 Python은 __repr__로 fall-back하지만, 역방향 fall-back은 없습니다.

실무에서는 최소한 __repr__은 항상 구현합니다. @dataclass는 자동으로 repr을 생성하지만, 커스텀 클래스에서는 직접 작성해야 로그와 디버거에서 의미있는 정보를 볼 수 있습니다. 또한 eval(repr(obj)) == obj를 만족시키도록 설계하는 것이 이상적이지만, 이 계약은 강제되지 않으며 복잡한 클래스에서는 현실적으로 어렵습니다."

약한 답변 예시 (왜 약한가)

"__str__은 문자열 변환이고 __repr__은 객체 정보를 보여줍니다."

✗ Fall-back 관계를 모른다. 언제 어느 것이 호출되는지 설명 없음. 컨테이너에서 repr이 쓰인다는 것을 모른다. 실무 적용(로깅, 디버깅, eval 계약)에 대한 언급이 없다.

코드 예시

class Point:
    def __init__(self, x: float, y: float) -> None:
        self.x = x
        self.y = y

    def __repr__(self) -> str:
        # eval(repr(p)) == p 가 성립하도록 작성
        return f"Point(x={self.x!r}, y={self.y!r})"

    def __str__(self) -> str:
        # 사람이 읽기 쉬운 형식
        return f"({self.x}, {self.y})"

p = Point(1.0, 2.0)
print(repr(p))    # Point(x=1.0, y=2.0)
print(str(p))     # (1.0, 2.0)
print([p])        # [Point(x=1.0, y=2.0)]  ← repr 사용

Senior-level follow-up

Q: __format__은 무엇이고 언제 필요한가?

f"{obj:.2f}"처럼 포맷 스펙을 줄 때 호출된다. __format__(self, format_spec: str)을 구현하지 않으면 format_spec이 있을 때 기본 __str__에서 오류가 발생할 수 있다. 수치 타입이나 날짜 타입 등 포맷이 중요한 클래스에서 구현한다.

Q: logging 모듈에서 lazy evaluation을 위해 어떤 dunder가 중요한가?

logging.debug("%s", obj)는 DEBUG 레벨이 비활성화면 str(obj)를 호출하지 않는다. 반면 logging.debug(f"{obj}")는 항상 __str__을 호출한다. 대형 객체나 연산 비용이 있는 __str__이 있을 때 성능 차이로 이어진다.

면접 질문 ② — __eq__와 __hash__의 계약

Python에서 __eq__를 오버라이드할 때 __hash__를 함께 정의해야 하는 이유를 설명하세요. 어떤 규칙을 따라야 하며, 잘못 구현하면 어떤 문제가 생기나요?

인터뷰어가 평가하는 신호

  • dict와 set의 내부 동작(해시 테이블)을 이해하는가
  • __eq__만 정의했을 때 Python이 __hash__를 None으로 만드는 이유를 아는가
  • 해시 불변성(hash invariant) 계약을 설명할 수 있는가
  • 가변 객체의 해시 가능성 문제를 인식하는가

전제 개념 — 해시 테이블이란

Python의 dict와 set은 내부적으로 해시 테이블로 구현된다. 키를 찾을 때는 먼저 hash(key)로 버킷을 찾고, 그 버킷 안에서 ==으로 정확한 키를 찾는다.

이 때문에 핵심 불변식(invariant)이 생긴다:

해시 계약 (Hash Contract)

a == b 이면 반드시 hash(a) == hash(b) 이어야 한다.

역은 성립하지 않는다: hash(a) == hash(b)여도 a != b일 수 있다(해시 충돌). 하지만 a == b인데 hash(a) != hash(b)라면 dict/set이 같은 키를 다른 버킷에서 찾으므로 찾지 못한다.

Python의 자동 동작 — 왜 __hash__가 None이 되는가

class BadPoint:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)
    # __hash__ 미정의!

p = BadPoint(1, 2)
hash(p)     # TypeError: unhashable type: 'BadPoint'
{p}         # TypeError: unhashable type: 'BadPoint'
Python 규칙 (Python Language Reference)
클래스에서 __eq__를 정의하면, Python은 자동으로 __hash__를 None으로 설정한다. 이는 의도적이다: __eq__를 바꿨는데 __hash__를 그대로 두면 계약이 깨질 수 있으므로, Python이 강제로 "hashable 아님"을 만든다.

올바른 구현

class Point:
    def __init__(self, x: float, y: float) -> None:
        self.x = x
        self.y = y

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Point):
            return NotImplemented  # bool 아닌 NotImplemented 반환
        return (self.x, self.y) == (other.x, other.y)

    def __hash__(self) -> int:
        # __eq__에서 비교하는 필드를 tuple로 묶어 hash
        return hash((self.x, self.y))

p1 = Point(1.0, 2.0)
p2 = Point(1.0, 2.0)
print(p1 == p2)           # True
print(hash(p1) == hash(p2))  # True — 계약 만족
print({p1, p2})           # {Point(x=1.0, y=2.0)} — 중복 제거됨

가변 객체의 딜레마

가변 객체(필드가 바뀔 수 있는 클래스)는 원칙적으로 해시 불가능하게 만들어야 한다. 필드가 바뀌면 hash 값이 바뀌어 dict/set에서 영구히 찾을 수 없게 되기 때문이다. 

class MutablePoint:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)

    # 가변이므로 명시적으로 unhashable 선언
    __hash__ = None
Trade-off

가변 객체를 hashable로 만들고 싶다면 identity 기반 hash를 쓸 수 있다: hash(id(self)). 이 경우 __eq__도 identity 비교여야 일관성이 유지된다. 즉 내용이 같아도 다른 객체면 hash가 다르다. 이는 list가 아닌 dict처럼 참조로 식별해야 하는 객체에 적합하다.

Python의 기본 object가 바로 이 방식이다: 기본 __eq__는 identity, 기본 __hash__는 id() 기반.

모범 답변 구조

강한 답변 예시

"Python의 dict와 set은 해시 테이블로 구현되어 있어서 키를 조회할 때 먼저 hash(key)로 버킷을 찾고, 그 버킷에서 ==으로 정확한 키를 찾습니다. 이 때문에 'a == b이면 반드시 hash(a) == hash(b)'라는 계약이 있습니다.

__eq__를 정의하면 Python이 자동으로 __hash__를 None으로 설정해서 이 클래스를 unhashable로 만듭니다. 계약이 깨질 수 있는 상황을 언어가 강제 방지하는 것입니다.

올바른 구현은 __eq__에서 비교하는 필드와 동일한 필드를 tuple로 묶어 hash()에 넘기는 것입니다. 가변 객체는 __hash__ = None으로 명시적으로 unhashable로 선언하거나, identity 기반 해시를 써야 합니다."

약한 답변 예시 (왜 약한가)

"__eq__를 쓰면 __hash__도 구현해야 한다고 들었습니다."

✗ 왜 그래야 하는지 설명이 없다. 해시 테이블 동작 원리 모름. 계약 위반시 실제 버그(dict에서 키를 못 찾는 상황)를 설명하지 못한다. NotImplemented를 반환해야 하는 이유도 모른다.

Senior-level follow-up

Q: __eq__에서 False가 아닌 NotImplemented를 반환하는 이유는?

NotImplemented를 반환하면 Python이 반대쪽 객체의 __eq__를 시도한다 (reflected operation). 예: a == b에서 a.__eq__(b)가 NotImplemented를 반환하면, Python은 b.__eq__(a)를 시도한다. False를 반환하면 이 기회가 없다. 다른 라이브러리 타입(예: numpy 배열)과 비교할 때 올바른 결과를 얻으려면 NotImplemented를 써야 한다.

Q: @dataclass(eq=True, frozen=True)는 무엇을 해주는가?

frozen=True면 dataclass의 필드를 수정할 수 없게 하고 자동으로 __hash__를 생성한다. eq=True(기본값)만 쓰면 __hash__ = None이 된다. 불변 값 객체(value object) 패턴을 만들 때 frozen=True가 핵심이다.

면접 질문 ③ — 시퀀스 프로토콜과 __len__ / __getitem__

커스텀 클래스를 Python의 시퀀스처럼 동작하게 만들려면 어떤 메서드를 구현해야 하나요? __len__과 __getitem__만 구현해도 충분한가요?

인터뷰어가 평가하는 신호

  • 시퀀스 프로토콜의 최소 요구사항을 아는가
  • __getitem__만 있어도 iteration이 가능한 이유를 아는가
  • __contains__, __reversed__ 등 선택적 메서드를 아는가
  • collections.abc.Sequence를 활용하는 방법을 아는가

상세 설명

Python의 iteration 메커니즘은 두 가지 프로토콜 중 하나를 찾는다:

  1. Iterator 프로토콜: __iter__ (→ iterator 객체 반환)
  2. 구식 시퀀스 프로토콜: __getitem__만 있어도 Python이 0부터 시작해서 IndexError가 날 때까지 반복
class SquareSequence:
    def __init__(self, n: int) -> None:
        self.n = n

    def __len__(self) -> int:
        return self.n

    def __getitem__(self, index: int) -> int:
        if index >= self.n:
            raise IndexError("index out of range")
        return index ** 2

sq = SquareSequence(5)
print(list(sq))       # [0, 1, 4, 9, 16] — __iter__ 없이도 됨
print(3 in sq)         # True — __contains__ 없이 선형 탐색으로 fallback
print(len(sq))        # 5
성능 함정
__iter__를 구현하지 않으면 in 연산자가 선형 탐색을 한다. 정렬된 시퀀스나 해시 기반 컬렉션이라면 __contains__를 별도 구현해서 O(log n) 또는 O(1) 탐색을 제공해야 한다.

collections.abc.Sequence 활용

from collections.abc import Sequence

class MySeq(Sequence):
    def __init__(self, data):
        self._data = data

    def __getitem__(self, index):
        return self._data[index]

    def __len__(self):
        return len(self._data)
    # Sequence ABC가 count(), index(), __contains__(), __iter__(), __reversed__() 자동 제공

collections.abc.Sequence를 상속하면 __getitem__과 __len__만 구현해도 나머지 시퀀스 메서드가 자동으로 구현된다. 경력자라면 이 ABC를 활용할 줄 알아야 한다.

강한 답변 예시

"__getitem__과 __len__을 구현하면 Python이 시퀀스처럼 취급합니다. __iter__ 없이도 Python이 0부터 IndexError까지 반복해서 iteration을 지원합니다. 하지만 in 연산자는 __contains__가 없으면 선형 탐색으로 fallback되어 성능 문제가 생길 수 있습니다.

실무에서는 collections.abc.Sequence를 상속해서 __getitem__과 __len__만 구현하는 것이 권장됩니다. 나머지 메서드를 ABC가 자동으로 제공하고, 타입 체커에서도 올바르게 인식됩니다."

Senior-level follow-up

Q: __getitem__이 슬라이싱을 지원하려면 무엇을 해야 하는가?

__getitem__의 index 인자가 slice 객체일 수 있다. isinstance(index, slice)로 분기해서 처리하거나, 내부 list에 위임하면 자동으로 처리된다. slice.indices(len(self))로 start, stop, step을 정규화할 수 있다.

면접 질문 ④ — __bool__과 암묵적 진리값 평가

Python에서 if my_obj:가 어떻게 동작하는지 설명하세요. 어떤 우선순위로 메서드가 호출되나요?

상세 설명

Python의 진리값 평가(truthiness) 우선순위:

  1. __bool__이 있으면 호출 — True/False 반환
  2. __bool__이 없고 __len__이 있으면 — len() != 0이면 True
  3. 둘 다 없으면 — 항상 True (기본 object 동작)
class Container:
    def __init__(self, items):
        self.items = items

    def __len__(self):
        return len(self.items)

c = Container([])
print(bool(c))    # False — __bool__ 없으므로 __len__ 기반

class AlwaysFalse:
    def __bool__(self):
        return False

    def __len__(self):
        return 100  # __bool__이 있으므로 이 값은 무시됨

print(bool(AlwaysFalse()))  # False
실무 함정
ORM 쿼리셋(Django QuerySet 등)은 __len__을 호출하면 즉시 DB 쿼리가 실행된다. if queryset:은 __bool__이 없으면 len(queryset)을 호출해서 전체 결과를 메모리에 올린다. Django는 이를 위해 __bool__을 별도 구현해 __len__과 다른 경로로 최적화한다.

자기 점검 Rubric

평가 항목상중하
__repr__ vs __str__ 상 fall-back 관계, 컨테이너 repr, eval 계약, logging lazy eval까지 설명 중 각 메서드 역할과 fall-back 방향 설명 하 "문자열 반환"만 아는 수준
__eq__ + __hash__ 계약 상 해시 테이블 원리, 자동 None 이유, NotImplemented 용도, 가변 객체 딜레마까지 중 계약(a==b → hash 같음), __hash__=None 이유 설명 하 "같이 써야 한다"만 아는 수준
시퀀스 프로토콜 상 ABC 활용, __contains__ 성능, 슬라이스 처리까지 중 __getitem__만으로 iteration 가능 이유 설명 하 __len__, __getitem__ 이름만 아는 수준
