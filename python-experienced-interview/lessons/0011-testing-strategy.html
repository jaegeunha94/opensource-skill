Testing 전략
pytest fixture, mock, property-based testing, 테스트 격리

파이썬 경력자 Interview · 2026-07-09 · 약 70–90분 소요

왜 면접에서 테스트 전략을 묻는가

assert 1 + 1 == 2 같은 테스트를 짤 줄 아는가는 경력자 인터뷰에서 거의 묻지 않는다. 면접관이 실제로 궁금해하는 건 "이 사람이 짠 테스트가 리팩토링을 막는 방해물이 되는가, 아니면 리팩토링을 가능하게 해주는 안전망이 되는가"다. 테스트를 많이 짜본 사람과 테스트를 잘 짜본 사람의 차이는, 테스트가 구현 세부사항에 결합(coupled)되어 있는지, 아니면 동작(behavior)에 결합되어 있는지에서 갈린다.

실무에서 이 감각이 갈리는 지점은 명확하다. 내부 함수 이름을 바꿨다는 이유만으로 테스트 수십 개가 깨지는 코드베이스는, 테스트가 있어도 리팩토링을 두렵게 만든다. 반대로 외부에서 관찰 가능한 동작만 검증하는 테스트는, 내부 구현을 자유롭게 바꿔도 안심하고 진행할 수 있는 기반이 된다. 이 차이를 설명할 수 있는가가 이 레슨의 핵심이다.

실무 연결
"외부 결제 API를 호출하는 서비스 로직을 테스트하고 싶은데, 실제 API를 호출하면 느리고 비용도 들고 네트워크 상태에 따라 실패할 수도 있습니다. 어떻게 테스트하시겠습니까?" — 이 질문에 mock의 역할과 한계, 그리고 "무엇을 mock하고 무엇을 mock하지 말아야 하는가"의 판단 기준을 설명할 수 있는가가 이 레슨의 핵심이다.

전제 개념 ① — 테스트가 "격리"되어야 한다는 말의 진짜 의미

테스트 두 개가 있는데, 하나를 실행한 순서나 결과가 다른 하나의 통과 여부에 영향을 준다면 그 테스트는 격리(isolation)되지 않은 것이다. 흔한 원인은 전역 상태(모듈 레벨 변수, 싱글턴), 파일 시스템에 남는 부작용, 테스트 간 공유되는 DB 레코드, 혹은 테스트 실행 순서에 의존하는 assumption이다.

격리가 왜 중요한가? 격리되지 않은 테스트는 "가끔 실패하는 테스트(flaky test)"를 만든다. 그리고 flaky한 테스트가 하나라도 있으면, 팀은 결국 "이 테스트는 원래 가끔 실패해요, 재실행하면 통과해요"라고 무시하는 습관이 생기고, 이 습관이 굳어지면 진짜 버그가 난 실패조차 재실행으로 넘겨버리는 문화적 위험이 생긴다. 즉 격리는 단순한 스타일 문제가 아니라 테스트 스위트의 신뢰성 자체를 좌우하는 성질이다. 

# 격리가 깨진 예 — 모듈 레벨 리스트를 여러 테스트가 공유
_cache = []

def add_item(item):
    _cache.append(item)
    return len(_cache)

def test_add_item_returns_one():
    assert add_item("a") == 1  # 단독 실행하면 통과

def test_add_another_item():
    assert add_item("b") == 1  # 위 테스트가 먼저 실행되면 실패 (_cache에 이미 "a"가 있음)
격리를 확인하는 실전 방법
pytest -p no:randomly로 순서를 고정해서 통과하는 테스트가, pytest-randomly 플러그인으로 순서를 무작위화했을 때도 항상 통과하는지 확인하는 것이 격리 여부를 검증하는 가장 확실한 실무 방법이다. "순서를 바꿔도 통과하는가"가 격리의 실질적 정의다.

면접 질문 ① — pytest fixture의 scope는 왜 있고, 잘못 고르면 어떤 문제가 생기나요?

DB 커넥션을 만드는 fixture를 작성하려고 합니다. 매 테스트마다 새 커넥션을 만들면 느린데, 그렇다고 전체 테스트 세션에서 하나의 커넥션만 재사용하면 어떤 위험이 있나요? scope를 어떻게 선택하시겠습니까?

인터뷰어가 평가하는 신호

  • fixture의 생명주기(setup/teardown 시점)와 scope의 관계를 이해하는가
  • "빠른 테스트"와 "격리된 테스트" 사이의 trade-off를 판단할 수 있는가
  • scope를 잘못 고르면 생기는 구체적인 실패 시나리오를 설명할 수 있는가

답변 구조 — 생각의 순서

  1. 문제 정의: 리소스 생성 비용(느림)과 테스트 간 격리(안전함)는 서로 당기는 힘이다
  2. scope의 역할: fixture가 몇 개의 테스트 사이에서 재사용되는지를 결정
  3. 선택 기준: 리소스가 상태를 갖는가(mutable)? 생성 비용이 얼마나 비싼가?
  4. 절충안: 비싼 리소스는 session/module scope로 만들되, 그 안에서 상태를 격리하는 방법을 별도로 마련
import pytest

@pytest.fixture(scope="session")
def db_engine():
    engine = create_engine(TEST_DB_URL)  # 비싼 연결 생성 — 세션 전체에서 1번만
    yield engine
    engine.dispose()

@pytest.fixture(scope="function")  # 기본값. 매 테스트마다 새로
def db_session(db_engine):
    connection = db_engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)
    yield session       # 테스트 함수가 이 시점에 실행됨
    session.close()
    transaction.rollback()   # 매 테스트 후 롤백 — 데이터가 다음 테스트로 새지 않음
    connection.close()
강한 답변

"커넥션 객체 자체(엔진, 커넥션 풀)는 생성 비용이 크니까 scope="session"으로 한 번만 만들고 테스트 전체에서 재사용합니다. 하지만 그 커넥션 위에서 벌어지는 트랜잭션은 scope="function" fixture로 매 테스트마다 새로 시작하고, 테스트가 끝나면 무조건 롤백합니다.

이렇게 나누는 이유는, 커넥션 생성 비용과 데이터 격리 요구사항이 서로 다른 계층에 있기 때문입니다. 커넥션은 '비싸지만 상태가 없는' 자원이라 재사용해도 안전하고, 트랜잭션은 '싸지만 상태를 담는' 자원이라 매번 초기화해야 합니다. 만약 커넥션과 트랜잭션을 구분하지 않고 세션 스코프 하나로 몰아넣으면, 한 테스트에서 INSERT한 데이터가 롤백되지 않고 다음 테스트까지 남아서 테스트 순서에 따라 결과가 달라지는 flaky 테스트가 됩니다."

"반대로 모든 fixture를 scope="function"으로만 쓰면 완벽히 격리되지만, 매 테스트마다 실제 DB 커넥션을 새로 맺어야 해서 테스트 스위트 전체 실행 시간이 크게 늘어나는 trade-off가 있습니다."

약한 답변

"그냥 scope="session"으로 하면 빠릅니다."

왜 약한가: 속도만 고려하고 격리 문제를 전혀 언급하지 않았다. 세션 스코프로만 처리하면 한 테스트의 부작용이 다른 테스트로 새어 나가는 위험이 있는데, 이 trade-off를 인지하지 못하고 있다는 신호로 읽힌다.

전제 개념 ② — mock, stub, fake는 다른 도구다

실무에서 "mock 쓰면 되죠"라는 말을 자주 듣지만, 테스트 대역(test double)에는 목적이 다른 여러 종류가 있다. 이 구분을 못 하면 "어떤 상황에 무엇을 써야 하는가"를 판단하지 못한다.

  • Stub — 정해진 값을 그냥 반환하는 대역. 호출됐는지 여부는 신경 쓰지 않음.
  • Mock — 값을 반환하는 것뿐 아니라, "어떻게 호출되었는가"를 검증하는 데 쓰는 대역 (예: 특정 인자로 정확히 1번 호출됐는가).
  • Fake — 실제와 비슷하게 동작하는 단순화된 구현 (예: 실제 DB 대신 인메모리 dict 기반 저장소).
from unittest.mock import Mock

# Mock으로 "호출 방식"까지 검증
payment_gateway = Mock()
payment_gateway.charge.return_value = {"status": "success"}

process_order(order, payment_gateway)

payment_gateway.charge.assert_called_once_with(amount=100, currency="USD")
# 단순히 결과만 검증하는 게 아니라, "정확히 이 인자로 1번 불렸는가"까지 검증
Mock을 과도하게 쓰면 생기는 문제
내부 구현의 호출 순서·인자까지 전부 assert_called_with로 검증하는 테스트는, 내부 구현을 리팩토링할 때마다 테스트도 함께 고쳐야 하는 "구현에 결합된 테스트"가 된다. Mock은 "외부 경계(외부 API, DB, 파일 시스템)"에서만 쓰고, 내부 협력 객체 사이의 상호작용까지 지나치게 세밀히 검증하는 건 오히려 리팩토링을 방해한다.

면접 질문 ② — 외부 결제 API를 호출하는 함수를 어떻게 테스트하시겠습니까?

process_payment(order) 함수가 내부에서 외부 결제 게이트웨이 API를 HTTP로 호출합니다. 이 함수의 유닛 테스트를 짜야 하는데, 실제 API를 호출하면 느리고, 비용이 들고, 네트워크 장애로 실패할 수도 있습니다. 어떻게 접근하시겠습니까?

인터뷰어가 평가하는 신호

  • 유닛 테스트와 통합 테스트의 책임 경계를 구분하는가
  • mock을 어디에(경계) 적용해야 하는지 판단할 수 있는가
  • "mock이 실제 API 응답 형식과 어긋나면 어떻게 하는가"라는 위험을 인지하는가

답변 구조 — 생각의 순서

  1. 경계 파악: 이 함수가 의존하는 것 중 "내가 통제할 수 없는 외부 시스템"이 무엇인지 식별
  2. 유닛 테스트 설계: 외부 HTTP 호출 지점을 mock으로 대체해 비즈니스 로직만 검증
  3. 공백 인지: mock은 "우리가 예상한 응답 형식"만 흉내 내므로, 실제 API가 이 형식과 다르면 유닛 테스트는 통과하지만 프로덕션은 깨질 수 있음
  4. 보완책: 별도의 통합 테스트(또는 계약 테스트, contract test)로 실제 API와의 스키마 일치를 주기적으로 검증
from unittest.mock import patch

def test_process_payment_success():
    with patch("myapp.payments.gateway_client.charge") as mock_charge:
        mock_charge.return_value = {"status": "success", "id": "tx_123"}

        result = process_payment(order=Order(amount=100))

        assert result.success is True
        mock_charge.assert_called_once_with(amount=100, currency="USD")
강한 답변

"먼저 내가 통제할 수 없는 경계가 어디인지 봅니다. 여기서는 외부 결제 게이트웨이의 HTTP 호출이 그 경계입니다. 유닛 테스트에서는 이 호출 지점을 mock이나 patch로 대체해서, 네트워크 없이도 process_payment의 비즈니스 로직(성공/실패 처리, 금액 검증, 예외 처리)만 빠르고 결정적으로 검증합니다.

다만 이 방식에는 한 가지 위험이 있다는 걸 같이 언급하고 싶습니다. mock은 '우리가 예상한 응답 형식'을 그대로 흉내 낼 뿐이라서, 실제 결제사가 응답 스키마를 바꾸거나 우리가 처음부터 잘못 이해하고 있었다면 유닛 테스트는 계속 통과하는데 프로덕션은 깨지는 상황이 생길 수 있습니다. 이걸 막으려면 실제 API(또는 sandbox 환경)를 호출하는 별도의 통합 테스트나 계약 테스트를 CI에 낮은 빈도로라도 포함시켜서, 두 테스트 계층이 서로 다른 걸 검증하도록 역할을 나눕니다."

"정리하면: 유닛 테스트는 '우리 로직이 옳은가'를, 통합/계약 테스트는 '우리 가정이 실제 외부 시스템과 맞는가'를 검증하는 서로 다른 책임을 갖습니다."

약한 답변

"mock.patch로 API 호출을 막아버리면 됩니다."

왜 약한가: mock을 쓰는 것 자체는 맞지만, "유닛 테스트가 검증하지 못하는 부분"(mock 응답이 실제 API와 어긋날 위험)을 전혀 인지하지 못한 채 mock 하나로 모든 검증이 끝났다고 생각하는 답변이다. 이건 mock의 한계를 모른다는 신호로 읽힌다.

면접 질문 ③ — 일반적인 예시 기반 테스트와 property-based testing은 무엇이 다른가요?

정렬 함수 my_sort(items)를 테스트하려고 합니다. assert my_sort([3,1,2]) == [1,2,3] 같은 테스트를 여러 개 쓰는 방법과, hypothesis 같은 property-based testing 라이브러리를 쓰는 방법은 어떤 차이가 있나요?

인터뷰어가 평가하는 신호

  • "예시 기반 테스트"의 근본적 한계(작성자가 생각한 케이스만 검증)를 이해하는가
  • "불변 성질(invariant)"을 정의해서 테스트할 수 있는 사고방식을 가졌는가
  • property-based testing이 만능이 아니라는 것도 아는가(느림, 디버깅 난이도)

답변 구조 — 생각의 순서

  1. 예시 기반 테스트의 한계: 작성자가 직접 떠올린 입력값만 검증 → 떠올리지 못한 edge case는 그냥 통과되지 않고 검사조차 안 됨
  2. 관점 전환: "이 함수의 결과는 항상 어떤 성질을 만족해야 하는가"를 먼저 정의
  3. 도구 도입: hypothesis가 무작위로 수백 개의 입력을 생성해 그 성질이 항상 성립하는지 검증
  4. 실패 시 동작: 실패하면 라이브러리가 "가장 작은 반례(minimal counterexample)"로 자동 축소(shrinking)해서 보여줌
from hypothesis import given
from hypothesis import strategies as st

@given(st.lists(st.integers()))
def test_sort_is_ordered(items):
    result = my_sort(items)
    # 성질 1: 결과는 항상 오름차순으로 정렬되어 있어야 한다
    assert all(result[i] <= result[i+1] for i in range(len(result)-1))
    # 성질 2: 원소 개수는 정렬 전후 같아야 한다 (원소가 사라지거나 늘어나면 안 됨)
    assert sorted(result) == sorted(items)
# hypothesis가 [], [0], [-5, 3, -5], 매우 긴 리스트 등을 자동으로 생성해 시도함
# 실패하면 예: [2, 1, 1] 같은 가장 작은 반례로 축소해서 보여줌
강한 답변

"예시 기반 테스트는 제가 떠올린 케이스만 검증합니다. 정렬 함수라면 [3,1,2]는 떠올려도, 빈 리스트, 원소가 모두 같은 리스트, 음수가 섞인 리스트, 이미 정렬된 리스트 같은 경우를 빠뜨릴 수 있습니다. 제가 미처 생각하지 못한 입력은 애초에 테스트조차 되지 않습니다.

Property-based testing은 반대로 접근합니다. '구체적인 입력-출력 쌍' 대신 '이 함수의 결과가 항상 만족해야 하는 성질(invariant)'을 정의합니다. 정렬이라면 '결과가 오름차순이다', '원소 집합이 정렬 전후 동일하다' 같은 성질이죠. 그리고 hypothesis 같은 라이브러리가 다양한 입력을 무작위로 대량 생성해서 이 성질이 항상 성립하는지 검증합니다.

가장 실용적인 장점은 실패했을 때입니다. 성질이 깨지는 입력을 찾으면, hypothesis는 그 입력을 계속 더 단순한 형태로 축소(shrinking)해서 '이해하기 쉬운 가장 작은 반례'를 보여줍니다. 다만 만능은 아닙니다 — 실행 시간이 예시 기반 테스트보다 길고, '이 함수가 만족해야 할 성질이 무엇인가'를 정의하는 것 자체가 어려운 로직(복잡한 비즈니스 규칙)도 있어서, 예시 기반 테스트와 상호 보완적으로 씁니다."

약한 답변

"hypothesis를 쓰면 테스트 케이스를 자동으로 만들어줘서 편합니다."

왜 약한가: "자동 생성"이라는 표면적 특징만 말했을 뿐, 왜 그게 예시 기반 테스트보다 근본적으로 다른 검증 방식인지("성질"을 검증한다는 관점 전환)를 설명하지 못했다. property-based testing의 핵심 가치를 이해하지 못한 채 도구 이름만 아는 상태로 읽힌다.

Senior-level Follow-up 질문

Follow-up 1

테스트 커버리지 100%가 버그 없음을 보장하지 않는 이유는 무엇인가요?

커버리지는 "이 코드 줄이 실행되었는가"만 측정하지, "이 코드 줄이 모든 가능한 입력과 상태 조합에서 올바르게 동작하는가"는 측정하지 않는다. 예를 들어 if x > 0: return 1 / x라는 줄이 x=5로 한 번 실행되어 커버리지 100%를 달성해도, x가 매우 작은 부동소수점이거나 예상치 못한 타입일 때의 동작은 전혀 검증되지 않는다. 조건 분기의 모든 경로가 실행되었는가(branch coverage)와, 각 경로가 여러 대표적인 입력으로 검증되었는가는 커버리지 수치와는 별개의 문제다. 커버리지는 "테스트가 부족한 곳을 찾는 도구"로는 유용하지만, "테스트가 충분하다"는 증명으로 오용하면 안 된다.

Follow-up 2

레거시 코드에 테스트가 하나도 없는 상태에서, 리팩토링 전에 어떤 순서로 테스트를 추가하시겠습니까?

가장 먼저 유닛 테스트를 촘촘히 짜려는 유혹을 피해야 한다. 테스트가 없는 레거시 코드는 보통 내부 구조가 테스트하기 어렵게(강하게 결합되어) 설계되어 있어서, 유닛 테스트를 먼저 짜려면 그 자체로 위험한 리팩토링이 선행되어야 하는 모순이 생긴다. 실무적으로는 (1) 먼저 시스템의 외부에서 관찰 가능한 입출력을 감싸는 굵직한 특성화 테스트(characterization test, 현재 동작을 있는 그대로 캡처)를 짠다, (2) 이 안전망 위에서 내부를 조금씩 리팩토링하며 특성화 테스트가 계속 통과하는지 확인한다, (3) 리팩토링으로 구조가 정리되면 그때 세밀한 유닛 테스트를 추가한다. 순서를 거꾸로 하면(먼저 잘게 쪼개려 시도) 안전망 없이 구조를 바꾸다가 회귀 버그를 만들 위험이 크다.

Follow-up 3

테스트가 실패했는데 "테스트 코드가 틀렸다"와 "구현이 틀렸다"를 어떻게 구분하시겠습니까?

먼저 그 테스트가 검증하려는 요구사항(스펙)을 테스트 코드가 아니라 원 요구사항 문서나 도메인 지식에서 다시 확인한다. 테스트가 요구사항을 정확히 반영하고 있다면 구현을 고치는 게 맞고, 요구사항이 변경되었는데 테스트가 옛 요구사항을 그대로 굳혀놓은 상태라면 테스트를 고치는 게 맞다. 실무에서 흔히 저지르는 실수는, 테스트가 실패하면 일단 "구현을 테스트에 맞춰 고쳐서" 통과시키는 것이다 — 이건 테스트가 실제로는 틀렸는데도 구현을 억지로 거기에 맞춰서 진짜 버그를 만들어내는 패턴이다. "이 assertion이 방금 실패했다"는 사실 자체는 무엇이 틀렸는지 알려주지 않으며, 그 판단은 항상 스펙에 대한 이해에서 나와야 한다.

Follow-up 4

시간에 의존하는 코드(datetime.now() 사용)는 어떻게 테스트하기 좋게 설계하시겠습니까?

가장 흔한 실수는 함수 내부에서 직접 datetime.now()를 호출하는 것이다 — 이러면 테스트가 "지금이 언제인가"라는 통제 불가능한 값에 의존하게 되어 결정적이지 않은 테스트가 된다. 실무적 해법은 시간을 의존성으로 주입하는 것이다: 함수 시그니처에 now: datetime 매개변수를 추가하거나(가장 명시적), 시간 제공자를 인터페이스로 감싼 객체를 주입하거나(Clock 프로토콜), freezegun 같은 라이브러리로 테스트 시점에 datetime.now()가 반환하는 값 자체를 고정한다. 핵심은 "현재 시각"을 함수가 스스로 알아내게 하지 않고, 테스트가 원하는 시각을 넣어줄 수 있게 만드는 것 — 이건 시간뿐 아니라 난수, 외부 I/O 등 "통제 불가능한 값에 의존하는 모든 코드"에 적용되는 일반 원칙이다.

Trade-off와 흔한 오해

오해 1 — "테스트가 많을수록 좋다"

같은 동작을 여러 각도에서 중복 검증하는 테스트가 많으면, 리팩토링할 때마다 고쳐야 할 테스트 수만 늘어나고 실질적인 버그 발견율은 늘지 않는다. 중요한 건 테스트 개수가 아니라, 각 테스트가 독립적인 실패 시나리오를 커버하는가다. 동일한 논리 경로를 검증하는 테스트가 5개라면, 그건 5배의 유지보수 비용을 내고도 검증 범위는 1개짜리와 같다.

오해 2 — "내부 함수까지 전부 유닛 테스트로 검증해야 한다"

비공개(private) 헬퍼 함수까지 하나하나 테스트하면, 그 헬퍼 함수의 이름이나 시그니처를 바꾸는 순간(동작은 그대로인데도) 테스트가 깨진다. 원칙적으로는 공개 API, 외부에서 관찰 가능한 동작을 검증 대상으로 삼고, 내부 헬퍼는 그 공개 API를 통해 간접적으로 검증되도록 하는 편이 리팩토링 내성이 훨씬 높다.

오해 3 — "mock으로 대체했으니 그 부분은 안전하게 검증됐다"

mock은 "우리가 그 의존성이 이렇게 동작할 거라고 가정한 것"을 검증할 뿐, 그 가정이 실제 의존성과 맞는지는 검증하지 않는다. 외부 시스템의 실제 응답 형식이 바뀌면 mock 기반 테스트는 계속 통과하는데 프로덕션만 깨지는 "거짓 안전감"이 생길 수 있다. 이걸 보완하려면 통합 테스트나 계약 테스트가 반드시 별도로 필요하다.

오해 4 — "테스트를 작성하는 데 시간을 오래 쓰는 건 비효율이다"

테스트 작성 시간은 "이번 기능을 만드는 비용"에서는 늘어나 보이지만, "이후 몇 달간 회귀 버그를 얼마나 빨리 잡아내는가"라는 관점에서는 오히려 총비용을 줄인다. 테스트가 없는 코드의 진짜 비용은 작성 시점이 아니라, 6개월 뒤 다른 사람이 그 코드를 건드릴 때 드러난다 — "이걸 고쳐도 되는지 확신할 수 없다"는 상태가 만드는 속도 저하가 테스트 비용보다 훨씬 크다.

핵심 요약 — 1분 복습 카드

오늘 반드시 기억할 것
  1. 테스트 격리는 전역 상태·부작용을 제거해 실행 순서와 무관하게 항상 같은 결과를 내는 성질이며, 깨지면 flaky test로 이어짐
  2. fixture scope는 "리소스 생성 비용"과 "테스트 간 격리"라는 두 힘의 절충점 — 비싼 자원은 상위 scope, 상태는 하위 scope로 분리
  3. Stub은 값만 반환, Mock은 호출 방식까지 검증, Fake는 단순화된 실제 구현 — 목적이 다른 도구
  4. mock은 내가 통제할 수 없는 외부 경계에만 적용하고, 내부 협력 객체까지 과도하게 mock하면 리팩토링을 방해하는 테스트가 됨
  5. mock 기반 테스트는 "우리 가정이 맞다면"이라는 전제 위에 서 있으므로, 실제 의존성과의 어긋남은 별도의 통합/계약 테스트로 보완해야 함
  6. property-based testing은 "예시"가 아니라 "항상 성립해야 할 성질"을 검증하는 방식이며, 실패 시 최소 반례로 자동 축소(shrinking)해줌
  7. 테스트 커버리지는 실행 여부만 측정할 뿐 정확성을 보장하지 않으며, 레거시 코드에는 유닛 테스트보다 특성화 테스트를 먼저 두는 것이 안전함
