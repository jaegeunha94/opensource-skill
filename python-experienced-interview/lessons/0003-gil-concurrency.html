GIL(Global Interpreter Lock)
동시성 한계와 우회 전략

파이썬 경력자 Interview · 2026-07-01 · 약 60–80분 소요

왜 면접에서 GIL을 묻는가

GIL은 Python 경력자 면접에서 가장 자주 등장하면서도 가장 자주 피상적으로 답변되는 주제다. "Python은 GIL 때문에 진짜 멀티스레딩이 안 된다"까지만 아는 지원자와, 왜 그런 설계를 했는지, 언제 문제가 되고 언제 문제가 안 되는지, 실제로 어떻게 우회하는지를 아는 지원자를 가르는 질문이다.

면접관이 GIL을 묻는 진짜 목적은 트리비아 확인이 아니다. 동시성 문제를 진단하고 올바른 해결책(threading vs multiprocessing vs asyncio vs C 확장)을 선택할 수 있는가를 보기 위해서다. 잘못된 도구를 선택하면 성능이 오히려 나빠지거나 코드 복잡도만 늘어나는 실무 사고로 이어지기 때문이다.

실무 연결
"API 응답이 느려서 스레드를 늘렸는데 왜 안 빨라지죠?" 라는 질문에 답할 수 있는가, 혹은 "이미지 리사이징 배치 작업을 스레드로 병렬화했는데 CPU 사용률이 1코어만 100%예요"라는 상황을 진단할 수 있는가가 이 레슨의 핵심이다. GIL을 모르면 이런 상황에서 잘못된 방향(스레드 수를 더 늘리기)으로 삽질하게 된다.

전제 개념 ① — GIL이란 정확히 무엇인가

GIL(Global Interpreter Lock)은 CPython 인터프리터 전체에 걸린 단 하나의 뮤텍스(mutex)다. "한 번에 하나의 스레드만 Python 바이트코드를 실행할 수 있다"는 규칙을 강제하는 락이다. 스레드가 몇 개든, CPU 코어가 몇 개든 상관없이 Python 바이트코드를 실행하는 시점에는 반드시 이 락을 쥔 스레드 하나만 동작한다.

먼저 정확히 구분할 것 — "Python"이 아니라 "CPython"의 특징
GIL은 Python 언어 명세가 아니라 CPython이라는 구현체의 설계 선택이다. Jython, IronPython은 GIL이 없다. PyPy는 STM(Software Transactional Memory) 실험을 거쳤고, Python 3.13부터는 CPython 자체도 No-GIL 빌드(PEP 703)를 선택적으로 제공한다. "Python은 원래 GIL이 있다"고 답하면 절반만 맞는 답이다.

왜 GIL이 존재하는가 — 근본 원인은 레퍼런스 카운팅

Day 2에서 다뤘듯 CPython은 모든 객체의 메모리 관리를 레퍼런스 카운팅으로 한다. 객체마다 정수 카운터가 있고, 참조가 생기거나 사라질 때마다 그 값을 증감시킨다.

문제는 카운트 증감이 원자적(atomic)이지 않다는 것이다. refcount += 1은 실제로는 "읽기 → 더하기 → 쓰기" 3단계로 이뤄진다. 멀티스레드 환경에서 두 스레드가 동시에 같은 객체의 카운트를 증감시키면 race condition이 발생해서 카운트가 잘못될 수 있다. 카운트가 실제보다 적게 나오면 아직 쓰이는 객체가 조기에 해제되어 use-after-free로 크래시가 나고, 더 많이 나오면 메모리가 영원히 해제되지 않는 누수가 된다. 

# 개념적으로 이런 상황 (실제 CPython 코드는 아님)
# Thread A와 Thread B가 동시에 같은 객체 obj를 참조 해제한다고 하자

# Thread A                      Thread B
# tmp = obj.refcount   (읽기: 2)
#                                tmp = obj.refcount   (읽기: 2, 아직 A가 안 씀)
# tmp -= 1              (1)
# obj.refcount = tmp    (쓰기: 1)
#                                tmp -= 1              (1, B도 자기 계산대로)
#                                obj.refcount = tmp    (쓰기: 1  ← 원래는 0이어야 함!)
# 결과: 두 스레드가 참조를 하나씩 없앴는데 refcount는 1로 남음 → 메모리 누수
# 반대 타이밍이면 두 번 감소가 겹쳐서 refcount가 실제보다 더 낮아질 수도 있음 → use-after-free

1992년 Python이 멀티코어가 흔하지 않던 시절에 설계될 때, 모든 객체마다 개별 락(fine-grained lock)을 거는 대신 인터프리터 전체에 락 하나를 거는 훨씬 단순한 방법을 선택했다. 이것이 GIL이다. 덕분에 CPython은 단일 스레드 성능이 빠르고, C 확장 모듈을 만들기 쉬웠으며, 메모리 관리 코드가 단순해졌다. 대신 여러 스레드가 CPU 코어를 동시에 쓰지 못한다는 대가를 치른다.

GIL은 "버그"가 아니라 "트레이드오프"다
면접에서 GIL을 "Python의 결함"이라고만 말하면 얕은 이해로 보인다. GIL은 단순성·안전성·단일 스레드 성능과 멀티코어 병렬성 사이의 의도적인 트레이드오프였다. Python이 인터프리터 언어로서 널리 쓰이게 된 이유 중 하나가 바로 이 단순한 메모리 모델 덕분에 C 확장을 쉽게 만들 수 있었기 때문이라는 점도 알아두면 좋다.

면접 질문 ① — GIL이 무엇이고 왜 존재하나요?

Python의 GIL이 무엇인지 설명하고, 왜 이런 설계를 했는지 말해보세요.

인터뷰어가 평가하는 신호

  • GIL을 "Python 언어의 한계"가 아니라 "CPython 구현의 선택"으로 정확히 위치시키는가
  • 레퍼런스 카운팅과의 인과관계(원자성 문제)를 설명할 수 있는가
  • GIL이 준 이득(단순성, C 확장 용이성, 단일 스레드 성능)까지 균형 있게 말하는가

답변 구조 — 생각의 순서

  1. 정의: CPython 인터프리터 전체에 걸린 뮤텍스, 한 번에 하나의 스레드만 바이트코드 실행
  2. 범위: 언어 명세가 아니라 CPython 구현의 특징 (다른 구현체는 다름)
  3. 근본 원인: 레퍼런스 카운팅의 비원자성 → race condition 방지
  4. 대가와 이득: 병렬성을 포기하는 대신 단순성·C 확장 호환성 확보
강한 답변

"GIL은 CPython 인터프리터 전체를 보호하는 하나의 뮤텍스입니다. 스레드가 몇 개든 Python 바이트코드를 실행하는 순간에는 이 락을 쥔 스레드 하나만 동작합니다. 중요한 건 이게 Python 언어의 특성이 아니라 CPython이라는 구현체의 설계라는 점입니다. Jython이나 IronPython에는 GIL이 없고, Python 3.13부터는 CPython도 No-GIL 빌드를 선택할 수 있습니다.

GIL이 필요한 근본 이유는 레퍼런스 카운팅의 원자성 문제입니다. CPython은 모든 객체의 메모리를 레퍼런스 카운트로 관리하는데, 카운트 증감은 읽기-계산-쓰기 세 단계라서 원자적이지 않습니다. 여러 스레드가 동시에 같은 객체의 카운트를 건드리면 레이스 컨디션으로 카운트가 틀어져서 use-after-free나 메모리 누수가 생길 수 있습니다.

1990년대 설계 당시 CPython은 객체마다 개별 락을 거는 fine-grained locking 대신 인터프리터 전체에 락 하나를 거는 방식을 선택했습니다. 그 결과 메모리 관리 로직이 단순해지고, 단일 스레드 성능이 빨라졌고, C 확장 모듈을 스레드 안전성 걱정 없이 만들 수 있게 됐습니다. 대신 여러 스레드가 CPU 코어를 동시에 활용하는 진짜 병렬 실행은 못 하게 된 것이 GIL의 트레이드오프입니다."

약한 답변

"GIL은 파이썬에서 한 번에 하나의 스레드만 실행되게 막는 락이에요. 그래서 파이썬은 멀티스레딩이 안 됩니다."

왜 약한가: "멀티스레딩이 안 된다"는 표현부터 부정확하다(스레드는 생성되고 스위칭되며, I/O 대기 중에는 병렬로 유용하게 동작한다). CPython 구현의 특성이라는 점, 레퍼런스 카운팅과의 연관성, 설계 트레이드오프에 대한 이해가 전혀 드러나지 않는다.

전제 개념 ② — CPU-bound vs I/O-bound, GIL의 실제 영향

GIL의 영향을 정확히 이해하려면 작업을 두 종류로 구분해야 한다. 이 구분이 "어떤 동시성 도구를 쓸 것인가"를 결정하는 가장 중요한 기준이다.

CPU-bound 작업

수치 계산, 이미지 처리, 데이터 파싱, 정렬처럼 CPU가 쉬지 않고 계산만 하는 작업. 이런 작업은 GIL을 계속 쥐고 있어야 진행되므로, 스레드를 여러 개 만들어도 결국 한 번에 하나씩 번갈아 실행될 뿐이다. 오히려 컨텍스트 스위칭 오버헤드 때문에 단일 스레드보다 느려질 수도 있다.

I/O-bound 작업

네트워크 요청, 파일 읽기/쓰기, DB 쿼리, sleep처럼 다른 시스템의 응답을 기다리는 작업. CPython은 I/O 대기에 들어가는 순간(예: socket.recv(), 파일 read 시스템 콜 호출 직전) GIL을 명시적으로 풀어준다. 그래서 한 스레드가 네트워크 응답을 기다리는 동안 다른 스레드가 GIL을 잡고 실행될 수 있다. threading이 I/O-bound 작업에는 실제로 효과가 있는 이유다.

실제로 확인해보기 — CPU-bound에서 스레드는 도움이 안 된다

import time
import threading

def cpu_task(n):
    # 순수 계산 — I/O 없음
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 30_000_000

# 단일 스레드로 2번 실행
start = time.perf_counter()
cpu_task(N); cpu_task(N)
print(f"순차 실행: {time.perf_counter() - start:.2f}s")

# 스레드 2개로 "병렬" 실행 시도
start = time.perf_counter()
t1 = threading.Thread(target=cpu_task, args=(N,))
t2 = threading.Thread(target=cpu_task, args=(N,))
t1.start(); t2.start()
t1.join(); t2.join()
print(f"스레드 2개: {time.perf_counter() - start:.2f}s")
# 결과: 스레드 버전이 순차 버전보다 빠르지 않다 (오히려 스위칭 비용으로 살짝 느릴 수 있음)
# CPU 사용률을 보면 코어 하나만 100%에 가깝게 나옴 (여러 코어가 있어도)

GIL이 스레드 사이를 오가는 방식 — switch interval

GIL을 쥔 스레드는 무한정 실행되지 않는다. CPython은 일정 간격(기본 5ms)마다 다른 스레드에게 GIL을 넘길 기회를 준다. 이를 sys.setswitchinterval()로 조절할 수 있다. 

import sys

print(sys.getswitchinterval())  # 기본값 0.005 (5ms)

# 값을 줄이면 스레드 전환이 잦아져 응답성은 좋아지지만 오버헤드 증가
sys.setswitchinterval(0.001)
CPU-bound 스레드가 여러 개면 오히려 성능이 나빠지는 이유
GIL을 확보하려는 스레드들이 서로 경쟁하면서 convoy effect(스레드들이 GIL을 뺏고 뺏기는 과정에서 캐시 지역성이 깨지고 OS 스케줄러 오버헤드가 커지는 현상)가 발생할 수 있다. David Beazley가 2010년에 발표해 유명해진 문제로, CPU-bound 멀티스레딩이 단일 스레드보다 느려질 수 있음을 실측으로 보여줬다.

면접 질문 ② — API 서버 성능 개선에 스레드를 늘렸는데 느려요

이미지 리사이징을 처리하는 API가 느려서 워커 스레드를 4개로 늘렸는데 처리량이 거의 그대로입니다. 왜 이런 일이 발생했고, 어떻게 해결하시겠습니까?

인터뷰어가 평가하는 신호

  • 이미지 리사이징을 CPU-bound 작업으로 즉시 분류하는가
  • 스레드를 늘려도 GIL 때문에 병렬 실행이 안 된다는 점을 원인으로 짚는가
  • multiprocessing, C 확장(Pillow-SIMD 등), 별도 프로세스 풀 등 구체적 대안을 제시하는가
  • 대안의 비용(프로세스 생성 오버헤드, 데이터 직렬화 비용)까지 함께 언급하는가

답변 구조 — 생각의 순서

  1. 작업 성격 판별: 이미지 리사이징 = CPU-bound (픽셀 연산, I/O 대기 거의 없음)
  2. 원인 진단: GIL 때문에 스레드가 늘어도 코어를 동시에 못 씀 → 처리량 정체
  3. 대안 비교: multiprocessing / C 확장(GIL 해제) / 별도 서비스로 오프로드
  4. 비용 고려: 프로세스 오버헤드, IPC 직렬화 비용, 운영 복잡도
강한 답변

"먼저 이미지 리사이징이 CPU-bound 작업이라는 걸 확인하겠습니다. 네트워크 대기가 아니라 픽셀 연산 자체가 병목이라면, 스레드를 늘려도 GIL 때문에 한 번에 한 스레드만 바이트코드를 실행하므로 코어 여러 개를 동시에 못 씁니다. 스레드 수를 늘려도 처리량이 그대로인 게 정확히 이 증상입니다.

해결책은 세 방향으로 검토합니다. 첫째, multiprocessing으로 워커를 프로세스 단위로 분리합니다. 각 프로세스는 독립적인 GIL과 인터프리터를 가지므로 코어 수만큼 진짜 병렬 처리가 가능합니다. 대신 프로세스 생성 비용과, 이미지 데이터를 프로세스 간에 넘길 때의 직렬화(pickle) 비용을 고려해야 합니다. 큰 이미지라면 multiprocessing.shared_memory로 복사 비용을 줄이는 것도 방법입니다.

둘째, Pillow 같은 이미지 라이브러리는 실제 리사이징 연산을 C로 구현하고 있어서 연산 도중 GIL을 해제합니다. 이미 최적화된 C 확장을 쓰고 있는지 먼저 확인하고, libvips 기반 pyvips처럼 더 빠른 대안도 검토합니다. 셋째, 트래픽이 크면 이미지 처리를 별도의 워커 서비스나 큐(Celery, RQ)로 분리해서 API 서버와 분리된 프로세스 풀에서 처리하는 아키텍처도 고려할 수 있습니다."

약한 답변

"GIL 때문에 스레드로는 성능이 안 나오니까 멀티프로세싱을 쓰면 됩니다."

왜 약한가: 결론은 맞지만 근거(왜 CPU-bound인지, GIL이 정확히 어떻게 막는지)가 없다. multiprocessing의 비용(직렬화, 프로세스 생성)을 언급하지 않아서 실제로 도입했을 때 예상치 못한 문제(느린 IPC, 메모리 사용량 증가)에 부딪힐 수 있다는 인상을 준다.

전제 개념 ③ — GIL을 우회하는 4가지 전략

"GIL이 있으니 Python은 못 쓴다"는 결론은 실무적으로 틀렸다. 실제로는 상황에 맞는 전략이 이미 존재한다.

전략 1 — threading: I/O-bound에 사용

import concurrent.futures
import requests

urls = ["https://api.example.com/a", "https://api.example.com/b", "https://api.example.com/c"]

def fetch(url):
    return requests.get(url, timeout=5).status_code

# I/O 대기 중에는 GIL이 풀리므로 스레드풀이 실제로 유용하다
with concurrent.futures.ThreadPoolExecutor(max_workers=10) as pool:
    results = list(pool.map(fetch, urls))
# 10개 요청을 순차로 하면 10 * 응답시간이지만
# 스레드풀로 하면 가장 느린 요청 시간에 근접하게 끝남

전략 2 — multiprocessing: CPU-bound에 사용

import concurrent.futures

def cpu_heavy(n):
    return sum(i * i for i in range(n))

# 프로세스마다 독립적인 GIL → 진짜 병렬 실행
with concurrent.futures.ProcessPoolExecutor(max_workers=4) as pool:
    results = list(pool.map(cpu_heavy, [10_000_000] * 4))
# CPU 4코어를 실제로 동시에 사용, 코어 수에 비례해 시간 단축
multiprocessing의 숨은 비용
  • 프로세스 생성 비용 — fork는 상대적으로 싸지만 spawn(macOS/Windows 기본)은 인터프리터를 새로 띄우므로 느리다.
  • 데이터 전달 비용 — 인자와 반환값이 pickle로 직렬화되어 프로세스 간에 복사된다. 큰 객체를 주고받으면 이 비용이 계산 이득을 상쇄할 수 있다.
  • 공유 상태 부재 — 프로세스는 메모리를 공유하지 않는다. 공유가 필요하면 multiprocessing.shared_memory, Manager, 또는 외부 저장소(Redis 등)가 필요하다.
  • 디버깅 난이도 — 프로세스마다 별도 스택 트레이스, 별도 로그 컨텍스트가 생겨 스레드보다 디버깅이 어렵다.

전략 3 — C 확장 / 네이티브 라이브러리: GIL을 명시적으로 해제

NumPy, Pandas, Pillow, hashlib 같은 라이브러리의 연산 상당수는 C로 구현되어 있고 연산 도중 GIL을 명시적으로 풀어준다. C 코드 안에서는 Python 객체를 건드리지 않으므로 레퍼런스 카운팅 걱정이 없어서 안전하게 GIL을 놓을 수 있는 것이다. 즉 "NumPy 배열 연산이 빠른 이유"에는 벡터화뿐 아니라 GIL 해제도 포함된다. 

# C 확장 함수 내부의 개념적 구조 (CPython C API)
// Py_BEGIN_ALLOW_THREADS 로 GIL을 풀고
// 순수 C 계산(넘파이 배열 곱셈 등)을 수행한 뒤
// Py_END_ALLOW_THREADS 로 GIL을 다시 획득

전략 4 — asyncio: 단일 스레드 협력형 동시성

GIL 자체를 우회하는 건 아니지만, I/O-bound 동시성을 스레드 없이 처리하는 방법이다. 이벤트 루프 하나가 여러 코루틴을 스위칭하며 실행하므로 GIL 경합 자체가 없다. 다음 Day 5에서 상세히 다룬다. Day 3에서는 "threading과 asyncio 둘 다 I/O-bound 해법이지만 메커니즘이 다르다"는 것만 짚고 넘어간다 — threading은 OS 스레드 + GIL 스위칭, asyncio는 단일 스레드 협력형 스위칭이다.

상황권장 도구이유
네트워크 I/O 다수 (API 호출, DB 쿼리)asyncio 또는 threadingI/O 대기 중 GIL 해제, 동시 처리 가능

면접 질문 ③ — Python 3.13의 No-GIL(PEP 703)에 대해 아는 대로 설명해보세요

Python 3.13부터 GIL을 끌 수 있는 빌드가 등장했다고 들었습니다. 어떻게 가능해졌고, 실무에 어떤 영향을 줄까요?

인터뷰어가 평가하는 신호

  • 최신 Python 발전 방향을 따라가고 있는가 (기술 트렌드에 대한 관심)
  • GIL 제거가 왜 어려운지(레퍼런스 카운팅 원자성 문제)를 다시 연결해서 설명하는가
  • No-GIL이 "즉시 모든 게 빨라진다"는 식의 과도한 낙관을 하지 않는가 (단일 스레드 성능 저하, 생태계 호환성 문제를 아는가)
강한 답변

"PEP 703은 Sam Gross가 제안한 선택적 No-GIL 빌드로, Python 3.13부터 python3.13t(free-threaded 빌드)로 실험적으로 제공되고, 이후 버전에서 점차 안정화되고 있습니다. 핵심은 앞서 말씀드린 레퍼런스 카운팅의 원자성 문제를 다른 방식으로 해결했다는 점입니다.

구체적으로는 biased reference counting(객체를 만든 스레드가 대부분의 증감을 락 없이 처리하고, 다른 스레드의 접근만 별도 카운터로 분리)과, 컨테이너 등 공유 자료구조에는 per-object lock을 세밀하게 적용하는 방식을 씁니다. 즉 인터프리터 전체 락 하나 대신 더 촘촘한 락 전략으로 바꾼 것입니다.

실무 영향은 아직 신중하게 봐야 합니다. No-GIL 빌드는 단일 스레드 성능이 GIL 빌드보다 낮습니다 (락 오버헤드 때문). 그리고 C 확장 생태계(NumPy, Pandas 등) 상당수가 GIL이 있다는 가정으로 작성되어서 No-GIL 대응이 되려면 시간이 걸립니다. 따라서 지금 당장 프로덕션에 적용하기보다는, CPU-bound 멀티스레딩이 필요한 워크로드에서 실험적으로 검토하고 생태계 성숙을 지켜보는 것이 합리적입니다."

약한 답변

"요즘 파이썬은 GIL을 끌 수 있어서 멀티스레딩이 진짜 빨라졌다고 들었어요."

왜 약한가: "들었다"는 표현부터 피상적인 인상을 준다. 어떻게 가능해졌는지(원자성 문제를 어떻게 풀었는지) 설명이 없고, 단일 스레드 성능 저하나 생태계 호환성 같은 현실적 제약을 전혀 언급하지 않아 기술을 비판적으로 평가하는 능력이 안 보인다.

Senior-level Follow-up 질문

Follow-up 1

"threading.Lock은 GIL과 무슨 관계인가요? GIL이 있는데 왜 Lock이 또 필요하죠?"

GIL은 "Python 바이트코드 실행"을 원자적으로 보호하지만, 여러 바이트코드로 이뤄진 연산 시퀀스는 보호하지 않는다. 예를 들어 counter += 1은 바이트코드 수준에서 LOAD → ADD → STORE 여러 단계이고, 이 사이에 GIL이 다른 스레드로 넘어갈 수 있다. 그래서 여러 단계로 이뤄진 "읽고-수정하고-쓰는" 로직에는 여전히 threading.Lock 같은 명시적 동기화가 필요하다. GIL은 인터프리터 내부 상태(메모리 관리)를 보호하는 것이지, 애플리케이션 로직의 논리적 원자성을 보장하지 않는다는 점이 핵심이다.

Follow-up 2

"asyncio와 threading을 동시에 쓰는 경우가 있나요? 왜 그렇게 하나요?"

블로킹 라이브러리(예: 동기식 DB 드라이버, 파일 I/O)를 asyncio 이벤트 루프 안에서 직접 호출하면 이벤트 루프 전체가 멈춘다. 이럴 때 loop.run_in_executor()로 블로킹 호출을 별도 스레드풀에 위임해서 이벤트 루프는 계속 다른 코루틴을 처리하게 한다. "비동기 코드 안에 동기 블로킹 코드가 섞여 있을 때 스레드풀로 격리한다"는 패턴을 아는지 확인하는 질문이다.

Follow-up 3

"multiprocessing.Pool과 concurrent.futures.ProcessPoolExecutor 중 어떤 걸 쓰겠습니까?"

기능적으로는 유사하지만 concurrent.futures는 ThreadPoolExecutor와 동일한 API(submit, map, Future)를 제공해서 스레드 풀과 프로세스 풀 사이를 전환하기 쉽고, 예외 처리와 타임아웃 제어가 더 명시적이다. 실무에서는 특별한 이유가 없다면 concurrent.futures를 기본으로 권장한다는 관점을 보여주면 좋다.

Follow-up 4

"fork와 spawn 방식의 차이가 multiprocessing 코드 작성에 어떤 영향을 주나요?"

fork(Linux 기본)는 부모 프로세스의 메모리를 그대로 복사해서 자식을 만들기 때문에 빠르고 전역 상태가 자동으로 전달되지만, 멀티스레드 상태에서 fork하면 락 상태가 꼬여 데드락 위험이 있다. spawn(Windows/macOS 기본)은 인터프리터를 새로 시작하므로 느리지만 더 안전하며, 전달하는 함수와 인자가 pickle 가능해야 한다는 제약이 생긴다. if __name__ == "__main__": 가드가 spawn 환경에서 왜 필수인지도 함께 설명하면 좋은 신호다.

Trade-off와 흔한 오해

오해 1 — "GIL 때문에 Python은 멀티스레딩을 아예 못 한다"

틀렸다. 스레드는 정상적으로 생성되고 스위칭되며, I/O-bound 작업에서는 실질적인 동시성 이득이 있다. "동시에 여러 스레드가 CPU 코어를 써서 계산하는 진짜 병렬성이 없다"고 정확히 말해야 한다.

오해 2 — "multiprocessing을 쓰면 항상 빨라진다"

프로세스 생성 비용과 IPC 직렬화 비용이 계산 이득보다 클 수 있다. 작은 작업을 여러 개의 프로세스로 쪼개면 오버헤드 때문에 단일 프로세스보다 느려지는 경우도 흔하다. 작업 단위가 충분히 크고(coarse-grained), 데이터 전달량이 적을 때 multiprocessing이 유리하다.

오해 3 — "NumPy를 쓰면 GIL이 자동으로 사라진다"

NumPy의 벡터화된 C 연산 구간에서만 GIL이 풀린다. Python 레벨의 반복문으로 NumPy 배열 원소를 하나씩 접근하는 코드는 여전히 순수 Python 코드라서 GIL의 영향을 그대로 받는다. "NumPy를 쓴다"가 아니라 "벡터화된 연산으로 작성했는가"가 핵심이다.

오해 4 — "No-GIL(PEP 703)이 나오면 GIL 지식은 필요 없어진다"

No-GIL은 선택적 빌드이고, 표준 GIL 빌드가 여전히 기본값이며 오래 공존할 것이다. 또한 No-GIL 환경에서도 race condition은 여전히 존재한다 — 오히려 GIL이 우연히 막아주던 일부 경합 상황이 No-GIL에서는 명시적 락 없이는 드러날 수 있다. GIL에 대한 이해는 당분간 계속 필요하다.
