Day 7

Descriptor 프로토콜
__get__, __set__, __delete__와 속성 접근의 실체

파이썬 경력자 Interview · 2026-07-05 · 약 70–90분 소요

왜 면접에서 Descriptor를 묻는가

@property는 거의 모든 파이썬 개발자가 쓴다. 하지만 "@property가 내부적으로 무엇으로 구현되어 있는지, 왜 인스턴스 __dict__에 같은 이름의 값을 넣어도 property가 덮어써지지 않는지, functools.cached_property는 그것과 뭐가 다른지"를 설명할 수 있는 사람은 훨씬 적다. 면접관은 property 문법을 아는지가 아니라, "obj.attr 하나가 실행될 때 파이썬 인터프리터가 정확히 어떤 순서로 어디를 뒤지는지"를 이해하고 있는지를 본다.

이 감각은 Day 1의 data model(dunder 메서드가 언어 문법을 대신 구현한다는 원리), Day 6의 context manager(프로토콜을 구현하면 문법이 자동으로 호출해준다는 패턴)와 직접 연결된다. ORM의 필드 정의 (models.CharField()), 검증 로직이 들어간 속성, functools.cached_property, @staticmethod/@classmethod까지 — 실무에서 매일 쓰는 도구들이 전부 descriptor 위에서 동작한다. 이걸 직접 설명하고 구현할 수 있는지가 "라이브러리를 쓸 줄 아는 사람"과 "왜 그렇게 동작하는지 아는 사람"을 가른다.

실무 연결
"여러 필드에 공통된 유효성 검증 로직(타입 체크, 범위 체크)이 필요한 클래스가 여러 개 있습니다. 매번 @property로 getter/setter를 반복 작성하지 않고 재사용 가능한 방식으로 만들 수 있을까요?" — 이런 질문에 "재사용 가능한 검증 로직은 descriptor 클래스 하나로 만들어서 여러 속성에 재사용한다"고 바로 연결할 수 있는가가 이 레슨의 핵심이다.

전제 개념 ① — obj.attr는 단순한 "값 읽기"가 아니라 정해진 탐색 순서를 따르는 함수 호출이다

많은 사람이 obj.attr를 "인스턴스가 들고 있는 딕셔너리에서 값을 꺼내는 것"이라고만 생각한다. 절반만 맞다. 실제로는 type(obj).__getattribute__(obj, "attr")라는 함수가 호출되고, 이 함수는 인스턴스의 __dict__와 클래스(및 부모 클래스)의 __dict__를 정해진 우선순위로 뒤지는 알고리즘을 실행한다. 이 우선순위를 정확히 아는 것이 descriptor를 이해하는 전제 조건이다.

obj.attr 조회 시 실제 탐색 순서

type(obj)의 MRO(Method Resolution Order)를 따라 attr이라는 이름이 클래스 쪽에 있는지 먼저 확인한 뒤, 그 결과에 따라 분기한다. 

1. type(obj)의 MRO에서 "attr"을 찾는다 → 클래스 쪽 값(cls_attr)이라고 하자
2. cls_attr가 data descriptor다 (즉 __set__ 또는 __delete__를 정의) →
   → 바로 cls_attr.__get__(obj, type(obj))을 호출해서 반환 (인스턴스 __dict__보다 우선!)
3. cls_attr가 없거나 data descriptor가 아니다 → obj.__dict__에 "attr"이 있는지 확인
   → 있으면 그 값을 그대로 반환
4. obj.__dict__에도 없다 → cls_attr가 non-data descriptor다 (__get__만 정의) →
   → cls_attr.__get__(obj, type(obj))을 호출해서 반환
5. cls_attr가 평범한 클래스 속성(함수, 값 등)이다 → cls_attr을 그대로 반환
6. 아무 데도 없다 → __getattr__이 정의되어 있으면 호출, 없으면 AttributeError

이 순서에서 가장 반직관적인 부분은 2번이다. 상식적으로는 "인스턴스 속성이 클래스 속성보다 우선한다"고 생각하기 쉬운데, data descriptor는 이 상식을 뒤집는다. 바로 이 규칙 때문에 @property로 만든 속성에 obj.attr = 5를 아무리 실행해도(사실은 그 대입 자체가 property.__set__을 호출하는 것이라) 값이 인스턴스 __dict__에 조용히 저장되어 property를 가려버리는 일이 일어나지 않는다.

Data descriptor vs Non-data descriptor — 딱 하나의 기준
기준은 단 하나, __set__ 또는 __delete__를 정의했는가이다.
  • Data descriptor: __get__ + (__set__ 또는 __delete__) 정의 → 인스턴스 __dict__보다 항상 우선
  • Non-data descriptor: __get__만 정의 → 인스턴스 __dict__에 값이 있으면 인스턴스 쪽이 우선
__get__의 유무가 아니라 __set__/__delete__의 유무가 우선순위를 가른다는 걸 놓치는 사람이 많다.

면접 질문 ① — @property는 내부적으로 어떻게 동작하나요?

@property를 쓰면 메서드를 속성처럼 접근할 수 있게 되는데, 이게 파이썬 언어 차원에서 어떤 메커니즘으로 동작하는 건가요? 그리고 obj.__dict__["x"] = 5처럼 인스턴스 딕셔너리에 직접 값을 넣으면 property로 정의된 x가 가려지나요?

인터뷰어가 평가하는 신호

  • property가 __get__/__set__/__delete__를 구현한 descriptor 클래스라는 걸 아는가
  • data descriptor의 우선순위 규칙을 정확히 아는가
  • obj.__dict__를 직접 조작하는 것과 obj.attr = value가 다르다는 걸 구분하는가
  • setter가 없는 read-only property도 여전히 data descriptor라는 미묘한 지점을 아는가

답변 구조 — 생각의 순서

  1. 정의: property는 descriptor 프로토콜을 구현한 내장 클래스
  2. 메커니즘: @property로 감싸면 클래스 속성이 property 인스턴스가 되고, obj.x는 __get__ 호출로 변환됨
  3. 우선순위: data descriptor이므로 인스턴스 __dict__보다 항상 먼저 검사됨
  4. 함정 짚기: obj.__dict__를 직접 만지면 우회 가능하지만, obj.x = v는 __set__을 호출할 뿐 __dict__에 직접 쓰지 않음
class Circle:
    def __init__(self, radius):
        self._radius = radius

    @property
    def radius(self):
        return self._radius

    @radius.setter
    def radius(self, value):
        if value <= 0:
            raise ValueError("radius는 양수여야 합니다")
        self._radius = value

# 위 코드는 대략 아래와 동일하게 동작한다
class _RadiusDescriptor:
    def __get__(self, instance, owner):
        return instance._radius
    def __set__(self, instance, value):
        if value <= 0:
            raise ValueError("radius는 양수여야 합니다")
        instance._radius = value

# Circle.radius는 클래스 속성이자 _RadiusDescriptor() 인스턴스
# c.radius = 10  ->  Circle.radius.__set__(c, 10) 이 호출됨 (c.__dict__["radius"]에 직접 쓰이지 않음)
강한 답변

"property는 __get__, __set__, __delete__를 구현한 내장 descriptor 클래스입니다. @property로 메서드를 감싸면, 클래스 네임스페이스에서 그 이름은 함수가 아니라 property 인스턴스가 됩니다. 그래서 obj.radius를 실행하면 파이썬은 인스턴스 __dict__를 보기 전에 먼저 Circle.radius가 data descriptor인지 확인하고, 맞다면 바로 Circle.radius.__get__(obj, Circle)을 호출합니다.

핵심은 property가 __set__을 정의하고 있기 때문에 data descriptor라는 겁니다. data descriptor는 인스턴스 __dict__보다 항상 우선하기 때문에, 이론적으로 obj.__dict__["radius"] = 999처럼 딕셔너리를 직접 조작해도 obj.radius를 읽으면 여전히 property의 __get__이 먼저 호출됩니다. 반면 obj.radius = 10이라는 코드는 애초에 obj.__dict__에 쓰는 게 아니라 __set__ 메서드를 호출하는 것이므로, '인스턴스 딕셔너리에 값을 직접 넣어서 property를 가린다'는 시나리오 자체가 정상적인 속성 대입 문법으로는 발생하지 않습니다."

"덧붙이면 setter를 정의하지 않은 읽기 전용 property도 여전히 __set__ 메서드 자체는 존재합니다 (호출되면 AttributeError를 던지도록 구현되어 있음). 그래서 setter가 없어도 data descriptor 지위는 유지되고, obj.attr = v를 시도하면 '속성을 설정할 수 없다'는 에러가 나지, 조용히 무시되거나 인스턴스 딕셔너리에 저장되는 일은 없습니다."

약한 답변

"@property는 메서드를 속성처럼 쓸 수 있게 해주는 데코레이터입니다."

왜 약한가: "무엇을 해주는지"는 맞지만 "어떻게" 그게 가능한지 전혀 설명하지 못한다. descriptor라는 단어도, __get__/__set__도, 인스턴스 __dict__보다 우선한다는 규칙도 언급이 없어 "문법을 쓸 줄만 아는" 수준으로 보인다.

전제 개념 ② — 커스텀 descriptor로 검증 로직을 여러 속성에 재사용하기

@property는 속성 하나마다 getter/setter를 새로 작성해야 한다. 같은 검증 로직 (양수 체크, 타입 체크, 범위 체크)이 여러 클래스, 여러 속성에 반복된다면 descriptor 클래스를 직접 만들어 재사용하는 게 훨씬 효율적이다. 이것이 실무에서 ORM(Django, SQLAlchemy)의 필드 정의가 동작하는 방식이기도 하다. 

class PositiveNumber:
    def __set_name__(self, owner, name):
        self.name = f"_" + name    # PEP 487: 클래스 body에서 대입될 때 속성 이름을 전달받음

    def __get__(self, instance, owner):
        if instance is None:
            return self             # Klass.attr로 클래스 자체에서 접근한 경우
        return getattr(instance, self.name)

    def __set__(self, instance, value):
        if value <= 0:
            raise ValueError(f"{self.name}은 양수여야 합니다")
        setattr(instance, self.name, value)

class Circle:
    radius = PositiveNumber()    # __set_name__(Circle, "radius")가 자동 호출됨

class Rectangle:
    width = PositiveNumber()
    height = PositiveNumber()     # 같은 검증 로직을 3개 속성에서 재사용
치명적인 함정 — descriptor 인스턴스에 값을 직접 저장하면 모든 인스턴스가 값을 공유한다

__set__에서 self.value = value처럼 descriptor 자기 자신에 값을 저장하면 큰 버그가 된다. radius = PositiveNumber()는 클래스 속성이라 Circle의 모든 인스턴스가 같은 PositiveNumber 인스턴스 하나를 공유한다. 따라서 circle_a.radius = 5를 하고 나서 circle_b.radius를 읽으면 5가 나온다 — 서로 다른 인스턴스인데 값이 공유되는 것이다.

위 예제가 getattr(instance, self.name)/setattr(instance, self.name, value)로 값을 인스턴스의 __dict__에 저장하는 이유가 바로 이 함정을 피하기 위해서다. instance 정보 없이 값을 저장할 자리가 필요하면 WeakKeyDictionary로 instance를 키로 삼아 관리하는 방법도 있다(순환 참조로 인한 메모리 누수를 피하려면 일반 dict가 아니라 weakref 기반이어야 함 — Day 2에서 다룬 내용과 직결).

면접 질문 ② — 여러 클래스에서 재사용 가능한 타입/범위 검증 속성을 설계해보세요

여러 모델 클래스에 "이 필드는 항상 양의 정수여야 한다", "이 필드는 특정 문자열 집합 중 하나여야 한다" 같은 검증 로직이 반복됩니다. 매번 @property로 getter/setter를 새로 작성하지 않고 재사용 가능하게 설계해보세요.

인터뷰어가 평가하는 신호

  • 재사용 가능한 검증 로직을 descriptor 클래스 하나로 뽑아내는 설계 감각
  • __set_name__으로 속성 이름을 자동으로 알아내는 방법을 아는가
  • "descriptor 인스턴스에 값을 저장하면 인스턴스 간 상태가 공유된다"는 함정을 스스로 피하는가
  • 이 패턴이 ORM의 필드 정의(models.IntegerField() 등)와 동일한 아이디어임을 연결하는가

답변 구조 — 생각의 순서

  1. 문제 재정의: 검증 로직 자체를 재사용 가능한 단위로 뽑아야 함
  2. 설계 선택: descriptor 클래스로 검증 로직을 캡슐화 — 클래스 속성으로 선언만 하면 여러 곳에서 재사용
  3. 구현: __set_name__으로 이름을 받고, __get__/__set__에서 인스턴스 __dict__에 위임
  4. 함정 방지: 값을 descriptor 자신이 아니라 인스턴스에 저장해서 공유 버그를 피함
강한 답변

"이런 반복되는 검증 로직은 descriptor 클래스로 한 번만 구현하고, 여러 클래스의 속성 선언에서 재사용하는 게 정답입니다. @property는 속성 하나마다 함수를 새로 작성해야 하지만, descriptor 클래스는 ClassAttr = Validator(...) 한 줄로 검증 로직을 재사용할 수 있습니다." 

class OneOf:
    def __init__(self, *choices):
        self.choices = set(choices)

    def __set_name__(self, owner, name):
        self.name = f"_" + name

    def __get__(self, instance, owner):
        if instance is None:
            return self
        return instance.__dict__[self.name]

    def __set__(self, instance, value):
        if value not in self.choices:
            raise ValueError(f"{value!r}는 {self.choices} 중 하나여야 합니다")
        instance.__dict__[self.name] = value

class Order:
    status = OneOf("pending", "paid", "shipped", "cancelled")

class Ticket:
    priority = OneOf("low", "medium", "high")  # 같은 클래스를 다른 값 집합으로 재사용

"여기서 중요한 건 값을 self(descriptor 자신)가 아니라 instance.__dict__에 저장한다는 점입니다. status = OneOf(...)는 클래스 속성이라 Order의 모든 인스턴스가 같은 OneOf 객체 하나를 공유하기 때문에, 값을 descriptor 자신에 저장하면 모든 인스턴스가 상태를 공유하는 심각한 버그가 됩니다. __set_name__은 이 descriptor가 클래스 body에서 어떤 이름으로 대입됐는지(PEP 487) 자동으로 알려주기 때문에, 저장 키를 하드코딩하지 않고 재사용 가능하게 만들 수 있습니다. 이 패턴이 Django의 models.CharField(), SQLAlchemy의 Column()이 동작하는 방식과 본질적으로 같습니다."

약한 답변

"각 클래스마다 @property로 getter/setter를 만들고, setter 안에 검증 코드를 복사해서 넣으면 됩니다."

왜 약한가: 요구사항인 "재사용 가능하게"를 정면으로 무시한 답변이다. 검증 로직이 여러 클래스에 복사-붙여넣기 되면, 로직을 수정할 때마다 모든 클래스를 찾아 고쳐야 하고, descriptor라는 재사용 메커니즘 자체를 모른다는 신호를 준다.

전제 개념 ③ — functools.cached_property가 non-data descriptor인 이유

property는 data descriptor다. 반면 functools.cached_property는 의도적으로 non-data descriptor로 설계되어 있다. 이 차이를 설명할 수 있는지가 "descriptor 우선순위 규칙을 실무 설계에 적용할 수 있는가"를 가르는 지점이다. 

import functools

class Report:
    def __init__(self, rows):
        self.rows = rows

    @functools.cached_property
    def summary(self):
        print("무거운 집계 연산 실행 중...")
        return sum(self.rows)  # 비용이 큰 계산이라고 가정

r = Report([1, 2, 3])
r.summary   # "무거운 집계 연산 실행 중..." 출력, 6 반환
            # 이때 __set__이 없는 non-data descriptor이므로,
            # __get__이 계산 결과를 r.__dict__["summary"] = 6 으로 직접 저장한다
r.summary   # 두 번째부터는 출력 없이 6만 반환
            # 왜? r.__dict__에 "summary"가 이미 있고, cached_property는 non-data descriptor라
            # "인스턴스 __dict__ 우선" 규칙에 따라 __get__ 호출 자체가 생략되기 때문
설계 의도 — 왜 __set__을 일부러 정의하지 않았는가

만약 cached_property가 property처럼 __set__을 정의한 data descriptor였다면, 인스턴스 __dict__에 값을 저장해도 매번 __get__이 먼저 호출되어 캐시가 무의미해진다. __set__을 정의하지 않음으로써 "첫 접근 때만 계산하고 그 결과를 인스턴스 __dict__에 심어서, 이후에는 descriptor를 거치지도 않고 바로 반환"하는 캐싱을 속성 우선순위 규칙 자체를 이용해서 구현한 것이다. 이건 "규칙을 아는 사람이 규칙을 역이용해 최적화를 설계한" 대표 사례다.

함정 — cached_property는 인스턴스가 __dict__를 가져야 동작한다
__slots__을 쓰는 클래스(Day 2에서 다룬 메모리 최적화 기법)는 기본적으로 인스턴스 __dict__가 없다. cached_property는 계산 결과를 instance.__dict__에 저장하는 방식으로 캐싱하므로, __slots__을 쓰는 클래스에서는 그대로 쓸 수 없다 (별도로 __dict__를 __slots__에 포함시키거나 캐시 슬롯을 직접 만들어야 함). "메모리 최적화를 위해 __slots__을 도입했는데 캐싱 데코레이터가 깨졌다"는 실무 이슈가 여기서 나온다.

면접 질문 ③ — property와 cached_property의 차이를 descriptor 관점에서 설명해보세요

property와 functools.cached_property는 둘 다 메서드를 속성처럼 접근하게 해주는데, descriptor 프로토콜 관점에서 어떤 차이가 있나요? 어떤 상황에서 각각을 선택해야 하나요?

인터뷰어가 평가하는 신호

  • property = data descriptor, cached_property = non-data descriptor라는 정확한 구분
  • 이 차이가 "매번 재계산 vs 최초 1회만 계산 후 캐싱"이라는 실질적 동작 차이로 이어진다는 걸 아는가
  • 가변 상태(mutable state)가 바뀌면 cached_property의 캐시가 무효화되지 않는다는 리스크를 아는가
  • __slots__과의 상호작용 같은 실무 제약을 아는가
강한 답변

"property는 __get__과 __set__을 둘 다 구현한 data descriptor이고, cached_property는 __get__만 구현한 non-data descriptor입니다. 이 차이가 우선순위 규칙과 맞물려 완전히 다른 동작을 만듭니다.

property는 data descriptor라서 인스턴스 __dict__보다 항상 우선하고, 그래서 접근할 때마다 매번 getter 함수가 실행됩니다 — 값이 바뀔 수 있는 상태를 매번 최신으로 계산해서 보여주기에 적합합니다. 반면 cached_property는 non-data descriptor라서, 최초 접근 시 계산 결과를 인스턴스 __dict__에 저장해두면 그 다음부터는 인스턴스 __dict__가 우선순위 규칙에 따라 먼저 발견되어 descriptor의 __get__ 자체가 호출되지 않습니다. 계산 비용이 크고, 계산에 쓰이는 입력이 인스턴스 생명주기 동안 바뀌지 않는 경우에 적합합니다.

선택 기준은 명확합니다 — 값이 매번 최신이어야 하면 property, 계산이 비싸고 입력이 불변이면 cached_property입니다. 다만 cached_property는 인스턴스 __dict__에 값을 써야 하므로 __slots__을 쓰는 클래스에서는 그대로 동작하지 않는다는 제약도 같이 언급하겠습니다."

약한 답변

"cached_property는 결과를 캐싱해주는 property입니다."

왜 약한가: "무엇을 하는지"만 말하고 "어떻게" 캐싱이 구현되는지, 왜 property와 구조적으로 다른 종류의 descriptor여야만 캐싱이 가능한지를 전혀 설명하지 못한다. 언제 쓰면 안 되는지(가변 상태, __slots__)도 언급이 없어 피상적인 "사용법 암기" 수준으로 보인다.

Senior-level Follow-up 질문

Follow-up 1

__get__의 두 번째 인자로 왜 instance뿐 아니라 owner(클래스)도 전달되나요?

ClassName.attr처럼 인스턴스 없이 클래스 자체에서 속성에 접근하는 경우를 지원하기 위해서다. 이때 instance는 None으로 전달되고, 관례적으로 descriptor는 instance is None이면 self(descriptor 자신)를 반환해서 "이 속성이 어떤 descriptor로 구현되어 있는지"를 introspection할 수 있게 한다. 실제로 Circle.radius(인스턴스 없이 클래스에서 직접 접근)를 호출하면 이 경로를 탄다 — property 객체 자체가 궁금할 때 쓰는 경로이며, 이걸 처리하지 않으면 클래스 레벨 접근에서 AttributeError나 예상치 못한 크래시가 난다.

Follow-up 2

일반 함수(메서드)도 descriptor인가요? self는 어떻게 자동으로 바인딩되나요?

그렇다. 파이썬의 함수 객체는 __get__만 구현한 non-data descriptor다. obj.method에 접근하면 함수의 __get__(obj, type(obj))가 호출되어 obj가 첫 인자로 바인딩된 bound method 객체를 새로 만들어 반환한다. self를 매번 명시적으로 넘기지 않아도 되는 이유가 바로 이 descriptor 메커니즘이다. @staticmethod는 이 자동 바인딩을 의도적으로 무력화하는 descriptor(전달받은 함수를 그대로 반환), @classmethod는 instance 대신 owner(클래스)를 바인딩하는 descriptor로 구현되어 있다 — 셋 다 같은 프로토콜의 변형이라는 걸 알면 "메서드, staticmethod, classmethod의 차이"를 훨씬 체계적으로 설명할 수 있다.

Follow-up 3

Descriptor와 Day 6의 context manager, Day 1의 data model은 어떤 공통점이 있나요?

셋 다 "특정 dunder 메서드를 구현하면, 특정 파이썬 문법이 자동으로 그 메서드를 호출해준다"는 동일한 설계 철학 위에 있다. with는 __enter__/__exit__를, 속성 접근은 __get__/__set__/__delete__를, 연산자는 __add__/__eq__ 등을 호출한다. 이 패턴을 "프로토콜 기반 설계"로 인식하면, 파이썬이 왜 클래스 상속보다 "이 메서드만 구현하면 이 동작을 지원한다"는 duck typing 방식의 확장을 선호하는지에 대한 답변까지 자연스럽게 이어진다.

Follow-up 4

descriptor를 인스턴스에(클래스가 아니라) 할당하면 동작하나요?

동작하지 않는다. descriptor 프로토콜은 type(obj).__getattribute__가 클래스(및 MRO) 쪽 값을 검사할 때만 발동하도록 CPython에 구현되어 있다. obj.__dict__["x"] = SomeDescriptor()처럼 인스턴스 딕셔너리에 넣은 descriptor는 그냥 평범한 객체로 취급되어 __get__이 호출되지 않는다. 이건 "descriptor는 반드시 클래스 속성으로 선언되어야 프로토콜이 발동한다"는 규칙을 정확히 아는지를 확인하는 질문이며, 앞서 다룬 "descriptor 자신에 상태를 저장하면 인스턴스 간 공유 버그가 난다"는 함정과 정확히 대칭되는 반대쪽 함정이다.

Trade-off와 흔한 오해

오해 1 — "descriptor는 @property보다 항상 무겁고 복잡한 대안이다"

속성 하나만 커스텀 getter/setter가 필요하면 @property가 훨씬 간결하고 명확하다. descriptor 클래스를 직접 만드는 건 같은 검증/변환 로직이 여러 속성 또는 여러 클래스에 반복될 때만 가치가 있다. "재사용 필요성"이 없는데도 커스텀 descriptor부터 만드는 건 불필요한 추상화다.

오해 2 — "non-data descriptor는 data descriptor보다 열등한 기능이다"

우열의 문제가 아니라 용도의 차이다. cached_property가 non-data descriptor인 건 의도된 설계이지 제약이 아니다. "매번 최신 값이 필요한가, 최초 1회 계산 후 고정값이면 충분한가"라는 요구사항이 어떤 종류의 descriptor를 써야 하는지를 결정한다.

오해 3 — "인스턴스 __dict__에 같은 이름으로 값을 넣으면 항상 클래스 속성을 가릴 수 있다"

일반 클래스 속성(함수가 아닌 값)이나 non-data descriptor라면 맞다. 하지만 data descriptor(대표적으로 property)는 정반대다 — 인스턴스 __dict__에 뭘 넣어도 우선순위에서 항상 밀린다. "인스턴스 속성이 클래스 속성보다 우선한다"는 명제는 일반적으로만 맞고, data descriptor 앞에서는 뒤집힌다.

오해 4 — "descriptor는 클래스 속성이니 self.value = x로 상태를 저장해도 된다"

가장 흔하고 치명적인 실수다. descriptor 인스턴스는 클래스에 딱 하나만 존재하며 모든 인스턴스가 공유한다. __set__/__get__ 안에서 상태를 저장할 때는 반드시 전달받은 instance 인자 쪽(그 인스턴스의 __dict__, 또는 instance를 키로 하는 WeakKeyDictionary)에 저장해야 한다.

핵심 요약 — 1분 복습 카드

오늘 반드시 기억할 것
  1. obj.attr는 단순 딕셔너리 조회가 아니라 정해진 우선순위로 클래스와 인스턴스를 탐색하는 알고리즘
  2. data descriptor(__set__/__delete__ 정의)는 인스턴스 __dict__보다 항상 우선
  3. non-data descriptor(__get__만 정의)는 인스턴스 __dict__에 값이 있으면 그쪽이 우선
  4. property = data descriptor(매번 재계산), cached_property = non-data descriptor(최초 1회 계산 후 캐싱)
  5. 커스텀 descriptor는 상태를 반드시 instance 쪽에 저장 — 자신에 저장하면 모든 인스턴스가 값을 공유하는 버그
  6. 함수, staticmethod, classmethod도 전부 같은 descriptor 프로토콜의 변형
