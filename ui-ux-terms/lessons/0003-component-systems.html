1. 디자인 시스템(Design System)

디자인 시스템(Design System)은 제품 전반에 걸쳐 일관된 사용자 경험을 제공하기 위해 디자인 원칙, 재사용 가능한 컴포넌트, 가이드라인, 코드를 하나의 공유 언어로 묶어 놓은 시스템이다. 단순한 스타일 가이드나 UI 키트와 달리 살아 있는 문서(living document)로 지속적으로 유지·관리된다.

비유: 디자인 시스템은 레고 블록 세트와 같다. 블록 하나하나가 정해진 규격(토큰)을 가지고, 이 블록들을 조합해 다양한 구조물(화면)을 빠르게 만들 수 있다. 규격이 통일되어 있기 때문에 디자이너와 개발자가 같은 언어로 소통할 수 있다.
Design Principles
디자인 원칙
제품의 모든 설계 결정을 이끄는 방향성. 예: "명확성 우선", "사용자의 실수를 용인하는 설계". 원칙이 명문화되면 새 팀원도 일관된 판단을 내릴 수 있다.
Component Library
컴포넌트 라이브러리
버튼, 입력 필드, 모달 등 재사용 가능한 UI 요소들의 집합. 디자인 도구(Figma)와 코드(React, Vue 등) 양쪽에서 동일한 컴포넌트를 참조하도록 유지한다.
Style Guide
스타일 가이드
색상, 타이포그래피, 여백 등 시각적 규칙을 문서화한 것. 디자인 시스템의 일부이지만, 스타일 가이드 단독으로는 컴포넌트 재사용이나 코드 연동이 보장되지 않는다.
Documentation
문서화
각 컴포넌트의 사용 방법, 금지 사항(Do/Don't), 접근성 고려사항을 기록한 가이드. 좋은 문서가 없으면 컴포넌트 라이브러리도 "어디서 뭘 써야 하는지 모르는" 상태가 된다.
종류 포함 내용 대표 사례
스타일 가이드 색상, 폰트, 여백 규칙 Airbnb 색상 팔레트 문서
UI 키트 Figma/Sketch 컴포넌트 파일 Material Design Figma Kit
컴포넌트 라이브러리 코드로 구현된 UI 컴포넌트 MUI, Ant Design, shadcn/ui
디자인 시스템 원칙 + 토큰 + 컴포넌트 + 문서 전체 Material Design, Apple HIG, Carbon

2. 원자 디자인(Atomic Design)

원자 디자인(Atomic Design)은 Brad Frost가 2013년에 제안한 UI 컴포넌트 설계 방법론이다. 화학의 원자→분자→유기체 개념을 UI에 적용해 5단계 계층으로 컴포넌트를 구성한다. "작은 것에서 큰 것으로" 조립하는 방식으로 일관성과 재사용성을 높인다.

Atoms (원자)
더 이상 나눌 수 없는 가장 작은 UI 요소
버튼, 입력 필드, 레이블, 아이콘, 색상 샘플, 폰트 스타일 등.

예: <Button>, <Input>, <Label>, <Icon>
Molecules (분자)
원자 2개 이상이 결합된 간단한 UI 그룹
레이블 + 입력 필드 + 버튼 = 검색 폼.

예: SearchBar, FormField, NavItem
Organisms (유기체)
여러 분자/원자가 결합된 복잡한 UI 섹션
로고 + 네비게이션 + 검색 폼 = 헤더.

예: Header, ProductCard, Footer
Templates (템플릿)
실제 콘텐츠 없이 레이아웃 구조만 있는 페이지 틀
헤더 + 사이드바 + 콘텐츠 영역 + 푸터가 배치된 그리드.

예: ArticleTemplate, DashboardTemplate
Pages (페이지)
실제 콘텐츠가 채워진 완성된 화면
템플릿에 실제 텍스트·이미지·데이터가 들어간 최종 결과물.

예: 특정 기사 상세 페이지, 사용자 프로필 페이지
주의: 원자 디자인은 엄격한 규칙이 아니라 사고 방식(mental model)이다. 팀마다 "어디까지가 원자이고 어디서부터 분자인가"에 대한 기준이 다를 수 있다. 중요한 것은 일관된 계층 기준을 팀 내에서 합의하는 것이다.

3. 디자인 토큰(Design Token)

디자인 토큰(Design Token)은 색상, 간격, 타이포그래피, 그림자 등 시각적 속성을 이름이 있는 변수로 저장한 값이다. 하드코딩된 값 대신 토큰을 사용하면 디자인과 코드 양쪽에서 하나의 변경만으로 전체 제품의 스타일을 일관되게 바꿀 수 있다.

Global Token
글로벌 토큰
원시 값을 담는 토큰. 색상 팔레트나 폰트 크기 스케일의 원본 값이다.

color-blue-500: #3b82f6
Alias Token
의미론적 토큰 (시멘틱 토큰)
용도(role)를 이름으로 표현하는 토큰. 글로벌 토큰을 참조한다.

color-action-primary: {color-blue-500}
Component Token
컴포넌트 토큰
특정 컴포넌트에만 적용되는 토큰. 의미론적 토큰을 참조한다.

button-primary-bg: {color-action-primary}

토큰을 3단계로 나누는 이유는 테마 전환(다크 모드, 브랜드 변경)을 쉽게 하기 위해서다. 의미론적 토큰의 참조만 바꾸면 컴포넌트 코드를 건드리지 않고도 전체 제품의 색상이 바뀐다.

토큰 레벨 이름 예시 값 예시 역할
글로벌 color-blue-500 #3b82f6 원시 팔레트 값 저장
시멘틱 color-action-primary {color-blue-500} 용도·의미 표현
컴포넌트 button-primary-bg {color-action-primary} 컴포넌트별 세부 바인딩
실무 팁: Figma Variables(2023+)는 디자인 토큰을 Figma 안에서 직접 정의하고 코드와 동기화할 수 있게 해 준다. W3C의 Design Token Community Group은 디자인 토큰의 표준 JSON 포맷(DTCG)을 정의하고 있으며, Style Dictionary·Token Transformer 같은 도구로 플랫폼별 변수 파일로 자동 변환할 수 있다.

4. 컴포넌트 라이브러리 실무 용어

디자인 시스템 논의에서 자주 등장하는 실무 용어들을 정리한다.

Variant
배리언트
하나의 컴포넌트가 가질 수 있는 시각적·기능적 변형. Figma에서는 컴포넌트 속성(Property)으로 관리한다.

예: Button의 variant — primary, secondary, ghost, destructive
State
상태
컴포넌트가 사용자 인터랙션에 따라 취하는 시각적 상태.

예: default, hover, focus, active, disabled, loading, error
Slot / Children
슬롯 / 자식 요소
컴포넌트 내부에 외부에서 콘텐츠를 주입할 수 있는 자리. 컴포넌트의 유연성을 높이면서도 구조 일관성을 유지하게 해 준다.
Prop / Property
프롭 / 속성
컴포넌트의 외부 인터페이스. 코드에서는 React props, Figma에서는 컴포넌트 Properties 패널에서 제어한다.

예: <Button size="sm" variant="primary" disabled />
Storybook
스토리북
UI 컴포넌트를 독립적으로 개발·문서화·테스트하는 오픈소스 도구. 컴포넌트 라이브러리의 "카탈로그" 역할을 하며 배리언트와 상태를 인터랙티브하게 확인할 수 있다.
Theming
테마 적용
디자인 토큰 값을 교체해 동일한 컴포넌트에 다른 시각적 스타일(다크 모드, 화이트라벨 등)을 적용하는 메커니즘. CSS custom properties나 JS theme 객체로 구현한다.
추천 1차 출처
Atomic Web Design — Brad Frost

원자 디자인 방법론의 원저자 블로그 글. 디자인 시스템 전반은 designsystems.com과 NN/g — Design Systems 101을 함께 읽으면 좋다. 디자인 토큰 표준은 W3C DTCG Spec을 참고하자.
