Day 12 · UI/UX 용어

다크 패턴(Dark Pattern)과 윤리적 디자인

Confirmshaming · Roach Motel · Sneak into Basket · Forced Continuity · Nagging

1. 왜 이 용어들이 중요한가

UI/UX의 설득 기법은 사용자를 돕는 방향으로도, 속이는 방향으로도 쓰일 수 있다. 다크 패턴(Dark Pattern)은 사용자의 의도가 아닌 방향으로 행동하도록 의도적으로 설계된 인터페이스를 가리키는 용어로, UX 디자이너 Harry Brignull이 2010년 darkpatterns.org(현 deceptive.design)를 통해 체계화했다.

디자인 리뷰에서 "이 플로우 해지가 너무 어렵다", "이 문구가 죄책감을 유발한다"는 지적이 나올 때, 어떤 다크 패턴 유형에 해당하는지 정확히 이름 붙일 수 있어야 개발자·디자이너·PM·법무가 같은 기준으로 리스크를 판단하고 대안을 논의할 수 있다. 또한 EU DSA(디지털서비스법), 미국 FTC 규제처럼 다크 패턴을 명시적으로 규제하는 법 조항이 늘고 있어, 실무에서 이 용어는 더 이상 이론이 아니라 컴플라이언스 이슈다.

2. 핵심 용어 카드

다크 패턴
Dark Pattern / Deceptive Pattern
사용자의 이익보다 비즈니스의 이익(가입, 결제, 데이터 제공)을 우선하도록 의도적으로 설계된 UI/UX. 단순한 나쁜 디자인(bad UX)과 구분되는 핵심은 "의도성"이다 — 우연히 불편한 것이 아니라, 사용자가 원치 않는 행동을 하도록 계획적으로 설계됐다는 점이 다크 패턴을 정의한다. 최근에는 업계 표준 용어로 Deceptive Pattern(기만적 패턴)이라는 표현도 함께 쓰인다.
컨펌셰이밍
Confirmshaming
거절 버튼에 사용자가 죄책감이나 부끄러움을 느끼게 하는 문구를 넣어 원하는 행동(구독, 동의)을 유도하는 패턴. 예: "구독하기" 버튼 옆에 "아니요, 저는 할인 받고 싶지 않아요" 같은 자기 비하성 거절 문구를 배치한다. 정상적인 거절 옵션은 중립적인 언어("나중에", "닫기")를 써야 한다는 것이 대비되는 원칙이다.
로치 모텔
Roach Motel
"바퀴벌레는 들어가기는 쉬워도 나오기는 어렵다"는 살충제 광고 문구에서 이름을 딴 패턴. 가입·구독은 클릭 한 번으로 쉽게 만들고, 해지는 전화 상담·다단계 확인·숨겨진 메뉴를 거치도록 설계해 이탈을 막는다. "가입은 온라인, 해지는 전화로만" 같은 비대칭 구조가 대표 사례이며, 미국 FTC는 이를 "Click to Cancel" 규칙으로 직접 규제하려 한 바 있다.
스니크 인투 바스켓
Sneak into Basket
사용자가 명시적으로 선택하지 않은 상품이나 옵션(보험, 보증 연장, 기부금 등)을 결제 직전 장바구니에 자동으로 추가하는 패턴. 흔히 체크박스가 기본으로 선택된 상태(pre-checked)로 제공되어 사용자가 직접 해제하지 않으면 그대로 결제가 진행된다. 윤리적 대안은 옵트인(opt-in) 기본값 — 사용자가 직접 체크해야만 추가되는 구조다.
포스드 컨티뉴이티
Forced Continuity
무료 체험(free trial) 신청 시 결제 카드 정보를 미리 받아두고, 체험 종료를 명확히 알리지 않은 채 자동으로 유료 결제로 전환하는 패턴. 해지 안내가 이메일 한 통에만 묻혀 있거나, 체험 종료일이 화면 어디에도 표시되지 않는 경우가 흔하다. 윤리적 설계는 체험 종료 며칠 전 알림과 명확한 해지 경로 제공을 포함한다.
내깅
Nagging
사용자가 이미 거절한 요청(알림 허용, 앱 평가, 뉴스레터 구독)을 세션마다 반복적으로 재요청하는 패턴. 한두 번은 정상적인 리마인더로 볼 수 있지만, 거절해도 계속 뜨는 팝업은 사용자의 명시적 의사를 무시하는 것으로 간주된다. 원칙은 "아니요"를 존중하고 재요청 빈도를 제한하는 것이다.
윤리적 디자인
Ethical Design
다크 패턴의 대응 개념. 사용자의 자율적 의사결정을 존중하고, 정보를 투명하게 제공하며, 취소·거절 경로를 가입·동의 경로와 동등하게 쉽게 만드는 설계 원칙. IDF(Interaction Design Foundation)와 NN/g는 이를 "사용자 신뢰가 장기적으로 비즈니스 지표(재구매율, 이탈률)에 더 유리하다"는 근거로 옹호한다.

3. 나쁜 패턴 vs 윤리적 대안

① 거절 버튼 문구 (Confirmshaming)

Dark Pattern
아니요, 저는 돈 아끼고 싶지 않아요
Ethical
나중에 할게요

거절 옵션은 수락 옵션과 동일하게 중립적이고 명확한 언어를 써야 한다.

② 구독 해지 경로 (Roach Motel)

Dark Pattern
가입: 버튼 1번 클릭 / 해지: 전화 상담 + 재직 확인 질문
Ethical
가입: 버튼 1번 클릭 / 해지: 설정 > 구독 관리 > 버튼 1번 클릭

가입과 해지에 드는 클릭 수·시간이 대칭적이어야 한다는 것이 업계 권고 기준이다.

③ 장바구니 추가 항목 (Sneak into Basket)

Dark Pattern
결제 보험 항목이 기본 체크된 상태로 자동 추가
Ethical
결제 보험 항목은 기본 미체크, 필요 시 사용자가 직접 선택

선택적 옵션은 옵트인이 기본이어야 하며, 옵트아웃(사전 체크 후 해제)을 기본값으로 쓰면 안 된다.

4. 다크 패턴을 판별하는 체크리스트

디자인 리뷰에서 아래 질문으로 다크 패턴 여부를 빠르게 점검할 수 있다.

  1. 의도성 — 사용자가 원치 않는 행동을 하도록 설계자가 의도했는가?
  2. 비대칭성 — 원하는 행동(가입)과 원치 않는 행동(해지)의 난이도 차이가 부자연스럽게 큰가?
  3. 투명성 — 사용자가 지금 무엇에 동의하는지, 무엇이 청구될지 명확히 알 수 있는가?
  4. 기본값 — 사용자에게 불리한 선택지가 기본값(pre-checked, opt-out)으로 설정되어 있는가?
  5. 존중 — 사용자가 이미 거절한 요청을 반복해서 요구하는가?
주의 — "그로스 해킹"과의 경계: 전환율을 높이는 UX 기법(사회적 증거, 긴급성 표시) 자체는 다크 패턴이 아니다. 하지만 "지금 3명이 이 상품을 보고 있습니다" 같은 문구를 사실이 아닌 숫자로 조작하면 다크 패턴(Fake Urgency/Scarcity)이 된다. 기준은 항상 "이 정보가 사실이고 사용자에게 투명한가"이다.
Tip — 법적 리스크로 접근하기: EU의 DSA·GDPR, 미국 FTC의 "Click to Cancel" 규칙처럼 다크 패턴을 직접 규제하는 법이 늘고 있다. 디자인 리뷰에서 다크 패턴 이슈를 제기할 때 "사용자 경험 문제"뿐 아니라 "컴플라이언스 리스크"로도 프레이밍하면 우선순위 조정에 설득력이 커진다.

6. 셀프 체크 퀴즈

Q1

가입은 버튼 1번으로 끝나지만, 해지는 전화 상담과 여러 확인 단계를 거쳐야 하는 패턴의 이름은?

  1. A. Confirmshaming
  2. B. Roach Motel
  3. C. Nagging
  4. D. Forced Continuity
정답 보기

정답: B — Roach Motel
"들어가기는 쉬워도 나오기는 어렵다"는 의미로 가입/해지 난이도의 비대칭 구조를 가리킨다. A(Confirmshaming)는 거절 버튼 문구로 죄책감을 유발하는 패턴이고, C(Nagging)는 거절한 요청을 반복 재요청하는 패턴이며, D(Forced Continuity)는 무료체험을 은근슬쩍 유료로 전환하는 패턴이라 이 사례에 해당하지 않는다.

Q2

결제 화면에서 여행자 보험 옵션이 기본으로 체크된 채 표시되어, 사용자가 직접 해제하지 않으면 함께 결제되는 경우에 해당하는 패턴은?

  1. A. Sneak into Basket
  2. B. Nagging
  3. C. Confirmshaming
  4. D. Roach Motel
정답 보기

정답: A — Sneak into Basket
사용자가 명시적으로 선택하지 않은 항목이 결제에 슬쩍 포함되는 패턴이다. B(Nagging)는 반복 재요청, C(Confirmshaming)는 거절 문구 조작, D(Roach Motel)는 해지 난이도 문제로 이 사례와는 다른 패턴이다.

Q3

다크 패턴과 단순한 "나쁜 UX(bad UX)"를 구분하는 핵심 기준은?

  1. A. 화면이 시각적으로 예쁜지 여부
  2. B. 사용자가 원치 않는 행동을 하도록 의도적으로 설계했는지 여부
  3. C. 로딩 속도가 느린지 여부
  4. D. 모바일에서 반응형으로 동작하는지 여부
정답 보기

정답: B — 의도성
다크 패턴은 우연한 실수가 아니라 사용자의 이익보다 비즈니스 이익을 우선하도록 계획적으로 설계됐다는 "의도성"이 핵심이다. A, C, D는 모두 일반적인 UX 품질 요소이지만, 의도적 기만 여부와는 무관하므로 다크 패턴을 정의하는 기준이 될 수 없다.

7. 다음 학습 예고
