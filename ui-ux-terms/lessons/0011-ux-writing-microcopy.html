Day 11 · UI/UX 용어

UX 글쓰기 — 마이크로카피와 인터페이스 언어

Microcopy · CTA · Error Message · Voice & Tone · Placeholder · Tooltip

1. 왜 이 용어들이 중요한가

버튼 하나에 적힌 단어, 오류가 났을 때 뜨는 한 줄 메시지, 빈 입력창에 흐리게 떠 있는 안내 문구. 이런 짧은 텍스트들을 마이크로카피(Microcopy)라고 부르며, 사용자가 제품을 신뢰하고 다음 행동을 결정하는 데 UI 비주얼만큼이나 큰 영향을 준다.

디자인 리뷰에서 "이 버튼 문구가 애매하다", "에러 메시지가 불친절하다"는 피드백이 나왔을 때, 정확한 용어로 문제를 짚고 개선 원칙을 제안할 수 있어야 개발자·디자이너·PM이 같은 기준으로 카피를 다듬을 수 있다.

2. 핵심 용어 카드

마이크로카피
Microcopy
버튼 레이블, 입력 힌트, 오류 메시지, 빈 상태 안내처럼 인터페이스 곳곳에 흩어져 있는 짧은 텍스트. Joshua Porter가 처음 이 용어를 대중화했으며, "제품의 목소리가 사용자와 직접 대화하는 순간"으로 정의된다. 문장 하나가 전환율(conversion)과 신뢰도에 직접적인 영향을 줄 수 있다.
CTA
Call-to-Action
사용자가 다음 행동을 하도록 유도하는 버튼·링크 문구. "제출하기" 같은 일반 동사보다 구체적인 결과를 담은 동사구("무료로 시작하기", "장바구니에 담기")가 클릭률을 높인다는 것이 실무에서 반복 검증된 원칙이다. CTA는 하나의 화면에 우선순위가 명확한 Primary CTA 하나만 두는 것이 일반적이다.
에러 메시지
Error Message / Humanized Error
시스템 오류나 사용자 입력 실수를 알리는 메시지. 좋은 에러 메시지는 ① 무엇이 잘못됐는지 ② 왜 그런지 ③ 어떻게 고치는지 세 요소를 포함한다. "Error 400" 같은 코드 노출이나 "잘못된 입력입니다" 같은 모호한 문구는 Humanized Error(사람이 이해할 수 있게 번역된 에러) 원칙을 위반한 사례로 자주 언급된다.
보이스 앤 톤
Voice & Tone
Voice(보이스)는 브랜드의 고정된 성격(예: 친근함, 전문성)으로 제품 전체에서 일관되게 유지된다. Tone(톤)은 같은 Voice를 상황에 맞게 조절한 어조로, 성공 메시지는 밝게, 에러나 결제 실패 상황은 차분하고 신중하게 조정한다. Mailchimp의 Content Style Guide가 이 개념을 정립한 대표 사례로 자주 인용된다.
플레이스홀더
Placeholder
입력 필드가 비어 있을 때 흐리게 표시되는 예시 텍스트. 값을 입력하면 사라지므로 라벨(Label) 대용으로 쓰면 안 된다 — 입력 중 사용자가 어떤 필드였는지 잊어버리는 접근성 문제가 발생한다(WCAG에서도 플레이스홀더만으로 라벨을 대체하지 말 것을 권고). 형식 예시("010-1234-5678")를 보여줄 때만 보조적으로 사용한다.
툴팁
Tooltip
아이콘이나 요소에 마우스를 올리거나(hover) 포커스했을 때 나타나는 짧은 보조 설명. 핵심 기능을 설명하는 데 의존하면 안 되며(모바일에는 hover가 없음), 부가 정보나 전문 용어 풀이처럼 없어도 사용에 지장 없는 정보에만 사용하는 것이 원칙이다.
빈 상태 카피
Empty State Copy
데이터가 아직 없는 화면(첫 방문, 검색 결과 없음, 목록 삭제 후)에 표시하는 안내 문구. 단순히 "결과 없음"이라고 알리는 대신, 왜 비어 있는지 설명하고 다음 행동을 제안하는 것이 좋은 Empty State 카피의 기준이다. 온보딩 단계에서는 첫 액션을 유도하는 CTA를 함께 배치한다.
확인 메시지
Confirmation Message
작업이 성공적으로 완료됐음을 알리는 메시지("저장되었습니다", "결제가 완료되었습니다"). 되돌릴 수 없는 작업(삭제, 결제) 전에는 Confirmation Dialog(확인 대화상자)로 사전 확인을 받고, 완료 후에는 명확한 결과 메시지로 마무리하는 것이 표준 흐름이다.

3. 나쁜 카피 vs 좋은 카피

① CTA 문구

Weak
제출
Strong
무료로 시작하기

"제출"은 어떤 결과가 일어나는지 알려주지 않는다. 결과를 명시한 동사구가 사용자의 불확실성을 줄여준다.

② 에러 메시지

Weak
오류가 발생했습니다. (Error 500)
Strong
저장에 실패했어요. 잠시 후 다시 시도하거나, 계속되면 고객센터로 문의해 주세요.

좋은 에러 메시지는 원인 코드를 그대로 노출하지 않고, 사용자가 취할 수 있는 다음 행동을 제시한다.

③ 빈 상태 카피

Weak
데이터가 없습니다.
Strong
아직 작성한 글이 없어요. 첫 글을 써 볼까요? [글쓰기]

비어 있는 이유와 다음 행동을 함께 제시하면 빈 화면이 막다른 길이 아니라 시작 지점으로 느껴진다.

4. 에러 메시지 설계 3요소

  1. 무엇이 잘못됐는가 — 사용자가 이해할 수 있는 말로 상황을 설명한다.
  2. 왜 그런가 — 원인을 간단히 밝힌다(비밀번호 형식 불일치, 네트워크 오류 등).
  3. 어떻게 고치는가 — 사용자가 바로 취할 수 있는 다음 행동을 제안한다.
Tip — 사용자 잘못이 아니라 시스템 언어로: "당신이 잘못 입력했습니다" 대신 "형식이 맞지 않아요"처럼 사용자를 탓하지 않는 어조를 쓴다. 비난조 카피는 신뢰도와 이탈률에 부정적인 영향을 준다.
주의 — 에러 코드 노출: "Error 500", "NullPointerException" 같은 내부 구현 정보를 그대로 노출하면 사용자는 물론 보안 관점에서도 바람직하지 않다. 사용자용 메시지와 개발자용 로그를 분리해서 설계해야 한다.

5. Voice & Tone 적용 예시

같은 Voice(친근하고 담백한 브랜드 성격)를 유지하면서도 상황에 따라 Tone을 다르게 조절하는 예시:

상황Tone예시 카피
회원가입 성공밝고 환영하는 톤가입을 환영해요! 첫 프로젝트를 만들어 볼까요?
결제 실패차분하고 신중한 톤결제가 처리되지 않았어요. 카드 정보를 다시 확인해 주세요.
계정 삭제 확인진지하고 명확한 톤계정을 삭제하면 모든 데이터가 영구적으로 사라져요. 계속할까요?
기능 안내 툴팁가볍고 도움을 주는 톤여기를 누르면 팀원을 초대할 수 있어요.

Voice는 브랜드마다 하나로 고정되지만, Tone은 같은 회사 안에서도 화면 목적에 따라 계속 달라진다는 점이 핵심 차이다.

7. 셀프 체크 퀴즈

Q1

좋은 에러 메시지가 반드시 포함해야 할 세 가지는?

  1. A. 에러 코드, 발생 시간, 서버 IP
  2. B. 무엇이 잘못됐는지, 왜 그런지, 어떻게 고치는지
  3. C. 사과 문구, 이모지, 브랜드 슬로건
  4. D. 재부팅 안내, 고객센터 전화번호, 약관 링크
정답 보기

정답: B — 무엇이 잘못됐는지, 왜 그런지, 어떻게 고치는지
이 세 요소가 빠지면 사용자는 문제를 이해하지도, 스스로 해결하지도 못한다. A는 개발자 디버깅용 정보로 사용자에게는 불필요하거나 보안상 노출하면 안 되는 정보다. C는 톤을 부드럽게 할 수는 있지만 그 자체가 핵심 요소는 아니다. D는 상황에 따라 도움이 될 수 있지만 모든 에러에 필요한 필수 요소는 아니다.

Q2

Voice와 Tone의 차이에 대한 설명으로 맞는 것은?

  1. A. Voice와 Tone은 같은 말을 다르게 부르는 것뿐이다
  2. B. Voice는 상황마다 바뀌고, Tone은 브랜드 전체에서 고정된다
  3. C. Voice는 브랜드의 고정된 성격이고, Tone은 상황에 따라 조절되는 어조다
  4. D. Tone은 오직 에러 메시지에만 적용되는 개념이다
정답 보기

정답: C — Voice는 고정된 성격, Tone은 상황별 어조
A는 오답 — 두 개념은 명확히 구분된다. B는 정의가 반대로 뒤바뀐 설명이다. D는 오답 — Tone은 성공 메시지, 안내 문구 등 모든 카피 상황에 적용된다.

Q3

플레이스홀더(Placeholder) 사용에 대한 설명으로 옳은 것은?

  1. A. 라벨(Label) 없이 플레이스홀더만으로 입력 필드를 설명해도 충분하다
  2. B. 플레이스홀더는 입력값이 채워지면 사라지므로 라벨 대용으로 쓰면 안 된다
  3. C. 플레이스홀더는 항상 굵은 글씨로 강조해야 한다
  4. D. 플레이스홀더는 모바일에서만 사용하는 요소다
정답 보기

정답: B — 라벨 대용으로 쓰면 안 된다
A는 오답 — 사용자가 입력을 시작하면 플레이스홀더가 사라져 어떤 필드였는지 잊어버리는 접근성 문제가 생긴다. WCAG도 이를 라벨 대체 수단으로 권장하지 않는다. C는 근거 없는 오답 — 오히려 플레이스홀더는 옅은 색으로 라벨과 시각적으로 구분하는 것이 일반적이다. D는 오답 — 데스크톱·모바일 모두에서 쓰이는 일반적인 UI 요소다.

8. 다음 학습 예고 — Day 12

다크 패턴(Dark Pattern)과 윤리적 디자인

마이크로카피와 UI 설계가 사용자를 돕는 대신 속이는 방향으로 쓰일 때를 다룬다. Confirmshaming(죄책감 유발 문구), Roach Motel(가입은 쉽고 해지는 어려운 구조), Sneak into Basket(장바구니 몰래 추가), Forced Continuity(무료체험 후 자동결제 은닉), Nagging(반복적인 재요청) 등 대표적인 다크 패턴 유형과, 이를 피하는 윤리적 카피·플로우 설계 원칙을 학습한다.
