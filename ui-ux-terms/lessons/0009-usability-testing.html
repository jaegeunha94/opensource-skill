이 레슨에서 배울 것

  • Usability Testing의 목적과 전형적인 진행 단계를 설명할 수 있다

1. Usability Testing(사용성 테스트)이란

사용성 테스트(Usability Testing)는 실제 사용자가 제품을 직접 사용하는 과정을 관찰해 어디서 막히고, 왜 어려워하는지를 발견하는 평가 방법이다. 디자이너나 개발자의 추측 대신, 실제 행동 데이터를 통해 UX 문제를 찾는다.

핵심 원칙: 사용성 테스트는 사용자를 테스트하는 것이 아니라 제품을 테스트하는 것이다. 진행 중 "잘하셨어요"는 금물 — 사용자 행동이 그대로 피드백이다.

왜 사용성 테스트를 하는가?

문제 발견 방법 신뢰도 비용
전문가 검토 (Heuristic Evaluation) 중간 — 실제 사용자 행동이 아님 낮음
사용성 테스트 높음 — 실제 사용자 행동 관찰 중간
A/B 테스트 높음 — 대규모 정량 데이터 높음 (트래픽 필요)

2. Think-Aloud Protocol (생각 말하기 프로토콜)

Think-Aloud Protocol은 사용자가 제품을 조작하면서 머릿속에 떠오르는 생각을 실시간으로 소리 내어 말하게 하는 기법이다. 1983년 Jakob Nielsen과 동료들이 소프트웨어 평가에 도입하면서 표준 기법이 됐다.

동시적 Think-Aloud (Concurrent)
  • 과업 수행 중 실시간으로 말함
  • 즉각적인 생각·감정을 포착할 수 있다
  • 말하면서 속도가 느려지는 단점
  • 가장 널리 사용되는 방식
회고적 Think-Aloud (Retrospective)
  • 과업 완료 후 세션 영상을 보며 설명
  • 수행 중 방해받지 않아 자연스러운 행동 관찰 가능
  • 기억이 흐려지거나 합리화될 수 있다
  • 복잡한 인터랙션 과업에 적합
진행자 안내 스크립트 예시
"지금부터 제품을 사용하시면서 머릿속에 떠오르는 모든 생각을 소리 내어 말씀해 주세요. '이 버튼이 뭘까?' '여기서 찾을 줄 알았는데 없네' 처럼 무엇이든 좋습니다. 저는 도움을 드리지 않을 거예요 — 어려우신 부분이 바로 저희가 개선할 부분입니다."
참가자에게 정답을 찾는 것이 목표가 아님을 명확히 전달하는 것이 중요하다.

3. Moderated vs Unmoderated Testing

사용성 테스트는 진행자(Moderator)의 존재 여부에 따라 두 가지로 나뉜다.

Moderated Testing (진행자 있음)
  • 연구자가 실시간으로 참여해 질문하고 탐색한다
  • 예상치 못한 행동에 즉시 "왜 그렇게 하셨나요?" 질문 가능
  • 대면(In-person) 또는 화상(Remote Moderated)으로 진행
  • 소수 참가자(5~8명)로 깊은 인사이트 도출에 적합
  • 진행 비용·시간이 많이 든다
Unmoderated Testing (진행자 없음)
  • 참가자가 혼자 과업을 수행하고 녹화·기록됨
  • UserTesting · Maze · Lookback 같은 툴로 자동화
  • 더 많은 참가자 (20~100명)에서 정량 데이터 수집 가능
  • 지역·시간 제약 없이 글로벌 참가자 모집 가능
  • 탐색적 질문(why)이 어렵다
기준 Moderated Unmoderated
참가자 수 5–8명 20–100명+
인사이트 유형 정성적(Why) 정량적(What/How many)
비용 높음 낮음~중간
적합한 단계 초기·탐색 단계 검증·스케일 단계
즉석 질문 가능 불가

4. 테스트 계획 — Test Plan & Task Scenario

사용성 테스트를 시작하기 전에 Test Plan(테스트 계획)을 작성한다. 계획 없이 진행하면 수집된 데이터가 일관성 없고 비교 불가능해진다.

1
연구 목표 정의 (Research Goal)

"사용자가 장바구니에서 주문 완료까지 얼마나 쉽게 이동하는가?"처럼 구체적으로 답할 질문을 먼저 정한다.

2
참가자 모집 (Screener)

Screener(선별 설문)로 목표 사용자 특성에 맞는 참가자를 필터링한다. 내부 직원·디자이너는 제외해야 편향이 줄어든다.

3
과업 시나리오 작성 (Task Scenario)

사용자에게 주어질 현실적인 상황 설명. "버튼을 클릭하세요"처럼 정답을 알려주는 지시가 아닌, 목표를 제시하는 시나리오여야 한다.

4
측정 지표 결정 (Metrics)

Task Success Rate, Time on Task, Error Rate, SUS 등 어떤 지표를 수집할지 미리 정해 둔다.

5
파일럿 테스트 (Pilot Test)

실제 참가자 모집 전 내부 팀원으로 리허설을 진행해 과업 설명의 불명확한 부분을 수정한다.

좋은 Task Scenario vs 나쁜 Task Scenario

❌ 나쁜 예 — 정답을 내포한 지시
"'장바구니' 버튼을 클릭하고 '결제하기' 버튼을 눌러 주문을 완료하세요."
버튼 이름을 그대로 알려 주므로 사용자가 레이블을 인식하는지 전혀 테스트할 수 없다.
✓ 좋은 예 — 목표 상황 제시
"선물로 줄 블루투스 이어폰을 구입하려고 합니다. 마음에 드는 제품을 골라 주문을 완료해 주세요."
사용자가 스스로 탐색하도록 유도하고 네비게이션·레이블·흐름을 자연스럽게 테스트할 수 있다.

5. 주요 측정 지표 (Metrics)

✅
Task Success Rate
과업 성공률
참가자 중 과업을 성공적으로 완료한 비율. 기준선(Benchmark)과 비교해 설계 개선 여부를 판단한다.
⏱️
Time on Task
과업 소요 시간
과업 시작부터 완료(또는 포기)까지 걸린 시간. 효율성(Efficiency)을 나타내는 핵심 지표다.
❌
Error Rate
오류율
과업 수행 중 발생한 실수·잘못된 클릭·되돌아가기 횟수. 오류가 많을수록 인터페이스가 혼란스럽다는 신호다.
😵
Confusion Points
혼란 포인트
사용자가 멈추거나 길을 잃는 특정 화면·요소. Think-Aloud와 함께 관찰해 구체적인 원인을 파악한다.
💬
Satisfaction (SAT)
만족도
리커트 척도(1–7) 또는 SUS로 측정하는 주관적 만족감. 객관적 성과 지표와 함께 해석해야 정확하다.
🔍
Findability
검색 가능성
사용자가 특정 정보나 기능을 찾을 수 있는지 여부. 네비게이션·IA(정보 구조) 평가에 주로 사용된다.

6. SUS — System Usability Scale (시스템 사용성 척도)

SUS(System Usability Scale)는 1986년 John Brooke이 개발한 10문항 설문으로, 빠르고 표준화된 방식으로 사용성을 수치화한다. 각 문항을 1~5점으로 평가하며, 공식 계산법에 따라 0~100점으로 환산한다.

0–50
Unacceptable
사용 불가
51–67
Poor
미흡
68
OK
업계 평균
69–84
Good
양호
85–100
Excellent
탁월
SUS 68점 = 업계 평균. 68점 미만이면 개선이 시급하다는 신호이고, 85점 이상이면 최상위 10% 수준의 사용성을 갖췄다고 볼 수 있다. (Bangor et al., 2008 기준)

SUS 10문항 (요약)

#문항 (요약)방향
1이 시스템을 자주 사용하고 싶다긍정
2이 시스템이 불필요하게 복잡하다부정
3이 시스템이 사용하기 쉽다고 생각한다긍정
4기술 지원이 필요할 것 같다부정
5이 시스템의 다양한 기능이 잘 통합돼 있다긍정
6이 시스템에 일관성이 없다부정
7대부분 사람이 빠르게 배울 수 있을 것 같다긍정
8이 시스템이 매우 불편하다부정
9이 시스템을 사용하면서 자신감을 느꼈다긍정
10사용하기 전에 많은 것을 배워야 했다부정

7. 결과 분석 — 관찰에서 인사이트로

테스트 후 수집된 데이터를 분석해 실행 가능한 인사이트(Actionable Insight)로 전환해야 한다.

1
관찰 노트 정리 (Observation Notes)

테스트 중 메모한 관찰 사항을 수집한다. "참가자 3명이 'My Account' 레이블을 찾지 못함" 처럼 구체적으로 기록한다.

2
어피니티 다이어그램 (Affinity Diagram)

Day 7에서 배운 어피니티 다이어그램을 활용해 관련 관찰 사항을 주제별로 클러스터링한다.

3
심각도 분류 (Severity Rating)

Nielsen의 0–4 심각도 척도로 각 문제를 분류한다. 4(사용 불가) → 3(심각) → 2(개선 필요) → 1(미미) → 0(실제 문제 아님).

4
개선 권고안 작성 (Design Recommendations)

각 문제에 대해 구체적인 디자인 개선안을 제안한다. "레이블을 바꾸세요"보다 "My Account를 '내 정보'로 변경해 직관성을 높이세요"처럼 구체적으로 작성한다.

8. 핵심 용어 정리

Usability Testing
사용성 테스트
실제 사용자가 제품을 사용하는 과정을 관찰해 UX 문제를 발견하는 평가 방법.
Think-Aloud Protocol
생각 말하기 프로토콜
사용자가 과업 수행 중 머릿속 생각을 실시간으로 소리 내어 말하게 하는 기법. 내면 상태를 직접 관찰할 수 있는 가장 효과적인 방법이다.
Moderated Testing
진행자 있는 테스트
연구자가 실시간으로 참여해 탐색 질문을 할 수 있는 사용성 테스트 방식. 정성적 인사이트 도출에 강점이 있다.
Unmoderated Testing
진행자 없는 테스트
참가자가 혼자 과업을 수행하고 자동으로 기록되는 방식. 대규모 정량 데이터 수집에 적합하다.
Task Success Rate
과업 성공률
사용성 테스트 참가자 중 주어진 과업을 성공적으로 완료한 비율. 효과성(Effectiveness)의 핵심 지표.
Time on Task
과업 소요 시간
과업 시작부터 완료까지 걸린 시간. 제품의 효율성(Efficiency)을 측정하는 지표로, 짧을수록 사용하기 쉬운 제품이다.
SUS (System Usability Scale)
시스템 사용성 척도
10개 문항으로 사용성을 0~100점으로 수치화하는 표준 설문. 68점이 업계 평균이며 85점 이상이 탁월 수준이다.
Screener
참가자 선별 설문
테스트 참가자 모집 시 목표 사용자 조건에 맞는지 사전에 필터링하기 위해 사용하는 설문지.
Task Scenario
과업 시나리오
사용성 테스트 참가자에게 주어지는 현실적인 상황 설명. 정답을 내포하지 않고 목표만 제시해야 유효한 테스트가 된다.
Severity Rating
심각도 분류
발견된 사용성 문제를 영향도와 발생 빈도에 따라 0(문제 없음)~4(사용 불가) 척도로 분류하는 방법.
