Day 15 · UI/UX 용어

알림 & 피드백 UI 컴포넌트 용어

Toast · Snackbar · Banner · Modal vs Non-modal Dialog · Alert Dialog · Badge

1. 왜 이 용어들이 중요한가

"저장됐다는 거 잠깐 보여주고 사라지게 해주세요" 같은 요청은 실무에서 정말 흔하지만, 정확히 어떤 컴포넌트를 말하는지는 사람마다 다르게 이해한다. 화면 하단에 잠깐 떴다 사라지는 것인지(Toast), 액션 버튼이 붙어 있는지 (Snackbar), 아예 다른 작업을 못 하게 막고 응답을 기다려야 하는지(Modal Dialog)에 따라 컴포넌트 구현과 접근성 처리 방식이 완전히 달라진다.

알림·피드백 UI 용어를 정확히 쓰면 "이건 Toast로 처리하고, 삭제 확인은 Alert Dialog로 막아야 한다"처럼 디자이너·개발자·PM이 화면 하나 없이도 구현 방식에 합의할 수 있다. 용어가 뒤섞이면 "가벼운 알림"이라고 요청했는데 개발자가 화면을 가로막는 모달을 만들어버리는 식의 재작업이 반복된다.

2. 핵심 용어 카드

토스트
Toast
화면 한쪽에 잠깐 나타났다가 일정 시간이 지나면 자동으로 사라지는 비침해적(non-intrusive) 알림. 사용자의 현재 작업을 막지 않으며, 보통 액션 버튼 없이 상태만 전달한다. 예: "저장되었습니다" 문구가 2~3초간 화면 하단에 떴다 사라지는 것.
스낵바
Snackbar
Toast와 형태는 비슷하지만 "실행 취소(Undo)" 같은 액션 버튼을 포함할 수 있다는 점이 다르다. Material Design에서 정식으로 정의한 용어로, "메일을 삭제했습니다 · 실행 취소"처럼 방금 한 행동을 되돌릴 짧은 시간을 준다. 실무에서는 Toast와 Snackbar를 같은 컴포넌트로 통칭해 쓰는 팀도 많으므로, 팀 컨벤션을 먼저 확인하는 것이 중요하다.
배너
Banner
Toast처럼 자동으로 사라지지 않고, 사용자가 직접 닫거나 조건이 해소될 때까지 화면 상단(또는 특정 영역)에 지속적으로 노출되는 알림. 시스템 점검 공지, 네트워크 끊김 경고, 결제 수단 만료 안내처럼 페이지를 이동해도 계속 인지시켜야 하는 정보에 쓴다.
모달 다이얼로그
Modal Dialog
화면 뒤를 어둡게 딤(dim) 처리하고 다이얼로그를 닫기 전까지 배경의 다른 요소와 상호작용을 못 하게 막는 팝업. 포커스가 다이얼로그 내부에 갇히는 포커스 트랩(Focus Trap) 처리가 필수이며, 사용자의 즉각적인 결정이 꼭 필요한 흐름(결제 확인, 삭제 확인)에만 제한적으로 써야 한다.
논모달 다이얼로그
Non-modal Dialog
다이얼로그가 떠 있어도 배경의 다른 요소와 계속 상호작용할 수 있는 팝업. 예: 채팅 위젯을 열어둔 채로 다른 화면을 스크롤하는 경우. Modal보다 사용자의 흐름을 덜 방해하지만, 다이얼로그의 존재를 놓치기 쉬워 중요한 결정에는 적합하지 않다.
얼럿 다이얼로그
Alert Dialog
Modal Dialog 중에서도 되돌리기 어려운 결정을 확인받기 위한 특수한 형태. "이 항목을 삭제하시겠습니까?"처럼 확인/취소 버튼만 제공하며, 일반 Modal Dialog보다 접근성 규격(role="alertdialog")이 더 엄격하게 요구된다.
배지
Badge
알림 아이콘이나 메뉴 항목 위에 붙어 읽지 않은 개수나 상태를 숫자·점(dot)으로 표시하는 작은 인디케이터. Toast·Banner처럼 메시지를 직접 전달하지 않고, "확인할 것이 있다"는 신호만 준다는 점에서 구분된다.

3. 헷갈리기 쉬운 개념 비교

① Toast vs Snackbar vs Banner

구분지속 시간액션 버튼대표 용도
Toast몇 초 후 자동 소멸보통 없음단순 상태 알림 ("저장됨")
Snackbar몇 초 후 자동 소멸(액션 시 연장 가능)있음 (실행 취소 등)되돌릴 수 있는 행동 직후 알림
Banner사용자가 닫거나 조건 해소까지 지속있을 수 있음지속적 시스템 공지, 경고

실무 팁: Toast/Snackbar는 몇 초 만에 사라지므로 스크린 리더 사용자가 놓치기 쉽다. 중요한 정보는 자동 소멸형 알림 하나에만 의존하지 말고, 필요하다면 별도 상태 페이지나 알림 센터에도 함께 기록해야 한다.

② Modal vs Non-modal vs Alert Dialog

써야 할 때
결제 확인, 삭제 확인처럼 사용자의 즉각적인 결정이 없으면 다음 단계로 진행할 수 없는 경우 → Modal / Alert Dialog
피해야 할 때
단순 안내나 부가 정보를 위해 배경 상호작용을 막는 경우 → Non-modal이나 Toast로 충분

Modal Dialog를 남용하면 사용자가 "닫기"만 반복해서 누르는 습관이 생겨, 정작 중요한 Alert Dialog도 제대로 읽지 않고 닫아버리는 경고 피로 (Alert Fatigue)가 발생한다.

③ Banner vs Badge

Badge
"알림이 3개 있다"는 신호만 주고, 내용은 클릭해야 확인 가능
Banner
클릭하지 않아도 메시지 내용 자체가 바로 화면에 노출됨

4. 알림 컴포넌트 리뷰 실전 체크리스트

디자인 리뷰나 개발 착수 전, 알림·피드백 화면에서 아래 항목을 팀과 함께 확인한다.

  1. 차단 여부 — 사용자의 다른 작업을 막아야 하는가(Modal), 막지 않아도 되는가(Toast/Banner/Non-modal)?
  2. 지속 시간 — 자동으로 사라져야 하는가, 사용자가 직접 닫아야 하는가?
  3. 되돌리기 — 방금 한 행동을 취소할 수단(Snackbar의 Undo)이 필요한가?
  4. 접근성 — 자동 소멸형 알림의 내용을 스크린 리더가 놓치지 않도록 aria-live 영역으로 처리했는가? Modal은 포커스 트랩과 Esc 닫기가 구현되어 있는가?
  5. 중복 노출 — 같은 정보를 Toast와 Banner로 동시에 띄워 사용자를 헷갈리게 하지 않는가?
  6. 경고 피로 — Modal/Alert Dialog를 꼭 필요한 결정에만 쓰고 있는가, 남용하고 있지 않은가?
Tip — 자동 소멸형 알림은 보조 수단일 뿐: Toast/Snackbar는 몇 초 만에 사라지므로, 사용자가 놓쳐도 문제없는 정보에만 쓰고 중요한 정보는 알림 센터나 상태 화면에도 남겨야 한다.
주의 — Modal 남용: 단순 안내를 위해 Modal Dialog를 쓰면 배경 상호작용이 막혀 사용자 흐름이 끊기고, 반복되면 경고 피로로 이어져 정작 중요한 Alert Dialog도 대충 닫아버리게 된다.

5. 용어 요약

용어한국어한 줄 설명
Toast토스트잠깐 떴다 자동으로 사라지는 비침해적 알림
Snackbar스낵바실행 취소 등 액션 버튼을 포함할 수 있는 Toast 유형
Banner배너닫거나 조건이 해소될 때까지 지속되는 알림
Modal Dialog모달 다이얼로그배경 상호작용을 막고 응답을 기다리는 팝업
Non-modal Dialog논모달 다이얼로그배경 상호작용을 막지 않는 팝업
Alert Dialog얼럿 다이얼로그되돌리기 어려운 결정을 확인받는 Modal 특수형
Badge배지읽지 않은 개수나 상태를 표시하는 작은 인디케이터

6. 셀프 체크 퀴즈

Q1

이메일을 삭제한 직후 화면 하단에 "메일을 삭제했습니다 · 실행 취소" 문구가 잠깐 나타나는 컴포넌트는?

  1. A. Banner
  2. B. Snackbar
  3. C. Modal Dialog
  4. D. Badge
정답 보기

정답: B — Snackbar
실행 취소 같은 액션 버튼을 포함하며 잠깐 떴다 사라지는 것이 Snackbar다. A(Banner)는 자동으로 사라지지 않고 지속 노출되는 알림, C(Modal Dialog)는 배경 상호작용을 막는 팝업, D(Badge)는 개수·상태만 표시하는 인디케이터라 이 상황과 맞지 않는다.

Q2

결제 취소처럼 되돌리기 어려운 작업을 실행하기 전, 사용자의 확인 없이는 다음 단계로 진행할 수 없게 만드는 컴포넌트는?

  1. A. Toast
  2. B. Non-modal Dialog
  3. C. Alert Dialog
  4. D. Banner
정답 보기

정답: C — Alert Dialog
되돌리기 어려운 결정을 확인받기 위해 배경 상호작용을 막고 응답을 기다리는 것이 Alert Dialog다. A(Toast)는 자동 소멸형 알림, B(Non-modal Dialog)는 배경 상호작용을 막지 않아 이런 결정에 부적합, D(Banner)는 지속 노출형 안내라 확인 절차를 강제하지 않는다.

Q3

네트워크 연결이 끊긴 상태를 사용자가 직접 닫거나 연결이 복구될 때까지 화면 상단에 계속 보여주는 것이 적절한 컴포넌트는?

  1. A. Toast
  2. B. Badge
  3. C. Banner
  4. D. Alert Dialog
정답 보기

정답: C — Banner
조건이 해소되거나 사용자가 닫을 때까지 지속적으로 노출되어야 하는 정보는 Banner가 적합하다. A(Toast)는 몇 초 만에 사라져 지속 경고에 부적합, B(Badge)는 내용을 직접 보여주지 않고 개수·상태만 표시, D(Alert Dialog)는 배경 작업을 막아버려 네트워크 복구를 기다리는 동안 다른 작업도 못 하게 되므로 과하다.
