Day 14 · UI/UX 용어

폼 & 인풋 UX 용어

Progressive Disclosure · Inline Validation · Error Prevention · Input Mask

1. 왜 이 용어들이 중요한가

회원가입, 결제, 설정 화면 등 실무에서 가장 자주 다시 만드는 화면이 바로 폼(Form)이다. 그런데 "이 필드는 입력하자마자 바로 검증해 주세요" 같은 요청은 디자이너·PM·개발자가 서로 다른 그림을 떠올리게 만든다. 언제 검증할지 (Inline Validation의 타이밍), 처음부터 얼마나 많은 필드를 보여줄지 (Progressive Disclosure), 애초에 실수를 못 하게 막을지 아니면 실수 후 안내할지(Error Prevention vs Error Correction)는 모두 구현 방식이 크게 달라지는 결정이다.

이 용어들을 정확히 쓰면 스프린트 기획 단계에서 "이 필드는 온블러(on blur) 검증인지 실시간 검증인지", "이 옵션 필드는 처음부터 노출할지 아니면 접어 둘지"를 회의 한 번으로 합의할 수 있다. 용어가 없으면 매번 화면을 그려서 보여줘야 하고, 스토리 포인트 산정도 어긋난다.

2. 핵심 용어 카드

점진적 정보 노출
Progressive Disclosure
모든 옵션과 필드를 한 번에 보여주지 않고, 핵심 정보만 먼저 보여준 뒤 필요할 때 추가 옵션을 펼치는 설계 원칙. 예: 회원가입 폼에서 "고급 설정"을 접어 두고, 클릭했을 때만 회사명·직책 같은 선택 필드를 펼쳐 보여준다. 목적은 인지 부하를 줄여 첫 화면의 완료율을 높이는 것이다.
인라인 유효성 검사
Inline Validation
폼 전체를 제출한 뒤가 아니라, 필드 단위로 그 자리에서 입력값의 유효성을 검사하고 결과를 보여주는 패턴. 검사 시점에 따라 실시간(입력 중), 온블러(필드를 벗어날 때), 제출 시로 나뉘며, 어느 시점을 쓰느냐에 따라 사용자 경험이 크게 달라진다. 다음 섹션의 비교표를 참고.
에러 방지
Error Prevention
사용자가 애초에 실수를 저지르지 못하도록 인터페이스로 미리 막는 설계 전략. 예: 시작일이 종료일보다 늦을 수 없도록 날짜 선택기 자체를 제한하거나, 숫자만 입력 가능한 필드에 문자 입력을 아예 막는 것. 실수가 일어난 뒤 에러 메시지로 안내하는 Error Correction(사후 교정)보다 사용자 부담이 적어, 가능하면 우선적으로 고려해야 하는 원칙이다.
필드 단위 피드백
Field-level Feedback
폼 상단에 "입력값을 확인해주세요"처럼 뭉뚱그려 안내하지 않고, 문제가 있는 바로 그 필드 옆에 구체적인 안내를 붙이는 패턴. 사용자가 어떤 필드를 고쳐야 하는지 다시 스캔할 필요가 없어 수정 시간이 단축된다. Inline Validation의 결과가 보통 Field-level Feedback 형태로 표시된다.
필수 입력 표시
Required Field Indicator
필수 필드와 선택 필드를 구분해 보여주는 시각 장치(보통 * 표시). 실무에서는 필수 필드가 더 많은 폼이라면 선택 필드에만 "(선택)"을 표시하는 편이 별표를 반복하는 것보다 스캔하기 쉽다는 점이 자주 논쟁거리가 된다.
인풋 마스크
Input Mask
전화번호, 카드번호, 날짜처럼 정해진 형식이 있는 입력값을 사용자가 타이핑하는 동안 자동으로 구분자(하이픈, 슬래시 등)를 넣어주며 형식을 강제하는 패턴. 형식 오류를 사전에 방지한다는 점에서 Error Prevention의 한 구현 방법이지만, 지나치게 엄격한 마스크는 붙여넣기(paste)를 방해해 오히려 사용자를 불편하게 만들 수 있다.
자동완성
Autofill / Autocomplete
브라우저나 OS에 저장된 정보(주소, 카드 정보, 비밀번호)를 활용해 필드를 자동으로 채워주는 기능. autocomplete HTML 속성과 name 속성을 표준에 맞게 지정하지 않으면 브라우저가 필드를 인식하지 못해 자동완성이 동작하지 않는다 — 디자인 문제가 아니라 마크업 문제로 발생하는 대표적인 사례.

3. 헷갈리기 쉬운 개념 비교

① 유효성 검사 타이밍 — Real-time vs On Blur vs On Submit

타이밍동작 시점장점단점
Real-time키를 입력할 때마다 즉시가장 빠른 피드백입력 중간에 "에러"가 깜빡여 방해될 수 있음(예: 이메일 입력 중 "@" 치기 전 에러 표시)
On Blur필드에서 포커스가 빠져나갈 때입력이 끝난 뒤 검사해 방해가 적음다음 필드로 넘어가기 전까지는 오류를 모름
On Submit폼 전체를 제출할 때구현이 가장 단순함여러 필드 오류를 한 번에 만나 수정 부담이 큼

실무 기본값으로 흔히 쓰이는 조합은 "On Blur로 첫 검사 + 이후 수정 중에는 Real-time으로 에러 해제"다. 즉 에러를 보여줄 때는 성급하지 않게, 에러를 없앨 때는 최대한 빠르게 반응한다.

② Error Prevention vs Error Correction

Error Prevention
날짜 선택기에서 과거 날짜를 아예 선택 불가로 비활성화
Error Correction
과거 날짜를 선택하게 둔 뒤 제출 시 "유효하지 않은 날짜입니다" 에러 표시

Error Correction이 나쁜 것은 아니다. 모든 실수를 UI로 막을 수는 없으므로 Prevention이 불가능하거나 비용이 큰 경우엔 명확한 Correction 메시지가 대안이 된다. 다만 Prevention이 가능한데 Correction으로만 처리하면 불필요하게 사용자 실수를 유발하는 설계로 간주된다.

③ Progressive Disclosure vs 그냥 필드 숨기기

단순 숨기기
사용자가 필요한 옵션이 있는지조차 모르게 완전히 감춤
Progressive Disclosure
"고급 옵션 보기" 같은 명확한 트리거를 노출해, 있다는 것은 알려주되 필요할 때만 펼침

4. 폼 리뷰 실전 체크리스트

디자인 리뷰나 개발 착수 전, 폼 화면에서 아래 항목을 팀과 함께 확인한다.

  1. 검증 타이밍 — 이 필드는 Real-time / On Blur / On Submit 중 무엇인가?
  2. Prevention 우선 — 에러 메시지로 안내하기 전에, UI 자체로 막을 방법은 없는가?
  3. 피드백 위치 — 에러 안내가 폼 상단 뭉뚱그린 문구가 아니라 필드 단위로 붙어 있는가?
  4. 노출 범위 — 처음부터 모든 필드를 보여줘야 하는가, 일부는 Progressive Disclosure로 접을 수 있는가?
  5. 마크업 속성 — autocomplete, name, type 속성이 표준에 맞게 지정되어 브라우저 자동완성이 동작하는가?
  6. 필수/선택 표기 — 필수·선택 표시 방식이 폼 전체에서 일관되는가?
Tip — Error Prevention이 항상 이긴다: "에러 메시지를 더 친절하게 쓰자"는 논의보다, "애초에 그 실수가 나지 않게 막을 수 있는가"를 먼저 검토하는 편이 대개 더 나은 해결책이다.
주의 — Real-time 검증 남용: 모든 필드를 실시간으로 검사하면, 아직 입력을 끝내지 않은 사용자에게 성급하게 에러를 보여줘 오히려 불안감을 준다. 최소 길이·형식이 명확한 필드(비밀번호 강도 등)에만 제한적으로 쓰는 것이 좋다.

5. 용어 요약

용어한국어한 줄 설명
Progressive Disclosure점진적 정보 노출핵심 정보만 먼저 보여주고 필요할 때 추가 옵션을 펼치는 원칙
Inline Validation인라인 유효성 검사필드 단위로 그 자리에서 입력값을 검사하는 패턴
Error Prevention에러 방지실수를 사후 교정이 아니라 UI로 사전에 막는 전략
Field-level Feedback필드 단위 피드백문제가 있는 필드 바로 옆에 구체적으로 안내하는 방식
Required Field Indicator필수 입력 표시필수·선택 필드를 구분해 보여주는 시각 장치
Input Mask인풋 마스크정해진 형식을 입력 중 자동으로 강제하는 패턴
Autofill / Autocomplete자동완성저장된 정보를 활용해 필드를 자동으로 채우는 기능

6. 셀프 체크 퀴즈

Q1

시작일이 종료일보다 늦은 날짜를 애초에 선택하지 못하도록 날짜 선택기 자체를 제한하는 설계는 어떤 원칙에 해당하는가?

  1. A. Error Correction
  2. B. Error Prevention
  3. C. Progressive Disclosure
  4. D. Field-level Feedback
정답 보기

정답: B — Error Prevention
실수가 일어난 뒤 안내하는 것이 아니라 UI 자체로 실수를 아예 못 하게 막는 것이 Error Prevention이다. A(Error Correction)는 실수 후 사후 안내, C(Progressive Disclosure)는 정보 노출 범위 조절, D(Field-level Feedback)는 에러를 보여주는 위치에 관한 개념이라 이 상황과 맞지 않는다.

Q2

사용자가 필드에서 포커스를 빼기 전까지는 검사하지 않고, 다른 필드로 넘어가는 순간 유효성을 검사하는 타이밍은?

  1. A. Real-time
  2. B. On Submit
  3. C. On Blur
  4. D. Progressive Disclosure
정답 보기

정답: C — On Blur
포커스가 필드를 벗어나는 시점에 검사하는 것이 On Blur다. A(Real-time)는 입력 중 즉시 검사, B(On Submit)는 폼 전체 제출 시 검사이며, D(Progressive Disclosure)는 검증과 무관하게 정보 노출 범위를 다루는 용어라 정답이 될 수 없다.

Q3

회원가입 폼에서 회사명·직책 같은 선택 필드를 "고급 설정 보기"를 눌러야만 펼쳐지도록 접어 두는 설계를 가리키는 용어는?

  1. A. Input Mask
  2. B. Progressive Disclosure
  3. C. Autofill
  4. D. Required Field Indicator
정답 보기

정답: B — Progressive Disclosure
핵심 정보만 먼저 보여주고 추가 옵션은 트리거를 눌러야 펼쳐지게 하는 원칙이 Progressive Disclosure다. A(Input Mask)는 입력 형식을 강제하는 패턴, C(Autofill)는 저장된 정보로 필드를 자동 채우는 기능, D(Required Field Indicator)는 필수 여부 표시 방식이라 이 문제와 맞지 않는다.
