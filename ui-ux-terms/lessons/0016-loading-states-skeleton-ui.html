Day 16 · UI/UX 용어

로딩 상태 & 스켈레톤 UI 용어

Skeleton Screen · Spinner · Progress Indicator · Optimistic UI · Perceived Performance

1. 왜 이 용어들이 중요한가

Day 15에서 다룬 Toast·Banner·Modal이 "작업이 끝난 뒤" 상태를 알리는 용어였다면, 이번 Day는 "데이터를 기다리는 동안" 화면이 어떻게 보여야 하는지를 다루는 용어다. "로딩 중인 거 좀 보기 좋게 해주세요" 같은 요청은 실무에서 매우 흔하지만, 회색 상자 모양의 뼈대를 먼저 보여줄지 (Skeleton Screen), 빙글빙글 도는 아이콘만 보여줄지 (Spinner), 진행률 숫자를 보여줄지(Progress Indicator)에 따라 사용자가 느끼는 대기 시간과 구현 난이도가 완전히 달라진다.

로딩 상태 용어를 정확히 쓰면 "이 화면은 실제 데이터 구조를 미리 보여주는 Skeleton으로, 저 버튼은 서버 응답을 기다리지 않고 Optimistic하게 처리한다"처럼 디자이너·개발자·PM이 화면 하나 없이도 체감 속도 (Perceived Performance) 전략에 합의할 수 있다. 용어가 뒤섞이면 실제로는 빠른 API인데도 불필요한 Spinner 때문에 사용자가 느리다고 느끼는 식의 체감 성능 손실이 반복된다.

2. 핵심 용어 카드

스켈레톤 스크린
Skeleton Screen
실제 콘텐츠가 도착하기 전에 텍스트·이미지가 들어갈 자리를 회색 블록(뼈대) 형태로 미리 보여주는 로딩 화면. 최종 레이아웃과 구조가 거의 같기 때문에, 사용자가 "무엇이 어디에 나타날지"를 미리 예측할 수 있어 스피너보다 체감 대기 시간이 짧게 느껴진다. 페이스북 뉴스피드나 유튜브 썸네일 목록 로딩에서 흔히 볼 수 있다.
스피너
Spinner (Loading Spinner)
회전하는 원형 아이콘으로 "처리 중"이라는 사실만 알릴 뿐, 얼마나 남았는지·무엇이 나타날지는 전혀 알려주지 않는 가장 단순한 로딩 인디케이터. 구현이 쉽고 어떤 상황에도 쓸 수 있지만, 3초 이상 지속되면 사용자가 실제보다 훨씬 오래 기다렸다고 느끼는 경향이 있다.
프로그레스 인디케이터
Progress Indicator
진행 정도를 구체적으로 보여주는 로딩 표시. 전체 대비 진행률을 알 수 있는 확정형(Determinate)(예: "63%", 진행 바)과, 진행률은 모르지만 계속 작업 중임을 보여주는 비확정형(Indeterminate)(예: 계속 움직이는 바)으로 나뉜다. 파일 업로드처럼 전체 크기를 아는 작업에는 확정형을, 서버 응답 시간을 예측할 수 없는 작업에는 비확정형을 쓴다.
셔머 이펙트
Shimmer Effect
Skeleton Screen의 회색 블록 위로 빛이 은은하게 훑고 지나가는 듯한 애니메이션 효과. 화면이 멈춰있지 않고 계속 무언가 진행되고 있다는 느낌을 줘서, 정적인 Skeleton보다 "로딩 중"이라는 신호를 더 명확하게 전달한다.
옵티미스틱 UI
Optimistic UI
서버 응답을 기다리지 않고 요청이 성공할 것이라고 가정하여 화면을 먼저 업데이트한 뒤, 실패하면 그때 되돌리고 에러를 보여주는 방식. "좋아요" 버튼을 누르자마자 즉시 하트가 채워지는 것이 대표적인 예다. 실패 확률이 낮고 되돌리기 쉬운 액션(좋아요, 팔로우, 읽음 처리)에 적합하며, 결제처럼 실패 시 되돌리기가 부담스러운 액션에는 신중해야 한다.
지연 로딩
Lazy Loading
화면에 보이는 시점까지 리소스(이미지, 데이터, 컴포넌트) 로딩을 미루는 기법. 스크롤해서 실제로 보이는 순간 이미지를 불러오는 무한 스크롤 목록이 대표적이다. 초기 로딩 속도를 높이지만, 스크롤 시 빈 공간이나 Skeleton이 잠깐 보일 수 있어 함께 설계해야 한다.
체감 성능
Perceived Performance
실제 서버 응답 속도(실측 성능)와 별개로, 사용자가 주관적으로 느끼는 대기 시간. 같은 2초라도 빈 화면에 스피너만 있을 때보다 Skeleton으로 레이아웃을 먼저 보여줄 때 더 빠르게 느껴진다. 로딩 UI 설계의 목표는 실측 성능을 줄이는 것뿐 아니라 체감 성능을 높이는 것도 포함한다.

3. 헷갈리기 쉬운 개념 비교

① Skeleton Screen vs Spinner vs Progress Indicator

구분보여주는 정보적합한 대기 시간대표 용도
Skeleton Screen최종 레이아웃 구조 예측1~3초 내외피드·목록·카드형 콘텐츠 첫 로딩
Spinner"처리 중"이라는 사실만짧고 예측 어려운 대기버튼 클릭 후 짧은 처리, 구조를 알 수 없는 화면
Progress Indicator구체적 진행률 또는 진행 신호길고 예측 가능한 대기파일 업로드/다운로드, 여러 단계 처리

실무 팁: 로딩이 200ms 이내로 끝나는 경우에는 어떤 인디케이터도 보여주지 않는 편이 낫다. 너무 짧게 깜빡이는 스피너나 스켈레톤은 오히려 화면이 깜박인다는 인상을 줘 체감 성능을 해친다.

② Optimistic UI vs Pessimistic UI

Optimistic UI
요청을 보내는 즉시 성공을 가정하고 화면을 먼저 바꾼다. 체감 속도가 매우 빠르지만, 실패 시 되돌리는 로직과 에러 메시지 설계가 반드시 필요하다.
Pessimistic UI
서버 응답을 받은 뒤에만 화면을 바꾼다. 구현은 단순하고 안전하지만, 모든 액션에 스피너나 버튼 비활성화가 들어가 상대적으로 느리게 느껴진다.

결제, 재고 차감처럼 실패했을 때 되돌리기 어렵거나 비용이 큰 액션은 Optimistic UI 대신 Pessimistic 방식을 쓰는 것이 안전하다.

③ Skeleton Screen vs Lazy Loading

Skeleton Screen
"무엇이 로딩되고 있는지"를 미리 보여주는 시각적 표현 방식
Lazy Loading
"언제 리소스를 불러올지" 타이밍을 결정하는 로딩 전략

두 개념은 함께 쓰이는 경우가 많다. Lazy Loading으로 로딩 시점을 미루고, 그 사이 빈 공간을 Skeleton Screen으로 채워 사용자가 빈 화면을 보지 않게 하는 조합이 대표적이다.

4. 로딩 UI 리뷰 실전 체크리스트

디자인 리뷰나 개발 착수 전, 로딩 상태 화면에서 아래 항목을 팀과 함께 확인한다.

  1. 대기 시간 예측 가능 여부 — 전체 크기·단계 수를 알 수 있는 작업인가(Progress Indicator), 알 수 없는 작업인가(Spinner)?
  2. 구조 예측 필요 여부 — 카드·목록처럼 레이아웃이 정해진 콘텐츠인가(Skeleton), 구조가 매번 달라지는 화면인가(Spinner)?
  3. 실패 가능성과 되돌리기 비용 — 이 액션에 Optimistic UI를 써도 되는가, 아니면 응답을 반드시 기다려야(Pessimistic) 하는가?
  4. 초단기 로딩 처리 — 200ms 이내로 끝나는 경우에도 인디케이터를 강제로 보여주고 있지 않은가?
  5. 접근성 — 로딩 상태 변화를 스크린 리더가 인지할 수 있도록 aria-busy, aria-live 처리가 되어 있는가?
  6. Optimistic 실패 처리 — Optimistic UI가 실패했을 때 되돌리는 애니메이션과 에러 메시지(Day 11 마이크로카피 참고)가 함께 설계되어 있는가?
Tip — 체감 성능은 실측 성능만큼 중요하다: 같은 API 응답 속도라도 Skeleton Screen과 Shimmer Effect를 쓰면 사용자가 더 짧게 기다렸다고 느낀다. "왜 느려 보이나요?"라는 피드백을 받으면 서버 속도뿐 아니라 로딩 인디케이터 종류부터 점검한다.
주의 — Optimistic UI 남용: 결제·재고 차감처럼 실패 시 되돌리기 어려운 액션에 Optimistic UI를 적용하면, 실패했을 때 사용자가 이미 성공했다고 믿은 상태를 다시 정정해야 해서 신뢰도가 크게 떨어진다.

6. 셀프 체크 퀴즈

Q1

SNS 피드가 로딩되는 동안, 실제 게시물 카드와 비슷한 모양의 회색 블록들을 먼저 보여주는 방식은?

  1. A. Spinner
  2. B. Skeleton Screen
  3. C. Progress Indicator
  4. D. Optimistic UI
정답 보기

정답: B — Skeleton Screen
최종 콘텐츠와 비슷한 레이아웃 구조를 회색 블록으로 미리 보여주는 것이 Skeleton Screen이다. A(Spinner)는 구조 정보 없이 처리 중임만 알리고, C(Progress Indicator)는 진행률·단계를 보여주는 것이며, D(Optimistic UI)는 로딩 표시가 아니라 응답 전에 화면을 먼저 바꾸는 방식이라 이 상황과 맞지 않는다.

Q2

게시글에 "좋아요"를 누르자마자 서버 응답을 기다리지 않고 즉시 하트가 채워지고, 실패하면 그때 되돌리는 방식은?

  1. A. Lazy Loading
  2. B. Shimmer Effect
  3. C. Optimistic UI
  4. D. Progress Indicator
정답 보기

정답: C — Optimistic UI
요청 성공을 가정하고 화면을 먼저 갱신한 뒤 실패 시 되돌리는 방식이 Optimistic UI다. A(Lazy Loading)는 리소스를 불러오는 시점을 미루는 전략, B(Shimmer Effect)는 스켈레톤 위 애니메이션, D(Progress Indicator)는 진행률 표시라 이 시나리오와 다르다.

Q3

대용량 파일을 업로드할 때, 전체 크기 대비 몇 %가 전송되었는지 숫자와 막대로 보여주는 것이 적절한 인디케이터는?

  1. A. Spinner
  2. B. 확정형(Determinate) Progress Indicator
  3. C. Skeleton Screen
  4. D. Optimistic UI
정답 보기

정답: B — 확정형(Determinate) Progress Indicator
전체 크기를 알 수 있는 작업은 진행률을 구체적으로 보여주는 확정형 Progress Indicator가 적합하다. A(Spinner)는 진행률 정보를 주지 않고, C(Skeleton Screen)는 콘텐츠 레이아웃 예측용이라 파일 전송에 맞지 않으며, D(Optimistic UI)는 애초에 응답을 기다리지 않는 방식이라 진행률 표시와는 무관하다.
