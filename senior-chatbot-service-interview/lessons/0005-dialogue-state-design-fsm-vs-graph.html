Day 5 · 챗봇 서비스 Interview

Dialogue State 설계: FSM vs Graph 기반 상태 모델

멀티턴 대화에서 "지금 대화가 어디에 와 있는가"를 어떤 자료구조로 표현할지 결정하고, 순수 상태 기계(FSM)의 한계를 스택/그래프 기반 모델로 어떻게 극복하는지 설계하는 능력
면접 질문
Q1 — 주력 질문 "멀티턴 챗봇의 dialogue state를 어떻게 설계하시겠습니까? 상태 기계(FSM)와 그래프 기반 상태 모델의 차이를 실제 설계 관점에서 설명해주세요."
Q2 — 변형 "사용자가 항공권 예약 플로우 중간에 갑자기 '아, 근데 취소 정책이 어떻게 되나요?'라고 다른 주제로 새는(digression) 경우, 상태를 어떻게 관리해서 원래 흐름으로 자연스럽게 복귀시키시겠습니까?"
Q3 — 변형 "State machine 기반으로 대화 흐름을 설계했다가 상태(state) 개수가 감당할 수 없이 늘어난 경험이 있다면, 원인이 무엇이었고 어떻게 해결하셨나요?"
전제 개념

Dialogue State란 정확히 무엇인가

"상태(state)"라는 말을 들으면 많은 사람이 현재_상태 = "배송조회_주문번호_대기중" 같은 문자열 하나를 떠올린다. 이 직관이 바로 dialogue state 설계에서 가장 먼저 깨져야 하는 오해다. 실제로 프로덕션 대화 시스템에서 dialogue state는 최소한 다음 네 가지를 포함하는 구조화된 데이터다.

  • 지금 진행 중인 목표(goal/task) — 예: "환불 요청 처리 중"
  • 그 목표를 위해 채워야 할 slot 값들 — Day 3에서 다룬 entity/slot과 연결
  • 목표가 중첩됐을 때의 순서 정보 — 사용자가 다른 주제로 샜다가 돌아올 원래 목표가 무엇인지
  • 대화 이력(history) — 이전 턴에서 무엇을 말했고 무엇을 확정했는지
핵심 정의: Dialogue state는 "지금 어떤 화면(스크린)에 있는가"가 아니라 "지금까지의 대화로부터 시스템이 다음 행동을 결정하기 위해 반드시 기억해야 하는 최소한의 정보 집합"이다. 이 정의를 먼저 세워야 FSM이냐 그래프냐 하는 구현 선택이 "무엇을 상태로 삼을 것인가"라는 더 근본적인 질문 아래의 하위 결정이라는 것이 명확해진다.

① 왜 순수 FSM은 실무에서 한계에 부딪히는가 — 상태 폭발(state explosion)

고전적인 유한 상태 기계(FSM)는 상태(state)와 전이(transition)를 명시적으로 나열하는 방식이다. "주문번호 입력 대기" → "배송지 확인 대기" → "완료" 같은 선형 흐름에는 직관적이고 예측 가능하다. 문제는 현실의 대화가 선형이 아니라는 점이다. 사용자는 중간에 질문을 바꾸고(digression), 이전 답을 정정하고(correction), 다른 작업을 요청했다가 원래 작업으로 돌아온다. 이런 각각의 경우의 수를 FSM의 "상태"로 명시적으로 표현하려고 하면, 상태 개수가 대화의 가능한 경로 수에 비례해 조합적으로 폭발한다.

실무에서 관찰되는 패턴: 규칙 기반 순수 상태 기계 접근은 대화 경험의 통제력은 크지만, 개발자가 가능한 모든 상태마다 규칙을 수작업으로 만들어야 하므로 결과적으로 지나치게 경직된(rigid) 대화 관리자가 된다는 것이 반복적으로 보고되는 실패 패턴이다. "환불 요청 중 배송조회로 새는 경우", "배송조회 중 환불로 새는 경우", "그 상태에서 다시 돌아오는 경우"를 각각 별도 상태로 정의하다 보면 조합이 기하급수적으로 늘어난다.
근거: 대화 관리 문헌에서 규칙 기반 상태 기계의 controllability-flexibility trade-off에 대한 반복적 관찰(상세 URL은 RESOURCES.md 참고)

② 스택 기반 상태 — FSM의 안정성과 인터럽션 지원을 동시에

2026년 현재 프로덕션 대화형 AI 프레임워크들이 수렴한 해법은 "상태를 하나의 값이 아니라 스택(LIFO)으로 관리"하는 것이다. 이 아이디어는 서로 다른 두 주요 프레임워크에서 독립적으로 관찰된다.

Rasa Pro CALM — Dialogue Frame Stack

  • Flow Policy는 dialogue frame의 LIFO 스택과 내부 slot으로 대화 상태를 관리한다
  • 사용자가 다른 flow로 새면(digression) 새 frame이 스택에 push되고, 원래 flow는 INTERRUPTED 상태로 유지된다
  • 새 flow가 끝나면 pattern_continue_interrupted로 원래 flow가 자동으로 스택 최상단으로 복귀한다

Dialogflow CX — Flow Stack

  • Flow에서 다른 Flow로 전이하면 각 flow가 flow 스택에 push된다
  • 하위 flow가 END_FLOW로 끝나면 스택에서 pop되어 호출한 상위 page로 자동 복귀한다
  • Page(하나의 flow 내부 상태)마다 정의된 state handler가 해당 턴의 이벤트를 평가해 다음 page로의 전이를 결정한다
왜 스택인가: 순수 FSM은 "현재 상태 하나"만 기억하므로 원래 하려던 일로 돌아갈 방법이 없다. 스택은 "지금 하는 일"과 "잠시 미뤄둔 일들의 순서"를 동시에 표현할 수 있어서, 인터럽션·디그레션·정정을 "예외 상황"이 아니라 스택 push/pop이라는 정규 연산으로 다룰 수 있게 한다. 이것이 순수 FSM 대비 상태 폭발 없이 인터럽션을 지원하는 핵심 아이디어다.
근거: Rasa Flow Policy 공식 문서; Rasa Pro Changelog(dialogue frame stack, LIFO, pattern_continue_interrupted); Dialogflow CX State handlers 공식 문서; Dialogflow CX Pages 공식 문서 (2026-06 갱신 확인)

③ 그래프 기반 상태 — LangGraph StateGraph로 더 일반화하기

스택 모델이 "진행 중인 작업들의 순서"를 다뤘다면, 그래프 기반 상태 모델은 한 걸음 더 나아가 상태 자체를 타입이 정의된 데이터 스키마로 다루고, 그 상태를 읽고 쓰는 노드(node)들을 조건부 엣지(edge)로 연결한다.

  • 타입 상태 스키마: StateGraph는 상태의 필드 타입을 명시적으로 정의한다. 이 스키마를 올바르게 설계하는 것이 그래프 설계에서 가장 중요한 결정으로 꼽히는데, 스키마를 나중에 바꾸면 이미 저장된 체크포인트가 무효화되기 때문이다.
  • 리듀서(reducer): 대화 이력처럼 "누적"되어야 하는 필드는 add_messages 같은 리듀서로 표시해 병렬 브랜치나 재호출 시에도 값이 덮어써지지 않도록 한다. 리듀서 표시가 없는 필드는 최신 업데이트로 단순 덮어쓰기된다.
  • 체크포인터(checkpointer): 모든 상태 전이가 자동으로 영속화되어, 일시정지/재개, 타임트래블 디버깅, 여러 인스턴스에 걸친 수평 확장이 "나중에 덧붙이는 기능"이 아니라 기본 제공 기능이 된다.
스택 모델과의 관계: 그래프 기반 상태는 스택 기반 모델의 상위 집합으로 볼 수 있다. 실제로 견고한 대화 에이전트를 만드는 방법은 "하나의 긴 프롬프트 루프" 대신 명시적인 상태·이벤트·전이를 정의해 대화가 분기하고, 실패에서 복구하고, 적응할 수 있게 만드는 것이라는 것이 2026년 LangGraph 실무 가이드의 반복적 결론이다. 즉 "그래프냐 스택이냐 FSM이냐"는 배타적 선택이 아니라, 상태를 얼마나 구조화해서 표현할 것인가의 스펙트럼 위에 있다.
근거: LangGraph Persistence 공식 문서(Checkpointer/Store, 2026-06 확인); LangGraph 1.2 릴리스(2026-05-11, durable graph execution); "A Practical Approach for Building Production-Grade Conversational Agents with Workflow Graphs" (arXiv 2505.23006, 2025) — LLM이 복잡한 조건부 규칙을 안정적으로 따르지 못하는 문제를 워크플로 그래프의 명시적 노드 구조로 보완한다는 논지 (상세 URL은 RESOURCES.md 참고)

④ 상태 전이 설계의 함정 — 값이 아니라 시스템으로 다루기

상태를 "무엇으로 표현할지" 정한 다음에는 "상태가 언제 어떻게 바뀌는지"를 운영 가능한 시스템으로 설계해야 한다. 이 단계에서 실무자가 자주 놓치는 네 가지가 있다.

함정 실무에서 드러나는 증상
스키마 변경과 체크포인트 무효화 운영 중 상태 스키마에 필드를 추가/변경했더니, 이미 저장돼 있던 진행 중 대화의 체크포인트가 역직렬화에 실패해 세션이 통째로 끊김
비멱등적(non-idempotent) 전이 재시도나 중복 이벤트 처리 시 같은 전이가 두 번 실행돼 slot 값이 중복 누적되거나 결제가 두 번 트리거됨
상태를 메모리에만 유지 서버 재시작/오토스케일링으로 인스턴스가 바뀌면 진행 중이던 대화의 상태가 유실돼 사용자가 처음부터 다시 말해야 함
"현재 상태" 하나만 저장하고 스택/이력을 버림 디그레션 이후 원래 작업으로 복귀할 정보가 없어, 사용자가 "아까 하던 거 계속할게요"라고 해도 시스템이 무엇을 하던 중이었는지 모름
설계 원칙: Dialogue state는 영속화(persist)·버전 관리(versioning)·멱등적 전이(idempotent transition)를 갖춘 데이터로 다뤄야 한다. "상태 = 세션 안의 변수 몇 개"라는 가벼운 취급은 트래픽이 작을 때는 문제가 드러나지 않다가, 재시작·재시도·스키마 변경이 실제로 발생하는 순간 장애로 나타난다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • State를 값이 아니라 구조로 이해하는가: "현재 상태 문자열 하나"라는 단순한 모델에 머물지 않고, 목표·slot·스택·이력을 포함한 구조화된 데이터로 설명하는가
  • FSM의 실패 모드를 구체적으로 아는가: "FSM은 유연성이 부족하다"는 추상적 답이 아니라, 상태 폭발이 왜 일어나는지 구체적 시나리오로 설명할 수 있는가
  • 인터럽션/디그레션을 정규 케이스로 설계하는가: 사용자가 주제를 바꾸는 것을 예외 처리가 아니라 스택 push/pop 같은 정규 연산으로 다루는 설계 감각이 있는가
  • 영속성과 운영 관점을 함께 고려하는가: 상태를 메모리 변수로만 생각하지 않고, 체크포인팅·스키마 마이그레이션·멱등성 같은 운영 이슈까지 고려하는가
  • 도구를 맹신하지 않는가: "LangGraph/Rasa를 쓰면 자동으로 해결된다"가 아니라, 그 도구들이 왜 그렇게 설계됐는지의 원리를 설명할 수 있는가
Senior 엔지니어는 dialogue state 설계를 "어떤 프레임워크를 쓰는가"의 문제가 아니라 "대화라는 비선형적 프로세스를 어떤 자료구조로 모델링하고, 그 자료구조를 어떻게 안전하게 영속화할 것인가"라는 시스템 설계 문제로 설명해야 한다.
실무 연결

실제 상황과 연결하기

상황 1: 상태 개수가 감당할 수 없이 늘어난 경우

초기에는 "주문조회 대기 → 배송지 확인 대기 → 완료" 3개 상태로 시작했지만, 기능이 늘면서 "환불 진행 중 배송조회로 새는 경우", "그 상태에서 다시 돌아오는 경우"처럼 조합이 생길 때마다 새 상태를 추가하다가 상태 정의 파일이 수백 줄로 불어나 아무도 전체 흐름을 파악하지 못하게 됐다. → "현재 하는 일"과 "잠시 미뤄둔 일"을 별도 상태로 나열하려 한 것이 원인이며, 스택 기반 모델로 전환해 "미뤄둔 일"을 스택 항목으로 표현하자 상태 정의 자체는 원래의 3~4개 수준으로 되돌아갔다.

상황 2: 배포 후 진행 중이던 대화가 전부 끊긴 경우

상태 스키마에 새 slot 필드를 추가하는 배포를 했더니, 배포 시점에 진행 중이던 모든 대화 세션이 상태 역직렬화 오류로 끊기고 사용자가 처음부터 다시 대화를 시작해야 했다. → 상태 스키마 변경이 기존 체크포인트를 무효화한다는 것을 고려하지 않은 배포 전략이 원인이며, 이후 스키마 버전 필드를 두고 구버전 체크포인트를 마이그레이션하거나 만료 처리하는 절차를 도입했다.

상황 3: 재시도로 같은 결제가 두 번 발생한 경우

네트워크 지연으로 클라이언트가 같은 확인 메시지를 두 번 전송했는데, 상태 전이 로직이 "결제 확인 상태로 전이 + 결제 API 호출"을 한 묶음으로 처리하면서 두 번째 이벤트도 동일하게 처리해 결제가 중복 발생했다. → 상태 전이를 멱등적으로 설계하지 않은 것이 원인이며, 전이 자체에 멱등키(idempotency key)를 부여해 이미 처리된 이벤트는 상태만 확인하고 부수 효과(side effect)는 재실행하지 않도록 고쳤다.

상황 4: 오토스케일링 중 대화 컨텍스트가 유실된 경우

상태를 애플리케이션 서버의 인메모리 딕셔너리로만 관리했는데, 트래픽 증가로 오토스케일러가 새 인스턴스를 띄우고 로드밸런서가 후속 요청을 다른 인스턴스로 보내면서 이전 인스턴스에만 있던 대화 상태에 접근할 수 없게 됐다. → 상태를 프로세스 로컬 메모리에만 둔 것이 원인이며, Redis 같은 외부 스토어나 체크포인터를 통한 영속화로 전환해 어떤 인스턴스가 요청을 받아도 동일한 상태에서 이어갈 수 있게 했다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
Dialogue state를 구조화된 데이터로 정의한다 "저는 dialogue state를 '현재 상태 문자열 하나'가 아니라, 진행 중인 목표, 채워진 slot 값, 진행 중인 목표들의 순서, 대화 이력을 포함하는 구조화된 데이터로 봅니다."
2
순수 FSM의 한계를 구체적으로 짚는다 "순수 상태 기계로 모든 인터럽션·디그레션 조합을 상태로 나열하면, 상태 개수가 대화 경로 수에 비례해 조합적으로 폭발합니다. 이건 실무에서 상태 정의 파일이 통제 불가능하게 커지는 형태로 나타납니다."
3
스택 기반 모델로 인터럽션을 정규 케이스화한다 (Senior 포인트) "그래서 저는 진행 중인 작업들을 스택으로 관리합니다. 사용자가 다른 주제로 새면 새 작업을 스택에 push하고, 그 작업이 끝나면 pop해서 원래 작업으로 자동 복귀시킵니다. Rasa CALM의 dialogue frame stack이나 Dialogflow CX의 flow stack이 이 패턴을 씁니다."
4
그래프 기반 상태로 일반화한다 "더 복잡한 에이전트형 대화에서는 상태를 타입이 정의된 스키마로 두고, 노드와 조건부 엣지로 전이를 구성하는 그래프 모델(LangGraph StateGraph 류)을 씁니다. 이때 체크포인터로 모든 전이를 영속화해 일시정지·재개· 디버깅이 기본으로 되게 합니다."
5
운영 관점의 함정을 언급한다 "상태 스키마를 바꾸면 기존 체크포인트가 무효화될 수 있다는 점, 전이를 멱등적으로 설계해야 재시도 시 부수 효과가 중복되지 않는다는 점, 상태를 프로세스 로컬 메모리가 아니라 외부 스토어에 영속화해야 스케일아웃과 재시작에 안전하다는 점을 함께 설계합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"저는 dialogue state를 '현재 상태를 나타내는 문자열 하나'로 보지 않습니다. 실제로는 지금 진행 중인 목표, 그 목표를 위해 채워진 slot 값들, 여러 목표가 중첩됐을 때의 순서 정보, 그리고 대화 이력을 포함하는 구조화된 데이터로 다룹니다.

순수 상태 기계로 이걸 표현하려고 하면 한계에 부딪힙니다. 사용자는 항상 선형적으로 대화하지 않고, 중간에 다른 질문으로 새거나 이전 답을 정정하기 때문에, 이런 모든 경우의 수를 별도 상태로 나열하면 상태 개수가 조합적으로 폭발합니다. 실무에서는 상태 정의 파일이 감당할 수 없이 커지는 형태로 이 문제가 드러납니다.

그래서 저는 진행 중인 작업들을 스택(LIFO)으로 관리하는 방식을 선호합니다. 사용자가 다른 주제로 새면 새 작업을 스택에 push하고 원래 작업은 잠시 대기 상태로 유지합니다. 새 작업이 끝나면 스택에서 pop되어 원래 작업으로 자동 복귀합니다. 이렇게 하면 인터럽션이나 디그레션을 예외 상황이 아니라 스택 연산이라는 정규 케이스로 처리할 수 있어서, 상태 개수 자체는 크게 늘지 않습니다. Rasa Pro의 dialogue frame stack이나 Dialogflow CX의 flow stack이 실제로 이 패턴을 씁니다.

더 복잡한 에이전트형 대화, 예를 들어 여러 tool을 호출하며 진행되는 작업에서는 상태를 타입이 정의된 스키마로 두고 노드와 조건부 엣지로 전이를 구성하는 그래프 기반 모델을 씁니다. LangGraph의 StateGraph가 대표적인데, 여기서는 모든 상태 전이가 체크포인터를 통해 자동으로 영속화되기 때문에 일시정지·재개·타임트래블 디버깅·수평 확장이 기본으로 따라옵니다.

마지막으로 상태를 운영 가능한 시스템으로 만들기 위한 함정들도 함께 설계합니다. 상태 스키마를 바꾸면 기존에 저장된 체크포인트가 무효화될 수 있기 때문에 스키마 버전 관리와 마이그레이션 절차를 두고, 재시도나 중복 이벤트로 같은 전이가 두 번 실행돼도 부수 효과가 중복되지 않도록 전이를 멱등적으로 설계하며, 상태를 프로세스 로컬 메모리가 아니라 Redis나 체크포인터 같은 외부 스토어에 영속화해서 인스턴스 재시작이나 오토스케일링에도 대화가 끊기지 않게 합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "스택에 쌓인 작업이 영원히 pop되지 않으면 어떻게 되나요?"

약한 답변: "그런 경우는 거의 없어서 별도 처리를 안 합니다."

강한 답변: "스택 항목마다 TTL(생존 시간)이나 최대 유휴 시간을 두고, 그 시간이 지나면 해당 항목을 자동으로 만료시켜 pop하거나 사용자에게 '아까 하시던 환불 요청은 계속 진행할까요?'라고 재확인합니다. 스택이 무한히 쌓이는 것을 방지하기 위해 스택 깊이 자체에도 상한을 두고, 상한을 넘으면 가장 오래된 항목을 정리하거나 새 작업을 시작하기 전에 사용자에게 명시적으로 우선순위를 묻습니다."

F2. "FSM과 그래프 모델을 같은 서비스 안에서 함께 쓸 수 있나요?"

강한 답변: "네, 오히려 그게 실무에서 흔한 구조입니다. 정형화되고 트래픽이 큰 플로우 (예: 비밀번호 재설정처럼 단계가 고정된 작업)는 검증이 쉬운 FSM으로 두고, 여러 tool을 호출하며 유연하게 분기해야 하는 복잡한 상담은 그래프 기반 상태로 둡니다. Day 4의 hybrid routing 원칙과 마찬가지로, 예측 가능성이 필요한 부분과 유연성이 필요한 부분을 구분해서 각각 다른 상태 모델을 적용합니다."

F3. "상태 스키마를 마이그레이션 없이 바꿔야 한다면 어떻게 하시겠어요?"

강한 답변: "가능하면 무중단 마이그레이션은 피하고, 새 필드는 기본값을 가진 optional 필드로 추가해 구버전 체크포인트도 역직렬화가 가능하게 합니다. 구조적으로 호환이 안 되는 변경이라면, 새 스키마 버전으로 새 세션만 받고 기존 세션은 만료될 때까지 구버전 로직으로 처리하는 블루/그린 방식의 전이 기간을 둡니다. '무중단으로 즉시 전환'을 목표로 삼지 않고, 진행 중인 대화를 끊지 않는 것을 우선순위로 둡니다."

F4. "상태를 외부 스토어에 영속화하면 지연시간이 늘어나지 않나요?"

강한 답변: "매 턴마다 전체 상태를 동기적으로 디스크에 쓰면 당연히 지연이 늘어납니다. 그래서 저는 상태를 Redis처럼 빠른 인메모리 스토어에 두고, 영속성이 중요한 체크포인트(예: 결제 확정 직전)에서만 더 강한 내구성 보장을 요구합니다. 즉 모든 전이를 똑같은 수준으로 영속화하지 않고, 전이의 위험도에 따라 저장 계층의 내구성 수준을 차등화합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • State를 구조화된 데이터(목표+slot+스택+이력)로 정의한다
  • FSM의 한계를 상태 폭발이라는 구체적 메커니즘으로 설명한다
  • 인터럽션/디그레션을 스택 push/pop이라는 정규 연산으로 다룬다
  • 스키마 변경·멱등성·영속화 같은 운영 함정을 함께 고려한다
  • FSM/스택/그래프를 배타적 선택이 아닌 스펙트럼으로 이해하고 상황별로 조합한다

약한 답변의 특징

  • State를 "현재 상태 문자열 하나"로만 이해한다
  • FSM의 한계를 "유연성이 부족하다"는 추상적 표현에 그친다
  • 디그레션을 "예외 처리로 따로 코딩"한다고만 말한다
  • 상태를 메모리 변수로만 생각하고 영속화·마이그레이션을 언급하지 않는다
  • "OO 프레임워크를 쓰면 된다"고만 말하고 그 프레임워크가 왜 그렇게 설계됐는지 원리를 설명하지 못한다
핵심 차이: 강한 답변은 dialogue state 설계를 "비선형적 대화를 안전하게 영속화 가능한 자료구조로 모델링하는 시스템 설계 문제"로 다루고, 약한 답변은 "대화 흐름을 코드로 옮기는 구현 작업"으로만 다룬다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "LLM 기반 챗봇에는 상태 관리가 필요 없다"

사실: LLM이 매 턴 대화 이력을 다시 읽더라도, 진행 중인 작업의 slot 값·확정 여부·중첩된 목표는 여전히 명시적으로 추적해야 한다. 그렇지 않으면 LLM이 이미 확인한 정보를 다시 묻거나, 취소된 작업을 계속 진행하는 등 일관성 없는 행동을 한다.

오해 2: "그래프 기반 상태가 항상 FSM보다 우월하다"

사실: 정형화되고 검증이 중요한 짧은 플로우(예: 본인 인증)는 오히려 단순한 FSM이 더 예측 가능하고 감사하기 쉽다. 그래프 기반 모델은 복잡도가 그만큼 크므로, 실제로 유연성이 필요한 부분에만 선택적으로 적용해야 한다.

오해 3: "스택에 쌓인 작업은 언젠가 알아서 정리된다"

사실: 명시적인 TTL이나 최대 깊이 제한이 없으면 스택은 끝없이 쌓일 수 있고, 오래된 미완료 작업이 세션 안에 계속 남아 이후 대화 해석에 잘못된 문맥으로 영향을 준다.

오해 4: "상태 스키마는 한 번 정하면 거의 바뀌지 않는다"

사실: 서비스가 성장하면 새 slot·새 목표·새 채널이 계속 추가되므로 스키마 변경은 예외가 아니라 정상적인 운영 활동이다. 변경이 드물 것이라 가정하고 마이그레이션 전략을 준비하지 않으면, 실제 변경이 필요한 시점에 급하게 임시방편으로 대응하게 된다.

주니어 State가 여러 값을 포함해야 한다는 것은 알지만, FSM의 한계(상태 폭발)나 스택/그래프 모델의 구체적 동작 방식은 설명하지 못함
중급 FSM의 한계와 스택 기반 인터럽션 처리를 설명할 수 있음. 다만 스키마 마이그레이션·멱등성·영속화 같은 운영 함정까지는 다루지 못함
시니어 구조화된 상태 정의, FSM/스택/그래프의 상황별 선택 기준, 스키마 마이그레이션·멱등성·외부 영속화까지 운영 관점을 포함해 일관되게 설명하고 구체적 실패 시나리오로 근거를 댐
자가 점검 질문:
1. Dialogue state에 포함돼야 할 최소 네 가지 요소를 말할 수 있는가?
2. 순수 FSM에서 상태 폭발이 왜 일어나는지 구체적 시나리오로 설명할 수 있는가?
3. 스택 기반 모델이 인터럽션을 예외가 아닌 정규 연산으로 만드는 원리를 설명할 수 있는가?
4. 상태 스키마 변경이 왜 체크포인트 무효화로 이어지는지, 그리고 이를 어떻게 완화하는지 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • Dialogue state는 문자열 하나가 아니라 목표·slot·스택·이력을 포함한 구조화된 데이터다.
  • 순수 FSM은 인터럽션·디그레션 조합을 상태로 나열하려 하면 상태 폭발에 빠진다.
  • 스택(LIFO) 기반 모델(Rasa dialogue frame stack, Dialogflow CX flow stack)은 인터럽션을 push/pop이라는 정규 연산으로 다뤄 상태 폭발 없이 유연성을 확보한다.
  • 그래프 기반 상태(LangGraph StateGraph)는 타입 스키마 + 노드/엣지 + 체크포인터로 더 복잡한 에이전트형 대화를 구조화하며, 일시정지· 재개·디버깅을 기본 제공한다.
  • FSM·스택·그래프는 배타적 선택이 아니라 스펙트럼이며, 정형화된 플로우와 유연한 상담을 구분해 상황별로 조합하는 것이 실무 표준이다.
  • 상태는 스키마 마이그레이션·멱등적 전이·외부 영속화를 갖춘 운영 가능한 시스템으로 설계해야 한다.
