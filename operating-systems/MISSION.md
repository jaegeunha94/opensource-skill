# 운영체제 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 개발자 기술 면접에서 운영체제 지식을 **성능, 동시성, 메모리, 파일 I/O, 장애 진단**과 연결해 설명할 수 있도록 훈련한다.

운영체제 면접은 "스케줄링 알고리즘을 아는가"만 보지 않는다. 경력자에게는 스레드 폭증, context switch 증가, lock contention, memory leak, page fault, disk I/O 병목 같은 실무 증상을 보고 원리를 기반으로 판단할 수 있는지를 묻는다.

## 평가 대상 역량

| 역량 | 면접에서 확인하려는 것 |
|------|------------------------|
| 프로세스/스레드 | 실행 단위, 주소 공간, context switch, 스레드 안정성을 실무와 연결하는가 |
| 동시성 제어 | mutex, semaphore, condition variable, race condition, deadlock을 설명하고 예방하는가 |
| 스케줄링 | CPU-bound/I/O-bound workload와 latency/throughput trade-off를 이해하는가 |
| 메모리 관리 | virtual memory, paging, TLB, page fault, swapping을 성능 진단에 연결하는가 |
| 파일/I/O | buffer cache, fsync, blocking/non-blocking I/O, DMA, interrupt를 이해하는가 |
| 장애 분석 | CPU 사용률, load average, thread dump, memory profile, I/O wait를 해석하는가 |
| 설계 판단 | 프로세스, 스레드, async, queue, worker pool 선택 기준을 말하는가 |

## Interview 레슨 작성 규칙

각 Day 레슨은 반드시 **면접 질문 중심**으로 작성한다.

1. **면접 질문**
   - 실무 경험이 드러나는 질문으로 시작한다.
   - 예: "서버의 CPU 사용률은 낮은데 응답이 느립니다. 운영체제 관점에서 무엇을 확인하시겠습니까?"
2. **전제 개념**
   - 어려운 개념은 먼저 쉬운 한국어로 설명한다.
   - 프로세스, 스레드, 주소 공간, 커널 모드 같은 용어는 필요한 배경부터 풀어쓴다.
3. **왜 면접에서 묻는가**
   - 기본 CS 지식 확인인지, 장애 대응력 확인인지, 고성능 서버 설계 경험 확인인지 구분한다.
4. **실무 연결**
   - 웹 서버, 배치 작업, DB 커넥션 풀, worker pool, 로그 처리, 파일 업로드 같은 상황과 연결한다.
5. **답변 사고 순서**
   - 증상 확인 -> OS 지표 확인 -> 실행 단위 확인 -> 병목 가설 -> 측정 -> 해결안 비교 순서로 답변하도록 안내한다.
6. **핵심 답변 예시**
   - 면접에서 바로 말할 수 있는 답변을 짧은 버전과 깊이 있는 버전으로 제공한다.
7. **Trade-off**
   - 성능, latency, throughput, 메모리 사용량, 구현 복잡도, 디버깅 난이도를 비교한다.
8. **흔한 오해**
   - "스레드는 항상 빠르다", "비동기는 스레드를 쓰지 않는다", "메모리가 남으면 page fault는 문제가 아니다" 같은 오해를 바로잡는다.
9. **Follow-up 질문**
   - 면접관이 추가로 물을 질문과 답변 방향을 포함한다.
10. **10년차 이상 답변 기준**
   - 정의 암기가 아니라 지표 해석, 병목 추론, 운영 경험이 드러나는지 평가한다.

## 완료 기준

- 운영체제 개념을 실제 서버 성능 문제와 연결해 설명할 수 있다.
- 동시성 문제를 재현, 진단, 완화하는 사고 순서를 말할 수 있다.
- OS 지표를 보고 병목 가설을 세울 수 있다.
- 약한 답변과 강한 답변의 차이를 스스로 설명할 수 있다.
