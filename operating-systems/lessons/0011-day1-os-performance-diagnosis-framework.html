OS 관점 성능 진단 프레임워크

CPU 사용률 하나만 보고 "서버가 건강하다"고 판단하는 것은 10년차 이상 면접에서 약한 답변으로 분류됩니다. 이 레슨에서는 OS 지표를 체계적으로 읽고 병목 가설을 세우는 프레임워크를 익힙니다.

오늘의 면접 질문

Senior Interview Question
"서버의 CPU 사용률은 20% 수준인데 API 응답 시간이 평소보다 5배 느려졌습니다. 운영체제 관점에서 무엇을 먼저 확인하겠습니까? 순서대로 설명해주세요."

이 질문은 단순히 명령어를 아는지가 아니라, OS 지표를 연결해서 병목 원인을 좁혀나가는 사고력을 보는 질문입니다. "top을 보겠습니다"에서 멈추면 약한 답변입니다.

전제 개념: 이것부터 알고 있어야 합니다

1. CPU 사용률이 낮아도 느릴 수 있는 이유

CPU 사용률(%cpu)은 CPU가 실제로 연산을 하고 있는 시간 비율입니다. 이 숫자가 낮다는 것은 CPU가 바쁘지 않다는 뜻입니다. 그렇다면 왜 느릴까요? CPU는 기다리고 있기 때문입니다.

스레드가 CPU를 쓰지 않고 대기하는 대표적인 경우:

  • I/O wait: 디스크 읽기/쓰기, 네트워크 응답을 기다리는 중
  • Lock wait: mutex, 데이터베이스 row lock, 파일 잠금을 기다리는 중
  • Swap I/O: 물리 메모리가 부족해 디스크 swap에서 페이지를 불러오는 중
  • System call wait: 커널이 자원을 준비하는 동안 유저 스레드가 블록된 상태

이 모든 경우에 CPU 사용률은 낮게 나옵니다. 기다리는 동안 CPU는 다른 일을 할 수 없거나 idle 상태에 있기 때문입니다.

2. Load Average란 무엇인가

uptime 명령어나 top 상단에 보이는 세 개의 숫자가 load average입니다. 

load average: 3.20, 2.91, 2.15  ← 1분, 5분, 15분 평균

이 숫자는 실행 대기 중이거나 I/O 대기 중인 프로세스/스레드의 평균 개수입니다. CPU 코어가 4개인 서버에서 load average가 4.0이면 코어가 꽉 찬 상태입니다. 8.0이면 처리 속도의 2배 분량의 요청이 쌓이고 있다는 의미입니다.

흔한 오해
Load average는 CPU 부하만 반영하지 않습니다. Disk I/O를 기다리는 프로세스(D 상태)도 포함됩니다. CPU 사용률이 낮아도 load average가 높다면 I/O 병목일 가능성이 높습니다.

3. 스레드 상태 (Thread State)

Linux 커널에서 프로세스(스레드)는 다음 상태 중 하나에 있습니다:

R — Running
CPU에서 실행 중이거나 실행 큐에서 대기 중. 이 상태가 많으면 CPU 부하.
S — Sleeping
이벤트를 기다리며 잠든 상태. 시그널로 깨울 수 있음. 일반적인 대기 상태.
D — Uninterruptible Sleep
I/O 완료를 기다리며 블록된 상태. 시그널로 깨울 수 없음. 이 상태가 많으면 I/O 병목.
Z — Zombie
종료됐지만 부모 프로세스가 wait()을 호출하지 않아 PCB만 남은 상태.
핵심 포인트
D 상태 스레드가 많다 = I/O 대기 병목이 가장 유력한 가설입니다. CPU 사용률이 낮고 load average가 높으며 D 상태 스레드가 보이면 디스크나 네트워크 I/O를 가장 먼저 의심하세요.

4. I/O Wait (iowait)

top의 %wa 또는 vmstat의 wa 컬럼입니다. CPU가 I/O 완료를 기다리느라 아무 일도 못 하고 있는 시간 비율입니다. iowait가 높으면 디스크 읽기/쓰기가 병목임을 의심합니다.

5. Context Switch

CPU가 하나의 스레드 실행을 멈추고 다른 스레드를 실행하기 위해 레지스터, 프로그램 카운터, 스택 포인터를 저장하고 복원하는 작업입니다. 이 작업 자체가 오버헤드입니다.

스레드가 많아지거나 락 경합이 심해지면 context switch 횟수가 급증하고, vmstat의 cs 컬럼이 비정상적으로 높아집니다.

왜 면접에서 이 질문을 묻는가

이 질문은 세 가지를 동시에 평가합니다:

  1. 장애 대응 경험이 있는가 — 운영 중 실제 성능 저하를 진단해본 사람은 어떤 지표를 어떤 순서로 봐야 하는지 안다. 이론만 아는 사람은 순서가 뒤죽박죽이다.
  2. OS 지표를 해석할 수 있는가 — "top 보겠습니다"에서 멈추지 않고, top에서 무엇을 읽고 어떤 다음 명령어로 이어가는지를 본다.
  3. 가설 기반 사고를 하는가 — 모든 지표를 무작위로 보는 것이 아니라, 증상에서 가능한 원인을 좁히고, 가설에 맞는 지표를 확인하고, 틀리면 다음 가설로 넘어가는지를 본다.

실무 연결

실제로 이런 상황은 자주 발생합니다:

  • 로그 파일 burst write: 트래픽 급증 시 로그를 디스크에 동기 쓰기(fsync)하면 디스크 I/O가 병목이 되어 API 스레드까지 블록된다. CPU 사용률은 낮지만 iowait가 튄다.
  • DB 연결 풀 고갈: DB 연결 풀이 모두 사용 중이면 API 요청 스레드들이 연결을 기다리며 sleep 상태에 빠진다. CPU 사용률은 낮고 active thread 수는 높다.
  • JVM GC 로 인한 Stop-the-World: Full GC가 수백 ms 동안 모든 스레드를 멈추면 CPU 사용률이 GC 순간에 치솟다가 다시 낮아진다. 평균 CPU는 낮지만 응답 시간은 높다.
  • swap 사용 증가: 메모리 누수로 물리 메모리가 차면 OS가 오래된 페이지를 swap으로 내보내고, 필요할 때 다시 불러오는 작업(major page fault)이 발생한다. 수십 ms~수백 ms latency가 추가된다.

답변 사고 순서

면접에서 이 사고 흐름을 보여주면 강한 답변이 됩니다:

  1. 증상 재확인 — 단서 파악
    "CPU 사용률이 낮다"는 것이 핵심 단서입니다. CPU가 놀고 있다는 뜻이므로 CPU 연산이 아닌 대기가 병목임을 전제합니다.
  2. Load Average와 스레드 상태 확인
    uptime 또는 top으로 load average를 봅니다. ps aux나 top에서 D 상태(uninterruptible sleep) 스레드 수를 봅니다.
    ps -eo state,pid,cmd | grep "^D"
  3. I/O 병목 여부 확인
    iostat -x 1으로 디스크 사용률(%util)과 wait time(await)을 봅니다. vmstat 1으로 wa(I/O wait)와 cs(context switch)를 봅니다.
  4. 메모리 압력 확인
    free -h로 available memory와 swap 사용량을 확인합니다. Swap이 사용 중이면 vmstat의 si(swap in), so(swap out)로 swap I/O 빈도를 확인합니다.
  5. 네트워크 병목 확인
    외부 서비스(DB, 캐시, 써드파티 API) 호출이 많은 서비스라면 ss -s로 소켓 상태를 봅니다. ESTABLISHED 연결 수, TIME_WAIT 폭증 여부를 확인합니다.
  6. Context Switch 이상 여부 확인
    pidstat -w 1 -p [pid]으로 문제 프로세스의 voluntary/involuntary context switch를 봅니다. involuntary cs가 많으면 CPU 경쟁이 심하고, voluntary cs가 많으면 I/O나 lock 대기가 많다는 신호입니다.
  7. 가설 설정과 다음 액션
    지표를 종합해 "디스크 I/O 병목", "DB 연결 대기", "swap 압력" 중 가장 유력한 가설을 선택하고, 애플리케이션 수준(스레드 덤프, APM 트레이스, 슬로우 쿼리 로그)으로 연결합니다.

핵심 명령어 레퍼런스

# 1. 전체 시스템 개요 (CPU, 메모리, swap, load average)
top

# 2. 1초 간격 시스템 지표 (cs, wa, si, so 포함)
vmstat 1

# 3. 디스크 I/O 상세 (장치별 %util, await, r/s, w/s)
iostat -x 1

# 4. D 상태(I/O wait) 프로세스 목록
ps -eo state,pid,ppid,cmd | awk '$1 == "D"'

# 5. 프로세스별 context switch 빈도
pidstat -w 1

# 6. 메모리 사용량 및 swap
free -h

# 7. 소켓 연결 상태 요약
ss -s

# 8. 네트워크 연결 상태 (TIME_WAIT, CLOSE_WAIT 등)
ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn
vmstat 읽는 법
vmstat 1 5 출력에서 주목할 컬럼:
r — 실행 대기 중인 프로세스 수 (CPU 수보다 크면 CPU 포화)
b — I/O 대기 중(D 상태) 프로세스 수
wa — I/O wait %
cs — 초당 context switch 횟수
si/so — swap in/out (0이 아니면 메모리 압력)

핵심 답변 예시

짧은 버전 (30초 답변)

강한 답변 (짧은 버전)

"CPU 사용률이 낮다는 게 핵심 단서입니다. CPU가 아닌 대기가 병목이라는 뜻입니다. 먼저 load average와 D 상태 스레드 수를 확인해 I/O 대기인지 파악하고, iostat -x로 디스크 utilization을 보겠습니다. 이와 함께 vmstat로 swap 사용 여부와 context switch 폭증을 확인합니다. 지표에 따라 I/O 병목, 메모리 압력, DB 연결 대기 중 원인을 좁혀가겠습니다."

깊이 있는 버전 (2분 답변)

강한 답변 (깊이 있는 버전)

"CPU 사용률 20%라는 수치에서 이미 방향이 정해집니다. CPU가 유휴 상태라면 연산이 아닌 블록킹 대기가 요청 처리 시간을 늘리고 있다고 가설을 세웁니다.

첫 번째로 uptime과 vmstat 1을 보겠습니다. Load average가 코어 수 대비 높고 b 컬럼(D 상태 프로세스)이 크면 I/O 대기가 주된 원인임을 확신할 수 있습니다. iostat -x 1으로 디스크의 await와 %util을 확인합니다. %util이 80% 이상이면 디스크가 포화 상태입니다.

두 번째로 메모리를 봅니다. free -h에서 available이 매우 작거나 vmstat의 si/so가 0이 아니면 swap 압력으로 인한 major page fault가 latency에 기여하고 있을 수 있습니다.

세 번째로 애플리케이션 레벨로 내려갑니다. I/O 병목이 맞다면 로그 쓰기에 fsync가 걸려 있는지, DB 커넥션 풀이 고갈된 것은 아닌지, 외부 API 응답이 느려진 것은 아닌지 애플리케이션 스레드 덤프나 APM 트레이스로 연결합니다. OS 지표와 애플리케이션 지표를 교차 검증하여 RCA를 완성하겠습니다."

좋은 답변과 약한 답변의 차이

약한 답변

"top 명령어로 CPU 사용률과 메모리를 확인하겠습니다. 그래도 원인을 모르면 재시작해보겠습니다."

왜 약한가: 이미 CPU 사용률이 낮다고 했는데 CPU를 다시 보겠다는 것은 단서를 무시한 것입니다. 재시작은 RCA 없이 증상만 임시로 해소하는 방법으로, 반복 발생을 막지 못합니다. 면접관은 "이 사람은 장애 대응 경험이 없구나"라고 판단합니다.

약한 답변

"I/O wait가 높은지, context switch가 많은지, swap이 있는지, 네트워크가 문제인지 모두 확인하겠습니다."

왜 약한가: 순서 없이 모든 걸 나열하는 것은 사고 체계가 없어 보입니다. "모두 봐야 한다"는 맞지만, 어떤 단서로 어떤 가설을 세우고, 어떤 순서로 좁혀가는지를 보여주지 않으면 운영 경험이 없는 것처럼 들립니다.

강한 답변의 패턴

① 증상에서 단서를 추출 → ② 단서로 가설을 좁힘 → ③ 가설 검증에 맞는 지표 선택 → ④ 지표 결과로 다음 가설로 이동 또는 확정 → ⑤ 애플리케이션 수준으로 연결

Senior-level Follow-up 질문

▶ "I/O wait가 높지 않은데도 응답이 느리다면 다음은 무엇을 보겠습니까?"
I/O wait가 낮으면 디스크 I/O 이외의 원인을 의심합니다.
  • Lock 경합: pidstat -w에서 voluntary context switch가 많다면 스레드들이 mutex/semaphore를 기다리며 스스로 CPU를 양보하는 것입니다. JVM이라면 jstack으로 스레드 덤프를 뜨고 WAITING/BLOCKED 상태 스레드와 잠긴 모니터를 확인합니다.
  • 네트워크 지연: 외부 DB, 캐시, API 호출 시간이 길어진 경우. ss -tan으로 연결 상태, netstat -s로 TCP 재전송 통계를 확인합니다.
  • GC 압력: JVM이라면 GC 로그를 보고 Stop-the-World 빈도와 시간을 확인합니다. GC가 잦으면 힙 사이즈 또는 객체 생성 패턴을 검토합니다.
▶ "Swap을 완전히 끄는 것과 켜두는 것의 trade-off는 무엇입니까?"
Swap OFF의 장점: swap I/O latency가 없어서 예측 가능한 응답 시간을 유지합니다. 메모리 부족 시 OOM Killer가 프로세스를 종료하므로, 느린 swap 때문에 전체 시스템이 응답 불능이 되는 상황을 막습니다.

Swap ON의 장점: 메모리 누수나 급격한 부하 증가 시 OOM 즉시 종료 대신 시간을 벌 수 있습니다. 버퍼링 용도로 OS가 메모리를 더 공격적으로 쓸 수 있습니다.

실무 선택: 레이턴시가 중요한 서비스(웹 API, 실시간 처리)는 swap OFF 또는 최소화(vm.swappiness=0)가 일반적입니다. 배치 작업이나 스테이트풀 서비스는 swap을 남겨두는 경우도 있습니다.
▶ "Context switch가 많다고 판단했을 때, 어떻게 줄이겠습니까?"
원인에 따라 접근이 다릅니다:
  • 스레드 과다: 스레드 풀 크기를 줄이거나 이벤트 기반(비동기) 방식으로 전환합니다. I/O-bound 작업이 많다면 async I/O가 효과적입니다.
  • Lock 경합: Critical section 범위를 줄이거나, 잠금 없는 자료구조(lock-free queue), CAS 기반 atomic 연산을 활용합니다.
  • CPU affinity 설정: taskset으로 특정 스레드를 특정 코어에 고정하면 캐시 재사용률이 올라가고 involuntary context switch가 줄어듭니다.
▶ "모니터링 툴 없이 현장에서 진단해야 한다면 어떻게 하겠습니까?"
Linux 기본 명령어만으로 충분합니다: 
# 30초 스냅샷
uptime && vmstat 1 5 && iostat -x 1 3 && free -h && ss -s

# D 상태 프로세스 확인
ps -eo state,pid,ppid,user,cmd | awk '$1=="D"' | head -20

# 특정 프로세스 context switch
pidstat -w 1 -p $(pgrep -d, java) 5
이 명령어들을 조합해 1~2분 안에 병목 카테고리를 식별하는 것이 현장 능력입니다. 스크립트로 저장해두면 더 빠릅니다.

지표 순서 load average → D 상태 → iowait → swap → context switch 순서로 체계적으로 설명했는가
명령어 구체성 막연히 "확인하겠다"가 아니라 실제 명령어를 2개 이상 언급했는가
가설 기반 사고 지표별로 "이 값이 높으면 이것이 원인"이라는 연결을 설명했는가
애플리케이션 연결 OS 지표 후 애플리케이션 레벨(스레드 덤프, APM, 슬로우 쿼리)로 연결했는가
Trade-off 언급 Swap on/off, 스레드 풀 크기 같은 판단 기준을 설명할 수 있는가
운영 경험 표현 실제 겪었거나 분석해본 유사 사례를 언급했는가
