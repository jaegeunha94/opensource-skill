메모리 관리 기초

CPU가 아무리 빠르더라도 실행할 프로그램이 메모리에 올라와 있지 않으면 동작하지 않습니다. OS는 한정된 메모리를 여러 프로세스가 안전하게 공유할 수 있도록 관리해야 합니다. 오늘은 주소 바인딩, 연속 메모리 할당, 그리고 단편화 문제를 배웁니다.

학습 목표
  • 주소 바인딩의 세 시점(컴파일, 적재, 실행)을 구분할 수 있다.
  • 연속 할당의 고정 분할 vs 가변 분할 차이를 설명할 수 있다.
  • 내부 단편화 vs 외부 단편화를 구별하고 압축(Compaction)의 한계를 이해한다.
  • 최초 적합·최적 적합·최악 적합 알고리즘을 비교할 수 있다.

1. 왜 메모리 관리가 필요한가?

초기 컴퓨터는 프로세스 하나가 메모리 전체를 독점했습니다. 현대 OS는 다중 프로그래밍(multiprogramming)을 지원하여 여러 프로세스를 동시에 메모리에 올리는데, 이 때 세 가지 목표를 달성해야 합니다.

목표설명
보호 (Protection)프로세스 A가 프로세스 B의 메모리 공간을 침범하지 못하게 한다.
공유 (Sharing)필요한 경우(예: 공유 라이브러리) 두 프로세스가 같은 물리 메모리 페이지를 사용하게 한다.
효율 (Efficiency)낭비 없이 메모리를 할당하고, 해제된 공간을 재사용한다.

2. 주소 바인딩 (Address Binding)

소스 코드의 변수·함수가 실제 물리 메모리 주소와 연결되는 시점을 바인딩이라 합니다. 바인딩은 세 시점에 발생할 수 있습니다.

2-1. 컴파일 시간 바인딩 (Compile-time Binding)

컴파일러가 절대 주소(absolute address)를 코드에 직접 삽입합니다. 프로그램이 항상 같은 물리 주소에 적재될 때만 사용할 수 있습니다. 주소가 바뀌면 재컴파일이 필요하므로 현대 OS에서는 거의 사용하지 않습니다.

[소스코드: int x at 0x0100] ↓ 컴파일 [기계코드: MOV 0x0100, ...] ← 주소가 고정

2-2. 적재 시간 바인딩 (Load-time Binding)

컴파일러는 재배치 가능한 주소(relocatable address)를 사용하고, 로더(loader)가 프로그램을 메모리에 올릴 때 실제 시작 주소를 기준으로 최종 주소를 결정합니다. 프로그램이 실행되는 동안에는 주소가 고정됩니다.

[컴파일 결과: MOV base+100, ...] ↓ 적재 (base = 0x3000) [메모리: MOV 0x3100, ...] ← 적재 시 계산

2-3. 실행 시간 바인딩 (Execution-time Binding)

프로세스가 실행 중에도 메모리 위치가 바뀔 수 있습니다. 하드웨어 지원(MMU, Memory Management Unit)이 필요하며, CPU가 메모리에 접근할 때마다 논리 주소(logical address)를 물리 주소(physical address)로 변환합니다.

CPU가 생성 → 논리 주소 0x0100 │ [MMU] (base register = 0x3000) │ 물리 주소 = 0x0100 + 0x3000 = 0x3100

그림 1 — MMU의 동적 주소 변환 (Dynamic Relocation)

핵심 개념
  • 논리 주소 (Logical / Virtual Address): CPU가 생성하는 주소. 프로세스 입장의 주소.
  • 물리 주소 (Physical Address): 실제 메모리 칩 위의 주소.
  • 재배치 레지스터 (Base Register): 논리 주소에 더해지는 오프셋.
  • 한계 레지스터 (Limit Register): 허용 범위를 넘으면 세그멘테이션 오류 발생 → 메모리 보호.

3. 연속 메모리 할당 (Contiguous Memory Allocation)

각 프로세스에 연속된 물리 메모리 블록을 할당하는 가장 단순한 방식입니다.

3-1. 고정 분할 (Fixed Partitioning)

메모리를 고정된 크기의 구역(partition)으로 나누어 각 프로세스에 하나씩 배정합니다.

┌──────────────────┐ 0x0000 │ OS 커널 │ ├──────────────────┤ 0x1000 │ Partition 1 │ 4 KB — 프로세스 A (3 KB 사용) ├──────────────────┤ 0x2000 │ Partition 2 │ 4 KB — 프로세스 B (1 KB 사용) ├──────────────────┤ 0x3000 │ Partition 3 │ 4 KB — 미사용 └──────────────────┘ 0x4000

그림 2 — 고정 분할: 파티션이 고정된 크기

단점: 프로세스보다 파티션이 크면 남는 공간이 생깁니다 → 내부 단편화.

3-2. 가변 분할 (Variable Partitioning)

프로세스 크기에 맞추어 정확한 크기로 파티션을 동적으로 생성합니다. 내부 단편화는 없지만, 프로세스가 종료되면 크기가 제각각인 빈 구멍(hole)이 생겨 외부 단편화가 발생합니다.

초기 상태: [OS][ 프로세스 A (200 KB) ][ 프로세스 B (150 KB) ][ 빈 공간 (50 KB) ] B 종료 후: [OS][ 프로세스 A (200 KB) ][ 빈 구멍 (150 KB) ][ 빈 공간 (50 KB) ] 총 빈 공간 = 200 KB이지만, 150 KB짜리 프로세스 C 적재 가능 총 빈 공간 = 200 KB이지만, 180 KB짜리 프로세스 D는 연속 공간 부족으로 적재 불가

그림 3 — 외부 단편화: 빈 공간이 흩어져 있어 큰 프로세스를 올릴 수 없음

4. 동적 메모리 할당 알고리즘

빈 구멍(hole) 목록에서 어느 구멍에 프로세스를 넣을지 결정하는 전략입니다.

알고리즘방식장점단점
최초 적합 (First Fit) 처음 만나는 충분히 큰 구멍에 배정 빠름 앞부분에 작은 조각이 쌓임
최적 적합 (Best Fit) 딱 맞는 가장 작은 구멍에 배정 낭비 최소화 (이론적) 전체 탐색, 아주 작은 잔여 조각 발생
최악 적합 (Worst Fit) 가장 큰 구멍에 배정 남은 공간이 커서 재활용 가능 전체 탐색, 대형 구멍 소진
실용 팁

시뮬레이션 연구에서는 최초 적합이 평균 성능이 가장 좋은 경우가 많습니다. 최적 적합은 직관적으로 유리해 보이지만 매우 작은 잔여 조각이 오히려 공간을 낭비합니다.

5. 단편화 (Fragmentation)

종류발생 상황예시
내부 단편화
(Internal Fragmentation)		 할당된 블록이 프로세스보다 커서 내부에 빈 공간이 생김 4 KB 파티션에 3 KB 프로세스 → 1 KB 낭비
외부 단편화
(External Fragmentation)		 전체 빈 공간은 충분하지만 연속된 빈 블록이 없어 할당 불가 50 KB + 60 KB 구멍이 있지만 100 KB 연속 공간 필요

5-1. 압축 (Compaction)

외부 단편화 해결책 중 하나. 사용 중인 프로세스들을 한 쪽으로 밀어 빈 공간을 하나로 합칩니다.

압축 전: [OS][A][빈][B][빈][C] 압축 후: [OS][A][B][C][ 큰 빈 공간 ]

그림 4 — 압축으로 외부 단편화 해소

압축의 한계
  • 프로세스를 이동하는 동안 해당 프로세스는 일시 중단되어야 합니다.
  • 대용량 메모리에서는 이동 비용이 매우 크고 지연이 길어집니다.
  • 그래서 현대 OS는 압축 대신 가상 메모리(페이징)를 사용합니다. → Day 8에서 학습.

6. 외부 단편화의 50% 규칙

Knuth가 분석한 경험 법칙에 따르면, 가변 분할에서 N개의 블록이 할당되어 있을 때 평균적으로 약 0.5N개의 빈 구멍이 생깁니다. 즉, 전체 메모리의 약 1/3이 단편화로 낭비될 수 있습니다. 이것이 페이징으로 넘어가는 핵심 동기입니다.
