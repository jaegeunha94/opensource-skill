동시성 제어와 Race Condition

"가끔씩 재고가 마이너스로 떨어진다", "포인트가 이중으로 적립된다" 같은 버그는 재현이 안 돼서 더 무섭습니다. 이 레슨은 그런 간헐적 버그를 race condition으로 알아보고, 발견·재현·예방하는 사고 과정을 훈련합니다.

오늘의 면접 질문

Senior Interview Question
"쿠폰 발급 API에 동시에 여러 요청이 들어오면 재고가 음수로 떨어지는 버그가 가끔 발생한다는 리포트를 받았습니다. 로컬에서는 재현이 안 됩니다. 이 버그를 어떻게 진단하고, race condition이라면 어떻게 예방하시겠습니까? 서로 다른 예방 방법들의 트레이드오프도 설명해주세요."

이 질문은 "lock을 걸면 된다"는 한 줄 답변으로는 통과되지 않습니다. 면접관은 왜 로컬에서 재현이 안 되는지, race condition을 어떤 절차로 특정하는지, 그리고 여러 해결책(mutex, atomic 연산, 낙관적 락, DB 제약조건) 중 상황에 맞는 것을 고를 수 있는지를 봅니다.

전제 개념: 이것부터 알고 있어야 합니다

1. Critical Section(임계 구역)이란 무엇인가

여러 실행 흐름(스레드/프로세스)이 공유 자원(변수, 파일, DB 행)에 접근해서 값을 읽고 수정하는 코드 구간을 critical section이라고 합니다. 문제는 "읽고 → 계산하고 → 쓰는" 이 세 단계가 하나의 원자적(atomic) 동작이 아니라는 점입니다. CPU 명령어 수준에서는 여러 단계로 쪼개져 있고, 그 사이에 다른 스레드가 끼어들 수 있습니다.

2. Race Condition은 정확히 어떤 상황인가

Race condition은 두 개 이상의 실행 흐름이 공유 자원에 동시에 접근할 때, 실행 순서(타이밍)에 따라 결과가 달라지는 상황입니다. "쿠폰 재고 차감"을 예로 들면:

// 재고 차감 로직 (의사코드) 1. stock = read(coupon.stock) // 재고 10 읽음 2. if (stock > 0): 3. write(coupon.stock, stock - 1) // 9로 갱신 4. issue_coupon()

재고가 1개 남은 상황에서 스레드 A와 B가 거의 동시에 1번을 실행하면, 둘 다 stock = 1을 읽습니다. 둘 다 1 > 0이 참이라고 판단해 각자 쿠폰을 발급하고 재고를 0으로 씁니다. 결과적으로 재고 1개인데 쿠폰이 2장 나가거나, 여러 요청이 겹치면 재고가 음수까지 내려갑니다.

흔한 오해
"read-then-write 로직이 짧으니 거의 안 겹칠 것이다"는 오해입니다. 트래픽이 낮으면 확률이 낮을 뿐 버그가 없는 것이 아닙니다. 이벤트/선착순 쿠폰처럼 순간적으로 요청이 몰리는 시점에는 이 좁은 시간 창(window)에 여러 요청이 들어올 확률이 오히려 급격히 높아집니다. "로컬에서 재현이 안 된다"는 것은 버그가 없다는 뜻이 아니라, 동시 요청 타이밍을 만들지 못했다는 뜻입니다.

3. 왜 로컬 테스트에서는 재현이 안 되는가

로컬 단위 테스트는 대개 요청을 순차적으로 보내거나, 동시에 보내더라도 스레드 수가 적고 CPU 코어 경쟁이 약해 두 스레드가 정확히 같은 명령어 사이 지점(critical section)에서 겹칠 확률이 낮습니다. 운영 환경은 다릅니다.

  • 동시 접속자 수가 훨씬 많다 — 같은 타이밍에 read를 실행할 확률(window collision)이 커집니다.
  • context switch가 임의 시점에 끼어든다 — Day 3에서 다룬 것처럼, 타임 슬라이스가 끝나거나 I/O 대기(네트워크 호출, DB 응답 대기)로 전환되는 순간이 critical section 중간일 수 있습니다. 스레드 A가 재고를 읽은 직후 context switch로 B에게 넘어가면, B가 같은 값을 읽고 쓴 뒤 다시 A로 돌아와 stale한 값으로 덮어쓰는 상황이 만들어집니다.
  • 멀티코어에서는 진짜 물리적 동시 실행이 일어난다 — 코어가 여러 개면 두 스레드가 "번갈아"가 아니라 정말 같은 순간에 각자의 코어에서 읽기 연산을 수행할 수 있습니다.

4. 예방 도구: Mutex, Semaphore, Atomic 연산, Condition Variable

Mutex (Lock)
critical section에 "한 번에 하나의 실행 흐름만" 들어가게 하는 상호 배제 도구. 락을 가진 흐름만 진입, 나머지는 대기(block).
Semaphore
카운터 기반으로 "동시에 N개까지" 진입을 허용. mutex는 N=1인 semaphore의 특수 경우로 볼 수 있음. 커넥션 풀 슬롯 제한 등에 사용.
Atomic 연산 (CAS)
Compare-And-Swap처럼 "읽고-비교하고-쓰기"를 하드웨어 수준에서 하나의 명령으로 처리. lock 없이도 단일 변수 갱신을 안전하게 만듦(lock-free).
Condition Variable
특정 조건이 될 때까지 스레드를 재우고, 조건이 충족되면 깨우는 도구. mutex와 함께 써서 "락은 걸되 CPU를 낭비하며 기다리지(busy-wait) 않게" 함.

5. 분산 환경/DB 레벨의 도구도 있다

단일 프로세스 내 lock만으로는 부족한 경우가 많습니다. 서버가 여러 대(수평 확장)라면 메모리 내 mutex는 다른 서버의 스레드를 막지 못합니다. 이때는:

  • DB 원자적 UPDATE + 조건절: UPDATE coupon SET stock = stock - 1 WHERE id = ? AND stock > 0처럼 "읽고 판단하고 쓰기"를 애플리케이션이 아니라 DB 엔진 내부에서 원자적으로 처리하도록 위임합니다. 영향받은 row 수가 0이면 재고 소진으로 판단합니다.
  • 비관적 락(pessimistic lock): SELECT ... FOR UPDATE로 행에 락을 걸어 다른 트랜잭션의 접근을 차단. 경쟁이 심하면 안전하지만 대기 시간과 처리량 저하가 생김.
  • 낙관적 락(optimistic lock): version 컬럼을 두고 갱신 시 WHERE version = 읽은값으로 조건을 걸어, 다른 트랜잭션이 먼저 바꿨으면 실패시키고 재시도. 경쟁이 드물 때 유리(락 대기 비용이 없음).
  • 분산 락(Redis, ZooKeeper 등): 여러 서버에 걸친 critical section을 외부 저장소의 락으로 직렬화. 락 획득/해제/만료(TTL) 관리의 복잡도가 추가로 생김.

왜 면접에서 이 질문을 묻는가

  1. 간헐적 버그를 원리로 진단하는지 확인 — "재현이 안 돼서 모르겠다"에서 멈추지 않고, 타이밍 의존적 버그라는 가설을 세우고 검증할 수 있는지를 봅니다.
  2. 여러 해결책의 트레이드오프를 아는지 확인 — "lock 걸면 끝"이 아니라 mutex, atomic, 낙관적/비관적 락, DB 제약조건 중 트래픽 패턴과 정합성 요구 수준에 맞는 선택을 하는지 봅니다.
  3. 분산 환경 이해도 확인 — 서버가 한 대일 때만 통하는 해법(in-memory mutex)의 한계를 알고, 다중 인스턴스 환경에서는 다른 접근이 필요하다는 것을 아는지 봅니다.

실무 연결

  • 선착순 쿠폰/한정 수량 이벤트: 재고 차감은 race condition의 교과서적 사례입니다. 실무에서는 Redis DECR 같은 원자적 연산이나 메시지 큐로 요청을 직렬화하는 방식을 많이 씁니다.
  • 포인트/잔액 이중 적립·차감: 결제 취소 API가 동시에 두 번 호출되면 환불이 중복 처리될 수 있습니다. 멱등키(idempotency key)와 DB unique 제약조건을 함께 쓰는 것이 표준 패턴입니다.
  • 캐시 스탬피드(cache stampede): 캐시가 만료된 순간 수많은 요청이 동시에 origin(DB)을 두드리는 현상도 일종의 race condition입니다. 락(단일 요청만 DB 조회) 또는 확률적 조기 갱신으로 완화합니다.
  • 싱글턴 초기화(lazy initialization): "커넥션 풀이 두 번 생성된다" 같은 버그는 double-checked locking 패턴이 잘못 구현됐을 때 흔히 발생합니다.
  • 배치 작업의 분산 락: 여러 서버 인스턴스가 동일한 크론잡을 동시에 실행하지 않도록 Redis나 DB 기반 분산 락으로 "이 배치는 한 인스턴스만 실행"을 보장합니다.

답변 사고 순서

  1. 증상 재정의
    "가끔 발생하고 로컬에서 재현이 안 된다"는 특징 자체가 타이밍(동시 실행) 의존적 버그의 전형적 신호임을 인지합니다. 결정론적(deterministic) 버그가 아니라 확률적으로 겹치는 실행 순서가 원인일 가능성을 먼저 세웁니다.
  2. 공유 자원과 critical section 특정
    코드에서 "읽고 → 판단하고 → 쓰는" 구간이 어디인지, 그 구간이 락 없이 실행되는지 확인합니다. 이번 사례라면 재고 조회와 차감 쿼리 사이입니다.
  3. 재현 전략 수립
    부하 테스트 도구(k6, JMeter, 또는 간단한 병렬 스크립트)로 동일 리소스에 대해 수십~수백 개의 동시 요청을 보내 인위적으로 window collision 확률을 높여 재현을 시도합니다. DB 로그나 애플리케이션 로그에서 거의 동시 타임스탬프의 두 요청이 같은 재고 값을 읽었는지 확인합니다.
  4. 단일 서버 vs 다중 서버 여부 확인
    서버가 한 대인지, 오토스케일링으로 여러 인스턴스가 떠 있는지 확인합니다. 이는 in-memory lock으로 충분한지, 외부 동기화(DB/분산 락)가 필요한지를 가르는 핵심 질문입니다.
  5. 해결책 후보 비교
    트래픽 패턴(경쟁이 잦은지), 정합성 요구 수준(절대 초과 발급 불가인지), 지연 허용치를 기준으로 비관적 락, 낙관적 락, DB 원자적 UPDATE, 분산 락 중 선택합니다.
  6. 검증과 재발 방지
    수정 후 동시 요청 부하 테스트로 재현이 사라지는지 확인하고, 유사한 read-modify-write 패턴이 다른 코드에도 있는지 점검(코드 리뷰 체크리스트화)합니다.

핵심 답변 예시

짧은 버전 (30초 답변)

강한 답변 (짧은 버전)

"가끔 발생하고 로컬에서 재현이 안 된다는 특징 자체가 race condition의 전형적 신호입니다. 재고를 읽고 판단하고 쓰는 구간이 원자적이지 않아서, 동시 요청이 같은 재고 값을 읽고 둘 다 발급해버리는 것으로 의심됩니다. 저는 부하 테스트로 동시 요청을 인위적으로 만들어 재현부터 확인하고, 서버가 여러 대라면 DB 원자적 UPDATE(WHERE stock > 0)나 분산 락으로, 단일 서버라면 mutex로 critical section을 직렬화하겠습니다."

깊이 있는 버전 (2분 답변)

강한 답변 (깊이 있는 버전)

"이 버그는 재고를 '읽고 → 0보다 큰지 판단하고 → 차감해서 쓰는' 세 단계가 하나의 원자적 연산이 아니기 때문에 발생하는 race condition으로 보입니다. 재고가 1개 남았을 때 두 요청이 거의 동시에 읽으면 둘 다 '1개 남았다'고 판단해 각자 차감을 진행하고, 결과적으로 초과 발급이 일어납니다.

로컬에서 재현이 안 되는 이유는 로컬 테스트가 이 좁은 시간 창(window)에 두 요청이 겹치는 상황을 잘 만들지 못하기 때문입니다. 운영 환경은 동시 접속자가 많고, 멀티코어에서는 물리적으로 같은 순간에 읽기 연산이 실행될 수 있어 이 확률이 훨씬 높습니다. 그래서 저는 먼저 k6 같은 도구로 동일 쿠폰에 수백 개의 동시 요청을 보내 인위적으로 재현을 시도하고, 애플리케이션/DB 로그에서 같은 재고 값을 읽은 두 요청의 타임스탬프가 겹치는지 확인하겠습니다.

해결책은 서버가 몇 대인지에 따라 달라집니다. 서버가 한 대라면 in-memory mutex로 critical section을 직렬화할 수 있지만, 오토스케일링으로 여러 인스턴스가 떠 있다면 메모리 내 락은 다른 인스턴스의 요청을 막지 못하므로 의미가 없습니다. 이 경우 DB의 원자적 UPDATE, 즉 UPDATE coupon SET stock = stock - 1 WHERE id = ? AND stock > 0처럼 read-modify-write를 DB 엔진에 위임하고 영향받은 row 수로 성공 여부를 판단하는 방식이 안전하고 구현도 단순합니다. 트래픽이 매우 높다면 Redis의 원자적 DECR로 재고를 관리하고 DB는 비동기로 정합성을 맞추는 방식도 고려하겠습니다. 수정 후에는 다시 동시 요청 부하 테스트로 재발 여부를 검증하고, 비슷한 read-modify-write 패턴이 결제 취소나 포인트 적립 같은 다른 API에도 있는지 점검하겠습니다."

좋은 답변과 약한 답변의 차이

약한 답변

"synchronized 키워드나 lock을 걸면 해결될 것 같습니다."

왜 약한가: 결론(락)만 말하고 왜 이 버그가 생기는지(read-modify-write의 비원자성), 어디에 락을 걸어야 하는지, 그리고 서버가 여러 대일 때는 in-memory 락이 무의미하다는 점을 전혀 다루지 않습니다. 실무에서는 이 답만으로는 다중 인스턴스 환경에서 여전히 버그가 재현됩니다.

약한 답변

"재현이 안 되니 일단 로그를 더 추가하고 지켜보겠습니다."

왜 약한가: 간헐적이고 로컬 재현이 안 된다는 특징이 타이밍 버그의 강력한 단서인데, 이를 가설로 연결하지 못하고 수동적으로 관찰만 하겠다는 답변입니다. 시니어라면 동시 요청 부하 테스트로 능동적으로 재현을 시도해야 합니다.

강한 답변의 패턴

① 증상(간헐적, 재현 어려움)에서 race condition 가설 도출 → ② critical section(읽고-판단-쓰기 구간) 특정 → ③ 부하 테스트로 능동적 재현 시도 → ④ 단일/다중 서버 여부로 해결책 범위 결정 → ⑤ 트래픽/정합성 요구에 맞는 구체적 해결책(원자적 UPDATE, 분산 락 등) 제시 → ⑥ 재발 방지 점검까지 포함

Senior-level Follow-up 질문

▶ "비관적 락과 낙관적 락 중 어떤 상황에서 무엇을 선택합니까?"
경쟁(contention)이 잦고 충돌 시 재시도 비용이 크다면 비관적 락(SELECT FOR UPDATE)이 유리합니다. 처음부터 락을 걸어 대기시키므로 충돌로 인한 낭비 작업이 없지만, 락 대기 시간 때문에 처리량이 떨어지고 락 순서를 잘못 관리하면 deadlock 위험도 생깁니다(Day 5에서 다룹니다). 경쟁이 드물고 대부분의 트랜잭션이 충돌 없이 끝난다면 낙관적 락(version 컬럼 비교)이 유리합니다. 락 대기 없이 진행하다가 실제 충돌이 났을 때만 재시도하므로 평균적으로 처리량이 높지만, 경쟁이 심한 환경에서는 재시도가 계속 실패하며 오히려 비효율적일 수 있습니다. 선착순 쿠폰처럼 특정 순간에 경쟁이 극도로 몰리는 경우는 낙관적 락의 재시도 폭증보다 DB 원자적 UPDATE나 Redis 기반 직렬화가 더 낫습니다.
▶ "atomic 연산(CAS)은 왜 lock보다 빠릅니까? 항상 CAS가 더 나은가요?"
CAS(Compare-And-Swap)는 CPU가 제공하는 하드웨어 명령으로, "메모리 값이 기대값과 같으면 새 값으로 교체"를 단일 명령으로 원자적으로 수행합니다. mutex처럼 커널에 진입해 스레드를 재우고 깨우는 비용(system call, context switch)이 없어 경쟁이 적을 때는 훨씬 빠릅니다. 하지만 항상 나은 것은 아닙니다 — 경쟁이 심하면 CAS가 계속 실패하며 재시도하는 스핀(spin)이 반복되어 CPU를 계속 소모합니다(busy-wait). 이 경우 스레드를 재워서 CPU를 양보하는 mutex가 오히려 전체 시스템에는 유리할 수 있습니다. 또한 CAS는 단일 변수 갱신에는 강력하지만, 여러 변수를 함께 원자적으로 바꿔야 하는 복잡한 critical section에는 적용하기 어렵고 이런 경우는 lock이 더 적합합니다.
▶ "race condition을 코드 리뷰나 테스트 단계에서 미리 잡으려면 어떻게 합니까?"
코드 리뷰에서는 "공유 상태(전역 변수, static 필드, DB 행, 캐시 키)를 읽고 판단하고 쓰는 코드가 있는가"를 체크리스트로 삼아 검토합니다. 특히 "read 결과로 조건 분기 후 write"하는 패턴은 항상 의심 대상입니다. 도구 측면에서는 Java의 ThreadSanitizer 계열, Go의 -race 플래그 (race detector), 정적 분석 도구가 동시 접근을 탐지해줍니다. 테스트 단계에서는 동시성 통합 테스트를 작성해 같은 리소스에 대해 수십~수백 개의 goroutine/thread를 동시에 실행시키고 최종 상태(재고, 잔액)의 정합성을 assert하는 방식이 효과적입니다. 다만 이런 테스트도 100% 보장은 아니므로, 프로덕션에서는 모니터링(음수 재고 알람 같은 불변조건 체크)을 함께 두는 것이 안전합니다.
▶ "멱등성(idempotency)은 race condition 예방과 어떤 관계가 있습니까?"
멱등성은 같은 요청이 여러 번 들어와도 결과가 한 번 처리된 것과 같아지는 성질입니다. race condition과는 다른 문제지만 실무에서는 자주 함께 다뤄집니다. 예를 들어 네트워크 재시도(클라이언트가 응답을 못 받아 같은 결제 요청을 두 번 보냄)로 인한 중복 처리는 race condition이 아니라 멱등성 부재가 원인입니다. 하지만 두 문제 모두 "같은 리소스에 대한 중복/동시 처리를 어떻게 하나로 취급할 것인가"라는 공통 뿌리를 가지므로, 멱등키를 DB의 unique 제약조건과 결합하면(예: 요청 ID에 unique 인덱스) race condition 예방과 중복 요청 방지를 동시에 해결하는 경우가 많습니다.
