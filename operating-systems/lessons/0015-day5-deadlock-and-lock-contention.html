Deadlock과 Lock Contention

"요청이 멈춰서 아예 안 끝난다"와 "요청이 느려지긴 하는데 결국은 끝난다"는 겉보기엔 비슷해 보여도 운영체제 관점에서는 완전히 다른 문제입니다. 이 레슨은 deadlock(교착 상태)과 lock contention(락 경합)을 구분해서 진단하고, thread dump로 원인을 특정하고, 재발을 막는 사고 과정을 훈련합니다.

오늘의 면접 질문

Senior Interview Question
"배포 후 특정 API의 응답 시간이 튀다가, 결국 일부 요청이 아예 응답하지 않고 멈추는 현상이 발생했습니다. 운영 중인 서버에서 이 문제가 deadlock인지 단순 lock contention(락 경합)인지 어떻게 구분하시겠습니까? 그리고 각각의 경우 어떻게 대응하시겠습니까?"

이 질문은 "락을 걸었으니 스레드가 멈췄다"는 한 줄 진단으로는 통과되지 않습니다. 면접관은 운영 중인 프로세스에서 실제로 무엇을 확인해서 deadlock과 contention을 구분하는지, thread dump를 읽을 줄 아는지, 그리고 즉시 조치(hotfix)와 근본 원인 조치(설계 변경)를 구분해서 제시하는지를 봅니다.

전제 개념: 이것부터 알고 있어야 합니다

1. Lock Contention(락 경합)이란 무엇인가

여러 스레드가 같은 락을 얻으려고 경쟁하지만, 결국은 순서대로 하나씩 락을 얻고 풀며 진행되는 상황을 lock contention이라고 합니다. Day 4에서 다룬 것처럼 critical section을 보호하려고 락을 걸면, 그 락을 기다리는 스레드는 대기(block) 상태가 됩니다. 락을 쥔 스레드가 빨리 끝내고 풀어주면 문제가 없지만, 락을 쥐고 있는 시간이 길어지거나 락을 요청하는 스레드 수가 많아지면 대기 줄이 길어지고 전체 처리량이 떨어집니다. 중요한 것은 이 상태는 결국 풀린다는 점입니다 — 느려질 뿐, 영원히 멈추지는 않습니다.

2. Deadlock(교착 상태)이란 무엇인가

Deadlock은 두 개 이상의 실행 흐름이 서로 상대방이 가진 자원을 기다리며 영원히 멈춰버리는 상황입니다. Contention과 결정적으로 다른 점은, 아무리 기다려도 스스로 풀리지 않는다는 것입니다. 고전적인 예시:

// 스레드 A lock(mutex_1) ... 잠깐 대기 ... lock(mutex_2) // B가 이미 갖고 있음 -> A는 여기서 영원히 대기 // 스레드 B (거의 동시에 실행) lock(mutex_2) ... 잠깐 대기 ... lock(mutex_1) // A가 이미 갖고 있음 -> B는 여기서 영원히 대기

A는 mutex_1을 쥔 채 mutex_2를 기다리고, B는 mutex_2를 쥔 채 mutex_1을 기다립니다. 둘 다 상대가 자원을 놓아주기를 기다리지만, 둘 다 이미 대기 상태라 아무도 놓아줄 수 없습니다. 이 상황은 외부 개입(강제 종료, timeout) 없이는 영원히 지속됩니다.

3. Deadlock이 성립하는 4가지 필요조건 (Coffman conditions)

Mutual Exclusion
자원(락)을 한 번에 하나의 스레드만 가질 수 있음. 애초에 배타적 자원이 아니면 deadlock 자체가 성립하지 않음.
Hold and Wait
이미 자원을 하나 쥔 채로, 다른 자원을 추가로 기다림. 위 예시에서 A가 mutex_1을 쥔 채 mutex_2를 기다리는 상태.
No Preemption
다른 스레드가 강제로 락을 빼앗을 수 없음. 락을 쥔 스레드가 스스로 풀어주기 전까지는 아무도 가져갈 수 없음.
Circular Wait
A→B→...→A처럼 대기 관계가 원형으로 이어짐. 이 4가지가 모두 성립해야 deadlock이 발생하며, 하나만 깨도 예방 가능.
흔한 오해
"락을 여러 개 쓰면 deadlock 위험이 있으니 락은 하나만 써야 한다"는 오해입니다. 락이 여러 개라서 위험한 게 아니라, 서로 다른 코드 경로에서 같은 락들을 다른 순서로 잠그기 때문에 위험한 것입니다. 락 획득 순서를 항상 동일하게 고정하면(lock ordering) 여러 락을 써도 circular wait 자체가 발생하지 않습니다.

4. Thread Dump로 무엇을 볼 수 있는가

운영 중인 프로세스에서 특정 시점의 모든 스레드가 지금 어떤 상태이고 어떤 코드 라인(스택)에서 멈춰 있는지를 스냅샷으로 찍은 것이 thread dump입니다(JVM은 jstack, Go는 pprof goroutine, 리눅스 프로세스 일반은 /proc/[pid]/task/*/stack 등). Deadlock 진단의 핵심은 이 덤프에서 "BLOCKED" 상태인 스레드들이 서로 상대가 쥔 락을 기다리는 원형 관계를 찾는 것입니다. JVM의 경우 jstack은 deadlock을 발견하면 Found one Java-level deadlock이라는 메시지와 함께 관련 스레드 스택을 직접 알려주기도 합니다.

5. Deadlock vs Contention을 구분하는 결정적 신호

Lock Contention
thread dump를 시간차를 두고 2~3번 떠보면, 대기 중인 스레드 목록이 계속 바뀐다. 어떤 스레드는 락을 얻고 빠져나가고, 새로운 스레드가 대기열에 들어온다. CPU 사용률은 낮을 수도 정상일 수도 있음.
Deadlock
시간차를 두고 덤프를 떠도 같은 스레드들이 계속 같은 지점에서 BLOCKED 상태로 정지해 있다. 진행이 전혀 없다. 해당 요청들의 latency는 거의 무한대로 수렴(timeout까지 응답 없음).

왜 면접에서 이 질문을 묻는가

  1. 운영 중인 프로세스를 직접 진단할 수 있는지 확인 — 이론으로 deadlock의 정의를 아는 것과, 실제로 thread dump를 떠서 원인 스레드를 짚어내는 것은 완전히 다른 역량입니다.
  2. 증상의 결(texture)을 구분하는지 확인 — "느려짐"과 "완전히 멈춤"을 같은 문제로 뭉뚱그리지 않고, 시간에 따른 변화 여부로 정확히 분류하는지를 봅니다.
  3. 즉시 조치와 근본 대책을 구분하는지 확인 — 장애 중에는 무엇을 하고(재시작, 특정 스레드 강제 종료), 재발 방지를 위해 무엇을 바꿔야 하는지(lock ordering, timeout 도입)를 나눠서 답하는지 봅니다.

실무 연결

  • DB 커넥션 풀 고갈과 유사 증상: 커넥션 풀이 부족해서 요청이 대기하는 것도 겉보기엔 "멈춘 것"처럼 보이지만, 이는 deadlock이 아니라 자원 고갈(pool exhaustion)입니다. 진단 시 thread dump와 함께 DB 커넥션 풀 지표(활성/대기 커넥션 수)를 같이 봐야 혼동하지 않습니다.
  • 여러 테이블을 다른 순서로 갱신하는 트랜잭션: 주문 처리 트랜잭션이 "재고→포인트" 순서로, 환불 트랜잭션이 "포인트→재고" 순서로 행 락을 건다면 DB 레벨 deadlock이 발생할 수 있습니다. 대부분의 RDBMS는 이를 자동 탐지해 한쪽 트랜잭션을 강제 롤백(deadlock victim)시킵니다.
  • 서비스 간 동기 호출의 순환 대기: 서비스 A가 B를 호출한 응답을 기다리는 스레드 풀을 다 써버렸는데, B가 다시 A를 호출하며 A의 스레드 풀이 이미 고갈된 상태라면 논리적으로 deadlock과 유사한 정체가 발생합니다. 이런 경우는 timeout과 circuit breaker로 완화합니다.
  • 배치 작업의 분산 락 미해제: 크론잡이 분산 락을 잡은 채 예외로 죽으면서 락을 반환하지 못하면, 다음 배치 실행이 영원히 락을 기다리는 상황이 생깁니다. 이는 애플리케이션 레벨 deadlock으로, 락에 TTL(자동 만료)을 걸어 예방합니다.
  • 장기 실행 락 경합으로 인한 캐스케이딩 타임아웃: 특정 API의 락 보유 시간이 길어지면 대기 스레드가 쌓이고, 스레드 풀이 고갈되면서 다른 무관한 API까지 응답 불가 상태가 되는 "장애 전파(cascading failure)"로 이어질 수 있습니다.

답변 사고 순서

  1. 증상의 시간적 패턴 확인
    영향받은 요청들이 "점점 느려지다가 결국 응답한다"인지 "특정 시점부터 아예 응답이 없다"인지를 APM/로그로 먼저 구분합니다. 후자일수록 deadlock 가능성이 높습니다.
  2. Thread Dump 확보(가능하면 시간차로 2회 이상)
    운영 중인 프로세스에서 즉시 thread dump를 뜨고, 몇 초~수십 초 후 한 번 더 뜹니다. 같은 스레드들이 같은 스택에 멈춰 있으면 deadlock, 대기열 구성이 바뀌면 contention입니다.
  3. BLOCKED 스레드의 대기 대상 추적
    BLOCKED 상태인 각 스레드가 "어떤 락을 기다리는지"와 "그 락을 지금 누가 쥐고 있는지"를 덤프에서 짚어 circular wait 관계(A는 B가 쥔 락을 기다리고, B는 A가 쥔 락을 기다림)를 확인합니다.
  4. 코드 경로에서 락 획득 순서 비교
    deadlock으로 확인되면, 관련된 두 코드 경로(예: 주문 처리 vs 환불 처리)가 같은 락들을 서로 다른 순서로 획득하는지 코드를 확인합니다. 이것이 대부분의 애플리케이션 레벨 deadlock의 근본 원인입니다.
  5. 즉시 조치와 근본 조치 분리
    장애 중에는 멈춘 스레드/프로세스를 재시작해 서비스를 우선 복구합니다. 이후 근본 대책으로 lock ordering 통일, 락 획득에 timeout 도입(tryLock), 락 보유 범위 최소화 중 적절한 방법을 선택합니다.
  6. 재발 방지와 조기 탐지 체계 구축
    락 대기 시간에 대한 임계치 알람, 정기적인 deadlock 자동 탐지 로그(DB는 대부분 기본 제공), 코드 리뷰 시 "여러 락을 잡는 코드는 획득 순서를 명시"하는 체크리스트를 추가합니다.

핵심 답변 예시

짧은 버전 (30초 답변)

강한 답변 (짧은 버전)

"먼저 영향받은 요청이 점점 느려지다 끝나는지, 아예 멈춰서 안 끝나는지를 구분하겠습니다. 후자라면 시간차를 두고 thread dump를 두 번 떠서 같은 스레드가 같은 지점에서 계속 BLOCKED 상태인지 확인하겠습니다. 같다면 deadlock이므로, 대기 관계를 따라가 circular wait를 찾고 관련 코드 경로가 락을 서로 다른 순서로 획득하는지 확인하겠습니다. 장애 중에는 프로세스 재시작으로 우선 복구하고, 이후 lock ordering을 통일하거나 락 획득에 timeout을 두는 방식으로 근본 원인을 고치겠습니다. 단순 contention이라면 재시작 없이도 시간이 지나면 풀리므로, 락 보유 시간을 줄이는 방향으로 접근하겠습니다."

깊이 있는 버전 (2분 답변)

강한 답변 (깊이 있는 버전)

"가장 먼저 할 일은 이 현상이 deadlock인지 단순 lock contention인지를 시간에 따른 변화로 구분하는 것입니다. Contention은 스레드들이 락을 기다리다가도 결국 순서대로 얻고 빠져나가므로 요청이 느릴 뿐 끝은 납니다. 반면 deadlock은 관련된 스레드들이 서로 상대가 쥔 자원을 기다리며 영원히 멈추므로, 해당 요청은 timeout이 걸릴 때까지 응답이 아예 없습니다. 지금 상황처럼 '결국 응답하지 않고 멈추는' 요청이 있다는 것은 deadlock 가능성을 강하게 시사합니다.

확인을 위해 운영 중인 프로세스에서 thread dump를 즉시 뜨고, 수십 초 뒤 한 번 더 뜹니다. 두 덤프에서 같은 스레드들이 같은 스택 지점에서 BLOCKED 상태로 정지해 있다면 deadlock으로 확진합니다. 이후 각 BLOCKED 스레드가 어떤 락 객체를 기다리는지, 그 락을 지금 누가 쥐고 있는지를 추적해 A는 B가 쥔 락을, B는 A가 쥔 락을 기다리는 순환 관계(circular wait)를 찾습니다. 그 다음 실제 코드에서 두 경로가 같은 락들을 서로 다른 순서로 획득하고 있는지 확인하는데, 이것이 애플리케이션 레벨 deadlock의 가장 흔한 원인입니다.

장애 대응은 두 단계로 나눕니다. 즉시는 서비스 복구가 우선이므로 멈춘 프로세스나 스레드를 재시작해 순환 대기를 강제로 끊습니다. 근본 원인 조치로는 모든 코드 경로가 여러 락을 항상 동일한 순서로 획득하도록 통일하거나(lock ordering), 락 획득에 무한 대기 대신 timeout을 둬서(tryLock) 실패 시 재시도나 롤백을 하도록 바꾸는 방법을 고려합니다. 만약 이게 단순 contention이었다면 재시작 없이도 자연히 풀리므로, 대신 락을 쥐고 있는 시간을 줄이거나(critical section 축소) 락을 잘게 쪼개는 방향으로 접근했을 것입니다. 마지막으로 재발 방지를 위해 락 대기 시간에 임계치 알람을 걸고, DB 레벨이라면 deadlock 발생 로그를 모니터링에 연동해 조기에 감지하도록 하겠습니다."

좋은 답변과 약한 답변의 차이

약한 답변

"deadlock인 것 같으니 서버를 재시작하겠습니다."

왜 약한가: 재시작은 증상 완화일 뿐 진단이 아닙니다. deadlock인지 contention인지 확인하는 절차(thread dump)를 전혀 언급하지 않았고, 근본 원인(락 획득 순서)에 대한 조치도 없어 같은 문제가 곧 재발합니다.

약한 답변

"락을 다 없애고 atomic 연산으로 바꾸겠습니다."

왜 약한가: 진단도 하기 전에 결론(락 제거)으로 건너뜁니다. 여러 자원에 걸친 복잡한 critical section은 atomic 연산 하나로 대체할 수 없는 경우가 많고, 무엇보다 지금 상황이 deadlock인지조차 확인하지 않았습니다.

강한 답변의 패턴

① 증상의 시간적 패턴(느려짐 vs 완전 정지)으로 deadlock/contention 가설 분리 → ② thread dump를 시간차로 확보해 진행 여부 확인 → ③ BLOCKED 스레드의 순환 대기 관계 추적 → ④ 코드에서 락 획득 순서 불일치 확인 → ⑤ 즉시 조치(재시작)와 근본 조치(lock ordering, timeout) 분리 → ⑥ 조기 탐지 체계(알람, 모니터링) 구축까지 포함

Senior-level Follow-up 질문

▶ "Deadlock을 예방하는 방법에는 어떤 것들이 있고, 각각의 트레이드오프는 무엇입니까?"
가장 실용적인 방법은 Lock Ordering(락 획득 순서 고정)입니다. 모든 코드 경로가 여러 락을 항상 같은 전역 순서(예: ID가 작은 자원부터)로 획득하도록 강제하면 circular wait 자체가 생기지 않습니다. 구현이 간단하지만, 코드베이스 전체에 걸쳐 규율을 지켜야 하고 락 종류가 늘어날수록 순서 관리가 번거로워집니다. 두 번째는 Lock Timeout(tryLock)입니다 — 무한정 기다리는 대신 일정 시간 후 실패 처리하고 재시도하거나 롤백합니다. 무한 대기를 원천 차단하지만, 재시도 로직과 실패 처리 복잡도가 늘어납니다. 세 번째는 Deadlock Detection(사후 탐지)으로, OS나 DB가 주기적으로 대기 그래프를 그려 순환을 발견하면 강제로 하나를 희생시킵니다(DB의 deadlock victim이 대표적). 구현 부담은 없지만(대부분 DB가 기본 제공) 이미 한 트랜잭션이 롤백되는 비용을 감수해야 합니다. 마지막으로 자원을 아예 하나만 쓰도록 설계 단순화(단일 락, 단일 자원 접근 경로)도 가장 근본적인 예방이지만 병렬성을 희생합니다.
▶ "Livelock은 deadlock과 무엇이 다릅니까?"
Deadlock은 스레드들이 완전히 정지(BLOCKED)한 채 아무 진행이 없는 상태입니다. Livelock은 스레드들이 계속 실행 중(RUNNABLE)이지만 실질적인 진행이 없는 상태입니다. 예를 들어 두 스레드가 서로에게 양보하려고 계속 자신의 락을 풀었다가 다시 시도하기를 반복하면, CPU는 계속 소모되지만 어느 쪽도 끝까지 진행하지 못합니다. thread dump로 보면 deadlock은 스레드가 BLOCKED/WAITING 상태로 고정돼 있고, livelock은 RUNNABLE 상태를 유지하며 스택이 계속 바뀌지만 같은 패턴을 반복합니다. Livelock은 그래서 CPU 사용률이 오히려 비정상적으로 높게 나타나는 경우가 많다는 점이 진단의 단서가 됩니다.
▶ "락 경합이 심한 코드를 발견했을 때, lock을 없애지 않고도 경합을 줄이는 방법은?"
가장 먼저 critical section의 범위를 최소화락 분할(lock striping/sharding)입니다 — 하나의 큰 락 대신 자원을 여러 구간으로 나누고 구간별로 다른 락을 사용해(예: 해시 버킷별 락) 서로 다른 구간에 대한 접근은 동시에 진행되게 합니다(Java의 ConcurrentHashMap이 이 방식을 씁니다). 세 번째는 읽기가 압도적으로 많은 워크로드라면 Read-Write Lock으로 바꿔 읽기끼리는 동시 진입을 허용합니다. 네 번째는 상황에 따라 lock-free 자료구조(CAS 기반)로 전환해 커널 개입 없이 경합을 처리하는 방법도 있지만, 구현 복잡도와 ABA 문제 같은 새로운 위험이 따릅니다.
▶ "DB 레벨 deadlock과 애플리케이션(in-process) deadlock은 대응 방식이 어떻게 다릅니까?"
DB 레벨 deadlock은 대부분의 RDBMS(MySQL InnoDB, PostgreSQL 등)가 내부적으로 대기 그래프를 감시하다가 순환을 감지하면 자동으로 한쪽 트랜잭션을 강제 롤백(deadlock victim)시키고 에러를 반환합니다. 따라서 애플리케이션은 이 에러를 잡아서 재시도 로직을 두는 것이 표준 대응입니다 — 즉 탐지와 해소는 DB가 해주고, 애플리케이션은 실패를 우아하게 처리하기만 하면 됩니다. 반면 애플리케이션 레벨(in-process) deadlock은 언어/런타임이 자동으로 풀어주지 않는 경우가 대부분이라(JVM의 jstack처럼 탐지 도구는 있어도 자동 해소는 안 함), 개발자가 lock ordering이나 timeout을 직접 설계에 반영해 예방해야 하고, 발생 시에도 수동 개입(재시작)이 필요한 경우가 많습니다. 이 차이 때문에 저는 가능하면 read-modify-write 로직을 DB 트랜잭션에 위임해서 DB의 자동 deadlock 처리 능력을 활용하는 것을 선호합니다.
