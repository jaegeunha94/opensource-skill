1. CPU 스케줄링이란?

CPU는 한 번에 하나의 프로세스만 실행할 수 있습니다 (단일 코어 기준). 그런데 OS에는 수십 개의 프로세스가 동시에 존재합니다. CPU 스케줄러(단기 스케줄러)는 준비 큐(Ready Queue)에 있는 프로세스 중 다음에 CPU를 받을 프로세스를 결정합니다.

핵심 개념
CPU 스케줄링 = "준비 상태의 여러 프로세스 중 누구에게 CPU를 줄 것인가?"를 결정하는 OS의 핵심 기능

스케줄링이 발생하는 시점

  • 프로세스가 실행(Running) → 대기(Waiting) 상태로 전환될 때 (I/O 요청)
  • 프로세스가 실행(Running) → 준비(Ready) 상태로 전환될 때 (타임 슬라이스 만료, 인터럽트)
  • 프로세스가 대기(Waiting) → 준비(Ready) 상태로 전환될 때 (I/O 완료)
  • 프로세스가 종료될 때
선점형 vs 비선점형
비선점형(Non-preemptive): CPU를 한 번 받으면 자발적으로 놓을 때(I/O, 종료)까지 계속 실행.
선점형(Preemptive): OS가 강제로 CPU를 빼앗아 다른 프로세스에게 줄 수 있음. 현대 OS(Linux, Windows)는 모두 선점형을 사용합니다.

2. 성능 지표 (Scheduling Criteria)

스케줄링 알고리즘의 품질을 측정하는 지표입니다.

지표 정의 목표
CPU 이용률 (Utilization) CPU가 실제로 작업에 사용된 비율 최대화
처리량 (Throughput) 단위 시간당 완료된 프로세스 수 최대화
반환 시간 (Turnaround Time) 프로세스 제출 ~ 완료까지 걸린 총 시간 최소화
대기 시간 (Waiting Time) 준비 큐에서 기다린 총 시간 최소화
응답 시간 (Response Time) 요청 제출 ~ 첫 번째 응답까지 걸린 시간 최소화
공식 기억법
반환 시간 = 완료 시간 − 도착 시간
대기 시간 = 반환 시간 − 실행 시간 (CPU 버스트 시간)
응답 시간 = 첫 번째 CPU 할당 시작 − 도착 시간

3. FCFS — First-Come, First-Served

도착 순서대로 CPU를 할당하는 가장 단순한 알고리즘입니다. 비선점형이며, FIFO 큐로 구현합니다.

예시

프로세스도착 시간CPU 버스트
P1024
P213
P323
간트 차트 (Gantt Chart) ┌──────────────────────────────┬─────┬─────┐ │ P1 │ P2 │ P3 │ └──────────────────────────────┴─────┴─────┘ 0 24 27 30 대기 시간: P1=0, P2=23, P3=25 평균 대기 시간 = (0 + 23 + 25) / 3 = 16ms

그림 1: FCFS 스케줄링 — P1이 오래 걸려 P2, P3가 오래 기다림

호위 효과 (Convoy Effect)
CPU 버스트가 긴 프로세스 뒤에 짧은 프로세스들이 줄지어 기다리는 현상. 전체 평균 대기 시간이 크게 늘어납니다. FCFS의 가장 큰 단점입니다.

4. SJF — Shortest-Job-First

CPU 버스트가 가장 짧은 프로세스를 먼저 실행합니다. 이론적으로 평균 대기 시간을 최소화하는 최적 알고리즘입니다.

예시 (비선점형 SJF)

프로세스도착 시간CPU 버스트
P106
P228
P347
P453
시간 0: 준비 큐 = [P1(6)] → P1 실행 시간 6: 준비 큐 = [P2(8), P3(7), P4(3)] → 가장 짧은 P4 실행 시간 9: 준비 큐 = [P2(8), P3(7)] → P3 실행 시간 16: 준비 큐 = [P2(8)] → P2 실행 ┌──────┬───┬───────┬────────┐ │ P1 │P4 │ P3 │ P2 │ └──────┴───┴───────┴────────┘ 0 6 9 16 24 대기 시간: P1=0, P2=16, P3=9, P4=4 평균 대기 시간 = (0 + 16 + 9 + 4) / 4 = 7.25ms (FCFS보다 훨씬 짧음!)

그림 2: 비선점형 SJF — 짧은 작업을 우선하여 평균 대기 시간 감소

SJF의 한계 — 기아 문제 (Starvation)
CPU 버스트가 짧은 프로세스가 계속 들어오면, 버스트가 긴 프로세스는 영원히 CPU를 받지 못할 수 있습니다.
해결책: 에이징(Aging) — 오래 기다린 프로세스의 우선순위를 점진적으로 높여줌.
또한, 실제로는 다음 CPU 버스트 시간을 미리 알 수 없어 지수 평균 예측을 사용합니다.

선점형 SJF — SRTF (Shortest Remaining Time First)

새 프로세스가 도착할 때마다 남은 시간을 비교해 더 짧은 프로세스가 있으면 선점합니다. 평균 대기 시간은 더 줄지만 컨텍스트 스위치 오버헤드가 증가합니다.

5. Round-Robin (RR)

각 프로세스에 일정한 타임 퀀텀(Time Quantum, q)만큼 CPU를 순서대로 할당합니다. 타임 퀀텀이 끝나면 현재 프로세스를 준비 큐 맨 뒤로 보내고 다음 프로세스를 실행합니다. 선점형이며, 현대 OS의 기본 스케줄링 방식입니다.

예시 (q = 4ms)

프로세스도착 시간CPU 버스트
P1024
P203
P303
┌────┬───┬───┬────┬────┬────┬────┐ │ P1 │P2 │P3 │ P1 │ P1 │ P1 │ P1 │ └────┴───┴───┴────┴────┴────┴────┘ 0 4 7 10 14 18 22 26 (단, P2·P3는 3ms 후 완료) 대기 시간: P1 = (4+4+4+4+4) = 20 (남은 시간이 있어 다시 큐로) P2 = 4ms, P3 = 7ms 평균 대기 시간 = (20 + 4 + 7) / 3 ≈ 10.3ms

그림 3: Round-Robin (q=4) — 응답성이 좋지만 컨텍스트 스위치 비용 발생

타임 퀀텀 크기의 영향

타임 퀀텀특성
매우 작음 (→ 0) 프로세서 공유처럼 느껴지지만, 컨텍스트 스위치 오버헤드가 폭증
매우 큼 (→ ∞) 사실상 FCFS와 동일해져 응답성 저하
적정 크기 (일반적으로 10~100ms) 응답성과 효율성의 균형. 컨텍스트 스위치는 전체 시간의 10% 미만 권장

6. 우선순위 스케줄링 (Priority Scheduling)

각 프로세스에 우선순위 번호를 부여하고, 가장 높은 우선순위를 가진 프로세스를 먼저 실행합니다. (관례상 낮은 숫자 = 높은 우선순위인 경우가 많음) 선점형 또는 비선점형으로 구현할 수 있습니다.

프로세스CPU 버스트우선순위
P1103
P211 (가장 높음)
P324
P415 (가장 낮음)
P552
실행 순서 (비선점형): P2 → P5 → P1 → P3 → P4 ┌───┬─────┬──────────┬────┬───┐ │P2 │ P5 │ P1 │ P3 │P4 │ └───┴─────┴──────────┴────┴───┘ 0 1 6 16 18 19 평균 대기 시간 = (6 + 0 + 16 + 18 + 1) / 5 = 8.2ms

그림 4: 우선순위 스케줄링 — 우선순위 높은 순으로 실행

기아 문제 (Starvation) & 에이징 (Aging)
우선순위가 낮은 프로세스는 높은 우선순위 프로세스가 계속 들어오면 영원히 실행되지 못합니다.
에이징: 대기 시간이 길어질수록 우선순위를 점진적으로 높여 기아를 방지합니다. 예: 15분마다 우선순위를 1씩 증가. Linux CFS는 가상 런타임(vruntime)으로 이 문제를 해결합니다.

7. 알고리즘 비교 요약

알고리즘 선점형 평균 대기 응답성 주요 문제
FCFS 비선점 나쁨 나쁨 호위 효과
SJF (비선점) 비선점 최소 (이론) 보통 기아, 버스트 시간 예측 불가
SRTF (선점 SJF) 선점 최소 (이론) 좋음 기아, 컨텍스트 스위치 오버헤드
Round-Robin 선점 보통 매우 좋음 타임 퀀텀 크기 결정 어려움
우선순위 선점/비선점 보통 보통 기아 (에이징으로 해결)
실제 OS는 다단계 큐를 사용
실제 OS는 단일 알고리즘이 아닌 다단계 피드백 큐(Multilevel Feedback Queue)를 사용합니다. 예: CPU 버스트가 짧은 프로세스는 높은 우선순위 큐로, 긴 프로세스는 낮은 우선순위 큐로 내려보냅니다. Linux의 CFS(Completely Fair Scheduler)는 Red-Black Tree와 가상 런타임으로 공정성을 보장합니다.
