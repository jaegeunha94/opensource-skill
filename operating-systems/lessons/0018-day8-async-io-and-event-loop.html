비동기 I/O와 이벤트 루프

"스레드를 늘렸는데 왜 처리량이 더 안 나오지?"와 "요청 하나에 스레드 하나씩 쓰면 왜 만 개 연결에서 무너지지?"는 같은 뿌리를 가진 질문입니다. 이 레슨은 블로킹 I/O가 스레드를 어떻게 점유하는지부터 시작해서, non-blocking I/O, I/O multiplexing(select/poll/epoll), 이벤트 루프까지 하나의 인과 사슬로 연결해서 "thread-per-request와 event-driven 중 무엇을 선택할지"를 근거 있게 답하는 사고 과정을 훈련합니다.

오늘의 면접 질문

Senior Interview Question
"동시 접속 1만 개를 처리해야 하는 API 서버를 설계한다고 합시다. thread-per-request 모델과 event-driven(비동기 I/O) 모델 중 어떤 것을 선택하시겠습니까? 운영체제 관점에서 각각의 한계와, 그 한계가 실제로 어떤 지표로 드러나는지 설명해 주세요."

이 질문은 "비동기가 항상 더 빠르다"는 유행어 수준의 답을 걸러내려는 목적을 가집니다. 면접관은 왜 스레드 1만 개가 스케줄러와 메모리에 부담을 주는지, 그리고 event-driven 모델이 그 부담을 어떻게 없애는 대신 무엇을 대가로 치르는지를 OS 메커니즘 수준에서 설명할 수 있는지를 확인합니다.

전제 개념: 이것부터 알고 있어야 합니다

1. Blocking I/O — 커널이 데이터를 줄 때까지 스레드를 재운다

일반적인 read()/write() 소켓 호출은 기본적으로 blocking입니다. 네트워크 소켓에서 아직 도착하지 않은 데이터를 read()로 요청하면, 커널은 그 스레드를 대기 상태(Day 1에서 다룬 D 또는 S 상태)로 재우고, 데이터가 도착하면 다시 실행 큐에 올려 깨웁니다. 이 자체는 CPU를 낭비하지 않는 정상적인 대기입니다. 문제는 이 대기를 누가 감당하느냐입니다 — thread-per-request 모델은 요청 하나마다 스레드 하나를 배정해서, 그 스레드가 I/O를 기다리는 동안 통째로 블로킹되어도 다른 요청 처리에 지장이 없게 만듭니다. 하지만 스레드 자체가 비싼 자원이라는 게 다음 문제로 이어집니다.

2. 왜 스레드를 무한정 늘릴 수 없는가 (Day 2, Day 3의 재등장)

Day 2에서 다룬 것처럼 스레드마다 커널 스택(보통 수백 KB~수 MB)이 할당되고, Day 3에서 다룬 것처럼 스레드 수가 CPU 코어 수를 초과하면 context switch 비용이 늘어납니다. 동시 연결이 1만 개인데 요청 대부분이 I/O 대기 상태(느린 DB 쿼리, 외부 API 호출)라면, 실제로 동시에 CPU를 쓰는 스레드는 소수인데도 스레드 1만 개가 메모리를 점유하고, 스케줄러가 그 중 실행 가능한 소수를 골라내는 데 드는 오버헤드가 누적됩니다. 이것이 "C10K 문제"라고 불리는 고전적인 한계입니다 — 연결 수가 늘어날수록 thread-per-request 모델의 자원 사용량이 거의 선형으로 증가합니다.

3. Non-blocking I/O — "당장 없으면 없다고 알려주고 바로 반환"

소켓을 O_NONBLOCK으로 설정하면, read()는 데이터가 없을 때 스레드를 재우는 대신 즉시 EAGAIN/EWOULDBLOCK 에러를 반환합니다. 이 자체만으로는 스레드가 계속 read()를 반복 호출하는 busy-wait가 되어 오히려 CPU만 낭비합니다. 그래서 "지금 어떤 소켓에 데이터가 준비됐는지"를 커널에게 물어보는 별도의 메커니즘이 필요한데, 이것이 I/O multiplexing입니다.

4. I/O Multiplexing — select/poll/epoll이 하는 일

select()와 poll()은 "이 파일 디스크립터 목록 중 읽기/쓰기 준비가 된 게 있으면 알려달라"고 커널에 요청하고, 준비될 때까지 스레드 하나만 블로킹됩니다. 즉 스레드 1만 개 대신 스레드 1개(또는 소수)가 1만 개의 연결을 동시에 감시할 수 있습니다. 다만 select/poll은 매번 감시 대상 목록 전체를 커널에 복사하고 커널이 그 전체를 순회(O(n))하기 때문에, 연결 수가 커지면 이 조회 자체가 비용이 됩니다. epoll(Linux)은 감시 목록을 커널 내부에 유지시켜 두고(epoll_ctl로 등록/해제), 실제로 이벤트가 발생한 fd만 반환하는 방식(epoll_wait)으로 이 오버헤드를 없애 대규모 연결에서도 효율적으로 동작합니다. kqueue(BSD/macOS), IOCP(Windows)도 같은 문제를 해결하는 각 OS의 방식입니다.

흔한 오해
"비동기 I/O는 스레드를 아예 안 쓴다"는 틀린 말입니다. epoll 기반 이벤트 루프도 결국 하나 이상의 OS 스레드 위에서 돌아갑니다. 차이는 "연결 수만큼 스레드를 쓰느냐, 소수의 스레드가 다수의 연결을 멀티플렉싱하느냐"입니다. 또한 "이벤트 루프는 항상 스레드 하나로 충분하다"도 틀린 말입니다 — CPU-bound 작업이 섞이면 이벤트 루프 스레드 자체가 막혀 다른 모든 연결의 이벤트 처리가 지연되므로, 실무에서는 이벤트 루프를 여러 개 두거나(Nginx의 워커 프로세스) CPU-bound 작업을 별도 스레드 풀로 위임하는 하이브리드 구조를 씁니다.

5. 이벤트 루프 — epoll 결과를 콜백/코루틴 재개로 연결하는 런타임

Node.js, Nginx, Redis 같은 event-driven 시스템은 이벤트 루프라는 반복 구조로 동작합니다: epoll_wait로 준비된 fd 목록을 받아온다 → 각 fd에 연결된 콜백(또는 정지된 코루틴)을 실행한다 → 그 콜백이 다시 I/O를 요청하면 등록만 해두고 즉시 루프의 다음 반복으로 넘어간다. 이 구조의 핵심 전제는 각 콜백이 절대 오래 블로킹하거나 CPU를 오래 점유해서는 안 된다는 것입니다 — 콜백 하나가 100ms짜리 동기 연산을 돌리면, 같은 루프를 공유하는 다른 모든 연결의 이벤트 처리가 그 100ms 동안 전부 지연됩니다.

thread-per-request
연결마다 스레드 1개. 코드가 직관적(동기 스타일)이고 디버깅(스택 트레이스)이 쉬움. 연결 수 증가 시 메모리·context switch 비용이 선형 증가. CPU-bound 작업에 유리.
event-driven (epoll + loop)
소수 스레드가 다수 연결을 멀티플렉싱. I/O-bound·대량 연결에 유리(메모리·context switch 절약). 콜백 하나가 루프를 블로킹하면 전체 지연. 콜백 지옥/비동기 스택 트레이스로 디버깅 난이도 상승.

왜 면접에서 이 질문을 묻는가

  1. "비동기가 항상 빠르다"는 유행어를 걷어내고 원리로 설명하는지 확인 — 스레드 자원 비용(메모리, context switch)과 epoll의 동작 방식을 근거로 언제 event-driven이 실제로 이득인지 설명할 수 있는지 봅니다.
  2. 워크로드 특성 구분 능력 확인 — I/O-bound(대기가 대부분)와 CPU-bound(연산이 대부분) 워크로드를 구분하고, 그에 따라 모델 선택이 달라져야 한다는 것을 아는지 봅니다.
  3. 이벤트 루프 블로킹이라는 실전 장애 패턴을 아는지 확인 — 콜백 하나가 루프를 막으면 전체 서비스가 멈추는 event-driven 모델 특유의 장애 형태를 경험적으로 이해하는지 봅니다.

실무 연결

  • Nginx vs Apache prefork: Nginx는 소수의 워커 프로세스가 각각 epoll 기반 이벤트 루프를 돌며 수만 개 연결을 처리합니다. Apache의 prefork MPM은 연결(또는 요청)마다 프로세스/스레드를 배정해 동시 연결이 늘어날수록 메모리 사용량이 급증합니다. 정적 파일 서빙, 리버스 프록시처럼 I/O 대기가 대부분인 워크로드에서 Nginx 구조가 유리한 이유입니다.
  • Node.js의 단일 이벤트 루프와 CPU-bound 작업: Node.js는 기본적으로 이벤트 루프가 하나이므로, 이미지 리사이징이나 대량 JSON 파싱 같은 CPU-bound 작업을 메인 루프에서 동기로 실행하면 그동안 다른 모든 요청이 멈춥니다. 이런 작업은 worker_threads나 별도 프로세스로 분리해야 합니다.
  • DB 커넥션 풀과 thread-per-request의 상호작용: thread-per-request 서버 (예: 전통적인 Java Servlet 컨테이너)에서 DB 쿼리가 느려지면, 그 쿼리를 기다리는 스레드가 풀에서 반환되지 않아 스레드 풀이 고갈되고, 이는 다른 무관한 요청까지 대기시킵니다. Day 5에서 다룬 lock contention과 유사하게, "자원 고갈"이 연쇄적으로 전체 서비스 latency에 영향을 줍니다.
  • Reactor 패턴과 비동기 프레임워크: Netty(Java), asyncio(Python), Tokio(Rust)는 모두 OS의 epoll/kqueue를 감싼 Reactor 패턴 위에서 동작합니다. 이런 프레임워크에서 "블로킹 함수를 async 컨텍스트에서 실수로 호출"하는 것이 실무에서 가장 흔한 성능 버그입니다 — 이벤트 루프 스레드 자체가 블로킹되어 전체 처리량이 무너집니다.
  • 커넥션 수 대비 스레드 수 모니터링: 운영 중인 서버의 스레드 수가 동시 연결 수에 비례해 계속 늘어나는지, 아니면 소수로 고정되어 있는지를 관찰하는 것만으로도 해당 서버가 thread-per-request인지 event-driven인지, 그리고 확장 한계가 어디인지 가늠할 수 있습니다.

답변 사고 순서

  1. 워크로드가 I/O-bound인지 CPU-bound인지 구분
    요청 처리 시간의 대부분이 네트워크/디스크 대기인지, 실제 연산(암호화, 이미지 처리, 복잡한 계산)인지 먼저 파악합니다. 이 구분이 두 모델의 유불리를 가르는 첫 갈림길입니다.
  2. 예상 동시 연결 규모 확인
    수백 단위인지 수만 단위인지에 따라 thread-per-request의 메모리·context switch 비용이 감당 가능한 수준인지 가늠합니다.
  3. thread-per-request 선택 시 한계 설명
    스레드 수 증가에 따른 메모리 사용량, context switch 오버헤드, 스레드 풀 고갈로 인한 연쇄 지연 가능성을 짚습니다.
  4. event-driven 선택 시 대가 설명
    콜백이 루프를 블로킹하면 전체 연결이 지연된다는 점, CPU-bound 작업은 별도 스레드/프로세스로 분리해야 한다는 점, 비동기 코드의 디버깅·스택 트레이스 난이도가 올라간다는 점을 짚습니다.
  5. 하이브리드 구조 고려
    실무 시스템 다수가 순수 단일 모델이 아니라 "이벤트 루프(I/O) + 워커 스레드 풀(CPU-bound 오프로드)"을 결합한다는 점을 언급합니다.
  6. 모니터링 지표로 검증 근거 제시
    스레드 수, context switch 비율(Day 3), 이벤트 루프 지연(Node.js의 event loop lag 등)을 운영 지표로 삼아 선택이 실제로 유효한지 사후 검증하는 방안을 제시합니다.

핵심 답변 예시

짧은 버전 (30초 답변)

강한 답변 (짧은 버전)

"요청 대부분이 DB나 외부 API를 기다리는 I/O-bound 워크로드이고 동시 연결이 1만 개 수준이라면, thread-per-request는 스레드 1만 개의 메모리와 context switch 비용을 그대로 감당해야 해서 비효율적입니다. epoll 기반 이벤트 루프를 쓰면 소수 스레드로 그 연결들을 멀티플렉싱할 수 있어 자원 효율이 훨씬 좋습니다. 다만 콜백 안에서 CPU-bound 연산을 동기로 돌리면 이벤트 루프 전체가 막히므로, 그런 작업은 별도 워커 스레드로 분리하는 하이브리드 구조로 설계하겠습니다."

깊이 있는 버전 (2분 답변)

강한 답변 (깊이 있는 버전)

"먼저 이 워크로드의 성격을 봐야 합니다. 1만 개 연결이 각각 DB 쿼리나 외부 API 응답을 기다리는 시간이 대부분이라면, 특정 순간에 실제로 CPU를 쓰는 연결은 소수입니다. thread-per-request 모델로 가면, 대기 중인 스레드도 커널 스택 메모리를 점유하고 스케줄러가 관리해야 할 대상이 되기 때문에, 실제 유효 작업량에 비해 자원 사용량이 과도해집니다. 게다가 스레드 수가 CPU 코어 수를 크게 넘어서면 context switch 오버헤드가 누적되어 처리량이 오히려 떨어지는 지점이 옵니다.

반면 event-driven 모델은 epoll_wait 하나로 수만 개 fd 중 실제로 준비된 것만 골라내기 때문에, 소수의 스레드(보통 CPU 코어 수 정도)만으로 대량의 연결을 처리할 수 있습니다. 이게 Nginx나 Node.js가 적은 자원으로 대규모 연결을 감당하는 이유입니다. 하지만 이 구조에는 명확한 대가가 있습니다 — 이벤트 루프는 '콜백이 절대 오래 블로킹하지 않는다'는 전제 위에 서 있습니다. 만약 콜백 안에서 동기 파일 I/O나 무거운 연산을 실행하면, 그 순간 같은 루프를 공유하는 다른 모든 연결의 이벤트 처리가 함께 지연됩니다. 이건 thread-per-request에서는 발생하지 않는, event-driven 모델 특유의 장애 패턴입니다.

그래서 저라면 순수 단일 모델보다는, I/O 대기가 지배적인 요청 경로는 이벤트 루프로 처리하고, 이미지 처리나 암호화처럼 CPU를 오래 쓰는 작업은 별도 워커 스레드 풀로 위임하는 하이브리드 구조를 택하겠습니다. 그리고 운영 중에는 스레드 수 추이, context switch 비율, 이벤트 루프 지연(예: Node.js의 event loop lag 지표)을 모니터링해서, 특정 요청 경로가 루프를 블로킹하고 있지는 않은지 조기에 잡아내는 체계를 함께 두겠습니다."

좋은 답변과 약한 답변의 차이

약한 답변

"비동기 I/O가 최신 기술이고 더 빠르니까 무조건 event-driven으로 가면 됩니다."

왜 약한가: 워크로드 특성(I/O-bound vs CPU-bound)에 대한 고려 없이 "최신 = 우월"이라는 근거 없는 전제로 결론을 냅니다. CPU-bound 작업이 섞였을 때 이벤트 루프가 오히려 전체 서비스를 멈출 수 있다는 event-driven 모델의 명확한 약점을 인식하지 못하고 있습니다.

약한 답변

"스레드를 최대한 많이 만들면 동시 처리량이 늘어나니까 스레드 수를 늘리면 됩니다."

왜 약한가: 스레드 수 증가가 항상 처리량 증가로 이어진다는 잘못된 가정입니다. 메모리 점유와 context switch 오버헤드가 스레드 수에 비례해 늘어나며, 어느 지점부터는 오히려 처리량이 감소한다는 Day 3의 핵심 원리를 무시하고 있습니다.

강한 답변의 패턴

① 워크로드가 I/O-bound인지 CPU-bound인지 먼저 구분 → ② thread-per-request의 자원 비용 (메모리, context switch)을 스레드 수와 연결지어 설명 → ③ epoll이 select/poll 대비 대량 연결에서 효율적인 이유를 동작 원리로 설명 → ④ 이벤트 루프가 콜백 블로킹에 취약하다는 구조적 대가를 명시 → ⑤ CPU-bound 작업을 별도 워커로 분리하는 하이브리드 구조 제시 → ⑥ 스레드 수, context switch, 이벤트 루프 지연을 지표로 사후 검증

Senior-level Follow-up 질문

▶ "select/poll 대신 epoll을 쓰는 이유를 알고리즘 복잡도 관점에서 설명해 보세요."
select()/poll()은 호출할 때마다 감시할 fd 목록 전체를 유저 공간에서 커널로 복사하고, 커널은 그 목록 전체를 순회하며 준비 상태를 확인합니다. 즉 감시 대상이 N개면 매 호출이 O(N)입니다. 연결 수가 수만 개로 늘어나면 이 순회 자체가 무시할 수 없는 오버헤드가 됩니다. epoll은 epoll_ctl로 감시할 fd를 커널 내부의 자료구조(레드-블랙 트리 등)에 한 번만 등록해두고, 이후 epoll_wait는 실제로 이벤트가 발생한 fd만 (커널이 내부적으로 유지하는 준비 리스트에서) 반환합니다. 매 호출마다 전체 목록을 복사하거나 순회할 필요가 없어, 연결 수가 늘어나도 호출 비용이 상대적으로 일정하게 유지됩니다.
▶ "async/await 같은 언어 차원의 비동기 문법은 스레드와 어떤 관계입니까?"
async/await는 OS 스레드를 대체하는 것이 아니라, 콜백 지옥을 코루틴 스타일로 감싼 문법적 편의에 가깝습니다. 내부적으로는 여전히 하나 이상의 이벤트 루프 스레드가 epoll 같은 메커니즘으로 I/O 완료를 기다리고, await 지점에서 함수 실행을 일시 정지했다가 해당 I/O가 준비되면 그 지점부터 재개합니다. 중요한 점은 이 재개도 결국 같은 이벤트 루프 스레드에서 순차적으로 일어난다는 것 — 그래서 async 함수 안에서 실수로 동기 블로킹 호출을 하면, "비동기 코드처럼 보이는" 코드가 실제로는 이벤트 루프 스레드 전체를 블로킹시키는 문제가 여전히 발생할 수 있습니다.
▶ "io_uring은 epoll과 무엇이 다르고 왜 등장했습니까?"
epoll은 "어떤 fd가 준비됐는지"만 알려줄 뿐, 실제 read()/write() 호출은 애플리케이션이 여전히 별도의 시스템 콜로 수행해야 합니다. 연결이 매우 많고 I/O 빈도가 높으면 이 시스템 콜 자체의 횟수(그리고 유저-커널 컨텍스트 전환 비용)가 병목이 될 수 있습니다. io_uring은 유저 공간과 커널이 공유하는 두 개의 링 버퍼(제출 큐, 완료 큐)를 통해, 시스템 콜 왕복 없이 I/O 요청을 제출하고 완료를 회수할 수 있게 합니다. 이는 "이벤트가 왔는지 알려주는" epoll의 통지 모델을 넘어, I/O 요청·완료 자체를 배치로 처리해 시스템 콜 오버헤드를 크게 줄이는 더 최신 접근입니다. 다만 커널 버전 요구사항이 있고 상대적으로 생태계가 아직 성숙 중이라는 점도 함께 언급하는 것이 좋습니다.
▶ "컨테이너 환경에서 이벤트 루프 기반 서비스가 CPU 제한(cgroup)에 걸리면 어떤 증상이 나타납니까?"
이벤트 루프는 하나의 스레드(또는 소수)가 계속 실행되어야 지연 없이 이벤트를 처리할 수 있습니다. cgroup의 CPU quota(예: cpu.cfs_quota_us)로 컨테이너의 CPU 사용을 제한하면, 이벤트 루프 스레드가 quota를 다 쓴 시점에 throttling되어 다음 period까지 강제로 실행이 멈춥니다. 이는 애플리케이션 코드에는 전혀 드러나지 않는 지연이라 "코드는 그대로인데 응답 latency가 주기적으로 튄다"는 증상으로 나타납니다. 컨테이너의 CPU throttling 지표(nr_throttled, throttled_time)를 확인하지 않으면, 애플리케이션 버그로 오인하기 쉬운 대표적인 함정입니다. 이벤트 루프 기반 서비스는 CPU limit을 너무 타이트하게 잡지 않거나, 워커 프로세스를 여러 개 두어 하나가 throttling 되어도 나머지가 요청을 받을 수 있게 설계하는 것이 실무적 대응입니다.
