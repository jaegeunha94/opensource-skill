컨테이너와 OS 격리

"컨테이너는 가벼운 VM"이라는 설명은 면접에서 감점 대상입니다. 이 레슨은 컨테이너가 하이퍼바이저 없이 namespace와 cgroup이라는 커널 기능만으로 격리를 흉내 내는 원리부터 시작해서, 그 격리가 완전하지 않기 때문에 실제 운영에서 발생하는 noisy neighbor, 커널 공유로 인한 부수 효과, 리소스 limit 설정 실수까지 하나의 인과 사슬로 연결해서 "컨테이너를 프로덕션에 왜, 어떻게 신뢰할 수 있는지"를 근거 있게 답하는 사고 과정을 훈련합니다.

오늘의 면접 질문

Senior Interview Question
"컨테이너는 VM과 무엇이 다르고, 운영 중에 어떤 문제가 생길 수 있습니까? 실제로 겪었거나 예상되는 장애 시나리오를 들어 설명해 주세요."

이 질문은 "컨테이너 = 가벼운 VM"이라는 마케팅 수준의 이해를 걸러내려는 목적을 가집니다. 면접관은 컨테이너가 왜 가벼운지(무엇을 공유하기 때문인지), 그리고 그 공유가 격리 경계를 어디까지만 보장하는지를 커널 메커니즘 수준에서 설명할 수 있는지, 그리고 그 한계가 실제로 어떤 운영 사고로 이어지는지를 확인합니다.

전제 개념: 이것부터 알고 있어야 합니다

1. VM은 하드웨어를 가상화하고, 컨테이너는 커널을 공유한다

VM(가상 머신)은 하이퍼바이저(예: KVM, VMware)가 물리 하드웨어를 가상화해서, 게스트마다 완전히 독립된 커널을 실행합니다. 게스트 OS 하나가 부팅부터 자체 커널, 자체 메모리 관리, 자체 디바이스 드라이버까지 전부 따로 가지므로 격리 수준이 매우 강하지만, 게스트 하나당 자체 커널을 부팅해야 해서 부팅 시간이 초 단위 이상, 메모리 오버헤드가 수백 MB 단위로 발생합니다.

컨테이너는 하이퍼바이저나 별도 커널 없이, 호스트 OS의 커널 하나를 여러 프로세스 그룹이 공유하면서 마치 독립된 시스템처럼 보이게 만드는 방식입니다. "컨테이너를 실행한다"는 것은 사실 호스트 커널 위에서 격리된 환경(namespace)과 자원 제한(cgroup)이 적용된 일반 프로세스를 실행하는 것과 다르지 않습니다. 별도 커널을 부팅하지 않으므로 시작 시간이 밀리초~수백 밀리초 단위이고, 메모리 오버헤드도 프로세스 하나 수준에 가깝습니다. 이 차이가 "컨테이너가 가벼운" 근본적인 이유입니다 — 그리고 동시에 "커널을 공유하기 때문에 격리가 VM만큼 강하지 않은" 근본적인 이유이기도 합니다.

VM
하이퍼바이저 + 게스트마다 독립 커널. 격리 강함(커널 취약점도 게스트 안에 격리). 부팅 초 단위, 메모리 오버헤드 큼. 서로 다른 커널 버전/OS 혼용 가능.
컨테이너
호스트 커널 공유 + namespace/cgroup. 시작 밀리초 단위, 오버헤드 작음. 커널 취약점이나 자원 경합이 호스트 전체에 영향 가능. 호스트와 다른 커널 버전 실행 불가.

2. Namespace — "네가 보는 세상을 좁혀준다"

리눅스 namespace는 프로세스가 시스템 자원의 전역 상태 중 자신에게 할당된 부분집합만 보이도록 시야를 좁히는 커널 기능입니다. "격리"라는 말이 추상적으로 느껴진다면, "실제로 존재하는 자원 목록에서 이 프로세스에게 보여줄 부분만 골라 보여주는 필터"라고 생각하면 됩니다. 대표적인 namespace 종류는 다음과 같습니다.

PID namespace — 컨테이너 안에서는 자신이 PID 1처럼 보임 (호스트에서는 다른 PID) Network ns — 자체 네트워크 인터페스, IP, 라우팅 테이블, 포트 공간을 가진 것처럼 보임 Mount ns — 자체 파일 시스템 마운트 트리를 가진 것처럼 보임 (chroot의 확장판) UTS ns — 자체 hostname을 가진 것처럼 보임 IPC ns — 자체 System V IPC, 메시지 큐를 가진 것처럼 보임 User ns — 컨테이너 안의 root(UID 0)를 호스트의 비특권 UID로 매핑 가능

중요한 점: namespace는 커널 자료구조 자체를 복제하지 않고, "이 프로세스 그룹에게는 이 부분집합만 보여준다"는 뷰(view)를 씌우는 것입니다. 실제 커널은 하나이고, 스케줄러도 하나이고, 물리 메모리 관리자도 하나입니다. 그래서 namespace만으로는 "내가 컨테이너 안에 있는 유일한 프로세스처럼 보인다"는 시야의 격리는 되지만, "내가 CPU나 메모리를 얼마나 쓸 수 있는가"라는 자원의 격리는 되지 않습니다. 그 역할은 cgroup이 맡습니다.

3. cgroup — "네가 쓸 수 있는 양을 제한한다"

cgroup(control group)은 프로세스 그룹이 사용할 수 있는 CPU, 메모리, 디스크 I/O, 네트워크 대역폭 등의 자원 사용량 상한을 강제하는 커널 기능입니다. Day 6에서 다룬 cgroup 메모리 한도가 바로 이것이고, Day 8에서 다룬 cpu.cfs_quota_us 기반 CPU throttling도 cgroup의 CPU 컨트롤러가 하는 일입니다.

namespace가 "무엇이 보이는가"를 결정한다면, cgroup은 "얼마나 쓸 수 있는가"를 결정합니다. 이 둘이 합쳐진 것이 컨테이너입니다: 컨테이너 런타임(Docker, containerd)은 새 프로세스를 시작할 때 여러 namespace를 새로 만들어 그 프로세스를 격리된 시야에 넣고, 동시에 그 프로세스를 특정 cgroup에 배치해 자원 사용을 제한합니다. 이미지 레이어(파일 시스템)와 이 둘을 합친 것이 우리가 흔히 "컨테이너"라고 부르는 실행 단위입니다.

흔한 오해
"컨테이너는 자체 커널을 가진 작은 VM이다"는 틀린 말입니다. 컨테이너는 호스트 커널을 그대로 공유하는 격리된 일반 프로세스입니다. 그래서 docker ps로 보이는 컨테이너의 프로세스는 호스트에서 ps aux로도 (다른 PID로) 보이며, 호스트 커널에 치명적 취약점이 있으면 그 취약점은 모든 컨테이너에 동시에 영향을 줄 수 있습니다. 또한 "컨테이너 안에서는 root라도 안전하다"도 틀린 말입니다 — User namespace로 UID를 매핑하지 않으면 컨테이너 안의 root는 특정 취약점(커널 버그, 잘못된 마운트 설정)을 통해 호스트의 root 권한으로 이어질 수 있습니다.

4. Noisy Neighbor — 같은 커널, 같은 자원을 나눠 쓰는 대가

namespace로 시야는 격리되어도, 물리 CPU, 메모리 대역폭, 디스크 I/O, 네트워크 대역폭은 결국 하나의 물리 자원을 여러 컨테이너가 나눠 씁니다. cgroup으로 상한을 걸어도, 완벽히 격리되지 않는 자원이 있습니다 — 예를 들어 CPU 코어 수 제한(quota)은 걸 수 있지만, 여러 컨테이너가 같은 물리 코어의 캐시(L1/L2/L3)나 메모리 버스 대역폭을 경합하는 것까지는 기본적으로 제어하지 못합니다. 한 컨테이너가 메모리를 매우 자주 읽고 써서 캐시를 오염시키면, 같은 노드의 다른 컨테이너가 cgroup 상 CPU quota는 충분히 남아있는데도 캐시 미스가 늘어 체감 성능이 떨어지는 현상이 생길 수 있습니다. 이것이 noisy neighbor 문제이며, 멀티테넌트 쿠버네티스 클러스터에서 "특정 시간대에 우리 서비스만 느려진다"는 증상의 흔한 원인 중 하나입니다.

왜 면접에서 이 질문을 묻는가

  1. "컨테이너 = 가벼운 VM"이라는 표면적 이해를 걷어내고 원리로 설명하는지 확인 — namespace와 cgroup이 각각 무엇을 격리하고 무엇을 격리하지 않는지 구분해서 설명할 수 있는지 봅니다.
  2. 격리의 한계를 실무 장애로 연결하는 능력 확인 — "커널 공유"라는 구조적 특성이 noisy neighbor, 커널 취약점 전파, 잘못된 자원 limit 설정 같은 실제 운영 리스크로 이어진다는 것을 아는지 봅니다.
  3. 컨테이너 오케스트레이션 환경에서의 자원 설계 경험 확인 — requests/limits를 설정하는 근거, OOM kill과의 관계, CPU throttling 진단 경험이 있는지 봅니다.

실무 연결

  • 쿠버네티스 requests/limits와 OOM kill: 컨테이너의 메모리 cgroup 한도를 초과하면 프로세스가 OOM killer에 의해 즉시 종료됩니다(Day 6의 swap/thrashing과 달리, 컨테이너 메모리 cgroup은 보통 스왑을 쓰지 않도록 구성되어 한도 초과 시 바로 kill로 이어집니다). "메모리가 남는 것 같은데 컨테이너가 갑자기 재시작된다"는 신고의 상당수가 이 현상이며, OOMKilled 상태와 cgroup 메모리 사용량 그래프를 함께 봐야 원인을 확정할 수 있습니다.
  • CPU limit과 스로틀링(Day 8 재등장): CPU limit을 너무 타이트하게 걸면, 순간적으로 짧게 CPU를 많이 쓰는 워크로드(예: GC pause 직후 몰아치는 처리)가 cpu.cfs_quota_us 소진으로 스로틀링되어 지연이 튀는 현상이 발생합니다. 이는 애플리케이션 로그에는 전혀 나타나지 않고 nr_throttled/throttled_time 지표로만 드러나는, 컨테이너 환경 특유의 함정입니다.
  • Noisy neighbor와 bin packing: 쿠버네티스 스케줄러가 같은 노드에 CPU를 많이 쓰는 워크로드 여러 개를 몰아 배치(bin packing)하면, 각 파드의 cgroup CPU quota는 지켜지더라도 노드 전체의 물리 자원(메모리 대역폭, 캐시, 디스크 I/O)이 경합해 전체적으로 성능이 떨어질 수 있습니다. 이런 경우 특정 노드 셀렉터, 어피니티/안티어피니티, 전용 노드 풀 분리 같은 배치 전략으로 완화합니다.
  • 커널 공유로 인한 보안 사고 범위: 컨테이너 이스케이프 취약점(예: 잘못된 마운트 설정, 커널 버그를 이용한 권한 상승)은 VM 이스케이프보다 훨씬 자주 보고됩니다. 호스트 커널 하나를 여러 테넌트가 공유하는 구조이기 때문에, 신뢰할 수 없는 워크로드를 함께 돌려야 한다면 컨테이너보다 강한 격리(gVisor, Kata Containers, 또는 아예 별도 VM)를 고려해야 합니다.
  • 이미지 레이어와 배포 속도: 컨테이너가 VM보다 빠르게 뜨는 것은 namespace/cgroup 설정이 가벼운 이유도 있지만, 이미지 레이어(overlay 파일 시스템)를 캐시해서 재사용하기 때문이기도 합니다. 배포 파이프라인에서 이미지 크기와 레이어 구성을 최적화하는 것이 배포 속도와 콜드 스타트 latency에 직접 영향을 줍니다.

답변 사고 순서

  1. VM과 컨테이너의 구조적 차이를 먼저 명확히
    VM은 하이퍼바이저 위 독립 커널, 컨테이너는 호스트 커널을 공유하는 격리된 프로세스라는 점을 분명히 짚습니다. 이게 이후 모든 trade-off의 근거가 됩니다.
  2. 격리를 담당하는 두 메커니즘을 분리해서 설명
    namespace(시야 격리)와 cgroup(자원 제한)의 역할을 구분합니다. 이 구분이 없으면 "격리"라는 말이 뭉뚱그려져 다음 단계인 "한계"를 설명할 근거가 사라집니다.
  3. 격리가 완전하지 않은 지점을 짚는다
    커널은 하나이고, 물리 자원(캐시, 메모리 대역폭)은 cgroup으로도 완벽히 나눌 수 없다는 점을 설명합니다.
  4. 그 한계가 실제로 어떤 증상으로 나타나는지 구체적 사례 제시
    noisy neighbor, OOM kill, CPU throttling처럼 지표로 확인 가능한 실제 장애 패턴을 듭니다.
  5. 완화 방안 제시
    적절한 requests/limits 설정, 노드 배치 전략, 필요 시 강한 격리(gVisor/Kata/VM)로의 전환까지 상황별 대응을 제시합니다.
  6. 모니터링 지표로 검증 근거 제시
    cgroup 메모리/CPU 사용량, nr_throttled, OOM 이벤트, 노드 레벨 자원 사용률을 운영 지표로 삼아 판단의 근거를 제시합니다.

핵심 답변 예시

짧은 버전 (30초 답변)

강한 답변 (짧은 버전)

"VM은 하이퍼바이저 위에서 게스트마다 독립된 커널을 부팅하지만, 컨테이너는 호스트 커널 하나를 공유하면서 namespace로 시야를, cgroup으로 자원 사용량을 제한하는 방식입니다. 그래서 컨테이너가 훨씬 가볍지만, 커널을 공유하기 때문에 완벽한 격리는 아닙니다. 실무에서는 이게 noisy neighbor(물리 자원 경합), CPU quota 스로틀링으로 인한 지연, 메모리 한도 초과 시 OOM kill 같은 형태로 나타나서, requests/limits를 신중히 설정하고 관련 지표를 모니터링하는 게 중요합니다."

깊이 있는 버전 (2분 답변)

강한 답변 (깊이 있는 버전)

"먼저 구조적 차이부터 짚겠습니다. VM은 하이퍼바이저가 하드웨어를 가상화해서 게스트마다 완전히 독립된 커널을 실행합니다. 컨테이너는 그런 별도 커널 부팅 과정이 없고, 호스트 OS의 커널 하나를 여러 프로세스 그룹이 그대로 공유합니다. 대신 리눅스는 namespace라는 기능으로 각 프로세스 그룹이 자신만의 PID 공간, 네트워크, 파일 시스템 마운트 트리를 가진 것처럼 보이게 시야를 좁혀주고, cgroup이라는 기능으로 그 프로세스 그룹이 쓸 수 있는 CPU, 메모리, I/O 상한을 강제합니다. 이 두 커널 기능의 조합이 컨테이너의 정체이고, 별도 커널을 부팅하지 않기 때문에 시작이 밀리초 단위로 빠르고 오버헤드도 작습니다.

문제는 이 가벼움의 대가입니다. namespace는 '보이는 것'만 제한할 뿐 커널 자체는 하나이기 때문에, 호스트 커널의 취약점은 그 위에서 도는 모든 컨테이너에 동시에 영향을 줄 수 있습니다. 그리고 cgroup으로 CPU나 메모리의 논리적 상한은 걸 수 있어도, 물리 코어의 캐시나 메모리 버스 대역폭처럼 하드웨어 수준에서 진짜로 나눌 수 없는 자원은 완벽히 격리되지 않습니다. 그래서 같은 노드에 배치된 다른 컨테이너가 캐시를 많이 오염시키는 워크로드를 돌리면, 제 컨테이너는 CPU quota가 충분히 남아있는데도 체감 성능이 떨어지는 noisy neighbor 현상을 겪을 수 있습니다.

실무에서 이 한계는 몇 가지 구체적인 증상으로 나타납니다. 메모리 cgroup 한도를 초과하면 스왑 없이 즉시 OOM killer가 프로세스를 종료시켜서, 로그상으로는 멀쩡했는데 컨테이너가 갑자기 재시작되는 것처럼 보입니다. CPU limit을 너무 빡빡하게 걸면, 순간적으로 CPU를 몰아 쓰는 구간에서 CFS quota가 소진되어 스로틀링되고, 이건 애플리케이션 로그가 아니라 nr_throttled 같은 cgroup 지표를 봐야만 확인할 수 있습니다. 그래서 저라면 requests/limits를 실측 기반으로 설정하고, OOM 이벤트와 스로틀링 지표를 모니터링에 포함시키고, 정말 신뢰할 수 없는 워크로드를 함께 돌려야 한다면 gVisor나 Kata Containers 같은 강한 격리, 또는 아예 별도 VM 분리까지 고려하겠습니다."

좋은 답변과 약한 답변의 차이

약한 답변

"컨테이너는 가벼운 VM이라서 더 빠르고 자원도 적게 씁니다."

왜 약한가: "가벼운 VM"이라는 표현 자체가 컨테이너와 VM의 구조적 차이(독립 커널 vs 커널 공유)를 무시한 틀린 비유입니다. 왜 가벼운지(커널 공유, namespace/cgroup)에 대한 설명이 없고, 그 가벼움이 격리 수준의 trade-off라는 점도 전혀 언급하지 못했습니다.

약한 답변

"컨테이너는 격리되어 있으니까 자원을 얼마나 쓰든 서로 영향을 주지 않습니다."

왜 약한가: namespace의 시야 격리와 cgroup의 논리적 자원 제한을, 물리 하드웨어 수준의 완전한 격리와 혼동하고 있습니다. noisy neighbor 문제, 커널 취약점 전파처럼 실제로 발생하는 격리의 한계를 전혀 인식하지 못한 답변입니다.

강한 답변의 패턴

① VM(독립 커널)과 컨테이너(커널 공유)의 구조적 차이를 먼저 명시 → ② namespace(시야 격리)와 cgroup(자원 제한)의 역할을 구분해서 설명 → ③ 커널 공유와 물리 자원 경합으로 인해 격리가 완전하지 않다는 한계를 명시 → ④ OOM kill, CPU throttling, noisy neighbor 같은 구체적 장애 패턴으로 연결 → ⑤ requests/limits 설정, 노드 배치, 강한 격리 런타임 같은 완화 방안 제시 → ⑥ cgroup 지표 기반 모니터링으로 검증 방안 제시

Senior-level Follow-up 질문

▶ "컨테이너 안에서 top이나 free를 실행하면 왜 호스트 전체의 CPU/메모리가 보이는 경우가 있습니까?"
top, free 같은 전통적인 도구는 /proc/meminfo, /proc/stat 같은 파일을 읽어 정보를 얻는데, 이 파일들은 기본적으로 PID namespace가 아니라 컨테이너에 마운트된 /proc가 어디까지 격리되어 있는지에 따라 호스트 전체의 값을 그대로 보여줄 수 있습니다. cgroup은 실제 자원 사용을 제한하지만, "지금 몇 개의 CPU를 쓸 수 있는가"를 알려주는 표준 인터페이스가 아니기 때문에, 컨테이너 안의 애플리케이션이 /proc/cpuinfo를 보고 호스트의 전체 코어 수를 감지해 스레드 풀 크기를 잘못 설정하는 사고가 실무에서 자주 발생합니다. 최근에는 cgroup-aware한 런타임 설정이나 애플리케이션이 cgroup 파일 (cpu.max 등)을 직접 읽어 실제 할당량을 반영하도록 수정하는 방식으로 대응합니다.
▶ "User namespace가 컨테이너 보안에 어떤 역할을 합니까?"
User namespace가 없으면 컨테이너 안의 root(UID 0)는 실제로 호스트의 UID 0와 동일한 권한을 가진 프로세스입니다. 그래서 컨테이너 이스케이프 취약점이 발생하면 곧바로 호스트 root 권한 획득으로 이어질 위험이 있습니다. User namespace를 사용하면 컨테이너 안의 UID 0을 호스트의 비특권 UID(예: 100000)로 매핑해서, 컨테이너 안에서는 root처럼 동작하지만 호스트 관점에서는 평범한 비특권 사용자로 취급되게 만들 수 있습니다. 이는 컨테이너 이스케이프가 발생하더라도 피해 범위를 호스트 root 권한이 아니라 매핑된 비특권 사용자 권한으로 제한하는 심층 방어(defense in depth) 기법입니다.
▶ "gVisor나 Kata Containers는 일반 컨테이너와 무엇이 다릅니까?"
일반 컨테이너(runc 기반)는 호스트 커널을 그대로 공유하며 namespace/cgroup으로만 격리합니다. Kata Containers는 컨테이너 하나마다 경량 VM(자체 커널)을 띄워서 실행함으로써, 컨테이너의 편의성(이미지, 오케스트레이션 API 호환)은 유지하면서 VM 수준의 커널 격리를 제공합니다 — 대신 VM을 띄우는 만큼 시작 시간과 오버헤드가 순수 컨테이너보다 커집니다. gVisor는 별도 커널을 띄우는 대신, 컨테이너의 시스템 콜을 가로채 유저 공간에 구현된 자체 커널 유사 계층(Sentry)이 대신 처리하게 만들어, 호스트 커널에 직접 닿는 시스템 콜 표면을 크게 줄입니다. 둘 다 "커널 공유로 인한 이스케이프 리스크"를 줄이기 위한 접근이지만, 방식(별도 커널 vs 시스템 콜 가로채기)과 그에 따른 성능/호환성 trade-off가 다릅니다. 신뢰할 수 없는 멀티테넌트 워크로드(예: 서드파티 코드 실행 플랫폼)를 돌려야 할 때 고려하는 선택지입니다.
▶ "쿠버네티스에서 requests와 limits를 다르게 설정하면 어떤 리스크가 생깁니까?"
requests는 스케줄러가 파드를 노드에 배치할 때 "이만큼은 보장되어야 한다"고 예약하는 값이고, limits는 cgroup이 실제로 강제하는 상한입니다. limits를 requests보다 훨씬 높게 잡으면(overcommit), 평소에는 여러 파드가 여유 자원을 나눠 쓰며 잘 동작하지만, 여러 파드가 동시에 순간적으로 limits에 가까운 자원을 요구하면 노드의 물리 자원이 부족해져 CPU는 스로틀링으로, 메모리는 OOM killer가 우선순위 낮은 파드부터 죽이는 방식으로 대응합니다. 이 시점에 어떤 파드가 먼저 죽는지는 QoS 클래스(Guaranteed, Burstable, BestEffort)에 따라 결정되므로, 중요한 서비스는 requests와 limits를 동일하게 설정해 Guaranteed QoS로 배치하는 것이 일반적인 실무 대응입니다.
