종합 모의 면접: 운영체제 지표 기반 RCA

실제 시니어 면접의 마지막 라운드는 단일 개념을 묻지 않습니다. 여러 원인이 동시에 얽혀 있는 장애 시나리오를 던지고, 후보를 어떻게 좁혀가며 어떤 순서로 검증하는지를 봅니다. 이 레슨은 Day 1~9에서 배운 CPU/스케줄링, 동시성, 메모리, 파일 I/O, 비동기 I/O, 컨테이너 격리를 하나의 장애 시나리오 안에서 순차적으로 단서를 해석하는 모의 면접 형식으로 통합합니다.

오늘의 면접 질문

Senior Interview Question — Live RCA
"결제 서비스를 쿠버네티스로 이전한 뒤, 평소에는 문제가 없는데 매일 저녁 피크 시간대에만 p99 레이턴시가 150ms에서 900ms로 튑니다. 평균 CPU 사용률은 45%, 메모리 사용량도 안정적이고 DB 쿼리 응답 시간도 정상입니다. 지금부터 지표를 하나씩 드릴 테니, 각 단계에서 어떤 가설을 세우고 다음에 무엇을 확인할지 실시간으로 말씀해 주세요."

이런 형식은 "정답을 아는가"가 아니라 "불완전한 정보 상태에서 다음 행동을 논리적으로 선택할 수 있는가"를 봅니다. 면접관이 단서를 하나씩 던지는 이유는, 실무 장애 대응이 정확히 이런 식(처음부터 전체 그림이 보이지 않고, 지표를 하나씩 확인하며 가설을 갱신하는 과정)으로 진행되기 때문입니다.

전제 개념: Day 1~9 진단 프레임워크 복습

이 레슨을 풀려면 지금까지의 레슨에서 각 계층이 어떤 증상과 지표로 나타나는지 한 번에 떠올릴 수 있어야 합니다. 아래 표는 Day 1의 4계층 진단 프레임워크에 Day 2~9의 세부 원인을 겹쳐 정리한 것입니다.

계층 대표 지표 의심할 원인 (관련 Day)
CPU / 스케줄링 vmstat r, cs / mpstat / load average run queue 적체, context switch 폭증, 스레드 풀 과다 (Day 1, 3)
동시성 thread dump, pidstat -w, lock wait 시간 race condition, 전역 락, lock contention, deadlock 근접 상태 (Day 4, 5)
메모리 vmstat si/so, page fault율, cgroup memory.stat 가상 메모리 압박, swap/thrashing, cgroup 메모리 한도 근접 (Day 6)
파일 I/O iostat -x await/%util, fsync 지연 동기 로그 쓰기, 저널링 오버헤드, 디스크 포화 (Day 7)
비동기 I/O / 이벤트 루프 이벤트 루프 lag, epoll 대기 큐 길이 이벤트 루프를 블로킹하는 동기 호출, worker 부족 (Day 8)
컨테이너 경계 cgroup cpu.stat nr_throttled, OOM 이벤트 CPU quota 스로틀링, noisy neighbor, requests/limits 오설정 (Day 9)
이 레슨에서 반드시 버려야 할 습관
"평균 CPU가 45%니까 CPU는 문제가 아니다"라는 결론은 성급합니다. 평균은 피크 시간대의 순간적인 포화를 가려버립니다. cgroup CPU throttling은 평균 사용률이 낮아도 짧은 순간에 quota를 다 써버리면 발생하기 때문에, "평균이 낮다 = CPU는 무죄"라는 결론 자체가 이 시나리오 전체를 놓치게 만드는 함정입니다.

왜 면접에서 이 질문을 묻는가

  1. 단일 원인 사고에서 벗어났는지 확인 — 실무 장애의 상당수는 한 가지 원인이 아니라 여러 요인이 특정 시간대에 겹쳐 임계점을 넘을 때 발생합니다. 후보를 하나만 찾고 멈추는지, 복합 원인 가능성을 열어두는지를 봅니다.
  2. 지표를 받았을 때 실시간으로 가설을 갱신하는 능력 확인 — 이론을 안다고 해서 실제 지표를 해석하는 것은 다른 능력입니다. "이 지표가 나오면 다음에 무엇을 볼 것인가"를 끊기지 않고 이어갈 수 있는지 봅니다.
  3. Day 1~9 개별 지식을 하나의 사고 체계로 통합했는지 확인 — 개별 개념을 설명하는 것과, 여러 계층에 걸친 증상을 하나의 타임라인으로 엮어 설명하는 것은 난이도가 다릅니다. 이 통합 능력이 진짜 시니어와 "암기한 시니어"를 가릅니다.

실무 연결: 모의 면접 전개 (단서를 하나씩 받는다)

아래는 실제 면접에서 면접관이 순서대로 단서를 공개하는 방식을 재현한 것입니다. 각 단서를 읽고 다음에 무엇을 확인할지 먼저 스스로 생각한 뒤 "판정" 박스를 확인하세요.

단서 1 — 시스템 레벨 지표

피크 시간대에 해당 파드에서 vmstat 1을 돌려보니 r(run queue) 값이 평소 2~3에서 순간적으로 15까지 튀고, cs(context switch)도 초당 3천 건에서 4만 건으로 급증합니다. us+sy는 크게 오르지 않습니다.

procs -----------memory---------- ---swap-- -----io---- --system-- ------cpu----- r b swpd free buff cache si so bi bo in cs us sy id wa 15 0 81920 210044 12288 512000 0 0 4 88 9200 41800 22 9 60 9
1차 판정: run queue와 context switch가 함께 튀는데 CPU 사용률(us+sy)은 크게 오르지 않는다는 것은, CPU가 계산을 많이 해서 바쁜 게 아니라 스케줄러가 짧은 시간 안에 많은 스레드를 번갈아 실행시키고 있다는 신호입니다(Day 3). 다음으로 확인할 것: 이 context switch가 voluntary(자발적, 뭔가를 기다리며 양보)인지 involuntary (스케줄러 강제 선점)인지를 구분해야 합니다.
단서 2 — 스레드 단위 지표

pidstat -w -p <pid> 1로 스레드별 context switch를 나눠보니, 결제 처리 스레드 풀 전체가 voluntary_ctxswitches가 비정상적으로 높습니다. 스레드 덤프를 떠 보니 다수의 스레드가 synchronized 블록 하나(재고 차감 로직)에서 BLOCKED 상태로 대기 중입니다.

2차 판정: voluntary context switch가 지배적이라는 것은 스레드들이 CPU를 강제로 빼앗기는 게 아니라 스스로 뭔가(락)를 기다리며 양보하고 있다는 뜻입니다 (Day 3에서 배운 voluntary/involuntary 구분). 스레드 덤프가 이를 확정합니다 — coarse-grained lock으로 인한 lock contention이 첫 번째 원인 후보입니다 (Day 4, 5). 하지만 여기서 멈추면 안 됩니다: 락 스코프를 좁혀도 문제가 완전히 사라지지 않는다면 다른 계층도 함께 봐야 합니다.
단서 3 — 락 스코프를 좁힌 뒤에도 남은 증상

재고 차감 로직을 세밀한 락(row-level 또는 CAS 기반 원자적 갱신)으로 바꿔 배포했더니 p99가 900ms에서 400ms로 줄었지만, 여전히 평소(150ms) 대비 높습니다. 해당 파드의 cgroup cpu.stat을 보니 피크 시간대에 nr_throttled가 분당 수백 건씩 증가하고 있습니다.

3차 판정: lock contention은 완화됐지만 남은 지연은 cgroup CPU quota 스로틀링입니다(Day 9). requests/limits가 평소 트래픽 기준으로 타이트하게 설정되어 있어서, 락 경합이 줄어 스레드들이 동시에 CPU를 요구할 수 있게 되자 오히려 cpu.cfs_quota_us를 짧은 주기 안에 소진해 스로틀링에 더 자주 걸린 것입니다 — 한 원인(락 경합)을 완화하니 다른 원인(CPU quota 부족)이 병목으로 드러난 전형적인 사례입니다.
단서 4 — 로그와 이벤트 루프

CPU limit을 완화한 뒤에도 가끔 개별 요청이 순간적으로 800ms 근처까지 튀는 롱테일이 남아있습니다. APM 분산 추적을 보니, 결제 완료 후 감사 로그를 JSON으로 동기 write하는 코드가 이벤트 루프 스레드 안에서 fsync를 호출하고 있었고, 디스크가 다른 파드들과 같은 EBS 볼륨을 공유하는 노드였습니다.

4차 판정: 이벤트 루프 스레드 안에서 동기 fsync를 호출하면 그 순간 이벤트 루프 전체가 블로킹되어, 대기 중이던 다른 모든 요청의 처리가 지연됩니다(Day 7, Day 8). 공유 디스크에 다른 파드의 I/O가 몰리는 순간과 겹치면 await가 튀면서 그 블로킹 시간이 더 길어집니다. 이것이 락 경합과 CPU 스로틀링을 모두 해결한 뒤에도 남아있던 롱테일의 정체입니다.

최종 그림: 이 장애는 단일 원인이 아니라 세 가지 원인이 같은 시간대(피크)에 겹쳐 있던 것이었습니다 — (1) 전역 락으로 인한 lock contention, (2) 타이트한 cgroup CPU limit으로 인한 스로틀링, (3) 이벤트 루프를 블로킹하는 동기 fsync 로깅. 세 가지 모두 "피크 시간대에만" 나타나는 이유는 각각 동시 요청 수가 임계점을 넘을 때만 증상이 드러나는 비선형적 병목이기 때문입니다.

답변 사고 순서

  1. 증상을 정밀화하고 "정상 지표"의 함정을 경계
    평균이 정상이라는 말에 안심하지 않습니다. p50/p95/p99를 나눠 보고, 증상이 시간대·엔드포인트· 노드 단위로 국소적인지 확인합니다.
  2. CPU/스케줄링 계층부터 순서대로 내려간다
    vmstat의 r, cs로 스케줄러 압박 여부를 먼저 봅니다(Day 1, 3). 이 단계에서 "바쁜 것"과 "기다리는 것"을 반드시 구분합니다.
  3. voluntary/involuntary context switch로 원인의 방향을 가른다
    자발적 양보가 많으면 락/IO 대기(Day 4, 5, 7, 8) 쪽을, 강제 선점이 많으면 스레드 과다/스케줄러 경합(Day 3) 쪽을 우선 조사합니다.
  4. 하나를 고쳐도 끝났다고 결론짓지 않는다
    한 원인을 완화한 뒤 지표가 얼마나 개선됐는지 다시 측정합니다. 완전히 해소되지 않으면 "다른 계층에 숨겨져 있던 원인이 드러난 것"으로 보고 계속 내려갑니다.
  5. 컨테이너 경계 지표를 빠뜨리지 않는다
    쿠버네티스 환경이라면 애플리케이션 지표가 정상이어도 cgroup cpu.stat, memory.stat, OOM 이벤트를 반드시 함께 봅니다(Day 9). 호스트 관점의 지표와 cgroup 관점의 지표가 다르게 보일 수 있습니다.
  6. 블로킹 지점을 이벤트 루프/스레드 모델 관점에서 마지막으로 점검
    동기 I/O 호출 한 줄이 비동기 처리 모델 전체를 블로킹할 수 있다는 것을 항상 의심 목록에 둡니다(Day 7, 8).
  7. 복합 원인 여부를 명시적으로 결론에 포함
    "원인은 A였다"가 아니라 "A, B, C가 피크 시간대에 겹쳐 있었고, 각각을 순서대로 완화하며 확인했다"는 형태로 답변을 마무리합니다.

핵심 답변 예시

짧은 버전 (30초 답변)

강한 답변 (짧은 버전)

"평균 지표가 정상이어도 피크 시간대의 순간적인 포화는 가려질 수 있어서, 먼저 vmstat으로 run queue와 context switch를 보고 CPU가 '바쁜 것'인지 '기다리는 것'인지부터 구분하겠습니다. voluntary context switch가 많으면 스레드 덤프로 락 경합을, 이후 cgroup cpu.stat으로 스로틀링을, 마지막으로 이벤트 루프를 블로킹하는 동기 I/O 호출을 순서대로 확인하겠습니다. 여러 원인이 동시에 겹쳐 있을 수 있다는 전제를 유지하고, 하나를 완화한 뒤에도 반드시 재측정하겠습니다."

깊이 있는 버전 (2분 답변)

강한 답변 (깊이 있는 버전)

"먼저 '평균 CPU 45%, 메모리 안정적'이라는 정보에 안심하지 않겠습니다. 피크 시간대에만 나타나는 문제는 평균에 묻히는 순간적 포화일 가능성이 높기 때문에, p50과 p99를 나눠서 보고 문제가 특정 엔드포인트나 특정 노드에 국한되는지부터 확인합니다.

다음으로 CPU/스케줄링 계층을 봅니다. vmstat에서 run queue와 context switch가 함께 튀는데 us+sy는 크게 오르지 않는다면, CPU가 계산으로 바쁜 게 아니라 스케줄러가 짧은 시간에 많은 스레드를 오가고 있다는 뜻입니다. 이때 voluntary context switch 비중이 높으면 락이나 I/O 대기를, involuntary 비중이 높으면 스레드 과다나 스케줄러 경합을 의심합니다.

스레드 덤프에서 다수 스레드가 같은 락에 몰려 있다면 lock contention을 1차 원인으로 잡고 락 스코프를 좁히거나 원자적 연산으로 대체합니다. 그런데 이걸 고쳤다고 바로 끝났다고 결론짓지 않습니다. 반드시 재측정해서 지표가 얼마나 개선됐는지 봅니다. 만약 락 경합이 줄었는데도 여전히 지연이 남아있다면, 락 경합에 가려져 있던 다른 병목 — 예를 들어 컨테이너 환경이라면 cgroup CPU quota 스로틀링 — 이 드러난 것일 수 있습니다. 스레드들이 락 대기 없이 동시에 CPU를 요구하게 되면서 오히려 quota를 더 빨리 소진할 수 있기 때문입니다.

여기까지 해결한 뒤에도 롱테일이 남는다면, 이벤트 루프나 워커 스레드 안에서 동기 I/O를 호출하는 코드가 없는지 봅니다. 로깅이나 감사 기록처럼 눈에 잘 안 띄는 코드가 fsync를 동기 호출해서 처리 스레드 전체를 블로킹시키는 경우가 실무에서 흔합니다. 특히 여러 파드가 같은 디스크 볼륨을 공유하는 환경에서는 다른 워크로드의 I/O 부하와 우연히 겹치는 순간에만 증상이 나타나서 재현이 어렵습니다.

결론적으로 이런 장애는 원인을 하나로 확정하기보다, '피크 시간대에 락 경합, CPU 스로틀링, 블로킹 I/O가 동시에 임계점을 넘었다'는 복합 원인으로 설명하고, 각 원인을 순서대로 완화하며 재측정한 근거를 함께 제시하겠습니다."

좋은 답변과 약한 답변의 차이

약한 답변

"스레드 덤프를 보니 락 경합이 있었습니다. 락을 없앴더니 해결됐습니다."

왜 약한가: 단서 3, 4에서 드러나듯 락 경합은 세 가지 원인 중 하나였을 뿐입니다. 첫 번째로 찾은 원인에서 탐색을 멈추면, 락을 고친 뒤에도 남아있는 스로틀링과 블로킹 I/O를 놓치게 됩니다. 실무에서 "고쳤는데 왜 또 느려지냐"는 재발의 상당수가 이런 조기 종료에서 나옵니다.

약한 답변

"평균 CPU가 45%밖에 안 되니까 CPU/컨테이너 쪽은 원인이 아닐 것 같습니다."

왜 약한가: cgroup CPU throttling은 평균 사용률이 아니라 짧은 주기(기본 100ms) 안에서의 순간적인 quota 소진으로 발생합니다. 평균 지표만으로 계층 하나를 통째로 배제하는 것은 이 시나리오의 세 번째 원인을 놓치게 만드는 전형적인 실수입니다.

강한 답변의 패턴

① 평균 지표의 함정을 경계하고 p99/국소성부터 확인 → ② CPU/스케줄링 계층에서 "바쁨"과 "기다림"을 구분 → ③ voluntary/involuntary context switch로 방향을 좁혀 락 경합을 1차 원인으로 특정 → ④ 완화 후 반드시 재측정하고, 남은 증상이 있으면 다음 계층(컨테이너 경계)으로 내려감 → ⑤ cgroup 지표로 스로틀링을 2차 원인으로 특정 → ⑥ 그래도 남는 롱테일은 이벤트 루프를 블로킹하는 I/O 호출로 좁혀 3차 원인을 확정 → ⑦ 복합 원인이라는 결론과 각 단계의 재측정 근거를 함께 제시

Senior-level Follow-up 질문

▶ "여러 원인이 동시에 관찰될 때, 어떤 순서로 검증할지 어떻게 정합니까?"
영향 범위와 확인 비용이 낮은 것부터 봅니다. vmstat, pidstat 같은 시스템 지표는 관찰 비용이 거의 없고 부작용도 없어 가장 먼저 확인합니다. 스레드 덤프처럼 순간적으로 애플리케이션을 멈출 수 있는 방법은 그다음, 실제 코드 변경(락 스코프 축소, limit 조정)처럼 되돌리기 번거로운 조치는 가장 나중에, 그것도 한 번에 하나씩만 적용해 각 변경의 효과를 독립적으로 측정합니다. 여러 변경을 동시에 적용하면 어떤 변경이 실제로 효과가 있었는지 알 수 없게 됩니다.
▶ "지표 사이의 상관관계와 인과관계를 어떻게 구분합니까?"
두 지표가 같은 시간대에 함께 튀는 것(상관관계)만으로는 원인을 확정할 수 없습니다. "이 지표를 인위적으로 변화시키면 증상도 따라 변하는가"를 확인하는 개입(intervention)이 필요합니다. 이 시나리오에서도 락 스코프를 좁히는 개입을 실제로 적용하고 지표 변화를 재측정했기 때문에 lock contention이 인과관계에 있다고 확정할 수 있었습니다. 단순히 "락 대기가 관찰됐다"는 관찰만으로 멈췄다면 상관관계와 인과관계를 혼동한 것입니다.
▶ "이런 간헐적 장애를 프로덕션에서 재현하기 어렵다면 어떻게 접근합니까?"
두 가지 축으로 접근합니다. 첫째, 상시 관측성을 강화해서 장애가 실제로 발생하는 순간의 지표를 자동으로 캡처해 둡니다 — 예를 들어 p99 latency가 임계값을 넘는 순간 자동으로 스레드 덤프와 cgroup 지표 스냅샷을 남기는 알림/자동화를 구성합니다. 둘째, 부하 테스트로 조건을 인위적으로 재현합니다 — 피크 시간대의 동시 요청 수, 락 경합 패턴, 로깅 빈도를 스테이징 환경에서 흉내 내어 같은 증상이 재현되는지 확인합니다. 두 방법 모두 "프로덕션에서만 재현된다"는 답변을 "특정 부하 조건에서 재현된다"는 구체적 조건으로 바꾸는 것이 목표입니다.
▶ "이 장애를 팀 전체와 경영진에게 각각 어떻게 다르게 설명하시겠습니까?"
팀에는 근본 원인의 기술적 인과 사슬(락 경합 → 스로틀링 → 블로킹 I/O)과 각 단계의 재측정 근거, 재발 방지를 위한 구체적 변경 사항(락 구조, requests/limits, 로깅 방식)을 상세히 공유합니다. 경영진에게는 기술 용어 대신 "결제 지연이 피크 시간대에만 발생했고, 세 가지 독립적인 원인이 동시에 겹친 복합 장애였다. 각 원인을 순차적으로 제거했고, 재발 방지를 위해 자동 감지 알림을 추가했다"는 식으로 영향 범위, 원인의 성격(복합적), 재발 방지 조치 중심으로 요약해 전달합니다. 기술적 세부사항의 나열이 아니라 "무엇을 했고 앞으로 어떻게 막는가"에 초점을 맞춥니다.
