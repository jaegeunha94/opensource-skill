교착 상태 (Deadlock)

둘 이상의 프로세스가 서로 상대방이 가진 자원을 기다리며 영원히 진행되지 못하는 상태를 교착 상태(Deadlock)라 합니다. 교착 상태가 왜 발생하는지, 어떻게 예방·회피·탐지·회복하는지 배웁니다.

학습 목표
  • 교착 상태의 4가지 필요 조건을 모두 열거할 수 있다.
  • 자원 할당 그래프(Resource-Allocation Graph)로 교착 상태를 판별할 수 있다.
  • 예방·회피·탐지·회복 전략의 차이를 설명할 수 있다.
  • 은행원 알고리즘(Banker's Algorithm)의 원리를 이해한다.

1. 교착 상태란?

Day 5에서 살펴본 생산자-소비자 예제를 떠올려 보세요. wait(mutex)를 wait(empty/full)보다 먼저 호출하면 교착 상태가 생긴다고 했습니다. 더 직관적인 예시를 봅시다.

프로세스 A: lock(mutex_1) → wait for mutex_2 … 프로세스 B: lock(mutex_2) → wait for mutex_1 … A는 B가 mutex_2를 놓기를 기다리고, B는 A가 mutex_1을 놓기를 기다린다. → 둘 다 영원히 대기 — 교착 상태 발생

2. 교착 상태의 4가지 필요 조건 (Coffman, 1971)

아래 조건이 동시에 모두 성립할 때만 교착 상태가 발생할 수 있습니다. 하나라도 깨면 교착 상태는 불가능합니다.

#조건설명
1 상호 배제 (Mutual Exclusion) 자원은 한 번에 하나의 프로세스만 사용할 수 있다.
2 점유와 대기 (Hold and Wait) 최소 하나의 자원을 점유한 채로 다른 자원을 기다린다.
3 비선점 (No Preemption) 자원을 강제로 빼앗을 수 없다. 프로세스가 자발적으로 해제해야 한다.
4 순환 대기 (Circular Wait) 프로세스 집합이 원형으로 서로의 자원을 기다린다.
암기 팁 — "상점비순"
상호 배제 → 점유와 대기 → 비선점 → 순환 대기

3. 자원 할당 그래프 (Resource-Allocation Graph)

자원 할당 그래프는 프로세스(원)와 자원(사각형)을 노드로, 할당·요청 관계를 방향 간선으로 표현합니다.

간선 방향의미
자원 → 프로세스 자원이 해당 프로세스에게 할당됨
프로세스 → 자원 프로세스가 해당 자원을 요청(대기) 중
자원 인스턴스가 1개인 경우: 그래프에 사이클 존재 → 교착 상태 확정 자원 인스턴스가 여러 개인 경우: 사이클 존재 → 교착 상태 가능(확정 아님) 사이클 없음 → 교착 상태 없음

그림 2 — 자원 할당 그래프 해석 규칙

4. 교착 상태 처리 전략

전략발생 시점아이디어비용
예방 (Prevention) 사전 4가지 조건 중 하나를 원천 차단 높음 (자원 낭비·성능 저하)
회피 (Avoidance) 사전 안전 상태(Safe State)인 경우에만 자원 할당 중간 (최대 사용량 정보 필요)
탐지 (Detection) 사후 주기적으로 교착 상태 여부 검사 낮음 (검사 오버헤드)
회복 (Recovery) 사후 탐지 후 프로세스 종료 또는 자원 선점 중간 (일부 작업 손실 가능)

5. 교착 상태 예방 (Prevention)

4가지 필요 조건 중 하나를 부정합니다.

깨는 조건방법단점
상호 배제 공유 가능한 자원(읽기 전용 파일 등)은 배제 안 함 모든 자원에 적용 불가
점유와 대기 실행 전 필요한 자원을 모두 미리 요청, 또는 자원 없을 때만 요청 자원 이용률 저하, 기아 가능
비선점 새 자원 요청 실패 시 보유 자원을 모두 반납 CPU/메모리 등 상태 저장이 필요한 자원에 부적합
순환 대기 모든 자원에 전역 번호 부여 후 번호 순서대로만 요청 허용 유연한 자원 사용 제한

6. 교착 상태 회피 — 은행원 알고리즘 (Banker's Algorithm)

Dijkstra(1965)가 제안. OS는 프로세스의 최대 자원 사용량을 미리 알고, 자원 요청 시 할당 후에도 안전 상태(Safe State)가 유지되는지 시뮬레이션합니다.

안전 상태 (Safe State)

모든 프로세스를 교착 상태 없이 완료할 수 있는 실행 순서(안전 순서)가 존재하는 상태. 안전 상태 → 교착 상태 발생 가능성 없음. 불안전 상태 → 교착 상태 발생 가능.

예시: 자원 A 10개, 프로세스 3개

           Max  Allocated  Need  Available
Process 0:  7       0        7      3
Process 1:  3       2        1
Process 2:  9       3        6

안전 순서 탐색:
  Available = 3
  P1의 Need(1) ≤ Available(3) → P1 실행 → 반납 후 Available = 5
  P0의 Need(7) ≤ Available(5)? 아니오
  P2의 Need(6) ≤ Available(5)? 아니오
  P0의 Need(7) ≤ Available(5)? 아직 불가 → 재검사
  ... (P0: 불가, P2: 불가)
  → 안전 순서 없음? 아니요 — P1 → P0 → P2 순서 재탐색 필요

  ※ 실제 계산은 이 단순 예시와 다를 수 있음. 알고리즘 흐름을 이해하는 것이 핵심.

은행원 알고리즘의 핵심 절차:

  1. 자원 요청이 들어오면 가상으로 할당한다.
  2. 안전 순서가 존재하는지 탐색한다 (모든 프로세스의 Need ≤ Available 순서 탐색).
  3. 안전 순서 존재 → 할당 확정. 불안전 → 요청 거부 후 대기.
한계
  • 프로세스 수와 최대 자원 사용량을 미리 알아야 한다.
  • 안전 순서 탐색 시간: O(n² × m) — 프로세스·자원 수 많을수록 비쌈.
  • 실제 범용 OS에서는 거의 사용되지 않음.

7. 교착 상태 탐지 및 회복

7-1. 탐지 (Detection)

OS가 주기적으로 대기 그래프(Wait-for Graph)를 생성해 사이클을 검사합니다. 자원 인스턴스가 여러 개면 은행원 알고리즘과 유사한 탐지 알고리즘(Shoshani-Coffman)을 사용합니다.

대기 그래프 = 자원 할당 그래프에서 자원 노드를 제거한 것 P_i → P_j : P_i가 P_j가 보유한 자원을 대기 중 사이클 탐지 (DFS, O(V+E)) → 교착 상태 확정

그림 3 — 대기 그래프 (Wait-for Graph)

7-2. 회복 (Recovery)

교착 상태가 탐지되면 두 가지 방법 중 하나로 회복합니다.

방법설명비고
프로세스 종료 교착 상태 관련 프로세스를 하나씩 또는 모두 종료 단순하지만 작업 손실 큼
자원 선점 (Preemption) 교착 상태를 깨는 데 필요한 자원을 강제로 빼앗아 다른 프로세스에 할당 체크포인트/롤백 필요

자원 선점 시 고려 사항:

  • 희생자 선택: 최소 비용(남은 작업, 보유 자원 수)으로 선점할 프로세스 선택.
  • 롤백: 선점된 프로세스를 안전 상태로 되돌려야 함 (최근 체크포인트로).
  • 기아 방지: 같은 프로세스가 반복 희생되지 않도록 카운터 관리.
