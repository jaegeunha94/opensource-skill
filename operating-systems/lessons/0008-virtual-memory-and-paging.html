가상 메모리와 페이징

Day 7에서 연속 메모리 할당의 외부 단편화 문제를 확인했습니다. 현대 OS는 이를 가상 메모리(Virtual Memory)와 페이징(Paging)으로 해결합니다. 오늘은 페이지 테이블의 구조, TLB의 역할, 그리고 요구 페이징의 동작 원리를 배웁니다.

학습 목표
  • 가상 메모리가 왜 필요한지, 어떤 문제를 해결하는지 설명할 수 있다.
  • 페이징의 기본 구조(페이지, 프레임, 페이지 테이블)를 그림으로 설명할 수 있다.
  • TLB(Translation Lookaside Buffer)가 주소 변환 성능을 어떻게 개선하는지 이해한다.
  • 요구 페이징(Demand Paging)과 페이지 폴트(Page Fault) 처리 과정을 순서대로 말할 수 있다.

1. 가상 메모리란?

프로세스는 자신이 메모리 전체를 독점한다는 환상(illusion) 속에서 실행됩니다. 실제 물리 메모리가 얼마나 있든 상관없이, 각 프로세스는 0번 주소부터 시작하는 넓은 가상 주소 공간(virtual address space)을 갖습니다.

프로세스 A 가상 공간 프로세스 B 가상 공간 ┌─────────────────┐ ┌─────────────────┐ │ 0xFFFF… 스택 │ │ 0xFFFF… 스택 │ │ ↓ │ │ ↓ │ │ (빈 공간) │ │ (빈 공간) │ │ ↑ │ │ ↑ │ │ 힙 │ │ 힙 │ │ BSS / 데이터 │ │ BSS / 데이터 │ │ 0x0000 텍스트 │ │ 0x0000 텍스트 │ └─────────────────┘ └─────────────────┘ │ MMU (페이지 테이블) │ └───────────┬──────────────────┘ ↓ ┌────────────────────┐ │ 물리 메모리 (RAM) │ │ (두 프로세스가 공유) │ └────────────────────┘

그림 1 — 각 프로세스는 독립된 가상 주소 공간을 가진다

가상 메모리의 핵심 이점
  • 격리(Isolation): 프로세스 A가 프로세스 B의 메모리를 직접 읽거나 쓸 수 없다.
  • 용량 초과 수용: 물리 메모리보다 큰 프로그램도 실행 가능하다 (디스크를 보조 메모리로 활용).
  • 공유 메모리: 공유 라이브러리 등 같은 물리 페이지를 여러 프로세스가 매핑할 수 있다.

2. 페이징(Paging)의 기본 구조

페이징은 가상 주소 공간과 물리 메모리를 같은 크기의 고정 단위로 나누어 관리합니다.

용어설명
페이지 (Page)가상 주소 공간을 나눈 고정 크기 단위 (예: 4 KB)
프레임 (Frame)물리 메모리를 나눈 같은 크기의 단위 (예: 4 KB)
페이지 테이블 (Page Table)페이지 번호 → 프레임 번호 매핑 테이블; 프로세스마다 하나씩 존재
페이지 크기 (Page Size)보통 4 KB ~ 4 MB; 2의 거듭제곱이어야 오프셋 계산이 단순해짐

2-1. 가상 주소의 구성

CPU가 생성하는 가상 주소는 두 필드로 나뉩니다.

가상 주소 (32비트, 페이지 크기 4 KB = 2¹²) ┌────────────────────────┬───────────────┐ │ 페이지 번호 (VPN) │ 오프셋 │ │ 상위 20비트 │ 하위 12비트 │ └────────────────────────┴───────────────┘ 페이지 테이블 인덱스 페이지 내 위치

그림 2 — 32비트 가상 주소의 페이지 번호와 오프셋 분리

2-2. 주소 변환 과정

1. CPU가 가상 주소 0x00002ABC를 생성 → VPN = 0x00002, offset = 0xABC 2. MMU가 페이지 테이블에서 VPN 2 항목 조회 → 프레임 번호 (PFN) = 7 3. 물리 주소 = PFN × 페이지크기 + offset = 7 × 4096 + 0xABC = 0x007ABC

그림 3 — 페이지 번호(VPN)로 프레임 번호(PFN)를 찾고 오프셋을 합친다

외부 단편화가 없는 이유

페이지는 고정 크기이므로 연속되지 않은 프레임들에 나누어 배치할 수 있습니다. 따라서 연속 할당에서 발생하던 외부 단편화가 원칙적으로 사라집니다.
내부 단편화는 마지막 페이지에서 미세하게 발생할 수 있지만 페이지 크기만큼만 최대 낭비됩니다.

3. 페이지 테이블 항목(PTE)의 구성

각 페이지 테이블 항목(Page Table Entry, PTE)은 프레임 번호 외에 여러 제어 비트를 포함합니다.

비트이름역할
VValid(Present) bit1이면 물리 메모리에 실제 존재, 0이면 페이지 폴트 트리거
R/WRead/Write bit쓰기 허용 여부; 0이면 쓰기 시 보호 오류 발생
U/SUser/Supervisor사용자 모드 프로세스의 접근 허용 여부
DDirty bit페이지 내용이 수정된 경우 1; 페이지 교체 시 디스크에 저장 여부 판단
AAccessed bit최근에 참조된 경우 1; LRU 근사 알고리즘에 사용

4. TLB — 주소 변환 캐시

페이지 테이블은 메모리(RAM)에 저장됩니다. 주소 변환마다 메모리를 한 번 더 읽어야 하므로 성능이 반 토막납니다. 이를 해결하기 위해 TLB(Translation Lookaside Buffer)라는 빠른 하드웨어 캐시를 CPU 내부에 둡니다.

CPU → 가상 주소 VPN 발생 │ ▼ ┌─────────────────────────────────────┐ │ TLB (CPU 내부, 수십~수백 항목) │ │ VPN → PFN 쌍을 캐싱 │ └──────────────┬──────────────────────┘ TLB Hit ◄──┤──► TLB Miss (수 ns) │ │ │ ▼ │ 메모리 내 페이지 테이블 조회 │ (수십~수백 ns) │ │ │ TLB에 항목 추가 │ │ └───────────┘ PFN 확보 → 물리 주소 계산

그림 4 — TLB Hit 시 메모리 접근 없이 즉시 PFN 획득

TLB 적중률(Hit Rate)과 실효 접근 시간

TLB 적중률이 높을수록 평균 메모리 접근 시간이 줄어듭니다.

실효 접근 시간(EAT) 계산 예시

TLB 접근 시간 = 20 ns, 메모리 접근 시간 = 100 ns, TLB 적중률 = 90%

EAT = 0.90 × (20 + 100) + 0.10 × (20 + 100 + 100)
    = 0.90 × 120 + 0.10 × 220
    = 108 + 22
    = 130 ns

TLB가 없을 때 200 ns → TLB 도입으로 약 35% 단축.

TLB 무효화(TLB Flush)

컨텍스트 스위치 시 다른 프로세스의 TLB 항목이 남아있으면 잘못된 변환이 발생합니다. OS는 컨텍스트 스위치마다 TLB를 전체 무효화하거나, ASID(Address Space Identifier)를 PTE에 태깅하여 선택적 무효화를 수행합니다.

5. 요구 페이징(Demand Paging)

프로세스를 실행할 때 전체 프로그램을 메모리에 올리지 않고, 실제로 필요한 페이지만 그때그때 적재하는 전략입니다. 나머지 페이지는 디스크(스왑 영역)에 머뭅니다.

5-1. 페이지 폴트(Page Fault) 처리 과정

유효 비트(V)가 0인 페이지에 접근하면 하드웨어가 페이지 폴트 인터럽트를 발생시킵니다.

1. CPU → PTE의 V=0 감지 → 페이지 폴트 인터럽트 2. OS 페이지 폴트 핸들러 실행 ├─ 페이지가 합법적 접근인지 확인 (불법이면 → SIGSEGV 종료) ├─ 빈 프레임 확보 (없으면 → 페이지 교체 알고리즘 실행) ├─ 디스크(스왑)에서 해당 페이지 읽기 (I/O, 느림!) └─ PTE 갱신: PFN 기록, V←1 3. 인터럽트를 발생시킨 명령 재실행 4. 이번엔 TLB Hit or PTE V=1 → 정상 실행

그림 5 — 페이지 폴트 처리 6단계

스래싱(Thrashing) 위험

너무 많은 프로세스가 동시에 실행되어 물리 메모리가 부족해지면, OS가 페이지 폴트를 처리하느라 유용한 작업보다 페이지 교체에 대부분의 시간을 씁니다. 이 현상을 스래싱이라 하며, CPU 이용률이 급격히 떨어집니다. → 해결책: 워킹셋(Working Set) 모델로 적정 프로세스 수를 제한하거나, 물리 메모리를 증설합니다.

6. 페이지 테이블의 크기 문제와 해결책

32비트 주소 공간에서 페이지 크기가 4 KB이면 페이지 테이블 항목이 2²⁰ ≈ 100만 개가 필요합니다. 항목당 4바이트면 프로세스 하나당 4 MB. 64비트에서는 수십 TB에 달합니다. 이를 해결하는 기술이 있습니다.

기법핵심 아이디어
계층적 페이지 테이블
(Multi-level Page Table)		 페이지 테이블 자체를 다시 페이지 단위로 나눈다. 사용 중인 영역의 테이블만 메모리에 상주시켜 공간 절약. (x86-64는 4단계 사용)
역방향 페이지 테이블
(Inverted Page Table)		 물리 프레임마다 항목 하나 (PID, VPN). 항목 수 = 물리 프레임 수. 공간은 획기적으로 줄지만, 조회 시 해시가 필요.
세그멘테이션 + 페이징
(Intel x86 과거 방식)		 먼저 세그먼트로 영역을 나눈 후 각 세그먼트 내에서 페이징 적용. 복잡성이 높아 현대 64비트 OS에서는 페이징 단독으로 대체.
