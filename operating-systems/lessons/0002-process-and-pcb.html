1. 프로세스란 무엇인가?

프로그램(Program)은 디스크에 저장된 수동적인 명령어 묶음입니다. 프로세스(Process)는 그 프로그램이 메모리에 적재되어 실행 중인 상태입니다. 같은 프로그램을 두 번 실행하면 두 개의 별도 프로세스가 생성됩니다.

핵심 정의
프로세스 = 실행 중인 프로그램 + 그 프로그램이 사용하는 자원 (메모리, 파일 핸들, CPU 레지스터 상태 등)

프로세스의 메모리 구조

┌───────────────────────────────────┐ 높은 주소 (High Address) │ Stack │ ← 함수 호출 프레임, 지역 변수 │ (↓ grows down) │ ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤ │ │ ← 빈 공간 (Stack ↔ Heap 사이) ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤ │ Heap │ ← 동적 메모리 (malloc / new) │ (↑ grows up) │ ├───────────────────────────────────┤ │ BSS │ ← 초기화되지 않은 전역/정적 변수 ├───────────────────────────────────┤ │ Data │ ← 초기화된 전역/정적 변수 ├───────────────────────────────────┤ │ Text (Code) │ ← 실행 가능한 명령어 (읽기 전용) └───────────────────────────────────┘ 낮은 주소 (Low Address)

그림 1: 프로세스의 가상 주소 공간 레이아웃

2. 프로세스 상태(Process State)와 전이

OS는 각 프로세스를 항상 다섯 가지 상태 중 하나로 분류합니다.

fork() │ ▼ ┌──────────────┐ │ New │ 생성됨, 초기화 중 └──────┬───────┘ │ admitted (OS가 메모리 할당) ▼ ┌──────────────┐ 스케줄러 선택 │ Ready │ ◄──────────────────────┐ └──────┬───────┘ │ │ scheduler dispatch │ ▼ │ ┌──────────────┐ I/O 또는 이벤트 요청 │ │ Running │ ──────────────────► ┌──┴───────┐ └──────┬───────┘ │ Waiting │ │ I/O 완료 ◄─────────────────└──────────┘ │ (interrupt) │ exit() / return ▼ ┌──────────────┐ │ Terminated │ 자원 해제, 부모 프로세스가 수거 └──────────────┘

그림 2: 프로세스 상태 전이 다이어그램

상태 설명 전이 조건
New 프로세스가 생성 중. OS가 PCB를 초기화하고 자원을 할당 → Ready: OS가 승인(admit)
Ready CPU를 할당받기를 기다리는 상태. Ready Queue에 대기 → Running: 스케줄러가 선택
Running 현재 CPU에서 명령어를 실행 중. 단일 CPU 시스템에서 한 번에 하나만 가능 → Waiting: I/O 요청
→ Ready: 타임아웃(선점)
→ Terminated: 종료
Waiting (Blocked) I/O 완료, 이벤트 신호 등 외부 조건을 기다리는 상태. CPU를 소비하지 않음 → Ready: I/O 완료 또는 이벤트 발생
Terminated 실행 완료 또는 강제 종료. OS가 자원 회수. 부모가 wait()으로 수거 전까지 Zombie로 잔존 종단 상태 (전이 없음)
Zombie vs Orphan 프로세스
Zombie: 종료했지만 부모가 wait()을 호출하지 않아 PCB가 아직 커널에 남아 있는 상태.
Orphan: 부모 프로세스가 먼저 종료되어 고아가 된 프로세스. Linux에서는 init(PID 1)이 입양하여 수거합니다.

3. PCB (Process Control Block)

OS가 프로세스 하나를 관리하기 위해 커널 메모리에 유지하는 자료구조입니다. 컨텍스트 스위치, 스케줄링, 자원 추적에 필요한 모든 정보가 담겨 있습니다.

┌─────────────────────────────────┐ │ PCB (per process) │ ├─────────────────────────────────┤ │ Process ID (PID) │ ← 고유 식별자 │ Process State │ ← New / Ready / Running / … │ Program Counter (PC) │ ← 다음 실행할 명령어 주소 │ CPU Registers │ ← AX, BX, SP, BP … 일체 │ Memory Management Info │ ← 페이지 테이블 포인터, 베이스/리밋 │ Scheduling Info │ ← 우선순위, CPU 사용 시간, 대기 시간 │ I/O Status │ ← 열린 파일 목록, 장치 목록 │ Accounting Info │ ← 사용자 ID, 그룹 ID, 누적 CPU 시간 │ Parent PID (PPID) │ ← fork() 한 부모 프로세스 PID └─────────────────────────────────┘

그림 3: PCB에 저장되는 주요 필드

Linux에서 PCB는 task_struct라는 구조체로 구현되어 있습니다. 커널 소스 include/linux/sched.h에서 수백 개의 필드를 확인할 수 있습니다. 프로세스 목록은 이중 연결 리스트로 연결되며, 스케줄러가 이를 탐색하여 다음 실행할 프로세스를 결정합니다.

4. 컨텍스트 스위치 (Context Switch)

CPU가 실행 중인 프로세스를 바꿀 때, 현재 프로세스의 상태를 PCB에 저장하고 다음 프로세스의 상태를 PCB에서 복원하는 과정입니다.

프로세스 A (Running) OS 커널 프로세스 B (Ready) │ │ │ │ 타임아웃 or 시스콜 │ │ │ ──────────────────────►│ │ │ │ A의 CPU 상태를 PCB-A에 저장 │ │ (PC, 레지스터, 스택 포인터) │ │ │ │ │ PCB-B에서 B의 상태 복원│ │ │ ──────────────────────►│ │ │ │ (Running)

그림 4: 컨텍스트 스위치 흐름

컨텍스트 스위치 오버헤드

스위치 중에는 유용한 작업이 전혀 수행되지 않습니다 — 순수한 오버헤드입니다. 일반적으로 수 마이크로초(μs) 수준이지만, 캐시(TLB, L1/L2) 무효화까지 포함하면 실질적인 성능 손실이 수십~수백 μs에 달할 수 있습니다.

원인 설명
타임슬라이스 만료 스케줄러가 할당한 CPU 시간이 끝남 (타이머 인터럽트)
I/O 시스템 콜 프로세스가 느린 작업을 요청 → Waiting 상태로 전환
더 높은 우선순위 프로세스 도착 선점형(preemptive) 스케줄러가 강제로 교체
yield() 호출 프로세스가 자발적으로 CPU를 양보

5. 프로세스 생성과 종료

fork() / exec() 패턴

/* Unix/Linux 프로세스 생성 */
pid_t pid = fork();   // 현재 프로세스를 복제 → 자식 프로세스 생성

if (pid == 0) {
    // 자식 프로세스
    execv("/bin/ls", args);  // 새 프로그램으로 교체 (PCB 재사용)
} else {
    // 부모 프로세스
    wait(NULL);  // 자식 종료까지 대기 (Zombie 방지)
}

fork(): 부모 프로세스의 주소 공간, PCB, 열린 파일 등을 복사하여 자식 프로세스를 생성합니다. 실제 메모리 복사는 Copy-on-Write(CoW)로 지연됩니다 — 쓰기가 발생할 때만 복사합니다.
exec(): 현재 프로세스의 주소 공간을 새 프로그램으로 덮어씁니다. PID는 유지됩니다.
