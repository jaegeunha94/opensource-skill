1. 스레드란 무엇인가?

스레드(Thread)는 프로세스 안에서 실제로 실행되는 흐름의 단위입니다. 하나의 프로세스는 반드시 하나 이상의 스레드를 포함합니다. 프로세스가 "작업의 컨테이너"라면, 스레드는 그 컨테이너 안에서 실제로 일을 하는 "작업자"입니다.

핵심 정의
스레드 = CPU 실행의 기본 단위 = 프로세스 내부의 독립적인 제어 흐름 (독립적인 PC, 스택, 레지스터 세트를 가짐)

단일 스레드 vs 멀티 스레드 프로세스

단일 스레드 프로세스 멀티 스레드 프로세스 ┌──────────────────────┐ ┌──────────────────────┐ │ Code │ │ Code (공유) │ │ Data / Heap (공유) │ │ Data / Heap (공유) │ ├──────────────────────┤ ├──────────┬───────────┤ │ Stack │ │ Stack 1 │ Stack 2 │ ← 각 스레드 독립 │ Registers │ │ Regs 1 │ Regs 2 │ │ PC │ │ PC 1 │ PC 2 │ └──────────────────────┘ └──────────┴───────────┘ 스레드 1개 스레드 2개

2. 스레드가 공유하는 것 vs 독립적으로 소유하는 것

같은 프로세스 내 스레드들은 프로세스의 자원을 공유합니다. 하지만 각 스레드마다 독립적으로 가지는 자원도 있습니다.

구분 항목 이유
공유 (Shared) 코드 영역 (Text) 모든 스레드가 같은 프로그램 코드를 실행
데이터 영역 (전역·정적 변수) 프로세스 전역으로 할당된 메모리
힙 (Heap) 동적으로 할당된 메모리는 모든 스레드가 접근 가능
열린 파일 디스크립터 프로세스 단위로 관리
시그널 핸들러 프로세스 전체에 적용
독립 (Private) 스택 (Stack) 각 스레드의 함수 호출 프레임·지역 변수가 다름
프로그램 카운터 (PC) 각 스레드가 다른 코드 지점을 실행 중일 수 있음
CPU 레지스터 세트 컨텍스트 스위치 시 각 스레드의 레지스터를 따로 저장·복원
스레드 ID (TID) OS가 스레드를 개별 식별하기 위해 부여
주의 — 공유가 양날의 검
힙과 전역 변수를 공유하므로 스레드 간 데이터 교환이 빠르지만, 여러 스레드가 동시에 같은 데이터를 수정하면 경쟁 조건(Race Condition)이 발생합니다. 동기화 없이 공유 데이터에 접근하는 것은 위험합니다. (Day 5에서 자세히 다룹니다.)

3. 멀티스레딩(Multithreading)의 이점과 위험

이점

  • 응답성(Responsiveness): UI 스레드와 작업 스레드를 분리하면 긴 작업 중에도 화면이 멈추지 않습니다.
  • 자원 공유(Resource Sharing): 새 프로세스를 생성하지 않고 스레드를 추가하면 메모리·파일 핸들 등을 재사용합니다.
  • 경제성(Economy): 스레드 생성·컨텍스트 스위치 비용이 프로세스보다 훨씬 저렴합니다. (Linux 기준 clone() vs fork())
  • 확장성(Scalability): 멀티코어 CPU에서 여러 스레드를 병렬 실행하면 처리량이 증가합니다.

위험

  • 경쟁 조건(Race Condition): 공유 데이터를 보호하지 않으면 예측 불가능한 결과가 발생합니다.
  • 교착 상태(Deadlock): 스레드들이 서로의 잠금을 기다리며 영원히 멈출 수 있습니다.
  • 디버깅 어려움: 스레드 스케줄링은 비결정적이어서 재현이 어려운 버그가 생깁니다.

4. 커널 레벨 스레드 vs 사용자 레벨 스레드

스레드를 누가 관리하느냐에 따라 두 종류로 나뉩니다.

항목 커널 레벨 스레드 (KLT) 사용자 레벨 스레드 (ULT)
관리 주체 OS 커널 사용자 공간 라이브러리
OS 인식 OS가 각 스레드를 개별 스케줄 단위로 인식 OS는 프로세스만 보고, 스레드 존재를 모름
블로킹 시스콜 한 스레드가 블로킹되어도 다른 스레드 실행 가능 한 스레드가 블로킹되면 프로세스 전체가 블로킹
멀티코어 활용 가능 (여러 코어에서 병렬 실행) 불가 (단일 프로세스는 하나의 코어만 사용)
스위치 비용 높음 (커널 모드 전환 필요) 낮음 (사용자 공간 내 컨텍스트 스위치)
예시 Linux pthreads, Windows Threads 초기 Java Green Threads, GNU Portable Threads
커널 레벨 스레드 (1:1 모델) 사용자 레벨 스레드 (N:1 모델) 사용자 T1 T2 T3 사용자 T1 T2 T3 공간 │ │ │ 공간 └───┴───┘ │ │ │ │ ---------kernel boundary-------- --------kernel boundary-------- 커널 │ │ │ 커널 │ 공간 KT1 KT2 KT3 KT1 (각 ULT가 KLT에 1:1 매핑) (모든 ULT가 단일 KLT에 N:1 매핑)

5. 스레딩 모델 — 1:1, N:1, M:N

모델 매핑 특징 사용 예
1:1 ULT 1개 → KLT 1개 멀티코어 활용 가능, 블로킹 안전. 스레드 수만큼 커널 자원 소모 Linux pthreads, Windows
N:1 ULT N개 → KLT 1개 전환 비용 낮음. 블로킹 시 전체 프로세스 멈춤. 멀티코어 미활용 초기 Solaris, Green Threads
M:N ULT M개 → KLT N개 (M≥N) 유연성 최대. 구현이 매우 복잡. 일부 언어 런타임(Go 고루틴)이 채택 Go 런타임, 초기 Solaris, Windows UMS
현대 OS의 선택
Linux와 Windows는 모두 1:1 모델을 채택합니다. 커널이 강력해져 오버헤드가 줄었고, 멀티코어 활용이 중요해졌기 때문입니다. Go 언어는 자체 런타임에서 M:N 모델을 구현하여 수백만 고루틴을 소수의 OS 스레드에 스케줄합니다.

6. 실전 예시 — POSIX Threads (pthreads)

#include <pthread.h>
#include <stdio.h>

void *worker(void *arg) {
    int id = *(int *)arg;
    printf("스레드 %d 실행 중\n", id);
    return NULL;
}

int main(void) {
    pthread_t t1, t2;
    int id1 = 1, id2 = 2;

    pthread_create(&t1, NULL, worker, &id1);  // 스레드 생성
    pthread_create(&t2, NULL, worker, &id2);

    pthread_join(t1, NULL);  // 스레드 종료 대기 (프로세스의 wait()와 유사)
    pthread_join(t2, NULL);

    printf("모든 스레드 완료\n");
    return 0;
}

pthread_create(): 새 스레드를 생성하고 지정한 함수를 실행합니다. Linux에서는 내부적으로 clone() 시스템 콜을 사용합니다.
pthread_join(): 대상 스레드가 종료될 때까지 현재 스레드를 블로킹합니다. 프로세스의 wait()와 동일한 역할입니다.

프로세스 생성 vs 스레드 생성 비용 비교
Linux 기준 fork()(프로세스 생성)는 pthread_create()(스레드 생성)보다 수십~수백 배 더 느립니다. 스레드는 코드·데이터·힙을 새로 복사하지 않기 때문입니다. 단, fork()도 CoW(Copy-on-Write) 덕분에 실제 메모리 복사는 지연됩니다.
