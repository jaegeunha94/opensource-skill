전제 개념

캐시가 존재하는 이유부터 이해하기

캐시란 무엇인가?

캐시(Cache)는 느린 저장소의 데이터를 빠른 저장소에 임시로 복사해두는 기술이다. 예를 들어 DB에서 상품 목록을 조회하면 디스크 I/O가 발생하고 네트워크도 탄다. 같은 상품 목록을 초당 1,000번 조회하면 DB에 1,000번의 쿼리가 나간다. 캐시를 쓰면 첫 번째 조회 때만 DB에서 가져오고, 이후 999번은 메모리에서 바로 돌려준다.

핵심 전제: 캐시가 효과를 내려면 두 가지 조건이 필요하다.
1. 같은 데이터를 반복해서 읽는다 (읽기 중심 패턴)
2. 약간 오래된 데이터를 보여줘도 괜찮다 (일관성 허용 범위)

캐시 히트(Cache Hit)와 캐시 미스(Cache Miss)

캐시에서 데이터를 찾으면 캐시 히트, 없으면 캐시 미스라고 한다. 캐시 히트가 많을수록 DB 부하가 줄어든다. 캐시 히트율이 낮은데 캐시를 도입하면 오히려 캐시 레이어만 추가된 복잡한 시스템이 된다.

주요 캐시 전략 3가지

① Cache-aside (Lazy Loading)

앱이 데이터를 읽을 때, 먼저 캐시를 확인한다. 없으면(캐시 미스) DB에서 읽어온 다음 캐시에 저장하고 반환한다. 가장 많이 쓰이는 패턴이다. 

# Cache-aside 흐름
def get_product_list(category_id):
    cache_key = f"products:{category_id}"

    # 1. 캐시 먼저 확인
    cached = redis.get(cache_key)
    if cached:
        return json.loads(cached)  # 캐시 히트

    # 2. 캐시 미스 → DB 조회
    products = db.query("SELECT * FROM products WHERE category=?", category_id)

    # 3. 캐시에 저장 (TTL 5분)
    redis.setex(cache_key, 300, json.dumps(products))

    return products
장점 단점 언제 쓰나
실제 읽는 데이터만 캐시됨 (메모리 효율) 첫 요청은 항상 느림 (cold start) 읽기 중심, 다양한 키가 산발적으로 조회될 때
DB 장애 시 캐시로 일부 서빙 가능 캐시와 DB 간 일시적 불일치 발생 가능 쇼핑몰 상품 목록, 뉴스 피드, 설정값

② Write-through

데이터를 저장할 때 캐시와 DB를 동시에 업데이트한다. 항상 캐시와 DB가 동기화되어 있어서 읽기 시 캐시 미스가 거의 없다. 

# Write-through 흐름
def update_product(product_id, data):
    # 1. DB 먼저 저장
    db.update("UPDATE products SET ... WHERE id=?", product_id, data)

    # 2. 캐시도 즉시 갱신
    cache_key = f"product:{product_id}"
    redis.setex(cache_key, 300, json.dumps(data))
장점 단점 언제 쓰나
데이터 일관성 보장 쓸 때마다 캐시도 갱신 → 쓰기 비용 증가 자주 읽고 가끔 쓰는 데이터
캐시 미스 발생 확률 낮음 읽히지 않는 데이터도 캐시에 올라감 세션, 사용자 프로필, 설정값

③ Write-behind (Write-back)

데이터를 캐시에만 먼저 쓰고, 일정 시간 후에 비동기로 DB에 반영한다. 쓰기 성능이 매우 높지만, 데이터 유실 위험이 있다.

주의: Write-behind는 Redis가 다운되면 캐시에만 있던 데이터가 유실될 수 있다. 금융 거래, 주문 등 데이터 손실이 용납되지 않는 영역에는 절대 사용하지 않는다.
장점 단점 언제 쓰나
쓰기 성능 극대화 Redis 장애 시 데이터 유실 가능 조회수, 좋아요 수, 게임 점수 (비중요 집계)
DB 부하 크게 감소 구현 복잡도 높음 유실 허용 가능한 임시 집계
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "Redis 명령어를 아느냐"가 아니라 "성능 문제를 어떻게 구조화해서 해결하느냐"를 본다.

  • 판단 근거가 있는가: "느리니까 캐시 붙인다"가 아니라, 읽기 빈도·데이터 변경 주기·허용 TTL을 따져서 결정하는가
  • 부작용을 아는가: 캐시를 붙이면 데이터 일관성 문제, cold start, 캐시 스탬피드 등의 새로운 문제가 생긴다는 것을 아는가
  • 운영을 생각하는가: 히트율 모니터링, 메모리 한도, 장애 시 동작을 설계에 포함하는가
  • Trade-off를 언어화하는가: "A를 선택하면 B가 희생된다"를 명확히 설명할 수 있는가
Senior 개발자는 "캐시를 쓴다/안 쓴다" 수준이 아니라, 어떤 캐시 전략을 / 어느 데이터에 / 어떤 TTL로 / 무효화는 어떻게 / 장애 시엔 어떻게를 한 번에 설명할 수 있어야 한다.
실무 연결

실제 상황과 연결하기

상황 1: 쇼핑몰 상품 목록 API (일 PV 1,000만)

상품 목록은 하루에도 수십 번 업데이트되지만, 조회는 수백만 번이다. 10초~1분 오래된 데이터를 보여줘도 사용자가 눈치채지 못한다. → Cache-aside + TTL 60초가 적합하다.

상황 2: 실시간 재고 수량

재고가 0인데 1로 보여서 주문이 들어오면 문제가 생긴다. 일관성이 매우 중요하다. → 캐시 미사용 또는 Write-through + 짧은 TTL. 혹은 캐시는 쓰되 재고 감소 시 즉시 무효화하는 전략을 쓴다.

상황 3: 사용자 세션

로그인 후 세션 데이터를 매 요청마다 DB에서 읽으면 부하가 크다. 세션은 자주 읽히고, 로그아웃 시에만 지우면 된다. → Write-through + TTL 30분 (세션 만료 시간과 맞춤).

상황 4: 오늘의 조회수 집계

게시글 조회수를 매 조회마다 DB에 UPDATE하면 부하가 크다. 조회수는 약간 부정확해도 허용된다. → Write-behind: Redis에 INCR로 카운트 후, 배치로 DB에 저장.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
캐시 도입 여부 판단 기준 먼저 말하기 "저는 세 가지를 먼저 봅니다: 읽기 빈도가 높은지, 데이터 변경 주기가 느린지, 약간 오래된 데이터를 보여줘도 되는지입니다."
2
지금 상황에 대입하기 "상품 목록은 하루 10번 업데이트되지만 초당 수천 번 조회됩니다. TTL 60초 정도의 불일치는 허용되므로 캐시를 도입하기 좋은 케이스입니다."
3
캐시 전략 선택 및 이유 설명 "Cache-aside 전략을 쓰겠습니다. 여러 카테고리가 있고 카테고리별로 읽히는 빈도가 다르기 때문에, 실제로 조회되는 데이터만 캐시에 올리는 게 메모리 효율이 높습니다."
4
TTL과 무효화 전략 언급 "TTL은 60초로 설정하고, 상품이 업데이트되면 해당 카테고리 캐시 키를 즉시 삭제(del)해서 다음 요청 시 최신 데이터를 DB에서 가져오게 합니다."
5
장애 시나리오 언급 (Senior 포인트) "Redis가 다운되면 캐시 미스가 발생하고 모든 요청이 DB로 직접 가게 됩니다. 이를 대비해서 Circuit Breaker 패턴이나 fallback 로직을 앱 레이어에 추가하거나, Redis Sentinel 구성으로 장애 시 자동 페일오버를 준비하겠습니다."
6
모니터링 언급 (운영 관점) "도입 후엔 히트율(hit ratio)을 모니터링합니다. 히트율이 80% 이하라면 TTL이 너무 짧거나 키 설계가 잘못된 것이므로 재검토합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"캐시 도입 전에 저는 세 가지를 먼저 확인합니다. 첫째, 같은 데이터를 반복해서 읽는지(읽기 편향), 둘째, 데이터 변경 주기가 조회 빈도보다 훨씬 낮은지, 셋째, 수 초~수 분 오래된 데이터를 보여줘도 비즈니스상 괜찮은지입니다.

상품 목록의 경우, 조회는 초당 수천 번이지만 업데이트는 하루 수십 번이고, 1분 전 데이터를 보여줘도 사용자 경험에 영향이 없습니다. 세 조건 모두 충족하므로 캐시 적합 케이스입니다.

전략은 Cache-aside를 선택합니다. 카테고리별 트래픽이 달라서 인기 있는 카테고리만 메모리에 올리는 게 효율적이고, DB 장애 시에도 캐시에 데이터가 남아 있으면 일부 서빙이 가능합니다. TTL은 60초, 상품 변경 시 해당 키를 즉시 삭제(Event-based invalidation)하는 방식으로 무효화합니다.

Redis 다운 시를 대비해서 앱 레이어에서 캐시 오류를 잡아 DB로 fallback하는 로직을 넣고, 운영 중에는 keyspace hit ratio를 지표로 모니터링합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Redis가 다운되면 어떻게 됩니까?"

약한 답변: "서비스가 느려집니다."

강한 답변: "Cache-aside에서 Redis가 다운되면 캐시 미스가 100%가 되어 모든 요청이 DB로 직접 가게 됩니다. DB가 이 부하를 감당하면 서비스는 느려지지만 동작합니다. 감당 못 하면 DB도 다운되는 cascade failure가 발생합니다. 이를 막기 위해 세 가지를 준비합니다: (1) Redis 연결 오류 시 캐시 없이 DB를 직접 조회하는 fallback 경로, (2) DB Connection Pool 한도를 적절히 제한해서 burst를 흡수, (3) Redis Sentinel 또는 Cluster 구성으로 장애 시 자동 페일오버."

F2. "캐시 히트율이 40%밖에 안 나오면 어떻게 하시겠어요?"

약한 답변: "TTL을 늘리겠습니다."

강한 답변: "히트율이 낮은 원인부터 파악합니다. 주로 세 가지입니다: (1) 키 설계 문제: 캐시 키가 너무 세분화되어 있어 같은 데이터도 다른 키로 조회되는 경우. 예를 들어 정렬 기준이나 페이지 번호가 키에 포함되면 히트율이 낮아집니다. (2) TTL이 너무 짧음: 데이터 변경 주기보다 TTL이 짧으면 자주 만료됩니다. (3) 데이터가 캐시 대상이 아님: 사용자마다 다른 개인화 데이터는 공유 캐시 효과가 없습니다. 진단 후 원인에 맞게 키 구조 개선, TTL 조정, 또는 캐시 대상에서 제외하는 결정을 합니다."

F3. "캐시를 쓰면 안 되는 경우는 무엇인가요?"

강한 답변: "세 가지 경우에는 캐시가 오히려 독이 됩니다. 첫째, 높은 일관성이 필요한 데이터: 결제 잔액, 재고 수량처럼 조금이라도 틀리면 비즈니스 오류가 나는 데이터. 둘째, 쓰기가 읽기보다 많은 데이터: 캐시 무효화가 너무 자주 일어나서 히트율이 0에 가까움. 셋째, 캐시 미스 시 DB 부하가 캐시 없을 때보다 클 경우: 캐시 스탬피드(한꺼번에 만료) 상황에서는 DB에 트래픽이 한 번에 몰려 장애로 이어질 수 있습니다."

F4. "Redis와 Memcached 중 무엇을 선택하시겠어요?"

강한 답변: "단순 키-값 캐싱만 필요하면 Memcached가 더 빠르고 가볍습니다. 하지만 실무에서 Redis를 주로 선택하는 이유는 세 가지입니다: (1) 자료구조 다양성: List, Hash, Set, Sorted Set을 활용한 랭킹, 세션, 분산 락 구현 가능, (2) Persistence 옵션: RDB/AOF로 재시작 후 데이터 복구 가능, (3) Pub/Sub과 Lua 스크립팅: 단순 캐시 이상의 용도로 확장 가능. 프로젝트 요구사항이 정말 단순한 캐시라면 Memcached도 좋지만, 대부분의 서비스에서는 Redis의 확장성이 장기적으로 유리합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 도입 판단 기준을 먼저 말한다 ("세 가지를 확인합니다")
  • 전략 선택에 이유가 있다 ("트래픽 분포가 다르므로 Cache-aside")
  • TTL을 임의로 정하지 않고 데이터 변경 주기와 비즈니스 허용치로 설명한다
  • 장애 시나리오를 먼저 생각한다
  • 도입 후 히트율 모니터링을 언급한다
  • 무효화 전략을 구체적으로 말한다 (TTL + event-based)

약한 답변의 특징

  • "느리니까 Redis 붙이면 됩니다" — 판단 기준 없음
  • "TTL을 적당히 설정합니다" — 근거 없는 설정
  • 캐시 전략 이름을 몰라서 설명이 두루뭉술함
  • Redis 다운 시 어떻게 되는지 생각 안 함
  • 캐시 무효화를 TTL에만 의존 ("알아서 만료됩니다")
  • 메모리 한도나 eviction을 전혀 언급 안 함
핵심 차이: 강한 답변은 "Redis를 안다"가 아니라 "문제를 어떻게 분석하고 설계 결정을 내리는가"를 보여준다. 면접관은 Redis 명령어 전문가를 찾는 게 아니라, 시스템 설계 판단력을 가진 사람을 찾는다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Redis는 DB 앞에 항상 붙이면 좋다"

사실: 캐시 히트율이 낮거나 데이터 일관성이 중요한 케이스에서는 오히려 복잡도만 늘어난다. 캐시를 잘못 쓰면 사용자에게 잘못된 데이터를 보여주는 버그의 원인이 된다.

오해 2: "TTL만 잘 설정하면 무효화는 자동으로 된다"

사실: TTL은 최후의 안전망이지 주 무효화 전략이 아니다. 데이터가 변경될 때 해당 캐시 키를 명시적으로 삭제하거나 갱신하지 않으면, TTL이 만료되기 전까지 잘못된 데이터가 계속 서빙된다.

오해 3: "Redis가 빠르니까 DB 대신 Redis만 써도 된다"

사실: Redis는 메모리에 데이터를 저장한다. 메모리는 비싸고 한정적이다. 또한 기본 설정에서는 재시작하면 데이터가 날아간다. Redis는 DB를 대체하는 게 아니라 DB 앞에서 부하를 줄이는 계층이다.

오해 4: "캐시 미스는 나쁘다"

사실: 캐시 미스 자체가 문제는 아니다. 히트율 100%를 목표로 TTL을 너무 길게 잡으면 데이터 신선도가 떨어진다. 비즈니스 허용 범위 안에서 적절한 히트율을 목표로 삼아야 한다 (보통 80~95%).

자기 점검 Rubric

내 답변 수준 확인하기

미흡 "Redis를 쓰면 빠릅니다"만 말하고, 왜 쓰는지, 어떤 전략인지 설명 못 함
주니어 Cache-aside 개념을 알고, GET → DB → SET 흐름을 설명할 수 있음. TTL 언급 가능
중급 Cache-aside / Write-through / Write-behind 차이를 상황별로 선택할 수 있음. 무효화 전략(TTL + 명시적 삭제)을 설명할 수 있음
시니어 도입 판단 기준(읽기 편향 / 변경 주기 / 허용 불일치)을 먼저 제시하고, 전략 선택 이유, TTL 근거, 장애 시 동작, 히트율 모니터링까지 자연스럽게 연결해서 답변함. 캐시가 적합하지 않은 경우도 명확히 설명할 수 있음

핵심 요약

오늘 반드시 기억할 것

  • 캐시 = 빠른 저장소에 복사. 읽기 편향 + 허용 불일치 + 반복 조회가 전제 조건이다.
  • Cache-aside: 실제 조회된 것만 캐시. 가장 범용적. 첫 요청은 느림.
  • Write-through: 쓸 때 캐시도 동시 갱신. 일관성 높음. 쓰기 비용 증가.
  • Write-behind: 캐시에 먼저 쓰고 DB는 나중에. 쓰기 성능 최고. 데이터 유실 위험.
  • TTL만으로는 부족하다. 데이터 변경 시 명시적 무효화(삭제/갱신)가 병행되어야 한다.
  • Redis가 다운되면 모든 요청이 DB로 가는 cascade failure 위험이 있다. 대비책을 설계해야 한다.
  • 히트율이 80% 이하면 키 설계나 TTL을 재검토해야 한다.
