Day 6 · Redis / 캐시 전략 Interview

랭킹 시스템과 Sorted Set

"순위"는 단순 정렬 문제가 아니다 — 실시간 갱신, 동점 처리, 대량 랭킹의 메모리 비용까지
면접 질문
Q1 — 주력 질문 "실시간으로 갱신되는 게임 랭킹(또는 인기글 랭킹) 시스템을 설계해야 합니다. 초당 수천 건씩 점수가 바뀌는 상황에서 '내 순위'와 '상위 100명'을 빠르게 조회하려면 어떻게 하시겠습니까? DB로도 할 수 있는데 왜 Redis Sorted Set을 쓰나요?"
Q2 — 변형 "점수가 동일한 사용자가 여러 명일 때 순위는 어떻게 결정하시겠습니까? Sorted Set 내부적으로 동점을 어떻게 처리하나요?"
Q3 — 변형 "사용자가 1억 명이고 랭킹도 1억 건이라면, Sorted Set 하나에 다 넣어도 괜찮을까요? 메모리와 성능 관점에서 어떤 문제가 생기고, 어떻게 완화하시겠습니까?"
전제 개념

왜 랭킹은 "그냥 정렬"로 풀리지 않는가

랭킹 기능을 처음 설계할 때 흔히 떠올리는 방식은 "DB에서 점수 컬럼으로 ORDER BY score DESC 하면 되지 않나?"다. 데이터가 작고 갱신이 드물면 실제로 이걸로 충분하다. 문제는 랭킹이 의미 있는 서비스일수록 점수가 실시간으로, 매우 자주 바뀐다는 점이다. 게임 점수, 좋아요 수, 실시간 조회수처럼 초당 수백~수천 건의 갱신이 발생하면, 매번 ORDER BY로 전체를 다시 정렬하는 방식은 인덱스가 있어도 쓰기 부하와 락 경합이 급격히 커진다.

Sorted Set — "정렬된 상태를 유지하는 자료구조"라는 개념

Redis의 Sorted Set(정렬 집합, ZSET)은 각 원소(member)에 score(점수)라는 숫자를 붙여 저장하는 자료구조인데, 핵심은 "저장할 때마다 정렬해서 넣는 것"이 아니라 내부적으로 항상 정렬된 상태를 유지한다는 점이다. 내부 구현은 skip list(스킵 리스트)와 해시 테이블을 함께 사용하는데, 이 덕분에 원소 추가/갱신은 O(log N), 순위 구간 조회도 O(log N + M) (M은 반환 개수)로 처리된다. 즉 "정렬을 매번 다시 하는 것"이 아니라 "이미 정렬된 구조에 하나를 끼워 넣는 것"이라 훨씬 저렴하다.

왜 이게 랭킹에 딱 맞는가: 랭킹 시스템이 필요로 하는 연산 — "점수 갱신", "내 순위 조회", "상위 N명 조회", "특정 순위 구간 조회" — 이 모두 Sorted Set의 기본 명령 하나로 O(log N) 수준에 해결된다는 것이 핵심이다. 
# 점수 등록/갱신: member의 score를 지정 값으로 설정 (있으면 갱신, 없으면 추가)
ZADD ranking:game:daily 1500 user:1001

# 점수 증감: 기존 점수에 delta만큼 더함 (좋아요 +1, 킬 스코어 +10 같은 상황에 적합)
ZINCRBY ranking:game:daily 10 user:1001

# 상위 100명 조회 (점수 내림차순, WITHSCORES로 점수까지 함께 반환)
ZREVRANGE ranking:game:daily 0 99 WITHSCORES

# 특정 유저의 현재 순위 조회 (0부터 시작, 내림차순 기준)
ZREVRANK ranking:game:daily user:1001

동점 처리 — Sorted Set은 어떻게 순서를 정하는가

Sorted Set은 score가 같은 원소들을 member 이름의 사전식(lexicographic) 순서로 정렬한다. 즉 점수가 같으면 "먼저 등록된 사람이 위"가 아니라 문자열 비교로 결정되는 임의의 순서다. 실무에서 "동점이면 먼저 달성한 사람이 위"라는 요구가 흔한데, 이걸 그대로 구현하면 안 되는 이유가 여기 있다. 해법은 score 자체에 시간 정보를 얹어 합성 점수(composite score)를 만드는 것이다. 

// 예: 점수를 상위 비트, (역전된) 타임스탬프를 하위 비트로 합성
// 점수가 같으면 타임스탬프가 작을수록(더 먼저 달성) 상위로 오도록 설계
composite_score = (score * 10_000_000_000) + (9_999_999_999 - timestamp_offset)

이렇게 하면 "점수가 곧 진짜 등수를 결정하는 유일한 값"이 되어, Sorted Set의 기본 정렬만으로 원하는 동점 처리 규칙까지 구현할 수 있다. 별도의 애플리케이션 로직으로 동점자를 다시 정렬할 필요가 없다는 것이 핵심 장점이다.

대량 랭킹의 진짜 문제 — "내 순위"가 아니라 "메모리와 브로드캐스트 비용"

사용자 1억 명의 점수를 Sorted Set 하나에 넣는 것 자체는 자료구조상 불가능하지 않다. 문제는 다음 두 가지다.

  • 메모리 사용량: Sorted Set의 원소 하나는 member 문자열 + score(8바이트 double) + skip list 포인터 오버헤드를 포함한다. 1억 개면 순수 데이터보다 구조적 오버헤드가 몇 배로 커질 수 있다. member를 짧은 정수 ID로 쓰는 것만으로도 메모리를 크게 아낄 수 있다.
  • ZREVRANGE의 비용은 반환 개수(M)에 비례한다: 상위 100명 조회는 빠르지만, "내 순위가 5천만 등일 때 앞뒤 10명을 보여줘" 같은 요청은 ZREVRANK + ZREVRANGE 조합으로 여전히 O(log N)이라 괜찮다. 진짜 비용은 랭킹판을 실시간으로 모든 접속자에게 브로드캐스트하려는 요구사항 쪽에서 생긴다 — 이건 Sorted Set의 문제가 아니라 Pub/Sub·polling 설계의 문제다.

여러 랭킹판 연산 — 합산 랭킹은 ZUNIONSTORE로

"일간 랭킹 + 주간 랭킹을 합친 종합 랭킹"처럼 여러 Sorted Set을 합쳐야 하는 경우, 각 원소를 애플리케이션에서 읽어와 더하는 대신 ZUNIONSTORE로 Redis 서버 안에서 합산해 새 Sorted Set으로 저장할 수 있다. 네트워크 왕복과 애플리케이션 계산 비용을 서버 쪽 단일 연산으로 옮기는 방식이다. 

# daily와 weekly 랭킹을 합산(SUM)해서 combined 키에 저장
ZUNIONSTORE ranking:combined 2 ranking:daily ranking:weekly WEIGHTS 1 1
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

랭킹 질문은 단순히 "ZADD 아세요?"가 아니라, 자료구조의 시간 복잡도가 실시간 요구사항과 어떻게 연결되는지, 그리고 정렬이라는 "당연해 보이는" 문제에도 숨은 함정(동점, 대량 데이터, 메모리)이 있다는 것을 인지하는지를 평가한다.

  • 복잡도 감각: DB의 ORDER BY 재정렬 비용과 Sorted Set의 O(log N) 삽입/조회 비용 차이를 구체적으로 설명할 수 있는가
  • 요구사항을 자료구조 설계로 번역하는 능력: "동점 시 먼저 달성한 사람이 위"라는 비기능 요구를 합성 점수라는 구체적 설계로 바꿀 수 있는가
  • 규모의 문제를 안다: "잘 되겠지"가 아니라, 데이터가 1억 건이 되었을 때 메모리와 연산 비용이 어떻게 변하는지 구체적 수치 감각으로 설명하는가
  • Redis 자체의 한계도 인지하는가: Sorted Set이 랭킹에 최적화된 도구이지만, 영속성이나 정합성 요구가 매우 강한 랭킹(예: 상금이 걸린 대회 최종 순위)이라면 DB와 병행하거나 검증 단계를 둬야 한다는 것을 아는가
실무에서 랭킹은 "화면에 예쁘게 보여주는 기능"처럼 보이지만, 실제로는 초당 쓰기량, 동점 규칙, 데이터 규모가 얽힌 성능/정합성 설계 문제다. 면접관은 지원자가 이 문제를 "정렬 알고리즘 지식"이 아니라 "자료구조 선택과 트레이드오프 설계"로 접근하는지를 본다.
실무 연결

실제 상황과 연결하기

상황 1: 실시간 게임 리더보드

게임 킬 스코어, 레이싱 기록처럼 초당 수백~수천 건의 갱신이 발생하는 랭킹은 DB 갱신으로는 감당하기 어렵다. Redis Sorted Set에 실시간 점수를 유지하고, DB에는 주기적으로(예: 5분마다) 스냅샷을 비동기로 기록해 영속성을 확보하는 방식이 흔히 쓰인다. Redis가 "현재 순위의 진실"을, DB가 "히스토리 기록"을 담당하는 역할 분리다.

상황 2: 인기글/트렌딩 랭킹

단순 조회수 누적이 아니라 "최근 N시간 내 활동에 가중치를 더 주는" 트렌딩 랭킹은, score 계산 자체에 시간 감쇠(time decay) 함수를 반영한다. 예를 들어 좋아요 수에 게시 시각 기반 감쇠 계수를 곱한 값을 score로 써서, 오래된 글이 자연스럽게 순위에서 밀려나도록 설계한다(Reddit의 hot 랭킹 알고리즘이 대표적 사례다).

상황 3: "내 순위" 조회가 몰리는 이벤트

랭킹 이벤트 종료 직전에는 사용자들이 "내 순위"를 몰아서 조회하는 트래픽 스파이크가 발생한다. ZREVRANK는 O(log N)이라 단건 조회 자체는 빠르지만, 동시 접속자가 급증하면 Redis 인스턴스의 CPU 단일 스레드 처리량이 병목이 될 수 있다. 이런 상황에서는 Redis Cluster로 랭킹 키를 여러 노드에 분산하거나, "내 순위"처럼 자주 조회되는 값은 짧은 TTL의 별도 캐시로 한 번 더 감싸는 것도 고려한다.

상황 4: 여러 조건의 랭킹(일간/주간/전체) 관리

랭킹판이 여러 개면 키를 ranking:game:daily:2026-07-05처럼 시간 단위로 분리해서 관리하고, 지난 날짜의 키는 TTL을 걸어 자동 만료시킨다. 이렇게 하면 "며칠 지난 일간 랭킹을 수동으로 지우는 배치"가 필요 없어지고, 필요하면 ZUNIONSTORE로 여러 기간의 랭킹을 합산해 종합 랭킹을 만든다.

상황 5: 랭킹 데이터의 정합성이 돈과 연결될 때

상금이 걸린 대회처럼 랭킹 결과가 실제 정산으로 이어지는 경우, Redis만 신뢰하는 것은 위험하다. Redis는 기본적으로 가용성과 속도에 최적화되어 있고, 장애나 데이터 손실 가능성이 완전히 0은 아니기 때문이다. 이런 도메인에서는 실시간 랭킹은 Redis로 보여주되, 최종 확정 순위는 DB 트랜잭션 기반으로 검증 후 확정하는 이원화 설계를 쓴다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
왜 DB 정렬이 아닌지부터 설명한다 "점수가 실시간으로 자주 바뀌는 랭킹은 매번 ORDER BY로 전체를 재정렬하는 비용이 크기 때문에, 항상 정렬된 상태를 유지하는 자료구조가 필요합니다."
2
Sorted Set의 복잡도 근거를 댄다 "Redis Sorted Set은 내부적으로 skip list를 써서 삽입/갱신이 O(log N)이고, 상위 N개 조회도 O(log N + M)이라 실시간 랭킹에 적합합니다."
3
동점 처리 요구사항을 설계로 번역한다 "동점 처리 규칙이 있다면 score에 시간 정보를 합성해서, Sorted Set의 기본 정렬만으로 원하는 순서가 나오게 만듭니다."
4
규모가 커졌을 때의 트레이드오프를 짚는다 "데이터가 수천만~억 단위로 커지면 member를 짧은 ID로 최적화하고, 조회가 특정 시점에 몰리는 트래픽 스파이크를 고려해 캐시나 Cluster 분산을 함께 검토합니다."
5
정합성이 중요한 도메인이면 이원화를 제안한다 (Senior 포인트) "정산이 걸린 랭킹이라면 Redis만 신뢰하지 않고, 실시간 화면은 Redis로, 최종 확정은 DB 트랜잭션 기반 검증으로 분리합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"실시간으로 자주 바뀌는 랭킹을 DB의 ORDER BY로 매번 재정렬하면, 데이터가 많아질수록 쓰기 부하와 락 경합이 급격히 커집니다. 그래서 저는 Redis Sorted Set을 씁니다. Sorted Set은 내부적으로 skip list 구조를 써서 항상 정렬된 상태를 유지하고, 점수 갱신과 순위 조회가 모두 O(log N) 수준으로 처리되기 때문에 전체 재정렬 없이 실시간 갱신을 감당할 수 있습니다.

동점 처리가 필요하다면, Sorted Set은 score가 같을 때 member 이름의 사전식 순서로 정렬한다는 점을 이용하지 않고, score 자체에 타임스탬프 같은 부가 정보를 합성해서 '점수가 같으면 먼저 달성한 사람이 위' 같은 비즈니스 규칙을 자료구조 차원에서 해결합니다.

데이터가 수천만~억 단위로 커지면, member를 문자열 대신 짧은 정수 ID로 써서 메모리를 아끼고, 이벤트 종료 직전처럼 '내 순위' 조회가 몰리는 트래픽 스파이크에 대비해 Redis Cluster로 분산하거나 자주 조회되는 구간을 짧은 TTL 캐시로 한 번 더 감쌉니다. 그리고 만약 이 랭킹 결과가 실제 상금 정산처럼 돈과 직결된다면, 저는 Redis 하나만 신뢰하지 않고 실시간 화면은 Redis로 보여주되 최종 확정 순위는 DB 트랜잭션 기반으로 검증 후 확정하는 이원화 구조로 설계합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "ZADD와 ZINCRBY의 차이는 무엇이고, 언제 각각을 쓰나요?"

강한 답변: "ZADD는 score를 절대값으로 설정하고, ZINCRBY는 기존 score에 delta를 더합니다. '좋아요 +1'처럼 상대적 증감이 필요하면 ZINCRBY가 원자적이라 안전하고, '레벨 만점 100점'처럼 절대값을 지정해야 하면 ZADD를 씁니다. 두 방식 모두 원자적 연산이라 별도 락 없이 동시 갱신이 안전합니다."

F2. "Sorted Set의 내부 구현이 skip list인 이유는 무엇인가요? 균형 이진 트리(B-Tree)를 쓰면 안 되나요?"

강한 답변: "skip list는 균형 트리와 비슷한 O(log N) 성능을 내면서도 구현이 훨씬 단순하고, 특히 '구간 조회(range query)'에서 연결 리스트처럼 순회가 쉬워 ZRANGE 같은 연산에 유리합니다. 또한 락을 거는 범위가 트리 재균형보다 작아서 동시성 처리에도 더 단순합니다."

F3. "랭킹 데이터가 1억 건일 때 메모리를 어떻게 줄이나요?"

강한 답변: "member를 긴 문자열(예: UUID)이 아니라 짧은 정수 ID로 쓰는 것이 가장 직접적입니다. 또한 랭킹이 굳이 전체 사용자를 다 담을 필요가 없다면, 상위 N만 유지하고 나머지는 별도로 관리하는 것도 고려합니다. Redis의 maxmemory 설정과 자료구조별 인코딩 임계값(예: zset-max-listpack-entries)도 함께 점검합니다."

F4. "실시간 랭킹판을 접속자 전체에게 갱신해서 보여줘야 한다면 어떻게 하나요?"

강한 답변: "이건 Sorted Set의 문제가 아니라 갱신 전파(broadcast)의 문제입니다. 클라이언트가 각자 폴링하면 랭킹판 변화가 없어도 트래픽이 낭비되므로, 점수 변화가 있을 때만 Redis Pub/Sub이나 WebSocket으로 서버가 먼저 알리는 방식을 검토합니다. 다만 초당 갱신이 매우 잦다면 매번 브로드캐스트하지 않고, 일정 주기(예: 1초)로 스냅샷을 묶어 보내는 throttling도 함께 고려합니다."

F5. "왜 굳이 Redis를 쓰나요? RDB에 인덱스 걸고 캐싱하면 안 되나요?"

강한 답변: "인덱스를 걸어도 RDB는 갱신마다 인덱스 구조(B-Tree)를 다시 조정해야 해서 쓰기 비용이 Sorted Set보다 큽니다. 또한 '상위 N개'나 '내 순위' 같은 연산은 Sorted Set이 자료구조 자체로 지원하지만, RDB에서는 ORDER BY ... LIMIT이나 윈도우 함수로 매번 계산해야 합니다. 다만 저는 Redis가 무조건 우월하다고 보지 않고, 갱신 빈도가 낮고 규모가 작다면 RDB만으로 충분하다는 것도 함께 판단합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • "Sorted Set 쓰면 됩니다"가 아니라 왜 O(log N)인지, 왜 이게 실시간 랭킹에 맞는지 근거를 댄다
  • 동점 처리를 합성 점수 설계로 구체화한다
  • 데이터 규모가 커졌을 때 메모리 최적화와 트래픽 스파이크 대응을 함께 이야기한다
  • 랭킹의 정합성 요구 수준에 따라 Redis 단독 vs DB 병행을 구분해 판단한다
  • Redis가 모든 상황에 우월하다고 단정하지 않고 규모/빈도에 따른 선택 기준을 제시한다

약한 답변의 특징

  • "Redis가 빠르니까 Sorted Set 쓰면 됩니다"라고만 하고 왜 빠른지 설명하지 못한다
  • 동점 처리를 애플리케이션에서 다시 정렬하는 식으로 비효율적으로 접근한다
  • 데이터 규모가 커져도 같은 설계로 문제없이 확장된다고 단정한다
  • 랭킹 데이터의 정합성/영속성 문제를 전혀 고려하지 않는다
  • "Redis가 무조건 DB보다 낫다"는 식으로 근거 없이 우열을 단정한다
핵심 차이: 강한 답변은 자료구조 선택의 이유를 복잡도와 실제 트래픽 패턴으로 설명하고, 규모가 커지거나 정합성이 중요해질 때 설계가 어떻게 바뀌어야 하는지까지 이야기한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Sorted Set은 그냥 정렬해서 저장하는 List다"

사실: Sorted Set은 원소 추가/조회마다 전체를 재정렬하는 것이 아니라, skip list와 해시 테이블 구조로 항상 정렬된 상태를 O(log N)에 유지한다. "정렬을 매번 다시 수행하는 자료구조"와 "이미 정렬된 구조를 유지하는 자료구조"는 완전히 다른 성능 특성을 가진다.

오해 2: "동점이면 순서가 알아서 공평하게 처리된다"

사실: score가 같으면 Sorted Set은 member 이름의 사전식 순서로 정렬한다. "먼저 달성한 사람이 위"같은 비즈니스 규칙은 자동으로 보장되지 않으며, score 자체에 시간 정보를 합성해서 설계해야 한다.

오해 3: "Sorted Set은 데이터가 아무리 많아져도 그대로 잘 동작한다"

사실: 자료구조의 시간 복잡도는 그대로 유지되지만, 메모리 사용량은 원소 수에 비례해 선형으로 증가한다. 억 단위 데이터에서는 member 크기 최적화, 인코딩 임계값 설정, 조회 트래픽 분산을 함께 고려하지 않으면 메모리 부족이나 특정 시점의 조회 폭주로 문제가 생길 수 있다.

오해 4: "랭킹은 Redis에만 있으면 충분하다"

사실: Redis는 속도와 가용성에 최적화되어 있지만, 랭킹 결과가 실제 정산·상금·법적 효력과 연결되는 도메인이라면 Redis 데이터만으로 최종 확정하는 것은 위험하다. 이런 경우 DB 트랜잭션 기반 검증을 병행하는 이원화 설계가 필요하다.

핵심 요약

오늘 반드시 기억할 것

  • Sorted Set은 skip list 기반으로 항상 정렬된 상태를 O(log N)에 유지해서 실시간 랭킹에 적합하다.
  • 동점 처리는 score에 시간 정보를 합성해서 자료구조 차원에서 해결하는 것이 애플리케이션 재정렬보다 낫다.
  • 데이터 규모가 커지면 member 크기 최적화, 인코딩 임계값, 조회 트래픽 분산을 함께 고려해야 한다.
  • 여러 랭킹판은 ZUNIONSTORE로 서버 쪽에서 합산해 네트워크/계산 비용을 줄인다.
  • 정산·상금과 직결된 랭킹은 Redis 단독으로 확정하지 말고 DB 트랜잭션 기반 검증을 병행한다.
