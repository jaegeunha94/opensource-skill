캐시 스탬피드와 Hot Key 문제

캐시가 오히려 DB를 무너뜨리는 순간을 막는 설계 — dog-pile effect, mutex lock, local cache 계층
면접 질문
Q1 — 주력 질문 "트래픽이 많은 캐시 키의 TTL이 만료되는 순간, 어떤 문제가 생길 수 있습니까? 겪어보셨거나 방지해보신 적이 있나요?"
Q2 — 변형 "타임세일이나 이벤트로 특정 상품 하나에 트래픽이 몰려서 Redis 노드 하나만 부하가 튀는 상황을 본 적 있나요? 어떻게 대응하시겠습니까?"
Q3 — 변형 "캐시 서버를 재시작했더니 DB가 순간적으로 뻗었습니다. 왜 이런 일이 생기고, 어떻게 미리 막을 수 있었을까요?"
전제 개념

캐시가 "DB를 지키는 방패"에서 "DB를 무너뜨리는 방아쇠"로 바뀌는 순간

캐시를 쓰는 이유는 DB에 몰릴 요청을 줄이기 위함이다. 그런데 특정 조건에서는 캐시가 있기 때문에 DB가 순간적으로 무너지는 역설적인 상황이 생긴다. 이 Day에서는 그 대표적인 두 가지 현상 — 캐시 스탬피드(cache stampede)와 hot key — 를 구분하고, 각각을 방어하는 방법을 다룬다.

캐시 스탬피드(Cache Stampede) = Dog-pile Effect

"stampede"는 소 떼가 한꺼번에 우르르 몰려 뛰는 것을, "dog-pile"은 여러 사람이 한 사람 위에 무더기로 올라타는 것을 뜻하는 영어 표현이다. 캐시 맥락에서는 인기 있는 캐시 키 하나가 만료(또는 삭제)되는 순간, 그 키를 요청하던 수백~수만 개의 동시 요청이 전부 캐시 미스를 만나 한꺼번에 DB로 몰려가는 현상을 말한다. 

// Cache-aside 패턴에서 흔히 쓰는 순진한 구현
function getProduct(id) {
  let data = redis.get(`product:${id}`);
  if (data == null) {
    data = db.query(`SELECT * FROM products WHERE id=?`, id); // 캐시 미스 시 DB 조회
    redis.setex(`product:${id}`, 60, data);
  }
  return data;
}
// 이 상품 키에 초당 5,000건이 몰리는 상황에서 TTL이 만료되는 그 1ms 동안
// 5,000개 요청이 "동시에" data == null 을 보고 전부 db.query()를 실행한다.
왜 위험한가: 캐시는 평소 초당 5,000건의 조회를 DB 대신 흡수해서 DB를 보호하고 있었다. 그런데 만료되는 그 순간만큼은 그 5,000건이 동시에 DB로 꽂힌다. 평소 초당 몇 건 수준의 조회만 받던 DB 커넥션 풀이 순식간에 고갈되고, 느려진 쿼리들이 서로 얽혀 DB 전체 장애로 번질 수 있다.

Hot Key — "만료"와 무관하게 발생하는 또 다른 문제

캐시 스탬피드가 "만료 순간"의 문제라면, hot key는 특정 키 하나에 대한 요청량이 다른 키들보다 압도적으로 많아서, 그 키를 처리하는 Redis 노드(샤드) 하나에만 부하가 쏠리는 현상이다. TTL이 남아있어도, 캐시가 정상적으로 히트해도 발생한다.

핵심 차이: Redis Cluster는 키를 해시 슬롯(hash slot) 단위로 여러 노드에 분산한다. 하지만 같은 키는 항상 같은 슬롯, 같은 노드에만 저장된다. 노드를 아무리 늘려도(scale-out) 특정 키 하나에 대한 요청은 결국 그 키를 담당하는 노드 하나로만 몰린다. 즉, hot key는 "서버를 늘리면 자동으로 풀리는 문제"가 아니다.

예를 들어 인플루언서 라이브 방송의 "현재 시청자 수" 카운터, 타임세일 시작 시각의 "재고 수량" 키, 급상승 뉴스 기사의 조회수 캐시 키는 그 키 하나에 초당 수만 건의 요청이 집중될 수 있다. 이때 Redis 클러스터 전체 CPU 평균은 낮아 보이는데, 그 키를 담당하는 노드 하나만 CPU 100%를 찍는 비대칭 현상이 나타난다.

완화 전략 ① Mutex Lock (Request Coalescing) — "한 명만 DB에 가고 나머지는 기다린다"

캐시 미스가 나면, 가장 먼저 도착한 요청 하나만 DB 조회를 하도록 락을 걸고, 나머지 요청은 그 결과를 기다리거나 잠깐 재시도한다. Redis의 SET key value NX EX ttl 명령으로 간단한 락을 구현할 수 있다. 

function getProduct(id) {
  let data = redis.get(`product:${id}`);
  if (data != null) return data;

  // 락 획득 시도 — 이미 누군가 락을 잡고 있으면 false
  const gotLock = redis.set(`lock:product:${id}`, "1", "NX", "EX", 5);
  if (gotLock) {
    try {
      data = db.query(`SELECT * FROM products WHERE id=?`, id);
      redis.setex(`product:${id}`, 60, data);
    } finally {
      redis.del(`lock:product:${id}`);
    }
  } else {
    // 락을 못 잡은 요청들: 짧게 대기 후 캐시 재조회 (또는 stale 값 반환)
    sleep(50);
    return getProduct(id);
  }
  return data;
}
락 자체도 실패 지점이 될 수 있다. DB 조회가 락 TTL보다 오래 걸리면 락이 먼저 풀려 다른 요청이 또 락을 잡고 중복 조회가 발생할 수 있고, 락을 잡은 프로세스가 죽으면 finally의 DEL이 실행되지 않아 락 TTL이 끝날 때까지 다른 요청이 계속 대기해야 한다. 락은 "완벽한 차단"이 아니라 "동시 조회 수를 크게 줄이는 완화책"으로 다뤄야 한다.

완화 전략 ② Stale-While-Revalidate — "옛 값을 먼저 주고, 뒤에서 몰래 갱신한다"

논리적 만료 시각(logical TTL)과 실제 삭제 시각(physical TTL)을 분리하는 방식이다. 캐시 값에 "진짜 최신인지" 여부를 같이 저장해두고, 논리적으로 만료되었어도 일단 그 값을 즉시 사용자에게 돌려준 뒤, 그중 딱 한 요청만 백그라운드에서 DB를 조회해 캐시를 갱신한다. 

// 캐시 값에 "logical_expire" 필드를 함께 저장
{ "data": {...}, "logical_expire": 1719900000 }

function getProduct(id) {
  const cached = redis.get(`product:${id}`); // physical TTL은 넉넉하게(예: 1시간) 설정
  if (cached.logical_expire > now()) {
    return cached.data; // 아직 신선함
  }
  // 논리적으로는 만료됨. 하지만 값은 즉시 반환한다 (사용자는 기다리지 않음)
  if (redis.set(`lock:product:${id}`, "1", "NX", "EX", 5)) {
    asyncRefreshInBackground(id); // 딱 한 요청만 백그라운드 갱신 시작
  }
  return cached.data; // 모든 요청은 즉시 (약간 오래된) 값을 받는다
}
Mutex lock은 "정확한 최신값을 위해 사용자를 잠깐 기다리게" 하는 방식이고, stale-while-revalidate는 "사용자는 절대 기다리지 않게 하고 대신 데이터가 몇 초 더 오래될 수 있음을 감수"하는 방식이다. 응답 지연을 절대 허용할 수 없는 트래픽(메인 페이지, 실시간 피드)에는 후자가 자주 쓰인다.

완화 전략 ③ 확률적 조기 갱신 (Probabilistic Early Expiration / XFetch)

TTL이 정확히 끝나는 순간이 아니라, 만료 시각에 가까워질수록 조기에 갱신을 시도할 확률을 점점 높이는 방식이다. 그러면 동시에 만료되는 대신, 만료 시각 이전 구간에 걸쳐 갱신 요청이 자연스럽게 분산된다. 하나의 요청이 "이번엔 내가 미리 갱신해볼게"라고 확률적으로 결정하기 때문에, 대부분의 요청은 평소처럼 캐시를 그냥 읽고 지나간다.

완화 전략 ④ Hot Key 전용 — Local Cache(L1) 계층과 키 복제(Key Replication)

Hot key는 애초에 Redis까지 요청이 도달하기 전에 흡수해야 근본적으로 완화된다.

  • Local Cache(L1, in-process cache): 애플리케이션 서버 프로세스 메모리 안에 아주 짧은 TTL(예: 1~3초)로 값을 캐싱한다. 서버가 100대면 Redis에 도달하는 요청이 이론상 100분의 1 수준으로 줄어든다.
  • 키 복제(Key Replication / Key Splitting): product:1 하나로 몰리는 요청을, product:1:0 ~ product:1:9처럼 같은 값을 여러 키에 복제해두고 요청 시 무작위로 분산 조회한다. 같은 노드일 수도 있지만, Redis Cluster에서는 해시 슬롯이 달라져 여러 노드로 물리적으로 분산될 수 있다.
Local cache를 추가하면 새로운 문제가 생긴다. 서버마다 로컬 캐시를 들고 있으므로, 값이 바뀌었을 때 모든 서버의 로컬 캐시를 동시에 무효화해야 한다. 이건 Redis 하나만 지우는 것보다 훨씬 어렵다 — 그래서 로컬 캐시 TTL은 아주 짧게 잡거나, Redis Pub/Sub으로 무효화 이벤트를 모든 서버에 브로드캐스트하는 방식을 함께 쓴다(Pub/Sub은 Day 7에서 더 다룬다).
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

캐시 스탬피드/hot key 질문은 "캐시를 붙이는 것"과 "트래픽 규모에서 캐시가 살아남게 설계하는 것"의 차이를 평가하는 질문이다. 트래픽이 적은 환경에서는 절대 드러나지 않는 문제이기 때문에, 이 질문에 구체적으로 답할 수 있다는 것 자체가 실제로 규모 있는 트래픽을 운영해본 경험의 신호가 된다.

  • 캐시의 역설을 이해하는가: 캐시가 평소엔 DB를 지키지만, 특정 순간엔 오히려 DB를 무너뜨릴 수 있다는 구조적 위험을 인지하는가
  • 스탬피드와 hot key를 구분하는가: 하나는 "만료 시점"의 문제, 하나는 "특정 키 자체"의 문제임을 구분해서 서로 다른 해법을 제시하는가
  • 단일 해법에 의존하지 않는가: "락 하나면 끝"이 아니라 mutex lock, stale-while-revalidate, jitter, local cache 등 상황에 맞는 여러 계층의 방어를 조합할 수 있는가
  • trade-off를 설명하는가: 정합성(항상 최신) vs 가용성/지연(사용자를 기다리게 하지 않음) 사이에서 비즈니스 요구에 맞게 선택할 수 있는가
이 주제는 실제로 대형 서비스의 장애 회고(postmortem)에 반복적으로 등장하는 패턴이다. "배포 직후 캐시가 비어서 DB가 죽었다", "타임세일 상품 하나 때문에 Redis 노드 하나가 다운됐다"는 이야기는 시니어 면접관 다수가 직접 겪어본 상황이기 때문에, 피상적인 답변은 바로 드러난다.
실무 연결

실제 상황과 연결하기

상황 1: 배포/재시작 직후 캐시가 텅 비어(Cold Cache) DB가 순간 뻗음

Redis를 재시작하거나 새 캐시 클러스터로 전환하면, 그 순간 모든 키가 캐시 미스 상태가 된다. 평소 캐시가 흡수하던 트래픽 전부가 한꺼번에 DB로 몰리는, 스탬피드의 극단적인 형태(cold start stampede)다. → 배포 전 캐시 워밍업(cache warm-up) 스크립트로 인기 키를 미리 채워두거나, 트래픽을 점진적으로 전환(canary)하며 DB 부하를 모니터링한다.

상황 2: 타임세일 시작 시각, 재고 수량 키 하나에 트래픽 집중

"10시 정각 한정판 100개" 이벤트에서 재고 확인 API가 특정 상품 key 하나에 초당 수만 건 몰린다. 해당 키를 담당하는 Redis 노드만 CPU가 튀고, 다른 노드는 한가하다. → 재고 수량처럼 자주 바뀌는 값은 local cache(1~2초 TTL)로 1차 흡수하고, 정확한 차감은 별도의 원자적 연산(예: DECR)으로만 처리해 값을 읽는 트래픽과 값을 깎는 트래픽을 분리한다.

상황 3: 인기 게시물의 "좋아요 수" 캐시가 만료되는 순간 DB 커넥션 풀 고갈

팔로워 수백만인 인플루언서 게시물의 좋아요 수 캐시가 만료되는 순간, 동시 조회 요청이 전부 DB로 몰려 DB 커넥션 풀이 바닥나고, 이 게시물과 무관한 다른 API까지 같은 DB를 쓰다가 함께 느려진다(장애의 전파, blast radius 확산). → mutex lock으로 DB 조회를 1건으로 제한하거나, stale-while-revalidate로 사용자에게는 즉시 옛 값을 보여주고 뒤에서 갱신한다.

상황 4: 뉴스 속보/급상승 검색어 — 갑자기 하나의 키가 hot key가 됨

평소엔 트래픽이 고르게 분산돼 있다가, 특정 이슈가 터지면 그 기사 하나의 조회수 캐시 키에 트래픽이 순간적으로 집중된다. → 사전에 예측 불가능한 hot key이므로, 상시 hot key 탐지 모니터링(특정 키의 초당 요청 수 추적)을 깔아두고, 감지되면 자동으로 그 키만 local cache TTL을 늘리거나 key replication을 적용하는 운영 자동화가 필요하다.

캐시 스탬피드 대응 — 동시 DB 조회를 1건으로 제한 "만료 시 mutex lock으로 첫 요청만 DB를 조회하게 하거나, 지연을 절대 허용할 수 없다면 stale-while-revalidate로 옛 값을 즉시 주고 백그라운드에서 갱신합니다."

4
Trade-off 명시 "mutex lock은 정확성을 지키지만 사용자를 잠깐 기다리게 할 수 있고, stale-while-revalidate는 지연은 없지만 잠깐 오래된 데이터를 보여줍니다. 어떤 방식을 쓸지는 데이터의 신선도 요구 수준에 따라 다릅니다."
5
단일 방어선에 의존하지 않는다는 것 강조 (Senior 포인트) "실무에서는 TTL jitter, mutex lock, local cache를 함께 겹쳐 씁니다. 하나가 뚫려도 다른 계층이 완충 역할을 하도록 defense in depth로 설계합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"캐시 스탬피드와 hot key는 원인이 다른 별개의 문제입니다. 캐시 스탬피드는 트래픽이 많은 키 하나의 TTL이 만료되는 순간, 그 키를 기다리던 요청들이 동시에 캐시 미스를 만나 전부 DB로 몰리는 현상입니다. Hot key는 만료와 무관하게 특정 키의 요청량 자체가 압도적으로 많아서, Redis Cluster에서 그 키를 담당하는 노드 하나에만 부하가 쏠리는 문제입니다. Redis는 같은 키를 항상 같은 노드에 두기 때문에, 노드를 늘려도 hot key는 자동으로 해결되지 않습니다.

스탬피드는 동시 DB 조회를 1건으로 제한하는 방식으로 완화합니다. Redis의 SET NX EX로 락을 잡아 가장 먼저 도착한 요청만 DB를 조회하고, 나머지는 짧게 대기 후 갱신된 캐시를 읽게 합니다. 응답 지연을 전혀 허용할 수 없는 트래픽이라면 stale-while-revalidate 방식으로, 논리적으로 만료된 값이라도 일단 즉시 반환하고 그중 한 요청만 백그라운드에서 캐시를 갱신하게 합니다.

Hot key는 애초에 Redis에 요청이 도달하기 전에 줄이는 게 핵심입니다. 애플리케이션 서버 프로세스 안에 1~2초짜리 짧은 local cache를 두어 서버 대수만큼 Redis 요청을 나누거나, 같은 값을 여러 키에 복제해두고 무작위로 분산 조회하는 key replication을 씁니다.

실무에서는 이 방법들을 하나만 쓰지 않고 겹쳐서 씁니다. TTL에 jitter를 줘서 동시 만료를 애초에 줄이고, 그래도 몰리는 순간엔 mutex lock으로 DB 조회를 제한하고, hot key로 의심되는 키는 모니터링으로 감지해서 local cache나 key replication을 적용하는 식으로 여러 계층의 방어선을 둡니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Mutex lock을 잡은 프로세스가 DB 조회 도중 죽어버리면 어떻게 되나요?"

약한 답변: "그런 경우는 별로 없을 것 같습니다."

강한 답변: "락에 EX로 TTL을 반드시 걸어둬야 합니다. 락을 잡은 프로세스가 죽어서 DEL이 실행되지 못해도, 락 TTL이 지나면 자동으로 풀려서 다음 요청이 다시 시도할 수 있습니다. 락 TTL이 없으면 그 키는 영원히 잠긴 채로 남아 모든 요청이 DB 조회도 캐시 히트도 못 하는 상태(deadlock에 가까운 상황)가 됩니다."

F2. "Stale-while-revalidate와 mutex lock을 같이 쓸 수도 있나요?"

강한 답변: "네, 오히려 이 둘을 조합하는 게 일반적입니다. 논리적 만료를 감지하면 값은 즉시 반환(stale-while-revalidate)하고, 백그라운드 갱신을 시작할 때 mutex lock으로 '갱신 작업 자체'가 중복 실행되지 않게 막습니다. 즉 사용자 응답 경로에는 락이 개입하지 않고, 갱신 경로에서만 중복 방지용으로 락을 씁니다."

F3. "TTL에 jitter(무작위 값)를 추가하면 스탬피드가 완전히 사라지나요?"

약한 답변: "네, jitter를 주면 스탬피드는 해결됩니다."

강한 답변: "jitter는 '동시에 세팅된 여러 키가 동시에 만료되는' 경우(cache avalanche)를 완화하는 데는 효과적이지만, 트래픽이 극단적으로 몰리는 단일 hot key 하나의 스탬피드는 여전히 발생할 수 있습니다. 그 키 하나가 만료되는 순간엔 jitter와 무관하게 동시 요청이 몰리기 때문에, hot key급 트래픽에는 mutex lock이나 stale-while-revalidate 같은 별도의 방어가 함께 필요합니다."

F4. "Hot key인지 아닌지는 어떻게 알아채나요?"

강한 답변: "Redis Cluster의 노드별 CPU/네트워크 사용량이 유독 한 노드만 튀는지를 먼저 봅니다. 그다음 redis-cli --hotkeys나 MONITOR, 최신 버전의 hotkey 감지 기능으로 특정 키의 접근 빈도를 직접 확인합니다. 운영 환경에서는 애플리케이션 레벨에서 키별 접근 횟수를 샘플링해서 임계값을 넘는 키를 알림으로 띄우는 방식도 함께 씁니다."

F5. "Local cache(L1)를 추가했을 때 생기는 데이터 정합성 문제는 어떻게 푸나요?"

강한 답변: "서버 대수만큼 로컬 캐시가 흩어져 있으니, 값이 바뀌어도 각 서버가 알아서 짧은 TTL(1~2초)로 만료될 때까지는 서버마다 다른 값을 볼 수 있습니다. 이 정도의 짧은 불일치를 감수할 수 있는 데이터(조회수, 좋아요 수 같은 근사치 허용 데이터)에만 local cache를 적용하고, 정확해야 하는 데이터는 local cache를 쓰지 않거나 Redis Pub/Sub으로 무효화 이벤트를 모든 서버에 브로드캐스트해서 능동적으로 지웁니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 캐시 스탬피드와 hot key를 원인이 다른 별개의 문제로 구분한다
  • "캐시가 DB를 지키지만, 만료 순간엔 오히려 DB를 무너뜨릴 수 있다"는 구조적 역설을 이해한다
  • mutex lock, stale-while-revalidate, jitter, local cache를 trade-off와 함께 설명한다
  • hot key는 서버를 늘려도 해결되지 않는다는 것을 안다(같은 키는 같은 노드)
  • 여러 방어 계층을 겹쳐서 쓴다는 관점(defense in depth)을 갖고 있다

약한 답변의 특징

  • "캐시를 쓰면 DB 부하가 줄어드니 문제없다"는 단순 낙관
  • 스탬피드와 hot key를 같은 문제로 뭉뚱그린다
  • "TTL을 늘리면 된다" 또는 "서버를 늘리면 된다"는 근본 원인을 못 짚은 해법
  • 락을 걸면 무조건 안전하다고 생각하고, 락 TTL·데드락 가능성을 고려하지 않는다
  • local cache를 추가했을 때 생기는 새로운 정합성 문제를 인지하지 못한다
핵심 차이: 강한 답변은 캐시를 "항상 안전한 장치"가 아니라, 특정 조건에서 위험이 집중되는 시스템 구성요소로 본다. 어떤 조건에서 위험이 커지는지 구체적인 메커니즘으로 설명하고, 그 조건을 좁히는 여러 대응책을 제시할 수 있는지가 시니어와 중급의 갈림길이다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "TTL을 늘리면 스탬피드 문제가 해결된다"

사실: TTL을 늘리면 만료가 발생하는 빈도는 줄지만, 만료가 발생하는 그 순간의 충격은 그대로다. 오히려 TTL이 길어지는 동안 트래픽이 더 쌓여서, 만료되는 순간의 동시 요청 수가 더 커질 수도 있다. 근본 해법은 만료 순간의 동시 조회 자체를 제한하는 것(mutex lock, stale-while-revalidate)이다.

오해 2: "Redis 노드를 늘리면(scale-out) hot key도 해결된다"

사실: Redis Cluster에서 하나의 키는 하나의 해시 슬롯에 고정되고, 그 슬롯은 하나의 노드에만 배정된다. 노드를 100개로 늘려도 특정 키 하나에 대한 요청은 여전히 그 키를 담당하는 노드 하나로만 몰린다. hot key는 애플리케이션 레벨의 요청 분산(local cache, key replication)으로만 근본적으로 완화된다.

오해 3: "Mutex lock을 걸면 완벽하게 안전하다"

사실: 락 TTL 설정을 빠뜨리면 락을 잡은 프로세스가 죽었을 때 다른 모든 요청이 락이 풀릴 때까지 막힌다. 또한 DB 조회가 락 TTL보다 오래 걸리면 락이 먼저 풀려 중복 조회가 일어날 수 있다. 락은 "동시 조회 수를 크게 줄이는 완화책"이지 "무조건적인 보장"이 아니다.

오해 4: "캐시가 있으니 DB 커넥션 풀은 여유 있게 잡을 필요 없다"

사실: 캐시가 평소 트래픽을 잘 흡수할수록, 오히려 DB는 "정말 적은 트래픽만 받는 상태"에 익숙해져 있다. 스탬피드가 발생하면 평소 대비 수십~수백 배의 트래픽이 갑자기 DB로 몰리는데, 이때 DB 커넥션 풀·쿼리 타임아웃 설정이 그 충격을 견딜 수 있는지 별도로 검증해둬야 한다. 캐시 설계와 DB 방어 설정은 같이 가야 한다.

실무 체크리스트

스탬피드 / Hot Key 방어 체크리스트

상황 권장 접근 주의
동일 TTL 키가 대량으로 동시 만료 TTL jitter (±10~20%) 단일 초고트래픽 키에는 부족
초고트래픽 키 하나의 만료 순간 Mutex lock 또는 Stale-while-revalidate 락에 TTL 없으면 deadlock 위험
응답 지연을 절대 허용 불가 Stale-while-revalidate 데이터가 몇 초 오래될 수 있음을 감수
Hot key로 특정 노드만 부하 집중 Local cache(L1) + Key replication 서버 scale-out만으로는 해결 안 됨

자기 점검 Rubric

내 답변 수준 확인하기

미흡 캐시 스탬피드와 hot key라는 용어 자체를 모르고, "캐시가 있으면 DB는 안전하다"고만 답함
주니어 캐시 만료 순간 동시 요청이 DB로 몰릴 수 있다는 것은 알지만, 구체적인 완화 방법을 제시하지 못함
중급 Mutex lock이나 TTL jitter 같은 방법을 하나 이상 설명할 수 있고, hot key와 스탬피드를 구분할 수 있음
시니어 mutex lock, stale-while-revalidate, local cache, key replication을 trade-off와 함께 설명하고, hot key가 scale-out으로 해결되지 않는 이유(해시 슬롯 고정)까지 근거를 들어 설명하며, 여러 방어 계층을 조합하는 defense in depth 관점을 자연스럽게 제시함
자가 점검 질문:
1. 캐시 스탬피드와 hot key의 원인 차이를 한 문장으로 설명할 수 있는가?
2. Mutex lock에 TTL이 반드시 필요한 이유를 설명할 수 있는가?
3. Stale-while-revalidate가 mutex lock과 다르게 사용자 경험을 어떻게 바꾸는지 설명할 수 있는가?
4. Hot key가 Redis 노드를 늘려도 해결되지 않는 이유를 해시 슬롯 개념으로 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • 캐시 스탬피드 = 만료 순간의 문제. 인기 키 하나가 만료되면 동시 요청이 전부 DB로 몰린다.
  • Hot key = 특정 키 자체의 문제. 만료와 무관하게 한 키에 트래픽이 집중돼 그 키를 담당하는 노드 하나만 부하가 튄다.
  • Hot key는 scale-out으로 해결되지 않는다. 같은 키는 항상 같은 해시 슬롯·노드에 고정된다.
  • Mutex lock은 동시 DB 조회를 1건으로 줄이지만, 락 TTL이 없으면 deadlock 위험이 있다.
  • Stale-while-revalidate는 사용자를 기다리게 하지 않는 대신, 짧은 시간 오래된 데이터를 감수한다.
  • Local cache(L1) + Key replication은 Redis에 요청이 도달하기 전에 hot key 트래픽을 흡수한다.
  • 단일 방어책에 의존하지 않는다. jitter, lock, local cache를 겹쳐 쓰는 defense in depth가 실무 정석이다.
