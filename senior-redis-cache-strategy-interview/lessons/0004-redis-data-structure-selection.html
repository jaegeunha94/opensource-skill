Redis 자료구조 실무 선택 기준

String, Hash, List, Set, Sorted Set — "무엇을 저장하는가"가 아니라 "어떻게 접근하는가"로 고르는 법
면접 질문
Q1 — 주력 질문 "Redis에는 String, Hash, List, Set, Sorted Set 같은 여러 자료구조가 있는데, 실무에서 어떤 기준으로 자료구조를 선택하시나요?"
Q2 — 변형 "사용자 프로필을 캐싱할 때 JSON 문자열 하나를 String에 넣는 것과, 필드별로 나눠서 Hash에 저장하는 것 중 어떤 걸 선택하시겠습니까? 이유는?"
Q3 — 변형 "팔로워 목록이나 태그 목록처럼 '중복 없는 목록'을 다룰 때 List 대신 Set을 쓰는 이유는 무엇인가요? 실무에서 Set 연산을 잘못 쓰면 어떤 문제가 생기나요?"
전제 개념

"어떤 값을 담을까"가 아니라 "어떻게 읽고 쓸까"를 먼저 묻는다

Redis 자료구조를 고를 때 가장 흔한 실수는 "이 데이터가 논리적으로 무엇인가"만 보고 정하는 것이다. 예를 들어 "사용자 정보니까 객체 하나 = String에 JSON으로" 라고 생각하는 식이다. 하지만 시니어 수준의 선택 기준은 다르다. 이 데이터에 접근하는 패턴(access pattern) — 전체를 통으로 읽고 쓰는지, 일부 필드만 바꾸는지, 순서가 중요한지, 중복을 허용하지 않아야 하는지, 정렬된 상태로 범위 조회가 필요한지 — 를 먼저 정의하고, 그 패턴에 맞는 자료구조를 고르는 것이 핵심이다.

String — 가장 단순하지만 "전체 교체"가 기본 단위

Redis의 String은 최대 512MB까지 담을 수 있는 바이너리 안전(binary-safe) 값이다. 숫자, 문자열, 직렬화된 JSON, 이미지 바이트까지 뭐든 담을 수 있다. INCR/DECR로 원자적 증감이 가능하고, SETBIT/BITCOUNT로 비트맵을, GETRANGE로 부분 읽기도 가능하다.

핵심 제약: String에 JSON을 통째로 넣으면, 필드 하나만 바꾸고 싶어도 전체 값을 다시 직렬화해서 통째로 덮어써야 한다. 동시에 여러 요청이 서로 다른 필드를 수정하려 하면, 나중에 쓴 요청이 먼저 쓴 요청의 변경을 덮어써버리는 lost update 문제가 생기기 쉽다.

Hash — "객체의 필드별 부분 갱신"이 필요할 때

Hash는 하나의 키 아래에 필드-값 쌍을 여러 개 담는 구조다. HSET, HGET, HINCRBY로 필드 단위 원자적 연산이 가능해서, "사용자 프로필의 닉네임만 바꾸고 싶다"는 요구에 정확히 맞는다. 

// String(JSON)으로 처리하면 필드 하나 바꿔도 전체를 다시 써야 함
SET user:1001 '{"name":"kim","age":30,"level":"gold"}'
// level만 바꾸려면 GET → JSON parse → 수정 → SET 전체 재기록 (원자적이지 않음, race condition 위험)

// Hash로 처리하면 필드 단위로 원자적 갱신
HSET user:1001 name kim age 30 level gold
HSET user:1001 level platinum  // level 필드만 원자적으로 교체, 다른 필드는 그대로
내부 인코딩(내부 구현) 관점: Redis는 Hash의 필드 수와 값 길이가 작을 때(기본 설정 기준 128개 필드, 값 64바이트 이하) 메모리 효율이 훨씬 좋은 listpack(구 ziplist) 인코딩을 자동으로 쓴다. 임계값을 넘으면 일반 해시 테이블(hashtable) 인코딩으로 전환된다. 즉, 작은 Hash는 String(JSON)보다 오히려 메모리를 덜 쓰면서 필드 단위 접근까지 가능하다는 것이 실무에서 자주 나오는 반전 포인트다.

List — 순서가 있고, 양 끝에서만 빠르게 넣고 빼는 구조

List는 연결 리스트 기반(내부적으로는 quicklist, 여러 개의 압축된 listpack을 이어붙인 구조)이라 LPUSH/RPUSH/LPOP/RPOP 같은 양 끝 연산은 O(1)이다. 반면 중간 인덱스 접근(LINDEX)이나 임의 위치 삽입은 O(N)에 가깝다. 최근 게시물 타임라인, 최근 알림 목록처럼 순서가 있고 최근 것만 유지하면 되는 데이터에 적합하다. 

LPUSH feed:user:1001 post:5001   // 맨 앞에 최신 글 추가
LTRIM feed:user:1001 0 99       // 최근 100개만 유지, 나머지는 잘라냄(메모리 무한 증가 방지)

Set — 순서 없음, 중복 없음, "포함 여부"를 O(1)로

Set은 정렬되지 않은 유일값(unique) 집합이다. SISMEMBER로 "이 값이 집합에 있는가"를 평균 O(1)에 확인할 수 있고, SINTER(교집합), SUNION(합집합), SDIFF(차집합) 같은 집합 연산을 지원한다. 팔로우/팔로워 목록, 태그, "이미 처리한 ID 집합" 같은 중복 제거와 소속 여부 확인이 핵심인 데이터에 적합하다.

위험 포인트: SINTER/SUNION/SMEMBERS는 집합 크기에 비례해 시간이 걸리는 O(N) (교집합은 최악의 경우 O(N*M)) 연산이다. Redis는 기본적으로 싱글 스레드로 명령을 처리하기 때문에, 원소 수백만 개짜리 Set에 이 명령을 날리면 그 순간 다른 모든 클라이언트의 요청까지 지연시키는 "느린 명령 하나가 전체를 막는" 사고로 이어질 수 있다.

Sorted Set — "정렬된 상태를 유지하며 범위로 조회"할 때

Sorted Set(ZSet)은 각 원소가 score라는 숫자를 갖고, 그 score 순으로 항상 정렬된 상태를 유지하는 구조다. 내부적으로 skip list와 hash table을 함께 사용해서, ZADD/ZSCORE(특정 원소 점수 조회)는 평균 O(1)~O(log N), ZRANGE/ZRANGEBYSCORE(범위 조회)는 O(log N + M)(M은 반환 개수)로 처리한다. 실시간 랭킹, 지연 큐(score = 실행 예정 시각), 정렬된 타임라인에 쓰인다. List와 헷갈리기 쉽지만, List는 삽입 순서를 유지하는 큐/스택이고 Sorted Set은 값(score) 기준으로 항상 재정렬되는 랭킹 구조라는 점이 근본적으로 다르다.

자료구조 핵심 특성 대표 명령 적합한 상황
String 바이너리 안전, 원자적 증감 GET/SET/INCR 단순 값, 카운터, 세션 토큰, 캐시 통짜 값
Hash 필드 단위 부분 갱신, 소규모 시 메모리 효율 HSET/HGET/HINCRBY 사용자 프로필처럼 필드가 자주 부분 변경되는 객체
List 순서 유지, 양 끝 O(1) LPUSH/RPOP/LTRIM 최근 N개 타임라인, 알림 목록, 단순 큐
Set 중복 없음, 소속 여부 O(1), 집합 연산 SADD/SISMEMBER/SINTER 팔로우 관계, 태그, 중복 방지 ID 집합
Sorted Set score 기준 정렬 유지, 범위 조회 ZADD/ZRANGE/ZINCRBY 실시간 랭킹, 지연 큐, 정렬된 피드
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

자료구조 선택 질문은 "Redis 명령어를 몇 개 아는가"를 묻는 것이 아니다. 데이터의 접근 패턴을 분석해서 알맞은 도구를 고르고, 그 도구의 내부 동작 방식이 만드는 성능/메모리 특성까지 예측할 수 있는가를 평가한다.

  • 접근 패턴 중심 사고: "무엇을 저장하나"가 아니라 "어떻게 읽고 쓰는가"로 자료구조를 고르는가
  • 내부 구현에 대한 이해: 작은 Hash의 listpack 인코딩, Sorted Set의 skip list, List의 quicklist처럼 내부 동작이 실무 성능/메모리에 미치는 영향을 아는가
  • 싱글 스레드 모델의 위험을 아는가: O(N) 명령을 큰 컬렉션에 잘못 쓰면 Redis 전체가 지연되는 구조적 위험을 이해하는가
  • 잘못된 선택의 대가를 구체적으로 설명할 수 있는가: "String에 JSON 넣으면 안 되는 이유"를 단순 암기가 아니라 lost update, 네트워크 페이로드, 동시성 문제로 풀어서 설명할 수 있는가
실무에서 Redis 장애의 상당수는 "잘못된 자료구조 + 큰 컬렉션 + 블로킹 명령"의 조합에서 나온다. 이 질문은 지원자가 그런 사고를 겪어봤는지, 혹은 사전에 막을 수 있는 설계 감각이 있는지를 동시에 확인하는 질문이다.
실무 연결

실제 상황과 연결하기

상황 1: 사용자 프로필 캐시 — String(JSON) vs Hash

프로필 화면은 전체 필드를 한 번에 보여주지만, "닉네임 변경", "포인트 증가" 같은 액션은 필드 하나만 바꾼다. String(JSON)으로 저장하면 필드 하나 바꾸는 데도 전체를 읽고 파싱하고 다시 써야 해서 네트워크 전송량과 동시성 위험이 커진다. Hash로 저장하면 HSET user:1001 nickname 새닉네임 한 번으로 끝나고, 다른 필드를 동시에 바꾸는 요청과 충돌하지 않는다.

상황 2: 최근 조회 상품 목록 — List + LTRIM

"최근 본 상품 20개"처럼 개수 제한이 있고 순서(최신순)가 중요한 데이터는 List가 자연스럽다. LPUSH로 앞에 추가하고 LTRIM으로 20개를 넘는 오래된 항목을 즉시 잘라내면, 메모리가 무한정 늘어나는 것을 막으면서 항상 최근 20개만 유지할 수 있다.

상황 3: 팔로우 관계와 "맞팔 여부" — Set과 SINTER

"A가 B를 팔로우하는가"는 SISMEMBER following:A B로 O(1)에 확인한다. "A와 B의 공통 팔로잉 목록"은 SINTER following:A following:B로 구할 수 있다. 다만 팔로잉 수가 수십만인 인플루언서 계정에 이 연산을 그대로 걸면 순간적으로 Redis를 지연시킬 수 있어, 실무에서는 팔로잉 수 상한을 두거나 샘플링/캐시된 결과를 함께 활용한다.

상황 4: 실시간 랭킹 — Sorted Set

게임 점수 랭킹, 인기 게시물 순위는 ZADD leaderboard 15000 user:1001로 점수를 갱신하고 ZREVRANGE leaderboard 0 9 WITHSCORES로 상위 10명을 즉시 가져온다. DB에서 매번 ORDER BY score DESC LIMIT 10 쿼리를 실행하는 것보다 훨씬 저렴하고, 점수가 바뀔 때마다 정렬 상태를 새로 계산할 필요가 없다(자세한 설계는 Day 6에서 더 다룬다).

상황 5: 일별 활성 사용자(DAU) 체크 — String 기반 Bitmap

"오늘 접속한 유저인지"를 사용자마다 1비트로 표현해 SETBIT dau:2026-07-03 1001 1처럼 기록하면, 수백만 사용자의 접속 여부를 String 하나(수 MB 수준)로 표현할 수 있고 BITCOUNT로 순간 DAU를, BITOP AND로 "이틀 연속 접속자"까지 구할 수 있다. 이런 비트 연산 활용은 String의 숨은 실무 가치를 보여주는 대표 사례다.

1
접근 패턴부터 정의한다 "먼저 이 데이터를 통째로 읽고 쓰는지, 필드 단위로 부분 갱신하는지, 순서가 중요한지, 중복을 허용하지 않아야 하는지, 정렬된 범위 조회가 필요한지를 확인합니다."
2

3
내부 구현과 컬렉션 크기를 함께 고려한다 "작은 Hash는 listpack 인코딩으로 메모리 효율이 좋지만, 필드 수가 많아지면 hashtable로 전환되며 메모리 사용량 패턴이 바뀝니다. Set/Sorted Set도 컬렉션이 커질수록 특정 명령의 비용이 커진다는 걸 감안합니다."
4
싱글 스레드 리스크를 점검한다 "SMEMBERS, SINTER, ZRANGE를 대규모 컬렉션에 그대로 쓰면 O(N) 이상 비용이 그 순간 Redis 전체를 지연시킬 수 있어, SCAN 계열 명령이나 컬렉션 크기 제한으로 방어합니다."
5
Trade-off로 마무리한다 (Senior 포인트) "자료구조 선택은 '더 좋은 것'을 고르는 게 아니라, 접근 패턴·메모리 특성·명령 복잡도 사이의 트레이드오프를 보고 그 상황에 맞는 것을 고르는 문제입니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Redis 자료구조는 '무엇을 저장하는가'가 아니라 '어떻게 접근하는가'를 기준으로 선택합니다. 예를 들어 사용자 프로필처럼 필드 하나만 자주 부분 갱신되는 데이터는 String에 JSON을 통째로 넣기보다 Hash로 필드 단위로 저장합니다. String(JSON)은 필드 하나만 바꿔도 전체를 읽고 파싱하고 다시 써야 해서 네트워크 비용이 크고, 동시에 여러 요청이 서로 다른 필드를 고치면 나중 쓰기가 먼저 쓰기를 덮어써버리는 lost update가 생기기 쉽습니다. Hash는 HSET으로 필드 단위 원자적 갱신이 가능하고, 필드 수가 적을 때는 listpack이라는 압축 인코딩을 써서 오히려 메모리 효율도 좋습니다.

순서가 중요하고 최근 N개만 유지하면 되는 데이터, 예를 들어 최근 조회 상품 목록은 List에 LPUSH와 LTRIM을 함께 씁니다. 중복을 허용하지 않고 소속 여부만 빠르게 확인하면 되는 팔로우 관계나 태그는 Set을 씁니다. 반대로 점수 기준으로 항상 정렬된 상태를 유지하며 범위로 조회해야 하는 랭킹이나 지연 큐는 Sorted Set을 씁니다. Sorted Set은 skip list와 hash table을 함께 써서, 특정 원소의 점수 조회와 범위 조회를 모두 효율적으로 처리합니다.

다만 자료구조를 고를 때 컬렉션의 크기도 함께 고려합니다. Redis는 기본적으로 싱글 스레드로 명령을 처리하기 때문에, SMEMBERS나 SINTER, ZRANGE 같은 명령을 원소가 수십만~수백만 개인 컬렉션에 그대로 실행하면 그 한 번의 명령이 다른 모든 클라이언트의 요청까지 지연시킬 수 있습니다. 그래서 큰 컬렉션은 SCAN 계열 명령으로 나눠서 순회하거나, 애초에 컬렉션 크기에 상한을 두는 설계를 함께 고려합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Hash 안에 필드가 수만 개까지 늘어나면 무슨 일이 생기나요?"

약한 답변: "Hash는 필드가 많아도 상관없이 잘 동작합니다."

강한 답변: "필드 수나 값 크기가 설정된 임계값(hash-max-listpack-entries, hash-max-listpack-value)을 넘으면 메모리 효율적인 listpack 인코딩에서 일반 해시 테이블 인코딩으로 자동 전환됩니다. 이후에도 개별 HGET/HSET은 평균 O(1)로 동작하지만, 메모리 사용량 패턴이 달라지고 HGETALL처럼 전체를 가져오는 연산은 필드 수에 비례해 비용이 커집니다. 그래서 필드가 계속 늘어날 가능성이 있는 데이터는 애초에 여러 개의 Hash 키로 나누는 버킷팅을 고려합니다."

F2. "SMEMBERS를 프로덕션에서 그대로 쓰면 안 되는 이유는?"

강한 답변: "SMEMBERS는 Set의 원소 전체를 한 번에 반환하는 O(N) 명령입니다. Redis는 싱글 스레드이기 때문에, 원소가 수십만 개인 Set에서 이 명령이 실행되는 동안 다른 모든 요청이 대기하게 됩니다. 전체 목록이 정말 필요하다면 SSCAN으로 커서 기반으로 나눠서 가져오거나, 애초에 큰 Set을 만들지 않도록 데이터를 분할하는 설계를 고려해야 합니다."

F3. "List를 메시지 큐로 쓸 때 뭘 조심해야 하나요?"

강한 답변: "LPUSH로 넣고 BRPOP으로 블로킹 방식으로 꺼내면 간단한 큐처럼 쓸 수 있지만, List는 메시지 큐 전용 구조가 아닙니다. 소비자가 메시지를 꺼내자마자 목록에서 사라지기 때문에, 처리 중 소비자가 죽으면 그 메시지는 유실됩니다. 컨슈머 그룹, 처리 확인(ack), 재처리 같은 기능이 필요하다면 List가 아니라 Redis Streams(XADD, XREADGROUP, XACK)를 쓰는 게 맞습니다. List는 단순하고 순서만 보장하면 되는 가벼운 용도에 한정해서 씁니다."

F4. "Sorted Set으로 랭킹을 만들 때 점수가 같은 동점자는 어떻게 정렬되나요?"

강한 답변: "Sorted Set은 score가 같으면 멤버 이름을 사전순으로 정렬합니다. 실무에서는 이걸 그대로 쓰기보다, 동점자를 먼저 도달한 순서로 보여주고 싶다면 실제 점수 * 큰 상수 - 타임스탬프 같은 방식으로 하나의 숫자에 '점수 우선, 시간 차선'을 함께 인코딩해서 score로 씁니다. 그러면 별도 정렬 로직 없이 ZRANGE 하나로 원하는 동점자 처리 순서까지 얻을 수 있습니다."

F5. "Set 대신 Sorted Set을 쓸 수도 있는데, 왜 굳이 Set을 따로 쓰나요?"

강한 답변: "정렬이나 점수가 필요 없다면 Set이 메모리와 명령 비용 모두에서 더 유리합니다. Sorted Set은 정렬 상태를 유지하기 위해 skip list를 추가로 관리하므로 같은 원소 수라도 메모리 오버헤드가 더 크고, 삽입 비용도 Set의 평균 O(1)에 비해 O(log N)입니다. '정렬이 필요한가'라는 질문 하나로 두 구조 중 무엇을 쓸지 결정할 수 있습니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 접근 패턴을 기준으로 자료구조를 선택하는 논리를 먼저 세운다
  • 각 자료구조의 내부 인코딩/구현(listpack, quicklist, skip list)이 성능·메모리에 미치는 영향을 안다
  • Redis가 싱글 스레드라는 제약이 큰 컬렉션에 대한 명령 선택에 미치는 영향을 설명한다
  • List를 메시지 큐로 쓸 때의 한계와 Streams와의 차이를 구분한다
  • "더 좋은 자료구조"가 아니라 trade-off 관점으로 답한다

약한 답변의 특징

  • "객체는 Hash, 목록은 List, 집합은 Set"처럼 명칭 매칭만 하고 근거가 없다
  • String(JSON)과 Hash의 차이를 메모리·동시성 관점에서 설명하지 못한다
  • SMEMBERS, SINTER 같은 명령을 대규모 데이터에 그대로 써도 된다고 생각한다
  • List = Queue로 단순화하고, 메시지 유실 가능성을 인지하지 못한다
  • Sorted Set과 List, Sorted Set과 Set의 차이를 명확히 구분하지 못한다
핵심 차이: 강한 답변은 자료구조를 "데이터의 모양"이 아니라 "데이터에 무슨 연산을, 얼마나 자주, 얼마나 큰 규모로 할 것인가"의 관점에서 고른다. 이 관점 전환이 주니어와 시니어를 가르는 지점이다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Hash가 항상 String(JSON)보다 메모리를 더 쓴다"

사실: 필드 수가 적고 값이 짧을 때 Hash는 listpack이라는 압축 인코딩을 쓰기 때문에, 오히려 JSON 문자열을 String으로 저장하는 것보다 메모리 효율이 좋을 수 있다. 다만 필드 수가 많아지면 hashtable 인코딩으로 전환되며 메모리 사용량이 늘어나므로, "항상 좋다/나쁘다"가 아니라 필드 수 규모에 따라 달라진다.

오해 2: "List는 곧 안전한 메시지 큐다"

사실: List 기반 큐는 소비자가 메시지를 꺼내는 순간 목록에서 사라지고, 처리 실패 시 재처리 메커니즘이 없다. 컨슈머 그룹, ack, 재처리, 순서 보장이 필요한 메시징에는 Redis Streams가 더 적합한 도구다.

오해 3: "Set 연산(SINTER, SUNION)은 항상 빠르다"

사실: Set 연산은 원소 수에 비례해 시간이 걸리는 O(N) 이상의 명령이다. Redis가 싱글 스레드로 명령을 처리하기 때문에, 대규모 Set에 이 연산을 그대로 실행하면 그 순간 다른 요청까지 지연되는 "느린 명령이 전체를 막는" 구조적 위험이 있다.

오해 4: "Sorted Set은 그냥 정렬된 List다"

사실: List는 삽입 순서를 그대로 유지하는 큐/스택 구조이고, Sorted Set은 score 값을 기준으로 항상 재정렬된 상태를 유지하는 랭킹 구조다. 내부적으로도 List는 quicklist, Sorted Set은 skip list + hash table로 전혀 다른 구현을 쓰며, 지원하는 연산(범위 조회, 점수 조회, 순위 조회)도 다르다.

핵심 요약

오늘 반드시 기억할 것

  • 자료구조 선택은 "무엇을 저장하나"가 아니라 "어떻게 접근하나"로 결정한다.
  • Hash는 필드 단위 부분 갱신이 필요할 때 String(JSON)보다 원자성과 메모리 효율 모두 유리할 수 있다.
  • List는 순서 있는 최근 N개 유지에 적합하지만, 안전한 메시지 큐 대체재는 아니다(Streams가 더 적합).
  • Set은 중복 제거·소속 확인에 O(1)로 강하지만, 대규모 Set에 SMEMBERS/SINTER를 쓰면 싱글 스레드 전체를 지연시킬 수 있다.
  • Sorted Set은 skip list + hash table로 정렬 상태를 유지하며 범위 조회를 지원하는, List/Set과 근본적으로 다른 구조다.
  • 모든 선택에는 내부 인코딩과 컬렉션 크기에 따른 trade-off가 따른다는 것을 항상 함께 설명한다.
