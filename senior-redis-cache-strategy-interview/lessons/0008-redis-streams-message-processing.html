Redis Streams와 메시지 처리

Day 7의 "유실돼도 되는 이벤트"를 넘어 — 놓치면 안 되는 이벤트를 Redis 하나로 감당하는 법
면접 질문
Q1 — 주력 질문 "주문 생성 이벤트를 여러 서비스(재고, 정산, 배송)에 전파해야 합니다. Kafka를 새로 도입하기엔 인프라가 부담스러워서 Redis Streams로 처리하자는 의견이 나왔습니다. 이게 합리적인 선택인지, 그리고 어떤 조건에서 위험해지는지 설명해 보세요."
Q2 — 변형 "Consumer Group으로 메시지를 처리하던 워커 하나가 메시지를 읽어간 직후 크래시했습니다. 그 메시지는 어떻게 되나요? 다른 워커가 대신 처리하게 하려면 무엇을 해야 합니까?"
Q3 — 변형 "Stream에 초당 수천 건씩 데이터가 쌓이는데 아무 트리밍(trimming)도 설정하지 않았습니다. 몇 주 뒤에 무슨 일이 생길까요?"
전제 개념

Streams는 "저장되는 Pub/Sub" — append-only 로그다

Day 7에서 Pub/Sub은 "방송"이라 메시지가 저장되지 않고, 구독자가 없으면 그대로 사라진다고 배웠다. Redis Streams는 정확히 그 약점을 메우기 위해 나온 자료구조다. Stream은 각 메시지(entry)에 고유 ID를 붙여 순서대로 로그에 계속 append하는 구조다. Kafka의 토픽(topic)과 개념적으로 거의 동일하다고 생각하면 된다 — 메시지가 사라지지 않고 로그에 남아 있으니, 나중에 연결한 컨슈머도 과거 메시지를 처음부터 다시 읽을 수 있다. 

# 주문 생성 이벤트를 orders 스트림에 추가 (ID는 자동 생성: timestamp-seq)
XADD orders * order_id 1001 status "created" amount 39000

# 스트림 전체를 처음부터 읽기 (과거 메시지도 그대로 남아 있음)
XRANGE orders - +

# 특정 ID 이후의 새 메시지만 블로킹으로 대기하며 읽기
XREAD BLOCK 5000 STREAMS orders $
ID의 의미: XADD가 자동 생성하는 ID는 <millisecond-timestamp>-<sequence> 형태다(예: 1720000000000-0). 이 ID는 항상 이전 ID보다 커야 하고, Stream 안의 메시지는 이 ID 순서로 엄격하게 정렬된다. 이 특성 덕분에 "이 ID 이후부터 다시 읽어달라"는 재처리(replay)가 가능해진다.

Consumer Group — 여러 워커가 나눠서, 중복 없이, 안전하게

Pub/Sub의 가장 큰 문제는 "구독자 전원이 모든 메시지를 받는다"는 것이었다. Streams는 Consumer Group이라는 개념으로 이를 해결한다. 하나의 그룹에 속한 여러 컨슈머(워커)는 같은 메시지를 딱 한 명만 배정받아 처리한다 — 즉 로드밸런싱이 된다. 그리고 각 컨슈머가 "이 메시지까지 읽었다"는 지점(offset 개념)을 그룹 단위로 기억하기 때문에, 워커를 몇 대로 늘리든 줄이든 중복 처리 없이 나눠 가진다. 

# orders 스트림에 order-processing이라는 Consumer Group 생성 (처음부터 읽기: $는 "지금부터"의미)
XGROUP CREATE orders order-processing 0

# worker-1이 그룹의 이름으로 아직 아무도 안 읽은 메시지를 하나 읽어감
XREADGROUP GROUP order-processing worker-1 COUNT 1 STREAMS orders >

# 처리가 끝났으면 반드시 ACK — 안 하면 이 메시지는 "미처리(pending)" 목록에 계속 남는다
XACK orders order-processing 1720000000000-0
핵심 함정: XREADGROUP으로 메시지를 읽어갔다고 처리가 끝난 게 아니다. Redis 입장에서는 "이 워커에게 배정했다"는 사실만 기록하고, 실제로 처리에 성공했다는 것은 XACK를 호출해야만 안다. 워커가 메시지를 읽은 직후 크래시하면, 그 메시지는 ACK되지 않은 채 PEL(Pending Entries List)에 "누군가에게 배정됐지만 아직 확인 안 됨" 상태로 남는다.

PEL과 XCLAIM/XAUTOCLAIM — 죽은 워커의 메시지를 되찾아오기

PEL에 남은 메시지를 그대로 두면 영원히 처리되지 않는다. 그래서 다른 워커(또는 감시자 프로세스)가 주기적으로 "일정 시간 이상 ACK되지 않은 메시지"를 찾아서 자신에게 재배정(claim)한다. 

# 그룹 내에서 60초 이상 idle 상태(ACK 안 됨)인 메시지 목록 확인
XPENDING orders order-processing IDLE 60000

# worker-2가 그 메시지들을 자신에게 재배정받아 다시 처리 시도 (Redis 6.2+ 권장 방식)
XAUTOCLAIM orders order-processing worker-2 60000 0

이 메커니즘 덕분에 Streams는 at-least-once(최소 한 번 처리)를 보장할 수 있다. 단, "최소 한 번"이라는 말은 같은 메시지가 두 번 처리될 수도 있다는 뜻이므로, 비즈니스 로직은 반드시 멱등성(idempotency)을 갖춰야 한다 — 예를 들어 재고 차감을 "차감한다"가 아니라 "이 order_id는 이미 차감했는지 확인 후 차감한다"로 설계해야 한다.

메모리 관리 — Stream은 자동으로 줄어들지 않는다

Stream은 로그이기 때문에 XADD를 계속 호출하면 메모리가 계속 늘어난다. TTL을 걸어도 Stream 전체가 한 번에 사라질 뿐, 항목 단위로 자동 삭제되지 않는다. 그래서 MAXLEN으로 최대 길이를 제한하거나, 처리 완료된 항목을 XTRIM/XDEL로 직접 정리해야 한다. 

# 최근 10만 건만 남기고 오래된 항목은 자동 삭제 (근사치 트리밍, 성능 우수)
XADD orders MAXLEN ~ 100000 * order_id 1002 status "created"
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • 전달 보장(delivery guarantee) 수준을 정확히 구분하는 능력: Pub/Sub의 at-most-once, Streams의 at-least-once, 그리고 "정확히 한 번(exactly-once)"은 애플리케이션 레벨의 멱등성으로만 근사할 수 있다는 것을 아는가
  • 장애 시나리오를 구체적으로 그릴 수 있는가: 워커 크래시, ACK 누락, PEL 적체 같은 실패 상황에서 시스템이 어떻게 스스로 회복하는지 설계했는가
  • 운영 비용 트레이드오프 판단: Kafka/RabbitMQ 같은 전용 메시징 인프라 대비, 이미 쓰고 있는 Redis로 처리 규모를 감당할 수 있는지 냉정하게 판단하는가
  • 무한 성장 자료구조에 대한 경계심: 트리밍 없이 방치하면 메모리를 잠식한다는 운영 감각이 있는가
이 주제는 "Streams 명령어를 아는가"가 아니라, "메시지가 유실되면 안 되는 상황에서 Redis만으로 충분한지, 언제 전용 메시징 시스템으로 넘어가야 하는지"를 판단하는 아키텍처 감각을 본다.
실무 연결

실제 상황과 연결하기

상황 1: 이미 Redis만 쓰는 소규모 서비스의 이벤트 전파

트래픽이 크지 않고, 별도 메시징 인프라(Kafka 클러스터 운영 인력, 모니터링)를 갖추기 부담스러운 팀에서는 이미 캐시/세션용으로 쓰고 있는 Redis에 Streams를 추가하는 것만으로 order-created, payment-completed 같은 이벤트를 안전하게 여러 서비스에 전파할 수 있다. 새로운 인프라 없이 at-least-once 보장을 얻는 실용적 선택이다.

상황 2: 활동 피드(activity feed) / 타임라인

"내가 팔로우한 사람들의 최근 활동"처럼 시간순 로그가 필요한 기능은 Stream의 ID 자체가 시간 순서를 보장하므로 잘 맞는다. XRANGE로 특정 구간을 페이지네이션하듯 조회할 수 있고, MAXLEN으로 사용자별 최근 N개만 유지하는 것도 자연스럽다.

상황 3: 여러 워커가 나눠 처리해야 하는 백그라운드 작업

이미지 리사이징, 알림 발송처럼 "작업을 여러 워커가 나눠서(load balancing), 하나는 딱 한 번만 처리"해야 하는 케이스는 Pub/Sub으로 구현하면 중복 처리 사고가 난다는 것을 Day 7에서 배웠다. Consumer Group을 쓰면 워커 대수를 늘리거나 줄여도 자동으로 작업이 재분배된다.

상황 4: 트래픽이 크게 성장한 이후 — Kafka로의 이관 시점

Streams는 단일 Redis 인스턴스(혹은 Cluster의 단일 슬롯)에 담기는 구조라, 파티션을 통해 처리량을 수평 확장하는 Kafka만큼 확장성이 크지 않다. 초당 수만 건 이상의 이벤트, 여러 팀이 공유하는 이벤트 버스, 장기 보관·재처리(replay) 요구가 커지면 Kafka로 이관을 검토해야 한다. 면접에서는 "무조건 Streams로 버틴다"가 아니라 이런 성장 시점을 미리 인지하고 있는지를 본다.

상황 5: 크래시 복구 — 결제 워커가 죽었을 때

결제 완료 이벤트를 처리하던 워커가 XREADGROUP으로 메시지를 읽어간 직후 크래시했다면, 그 메시지는 PEL에 남는다. 감시(watchdog) 프로세스가 주기적으로 XAUTOCLAIM을 돌려 idle 시간이 긴 메시지를 다른 워커에게 넘겨 처리를 이어가야 한다. 이 복구 로직이 없으면 워커 하나가 죽는 순간 그 메시지는 영원히 처리되지 않고 방치된다.

3

핵심 답변 예시

면접 현장 답변 예시 (전체)

"네, 조건부로 합리적인 선택이라고 생각합니다. Redis Streams는 Pub/Sub과 달리 메시지를 로그처럼 영속적으로 저장하고, Consumer Group과 ACK 메커니즘으로 at-least-once 전달을 보장하기 때문에 주문 생성처럼 '놓치면 안 되는' 이벤트에도 쓸 수 있습니다. 이미 캐시나 세션용으로 Redis 인프라와 운영 노하우가 있는 팀이라면, 새로 Kafka 클러스터를 구축하고 운영 인력을 투입하는 것보다 현재 트래픽 규모에서는 더 실용적일 수 있습니다.

다만 몇 가지 조건을 확인해야 합니다. 첫째, 처리 워커가 죽었을 때의 회복 전략 — ACK 안 된 메시지는 PEL(Pending Entries List)에 남고, 이를 XAUTOCLAIM으로 다른 워커가 넘겨받아 재처리하도록 설계해야 합니다. 이 과정에서 메시지가 두 번 처리될 수 있으므로 재고 차감, 정산 로직은 반드시 멱등하게 만들어야 합니다. 둘째, Stream은 자동으로 줄어들지 않으므로 MAXLEN 트리밍을 설정하지 않으면 메모리가 계속 늘어난다는 점을 운영에 반영해야 합니다.

마지막으로, Streams는 사실상 단일 Redis 인스턴스에 담기는 구조라 Kafka의 파티션 기반 수평 확장만큼 처리량을 키우기는 어렵습니다. 트래픽이 초당 수만 건 이상으로 커지거나, 여러 팀이 공유하는 이벤트 버스가 필요해지면 그 시점에는 Kafka로의 이관을 검토하는 것이 맞다고 봅니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Consumer Group에서 동일 메시지가 중복 처리되는 걸 완전히 막을 방법이 있나요?"

강한 답변: "Streams의 보장 수준 자체가 at-least-once이기 때문에 프로토콜 레벨에서 중복을 완전히 막을 수는 없습니다. 대신 메시지에 담긴 order_id 같은 고유 키를 기준으로 '이미 처리했는지' 확인 후 처리하는 멱등성 로직으로 애플리케이션 레벨에서 exactly-once에 가깝게 근사합니다. 예를 들어 처리 결과를 SETNX로 먼저 마킹하고, 이미 마킹되어 있으면 스킵하는 방식입니다."

F2. "XREADGROUP과 XREAD의 차이는 무엇이고, 언제 XREAD를 써야 하나요?"

강한 답변: "XREAD는 Consumer Group 없이 스트림을 직접 읽는 방식으로, 각 컨슈머가 독립적으로 전체 메시지를 다 받아야 하는 상황(예: 여러 서비스가 각자 전체 이벤트를 다 봐야 하는 fan-out)에 적합합니다. XREADGROUP은 그룹 내에서 메시지를 나눠 가지며 ACK/PEL로 처리 상태를 추적하므로, 같은 작업을 여러 워커가 분담해야 하는 로드밸런싱 상황에 적합합니다."

F3. "Stream이 계속 커지면 메모리가 부족해질 텐데, 어떻게 대응하나요?"

강한 답변: "XADD에 MAXLEN ~ 옵션을 걸어 근사치 트리밍을 상시 적용하거나, 모든 Consumer Group이 처리를 마친 항목을 주기적으로 XTRIM/XDEL로 정리합니다. 트리밍 기준을 너무 짧게 잡으면 아직 ACK되지 않은 메시지가 삭제될 위험이 있으니, 모든 그룹의 최소 확인 지점(XINFO GROUPS로 확인 가능한 last-delivered-id)보다 뒤쪽만 트리밍하도록 설계합니다."

F4. "Redis Cluster 환경에서 Streams를 쓸 때 주의할 점은?"

강한 답변: "Stream 키 하나는 하나의 슬롯(그리고 결국 하나의 노드)에만 존재합니다. 즉 Kafka 파티션처럼 하나의 논리적 스트림이 여러 노드에 자동으로 분산되지 않습니다. 처리량을 늘리려면 애플리케이션 레벨에서 order:{shard} 처럼 여러 개의 Stream 키로 직접 샤딩해야 하고, 이는 곧 여러 개의 독립된 Consumer Group 관리 부담으로 이어집니다."

F5. "그냥 DB 테이블에 outbox 패턴으로 이벤트를 쌓고 폴링하면 안 되나요?"

강한 답변: "Transactional Outbox 패턴은 DB 트랜잭션과 이벤트 발행을 원자적으로 묶을 수 있어 신뢰성 면에서는 더 강력합니다. 다만 폴링 주기만큼 지연이 생기고 DB에 추가 부하가 갑니다. Streams는 지연이 훨씬 낮고 Consumer Group으로 실시간에 가깝게 처리할 수 있다는 장점이 있어, '정합성이 최우선이면 outbox, 지연시간이 중요하면 Streams'로 상황에 따라 선택하거나 두 패턴을 병행하기도 합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Streams의 at-least-once 보장과 그 근거(로그 저장 + PEL/ACK)를 정확히 설명한다
  • 워커 크래시 시 PEL과 XAUTOCLAIM으로 이어지는 복구 경로를 구체적으로 그린다
  • 중복 처리 가능성을 인정하고 멱등성 설계를 함께 제시한다
  • 메모리 무한 성장 문제와 트리밍 전략을 스스로 짚는다
  • Kafka 대비 확장성 한계와 이관 시점을 명확히 인지한다

약한 답변의 특징

  • "Streams는 Pub/Sub보다 좋은 버전"이라고만 하고 보장 수준의 근거를 설명 못한다
  • ACK 개념을 모르고 XREADGROUP만으로 처리가 끝난다고 착각한다
  • 워커 크래시 시 메시지가 어떻게 되는지 답하지 못한다
  • Stream이 자동으로 정리된다고 잘못 알고 있다
  • 무조건 "Redis로 다 되니 Kafka는 필요 없다"고 확장 한계를 고려하지 않는다
핵심 차이: 강한 답변은 Streams를 "메시지가 저장되는 Pub/Sub"으로 끝내지 않고, "장애 상황에서 무엇이 보장되고, 그 대가로 무엇을 애플리케이션이 책임져야 하는가"까지 말한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "XREADGROUP으로 메시지를 읽어가면 처리가 끝난 것이다"

사실: 읽어간 것은 "배정받았다"는 뜻일 뿐이다. 실제 처리 완료는 XACK를 호출해야 Redis가 안다. ACK를 빼먹으면 정상적으로 처리했더라도 그 메시지는 PEL에 계속 남아 나중에 다른 워커가 중복 처리하게 된다.

오해 2: "Streams는 exactly-once를 보장한다"

사실: Streams가 프로토콜 레벨에서 보장하는 것은 at-least-once다. 워커 크래시 후 재배정(XCLAIM/XAUTOCLAIM) 과정에서 같은 메시지가 두 번 처리될 수 있다. exactly-once에 가까운 동작은 애플리케이션이 멱등성 로직을 추가해야만 얻을 수 있다.

오해 3: "TTL을 걸어두면 오래된 메시지가 알아서 지워진다"

사실: 일반 키의 TTL은 Stream 키 전체에만 적용되고, 만료 시 스트림 전체가 사라진다. 항목 단위 정리는 TTL이 아니라 MAXLEN 트리밍이나 XTRIM/XDEL을 직접 호출해야만 이뤄진다.

오해 4: "Redis Streams만 있으면 Kafka는 아예 필요 없다"

사실: Stream 키는 하나의 슬롯/노드에 묶이므로 Kafka의 파티션 기반 수평 확장만큼 처리량을 키우기 어렵다. 이벤트량이 크게 늘거나 여러 팀이 공유하는 대규모 이벤트 버스가 필요해지면 Kafka 같은 전용 메시징 인프라로 이관하는 것이 맞다.

핵심 요약

오늘 반드시 기억할 것

  • Redis Streams는 append-only 로그로 메시지를 영속 저장해, Pub/Sub과 달리 재처리(replay)가 가능하다.
  • Consumer Group은 ACK/PEL로 처리 상태를 추적해 at-least-once를 보장한다 — 워커가 죽어도 XAUTOCLAIM으로 다른 워커가 이어받는다.
  • at-least-once는 중복 처리 가능성을 내포하므로 비즈니스 로직은 반드시 멱등하게 설계해야 한다.
  • Stream은 자동으로 줄어들지 않는다 — MAXLEN 트리밍이나 XTRIM으로 직접 관리해야 한다.
  • Stream 키는 단일 슬롯/노드에 묶이므로 Kafka만큼 수평 확장이 쉽지 않다 — 대규모 트래픽에서는 이관 시점을 미리 판단해야 한다.
