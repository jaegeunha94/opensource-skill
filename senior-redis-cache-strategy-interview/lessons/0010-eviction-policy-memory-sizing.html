Day 10 · Redis / 캐시 전략 Interview

Eviction Policy와 메모리 사이징

"메모리가 가득 차면 Redis는 어떻게 되는가" — maxmemory 도달 시 동작을 설계하고, 필요한 메모리를 정확히 산정하는 법
면접 질문
Q1 — 주력 질문 "운영 중인 Redis가 maxmemory 한계에 도달했습니다. 지금 어떤 maxmemory-policy가 설정돼 있는지에 따라 서비스에 각각 어떤 일이 벌어지는지 설명해 주세요. 그리고 이 Redis 인스턴스가 순수 캐시인지, 세션 데이터도 함께 담고 있는지에 따라 어떤 정책을 선택하시겠습니까?"
Q2 — 변형 "Redis의 LRU는 '완벽한' LRU가 아니라고 들었습니다. 실제로는 어떻게 동작하고, 이게 왜 정확도와 성능의 트레이드오프가 되는지 설명해 주세요."
Q3 — 변형 "서버 물리 메모리가 64GB인 신규 Redis 클러스터를 구축합니다. maxmemory를 몇 GB로 잡으시겠습니까? 그 숫자를 어떻게 산정하는지 근거를 말씀해 주세요."
전제 개념

Redis가 메모리 한계에 부딪히면 무슨 일이 벌어지는가

Redis는 데이터를 전부 RAM에 올려두고 쓰는 인메모리 데이터베이스다. RAM은 디스크와 달리 용량이 제한적이고 비싸기 때문에, Redis 인스턴스에는 보통 maxmemory로 사용할 수 있는 메모리 상한을 명시적으로 걸어둔다. 문제는 실제로 그 상한에 도달했을 때 Redis가 무엇을 할지는 기본값만으로는 원하는 대로 동작하지 않는다는 점이다.

가장 많이 놓치는 사실: Redis의 maxmemory-policy 기본값은 noeviction이다. 즉, 아무것도 설정하지 않으면 Redis는 알아서 오래된 키를 지우지 않는다. 대신 메모리가 꽉 찬 상태에서 새 쓰기 명령이 들어오면 그 명령을 거부하고 OOM command not allowed 에러를 반환한다. "캐시니까 알아서 오래된 것부터 지워지겠지"라고 가정하고 아무 설정도 하지 않으면, 트래픽이 몰리는 순간 쓰기가 전부 실패하는 장애로 이어진다.

maxmemory-policy — 8가지 선택지

Redis가 메모리 한계에 도달했을 때 취할 행동은 maxmemory-policy로 선택한다. 두 축으로 나눠서 이해하면 외우기 쉽다: 어떤 키를 대상으로 삼을지(allkeys vs volatile)와 어떤 기준으로 고를지(LRU / LFU / random / TTL).

정책 대상 키 선택 기준
noeviction 없음 (evict 안 함) 쓰기 명령을 거부하고 에러 반환 (읽기는 계속 가능)
allkeys-lru 전체 키 가장 오래 접근 안 된 키부터 제거 (근사 LRU)
allkeys-lfu 전체 키 접근 빈도가 가장 낮은 키부터 제거 (근사 LFU)
allkeys-random 전체 키 무작위로 제거
volatile-lru TTL이 설정된 키만 가장 오래 접근 안 된 키부터 제거
volatile-lfu TTL이 설정된 키만 접근 빈도가 가장 낮은 키부터 제거
volatile-random TTL이 설정된 키만 무작위로 제거
volatile-ttl TTL이 설정된 키만 남은 TTL이 가장 짧은 키부터 제거
allkeys vs volatile의 핵심 차이: volatile-* 계열은 TTL이 걸려 있지 않은 키는 절대 건드리지 않는다. 이 특성 덕분에, 같은 Redis 인스턴스에 "만료돼도 되는 캐시 데이터(TTL 있음)"와 "절대 사라지면 안 되는 원본 데이터(TTL 없음, 예: 세션·설정값)"를 함께 둘 때 후자를 보호하는 용도로 쓸 수 있다. 반대로 모든 키가 TTL을 갖고 있다면 volatile-lru는 사실상 allkeys-lru와 동일하게 동작한다.

Redis의 LRU는 "근사(approximated) LRU"다

진짜 LRU를 구현하려면 모든 키를 접근 순서대로 정렬한 연결 리스트를 유지하고, 접근할 때마다 그 키를 리스트 맨 앞으로 옮겨야 한다. 이건 정확하지만 모든 읽기/쓰기마다 추가 연산과 메모리 오버헤드가 발생한다. Redis는 이 비용을 감당하지 않기 위해 다른 전략을 쓴다:

  1. 모든 키 객체에 최근 접근 시각을 담는 24비트 타임스탬프(LRU clock)를 붙여둔다.
  2. evict가 필요해지면, 전체 키를 순회하는 대신 무작위로 N개(기본값 5, maxmemory-samples)를 뽑아 그중 가장 오래된 것을 제거한다.
  3. Redis 4 이상에서는 후보 풀(pool)을 유지해 샘플링 정확도를 조금 더 높인다.
# redis.conf — 샘플 개수. 클수록 진짜 LRU에 가까워지지만 CPU 비용 증가
maxmemory-samples 5  # 기본값. 실무에서 정확도를 높이려면 10 정도로 조정

즉 "가장 오래 안 쓴 키"가 아니라 "무작위 표본 중에서 가장 오래 안 쓴 키"가 제거된다. 샘플 수가 적을수록 오차가 커지지만 CPU 비용은 낮고, 샘플 수를 늘리면 진짜 LRU에 가까워지는 대신 evict 연산 자체의 비용이 늘어난다 — 이것도 하나의 trade-off다.

LFU — "최근성"이 아니라 "빈도"로 판단

LRU의 약점은 아주 자주 쓰이는 키라도 잠깐 접근이 뜸하면 evict 후보가 될 수 있다는 점이다. 예를 들어 인기 상품 정보처럼 하루 종일 꾸준히 조회되던 키가, 마침 direct 접근이 몇 초 뜸한 타이밍에 무작위 샘플에 걸리면 지워질 수 있다. LFU(Least Frequently Used)는 "얼마나 자주 쓰였는가"를 기준으로 삼아 이런 상황을 줄인다. Redis는 각 키에 8비트 확률적 카운터(probabilistic counter, Morris counter 방식)를 붙여 접근 빈도를 근사적으로 추적하고, 시간이 지나면 카운터 값을 서서히 감쇠(decay)시켜 "예전엔 자주 쓰였지만 지금은 아닌" 키도 결국 후보에 오르게 한다.

메모리 사이징 — "데이터 크기 = maxmemory"가 아니다

메모리 용량을 산정할 때 가장 흔한 실수는 저장할 값(value)들의 크기만 더해서 그게 곧 필요한 메모리라고 계산하는 것이다. 실제로 Redis가 소비하는 메모리는 그보다 훨씬 크다.

  • 키당 오버헤드: 모든 키는 dictEntry, robj(Redis Object) 헤더, 키 이름 자체의 SDS(Simple Dynamic String) 구조체 등 부가 구조를 함께 갖는다. 작은 값(예: 정수 하나)을 저장해도 키당 수십 바이트의 고정 오버헤드가 붙는다 — 값 자체보다 오버헤드가 더 클 수도 있다.
  • 인코딩 임계값: Hash/List/Set/Sorted Set은 요소 수가 적을 때 listpack(구 ziplist)처럼 압축된 표현을 쓰다가, hash-max-listpack-entries 같은 임계값을 넘으면 해시테이블 같은 완전한 구조로 전환된다. 이 전환 시점에 메모리 사용량이 계단식으로 뛴다.
  • 메모리 파편화(fragmentation): Redis의 기본 메모리 할당자(jemalloc)는 할당/해제가 반복되면 실제 사용량보다 더 많은 물리 메모리를 점유하게 된다. mem_fragmentation_ratio로 확인할 수 있고, 이 값이 1.5를 넘으면 파편화가 심하다는 신호다.
  • Replication backlog / client output buffer: 복제본에 변경분을 전달하기 위한 백로그 버퍼, Pub/Sub 구독자나 느린 클라이언트를 위한 output buffer도 maxmemory 예산 안에서 함께 소비된다.
  • fork 시 Copy-on-Write 스파이크(Day 9 연결): BGSAVE/ BGREWRITEAOF 도중에는 쓰기가 활발할수록 COW로 메모리 사용량이 순간적으로 크게 늘어난다.
왜 이게 위험한가: maxmemory를 물리 메모리 100%에 딱 맞춰 잡으면, fork COW 스파이크나 replication 버퍼 증가가 물리 메모리 자체를 초과시켜 OS가 Redis 프로세스를 OOM Killer로 강제 종료시킬 수 있다. 이건 Redis 내부의 eviction 정책과는 별개의 문제 — Redis 프로세스 자체가 죽는 것이라 maxmemory-policy가 무엇이든 소용없다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • "기본 설정을 그대로 쓰면 무슨 일이 벌어지는지" 아는가: noeviction이 기본값이라는 사실과, 그로 인해 캐시 용도 인스턴스에서 트래픽 피크에 쓰기 실패가 날 수 있다는 구체적 장애 시나리오를 아는지
  • eviction 대상 범위를 데이터 성격에 맞춰 설계하는가: 같은 인스턴스에 캐시와 원본 데이터가 섞여 있을 때 volatile-*와 TTL 유무를 조합해 원본 데이터를 보호할 수 있는지
  • Redis의 LRU/LFU가 "근사치"라는 내부 동작을 이해하는가: 샘플링 기반이라는 것, 정확도와 성능 사이의 trade-off를 maxmemory-samples 같은 파라미터로 조정할 수 있다는 것을 아는지
  • 메모리 사이징을 정량적으로 산정할 수 있는가: 데이터 크기뿐 아니라 키당 오버헤드, 인코딩 임계값, 파편화, fork COW 여유까지 감안해 maxmemory를 물리 메모리보다 낮게 잡아야 한다는 걸 아는지
이 주제는 "LRU와 LFU 차이를 아는가"로 끝나지 않는다. "이 Redis 인스턴스가 가득 찼을 때 비즈니스적으로 어떤 결과가 나야 정상인지 스스로 설계하고, 그 설계에 맞는 메모리 여유를 미리 확보할 수 있는가"를 평가한다.
실무 연결

실제 상황과 연결하기

상황 1: 순수 캐시 레이어에서 기본값(noeviction) 그대로 운영

DB 조회 결과만 캐싱하는 인스턴스에 maxmemory-policy를 별도로 설정하지 않으면 기본값인 noeviction이 그대로 적용된다. 트래픽이 늘어 캐시 키가 계속 쌓이다가 메모리가 꽉 차면, Redis는 알아서 오래된 캐시를 지우는 게 아니라 새 쓰기(SET)를 전부 거부한다. 캐시 워밍이나 신규 캐시 채우기 로직이 갑자기 에러를 뱉기 시작하고, 최악의 경우 애플리케이션이 그 에러를 제대로 처리하지 못해 요청 자체가 실패한다. 이런 용도라면 allkeys-lru 또는 allkeys-lfu로 명시적으로 바꿔야 한다.

상황 2: 캐시와 세션이 한 인스턴스에 공존

비용 절감을 위해 캐시(TTL 있음)와 세션 데이터(TTL 있을 수도, 없을 수도)를 같은 Redis 인스턴스에 두는 경우가 많다. 이때 allkeys-lru를 쓰면 접근 패턴에 따라 세션 데이터도 evict 대상이 되어 사용자가 갑자기 로그아웃될 수 있다. 세션은 만료 시점을 애플리케이션이 명시적으로 관리해야 하므로(예: 로그인 시 TTL 갱신) TTL을 걸어두되, 캐시 키만 evict 대상이 되도록 세션 키에는 애초에 TTL을 걸지 않고 volatile-lru를 쓰는 식의 구분이 필요하다 — 다만 이 경우 세션 만료는 애플리케이션 레벨(예: 별도 배치)로 처리해야 한다는 trade-off가 생긴다.

상황 3: 특정 상품/유저 데이터가 예측 못한 타이밍에 자주 사라짐

allkeys-lru로 운영 중인데, 하루 중 특정 시간대에 접근이 뜸했다가 다시 몰리는 인기 상품 데이터가 그 사이 evict돼서 캐시 미스가 튀는 패턴이 반복된다면 — 이건 LRU의 전형적인 약점이다. "자주 쓰이지만 순간적으로 조회가 뜸했던" 키가 "정말 안 쓰이는" 키와 구분되지 않기 때문이다. 이런 워크로드에서는 allkeys-lfu로 전환해 "빈도" 기준으로 판단하게 하면 개선되는 경우가 많다.

상황 4: BGSAVE 직후 메모리 사용량 급증으로 evict가 과도하게 발생

Day 9에서 다룬 것처럼 BGSAVE는 fork/COW로 순간적인 메모리 스파이크를 유발한다. maxmemory를 여유 없이 빡빡하게 잡아둔 상태에서 이 스파이크가 발생하면, 실제로는 필요하지 않은 시점에 evict가 과도하게 트리거되어 캐시 히트율이 급락하는 부작용이 생길 수 있다. 메모리 사이징 시 fork COW 여유를 별도로 감안해야 하는 이유다.

상황 5: 신규 클러스터 용량 산정 미팅

"예상 데이터 총량이 20GB니까 32GB 인스턴스면 충분하다"는 계산은 키당 오버헤드, 파편화, replication buffer, fork COW 여유를 전혀 고려하지 않은 것이다. 실무에서는 순수 데이터 크기 추정값에 1.5~2배 정도의 여유를 곱해 인스턴스 크기를 정하고, maxmemory는 그 물리 메모리보다 더 낮은 값(예: 물리 메모리의 70~80%)으로 설정해 OS 레벨 OOM을 방지하는 것이 표준적인 접근이다.

모니터링 지표로 검증 루프를 닫는다 (Senior 포인트) "설정 이후에는 evicted_keys, used_memory, mem_fragmentation_ratio, OOM 에러율 같은 지표를 대시보드로 계속 관찰해서, 설계 당시의 가정이 실제 트래픽과 맞는지 검증합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"먼저 이 Redis가 어떤 데이터를 담고 있는지부터 구분합니다. 순수 DB 캐시라면 재생성이 가능하므로 allkeys-lru나 allkeys-lfu를 명시적으로 설정합니다. 기본값인 noeviction을 그대로 두면 메모리가 찼을 때 오래된 키를 지우는 대신 새 쓰기 자체를 거부하기 때문에, 트래픽 피크에 캐시 갱신이 통째로 실패하는 장애로 이어질 수 있습니다.

반대로 같은 인스턴스에 세션처럼 원본 데이터가 함께 있다면, 그 키에는 TTL을 걸지 않고 volatile-lru를 사용해서 evict 대상에서 제외합니다. LRU와 LFU 중에서는 접근 패턴이 최근성 위주인지, 특정 인기 키가 시간대와 무관하게 꾸준히 조회되는지에 따라 선택합니다. 다만 Redis의 LRU/LFU는 전체 키를 정렬하는 게 아니라 maxmemory-samples 개수만큼 무작위로 뽑아 그중 가장 오래됐거나 빈도가 낮은 걸 지우는 근사치이기 때문에, 정확도가 중요하다면 샘플 수를 늘리는 것도 고려합니다.

메모리 용량은 저장할 값의 크기만 더해서 계산하지 않습니다. 키당 dictEntry/robj 오버헤드, Hash나 Sorted Set이 listpack에서 해시테이블로 전환될 때의 메모리 증가, jemalloc 파편화, replication backlog, 그리고 BGSAVE 시 fork COW로 인한 순간적인 메모리 스파이크까지 감안해서, 실제 데이터 크기 추정값에 여유를 두고 maxmemory를 물리 메모리의 70~80% 선으로 설정합니다. 이렇게 하면 Redis 내부 eviction 로직과 별개로 OS OOM Killer에 의해 프로세스 전체가 죽는 상황을 막을 수 있습니다. 설정 이후에는 evicted_keys와 mem_fragmentation_ratio를 계속 모니터링해서 설계 가정이 실제와 맞는지 확인합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "noeviction이 기본값인 게 왜 위험한가요? 그렇다면 왜 Redis는 이걸 기본값으로 뒀을까요?"

강한 답변: "위험한 이유는 캐시 용도로 쓰는 팀이 이 사실을 놓치면 쓰기가 조용히 실패하기 시작한다는 점입니다. 다만 Redis 입장에서는 noeviction이 안전한 기본값입니다 — Redis가 원본 데이터 저장소로 쓰일 수도 있는데, 임의로 데이터를 지워버리는 정책을 기본값으로 두면 사용자가 의도치 않게 데이터를 잃을 수 있기 때문입니다. 그래서 '어떤 데이터가 사라져도 되는지'는 사용자가 명시적으로 선택하게 만든 것이라고 이해하고 있습니다."

F2. "maxmemory-samples를 5에서 10으로 늘리면 무엇이 좋아지고 무엇이 나빠지나요?"

강한 답변: "샘플 수를 늘리면 evict 후보를 더 넓게 살펴보는 것이므로 실제 LRU/LFU 순서에 더 가까운 키를 골라내게 되어 정확도가 올라갑니다. 대신 evict가 일어날 때마다 더 많은 키를 조사해야 하므로 그 순간의 CPU 비용이 늘어납니다. eviction이 자주 일어나는, 즉 메모리가 빡빡한 인스턴스일수록 이 비용 증가가 체감됩니다. 그래서 무조건 높이기보다는, 실제로 '엉뚱한 키가 evict된다'는 문제가 관찰될 때 조정하는 편입니다."

F3. "volatile-lru로 설정했는데 evict가 필요한 시점에 TTL이 걸린 키가 하나도 없다면 어떻게 되나요?"

강한 답변: "그 경우 evict할 후보 자체가 없으므로 noeviction과 동일하게 동작합니다 — 즉 쓰기가 거부됩니다. volatile-* 정책을 쓸 때는 캐시로 쓰이는 키에 TTL이 실제로 잘 걸리고 있는지, 애플리케이션 배포 실수로 TTL 없이 SET된 키가 쌓이고 있지 않은지를 함께 모니터링해야 한다고 생각합니다."

F4. "used_memory가 실제 데이터 총량보다 훨씬 크게 나옵니다. 왜 그럴까요?"

강한 답변: "여러 원인이 있을 수 있습니다. 첫째, 키당 dictEntry·robj 헤더 같은 고정 오버헤드가 값 자체보다 클 수 있습니다. 둘째, Hash나 Set 같은 컬렉션이 hash-max-listpack-entries 같은 임계값을 넘어 압축된 listpack에서 완전한 해시테이블로 전환되면 메모리 사용량이 계단식으로 뜁니다. 셋째, jemalloc 할당/해제 패턴에 따른 파편화입니다. MEMORY USAGE로 개별 키의 실제 점유량을, INFO memory의 mem_fragmentation_ratio로 파편화 정도를 확인해서 원인을 좁혀갑니다."

F5. "eviction과 데이터 만료(expire)는 어떻게 다른가요? 같은 걸로 봐도 되나요?"

강한 답변: "다릅니다. 만료(expire)는 TTL이 다 돼서 키가 자연스럽게 사라지는 것으로, 메모리 여유와 무관하게 항상 일어납니다. Eviction은 maxmemory 한계에 도달했을 때 메모리를 확보하기 위해 아직 TTL이 남았거나 TTL이 아예 없는 키까지도 강제로 지우는 것입니다. 그래서 evicted_keys와 expired_keys 지표를 구분해서 봐야 합니다 — evicted_keys가 계속 늘어난다는 건 메모리가 부족하다는 신호이고, expired_keys는 정상적인 TTL 설계의 결과입니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 기본값이 noeviction이라는 사실과 그로 인한 구체적 장애 시나리오를 안다
  • 인스턴스에 담긴 데이터 성격(캐시 vs 원본)에 따라 allkeys/volatile을 구분해서 설계한다
  • Redis의 LRU/LFU가 샘플링 기반 근사치라는 내부 동작을 알고, maxmemory-samples의 trade-off를 설명한다
  • 메모리 사이징을 키당 오버헤드·인코딩 전환·파편화·fork COW까지 감안해 수치로 산정한다
  • evicted_keys, mem_fragmentation_ratio 같은 모니터링 지표로 설계를 검증하는 루프까지 언급한다

약한 답변의 특징

  • "LRU는 오래 안 쓴 걸 지우는 것"이라는 정의만 나열하고 끝낸다
  • Redis는 메모리가 차면 자동으로 알아서 지운다고 착각한다 (기본값을 모른다)
  • Redis의 LRU가 정확한 진짜 LRU라고 착각한다
  • 메모리 사이징을 "데이터 크기 = maxmemory"로 단순 계산한다
  • maxmemory를 물리 메모리와 동일하게 설정해도 된다고 생각한다
핵심 차이: 강한 답변은 "eviction 정책이 뭐가 있는가"로 답하지 않는다. "이 인스턴스가 가득 찼을 때 어떤 데이터는 지워져도 되고 어떤 데이터는 절대 지워지면 안 되는지 구분하고, 그 경계를 설정으로 강제할 수 있는가"라는 질문으로 바꿔서 설계한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Redis는 캐시니까 메모리가 차면 알아서 오래된 것부터 지운다"

사실: 기본 정책은 noeviction이다. 명시적으로 allkeys-lru 등으로 바꾸지 않으면, 메모리가 차는 순간 자동 삭제가 아니라 쓰기 실패가 발생한다.

오해 2: "Redis의 LRU는 정확히 가장 오래 안 쓴 키를 찾아낸다"

사실: 전체 키를 정렬해서 관리하지 않는다. maxmemory-samples개만 무작위로 뽑아 그중 가장 오래된 걸 지우는 근사치다. 샘플 수가 적을수록 실제 LRU 순서와 어긋날 가능성이 커진다.

오해 3: "volatile-lru를 쓰면 세션 데이터도 안전하다"

사실: 세션 키에 TTL이 걸려 있다면 volatile-lru도 그 키를 evict 대상으로 삼는다. 보호하려면 해당 키에 TTL을 걸지 않아야 하며, 이 경우 만료는 애플리케이션이 별도로 관리해야 한다.

오해 4: "메모리 용량 산정은 저장할 데이터 크기만 더하면 된다"

사실: 키당 오버헤드, 인코딩 전환 임계값, 파편화, replication buffer, fork COW 스파이크까지 감안해야 한다. 실무에서는 순수 데이터 크기 추정값에 1.5~2배 여유를 두고, maxmemory는 물리 메모리보다 낮게(예: 70~80%) 설정하는 것이 표준이다.

핵심 요약

오늘 반드시 기억할 것

  • maxmemory-policy의 기본값은 noeviction이다 — 캐시 용도라면 명시적으로 allkeys-lru/allkeys-lfu 등으로 바꿔야 한다.
  • volatile-* 계열은 TTL이 걸린 키만 evict 대상으로 삼는다 — 원본 데이터를 보호하려면 TTL을 걸지 않아야 한다.
  • Redis의 LRU/LFU는 전체 정렬이 아니라 무작위 샘플링 기반의 근사치이며, maxmemory-samples로 정확도와 CPU 비용을 조정한다.
  • LFU는 접근 빈도를, LRU는 최근성을 기준으로 삼는다 — hot key가 꾸준히 반복되는 워크로드에는 LFU가 더 적합할 수 있다.
  • 메모리 사이징은 데이터 크기 + 키당 오버헤드 + 인코딩 전환 + 파편화 + fork COW 여유까지 감안해야 하며, maxmemory는 물리 메모리보다 낮게(예: 70~80%) 잡는 것이 표준이다.
