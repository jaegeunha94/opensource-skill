Day 7 · Redis / 캐시 전략 Interview

Pub/Sub과 이벤트 처리

"메시지를 보낸다"는 말 안에 숨은 함정 — Redis Pub/Sub은 메시지 큐가 아니다
면접 질문
Q1 — 주력 질문 "여러 대의 WebSocket 서버에 연결된 사용자들에게 실시간 알림을 뿌려야 합니다. Redis Pub/Sub으로 구현하겠다고 하셨는데, 왜 Kafka나 RabbitMQ가 아니라 Pub/Sub인가요? 그리고 이 선택이 위험해지는 상황은 언제인가요?"
Q2 — 변형 "Pub/Sub으로 알림을 보내는 도중에 구독 중이던 서버 인스턴스가 재시작되었습니다. 그 사이에 발행된 메시지는 어떻게 되나요? 이게 문제가 되는 서비스와 안 되는 서비스를 구분해 보세요."
Q3 — 변형 "주문 생성 이벤트를 Pub/Sub으로 다른 서비스에 전파하려는 팀원이 있다면, 어떻게 조언하시겠습니까?"
전제 개념

Pub/Sub은 "큐"가 아니라 "방송"이다

Redis 문서에도 PUBLISH/SUBSCRIBE가 나오기 때문에 처음 접하면 "Redis로 메시지 큐를 만들 수 있구나"라고 오해하기 쉽다. 하지만 Pub/Sub의 동작 방식을 자세히 보면 큐(queue)가 아니라 라디오 방송(broadcast)에 가깝다는 걸 알 수 있다. 라디오는 지금 채널을 듣고 있는 사람에게만 전파가 도달하고, 방송 시점에 꺼져 있던 사람은 그 내용을 다시 들을 방법이 없다. Pub/Sub도 정확히 이렇게 동작한다.

가장 중요한 특성: PUBLISH된 메시지는 그 순간 채널을 구독(SUBSCRIBE) 중인 클라이언트에게만 즉시 전달되고, 어디에도 저장되지 않는다. 구독자가 하나도 없으면 메시지는 아무에게도 전달되지 않고 그냥 사라진다. 이후에 누군가 구독을 시작해도 그 이전 메시지를 받을 방법이 없다. 
# 터미널 A: notifications 채널 구독 시작
SUBSCRIBE notifications:user:1001

# 터미널 B: 채널에 메시지 발행
PUBLISH notifications:user:1001 "주문이 배송을 시작했습니다"

# 패턴 매칭 구독: notifications:* 형태의 모든 채널을 한 번에 구독
PSUBSCRIBE notifications:*

메시지 큐(Message Queue)와 무엇이 근본적으로 다른가

RabbitMQ나 Kafka 같은 메시지 큐는 메시지를 브로커가 보관한다. 컨슈머가 잠시 죽어 있어도 큐에 메시지가 쌓여 있다가, 컨슈머가 다시 살아나면 이어서 처리(consume)할 수 있다. 반면 Pub/Sub은 브로커(Redis)가 메시지를 전혀 보관하지 않는다 — 구독자가 연결되어 있는 그 순간에만 의미가 있는 "지금 이 순간의 신호 전달" 도구다. 이 차이 때문에 Pub/Sub은 at-most-once(최대 한 번, 유실 가능) 전달만 보장하고, 메시지 큐는 설계에 따라 at-least-once(최소 한 번, 재처리로 보완)를 보장할 수 있다.

왜 Redis에 이런 "불완전해 보이는" 기능이 있는가

Pub/Sub은 "신뢰성 있는 메시지 전달"을 목표로 만들어진 기능이 아니라, 지금 연결되어 있는 대상에게 최대한 빠르게, 최소한의 오버헤드로 신호를 알리는 것이 목표다. 여러 애플리케이션 서버 인스턴스가 "같은 정보를 동시에 알아야 하는" 상황 — 예를 들어 "이 사용자에게 새 메시지가 왔다" 같은 신호 — 에서 메시지 자체의 영속성보다 낮은 지연시간과 단순함이 더 중요할 때 쓰라고 만든 도구다.

Consumer Group이 없다는 것의 의미

Kafka나 Redis Streams에는 Consumer Group이 있어서, 여러 컨슈머가 같은 메시지를 중복 없이 나눠 처리(load balancing)하거나, 각자 독립적으로 전체 메시지를 다 받을 수도 있다. Pub/Sub에는 이 개념이 없다 — 구독 중인 모든 클라이언트가 같은 메시지를 전부 받는다. "여러 워커 중 하나만 처리해야 하는 작업(job)"을 Pub/Sub으로 구현하면, 구독자가 3대면 같은 작업이 3번 처리되는 사고로 이어진다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

Pub/Sub 질문은 명령어 사용법이 아니라, "메시지 전달 보장 수준(delivery guarantee)"을 이해하고 요구사항에 맞는 도구를 고르는 판단력을 평가하는 질문이다.

  • 유실 허용 여부를 구분하는 능력: "이 이벤트가 유실되면 실제로 문제가 되는가?"를 먼저 따지는 습관이 있는가
  • 도구 선택의 근거: Pub/Sub, Streams, Kafka, RabbitMQ 각각이 어떤 보장 수준과 비용 트레이드오프를 가지는지 구체적으로 비교할 수 있는가
  • 실무 사고 경험: "Pub/Sub으로 중요한 이벤트를 보냈다가 유실돼서 장애가 났다" 류의 실패 패턴을 미리 인지하고 있는가
  • 과대 설계 방지: 반대로, 유실돼도 무방한 신호성 이벤트에 굳이 Kafka 같은 무거운 인프라를 끌어오지 않는 실용적 판단을 하는가
실무에서 신입~주니어가 가장 흔히 저지르는 실수는 "Redis에 PUBLISH/SUBSCRIBE가 있으니 메시지 큐 대신 쓰면 되겠다"는 판단이다. 면접관은 지원자가 이 함정을 아는지, 그리고 어떤 이벤트는 유실돼도 되고 어떤 이벤트는 절대 안 되는지 경계선을 그을 수 있는지를 본다.
실무 연결

실제 상황과 연결하기

상황 1: 여러 WebSocket 서버 간 실시간 알림 fan-out

사용자가 여러 대의 WebSocket 서버 중 아무 곳에나 연결될 수 있는 구조에서, "이 사용자에게 새 메시지가 왔다"는 이벤트를 어느 서버가 처리할지 알 수 없다. 이때 모든 서버 인스턴스가 공통 채널을 구독해 두고, 이벤트가 발생하면 Pub/Sub으로 전체 서버에 뿌린 뒤, 실제로 그 사용자의 연결을 들고 있는 서버만 최종 전달을 처리하는 패턴이 흔히 쓰인다. 이 경우 연결이 없는 서버는 메시지를 그냥 버리면 되고, 유실돼도 재시도할 필요가 없는 신호성 데이터라 Pub/Sub의 특성과 잘 맞는다.

상황 2: 분산 캐시(로컬 캐시)의 무효화 브로드캐스트

각 애플리케이션 서버가 자체 in-process 캐시(예: Caffeine, 로컬 Map)를 갖고 있을 때, 특정 데이터가 갱신되면 "모든 서버의 로컬 캐시를 지워야 한다"는 신호를 보내야 한다. 이때 Pub/Sub으로 무효화 이벤트를 브로드캐스트하고, 각 서버가 이를 받아 자신의 로컬 캐시 항목을 지우는 방식을 쓴다. 이 신호를 놓친 서버가 있어도 다음 TTL 만료나 다음 갱신 이벤트에서 다시 맞춰지므로, 완벽한 전달 보장이 없어도 실무적으로 감당 가능하다.

상황 3: 실시간 랭킹판 갱신 알림 (Day 6과 연결)

Day 6에서 다룬 실시간 랭킹은 Sorted Set에 점수를 저장하지만, "점수가 바뀌었으니 화면을 갱신하라"는 신호 자체는 Pub/Sub으로 전파하는 경우가 많다. 클라이언트가 계속 폴링하는 대신, 점수 변화가 있을 때만 서버가 먼저 알려주는 방식이다. 여기서도 핵심은 "화면에 최신 순위가 보이면 되고, 중간에 알림 하나를 놓쳐도 다음 알림이 오면 자동으로 맞춰진다"는 점 — 유실 허용 가능한 이벤트라는 것이다.

상황 4: 절대 Pub/Sub으로 하면 안 되는 것 — 주문/결제 이벤트

"주문이 생성되었다"는 이벤트를 Pub/Sub으로 다른 서비스(재고 차감, 배송 시작, 정산)에 전파한다고 하면 위험하다. 이벤트를 받는 서비스가 그 순간 재시작 중이거나 네트워크가 잠깐 끊겨 있으면, 그 주문은 영원히 재고 차감이나 배송 처리가 되지 않는다. 이런 이벤트는 반드시 영속성과 재처리(replay)가 보장되는 Redis Streams, Kafka, RabbitMQ 같은 메시지 큐로 처리해야 한다.

상황 5: Redis Cluster 환경에서 Pub/Sub을 쓸 때

Redis Cluster는 데이터를 키 슬롯 기준으로 여러 노드에 분산하지만, 전통적인 Pub/Sub 메시지는 이 샤딩과 무관하게 클러스터 전체 노드에 브로드캐스트된다. 즉 어느 노드에 PUBLISH하든 모든 노드가 그 메시지를 전파해야 하므로, 메시지량이 많으면 클러스터 전체에 부하가 걸릴 수 있다. Redis 7부터 제공되는 Sharded Pub/Sub(SPUBLISH/SSUBSCRIBE)은 채널을 슬롯에 매핑해 특정 노드에만 부하를 집중시켜 이 문제를 완화한다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"Redis Pub/Sub은 메시지 큐가 아니라 방송에 가깝습니다. 발행된 메시지는 그 순간 구독 중인 클라이언트에게만 전달되고 어디에도 저장되지 않기 때문에, 구독자가 없으면 메시지는 그냥 사라집니다. 그래서 저는 이 이벤트가 '유실돼도 되는가'를 먼저 판단 기준으로 삼습니다.

여러 WebSocket 서버에 흩어져 있는 사용자에게 실시간 알림을 뿌리거나, 분산된 로컬 캐시를 무효화하는 신호처럼 '지금 연결된 대상에게만 알리면 되고, 놓쳐도 다음 이벤트나 TTL로 자연스럽게 맞춰지는' 경우에는 Pub/Sub이 구현이 단순하고 지연시간도 낮아 적합합니다. 반면 주문 생성이나 결제 완료처럼 놓치면 실제 비즈니스 손실로 이어지는 이벤트는 Pub/Sub으로 처리하면 안 되고, Redis Streams나 Kafka처럼 메시지가 영속적으로 저장되고 재처리될 수 있는 도구를 씁니다.

그리고 Redis Cluster 환경이라면, 일반 Pub/Sub 메시지는 샤딩과 무관하게 클러스터 전체 노드에 브로드캐스트된다는 점도 고려합니다. 메시지량이 많아지면 전체 노드에 부하가 걸릴 수 있어서, Redis 7의 Sharded Pub/Sub이나 별도 메시징 인프라 분리를 검토합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Pub/Sub과 Redis Streams의 근본적인 차이는 무엇인가요?"

강한 답변: "Pub/Sub은 메시지를 저장하지 않는 fire-and-forget 방식이고, 구독자가 그 순간 없으면 메시지가 사라집니다. 반면 Streams는 메시지를 로그처럼 순서대로 저장하고, Consumer Group으로 여러 워커가 나눠 처리하거나 각자 독립적으로 다시 읽을 수 있습니다. 즉 Pub/Sub은 '지금 이 순간의 신호'이고, Streams는 '재생 가능한 이벤트 로그'라는 점이 근본적으로 다릅니다."

F2. "구독자(subscriber)가 메시지를 처리하는 속도가 느려지면 어떻게 되나요?"

강한 답변: "Redis는 구독자에게 보낼 메시지를 client output buffer에 쌓아두는데, 구독자가 느리게 소비하면 이 버퍼가 계속 커집니다. 설정된 한계(client-output-buffer-limit pubsub)를 넘으면 Redis는 해당 클라이언트 연결을 강제로 끊어버립니다. 그래서 구독자 쪽 처리 로직이 무겁다면 Pub/Sub 콜백 안에서 직접 무거운 작업을 하지 않고, 큐에 넘겨 비동기로 처리하도록 설계합니다."

F3. "Redis Cluster에서 Pub/Sub은 어떻게 동작하나요?"

강한 답변: "일반 PUBLISH/SUBSCRIBE는 키 슬롯 기반 샤딩과 무관하게, 클러스터의 모든 노드가 메시지를 전파합니다. 즉 특정 노드 하나에만 부하가 걸리는 게 아니라 전체 노드가 함께 브로드캐스트 비용을 부담합니다. Redis 7부터는 SPUBLISH/SSUBSCRIBE로 채널을 슬롯에 매핑하는 Sharded Pub/Sub을 제공해서, 특정 채널의 트래픽을 담당 노드로 한정시킬 수 있습니다."

F4. "Pub/Sub으로 분산 캐시 무효화를 구현할 때 주의할 점은?"

강한 답변: "무효화 신호를 놓친 서버가 있으면 그 서버만 오래된 로컬 캐시 데이터를 계속 들고 있을 수 있습니다. 그래서 로컬 캐시에도 짧은 TTL을 함께 걸어서, 무효화 신호를 놓치더라도 일정 시간 후엔 자연스럽게 최신 데이터로 맞춰지도록 이중 안전장치를 둡니다. Pub/Sub 하나만 신뢰하는 설계는 하지 않습니다."

F5. "굳이 Redis Pub/Sub을 쓸 이유가 있나요? 이미 Kafka를 쓰고 있다면요?"

강한 답변: "Kafka는 안정적이지만 토픽 생성, 파티션 설계, 컨슈머 그룹 관리 등 운영 비용이 있고, 지연시간도 Pub/Sub보다 상대적으로 큽니다. 이미 캐시나 세션 용도로 Redis를 쓰고 있는 서비스에서, 유실돼도 무방한 가벼운 신호성 이벤트까지 Kafka로 보내는 것은 과대 설계일 수 있습니다. 다만 팀에 이미 Kafka 인프라와 운영 노하우가 갖춰져 있고 이벤트 종류를 통일하고 싶다면, 굳이 두 가지 메시징 도구를 병행하지 않고 Kafka로 일원화하는 것도 합리적인 선택입니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Pub/Sub의 전달 보장 수준(at-most-once)을 정확히 설명한다
  • "유실 허용 여부"를 도구 선택의 판단 기준으로 명시적으로 제시한다
  • 적합한 사례와 부적합한 사례를 구체적인 실무 예시로 대비한다
  • Cluster 환경, slow subscriber 같은 운영 관점의 함정까지 인지한다
  • "Redis만 있으면 다 된다"가 아니라 다른 메시징 도구와의 경계를 분명히 한다

약한 답변의 특징

  • "PUBLISH/SUBSCRIBE 쓰면 됩니다"라고만 하고 메시지가 저장되지 않는다는 점을 언급하지 않는다
  • Pub/Sub을 메시지 큐와 동일한 것으로 착각하고 아무 이벤트에나 적용한다
  • 주문/결제 같은 신뢰성이 필요한 이벤트에도 Pub/Sub을 그대로 제안한다
  • 구독자가 없을 때 메시지가 어떻게 되는지 설명하지 못한다
  • Cluster 환경에서의 브로드캐스트 부하 같은 운영 이슈를 전혀 고려하지 않는다
핵심 차이: 강한 답변은 Pub/Sub을 "쓸 수 있는 기능"이 아니라 "특정 보장 수준을 가진 도구"로 이해하고, 그 보장 수준이 요구사항과 맞는지를 먼저 검증한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Redis Pub/Sub도 메시지 큐의 일종이라 메시지가 쌓여 있다가 처리된다"

사실: Pub/Sub은 메시지를 전혀 저장하지 않는다. 발행 시점에 구독자가 없으면 메시지는 그대로 사라지고, 이후 구독을 시작해도 과거 메시지를 받을 방법이 없다. "쌓인다"는 개념 자체가 없다.

오해 2: "Pub/Sub은 신뢰할 수 없으니 아예 쓰면 안 된다"

사실: Pub/Sub은 신뢰성이 필요 없는 상황을 위해 설계된 도구이지, 결함이 있는 도구가 아니다. "지금 연결된 대상에게만 알리면 충분한" 신호성 이벤트에는 오히려 무거운 메시지 큐보다 더 적합하다. 도구 자체의 문제가 아니라 맞지 않는 곳에 쓰는 것이 문제다.

오해 3: "구독자가 여러 개면 메시지를 나눠서 처리한다"

사실: Pub/Sub에는 Consumer Group 개념이 없다. 채널을 구독 중인 모든 클라이언트가 같은 메시지를 각각 전부 받는다. "여러 워커 중 하나만 처리"해야 하는 작업 분배 용도로 쓰면 중복 처리 사고가 난다.

오해 4: "Redis Cluster에서는 Pub/Sub도 키처럼 샤딩되어 특정 노드에만 부하가 간다"

사실: 일반 Pub/Sub 메시지는 클러스터의 모든 노드에 브로드캐스트된다. 키 슬롯 기반 샤딩과는 무관하게 동작하며, 이 부하를 줄이려면 Redis 7 이상의 Sharded Pub/Sub (SPUBLISH/SSUBSCRIBE)을 별도로 도입해야 한다.

핵심 요약

오늘 반드시 기억할 것

  • Redis Pub/Sub은 메시지를 저장하지 않는 fire-and-forget 방송이지, 메시지 큐가 아니다.
  • 도구 선택의 기준은 "이 이벤트가 유실돼도 되는가"다 — 신호성 이벤트는 Pub/Sub, 신뢰성이 필요한 이벤트는 Streams/Kafka로 분리한다.
  • Pub/Sub에는 Consumer Group이 없어 구독자 전원이 같은 메시지를 중복으로 받는다.
  • 구독자가 느리면 client output buffer 한계로 강제 연결 종료될 수 있다.
  • Redis Cluster에서 일반 Pub/Sub은 전체 노드로 브로드캐스트되며, 부하 완화를 위해 Redis 7+의 Sharded Pub/Sub을 검토할 수 있다.
