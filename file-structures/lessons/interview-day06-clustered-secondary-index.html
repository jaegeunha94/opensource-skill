클러스터링과 보조 인덱스 — Data Locality, Write Amplification, Secondary Lookup

인덱스를 하나 더 만들면 조회는 빨라지는데, 왜 삽입은 느려지고 조회는 "두 번" 일어날 수 있는가

"클러스터드 인덱스는 읽기 성능을 어떻게 바꾸고, 그 대가로 어떤 비용을 만듭니까?"

전제 개념 — 쉬운 한국어로 먼저 이해하기

6-1. 클러스터드 인덱스란 — "인덱스 순서 = 실제 데이터가 저장된 순서"

Day 3~4에서 다룬 B+Tree는 "정렬된 키를 따라가면 leaf에서 원하는 레코드를 찾는다"는 구조였습니다. 그런데 leaf에 무엇이 저장되는지는 두 가지 방식이 있습니다.

  • Clustered Index (클러스터드 인덱스): leaf 노드 자체가 실제 데이터 행(row)을 담습니다. 즉 테이블의 물리적 저장 순서 자체가 인덱스 키 순서와 같습니다.
  • Non-clustered / Secondary Index (보조 인덱스): leaf 노드는 실제 데이터가 아니라 "진짜 데이터가 어디 있는지 가리키는 포인터"만 담습니다.

비유하자면, 클러스터드 인덱스는 사전 자체가 알파벳 순으로 인쇄된 책입니다. 반면 보조 인덱스는 책 뒤의 찾아보기(색인)입니다 — 색인은 정렬돼 있지만, 실제 내용을 보려면 색인이 가리키는 페이지 번호로 다시 넘어가야 합니다.

[ Clustered Index — leaf = 실제 데이터 ] B+Tree leaf: [id=10, name="Kim", ...전체 row 데이터...] → [id=11, ...] → [id=12, ...] (정렬된 키 순서 == 디스크에 물리적으로 저장된 순서) id=11 조회: root → internal → leaf 도달 = 바로 전체 row 확보 (추가 I/O 없음) [ Secondary Index — leaf = 포인터만 ] B+Tree leaf: [email="a@x.com" → id=11] → [email="b@x.com" → id=27] → ... (leaf에는 실제 row가 아니라 "클러스터드 키(id)" 또는 "row 위치"만 있음) email="a@x.com" 조회: 1) secondary index B+Tree 탐색 → id=11 확보 (I/O: 트리 높이만큼) 2) id=11로 clustered index를 "다시" 탐색 → 실제 row 확보 (I/O: 트리 높이만큼, 추가) => 총 I/O = secondary 탐색 + clustered 재탐색 (이중 조회, "bookmark lookup")

6-2. 왜 클러스터드 인덱스는 테이블당 하나뿐인가

데이터는 디스크의 물리적 페이지에 한 가지 순서로만 저장될 수 있습니다. "id 순으로 정렬해서 저장"하면서 동시에 "email 순으로 정렬해서 저장"할 수는 없습니다 — 물리적 저장 위치는 하나뿐이기 때문입니다. 그래서 클러스터드 인덱스는 테이블당 정확히 하나만 존재하고, 나머지 조회 조건(email, 이름 등)은 모두 보조 인덱스로 만들어 "정렬된 포인터 목록"만 유지합니다.

💡 DBMS별 차이: MySQL InnoDB는 기본키(PK)를 자동으로 클러스터드 인덱스로 사용합니다 (PK가 없으면 내부적으로 숨겨진 행 ID를 만들어 클러스터링합니다). 반면 PostgreSQL은 테이블을 기본적으로 heap 구조(정렬 없는 삽입 순서 저장)로 두고, 모든 인덱스(PK 포함)를 "포인터만 가진" 구조로 취급합니다. CLUSTER 명령으로 한 번 물리적으로 재정렬할 수는 있지만, 이후 삽입에서 그 순서를 자동으로 유지해주지는 않습니다. 이 차이를 아는 것 자체가 "DBMS 내부 구조를 이해하고 있다"는 신호입니다.

6-3. Secondary Index Lookup이 "두 번" 일어나는 이유

클러스터드 인덱스가 있는 테이블에서 보조 인덱스로 조회하면, 보조 인덱스의 leaf에는 실제 row가 아니라 클러스터드 키 값(예: id)이 들어 있습니다(MySQL InnoDB 방식). 그래서 조회는 "보조 인덱스에서 id 찾기" → "그 id로 클러스터드 인덱스를 다시 타고 내려가 실제 row 찾기"라는 두 단계를 거칩니다. 이를 bookmark lookup 또는 random I/O 재조회라 부릅니다. 문제는 두 번째 단계가 정렬 순서가 다른 곳으로 점프하는 것이라, 여러 건을 조회할 때 순차 I/O가 아니라 건마다 랜덤 I/O가 발생한다는 점입니다.

[ Secondary index로 100건 조회 시 I/O 패턴 ] secondary index leaf 스캔: email 순서대로 정렬 → 순차 I/O로 100개의 id를 빠르게 수집 각 id로 clustered index 재조회: id 값들은 secondary index 순서와 무관하게 흩어져 있음 → 100번의 "서로 다른 페이지로의" 랜덤 I/O 발생 => "인덱스를 탔다"고 해서 항상 빠른 것이 아니라, 보조 인덱스를 통한 대량 조회는 랜덤 I/O 폭증으로 이어질 수 있음 (이것이 옵티마이저가 "인덱스 대신 풀스캔"을 선택하는 이유 중 하나)

6-4. Write Amplification — 클러스터드 키 선택이 삽입 비용을 좌우한다

클러스터드 인덱스는 "물리적 저장 순서 = 키 순서"를 유지해야 하므로, 새 row를 삽입할 때 정렬 순서상 맞는 위치에 끼워 넣어야 합니다. 이때 키 값의 패턴이 성능을 크게 좌우합니다.

  • Auto-increment(단조 증가) 키: 새 값은 항상 마지막 페이지 뒤에 추가되므로, 대부분 순차 append입니다. Page split이 거의 발생하지 않습니다.
  • UUID v4(무작위) 키: 새 값이 트리 전체에 무작위로 흩어져 삽입되므로, 거의 매번 이미 꽉 찬 페이지 중간에 끼워 넣어야 합니다. 이는 page split(페이지를 둘로 나누는 작업)을 유발하고, split은 추가 쓰기 I/O와 디스크 단편화(fragmentation)를 만듭니다.
[ Auto-increment vs UUID v4 삽입 패턴 ] Auto-increment (1, 2, 3, 4, ...): 새 row는 항상 맨 끝 leaf 페이지 뒤에 추가 → 순차 append, page split 드묾, 디스크 단편화 최소 UUID v4 (무작위, 예: 7f3a..., 2b91..., e04c...): 새 row가 트리 전역에 무작위 위치로 삽입됨 → 이미 가득 찬 중간 페이지에 끼어들어야 함 → page split 빈번 → split마다 추가 쓰기 I/O + 페이지 내 공간 낭비(fragmentation) → 캐시 지역성도 나빠짐 (최근 삽입 데이터가 여기저기 흩어짐)
⚠️ 면접관이 보는 것: "UUID는 충돌 위험이 없어서 좋다"는 답만으로는 부족합니다. "클러스터드 키의 삽입 패턴이 물리적 페이지 재배치 비용(write amplification)에 어떤 영향을 주는지"까지 연결해야 시니어 답변입니다. 이 때문에 실무에서는 UUID v7이나 ULID처럼 "무작위성은 유지하되 시간 순서로는 단조 증가하는" 키 설계가 대안으로 쓰입니다.

왜 면접에서 이 질문을 묻는가

"클러스터드 인덱스가 뭔지 정의를 아는가"가 아니라, "인덱스 설계 결정이 읽기와 쓰기 성능에 어떤 구조적 trade-off를 만드는지 인과관계로 설명할 수 있는가"를 봅니다.

  1. 왜 클러스터드 인덱스가 하나뿐인지 물리적 근거로 설명할 수 있는가
    "그렇게 정해져 있다"가 아니라, 물리적 저장 순서는 하나뿐이라는 제약에서 도출된다는 것을 말해야 합니다.
  2. 보조 인덱스 조회의 실제 비용을 구조적으로 설명할 수 있는가
    "인덱스를 타면 빠르다"가 아니라, 클러스터드 키를 통한 재조회(bookmark lookup)와 그로 인한 랜덤 I/O를 설명해야 합니다.
  3. 기본키 설계가 실무 성능 이슈로 이어진다는 것을 아는가
    UUID 기본키가 실제로 삽입 성능 저하와 인덱스 비대화(secondary index가 더 긴 클러스터드 키를 저장해야 함)를 만든다는 것을 사례로 연결할 수 있어야 합니다.
⚠️ 주니어 수준 답변: "클러스터드 인덱스는 데이터가 정렬돼 저장되는 인덱스이고, 보조 인덱스는 별도의 인덱스입니다."
10년차 기대 답변: "클러스터드 인덱스는 leaf가 곧 실제 row라서 물리적 저장 순서를 결정하고, 그래서 테이블당 하나만 가능합니다. 보조 인덱스는 leaf에 클러스터드 키만 저장하므로 조회 시 secondary 탐색 후 클러스터드 인덱스를 다시 타는 이중 조회(bookmark lookup)가 필요하고, 이는 랜덤 I/O로 이어집니다. 또한 클러스터드 키가 UUID처럼 무작위면 삽입마다 page split이 발생해 write amplification이 커지므로, 저는 auto-increment나 시간 순서를 보존하는 UUID v7 같은 키를 우선 고려합니다."
⚙

실무 연결

실무 상황 클러스터링/보조 인덱스 판단
MySQL InnoDB 기본키 설계 PK가 클러스터드 인덱스이므로, UUID v4를 PK로 쓰면 삽입마다 page split과 fragmentation이 누적됩니다. 실무에서는 auto-increment 서로게이트 키를 PK로 쓰고, UUID는 별도의 unique secondary index로 두는 패턴이 흔합니다.
Covering Index (커버링 인덱스) 자주 조회하는 컬럼을 보조 인덱스에 함께 저장(INCLUDE 또는 복합 인덱스)하면, 클러스터드 인덱스로 재조회할 필요 없이 secondary index만으로 결과를 반환할 수 있습니다. bookmark lookup 비용을 없애는 대표적인 실무 최적화입니다.
PostgreSQL vs MySQL 마이그레이션 PostgreSQL은 heap 구조라 PK도 사실상 보조 인덱스처럼 동작해, PK 조회도 heap을 한 번 더 찾아가는 구조입니다. MySQL 기준으로 설계된 스키마를 PostgreSQL로 옮길 때 이 구조적 차이가 성능 특성을 바꿀 수 있습니다.
대량 삽입(bulk insert) 배치 작업 로그, 이벤트 데이터를 대량 삽입할 때 클러스터드 키를 시간 기반 단조 증가 값으로 설계하면 page split 없이 append 위주로 처리되어 처리량이 크게 개선됩니다.
인덱스 비대화(bloat)와 재구성 무작위 키로 인한 page split이 누적되면 인덱스 크기가 필요 이상으로 커집니다. 정기적인 인덱스 재구성(rebuild)이나 fill factor 조정으로 완화하지만, 근본 해법은 키 설계 자체를 바꾸는 것입니다.

답변 사고 순서 — 면접에서 이렇게 생각하면서 말한다

  1. 클러스터드 인덱스의 정의부터 물리적 근거로 설명
    leaf가 곧 실제 데이터이며, 그래서 물리적 저장 순서를 결정한다는 점에서 시작합니다.
  2. 테이블당 하나만 가능한 이유를 도출
    물리적 저장 위치는 하나뿐이라는 제약에서 "왜 하나뿐인가"의 답이 자연스럽게 나온다는 것을 보여줍니다.
  3. 보조 인덱스 조회의 이중 탐색 구조를 설명
    secondary index 탐색 후 클러스터드 키로 재조회하는 bookmark lookup과, 이로 인한 랜덤 I/O를 설명합니다.
  4. 삽입 시 write amplification을 키 선택과 연결
    단조 증가 키와 무작위 키의 page split 빈도 차이를 원인-결과로 설명합니다.
  5. 실무 대응 방안으로 마무리
    covering index, 서로게이트 auto-increment 키, 시간 순서 보존 UUID(v7/ULID) 같은 완화 전략을 제시합니다.

핵심 답변 예시

짧은 버전 (1~2분 답변)

강한 답변 (Short)

클러스터드 인덱스는 leaf 노드 자체가 실제 데이터 row라서, 인덱스 키 순서가 곧 디스크에 저장된 물리적 순서가 됩니다. 그래서 순차 스캔과 범위 조회가 빠르지만, 물리적 순서는 하나뿐이라 테이블당 하나만 만들 수 있습니다.

보조 인덱스는 leaf에 실제 row 대신 클러스터드 키만 저장하므로, 조회 시 secondary index를 먼저 탐색하고 그 키로 클러스터드 인덱스를 다시 타는 이중 조회(bookmark lookup)가 필요합니다. 또한 클러스터드 키가 UUID처럼 무작위면 삽입마다 페이지 중간에 끼어들며 page split이 자주 발생해 쓰기 비용이 커집니다.

깊이 있는 버전 (3~5분 답변)

강한 답변 (Deep)

클러스터드 인덱스와 보조 인덱스의 차이는 "B+Tree leaf에 무엇을 저장하는가"에서 시작합니다. 클러스터드 인덱스는 leaf가 곧 전체 row 데이터이므로, 인덱스를 통해 도달한 순간 바로 원하는 데이터를 얻습니다. 반면 보조 인덱스는 leaf에 클러스터드 키(또는 row 위치)만 저장하기 때문에, secondary index 탐색으로 얻은 값을 가지고 클러스터드 인덱스를 다시 한번 타야 실제 row를 얻을 수 있습니다. 이 두 번째 탐색을 bookmark lookup이라 부르고, 여러 건을 조회할 때는 이 재조회가 정렬 순서와 무관한 위치로 점프하는 랜덤 I/O가 되어, "인덱스를 탔다"고 해서 항상 빠르지는 않은 이유가 됩니다.

데이터의 물리적 저장 위치는 하나뿐이므로, 클러스터드 인덱스는 테이블당 정확히 하나만 존재할 수 있습니다. MySQL InnoDB는 기본키를 자동으로 클러스터드 인덱스로 쓰고, PostgreSQL은 테이블을 기본적으로 heap 구조로 두고 PK를 포함한 모든 인덱스를 포인터 기반으로 취급한다는 점에서 DBMS별 구조 차이가 있습니다.

삽입 관점에서는 클러스터드 키의 패턴이 성능을 좌우합니다. auto-increment처럼 단조 증가하는 키는 새 row가 항상 마지막 페이지 뒤에 추가되어 순차 append에 가깝지만, UUID v4처럼 무작위 키는 매번 이미 가득 찬 중간 페이지에 끼어들어야 해서 page split이 빈번하게 발생하고, 이는 추가 쓰기 I/O와 디스크 단편화로 이어지는 write amplification을 만듭니다. 그래서 저는 실무에서 서로게이트 auto-increment 키를 클러스터드 키로 쓰고, 전역 고유성이 필요하면 UUID v7이나 ULID처럼 "무작위성은 유지하되 생성 시간 순으로는 단조 증가하는" 값을 사용해 write amplification과 고유성 요구를 동시에 만족시키는 방향으로 판단합니다. 그리고 자주 조회하는 컬럼 조합은 covering index로 묶어 bookmark lookup 자체를 없애는 것도 함께 고려합니다.

약한 답변

"클러스터드 인덱스는 데이터가 정렬돼서 저장되는 인덱스이고, 보조 인덱스는 그냥 추가로 만드는 인덱스입니다. PK를 클러스터드 인덱스로 씁니다."

왜 약한가: 왜 테이블당 하나만 가능한지 물리적 근거가 없고, 보조 인덱스 조회에서 이중 탐색(bookmark lookup)과 랜덤 I/O가 발생한다는 인과관계도 없습니다. 삽입 성능과 키 선택(auto-increment vs UUID)의 연결도 빠져 있어, 정의만 나열한 수준에 그칩니다.

⇄

Trade-off 정리

기준 Clustered Index Secondary Index
단건 조회(해당 키) leaf 도달 = row 확보, 추가 I/O 없음 secondary 탐색 + 클러스터드 재탐색 (2단계)
범위 스캔 물리적으로 인접, 순차 I/O로 빠름 포인터는 정렬돼도 실제 row는 흩어져 랜덤 I/O
테이블당 개수 정확히 1개 (물리 저장 순서는 하나뿐) 여러 개 생성 가능
삽입 비용(키가 무작위일 때) page split 빈번, write amplification 큼 leaf가 작아 상대적으로 split 부담 적음
저장 공간 row 전체 보유 키 + 클러스터드 키(또는 포인터)만 보유, 상대적으로 작음
완화 전략 단조 증가 키, fill factor 여유 확보 covering index로 bookmark lookup 제거
💡 클러스터드 인덱스는 "자주 범위로 조회하거나 정렬 순회가 필요한 키"에 배정하고, 나머지 조회 조건은 보조 인덱스로 보완하되 bookmark lookup 비용이 문제라면 covering index로 회피하는 것이 실무에서 반복되는 패턴입니다.
✗

흔한 오해와 바로잡기

오해 1: "기본키(Primary Key)는 항상 클러스터드 인덱스다"

MySQL InnoDB에서는 사실이지만, PostgreSQL은 기본적으로 heap 구조를 쓰고 PK도 포인터 기반 인덱스로 취급합니다. "PK = 클러스터드 인덱스"는 특정 DBMS의 구현 선택이지, 일반 원리가 아닙니다. 미신

오해 2: "인덱스를 타면 항상 풀스캔보다 빠르다"

보조 인덱스로 많은 건수를 조회하면 bookmark lookup으로 인한 랜덤 I/O가 누적되어, 오히려 순차 I/O인 풀스캔보다 느릴 수 있습니다. 옵티마이저가 조회 비율(selectivity)에 따라 인덱스 대신 풀스캔을 선택하는 이유입니다. 미신

오해 3: "UUID는 충돌 위험이 없으니 PK로 쓰기에 항상 좋다"

전역 고유성은 장점이지만, UUID v4의 무작위성은 클러스터드 인덱스의 page split을 유발해 삽입 성능과 디스크 단편화에 실질적인 비용을 만듭니다. "고유성"과 "삽입 지역성"은 별개의 요구이며, UUID v7/ULID처럼 두 요구를 함께 만족하는 대안을 우선 검토해야 합니다. 미신

오해 4: "테이블에 인덱스를 여러 개 만들수록 무조건 좋다"

각 보조 인덱스는 삽입/수정/삭제 시마다 함께 갱신되어야 하므로, 인덱스가 많을수록 쓰기 비용이 누적됩니다. 조회 이득과 쓰기 비용 사이의 trade-off이지, "많을수록 좋다"가 아닙니다. 미신

Senior-Level Follow-up 질문과 답변 방향

"Covering Index는 정확히 어떻게 bookmark lookup을 없앱니까?"

답변 방향: 쿼리가 필요로 하는 모든 컬럼(SELECT, WHERE, ORDER BY 등)이 보조 인덱스 자체에 포함돼 있으면, 클러스터드 인덱스로 재조회할 필요 없이 secondary index의 leaf만으로 결과를 완성할 수 있습니다. 이를 "index-only scan"이라 부르며, bookmark lookup 단계 자체를 제거해 랜덤 I/O를 없앱니다.

"UUID v7과 UUID v4는 클러스터드 키 관점에서 무엇이 다릅니까?"

답변 방향: UUID v4는 전체 비트가 무작위라 삽입 위치가 트리 전역에 흩어집니다. UUID v7은 앞부분에 타임스탬프를 포함해 생성 시간 순으로는 단조 증가하므로, 삽입이 대체로 마지막 페이지 근처에서 일어나 auto-increment와 비슷한 삽입 지역성을 얻으면서도 전역 고유성을 유지합니다.

"PostgreSQL처럼 heap 구조인 DBMS에서는 클러스터드 인덱스의 이점을 아예 못 얻습니까?"

답변 방향: CLUSTER 명령으로 특정 인덱스 순서에 맞춰 테이블을 한 번 물리적으로 재정렬할 수 있지만, 이후 삽입/갱신에서 그 순서를 자동으로 유지하지는 않습니다. 즉 "일회성 재정렬"은 가능해도 MySQL InnoDB처럼 "항상 유지되는 클러스터링"은 구조적으로 없다는 점을 짚어야 합니다.

"보조 인덱스가 여러 개 있는 테이블에서 UPDATE가 느려지는 이유는 무엇입니까?"

답변 방향: 인덱싱된 컬럼 값이 바뀌면 클러스터드 인덱스뿐 아니라 관련된 모든 보조 인덱스도 함께 갱신해야 합니다. 보조 인덱스 수만큼 추가 쓰기 I/O가 발생하므로, 인덱스가 많을수록 쓰기 경로의 비용이 선형적으로 늘어난다는 것이 핵심입니다.
