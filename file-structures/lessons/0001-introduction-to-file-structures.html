1. 왜 파일구조를 배우는가?

프로그램이 처리하는 데이터는 대부분 메인 메모리(RAM)에 올라와 있다. 그러나 메인 메모리는 전원을 끄면 사라지고, 용량에 한계가 있다. 수십억 건의 고객 정보나 수년치 거래 기록은 메모리에 모두 올릴 수 없다.

이 때 등장하는 것이 보조기억장치(Secondary Storage) — 하드디스크(HDD), SSD —다. 보조기억장치는 전원이 꺼져도 데이터가 남고, 용량이 훨씬 크지만, 메모리에 비해 접근 속도가 수천~수만 배 느리다.

구분주기억장치 (RAM)보조기억장치 (HDD/SSD)
속도 나노초(ns) 단위 밀리초(ms) 단위 (HDD 기준)
영속성 휘발성 — 전원 OFF 시 소멸 비휘발성 — 전원 OFF 후 보존
용량 (일반 PC) 8 GB ~ 64 GB 수 TB 이상
비용 (GB당) 높음 낮음
접근 방식 임의 접근 (byte 단위) 블록 단위 I/O
핵심 통찰 보조기억장치는 블록(Block) 단위로 읽고 쓴다. 필요한 데이터 1 바이트를 얻으려 해도 4 KB ~ 512 KB 크기의 블록 전체를 읽어야 한다. 따라서 얼마나 적은 블록 I/O로 원하는 데이터를 찾느냐가 파일구조 설계의 핵심 목표다.

파일구조(File Structures)는 보조기억장치에 데이터를 어떻게 배치하고, 어떻게 효율적으로 검색·삽입·삭제하는지를 다루는 학문이다. 잘 설계된 파일구조는 디스크 접근 횟수를 최소화하여 성능을 수백 배 높일 수 있다.

2. 데이터의 계층 구조

파일구조에서 데이터는 다음과 같은 4단계 계층으로 표현된다.

┌─────────────────────────────────────────────────────┐ │ 데이터베이스 (Database) │ │ ┌─────────────────────────────────────────────┐ │ │ │ 파일 (File) — 관련 레코드의 집합 │ │ │ │ ┌───────────────────────────────────────┐ │ │ │ │ │ 레코드 (Record) — 관련 필드의 집합 │ │ │ │ │ │ ┌──────┐ ┌──────┐ ┌──────────┐ │ │ │ │ │ │ │ 필드 │ │ 필드 │ │ 필드 │ │ │ │ │ │ │ │(이름)│ │(나이)│ │(전화번호)│ │ │ │ │ │ │ └──────┘ └──────┘ └──────────┘ │ │ │ │ │ └───────────────────────────────────────┘ │ │ │ └─────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────┘

각 단계의 정의

  • 필드 (Field): 데이터의 가장 작은 논리적 단위. 하나의 속성을 표현한다. 예: 이름, 학번, 생년월일
  • 레코드 (Record): 하나의 개체(Entity)에 대한 관련 필드들의 집합. 예: 한 학생의 {학번, 이름, 학과, 학년} 묶음
  • 파일 (File): 같은 유형의 레코드들의 집합. 예: 전체 학생 레코드를 담은 students.dat
  • 데이터베이스 (Database): 서로 관련된 여러 파일들의 집합. 예: 학생, 수강, 강의 파일을 통합한 학사관리 DB

키 (Key)

레코드 중에서 키(Key)는 특별한 역할을 한다. 키는 각 레코드를 유일하게 식별하는 필드(또는 필드 조합)다.

  • 기본 키 (Primary Key): 레코드를 유일하게 식별. 예: 학번, 주민등록번호
  • 보조 키 (Secondary Key): 검색에 사용되나 유일성 보장 안 함. 예: 이름, 학과
암기 팁 필드 → 레코드 → 파일 → 데이터베이스: "필레파데"로 기억하라. 각각 속성 하나 → 개체 하나 → 같은 유형 모음 → 관련 파일 전체.

3. 레코드의 종류

고정 길이 레코드 (Fixed-Length Record)

모든 레코드가 동일한 크기를 가진다. 이름(20B) + 나이(4B) + 전화(15B) = 39B 처럼 항상 39바이트.

  • 장점 레코드 번호로 위치 계산 쉬움: 위치 = 레코드번호 × 레코드크기
  • 단점 가변 길이 데이터(이름, 설명 등)에서 공간 낭비

가변 길이 레코드 (Variable-Length Record)

레코드마다 크기가 다르다. 구분자(Delimiter)나 길이 필드를 사용하여 경계를 표시한다.

  • 장점 공간 효율적 — 실제 데이터 크기만큼만 사용
  • 단점 특정 레코드 위치 계산이 복잡함 (순차 탐색 필요)

4. 세 가지 기본 파일 조직 방법

파일 안에서 레코드를 어떻게 배치하고 검색하느냐에 따라 크게 세 가지 방식으로 나뉜다. 이것이 파일구조 과목의 핵심 축이다.

방식 원리 검색 성능 삽입 성능 주요 활용
순차 파일
(Sequential)		 레코드를 순서대로 저장 O(n) 선형 탐색 느림 (정렬 유지 시) 일괄 처리(Batch)
인덱스 파일
(Indexed)		 별도 인덱스로 위치 저장 빠름 인덱스 탐색 후 직접 접근 중간 범위 검색, DB 인덱스
직접 파일
(Direct/Hash)		 키로 저장 위치 계산(해시) O(1) 평균 빠름 키 기반 단건 검색
