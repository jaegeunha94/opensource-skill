Day 9: 외부 정렬 (External Sorting)

📅 2026-06-29 ⏱ 약 60분 📚 파일구조 과목 🔢 Lesson 9 / 10
학습 목표 이 레슨을 마치면: (1) 메모리에 올릴 수 없는 대용량 데이터를 정렬하는 외부 정렬의 두 단계(런 생성·합병)를 설명할 수 있다. (2) k-way 합병 정렬의 패스 수 계산 공식을 적용할 수 있다. (3) 교체 선택(Replacement Selection)이 런 길이를 늘리는 원리를 이해한다. (4) 폴리페이즈(Polyphase) 합병이 I/O를 줄이는 방식을 설명할 수 있다.

0. 왜 외부 정렬이 필요한가?

배열이 메모리보다 작으면 퀵소트·힙소트로 충분하다. 그러나 수 TB의 로그 파일, 수억 건의 거래 내역은 RAM에 절대 들어가지 않는다. 외부 정렬(External Sorting)은 디스크를 보조 기억장치로 활용해 데이터 전체가 메모리를 초과해도 정렬을 완료하는 기법이다.

핵심 제약 외부 정렬의 비용은 디스크 I/O 횟수로 측정한다. CPU 연산은 디스크 접근보다 10만 배 이상 빠르므로 I/O 최소화가 설계의 핵심이다.

외부 정렬은 두 단계로 나뉜다.

  1. 런 생성(Run Generation): 메모리에 들어가는 크기씩 읽어 내부 정렬 후 디스크에 기록. 이 정렬된 조각을 런(Run)이라 한다.
  2. 합병(Merge): 여러 런을 반복적으로 합쳐 하나의 정렬 파일로 만든다.

1. 기본 2-way 외부 합병 정렬

가장 단순한 형태. 메모리에 버퍼 2개를 두고 두 런을 한 번에 합병한다.

전체 16블록, 메모리 4블록 가정 ────────────────────────────────────────────────────── 패스 0 (런 생성): 4블록씩 읽어 정렬 → 런 4개 R1[B1..B4] R2[B5..B8] R3[B9..B12] R4[B13..B16] ────────────────────────────────────────────────────── 패스 1 (2-way 합병): R1+R2→R12, R3+R4→R34 R12[B1..B8] R34[B9..B16] ────────────────────────────────────────────────────── 패스 2: R12+R34 → 최종 정렬 파일 ✓

1-1. 패스 수 공식 (2-way)

  • 초기 런 수: R = ⌈N / M⌉
  • 합병 패스 수: ⌈log₂ R⌉
  • 총 패스 수: 1 + ⌈log₂ ⌈N/M⌉⌉
예시 N=1000블록, M=10블록 → 런 수=100 → 합병 패스=⌈log₂ 100⌉=7 → 총 패스=8
각 패스마다 전체 데이터를 읽고 씀 → 총 I/O ≈ 8 × 2 × 1000 = 16,000 블록 접근

2. k-way 합병 정렬

패스를 줄이려면 한 번에 k개 런을 동시에 합병하면 된다.

2-1. 메모리 분할

메모리 M=12블록, k=5 합병 [입력1: 2블록] [입력2: 2블록] [입력3: 2블록] [입력4: 2블록] [입력5: 2블록] [출력: 2블록] 런 R1 런 R2 런 R3 런 R4 런 R5 → 디스크

각 단계에서 k개 런의 선두 원소를 최솟값 힙(Min-Heap)으로 비교해 가장 작은 원소를 출력 버퍼에 보낸다.

k=3 합병 예시 R1 → [3, 7, 12, ...] R2 → [1, 5, 9, ...] R3 → [4, 6, 11, ...] Min-Heap: {1(R2), 3(R1), 4(R3)} → 출력: 1 R2 다음 원소 5 삽입 → {3(R1), 4(R3), 5(R2)} → 출력: 3 R1 다음 원소 7 삽입 → {4(R3), 5(R2), 7(R1)} → 출력: 4 ...

2-2. k-way 패스 수 공식

변수의미공식
R초기 런 수⌈N / M⌉
k합병 차수 (최대)M - 1 (출력 버퍼 1개 제외)
합병 패스 수⌈logₖ R⌉
총 패스 수1 + ⌈logₖ ⌈N/M⌉⌉
k 증가의 트레이드오프 k가 크면 패스 수가 줄어 I/O가 감소한다. 그러나 입력 버퍼 크기가 작아지면 디스크 탐색 횟수가 늘어나는 역효과가 있다. 실제 DBMS는 사용 가능 메모리와 런 수에 따라 k를 동적으로 결정한다.

3. 교체 선택(Replacement Selection)으로 런 길이 늘리기

런 생성 단계에서 단순 내부 정렬 대신 교체 선택을 쓰면 입력이 무작위일 때 런의 평균 길이를 2M으로 늘릴 수 있다.

3-1. 알고리즘

  1. 메모리를 힙(Min-Heap)으로 유지한다.
  2. 힙의 최솟값을 출력 후 디스크에서 새 원소 하나를 읽는다.
  3. 새 원소 ≥ 방금 출력한 값 → 현재 런에 추가(힙 삽입).
  4. 새 원소 < 방금 출력한 값 → 보류 집합에 보관. 힙이 빌 때 새 런을 시작한다.
M=3, 입력: 5, 1, 8, 3, 9, 2, 7, 4, 6 초기 힙: {1, 5, 8} 출력 1 → 다음 3 읽음. 3 ≥ 1 → 힙 유지. 힙:{3,5,8} 출력:[1] 출력 3 → 다음 9 읽음. 9 ≥ 3 → 힙 유지. 힙:{5,8,9} 출력:[1,3] 출력 5 → 다음 2 읽음. 2 < 5 → 보류. 힙:{8,9} 보류:{2} 출력:[1,3,5] 출력 8 → 다음 7 읽음. 7 < 8 → 보류. 힙:{9} 보류:{2,7} 출력 9 → 다음 4 읽음. 4 < 9 → 보류. 힙 비어있음 → 런 1 종료 런 1: [1, 3, 5, 8, 9] ← 길이 5 (단순 M=3 정렬이면 길이 3) 보류 {2,4,7} → 런 2 시작
평균 런 길이 = 2M 무작위 입력에서 교체 선택으로 생성한 런의 평균 길이는 2M이다. 런 수가 절반으로 줄면 합병 패스 수도 하나 줄어 전체 I/O 비용이 크게 감소한다.

4. 폴리페이즈 합병(Polyphase Merge)

일반 k-way 합병은 합병 후 런을 다시 분산시키는 패스가 필요하다. 폴리페이즈 합병은 테이프(파일)에 피보나치 수열 비율로 런을 배분해 분산 패스 없이 합병만 반복한다.

3-way 폴리페이즈 (테이프 T1, T2, T3, 초기 런 13개) 피보나치 분배: T1=8개, T2=5개, T3=0(출력) 패스 1: T1(8)+T2(5) → T3 → T3:5개, T1:3개 남음 패스 2: T1(3)+T3(5) → T2 → T2:3개, T3:2개 남음 패스 3: T2(3)+T3(2) → T1 → T1:2개, T2:1개 남음 패스 4: T1(2)+T2(1) → T3 → T3:1개, T1:1개 남음 패스 5: T1(1)+T3(1) → T2 → T2:1개 (최종 정렬 파일) ✓
방식3-way, 런 13개일 때특징
균등 합병⌈log₂ 13⌉=4 패스 + 분산 패스구현 단순, I/O 낭비
폴리페이즈 합병5 패스 (분산 없음)실제 I/O 횟수 더 적음
실무 적용 PostgreSQL의 external merge sort, MySQL InnoDB의 정렬 버퍼가 유사한 원리로 작동한다. sort_buffer_size 파라미터가 곧 M(메모리 크기)에 해당한다.

5. 정렬 방식별 I/O 비용 요약

방식총 패스 수총 I/O (N=1000, M=10 기준)
2-way 합병1+⌈log₂ 100⌉=88×2×1000=16,000
k=9-way 합병1+⌈log₉ 100⌉=33×2×1000=6,000
교체 선택+k=9-way1+⌈log₉ 50⌉=3≈6,000 (런 수 절반)
