Day 6: B+-트리 (B+-Tree)

📅 2026-06-26 ⏱ 약 50분 📚 파일구조 과목 🔢 Lesson 6 / 10
학습 목표 이 레슨을 마치면: (1) B+-트리가 B-트리와 어떻게 다른지 구조적으로 설명할 수 있다. (2) 리프 노드 연결 리스트가 범위 검색에 왜 유리한지 이해한다. (3) B+-트리의 삽입·삭제 과정을 단계별로 따라갈 수 있다. (4) 실제 DBMS가 B+-트리를 선택하는 이유를 설명할 수 있다.

1. B-트리의 한계와 B+-트리의 등장

Day 5에서 B-트리는 균형 다방향 탐색 트리로서 디스크 I/O를 획기적으로 줄인다고 배웠다. 그런데 B-트리에는 범위 검색(Range Search)에서 중요한 비효율이 있다.

B-트리의 범위 검색 문제 B-트리에서 "나이가 20~30인 모든 레코드"를 찾으려면, 20을 찾은 뒤 다음 키로 이동하기 위해 매번 루트에서 다시 탐색해야 한다. 키가 내부 노드와 리프 노드에 모두 흩어져 있기 때문에 순차 스캔이 불가능하다.

B+-트리(B+ Tree)는 이 문제를 해결한다. 핵심 아이디어는 두 가지다:

  1. 모든 실제 데이터(또는 레코드 포인터)는 리프 노드에만 저장한다.
  2. 리프 노드들을 연결 리스트(Linked List)로 이어 순차 접근을 가능하게 한다.
B-트리 (키가 내부·리프 모두 존재): [30 | 50] ↙ ↓ ↘ [10|20] [35|40] [60|70] ← 키가 여기저기 흩어짐 B+-트리 (내부 노드 = 라우팅 키, 리프 = 실제 데이터): [30 | 50] ← 내부 노드: 안내 역할만 ↙ ↓ ↘ [10|20|30] ↔ [35|40|50] ↔ [60|70] ← 리프 연결!

2. B+-트리 구조 상세

2-1. 내부 노드 (Internal Node)

내부 노드는 탐색을 안내하는 라우팅 키(Routing Key)만 저장한다. 실제 레코드 포인터는 없다. 차수 m인 B+-트리에서 내부 노드는 B-트리와 동일한 규칙을 따른다:

  • 최대 m-1개의 키, 최대 m개의 자식 포인터
  • 루트 외 내부 노드: 최소 ⌈m/2⌉개의 자식

2-2. 리프 노드 (Leaf Node)

리프 노드는 B-트리와 완전히 다르다:

  • 모든 탐색 키(또는 키+레코드 포인터 쌍)를 정렬 순서로 저장한다.
  • 마지막 슬롯에 다음 리프 포인터(Next Leaf Pointer)를 저장한다.
  • 리프 노드들이 이중 또는 단방향 연결 리스트를 형성한다.
차수 m=4, 리프 노드 구조 (슬롯 3개 + 다음 포인터 1개): ┌──────┬──────┬──────┬──────────┐ │ key₁ │ key₂ │ key₃ │ next → │ │ rec₁ │ rec₂ │ rec₃ │ │ └──────┴──────┴──────┴──────────┘ ↓ ↓ ↓ 실제 레코드 / 디스크 블록 주소

2-3. B-트리 vs B+-트리 비교

항목 B-트리 B+-트리
데이터 위치 내부 노드 + 리프 노드 리프 노드만
내부 노드 역할 키 저장 + 탐색 안내 탐색 안내(라우팅)만
리프 노드 연결 없음 연결 리스트로 연결
범위 검색 비효율적 (매번 루트부터) 효율적 (리프 순차 스캔)
단건 탐색 최악 경우 O(log n) — 내부 노드에서 발견 가능 O(log n) — 항상 리프까지 내려감
공간 효율 중간 (내부에 데이터 포함) 높음 (내부 노드에 더 많은 키)
실제 DBMS 사용 거의 없음 MySQL InnoDB, PostgreSQL 등
왜 B+-트리가 표준인가? 내부 노드에 데이터 포인터가 없으므로 같은 노드 크기(=디스크 블록 크기)에 더 많은 라우팅 키를 담을 수 있다. 즉, 트리 높이가 낮아지고 디스크 I/O가 줄어든다. 리프 연결 덕분에 범위 검색·전체 스캔도 빠르다.

3. 탐색 (Search)

3-1. 단건 탐색

B-트리와 거의 같지만, B+-트리는 항상 리프까지 내려간다. 내부 노드의 키가 라우팅용이므로, 내부 노드에서 일치하는 키를 발견해도 계속 내려가야 한다.

B+-트리 (m=4) 에서 key=35 탐색 루트: [30 | 50] ↙ ↘ [10|20|30] [35|40|50] [60|70] 35 ≥ 30 → 오른쪽 자식으로 리프 [35|40|50] 에서 35 발견 → 해당 레코드 포인터 반환

3-2. 범위 탐색 (20 ≤ key ≤ 45)

  1. 20을 단건 탐색하여 20이 있는 리프를 찾는다.
  2. 리프에서 20부터 오른쪽으로 스캔한다.
  3. 현재 리프를 다 읽으면 next 포인터를 따라 다음 리프로 이동한다.
  4. 45보다 큰 키가 나오면 종료한다.
범위 탐색 20~45: [10|20|30] →next→ [35|40|50] →next→ [60|70] ↑ ↑ 20 발견 40까지 수집, 50 > 45이면 종료 결과: {20, 30, 35, 40} — 루트 재탐색 없이 리프 연결만으로 해결!

4. 삽입 (Insertion)

삽입은 항상 리프 노드에서 이루어진다. B-트리 삽입과 비슷하지만 리프 분할 방식이 다르다.

4-1. 삽입 절차

  1. 삽입할 리프 노드를 탐색으로 찾는다.
  2. 리프에 빈 슬롯이 있으면 정렬 순서로 삽입 → 완료.
  3. 리프가 꽉 차 있으면 리프 분할을 수행한다.

4-2. 리프 분할

B-트리와의 결정적 차이: B+-트리에서는 리프 분할 시 중간 키를 부모에 복사(copy up)한다. B-트리처럼 올리고(push up) 리프에서 제거하지 않는다. 리프에 모든 데이터가 있어야 하기 때문이다.

차수 m=4, 리프 슬롯 3개. key=25 삽입, 리프 [10|20|30] 가 꽉 참. 삽입 시도: [10|20|25|30] → 오버플로우 → 분할 필요 분할: ⌈4/2⌉ = 2 → 왼쪽 2개 / 오른쪽 2개 왼쪽 리프: [10 | 20] 오른쪽 리프: [25 | 30] 부모에 오른쪽 리프의 첫 번째 키(25)를 "복사(copy up)": 부모: [25 | 50] ← 25는 리프에도 여전히 존재! ↙ ↓ ↘ [10|20] [25|30] [35|40|50] (B-트리와 달리 25가 리프에서 사라지지 않음)

4-3. 내부 노드 분할

내부 노드가 꽉 차면 B-트리와 동일하게 중간 키를 위로 올리고(push up) 내부 노드에서 제거한다. 내부 노드는 데이터가 아니라 라우팅 키만 갖기 때문이다.

Copy Up vs Push Up 정리 리프 분할 → copy up (중간 키를 부모에 복사, 리프에도 남김)
내부 노드 분할 → push up (중간 키를 부모로 올리고 현재 노드에서 제거)

5. 삭제 (Deletion)

삭제는 B-트리보다 단순한 면이 있다. 데이터는 항상 리프에만 있으므로, 삭제 대상을 찾으면 리프에서만 제거하면 된다. 내부 노드에 같은 키가 있어도 라우팅 역할을 하므로 즉시 제거하지 않아도 된다.

5-1. 삭제 절차

  1. 삭제할 키가 있는 리프를 탐색으로 찾는다.
  2. 리프에서 키를 삭제한다.
  3. 삭제 후 리프의 키 수가 최소(⌈m/2⌉ - 1) 이상이면 완료.
  4. 미달이면 형제에서 빌림(Borrow) 또는 형제와 합병(Merge)을 한다.

5-2. 빌림과 합병

빌림 예시 (m=4, 리프 슬롯 3개, 최솟값 ⌈4/2⌉-1 = 1) 삭제 전: 부모: [30 | 50] ↙ ↓ ↘ [10|20|30] ↔ [35|40|50] ↔ [60|70] [35|40|50]에서 35 삭제 → [40|50] → 키 2개 ≥ 1 → 완료 (최솟값=1이므로 2개는 문제없음 — 빌림 불필요)
합병 예시 (m=4, 최솟값 = 1) 삭제 전: 부모: [30 | 50] ↙ ↓ ↘ [10|20] ↔ [35] ↔ [60|70] [35]에서 35 삭제 → [] → 키 0개 < 1 → 재조정 필요 형제 [10|20] 에서도 빌릴 키가 없으면 합병: [10|20] + [] → [10|20] (오른쪽 빈 리프 제거) 부모에서 구분자 키(30) 제거: 부모: [50] ↙ ↘ [10|20] ↔ [60|70]
내부 노드의 잔여 키 처리 합병 후 내부 노드에서 구분자 키를 제거할 때, B+-트리는 해당 키가 아직 리프에 남아 있을 수 있다. 리프의 키는 그대로 두어도 된다 — 내부 노드는 라우팅용이므로 일치하지 않아도 탐색에 지장이 없다.

6. B+-트리 높이와 팬아웃

실제 DBMS에서 디스크 블록 크기는 보통 4KB~16KB이다. 키+포인터 쌍이 16바이트라면 하나의 내부 노드에 수백 개의 자식을 가질 수 있다. 

블록 크기 8KB, 키+포인터 = 16바이트
팬아웃(fan-out) ≈ 8192 / 16 = 512

n = 1,000,000 레코드일 때:
높이 ≤ log₅₁₂(1,000,000) ≈ 2.06  →  최악 3번의 I/O

루트를 메모리에 캐시하면 실제로는 1~2번의 디스크 I/O만으로 수백만 건 데이터에서 원하는 레코드를 찾는다.

파일 구조 단건 탐색 범위 탐색 삽입 삭제 실사용
순차 파일 O(log n) O(log n + k) O(n) O(n) 읽기 전용 배치
해싱 O(1) 불가 O(1) O(1) 동등 조건 검색
B-트리 O(log n) O(k·log n) O(log n) O(log n) 제한적
B+-트리 O(log n) O(log n + k) O(log n) O(log n) DBMS 기본 인덱스

k = 범위 내 레코드 수

7. 실제 적용 — DBMS 인덱스

MySQL InnoDB의 클러스터드 인덱스(Clustered Index)는 B+-트리로 구현된다. 리프 노드에 실제 행 데이터(Row Data)가 저장되고, 리프 간 연결로 범위 쿼리가 빠르다. 

-- 이 쿼리는 B+-트리 범위 탐색을 이용한다:
SELECT * FROM orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31'
ORDER BY order_date;

-- 실행 계획: Index Range Scan
--   → 시작 리프 탐색 후 next 포인터로 순차 스캔
--   → 정렬 비용 없음 (이미 리프가 정렬됨)
PostgreSQL의 경우 PostgreSQL도 기본 인덱스 타입이 B-트리(내부적으로는 B+-트리 변형)다. CREATE INDEX를 실행하면 자동으로 B+-트리 인덱스가 생성된다.

A. Push Up — 중간 키를 부모로 올리고 리프에서 제거한다.

확인 문제 3

A. 내부 노드에도 데이터가 있어 중간에 탐색을 멈출 수 있기 때문이다.

C. 리프 노드가 연결 리스트로 연결되어 있어 시작 리프 탐색 후 순차 스캔이 가능하기 때문이다.
D. B+-트리는 키를 정렬하지 않고 저장해 탐색 비교 횟수가 적기 때문이다.

확인 문제 4

B+-트리에서 단건 탐색(Point Query)의 시간복잡도는?
A. O(1) — 리프 노드 연결로 즉시 접근 가능하다.
B. O(n) — 모든 리프를 스캔해야 한다.
C. O(log n) — 루트에서 리프까지 트리 높이만큼 탐색한다.
D. O(n log n) — 분할이 발생할 때마다 재정렬이 필요하다.
