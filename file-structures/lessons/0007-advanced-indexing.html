Day 7: 고급 인덱싱 (Advanced Indexing)

📅 2026-06-27 ⏱ 약 55분 📚 파일구조 과목 🔢 Lesson 7 / 10
학습 목표 이 레슨을 마치면: (1) 다중 키 인덱스의 구조와 복합 조건 검색 최적화를 설명할 수 있다. (2) 비트맵 인덱스가 낮은 카디널리티 컬럼에 유리한 이유를 이해한다. (3) 주어진 쿼리 패턴에 맞는 인덱스 구조를 선택하고 그 근거를 논리적으로 설명할 수 있다. (4) 인덱스가 쓰기 성능에 미치는 트레이드오프를 인식한다.

1. 다중 키 인덱스 (Multi-Key Index)

Day 3에서 배운 단순 인덱스는 하나의 탐색 키(Search Key)만 사용한다. 그런데 실제 쿼리는 여러 컬럼을 동시에 조건으로 사용하는 경우가 많다. 다중 키 인덱스는 두 개 이상의 필드를 묶어 하나의 인덱스를 구성한다. 

-- 두 컬럼을 동시에 조건으로 사용하는 쿼리 예시
SELECT * FROM employees
WHERE department = 'Engineering'
  AND salary BETWEEN 60000 AND 90000;

1-1. 복합 인덱스 (Composite Index)

가장 일반적인 다중 키 인덱스다. 여러 컬럼을 순서대로 연결해 하나의 탐색 키로 만든다. 예를 들어 (department, salary) 복합 인덱스는 다음 형태로 레코드를 정렬한다:

복합 인덱스 (department, salary) 의 정렬 순서 (Engineering, 55000) → 레코드 A (Engineering, 72000) → 레코드 B (Engineering, 90000) → 레코드 C (HR, 48000) → 레코드 D (HR, 63000) → 레코드 E (Marketing, 51000) → 레코드 F → department 기준 1차 정렬, 같은 department 내 salary 기준 2차 정렬

1-2. 선두 컬럼 규칙 (Leading Column Rule)

복합 인덱스는 왼쪽(선두) 컬럼부터 순서대로 인덱스를 활용할 수 있다. 중간 컬럼을 건너뛰면 인덱스를 온전히 사용할 수 없다.

쿼리 조건 인덱스 (dept, salary) 활용 여부 설명
WHERE dept = 'Eng' 활용 가능 선두 컬럼만 사용 — 인덱스 범위 스캔
WHERE dept = 'Eng' AND salary = 72000 최대 활용 두 컬럼 모두 인덱스 조건으로 사용
WHERE dept = 'Eng' AND salary > 60000 활용 가능 선두 등치 + 후속 범위 — 인덱스 범위 스캔
WHERE salary = 72000 활용 불가 선두 컬럼 없이 후속 컬럼만 — 풀 스캔 필요
WHERE salary > 60000 AND dept = 'Eng' 부분 활용 옵티마이저가 dept 조건으로 필터 후 나머지 조건 적용
컬럼 순서가 핵심이다 (dept, salary) 인덱스와 (salary, dept) 인덱스는 서로 다른 쿼리 패턴에 최적화된다. 인덱스 설계 전에 어떤 컬럼이 조건에 자주 등장하는지, 등치 조건인지 범위 조건인지 반드시 파악해야 한다.

1-3. 다중 인덱스 교차 (Index Intersection)

여러 개의 단일 컬럼 인덱스를 교집합(Intersection)으로 결합해 복합 조건을 처리하는 방법이다. 복합 인덱스를 만들기 어렵거나 조건이 동적으로 바뀔 때 사용한다.

단일 인덱스 교차 처리 과정 (dept 조건 + salary 조건) 1. 인덱스 A (dept): dept='Engineering' → {RID: 101, 102, 105, 108} 2. 인덱스 B (salary): salary BETWEEN 60000 AND 90000 → {RID: 102, 104, 105, 109} 3. 교집합(Intersection): {102, 105} 4. 두 RID에 해당하는 레코드만 테이블에서 가져옴
복합 인덱스 vs 인덱스 교차 복합 인덱스는 단일 B+-트리 순회로 처리하므로 더 효율적이다. 인덱스 교차는 두 인덱스를 각각 순회하고 RID 집합을 정렬·교집합해야 하므로 오버헤드가 크다. 자주 쓰이는 복합 조건은 복합 인덱스가 우선이다.

2. 비트맵 인덱스 (Bitmap Index)

비트맵 인덱스는 카디널리티(Cardinality)가 낮은 컬럼, 즉 값의 종류가 적은 컬럼에 특화된 인덱스다. 성별, 지역, 등급처럼 값이 몇 가지뿐인 컬럼에서 B+-트리보다 훨씬 효율적이다.

2-1. 비트맵 인덱스 구조

각 컬럼 값(Distinct Value)마다 비트 벡터(Bit Vector)를 하나 생성한다. 비트 벡터의 각 비트는 테이블의 한 행(Row)에 대응하며, 해당 행이 그 값을 가지면 1, 아니면 0이다.

employees 테이블 (5개 행) Row | dept | gender -----|-------------|------- 1 | Engineering | M 2 | Engineering | F 3 | HR | F 4 | Marketing | M 5 | HR | M 비트맵 인덱스 — dept 컬럼: dept=Engineering : [1, 1, 0, 0, 0] dept=HR : [0, 0, 1, 0, 1] dept=Marketing : [0, 0, 0, 1, 0] 비트맵 인덱스 — gender 컬럼: gender=M : [1, 0, 0, 1, 1] gender=F : [0, 1, 1, 0, 0]

2-2. 비트 연산으로 쿼리 처리

비트맵 인덱스의 강점은 복합 조건 쿼리를 비트 AND/OR 연산으로 처리한다는 점이다. 비트 연산은 CPU 수준에서 병렬 처리되어 매우 빠르다.

쿼리: WHERE dept = 'HR' AND gender = 'F' dept=HR : [0, 0, 1, 0, 1] gender=F : [0, 1, 1, 0, 0] AND 연산 ────────────── 결과 : [0, 0, 1, 0, 0] → Row 3만 해당 쿼리: WHERE dept = 'Engineering' OR dept = 'HR' dept=Engineering : [1, 1, 0, 0, 0] dept=HR : [0, 0, 1, 0, 1] OR 연산 ────────────── 결과 : [1, 1, 1, 0, 1] → Row 1, 2, 3, 5 해당

2-3. 비트맵 인덱스의 장단점

항목 비트맵 인덱스 B+-트리 인덱스
적합한 카디널리티 낮음 (성별, 지역, 등급 등) 높음 (이름, 주문번호, 날짜 등)
복합 조건 처리 비트 AND/OR로 초고속 인덱스 교차 필요
저장 공간 매우 작음 (비트 단위) 중간
동시 쓰기 성능 매우 나쁨 (행 삽입 시 모든 비트벡터 잠금) 양호 (해당 노드만 잠금)
주요 사용 환경 데이터 웨어하우스(DW), OLAP, 읽기 집약적 분석 OLTP, 빈번한 업데이트가 있는 시스템
집계 쿼리 (COUNT, SUM) 비트 카운트(popcount)로 초고속 인덱스 스캔 필요
비트맵 인덱스와 동시성 단 한 행이 삽입·수정·삭제될 때도 해당 컬럼의 모든 비트 벡터를 변경해야 한다. 이로 인해 OLTP 환경(잦은 쓰기)에서는 심각한 잠금(Lock) 경합이 발생한다. Oracle Database는 비트맵 인덱스를 지원하지만, MySQL InnoDB는 지원하지 않는 이유가 바로 이것이다.

3. 인덱스 선택 기준

인덱스가 많을수록 검색은 빠르지만 삽입·수정·삭제 시 유지 비용이 증가한다. 또한 인덱스 자체가 저장 공간을 차지한다. 다음 기준을 체계적으로 적용해 필요한 인덱스만 생성해야 한다.

3-1. 쿼리 패턴 분석

먼저 시스템에서 자주 실행되는 쿼리를 분류한다.

  • 점 쿼리 (Point Query): WHERE id = 42 — 단건 등치 검색. B+-트리나 해싱 모두 적합.
  • 범위 쿼리 (Range Query): WHERE date BETWEEN A AND B — B+-트리가 유일한 실용적 선택.
  • 다중 조건 쿼리 (Multi-Condition): 복합 인덱스 설계 필요.
  • 집계 쿼리 (Aggregate): COUNT, SUM — 커버링 인덱스 또는 비트맵 인덱스.
  • 조인 쿼리 (Join): 조인 키 컬럼에 인덱스를 두어 Nested Loop Join을 최적화.

3-2. 인덱스 선택 체크리스트

질문 YES이면 NO이면
이 컬럼이 WHERE 절에 자주 등장하는가? 인덱스 후보 인덱스 불필요
카디널리티가 높은가? (값이 다양한가?) B+-트리 인덱스 유리 비트맵 인덱스 고려
범위 조건 (>, <, BETWEEN)이 자주 쓰이는가? B+-트리 필수 해싱도 가능
테이블이 자주 업데이트되는가? 인덱스 수를 최소화 읽기 최적화 우선
인덱스 컬럼만으로 쿼리가 해결되는가? 커버링 인덱스 설계 테이블 접근(Heap Fetch) 필요
두 컬럼이 항상 함께 조건에 등장하는가? 복합 인덱스 설계 단일 인덱스 분리

3-3. 선택도 (Selectivity)와 인덱스 효율

선택도(Selectivity) = (조건에 맞는 행 수) / (전체 행 수)

선택도가 낮을수록(= 조건에 맞는 행이 적을수록) 인덱스가 더 효과적이다. 반대로 선택도가 높으면(= 거의 모든 행이 조건에 해당) 인덱스를 타더라도 결국 대부분 행을 읽게 되어 풀 테이블 스캔(Full Table Scan)보다 오히려 느릴 수 있다.

선택도 예시 (100만 건 테이블) 컬럼 A (gender — 값 2종류): 선택도 ≈ 0.5 (50만 건 해당) → 인덱스 효과 미미 → 비트맵 고려 단, 복합 조건과 함께라면 AND로 필터링 효과 있음 컬럼 B (user_id — 값 100만 종류): 선택도 ≈ 0.000001 (1건 해당) → B+-트리 인덱스 매우 효과적 컬럼 C (status — 값 4종류, 95%가 'completed'): 선택도 ≈ 0.95 (WHERE status='completed') → 인덱스보다 풀 스캔이 나을 수 있음 반면 WHERE status='pending'은 선택도 ≈ 0.01 → 인덱스 효과적

3-4. 커버링 인덱스 (Covering Index)

쿼리가 필요로 하는 모든 컬럼이 인덱스에 포함되어 있을 때, 데이터베이스는 실제 테이블을 읽지 않고 인덱스만으로 결과를 반환할 수 있다. 이를 커버링 인덱스 또는 Index-Only Scan이라고 한다. 

-- 인덱스: (dept, salary, name)
SELECT name, salary
FROM employees
WHERE dept = 'Engineering';

-- 위 쿼리는 dept, salary, name 모두 인덱스에 있음
-- → 테이블 접근(Heap Fetch) 불필요 → I/O 대폭 절감
INCLUDE 컬럼 (PostgreSQL, SQL Server) 일부 DBMS는 정렬 키에 포함시키지 않고 리프 노드에만 추가 컬럼을 저장하는 INCLUDE 절을 지원한다. 커버링 인덱스를 구성할 때 불필요한 키 블로트(bloat) 없이 원하는 컬럼을 인덱스에 포함시킬 수 있다.

3-5. 인덱스 설계 원칙 요약

인덱스 설계 5원칙
  1. 쿼리 주도(Query-Driven): 실제 자주 실행되는 쿼리를 먼저 파악한다.
  2. 고선택도 우선: 선택도가 낮은(결과 행이 적은) 컬럼에 인덱스를 건다.
  3. 등치 → 범위 순서: 복합 인덱스에서 등치 조건 컬럼을 앞에, 범위 조건 컬럼을 뒤에 배치한다.
  4. 쓰기 비용 인식: 인덱스 하나 추가 = 모든 INSERT/UPDATE/DELETE마다 인덱스 유지 비용 추가.
  5. 커버링 인덱스 고려: 자주 실행되는 쿼리는 필요한 컬럼을 인덱스에 포함해 테이블 접근을 줄인다.

4. 파일구조별 인덱스 적용 종합 비교

인덱스 유형 점 쿼리 범위 쿼리 복합 조건 쓰기 부담 주요 적용처
단일 B+-트리 O(log n) O(log n + k) 인덱스 교차 중간 OLTP 기본 인덱스
복합 B+-트리 O(log n) O(log n + k) 단일 스캔 중간 다중 조건 OLTP
해시 인덱스 O(1) 불가 미지원 낮음 동등 검색 전용
비트맵 인덱스 비트 스캔 가능(비효율) 비트 AND/OR 매우 높음 DW·OLAP 분석
