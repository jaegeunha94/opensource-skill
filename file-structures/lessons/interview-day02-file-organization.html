파일 조직 선택 — Sequential / Indexed / Hash

조회, 삽입, 삭제, range query 요구사항으로 저장 구조를 고르는 판단력 훈련

Q

  • "삽입이 잦은 로그 파일과 조회가 잦은 마스터 테이블은 왜 다른 구조를 써야 합니까?"
  • "정렬된 순차 파일에 새 레코드를 삽입할 때 어떤 문제가 생깁니까?"
  • "해시 인덱스는 조회가 O(1)인데, 왜 모든 테이블에 해시 인덱스를 쓰지 않습니까?"

전제 개념 — 쉬운 한국어로 먼저 이해하기

2-1. 파일 조직(File Organization)이란

파일 조직은 레코드를 디스크 블록에 물리적으로 어떻게 배치할 것인가에 대한 설계입니다. Day 1에서 다룬 것처럼 디스크는 block 단위로 읽고 쓰기 때문에, 같은 데이터라도 어떻게 배치하느냐에 따라 조회/삽입/삭제 비용이 완전히 달라집니다. 크게 네 가지 방식이 있습니다: heap(pile) file, sequential file, indexed file, hashed file.

2-2. Heap File (Pile File) — 정렬 없이 그냥 쌓기

레코드를 들어오는 순서대로 파일 끝에 계속 추가(append)만 하는 방식입니다. 정렬도, 인덱스도 없습니다.

[ Heap File ] Block 0 Block 1 Block 2 (마지막) ┌────────────┐ ┌────────────┐ ┌────────────┐ │ id=57 │ │ id=12 │ │ id=91 │ ← 순서 무작위 │ id=3 │ │ id=88 │ │ │ ← 새 레코드는 항상 여기 추가 │ id=200 │ │ id=44 │ │ │ └────────────┘ └────────────┘ └────────────┘ 삽입: 마지막 block에 추가 → O(1) I/O (순차 쓰기) 조회(특정 id): 어디 있는지 몰라 전체 스캔 → O(N/B) I/O (B = block당 레코드 수) 삭제: 위치를 찾은 뒤 삭제 표시(tombstone) 또는 마지막 레코드로 대체

삽입은 가장 저렴하지만(항상 끝에 순차 쓰기), 조회는 가장 비쌉니다(항상 전체 스캔). 로그 적재나 임시 적재 영역(staging)처럼 "일단 빨리 쓰고, 나중에 한꺼번에 처리"하는 상황에 맞습니다.

2-3. Sequential File — 정렬해서 저장하기

특정 키(예: id) 기준으로 물리적으로 정렬된 상태로 레코드를 저장하는 방식입니다. 정렬되어 있으므로 이진 탐색으로 검색할 수 있고, 순서대로 읽으면 범위 검색이 순차 I/O로 처리됩니다.

[ Sequential File — key 기준 정렬 ] Block 0 Block 1 Block 2 ┌────────────┐ ┌────────────┐ ┌────────────┐ │ id=3 │ │ id=44 │ │ id=91 │ │ id=12 │ │ id=57 │ │ id=200 │ └────────────┘ ┌────────────┘ └────────────┘ 조회(id=57): 정렬되어 있으므로 이진 탐색 → O(log₂(N/B)) I/O range(id 12~91): 시작점만 찾고 이후 순차 스캔 → 매우 저렴 (순차 I/O) 삽입(id=50): 정렬 위치를 지켜야 함 → 뒤 레코드를 전부 밀어야 함 → O(N/B) I/O [ 실무 대안: Overflow Area ] Main File (정렬 유지) Overflow Area (임시, 미정렬) ┌────────────┐ ┌────────────┐ │ id=3 │ │ id=50 (신규)│ ← main을 밀지 않고 여기 먼저 추가 │ id=12 │ ─┬link┐ │ id=61 (신규)│ │ id=44 │ │ └───▶│ │ │ id=57 │◀──┘ └────────────┘ └────────────┘ 주기적으로 main + overflow를 merge하여 재정렬(reorganization)
⚠️ 면접관이 보는 것: "정렬돼 있으니 삽입도 쉽겠다"는 흔한 착각입니다. 정렬을 유지하려면 삽입마다 뒤 레코드를 물리적으로 이동시켜야 하고, 이 비용을 피하려고 overflow area와 주기적 재구성(reorganization)을 쓴다는 것까지 설명해야 합니다.

2-4. Indexed File — 별도의 지도(index)를 두기

데이터 파일 자체는 어떤 순서든 상관없이 두고, 대신 키 → 레코드 위치를 매핑하는 작은 별도 구조(index)를 만들어 검색을 가속하는 방식입니다. 인덱스는 데이터 파일보다 훨씬 작아서(키 + 포인터만 저장) 메모리에 올리기 쉽고, 인덱스에서 위치를 찾은 뒤 데이터 파일에 한 번만 접근하면 됩니다.

[ Indexed File ] Index (key, block 주소만 저장 — 작고 빠름) ┌───────────────┐ │ id=3 → B0 │ │ id=44 → B1 │ │ id=91 → B2 │ └───────────────┘ │ 인덱스에서 위치를 찾은 뒤 딱 한 번 접근 ▼ Data File (블록 단위 저장, 순서는 자유) Block 0 Block 1 Block 2 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ id=3 ... │ │ id=44... │ │ id=91... │ └──────────┘ └──────────┘ └──────────┘ 조회: 인덱스 탐색(대부분 메모리) + 데이터 block 1회 I/O → 매우 빠름 삽입: 데이터는 아무 데나 추가 가능하지만, 인덱스에도 항목을 추가/재정렬해야 함 → 인덱스 유지(maintenance) 비용이 발생

인덱스 자체도 결국 정렬된 파일이거나 트리 구조입니다. 이 인덱스를 dense/sparse로 나누고, 트리로 발전시킨 것이 Day 3~4에서 다룰 B-Tree/B+Tree입니다. 여기서는 "인덱스는 공짜가 아니라 유지 비용이 있는 또 하나의 자료구조"라는 점이 핵심입니다.

2-5. Hashed File (Direct File) — 계산으로 바로 위치 찾기

키를 해시 함수에 넣어 어느 block(bucket)에 저장될지 직접 계산하는 방식입니다. 탐색이 필요 없고, 계산만으로 위치를 알아내므로 평균적으로 O(1) I/O입니다.

[ Hashed File ] hash(id) mod 4 = bucket 번호 Bucket 0 Bucket 1 Bucket 2 Bucket 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ id=44 │ │ id=57 │ │ id=3 │ │ id=91 │ │ id=12 │ │ │ │ id=200 │ │ │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ ↑ collision(충돌): id=3과 id=200이 같은 bucket으로 계산됨 → overflow chain 또는 open addressing으로 처리 조회(id=X): hash(X) 계산 → 해당 bucket 1회 I/O → O(1) 평균 range(id 12~91): 해시값은 원래 키의 순서를 보존하지 않음 → "12보다 크고 91보다 작은 모든 id"를 찾으려면 전체 스캔 필요 → range query 불가

해시는 동등 비교(equality lookup, "id = 57")에는 이론상 최고 성능이지만, 범위 검색(range query, "id BETWEEN 12 AND 91")에는 근본적으로 쓸 수 없습니다. 키의 순서 정보를 해시 값이 보존하지 않기 때문입니다. 이것이 dense index/B+Tree와의 결정적 차이입니다.

왜 면접에서 이 질문을 묻는가

이 질문은 자료구조 이름을 나열할 수 있는지가 아니라, 워크로드를 분석해서 구조를 고르는 판단 과정을 봅니다.

  1. 접근 패턴을 먼저 묻는 습관이 있는가
    "인덱스 걸면 되죠"가 아니라, 조회/삽입/삭제/range 비율을 먼저 확인하는 사고를 하는지 봅니다.
  2. 쓰기 비용을 무시하지 않는가
    읽기만 빨라지는 구조는 없습니다. 인덱스, 정렬 유지에는 항상 쓰기 비용이 따른다는 것을 아는지 확인합니다.
  3. 실제 시스템 설계 경험과 연결되는가
    로그 적재(heap 유사), 마스터 테이블 조회(indexed), 세션/캐시(hashed) 같은 실무 사례를 즉시 연결할 수 있어야 합니다.
⚠️ 주니어 수준 답변: "인덱스를 걸면 검색이 빨라집니다."
10년차 기대 답변: "조회/삽입 비율과 range query 필요 여부를 먼저 확인합니다. 쓰기가 압도적으로 많고 조회가 드물면 heap에 가까운 append 구조가 낫고, 조회가 많고 동등 비교 위주면 hash, 범위 검색이 필요하면 정렬 기반 인덱스를 씁니다."

실무 연결

실무 상황 적합한 구조와 이유
애플리케이션 로그 / 이벤트 적재 Heap에 가까운 append-only 쓰기. 조회는 드물거나 나중에 배치로 처리하므로 쓰기 비용을 최소화하는 것이 우선입니다. Kafka log segment가 대표적입니다.
게시판 / 상품 검색 제목, 카테고리 등 범위 검색·정렬·LIKE 검색이 필요합니다. 정렬 기반 인덱스(B+Tree)가 필요하고, 해시 인덱스만으로는 부족합니다.
세션 스토어, 캐시 (Redis) 거의 항상 key로 동등 조회(GET session:1234)만 하고 range query가 없습니다. 해시 기반 구조(hash table)가 이론적으로 최적입니다.
파티션 키 설계 (DynamoDB, Kafka) 파티션 키는 해시로 분산시켜 균등 분배하지만, 정렬/범위 조회가 필요한 부분은 별도의 정렬 키(sort key)를 둡니다. 이것이 hash와 sequential 구조를 함께 쓰는 하이브리드 설계입니다.
대량 데이터 마이그레이션 / 초기 적재(bulk load) 인덱스를 미리 걸어두고 삽입하면 삽입마다 인덱스 유지 비용이 발생해 매우 느립니다. 인덱스를 제거한 채 heap/sequential로 대량 적재한 뒤, 마지막에 인덱스를 한 번에 재생성합니다.

답변 사고 순서 — 면접에서 이렇게 생각하면서 말한다

  1. 접근 패턴 확인
    조회 대 삽입 대 삭제 비율은? 동등 조회(=)인가, 범위 조회(BETWEEN, >, ORDER BY)인가? 쓰기가 몰아서 발생하는가(batch), 지속적으로 발생하는가?
  2. 저장장치 비용 모델 재확인 (Day 1 프레임워크 적용)
    어떤 구조든 결국 I/O 횟수로 평가합니다. 정렬 유지, 인덱스 갱신도 결국 추가 I/O라는 것을 전제로 둡니다.
  3. 후보 구조 비교
    heap(쓰기 최적) / sequential(range에 강함, 삽입 비쌈) / indexed(균형, 유지비용 있음) / hash(동등 조회 최강, range 불가) 네 가지를 조회·삽입·삭제·range 축으로 비교합니다.
  4. I/O 비용 추정
    예상 레코드 수, block당 레코드 수를 근거로 대략적인 I/O 횟수를 어림잡아 말합니다 (예: "정렬 파일이면 이진 탐색으로 log₂(N/B) 정도").
  5. 운영 trade-off 설명
    선택한 구조의 유지보수 비용(재구성 주기, 인덱스 리빌드, collision 처리)까지 언급합니다.

핵심 답변 예시

짧은 버전 (1~2분 답변)

강한 답변 (Short)

먼저 접근 패턴을 봅니다. 쓰기가 압도적으로 많고 조회가 거의 없다면 heap처럼 그냥 append하는 구조가 가장 저렴합니다. 조회가 잦고 동등 비교 위주라면 해시 구조가 이론상 최고의 조회 성능을 냅니다. 하지만 범위 검색이나 정렬된 출력이 필요하다면 해시는 쓸 수 없고, 정렬 기반 인덱스 파일을 써야 합니다.

실무에서는 거의 항상 순수 heap이나 순수 sequential file을 그대로 쓰지 않고, 정렬된 데이터 파일 위에 인덱스를 얹거나, B+Tree 인덱스로 조회와 range를 함께 지원하는 쪽을 선택합니다. 다만 인덱스는 공짜가 아니라서, 삽입마다 인덱스도 갱신해야 하는 쓰기 비용이 따른다는 점을 함께 고려합니다.

깊이 있는 버전 (3~5분 답변)

강한 답변 (Deep)

네 가지 조직 방식을 조회/삽입/삭제/range 네 축으로 비교해서 판단합니다. Heap file은 삽입이 항상 파일 끝에 순차 쓰기라 O(1) I/O로 가장 저렴하지만, 조회는 위치 정보가 없어 전체 스캔(O(N/B))이 필요합니다. 그래서 조회가 드문 로그·이벤트 적재에 적합합니다.

Sequential file은 정렬되어 있어 이진 탐색으로 O(log₂(N/B)) 조회가 가능하고, range query는 시작점만 찾으면 나머지는 순차 스캔이라 매우 저렴합니다. 문제는 삽입입니다. 정렬을 유지하려면 뒤 레코드를 밀어야 하므로 실무에서는 이 비용을 감추기 위해 별도의 overflow area에 신규 레코드를 먼저 쌓고, 주기적으로 main file과 병합해 재정렬(reorganization)합니다. 이 재구성 주기 자체가 운영 비용입니다.

Indexed file은 데이터 파일과 별도로 (key → 위치) 매핑만 담은 작은 인덱스를 둡니다. 인덱스는 데이터보다 훨씬 작아 메모리에 상주하기 쉽고, 인덱스 탐색 후 데이터에 1회만 접근하면 됩니다. 하지만 인덱스도 자료구조이므로 삽입/삭제 때마다 유지 비용이 발생하고, 이 인덱스를 정렬 배열로 둘지 B-Tree로 둘지에 따라 유지 비용과 조회 성능이 또 달라집니다.

Hashed file은 해시 함수로 위치를 직접 계산하기 때문에 동등 조회에서 평균 O(1) I/O로 최고 성능을 냅니다. 하지만 해시 값은 원래 키의 순서를 보존하지 않으므로, "특정 범위의 레코드를 모두 찾아라"는 요청에는 전체 스캔밖에 방법이 없습니다. 그래서 세션 스토어나 캐시처럼 동등 조회만 있는 워크로드에는 최적이지만, 게시판 검색이나 정렬된 리포트가 필요한 테이블에는 부적합합니다.

결론적으로 저는 조회 패턴이 range를 포함하면 정렬 기반 인덱스(B+Tree)를, 순수 동등 조회 + 초고속 캐시성 접근이면 해시를, 쓰기가 압도적이고 조회가 드물면 append-only 구조를 우선 검토합니다. 실무에서는 종종 두 가지를 함께 씁니다. 예를 들어 DynamoDB 파티션 키는 해시로 분산하고 정렬 키(sort key)는 B-Tree 유사 구조로 범위 조회를 지원합니다.

약한 답변

"인덱스를 걸면 검색이 빨라지니까 다 인덱스를 겁니다. 해시는 O(1)이니까 제일 빠르고요."

왜 약한가: 삽입/삭제 비용을 전혀 고려하지 않았습니다. range query가 불가능한 해시의 한계를 모릅니다. "빠르다"는 결론만 있고, 조회/삽입/삭제/range라는 다차원 trade-off로 구조를 비교하는 사고 과정이 없습니다. 모든 상황에 하나의 정답(인덱스, 또는 해시)을 대입하는 것 자체가 위험 신호입니다.

⇄

Trade-off 정리

조직 방식 조회(동등) Range Query 삽입 삭제 공간 오버헤드
Heap O(N/B) 전체 스캔 불가 (스캔 필요) O(1) 매우 저렴 tombstone 또는 이동 없음
Sequential O(log₂(N/B)) 이진탐색 매우 저렴 (순차 스캔) 비쌈 (재정렬/overflow) 비쌈 (shift 또는 표시) overflow area 필요
Indexed 매우 빠름 (인덱스 + 1회 접근) 인덱스 구조에 따라 다름 보통 (인덱스 갱신 비용) 보통 (인덱스 갱신 비용) 인덱스 저장 공간 추가
Hashed 평균 O(1) 최고 불가 (순서 보존 안 됨) 평균 O(1), collision 시 비쌈 평균 O(1), collision 정리 필요 bucket 여유 공간 필요
💡 네 구조 모두 "완벽한 구조"는 없습니다. 조회를 빠르게 하면 삽입이 느려지고, 삽입을 빠르게 하면 조회가 느려집니다. 이 trade-off를 명시적으로 말하는 것 자체가 10년차 답변의 핵심입니다.
✗

흔한 오해와 바로잡기

오해 1: "인덱스를 많이 걸수록 무조건 좋다"

인덱스는 조회를 빠르게 하는 대신, 모든 INSERT/UPDATE/DELETE마다 인덱스도 함께 갱신해야 합니다. 인덱스가 많을수록 쓰기 성능이 떨어지고 저장 공간도 늘어납니다. 실무에서는 실제 쿼리 패턴을 보고 필요한 인덱스만 만듭니다. 미신

오해 2: "정렬된 파일이니 삽입도 빠를 것이다"

정렬을 물리적으로 유지하려면 삽입 위치 뒤의 레코드를 전부 밀어야 합니다. 실무 시스템은 이 비용을 피하려고 overflow area, 주기적 reorganization, 또는 애초에 트리 기반 구조(B+Tree)를 써서 삽입 비용을 O(log N)으로 낮춥니다. 미신

오해 3: "해시가 O(1)이니 항상 B+Tree보다 빠르다"

동등 조회 단일 케이스만 보면 맞을 수 있지만, 해시는 순서 정보를 버리기 때문에 range query, ORDER BY, 정렬된 결과가 필요한 순간 전체 스캔으로 퇴화합니다. DB 인덱스가 기본으로 B+Tree인 이유는 동등 조회와 range 조회를 모두 균형 있게 지원하기 때문입니다. 미신

오해 4: "heap file은 정렬도 인덱스도 없으니 실무에서 쓸모없다"

오히려 로그 적재, 이벤트 스트림, 대량 bulk insert의 staging 영역에서는 heap에 가까운 append-only 구조가 최선입니다. 조회가 필요 없거나 나중에 배치로 처리한다면, 정렬/인덱스 유지 비용 자체가 낭비입니다. 미신

Senior-Level Follow-up 질문과 답변 방향

"정렬 순서를 유지하면서 삽입 비용을 줄이는 방법이 있습니까?"

답변 방향: 두 가지 접근이 있습니다. 첫째는 overflow area를 두고 신규 레코드를 임시로 미정렬 상태로 쌓았다가 주기적으로 main file과 병합·재정렬하는 것입니다(재구성 비용은 배치로 상환). 둘째는 애초에 배열이 아닌 트리 구조(B-Tree/B+Tree)를 쓰는 것입니다. 트리는 삽입할 때 리프 노드 재분배(split)만으로 정렬 순서를 O(log N)에 유지할 수 있어 배열식 정렬 파일의 O(N) 삽입 비용을 근본적으로 피합니다. (Day 4에서 자세히 다룹니다.)

"해시 파일에서 collision은 어떻게 처리합니까?"

답변 방향: 대표적으로 chaining(같은 bucket 안에 overflow list 연결)과 open addressing(다른 빈 slot을 탐사)이 있습니다. 디스크 기반 해시 파일에서는 보통 overflow bucket을 체인으로 연결하는 방식을 씁니다. 더 나아가 bucket이 가득 찼을 때 전체를 rehash하는 비용을 피하기 위해 extendible hashing이나 linear hashing처럼 점진적으로 bucket을 늘리는 기법을 씁니다. (Day 5에서 다룹니다.)

"인덱스 파일 자체도 파일인데, 그 인덱스는 어떻게 검색합니까?"

답변 방향: 인덱스가 작으면 메모리에 올려 선형/이진 탐색으로 충분하지만, 인덱스 자체가 커지면 인덱스도 여러 block에 걸쳐 저장되고, 그 인덱스를 찾기 위한 또 다른(더 작은) 인덱스가 필요해집니다. 이렇게 인덱스를 계층적으로 쌓아 올린 것이 multi-level index이고, 이를 자연스럽게 균형 트리로 일반화한 것이 B-Tree/B+Tree입니다.

"실시간 삽입이 매우 잦은 시스템에서 인덱스 구조를 어떻게 유지합니까?"

답변 방향: 매 삽입마다 인덱스를 즉시 갱신하는 대신, 쓰기를 메모리 버퍼(memtable)에 먼저 모았다가 일정 크기가 되면 정렬된 파일로 순차 flush하고, 백그라운드에서 파일들을 병합(compaction)하는 LSM-Tree 방식이 있습니다. 랜덤 쓰기를 순차 쓰기로 바꿔 쓰기 처리량을 크게 높입니다. RocksDB, Cassandra, LevelDB가 이 방식을 씁니다. (Day 9 append-only 구조에서 더 다룹니다.)
