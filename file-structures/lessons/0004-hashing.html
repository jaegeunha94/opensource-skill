Day 4: 직접 파일 / 해싱

학습 목표 이 레슨을 마치면: (1) 직접 파일(Direct File)의 개념과 인덱스 기반 파일과의 차이를 설명할 수 있다. (2) 해시 함수의 역할과 나머지 연산·접기·중간-제곱 기법을 이해한다. (3) 충돌(Collision) 발생 원인과 체이닝·오픈 어드레싱 두 가지 해결 전략을 비교할 수 있다. (4) 버킷 개념과 적재율(Load Factor)이 성능에 미치는 영향을 계산하고 설명할 수 있다.

1. 직접 파일이란?

Day 1~3에서 배운 순차 파일과 인덱스 파일은 모두 키 값을 비교해서 레코드를 찾는다. 순차 파일은 처음부터 순서대로 훑고, 인덱스 파일은 인덱스를 이진 탐색한 뒤 블록을 읽는다.

직접 파일(Direct File)은 다른 접근법을 택한다. 키 값을 수학 함수(해시 함수)에 넣어 저장 위치(주소)를 직접 계산하고, 그 주소로 바로 이동한다. 이진 탐색도, 인덱스 순회도 필요 없다.

키 값 → [해시 함수] → 버킷 주소 → 직접 접근 예시: 학번 20301 → h(20301) = 20301 % 100 = 1 → 버킷 #1로 바로 이동 학번 20455 → h(20455) = 20455 % 100 = 55 → 버킷 #55로 바로 이동 학번 20801 → h(20801) = 20801 % 100 = 1 → 버킷 #1 (충돌!) 순차/인덱스 파일: O(log B) ~ O(B) 블록 접근 직접 파일 (이상적): O(1) — 한 번의 블록 접근
직접 파일의 핵심 트레이드오프 검색이 O(1)에 가깝지만 해시 충돌이라는 문제가 발생하며, 키 공간(가능한 키 값 범위)에 비해 실제 사용 주소 공간이 낭비될 수 있다. 또한 범위 검색(예: 학번 20000~20500 전체)에는 불리하다.

2. 해시 함수

해시 함수(Hash Function)는 임의의 키 값을 받아 0 이상 B 미만(B = 버킷 수)의 정수 주소를 반환하는 함수다.

2-1. 나머지 연산 (Division / Modulo)

가장 단순하고 널리 쓰이는 방법. h(k) = k mod M (M = 버킷 수)

M = 7 (버킷 수) h(23) = 23 mod 7 = 2 → 버킷 #2 h(47) = 47 mod 7 = 5 → 버킷 #5 h(100) = 100 mod 7 = 2 → 버킷 #2 (충돌) h(7) = 7 mod 7 = 0 → 버킷 #0
M 선택 팁 M을 소수(Prime number)로 선택하면 키 값이 특정 패턴을 가질 때 충돌이 분산된다. 예를 들어 키가 짝수만 있을 때 M=10이면 짝수 버킷만 사용되지만, M=7이면 모든 버킷이 고르게 사용된다.

2-2. 접기 (Folding)

키를 여러 부분으로 나눈 뒤 합산(또는 XOR)하여 주소를 만든다. 키가 긴 문자열이거나 자릿수가 많은 숫자일 때 유용하다.

키 = 987654321 (9자리), 버킷 수 = 1000 (주소 3자리) 3자리씩 분할: 987 | 654 | 321 합산: 987 + 654 + 321 = 1962 mod 1000: 1962 mod 1000 = 962 → 버킷 #962

2-3. 중간-제곱법 (Mid-Square)

키를 제곱한 뒤 중간 자리를 꺼내 주소로 사용한다. 키의 모든 자릿수가 결과에 영향을 주어 분산이 좋다.

키 = 1234, 버킷 수 = 100 (주소 2자리) 1234² = 1,522,756 중간 2자리 추출: 1-52-2756 → 52 주소 = 52 → 버킷 #52

3. 버킷 (Bucket)

해시 파일의 저장 단위를 버킷(Bucket)이라고 한다. 버킷은 하나 이상의 레코드를 담을 수 있는 고정 크기 슬롯이다. 보통 버킷 하나 = 디스크 블록 하나에 대응한다.

해시 파일 (M=5 버킷, 버킷당 최대 2개 레코드) 버킷 #0: [ 학번=20005 ] [ 학번=20010 ] ← 만원 버킷 #1: [ 학번=20001 ] [ 비어있음 ] 버킷 #2: [ 학번=20002 ] [ 학번=20007 ] ← 만원 버킷 #3: [ 비어있음 ] [ 비어있음 ] 버킷 #4: [ 학번=20004 ] [ 비어있음 ] h(20005) = 20005 mod 5 = 0 → 버킷 #0 ✓ h(20010) = 20010 mod 5 = 0 → 버킷 #0 ✓ (같은 버킷, 충돌 없음) h(20015) = 20015 mod 5 = 0 → 버킷 #0 ✗ (오버플로우 발생!)
용어 정리
  • 버킷 용량(Bucket Capacity): 버킷 하나에 들어갈 수 있는 최대 레코드 수
  • 오버플로우(Overflow): 버킷이 꽉 찼는데 새 레코드가 같은 주소로 해싱될 때
  • 충돌(Collision): 두 개 이상의 키가 같은 버킷 주소를 가질 때 (오버플로우의 원인)

4. 적재율 (Load Factor)

적재율(Load Factor, α)은 해시 파일이 얼마나 채워져 있는지를 나타내는 비율이다.

α = (저장된 레코드 수) / (전체 버킷 용량) = n / (M × 버킷 용량) 예: 레코드 70개, 버킷 10개, 버킷당 10칸 α = 70 / (10 × 10) = 70 / 100 = 0.70 (70%)
적재율 범위 충돌 빈도 공간 낭비 권장 여부
α < 0.5 낮음 큼 (50% 이상 비어있음) 공간 낭비 심함
0.7 ≤ α ≤ 0.8 적절 적절 권장 범위
α > 0.9 매우 높음 낮음 성능 저하 심각
실무에서의 적재율 대부분의 해시 테이블 구현체는 α가 0.75를 넘으면 자동으로 리해싱(Rehashing)을 수행한다. 리해싱은 버킷 수를 2배로 늘리고 모든 레코드를 재배치하는 비용이 크지만, 충돌율을 낮춰 이후 성능을 보장한다.

5. 충돌 해결 전략

충돌은 불가피하다. 두 가지 주요 전략이 있다: 체이닝과 오픈 어드레싱.

5-1. 체이닝 (Chaining / Separate Chaining)

각 버킷에 연결 리스트(Linked List)를 두고, 오버플로우가 발생하면 오버플로우 영역에 레코드를 추가하여 체인으로 연결한다.

버킷 #0: [ REC-A ] → [ REC-E ] → [ REC-I ] → null (메인 영역) (오버플로우 블록 1) (오버플로우 블록 2) 버킷 #1: [ REC-B ] → null 버킷 #2: [ REC-C ] → [ REC-F ] → null 버킷 #3: [ 비어 ] → null 버킷 #4: [ REC-D ] → null 검색(REC-I): h(I) = 0 → 버킷#0 읽기 → 없음 → 체인 따라가기 → 블록1 읽기 → 없음 → 블록2 읽기 → REC-I 발견! (블록 접근 3회)
체이닝 특징
  • 삭제가 비교적 간단 (체인에서 해당 노드 제거)
  • 오버플로우 영역이 길어질수록 성능 저하 (체인 길이 비례)
  • 파일 기반 해싱에서는 오버플로우 블록이 별도 영역에 할당됨
  • 오버플로우 블록을 주기적으로 재구성(Reorganization)해야 성능 유지

5-2. 오픈 어드레싱 (Open Addressing)

오버플로우가 발생하면 다음 빈 버킷을 찾아 그 자리에 저장한다. 체인 포인터가 필요 없으므로 메모리/포인터 오버헤드가 없다.

선형 탐사 (Linear Probing)

M = 7, h(k) = k mod 7 삽입 순서: 15, 22, 29 → h(15)=1, h(22)=1(충돌!), h(29)=1(충돌!) [0] 비어있음 [1] 15 ← h(15)=1, 저장 [2] 22 ← h(22)=1 충돌 → 다음 빈 칸 [2]에 저장 [3] 29 ← h(29)=1 충돌 → [2] 충돌 → 다음 빈 칸 [3]에 저장 [4] 비어있음 [5] 비어있음 [6] 비어있음 검색(29): h(29)=1 → [1]은 15 (≠29) → [2]는 22 (≠29) → [3]은 29 ✓ (3번 비교)
클러스터링 문제 (Primary Clustering) 선형 탐사는 같은 해시 주소의 키들이 인접 슬롯에 몰려 클러스터(덩어리)를 형성한다. 클러스터가 커질수록 탐사 횟수가 급격히 증가한다. 이를 해결하기 위해 이차 탐사(Quadratic Probing)나 이중 해싱(Double Hashing)을 사용한다.

이차 탐사 (Quadratic Probing)

충돌 시 탐사 간격을 1, 4, 9, 16... (i²) 씩 늘려가며 빈 자리를 찾는다. h(k)=1 충돌 → [1+1²] = [2] 시도 → 충돌 → [1+2²] = [5] 시도 → 성공 선형 탐사보다 클러스터링이 덜 발생하지만, 이차 클러스터링(Secondary Clustering)은 남는다.

이중 해싱 (Double Hashing)

두 번째 해시 함수 h₂(k)를 탐사 간격으로 사용한다. h₁(k) = k mod M (기본 주소) h₂(k) = 1 + (k mod (M-2)) (탐사 간격, 항상 양수) 충돌 시: (h₁(k) + i × h₂(k)) mod M (i = 1, 2, 3, ...) k=22, M=7: h₁(22)=1, h₂(22)=1+(22 mod 5)=1+2=3 충돌 시 탐사: [1+3] = [4], [1+6] = [0], ... → 가장 균일한 분산, 클러스터링 최소화

6. 체이닝 vs 오픈 어드레싱 비교

비교 항목 체이닝 (Chaining) 오픈 어드레싱
구조 메인 버킷 + 오버플로우 체인 단일 배열/파일 내 빈 슬롯 탐사
공간 효율 포인터 추가 오버헤드 있음 포인터 불필요 (공간 절약)
삭제 처리 간단 — 체인에서 노드 제거 복잡 — Tombstone 마킹 필요

성능 최악 모든 키가 같은 버킷 → 체인 탐색 O(n) 모든 슬롯 찾아야 → O(n)
실제 파일 시스템 오버플로우 블록 체인 일반적 메모리 해시 테이블에서 주로 사용
