Day 2: 순차 파일 구조

📅 2026-06-22 ⏱ 약 30분 📚 파일구조 과목 🔢 Lesson 2 / 10
학습 목표 이 레슨을 마치면: (1) 순차 파일과 직렬 파일의 차이를 설명할 수 있다. (2) 고정/가변 길이 레코드에서 순차 파일이 어떻게 저장되는지 그릴 수 있다. (3) 순차 파일의 삽입·삭제 연산 비용을 분석할 수 있다. (4) 순차 파일이 적합한 상황과 부적합한 상황을 판단할 수 있다.

1. 순차 파일이란?

순차 파일(Sequential File)은 레코드를 특정 키(Key) 순서대로 저장하는 파일 조직 방법이다. 새로운 레코드는 키 순서를 유지하는 위치에만 삽입된다.

이와 유사한 개념으로 직렬 파일(Serial File)이 있는데, 둘의 차이는 다음과 같다.

구분직렬 파일 (Serial File)순차 파일 (Sequential File)
저장 순서 입력된 순서대로 저장 (정렬 없음) 키(Key) 값 기준으로 정렬하여 저장
검색 처음부터 끝까지 선형 탐색 이진 탐색 가능 (블록 단위)
삽입 빠름 — 파일 끝에 추가 느림 — 정렬 위치 찾아 삽입
활용 예시 로그 파일, 임시 데이터 급여 파일, 학적 파일
혼동 주의 "순차"라는 말 때문에 직렬 파일과 혼동하기 쉽다. 순차 파일의 핵심은 정렬된 순서 유지다. 정렬 없이 순서대로만 저장하면 직렬 파일이다.

2. 순차 파일의 물리적 구조

순차 파일은 디스크에서 블록(Block) 단위로 저장된다. 각 블록은 여러 개의 레코드를 담는다.

고정 길이 레코드 순차 파일

블록 1 (512 bytes) 블록 2 (512 bytes) ┌──────────────────────┐ ┌──────────────────────┐ │ REC#01 | key=1003 │ │ REC#06 | key=2010 │ │ REC#02 | key=1027 │ │ REC#07 | key=2055 │ │ REC#03 | key=1089 │ │ REC#08 | key=2099 │ │ REC#04 | key=1145 │ │ REC#09 | key=2150 │ │ REC#05 | key=1902 │ │ REC#10 | key=2200 │ └──────────────────────┘ └──────────────────────┘ ↓ 오름차순 정렬 유지 ↓

고정 길이 레코드에서 n번째 레코드의 위치는 수식으로 바로 계산할 수 있다: 

위치(바이트) = (레코드번호 - 1) × 레코드크기

단, 실제 접근 단위는 블록이므로:

블록번호 = ⌊(레코드번호 - 1) × 레코드크기 / 블록크기⌋

가변 길이 레코드 순차 파일

레코드 크기가 다를 때는 위치를 공식으로 계산할 수 없다. 대신 두 가지 방식 중 하나를 사용한다.

방법 1 — 구분자(Delimiter) 방식 ┌─────────────────────────────────────────────────────┐ │ 김철수|23|010-1111-2222 § 이영희|25|010-3333-4444 § … │ └─────────────────────────────────────────────────────┘ ↑ § : 레코드 구분자 방법 2 — 길이 필드(Length Field) 방식 ┌────────────────────────────────────────────────────┐ │ [18] 김철수|23|010 [20] 이영희|25|010-333 [17] … │ └────────────────────────────────────────────────────┘ ↑ 앞에 길이(bytes)를 먼저 기록
길이 필드 방식의 장점 구분자 방식은 데이터 안에 구분자 문자가 포함되면 오류가 발생한다. 길이 필드 방식은 이 문제가 없어 실제 파일 시스템에서 더 많이 쓰인다.

3. 순차 파일의 연산

3-1. 검색 (Search)

순차 파일은 정렬되어 있으므로 이진 탐색(Binary Search)을 적용할 수 있다.

탐색 방법비교 횟수적용 조건
선형 탐색 평균 n/2, 최대 n 정렬 불필요 (직렬 파일도 가능)
이진 탐색 최대 ⌈log₂ n⌉ 정렬된 파일 필수

단, 디스크에서의 이진 탐색은 메모리 이진 탐색과 다르다. 레코드 단위가 아니라 블록 단위로 접근하므로 실제 탐색 비용은 블록 수 기준으로 계산한다.

블록 수 기준 이진 탐색 비용 총 레코드 수 = n, 블록당 레코드 수 = b 이면:
총 블록 수 B = ⌈n / b⌉
이진 탐색 블록 접근 횟수 ≈ ⌈log₂ B⌉

3-2. 삽입 (Insert)

순차 파일에 새 레코드를 삽입하려면 정렬 순서를 유지해야 한다. 삽입 위치를 찾은 뒤, 그 이후의 레코드를 모두 한 칸씩 뒤로 밀어야 한다.

삽입 전: key = 1003, 1027, 1089, 1145 ↑ 여기에 key=1060 삽입 삽입 후: key = 1003, 1027, 1060, 1089, 1145 ↑ 새 레코드 ↑↑ 이 레코드들이 한 칸씩 뒤로 이동

최악의 경우 파일 전체 레코드를 이동해야 하므로 삽입 비용은 O(n)이다. 이 때문에 빈번한 삽입이 있는 환경에서는 순차 파일이 부적합하다.

3-3. 삭제 (Delete)

삭제에는 두 가지 전략이 있다.

  • 즉시 삭제: 레코드를 삭제하고 이후 레코드를 앞으로 이동. 역시 O(n) 비용. 파일이 항상 빈틈 없이 유지된다.
  • 논리적 삭제 (Deletion Mark): 삭제 플래그만 세우고 나중에 일괄 정리. 삭제 자체는 O(1)이지만, 파일에 "죽은" 레코드가 남아 낭비가 발생한다. 주기적인 파일 재구성(Reorganization)이 필요하다.
논리적 삭제 예시 (✗ = 삭제 마크) ┌──────┬──────┬──────┬──────┬──────┐ │ 1003 │ 1027✗│ 1060 │ 1089 │ 1145 │ └──────┴──────┴──────┴──────┴──────┘ ↑ 삭제 마크만 있고 공간은 그대로 파일 재구성 후: ┌──────┬──────┬──────┬──────┐ │ 1003 │ 1060 │ 1089 │ 1145 │ └──────┴──────┴──────┴──────┘

3-4. 수정 (Update)

키가 아닌 일반 필드의 수정은 해당 레코드를 찾아 내용을 바꾸면 된다. 고정 길이 레코드라면 같은 위치에 덮어쓸 수 있다. 그러나 키 값을 수정하면 정렬 순서가 깨지므로, 기존 레코드를 삭제하고 새 키로 다시 삽입하는 두 단계를 거쳐야 한다.

4. 일괄 처리와 오버플로우 영역

순차 파일의 최적 활용: 일괄 처리 (Batch Processing)

순차 파일은 전체 파일을 처음부터 끝까지 읽는 일괄 처리에 매우 효율적이다. 예: 월급 계산, 청구서 발행, 백업.

일괄 처리 시 트랜잭션 파일(변경 사항)을 마스터 파일(원본)과 합치는 파일 합병(File Merging) 기법을 사용한다. 두 파일이 같은 키로 정렬되어 있으면 한 번의 순차 읽기로 처리할 수 있다.

오버플로우 영역 (Overflow Area)

삽입마다 레코드를 이동하는 비용을 줄이기 위해 오버플로우 영역을 예약하는 방법이 있다. 새 레코드는 일단 오버플로우 영역에 저장하고, 나중에 일괄 재구성한다.

기본 영역 (정렬된 레코드) 오버플로우 영역 ┌────┬────┬────┬────┐ ┌────┬────┐ │1003│1060│1089│1145│ ←링크→ │1027│1075│ └────┴────┴────┴────┘ └────┴────┘ 오버플로우가 쌓이면 재구성 필요
📖 Primary Source Folk, Zoellick & Riccardi, File Structures — Chapter 3: Secondary Storage and System Software. 블록 I/O와 버퍼링, 그리고 순차 파일이 디스크에서 어떻게 배치되는지 설명한다. 특히 “buffering strategies” 절을 중점적으로 읽어라.

5. 순차 파일 성능 요약

연산비용 (레코드 수 n 기준)비고
순차 읽기 (전체) O(n) 일괄 처리에 최적
이진 탐색 (단건) O(log n) 정렬된 경우에만 가능
삽입 O(n) 정렬 위치 이후 레코드 이동
삭제 (즉시) O(n) 이후 레코드 앞으로 이동
삭제 (논리적) O(log n) 탐색 후 플래그만 설정
수정 (비키 필드) O(log n) 탐색 후 덮어쓰기
순차 파일이 적합한 경우
  • 전체 레코드를 주기적으로 처리하는 일괄 업무 (급여, 청구서)
  • 순서대로 생성·소비되는 로그 데이터
  • 삽입/삭제가 드물고 순차 읽기가 대부분인 경우
순차 파일이 부적합한 경우
  • 특정 레코드를 즉시 찾아야 하는 온라인 트랜잭션 처리 (OLTP)
  • 삽입·삭제가 빈번한 동적 데이터
  • 임의 접근(Random Access)이 주된 접근 패턴인 경우
