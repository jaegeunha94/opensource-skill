Interview Day 9

대용량 로그/이벤트 저장 — Append-only 구조, WAL, LSM-Tree

"수정을 허용하지 않고 끝에만 쓴다"는 제약이 오히려 쓰기 성능·복구·동시성을 어떻게 단순화하는지, 그리고 그 대가로 무엇을 잃는지

Q

면접 질문

"Append-only 파일 구조는 어떤 장점과 단점이 있습니까?"

변형 질문 유형

  • "DB는 왜 데이터를 바로 갱신하지 않고 WAL(Write-Ahead Log)에 먼저 기록합니까?"
  • "LSM-Tree는 B+Tree와 비교해 쓰기 성능이 왜 더 좋다고 알려져 있습니까? 대신 무엇을 희생합니까?"
  • "Append-only 구조에서 삭제(delete)나 갱신(update)은 실제로 어떻게 처리됩니까?"
  • "Kafka 같은 메시지 로그는 왜 append-only 구조를 쓰고, 그게 대용량 이벤트 처리에 왜 유리합니까?"
📖

전제 개념 — 쉬운 한국어로 먼저 이해하기

9-1. "제자리 수정(in-place update)"이 디스크에서 왜 비싼가

Day 1에서 다뤘듯 디스크(특히 HDD)는 순차 접근이 랜덤 접근보다 훨씬 빠릅니다. 전통적인 파일이나 B+Tree 기반 저장 구조는 "레코드를 바꾸려면 그 레코드가 있는 원래 위치를 다시 찾아가서 덮어쓴다"는 in-place update 방식을 씁니다. 문제는 갱신이 몰릴수록 디스크 여기저기를 랜덤하게 찾아가서 쓰는(random write) 패턴이 반복된다는 점입니다. HDD는 이때마다 헤드가 이동(seek)해야 하고, SSD도 랜덤 쓰기가 순차 쓰기보다 느리고 쓰기 증폭(write amplification)이 커지는 특성이 있습니다.

💡 Append-only의 핵심 아이디어: "기존 데이터를 절대 덮어쓰지 않고, 항상 파일의 맨 끝에만 새로 쓴다." 갱신도, 삭제도 전부 "새로운 레코드를 끝에 추가하는 것"으로 표현합니다. 그 결과 모든 쓰기가 순차 쓰기가 되어 디스크에 가장 유리한 접근 패턴만 사용하게 됩니다.

9-2. Append-only가 갱신/삭제를 표현하는 방법 — Tombstone과 최신 버전 우선

키 user:42의 값을 "서울"에서 "부산"으로 바꾼다고 해봅시다. In-place 구조라면 기존 레코드를 찾아 덮어쓰지만, append-only 구조는 같은 키로 새 값을 파일 끝에 하나 더 씁니다. 읽을 때는 "그 키로 기록된 것 중 가장 최근 것"을 유효한 값으로 취급합니다. 삭제는 값 대신 tombstone(묘비 마커)이라는 특별한 레코드를 끝에 추가해서 "이 키는 삭제됨"을 표시합니다.

[ Append-only 로그 예시 — 시간 순으로 쌓임 ] offset 0: PUT user:42 = "서울" offset 1: PUT user:7 = "대전" offset 2: PUT user:42 = "부산" <- user:42 갱신 (기존 값은 그대로 남아있음) offset 3: DEL user:7 <- user:7 삭제 (tombstone) offset 4: PUT user:42 = "인천" <- 다시 갱신 => user:42 조회 시: offset 4("인천")가 가장 최근 값이므로 이것이 유효값 => user:7 조회 시: offset 3에 tombstone이 있으므로 "삭제됨"으로 처리 => offset 0, 1, 2는 더 이상 유효하지 않지만 파일에는 그대로 남아 공간을 차지함
⚠️ 핵심 문제: 이렇게 쌓기만 하면 무효화된 옛날 레코드가 파일에 계속 쌓여 파일이 무한히 커지고, 같은 키를 여러 번 읽어 최신 값을 찾는 비용도 늘어납니다. 이것을 정리하는 과정이 compaction입니다.

9-3. Compaction — 쌓인 로그에서 "유효한 최신 값만" 남기기

Compaction은 append-only 로그(또는 여러 개의 로그 파일)를 읽어서, 같은 키에 대해 가장 최신 값만 남기고 나머지(무효화된 값, tombstone까지 포함해 충분히 오래된 것)를 제거한 새 파일을 만드는 백그라운드 작업입니다. 이 작업도 결과물을 새 파일로 순차 쓰기 때문에, compaction 자체도 random write를 유발하지 않습니다.

[ Compaction 전/후 ] Compaction 전 (원본 로그, 4개 레코드): user:42="서울", user:42="부산", user:7="대전", user:42="인천" Compaction 후 (같은 키는 최신 값만 남김, 2개 레코드): user:7="대전", user:42="인천" => 파일 크기 감소, 특정 키를 찾을 때 훑어야 할 레코드 수 감소

9-4. WAL(Write-Ahead Log) — "데이터를 바꾸기 전에, 바꾸겠다는 사실부터 순차 기록"

WAL은 append-only 아이디어를 장애 복구(durability)에 적용한 것입니다. DB가 실제 데이터 파일(보통 B+Tree 기반, in-place update)을 바로 수정하기 전에, "무엇을 바꿀 것인지"를 먼저 별도의 append-only 로그 파일에 순차적으로 기록합니다. 커밋은 "WAL에 순차 쓰기 + fsync가 끝났을 때" 완료된 것으로 간주하고, 실제 데이터 파일 반영(체크포인트)은 나중에 몰아서 처리해도 됩니다.

[ WAL을 사용하는 쓰기 흐름 ] 1. 트랜잭션: "계좌 A에서 1000원 차감" 요청 2. WAL에 순차 append: "계좌A -= 1000" 기록 + fsync 3. 클라이언트에 커밋 완료 응답 (이 시점에 durability 보장됨) 4. (나중에, 배치로) 실제 데이터 페이지(B+Tree 등)에 반영 = 체크포인트 정전 발생 시 복구: 마지막 체크포인트 이후의 WAL 레코드를 순서대로 재생(replay)하면 정전 직전 상태를 그대로 복원할 수 있음

즉 WAL은 "느린 랜덤 쓰기(실제 데이터 파일 반영)"를 커밋 경로에서 빼고, 커밋의 durability를 빠른 순차 쓰기(로그)만으로 보장하는 기법입니다. 이는 Day 8에서 다룬 저널링 파일 시스템의 저널과 정확히 같은 원리를 DBMS 레벨에 적용한 것입니다.

9-5. LSM-Tree — Append-only 쓰기와 인덱스를 결합한 저장 엔진 구조

LSM-Tree(Log-Structured Merge-Tree)는 "쓰기는 항상 순차적으로, 읽기는 인덱스로 빠르게"를 동시에 노리는 저장 엔진 설계입니다. B+Tree가 "쓰기 시점에 정렬된 구조를 바로 유지"하는 것과 달리, LSM-Tree는 쓰기를 메모리에 먼저 모았다가, 다 차면 정렬된 파일로 통째로 순차 디스크에 쓰고, 여러 개의 그런 파일을 나중에 백그라운드에서 병합(merge)합니다.

[ LSM-Tree 쓰기 경로 ] 1. 쓰기 요청 -> WAL에 순차 append (durability 확보) 2. 동시에 메모리 내 정렬 구조(memtable, 보통 skip list/균형트리)에 반영 3. memtable이 일정 크기를 넘으면 -> 정렬된 상태 그대로 디스크에 하나의 불변(immutable) 파일로 순차 쓰기 = SSTable 4. SSTable이 여러 개 쌓이면 -> 백그라운드 compaction이 여러 SSTable을 병합해 더 적은 수의, 최신 값만 남은 SSTable로 재구성 [ LSM-Tree 읽기 경로 ] 1. memtable 확인 (가장 최신) 2. 최근 SSTable부터 오래된 순서로 확인 3. 각 SSTable 탐색 전 Bloom filter로 "이 SSTable에 키가 없을 가능성"을 빠르게 걸러냄 4. 발견하면 그 값이 최신값 (더 최신 SSTable/memtable에 없었으므로)

핵심은 쓰기 경로에는 랜덤 디스크 쓰기가 전혀 없다는 것입니다. WAL도, SSTable 생성도 전부 순차 쓰기입니다. 대신 읽기는 memtable + 여러 SSTable을 순서대로 확인해야 할 수 있어 B+Tree 단일 구조보다 느려질 수 있고, 이를 Bloom filter와 compaction으로 완화합니다.

9-6. RUM Conjecture — Read, Update(Write), Memory/Space 세 마리 토끼를 동시에 잡을 수 없다

저장 구조를 비교할 때 시니어 면접에서 특히 유용한 프레임이 Read amplification, Write amplification, Space amplification 세 가지입니다.

  • Read amplification — 논리적으로 1개의 값을 찾기 위해 실제로 몇 개의 파일/블록을 읽어야 하는가 (LSM-Tree는 여러 SSTable을 뒤질 수 있어 이 값이 큼)
  • Write amplification — 논리적으로 1바이트를 쓰기 위해 실제로 디스크에 몇 바이트가 쓰이는가 (compaction이 반복되며 같은 데이터를 여러 번 다시 씀)
  • Space amplification — 논리적 데이터 크기 대비 실제 디스크 사용량 (무효화된 옛 버전이 compaction 전까지 공간을 차지)

B+Tree(in-place)는 read amplification이 낮고 space amplification도 낮지만 random write 비용이 크고, LSM-Tree는 write 경로가 순차적이라 쓰기는 빠르지만 compaction으로 인한 write amplification과 다중 SSTable 탐색으로 인한 read amplification을 대가로 치릅니다. 셋을 동시에 최소화할 수 없고, 워크로드에 맞춰 어느 것을 희생할지 선택하는 것이 저장 엔진 설계의 핵심입니다.

왜 면접에서 이 질문을 묻는가

"LSM-Tree라는 단어를 아는가"가 아니라, "쓰기 성능을 높이기 위해 어떤 것을 포기해야 하는지 스스로 추론하고, 그것이 실제 대용량 로그/이벤트/DB 시스템 설계에서 어떤 선택으로 이어지는지 설명할 수 있는가"를 봅니다. 이는 Day 1의 순차/랜덤 접근 비용 모델, Day 4의 B+Tree 구조 이해를 "쓰기가 매우 잦은 워크로드"라는 새로운 제약 조건에 적용할 수 있는지 확인하는 질문입니다.

  1. 왜 순차 쓰기만 허용하면 쓰기 성능이 좋아지는지 원리를 설명할 수 있는가
    random write를 제거해 디스크/SSD의 물리적 특성에 맞춘 설계라는 점을 근거로 들 수 있어야 합니다.
  2. "공짜 점심은 없다"를 이해하는가 — 무엇을 대가로 치르는지 아는가
    Read/Write/Space amplification 중 무엇을 얻고 무엇을 잃는지 설명할 수 있어야 합니다.
  3. WAL과 LSM-Tree가 같은 원리(순차 쓰기 우선)의 서로 다른 적용이라는 것을 연결할 수 있는가
    WAL은 durability, LSM-Tree는 저장 엔진 전체 구조라는 적용 범위 차이를 구분해야 합니다.
  4. 실제 시스템(Kafka, RocksDB, Cassandra, DB의 redo log)과 연결해 말할 수 있는가
    추상적 정의가 아니라 실무에서 마주치는 구체적 시스템의 설계 이유로 답할 수 있어야 합니다.
⚠️ 주니어 수준 답변: "append-only는 그냥 계속 뒤에 쓰는 거고, 로그 파일 같은 거예요."
10년차 기대 답변: "append-only는 random write를 없애 순차 쓰기 성능(HDD의 seek 제거, SSD의 쓰기 증폭 완화)을 얻는 대신, 무효화된 옛 버전이 계속 쌓여 공간과 읽기 비용이 늘어나는 문제가 생깁니다. 이를 compaction으로 주기적으로 정리하고, 그 과정에서 write amplification이라는 새로운 비용이 발생합니다. WAL은 이 아이디어를 durability에, LSM-Tree는 저장 엔진 전체 구조에 적용한 것이고, 쓰기가 매우 잦고 읽기는 상대적으로 덜 민감한 워크로드(로그 수집, 이벤트 스트림, 쓰기 위주 DB)에서 B+Tree 기반 in-place 구조보다 유리합니다."

실무 연결

실무 상황 Append-only / WAL / LSM-Tree 판단
RDBMS의 durability 보장 (PostgreSQL WAL, MySQL InnoDB redo log) 트랜잭션 커밋 시 실제 데이터 페이지가 아니라 WAL을 순차 fsync하는 것만으로 커밋을 확정합니다. 정전 후에는 마지막 체크포인트 이후 WAL을 재생해 복구합니다. "왜 커밋이 상대적으로 빠른가"를 설명할 때 핵심 근거가 됩니다.
쓰기 위주(write-heavy) 워크로드의 저장 엔진 선택 (RocksDB, Cassandra, LevelDB) 시계열 데이터, 이벤트 로그, IoT 센서 데이터처럼 쓰기 비율이 압도적으로 높고 최신 데이터 위주로 조회하는 워크로드는 LSM-Tree 기반 스토리지가 B+Tree 기반보다 쓰기 처리량에서 유리한 경우가 많습니다.
Kafka 같은 메시지/이벤트 로그 저장 Kafka의 파티션 로그는 append-only이며 메시지는 절대 수정되지 않고 offset으로만 참조됩니다. 순차 쓰기/읽기 위주로 설계되어 있어 매우 높은 처리량을 낼 수 있고, 오래된 세그먼트는 시간/용량 기준으로 통째로 삭제(로그 세그먼트 단위 정리)하는 방식으로 "compaction"과 유사한 정리를 수행합니다.
Compaction 전략으로 인한 운영 이슈 대응 LSM-Tree 기반 DB(RocksDB, Cassandra 등)를 운영하다 보면 compaction이 CPU/디스크 I/O를 급격히 소모해 지연시간 스파이크를 유발하는 경우가 있습니다. compaction 전략(leveled vs size-tiered)과 throttling 설정을 조정해 이 trade-off를 관리하는 것이 실무 튜닝 포인트입니다.
Event Sourcing / 감사 로그(audit log) 설계 "상태를 직접 갱신하지 않고 발생한 이벤트를 append-only로 쌓은 뒤 현재 상태를 재생으로 계산한다"는 아키텍처는 append-only 파일 구조의 아이디어를 애플리케이션 데이터 모델 레벨로 확장한 것입니다. 변경 이력이 그대로 보존되어 감사·재현·디버깅에 유리합니다.

답변 사고 순서 — 면접에서 이렇게 생각하면서 말한다

  1. 왜 순차 쓰기가 유리한지 원리부터 짚는다
    Day 1의 sequential vs random access 비용 차이를 다시 소환해 "append-only = random write 제거"라는 핵심 아이디어로 연결합니다.
  2. 갱신/삭제를 어떻게 표현하는지 설명한다
    같은 키의 새 레코드 추가, tombstone을 통한 삭제 표현, "최신 값 우선" 규칙을 짚습니다.
  3. 쌓인 무효 데이터를 어떻게 정리하는지(compaction) 설명한다
    compaction이 공간을 회수하고 읽기 비용을 낮추지만 새로운 I/O(write amplification)를 유발한다는 점을 짚습니다.
  4. 같은 원리의 두 가지 적용(WAL, LSM-Tree)으로 확장한다
    WAL은 durability에, LSM-Tree는 저장 엔진 전체 구조에 적용된 사례임을 구분해 설명합니다.
  5. Read/Write/Space amplification trade-off로 마무리한다
    "무엇을 얻고 무엇을 잃는가"를 명시적으로 말하고, 워크로드 특성(쓰기 위주 vs 읽기 위주)에 따라 선택이 달라진다는 결론을 냅니다.

짧은 버전 (1~2분 답변)

강한 답변 (Short)

Append-only 구조는 기존 데이터를 절대 덮어쓰지 않고 항상 파일 끝에만 씁니다. 갱신은 같은 키로 새 레코드를 추가하고, 삭제는 tombstone을 추가하는 방식으로 표현하며, 읽을 때는 가장 최신 레코드를 유효값으로 취급합니다. 모든 쓰기가 순차 쓰기가 되어 랜덤 쓰기 비용을 없애는 대신, 무효화된 옛 버전이 계속 쌓이므로 compaction으로 주기적으로 정리해야 합니다. WAL은 이 아이디어를 트랜잭션 durability에, LSM-Tree는 저장 엔진 전체 구조에 적용한 사례이고, 쓰기가 매우 잦은 워크로드에서 B+Tree 기반 in-place 구조보다 쓰기 성능이 유리한 대신 읽기와 공간 사용에서 대가를 치릅니다.

깊이 있는 버전 (3~5분 답변)

강한 답변 (Deep)

Append-only 구조의 출발점은 "디스크는 순차 접근이 랜덤 접근보다 훨씬 빠르다"는 물리적 특성입니다. 전통적인 in-place update 구조는 갱신할 때마다 원래 레코드 위치를 찾아가 덮어써야 해서, 갱신이 잦을수록 랜덤 쓰기가 반복됩니다. Append-only는 이 문제를 근본적으로 없앱니다 — 갱신도 삭제도 전부 "파일 끝에 새 레코드를 추가하는 것"으로 표현하기 때문에 디스크에는 오직 순차 쓰기만 발생합니다.

구체적으로는 같은 키에 대해 새 값을 append하면 그것이 최신값이 되고, 삭제는 tombstone이라는 특수 레코드를 append해서 표현합니다. 읽을 때는 그 키에 대해 기록된 것 중 가장 최근 것을 유효값으로 판단합니다. 문제는 이렇게 쌓기만 하면 무효화된 과거 버전이 파일에 계속 남아 공간을 낭비하고, 특정 키의 최신 값을 찾기 위해 훑어야 할 데이터도 늘어난다는 점입니다. 이를 해결하는 것이 compaction — 여러 레코드/파일을 읽어 키별 최신 값만 남긴 새 파일을 만드는 백그라운드 작업입니다. compaction 자체도 결과물을 순차 쓰기 때문에 랜덤 쓰기를 유발하지 않지만, 같은 데이터를 여러 번 다시 쓰게 되므로 write amplification이라는 비용이 발생합니다.

이 원리는 두 가지 층위에 적용됩니다. 첫째, WAL(Write-Ahead Log)은 DBMS가 실제 데이터 페이지를 수정하기 전에 "무엇을 바꿀 것인지"를 먼저 순차 로그에 기록하고, 그 로그의 fsync가 끝나면 커밋을 확정합니다. 실제 데이터 파일 반영(체크포인트)은 나중에 몰아서 처리해도 되므로, 커밋 경로에서 느린 랜덤 쓰기를 제거하면서도 정전 시에는 로그를 재생해 durability를 보장할 수 있습니다. 둘째, LSM-Tree는 이 아이디어를 저장 엔진 전체 구조로 확장한 것입니다. 쓰기는 메모리 내 정렬 구조 (memtable)에 먼저 반영되고, 일정 크기가 차면 정렬된 상태 그대로 디스크에 불변 파일(SSTable)로 순차 저장됩니다. 이 SSTable들이 쌓이면 백그라운드 compaction이 병합해 최신 값만 남은 더 적은 수의 SSTable로 재구성합니다. 읽기는 memtable부터 최근 SSTable 순으로 확인하며, Bloom filter로 불필요한 SSTable 탐색을 걸러냅니다.

결국 이것은 Read amplification, Write amplification, Space amplification 세 가지를 동시에 최소화할 수 없다는 trade-off로 요약됩니다. B+Tree 기반 in-place 구조는 read/space amplification이 낮은 대신 random write 비용이 크고, LSM-Tree는 순차 쓰기로 쓰기 처리량은 높이지만 compaction으로 인한 write amplification과 다중 SSTable 탐색으로 인한 read amplification을 대가로 치릅니다. 그래서 쓰기가 매우 잦고 최신 데이터 위주로 조회하는 워크로드(로그 수집, 이벤트 스트림, 시계열 데이터)는 LSM-Tree/append-only 계열이 유리하고, 읽기 비율이 높고 갱신이 상대적으로 적은 워크로드는 B+Tree 기반이 여전히 유리한 경우가 많습니다.

약한 답변

"append-only는 그냥 파일 뒤에 계속 쓰는 거라 빠릅니다. 로그 파일이나 Kafka 같은 데서 씁니다."

왜 약한가: "빠르다"의 근거(random write 제거)를 설명하지 못했고, 갱신/삭제를 어떻게 표현하는지, 쌓인 무효 데이터를 어떻게 정리하는지(compaction), 그 대가로 무엇을 잃는지 (write/read/space amplification)를 전혀 다루지 못했습니다. "빠르다"는 결론만 있고 trade-off가 없는 답변은 10년차 기준에서 정의 암기 수준으로 평가됩니다.

Trade-off 정리

기준 In-place update (B+Tree 등) Append-only / LSM-Tree
쓰기 패턴 갱신 위치를 찾아 덮어씀 — random write 발생 가능 항상 파일 끝에 추가 — 순차 쓰기만 발생
쓰기 처리량(특히 쓰기 위주 워크로드) 랜덤 쓰기 비용에 제약받음 순차 쓰기 위주라 일반적으로 더 높음
Read amplification 낮음 — 인덱스로 바로 위치 확인 높아질 수 있음 — memtable + 여러 SSTable 확인 필요
Write amplification 상대적으로 낮음 — 필요한 페이지만 갱신 compaction으로 같은 데이터를 반복해서 다시 씀
Space amplification 낮음 — 최신 값만 저장 compaction 전까지 무효화된 옛 버전이 공간 차지
삭제/갱신 표현 제자리 수정/삭제 새 레코드 추가 / tombstone 추가
백그라운드 유지보수 비용 상대적으로 적음 compaction이 CPU/I/O를 상당히 소모
대표 사용처 읽기 비율 높은 OLTP, 전통적 RDBMS 인덱스 쓰기 위주 워크로드(시계열, 로그, 이벤트), RocksDB/Cassandra/LevelDB, Kafka
💡 면접에서는 "LSM-Tree가 B+Tree보다 항상 좋다"가 아니라, "쓰기/읽기/공간 중 무엇을 우선하느냐에 따라 선택이 달라진다"(RUM conjecture)는 점을 강조하는 것이 중요합니다.

흔한 오해와 바로잡기

오해 1: "append-only는 그냥 순차로 쓰기만 하니 인덱스가 필요 없다"

Append-only 로그만 있으면 특정 키를 찾기 위해 파일 전체를 훑어야(O(n)) 합니다. 실무 시스템은 append-only 로그 위에 별도의 인메모리 인덱스(hash map, memtable의 skip list 등)나 SSTable 내부 정렬 + sparse index를 결합해서 조회 성능을 확보합니다. "순차 쓰기 = 인덱스 불필요"는 잘못된 일반화입니다. 미신

오해 2: "LSM-Tree는 무조건 B+Tree보다 빠르다"

LSM-Tree는 쓰기 처리량에서 유리한 경우가 많을 뿐, 읽기(특히 점 조회 read 비율이 높은 워크로드)에서는 여러 SSTable을 확인해야 해서 오히려 B+Tree보다 느릴 수 있습니다. Bloom filter, 캐싱, compaction 전략으로 이를 완화하지만 근본적인 read amplification trade-off는 여전히 존재합니다. 미신

오해 3: "WAL이 있으면 체크포인트(실제 데이터 반영)는 안 해도 된다"

WAL만 계속 쌓이고 체크포인트가 없으면 로그가 무한히 커지고, 장애 복구 시 재생해야 할 로그 양도 늘어나 복구 시간이 길어집니다. 체크포인트는 "이 시점까지의 WAL 내용은 데이터 파일에 이미 반영됐다"는 지점을 만들어 로그를 자를 수 있게 하는 필수 과정입니다. 미신

오해 4: "compaction은 선택 사항이며, 안 해도 정합성에는 문제가 없다"

정합성(어떤 값이 최신인가) 자체는 compaction 없이도 "가장 최근 레코드 우선" 규칙으로 유지됩니다. 하지만 compaction 없이는 파일이 무한히 커지고 조회 시 훑어야 할 무효 레코드가 계속 늘어나 실질적으로 운영이 불가능해집니다. compaction은 정합성이 아니라 공간/성능 유지를 위한 필수 유지보수 작업입니다. 미신

Senior-Level Follow-up 질문과 답변 방향

"Leveled compaction과 Size-tiered compaction은 어떤 trade-off 차이가 있습니까?"

답변 방향: Size-tiered는 비슷한 크기의 SSTable들을 모아 하나로 합치는 방식으로, 쓰기 증폭은 상대적으로 적지만 같은 키가 여러 레벨의 SSTable에 중복될 수 있어 space amplification과 read amplification이 커지기 쉽습니다. Leveled compaction(RocksDB의 기본 전략 등)은 각 레벨 내에서는 키 범위가 겹치지 않도록 유지해 read/space amplification을 줄이지만, 데이터를 레벨 간 이동시키며 다시 쓰는 빈도가 높아 write amplification이 커집니다. 쓰기 위주 워크로드는 size-tiered, 읽기/공간 효율이 중요한 워크로드는 leveled를 선호하는 경향이 있습니다.

"Bloom filter는 LSM-Tree의 읽기 성능을 어떻게, 왜 개선합니까?"

답변 방향: Bloom filter는 "이 키가 이 SSTable에 확실히 없다" 또는 "있을 수도 있다"를 매우 빠르게(디스크 접근 없이 메모리에서) 판단하는 확률적 자료구조입니다. false negative는 없고 false positive만 존재하므로, "없다"는 답이 나오면 그 SSTable을 디스크에서 읽을 필요 없이 건너뛸 수 있습니다. SSTable이 많을수록 불필요한 디스크 접근을 크게 줄여 read amplification을 완화하는 핵심 도구입니다.

"WAL을 쓰는 DB에서 fsync를 매 커밋마다 하지 않으면 무엇을 잃습니까?"

답변 방향: fsync는 OS 버퍼 캐시에 있는 데이터를 실제 물리 디스크에 확실히 내려쓰도록 강제하는 호출입니다. 매 커밋마다 fsync하지 않으면 "커밋 완료" 응답 이후에도 데이터가 실제로는 아직 디스크가 아니라 메모리(OS 캐시)에만 있을 수 있고, 그 사이 정전이 나면 커밋했다고 응답받은 트랜잭션이 사라질 수 있습니다. 이는 durability(ACID의 D)를 잃는 것이며, 대신 커밋 지연시간과 처리량을 얻는 명시적인 trade-off(예: PostgreSQL의 synchronous_commit 옵션)입니다.

"Kafka 파티션 로그가 append-only라는 것이 컨슈머(consumer) 설계에 어떤 영향을 줍니까?"

답변 방향: 메시지가 절대 수정되지 않고 순서가 고정된 offset으로만 참조되므로, 컨슈머는 "어디까지 읽었는지"를 단순한 offset 커밋만으로 추적할 수 있고, 장애 후에도 마지막 커밋 offset부터 재개하면 됩니다. 또한 여러 컨슈머가 같은 로그를 독립적으로, 서로 다른 속도로 읽어도 로그 자체는 변하지 않으므로 재처리(replay)나 여러 소비자 그룹 지원이 자연스럽게 가능해집니다. 이는 append-only가 동시성 제어를 단순화하는 대표적인 사례입니다.
