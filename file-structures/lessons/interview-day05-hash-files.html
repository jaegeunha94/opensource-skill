해싱 기반 파일 — Collision, Overflow, Rehashing, Range Query 한계

면접 질문

"해시 인덱스가 equality lookup에서 이론상 O(1)에 가까운데도, 왜 대부분의 DB는 기본 인덱스를 B+Tree로 만들고 해시는 보조적으로만 씁니까?"

전제 개념 — 쉬운 한국어로 먼저 이해하기

5-1. 해시 파일이란 — "계산해서 바로 위치를 찾는다"

Day 2~4에서 다룬 sequential file과 B+Tree는 모두 정렬된 순서를 전제로 탐색 경로를 좁혀갑니다. 반면 해시 파일은 정렬을 아예 포기하는 대신, 키 값을 해시 함수에 넣어 "이 레코드가 저장된 블록(bucket) 번호"를 즉시 계산합니다. 예를 들어 h(key) = key % 버킷_개수 같은 함수로 계산하면, 탐색이 아니라 계산 한 번 + 디스크 I/O 한 번으로 레코드를 찾을 수 있습니다. B+Tree가 "트리 높이만큼 I/O"였다면, 해시는 이상적으로 "I/O 1회"입니다.

[ Hash File — key를 계산해서 바로 bucket 위치로 이동 ] key = 12345 h(key) = 12345 % 100 = 45 ← 계산만으로 bucket 번호 결정 Bucket 45 위치로 디스크 I/O 1회 접근 → 레코드 확인 (B+Tree라면 root → internal → leaf, 여러 번 I/O가 필요했던 것과 대비)

5-2. Collision(충돌)과 Overflow — 계산이 겹치는 순간부터 생기는 비용

해시 함수는 서로 다른 키를 같은 bucket 번호로 계산하는 경우가 반드시 생깁니다. 이를 collision(충돌)이라 합니다. bucket 하나(디스크 블록 하나)에 담을 수 있는 레코드 수는 블록 크기로 제한되므로, 충돌한 레코드가 계속 들어오면 bucket이 꽉 찹니다. 이때 overflow bucket(별도의 추가 블록)을 연결해 넘치는 레코드를 저장합니다.

[ Bucket이 꽉 차면 overflow chain으로 연결 ] Bucket 45 (가득 참) → overflow #1 (가득 참) → overflow #2 (일부 사용) key=12345 조회 시: best case: bucket 45에서 바로 발견 → I/O 1회 worst case: overflow chain을 끝까지 따라가야 함 → I/O 3회 이상 overflow chain이 길어질수록 "이상적인 O(1)"이 무너지고, 결국 B+Tree의 log(n)과 비슷하거나 더 나빠질 수 있음
⚠️ 면접관이 보는 것: "해시는 O(1)이다"라는 이론적 명제가 아니라, "충돌과 overflow chain이 쌓이면 실제 I/O 성능이 어떻게 저하되는지"를 압니다. 해시의 O(1)은 "충돌이 거의 없는 이상적인 상황"이라는 전제 위에 있다는 것을 짚어야 합니다.

5-3. Static Hashing의 근본 문제 — bucket 개수를 미리 정해야 한다

가장 단순한 해싱(static hashing)은 파일을 만들 때 bucket 개수를 고정합니다. 그런데 데이터가 예상보다 많이 늘어나면, 고정된 bucket 개수로는 감당이 안 되어 overflow chain이 걷잡을 수 없이 길어집니다. 반대로 bucket을 넉넉하게 미리 잡아두면, 데이터가 적을 때는 디스크 공간을 낭비합니다. 이것이 static hashing의 근본적인 딜레마입니다: "미래의 데이터 크기를 미리 알아야 한다"는 비현실적인 전제가 깔려 있습니다.

5-4. Dynamic Hashing — bucket 수를 데이터 양에 맞춰 늘린다

이 문제를 해결하기 위해 extendible hashing과 linear hashing 같은 dynamic hashing 기법이 나왔습니다. 핵심 아이디어는 "bucket이 꽉 찼을 때 그 순간에만 필요한 부분을 나누어(split) bucket 수를 점진적으로 늘린다"는 것입니다. Extendible hashing은 directory(주소 테이블)를 두어 필요한 bucket만 두 배로 분리하고, linear hashing은 directory 없이 순서대로 하나씩 bucket을 분리해 나갑니다. 둘 다 "전체 재해싱(rehash) 없이 점진적으로 확장"한다는 점에서 static hashing의 딜레마를 완화합니다.

[ Static Hashing vs Dynamic Hashing ] Static Hashing: bucket 수 고정 (예: 100개) → 데이터 급증 시 overflow chain 폭증 → 해결하려면 전체 재해싱(모든 레코드를 다시 계산해 재배치) 필요 → 재해싱 동안 서비스 영향(락, 대량 I/O) 발생 Dynamic Hashing (Extendible / Linear): bucket이 꽉 찰 때마다 "그 bucket만" 분리(split) → 전체 재해싱 없이 점진적으로 bucket 수 증가 → 순간적인 대량 I/O 부담을 시간에 걸쳐 분산

다만 "전체 재해싱을 피한다"고 해서 비용이 사라지는 것은 아닙니다. extendible hashing은 directory 크기가 커지면 그 자체가 메모리/디스크 부담이 되고, linear hashing은 split 순서가 고정돼 있어 아직 충돌이 없는 bucket까지 먼저 분리하는 비효율이 생길 수 있습니다. "재해싱 비용을 없앤다"가 아니라 "한 번에 몰아서 낼 비용을 잘게 쪼개 분산한다"는 표현이 더 정확합니다.

5-5. 왜 해시는 Range Query를 근본적으로 지원할 수 없는가

B+Tree는 정렬 순서를 유지하기 때문에 "30 이상 70 이하"처럼 인접한 키가 물리적으로도 가까이 있습니다. 하지만 좋은 해시 함수는 정반대의 목표를 가집니다: 비슷한 키라도 완전히 다른 bucket에 흩어지도록 만들어야 충돌을 줄일 수 있습니다. 즉 해시 함수는 의도적으로 순서 정보를 파괴합니다. 그 결과 key=30과 key=31이 전혀 다른 bucket에 저장될 수 있고, "30부터 70까지"를 조회하려면 이론적으로 그 범위 안의 모든 키 값에 대해 개별적으로 해시를 계산해 bucket을 하나씩 확인해야 합니다. 이는 사실상 풀스캔과 다를 바 없습니다.

[ 해시는 "순서 보존"을 포기하고 "충돌 최소화"를 택한 구조 ] B+Tree: key 30 → leaf[30~50] 근처 key 31 → leaf[30~50] 근처 (인접 저장) Hash: key 30 → bucket 7 key 31 → bucket 62 (완전히 흩어짐) range scan(30~70) 시: B+Tree: leaf 하나 찾고 → 연결 리스트 따라 순차 읽기 (Day 4) Hash: 30~70 사이 "존재 가능한 모든 키"를 개별 계산해야 함 → 사실상 불가능/비효율
?

왜 면접에서 이 질문을 묻는가

해시 테이블 구현 코드를 암기했는지가 아니라, "O(1)이라는 이론적 성능이 실무에서 언제 깨지고, 그 대가로 무엇을 포기하는지 구조적으로 설명할 수 있는가"를 봅니다.

  1. 이상적인 O(1)과 실제 성능의 차이를 아는가
    "해시는 무조건 빠르다"가 아니라, load factor가 높아지고 overflow chain이 길어지면 성능이 저하된다는 것을 정량적으로 설명해야 합니다.
  2. Static/Dynamic hashing의 trade-off를 구조적으로 설명할 수 있는가
    "그냥 동적으로 늘어난다"가 아니라, 전체 재해싱을 피하기 위해 점진적 split을 택했고 그 대신 directory 관리나 split 순서 비효율 같은 대가가 있다는 것을 말할 수 있어야 합니다.
  3. Range query가 왜 "느리다"가 아니라 "근본적으로 안 되는지"를 설명할 수 있는가
    해시 함수가 순서를 의도적으로 파괴하는 설계라는 인과관계까지 짚어야 합니다.
⚠️ 주니어 수준 답변: "해시는 O(1)이라 빠르지만 범위 검색이 안 돼서 B+Tree를 씁니다."
10년차 기대 답변: "해시의 O(1)은 충돌이 거의 없는 이상적 상황에서만 성립하며, load factor가 높아지면 overflow chain 때문에 실제 I/O 횟수가 늘어납니다. 게다가 해시 함수는 충돌을 줄이기 위해 의도적으로 키의 순서 정보를 흩어버리므로, 인접한 키가 물리적으로 인접해 있다는 보장이 없어 range scan을 구조적으로 지원할 수 없습니다. 그래서 equality lookup 위주 워크로드에는 해시가 유리하지만, 정렬·범위 조회가 잦은 일반적인 DB 인덱스는 B+Tree가 기본이 됩니다."

답변 사고 순서 — 면접에서 이렇게 생각하면서 말한다

  1. 워크로드의 접근 패턴부터 확인
    equality lookup 위주인지, range query가 필요한지부터 구분합니다. 이 판단이 해시와 B+Tree 중 무엇을 선택할지의 출발점입니다.
  2. 해시의 이상적 성능과 그 전제를 설명
    충돌이 적을 때 O(1)에 가깝다는 이론적 장점과, 그 전제가 무너지는 조건(load factor 상승)을 함께 짚습니다.
  3. Collision/overflow가 실제 I/O에 미치는 영향을 설명
    overflow chain이 길어지면 이상적 O(1)이 사실상 여러 번의 I/O로 저하된다는 인과관계를 제시합니다.
  4. Static vs Dynamic hashing의 trade-off를 설명
    전체 재해싱을 피하기 위한 점진적 split 방식과, 그 대신 감수하는 관리 비용을 균형 있게 설명합니다.
  5. Range query가 구조적으로 불가능한 이유로 마무리
    해시 함수가 순서를 의도적으로 파괴하는 설계라는 점을 근거로, 대부분의 범용 인덱스가 B+Tree를 기본으로 삼는 이유를 정리합니다.

핵심 답변 예시

짧은 버전 (1~2분 답변)

강한 답변 (Short)

해시 인덱스는 키를 해시 함수로 계산해 바로 bucket 위치를 찾기 때문에, 충돌이 적으면 이론적으로 I/O 1회에 가까운 조회가 가능합니다. 하지만 데이터가 늘어나 충돌과 overflow chain이 쌓이면 실제 성능은 저하되고, 이를 막기 위해 static hashing은 전체 재해싱을, dynamic hashing은 점진적 bucket split을 택합니다.

더 근본적인 문제는, 해시 함수가 충돌을 줄이기 위해 의도적으로 키의 순서 정보를 흩어버린다는 점입니다. 그래서 인접한 키가 물리적으로도 인접해 있다는 보장이 없고, 범위 검색을 구조적으로 지원할 수 없습니다. 그래서 equality lookup 위주 워크로드에는 해시가 유리하지만, 범용 DB 인덱스는 대부분 B+Tree를 기본으로 씁니다.

깊이 있는 버전 (3~5분 답변)

강한 답변 (Deep)

해시 기반 파일은 정렬을 포기하는 대신, 키를 해시 함수에 넣어 저장 위치(bucket)를 직접 계산합니다. 탐색 경로를 좁혀가는 B+Tree와 달리, 이상적인 상황에서는 계산 한 번과 디스크 I/O 한 번으로 레코드를 찾을 수 있어 이론적으로 O(1)에 가깝습니다.

문제는 이 O(1)이 "충돌이 거의 없는 상황"이라는 전제 위에서만 성립한다는 점입니다. 서로 다른 키가 같은 bucket으로 계산되는 충돌은 필연적으로 발생하고, bucket이 블록 크기 제한으로 꽉 차면 overflow bucket을 체인으로 연결합니다. 데이터가 늘어나 load factor가 높아질수록 이 overflow chain이 길어지고, 조회는 chain을 끝까지 따라가야 할 수도 있어 실제 I/O 횟수가 늘어납니다.

Static hashing은 bucket 개수를 파일 생성 시 고정하므로, 데이터 증가를 미리 정확히 예측해야 한다는 비현실적인 전제를 가집니다. 예측이 틀리면 overflow chain이 폭증하거나, 반대로 공간을 낭비합니다. 이를 완화하기 위해 extendible hashing이나 linear hashing 같은 dynamic hashing은 bucket이 꽉 찰 때마다 그 부분만 분리(split)해, 전체 재해싱 없이 점진적으로 bucket 수를 늘립니다. 다만 이것이 비용을 없애는 것은 아니고, directory 관리 비용이나 split 순서에 따른 비효율처럼 "한 번에 몰릴 비용을 시간에 걸쳐 분산"하는 것에 가깝습니다.

가장 근본적인 한계는 range query입니다. 좋은 해시 함수는 충돌을 줄이기 위해 비슷한 키라도 완전히 다른 bucket에 흩어지도록 설계됩니다. 즉 순서 정보를 의도적으로 파괴하는 것이 해시 함수의 목표입니다. 그 결과 "30 이상 70 이하" 같은 범위 조회를 하려면 그 범위 안의 모든 키 값에 대해 개별적으로 해시를 계산해 bucket을 확인해야 하며, 이는 사실상 풀스캔과 다르지 않습니다. 그래서 저는 해시와 B+Tree의 선택 기준을 "워크로드가 equality lookup 위주인지, 정렬·범위 조회가 필요한지"로 판단합니다. 세션 조회나 캐시 키처럼 항상 정확히 일치하는 값만 찾는다면 해시가 유리하고, 정렬·범위 조회가 잦은 범용 DB 인덱스는 B+Tree가 기본이 됩니다.

약한 답변

"해시는 O(1)이라 빠르지만 범위 검색은 못 해서 B+Tree를 씁니다."

왜 약한가: 해시가 왜 범위 검색을 못 하는지(순서 정보를 의도적으로 파괴하는 설계)에 대한 근거가 없고, O(1)이 무너지는 조건(충돌, overflow chain, load factor)도 설명하지 않았습니다. static/dynamic hashing의 trade-off도 언급하지 않아, 결론만 있고 구조적 이해가 드러나지 않습니다.

⇄

Trade-off 정리

기준 Hash File B+Tree
Equality lookup 이상적으로 O(1) (충돌 적을 때) O(log n), 트리 높이만큼 I/O
Range query 구조적으로 불가능 (순서 파괴) leaf 연결로 순차 I/O 지원
정렬 순회(ORDER BY) 불가능, 별도 정렬 필요 인덱스 자체가 정렬 순서 제공
데이터 증가 대응 static: 재해싱 필요 / dynamic: 점진적 split 노드 분할(split)로 자연스럽게 성장
저장 공간 예측 bucket 수 설계에 민감 상대적으로 유연
구현/운영 복잡도 충돌 처리, overflow, rehashing 관리 필요 split/merge, fill factor 관리 필요
실무 채택 key-value 캐시, 세션 조회, 샤딩 키 대부분의 RDBMS 기본 인덱스
💡 해시와 B+Tree의 선택은 "이 컬럼에 range query나 정렬이 필요한가" 한 가지 질문으로 대부분 결정됩니다. equality 전용 워크로드가 확실하다면 해시가 이론적으로 유리하지만, 미래에 범위 조회가 추가될 가능성이 있다면 B+Tree가 더 안전한 기본 선택입니다.
✗

흔한 오해와 바로잡기

오해 1: "해시가 O(1)이니 항상 B+Tree보다 빠르다"

O(1)은 충돌이 거의 없는 이상적 상황을 전제로 합니다. load factor가 높아지고 overflow chain이 길어지면 실제 I/O 횟수는 B+Tree보다 나빠질 수 있습니다. Big-O는 평균/이상적 케이스를 나타낼 뿐, worst case를 보장하지 않습니다. 미신

오해 2: "해시 함수를 잘 만들면 충돌이 아예 없앨 수 있다"

비둘기집 원리(pigeonhole principle)에 따라, 가능한 키의 수가 bucket 수보다 많으면 충돌은 수학적으로 피할 수 없습니다. 좋은 해시 함수는 충돌을 "줄이고 고르게 분산"시킬 뿐, 완전히 없애지 못합니다. 미신

오해 3: "dynamic hashing을 쓰면 재해싱 비용이 완전히 사라진다"

dynamic hashing은 "전체를 한 번에" 재해싱하는 대신 "필요한 부분만 점진적으로" 분리하는 것입니다. 비용을 없앤 것이 아니라 시간에 걸쳐 분산시킨 것이며, directory 관리나 split 순서에 따른 비효율 같은 새로운 비용도 함께 생깁니다. 미신

오해 4: "해시 인덱스는 정렬이 조금 느릴 뿐이지 정렬 자체는 가능하다"

해시는 정렬 순서를 저장하지 않으므로 "조금 느린 정렬"이 아니라 "정렬 정보 자체가 없는" 상태 입니다. ORDER BY가 필요하다면 해시 인덱스만으로는 불가능하고, 별도의 전체 정렬(external sort, Day 7)이 필요합니다. 미신

Senior-Level Follow-up 질문과 답변 방향

"Chaining과 Open Addressing의 차이는 무엇이고, 디스크 기반 파일에서는 왜 chaining(overflow bucket)이 더 흔합니까?"

답변 방향: Open addressing은 충돌 시 다른 슬롯을 탐사(probing)하는 방식으로, 메모리 내 해시 테이블에서는 캐시 지역성 덕분에 유리합니다. 하지만 디스크 기반 파일에서는 탐사 자체가 추가 블록 I/O를 유발할 수 있고, 삭제 시 tombstone 처리가 복잡해집니다. Overflow chaining은 원래 bucket 하나만 우선 접근하고, 넘칠 때만 추가 I/O를 감수하는 구조라 디스크 I/O 비용 모델에 더 잘 맞습니다.

"Extendible hashing에서 directory 크기가 계속 커지면 어떤 문제가 생깁니까?"

답변 방향: directory는 보통 메모리에 상주시켜 빠른 bucket 위치 계산을 노리는데, 데이터가 매우 많아지면 directory 자체가 커져 메모리 부담이 되거나 디스크에 걸쳐야 할 수 있습니다. 이 경우 directory 접근 자체가 추가 I/O가 되어 "계산 한 번으로 끝난다"는 해시의 이점이 줄어듭니다.

"분산 시스템에서 consistent hashing을 쓰는 이유는 무엇이고, Day 5의 해시 충돌 문제와 어떻게 다릅니까?"

답변 방향: 단일 파일의 해싱은 "충돌을 줄이고 빠른 조회"가 목표지만, 분산 시스템의 consistent hashing은 "노드가 추가/제거될 때 재배치되는 키의 비율을 최소화"하는 것이 목표입니다. 문제의 축이 다르지만, "전체를 다시 계산하지 않고 일부만 옮긴다"는 점에서 dynamic hashing의 점진적 split 아이디어와 철학적으로 닮아 있습니다.

"해시 인덱스에 LIKE 'abc%' 같은 접두사 검색을 지원할 수 있습니까?"

답변 방향: 불가능합니다. 접두사 검색도 결국 정렬 순서를 이용한 범위 검색의 한 형태이기 때문입니다. 해시 함수는 접두사가 같아도 전혀 다른 값으로 계산하므로, 이런 패턴 매칭이 필요한 컬럼은 애초에 B+Tree(또는 별도의 텍스트 인덱스)를 선택해야 합니다.
