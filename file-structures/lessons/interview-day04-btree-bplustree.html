Interview Day 4

B-Tree와 B+Tree — Fanout, Height, Range Scan

Multi-level index를 균형 트리로 일반화할 때, 왜 DB는 B-Tree가 아닌 B+Tree를 선택하는가

Q

면접 질문

"왜 대부분의 DB 인덱스는 B-Tree가 아니라 B+Tree를 사용합니까? 두 구조의 차이가 실제 조회 성능에 어떤 영향을 줍니까?"

변형 질문 유형

  • "B+Tree의 fanout이 클수록 왜 유리합니까? fanout은 무엇으로 결정됩니까?"
  • "1억 건 레코드에 B+Tree 인덱스를 걸면 트리의 높이는 대략 몇입니까?"
  • "B+Tree에서 range scan(범위 검색)이 B-Tree보다 빠른 이유는 무엇입니까?"
  • "B+Tree의 삽입/삭제 시 노드 분할(split)과 병합(merge)이 성능에 어떤 영향을 줍니까?"

전제 개념 — 쉬운 한국어로 먼저 이해하기

4-1. Day 3의 문제의식 다시 보기 — sparse index도 결국 커진다

Day 3에서 sparse index는 블록 수만큼만 항목을 두어 저장 공간을 아꼈습니다. 그런데 레코드가 수억 건이면 블록 수도 수백만 개가 될 수 있고, 그러면 sparse index 자체도 하나의 파일로는 감당하기 어려운 크기가 됩니다. 이때 "그 sparse index 파일을 다시 정렬된 파일로 보고, 그 위에 한 단계 더 작은 인덱스를 얹는다"는 아이디어가 나옵니다. 이렇게 인덱스를 계층적으로 쌓아 올리는 것을 multi-level index라고 합니다. B-Tree와 B+Tree는 이 multi-level index를 "삽입/삭제가 있어도 항상 균형을 유지하는 트리 구조"로 일반화한 것입니다.

4-2. 왜 이진 트리(BST)가 아니라 다지 트리(m-way tree)인가 — Fanout이라는 개념

메모리 안에서만 동작하는 이진 탐색 트리(BST)는 노드 하나가 자식을 최대 2개만 가집니다. 그런데 디스크 기반 인덱스에서 성능을 좌우하는 것은 "비교 횟수"가 아니라 "디스크에서 블록을 몇 번 읽는가(I/O 횟수)"입니다. Day 1에서 다룬 것처럼 디스크 I/O 1회 비용은 메모리 접근보다 수만 배 비쌉니다. 그래서 트리 노드 1개를 디스크 블록(페이지) 1개에 맞추고, 그 노드 안에 최대한 많은 키를 채워 넣어 한 번의 I/O로 더 많은 정보를 얻도록 설계합니다. 이때 노드 하나가 가질 수 있는 자식 수를 fanout(분기 수)이라고 부릅니다.

[ BST(이진, fanout=2) vs B+Tree(다지, fanout=100) — 100만 건 기준 높이 비교 ] BST (fanout=2): 높이 ≈ log₂(1,000,000) ≈ 20 → 조회 시 최대 20번 I/O B+Tree(fanout=100): 높이 ≈ log₁₀₀(1,000,000) ≈ 3 → 조회 시 최대 3번 I/O 노드 하나 = 디스크 블록(페이지) 하나 fanout이 크다 = 한 페이지 안에 더 많은 키를 채운다 = 트리가 더 "낮고 넓어진다" 트리가 낮을수록 루트에서 리프까지 가는 데 필요한 I/O 횟수가 줄어든다

즉 fanout을 늘리는 것은 단순한 최적화가 아니라, "I/O 횟수 = 트리 높이"라는 등식을 최소화하는 핵심 레버입니다. fanout은 "페이지 크기 ÷ (키 크기 + 포인터 크기)"로 대략 결정되므로, 키를 작게(예: 정수 PK) 유지하는 것도 실무에서 fanout을 키우는 방법 중 하나입니다.

4-3. B-Tree — 모든 노드에 데이터를 둔다

B-Tree는 internal node(중간 노드)에도 실제 레코드(또는 레코드 포인터)를 저장합니다. 그래서 루트나 중간 노드에서 찾는 키를 만나면, 리프까지 내려가지 않고 바로 그 자리에서 조회를 끝낼 수 있습니다. "운이 좋으면 더 빨리 끝난다"는 장점이 있습니다.

[ B-Tree — internal node에도 데이터가 있음 ] [ 30:data | 60:data ] / | \ [10:data|20:data] [40:data|50:data] [70:data|80:data] 키=30 조회 → 루트에서 바로 발견 (1 I/O로 종료, 운이 좋은 경우) 키=45 조회 → 루트 → 중간 리프까지 내려가야 함 (여러 I/O) range scan(30~70) → 트리를 오르내리며 순회해야 함 (leaf끼리 연결 안 됨)

4-4. B+Tree — 데이터는 오직 리프에만, internal node는 이정표 역할만

B+Tree는 internal node에는 오직 "키 값과 자식을 찾아가기 위한 포인터"만 두고, 실제 레코드(또는 레코드 포인터)는 리프 노드에만 저장합니다. 그리고 결정적으로, 모든 리프 노드를 옆으로 연결된 linked list로 묶어 둡니다.

[ B+Tree — 데이터는 leaf에만, leaf는 서로 연결됨 ] [ 30 | 60 ] ← internal node: 이정표(키)만 있음 / | \ [10|20]→[30|40|50]→[60|70]→[80|90] ← leaf: 실제 데이터 + 다음 leaf로의 포인터 키=45 조회 → 항상 루트에서 리프까지 끝까지 내려감 (I/O 횟수가 항상 "트리 높이"로 일정) range scan(30~70) → leaf인 [30|40|50]을 찾은 뒤, → 포인터를 따라 옆으로만 쭉 읽으면 끝 (트리를 다시 오르내릴 필요 없음)
⚠️ 면접관이 보는 것: "B+Tree가 무조건 더 빠르다"는 결과 암기가 아니라, "internal node를 이정표 전용으로 비워서 fanout을 극대화하고, leaf를 연결해 range scan을 순차 I/O로 바꿨다"는 설계 의도를 아는지를 봅니다. B-Tree가 "가끔 더 빠를 수도 있다"는 점과, B+Tree가 그 대신 무엇을 얻었는지를 함께 설명해야 합니다.

4-5. 왜 B+Tree가 "이정표만 남겨서" 더 유리한가 — fanout과 range scan 두 가지 이유

첫째, B-Tree는 internal node 안에 데이터(또는 데이터 포인터)까지 같이 저장하므로, 같은 크기의 페이지 안에 담을 수 있는 키의 개수가 줄어듭니다. 반면 B+Tree의 internal node는 키와 자식 포인터만 있어 훨씬 가볍고, 그만큼 같은 페이지에 더 많은 키를 채워 fanout이 커집니다. fanout이 크면 트리 높이가 낮아지고, 트리 높이가 낮으면 조회에 필요한 I/O가 줄어듭니다.

둘째, B-Tree는 데이터가 트리 전체에 흩어져 있어 range scan을 하려면 트리를 오르내리며 순회해야 합니다. B+Tree는 데이터가 오직 리프에만 있고, 그 리프들이 정렬된 순서로 서로 연결돼 있으므로 range scan은 리프 하나만 먼저 찾고 나면 그 다음부터는 포인터를 따라 옆으로 읽기만 하면 됩니다. 이것이 사실상 sequential access이므로, Day 1에서 다룬 "순차 접근이 랜덤 접근보다 훨씬 싸다"는 원리를 그대로 활용하는 것입니다.

왜 면접에서 이 질문을 묻는가

B-Tree/B+Tree 삽입 알고리즘을 손으로 그릴 수 있는지가 아니라, "트리 구조 설계가 실제 I/O 비용에 어떤 영향을 주는지 인과관계로 설명할 수 있는가"를 봅니다.

  1. I/O 횟수와 트리 높이의 관계를 정량적으로 아는가
    "fanout이 크면 빠르다"를 넘어서, 1억 건 레코드에서 fanout 100이면 높이가 3~4 정도라는 감각적인 계산을 할 수 있어야 합니다.
  2. B+Tree의 설계 의도를 구조적으로 설명할 수 있는가
    "그냥 B+Tree가 더 좋다고 배웠다"가 아니라, internal node를 비워 fanout을 늘리고 leaf를 연결해 range scan을 순차 I/O로 바꿨다는 두 가지 이유를 각각 말할 수 있어야 합니다.
  3. Day 1~3의 개념과 연결되는가
    B+Tree는 결국 Day 1의 block I/O 비용 모델, Day 3의 multi-level sparse index를 이어받은 구조입니다. 이 연결을 짚지 못하면 암기처럼 들립니다.
⚠️ 주니어 수준 답변: "B+Tree는 리프에 데이터가 다 있고 리프끼리 연결돼 있어서 B-Tree보다 좋습니다."
10년차 기대 답변: "internal node를 키/포인터 전용으로 비워 같은 페이지 크기에서 fanout을 극대화하고, 그 결과 트리 높이가 낮아져 point lookup의 I/O 횟수가 줄어듭니다. 또한 데이터를 리프에만 두고 리프를 정렬된 순서로 연결해, range scan을 트리 순회가 아니라 순차 I/O로 바꾸었습니다. B-Tree는 운이 좋으면 internal node에서 더 빨리 끝날 수 있지만, 그 대가로 fanout이 줄고 range scan이 비효율적이라는 trade-off를 안습니다."

실무 연결

실무 상황 B-Tree / B+Tree 판단
InnoDB(MySQL) 클러스터드 인덱스 PK 기준 B+Tree이며 leaf 자체가 실제 데이터 페이지입니다. leaf 연결 리스트 덕분에 WHERE id BETWEEN a AND b 같은 range scan이 순차 I/O로 처리됩니다.
페이지 크기(innodb_page_size)와 fanout 16KB 페이지에 PK(정수 8바이트) + 포인터(6바이트) 기준이면 fanout이 수백 단위가 되어, 수억 건도 트리 높이 3~4로 처리 가능합니다. 페이지 크기를 키우거나 키를 작게 유지하는 것이 실무에서 fanout을 관리하는 방법입니다.
ORDER BY + LIMIT 페이지네이션 정렬 기준 컬럼이 B+Tree 인덱스라면, leaf 연결 리스트를 따라가며 필요한 만큼만 읽으면 되어 매우 효율적입니다. 반대로 인덱스 없는 컬럼으로 정렬하면 전체를 읽어 정렬해야 하는 비용이 발생합니다.
대량 INSERT 시 노드 분할(page split) 정렬 순서와 무관한 키(UUID 등)로 대량 삽입하면 리프 중간에 계속 끼워 넣어야 해서 page split이 빈번해지고 쓰기 성능이 떨어집니다. auto-increment나 시간 정렬 가능한 키를 선호하는 이유입니다.
인덱스 재구성(OPTIMIZE TABLE 등) 잦은 삭제/삽입으로 리프의 채움률(fill factor)이 낮아지면 fanout 효과가 떨어져 트리 높이가 실질적으로 늘어날 수 있습니다. 정기적인 인덱스 재구성으로 채움률을 회복합니다.

답변 사고 순서 — 면접에서 이렇게 생각하면서 말한다

  1. 왜 이진 트리가 아니라 다지 트리인지부터 설명
    디스크 I/O 비용 모델에서는 비교 횟수가 아니라 I/O 횟수가 성능을 좌우하므로, 노드당 fanout을 키워 트리 높이를 낮추는 것이 목표라는 점을 먼저 짚습니다.
  2. B-Tree와 B+Tree의 구조적 차이를 정의
    B-Tree는 internal node에도 데이터를 두고, B+Tree는 데이터를 리프에만 두고 리프를 연결한다는 차이를 명확히 말합니다.
  3. fanout 관점에서 우위를 설명
    internal node가 가벼울수록 같은 페이지에 더 많은 키를 담아 fanout이 커지고 트리 높이가 낮아진다는 인과관계를 제시합니다.
  4. range scan 관점에서 우위를 설명
    leaf 연결 리스트 덕분에 range scan이 트리 순회가 아니라 순차 I/O가 된다는 점을 강조합니다.
  5. B-Tree가 유리한 경우도 인정
    point lookup에서 운 좋게 internal node에서 바로 발견되는 경우가 있지만, 이는 평균적 이득이 아니라 최악의 경우 대비가 어렵다는 점까지 균형 있게 마무리합니다.

핵심 답변 예시

짧은 버전 (1~2분 답변)

강한 답변 (Short)

B-Tree는 internal node에도 데이터를 저장해서 운이 좋으면 조회가 빨리 끝나지만, 그만큼 한 페이지에 담을 수 있는 키 수(fanout)가 줄어 트리가 더 높아집니다. B+Tree는 데이터를 리프에만 두고 internal node는 이정표 역할만 하도록 가볍게 만들어 fanout을 극대화하고, 그 결과 트리 높이가 낮아져 point lookup의 I/O 횟수가 줄어듭니다.

게다가 B+Tree는 모든 리프를 정렬된 순서로 연결해 둬서, range scan을 할 때 리프 하나만 먼저 찾으면 그 다음부터는 트리를 다시 타지 않고 옆으로 순차적으로 읽으면 됩니다. 이 두 가지 이유로 대부분의 DB 인덱스가 B+Tree를 채택합니다.

깊이 있는 버전 (3~5분 답변)

강한 답변 (Deep)

디스크 기반 인덱스에서 성능을 좌우하는 것은 비교 연산 횟수가 아니라 디스크 I/O 횟수입니다. 트리 노드 하나를 디스크 페이지 하나에 맞춘다면, 노드 하나가 얼마나 많은 자식을 가질 수 있는가(fanout)가 트리의 높이, 곧 조회에 필요한 I/O 횟수를 결정합니다. 이진 트리는 fanout이 2뿐이라 백만 건만 되어도 높이가 20에 가깝지만, fanout을 100 수준으로 키우면 같은 데이터를 높이 3 정도로 다룰 수 있습니다.

B-Tree는 internal node에도 실제 레코드나 그 포인터를 함께 저장합니다. 그래서 찾는 키가 운 좋게 중간 노드에 있으면 리프까지 내려가지 않고 끝낼 수 있다는 장점이 있습니다. 하지만 이 데이터가 내부 노드의 공간을 차지하기 때문에, 같은 페이지 크기 안에 담을 수 있는 키의 개수, 즉 fanout이 줄어듭니다. 게다가 데이터가 트리 전체에 흩어져 있어서 범위 검색을 하려면 트리를 오르내리며 순회해야 하는 부담이 있습니다.

B+Tree는 이 두 가지 문제를 구조적으로 해결합니다. internal node는 키 값과 자식 포인터만 저장해 매우 가볍게 유지하고, 그 덕분에 같은 페이지 크기에서 훨씬 많은 키를 담아 fanout을 극대화합니다. 실제 레코드는 오직 리프 노드에만 저장하고, 모든 리프를 정렬된 순서로 연결된 리스트로 묶습니다. 그 결과 point lookup은 항상 루트에서 리프까지 일정한 횟수의 I/O로 끝나고(worst case가 곧 일반적인 경우), range scan은 시작 리프를 찾은 뒤부터는 트리를 다시 타지 않고 리프 포인터를 따라 순차적으로 읽기만 하면 됩니다. 이는 디스크에서 순차 접근이 랜덤 접근보다 훨씬 저렴하다는 원리를 그대로 활용하는 설계입니다.

결론적으로 저는 이 선택을 "B+Tree가 무조건 더 좋다"가 아니라, "internal node를 비워 fanout을 키우고, 리프를 연결해 range scan을 순차 I/O로 바꾼 대가로 point lookup에서의 우연한 조기 종료 가능성을 포기한 것"이라는 trade-off로 설명합니다. 대부분의 실무 워크로드는 range scan과 정렬된 순회의 빈도가 높기 때문에 이 trade-off가 유리하게 작용합니다.

약한 답변

"B+Tree는 리프 노드에만 데이터가 있고 리프끼리 연결돼 있어서 검색이 빠릅니다. 그래서 DB는 B+Tree를 씁니다."

왜 약한가: "빠르다"는 결론만 있고 왜 빠른지에 대한 인과관계(fanout 증가로 인한 트리 높이 감소, 순차 I/O로 전환된 range scan)가 빠져 있습니다. B-Tree가 어떤 상황에서는 유리할 수 있다는 것도 언급하지 않아, 단순 비교가 아니라 trade-off를 이해하고 있는지 확인할 수 없습니다.

⇄

Trade-off 정리

기준 B-Tree B+Tree
데이터 저장 위치 internal node + leaf 모두 leaf에만
같은 페이지 크기의 fanout 작음 (데이터가 공간을 차지) 큼 (internal node가 가벼움)
Point lookup 최선의 경우 internal node에서 조기 종료 가능 항상 리프까지 내려감 (일정한 비용)
Point lookup 평균/최악의 경우 트리가 더 높아 상대적으로 불리 fanout이 커서 유리
Range scan 트리 순회 필요 (비효율적) leaf 연결 리스트로 순차 I/O
구현 복잡도 상대적으로 단순 (중복 저장 없음) 리프 연결 리스트 유지 로직 추가
실무 채택 파일 시스템 일부(NTFS 등), 특정 인메모리 구조 대부분의 RDBMS 인덱스(InnoDB, PostgreSQL 등)
💡 B-Tree와 B+Tree의 우열은 워크로드의 range scan 비중에 달려 있습니다. 정렬 순회와 범위 검색이 잦은 DBMS 워크로드에서는 B+Tree가 압도적으로 유리하지만, point lookup 위주이고 range scan이 거의 없는 워크로드라면 B-Tree의 조기 종료 이점이 상대적으로 부각될 수 있습니다.
✗

흔한 오해와 바로잡기

오해 1: "B+Tree는 그냥 B-Tree에 리프 연결 리스트만 추가한 것이다"

리프 연결은 결과이지 원인이 아닙니다. 더 근본적인 차이는 internal node에 데이터를 두는지 여부이고, 이것이 fanout과 트리 높이를 결정합니다. 리프 연결은 "데이터가 리프에만 있다"는 구조 위에서 range scan을 최적화하기 위해 추가된 것입니다. 미신

오해 2: "트리 높이가 O(log n)이니 데이터가 아무리 늘어나도 성능은 거의 그대로다"

O(log n)이라도 fanout이 작으면 실제 I/O 횟수 차이가 큽니다. fanout 2와 fanout 100은 같은 O(log n)이어도 백만 건 기준으로 20 vs 3이라는 실질적 차이를 만듭니다. Big-O 표기만으로는 실무 인덱스 설계에서 fanout 관리의 중요성이 드러나지 않습니다. 미신

오해 3: "B+Tree는 항상 B-Tree보다 빠르다"

point lookup만 반복되고 range scan이 전혀 없는 워크로드라면, B-Tree가 일부 조회에서 internal node 조기 종료로 더 빠를 수 있습니다. B+Tree의 우위는 "range scan과 정렬 순회가 잦은 실무 워크로드"라는 전제 위에서 성립하는 것이지, 무조건적인 우열이 아닙니다. 미신

오해 4: "인덱스 컬럼 개수를 늘려도 fanout에는 영향이 없다"

복합 인덱스(composite index)는 키 크기가 커지므로 같은 페이지에 담을 수 있는 키 개수, 즉 fanout이 줄어듭니다. 컬럼을 무분별하게 추가하면 트리 높이가 늘어나 조회 I/O가 증가할 수 있다는 점을 실무에서 놓치기 쉽습니다. 미신

Senior-Level Follow-up 질문과 답변 방향

"B+Tree 삽입 중 노드가 꽉 차면 어떤 일이 일어납니까? 이것이 성능에 어떤 영향을 줍니까?"

답변 방향: 노드가 가득 차면 노드를 둘로 나누는 page split이 일어나고, 그 여파가 부모 노드까지 전파될 수 있습니다(cascading split). split은 추가 쓰기 I/O를 유발하고, 정렬 순서와 무관한 키로 삽입할 때 빈번히 발생합니다. 이를 줄이기 위해 노드를 처음부터 일부러 100%가 아니라 60~70% 정도만 채우는 fill factor 조정을 사용하기도 합니다.

"삭제가 반복되면 트리가 어떻게 변형됩니까?"

답변 방향: 노드의 채움률이 최소 기준 이하로 떨어지면 인접 노드와 병합(merge)하거나 키를 재분배 (redistribution)합니다. 병합/재분배가 잦으면 트리 구조가 계속 바뀌며 쓰기 오버헤드가 커지므로, 삭제가 많은 워크로드에서는 주기적인 인덱스 재구성(rebuild)을 고려합니다.

"메모리에 트리 전체를 올릴 수 있다면, 그래도 B+Tree가 최선입니까?"

답변 방향: 메모리 안에서는 디스크 I/O 비용이 사라지므로 fanout을 극대화할 필요성이 줄고, 오히려 캐시 라인 크기에 맞춘 더 작은 fanout이나 이진 탐색 트리 계열(레드-블랙 트리 등)이 유리할 수 있습니다. B+Tree의 설계는 디스크 I/O 비용 모델을 전제로 한 선택이므로, in-memory 구조에서는 같은 결론이 그대로 적용되지 않습니다. (다만 in-memory에서도 cache line 접근을 블록처럼 취급하는 cache-oblivious B-tree 계열은 여전히 유효합니다.)

"B+Tree 인덱스에서 매우 선택도(selectivity)가 낮은 컬럼(예: boolean)에 인덱스를 걸면 어떻게 됩니까?"

답변 방향: 선택도가 낮으면(값의 종류가 적으면) 인덱스로 걸러지는 레코드 비율이 작아, 결국 데이터 페이지의 대부분을 다시 읽어야 해서 순차 풀스캔보다 느려질 수 있습니다. 옵티마이저가 이런 경우 인덱스를 무시하고 full scan을 선택하는 이유이기도 합니다. B+Tree 자체의 문제가 아니라 "인덱스를 쓸 가치가 있는 선택도인가"를 먼저 판단해야 합니다.
