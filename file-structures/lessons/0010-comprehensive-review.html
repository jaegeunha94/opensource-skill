0. 왜 파일구조 선택이 결정적인가?

데이터를 어떤 구조로 저장하느냐는 시스템 성능의 가장 근본적인 결정이다. 동일한 하드웨어와 동일한 1,000만 건 데이터를 다룰 때, 순차 파일은 임의 탐색에 O(N) I/O가 들지만 B+-트리는 O(log N)·3–4 I/O로 끝낸다. 대규모 시스템에서 이 차이는 수십 배의 응답 시간 차이로 직결된다.

오해 주의 "B+-트리가 항상 최선"이라는 오해가 흔하다. 순수 순차 스캔이 필요한 배치 ETL에서는 순차 파일이 더 빠르고, 균등 분포 데이터의 등치 탐색에서는 해싱이 B+-트리보다 빠르다. 요구사항이 구조를 결정한다.

2. 파일구조 선택 결정 프레임워크

다음 네 가지 질문에 순서대로 답하면 최적 구조를 좁힐 수 있다.

질문 1: 탐색 유형은 무엇인가?

등치 탐색만 (WHERE key = ?) ├── 해싱 → 평균 O(1), 최선 선택 └── B+-트리 → O(log N), 범위 탐색 동시 지원 시 선택 범위 탐색 포함 (WHERE key BETWEEN a AND b) ├── B+-트리 → 리프 연결로 연속 스캔, 최선 선택 └── 정렬 순차 파일 → 갱신이 없고 읽기만 할 때 전체 스캔 / 대량 배치 └── 순차 파일 (정렬·비정렬) → 연속 I/O로 최고 처리량

질문 2: 갱신(삽입·삭제·수정) 빈도는?

  • 읽기 전용 또는 희귀 갱신: 정렬 순차 파일 + 덴스 인덱스 충분
  • 잦은 갱신: B+-트리 또는 동적 해싱. 정적 순차 파일은 재구성 비용 때문에 부적합
  • 삽입만 빈번, 삭제 없음: 직렬 파일 + 주기적 정렬 재구성도 실용적

질문 3: 파일 크기와 메모리 비율은?

  • 데이터가 메모리보다 훨씬 큼: 블록 I/O 최소화가 핵심 → B+-트리 or 인덱스
  • 데이터가 메모리에 거의 들어감: 구조 오버헤드보다 인메모리 캐시 활용이 유리
  • 정렬이 필요한 대용량: 외부 정렬(k-way 합병)을 선행 후 순차 파일로 저장

질문 4: 순서 보장이 필요한가?

  • 순서 출력 필요: 정렬 순차 파일, B-트리, B+-트리 중 선택
  • 순서 불필요: 해싱 가능 (단, 범위 탐색 불가 감수)
결정 요약 규칙 (1) 범위 탐색 → B+-트리 우선. (2) 등치만 + 높은 갱신 → 해싱. (3) 읽기 전용 배치 → 정렬 순차. (4) 복합 조건 → 보조 인덱스·복합 인덱스. (5) 대용량 정렬 전처리 → 외부 정렬.

3. 실전 시나리오: 어떤 구조를 쓰는가?

시나리오 A: 관계형 DBMS (PostgreSQL, MySQL InnoDB)

InnoDB의 기본 저장 구조는 클러스터드 B+-트리다. 기본 키를 리프 노드에 데이터 행 자체가 들어가는 구조로 저장한다. 보조 인덱스는 기본 키 값을 저장해 이중 탐색으로 행을 조회한다. 등치·범위·정렬 출력 모두를 단일 구조로 지원하기 때문이다.

InnoDB B+-트리 (클러스터드) 루트 페이지 ├── 내부 노드 (키 + 자식 포인터) └── 리프 노드 [pk1 | 행데이터] ↔ [pk2 | 행데이터] ↔ ... ↑ 범위 스캔은 리프를 좌→우로 직접 연결해 스캔

시나리오 B: NoSQL 키-값 저장소 (Redis, DynamoDB)

등치 탐색이 압도적으로 많고 범위 탐색이 드물 때 해싱이 유리하다. Redis는 인메모리 해시 테이블로 O(1) 조회를 제공하고, DynamoDB는 파티션 키를 해시하여 물리적 파티션을 선택한 뒤 선택적으로 정렬 키(Sort Key)에 B+-트리를 적용해 범위 탐색을 지원한다.

시나리오 C: 파일시스템 (ext4, NTFS)

Day 8에서 배운 대로 ext4는 inode + 다중 간접 포인터로 파일 블록을 관리하고, 디렉터리 엔트리는 해시 트리(HTree, B-트리 변형)로 빠른 파일 이름 탐색을 지원한다. 대용량 디렉터리에서 선형 탐색 O(N) 대신 O(log N)을 달성한다.

시나리오 D: 검색 엔진 역방향 인덱스 (Elasticsearch)

검색 엔진은 역방향 인덱스(Inverted Index)를 사용한다. 이는 보조 인덱스의 특수한 형태로, 단어(Term)를 키로, 해당 단어가 등장하는 문서 ID 목록을 값으로 저장한다. 세그먼트별로 정렬된 순차 파일 형태로 디스크에 기록하고, 조회 시 병합한다. 인덱스 갱신은 새 세그먼트를 추가(불변 쓰기)하는 방식으로 B+-트리의 잦은 분할을 피한다.

시나리오 E: 대용량 배치 정렬 파이프라인

Day 9에서 배운 외부 정렬의 전형적 사용처다. 예: 수억 건의 클릭로그를 사용자 ID 기준으로 정렬해 추천 모델에 입력할 때. 교체 선택으로 초기 런 길이를 2M으로 늘리고, 폴리페이즈 합병으로 패스 수를 줄여 전체 I/O를 N·logk(N/M)에서 최적화한다.

5. 트레이드오프 심화: B+-트리 vs 해싱

실무에서 가장 자주 나오는 선택지다. 아래 두 특성을 기준으로 판단한다.

요구사항 매트릭스 | 등치 탐색 | 범위 탐색 | 순서 출력 ───────────────┼─────────────┼────────────┼────────── 정적 해싱 | ★★★★★ | ✗ | ✗ 동적 해싱 | ★★★★☆ | ✗ | ✗ B-트리 | ★★★★☆ | ★★★☆☆ | ★★★★☆ B+-트리 | ★★★★☆ | ★★★★★ | ★★★★★ 정렬 순차 파일 | ★★★☆☆ | ★★★★☆ | ★★★★★ ── 갱신 비용 ────────────────────────────────────── 정적 해싱 | 오버플로우 체인 증가 → 재해싱 필요 동적 해싱 | 버킷 분할·디렉터리 확장, 단 점진적 B-트리/B+-트리 | 분할·합병 O(log N), 예측 가능 정렬 순차 파일 | 오버플로우 영역 → 주기적 재구성 O(N)

해싱이 B+-트리를 이기는 조건

  • 탐색의 99% 이상이 WHERE pk = ? 형태
  • 데이터 분포가 균등 (해시 충돌 최소)
  • 파일 크기 변화가 예측 가능하거나 동적 해싱 사용 가능
  • 정렬 출력이 전혀 필요 없음

B+-트리가 해싱을 이기는 조건

  • 범위 탐색(BETWEEN, >, <)이 포함됨
  • ORDER BY 없이도 정렬된 결과가 필요
  • 데이터 분포가 편중(skewed)되어 해시 충돌 위험
  • 파일 크기 변화가 예측 불가
