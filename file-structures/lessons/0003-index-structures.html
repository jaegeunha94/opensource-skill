Day 3: 인덱스 구조 기초

📅 2026-06-23 ⏱ 약 35분 📚 파일구조 과목 🔢 Lesson 3 / 10
학습 목표 이 레슨을 마치면: (1) 인덱스가 왜 필요한지, 순차 파일의 한계와 연결하여 설명할 수 있다. (2) 기본 인덱스(Primary Index)와 보조 인덱스(Secondary Index)의 차이를 구분할 수 있다. (3) 덴스(Dense) 인덱스와 스파스(Sparse) 인덱스의 구조와 트레이드오프를 설명할 수 있다. (4) 클러스터링 인덱스(Clustering Index)의 개념과 효과를 이해한다.

1. 인덱스란 무엇인가?

인덱스(Index)는 파일에서 레코드를 빠르게 찾기 위한 보조 데이터 구조다. 책의 색인(索引)과 동일한 원리다 — 책 전체를 읽지 않고 색인 페이지에서 키워드를 찾아 해당 페이지로 바로 이동하는 것처럼, 인덱스를 통해 전체 파일을 순차 탐색하지 않고 원하는 레코드 위치로 바로 접근할 수 있다.

파일 (10,000 레코드) 인덱스 (작은 보조 파일) ┌──────────────────────┐ ┌──────────────────────────┐ │ REC#0001 | key=1003 │ │ key=1003 → 블록 1, 오프셋 0 │ │ REC#0002 | key=1027 │ │ key=2055 → 블록 3, 오프셋 2 │ │ ... │◄───────│ key=5100 → 블록 8, 오프셋 4 │ │ REC#9999 | key=9998 │ │ ... │ └──────────────────────┘ └──────────────────────────┘ ↓ 인덱스는 작고 정렬됨 이진 탐색으로 빠르게 찾기!

인덱스는 항상 <키값, 포인터> 쌍으로 구성된다. 포인터는 레코드가 저장된 블록 번호나 파일 내 오프셋을 가리킨다. 인덱스 파일 자체는 원본 파일보다 훨씬 작기 때문에, 메모리에 올려두거나 빠르게 이진 탐색할 수 있다.

왜 인덱스가 필요한가? Day 2에서 배운 순차 파일은 이진 탐색으로 O(log B) 블록 접근이 가능하지만, 정렬 키 이외의 필드로 검색하면 전체 파일을 순차 읽기(O(B))해야 한다. 인덱스는 이 문제를 해결하여 어떤 필드에 대해서도 빠른 검색을 가능하게 한다.

2. 기본 인덱스 vs 보조 인덱스

인덱스는 파일의 정렬 키와 일치하는지에 따라 두 종류로 나뉜다.

구분 기본 인덱스 (Primary Index) 보조 인덱스 (Secondary Index)
인덱스 키 파일의 정렬 키(Ordering Key)와 동일 파일의 정렬 키가 아닌 필드
파일 정렬 인덱스 키 기준으로 정렬된 파일 필요 파일 정렬 상태와 무관
인덱스 수 파일당 최대 1개 파일당 여러 개 가능
포인터 대상 각 블록의 첫 레코드 (스파스 방식 일반적) 각 레코드 또는 레코드 집합
활용 예시 학번으로 정렬된 학생 파일 → 학번 인덱스 이름, 학과명 등 추가 검색 필드

2-1. 기본 인덱스의 구조

기본 인덱스는 파일이 이미 정렬되어 있으므로, 모든 레코드마다 인덱스 항목을 만들 필요가 없다. 보통 각 블록의 첫 번째 레코드(앵커 레코드, Anchor Record)에 대해서만 인덱스 항목을 만든다.

데이터 파일 (블록 단위) ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │블록 1 │ │블록 2 │ │블록 3 │ │ key=1003 ←┐ │ │ key=2010 ←┐ │ │ key=3001 ←┐ │ │ key=1027 │ │ │ key=2055 │ │ │ key=3045 │ │ │ key=1089 │ │ │ key=2099 │ │ │ key=3088 │ │ └─────────────┘ └─────────────┘ └─────────────┘ ↑ ↑ ↑ │ │ │ 기본 인덱스 파일 (앵커 레코드만) ┌─────────────────────┐ │ key=1003 → 블록 1 │ ← 항목 1 │ key=2010 → 블록 2 │ ← 항목 2 │ key=3001 → 블록 3 │ ← 항목 3 └─────────────────────┘

인덱스 파일의 크기 = 블록 수만큼의 항목. 원본 파일보다 훨씬 작다. 이진 탐색으로 인덱스를 탐색한 뒤 해당 블록만 읽으면 된다.

2-2. 보조 인덱스의 구조

보조 인덱스는 정렬되지 않은 필드에 대해 만든다. 예를 들어 학번(기본 키) 순으로 정렬된 파일에서 학과명으로 검색하려면 보조 인덱스가 필요하다.

데이터 파일 (학번 기준 정렬) ┌──────────────────────────┐ │ 학번=20001 | 학과=컴퓨터 │ ← 블록 1 │ 학번=20002 | 학과=수학 │ │ 학번=20003 | 학과=컴퓨터 │ ├──────────────────────────┤ │ 학번=20004 | 학과=물리 │ ← 블록 2 │ 학번=20005 | 학과=컴퓨터 │ └──────────────────────────┘ 보조 인덱스 (학과명 기준, 정렬됨) ┌────────────────────────────────────────────┐ │ 학과=물리 → [블록2, 오프셋3] │ │ 학과=수학 → [블록1, 오프셋1] │ │ 학과=컴퓨터 → [블록1, 오프셋0], [블록1, 오프셋2], [블록2, 오프셋4] │ └────────────────────────────────────────────┘
보조 인덱스의 비용 보조 인덱스는 같은 키값을 가진 레코드가 여러 곳에 흩어져 있어, 검색 시 여러 블록을 비순차적으로 접근해야 할 수 있다. 레코드가 n개라면 최대 n번의 블록 접근이 필요할 수 있다 (임의 접근).

3. 덴스 인덱스 vs 스파스 인덱스

인덱스 항목 수가 레코드 수와 같으냐, 블록 수와 같으냐에 따라 두 가지로 나뉜다.

구분 덴스 인덱스 (Dense Index) 스파스 인덱스 (Sparse Index)
항목 수 모든 레코드마다 1개 (레코드 수 = 인덱스 항목 수) 각 블록마다 1개 (블록 수 = 인덱스 항목 수)
인덱스 크기 큼 — 레코드 수에 비례 작음 — 블록 수에 비례
검색 속도 빠름 — 정확한 위치로 바로 이동 약간 느림 — 블록 안에서 추가 탐색 필요
파일 정렬 필요 불필요 (비정렬 파일도 가능) 필수 — 정렬된 파일에서만 의미 있음
적용 사례 보조 인덱스, 소규모 파일 기본 인덱스, 대규모 정렬 파일

덴스 인덱스 구조

데이터 파일 덴스 인덱스 ┌──────────────────┐ ┌──────────────────────┐ │블록 1 │ │ key=1003 → 블록1, 0번 │ │ [0] key=1003 │◄───────│ key=1027 → 블록1, 1번 │ │ [1] key=1027 │◄───────│ key=1089 → 블록1, 2번 │ │ [2] key=1089 │◄───────│ key=2010 → 블록2, 0번 │ ├──────────────────┤ │ key=2055 → 블록2, 1번 │ │블록 2 │ │ ... │ │ [0] key=2010 │◄───────└──────────────────────┘ │ [1] key=2055 │◄─────── 레코드 수 = 인덱스 항목 수 └──────────────────┘

스파스 인덱스 구조

데이터 파일 스파스 인덱스 ┌──────────────────┐ ┌──────────────────────┐ │블록 1 │ │ key=1003 → 블록 1 │ │ [0] key=1003 │◄───────│ key=2010 → 블록 2 │ │ [1] key=1027 │ │ key=3001 → 블록 3 │ │ [2] key=1089 │ └──────────────────────┘ ├──────────────────┤ 블록 수 = 인덱스 항목 수 │블록 2 │ │ [0] key=2010 │◄─── 블록 첫 항목(앵커)만 인덱스에 있음 │ [1] key=2055 │ 찾는 key가 블록 안 어디에 있는지는 │ [2] key=2099 │ 블록 읽은 후 순차 탐색으로 확인 ├──────────────────┤ │블록 3 │ │ [0] key=3001 │◄─── │ ... │ └──────────────────┘
스파스 인덱스 검색 절차
  1. 인덱스에서 이진 탐색: 찾는 key보다 크지 않은 가장 큰 앵커 키를 찾는다.
  2. 그 앵커가 가리키는 블록을 디스크에서 읽는다.
  3. 블록 안에서 순차 탐색으로 원하는 레코드를 찾는다.

4. 클러스터링 인덱스

클러스터링 인덱스(Clustering Index)는 파일의 레코드가 특정 필드(클러스터링 필드) 기준으로 물리적으로 묶여(클러스터) 저장되어 있을 때 만드는 인덱스다.

기본 인덱스가 유일한(Unique) 정렬 키에 대한 인덱스라면, 클러스터링 인덱스는 중복 값을 허용하는 정렬 필드에 대한 인덱스다.

데이터 파일 (학과별로 클러스터링) ┌─────────────────────────────────────┐ │블록 1 — 컴퓨터과 │ │ 학번=20001 | 학과=컴퓨터 | 이름=김A │ │ 학번=20003 | 학과=컴퓨터 | 이름=이B │ │ 학번=20005 | 학과=컴퓨터 | 이름=박C │ ├─────────────────────────────────────┤ │블록 2 — 수학과 │ │ 학번=20002 | 학과=수학 | 이름=최D │ │ 학번=20007 | 학과=수학 | 이름=정E │ ├─────────────────────────────────────┤ │블록 3 — 물리과 │ │ 학번=20004 | 학과=물리 | 이름=강F │ │ 학번=20006 | 학과=물리 | 이름=조G │ └─────────────────────────────────────┘ 클러스터링 인덱스 (학과 기준) ┌──────────────────────┐ │ 학과=물리 → 블록 3 │ │ 학과=수학 → 블록 2 │ │ 학과=컴퓨터 → 블록 1 │ └──────────────────────┘

같은 학과 레코드들이 물리적으로 같은 블록에 모여 있으므로, "컴퓨터과 학생 전체 조회" 같은 범위 쿼리에서 블록 접근을 최소화할 수 있다.

클러스터링 인덱스의 장점
  • 같은 값을 가진 레코드가 같은 블록(혹은 인접 블록)에 있어 범위 검색이 매우 효율적
  • 한 번의 블록 접근으로 여러 레코드를 동시에 가져올 수 있음
클러스터링 인덱스의 제약
  • 파일은 클러스터링 필드 기준으로 물리적으로 정렬되어 있어야 한다
  • 한 파일에 클러스터링 인덱스는 하나만 가능 (물리 순서가 하나이므로)
  • 클러스터링 필드로 정렬하면 다른 필드 기준 순차 접근 성능은 떨어짐

5. 인덱스 유형 총정리

인덱스 유형 파일 정렬 조건 항목 수 주요 특징
기본 인덱스
(Primary)		 정렬 키와 동일 블록 수 (스파스) 파일당 1개, 앵커 레코드 기반
보조 인덱스
(Secondary)		 무관 (비정렬 가능) 레코드 수 (덴스) 파일당 여러 개, 임의 접근 발생
클러스터링
(Clustering)		 중복 허용 정렬 필드 고유 값 수 (스파스) 범위 검색에 최적, 파일당 1개
덴스
(Dense)		 무관 레코드 수 빠르지만 인덱스 크기가 큼
스파스
(Sparse)		 정렬 필수 블록 수 작지만 추가 블록 내 탐색 필요
📖 Primary Source Folk, Zoellick & Riccardi, File Structures — Chapter 8: Indexing. 기본 인덱스, 보조 인덱스, 클러스터링 인덱스를 예제 파일과 함께 상세히 설명한다. 특히 “dense vs sparse” 비교 표와 “secondary index pointer chains” 절을 집중적으로 읽어라.
