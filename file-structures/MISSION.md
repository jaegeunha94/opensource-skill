# 파일구조 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 개발자 기술 면접에서 파일구조 지식을 **디스크 기반 저장, 인덱스 설계, 대용량 데이터 처리, DBMS 내부 구조, 파일 시스템 성능**과 연결해 설명할 수 있도록 훈련한다.

파일구조 면접은 B-트리 삽입 과정을 외웠는지보다, 왜 디스크/SSD에서는 순차 접근과 랜덤 접근 비용이 다르고, 왜 B+Tree가 DB 인덱스에 적합하며, 왜 외부 정렬과 버퍼링이 대용량 처리에서 중요한지를 실무 언어로 설명할 수 있는지를 본다.

## 평가 대상 역량

| 역량 | 면접에서 확인하려는 것 |
|------|------------------------|
| 저장장치 모델 | block I/O, sequential/random access, page, buffer 개념을 이해하는가 |
| 파일 조직 | sequential, indexed, hashed file의 장단점과 선택 기준을 말하는가 |
| 인덱스 구조 | dense/sparse index, primary/secondary index, clustered index를 구분하는가 |
| 트리 인덱스 | B-Tree, B+Tree의 높이, fanout, range scan 장점을 설명하는가 |
| 해싱 | hash collision, overflow, rehashing, range query 한계를 이해하는가 |
| 외부 정렬 | 메모리보다 큰 데이터를 정렬하는 과정과 I/O 비용을 설명하는가 |
| 실무 연결 | DB 인덱스, 로그 저장, 검색, batch processing, 파일 시스템 설계와 연결하는가 |

## Interview 레슨 작성 규칙

각 Day 레슨은 반드시 **면접 질문 중심**으로 작성한다.

1. **면접 질문**
   - 저장 구조 선택과 성능 판단을 요구하는 질문으로 시작한다.
   - 예: "왜 DB 인덱스는 보통 B+Tree를 사용하고, 해시 테이블만으로 대체하지 않을까요?"
2. **전제 개념**
   - 어려운 개념은 먼저 쉬운 한국어로 설명한다.
   - 레코드, 블록, 페이지, 인덱스, fanout, range scan 같은 용어를 필요한 배경부터 풀어쓴다.
3. **왜 면접에서 묻는가**
   - 대용량 데이터 처리와 저장 구조 선택의 원리를 이해하는지 확인하는 이유를 적는다.
4. **실무 연결**
   - DB 인덱스, 로그 파일, 검색 기능, batch job, 데이터 마이그레이션, 파일 시스템과 연결한다.
5. **답변 사고 순서**
   - 접근 패턴 확인 -> 저장장치 비용 모델 확인 -> 후보 구조 비교 -> I/O 비용 추정 -> 운영 trade-off 설명 순서로 답변하도록 안내한다.
6. **핵심 답변 예시**
   - 면접에서 바로 말할 수 있는 답변을 짧은 버전과 깊이 있는 버전으로 제공한다.
7. **Trade-off**
   - 조회, 삽입, 삭제, 범위 검색, 저장 공간, 재구성 비용, 구현 복잡도를 비교한다.
8. **흔한 오해**
   - "해시가 O(1)이니 항상 B+Tree보다 빠르다", "파일은 그냥 byte 배열이다", "메모리 알고리즘을 디스크에 그대로 쓰면 된다" 같은 오해를 바로잡는다.
9. **Follow-up 질문**
   - 면접관이 더 깊게 물을 질문과 답변 방향을 포함한다.
10. **10년차 이상 답변 기준**
   - 자료구조 이름 암기가 아니라 저장장치 특성과 접근 패턴 기반 선택이 드러나는지 평가한다.

## 완료 기준

- 파일구조를 DB 인덱스와 대용량 데이터 처리 문제로 연결해 설명할 수 있다.
- B+Tree, hash, sequential file의 선택 기준을 접근 패턴으로 설명할 수 있다.
- I/O 비용과 운영 복잡도를 포함해 trade-off를 말할 수 있다.
- 약한 답변과 강한 답변의 차이를 스스로 설명할 수 있다.
