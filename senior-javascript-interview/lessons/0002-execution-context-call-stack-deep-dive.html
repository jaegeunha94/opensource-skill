1. 핵심 면접 질문

Primary Question

"다음 코드에서 console.log(a)는 왜 에러가 나고, console.log(b)는 왜 undefined가 나올까요?"

console.log(a);
let a = 1;

console.log(b);
var b = 2;
Variant 1

"함수 선언문(function foo(){})과 함수 표현식(const foo = function(){})의 호이스팅 차이를 설명해주세요."

Variant 2

"재귀 함수를 호출하다가 Maximum call stack size exceeded 에러를 만난 적이 있나요? 왜 발생하고, 어떻게 고쳤나요?"

Variant 3

"함수가 호출될 때 실행 컨텍스트 내부에서 어떤 일이 순서대로 일어나는지 단계별로 설명해주세요."

2. 왜 면접에서 묻는가

이 질문은 "호이스팅이라는 단어를 아는가"가 아니라 "코드가 실행되기 전에 엔진이 무엇을 미리 준비해두는지 원리로 아는가"를 테스트합니다. "함수랑 var는 위로 끌어올려진다"는 표현만 외운 사람은 let/const의 TDZ 앞에서 바로 막힙니다.

면접관이 이 질문으로 평가하는 것:

  • Creation Phase와 Execution Phase를 구분하는가 — 호이스팅이 "코드가 물리적으로 이동하는 것"이 아니라 "실행 전 메모리 등록 단계"의 결과라는 것을 아는가.
  • 선언 종류별 차이를 정확히 아는가 — var/let/const/함수 선언/함수 표현식/class가 Creation Phase에서 각각 다르게 취급된다는 것을 설명할 수 있는가.
  • 런타임 에러 메시지를 근거로 추론할 수 있는가 — ReferenceError와 undefined라는 서로 다른 결과가 왜 나오는지, 프로덕션 에러 로그를 볼 때 이 구분이 왜 중요한지 아는가.
  • Call Stack의 물리적 한계를 이해하는가 — 재귀가 실제로 메모리 어디에 무엇을 쌓는지 알고, 대용량 데이터(깊은 트리, 중첩 JSON)를 다룰 때 왜 재귀가 위험할 수 있는지 판단할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "let은 호이스팅이 안 돼요"라고 단정적으로 말하는 답변입니다. 이는 절반만 맞습니다 — let도 Creation Phase에서 바인딩 자체는 등록되지만(그래서 스코프 체인 상 존재는 함), 초기화가 되지 않은 상태(TDZ)로 남아있다가 실제 선언문에 도달해야 초기화됩니다. 시니어는 "호이스팅 여부"가 아니라 "바인딩 등록과 초기화 시점의 차이"로 설명합니다.

3. 전제 개념 — 쉬운 한국어로

Execution Context(실행 컨텍스트)란 무엇인가

실행 컨텍스트는 "지금 실행 중인 코드가 참조할 수 있는 모든 것을 담아둔 상자"입니다. 여기에는 (1) 어떤 변수들이 있는지, (2) this가 무엇을 가리키는지, (3) 바깥 스코프는 어디인지가 들어 있습니다. JS 엔진은 코드를 실행할 때마다 — 전역 코드가 시작될 때, 함수가 호출될 때, eval이 실행될 때 — 이 상자를 하나씩 새로 만듭니다.

왜 "생성 단계"와 "실행 단계"로 나뉘는가

이사를 예로 들면 이해하기 쉽습니다. 이사 갈 집(실행 컨텍스트)에 들어가기 전, 미리 방 배치도를 그리고 어느 방에 어떤 가구가 들어갈지 표시해두는 단계(Creation Phase)가 있고, 그다음 실제로 가구를 옮기고 짐을 채워 넣는 단계(Execution Phase)가 있습니다. Creation Phase에서는 "이 자리에 a라는 변수가 있을 것이다"라는 자리만 미리 잡아두고, Execution Phase에 가서야 실제 값이 채워집니다. 호이스팅은 바로 이 "자리를 미리 잡아두는 행위"가 겉으로 보이는 현상입니다 — 코드가 실제로 위로 이동하는 것이 아닙니다.

var와 let/const는 "자리 잡기"를 다르게 한다

Creation Phase에서 var로 선언된 변수는 자리를 잡으면서 곧바로 undefined 값으로 초기화됩니다. 그래서 실제 선언문 이전에 접근해도 에러 없이 undefined가 나옵니다. 반면 let/const(그리고 class)는 자리는 잡히지만 초기화되지 않은 상태로 남습니다. 이 "자리는 있지만 아직 쓸 수 없는" 구간을 TDZ(Temporal Dead Zone, 일시적 사각지대)라고 부릅니다. TDZ 상태에서 접근하면 엔진이 "이 변수, 존재는 알지만 아직 준비가 안 됐다"는 의미로 ReferenceError: Cannot access 'a' before initialization을 던집니다.

Call Stack — 실행 컨텍스트를 쌓아 올리는 자료구조

함수가 호출될 때마다 새 실행 컨텍스트가 만들어지고, 이것이 Call Stack이라는 후입선출(LIFO) 구조 위에 쌓입니다. 함수가 return하면 그 함수의 실행 컨텍스트가 스택에서 제거(pop)됩니다. 이 스택은 힙(heap) 메모리와 별개로, 크기가 제한된 전용 메모리 영역입니다. 재귀 함수가 종료 조건 없이 계속 자신을 호출하면 이 스택에 실행 컨텍스트가 무한정 쌓이다가 한계를 초과해 RangeError: Maximum call stack size exceeded가 발생합니다.

왜 이 개념이 Day 1 다음에 필요한가

Day 1에서 "코드가 어떤 순서로 실행되는가"를 다뤘다면, Day 2는 그 실행이 시작되기 직전에 엔진이 무엇을 준비해두는지를 다룹니다. 이 Creation Phase 개념은 Day 3(Scope Chain)과 Day 4(Closure)의 기반이 됩니다 — 클로저가 "기억하는" 대상이 바로 이 실행 컨텍스트 안의 변수 환경(Variable Environment)이기 때문입니다.

4. Execution Context의 2단계 생명주기

모든 실행 컨텍스트(전역이든 함수든)는 반드시 이 두 단계를 거칩니다.

단계 하는 일 결과
Creation Phase
코드 실행 직전		 1) this 바인딩 결정
2) Lexical Environment 생성 (스코프 체인 연결)
3) 코드를 한 번 스캔하며 변수/함수 선언을 미리 등록 		변수·함수의 "자리"가 메모리에 먼저 잡힘 (= 호이스팅으로 관찰되는 현상)
Execution Phase
코드가 위에서 아래로 실제 실행		 1) 변수에 실제 값 대입
2) 함수 호출 시 새 실행 컨텍스트 생성 → Call Stack에 push
3) 함수 종료 시 Call Stack에서 pop 		실제 로직이 수행되고 값이 채워짐

실무에서 이 2단계 구분이 왜 중요한가

번들러(Webpack, esbuild 등)가 코드를 압축·재배치하면서 변수 선언 순서가 바뀌는 경우가 있습니다. 이때 로컬 개발 환경에서는 문제없다가 프로덕션 빌드에서만 TDZ 에러가 터지는 사례가 실제로 자주 발생합니다. Creation Phase에서 "자리는 잡히지만 초기화는 안 된다"는 원리를 알아야, 왜 특정 빌드 설정(예: 모듈 순환 참조, 트리쉐이킹으로 인한 순서 변경)에서만 이런 에러가 재현되는지 진단할 수 있습니다.

5. 호이스팅 분류표 — 선언 종류별 동작 차이

이 표를 원리(Creation Phase에서 "자리만 잡히는가" vs "값까지 채워지는가")로 이해하면, 새로운 선언 키워드가 나와도 스스로 분류할 수 있습니다.

선언 종류 Creation Phase 동작 선언 이전 접근 시 실무 함정
var 자리 등록 + undefined로 즉시 초기화 undefined (에러 없음) 오타로 인한 미선언 변수 사용을 조용히 숨김 — 버그가 런타임 늦게까지 드러나지 않음
let / const 자리 등록만, 초기화는 실제 선언문 도달 시 ReferenceError (TDZ) 에러가 빨리, 명확하게 드러남 — 대신 모듈 순환 참조 시 TDZ 에러의 원인이 되기도 함
함수 선언문 (function foo(){}) 자리 등록 + 함수 전체(본문 포함)를 즉시 초기화 정상 호출 가능 선언 위치와 무관하게 파일 어디서든 호출 가능해 코드 흐름 추적을 어렵게 만들 수 있음
함수 표현식 (const foo = function(){}) 변수 부분만 let/const/var 규칙을 그대로 따름 선언 키워드에 따라 다름 (TDZ 또는 undefined) "함수니까 호이스팅될 것"이라고 착각하고 선언 전에 호출해 에러를 만남
class 자리 등록만, 초기화는 let/const와 동일하게 TDZ 적용 ReferenceError (TDZ) "클래스도 함수 선언처럼 끌어올려질 것"이라는 오해 — 실제로는 let과 동일하게 취급

6. Call Stack과 스택 오버플로우

재귀 함수를 호출할 때마다 새 실행 컨텍스트가 Call Stack에 쌓입니다. 종료 조건이 없거나, 입력 데이터가 예상보다 훨씬 깊으면(예: 매우 깊게 중첩된 JSON, 매우 긴 연결 리스트) 스택 프레임이 한계를 초과합니다.

function countDown(n) {
  if (n <= 0) return;
  countDown(n - 1); // 매번 새 실행 컨텍스트가 Call Stack에 push
}

countDown(100000);
// RangeError: Maximum call stack size exceeded

실무 연결 — 왜 이게 프로덕션 버그가 되는가

테스트할 때는 얕은 트리(댓글 3단 중첩, 작은 JSON)로만 확인하고 배포했는데, 실제 사용자 데이터에서 훨씬 깊은 중첩(수만 단계로 중첩된 댓글, 재귀적으로 참조하는 조직도, 매우 큰 재귀적 파싱 대상)을 만나면 스택 오버플로우가 발생합니다. 이런 버그는 로컬 재현이 어렵고, 특정 사용자 데이터에서만 터지는 특성 때문에 진단이 까다롭습니다.

실무 대응 방향: 재귀 깊이가 데이터에 따라 무한정 커질 수 있는 로직(트리 순회, 깊은 객체 복제)은 명시적인 스택 자료구조를 쓰는 반복문(iterative) 형태로 바꾸는 것이 안전합니다. JavaScript(V8 포함 대부분의 엔진)는 꼬리 호출 최적화(Tail Call Optimization, TCO)를 스펙에는 포함하고 있지만(ES2015) 실질적으로 구현한 주요 엔진이 거의 없으므로, "꼬리 재귀로 작성하면 스택이 안 쌓인다"는 가정에 의존하면 안 됩니다.

7. 답변 사고 순서

  1. Creation Phase와 Execution Phase를 먼저 구분해 언급합니다. 실행 전에 "자리 잡기" 단계가 있다는 전제를 깝니다.
  2. 문제의 각 변수/함수가 어떤 선언 종류인지 식별합니다. var인지 let/const인지 함수 선언인지 표현식인지 구분합니다.
  3. Creation Phase에서 각각 어떻게 등록되는지 적용합니다. "자리만 잡히는가, 값까지 채워지는가"를 표(5절) 기준으로 판단합니다.
  4. 접근 시점이 초기화 이전인지 이후인지 확인합니다. 초기화 이전이면 TDZ(에러) 또는 undefined, 이후면 실제 값입니다.
  5. 결과와 에러 메시지 종류까지 구체적으로 말합니다. 단순히 "에러가 난다"가 아니라 ReferenceError인지 값이 undefined인지 구분해서 답합니다.

8. 핵심 답변 예시

면접관: "다음 코드에서 console.log(a)는 왜 에러가 나고, console.log(b)는 왜 undefined가 나올까요?"

console.log(a);
let a = 1;

console.log(b);
var b = 2;
STRONG ANSWER

이 코드가 담긴 실행 컨텍스트(여기서는 전역 컨텍스트)는 실제 실행 전에 Creation Phase를 거칩니다. 이 단계에서 엔진은 코드를 한 번 스캔하며 a와 b 모두 이 스코프에 존재한다는 "자리"를 미리 등록합니다. 하지만 등록 방식이 선언 키워드에 따라 다릅니다.


b는 var로 선언됐기 때문에, Creation Phase에서 자리가 잡히면서 곧바로 undefined라는 값으로 초기화됩니다. 그래서 실제 var b = 2라는 대입문에 도달하기 전에 console.log(b)를 실행해도 에러 없이 undefined가 출력됩니다.


반면 a는 let으로 선언됐기 때문에, Creation Phase에서 자리는 잡히지만 값으로 초기화되지는 않습니다. 실제 let a = 1 선언문에 도달해야 초기화가 일어납니다. 그 전까지는 a가 "존재는 하지만 아직 접근할 수 없는" 상태인데, 이 구간을 TDZ(Temporal Dead Zone)라고 부릅니다. 그래서 console.log(a)는 ReferenceError: Cannot access 'a' before initialization을 던집니다 — 변수가 아예 없다는 의미의 is not defined가 아니라, "존재는 알지만 아직 준비가 안 됐다"는 다른 종류의 에러라는 점이 핵심입니다.

이 답변이 좋은 이유:

  • Creation/Execution 2단계 모델을 먼저 제시하고 그 위에서 두 변수를 각각 설명함
  • "자리 등록"과 "값 초기화"를 명확히 구분함
  • 단순히 결과("에러 난다", "undefined 나온다")만 말하지 않고 정확한 에러 메시지 종류까지 짚음
  • TDZ가 "존재하지 않음"이 아니라 "아직 초기화되지 않음"이라는 의미론적 차이를 정확히 설명함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"let은 호이스팅이 안 되고 var는 호이스팅이 되니까 그래요."

이 답변이 약한 이유

  • 사실 let도 호이스팅됩니다 — 스코프 체인 상에는 이미 존재를 등록하고 있고, 그래서 "밖에 있는 같은 이름의 변수를 실수로 참조하는" 섀도잉 관련 버그도 이 등록 때문에 생깁니다. "안 된다/된다"는 이분법은 정확하지 않습니다.
  • 왜 a는 ReferenceError이고 b는 undefined인지, 즉 두 결과의 차이가 나는 이유를 설명하지 못했습니다.
  • TDZ라는 개념 자체를 언급하지 않아 후속 질문("그럼 TDZ가 뭔가요?")에 막힐 가능성이 높습니다.
WEAK ANSWER — 또 다른 패턴

"자바스크립트 엔진이 코드를 실행하기 전에 변수 선언문을 파일 맨 위로 옮겨서 실행하기 때문이에요."

이 답변이 약한 이유

  • "코드가 물리적으로 이동한다"는 설명은 흔한 오해이자 부정확한 멘탈 모델입니다. 실제로는 코드 이동이 아니라 Creation Phase에서 메모리에 미리 등록되는 것입니다.
  • 이 멘탈 모델로는 함수 선언문과 함수 표현식의 차이, class가 TDZ를 따르는 이유 등을 설명할 수 없습니다.
STRONG ANSWER 요소 체크리스트
  • Creation Phase / Execution Phase 2단계 모델을 먼저 언급한다
  • "코드 이동"이 아니라 "메모리에 자리를 미리 등록"이라는 정확한 표현을 쓴다
  • var는 즉시 undefined로 초기화, let/const/class는 TDZ라는 차이를 명시한다
  • ReferenceError와 undefined라는 서로 다른 결과를 근거와 함께 구분한다
  • 함수 선언문과 함수 표현식의 차이도 같은 원리(자리 등록 방식의 차이)로 설명할 수 있다

10. Senior-level Follow-up Q&A

Q: 함수 선언문은 왜 본문까지 통째로 호이스팅되는데, 함수 표현식은 그렇지 않나요?

모범 답변: 함수 선언문(function foo(){})은 ECMAScript 스펙상 Creation Phase에서 "함수 자체"가 하나의 완결된 단위로 등록되기 때문입니다 — 이름과 본문이 함께 메모리에 준비됩니다. 반면 함수 표현식(const foo = function(){})은 "표현식의 결과를 변수에 대입"하는 문장이기 때문에, Creation Phase에서는 변수 foo의 "자리"만 (선언 키워드 규칙에 따라) 등록되고, 실제 함수 객체가 만들어져 대입되는 것은 Execution Phase에서 그 줄에 도달했을 때입니다. 즉 "함수라서 특별 취급"이 아니라 "선언문이냐 대입 표현식이냐"의 차이입니다.

Q: TDZ는 성능에도 영향을 주나요, 아니면 순수하게 에러를 미리 잡기 위한 안전장치인가요?

모범 답변: TDZ 자체가 별도의 런타임 비용을 크게 유발하지는 않습니다. 핵심 의도는 성능이 아니라 안전성입니다 — var의 "선언 전에도 조용히 undefined로 접근 가능"한 동작이 실무에서 오타·순서 버그를 늦게 드러내는 원인이 되어 왔기 때문에, let/const는 의도적으로 "준비 안 된 상태에서 접근하면 즉시 명확한 에러를 던진다"는 설계를 택한 것입니다. 실패를 늦추는 대신 빠르게 드러내는(fail-fast) 설계 철학의 산물이라고 설명하면 좋습니다.

Q: Call Stack 크기 제한은 왜 존재하나요? 늘릴 수 있나요?

모범 답변: Call Stack은 힙과 분리된 고정 크기 메모리 영역으로, 각 스택 프레임(실행 컨텍스트)마다 지역 변수·매개변수·반환 주소 등을 저장할 공간이 필요합니다. 이 영역을 무제한으로 두면 잘못된 재귀 하나가 프로세스 전체의 메모리를 소진시켜 다른 정상 동작까지 위협할 수 있기 때문에 엔진이 상한선을 둡니다. Node.js에서는 --stack-size 플래그로 조정할 수 있지만, 이는 근본 해결책이 아니라 임시방편입니다 — 애초에 입력 크기에 비례해 재귀 깊이가 무한정 늘어날 수 있는 로직이라면, 반복문 기반 순회로 바꾸는 것이 올바른 해결 방향입니다.

Q: 꼬리 호출 최적화(TCO)로 재귀를 안전하게 만들 수 있다고 들었는데, 실무에서 믿고 써도 되나요?

모범 답변: ES2015 스펙은 Proper Tail Calls(PTC)를 정의하고 있지만, 실질적으로 주요 브라우저 엔진(V8 포함) 대부분이 이를 구현하지 않았습니다. Safari(JavaScriptCore)만 제한적으로 지원한 적이 있습니다. 따라서 "꼬리 재귀 형태로 작성하면 엔진이 알아서 스택을 재사용해줄 것"이라는 가정에 프로덕션 코드를 의존하면 안 됩니다. 스택 오버플로우 위험이 있는 재귀는 트램폴린(trampoline) 패턴이나 명시적 스택을 쓰는 반복문으로 직접 바꾸는 것이 실무에서 신뢰할 수 있는 해법입니다.

Q: for (let i = 0; ...)에서 매 반복마다 i가 새로운 바인딩을 갖는다는 것도 Execution Context와 관련이 있나요?

모범 답변: 관련이 있습니다. let으로 선언된 반복 변수는 스펙상 매 반복(iteration)마다 새로운 렉시컬 환경(그리고 그 안에 새 바인딩)을 생성하도록 정의되어 있습니다. 그래서 반복문 안에서 클로저를 만들면 각 클로저가 서로 다른 i 값을 "기억"하게 됩니다. var는 함수 스코프(또는 전역 스코프) 하나만 쓰기 때문에 모든 반복이 같은 바인딩을 공유합니다. 이 차이는 Day 4(클로저)에서 "반복문 클로저 버그"로 더 깊게 다룰 주제이지만, 근본 원인은 지금 배운 것처럼 "각 실행 컨텍스트/환경이 언제, 몇 개 생성되는가"에 있습니다.

11. 흔한 오해

오해 1

"호이스팅은 코드가 실제로 파일 맨 위로 이동하는 것이다"

코드는 물리적으로 이동하지 않습니다. Creation Phase에서 엔진이 미리 스캔하며 변수/함수의 "자리"를 메모리에 등록해두는 것이 호이스팅처럼 관찰될 뿐입니다. 이 오해로는 var와 let의 동작 차이를 설명할 수 없습니다.

오해 2

"let/const는 호이스팅되지 않는다"

정확히는 let/const도 Creation Phase에서 바인딩(자리)이 등록됩니다. 다만 그 자리가 초기화되지 않은 채로 TDZ에 놓여 있다가, 실제 선언문에 도달해야 초기화될 뿐입니다. "호이스팅이 안 된다"가 아니라 "초기화 시점이 다르다"가 정확한 설명입니다.

오해 3

"스택 오버플로우는 메모리가 부족해서 생기는 문제다"

힙 메모리가 충분히 남아 있어도 스택 오버플로우는 발생할 수 있습니다. Call Stack은 힙과 별개로 크기가 고정된 전용 영역이기 때문에, 재귀 깊이(쌓이는 실행 컨텍스트 개수)가 이 한계를 넘으면 다른 메모리 여유와 무관하게 RangeError가 발생합니다.

오해 4

"함수 표현식도 함수니까 함수 선언문처럼 어디서든 먼저 호출할 수 있다"

함수 표현식은 "함수 정의"가 아니라 "변수 대입문"으로 취급됩니다. 따라서 그 변수가 var/let/const 중 무엇으로 선언됐는지에 따라 선언 이전 접근 시 동작(undefined 또는 TDZ 에러)이 완전히 달라집니다.

오해 5

"TDZ는 let/const만의 특수 예외 규칙이다"

TDZ는 예외가 아니라, "바인딩은 Creation Phase에 등록되지만 초기화는 실행 흐름이 실제로 그 지점에 도달해야 일어난다"는 일관된 원리의 자연스러운 결과입니다. class 선언도 동일한 원리로 TDZ의 적용을 받습니다.
