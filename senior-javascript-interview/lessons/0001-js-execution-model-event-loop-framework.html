1. 핵심 면접 질문

Primary Question

"다음 코드의 출력 순서를 설명하고, 왜 그 순서가 나오는지 이유를 말씀해주세요."

console.log('1');

setTimeout(() => console.log('2'), 0);

Promise.resolve().then(() => console.log('3'));

console.log('4');
Variant 1

"setTimeout(fn, 0)은 0ms 뒤에 즉시 실행되는 건가요? 아니라면 왜 그런가요?"

Variant 2

"무거운 동기 함수(예: 5초 걸리는 반복문)가 실행되는 동안 화면이 멈추는 이유를 이벤트 루프 관점에서 설명해보세요."

Variant 3

"async/await과 Promise.then이 섞여 있을 때 실행 순서를 어떻게 예측하시나요?"

2. 왜 면접에서 묻는가

이 질문은 "비동기 API를 써봤는가"가 아니라 "JavaScript가 코드를 실제로 어떤 순서로 실행하는지 원리로 설명할 수 있는가"를 테스트합니다. setTimeout과 Promise를 수백 번 써봤어도, 왜 특정 순서로 실행되는지 원리를 설명하지 못하는 개발자가 실제로 매우 많습니다.

면접관이 이 질문으로 평가하는 것:

  • 실행 모델을 원리로 아는가 — 콜스택, Web API, 태스크 큐, 마이크로태스크 큐가 실제로 어떻게 상호작용하는지 그릴 수 있는가, 아니면 "Promise가 항상 먼저"라는 패턴만 암기했는가.
  • 싱글 스레드라는 제약을 이해하는가 — JavaScript 엔진이 한 번에 하나의 코드만 실행한다는 사실이 실무에서 어떤 결과(UI 멈춤, 렌더링 지연)로 이어지는지 연결할 수 있는가.
  • 비동기 버그를 진단할 기초 체력이 있는가 — 프로덕션에서 "이 콜백이 예상보다 늦게/빠르게 실행돼요" 같은 버그를 만났을 때, 실행 모델을 근거로 원인을 추론할 수 있는가.
  • 새로운 API를 만나도 같은 원리를 적용할 수 있는가 — queueMicrotask, MutationObserver, requestAnimationFrame 같은 처음 보는 API가 나와도 "이건 어느 큐에 들어갈까"를 원리로 추론할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "Promise가 setTimeout보다 우선순위가 높아서요" 처럼 결론은 맞지만 이유를 설명하지 못하는 답변입니다. 시니어는 콜스택 → Web API → 큐 → 이벤트 루프라는 전체 그림을 근거로 순서를 유도해서 설명합니다.

3. 전제 개념 — 쉬운 한국어로

JavaScript는 왜 "싱글 스레드"인가

JavaScript 엔진은 한 순간에 딱 하나의 코드 조각만 실행할 수 있습니다. 이것을 Call Stack(콜스택)이라고 부르는 자료구조 하나로 관리합니다. 콜스택은 접시를 쌓아 올리는 것과 같습니다 — 함수가 호출되면 접시(실행 컨텍스트)가 하나 쌓이고, 함수가 끝나면(return) 맨 위 접시부터 빠집니다. 콜스택이 완전히 비어야만 "다음 할 일"을 시작할 수 있습니다.

비동기 API는 어떻게 "동시에" 실행되는 것처럼 보이는가

JS 엔진 자체는 싱글 스레드지만, setTimeout, fetch, DOM 이벤트 같은 것은 JS 엔진이 직접 처리하지 않습니다. 브라우저(또는 Node.js의 libuv)가 별도로 처리해줍니다. 식당에 비유하면, 콜스택은 "한 번에 한 손님만 응대하는 웨이터"이고, 오래 걸리는 요리(비동기 작업)는 웨이터가 직접 만들지 않고 주방(브라우저의 Web API 영역)에 맡깁니다. 웨이터는 요리가 끝나길 기다리지 않고 바로 다음 손님(다음 줄의 코드)을 응대합니다.

콜백은 끝나면 어디로 가는가 — 큐(Queue)

주방(Web API)에서 요리(비동기 작업)가 끝나면, 그 결과를 처리할 콜백 함수가 곧바로 실행되는 것이 아니라 대기열(Queue)에 줄을 섭니다. 이 대기열은 하나가 아니라 두 종류입니다.

  • Macrotask Queue(콜백 큐/태스크 큐) — setTimeout, setInterval, DOM 이벤트 콜백, I/O 콜백이 줄 서는 "일반 대기열"입니다.
  • Microtask Queue — Promise.then/catch/finally, async 함수의 await 이후 코드, queueMicrotask가 줄 서는 "VIP 대기열"입니다. 이 큐는 항상 Macrotask보다 먼저, 그리고 완전히 빌 때까지 처리됩니다.

이벤트 루프 — 웨이터의 반복 행동

Event Loop(이벤트 루프)는 특별한 코드가 아니라 하나의 단순한 반복 규칙입니다: "콜스택이 비었는가? 비었다면 Microtask 큐를 전부 처리한다. 그다음 Macrotask 큐에서 딱 하나만 꺼내 실행한다. 그리고 다시 처음부터 반복한다." 이 반복 규칙 하나가 JavaScript의 모든 비동기 실행 순서를 결정합니다.

왜 이 개념부터 시작하는가

클로저, 프로토타입, this, 모듈, 메모리 누수 등 앞으로 다룰 모든 주제는 결국 "이 코드가 언제, 어떤 순서로 실행되는가"라는 실행 모델 위에서 벌어지는 일입니다. 실행 모델을 먼저 원리로 잡아두면, 이후 레슨에서 나오는 개념들이 훨씬 쉽게 연결됩니다.

4. JavaScript 실행 모델 전체 지도

실무에서 코드 한 줄이 실행되는 여정은 다음 5개 구성 요소를 순서대로(또는 반복해서) 거칩니다. 이 지도는 이후 Day 8~11에서 각 구성 요소를 깊게 다룰 때의 기준점 역할을 합니다.

구성 요소 역할 대표 예시 흔한 오해
Call Stack 현재 실행 중인 실행 컨텍스트를 LIFO(후입선출)로 관리 일반 함수 호출, 동기 코드 전체 "함수가 여러 개 동시에 실행된다"는 착각 — 실제로는 하나씩 순차 실행
Web API / Node API JS 엔진 밖에서 비동기 작업을 대신 처리 (타이머, 네트워크, DOM 이벤트 감지) setTimeout, fetch, addEventListener "비동기 함수도 JS 엔진이 병렬로 실행한다"는 착각 — 실제로는 브라우저/Node가 처리
Macrotask Queue Web API 작업이 끝난 콜백이 순서대로 대기 setTimeout 콜백, click 이벤트 콜백 "큐에 들어가는 즉시 실행된다"는 착각 — 콜스택이 비고, 순번이 될 때까지 대기
Microtask Queue Promise 관련 후속 작업이 대기 — Macrotask보다 항상 먼저, 전부 소진될 때까지 처리 Promise.then, await 이후 코드, queueMicrotask "microtask 하나 처리하면 바로 macrotask로 넘어간다"는 착각 — 실제로는 큐가 빌 때까지 반복
Event Loop 콜스택이 비는 순간을 계속 감시하고, Microtask → Macrotask 순서로 큐를 처리하는 반복 규칙 엔진 내부 동작 (직접 호출하는 API 없음) "이벤트 루프는 별도의 스레드다"는 착각 — 실제로는 엔진의 실행 규칙일 뿐

이 지도가 실무에서 중요한 이유

"이 콜백이 왜 이 타이밍에 실행되지 않았지?"라는 버그를 만났을 때, 이 5개 구성 요소 중 어디서 문제가 생겼는지 특정하지 않으면 원인 없는 코드를 계속 고치게 됩니다. 예를 들어 실제 원인이 "콜스택이 무거운 동기 작업으로 계속 차 있어서 이벤트 루프가 macrotask로 못 넘어간 것"인데, 콜백 함수 코드 자체를 아무리 고쳐도 해결되지 않습니다.

5. 6단계 실행 순서 예측 프레임워크

시니어가 setTimeout/Promise/async가 섞인 코드를 받았을 때 실제로 따르는 순서입니다. 이후 커리큘럼(Day 2~20)의 비동기·성능·디버깅 주제 대부분은 이 프레임워크의 한 단계를 깊게 파는 것과 같습니다.

  • 1
    동기 코드에 순서 번호를 매긴다 (Synchronous First Pass)

    위에서 아래로, 콜스택에 쌓였다 빠지는 코드(함수 호출, 일반 console.log)를 먼저 실행 순서대로 표시합니다. 비동기 API 호출문 자체(예: setTimeout(fn, 0)이라는 "호출")는 동기적으로 실행되지만, fn의 내용은 지금 실행되지 않는다는 것을 구분합니다.

  • 2
    비동기 호출을 "위임" 지점으로 표시한다 (Delegation Point)

    setTimeout, fetch, addEventListener 등을 만나면, 그 콜백이 Web API로 넘겨지는 지점이라고 표시합니다. 이 시점에서 콜백 자체는 실행되지 않고 등록만 됩니다.

  • 3
    콜백이 어느 큐로 가는지 분류한다 (Queue Classification)

    타이머/이벤트 콜백 → Macrotask 큐, Promise.then/await 이후 코드 → Microtask 큐로 분류합니다. (6절의 분류표를 기준으로 삼습니다.)

  • 4
    최상위 동기 코드가 전부 끝나는 지점을 찾는다 (Stack-Empty Point)

    스크립트 최상위 레벨의 동기 코드 실행이 모두 끝나 콜스택이 완전히 비는 지점을 찾습니다. 이 지점 이전에는 어떤 큐도 처리되지 않습니다 — 이것이 "동기 코드가 항상 비동기 콜백보다 먼저 실행된다"는 규칙의 근거입니다.

  • 5
    Microtask 큐를 완전히 비운다 (Drain Microtasks)

    콜스택이 비면 Microtask 큐에 있는 작업을 순서대로 전부 실행합니다. 이때 실행 도중 새로운 microtask가 추가되면(예: .then 체인이 이어지는 경우) 그것도 같은 사이클에서 계속 처리합니다 — Macrotask로 넘어가기 전까지 절대 끊기지 않습니다.

  • 6
    Macrotask를 하나 꺼내 실행하고 5번으로 되돌아간다 (One Macrotask, Repeat)

    Macrotask 큐에서 딱 하나만 꺼내 실행합니다. 그 하나가 끝나면 다시 5번(Microtask 비우기)부터 반복합니다. 브라우저 환경에서는 이 사이 어딘가에 필요 시 렌더링(리페인트) 기회가 끼어들 수 있습니다.

6. 큐 분류표 — 무엇이 어디로 들어가는가

처음 보는 API가 나와도 이 표의 원리(누가 그 작업을 "예약"하는가)로 분류를 추론할 수 있어야 합니다.

분류 대표 API 처리 시점 실무 의미
Macrotask setTimeout, setInterval, DOM 이벤트 콜백, I/O 콜백 Microtask 큐가 완전히 빈 뒤, 한 번에 하나씩 여러 개의 setTimeout(fn, 0)이 있어도 한 사이클에 하나씩만 실행 — 중간에 렌더링 기회가 생길 수 있음
Microtask Promise.then/catch/finally, await 이후 코드, queueMicrotask, MutationObserver 콜스택이 빌 때마다, 큐가 빌 때까지 전부 Promise 체인이 길면 그동안 렌더링/타이머가 지연될 수 있음 (microtask starvation)
Animation Frame requestAnimationFrame Microtask 큐를 비운 뒤, 다음 화면을 그리기 직전 Macrotask도 Microtask도 아닌 별도 타이밍 — 화면에 반영되기 직전 값을 다룰 때 사용
Idle Callback requestIdleCallback 메인 스레드가 한가할 때(우선순위 가장 낮음) 중요도 낮은 작업(로깅, prefetch)을 렌더링을 방해하지 않고 실행 (Day 14에서 심화)

7. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 됩니다.

  1. 동기 코드부터 순서를 매깁니다. 위에서 아래로 실행되는 부분을 먼저 실행 순서에 넣습니다.
  2. 비동기 호출을 위임 지점으로 표시합니다. setTimeout/Promise 등을 만나면 콜백이 지금 실행되는 게 아니라 등록만 된다는 것을 짚습니다.
  3. 콜백을 Macrotask/Microtask로 분류합니다. 타이머·이벤트는 Macrotask, Promise 관련은 Microtask라고 명시합니다.
  4. 동기 코드가 모두 끝나는 지점을 확인합니다. 이 지점 전에는 어떤 콜백도 실행되지 않는다는 원칙을 말합니다.
  5. Microtask 큐를 먼저, 완전히 비웁니다. Macrotask보다 항상 우선하고, 큐가 빌 때까지 처리된다는 규칙을 적용합니다.
  6. Macrotask를 하나씩 꺼내며 반복합니다. 이 사이클을 실제 코드에 적용해 최종 출력 순서를 말합니다.

8. 핵심 답변 예시

면접관: "다음 코드의 출력 순서를 설명해주세요."

console.log('A');

setTimeout(() => console.log('B'), 0);

async function foo() {
  console.log('C');
  await null;
  console.log('D');
}
foo();

Promise.resolve().then(() => console.log('E'));

console.log('F');
STRONG ANSWER

먼저 동기 코드에 순서를 매깁니다. console.log('A')가 즉시 실행됩니다. 그다음 setTimeout은 콜백을 등록만 하고 Web API로 위임한 뒤 바로 다음 줄로 넘어갑니다 — 이 콜백은 나중에 Macrotask 큐에 들어갑니다.


foo()를 호출하면 async 함수라도 첫 await 이전 코드는 동기적으로 실행되므로 console.log('C')가 바로 실행됩니다. await null을 만나는 순간 foo의 나머지 부분(console.log('D'))은 Microtask 큐에 등록되고, 함수 실행은 여기서 일시 중단되며 제어권이 호출자에게 돌아갑니다.


그다음 Promise.resolve().then(...)도 콜백을 Microtask 큐에 등록만 하고 넘어갑니다. 이 시점에서 Microtask 큐에는 순서대로 [D 이어서 실행할 부분, E 콜백] 두 개가 쌓여 있습니다 — D가 E보다 먼저 등록됐기 때문입니다. 마지막으로 console.log('F')가 동기적으로 실행됩니다.


이제 최상위 동기 코드가 모두 끝나 콜스택이 비었으므로, Microtask 큐를 순서대로 전부 비웁니다: D, 그다음 E가 실행됩니다. Microtask 큐가 완전히 빈 뒤에야 Macrotask 큐의 setTimeout 콜백이 실행되어 B가 마지막에 출력됩니다.


따라서 최종 출력 순서는 A, C, F, D, E, B입니다.

이 답변이 좋은 이유:

  • 6단계 프레임워크를 코드에 그대로 적용하며 순서를 "유도"함 (암기가 아니라 추론)
  • async 함수도 첫 await 이전은 동기라는 것을 정확히 짚음
  • Microtask 큐 안에서의 등록 순서(D가 E보다 먼저 큐에 들어간다는 것)까지 근거로 설명
  • Macrotask는 Microtask 큐가 완전히 빈 뒤에만 실행된다는 규칙을 명시적으로 적용
  • 결과(순서)뿐 아니라 각 단계에서 "왜"를 함께 말함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"Promise는 항상 setTimeout보다 먼저 실행되니까 A, C, F, D, E, B 순서일 것 같아요."

이 답변이 약한 이유

  • 결론(순서)은 우연히 맞을 수 있지만 "왜"에 대한 설명이 전혀 없습니다.
  • "Promise가 항상 먼저"라는 규칙만 외웠을 뿐, Microtask 큐 자체의 존재와 동작 원리를 언급하지 않았습니다.
  • 코드가 조금만 복잡해지면(예: Promise 체인이 여러 단계이거나 await가 여러 번 나오면) 이 암기 규칙만으로는 틀린 답을 내게 됩니다.
  • 면접관이 "왜 그런가요?"라고 한 번만 더 물어도 답변이 막힙니다.
WEAK ANSWER — 또 다른 패턴

"비동기 코드라 순서를 정확히 예측하기 어렵고, 실행 환경에 따라 달라질 수 있습니다."

이 답변이 약한 이유

  • 완전히 틀렸습니다 — 이벤트 루프의 실행 순서는 명세(spec)로 정의된 결정적(deterministic) 동작입니다. "환경에 따라 달라진다"는 것은 실행 모델을 모른다는 신호로 읽힙니다.
  • 비동기 = 예측 불가능이라는 오해를 그대로 드러냅니다.
STRONG ANSWER 요소 체크리스트
  • 동기 코드와 비동기 위임 지점을 구분해서 짚는다
  • Macrotask와 Microtask를 구분하고, 각각 어떤 API가 어디에 들어가는지 분류한다
  • "콜스택이 완전히 빈 뒤"라는 조건을 명시적으로 언급한다
  • Microtask 큐가 완전히 빌 때까지 처리된다는 규칙을 근거로 사용한다
  • async/await의 동기 구간과 비동기 전환 지점을 정확히 구분한다
  • 결론만이 아니라 각 단계의 "왜"를 함께 설명한다

10. Senior-level Follow-up Q&A

Q: 왜 Microtask 큐를 Macrotask보다 먼저, 그리고 완전히 비우도록 설계했을까요?

모범 답변: Promise 체인은 하나의 논리적인 비동기 흐름을 표현합니다. 만약 체인 중간에 타이머 콜백이나 렌더링이 끼어들 수 있다면, 애플리케이션이 일관되지 않은 중간 상태(예: 일부 데이터만 업데이트된 화면)를 사용자에게 보여주거나, 다른 콜백이 아직 완전히 처리되지 않은 Promise 결과에 의존하게 되는 문제가 생길 수 있습니다. Microtask 큐를 완전히 비우는 규칙은 "하나의 비동기 흐름은 중간에 끊기지 않고 끝까지 처리된다"는 일관성을 보장하기 위한 설계입니다. 다만 이 규칙 때문에 Promise 체인이 스스로 계속 새로운 microtask를 만들어내면(예: 재귀적인 .then) 이벤트 루프가 Macrotask로 영영 넘어가지 못하는 "microtask starvation" 현상도 발생할 수 있습니다.

Q: requestAnimationFrame은 Microtask인가요 Macrotask인가요?

모범 답변: 둘 다 아닙니다. requestAnimationFrame 콜백은 브라우저가 Microtask 큐를 완전히 비운 뒤, 다음 프레임을 화면에 그리기 직전이라는 별도의 타이밍에 실행됩니다. 그래서 화면에 반영되기 직전의 최신 값을 다루기에 적합합니다. 이 타이밍은 (Day 14에서 다룰) requestIdleCallback과도 다른데, requestIdleCallback은 렌더링과 무관하게 "메인 스레드가 한가할 때" 실행되는 가장 낮은 우선순위 콜백입니다.

Q: 5초짜리 무거운 동기 반복문이 있으면, 그동안 등록된 setTimeout(fn, 0)은 언제 실행되나요?

모범 답변: 콜스택이 그 반복문 실행으로 계속 차 있는 5초 동안은 이벤트 루프가 다음 단계(Microtask 비우기, Macrotask 하나 실행)로 절대 넘어가지 못합니다. 따라서 setTimeout(fn, 0)으로 등록된 콜백은 최소 5초 뒤, 반복문이 끝나 콜스택이 비고 나서야 실행됩니다. 이것이 실무에서 "무거운 동기 연산이 UI를 멈추게 한다"는 현상의 근본 원인입니다 — 렌더링도 결국 이벤트 루프의 한 지점에서 기회를 얻는 것이므로, 콜스택이 계속 차 있으면 렌더링 자체도 밀립니다.

Q: 옛날 V8과 최신 V8에서 async/await이 소비하는 microtask tick 수가 다르다고 들었습니다. 아시나요?

모범 답변: 네. ES2017 스펙 초기 구현에서는 await 하나당 여러 번의 microtask tick(최대 3틱)이 소비되어 Promise.then 체인보다 미묘하게 느린 순서로 이어지는 경우가 있었습니다. 2019년경 스펙과 V8 구현이 개선되면서(await가 네이티브 Promise 처리와 동일하게) 1틱으로 줄었습니다. 실무에서 직접 이 틱 수를 세는 일은 드물지만, "같은 것처럼 보이는 await와 .then 체인도 엔진 버전에 따라 미묘하게 순서가 달라질 수 있었다"는 사실은 실행 모델이 스펙과 엔진 구현 양쪽에 의해 결정된다는 것을 보여주는 좋은 사례입니다.

Q: Node.js의 이벤트 루프는 브라우저와 완전히 같나요?

모범 답변: 기본 원리(콜스택 → 비동기 위임 → 큐 → 반복)는 같지만 세부 구조가 다릅니다. Node.js는 libuv 기반으로 timers, pending callbacks, poll, check, close callbacks 같은 여러 "phase"로 나뉘어 있고, 각 phase 사이마다 microtask 큐(그리고 process.nextTick 큐 — 이건 Promise microtask보다도 우선순위가 높습니다)를 비웁니다. 브라우저는 이런 phase 구분 없이 더 단순한 구조입니다. 실무에서는 "Node에도 이벤트 루프가 있지만 phase가 여러 개이고, process.nextTick이 가장 먼저 처리된다"는 정도만 구분해도 충분합니다.

11. 흔한 오해

오해 1

"setTimeout(fn, 0)은 0ms 뒤 즉시 실행된다"

0은 "최소 지연 시간"일 뿐입니다. 실제 실행 시점은 (1) 콜스택이 완전히 빌 것, (2) Microtask 큐가 전부 처리될 것, (3) Macrotask 큐에서 자기 순번이 올 것 — 이 세 조건을 모두 만족해야 합니다. 무거운 동기 코드가 있으면 수 초씩 밀릴 수 있습니다.

오해 2

"Promise가 setTimeout보다 빠른 건 우선순위가 더 높은 특별한 타이머라서다"

Promise는 타이머가 아닙니다. 애초에 Macrotask 큐가 아니라 완전히 별도인 Microtask 큐에 들어가고, 그 큐가 Macrotask보다 먼저·전부 처리되도록 이벤트 루프가 설계되어 있기 때문입니다.

오해 3

"await는 코드를 진짜로 멈추고(blocking) 기다린다"

await를 만나면 해당 async 함수의 실행만 일시 중단되고, 콜스택이 비면서 제어권이 호출자(그리고 이벤트 루프)에게 돌아갑니다. 그동안 다른 동기 코드나 다른 콜백이 실행될 수 있습니다. 브라우저나 Node.js 프로세스 전체가 멈추는 것이 아닙니다.

오해 4

"Microtask 하나 처리하면 곧바로 Macrotask로 넘어간다"

실제로는 Microtask 큐가 완전히 빌 때까지 반복 처리됩니다. 처리 도중 새로운 microtask가 추가되면(예: 체이닝된 .then) 그것도 같은 사이클에서 계속 처리합니다. 이 때문에 재귀적으로 microtask를 계속 만들어내는 코드는 Macrotask(타이머, 렌더링 포함)를 무한정 굶길 수 있습니다.

오해 5

"이벤트 루프는 별도의 스레드에서 돌아간다"

이벤트 루프는 별도 스레드가 아니라, JS 엔진이 콜스택과 큐를 관리하는 하나의 반복 실행 규칙일 뿐입니다. JS 실행 자체는 여전히 싱글 스레드입니다. (Web Worker는 별도 스레드를 쓰지만 그것은 완전히 별개의 실행 컨텍스트이며 Day 14에서 다룹니다.)
