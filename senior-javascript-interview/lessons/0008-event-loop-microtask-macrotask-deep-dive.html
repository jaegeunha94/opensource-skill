1. 핵심 면접 질문

Primary Question

"버튼 클릭 시 요소를 두 번 이동시키는 다음 코드가 있습니다. requestAnimationFrame 콜백에서 찍힌 transform 값은 무엇이고, 왜 그런가요?"

button.addEventListener('click', () => {
  box.style.transform = 'translateX(100px)';

  Promise.resolve().then(() => {
    box.style.transform = 'translateX(200px)';
  });

  requestAnimationFrame(() => {
    console.log(getComputedStyle(box).transform);
  });

  console.log('click handler 끝');
});
Variant 1

"Promise 체인이 계속 스스로를 재귀적으로 호출하면 화면이 멈춘 것처럼 보일 수 있다고 하는데, 왜 그런가요? setTimeout 반복문과는 뭐가 다른가요?"

Variant 2

"requestAnimationFrame, requestIdleCallback, queueMicrotask, setTimeout(fn, 0)을 실행 우선순위 순서로 나열하고 이유를 설명해주세요."

Variant 3

"Chrome DevTools Performance 탭에서 'Long Task'로 표시되는 구간이 있는데, 이게 왜 Microtask starvation과 관련될 수 있나요?"

2. 왜 면접에서 묻는가

Day 1에서 "Microtask가 Macrotask보다 먼저 실행된다"는 규칙 자체는 대부분의 지원자가 압니다. Day 8 수준 질문이 실제로 가려내려는 것은 그 규칙이 렌더링·애니메이션·체감 성능과 어떻게 물리적으로 연결되는지를 아는가입니다. "Promise가 먼저 실행된다"는 것과 "그래서 화면이 어떻게 보이는가"는 완전히 다른 층위의 지식입니다.

면접관이 이 질문으로 평가하는 것:

  • 큐 규칙을 렌더링 파이프라인에 연결할 수 있는가 — Microtask 처리 시점과 requestAnimationFrame, 실제 Style/Layout/Paint 단계가 한 프레임 안에서 어떤 순서로 얽혀 있는지 그릴 수 있는가.
  • "성능 문제"를 큐 이론으로 설명할 수 있는가 — "애니메이션이 끊긴다", "클릭 반응이 느리다" 같은 리포트를 받았을 때, 그 원인이 Long Task(무거운 동기 코드)인지 Microtask starvation(재귀적 Promise 체인)인지 구분해서 진단할 수 있는가.
  • 브라우저 스케줄링에 대한 최신 이해가 있는가 — 단일한 "매크로태스크 큐" 하나가 아니라, 브라우저가 우선순위가 다른 여러 태스크 소스를 두고 있으며(입력 이벤트가 타이머보다 우선될 수 있다는 것 등) 최신 Scheduler API 흐름까지 아는가.
  • 실무 디버깅 도구와 연결할 수 있는가 — DevTools Performance 탭에서 Long Task, microtask 체인, rAF 콜백이 타임라인에 각각 어떻게 나타나는지 실전 경험이 있는가.

10년차 이하에서 흔한 답변: "Promise가 setTimeout보다 먼저 실행돼요"까지만 말하고 멈춥니다. 시니어는 여기서 한 걸음 더 나아가 "그래서 microtask 체인이 길어지면 rAF와 페인트가 밀리고, 이게 실제로 애니메이션 프레임 드랍으로 나타난다"는 원인-결과를 프로덕션 증상과 연결해서 설명합니다.

3. 전제 개념 — 쉬운 한국어로

"매크로태스크 큐"는 사실 하나가 아니다

Day 1에서는 이해를 돕기 위해 "Macrotask Queue"를 하나의 대기열처럼 설명했습니다. 실제로 HTML 표준(WHATWG)은 브라우저가 여러 개의 task queue(태스크 소스별로 분리된 큐)를 가질 수 있다고 정의합니다. 예를 들어 "타이머용 큐", "사용자 입력(클릭 등)용 큐", "네트워크 콜백용 큐"가 개념적으로 분리되어 있고, 브라우저는 매 반복(tick)마다 이 큐들 중 하나를 구현체가 정한 우선순위 규칙으로 골라 하나의 task를 실행합니다. 크롬 같은 최신 브라우저는 실제로 "사용자 입력에 대한 응답"을 타이머 콜백보다 우선시키는 스케줄링 로직을 갖고 있습니다. 즉 "타이머가 등록된 순서대로만 실행된다"는 것도 정확한 설명이 아닙니다.

Microtask는 "끼어들기"가 아니라 "턴이 끝날 때 정리하기"다

Microtask 큐를 "우선순위가 높아서 새치기하는 큐"로 이해하면 반쯤만 맞습니다. 더 정확한 비유는 이렇습니다 — 회의(하나의 task)가 끝날 때마다 "오늘 나온 액션 아이템을 지금 바로 전부 처리하고 다음 회의로 넘어간다"는 규칙입니다. 하나의 동기 코드 블록(task)이 완전히 끝난 "직후", 다음 task로 넘어가기 전에 반드시 microtask 큐를 바닥까지 비웁니다. 이 "직후"가 정확히 언제인가가 이번 레슨의 핵심입니다.

"프레임(Frame)"이라는 렌더링의 기본 단위

화면은 보통 초당 60번(디스플레이에 따라 다를 수 있음, 대략 16.6ms마다 한 번) 다시 그려집니다. 이 한 번의 그리기를 "프레임"이라고 부릅니다. 브라우저는 이 16.6ms 예산 안에서 (1) JS 실행, (2) 스타일 재계산(Style), (3) 레이아웃 계산(Layout), (4) 페인트(Paint), (5) 합성(Composite)을 모두 마쳐야 사용자 눈에 부드러운 애니메이션으로 보입니다. 이 중 하나라도 예산을 넘기면 "프레임 드랍(jank)"이 발생합니다.

requestAnimationFrame은 언제 실행되는가

requestAnimationFrame(cb)은 "다음 프레임을 그리기 직전, 그러나 이번 tick의 Microtask 큐가 완전히 비워진 뒤"라는 정확한 타이밍에 실행됩니다. 즉 순서는 항상 Task 실행 → Microtask 전부 비우기 → (필요하다면) rAF 콜백들 실행 → Style/Layout/Paint → Composite입니다. rAF 콜백 안에서 DOM 스타일을 바꾸면 그 변경이 이번 프레임의 Layout/Paint에 반영될 기회를 얻습니다 — 그래서 "이번 프레임에 반영되어야 하는 시각적 변경"은 setTimeout이 아니라 rAF에 넣는 것이 정석입니다.

왜 이 개념부터 시작하는가

Day 17(Performance Profiling)에서 다룰 Long Task, Web Vitals(INP)는 결국 "이 프레임 예산 안에서 무엇이 얼마나 걸렸는가"를 측정하는 것입니다. 이번 레슨에서 큐 우선순위와 프레임 파이프라인을 원리로 연결해두면, 이후 프로파일링 데이터를 읽을 때 "이 구간이 왜 문제인지"를 근거로 설명할 수 있습니다.

4. 한 프레임 안에서 실제로 벌어지는 일

Day 1의 6단계 프레임워크에 "언제 화면이 그려지는가"를 끼워 넣으면 다음과 같은 확장된 순서가 됩니다.

  • 1
    Macrotask 큐에서 하나를 골라 실행한다

    클릭 이벤트, 타이머 콜백, 네트워크 콜백 등 하나의 task가 콜스택에서 완전히 실행됩니다. 이 task 실행 도중 .then이나 await로 등록된 콜백은 지금 실행되지 않고 microtask 큐에 쌓이기만 합니다.

  • 2
    Microtask 큐를 완전히 비운다

    방금 실행한 task가 끝나는 즉시, 다음 task로 넘어가기 전에 microtask 큐를 큐가 빌 때까지 반복 처리합니다. 처리 도중 새 microtask가 추가되면(체이닝) 같은 사이클에서 계속 처리합니다.

  • 3
    "렌더링 기회"인지 판단한다

    브라우저는 매 microtask 비움 이후 "지금 화면을 다시 그릴 시점인가"를 판단합니다. 이 판단은 프레임 예산(약 16.6ms)과 실제로 변경된 스타일이 있는지에 달려 있습니다 — 변경 사항이 없으면 이 단계는 스킵될 수 있습니다.

  • 4
    등록된 requestAnimationFrame 콜백을 전부 실행한다

    렌더링 기회로 판단되면, 그 시점까지 등록된 rAF 콜백들을 실행합니다. 이 콜백 안에서 만든 스타일 변경은 이번 프레임의 Layout/Paint 대상에 포함됩니다. rAF 콜백 안에서 다시 Promise.resolve().then()을 쓰면 그 microtask는 rAF 콜백들이 모두 끝난 직후, Layout 전에 비워집니다.

  • 5
    Style 재계산 → Layout → Paint → Composite

    변경된 스타일을 반영해 레이아웃을 다시 계산하고, 실제 픽셀을 그리고(Paint), 레이어를 합성(Composite)해 화면에 반영합니다. 이 단계가 무거우면(예: 매 프레임 offsetHeight를 읽고 쓰는 "레이아웃 스래싱") 프레임 예산을 넘겨 jank가 발생합니다.

  • 6
    다음 Macrotask로 돌아가 1번부터 반복한다

    이 전체 사이클이 초당 60번 가까이 반복되어야 "부드러운" 애플리케이션으로 체감됩니다. Microtask 체인이 스스로를 계속 재생성하면 2번 단계에서 영영 빠져나오지 못해 3~5번 자체가 실행되지 않고, 결과적으로 화면이 멈춘 것처럼 보입니다 — 이것이 "Microtask starvation이 렌더링을 막는다"는 말의 정확한 의미입니다.

Primary Question 정답 근거

클릭 핸들러(task)가 translateX(100px)를 설정하고, .then 콜백(microtask)과 rAF 콜백을 등록한 뒤 끝납니다. task가 끝나자마자(3~4단계 이전) microtask 큐부터 비우므로, translateX(200px)로 덮어쓰는 코드가 먼저 실행됩니다. 그 다음에야 rAF 콜백이 실행되므로, rAF 안에서 읽은 transform 값은 matrix(1, 0, 0, 1, 200, 0)(즉 200px)입니다 — 100px는 화면에 반영될 기회조차 없이 microtask 단계에서 덮어써집니다.

5. 큐 우선순위 심화표

우선순위 대상 실행 시점 실무 시사점
0 (최우선) Node.js process.nextTick 현재 동기 코드 종료 직후, Promise microtask보다도 먼저 남용하면 다른 microtask와 I/O 콜백을 굶길 수 있어 Node 공식 문서도 사용을 제한적으로 권장
1 Microtask (Promise.then, await 이후, queueMicrotask, MutationObserver) 매 task 종료 직후, 큐가 빌 때까지 전부 체인이 재귀적이면 Macrotask·rAF·Paint를 무한정 지연시킬 수 있음(starvation)
2 requestAnimationFrame Microtask 소진 후, 다음 프레임을 그리기 직전 (렌더링 기회가 있을 때만) 이번 프레임에 반영돼야 하는 시각적 변경은 여기에 넣는 것이 정석
3 일반 Macrotask (setTimeout, setInterval, 대부분의 DOM 이벤트, I/O) Microtask/렌더링 이후, 한 사이클에 하나씩. 단, 사용자 입력 이벤트는 구현체 스케줄러가 다른 타이머보다 우선 처리할 수 있음 "등록 순서 = 실행 순서"라고 단정할 수 없음 — 우선순위가 다른 소스가 섞이면 순서가 재조정될 수 있음
4 requestIdleCallback / Scheduler.postTask({priority:'background'}) 메인 스레드가 한가할 때(프레임 예산이 남았을 때만), 가장 낮은 우선순위 로깅, prefetch, 분석 전송처럼 지연돼도 되는 작업 전용. Day 14에서 심화

Node.js setImmediate vs setTimeout(fn, 0)

둘 다 매크로태스크지만 libuv의 서로 다른 phase에 속합니다. setImmediate는 "check" phase에서, setTimeout(fn, 0)은 "timers" phase에서 실행됩니다. 일반 동기 코드에서 순서는 실행 환경(타이머 정밀도 등)에 따라 달라질 수 있지만, I/O 콜백 내부에서 등록하면 setImmediate가 항상 setTimeout(fn, 0)보다 먼저 실행됨이 보장됩니다 — I/O 콜백은 poll phase에서 실행되고, 그다음 phase가 check(setImmediate)이기 때문입니다.

6. 답변 사고 순서

  1. 어떤 task가 지금 실행 중인지 특정합니다. 클릭 핸들러, 타이머 콜백 등 현재 콜스택을 채우고 있는 동기 코드 블록을 명시합니다.
  2. 그 안에서 등록되는 microtask와 rAF 콜백을 구분합니다. .then/await는 microtask, requestAnimationFrame은 별도 렌더링 타이밍이라고 명시합니다.
  3. task 종료 직후 microtask가 전부 비워짐을 적용합니다. 이 시점에 DOM 상태나 변수 값이 어떻게 바뀌는지 추론합니다.
  4. 렌더링 기회 여부를 판단합니다. 변경된 스타일이 있는지, 프레임 예산이 남아있는지에 따라 rAF·Paint가 이번 tick에 실행될지 결정합니다.
  5. 결과 값을 프레임 파이프라인 순서(Task→Microtask→rAF→Style/Layout/Paint)에 대입해 도출합니다.
  6. 성능 관점의 함의를 덧붙입니다. 이 순서가 실무에서 왜 "애니메이션이 끊긴다"는 증상과 연결되는지 한 문장으로 마무리합니다.

7. 핵심 답변 예시

면접관: "Promise 체인이 스스로를 재귀적으로 계속 호출하면 화면이 멈춘 것처럼 보일 수 있다고 하셨는데, 원리를 설명해주세요."

function spin() {
  Promise.resolve().then(spin); // 계속 새로운 microtask를 만들어냄
}
spin();

requestAnimationFrame(() => {
  console.log('이 로그는 언제 찍힐까?');
});
STRONG ANSWER

이벤트 루프는 하나의 task가 끝나면 다음 단계(렌더링 기회 확인, rAF 콜백 실행)로 넘어가기 전에 반드시 microtask 큐를 완전히 비웁니다. 그런데 spin 함수는 .then(spin)으로 자기 자신을 다시 microtask 큐에 등록하기 때문에, microtask 큐가 절대 비워지지 않는 상태가 됩니다.


이벤트 루프 입장에서는 "microtask 큐가 아직 안 비었으니 다음 단계로 못 간다"는 규칙을 계속 지키고 있을 뿐입니다. 그 결과 3~5단계(렌더링 기회 판단, rAF 실행, Style/Layout/Paint)로 영원히 넘어가지 못하고, requestAnimationFrame에 등록한 콜백도 실행되지 않습니다. 이것이 "Promise 체인이 화면을 멈추게 한다"는 현상의 원리입니다.


반면 setTimeout으로 같은 재귀를 만들면(function loop(){ setTimeout(loop, 0); }) 매번 새로운 macrotask로 큐에 다시 들어가기 때문에, 한 번 실행될 때마다 이벤트 루프가 렌더링 기회를 확인할 수 있는 지점을 거치게 됩니다. 그래서 setTimeout 재귀는 CPU를 계속 점유해도 화면이 "완전히 멈추지"는 않고, Promise 재귀는 완전히 멈춥니다 — 같은 "무한 반복"처럼 보여도 큐의 종류가 다르면 사용자 체감이 근본적으로 다릅니다.

이 답변이 좋은 이유:

  • "Microtask가 계속 재생성되면 다음 단계로 못 넘어간다"는 규칙을 정확히 짚음
  • rAF/렌더링이 그 "다음 단계"에 포함된다는 것까지 연결함
  • setTimeout 재귀와의 대조를 통해 "같은 무한 루프도 큐 종류에 따라 증상이 다르다"는 실무 함의를 도출함
  • 추상적 규칙(큐 이론)을 구체적 증상(화면 멈춤)까지 끝까지 연결함

8. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"Promise가 계속 실행되니까 CPU를 많이 써서 느려지는 거예요. setTimeout도 무한 반복이면 마찬가지로 느려질 거고요."

이 답변이 약한 이유

  • "CPU를 많이 쓴다"는 것과 "렌더링이 아예 실행되지 못한다"는 것은 다른 현상입니다 — 이 답변은 그 차이를 구분하지 못합니다.
  • setTimeout 재귀와 Promise 재귀가 "마찬가지"라고 뭉뚱그렸지만, 실제로는 렌더링 기회 확보 여부에서 근본적으로 다릅니다.
  • 큐 구조(microtask가 완전히 비워져야 다음 단계로 간다는 규칙)를 근거로 들지 않고 "느려진다"는 결과만 말했습니다.
STRONG ANSWER 요소 체크리스트
  • Microtask 큐가 "완전히 빌 때까지" 다음 단계로 못 넘어간다는 규칙을 명시한다
  • 그 "다음 단계"에 rAF와 렌더링 기회 판단이 포함된다는 것을 연결한다
  • setTimeout 재귀와의 차이(매번 새 macrotask vs 계속 같은 microtask 사이클)를 대조한다
  • "화면이 멈춘 것처럼 보인다"는 사용자 체감 증상까지 원리로 도출한다

9. Senior-level Follow-up Q&A

Q: requestAnimationFrame 콜백 안에서 element.offsetHeight를 읽고 바로 style.height를 쓰는 코드를 반복문으로 여러 요소에 적용하면 무슨 문제가 생기나요?

모범 답변: "레이아웃 스래싱(layout thrashing)" 또는 강제 동기 레이아웃(forced synchronous layout)이 발생합니다. 원래는 Style/Layout 계산이 한 프레임에 한 번, 여러 스타일 변경을 모아서 처리됩니다. 하지만 스타일을 쓴 직후 곧바로 offsetHeight 같은 레이아웃 의존 값을 읽으면, 브라우저는 정확한 값을 주기 위해 그 자리에서 즉시 레이아웃을 다시 계산해야 합니다. 이걸 쓰기-읽기-쓰기-읽기로 반복하면 한 프레임 안에서 레이아웃 계산이 요소 수만큼 여러 번 실행되어 프레임 예산을 크게 초과하게 됩니다. 해결책은 모든 읽기(offsetHeight 등)를 먼저 배치로 끝내고, 그다음에 모든 쓰기(style.height 등)를 배치로 처리하는 것입니다.

Q: DevTools Performance 탭에서 "Long Task"는 정확히 무엇을 기준으로 표시되나요?

모범 답변: 하나의 task(콜스택이 비기 전까지 실행되는 동기 코드 블록, microtask 처리 포함)가 50ms를 넘기면 Long Task로 분류됩니다. PerformanceObserver의 longtask entry type으로 코드에서도 직접 관찰할 수 있습니다. 50ms라는 기준은 사람이 입력에 대한 반응 지연을 인지하기 시작하는 경험적 임계값에서 나왔고, Web Vitals의 INP(Interaction to Next Paint) 지표와 직접 연결됩니다 — Long Task가 자주 발생하면 클릭·탭 같은 상호작용 후 다음 페인트까지의 지연이 길어져 INP 점수가 나빠집니다.

Q: 브라우저가 사용자 입력 이벤트를 타이머보다 먼저 처리하도록 스케줄링을 조정한다고 하셨는데, 이게 스펙으로 보장되나요?

모범 답변: HTML 표준은 "task queue가 여러 개일 수 있고, 각 큐에서 다음에 실행할 task를 어떻게 고를지는 구현체(브라우저)에 맡긴다"는 수준까지만 규정합니다. 즉 정확한 우선순위 알고리즘은 스펙이 아니라 구현 세부사항입니다. Chrome의 경우 사용자 입력에 대한 응답성을 높이기 위해 입력 관련 task의 우선순위를 높이는 스케줄러 로직이 실제로 존재하고, isInputPending() API나 Scheduler.postTask()의 priority 옵션으로 개발자가 이 우선순위 개념을 직접 활용할 수도 있습니다. 다만 "정확히 어떤 순서로 실행된다"를 스펙 차원에서 100% 보장하는 것은 아니므로, 면접에서는 "구현체마다 세부 스케줄링은 다를 수 있지만, 큰 원칙(microtask는 항상 macrotask보다 먼저, 완전히 비워질 때까지)은 스펙으로 고정돼 있다"고 구분해서 답하는 것이 정확합니다.

Q: MutationObserver 콜백은 왜 microtask로 분류되나요? DOM 변경이 이벤트처럼 느껴지는데요.

모범 답변: MutationObserver는 설계 의도 자체가 "DOM 변경 사항을 최대한 빠르게, 그러나 현재 실행 흐름을 방해하지 않고 일괄로 모아서(batch) 알려주는 것"입니다. 만약 이걸 macrotask로 처리하면 여러 개의 다른 macrotask(타이머, 다른 이벤트) 사이에 끼어 순서를 예측하기 어렵고 지연도 커집니다. Microtask로 처리하면 현재 실행 중이던 동기 코드 블록이 끝나는 즉시, 다른 어떤 macrotask보다도 먼저 변경 사항을 일괄로 받을 수 있습니다. 이는 Promise와 같은 우선순위를 공유하므로, 여러 개의 DOM 변경이 한 task 안에서 발생해도 콜백은 한 번(batch)만 호출되는 특성과도 맞물립니다.

Q: Scheduler.postTask()는 기존 setTimeout/requestIdleCallback과 무엇이 다른가요?

모범 답변: Scheduler.postTask(callback, { priority })는 'user-blocking', 'user-visible', 'background' 세 가지 명시적 우선순위를 코드에서 직접 지정할 수 있게 해주는 최신 API입니다. 기존에는 "이 작업이 얼마나 급한가"를 setTimeout의 지연 시간이나 requestIdleCallback 사용 여부로 간접적으로만 표현할 수 있었는데, postTask는 브라우저의 실제 내부 스케줄러에게 우선순위를 직접 전달합니다. 예를 들어 사용자가 방금 클릭한 것에 대한 반응은 'user-blocking'으로, 분석 로그 전송은 'background'로 등록하면 브라우저가 프레임 예산 안에서 더 똑똑하게 순서를 조정할 수 있습니다. 아직 모든 브라우저에서 지원되지는 않으므로 폴백 전략(예: requestIdleCallback/setTimeout)이 함께 필요합니다.

10. 흔한 오해

오해 1

"Microtask는 항상 빠르게 끝나니까 성능에 나쁜 영향을 줄 수 없다"

Microtask 하나하나는 짧을 수 있지만, 재귀적으로 계속 새 microtask를 만들어내면(체이닝) 큐가 영원히 안 비워져서 렌더링·타이머 전체가 지연될 수 있습니다("microtask starvation"). "짧다"와 "큐 전체가 빨리 빈다"는 다른 이야기입니다.

오해 2

"브라우저는 매 프레임(16.6ms)마다 반드시 화면을 다시 그린다"

변경된 스타일이 없으면 브라우저는 Style/Layout/Paint 단계를 건너뛸 수 있습니다. "프레임 주기가 있다"는 것과 "매 프레임 무조건 렌더링 파이프라인 전체가 실행된다"는 것은 다릅니다. 또한 여러 rAF 콜백이 쌓여도 실제로는 하나의 렌더링 기회에 한꺼번에 처리될 수 있습니다.

오해 3

"매크로태스크는 등록된 순서대로만 실행된다"

같은 소스(예: 여러 개의 setTimeout)라면 등록 순서(및 지연 시간)가 큰 영향을 주지만, 소스가 다르면(예: 사용자 입력 이벤트 vs 타이머) 구현체의 스케줄링 우선순위에 따라 순서가 조정될 수 있습니다. 스펙은 "여러 task queue가 있을 수 있다"고만 정의하고, 그 사이 우선순위는 구현에 맡깁니다.

오해 4

"setTimeout 재귀와 Promise 재귀는 둘 다 무한 루프이므로 증상이 같다"

setTimeout 재귀는 매번 새로운 macrotask로 큐에 들어가므로 그 사이사이에 이벤트 루프가 렌더링 기회를 확인할 지점을 거칩니다. Promise(microtask) 재귀는 현재 사이클의 microtask 큐가 절대 안 비워지므로 렌더링 기회 확인 자체가 실행되지 않습니다. 전자는 CPU를 많이 쓰며 느려지고, 후자는 화면이 완전히 멈춥니다.

오해 5

"requestAnimationFrame은 setTimeout보다 그냥 더 빠른 타이머다"

rAF는 시간 기반 타이머가 아니라 "다음 렌더링 시점"에 맞춰 호출되는 콜백입니다. 탭이 백그라운드에 있으면 rAF는 실행 빈도가 크게 줄거나 멈추지만(배터리·성능 절약), setTimeout은 (지연은 될 수 있어도) 계속 실행될 수 있습니다. 즉 우선순위 차이가 아니라 "무엇에 맞춰 스케줄링되는가"가 근본적으로 다릅니다.
