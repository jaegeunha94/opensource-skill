1. 핵심 면접 질문

Primary Question

"다음 코드에서 inner()가 실행될 때 console.log(x)는 무엇을 출력할까요? 왜 1도 2도 아닐까요?"

var x = 1;

function outer() {
  var x = 2;

  function inner() {
    console.log(x); // ?
    var x = 3;
  }

  inner();
}

outer();
Variant 1

"다음 코드는 왜 ReferenceError가 날까요?"

if (true) {
  let y = 10;
}
console.log(y);
Variant 2

"자바스크립트가 '렉시컬 스코프(lexical scope) 언어'라는 말이 정확히 무슨 뜻인가요? 함수를 어디서 호출하느냐가 스코프에 영향을 주지 않나요?"

Variant 3

"중첩된 함수 안에서 바깥 스코프의 변수를 계속 참조하는 코드가 있는데, 이게 성능에 영향을 줄 수 있나요?"

2. 왜 면접에서 묻는가

이 질문은 "스코프란 무엇인가"를 정의로 아는가가 아니라, 여러 스코프가 중첩되고 이름이 겹칠 때 엔진이 정확히 어떤 순서로, 언제 결정된 규칙에 따라 변수를 찾는지 원리로 아는가를 테스트합니다. "안쪽에서 바깥쪽으로 찾아 나간다"는 문장만 외운 사람은, 변수 섀도잉(shadowing)이나 함수가 정의된 곳과 다른 곳에서 호출되는 상황 앞에서 바로 틀린 답을 냅니다.

면접관이 이 질문으로 평가하는 것:

  • 스코프 체인이 "정적으로 결정"된다는 것을 아는가 — 스코프 체인은 코드가 실행되는 시점이 아니라 코드가 작성(정의)되는 시점에 이미 확정된다는 것, 즉 Day 2에서 배운 Creation Phase의 Lexical Environment 연결과 직접 이어진다는 것을 아는가.
  • 섀도잉(shadowing)의 함정을 아는가 — 안쪽 스코프에 같은 이름의 변수가 "선언만" 되어 있어도(호이스팅된 상태라도), 조회는 그 스코프에서 멈추고 바깥으로 나가지 않는다는 것을 아는가.
  • 함수 스코프와 블록 스코프의 실무 차이를 설명할 수 있는가 — var는 함수 단위, let/const는 블록({}) 단위로 스코프가 나뉜다는 차이가 실제 어떤 버그로 이어지는지 사례를 말할 수 있는가.
  • 렉시컬 스코프와 동적 스코프의 차이를 개념적으로 구분하는가 — 자바스크립트가 왜 "호출 위치"가 아니라 "정의 위치" 기반으로 스코프를 결정하는 언어인지, 이것이 코드 예측 가능성에 왜 중요한지 아는가.
  • 스코프 체인 조회가 "공짜가 아니다"라는 것을 아는가 — 반복적으로 실행되는 함수 안에서 매번 바깥(특히 전역) 스코프까지 조회를 타는 것이 실제로 어떤 상황에서 성능에 영향을 줄 수 있는지 판단할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "가까운 스코프부터 찾다가 없으면 바깥으로 나간다"는 절차만 말하고 멈추는 답변입니다. 이는 맞지만 절반입니다 — 시니어는 여기에 "이 체인이 언제, 어떻게 결정되는지(정의 시점, 정적)"와 "섀도잉이 왜 종종 버그의 원인이 되는지"까지 이어서 설명합니다.

3. 전제 개념 — 쉬운 한국어로

스코프(Scope)란 무엇인가

스코프는 "이 변수 이름이 어디까지 유효한가"를 정하는 규칙입니다. 도서관에 비유하면, 스코프는 "책이 꽂혀 있는 서가의 범위"와 같습니다. 특정 서가(스코프) 안에서 책(변수)을 찾다가 없으면, 그 서가를 관리하는 상위 구역(바깥 스코프)으로 나가서 다시 찾습니다. 이 "찾아 나가는 경로"가 바로 Scope Chain(스코프 체인)입니다.

왜 "렉시컬(Lexical)" 스코프라고 부르는가

"렉시컬"은 "코드가 눈에 보이는 그대로, 글자(text)로 작성된 위치 기준"이라는 뜻입니다. 즉 함수가 어디서 호출되는지가 아니라, 함수가 코드상 어디에 "써져 있는지"가 스코프 체인을 결정합니다. 이것이 Day 2에서 배운 Creation Phase와 연결됩니다 — 함수가 정의되는 순간(코드가 파싱되는 시점), 엔진은 이미 "이 함수의 바깥 스코프는 여기다"라고 하나의 참조를 고정해 둡니다. 함수가 나중에 전혀 다른 곳에서 호출되어도 이 참조는 바뀌지 않습니다.

비교를 위한 개념: 동적 스코프(Dynamic Scope)

일부 언어(예: 초기 Bash 스크립트 변수 규칙)는 "동적 스코프"를 씁니다 — 이 경우 변수 조회가 "코드가 어디에 써 있는가"가 아니라 "지금 누가 이 함수를 호출했는가(호출 스택)"를 기준으로 결정됩니다. 자바스크립트는 이 방식을 쓰지 않습니다. 자바스크립트에서 함수 안의 자유 변수(free variable)가 무엇을 가리키는지는, 그 함수를 누가 호출하든 상관없이 함수가 정의된 위치만 보고 항상 동일하게 결정됩니다. 이 예측 가능성이 렉시컬 스코프의 핵심 가치입니다.

Scope Chain은 "링크드 리스트"처럼 연결되어 있다

실무적으로는, 각 실행 컨텍스트(Day 2 참고)가 만들어질 때 자신의 Lexical Environment 안에 "바깥 Lexical Environment에 대한 참조"를 하나씩 갖습니다. 변수를 조회할 때 엔진은 현재 스코프의 변수 목록을 먼저 보고, 없으면 이 참조를 따라 바깥으로, 또 없으면 그 바깥으로 — 결국 전역 스코프까지 — 순서대로 훑습니다. 전역까지 가도 없으면 ReferenceError: x is not defined가 발생합니다.

왜 이 개념이 Day 2 다음에 필요한가

Day 2에서 "하나의 실행 컨텍스트 안에서 변수의 자리가 어떻게 등록되는가"를 다뤘다면, Day 3은 "여러 실행 컨텍스트가 중첩되어 있을 때, 그 자리들을 어떤 순서로 찾아가는가"를 다룹니다. 이 스코프 체인 개념은 Day 4(클로저)의 정확한 기반이 됩니다 — 클로저란 결국 "함수가 자신이 정의된 스코프 체인에 대한 참조를 계속 붙잡고 있는 것"이기 때문입니다.

4. Scope Chain 조회 규칙 — 정적 구조, 동적 값

Primary Question 코드를 다시 봅시다.

var x = 1;

function outer() {
  var x = 2;

  function inner() {
    console.log(x); // undefined
    var x = 3;
  }

  inner();
}

outer();
단계 무슨 일이 일어나는가
1. 스코프 체인은 정의 시점에 고정 inner는 outer 내부에 "써 있으므로", inner의 스코프 체인은 inner 자신 → outer → 전역으로 코드 작성 시점에 이미 정해집니다.
2. inner 실행 시 자신의 스코프부터 조회 시작 console.log(x)가 실행될 때 엔진은 먼저 inner 자신의 스코프에 x라는 이름이 있는지 봅니다.
3. inner 안에 var x = 3이 있다 → 조회는 여기서 멈춘다 Day 2에서 배운 대로 var x는 Creation Phase에서 inner 스코프 안에 이미 자리가 등록되고 undefined로 초기화되어 있습니다. 조회는 "이름이 있는가"만 보지, "실행 흐름상 그 줄을 이미 지났는가"는 보지 않습니다. 따라서 바깥(outer의 x = 2)으로 나가지 않고 inner 자신의 (아직 undefined인) x를 찾아 멈춥니다.
4. 결과 1도 2도 아닌 undefined가 출력됩니다.

핵심 원리 — 스코프 체인은 "구조"가 정적이고, 그 안의 "값"만 동적이다

스코프 체인이라는 탐색 경로(어느 스코프에서 어느 스코프로 나가는가)는 코드가 작성된 구조만 보고 정해지는 정적인 것입니다. 반면 그 경로 위에서 실제로 어떤 값이 채워져 있는지는 실행 흐름(언제 그 줄에 도달했는지)에 따라 달라지는 동적인 것입니다. 이 둘을 분리해서 이해하는 것이 시니어급 설명의 핵심입니다 — "구조는 정적, 값은 동적".

5. 함수 스코프(var) vs 블록 스코프(let/const) 실무 함정

var는 함수(또는 전역) 단위로만 스코프를 나눕니다 — if, for, {} 블록은 var에게 새로운 스코프가 아닙니다. 반면 let/const는 블록({}) 단위로 스코프를 나눕니다.

function example() {
  if (true) {
    var a = 1;   // 함수 스코프 → if 블록 밖에서도 접근 가능
    let b = 2;   // 블록 스코프 → if 블록을 벗어나면 접근 불가
  }
  console.log(a); // 1
  console.log(b); // ReferenceError: b is not defined
}
상황 var의 문제 let/const가 해결하는 방식
조건문/반복문 내부 임시 변수 블록을 벗어나도 살아있어, 의도치 않게 바깥 코드에서 접근·덮어쓰기가 가능함 블록을 벗어나면 자동으로 스코프가 끝나 실수를 컴파일(파싱) 단계에 가깝게 조기에 드러냄
반복문에서 클로저 생성 모든 반복이 같은 var 바인딩을 공유해, 비동기 콜백이 실행될 때 마지막 값만 참조하는 버그 발생 매 반복마다 새로운 렉시컬 환경(새 바인딩)이 생성되어 각 클로저가 서로 다른 값을 기억함 (Day 4에서 심화)
중첩된 블록에서의 이름 재사용 같은 함수 스코프 안에서 var를 여러 번 선언해도 에러가 나지 않아 실수를 감춤 같은 블록 안에서 let/const를 중복 선언하면 SyntaxError로 즉시 드러남

실무 연결 — 왜 이게 프로덕션 버그가 되는가

레거시 코드베이스에서 try/catch 블록이나 반복문 안에서 var로 선언한 임시 변수가, 코드가 길어지면서 아래쪽 로직이 그 이름을 다시 참조하며 값을 덮어쓰는 사고가 실제로 자주 발생합니다. 함수가 길어질수록(특히 리팩터링 중 코드가 이동·병합될 때) var의 넓은 스코프는 "이 변수가 지금 어떤 값인지" 추적하기 어렵게 만듭니다. 이런 이유로 실무에서는 var 사용을 지양하고 const를 기본, 재할당이 필요할 때만 let을 쓰는 관례가 정착했습니다.

6. Scope Chain 조회 비용과 성능

변수 하나를 조회할 때 엔진은 스코프 체인을 따라 "현재 스코프 → 바깥 스코프 → ... → 전역 스코프" 순으로 이름을 찾습니다. 체인이 길수록, 그리고 그 조회가 많이 반복 실행되는 코드(hot path) 안에 있을수록 누적 비용이 커질 수 있습니다.

// 매 반복마다 전역(또는 훨씬 바깥) 스코프까지 조회해야 하는 패턴
function processLargeArray(items) {
  for (let i = 0; i < items.length; i++) {
    // globalConfig가 여러 단계 바깥(전역) 스코프에 있다면
    // 이 줄이 실행될 때마다 스코프 체인을 끝까지 훑어야 함
    doSomething(items[i], globalConfig);
  }
}

실무 연결 — 언제 실제로 신경 써야 하는가

현대 JS 엔진(V8 등)은 이런 조회를 상당 부분 최적화(인라인 캐싱 등)하기 때문에, 대부분의 일반적인 코드에서는 스코프 체인 깊이가 체감 가능한 성능 문제로 이어지지 않습니다. 다만 수백만 번 반복되는 루프의 내부(hot path)에서 매번 여러 단계 바깥의 변수를 참조하는 패턴은, 그 변수를 루프 진입 전에 지역 변수로 한 번 캐싱해두는 것만으로 조회 비용을 줄일 수 있습니다. 이는 "무조건 스코프를 얕게 만들어야 한다"는 규칙이 아니라, 프로파일링으로 실제 hot path임이 확인된 경우에만 적용할 최적화라는 점이 중요합니다 — 근거 없이 미리 최적화하면 가독성만 해칩니다.

7. 답변 사고 순서

  1. 스코프 체인은 정의(작성) 시점에 정적으로 고정된다는 원칙을 먼저 언급합니다. 호출 위치가 아니라 코드가 써진 위치가 기준임을 명시합니다.
  2. 문제의 변수가 어느 스코프에 선언되어 있는지 안쪽부터 나열합니다. 가장 안쪽 스코프부터 전역까지 계층을 그려봅니다.
  3. 가장 안쪽 스코프에 같은 이름이 "선언"되어 있는지 확인합니다(섀도잉 여부). 선언되어 있다면 실행 흐름상 아직 값이 대입되지 않았어도 조회는 거기서 멈춥니다.
  4. 그 자리의 초기화 상태(Day 2의 Creation/Execution Phase)를 적용합니다. var면 undefined, let/const면 TDZ로 인한 에러입니다.
  5. 선언이 없다면 바깥 스코프로 한 단계씩 이동하며 같은 절차를 반복합니다. 전역까지 가도 없으면 ReferenceError입니다.

8. 핵심 답변 예시

면접관: "다음 코드에서 inner()가 실행될 때 console.log(x)는 무엇을 출력할까요?"

var x = 1;

function outer() {
  var x = 2;

  function inner() {
    console.log(x); // ?
    var x = 3;
  }

  inner();
}

outer();
STRONG ANSWER

inner 함수는 outer 함수 내부에 정의되어 있기 때문에, 자바스크립트는 렉시컬 스코프 언어이므로 inner의 스코프 체인은 코드가 작성된 구조 그대로 inner 자신 → outer → 전역으로 정의 시점에 고정됩니다. 이는 inner가 나중에 어디서 호출되는지와 무관합니다.


console.log(x)를 실행할 때 엔진은 먼저 inner 자신의 스코프에 x라는 이름이 있는지를 봅니다. inner 안에는 var x = 3이라는 선언이 있고, Day 2에서 다룬 Creation Phase 규칙에 따라 이 x는 이미 inner 스코프에 자리를 잡고 undefined로 초기화되어 있습니다. 스코프 체인 조회는 "이 스코프에 그 이름이 선언되어 있는가"만 확인하지, "실행 흐름상 대입문을 이미 지났는가"는 신경 쓰지 않습니다.


따라서 조회는 inner 스코프에서 멈추고, outer의 x = 2나 전역의 x = 1까지 나가지 않습니다. 결과적으로 console.log(x)는 undefined를 출력합니다. 이것이 흔히 "변수 섀도잉(shadowing)"이라고 부르는 현상이며, 스코프 체인이 값이 아니라 "선언의 존재 여부"를 기준으로 조회를 멈춘다는 것을 보여주는 대표적인 예입니다.

이 답변이 좋은 이유:

  • 렉시컬 스코프의 정의(정적 고정, 정의 시점 기준)를 먼저 명시함
  • 스코프 체인 조회가 "이름의 존재"를 기준으로 멈춘다는 원리를 정확히 짚음
  • Day 2의 호이스팅/Creation Phase 개념과 연결해 undefined가 나오는 근거를 설명함
  • 이 현상에 "섀도잉"이라는 정확한 용어를 붙여 후속 질문에 대비함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"가장 가까운 outer의 x를 찾으니까 2가 출력돼요."

이 답변이 약한 이유

  • 스코프 체인 조회가 "값이 있는 가장 가까운 스코프"를 찾는 것이 아니라 "이름이 선언된 가장 가까운 스코프"에서 멈춘다는 것을 놓치고 있습니다.
  • inner 자신의 스코프에 x가 (비록 아직 초기화 전이라도) 선언되어 있다는 사실, 즉 섀도잉이 일어나고 있다는 것을 확인하지 않았습니다.
  • Day 2의 호이스팅 개념과 연결하지 못해 왜 undefined인지 근거를 대지 못합니다.
WEAK ANSWER — 또 다른 패턴

"함수를 호출한 위치에 따라 스코프가 결정되니까..."

이 답변이 약한 이유

  • 자바스크립트는 동적 스코프가 아니라 렉시컬(정적) 스코프 언어라는 근본 전제를 틀리게 말하고 있습니다. 이 오해로는 콜백 함수나 이벤트 핸들러가 왜 정의된 곳의 변수를 계속 참조하는지 설명할 수 없습니다.
STRONG ANSWER 요소 체크리스트
  • 자바스크립트가 렉시컬(정적) 스코프 언어라는 전제를 먼저 명시한다
  • 스코프 체인이 "정의 시점"에 정해지고, "호출 위치"와 무관하다는 것을 구분한다
  • 조회가 "값의 존재"가 아니라 "이름의 선언 여부"를 기준으로 멈춘다는 것(섀도잉)을 설명한다
  • Day 2의 Creation/Execution Phase, 호이스팅과 연결해 결과값의 근거를 댄다
  • 이 현상에 정확한 용어(섀도잉)를 붙여 설명한다

10. Senior-level Follow-up Q&A

Q: 화살표 함수는 스코프 체인이 일반 함수와 다르게 동작하나요?

모범 답변: 변수 조회를 위한 스코프 체인 규칙 자체는 화살표 함수와 일반 함수가 동일합니다 — 둘 다 렉시컬 스코프를 따릅니다. 다만 화살표 함수는 자기 자신만의 this와 arguments를 만들지 않고, 이 두 가지를 렉시컬 스코프 체인을 따라 바깥에서 그대로 가져와 씁니다. 즉 일반 변수 조회 규칙은 같지만, this/arguments라는 특수한 바인딩만 화살표 함수에서 예외적으로 동작합니다. 이 주제는 Day 6(this 바인딩)에서 더 깊게 다룹니다.

Q: eval이나 with 문이 스코프 체인에 영향을 준다고 들었는데, 실무에서 신경 써야 하나요?

모범 답변: eval과 with는 런타임에 스코프 체인에 새로운 스코프를 동적으로 끼워 넣을 수 있어, "스코프는 정적으로 정해진다"는 원칙을 사실상 깨는 예외적인 기능입니다. 이 때문에 두 기능 모두 엔진이 스코프 조회를 미리 최적화하기 어렵게 만들고(엔진이 "혹시 런타임에 스코프가 바뀔 수도 있으니" 안전한 경로로 폴백해야 함), 코드 예측 가능성도 크게 떨어뜨립니다. Strict mode에서는 with 자체가 금지되어 있고, 실무에서는 eval과 with 둘 다 사용하지 않는 것이 사실상 표준 관례입니다.

Q: 클로저와 스코프 체인은 같은 개념인가요?

모범 답변: 같지 않지만 밀접하게 연결되어 있습니다. 스코프 체인은 "변수를 조회하는 경로 규칙" 그 자체이고, 클로저는 "함수가 자신이 정의될 때의 스코프 체인에 대한 참조를 실행이 끝난 뒤에도 계속 붙잡고 있는 현상"입니다. 즉 클로저는 스코프 체인이라는 메커니즘이 만들어내는 결과이지, 별개의 새로운 자료구조가 아닙니다. 이 관계는 Day 4에서 클로저의 실제 메모리 모델과 함께 자세히 다룹니다.

Q: let이 블록 스코프라는 것이 for 반복문에서는 어떻게 특별하게 적용되나요?

모범 답변: for (let i = 0; ...) 형태에서는 스펙상 매 반복(iteration)마다 i에 대한 새로운 렉시컬 환경(새 바인딩)이 만들어지도록 정의되어 있습니다. 단순히 "블록 스코프라서 반복문 밖에서 못 쓴다"는 차원을 넘어, 반복마다 물리적으로 다른 i 바인딩이 생긴다는 의미입니다. 그래서 반복문 안에서 비동기 콜백이 i를 클로저로 캡처하면, 각 콜백이 자신이 생성될 당시의 i 값을 서로 다르게 기억합니다. var i를 썼다면 함수(또는 전역) 스코프 하나만 존재하므로 모든 콜백이 같은 바인딩을 공유해 최종값 하나만 참조하게 됩니다.

11. 흔한 오해

오해 1

"스코프 체인은 함수가 호출될 때마다 다시 계산된다"

스코프 체인의 "구조"(어떤 순서로 바깥 스코프를 찾아 나가는지)는 함수가 정의될 때 이미 고정됩니다. 함수가 호출될 때 새로 만들어지는 것은 그 함수 자신의 지역 스코프(변수들의 새 자리)일 뿐, 바깥으로 이어지는 체인의 순서 자체는 바뀌지 않습니다.

오해 2

"스코프 체인 조회는 '값이 존재하는' 가장 가까운 스코프를 찾는다"

실제로는 "이름이 선언되어 있는" 가장 가까운 스코프에서 멈춥니다. 그 자리의 값이 아직 초기화되지 않았어도(예: var로 인한 undefined, 또는 let/const의 TDZ) 조회는 거기서 멈추고 바깥으로 나가지 않습니다. 이것이 섀도잉이 만드는 함정입니다.

오해 3

"블록({})은 항상 새로운 스코프를 만든다"

let/const/class에게는 맞지만, var와 함수 선언(비-strict 모드의 일부 레거시 동작 제외)에게는 블록이 새로운 스코프가 아닙니다. "블록 스코프"라는 표현은 let/const 기준이며, var는 여전히 함수 스코프라는 것을 구분해야 합니다.

오해 4

"자바스크립트도 결국 호출 스택을 보고 변수를 찾으니 동적 스코프 아닌가?"

호출 스택(Call Stack, Day 2)은 "어떤 함수가 어떤 함수를 호출했는가"의 실행 순서를 관리하는 자료구조이고, 스코프 체인은 "변수 이름을 찾는 경로"를 관리하는 별개의 개념입니다. 둘을 혼동하면 안 됩니다 — 자바스크립트의 변수 조회는 호출 스택이 아니라 코드가 정의된 위치(렉시컬 구조)만을 따라갑니다.
