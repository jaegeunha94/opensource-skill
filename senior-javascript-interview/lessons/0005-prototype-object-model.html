1. 핵심 면접 질문

Primary Question

"다음 코드에서 d.speak()는 무엇을 출력하고, 왜 그렇게 되나요? 그리고 class Dog extends Animal이 내부적으로 어떤 객체 구조를 만드는지 프로토타입 체인으로 그려서 설명해주세요."

function Animal(name) {
  this.name = name;
}
Animal.prototype.speak = function () {
  return `${this.name} makes a noise.`;
};

class Dog extends Animal {
  speak() {
    return `${this.name} barks.`;
  }
}

const d = new Dog('Rex');
console.log(d.speak());
console.log(Object.getPrototypeOf(d) === Dog.prototype);
console.log(Object.getPrototypeOf(Dog.prototype) === Animal.prototype);
Variant 1

"ES5 생성자 함수 + prototype 방식과 ES6 class 방식이 결과적으로 완전히 동일한가요, 아니면 실제로 다른 점이 있나요?"

Variant 2

"Object.create(null)로 만든 객체와 {}로 만든 객체는 무엇이 다른가요? 실무에서 Object.create(null)을 언제 쓰나요?"

Variant 3

"다음 코드의 세 console.log는 각각 무엇을 출력할까요?"

const base = { greeting: 'hello' };
const obj = Object.create(base);

obj.greeting = 'hi';
console.log(obj.greeting);   // ?
console.log(base.greeting);  // ?

delete obj.greeting;
console.log(obj.greeting);   // ?

2. 왜 면접에서 묻는가

"자바스크립트는 프로토타입 기반 언어다"라는 문장은 누구나 말할 수 있습니다. 하지만 대부분의 실무 개발자는 하루종일 class 문법만 쓰고, 그 아래에서 실제로 어떤 객체 구조가 만들어지는지 한 번도 들여다보지 않습니다. 시니어 인터뷰가 이 주제를 묻는 이유는, class라는 "익숙한 포장지"를 벗겨내고 그 안의 실제 동작(프로토타입 체인을 통한 속성 조회, this 바인딩, 메모리 공유 구조)을 원리로 설명할 수 있는지를 확인하기 위함입니다.

면접관이 이 질문으로 평가하는 것:

  • 속성 조회가 "복사"가 아니라 "체인을 따라 위로 찾아 올라가는 알고리즘"이라는 것을 아는가 — 인스턴스에 없는 속성/메서드를 호출하면 엔진이 [[Prototype]] 링크를 따라 얼마나 올라가서 찾는지 단계별로 설명할 수 있는가.
  • class가 새로운 상속 모델이 아니라 기존 프로토타입 메커니즘의 문법 설탕이라는 것을 아는가 — class 문법이 감추고 있는 prototype 객체, constructor, extends가 실제로 무엇을 연결하는지 설명할 수 있는가. 단, 완전히 동일하지는 않다는 것(TDZ, strict mode, non-enumerable 메서드, new 없이 호출 불가)까지 구분하는가.
  • 속성 섀도잉(shadowing)과 프로토타입 체인 조회를 구분하는가 — 인스턴스에 같은 이름의 속성을 쓰면 프로토타입의 값을 "덮어쓰는" 것이 아니라 인스턴스 자신의 새 속성이 생겨 체인 조회를 가리는(shadow) 것이라는 차이를 아는가.
  • 메모리·성능 관점에서 프로토타입의 역할을 판단할 수 있는가 — 메서드를 프로토타입에 두는 것이 왜 인스턴스마다 메서드를 복사하는 것보다 효율적인지, 반대로 체인이 길어지면 조회 비용이 어떻게 되는지 설명할 수 있는가.
  • Object.create처럼 프로토타입 체인을 직접 다루는 저수준 API를 실무에서 왜/언제 쓰는지 아는가 — 순수 사전(dictionary) 객체를 만들 때, 또는 class 없이 상속 구조를 구성해야 할 때의 판단력.

10년차 이하에서 자주 나오는 답변 패턴: "class가 상속을 지원한다"까지만 말하고, 그 class가 실제로 어떤 객체를 만들어내는지, 인스턴스와 prototype과 constructor가 어떻게 서로를 참조하는지는 그려내지 못하는 답변입니다. 시니어는 이 구조를 화이트보드에 실제로 그릴 수 있어야 합니다.

3. 전제 개념 — 쉬운 한국어로

프로토타입(Prototype)이란 무엇인가 — "물려받는 것"이 아니라 "빌려 쓰는 것"

클래스 기반 언어(Java, C# 등)에서 상속은 "부모 클래스의 내용을 자식 클래스에 복사해서 붙여넣는" 것에 가깝게 동작합니다(정확히는 컴파일 시점에 구조가 결정됩니다). 자바스크립트는 다릅니다. 모든 객체는 [[Prototype]]이라는 숨겨진 내부 링크로 다른 객체 하나(또는 null)를 가리킵니다. 어떤 속성이나 메서드를 인스턴스에서 찾을 때, 자바스크립트 엔진은 그 값을 인스턴스에 "복사해서 갖고 있는지" 확인하는 것이 아니라, 먼저 인스턴스 자신을 보고, 없으면 [[Prototype]]이 가리키는 객체로 이동해서 다시 찾고, 그래도 없으면 또 그 위의 [[Prototype]]으로 이동하는 과정을 null에 도달할 때까지 반복합니다. 이 링크로 연결된 객체들의 연쇄를 프로토타입 체인(Prototype Chain)이라 부릅니다.

비유 — 회사의 "위임 결재선"

신입 사원(인스턴스)에게 결정 권한이 없는 안건이 오면, 그 사람은 먼저 자기 권한 안에서 처리할 수 있는지 봅니다(자기 자신의 속성 확인). 처리할 수 없으면 바로 위 팀장(프로토타입)에게 결재를 올립니다. 팀장도 처리 권한이 없으면 그 위 본부장에게 다시 올라갑니다. 최상위 결재선(Object.prototype, 그리고 그 위는 null)까지 갔는데도 처리할 수 없으면 그 안건은 반려(undefined 또는 TypeError)됩니다. 중요한 것은 신입 사원이 팀장의 권한을 "복사해서 갖고 있는" 게 아니라, 필요할 때마다 결재선을 타고 올라가 위임받아 처리한다는 점입니다.

"객체 리터럴"과 "프로토타입"의 관계 — 이미 매 순간 프로토타입을 쓰고 있다

const obj = {}처럼 아무 생각 없이 만든 객체조차 프로토타입 체인 위에 있습니다. obj.toString()이 동작하는 이유는 obj 자신에게 toString이라는 메서드가 있어서가 아니라, obj의 [[Prototype]]이 Object.prototype을 가리키고, 그 Object.prototype에 toString이 정의되어 있기 때문입니다. 즉 자바스크립트에서 "평범한 객체"란 존재하지 않고, 모든 객체는 이미 어떤 프로토타입 체인 위에 놓여있는 것입니다.

실무 연결 — 이 개념이 왜 프로덕션에서 중요한가

프로토타입 체인은 class 문법 뒤에 숨어 실무 코드 어디에나 존재합니다. 상속 구조를 잘못 설계하면(예: 체인을 너무 깊게 쌓거나, 프로토타입 오염(prototype pollution) 취약점을 만들거나) 예상치 못한 속성이 갑자기 "생겨나는" 버그, 혹은 보안 취약점으로 이어집니다. 반대로 프로토타입 체인이 메서드를 인스턴스 간에 공유하는 원리를 이해하면, 수만 개의 인스턴스를 만들 때 메모리를 어떻게 아끼는지, 왜 메서드를 생성자 함수 안에서 this.method = function() {...}로 정의하면 안 되는지를 원리로 설명할 수 있습니다.

4. Prototype Chain의 실제 동작 — 속성 조회 알고리즘

function Animal(name) {
  this.name = name; // 인스턴스 자신의 속성(own property)
}
Animal.prototype.speak = function () {
  return `${this.name} makes a noise.`;
};

const a = new Animal('Generic');
console.log(a.speak()); // "Generic makes a noise."
단계 속성 조회 알고리즘 (a.speak를 찾는 과정)
1. a 자신의 own property를 확인 a는 { name: 'Generic' }만 자신의 속성으로 갖고 있습니다. speak는 a 자신에게 없습니다.
2. a의 [[Prototype]]으로 이동 new Animal(...)로 인스턴스를 만들면, 엔진은 자동으로 a의 [[Prototype]]을 Animal.prototype으로 연결합니다. 즉 Object.getPrototypeOf(a) === Animal.prototype입니다.
3. Animal.prototype에서 speak를 발견 Animal.prototype.speak가 정의되어 있으므로 여기서 조회가 멈추고 이 함수가 호출됩니다. 이때 this는 여전히 a를 가리킵니다(호출 시점의 수신자 기준 — Day 6에서 다룰 규칙).
4. (발견되지 않았다면) Animal.prototype의 [[Prototype]]인 Object.prototype으로 계속 이동 만약 speak가 Animal.prototype에도 없었다면, 조회는 Object.prototype까지 올라가고, 거기에도 없으면 Object.prototype의 [[Prototype]]인 null에 도달해 조회가 끝나 undefined가 됩니다.

섀도잉(Shadowing) — Variant 3 문제 풀이

const base = { greeting: 'hello' };
const obj = Object.create(base); // obj의 [[Prototype]]을 base로 설정

obj.greeting = 'hi';
console.log(obj.greeting);   // "hi"   — obj 자신의 새 own property가 생김
console.log(base.greeting);  // "hello" — base는 전혀 변경되지 않음

delete obj.greeting;
console.log(obj.greeting);   // "hello" — own property가 사라져 프로토타입 체인의 base.greeting이 다시 보임

핵심 원리 — 대입(=)은 체인을 타고 올라가 수정하지 않는다

obj.greeting = 'hi'는 프로토타입 체인을 올라가 base.greeting을 찾아서 그 값을 바꾸는 것이 아닙니다. 대입 연산은 (일반 데이터 속성인 경우) 항상 obj 자신에게 새로운 own property를 생성합니다. 이렇게 생긴 obj.greeting은 앞으로의 조회에서 프로토타입 체인을 타고 올라갈 필요조차 없이 1단계에서 바로 발견되므로, base.greeting을 가려버립니다(shadow). 반대로 조회(읽기)는 own property가 없을 때만 체인을 타고 올라갑니다. 이 "쓰기는 항상 자신에게, 읽기는 없으면 체인을 타고 올라간다"는 비대칭이 프로토타입 모델의 핵심입니다. delete obj.greeting으로 own property를 없애면, 그 이후 조회는 다시 체인을 타고 올라가 base.greeting을 찾아냅니다.

5. class는 무엇의 syntactic sugar인가

Primary Question의 코드를 ES5 방식으로 완전히 풀어서 써보면 class가 실제로 무엇을 하는지 보입니다.

// class Dog extends Animal { speak() {...} } 와 동일한 구조를 ES5로 직접 구성
function Dog(name) {
  Animal.call(this, name); // super(name)에 해당
}

// 1) Dog.prototype의 [[Prototype]]을 Animal.prototype으로 연결 (extends가 하는 일)
Object.setPrototypeOf(Dog.prototype, Animal.prototype);

// 2) Dog.prototype에 메서드를 정의 (class 바디의 메서드가 가는 곳)
Dog.prototype.speak = function () {
  return `${this.name} barks.`;
};

// 3) Dog 자신도 정적 메서드 상속을 위해 [[Prototype]]이 Animal을 가리킴
Object.setPrototypeOf(Dog, Animal);

const d = new Dog('Rex');
console.log(d.speak());                                     // "Rex barks."
console.log(Object.getPrototypeOf(d) === Dog.prototype);              // true
console.log(Object.getPrototypeOf(Dog.prototype) === Animal.prototype); // true
class 문법 실제로 연결되는 프로토타입 구조
class Dog extends Animal Object.setPrototypeOf(Dog.prototype, Animal.prototype) — 인스턴스 메서드 상속 체인을 연결. 동시에 Object.setPrototypeOf(Dog, Animal)도 일어나 정적(static) 메서드 상속도 연결됩니다.
constructor(name) { super(name); ... } Animal.call(this, name)과 유사하게 부모 생성자를 현재 this로 호출. 단 super()를 호출하기 전에는 this에 접근할 수 없다는 제약이 추가됩니다(파생 클래스의 this는 부모 생성자가 만들어줘야 하기 때문).
클래스 바디의 speak() {...} Dog.prototype.speak = function() {...}과 같은 위치에 정의되지만, 기본적으로 enumerable: false로 정의됩니다(for...in이나 Object.keys(Dog.prototype)에 노출되지 않음). 일반 함수 표현식으로 프로토타입에 붙인 메서드는 기본적으로 enumerable이라는 점에서 다릅니다.

완전히 같지는 않다 — class가 추가로 강제하는 것들

  • Temporal Dead Zone(TDZ) — class 선언은 let/const처럼 호이스팅되지만 선언 이전에 접근하면 ReferenceError가 발생합니다(Day 2에서 다룬 TDZ와 동일 원리). 함수 선언(function Animal() {})은 선언 전에도 호출 가능한 것과 다릅니다.
  • 강제 strict mode — 클래스 바디 내부는 항상 strict mode로 실행됩니다. 실수로 this가 전역 객체를 가리키는 등의 non-strict 동작이 발생하지 않습니다.
  • new 없이 호출 불가 — Dog('Rex')처럼 new 없이 호출하면 TypeError가 발생합니다. 반면 일반 생성자 함수는 new 없이 호출해도 에러 없이 조용히 잘못된 동작(예: 전역 객체 오염)을 할 수 있습니다.
  • 메서드가 non-enumerable — 위 표에서 언급한 대로, class로 만든 프로토타입 메서드는 for...in으로 순회되지 않습니다. 이는 실무에서 "왜 이 객체를 순회했더니 상속받은 메서드까지 나온다/안 나온다"는 혼란을 줄여주는 의도적 설계입니다.

실무 연결 — "완전히 동일한 문법 설탕"이 아니라는 것을 아는 것의 가치

레거시 코드(특히 오래된 라이브러리나 프레임워크 내부 코드)에서는 여전히 생성자 함수 + prototype 패턴이 남아있습니다. class와 완전히 같다고 착각하면, TDZ로 인한 초기화 순서 버그나 new 강제 여부 차이로 인한 런타임 동작 차이를 디버깅할 때 원인을 잘못 짚게 됩니다. "class는 대부분의 경우 기존 프로토타입 메커니즘 위에 놓인 문법 설탕이지만, 몇 가지 안전장치를 스펙 차원에서 추가로 강제한다"는 정확한 이해가 필요합니다.

6. Object.create로 체인을 직접 다루기

const proto = {
  greet() {
    return `Hello, ${this.name}`;
  },
};

const person = Object.create(proto); // [[Prototype]]을 proto로 지정한 새 객체 생성
person.name = 'Alice';
console.log(person.greet()); // "Hello, Alice"

console.log(Object.getPrototypeOf(person) === proto); // true

Object.create(proto)는 class나 생성자 함수 없이, 어떤 객체를 프로토타입으로 삼을지 직접 지정해서 새 객체를 만듭니다. class/생성자 함수는 결국 이 저수준 메커니즘 위에 관습적인 문법을 얹은 것입니다.

Object.create(null) — 프로토타입이 없는 "진짜 빈" 객체

const dict = Object.create(null);
dict.toString = 'a string value'; // 문제 없음 — Object.prototype.toString과 충돌하지 않음
console.log(dict.hasOwnProperty); // undefined — Object.prototype이 아예 없으므로 상속된 메서드가 없음
console.log(Object.getPrototypeOf(dict)); // null
비교 {} (또는 Object.create(Object.prototype)) Object.create(null)
[[Prototype]] Object.prototype을 가리킴 null — 체인이 아예 없음
toString, hasOwnProperty, valueOf 등 모두 상속받아 사용 가능 전혀 없음 — 직접 Object.prototype.hasOwnProperty.call(dict, key)처럼 호출해야 함
사용 사례 일반적인 객체 외부에서 들어온 키를 그대로 저장하는 "순수 사전(dictionary)". toString, constructor 같은 이름의 키가 들어와도 프로토타입 메서드와 충돌하거나 덮어써질 위험이 없음

실무 연결 — 왜 "진짜 빈" 객체가 필요한가

사용자 입력이나 외부 데이터를 키로 삼아 맵처럼 쓰는 객체(예: 캐시, 카운터, 룩업 테이블)를 {}로 만들면, 우연히 "constructor", "toString", "__proto__" 같은 키가 들어왔을 때 예상치 못하게 프로토타입 체인의 기존 값과 충돌하거나(과거 "prototype pollution" 계열 취약점의 근본 원인 중 하나), obj["__proto__"] = maliciousObj 같은 조작으로 객체의 프로토타입 자체가 바뀌는 보안 이슈로 이어질 수 있습니다. Object.create(null)은 애초에 체인이 없으므로 이런 충돌 가능성을 구조적으로 차단합니다. 다만 이제는 이런 용도에는 프로토타입 오염 위험이 아예 없는 Map을 쓰는 것이 더 널리 권장되는 실무 관행입니다.

7. 실무 연결 — 메모리, Mixin, 성능

메서드를 프로토타입에 두는 것이 왜 메모리를 아끼는가

// 안티패턴 — 인스턴스마다 메서드를 새로 생성
function BadAnimal(name) {
  this.name = name;
  this.speak = function () { // 인스턴스를 만들 때마다 새 함수 객체가 생성됨
    return `${this.name} makes a noise.`;
  };
}

// 좋은 패턴 — 프로토타입에 메서드를 한 번만 정의
function GoodAnimal(name) {
  this.name = name;
}
GoodAnimal.prototype.speak = function () { // 모든 인스턴스가 이 하나의 함수 객체를 공유
  return `${this.name} makes a noise.`;
};

BadAnimal 방식은 인스턴스를 10만 개 만들면 10만 개의 서로 다른 speak 함수 객체가 메모리에 생깁니다(참고로 이 함수들은 매번 새 클로저이기도 합니다 — Day 4). GoodAnimal 방식은 Animal.prototype.speak이라는 단 하나의 함수 객체를 모든 인스턴스가 프로토타입 체인을 통해 공유합니다. 대량의 인스턴스를 만드는 코드(예: 리스트 렌더링에서 각 행마다 만드는 뷰모델 객체)에서 이 차이는 실제 메모리 사용량에 직접적인 영향을 줍니다. class 문법을 쓰면 메서드가 자동으로 프로토타입에 정의되므로 이 안티패턴을 방지해주지만, 클로저로 프라이빗 상태를 흉내내려다 메서드까지 인스턴스 필드로 옮기는 실수는 여전히 발생합니다.

Mixin — 단일 상속의 한계를 우회하는 실무 패턴

const CanFly = (Base) => class extends Base {
  fly() { return `${this.name} flies.`; }
};
const CanSwim = (Base) => class extends Base {
  swim() { return `${this.name} swims.`; }
};

class Animal {
  constructor(name) { this.name = name; }
}
class Duck extends CanSwim(CanFly(Animal)) {}

const duck = new Duck('Donald');
console.log(duck.fly());  // "Donald flies."
console.log(duck.swim()); // "Donald swims."

자바스크립트는 클래스 하나가 오직 하나의 부모만 extends할 수 있는 단일 상속 모델입니다. 여러 "능력"을 조합해야 할 때는 이렇게 클래스를 인자로 받아 확장된 클래스를 반환하는 함수(Mixin)를 체인처럼 감싸서, 실질적으로 여러 겹의 프로토타입 체인을 만들어 다중 상속과 유사한 효과를 냅니다. 각 Mixin이 만드는 중간 클래스가 프로토타입 체인의 한 칸씩을 차지한다는 것을 이해하면, 이 패턴이 "새로운 상속 모델"이 아니라 Section 4의 체인 조회 알고리즘을 여러 겹 쌓아 활용하는 것임을 알 수 있습니다.

성능 예고 — 체인이 길어지면 조회 비용이 늘어난다

Section 4의 알고리즘대로, 인스턴스에 없는 속성을 찾을 때마다 엔진은 체인을 한 칸씩 타고 올라갑니다. Mixin을 여러 겹 쌓거나 상속 계층을 깊게 만들면, 자주 호출되는 메서드일수록 체인 최하단에 두는 것이 이론상 유리합니다. 다만 V8 같은 실제 엔진은 "히든 클래스(Hidden Class)"와 인라인 캐싱(Inline Caching) 최적화로 이 비용을 상당 부분 상쇄하므로, "체인이 길면 무조건 느리다"고 단순화하면 안 됩니다. 이 최적화의 구체적인 동작과 실제 프로파일링 방법은 Day 17(Performance Profiling)과 Day 18(Garbage Collection 심화)에서 다룹니다.

8. 답변 사고 순서

  1. 인스턴스 자신이 가진 own property부터 확인합니다. 질문에 등장하는 속성/메서드가 생성자 함수(또는 constructor) 안에서 this.x = ... 형태로 만들어진 것인지 먼저 구분합니다.
  2. own property에 없다면 [[Prototype]] 체인을 한 칸씩 따라 올라갑니다. 어떤 객체가 Object.getPrototypeOf로 연결되어 있는지 순서대로 그립니다(인스턴스 → 생성자의 prototype → 그 위 부모의 prototype → ... → Object.prototype → null).
  3. class가 쓰였다면 이 체인이 extends에 의해 어떻게 연결됐는지로 번역합니다. "class라서 특별한 상속"이 아니라 "Object.setPrototypeOf로 체인이 연결된 것"이라고 설명합니다.
  4. 쓰기(대입) 연산이 섞여 있다면 섀도잉 규칙을 적용합니다. 대입은 항상 자신에게 own property를 만든다는 규칙으로, 프로토타입 값이 실제로 바뀌었는지 아니면 가려졌을 뿐인지 구분합니다.
  5. 필요하다면 메모리·성능 함의까지 확장합니다. 이 구조가 인스턴스 간 메서드 공유, 혹은 체인 깊이에 따른 조회 비용과 어떻게 연결되는지 짚습니다.

9. 핵심 답변 예시

면접관: "Primary Question 코드에서 d.speak()는 무엇을 출력하고, class Dog extends Animal이 만드는 프로토타입 구조를 설명해주세요."

STRONG ANSWER

d.speak()는 "Rex barks."를 출력합니다. 그 이유는 자바스크립트의 속성 조회가 인스턴스 자신부터 시작해 [[Prototype]] 체인을 따라 올라가는 알고리즘이기 때문입니다. d는 Dog의 인스턴스이므로 먼저 d 자신에게 speak가 있는지 봅니다. 없으므로 d의 [[Prototype]]인 Dog.prototype으로 이동하는데, class Dog 바디에 정의한 speak 메서드가 바로 여기, Dog.prototype.speak에 있으므로 여기서 조회가 멈추고 이 메서드가 실행됩니다.


class Dog extends Animal은 새로운 상속 모델이 아니라, 기존 생성자 함수 + prototype 패턴을 문법적으로 정리한 것입니다. 구체적으로 extends는 Object.setPrototypeOf(Dog.prototype, Animal.prototype)과 동등한 연결을 만듭니다. 그래서 Object.getPrototypeOf(d) === Dog.prototype이 true이고, Object.getPrototypeOf(Dog.prototype) === Animal.prototype도 true입니다. 즉 체인은 d → Dog.prototype → Animal.prototype → Object.prototype → null 순서로 이어집니다. 만약 Dog가 speak를 오버라이드하지 않았다면, 조회는 Dog.prototype에서 멈추지 않고 한 칸 더 올라가 Animal.prototype.speak("Rex makes a noise.")를 실행했을 것입니다.


다만 class가 기존 프로토타입 메커니즘과 완전히 동일하지는 않습니다. class 선언은 TDZ의 적용을 받아 선언 전 참조 시 ReferenceError가 나고, 클래스 바디는 항상 strict mode로 실행되며, new 없이 호출하면 TypeError가 발생하고, 프로토타입에 정의된 메서드는 기본적으로 non-enumerable입니다 — 이런 안전장치들은 생성자 함수 방식에는 스펙상 강제되지 않습니다.

이 답변이 좋은 이유:

  • 결과(출력값)만 말하지 않고 체인 조회 알고리즘을 단계별로 그려서 근거를 댐
  • class를 "새로운 개념"이 아니라 기존 프로토타입 모델의 문법 설탕으로 정확히 위치시킴
  • Object.getPrototypeOf 검증 코드가 왜 true인지까지 연결해 설명함
  • "완전히 동일하지는 않다"는 것까지 짚어 표면적 암기가 아님을 보여줌

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"'Rex barks.'가 나와요. Dog가 Animal을 상속받아서 speak를 오버라이드했으니까요."

이 답변이 약한 이유

  • "상속받아서 오버라이드했다"는 결과 서술일 뿐, 실제로 어떤 객체 구조([[Prototype]] 링크)가 만들어졌고, 조회가 어떤 순서로 진행되는지를 전혀 설명하지 못합니다.
  • 변형 질문(Object.getPrototypeOf(d) === Dog.prototype이 왜 true인지, class와 생성자 함수가 완전히 같은지)에 막힐 가능성이 높습니다.
WEAK ANSWER — 또 다른 패턴

"class는 완전히 새로운 문법이라 ES5의 프로토타입이랑은 별개로 동작해요."

이 답변이 약한 이유

  • 사실과 반대입니다. class는 여전히 [[Prototype]] 체인 위에서 동작하는 문법 설탕입니다. 이렇게 답하면 "그럼 Object.getPrototypeOf(Dog.prototype)은 무엇을 가리키나요?" 같은 후속 질문에 전혀 대응하지 못합니다.
STRONG ANSWER 요소 체크리스트
  • 속성 조회를 "own property 확인 → 없으면 [[Prototype]]을 따라 이동 → 반복"이라는 알고리즘으로 설명한다
  • class가 기존 프로토타입 메커니즘 위의 문법 설탕임을 extends → Object.setPrototypeOf 대응 관계로 설명한다
  • 실제 체인 구조(d → Dog.prototype → Animal.prototype → ...)를 순서대로 그릴 수 있다
  • class와 생성자 함수 방식의 차이점(TDZ, strict mode, new 강제, non-enumerable)까지 구분한다

11. Senior-level Follow-up Q&A

Q: obj.__proto__와 Object.getPrototypeOf(obj)는 같은 것을 반환하나요? 실무에서 어떤 것을 써야 하나요?

모범 답변: 둘 다 결과적으로 같은 [[Prototype]] 링크를 반환하지만, __proto__는 원래 특정 엔진의 비표준 구현이 관행적으로 굳어져 나중에 스펙에 "레거시 호환용"으로만 표준화된 접근자(accessor)입니다. Object.getPrototypeOf() / Object.setPrototypeOf()가 명시적으로 스펙에 정의된 정식 API이므로 실무 코드에서는 이쪽을 써야 합니다. __proto__는 레거시 코드를 읽을 때 알아야 하는 지식이지, 새 코드에 쓸 이유는 없습니다.

Q: for...in으로 객체를 순회하면 프로토타입 체인의 속성까지 나오나요?

모범 답변: 네, for...in은 own property뿐 아니라 enumerable한 상속 속성까지 순회 대상에 포함합니다. 그래서 실무에서는 for...in 안에 if (Object.hasOwn(obj, key))(또는 obj.hasOwnProperty(key)) 체크를 넣어 own property만 거르는 관용구를 씁니다. Object.keys(), Object.entries(), Object.values()는 처음부터 own enumerable property만 반환하도록 설계되어 이런 실수를 구조적으로 방지하므로, 신규 코드에서는 대체로 이쪽이 더 안전한 선택입니다. 또한 Section 5에서 다뤘듯 class로 만든 메서드는 애초에 non-enumerable이라 이 문제에서 자유롭습니다.

Q: 프로토타입 체인을 실행 중에 바꾸는 Object.setPrototypeOf를 자주 써도 괜찮나요?

모범 답변: 권장하지 않습니다. 대부분의 자바스크립트 엔진은 객체가 생성된 시점의 프로토타입 구조를 기반으로 내부 최적화(V8의 히든 클래스 등)를 수행하는데, 객체 생성 이후에 [[Prototype]]을 바꾸면 이 최적화 경로를 이탈시켜 해당 객체와 관련된 코드 경로 전체가 비최적화(deoptimize)될 수 있습니다. 상속 구조는 Object.create나 class/extends로 객체 생성 시점에 확정하는 것이 정석이며, setPrototypeOf는 프레임워크 내부 구현 등 꼭 필요한 경우로 한정해야 합니다.

Q: 프로토타입 오염(Prototype Pollution) 취약점이 정확히 무엇이고, 이 프로토타입 모델과 어떻게 연결되나요?

모범 답변: 외부 입력(예: JSON body)을 재귀적으로 병합(merge)하는 코드가 키 이름을 검증 없이 그대로 써서 obj["__proto__"]["polluted"] = true 같은 대입을 허용하면, 공격자는 Object.prototype 자체에 속성을 주입할 수 있습니다. Object.prototype은 애플리케이션의 모든 일반 객체가 체인의 최상단에서 공유하는 대상이므로, 이 오염은 전혀 관련 없어 보이는 코드 경로의 동작까지 조용히 바꿔버립니다(예: if (obj.isAdmin) 같은 체크가 프로토타입에 주입된 값 때문에 모든 객체에서 true가 되는 식). 병합 유틸리티에서 __proto__, constructor, prototype 같은 키를 명시적으로 차단하거나,애초에 신뢰할 수 없는 키-값 저장에는 Object.create(null) 기반 객체나 Map을 쓰는 것이 근본적인 방어입니다.

12. 흔한 오해

오해 1

"인스턴스는 프로토타입의 메서드를 자기 것으로 복사해서 갖는다"

복사가 일어나지 않습니다. 인스턴스는 프로토타입 객체에 대한 참조([[Prototype]] 링크)만 가지고 있고, 메서드를 호출할 때마다 그 링크를 따라가 조회합니다. 그래서 Animal.prototype.speak를 나중에 바꾸면, 이미 만들어진 인스턴스들도 즉시 바뀐 동작을 보게 됩니다 — 복사본이었다면 불가능한 일입니다.

오해 2

"class는 ES5의 프로토타입 상속과 완전히 다른, 진짜 클래스 기반 모델을 도입했다"

실제로는 여전히 같은 [[Prototype]] 체인 위에서 동작하는 문법 설탕입니다. Section 5에서 다룬 TDZ, strict mode, new 강제, non-enumerable 메서드 같은 추가 안전장치는 있지만, 속성 조회 알고리즘 자체(Section 4)는 class를 쓰든 생성자 함수를 쓰든 동일합니다.

오해 3

"프로토타입 체인이 길면 무조건 성능이 나쁘다"

이론적으로 체인이 길수록 조회에 필요한 단계가 늘어나는 것은 맞지만, V8 등 실제 엔진은 히든 클래스(Hidden Class)와 인라인 캐싱(Inline Caching)으로 반복 호출되는 조회 경로를 캐싱해 이 비용을 크게 상쇄합니다. "체인이 길면 항상 느리다"고 단정하기보다, 실제로 병목이 의심될 때 Day 17의 프로파일링 절차로 측정해서 확인하는 것이 시니어다운 태도입니다.

오해 4

"obj.prop = value는 프로토타입 체인에 있는 기존 값을 찾아서 그 자리를 수정한다"

Section 4에서 다뤘듯, 일반적인 데이터 속성 대입은 항상 대상 객체 자신에게 새 own property를 생성합니다(체인 위쪽의 값을 수정하지 않고 가릴 뿐입니다). 체인 위쪽의 값이 실제로 수정되는 것은 그 속성이 setter가 정의된 접근자 속성(accessor property)인, 상대적으로 드문 경우뿐입니다.
