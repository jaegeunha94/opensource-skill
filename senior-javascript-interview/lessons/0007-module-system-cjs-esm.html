1. 핵심 면접 질문

Primary Question

"다음은 Node.js 공식 문서에 나오는 순환 참조(circular dependency) 예제를 CommonJS로 작성한 것입니다. 실행했을 때 콘솔에 무엇이 출력되는지, 그리고 왜 b.js 안에서 a.done이 false로 찍히는지 설명해주세요. 같은 문제를 ESM으로 작성하면 결과가 왜 달라질 수 있나요?"

// main.js
require('./a.js');

// a.js
console.log('a starting');
exports.done = false;
const b = require('./b.js');
console.log('in a, b.done = %j', b.done);
exports.done = true;
console.log('a done');

// b.js
console.log('b starting');
exports.done = false;
const a = require('./a.js');
console.log('in b, a.done = %j', a.done); // ?
exports.done = true;
console.log('b done');
Variant 1

"webpack이나 Rollup 같은 번들러가 ESM에서는 트리쉐이킹으로 사용하지 않는 export를 제거할 수 있는데, CommonJS(require/module.exports)로 작성된 라이브러리는 왜 트리쉐이킹이 잘 안 된다는 말을 듣나요?"

Variant 2

"사내 라이브러리를 npm 패키지로 배포하려고 합니다. 이 패키지를 CommonJS 프로젝트와 ESM 프로젝트 양쪽에서 모두 import/require할 수 있게 만들려면 무엇을 고려해야 하나요?"

Variant 3

"ESM에서 import { count } from './counter.mjs'로 가져온 count가, 원본 모듈에서 값이 바뀌면 가져온 쪽에서도 자동으로 갱신되는 이유는 무엇인가요? CommonJS의 require와 무엇이 근본적으로 다른가요?"

2. 왜 면접에서 묻는가

대부분의 개발자는 import와 require를 "문법이 다를 뿐 결국 같은 일을 하는 것"으로 취급합니다. 실무에서는 이 둘이 모듈을 로드하는 시점(동기 vs 비동기), 내보낸 값이 원본과 연결되는 방식(값 복사 vs 실시간 바인딩), 그리고 그 구조가 정적으로 분석 가능한지 여부가 근본적으로 다릅니다. 시니어 인터뷰가 이 주제를 묻는 이유는, 번들 크기가 비정상적으로 크거나("트리쉐이킹이 안 먹는다"), 순환 참조가 있는 모듈에서 undefined가 튀어나오거나, 라이브러리를 배포했는데 한쪽 생태계에서만 깨지는 문제를 모듈 시스템의 실행 모델 차이로 설명하고 대응할 수 있는지를 확인하기 위함입니다.

면접관이 이 질문으로 평가하는 것:

  • 모듈 로딩이 동기(CJS)인지 비동기·정적(ESM)인지, 그 차이가 왜 생기는지 아는가 — require는 함수 호출이라 조건문 안에서도 쓸 수 있지만, import는 선언이라 파일 최상위에서만, 그리고 실행 전에 먼저 분석된다는 것을 아는가.
  • export된 값이 "복사본"인지 "실시간 참조(live binding)"인지 구분하는가 — CommonJS는 require 시점의 module.exports 스냅샷을 가져오고, ESM은 원본 바인딩 자체를 참조해서 나중에 바뀌어도 갱신된다는 차이를 아는가.
  • 순환 참조가 왜, 그리고 어떻게 다르게 깨지는지 설명할 수 있는가 — CJS는 순환이 발생하면 "그 시점까지 완성된 부분"만 돌려주는 반면, ESM은 바인딩을 미리 연결해두고 나중에 값이 채워지는 방식이라는 원리를 아는가.
  • 트리쉐이킹이 왜 ESM 생태계의 산물인지 설명할 수 있는가 — "정적 분석이 가능하다"는 표현을 실제로 무엇을 의미하는지(실행하지 않고도 import/export 그래프를 결정할 수 있다는 것) 풀어서 설명할 수 있는가.
  • 실무 배포/빌드 문제로 연결할 수 있는가 — dual package(CJS+ESM 동시 배포), package.json의 "type"/exports 필드, 번들러 interop이 왜 필요한지 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "ESM이 최신 문법이고 CommonJS는 옛날 Node 방식"이라는 표면적 구분까지만 말하고, 왜 ESM에서만 트리쉐이킹이 안정적으로 되는지, 순환 참조에서 실제로 어떤 값이 찍히는지는 설명하지 못하는 답변입니다. 시니어는 두 시스템의 실행 모델 차이를 원인으로 삼아 결과를 예측할 수 있어야 합니다.

3. 전제 개념 — 쉬운 한국어로

모듈은 "파일 분리"가 아니라 "독립된 스코프 + 의존성 계약"이다

모듈 시스템이 없던 시절(초기 브라우저 JS)에는 <script> 태그로 파일을 여러 개 불러오면 모든 코드가 하나의 전역 스코프를 공유했습니다. A 파일에서 선언한 변수 이름을 B 파일이 실수로 덮어써도 아무 에러 없이 조용히 버그가 생겼습니다. 모듈 시스템의 핵심 목적은 각 파일이 자기만의 스코프를 갖게 하고, "이 모듈이 무엇을 밖으로 내보내는지(export)"와 "이 모듈이 무엇을 필요로 하는지(import)"를 명시적인 계약으로 만드는 것입니다.

비유 — 밀폐된 방과 창구

모듈이 없던 시절은 벽 없는 사무실 하나에 모든 팀이 함께 앉아 있는 것과 같습니다. 누구나 남의 책상(변수)을 건드릴 수 있습니다. 모듈 시스템은 각 팀을 독립된 방(스코프)에 넣고, 방마다 "내보내는 창구(export)"와 "필요한 것을 요청하는 창구(import)"만 열어둔 것입니다. CommonJS와 ESM의 차이는, 이 창구가 언제, 어떻게 열리고 정보가 오가는지에 있습니다 — 이것이 이번 레슨 전체를 관통하는 축입니다.

"동기/비동기"와 "정적/동적"은 다른 축이다

CommonJS(require)는 동기적이고 동적입니다. 파일을 읽어 그 자리에서 즉시 실행하고 결과를 반환하며, require는 평범한 함수 호출이라 if 문 안에서도, 변수로 만든 경로 문자열로도 호출할 수 있습니다. ESM(import)은 정적입니다 — import 선언은 반드시 파일 최상위에 있어야 하고, 경로도 문자열 리터럴이어야 합니다. 그 덕분에 엔진/번들러는 코드를 한 줄도 실행하지 않고도 "이 파일이 어떤 파일들을 필요로 하는지"의 전체 그래프를 미리 그릴 수 있습니다. 이 "정적으로 분석 가능하다"는 성질이 Section 6(트리쉐이킹)의 근거가 됩니다.

실무 연결 — 이 개념이 왜 프로덕션에서 중요한가

실무에서 모듈 시스템 문제는 대부분 "왜 번들 크기가 안 줄어들지?", "왜 이 라이브러리를 import하니 순환 참조 경고가 뜨지?", "왜 같은 라이브러리인데 프론트엔드에서는 되고 백엔드 스크립트에서는 require가 안 먹지?" 같은 형태로 나타납니다. 이 문제들은 전부 "CJS와 ESM이 모듈을 로드하고 값을 연결하는 방식이 다르다"는 하나의 원리에서 비롯됩니다. 원리를 모르면 매번 스택오버플로우에서 임시방편(esModuleInterop: true를 이유도 모른 채 켜는 것 등)만 복사하게 됩니다.

4. CommonJS vs ESM 핵심 차이 — 동적/동기 vs 정적/비동기

구분 CommonJS (require/module.exports) ESM (import/export)
로딩 방식 동기적 — require 호출 시점에 파일을 읽고 즉시 실행 비동기적 지원 전제 — 파싱 단계에서 의존성 그래프를 먼저 구성(모듈 레코드 생성) 후 평가(evaluate)
선언 위치 require는 함수 호출 — 조건문/함수 내부 어디서나 가능, 경로를 변수로 계산 가능 import는 선언(statement) — 반드시 파일 최상위, 경로는 문자열 리터럴 고정(동적 로딩은 import() 별도 문법)
분석 시점 런타임에만 실제 의존성이 드러남(동적이므로 실행 전 100% 예측 불가) 파싱 시점에 정적으로 전체 import/export 그래프 확정 가능
내보낸 값의 성격 값 복사 — require 호출 시점의 module.exports 객체(참조)를 가져옴. 원시값을 구조 분해하면 그 시점의 스냅샷. 실시간 바인딩(live binding) — import한 이름은 원본 변수 바인딩 자체를 가리켜, 원본이 나중에 바뀌면 import한 쪽에서도 자동 반영
실행 컨텍스트 모든 코드가 함수 스코프 안에서 즉시 순서대로 실행(top-level await 없음, Node 최신 버전 일부 예외 제외) import/export 선언은 호이스팅되어 평가 전에 바인딩부터 연결. 최상위 await 지원
this(모듈 최상위) module.exports(초기값) undefined

Variant 3 풀이 — 왜 ESM의 import는 "실시간"인가

// counter.mjs
export let count = 0;
export function increment() { count++; }

// main.mjs
import { count, increment } from './counter.mjs';
console.log(count); // 0
increment();
console.log(count); // 1 — count는 "값"이 아니라 counter.mjs의 count 변수에 대한 바인딩 자체다

ESM의 import { count }는 "지금 이 순간의 count 값을 복사해오는 것"이 아니라, counter.mjs 내부의 count라는 변수 자리(바인딩)에 대한 읽기 전용 참조를 만드는 것입니다. 그래서 counter.mjs 안에서 count가 바뀌면, 이를 import한 모든 곳에서 다시 읽을 때 자동으로 최신 값이 보입니다. 반면 CommonJS로 같은 것을 흉내내면(const { count } = require('./counter.js')), count는 require 시점에 복사된 원시값(숫자)일 뿐이라 원본이 나중에 바뀌어도 이 지역 변수는 그대로입니다.

5. 순환 참조(Circular Dependency)가 다르게 처리되는 이유

CommonJS — "그 시점까지 완성된 것"만 돌려준다

Node.js는 require가 처음 호출되는 모듈마다 모듈 캐시에 항목을 만들고, 그 모듈의 코드를 위에서 아래로 순서대로, 동기적으로 실행합니다. 만약 실행 도중 자기 자신을 순환적으로 다시 require하게 되면, Node는 무한 루프에 빠지는 대신 그 시점까지 module.exports에 쌓인 것을 그대로 돌려주고 실행을 계속 진행합니다.

Primary Question 풀이 — 단계별 추적

  1. main.js가 a.js를 require → a.js 실행 시작, "a starting" 출력, exports.done = false 설정.
  2. a.js가 b.js를 require → b.js 실행 시작, "b starting" 출력, exports.done = false 설정.
  3. b.js가 다시 a.js를 require → a.js는 이미 모듈 캐시에 "로딩 중"으로 등록되어 있으므로, Node는 a.js를 처음부터 다시 실행하지 않고, 지금까지 a.js의 module.exports에 쌓인 값({ done: false })을 즉시 돌려준다.
  4. 그래서 b.js의 console.log('in b, a.done = %j', a.done)는 false를 출력한다 — a.js가 아직 exports.done = true까지 도달하지 못했기 때문이다.
  5. b.js는 exports.done = true를 설정하고 "b done"을 출력한 뒤 종료, 제어권이 a.js로 돌아온다.
  6. a.js는 이제 완전히 로드된 b.js의 module.exports({ done: true })를 b 변수로 받으므로, console.log('in a, b.done = %j', b.done)는 true를 출력한다.
  7. a.js는 exports.done = true를 설정하고 "a done"을 출력하며 종료한다.

핵심 원리: CommonJS의 순환 참조는 "실행 순서상 나중에 요청된 쪽이, 상대방이 아직 다 완성하지 못한 부분적인(partial) exports 스냅샷을 받는다"는 규칙으로 항상 설명됩니다. 이때 함수처럼 이미 정의된 프로퍼티는 문제없이 쓸 수 있지만, 아직 할당되지 않은 프로퍼티는 undefined로 보이거나 아예 존재하지 않습니다.

ESM — 바인딩을 먼저 연결하고, 값은 나중에 채운다

ESM은 모듈 로딩을 3단계(Construction → Instantiation → Evaluation)로 나눕니다. Instantiation 단계에서는 아직 어떤 코드도 실행하지 않고, 전체 모듈 그래프를 훑어 모든 import/export 바인딩의 "연결 관계"만 먼저 확정합니다(마치 변수 이름표만 미리 걸어두는 것). 실제 값을 채우는 것은 그다음 Evaluation 단계입니다. 그래서 순환 참조가 있어도, 함수 선언처럼 호이스팅되어 Evaluation 이전에 이미 값이 채워지는 것은 순환 상황에서도 정상적으로 참조할 수 있습니다.

// odd.mjs
import { even } from './even.mjs';
export function odd(n) { return n === 0 ? false : even(n - 1); }

// even.mjs
import { odd } from './odd.mjs';
export function even(n) { return n === 0 ? true : odd(n - 1); }

// main.mjs
import { odd } from './odd.mjs';
console.log(odd(10)); // true — 순환 참조가 있지만 정상 동작
// odd.mjs와 even.mjs가 서로를 import하지만, 함수 선언은 호이스팅되어
// Evaluation 단계 이전에 이미 바인딩이 채워져 있으므로, odd(10)이 실제로
// "호출"되는 시점(main.mjs 실행 시점)에는 양쪽 모듈이 이미 완전히 평가된 뒤다.

단, ESM도 순환 참조에서 안전하지 않은 경우가 있다

let/const로 선언한 값을 모듈 최상위 레벨에서, 아직 초기화되기 전에 순환적으로 읽으려 하면 ReferenceError가 납니다(Day 2에서 다룬 TDZ와 같은 원리). ESM의 순환 참조는 "무조건 안전하다"가 아니라, 실제로 값을 읽는 시점이 그 값이 초기화된 이후인지에 달려 있습니다 — 함수 선언(호이스팅됨)은 안전하지만, 모듈 최상위에서 즉시 실행되는 값 참조는 여전히 위험할 수 있습니다.

비교 항목CommonJSESM
순환 발생 시 반환값 그 시점까지 완성된 module.exports의 부분 스냅샷 모든 바인딩이 먼저 연결되고, 값은 실제 사용 시점에 평가(늦게 읽으면 최신 값)
실패 형태 조용히 undefined를 반환(에러 없이 버그로 이어짐) 초기화 전 접근 시 명시적 ReferenceError(TDZ)
디버깅 난이도 높음 — 어느 시점의 스냅샷인지 실행 순서를 손으로 추적해야 함 상대적으로 낮음 — 에러가 명시적으로 발생하는 경우가 많음(단, 함수 호이스팅 케이스는 조용히 성공하기도 함)

6. 트리쉐이킹이 ESM에서만 안정적으로 가능한 이유

트리쉐이킹은 "번들에 실제로 쓰이지 않는 export를 정적 분석으로 찾아내 제거하는 것"입니다. 이것이 성립하려면 번들러가 코드를 한 줄도 실행하지 않고도 "이 모듈이 무엇을 export하고, 어느 모듈이 그중 무엇을 import해서 실제로 쓰는지"를 100% 확정할 수 있어야 합니다. Section 4에서 다룬 대로 ESM의 import/export는 정적이므로 이 조건을 만족하지만, CommonJS의 require/module.exports는 런타임에만 결정되는 동적 구조이므로 이 조건을 만족하지 못합니다.

CommonJS에서 정적 분석이 실패하는 대표 패턴

// 조건부 require — 번들 시점에는 어떤 분기를 탈지 알 수 없다
const utils = condition ? require('./utils-a') : require('./utils-b');

// 계산된 경로 — 문자열이 런타임에 완성된다
const plugin = require(`./plugins/${pluginName}`);

// module.exports를 런타임에 동적으로 변형
module.exports = { ...baseExports, ...(isDev ? devExtras : {}) };

이런 코드는 파일을 실제로 실행해봐야만 무엇이 최종적으로 export되는지 알 수 있습니다. 번들러가 안전하게 "이 export는 아무도 안 쓰니 지워도 된다"고 판단하려면, 코드를 실행하지 않고도 그 결론이 항상 참이어야 하는데, 위와 같은 동적 패턴에서는 그것을 보장할 수 없습니다. 그래서 번들러는 "혹시 몰라서" 안전한 쪽(제거하지 않음)을 택하게 되고, 결과적으로 죽은 코드가 번들에 그대로 남습니다.

ESM이 정적 분석에 유리한 이유를 한 문장으로

export/import 선언은 파일 최상위에 고정된 문법이고 경로도 문자열 리터럴로 고정되므로, 번들러는 모든 모듈의 소스 텍스트를 파싱하는 것만으로(실행하지 않고) 전체 의존성 그래프와 각 모듈이 실제로 사용하는 export 목록을 정확히 그릴 수 있습니다. 이 "실행 없이 그래프를 확정할 수 있다"는 성질이 트리쉐이킹의 전제조건입니다.

트리쉐이킹을 방해하는 추가 요인 — Side Effect

정적 구조라도 모듈을 import하는 것 자체가 부수효과(side effect, 예: 전역 CSS 삽입, polyfill 등록)를 갖는다면, 번들러는 "export를 안 쓴다고 해서 이 모듈을 통째로 제거해도 되는지" 확신할 수 없습니다. 그래서 라이브러리는 package.json에 "sideEffects": false(또는 부수효과가 있는 파일만 배열로 명시)를 선언해, "이 패키지의 모듈들은 import만으로는 아무 부수효과가 없으니 안전하게 제거해도 된다"는 것을 번들러에게 명시적으로 보장합니다.

7. 번들러가 다루는 경계 — Interop, Dual Package Hazard

실무 코드베이스와 npm 생태계에는 CommonJS와 ESM이 동시에 존재합니다. 번들러/Node.js는 이 둘을 오가야 하는 "경계"에서 다음과 같은 문제를 해결해야 합니다.

7-1. Default Export Interop

// CJS로 작성된 라이브러리
module.exports = function lib() { /* ... */ };

// ESM에서 이걸 import하면?
import lib from 'legacy-lib'; // 동작하지만, 번들러/런타임이 "CJS의 module.exports 전체를
                               // ESM의 default export처럼" 취급해주는 interop 계층이 필요하다

ESM에는 "default export"라는 개념이 있지만 CJS에는 없습니다(그냥 module.exports 하나뿐). 그래서 번들러/Node는 CJS 모듈을 ESM에서 import할 때 암묵적으로 module.exports 전체를 default export로 감싸는 규칙을 적용합니다. TypeScript의 esModuleInterop, allowSyntheticDefaultImports 옵션, 그리고 트랜스파일된 코드에서 자주 보이는 __esModule 플래그, .default 접근이 필요해지는 현상이 모두 이 지점에서 나옵니다.

7-2. Dual Package Hazard

같은 패키지를 CJS 버전과 ESM 버전 두 가지로 배포할 때, 한 프로세스 안에서 같은 패키지가 두 번 로드되어 서로 다른 모듈 인스턴스가 되는 문제가 생길 수 있습니다. 예를 들어 상태를 싱글턴으로 관리하는 라이브러리(예: 전역 캐시, 이벤트 emitter)를 어떤 코드는 require로, 어떤 코드는 import로 불러오면, 겉보기엔 같은 패키지인데 실제로는 서로 다른 인스턴스를 참조해 상태가 공유되지 않는 버그가 생깁니다. package.json의 exports 필드로 require/import 조건에 따라 다른 파일을 매핑하는 것이 이 문제를 관리하는 표준 방법입니다.

{
  "name": "my-lib",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}

7-3. 확장자와 "type" 필드

설정Node.js가 해석하는 방식
.mjspackage.json과 무관하게 항상 ESM
.cjspackage.json과 무관하게 항상 CommonJS
.js + "type": "module"ESM으로 해석
.js + "type": "commonjs"(기본값)CommonJS로 해석

8. 실무 연결 — 라이브러리 배포와 순환 참조 리팩터링

시나리오 1 — "번들 크기가 안 줄어든다"

디자인 시스템 컴포넌트 라이브러리에서 아이콘 컴포넌트 하나만 import했는데도 번들에 아이콘 전체 세트가 딸려 들어오는 문제가 있다면, 다음을 순서대로 의심합니다: (1) 그 라이브러리가 CJS로만 배포되고 있는가(패키지의 main만 있고 module/exports.import가 없는가), (2) index 파일이 export * from './icons/all'처럼 배럴(barrel) 형태로 되어 있어 정적으로는 분리 가능해도 번들러 설정(sideEffects 미선언 등)이 이를 막고 있는가.

시나리오 2 — 순환 참조로 인한 undefined 버그

대규모 코드베이스에서 services/user.js와 services/auth.js가 서로를 require하다가, 한쪽에서 다른 쪽의 함수가 undefined로 보이는 버그가 발생하면, Section 5의 원리대로 "어느 파일이 먼저 로드되었고, 순환이 걸린 시점에 상대방이 아직 export하지 못한 부분이 무엇인지"를 실행 순서를 따라 손으로 추적합니다. 근본 해결책은 보통 순환을 유발하는 공통 의존성을 제3의 모듈로 분리하는 것이며, "함수 내부에서 지연 require하기" 같은 임시방편은 문제를 감출 뿐 설계 결함은 남습니다.

시나리오 3 — dual package 라이브러리 배포

사내 유틸 라이브러리를 CJS 백엔드 프로젝트와 ESM 프론트엔드 프로젝트 양쪽에서 쓰게 하려면, 빌드 시 CJS/ESM 두 버전을 모두 산출하고 package.json의 exports 필드로 조건부 매핑을 제공해야 합니다. 이때 라이브러리 내부에 싱글턴 상태(예: 캐시 인스턴스)가 있다면 Section 7-2의 dual package hazard를 반드시 검토해야 합니다.

9. 답변 사고 순서

  1. 어느 모듈 시스템인지부터 확정합니다. require/module.exports인지 import/export인지에 따라 이후 규칙이 전부 달라집니다.
  2. 로딩이 동기인지 정적인지를 근거로 "언제 값이 확정되는가"를 판단합니다. CJS는 실행 순서(동기), ESM은 바인딩 연결(Instantiation) 후 평가(Evaluation) 2단계로 나눠 생각합니다.
  3. 순환 참조가 있다면, "지금 값을 읽는 시점"과 "그 값이 실제로 채워지는 시점"의 선후 관계를 손으로 추적합니다. CJS는 부분 스냅샷, ESM은 TDZ 또는 호이스팅된 함수 여부로 나눠 판단합니다.
  4. 트리쉐이킹/번들 크기 질문이면 "정적으로 분석 가능한가"를 축으로 설명합니다. 동적 require, 배럴 파일, side effect 선언 누락을 원인 후보로 순서대로 짚습니다.
  5. 배포/interop 질문이면 default export 변환 규칙과 dual package hazard, exports 필드를 근거로 답합니다.

10. 핵심 답변 예시

면접관: "Primary Question의 CommonJS 순환 참조 예제에서 b.js가 a.done을 false로 읽는 이유를 설명하고, 같은 문제가 ESM에서는 왜 다르게 나타날 수 있는지 말해주세요."

STRONG ANSWER

Node는 require가 처음 호출된 모듈마다 모듈 캐시에 항목을 만들고 코드를 위에서 아래로 동기적으로 실행합니다. a.js가 b.js를 require하고, b.js가 다시 a.js를 require하는 순간, Node는 a.js가 이미 "로딩 중"이라는 것을 캐시에서 확인하고 처음부터 다시 실행하는 대신 그 시점까지 a.js의 module.exports에 쌓인 값을 즉시 돌려줍니다. a.js는 b.js를 require하기 직전에 exports.done = false만 설정해둔 상태였으므로, b.js가 받는 a는 { done: false }인 부분 스냅샷입니다. 그래서 a.done이 false로 출력됩니다.


ESM에서는 로딩이 Instantiation과 Evaluation 두 단계로 나뉘고, import한 이름은 값의 복사본이 아니라 원본 변수 바인딩에 대한 실시간 참조입니다. 만약 함수처럼 호이스팅되는 값이라면, 순환 참조가 있어도 실제로 그 값을 "사용"하는 시점(대개 모듈 평가가 모두 끝난 뒤 호출되는 시점)에는 이미 바인딩이 채워져 있어 정상적으로 동작합니다. 반면 let/const로 선언된 값을 초기화 이전에 순환적으로 최상위에서 읽으려 하면, CJS처럼 조용히 undefined를 주는 대신 ReferenceError가 발생합니다 — 즉 CJS는 "잘못된 값을 조용히 주는" 방식이고, ESM은 "아직 준비 안 됐다"를 명시적인 에러로 알려주는 방식이라는 차이가 있습니다.

이 답변이 좋은 이유:

  • "순환 참조라서 문제"가 아니라 모듈 캐시와 실행 순서라는 구체적 메커니즘으로 원인을 설명함
  • 부분 스냅샷이라는 정확한 용어와, 그것이 왜 false로 나타나는지 실행 순서를 근거로 짚음
  • ESM과의 차이를 "무엇이 다른가"가 아니라 "왜 다른가"(2단계 로딩, live binding, TDZ)로 설명함
  • 두 시스템의 실패 방식을 "조용한 실패 vs 명시적 에러"로 대비시켜 실무 디버깅 함의까지 짚음

11. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"순환 참조라서 b.js에서 a.js가 아직 다 안 끝나서 그런 것 같아요. ESM은 최신 문법이라 이런 문제가 없어요."

이 답변이 약한 이유

  • "아직 다 안 끝나서"까지는 맞지만, 왜 Node가 다시 처음부터 실행하지 않고 부분 스냅샷을 돌려주는지(모듈 캐시 메커니즘)에 대한 근거가 없습니다.
  • "ESM은 이런 문제가 없다"는 틀린 일반화입니다. ESM도 순환 참조에서 TDZ로 인한 ReferenceError가 날 수 있습니다 — 문제가 "없는" 것이 아니라 "다르게, 더 명시적으로" 드러날 뿐입니다.
WEAK ANSWER — 또 다른 패턴

"트리쉐이킹은 웹팩이 알아서 안 쓰는 코드를 지워주는 최적화 기능이에요."

이 답변이 약한 이유

  • 결과(기능)만 설명하고 전제조건(정적 분석이 가능해야 한다는 것, 그래서 CJS에서는 잘 안 된다는 것)을 설명하지 않았습니다.
  • "웹팩이 알아서"라는 표현은 마치 항상 동작하는 것처럼 들리지만, 실무에서는 배럴 파일, 동적 require, side effect 미선언 때문에 트리쉐이킹이 실패하는 경우가 훨씬 흔하며 이런 원인 진단 능력이 시니어에게 요구됩니다.
STRONG ANSWER 요소 체크리스트
  • CJS는 동기/동적, ESM은 정적/2단계 로딩이라는 실행 모델 차이를 먼저 세운다
  • 순환 참조 결과를 실행 순서를 손으로 추적해 설명한다(부분 스냅샷 vs live binding/TDZ)
  • 트리쉐이킹의 전제조건(정적 분석 가능성, side effect 선언)까지 짚는다
  • 실무 실패 사례(번들 크기, dual package hazard, interop)로 연결한다
  • "어느 쪽이 무조건 낫다"가 아니라 각 시스템이 실패하는 방식의 차이(조용한 실패 vs 명시적 에러)까지 비교한다

12. Senior-level Follow-up Q&A

Q: import()(동적 import)는 정적 분석 원칙에 어긋나지 않나요?

모범 답변: import()는 함수처럼 호출되고 Promise를 반환하며 조건문 안에서도 쓸 수 있어 require와 비슷해 보이지만, 목적이 다릅니다. 정적 import 선언이 "빌드 타임에 반드시 알아야 하는 고정 의존성"을 위한 것이라면, import()는 런타임에 필요할 때만 로드하는 코드 스플리팅을 위한 것입니다. 번들러는 import()의 인자가 정적 문자열이면(변수가 아니면) 그 시점에서 별도 청크(chunk)로 분리해 지연 로딩하도록 처리할 수 있습니다. 즉 트리쉐이킹의 정적 분석 원칙과 충돌하는 것이 아니라, "지금 당장 필요 없는 코드를 나중으로 미루는" 별개의 최적화 축입니다.

Q: TypeScript의 esModuleInterop은 정확히 무엇을 바꾸나요?

모범 답변: 이 옵션이 꺼져 있으면 import * as React from 'react'처럼 namespace import 문법을 강제로 써야 CJS 모듈을 가져올 수 있습니다. 켜면 import React from 'react'처럼 default import 문법을 쓸 수 있도록, 컴파일된 코드에 module.exports 전체를 { default: module.exports, ...나머지 프로퍼티 } 형태로 감싸는 헬퍼(__importDefault 등)를 자동으로 삽입합니다. 즉 Section 7-1에서 다룬 "CJS에는 default export 개념이 없다"는 근본 차이를 컴파일 타임에 메워주는 옵션입니다. 런타임 동작 자체를 바꾸는 것이 아니라, 컴파일된 JS에 interop 코드를 추가하는 것이라는 점이 핵심입니다.

Q: 순환 참조 자체를 아예 안 만드는 것이 항상 가능한가요, 아니면 어느 정도는 감수해야 하나요?

모범 답변: 완전히 제거하는 것이 이상적이지만, 큰 코드베이스에서 관련 도메인 모델끼리(예: User가 Order를 참조하고 Order가 User를 참조) 순환이 자연스럽게 생기는 경우가 실무에서는 드물지 않습니다. 대응 전략은 우선순위대로: (1) 공통으로 필요한 타입/유틸을 제3의 모듈로 추출해 순환 자체를 끊는다, (2) 어쩔 수 없다면 순환이 있는 두 모듈 중 어느 쪽이 "완성된 값"을 필요로 하는 시점이 늦는지 설계해서 부분 스냅샷 문제를 피한다, (3) 정말 급하면 함수 내부에서 지연 require/동적 import()로 실제 호출 시점까지 로딩을 미룬다(단, 이건 구조적 결함을 가리는 임시방편임을 인지해야 한다). 시니어는 순환을 "없앨 수 있는데 안 없앤 것"과 "도메인 특성상 불가피한 것"을 구분해서 설명할 수 있어야 합니다.

Q: "sideEffects": false를 잘못 선언하면 어떤 문제가 생기나요?

모범 답변: 실제로는 부수효과가 있는 모듈(예: import 'polyfill.js'처럼 전역에 무언가를 등록하는 모듈, 또는 CSS를 import하면서 스타일을 주입하는 모듈)을 sideEffects: false로 선언하면, 번들러는 "이 모듈이 export를 안 쓰이면 통째로 제거해도 된다"고 오판해 필요한 부수효과 코드까지 트리쉐이킹으로 삭제해버릴 수 있습니다. 결과적으로 프로덕션 빌드에서만 polyfill이 빠지거나 스타일이 깨지는, 로컬 개발 환경(번들링을 다르게 하거나 트리쉐이킹을 안 하는 dev 모드)에서는 재현되지 않는 버그가 생깁니다. 그래서 부수효과가 있는 파일은 "sideEffects": ["./src/polyfill.js", "*.css"]처럼 예외 목록으로 명시해야 합니다.

13. 흔한 오해

오해 1

"ESM은 CommonJS의 최신 버전이라 기능상 완전한 상위호환이다"

문법적으로 더 표준화되어 있지만 "상위호환"은 아닙니다. CJS의 require는 동적/조건부 로딩이 자유롭다는 장점이 있고(레거시 Node 스크립트, 조건부 플러그인 로딩 등에서 여전히 실용적), ESM은 정적 분석·트리쉐이킹·live binding이라는 다른 장점을 갖는 서로 다른 설계 목표를 가진 시스템입니다. 실무에서는 여전히 두 시스템의 상호운용(Section 7)을 계속 다뤄야 합니다.

오해 2

"트리쉐이킹은 웹팩/번들러의 자체 기능이라 아무 모듈 시스템에서나 동작한다"

트리쉐이킹은 번들러의 기능이 맞지만, ESM의 정적 구조라는 전제조건이 있어야 안전하게 동작합니다. Section 6에서 다뤘듯 CJS의 동적 require/module.exports 변형 패턴에서는 번들러가 안전을 보장할 수 없어 트리쉐이킹을 포기하는 경우가 많습니다.

오해 3

"순환 참조가 있으면 무조건 런타임 에러가 나거나 프로그램이 죽는다"

Section 5에서 봤듯 CJS는 에러 없이 조용히 부분 스냅샷을 돌려주고, ESM도 함수 호이스팅 케이스처럼 정상 동작하는 경우가 많습니다. 순환 참조 자체가 항상 치명적인 것이 아니라, "아직 준비되지 않은 값을 너무 이른 시점에 사용하려 할 때"만 문제가 됩니다. 다만 코드 구조를 이해하기 어렵게 만들고 잠재적 버그의 원인이 되므로 가능하면 피하는 것이 좋은 설계입니다.

오해 4

"package.json에 "type": "module"만 추가하면 프로젝트 전체가 자동으로 ESM 이점(트리쉐이킹 등)을 얻는다"

"type": "module"은 Node.js가 .js 파일을 어떤 문법으로 해석할지 결정하는 런타임 설정일 뿐입니다. 트리쉐이킹은 번들러가 빌드 시점에 수행하는 최적화이며, 소스 코드가 실제로 정적 import/export를 사용하고, 배럴 파일·동적 require 같은 방해 패턴이 없고, sideEffects가 올바르게 선언되어 있어야 실질적인 효과를 봅니다. 설정 하나로 자동 해결되는 문제가 아닙니다.
