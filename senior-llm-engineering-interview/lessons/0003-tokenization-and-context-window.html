Day 3 — Tokenization과 Context Window

"큰 context window"가 왜 "다 넣어도 된다"는 뜻이 아닌지 수치로 설명하는 사고법 · 2026-07-05

최신성 안내 — 이 레슨은 2026년 7월 기준 Anthropic Claude Platform Docs(context windows, token counting), OpenAI 공식 changelog, Chroma의 context rot 연구를 확인해 작성되었습니다. 2026년 중반 기준 프론티어 모델(Claude Opus/Sonnet 최신 세대, GPT-5.5)은 API 기본값으로 최대 1M 토큰 context window를 제공하며, Claude는 Opus 4.7 이후·Sonnet 5 등에서 새 토크나이저를 적용해 동일한 텍스트가 이전 대비 약 30% 더 많은 토큰으로 계산됩니다. 구체적 토큰 수·모델명·가격은 계속 바뀌므로, 이 레슨에서 암기할 것은 숫자 자체가 아니라 "context window 크기와 실제로 안정적으로 활용 가능한 크기(effective context)는 다르다"는 판단 프레임워크입니다.

1. 핵심 면접 질문

Primary Question

"context window가 100만 토큰인 모델이 있습니다. 우리 사내 문서를 전부 넣으면 되는 것 아닌가요? 굳이 RAG나 청킹이 필요한 이유가 있나요?"

Variant 1

"토큰(token)이 정확히 무엇인가요? 왜 한국어가 영어보다 토큰 비용이 더 많이 든다는 이야기가 있나요?"

Variant 2

"긴 문서를 컨텍스트에 넣었더니, 앞부분과 뒷부분 내용은 잘 답하는데 중간에 있는 내용은 자꾸 놓칩니다. 왜 이런 현상이 생기나요?"

Variant 3

"멀티턴 대화 에이전트가 오래 실행되다 보니 컨텍스트가 계속 쌓여서 언젠가 한도를 넘습니다. 어떻게 설계하시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 "context window는 크면 클수록 무조건 좋다"는 마케팅 숫자를 그대로 믿는지, 아니면 그 숫자가 실제 프로덕션 품질과 어떤 간극이 있는지 아는지를 테스트합니다. 벤더의 발표 자료에는 "100만 토큰까지 지원"이라는 문구가 크게 적혀 있지만, 실제로 그 100만 토큰을 다 채웠을 때 모델이 그 안의 모든 정보를 똑같이 신뢰성 있게 활용한다는 뜻은 아닙니다. 시니어와 주니어의 차이는, 이 숫자를 스펙 그대로 받아들여 설계하는지, 아니면 "이 태스크에서 실제로 안정적으로 쓸 수 있는 컨텍스트 크기는 얼마인가"를 직접 검증하고 그에 맞춰 아키텍처(RAG, 청킹, 압축, 정보 배치)를 선택하는지입니다.

면접관이 이 질문으로 평가하는 것:

  • 토큰 단위에 대한 정확한 이해 — 토큰이 글자도 단어도 아닌 서브워드 단위라는 것, 그리고 이것이 비용·언어별 형평성에 미치는 영향을 아는가.
  • 스펙과 실제 성능의 간극을 아는가 — "claimed context window"(광고된 크기)와 "effective context window"(실제로 정확도가 유지되는 크기)가 다르다는 것을 수치·연구 근거로 설명할 수 있는가.
  • 정보 배치 전략 — 컨텍스트 내 정보의 "위치"가 모델의 활용도에 영향을 준다는 것(포지션 편향)을 이해하고, 중요한 정보를 어디에 배치할지 설계할 수 있는가.
  • RAG vs 긴 컨텍스트 stuffing 판단 — "컨텍스트가 크니까 다 넣자"가 아니라, 관련성 있는 정보만 선별해 넣는 것이 왜 더 나은 선택인지 비용·정확도 양쪽으로 논증할 수 있는가.
  • 장기 실행 에이전트 설계 감각 — 대화·에이전트 루프가 길어질 때 컨텍스트가 계속 누적된다는 문제를 압축(compaction), 컨텍스트 편집, 오래된 tool 결과 정리 같은 실무 기법으로 대응할 수 있는가.

주니어~중니어에서 자주 나오는 답변 패턴: "context window가 100만 토큰이니까 문서를 다 넣으면 됩니다." 이 답변은 "들어간다"와 "잘 활용된다"를 혼동한 것입니다. 시니어는 컨텍스트에 넣을 수 있는 최대치와, 그 안에서 모델이 실제로 정확하게 추론할 수 있는 범위가 다르다는 것을 알고, 태스크별로 이를 직접 측정한 뒤 설계에 반영합니다.

3. 전제 개념 — 쉬운 한국어로

토큰(token)이란 정확히 무엇인가

토큰은 "글자"도 "단어"도 아닙니다. LLM은 텍스트를 서브워드(subword) 단위로 잘라서 처리하는데, 이 자르는 방식을 토큰화(tokenization)라고 하고, 대표적인 방식이 BPE(Byte Pair Encoding) 계열 알고리즘입니다. 예를 들어 영어 단어 "unbelievable"은 통째로 하나의 토큰이 아니라 un + believ + able 같은 조각으로 나뉠 수 있습니다. 자주 등장하는 글자 조합(영어의 흔한 접두사·접미사 등)은 하나의 토큰으로 묶이고, 드물게 등장하는 조합은 더 잘게 쪼개집니다. 이 학습 데이터가 영어 중심으로 만들어졌기 때문에, 한국어 같은 언어는 상대적으로 같은 의미를 표현하는 데 더 많은 토큰이 필요한 경우가 흔합니다. 이것이 "한국어가 영어보다 토큰 비용이 더 든다"는 이야기의 실제 원인입니다. 인터뷰에서는 "토큰 = 대략 단어의 일부"라는 감(글자 수가 아니라)과, 이것이 비용·컨텍스트 소모량에 직접 영향을 준다는 연결을 함께 말할 수 있어야 합니다.

Context window란 무엇인가 — "작업 기억"이라는 비유

Context window는 모델이 응답을 생성하는 그 순간에 "동시에 참고할 수 있는" 텍스트의 최대 범위입니다. 모델이 학습 때 본 방대한 지식(하드디스크에 저장된 지식)과는 다른 개념으로, "지금 책상 위에 펼쳐 놓고 볼 수 있는 자료의 양"에 비유하면 이해가 쉽습니다. 중요한 것은, 시스템 프롬프트, 대화 기록, 첨부 문서, 이미지, tool 정의, tool 실행 결과, (extended thinking을 쓰는 경우) 모델의 사고 과정까지 요청에 들어가는 거의 모든 것이 이 window를 함께 소비한다는 점입니다. 그래서 "대화가 길어질수록", "도구를 많이 등록할수록", "문서를 많이 첨부할수록" 컨텍스트는 예상보다 훨씬 빨리 찹니다.

왜 "context window가 크다" ≠ "다 넣어도 성능이 유지된다"인가

이것이 이번 레슨의 핵심 개념입니다. Context window 크기는 "물리적으로 넣을 수 있는 최대 용량"을 말할 뿐, "그 용량을 다 채워도 모델이 모든 부분을 동일한 정확도로 활용한다"는 보장이 아닙니다. 실제로는 컨텍스트에 들어가는 토큰 수가 늘어날수록 관련 정보를 찾아내고 정확히 추론하는 능력이 점점 떨어지는 현상이 관찰되는데, 이를 최근 업계·연구에서는 "context rot"(컨텍스트 부패)이라고 부릅니다. Anthropic의 공식 문서도 "더 많은 컨텍스트가 자동으로 더 좋은 것은 아니며, 토큰 수가 늘어날수록 정확도와 회상(recall)이 저하되는 현상을 context rot이라 부른다"고 명시하고 있습니다. 도서관에 비유하면, 책 한 권만 있을 때는 원하는 문장을 금방 찾지만, 책이 백만 권 쌓인 창고에서는 (설령 그 창고에 다 들어갈 "공간"이 있다 해도) 원하는 문장을 정확히 찾아내는 것이 훨씬 어려워지는 것과 비슷합니다.

"포지션 편향(position bias)" 또는 "lost in the middle"이란 무엇인가

여러 문서를 컨텍스트에 넣고 질문했을 때, 정답 근거가 컨텍스트의 맨 앞이나 맨 뒤에 있을 때는 잘 찾아내지만, 중간에 있을 때는 정확도가 크게 떨어지는 경향이 여러 모델에서 반복적으로 관찰됩니다. 이를 그래프로 그리면 U자 모양이 나온다고 해서 U-shaped position bias라고도 부릅니다. 사람이 긴 회의록을 읽을 때 도입부와 결론은 잘 기억하지만 중간에 나온 세부 사항은 놓치기 쉬운 것과 비슷한 현상이라고 이해하면 됩니다. 이 특성을 알면, 컨텍스트를 구성할 때 "가장 중요한 정보를 앞이나 뒤에 배치"하거나 "애초에 관련 없는 정보는 넣지 않는" 설계 판단으로 이어집니다.

"Claimed context window"와 "Effective context window"의 차이

Claimed context window는 벤더가 공식적으로 발표한 최대 크기(예: "1M 토큰 지원")이고, effective context window는 실제로 정확도가 크게 저하되지 않고 안정적으로 활용 가능한 크기입니다. 2026년 기준 공개 연구(Chroma의 다중 모델 벤치마크 등)에 따르면, 여러 프론티어 모델에서 effective context window가 claimed 값보다 30~40% 이상 이른 지점부터 정확도가 무너지기 시작하는 경우가 흔하고, 복잡한 태스크에서는 그 간극이 훨씬 클 수 있다고 보고됩니다. 즉 "200K 토큰까지 지원"이라는 모델이라도, 실무 태스크 기준으로는 그보다 훨씬 이전 지점부터 신뢰도가 떨어질 수 있다는 뜻입니다. 시니어 엔지니어는 이 간극을 벤더 스펙이 아니라 우리 태스크에 대한 자체 eval로 직접 측정해야 한다고 봅니다 (Day 1의 task-fit eval 원칙과 정확히 연결됩니다).

4. 최신 정보 스냅샷 — 2026년 7월 기준 context window 지형

주요 벤더 context window 크기 (2026-07 기준, 공식 문서 확인)

Anthropic Claude Platform Docs에 따르면, Claude Opus 최신 세대(4.6/4.7/4.8)와 Claude Sonnet 5·Sonnet 4.6은 Claude API·Amazon Bedrock·Google Cloud·Microsoft Foundry에서 1M 토큰 context window를 기본값으로 제공하며(별도 beta 헤더 불필요), 장문 컨텍스트도 표준 요금으로 청구됩니다. 다만 Claude Sonnet 4.5 등 일부 모델은 여전히 200K 토큰이 상한입니다. OpenAI 측도 GPT-5.5에서 1M 토큰 컨텍스트를 지원한다고 공개했습니다. 중요한 것은 "1M"이라는 숫자를 암기하는 것이 아니라, 인터뷰 시점에 각 모델의 공식 스펙을 다시 확인해야 한다는 습관입니다 — 이 숫자는 이 레슨이 쓰인 시점 이후에도 계속 바뀝니다.

토크나이저 세대 교체와 마이그레이션 함정

Anthropic 문서는 Claude Opus 4.7 이후 세대와 Sonnet 5 등이 새로운 토크나이저를 사용하며, 동일한 입력 텍스트가 이전 세대 대비 약 30% 더 많은 토큰으로 계산된다고 명시합니다. 이는 실무에서 매우 중요한 시사점을 줍니다 — 모델 버전을 올렸을 때 "품질은 그대로인데 비용이 갑자기 늘었다"거나 "이전에는 문제없던 프롬프트가 갑자기 context window를 초과한다"는 장애가 실제로 발생할 수 있습니다. 시니어는 모델 마이그레이션 시 토크나이저 변경 여부와 토큰 수 변화를 반드시 사전에 확인하고, 토큰 카운팅 API로 실측한 뒤 배포해야 한다고 판단합니다.

Context rot 연구 — "context window가 클수록 좋다"는 가정에 대한 반증

Chroma가 GPT, Claude, Gemini, Qwen 등 다수의 프론티어 모델을 대상으로 진행한 벤치마크는, 테스트한 모든 모델이 입력 길이가 늘어날수록 정확도가 저하되며, 어떤 경우에는 공식 한도에 도달하기 훨씬 전(예: 200K 한도 모델이 5만 토큰 근처)부터 30~50%의 정확도 저하가 나타난다고 보고합니다. 관련 위치 편향 연구들도 정답 근거가 컨텍스트 중간에 있을 때 정확도가 U자형으로 떨어지는 현상을 반복 관찰했습니다. 이 연구는 "context window가 크면 무조건 좋다"는 인터뷰 답변이 왜 얕은 답변인지에 대한 구체적 근거로 인용할 수 있습니다.

장기 실행 대화·에이전트를 위한 공식 대응 기능들

2026년 기준 Anthropic은 서버 사이드 compaction(대화 앞부분을 서버에서 자동 요약해 한도를 넘어서도 대화를 이어가는 기능), context editing(오래된 tool 실행 결과나 thinking 블록을 정리), context awareness(모델이 시스템 프롬프트에 주입된 토큰 예산을 스스로 추적해 남은 여유를 인식) 기능을 공식 제공합니다. OpenAI 쪽도 API가 대화 기록을 자동 요약·압축해 토큰 한도 내로 유지하는 기능을 발표했습니다. 즉 "컨텍스트가 쌓이면 잘라버린다"는 수동적 대응을 넘어, 벤더 차원에서 압축·편집을 표준 기능으로 제공하는 흐름이 자리잡고 있습니다. 인터뷰에서 "장기 실행 에이전트의 컨텍스트를 어떻게 관리하�나요?"라는 질문에는, 직접 처음부터 구현하는 방법뿐 아니라 이런 플랫폼 기능을 먼저 검토한다고 답하는 것이 최신 실무 감각을 보여줍니다.

이 스냅샷에서 인터뷰용으로 가져가야 할 한 문장: "context window는 계속 커지고 있지만(2026년 기준 프론티어 모델은 1M 토큰 수준), 실제 활용 가능한 범위(effective context)는 광고된 크기보다 훨씬 작을 수 있다는 것이 여러 연구로 반복 확인되었고, 그래서 저는 '얼마나 큰가'가 아니라 '우리 태스크에서 얼마나 안정적으로 쓸 수 있는가'를 직접 측정해 설계에 반영합니다."

5. Context Rot 프레임워크 — "claimed" vs "effective" context window

인터뷰에서 이 개념을 설명할 때는 아래 3개 축으로 구조화해서 말하면 신호가 명확해집니다.

축 정의 실무 시사점
① Claimed context window 벤더가 공식 발표한 최대 토큰 수 (예: "1M 토큰 지원") 계약·가격·요청 제한의 기준일 뿐, 품질 보장의 기준이 아님
② Effective context window 우리 태스크 기준으로 정확도가 허용 가능한 수준을 유지하는 실제 토큰 범위 자체 eval로만 측정 가능. 태스크·모델·프롬프트 구조에 따라 달라짐
③ 포지션 편향 (Position bias) 같은 토큰 수 안에서도 정보의 "위치"에 따라 회상률이 달라지는 현상 중요한 지시·근거는 컨텍스트 앞 또는 뒤에 배치, 관련 없는 정보는 애초에 제외

왜 이 구분이 실무에서 중요한가

"우리 RAG 시스템이 답을 못 찾는다"는 보고를 받았을 때, 원인이 ①claimed window를 초과해 문서가 아예 잘렸는지, ②effective window를 넘어서 정확도가 이미 저하된 구간인지, ③문서 순서상 정답 근거가 중간에 위치해 포지션 편향으로 놓쳤는지에 따라 대응이 완전히 다릅니다. ①이라면 청킹·요약이 필요하고, ②라면 애초에 넣는 문서 양을 줄이거나 RAG로 관련 문서만 선별해야 하며, ③이라면 검색 결과 재정렬(가장 관련성 높은 문서를 맨 앞/맨 뒤로) 같은 배치 전략이 필요합니다. 세 가지를 구분하지 않고 "컨텍스트를 늘리면 되겠지"라고만 대응하면 오히려 ②의 문제를 악화시킬 수 있습니다.

6. 상황별 전략 매핑

  • 1
    "관련 문서가 매우 많고 계속 늘어난다" → RAG(검색 기반 주입)

    전체 지식 베이스를 매번 컨텍스트에 넣는 대신, 질문과 관련성이 높은 일부만 검색해 주입합니다. 이는 토큰 비용을 줄일 뿐 아니라, 관련 없는 정보를 컨텍스트에서 배제해 effective context 문제와 포지션 편향 문제를 동시에 줄여줍니다. (세부 청킹·리랭킹 전략은 senior-rag-interview에서 깊게 다룸)

  • 2
    "문서 하나(계약서, 대용량 코드베이스 등)를 통째로 이해해야 한다" → Long-context stuffing, 단 검증 필수

    문서 하나의 흐름 전체를 이해해야 하는 태스크는 단편적 청크로 쪼개면 문맥이 끊깁니다. 이런 경우는 long-context에 통째로 넣는 것이 타당하지만, 반드시 그 문서 크기에서 정확도가 유지되는지 우리 태스크 eval로 검증하고, 핵심 지시·질문은 컨텍스트 앞/뒤에 배치합니다.

  • 3
    "멀티턴 대화·에이전트가 오래 실행되어 컨텍스트가 계속 쌓인다" → Compaction / Context editing

    오래된 대화 턴이나 이미 소비된 tool 실행 결과를 서버 사이드 압축(요약)하거나 정리해 컨텍스트를 재확보합니다. 처음부터 직접 구현하기보다, 플랫폼이 제공하는 compaction·context editing 기능을 먼저 검토합니다.

  • 4
    "같은 대용량 시스템 프롬프트/문서를 반복 호출한다" → Prompt caching과 결합

    긴 컨텍스트를 반복적으로 재사용하는 워크로드는 비용·지연시간 문제가 커지므로, 캐시 가능한 접두어(시스템 프롬프트, 고정 문서)를 앞쪽에 고정해 캐시 적중률을 높입니다. (Day 15 latency/cost 최적화에서 심화)

  • 5
    "토큰 비용·성능을 사전에 검증해야 한다" → 토큰 카운팅 API + 자체 eval

    배포 전 토큰 카운팅 API로 실제 요청의 토큰 수를 추정하고, 다양한 컨텍스트 길이 구간에서 우리 태스크의 정확도를 측정하는 자체 eval(needle-in-a-haystack류 테스트 포함)을 구축해 effective context window를 직접 파악합니다.

7. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 됩니다.

  1. "토큰"과 "context window"의 정의를 정확히 짚습니다. 토큰은 서브워드 단위이고, context window는 시스템 프롬프트·대화·문서·tool 정의·tool 결과·thinking까지 포함하는 "요청 전체가 공유하는 용량"임을 언급합니다.
  2. "큰 context window = 다 넣어도 안전"이라는 가정을 반박합니다. claimed context window와 effective context window가 다르다는 것을, context rot 연구를 근거로 설명합니다.
  3. 포지션 편향(U-shaped)을 언급합니다. 같은 토큰 수 안에서도 정보 위치에 따라 회상률이 달라진다는 것을 설명합니다.
  4. 우리 상황(RAG 필요 여부, long-context stuffing 여부, 장기 실행 에이전트 여부)에 맞는 전략을 매핑합니다. "무조건 RAG"도 "무조건 long-context"도 아니라 태스크 특성에 따라 선택한다고 말합니다.
  5. 이 판단을 벤더 스펙이 아니라 자체 eval로 검증하겠다고 마무리합니다. 토큰 카운팅 API와 태스크별 정확도 측정을 함께 언급합니다.

8. 핵심 답변 예시

면접관: "context window가 100만 토큰인 모델이 있습니다. 우리 사내 문서를 전부 넣으면 되는 것 아닌가요? 굳이 RAG나 청킹이 필요한 이유가 있나요?"

STRONG ANSWER

먼저 전제를 짚겠습니다. context window 크기는 "물리적으로 넣을 수 있는 최대치"일 뿐, "그 안의 모든 내용을 동일한 정확도로 활용한다"는 보장이 아닙니다. 2026년 기준 여러 프론티어 모델을 대상으로 한 연구(예: Chroma의 context rot 벤치마크)에서, 입력 길이가 늘어날수록 정확도가 저하되며 어떤 모델은 공식 한도의 절반도 안 되는 지점부터 이미 성능이 크게 떨어진다는 결과가 보고되었습니다. 게다가 정답 근거가 컨텍스트 중간에 위치할 때 회상률이 크게 떨어지는 포지션 편향도 여러 모델에서 반복 관찰됩니다.


그래서 "1M 토큰이 들어가니 문서를 전부 넣자"는 접근은 두 가지 리스크가 있습니다. 첫째, 실제 정확도가 광고된 스펙만큼 나오지 않을 수 있다는 것(effective context window가 claimed보다 훨씬 작을 수 있음). 둘째, 관련 없는 문서까지 다 넣으면 오히려 진짜 관련 있는 정보를 찾아내는 것을 방해할 수 있다는 것입니다.


따라서 저는 문서 전체를 한 번에 넣기보다, 질문과 관련성이 높은 부분만 검색해 주입하는 RAG 구조를 기본으로 채택합니다. 다만 예외적으로 계약서 한 건처럼 문서 하나의 전체 흐름을 이해해야 하는 태스크는 long-context에 통째로 넣는 것이 더 적합할 수 있는데, 이 경우에도 반드시 그 문서 길이에서 우리 태스크 기준 정확도가 유지되는지 자체 eval로 검증한 뒤 배포합니다. 그리고 핵심 질문이나 지시는 컨텍스트의 맨 앞이나 맨 뒤에 배치해 포지션 편향의 영향을 줄입니다.

이 답변이 좋은 이유:

  • "들어간다"와 "정확하게 활용된다"를 명확히 구분해 설명함
  • claimed context window와 effective context window 개념을 연구 근거와 함께 제시함
  • "무조건 RAG"가 아니라 상황(전체 문서 이해 필요 여부)에 따라 long-context stuffing도 선택지로 열어둠
  • 포지션 편향까지 고려한 구체적 배치 전략을 제시함
  • 벤더 스펙을 그대로 믿지 않고 자체 eval로 검증하겠다는 태도로 마무리함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"네, context window가 100만 토큰이면 충분히 크니까 문서를 전부 넣으면 됩니다. RAG는 컨텍스트가 작았을 때나 필요했던 기술이죠."

이 답변이 약한 이유

  • claimed context window와 effective context window를 구분하지 못했습니다.
  • context rot·포지션 편향 같은, 컨텍스트가 커질수록 오히려 나타나는 문제를 전혀 언급하지 않았습니다.
  • RAG를 "컨텍스트가 작을 때의 임시방편"으로 오해하고 있습니다 — 실제로는 관련성 필터링과 비용 효율성 측면에서 여전히 유효한 아키텍처 선택입니다.
WEAK ANSWER — 또 다른 패턴

"토큰이요? 그냥 단어 하나라고 보면 됩니다."

이 답변이 약한 이유

  • 토큰은 단어 단위가 아니라 서브워드 단위이며, 이 차이가 언어별 비용 형평성(한국어가 더 비쌀 수 있음), 정확한 비용 추정 등 실무 판단에 직접 영향을 준다는 것을 놓쳤습니다.
  • 단순화된 정의만 말하고, 이것이 왜 중요한지 실무 연결을 전혀 하지 못했습니다.
STRONG ANSWER 요소 체크리스트
  • 토큰을 서브워드 단위로 정확히 정의하고 언어별 비용 차이까지 연결한다
  • claimed context window와 effective context window의 차이를 구체적 연구 근거로 설명한다
  • 포지션 편향(U-shaped) 현상과 그 대응(정보 배치)을 언급한다
  • RAG와 long-context stuffing을 경쟁 관계가 아니라 상황별 도구로 설명한다
  • 최종 판단은 벤더 스펙이 아니라 자체 eval로 검증하겠다고 마무리한다

10. Senior-level Follow-up Q&A

Q: 토큰이 정확히 무엇인가요? 왜 한국어가 영어보다 토큰 비용이 더 많이 든다는 이야기가 있나요?

모범 답변: 토큰은 BPE류 알고리즘으로 텍스트를 잘게 쪼갠 서브워드 단위입니다. 학습 데이터의 통계적 빈도를 기반으로 자주 나오는 글자 조합을 하나의 토큰으로 묶기 때문에, 학습 데이터에 영어 비중이 높았던 토크나이저는 영어 단어·접사 조합은 적은 토큰으로 표현하지만, 한국어 같은 언어는 상대적으로 더 잘게 쪼개져 같은 의미를 표현하는 데 더 많은 토큰이 필요한 경우가 흔합니다. 실무적으로는 다국어 서비스의 토큰 비용·context 소모량을 산정할 때 언어별로 실측해야 하며, "영어 기준으로 계산한 토큰 예산"을 그대로 다른 언어에 적용하면 과소 추정될 수 있다는 점을 유의해야 합니다.

Q: 긴 문서를 컨텍스트에 넣었더니 중간 내용을 자꾸 놓칩니다. 왜 이런 현상이 생기나요?

모범 답변: 여러 모델에서 반복 관찰되는 포지션 편향(위치 편향) 현상으로, 정답 근거가 컨텍스트의 맨 앞이나 맨 뒤에 있을 때는 잘 찾아내지만 중간에 있을 때 회상률이 크게 떨어지는 U자형 경향입니다. 대응 방법으로는 (1) 가장 중요한 정보나 질문 자체를 컨텍스트의 앞 또는 뒤에 재배치하고, (2) 검색 기반 구조라면 관련성 높은 문서를 재정렬(reranking)해 상대적으로 유리한 위치에 놓으며, (3) 애초에 관련 없는 정보를 컨텍스트에서 제외해 "중간"이라는 위치 자체가 덜 위험하도록 컨텍스트 총량을 줄이는 방법이 있습니다. 다만 이 현상이 모든 태스크·모델에 동일한 강도로 나타나는 것은 아니므로, 우리 태스크로 직접 재현·측정한 뒤 대응 수준을 정하는 것이 중요합니다.

Q: 멀티턴 대화 에이전트가 오래 실행되다 보니 컨텍스트가 계속 쌓여서 언젠가 한도를 넘습니다. 어떻게 설계하시겠어요?

모범 답변: 우선 무엇이 컨텍스트를 실제로 소모하는지 분해합니다 — 대화 이력 자체인지, 반복적으로 쌓이는 tool 실행 결과인지, 아니면 매 턴 다시 포함되는 대용량 시스템 프롬프트나 tool 정의인지 구분합니다. 원인에 따라 (1) 오래된 대화 턴은 서버 사이드 compaction으로 자동 요약하고, (2) 이미 소비된 tool 결과나 thinking 블록은 context editing으로 정리하며, (3) 반복되는 고정 프리픽스(시스템 프롬프트 등)는 prompt caching과 결합해 비용을 줄입니다. 직접 잘라내는 로직을 처음부터 구현하기보다, 플랫폼이 제공하는 compaction·context editing 기능을 먼저 검토하고, 그 기능만으로 부족한 부분(우리 도메인 특화 요약 기준 등)만 커스텀 로직으로 보완합니다.

Q (비기술적): "저장 공간이 1M 토큰이나 되는데 왜 자꾸 검색(RAG)이 필요하다고 하나요?"라고 비개발 조직이 묻는다면 어떻게 설명하시겠어요?

모범 답변: 비유로 설명합니다. "1M 토큰 컨텍스트는 책상 위에 한 번에 펼쳐 놓을 수 있는 서류의 최대 양이라고 보시면 됩니다. 그런데 서류를 책상 위에 아무리 많이 쌓아 놔도, 그 안에서 원하는 한 문장을 정확히 찾아내는 능력은 서류가 많아질수록 오히려 떨어집니다. 그래서 저희는 모든 서류를 매번 책상에 다 펼치기보다, 질문과 관련된 몇 장만 먼저 골라서 책상에 올려놓는 방식(검색, RAG)을 씁니다. 이렇게 하면 더 빠르고, 더 저렴하고, 실제로 더 정확한 답을 받을 수 있습니다." 이후 구체적 사례(관련 없는 문서가 섞였을 때 오히려 답이 부정확해졌던 경험 등)로 이어 설명하면 설득력이 높아집니다.

Q: 모델을 신버전으로 업그레이드했더니 갑자기 토큰 비용이 늘고, 예전엔 문제없던 요청이 context window 초과 오류를 냅니다. 무엇을 먼저 확인하시겠어요?

모범 답변: 가장 먼저 토크나이저 세대가 바뀌었는지부터 확인합니다. 실제로 최근 모델 세대 교체에서 새 토크나이저가 도입되어 동일한 텍스트가 이전 대비 훨씬 많은(예: 약 30% 더 많은) 토큰으로 계산되는 사례가 있었습니다. 이를 놓치면 "품질은 같은데 비용이 늘었다"거나 "이전에 문제없던 프롬프트가 갑자기 한도를 초과한다"는 증상으로 나타납니다. 대응으로는 (1) 토큰 카운팅 API로 마이그레이션 전후 토큰 수를 실측 비교하고, (2) 고정 프롬프트·문서 길이를 새 토크나이저 기준으로 재산정하며, (3) 비용·한도에 여유 마진을 두고 배포 전 회귀 테스트에 토큰 수 검증을 포함시킵니다. 모델 마이그레이션은 성능 변화뿐 아니라 토큰화 방식 변화까지 함께 점검해야 하는 이벤트라는 것이 핵심입니다.

11. 흔한 오해

오해 1

"context window가 크면 클수록 무조건 좋다"

context window가 커질수록 회상 정확도가 저하되는 context rot 현상이 여러 프론티어 모델에서 공통으로 관찰됩니다. 크기는 "가능성의 상한"일 뿐, 품질을 보장하지 않습니다.

오해 2

"토큰 수 = 단어 수 또는 글자 수"

토큰은 서브워드 단위로, 언어·표현 방식에 따라 같은 의미라도 토큰 수가 크게 달라집니다. 특히 영어 중심으로 학습된 토크나이저에서는 한국어 등 비영어권 언어가 상대적으로 더 많은 토큰을 소비하는 경향이 있습니다.

오해 3

"RAG는 context window가 작았던 시절의 임시방편이고, long-context 모델이 나오면 필요 없어진다"

RAG는 단순히 "다 못 넣으니까 쓰는" 기술이 아니라, 관련성 있는 정보만 선별해 넣어 정확도와 비용 모두를 개선하는 아키텍처입니다. context window가 커져도, 관련 없는 정보를 걸러내는 가치는 그대로 남습니다.

오해 4

"벤더가 발표한 context window 크기를 그대로 설계 기준으로 써도 된다"

claimed context window와 effective context window는 다를 수 있습니다. 특히 복잡한 추론 태스크일수록 그 간극이 커질 수 있으므로, 실제 설계 기준은 자체 eval로 측정한 effective context window여야 합니다.

오해 5

"컨텍스트 안에서는 정보의 위치가 중요하지 않다 — 어디에 있든 모델이 다 찾아낼 것이다"

여러 연구에서 정답 근거의 위치(맨 앞/맨 뒤 vs 중간)에 따라 회상률이 크게 달라지는 포지션 편향이 관찰됩니다. 중요한 지시나 근거는 의도적으로 유리한 위치에 배치하는 것이 실무적으로 유효한 전략입니다.
