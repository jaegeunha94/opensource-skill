Day 6 — Structured Output 설계

"JSON으로 답해줘"라는 부탁과 "스키마를 어길 수 없게 만드는 것"은 전혀 다른 기술이라는 것부터 시작하는 출력 계약 설계 · 2026-07-08

최신성 안내 — 이 레슨은 2026년 7월 기준 Anthropic Claude Platform Docs의 Structured Outputs 공식 문서(output_config.format, strict tool use), OpenAI의 Structured Outputs 공식 가이드(strict mode, 디코더 레벨 스키마 강제), Google Gemini의 구조화 출력 공식 문서(JSON Schema 지원 확대, propertyOrdering)를 확인해 작성되었습니다. 세 벤더 모두 "프롬프트로 JSON을 부탁하는 방식"에서 "샘플링 단계에서 스키마 위반 토큰 자체를 생성 불가능하게 만드는 constrained decoding 방식"으로 이미 전환을 마쳤고, 이제 차이는 세부 지원 범위(재귀 스키마, 정규식, 파라미터 개수 제한 등)에 있습니다. 구체적 파라미터 이름·제한 수치·모델명은 계속 바뀌므로, 암기 대상은 "구조화 출력은 스키마 준수를 보장하지만 내용의 정확성은 보장하지 않으며, 출력은 코드처럼 버저닝하고 검증해야 하는 계약(contract)"이라는 프레임워크입니다.

1. 핵심 면접 질문

Primary Question

"모델이 반환한 JSON이 가끔 스키마를 깨뜨려서 다운스트림 파싱이 실패합니다. 이 문제를 임시방편이 아니라 구조적으로 어떻게 해결하시겠어요?"

Variant 1

"OpenAI나 Anthropic의 structured output/strict mode를 쓰면 파싱 실패가 완전히 사라지나요? 그래도 여전히 남는 리스크는 무엇인가요?"

Variant 2

"여러 단계를 거치는 파이프라인에서 한 단계의 출력이 다음 단계의 입력이 됩니다. 스키마가 깨지거나 변경되면 전체 파이프라인이 어떻게 실패하고, 어떻게 설계해야 안전한가요?"

Variant 3

"구조화 출력이 스키마를 100% 지킨다고 해도 내용 자체가 틀릴 수 있습니다. 이 두 가지 문제를 어떻게 구분해서 다루시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 지원자가 "JSON으로 답해줘"라고 프롬프트에 적는 것과, 벤더가 제공하는 constrained decoding(샘플링 단계에서 스키마 위반 자체를 원천 차단하는 기법)의 차이를 정확히 아는지를 가른다. 더 중요한 것은, 구조화 출력 기능을 쓰면 "이제 파싱 문제는 끝났다"고 안심하는지, 아니면 "스키마 준수"와 "내용의 정확성"은 완전히 다른 층위의 문제라는 것을 알고 남은 리스크(refusal, 길이 초과, 환각된 값, 벤더별 스키마 지원 한계)까지 방어 설계를 하는지다. 실무에서 구조화 출력은 한 번 설정하고 끝나는 기능이 아니라, 여러 서비스가 공유하는 출력 계약(contract)이며, 이 계약이 깨질 때 파이프라인 전체가 어떻게 실패하는지를 설계해야 하는 문제다.

면접관이 이 질문으로 평가하는 것:

  • 메커니즘의 차이를 아는가 — 프롬프트로 형식을 "부탁"하는 것과, 디코더가 스키마를 위반하는 토큰 자체를 생성할 수 없게 만드는 것의 근본적 차이를 설명할 수 있는가.
  • "스키마 준수 ≠ 내용 정확성"을 구분하는가 — 구조화 출력이 보장하는 것(파싱 가능한 형식, 필수 필드 존재, 타입 일치)과 보장하지 않는 것(그 값이 사실인지, 존재하는 레코드를 가리키는지)을 명확히 구분하는가.
  • 벤더 제약을 실제로 아는가 — "아무 JSON Schema나 다 지원한다"는 막연한 인식이 아니라, 재귀 스키마·복잡한 정규식·파라미터 개수 등에 실제 제한이 있고 이를 설계 시 고려해야 한다는 것을 아는가.
  • 실패를 유형별로 나눠 다르게 대응하는가 — 스키마 위반(파싱 불가), refusal(안전 거부), 길이 초과(잘림), 내용 오류(스키마는 맞지만 값이 틀림)를 뭉뚱그리지 않고 각각 다른 방어 전략이 필요하다는 것을 아는가.
  • 출력을 "계약"으로 다루는 시스템 사고를 하는가 — 스키마를 코드처럼 버전 관리하고, 여러 단계·여러 팀이 공유하는 계약이 바뀔 때의 파급을 관리하는 절차를 제시하는가.

주니어~중니어에서 자주 나오는 답변 패턴: "JSON.parse가 실패하면 그냥 다시 요청하면 됩니다." 이 답변은 실패를 하나의 뭉뚱그려진 예외로만 다루고, 실패 유형을 구분하지 않으며, 재시도 후에도 같은 실패가 반복될 수 있다는 것과 벤더의 네이티브 구조화 출력 기능으로 애초에 그 실패의 상당 부분을 원천 차단할 수 있다는 것을 모른다는 신호다.

3. 전제 개념 — 쉬운 한국어로

Structured output(구조화 출력)이란 무엇인가

LLM은 원래 사람이 읽는 자연어 문장을 만들도록 학습된 모델이다. 그런데 실제 서비스에서는 모델의 출력을 사람이 아니라 코드가 파싱해서 다음 로직에 넘겨야 하는 경우가 훨씬 많다 — 예를 들어 "이 리뷰의 감정을 분석해서 {"sentiment": "positive", "confidence": 0.9} 형태로 달라"는 요청이다. Structured output은 모델의 출력이 미리 정의한 JSON Schema를 반드시 따르도록 강제하는 기능이다. 비유하면, 자유 서술형 답안지에 "표 형식으로 써주세요"라고 부탁하는 것과, 아예 정해진 칸이 있는 OMR 답안지를 주고 그 칸 안에만 표시하게 만드는 것의 차이다.

왜 "JSON으로 답해줘"라는 프롬프팅만으로는 부족한가

가장 단순한 방법은 프롬프트에 "반드시 이런 JSON 형식으로 답하라"고 적는 것이다. 하지만 이것은 모델에게 부탁하는 것이지 강제하는 것이 아니다. 모델은 여전히 자유롭게 토큰을 생성하기 때문에, 코드블록으로 감싸거나("```json ... ```"), 앞에 "다음은 요청하신 결과입니다:" 같은 서론을 붙이거나, 따옴표를 깜빡하거나, 필수 필드를 빠뜨릴 수 있다. OpenAI가 이 기능을 처음 발표할 당시 공개한 자체 평가 수치가 이 차이를 잘 보여준다 — 네이티브 구조화 출력 없이 프롬프트만으로 복잡한 스키마를 따르게 했을 때 정확도는 40% 미만이었지만, 디코더 단에서 스키마를 강제하는 기능을 적용하자 100%로 올라갔다. 이 격차가 바로 "부탁"과 "강제"의 차이다.

Constrained decoding(제약된 디코딩)의 원리

모델은 다음 토큰을 생성할 때, 가능한 모든 토큰 각각에 확률을 매기고 그중 하나를 선택한다. Constrained decoding은 이 선택 단계에서, 스키마를 위반하게 되는 토큰의 확률을 0으로 만들어 애초에 선택될 수 없게 만드는 기법이다. 예를 들어 스키마가 "status" 필드를 "active" 또는 "inactive" 중 하나로 요구한다면, 모델이 그 필드를 채울 차례가 됐을 때 이 두 문자열을 구성하는 토큰 외에는 애초에 생성 후보에서 제외된다. 사람으로 비유하면, 객관식 시험에서 답안지의 정답 칸 외에는 물리적으로 펜이 닿지 않게 만드는 것과 같다 — 수험생이 실수로 다른 곳에 쓸 가능성 자체가 없어진다. 이 방식은 생성 후 결과를 검사해서 걸러내는 후처리 검증과 근본적으로 다르다.

가장 중요한 구분: "스키마를 지키는 것"과 "내용이 맞는 것"은 다른 문제다. Constrained decoding은 모델이 형식적으로 유효한 JSON을, 스키마가 요구하는 타입으로, 필수 필드를 빠짐없이 채우도록 보장한다. 하지만 그 필드에 들어간 값이 사실인지, 실제로 존재하는 고객 ID를 가리키는지, 정확한 계산 결과인지는 전혀 보장하지 않는다. 모델은 스키마가 요구하는 형식을 정확히 지키면서도 완전히 틀리거나 존재하지 않는 값을 자신 있게 채울 수 있다. 이 구분을 놓치면, "구조화 출력을 도입했으니 이제 안전하다"는 잘못된 안도감에 빠지기 쉽다.

Strict tool use vs JSON output — 강제 대상이 다르다

구조화 출력은 크게 두 위치에 적용된다. 하나는 최종 응답 자체의 형식을 강제하는 것(JSON output)이고, 다른 하나는 모델이 도구(tool)를 호출할 때 넘기는 파라미터를 강제하는 것(strict tool use)이다. 후자는 Day 7에서 다룰 tool calling과 맞닿아 있지만, Day 6에서는 "출력 스키마를 신뢰 가능하게 만드는 메커니즘 자체"에 집중하고, 멀티스텝 tool calling의 신뢰성·실패 복구·에이전틱 아키텍처는 Day 7에서 깊게 다룬다.

4. 최신 정보 스냅샷 — 2026년 7월 기준 구조화 출력 지형

Anthropic — output_config.format과 strict tool use

Anthropic 공식 문서에 따르면, JSON 출력 강제는 output_config.format(과거의 output_format 파라미터에서 이동, 과거 방식도 전환 기간 동안 계속 동작)에 JSON Schema를 지정하는 방식으로 이루어지며, 도구 호출 파라미터를 강제하는 strict tool use(strict: true)와 함께 쓸 수 있다. Claude Fable 5, Opus 4.8/4.7/4.6, Sonnet 5/4.6/4.5, Opus 4.5, Haiku 4.5 등 주요 모델에서 GA로 제공된다. 다만 지원되는 JSON Schema 범위에는 명확한 한계가 있다 — 재귀 스키마 미지원, 숫자/문자열 길이 제약(minimum/maximum/minLength/maxLength) 미지원, 복잡한 정규식(백레퍼런스·룩어헤드) 미지원, additionalProperties는 반드시 false여야 함 등이다. 요청당 strict tool 최대 20개, 옵셔널 파라미터 총 24개, anyOf 유니온 타입 파라미터 최대 16개라는 복잡도 한도도 있다. 또한 refusal(안전상 이유로 스키마를 따르지 않고 거부, 그래도 과금됨)과 길이 초과(max_tokens 도달 시 출력이 스키마를 어길 수 있음) 두 가지는 구조화 출력이 강제해도 여전히 발생할 수 있는 실패로 문서에 명시되어 있다. enum 값의 대소문자까지는 보장하지 않는다는 세부 사항도 있다.

OpenAI — Strict mode: "디코더가 물리적으로 스키마 위반 토큰을 낼 수 없다"

OpenAI의 구조화 출력은 strict: true와 함께 JSON Schema를 지정하는 방식이며, 공식 문서와 업계 설명 모두 이를 "디코더가 스키마를 어기는 토큰을 애초에 생성할 수 없다"는 샘플링 레벨의 보장으로 설명한다. 2026년 기준 GPT-5.5 등 최신 모델 계열에서 strict mode가 지원되며, 형식만 유효한 JSON을 보장하고 스키마 준수는 보장하지 않는 과거의 JSON mode(json_object)는 레거시로 격하되었다. 모델이 스키마를 따르는 대신 요청을 거부할 경우를 프로그램적으로 감지할 수 있도록 refusal 필드가 응답에 포함된다. 이 기능의 배경이 된 OpenAI 자체 평가 수치는 여전히 유효한 참고점이다 — 네이티브 구조화 출력 없이 복잡한 스키마를 프롬프트만으로 따르게 했을 때는 정확도가 40% 미만이었지만, 구조화 출력을 적용하자 100%로 올랐다.

Google Gemini — JSON Schema 지원 확대와 propertyOrdering

Gemini API는 response_schema(및 responseMimeType: application/json)로 구조화 출력을 강제하며, 지원 모델 전반에 걸쳐 JSON Schema 지원 범위가 확대되어 Pydantic(Python)·Zod(TypeScript) 같은 라이브러리가 별도 변환 없이 바로 연동된다. 필드 순서를 스키마에 정의한 순서와 동일하게 유지하는 propertyOrdering 개념도 제공되는데, 이는 멀티스텝 파이프라인에서 "이전 필드 값을 참고해 다음 필드를 채우는" 방식의 추론 품질에 실질적인 영향을 준다.

오픈웨이트 모델 자체 호스팅 — Outlines / llguidance류 grammar 기반 라이브러리

벤더 API가 아니라 오픈웨이트 모델을 직접 서빙하는 경우에는 네이티브 구조화 출력이 없으므로, Outlines, llguidance 같은 grammar 기반 constrained decoding 라이브러리를 추론 서버(vLLM, TGI 등)에 통합해 동일한 원리(스키마 위반 토큰의 확률을 0으로 마스킹)를 자체 구현하는 것이 표준적인 방법이다. 벤더 네이티브 기능이 있는지 여부가 "구조화 출력을 어떻게 확보할 것인가"의 첫 번째 분기점이 된다.

Instructor류 retry-with-error-feedback 라이브러리 — 네이티브 기능을 보완하는 계층

네이티브 구조화 출력이 스키마 위반 자체는 막아주지만, refusal·길이 초과·비즈니스 규칙 위반(참조 무결성 등)까지 막아주지는 않는다. Instructor 같은 라이브러리가 대중화한 패턴은, Pydantic 등으로 검증에 실패했을 때 실패 원인을 모델에게 다시 알려주고 재시도시키는 피드백 루프다(보통 최대 2~3회 제한). 이는 네이티브 기능을 대체하는 것이 아니라, 네이티브 기능이 커버하지 못하는 남은 실패 유형을 처리하는 보완 계층으로 이해해야 한다.

실무 사례 — 검증 계층 없이 도구 호출 결과를 그대로 실행한 사고

업계에 보고된 사례로, 어떤 프로덕션 시스템이 모델이 반환한 고객 ID를 그대로 신뢰해 환불 API를 호출하도록 구성했는데, 모델이 존재하지 않는 고객 ID를 스키마 형식은 완벽하게 지킨 채로 생성해 실제 환불이 처리되었고, 모니터링 알림이 울릴 때까지 상당한 금전적 손실이 발생한 사례가 보고된다. 근본 원인은 스키마 검증(형식이 맞는가)과 비즈니스 규칙 검증(그 고객 ID가 실제로 존재하는가, 그 금액이 정책상 허용 범위인가)을 구분하지 않고, 스키마만 통과하면 바로 실행에 넘긴 설계였다. 이 사례는 "구조화 출력=안전"이라는 오해가 실제로 어떤 손실로 이어질 수 있는지를 보여주는 대표적인 패턴이다.

이 스냅샷에서 인터뷰용으로 가져가야 할 한 문장: "주요 벤더가 프롬프트 기반 JSON 유도에서 디코더 레벨 constrained decoding으로 전환하면서 스키마 위반으로 인한 파싱 실패는 사실상 해소되었지만, 이는 형식의 문제를 해결한 것이지 내용의 정확성이나 비즈니스 규칙 위반까지 해결한 것은 아니므로, 스키마 검증과 별개로 비즈니스 규칙 검증 계층을 반드시 두어야 한다는 것이 지금 업계가 확인한 표준입니다."

5. 출력 계약 설계 프레임워크

인터뷰에서 구조화 출력을 이야기할 때는 "어떻게 강제할 것인가"와 "그래도 남는 실패를 어떻게 처리할 것인가"를 분리해서 구조화하면 신호가 명확해진다.

축 핵심 질문 실무 기법
① 강제 방식 선택 네이티브 constrained decoding을 쓸 수 있는가? 벤더 API는 네이티브 기능 우선, 자체 호스팅은 Outlines/llguidance로 자체 구현
② 실패 유형 분류 스키마 위반 / refusal / 길이 초과 / 내용 오류를 구분했는가? 4가지 실패 유형별로 원인이 다르므로 각각 다른 대응 전략 설계
③ 이중 검증 계층 형식 검증과 비즈니스 규칙 검증을 분리했는가? 스키마 검증(타입/필수필드) + 참조 무결성·범위·정책 검증을 별도 단계로 실행
④ 재시도·폴백 전략 검증 실패 시 무한 재시도가 아니라 언제 멈추는가? 에러를 모델에 되돌려주는 피드백 루프(최대 N회) 후 폴백 경로(기본값/사람 개입/실패 응답)
⑤ 계약 버저닝 스키마를 바꿀 때 하위 호환을 어떻게 유지하는가? 스키마를 코드처럼 버전 관리, 필드 추가는 옵셔널로, 제거는 단계적 deprecation
⑥ 관측 실패율·재시도율이 회귀했는지 어떻게 아는가? 스키마 위반율, refusal율, 재시도 횟수 분포를 지표화해 배포 전후 비교

왜 ①(강제 방식)만으로는 부족한가

많은 팀이 "네이티브 구조화 출력을 켰다"에서 설계를 멈춘다. 그러나 Day 6 앞부분에서 짚었듯 구조화 출력은 형식만 보장하지 내용을 보장하지 않는다. 시니어 엔지니어의 답변은 ①에서 그치지 않고, ②~⑥(남은 실패를 유형별로 어떻게 처리하고, 출력을 계약으로서 어떻게 운영하는지)까지 포함해야 완결된다.

6. 상황별 전략 매핑

  • 1
    "벤더 API(Claude/GPT/Gemini)를 쓰고 있다" → 네이티브 구조화 출력을 우선 적용

    프롬프트로 JSON 형식을 설명하는 방식 대신 output_config.format/strict: true/response_schema 등 네이티브 기능을 우선 사용해 스키마 위반 실패 자체를 원천 차단한다.

  • 2
    "오픈웨이트 모델을 자체 호스팅한다" → Grammar 기반 constrained decoding 라이브러리로 자체 구현

    벤더 네이티브 기능이 없으므로 Outlines·llguidance 등을 추론 서버에 통합해 동일한 원리(위반 토큰 마스킹)를 직접 확보한다. 순수 프롬프트+파싱+재시도만으로는 벤더 API 수준의 신뢰성에 못 미친다.

  • 3
    "스키마가 벤더 제약(재귀·정규식·필드 수)을 초과한다" → 스키마를 단순화하고 후처리로 보완

    재귀 구조를 평탄화하거나 여러 번의 호출로 나누고, 벤더가 지원하지 않는 세밀한 제약(길이·범위·복잡한 패턴)은 스키마 강제 밖에서 별도의 비즈니스 규칙 검증으로 처리한다.

  • 4
    "여러 단계 파이프라인에서 한 단계 출력이 다음 단계 입력이 된다" → 각 경계에 검증 게이트 설치

    단계 경계마다 스키마 검증과 비즈니스 규칙 검증을 모두 통과해야 다음 단계로 넘기고, 실패 시 그 경계에서 멈춰 재시도하거나 폴백한다. 검증 없이 통과시키면 실패가 여러 단계 뒤에서야 드러나 원인 추적이 어려워진다.

  • 5
    "스키마는 맞는데 값이 틀렸다(환각)" → 이것은 구조화 출력의 책임 범위 밖임을 인지

    형식이 맞다고 값을 그대로 신뢰하지 않는다. 참조 무결성(실제 존재하는 ID인가), 정책 범위(허용된 금액·권한인가) 검증을 별도로 두고, 근본적인 환각 완화는 Day 13에서 grounding·인용 전략으로 深化한다.

7. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 된다.

  1. 프롬프트 기반 JSON 유도와 네이티브 constrained decoding을 구분해서 짚습니다. "부탁"과 "강제"의 메커니즘 차이를 먼저 설명합니다.
  2. 실패를 유형별로 분류합니다. 스키마 위반(형식 자체가 깨짐), refusal(안전 거부), 길이 초과(잘림), 내용 오류(형식은 맞지만 값이 틀림) — 각각 원인과 대응이 다릅니다.
  3. 벤더 네이티브 기능으로 스키마 위반을 원천 차단할 수 있는지 확인하되, 그 기능의 지원 범위 한계(재귀 불가 등)도 함께 고려합니다.
  4. 남은 실패 유형에 대한 이중 검증 계층(형식+비즈니스 규칙)과 재시도·폴백 절차를 설계합니다.
  5. 출력을 "계약"으로 다루는 버저닝·관측 체계로 마무리합니다. 스키마 변경이 여러 단계·여러 팀에 미치는 파급을 관리하는 절차를 언급합니다.

8. 핵심 답변 예시

면접관: "모델이 반환한 JSON이 가끔 스키마를 깨뜨려서 다운스트림 파싱이 실패합니다. 이 문제를 임시방편이 아니라 구조적으로 어떻게 해결하시겠어요?"

STRONG ANSWER

먼저 지금 방식이 프롬프트로 "JSON 형식으로 답해달라"고 부탁하는 방식인지 확인하겠습니다. 그렇다면 근본 원인은 명확합니다 — 프롬프트 유도는 모델에게 형식을 부탁하는 것이지 강제하는 것이 아니라서, 코드블록으로 감싸거나 서론을 붙이거나 필드를 빠뜨리는 실패가 구조적으로 발생할 수밖에 없습니다. 저라면 벤더가 제공하는 네이티브 구조화 출력(Anthropic의 output_config.format, OpenAI의 strict mode, Gemini의 response_schema)으로 전환합니다. 이 방식은 프롬프트 유도와 달리 디코더가 스키마를 위반하는 토큰 자체를 생성하지 못하도록 샘플링 단계에서 막기 때문에, 형식적으로 유효하지 않은 JSON이 나오는 실패는 사실상 사라집니다.


다만 여기서 멈추지 않습니다. 구조화 출력이 보장하는 것은 형식이지 내용이 아닙니다. 스키마를 완벽히 지키면서도 존재하지 않는 ID나 틀린 값을 채울 수 있고, 안전상 이유로 refusal이 나오거나, 출력이 길어져 max_tokens에 걸리면 여전히 스키마가 깨질 수 있습니다. 그래서 검증을 두 층으로 나눕니다. 첫 번째 층은 스키마 검증(타입·필수 필드가 맞는가)이고, 두 번째 층은 비즈니스 규칙 검증(그 ID가 실제로 존재하는가, 그 금액이 정책 범위 안인가)입니다. 두 층을 모두 통과해야 실제 실행이나 다음 단계로 넘어가고, 실패하면 에러 원인을 모델에게 되돌려주는 재시도를 최대 2~3회로 제한한 뒤, 그래도 실패하면 사람 개입이나 안전한 기본값으로 폴백합니다.


마지막으로 이 스키마를 일회성 설정이 아니라 계약으로 다룹니다. 여러 단계나 여러 팀이 이 스키마를 공유한다면, 필드를 추가할 때는 옵셔널로 시작하고 제거할 때는 단계적으로 deprecation하며, 스키마 위반율·refusal율·재시도 횟수를 지표로 남겨 배포 전후 회귀를 관측합니다. 이렇게 하면 "가끔 깨진다"는 증상이 아니라, 어떤 유형의 실패가 얼마나 자주 발생하는지를 근거로 계속 개선할 수 있는 시스템이 됩니다.

이 답변이 좋은 이유:

  • 프롬프트 유도와 constrained decoding의 메커니즘 차이를 정확히 짚고 원인 진단부터 시작함
  • "구조화 출력=완전 해결"이라는 함정에 빠지지 않고 형식과 내용을 구분함
  • 이중 검증 계층(스키마+비즈니스 규칙)과 제한된 재시도+폴백을 구체적으로 설계함
  • 출력을 계약으로 다루는 버저닝·관측 관점까지 포함해 일회성 수정이 아닌 시스템 사고를 보여줌

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"JSON.parse가 실패하면 catch에서 잡아서 그냥 다시 요청을 보내면 됩니다."

이 답변이 약한 이유

  • 실패 원인을 진단하지 않고 즉시 재시도로 넘어가, 프롬프트 유도 방식의 근본 문제를 그대로 방치합니다.
  • 실패 유형(스키마 위반/refusal/길이초과/내용오류)을 구분하지 않아, 재시도로 해결되지 않는 유형(예: 반복적인 refusal)에도 무한정 재시도할 위험이 있습니다.
  • 재시도 횟수 제한이나 폴백 경로가 없어 장애 시 무한 루프나 비용 폭증으로 이어질 수 있습니다.
WEAK ANSWER — 또 다른 패턴

"벤더의 structured output 기능을 켜면 스키마가 100% 보장되니까 이제 검증 로직은 따로 필요 없습니다."

이 답변이 약한 이유

  • "스키마 준수"와 "내용 정확성"을 구분하지 못해, 형식은 맞지만 값이 틀리거나 존재하지 않는 참조를 그대로 실행에 넘기는 리스크를 방치합니다.
  • refusal·길이 초과처럼 구조화 출력을 켜도 여전히 발생할 수 있는 실패 유형을 놓치고 있습니다.
STRONG ANSWER 요소 체크리스트
  • 프롬프트 기반 JSON 유도와 constrained decoding의 메커니즘 차이를 설명한다
  • 실패를 스키마 위반/refusal/길이초과/내용오류로 구분하고 각각 다른 대응을 제시한다
  • 스키마 검증과 비즈니스 규칙 검증을 별도 계층으로 설계한다
  • 재시도 횟수를 제한하고 폴백 경로(사람 개입/기본값)를 둔다
  • 스키마를 버전 관리하고 위반율·재시도율을 관측 지표로 남긴다

10. Senior-level Follow-up Q&A

Q: 구조화 출력이 스키마를 100% 지킨다면, 왜 여전히 검증 계층이 필요한가요?

모범 답변: 구조화 출력이 강제하는 것은 형식(타입, 필수 필드, 값의 후보군)이지, 그 값이 사실인지·실제로 존재하는 레코드를 가리키는지가 아닙니다. 모델은 스키마를 완벽히 지키면서도 존재하지 않는 고객 ID나 틀린 계산 결과를 자신 있게 채울 수 있습니다. 그래서 스키마 검증과 별개로, 그 값이 실제 시스템 상태와 일치하는지 확인하는 비즈니스 규칙 검증(참조 무결성, 권한·정책 범위 확인)이 반드시 필요합니다. 형식 검증을 통과했다고 바로 실행에 넘기는 것은 "스키마 준수"와 "안전한 실행"을 혼동하는 것입니다.

Q: 멀티스텝 파이프라인에서 스키마를 바꿔야 할 때, 어떻게 안전하게 마이그레이션하나요?

모범 답변: 스키마를 코드처럼 버전 관리합니다. 필드를 추가할 때는 처음에 반드시 옵셔널로 넣어 기존 소비자가 깨지지 않게 하고, 일정 기간 두 버전을 동시에 지원한 뒤 소비자들이 전환을 마치면 이전 버전을 제거합니다. 필드를 제거하거나 타입을 바꾸는 것은 더 신중해야 하는데, 그 필드를 참조하는 모든 다운스트림 단계를 먼저 파악하고, deprecation 경고 기간을 두고, 스키마 위반율·파싱 실패율을 배포 전후로 비교해 회귀가 없는지 확인한 뒤 완전히 전환합니다. 여러 팀이 같은 스키마를 공유한다면 이 변경을 API 계약 변경과 동일한 수준의 리뷰·공지 절차로 다룹니다.

Q: 오픈웨이트 모델을 자체 호스팅할 때는 벤더 네이티브 기능이 없는데, 어떻게 신뢰성을 확보하나요?

모범 답변: 벤더 API의 구조화 출력과 동일한 원리 — 샘플링 단계에서 스키마 위반 토큰의 확률을 마스킹하는 constrained decoding — 을 Outlines나 llguidance 같은 grammar 기반 라이브러리로 추론 서버에 직접 통합합니다. 단순히 프롬프트로 형식을 설명하고 파싱 실패 시 재시도하는 방식만으로는 벤더 API 수준의 신뢰성에 미치지 못한다는 것을 알고, 자체 호스팅 환경에서도 동일한 수준의 강제 메커니즘을 확보하는 것을 목표로 삼습니다. 다만 이 경우 스키마 컴파일에 따른 첫 요청 지연이나 지원 문법의 범위 등을 별도로 검증해야 합니다.

Q: refusal이나 길이 초과(max_tokens)로 인한 실패는 재시도로 해결되나요?

모범 답변: 둘 다 단순 재시도만으로는 근본적으로 해결되지 않는 경우가 많습니다. Refusal은 모델이 안전상 이유로 스키마를 따르지 않고 거부하는 것이라, 같은 입력으로 재시도해도 같은 이유로 다시 거부될 가능성이 높습니다 — 이 경우는 입력 자체(민감한 요청인지)를 재검토하거나 사람에게 에스컬레이션하는 것이 맞는 대응입니다. 길이 초과는 max_tokens를 늘려 재시도하는 것이 합리적인 해결책이지만, 무제한으로 늘리면 비용이 커지므로 스키마가 요구하는 최대 크기를 사전에 추정해 적절한 상한을 두는 것이 우선입니다. 두 경우 모두 "일단 재시도"가 아니라 실패 원인에 맞는 대응을 먼저 판단해야 합니다.

Q (비기술적): 비개발 조직에게 "AI가 만든 표(데이터)가 가끔 형식이 깨진다"는 문제를 어떻게 설명하시겠어요?

모범 답변: 비유로 설명합니다. "예전에는 AI에게 '이런 양식으로 채워주세요'라고 말로 부탁하는 방식이었는데, 사람도 말로 부탁받으면 가끔 양식을 깜빡하듯 AI도 그랬습니다. 최근에는 아예 정해진 칸이 있는 서식지를 주고 그 칸 밖에는 물리적으로 쓸 수 없게 만드는 방식으로 바뀌어서, 양식 자체가 깨지는 문제는 거의 사라졌습니다. 다만 이건 '양식을 지키는 것'이지 '내용이 맞는 것'을 보장하지는 않아서, 저희는 별도로 그 내용이 실제 데이터와 맞는지 한 번 더 확인하는 절차를 두고 있습니다."

11. 흔한 오해

오해 1

"JSON.parse()가 성공하면 내용도 정확하다는 뜻이다"

구조화 출력이 보장하는 것은 형식(파싱 가능, 필수 필드, 타입)이지 값의 사실성이나 참조 무결성이 아니다. 형식이 완벽해도 존재하지 않는 값이나 틀린 계산 결과가 들어있을 수 있다.

오해 2

"structured output/strict mode를 켜면 이제 검증·재시도 로직은 필요 없다"

refusal, 길이 초과로 인한 잘림, 벤더가 지원하지 않는 스키마 제약 초과 등은 네이티브 기능을 켜도 여전히 발생할 수 있는 실패다. 이중 검증 계층과 제한된 재시도·폴백은 여전히 필요하다.

오해 3

"스키마는 한 번 짜면 바뀌지 않는다"

제품 요구사항이 바뀌면 스키마도 바뀐다. 여러 단계·여러 팀이 공유하는 출력 계약이라면 API 버저닝과 동일한 수준의 하위 호환 관리(옵셔널 필드 추가, 단계적 deprecation)가 필요하다.

오해 4

"모든 벤더가 임의로 복잡한 JSON Schema를 다 지원한다"

실제로는 재귀 스키마, 복잡한 정규식, 숫자/문자열 길이 제약, 파라미터 개수 등에 벤더별 제한이 있다. 스키마를 설계할 때 이 제약을 미리 확인하지 않으면 배포 후에 실패한다.

오해 5

"프롬프트에 형식을 잘 설명하면 네이티브 구조화 출력과 비슷한 신뢰성을 얻을 수 있다"

프롬프트 유도는 모델에게 형식을 부탁하는 것이고, 네이티브 구조화 출력은 디코더 단에서 위반 자체를 원천 차단하는 것이다. 복잡한 스키마에서는 이 차이가 40% 미만 대 100%라는 격차로 나타난 바 있다.
