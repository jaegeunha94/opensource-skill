2. 전제 개념 — 쉬운 한국어로

왜 권한 모델을 별도로 심화해야 하는가

Day 1에서 "checkpoint(파일 되돌리기)와 permission(승인 통제)의 역할이 분리된다"는 것과 4대 권한 모드의 개요를, Day 3에서 "CLAUDE.md는 강제 계층이 아니므로 절대 규칙은 훅/권한으로 이중화해야 한다"는 것을 배웠다. 이번 Day는 그 "권한/훅 계층"이 실제로 어떤 문법과 순서로 판단을 내리는지를 심화한다. 시니어 인터뷰에서 이 부분이 중요한 이유는, 권한 규칙은 "그럴듯해 보이지만 실제로는 우회되는 규칙"을 쓰기 쉬운 영역이기 때문이다 — 문법을 정확히 알아야 실제로 안전한 규칙을 설계할 수 있다.

6개 Permission Mode — "얼마나 자주 멈추고 물어보는가"의 스펙트럼

모드프롬프트 없이 실행되는 것적합한 상황
default (CLI/IDE 라벨: Manual, v2.1.200+)읽기 전용 작업만시작 단계, 민감한 작업
acceptEdits파일 편집 + 일부 안전한 파일시스템 Bash(mkdir/touch/rm/rmdir/mv/cp/sed, 안전한 환경변수·프로세스 래퍼 접두 포함)리뷰 예정인 코드를 반복 수정할 때
plan읽기 전용(탐색만, 편집 없음)변경 전 코드베이스 분석
auto (research preview)전부, 단 백그라운드 안전성 classifier가 검증긴 작업, 프롬프트 피로도 감소
dontAsk사전 승인된 도구만(permissions.allow 매치 + 읽기 전용 Bash)잠금된 CI/스크립트 환경
bypassPermissions전부(단 protected path 예외 있음, 아래 참고)격리된 컨테이너/VM 전용

default의 CLI/IDE 표시 이름이 v2.1.200부터 "Manual"로 바뀐 것이 최근 변화다. 설정값 자체(defaultMode, hooks, SDK가 참조하는 값)는 여전히 default이고, CLI가 manual을 별칭으로 허용할 뿐이다. 모드는 "기준선(baseline)"이고, 그 위에 allow/ask/deny 규칙을 얹어 특정 도구를 사전 승인하거나 차단한다 — 즉 모드와 규칙은 서로 다른 층이다.

모드 전환 — surface마다 사용 가능한 모드가 다르다

CLI에서는 Shift+Tab으로 default → acceptEdits → plan을 순환하고, bypassPermissions/auto는 활성화 조건을 충족해야 순환 목록에 추가된다. 흥미로운 점은 surface에 따라 애초에 제공되는 모드 목록이 다르다는 것이다.

Surface제공되는 모드
CLI / VS Code / JetBrains / Desktop전체 6종(단 auto/bypassPermissions는 별도 활성화 필요)
Web(claude.ai/code, Cloud 세션)Accept edits · Plan · Auto — bypassPermissions 불가. Cloud 환경은 파일 편집을 이미 사전 승인하므로 default가 "Accept edits"로 표시
Remote Control 세션Ask permissions · Auto accept edits · Plan — auto/bypassPermissions 불가

Web 세션은 defaultMode: "bypassPermissions"나 "dontAsk"를 설정에 적어도 조용히 무시한다 — 저장소에 커밋된 설정 파일이 클라우드 세션을 bypass 모드로 몰래 띄울 수 없게 막는 설계다.

규칙 3종과 평가 순서 — "구체성은 순서를 바꾸지 않는다"

allow(승인 없이 실행), ask(항상 프롬프트), deny(차단) 세 종류가 있고, 평가 순서는 항상 deny → ask → allow다. 가장 오해하기 쉬운 지점: 규칙이 더 구체적이라고 해서 순서가 바뀌지 않는다.

구체성이 순서를 못 이기는 예시

deny: ["Bash(aws *)"] + allow: ["Bash(aws s3 ls)"]를 함께 설정해도, aws s3 ls는 차단된다. 넓은 deny가 먼저 평가되어 매치되면 그 시점에서 끝나고, 더 구체적인 allow는 아예 평가되지 않는다. 즉 deny 규칙에는 "예외"를 못 둔다 — 예외가 필요하면 deny 규칙 자체를 더 좁게 다시 써야 한다.

bare 도구명(예: Bash)을 deny하면 그 도구 자체가 Claude의 컨텍스트에서 완전히 제거되어(Claude가 그런 도구가 있는지조차 모르게 됨), scoped deny(예: Bash(rm *))는 도구는 그대로 남기고 매칭되는 특정 호출만 차단한다는 차이도 있다.

Tool별 규칙 문법 — 안전해 보이지만 실제로는 함정인 지점들

Bash — wildcard, compound command, 프로세스 래퍼

  • Bash(npm run *)처럼 *는 어느 위치에나 올 수 있고 공백 포함 임의 길이를 매치한다. Bash(ls *)(공백 있음)는 ls -la는 매치하되 lsof는 매치하지 않는다(단어 경계 강제) — 반면 Bash(ls*)(공백 없음)는 둘 다 매치한다.
  • compound command(&&, ||, ;, |, |&, &, 줄바꿈으로 연결)는 각 하위 명령이 독립적으로 규칙에 매치되어야 한다. Bash(safe-cmd *)는 safe-cmd && other-cmd 전체를 승인하지 않는다. 사용자가 "Yes, don't ask again"으로 compound command를 승인하면, 시스템은 전체 문자열이 아니라 하위 명령별로 최대 5개 규칙을 따로 저장한다.
  • timeout/time/nice/nohup/stdbuf와 무옵션 xargs는 매칭 전에 자동으로 벗겨지는 내장 wrapper라 Bash(npm test *) 규칙이 timeout 30 npm test에도 적용된다. 반면 direnv exec, devbox run, npx, docker exec 같은 환경 러너는 벗겨지지 않는다 — Bash(devbox run *)는 devbox run rm -rf .까지 그대로 허용해 버리는 함정이 된다. 이런 러너를 쓸 때는 Bash(devbox run npm test)처럼 러너와 내부 명령을 통째로 규칙에 박아야 한다.
  • ls, cat, grep, find, git의 읽기 전용 형태 등은 어떤 모드에서도 프롬프트 없이 실행되는 내장 read-only 목록이며, 설정으로 끌 수 없다(막으려면 ask/deny 규칙 추가).
curl 필터링의 구조적 함정

Bash(curl http://github.com/ *)로 curl을 GitHub로만 제한하려는 시도는 실패하기 쉽다: 옵션을 앞에 붙이거나(curl -X GET http://github.com/...), 프로토콜을 바꾸거나(https://), 리다이렉트를 태우거나(bit.ly → GitHub로 302), 변수로 치환하면(URL=http://github.com && curl $URL) 전부 우회된다. 공식 문서가 권장하는 대안은 ① Bash에서 curl/wget류를 deny하고 WebFetch(domain:example.com) allow로 대체 ② PreToolUse 훅으로 URL을 직접 검증 ③ CLAUDE.md 안내(강제력 없는 보조 수단일 뿐)의 조합이다.

Read / Edit — gitignore 스타일 4종 패턴과 심볼릭 링크

패턴의미예시
//path파일시스템 루트 기준 절대경로Read(//Users/alice/secrets/**)
~/path홈 디렉토리 기준Read(~/Documents/*.pdf)
/path규칙이 정의된 settings 소스 기준 앵커(프로젝트 설정이면 프로젝트 루트, 사용자 설정이면 ~/.claude/)Edit(/src/**/*.ts)
path / ./path현재 디렉토리 기준Read(*.env)

/Users/alice/file처럼 슬래시 하나로 시작하는 패턴은 절대경로가 아니다 — settings 소스 기준 앵커이므로, 정말 파일시스템 루트를 지정하려면 //Users/alice/file처럼 슬래시 두 개를 써야 한다. 심볼릭 링크는 allow와 deny가 비대칭으로 동작한다: allow 규칙은 심볼릭 링크 경로와 그 타깃 경로가 둘 다 매치해야 자동 승인되고(하나만 매치하면 프롬프트로 폴백), deny 규칙은 둘 중 하나만 매치해도 차단한다. 즉 ./project/**가 allow, ~/.ssh/**가 deny인 상태에서 ./project/key가 ~/.ssh/id_rsa를 가리키는 심볼릭 링크라면, 타깃이 deny에 걸려 차단된다.

WebFetch / MCP / Agent / Cd

  • WebFetch: WebFetch(domain:example.com)는 호스트네임을 매치하고, domain:*.example.com은 임의 깊이의 서브도메인을 매치하되 example.com 자체는 매치하지 않는다. example.*처럼 점(.)을 넘어가는 와일드카드는 매치되지 않도록 설계되어 있다(공격자가 임의 도메인을 등록해 우회하는 것을 막기 위함) — 단 맨 앞의 *.나 단독 *는 예외.
  • MCP: mcp__서버명은 그 서버의 모든 도구, mcp__서버명__*도 동일, mcp__서버명__도구명은 특정 도구 하나. allow 규칙은 서버 세그먼트가 글롭 없이 명시된 뒤에만 와일드카드를 허용한다(임의 서버까지 걸치는 "*" 같은 allow 글롭은 경고와 함께 무시됨).
  • Agent: Agent(Explore)처럼 서브에이전트 이름으로 차단할 수 있다. 특정 서브에이전트를 조직 정책으로 막을 때 쓴다.
  • Cd(신규): /cd 명령이 이동 가능한 디렉토리를 통제한다. 모델이 직접 호출할 수 있는 도구가 아니라 사용자가 /cd를 칠 때만 적용되는 규칙이며, allow 규칙을 하나라도 추가하면 /cd가 허용목록 모드로 전환된다.

Tool(param:value) — 매개변수 기반 매칭(v2.1.178+)

deny/ask 규칙은 도구의 최상위 스칼라 파라미터를 직접 매치할 수 있다: Agent(model:opus)(Opus 모델을 요청하는 서브에이전트 차단), Agent(isolation:worktree), Bash(run_in_background:true). 단 canonicalizing 필드(Bash/PowerShell의 command, Read/Edit/Write의 file_path, Grep/Glob의 path, WebFetch의 url)는 이 방식으로 매치할 수 없다 — 예를 들어 Bash(command:rm *)는 compound command로 쉽게 우회되므로 시작 시 경고와 함께 무시되고, 대신 전용 문법인 Bash(rm *)를 써야 한다.

Protected Paths — allow 규칙조차 뚫지 못하는 안전망

.git, .config/git, .vscode, .idea, .husky, .cargo, .devcontainer, .yarn, .mvn, .claude(단 .claude/worktrees 제외) 같은 디렉토리와 .gitconfig, 각종 shell rc 파일, .npmrc, .mcp.json, .claude.json 등의 쓰기는 bypassPermissions를 제외한 모든 모드에서 자동 승인되지 않는다.

모드protected path 쓰기 처리
default, acceptEdits, plan프롬프트
autoclassifier로 라우팅(allow 규칙이 매치해도 classifier를 거침)
dontAsk거부
bypassPermissions허용(v2.1.126부터 — 이전 버전은 이 모드에서도 프롬프트했음)

핵심은 ~/.claude/settings.json이나 .claude/settings.json에 Edit(.claude/**) 같은 allow 규칙을 넣어도 이 표의 결과가 바뀌지 않는다는 것 — 안전 검사가 allow 규칙 평가보다 먼저 실행되기 때문이다. bypassPermissions에서도 rm -rf /, rm -rf ~는 모델 오류에 대비한 circuit breaker로 여전히 프롬프트한다.

Hooks와 권한의 상호작용 — 누가 누구를 이기는가

PreToolUse 훅과 permission 규칙의 우선순위

PreToolUse 훅은 permission 프롬프트보다 먼저 실행된다. 하지만 훅이 "allow"나 "ask"를 반환해도, 매칭되는 permission deny/ask 규칙은 여전히 강제된다 — 훅의 판정이 규칙을 무시하게 만들지 못한다. 반대로 훅이 exit code 2로 차단하면, 그 차단은 permission 규칙 평가 자체보다도 먼저 적용된다(즉 훅의 차단은 allow 규칙보다 우선한다). 이 비대칭 덕분에 "Bash 전체를 allow해 두고, 위험한 특정 명령만 훅으로 골라 차단"하는 패턴이 가능해진다.

Settings Precedence — 5단계와 Managed-only 설정

규칙이 저장되는 설정 파일에는 우선순위가 있고, 이 순서는 permission 규칙에도 동일하게 적용된다.

순위레벨비고
1(최상위)Managed settings다른 어떤 레벨도, CLI 인자로도 못 넘음
2Command line arguments세션 한정 임시 override
3Local project (.claude/settings.local.json)
4Shared project (.claude/settings.json)버전 관리 대상
5User (~/.claude/settings.json)

어느 레벨에서든 deny되면 다른 레벨이 allow할 수 없다 — managed deny는 --allowedTools로도 못 뒤집고, 반대로 사용자 레벨 deny도 프로젝트 레벨 allow를 이긴다(deny는 항상 먼저 평가되기 때문). 조직이 "개발자가 절대 못 바꾸게" 강제하고 싶은 설정은 managed-only 설정으로 존재한다: allowManagedPermissionRulesOnly(사용자/프로젝트가 권한 규칙 자체를 정의 못 하게), disableBypassPermissionsMode, disableAutoMode, strictPluginOnlyCustomization, strictKnownMarketplaces 등은 managed settings가 아닌 곳에 적으면 아무 효과가 없다.

Workspace Trust — "확장은 신중하게, 축소는 즉시"

프로젝트 .claude/settings.json의 permissions.allow와 additionalDirectories는 워크스페이스 신뢰 다이얼로그를 수락하기 전까지 읽히기만 하고 적용되지 않는다. 반면 deny/ask 규칙은 신뢰 여부와 무관하게 즉시 적용된다 — 저장소가 스스로 권한을 넓히는 것은 사용자 확인이 필요하지만, 좁히는 것은 확인 없이 바로 걸리는 비대칭 설계다. -p(non-interactive) 모드에서는 신뢰 다이얼로그 자체가 뜨지 않고 신뢰 안 된 allow 규칙은 그대로 무시된 채 진행된다 — CI 파이프라인에서 프로젝트의 allow 규칙이 조용히 무시되는 지점이므로, CI에는 managed settings나 ~/.claude/settings.json(신뢰 절차 불필요)에 규칙을 두는 것이 안전하다.

Auto Mode — 세 단계 결정 순서와 classifier

auto 모드(아직 research preview)는 액션마다 다음 순서로 결정한다: ① allow/deny 규칙(protected path 쓰기는 예외) ② 작업 디렉토리 내 읽기/편집 자동 승인(protected path 제외) ③ 그 외 전부는 별도 classifier 모델이 검증. 대화 중 사용자가 말로 정한 경계("push하지 마")는 classifier가 소프트 차단 신호로 받아들이지만 규칙으로 저장되지 않으므로 context compaction으로 사라질 수 있다 — 하드 보장이 필요하면 반드시 deny 규칙을 쓴다. classifier가 연속 3회 또는 총 20회 차단하면 auto mode가 일시 정지되고 프롬프트 모드로 복귀하며, 이 임계값은 설정으로 바꿀 수 없다. 서브에이전트는 스폰 전(작업 설명 자체를 평가) → 실행 중(각 액션을 부모와 동일 규칙으로 검사) → 종료 후(전체 히스토리 재검토) 3단계로 검사된다.

Sandbox와의 layered defense — 권한이 필요 없어지는 것이 아니다

샌드박스(Day 5에서 심화)가 켜져 있고 autoAllowBashIfSandboxed(기본값 true)가 활성화되어 있으면, 샌드박스된 Bash 명령은 bare Bash/Bash(*) ask 규칙이 있어도 프롬프트 없이 실행된다 — 샌드박스 경계 자체가 그 프롬프트를 대신하는 것이다. 하지만 이것이 "샌드박스가 권한을 대체한다"는 뜻은 아니다: Bash(git push *) 같은 content-scoped ask 규칙, 명시적 deny 규칙, 루트/홈 대상 rm은 샌드박스 여부와 무관하게 여전히 프롬프트한다. 파일시스템 경계는 sandbox.filesystem 설정과 Read/Edit deny 규칙이 병합되고, 네트워크 경계는 WebFetch 권한 규칙과 샌드박스의 allowedDomains/deniedDomains가 병합된다 — 두 계층이 서로를 대체하는 게 아니라 합쳐져서 최종 경계를 만든다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • "규칙 문법을 외웠는가"가 아니라 "규칙이 실제로 안전한지 검증할 수 있는가": curl 필터링 함정처럼, 그럴듯해 보이는 규칙이 실제로는 우회 가능하다는 것을 짚어낼 수 있는지를 본다.
  • deny-first 설계 원칙의 함의를 이해하는가: "구체성이 이긴다"고 잘못 답하면 프로덕션에서 "예외를 뒀다고 생각했는데 실제로는 전체가 막혔다/뚫렸다"는 사고로 이어질 수 있는 위험 신호다.
  • 계층 간 상호작용을 정확히 아는가: 훅↔권한, 샌드박스↔권한, managed↔user 설정처럼 여러 통제 계층이 겹칠 때 무엇이 우선하는지 예측할 수 있어야 조직 보안 정책을 안전하게 설계할 수 있다.
  • 조직 거버넌스 감각: managed-only 설정과 workspace trust의 비대칭 설계("확장은 신중, 축소는 즉시")를 이해하는 것은 실제로 여러 팀에 권한 정책을 배포해 본 경험을 드러낸다.

이 질문은 "어떤 모드가 있는지"를 묻지 않는다. "여러 통제 계층이 겹칠 때 최종 결과를 정확히 예측하고, 겉보기엔 안전해 보이지만 실제로는 우회 가능한 규칙을 식별할 수 있는가"를 묻는다.

4. 실무 연결

실제로 이 지식이 필요한 순간들

시나리오 1 — curl deny 규칙이 뚫린 사례

한 팀이 "Claude가 외부 네트워크에 함부로 접근하지 못하게 하자"며 Bash(curl http://github.com/ *)류의 allow-only 규칙과 Bash(curl *) deny를 섞어 썼다. 처음엔 잘 작동하는 듯했지만, 어느 세션에서 Claude가 우연히 읽은 스크립트 안의 리다이렉트 URL(bit.ly 단축 링크)을 통해 의도치 않은 외부 호스트로 데이터를 전송하는 명령을 실행할 뻔했다.

사후 분석에서 팀은 Bash 기반 URL 필터링이 프로토콜 변경·리다이렉트·변수 치환 앞에서 근본적으로 취약하다는 것을 확인했다. 해결책은 Bash에서 curl/wget류를 전부 deny하고, 허용해야 하는 도메인은 WebFetch(domain:...) allow 규칙으로 옮기는 것이었다 — WebFetch 도구 자체가 도메인을 구조적으로 검증하기 때문에 Bash 명령줄 패턴 매칭보다 훨씬 견고했다.

시나리오 2 — "구체적인 allow 규칙이 예외를 만들어 줄 것"이라는 착각

한 팀이 Bash(aws *)를 deny하면서도 "읽기 전용 조회 명령 정도는 예외로 허용하고 싶다"며 Bash(aws s3 ls), Bash(aws sts get-caller-identity) 같은 allow 규칙을 함께 추가했다. 실제 세션에서 이 명령들이 전부 차단되자 팀은 "권한 시스템이 버그가 있다"고 오해했다.

원인은 버그가 아니라 deny → ask → allow 순서상 deny가 항상 먼저 평가되고, 매치되면 그 자리에서 확정되기 때문이었다. 팀은 규칙 설계를 "넓게 막고 구체적으로 예외를 둔다"에서 "필요한 것만 좁게 allow하고 deny는 정말 절대 허용 불가능한 것에만 쓴다"로 바꿨다. 이 사례는 오히려 이 설계가 팀에 유리하다는 것도 보여준다 — 나중에 실수로 넓은 allow 규칙이 추가되어도, 기존 deny 규칙이 절대 원칙을 계속 지켜준다는 뜻이기 때문이다.

시나리오 3 — CI에서 프로젝트 allow 규칙이 조용히 무시된 사례

한 팀이 저장소 .claude/settings.json에 CI 전용 allow 규칙(특정 배포 스크립트 실행 허용)을 커밋해 두고, GitHub Actions에서 claude -p로 파이프라인을 실행했다. 그런데 실제 실행에서는 그 규칙이 적용되지 않는 것처럼 동작했고, 팀은 "설정 파일을 잘못 커밋했나" 하며 한참을 디버깅했다.

원인은 프로젝트 allow 규칙이 워크스페이스 신뢰(workspace trust)를 통과해야 적용되는데, -p non-interactive 모드에서는 신뢰 다이얼로그 자체가 뜨지 않아 규칙이 계속 "미신뢰" 상태로 무시되고 있었기 때문이었다. 해결책은 CI 전용 규칙을 신뢰 절차가 필요 없는 ~/.claude/settings.json(CI 러너의 홈 디렉토리) 또는 managed settings로 옮기는 것이었다. 이 사례는 "설정 파일에 규칙을 적으면 무조건 즉시 적용된다"는 가정이 왜 위험한지 보여준다.

5. 답변 사고 순서

면접에서 "권한 모드/규칙을 어떻게 설계하는가"류의 질문을 받으면, 아래 순서로 답을 구성하라.

  1. 모드(baseline)와 규칙(rule layer)을 구분한다

    6개 모드가 "얼마나 자주 멈추는가"의 기준선이고, allow/ask/deny 규칙이 그 위에서 특정 도구 호출을 세밀하게 통제한다는 계층 구조를 먼저 밝힌다.

  2. 평가 순서를 정확히 설명한다

    deny → ask → allow 순서와 "구체성이 순서를 바꾸지 않는다"는 원칙을 구체적 예시(Bash(aws *) deny가 Bash(aws s3 ls) allow를 이긴다)로 보여준다.

  3. Tool별 문법의 함정을 짚는다

    특히 Bash 기반 URL/명령 필터링이 구조적으로 취약하다는 것과, WebFetch/전용 도구 규칙 또는 훅으로 보완해야 한다는 것을 예시로 든다.

  4. 계층 간 상호작용으로 확장한다

    훅↔권한(훅의 allow가 deny 규칙을 못 이김, 훅의 차단은 allow보다 우선), 샌드박스↔권한(대체가 아니라 병합), settings precedence(managed가 최상위)를 짚는다.

  5. 조직 운영 관점(workspace trust, managed-only 설정)으로 마무리한다

    "확장은 신중, 축소는 즉시"라는 비대칭 설계와, CI/조직 환경에서 규칙을 어디에 둬야 실제로 적용되는지로 답을 맺는다.

핵심 팁

규칙 문법을 나열하는 데서 멈추지 말고, 반드시 "이 규칙이 실제로 우회 불가능한가"를 스스로 검증하는 태도를 보여줘라. 인터뷰어는 문법 암기보다 "설계한 규칙을 공격자 관점에서 재검토하는 습관"에 더 높은 점수를 준다.

6. 핵심 답변 예시

면접 답변 예시 — "npm은 자유롭게, git push는 절대 금지"를 어떻게 설계합니까?

"먼저 모드와 규칙을 분리해서 생각합니다. 이 프로젝트가 반복적인 코드 수정이 많다면 acceptEdits 모드를 기준선으로 두어 파일 편집과 안전한 파일시스템 명령은 자동 승인되게 하고, 그 위에 규칙을 얹습니다. npm은 allow: [\"Bash(npm *)\"]로 자유롭게 허용하고, git push는 deny: [\"Bash(git push *)\"]로 명시적으로 막습니다.

이 설계가 안전한 이유는 평가 순서 때문입니다. deny가 먼저 평가되고, 규칙의 구체성은 순서에 영향을 주지 않기 때문에, 나중에 누군가 실수로 Bash(git push origin main) 같은 더 구체적인 allow 규칙을 추가해도 git push 전체를 막는 deny가 여전히 이깁니다. 다만 이것만으로 끝내지는 않습니다. Bash 기반 패턴 매칭은 &&로 연결된 compound command나 환경 러너(devbox run git push 같은)를 통해 우회될 수 있기 때문에, 정말 절대 금지가 필요하다면 PreToolUse 훅으로 git push가 포함된 명령을 한 번 더 검증하도록 이중화합니다. 훅의 차단은 permission 규칙 평가보다도 먼저 적용되기 때문에 가장 강력한 마지막 방어선이 됩니다.

조직 차원에서는 이 규칙들이 개발자 개인 설정으로 덮어써지면 안 되므로 managed settings에 두거나 allowManagedPermissionRulesOnly를 켜서 사용자/프로젝트 레벨에서 아예 권한 규칙을 정의하지 못하게 잠급니다. CI 파이프라인에서 같은 규칙을 쓴다면, 프로젝트 .claude/settings.json의 allow 규칙은 워크스페이스 신뢰를 통과해야 적용되는데 -p 모드에서는 신뢰 다이얼로그가 뜨지 않는다는 점을 고려해서, CI 전용 규칙은 managed settings나 CI 러너의 ~/.claude/settings.json에 둡니다."

이 답변이 좋은 이유: ① 모드(baseline)와 규칙(rule layer)을 명확히 분리 ② deny-first·구체성 무관 원칙을 구체적 위험 시나리오로 검증 ③ Bash 패턴 매칭의 구조적 한계를 알고 훅으로 이중화 ④ 훅↔권한 우선순위(차단은 우선, allow는 무효)를 정확히 적용 ⑤ 조직 거버넌스(managed-only, workspace trust)까지 연결해 실무 배포 경험을 드러냄

7. Senior-level Follow-up Q&A

Q1. "deny 규칙이 구체적인 allow 규칙보다 왜 항상 이깁니까? 이 설계의 장단점은 무엇입니까?"

평가 자체가 deny → ask → allow 순서로 고정되어 있고, 첫 매치가 결과를 확정하기 때문입니다. 규칙의 "구체성"은 이 순서에 관여하지 않습니다 — 만약 구체성이 순서를 바꿀 수 있다면, "가장 구체적인 규칙이 무엇인가"를 사람이 매번 다시 계산해야 하고, 그 계산이 팀마다 다르게 이해될 위험이 커집니다.

장점은 "안전 규칙(deny)에는 절대 예외가 뚫리지 않는다"는 강한 보장을 준다는 것입니다. 단점은 "이 특정 케이스만 예외로 허용하고 싶다"는 요구를 deny+allow 조합으로 표현할 수 없다는 것 — 이런 경우엔 deny 규칙 자체를 원하는 예외만큼 좁게 다시 써야 합니다(예: 전체 aws를 막는 대신 위험한 하위 명령만 개별적으로 deny).

Q2. "PreToolUse 훅에서 allow를 반환했는데 permission ask 규칙이 있으면 어떻게 됩니까?"

훅의 판정과 무관하게 매칭되는 ask 규칙은 여전히 프롬프트를 띄웁니다. 훅의 역할은 "permission 규칙보다 먼저 개입해 차단하거나, 규칙이 프롬프트를 요구하지 않는 경우에 한해 통과를 돕는 것"이지, 규칙 자체를 무효화하는 게 아닙니다.

반대로 훅이 exit code 2로 차단하면 그 차단은 permission 규칙 평가보다 먼저 적용되어, 심지어 allow 규칙이 매치하더라도 그 호출을 막습니다. 이 비대칭 때문에 "훅은 추가 차단을 위한 도구이지 추가 허용을 위한 도구가 아니다"라고 이해하는 것이 정확합니다.

Q3. "샌드박스를 켜면 permission 규칙이 필요 없어집니까?"

아닙니다. autoAllowBashIfSandboxed(기본 켜짐)는 bare Bash ask 규칙만 프롬프트 없이 지나가게 하는 것이지, content-scoped ask 규칙(Bash(git push *) 등)이나 명시적 deny 규칙까지 무력화하지 않습니다. 또한 샌드박스는 Bash 도구와 그 자식 프로세스에만 적용되는 OS 레벨 격리이고, permission은 Read/Edit/WebFetch/MCP/Agent 등 모든 도구에 적용됩니다 — 적용 범위 자체가 다릅니다.

실무적으로는 "샌드박스가 있으니 권한 규칙은 대충 둬도 된다"가 아니라, 파일시스템/네트워크 경계는 두 계층의 설정이 병합되어 최종 경계가 만들어진다고 이해하는 것이 맞습니다. 하나가 느슨해도 다른 하나가 막아주는 defense-in-depth 관계입니다.

Q4. "워크스페이스 신뢰(workspace trust)가 CI 파이프라인(-p 모드)에서는 어떻게 동작합니까?"

-p non-interactive 모드에서는 신뢰 다이얼로그 자체가 뜨지 않습니다. 그래서 아직 신뢰되지 않은 프로젝트의 permissions.allow나 additionalDirectories는 다이얼로그 없이 계속 무시된 채로 세션이 진행됩니다 — 에러가 나거나 멈추는 게 아니라 조용히 규칙이 적용 안 된 상태로 흘러가기 때문에 디버깅이 까다롭습니다.

그래서 CI에서 재현 가능하게 규칙을 걸려면, 신뢰 절차가 필요 없는 위치(CI 러너 홈의 ~/.claude/settings.json) 또는 관리형 설정(managed settings)에 규칙을 두는 것이 안전합니다. deny/ask 규칙은 애초에 신뢰 여부와 무관하게 항상 적용되므로 이 문제에서 자유롭습니다.

Q5. "auto mode의 classifier가 반복적으로 정상 작업을 차단합니다. 어떻게 대응합니까?"

연속 3회 또는 총 20회 차단되면 auto mode 자체가 일시 정지되고 프롬프트 모드로 돌아오므로, 우선 그 지점에서 승인해 재개할 수 있습니다. 다만 반복적인 차단은 대개 classifier가 조직의 인프라 맥락(어떤 리포지토리·클라우드 리소스가 "신뢰 대상"인지)을 모르기 때문에 발생하므로, 관리자가 신뢰 인프라 목록을 설정하는 것이 근본적인 해결책입니다.

더 강한 보장이 필요한 개별 액션이라면 auto mode의 소프트한 대화 맥락 차단에 의존하지 말고, 명시적 deny 규칙을 추가하는 것이 안전합니다 — 대화 중에 말로 정한 경계는 classifier가 인식은 하지만 규칙으로 저장되지 않아 context compaction 이후 사라질 수 있기 때문입니다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 모드(baseline)와 규칙(rule layer)을 계층으로 명확히 분리한다
  • deny → ask → allow 순서와 "구체성 무관" 원칙을 구체적 예시로 검증한다
  • Bash 기반 패턴 매칭의 구조적 취약점(compound command, 환경 러너, curl 필터링)을 알고 WebFetch/훅으로 보완한다
  • 훅↔권한, 샌드박스↔권한의 상호작용을 정확한 우선순위로 설명한다
  • workspace trust, managed-only 설정 같은 조직 거버넌스 메커니즘까지 연결한다
약한 답변
  • "규칙을 구체적으로 쓰면 우선한다"고 잘못 답한다
  • Bash(curl http://domain/ *) 같은 규칙이 안전하다고 믿는다
  • "샌드박스를 켜면 권한 설정이 필요 없다"고 두 계층을 혼동한다
  • 훅이 permission 규칙을 완전히 대체하거나 무효화할 수 있다고 답한다
  • 조직 배포 질문에 "설정 파일에 적으면 된다"로 답해 workspace trust/managed settings를 언급하지 못한다
왜 약한 답변이 약한가

permission mode/규칙이 존재한다는 것 자체는 누구나 안다. 시니어 면접이 확인하려는 건 "내가 설계한 규칙이 실제로 우회 불가능한지 스스로 검증할 수 있는가", 그리고 "여러 통제 계층이 겹칠 때 최종 판정을 정확히 예측할 수 있는가"다. 특히 curl 필터링 같은 함정을 안전하다고 믿는 답변은 실제 보안 사고로 직결될 수 있는 위험한 착각이라 감점 폭이 크다.

9. 흔한 오해

오해 1

"규칙이 더 구체적이면 더 넓은 규칙보다 우선한다."

실제

평가 순서는 항상 deny → ask → allow이고 구체성은 이 순서에 영향을 주지 않는다. 넓은 deny가 더 구체적인 allow보다 먼저 매치되면 그 시점에서 차단이 확정된다.

오해 2

"Bash(curl http://github.com/ *) 같은 규칙으로 curl을 특정 도메인에만 묶을 수 있다."

실제

옵션 순서, 프로토콜, 리다이렉트, 변수 치환으로 쉽게 우회된다. 도메인 제한이 필요하면 Bash의 curl/wget류를 deny하고 WebFetch(domain:...)로 대체하거나 PreToolUse 훅으로 검증해야 한다.

오해 3

"bypassPermissions는 정말 아무 것도 막지 않는다."

실제

명시적 ask 규칙은 이 모드에서도 여전히 프롬프트하고, 루트/홈 삭제(rm -rf /, rm -rf ~)는 모델 오류 대비 circuit breaker로 항상 프롬프트한다. root/sudo로 시작하면 이 모드 진입 자체가 거부된다(샌드박스 내부는 예외).

오해 4

"샌드박스를 켜면 permission 규칙은 신경 쓰지 않아도 된다."

실제

샌드박스는 Bash 도구·자식 프로세스에만 적용되는 별개의 OS 레벨 계층이다. autoAllowBashIfSandboxed는 bare ask 규칙만 생략시킬 뿐, content-scoped ask/deny 규칙은 그대로 강제되며, 파일시스템/네트워크 경계는 두 계층 설정이 병합되어 최종 결정된다.

오해 5

"PreToolUse 훅이 allow를 반환하면 permission 규칙과 무관하게 실행된다."

실제

훅의 allow/ask 판정은 매칭되는 permission deny/ask 규칙을 무효화하지 못한다. 반대로 훅의 차단(exit code 2)은 permission 규칙 평가 자체보다 먼저 적용되어 allow 규칙까지 이긴다 — 훅은 "추가 차단" 도구이지 "추가 허용" 도구가 아니다.

오해 6

"프로젝트 .claude/settings.json에 allow 규칙을 커밋하면 CI에서도 곧바로 적용된다."

실제

프로젝트 allow 규칙과 additionalDirectories는 워크스페이스 신뢰를 통과해야 적용되고, -p non-interactive 모드에서는 신뢰 다이얼로그가 뜨지 않아 계속 무시된 채로 진행된다. CI에는 managed settings나 러너의 ~/.claude/settings.json을 쓰는 것이 안전하다.
