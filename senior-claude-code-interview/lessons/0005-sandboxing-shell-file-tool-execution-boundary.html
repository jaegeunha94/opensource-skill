2. 전제 개념 — 쉬운 한국어로

왜 권한 모델(Day 4)과 별개로 샌드박스를 다뤄야 하는가

Day 4에서 permission mode와 allow/ask/deny 규칙을 "누가 이 행동을 승인할지 결정하는 문지기 층"으로 배웠다. 이번 Day가 다루는 샌드박스는 그 문지기를 통과한 이후에 작동하는 완전히 다른 층이다 — "승인받은 명령이 실제로 어디까지 손댈 수 있는가"를 운영체제가 직접 강제한다. 이 둘을 같은 것으로 혼동하면 "권한 규칙만 잘 짜면 안전하다"는 잘못된 설계로 이어지기 쉽다. 시니어 면접에서 이 구분을 중요하게 보는 이유는, 실제 프로덕션 사고의 상당수가 "문지기는 통과했지만 그 다음 층이 없어서" 벌어지기 때문이다 — 예를 들어 프롬프트 인젝션으로 위험한 명령이 "안전한 것처럼" 포장되어 문지기를 속이는 데 성공해도, 울타리(샌드박스)가 있으면 실제 피해 범위가 좁아진다.

가장 먼저 짚어야 할 것 — 샌드박스는 Bash만 격리한다 ("Shell/File 도구 실행 경계")

Scope — 무엇이 격리되고 무엇이 격리되지 않는가
구성 요소격리 여부실제로 통제하는 계층
Bash 도구와 그 자식 프로세스OS 레벨로 격리됨샌드박스(Seatbelt/bubblewrap) + permission 규칙(병합)
Read / Edit / Write격리 안 됨 — Claude Code 프로세스 안에서 직접 실행permission 규칙(Read/Edit allow·deny)만
WebFetch격리 안 됨 — 임의 코드를 스폰하지 않음permission 규칙(domain allow·deny)만
MCP 서버격리 안 됨 — host에서 별도 프로세스로 무제한 실행permission 규칙(mcp__서버 allow·deny)만
Hooks격리 안 됨 — host에서 별도 프로세스로 무제한 실행훅 스크립트 자체의 신뢰성에 의존
Subagent 내부 Bash부모와 같은 프로세스, 부모의 샌드박스 설정 그대로 적용부모와 동일

이 표가 "shell/file 도구 실행 경계"의 핵심이다. Claude Code에 내장된 샌드박스(sandboxed Bash tool)는 Bash 명령과 그 자식 프로세스만 OS 레벨로 제한한다. 파일 조작은 Read/Edit/Write라는 별도 내장 도구로 이뤄지며, 이들은 애초에 임의 코드를 스폰하지 않으므로 샌드박스가 아니라 permission 규칙(경로 allow/deny)만으로 통제된다. MCP 서버와 Hooks는 host에서 직접 실행되는 별도 프로세스라서 이 샌드박스의 보호 범위 밖에 있다 — 이것이 "샌드박스를 켜면 다 안전해진다"는 흔한 착각의 근본 원인이다.

OS 레벨 강제 — 두 개의 서로 다른 커널 프리미티브

플랫폼구현비고
macOSSeatbelt(내장 샌드박스 프레임워크)별도 설치 불필요
Linuxbubblewrap(파일시스템) + socat(네트워크 프록시 릴레이)패키지 설치 필요(apt/dnf install bubblewrap socat)
WSL2Linux와 동일(bubblewrap)WSL1은 커널 기능 부족으로 미지원
네이티브 Windows미지원WSL2 안에서 실행해야 함

Linux에서는 seccomp 필터(Unix domain socket 차단용, 선택 사항이며 @anthropic-ai/sandbox-runtime 설치로 추가)와 Ubuntu 24.04+의 AppArmor가 bwrap의 user namespace 생성을 막는 문제(별도 AppArmor 프로필 필요) 같은 배포 환경별 세부사항이 있다 — "그냥 켜면 된다"가 아니라 플랫폼별 의존성 확인이 필요한 기능이라는 점이 실무에서 자주 놓치는 지점이다.

Sandbox Mode — auto-allow에서도 여전히 살아있는 예외들

/sandbox 패널에서 두 모드 중 하나를 고른다.

모드동작
Auto-allow샌드박스 안에서 실행 가능한 Bash 명령은 프롬프트 없이 자동 승인. 샌드박스로 실행 불가능한 명령(허용 안 된 도메인 접근 등)은 일반 permission 흐름으로 폴백
Regular permissions샌드박스 여부와 무관하게 모든 Bash 명령이 일반 permission 흐름을 거침
Auto-allow에서도 여전히 적용되는 것
  • 명시적 deny 규칙은 항상 존중된다
  • /, 홈 디렉토리, 기타 critical path를 대상으로 하는 rm/rmdir은 여전히 프롬프트한다
  • Bash(git push *) 같은 content-scoped ask 규칙은 샌드박스된 명령에도 프롬프트를 강제한다
  • bare Bash/Bash(*) ask 규칙만 "샌드박스된 명령에 한해" 생략된다 — 일반 permission 흐름으로 폴백한 명령에는 여전히 적용된다

즉 auto-allow는 "샌드박스 경계 자체가 그 프롬프트를 대신한다"는 논리이지, "무제한 실행을 허용한다"는 뜻이 아니다. 또한 이 auto-allow는 permission mode 목록(Day 4의 auto mode)과 이름은 비슷하지만 완전히 다른 메커니즘이다 — auto-allow는 OS 경계가 프롬프트를 대신하는 것이고, auto mode는 별도 classifier 모델이 액션을 검토하는 것이다. 둘은 독립적으로 동작하며 함께 켤 수 있다.

파일시스템 격리 — 기본값의 함정

기본 정책, 그리고 흔히 놓치는 지점
  • 쓰기(write): 현재 작업 디렉토리 + 세션 임시 디렉토리($TMPDIR)로만 제한
  • 읽기(read): 컴퓨터 전체를 기본적으로 읽을 수 있다(일부 차단된 디렉토리 제외) — 여기에는 ~/.aws/credentials, ~/.ssh/ 같은 credential 파일도 기본적으로 포함된다. 별도로 sandbox.credentials나 denyRead를 설정하지 않으면 읽힌다.

이 read 기본 정책은 "샌드박스가 곧 credential 보호"라고 오해하기 쉬운 지점이다. 샌드박스의 기본 목적은 "쓰기와 네트워크 확산"을 막는 것이지, "민감 파일 읽기"를 기본으로 막는 것이 아니다.

sandbox.filesystem.allowWrite(작업 디렉토리 밖 쓰기 허용, 예: kubectl/terraform이 ~/.kube에 써야 할 때), denyWrite/denyRead(특정 경로 차단), allowRead(denyRead로 막힌 영역 내에서 특정 경로만 재허용)로 세밀하게 조정한다. 이 값들은 여러 설정 스코프에 걸쳐 있으면 병합(merge)된다(하나가 다른 것을 덮어쓰지 않음).

경로 프리픽스 — Day 4의 Read/Edit 규칙 문법과 다르다
프리픽스의미
/path파일시스템 루트 기준 절대경로(예: /tmp/build)
~/path홈 디렉토리 기준
./path 또는 프리픽스 없음프로젝트 설정이면 프로젝트 루트, 사용자 설정이면 ~/.claude 기준

Day 4에서 배운 permission Read/Edit 규칙은 //path가 절대경로, /path가 settings 소스 기준 앵커였다. 샌드박스 sandbox.filesystem.* 설정은 표준 관례를 따라 /path 자체가 절대경로다 — 슬래시 개수가 하나냐 둘이냐로 의미가 완전히 달라지는 두 문법을 혼동하면 의도와 정반대의 경로를 여는 실수를 하기 쉽다.

Credential 보호 — sandbox.credentials: deny vs mask

기본 read 정책이 credential 파일을 막지 않으므로, 실무에서는 sandbox.credentials로 명시적으로 보호해야 한다. 두 가지 모드가 있다(v2.1.187+에서 도입).

모드동작트레이드오프
deny파일 read 차단(denyRead와 동일 효과), 환경변수는 명령 실행 전 unset완전 차단이지만 그 credential로 인증해야 하는 도구(gh, npm 등)가 깨진다
mask(환경변수 전용, v2.1.199+)샌드박스 안에서는 세션별 sentinel 값만 보이고, 요청이 injectHosts로 나갈 때 샌드박스 프록시가 실제 값으로 치환도구는 정상 인증하지만, 실제 값이 프록시가 TLS를 종료(network.tlsTerminate)해야만 치환 가능 — 없으면 fail-closed(인증 실패)

mask는 "명령과 그 로그는 실제 credential을 절대 들고 있지 않지만, 인증은 정상 동작해야 한다"는 요구를 푸는 정교한 기능이다. 단 network.tlsTerminate 설정 없이 mask만 켜면 프록시가 암호화된 요청 내용을 볼 수 없어 sentinel이 그대로 서버에 도달해 인증이 조용히 실패한다 — Claude Code는 이 오설정을 시작 시점과 /doctor에서 경고한다. 또한 mask·tlsTerminate·credentials.allowPlaintextInject는 저장소에 커밋되는 .claude/settings.json/settings.local.json에서는 무시되고 사용자/managed 설정, --settings CLI 플래그에서만 유효하다 — 저장소를 체크아웃한 누구나 실제 credential을 프록시로 흘려보내게 만드는 규칙을 커밋할 수 없도록 막는 설계다.

네트워크 격리 — 프록시, 그리고 "내용은 보지 않는다"는 한계

네트워크는 샌드박스 바깥에서 실행되는 프록시 서버가 통제한다. 기본적으로 사전 허용된 도메인은 없고, 명령이 새 도메인이 필요할 때마다 프롬프트한다(v2.1.191부터는 한 번 승인하면 그 세션 동안 같은 호스트는 다시 묻지 않는다). sandbox.network.allowedDomains/deniedDomains로 사전 허용/차단 목록을 정의하고, allowManagedDomainsOnly(managed 설정 전용)를 켜면 관리자가 정의한 도메인 외에는 자동 차단된다.

가장 중요한 보안 한계 — 기본적으로 TLS를 검사하지 않는다

프록시는 요청 hostname 기준으로 allow 여부를 판단할 뿐, 기본적으로 HTTPS 트래픽을 종료(terminate)하거나 내용을 검사하지 않는다. 즉 github.com처럼 넓은 도메인을 허용하면, 그 안에서 오가는 실제 데이터는 검사되지 않는다. 클라이언트가 제시하는 hostname만으로 판단하기 때문에 domain fronting 같은 기법으로 허용 목록 밖의 호스트에 실질적으로 도달할 가능성도 남는다. network.tlsTerminate(실험적, v2.1.199+)는 mask credential 치환을 위해 프록시가 TLS를 스스로 종료하게 하지만, 이것이 콘텐츠 필터링을 추가하는 것은 아니다. 진짜 강한 보장이 필요하면 자체 CA를 심는 custom proxy(httpProxyPort/socksProxyPort)로 트래픽을 검사해야 한다.

탈출구(Escape Hatch)와 Strict Mode

샌드박스 안에서 실행 불가능한 명령(예: 도구 자체가 격리와 호환되지 않거나, 허용되지 않은 호스트가 필요한 경우)은 실패로 끝나는 대신 Claude가 실패를 분석해 dangerouslyDisableSandbox 파라미터로 재시도할 수 있다 — 이 재시도는 샌드박스 밖에서 실행되므로 일반 permission 흐름(승인 필요)을 그대로 거친다. allowUnsandboxedCommands: false(/sandbox 패널의 Strict sandbox mode)로 이 탈출구 자체를 봉쇄하면, 모든 명령은 반드시 샌드박스 안에서 실행되거나 excludedCommands에 명시적으로 등록된 것만 예외로 통과한다.

격리 스펙트럼 — 샌드박스는 여러 옵션 중 하나일 뿐이다

내장 샌드박스(Bash만 격리)는 "MCP 서버와 Hooks까지 같은 OS 경계 안에 두고 싶다"는 요구를 채우지 못한다 — 이때는 sandbox-runtime으로 Claude Code 프로세스 전체를 감싸거나, devcontainer/custom container/VM으로 확장해야 한다. 특히 --dangerously-skip-permissions(bypassPermissions)는 명시적 ask 규칙과 protected path circuit breaker를 제외한 모든 사전 검토를 제거하므로, 반드시 컨테이너·VM·sandbox-runtime 같은 전체 프로세스 격리 안에서만 사용해야 한다 — 내장 Bash 샌드박스만으로는 file 도구/MCP/훅이 여전히 무방비이기 때문이다. 반면 auto mode(Day 4의 classifier)는 프롬프트를 대체하는 별도 판단 계층이라 완전한 review 제거가 아니므로, 격리 강도 요구가 상대적으로 낮다.

조직 강제 — Managed Settings로 잠그기

sandbox.enabled, failIfUnavailable(의존성 없으면 warning 대신 시작 자체를 차단), allowUnsandboxedCommands: false를 managed settings로 배포하면 조직 전체에 강제할 수 있다. boolean 키(enabled, failIfUnavailable)는 managed 값이 항상 이기지만, excludedCommands나 allowRead 같은 배열 키는 스코프 간 병합되므로 개발자가 항목을 추가해 정책을 넓힐 수 있다. 이를 막으려면 allowManagedReadPathsOnly(managed의 allowRead만 유효화), allowManagedDomainsOnly(도메인도 동일)를 쓴다. 단 excludedCommands는 이런 managed-only 잠금 장치가 없어 개발자가 항상 항목을 추가할 수 있다는 한계가 남는다 — managed 목록을 최소한으로 유지하는 것이 실무 권장이다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • 권한과 샌드박스를 같은 것으로 혼동하지 않는가: "샌드박스 켰으니 안전하다"는 답은 두 계층의 역할을 이해하지 못했다는 신호다.
  • scope 경계를 정확히 아는가: Bash만 격리되고 Read/Edit/WebFetch/MCP/Hooks는 host에서 직접 실행된다는 것을 알아야, 완전 자율 실행 시나리오에서 무엇을 추가로 격리해야 하는지 설계할 수 있다.
  • "기본값 = 안전"이라는 가정을 검증하는 태도: 기본 read 정책이 credential 파일을 막지 않는다는 것처럼, 그럴듯해 보이는 기본값의 실제 동작을 확인하는 습관이 있는가.
  • 네트워크 격리의 한계를 이해하고 threat model에 맞게 강도를 고르는가: TLS 미검사·domain fronting 리스크를 알고, 필요하면 custom proxy까지 고려할 수 있는가.
  • 조직 enforcement 설계 감각: managed-only 잠금과 그 한계(excludedCommands는 항상 넓힐 수 있음)까지 아는 것은 실제로 여러 팀에 격리 정책을 배포해 본 경험을 드러낸다.

이 질문은 "샌드박스 기능이 있다는 것"을 묻지 않는다. "권한 승인과 실행 후 접근 제한이라는 서로 다른 두 계층을 구분하고, 그 격리 경계가 정확히 어디까지인지 알며, 남은 위험(TLS 미검사, credential 기본 read 허용)에 대한 대응책까지 설계할 수 있는가"를 묻는다.

4. 실무 연결

실제로 이 지식이 필요한 순간들

시나리오 1 — "샌드박스를 켰으니 완전 자율 실행도 안전하다"는 착각

한 팀이 야간 배치 작업에 --dangerously-skip-permissions와 내장 Bash 샌드박스(sandbox.enabled: true)를 함께 켜고 "이제 안전하게 무인 실행이 가능하다"고 판단했다. 그런데 세션에 연결된 MCP 서버 하나가 host에서 직접 실행되며 프로젝트 밖의 설정 파일을 읽어 외부로 전송하는 동작을 했고, 이는 Bash 샌드박스의 보호 범위 밖이라 전혀 차단되지 않았다.

사후 분석에서 팀은 "샌드박스는 Bash와 그 자식 프로세스만 격리하고, MCP 서버·Hooks·Read/Edit는 host에서 직접 실행된다"는 scope 경계를 놓쳤다는 것을 확인했다. 해결책은 Claude Code 프로세스 전체를 sandbox-runtime이나 컨테이너로 한 번 더 감싸, MCP/훅까지 같은 OS 경계 안에 두는 것이었다. 이 사례는 "--dangerously-skip-permissions는 반드시 전체 프로세스 격리 안에서만 쓴다"는 원칙이 왜 필요한지 보여준다.

시나리오 2 — credential mask가 조용히 fail-closed된 사례

한 팀이 NPM_TOKEN을 sandbox.credentials의 mask 모드로 보호하려 했다. 코드는 정상 실행되는 것처럼 보였지만, npm 배포 명령이 계속 인증 실패로 끝났다. 팀은 처음에 토큰 자체가 만료된 줄 알고 토큰을 재발급했지만 같은 문제가 반복됐다.

원인은 network.tlsTerminate를 설정하지 않은 채 mask만 켠 것이었다 — 프록시가 암호화된 요청 내용을 볼 수 없어 sentinel 값을 실제 토큰으로 치환하지 못하고, 그대로 서버에 전달되어 인증이 fail-closed된 것이다. /doctor가 이 오설정을 경고하고 있었지만 팀이 처음엔 확인하지 않았다. 이 사례는 "보안 기능을 켰는데 애플리케이션이 깨진다"는 신호가 오설정 때문일 수 있다는 것과, 새 기능을 켤 때 전제 조건까지 함께 확인해야 한다는 교훈을 준다.

시나리오 3 — 넓은 도메인 허용이 만든 잠재적 exfiltration 경로

한 팀이 CI 자동화에서 github.com을 sandbox.network.allowedDomains에 통째로 허용했다. 보안 검토 과정에서, 프록시가 hostname만으로 판단하고 TLS 내용을 검사하지 않기 때문에 이론적으로 domain fronting 같은 기법으로 허용 목록 밖 호스트에 데이터가 흘러갈 수 있는 경로가 남아 있다는 것을 발견했다.

팀은 threat model을 다시 검토해, 정말 필요한 서브도메인만 allowedDomains에 최소한으로 등록하고, 규제 대상 데이터를 다루는 파이프라인에는 TLS를 종료하고 내용을 검사하는 custom proxy를 추가로 배치했다. 이 사례는 "네트워크 허용 목록이 있으니 안전하다"가 threat model에 따라 충분하지 않을 수 있다는 것을 보여준다.

5. 답변 사고 순서

면접에서 "샌드박스/격리를 어떻게 설계하는가"류의 질문을 받으면, 아래 순서로 답을 구성하라.

  1. 권한(승인 여부)과 샌드박스(접근 범위)를 계층으로 분리한다

    permission이 "이 행동을 허락할지"를 결정하는 문지기라면, 샌드박스는 "허락받은 뒤 실제로 무엇에 닿을 수 있는가"를 OS가 강제하는 울타리라는 것을 먼저 밝힌다.

  2. scope 경계를 명시한다

    내장 샌드박스는 Bash와 자식 프로세스만 격리하고, Read/Edit/WebFetch/MCP/Hooks는 host에서 직접 실행되어 permission 규칙으로만 통제된다는 것을 구체적으로 짚는다.

  3. 기본값의 한계를 짚는다

    기본 read 정책이 credential 파일을 막지 않는다는 것, 네트워크 격리가 기본적으로 TLS 내용을 검사하지 않는다는 것을 예시로 설명한다.

  4. threat model에 맞는 격리 스펙트럼을 선택한다

    Bash만 필요한지, 전체 프로세스(sandbox-runtime/컨테이너/VM)까지 필요한지를 요구사항(특히 --dangerously-skip-permissions 사용 여부)에 맞춰 판단한다.

  5. 조직 enforcement로 마무리한다

    managed settings로 강제하는 방법과, excludedCommands처럼 잠글 수 없는 한계까지 짚어 실무 감각을 보여준다.

핵심 팁

"샌드박스를 켰다"는 사실만 말하지 말고, 반드시 "이 격리가 정확히 무엇을 막고 무엇은 막지 못하는가"까지 스스로 검증하는 태도를 보여줘라. 인터뷰어는 기능 나열보다 "격리 경계를 공격자 관점에서 재검토하는 습관"에 더 높은 점수를 준다.

6. 핵심 답변 예시

면접 답변 예시 — "완전 자율 실행을 안전하게 만들려면?"

"먼저 권한과 샌드박스를 분리해서 생각합니다. --dangerously-skip-permissions는 protected path와 루트/홈 삭제에 대한 circuit breaker를 제외하면 사실상 모든 사전 승인을 제거하는 모드이므로, 이 모드를 쓰는 순간부터는 '무엇을 승인할지'가 아니라 '승인 없이도 실제로 얼마나 접근할 수 있는가'가 유일한 안전장치가 됩니다.

그런데 Claude Code 내장 샌드박스(sandboxed Bash tool)는 Bash 명령과 그 자식 프로세스만 OS 레벨로 격리합니다. Read/Edit/Write는 애초에 임의 코드를 스폰하지 않아 permission 규칙으로만 통제되고, MCP 서버와 Hooks는 host에서 별도 프로세스로 무제한 실행됩니다. 즉 Bash 샌드박스만 켜고 나머지 도구를 그대로 두면, 완전 자율 실행 시 MCP나 훅을 통한 우회 경로가 남습니다. 그래서 이 요구에는 Bash 샌드박스가 아니라 Claude Code 프로세스 전체를 감싸는 격리 — sandbox-runtime이나 devcontainer/커스텀 컨테이너 — 를 골라야 합니다.

그 안에서도 두 가지를 추가로 점검합니다. 첫째, 기본 파일시스템 read 정책은 credential 파일을 막지 않으므로 sandbox.credentials로 ~/.aws, ~/.ssh 같은 경로와 토큰 환경변수를 명시적으로 deny하거나, 도구가 그 인증을 계속 써야 한다면 mask 모드로 sentinel 치환을 씁니다 — 단 mask는 network.tlsTerminate 없이는 fail-closed된다는 점을 확인합니다. 둘째, 네트워크 프록시는 기본적으로 hostname만 보고 TLS 내용은 검사하지 않으므로, github.com 같은 넓은 도메인을 허용하면 domain fronting류의 이론적 exfiltration 경로가 남는다는 것을 팀에 명시하고, 규제 데이터를 다루는 파이프라인이라면 TLS를 종료하는 custom proxy까지 검토합니다.

마지막으로 이 정책을 개인 설정에 맡기지 않고 managed settings로 sandbox.enabled, failIfUnavailable, allowUnsandboxedCommands: false를 강제하고, allowManagedReadPathsOnly/allowManagedDomainsOnly로 개발자가 read 경로나 도메인을 임의로 넓히지 못하게 잠급니다. 다만 excludedCommands는 managed-only 잠금이 없어 개발자가 항상 항목을 추가할 수 있다는 한계가 남으므로, 이 목록을 최소화하고 정기적으로 감사하는 것을 조직 프로세스에 포함시킵니다."

이 답변이 좋은 이유: ① 권한(승인)과 샌드박스(접근 범위)를 명확히 분리 ② Bash 샌드박스의 scope 한계(MCP/훅 미포함)를 정확히 알고 전체 프로세스 격리로 확장 ③ 기본값의 숨은 위험(credential read 허용, TLS 미검사)을 스스로 검증 ④ mask의 전제 조건(tlsTerminate)까지 아는 세부 지식 ⑤ 조직 enforcement와 그 한계(excludedCommands)까지 연결해 실무 배포 경험을 드러냄

7. Senior-level Follow-up Q&A

Q1. "샌드박스가 있는데 permission 규칙도 따로 필요한 이유가 무엇입니까?"

둘은 적용 범위와 판단 시점이 다릅니다. permission 규칙은 Bash뿐 아니라 Read/Edit/WebFetch/MCP/Agent 등 모든 도구에 적용되고, 명령이 실행되기 전에 명령 문자열(또는 auto mode의 classifier 판단) 기준으로 결정됩니다. 샌드박스는 Bash와 그 자식 프로세스에만 적용되고, 이미 실행 중인 프로세스에 대해 운영체제가 실제 접근 시점에 강제합니다 — 그래서 모델이 무엇을 실행했다고 "말했는지"와 무관하게, 그 프로세스가 실제로 무엇을 하든 경계가 유지됩니다.

즉 permission이 없으면 애초에 위험한 명령이 시도조차 안 되게 막고, 샌드박스는 그 명령이 시도된 이후에도 피해 범위를 제한합니다. 두 계층 다 있어야 defense-in-depth가 완성됩니다.

Q2. "MCP 서버는 왜 이 샌드박스로 보호되지 않습니까? 그럼 어떻게 보호합니까?"

내장 샌드박스는 Bash 도구 호출을 가로채 자식 프로세스를 격리된 네임스페이스 안에서 실행하는 방식으로 동작합니다. MCP 서버는 Claude Code가 시작 시점에 별도 프로세스로 띄우는 host 상의 독립 프로세스라서, 이 Bash 전용 가로채기 지점을 거치지 않습니다. 그래서 MCP 서버 자체의 코드 신뢰성과 permission 규칙(mcp__서버명 allow/deny)에만 의존하게 됩니다.

MCP 서버까지 OS 레벨로 격리하려면 Claude Code 프로세스 전체를 감싸는 방식이 필요합니다 — @anthropic-ai/sandbox-runtime으로 전체 프로세스를 wrapping하거나, devcontainer/커스텀 컨테이너/VM 안에서 Claude Code(와 그 MCP 서버들)를 통째로 실행하는 것입니다.

Q3. "샌드박스의 네트워크 격리가 TLS를 검사하지 않는다면, 이건 사실상 없는 것과 같지 않습니까?"

완전히 무력하지는 않습니다. hostname 기준 allowlist만으로도 "임의의 아무 도메인에나 연결"하는 것은 막습니다 — 예를 들어 공격자가 완전히 새로운 미등록 도메인으로 데이터를 보내려는 시도는 차단됩니다. 다만 이미 허용된 도메인 내부에서 오가는 내용은 검사되지 않고, hostname 자체를 속이는 domain fronting류의 기법에는 취약할 수 있다는 한계가 있습니다.

그래서 "어떤 도메인을 허용 목록에 넣는가"가 실질적인 보안 결정이 됩니다. github.com처럼 넓은 도메인보다 필요한 서브도메인만 좁게 허용하는 것이 낫고, 규제 데이터처럼 더 강한 보장이 필요하면 자체 CA로 TLS를 종료해 내용까지 검사하는 custom proxy를 추가로 두어야 합니다. "TLS를 검사 안 하니 의미 없다"가 아니라 "hostname 레벨 방어이지 콘텐츠 레벨 방어는 아니다"라고 정확히 이해하는 것이 중요합니다.

Q4. "sandbox.credentials의 mask 모드는 deny보다 왜 더 정교합니까? 언제 deny 대신 mask를 씁니까?"

deny는 환경변수를 완전히 제거하므로 그 credential로 인증해야 하는 gh, npm 같은 도구 자체가 깨집니다. mask는 샌드박스 안에서는 세션별 sentinel 값만 보이게 하고, 요청이 injectHosts로 지정된 신뢰 호스트로 나갈 때만 프록시가 실제 값으로 치환합니다 — 그래서 명령과 그 로그는 실제 값을 절대 들고 있지 않으면서도, 정당한 호스트로의 인증은 그대로 동작합니다.

"이 도구가 반드시 이 credential로 인증해야 동작하는데, 그 값을 모델 컨텍스트나 로그에 노출시키고 싶지 않다"는 요구가 있을 때 mask를 선택합니다. 반대로 그 도구/명령이 애초에 그 credential 없이도 동작해야 한다면(혹은 격리된 세션에서 그 인증 자체를 아예 막고 싶다면) deny가 더 단순하고 안전합니다. mask는 network.tlsTerminate가 전제 조건이라는 것, 그리고 저장소에 커밋되는 프로젝트 설정에서는 무시된다는 것도 함께 기억해야 합니다.

Q5. "조직에서 샌드박스를 강제했는데, 개발자가 excludedCommands에 항목을 계속 추가해서 사실상 정책이 무력화되고 있습니다. 어떻게 대응합니까?"

excludedCommands는 다른 배열 설정과 달리 managed-only 잠금 장치가 없어, 어느 스코프에서든 추가된 항목이 병합됩니다. 이 설정 자체로는 막을 방법이 없다는 것을 먼저 인정하고, 대신 프로세스적 통제로 보완합니다 — 예를 들어 .claude/settings.json 변경을 코드 리뷰 대상에 포함시키거나, CI에서 excludedCommands 목록의 diff를 감지해 보안 담당자 승인을 요구하는 훅/파이프라인을 추가하는 방식입니다.

동시에 "왜 자꾸 추가되는지"를 파악하는 것도 중요합니다 — docker, `jest --watchman`처럼 샌드박스와 근본적으로 호환되지 않는 도구 때문이라면, 그 도구들만 예외로 남기고 나머지 요청은 allowWrite/allowedDomains로 더 정밀하게 허용하는 방향으로 유도해 예외 목록이 무분별하게 커지는 것을 막습니다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • 권한(승인 계층)과 샌드박스(접근 범위 계층)를 명확히 분리해서 설명한다
  • 내장 샌드박스가 Bash만 격리하고 MCP/훅/파일 도구는 host에서 직접 실행된다는 scope 경계를 정확히 안다
  • credential 기본 read 허용, TLS 미검사 같은 기본값의 숨은 한계를 스스로 짚는다
  • threat model에 맞춰 Bash 샌드박스 ↔ sandbox-runtime ↔ 컨테이너/VM 중 적절한 격리 강도를 고른다
  • 조직 enforcement(managed settings)와 그 한계(excludedCommands 잠금 불가)까지 연결한다
약한 답변
  • "샌드박스를 켰으니 안전하다"고 권한/샌드박스를 하나로 뭉뚱그린다
  • MCP 서버나 Hooks도 자동으로 샌드박스에 포함된다고 잘못 답한다
  • 기본 read 정책이 credential 파일을 막아준다고 착각한다
  • 네트워크 격리가 트래픽 내용까지 검사한다고 잘못 이해한다
  • "--dangerously-skip-permissions + 샌드박스"만으로 완전 자율 실행이 안전해진다고 답한다
왜 약한 답변이 약한가

샌드박스라는 기능이 존재한다는 것 자체는 누구나 안다. 시니어 면접이 확인하려는 건 "이 격리가 정확히 어디까지 미치고 어디서부터 뚫리는지 스스로 검증할 수 있는가"다. 특히 "샌드박스 + skip-permissions면 안전하다"는 답은 MCP/훅이라는 실제 우회 경로를 놓친 것이라 프로덕션에서 그대로 사고로 이어질 수 있는 위험한 착각이며, 감점 폭이 크다.

9. 흔한 오해

오해 1

"샌드박스를 켜면 Claude Code의 모든 도구 실행이 격리된다."

실제

내장 샌드박스(sandboxed Bash tool)는 Bash 명령과 그 자식 프로세스만 격리한다. Read/Edit/Write/WebFetch/MCP/Hooks는 Claude Code 프로세스 안에서 직접 실행되거나 host에서 별도 프로세스로 무제한 실행되며, permission 규칙으로만 통제된다. 전체를 격리하려면 sandbox-runtime이나 컨테이너/VM이 필요하다.

오해 2

"샌드박스의 기본 read 정책이 ~/.aws/credentials, ~/.ssh 같은 민감 파일을 자동으로 막아준다."

실제

기본 read 정책은 컴퓨터 전체를 허용하며, 여기에는 credential 파일도 포함된다. 이를 막으려면 sandbox.credentials의 deny/denyRead를 명시적으로 설정해야 한다.

오해 3

"네트워크 격리가 허용된 도메인 안에서 오가는 데이터의 내용까지 검사한다."

실제

기본적으로 프록시는 hostname만 보고 판단하며 TLS 트래픽 내용은 검사하지 않는다. 넓은 도메인을 허용하면 그 안에서의 실제 요청 내용은 통제 밖에 남고, domain fronting류의 우회 가능성도 이론적으로 존재한다. 콘텐츠 검사가 필요하면 자체 CA로 TLS를 종료하는 custom proxy가 필요하다.

오해 4

"--dangerously-skip-permissions와 내장 Bash 샌드박스를 함께 켜면 완전 자율 실행이 안전하다."

실제

내장 Bash 샌드박스만으로는 MCP 서버와 Hooks가 여전히 host에서 무제한으로 실행된다. --dangerously-skip-permissions는 반드시 sandbox-runtime, 컨테이너, VM 같은 전체 프로세스 격리 안에서 사용해야 한다.

오해 5

"sandbox.credentials의 mask를 켜기만 하면 credential이 안전하게 치환된다."

실제

mask는 network.tlsTerminate가 함께 설정되어야 프록시가 요청 내용을 보고 sentinel을 실제 값으로 치환할 수 있다. tlsTerminate 없이 mask만 켜면 sentinel이 그대로 서버에 전달되어 인증이 조용히 fail-closed된다.

오해 6

"샌드박스 관련 설정(excludedCommands 포함)은 managed settings로 모두 잠글 수 있다."

실제

enabled/failIfUnavailable 같은 boolean 키는 managed 값이 절대적으로 이기지만, excludedCommands는 managed-only 잠금 장치가 없어 어느 스코프에서든 추가된 항목이 병합된다. 이 부분은 설정만으로 막을 수 없고 리뷰 프로세스 같은 조직적 통제로 보완해야 한다.
