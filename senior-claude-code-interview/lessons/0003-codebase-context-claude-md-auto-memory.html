2. 전제 개념 — 쉬운 한국어로

왜 "코드베이스 컨텍스트 설계"라는 별도 주제가 필요한가

Day 1에서 Claude Code를 "매 세션 빈 컨텍스트 윈도우로 시작하는 agentic harness"라고 정의했다. 그런데 실제로 팀에서 코드를 다루려면 "이 프로젝트는 어떤 빌드 명령을 쓰는지, 어떤 코딩 컨벤션을 따르는지, 어떤 부분을 건드리면 안 되는지"를 Claude가 매번 새로 배우게 할 수는 없다. CLAUDE.md(사람이 쓰는 지침)와 auto memory(Claude가 스스로 기록하는 학습 노트)는 이 "세션 간 지식 전달"을 담당하는 두 축이고, 이 둘을 어떻게 설계하느냐가 "결과의 신뢰성"과 "컨텍스트 비용" 사이의 trade-off를 직접 결정한다.

CLAUDE.md — 사람이 쓰는 지침
누가 쓰는가: 사람. 무엇을: 규칙과 워크플로우. 범위: 프로젝트/사용자/조직. 용도: 코딩 표준, 빌드 명령, 아키텍처 결정, "항상 X 하라" 규칙.
Auto Memory — Claude가 쓰는 학습 노트
누가 쓰는가: Claude 자신. 무엇을: 사용자의 교정/선호에서 학습한 내용. 범위: 저장소 단위(머신 로컬). 용도: 디버깅 인사이트, 빌드 명령 발견, 사용자가 매번 반복 교정했던 습관.

두 시스템 모두 매 대화 시작 시 로드되지만, Claude는 이 둘을 "강제로 지켜야 하는 설정"이 아니라 "컨텍스트로 참고하는 정보"로 취급한다. 이 구분이 이번 Day 전체를 관통하는 가장 중요한 전제다 — 어길 수 없는 통제가 필요하면 CLAUDE.md가 아니라 PreToolUse 훅(Day 9)을 써야 한다.

CLAUDE.md 4단계 스코프 — 어디에 두느냐가 곧 누구와 공유하느냐다

공식 문서는 CLAUDE.md가 저장될 수 있는 위치를 4단계로 구분하고, 이 순서를 "가장 넓은 범위 → 가장 좁은 범위" 순서, 즉 로드 순서로 명시한다.

스코프위치용도공유 대상
Managed policymacOS /Library/Application Support/ClaudeCode/CLAUDE.md 등 OS별 고정 경로 (또는 managed-settings.json의 claudeMd 키)조직 전체 코딩 표준, 보안 정책, 컴플라이언스 요건조직 내 모든 사용자, 개인 설정으로 제외 불가
User~/.claude/CLAUDE.md개인 취향, 개인 툴링 단축 명령나 자신 (모든 프로젝트)
Project./CLAUDE.md 또는 ./.claude/CLAUDE.md프로젝트 아키텍처, 코딩 표준, 공통 워크플로우버전 관리를 통한 팀 전체
Local./CLAUDE.local.md개인 프로젝트별 설정(샌드박스 URL, 테스트 데이터)나 자신 (현재 프로젝트, .gitignore 대상)

네 파일 모두 로드되면 서로 덮어쓰는 게 아니라 이어붙여진다(concatenate). 순서는 "가장 넓은 범위가 먼저, 가장 좁은 범위(=지금 작업 중인 위치에 가까운 지침)가 마지막"이다 — user보다 project가 나중에 나타나고, 같은 디렉토리 안에서는 CLAUDE.md 다음에 CLAUDE.local.md가 나중에 온다. "나중에 읽은 지침이 더 구체적이고 최신 맥락에 가깝다"는 순서 설계다.

디렉토리 트리 로드 순서 — 즉시 로드 vs 온디맨드 로드

이 부분이 이번 Day에서 가장 자주 오해되는 지점이다. Claude Code는 현재 작업 디렉토리부터 상위로 거슬러 올라가며 각 디렉토리의 CLAUDE.md/CLAUDE.local.md를 찾는다. 하지만 "서브디렉토리"의 CLAUDE.md는 이야기가 다르다:

  • 작업 디렉토리 자체와 그 모든 상위 디렉토리의 CLAUDE.md/CLAUDE.local.md는 세션 시작(launch) 시점에 전부 로드된다.
  • 작업 디렉토리 아래의 서브디렉토리에 있는 CLAUDE.md는 세션 시작 시 로드되지 않는다. Claude가 그 서브디렉토리 안의 파일을 읽을 때 비로소 온디맨드로 로드된다.

즉 저장소 루트에서 세션을 시작하면 루트 CLAUDE.md만 즉시 로드되고, packages/api/CLAUDE.md는 Claude가 실제로 packages/api/ 안의 파일을 열어볼 때 로드된다. 반대로 packages/api/에서 세션을 시작하면 그 디렉토리와 모든 상위 디렉토리(결국 루트까지)의 CLAUDE.md가 즉시 로드되고, packages/web/의 CLAUDE.md는 애초에 그쪽 파일을 읽지 않는 한 컨텍스트에 들어오지 않는다. 이 메커니즘 덕분에 "필요한 서브시스템의 컨텍스트만 자연스럽게 로드된다."

.claude/rules/ — path-scoped rules, 새로운 세 번째 축

per-directory CLAUDE.md와 별개로, 공식 문서는 .claude/rules/ 디렉토리에 마크다운 파일을 여러 개 두는 방식을 제공한다. 각 파일은 하나의 주제를 다루고, YAML frontmatter의 paths 필드로 글롭 패턴에 매칭되는 파일을 다룰 때만 조건부로 로드되도록 스코프를 지정할 수 있다. 

---
paths:
  - "src/api/**/*.ts"
---

# API Development Rules
- All API endpoints must include input validation
- Use the standard error response format

paths 없는 규칙 파일은 .claude/CLAUDE.md와 동일한 우선순위로 launch 시 로드된다. path-scoped 규칙은 Claude가 매칭되는 파일을 읽을 때 트리거되지, 매 도구 사용마다 재평가되는 게 아니다. 사용자 레벨 규칙(~/.claude/rules/)은 프로젝트 규칙보다 먼저 로드되어(즉 프로젝트 규칙이 더 나중에, 더 구체적으로 취급되어) 우선순위가 더 높다. 심볼릭 링크도 지원되므로 여러 프로젝트에 공유 규칙 세트를 링크해 재사용할 수 있다.

Per-directory CLAUDE.md vs .claude/rules/ — 언제 무엇을 쓰는가
방식파일 위치로드 시점적합한 경우
Per-directory CLAUDE.md해당 코드 옆(디렉토리 안)그 디렉토리에서 시작 시, 또는 그 디렉토리 파일을 읽을 때디렉토리 소유팀이 자기 컨벤션을 코드와 함께 버전 관리
.claude/rules/ path-scoped rule저장소 루트의 중앙 .claude/패턴에 매칭되는 파일을 다룰 때규칙을 한 곳에 모으고 싶거나, 같은 규칙이 흩어진 여러 경로에 적용될 때

Import(@path)와 AGENTS.md 상호운용

CLAUDE.md는 @path/to/file 문법으로 다른 파일을 불러올 수 있다(상대/절대 경로 모두 가능, 재귀 import 최대 4단계, 백틱 안에 쓰면 import 안 됨). 다만 import된 파일도 launch 시 그대로 컨텍스트에 로드되므로, "여러 파일로 쪼개면 컨텍스트가 줄어든다"는 것은 착각이다 — import는 조직(organization) 목적이지 비용 절감 목적이 아니다. 또한 Claude Code는 AGENTS.md를 직접 읽지 않는다(다른 코딩 에이전트가 쓰는 파일). 이미 AGENTS.md를 쓰는 저장소라면 CLAUDE.md에서 @AGENTS.md로 import하거나 심볼릭 링크를 걸어 중복 없이 공유한다.

컨텍스트 비용 관리 — "무엇이 실제로 비용을 줄이는가"

전략실제 효과
CLAUDE.md를 200줄 이하로 유지효과 있음 — 길수록 컨텍스트 소모가 크고 준수율(adherence)도 낮아진다는 것이 공식 가이드
path-scoped rules(.claude/rules/)로 분리효과 있음 — 매칭되는 파일을 다룰 때만 로드되므로 무관한 작업에서는 컨텍스트를 아예 쓰지 않는다
서브디렉토리별 CLAUDE.md 분리효과 있음 — 루트에서 시작해도 실제로 건드리는 서브디렉토리의 것만 온디맨드로 로드된다
@path import로 파일 쪼개기효과 없음 — 정리(organization) 목적일 뿐, import된 내용도 launch 시 전부 로드된다
claudeMdExcludes로 무관한 팀의 CLAUDE.md 제외효과 있음 — 모노레포에서 안 건드리는 패키지의 지침을 아예 로드 대상에서 제거

Auto Memory — MEMORY.md와 200줄/25KB 임계값

Auto memory는 Claude가 사용자의 교정이나 선호를 스스로 저장하는 시스템이다(v2.1.59 이상, 기본 켜짐). 저장 위치는 ~/.claude/projects/<project>/memory/이고, <project>는 git 저장소 기준으로 파생되므로 같은 저장소의 모든 worktree와 서브디렉토리가 하나의 메모리 디렉토리를 공유한다. 단, 이 디렉토리는 머신 로컬(machine-local)이다 — 다른 컴퓨터나 클라우드 환경과는 공유되지 않는다. 

~/.claude/projects/<project>/memory/
├── MEMORY.md          # 인덱스 역할, 매 세션 로드
├── debugging.md       # 상세 노트 (온디맨드)
├── api-conventions.md # 상세 노트 (온디맨드)
└── ...

MEMORY.md의 처음 200줄 또는 25KB(둘 중 먼저 도달하는 쪽)만 매 대화 시작 시 로드된다. 그 이상은 로드되지 않으며, Claude는 상세 내용을 별도 주제 파일로 옮겨 MEMORY.md를 인덱스처럼 간결하게 유지한다. 이 200줄/25KB 제한은 MEMORY.md에만 적용되고, CLAUDE.md는 길이와 무관하게 항상 전체가 로드된다(다만 짧을수록 준수율이 좋다는 것은 동일하게 적용). /memory 명령으로 로드된 모든 CLAUDE.md/rules/auto memory 폴더를 확인·편집할 수 있다.

강제(Enforcement)가 아니라 컨텍스트다

공식 문서가 명시하는 핵심 경계

"CLAUDE.md 내용은 시스템 프롬프트의 일부가 아니라, 시스템 프롬프트 뒤에 오는 하나의 사용자 메시지로 전달된다. Claude는 이를 읽고 따르려 하지만, 엄격한 준수가 보장되지는 않는다 — 특히 모호하거나 서로 충돌하는 지침일수록 그렇다." 즉 CLAUDE.md와 auto memory는 둘 다 Claude의 판단에 영향을 주는 컨텍스트이지, Claude가 무엇을 하든 강제로 막는 계층이 아니다. 특정 시점에 반드시 실행되어야 하거나(커밋 전 lint 등) 절대 우회되면 안 되는 규칙은 PreToolUse 훅(Day 9)이나 permission deny 규칙(Day 4)으로 옮겨야 한다.

Compaction과의 상호작용

/compact 이후 프로젝트 루트 CLAUDE.md는 디스크에서 다시 읽혀 재주입된다. 하지만 서브디렉토리의 CLAUDE.md는 자동으로 재주입되지 않는다 — Claude가 그 서브디렉토리의 파일을 다시 읽을 때 재로드된다. 대화 중에만 구두로 준 지침("이번 세션에서는 이렇게 해줘")은 compaction 이후 사라질 수 있으므로, 계속 유지해야 하는 지침이라면 CLAUDE.md에 적어두는 것이 안전하다(Day 18에서 compaction 메커니즘을 더 깊이 다룬다).

대규모 코드베이스/모노레포 전략 — 레이어링이 한계에 부딪힐 때

공식 가이드는 코드베이스가 커질수록 아래 도구들을 층층이 조합하도록 안내한다.

필요수단
손대는 코드의 컨벤션만 로드서브디렉토리별 CLAUDE.md (온디맨드 로드)
절대 안 건드리는 패키지의 CLAUDE.md 제외claudeMdExcludes (글롭 패턴, 여러 설정 스코프에서 병합됨)
빌드 산출물/벤더 코드를 아예 못 읽게 차단permissions.deny의 Read(...) 규칙
파일을 훑지 않고 심볼 정의/참조를 바로 찾기code intelligence 플러그인(언어 서버 연동)
서브에이전트/worktree가 저장소 전체를 체크아웃하지 않게worktree.sparsePaths + symlinkDirectories
형제 패키지나 다른 저장소도 함께 수정additionalDirectories 설정 또는 --add-dir
영역별 절차를 온디맨드로만 로드디렉토리별 .claude/skills/
레이어링 자체가 거버넌스 부담이 될 때플러그인/MCP 서버로 컨벤션을 중앙화(팀이 버전 관리)

중요한 세부사항 하나: additionalDirectories 설정으로 추가한 디렉토리는 CLAUDE.md/rules/skills를 전혀 로드하지 않는다(파일 접근 권한만 부여). 반면 --add-dir 플래그나 /add-dir 명령으로 추가하면 skills는 로드되고, CLAUDE.md/rules는 CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD 환경 변수를 켜야 로드된다. "권한 부여"와 "컨텍스트 로드"가 별개로 제어된다는 것이 이 설정의 함정이다.

3. 왜 면접에서 묻는가

인터뷰어가 평가하는 것
  • "설정 파일 문법을 아는가"가 아니라 "컨텍스트 비용 vs 신뢰성의 trade-off를 판단할 수 있는가": CLAUDE.md를 무한정 길게 쓰면 지침은 촘촘해지지만 컨텍스트를 잡아먹고 준수율도 떨어진다. 이 긴장 관계를 이해하고 있는지를 본다.
  • Enforcement의 한계를 아는가: "CLAUDE.md에 금지 규칙을 적었으니 안전하다"고 답하면 위험 신호다. 컨텍스트 기반 지침과 강제 계층(훅/권한)의 경계를 정확히 아는 것은 프로덕션 사고를 막는 핵심 판단력이다.
  • 대규모 코드베이스 운영 경험: 개인 프로젝트 수준의 CLAUDE.md 하나로는 충분하지 않다. 모노레포에서 레이어링·제외·중앙화를 조합해 설계할 수 있는지가 시니어와 미드레벨을 가르는 지점이다.
  • 세션/컴팩션 라이프사이클 이해: "왜 어떤 지침은 유지되고 어떤 지침은 사라지는가"를 설명할 수 있어야, 장시간 세션에서 발생하는 예측 불가능한 동작을 디버깅할 수 있다.

이 질문은 "CLAUDE.md 문법을 외웠는가"를 묻지 않는다. "컨텍스트를 설계 대상으로 다루고, 그 설계가 결과의 신뢰성·비용·강제력에 미치는 영향을 예측할 수 있는가"를 묻는다.

4. 실무 연결

실제로 이 지식이 필요한 순간들

시나리오 1 — 루트 CLAUDE.md가 800줄로 불어나 지침을 무시하기 시작한 사례

한 팀이 모노레포 루트 CLAUDE.md에 API 서버, 프론트엔드, 배치 잡, 인프라 스크립트의 컨벤션을 전부 적어 넣었다. 처음엔 문제가 없어 보였지만, 파일이 800줄을 넘어가면서 프론트엔드 작업 중에도 배치 잡 규칙이 컨텍스트에 함께 실려 다니게 됐고, 팀은 "최근 들어 Claude가 자꾸 정해둔 컨벤션을 놓친다"는 불만을 반복적으로 겪었다.

원인은 CLAUDE.md의 길이가 늘어날수록 준수율이 떨어진다는 공식 가이드와 정확히 일치했다. 해결책은 루트에는 저장소 전체에 적용되는 규칙만 남기고, 패키지별 규칙은 각 패키지 디렉토리의 CLAUDE.md로, 파일 타입별 규칙(예: 테스트 파일 컨벤션)은 .claude/rules/의 path-scoped 규칙으로 분리하는 것이었다. 그 결과 특정 패키지에서 작업할 때는 그 패키지와 무관한 규칙이 컨텍스트에 실리지 않게 됐다.

시나리오 2 — "CLAUDE.md에 적어뒀으니 안전하다"고 믿었던 사고

한 팀이 CLAUDE.md에 "프로덕션 데이터베이스에는 절대 쓰기 작업을 실행하지 말 것"이라고 굵게 적어두고 안심하고 있었다. 그런데 한 세션에서 사용자가 모호하게 "이 마이그레이션 스크립트 실행해줘"라고 요청했고, 그 스크립트가 우연히 프로덕션 접속 정보를 기본값으로 물고 있던 환경에서 Claude가 그대로 실행해 버렸다.

사후 분석에서 팀이 깨달은 것은, CLAUDE.md 지침은 시스템 프롬프트가 아니라 대화 맥락으로 전달되는 컨텍스트이고, 모호한 지시나 상충하는 신호 앞에서는 절대적으로 지켜진다는 보장이 없다는 점이었다. 이후 팀은 같은 규칙을 permissions.deny로 프로덕션 접속 문자열 패턴에 대한 Bash 실행을 차단하고, PreToolUse 훅으로 위험한 명령을 이중으로 검증하도록 바꿨다. "CLAUDE.md는 방향을 제시하고, 훅/권한은 경계를 강제한다"는 역할 분리를 실제 사고 이후에야 체득한 사례다.

시나리오 3 — auto memory가 머신 로컬이라는 것을 모르고 온보딩에서 혼란을 겪은 사례

신입 팀원이 "선임 개발자가 예전에 Claude에게 알려준 이 프로젝트의 특이한 빌드 순서를, 내 세션에서는 왜 Claude가 전혀 모르는지" 의아해했다. 알고 보니 그 지식은 선임 개발자의 auto memory(~/.claude/projects/.../memory/)에만 저장되어 있었고, auto memory는 머신 로컬이라 다른 개발자의 컴퓨터로 자동 전파되지 않았다.

이 사례를 계기로 팀은 규칙을 세웠다: "여러 세션에 걸쳐 반복적으로 필요하지만 개인적인 디버깅 습관 수준인 것"은 auto memory에 맡겨두되, "팀 전체가 알아야 하는 지식"은 반드시 사람이 검토해 CLAUDE.md로 승격시킨다는 워크플로우다. 즉 auto memory는 "개인의 학습 축적"이고 CLAUDE.md는 "팀의 공식 지식"이라는 구분을 운영 정책으로 명문화한 것이다.

5. 답변 사고 순서

면접에서 "CLAUDE.md/auto memory를 어떻게 설계하는가"류의 질문을 받으면, 아래 순서로 답을 구성하라.

  1. CLAUDE.md와 auto memory를 "누가 쓰는가" 기준으로 먼저 구분한다

    사람이 쓰는 명시적 지침(CLAUDE.md)과 Claude가 스스로 기록하는 학습 노트(auto memory)라는 역할 차이를 먼저 규정한다.

  2. 로드 순서와 타이밍을 설명한다

    4단계 스코프(managed → user → project → local)가 이어붙여지는 순서, 작업 디렉토리와 상위는 즉시 로드되지만 서브디렉토리는 온디맨드로 로드된다는 것을 짚는다.

  3. 컨텍스트 비용을 낮추는 실제 수단과 가짜 수단을 구분한다

    path-scoped rules와 서브디렉토리 분리는 진짜로 비용을 줄이지만, @path import는 정리 목적일 뿐 비용을 줄이지 않는다는 것을 명확히 한다.

  4. Enforcement의 한계를 짚는다

    CLAUDE.md/auto memory는 컨텍스트이지 강제 계층이 아니라는 것, 절대 어겨서는 안 되는 규칙은 훅/권한으로 옮겨야 한다는 것을 설명한다.

  5. 대규모 코드베이스 전략으로 확장한다

    서브디렉토리별 분리, claudeMdExcludes, Read deny 규칙, code intelligence, sparse worktree, 그리고 레이어링이 거버넌스 부담이 될 때 플러그인/MCP로 중앙화하는 흐름을 제시한다.

  6. 팀 운영 원칙으로 마무리한다

    "팀 지식은 CLAUDE.md로 승격, 개인 학습은 auto memory에 맡긴다"처럼 구체적인 운영 규칙으로 답을 맺는다.

핵심 팁

CLAUDE.md 문법을 나열하는 데서 멈추지 말고, 반드시 "그래서 이 설계가 컨텍스트 비용과 신뢰성 사이에서 어떤 trade-off를 만드는가"로 연결하라. 이 연결이 없으면 인터뷰어에게는 "문서를 읽었다" 이상의 신호를 주지 못한다.

6. 핵심 답변 예시

면접 답변 예시 — "CLAUDE.md와 auto memory를 대규모 코드베이스에서 어떻게 설계합니까?"

"먼저 두 시스템의 역할을 구분합니다. CLAUDE.md는 사람이 직접 쓰는 명시적 지침이고, auto memory는 Claude가 세션 중 사용자의 교정이나 선호에서 스스로 학습해 MEMORY.md라는 인덱스 파일에 기록하는 것입니다. 둘 다 매 세션 시작 시 로드되지만, auto memory는 MEMORY.md의 처음 200줄 또는 25KB만 로드되고 나머지 상세 파일은 필요할 때 온디맨드로 읽습니다.

로드 순서도 중요한데, CLAUDE.md는 managed policy → user → project → local 순으로 이어붙여지고, 이 중 작업 디렉토리와 그 상위 디렉토리들의 CLAUDE.md는 세션 시작 시 즉시 로드되지만, 서브디렉토리의 CLAUDE.md는 Claude가 그 서브디렉토리 파일을 실제로 읽을 때가 되어서야 로드됩니다. 그래서 모노레포 루트에서 세션을 시작해도, 실제로 API 패키지를 건드리기 전까지는 프론트엔드 패키지의 지침이 컨텍스트를 차지하지 않습니다.

컨텍스트 비용 관점에서는, 저희 팀은 세 가지를 조합합니다. 첫째, 루트 CLAUDE.md는 저장소 전체에 적용되는 규칙만 200줄 이내로 유지합니다. 둘째, 패키지별 컨벤션은 각 패키지 디렉토리의 CLAUDE.md로, 특정 파일 타입에만 적용되는 규칙은 .claude/rules/의 path-scoped 규칙으로 분리합니다. 셋째, 안 건드리는 패키지는 claudeMdExcludes로 아예 로드 대상에서 뺍니다. 반대로 @path import는 파일을 여러 개로 정리하는 데는 유용하지만 컨텍스트 비용을 줄여주지는 않는다는 것도 알고 있습니다 — import된 내용도 launch 시 그대로 로드되기 때문입니다.

마지막으로 가장 중요하게 여기는 원칙은, CLAUDE.md와 auto memory 둘 다 컨텍스트이지 강제 계층이 아니라는 것입니다. 그래서 '절대 프로덕션에 쓰기 금지' 같은 규칙은 CLAUDE.md에 적는 것으로 끝내지 않고, permissions.deny나 PreToolUse 훅으로 실제로 차단되게 이중화합니다. 그리고 auto memory는 머신 로컬이라 팀원 간에 자동으로 공유되지 않기 때문에, 개인 세션에서 반복적으로 쓸모 있다고 확인된 지식은 검토를 거쳐 CLAUDE.md로 승격시키는 운영 워크플로우를 팀 규칙으로 두고 있습니다."

이 답변이 좋은 이유: ① CLAUDE.md와 auto memory를 역할·로드 방식으로 명확히 구분 ② 서브디렉토리 온디맨드 로딩이라는 자주 틀리는 지점을 정확히 설명 ③ 진짜 비용 절감 수단과 가짜(import) 수단을 구분해 깊이를 드러냄 ④ enforcement 한계를 짚고 훅/권한과의 역할 분리로 연결 ⑤ auto memory의 머신 로컬 특성을 팀 운영 정책(승격 워크플로우)으로 구체화

7. Senior-level Follow-up Q&A

Q1. "서브디렉토리 CLAUDE.md가 온디맨드로 로드된다고 했는데, 그러면 처음부터 필요한 서브디렉토리를 다 안다면 어떻게 미리 로드시킬 수 있습니까?"

가장 확실한 방법은 애초에 그 서브디렉토리에서 세션을 시작하는 것입니다 — 그러면 그 디렉토리와 모든 상위 디렉토리의 CLAUDE.md가 launch 시 즉시 로드됩니다. 저장소 루트에서 시작해야 하는 상황이라면, 세션 초반에 해당 서브디렉토리의 파일을 한 번 읽게 하는 것만으로도 그 시점부터 온디맨드 로딩이 트리거됩니다.

다만 "여러 서브디렉토리에 걸친 작업을 처음부터 알고 있다"면, 애초에 그 규칙들을 .claude/rules/의 path-scoped 규칙으로 옮겨서 관련 파일 패턴에 매칭될 때 자동으로 로드되게 설계하는 것이 더 견고합니다. 특정 시작 위치에 의존하지 않기 때문입니다.

Q2. "additionalDirectories 설정과 --add-dir 플래그가 둘 다 디렉토리 접근을 부여하는데, 왜 다르게 동작합니까?"

additionalDirectories 설정으로 추가한 디렉토리는 파일 읽기/쓰기 권한만 부여하고, 그 디렉토리의 CLAUDE.md·rules·skills는 전혀 로드하지 않습니다. 반면 --add-dir 플래그나 /add-dir 명령으로 런타임에 추가한 디렉토리는 skills는 로드하고, CLAUDE.md/rules는 CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD 환경 변수를 켜야만 로드합니다.

이 차이가 실무에서 중요한 이유는, "권한을 줬으니 당연히 그 디렉토리의 컨벤션도 따르겠지"라고 착각하기 쉽기 때문입니다. 형제 패키지에 접근 권한만 주고 그 패키지의 CLAUDE.md는 로드하지 않으면, Claude가 그 패키지의 컨벤션을 모른 채로 파일을 수정할 위험이 있습니다. 그래서 형제 패키지 수정이 잦은 워크플로우라면 환경 변수를 명시적으로 켜거나, 아예 그 패키지도 함께 스코프에 넣고 세션을 설계해야 합니다.

Q3. "CLAUDE.md 파일들이 서로 충돌하는 지침을 담고 있으면 어떻게 됩니까?"

파일들이 서로를 덮어쓰는 메커니즘이 없기 때문에, 충돌하는 지침은 전부 컨텍스트에 함께 실립니다. 이 경우 Claude가 둘 중 하나를 임의로 선택할 수 있다는 것이 공식 문서의 설명입니다 — 즉 시스템이 자동으로 "더 구체적인 지침이 이긴다"는 식으로 중재해주지 않습니다.

그래서 실무에서는 CLAUDE.md들을 주기적으로 리뷰해서 오래되거나 상충하는 지침을 정리해야 합니다. 특히 모노레포처럼 여러 팀이 각자 CLAUDE.md를 유지하는 구조에서는, 리뷰 프로세스에 CLAUDE.md 변경도 일반 코드 리뷰처럼 포함시키는 것이 권장됩니다 — 코드가 바뀌면 컨벤션 문서도 같이 바뀌어야 드리프트가 안 생기기 때문입니다.

Q4. "auto memory에 잘못된 정보가 저장되면 어떻게 고칩니까? 그리고 이게 팀 전체에 영향을 줍니까?"

auto memory 파일들은 전부 평범한 마크다운이라 /memory 명령으로 열어 직접 읽고 고치거나 지울 수 있습니다. 그리고 auto memory는 머신 로컬이므로, 잘못된 정보가 저장돼도 그건 그 컴퓨터의 그 저장소 세션에만 영향을 줍니다 — 다른 팀원의 머신이나 다른 저장소로 전파되지 않습니다.

다만 이 격리성이 반대 방향의 리스크도 만듭니다: "이 팀원의 로컬에서만 통했던 잘못된 지름길"이 auto memory에 남아, 그 팀원의 다음 세션들에 계속 영향을 줄 수 있습니다. 그래서 auto memory는 주기적으로 감사(audit)하는 습관이 필요하고, 정말 유용하다고 검증된 내용만 CLAUDE.md로 승격시켜 팀 전체가 공유하는 지식으로 만드는 것이 안전한 운영 방식입니다.

Q5. "레이어링(서브디렉토리 CLAUDE.md, rules, excludes)이 이미 있는데도 모노레포 컨텍스트 관리가 안 될 때는 무엇을 합니까?"

공식 가이드가 명시하는 다음 단계는 "레이어링을 더 잘게 쪼개는 것"이 아니라 컨벤션을 항상-로드되는 CLAUDE.md 바깥으로 완전히 빼내는 것입니다. 온디맨드로만 로드되는 skills로 참고 자료를 옮기거나, 플러그인으로 묶어서 플랫폼 팀이 버전 관리·중앙 배포하게 하거나, 이미 코드 검색/RAG 인덱스가 있는 조직이라면 그것을 MCP 서버로 노출해서 Claude가 파일을 직접 훑는 대신 쿼리하게 만드는 방식입니다.

이 지점이 "개별 개발자가 자기 디렉토리 컨벤션을 관리하는 문제"에서 "저장소 전체의 Claude Code 설정을 소유하는 팀이 필요한 문제"로 전환되는 신호입니다. 시니어 면접에서 이 전환 시점을 인지하고 있는지가 실제 대규모 조직 경험을 드러내는 지점입니다.

8. 좋은 답변 vs 약한 답변

좋은 답변
  • CLAUDE.md(사람이 씀)와 auto memory(Claude가 씀)를 역할과 로드 방식으로 정확히 구분한다
  • 작업 디렉토리/상위는 즉시 로드, 서브디렉토리는 온디맨드 로드라는 타이밍 차이를 정확히 짚는다
  • path-scoped rules/서브디렉토리 분리(진짜 비용 절감)와 @path import(비용 절감 아님)를 구분한다
  • CLAUDE.md/auto memory가 강제 계층이 아니라는 것을 알고 훅/권한과의 역할 분리를 설명한다
  • 모노레포 전략(exclude, deny, code intelligence, 플러그인 중앙화)을 상황에 맞게 조합해서 제시한다
약한 답변
  • "CLAUDE.md에 규칙을 적으면 된다"는 수준에서 멈춘다
  • 모든 CLAUDE.md가 세션 시작 시 한꺼번에 로드된다고 잘못 설명한다
  • @path import를 컨텍스트 비용 절감 수단이라고 잘못 말한다
  • "CLAUDE.md에 금지 규칙을 적어두면 안전하다"고 답해 enforcement 한계를 놓친다
  • 대규모 코드베이스 질문에 "CLAUDE.md 하나 잘 쓰면 된다"로 답해 모노레포 전략을 제시하지 못한다
왜 약한 답변이 약한가

CLAUDE.md 존재 자체는 누구나 안다. 시니어 면접이 확인하려는 건 "컨텍스트가 유한한 자원이라는 것을 이해하고, 그 자원을 어디에 어떻게 배분할지 설계할 수 있는가", 그리고 "지침(guidance)과 강제(enforcement)를 구분해 실제 사고를 막을 수 있는가"다. 특히 enforcement 한계를 놓치는 답변은 실제 프로덕션 사고로 이어질 수 있는 위험한 착각이기 때문에 감점 폭이 크다.

9. 흔한 오해

오해 1

"CLAUDE.md에 '절대 X 하지 마라'라고 적으면 Claude는 반드시 그 규칙을 지킨다."

실제

CLAUDE.md는 시스템 프롬프트가 아니라 대화 컨텍스트로 전달되는 지침이다. Claude는 이를 따르려 하지만 모호하거나 상충하는 상황에서는 엄격한 준수가 보장되지 않는다. 절대 어겨서는 안 되는 규칙은 PreToolUse 훅이나 permission deny 규칙 같은 강제 계층으로 옮겨야 한다.

오해 2

"서브디렉토리의 CLAUDE.md도 세션 시작 시 전부 로드된다."

실제

작업 디렉토리와 그 상위 디렉토리들의 CLAUDE.md만 launch 시 즉시 로드된다. 서브디렉토리의 CLAUDE.md는 Claude가 그 서브디렉토리 안의 파일을 실제로 읽을 때 비로소 온디맨드로 로드된다.

오해 3

"@path import로 CLAUDE.md를 여러 파일로 쪼개면 컨텍스트 비용이 줄어든다."

실제

import된 파일도 launch 시 그대로 로드된다. 이 기능은 내용을 정리(organization)하는 데는 유용하지만 컨텍스트 비용을 줄이지는 않는다. 비용을 실제로 줄이려면 path-scoped rules나 서브디렉토리별 온디맨드 로딩을 활용해야 한다.

오해 4

"auto memory는 팀 전체가 공유하는 지식 베이스다."

실제

auto memory는 머신 로컬이다 — 같은 저장소의 모든 worktree/서브디렉토리끼리는 공유되지만, 다른 컴퓨터나 팀원에게는 자동으로 전파되지 않는다. 팀 전체가 공유해야 할 지식은 사람이 검토해 CLAUDE.md로 승격시켜야 한다.

오해 5

"AGENTS.md를 저장소에 두면 Claude Code가 자동으로 그 파일도 읽는다."

실제

Claude Code는 CLAUDE.md만 읽고 AGENTS.md는 직접 읽지 않는다. 이미 AGENTS.md를 쓰는 저장소라면 CLAUDE.md에서 @AGENTS.md로 import하거나 심볼릭 링크를 만들어 두 도구가 같은 지침을 공유하게 해야 한다.

오해 6

"additionalDirectories로 형제 디렉토리에 접근 권한을 주면 그 디렉토리의 CLAUDE.md/skills도 당연히 로드된다."

실제

additionalDirectories 설정은 파일 접근 권한만 부여하고 CLAUDE.md/rules/skills는 전혀 로드하지 않는다. --add-dir//add-dir은 skills는 로드하지만 CLAUDE.md/rules는 별도 환경 변수를 켜야 로드된다. "접근 권한"과 "컨텍스트 로드"는 별개로 제어되는 축이다.
