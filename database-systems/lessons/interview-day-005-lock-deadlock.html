💬 면접 질문

Core Question

"운영 중 deadlock이 발생하면 어떻게 분석하고 줄입니까?"

함께 출제되는 변형 질문

  • "lock wait timeout과 deadlock의 차이를 설명해주세요."
  • "공유 락(shared lock)과 배타 락(exclusive lock)은 어떻게 다르고, 언제 서로 충돌합니까?"
  • "같은 UPDATE 문인데 어떤 날은 빠르고 어떤 날은 lock 대기가 길어집니다. 무엇을 먼저 확인하겠습니까?"
  • "deadlock을 애초에 발생시키지 않으려면 애플리케이션 코드를 어떻게 작성해야 합니까?"

🎯 왜 면접에서 묻는가

이 질문은 "deadlock이 뭔지 정의를 아는가"가 아니라 운영 중 실제로 로그를 열람하고, lock이 걸린 순서를 재구성하고, 근본 원인을 코드 레벨에서 고쳐낼 수 있는가를 확인하기 위한 것입니다.


  • 장애 대응 역량: deadlock은 새벽에 알림으로 터지는 대표적인 운영 이슈입니다. 로그를 읽고 몇 분 안에 "어느 트랜잭션들이 어떤 순서로 무엇을 잠갔는지"를 재구성할 수 있는지를 봅니다.
  • 근본 원인 vs 임시 처방 구분: 주니어는 "재시도 로직을 추가하면 된다"고 답하지만, 시니어는 재시도는 증상 완화일 뿐이고 lock 획득 순서 불일치 같은 근본 원인을 찾아 고치려 합니다.
  • lock 내부 동작 이해: 인덱스 설계, WHERE 조건, 격리 수준이 실제로 몇 개의 행/갭에 lock을 거는지 알아야 deadlock 발생 지점을 예측하고 설명할 수 있습니다. (Day 3 인덱스, Day 4 격리 수준과 직접 연결됩니다.)
  • 재발 방지 설계력: deadlock 자체보다, 같은 클래스의 deadlock이 다시 발생하지 않도록 팀 전체의 접근 순서 규칙이나 코드 리뷰 기준을 만들 수 있는지를 봅니다.

📚 전제 개념

Lock이 왜 필요한가 — 쉬운 말로

Day 4에서 다룬 MVCC 덕분에 일반적인 SELECT(읽기)는 lock을 거의 잡지 않습니다. 하지만 같은 행을 동시에 수정하려는 두 트랜잭션은 여전히 순서를 정해야 합니다. 두 사람이 동시에 같은 문서를 편집하면 안 되는 것과 같은 이유입니다. Lock은 "이 행(또는 이 범위)은 지금 내가 쓰고 있으니, 내가 끝날 때까지 다른 트랜잭션은 기다리라"는 표시입니다.

공유 락(Shared Lock, S) vs 배타 락(Exclusive Lock, X)

InnoDB, PostgreSQL 등 대부분의 RDB는 lock을 두 종류로 나눕니다.

  • 공유 락(S): "나는 이 행을 읽기만 할 거야"라는 표시. 다른 트랜잭션도 동시에 공유 락을 잡고 읽을 수 있습니다(읽기끼리는 서로 막지 않음). SELECT ... FOR SHARE가 대표적입니다.
  • 배타 락(X): "나는 이 행을 수정할 거야"라는 표시. 다른 트랜잭션은 이 행에 공유 락도, 배타 락도 잡을 수 없습니다. UPDATE, DELETE, SELECT ... FOR UPDATE가 배타 락을 겁니다.
이미 걸린 락 \ 요청하는 락공유 락(S) 요청배타 락(X) 요청
공유 락(S) 보유 중허용(같이 읽기 가능)대기
배타 락(X) 보유 중대기대기

즉 "읽기 vs 읽기"만 서로 막지 않고, "쓰기가 하나라도 끼면" 반드시 대기가 발생합니다. deadlock은 이 대기 관계가 서로 맞물릴 때 생깁니다.

Lock의 범위 — Row Lock, Gap Lock, Next-key Lock, Intention Lock

lock은 "행 하나"에만 걸리는 것이 아니라 범위와 계층에 따라 여러 형태로 걸립니다. Day 3(인덱스)에서 배운 인덱스 구조와 직접 연결됩니다.

  • Record Lock(행 락): 인덱스의 특정 행 하나에 거는 락. WHERE 조건이 인덱스로 정확히 한 행을 특정할 때 이 락만 걸립니다.
  • Gap Lock(갭 락): 인덱스 값 "사이의 빈 공간"에 거는 락. 새로운 행이 그 범위에 삽입되는 것을 막습니다(Day 4에서 다룬 phantom read 방지 메커니즘).
  • Next-key Lock: Record Lock + 그 앞의 Gap Lock을 합친 것. InnoDB REPEATABLE READ의 기본 락 방식입니다.
  • Intention Lock(의도 락): "나는 이 테이블 안의 어떤 행들에 락을 걸 예정"임을 테이블 레벨에 미리 표시하는 락. 테이블 전체에 락을 걸려는 다른 트랜잭션(예: LOCK TABLES, DDL)이 행 단위로 일일이 확인하지 않고도 충돌 여부를 빠르게 판단하게 해줍니다.
주의: WHERE 조건에 인덱스가 없으면 DB는 조건에 맞는 행을 찾기 위해 스캔한 모든 행(과 그 갭)에 락을 걸 수 있습니다. "느린 UPDATE 하나가 관련 없어 보이는 다른 트랜잭션까지 멈추게 하는" 흔한 원인이 바로 이것입니다.

Lock Wait Timeout vs Deadlock — 무엇이 다른가

둘 다 "트랜잭션이 서로 기다리다 실패한다"는 점은 같지만 원인과 처리 방식이 다릅니다.

  • Lock Wait Timeout: 트랜잭션 A가 트랜잭션 B가 가진 lock을 기다리는데, B가 끝나지 않아 정해진 시간(innodb_lock_wait_timeout 등)을 넘긴 경우. A가 "너무 오래 기다렸다"는 이유로 실패합니다. B는 정상적으로 진행 중일 수 있습니다.
  • Deadlock: 트랜잭션 A는 B가 가진 lock을, B는 A가 가진 lock을 서로 기다리는 순환 대기(circular wait) 상태. 시간이 아무리 지나도 둘 다 끝날 수 없으므로, DB는 대기를 기다리지 않고 즉시 한쪽을 골라 강제로 롤백시킵니다(victim 선정).
실무 팁: MySQL InnoDB는 wait-for graph(누가 누구를 기다리는지 나타낸 그래프)에서 순환을 발견하면 그 즉시 deadlock으로 판단하고, 일반적으로 "되돌릴 작업량이 더 적은(undo log가 더 짧은) 트랜잭션"을 victim으로 선택해 롤백시킵니다. PostgreSQL은 주기적으로(deadlock_timeout 간격) 순환을 검사하는 방식입니다.

Deadlock이 실제로 생기는 전형적 패턴 — 순서가 다른 두 트랜잭션

가장 흔한 deadlock 패턴은 두 트랜잭션이 같은 두 행을 서로 다른 순서로 잠그려 할 때입니다.

-- 트랜잭션 A
UPDATE accounts SET balance = balance - 100 WHERE id = 1; -- id=1 X lock 획득
UPDATE accounts SET balance = balance + 100 WHERE id = 2; -- id=2 X lock 대기 (B가 보유 중)

-- 트랜잭션 B (거의 동시에 실행)
UPDATE accounts SET balance = balance - 50 WHERE id = 2;  -- id=2 X lock 획득
UPDATE accounts SET balance = balance + 50 WHERE id = 1;  -- id=1 X lock 대기 (A가 보유 중) → 순환 대기 → Deadlock

A는 id=1을 잠근 채 id=2를 기다리고, B는 id=2를 잠근 채 id=1을 기다리므로 서로 영원히 풀리지 않는 순환이 생깁니다. 계좌 이체처럼 "두 행 이상을 갱신하는 로직"에서 접근 순서가 코드마다 다르면 거의 반드시 이 패턴을 만납니다.

🏭 실무 연결

시나리오: 계좌 이체 — 접근 순서 불일치로 인한 반복 deadlock

송금 API는 "보내는 계좌 → 받는 계좌" 순서로 UPDATE를 실행하도록 짰지만, 정산 배치는 계좌 ID 순서와 무관하게 조회된 순서대로 처리하도록 짜여 있었습니다. 트래픽이 적을 때는 우연히 순서가 겹치지 않아 문제가 없었지만, 트래픽이 늘면서 하루 수십 건씩 deadlock 에러 로그가 쌓이기 시작했습니다. 해결책은 모든 다중 행 갱신 로직에서 "행 ID 오름차순으로 정렬 후 잠그기"라는 공통 규칙을 정하고, 정산 배치 코드를 그 규칙에 맞춰 수정한 것이었습니다.

시나리오: 인덱스 누락으로 인한 광범위 gap lock

UPDATE orders SET status = 'CANCELLED' WHERE customer_id = ? AND created_at > ? 쿼리에서 created_at에 인덱스가 없어 DB가 많은 행을 스캔하며 그 과정에서 여러 행과 갭에 lock을 걸었습니다. 평소엔 짧게 끝나 문제가 안 보이다가, 배치 작업이 겹치는 시간대에 서로 다른 고객의 주문 갱신 트랜잭션들이 이 넓은 lock 범위 안에서 충돌하며 deadlock과 lock wait timeout이 동시에 폭증했습니다. (customer_id, created_at) 복합 인덱스를 추가해 스캔 범위를 좁히자 lock 경합이 크게 줄었습니다. Day 3에서 다룬 "인덱스 설계가 곧 lock 설계"라는 관점이 여기서 실전으로 이어집니다.

시나리오: 긴 트랜잭션 안의 외부 호출 — lock을 쥔 채 대기

재고 차감 트랜잭션 안에서 SELECT ... FOR UPDATE로 행을 잠근 뒤, 커밋 전에 외부 결제 API를 동기 호출하도록 구현되어 있었습니다. 결제 API 응답이 느려지자 그 트랜잭션이 재고 행을 잠근 채 몇 초씩 대기했고, 같은 상품을 주문하려는 다른 요청들이 줄줄이 대기(lock convoy)하다가 일부는 timeout으로 실패했습니다. 원칙은 lock을 잡은 트랜잭션 안에서 네트워크 호출이나 느린 연산을 하지 않는 것이며, 외부 호출은 lock을 커밋/해제한 이후로 옮겨야 합니다.

🔍 답변 사고 순서

Deadlock 질문은 "무슨 뜻인지 아는가"가 아니라, 실제 장애가 터졌을 때 어떤 순서로 로그를 보고 원인을 좁혀가는지 그 절차를 보여주는 것이 핵심입니다.
  1. 1단계 — 증상 확인: deadlock인지 lock wait timeout인지 구분

    에러 메시지와 로그(MySQL SHOW ENGINE INNODB STATUS의 LATEST DETECTED DEADLOCK, PostgreSQL pg_stat_activity/로그)를 확인해 순환 대기인지 단순 대기 초과인지 먼저 구분한다.

  2. 2단계 — 관련된 트랜잭션들의 SQL과 락 순서 재구성

    deadlock 로그에 남은 각 트랜잭션의 실행 SQL과 보유/대기 중이던 lock을 시간 순서로 나열해 "누가 무엇을 먼저 잠그고 무엇을 기다렸는지" 그림을 그린다.

  3. 3단계 — 접근 순서 불일치 여부 확인

    같은 두 개 이상의 행/테이블을 서로 다른 순서로 갱신하는 코드 경로가 있는지 애플리케이션 코드에서 찾는다. 이것이 가장 흔한 근본 원인이다.

  4. 4단계 — lock 범위가 의도보다 넓은지 확인

    WHERE 조건에 걸맞은 인덱스가 있는지, 없다면 얼마나 많은 행/갭에 불필요하게 락이 걸렸는지 EXPLAIN으로 확인한다.

  5. 5단계 — 트랜잭션 지속 시간 확인

    lock을 잡은 트랜잭션 안에 외부 API 호출, 느린 연산, 불필요하게 넓은 범위의 작업이 섞여 있는지 확인한다.

  6. 6단계 — 근본 원인에 맞는 수정 적용

    접근 순서 통일, 인덱스 추가, 트랜잭션 범위 축소, 필요 시 배타적 잠금 대신 낙관적 락으로 전환 등 원인에 맞는 구조적 수정을 적용한다.

  7. 7단계 — 애플리케이션 레벨 재시도로 보완

    deadlock은 트래픽이 늘면 구조적으로 완전히 제거하기 어려울 수 있으므로, 근본 수정과 별개로 deadlock 에러(예: MySQL 1213)를 감지해 짧은 backoff 후 자동 재시도하는 안전망을 둔다.

💡 핵심 답변 예시

짧은 버전 (1분)

"먼저 로그로 deadlock인지 단순 lock wait timeout인지부터 구분합니다. deadlock이면 로그에 남은 각 트랜잭션의 SQL과 lock 순서를 재구성해서, 같은 행들을 서로 다른 순서로 갱신하는 코드 경로가 있는지 확인합니다. 실무에서는 대부분 이 접근 순서 불일치가 원인이라, 모든 다중 행 갱신 로직에 'ID 오름차순으로 잠그기' 같은 공통 규칙을 정해 근본적으로 줄입니다. 동시에 인덱스가 없어 lock 범위가 불필요하게 넓어진 건 아닌지도 함께 확인하고, 그래도 남는 낮은 빈도의 deadlock은 애플리케이션에서 짧은 backoff 후 재시도하도록 방어합니다."

깊이 있는 버전 (3~4분)

"deadlock 분석은 '재시도 로직을 넣으면 되지 않느냐'는 임시 처방에서 멈추지 않고, 근본 원인을 코드 레벨에서 찾는 것을 목표로 합니다.

첫째, deadlock인지 단순 lock wait timeout인지 구분합니다. lock wait timeout은 순환 대기가 아니라 한쪽이 오래 걸려 생긴 것이라 원인이 다릅니다. deadlock이면 MySQL 기준 SHOW ENGINE INNODB STATUS의 LATEST DETECTED DEADLOCK 섹션에서 관련된 두 트랜잭션의 SQL, 보유 lock, 대기 lock을 확인합니다.

둘째, 그 정보로 순환 대기 그래프를 재구성합니다. 대부분의 경우 두 트랜잭션이 같은 두 개 이상의 행을 서로 다른 순서로 갱신하려다 발생한 것을 확인할 수 있습니다. 계좌 이체처럼 여러 행을 갱신하는 로직이라면, 코드 전체에서 '행을 잠그는 순서'가 일관되게 정렬되어 있는지 점검합니다.

셋째, lock 범위가 의도보다 넓어지진 않았는지 확인합니다. WHERE 조건에 맞는 인덱스가 없으면 스캔한 행과 갭 전체에 lock이 걸려, 원래는 무관해야 할 트랜잭션끼리도 충돌할 수 있습니다. EXPLAIN으로 실행 계획을 확인하고 필요하면 인덱스를 추가합니다.

넷째, lock을 쥔 상태로 오래 머무는 코드가 있는지 봅니다. 트랜잭션 안에서 외부 API 호출이나 느린 연산이 섞여 있으면 lock 보유 시간이 길어져 deadlock과 lock convoy(대기 행렬) 위험이 함께 커집니다.

마지막으로, 이 모든 구조적 수정 이후에도 남는 낮은 빈도의 deadlock에는 재시도 안전망을 둡니다. deadlock은 트래픽 패턴에 따라 완전히 0으로 만들기 어려울 수 있으므로, 이를 정상적으로 발생 가능한 상황으로 인정하고 감지 → 짧은 backoff → 재시도 흐름을 애플리케이션에 갖추는 것을 마지막 방어선으로 둡니다."

⚖️ Trade-off

배타 락(SELECT FOR UPDATE) vs 낙관적 락

  • 배타 락 이득: 충돌을 사전에 원천 차단, 로직이 단순함
  • 배타 락 비용: lock 대기 시간 발생, 여러 행을 잠그는 순서가 어긋나면 deadlock 위험
  • 낙관적 락 이득: lock 없이 동시 처리량을 높게 유지, deadlock 자체가 구조적으로 발생하지 않음
  • 낙관적 락 비용: 충돌 시 재시도 로직을 애플리케이션이 직접 구현해야 함
  • 판단 기준: 다중 행을 함께 갱신해야 하는 로직(계좌 이체)은 배타 락 + 접근 순서 통일, 단일 행 갱신 위주 로직은 낙관적 락으로 deadlock 위험 자체를 줄이는 편이 유리.

넓은 lock 범위(테이블/큰 갭) vs 좁은 lock 범위(정확한 행)

  • 넓은 범위 이득: 락 관리 오버헤드가 적고 구현이 단순함(예: 배치 작업에서 테이블 락)
  • 넓은 범위 비용: 관련 없는 트랜잭션까지 대기시켜 처리량이 급격히 떨어지고 deadlock 표면적이 커짐
  • 판단 기준: 인덱스를 통해 lock 범위를 필요한 행으로 최소화하는 것이 기본 전략. 배치처럼 대량 갱신이 불가피한 작업은 트래픽이 적은 시간대로 분리하거나 작은 단위로 쪼갠다.

즉시 재시도 vs 지수 백오프 재시도

  • 즉시 재시도 이득: 구현이 단순하고 지연이 거의 없음
  • 즉시 재시도 비용: deadlock을 유발한 조건(동시 접근 패턴)이 그대로 남아 있으면 같은 실패가 반복될 확률이 높음
  • 지수 백오프 이득: 재시도 시점을 분산시켜 같은 충돌이 반복될 확률을 낮춤
  • 판단 기준: deadlock은 순간적인 경합이므로 짧은 무작위 지연(jitter)을 섞은 재시도가 안전하며, 재시도 횟수 상한을 반드시 둔다.

🚫 흔한 오해

Deadlock은 DB 버그이거나 운이 나빠서 생기는 것이다.
대부분의 deadlock은 애플리케이션이 여러 행을 서로 다른 순서로 잠그도록 짜여 있어서 생기는, 구조적으로 예측 가능한 문제입니다. DB는 이를 감지해 한쪽을 롤백시켜 시스템이 멈추지 않게 보호하는 정상 동작을 한 것뿐입니다.
재시도 로직만 넣으면 deadlock 문제는 해결된 것이다.
재시도는 증상을 완화할 뿐 원인을 없애지 않습니다. 트래픽이 늘면 재시도로도 감당 못 할 만큼 deadlock 빈도가 늘 수 있고, 재시도 자체가 추가 부하를 만듭니다. 접근 순서 통일이나 인덱스 개선 같은 근본 조치를 우선해야 합니다.
읽기(SELECT)는 lock을 안 거니 deadlock과 무관하다.
일반 SELECT는 대부분 무관하지만, SELECT ... FOR UPDATE나 SELECT ... FOR SHARE처럼 명시적으로 lock을 요청하는 조회는 다른 트랜잭션과 똑같이 lock 경합과 deadlock에 관여합니다. "SELECT라서 안전하다"고 단정하면 안 됩니다.
격리 수준을 낮추면 deadlock이 줄어든다.
격리 수준은 "읽기가 무엇을 볼 수 있는가"를 결정하는 것이지, 쓰기 계열 작업의 배타 락 자체를 없애주지 않습니다. 격리 수준을 낮춰도 두 트랜잭션이 같은 행을 다른 순서로 갱신하면 여전히 deadlock이 발생합니다. 다만 InnoDB에서 gap lock의 범위는 격리 수준에 따라 달라지므로(예: READ COMMITTED는 REPEATABLE READ보다 gap lock을 덜 사용), 어느 정도 영향을 줄 수는 있습니다.
인덱스를 추가하면 무조건 lock 경합이 줄어든다.
적절한 인덱스는 스캔 범위를 좁혀 lock 경합을 줄이는 경우가 많지만, 인덱스 선택도(index selectivity)가 낮거나 잘못된 컬럼에 인덱스를 걸면 오히려 예상치 못한 gap lock 패턴이 생길 수 있습니다. 인덱스 설계는 반드시 실행 계획(EXPLAIN)으로 실제 lock 범위 축소 효과를 확인해야 합니다.

📊 좋은 답변 vs 약한 답변

강한 답변

"먼저 로그로 순환 대기(deadlock)인지 단순 lock wait timeout인지 구분합니다. deadlock이면 관련 트랜잭션들의 SQL과 lock 순서를 재구성해서 같은 행을 다른 순서로 갱신하는 코드 경로가 있는지 확인하고, 있다면 접근 순서를 통일합니다. 동시에 인덱스가 없어 lock 범위가 불필요하게 넓어진 건 아닌지 EXPLAIN으로 확인하고, 그래도 남는 저빈도 deadlock은 짧은 backoff 재시도로 방어합니다."

왜 강한가: deadlock과 lock wait timeout을 구분하고, 근본 원인(접근 순서, lock 범위)을 코드/인덱스 레벨에서 좁혀가며, 재시도를 근본 해결책이 아니라 마지막 방어선으로 위치시킵니다.

약한 답변

"deadlock이 발생하면 그 트랜잭션을 재시도하도록 처리하면 됩니다. DB가 알아서 하나를 롤백시켜주니까요."

왜 약한가: DB의 방어 동작(victim 롤백)과 재시도를 "해결책"으로 착각하고 있으며, 왜 deadlock이 반복되는지 원인 분석 과정이 전혀 없습니다.

🎓 Senior-level Follow-up 질문

Q. MySQL은 deadlock을 어떻게 즉시 감지합니까? PostgreSQL과 무엇이 다릅니까?
의도: "DB가 알아서 처리해준다"는 결론만 아는 게 아니라, wait-for graph 기반 감지와 timeout 기반 감지의 차이를 구조적으로 이해하는지 확인합니다.
모범 사고 경로: InnoDB는 각 트랜잭션이 누구의 lock을 기다리는지 wait-for graph를 실시간으로 유지하다가, 새로운 대기가 이 그래프에 순환을 만드는 순간 즉시 deadlock으로 판정하고 그 자리에서 victim을 롤백시킵니다. 반면 PostgreSQL은 매 대기마다 그래프를 갱신하는 대신 deadlock_timeout(기본 1초) 동안 기다린 뒤에도 여전히 막혀 있으면 그때 순환 여부를 검사합니다. 그래서 PostgreSQL의 deadlock 감지는 MySQL보다 약간 지연될 수 있습니다.
Q. victim으로 선택되는 트랜잭션은 어떤 기준으로 정해집니까? 이걸 애플리케이션이 통제할 수 있습니까?
의도: deadlock 처리를 "블랙박스"로 받아들이는지, 아니면 그 기준을 알고 재시도/우선순위 설계에 반영할 수 있는지 확인합니다.
모범 사고 경로: InnoDB는 일반적으로 되돌릴 작업량(undo log 양)이 더 적은, 즉 "더 최근에 시작했거나 변경량이 적은" 트랜잭션을 victim으로 선택해 롤백 비용을 최소화하려 합니다. 애플리케이션이 이를 직접 제어하긴 어렵지만, 트랜잭션을 짧게 유지하면 자신이 victim이 되어도 손실이 적고, 모든 클라이언트가 deadlock 에러를 감지해 재시도하도록 일관되게 구현해두면 어느 쪽이 희생되어도 시스템 전체는 정상 동작합니다.
Q. deadlock이 발생하지 않는데도 특정 API가 간헐적으로 느립니다. lock과 관련이 있을 수 있습니까?
의도: deadlock만이 lock 문제의 전부라고 생각하는지, lock wait(대기)로 인한 지연도 같은 근본 원인 분석 대상으로 볼 수 있는지 확인합니다.
모범 사고 경로: 순환 대기가 아니어도 특정 행에 lock이 오래 걸려 있으면 그 행을 필요로 하는 다른 요청들이 줄줄이 대기(lock convoy)하며 지연될 수 있습니다. deadlock 로그에는 안 잡히지만 SHOW ENGINE INNODB STATUS의 lock wait 정보나 performance_schema, PostgreSQL의 pg_locks/pg_stat_activity로 "누가 무엇을 오래 기다리는지" 확인해야 합니다. 원인은 대개 lock을 쥔 채 오래 머무는 트랜잭션(외부 호출, 느린 연산 포함)입니다.
Q. 애플리케이션 코드에서 "행 ID 오름차순으로 잠그기" 규칙을 지키기 어려운 경우(예: ID가 아닌 조건으로 여러 행을 갱신)에는 어떻게 접근 순서를 통일합니까?
의도: "정렬해서 잠그라"는 규칙을 단순 암기했는지, 그 규칙이 적용되기 어려운 실제 코드에서 대안을 설계할 수 있는지 확인합니다.
모범 사고 경로: 조건으로 여러 행을 갱신해야 한다면, 먼저 SELECT ... FOR UPDATE로 대상 행들을 조회해 ID 기준으로 정렬한 뒤, 그 정렬된 순서대로 갱신 문을 실행하도록 코드를 재구성합니다. 이것이 어렵다면 애플리케이션 레벨에서 리소스별 논리적 락(advisory lock)이나 큐를 통해 같은 자원에 대한 접근을 직렬화하는 방법도 검토할 수 있지만, 이는 별도의 복잡도를 추가하므로 정렬 잠금으로 해결되지 않을 때의 차선책으로 접근합니다.
