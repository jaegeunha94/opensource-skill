💬 면접 질문

Core Question

"복합 인덱스 순서는 어떤 기준으로 정합니까?"

함께 출제되는 변형 질문

  • "인덱스를 걸었는데 옵티마이저가 안 탑니다. 왜 그럴 수 있습니까?"
  • "인덱스를 추가했더니 조회는 빨라졌는데 쓰기가 느려졌습니다. 어떻게 판단하시겠습니까?"
  • "WHERE와 ORDER BY가 함께 있는 쿼리는 인덱스를 어떻게 설계합니까?"
  • "covering index가 뭔지, 언제 쓰는지 설명해주세요."

🎯 왜 면접에서 묻는가

이 질문은 "인덱스를 걸 줄 아는가"가 아니라 인덱스 하나하나가 읽기/쓰기 비용의 균형점을 바꾼다는 것을 이해하고 설계하는가를 확인하기 위한 것입니다.


  • 자료구조 이해: B+Tree가 왜 정렬된 구조이고, 왜 "왼쪽부터" 순서대로 값을 찾을 수 있는지 이해해야 복합 인덱스 순서를 설명할 수 있습니다. 이걸 모르면 "그냥 자주 쓰는 컬럼을 앞에 둔다"는 감으로만 답하게 됩니다.
  • 비용 감각: 주니어는 "느리면 인덱스 추가"로 끝내지만, 시니어는 인덱스가 쓰기마다 갱신되는 자료구조라는 것을 알고 "이 인덱스가 얻는 조회 이득이 잃는 쓰기 비용보다 큰가"를 판단합니다.
  • 운영 판단력: 인덱스가 너무 많으면 INSERT/UPDATE가 느려지고 저장 공간도 늘어납니다. 어떤 인덱스를 남기고 어떤 인덱스를 제거할지 판단하는 것 자체가 시니어의 역할입니다.
  • 실행 계획 연결력: 인덱스를 만들어도 옵티마이저가 안 탈 수 있습니다. 이 질문은 결국 다음 Day인 EXPLAIN 해석 능력과 이어지는 기초 체력을 확인합니다.

📚 전제 개념

인덱스는 왜 빠른가? — B+Tree를 쉬운 말로

인덱스가 없으면 DB는 원하는 행을 찾기 위해 테이블 전체를 처음부터 끝까지 훑어야 합니다(Full Table Scan). 인덱스는 특정 컬럼 값을 미리 정렬해서 별도의 자료구조에 저장해두고, 그 값을 빠르게 찾을 수 있는 "책의 색인"입니다.


대부분의 RDB 인덱스는 B+Tree 구조를 씁니다. 핵심 특징은 세 가지입니다.

  • 정렬되어 있다: 리프 노드(맨 아래 층)에 있는 값들이 오름차순으로 정렬되어 있습니다. 그래서 WHERE age > 30처럼 범위 검색을 할 때도, 값을 하나 찾은 다음 옆으로 쭉 읽기만 하면 됩니다.
  • 균형 트리다: 어떤 값을 찾든 루트에서 리프까지 이동하는 깊이가 거의 같습니다. 데이터가 100만 건이든 1억 건이든, 트리 깊이는 로그 스케일로만 늘어나서 탐색 비용이 거의 일정하게 유지됩니다.
  • 리프 노드가 서로 연결되어 있다: 리프 노드끼리 링크드 리스트처럼 연결돼 있어서, 범위 검색이나 정렬된 결과를 순차적으로 읽는 데 유리합니다. ORDER BY가 인덱스 순서와 일치하면 별도 정렬 없이 리프를 그대로 읽으면 됩니다.

그리고 반드시 기억할 것: 인덱스는 공짜가 아닙니다. INSERT/UPDATE/DELETE가 일어날 때마다 인덱스도 함께 갱신해야 하므로, 인덱스가 많을수록 쓰기 비용이 커집니다.

Selectivity(선택도)란?

selectivity는 "이 컬럼 값으로 얼마나 데이터를 좁힐 수 있는가"를 나타냅니다. 고유값 개수 / 전체 행 수로 계산하며, 1에 가까울수록(고유값이 많을수록) selectivity가 높습니다.


  • selectivity가 높은 예: 주민등록번호, 이메일 — 값 하나가 거의 한 행만 가리킴. 인덱스 효율이 매우 좋음.
  • selectivity가 낮은 예: 성별, 상태값(활성/비활성) — 값 하나가 테이블의 절반, 또는 대부분을 가리킴. 인덱스를 걸어도 옵티마이저가 Full Scan을 선택할 수 있음(전체의 30~40% 이상을 읽어야 한다면 인덱스로 한 건씩 찾아가는 것보다 순차적으로 다 읽는 게 더 빠를 수 있기 때문).

복합 인덱스를 설계할 때 selectivity가 높은 컬럼을 일반적으로 앞쪽에 두면 초반에 후보 행을 크게 줄일 수 있습니다. 다만 이는 "항상 옳은 규칙"이 아니라 실제 쿼리 패턴과 함께 고려해야 하는 기준 중 하나입니다(아래 Leftmost Prefix 참고).

Leftmost Prefix Rule (최좌측 접두 규칙)

복합 인덱스 (a, b, c)는 컬럼 a를 기준으로 먼저 정렬하고, 같은 a 값 안에서 b로 정렬하고, 같은 a, b 안에서 c로 정렬한 하나의 구조입니다. 전화번호부가 "성 → 이름" 순으로 정렬되어 있으면 성으로는 바로 찾을 수 있지만, 이름만으로는 전화번호부를 처음부터 훑어야 하는 것과 같은 원리입니다.


그래서 이 인덱스는 다음 조건에서만 온전히 활용됩니다:

  • WHERE a = ? → 사용 가능
  • WHERE a = ? AND b = ? → 사용 가능
  • WHERE a = ? AND b = ? AND c = ? → 완전히 사용 가능
  • WHERE b = ? (a 없이 b만) → 기본적으로 사용 불가 (인덱스를 처음부터 스캔해야 함)
  • WHERE a = ? AND c = ? (중간 b 생략) → a까지만 활용, c는 인덱스 내에서 필터링만 가능(범위 탐색 자체는 못 좁힘)

이 규칙 때문에 복합 인덱스 순서를 정할 때는 "이 인덱스를 어떤 쿼리들이 쓸 것인가"를 먼저 나열하고, 그 쿼리들의 WHERE 조건에서 공통으로 등호(=)로 쓰이는 컬럼을 앞에 두는 것이 selectivity보다 우선하는 경우가 많습니다.

Covering Index(커버링 인덱스)란?

쿼리가 요구하는 모든 컬럼(SELECT, WHERE, ORDER BY에 쓰인 컬럼)이 인덱스 자체에 다 포함되어 있으면, DB는 실제 테이블 데이터(원본 행)를 읽지 않고 인덱스만 읽고 응답할 수 있습니다. 이를 "인덱스만으로 커버된다"고 해서 covering index라고 부릅니다.


일반적으로 인덱스에서 조건에 맞는 위치를 찾은 뒤, 실제 컬럼 값을 읽으려면 원본 테이블의 해당 행을 다시 찾아가야 합니다(이 과정을 흔히 "북마크 조회" 또는 InnoDB 기준으로는 2차 인덱스 → 클러스터드 인덱스 조회라고 부릅니다). 이 추가 조회가 랜덤 I/O라서 비용이 큽니다. covering index는 이 추가 조회 자체를 없애버립니다.

실무 팁: 자주 실행되는 조회 쿼리에서 SELECT절 컬럼 수가 적다면, WHERE 조건 컬럼 뒤에 조회하는 컬럼을 추가로 포함시켜 covering index로 만드는 것이 효과적인 튜닝 기법입니다.

클러스터드 인덱스 vs 세컨더리(보조) 인덱스

클러스터드 인덱스는 테이블의 실제 데이터 행 자체가 인덱스(보통 PK) 순서대로 물리적으로 정렬되어 저장되는 방식입니다(MySQL InnoDB가 대표적). 테이블당 하나만 가능합니다. 세컨더리 인덱스는 그 외의 인덱스로, 리프 노드에 실제 데이터 대신 "이 값을 가진 행은 PK가 얼마다"라는 포인터(InnoDB 기준으로는 PK 값 자체)를 저장합니다.


그래서 세컨더리 인덱스로 조회하면 리프에서 PK를 얻은 뒤, 그 PK로 클러스터드 인덱스를 한 번 더 타서 실제 데이터를 가져옵니다. 이것이 covering index가 필요한 이유입니다 — covering index는 이 "한 번 더 타는" 과정을 생략시켜줍니다.

쓰기 비용: 인덱스는 왜 "많을수록 좋다"가 아닌가

INSERT 한 번은 테이블에 값을 추가하는 것뿐 아니라, 그 테이블에 걸린 모든 인덱스에도 값을 추가하는 작업입니다. 인덱스가 5개면 INSERT 한 번에 최대 5개의 B+Tree를 갱신해야 하고, 트리 재조정(페이지 분할 등)이 일어날 수 있어 쓰기 지연이 커집니다. UPDATE도 변경된 컬럼이 인덱스에 포함되어 있으면 해당 인덱스를 함께 갱신해야 합니다.

🏭 실무 연결

시나리오: 주문 목록 API — "특정 유저의 최근 주문 조회"

쿼리가 WHERE user_id = ? ORDER BY created_at DESC LIMIT 20 형태라면, 복합 인덱스 (user_id, created_at)를 고려합니다. user_id로 먼저 좁히고, 같은 user_id 안에서는 created_at으로 이미 정렬되어 있으므로 별도 정렬 없이 리프를 뒤에서부터(또는 DESC 인덱스 방향으로) 읽기만 하면 됩니다. 만약 인덱스를 (created_at, user_id) 순서로 걸면 created_at 전체를 훑으면서 user_id를 걸러야 해서 selectivity 이득을 못 봅니다.

시나리오: 관리자 검색 화면 — 여러 조건 조합 검색

상태값, 날짜 범위, 담당자 등 여러 조건을 관리자가 자유롭게 조합해서 검색하는 화면은 인덱스 설계가 까다롭습니다. 모든 조합에 맞는 인덱스를 다 만들면 인덱스 개수가 폭발적으로 늘어나서 쓰기 성능이 크게 저하됩니다.

  • 실제 로그에서 가장 많이 쓰이는 조합 2~3개를 추려서 복합 인덱스를 설계
  • 드물게 쓰이는 조합은 Full Scan을 감수하거나, 검색 전용 저장소(Elasticsearch 등)로 분리
  • 관리자 화면처럼 트래픽이 낮고 지연에 덜 민감한 경우, 모든 조건에 완벽한 인덱스를 만드는 것보다 "허용 가능한 지연"을 먼저 합의하는 것이 현실적

시나리오: 인덱스를 걸었는데도 느린 경우

컬럼에 LOWER(email) = ?처럼 함수를 씌우거나, LIKE '%keyword%'처럼 앞에 와일드카드를 쓰면 일반 B+Tree 인덱스를 탈 수 없습니다(정렬된 순서를 활용할 수 없기 때문). 이럴 때는 함수 기반 인덱스(표현식 인덱스)를 만들거나, 검색 요구사항 자체를 전문 검색(full-text search) 엔진으로 옮기는 것을 검토합니다.

🔍 답변 사고 순서

인덱스 설계 질문은 "어떤 컬럼에 인덱스를 걸지"가 아니라, 그 인덱스를 실제로 사용할 쿼리 패턴부터 역산하는 사고 과정을 보여주는 것이 핵심입니다.
  1. 1단계 — 이 인덱스를 탈 쿼리 목록 나열

    어떤 쿼리(들)가 이 인덱스를 사용할 것인지 먼저 구체화한다. 하나의 쿼리만 보고 결정하지 않고, 같은 테이블에 자주 실행되는 쿼리들을 함께 고려한다.

  2. 2단계 — WHERE 등호 조건 컬럼 식별

    쿼리들에서 공통으로 등호(=)로 필터링하는 컬럼을 찾는다. 이 컬럼들이 복합 인덱스의 앞쪽 후보가 된다.

  3. 3단계 — Selectivity와 leftmost prefix 함께 고려

    등호 조건 컬럼이 여러 개면, selectivity가 높은 컬럼을 우선 배치하되, 여러 쿼리가 이 인덱스를 공유해야 한다면 leftmost prefix 규칙상 "가장 많은 쿼리에서 공통으로 쓰이는 컬럼"을 더 앞에 둘지 판단한다.

  4. 4단계 — 정렬(ORDER BY) 요구사항 반영

    쿼리에 ORDER BY가 있다면, 등호 조건 컬럼 뒤에 정렬 기준 컬럼을 이어 붙여서 별도 정렬(filesort) 없이 인덱스 순서 그대로 결과를 낼 수 있는지 확인한다.

  5. 5단계 — Covering index 가능성 검토

    SELECT절에서 조회하는 컬럼 수가 적다면, 그 컬럼들을 인덱스에 포함시켜 테이블 재조회(북마크 조회) 자체를 없앨 수 있는지 검토한다.

  6. 6단계 — 쓰기 비용과 기존 인덱스 중복 확인

    이 테이블의 쓰기 빈도를 확인하고, 새 인덱스가 기존 인덱스와 겹치거나(예: 기존 인덱스의 leftmost prefix로 이미 커버되는 경우) 불필요한 중복은 아닌지 점검한다.

  7. 7단계 — 실제 데이터로 검증

    실제(또는 유사 규모의) 데이터로 EXPLAIN을 실행해서 예상한 인덱스가 실제로 선택되는지, rows 예측치와 실제 실행 시간이 기대한 대로인지 확인한다. 추측으로 끝내지 않는다.

💡 핵심 답변 예시

짧은 버전 (1분)

"복합 인덱스 순서는 selectivity만 보고 정하지 않습니다. 먼저 이 인덱스를 탈 쿼리들의 WHERE 절에서 등호로 비교되는 컬럼이 뭔지 확인하고, 그 컬럼들을 앞에 둡니다. 여러 컬럼이 등호 조건이면 그중 selectivity가 높은 컬럼을 우선 배치합니다. 그리고 ORDER BY가 있으면 그 정렬 컬럼을 뒤에 이어 붙여서 별도 정렬 비용을 없앨 수 있는지 확인합니다. 마지막으로 SELECT절 컬럼까지 인덱스에 포함시켜 covering index로 만들 수 있는지 보고, 쓰기 비용 대비 이득이 충분한지 EXPLAIN으로 검증합니다."

깊이 있는 버전 (3~4분)

"인덱스 설계는 '이 컬럼이 자주 조회된다'가 아니라 '이 인덱스가 실제로 어떤 쿼리에서, 어떻게 스캔될 것인가'를 기준으로 접근합니다.

첫째, B+Tree는 정렬된 구조이기 때문에 복합 인덱스는 leftmost prefix rule을 따릅니다. 예를 들어 (user_id, created_at) 인덱스가 있으면 WHERE user_id = ?는 인덱스를 탈 수 있지만 WHERE created_at = ?만으로는 인덱스 전체를 훑어야 합니다. 그래서 저는 이 인덱스를 사용할 쿼리들의 WHERE 절을 먼저 나열하고, 공통으로 등호 조건에 쓰이는 컬럼을 앞쪽에 배치합니다.

둘째, selectivity를 함께 고려합니다. 등호 조건 컬럼이 여러 개라면, 고유값이 많아 후보 행을 크게 줄여주는 컬럼을 앞에 두는 것이 일반적으로 유리합니다. 다만 여러 쿼리가 같은 인덱스를 재사용해야 하는 상황이면, selectivity보다 '여러 쿼리에서 공통으로 등호 조건에 쓰이는가'를 더 우선할 때도 있습니다.

셋째, ORDER BY와 covering index를 검토합니다. 정렬 요구사항이 있으면 등호 조건 뒤에 정렬 컬럼을 붙여서 filesort를 없애고, SELECT절 컬럼 수가 적다면 그 컬럼까지 인덱스에 포함시켜 테이블 재조회(북마크 조회) 자체를 생략시킵니다.

마지막으로, 이 모든 판단을 실제 데이터로 검증합니다. 인덱스는 쓰기 비용을 늘리는 트레이드오프이기 때문에, 조회 이득이 실제로 EXPLAIN과 실행 시간 측정에서 확인될 때만 인덱스를 추가하고, 기존 인덱스로 이미 커버되는 중복 인덱스는 정리합니다."

⚖️ Trade-off

인덱스 추가 vs 유지

  • 인덱스 추가 이득: 특정 조회 패턴의 응답 속도 대폭 개선, Full Scan 회피
  • 인덱스 추가 비용: INSERT/UPDATE/DELETE마다 인덱스 갱신 필요(쓰기 지연 증가), 저장 공간 증가, 옵티마이저의 실행 계획 선택 후보가 늘어나 통계 관리 부담 증가
  • 판단 기준: 읽기가 압도적으로 많은 테이블(카탈로그, 이력 조회)은 인덱스를 적극 활용. 쓰기가 매우 잦은 테이블(로그 적재, 카운터)은 인덱스를 최소한으로 유지.

Selectivity 우선 vs Leftmost Prefix 재사용 우선

  • Selectivity 우선: 단일 쿼리 관점에서는 가장 빠르게 후보를 좁히는 순서. 다만 이 인덱스를 다른 쿼리가 재사용하지 못할 수 있음.
  • 공통 컬럼 우선(재사용 우선): 인덱스 개수를 줄이고 쓰기 비용을 아낄 수 있음. 다만 특정 쿼리 입장에서는 이론상 최적 순서가 아닐 수 있음.
  • 판단 기준: 인덱스가 단일 핵심 쿼리 전용이면 selectivity 우선, 여러 쿼리가 동일 컬럼 조합을 공유하면 재사용 우선으로 인덱스 개수를 줄이는 쪽이 실무에서 더 자주 선택됨.

Covering Index로 넓히기 vs 인덱스 크기 유지

  • 컬럼을 더 포함(covering) 이득: 테이블 재조회 제거로 조회 속도 향상, 특히 랜덤 I/O가 비싼 디스크 기반 스토리지에서 효과 큼
  • 컬럼을 더 포함 비용: 인덱스 자체 크기 증가 → 메모리(버퍼 풀)에 올라가는 인덱스 페이지 수 증가, 쓰기 시 갱신해야 할 데이터량 증가
  • 판단 기준: 매우 자주 실행되는 조회 쿼리이고 SELECT절 컬럼이 몇 개 안 되면 covering index로 확장할 가치가 큼. 가끔 실행되거나 SELECT절이 넓으면(예: SELECT *) covering index로 만드는 비용이 이득보다 클 수 있음.

🚫 흔한 오해

인덱스는 많이 걸어둘수록 조회가 빨라지니 좋다.
인덱스는 쓰기 작업마다 함께 갱신되는 자료구조입니다. 인덱스가 많을수록 INSERT/UPDATE/DELETE가 느려지고 저장 공간도 늘어납니다. 실제로 사용되지 않는 인덱스는 이득 없이 비용만 발생시키므로 주기적으로 사용 통계를 확인해 정리해야 합니다.
WHERE 절에 있는 컬럼은 다 인덱스에 넣으면 된다.
복합 인덱스는 순서가 있는 구조입니다. leftmost prefix rule 때문에 컬럼을 아무 순서로나 넣으면 일부 쿼리에서 인덱스를 활용하지 못합니다. 어떤 컬럼이 등호 조건으로 쓰이는지, 범위 조건으로 쓰이는지에 따라 순서를 다르게 설계해야 합니다(일반적으로 등호 조건 컬럼을 범위 조건 컬럼보다 앞에 둡니다).
selectivity가 높은 컬럼을 무조건 맨 앞에 두면 된다.
selectivity는 인덱스 순서를 정하는 중요한 기준이지만 유일한 기준은 아닙니다. 여러 쿼리가 같은 인덱스를 공유해야 하거나, ORDER BY 요구사항이 있는 경우에는 selectivity보다 실제 쿼리 패턴과의 정합성이 더 중요할 수 있습니다.
인덱스를 걸었으면 옵티마이저가 항상 그 인덱스를 사용한다.
옵티마이저는 통계 정보를 바탕으로 여러 실행 계획의 비용을 추정해서 더 싼 쪽을 선택합니다. selectivity가 낮은 컬럼이거나, 통계가 오래되어 실제 분포와 다르거나, 컬럼에 함수를 씌우면 인덱스가 있어도 Full Scan을 선택할 수 있습니다.
PRIMARY KEY에는 인덱스가 자동으로 있으니 신경 쓸 게 없다.
PK에 인덱스(주로 클러스터드 인덱스)가 자동 생성되는 것은 맞지만, PK 선택 자체가 성능에 영향을 줍니다. 예를 들어 무작위(UUID 등) 값을 PK로 쓰면 삽입 시 트리 곳곳에 랜덤하게 끼워 넣어야 해서 페이지 분할이 잦아지고 쓰기 성능이 저하될 수 있습니다. 순차 증가 값(auto-increment) 대비 이런 trade-off를 인지해야 합니다.

📊 좋은 답변 vs 약한 답변

강한 답변

"이 인덱스를 탈 쿼리들의 WHERE 절을 먼저 나열하고, 등호 조건으로 쓰이는 컬럼을 앞에 둡니다. 여러 등호 조건이 있으면 selectivity가 높은 컬럼을 우선하되, 여러 쿼리가 이 인덱스를 공유해야 하면 공통 컬럼을 우선합니다. ORDER BY가 있으면 정렬 컬럼을 뒤에 이어 붙이고, SELECT절 컬럼이 적으면 covering index로 확장할지 검토합니다. 마지막으로 EXPLAIN으로 실제 실행 계획을 확인하고, 쓰기 비용 대비 이득이 충분한지 판단합니다."

왜 강한가: 순서 결정 기준(leftmost prefix, selectivity, 재사용성)을 명확히 언급하고, ORDER BY·covering index까지 확장해서 생각합니다. 마지막에 검증 단계까지 포함해 "추측이 아니라 근거로 판단"하는 태도를 보여줍니다.

약한 답변

"자주 조회되는 컬럼을 앞에 두고 나머지를 뒤에 붙입니다. WHERE 절에 있는 컬럼은 다 인덱스에 넣어주면 됩니다."

왜 약한가: leftmost prefix rule을 언급하지 않아 "왜 그 순서인지"에 대한 근거가 없습니다. selectivity, ORDER BY, covering index, 쓰기 비용 같은 트레이드오프도 전혀 언급하지 않습니다.

🎓 Senior-level Follow-up 질문

Q. (a, b, c) 복합 인덱스가 있는데 쿼리는 WHERE a = ? AND c = ?입니다. 이 인덱스가 얼마나 도움이 됩니까?
의도: leftmost prefix rule을 기계적으로 외웠는지, 아니면 "부분적으로 도움이 되는 상황"까지 세밀하게 이해하는지 확인합니다.
모범 사고 경로: a까지는 인덱스 탐색(range/ref access)으로 후보를 좁힐 수 있습니다. 하지만 b 조건이 없으므로 c는 인덱스 순서상 바로 좁혀지지 않고, a로 좁혀진 범위 안에서 c 값을 하나씩 필터링(index condition pushdown 등으로 인덱스 내부에서 걸러지긴 하지만, 트리 탐색으로 직접 좁히는 것과는 다름)하는 방식으로 동작합니다. b의 selectivity가 낮다면 인덱스를 (a, c, b)로 바꾸는 것이 이 쿼리에는 더 유리할 수 있습니다 — 다만 이는 다른 쿼리가 b를 a 다음에 필요로 하지 않는지 함께 확인해야 합니다.
Q. 인덱스를 새로 추가하기 전에 기존 인덱스와 중복되는지 어떻게 확인하시겠습니까?
의도: 인덱스를 무작정 추가하는지, 기존 인덱스 구조를 먼저 점검하는 습관이 있는지 확인합니다.
모범 사고 경로: 기존 복합 인덱스의 leftmost prefix로 새 인덱스 요구사항이 이미 커버되는지 먼저 확인합니다. 예를 들어 (a, b) 인덱스가 있는데 WHERE a = ? 조회만 필요하다면 별도로 (a) 인덱스를 추가할 필요가 없습니다. 시스템 카탈로그(예: information_schema.statistics, pg_indexes)나 인덱스 사용 통계(예: sys.schema_unused_indexes, PostgreSQL의 pg_stat_user_indexes)로 실제 사용 여부도 함께 점검합니다.
Q. UUID를 PK로 쓰는 것과 auto-increment 정수를 PK로 쓰는 것, 인덱스 관점에서 어떤 차이가 있습니까?
의도: PK 선택이 클러스터드 인덱스 구조와 쓰기 성능에 미치는 영향까지 이해하는지 확인합니다.
모범 사고 경로: 클러스터드 인덱스를 쓰는 엔진(예: InnoDB)에서는 PK 순서대로 데이터가 물리적으로 정렬 저장됩니다. auto-increment는 항상 오른쪽 끝에 순차적으로 추가되므로 페이지 분할이 적지만, UUID(특히 랜덤 UUID)는 값이 무작위라 트리 중간 곳곳에 삽입되면서 페이지 분할과 단편화가 잦아져 쓰기 성능이 저하되고 캐시 효율도 떨어집니다. 분산 환경에서 전역 유일성이 필요해 UUID를 꼭 써야 한다면, 시간 정렬성을 가진 UUID(UUIDv7 등)나 별도의 순차 대체키를 함께 검토합니다.
Q. 인덱스 추가 이후 쓰기 성능이 얼마나 나빠질지 어떻게 사전에 가늠하시겠습니까?
의도: 인덱스 추가를 "일단 걸어보고 문제되면 빼자"는 식으로 접근하는지, 사전에 근거를 갖고 판단하는지 봅니다.
모범 사고 경로: 해당 테이블의 초당 쓰기 건수(INSERT/UPDATE/DELETE)와 현재 인덱스 개수를 파악합니다. staging 환경에서 실제(또는 유사 규모의) 데이터로 인덱스 추가 전후 쓰기 처리량과 지연을 벤치마크로 비교합니다. 특히 대량 배치 INSERT가 있는 테이블이라면 그 배치 작업 시간이 얼마나 늘어나는지도 함께 측정합니다.
