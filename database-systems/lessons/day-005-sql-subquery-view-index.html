2. 서브쿼리(Subquery)

서브쿼리는 다른 SQL 문 안에 중첩된 SELECT 문이다. 결과 형태에 따라 세 종류로 구분한다.

종류반환 값위치예시 상황
스칼라(Scalar) 단일 행·단일 열 (값 1개) SELECT·WHERE·HAVING 전체 평균 급여를 비교 기준으로 사용
행(Row) 단일 행·여러 열 WHERE + 행 비교 특정 직원과 동일한 (부서, 급여)인 직원 검색
테이블(Table) 여러 행·여러 열 FROM (인라인 뷰), IN·ANY·ALL 부서별 평균 급여 결과를 다시 JOIN

2-1. 스칼라 서브쿼리

전체 평균 급여보다 높은 직원을 조회한다.

-- 전체 직원 평균 급여보다 높은 직원
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees);

-- 결과: 이영희(7,500,000), 한지민(6,800,000)

포인트: 스칼라 서브쿼리는 반드시 행 1개, 열 1개를 반환해야 한다. 여러 행이 반환되면 오류가 발생한다.

2-2. IN 서브쿼리

서브쿼리 결과 목록 안에 값이 존재하는지 확인한다.

-- 직원이 한 명 이상 있는 부서 이름 조회
SELECT dept_name
FROM   departments
WHERE  dept_id IN (
  SELECT DISTINCT dept_id
  FROM   employees
  WHERE  dept_id IS NOT NULL
);

-- 결과: 개발팀, 마케팅팀, 인사팀  (기획팀 제외)

2-3. 테이블 서브쿼리 (인라인 뷰)

FROM 절에 서브쿼리를 배치하면 임시 테이블처럼 사용할 수 있다. 반드시 별칭(alias)을 붙여야 한다.

-- 부서별 평균 급여를 인라인 뷰로 만들어 부서명과 JOIN
SELECT d.dept_name, avg_info.avg_salary
FROM   departments d
JOIN   (
  SELECT dept_id, ROUND(AVG(salary)) AS avg_salary
  FROM   employees
  WHERE  dept_id IS NOT NULL
  GROUP BY dept_id
) AS avg_info ON d.dept_id = avg_info.dept_id;

-- 결과 예시
-- 개발팀    | 6,750,000
-- 마케팅팀  | 5,250,000
-- 인사팀    | 6,200,000

3. 상관 서브쿼리(Correlated Subquery)

상관 서브쿼리는 외부 쿼리의 현재 행을 참조하는 서브쿼리다. 외부 행마다 한 번씩 실행되므로 일반 서브쿼리보다 느릴 수 있다.

외부 쿼리가 employees 행을 하나씩 처리할 때마다 → 내부 서브쿼리가 현재 행의 dept_id를 사용해 실행 → 내부 결과로 조건을 평가 → 조건 통과 시 외부 결과에 포함

3-1. 자신의 부서 평균보다 높은 급여를 받는 직원

SELECT e.name, e.salary, e.dept_id
FROM   employees e
WHERE  e.salary > (
  SELECT AVG(e2.salary)
  FROM   employees e2
  WHERE  e2.dept_id = e.dept_id   -- 외부 쿼리의 e.dept_id 참조
);

-- 결과: 이영희(개발팀, 7,500,000), 한지민(NULL — 부서 없으므로 NULL 비교 → 제외)

주의: e.dept_id가 NULL인 행은 서브쿼리 내 WHERE dept_id = NULL이 항상 거짓이므로 비교 결과가 NULL이 되어 조건을 통과하지 못한다.

3-2. 부서 평균 급여와 각 직원 급여를 나란히 표시

SELECT
  e.name,
  e.salary,
  (SELECT ROUND(AVG(e2.salary))
   FROM   employees e2
   WHERE  e2.dept_id = e.dept_id) AS dept_avg
FROM employees e;

4. EXISTS / NOT EXISTS

EXISTS는 서브쿼리가 한 행이라도 반환하면 TRUE를 돌려준다. 실제 값을 반환하지 않고 존재 여부만 확인하므로 IN보다 대규모 데이터에서 빠를 때가 많다.

4-1. EXISTS — 직원이 존재하는 부서

SELECT dept_name
FROM   departments d
WHERE  EXISTS (
  SELECT 1
  FROM   employees e
  WHERE  e.dept_id = d.dept_id
);

-- 결과: 개발팀, 마케팅팀, 인사팀

관례: EXISTS 내부의 SELECT는 SELECT 1 또는 SELECT *로 쓴다. 어떤 값을 반환하느냐는 중요하지 않고 행의 존재 여부만 본다.

4-2. NOT EXISTS — 직원이 없는 부서

SELECT dept_name
FROM   departments d
WHERE  NOT EXISTS (
  SELECT 1
  FROM   employees e
  WHERE  e.dept_id = d.dept_id
);

-- 결과: 기획팀

4-3. IN vs EXISTS 비교

구분INEXISTS
반환서브쿼리 결과 목록과 일치 여부행 존재 여부(TRUE/FALSE)
NULL 처리결과에 NULL 포함 시 NOT IN 오작동 가능NULL에 안전
성능소규모에 적합대규모 외부 테이블에 유리
가독성단순 목록 비교 시 직관적상관 서브쿼리와 결합 시 명확

5. 뷰(VIEW)

뷰는 이름이 붙은 저장된 SELECT 문이다. 실제 데이터를 복사하지 않고 쿼리를 재사용·단순화하는 가상 테이블이다.

[뷰 정의] [뷰 조회] CREATE VIEW v AS SELECT * FROM v; SELECT ...; ↓ DB 내부에서 원본 쿼리를 실행 → 항상 최신 데이터 반환

5-1. 기본 문법

CREATE VIEW view_name AS
  SELECT ...;

-- 조회
SELECT * FROM view_name;

-- 삭제
DROP VIEW view_name;

-- 재정의 (PostgreSQL·MySQL)
CREATE OR REPLACE VIEW view_name AS
  SELECT ...;

5-2. 부서별 직원 정보 뷰

CREATE VIEW v_emp_dept AS
  SELECT
    e.emp_id,
    e.name        AS emp_name,
    e.salary,
    d.dept_name
  FROM employees e
  LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- 이후 단순하게 조회
SELECT dept_name, COUNT(*) AS headcount, AVG(salary) AS avg_salary
FROM   v_emp_dept
GROUP  BY dept_name;

5-3. 뷰의 장점과 한계

장점한계
복잡한 JOIN을 숨겨 쿼리 단순화성능 개선 효과 없음 (매번 원본 실행)
접근 제어 — 민감 컬럼 제외한 뷰 노출일부 DB에서 INSERT/UPDATE 불가
논리적 데이터 독립성 — 테이블 구조 변경 시 뷰만 수정뷰에 ORDER BY·LIMIT 포함 시 중첩 주의

Materialized View: PostgreSQL 등에서 지원하는 구체화 뷰는 결과를 물리적으로 저장해 읽기 성능을 크게 높인다. 데이터는 REFRESH MATERIALIZED VIEW로 갱신한다.

6. 인덱스(INDEX)

인덱스는 특정 컬럼의 값을 정렬·구조화하여 저장한 별도 자료구조다. WHERE·JOIN·ORDER BY 성능을 극적으로 높일 수 있다.

6-1. 인덱스가 없을 때 vs 있을 때

인덱스 없음 (Full Table Scan) employees 테이블 (100만 행) ┌──────────────────────────────────────────┐ │ 처음부터 끝까지 모든 행을 읽어 비교 │ │ → 최악의 경우 100만 번 비교 │ └──────────────────────────────────────────┘ 인덱스 있음 (Index Scan) employees.name 인덱스 (B-Tree) [박] / \ [김] [이] / \ / \ [김철수][김민][이영희][이순신] → 트리 순회만으로 O(log n) 탐색

6-2. 기본 문법

-- 단일 컬럼 인덱스
CREATE INDEX idx_emp_name ON employees(name);

-- 복합 컬럼 인덱스 (왼쪽 컬럼부터 순서 중요)
CREATE INDEX idx_emp_dept_salary ON employees(dept_id, salary);

-- UNIQUE 인덱스
CREATE UNIQUE INDEX idx_emp_email ON employees(email);

-- 인덱스 삭제
DROP INDEX idx_emp_name;          -- PostgreSQL
DROP INDEX idx_emp_name ON employees;  -- MySQL

6-3. EXPLAIN으로 실행 계획 확인

EXPLAIN은 쿼리가 어떤 방식으로 실행될지 미리 보여준다. 인덱스 사용 여부를 검증할 때 필수다.

EXPLAIN SELECT * FROM employees WHERE name = '김철수';

-- 인덱스 미생성 시
-- Seq Scan on employees  (cost=0.00..12.10 rows=1 width=52)
--   Filter: ((name)::text = '김철수')

-- 인덱스 생성 후
CREATE INDEX idx_emp_name ON employees(name);
EXPLAIN SELECT * FROM employees WHERE name = '김철수';
-- Index Scan using idx_emp_name on employees  (cost=0.15..8.17 rows=1 width=52)
--   Index Cond: ((name)::text = '김철수')

6-4. 인덱스가 효과 없는 상황

상황예시이유
함수 적용WHERE UPPER(name) = 'KIM'인덱스는 원본 값으로 구성됨
선행 와일드카드WHERE name LIKE '%철수'B-Tree는 접두사 기준으로 정렬됨
묵시적 형변환WHERE emp_id = '101' (INT 컬럼에 문자열)타입 캐스팅이 인덱스를 무력화
복합 인덱스 왼쪽 컬럼 누락idx(dept_id, salary)에서 WHERE salary = 5000000선두 컬럼 없이 두 번째 컬럼만 사용
선택도가 낮은 컬럼boolean 컬럼, 성별 등전체 대비 매칭 행이 너무 많으면 Full Scan이 유리

6-5. 커버링 인덱스

쿼리에 필요한 모든 컬럼이 인덱스에 포함되면 테이블을 읽지 않고 인덱스만으로 결과를 반환한다. 이를 커버링 인덱스(Covering Index)라 한다.

CREATE INDEX idx_emp_dept_name ON employees(dept_id, name);

-- dept_id와 name만 필요한 쿼리 → 테이블 접근 없이 인덱스만 사용
SELECT name FROM employees WHERE dept_id = 1;

PostgreSQL에서는 INCLUDE 절로 인덱스에 추가 컬럼을 포함할 수 있다: CREATE INDEX idx ON t(a) INCLUDE (b, c);

7. 종합 실습

서브쿼리, 뷰, 인덱스를 조합한 시나리오다.

시나리오: 부서 평균보다 높은 급여를 받는 직원 목록 뷰 생성

-- 1단계: 뷰 생성
CREATE VIEW v_high_earners AS
  SELECT e.emp_id, e.name, e.salary, d.dept_name,
         dept_avg.avg_salary
  FROM   employees e
  JOIN   departments d ON e.dept_id = d.dept_id
  JOIN   (
    SELECT dept_id, ROUND(AVG(salary)) AS avg_salary
    FROM   employees
    WHERE  dept_id IS NOT NULL
    GROUP  BY dept_id
  ) AS dept_avg ON e.dept_id = dept_avg.dept_id
  WHERE  e.salary > dept_avg.avg_salary;

-- 2단계: 인덱스로 성능 최적화
CREATE INDEX idx_emp_dept ON employees(dept_id);
CREATE INDEX idx_emp_salary ON employees(salary);

-- 3단계: 뷰 조회
SELECT dept_name, name, salary, avg_salary
FROM   v_high_earners
ORDER  BY dept_name, salary DESC;

-- 기대 결과
-- 개발팀  | 이영희 | 7,500,000 | 6,750,000

퀴즈

Q1. 급여가 전체 평균보다 낮은 직원의 이름과 급여를 구하시오. 
SELECT name, salary
FROM   employees
WHERE  salary < (SELECT AVG(salary) FROM employees);

전체 평균 급여 ≈ 6,166,667 → 박민준(5,500,000), 정도윤(5,000,000)

Q2. EXISTS를 사용해 직원이 없는 부서 이름을 조회하시오. 
SELECT dept_name
FROM   departments d
WHERE  NOT EXISTS (
  SELECT 1 FROM employees e WHERE e.dept_id = d.dept_id
);

결과: 기획팀

Q3. 다음 중 인덱스가 사용되지 않는 쿼리는? (idx_emp_name 인덱스 존재 가정)
  1. WHERE name = '김철수'
  2. WHERE name LIKE '김%'
  3. WHERE name LIKE '%철수'
  4. WHERE name > '나'

정답: c. 선행 와일드카드(%철수)는 B-Tree 인덱스를 사용할 수 없다.
