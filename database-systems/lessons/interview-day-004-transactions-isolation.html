💬 면접 질문

Core Question

"READ COMMITTED와 REPEATABLE READ를 어떻게 선택합니까?"

함께 출제되는 변형 질문

  • "Dirty read, non-repeatable read, phantom read의 차이를 설명해주세요."
  • "MVCC가 뭔지, lock 기반 동시성 제어와 어떻게 다른지 설명해주세요."
  • "격리 수준을 SERIALIZABLE로 올리면 항상 더 안전합니까?"
  • "재고 차감 로직에서 두 요청이 동시에 들어와 재고가 잘못 줄어드는 문제(lost update)를 어떻게 방지하시겠습니까?"

🎯 왜 면접에서 묻는가

이 질문은 "격리 수준 4단계 이름을 외웠는가"가 아니라 동시에 실행되는 트랜잭션들이 서로의 데이터를 어떻게 침범할 수 있는지 이해하고, 그 위험을 감수 가능한 수준으로 통제할 수 있는가를 확인하기 위한 것입니다.


  • 동시성 버그 감각: 실무에서 가장 재현하기 어려운 버그는 동시 요청이 겹칠 때만 발생하는 정합성 버그입니다. 이 질문은 그런 버그를 사전에 예측할 수 있는지를 확인합니다.
  • 비용 감각: 주니어는 "정합성이 중요하니 SERIALIZABLE을 쓰겠다"고 답하지만, 시니어는 격리 수준을 올릴수록 동시성이 줄고 lock 경합·재시도 비용이 커진다는 것을 알고 필요한 만큼만 올립니다.
  • 내부 동작 이해: MVCC와 lock이 실제로 어떻게 동작하는지 알아야 "왜 이 격리 수준에서 이 이상 현상이 막히는지 안 막히는지"를 구조적으로 설명할 수 있습니다.
  • 벤더 차이 인지: SQL 표준의 격리 수준 정의와 실제 DB(특히 MySQL InnoDB)의 구현이 다르다는 것을 아는지 확인합니다. 이걸 모르면 실제 운영에서 잘못된 가정으로 버그를 만듭니다.

📚 전제 개념

트랜잭션이 왜 필요한가 — ACID를 쉬운 말로

여러 SQL 문을 "하나의 작업 단위"로 묶어서, 중간에 실패해도 부분 반영이 남지 않게 하는 것이 트랜잭션입니다. ACID는 이 작업 단위가 지켜야 할 네 가지 약속입니다.

  • Atomicity(원자성): 트랜잭션 안의 여러 작업이 전부 반영되거나 전부 반영되지 않아야 합니다. 계좌 이체에서 출금만 되고 입금이 안 되는 상황을 막습니다.
  • Consistency(일관성): 트랜잭션 전후로 데이터가 정의된 제약(외래키, unique, 도메인 규칙)을 항상 만족해야 합니다.
  • Isolation(격리성): 동시에 실행되는 트랜잭션들이 서로의 중간 상태를 침범하지 않아야 합니다. 오늘 레슨의 핵심 주제입니다.
  • Durability(지속성): 커밋된 데이터는 장애가 나도(전원이 꺼져도) 사라지지 않아야 합니다. WAL(Write-Ahead Log) 같은 구조로 보장됩니다.

이 중 Isolation은 "완벽히 지키면 지킬수록 좋은 것"이 아니라 성능과 맞바꾸는 손잡이(knob)입니다. 오늘 다루는 격리 수준은 이 손잡이를 어디까지 돌릴지 정하는 기준입니다.

동시 실행이 만드는 4가지 이상 현상(Anomaly)

격리 수준을 낮게 잡으면 트랜잭션들이 서로의 데이터를 얼마나 "볼 수 있는가"가 느슨해지면서 다음 이상 현상이 발생할 수 있습니다.

  • Dirty Read(더티 리드): 아직 커밋되지 않은 다른 트랜잭션의 변경을 읽어버림. 그 트랜잭션이 롤백되면 존재한 적 없는 데이터를 읽은 셈이 됩니다.
  • Non-repeatable Read(반복 불가능한 읽기): 같은 트랜잭션 안에서 같은 행을 두 번 읽었는데 값이 다름. 중간에 다른 트랜잭션이 그 행을 커밋해버렸기 때문입니다.
  • Phantom Read(팬텀 리드): 같은 조건으로 두 번 조회했는데 행의 개수(집합)가 달라짐. 중간에 다른 트랜잭션이 조건에 맞는 행을 새로 삽입/삭제했기 때문입니다. Non-repeatable read가 "행의 값"이 바뀌는 것이라면, phantom read는 "행의 존재 여부(집합)"가 바뀌는 것입니다.
  • Lost Update(갱신 손실): 두 트랜잭션이 같은 값을 동시에 읽고, 각자 계산한 결과로 덮어쓰면서 한쪽의 변경이 사라짐. 표준 격리 수준 정의에는 포함되지 않지만 실무에서 가장 자주 만나는 버그입니다(아래 재고 차감 시나리오 참고).

4단계 격리 수준과 이상 현상 매트릭스 (SQL 표준 기준)

SQL 표준은 어떤 이상 현상을 허용하는지에 따라 격리 수준을 4단계로 정의합니다. 위로 갈수록 정합성은 강해지고 동시성(처리량)은 낮아집니다.

격리 수준Dirty ReadNon-repeatable ReadPhantom Read
READ UNCOMMITTED허용허용허용
READ COMMITTED방지허용허용
REPEATABLE READ방지방지표준상 허용 (구현별 상이)
SERIALIZABLE방지방지방지
실무 팁: PostgreSQL과 MySQL(InnoDB) 모두 기본값이 다릅니다. PostgreSQL은 READ COMMITTED, MySQL은 REPEATABLE READ가 기본값입니다. 면접에서 "기본값이 뭐냐"고 물으면 DB 종류를 반드시 확인하고 답해야 합니다.

MVCC(Multi-Version Concurrency Control)란? — 쉬운 말로

lock만으로 격리성을 구현하면 "읽는 트랜잭션이 쓰는 트랜잭션을 막고, 쓰는 트랜잭션이 읽는 트랜잭션을 막는" 구조가 되어 동시성이 크게 떨어집니다. MVCC는 행이 수정될 때 기존 값을 지우지 않고 버전(스냅샷)으로 남겨두는 방식으로 이 문제를 해결합니다.


비유하면, 도서관에서 책을 대출한 사람이 책에 메모를 남기는 중이어도, 다른 사람은 "메모 남기기 전 원본 사본"을 그대로 읽을 수 있게 해주는 것과 비슷합니다. 그래서 읽기가 쓰기를 막지 않고, 쓰기가 읽기를 막지 않습니다.

  • PostgreSQL: 행을 UPDATE하면 기존 행을 바로 지우지 않고 새 버전을 추가로 저장합니다(오래된 버전은 나중에 VACUUM이 정리). 각 트랜잭션은 자신이 시작된 시점 기준으로 "볼 수 있는 버전"만 읽습니다.
  • MySQL(InnoDB): 변경 전 값을 undo log에 남겨두고, 트랜잭션은 자신의 스냅샷 시점보다 이후에 변경된 값을 만나면 undo log를 따라가며 그 시점의 값을 재구성해서 읽습니다.

즉 MVCC 덕분에 일반적인 SELECT(읽기)는 lock을 거의 잡지 않고도 dirty read와 non-repeatable read를 방지할 수 있습니다. "격리 수준을 올리면 lock을 더 많이 잡는다"는 생각은 절반만 맞는 오해입니다(아래 오해 섹션 참고).

InnoDB REPEATABLE READ가 Phantom Read를 막는 방법 — Gap Lock / Next-key Lock

SQL 표준상 REPEATABLE READ는 phantom read를 허용해도 되지만, MySQL InnoDB는 REPEATABLE READ에서도 gap lock(인덱스 값 "사이의 빈 공간"에 거는 락)과 next-key lock(레코드 락 + gap lock 조합)을 이용해 새로운 행이 그 범위에 끼어드는 것 자체를 막습니다. 그래서 InnoDB의 REPEATABLE READ는 표준 정의보다 더 강한 정합성을 제공합니다(단, 이는 일반 SELECT가 아니라 SELECT ... FOR UPDATE, UPDATE, DELETE처럼 lock을 거는 쓰기 계열 작업에 해당합니다).

주의: 이는 InnoDB의 구현 특성이지 SQL 표준의 보장이 아닙니다. 다른 RDB(PostgreSQL 등)에서 REPEATABLE READ를 쓸 때 같은 동작을 기대하면 안 됩니다.

비관적 락(Pessimistic Lock) vs 낙관적 락(Optimistic Lock)

격리 수준과 별개로, 애플리케이션 레벨에서 동시 갱신 충돌을 막는 두 가지 전략이 있습니다.

  • 비관적 락: "충돌이 자주 일어날 것"이라 가정하고 SELECT ... FOR UPDATE로 행을 미리 잠근 뒤 수정합니다. 충돌 가능성이 높고 재시도 비용이 큰 로직(재고 차감, 좌석 예약)에 적합합니다.
  • 낙관적 락: "충돌이 드물 것"이라 가정하고 잠그지 않은 채 읽고, 수정할 때 version 컬럼(또는 updated_at)을 조건에 포함해 UPDATE ... WHERE id = ? AND version = ? 형태로 반영합니다. 영향받은 행이 0건이면 충돌로 간주하고 재시도합니다. 충돌이 드물고 lock 경합을 피하고 싶은 로직(게시글 수정, 프로필 수정)에 적합합니다.

🏭 실무 연결

시나리오: 재고 차감 — Lost Update

두 사용자가 동시에 재고 1개 남은 상품을 주문합니다. 두 트랜잭션 모두 SELECT stock FROM products WHERE id = ?로 stock = 1을 읽고, 애플리케이션에서 1 - 1 = 0을 계산해서 각자 UPDATE products SET stock = 0을 실행합니다. 결과적으로 재고 하나가 두 번 팔린 셈이 되지만 DB 상 재고는 정확히 0으로 보여서 문제를 알아채기 어렵습니다.

이는 격리 수준을 SERIALIZABLE로 올려도 "읽고 → 계산하고 → 쓰는" 애플리케이션 로직 자체가 원자적이지 않으면 재현될 수 있습니다(구현에 따라 SERIALIZABLE이 직렬화 실패로 한쪽을 롤백시켜 막아주기도 하지만, 그 경우 재시도 로직이 필요합니다). 실무에서는 UPDATE products SET stock = stock - 1 WHERE id = ? AND stock >= 1처럼 읽기와 쓰기를 하나의 원자적 SQL 문으로 합치거나, SELECT FOR UPDATE로 비관적 락을 거는 방식으로 해결합니다.

시나리오: 정산 배치 — 긴 트랜잭션의 MVCC 부작용

REPEATABLE READ에서 시작한 정산 배치 트랜잭션이 몇 시간 동안 열려 있으면, 그 스냅샷 시점 이전 데이터를 계속 참조할 수 있어야 하므로 DB는 그 사이 변경된 오래된 버전(undo log / dead tuple)을 지우지 못합니다. 결과적으로 undo 영역이 커지거나(MySQL), 테이블에 dead tuple이 쌓여 VACUUM이 밀리고(PostgreSQL) 전체 쿼리 성능이 저하됩니다. "격리 수준을 올렸더니 며칠 뒤 전체 DB가 느려졌다"는 장애는 실무에서 드물지 않은 패턴입니다.

시나리오: 결제 상태 전이 — Dirty Read로 인한 이중 처리

결제 승인 트랜잭션이 status = 'PAID'로 갱신 중인데 아직 커밋 전인 상태에서, 알림 발송 프로세스가 READ UNCOMMITTED로 그 값을 읽어버리면 실제로는 승인이 실패해 롤백될 수 있는 상태를 "성공"으로 알림을 보낼 수 있습니다. 결제·정산처럼 금전이 얽힌 로직에서는 READ UNCOMMITTED를 쓸 이유가 사실상 없고, 최소 READ COMMITTED 이상을 기본으로 가져가야 합니다.

🔍 답변 사고 순서

격리 수준 질문은 "어느 것이 더 안전한가"가 아니라, 그 로직이 실제로 어떤 이상 현상에 취약한지부터 역산하는 사고 과정을 보여주는 것이 핵심입니다.
  1. 1단계 — 이 로직이 요구하는 정합성 수준 정의

    이 트랜잭션이 실패하면 어떤 비즈니스 피해가 생기는지 먼저 정의한다(중복 차감, 이중 결제, 잘못된 집계 등).

  2. 2단계 — 취약한 이상 현상 식별

    이 로직이 dirty read, non-repeatable read, phantom read, lost update 중 어느 것에 노출되는지 구체적인 시나리오로 확인한다.

  3. 3단계 — 격리 수준 후보 선택

    식별한 이상 현상을 막는 최소한의 격리 수준을 고른다. "가장 안전한 수준"이 아니라 "필요한 만큼의 수준"을 목표로 한다.

  4. 4단계 — 실제 DB의 구현 특성 확인

    사용 중인 DB(MySQL InnoDB vs PostgreSQL 등)에서 그 격리 수준이 표준과 동일하게 동작하는지, gap lock 같은 벤더별 보강이 있는지 확인한다.

  5. 5단계 — 격리 수준만으로 부족한 부분은 애플리케이션 레벨로 보강

    lost update처럼 격리 수준만으로 막을 수 없는 문제는 원자적 UPDATE, SELECT FOR UPDATE, 낙관적 락(version 컬럼) 중 상황에 맞는 방식을 선택한다.

  6. 6단계 — 동시성 비용과 트랜잭션 길이 점검

    선택한 격리 수준과 lock 전략이 처리량에 미치는 영향을 가늠하고, 트랜잭션을 최대한 짧게 유지해 lock 보유 시간과 MVCC 버전 누적을 줄인다.

  7. 7단계 — 재현 테스트로 검증

    동시 요청을 인위적으로 겹치게 만드는 테스트(두 커넥션을 나란히 실행)로 실제로 이상 현상이 막히는지, 또는 예상한 재시도/실패가 발생하는지 확인한다.

💡 핵심 답변 예시

짧은 버전 (1분)

"READ COMMITTED와 REPEATABLE READ 중 하나를 고를 때는 이 트랜잭션이 non-repeatable read에 취약한지를 먼저 봅니다. 같은 트랜잭션 안에서 같은 조건을 여러 번 조회하고 그 값들이 일관되게 유지돼야 하는 로직(예: 정산 계산)이면 REPEATABLE READ가 필요합니다. 반대로 매 조회가 최신 값을 봐도 되는 일반적인 API 요청 처리라면 READ COMMITTED로 충분하고, lock 경합과 MVCC 버전 누적을 줄일 수 있어 처리량 측면에서 유리합니다. 다만 재고 차감처럼 lost update가 우려되는 로직은 격리 수준과 별개로 원자적 UPDATE나 SELECT FOR UPDATE로 따로 방어합니다."

깊이 있는 버전 (3~4분)

"격리 수준 선택은 '더 안전한 걸 고른다'가 아니라 '이 로직이 실제로 어떤 이상 현상에 노출되는가'를 먼저 정의하는 것에서 시작합니다.

첫째, 이 트랜잭션이 dirty read, non-repeatable read, phantom read, lost update 중 무엇에 취약한지 구체적으로 그려봅니다. READ COMMITTED는 dirty read는 막아주지만, 같은 트랜잭션 안에서 두 번 읽은 값이 달라질 수 있는(non-repeatable read) 상황은 막지 못합니다. 정산처럼 한 트랜잭션 안에서 일관된 스냅샷이 필요한 로직이면 REPEATABLE READ가 필요합니다.

둘째, 사용 중인 DB의 실제 구현을 확인합니다. MySQL InnoDB의 REPEATABLE READ는 gap lock과 next-key lock으로 표준보다 더 강하게 phantom read를 막아주지만, PostgreSQL의 REPEATABLE READ는 표준에 가깝게 동작하며 대신 쓰기 충돌 시 직렬화 오류로 실패시키는 방식으로 정합성을 지킵니다. 이 차이를 모르고 DB를 바꾸면 같은 코드가 다르게 동작할 수 있습니다.

셋째, lost update처럼 격리 수준만으로 해결되지 않는 문제는 별도로 방어합니다. 재고 차감 같은 로직은 SERIALIZABLE로 올려도 애플리케이션이 '읽고 계산해서 쓰는' 구조라면 여전히 취약할 수 있어서, 저는 UPDATE ... WHERE stock >= 1처럼 조건부 원자적 UPDATE로 만들거나, 충돌 빈도가 낮은 로직이면 버전 컬럼 기반 낙관적 락을 씁니다.

마지막으로, 격리 수준을 올리는 것이 공짜가 아니라는 점을 함께 고려합니다. lock 경합이 늘고 트랜잭션이 길어질수록 MVCC 버전(undo log, dead tuple)이 누적돼 전체 DB 성능에 영향을 줄 수 있으므로, 트랜잭션을 최대한 짧게 유지하는 것도 격리 수준 선택만큼 중요하게 봅니다."

⚖️ Trade-off

격리 수준을 올리기 vs 낮게 유지하기

  • 올릴 때 이득: non-repeatable read, phantom read 같은 정합성 문제를 DB 레벨에서 방지
  • 올릴 때 비용: lock 경합 증가, 직렬화 실패(serialization failure)로 인한 트랜잭션 재시도 필요, 동시 처리량 감소
  • 판단 기준: 정산·회계처럼 한 트랜잭션 내 스냅샷 일관성이 중요한 로직은 REPEATABLE READ 이상, 일반적인 조회/갱신 API는 READ COMMITTED로 처리량을 우선.

비관적 락 vs 낙관적 락

  • 비관적 락 이득: 충돌을 사전에 원천 차단, 재시도 로직이 단순함
  • 비관적 락 비용: lock 대기 시간 발생, 트랜잭션이 길어지면 다른 트랜잭션까지 줄줄이 대기(lock convoy), deadlock 가능성 증가
  • 낙관적 락 이득: lock 없이 동시 처리량을 높게 유지
  • 낙관적 락 비용: 충돌 시 재시도 로직을 애플리케이션이 직접 구현해야 함, 충돌이 잦은 로직에는 오히려 비효율적
  • 판단 기준: 충돌 빈도가 높고 실패 비용이 큰 로직(한정 수량 재고, 좌석 예약)은 비관적 락, 충돌이 드문 로직(게시글 수정)은 낙관적 락.

MVCC(스냅샷 기반) vs 순수 2단계 잠금(2PL)

  • MVCC 이득: 읽기가 쓰기를 막지 않아 읽기 위주 워크로드에서 동시성이 크게 좋음
  • MVCC 비용: 오래된 버전을 정리하는 부가 작업(VACUUM, purge)이 필요하고, 긴 트랜잭션이 있으면 이 정리가 밀려 성능이 저하됨
  • 판단 기준: 대부분의 OLTP 워크로드(읽기 비중 높음)는 MVCC 기반 DB가 유리. 트랜잭션을 짧게 유지하는 운영 규율이 MVCC 이득을 지키는 전제 조건.

🚫 흔한 오해

정합성이 중요하니 무조건 SERIALIZABLE을 쓰면 안전하다.
SERIALIZABLE은 정합성은 가장 강하지만 동시성 저하와 직렬화 실패로 인한 재시도가 크게 늘어납니다. 재시도 로직 없이 SERIALIZABLE만 켜면 트래픽이 늘 때 오류율이 급증할 수 있습니다. "필요한 만큼만" 올리는 것이 시니어의 판단입니다.
격리 수준을 높이면 항상 lock을 더 많이, 더 오래 잡는다.
MVCC 기반 DB에서는 일반 SELECT(읽기)는 격리 수준을 올려도 lock을 거의 잡지 않습니다. 스냅샷으로 과거 버전을 읽기 때문입니다. lock이 늘어나는 것은 주로 쓰기 계열 작업(UPDATE, SELECT FOR UPDATE)이 gap lock 등을 추가로 요구할 때입니다.
REPEATABLE READ면 phantom read는 어떤 DB에서도 항상 안 생긴다.
SQL 표준상 REPEATABLE READ는 phantom read를 허용해도 되는 수준입니다. MySQL InnoDB는 gap lock으로 이를 추가로 막아주지만, 이는 InnoDB의 구현 특성이지 표준의 보장이 아닙니다. 다른 DB에서는 REPEATABLE READ에서도 phantom read가 발생할 수 있습니다.
트랜잭션 격리 수준만 맞게 설정하면 동시성 버그는 다 막힌다.
Lost update처럼 "읽고 → 애플리케이션에서 계산 → 쓰기"로 이루어진 로직은 격리 수준만으로 막기 어렵습니다. 원자적 SQL 문(WHERE 조건에 현재 값 포함)이나 낙관적/비관적 락을 애플리케이션 레벨에서 함께 설계해야 합니다.
NoSQL은 트랜잭션이 없으니 이런 고민을 할 필요가 없다.
많은 NoSQL도 문서/파티션 단위 원자성이나 자체 격리 모델(예: MongoDB의 다중 문서 트랜잭션)을 제공합니다. "트랜잭션이 없다"가 아니라 "범위와 보장 수준이 RDB와 다르다"고 이해해야 하며, 이는 Day 9(RDB vs NoSQL)에서 더 다룹니다.

📊 좋은 답변 vs 약한 답변

강한 답변

"이 트랜잭션이 non-repeatable read에 취약한지부터 확인합니다. 한 트랜잭션 안에서 같은 조건을 여러 번 읽고 그 값이 일관돼야 하면 REPEATABLE READ, 아니라면 READ COMMITTED로 처리량을 우선합니다. 사용 중인 DB가 MySQL이면 InnoDB의 gap lock 때문에 REPEATABLE READ가 표준보다 강하게 phantom read를 막아준다는 점도 함께 고려합니다. 다만 lost update는 격리 수준과 별개 문제라서 재고 차감 같은 로직은 원자적 UPDATE나 SELECT FOR UPDATE로 따로 방어합니다."

왜 강한가: 이상 현상을 구체적으로 식별하고, 격리 수준의 표준 정의와 실제 DB 구현 차이까지 구분하며, lost update가 격리 수준만으로 해결되지 않는 별개 문제임을 명확히 알고 있습니다.

약한 답변

"REPEATABLE READ가 더 안전하니까 항상 REPEATABLE READ를 씁니다. 정합성이 제일 중요합니다."

왜 약한가: 어떤 이상 현상을 막으려는지 구체적인 근거가 없고, 성능/동시성 비용을 전혀 고려하지 않았습니다. 격리 수준만으로 막을 수 없는 lost update 같은 문제도 언급하지 못합니다.

🎓 Senior-level Follow-up 질문

Q. MVCC를 쓰는 DB에서 SELECT는 lock을 안 잡는다고 하셨는데, 그러면 dirty read는 어떻게 막습니까?
의도: MVCC의 동작 원리를 "lock이 없다"는 결론만 외운 게 아니라, 스냅샷/버전 관리로 어떻게 격리성을 달성하는지 구조적으로 이해하는지 확인합니다.
모범 사고 경로: lock으로 다른 트랜잭션의 접근을 막는 대신, 각 트랜잭션이 자신의 시작 시점(또는 문장 시작 시점) 기준 스냅샷만 읽도록 버전을 분리합니다. 커밋되지 않은 변경은 아직 그 트랜잭션의 스냅샷에 "보이지 않는" 버전이므로 애초에 읽을 수가 없습니다. lock으로 막는 게 아니라 애초에 안 보이게 설계한 것이 MVCC의 핵심입니다.
Q. 낙관적 락으로 구현한 UPDATE가 0건 영향을 받았습니다. 이걸 애플리케이션에서 어떻게 처리해야 합니까?
의도: 낙관적 락을 "쓸 줄 아는가"가 아니라 충돌 발생 시 실제 운영 흐름(재시도, 사용자 안내)까지 설계할 수 있는지 확인합니다.
모범 사고 경로: 0건이면 그 사이 다른 트랜잭션이 먼저 커밋했다는 뜻이므로, 최신 값을 다시 읽어서 재계산 후 재시도하거나(자동 재시도가 안전한 로직), 재고 소진처럼 재시도가 무의미한 로직이면 사용자에게 "이미 반영된 정보와 다릅니다"를 알리고 새로고침을 유도합니다. 무한 재시도 루프를 막기 위해 재시도 횟수 상한과 백오프를 둡니다.
Q. 정산 배치처럼 오래 걸리는 트랜잭션을 REPEATABLE READ로 오래 열어두면 어떤 문제가 생길 수 있습니까?
의도: 격리 수준 선택이 "그 순간의 정합성"만이 아니라 시스템 전체의 자원(undo log, VACUUM)에 미치는 영향까지 고려하는지 확인합니다.
모범 사고 경로: 그 트랜잭션의 스냅샷 시점 이전 버전을 계속 유지해야 하므로, 그 사이 다른 트랜잭션들이 만든 오래된 버전(undo log, dead tuple)을 DB가 정리하지 못하고 쌓아둡니다. 이는 저장 공간 증가와 정리 작업 지연으로 이어져 전체 DB 성능에 영향을 줄 수 있습니다. 정산처럼 오래 걸리는 읽기 작업은 REPEATABLE READ 트랜잭션을 오래 유지하기보다, 읽기 전용 복제본에서 실행하거나 작은 배치로 쪼개는 것을 우선 검토합니다.
Q. SERIALIZABLE에서 "직렬화 실패(serialization failure)"란 무엇이고, 애플리케이션은 이를 어떻게 다뤄야 합니까?
의도: SERIALIZABLE을 "가장 안전한 설정"으로만 알고 있는지, 실제로 그 대가가 무엇인지(그리고 대가를 감당할 준비가 되어 있는지) 확인합니다.
모범 사고 경로: DB가 여러 트랜잭션을 마치 순서대로 실행한 것과 같은 결과를 보장하려다, 실제로는 동시 실행된 트랜잭션들의 순서를 정할 수 없다고 판단하면 그중 하나를 강제로 실패(롤백)시킵니다. 이것이 직렬화 실패입니다. 애플리케이션은 이 실패를 예외로 잡아 재시도하는 로직을 반드시 갖춰야 하며, 이 재시도 비용까지 감안해야 SERIALIZABLE 도입이 실질적으로 이득인지 판단할 수 있습니다.
