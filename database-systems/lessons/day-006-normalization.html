Day 6 — 정규화: 1NF·2NF·3NF·BCNF

날짜: 2026-06-26

소요 시간: 약 80분

선행 조건: Day 1 — 관계형 모델, Day 2 — SQL DDL

학습 목표

이 레슨을 마치면 다음을 할 수 있다:

  1. 함수적 종속(Functional Dependency)의 개념과 종류를 설명할 수 있다
  2. 1NF·2NF·3NF·BCNF의 위반 사례를 식별하고 분해(Decomposition)할 수 있다
  3. 이상(Anomaly)이 무엇인지, 정규화가 왜 이를 해결하는지 설명할 수 있다
  4. 역정규화(Denormalization)가 필요한 상황과 절충점을 논할 수 있다

1. 왜 정규화가 필요한가? — 이상(Anomaly)

잘못 설계된 테이블은 세 가지 이상 현상을 일으킨다.

문제 테이블 예시

아래 테이블은 수강(Enrollment) 정보를 한 테이블에 모두 담으려 했을 때의 모습이다.

student_idstudent_namecourse_idcourse_nameinstructorinstructor_phone
S01김민준C101데이터베이스이교수010-1111-2222
S01김민준C102운영체제박교수010-3333-4444
S02이서연C101데이터베이스이교수010-1111-2222
S03박지호C103알고리즘최교수010-5555-6666
이상 종류설명위 예시에서
삽입 이상
(Insertion Anomaly)		 수강생 없이 강좌만 추가 불가 학생이 아직 없는 새 강좌 C104를 등록하면 student_id에 NULL이 들어가거나 행 자체를 넣을 수 없다
갱신 이상
(Update Anomaly)		 중복 저장된 값 일부만 수정 시 불일치 발생 이교수 전화번호가 바뀌면 C101을 수강하는 모든 행을 함께 수정해야 한다. 한 행만 빠뜨리면 데이터가 불일치한다
삭제 이상
(Deletion Anomaly)		 일부 정보 삭제 시 다른 정보도 함께 사라짐 S03이 C103 수강 취소 → 최교수 정보가 테이블에서 완전히 사라진다

핵심: 이상 현상의 근본 원인은 하나의 테이블에 서로 다른 주제(학생, 강좌, 교수)가 뒤섞인 것이다. 정규화는 이를 의미 단위로 분리한다.

2. 함수적 종속 (Functional Dependency)

정규화를 이해하려면 먼저 함수적 종속(FD)을 알아야 한다.

속성 집합 X가 속성 집합 Y를 함수적으로 결정한다(X → Y):
같은 X 값을 가진 모든 튜플은 반드시 같은 Y 값을 가진다.

종류

종류정의예시
완전 함수적 종속
(Full FD)		 Y가 X 전체에 종속되고, X의 어떤 진부분집합에도 종속되지 않음 (student_id, course_id) → grade
grade는 두 속성 모두 있어야 결정된다
부분 함수적 종속
(Partial FD)		 Y가 X의 진부분집합에 종속됨 (student_id, course_id) → student_name
student_name은 student_id만으로 결정된다
이행적 종속
(Transitive FD)		 X → Z가 X → Y, Y → Z를 통해 간접적으로 성립 student_id → course_id → instructor
(student_id가 instructor를 간접 결정)
완전 FD: (student_id, course_id) ──────────► grade 부분 FD: (student_id, course_id) ─student_id─► student_name └─course_id──► course_name 이행적 FD: student_id ──► dept_id ──► dept_name (student_id가 dept_name을 이행 결정)

3. 제1 정규형 (1NF)

모든 속성값이 원자값(Atomic Value)이어야 한다. 반복 그룹이나 다중값 속성을 허용하지 않는다.

1NF 위반 예시

student_idstudent_namecourses (다중값)
S01김민준C101, C102
S02이서연C101

courses 컬럼에 쉼표로 구분된 여러 값이 들어있다 → 1NF 위반

1NF 변환 (행으로 분리)

student_idstudent_namecourse_id
S01김민준C101
S01김민준C102
S02이서연C101

주의: 현대 RDBMS의 배열 타입(PostgreSQL ARRAY, MySQL JSON 배열 등)은 1NF를 기술적으로 위반하는 저장 형태다. 검색·조인이 복잡해지므로 정규화된 설계를 우선 고려하자.

4. 제2 정규형 (2NF)

1NF를 만족하고, 기본키의 일부에만 종속되는 부분 함수적 종속이 없어야 한다.
즉, 모든 비키(non-key) 속성이 기본키 전체에 완전 함수적으로 종속되어야 한다.

기본키가 단일 컬럼이면 부분 종속이 원래 불가능하므로, 2NF는 복합 기본키일 때만 문제가 된다.

2NF 위반 예시 (복합 PK)

Enrollment(student_id, course_id, grade, student_name, course_name) FD 목록: (student_id, course_id) → grade ← 완전 FD ✔ student_id → student_name ← 부분 FD ✗ course_id → course_name ← 부분 FD ✗

2NF 변환 (부분 종속을 별도 테이블로 분리)

-- 수강 테이블: 완전 FD만 남김
CREATE TABLE Enrollment (
  student_id CHAR(4),
  course_id  CHAR(4),
  grade      CHAR(2),
  PRIMARY KEY (student_id, course_id)
);

-- 학생 테이블: student_id → student_name 분리
CREATE TABLE Student (
  student_id   CHAR(4) PRIMARY KEY,
  student_name VARCHAR(50) NOT NULL
);

-- 강좌 테이블: course_id → course_name 분리
CREATE TABLE Course (
  course_id   CHAR(4) PRIMARY KEY,
  course_name VARCHAR(100) NOT NULL
);

결과: 학생 이름을 수정할 때 Student 테이블 한 행만 바꾸면 된다 → 갱신 이상 제거.

5. 제3 정규형 (3NF)

2NF를 만족하고, 이행적 함수적 종속이 없어야 한다.
즉, 기본키가 아닌 속성이 다른 비키 속성을 통해 간접적으로 결정되면 안 된다.

3NF 위반 예시

Student(student_id, student_name, dept_id, dept_name) FD 목록: student_id → dept_id ← 직접 종속 ✔ dept_id → dept_name ← dept_id는 비키 속성 ✗ (이행적 종속) student_id → dept_name ← student_id가 dept_name을 이행 결정

dept_name이 기본키(student_id)가 아닌 비키 속성 dept_id에 종속되므로 3NF 위반.

3NF 변환 (이행 종속을 분리)

-- 학과 테이블: dept_id → dept_name 분리
CREATE TABLE Department (
  dept_id   CHAR(4) PRIMARY KEY,
  dept_name VARCHAR(100) NOT NULL
);

-- 학생 테이블: dept_name 제거, dept_id(FK)만 유지
CREATE TABLE Student (
  student_id   CHAR(4) PRIMARY KEY,
  student_name VARCHAR(50) NOT NULL,
  dept_id      CHAR(4) REFERENCES Department(dept_id)
);

결과: 학과 이름이 바뀌면 Department 한 행만 수정. 삭제 이상도 사라진다.

6. 보이스-코드 정규형 (BCNF)

3NF보다 엄격한 형태.
모든 함수적 종속 X → Y에서 X가 슈퍼키(Superkey)여야 한다.

3NF는 비키 속성이 다른 비키 속성을 결정하는 것만 금지하지만, BCNF는 키가 아닌 속성이 후보키의 일부를 결정하는 경우도 금지한다.

BCNF 위반이지만 3NF를 만족하는 예시

Course_Instructor(student_id, subject, instructor) 후보키: (student_id, subject), (student_id, instructor) FD 목록: (student_id, subject) → instructor ← 슈퍼키 결정 ✔ instructor → subject ← instructor는 슈퍼키가 아님 ✗

instructor → subject에서 instructor는 슈퍼키가 아니므로 BCNF 위반.

BCNF 변환

-- 교수가 어떤 과목을 담당하는지
CREATE TABLE Instructor_Subject (
  instructor VARCHAR(50) PRIMARY KEY,
  subject    VARCHAR(100) NOT NULL
);

-- 학생이 어떤 교수의 강의를 수강하는지
CREATE TABLE Student_Instructor (
  student_id CHAR(4),
  instructor VARCHAR(50) REFERENCES Instructor_Subject(instructor),
  PRIMARY KEY (student_id, instructor)
);

트레이드오프: BCNF 분해는 손실 없는 분해(Lossless Decomposition)를 보장하지만, 일부 FD를 보존하지 못할 수 있다. 3NF는 FD 보존을 항상 보장한다. 실무에서는 3NF까지 적용하는 경우가 많다.

7. 정규형 요약 비교

정규형핵심 조건제거되는 이상
1NF모든 속성값이 원자값 (반복 그룹 없음)다중값 속성으로 인한 이상
2NF1NF + 부분 함수적 종속 제거복합키 일부에 의존한 갱신/삽입/삭제 이상
3NF2NF + 이행적 함수적 종속 제거비키 속성 간 종속으로 인한 이상
BCNF3NF + 모든 결정자가 슈퍼키후보키를 부분 결정하는 비키 속성으로 인한 이상
비정규형 ─► 1NF ─► 2NF ─► 3NF ─► BCNF 원자값 부분FD 이행FD 슈퍼키만 결정자

8. 역정규화 (Denormalization)

정규화가 항상 최선은 아니다. 조인이 많아질수록 조회 성능이 저하될 수 있다.

정규화역정규화
데이터 중복 최소화의도적 중복 허용
쓰기(INSERT/UPDATE) 성능 유리읽기(SELECT) 성능 유리
갱신 이상 없음갱신 이상 위험 존재
조인 많아짐조인 줄어듦

역정규화가 적절한 상황

  • 읽기 비율이 압도적으로 높은 분석/리포팅 시스템 (OLAP)
  • 복잡한 다단 조인이 성능 병목임을 프로파일링으로 확인한 경우
  • 집계값을 미리 계산해 저장하는 요약 테이블 (Summary Table)

원칙: 역정규화는 측정 후 적용한다. 추측으로 먼저 적용하면 유지보수 복잡도만 늘어난다. 정규화 → 인덱스 최적화 → 필요 시 역정규화 순서가 권장된다.

9. 실습: 정규화 단계별 적용

다음 비정규 테이블을 BCNF까지 분해해 보자.

-- 비정규 테이블
OrderDetail(order_id, product_ids, customer_id, customer_city, product_name, unit_price, quantity)

-- 샘플 데이터
-- order_id | product_ids   | customer_id | customer_city | product_name | unit_price | quantity
-- O001     | P01,P02       | C01         | 서울           | (다중값 문제)  | ...        | ...

단계 1: 1NF — 원자값 보장

OrderDetail(order_id, product_id, customer_id, customer_city, product_name, unit_price, quantity)
-- PK: (order_id, product_id)

단계 2: 2NF — 부분 종속 제거

-- 부분 FD 발견:
--   customer_id → customer_city  (order_id만으로 결정 → 분리)
--   product_id  → product_name, unit_price (product_id만으로 결정 → 분리)

OrderItem(order_id, product_id, quantity)   -- PK: (order_id, product_id)
Order(order_id, customer_id)                -- PK: order_id
Customer(customer_id, customer_city)        -- PK: customer_id
Product(product_id, product_name, unit_price) -- PK: product_id

단계 3: 3NF — 이행 종속 확인

-- customer_id → zip_code → customer_city 같은 이행 종속이 있다면 추가 분리
-- 현재 예시에서는 customer_id → customer_city가 직접 종속이므로 3NF 만족

결과: 4개의 깔끔한 테이블 — 각 테이블은 하나의 주제만 담당하고, 이상 현상이 없다.

이해도 확인 퀴즈

Q1. 다음 함수적 종속에서 이행적 종속에 해당하는 것은?

해설 보기

이행적 종속은 X → Y → Z 형태로 X가 Z를 간접 결정하는 것이다. student_id → dept_id이고 dept_id → dept_name이므로 student_id가 dept_name을 이행 결정한다. 이를 제거하는 것이 3NF의 핵심이다.

Q2. 복합 기본키 (A, B)를 가진 테이블에서 비키 속성 C가 A만으로 결정될 때, 위반하는 정규형은?

해설 보기

C가 기본키 전체(A, B)가 아닌 일부(A)에 종속되는 것은 부분 함수적 종속이다. 2NF는 이러한 부분 종속을 금지한다.

Q3. BCNF와 3NF의 가장 중요한 차이점은?

해설 보기

BCNF는 3NF보다 엄격하다. 3NF를 만족하더라도, 비키 속성이 후보키의 일부를 결정(Instructor → Subject)하는 경우 BCNF를 위반한다. BCNF에서는 FD의 좌변이 반드시 슈퍼키여야 한다.

Q4. 역정규화(Denormalization)를 적용하기에 가장 적합한 상황은?

해설 보기

역정규화는 읽기 성능을 높이기 위해 의도적으로 중복을 허용하는 기법이다. 측정(프로파일링)을 먼저 하고, 실제 병목이 조인임을 확인한 후 적용해야 한다. OLTP 시스템에서는 갱신 이상이 심화되므로 일반적으로 적합하지 않다.

핵심 개념 요약

  • 함수적 종속 (FD): X → Y, X값이 같으면 Y값도 반드시 같다
  • 완전 FD: Y가 복합키 전체에 종속 / 부분 FD: Y가 키의 일부에 종속
  • 이행적 FD: X → Y → Z (X가 Z를 간접 결정)
  • 1NF: 원자값 보장 — 다중값·반복 그룹 제거
  • 2NF: 1NF + 부분 FD 제거 (복합키 테이블에서만 문제)
  • 3NF: 2NF + 이행적 FD 제거
  • BCNF: 3NF + 모든 결정자가 슈퍼키 (3NF보다 엄격)
  • 역정규화: 읽기 성능을 위해 의도적 중복 — 측정 후 적용
