1. 서버 측 프로그래밍 개요

애플리케이션이 매번 SQL 문을 네트워크로 전송하는 방식은 왕복 지연(round-trip)과 파싱·최적화 비용이 반복됩니다. 데이터베이스 서버 안에 로직을 저장해 두면 이 비용을 줄이고, 재사용성과 보안을 높일 수 있습니다.

저장 프로시저
이름 붙인 SQL + 로직 블록. 반환값 없거나 여러 결과셋 반환 가능.
사용자 정의 함수
단일 값(스칼라) 또는 테이블 반환. SELECT 절에서 호출 가능.
트리거
DML 이벤트(INSERT/UPDATE/DELETE)에 자동 반응하는 서버 측 핸들러.
클라이언트 DB 서버 ─────────────────────────────────────────────────── CALL get_order_summary(42) → [저장 프로시저 실행] ├─ 파싱: 1회만 수행 ├─ 최적화 계획 캐시 └─ 결과 반환 ←────────────────────────────────────────────────
MySQL vs PostgreSQL 문법 차이

이 레슨은 MySQL 8 / MariaDB 기준 문법을 주로 사용합니다. PostgreSQL은 PL/pgSQL 언어로 작성하며 CREATE FUNCTION으로 프로시저 역할까지 수행합니다. 핵심 개념은 같으나 키워드가 다를 수 있습니다.

2. 저장 프로시저 (Stored Procedure)

저장 프로시저는 미리 컴파일되어 DB 서버에 저장된 SQL 코드 블록입니다. CALL 문으로 호출하며, IN / OUT / INOUT 파라미터를 사용해 값을 주고받습니다.

기본 문법

-- 구분자 변경: 프로시저 본문 내 세미콜론과 구분하기 위함
DELIMITER $$

CREATE PROCEDURE procedure_name (
    IN  param1 INT,
    OUT param2 VARCHAR(100),
    INOUT param3 DECIMAL(10,2)
)
BEGIN
    -- SQL 로직
    SELECT name INTO param2 FROM products WHERE id = param1;
    SET param3 = param3 * 1.1;
END$$

DELIMITER ;

-- 호출
CALL procedure_name(10, @out_name, @amount);
SELECT @out_name, @amount;

파라미터 종류

종류방향설명
IN호출자 → 프로시저입력값. 프로시저 내부 변경이 호출자에 반영되지 않음.
OUT프로시저 → 호출자출력값. 초기값은 NULL.
INOUT양방향입력과 출력 모두 사용.

실전 예제 — 주문 처리 프로시저

DELIMITER $$

CREATE PROCEDURE place_order(
    IN  p_customer_id INT,
    IN  p_product_id  INT,
    IN  p_qty         INT,
    OUT p_order_id    INT,
    OUT p_message     VARCHAR(200)
)
BEGIN
    DECLARE v_stock INT;
    DECLARE v_price DECIMAL(10,2);

    -- 재고 확인
    SELECT stock_qty, price
    INTO   v_stock, v_price
    FROM   products
    WHERE  id = p_product_id
    FOR UPDATE;    -- 동시성 보호: 행 잠금

    IF v_stock < p_qty THEN
        SET p_order_id = -1;
        SET p_message  = '재고 부족';
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '재고 부족';
    END IF;

    -- 주문 생성
    INSERT INTO orders (customer_id, product_id, qty, total_price, created_at)
    VALUES (p_customer_id, p_product_id, p_qty, v_price * p_qty, NOW());

    SET p_order_id = LAST_INSERT_ID();

    -- 재고 차감
    UPDATE products SET stock_qty = stock_qty - p_qty WHERE id = p_product_id;

    SET p_message = CONCAT('주문 완료, ID: ', p_order_id);
END$$

DELIMITER ;

관리 명령어

-- 목록 조회
SHOW PROCEDURE STATUS WHERE Db = 'mydb';

-- 정의 확인
SHOW CREATE PROCEDURE place_order;

-- 삭제
DROP PROCEDURE IF EXISTS place_order;

3. 사용자 정의 함수 (User-Defined Function)

함수는 항상 단일 값을 반환하며, SELECT 절·WHERE 절 등 SQL 표현식 어디서든 호출할 수 있습니다. 반면 부작용(데이터 변경)은 원칙적으로 허용되지 않습니다.

스칼라 함수

DELIMITER $$

CREATE FUNCTION format_phone(raw_phone VARCHAR(20))
RETURNS VARCHAR(13)
DETERMINISTIC          -- 같은 입력 → 항상 같은 출력
NO SQL                 -- SQL 문 없음 (순수 로직)
BEGIN
    -- 숫자만 남기기
    DECLARE digits VARCHAR(20);
    SET digits = REGEXP_REPLACE(raw_phone, '[^0-9]', '');

    -- 010-XXXX-XXXX 형식
    IF LENGTH(digits) = 11 THEN
        RETURN CONCAT(
            SUBSTRING(digits, 1, 3), '-',
            SUBSTRING(digits, 4, 4), '-',
            SUBSTRING(digits, 8, 4)
        );
    END IF;
    RETURN raw_phone;   -- 변환 불가 시 원본 반환
END$$

DELIMITER ;

-- 사용
SELECT name, format_phone(phone) AS formatted_phone
FROM customers;

특성 지시자 (Characteristic)

지시자의미
DETERMINISTIC동일 입력 → 동일 출력. 쿼리 캐시 최적화 허용.
NOT DETERMINISTIC결과가 달라질 수 있음 (기본값).
NO SQLSQL 문을 포함하지 않음.
READS SQL DATASELECT만 수행.
MODIFIES SQL DATAINSERT/UPDATE/DELETE 포함.
MySQL 보안 설정

MySQL에서 UDF를 생성하려면 log_bin_trust_function_creators = 1 설정이 필요하거나 SUPER 권한이 있어야 합니다. 복제 환경에서는 바이너리 로그 안전성을 위해 반드시 DETERMINISTIC 또는 READS SQL DATA를 명시하세요.

4. 프로시저 vs 함수 비교

항목저장 프로시저사용자 정의 함수
반환값없거나 OUT 파라미터, 여러 결과셋반드시 단일 스칼라 값
호출 방법CALL proc()SELECT func() 또는 표현식
DML 허용가능 (INSERT/UPDATE/DELETE)제한적 (READS SQL DATA 이하 권장)
트랜잭션 제어COMMIT / ROLLBACK 가능불가
SQL 내 인라인 사용불가가능 (WHERE col = func(x))
주 용도복잡한 비즈니스 로직, 배치 처리값 변환, 계산, 포맷팅
언제 무엇을 쓸까?

데이터를 변경하거나 여러 단계 트랜잭션이 필요하면 프로시저. SELECT 절에서 컬럼처럼 쓰고 싶은 변환·계산은 함수.

5. 트리거 (Trigger)

트리거는 특정 테이블에 DML 이벤트가 발생할 때 자동으로 실행되는 코드 블록입니다. 명시적 호출 없이 DB 엔진이 자동 발동시킵니다.

기본 문법

DELIMITER $$

CREATE TRIGGER trigger_name
    {BEFORE | AFTER}
    {INSERT | UPDATE | DELETE}
ON table_name
FOR EACH ROW
BEGIN
    -- OLD.column : 변경 전 값 (UPDATE, DELETE)
    -- NEW.column : 변경 후 값 (INSERT, UPDATE)
    ...
END$$

DELIMITER ;

OLD / NEW 의사 레코드

이벤트OLD 사용 가능NEW 사용 가능
INSERT❌✅ (삽입될 값)
UPDATE✅ (수정 전 값)✅ (수정될 값)
DELETE✅ (삭제될 값)❌

실전 예제 — 가격 변경 이력 기록

-- 이력 테이블
CREATE TABLE product_price_log (
    log_id      INT AUTO_INCREMENT PRIMARY KEY,
    product_id  INT,
    old_price   DECIMAL(10,2),
    new_price   DECIMAL(10,2),
    changed_by  VARCHAR(50),
    changed_at  DATETIME
);

-- AFTER UPDATE 트리거
DELIMITER $$

CREATE TRIGGER trg_product_price_log
AFTER UPDATE ON products
FOR EACH ROW
BEGIN
    IF OLD.price <> NEW.price THEN
        INSERT INTO product_price_log
            (product_id, old_price, new_price, changed_by, changed_at)
        VALUES
            (OLD.id, OLD.price, NEW.price, USER(), NOW());
    END IF;
END$$

DELIMITER ;
BEFORE vs AFTER

BEFORE: 실제 행 변경 전 실행. NEW.column을 수정해 입력값 검증·보정 가능.
AFTER: 행 변경 후 실행. 이력 기록, 연관 테이블 갱신에 적합. NEW.column 수정 불가.

6. 트리거 종류와 활용 패턴

BEFORE INSERT — 데이터 검증 및 보정

DELIMITER $$

CREATE TRIGGER trg_orders_before_insert
BEFORE INSERT ON orders
FOR EACH ROW
BEGIN
    -- 수량이 0 이하이면 오류 발생
    IF NEW.qty <= 0 THEN
        SIGNAL SQLSTATE '45000'
            SET MESSAGE_TEXT = '주문 수량은 1 이상이어야 합니다.';
    END IF;

    -- 생성 시각 자동 설정
    SET NEW.created_at = IFNULL(NEW.created_at, NOW());
END$$

DELIMITER ;

AFTER DELETE — 소프트 삭제 이력

DELIMITER $$

CREATE TRIGGER trg_users_after_delete
AFTER DELETE ON users
FOR EACH ROW
BEGIN
    INSERT INTO deleted_users_log (user_id, email, deleted_at)
    VALUES (OLD.id, OLD.email, NOW());
END$$

DELIMITER ;

AFTER UPDATE — 파생 컬럼 동기화

-- orders 테이블에 total_amount 파생 컬럼 자동 갱신
DELIMITER $$

CREATE TRIGGER trg_order_items_sync
AFTER INSERT ON order_items
FOR EACH ROW
BEGIN
    UPDATE orders
    SET total_amount = (
        SELECT SUM(qty * unit_price) FROM order_items WHERE order_id = NEW.order_id
    )
    WHERE id = NEW.order_id;
END$$

DELIMITER ;
트리거 주의사항
  • 숨겨진 부작용: 트리거는 DML 실행 시 자동 발동되므로 디버깅이 어렵습니다.
  • 연쇄 트리거: 트리거 내 DML이 다른 트리거를 호출할 수 있어 실행 순서가 복잡해집니다.
  • 성능 영향: 모든 관련 행 변경에 실행되므로 대량 INSERT/UPDATE 시 성능 저하.
  • MySQL 제한: 같은 테이블의 같은 이벤트에 같은 타이밍(BEFORE/AFTER)을 가진 트리거는 하나만 허용 (MySQL 8.0.1 이전).

트리거 관리

-- 목록 확인
SHOW TRIGGERS FROM mydb;

-- 특정 테이블 트리거
SHOW TRIGGERS FROM mydb LIKE 'orders';

-- 삭제
DROP TRIGGER IF EXISTS trg_product_price_log;

7. 변수 · 제어 흐름

저장 프로시저와 함수 내에서 사용하는 변수와 제어 구조입니다.

변수 선언과 할당

BEGIN
    -- 로컬 변수: DECLARE는 BEGIN 직후에 위치해야 함
    DECLARE v_total     INT      DEFAULT 0;
    DECLARE v_name      VARCHAR(100);
    DECLARE v_done      BOOLEAN  DEFAULT FALSE;

    -- SET 으로 할당
    SET v_total = 100;

    -- SELECT INTO 로 할당
    SELECT COUNT(*) INTO v_total FROM orders WHERE customer_id = 1;

    -- 세션 변수 (@ 접두사, DECLARE 불필요)
    SET @session_var = 'hello';
END

조건문

-- IF / ELSEIF / ELSE
IF v_total > 1000 THEN
    SET v_grade = 'VIP';
ELSEIF v_total > 500 THEN
    SET v_grade = 'GOLD';
ELSE
    SET v_grade = 'REGULAR';
END IF;

-- CASE
CASE v_status
    WHEN 1 THEN SET v_label = '대기';
    WHEN 2 THEN SET v_label = '처리중';
    WHEN 3 THEN SET v_label = '완료';
    ELSE        SET v_label = '알 수 없음';
END CASE;

반복문

-- WHILE
SET v_i = 1;
WHILE v_i <= 10 DO
    INSERT INTO temp_numbers (n) VALUES (v_i);
    SET v_i = v_i + 1;
END WHILE;

-- REPEAT (do-while 형태)
REPEAT
    SET v_i = v_i + 1;
UNTIL v_i > 10
END REPEAT;

-- LOOP + LEAVE (break)
my_loop: LOOP
    IF v_i > 10 THEN
        LEAVE my_loop;
    END IF;
    SET v_i = v_i + 1;
END LOOP my_loop;

커서 (Cursor)

DELIMITER $$

CREATE PROCEDURE process_overdue_orders()
BEGIN
    DECLARE v_order_id  INT;
    DECLARE v_done      BOOLEAN DEFAULT FALSE;

    -- 커서 선언
    DECLARE cur CURSOR FOR
        SELECT id FROM orders WHERE status = 'PENDING' AND created_at < NOW() - INTERVAL 7 DAY;

    -- NOT FOUND 핸들러: 커서 끝 감지
    DECLARE CONTINUE HANDLER FOR NOT FOUND SET v_done = TRUE;

    OPEN cur;

    read_loop: LOOP
        FETCH cur INTO v_order_id;
        IF v_done THEN
            LEAVE read_loop;
        END IF;

        UPDATE orders SET status = 'CANCELLED' WHERE id = v_order_id;
    END LOOP;

    CLOSE cur;
END$$

DELIMITER ;

8. 예외 처리

DECLARE HANDLER

DELIMITER $$

CREATE PROCEDURE safe_transfer(
    IN p_from_account INT,
    IN p_to_account   INT,
    IN p_amount       DECIMAL(15,2)
)
BEGIN
    DECLARE v_balance DECIMAL(15,2);
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        -- 오류 발생 시 롤백 후 오류 재발생
        ROLLBACK;
        RESIGNAL;
    END;

    START TRANSACTION;

    SELECT balance INTO v_balance FROM accounts WHERE id = p_from_account FOR UPDATE;

    IF v_balance < p_amount THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '잔액 부족';
    END IF;

    UPDATE accounts SET balance = balance - p_amount WHERE id = p_from_account;
    UPDATE accounts SET balance = balance + p_amount WHERE id = p_to_account;

    COMMIT;
END$$

DELIMITER ;
핸들러 종류동작
EXIT HANDLER핸들러 실행 후 현재 BEGIN…END 블록 종료.
CONTINUE HANDLER핸들러 실행 후 오류 발생 다음 문장부터 계속 실행.

SIGNAL — 사용자 정의 오류 발생

-- SQLSTATE 값: '45000'은 사용자 정의 오류용 범용 코드
SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = '유효하지 않은 파라미터',
        MYSQL_ERRNO  = 1644;

9. 설계 지침과 주의사항

언제 서버 측 로직을 써야 하나?

적합한 경우
  • 복수 테이블에 걸친 원자적 처리
  • 재사용되는 비즈니스 규칙 강제
  • 보안상 직접 테이블 접근 제한
  • 복잡한 이력·감사 로그
피해야 하는 경우
  • 버전 관리가 어려운 복잡한 비즈니스 로직
  • 애플리케이션 레이어 테스트가 중요한 경우
  • DB 벤더 이식성이 필요한 경우
  • 단순 CRUD (불필요한 복잡성)

핵심 체크리스트

  • 프로시저·함수 이름은 동사형으로: get_, create_, process_
  • 트리거 이름에 테이블·타이밍·이벤트 포함: trg_orders_before_insert
  • 트리거 내에서 같은 테이블을 UPDATE하지 않기 (무한 루프 위험)
  • 프로시저는 단일 책임 원칙: 하나의 목적만 수행
  • 커서 대신 집합 연산(SET-based) 우선 고려: 성능 차이 큼
  • SHOW CREATE PROCEDURE 또는 information_schema.ROUTINES로 정의 관리
PostgreSQL에서의 동등 구조

PostgreSQL은 CREATE FUNCTION ... LANGUAGE plpgsql으로 프로시저와 함수 모두 구현합니다. 트리거 함수는 RETURNS TRIGGER로 선언하고, CREATE TRIGGER로 연결합니다. NEW/OLD 레코드와 TG_OP로 이벤트 종류를 확인합니다.

10. 확인 문제

Q1. 저장 프로시저와 함수의 가장 큰 차이점은?

정답 보기

함수는 반드시 단일 값을 반환하며 SQL 표현식 안에서 호출할 수 있습니다. 프로시저는 반환값이 없거나 OUT 파라미터와 여러 결과셋을 돌려줄 수 있으며, CALL로만 호출합니다. 또한 프로시저는 트랜잭션 제어(COMMIT/ROLLBACK)가 가능하지만 함수는 불가합니다.

Q2. BEFORE INSERT 트리거에서 NEW.column을 수정하면 어떻게 됩니까?

정답 보기

BEFORE 트리거에서 NEW.column = 값으로 변경하면, 실제 INSERT되는 행의 값이 해당 값으로 치환됩니다. 이를 이용해 입력 전 검증이나 기본값 자동 설정을 구현할 수 있습니다. AFTER 트리거에서는 NEW.column을 수정할 수 없습니다.

Q3. 커서(Cursor) 사용 시 NOT FOUND 핸들러가 필요한 이유는?

정답 보기

커서로 마지막 행까지 FETCH한 후 다시 FETCH하면 MySQL은 NOT FOUND 조건을 발생시킵니다. 이를 처리하는 CONTINUE HANDLER FOR NOT FOUND가 없으면 루프를 종료할 방법이 없어 무한 루프 또는 오류가 발생합니다. 핸들러에서 플래그 변수를 TRUE로 설정하고 루프에서 해당 플래그를 확인해 LEAVE로 빠져나옵니다.

Q4. 다음 중 트리거 사용이 부적절한 상황은? (복수 선택 가능)

① 가격 변경 이력 자동 기록
② 주문 수량 음수 방지
