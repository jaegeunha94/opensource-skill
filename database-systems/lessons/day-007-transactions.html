Day 7 — 트랜잭션과 동시성 제어

ACID 격리 수준 Lock MVCC

학습 목표
트랜잭션의 개념과 ACID 속성을 설명할 수 있다.
Commit·Rollback·Savepoint를 올바르게 사용할 수 있다.
4가지 격리 수준과 각각에서 발생 가능한 이상 현상(Dirty Read·Non-repeatable Read·Phantom Read)을 구분할 수 있다.
Lock 기반 동시성 제어와 MVCC의 차이를 이해한다.

1. 트랜잭션이란?

트랜잭션(Transaction)은 하나의 논리적 작업 단위로 묶인 SQL 문의 집합이다. "전부 성공하거나, 전부 실패하거나(All or Nothing)"가 핵심 원칙이다.

1.1 고전적 예시 — 계좌 이체

-- 계좌 A → B로 50,000원 이체
BEGIN;

UPDATE accounts SET balance = balance - 50000 WHERE id = 'A';
UPDATE accounts SET balance = balance + 50000 WHERE id = 'B';

COMMIT;

첫 번째 UPDATE 성공 후 두 번째 UPDATE 전에 오류가 발생하면?
트랜잭션 없이는 A 잔액만 감소하고 B는 그대로인 데이터 불일치가 남는다.
트랜잭션 안에서는 ROLLBACK으로 A도 원래 잔액으로 복구된다.

[BEGIN] │ ├─ UPDATE A: balance - 50000 ← OK │ ├─ UPDATE B: balance + 50000 ← 오류 발생! │ └─ ROLLBACK → A 잔액 복구, 아무 변경도 없던 상태로 회귀

2. ACID 속성

데이터베이스 트랜잭션이 보장해야 하는 4가지 속성이다.

① Atomicity — 원자성

트랜잭션 안의 모든 작업은 전부 완료되거나 전부 취소된다. 부분 완료는 허용되지 않는다.

구현: Undo Log (롤백 시 이전 값 복원)

② Consistency — 일관성

트랜잭션 실행 전후에 데이터베이스는 항상 정의된 무결성 제약조건을 만족해야 한다.

예: 잔액은 음수가 될 수 없다 (CHECK 제약).

③ Isolation — 격리성

동시에 실행 중인 트랜잭션들은 서로 영향을 주지 않는다. 각 트랜잭션은 혼자 실행되는 것처럼 보여야 한다.

구현: Lock, MVCC, 격리 수준 설정

④ Durability — 지속성

커밋된 트랜잭션의 결과는 시스템 장애가 발생해도 영구적으로 유지된다.

구현: Redo Log (WAL — Write-Ahead Logging)

ACID vs BASE
NoSQL 시스템은 가용성과 확장성을 위해 ACID 대신 BASE(Basically Available, Soft state, Eventual consistency)를 선택하는 경우가 많다. Day 10(NoSQL)에서 자세히 다룬다.

3. Commit · Rollback · Savepoint

3.1 기본 제어 명령

명령의미언제 사용
BEGIN / START TRANSACTION트랜잭션 시작명시적 트랜잭션 개시
COMMIT변경 사항을 영구 반영모든 작업이 정상 완료되었을 때
ROLLBACK변경 사항을 전부 취소오류 발생 또는 의도적 취소
SAVEPOINT name중간 저장점 생성부분 롤백이 필요할 때
ROLLBACK TO SAVEPOINT name해당 저장점까지만 롤백전체 취소 없이 일부만 되돌릴 때
RELEASE SAVEPOINT name저장점 삭제더 이상 필요 없을 때

3.2 Savepoint 예시

BEGIN;

INSERT INTO orders (id, product, qty) VALUES (1001, 'notebook', 2);
SAVEPOINT after_order;       -- 저장점 생성

INSERT INTO payments (order_id, amount) VALUES (1001, 30000);
-- 결제 처리 중 오류 감지

ROLLBACK TO SAVEPOINT after_order;   -- 결제만 취소, 주문은 유지

-- 재시도 또는 다른 처리...
COMMIT;

3.3 자동 커밋 (Auto-commit)

MySQL, PostgreSQL 등 대부분의 DBMS는 기본적으로 Auto-commit ON 상태이다. 이 경우 개별 SQL 문 하나하나가 독립 트랜잭션으로 처리된다. 여러 SQL을 하나의 트랜잭션으로 묶으려면 반드시 BEGIN으로 명시해야 한다. 

-- MySQL: 자동 커밋 끄기
SET autocommit = 0;

-- PostgreSQL: 기본값 ON. BEGIN 호출 시 트랜잭션 블록 진입
BEGIN;
...
COMMIT;

4. 동시성 문제 — 이상 현상

여러 트랜잭션이 동시에 실행될 때 격리성이 부족하면 다음 세 가지 이상 현상이 발생한다.

4.1 Dirty Read (오손 읽기)

아직 커밋되지 않은 트랜잭션의 변경 내용을 다른 트랜잭션이 읽는다.

T1: BEGIN T1: UPDATE accounts SET balance = 100000 WHERE id = 'A'; -- 아직 커밋 안 함 T2: SELECT balance FROM accounts WHERE id = 'A'; -- 100000 읽음 (더티 리드!) T1: ROLLBACK -- 원래 80000으로 복구 T2가 읽은 100000은 존재하지 않았던 값 → 불일치

4.2 Non-repeatable Read (반복 불가 읽기)

같은 트랜잭션 안에서 같은 행을 두 번 읽었을 때 값이 달라진다.

T1: SELECT balance FROM accounts WHERE id = 'A'; -- 결과: 80000 T2: UPDATE accounts SET balance = 90000 WHERE id = 'A'; COMMIT; T1: SELECT balance FROM accounts WHERE id = 'A'; -- 결과: 90000 ← 달라짐!

4.3 Phantom Read (유령 읽기)

같은 트랜잭션 안에서 같은 조건으로 두 번 SELECT 했을 때 결과 행의 수가 달라진다.

T1: SELECT * FROM orders WHERE amount > 10000; -- 5행 반환 T2: INSERT INTO orders (amount) VALUES (15000); COMMIT; T1: SELECT * FROM orders WHERE amount > 10000; -- 6행 반환 ← 유령 행 등장!

Non-repeatable Read vs Phantom Read
Non-repeatable Read는 기존 행의 값이 변경되는 문제,
Phantom Read는 행의 개수(집합)가 달라지는 문제다.

5. 트랜잭션 격리 수준

SQL 표준(ISO/IEC 9075)은 이상 현상을 허용 범위에 따라 4가지 격리 수준을 정의한다. 수준이 높을수록 데이터 정확성은 높아지고, 동시성(성능)은 낮아진다.

격리 수준 Dirty Read Non-repeatable Read Phantom Read 성능
READ UNCOMMITTED 발생 가능 발생 가능 발생 가능 최고
READ COMMITTED 방지 발생 가능 발생 가능 높음
REPEATABLE READ 방지 방지 발생 가능
(InnoDB는 방지)		 중간
SERIALIZABLE 방지 방지 방지 최저

5.1 격리 수준 설정

-- MySQL / MariaDB
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- PostgreSQL
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 세션 전체에 적용
ALTER SESSION SET ISOLATION LEVEL SERIALIZABLE;  -- Oracle

5.2 READ UNCOMMITTED

커밋되지 않은 데이터도 읽는다. 가장 낮은 격리 수준. Dirty Read가 발생하므로 데이터 정합성이 중요하지 않은 통계성 읽기 등에 제한적으로 사용한다.

5.3 READ COMMITTED

커밋된 데이터만 읽는다. Dirty Read는 방지하지만 Non-repeatable Read는 허용한다. Oracle, PostgreSQL의 기본값. 일반적인 OLTP 환경에서 널리 사용된다.

5.4 REPEATABLE READ

같은 트랜잭션 안에서 같은 행을 반복 조회하면 항상 같은 값을 반환한다. MySQL InnoDB의 기본값. InnoDB는 MVCC(갭 락 포함)를 통해 Phantom Read까지 방지한다.

5.5 SERIALIZABLE

가장 높은 격리 수준. 트랜잭션들이 마치 순차적으로 실행된 것처럼 보장한다. 모든 이상 현상을 방지하지만 Lock 경합으로 인한 성능 저하가 크다. 금융, 재고 관리처럼 정합성이 절대적으로 중요한 경우에 사용한다.

6. 동시성 제어 메커니즘

6.1 Lock 기반 제어

트랜잭션이 데이터에 접근하기 전에 Lock을 획득하는 방식이다.

Lock 종류다른 명칭읽기 허용쓰기 허용
공유 잠금 (Shared Lock) S Lock, 읽기 Lock O (여러 트랜잭션 동시 가능) X
배타 잠금 (Exclusive Lock) X Lock, 쓰기 Lock X X (오직 해당 트랜잭션만)
호환성 행렬: S Lock X Lock S Lock 허용 충돌 X Lock 충돌 충돌

6.2 Lock 세분화

범위설명장점단점
테이블 락 테이블 전체에 Lock 오버헤드 낮음 동시성 극도로 낮음
행 락 (Row Lock) 개별 행에만 Lock 동시성 높음 많은 Lock 관리 비용
페이지 락 디스크 페이지 단위 중간 절충 구현 DBMS 제한적
갭 락 (Gap Lock) 인덱스 범위 사이 공간 Phantom Read 방지 InnoDB REPEATABLE READ 이상에서만

6.3 교착 상태 (Deadlock)

두 트랜잭션이 서로 상대방의 Lock 해제를 기다리는 상태이다.

T1: Lock(A) 획득 → Lock(B) 요청 → 대기 T2: Lock(B) 획득 → Lock(A) 요청 → 대기 → 영원히 대기 = 교착 상태

DBMS는 Deadlock Detector가 주기적으로 탐지하여 희생자(Victim) 트랜잭션을 롤백시킨다.

Deadlock 예방 팁
① Lock 획득 순서를 항상 동일하게 통일한다.
② 트랜잭션을 짧게 유지하고 빠르게 커밋한다.
③ 불필요한 대량 Lock을 피한다.

6.4 MVCC (Multi-Version Concurrency Control)

Lock 없이 데이터의 여러 버전을 관리하여 읽기와 쓰기가 서로를 차단하지 않도록 한다. 읽기 트랜잭션은 스냅샷을 통해 자신이 시작된 시점의 데이터를 본다.

시간 ──────────────────────────────────────────────────▶ T_write: BEGIN → UPDATE row X (값 80→90) → COMMIT T_read: BEGIN (스냅샷 T=10 기준) → SELECT row X → 80 반환 (이전 버전) ← 쓰기가 커밋되었어도 영향 없음!
Lock 기반MVCC
읽기-쓰기 충돌 읽기도 대기 읽기 대기 없음
오래된 버전 관리 불필요 Undo 세그먼트 사용
대표 DBMS MyISAM, 일부 NoSQL InnoDB, PostgreSQL, Oracle

PostgreSQL의 MVCC
PostgreSQL은 UPDATE를 내부적으로 "이전 행 삭제 + 새 행 삽입"으로 처리한다. 오래된 버전 행은 VACUUM이 정리한다. MySQL InnoDB는 Undo Log에 이전 버전을 별도로 저장하는 방식을 쓴다.

7. 실전 패턴 — 격리 수준별 SQL

7.1 SELECT FOR UPDATE (비관적 잠금)

읽은 행에 즉시 배타 락을 거는 패턴. 이후 변경이 확실한 경우 사용한다.

BEGIN;

-- 재고를 읽으면서 동시에 잠금
SELECT stock FROM products WHERE id = 42 FOR UPDATE;

-- 재고 감소
UPDATE products SET stock = stock - 1 WHERE id = 42;

COMMIT;

7.2 낙관적 잠금 (Optimistic Locking)

Lock 대신 버전 번호(version) 컬럼으로 충돌을 감지한다.

-- 1. 데이터 조회
SELECT id, stock, version FROM products WHERE id = 42;
-- 결과: (42, 10, 5)

-- 2. 업데이트 시도 — 버전이 일치할 때만 성공
UPDATE products
SET stock = 9, version = version + 1
WHERE id = 42 AND version = 5;

-- 영향받은 행이 0 → 동시 수정 감지 → 재시도 또는 오류 처리

7.3 SKIP LOCKED — 큐 처리

이미 잠긴 행을 건너뛰고 처리 가능한 행만 선택. 작업 큐 패턴에서 유용하다.

-- PostgreSQL / MySQL 8+
BEGIN;
SELECT * FROM job_queue
WHERE status = 'pending'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;

UPDATE job_queue SET status = 'processing' WHERE id = :picked_id;
COMMIT;

8. 트랜잭션 설계 원칙

짧게 유지 — 트랜잭션이 길수록 Lock 점유 시간이 늘어 동시성이 저하된다.
최소한의 범위 — 꼭 필요한 데이터에만 Lock을 건다.
일관된 Lock 순서 — Deadlock 예방을 위해 여러 테이블을 접근할 때 항상 같은 순서로 Lock을 획득한다.
읽기 전용 트랜잭션 구분 — 읽기만 하는 경우 SET TRANSACTION READ ONLY를 선언하면 DBMS 최적화 힌트가 된다.
오류 처리 — 예외 발생 시 반드시 ROLLBACK을 호출해 자원을 해제한다.

확인 퀴즈

Q1. ACID의 I(Isolation)를 구현하는 두 가지 주요 메커니즘은?

Lock 기반 제어와 MVCC(Multi-Version Concurrency Control).

Q2. READ COMMITTED 격리 수준에서 방지되지 않는 이상 현상 두 가지는?

Non-repeatable Read와 Phantom Read. Dirty Read는 방지되지만 나머지 두 이상 현상은 허용된다.

Q3. Dirty Read가 발생할 수 있는 격리 수준은?

READ UNCOMMITTED. 이 수준에서만 아직 커밋되지 않은 데이터를 읽을 수 있다.

Q4. 두 트랜잭션이 서로 상대방의 Lock 해제를 기다리는 상황을 무엇이라 하며, DBMS는 이를 어떻게 해결하나?

교착 상태(Deadlock). DBMS의 Deadlock Detector가 주기적으로 순환 대기를 탐지하고, 희생자(Victim) 트랜잭션을 선택하여 강제 롤백시킨다.

Q5. MVCC에서 읽기 트랜잭션이 쓰기 트랜잭션과 충돌하지 않는 이유는?

읽기 트랜잭션은 자신이 시작된 시점의 스냅샷(이전 버전 데이터)을 사용하기 때문에 쓰기 트랜잭션이 동시에 수정하더라도 Lock을 기다리지 않는다.

Q6. MySQL InnoDB의 기본 격리 수준은 무엇이며, Phantom Read를 어떻게 방지하나?

REPEATABLE READ가 기본값이다. InnoDB는 MVCC와 Gap Lock(갭 락)을 함께 사용하여 인덱스 범위 사이의 삽입을 차단함으로써 Phantom Read까지 방지한다.
