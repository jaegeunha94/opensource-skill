1. 인덱스가 필요한 이유

인덱스(Index)가 없으면 DB 엔진은 풀 테이블 스캔(Full Table Scan) — 즉 모든 행을 하나씩 검사해야 합니다. 행이 100만 개면 100만 번의 비교가 필요합니다.

핵심 비유
두꺼운 사전에서 단어를 찾을 때, 처음 페이지부터 넘기는 것이 풀 스캔, 맨 뒤 색인 페이지를 보는 것이 인덱스 스캔입니다.

시간 복잡도 비교

방법시간 복잡도예시 (1억 건)
풀 테이블 스캔O(N)최악 1억 번 비교
B-Tree 인덱스O(log N)약 27번 비교
해시 인덱스O(1)1번 (등가 검색만)

인덱스의 대가

  • 저장 공간 증가: 인덱스 자체도 디스크를 차지합니다.
  • 쓰기 성능 저하: INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신됩니다.
  • 유지보수 비용: 인덱스가 많을수록 옵티마이저가 최적 계획을 고르는 데 오래 걸립니다.
경험 법칙
읽기(SELECT) 가 잦고 쓰기(INSERT/UPDATE/DELETE) 가 적은 컬럼에 인덱스를 걸어야 효과적입니다.

2. B-Tree 인덱스 구조

MySQL(InnoDB), PostgreSQL 등 대부분의 RDBMS는 기본 인덱스로 B+Tree를 사용합니다.

B+Tree란?

B+Tree는 B-Tree의 변형으로 다음 특성을 가집니다.

  • 균형 트리(Balanced Tree): 모든 리프 노드가 같은 깊이에 위치합니다.
  • 내부 노드: 키(key)만 저장하고, 자식 노드로 향하는 포인터를 가집니다.
  • 리프 노드: 실제 키와 행 포인터(또는 PK)를 저장하며, 링크드 리스트로 연결됩니다.
  • 리프 링크 덕분에 범위 검색(BETWEEN, <, >, ORDER BY)이 빠릅니다.
[ 30 | 70 ] ← 루트 노드 / | \ [10|20] [40|60] [80|90] ← 내부 노드 / | \ ... [1..9] [10..19] [20..29] ... ← 리프 노드 (← → 링크드 리스트) ↓ ↓ ↓ 행 포인터 / PK 값

클러스터링 인덱스 vs 세컨더리 인덱스

구분클러스터링 인덱스세컨더리 인덱스
정의행 데이터 자체를 인덱스 순서로 물리적으로 정렬별도 트리, 리프에 PK 값 저장
개수테이블당 1개 (보통 PRIMARY KEY)여러 개 생성 가능
InnoDB 특성PK = 클러스터링 인덱스PK를 통해 실제 행 조회
PK 없을 때내부적으로 rowid 자동 생성-
InnoDB 핵심
InnoDB에서 세컨더리 인덱스 리프에는 실제 행 주소 대신 PK 값이 저장됩니다. 따라서 세컨더리 인덱스로 검색하면 리프 도달 후 다시 클러스터링 인덱스를 탐색해 실제 행을 가져옵니다 — 이를 이중 탐색(double lookup)이라 합니다.

인덱스 생성 문법

-- 단일 컬럼 인덱스
CREATE INDEX idx_users_email ON users(email);

-- 유니크 인덱스
CREATE UNIQUE INDEX idx_users_email_uniq ON users(email);

-- 복합 인덱스
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);

-- 인덱스 삭제
DROP INDEX idx_users_email ON users;   -- MySQL
DROP INDEX idx_users_email;            -- PostgreSQL

3. 인덱스 종류

B-Tree
기본. 범위·등가·정렬 모두 지원. 대부분의 상황에 적합.
Hash
등가(=) 검색만 O(1). 범위·ORDER BY 불가. Memory 엔진에서 사용.
Full-Text
자연어 텍스트 검색. LIKE '%keyword%' 대신 MATCH…AGAINST 사용.
Spatial (R-Tree)
지리 데이터(GIS). 위치 기반 검색. PostgreSQL PostGIS 등.
함수 기반 인덱스
표현식 결과에 인덱스. MySQL 8+, PostgreSQL. LOWER(email) 등.
부분 인덱스
WHERE 조건을 만족하는 행에만 인덱스. PostgreSQL 지원.

Full-Text 인덱스 예시

-- MySQL Full-Text
ALTER TABLE articles ADD FULLTEXT INDEX idx_ft_content(title, body);

SELECT * FROM articles
WHERE MATCH(title, body) AGAINST ('데이터베이스 인덱스' IN BOOLEAN MODE);

-- LIKE '%...%' 는 앞에 와일드카드가 있으면 인덱스를 사용하지 못함
-- Full-Text로 대체해야 성능 확보 가능

부분 인덱스 예시 (PostgreSQL)

-- 미처리 주문만 인덱스 (status = 'pending')
CREATE INDEX idx_orders_pending ON orders(created_at)
WHERE status = 'pending';

-- 전체 주문 중 미처리가 소수일 때 매우 효과적

4. 복합 인덱스와 열 순서

복합 인덱스(Composite Index)는 여러 컬럼을 하나의 인덱스로 결합합니다. 열의 순서가 성능을 결정합니다.

왼쪽 접두사 원칙 (Leftmost Prefix Rule)

CREATE INDEX idx ON t(a, b, c) 인덱스가 있을 때:

WHERE 조건인덱스 사용 여부사용되는 컬럼
WHERE a = 1✅ 사용a
WHERE a = 1 AND b = 2✅ 사용a, b
WHERE a = 1 AND b = 2 AND c = 3✅ 사용a, b, c
WHERE b = 2❌ 사용 불가—
WHERE a = 1 AND c = 3⚠️ 부분 사용a만
WHERE a BETWEEN 1 AND 5 AND b = 2⚠️ 부분 사용a 범위 후 b 필터링

컬럼 순서 설계 원칙

  1. 등가 조건(=)이 자주 쓰이는 컬럼을 앞에 배치합니다.
  2. 카디널리티가 높은 컬럼(=고유값이 많은)을 앞에 배치하면 초기 필터링 효과가 큽니다.
  3. 범위 조건(BETWEEN, <, LIKE 'abc%')이 있는 컬럼은 뒤에 배치합니다.
  4. ORDER BY에 쓰이는 컬럼도 인덱스에 포함하면 정렬 비용을 절감할 수 있습니다.

실전 예시

-- 주문 테이블 예시: user_id로 필터링 후 최신순 정렬
SELECT * FROM orders
WHERE user_id = 42 AND status = 'paid'
ORDER BY created_at DESC
LIMIT 10;

-- 좋은 인덱스 설계
CREATE INDEX idx_orders_user_status_date
ON orders(user_id, status, created_at DESC);
-- user_id (등가) → status (등가) → created_at (정렬) 순서

5. 커버링 인덱스 (Covering Index)

커버링 인덱스란 쿼리가 필요로 하는 모든 컬럼이 인덱스 안에 포함되어 있어, 실제 테이블 행을 전혀 읽지 않고 인덱스만으로 결과를 반환하는 경우를 말합니다.

일반 인덱스 조회: ① 인덱스 B+Tree 탐색 → 리프에서 PK 획득 ② PK로 클러스터링 인덱스 재탐색 → 실제 행 읽기 ← 추가 I/O 발생 커버링 인덱스 조회: ① 인덱스 B+Tree 탐색 → 리프에서 바로 결과 반환 ← I/O 절감!

커버링 인덱스 구현

-- 쿼리: 특정 사용자의 최근 주문 ID와 금액만 조회
SELECT order_id, amount FROM orders
WHERE user_id = 42
ORDER BY created_at DESC
LIMIT 5;

-- user_id, created_at, order_id, amount 모두 인덱스에 포함
CREATE INDEX idx_covering ON orders(user_id, created_at DESC, order_id, amount);
-- SELECT 컬럼(order_id, amount)까지 인덱스에 넣어야 커버링 인덱스로 동작

-- EXPLAIN 결과에서 "Using index" 표시 → 커버링 인덱스 사용 중
-- "Using index condition" → 인덱스로 필터링하나 행 접근 필요
PostgreSQL Include 컬럼
PostgreSQL 11+에서는 INCLUDE 절로 인덱스 트리 탐색에는 사용하지 않지만 커버링 목적으로 포함할 컬럼을 지정할 수 있습니다. 
CREATE INDEX idx_cov ON orders(user_id, created_at) INCLUDE (order_id, amount);

6. EXPLAIN으로 실행 계획 읽기

EXPLAIN은 옵티마이저가 어떤 순서로, 어떤 접근 방법으로 쿼리를 실행할지 보여줍니다. 쿼리 튜닝의 첫 단계입니다.

MySQL EXPLAIN 주요 컬럼

EXPLAIN SELECT o.order_id, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'pending'
ORDER BY o.created_at DESC;
컬럼의미체크 포인트

rows읽어야 하는 예상 행 수작을수록 좋음
filtered조건 후 남는 행의 비율(%)높을수록 효율적

EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 42 AND status = 'pending';

7. 슬로우 쿼리 분석과 튜닝

MySQL 슬로우 쿼리 로그 활성화

-- 현재 설정 확인
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 활성화 (my.cnf 또는 동적 설정)
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;           -- 1초 이상 걸리는 쿼리 기록
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- 인덱스 미사용 쿼리도 기록
SET GLOBAL log_queries_not_using_indexes = 'ON';

mysqldumpslow 분석

# 상위 10개 느린 쿼리 (평균 실행 시간 순)
mysqldumpslow -s at -t 10 /var/log/mysql/slow.log

# 호출 횟수 기준 상위 10개
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

단계별 쿼리 튜닝 프로세스

  1. 슬로우 로그에서 문제 쿼리 식별 — 빈도 × 실행 시간 기준으로 우선순위 결정
  2. EXPLAIN 실행 — type=ALL, Using filesort, Using temporary 등 확인
  3. 인덱스 추가/수정 — WHERE, JOIN, ORDER BY, GROUP BY 컬럼 기준
  4. 쿼리 재작성 — 서브쿼리 → JOIN, 불필요한 함수 제거
  5. 스키마 개선 — 정규화/역정규화 재검토, 파티셔닝 고려
  6. 측정 후 확인 — EXPLAIN ANALYZE로 before/after 비교

자주 발생하는 슬로우 쿼리 패턴

패턴 1: 인덱스 컬럼에 함수 적용

-- ❌ 인덱스 미사용 (email 컬럼 인덱스가 있어도 무력화)
SELECT * FROM users WHERE UPPER(email) = 'USER@EXAMPLE.COM';

-- ✅ 개선: 데이터를 소문자로 저장하거나 함수 기반 인덱스 사용
SELECT * FROM users WHERE email = LOWER('USER@EXAMPLE.COM');

-- 또는 함수 기반 인덱스 (MySQL 8+)
CREATE INDEX idx_email_lower ON users((LOWER(email)));

패턴 2: 선행 와일드카드 LIKE

-- ❌ 인덱스 미사용
SELECT * FROM products WHERE name LIKE '%phone%';

-- ✅ Full-Text 인덱스로 대체
SELECT * FROM products
WHERE MATCH(name) AGAINST ('phone' IN BOOLEAN MODE);

패턴 3: 암묵적 형변환

-- ❌ phone_number가 VARCHAR인데 숫자로 비교 → 형변환 → 인덱스 무력화
SELECT * FROM users WHERE phone_number = 01012345678;

-- ✅ 타입 일치
SELECT * FROM users WHERE phone_number = '010-1234-5678';

패턴 4: N+1 쿼리 문제

-- ❌ 애플리케이션에서 루프로 쿼리 반복 (N+1)
-- 사용자 100명 조회 → 각 사용자 주문 100번 조회 = 101번 쿼리

-- ✅ JOIN으로 한 번에
SELECT u.id, u.name, COUNT(o.order_id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name;

패턴 5: SELECT * 남용

-- ❌ 불필요한 컬럼까지 모두 읽음
SELECT * FROM orders WHERE user_id = 42;

-- ✅ 필요한 컬럼만 지정 (커버링 인덱스 가능성도 높아짐)
SELECT order_id, status, total_amount FROM orders WHERE user_id = 42;

8. 인덱스 안티패턴

안티패턴문제해결책
인덱스 과다 생성 INSERT/UPDATE/DELETE 속도 저하, 옵티마이저 혼란 실제 쿼리 패턴 분석 후 필요한 것만 생성
카디널리티 낮은 컬럼 단독 인덱스 성별(M/F), 불리언 등 — 풀 스캔과 차이 없음 다른 컬럼과 복합 인덱스로 결합
PK를 UUID(v4)로 사용 완전 무작위 → B+Tree 삽입 시 페이지 분할 빈번 UUID v7(시간 정렬), ULID, 또는 AUTO_INCREMENT 사용
인덱스 컬럼 OR 조건 WHERE a = 1 OR b = 2 — 인덱스 효율 저하 각 조건별 쿼리를 UNION ALL로 합치기
대용량 컬럼 인덱스 TEXT/BLOB 전체 인덱스 불가 (prefix 길이 제한) prefix 인덱스 또는 Full-Text 인덱스 사용

OR 조건 개선 예시

-- ❌ OR 조건 — 인덱스가 있어도 비효율
SELECT * FROM products WHERE category_id = 1 OR brand_id = 5;

-- ✅ UNION ALL로 각각 인덱스 활용
SELECT * FROM products WHERE category_id = 1
UNION ALL
SELECT * FROM products WHERE brand_id = 5 AND category_id != 1;

인덱스 카디널리티 확인

-- 컬럼별 유일값 개수 확인 (카디널리티)
SELECT
  column_name,
  COUNT(DISTINCT column_name) AS cardinality,
  COUNT(*) AS total_rows,
  ROUND(COUNT(DISTINCT column_name) / COUNT(*) * 100, 2) AS selectivity_pct
FROM information_schema.columns
-- 실제로는 테이블에서 직접 조회
;

-- 예시
SELECT
  COUNT(DISTINCT status) AS status_cardinality,        -- 낮음 → 단독 인덱스 비효율
  COUNT(DISTINCT user_id) AS user_cardinality,         -- 높음 → 인덱스 효과적
  COUNT(*) AS total
FROM orders;

9. 핵심 정리

B+Tree
균형 트리, 리프 링크드 리스트로 범위 검색 O(log N). 대부분 RDBMS 기본.
클러스터링 인덱스
행 데이터 자체 정렬. InnoDB에서 PK = 클러스터링 인덱스.
왼쪽 접두사 원칙
복합 인덱스 (a,b,c): b만으로는 인덱스 사용 불가. 등가 컬럼을 앞에.
커버링 인덱스
SELECT 컬럼까지 인덱스에 포함 → 테이블 접근 없이 결과 반환.
EXPLAIN type=ALL
풀 테이블 스캔 → 대용량 테이블에서 즉시 인덱스 추가 검토.
슬로우 쿼리 3대 원인
인덱스 미사용, 함수 적용으로 인덱스 무력화, N+1 쿼리.

한 문장 요약

요약
인덱스는 읽기를 빠르게 하지만 쓰기를 느리게 한다 — EXPLAIN으로 실행 계획을 검증하고, 슬로우 쿼리 로그로 병목을 식별하며, 커버링 인덱스와 복합 인덱스 컬럼 순서로 추가 최적화를 달성한다.

10. 확인 문제

Q1. B+Tree vs B-Tree

B+Tree가 범위 검색에 유리한 이유는 무엇인가?

정답 보기

B+Tree는 모든 실제 데이터가 리프 노드에 있고, 리프 노드들이 링크드 리스트로 연결되어 있기 때문입니다. 범위 검색 시 시작 리프를 찾은 후 포인터를 순서대로 따라가면 O(k) (k = 결과 건수) 추가 탐색만으로 범위 내 모든 데이터를 읽을 수 있습니다. B-Tree는 내부 노드에도 데이터가 있어 범위 순회 시 트리를 재탐색해야 합니다.

Q2. 복합 인덱스 컬럼 순서

다음 쿼리에 대해 최적 인덱스를 설계하시오.

SELECT product_id, price FROM products
WHERE category_id = 5 AND status = 'active'
ORDER BY price ASC
LIMIT 20;
정답 보기

CREATE INDEX idx ON products(category_id, status, price) INCLUDE (product_id); (PostgreSQL) 또는 CREATE INDEX idx ON products(category_id, status, price, product_id); (MySQL)

이유: ① category_id = 5 등가 → 앞, ② status = 'active' 등가 → 다음, ③ ORDER BY price ASC → 마지막 컬럼으로 정렬 비용 제거 (Using filesort 방지). product_id와 price까지 포함하면 커버링 인덱스가 됩니다.

Q3. EXPLAIN 분석

EXPLAIN 결과의 Extra 컬럼에 Using filesort가 표시될 때 어떤 의미이며, 어떻게 해결할 수 있나?

정답 보기

Using filesort는 인덱스를 이용한 정렬이 불가능해 별도 정렬 작업(메모리 또는 임시 파일)을 수행한다는 의미입니다. 해결 방법: ORDER BY에 사용되는 컬럼을 인덱스에 포함시키고, 인덱스 컬럼 순서가 ORDER BY 컬럼 순서와 일치하도록 설계합니다. 다만 WHERE 조건에 범위 연산자가 있거나 ORDER BY 방향이 혼합(ASC/DESC)이면 인덱스 정렬이 불가할 수 있습니다.

Q4. 인덱스 무력화 원인 찾기

아래 쿼리에서 인덱스가 사용되지 않는 이유를 설명하고 수정하시오. (created_at에 인덱스 있음)

SELECT * FROM orders WHERE YEAR(created_at) = 2025;
정답 보기

YEAR() 함수를 인덱스 컬럼에 적용하면 옵티마이저가 인덱스를 사용할 수 없습니다 (함수 적용으로 인덱스 무력화). 수정: 

SELECT * FROM orders
WHERE created_at >= '2025-01-01 00:00:00'
  AND created_at <  '2026-01-01 00:00:00';
이렇게 범위 조건으로 바꾸면 range 타입으로 인덱스를 활용합니다.
