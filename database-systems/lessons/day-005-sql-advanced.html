2. 서브쿼리란?

서브쿼리(Subquery)는 다른 SQL 문 안에 중첩된 SELECT 문이다. 쿼리 결과를 즉석에서 다른 쿼리의 조건이나 데이터 원본으로 활용할 때 사용한다.

SELECT name FROM employees WHERE salary > (SELECT AVG(salary) FROM employees); └──────────── 서브쿼리 ────────────┘

2-1. 위치에 따른 분류

위치이름반환 형태예시
WHERE / HAVING조건 서브쿼리스칼라·행·집합WHERE salary > (SELECT AVG(...))
FROM인라인 뷰 (파생 테이블)테이블FROM (SELECT ...) AS t
SELECT스칼라 서브쿼리단일 값SELECT (SELECT MAX(...)) AS 최대값

2-2. 스칼라 서브쿼리 — 단일 값 반환

스칼라 서브쿼리는 반드시 정확히 1행 1열을 반환해야 한다. 전체 평균보다 급여가 높은 직원을 조회한다.

SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees)
ORDER BY salary DESC;
namesalary
이영희7500000
한지민6800000
최수진6200000
김철수6000000

전체 평균: SELECT AVG(salary) FROM employees = 6,166,667원. 이 값보다 높은 직원 4명이 결과에 포함된다.

2-3. 집합 서브쿼리 — IN / NOT IN

서브쿼리가 여러 행을 반환할 때 IN 또는 NOT IN으로 비교한다.

-- 개발팀(1) 또는 인사팀(3) 소속 직원 조회
SELECT name, dept_id
FROM   employees
WHERE  dept_id IN (SELECT dept_id FROM departments WHERE dept_name IN ('개발팀','인사팀'));

-- 직원이 한 명도 없는 부서 조회
SELECT dept_name
FROM   departments
WHERE  dept_id NOT IN (SELECT dept_id FROM employees WHERE dept_id IS NOT NULL);

NOT IN과 NULL 함정: 서브쿼리 결과에 NULL이 포함되면 NOT IN은 항상 FALSE를 반환한다. 반드시 WHERE dept_id IS NOT NULL로 NULL을 제거하거나, 대신 NOT EXISTS를 사용하라.

2-4. 인라인 뷰 — FROM 절 서브쿼리

서브쿼리를 FROM 절에서 임시 테이블처럼 사용한다. 반드시 별칭을 붙여야 한다.

-- 부서별 평균 급여를 먼저 구하고, 그 중 평균이 600만 원 이상인 부서만 조회
SELECT dept_id, avg_salary
FROM   (
  SELECT dept_id, ROUND(AVG(salary), 0) AS avg_salary
  FROM   employees
  WHERE  dept_id IS NOT NULL
  GROUP BY dept_id
) AS dept_avg
WHERE  avg_salary >= 6000000;

3. 상관 서브쿼리 (Correlated Subquery)

상관 서브쿼리는 바깥 쿼리의 현재 행을 참조한다. 바깥 쿼리가 한 행씩 처리될 때마다 서브쿼리가 다시 실행되므로, 큰 테이블에서는 성능에 주의해야 한다.

SELECT name, salary, dept_id FROM employees e_outer -- ① 바깥 쿼리가 한 행(e_outer) 선택 WHERE salary > ( SELECT AVG(salary) FROM employees e_inner WHERE e_inner.dept_id = e_outer.dept_id -- ② e_outer 행의 dept_id 참조 ); -- ③ ①②를 모든 행에 반복

3-1. 소속 부서 평균보다 급여가 높은 직원

SELECT e.name, e.salary, d.dept_name
FROM   employees e
INNER JOIN departments d ON e.dept_id = d.dept_id
WHERE  e.salary > (
  SELECT AVG(e2.salary)
  FROM   employees e2
  WHERE  e2.dept_id = e.dept_id
)
ORDER BY d.dept_name, e.salary DESC;
namesalarydept_name
이영희7500000개발팀
박민준5500000마케팅팀

개발팀 평균은 6,750,000원 → 이영희(7,500,000)만 해당. 마케팅팀 평균은 5,250,000원 → 박민준(5,500,000)만 해당.

3-2. 각 부서에서 가장 최근 입사한 직원

SELECT e.name, d.dept_name, e.hire_date
FROM   employees e
INNER JOIN departments d ON e.dept_id = d.dept_id
WHERE  e.hire_date = (
  SELECT MAX(e2.hire_date)
  FROM   employees e2
  WHERE  e2.dept_id = e.dept_id
)
ORDER BY d.dept_name;

4. EXISTS / NOT EXISTS

EXISTS는 서브쿼리가 결과를 하나라도 반환하면 TRUE, 아니면 FALSE다. IN과 달리 실제 값이 아닌 존재 여부만 확인하므로 NULL에 안전하고 큰 결과 집합에서 더 효율적인 경우가 많다.

4-1. EXISTS — 직원이 있는 부서만 조회

SELECT dept_name
FROM   departments d
WHERE  EXISTS (
  SELECT 1
  FROM   employees e
  WHERE  e.dept_id = d.dept_id
);
dept_name
개발팀
마케팅팀
인사팀

서브쿼리 내 SELECT 1은 관용적 표현이다. 반환값은 중요하지 않으므로 SELECT *나 SELECT emp_id를 써도 동일하다.

4-2. NOT EXISTS — 직원이 없는 부서만 조회

SELECT dept_name
FROM   departments d
WHERE  NOT EXISTS (
  SELECT 1
  FROM   employees e
  WHERE  e.dept_id = d.dept_id
);
dept_name
기획팀

5. 뷰 (VIEW)

뷰는 SELECT 쿼리에 이름을 붙여 저장한 가상 테이블이다. 실제 데이터를 복사하지 않고 쿼리 정의만 저장하며, 사용할 때마다 원본 테이블을 조회한다.

5-1. CREATE VIEW

-- 직원 이름, 부서명, 급여를 함께 보여주는 뷰
CREATE VIEW v_employee_detail AS
SELECT e.emp_id,
       e.name,
       d.dept_name,
       e.salary,
       e.hire_date
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

5-2. 뷰 조회 — 일반 테이블처럼 사용

-- 뷰를 테이블처럼 SELECT
SELECT * FROM v_employee_detail ORDER BY salary DESC;

-- 뷰에 추가 조건 적용
SELECT name, dept_name, salary
FROM   v_employee_detail
WHERE  salary >= 6000000;

5-3. 뷰 장점과 한계

장점한계
복잡한 JOIN/집계 쿼리를 단순화Materialized View가 아닌 경우 매번 원본 쿼리 실행
특정 컬럼만 노출해 보안 강화일부 DBMS는 뷰를 통한 UPDATE/DELETE 제한
쿼리 재사용성 향상뷰 안에 뷰를 중첩하면 성능 문제 발생 가능

5-4. 뷰 수정 및 삭제

-- 뷰 재정의 (PostgreSQL/MySQL 방식)
CREATE OR REPLACE VIEW v_employee_detail AS
SELECT e.emp_id,
       e.name,
       COALESCE(d.dept_name, '미배정') AS dept_name,
       e.salary,
       e.hire_date
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- 뷰 삭제
DROP VIEW v_employee_detail;

Materialized View: PostgreSQL 등 일부 DBMS는 결과를 물리적으로 저장하는 Materialized View를 지원한다. 조회 속도는 빠르지만 원본 데이터 변경 시 수동 또는 자동 갱신(REFRESH)이 필요하다.

6. 인덱스 (INDEX)

인덱스는 특정 컬럼의 값을 정렬된 자료구조(주로 B-Tree)에 저장해 풀 스캔(Full Scan) 없이 빠른 검색을 가능하게 한다. 책의 색인과 같은 역할이다.

6-1. 인덱스 없이 조회

employees 테이블 (6행) ┌────────┬──────────┬─────────┬─────────┐ │ emp_id │ name │ dept_id │ salary │ ├────────┼──────────┼─────────┼─────────┤ │ 101 │ 김철수 │ 1 │ 6000000 │ ← 순서대로 스캔 │ 102 │ 이영희 │ 1 │ 7500000 │ ← 순서대로 스캔 │ 103 │ 박민준 │ 2 │ 5500000 │ ← 순서대로 스캔 │ 104 │ 최수진 │ 3 │ 6200000 │ ← 순서대로 스캔 │ 105 │ 정도윤 │ 2 │ 5000000 │ ← 순서대로 스캔 │ 106 │ 한지민 │ NULL │ 6800000 │ ← 순서대로 스캔 └────────┴──────────┴─────────┴─────────┘ WHERE salary = 6200000 → 모든 행 검사 (Full Table Scan)

6-2. B-Tree 인덱스 동작 원리

salary 인덱스 (B-Tree) [6000000] / \ [5000000] [6800000] / \ / \ [5000000] [5500000] [6200000] [7500000] (105) (103) (104) (102) WHERE salary = 6200000 → 루트에서 비교하며 3단계 탐색으로 결과 발견 → Full Scan O(n) → 인덱스 탐색 O(log n)

6-3. CREATE INDEX

-- 단일 컬럼 인덱스
CREATE INDEX idx_employees_salary ON employees(salary);

-- 복합 컬럼 인덱스 (dept_id로 먼저 필터, 그 안에서 salary 정렬)
CREATE INDEX idx_employees_dept_salary ON employees(dept_id, salary);

-- UNIQUE 인덱스 (중복 불허)
CREATE UNIQUE INDEX idx_employees_name ON employees(name);

6-4. 인덱스 사용 여부 확인 (EXPLAIN)

-- PostgreSQL
EXPLAIN SELECT * FROM employees WHERE salary = 6200000;

-- MySQL
EXPLAIN SELECT * FROM employees WHERE salary = 6200000;

-- SQLite
EXPLAIN QUERY PLAN SELECT * FROM employees WHERE salary = 6200000;

EXPLAIN 결과에서 Index Scan(인덱스 사용) vs Seq Scan(순차 탐색) 구분이 핵심이다.

6-5. 인덱스 장단점

장점단점
WHERE·JOIN·ORDER BY 조회 속도 향상INSERT·UPDATE·DELETE 시 인덱스도 갱신 → 쓰기 성능 저하
정렬 연산 최적화추가 디스크 공간 사용
UNIQUE 제약조건 구현 수단선택도(Cardinality)가 낮으면 오히려 비효율

인덱스를 만들어야 할 곳: WHERE·JOIN 조건으로 자주 사용되는 컬럼, ORDER BY 기준 컬럼, 외래 키(FK) 컬럼. 반면 값 종류가 적은 컬럼(예: 성별, boolean)은 인덱스 효과가 미미하다.

6-6. 인덱스 삭제

DROP INDEX idx_employees_salary;
-- MySQL은 테이블 지정 필요
DROP INDEX idx_employees_salary ON employees;

7. 종합 실습

문제 1

전체 직원 평균 급여보다 낮은 급여를 받는 직원의 이름, 부서명, 급여를 조회하라. 부서 미배정 직원도 포함하라.

정답 보기 
SELECT e.name,
       COALESCE(d.dept_name, '미배정') AS dept_name,
       e.salary
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE  e.salary < (SELECT AVG(salary) FROM employees)
ORDER BY e.salary;

문제 2

각 부서에서 급여가 가장 높은 직원의 이름과 급여를 상관 서브쿼리로 조회하라.

정답 보기 
SELECT e.name, d.dept_name, e.salary
FROM   employees e
INNER JOIN departments d ON e.dept_id = d.dept_id
WHERE  e.salary = (
  SELECT MAX(e2.salary)
  FROM   employees e2
  WHERE  e2.dept_id = e.dept_id
)
ORDER BY d.dept_name;

문제 3

직원이 2명 이상인 부서만 EXISTS를 사용해 조회하라.

정답 보기 
SELECT dept_name
FROM   departments d
WHERE  EXISTS (
  SELECT 1
  FROM   employees e
  WHERE  e.dept_id = d.dept_id
  HAVING COUNT(*) >= 2
);

-- 또는 (더 명확한 방식)
SELECT dept_name
FROM   departments d
WHERE  (SELECT COUNT(*) FROM employees e WHERE e.dept_id = d.dept_id) >= 2;

문제 4

v_employee_detail 뷰를 생성하고, 그 뷰를 사용해 급여가 600만 원 이상인 직원의 이름과 부서명을 조회하라.

정답 보기 
-- 뷰 생성
CREATE VIEW v_employee_detail AS
SELECT e.emp_id, e.name,
       COALESCE(d.dept_name, '미배정') AS dept_name,
       e.salary, e.hire_date
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- 뷰 조회
SELECT name, dept_name
FROM   v_employee_detail

8. 핵심 요약

개념핵심 포인트
스칼라 서브쿼리단일 값 반환 — WHERE·SELECT·HAVING에서 값 대신 사용
집합 서브쿼리여러 행 반환 — IN/NOT IN으로 비교, NOT IN에서 NULL 주의
인라인 뷰FROM 절의 임시 테이블 — 반드시 별칭 필요
상관 서브쿼리바깥 쿼리의 행을 참조 — 행마다 재실행되므로 성능 주의
EXISTS존재 여부만 확인, Short-circuit — NOT IN보다 NULL에 안전
CREATE VIEW쿼리에 이름 붙이기 — 실제 데이터 저장 없음, 조회 시 원본 실행
CREATE INDEXB-Tree로 빠른 검색 — 쓰기 성능·공간 트레이드오프, 선택도 고려

자주 하는 실수: 스칼라 서브쿼리에서 결과가 2행 이상 반환되면 런타임 오류 발생. 반드시 단일 행만 반환하도록 MAX, MIN, LIMIT 1 등을 사용하라.
