💬 면접 질문

Core Question

"트래픽이 늘면서 특정 API의 응답 속도가 갑자기 느려졌습니다. DB 관점에서 어떤 순서로 어떤 것들을 확인하시겠습니까?"

함께 출제되는 변형 질문

  • "주문 목록 API가 이전보다 5배 느려졌습니다. DB에서 무엇부터 보시겠습니까?"
  • "슬로우 쿼리 로그에 특정 쿼리가 반복해서 잡힙니다. 어떻게 분석하시겠습니까?"
  • "배포 후 CPU가 급등했습니다. DB가 원인인지 어떻게 판단하시겠습니까?"
  • "EXPLAIN을 봤는데 rows가 100만 이상 나옵니다. 무엇을 의심해야 합니까?"

🎯 왜 면접에서 묻는가

이 질문은 지식 암기가 아니라 체계적 사고 과정을 보기 위한 것입니다.


  • 장애 대응 경험: 실제로 느린 쿼리를 다뤄본 사람은 "어디부터 볼지"가 이미 몸에 배어 있습니다. 면접관은 그 순서를 묻는 것으로 실전 경험을 확인합니다.
  • 가설 기반 사고: DB 성능 문제는 원인이 여러 겹입니다. 수십 개의 가능성을 무작위로 확인하는 사람과, 증거를 보고 빠르게 가설을 좁히는 사람을 구별합니다.
  • 범위 설정 능력: DB만의 문제인지, 애플리케이션·네트워크·인프라와 얽힌 문제인지 먼저 분리할 수 있는가. 이 질문으로 시스템 전체를 보는 시야를 가졌는지 봅니다.
  • 운영 리스크 감각: "원인 찾기"와 "찾는 중에 프로덕션 안 건드리기"를 동시에 고려하는가. 진단 도구 자체가 부하를 주거나 서비스에 영향을 줄 수 있기 때문입니다.

📚 전제 개념

DB는 왜 갑자기 느려지는가?

DB가 갑자기 느려지는 원인은 크게 네 가지 층으로 나뉩니다. 이 네 층을 머릿속에 두고 좁혀가면 원인을 체계적으로 찾을 수 있습니다.


  • 쿼리 층: 실행 계획이 나쁜 쿼리, 인덱스를 타지 않는 쿼리, 한 번에 수십만 행을 스캔하는 쿼리
  • Lock 층: 트랜잭션이 lock을 오래 잡거나 deadlock으로 재시도가 반복되는 상황
  • 리소스 층: CPU·메모리·디스크 I/O·네트워크가 포화 상태인 경우
  • 데이터 층: 테이블 크기가 임계치를 넘어 기존 인덱스가 더 이상 효과적이지 않은 경우

실행 계획(Execution Plan)이란?

DB 엔진은 SQL을 받으면 "어떤 순서로 어떤 방법으로 데이터를 읽을지" 계획을 세웁니다. 이것이 실행 계획입니다. EXPLAIN 또는 EXPLAIN ANALYZE 명령으로 볼 수 있습니다.


핵심 항목:

  • type (접근 방식): ALL(풀 스캔) → index → range → ref → const 순서로 효율이 높아집니다. ALL이 나오면 인덱스가 전혀 사용되지 않았다는 신호입니다.
  • rows: 실제로 읽을 행 수 추정치. 이 숫자가 테이블 전체 행 수에 가까우면 문제입니다.
  • key: 실제로 사용 중인 인덱스 이름. NULL이면 인덱스 없이 풀 스캔합니다.
  • Extra: Using filesort, Using temporary가 나오면 정렬이나 임시 테이블이 발생해 추가 비용이 생깁니다.

슬로우 쿼리 로그(Slow Query Log)란?

설정된 시간(예: long_query_time = 1초)을 초과하는 쿼리를 자동으로 기록하는 기능입니다. MySQL/MariaDB·PostgreSQL 모두 지원합니다. 성능 문제가 터졌을 때 가장 먼저 봐야 하는 로그입니다. 

-- MySQL: 슬로우 쿼리 설정 확인
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';

Lock Wait란?

두 트랜잭션이 같은 행이나 범위를 동시에 수정하려 할 때, 한쪽이 상대방의 lock이 풀릴 때까지 기다리는 현상입니다. 대기 시간이 길면 타임아웃으로 에러가 되고, 여러 트랜잭션이 서로 기다리면 deadlock이 됩니다. 트래픽이 늘면 동시 요청이 늘고, 짧은 lock wait도 쌓이면 큰 지연으로 나타납니다.

cardinality와 selectivity란?

cardinality는 컬럼에 있는 고유 값의 수입니다. selectivity는 "인덱스를 통해 걸러낼 수 있는 비율"입니다. selectivity = 고유 값 수 / 전체 행 수. 값이 1에 가까울수록 인덱스가 효과적입니다.


예: 1,000만 명 중 성별 컬럼(M/F)의 selectivity = 2/10,000,000 ≈ 0. 인덱스 효과 없음. 주문 ID의 selectivity = 10,000,000/10,000,000 = 1. 인덱스 매우 효과적.

🏭 실무 연결

시나리오: 주문 목록 API 응답이 3초로 급등

대규모 이커머스에서 주문 목록 API 응답 시간이 평균 80ms에서 갑자기 3,000ms로 치솟는 상황을 가정합니다. 이 상황에서 아래와 같은 층위별 원인이 있을 수 있습니다.


  • 쿼리 층: 주문 테이블에 5천만 행이 쌓이면서 status 컬럼 단독 인덱스가 selectivity를 잃음. EXPLAIN이 ALL로 바뀜.
  • Lock 층: 재고 차감 트랜잭션이 lock을 오래 보유하면서 조회 쿼리들이 lock wait 대기열에 쌓임.
  • 리소스 층: 대용량 집계 쿼리가 버퍼 풀(buffer pool)을 점유해서 다른 쿼리들이 디스크 I/O를 강제로 발생시킴.
  • 데이터 층: 대규모 배치가 실행되면서 인덱스 통계가 오래됨. 옵티마이저가 잘못된 실행 계획을 선택함.

배포 후 슬로우 쿼리가 갑자기 생겼을 때

  • 새로 배포한 기능이 기존에 없던 쿼리 패턴을 추가했는가
  • ORM이 예상과 다른 쿼리를 생성했는가 (N+1 문제, 불필요한 JOIN)
  • 새 기능이 기존 테이블의 write 패턴을 바꾸면서 lock 경합이 늘었는가

🔍 답변 사고 순서

면접에서 이 순서를 소리 내어 설명하는 것만으로도 강한 답변이 됩니다. "저는 A부터 확인한 다음 B로 좁혀가겠습니다"처럼 구체적으로 말하세요.
  1. 1단계 — 증상 범위 분리: DB 문제인가, 아닌가

    API 지연이 DB에서 비롯된 것인지 먼저 확인합니다. APM(Datadog, New Relic, Pinpoint)에서 DB 응답 시간 비율을 봅니다. DB 호출 시간이 전체 API 지연의 80% 이상이면 DB 문제로 가정하고 진단합니다.

    • DB 연결 풀이 고갈됐는가 (connection pool exhausted)
    • DB 서버 CPU·메모리·디스크 I/O가 포화 상태인가
    • 네트워크 레이턴시 급등은 없는가
  2. 2단계 — 슬로우 쿼리 로그 확인

    어떤 쿼리가 느린지 특정합니다. 슬로우 쿼리 로그에서 가장 자주 등장하거나 실행 시간이 긴 쿼리를 뽑습니다. pt-query-digest(MySQL), pg_stat_statements(PostgreSQL)를 사용합니다.

    -- PostgreSQL: 느린 쿼리 상위 10개
SELECT query, calls, total_exec_time, mean_exec_time, rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
  3. 3단계 — 실행 계획(EXPLAIN) 분석

    느린 쿼리를 특정했으면 EXPLAIN으로 실행 계획을 확인합니다. 풀 스캔 여부, 인덱스 사용 여부, rows 추정치를 봅니다.

    -- MySQL: 실행 계획 확인
EXPLAIN SELECT * FROM orders
WHERE user_id = 12345 AND status = 'PAID'
ORDER BY created_at DESC
LIMIT 20;

    확인 항목:

    • type이 ALL이면 인덱스 없는 풀 스캔 → 인덱스 추가 검토
    • rows 추정치가 실제보다 크게 틀리면 통계 갱신 필요 (ANALYZE TABLE)
    • Extra: Using filesort → ORDER BY에 인덱스가 없거나 커버링 인덱스 추가 검토
  4. 4단계 — Lock 및 동시성 확인

    특정 시간대에 지연이 집중된다면 lock wait를 의심합니다. 현재 대기 중인 lock 상황을 봅니다.

    -- MySQL InnoDB: lock wait 현황
SELECT * FROM information_schema.INNODB_TRX;
SELECT * FROM performance_schema.data_lock_waits;

-- PostgreSQL: 대기 중인 lock
SELECT pid, wait_event_type, wait_event, state, query
FROM pg_stat_activity
WHERE wait_event_type = 'Lock';

    lock wait가 많다면 트랜잭션 scope를 줄이거나, lock 순서를 일관되게 하거나, 격리 수준을 재검토합니다.

  5. 5단계 — 데이터 볼륨 및 통계 확인

    테이블 크기와 인덱스 통계가 최신인지 확인합니다. 데이터가 크게 늘었는데 통계가 오래됐으면 옵티마이저가 잘못된 계획을 선택합니다.

    -- MySQL: 테이블 크기 확인
SELECT table_name,
       ROUND((data_length + index_length) / 1024 / 1024, 2) AS size_mb,
       table_rows
FROM information_schema.tables
WHERE table_schema = 'your_db'
ORDER BY size_mb DESC;
  6. 6단계 — 개선안 도출 및 리스크 검토

    원인을 특정했으면 개선안을 제안합니다. 하지만 반드시 프로덕션 영향을 먼저 확인합니다.

    • 인덱스 추가: 운영 중 CREATE INDEX ... ONLINE (잠금 없이) vs. 트래픽 낮을 때
    • 쿼리 리라이팅: ORM에서 raw 쿼리로 교체, 서브쿼리 → JOIN 변환
    • 임시 캐싱: Redis로 집계 결과 캐싱, TTL 설정
    • 읽기 분산: read replica 추가
    프로덕션에 직접 ALTER TABLE이나 인덱스 생성 시 테이블 크기에 따라 수분~수십 분 잠금이 발생할 수 있습니다. 반드시 staging에서 실행 시간을 측정하고, 트래픽이 가장 낮은 시간대에 적용하거나 Online DDL을 사용하세요.

💡 핵심 답변 예시

짧은 버전 (1분)

"우선 DB 문제인지 먼저 분리합니다. APM에서 DB 응답 비율을 확인해 DB가 병목이 맞으면, 슬로우 쿼리 로그나 pg_stat_statements로 느린 쿼리를 특정합니다. 그 쿼리에 EXPLAIN을 실행해 풀 스캔인지, 인덱스를 타는데 rows가 과도한지, filesort가 발생하는지 봅니다. 동시에 lock wait가 없는지 pg_stat_activity로 확인합니다. 원인에 따라 인덱스 추가, 쿼리 리라이팅, 캐싱 중 하나를 선택하되, 프로덕션 적용 전 staging에서 실행 계획 변화와 실행 시간을 검증합니다."

깊이 있는 버전 (3~4분)

"저는 이런 상황에서 4개 층위로 나누어 원인을 좁힙니다.

첫 번째로 증상 범위를 분리합니다. API 지연이 DB 호출에서 비롯된 것인지 APM 트레이스로 확인합니다. DB 호출이 전체 레이턴시의 대부분을 차지한다면 DB로 범위를 좁히고, 그렇지 않다면 애플리케이션 로직이나 외부 서비스를 먼저 봅니다.

두 번째로 슬로우 쿼리를 특정합니다. MySQL이라면 슬로우 쿼리 로그와 pt-query-digest로 가장 무거운 쿼리를 뽑습니다. PostgreSQL이라면 pg_stat_statements에서 total_exec_time 기준으로 상위 쿼리를 봅니다.

세 번째로 EXPLAIN ANALYZE로 실행 계획을 확인합니다. 풀 스캔이면 인덱스 추가나 컬럼 변환 제거를 검토합니다. 인덱스를 타는데도 느리다면 rows 추정치가 실제와 다른지, 즉 통계가 오래됐는지 확인합니다. Using filesort가 나오면 정렬 컬럼 포함 복합 인덱스를 검토합니다.

네 번째로 lock wait를 확인합니다. 트래픽이 급증한 시점에 lock wait가 쌓였다면, 트랜잭션 scope를 줄이거나 낙관적 lock으로 바꾸거나, 격리 수준을 READ COMMITTED로 낮추는 방향을 검토합니다.

개선안을 적용할 때는 반드시 staging에서 실행 계획이 바뀌는지 확인하고, 온라인 인덱스 생성이 가능한지 먼저 검토합니다. 트래픽이 높은 시간에 테이블 잠금이 발생하면 문제가 더 커질 수 있어서 적용 타이밍도 중요합니다."

⚖️ Trade-off

인덱스 추가

  • 읽기 이득: 조회 속도 크게 향상
  • 쓰기 비용: INSERT/UPDATE/DELETE 시 인덱스도 갱신. 쓰기가 많은 테이블은 인덱스가 늘수록 쓰기가 느려짐
  • 스토리지: 인덱스 자체가 공간을 차지함. 대형 테이블에서 수 GB가 될 수 있음
  • 운영: 온라인 인덱스 생성이 가능해도 CPU/I/O 부하가 발생. 트래픽 낮은 시간대가 선호됨

격리 수준 낮추기 (SERIALIZABLE → READ COMMITTED)

  • 동시성 이득: lock wait 감소, throughput 향상
  • 정합성 리스크: phantom read, non-repeatable read 발생 가능. 금융·재고처럼 정합성이 중요한 도메인에서는 허용 불가
  • 판단 기준: 읽기 전용 조회는 낮은 격리 수준이 유리. 재고/포인트 차감처럼 lost update가 생기면 안 되는 연산은 높은 격리 수준 유지

캐싱(Redis) 도입

  • 응답 속도 이득: DB 호출 제거 → ms 단위 응답
  • 정합성 리스크: TTL 동안 최신 데이터 보장 안 됨. 재고·잔액처럼 실시간 정합성이 필요한 데이터에는 부적합
  • 운영 복잡도: 캐시 무효화(cache invalidation) 전략, 캐시 스탬피드(cache stampede) 방지가 추가로 필요

🚫 흔한 오해

느리면 일단 인덱스부터 추가하면 된다.
인덱스가 있어도 selectivity가 낮거나, 컬럼에 함수를 씌우면 인덱스가 동작하지 않습니다. 먼저 EXPLAIN으로 원인을 확인하고 추가 여부를 판단해야 합니다. 불필요한 인덱스는 쓰기 성능을 저하시킵니다.
EXPLAIN에서 key가 잡히면 쿼리가 빠르다.
인덱스를 쓴다고 무조건 빠른 것이 아닙니다. rows 추정치가 크거나, 인덱스 범위 스캔 후 필터링해야 할 행이 많으면 인덱스를 써도 느릴 수 있습니다. EXPLAIN ANALYZE로 실제 실행 시간과 추정치 오차를 함께 확인하세요.
슬로우 쿼리 로그에 없으면 DB 문제가 아니다.
lock wait는 쿼리 실행 시간에 포함되지 않는 경우가 있습니다. 또한 단건 쿼리는 빠르지만 N+1 패턴으로 수천 번 반복 실행돼 전체 지연이 생기는 경우도 슬로우 쿼리 로그에 잘 잡히지 않습니다.
통계는 DB가 자동으로 관리하므로 신경 쓸 필요 없다.
대량 INSERT/DELETE 후 자동 통계 갱신이 즉시 반영되지 않거나, 갱신 임계치를 넘지 못해 오래된 통계가 남아 있을 수 있습니다. 특히 배치 처리 후에는 수동으로 ANALYZE를 실행하는 것이 안전합니다.

📊 좋은 답변 vs 약한 답변

강한 답변

"먼저 APM에서 DB 호출 비율을 보고 DB 문제인지 분리합니다. 슬로우 쿼리 로그에서 가장 무거운 쿼리를 특정한 다음 EXPLAIN ANALYZE로 풀 스캔인지, lock wait인지 확인합니다. 원인이 인덱스라면 selectivity를 계산해서 효과를 검증하고, 운영 중 온라인 인덱스 생성이 가능한지, 쓰기 성능에 영향은 없는지 같이 검토합니다."

왜 강한가: ① 범위를 먼저 분리 → ② 특정 쿼리 → ③ 실행 계획 → ④ 개선안 + 리스크 순서가 논리적입니다. 슬로우 쿼리 로그, EXPLAIN, selectivity 같은 구체적 도구와 지표를 언급합니다. 개선 시 trade-off(쓰기 성능, 온라인 여부)도 함께 고려합니다.

약한 답변

"인덱스를 추가하면 될 것 같습니다. 아니면 캐시를 써볼 수 있습니다. EXPLAIN을 돌려보는 것도 좋을 것 같습니다."

왜 약한가: 순서 없이 해결책을 나열합니다. 왜 그 순서로 확인하는지 근거가 없습니다. 문제가 인덱스인지, lock인지, 데이터 볼륨인지 진단 전에 답부터 제시합니다. 면접관은 "인덱스가 왜 안 될 수도 있는지 생각해봤나?" 한 마디로 이 답변을 무너뜨릴 수 있습니다.

🎓 Senior-level Follow-up 질문

Q. EXPLAIN에서 rows가 100만인데 key는 잡혀 있습니다. 어떤 상황이고 어떻게 대응하시겠습니까?
의도: 인덱스가 있어도 충분하지 않을 수 있다는 것을 이해하는지 확인합니다. 인덱스 설계 문제와 selectivity 개념을 연결하는지 봅니다.
모범 사고 경로: 인덱스를 쓰지만 범위 스캔 후 조건으로 거르는 행이 많다는 의미입니다. 이 경우 ① 해당 컬럼의 selectivity를 확인합니다 — 고유 값이 적다면 인덱스 효과가 없습니다. ② 복합 인덱스로 바꿔서 조건 컬럼을 모두 포함시키거나 ③ 커버링 인덱스로 정렬/필터 컬럼까지 포함시킵니다. ④ 또는 쿼리 자체를 다시 작성해서 더 선택적인 조건이 먼저 오도록 합니다.
Q. 슬로우 쿼리가 특정 시간대에만 집중됩니다. 항상 느린 게 아닙니다. 무엇을 의심하시겠습니까?
의도: lock wait, 배치 작업 간섭, 버퍼 풀 경합처럼 동시성과 관련된 원인을 떠올릴 수 있는지 봅니다.
모범 사고 경로: 특정 시간대 집중이면 ① 그 시간대에 배치 작업이나 집계 쿼리가 실행되는지 확인합니다 — 대용량 읽기가 버퍼 풀을 차지하면 다른 쿼리가 디스크 I/O를 씁니다. ② lock wait 시간대와 일치하는지 봅니다 — 재고·포인트 차감 배치가 row-level lock을 오래 잡으면 조회 쿼리가 대기합니다. ③ 해당 시간대에 connection 수가 급증하는지 확인합니다 — connection pool 부족이 원인일 수도 있습니다.
Q. 인덱스를 추가하려는데 테이블이 3억 행입니다. 어떻게 하시겠습니까?
의도: 대형 테이블에서 DDL 운영 리스크를 인지하는지 봅니다. Online DDL, pt-online-schema-change 같은 도구를 알고 있는지 확인합니다.
모범 사고 경로: MySQL 5.6+는 InnoDB Online DDL로 테이블 잠금 없이 인덱스를 생성할 수 있습니다. 하지만 3억 행은 수십 분이 걸릴 수 있어서 ① 먼저 staging 환경에서 실행 시간을 측정합니다. ② 트래픽이 가장 낮은 새벽 시간대에 적용합니다. ③ 잠금을 최소화하려면 pt-online-schema-change(Percona)나 gh-ost(GitHub)처럼 binlog 기반으로 잠금 없이 복사하는 도구를 씁니다. ④ 적용 후 실행 계획이 바뀌는지 반드시 확인합니다.
Q. 문제 쿼리를 빠르게 개선했습니다. 개선 효과를 어떻게 측정하고 어떻게 검증하시겠습니까?
의도: 성능 개선이 체계적인지, "느낌상 빨라진 것 같다"가 아니라 측정 기반으로 판단하는지 봅니다.
모범 사고 경로: ① EXPLAIN ANALYZE로 변경 전/후 실행 계획과 실제 실행 시간을 비교합니다. ② staging 환경에서 프로덕션 데이터 볼륨을 재현해서 부하 테스트합니다. ③ 프로덕션 배포 후 APM에서 p50/p95/p99 레이턴시 추이를 최소 1시간 이상 모니터링합니다. ④ 슬로우 쿼리 로그에서 해당 쿼리가 사라졌는지 확인합니다. ⑤ 예상치 못한 side effect(쓰기 지연, 다른 쿼리의 실행 계획 변화)가 없는지 확인합니다.
