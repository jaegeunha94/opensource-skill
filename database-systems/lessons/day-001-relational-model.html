1. 왜 데이터베이스가 필요한가?

파일 시스템의 한계

파일 기반 데이터 관리에는 구조적인 문제가 있다.

문제설명
데이터 중복같은 정보가 여러 파일에 저장되어 불일치 발생
데이터 종속성파일 구조 변경 시 모든 프로그램 수정 필요
동시 접근 불가여러 프로그램이 동시에 파일을 안전하게 수정하기 어려움
무결성 보장 없음"나이는 양수여야 한다" 같은 규칙을 코드마다 따로 구현해야 함
보안 제어 어려움파일 수준 권한만 가능, 레코드·컬럼 단위 제어 불가

데이터베이스 시스템의 해결책

  • DBMS(Database Management System): 데이터를 체계적으로 저장·관리·검색하는 소프트웨어 계층
  • 데이터 독립성: 물리적 저장 방식이 바뀌어도 애플리케이션 코드 불변
  • 동시성 제어: 여러 사용자가 동시에 접근해도 일관성 유지
  • 무결성 제약: DB 수준에서 규칙을 강제
  • 백업 & 복구: 장애 발생 시 일관된 상태로 복원

핵심 구분: 데이터베이스(DB) = 데이터 집합 자체 / DBMS = 그것을 관리하는 소프트웨어 (PostgreSQL, MySQL, SQLite 등)

2. 관계형 모델 (Relational Model)

1970년 Edgar Codd가 제안. 현재도 가장 널리 쓰이는 DB 모델.

핵심 용어

릴레이션(Relation) = 테이블
    ├── 속성(Attribute)  = 열(Column)  — 예: 이름, 나이, 학번
    ├── 튜플(Tuple)      = 행(Row)     — 예: ('홍길동', 20, '2024001')
    └── 도메인(Domain)   = 속성이 취할 수 있는 값의 집합
                          예: 나이 도메인 = 0 이상의 정수

실제 예시

학생(Student) 릴레이션

학번 (PK)이름나이전공
2024001홍길동20컴퓨터공학
2024002김철수22경영학
2024003이영희21컴퓨터공학
  • 릴레이션 스키마: Student(학번, 이름, 나이, 전공) — 구조 정의
  • 릴레이션 인스턴스: 현재 저장된 튜플의 집합 — 위 표의 3개 행

릴레이션의 수학적 성질

  1. 튜플의 순서는 무의미하다 (집합이므로)
  2. 속성의 순서도 무의미하다
  3. 튜플은 중복될 수 없다 (기본키 제약이 이를 보장)
  4. 각 속성값은 원자적(Atomic)이어야 한다 — 1NF의 기초

3. 키(Key)의 종류

데이터를 식별하고 관계를 맺는 핵심 개념.

슈퍼키 (Superkey)

튜플을 유일하게 식별할 수 있는 속성 집합 (단독 또는 복합).

예: {학번}, {학번, 이름}, {학번, 이름, 나이} 모두 슈퍼키

후보키 (Candidate Key)

슈퍼키 중 최소성을 만족하는 것 — 어떤 속성도 제거하면 식별력이 깨짐.

예: {학번} 은 후보키 / {학번, 이름} 은 후보키가 아님 (이름 제거해도 학번만으로 식별 가능)

기본키 (Primary Key, PK)

후보키 중 선택된 하나. 절대 NULL 불허, 변경 최소화.

외래키 (Foreign Key, FK)

다른 릴레이션의 기본키를 참조하는 속성. 두 테이블 사이의 참조 무결성 보장.

수강(학번 FK→Student.학번, 과목코드 FK→Course.과목코드, 성적)

4. 관계 대수 (Relational Algebra) 기초

SQL의 수학적 토대. 릴레이션을 입력받아 릴레이션을 반환하는 연산들.

단항 연산

연산기호의미SQL 대응
선택(Selection)σ조건에 맞는 행 반환WHERE
투영(Projection)π지정한 열만 반환SELECT col1, col2
이름변경(Rename)ρ릴레이션/속성 이름 변경AS

예시:

  • σ 전공='컴퓨터공학'(Student) → 전공이 컴퓨터공학인 행들
  • π 이름,나이(Student) → 이름과 나이 열만 추출

이항 연산

연산기호의미SQL 대응
합집합(Union)∪두 릴레이션의 합UNION
교집합(Intersection)∩공통 튜플INTERSECT
차집합(Difference)−한쪽에만 있는 튜플EXCEPT
카티션 곱(Cartesian Product)×모든 조합CROSS JOIN
자연 조인(Natural Join)⋈공통 속성 기준 결합JOIN ... ON

5. ERD (Entity-Relationship Diagram)

DB 설계 단계에서 개념적 구조를 시각화하는 도구.

구성 요소

엔티티(Entity)     □ — 관리할 대상 (학생, 강좌, 교수)
속성(Attribute)    ○ — 엔티티의 특성 (이름, 나이)
관계(Relationship) ◇ — 엔티티 간의 연관 (수강하다, 담당하다)

카디널리티 (Cardinality)

관계에 참여하는 개체 수의 비율.

표기의미
1:1한 교수가 한 학과를 담당
1:N한 학과에 여러 학생 소속
M:N한 학생이 여러 강좌 수강, 한 강좌에 여러 학생

ERD → 릴레이션 변환 규칙

  1. 엔티티 → 테이블
  2. 단순 속성 → 컬럼
  3. 1:N 관계 → N 쪽 테이블에 FK 추가
  4. M:N 관계 → 별도 교차 테이블 생성 (학생-수강-강좌)

6. 실습 과제

다음 시나리오를 보고 아래 질문에 답해 보세요.

시나리오: 온라인 서점 시스템

  • 고객(고객ID, 이름, 이메일)
  • 도서(ISBN, 제목, 저자, 가격)
  • 주문(주문번호, 주문일, 고객ID)
  • 주문상세(주문번호, ISBN, 수량)

질문

  1. 각 테이블의 기본키는 무엇인가?
  2. 외래키 관계를 모두 나열하라.
  3. 고객과 도서의 관계는 1:1, 1:N, M:N 중 무엇인가? 왜?
  4. σ 가격>20000(도서) 를 SQL로 표현하면?
  5. π 이름,이메일(고객) 를 SQL로 표현하면?
정답 보기
  1. 기본키: 고객(고객ID), 도서(ISBN), 주문(주문번호), 주문상세(주문번호+ISBN 복합키)
  2. 외래키: 주문.고객ID → 고객.고객ID / 주문상세.주문번호 → 주문.주문번호 / 주문상세.ISBN → 도서.ISBN
  3. M:N — 한 고객이 여러 도서 주문 가능, 한 도서를 여러 고객이 주문 가능
  4. SELECT * FROM 도서 WHERE 가격 > 20000;
  5. SELECT 이름, 이메일 FROM 고객;

7. 핵심 요약

데이터베이스 시스템
├── 파일 시스템 한계 해결: 중복·종속성·동시성·무결성·보안
├── 관계형 모델: 릴레이션(테이블) = 튜플(행) + 속성(열) + 도메인
├── 키: 슈퍼키 ⊃ 후보키 → 기본키 선택 / 외래키로 참조 무결성
├── 관계 대수: σ(선택) π(투영) ⋈(조인) — SQL의 수학적 기반
└── ERD: 개념적 설계 도구, 엔티티·속성·관계·카디널리티
