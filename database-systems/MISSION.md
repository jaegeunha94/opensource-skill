# 데이터베이스 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 개발자 기술 면접에서 데이터베이스 지식을 **스키마 설계, 트랜잭션, 인덱스, 쿼리 튜닝, 복제, 장애 대응**과 연결해 설명할 수 있도록 훈련한다.

경력자 DB 면접은 SQL 문법 암기를 확인하는 자리가 아니다. 면접관은 느린 쿼리, lock wait, deadlock, 데이터 불일치, 인덱스 설계 실패, read/write 분리, 마이그레이션 리스크 같은 실제 문제에서 어떤 근거로 판단하는지 본다.

## 평가 대상 역량

| 역량 | 면접에서 확인하려는 것 |
|------|------------------------|
| 모델링 | 요구사항을 정규화/역정규화, 제약조건, 관계 설계로 바꿀 수 있는가 |
| 트랜잭션 | ACID, isolation level, MVCC, lock을 실무 이상 현상과 연결하는가 |
| 인덱스 | B+Tree, composite index, covering index, selectivity, 실행 계획을 설명하는가 |
| 쿼리 튜닝 | EXPLAIN을 읽고 병목을 좁히며 개선안을 검증하는가 |
| 동시성 | lock wait, deadlock, lost update, phantom read 대응을 말할 수 있는가 |
| 확장성 | replication, sharding, partitioning, caching의 trade-off를 이해하는가 |
| 운영 | migration, backup/restore, 장애 조치, 데이터 정합성 검증을 고려하는가 |

## Interview 레슨 작성 규칙

각 Day 레슨은 반드시 **면접 질문 중심**으로 작성한다.

1. **면접 질문**
   - 실무 판단을 요구하는 질문으로 시작한다.
   - 예: "트래픽이 늘면서 특정 API가 느려졌습니다. DB 관점에서 어떤 순서로 확인하시겠습니까?"
2. **전제 개념**
   - 어려운 개념은 먼저 쉬운 한국어로 설명한다.
   - 정규화, 인덱스, MVCC, 격리 수준 같은 개념은 필요한 배경부터 설명한다.
3. **왜 면접에서 묻는가**
   - 데이터 안정성, 성능, 운영 리스크, 설계 판단력 중 무엇을 확인하는 질문인지 적는다.
4. **실무 연결**
   - 주문, 결제, 재고, 알림, 로그, 대시보드, 관리자 검색 같은 도메인과 연결한다.
5. **답변 사고 순서**
   - 증상 정의 -> 데이터 모델 확인 -> 쿼리/실행 계획 확인 -> 인덱스/락 확인 -> 개선안 검증 -> 운영 리스크 점검 순서로 답변하도록 안내한다.
6. **핵심 답변 예시**
   - 면접에서 바로 말할 수 있는 답변을 짧은 버전과 깊이 있는 버전으로 제공한다.
7. **Trade-off**
   - 읽기 성능, 쓰기 성능, 저장 공간, 정합성, 운영 복잡도, 장애 복구 난이도를 비교한다.
8. **흔한 오해**
   - "인덱스는 많을수록 좋다", "격리 수준을 높이면 항상 안전하다", "NoSQL은 스키마가 없다" 같은 오해를 바로잡는다.
9. **Follow-up 질문**
   - 면접관이 더 깊게 물을 질문과 그 질문의 의도를 포함한다.
10. **10년차 이상 답변 기준**
   - 정의 암기가 아니라 데이터 무결성, 성능, 운영 안정성을 함께 고려하는지 평가한다.

## 완료 기준

- DB 문제를 쿼리 하나가 아니라 모델, 인덱스, 트랜잭션, 운영 리스크로 나누어 설명할 수 있다.
- EXPLAIN과 lock 현상을 바탕으로 병목 가설을 세울 수 있다.
- 정규화/역정규화, RDB/NoSQL, 동기/비동기 복제의 trade-off를 말할 수 있다.
- 약한 답변과 강한 답변의 차이를 스스로 설명할 수 있다.
