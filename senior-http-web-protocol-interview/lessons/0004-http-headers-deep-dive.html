1. 핵심 면접 질문

Primary Question

"클라이언트의 진짜 IP를 어떻게 신뢰성 있게 알아내나요? 왜 X-Forwarded-For를 그대로 믿으면 안 되나요?"

Variant 1

"hop-by-hop 헤더와 end-to-end 헤더의 차이는 무엇이고, 이 구분이 리버스 프록시를 설계할 때 왜 중요한가요?"

Variant 2

"여러 마이크로서비스를 거치는 요청 하나를 어떻게 추적하시겠어요? 헤더를 어떻게 설계하시나요?"

2. 왜 면접에서 묻는가

이 질문은 "헤더 이름을 몇 개 아는가"가 아니라 "헤더 값을 누가 썼는지, 그 값을 신뢰해도 되는지를 판단할 수 있는가"를 테스트합니다. "X-Forwarded-For에 클라이언트 IP가 들어 있습니다"라는 설명은 신입도 합니다. 시니어를 가르는 지점은 그 값이 조작 가능한 텍스트에 불과하다는 것을 알고, 어느 지점에서부터 그 값을 신뢰할 수 있는지 설계할 수 있는가입니다.

면접관이 이 질문으로 평가하는 것:

  • 신뢰 경계(trust boundary) 감각 — Day 1에서 그린 트래픽 경로(클라이언트 → CDN → 리버스 프록시 → 게이트웨이 → 앱 서버) 위에서, "이 헤더 값은 어느 구간부터 조작 불가능해지는가"를 특정할 수 있는가.
  • hop-by-hop / end-to-end 구분 — 프록시를 직접 설계하거나 설정해본 사람만 자연스럽게 아는 구분입니다. 이 구분을 모르면 프록시가 Connection 같은 헤더를 그대로 전달해 연결 관리가 깨지는 실수를 할 수 있습니다.
  • 분산 추적 설계 경험 — 마이크로서비스 환경에서 요청 하나가 여러 서비스를 거칠 때, 로그를 어떻게 하나로 묶는지(X-Request-Id, traceparent)를 실제로 설계해봤는가.
  • 보안 판단 — 조작 가능한 헤더 값을 IP 기반 접근 제어나 rate limiting의 근거로 그대로 쓰면 우회당한다는 것을 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "X-Forwarded-For 헤더 첫 번째 값을 클라이언트 IP로 씁니다." 방향은 맞지만 그 값을 누가 처음 썼는지, 클라이언트가 그 값을 스스로 조작해서 넣을 수 있다는 사실을 언급하지 않습니다. 시니어는 "이 헤더는 신뢰할 수 있는 지점에서 덮어써야 한다"는 원칙을 먼저 세우고, 그다음에 값을 읽습니다.

3. 전제 개념 — 쉬운 한국어로

헤더는 "본문의 부속품"이 아니라 "요청에 붙은 메타데이터 텍스트"다

HTTP 헤더는 이름: 값 형태의 텍스트 줄일 뿐입니다. 요청을 보내는 쪽(브라우저든, 서버든, curl이든)은 원하는 이름과 값을 얼마든지 자유롭게 넣을 수 있습니다. 이 사실이 이 레슨 전체의 출발점입니다 — 헤더 값은 그 자체로는 아무 신뢰도가 없는 자기 신고(self-declared) 정보입니다. 신분증에 비유하면, 헤더는 사람이 스스로 "저는 홍길동입니다"라고 말로 하는 자기소개와 같습니다. 실제로 그 사람이 홍길동인지 확인하려면(신뢰하려면) 중간에 신분증을 검사하는 절차(우리가 통제하는 인프라 지점)가 있어야 합니다.

왜 "신뢰 경계"라는 개념이 필요한가

Day 1에서 그린 트래픽 경로를 다시 떠올려 보면: 클라이언트 → CDN → 리버스 프록시/로드밸런서 → API 게이트웨이 → 앱 서버. 이 경로에서 클라이언트가 보낸 원본 요청은 우리가 전혀 통제할 수 없는 영역이고, CDN부터 앱 서버까지는 우리가 직접 운영하거나 계약을 맺은 신뢰 가능한 인프라입니다. 이 둘의 경계선을 "신뢰 경계"라고 부릅니다. 신뢰 경계를 넘어오기 전에 우리 인프라가 검증하거나 덮어쓰지 않은 값은, 아무리 그럴듯해 보여도 클라이언트가 마음대로 지어낸 값일 수 있습니다.

Hop-by-hop vs End-to-end — 헤더가 "한 구간용"인가 "전체 경로용"인가

요청이 여러 프록시를 거쳐 갈 때, 어떤 헤더는 바로 다음 연결 구간(hop)에서만 의미가 있고 다음 홉으로 전달되면 안 되는 헤더(hop-by-hop)이고, 어떤 헤더는 클라이언트에서 최종 서버까지 변형 없이 그대로 유지되어야 하는 헤더(end-to-end)입니다. 우체국 배송에 비유하면, hop-by-hop 헤더는 "이번 트럭 기사가 다음 트럭 기사에게 전달할 필요 없는, 이번 구간에서만 쓰는 배송 메모"(예: "이 트럭은 몇 번째 경유지인가")이고, end-to-end 헤더는 "상자 안에 든 실제 배송 내용물 정보"(수신인, 상품명)처럼 최종 목적지까지 그대로 유지되어야 하는 정보입니다.

왜 이 구분이 프록시 설계와 직결되는가

리버스 프록시나 게이트웨이를 직접 구현/설정한다면, hop-by-hop 헤더는 다음 서버로 그대로 전달하면 안 됩니다 — 예를 들어 클라이언트가 프록시와 맺은 연결 관리 정보(Connection: keep-alive)를 프록시가 업스트림 서버에도 그대로 전달하면, 서로 다른 연결의 상태 정보가 뒤섞여 연결 관리가 깨질 수 있습니다. 이 구분을 모르면 "헤더는 그냥 다 전달하면 되는 것 아닌가"라고 생각하기 쉽고, 실제로 이는 흔한 프록시 설정 버그의 원인입니다.

4. Hop-by-hop vs End-to-end 헤더 판단표

헤더 종류 이유 프록시의 처리 방법
Connection Hop-by-hop 이 TCP 연결을 유지할지 말지를 지정 — 연결은 각 구간마다 별개 다음 홉으로 전달하지 않음. 프록시가 각 구간별로 독립적으로 판단
Keep-Alive Hop-by-hop 연결 유지 파라미터(timeout, max) — 구간별로 다를 수 있음 다음 홉으로 전달하지 않음
Transfer-Encoding Hop-by-hop 이 구간에서 메시지를 어떻게 인코딩해 전송했는지 (chunked 등) 다음 홉에서 재인코딩 여부를 독립적으로 결정
Upgrade Hop-by-hop 이 연결을 다른 프로토콜(WebSocket 등)로 전환 요청 — 구간별 협상 다음 홉으로 그대로 전달하지 않고 프록시가 별도로 처리
Proxy-Authorization Hop-by-hop 바로 다음 프록시에 대한 인증 정보 — 그 뒤로 전달되면 안 됨 해당 프록시가 소비하고 다음 홉에는 제거
Content-Type End-to-end 실제 메시지 본문의 형식 — 최종 수신자가 알아야 할 정보 모든 프록시가 변형 없이 그대로 전달
Authorization End-to-end 최종 리소스 서버가 검증해야 할 클라이언트 인증 정보 그대로 전달(단, API 게이트웨이가 검증 후 내부 토큰으로 교체하는 설계도 흔함 — Day 17)
Cache-Control End-to-end 캐시 정책은 경로상의 모든 캐시(브라우저·CDN·프록시)가 동일하게 해석해야 함 그대로 전달 (Day 8에서 심화)
Host End-to-end (단, 재작성 흔함) 원래 클라이언트가 요청한 도메인 정보 프록시가 업스트림 라우팅을 위해 재작성하되, 원본 값은 X-Forwarded-Host로 보존

RFC 9110의 정의 요약

RFC 9110(HTTP Semantics)은 hop-by-hop 헤더를 "단일 전송 계층 연결에 대해서만 의미가 있으며, 프록시에 의해 전달되어서는 안 되는 헤더"로 정의합니다. 프록시가 이 원칙을 지키지 않으면, 서로 다른 구간의 연결 상태 정보가 섞여 커넥션 재사용(Day 11 keep-alive)이 오작동하거나 프로토콜 업그레이드가 실패하는 등의 문제가 생깁니다.

5. X-Forwarded-For 신뢰 경계 — 스푸핑 방지 설계

X-Forwarded-For는 정식 RFC 표준이 아니라 업계에서 굳어진 사실상의 관례(de-facto convention)입니다(표준화를 시도한 것이 RFC 7239의 Forwarded 헤더이지만 실무에서는 여전히 X-Forwarded-For가 압도적으로 많이 쓰입니다). 이 헤더는 프록시를 거칠 때마다 "이 요청이 어떤 IP에서 왔는지"를 콤마로 구분해 누적합니다.

X-Forwarded-For: <client-claimed-ip>, <proxy1-added-ip>, <proxy2-added-ip>

문제는 이 헤더의 맨 앞 값을 클라이언트가 스스로 채워서 보낼 수 있다는 것입니다. 다음 요청은 완전히 유효한 HTTP 요청입니다:

curl -H "X-Forwarded-For: 8.8.8.8" https://api.example.com/orders

서버가 이 헤더의 첫 값을 그대로 "클라이언트 IP"로 신뢰하면, IP 기반 rate limiting이나 지역 차단, 화이트리스트 접근 제어를 손쉽게 우회당합니다.

올바른 설계 절차

  • 1
    "최초 신뢰 지점"을 특정한다. 우리 인프라 중 클라이언트와 직접 TCP 연결을 맺는 첫 번째 지점(CDN edge, 또는 CDN이 없다면 우리가 운영하는 첫 로드밸런서)이 최초 신뢰 지점입니다.
  • 2
    최초 신뢰 지점에서 기존 X-Forwarded-For 값을 신뢰하지 않고, 실제 TCP 연결의 소스 IP로 덮어쓰거나 그 값을 새로 append한다. 이 시점부터는 값이 우리 인프라가 직접 관찰한 사실이 됩니다.
  • 3
    그 이후의 모든 내부 홉(리버스 프록시, 게이트웨이)은 기존 값에 자신의 관찰값을 append만 하고, 앞의 값을 지우거나 바꾸지 않는다. 이렇게 하면 헤더 전체가 "요청이 실제로 거쳐온 경로의 로그"가 됩니다.
  • 4
    앱 서버는 "신뢰하는 내부 홉 수"를 미리 알고, 끝에서부터 그 수만큼을 제외한 위치의 값을 실제 클라이언트 IP로 취급한다. 예를 들어 CDN → 리버스 프록시 → 앱 서버 구조에서 리버스 프록시가 마지막으로 append하므로, 앱 서버는 "끝에서 두 번째 값"을 신뢰합니다(리버스 프록시 자신의 IP는 신뢰 대상이 아니므로 제외).

실무 구현 예시 — Nginx

Nginx는 real_ip_module의 set_real_ip_from과 real_ip_recursive 설정으로 이 신뢰 경계를 구현합니다.

\# 우리가 운영하는 CDN/로드밸런서의 IP 대역만 신뢰
set_real_ip_from 10.0.0.0/8;
real_ip_header X-Forwarded-For;
real_ip_recursive on;

set_real_ip_from에 명시된 IP 대역에서 온 연결일 때만 X-Forwarded-For 값을 신뢰하고, 그 대역 밖(즉 인터넷의 임의 클라이언트)에서 직접 이 헤더를 위조해 보내는 경우는 무시합니다.

한 줄 원칙: "이 값을 마지막으로 누가 썼는가?"를 항상 먼저 묻습니다. 우리가 통제하는 신뢰 지점이 마지막으로 썼다면 신뢰할 수 있고, 클라이언트가 마지막으로 쓴 것이라면 신뢰할 수 없습니다.

6. Via / X-Request-Id로 트래픽 경로 재구성

Via — 표준화된 프록시 경유 기록

Via는 X-Forwarded-For와 달리 RFC 9110에 정의된 공식 표준 헤더입니다. 요청이나 응답이 프록시를 거칠 때마다, 각 프록시가 자신의 프로토콜 버전과 식별자를 콤마로 추가합니다.

Via: 1.1 cdn-edge-42, 1.1 internal-proxy-7

이 헤더는 "이 요청이 실제로 몇 개의 프록시를 거쳤는지, 어떤 프록시들인지"를 디버깅할 때 유용합니다 — 예를 들어 예상보다 Via 항목이 하나 더 많다면, 우리가 모르는 중간 프록시(회사 방화벽, ISP 캐싱 프록시 등)가 개입했다는 신호일 수 있습니다.

X-Request-Id / traceparent — 분산 환경에서 하나의 요청을 추적하기

마이크로서비스 환경에서는 클라이언트의 요청 하나가 API 게이트웨이 → 서비스 A → 서비스 B → 데이터베이스처럼 여러 홉을 거칩니다. 각 서비스는 독립적으로 로그를 남기므로, 이 로그들을 나중에 "같은 요청에서 나온 것"으로 묶으려면 요청 전체에 걸쳐 동일하게 유지되는 고유 ID가 필요합니다.

  • 1
    ID는 최초 신뢰 지점(보통 API 게이트웨이/edge)에서 생성한다. 클라이언트가 이미 X-Request-Id를 보냈더라도 그대로 신뢰하지 않고, 검증 후 사용하거나 별도 필드(X-Client-Request-Id)로 구분해서 남긴다 — 그렇지 않으면 클라이언트가 의도적으로 중복된 ID를 보내 로그 상관관계를 오염시킬 수 있습니다.
  • 2
    이후 모든 다운스트림 서비스 호출에 동일한 ID를 그대로 전달한다. 이 값은 end-to-end 헤더로 취급되어야 합니다 — 중간에 사라지거나 바뀌면 추적이 끊깁니다.
  • 3
    모든 서비스가 로그의 모든 라인에 이 ID를 포함하도록 강제한다. 로깅 미들웨어/인터셉터 레벨에서 자동으로 주입하는 것이 실무 표준입니다.

표준화 흐름 — W3C Trace Context

커스텀 X-Request-Id 관례는 회사마다 형식이 제각각이어서, 여러 벤더의 서비스를 함께 쓸 때 상호운용이 어렵습니다. 이를 표준화한 것이 W3C Trace Context 스펙의 traceparent 헤더입니다.

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │  │                                │                │
             │  trace-id (요청 전체 공통)          span-id(이 구간) flags

trace-id는 요청 전체에서 동일하고, span-id는 각 서비스 호출 구간마다 새로 생성됩니다. OpenTelemetry 같은 관측성 도구(Day 20에서 심화)는 이 표준을 기본으로 채택하고 있어, 신규 시스템을 설계한다면 커스텀 X-Request-Id보다 traceparent를 우선 검토하는 것이 실무 트렌드입니다.

7. 커스텀 헤더 설계 규칙

규칙 1 — X- 접두사를 새 헤더에 강제하지 않는다

RFC 6648(2012)은 비표준 헤더에 X- 접두사를 붙이는 관례를 공식적으로 폐기 권고했습니다. 이유는 X- 헤더가 실무에서 너무 널리 쓰이다 보니 나중에 표준으로 승격될 때(예: 예전의 X-Content-Type-Options) 접두사를 떼는 이름 변경이 발생해 하위 호환성 문제가 반복됐기 때문입니다. 다만 X-Forwarded-For, X-Request-Id처럼 이미 업계에 굳어진 관례는 그대로 유지하되, 새로 설계하는 헤더는 X- 없이 명확한 네임스페이스(회사/서비스명)를 접두어로 쓰는 것이 권장됩니다.

규칙 2 — 헤더 이름은 대소문자를 구분하지 않는다

Content-Type과 content-type은 같은 헤더입니다. 특히 HTTP/2/HTTP/3에서는 헤더 이름을 항상 소문자로 강제합니다(Day 14에서 심화). 클라이언트/서버 코드에서 헤더 이름을 문자열 그대로 비교(대소문자 구분)하면 프로토콜 버전에 따라 동작이 달라지는 버그가 생길 수 있습니다.

규칙 3 — 값에 민감정보를 넣지 않는다

헤더는 프록시, 게이트웨이, 로드밸런서, APM 도구 등 여러 지점에서 그대로 로그에 남는 경우가 흔합니다. 세션 토큰이나 개인정보를 커스텀 헤더 값으로 직접 넣으면, 우리가 통제하지 못하는 중간 로그 시스템에 그대로 노출될 위험이 있습니다.

규칙 4 — 헤더 크기와 개수에는 실질적 제한이 있다

Nginx, 대부분의 클라우드 로드밸런서는 기본적으로 요청 헤더 전체 크기에 상한(예: 8KB)을 둡니다. 트레이스 컨텍스트나 커스텀 메타데이터를 헤더에 과도하게 누적하면 431 Request Header Fields Too Large 에러로 이어질 수 있습니다 — 특히 Via나 X-Forwarded-For처럼 매 홉마다 append되는 헤더는 프록시 체인이 길어질수록 커진다는 점을 설계 시 고려해야 합니다.

8. 답변 사고 순서

면접에서 "이 헤더를 어떻게 신뢰하고 처리하시겠어요?" 류의 질문을 받으면 이 순서로 사고합니다.

  1. 먼저 이 헤더가 hop-by-hop인지 end-to-end인지 구분합니다. 프록시가 그대로 전달해도 되는 값인지, 각 구간에서 새로 협상되어야 하는 값인지가 첫 판단 기준입니다.
  2. 이 값을 클라이언트가 직접 조작할 수 있는지 확인합니다. 모든 요청 헤더는 원칙적으로 클라이언트가 임의로 채울 수 있다는 전제에서 출발합니다.
  3. 우리 인프라의 "최초 신뢰 지점"이 어디인지 특정합니다. CDN edge인지, 우리가 직접 운영하는 로드밸런서인지 트래픽 경로 위에서 짚습니다.
  4. 그 지점에서 값을 덮어쓰거나 append하는 구체적 절차를 설명합니다. Nginx의 set_real_ip_from처럼 실제 구현 메커니즘을 언급하면 답변의 구체성이 올라갑니다.
  5. 다운스트림에서 이 값을 쓸 때 "몇 번째 홉까지가 신뢰 가능한가"를 계산하는 방법을 제시합니다.
  6. 보안/운영 관점(우회 가능성, 로그 상관관계)까지 연결하면 답변이 완성됩니다.

9. 핵심 답변 예시

면접관: "클라이언트의 진짜 IP를 어떻게 신뢰성 있게 알아내나요? 왜 X-Forwarded-For를 그대로 믿으면 안 되나요?"

STRONG ANSWER

X-Forwarded-For는 표준이 아니라 관례이고, 무엇보다 모든 요청 헤더는 클라이언트가 스스로 채울 수 있는 값이라는 게 출발점입니다. 클라이언트가 curl -H "X-Forwarded-For: 8.8.8.8"처럼 임의의 IP를 넣어 보내면, 이 값을 그대로 믿는 서버는 IP 기반 접근 제어나 rate limiting을 손쉽게 우회당합니다.


그래서 저는 "이 값을 마지막으로 누가 썼는가"를 기준으로 신뢰 경계를 설계합니다. 우리 인프라 중 클라이언트와 직접 TCP 연결을 맺는 첫 지점 — 보통 CDN edge나 우리가 운영하는 첫 로드밸런서 — 을 "최초 신뢰 지점"으로 정하고, 그 지점에서 기존 X-Forwarded-For 값을 신뢰하지 않고 실제 TCP 연결의 소스 IP로 새로 append합니다. 이후의 내부 홉들은 이 값을 신뢰하고 계속 append만 합니다.


앱 서버 쪽에서는 신뢰하는 내부 홉의 개수를 알고 있어야 합니다. 예를 들어 리버스 프록시 하나만 거친다면 끝에서 두 번째 값이 실제 클라이언트 IP입니다 — 마지막 값은 리버스 프록시 자신의 IP이므로 제외합니다. Nginx라면 real_ip_module의 set_real_ip_from으로 신뢰하는 IP 대역을 명시해서 이 로직을 구현합니다.


더 근본적으로는 RFC 7239 표준 Forwarded 헤더가 이 문제를 해결하려는 시도였지만, 실무에서는 여전히 X-Forwarded-For가 더 널리 쓰이고 있어서, 결국 헤더 이름보다 "신뢰 경계를 어디에 긋는가"가 실질적인 보안 설계의 핵심이라고 생각합니다.

이 답변이 좋은 이유:

  • "헤더는 클라이언트가 조작 가능하다"는 근본 원리에서 답변을 시작함
  • 신뢰 경계라는 명확한 설계 개념으로 문제를 구조화함
  • Nginx real_ip_module처럼 실제 구현 메커니즘을 구체적으로 언급함
  • "끝에서 몇 번째 값을 신뢰하는가"라는 계산 가능한 기준을 제시함
  • RFC 7239와의 관계까지 짚어 스펙 지식의 폭을 보여줌

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"X-Forwarded-For 헤더의 첫 번째 값을 클라이언트 IP로 사용하면 됩니다."

이 답변이 약한 이유

  • 이 헤더가 클라이언트에 의해 조작될 수 있다는 사실을 전혀 언급하지 않았습니다.
  • "첫 번째 값"이 항상 클라이언트 IP라는 보장이 없습니다 — 신뢰 경계 설계 없이는 그 값이 공격자가 지어낸 값일 수 있습니다.
  • 신뢰 지점을 어디에 두어야 하는지에 대한 설계가 전혀 없습니다.
WEAK ANSWER — 또 다른 패턴

"TCP 연결의 소스 IP를 그대로 쓰면 헤더 조작 문제가 아예 없어서 더 안전합니다."

이 답변이 약한 이유

  • 실제 프로덕션 환경에서는 클라이언트가 CDN이나 로드밸런서를 거쳐 들어오므로, TCP 연결의 소스 IP는 클라이언트가 아니라 바로 앞 프록시의 IP입니다. 이 사실을 놓치면 실무에 적용 불가능한 답이 됩니다.
  • 결국 프록시를 거치는 구조에서는 "프록시가 관찰한 클라이언트 IP를 헤더로 전달"하는 메커니즘 자체가 필요하다는 것을 인지하지 못했습니다.
STRONG ANSWER 요소 체크리스트
  • 모든 요청 헤더가 클라이언트에 의해 조작 가능한 자기 신고 값이라는 원리에서 시작한다
  • "신뢰 경계"라는 개념으로 트래픽 경로 위의 특정 지점을 지목한다
  • 최초 신뢰 지점에서 값을 덮어쓰거나 append하는 구체적 메커니즘을 설명한다
  • "끝에서 몇 번째 값을 신뢰하는가"처럼 다운스트림에서의 계산 방법을 제시한다
  • 실제 구현(Nginx real_ip_module 등)이나 관련 표준(RFC 7239)을 언급한다
  • 이 신뢰 경계가 왜 보안(rate limiting/접근 제어 우회 방지)과 직결되는지 연결한다

11. Senior-level Follow-up Q&A

Q: CDN을 여러 겹 쓰는 경우(예: Cloudflare 뒤에 우리 자체 프록시가 하나 더 있는 경우), X-Forwarded-For를 어떻게 관리해야 하나요?

모범 답변: 신뢰 경계는 "우리가 통제하는 첫 지점"이지 반드시 인터넷과 맞닿은 첫 지점이 아닙니다. Cloudflare 같은 외부 CDN은 우리가 직접 운영하지는 않지만 계약을 통해 신뢰하는 파트너 인프라이므로, Cloudflare가 채워주는 CF-Connecting-IP 같은 벤더 전용 헤더나 Cloudflare가 재작성한 X-Forwarded-For를 신뢰 가능한 값으로 간주하되, 반드시 "이 요청이 실제로 Cloudflare의 IP 대역에서 왔는지"를 우리 프록시 레벨에서 검증해야 합니다. 검증 없이 벤더 헤더를 그대로 믿으면, 공격자가 Cloudflare를 거치지 않고 우리 서버에 직접 요청을 보내면서 Cloudflare 전용 헤더를 위조하는 경로가 남습니다.

Q: X-Forwarded-For 대신 표준화된 Forwarded 헤더(RFC 7239)를 실무에서 잘 쓰지 않는 이유는 무엇인가요?

모범 답변: Forwarded 헤더는 for=, proto=, host= 같은 파라미터를 한 헤더에 구조화해서 담을 수 있어 스펙상으로는 더 우수합니다. 하지만 X-Forwarded-For가 이미 수십 년간 광범위하게 채택되어 대부분의 프록시/로드밸런서/CDN/APM 도구의 기본 설정과 문서, 파싱 라이브러리가 이 관례를 전제로 만들어져 있습니다. 새 표준으로 전환하는 비용(모든 중간 인프라의 설정 변경, 로그 파이프라인 수정)이 얻는 이득보다 커서 관성이 유지되는 전형적인 사례입니다. 신규 그린필드 프로젝트라면 Forwarded 헤더 채택을 검토할 가치가 있지만, 기존 생태계와의 상호운용을 위해 X-Forwarded-For를 당분간 병행하는 것이 현실적입니다.

Q: Authorization 헤더는 hop-by-hop인가요, end-to-end인가요? 프록시가 이 헤더 내용을 봐야 하는 경우는 언제인가요?

모범 답변: Authorization은 end-to-end 헤더입니다 — 최종적으로 리소스를 소유한 서버가 검증해야 할 정보이기 때문에 중간에 임의로 제거되면 안 됩니다. 다만 실무에서는 API 게이트웨이가 이 헤더를 "그냥 통과"시키는 것을 넘어, 게이트웨이 레벨에서 토큰을 검증하고 그 결과(예: 사용자 ID, 권한 스코프)를 내부 전용 헤더(X-User-Id, X-Auth-Scope)로 변환해 다운스트림 서비스에 전달하는 설계가 흔합니다(Day 17 API Gateway에서 심화). 이렇게 하면 각 다운스트림 서비스가 토큰 검증 로직을 중복 구현하지 않아도 되지만, 이 내부 헤더들은 게이트웨이 뒤에서만 신뢰되어야 하고 외부에서 직접 위조해 들어올 수 없도록 게이트웨이가 유일한 진입점이 되도록 네트워크 경계를 막아야 합니다.

Q: X-Request-Id를 클라이언트가 직접 생성해서 요청에 실어 보내면 어떤 문제가 생기나요?

모범 답변: 두 가지 문제가 있습니다. 첫째, 여러 클라이언트가 우연히 또는 의도적으로 같은 ID를 보내면 서로 다른 요청의 로그가 같은 ID 아래 섞여, 트레이스 조회 시 잘못된 요청의 로그까지 함께 나오는 상관관계 오염이 생깁니다. 둘째, 악의적인 클라이언트가 의도적으로 다른 요청과 동일한 ID를 반복해서 보내 로그 분석이나 이상 탐지 시스템을 혼란시키는 것도 이론적으로 가능합니다. 그래서 신뢰 지점(게이트웨이)에서 새 ID를 생성하는 것이 원칙이고, 클라이언트가 이미 보낸 값을 완전히 버리기보다는 X-Client-Request-Id 같은 별도 필드로 남겨 "클라이언트가 자기 쪽에서 어떤 요청이라고 주장했는지"를 참고 정보로만 유지하는 것이 실무적인 절충안입니다.

Q: HTTP/2에서는 헤더가 HPACK으로 압축된다고 들었습니다. 이게 커스텀 헤더 설계에 어떤 영향을 주나요?

모범 답변: HPACK은 이전에 전송된 헤더 이름/값 쌍을 정적·동적 테이블에 캐싱해서, 같은 연결 안에서 반복되는 헤더를 다시 통째로 보내지 않고 짧은 인덱스 참조로 대체합니다(Day 14에서 심화). 이 구조 때문에 매 요청마다 값이 바뀌는 헤더(예: 요청마다 새로 생성되는 X-Request-Id, 매번 달라지는 타임스탬프)는 압축 효율이 거의 없어 오히려 일반 헤더보다 오버헤드가 클 수 있습니다. 반면 User-Agent나 Authorization처럼 같은 연결 안에서 반복되는 값은 압축 이득이 큽니다. 그래서 커스텀 헤더를 설계할 때는 "이 값이 매 요청마다 바뀌는가"를 인지하고, 자주 바뀌는 값은 헤더 개수를 최소화하거나 필요한 경우에만 포함시키는 것이 HTTP/2 이상 환경에서의 성능 감각입니다.

12. 흔한 오해

오해 1

"X-Forwarded-For의 첫 번째 값은 항상 진짜 클라이언트 IP다"

신뢰 경계 설계 없이는 첫 번째 값은 클라이언트가 스스로 채운 임의의 문자열일 수 있습니다. 신뢰 가능한 값은 우리 인프라의 최초 신뢰 지점이 직접 관찰해 기록한 값뿐입니다.

오해 2

"Via 헤더가 응답에 붙으면 캐시가 동작하지 않는다"

Via는 단순히 이 응답이 어떤 프록시들을 거쳤는지 기록하는 진단용 헤더일 뿐, 캐시 가능 여부와는 무관합니다. 캐시 동작은 Cache-Control과 Vary가 결정합니다(Day 5, Day 8에서 심화).

오해 3

"커스텀 헤더 이름은 아무렇게나 지어도 상관없다"

이름 자체는 자유롭지만, 로그 파이프라인·프록시 헤더 크기 제한·HTTP/2 소문자 강제 같은 실무 제약이 있습니다. 특히 프록시 체인마다 append되는 헤더(X-Forwarded-For, Via)는 체인이 길어질수록 커진다는 점을 감안해야 합니다.

오해 4

"클라이언트가 준 X-Request-Id를 그대로 써도 안전하다"

클라이언트가 준 값을 검증 없이 그대로 신뢰하면 로그 상관관계가 오염될 수 있습니다. 신뢰 지점에서 새 ID를 생성하거나, 클라이언트 값은 참고용 별도 필드로 분리하는 것이 원칙입니다.

오해 5

"프록시는 헤더를 그대로 다 전달해야 한다"

hop-by-hop 헤더(Connection, Keep-Alive, Transfer-Encoding, Upgrade 등)는 다음 홉으로 전달되면 안 됩니다. 이 원칙을 지키지 않으면 연결 관리, 프로토콜 업그레이드가 오작동할 수 있습니다.
