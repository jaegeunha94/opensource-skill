1. 핵심 면접 질문

Primary Question

"네트워크 오류로 응답을 못 받은 POST 요청을 재시도해도 안전한가요? 안전하게 만들려면 어떻게 설계해야 하나요?"

Variant 1

"리소스를 생성할 때 PUT과 POST 중 무엇을 쓸지 어떤 기준으로 결정하시나요?"

Variant 2

"DELETE 요청을 두 번 보내면 두 번째는 404가 나올 텐데, 그래도 DELETE를 idempotent하다고 부르는 이유가 뭔가요?"

2. 왜 면접에서 묻는가

이 질문은 "메서드 이름을 암기했는가"가 아니라 "분산 환경에서 네트워크 실패가 실제로 어떤 모양을 하는지, 그리고 그 실패에 재시도로 대응해도 안전한지 판단할 수 있는가"를 테스트합니다. GET은 조회, POST는 생성이라는 사전적 정의는 신입도 말합니다. 시니어를 가르는 지점은 "이 메서드로 재시도했을 때 부작용이 몇 번 일어나는가"를 정확히 추론하는 능력입니다.

면접관이 이 질문으로 평가하는 것:

  • 네트워크 불확실성에 대한 이해 — 타임아웃이 "요청이 실패했다"는 뜻이 아니라 "응답을 못 받았을 뿐, 서버는 이미 처리했을 수도 있다"는 사실을 아는가.
  • Safe/Idempotent 개념의 실무 적용 — 스펙의 정의를 아는 것을 넘어, 이 구분이 로드밸런서·클라이언트 라이브러리·API 게이트웨이의 자동 재시도 정책과 직결된다는 것을 아는가.
  • 설계 판단력 — POST처럼 원래 idempotent하지 않은 메서드도 idempotency key 같은 장치로 안전하게 만들 수 있다는 것을 알고, 언제 그 장치가 필요한지 판단할 수 있는가.
  • 스펙과 현실의 괴리를 인식하는가 — DELETE가 "idempotent하다"는 스펙상의 의미와, 실제로 두 번째 호출이 404를 반환하는 현실이 모순되지 않는 이유를 설명할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "GET은 조회라 여러 번 해도 되고 POST는 생성이라 조심해야 합니다." 이 답변은 방향은 맞지만 "왜 조심해야 하는지", "구체적으로 어떤 장애로 이어지는지"에 대한 근거가 없습니다. 시니어는 이걸 "재시도했을 때 실제로 중복 결제가 발생하는 구체적 시나리오"로 설명합니다.

3. 전제 개념 — 쉬운 한국어로

왜 "재시도"라는 문제가 생기는가

분산 시스템에서 클라이언트가 서버로 요청을 보내고 응답을 기다리다가 타임아웃이 나면, 클라이언트는 다음 셋 중 무엇이 실제로 일어났는지 알 수 없습니다:

  1. 요청이 서버에 도착하기 전에 네트워크에서 유실됨 (서버는 아무것도 안 함)
  2. 요청은 도착해서 서버가 처리를 끝냈지만, 응답이 돌아오는 길에 유실됨 (서버는 이미 처리함)
  3. 서버가 아직 처리 중인데 클라이언트가 먼저 타임아웃을 냄 (처리는 나중에 완료될 수 있음)

택배로 비유하면, 문 앞에 놓고 간 택배 기사가 "배송 완료" 문자를 보냈는데 그 문자가 통신 장애로 나에게 오지 않은 상황과 같습니다. 나는 "택배가 안 왔다"고 생각해서 다시 주문하지만, 실제로는 이미 집 앞에 와 있을 수 있습니다. 클라이언트 입장에서 "응답을 못 받음"과 "서버가 처리 안 함"은 절대 같은 뜻이 아닙니다. 이 불확실성 때문에 "재시도해도 안전한가"라는 질문이 성립합니다.

Safe 메서드란 무엇인가

Safe는 "서버 상태를 변경하지 않는다"는 뜻입니다. GET, HEAD, OPTIONS가 대표적입니다. 도서관에서 책 목록을 조회하는 것과 같습니다 — 몇 번을 조회하든 책장의 책은 그대로입니다. Safe한 메서드는 몇 번을 호출해도 서버 데이터에 부작용이 없으므로, 브라우저의 프리페치(prefetch)나 크롤러가 사용자 동의 없이 호출해도 괜찮다는 전제가 깔려 있습니다.

Idempotent 메서드란 무엇인가

Idempotent는 "같은 요청을 여러 번 보내도 서버의 최종 상태가 한 번 보낸 것과 동일하다"는 뜻입니다. 여기서 핵심은 "결과 상태"이지 "부작용이 아예 없다"가 아닙니다. 이 둘을 구분하지 못하는 것이 가장 흔한 오해입니다 (11절에서 다시 다룹니다).

엘리베이터 버튼에 비유하면, 이미 눌려서 켜진 버튼을 다시 눌러도 엘리베이터는 여전히 그 층으로 오도록 예약된 상태 그대로입니다. 버튼을 5번 누르든 1번 누르든 "그 층에 엘리베이터가 오도록 예약됨"이라는 최종 상태는 같습니다. 반면 초인종은 누를 때마다 소리가 새로 울리므로(부작용이 매번 발생) idempotent하지 않습니다.

PUT /users/1 { name: "Alice" }를 세 번 보내도 사용자 1번의 이름은 계속 "Alice"입니다 — idempotent합니다. 반면 POST /orders { item: "book" }를 세 번 보내면 주문이 세 개 생길 수 있습니다 — idempotent하지 않습니다.

Safe와 Idempotent는 다른 개념이다

Safe한 메서드는 모두 idempotent합니다(부작용이 없으니 당연히 여러 번 해도 상태가 같음). 하지만 idempotent하다고 모두 safe한 것은 아닙니다. PUT과 DELETE는 서버 상태를 바꾸므로 safe하지 않지만, 여러 번 반복해도 최종 상태가 동일하므로 idempotent합니다. 이 둘을 같은 개념으로 혼동하면 "PUT은 상태를 안 바꾸는 거 아니냐"는 잘못된 결론에 이르게 됩니다.

왜 이 구분이 재시도 설계로 직결되는가

클라이언트 라이브러리, 로드밸런서, API 게이트웨이는 흔히 "타임아웃 나면 자동으로 재시도"하는 정책을 갖고 있습니다. 이 자동 재시도가 안전한지는 전적으로 그 요청의 메서드가 idempotent한지에 달려 있습니다. GET을 자동 재시도하는 것은 안전하지만, 멱등성이 보장되지 않은 POST를 무작정 자동 재시도하면 중복 주문, 중복 결제 같은 실질적 장애로 이어집니다. 이것이 "메서드 의미론"이 단순 문법이 아니라 신뢰성 설계의 근거가 되는 이유입니다.

4. 메서드별 Safe/Idempotent/Cacheable 분류표

메서드 Safe Idempotent Cacheable 일반적 용도
GET O O O 리소스 조회
HEAD O O O 바디 없이 헤더만 조회 (존재 확인, 크기 확인)
OPTIONS O O X 지원 메서드 확인, CORS Preflight (Day 7)
PUT X O X 리소스 전체 교체 (없으면 생성, 있으면 덮어쓰기)
DELETE X O X 리소스 삭제
POST X X 조건부(명시적 캐시 지시자 있을 때만) 리소스 생성, 처리 트리거(결제, 메일 발송 등)
PATCH X X (설계에 따라 O 가능) X 리소스 부분 수정

PATCH가 "X (설계에 따라 O 가능)"인 이유

PATCH는 스펙상 idempotency를 보장하지 않습니다. {"likes": "+1"} 같은 상대적 변경을 표현하는 PATCH는 여러 번 호출하면 값이 계속 증가하므로 idempotent하지 않습니다. 반면 {"status": "shipped"}처럼 절대값을 지정하는 PATCH는 여러 번 호출해도 결과가 같으므로 idempotent하게 설계할 수 있습니다. 즉 PATCH의 idempotency 여부는 메서드 자체가 아니라 API가 요청 바디를 상대적 변경으로 설계했는지, 절대값 지정으로 설계했는지에 달려 있습니다.

5. PUT vs POST 설계 판단

"리소스를 생성할 때 PUT을 쓸지 POST를 쓸지"는 취향 문제가 아니라 클라이언트가 리소스 식별자(ID)를 미리 알고 지정하는지 여부로 결정되는 설계 문제입니다.

기준 PUT POST
리소스 식별자(URI) 클라이언트가 지정 — PUT /users/1024 서버가 생성 후 결정 — POST /users → 서버가 새 ID 발급
의미 "이 URI의 리소스를 이 내용으로 완전히 교체(또는 생성)해라" "이 컬렉션에 새로운 항목을 추가/처리해라"
재시도 안전성 기본적으로 안전 (idempotent) 기본적으로 위험 — 별도 장치 필요 (6절)
대표 사례 S3 오브젝트 업로드(키를 클라이언트가 지정), 설정 파일 저장 주문 생성, 결제 요청, 파일 업로드(서버가 ID 발급)

실무 판단 기준 한 줄 요약

"클라이언트가 이 리소스의 최종 URI를 이미 알고 있고, 그 URI 전체를 이 내용으로 덮어써도 되는가?" → 그렇다면 PUT. "서버가 새 식별자를 발급해야 하거나, 요청이 단순 데이터 저장이 아니라 사이드 이펙트를 가진 처리(결제, 이메일 발송)를 트리거하는가?" → 그렇다면 POST이고, 재시도 안전성을 별도로 설계해야 합니다.

6. POST를 안전하게 재시도하는 법 — Idempotency Key

POST는 스펙상 idempotent하지 않지만, 결제 API 같은 실무 시스템에서는 "네트워크가 불안정해도 중복 결제 없이 안전하게 재시도"해야 하는 요구가 항상 존재합니다. 이때 쓰는 패턴이 Idempotency Key입니다.

  • 1
    클라이언트가 요청마다 고유한 키를 생성

    보통 UUID를 사용합니다. 같은 논리적 작업(예: "이 주문 하나 생성")에 대해서는 재시도하더라도 동일한 키를 재사용합니다.

    POST /payments
Idempotency-Key: 3f29a1c2-8b4d-4e6a-9c1a-7d5e8f0b2c11
{ "amount": 50000, "currency": "KRW" }
  • 2
    서버가 (키, 요청 결과)를 저장

    서버는 이 키로 요청을 처음 처리할 때 결제를 실행하고, 그 결과(성공/실패, 응답 바디)를 키와 함께 저장해 둡니다.

  • 3
    같은 키로 재요청이 오면 실제 처리를 건너뛰고 저장된 결과를 반환

    타임아웃 때문에 클라이언트가 같은 요청을 다시 보내도, 서버는 이미 처리한 키임을 확인하고 결제를 다시 실행하지 않은 채 이전 응답을 그대로 돌려줍니다. 이렇게 하면 POST라는 메서드 자체는 idempotent하지 않아도, 애플리케이션 레벨에서 idempotency를 흉내낼 수 있습니다.

이 패턴에서 시니어가 짚어야 할 트레이드오프

  • 키 저장소의 신뢰성 — (키, 결과) 저장 자체도 원자적이어야 합니다. "결제 실행"과 "키 저장"이 따로 커밋되면 그 사이에 서버가 죽었을 때 다시 중복 결제가 날 수 있습니다. 트랜잭션으로 묶거나 저장을 먼저 성공시킨 뒤 처리하는 순서 설계가 필요합니다.
  • 키의 TTL — 키를 영구 보관할 수는 없으므로 만료 시간을 둡니다. 그 만료 시간보다 늦게 재시도가 들어오면 다시 새로운 요청으로 처리됩니다.
  • 키 생성 책임은 클라이언트에 있다 — 서버가 키를 만들어주면 재시도 자체가 새로운 키를 받게 되어 의미가 없습니다. 클라이언트(또는 클라이언트 SDK)가 "이건 같은 논리적 요청이다"를 알고 키를 재사용해야 합니다.

7. 답변 사고 순서

면접에서 "이 요청 재시도해도 되나요?" 류의 질문을 받으면 이 순서로 사고합니다.

  1. 먼저 "응답을 못 받음"과 "서버가 처리 안 함"은 다르다는 전제를 명시합니다. 타임아웃은 결과를 모르는 상태이지 실패가 확정된 상태가 아닙니다.
  2. 해당 메서드가 idempotent한지 스펙 기준으로 판단합니다. GET/PUT/DELETE라면 원칙적으로 재시도가 안전합니다.
  3. POST/PATCH라면 API 설계(상대적 변경 vs 절대값 지정)를 확인합니다. 상대적 변경이면 재시도 시 값이 누적되어 위험합니다.
  4. 재시도가 필요한데 메서드 자체가 idempotent하지 않다면 Idempotency Key 같은 애플리케이션 레벨 장치를 제안합니다.
  5. 재시도 정책 자체의 설계(횟수 제한, 백오프, 어느 계층이 재시도를 수행하는지)까지 언급하면 답변이 완성됩니다. (재시도 정책 자체는 Day 13에서 더 깊게 다룹니다.)

8. 핵심 답변 예시

면접관: "네트워크 오류로 응답을 못 받은 POST 요청을 재시도해도 안전한가요? 안전하게 만들려면 어떻게 설계해야 하나요?"

STRONG ANSWER

먼저 전제를 짚고 갑니다. 클라이언트가 응답을 못 받았다는 것은 서버가 처리하지 않았다는 뜻이 아닙니다. 요청이 서버에 도착해 처리까지 끝났는데 응답만 유실됐을 가능성이 항상 있습니다. 그래서 "안전한가"라는 질문은 곧 "이 메서드가 idempotent한가"로 바뀝니다.


POST는 스펙상 idempotent하지 않습니다. 예를 들어 POST /orders가 주문을 새로 생성하는 API라면, 첫 번째 시도가 실제로는 성공했는데 응답만 유실된 상황에서 그대로 재시도하면 주문이 하나 더 생깁니다. 그러니 이 메서드를 그냥 재시도하는 것은 안전하지 않습니다.


안전하게 만들려면 Idempotency Key 패턴을 씁니다. 클라이언트가 논리적으로 동일한 요청에 대해 UUID 같은 고유 키를 생성해 Idempotency-Key 헤더로 보내고, 같은 키로는 재시도해도 같은 키를 재사용합니다. 서버는 이 키로 처음 처리할 때 결과를 저장해 두고, 같은 키로 재요청이 오면 실제 처리(예: 결제 실행)를 건너뛰고 저장된 결과를 그대로 돌려줍니다.


이때 중요한 것은 "처리 실행"과 "키 저장"이 원자적으로 묶여야 한다는 점입니다. 둘이 따로 커밋되면 그 사이에 서버가 죽었을 때 다시 중복 처리가 발생할 수 있습니다. 그래서 저장소 트랜잭션 설계까지 함께 고려해야 완전한 답이 됩니다.

이 답변이 좋은 이유:

  • "응답 유실 ≠ 처리 실패"라는 분산 시스템의 근본 전제를 먼저 짚음
  • POST가 왜 위험한지 구체적 시나리오(중복 주문)로 설명
  • Idempotency Key라는 실제 업계 표준 패턴을 구체적으로 제시
  • 패턴의 구현 디테일(원자적 저장)까지 파고들어 트레이드오프를 짚음

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"POST는 생성 요청이라 재시도하면 안 되고, GET은 조회라 재시도해도 됩니다."

이 답변이 약한 이유

  • "안 된다"고만 하고 그 이유(왜 위험한지, 구체적으로 어떤 장애가 나는지)를 설명하지 않았습니다.
  • "안 되면 어떻게 해야 안전하게 만들 수 있는지"에 대한 후속 답이 없습니다. 질문은 "안전하게 만들려면?"까지 묻고 있는데 이 부분을 답하지 않았습니다.
  • PUT/DELETE도 상태를 바꾸지만 idempotent해서 재시도가 안전하다는 것을 언급하지 않아, "상태를 바꾸는 메서드 = 위험"이라는 부정확한 이분법으로 들립니다.
WEAK ANSWER — 또 다른 패턴

"클라이언트에서 재시도 로직을 아예 안 쓰면 됩니다."

이 답변이 약한 이유

  • 문제 자체를 회피했습니다. 실무에서는 네트워크가 불안정한 환경(모바일, 클라우드 리전 간 통신)에서 재시도 없이는 가용성을 유지하기 어렵습니다.
  • "재시도를 안 쓴다"는 트레이드오프(가용성 저하)를 인지하지 못한 채 안전 쪽으로만 치우친 결론입니다.
  • 이 문제를 실제로 해결하는 업계 표준 패턴(Idempotency Key)을 모르는 것으로 보입니다.
STRONG ANSWER 요소 체크리스트
  • "응답 유실 ≠ 서버가 처리 안 함"이라는 전제를 먼저 짚는다
  • 메서드별 idempotency를 스펙 기준으로 정확히 구분한다
  • PATCH처럼 설계에 따라 달라지는 케이스까지 언급한다
  • POST를 안전하게 재시도하는 구체적 패턴(Idempotency Key)을 제시한다
  • 그 패턴 구현의 트레이드오프(원자성, TTL)까지 짚는다

10. Senior-level Follow-up Q&A

Q: DELETE 요청을 두 번 보내면 두 번째는 404가 나올 텐데, 그래도 DELETE를 idempotent하다고 부르는 이유가 뭔가요?

모범 답변: Idempotency는 "응답 코드가 항상 같다"는 뜻이 아니라 "서버의 최종 상태가 같다"는 뜻입니다. DELETE를 첫 번째 호출하면 리소스가 삭제되어 200/204가 오고, 두 번째 호출하면 이미 없으므로 404가 올 수 있습니다. 응답 코드는 다르지만 "그 리소스는 존재하지 않는다"는 서버 상태 자체는 몇 번을 호출하든 동일합니다. 그래서 응답이 달라도 idempotent하다고 부릅니다. 이 구분(상태 vs 응답)을 못 하는 것이 이 주제에서 가장 흔한 오해입니다.

Q: 로드밸런서나 HTTP 클라이언트 라이브러리가 자동 재시도를 지원한다면, 어떤 메서드에 대해 그 기능을 켜도 안전한가요?

모범 답변: 기본적으로 GET, HEAD, PUT, DELETE, OPTIONS처럼 스펙상 idempotent가 보장된 메서드에 대해서만 자동 재시도를 켜는 것이 안전합니다. POST와 PATCH(상대적 변경으로 구현된 경우)는 자동 재시도 대상에서 기본적으로 제외해야 하며, 꼭 필요하다면 Idempotency Key가 있는 요청에 한해서만 재시도를 허용하도록 구성해야 합니다. 실제로 많은 HTTP 클라이언트 라이브러리(예: 일부 로드밸런서의 retry policy)가 기본값으로 idempotent 메서드에만 재시도를 적용하는 것도 이 이유 때문입니다.

Q: PATCH API를 설계할 때 idempotent하게 만들려면 구체적으로 어떻게 설계해야 하나요?

모범 답변: 요청 바디를 "상대적 변경"이 아니라 "절대값 지정"으로 설계합니다. 예를 들어 재고를 줄이는 API를 {"stock_delta": -1}로 설계하면 재시도할 때마다 재고가 계속 줄어들어 idempotent하지 않습니다. 대신 {"stock": 41}처럼 최종값을 지정하게 하거나, 버전/조건부 요청(If-Match 등 ETag 기반 조건부 갱신, Day 9에서 심화)과 결합해 "현재 버전이 N일 때만 이 값으로 바꿔라"는 형태로 설계하면 재시도해도 같은 결과가 보장됩니다.

Q: GET 요청에 부작용이 있는 API(예: 호출할 때마다 조회수를 증가시키는 API)를 봤습니다. 이건 스펙 위반인가요?

모범 답변: 스펙 관점에서는 안전(safe) 원칙 위반입니다. GET은 서버 상태를 변경하지 않는다는 것이 전제이기 때문에, 브라우저 프리페치, 크롤러, 프록시의 재시도 로직이 부작용 없이 GET을 여러 번 호출할 수 있다고 가정합니다. 조회수 증가 같은 부작용을 GET에 넣으면 봇 트래픽이나 프리페치만으로 조회수가 왜곡되는 실제 장애로 이어질 수 있습니다. 조회수 증가처럼 의도적인 부작용이 필요하다면 별도의 POST 엔드포인트로 분리하는 것이 원칙에 맞는 설계입니다.

Q: Idempotency Key를 헤더로 받는 API를 설계할 때, 키 저장소로 무엇을 고려하고 어떤 실패 모드를 방어해야 하나요?

모범 답변: 키 저장소는 (키, 처리 상태, 응답 결과)를 원자적으로 기록할 수 있어야 하며, 보통 Redis(TTL 지원)나 RDB의 유니크 제약(키 컬럼 unique)을 사용합니다. 방어해야 할 실패 모드는 두 가지입니다: 첫째, "처리 실행"과 "키 저장"이 분리되어 그 사이에 프로세스가 죽으면 재시도 시 중복 처리가 일어날 수 있으므로, 키를 먼저 "처리 중" 상태로 기록(유니크 제약으로 동시 요청 차단)한 뒤 실제 처리를 하고 마지막에 결과를 갱신하는 순서가 필요합니다. 둘째, 동시에 같은 키로 두 요청이 들어오는 race condition — 이 경우 유니크 제약이나 분산 락으로 하나만 실제 처리하고 나머지는 대기 후 저장된 결과를 반환하도록 만들어야 합니다.

11. 흔한 오해

오해 1

"Idempotent = 부작용이 전혀 없다"

Idempotent는 "여러 번 실행해도 서버의 최종 상태가 같다"는 뜻이지 "부작용이 없다"는 뜻이 아닙니다. PUT과 DELETE는 분명히 서버 상태를 바꾸는(부작용이 있는) 메서드지만, 그 결과가 몇 번을 반복해도 동일하므로 idempotent합니다.

오해 2

"Safe와 Idempotent는 같은 개념이다"

Safe(상태를 안 바꿈)한 메서드는 모두 idempotent하지만, 역은 성립하지 않습니다. PUT/DELETE는 상태를 바꾸므로 safe하지 않지만 idempotent합니다. 두 개념을 같다고 생각하면 "PUT은 조회처럼 안전한 것"이라는 잘못된 결론에 이릅니다.

오해 3

"응답이 다르면 idempotent하지 않은 것이다"

DELETE의 두 번째 호출이 404를 반환하는 것처럼, 응답 코드나 응답 바디는 호출마다 달라질 수 있습니다. Idempotency의 기준은 응답이 아니라 서버에 남는 최종 상태입니다.

오해 4

"POST는 항상 위험하니 절대 자동 재시도하면 안 된다"

POST 자체는 스펙상 idempotent를 보장하지 않지만, Idempotency Key 같은 애플리케이션 레벨 장치를 붙이면 안전하게 재시도할 수 있습니다. "POST는 무조건 위험하다"가 아니라 "기본적으로는 위험하고, 안전하게 만들려면 별도 설계가 필요하다"가 정확한 진술입니다.

오해 5

"타임아웃이 났다는 것은 요청이 실패했다는 뜻이다"

타임아웃은 "정해진 시간 안에 응답을 받지 못했다"는 뜻일 뿐, 서버가 요청을 처리했는지 여부와는 무관합니다. 이 불확실성을 인지하지 못하면 idempotency가 왜 중요한 문제인지 자체를 이해하지 못하게 됩니다.
