1. 핵심 면접 질문

Primary Question

"프론트엔드에서 API를 호출하는데 브라우저 콘솔에 CORS 에러가 뜹니다. 어떤 순서로 원인을 좁혀 나가시겠어요?"

Variant 1

"Preflight(OPTIONS) 요청은 왜 필요하고, 어떤 조건의 요청에서 발생하나요?"

Variant 2

"CORS 설정에 Access-Control-Allow-Origin: *를 쓰면 안 되는 경우는 언제이고, 왜 그런가요?"

2. 왜 면접에서 묻는가

이 질문은 "Access-Control-Allow-Origin 헤더를 추가하면 된다"는 처방을 아는가가 아니라, CORS가 왜 존재하고 정확히 무엇을 막고 있으며, 그것이 서버가 아닌 브라우저에서 강제되는 규칙이라는 사실을 이해하는가를 테스트합니다. 실무에서 "CORS 에러"라는 한 줄의 콘솔 메시지 뒤에는 origin 불일치, preflight 실패, credentials 설정 충돌, 캐시된 CORS 헤더 오염 등 서로 다른 원인이 숨어 있고, 이를 구분하지 못하면 Access-Control-Allow-Origin: *을 무작정 추가해 인증이 필요한 API를 열어버리는 보안 사고로 이어질 수 있습니다.

면접관이 이 질문으로 평가하는 것:

  • Same-Origin Policy와 CORS의 관계에 대한 이해 — 브라우저가 기본적으로 크로스 오리진 응답 읽기를 차단하고, CORS는 서버가 "이 오리진에게는 읽기를 허용한다"고 명시적으로 완화해주는 메커니즘이라는 것을 아는가.
  • CORS가 서버가 아니라 브라우저가 강제하는 규칙이라는 것 — 서버는 여전히 응답을 만들어 보내지만, 브라우저가 JavaScript에게 그 응답을 넘겨주지 않을 뿐이라는 사실을 아는가. 이것이 curl이나 서버 간 통신에서는 "CORS 에러"가 존재하지 않는 이유입니다.
  • Simple request와 Preflighted request를 구분하는 기준 — 어떤 조건에서 브라우저가 실제 요청 전에 OPTIONS 요청(preflight)을 먼저 보내는지, 그리고 이것이 지연 시간(추가 왕복)에 미치는 영향을 아는가.
  • Credentials(쿠키/인증 헤더)가 걸리면 규칙이 더 엄격해진다는 것 — Access-Control-Allow-Origin: *과 credentials가 함께 쓰일 수 없는 이유를 보안 관점에서 설명할 수 있는가.
  • 실제 장애를 헤더 단위로 재현·격리하는 능력 — "CORS 에러"라는 하나의 증상을 여러 원인 후보로 나누어 순서대로 배제해나가는 절차적 사고를 보여주는가.

10년차 이하에서 자주 나오는 답변 패턴: "서버에 Access-Control-Allow-Origin 헤더를 추가하면 됩니다." 방향은 맞지만 왜 이 헤더가 필요한지, 어떤 값을 넣어야 하는지(구체적 오리진 vs *), credentials가 걸린 요청에서는 왜 *가 통하지 않는지를 설명하지 못합니다. 시니어는 "이 요청이 인증 정보를 포함하는가"부터 확인하고, 그에 따라 허용 가능한 설정의 범위가 달라진다는 것을 판단 기준으로 삼습니다.

3. 전제 개념 — 쉬운 한국어로

Same-Origin Policy란 — 브라우저의 기본 원칙 "남의 집 우편함은 못 연다"

브라우저는 기본적으로 a.com에서 로드된 JavaScript가 b.com의 응답 내용을 읽지 못하게 막습니다. 이것이 Same-Origin Policy입니다. Origin은 스킴(http/https) + 호스트 + 포트 세 가지가 모두 같아야 동일하다고 인정되며, 하나라도 다르면 크로스 오리진입니다. 아파트 비유로 설명하면, 우편함(응답)은 배달(요청 전송) 자체는 막지 않습니다 — 편지는 실제로 배달되어 우편함에 들어갑니다. 다만 다른 집 사람이 남의 우편함을 열어 내용물을 꺼내 읽는 것을 막는 규칙입니다. 이 비유가 중요한 이유는, CORS 에러가 나도 서버는 이미 요청을 처리하고 응답을 만들어 보냈을 수 있다는 점을 이해하는 데 핵심이기 때문입니다 — 문제는 "요청이 서버에 도달했는가"가 아니라 "브라우저가 그 응답을 JavaScript에게 넘겨주는가"입니다.

CORS란 — "이 오리진에게는 우편함을 열어봐도 된다"는 서버의 공식 허가

CORS(Cross-Origin Resource Sharing)는 서버가 응답 헤더에 Access-Control-Allow-Origin: https://a.com처럼 "이 오리진의 요청이라면 브라우저야, 응답을 JavaScript에게 넘겨줘도 된다"는 허가를 명시하는 메커니즘입니다. 이 헤더가 없거나 요청을 보낸 오리진과 일치하지 않으면, 브라우저는 응답이 이미 도착했더라도 JavaScript 코드(예: fetch().then())에게 그 내용을 넘기지 않고 에러를 던집니다. 핵심은 이 판단과 차단이 전적으로 브라우저에서 일어난다는 것입니다. 같은 요청을 Postman이나 curl, 또는 서버 간 통신으로 보내면 CORS 에러는 애초에 발생하지 않습니다 — CORS는 "브라우저에서 실행되는 JavaScript"라는 특정 클라이언트 환경을 겨냥한 보호 장치이기 때문입니다.

왜 이런 제약이 필요한가 — 쿠키 자동 전송과의 조합이 문제

Day 6에서 다뤘듯, 브라우저는 사용자가 로그인한 사이트로 가는 요청에 쿠키를 자동으로 실어 보냅니다. 만약 크로스 오리진 응답 읽기에 아무 제약이 없다면, 사용자가 bank.com에 로그인한 상태에서 evil.com을 방문했을 때, evil.com의 JavaScript가 fetch('https://bank.com/api/account')를 호출해 사용자의 로그인 쿠키가 실린 요청을 보내고, 그 응답(계좌 정보)까지 그대로 읽어갈 수 있게 됩니다. Same-Origin Policy와 CORS는 정확히 이 마지막 단계 — "응답을 읽어가는 것"을 막는 방어선입니다.

SameSite(Day 6)와 CORS(Day 7)는 서로 다른 질문에 답한다

두 메커니즘을 혼동하는 것이 실무에서 가장 흔한 실수입니다. SameSite는 "이 요청에 쿠키를 실어 보낼 것인가"를 결정하고, CORS는 "이 응답을 JavaScript가 읽도록 허용할 것인가"를 결정합니다. 요청 단계의 문제와 응답 읽기 단계의 문제이며, 크로스 오리진 API를 쿠키 인증으로 호출하려면 두 메커니즘 모두를 통과해야 합니다 — SameSite=None; Secure로 쿠키가 실려 가야 하고, 서버는 CORS 헤더로 그 오리진의 응답 읽기를 허용해야 합니다.

핵심 요약: CORS는 서버를 보호하는 것이 아니라 사용자의 브라우저(그리고 그 브라우저에 저장된 인증 정보)를 보호하는 메커니즘입니다. 서버는 요청을 이미 받아 처리했고, CORS 헤더가 없다는 것은 "이 오리진에서 온 JavaScript에게는 결과를 보여주지 않겠다"는 브라우저의 뒤늦은 차단일 뿐입니다. 이 전제를 이해해야 뒤에 나올 모든 원인 분석(요청은 200으로 도착했는데 왜 콘솔에는 에러가 뜨는가)이 논리적으로 연결됩니다.

4. CORS 헤더 판단표

헤더 어느 쪽에서 보내는가 역할 실무 주의점
Origin 브라우저 → 서버 (요청) 이 요청을 시작한 오리진을 서버에 알림. 브라우저가 자동으로 붙이며 JavaScript에서 조작 불가 서버는 이 값을 보고 화이트리스트와 대조해 Access-Control-Allow-Origin을 동적으로 결정하는 경우가 많음
Access-Control-Allow-Origin 서버 → 브라우저 (응답) 이 오리진(또는 *)의 요청이라면 응답 읽기를 허용한다고 선언 credentials가 걸린 요청에는 * 사용 불가 — 반드시 구체적 오리진을 반환해야 함(아래 5번 섹션)
Access-Control-Allow-Credentials 서버 → 브라우저 (응답) 쿠키/인증 헤더가 실린 요청의 응답을 JavaScript가 읽도록 허용 값은 반드시 true 문자열이어야 하며, 이 헤더가 없으면 클라이언트가 credentials: 'include'를 보내도 응답을 읽을 수 없음
Access-Control-Request-Method 브라우저 → 서버 (preflight) 실제로 보낼 요청의 메서드(예: PUT)를 preflight에서 미리 알림 서버는 preflight 응답에서 이 메서드를 Access-Control-Allow-Methods로 허용해야 함
Access-Control-Request-Headers 브라우저 → 서버 (preflight) 실제 요청에 실릴 커스텀 헤더 목록(예: Authorization, Content-Type)을 미리 알림 서버는 이 목록을 Access-Control-Allow-Headers로 허용해야 하며, 하나라도 누락되면 preflight가 실패함
Access-Control-Allow-Methods 서버 → 브라우저 (preflight 응답) 이 엔드포인트가 허용하는 메서드 목록을 선언 실제 요청에서 쓸 메서드가 포함되어야 하며, 와일드카드는 credentials 모드에서 지원되지 않음
Access-Control-Allow-Headers 서버 → 브라우저 (preflight 응답) 이 엔드포인트가 허용하는 요청 헤더 목록을 선언 커스텀 인증 헤더(예: X-Api-Key)를 쓴다면 반드시 이 목록에 명시해야 함
Access-Control-Max-Age 서버 → 브라우저 (preflight 응답) 이 preflight 결과를 브라우저가 캐시할 시간(초)을 지정 값을 늘리면 매 요청마다 preflight 왕복이 생기는 것을 줄여 지연 시간을 절감하지만, 브라우저별 상한(Chromium 계열은 최대 2시간)이 있어 무한정 늘릴 수 없음
Vary: Origin 서버 → 브라우저/캐시 (응답) 응답이 Origin 값에 따라 달라진다는 것을 캐시 계층에 알림 화이트리스트 기반으로 Access-Control-Allow-Origin을 동적으로 반환하면서 이 헤더를 빠뜨리면, Day 5에서 다룬 것과 동일한 캐시 오염이 발생 — 한 오리진용으로 캐시된 응답이 다른 오리진에게 그대로 재사용됨

5. Preflight의 동작 원리 — 언제 발생하고 언제 생략되는가

모든 크로스 오리진 요청이 OPTIONS preflight를 거치는 것은 아닙니다. 브라우저는 요청이 "Simple Request"의 조건을 모두 만족하면 preflight 없이 바로 실제 요청을 보내고, 하나라도 벗어나면 먼저 OPTIONS로 서버의 허가를 확인합니다.

조건 Simple Request (preflight 생략) Preflighted Request (OPTIONS 선행)
메서드 GET, HEAD, POST PUT, PATCH, DELETE 등 그 외 모든 메서드
요청 헤더 Accept, Accept-Language, Content-Language, Content-Type(제한된 값만) 등 "CORS-safelisted" 헤더만 사용 Authorization, X-Requested-With, 커스텀 헤더 등 safelist 밖의 헤더가 하나라도 포함
Content-Type 값 application/x-www-form-urlencoded, multipart/form-data, text/plain 중 하나 application/json 등 그 외 모든 값 — REST API가 JSON을 쓰는 이상 사실상 대부분 preflight 대상

실무에서 사실상 대부분의 API 호출에 preflight가 붙는 이유

JSON 기반 REST API는 Content-Type: application/json을 쓰고, 인증에는 Authorization: Bearer <token> 헤더를 쓰는 경우가 많습니다. 이 둘 중 하나만 있어도 Simple Request 조건을 벗어나므로, 현대적인 JSON API를 크로스 오리진으로 호출하는 GET/POST 요청조차 실제로는 대부분 preflight를 거칩니다. "GET은 원래 preflight가 없다"는 말은 조건부로만 맞으며, 커스텀 헤더가 붙는 순간 깨집니다.

성능 관점 — Preflight의 비용: Preflight는 실제 요청 전에 발생하는 추가 왕복(round trip)입니다. 지연 시간이 긴 네트워크(모바일, 지역 간 API 호출)에서는 이 왕복 하나가 체감 응답 시간에 그대로 더해집니다. Access-Control-Max-Age로 브라우저가 preflight 결과를 캐시하게 만들면, 같은 오리진·메서드·헤더 조합에 대해서는 캐시 유효 기간 동안 preflight를 생략할 수 있어 이 비용을 줄일 수 있습니다. 다만 이는 "요청 하나"가 아니라 "브라우저 세션 단위의 preflight 캐시"이므로, 사용자가 새 탭을 열거나 브라우저를 재시작하면 다시 발생합니다.

6. Credentials와 쿠키 — CORS의 특수 규칙

기본적으로 fetch나 XMLHttpRequest는 크로스 오리진 요청에 쿠키를 실어 보내지 않습니다. 클라이언트가 명시적으로 credentials: 'include'(fetch) 또는 withCredentials = true(XHR)를 설정해야만 쿠키가 실립니다. 이 순간부터 CORS 규칙이 눈에 띄게 엄격해집니다.

왜 Access-Control-Allow-Origin: *와 credentials가 함께 쓰일 수 없는가

*는 "어떤 오리진이든 이 응답을 읽어도 된다"는 뜻입니다. 만약 여기에 쿠키(로그인 정보)까지 실려서 응답을 읽을 수 있다면, 인터넷의 모든 사이트가 사용자 대신 로그인 쿠키를 실은 요청을 보내고 그 응답(개인정보, 계좌 정보 등)을 읽어갈 수 있는 최악의 시나리오가 됩니다. 브라우저 스펙은 이를 원천 차단하기 위해, credentials가 걸린 요청에서는 Access-Control-Allow-Origin에 반드시 요청을 보낸 구체적인 오리진 값이 와야 하고(* 불가), Access-Control-Allow-Credentials: true도 함께 있어야만 응답 읽기를 허용합니다. 이 두 조건 중 하나라도 빠지면 브라우저는 응답을 조용히 차단합니다.

동적 Allow-Origin 반환 패턴과 그 위험

여러 오리진(예: 여러 서브도메인, 여러 파트너사 도메인)에서의 접근을 허용해야 하면, 서버는 요청의 Origin 헤더 값을 읽어 화이트리스트에 있는지 검사한 뒤, 있으면 그 값을 그대로 Access-Control-Allow-Origin에 반영하는 패턴을 씁니다. 이 패턴 자체는 정상이지만, 화이트리스트 검사 없이 Origin 헤더 값을 무조건 반사(reflect)하면서 Access-Control-Allow-Credentials: true까지 함께 켜두면, 사실상 모든 오리진에게 인증된 API를 열어주는 것과 같은 심각한 보안 취약점이 됩니다. 이 경우 반드시 Vary: Origin을 함께 반환해 캐시가 오리진별 응답을 뒤섞지 않도록 해야 합니다.

7. 실전 장애 사례 — "브라우저 콘솔에 CORS 에러가 뜹니다"

원인 후보와 진단 절차

  • 1
    브라우저 개발자 도구의 Network 탭에서 실제 요청/응답이 발생했는지 확인한다. Status가 아예 안 찍히고 "(failed) net::ERR_..."로 보인다면 CORS가 아니라 네트워크 자체 실패(DNS, 연결 거부, Mixed Content — HTTPS 페이지에서 HTTP API 호출)일 가능성이 큽니다. CORS 에러는 요청 자체는 완료되고(Status 코드가 보임) 콘솔에만 에러가 뜨는 경우가 대부분입니다.
  • 2
    OPTIONS(preflight) 요청이 별도로 있는지, 있다면 그 응답 상태 코드를 확인한다. OPTIONS가 404나 405로 실패한다면, 서버 라우팅에 OPTIONS 메서드 핸들러가 아예 없는 것 — 애플리케이션 코드나 프레임워크의 CORS 미들웨어 설정 누락이 원인입니다.
  • 3
    실제 요청(또는 preflight 응답)에 Access-Control-Allow-Origin 헤더 자체가 있는지 확인한다. 없다면 서버가 CORS를 아예 처리하지 않는 것입니다. 있다면 그 값이 요청을 보낸 오리진(스킴+호스트+포트)과 정확히 일치하는지 문자 단위로 비교합니다 — http와 https, www.example.com과 example.com, 포트 유무 차이도 모두 다른 오리진으로 취급됩니다.
  • 4
    요청이 credentials(쿠키/Authorization)를 포함하는지 확인하고, 포함한다면 Access-Control-Allow-Origin이 *인지 확인한다. *이면서 credentials가 걸려 있으면 그 자체가 원인입니다 — 구체적 오리진 값과 Access-Control-Allow-Credentials: true로 바꿔야 합니다.
  • 5
    커스텀 헤더(Authorization, X-Api-Key 등)를 쓴다면 preflight 응답의 Access-Control-Allow-Headers에 그 헤더가 명시되어 있는지 확인한다. 하나라도 누락되면 preflight 자체는 200으로 응답해도 브라우저가 실제 요청을 차단합니다.
  • 6
    CDN/리버스 프록시/API 게이트웨이가 중간에서 CORS 헤더를 제거하거나 캐싱하고 있는지 확인한다. 애플리케이션 서버는 올바른 CORS 헤더를 보냈는데, 앞단의 캐시가 오리진 A의 요청에 대한 응답을 캐시해뒀다가 Vary: Origin 없이 오리진 B에게 그대로 재사용하면, B는 A용 Access-Control-Allow-Origin 값을 받아 여전히 차단당합니다(Day 5의 캐시 오염과 동일한 근본 원인).
  • 7
    로컬 개발 환경과 프로덕션의 오리진 차이를 점검한다. localhost:3000에서는 되던 호출이 배포 후 다른 서브도메인에서 안 되는 경우, CORS 화이트리스트에 프로덕션 오리진이 등록되지 않은 것이 흔한 원인입니다.

8. 답변 사고 순서

면접에서 CORS 관련 질문을 받으면 이 순서로 사고합니다.

  1. 먼저 CORS가 서버가 아니라 브라우저가 강제하는 규칙이라는 전제를 짚습니다. 이 전제가 있어야 "요청은 성공했는데 왜 콘솔에는 에러가 뜨는가"라는 현상이 논리적으로 설명됩니다.
  2. Simple Request인지 Preflighted Request인지 먼저 구분합니다. 메서드, 헤더, Content-Type 세 가지 조건으로 판단하며, 이는 원인 분석의 첫 번째 분기점입니다.
  3. 요청이 credentials를 포함하는지 확인합니다. 이 여부가 Access-Control-Allow-Origin에 *를 쓸 수 있는지 없는지를 결정하는 핵심 분기입니다.
  4. Network 탭에서 실제 응답 헤더를 직접 눈으로 확인하는 절차를 설명합니다. 추측이 아니라 헤더 값을 직접 대조하는 것이 시니어의 진단 방식입니다.
  5. 트래픽 경로 위의 다른 계층(CDN, 리버스 프록시)이 CORS 헤더를 제거·캐싱할 가능성을 함께 짚습니다. Day 1~5의 계층별 사고 프레임워크와 연결합니다.
  6. 보안 함의(와일드카드 + credentials 조합의 위험성, Origin 무조건 반사의 위험성)를 마지막에 언급해 단순 트러블슈팅을 넘어선 판단력을 보여줍니다.

9. 핵심 답변 예시

면접관: "프론트엔드에서 API를 호출하는데 브라우저 콘솔에 CORS 에러가 뜹니다. 어떤 순서로 원인을 좁혀 나가시겠어요?"

STRONG ANSWER

먼저 짚고 싶은 전제는, CORS 에러는 서버가 요청을 거부했다는 뜻이 아니라 브라우저가 이미 도착한 응답을 JavaScript에게 넘겨주지 않았다는 뜻이라는 점입니다. 그래서 저는 가장 먼저 Network 탭에서 그 요청의 Status 코드를 확인합니다. 200이 찍혀 있다면 서버까지는 정상적으로 다녀온 것이고, 문제는 응답 헤더 쪽에 있다고 판단합니다.


다음으로 요청 방식이 Simple Request인지 확인합니다. Content-Type: application/json이나 Authorization 헤더를 쓰고 있다면 preflight가 발생했을 것이므로, OPTIONS 요청이 따로 있는지, 그 응답이 200인지, 그리고 Access-Control-Allow-Methods와 Access-Control-Allow-Headers에 제가 쓰는 메서드와 헤더가 실제로 포함되어 있는지 하나씩 대조합니다.


그 다음 확인하는 건 이 요청이 쿠키나 인증 헤더 같은 credentials를 포함하는가입니다. 포함한다면 Access-Control-Allow-Origin이 *로 되어 있는지부터 의심합니다. credentials가 걸린 요청에서는 브라우저가 *를 아예 허용하지 않기 때문에, 이 조합 자체가 가장 흔한 원인 중 하나입니다. 이 경우 서버가 Origin 헤더 값을 화이트리스트와 대조해 구체적인 값으로 반환하고, Access-Control-Allow-Credentials: true도 함께 보내도록 고쳐야 합니다.


여기까지 확인해도 원인이 안 보이면, 이 요청이 CDN이나 리버스 프록시를 거치는지 확인합니다. 애플리케이션은 올바른 CORS 헤더를 보냈는데 앞단 캐시가 다른 오리진용으로 캐시된 응답을 재사용하면서 CORS 헤더가 어긋나는 경우가 실무에서 꽤 흔했기 때문에, 이 계층에서 Vary: Origin이 빠져 있지는 않은지도 함께 봅니다.

이 답변이 좋은 이유:

  • CORS가 브라우저 쪽 차단이라는 전제를 먼저 짚어 "요청은 도착했는데 왜 에러인가"를 논리적으로 설명함
  • Simple vs Preflighted 구분을 진단 절차의 분기점으로 명시적으로 사용함
  • credentials + 와일드카드 조합이라는 가장 흔한 원인을 구체적 메커니즘과 함께 제시함
  • 단일 계층이 아니라 CDN/프록시까지 포함한 전체 트래픽 경로 관점에서 원인을 탐색함
  • 순서 있는 절차(요청 확인 → 헤더 대조 → credentials 확인 → 상위 계층 확인)로 답해 실전 디버깅 경험을 보여줌

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"서버에 Access-Control-Allow-Origin: *를 추가하면 CORS 에러가 해결됩니다."

이 답변이 약한 이유

  • credentials가 걸린 요청에서는 *가 애초에 작동하지 않는다는 것을 모르고 있어, 실제로는 문제를 해결하지 못하는 경우가 많습니다.
  • 모든 오리진에게 응답 읽기를 허용하는 것의 보안적 의미(왜 이것이 종종 잘못된 처방인지)를 전혀 고려하지 않습니다.
  • "에러가 뜨면 이 헤더를 붙인다"는 암기된 해결책일 뿐, 원인을 진단하는 절차가 없습니다.
WEAK ANSWER — 또 다른 패턴

"CORS는 프론트엔드 문제니까 프론트엔드 팀에서 처리해야 합니다."

이 답변이 약한 이유

  • CORS 헤더는 서버(백엔드)가 응답에 실어 보내는 것이므로, 실제 수정은 백엔드 쪽 책임인 경우가 대부분입니다 — 책임 소재를 잘못 이해하고 있습니다.
  • 문제 해결에 대한 사고 없이 책임을 회피하는 답변으로 읽혀, 시니어에게 기대되는 오너십을 보여주지 못합니다.
STRONG ANSWER 요소 체크리스트
  • CORS가 브라우저 쪽 차단이며 서버 응답 자체는 이미 도착했을 수 있다는 전제를 제시한다
  • Simple Request와 Preflighted Request를 조건 기준으로 구분한다
  • credentials 포함 여부가 * 허용 가능성을 결정하는 핵심 분기라는 것을 안다
  • Network 탭에서 실제 헤더 값을 직접 대조하는 구체적 절차를 제시한다
  • CDN/프록시 등 상위 계층에서의 CORS 헤더 캐싱/제거 가능성까지 고려한다
  • 와일드카드 + credentials, Origin 무조건 반사의 보안 위험을 인지하고 있다

11. Senior-level Follow-up Q&A

Q: 서버 로그를 보면 요청이 200으로 정상 처리됐는데, 왜 프론트엔드에서는 요청이 실패한 것처럼 보이나요?

모범 답변: CORS 차단은 브라우저에서 일어나기 때문에 서버 입장에서는 요청을 정상적으로 받아 처리하고 200을 응답한 것이 맞습니다. 다만 브라우저가 그 응답을 JavaScript 실행 컨텍스트에 넘겨주지 않고 콘솔에 에러를 던지기 때문에, fetch의 Promise는 reject되고 애플리케이션 코드 입장에서는 "요청이 실패했다"고 보입니다. 이 비대칭 — 서버는 성공, 클라이언트는 실패 — 이 CORS 디버깅에서 가장 헷갈리는 지점이라, 서버 로그만 보고 "우리 쪽은 문제없다"고 결론 내리면 안 되고 반드시 브라우저 Network 탭의 응답 헤더까지 함께 확인해야 합니다.

Q: Preflight 요청 자체도 서버 부하나 지연에 영향을 주나요? 어떻게 줄일 수 있나요?

모범 답변: Preflight는 실제 요청 전에 발생하는 별도의 왕복이므로, 매번 발생하면 API 하나당 지연 시간이 사실상 두 배가 됩니다. 이를 줄이는 방법은 Access-Control-Max-Age를 적절히 설정해 브라우저가 같은 오리진·메서드·헤더 조합에 대한 preflight 결과를 세션 동안 캐시하게 하는 것입니다. 다만 이 값을 브라우저 상한(예: Chromium 계열 2시간)보다 크게 설정해도 상한에서 잘리므로 무한정 늘릴 수는 없고, 또한 CORS 설정을 바꿨을 때 캐시된 사용자 브라우저에는 변경 사항이 최대 그 시간만큼 늦게 반영될 수 있다는 것도 함께 고려해야 합니다. 근본적으로는 커스텀 헤더 개수를 줄이거나 Simple Request 조건에 맞추는 설계로 preflight 자체를 줄이는 것도 방법이지만, 인증 헤더가 필요한 API에서는 현실적으로 한계가 있습니다.

Q: CDN 뒤에 있는 API가 여러 오리진에서 호출됩니다. Access-Control-Allow-Origin을 어떻게 설정해야 캐시 오염 없이 안전하게 처리할 수 있나요?

모범 답변: 서버는 요청의 Origin 헤더를 화이트리스트와 대조해, 허용된 오리진이면 그 값을 그대로 Access-Control-Allow-Origin에 반환하는 동적 반사 패턴을 씁니다. 이때 반드시 Vary: Origin을 함께 응답해야 하는데, 이렇게 해야 CDN이나 공유 캐시가 "이 응답은 Origin 헤더 값에 따라 내용이 달라진다"는 것을 인지하고 오리진별로 캐시를 분리해서 저장합니다. Vary: Origin이 없으면 오리진 A의 요청에 대해 캐시된 응답이 오리진 B에게 그대로 재사용되어, B는 A용 Access-Control-Allow-Origin 값을 받아 여전히 차단당하거나, 반대로 화이트리스트에 없는 오리진이 캐시된 허용 헤더를 우연히 받아가는 상황까지 생길 수 있습니다. 이는 Day 5에서 다룬 Vary 캐시 오염과 동일한 근본 원리입니다.

Q: 화이트리스트 검사 없이 요청의 Origin 헤더 값을 그대로 반사하면서 Allow-Credentials: true를 켜두면 왜 위험한가요?

모범 답변: 이 조합은 사실상 "어떤 오리진이든 credentials(쿠키/인증 헤더)를 실어 보내면 응답을 읽을 수 있다"는 것과 동일합니다. 공격자가 자신의 사이트에 사용자를 유인해 로그인 상태의 사용자 브라우저로 대상 API를 호출하게 만들면, 응답(개인정보, 계좌 정보, 관리 기능 결과 등)을 그대로 읽어갈 수 있는 CSRF와 유사하지만 더 강력한 정보 유출 취약점이 됩니다. CORS는 서버가 명시적으로 허용을 선언하는 화이트리스트 모델로 설계되어 있으므로, 반사 패턴을 쓰더라도 반드시 사전에 정의된 화이트리스트와 대조하는 검증 단계가 있어야 합니다.

Q: 서버 간 통신(마이크로서비스 내부 호출)에서도 CORS 설정이 필요한가요?

모범 답변: 아닙니다. CORS는 브라우저가 강제하는 규칙이므로, 브라우저를 거치지 않는 서버 간 HTTP 호출(백엔드 서비스 간 통신, 배치 작업, curl 등)에는 애초에 적용되지 않습니다. CORS 설정은 오직 "브라우저에서 실행되는 JavaScript가 다른 오리진의 API를 호출하는" 시나리오에서만 의미가 있습니다. 이 구분을 못 하면 내부 서비스 간 통신에까지 불필요하게 CORS 헤더를 추가하거나, 반대로 프론트엔드에서 직접 호출하는 엔드포인트에 CORS 설정을 빠뜨리는 실수를 하게 됩니다.

12. 흔한 오해

오해 1

"CORS 에러가 뜨면 서버가 요청을 거부한 것이다"

서버는 대부분 요청을 정상적으로 처리하고 응답까지 보냈습니다. CORS 에러는 브라우저가 그 응답을 JavaScript에게 넘겨주지 않는 것이며, 원인 분석은 항상 "서버가 응답 헤더에 무엇을 담아 보냈는가"에서 시작해야 합니다.

오해 2

"Access-Control-Allow-Origin: *을 쓰면 항상 문제가 해결된다"

credentials(쿠키, Authorization 헤더)가 걸린 요청에서는 *가 브라우저 스펙상 아예 허용되지 않습니다. 이 경우 구체적인 오리진 값과 Access-Control-Allow-Credentials: true가 함께 필요합니다.

오해 3

"CORS를 설정하면 API가 안전해진다"

CORS는 브라우저에서 실행되는 JavaScript의 응답 읽기를 제어할 뿐, curl이나 서버 간 요청, 또는 브라우저 확장 프로그램 등 CORS를 우회할 수 있는 경로를 막지 못합니다. API 자체의 인증/인가는 CORS와 별개로 반드시 구현해야 하는 방어선입니다.

오해 4

"GET 요청은 항상 preflight 없이 바로 나간다"

메서드만이 아니라 헤더와 Content-Type도 Simple Request 조건에 포함됩니다. GET이라도 Authorization 헤더나 application/json이 아닌 커스텀 Content-Type을 쓰면 preflight가 발생합니다.

오해 5

"SameSite를 올바르게 설정했으니 CORS는 신경 쓰지 않아도 된다"

Day 6에서 다룬 SameSite는 "쿠키를 요청에 실을 것인가"를, CORS는 "응답을 JavaScript가 읽도록 허용할 것인가"를 결정하는 완전히 다른 메커니즘입니다. 크로스 오리진에서 쿠키 기반 인증 API를 호출하려면 둘 다 올바르게 설정되어야 합니다.

13. 자기 점검 Rubric

다음 항목을 말로 설명할 수 있으면 Day 7을 이해한 것입니다.

☐
Same-Origin Policy가 무엇을 막고, CORS가 어떻게 그것을 명시적으로 완화하는지 설명할 수 있다.
☐
CORS 차단이 서버가 아니라 브라우저에서 일어나며, 요청 자체는 서버에 정상 도달했을 수 있다는 것을 설명할 수 있다.
☐
Simple Request와 Preflighted Request를 메서드/헤더/Content-Type 기준으로 구분할 수 있다.
☐
Preflight가 추가 왕복을 발생시켜 지연 시간에 영향을 준다는 것과, Access-Control-Max-Age로 이를 완화하는 원리를 설명할 수 있다.
☐
credentials가 걸린 요청에서 왜 Access-Control-Allow-Origin에 *를 쓸 수 없는지 보안 관점에서 설명할 수 있다.
☐
Origin 헤더 값을 화이트리스트 없이 무조건 반사하는 것이 왜 취약점이 되는지 설명할 수 있다.
☐
동적으로 Access-Control-Allow-Origin을 반환할 때 Vary: Origin이 왜 필요한지, 없으면 어떤 캐시 오염이 생기는지 설명할 수 있다.
☐
"CORS 에러" 증상을 네트워크 실패, preflight 실패, origin 불일치, credentials 설정 오류, 상위 캐시 계층 오염 등 원인별로 진단할 수 있다.
☐
SameSite(Day 6)와 CORS(Day 7)가 각각 "쿠키 전송 여부"와 "응답 읽기 허용 여부"라는 다른 문제를 다룬다는 것을 구분할 수 있다.
☐
서버 간 통신에는 왜 CORS가 적용되지 않는지, CORS가 특정 클라이언트 환경(브라우저)을 겨냥한 방어인지 설명할 수 있다.
