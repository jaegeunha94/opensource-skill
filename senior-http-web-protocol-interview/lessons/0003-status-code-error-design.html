1. 핵심 면접 질문

Primary Question

"429와 503을 받았을 때 클라이언트는 각각 어떻게 다르게 대응해야 하나요?"

Variant 1

"400과 422는 둘 다 '요청이 잘못됐다'는 뜻인데, 왜 스펙에 두 코드가 따로 존재하나요? 실무에서는 어떻게 구분해서 쓰시나요?"

Variant 2

"클라이언트 라이브러리가 상태 코드만 보고 자동으로 재시도할지 말지 결정하게 만들려고 합니다. 어떤 코드는 재시도해도 되고 어떤 코드는 안 되나요? 그 기준은 뭔가요?"

2. 왜 면접에서 묻는가

이 질문은 "상태 코드의 숫자와 이름을 외웠는가"가 아니라 "상태 코드를 근거로 클라이언트가 무엇을 해야 하는지 자동화된 판단을 내릴 수 있는가"를 테스트합니다. "200은 성공, 404는 없음, 500은 서버 에러"라는 설명은 신입도 합니다. 시니어를 가르는 지점은 이 코드를 기계가 읽고 행동을 결정하는 신호로 다룰 수 있는지입니다 — 재시도할지, 즉시 실패 처리할지, 사용자에게 무엇을 보여줄지, 알림을 울릴지.

면접관이 이 질문으로 평가하는 것:

  • 재시도 가능성 판단력 — Day 2에서 "메서드가 idempotent한가"를 봤다면, 이번엔 "응답이 재시도해서 성공할 가능성이 있는 실패인가, 아니면 재시도해도 똑같이 실패할 실패인가"를 상태 코드로 구분할 수 있는가.
  • 4xx와 5xx의 책임 소재 구분 — 4xx는 원칙적으로 "클라이언트가 고쳐야 하는 문제"이고 5xx는 "서버가 고쳐야 하는 문제"라는 책임 경계를 실무 코드/알림 설계에 반영할 수 있는가.
  • 같은 카테고리 안의 세부 구분 — 429(Too Many Requests)와 503(Service Unavailable)은 둘 다 "지금 당장은 처리 못 함"이라는 점에서 비슷해 보이지만, 원인과 대응이 다릅니다. 이 세부 차이를 설명할 수 있는가.
  • API 설계자로서의 감각 — 표준 코드만으로는 충분하지 않을 때 에러 응답 바디를 어떻게 설계해야 클라이언트가 사람이 아니라 프로그램으로 올바르게 분기 처리할 수 있는지 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "4xx는 클라이언트 에러, 5xx는 서버 에러입니다." 방향은 맞지만 이 구분이 왜 중요한지(재시도 정책, 알림 설계, SLO 계산과 직결된다는 것)를 연결하지 못합니다. 시니어는 "5xx가 늘면 온콜을 깨워야 하고 4xx가 늘면 API 문서나 클라이언트 버그를 봐야 한다"처럼 코드를 운영 의사결정과 바로 연결합니다.

3. 전제 개념 — 쉬운 한국어로

상태 코드는 "설명"이 아니라 "신호"다

상태 코드를 사람이 읽는 메시지로만 생각하면 이 주제를 오해하기 쉽습니다. 실제로 상태 코드를 가장 많이 "읽는" 주체는 사람이 아니라 프로그램입니다 — 클라이언트 SDK, 로드밸런서, API 게이트웨이, 모니터링 시스템이 응답 바디를 파싱하지 않고도 상태 코드 숫자 하나만으로 즉각적인 행동을 결정합니다. 우편함에 비유하면, 상태 코드는 편지 내용이 아니라 봉투에 찍힌 도장입니다 — "반송", "수취인 불명", "배달 완료" 같은 도장만 보고도 우체국 시스템이 다음 행동(재배달, 폐기, 완료 처리)을 자동으로 정할 수 있는 것과 같습니다.

왜 "1xx~5xx 계열"이 재시도 판단의 출발점인가

Day 2에서 다룬 idempotency는 "이 요청을 다시 보내도 괜찮은가"를 요청(메서드) 기준으로 판단했습니다. 이번 Day 3은 그 반대편, "이 응답을 보고 다시 보낼 가치가 있는가"를 응답(상태 코드) 기준으로 판단합니다. 두 판단이 합쳐져야 완전한 재시도 정책이 됩니다: idempotent한 메서드로 재시도 가능한 상태 코드를 받았을 때만 자동 재시도가 안전합니다.

4xx vs 5xx — 책임 소재라는 축

4xx는 "클라이언트가 보낸 요청 자체에 문제가 있다"는 뜻입니다. 식당에 비유하면 손님이 메뉴에 없는 음식을 주문한 상황(400), 예약 없이 만석인 시간에 온 상황(429), 출입 권한이 없는 방에 들어가려는 상황(403)과 같습니다. 이 경우 같은 요청을 그대로 다시 보내봐야 결과는 똑같습니다 — 클라이언트가 요청 내용 자체를 바꿔야 합니다.

5xx는 "서버 쪽에 문제가 있다"는 뜻입니다. 손님의 주문은 정상인데 주방에 불이 나서(500) 또는 주방이 너무 바빠서(503) 음식을 못 내주는 상황입니다. 이 경우 손님(클라이언트)의 요청 자체는 문제가 없으므로, 서버 문제가 해소된 뒤 같은 요청을 다시 보내면 성공할 가능성이 있습니다. 이것이 5xx가 일반적으로 "재시도 후보"이고 4xx는 (429, 408 같은 예외를 빼면) "재시도 비후보"인 이유입니다.

왜 이 구분이 SLO/온콜과 직결되는가

운영 현장에서 5xx 비율은 보통 서비스 자체의 에러율(SLO의 핵심 지표)로 집계되고, 임계치를 넘으면 온콜 알림이 울립니다. 반면 4xx는 클라이언트 잘못이거나 정상적인 비즈니스 흐름(예: 재고 없음 404, 인증 만료 401)일 수 있어 보통 별도로 집계하거나 알림에서 제외합니다. 만약 4xx와 5xx를 구분하지 않고 "에러율"이라는 지표 하나로 뭉뚱그리면, 클라이언트가 잘못된 요청을 대량으로 보낸 상황(4xx 폭증, 서버는 정상)에도 온콜이 울리는 잘못된 알림 설계가 됩니다.

4. 상태 코드 계열별 재시도 판단표

코드 의미 책임 소재 재시도 판단 실무 대응
400 Bad Request 요청 구문/형식 자체가 잘못됨 클라이언트 재시도 금지 요청 바디/파라미터를 고쳐야 함. 그대로 재시도하면 영원히 같은 400
401 Unauthorized 인증 정보 없음/만료 클라이언트 토큰 갱신 후 1회 재시도 그대로 재시도 금지, 토큰 재발급 후에만 재시도
403 Forbidden 인증은 됐지만 권한 없음 클라이언트 재시도 금지 권한 자체가 없으므로 재시도해도 동일 결과
404 Not Found 리소스 없음 클라이언트(또는 정상 흐름) 재시도 금지 URI가 틀렸거나 리소스가 실제로 없는 것. 상태 변화를 기다리는 폴링이 아니라면 재시도 무의미
408 Request Timeout 서버가 클라이언트 요청을 기다리다 타임아웃 네트워크/타이밍 재시도 가능 Idempotent 메서드라면 백오프 후 재시도 권장
409 Conflict 현재 리소스 상태와 요청이 충돌 (버전 충돌 등) 클라이언트(경합 상황) 조건부 재시도 최신 상태를 다시 읽고 요청을 재구성한 뒤 재시도(그대로 재전송은 무의미)
422 Unprocessable Entity 구문은 맞지만 의미상 처리 불가 (비즈니스 검증 실패) 클라이언트 재시도 금지 입력값 자체를 수정해야 함 (예: 이미 취소된 주문을 다시 취소 요청)
429 Too Many Requests Rate limit 초과 클라이언트(트래픽량) 백오프 후 재시도 가능 Retry-After 헤더를 준수해 대기 후 재시도 (5절에서 심화)
500 Internal Server Error 서버 내부 미처리 예외 서버 제한적 재시도 가능 일시적 버그/장애일 수 있어 idempotent 요청이면 1~2회 재시도, 반복되면 알림
502 Bad Gateway 프록시가 업스트림으로부터 잘못된 응답을 받음 서버(인프라) 재시도 가능 업스트림 앱 서버 재시작/배포 중 흔함. 백오프 재시도로 대부분 해소
503 Service Unavailable 서버가 일시적으로 요청 처리 불가 (과부하, 유지보수) 서버(용량) 백오프 후 재시도 가능 Retry-After 헤더 준수, 서킷 브레이커와 결합 권장 (5절에서 심화)
504 Gateway Timeout 프록시가 업스트림 응답을 기다리다 타임아웃 서버(인프라/성능) 제한적 재시도 가능 Idempotent 요청만 재시도. 아니라면 이미 처리됐을 가능성 확인 필요 (Day 2 원칙과 결합)

일반 원칙 요약

5xx 계열은 "서버 쪽 문제이므로 시간이 지나면 해소될 수 있다"는 전제로 재시도 후보가 되고, 4xx 계열은 "요청 자체의 문제이므로 그대로 다시 보내봐야 소용없다"는 전제로 재시도 비후보가 됩니다. 예외는 408, 409, 429처럼 "일시적 상태/타이밍" 문제인 4xx입니다. 이 예외를 기억하지 못하고 "4xx는 무조건 재시도 금지"라고 답하면 429/Retry-After를 놓치는 흔한 실수로 이어집니다.

5. 429 vs 503 — 같은 "지금은 안 됨"인데 왜 다른가

429와 503은 둘 다 "지금 당장 처리할 수 없다"는 신호이고 둘 다 Retry-After 헤더를 함께 보낼 수 있어서 혼동하기 쉽습니다. 하지만 원인의 위치가 다릅니다.

기준 429 Too Many Requests 503 Service Unavailable
원인의 위치 이 클라이언트(또는 이 API 키)가 허용된 요청량을 초과함 서버 전체가 과부하 상태이거나 유지보수 중이라 모든 요청을 처리하지 못함
다른 클라이언트에게 영향 없음 — 이 클라이언트만 차단됨, 다른 클라이언트는 정상 응답 받음 있음 — 서버가 다운되었거나 과부하이므로 다른 클라이언트도 영향받을 가능성 높음
계층 보통 API 게이트웨이/애플리케이션 레벨의 rate limiter (Day 17에서 심화) 보통 앱 서버 자체 또는 로드밸런서가 헬스체크 실패로 라우팅 차단
클라이언트가 해야 할 일 자신의 요청 빈도를 낮추고 Retry-After만큼 대기 서버 상태와 무관하게 대기하고, 반복되면 서킷 브레이커로 해당 엔드포인트 호출 자체를 일시 중단

실무 판단 기준 한 줄 요약

"문제가 나(이 클라이언트)에게 있는가, 서버 전체에 있는가?" → 나에게 있으면 429(속도를 줄여라), 서버에 있으면 503(잠시 후 다시 오거나 다른 경로를 찾아라). 온콜 관점에서도 429 폭증은 "클라이언트 트래픽 패턴 이상"으로, 503 폭증은 "서버 용량/장애"로 서로 다른 사람이 조사해야 하는 신호입니다.

6. 커스텀 에러 바디 설계 — RFC 9457 Problem Details

상태 코드만으로는 "왜 실패했는지", "무엇을 고쳐야 하는지"를 클라이언트가 정확히 알기 어렵습니다. 예를 들어 422 하나만으로는 "이메일 형식이 틀렸다"인지 "비밀번호가 너무 짧다"인지 구분할 수 없습니다. 그래서 API는 상태 코드 + 구조화된 에러 바디를 함께 설계해야 합니다.

  • 1
    일관된 에러 응답 스키마를 만든다

    모든 엔드포인트가 서로 다른 에러 바디 형식을 쓰면 클라이언트가 매 엔드포인트마다 파싱 로직을 새로 짜야 합니다. 업계에서 이 문제를 해결하기 위한 표준이 RFC 9457 (구 RFC 7807) Problem Details for HTTP APIs입니다.

    HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type": "https://api.example.com/errors/invalid-email",
  "title": "이메일 형식이 올바르지 않습니다",
  "status": 422,
  "detail": "email 필드 값 'not-an-email'은 유효한 이메일 형식이 아닙니다.",
  "instance": "/users/register",
  "errors": [
    { "field": "email", "code": "invalid_format" }
  ]
}
  • 2
    "기계가 분기할 필드"와 "사람이 읽을 필드"를 분리한다

    type과 errors[].code는 클라이언트 코드가 switch문으로 분기하는 안정적 식별자여야 합니다. title, detail은 사람이 읽는 메시지이므로 문구가 바뀌어도 클라이언트 로직이 깨지면 안 됩니다. 이 둘을 분리하지 않고 message 문자열 하나로만 에러를 표현하면, 문구를 다국어화하거나 개선할 때마다 클라이언트의 문자열 매칭 로직이 함께 깨지는 결합이 생깁니다.

  • 3
    재시도 가능 여부를 응답에 명시적으로 드러낸다

    429/503처럼 재시도가 유효한 응답에는 Retry-After 헤더(초 단위 또는 날짜)를 반드시 포함시켜, 클라이언트가 "얼마나 기다려야 하는지"를 추측하지 않도록 합니다. 헤더가 없으면 클라이언트마다 제각각의 백오프 전략을 쓰게 되어 예측 가능한 부하 제어가 어려워집니다.

이 설계에서 시니어가 짚어야 할 트레이드오프

  • 세분화 수준 — 에러 코드를 너무 잘게 쪼개면 클라이언트가 관리해야 할 케이스가 폭발적으로 늘고, 너무 뭉뚱그리면 클라이언트가 세부 대응을 못 합니다. "클라이언트가 실제로 다르게 행동해야 하는 경우"만 별도 코드로 분리하는 것이 실용적입니다.
  • 에러 스펙의 하위 호환성 — errors[].code 같은 식별자는 사실상 공개 API 계약의 일부이므로, 한번 노출하면 변경이 브레이킹 체인지가 됩니다. 신규 필드 추가는 안전하지만 기존 코드 값의 의미 변경은 위험합니다.
  • 내부 정보 노출 금지 — 500 에러의 detail에 스택 트레이스나 내부 쿼리를 그대로 노출하면 보안 취약점이 됩니다(Day 19 TLS/경계 보안과도 연결). 내부 에러는 상세 정보를 로그에만 남기고 클라이언트에는 추적 가능한 요청 ID만 반환하는 것이 원칙입니다.

7. 답변 사고 순서

면접에서 "이 상태 코드를 받았을 때 어떻게 대응해야 하나요?" 류의 질문을 받으면 이 순서로 사고합니다.

  1. 먼저 4xx/5xx 계열로 책임 소재를 나눕니다. 클라이언트 문제인지 서버 문제인지가 재시도 여부의 1차 기준입니다.
  2. 예외 코드(408, 409, 429)가 있는지 확인합니다. 4xx라고 무조건 재시도 금지가 아니라, 타이밍/경합/속도 제한처럼 "일시적" 성격의 4xx는 재시도 후보임을 짚습니다.
  3. Day 2의 idempotency 판단과 결합합니다. "재시도해도 되는 상태 코드"이면서 "재시도해도 안전한 메서드"일 때만 자동 재시도를 설계합니다.
  4. Retry-After 헤더 유무를 확인합니다. 있으면 그 값을 준수하고, 없으면 지수 백오프(exponential backoff) + jitter로 스스로 대기 시간을 정합니다.
  5. 운영 관점(알림/SLO)까지 연결하면 답변이 완성됩니다. 5xx 폭증은 온콜 알림 대상, 4xx 폭증은 클라이언트/문서 점검 대상이라는 구분을 언급합니다.

8. 핵심 답변 예시

면접관: "429와 503을 받았을 때 클라이언트는 각각 어떻게 다르게 대응해야 하나요?"

STRONG ANSWER

둘 다 "지금 당장은 처리할 수 없다"는 신호라는 공통점이 있지만, 원인의 위치가 다릅니다. 429는 이 클라이언트(또는 이 API 키)가 허용된 요청 속도를 초과했다는 뜻이라 문제가 저에게 있고, 503은 서버 전체가 과부하이거나 일시적으로 서비스 불가 상태라는 뜻이라 문제가 서버에 있습니다.


429를 받으면 제 쪽에서 요청 빈도를 낮추고, 응답에 Retry-After 헤더가 있으면 그 값만큼 대기한 뒤 재시도합니다. 503도 마찬가지로 Retry-After를 준수하지만, 503은 서버 자체의 문제이므로 반복해서 503이 온다면 단순 재시도보다 서킷 브레이커를 열어서 해당 엔드포인트 호출 자체를 일정 시간 중단하는 게 낫습니다 — 계속 재시도하면 이미 과부하인 서버에 부하를 더 얹는 꼴이 되니까요.


둘 다 재시도 전에 이 요청의 메서드가 idempotent한지도 함께 확인합니다. GET이나 PUT처럼 idempotent한 요청이면 재시도가 안전하지만, POST라면 Idempotency Key가 없는 한 재시도 자체를 신중하게 판단해야 합니다.


운영 관점에서도 이 둘은 다르게 다룹니다. 429가 늘면 특정 클라이언트의 트래픽 패턴을 조사하고, 503이 늘면 서버 용량이나 배포 문제로 보고 온콜 알림을 켭니다.

이 답변이 좋은 이유:

  • 429/503의 차이를 "문제의 위치"라는 명확한 축으로 구분함
  • Retry-After 준수라는 구체적 프로토콜 메커니즘을 언급함
  • 단순 재시도를 넘어 서킷 브레이커라는 더 견고한 패턴까지 제시함
  • Day 2의 idempotency 판단과 자연스럽게 결합해 완결된 재시도 정책을 설명함
  • 운영/온콜 관점까지 연결해 실무 판단력을 보여줌

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"둘 다 4xx/5xx 에러니까 잠깐 기다렸다가 다시 요청하면 됩니다."

이 답변이 약한 이유

  • 429는 4xx, 503은 5xx라는 사실조차 구분하지 못했습니다 — 계열이 다르다는 것 자체를 놓쳤습니다.
  • "잠깐 기다렸다가"라는 표현이 구체성이 없습니다. Retry-After 헤더라는 실제 메커니즘을 언급하지 않았습니다.
  • 두 코드의 원인이 다르다는 것(클라이언트 속도 vs 서버 과부하)을 짚지 않아 대응이 왜 달라야 하는지 근거가 없습니다.
WEAK ANSWER — 또 다른 패턴

"5xx는 서버 에러니까 무조건 재시도하면 되고, 4xx는 클라이언트 에러니까 절대 재시도하면 안 됩니다."

이 답변이 약한 이유

  • "무조건"과 "절대"라는 이분법이 틀렸습니다. 429는 4xx지만 재시도 후보이고, 500은 5xx지만 무한 재시도하면 안 됩니다(반복되면 알림으로 전환해야 함).
  • idempotency와의 결합을 언급하지 않았습니다. 5xx라도 POST처럼 idempotent하지 않은 메서드라면 무작정 재시도가 위험할 수 있습니다.
  • 재시도 "횟수 제한"이나 "백오프 전략" 없이 "재시도하면 된다"고만 답해 실무에서 바로 쓸 수 있는 답이 아닙니다.
STRONG ANSWER 요소 체크리스트
  • 4xx/5xx의 책임 소재 구분과 함께 408/409/429 같은 예외를 인지한다
  • 429와 503을 "원인의 위치(클라이언트 vs 서버)"로 구분한다
  • Retry-After 헤더를 언급하고 없을 때의 대안(지수 백오프)을 안다
  • Day 2의 idempotency 판단과 결합해 완결된 재시도 정책을 설명한다
  • 단순 재시도를 넘어 서킷 브레이커 같은 더 견고한 패턴을 언급한다
  • 운영(온콜/SLO) 관점까지 연결한다

10. Senior-level Follow-up Q&A

Q: 400과 422는 둘 다 "요청이 잘못됐다"는 뜻인데, 왜 스펙에 두 코드가 따로 존재하나요?

모범 답변: 400은 요청 자체의 구문/형식이 잘못된 경우입니다 — JSON 파싱이 안 되거나, 필수 필드가 아예 빠졌거나, 타입이 완전히 틀린 경우입니다. 422는 구문은 완전히 올바르지만 의미(비즈니스 규칙)상 처리할 수 없는 경우입니다 — 예를 들어 이메일 필드에 유효한 문자열이 들어왔지만 이메일 형식이 아니거나, 이미 처리 완료된 주문에 대해 다시 "취소" 요청을 보내는 경우입니다. 이 둘을 구분하면 클라이언트가 "요청을 아예 다시 만들어야 하는지(400)" 아니면 "입력값 검증 메시지를 사용자에게 보여줘야 하는지(422)"를 서로 다른 UI 로직으로 처리할 수 있습니다.

Q: 클라이언트 라이브러리가 상태 코드만 보고 자동으로 재시도할지 말지 결정하게 만들려고 합니다. 어떤 기준으로 목록을 만들어야 하나요?

모범 답변: 기본 재시도 허용 목록은 408, 429, 500, 502, 503, 504로 구성하는 것이 일반적입니다. 여기에 두 가지 조건을 추가로 걸어야 합니다. 첫째, 원래 요청의 메서드가 idempotent해야 합니다(GET/PUT/DELETE, 또는 Idempotency Key가 있는 POST). 둘째, 재시도 횟수와 백오프 전략(지수 백오프 + jitter)을 반드시 함께 설계해서, 서버가 이미 과부하인 상황(503)에 재시도가 오히려 부하를 가중시키지 않도록 해야 합니다. 400, 401(재발급 없이), 403, 404, 409, 422는 기본적으로 재시도 목록에서 제외합니다 — 같은 요청을 그대로 다시 보내도 근본 원인이 해소되지 않기 때문입니다.

Q: 401을 받았는데 재시도해도 되는 경우가 있다고 하셨습니다. 어떤 경우인가요?

모범 답변: 401 자체를 그대로 재시도하는 것은 의미가 없습니다 — 같은 만료된 토큰을 다시 보내면 또 401이 나옵니다. 하지만 실무에서는 "401을 받으면 토큰을 갱신(refresh)한 뒤, 새 토큰으로 원래 요청을 1회 재시도"하는 패턴이 흔합니다. 이건 엄밀히 말하면 "같은 요청의 재시도"가 아니라 "인증 정보를 교체한 새로운 시도"입니다. 이 구분을 명확히 하지 않으면 토큰 갱신 로직 없이 401을 무한 재시도하는 버그(무한 루프)로 이어질 수 있습니다.

Q: 500 Internal Server Error가 특정 엔드포인트에서 지속적으로 발생합니다. 이게 재시도로 해결될 문제인지 아닌지 어떻게 판단하나요?

모범 답변: 500은 "서버 내부에서 처리되지 않은 예외가 발생했다"는 뜻일 뿐, 그 원인이 일시적인지(예: 순간적인 DB 커넥션 풀 고갈) 영구적인지(예: 특정 입력값에 대한 코드 버그, null pointer)는 코드만으로 알 수 없습니다. 판단 기준은 재현성입니다 — 같은 요청을 다른 시점에 다시 보냈을 때도 계속 500이 난다면 코드 버그일 가능성이 높고(재시도로 해결 안 됨), 시간을 두고 보냈을 때 성공한다면 일시적 리소스 부족일 가능성이 높습니다(재시도로 해결됨). 그래서 500에 대한 재시도는 1~2회로 제한하고, 그 이후에도 계속 실패하면 재시도를 멈추고 로그/트레이싱(Day 20)으로 근본 원인을 봐야 합니다 — 무한 재시도는 버그를 감추고 장애를 지연시킬 뿐입니다.

Q: 502 Bad Gateway를 봤다면 어느 계층의 문제로 좁혀가야 하나요?

모범 답변: 502는 리버스 프록시/로드밸런서(Day 16에서 심화)가 업스트림 앱 서버에 요청을 전달했는데 유효하지 않은 응답을 받았다는 뜻입니다 — 즉 문제는 "프록시 자체"가 아니라 "프록시 뒤에 있는 앱 서버"일 가능성이 높습니다. 흔한 원인은 앱 서버가 재시작/배포 중이라 순간적으로 연결을 거부했거나, 앱 서버가 크래시했거나, 앱 서버가 프록시가 이해할 수 없는 형식(불완전한 HTTP 응답 등)으로 응답한 경우입니다. 조사 순서는 프록시 로그에서 502가 발생한 시점의 업스트림 상태(배포 이력, 헬스체크 결과)를 먼저 확인하고, 그 다음 앱 서버 자체의 로그와 리소스(메모리/CPU)를 보는 것입니다.

11. 흔한 오해

오해 1

"4xx는 재시도하면 절대 안 되고 5xx는 재시도하면 항상 된다"

429(4xx)는 백오프 후 재시도가 정석이고, 500(5xx)이라도 idempotent하지 않은 메서드에 무한 재시도하면 위험합니다. 계열(4xx/5xx)은 출발점일 뿐, 실제 판단은 개별 코드와 메서드의 idempotency를 함께 봐야 합니다.

오해 2

"429와 503은 같은 상황에서 쓰는 코드다"

둘 다 "지금 당장 처리 못 함"이라는 표면은 비슷하지만, 429는 "이 클라이언트가 너무 많이 요청했다"(클라이언트 속도 문제), 503은 "서버 전체가 과부하/점검 중이다"(서버 용량 문제)로 원인의 위치가 다릅니다. 이 차이를 모르면 429 폭증을 서버 장애로 오인해 불필요한 온콜을 울릴 수 있습니다.

오해 3

"에러 응답 바디는 사람이 읽을 메시지만 있으면 충분하다"

클라이언트 코드가 message 문자열을 파싱해서 분기하는 것은 매우 취약한 설계입니다. 문구가 다국어화되거나 개선되면 클라이언트 로직이 깨집니다. type, code 같은 안정적인 기계 판독 필드와 사람이 읽는 필드를 분리해야 합니다.

오해 4

"Retry-After가 없으면 원하는 만큼 자주 재시도해도 된다"

Retry-After가 없다고 즉시 재시도를 반복하면, 이미 과부하이거나 rate limit이 걸린 서버에 부하를 더 가중시킵니다. 헤더가 없을 때는 지수 백오프(exponential backoff)와 jitter(무작위 지연)를 스스로 적용해야 합니다.

오해 5

"500이 나면 무조건 서버 코드에 버그가 있는 것이다"

500은 "서버가 예외를 처리하지 못했다"는 신호일 뿐, 원인은 코드 버그일 수도 있고 순간적인 리소스 고갈(DB 커넥션 풀 고갈 등) 같은 일시적 상황일 수도 있습니다. 재현성을 확인해야 어느 쪽인지 판단할 수 있습니다.
