CI/CD와 배포 전략 인터뷰 · Day 5
Artifact 관리와 버전 관리 전략
이미지 레지스트리, 태깅 전략, semantic versioning, artifact 불변성, 프로모션까지 — "빌드를 잘한다"가 아니라 "한 번 빌드한 것을 신뢰할 수 있게 추적하고 승격시킨다"는 것을 증명하는 수준을 목표로 한다.

🎯 오늘의 핵심 면접 질문

주질문 (Main Question)
"저희 팀은 각 환경(dev → staging → production)마다 소스 코드를 다시 체크아웃해서 이미지를 새로 빌드해 배포합니다.
지난주 staging에서는 문제없이 통과했던 버전이 production에서 장애를 일으켰는데, 원인을 찾아보니 두 환경의 이미지가 미묘하게 달랐습니다.
원인을 어떻게 진단하고, 재발을 막기 위해 artifact 관리 체계를 어떻게 다시 설계하시겠습니까?"
변형 질문 (Variation)
"배포 시 이미지 태그로 latest를 쓰고 있는데, 인시던트가 발생해 롤백하려는 순간 '정확히 몇 분 전의 어떤 버전으로 돌아가야 하는지' 아무도 확신하지 못했습니다.
태깅 전략과 롤백 가능성을 어떻게 다시 설계하시겠습니까?"

🔍 왜 면접에서 이 질문을 하는가

Artifact 관리는 "Docker 이미지를 빌드할 줄 아는가"를 묻는 질문이 아니다. 면접관은 이 질문으로 지원자가 "빌드"와 "배포"를 같은 것으로 착각하는지, 아니면 별개의 문제로 분리해서 다루는지를 구분한다. 환경마다 다시 빌드하는 팀은 재현성(reproducibility)을 스스로 포기한 것이고, 그 결과 "staging에서 됐는데 production에서 안 된다"는 가장 흔하면서도 가장 신뢰를 무너뜨리는 장애를 반복한다.

면접관이 실제로 평가하는 것
  • Build Once 원칙 이해: 하나의 artifact를 한 번만 빌드하고, 환경 간에는 그 artifact를 "승격(promotion)"만 시킨다는 원칙을 알고 있는가
  • 불변성(Immutability) 감각: 배포된 이미지가 나중에 몰래 바뀌지 않는다는 것을 어떻게 보장하는지 설명할 수 있는가
  • 추적성(Traceability): 프로덕션에서 문제가 생겼을 때, "이 artifact가 어떤 커밋에서, 어떤 파이프라인으로, 언제 만들어졌는가"를 즉시 되짚을 수 있는가
  • 태깅 규율: latest 같은 가변 태그의 위험성을 알고, 롤백이 실제로 가능한 태깅 체계를 설계할 수 있는가
  • 레지스트리 운영 감각: 저장 비용, retention 정책, 취약점 스캔, 서명(signing) 같은 운영 이슈까지 고려하는가
⚠️ 주니어 vs 시니어 차이: 주니어는 "Dockerfile을 작성하고 이미지를 빌드해서 레지스트리에 push했습니다"에서 멈춘다. 시니어는 "그 이미지가 어떤 태그로 불변하게 저장되고, 어떤 근거로 프로덕션까지 승격되며, 문제가 생기면 정확히 몇 초 만에 이전 버전으로 되돌릴 수 있는지"까지 설계한다.

📚 전제 개념: Artifact란 무엇이고 왜 "다시 만들면" 안 되는가

Artifact — 쉬운 설명

Artifact는 소스 코드를 빌드해서 나온 배포 가능한 산출물이다. Docker 이미지, JAR/WAR 파일, 압축된 정적 파일 번들, npm 패키지 등이 모두 artifact다. 핵심은 "소스 코드"가 아니라 "빌드가 끝난 결과물"이라는 점이다 — 이 결과물 안에는 의존성 버전, 컴파일 결과, 런타임 설정까지 이미 고정되어 있다.

왜 환경마다 다시 빌드하면 안 되는가

  • 의존성 드리프트: 같은 package.json이라도 시점에 따라 patch 버전이 자동으로 올라간 의존성이 설치될 수 있다. Lockfile이 있어도 베이스 이미지 태그(예: node:20)가 그 사이 업데이트됐다면 결과물이 달라진다
  • 빌드 환경 차이: CI 러너의 OS 패치, 캐시 상태, 환경 변수가 빌드 시점마다 미묘하게 다를 수 있다
  • "staging에서 통과, production에서 실패"의 근본 원인: 두 환경이 사실은 "같은 소스로 각각 다시 빌드한, 이름만 같은 서로 다른 산출물"을 실행하고 있었기 때문이다

이 문제를 해결하는 원칙이 Build Once, Deploy Many(한 번 빌드해서 여러 환경에 배포)다. dev에 배포한 것과 정확히 동일한 바이너리/이미지가 staging을 거쳐 production까지 그대로 이동해야, "테스트한 것 = 배포되는 것"이라는 신뢰가 성립한다.

잘못된 방식 (환경마다 재빌드): [소스 코드] --빌드--> [dev용 이미지] (커밋 abc123, 오전 9시 빌드) [소스 코드] --빌드--> [staging용 이미지] (같은 abc123, 오후 2시 재빌드 → 의존성 달라짐) [소스 코드] --빌드--> [prod용 이미지] (같은 abc123, 오후 6시 재빌드 → 또 달라짐) → 세 이미지가 "이름은 같지만 실제로는 다른 산출물" 올바른 방식 (Build Once, Promote Many): [소스 코드] --빌드(1회)--> [이미지 abc123:sha-f4e21a] │ ┌────────────────────┼────────────────────┐ ▼ ▼ ▼ [dev 배포] [staging 배포] [prod 배포] (동일 이미지 승격) (동일 이미지 승격) (동일 이미지 승격)

Artifact 불변성(Immutability) — "한 번 만들면 절대 바뀌지 않는다"

  • 정의: 특정 태그/버전으로 레지스트리에 push된 artifact는 이후 절대 덮어쓰지 않는다는 원칙
  • 왜 필요한가: 불변하지 않으면 "지금 프로덕션에 떠 있는 v2.3.1"이 어제 검증했던 v2.3.1과 실제로 같은 물건인지 아무도 보장할 수 없다
  • 구현 방법: 대부분의 레지스트리(ECR, GCR, Harbor, Docker Hub)는 "태그 불변성(tag immutability)" 옵션을 제공해, 같은 태그로 재push를 시도하면 거부하도록 설정할 수 있다

태깅 전략 — latest가 위험한 이유

  • latest 태그의 문제: 이름 그대로 "가장 최근"을 가리키는 가변 포인터라서, 시간이 지나면 같은 태그가 다른 이미지를 가리키게 된다. 롤백하려 해도 "이전의 latest가 정확히 무엇이었는지" 기록이 남지 않는다
  • Git SHA 태그: myapp:sha-f4e21a처럼 커밋 해시를 태그에 포함시키면, 이미지와 소스 코드 시점이 1:1로 대응되어 추적이 명확해진다
  • Semantic Versioning(semver): MAJOR.MINOR.PATCH 형식(예: 2.3.1)으로, MAJOR는 하위 호환이 깨지는 변경, MINOR는 하위 호환되는 기능 추가, PATCH는 버그 수정을 의미한다. 사람이 읽고 "이번 배포가 얼마나 위험한 변경인지" 가늠하는 데 유리하다
  • 실무 권장: 두 가지를 함께 쓴다 — myapp:2.3.1(사람이 읽는 릴리스 버전)과 myapp:sha-f4e21a(정확한 빌드 추적용)를 같은 이미지에 이중으로 태깅해 두 목적을 동시에 만족시킨다
⚠️ 흔한 착각: "latest를 쓰면 항상 최신 버전이 배포되니 편하다"고 생각하기 쉽지만, 실제로는 "지금 무엇이 배포되어 있는지, 롤백 시 무엇으로 돌아가야 하는지"를 아무도 확신할 수 없게 만드는 가장 흔한 사고 원인이다.

Promotion(프로모션) — 승격이지 재빌드가 아니다

Promotion은 이미 빌드된 동일한 artifact를 dev → staging → production 순으로 이동시키는 것이다. 이때 "이동"은 이미지를 다시 빌드하는 게 아니라, 같은 이미지에 새로운 태그를 추가하거나(예: staging-verified, prod-approved) 배포 매니페스트가 가리키는 이미지 참조를 바꾸는 것을 의미한다. 각 단계의 검증(테스트, 승인)을 통과해야만 다음 태그가 부여되므로, "이 artifact가 어느 단계까지 검증되었는지"가 태그 자체로 드러난다.

🏗️ Artifact 관리 설계의 네 가지 축

1. 레지스트리 선택과 접근 통제

  • 선택 기준: 클라우드 네이티브 환경이면 관리형 레지스트리(ECR, GCR, ACR)가 IAM 연동과 운영 부담 측면에서 유리하고, 멀티 클라우드/온프렘 혼합 환경이면 Harbor 같은 자체 호스팅 레지스트리가 유리하다
  • 접근 통제: 누가 push할 수 있는지(CI 파이프라인만 허용, 사람은 read-only)와 누가 pull할 수 있는지(프로덕션 클러스터만 특정 레지스트리 접근)를 최소 권한 원칙으로 분리한다
  • 취약점 스캔: push 시점에 이미지 내 알려진 CVE를 자동 스캔하고, critical 취약점이 있으면 프로모션을 차단한다

2. 태깅 전략 실전 설계

  • 불변 태그 강제: 레지스트리의 tag immutability 옵션을 켜서, 같은 태그로 실수로 덮어쓰는 사고를 원천 차단한다
  • 이중 태깅: git SHA(정확한 추적용)와 semver(사람이 읽는 릴리스 버전용)를 함께 부여한다
  • 환경 승격 태그: 검증 단계를 통과할 때마다 candidate → staging-verified → prod-approved 같은 태그를 같은 이미지에 추가로 붙여, 태그만 보고도 검증 이력을 알 수 있게 한다
  • latest 사용 제한: 로컬 개발 편의를 위한 latest는 유지하되, 실제 배포 매니페스트에서는 절대 latest를 참조하지 않고 구체적인 태그를 명시한다

3. Promotion 파이프라인 설계

[1] 소스 커밋 → CI 빌드 (1회) → 이미지 생성 + git SHA 태깅 [2] dev 환경에 배포 → 자동 테스트 통과 → "candidate" 태그 추가 [3] staging 환경에 동일 이미지 배포 → 통합/회귀 테스트 통과 → "staging-verified" 태그 추가 [4] 수동 승인 또는 카나리 지표 확인 → "prod-approved" 태그 추가 [5] production에 동일 이미지 배포 (재빌드 없음, 참조만 변경)

이 구조의 핵심은 1번 단계 이후로는 다시 컴파일/빌드하지 않는다는 것이다. 각 환경 전환은 "새 산출물을 만드는 것"이 아니라 "이미 검증된 산출물의 참조를 바꾸는 것"이어야 한다.

4. Retention 정책과 Provenance(출처 증명)

  • Retention 정책: 모든 artifact를 무한정 보관하면 스토리지 비용이 계속 늘어난다. "프로덕션에 배포된 적 있는 태그는 최소 N개월 보관, 배포된 적 없는 candidate는 N주 후 자동 정리" 같은 정책을 둔다
  • Provenance/SBOM: 이미지가 어떤 소스 커밋, 어떤 빌드 파이프라인 실행, 어떤 의존성 목록(Software Bill of Materials)으로 만들어졌는지 메타데이터로 함께 저장한다
  • 서명(Signing): Cosign 같은 도구로 이미지에 서명하고, 배포 클러스터에서 서명 검증을 강제해 "승인되지 않은 경로로 만들어진 이미지"가 배포되는 것을 막는다
💡 실무 판단: "왜 staging은 됐는데 production은 안 됐는가"라는 질문에 "정확히 같은 이미지였는가"로 즉시 답할 수 있어야 한다. 답이 "아마 같았을 것"이라면 이미 artifact 관리 체계가 실패한 것이다.

🧠 답변 사고 순서

면접 질문을 받았을 때 아래 순서로 생각을 전개하면 시니어 수준의 답변 구조가 된다.

  1. 재빌드가 원인인지부터 확인
    staging과 production 배포가 "같은 이미지"였는지, 아니면 각각 다시 빌드한 것이었는지부터 진단한다. 대부분의 "환경 간 불일치" 장애는 여기서 시작한다.
  2. Build Once, Deploy Many 구조로 전환
    CI 파이프라인을 "환경마다 빌드"에서 "한 번 빌드하고 이후 단계는 동일 artifact를 승격"하는 구조로 재설계한다.
  3. 태깅 체계를 불변하고 추적 가능하게 재설계
    latest 의존을 제거하고, git SHA + semver 이중 태깅과 태그 불변성 옵션을 적용한다.
  4. Promotion 태그로 검증 이력을 드러낸다
    candidate → staging-verified → prod-approved 같은 승격 태그로, 태그만 보고도 어느 단계까지 검증되었는지 알 수 있게 만든다.
  5. Retention·서명·스캔으로 운영 신뢰도를 마무리
    취약점 스캔, provenance 기록, 서명 검증, retention 정책까지 언급하며 "추적 가능하고 재현 가능한 artifact 파이프라인"으로 답을 정리한다.

💬 핵심 답변 예시

답변 예시 (시니어 수준)

"먼저 확인할 것은 staging에서 통과한 이미지와 production에 배포된 이미지가 정말로 동일한 바이너리였는가입니다. 현재 파이프라인이 환경마다 소스를 다시 체크아웃해서 빌드하고 있다면, 그 자체가 원인일 가능성이 매우 큽니다. 베이스 이미지 태그가 그 사이 업데이트됐거나, lockfile에 잡히지 않는 시스템 패키지 버전이 달라졌을 수 있습니다.

이를 근본적으로 막기 위해 Build Once, Deploy Many 원칙으로 파이프라인을 재설계하겠습니다. 소스 커밋이 들어오면 딱 한 번만 이미지를 빌드하고, 이후 dev·staging·production으로의 전환은 재빌드가 아니라 동일 이미지에 대한 '승격'으로 처리합니다. 예를 들어 dev 테스트를 통과하면 candidate 태그를, staging 통합 테스트를 통과하면 staging-verified 태그를, 최종 승인이 나면 prod-approved 태그를 같은 이미지에 추가로 붙입니다.

태깅 체계도 함께 정리합니다. 배포 매니페스트에서 latest 참조를 전부 제거하고, git SHA 기반 태그(예: sha-f4e21a)와 semantic versioning 태그(예: 2.3.1)를 이중으로 부여합니다. SHA 태그는 '정확히 어떤 커밋에서 만들어졌는가'를 추적하는 용도이고, semver 태그는 사람이 읽고 이번 배포의 위험도를 가늠하는 용도입니다. 레지스트리의 태그 불변성 옵션을 켜서, 같은 태그로 실수로 덮어쓰는 사고 자체를 차단합니다.

마지막으로 운영 신뢰도를 위해 push 시점에 취약점 스캔을 자동화하고, 이미지에 어떤 커밋·파이프라인 실행·의존성 목록으로 만들어졌는지 provenance 메타데이터를 남기며, Cosign 같은 도구로 서명을 적용해 승인되지 않은 경로의 이미지가 배포되지 않도록 클러스터에서 서명 검증을 강제하겠습니다. 이렇게 하면 '왜 staging은 됐는데 production은 안 됐는가'라는 질문 자체가 애초에 발생하지 않는 구조가 됩니다."

🔺 Senior-Level Follow-up 질문

Q. "Build Once 원칙을 적용하려는데, 환경마다 설정값(DB 접속 정보, feature flag 등)이 다릅니다. 이미지를 재빌드하지 않고 어떻게 다른 설정을 적용하시겠습니까?"
사고 방향: "설정이 다르다"와 "코드가 다르다"를 분리해서 생각해야 한다. 설정 차이 때문에 이미지를 다시 빌드하는 것은 Build Once 원칙을 무너뜨리는 흔한 함정이다.

핵심 답변: 이미지 안에는 실행 가능한 코드와 런타임만 담고, 환경별 설정값은 환경 변수, ConfigMap, Secret 같은 외부 주입 메커니즘으로 분리한다(12-factor app의 config 원칙). 동일 이미지가 dev에서는 dev용 환경 변수를, production에서는 production용 환경 변수를 주입받아 실행되므로, 이미지 자체는 전혀 바뀌지 않는다. Feature flag도 이미지 안에 하드코딩하지 않고 런타임에 조회하는 외부 서비스(LaunchDarkly 등)나 ConfigMap으로 관리한다.
Q. "Semantic versioning을 도입했는데, 팀원들이 MAJOR/MINOR/PATCH 구분 기준을 자꾸 다르게 판단합니다. 어떻게 일관성을 확보하시겠습니까?"
사고 방향: semver는 사람의 주관적 판단에 의존하므로, 판단을 자동화하거나 명확한 계약으로 강제해야 일관성이 생긴다.

핵심 답변: API 계약이나 공개 인터페이스가 있다면, 스키마 비교 도구로 breaking change를 자동 탐지해 MAJOR 여부를 사람이 아니라 도구가 판단하게 만든다. 커밋 메시지 컨벤션(Conventional Commits 등)을 강제해서 feat:는 MINOR, fix:는 PATCH, BREAKING CHANGE:는 MAJOR로 자동 계산되는 semantic-release 같은 도구를 CI에 연결한다. 사람의 주관적 판단에 맡기는 부분을 최소화하는 것이 핵심이다.
Q. "레지스트리 저장 비용이 계속 늘어나고 있습니다. 그렇다고 무작정 오래된 이미지를 지웠다가 롤백이 필요한 순간에 없으면 어떻게 하시겠습니까?"
사고 방향: "모든 이미지를 지운다/보관한다"의 이분법이 아니라, artifact의 '역할'에 따라 보관 정책을 다르게 가져가야 한다.

핵심 답변: 태그를 역할별로 구분해 정책을 다르게 적용한다 — 프로덕션에 배포된 적 있는 prod-approved 태그는 롤백 대비를 위해 최소 몇 개월(또는 몇 개 최신 버전)을 보관하고, 한 번도 승격되지 못한 candidate 태그는 1~2주 후 자동 삭제한다. 또한 이미지 레이어는 다수 버전이 공통 베이스 레이어를 공유하므로, 실제 저장 비용 증가분은 단순히 '이미지 개수 × 전체 크기'보다 작은 경우가 많다는 점도 비용 산정 시 함께 고려한다.
Q. "이미지 서명과 provenance를 도입하려는데, 팀에서 '배포 속도만 느려진다'는 반발이 있습니다. 어떻게 설득하시겠습니까?"
사고 방향: 서명·provenance를 "보안팀만 원하는 절차"가 아니라 "개발팀 스스로에게 필요한 사고 대응 도구"로 프레이밍해야 한다.

핵심 답변: 서명 검증은 CI 파이프라인에 자동화되어 몇 초 수준의 오버헤드만 추가되며, 수동 검토 절차를 대체하는 것이지 새로 추가하는 것이 아니다. 더 중요한 것은 인시던트 발생 시의 가치다 — provenance 메타데이터가 있으면 "이 프로덕션 이미지가 정확히 어떤 커밋, 어떤 의존성 버전으로 만들어졌는가"를 즉시 알 수 있어 원인 분석 시간이 크게 줄어든다. 이 절차가 없으면 오히려 장애 발생 시 원인 추적에 훨씬 더 많은 시간이 든다는 점을 실제 사례나 예상 시나리오로 설명한다.

⚖️ 좋은 답변과 약한 답변의 차이

✅ 좋은 답변
  • 환경 간 불일치의 원인을 "재빌드" 여부부터 진단
  • Build Once, Deploy Many 원칙을 구체적인 파이프라인 구조로 설명
  • latest의 위험성과 불변 태그(git SHA + semver)의 필요성을 명확히 구분
  • Promotion을 재빌드가 아니라 태그/참조 승격으로 설계
  • Retention, provenance, 서명 같은 운영 이슈까지 균형 있게 고려
  • 설정값과 이미지 콘텐츠를 분리해서 사고 (12-factor config)
❌ 약한 답변
  • "이미지를 다시 빌드해서 재배포하겠습니다" — 근본 원인 진단 없음
  • latest 태그를 계속 사용하며 문제의식 없음
  • 태그를 단순한 라벨로만 취급하고 추적/불변성 개념 없음
  • Promotion과 재빌드를 구분하지 못함
  • 저장 비용 절감을 위해 모든 오래된 이미지를 무차별 삭제
  • 환경별 설정 차이를 이유로 이미지를 환경마다 다시 빌드
💡 핵심 차이: 좋은 답변은 artifact를 "한 번 만들어 신뢰를 유지하며 이동시키는 자산"으로 다룬다. 약한 답변은 artifact를 "매번 다시 만들어도 되는 부산물"로 다룬다.

🏭 실무 연결 — 이런 상황에서 이렇게 설계

상황별 Artifact 관리 가이드
상황설계 포인트이유
staging은 통과했는데 production에서 장애 발생 Build Once, Deploy Many로 전환 — 동일 이미지를 승격만 시킴 환경 간 산출물 불일치를 구조적으로 제거
롤백 시 어떤 버전으로 돌아가야 할지 불확실 latest 제거, git SHA + semver 이중 태깅, 태그 불변성 적용 배포된 버전을 정확히 특정하고 되돌릴 수 있게 함
환경마다 DB/설정이 다름 이미지에는 코드만, 설정은 환경 변수/ConfigMap으로 외부 주입 설정 차이가 재빌드의 이유가 되지 않게 분리
레지스트리 저장 비용이 계속 증가 태그 역할별 retention 정책(prod-approved는 장기, candidate는 단기) 롤백 대비와 비용 절감을 동시에 만족
인시던트 시 원인 파이프라인/커밋 추적이 느림 Provenance 메타데이터와 서명 검증 도입 원인 분석 시간을 단축하고 비인가 이미지 배포 차단

셀프 체크리스트

  • ☐ "staging은 됐는데 production은 안 됐다"는 문제를 재빌드 여부로 진단할 수 있는가
  • ☐ Build Once, Deploy Many 원칙을 구체적인 파이프라인 단계로 설명할 수 있는가
  • ☐ latest 태그가 왜 롤백을 불가능하게 만드는지 설명할 수 있는가
  • ☐ git SHA 태그와 semver 태그를 각각 언제/왜 쓰는지 구분할 수 있는가
  • ☐ Promotion을 재빌드가 아니라 태그/참조 승격으로 설계할 수 있는가
  • ☐ Retention 정책을 artifact 역할(candidate vs prod-approved)에 따라 다르게 설계할 수 있는가

🚫 흔한 오해와 교정

오해 1

"각 환경에 배포할 때마다 최신 소스로 다시 빌드하는 것이 최신 상태를 유지하는 안전한 방법이다."

→ 오히려 반대다. 환경마다 다시 빌드하면 시점에 따라 의존성·베이스 이미지가 달라질 수 있어 "같은 이름이지만 실제로는 다른 산출물"이 배포된다. 안전한 방법은 한 번 빌드한 동일 artifact를 환경 간에 그대로 승격시키는 것이다.

오해 2

"latest 태그를 쓰면 항상 최신 버전이 배포되니 편리하고 안전하다."

→ latest는 시간이 지나면 다른 이미지를 가리키는 가변 포인터일 뿐이다. 지금 프로덕션에 떠 있는 것이 정확히 무엇인지, 롤백 시 무엇으로 돌아가야 하는지 아무도 확신할 수 없게 만드는 가장 흔한 사고 원인이다.

오해 3

"Semantic Versioning은 그냥 버전 번호를 매기는 관례일 뿐 큰 의미가 없다."

→ semver는 MAJOR/MINOR/PATCH 구분을 통해 "이번 배포가 하위 호환을 깨는지, 기능 추가인지, 버그 수정인지"를 숫자만으로 전달하는 커뮤니케이션 도구다. 의존하는 다른 팀이나 클라이언트가 이 번호만 보고 업그레이드 위험도를 판단하므로, 관례가 아니라 계약에 가깝다.

오해 4

"환경별 설정이 다르면 이미지도 환경별로 다르게 빌드해야 한다."

→ 설정 차이는 이미지를 다시 빌드할 이유가 되지 않는다. 이미지에는 코드와 런타임만 담고, DB 접속 정보나 feature flag 같은 환경별 값은 환경 변수나 ConfigMap으로 실행 시점에 주입하는 것이 올바른 분리다.
