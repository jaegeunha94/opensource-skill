Rolling 배포와 점진적 릴리스
Rolling 배포는 가장 흔히 쓰이는 기본 전략이라 "그냥 씁니다"로 끝내기 쉽다. 인스턴스를 몇 개씩, 어떤 순서로 교체할지, 실패를 어떻게 감지해 자동으로 멈출지, 신구 버전이 동시에 떠 있는 동안 무엇을 지켜야 하는지까지 다룬다.

🎯 오늘의 핵심 면접 질문

주질문 (Main Question)
"Rolling 배포에서 인스턴스를 몇 개씩, 어떤 순서로 교체하시겠습니까?
그리고 배포 도중 문제가 생겼다는 것을 어떻게 판단해서 자동으로 멈추시겠습니까?"
변형 질문 (Variation)
"Rolling 배포와 Blue-Green의 근본적인 차이는 무엇입니까?
그리고 배포 도중 신버전과 구버전이 동시에 트래픽을 받는 상황을 어떻게 안전하게 만드시겠습니까?"

🔍 왜 면접에서 이 질문을 하는가

Rolling 배포는 Kubernetes Deployment의 기본값이고, 대부분의 팀이 별다른 고민 없이 쓴다. 그래서 이 질문은 "Rolling을 아는가"가 아니라 "기본값을 그대로 쓰면서도 왜 그렇게 설정했는지 설명할 수 있는가"를 검증한다. 배치 크기 하나, health check 하나를 잘못 설정해도 장애로 이어질 수 있는데, 이 디테일을 얼마나 의식하고 있는지가 시니어와 주니어를 가른다.

면접관이 실제로 평가하는 것
  • 배치 크기 감각: "한 번에 몇 대씩 바꾸는가"가 배포 속도와 장애 반경(blast radius) 사이의 trade-off라는 것을 이해하는가
  • Health check 설계력: readiness와 liveness의 차이를 알고, 새 인스턴스가 "정말 트래픽을 받을 준비가 됐는지"를 어떻게 검증하는가
  • 실패 감지와 자동 중단: 배포 도중 에러율이 튀는 것을 어떻게 감지하고, 사람이 개입하기 전에 자동으로 멈추는 장치가 있는가
  • 신구 버전 공존 감각: Rolling은 Blue-Green보다 "신구 버전이 동시에 트래픽을 받는 시간"이 훨씬 길다는 것을 알고, 그 기간의 API/DB 호환성을 어떻게 관리하는가
  • 롤백 방식 이해: Rolling의 롤백도 즉시 스위치가 아니라 다시 롤링으로 되돌아간다는 것을 아는가
⚠️ 주니어 vs 시니어 차이: 주니어는 "Rolling 업데이트는 인스턴스를 하나씩 바꾸는 겁니다"에서 멈춘다. 시니어는 배치 크기를 어떤 기준으로 정하는지, readiness probe가 무엇을 검증해야 하는지, 실패율이 몇 %를 넘으면 배포를 자동으로 멈추는지까지 수치와 근거를 들어 설명한다.

📚 전제 개념: Rolling 배포가 실제로 하는 일

Rolling 배포란 무엇인가 (쉬운 설명)

서버가 10대 떠 있다고 하자. Blue-Green은 신버전 10대를 완전히 새로 띄운 뒤 트래픽을 통째로 옮긴다. Rolling은 그렇게 하지 않는다. 기존 10대 중 일부(예: 2대)만 신버전으로 바꾸고, 그 2대가 정상 동작하는 것을 확인한 뒤 다음 2대를 바꾸는 식으로 점진적으로 전체를 교체한다. 인프라를 2배로 쓸 필요가 없다는 것이 가장 큰 장점이다.

Rolling과 Blue-Green의 근본적인 차이

Blue-Green
  • 완전히 분리된 두 환경
  • 전환은 순간적 (초 단위)
  • 신구 버전 공존 시간 = 매우 짧음
  • 인프라 비용 = 일시적으로 2배
Rolling
  • 하나의 환경 안에서 점진적 교체
  • 전환은 배치 단위로 여러 분~수십 분
  • 신구 버전 공존 시간 = 배포 내내 (김)
  • 인프라 비용 = 추가 비용 거의 없음

이 차이에서 나오는 가장 중요한 결론: Rolling 배포는 "배포하는 동안 계속" 신버전과 구버전이 같은 트래픽을 나눠 받는다. Blue-Green은 이 기간이 짧아서 대충 넘어갈 수 있었던 API 호환성, DB 스키마 호환성 문제가 Rolling에서는 훨씬 더 중요해진다.

maxSurge와 maxUnavailable — 배치 크기를 결정하는 두 축

  • maxUnavailable: 배포 도중 "동시에 얼마나 많은 인스턴스가 트래픽을 못 받는 상태"를 허용할지. 예를 들어 10대 중 maxUnavailable: 2면, 항상 최소 8대는 살아있다.
  • maxSurge: 기존 인스턴스 수보다 "얼마나 더 많이 새로 띄울 수 있는지". maxSurge: 2면 일시적으로 12대까지 늘어날 수 있다.
  • 이 두 값의 조합이 배포 속도, 리소스 사용량, 장애 반경을 동시에 결정한다.

Readiness Probe vs Liveness Probe — 왜 둘 다 필요한가

Rolling 배포가 안전하려면 "이 새 인스턴스가 트래픽을 받아도 되는가"를 정확히 판단해야 한다. 이 판단을 하는 것이 readiness probe다.

  • Liveness probe: "이 프로세스가 살아있는가"만 확인한다. 실패하면 컨테이너를 재시작한다. 하지만 "재시작해도 트래픽을 받을 준비가 됐는가"는 별개 문제다.
  • Readiness probe: "지금 트래픽을 받아도 정상 응답할 수 있는가"를 확인한다. 실패하면 컨테이너를 재시작하지 않고, 대신 로드밸런서/서비스에서 제외한다. Rolling 배포에서 다음 배치로 넘어갈지 결정하는 것이 바로 이 readiness 상태다.
  • 둘을 혼동해서 liveness만 설정하면, 아직 초기화(DB 커넥션 풀 생성, 캐시 워밍)가 끝나지 않은 인스턴스가 트래픽을 받아 에러를 뱉는 문제가 생긴다.
⚠️ 흔한 오해: "readiness probe가 통과했다 = 다음 배치로 넘어가도 안전하다"고 단순화하기 쉽다. 하지만 readiness는 "이 인스턴스 하나가 응답 가능한가"만 본다. 실제 트래픽을 받았을 때 에러율이 튀는지는 별도의 관찰 기간과 지표(에러율, 지연시간)로 확인해야 한다.

🔁 배치 크기 결정과 순차 교체 절차

전체 10 Pod, maxUnavailable=1, maxSurge=1 인 경우 [배치 1] - 신규 Pod 1개 생성 (총 11개, surge) - readiness probe 통과 대기 - 통과하면 구버전 Pod 1개 종료 (총 10개로 복귀) - 관찰 기간(예: 60초) 동안 에러율/지연시간 모니터링 ↓ 이상 없음 [배치 2] → [배치 3] → ... → [배치 10] - 위 과정을 반복 - 배치 사이마다 이전 배치가 "진짜 안정적인지" 확인 후 다음 배치 시작 [중단 조건 — 어느 배치에서든 발동 가능] - 신규 Pod의 에러율이 임계치(예: 5xx 비율 1% 초과)를 넘으면 - 배포 파이프라인이 자동으로 롤아웃을 멈추고 알림 발송 - 사람이 진행/롤백을 판단

배치 크기를 어떤 기준으로 정하는가

배치 크기 결정 기준
  • 장애 반경(blast radius): 배치가 클수록 한 번에 많은 인스턴스가 바뀌어 배포는 빠르지만, 문제가 생겼을 때 영향받는 트래픽 비율도 커진다.
  • 전체 인스턴스 수: 인스턴스가 3대뿐이면 배치 크기를 1로 해도 이미 33%가 한 번에 바뀐다. 인스턴스 수가 적을수록 신중해야 한다.
  • 배포 빈도: 하루에도 여러 번 배포하는 팀은 배치를 작게, 관찰 기간을 짧게 가져가 전체 배포 시간을 관리한다.
  • 서비스의 중요도: 결제처럼 실패 비용이 큰 서비스는 배치를 작게(예: 10%씩), 내부 배치 서비스는 배치를 크게(예: 50%씩) 가져가는 등 서비스별로 다르게 설정한다.

🚨 실패 감지와 자동 롤아웃 중단

Rolling 배포에서 가장 위험한 실수는 "readiness probe만 통과하면 계속 진행"하는 설정이다. Probe는 인스턴스 하나의 헬스만 보기 때문에, 실제 트래픽을 받았을 때 나타나는 문제(특정 API 경로 에러, DB 커넥션 고갈, 메모리 누수)는 잡지 못한다.

진짜 실패 감지가 봐야 하는 지표

  • 에러율(error rate): 5xx 응답 비율이 배포 전 baseline 대비 얼마나 튀는지
  • 지연시간(latency): p95/p99 응답 시간이 임계치를 넘는지
  • 포화도(saturation): 신버전 Pod의 CPU/메모리가 비정상적으로 치솟는지
  • 비즈니스 지표: 결제 성공률, 로그인 성공률처럼 사용자 영향과 직결된 지표

이 지표들을 배포 도구(Argo Rollouts, Flagger, 또는 자체 CI/CD 스크립트)가 배치와 배치 사이의 관찰 기간 동안 자동으로 조회하고, 임계치를 넘으면 다음 배치로 진행하지 않고 롤아웃을 자동으로 멈추거나 이전 상태로 되돌린다.

💡 실무 팁: Kubernetes 기본 Deployment는 이 자동 지표 기반 중단 기능이 없다. 이걸 원한다면 Argo Rollouts나 Flagger 같은 Progressive Delivery 도구를 도입해 Prometheus 지표와 연동한 자동 분석(analysis) 단계를 배치 사이에 넣어야 한다.

"배치 하나에서 일부만 실패하면?" — 부분 실패 판단

예를 들어 신규 Pod 2개 중 1개만 readiness에 실패했다면, 그 배치 전체를 실패로 간주해야 한다. 부분 성공을 "일단 진행"으로 처리하면, 다음 배치에서 더 많은 문제 인스턴스가 늘어나는 위험한 패턴이 된다. 하나라도 실패하면 그 배치를 재시도하거나 롤아웃을 멈추고 원인을 조사하는 것이 원칙이다.

🔗 신구 버전 공존 기간 — 호환성이 선택이 아닌 이유

Rolling 배포 도중에는 배포 시간 내내 신버전과 구버전이 같은 로드밸런서 뒤에서 동시에 트래픽을 받는다. 이 시간이 Blue-Green보다 훨씬 길기 때문에(수 분~수십 분), 신구 버전이 서로 호환되지 않으면 배포 자체가 곧 장애가 된다.

N-1 호환성 규칙

지켜야 할 원칙
  • API 호환성: 신버전이 응답 스키마를 바꾼다면, 구버전 클라이언트(또는 구버전 서버가 호출하는 다른 서비스)가 새 스키마를 이해하지 못해도 동작해야 한다. 필드 추가는 안전하지만, 필드 삭제나 타입 변경은 위험하다.
  • DB 스키마 호환성: Blue-Green과 동일하게 Expand-Contract 원칙이 적용된다. 다만 Rolling에서는 신구 버전이 같은 시각에 같은 트래픽 아래 공존하므로, "배포가 끝날 때까지" 구버전이 새 스키마를 이해하지 못해도 깨지지 않아야 한다는 조건이 더 엄격하게 요구된다.
  • 메시지 큐 계약: 신버전이 새 이벤트 필드를 추가해 발행하면, 아직 배포되지 않은 구버전 컨슈머가 그 필드를 무시하고도 정상 동작해야 한다.
⚠️ 흔한 오해: "Rolling은 자동으로 무중단이니 신구 버전 호환성은 신경 안 써도 된다." 실제로는 정반대다. Rolling은 신구 버전 공존 시간이 가장 길기 때문에, 배포 전략 중에서 호환성 규칙을 가장 엄격하게 지켜야 하는 전략이다.

🧠 답변 사고 순서

"인스턴스를 몇 개씩, 어떤 순서로 교체하고, 실패를 어떻게 감지하시겠습니까"라는 질문을 받았을 때 아래 순서로 답변을 전개한다.

  1. 배치 크기 기준 제시
    전체 인스턴스 수, 서비스 중요도, 배포 빈도를 근거로 maxUnavailable/maxSurge 값을 어떻게 정하는지 설명한다.
  2. Readiness probe 설계 언급
    liveness와 readiness를 구분하고, 다음 배치로 넘어가는 기준이 readiness 통과라는 것을 명시한다.
  3. 배치 사이 관찰 기간과 실패 감지 지표
    에러율, 지연시간, 포화도, 비즈니스 지표 등 구체적인 지표를 들고, 임계치를 넘으면 자동으로 롤아웃이 멈추는 구조를 설명한다.
  4. 신구 버전 공존 기간의 호환성 전제
    N-1 호환성 원칙(API, DB, 메시지 계약)을 언급하고, Rolling이 이 원칙을 가장 엄격하게 요구한다는 점을 짚는다.
  5. 롤백 방식 설명
    Rolling의 롤백도 즉시 스위치가 아니라 "이전 버전으로 다시 롤링"하는 방식임을 설명하고, 그래서 롤백 자체도 시간이 걸린다는 한계를 인정한다.

💬 핵심 답변 예시

답변 예시 (시니어 수준)

"저는 전체 인스턴스 수와 서비스 중요도를 기준으로 배치 크기를 정합니다. 예를 들어 결제처럼 실패 비용이 큰 서비스는 maxUnavailable: 10%, maxSurge: 10% 정도로 작게 잡아서 한 번에 바뀌는 비율을 최소화하고, 내부용 배치 서비스는 조금 더 크게 잡아 배포 속도를 우선합니다.

새 인스턴스가 트래픽을 받아도 되는지는 readiness probe로 판단합니다. liveness probe는 프로세스 생존만 확인하기 때문에, 다음 배치로 넘어갈지는 반드시 readiness 기준으로 판단합니다. 다만 readiness 통과가 곧 안전을 의미하진 않기 때문에, 배치와 배치 사이에 관찰 기간을 두고 에러율과 p99 지연시간을 확인합니다. Kubernetes 기본 Deployment에는 이 자동 지표 분석이 없어서, Argo Rollouts를 도입해 Prometheus 지표 기반으로 임계치를 넘으면 자동으로 롤아웃을 멈추도록 구성합니다.

Rolling은 Blue-Green과 달리 배포 시간 내내 신구 버전이 같은 트래픽을 나눠 받기 때문에, API 응답 스키마나 DB 스키마 변경은 N-1 호환성을 지켜야 합니다. 신버전이 먼저 배포돼도 구버전이 문제없이 동작해야 하고, 그 반대도 마찬가지입니다. 필드를 삭제하거나 타입을 바꾸는 대신, 항상 추가 방식으로 변경하고 정리는 배포가 완전히 끝난 뒤로 미룹니다.

그리고 롤백도 즉시 스위치가 아니라는 걸 인지하고 있습니다. Rolling의 롤백은 다시 이전 버전으로 롤링하는 것이기 때문에 시간이 걸리고, 그 동안에도 신구 버전이 공존합니다. 그래서 저는 문제를 조기에 감지해서 배포 초반 배치에서 멈추는 것이 가장 중요하다고 생각하고, 자동 중단 임계치를 보수적으로 설정합니다."

🔺 Senior-Level Follow-up 질문

Q. "maxUnavailable을 0으로 설정하면 무슨 일이 생깁니까?"
사고 방향: maxUnavailable이 0이면 기존 인스턴스를 하나도 내리지 않고 새 인스턴스를 먼저 띄워야 하므로, 사실상 maxSurge에 의존하게 된다.

핵심 답변: maxUnavailable을 0으로 하면 배포 도중 가용 용량이 절대 줄어들지 않지만, 대신 반드시 maxSurge를 1 이상으로 설정해야 새 인스턴스를 추가로 띄울 수 있다. 만약 maxSurge도 0이면 배포 자체가 멈춰버린다. 또한 리소스 여유가 없는 클러스터에서는 surge Pod를 위한 추가 리소스를 확보하지 못해 배포가 대기 상태에 걸릴 수 있다는 것도 언급해야 한다.
Q. "Rolling 배포에서 카나리아처럼 특정 사용자 그룹만 신버전을 먼저 보게 할 수 있습니까?"
사고 방향: 기본 Rolling은 "어떤 사용자가 신버전을 받을지"를 제어하지 못한다. 트래픽이 인스턴스 단위로 무작위 분산되기 때문이다.

핵심 답변: 순수한 Rolling으로는 불가능하고, 이 요구가 있다면 사실상 Canary 전략(가중치 기반 트래픽 분배, 또는 특정 헤더/쿠키 기반 라우팅)이 필요하다. Rolling과 Canary는 배타적이지 않아서, Argo Rollouts처럼 Canary 단계(가중치 조절)를 거친 뒤 나머지를 Rolling으로 마무리하는 하이브리드 방식도 실무에서 쓰인다.
Q. "DB 마이그레이션과 Rolling 배포를 함께 진행할 때 가장 위험한 부분은 무엇입니까?"
사고 방향: Rolling은 신구 버전 공존 시간이 길기 때문에, 배포 도중 어느 순간이든 구버전 코드가 스키마를 건드릴 가능성이 있다는 것을 전제해야 한다.

핵심 답변: 배포가 완료되기 전까지는 구버전 인스턴스가 계속 트래픽을 받고 있으므로, 구버전이 이해하지 못하는 방식으로 스키마를 바꾸면 그 순간부터 배포가 끝날 때까지 에러가 발생한다. 그래서 Expand 단계(하위 호환 가능한 추가)는 반드시 배포 시작 전에 끝내고, Contract 단계(구 컬럼 삭제 등)는 전체 배치가 100% 완료되고 구버전 인스턴스가 완전히 사라진 것을 확인한 뒤에만 수행한다.
Q. "왜 Rolling 배포 도중 갑자기 응답 시간이 늘어나는 현상이 생길 수 있습니까?"
사고 방향: 배포 도중에는 인스턴스 수가 일시적으로 줄거나(maxUnavailable), 늘어난 인스턴스(maxSurge)가 아직 워밍업되지 않은 상태로 트래픽을 받는다는 것을 떠올려야 한다.

핵심 답변: 두 가지 원인이 있다. 첫째, maxUnavailable만큼 가용 인스턴스가 줄어드는 순간 남은 인스턴스의 부하가 일시적으로 늘어난다. 둘째, 새로 뜬 인스턴스는 JIT 컴파일, 커넥션 풀 초기화, 캐시 워밍이 안 된 상태라 readiness는 통과해도 초기 응답이 느릴 수 있다. 이를 완화하려면 readiness probe에 초기 워밍업 시간을 반영하거나, 트래픽을 점진적으로만 흘려보내는 startup 지연 로직을 넣는다.

⚖️ 좋은 답변과 약한 답변의 차이

✅ 좋은 답변
  • 배치 크기를 서비스 중요도와 인스턴스 수 기준으로 구체적으로 설명
  • readiness와 liveness probe의 역할 차이를 명확히 구분
  • 에러율/지연시간 같은 구체적 지표로 자동 중단 기준을 제시
  • Rolling이 신구 버전 공존 시간이 가장 길다는 것을 인지하고 N-1 호환성을 언급
  • 롤백도 즉시가 아니라 다시 롤링해야 한다는 한계를 인정
❌ 약한 답변
  • "인스턴스를 하나씩 순서대로 바꿉니다"에서 설명이 끝남
  • readiness와 liveness를 구분하지 못하거나 언급하지 않음
  • "probe만 통과하면 안전합니다"라고 단정
  • 신구 버전 호환성 문제를 전혀 고려하지 않음
  • "문제가 생기면 그냥 롤백하면 됩니다"라고 단순화
💡 핵심 차이: 약한 답변은 Rolling을 "인스턴스를 순차적으로 바꾸는 절차"로만 본다. 좋은 답변은 배치 크기, 헬스체크 종류, 자동 실패 감지, 호환성 요구사항까지 하나의 시스템으로 엮어서 설명한다.

아래 기준으로 자신의 답변 수준을 체크하라.

레벨 할 수 있는 것 부족한 것
Lv 1 "인스턴스를 하나씩 순서대로 바꾼다"고 말할 수 있다 배치 크기 기준, probe 종류 구분, 실패 감지 지표를 모름
Lv 2 maxSurge/maxUnavailable, readiness/liveness 차이를 설명할 수 있다 구체적인 실패 감지 지표, 자동 중단 임계치 설계 부족
Lv 3 배치별 관찰 기간과 지표 기반 자동 중단을 설명할 수 있다 신구 버전 호환성(N-1 규칙)에 대한 인식 부족
Lv 4 (목표) 배치 크기 기준 + probe 설계 + 자동 실패 감지 + N-1 호환성 + 롤백 한계까지 통합 설명 + 플랫폼별 구현 차이 연결 —

셀프 체크리스트

  • ☐ maxSurge와 maxUnavailable의 역할 차이를 설명할 수 있는가
  • ☐ readiness probe와 liveness probe의 역할 차이를 설명할 수 있는가
  • ☐ 배치 사이 관찰 기간에 어떤 지표를 확인해야 하는지 구체적으로 말할 수 있는가
  • ☐ Kubernetes 기본 Deployment가 지표 기반 자동 중단을 지원하지 않는다는 것을 아는가
  • ☐ Rolling이 Blue-Green보다 신구 버전 공존 시간이 길다는 것을 설명할 수 있는가
  • ☐ N-1 호환성 규칙(API, DB, 메시지 계약)을 말할 수 있는가
  • ☐ Rolling의 롤백이 즉시 스위치가 아니라 다시 롤링해야 한다는 것을 인지하는가

🚫 흔한 오해와 교정

오해 1

"Rolling 배포는 기본값이라 별도 설정 없이도 안전하다."

→ Kubernetes의 기본 Rolling 설정은 인프라 관점의 가용성(몇 대가 살아있는가)만 보장할 뿐, 애플리케이션 관점의 정상 동작(에러율, 지연시간)은 확인하지 않는다. 지표 기반 자동 분석 없이는 문제 있는 버전이 배치를 넘어 계속 확산될 수 있다.

오해 2

"readiness probe만 통과하면 다음 배치로 넘어가도 안전하다."

→ readiness는 인스턴스 하나가 요청에 응답할 수 있는지만 본다. 실제 트래픽 패턴에서 나타나는 에러율 상승이나 지연시간 증가는 별도의 관찰 기간과 지표 확인이 있어야 잡을 수 있다.

오해 3

"Rolling 배포는 인스턴스만 순차적으로 바꾸면 되므로 API/DB 호환성은 크게 신경 쓰지 않아도 된다."

→ 정반대다. Rolling은 신구 버전이 배포 시간 내내 동시에 트래픽을 받기 때문에, 어떤 배포 전략보다도 N-1 호환성(하위 호환 가능한 API/DB 변경)을 엄격하게 지켜야 한다.

오해 4

"배치 크기는 작을수록 무조건 안전하고 좋다."

→ 배치를 작게 하면 장애 반경은 줄지만, 전체 배포 시간이 길어지고 그만큼 신구 버전 공존 시간도 길어진다. 서비스 중요도와 배포 빈도를 함께 고려해 적정 배치 크기를 정해야 하며, 무조건 작은 것이 정답은 아니다.
