CI/CD와 배포 전략 인터뷰 · Day 4
테스트 자동화 전략과 CI 품질 게이트
테스트 피라미드, 단위/통합/E2E의 역할 분담, flaky test 진단과 격리, 커버리지 임계값의 함정까지 — "테스트를 많이 쓴다"가 아니라 "CI의 신호를 신뢰할 수 있게 설계한다"는 것을 증명하는 수준을 목표로 한다.

🎯 오늘의 핵심 면접 질문

주질문 (Main Question)
"CI 파이프라인 테스트가 40분 넘게 걸리고, 가끔 실패했다가 재실행하면 통과하는 테스트(flaky test)가 많아졌습니다.
팀이 CI 결과를 신뢰하지 않고 실패해도 재실행부터 누르는 문화가 생겼습니다. 어떻게 개선하시겠습니까?"
변형 질문 (Variation)
"커버리지 90% 미만이면 merge를 막는 정책이 있는데, 오히려 의미 없는 테스트만 늘고 개발 속도가 느려졌다는 불만이 나옵니다.
이 품질 게이트를 어떻게 재설계하시겠습니까?"

🔍 왜 면접에서 이 질문을 하는가

테스트 자동화는 "테스트 코드를 짤 줄 아는가"를 묻는 질문이 아니다. 면접관은 이 질문으로 지원자가 CI의 신호(signal)를 하나의 자산으로 다루는지, 아니면 그저 통과시켜야 할 관문으로만 보는지를 구분한다. 테스트가 많아도 신뢰할 수 없으면 오히려 해롭다 — 실패를 무시하는 습관이 팀 전체에 퍼지면, 진짜 버그가 섞여 들어와도 아무도 알아채지 못한다.

면접관이 실제로 평가하는 것
  • 피라미드 감각: 왜 unit test가 많고 E2E가 적어야 하는지, 그 비율이 무너졌을 때 무슨 일이 생기는지 아는가
  • Flaky test 진단력: "가끔 실패한다"를 우연으로 넘기지 않고, race condition·외부 의존성·상태 공유 같은 근본 원인으로 분해할 수 있는가
  • 지표 비판력: 커버리지 %를 목표로 착각하지 않고, 그 숫자가 무엇을 보장하고 무엇을 보장하지 않는지 아는가
  • 게이트 설계력: 무엇을 merge 차단 기준으로 삼고, 무엇을 경고 수준으로만 둘지 판단할 수 있는가
  • 조직 영향 감각: CI 시간과 신뢰도가 개발자 행동(로컬 테스트 스킵, 실패 무시)에 어떤 영향을 주는지 이해하는가
⚠️ 주니어 vs 시니어 차이: 주니어는 "테스트 커버리지를 90%로 올렸습니다"에서 멈춘다. 시니어는 "어떤 계층의 테스트를 늘렸는지, flaky test를 어떻게 격리하고 추적했는지, 커버리지가 실제 결함 검출력과 어떻게 다른지"까지 구체적으로 설명한다.

📚 전제 개념: 왜 테스트는 "많이"가 아니라 "구조"가 중요한가

테스트 피라미드 — 쉬운 설명

테스트 피라미드는 "어떤 종류의 테스트를 얼마나 많이 만들어야 하는가"에 대한 오래된 원칙이다. 핵심 아이디어는 단순하다: 빠르고 격리된 테스트는 많이, 느리고 깨지기 쉬운 테스트는 적게 만들라는 것이다.

Unit Test (많이)

함수/클래스 하나를 외부 의존성 없이 검증. 밀리초 단위로 실행되고, 실패 지점을 정확히 짚어준다. 대부분의 로직 결함을 여기서 잡는다.

Integration Test (중간)

DB, 캐시, 외부 API 등 실제 컴포넌트 간 연동을 검증. 초 단위로 실행되며, 계층 간 계약이 깨졌는지 확인한다.

E2E Test (적게)

사용자 관점에서 전체 흐름(로그인 → 주문 → 결제)을 검증. 분 단위로 실행되고, 환경 의존성이 커서 가장 깨지기 쉽다.

이 비율이 뒤집힌 상태를 "역피라미드(ice-cream cone)"라고 부른다. E2E 테스트만 잔뜩 쌓아두면, 실제 로직 결함이 아니라 네트워크 타이밍이나 브라우저 렌더링 지연 때문에 실패하는 경우가 많아지고, CI는 느리면서도 신뢰할 수 없는 상태가 된다.

Flaky Test — "같은 코드인데 랜덤하게 실패하는" 테스트

  • 정의: 코드도, 환경도 바뀌지 않았는데 같은 테스트가 통과했다 실패했다를 반복하는 것
  • 흔한 원인 1 — Race condition: 비동기 작업이 끝나기 전에 결과를 검증해서 타이밍에 따라 결과가 달라짐
  • 흔한 원인 2 — 외부 의존성: 실제 네트워크, 실제 시간(Date.now()), 실제 랜덤값에 의존해 환경마다 값이 달라짐
  • 흔한 원인 3 — 테스트 간 상태 공유: 이전 테스트가 남긴 DB 레코드나 전역 변수가 다음 테스트 결과에 영향을 줌
  • 흔한 원인 4 — 병렬 실행 부작용: 여러 테스트가 동시에 같은 리소스(파일, 포트, 테이블)를 건드려 충돌

Flaky test가 위험한 이유는 버그 자체보다 신뢰의 붕괴에 있다. 실패를 보고도 "또 flaky겠지"라며 재실행 버튼부터 누르는 습관이 생기면, 진짜 회귀 버그가 섞여 들어와도 같은 방식으로 무시된다.

커버리지(Coverage) — 숫자가 보장하는 것과 보장하지 않는 것

  • 커버리지가 보장하는 것: 그 코드 라인이 테스트 실행 중 "실행은 되었다"는 사실
  • 커버리지가 보장하지 않는 것: 그 실행 결과가 제대로 검증(assert)되었는가. assertion 없는 테스트도 커버리지에는 100% 잡힌다
  • Diff Coverage: 전체 커버리지 대신, 이번 PR에서 변경된 라인만의 커버리지를 보는 지표. 레거시 코드 전체를 갑자기 90%로 끌어올리라는 비현실적 압박을 피하면서, 새 코드의 품질은 지킬 수 있다
⚠️ 흔한 착각: "커버리지 90%"라는 숫자 자체는 결함 검출력을 보장하지 않는다. assertion이 약하거나 없는 테스트로 채운 90%보다, assertion이 명확한 70%가 실무적으로 더 신뢰할 수 있다.
// 커버리지는 채우지만 아무것도 검증하지 않는 테스트의 예 test('calculateDiscount runs without error', () => { calculateDiscount(100, 'VIP'); // 실행만 하고 결과를 검증하지 않음 → 커버리지는 오름 }); // 실제로 신뢰할 수 있는 테스트 test('VIP 회원은 20% 할인이 적용된다', () => { expect(calculateDiscount(100, 'VIP')).toBe(80); // 결과를 명확히 검증 expect(calculateDiscount(100, 'NORMAL')).toBe(100); });

🏗️ CI 품질 게이트 설계의 네 가지 축

1. 테스트 계층 재조정 — "무엇을 어느 계층에서 검증할 것인가"

[E2E] 소수의 핵심 사용자 흐름만 (로그인, 결제 등) — 분 단위, 느림 ─────── [Integration] DB/외부 연동 계약 검증 — 초 단위, 중간 속도 ───────────── [Unit] 로직 대부분 여기서 — 밀리초 단위, 빠르고 안정적

기존 테스트 중 상당수가 E2E로 쏠려 있다면, 같은 시나리오를 unit/integration 레벨로 "내려서" 다시 작성할 수 있는지부터 검토한다. 예를 들어 "할인율 계산이 맞는지"는 브라우저를 띄우는 E2E가 아니라 unit test로 충분히 검증된다.

2. Flaky Test 격리(Quarantine) 프로세스

  • 탐지: 재실행 시 결과가 바뀐 테스트를 CI 시스템에서 자동으로 태깅하거나, 별도 flaky test 대시보드로 추적한다
  • 격리: 확인된 flaky test는 즉시 별도 스위트(quarantine)로 옮겨 merge 게이트에서 제외한다 — 단, "제외"가 "방치"가 되지 않도록 소유자와 기한을 명시적으로 부여한다
  • 추적: 격리된 테스트는 이슈 트래커에 등록하고, 일정 기간(예: 2주) 내 원인 수정 또는 삭제가 없으면 자동 알림이 가도록 한다
  • 재통합: 원인을 고친 뒤에는 다시 메인 스위트로 복귀시키고, 동일 문제 재발을 막기 위한 회고를 짧게 남긴다

3. 품질 게이트 재정의 — "무엇을 막고, 무엇을 경고만 할 것인가"

  • Merge를 막는 기준(Hard Gate): 신규/변경 코드의 diff coverage 미달, 핵심 회귀 테스트 실패, 보안 스캔의 critical 이슈
  • 경고만 하는 기준(Soft Gate): 전체 저장소의 절대 커버리지 하락 추세, lint warning, 성능 회귀(임계값 이내)
  • 정책 변경의 근거: "숫자를 못 채워서 막는다"가 아니라 "이 실패가 실제로 프로덕션 리스크와 연결되는가"를 기준으로 hard/soft를 나눈다

4. CI 실행 시간 최적화

  • 병렬화(Sharding): 테스트 스위트를 여러 runner에 나눠 동시에 실행. 실행 시간이 shard 수에 비례해 줄어든다
  • 의존성/빌드 캐싱: 패키지 설치, 컴파일 결과를 캐싱해 매 실행마다 처음부터 다시 받지 않게 한다
  • 영향받은 테스트만 선택 실행(Test Selection): 변경된 파일과 의존 그래프를 분석해, 영향 가능성이 낮은 테스트는 매 PR마다 돌리지 않고 주기적으로만 전체 실행한다
  • 느린 테스트 모니터링: 개별 테스트 실행 시간을 추적해, 특정 테스트가 전체 파이프라인 시간을 지배하지 않는지 정기적으로 점검한다
💡 실무 판단: CI 시간이 늘어나는 근본 원인은 대개 "테스트가 많아서"가 아니라 "느린 계층(E2E, 통합)에 로직 검증까지 몰려 있어서"다. 계층을 재조정하면 테스트 개수는 늘어도 전체 시간은 오히려 줄어드는 경우가 많다.

🧠 답변 사고 순서

면접 질문을 받았을 때 아래 순서로 생각을 전개하면 시니어 수준의 답변 구조가 된다.

  1. 현재 테스트 계층 분포부터 진단
    E2E/통합/단위 테스트 비율이 역피라미드인지 먼저 확인한다고 말한다. CI 시간의 대부분이 어느 계층에서 소모되는지부터 파악한다.
  2. Flaky test를 우연이 아니라 진단 대상으로 다룬다
    재실행 이력을 추적해 flaky 테스트를 식별하고, 즉시 quarantine하되 소유자와 기한을 부여해 방치되지 않게 한다.
  3. 커버리지를 목표가 아니라 신호로 재정의
    절대 커버리지 강제 대신 diff coverage와 assertion 품질을 리뷰 기준에 포함시킨다.
  4. Hard Gate와 Soft Gate를 분리
    실제 프로덕션 리스크와 직결되는 실패만 merge를 막고, 나머지는 경고로 낮춰 개발 속도와 신뢰도 사이 균형을 잡는다.
  5. CI 실행 시간 최적화로 마무리
    병렬화, 캐싱, test selection을 언급하며 "빠르고 신뢰할 수 있는 CI"라는 목표로 답을 정리한다.

💬 핵심 답변 예시

답변 예시 (시니어 수준)

"먼저 CI 40분 중 어느 계층의 테스트가 시간을 지배하는지부터 확인하겠습니다. 대개 이런 상황은 E2E나 통합 테스트에 로직 검증까지 몰려 있는 역피라미드 구조에서 발생합니다. 브라우저나 실제 DB를 띄우지 않아도 검증 가능한 로직은 unit test로 내려서 다시 작성하고, E2E는 로그인·결제 같은 핵심 사용자 흐름 몇 개로만 좁힙니다.

Flaky test는 우연으로 넘기지 않고, CI 히스토리에서 재실행 시 결과가 바뀐 테스트를 자동으로 태깅해 추적합니다. 확인된 flaky test는 즉시 별도 quarantine 스위트로 옮겨 merge 게이트에서 제외하되, 담당자와 2주 기한을 명시적으로 부여해서 방치되지 않게 관리합니다. 원인은 대부분 비동기 타이밍, 외부 의존성, 테스트 간 상태 공유 세 가지 중 하나이므로 이 순서로 진단합니다.

커버리지 90% 강제 같은 정책은 절대 수치 대신 diff coverage로 바꾸는 것을 제안합니다. 이번 PR에서 변경된 코드만 일정 커버리지를 요구하고, 레거시 코드 전체를 갑자기 끌어올리라는 압박은 제거합니다. 동시에 코드 리뷰에서 assertion이 실질적으로 검증하는지도 함께 확인해서, 숫자만 채우는 테스트를 걸러냅니다.

품질 게이트는 hard gate와 soft gate로 나눕니다. 신규 코드의 diff coverage 미달이나 핵심 회귀 테스트 실패는 merge를 막고, 전체 저장소의 절대 커버리지 추세나 성능 경고는 알림만 주는 soft gate로 둡니다. 마지막으로 테스트를 shard로 나눠 병렬 실행하고 의존성 캐싱을 적용해서, 계층을 재조정한 뒤에도 CI가 빠르고 신뢰할 수 있는 상태를 유지하도록 만들겠습니다."

🔺 Senior-Level Follow-up 질문

Q. "커버리지가 90%인데도 프로덕션 버그가 자주 발생합니다. 왜 그럴까요?"
사고 방향: 커버리지는 "실행됨"을 측정하지 "검증됨"을 측정하지 않는다는 점을 먼저 짚어야 한다.

핵심 답변: 실행 커버리지가 높아도 assertion이 약하면(예: 예외만 안 나면 통과) 실제 로직 오류를 잡지 못한다. 또한 커버리지는 경계값·예외 흐름·동시성 문제 같은 케이스를 강제하지 않으므로, happy path만 반복 검증했을 가능성이 크다. 개선책으로는 mutation testing(코드를 일부러 변형해 테스트가 실패로 잡아내는지 확인)으로 assertion 품질을 검증하거나, 코드 리뷰에서 "이 테스트가 실패하는 경우를 상상할 수 있는가"를 체크리스트로 넣는 방법을 제시한다.
Q. "Flaky test를 quarantine하면 팀이 고치지 않고 방치하는 경향이 생깁니다. 어떻게 막으시겠습니까?"
사고 방향: quarantine 자체가 해결책이 아니라 임시 조치라는 것을 분명히 하고, 방치를 막는 강제 장치를 설계해야 한다.

핵심 답변: quarantine에 들어가는 순간 담당자와 기한을 자동으로 배정하고, 대시보드에 quarantine 목록과 경과 일수를 상시 노출한다. 기한을 넘긴 테스트는 팀 스탠드업이나 주간 리포트에서 자동으로 언급되게 하고, 일정 기간(예: 한 달) 이상 방치되면 테스트를 완전히 삭제할지 팀이 명시적으로 결정하도록 강제한다. "quarantine = 언젠가 고침"이 아니라 "quarantine = 만료 시한이 있는 부채"로 취급한다는 점이 핵심이다.
Q. "E2E 테스트가 너무 느려서 매 PR마다 돌리기 부담스럽습니다. 어떻게 하시겠습니까?"
사고 방향: "E2E를 없앤다"가 아니라 실행 시점과 범위를 재설계하는 방향으로 답한다.

핵심 답변: 모든 E2E를 매 PR마다 돌리는 대신, 핵심 사용자 흐름(로그인, 결제 등) 소수만 PR 게이트에 남기고 나머지는 merge 이후 스테이징 환경에서 비동기로, 혹은 야간 배치로 실행한다. 병렬 실행(shard)과 헤드리스 브라우저 캐싱으로 개별 실행 시간을 줄이고, PR 게이트용 스위트와 회귀 검증용 전체 스위트를 분리해서 "빠른 피드백"과 "폭넓은 검증"이라는 서로 다른 목적을 각각 만족시킨다.
Q. "조직이 커지고 서비스가 늘어날수록 CI 시간이 계속 길어집니다. 장기적으로 어떻게 관리하시겠습니까?"
사고 방향: 일회성 최적화가 아니라 CI 시간을 지속적으로 관리하는 프로세스를 만드는 방향으로 답해야 한다.

핵심 답변: 테스트 실행 시간을 추적하는 대시보드를 두고, 일정 임계값(예: 특정 테스트가 전체 시간의 5% 이상)을 넘는 테스트는 자동으로 플래그되어 소유 팀이 최적화하도록 한다. 변경된 파일의 의존 그래프를 기반으로 영향받는 테스트만 선택 실행하는 test selection을 도입해, 저장소가 커져도 PR당 실행 시간이 서비스 전체 크기에 비례해 늘지 않도록 한다. 새로운 테스트를 추가할 때 어느 계층에 속하는지 가이드라인으로 강제해서, 역피라미드가 다시 쌓이는 것을 구조적으로 예방한다.

⚖️ 좋은 답변과 약한 답변의 차이

✅ 좋은 답변
  • 테스트 계층 비율(피라미드/역피라미드)부터 진단하고 재조정 방향을 제시
  • Flaky test를 진단 가능한 원인(레이스 컨디션, 외부 의존성 등)으로 분해
  • 커버리지를 목표가 아니라 diff coverage 같은 실질적 신호로 재정의
  • Merge를 막는 기준과 경고만 하는 기준을 명시적으로 분리
  • CI 시간 최적화를 병렬화·캐싱·test selection 등 구체적 기법으로 설명
  • Quarantine에 소유자와 기한을 부여해 방치를 구조적으로 방지
❌ 약한 답변
  • "테스트를 더 많이 짜면 됩니다" — 어느 계층인지 구분 없음
  • Flaky test는 "그냥 재실행하면 된다"로 넘김
  • 커버리지 %를 절대적 품질 지표로 착각
  • 모든 실패를 무조건 merge 차단으로 처리해 개발 속도 저하
  • CI가 느린 이유를 "테스트가 많아서"로만 설명하고 계층 분석 없음
  • Quarantine한 테스트를 사실상 영구 제외로 방치
💡 핵심 차이: 좋은 답변은 CI를 "신뢰할 수 있는 신호 시스템"으로 다룬다. 약한 답변은 CI를 "통과시켜야 할 관문"으로만 다룬다.

🏭 실무 연결 — 이런 상황에서 이렇게 설계

상황별 테스트 자동화 가이드
상황설계 포인트이유
CI가 40분 넘게 걸린다 느린 계층(E2E/통합)에 몰린 로직 검증을 unit test로 이관 + 병렬 shard 실행 계층 재조정과 병렬화로 근본적인 시간 단축
재실행하면 통과하는 테스트가 반복된다 Flaky 태깅 → quarantine(소유자/기한 부여) → 원인 수정 후 재통합 신호 신뢰도 회복, 방치 방지

아래 기준으로 자신의 답변 수준을 체크하라.

레벨 할 수 있는 것 부족한 것
Lv 1 unit test, 통합 test, E2E test를 각각 작성할 수 있다 계층 비율, flaky test 진단, 커버리지 지표 비판력 없음
Lv 2 테스트 피라미드 개념을 알고 CI에 커버리지 임계값을 설정할 수 있다 Flaky test를 체계적으로 추적/격리하는 프로세스 부족
Lv 3 Flaky test를 quarantine하고, diff coverage 같은 실질적 지표를 적용한다 Hard/soft gate 분리, test selection 등 조직 규모 대응 경험 부족
Lv 4 (목표) 계층 재조정 + flaky 격리·재통합 프로세스 + 지표 비판력 + gate 분리 설계 + CI 시간을 조직 성장에도 일정하게 유지하는 구조 설계 —

셀프 체크리스트

  • ☐ 역피라미드가 왜 CI를 느리고 불안정하게 만드는지 설명할 수 있는가
  • ☐ Flaky test의 흔한 원인 네 가지를 구분해서 진단할 수 있는가
  • ☐ 커버리지 %가 보장하는 것과 보장하지 않는 것을 명확히 구분할 수 있는가
  • ☐ Diff coverage가 절대 커버리지보다 나은 이유를 설명할 수 있는가
  • ☐ Merge를 막는 기준과 경고만 하는 기준을 나눠서 설계할 수 있는가
  • ☐ CI 시간을 조직 성장에도 일정하게 유지하는 구조(병렬화, test selection)를 그릴 수 있는가

🚫 흔한 오해와 교정

오해 1

"테스트 커버리지가 높을수록 코드 품질이 좋다."

→ 커버리지는 "실행됨"을 측정할 뿐 "검증됨"을 측정하지 않는다. Assertion이 약한 테스트로 채운 높은 커버리지는 실제 결함 검출력이 낮을 수 있다. 커버리지는 신호일 뿐, 목표 자체가 되면 오히려 무의미한 테스트를 양산한다.

오해 2

"E2E 테스트가 가장 신뢰할 수 있는 테스트다. 많을수록 좋다."

→ E2E는 실제 사용자 흐름을 검증한다는 장점이 있지만, 환경 의존성이 커서 가장 깨지기 쉽고(flaky) 실행도 가장 느리다. 로직 검증은 대부분 unit test로 충분하며, E2E는 핵심 흐름 소수에만 집중하는 것이 CI 안정성과 속도 모두에 유리하다.

오해 3

"Flaky test는 재실행하면 되니까 큰 문제가 아니다."

→ 재실행으로 통과시키는 습관이 굳어지면, 진짜 회귀 버그가 발생해도 "또 flaky겠지"라며 같은 방식으로 무시하게 된다. Flaky test의 진짜 위험은 실패 자체가 아니라 팀이 CI 실패를 신뢰하지 않게 되는 문화적 침식이다.

오해 4

"모든 테스트 실패는 무조건 merge를 막아야 한다."

→ 모든 실패를 동일한 무게로 다루면, 사소한 경고성 실패까지 개발 속도를 저해한다. 실제 프로덕션 리스크와 직결되는 실패만 hard gate로 막고, 나머지는 추적 가능한 soft gate로 낮추는 것이 지속 가능한 품질 관리다.
