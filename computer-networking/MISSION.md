# 네트워크 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 개발자 기술 면접에서 네트워크 지식을 단순 암기가 아니라 **장애 분석, 성능 진단, 보안 판단, 아키텍처 의사결정**으로 연결해 설명할 수 있도록 훈련한다.

면접 답변은 "OSI 7계층을 외운다"가 아니라, 실제 서비스에서 지연, 패킷 손실, TLS 오류, DNS 장애, 로드밸런서 설정 문제, 클라우드 네트워크 병목이 발생했을 때 어떤 순서로 원인을 좁히고 어떤 trade-off로 해결할지 보여주는 방향으로 만든다.

## 평가 대상 역량

| 역량 | 면접에서 확인하려는 것 |
|------|------------------------|
| 네트워크 기본기 | OSI/TCP-IP 계층, 캡슐화, 주소 체계, 포트, 프로토콜 역할을 정확히 설명하는가 |
| 트러블슈팅 | DNS, TCP, TLS, HTTP, 라우팅 문제를 계층별로 분해해 추적하는가 |
| 성능 판단 | RTT, 대역폭, 혼잡 제어, 커넥션 재사용, keep-alive, HTTP/2/3 선택 근거를 말하는가 |
| 신뢰성 | timeout, retry, backoff, circuit breaker, load balancing을 네트워크 관점에서 설명하는가 |
| 보안 | TLS, 인증서 체인, 방화벽, NAT, VPN, zero trust의 실무 의미를 이해하는가 |
| 클라우드 네트워크 | VPC, subnet, route table, security group, NAT gateway, CDN을 운영 관점에서 다루는가 |
| 커뮤니케이션 | 장애 상황을 개발자, SRE, 보안팀, 비즈니스 담당자에게 각각 다른 언어로 설명할 수 있는가 |

## Interview 레슨 작성 규칙

각 Day 레슨은 반드시 **면접 질문 중심**으로 작성한다.

1. **면접 질문**
   - 10년차 이상 개발자에게 실제로 물을 법한 질문으로 시작한다.
   - 예: "사용자는 API가 느리다고 하는데 서버 CPU는 정상입니다. 네트워크 관점에서 무엇부터 확인하시겠습니까?"
2. **전제 개념**
   - 어려운 개념은 먼저 쉬운 한국어로 설명한다.
   - 전문 용어를 바로 던지지 말고, 왜 그 개념이 필요한지 일상적인 비유나 간단한 흐름으로 풀어쓴다.
3. **왜 면접에서 묻는가**
   - 이 질문이 기본기 확인인지, 장애 대응 경험 확인인지, 설계 판단력 확인인지 명확히 적는다.
4. **실무 연결**
   - 실제 운영 상황, 장애 사례, 성능 병목, 보안 사고, 클라우드 구성과 연결한다.
5. **답변 사고 순서**
   - 답변을 어떻게 구조화해야 하는지 단계별로 제시한다.
   - 예: 증상 정의 -> 계층 분리 -> 가설 수립 -> 측정 도구 선택 -> 해결안 비교 -> 재발 방지.
6. **핵심 답변 예시**
   - 면접에서 바로 말할 수 있는 수준의 답변을 한국어로 제공한다.
   - 짧은 답변과 깊이 있는 답변을 구분한다.
7. **Trade-off**
   - 성능, 비용, 복잡도, 보안, 운영 난이도 사이의 절충을 반드시 설명한다.
8. **흔한 오해**
   - 경력자도 자주 틀리는 설명, 과도한 일반화, 암기식 답변의 한계를 짚는다.
9. **Follow-up 질문**
   - 면접관이 더 깊게 물을 질문과 그 질문의 의도를 포함한다.
10. **10년차 이상 답변 기준**
   - 단순 정답이 아니라 실무 판단, 운영 경험, 장애 대응 사고가 드러나는지 평가한다.

## 완료 기준

- 네트워크 개념을 계층별로 외우는 데서 끝내지 않고 장애 원인 분석 순서로 재구성할 수 있다.
- 프로토콜 선택과 설정 변경의 trade-off를 말할 수 있다.
- "어떤 도구로 무엇을 확인할 것인가"를 구체적으로 답변할 수 있다.
- 약한 답변과 강한 답변의 차이를 스스로 설명할 수 있다.
