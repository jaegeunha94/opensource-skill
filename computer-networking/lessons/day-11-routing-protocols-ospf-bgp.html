1. 왜 동적 라우팅이 필요한가?

Day 3에서 라우팅 테이블과 Longest Prefix Match를 배웠다. 당시에는 관리자가 직접 경로를 설정하는 정적 라우팅(static routing)을 다뤘다. 소규모 네트워크에서는 충분하지만, 수천 개의 라우터가 서로 연결된 인터넷에서는 수동 관리가 불가능하다.

핵심 질문
라우터 A가 라우터 Z까지 가는 경로가 두 가지인데, 중간 링크 하나가 끊어졌을 때 어떻게 자동으로 우회 경로를 찾을 수 있을까?

동적 라우팅 프로토콜은 라우터들이 서로 경로 정보를 교환하여 네트워크 변화에 자동으로 적응한다. 동적 라우팅 프로토콜은 사용 범위에 따라 두 가지로 나뉜다:

구분 풀네임 사용 범위 대표 프로토콜
IGP Interior Gateway Protocol 자율 시스템(AS) 내부 OSPF, RIP, EIGRP
EGP Exterior Gateway Protocol 자율 시스템(AS) 간 BGP

자율 시스템(AS)이란?

자율 시스템(Autonomous System, AS)은 단일 관리 정책 하에 운영되는 라우터 집합이다. KT, SK브로드밴드, AWS, Google 각각이 하나의 AS를 운영한다. 각 AS에는 고유한 AS 번호(ASN)가 부여된다(예: AS9318은 SK브로드밴드).

인터넷 구조 (개념도) ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ AS 9318 (SKT) │ │ AS 7018 (AT&T) │ │ AS 16509 (AWS) │ │ │ │ │ │ │ │ R1 ─── R2 │BGP │ R5 ─── R6 │BGP │ R9 ─── R10 │ │ │ │ │◄──►│ │ │ │◄──►│ │ │ │ │ R3 ─── R4 │ │ R7 ─── R8 │ │ R11 ─── R12 │ │ [OSPF 사용] │ │ [OSPF 사용] │ │ [OSPF 사용] │ └──────────────────┘ └──────────────────┘ └──────────────────┘ IGP (내부) IGP (내부) IGP (내부) ←────────────────────── BGP (AS 간 경계) ───────────────────────►
요약
AS 내부는 OSPF 같은 IGP로 빠르게 경로를 교환하고, AS 경계에서는 BGP로 인터넷 전체의 경로 정보를 교환한다.

2. OSPF — 링크 상태 라우팅

OSPF(Open Shortest Path First)는 AS 내부에서 사용하는 링크 상태 기반 라우팅 프로토콜이다. "Open"은 오픈 표준(RFC 2328 / RFC 5340)을 의미한다.

링크 상태 vs 거리 벡터

방식 원리 정보 공유 대상 대표 프로토콜 수렴 속도
링크 상태 전체 네트워크 지도를 그리고 Dijkstra 최단 경로 계산 모든 라우터에 동일한 LSDB OSPF, IS-IS 빠름
거리 벡터 이웃에게 "나는 X까지 N홉"이라고 알림 인접(neighbor) 라우터에만 공유 RIP, EIGRP 느림 (수렴 문제)

OSPF 핵심 개념

LSA (Link State Advertisement)

각 라우터가 자신과 연결된 링크 정보를 광고하는 패킷. "나는 라우터 A이고, B와 비용 10으로 연결, C와 비용 5로 연결"을 전체 네트워크에 전파한다.

LSDB (Link State Database)

수집한 모든 LSA를 저장하는 데이터베이스. 같은 영역(Area) 내의 모든 라우터는 동일한 LSDB를 갖는다. 이 지도를 바탕으로 최단 경로를 계산한다.

SPF 알고리즘 (Dijkstra)

LSDB(전체 지도)를 바탕으로 자신을 루트로 하는 최단 경로 트리를 계산한다. 비용(Cost)은 인터페이스 대역폭에 반비례한다(10Gbps < 1Gbps < 100Mbps 순서로 Cost 증가).

OSPF 동작 과정 (단계별)

  1. 이웃 탐색 (Neighbor Discovery)
    라우터가 Hello 패킷을 멀티캐스트(224.0.0.5)로 전송하여 OSPF 이웃을 발견한다.
  2. 인접 관계 수립 (Adjacency Formation)
    같은 링크의 라우터들이 Hello 매개변수를 교환하고 Master/Slave 역할을 결정한다. 멀티액세스 네트워크에서는 DR(Designated Router)과 BDR(Backup DR)을 선출하여 LSA 교환을 최소화한다.
  3. LSDB 동기화 (Database Exchange)
    인접 라우터끼리 DBD(Database Description) 패킷을 교환하고 상대가 모르는 LSA를 요청(LSR)하여 수신(LSU)한다.
  4. SPF 계산 (Route Calculation)
    LSDB가 완성되면 Dijkstra 알고리즘으로 자신을 루트로 하는 최단 경로 트리를 계산하여 라우팅 테이블에 반영한다.
  5. 변화 감지 및 재계산 (Convergence)
    링크가 끊어지면 해당 라우터가 새 LSA를 플러딩(flooding)하고, 모든 라우터가 SPF를 재계산한다. RIP보다 수렴 속도가 훨씬 빠르다.

OSPF Area와 계층 구조

대규모 네트워크에서 LSDB가 너무 커지면 SPF 계산 부하가 증가한다. OSPF는 Area로 네트워크를 분할하여 이 문제를 해결한다.

OSPF Area 구조 ┌─────── Area 0 (Backbone) ───────┐ │ │ Area 1 ─── ABR1 ─── R_backbone ─── ABR2 ─── Area 2 (일반 Area) │ │ (일반 Area) └────────────────────────┘ ABR (Area Border Router): 두 Area에 동시에 속하는 라우터 Area 0: 백본 영역 (모든 Area가 연결) Area 1, 2: 일반 영역 (내부 LSA를 Area 0으로만 요약 전송)
  • Area 0 (Backbone Area): 모든 OSPF Area의 중심. 다른 Area는 반드시 Area 0을 통해 연결된다.
  • ABR (Area Border Router): 두 Area에 걸쳐 있는 라우터. 각 Area의 LSA를 요약(Summary LSA)하여 다른 Area로 전달.
  • Area를 나누면 각 Area 내부의 LSDB 크기가 줄어 SPF 계산이 가벼워진다.
주의
OSPF의 Cost는 기본적으로 100Mbps / 인터페이스 대역폭으로 계산된다. 기가비트 이상 링크에서는 모두 Cost 1이 되어 구분이 안 되므로, 고속 환경에서는 레퍼런스 대역폭을 높게 재설정해야 한다.

3. BGP — 인터넷의 라우팅 프로토콜

BGP(Border Gateway Protocol, RFC 4271)는 AS 간 경로를 교환하는 EGP다. 현재 버전은 BGP-4로, 인터넷의 "지도"를 그리는 유일한 프로토콜이다.

BGP의 특징
BGP는 최단 경로가 아니라 정책 기반으로 경로를 선택한다. AS 간에는 비용뿐 아니라 비즈니스 관계(피어링, 트랜짓 비용)가 경로 결정에 영향을 미친다.

iBGP vs eBGP

구분 전체 이름 사용 범위 특징
eBGP External BGP 서로 다른 AS 사이 AS 경계 라우터(ASBR) 간 연결. TTL=1로 직접 연결만 허용(기본값)
iBGP Internal BGP 같은 AS 내부 AS 내의 BGP 라우터 간 전파. Full-mesh 또는 Route Reflector 사용

BGP 세션 수립 과정

BGP는 TCP 179 포트를 사용하여 신뢰성 있는 세션을 수립한다. OSPF처럼 자동 탐색이 아니라, 관리자가 피어(Peer)를 명시적으로 설정해야 한다.

  1. TCP 연결 수립
    BGP 피어가 TCP 3-way handshake로 연결을 맺는다 (포트 179).
  2. OPEN 메시지 교환
    AS 번호, BGP 버전, Hold Time, BGP 라우터 ID를 교환한다. Hold Time이 합의되면 Keepalive를 해당 시간 내에 보내지 않으면 세션이 종료된다.
  3. UPDATE 메시지 교환
    경로 정보(NLRI: Network Layer Reachability Information)와 경로 속성을 교환한다. 새 경로 광고와 철회(Withdraw)가 여기서 이루어진다.
  4. KEEPALIVE 메시지
    세션을 유지하기 위해 주기적으로 교환하는 빈 메시지 (Hold Timer 리셋).
  5. NOTIFICATION 메시지
    오류 발생 시 전송 후 세션 종료. 오류 코드와 세부 코드로 원인을 알린다.

BGP 핵심 속성 (Path Attributes)

BGP는 경로 선택을 위해 여러 경로 속성(Path Attribute)을 사용한다. 관리자는 이 속성을 조작해 트래픽 경로를 제어한다.

AS-PATH

해당 경로가 거쳐온 AS 번호의 순서 목록. 예: AS-PATH: 7018 9318 4134는 AS7018 → AS9318 → AS4134를 경유한다는 뜻. 루프 방지에 사용되며, 일반적으로 AS-PATH가 짧을수록 선호된다.

NEXT-HOP

해당 경로에 도달하기 위해 다음에 보내야 할 라우터의 IP. eBGP에서는 피어의 IP, iBGP에서는 eBGP 세션 엔드포인트의 IP가 설정된다.

LOCAL_PREF (로컬 선호도)

AS 내부에서만 사용하는 속성(iBGP). 값이 높을수록 선호. 여러 eBGP 피어를 통해 같은 목적지에 도달할 수 있을 때, 어떤 출구를 선택할지 결정한다. (기본값 100)

MED (Multi-Exit Discriminator)

상대 AS에게 "우리 네트워크에 들어올 때 이 입구를 선호해 달라"고 힌트를 주는 속성. 값이 낮을수록 선호. LOCAL_PREF와 반대 방향으로 영향을 미친다.

BGP 경로 선택 알고리즘 (Best Path Selection)

같은 목적지로 가는 경로가 여러 개일 때 BGP는 다음 순서로 최적 경로를 선택한다:

1. Weight (Cisco 전용, 값이 높을수록 선호)
2. LOCAL_PREF (값이 높을수록 선호, AS 내부 정책)
3. locally originated (자신이 직접 생성한 경로 선호)
4. AS-PATH 길이 (짧을수록 선호)
5. ORIGIN type (IGP > EGP > incomplete)
6. MED (값이 낮을수록 선호)
7. eBGP > iBGP
8. 가장 낮은 IGP metric to NEXT-HOP
9. 가장 낮은 BGP Router ID (tie-breaker)
실무 팁
실제 운영에서 경로를 제어할 때는 대부분 LOCAL_PREF(아웃바운드 제어)와 AS-PATH prepending(인바운드 제어)을 사용한다. MED는 같은 AS에서 온 경로에만 비교되어 신뢰하기 어렵다.

BGP 보안 이슈 — 라우트 하이재킹

BGP는 경로 광고를 인증하는 메커니즘이 원래 없었다. 이로 인해 BGP 하이재킹(Route Hijacking)이 발생할 수 있다.

BGP 하이재킹 사례 (2010년 중국 China Telecom) [정상 상황] AS16509(AWS) 192.0.2.0/24 광고 → 전 세계 인터넷이 AWS로 트래픽 전송 [하이재킹] AS4134(China Telecom)가 192.0.2.0/24를 자신이 갖고 있다고 허위 광고 → 더 구체적인 prefix 또는 짧은 AS-PATH로 인해 일부 트래픽이 잘못 라우팅 → 중국을 경유하거나 블랙홀에 빠질 수 있음

현재는 RPKI(Resource Public Key Infrastructure)를 통해 AS가 특정 IP prefix를 광고할 권한이 있는지를 암호학적으로 검증하는 방식으로 대응한다.

5. 실전 시나리오 — 트래픽 경로 추적

한국에서 AWS 서버(54.230.x.x)에 접속할 때 라우팅이 어떻게 이루어지는지 추적해 보자.

  1. 내 PC → 기업/가정 라우터
    기본 게이트웨이로 패킷이 전달된다. 내 PC의 라우팅 테이블에는 0.0.0.0/0 → 게이트웨이 경로가 있다.
  2. 게이트웨이 → ISP (예: SKT, AS9318)
    ISP 라우터가 BGP 테이블에서 54.230.0.0/15(AWS prefix)를 찾아 다음 홉을 결정한다.
  3. ISP 내부 라우팅 (OSPF)
    SKT 내부에서 OSPF가 패킷을 국제 관문 라우터(ASBR)까지 전달한다.
  4. 국제 BGP 피어링
    한국 ISP의 ASBR이 eBGP로 연결된 AWS(AS16509) 또는 중간 경유 AS의 ASBR에 패킷을 전달한다. 경로는 BGP 정책에 따라 결정된다.
  5. AWS 내부 (OSPF 또는 사내 라우팅)
    AWS 내부에서 목적지 서버까지 최종 전달. EC2의 Virtual Private Cloud(VPC) 라우팅 테이블을 통해 도달한다.

traceroute로 AS 경계 확인하기

# Linux/macOS
traceroute 54.230.0.1

# 출력 예시
 1  192.168.1.1      (내 게이트웨이)
 2  10.x.x.x         (ISP 내부 — OSPF 경유)
 3  203.x.x.x        (ISP 국내 백본)
 4  202.x.x.x        (ISP 국제 관문 — eBGP 경계)
 5  54.239.x.x       (AWS 네트워크 진입)
 6  54.230.0.1       (목적지 도달)

# whois로 AS 확인
whois -h whois.cymru.com " -v 54.230.0.1"
# → AS16509 | Amazon.com, Inc.
실습 포인트
traceroute 출력에서 홉마다 IP가 크게 바뀌거나 지연이 급증하는 지점이 AS 경계(eBGP 피어링 포인트)일 가능성이 높다.

6. 이해도 확인 — 퀴즈

Q1. OSPF가 링크 상태를 전달하는 패킷 단위는?
  • ① BGP UPDATE
  • ② LSA (Link State Advertisement)
  • ③ RIP Response
  • ④ LSDB Packet
정답 보기
② LSA. OSPF에서 각 라우터는 자신의 링크 상태를 LSA 형태로 플러딩(flooding)한다. LSDB는 수집된 모든 LSA를 담는 데이터베이스이지, 패킷 단위가 아니다. BGP UPDATE는 BGP 전용 패킷이다.
Q2. BGP에서 AS 간 루프를 방지하는 속성은?
  • ① LOCAL_PREF
  • ② MED
  • ③ AS-PATH
  • ④ NEXT-HOP
정답 보기
③ AS-PATH. BGP 라우터는 UPDATE를 수신할 때 AS-PATH에 자신의 AS 번호가 이미 포함되어 있으면 해당 경로를 무시(루프 방지)한다. LOCAL_PREF는 AS 내부 출구 선택, MED는 타 AS에 입구 힌트를 주는 속성이다.
Q3. 다음 중 OSPF가 경로를 선택하는 기준은?
  • ① AS-PATH 길이가 짧은 경로
  • ② 비즈니스 정책에 따른 선호도
  • ③ 인터페이스 대역폭 기반 Cost의 합이 작은 경로
  • ④ BGP Weight 값이 높은 경로
정답 보기
③ Cost의 합이 작은 경로. OSPF는 각 인터페이스의 Cost(= 레퍼런스 대역폭 / 인터페이스 대역폭)를 합산하여 최솟값인 경로를 Dijkstra 알고리즘으로 선택한다. AS-PATH는 BGP, BGP Weight도 BGP(Cisco 전용) 속성이다.
Q4. OSPF Area에 대한 설명으로 옳은 것은?
  • ① 모든 Area는 서로 직접 연결되어야 한다
  • ② Area 0은 백본 영역으로, 모든 다른 Area가 여기에 연결되어야 한다
  • ③ ABR은 하나의 Area에만 속할 수 있다
  • ④ Area를 나누면 LSDB 크기가 커진다
정답 보기
② Area 0이 백본. OSPF의 Area 0은 백본 영역이며 모든 다른 Area는 Area 0에 직접 또는 virtual link로 연결되어야 한다. ABR은 두 Area에 동시에 속한다. Area를 나누면 각 Area의 LSDB 크기가 줄어 SPF 계산 부하가 감소한다.

7. 핵심 개념 요약

용어 설명
AS (Autonomous System) 단일 관리 정책으로 운영되는 라우터 집합. 고유 ASN 부여.
IGP AS 내부 라우팅 프로토콜 (OSPF, RIP, EIGRP)
EGP AS 간 라우팅 프로토콜 (BGP)
LSA OSPF 링크 상태 광고 패킷
LSDB OSPF 전체 네트워크 지도 데이터베이스
SPF Dijkstra 알고리즘으로 최단 경로 트리 계산
DR / BDR 멀티액세스 네트워크에서 LSA 교환을 중앙화하는 Designated Router
ABR OSPF Area 경계 라우터 (두 Area 동시 소속)
eBGP 서로 다른 AS 간 BGP 세션
iBGP 같은 AS 내 BGP 라우터 간 세션
AS-PATH BGP 경로가 거쳐온 AS 번호 목록 (루프 방지 + 경로 길이 기준)
LOCAL_PREF AS 내 아웃바운드 경로 선택 선호도 (높을수록 선호)
MED 타 AS에게 인바운드 입구 힌트 (낮을수록 선호)
RPKI BGP 경로 원본 인증 — AS가 특정 prefix를 광고할 권한 검증
