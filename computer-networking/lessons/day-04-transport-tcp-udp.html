Day 4 — 전송 계층 : TCP vs UDP

학습 목표

- TCP와 UDP의 차이를 설명하고 적절한 사용 사례를 고를 수 있다.

- TCP 3-way handshake의 각 단계(SYN → SYN-ACK → ACK)를 순서대로 설명할 수 있다.

- TCP 흐름 제어(슬라이딩 윈도우)와 혼잡 제어의 역할을 구분할 수 있다.

- 포트 번호의 역할과 잘 알려진 포트를 안다.

1. 전송 계층이란?

Day 3의 네트워크 계층(IP)은 패킷을 목적지 호스트까지 전달하는 역할만 한다. 하지만 한 컴퓨터에서 웹 브라우저, 이메일 클라이언트, 게임이 동시에 네트워크를 사용하고 있다면 어떤 데이터가 어느 애플리케이션에 속하는지 구분해야 한다.

전송 계층(Transport Layer)은 이 프로세스 간 통신을 담당한다. OSI 4계층 / TCP/IP 전송 계층에 해당하며 핵심 프로토콜은 TCP와 UDP다.

역할설명
포트(Port) 번호로 다중화한 IP 주소 위에서 여러 애플리케이션을 구분
세그멘테이션(Segmentation)대용량 데이터를 작은 세그먼트로 분할하여 전송
신뢰성(선택)TCP는 손실/순서 오류 복구, UDP는 빠른 전달 우선

2. 포트 번호

포트(port)는 0~65535 범위의 16비트 숫자다. IP 주소가 건물 주소라면, 포트는 건물 내 층수·호수와 같다.

출발지 : 192.168.1.10 : 54321  (임시 포트 / Ephemeral port)
목적지 : 93.184.216.34 : 443   (HTTPS 서버)

소켓(Socket) — 네트워크 통신의 끝점. (IP 주소, 포트 번호, 프로토콜) 3-튜플로 표현한다. 예: TCP:192.168.1.10:54321. 두 소켓 쌍(5-튜플: 프로토콜, 출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)이 하나의 연결을 식별한다.

언제 UDP를 쓰는가? — 속도가 중요하고 약간의 손실을 허용할 수 있을 때. 영상 스트리밍 중 한 프레임이 빠지더라도 재전송 기다리는 것보다 다음 프레임을 빠르게 받는 편이 낫다. 게임도 마찬가지다. DNS 조회는 요청·응답이 단 한 번의 교환으로 끝나므로 TCP 핸드셰이크 비용을 지불할 필요가 없다.

4. TCP 세그먼트 구조

TCP는 데이터를 세그먼트(Segment) 단위로 전송하며, 각 세그먼트에는 20바이트 이상의 헤더가 붙는다.

0        8        16       24       32 (bit)
├────────────────┼────────────────────┤
│   Source Port  │  Destination Port  │
├────────────────┼────────────────────┤
│          Sequence Number            │
├─────────────────────────────────────┤
│        Acknowledgment Number        │
├────┬───────────┼────────────────────┤
│Data│ Reserved  │ Flags (SYN/ACK/..) │
│Offs│           │ URG ACK PSH RST    │
│    │           │ SYN FIN            │
├────┴───────────┼────────────────────┤
│   Window Size  │     Checksum       │
├────────────────┼────────────────────┤
│ Urgent Pointer │    Options (…)     │
└────────────────────────────────────┘
필드설명
Sequence Number이 세그먼트가 보내는 데이터의 첫 바이트 번호 (순서 보장·재조합용)
Acknowledgment Number다음에 받기를 기대하는 바이트 번호 (수신 확인)

Window Size수신자가 받을 수 있는 데이터 양 → 흐름 제어

5. TCP 3-way Handshake (연결 설정)

TCP 연결은 3단계 핸드셰이크로 수립된다. 클라이언트와 서버가 서로 시퀀스 번호를 교환하며 신뢰 있는 연결 채널을 만드는 과정이다.

클라이언트                          서버
    │                                │
    │ ──── SYN (seq=x) ────────────► │  [SYN_SENT]
    │                                │  [SYN_RECEIVED]
    │ ◄─── SYN-ACK (seq=y, ack=x+1) │
    │ [ESTABLISHED]                  │
    │ ──── ACK (ack=y+1) ──────────► │  [ESTABLISHED]
    │                                │
    │ ◄──────── 데이터 교환 ─────────► │
단계보내는 쪽플래그의미
1클라이언트 → 서버SYN"연결하고 싶어. 내 초기 시퀀스 번호는 x야."
2서버 → 클라이언트SYN + ACK"알겠어(x+1 받을게). 내 초기 시퀀스 번호는 y야."
3클라이언트 → 서버ACK"알겠어(y+1 받을게). 이제 연결됐어."

왜 3단계인가? — 양방향 통신을 열려면 각 방향마다 SYN + ACK가 필요하다. 1단계(클→서 SYN), 2단계(서→클 SYN·ACK 동시), 3단계(클→서 ACK)로 두 방향을 최소 3회에 처리한다. 2단계만으론 서버가 자신의 SYN에 대한 확인을 받지 못한다.

6. TCP 4-way Handshake (연결 종료)

연결 종료는 4단계가 필요하다. 각 방향이 독립적으로 종료되기 때문이다.

클라이언트                          서버
    │                                │
    │ ──── FIN ────────────────────► │  (클라이언트 송신 종료)
    │ ◄─── ACK ───────────────────── │
    │                                │  (서버 아직 데이터 보낼 수 있음)
    │ ◄─── FIN ───────────────────── │  (서버 송신 종료)
    │ ──── ACK ────────────────────► │
    │  [TIME_WAIT 2*MSL 후 CLOSED]   │  [CLOSED]

TIME_WAIT 상태 — 마지막 ACK가 손실될 경우 서버가 FIN을 재전송할 수 있도록 클라이언트가 일정 시간(2×MSL, 보통 60초~4분) 대기하는 상태. 이 기간 동안 동일한 포트를 재사용하면 이전 연결의 지연 패킷과 충돌할 수 있다.

7. 흐름 제어 — 슬라이딩 윈도우

흐름 제어(Flow Control)는 수신자가 처리할 수 있는 속도보다 빠르게 송신자가 데이터를 보내지 않도록 조절하는 메커니즘이다.

수신 윈도우(rwnd)

수신자는 TCP 헤더의 Window Size 필드에 현재 수신 버퍼의 여유 공간을 담아 ACK와 함께 보낸다. 송신자는 이 값만큼만 한 번에 보낼 수 있다.

송신자 버퍼                수신자 버퍼 (rwnd=3)
┌───┬───┬───┬───┬───┐    ┌───┬───┬───┐
│ 1 │ 2 │ 3 │ 4 │ 5 │    │ 1 │ 2 │ 3 │  ← 지금 채울 수 있는 공간
└───┴───┴───┴───┴───┘    └───┴───┴───┘
  ↑         ↑
보낼 수 있는 윈도우 (ACK 없이 동시에 3개까지)

         rwnd
  ├──────────────┤
  │  1  │  2  │  3  │  4  │  5  │
      ACK=1 받음 → 윈도우 슬라이드:
  │  2  │  3  │  4  │  5  │  6  │
            ├──────────────┤

수신자가 버퍼 부족으로 rwnd = 0을 보내면 송신자는 전송을 멈춘다. 이후 수신자가 버퍼를 비우면 Window Update 패킷을 보내 재개를 알린다.

8. 혼잡 제어 — 네트워크 병목 방지

혼잡 제어(Congestion Control)는 네트워크 중간의 라우터·링크가 과부하되지 않도록 송신 속도를 조절한다. 흐름 제어(수신자 버퍼)와 다르게 네트워크 전체의 상태를 대상으로 한다.

혼잡 윈도우(cwnd)와 Slow Start

전송 가능 데이터 = min(rwnd, cwnd)

Slow Start 단계:
  Round 1: cwnd = 1 MSS       → 1 세그먼트 전송
  Round 2: cwnd = 2 MSS       → 2 세그먼트 전송
  Round 3: cwnd = 4 MSS       → 4 세그먼트 (지수 증가)
  …                           → ssthresh(임계값) 도달 시 선형 증가(Congestion Avoidance)
알고리즘설명
Slow Start초기에는 cwnd를 1로 시작, ACK마다 1씩 증가(지수). ssthresh에 도달하면 선형 증가로 전환
Congestion AvoidanceRTT마다 cwnd += 1 MSS (선형 증가)
Fast Retransmit같은 ACK가 3회 연속(Triple Duplicate ACK) → 타임아웃 기다리지 않고 즉시 재전송
Fast RecoveryTriple Dup ACK 후 cwnd = ssthresh + 3, 선형 증가 재개 (타임아웃보다 덜 급격히 줄임)

AIMD(Additive Increase, Multiplicative Decrease) — 혼잡이 없으면 cwnd를 조금씩 늘리고(+1), 혼잡 감지(타임아웃)되면 절반으로 줄인다(×0.5). 이 원칙이 여러 TCP 흐름이 공정하게 대역폭을 나눠 쓰는 수학적 근거다.

9. UDP 세그먼트 구조

UDP 헤더는 단 8바이트(고정)다. 연결 설정도, 재전송도, 혼잡 제어도 없다.

0        8        16       24       32 (bit)
├────────────────┼────────────────────┤
│   Source Port  │  Destination Port  │
├────────────────┼────────────────────┤
│    Length      │     Checksum       │
├────────────────────────────────────┤
│                Data …               │
└────────────────────────────────────┘

체크섬(Checksum)만 있어 기본 오류 검출은 하지만, 오류 복구는 하지 않는다. 신뢰성이 필요하다면 애플리케이션 계층에서 직접 구현해야 한다(예: QUIC 프로토콜이 UDP 위에서 신뢰성을 추가한 사례).

UDP가 선호되는 실제 사례

애플리케이션이유
DNS 조회단순 요청-응답, 재전송이 필요하면 애플리케이션이 재질의
온라인 게임최신 위치 정보만 필요, 이전 데이터 재전송은 낭비
라이브 스트리밍·VoIP지연(latency) 최소화가 손실 복구보다 중요
DHCPIP 할당 전이므로 브로드캐스트 필요, TCP 연결 불가
QUIC (HTTP/3)UDP 위에 자체 신뢰성·암호화 구현, TCP의 Head-of-Line 차단 해결

10. 오늘의 핵심 정리

개념한 줄 요약
포트 번호한 IP 위에서 여러 애플리케이션을 구분하는 16비트 숫자.
TCP연결 지향·신뢰성·순서 보장. 오버헤드가 크지만 완전한 전달 보증.
UDP비연결·8바이트 헤더. 빠르지만 신뢰성은 애플리케이션 몫.
3-way handshakeSYN → SYN-ACK → ACK. 양방향 시퀀스 번호 교환으로 연결 수립.
흐름 제어수신자의 rwnd(Window Size)로 송신 속도 제한 → 버퍼 오버플로 방지.
혼잡 제어네트워크 내 라우터 과부하 방지. Slow Start → AIMD → Fast Retransmit.
TIME_WAIT연결 종료 후 클라이언트가 2×MSL 대기 → 지연 패킷·재전송 FIN 처리.
