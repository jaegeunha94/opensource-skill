Day 6 — 응용 계층 : HTTP/HTTPS

학습 목표

- HTTP 요청·응답 구조와 주요 메서드(GET, POST, PUT, DELETE)를 설명할 수 있다.

- HTTP 상태 코드 계열(1xx~5xx)과 자주 쓰이는 코드의 의미를 안다.

- 쿠키와 세션의 차이를 구분하고 각각이 언제 쓰이는지 설명할 수 있다.

- TLS 핸드셰이크의 단계별 흐름(ClientHello → ServerHello → 키 교환 → 완료)을 순서대로 설명할 수 있다.

1. HTTP란?

HTTP(HyperText Transfer Protocol)는 웹 브라우저와 서버 사이에서 데이터를 주고받는 응용 계층 프로토콜이다. Day 4에서 배운 TCP(포트 80) 위에서 동작하며, 요청(Request) / 응답(Response) 모델을 따른다.

HTTP는 무상태(Stateless) 프로토콜이다 — 서버는 각 요청을 독립적으로 처리하며, 이전 요청을 기억하지 않는다. 로그인 상태 유지 등 "기억"이 필요한 경우를 위해 쿠키와 세션이 등장한다.

HTTP/1.0

요청마다 새 TCP 연결. 매번 핸드셰이크 비용 발생.

HTTP/1.1

지속 연결(Keep-Alive) 기본. 파이프라이닝 지원. 현재도 널리 사용.

HTTP/2

바이너리 프레이밍. 멀티플렉싱(한 연결에 여러 스트림). 헤더 압축(HPACK).

HTTP/3

UDP 기반 QUIC 사용. TCP Head-of-Line 차단 해소. 더 빠른 연결 수립.

2. HTTP 요청 구조

GET /index.html HTTP/1.1          ← 요청 라인: 메서드, 경로, 버전
Host: www.example.com             ← 헤더
User-Agent: Mozilla/5.0
Accept: text/html
Accept-Encoding: gzip, deflate
Connection: keep-alive
                                  ← 빈 줄 (헤더 종료 구분자)
[본문(Body) — GET은 보통 없음]

요청 구성요소

구성요소설명예시
요청 라인메서드, URL 경로, HTTP 버전GET /api/users HTTP/1.1
헤더부가 메타데이터. 키:값 형태Content-Type: application/json
빈 줄헤더와 본문 구분. 필수
본문(Body)전송 데이터. POST/PUT에서 사용{"name": "Alice"}

4. HTTP 메서드

메서드의미본문멱등성주요 사용처
GET리소스 조회없음예웹 페이지 읽기, 데이터 검색
POST리소스 생성있음아니오회원 가입, 댓글 작성
PUT리소스 전체 교체있음예사용자 정보 전체 수정
PATCH리소스 일부 수정있음조건부이메일 주소만 변경
DELETE리소스 삭제없음(선택)예게시글 삭제
HEAD헤더만 조회 (본문 없음)없음예파일 크기 확인, 캐시 유효성 검사
OPTIONS지원 메서드 조회없음예CORS preflight 요청

멱등성(Idempotency) — 같은 요청을 여러 번 보내도 결과가 동일한 성질. GET·PUT·DELETE는 멱등. POST는 멱등하지 않음(요청마다 새 리소스가 생성될 수 있음). 네트워크 오류 시 재시도 로직 설계에 중요하다.

5. HTTP 상태 코드

상태 코드는 3자리 숫자로 응답의 결과를 나타낸다. 첫 자리가 계열을 구분한다.

계열의미자주 쓰이는 코드
1xx 정보 요청 수신, 처리 중 100 Continue — 요청 본문을 계속 보내도 됨
2xx 성공 요청 정상 처리 200 OK — 성공
201 Created — 리소스 생성됨
204 No Content — 성공, 응답 본문 없음
3xx 리다이렉션 다른 위치로 이동 필요 301 Moved Permanently — URL 영구 변경
302 Found — 임시 리다이렉트
304 Not Modified — 캐시 사용 가능
4xx 클라이언트 오류 잘못된 요청 400 Bad Request — 잘못된 문법
401 Unauthorized — 인증 필요
403 Forbidden — 권한 없음
404 Not Found — 리소스 없음
429 Too Many Requests — 요청 한도 초과
5xx 서버 오류 서버 측 문제 500 Internal Server Error — 서버 내부 오류
502 Bad Gateway — 게이트웨이 오류
503 Service Unavailable — 서버 과부하/점검
504 Gateway Timeout — 게이트웨이 타임아웃
401 vs 403 — 헷갈리는 차이

401 Unauthorized는 이름과 달리 인증(Authentication) 문제다. "누군지 모르겠어 — 로그인해줘." 응답에 WWW-Authenticate 헤더를 포함해야 한다.

403 Forbidden은 인가(Authorization) 문제다. "누군지는 알지만, 이 리소스에 접근할 권한이 없어." 로그인해도 허용되지 않는다는 뜻.

6. 쿠키(Cookie)와 세션(Session)

HTTP가 무상태(Stateless)이므로 로그인 유지, 장바구니 등 "상태"가 필요한 웹 기능은 쿠키와 세션으로 구현한다.

쿠키

서버가 Set-Cookie 헤더로 클라이언트 브라우저에 작은 데이터를 저장한다. 이후 같은 서버로 요청 시 Cookie 헤더에 자동으로 포함된다.

서버 → 클라이언트

HTTP/1.1 200 OK
Set-Cookie: session_id=abc123;
  HttpOnly;
  Secure;
  SameSite=Lax;
  Max-Age=3600

클라이언트 → 서버 (이후 요청)

GET /profile HTTP/1.1
Host: example.com
Cookie: session_id=abc123
쿠키 속성의미
HttpOnlyJavaScript(document.cookie)로 접근 불가 → XSS 공격 방지
SecureHTTPS 연결에서만 전송 → 평문 노출 방지
SameSite크로스사이트 요청 시 쿠키 전송 제어 → CSRF 공격 방지
Max-Age / Expires쿠키 유효 기간. 없으면 브라우저 닫을 때 삭제(세션 쿠키)
Domain / Path쿠키를 전송할 도메인/경로 범위 지정

세션

서버 측에 사용자 상태를 저장하고, 클라이언트에는 세션 ID만 쿠키로 전달하는 방식이다.

① 사용자 로그인 요청 (아이디·비밀번호)
      │
      ▼
② 서버가 인증 후 메모리/DB에 세션 저장: { session_id: "abc123", user: "Alice", expires: ... }
      │
      ▼
③ Set-Cookie: session_id=abc123; HttpOnly; Secure
      │
      ▼
④ 이후 요청마다 Cookie: session_id=abc123 자동 포함
      │
      ▼
⑤ 서버가 세션 저장소에서 "abc123" 찾아 사용자 상태 복원
쿠키세션
저장 위치클라이언트 브라우저서버 (메모리, Redis, DB)
보안클라이언트 조작 가능 (서명 없으면 위험)ID만 노출, 실제 데이터는 서버에 안전하게 보관
확장성서버 상태 없음 → 수평 확장 쉬움서버 공유 세션 저장소 필요 → Redis 등 활용
만료Max-Age/Expires 설정서버에서 명시적 관리

JWT(JSON Web Token) — 세션 저장소 없이 서버가 서명한 토큰을 클라이언트에 발급하는 방식. 토큰 자체에 사용자 정보가 담겨 있어 서버 상태가 필요 없다(무상태 인증). 그러나 토큰 취소(revocation)가 어렵다는 트레이드오프가 있다.

7. HTTPS와 TLS

HTTPS(HTTP Secure)는 HTTP에 TLS(Transport Layer Security)를 더한 것이다. 전송 데이터를 암호화하여 도청·위변조·피싱을 방지한다. TCP 포트 443을 사용한다.

보호 목표설명
기밀성(Confidentiality)대칭 암호화로 데이터를 암호화해 도청 방지
무결성(Integrity)MAC(메시지 인증 코드)으로 데이터 변조 감지
인증(Authentication)서버 인증서(X.509)로 접속 서버가 진짜임을 증명

TLS 버전

버전상태비고
SSL 2.0 / 3.0사용 금지취약점(POODLE 등) 다수 발견
TLS 1.0 / 1.1사용 지양2021년 RFC 8996으로 공식 폐기
TLS 1.2사용 중현재 가장 널리 배포
TLS 1.3권장핸드셰이크 1-RTT, 불필요한 알고리즘 제거, 더 빠르고 안전

8. TLS 핸드셰이크 (TLS 1.2 기준)

TLS 핸드셰이크는 암호화 통신을 시작하기 전 암호 알고리즘 협상, 서버 인증, 세션 키 교환을 수행하는 과정이다.

핸드셰이크 시뮬레이터

단계별로 TLS 1.2 핸드셰이크를 따라가 보자.

클라이언트
서버
──── ClientHello ────────────────────►
◄─── ServerHello ────────────────────
◄─── Certificate (서버 인증서) ───────
◄─── ServerHelloDone ────────────────
──── ClientKeyExchange ──────────────►
──── ChangeCipherSpec ───────────────►
──── Finished (암호화) ──────────────►
◄─── ChangeCipherSpec ───────────────
◄─── Finished (암호화) ──────────────
🔒 암호화된 HTTP 데이터 교환
TCP 연결이 수립된 직후 TLS 핸드셰이크가 시작된다. 다음 ▶을 눌러 각 단계를 살펴보자.

핸드셰이크 단계 상세

단계메시지내용
1ClientHello클라이언트가 지원하는 TLS 버전, 암호 스위트(Cipher Suite) 목록, 랜덤값 전송
2ServerHello서버가 선택한 TLS 버전·암호 스위트, 서버 랜덤값 전송
3Certificate서버가 공개키가 포함된 X.509 인증서 전송. 클라이언트는 CA 체인으로 서버 신원 검증
4ServerHelloDone서버 협상 완료 신호
5ClientKeyExchange클라이언트가 Pre-Master Secret(RSA 암호화 또는 DH 공개값) 전송
6~7ChangeCipherSpec + Finished클라이언트: "이제 암호화 시작". Finished 메시지를 협상된 키로 암호화하여 전송
8~9ChangeCipherSpec + Finished서버도 동일하게 응답. 양쪽 모두 세션 키 도출 확인
10암호화 HTTP이후 HTTP 데이터는 대칭키(AES-GCM 등)로 암호화되어 전송

TLS 1.3의 개선 — TLS 1.2는 핸드셰이크에 2-RTT가 필요하지만, TLS 1.3은 1-RTT로 단축했다(또는 세션 재개 시 0-RTT도 가능). 취약한 암호 알고리즘(RSA 키 교환, RC4, MD5 등)을 제거하고 Perfect Forward Secrecy를 필수화했다.

인증서 체인 (Certificate Chain)

루트 CA (예: DigiCert 루트, OS·브라우저에 내장)
  └── 중간 CA (예: DigiCert SHA2 Secure Server CA)
        └── 서버 인증서 (example.com)
              ├── 공개키 (서버의 RSA/ECDSA 공개키)
              ├── 발급 도메인 (CN, SAN)
              └── 서명 (중간 CA의 개인키로 서명)

클라이언트는 서버 인증서 → 중간 CA → 루트 CA 순서로 신뢰 체인을 검증한다. 루트 CA는 OS·브라우저에 내장되어 있어 신뢰 앵커 역할을 한다.

9. HTTP vs HTTPS 전체 요청 흐름

HTTPS 요청: https://www.example.com/page

① DNS 조회 (Day 5)
   www.example.com → 93.184.216.34

② TCP 3-way handshake (Day 4)
   포트 443으로 연결

③ TLS 핸드셰이크 (Day 6)
   암호 스위트 협상 + 서버 인증 + 세션 키 교환

④ HTTP 요청 (TLS 암호화)
   GET /page HTTP/1.1
   Host: www.example.com
   Cookie: session_id=...

⑤ HTTP 응답 (TLS 암호화)
   HTTP/1.1 200 OK
   Content-Type: text/html
   Set-Cookie: ...
   [HTML 본문]

⑥ (재방문) 브라우저 쿠키 → 서버가 세션 복원

10. 오늘의 핵심 정리

개념한 줄 요약
HTTP무상태 요청/응답 프로토콜. TCP 80. 텍스트 기반.
HTTP 메서드GET(조회), POST(생성), PUT(교체), PATCH(수정), DELETE(삭제). GET·PUT·DELETE는 멱등.
상태 코드2xx 성공, 3xx 리다이렉트, 4xx 클라이언트 오류, 5xx 서버 오류.
쿠키서버가 클라이언트에 저장하는 작은 데이터. HttpOnly·Secure·SameSite 속성으로 보안 강화.
세션상태를 서버에 저장, 클라이언트엔 ID만 쿠키로 전달. 서버 측 보안이 높음.
HTTPSHTTP + TLS. 기밀성·무결성·인증 세 가지를 보장. TCP 443.
TLS 핸드셰이크ClientHello → ServerHello·인증서 → 키 교환 → ChangeCipherSpec·Finished. 이후 대칭키 암호화.
