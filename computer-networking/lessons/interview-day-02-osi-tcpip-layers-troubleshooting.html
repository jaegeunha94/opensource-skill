1. 면접 질문

"OSI 7계층을 설명해보세요." — 라고 물은 뒤 면접관이 바로 이어서: "그런데 그게 실제 장애 분석에서 어떻게 쓰입니까? 어제 겪었던 네트워크 이슈를 계층 관점에서 설명해보세요."

앞부분("7계층을 나열하라")은 사실 미끼다. 대부분의 지원자가 여기서 OSI 계층 이름 암기 발표로 답을 끝낸다. 진짜 평가는 뒷부분에서 시작된다 — 계층 모델을 실제 원인 분리 도구로 쓸 수 있는가다. 10년차 이상에게는 "계층이 몇 개인가"가 아니라 "계층을 어떻게 무기로 쓰는가"를 묻는다.

2. 전제 개념 — 알아야 할 기초

2-1. OSI 7계층은 왜 만들어졌는가

1980년대에는 컴퓨터 제조사마다 통신 방식이 달라 서로 다른 장비끼리 통신이 안 됐다. OSI(Open Systems Interconnection)는 "통신 기능을 역할별로 나누면 어느 계층이든 표준만 지키면 서로 다른 회사 장비도 통신할 수 있다"는 아이디어에서 나온 참조 모델(reference model)이다. 즉 실제로 동작하는 프로토콜 스택이 아니라, 문제를 나눠서 생각하기 위한 사고의 틀이다.

비유: 택배가 늦게 온다고 하자. "포장 문제인가(물건을 담는 방식) vs 운송 문제인가(트럭 경로) vs 주소 문제인가(배송지 오기입) vs 수령인 문제인가(사람이 집에 없음)"를 나눠 생각하지 않으면 어디서부터 고쳐야 할지 모른다. OSI 계층은 이 "나눠서 생각하기"를 네트워크에 적용한 것이다.

2-2. 실무에서는 OSI 7계층이 아니라 TCP/IP 4~5계층으로 생각한다

교과서는 OSI 7계층(물리-데이터링크-네트워크-전송-세션-표현-응용)을 가르치지만, 실제 인터넷은 TCP/IP 4계층(링크-인터넷-전송-응용)으로 구현되어 있다. 세션/표현 계층은 실무에서 별도 계층으로 다뤄지지 않고 응용 계층 프로토콜(TLS, HTTP 등) 안에 흡수돼 있다. 면접에서 7계층을 순서대로 읊는 것보다, 실무에서 쓰는 4~5계층 매핑을 아는 것이 더 시니어다운 답변이다.

2-3. 캡슐화(Encapsulation)란 무엇이고 왜 장애 분석에 중요한가

데이터가 응용 계층에서 만들어져 물리 계층으로 내려갈 때마다 각 계층은 자기 헤더를 덧붙인다. HTTP 요청(응용 데이터)은 TCP 세그먼트가 되고, IP 패킷이 되고, 이더넷 프레임이 되어 전선을 탄다. 이 순서를 알아야 "어느 계층에서 문제가 생겼을 때 위/아래 계층이 각각 어떻게 보이는지"를 추론할 수 있다.

[HTTP 요청]
   ↓ TCP 헤더 추가 (포트, 시퀀스 번호)
[TCP 세그먼트]
   ↓ IP 헤더 추가 (출발지/목적지 IP, TTL)
[IP 패킷]
   ↓ 이더넷 헤더 추가 (출발지/목적지 MAC)
[이더넷 프레임]
   ↓ 전기/광 신호로 변환
[물리 신호]

예를 들어 MTU(최대 전송 단위)보다 큰 패킷은 IP 계층에서 단편화(fragmentation)되거나 드롭된다. 이건 TCP 계층에서는 "재전송이 계속된다"로만 보이고, 응용 계층에서는 "가끔 응답이 안 온다"로만 보인다. 같은 문제가 계층마다 다른 증상으로 보이기 때문에, 캡슐화 순서를 알아야 증상에서 원인 계층을 역추론할 수 있다.

2-4. "계층 분리(layer isolation)"라는 실무 사고 도구

장애 상황에서 "어느 계층까지는 정상이고 어느 계층부터 비정상인가"를 경계 짓는 작업이다. 예를 들어 ping이 성공하면 L3(네트워크 계층)까지는 정상이라는 뜻이다. telnet host 443으로 TCP 연결이 되면 L4까지 정상이다. 그런데 HTTPS 요청이 실패하면 문제는 L5~L7(TLS/HTTP) 어딘가에 있다는 뜻이다. 이렇게 아래 계층부터 하나씩 "여기까지는 정상"을 확인하며 위로 올라가는 것이 장애 분석의 기본 골격이다.

3. 왜 면접에서 이걸 묻는가

이 질문은 세 단계로 필터링한다:

  1. 1단계 필터 (교과서 암기 vs 이해) — 7계층 이름을 순서대로 말하는 것 자체는 통과 기준이 아니다. 이름을 말하면서 각 계층이 "왜" 그 역할을 맡는지 설명할 수 있는지 본다.
  2. 2단계 필터 (이해 vs 적용) — "그래서 실제 장애에서 어떻게 썼는가"를 물어서, 계층 모델을 실전에서 도구로 써봤는지 확인한다. 여기서 답을 못하면 이론만 아는 사람으로 분류된다.
  3. 3단계 필터 (적용 vs 사고 체계화) — follow-up으로 "그럼 어떤 계층부터 확인할지 순서를 어떻게 정합니까"를 물어, 즉흥적으로 도구를 쓰는 게 아니라 재사용 가능한 방법론을 가지고 있는지 본다.
면접관이 피하고 싶은 답변: "물리, 데이터링크, 네트워크, 전송, 세션, 표현, 응용 계층이 있습니다" 하고 끝내는 답변. 이건 위키백과를 읽은 사람과 구분이 안 된다. 10년차 면접에서는 이 다음 문장이 없으면 감점이다.

4. 실무 연결

실제 장애 시나리오 A — "핑은 되는데 사이트가 안 열려요"

고객사에서 "우리 서버에 ping은 잘 되는데 웹사이트가 안 열린다"고 신고했다. ping(ICMP, L3)이 성공했다는 것은 네트워크 경로와 IP 도달성은 정상이라는 뜻이다. 문제는 그 위 계층에 있다는 것을 바로 추론할 수 있었다. telnet host 443으로 확인하니 TCP 연결(L4)도 정상. 마지막으로 openssl s_client -connect host:443으로 확인하니 인증서 만료로 TLS 핸드셰이크(L5/L6)가 실패하고 있었다. 계층을 하나씩 배제하지 않았다면 "네트워크는 정상인데 왜 안 되지"에서 멈췄을 것이다.

실제 장애 시나리오 B — 사내망에서만 특정 서비스가 느림

사내 VPN을 통해 접속하면 외부 API 호출이 느리다는 보고가 있었다. L3(라우팅)는 traceroute로 정상 경로 확인. L4(TCP 연결)도 빠르게 맺어짐. 그런데 대용량 응답에서만 느려졌다. 원인은 VPN 터널의 MTU가 1400인데 서버가 1500 바이트 패킷을 그대로 보내 L3에서 단편화(fragmentation)가 발생하고 있었던 것. 이건 L2/L3 경계에서 생긴 문제가 L7(응용) 증상으로만 보인 전형적인 사례다.

실제 장애 시나리오 C — 컨테이너 배포 후 서비스 간 통신 실패

Kubernetes 클러스터에서 새 Pod를 배포했더니 다른 서비스와 통신이 안 됐다. L3(Pod IP 라우팅)까지는 CNI 플러그인에서 정상 확인. 문제는 L2에 가까운 계층 — NetworkPolicy(방화벽 규칙, 실질적으로 L3/L4 필터)가 새 네임스페이스의 트래픽을 차단하고 있었다. 클라우드 네이티브 환경에서도 결국 문제는 전통적인 계층 모델로 분해된다는 것을 보여주는 사례다.

5. 답변 사고 순서

계층 모델 질문에 답할 때는 아래 순서로 구조화한다. "계층을 나열"하는 것이 아니라 "계층을 진단에 쓰는 절차"를 보여주는 것이 핵심이다.

  1. 계층 모델을 실무 언어로 재정의
    OSI 7계층을 그대로 나열하지 않고, TCP/IP 4~5계층 + 각 계층의 실무 구성요소(스위치, 라우터, TCP, TLS, HTTP)로 바로 연결한다.
  2. "위에서 아래로" 혹은 "아래에서 위로" 접근 방향을 명시
    일반적으로는 아래에서 위로(bottom-up) 확인한다. 물리적으로 연결이 되는지(L1/L2) → IP가 도달하는지(L3) → 포트가 열려 있는지(L4) → 애플리케이션이 정상 응답하는지(L7). 각 계층에서 "정상"을 확인하면 다음 계층으로 넘어간다.
  3. 각 계층에서 쓰는 도구와 정상/비정상 기준을 구체적으로 제시 
    L1/L2: ip link show / ethtool eth0        → 링크 상태 UP, 에러 카운터 0
L3:    ping / traceroute / mtr             → 목적지 도달, 손실률 0%에 가까움
L4:    telnet host port / ss -tn           → 연결 성공, 재전송 없음
L7:    curl -v / openssl s_client          → TLS 핸드셰이크 성공, HTTP 상태 코드 정상
  4. 계층 경계에서 증상이 "번역"된다는 점을 설명
    L3 문제(예: MTU mismatch)가 L7에서는 "가끔 큰 응답만 느리다"로 보일 수 있다는 것을 명시적으로 언급해, 계층 간 증상 전이를 이해하고 있음을 보여준다.
  5. 실제 겪은 사례로 연결
    이론 설명 후 반드시 실제 사례 1개를 계층 순서에 따라 재구성해서 말한다. "핑은 됐지만 TLS 인증서 문제였다" 같은 구체적 경험이 신뢰도를 만든다.
  6. 이 사고방식이 왜 효율적인지 요약
    계층별 배제법을 쓰면 "가능성이 낮은 원인에 시간을 낭비하지 않고, 정상/비정상 경계를 빠르게 좁힐 수 있다"는 점을 강조한다.

6. 핵심 답변 예시

짧은 답변 (60초 버전)

"저는 OSI 7계층을 그대로 쓰기보다 TCP/IP 4~5계층으로 단순화해서 씁니다 — 링크, 네트워크, 전송, 응용. 장애가 나면 아래에서 위로 확인합니다. 먼저 ping으로 IP 도달성(L3)을 확인하고, telnet이나 curl로 포트 연결(L4)을 확인하고, 마지막에 TLS와 HTTP 응답(L7)을 봅니다. 예를 들어 지난번 '핑은 되는데 사이트가 안 열린다'는 장애는 L3까지는 정상이고 L7의 TLS 인증서 만료가 원인이었습니다. 계층별로 정상 구간을 좁혀가면 원인 후보를 빠르게 줄일 수 있습니다."

깊이 있는 답변 (면접관이 "더 자세히"라고 물을 때)

"OSI 7계층은 원래 서로 다른 벤더 장비가 상호운용되도록 만든 참조 모델이라, 실제 구현체인 TCP/IP 스택과 1:1로 대응하지 않습니다. 세션/표현 계층은 실무에서 TLS나 애플리케이션 프로토콜 안에 흡수돼 있다고 보는 게 맞습니다. 그래서 저는 실무에서는 링크-네트워크-전송-응용의 4계층으로 단순화해서 생각합니다. 장애 분석에서는 캡슐화 순서를 거꾸로 이용합니다. 데이터가 응용에서 물리로 내려가며 헤더가 쌓이기 때문에, 하위 계층에서 문제가 생기면 상위 계층에서는 전혀 다른 증상으로 보입니다. 예를 들어 MTU mismatch(L3)는 L7에서는 '큰 응답만 가끔 느리다'로 나타나고, TCP TIME_WAIT 고갈(L4)은 L7에서는 '연결이 간헐적으로 거부된다'로 나타납니다. 그래서 증상만 보고 바로 상위 계층을 의심하면 안 되고, 아래에서부터 순서대로 배제해야 합니다. 실제로 겪은 사례를 말씀드리면, 사내 VPN을 통한 외부 API 호출이 큰 응답에서만 느려진 적이 있습니다. traceroute로 L3 경로는 정상, telnet으로 L4 연결도 빨랐는데, tcpdump로 캡처해보니 VPN MTU(1400)보다 큰 패킷이 반복 재전송되고 있었습니다. 서버 측 MSS clamping으로 해결했습니다. 이 경험에서 배운 건, 계층 모델은 단순히 지식이 아니라 '어디까지 확인했으니 다음은 여기를 봐야 한다'는 순서를 주는 도구라는 점입니다."

7. Trade-off

선택장점단점 / 주의사항
Bottom-up 진단 (아래에서 위로) 확실한 사실부터 쌓아가므로 오판 가능성이 낮음. 물리/네트워크 계층은 확인이 빠르고 명확함 L7 문제(애플리케이션 로직 버그 등)일 때는 하위 계층 확인이 시간 낭비가 될 수 있음
Top-down 진단 (위에서 아래로) 사용자가 실제로 겪는 증상(HTTP 에러 코드 등)에서 바로 시작해 빠르게 후보를 좁힐 수 있음 하위 계층 문제(MTU, 라우팅)를 애플리케이션 버그로 오판할 위험이 있음
계층 모델을 엄밀하게 지키기 사고가 체계적이고 다른 사람에게 설명하기 쉬움 실제 프로토콜은 계층을 넘나듦(예: TCP가 라우팅 정보에 영향받음). 지나친 도식화는 실제 원인을 놓칠 수 있음
도구 기반 확인 (ping/traceroute/curl) 빠르고 대부분 상황에서 충분함 ICMP가 방화벽에서 차단되거나 우선순위가 낮게 처리되면 오판 가능 (Day 1의 오해 참고)

8. 흔한 오해

오해 1 "OSI 7계층은 실제 인터넷이 동작하는 방식이다"

OSI는 참조 모델(reference model)이지 구현체가 아니다. 실제 인터넷은 TCP/IP 스택으로 구현되어 있고 TCP/IP는 4~5계층으로 설명된다. 세션/표현 계층은 별도 프로토콜 계층으로 존재하지 않고 응용 계층 프로토콜(TLS, HTTP 등)에 기능이 흡수돼 있다. 면접에서 "OSI 7계층대로 인터넷이 동작한다"고 말하면 개념 이해가 얕다는 신호로 읽힌다.

오해 2 "계층은 항상 순서대로, 완전히 독립적으로 문제가 생긴다"

실무에서는 계층 경계가 깔끔하지 않다. 예를 들어 NAT(L3/L4 경계)는 IP와 포트를 동시에 바꾸고, 로드밸런서는 L4/L7을 동시에 다룬다(L4 LB vs L7 LB). "계층은 서로 완전히 독립적"이라는 전제로만 접근하면 NAT, 로드밸런서, 방화벽처럼 계층을 넘나드는 장비에서 헤맬 수 있다.

오해 3 "낮은 계층부터 확인하면 항상 가장 효율적이다"

모든 상황에서 bottom-up이 정답은 아니다. HTTP 500 에러처럼 애플리케이션 로그에 이미 원인이 찍혀 있는 경우, L1부터 순서대로 확인하는 것은 시간 낭비다. 계층 모델은 "증상이 애매하고 원인 후보가 넓을 때" 쓰는 방법론이지, 모든 장애에 기계적으로 적용하는 절차가 아니다.

오해 4 "캡슐화는 그냥 헤더를 더하는 것뿐, 장애 분석과 무관하다"

캡슐화 순서를 모르면 "왜 L3 문제가 L7 증상으로 보이는지"를 설명할 수 없다. MTU mismatch(L3)가 왜 애플리케이션 타임아웃(L7)으로 나타나는지 설명하지 못하면, 서로 다른 계층의 문제를 같은 계층 문제로 착각해 잘못된 곳을 고치게 된다.

9. Senior-level Follow-up 질문

Follow-up 질문면접관의 의도답변 포인트
"L4 로드밸런서와 L7 로드밸런서의 차이를 계층 관점에서 설명해보세요." 계층 모델을 실제 인프라 컴포넌트에 적용할 수 있는지 L4는 IP/포트만 보고 라우팅(빠름, 프로토콜 비인식). L7은 HTTP 헤더/경로까지 보고 라우팅(느리지만 콘텐츠 기반 라우팅, TLS termination 가능).
"NAT는 어느 계층에서 동작합니까? 그리고 왜 그것이 P2P 연결을 어렵게 만듭니까?" 계층 경계를 넘나드는 컴포넌트 이해 NAT는 L3(IP)와 L4(포트)를 동시에 변환한다. 내부 IP가 외부에서 보이지 않아 외부에서 내부로 직접 연결을 시작할 수 없다(NAT traversal 문제, STUN/TURN 필요).
"컨테이너 네트워크(Overlay network)는 계층 모델에서 어떻게 표현됩니까?" 최신 인프라를 전통 모델로 설명할 수 있는지 오버레이 네트워크는 기존 L3 위에 가상 L2/L3를 캡슐화(VXLAN 등)해서 만든다. 즉 "터널링"은 계층을 하나 더 쌓는 것과 같다 — 이게 MTU 문제를 유발하는 이유다.
"같은 장애 증상이 L4 문제일 수도, L7 문제일 수도 있다면 무엇으로 구분합니까?" 모호한 상황에서의 구조화된 판단력 TCP 연결 자체(handshake)는 성공하는지(L4), 연결 후 데이터 교환에서 실패하는지(L7)를 tcpdump나 curl -v의 단계별 타임스탬프로 구분한다.
"주니어 개발자가 '핑이 되니까 네트워크는 문제 없다'고 말하면 어떻게 지도하겠습니까?" 기술 리딩/멘토링 역량 ping은 L3(ICMP)만 검증한다는 것, L4/L7은 별도 프로토콜이라 별도로 확인해야 한다는 것을 계층 다이어그램과 실제 사례로 설명한다고 답한다.

10. 자기 점검 Rubric

약한 답변 (Junior 수준)

  • "물리, 데이터링크, 네트워크, 전송, 세션, 표현, 응용 계층이 있습니다"로 답을 끝냄
  • OSI 7계층과 TCP/IP 계층의 관계를 설명하지 못함
  • 계층별로 어떤 도구를 쓰는지 구체적으로 말하지 못함
  • 실제 장애 사례와 연결하지 못하고 이론만 나열
  • 계층 경계에서 증상이 어떻게 "번역"되는지 개념이 없음

강한 답변 (Senior 수준)

  • OSI가 참조 모델이고 실무에서는 TCP/IP 4~5계층으로 단순화해서 쓴다고 설명
  • 계층별로 구체적인 도구와 정상/비정상 판단 기준을 제시
  • 캡슐화 순서를 근거로 "왜 하위 계층 문제가 상위 계층 증상으로 보이는지" 설명
  • bottom-up/top-down 접근의 trade-off를 상황에 맞게 선택
  • 실제 겪은 장애를 계층 순서로 재구성해 말함
  • NAT, 로드밸런서처럼 계층을 넘나드는 컴포넌트도 설명할 수 있음
