## 1. TCP/IP 4계층 모델 심화

[Day 1](./day-01-network-basics-osi.md)에서 잠깐 비교했던 TCP/IP 모델을 이제 깊이 살펴본다.

### TCP/IP 모델이 왜 중요한가?

OSI 7계층은 **이론적 참조 모델**이다. 실제 인터넷은 1970년대 ARPANET에서 발전한 **TCP/IP 프로토콜 스택** 위에 구축되어 있다. 실무에서 패킷을 분석하거나 문제를 디버깅할 때는 TCP/IP 4계층이 기준이다.

### 4계층 구조

```
┌──────────────────────────────────────────┐
│  4. 응용 계층 (Application Layer)         │  HTTP, HTTPS, DNS, SMTP, FTP, SSH
├──────────────────────────────────────────┤
│  3. 전송 계층 (Transport Layer)           │  TCP, UDP
├──────────────────────────────────────────┤
│  2. 인터넷 계층 (Internet Layer)          │  IP, ICMP, ARP (RFC 826)
├──────────────────────────────────────────┤
│  1. 네트워크 액세스 계층                   │  이더넷, Wi-Fi (802.11), PPP
│     (Network Access / Link Layer)        │
└──────────────────────────────────────────┘
```

### OSI vs TCP/IP 계층 대응표

| OSI 계층 | TCP/IP 계층 | 주요 프로토콜 |
|----------|-------------|---------------|
| 7 응용 / 6 표현 / 5 세션 | 응용 계층 | HTTP, DNS, SMTP, FTP |
| 4 전송 | 전송 계층 | TCP, UDP |
| 3 네트워크 | 인터넷 계층 | IPv4, IPv6, ICMP |
| 2 데이터링크 + 1 물리 | 네트워크 액세스 계층 | Ethernet, Wi-Fi |

> **핵심**: TCP/IP 모델은 하위 2계층(OSI 1,2번)을 하나로 묶어 "네트워크 하드웨어 세부 사항에 신경 쓰지 않겠다"는 입장을 취한다.

## 2. 물리 계층 — 비트가 전선을 달리는 방법

### 신호 유형

물리 계층은 디지털 비트(0/1)를 물리적 신호로 변환한다.

| 매체 | 신호 유형 | 특징 |
|------|-----------|------|
| 동축/트위스티드 페어 케이블 | 전기 신호 | 저렴, 간섭 취약 |
| 광섬유 (Fiber Optic) | 빛(광 신호) | 고속·장거리, 간섭 없음 |
| 무선 (Wi-Fi, LTE) | 전자기파 | 이동성 우수, 대역 공유 |

### 이더넷 케이블 카테고리

| 카테고리 | 최대 속도 | 최대 거리 | 용도 |
|----------|-----------|-----------|------|
| Cat5e | 1 Gbps | 100 m | 일반 사무실 |
| Cat6 | 10 Gbps | 55 m | 데이터센터 |
| Cat6A | 10 Gbps | 100 m | 고성능 네트워크 |
| Cat8 | 40 Gbps | 30 m | 서버 룸 |

### 전이중(Full-Duplex) vs 반이중(Half-Duplex)

```
반이중: [A] ──────> [B]  또는  [A] <────── [B]  (동시 불가)
전이중: [A] ──────> [B]
        [A] <────── [B]   (동시 가능, 현대 스위치 기본)
```

## 3. 데이터링크 계층 — 같은 네트워크 안에서의 통신

데이터링크 계층은 **같은 네트워크 세그먼트(링크)** 내 두 장치 간 신뢰성 있는 통신을 담당한다.

두 개의 하위 계층으로 나뉜다:

```
┌─────────────────────────────────────────┐
│  LLC (Logical Link Control)             │  흐름 제어, 오류 통지
├─────────────────────────────────────────┤
│  MAC (Media Access Control)             │  물리 주소, 매체 접근 제어
└─────────────────────────────────────────┘
```

---

## 4. MAC 주소 (물리 주소)

### MAC 주소란?

**MAC(Media Access Control) 주소**는 네트워크 인터페이스 카드(NIC)에 제조사가 부여하는 **48비트(6바이트) 고유 식별자**다.

```
예: 00:1A:2B:3C:4D:5E
     │  │  └─────────── OUI (제조사 식별, 3바이트)
     │  └────────────── 비트 플래그
     └───────────────── 장치 고유 번호 (3바이트)
```

### OUI (Organizationally Unique Identifier)

앞 3바이트는 IEEE가 제조사에게 할당한다:

| OUI | 제조사 |
|-----|--------|
| `00:50:56` | VMware |
| `3C:22:FB` | Apple |
| `00:1B:63` | Apple |
| `FC:FB:FB` | Cisco |

### 특수 MAC 주소

| 주소 | 의미 |
|------|------|
| `FF:FF:FF:FF:FF:FF` | 브로드캐스트 (같은 세그먼트 전체) |
| `01:00:5E:xx:xx:xx` | 멀티캐스트 (IPv4) |
| `33:33:xx:xx:xx:xx` | 멀티캐스트 (IPv6) |

> **MAC vs IP**: MAC 주소는 같은 네트워크 내 **물리적 위치**, IP 주소는 인터넷 전체에서의 **논리적 위치**를 나타낸다. MAC은 라우터를 넘어가지 못한다.

---

## 5. 이더넷(Ethernet) 프레임 구조

이더넷은 현재 유선 LAN의 사실상 표준이다(IEEE 802.3).

### 이더넷 II 프레임 구조

```
┌──────────┬──────────┬───────┬────────────────┬─────┐
│ 목적지   │ 출발지   │ 타입  │    페이로드    │ FCS │
│  MAC     │  MAC     │(EType)│   (데이터)     │     │
│  6바이트 │  6바이트 │ 2바이트│ 46~1500 바이트 │4바이트│
└──────────┴──────────┴───────┴────────────────┴─────┘
```

| 필드 | 크기 | 설명 |
|------|------|------|
| 목적지 MAC | 6 bytes | 수신 장치의 MAC 주소 |
| 출발지 MAC | 6 bytes | 송신 장치의 MAC 주소 |
| EtherType | 2 bytes | 상위 프로토콜 식별 (0x0800=IPv4, 0x0806=ARP, 0x86DD=IPv6) |
| 페이로드 | 46~1500 bytes | 실제 전송 데이터 (IP 패킷 등) |
| FCS | 4 bytes | CRC 오류 감지 값 |

### MTU (Maximum Transmission Unit)

이더넷의 기본 MTU = **1500 바이트**. 이것이 IP 패킷이 기본적으로 1500 바이트로 제한되는 이유다.

---

## 6. ARP — MAC 주소를 찾는 방법

### ARP가 왜 필요한가?

IP 패킷을 보내려면 목적지의 IP 주소를 안다. 하지만 이더넷 프레임을 만들려면 **목적지 MAC 주소**도 알아야 한다.

> **ARP(Address Resolution Protocol)**: IP 주소 → MAC 주소 변환 프로토콜

### ARP 동작 과정 (단계별)

```
시나리오: PC-A(192.168.1.10)가 PC-B(192.168.1.20)에게 패킷을 보내려 한다.

Step 1 — ARP 캐시 확인
  PC-A가 자신의 ARP 테이블을 검색: 192.168.1.20의 MAC 주소가 있는가?
  → 없으면 Step 2로.

Step 2 — ARP 요청 (브로드캐스트)
  PC-A → [목적지 MAC: FF:FF:FF:FF:FF:FF] 브로드캐스트 전송
  메시지: "192.168.1.20을 사용하는 분, MAC 주소를 알려주세요!"
  같은 네트워크의 모든 장치가 수신함.

Step 3 — ARP 응답 (유니캐스트)
  PC-B만 자신의 IP와 일치함을 확인하고 응답.
  PC-B → PC-A에게 유니캐스트 전송
  메시지: "저입니다! 제 MAC은 AA:BB:CC:DD:EE:FF입니다."

Step 4 — ARP 캐시 저장
  PC-A가 192.168.1.20 → AA:BB:CC:DD:EE:FF를 ARP 테이블에 저장.
  이후 통신에서는 ARP 과정 없이 바로 사용.
```

```
네트워크 트래픽 시각화:

PC-A ──[ARP Request: "192.168.1.20은 누구?"]──> 브로드캐스트
PC-A <──[ARP Reply: "저예요! AA:BB:CC:DD:EE:FF"]── PC-B
```

### ARP 테이블 확인 명령

```bash
# Linux / macOS
arp -n

# Windows
arp -a

# 출력 예시:
# 주소              HW 주소             플래그 마스크          인터페이스
# 192.168.1.1     00:50:56:c0:00:08    C                     eth0
# 192.168.1.20    AA:BB:CC:DD:EE:FF    C                     eth0
```

### ARP 캐시 타이머

ARP 엔트리는 일정 시간 후 만료된다 (리눅스 기본: 약 60초 ~ 20분). 만료 후에는 다시 ARP 요청을 보낸다.

## 7. 스위치의 MAC 주소 테이블 학습

### 스위치 vs 허브

| | 허브 (Hub) | 스위치 (Switch) |
|--|------------|-----------------|
| 계층 | 물리 계층 (1계층) | 데이터링크 계층 (2계층) |
| 동작 | 모든 포트로 전달 | MAC 주소 기반 선택 전달 |
| 충돌 도메인 | 전체 공유 | 포트별 분리 |
| 대역폭 | 공유 | 포트별 독립 |

### MAC 주소 테이블 학습 과정

스위치는 처음 켜졌을 때 MAC 주소 테이블이 비어있다. 프레임이 들어올 때마다 학습한다.

```
초기 상태 — 스위치의 MAC 테이블: 비어있음

Step 1: PC-A(MAC: AA:AA)가 포트 1로 프레임 전송
  → 스위치: "포트 1에서 AA:AA가 왔다" → 테이블에 기록
  테이블: [AA:AA → 포트 1]
  
  목적지 MAC을 모르면 → 플러딩(Flooding): 포트 1 제외 모든 포트로 전달

Step 2: PC-B(MAC: BB:BB)가 포트 2로 응답 전송
  → 스위치: "포트 2에서 BB:BB가 왔다" → 테이블에 기록
  테이블: [AA:AA → 포트 1] [BB:BB → 포트 2]
  
  목적지 AA:AA = 포트 1 → 포트 1로만 전달 (유니캐스트 포워딩)
```

```
스위치 내부 구조 (예):

포트 1 ── [PC-A: AA:AA]
포트 2 ── [PC-B: BB:BB]
포트 3 ── [PC-C: CC:CC]
포트 4 ── [라우터: 00:11]

MAC 주소 테이블:
┌──────────────┬──────┐
│ MAC 주소     │ 포트 │
├──────────────┼──────┤
│ AA:AA:..     │  1   │
│ BB:BB:..     │  2   │
│ CC:CC:..     │  3   │
│ 00:11:..     │  4   │
└──────────────┴──────┘
```

### 스위치의 3가지 동작

| 동작 | 조건 | 처리 방법 |
|------|------|-----------|
| **학습(Learning)** | 출발지 MAC 처음 봄 | 테이블에 포트 기록 |
| **포워딩(Forwarding)** | 목적지 MAC을 알고 있음 | 해당 포트로만 전달 |
| **플러딩(Flooding)** | 목적지 MAC 모름 or 브로드캐스트 | 모든 포트(입력 포트 제외)로 전달 |

> **브로드캐스트 도메인**: 스위치는 브로드캐스트를 모든 포트로 전달한다. 브로드캐스트 도메인을 분리하려면 **라우터** 또는 **VLAN**이 필요하다.

## 8. CSMA/CD — 이더넷의 충돌 처리 (반이중 환경)

과거 허브 기반 반이중 환경에서는 여러 장치가 동시에 전송하면 충돌이 발생했다.

**CSMA/CD (Carrier Sense Multiple Access / Collision Detection)**:

1. **Carrier Sense**: 전송 전에 채널 사용 중인지 확인
2. **Collision Detection**: 충돌 감지 시 즉시 전송 중단
3. **Jam Signal**: 충돌 발생 알림 신호 전송
4. **Backoff**: 무작위 시간 대기 후 재전송

> 현대의 전이중(Full-Duplex) 스위치 환경에서는 CSMA/CD가 필요 없다. 포트별로 독립된 충돌 도메인이므로 충돌 자체가 발생하지 않는다.

## 9. VLAN — 논리적 네트워크 분리

**VLAN(Virtual LAN)**: 물리적으로 같은 스위치에 연결되어 있어도 **논리적으로 별도 네트워크**처럼 분리.

```
물리적 연결:           논리적 분리:
스위치                 VLAN 10 (개발팀)  VLAN 20 (마케팅)
├── 포트 1 (PC-A) ──> [PC-A]           
├── 포트 2 (PC-B) ──>                  [PC-B]
├── 포트 3 (PC-C) ──> [PC-C]           
└── 포트 4 (PC-D) ──>                  [PC-D]
```

- VLAN 10의 PC-A와 PC-C는 서로 통신 가능.
- VLAN 10과 VLAN 20 간 통신은 라우터(또는 L3 스위치)를 거쳐야 한다.

## 11. 핵심 요약

- **TCP/IP 4계층**은 실제 인터넷의 구현 모델이며, OSI 하위 2계층을 '네트워크 액세스 계층' 하나로 통합한다.
- **MAC 주소**는 48비트 물리 주소로, 같은 네트워크 내 장치 식별에 사용된다.
- **ARP**는 IP 주소를 MAC 주소로 변환하는 프로토콜이다. 브로드캐스트 요청 → 유니캐스트 응답 방식으로 동작한다.
- **스위치**는 MAC 주소 테이블을 학습하여 플러딩 없이 목적지 포트로만 프레임을 전달한다.
- **이더넷 프레임**의 기본 MTU는 1500 바이트이며, CRC 기반 FCS로 오류를 감지한다.
