1. 면접 질문

"특정 리전의 일부 사용자에게서만 API가 502/타임아웃으로 실패한다는 보고가 들어왔습니다. 서버 CPU, 메모리, 애플리케이션 로그는 모두 정상입니다. 이것이 DNS 문제인지 애플리케이션 문제인지 어떤 순서로, 어떤 도구로 구분하시겠습니까? 그리고 원인이 DNS라면 왜 '일부' 사용자에게만 나타났을지 설명해보세요."

이 질문이 어려운 이유는 DNS 장애가 "전체 다운"이 아니라 "부분적이고 시간차를 두고 나타나는" 형태로 오는 경우가 많기 때문이다. 많은 지원자가 "DNS는 이름을 IP로 바꿔주는 것"까지는 안다. 하지만 면접관이 진짜 확인하려는 것은 캐시 계층이 여러 겹으로 존재해서 같은 도메인이라도 클라이언트마다 다른 시점에 다른 답을 볼 수 있다는 것, 그리고 그 사실을 이용해 "왜 일부만 실패하는가"를 논리적으로 설명할 수 있는가다.

2. 전제 개념 — 알아야 할 기초

2-1. DNS는 "질문"이 아니라 "질문의 릴레이"다

DNS는 하나의 서버가 모든 답을 아는 구조가 아니다. 도메인 이름은 계층 구조(www.example.com → com → example.com → www)로 나뉘어 있고, 각 계층을 책임지는 서버가 따로 있다. 사용자의 컴퓨터가 직접 이 계층을 다 돌아다니지 않고, 보통 재귀 리졸버(recursive resolver)라는 대리인에게 "이 이름의 IP를 알려줘"라고 맡기면, 그 대리인이 root → TLD(.com) → 권한 있는 서버(authoritative server) 순으로 대신 물어보고 최종 답을 가져온다.

비유: 내가 직접 도서관 전체를 뒤지지 않고 사서(재귀 리졸버)에게 "이 책 어디 있어요?"라고 물으면, 사서가 대신 안내 데스크(root) → 해당 서가 표지판(TLD) → 그 서가 담당 사서(authoritative server)를 순서대로 거쳐 책 위치를 찾아 나에게 알려주는 것과 같다. 나는 그 과정을 보지 못하고 최종 답만 받는다.

2-2. 캐시는 한 곳이 아니라 여러 겹으로 존재한다

이 질문에 강한 답을 하려면 "DNS 캐시"를 하나의 캐시로 뭉뚱그리지 않아야 한다. 실제로는 아래처럼 여러 층이 각자 다른 시점에 캐시를 갱신한다.

캐시 위치누가 갖고 있나실무 함의
애플리케이션/라이브러리 캐시브라우저, HTTP 클라이언트, JVM(기본적으로 영구 캐시하는 설정도 있음)애플리케이션이 TTL을 아예 무시하고 자체 캐시 정책을 쓰는 경우가 많아, "TTL을 줄였는데도 안 바뀐다"는 문제의 원인이 되기도 한다.
OS 리졸버 캐시운영체제(stub resolver), 컨테이너의 nsswitch/resolv.conf 경로컨테이너/VM을 재시작하지 않으면 오래 남아있을 수 있다.
재귀 리졸버 캐시사내 DNS 서버, ISP DNS, 퍼블릭 DNS(8.8.8.8 등)사용자가 어떤 리졸버를 쓰는지에 따라 같은 시각에도 다른 IP를 받을 수 있다 — 이게 "일부 사용자만" 문제가 나타나는 핵심 이유 중 하나다.
Negative 캐시재귀 리졸버가 SOA 레코드에 정의된 negative TTL만큼 "이 이름은 없다/에러"라는 응답 자체를 캐시레코드를 고쳐도 negative TTL이 끝날 때까지 실패가 계속 재현될 수 있다.

핵심: 같은 순간에도 사용자 A는 새 IP를, 사용자 B는 아직 옛 IP를 캐시하고 있을 수 있다. 이것이 "일부 리전, 일부 사용자만" 문제가 발생하는 가장 흔한 구조적 이유다 — 실제로는 리전 문제가 아니라 어느 리졸버 캐시를 거쳤는지의 문제인 경우가 많다.

2-3. TTL은 "약속"이지 "강제"가 아니다

TTL(Time To Live)은 권한 서버가 "이 답을 몇 초 동안 캐시해도 좋다"고 알려주는 값이다. 하지만 모든 리졸버가 이 값을 정확히 지키는 것은 아니다. 일부 ISP나 사내 리졸버는 성능/부하 이유로 최소 캐싱 시간을 강제하거나, 반대로 애플리케이션 코드(특히 오래된 JVM 기본 설정)는 TTL과 무관하게 무한 캐싱하기도 한다. "TTL을 60초로 낮췄으니 1분 안에 전 세계에 반영된다"는 가정은 실무에서 자주 깨진다.

2-4. 레코드 타입과 그 의미

레코드역할실무에서 자주 쓰는 곳
A / AAAA도메인 → IPv4 / IPv6 주소가장 기본적인 매핑
CNAME도메인 → 다른 도메인 이름(별칭)CDN, 로드밸런서(ALB 등)처럼 IP가 자주 바뀌는 대상을 가리킬 때
NS이 영역(zone)을 관리하는 권한 서버 지정DNS 위임(delegation) 구조 확인
SOA영역의 관리 정보(직렬번호, refresh, negative TTL 등)negative 캐싱 시간, 보조 서버 동기화 주기 확인
SRV호스트+포트+우선순위까지 지정내부 서비스 디스커버리(예: Kubernetes headless service, Consul)
TXT임의 텍스트 — 검증/정책용도메인 소유권 검증, SPF/DKIM(이메일 스푸핑 방지)

2-5. "로드밸런싱"처럼 보이지만 아닌 것 — Round-robin DNS

같은 도메인에 여러 A 레코드를 등록하면 리졸버가 순서를 돌려가며 응답하는데, 이것이 로드밸런싱처럼 보이지만 헬스 체크가 없다. 죽은 서버의 IP도 그대로 캐시되어 계속 반환될 수 있다. 반면 GSLB(Global Server Load Balancing)나 상태 기반 DNS 페일오버는 헬스 체크 결과에 따라 응답을 동적으로 바꾸지만, 이 역시 클라이언트/리졸버 캐시가 만료되어야 새 응답이 반영된다는 한계는 그대로 남는다. Anycast(같은 IP를 여러 위치에서 광고해 라우팅 계층에서 가장 가까운 곳으로 보내는 방식, 예: 8.8.8.8, CDN 엣지)는 아예 DNS 캐시 문제를 우회하는 다른 접근이다 — IP는 그대로인데 그 IP로 가는 "길"이 바뀐다.

2-6. Split-horizon DNS — 같은 이름, 다른 답

내부망(VPC)에서 api.example.com을 조회하면 내부 IP(사설 IP, VPC 내부 로드밸런서)를, 외부 인터넷에서 조회하면 공인 IP를 반환하도록 같은 도메인 이름에 대해 조회 위치에 따라 다른 응답을 주는 구성이다. 이렇게 하면 내부 서비스 간 통신이 인터넷을 거치지 않고 VPC 내부에서 끝나 지연과 비용, 노출 표면을 줄일 수 있다. 하지만 설정이 꼬이면 "내부에서 접근해야 할 서비스가 외부 DNS로 풀려서 NAT/보안 그룹을 거치다 막히는" 형태의 장애가 생긴다.

2-7. 컨테이너/클러스터 환경의 서비스 디스커버리

Kubernetes 같은 환경에서는 서비스 이름 해석에 클러스터 내부 DNS(CoreDNS)가 관여하고, search domain과 ndots 설정 때문에 짧은 이름 하나를 풀기 위해 여러 번의 추가 쿼리가 발생할 수 있다(예: myservice → myservice.ns.svc.cluster.local을 시도하기 전에 여러 접미사를 순서대로 시도). 이 구조를 모르면 "가끔 DNS 조회가 느리다"는 현상을 CPU나 네트워크 탓으로 잘못 짚기 쉽다.

3. 왜 면접에서 이걸 묻는가

  1. 1단계 필터 (정의 vs 구조 이해) — "DNS는 이름을 IP로 바꿔준다"에서 멈추는지, 재귀 질의 경로와 여러 겹의 캐시 구조까지 설명할 수 있는지 확인한다.
  2. 2단계 필터 (지식 vs 진단 능력) — 증상(부분적, 시간차 있는 실패)을 보고 원인 후보(캐시 불일치, negative caching, 페일오버 지연)를 논리적으로 좁혀나갈 수 있는지 본다.
  3. 3단계 필터 (진단 vs 실제 도구 사용 경험) — dig, nslookup, curl --resolve 같은 도구로 "이름 해석 문제"와 "연결 문제"를 실제로 분리해본 경험이 있는지, TTL/failover 설계를 직접 판단해본 적이 있는지 follow-up으로 확인한다.
면접관이 피하고 싶은 답변: "DNS는 도메인을 IP로 변환해주는 시스템입니다. 재귀 질의와 반복 질의가 있고, A, CNAME, MX 같은 레코드 타입이 있습니다." 교과서 정의는 맞지만, 왜 캐시 때문에 부분 장애가 발생하는지, 어떤 도구로 무엇을 확인해야 하는지가 전혀 없다. 실제 장애 상황에서 무엇을 할지 알 수 없는 답변이다.

4. 실무 연결

실제 장애 시나리오 A — "장애 복구를 선언했는데 일부 사용자는 30분 뒤에도 계속 실패"

장애 서버를 새 IP로 교체하고 DNS 레코드를 변경했지만, 일부 사용자 그룹에서 계속 이전 IP로 연결을 시도하며 타임아웃이 발생했다. dig로 여러 퍼블릭 리졸버(8.8.8.8, 1.1.1.1, 사내 DNS)에 직접 질의해보니, 사내 리졸버 하나가 TTL을 훨씬 넘겨서도 옛 IP를 캐시하고 있었다. 원인은 레코드에 설정한 TTL(60초)보다 사내 리졸버의 최소 캐싱 정책(강제 TTL 하한)이 더 길게 설정되어 있었던 것이다. "레코드를 바꿨으니 곧 반영된다"는 가정이 틀렸음을 실제 리졸버별 질의로 확인하고 나서야 원인을 좁힐 수 있었다.

실제 장애 시나리오 B — 오타로 배포된 레코드 하나가 몇 시간 동안 여파를 남김

배포 자동화 스크립트가 실수로 존재하지 않는 이름에 대한 조회를 여러 번 발생시켰고, 그 결과가 SOA에 정의된 negative TTL(예: 1시간)만큼 여러 재귀 리졸버에 "이 이름은 없다"로 캐시됐다. 원인이 되는 배포는 5분 만에 롤백했지만, negative 캐시가 살아있는 리졸버를 쓰는 사용자들은 이후 최대 1시간까지 실패를 계속 겪었다. 이 경험 이후로는 SOA의 negative TTL 값을 별도로 관리 대상에 넣고, 급한 변경이 필요할 때는 "캐시가 자연 만료될 시간"까지 포함해서 복구 예상 시간을 커뮤니케이션하게 됐다.

실제 장애 시나리오 C — Split-horizon 설정 실수로 내부 트래픽이 외부로 나가 막힘

내부 서비스 간 호출에 쓰던 도메인의 내부 zone 설정이 실수로 삭제되면서, 같은 이름이 외부(공인) DNS로 풀리기 시작했다. 트래픽이 VPC를 벗어나 인터넷 경유로 나가려다 보안 그룹/NACL에 막혀 연결이 실패했다. 애플리케이션 로그에는 "connection refused/timeout"만 남아 처음엔 방화벽 문제로 오인했지만, dig로 해석된 IP가 내부 대역이 아니라 공인 IP라는 것을 확인하고서야 DNS 위임 구조 문제였음을 알 수 있었다.

5. 답변 사고 순서

  1. 증상의 범위를 먼저 정의한다
    전체 사용자인지, 특정 리전/ISP/사내망인지, 시간이 지나면서 줄어드는지 늘어나는지부터 확인한다. "일부만, 시간차 있게"라는 패턴 자체가 DNS 캐시 계층을 의심할 강한 단서다.
  2. 이름 해석과 연결을 분리해서 확인한다
    dig/nslookup으로 이름이 어떤 IP로 풀리는지 먼저 확인하고, 그 IP로 curl --resolve나 직접 telnet/포트 접속을 시도해 "이름이 틀린 곳으로 풀리는지" vs "맞게 풀렸는데 그 IP가 응답하지 않는지"를 나눈다.
  3. 여러 리졸버에서 같은 질의를 비교한다
    퍼블릭 리졸버(8.8.8.8, 1.1.1.1)와 문제를 겪는 사용자가 실제로 쓰는 리졸버(사내 DNS, ISP DNS)의 응답이 다른지 비교해, 캐시 불일치인지 권한 서버 자체의 문제인지 좁힌다.
  4. TTL과 negative caching을 확인한다
    레코드의 TTL, 그리고 SOA의 negative TTL을 확인해 "언제쯤 자연적으로 해소될지"를 계산한다. 이는 원인 규명뿐 아니라 장애 커뮤니케이션(ETA 안내)에도 필요하다.
  5. 구성 변경 이력을 함께 본다
    최근에 레코드 변경, failover 전환, split-horizon 설정 변경이 있었는지 확인한다. 증상 시작 시점과 변경 이력의 시간 상관관계를 맞춰본다.
  6. 재발 방지책까지 제시한다
    TTL 정책 재검토, 헬스 체크 기반 자동 페일오버 도입, 배포 전 DNS 변경 사전 공지/모니터링, negative caching 값 관리 등으로 마무리한다.

6. 핵심 답변 예시

짧은 답변 (60초 버전)

"먼저 실패가 전체가 아니라 일부라는 점에 주목하겠습니다. 이건 DNS 캐시가 여러 계층(OS, 사내/ISP 리졸버, 애플리케이션 자체 캐시)에 나뉘어 있어서, 어떤 캐시를 거쳤는지에 따라 사용자마다 다른 시점에 다른 IP를 볼 수 있다는 신호이기 때문입니다. 그래서 dig로 도메인이 실제로 어떤 IP로 풀리는지 확인하고, 그 IP로 직접 연결을 시도해서 '이름이 잘못 풀렸는지' 아니면 '이름은 맞게 풀렸는데 그 서버가 문제인지'를 먼저 나눕니다. 그다음 문제를 겪는 사용자군이 쓰는 리졸버와 정상 사용자군이 쓰는 리졸버의 응답을 비교해서 캐시 불일치인지 확인하고, TTL과 negative TTL 값을 봐서 언제 자연 해소될지도 같이 계산합니다."

깊이 있는 답변 (면접관이 "더 자세히"라고 물을 때)

"조금 더 구체적으로 말씀드리면, 저는 이 문제를 세 단계로 나눠서 봅니다. 첫 번째는 '이름 해석 계층'이고, 두 번째는 '전송 계층 연결', 세 번째는 '애플리케이션 계층'입니다. dig +trace로 root부터 authoritative까지 실제 위임 경로를 따라가 보고, 여러 퍼블릭 리졸버와 문제 사용자가 쓰는 리졸버의 응답을 비교합니다. 만약 문제를 겪는 사용자 쪽 리졸버만 옛 IP를 주고 있다면, 그건 서버 문제가 아니라 그 리졸버의 캐시가 아직 안 비워진 것입니다. 이때 TTL을 확인해서 이미 만료됐어야 하는데도 옛 값을 주고 있다면, 그 리졸버가 TTL을 지키지 않거나 최소 캐싱 정책을 강제하고 있을 가능성을 의심합니다. 반대로 모든 리졸버가 같은(올바른) IP를 준다면, 문제는 DNS가 아니라 그 IP로의 연결 자체입니다 — 이 경우 curl -v나 직접 소켓 연결을 시도해서 TCP 핸드셰이크 단계에서 막히는지, TLS 단계에서 막히는지, 아니면 HTTP 응답이 느린지를 봅니다. 그리고 최근 변경 이력을 꼭 확인하는데, 실제로 겪었던 사례는 negative caching 때문에 오타로 배포된 존재하지 않는 이름 조회가 SOA에 정의된 negative TTL만큼(1시간) 여러 리졸버에 '없음'으로 캐시돼서, 원인 배포를 5분 만에 롤백했는데도 사용자 체감 장애는 훨씬 오래갔던 경우입니다. 그 이후로는 장애 복구 시점을 안내할 때 애플리케이션 배포 완료 시각이 아니라 TTL/negative TTL까지 고려한 캐시 자연 만료 시각까지 함께 계산해서 공지합니다."

7. Trade-off

선택장점단점 / 주의사항
짧은 TTL 장애 복구/페일오버가 빠르게 전파됨, 인프라 변경에 유연하게 대응 가능 리졸버 질의량 증가, 매 요청마다 조회가 캐시 미스로 이어져 지연 증가 가능, 일부 리졸버는 짧은 TTL을 무시하고 자체 하한을 강제하기도 함
긴 TTL 질의량 감소, 캐시 히트로 지연 감소, 리졸버/네트워크 부하 감소 장애 시 전환이 느리게 전파됨, 최악의 경우 캐시가 만료될 때까지 일부 사용자가 계속 실패
DNS 기반 페일오버(GSLB) 추가 인프라 없이 지역/헬스 기반 라우팅 가능, 구성이 비교적 단순 캐시 만료 전까지 전환이 지연됨, 세밀한 트래픽 제어(가중치, 세션 유지)는 L4/L7 로드밸런서보다 약함
Split-horizon DNS 내부 트래픽이 인터넷을 거치지 않아 지연·비용·노출 표면 감소 내부/외부 zone을 동시에 정확히 관리해야 하는 운영 복잡도, 설정 실수 시 내부 트래픽이 외부로 새거나 반대로 외부 접근이 내부 IP로 잘못 해석되는 위험
Anycast 기반 라우팅 DNS 캐시 만료를 기다릴 필요 없이 라우팅 계층에서 즉시 트래픽 전환 가능 구축·운영 복잡도가 높고, 일반적으로 CDN/DNS 사업자 인프라 수준에서만 실용적

8. 흔한 오해

오해 1 "TTL을 줄이면 변경사항이 그 시간 안에 전 세계에 반영된다"

TTL은 권한 서버가 제시하는 "권장 캐싱 시간"일 뿐, 모든 리졸버와 애플리케이션이 정확히 지킨다는 보장이 없다. 일부 리졸버는 최소 캐싱 시간을 강제하고, 일부 애플리케이션(특히 오래된 JVM 기본 설정)은 TTL과 무관하게 자체적으로 오래 캐시한다. TTL은 "이 시간 안에는 반드시 반영된다"가 아니라 "이 시간 이후에는 반영될 가능성이 높아진다" 정도로 이해해야 한다.

오해 2 "DNS 장애는 항상 전체 다운으로 나타난다"

캐시 계층이 여러 겹이기 때문에, 실제로는 캐시를 이미 가진 사용자는 문제를 못 느끼고 새로 조회하는 사용자만 영향을 받는 부분적·비균일한 장애로 나타나는 경우가 훨씬 흔하다. "일부 사용자만 겪는다"는 이유로 DNS를 원인 후보에서 제외하는 것은 반대로 가야 할 판단이다.

오해 3 "라운드로빈 DNS는 로드밸런서와 같은 기능이다"

라운드로빈 DNS는 여러 A 레코드를 순서대로 돌려가며 응답할 뿐, 대상 서버의 상태를 확인하지 않는다. 죽은 서버의 IP도 그대로 캐시되어 계속 반환될 수 있다. 헬스 체크, 세션 유지, 세밀한 트래픽 분배가 필요하다면 L4/L7 로드밸런서나 헬스 체크 기반 GSLB가 필요하다.

오해 4 "레코드를 수정하면 문제가 된 옛 응답도 바로 사라진다"

SOA에 정의된 negative TTL 때문에 "이 이름은 없다/에러다"라는 응답 자체도 별도로 캐시된다. 원인이 되는 설정을 즉시 되돌려도, 이미 negative 캐시를 가진 리졸버들은 그 캐시가 자연 만료될 때까지 계속 실패 응답을 줄 수 있다. 장애 복구 시각을 안내할 때 이 시간을 빠뜨리면 "고쳤다는데 왜 아직도 안 되냐"는 혼란이 생긴다.

오해 5 "Split-horizon DNS를 쓰면 내부망 보안은 저절로 해결된다"

Split-horizon DNS는 트래픽 경로를 최적화하는 장치이지 접근 제어 장치가 아니다. 이름이 내부 IP로 풀린다고 해서 그 통신이 안전하다는 뜻은 아니며, 실제 접근 통제는 보안 그룹/NACL/방화벽 같은 네트워크 계층에서 별도로 강제해야 한다. 반대로 설정 실수로 zone 위임이 깨지면 내부 트래픽이 외부로 새어 나가거나 그 반대의 상황이 벌어질 수 있다.

9. Senior-level Follow-up 질문

Follow-up 질문면접관의 의도답변 포인트
"TTL을 아주 짧게(예: 5초)로 설정하면 어떤 부작용이 있습니까?" 지식을 실무 규모(트래픽 볼륨)로 확장해서 사고할 수 있는지 권한 서버와 재귀 리졸버에 대한 질의량이 급증해 DNS 자체가 병목/장애 지점이 될 수 있고, 매 요청마다 캐시 미스로 인한 해석 지연이 누적된다. 페일오버 속도와 조회 비용 사이의 trade-off를 구체적으로 설명해야 한다.
"DNS 응답이 정상적으로 왔는데도 연결이 실패한다면 다음에 무엇을 확인합니까?" DNS 문제와 그 이후 계층(TCP, TLS, 애플리케이션) 문제를 분리할 수 있는지 해석된 IP로 직접 TCP 연결(telnet/curl -v)을 시도해 SYN 응답 여부를 확인하고, 방화벽/보안 그룹, 서버 프로세스 상태, TLS 핸드셰이크 단계까지 순차적으로 좁혀나간다고 답한다.
"Anycast와 GSLB(DNS 기반 페일오버)의 차이는 무엇이고, 언제 Anycast가 필요합니까?" DNS 캐시의 본질적 한계와 이를 우회하는 다른 접근을 이해하는지 GSLB는 DNS 응답 자체를 상태에 따라 바꾸는 방식이라 클라이언트/리졸버 캐시가 만료돼야 반영되지만, Anycast는 IP는 그대로 두고 라우팅 계층에서 가장 가까운/가용한 위치로 보내므로 캐시 만료를 기다릴 필요가 없다. 초저지연 페일오버(DDoS 완화, 글로벌 CDN)에는 Anycast가 유리하다.
"Kubernetes 클러스터에서 특정 파드만 서비스 이름 해석이 간헐적으로 느리거나 실패합니다. 어떻게 접근하시겠습니까?" 클러스터 환경의 서비스 디스커버리 세부 동작(ndots, search domain, CoreDNS)을 실무에서 다뤄봤는지 ndots 설정 때문에 짧은 서비스 이름 하나를 풀기 위해 여러 search domain을 순차 시도하며 불필요한 쿼리가 늘어나는 경우, UDP 기반 DNS 응답이 conntrack 테이블 경합으로 유실되는 경우 등을 원인 후보로 들고, CoreDNS 파드 상태·리소스, dnsConfig 설정, 특정 노드 편중 여부를 확인한다고 답한다.
"DNS 기반 서비스 디스커버리 대신 서비스 메시(사이드카)를 쓰는 이유는 무엇이라고 생각합니까?" DNS의 한계(헬스 인지 부족, 캐시 지연)를 더 정교한 대안과 비교해서 설명할 수 있는지 DNS는 캐시 계층 때문에 실시간 상태 반영이 느리고 세밀한 트래픽 제어(가중치 라우팅, 서킷 브레이커, 재시도 정책)를 표현하기 어렵다. 서비스 메시는 사이드카가 실시간으로 엔드포인트 상태를 알고 있어 더 빠르고 정교한 라우팅이 가능하지만, 그만큼 운영 복잡도와 리소스 오버헤드가 추가된다는 trade-off를 짚는다.

10. 자기 점검 Rubric

약한 답변 (Junior 수준)

  • "DNS는 도메인을 IP로 바꿔주는 시스템"이라는 정의에서 멈춤
  • 캐시가 여러 계층(OS, 리졸버, 애플리케이션)에 나뉘어 있다는 사실을 모름
  • "일부 사용자만 실패"라는 증상과 DNS 캐시 불일치를 연결하지 못함
  • TTL을 "정확히 지켜지는 강제 규칙"으로 오해
  • negative caching의 존재를 몰라 "롤백했는데 왜 계속 실패하냐"를 설명 못함
  • 이름 해석 문제와 연결(TCP/TLS) 문제를 구분할 진단 절차가 없음

강한 답변 (Senior 수준)

  • 재귀 질의 경로와 여러 겹의 캐시 계층을 구조적으로 설명
  • "부분적, 시간차 있는 실패"를 캐시 불일치의 신호로 즉시 연결
  • dig/nslookup으로 여러 리졸버 응답을 비교해 원인을 좁히는 절차를 안다
  • TTL과 negative TTL을 구분하고, 복구 ETA 계산에 반영한다
  • Round-robin DNS, GSLB, Anycast의 차이와 각각의 한계를 설명
  • Split-horizon과 클러스터 내부 DNS(ndots 등)까지 실무 사례로 연결

자기 점검 체크리스트

  • ☐ 재귀 질의와 여러 겹의 캐시(OS/리졸버/애플리케이션/negative) 구조를 설명했는가?
  • ☐ "일부 사용자만 실패"하는 증상을 캐시 계층 불일치로 논리적으로 연결했는가?
  • ☐ 이름 해석 문제와 그 이후(TCP/TLS/애플리케이션) 문제를 분리하는 진단 절차를 제시했는가?
  • ☐ TTL과 negative TTL의 차이를 구분하고 복구 시간 계산에 반영했는가?
  • ☐ Round-robin DNS, GSLB, Anycast의 차이를 설명했는가?
  • ☐ Split-horizon DNS나 클러스터 내부 서비스 디스커버리 같은 실무 구조를 언급했는가?
