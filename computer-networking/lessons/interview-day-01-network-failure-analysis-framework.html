1. 면접 질문

"사용자들이 우리 API가 느리다고 합니다. 서버 CPU 사용률은 5%, DB 쿼리 응답도 정상입니다. 네트워크 관점에서 무엇부터 확인하시겠습니까?"

위 질문은 10년차 이상 개발자 인터뷰의 전형적인 오픈형 문제다. 정답이 하나인 퀴즈가 아니라, 당신이 어떤 순서로 생각하고 어떤 도구를 선택하는지 보는 질문이다.

2. 전제 개념 — 알아야 할 기초

2-1. 왜 "계층(layer)"으로 생각하는가

네트워크 문제는 원인이 한 군데에 있지 않다. 같은 증상("느리다")이라도 DNS 응답 지연일 수도, TCP 재전송일 수도, TLS 핸드셰이크 비용일 수도 있다. 계층별로 분리해서 하나씩 배제(exclude)하지 않으면 어디서부터 고쳐야 할지 알 수 없다.

비유: 식당에서 요리가 늦게 나올 때, "주방 문제인가 vs 홀 서빙 문제인가 vs 주문 입력 문제인가"를 구분하지 않고 무작정 요리사를 바꾸면 실제 원인을 놓친다. 네트워크도 마찬가지다.

2-3. RTT(Round Trip Time)란

요청을 보낸 뒤 응답의 첫 비트가 돌아오기까지 걸리는 시간이다. RTT = 전파 지연 + 처리 지연 + 큐 지연. 서버 처리 시간이 아무리 짧아도 RTT가 크면 사용자는 느리다고 느낀다.

# 기본 RTT 측정 (ICMP)
$ ping api.example.com -c 10
rtt min/avg/max/mdev = 18.2/21.4/38.1/5.2 ms

# 패킷 손실 포함 경로 분석
$ mtr --report api.example.com

3. 왜 면접에서 이걸 묻는가

이 질문의 진짜 목적은 세 가지다:

  1. 구조화된 사고력 확인 — "느리다"는 증상을 계층별로 분해하는 훈련이 되어 있는가. 경험 없는 개발자는 "서버 스펙을 올리겠다" "코드를 최적화하겠다"로 뛰어든다.
  2. 도구 선택 판단력 — 실제로 어떤 도구로 무엇을 측정할지 구체적으로 말할 수 있는가.
  3. 장애 대응 경험 유무 — 각 단계에서 어떤 결과를 보고 다음 가설로 넘어가는지 경험에 기반해 말하는가.
면접관이 피하고 싶은 답변: "서버 로그를 먼저 봅니다" → 이미 CPU/DB가 정상이라고 말했는데 같은 방향을 반복하는 것은 문제 분해가 안 된다는 신호다.

4. 실무 연결

실제 장애 시나리오 A — DNS TTL이 너무 짧은 케이스

신규 서비스 오픈 직후 "페이지가 처음 열릴 때만 2~3초 느리다"는 민원이 들어왔다. CPU, DB 모두 정상. APM을 보니 서버 처리 시간은 50ms. 실제 원인은 DNS TTL이 60초로 설정되어 있어 클라이언트마다 DNS를 매 분 재조회하고, DNS 서버 응답 지연이 1~2초였다. TTL을 300초로 올리고 DNS 서버를 CDN 제공자의 AnyCast DNS로 교체하자 해결됐다.

실제 장애 시나리오 B — TCP TIME_WAIT 고갈

트래픽이 올라가면서 API 응답 시간이 갑자기 급등했다. 서버 측에서 ss -s를 보니 TIME_WAIT 상태 소켓이 수만 개였다. HTTP/1.1 기반 단기 연결(keep-alive 미사용)을 사용하고 있어 연결마다 TIME_WAIT이 생겼고, 포트 고갈이 임박했다. keep-alive 활성화 + connection pool로 해결.

실제 장애 시나리오 C — MTU mismatch로 인한 간헐적 패킷 손실

VPN 터널을 통해 내부 서비스를 호출하는데 특정 크기 이상의 응답만 느렸다. Wireshark로 보니 1500 바이트 초과 패킷이 재전송을 반복하고 있었다. VPN MTU가 1400인데 서버 측 MTU는 1500 그대로여서 단편화(fragmentation)가 일어나거나 패킷이 드롭되고 있었다. MSS clamping으로 해결.

5. 답변 사고 순서

면접 답변을 구조화할 때 아래 순서로 전개한다. 이 순서 자체가 시니어 개발자의 트러블슈팅 사고를 보여준다.

  1. 증상 정밀화
    "느리다"를 수치로 바꾼다. 어느 지역에서? 어떤 엔드포인트에서? 첫 바이트가 느린가, 전체 응답이 느린가? 모든 사용자인가 일부인가? 언제부터 발생했나?
  2. L7 — DNS 조회 시간 확인
    dig, nslookup으로 도메인 조회 시간을 측정. APM에서 TTFB(Time to First Byte)를 보고 DNS 지연이 포함되어 있는지 분리한다. 
    $ dig api.example.com +stats
;; Query time: 230 msec   ← 정상은 1~10ms 이내
  3. L4 — TCP 연결 및 TLS 핸드셰이크 시간 확인
    curl -w "@curl-format.txt"로 각 단계 시간을 분리해 측정. time_connect와 time_appconnect를 비교하면 TLS 비용을 알 수 있다. 
    $ curl -w "\n
dns_resolution:   %{time_namelookup}s
tcp_connect:      %{time_connect}s
tls_handshake:    %{time_appconnect}s
server_response:  %{time_starttransfer}s
total:            %{time_total}s\n" \
  -o /dev/null -s https://api.example.com
  4. L3 — 경로와 패킷 손실 확인
    mtr으로 경로 각 홉의 RTT와 손실률을 측정. 특정 홉에서 RTT가 급증하거나 손실이 발생하면 그 구간이 병목이다. 
    $ mtr --report --report-cycles 30 api.example.com
Host                    Loss%   Snt   Last   Avg  Best  Wrst
1. gateway (192.168.1.1) 0.0%    30    1.2   1.3   1.1   2.1
2. isp-router           0.0%    30    8.4   8.6   8.1   9.2
3. core-1.example.com   0.0%    30   19.3  20.1  18.9  38.4
4. api.example.com      0.0%    30   21.1  21.4  20.8  38.9
  5. L4 — TCP 상태 및 재전송 확인
    서버에서 ss -s, netstat -s | grep retransmit로 재전송 횟수와 소켓 상태 분포를 확인한다. TIME_WAIT이 과다하거나 재전송이 비정상적으로 많은지 체크. 
    $ ss -s
Total: 142
TCP:   89 (estab 12, closed 60, orphaned 0, timewait 58)

$ netstat -s | grep -i retransmit
    245 segments retransmitted   ← 재전송이 많으면 패킷 손실 의심
  6. 로드밸런서 / 프록시 계층 확인
    로드밸런서 액세스 로그에서 upstream 응답 시간 vs 클라이언트 응답 시간 차이를 비교한다. 차이가 크다면 LB 자체 처리 지연이나 keepalive 설정 문제일 수 있다.
  7. 가설 수립 및 재현 테스트
    수집한 데이터를 바탕으로 "DNS TTL 문제", "TLS 세션 재개 미사용", "keep-alive 미설정" 중 가능성이 높은 순으로 가설을 세우고 한 번에 하나씩 변경해 효과를 확인한다.
  8. 재발 방지 — 관측성(Observability) 강화
    원인을 찾았으면 다음 번에 더 빠르게 진단할 수 있도록 메트릭, 알림, 분산 추적을 추가한다. DNS 조회 시간, TCP 연결 수, 재전송율을 대시보드에 올린다.

6. 핵심 답변 예시

짧은 답변 (60초 버전)

"먼저 증상을 수치로 좁히겠습니다. curl -w로 DNS 조회, TCP 연결, TLS 핸드셰이크, 서버 응답 각 단계를 분리해 측정합니다. 어느 단계에서 시간이 소비되는지 알면 범위가 좁혀집니다. 예를 들어 DNS가 200ms라면 TTL이나 DNS 서버 문제, TCP 연결이 300ms라면 keep-alive 미사용이나 연결 풀 문제일 가능성이 높습니다. 각 단계를 배제해 가면서 병목 계층을 특정하겠습니다."

깊이 있는 답변 (면접관이 "더 자세히"라고 물을 때)

"우선 질문 몇 가지로 증상을 좁히겠습니다. '느리다'가 모든 사용자인가 특정 리전인가, 첫 요청만 느린가 지속적으로 느린가, 언제부터 발생했는가를 확인합니다. 특정 리전에서만 느리다면 물리적 RTT나 CDN 미적용이 원인일 가능성이 높습니다. 그런 다음 APM(Datadog, OpenTelemetry 등)에서 TTFB 분포를 보고 서버 처리 전단계에서 지연이 있는지 확인합니다. 서버 처리 시간이 30ms인데 TTFB가 500ms라면 나머지 470ms는 네트워크 또는 로드밸런서에서 소비됩니다. 다음으로 curl -w로 단계별 시간을 측정합니다. DNS가 길면 TTL 조정이나 AnyCast DNS 전환을 검토하고, TLS가 길면 session resumption(session ticket 또는 session ID)이나 OCSP stapling 설정을 확인합니다. TCP 연결이 길면 keep-alive 설정과 connection pool 크기를 점검합니다. 그 다음 mtr로 경로상 패킷 손실이 있는지 확인합니다. 재전송이 많으면 MTU mismatch나 링크 품질 문제일 수 있습니다. 서버 측에서는 ss -s로 TIME_WAIT 과다, backlog 큐 가득 참, 재전송 카운터를 확인합니다. 원인을 좁힌 뒤에는 한 번에 하나씩만 변경해서 효과를 측정하고, 같은 문제가 다시 발생하지 않도록 이 지표들을 대시보드와 알림으로 구성합니다."

7. Trade-off

선택장점단점 / 주의사항
DNS TTL 증가 DNS 조회 횟수 감소, 첫 응답 속도 개선 IP 변경 시 전파 지연이 길어짐 (장애 시 빠른 전환 불가)
HTTP keep-alive 활성화 TCP/TLS 핸드셰이크 비용 제거, 지연 감소 서버 측 열린 연결 수 증가 → fd(파일 디스크립터), 메모리 비용
TLS 세션 재개 (session ticket) TLS 핸드셰이크 RTT 1회 절감 session ticket key 유출 시 과거 트래픽 복호화 가능 (PFS 약화)
CDN 도입 물리적 RTT 감소, 엣지 캐싱, DDoS 완화 비용, CDN 장애 시 전체 영향, 동적 콘텐츠는 효과 제한적
HTTP/2 전환 multiplexing으로 다수 요청 병렬화, 헤더 압축 L4 로드밸런서는 HTTP/2 인식 못함, gRPC 등 특수 고려 필요
패킷 캡처 (tcpdump/Wireshark) 가장 정확한 원인 분석 프로덕션 부하, 암호화된 트래픽은 별도 복호화 환경 필요

8. 흔한 오해

오해 1 "ping이 빠르면 네트워크는 정상이다"

ping은 ICMP 응답만 본다. HTTP 응답이 느릴 때 DNS, TLS, HTTP 처리 시간은 ping에 포함되지 않는다. ping RTT가 1ms여도 DNS 조회가 500ms일 수 있다.

오해 2 "traceroute 홉의 응답 시간이 높으면 그 구간이 병목이다"

일부 라우터는 자신을 향한 ICMP/UDP를 낮은 우선순위로 처리한다. traceroute의 높은 RTT가 실제 경유 트래픽의 지연을 반영하지 않을 수 있다. mtr로 30회 이상 측정해 패킷 손실율을 함께 확인해야 한다.

오해 3 "서버가 같은 리전에 있으면 RTT는 무시해도 된다"

같은 IDC 안에서도 L3 스위치를 거치면 0.1~1ms가 쌓인다. 마이크로서비스가 100개의 내부 API를 순차 호출하면 총 RTT가 100~1000ms에 달할 수 있다. 동일 리전이라도 RTT는 설계의 대상이다.

오해 4 "TCP는 신뢰성 있으니 패킷 손실을 신경 쓰지 않아도 된다"

TCP는 패킷 손실을 재전송으로 복구하지만, 재전송 타이머(RTO)는 최소 200ms다. 1% 손실도 Throughput과 Latency에 큰 영향을 준다. 신뢰성 ≠ 성능 무관.

오해 5 "HTTPS는 HTTP보다 무조건 느리다"

TLS 1.3 + 세션 재개 + HTTP/2를 조합하면 오히려 HTTP/1.1보다 빠를 수 있다. HTTP/2 multiplexing이 여러 요청을 하나의 연결로 처리해 head-of-line blocking을 줄이기 때문이다. "HTTPS = 느리다"는 TLS 1.2 시대의 상식이다.

9. Senior-level Follow-up 질문

Follow-up 질문면접관의 의도답변 포인트
"특정 사용자만 느린 경우와 전체가 느린 경우를 어떻게 다르게 접근합니까?" 범위 특정 능력 특정 사용자 → 리전/ISP/클라이언트 환경 분리. 전체 → 서버 측 공통 병목 가능성 높음.
"curl -w로는 재현이 안 되는데 사용자 환경에서만 느리다면?" 클라이언트 측 관측 경험 RUM(Real User Monitoring), Navigation Timing API, CDN 엣지 로그로 클라이언트 측 데이터를 수집한다.
"연결 수가 많은 환경에서 ss와 netstat 중 어떤 걸 쓰고 그 이유는?" 도구 이해도 ss는 netlink를 직접 조회해 대용량 연결 환경에서도 빠름. netstat는 /proc 파싱으로 느릴 수 있음.
"프로덕션에서 tcpdump를 써도 되는 상황과 그렇지 않은 상황을 구분해보세요." 운영 위험 판단 CPU 부하, 암호화 트래픽 한계, 개인정보 규정. 프로덕션에서는 샘플링, 짧은 캡처 시간, PCAP 보안 처리 필요.
"로드밸런서 뒤에 서버가 여러 대라면 어느 서버 인스턴스에서 캡처해야 합니까?" 분산 환경 트러블슈팅 먼저 LB 로그에서 어떤 인스턴스로 라우팅됐는지 확인, 해당 인스턴스에서만 캡처. 또는 LB 자체에서 upstream 응답 시간을 비교.
"이 문제를 처음 겪는 SRE에게 어떻게 설명하겠습니까?" 커뮤니케이션 능력 계층별 분해 순서와 각 단계에서 봐야 할 지표를 런북(runbook)으로 정리해 공유한다. 도구 명령어가 아니라 "무엇을 확인해 무엇을 배제하는가"를 설명한다.

10. 자기 점검 Rubric

약한 답변 (Junior 수준)

  • "서버 로그를 다시 확인하겠습니다"
  • "코드 레벨에서 최적화를 하겠습니다"
  • "ping을 쳐봤는데 빠릅니다 → 네트워크는 정상입니다"
  • 계층을 언급하지 않고 바로 특정 도구로 점프
  • 측정 없이 "keep-alive를 켜면 됩니다" 같은 해결책을 먼저 제시
  • 도구 이름만 나열하고 무엇을 확인하려는지 말 못함

강한 답변 (Senior 수준)

  • 증상을 "느리다"에서 수치와 범위로 먼저 좁힘
  • 계층별로 순서 있게 분해하고 각 단계에서 배제 기준을 명확히 함
  • 각 도구로 무엇을 측정하고 어떤 숫자가 비정상인지 말할 수 있음
  • 실무에서 실제 겪은 유사 사례를 연결해 설명
  • trade-off를 언급하며 개선 방법의 한계를 함께 말함
  • 해결 후 재발 방지를 위한 관측성 강화를 언급

자기 점검 체크리스트

  • ☐ 증상을 수치화해 좁혔는가?
  • ☐ 계층 순서로 접근했는가 (L7부터 혹은 가능성 높은 계층부터 논리적으로)?
  • ☐ 각 도구로 무엇을 보려는지 말했는가?
  • ☐ 적어도 2개 이상의 trade-off를 언급했는가?
  • ☐ 해결 후 재발 방지를 언급했는가?
  • ☐ 실무 경험이나 유사 사례를 연결했는가?
