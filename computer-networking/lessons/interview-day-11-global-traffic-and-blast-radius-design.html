1. 면접 질문

"여러 리전에 서비스를 배포한다면 트래픽 라우팅, 장애 격리(blast radius 축소), 데이터 일관성을 네트워크 관점에서 어떻게 설계하시겠습니까?"

Variant 1 — "서울 리전이 완전히 다운됐습니다. 사용자가 도쿄 리전으로 자동 전환되게 하려면 무엇을 미리 설계해둬야 합니까?"

Variant 2 — "DNS 기반 failover와 Anycast 기반 failover의 차이를 설명하고, 어떤 상황에서 어떤 것을 선택하시겠습니까?"

이 질문은 Day 1~10에서 다룬 개별 계층 지식(DNS, TCP, LB, VPC, 보안, 성능)을 "하나의 서비스가 아니라 여러 리전에 걸친 전체 시스템" 단위로 확장해서 묻는다. 지원자가 단일 서버/단일 리전을 넘어, 리전 전체가 사라지는 상황까지 가정하고 트래픽과 데이터를 설계할 수 있는지를 본다.

2. 전제 개념 — 멀티 리전 트래픽 분산의 기본기

2-1. 왜 여러 리전에 배포하는가 — "달걀을 한 바구니에 담지 않는다"

하나의 데이터센터(리전)에만 서비스를 두면, 그 리전에 전력 장애, 네트워크 단절, 대규모 정전이 발생하면 서비스 전체가 멈춘다. 여러 도시에 지점을 둔 프랜차이즈가 한 지점에 불이 나도 다른 지점은 영업을 계속하는 것과 같은 원리다. 또한 사용자와 물리적으로 가까운 리전에서 응답하면 왕복 지연(RTT)도 줄어든다. 즉 멀티 리전은 가용성(재해 대응)과 성능(지연 감소) 두 가지 목적을 동시에 가진다.

2-2. GSLB(Global Server Load Balancing) — "어느 리전으로 보낼지 결정하는 장치"

사용자의 요청을 여러 리전 중 어디로 보낼지 결정하는 장치를 GSLB라고 부른다. 우체국이 여러 물류센터 중 배송지에서 가장 가까운 곳을 골라 소포를 넘기는 것과 비슷하다. GSLB는 보통 두 가지 방식 중 하나로 구현된다.

방식동작 원리장점단점
DNS 기반 GSLB
(Route53 Latency/Geo Routing 등)		 같은 도메인을 조회해도 사용자의 위치나 리전별 상태에 따라 서로 다른 IP를 응답으로 돌려줌 구현이 쉽고 대부분의 클라우드에서 기본 제공, 리전별로 완전히 다른 IP 대역 사용 가능 DNS 캐싱/TTL 때문에 failover 반영이 즉시 되지 않음 — 클라이언트/리졸버가 TTL이 지날 때까지 이전 IP를 계속 사용
Anycast 기반 라우팅
(Cloudflare, 대형 CDN 등)		 여러 리전의 서버가 동일한 IP 주소를 BGP로 광고하고, 인터넷 라우팅이 자연스럽게 가장 가까운(또는 네트워크상 가장 짧은 경로의) 리전으로 패킷을 전달 DNS TTL과 무관하게 라우팅 테이블 갱신만으로 즉시 failover, 사용자는 IP를 그대로 유지 BGP 라우팅에 의존하므로 세밀한 트래픽 비율 조정이 어렵고, 자체 구축 난이도가 높음(보통 CDN/클라우드 제공 기능 이용)
비유: DNS 기반은 "안내 데스크에서 매번 어느 창구로 가라고 알려주는 방식"이고, Anycast는 "모든 창구 입구에 같은 번호판을 붙여두고 손님이 자연스럽게 가장 가까운 입구로 걸어 들어오게 하는 방식"이다.

2-3. Active-Active vs Active-Passive — "모든 리전이 일하는가, 대기만 하는가"

  • Active-Active: 모든 리전이 평소에도 트래픽을 처리한다. 한 리전이 죽으면 나머지 리전이 트래픽을 흡수한다. 평상시 자원 활용도가 높지만, 리전 간 데이터 동기화와 쓰기 충돌 처리가 복잡하다.
  • Active-Passive: 평소에는 하나(또는 일부) 리전만 트래픽을 처리하고, 나머지는 대기(standby) 상태로 최신 데이터만 복제받는다. 설계가 단순하지만 평소 자원이 놀고, failover 시 실제로 전환이 되는지 정기적으로 검증(chaos drill)하지 않으면 "막상 필요할 때 안 되는" 위험이 있다.

2-4. Blast Radius(폭발 반경) — "장애가 얼마나 넓게 퍼지는가"

Blast Radius는 하나의 장애가 영향을 미치는 범위를 뜻한다. 폭탄이 터졌을 때 피해 반경을 뜻하는 군사 용어에서 온 표현이다. 모든 리전이 하나의 공유 컴포넌트(예: 전역 인증 서버, 전역 설정 서버)에 의존하면, 그 컴포넌트 하나가 죽었을 때 모든 리전이 동시에 영향을 받는다 — blast radius가 "전체"가 되는 것이다. 반대로 각 리전(또는 셀 단위)이 서로 독립적으로 동작하도록 설계(cell-based architecture / shared-nothing)하면, 한 셀의 장애가 다른 셀로 전파되지 않는다.

2-5. 데이터 일관성과 CAP — "여러 리전이 같은 데이터를 볼 것인가, 빠르게 응답할 것인가"

여러 리전에 데이터를 복제할 때, 모든 리전이 항상 100% 동일한 최신 데이터를 보게(강한 일관성) 하려면 리전 간 통신을 기다려야 해서 지연이 늘고, 리전 간 네트워크가 끊기면 쓰기 자체가 막힐 수 있다(CAP 이론에서 Partition 상황의 Consistency vs Availability 선택). 반대로 각 리전이 자신의 데이터로 즉시 응답(최종 일관성, eventual consistency)하게 하면 빠르고 가용성이 높지만, 리전 간 데이터가 잠시 다를 수 있고 두 리전에서 동시에 같은 데이터를 수정하면 충돌(split-brain)이 발생할 수 있다.

3. 왜 면접에서 이걸 묻는가

  1. 아키텍처 의사결정 능력 — 단일 서버 최적화를 넘어, 여러 리전에 걸친 트래픽과 데이터를 설계할 수 있는지 확인한다. 이는 시스템 설계 면접에서 "10년차 이상"과 "5~7년차"를 가장 뚜렷하게 가르는 지점이다.
  2. 일관성/가용성 trade-off 판단력 — 모든 서비스에 강한 일관성이 필요한 것은 아니다. 결제처럼 정합성이 중요한 데이터와, 상품 목록처럼 약간의 지연 반영이 허용되는 데이터를 구분해 다른 전략을 적용할 수 있는지 본다.
  3. 재해 대응 설계 경험 — 실제로 리전 장애를 경험했거나, failover가 "설계는 되어 있지만 실전에서 작동하지 않는" 흔한 함정을 알고 있는지 확인한다.
면접관이 경계하는 답변: "멀티 리전으로 배포하고 로드밸런서를 두면 됩니다"처럼 트래픽 분산만 언급하고 데이터 일관성, blast radius, failover 검증을 전혀 언급하지 않는 답변. 실제로 멀티 리전 설계의 어려움은 트래픽 라우팅이 아니라 데이터와 상태 관리에 있다.

4. 실무 연결 — 리전 장애 시나리오

실제 서비스에서 흔히 나타나는 리전 장애 대응 타임라인이다. 면접에서 이런 흐름을 구체적으로 그릴 수 있으면 강한 신호가 된다.

  • 서울 리전의 가용영역(AZ) 하나가 아니라 리전 전체 네트워크가 클라우드 사업자 측 장애로 단절됨
  • 서울 리전 헬스체크가 연속 실패, GSLB(Route53 헬스체크 연동 Failover 레코드)가 해당 리전을 라우팅 대상에서 제외 시작
  • 신규 DNS 조회는 도쿄 리전 IP를 반환하지만, TTL(예: 60초)이 지나지 않은 기존 클라이언트는 여전히 서울 리전 IP로 재시도하며 타임아웃 발생
  • 도쿄 리전 트래픽이 급증, 평소 대비 인스턴스 수가 부족해 오토스케일링이 뒤늦게 트리거됨(Day 10에서 다룬 워밍업 문제와 동일 패턴 발생 가능)
  • 도쿄 리전 DB가 서울 리전의 쓰기를 비동기로 복제받던 read replica였다면, failover 시 read replica를 쓰기 가능한 primary로 승격(promotion)하는 과정에서 복제 지연분(RPO)만큼의 데이터 유실 가능성 발생
  • 모든 트래픽이 도쿄 리전으로 안정화, 서울 리전 복구 후 재동기화 및 트래픽 점진적 복귀(failback) 계획 수립

이 타임라인에서 핵심은 "failover는 트래픽 전환 하나로 끝나지 않는다"는 것이다. DNS TTL로 인한 전환 지연, 새 리전의 스케일 준비 부족, DB primary 승격 시 데이터 유실 가능성까지 모두 사전에 설계하고 정기적으로 실전 훈련(chaos engineering, game day)으로 검증해야 실제로 작동한다.

5. 답변 사고 순서

  1. 서비스의 일관성 요구 수준을 먼저 구분한다
    모든 데이터를 동일한 전략으로 다루지 않는다. 결제/잔액처럼 강한 일관성이 필요한 데이터와, 상품 조회/설정처럼 최종 일관성으로 충분한 데이터를 나눈다는 것을 밝힌다.
  2. Active-Active/Active-Passive 중 서비스 특성에 맞는 배포 모델을 선택한다
    쓰기 충돌 처리 비용을 감수하고 가용성/지연을 최대화할지, 단순한 대기계 구조로 운영 복잡도를 낮출지 근거를 들어 설명한다.
  3. 트래픽 라우팅 방식을 결정한다
    DNS 기반(Route53 Latency/Failover Routing)과 Anycast 기반(CDN/Global Accelerator) 중 failover 속도와 구축 난이도를 근거로 선택한다는 것을 밝힌다. 필요하면 DNS TTL을 낮춰 failover 지연을 줄이는 트레이드오프도 언급한다.
  4. Blast Radius를 축소하는 아키텍처를 설계한다
    전역 공유 컴포넌트(전역 인증 서버, 전역 설정)를 최소화하고, 리전 또는 셀 단위로 독립적으로 동작하도록(shared-nothing) 설계한다는 것을 짚는다.
  5. 데이터 복제 전략과 RPO/RTO 목표를 정의한다
    동기 복제(강한 일관성, 지연 증가)와 비동기 복제(빠름, 유실 가능) 중 서비스별로 선택하고, 허용 가능한 데이터 유실 시간(RPO)과 복구 목표 시간(RTO)을 구체적 숫자로 제시한다.
  6. Split-brain 방지 메커니즘을 포함한다
    두 리전이 동시에 자신이 primary라고 착각해 서로 다른 쓰기를 받아들이는 상황을 막기 위한 합의 메커니즘(quorum, fencing)이나 단일 진실 공급원(single source of truth) 설계를 언급한다.
  7. 정기적인 failover 검증 계획을 제시한다
    설계만 해두고 실제로 테스트하지 않으면 진짜 장애 때 작동하지 않을 수 있다는 것을 인지하고, game day/chaos engineering으로 주기적으로 실제 리전 차단을 시뮬레이션한다는 것을 포함한다.

6. 핵심 답변 예시

짧은 답변 (60초 버전)

"먼저 모든 데이터를 동일하게 다루지 않겠습니다. 결제나 잔액처럼 정합성이 중요한 데이터는 단일 리전에서 쓰기를 처리하고 다른 리전은 읽기 전용 복제본을 두는 구조로, 상품 목록이나 정적 콘텐츠처럼 약간의 지연 반영이 허용되는 데이터는 여러 리전에서 모두 쓰기 가능한 Active-Active 구조로 나누겠습니다. 트래픽 라우팅은 failover 속도가 중요하므로 헬스체크와 연동된 DNS Failover Routing을 기본으로 하되, TTL을 짧게(예: 60초 이내) 설정해 전환 지연을 줄이고, 정적 자산이나 API 게이트웨이 앞단은 Anycast 기반 CDN으로 더 빠른 전환을 확보하겠습니다. 전역 인증 서버처럼 모든 리전이 의존하는 단일 컴포넌트는 최소화해서 한 리전 장애가 전체 서비스로 번지는 blast radius를 줄이고, primary 승격 시 발생할 수 있는 split-brain을 막기 위해 쿼럼 기반 합의를 두겠습니다. 마지막으로 이 전체 failover가 실제로 작동하는지 정기적으로 리전 차단 훈련(game day)을 통해 검증하겠습니다."

깊이 있는 답변 (면접관이 "더 자세히"라고 물을 때)

"조금 더 구체적으로 나누면, 우선 데이터 계층을 서비스별 일관성 요구 수준으로 분류합니다. 결제/재고처럼 초과 판매나 이중 차감이 치명적인 데이터는 단일 리전에 쓰기 권한을 집중(single-writer)하고 다른 리전은 비동기 read replica로 두되, RPO(허용 가능한 데이터 유실 시간)를 예를 들어 5초 이내로 목표하고 복제 지연을 모니터링합니다. 반대로 조회 위주의 데이터는 여러 리전에서 모두 쓰기가 가능한 다중 마스터 구조나, 아예 CRDT 같은 충돌 자동 해결 구조를 검토합니다. 트래픽 라우팅은 두 계층으로 나눕니다. CDN/정적 자산 앞단은 Anycast 기반으로 두어 라우팅 테이블 갱신만으로 즉시 failover되게 하고, API 트래픽은 Route53 같은 DNS 기반 Failover/Latency Routing에 리전별 헬스체크를 연동합니다. 다만 DNS 기반은 TTL이 지나기 전까지는 일부 클라이언트가 여전히 죽은 리전으로 요청을 보낼 수 있다는 한계가 있으므로, 클라이언트 SDK 차원에서도 특정 리전 연결이 반복 실패하면 다른 리전 엔드포인트로 재시도하는 클라이언트 사이드 failover 로직을 함께 둡니다. Blast Radius 관점에서는 전역으로 공유되는 컴포넌트(전역 인증 서버, 전역 설정 저장소, 전역 레이트리미터)를 최대한 리전별로 독립시켜 셀 기반(cell-based) 구조로 만듭니다. 이렇게 하면 서울 리전의 인증 서버 장애가 도쿄 리전에는 전혀 영향을 주지 않습니다. Primary 승격 과정에서는 반드시 쿼럼(quorum) 기반 합의를 거쳐, 네트워크 파티션 중에 두 리전이 동시에 자신을 primary로 착각해 쓰기를 받는 split-brain을 방지합니다. 마지막으로, 이 모든 설계는 실제로 작동한다는 것을 검증하지 않으면 의미가 없습니다. 정기적으로 프로덕션의 일부 트래픽에 대해 실제 리전을 격리하는 game day를 진행해서, DNS failover 반영 시간, 신규 리전의 오토스케일링 워밍업 시간, DB 승격 소요 시간을 실측하고 RTO 목표와 비교해 개선합니다."

7. Trade-off

선택장점단점 / 주의사항
DNS 기반 GSLB 구축이 쉽고 대부분의 클라우드/DNS 서비스가 기본 지원, 리전별 IP 대역을 자유롭게 운영 가능 TTL 때문에 failover가 즉시 반영되지 않음, 클라이언트/리졸버의 DNS 캐싱 동작에 따라 전환 속도가 들쭉날쭉함
Anycast 기반 라우팅 TTL과 무관하게 라우팅 테이블 갱신만으로 즉시 failover, 사용자 IP가 그대로 유지되어 커넥션 재사용에 유리 세밀한 트래픽 비율 조정이 어렵고, 자체 구축은 BGP 피어링 등 네트워크 엔지니어링 난이도가 높아 보통 CDN/클라우드가 제공하는 기능에 의존
Active-Active 배포 평시 자원 활용도가 높고, 리전 장애 시 즉시 트래픽 흡수 — 사용자 체감 다운타임이 짧음 다중 쓰기 충돌 해결(합의, CRDT 등)이 필요해 설계·운영 복잡도가 크게 증가
Active-Passive 배포 단일 쓰기 지점으로 데이터 정합성 관리가 단순함 대기 리전이 평소 검증되지 않아 실제 failover 시 "생각보다 안 됨" 위험, 정기 훈련 없이는 신뢰할 수 없음
동기(synchronous) 리전 간 복제 데이터 유실 없이 강한 일관성 보장 리전 간 물리적 거리로 인한 왕복 지연이 쓰기 지연에 그대로 반영되고, 네트워크 파티션 시 쓰기 자체가 막힐 수 있음(CAP의 C 선택)
비동기(asynchronous) 리전 간 복제 쓰기 지연이 낮고 네트워크 파티션에도 로컬 쓰기는 계속 가능(CAP의 A 선택) failover 시 아직 복제되지 않은 데이터(RPO만큼)가 유실될 수 있고, 복제 지연이 커지면 오래된 데이터를 정상 데이터로 착각할 위험

8. 흔한 오해

오해 1 "멀티 리전으로만 배포하면 자동으로 고가용성이 확보된다"

트래픽을 여러 리전에 분산하는 것과, 실제 장애 시 자동으로 안전하게 failover되는 것은 다른 문제다. 헬스체크 설계, DNS TTL, 신규 리전의 스케일 준비, DB 승격 절차까지 모두 갖춰야 실제로 작동한다. 이 중 하나라도 빠지면 "리전은 여러 개인데 장애는 여전히 전체 서비스로 번지는" 상황이 발생한다.

오해 2 "강한 일관성이 항상 더 안전하고 좋은 선택이다"

강한 일관성은 리전 간 통신을 기다려야 하므로 지연이 늘고, 네트워크 파티션 상황에서는 가용성을 포기해야 한다(CAP 이론). 모든 데이터에 강한 일관성을 적용하면 오히려 평상시 성능이 나빠지고, 파티션 상황에서 서비스 전체가 응답 불가 상태에 빠질 수 있다. 데이터 특성에 맞게 선택적으로 적용해야 한다.

오해 3 "DNS TTL을 0에 가깝게 낮추면 문제가 해결된다"

TTL을 극단적으로 낮추면 failover는 빨라지지만, 모든 요청마다 DNS 조회가 발생해 DNS 인프라 자체에 부하가 커지고, 일부 리졸버는 아주 짧은 TTL을 무시하고 자체 최소값을 적용하기도 한다. TTL은 "빠른 전환"과 "DNS 부하/신뢰성" 사이의 트레이드오프이지 무조건 낮추는 것이 정답이 아니다.

오해 4 "Failover는 설계해두면 실제 장애 때도 그대로 작동한다"

대기(standby) 리전은 평소 실제 트래픽을 받아본 적이 없어서, 스케일링 설정 오류, 만료된 인증서, 잘못된 설정값 등이 실전에서야 발견되는 경우가 매우 흔하다. 정기적인 game day/chaos engineering으로 실제 트래픽 일부를 격리해 검증하지 않으면, 설계 문서상의 failover와 실제 failover는 다를 수 있다.

오해 5 "Active-Active가 항상 Active-Passive보다 우월한 선택이다"

Active-Active는 다중 쓰기 충돌 처리, 데이터 동기화, 운영 복잡도 측면에서 비용이 훨씬 크다. 트래픽이 크지 않거나 쓰기 충돌 해결 비용이 감당되지 않는 서비스라면 단순한 Active-Passive가 더 합리적인 선택일 수 있다. "더 정교해 보이는 구조"가 항상 더 나은 구조는 아니다.

9. Senior-level Follow-up 질문

Follow-up 질문면접관의 의도답변 포인트
"Anycast와 DNS 기반 GSLB를 함께 쓸 수 있습니까?" 단일 기법에 국한되지 않고 계층별로 조합하는 설계 능력 확인 CDN/엣지 레이어는 Anycast로 즉시 라우팅하고, 그 뒤의 오리진/API 레이어는 DNS 기반 Failover Routing으로 리전을 선택하는 계층적 조합이 실무에서 흔하다는 것을 설명한다.
"Split-brain을 구체적으로 어떻게 막으시겠습니까?" 합의 알고리즘/쿼럼 개념을 실제로 이해하고 있는지 확인 과반수(quorum) 합의 없이는 어떤 리전도 자신을 primary로 승격하지 못하게 하거나, 외부 합의 서비스(예: 별도 리전의 락 서비스)를 통해 "지금 누가 primary인지"에 대한 단일 진실 공급원을 유지한다는 것을 설명한다.
"RPO와 RTO를 어떻게 구체적인 숫자로 정의하고 검증하시겠습니까?" 추상적 원칙을 측정 가능한 목표로 구체화하는 능력 확인 서비스별로 허용 가능한 데이터 유실 시간(RPO)과 복구 목표 시간(RTO)을 SLA/SLO 형태로 정의하고, game day에서 실제 failover 소요 시간을 측정해 목표 대비 실측치를 비교·개선한다는 것을 설명한다.
"모든 리전이 의존하는 전역 인증 서버가 있다면 blast radius를 어떻게 줄이시겠습니까?" 공유 컴포넌트가 만드는 숨은 단일 장애점(SPOF)을 찾아낼 수 있는지 확인 인증 토큰 검증을 리전별로 로컬에서 캐싱/자체 검증(예: 공개키 기반 서명 검증) 가능하도록 만들어 전역 서버 의존을 줄이거나, 전역 인증 서버 자체를 다중 리전 Active-Active로 별도 설계한다는 방향을 제시한다.
"이 아키텍처의 월간 인프라 비용 증가를 경영진에게 어떻게 설명하시겠습니까?" 기술적 결정을 비즈니스 언어(비용 대비 리스크)로 전달하는 능력 확인 과거 장애로 인한 매출/신뢰 손실 추정치와 멀티 리전 운영 비용을 비교하고, 일관성이 필요한 데이터에만 선택적으로 비용이 큰 전략(동기 복제, Active-Active)을 적용해 전체 비용을 최적화했다는 점을 설명한다.

10. 자기 점검 Rubric

약한 답변 (Junior 수준)

  • "멀티 리전에 배포하고 로드밸런서를 둔다"는 수준에서 답변이 끝남
  • 모든 데이터에 동일한 일관성 전략을 적용한다고 답함 (구분 없음)
  • DNS 기반과 Anycast 기반 라우팅의 차이를 설명하지 못함
  • Blast Radius나 전역 공유 컴포넌트로 인한 숨은 단일 장애점을 고려하지 않음
  • Failover 설계를 실제로 검증(game day)해야 한다는 개념이 없음

강한 답변 (Senior 수준)

  • 서비스/데이터 특성별로 일관성 요구 수준을 나누어 다른 복제 전략을 적용함
  • DNS 기반과 Anycast 기반 라우팅을 계층별로 조합하고 각각의 한계(TTL vs BGP 의존)를 설명함
  • Cell-based/shared-nothing 구조로 blast radius를 축소하는 구체적 방법을 제시함
  • RPO/RTO를 구체적 숫자로 정의하고 split-brain 방지 메커니즘(quorum)을 설명함
  • 정기적인 game day/chaos engineering으로 failover를 실전 검증한다는 것을 포함함
  • 기술적 결정을 비용/리스크 관점에서 비즈니스 언어로도 설명할 수 있음

자기 점검 체크리스트

  • ☐ 모든 데이터에 동일한 일관성 전략을 적용하지 않고 서비스 특성별로 구분했는가?
  • ☐ DNS 기반 GSLB와 Anycast 기반 라우팅의 차이와 각각의 한계를 설명했는가?
  • ☐ Active-Active/Active-Passive의 trade-off를 근거를 들어 선택했는가?
  • ☐ 전역 공유 컴포넌트로 인한 blast radius 확대를 인식하고 축소 방법을 제시했는가?
  • ☐ RPO/RTO를 구체적인 목표로 정의했는가?
  • ☐ Split-brain 방지 메커니즘을 언급했는가?
  • ☐ 설계를 실제로 검증하는 정기 훈련(game day)까지 답변에 포함했는가?
