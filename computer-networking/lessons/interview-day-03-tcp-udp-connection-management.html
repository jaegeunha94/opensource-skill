1. 면접 질문

"TCP가 UDP와 다르게 '신뢰성 있는 전송'을 제공한다고 하죠. 정확히 무엇을 보장하고, 그 보장을 위해 어떤 대가를 치릅니까? 그리고 그 대가가 실제 서비스에서 문제가 된 적이 있습니까?"

"TCP는 신뢰성 있고 UDP는 신뢰성 없다"는 문장은 교과서 첫 줄에 나온다. 그래서 대부분의 지원자가 이 문장을 그대로 되풀이하는 것으로 답을 끝낸다. 하지만 면접관이 진짜 알고 싶은 것은 "신뢰성"이라는 단어가 구체적으로 어떤 메커니즘(순서 보장, 손실 복구, 흐름 제어, 혼잡 제어)의 합인지, 그리고 그 메커니즘 하나하나가 지연·처리량·구현 복잡도에 어떤 대가를 요구하는지다. 이걸 설명하지 못하면 "TCP는 좋고 UDP는 나쁘다"는 이분법에 갇힌 사람으로 보인다.

2. 전제 개념 — 알아야 할 기초

2-1. "연결(connection)"이라는 것은 실제로 무엇을 의미하는가

IP 계층은 각 패킷을 독립적으로 취급한다 — 이전 패킷과 지금 패킷이 같은 대화의 일부인지 IP는 모른다(비연결형, connectionless). TCP는 그 위에 "이 두 지점 사이에 순서가 있는 대화가 진행 중"이라는 상태(state)를 양쪽 호스트의 메모리에 만든다. 이 상태에는 시퀀스 번호, 윈도우 크기, 혼잡 윈도우, 타이머 등이 들어있다. UDP는 이런 상태를 전혀 만들지 않고 패킷을 그냥 보낸다.

비유: UDP로 메시지를 보내는 것은 엽서를 우체통에 넣는 것과 같다 — 보내고 나면 도착했는지, 순서가 맞는지 알 방법이 없다. TCP는 전화 통화와 비슷하다 — 먼저 상대가 받았는지 확인하고("여보세요?"), 대화 중에는 "네, 들려요"를 계속 주고받으며, 잡음이 있으면 "방금 뭐라고 하셨죠?"라고 되묻는다. 이 확인 절차 자체가 신뢰성이고, 동시에 통화 시간(지연)을 늘리는 원인이다.

2-2. TCP의 "신뢰성"은 하나의 기능이 아니라 4가지 메커니즘의 합이다

면접에서 좋은 인상을 주는 핵심은 "신뢰성"을 뭉뚱그리지 않고 아래처럼 쪼개서 설명하는 것이다.

메커니즘보장하는 것핵심 동작대가
연결 수립양쪽이 통신할 준비가 됐다는 상호 확인3-way handshake (SYN → SYN-ACK → ACK)데이터 전송 전 1 RTT 지연
순서 보장 + 손실 복구보낸 순서대로, 빠짐없이 도착시퀀스 번호, ACK, 타임아웃 기반 재전송, 중복 ACK 기반 빠른 재전송손실 시 재전송 대기(RTO)로 지연 발생, 순서 보장을 위해 먼저 도착한 패킷도 대기(HOL blocking)
흐름 제어(Flow Control)수신자가 처리할 수 있는 만큼만 전송 (수신자 보호)슬라이딩 윈도우, 수신 윈도우(rwnd) 광고수신자 버퍼가 작으면 송신자가 아무리 빨라도 전송 속도 제한
혼잡 제어(Congestion Control)네트워크 전체가 감당할 수 있는 만큼만 전송 (네트워크 보호)slow start, congestion avoidance, fast retransmit/recovery, cwnd연결 초기와 손실 직후에는 대역폭을 다 쓰지 못함

여기서 중요한 구분: 흐름 제어는 "상대방(수신자)이 못 버티게 만들지 않기 위한" 제어이고, 혼잡 제어는 "네트워크 경로(라우터, 링크) 전체가 못 버티게 만들지 않기 위한" 제어다. 둘 다 "너무 빨리 보내지 마라"는 결과로 보이지만 보호 대상이 다르다. 이 구분을 못하면 시니어 답변으로 인정받기 어렵다.

2-3. 3-way handshake — 왜 2번이 아니라 3번인가

Client                          Server
  |------ SYN (seq=x) --------->|   "나는 x부터 시작할 거야"
  |<--- SYN-ACK (seq=y, ack=x+1)-|   "알겠고, 나는 y부터 시작할게. 네 x는 받았어"
  |------ ACK (ack=y+1) -------->|   "네 y도 받았어. 이제 대화 시작"

2-way(SYN → ACK)로는 부족한 이유는, 서버가 클라이언트의 SYN을 받았다는 사실을 클라이언트가 확인할 방법이 없기 때문이다. 3번째 ACK가 있어야 양쪽 모두 "상대가 내 시작 번호를 알고 있다"는 것을 확신하고 대화를 시작할 수 있다. 이것이 "신뢰성"의 가장 첫 단계 — 대화를 시작하기 전에 서로의 상태를 동기화하는 것이다.

2-4. 재전송은 "언제" 일어나는가 — 두 가지 트리거

  • 타임아웃 기반 재전송(RTO): ACK가 정해진 시간(RTO, Retransmission Timeout) 안에 오지 않으면 재전송한다. RTO는 고정값이 아니라 RTT의 평균과 변동성(RTT variance)을 계속 측정해서 동적으로 조정된다. RTO가 지나야 재전송하므로 가장 느린 복구 방법이다.
  • 빠른 재전송(Fast Retransmit): 같은 ACK 번호가 3번 중복으로 오면("나는 아직 세그먼트 5를 못 받았어"를 3번 반복해서 말하는 것) 타임아웃을 기다리지 않고 즉시 재전송한다. 이게 훨씬 빠르다.

실무에서 "TCP 재전송이 느리다"고 느껴지는 대부분의 경우는 타임아웃 기반 재전송에 걸린 것이다. RTO는 보통 200ms~1초 수준에서 시작하므로, 패킷 손실이 발생하면 사용자는 그 시간만큼 응답 지연을 체감한다.

2-5. 슬라이딩 윈도우와 혼잡 윈도우 — 왜 처음엔 느리다가 빨라지는가

연결 초기에는 네트워크 상태를 모르기 때문에 아주 작은 혼잡 윈도우(cwnd)로 시작해서(slow start), ACK를 받을 때마다 지수적으로 늘린다. 어느 시점(임계값, ssthresh)부터는 선형으로 천천히 늘린다(congestion avoidance). 손실이 감지되면 cwnd를 큰 폭으로 줄인다(TCP Reno/Cubic 등 알고리즘마다 감소 폭이 다름). 실제 전송 속도는 min(수신 윈도우, 혼잡 윈도우)로 결정된다 — 즉 수신자가 아무리 버퍼가 커도 네트워크가 혼잡하다고 판단되면 속도는 못 올라간다.

실무 함의: 짧은 연결(예: 매 요청마다 새 TCP 연결)에서는 slow start가 끝나기도 전에 전송이 끝나버려 대역폭을 제대로 활용하지 못한다. 이것이 "커넥션 재사용(keep-alive, connection pooling)"이 성능에 중요한 이유 중 하나 — 한 번 데워진(warm) 연결의 cwnd를 재사용할 수 있기 때문이다.

2-6. UDP는 무엇을 "포기"하고 무엇을 "얻는가"

UDP는 3-way handshake, 재전송, 순서 보장, 흐름/혼잡 제어를 모두 하지 않는다. 그 대신 지연이 없고(핸드셰이크 불필요), 헤더가 작고(8바이트), 순서가 뒤바뀌어도 애플리케이션이 즉시 처리할 수 있다. 이건 "품질이 낮다"가 아니라 "신뢰성을 애플리케이션 계층으로 넘긴다"는 설계 선택이다. DNS, 실시간 영상/음성(WebRTC), 게임 서버, QUIC(HTTP/3의 기반)이 UDP를 쓰는 이유가 여기 있다 — 오래된 데이터를 재전송받는 것보다 최신 데이터를 빨리 받는 게 더 중요한 경우가 있기 때문이다.

3. 왜 면접에서 이걸 묻는가

  1. 1단계 필터 (암기 vs 분해) — "TCP는 신뢰성, UDP는 비신뢰성"이라는 이분법에서 멈추는지, 아니면 신뢰성을 연결 수립/순서 보장/흐름 제어/혼잡 제어로 쪼개서 설명할 수 있는지 확인한다.
  2. 2단계 필터 (지식 vs 대가 인식) — 신뢰성에는 반드시 대가(지연, 처리량 제한, HOL blocking)가 따른다는 것을 알고 있는지 본다. "TCP가 무조건 더 좋다"고 말하면 trade-off 감각이 없다는 신호다.
  3. 3단계 필터 (지식 vs 실전 판단) — 실제로 프로토콜 선택이나 설정(keep-alive, timeout, buffer size) 때문에 장애나 성능 문제를 겪어본 적이 있는지, 그리고 그때 어떻게 진단했는지를 follow-up으로 확인한다.
면접관이 피하고 싶은 답변: "TCP는 연결형이고 신뢰성이 있어서 데이터가 순서대로 정확하게 전달됩니다. UDP는 비연결형이고 빠르지만 신뢰성이 없습니다." 이 문장에는 오답이 하나도 없지만, 왜 그런지, 무엇을 대가로 치르는지, 언제 반대 선택을 하는지가 전혀 없다. 위키백과 요약과 구분되지 않는다.

4. 실무 연결

실제 장애 시나리오 A — "가끔 한 요청만 3초씩 느려진다"

API 응답 시간 대시보드에서 p50은 정상인데 p99만 튀는 패턴이 있었다. tcpdump로 캡처해보니 특정 요청에서만 패킷 손실 후 RTO 기반 재전송이 발생하고 있었고, 그 RTO가 약 1초로 잡혀 있어 재전송이 2~3회 겹치면 3초까지 늘어났다. 애플리케이션 로그만 봤다면 "가끔 느린 원인 불명"으로 끝났겠지만, TCP 재전송 타이머의 존재를 알고 있었기 때문에 네트워크 구간(특정 AZ 간 손실률 증가)으로 원인을 좁힐 수 있었다.

실제 장애 시나리오 B — 대량 파일 업로드에서 처리량이 기대보다 낮음

고대역폭·고지연(장거리 리전 간) 구간에서 대용량 업로드 처리량이 이론적 대역폭의 일부밖에 안 나온 적이 있다. 원인은 TCP 수신 윈도우 크기가 BDP(Bandwidth-Delay Product = 대역폭 × RTT)보다 작아서 송신자가 ACK를 기다리며 노는 시간이 생긴 것이었다. 소켓 버퍼 크기(윈도우 스케일링 포함)를 조정하거나 병렬 연결을 사용해서 해결했다. 이건 "네트워크가 느리다"가 아니라 "설정된 윈도우가 이 RTT/대역폭 조합에 비해 작다"는 정량적 문제였다.

실제 장애 시나리오 C — 실시간 화상회의 품질 저하를 TCP로 "고쳐서" 오히려 나빠진 사례

초기 설계에서 화상회의 미디어 스트림을 신뢰성을 위해 TCP로 보냈더니, 손실이 발생할 때마다 재전송을 기다리며 뒤에 있는 최신 프레임까지 함께 지연되는 현상(Head-of-Line blocking)이 발생했다. 사용자 입장에서는 화면이 끊기다가 갑자기 몰아서 재생되는 형태로 체감됐다. UDP 기반(WebRTC)으로 바꾸고 애플리케이션 레벨에서 "오래된 프레임은 버리고 최신 프레임 우선"으로 처리하도록 바꾸자 체감 품질이 개선됐다. 신뢰성이 항상 사용자 경험을 개선하는 것은 아니라는 것을 보여주는 사례다.

5. 답변 사고 순서

  1. "신뢰성"을 정의부터 분해한다
    뭉뚱그리지 않고 연결 수립 / 순서 보장·손실 복구 / 흐름 제어 / 혼잡 제어 4가지로 나눠 말한다.
  2. 각 메커니즘의 동작 원리를 한 문장으로 압축
    3-way handshake, 시퀀스 번호+ACK+재전송, 슬라이딩 윈도우, cwnd(slow start/congestion avoidance)를 순서대로 짚는다.
  3. 각 메커니즘의 "대가"를 바로 이어서 말한다
    연결 수립 → 1 RTT 지연, 순서 보장 → HOL blocking, 재전송 → RTO 지연, 혼잡 제어 → 초기 저속.
  4. UDP가 이 모든 걸 포기하는 이유(설계 의도)를 설명
    신뢰성을 포기하는 게 아니라 "신뢰성 구현 책임을 애플리케이션으로 옮기는 것"이라는 프레이밍을 사용한다.
  5. 실제 겪은 사례로 연결
    재전송 지연, 윈도우 크기, 또는 TCP → UDP 전환 경험 중 하나를 구체적 수치와 함께 제시한다.
  6. "어떻게 선택하는가"로 마무리
    손실 허용 가능 여부, 지연 민감도, 순서 중요도, 트래픽 패턴(짧은 요청 vs 스트리밍)을 기준으로 프로토콜/설정을 고른다고 요약한다.

6. 핵심 답변 예시

짧은 답변 (60초 버전)

"TCP의 신뢰성은 하나의 기능이 아니라 네 가지 메커니즘의 합이라고 생각합니다. 3-way handshake로 연결 상태를 동기화하고, 시퀀스 번호와 ACK로 순서와 손실을 관리하고, 흐름 제어로 수신자를 보호하고, 혼잡 제어로 네트워크 경로를 보호합니다. 각각은 대가가 있습니다 — 핸드셰이크는 1 RTT 지연을, 순서 보장은 head-of-line blocking을, 혼잡 제어는 연결 초반 저속 구간을 만듭니다. UDP는 이걸 다 포기하는 대신 지연 없이 바로 보내고, 신뢰성이 필요하면 애플리케이션이 직접 구현하도록 넘깁니다. 그래서 저는 '순서와 손실 복구가 중요한가 vs 최신성이 더 중요한가'를 기준으로 프로토콜을 고릅니다."

깊이 있는 답변 (면접관이 "더 자세히"라고 물을 때)

"조금 더 구체적으로 말씀드리면, TCP의 재전송에는 두 가지 경로가 있습니다. 하나는 RTO(타임아웃) 기반 재전송인데, 이건 RTT 평균과 변동성을 계속 추정해서 타이머를 동적으로 잡기 때문에 최소 200ms에서 수 초까지 걸릴 수 있습니다. 다른 하나는 같은 ACK가 3번 중복되면 타임아웃을 기다리지 않고 즉시 재전송하는 빠른 재전송인데, 이게 훨씬 빠릅니다. 실무에서 'p99만 몇 초씩 튄다'는 증상은 대부분 전자, 즉 RTO 기반 재전송에 걸렸을 가능성이 높습니다. 흐름 제어와 혼잡 제어도 구분해서 설명드리고 싶은데, 흐름 제어는 수신자의 처리 능력(rwnd)을 보호하는 것이고 혼잡 제어는 중간 네트워크 경로의 용량(cwnd)을 보호하는 것입니다. 실제 전송 속도는 이 둘 중 작은 값으로 제한됩니다. 그래서 대역폭도 충분하고 수신 버퍼도 큰데 처리량이 안 나온다면, RTT가 큰 구간에서 윈도우 크기가 BDP(대역폭×RTT)보다 작게 설정된 건 아닌지 확인해야 합니다. 실제로 리전 간 대용량 업로드에서 이 문제를 겪었는데, 소켓 버퍼와 윈도우 스케일링을 조정하고 나서야 이론 대역폭에 근접했습니다. 반대로 화상회의 미디어 스트림을 TCP로 보냈을 때는 손실 시 재전송을 기다리느라 최신 프레임까지 지연되는 head-of-line blocking이 체감 품질을 떨어뜨려서, UDP 기반으로 바꾸고 오래된 프레임은 버리는 방식으로 개선한 경험도 있습니다. 결국 '신뢰성'은 공짜가 아니고, 트래픽 특성에 따라 어떤 대가를 감수할지 선택하는 문제라고 봅니다."

7. Trade-off

선택장점단점 / 주의사항
TCP 사용 순서 보장, 손실 자동 복구, 혼잡 상황에서 네트워크 전체 안정성에 기여 연결 수립 지연(1 RTT, TLS까지 포함하면 더 늘어남), head-of-line blocking, 짧은 연결에서는 slow start로 대역폭 활용도 낮음
UDP 사용 지연 최소화, 헤더 오버헤드 작음, 최신 데이터 우선 전달 가능 순서/손실 복구를 애플리케이션이 직접 구현해야 함, 혼잡 제어가 없어 네트워크를 과도하게 점유할 위험(그래서 QUIC 등은 자체 혼잡 제어를 둠)
커넥션 재사용 (Keep-alive / Pooling) 핸드셰이크·slow start 반복을 피해 지연과 처리량 모두 개선 유휴 연결이 서버 자원(파일 디스크립터, 메모리)을 점유, 방화벽/LB의 idle timeout과 충돌 가능
재전송 타이머를 짧게 설정 손실 복구가 빨라짐 정상 지연을 손실로 오판해 불필요한 재전송(spurious retransmission) 증가, 네트워크 부하 가중

8. 흔한 오해

오해 1 "TCP는 신뢰성이 있으니 무조건 UDP보다 좋은 선택이다"

신뢰성은 순서와 손실 복구를 보장하지만, 그 대가로 지연과 head-of-line blocking을 만든다. 실시간성이 정확성보다 중요한 경우(음성/영상, 게임 상태 동기화)에는 오래된 데이터를 재전송받는 것보다 최신 데이터를 즉시 받는 것이 사용자 경험에 더 좋다. "신뢰성 있음 = 항상 우월함"은 성립하지 않는다.

오해 2 "흐름 제어와 혼잡 제어는 같은 것이다"

흐름 제어는 수신자의 버퍼(처리 능력)를 보호하기 위한 것이고, 혼잡 제어는 중간 네트워크 경로(라우터, 링크)의 용량을 보호하기 위한 것이다. 하나는 종단 간(end-to-end) 문제고 다른 하나는 네트워크 전체의 문제다. 실제 전송 속도는 두 윈도우 중 더 작은 값(min(rwnd, cwnd))으로 결정되므로, 성능 문제를 진단할 때 어느 쪽이 병목인지 구분해야 한다.

오해 3 "패킷 손실이 생기면 TCP가 항상 빠르게 재전송해서 복구한다"

빠른 재전송(중복 ACK 3회)은 상대적으로 빠르지만, 손실된 패킷이 스트림의 마지막 부분이거나 후속 패킷이 충분히 도착하지 않아 중복 ACK가 3개 안 모이면 타임아웃 기반 재전송(RTO)으로 넘어가고, 이건 수백 ms에서 초 단위까지 걸릴 수 있다. "TCP니까 알아서 빠르게 복구된다"는 전제로 지연 문제를 접근하면 원인을 놓친다.

오해 4 "UDP는 신뢰성이 전혀 없으니 중요한 데이터에는 절대 쓸 수 없다"

DNS, QUIC(HTTP/3), WebRTC 모두 UDP 기반이지만 매우 중요한 트래픽을 처리한다. 이들은 UDP 위에 애플리케이션 레벨에서 필요한 만큼만 재전송, 순서 보장, 심지어 자체 혼잡 제어(QUIC)를 구현한다. "UDP = 아무 보장 없음"이 아니라 "OS 커널이 아니라 애플리케이션이 무엇을 보장할지 선택할 수 있다"가 정확한 이해다.

오해 5 "연결이 많아지면 그냥 커넥션 풀을 키우면 된다"

커넥션을 늘리면 핸드셰이크·slow start 비용은 줄지만, 각 연결이 유휴 상태로 서버의 파일 디스크립터와 메모리를 점유한다. 또한 클라이언트-서버 사이에 로드밸런서나 방화벽의 idle timeout이 있으면, 응용 계층의 keep-alive 주기와 그 timeout이 어긋나 예상치 못한 연결 끊김(RST)이 발생할 수 있다. 커넥션 풀 크기는 "많을수록 좋다"가 아니라 서버 자원, 중간 장비의 timeout, 트래픽 패턴을 함께 고려해서 정해야 한다.

9. Senior-level Follow-up 질문

Follow-up 질문면접관의 의도답변 포인트
"TCP 연결이 4-way handshake로 종료되는 이유는? 그리고 TIME_WAIT은 왜 필요합니까?" 연결 생명주기 전체를 이해하는지, 상태 관리 실무 이슈까지 아는지 각 방향(양방향 연결이므로)이 독립적으로 FIN을 보내야 하므로 4단계다. TIME_WAIT은 마지막 ACK가 유실됐을 때 재전송된 FIN을 처리하고, 지연 도착한 옛 패킷이 새 연결과 혼동되지 않도록 하기 위한 대기 시간이다. 대량 요청 서버에서 TIME_WAIT 소켓이 쌓여 포트가 고갈되는 문제(특히 짧은 연결을 많이 여는 구조)와 연결 지어 설명한다.
"슬로우 스타트가 매번 처음부터 다시 시작되는 게 왜 문제가 됩니까? 어떻게 완화합니까?" 프로토콜 지식을 성능 최적화 실무로 연결할 수 있는지 짧은 요청/응답이 반복되는 서비스는 매번 처음부터 slow start를 거치느라 대역폭을 다 못 쓴다. HTTP keep-alive/커넥션 풀링, HTTP/2의 멀티플렉싱(연결 하나 재사용), TCP Fast Open 등으로 완화한다.
"멀티플렉싱을 도입한 HTTP/2가 특정 상황에서 오히려 성능이 나빠질 수 있는 이유는?" TCP의 순서 보장 특성을 상위 프로토콜 설계와 연결할 수 있는지 HTTP/2는 하나의 TCP 연결 위에서 여러 스트림을 멀티플렉싱하는데, TCP 자체는 세그먼트 순서를 보장하므로 한 스트림의 패킷 손실이 같은 연결의 다른 스트림까지 막는다(TCP 레벨 head-of-line blocking). 이게 QUIC/HTTP3가 UDP 기반으로 스트림별 독립적인 손실 복구를 하는 이유다.
"수신자가 처리 속도를 못 따라가는데 송신자가 계속 보내면 어떤 일이 벌어집니까?" 흐름 제어의 실제 동작(Zero Window)까지 아는지 수신 버퍼가 가득 차면 수신자는 윈도우 크기 0(Zero Window)을 광고해 송신을 멈추게 한다. 이후 여유가 생기면 Window Update를 보내 재개시킨다. 이 과정에서 Window Update 패킷이 유실되면 데드락처럼 보이는 지연이 생길 수 있어 송신자는 주기적으로 probe를 보낸다.
"주니어 개발자가 '손실 안 나면 되니까 그냥 다 TCP로 통일하자'고 하면 어떻게 지도하겠습니까?" 기술 리딩 역량, trade-off를 언어화해서 설득하는 능력 트래픽 특성(실시간성 vs 정확성 요구, 요청 크기와 빈도, RTT)에 따라 적합한 프로토콜이 다르다는 것을 구체적 사례(화상회의, DNS, 대용량 배치 전송)로 설명하고, "통일"보다 "용도별 선택 기준을 문서화"하는 방향으로 유도한다고 답한다.

10. 자기 점검 Rubric

약한 답변 (Junior 수준)

  • "TCP는 연결형·신뢰성, UDP는 비연결형·비신뢰성"이라고 말하고 끝냄
  • 흐름 제어와 혼잡 제어를 구분하지 못하거나 같은 것으로 설명
  • 재전송이 "언제, 얼마나 걸려서" 일어나는지 구체적으로 말하지 못함
  • 신뢰성의 대가(지연, HOL blocking, 초기 저속)를 언급하지 않음
  • UDP를 "품질이 낮은 선택"으로만 이해하고 설계 의도를 설명하지 못함

강한 답변 (Senior 수준)

  • 신뢰성을 연결 수립/순서·손실 복구/흐름 제어/혼잡 제어로 분해해서 설명
  • RTO 기반 재전송과 빠른 재전송의 차이와 각각의 지연 규모를 안다
  • 흐름 제어(수신자 보호)와 혼잡 제어(네트워크 보호)를 구분해서 설명
  • 커넥션 재사용이 slow start 회피와 어떻게 연결되는지 성능 관점에서 설명
  • UDP를 "신뢰성을 애플리케이션으로 위임하는 설계"로 프레이밍
  • 실제 겪은 지연/처리량/품질 문제를 프로토콜 메커니즘으로 설명 가능

자기 점검 체크리스트

  • ☐ "신뢰성"을 4가지 메커니즘(연결 수립, 순서/손실 복구, 흐름 제어, 혼잡 제어)으로 쪼개 설명했는가?
  • ☐ 각 메커니즘의 대가(지연, HOL blocking, 초기 저속 등)를 함께 언급했는가?
  • ☐ RTO 기반 재전송과 빠른 재전송의 차이를 설명했는가?
  • ☐ 흐름 제어와 혼잡 제어를 명확히 구분했는가?
  • ☐ UDP를 "포기"가 아니라 "책임 위임"으로 설명했는가?
  • ☐ 실무 사례(지연, 처리량, 실시간성 문제)를 구체적 수치나 상황으로 제시했는가?
