Day 9 — 무선 네트워크 & 최신 트렌드

오늘은 Wi-Fi(IEEE 802.11) 표준의 발전 과정, SDN(Software Defined Networking) 개념, 그리고 클라우드 네트워킹 기초를 학습합니다. Day 1~8에서 쌓은 OSI/TCP-IP, IP 라우팅, 보안 지식이 여기서 연결됩니다.

오늘의 학습 목표

  • Wi-Fi 세대별 표준(802.11 b/g/n/ac/ax)의 주파수·속도 차이를 설명한다.
  • CSMA/CA, SSID, BSS, AP 로밍 개념을 이해한다.
  • SDN의 제어 평면·데이터 평면 분리 원리와 OpenFlow를 설명한다.
  • VPC, CDN, 로드밸런서, 서비스 메시 개념을 구분한다.

1. 무선 네트워크 기초

1-1. 유선 vs 무선의 차이

유선 이더넷은 CSMA/CD(충돌 감지)를 사용하지만, 무선은 숨겨진 노드 문제(Hidden Node Problem) 때문에 충돌을 감지하기 어렵습니다. 그래서 무선은 CSMA/CA(Collision Avoidance)를 사용합니다.

항목유선(이더넷)무선(Wi-Fi)
매체 접근CSMA/CDCSMA/CA
충돌 처리충돌 감지 후 재전송충돌 회피 — RTS/CTS 핸드셰이크
보안 위협물리 접근 필요전파 도청 가능 — WPA3 암호화 필요
대역폭 공유스위치로 전이중(full-duplex)AP에 연결된 모든 기기가 채널 공유

1-2. CSMA/CA 동작 원리

  1. 전송 전 채널이 비어 있는지 감지(Listen Before Talk).
  2. 비어 있으면 DIFS(Distributed Inter-Frame Space) 대기.
  3. 무작위 Backoff 카운터 설정 → 카운터가 0이 되면 전송.
  4. 선택적으로 RTS(Request to Send) / CTS(Clear to Send)를 교환하여 숨겨진 노드 문제 완화.
  5. 수신자가 ACK를 반환 → 없으면 재전송.

숨겨진 노드 문제: A와 C가 모두 B에게 전송 중이지만, A와 C는 서로의 전파를 감지하지 못하는 경우. RTS/CTS로 AP가 중재합니다.

3. SDN (Software Defined Networking)

3-1. 전통적 네트워크의 한계

전통 네트워크에서 스위치·라우터는 제어 평면(Control Plane)과 데이터 평면(Data Plane)이 같은 장비 안에 있습니다. 설정 변경마다 장비 하나씩 SSH로 접속해 CLI를 입력해야 합니다. 수백 대 장비를 운영하면 설정 불일치가 생기기 쉽습니다.

3-2. SDN 핵심 개념

SDN은 제어 평면을 중앙 집중 컨트롤러로 분리합니다.

┌─────────────────────────────────┐
│      애플리케이션 평면           │   (트래픽 분석, 정책 앱)
│  Application Plane              │
└──────────────┬──────────────────┘
               │ Northbound API (REST)
┌──────────────▼──────────────────┐
│      제어 평면 (Controller)      │   (OpenDaylight, ONOS, Floodlight)
│  Control Plane                  │
└──────────────┬──────────────────┘
               │ Southbound API (OpenFlow, NETCONF)
┌──────────────▼──────────────────┐
│      데이터 평면 (Switches)      │   (패킷 포워딩만 담당)
│  Data Plane / Forwarding Plane  │
└─────────────────────────────────┘
평면역할예시
애플리케이션 평면비즈니스 정책 정의, 트래픽 분석로드밸런서 앱, 방화벽 정책 앱
제어 평면라우팅 결정, 플로우 테이블 생성·배포ONOS, OpenDaylight
데이터 평면플로우 테이블에 따라 패킷 포워딩OpenFlow 지원 스위치, Open vSwitch

3-3. OpenFlow 플로우 테이블

SDN 스위치는 각 패킷을 플로우(Flow) 단위로 처리합니다. 플로우 테이블 항목은 매칭 조건 + 액션으로 구성됩니다.

매칭 필드액션우선순위
dst_ip=10.0.0.2, proto=TCP, dst_port=80포트 3으로 전달100
src_ip=192.168.1.0/24드롭(보안 정책)200
*컨트롤러에 문의(Packet-In)0

3-4. SDN 시뮬레이터

SDN 플로우 테이블 시뮬레이터

컨트롤러 (제어 평면)

ONOS Controller
관리 스위치: SW1, SW2, SW3
등록 플로우: 3개

SW1 플로우 테이블 (데이터 평면)

#매칭 조건액션우선순위

3-5. NFV (Network Function Virtualization)

SDN과 함께 자주 등장하는 개념입니다. 전용 하드웨어(방화벽 어플라이언스, 로드밸런서 장비)를 소프트웨어로 구현해 범용 서버에서 실행합니다.

기존 방식NFV
전용 방화벽 장비 구매·설치VM/컨테이너로 vFirewall 실행
로드밸런서 어플라이언스소프트웨어 로드밸런서 (HAProxy, Nginx)
WAN 최적화 박스vWAN 소프트웨어
확장: 장비 추가 구매확장: VM 인스턴스 증가(탄력적 스케일)

4. 클라우드 네트워킹 기초

4-1. VPC (Virtual Private Cloud)

클라우드 제공자(AWS, GCP, Azure) 내에서 논리적으로 격리된 사설 네트워크 공간입니다. Day 3의 IP 서브넷 개념이 그대로 적용됩니다.

VPC (10.0.0.0/16)
├── Public Subnet  (10.0.1.0/24)  ← 인터넷 게이트웨이 연결
│     └── EC2 웹 서버 (10.0.1.10)
└── Private Subnet (10.0.2.0/24)  ← 외부 직접 접근 불가
      └── RDS DB 서버 (10.0.2.20)
            ↑
        NAT Gateway를 통해서만 아웃바운드 가능
컴포넌트설명Day 연결
Internet GatewayVPC ↔ 인터넷 연결점Day 3 라우터 개념
NAT GatewayPrivate 서브넷 → 인터넷 아웃바운드Day 7 NAT
Security Group인스턴스 단위 스테이트풀 방화벽Day 7 방화벽
NACL서브넷 단위 스테이트리스 ACLDay 7 방화벽
VPC Peering두 VPC 간 프라이빗 통신Day 3 라우팅
VPN Gateway온프레미스 ↔ VPC IPSec 터널Day 7 VPN

4-2. 로드밸런서 (Load Balancer)

트래픽을 여러 서버에 분산해 가용성과 처리량을 높입니다.

종류OSI 계층분산 기준예시
L4 LB (Transport)4계층IP:Port 기반AWS NLB, HAProxy TCP
L7 LB (Application)7계층URL 경로, 헤더, 쿠키AWS ALB, Nginx, Traefik

L7 LB 예시: /api/* → 백엔드 서버 그룹 A, /static/* → CDN으로 리다이렉트.

4-3. CDN (Content Delivery Network)

전 세계 엣지 서버(PoP, Point of Presence)에 정적 콘텐츠를 캐싱해 사용자와 가까운 서버에서 응답합니다. Day 6의 HTTP 캐시(Cache-Control) 헤더가 CDN 동작의 기반입니다.

사용자 (서울) → CDN PoP (서울) → [캐시 히트] → 즉시 응답
                              → [캐시 미스] → 오리진 서버 (미국) 요청 후 캐싱

4-4. 서비스 메시 (Service Mesh)

마이크로서비스 환경에서 서비스 간 통신을 관리하는 인프라 계층입니다. 각 서비스 옆에 사이드카 프록시(Envoy 등)를 배치해 로드밸런싱·암호화·관찰 가능성을 처리합니다.

기능설명
트래픽 관리A/B 테스트, 카나리 배포, 타임아웃/재시도
mTLS서비스 간 상호 TLS 인증 자동화
관찰 가능성분산 트레이싱(Jaeger), 메트릭(Prometheus)
대표 구현Istio, Linkerd, Consul Connect
