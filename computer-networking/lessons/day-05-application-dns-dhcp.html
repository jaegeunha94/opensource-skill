Day 5 — 응용 계층 : DNS와 DHCP

학습 목표

- DNS가 도메인 이름을 IP 주소로 변환하는 계층적 과정(재귀 쿼리)을 순서대로 설명할 수 있다.

- DNS 레코드 타입(A, AAAA, CNAME, MX, NS, TXT)의 역할을 구분할 수 있다.

- DHCP DORA 4단계를 순서대로 말하고 각 단계의 의미를 설명할 수 있다.

- DNS 캐싱과 TTL이 네트워크 성능에 어떻게 기여하는지 이해한다.

1. 응용 계층이란?

Day 4에서 TCP/UDP가 프로세스 간 통신을 담당한다는 것을 배웠다. 응용 계층(Application Layer)은 그 위에 위치하며, 실제 사용자가 직접 상호작용하는 서비스가 정의되는 계층이다.

프로토콜포트전송 계층역할
DNS53UDP (주로) / TCP도메인 → IP 변환
DHCP67 / 68UDPIP 주소 자동 할당
HTTP80TCP웹 페이지 전송
HTTPS443TCP암호화된 웹 페이지 전송
SMTP25TCP이메일 전송
SSH22TCP원격 보안 접속

오늘은 네트워크가 작동하기 위한 인프라 프로토콜인 DNS와 DHCP를 집중적으로 살펴본다.

2. DNS — 인터넷의 전화번호부

사람은 www.google.com 같은 이름을 기억하기 쉽지만, 네트워크는 142.250.196.36 같은 IP 주소로 통신한다. DNS(Domain Name System)는 이 둘 사이의 번역을 담당한다.

왜 UDP인가? — DNS 조회는 대부분 단 한 번의 요청-응답 교환으로 끝난다. TCP의 3-way handshake 오버헤드(3개 패킷)를 감수하기보다 UDP 한 패킷으로 질문하고 한 패킷으로 답받는 편이 훨씬 빠르다. 응답이 오지 않으면 애플리케이션이 재질의한다. 단, 응답이 512바이트를 초과하거나 영역 이전(Zone Transfer)에는 TCP를 사용한다.

4. DNS 캐싱과 TTL

DNS 조회는 매번 루트 서버부터 시작하면 느리고 비효율적이다. 이 문제를 캐싱(Caching)으로 해결한다.

  • 로컬 리졸버, 운영체제, 브라우저 모두 DNS 결과를 캐시한다.
  • TTL(Time To Live) — 레코드마다 부여된 캐시 유효 시간(초 단위). TTL이 지나면 다시 조회한다.
  • TTL이 짧으면 → 변경사항이 빠르게 반영되지만 DNS 트래픽 증가.
  • TTL이 길면 → 캐시 히트율이 높아 빠르지만 레코드 변경이 늦게 전파된다.
# 명령줄로 DNS 레코드와 TTL 확인
$ dig www.example.com

;; ANSWER SECTION:
www.example.com.   3600   IN   A   93.184.216.34
                    ^^^^
                    TTL (3600초 = 1시간)

로컬 DNS 캐시 확인 / 초기화
Linux: resolvectl statistics / resolvectl flush-caches
macOS: sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
Windows: ipconfig /displaydns / ipconfig /flushdns

5. DNS 레코드 타입

DNS에 저장되는 데이터의 단위를 리소스 레코드(Resource Record, RR)라 하며 타입에 따라 역할이 다르다.

레코드 타입의미예시
A도메인 → IPv4 주소example.com. A 93.184.216.34
AAAA도메인 → IPv6 주소example.com. AAAA 2606:2800:220:1:248:1893:25c8:1946
CNAME도메인의 별칭(Canonical Name). 체인으로 연결 가능.www.example.com. CNAME example.com.
MX이메일을 받을 서버. 우선순위(priority) 숫자 낮을수록 우선.example.com. MX 10 mail.example.com.
NS해당 도메인의 권한 있는 네임서버 지정example.com. NS ns1.example.com.
TXT임의 텍스트. SPF(이메일 위조방지), DKIM, 도메인 소유 확인 등에 사용.example.com. TXT "v=spf1 include:_spf.google.com ~all"
PTRIP → 도메인(역방향 조회, Reverse DNS)34.216.184.93.in-addr.arpa. PTR example.com.

CNAME vs A 레코드 — www.example.com을 example.com의 CNAME으로 만들면, example.com의 A 레코드만 바꿔도 두 도메인이 함께 업데이트된다. 단, 루트 도메인(@)에는 CNAME을 쓸 수 없다(RFC 표준 제한).

6. DHCP — IP 주소 자동 할당

컴퓨터가 네트워크에 연결될 때 IP 주소, 서브넷 마스크, 기본 게이트웨이, DNS 서버 주소를 수동으로 설정하면 번거롭고 오류가 많다. DHCP(Dynamic Host Configuration Protocol)는 이 과정을 자동화한다.

DHCP는 브로드캐스트 기반이므로 TCP 연결이 불가능한 상태(IP 없음)에서도 동작해야 한다. 따라서 UDP를 사용하며, 클라이언트 포트 68 → 서버 포트 67로 통신한다.

DORA — DHCP 4단계

DHCP 과정을 DORA라는 약자로 기억하자: Discover → Offer → Request → Acknowledge.

클라이언트                            DHCP 서버
    │                                      │
    │ 1. DHCPDISCOVER (브로드캐스트) ──────►│  "IP 달라!"
    │                                      │
    │ ◄──── 2. DHCPOFFER ────────────────── │  "192.168.1.100 어때? (임시 제안)"
    │                                      │
    │ 3. DHCPREQUEST (브로드캐스트) ───────►│  "그 IP 쓸게!"
    │                                      │
    │ ◄──── 4. DHCPACK ──────────────────── │  "확정! 임대 기간: 24시간"
    │                                      │
    │ (IP: 192.168.1.100, GW: 192.168.1.1) │
단계메시지방향내용
1DHCPDISCOVER클라이언트 → 브로드캐스트네트워크에 DHCP 서버가 있는지 탐색. 출발지 IP: 0.0.0.0, 목적지 IP: 255.255.255.255
2DHCPOFFER서버 → 클라이언트사용 가능한 IP 주소와 임대 조건 제안. 아직 확정 아님.
3DHCPREQUEST클라이언트 → 브로드캐스트제안받은 IP 주소를 사용하겠다고 선택 공지. 다른 서버들에게도 알림(묵시적 거절).
4DHCPACK서버 → 클라이언트IP 주소, 서브넷 마스크, 게이트웨이, DNS 서버, 임대 기간 등 최종 확인.

7. DHCP 임대(Lease)와 갱신

DHCP 서버는 IP 주소를 영구적으로 주는 것이 아니라 일정 기간 동안 임대(Lease)한다.

  • 임대 기간이 절반 지나면(T1) 클라이언트가 서버에 갱신(Renewal)을 요청한다.
  • 임대 기간의 7/8이 지나면(T2) 브로드캐스트로 다시 시도한다(기존 서버 응답 없을 때).
  • 만료될 때까지 갱신이 안 되면 클라이언트는 IP 사용을 포기하고 DORA를 처음부터 다시 시작한다.
임대 기간 = 24시간 예시
  0h ──────── 12h (T1: Renewal) ──── 21h (T2: Rebind) ──── 24h (만료)
  │            │                      │                      │
 DORA         갱신 시도               브로드캐스트 갱신       IP 포기

DHCP와 정적 IP — 서버, 프린터처럼 항상 같은 IP를 써야 하는 기기는 DHCP 서버에서 MAC 주소 기반 IP 예약(Reservation)을 설정하거나 정적 IP를 수동으로 부여한다. 이 방식은 일관성은 높지만 관리 부담이 있다.
