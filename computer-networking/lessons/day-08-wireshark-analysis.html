Day 8 — Wireshark 패킷 분석

오늘은 네트워크 분석 도구의 대표 주자 Wireshark를 사용해 실제 패킷을 들여다봅니다. 필터 문법, TCP 스트림 추적, DNS·HTTP 트래픽 해석까지 익히고 나면 앞서 배운 모든 프로토콜이 "살아 움직이는" 형태로 보이기 시작합니다.

학습 목표
Wireshark 인터페이스 구조 이해 · 디스플레이 필터 문법 작성 · TCP 스트림 추적으로 HTTP 요청/응답 복원 · DNS 쿼리/응답 패킷 해석 · TLS Handshake 패킷 흐름 확인

1. Wireshark 인터페이스 구조

Wireshark는 NIC(Network Interface Card)를 통해 흐르는 패킷을 실시간으로 캡처하고, 프로토콜을 자동으로 분석해서 보여주는 GUI 기반 패킷 분석기입니다.

패널내용
패킷 목록 (상단) 캡처된 패킷 번호, 시간, 출발/목적지 IP, 프로토콜, 요약 정보를 한 줄씩 표시
패킷 상세 (중간) 선택한 패킷의 계층별 헤더를 펼쳐볼 수 있는 트리 뷰 (Ethernet → IP → TCP → HTTP 순)
패킷 바이트 (하단) 원시 헥스(Hex) + ASCII 데이터. 상세 뷰의 필드를 클릭하면 해당 바이트가 하이라이트됨

캡처 필터 vs 디스플레이 필터 — 캡처 필터는 캡처 시작 전에 적용해 저장할 패킷 자체를 줄입니다 (BPF 문법). 디스플레이 필터는 이미 캡처된 패킷 중 화면에 표시할 것만 걸러냅니다. 오늘은 디스플레이 필터를 주로 다룹니다.

1-1. .pcap / .pcapng 파일

Wireshark는 패킷을 .pcap 또는 .pcapng 형식으로 저장합니다. CTF, 네트워크 트러블슈팅, 사고 조사에서 표준적으로 사용되는 포맷입니다. 파일을 열면 캡처 당시의 패킷을 그대로 재생할 수 있습니다.

2. 디스플레이 필터 (Display Filter)

Wireshark 상단 필터 바에 입력해 사용합니다. 올바른 문법이면 초록색, 오류면 빨간색으로 표시됩니다.

2-1. 기본 문법

패턴예시설명
프로토콜dnsDNS 패킷만 표시
필드 비교ip.addr == 8.8.8.8출발지 또는 목적지가 8.8.8.8인 패킷
출발지 IPip.src == 192.168.1.10해당 IP에서 보낸 패킷만
목적지 포트tcp.dstport == 443443 포트로 가는 TCP 패킷
포트 범위tcp.port == 80포트 80 (출발 또는 목적지)
ANDip.addr == 1.1.1.1 && dns두 조건 모두 만족
ORdns || http둘 중 하나라도 만족
NOT!arpARP 패킷 제외
문자열 포함http.request.uri contains "login"URI에 "login"이 포함된 요청
TCP 플래그tcp.flags.syn == 1 && tcp.flags.ack == 0SYN 패킷만 (3-way handshake 시작)

2-2. 자주 쓰는 필터 레시피

# 특정 호스트 간 통신만 보기
ip.addr == 192.168.1.10 && ip.addr == 8.8.8.8

# HTTP GET 요청만
http.request.method == "GET"

# DNS 응답만 (qr 플래그 = 1)
dns.flags.response == 1

# TLS Client Hello 패킷
tls.handshake.type == 1

# TCP 재전송 패킷 (네트워크 품질 확인)
tcp.analysis.retransmission

# 큰 패킷 (1400 바이트 초과)
frame.len > 1400

2-3. 인터랙티브 — 필터 시뮬레이터

아래에서 필터를 입력하면 가상 패킷 목록이 필터링됩니다. 실제 Wireshark 없이 필터 문법을 연습해보세요.

힌트: dns / http / tls / arp / icmp / tcp / tcp.port == 443 / ip.addr == 8.8.8.8 / !arp

3. DNS 트래픽 분석

필터 dns를 입력하면 DNS 쿼리(Query)와 응답(Response)을 볼 수 있습니다. 패킷을 선택하면 하단 상세 뷰에서 다음 정보를 확인할 수 있습니다.

Wireshark 필드의미
dns.flags.response0 = 쿼리, 1 = 응답
dns.qry.name질의한 도메인 이름
dns.qry.type레코드 타입 (1=A, 28=AAAA, 5=CNAME …)
dns.a응답에 포함된 IPv4 주소 (A 레코드)
dns.time쿼리 → 응답 왕복 시간 (RTT)

3-1. DNS 쿼리/응답 패킷 예시

[패킷 #42] 클라이언트 → DNS 서버 (쿼리)
  Src: 192.168.1.5:53412  →  Dst: 8.8.8.8:53  UDP
  Domain Name System (query)
    Transaction ID: 0xa3f1
    Flags: 0x0100 (Standard query)
    Questions: 1
      www.example.com: type A, class IN

[패킷 #43] DNS 서버 → 클라이언트 (응답)
  Src: 8.8.8.8:53  →  Dst: 192.168.1.5:53412  UDP
  Domain Name System (response)
    Transaction ID: 0xa3f1         ← 쿼리와 동일한 ID로 매칭
    Flags: 0x8180 (Standard query response, No error)
    Answers: 1
      www.example.com: type A, class IN, addr 93.184.216.34
    Time: 0.023 sec

Transaction ID로 쿼리-응답 쌍을 연결합니다. 같은 ID를 가진 패킷 두 개가 DNS 요청 한 쌍입니다. dns && ip.addr == 8.8.8.8 필터로 특정 리졸버와의 통신만 볼 수 있습니다.

4. TCP 스트림 추적 (Follow TCP Stream)

HTTP 통신은 여러 TCP 패킷에 분산되어 있습니다. Wireshark의 Follow TCP Stream 기능을 쓰면 한 연결의 모든 패킷을 모아 사람이 읽을 수 있는 텍스트로 재구성합니다.

사용 방법

  1. 패킷 목록에서 HTTP 패킷 우클릭
  2. Follow → TCP Stream 선택
  3. 클라이언트 데이터(빨간색/초록색)와 서버 데이터(파란색)를 구분해 읽기

5. HTTPS / TLS 패킷 분석

HTTPS 트래픽은 암호화되어 내용을 바로 볼 수 없지만, TLS 핸드셰이크 과정은 평문이라 Wireshark에서 분석할 수 있습니다.

패킷 (Handshake 순서)Wireshark 표시주요 정보
① Client HelloTLSv1.3 Client Hello지원 암호화 스위트, SNI(서버 도메인명), 클라이언트 난수
② Server HelloTLSv1.3 Server Hello선택된 암호화 스위트, 서버 난수
③ CertificateCertificate서버 인증서 (발급자, 유효기간, 공개키)
④ Application DataApplication Data암호화된 HTTP 페이로드 (내용 불가 확인)

SNI (Server Name Indication) — Client Hello에 포함된 도메인명으로, 하나의 IP가 여러 HTTPS 사이트를 호스팅할 때 서버가 올바른 인증서를 선택하는 데 쓰입니다. Wireshark에서 tls.handshake.extensions_server_name 필드로 확인할 수 있습니다.

5-1. TLS 필터 조합

# Client Hello만 보기 (새 TLS 연결 탐지)
tls.handshake.type == 1

# 특정 도메인으로의 TLS 연결
tls.handshake.extensions_server_name == "api.example.com"

# 암호화된 데이터 전송 패킷만
tls.record.content_type == 23

# TLS 경고 (연결 오류 탐지)
tls.alert_message.level == 2

6. 실전 패킷 분석 절차

실제 트러블슈팅 시 권장하는 순서입니다.

  1. 전체 트래픽 파악 — 필터 없이 Statistics → Conversations로 가장 많은 트래픽을 주고받는 호스트 쌍 확인
  2. 이상 패킷 포착 — tcp.analysis.flags (재전송, 중복 ACK, 윈도우 가득 참 등)로 TCP 문제 탐지
  3. 관심 스트림 격리 — ip.addr == x.x.x.x로 관심 호스트 패킷만 남기기
  4. 프로토콜 심층 분석 — Follow TCP/UDP Stream, 또는 해당 프로토콜 필터로 상세 확인
  5. 타임라인 재구성 — Statistics → Flow Graph로 패킷 흐름 시각화

6-1. 주요 Statistics 메뉴

메뉴용도
Statistics → I/O Graphs시간에 따른 트래픽 양 그래프 (대역폭 스파이크 탐지)
Statistics → Protocol Hierarchy캡처 내 프로토콜 분포 (어떤 프로토콜이 얼마나 차지하는지)
Statistics → EndpointsIP별, MAC별 트래픽 통계
Statistics → ConversationsIP 쌍 간 총 패킷/바이트 수
Analyze → Expert Information재전송, 오류, 경고를 자동으로 분류해서 보여줌
