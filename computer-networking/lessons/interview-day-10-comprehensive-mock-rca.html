1. 면접 질문 — 장애 시나리오

"오늘 오후 2시부터 결제 API의 에러율이 0.1%에서 8%로 튀었고, 응답 지연 p99가 300ms에서 4초로 치솟았습니다. 담당 SRE가 확인한 사실은 다음과 같습니다: (1) 결제 서버 CPU/메모리는 정상, (2) 애플리케이션 로그에는 'connection reset' 과 'upstream timeout' 이 섞여 있음, (3) 문제는 특정 리전(도쿄)의 로드밸런서 뒤에 있는 인스턴스 그룹에서만 발생, (4) 같은 시각에 해당 리전에서 DNS 조회 지연이 늘었다는 모니터링 알림도 떴음, (5) 30분 전 해당 리전 오토스케일링이 트래픽 증가로 새 인스턴스 6대를 추가 기동함. 당신이 온콜 엔지니어라면 어떤 순서로 원인을 좁히고, 어떤 계층에서 무엇을 확인하며, 최종적으로 어떻게 RCA를 정리해서 팀에 공유하시겠습니까?"

이 질문이 Day 1~9와 다른 점은, 단일 개념(DNS만, TLS만, LB만)을 묻지 않고 여러 계층의 단서를 동시에 던져놓고 지원자가 스스로 우선순위를 정해 좁혀나가는지를 본다는 것이다. 실제 장애는 항상 이렇게 여러 신호가 뒤섞여 온다 — DNS 지연 알림이 진짜 원인일 수도 있고, 오토스케일링으로 새로 붙은 인스턴스의 워밍업 문제(콜드 커넥션 풀, 캐시 미스)일 수도 있고, 로드밸런서의 헬스체크 설정 문제일 수도 있다. 면접관은 "정답 하나"를 맞히는 것보다, 근거 없이 하나를 찍지 않고 가설을 세우고 데이터로 검증하며 좁혀가는 절차를 보여주는지를 평가한다.

2. 전제 개념 — RCA를 구조화하는 프레임

2-1. RCA(Root Cause Analysis)란 — "증상이 아니라 근본 원인을 찾는 절차"

RCA는 장애가 발생했을 때 눈에 보이는 증상(에러율 상승, 지연 증가)에서 멈추지 않고, 그 증상을 유발한 근본 원인을 찾아 재발을 막는 절차다. 병원에서 "열이 난다"는 증상만 보고 해열제를 주는 것과, 왜 열이 나는지 원인(감염, 염증)을 찾아 치료하는 것의 차이와 같다. 네트워크 장애 RCA는 특히 "여러 계층에서 동시에 이상 신호가 보일 때, 어느 것이 원인이고 어느 것이 결과(증상의 파급)인지"를 구분하는 것이 핵심이다.

2-2. 계층적 가설 트리 — "넓게 좁히고, 좁게 파고든다"

장애 초기에는 원인 후보가 너무 많다. 이걸 한 번에 다 조사하면 시간을 낭비한다. 그래서 범위를 좁히는 질문(어디까지 영향을 받았나) → 계층을 좁히는 질문(어느 계층 문제인가) → 원인을 좁히는 질문(그 계층에서 무엇이 바뀌었나) 순서로 가설 트리를 좁혀나간다.

단계질문이 문제 시나리오에서 확인할 것
1. 범위(Blast Radius)전체 리전인가, 특정 리전인가? 전체 트래픽인가, 특정 경로인가?도쿄 리전만 영향 → 글로벌 코드 배포/DB 문제가 아니라 리전 국소적 원인일 가능성이 큼
2. 계층(Layer)L3/L4(네트워크·전송) 문제인가, L7(애플리케이션·프로토콜) 문제인가, 이름 해석(DNS) 문제인가, 인프라(오토스케일링, LB) 문제인가?'connection reset', 'upstream timeout'은 L4~L7 경계, DNS 지연 알림은 이름 해석 계층, 오토스케일링 이벤트는 인프라 계층 — 세 계층이 동시에 등장
3. 타이밍 상관관계(Correlation)증상 발생 시각과 인프라 변경 시각이 정확히 겹치는가?오토스케일링(30분 전)과 에러율 급증(2시)의 시간차를 확인 — 정확히 같은 시각이면 강한 단서, 시간차가 크면 별개 사건일 가능성
4. 근본 원인(Root Cause)왜 그 변경이 이 증상을 유발했는가?새 인스턴스의 헬스체크 통과 전 트래픽 유입, 콜드 커넥션 풀, 리전 DNS 리졸버 부하 등 구체적 메커니즘 확인
핵심: 여러 단서가 동시에 보일 때 초보자는 하나를 "직감"으로 찍어 조사하지만, 숙련자는 범위 → 계층 → 타이밍 → 메커니즘 순으로 후보를 줄이며, 각 단계에서 반증 가능한 데이터를 확인한다. "이게 원인이 아니라면 어떤 데이터가 그것을 보여줄까"를 항상 함께 생각하는 것이 RCA 사고의 핵심이다.

3. 왜 면접에서 이걸 묻는가

  1. 1단계 필터 (통합 사고력) — 개별 개념(DNS, TCP, LB)을 아는 것과, 여러 계층의 단서가 동시에 주어졌을 때 이를 하나의 인과관계로 재구성할 수 있는 것은 다른 능력이다. 면접관은 지원자가 "지식의 목록"이 아니라 "장애를 다루는 사고 절차"를 가지고 있는지 확인한다.
  2. 2단계 필터 (우선순위 판단력) — 5개의 단서를 동시에 받았을 때 어느 것부터 확인할지, 왜 그 순서인지 설명할 수 있는지 본다. 모든 것을 동시에 조사하겠다는 답변은 실무에서 통하지 않는다 — 장애 중에는 시간이 곧 비용이다.
  3. 3단계 필터 (커뮤니케이션과 재발 방지) — 원인을 찾는 것으로 끝나지 않고, 이를 팀에 어떻게 공유하고 재발을 어떻게 막을지까지 설계하는지 확인한다. 이는 10년차 이상에게 기대하는 "장애를 조직의 자산으로 만드는" 역량이다.
면접관이 피하고 싶은 답변: "DNS 알림이 떴으니 DNS가 원인입니다"처럼 하나의 단서만 보고 단정하거나, "일단 서버를 재시작하겠습니다"처럼 근본 원인 분석 없이 증상만 완화하는 답변. 둘 다 "왜 그런지"에 대한 근거 있는 추론이 빠져 있다.

4. 실무 연결 — 타임라인 기반 장애 재구성

실제로 이런 시나리오를 조사하면 아래와 같은 타임라인이 나오는 경우가 많다. 면접에서는 이런 식으로 시간 순서를 재구성해 보여주는 것 자체가 강력한 신호다.

  • 도쿄 리전 트래픽 증가로 오토스케일링 트리거, 신규 인스턴스 6대 기동 시작
  • 신규 인스턴스가 부팅 완료, 헬스체크 엔드포인트가 200 응답을 반환하기 시작 (그러나 내부적으로 DB 커넥션 풀은 아직 채워지지 않은 상태)
  • LB가 헬스체크 통과를 확인하고 신규 인스턴스로 트래픽 라우팅 시작
  • 신규 인스턴스가 첫 요청을 받아 DB/캐시 연결을 그제서야 새로 맺기 시작 — 이 과정에서 내부 서비스 디스커버리(DNS)에 짧은 시간 다수의 조회가 몰림
  • 내부 DNS 리졸버가 순간적으로 부하를 받아 조회 지연 발생 (모니터링 알림 발생 시점)
  • DNS 지연으로 신규 인스턴스의 DB 커넥션 수립이 느려지며 LB의 upstream timeout 임계치를 초과하는 요청 증가, 일부는 LB가 연결을 강제 종료(connection reset)
  • 에러율 8%, p99 4초로 관측 — 온콜 알림 발생

이 타임라인을 재구성하면 근본 원인은 "신규 인스턴스가 실제로 트래픽을 처리할 준비(warm-up)가 되기 전에 헬스체크만 통과한 상태로 LB가 트래픽을 흘려보냈고, 그 워밍업 과정(다수의 DB/캐시 연결 수립)이 내부 DNS에 순간 부하를 유발해 연쇄적으로 타임아웃이 발생한 것"이다. DNS 지연은 원인이 아니라 동일한 근본 원인(부실한 워밍업 절차)의 또 다른 증상이었던 셈이다. 이런 "증상처럼 보이지만 사실은 같은 원인의 파생 결과"를 구분하는 것이 RCA의 핵심 난이도다.

5. 답변 사고 순서

  1. 영향 범위(Blast Radius)를 먼저 확정한다
    전체 리전/글로벌인지 도쿄 리전 국소인지, 전체 API인지 결제 API만인지 대시보드로 확인한다. "도쿄만"이라는 사실 자체가 글로벌 배포나 DB 스키마 변경 같은 광범위 원인을 1차로 배제시킨다는 것을 밝힌다.
  2. 타이밍 상관관계를 확인한다
    에러율 급증 시각(14:00)과 오토스케일링 시각(13:30), DNS 지연 알림 시각을 나란히 놓고 시간차를 본다. 30분의 간격이 있다면 오토스케일링이 "직접" 원인이 아니라 "간접적으로" 원인일 가능성(워밍업 지연)을 가설로 세운다는 것을 짚는다.
  3. 가장 최근 변경(오토스케일링)을 최우선 용의자로 놓고 검증한다
    장애 직전 변경 사항이 가장 흔한 원인이라는 원칙(change correlation)에 따라, 신규 인스턴스 6대의 헬스체크 grace period, 커넥션 풀 초기화 로직, 워밍업 절차를 먼저 확인한다.
  4. DNS 지연을 원인 후보와 파생 증상 후보로 동시에 열어둔다
    DNS 지연이 "독립적인 원인"인지 "신규 인스턴스의 워밍업이 유발한 결과"인지, 내부 DNS 리졸버의 쿼리량 그래프를 확인해 오토스케일링 시각과 겹치는지로 구분한다는 것을 밝힌다.
  5. LB/헬스체크 설정에서 근본 메커니즘을 확정한다
    헬스체크가 "프로세스가 떠 있는지"만 확인하고 "실제 의존 리소스(DB, 캐시) 연결까지 완료됐는지"는 확인하지 못했다면, 이것이 근본 원인이라는 것을 구체적으로 짚는다 (얕은 헬스체크 vs readiness probe의 차이).
  6. 즉시 완화(mitigation)와 근본 수정(fix)을 구분해서 제시한다
    즉시 완화: 신규 인스턴스를 일시적으로 LB에서 제외하거나 트래픽 유입 속도를 늦춤(connection draining/slow start). 근본 수정: 헬스체크를 readiness 기준으로 강화하고, 워밍업 단계에서 DNS/DB 연결을 사전에(pre-warm) 수립하도록 배포 절차를 바꾼다는 것까지 제시한다.
  7. RCA 문서화와 재발 방지 항목을 포함해 마무리한다
    타임라인, 근본 원인, 완화 조치, 재발 방지(readiness probe 도입, DNS 캐싱/커넥션 풀 사전 워밍업, 오토스케일링 트래픽 유입 속도 제한)를 문서화하고 팀에 공유한다는 것을 밝힌다.

6. 핵심 답변 예시

짧은 답변 (60초 버전)

"먼저 영향 범위를 확인하겠습니다. 도쿄 리전에서만 발생했다는 것은 글로벌 배포나 DB 문제보다는 리전 국소적 원인일 가능성을 높입니다. 그다음 타이밍을 보면 오토스케일링(13:30)과 에러율 급증(14:00) 사이에 30분의 시간차가 있는데, 이건 오토스케일링이 직접 원인이 아니라 신규 인스턴스의 워밍업 과정에서 뭔가 지연이 누적됐을 가능성을 시사합니다. 'upstream timeout'과 'connection reset'은 LB가 백엔드 응답을 기다리다 실패한 신호이고, DNS 지연 알림은 신규 인스턴스가 DB/캐시 연결을 새로 맺으면서 내부 DNS에 부하를 준 결과일 가능성이 큽니다. 즉 근본 원인은 헬스체크가 '프로세스 기동'만 확인하고 '실제 의존 리소스 연결 완료'는 확인하지 않은 상태에서 LB가 트래픽을 흘려보낸 것이라고 보고, 즉시로는 신규 인스턴스를 LB에서 빼거나 트래픽 유입을 느리게 하고, 근본적으로는 readiness 기준의 헬스체크와 사전 워밍업 절차를 도입하겠습니다."

깊이 있는 답변 (면접관이 "더 자세히"라고 물을 때)

"조금 더 구체적으로 조사 순서를 말씀드리면, 우선 APM/대시보드에서 에러율과 지연을 리전별, 엔드포인트별, 인스턴스별로 쪼개서 봅니다. 만약 신규로 추가된 6대의 인스턴스에서만 에러가 집중된다면 오토스케일링이 강한 용의자로 확정되고, 기존 인스턴스에도 고르게 에러가 퍼져 있다면 공유 의존성(예: 리전 공용 DB, 공유 캐시 클러스터)이 원인일 가능성으로 방향을 틉니다. 신규 인스턴스가 원인으로 확인되면, 다음으로 헬스체크 설정을 봅니다. 많은 팀이 헬스체크를 '프로세스가 떠서 HTTP 200을 반환하는가'로만 구현하는데, 이건 liveness(살아있는가) 확인일 뿐 readiness(트래픽을 받을 준비가 됐는가) 확인이 아닙니다. 신규 인스턴스가 부팅 직후 헬스체크는 통과했지만 DB 커넥션 풀, 캐시 클라이언트, 서비스 디스커버리 등록이 완료되지 않은 상태였다면, LB는 '건강하다'고 판단해 트래픽을 흘려보내고 실제로는 백엔드가 준비되지 않아 타임아웃이 발생합니다. DNS 지연은 이 워밍업 과정과 정확히 겹치는지 내부 DNS 리졸버의 QPS 그래프로 확인합니다. 만약 겹친다면, 6대의 인스턴스가 동시에 부팅하며 각각 여러 개의 DB/캐시/내부 서비스 이름을 짧은 시간에 조회했고, 이게 리졸버에 순간 부하를 줘 지연을 유발한 뒤, 그 지연이 다시 커넥션 수립을 늦춰 LB 타임아웃으로 이어지는 연쇄였다고 결론짓습니다. 이 경우 DNS는 원인이 아니라 같은 근본 원인(부실한 워밍업 설계)의 또 다른 증상입니다. 즉시 조치로는 신규 인스턴스를 connection draining으로 서서히 빼거나, 오토스케일링의 트래픽 유입을 slow start(초기에는 적은 비중의 트래픽만 라우팅 후 점진적으로 증가)로 설정해 급격한 부하를 막습니다. 근본 수정으로는 헬스체크를 readiness probe로 강화해 DB/캐시 연결까지 확인한 뒤에만 트래픽을 받도록 하고, 배포/스케일 아웃 시 커넥션 풀을 사전에(pre-warm) 채우는 절차를 표준화합니다. 마지막으로 이 전체 타임라인, 근본 원인, 조치, 재발 방지책을 postmortem으로 문서화해서 팀과 공유하고, 유사 패턴(동시 다발적 워밍업)이 다른 서비스에도 있는지 점검하는 것까지가 RCA의 마무리입니다."

7. Trade-off — 원인 후보 간 진단 우선순위

진단 접근장점단점 / 주의사항
변경 상관관계 우선(가장 최근 변경을 먼저 의심) 실무에서 장애 원인의 대다수가 최근 변경(배포, 스케일링, 설정 변경)이므로 적중률이 높고 빠름 우연히 시간이 겹쳤을 뿐인 무관한 변경을 원인으로 오판할 위험, 반드시 메커니즘까지 확인해야 함
영향 범위 우선(Blast Radius로 좁히기) 광범위 원인(글로벌 배포, 공유 DB)과 국소 원인(특정 리전/인스턴스)을 빠르게 구분해 조사 방향을 절반으로 줄임 범위 판단 자체가 모니터링 대시보드의 세분화 수준에 의존 — 리전/인스턴스별 지표가 없으면 이 접근이 불가능
계층별 순차 배제(L3→L4→L7→인프라) 체계적이고 놓치는 계층이 없음, 신입 온콜도 따라 하기 쉬운 절차 장애 중 시간이 오래 걸릴 수 있음 — 급한 장애에서는 가장 가능성 높은 계층부터 건너뛰어 시작하는 판단력이 필요
즉시 완화 우선(원인 규명보다 증상 완화 먼저) 사용자 영향을 빠르게 줄임 (예: 신규 인스턴스 롤백, 트래픽 우회) 근본 원인을 놓치고 재발할 위험, "완화됐으니 끝"이라는 착각으로 postmortem을 생략하면 같은 장애가 반복됨
readiness probe 도입(헬스체크 강화) 워밍업 미완료 인스턴스로 트래픽이 흐르는 근본 원인을 차단 readiness 판정 기준(어떤 의존성까지 확인할지)이 너무 엄격하면 스케일 아웃 속도가 느려져 트래픽 급증에 대한 대응력이 떨어질 수 있음

8. 흔한 오해

오해 1 "가장 눈에 띄는 알림(DNS 지연)이 원인이다"

모니터링 알림은 "무언가 이상하다"는 신호일 뿐 인과관계를 말해주지 않는다. 여러 알림이 동시에 뜨면, 그중 어느 것이 원인이고 어느 것이 같은 원인의 파생 증상인지 타이밍과 메커니즘으로 검증해야 한다. 가장 시끄러운 알림이 항상 근본 원인은 아니다.

오해 2 "서버 CPU/메모리가 정상이니 인프라 문제가 아니다"

CPU/메모리는 "리소스 고갈형" 장애만 보여준다. 커넥션 풀 고갈, 헬스체크와 실제 준비 상태의 불일치, DNS 지연처럼 리소스 사용량과 무관한 원인은 CPU/메모리 지표에 드러나지 않는다. "리소스는 정상"이라는 진술은 원인 후보를 줄여줄 뿐, 네트워크·인프라 문제 전체를 배제하지 않는다.

오해 3 "헬스체크를 통과했으면 트래픽을 받을 준비가 된 것이다"

많은 시스템의 헬스체크는 liveness(프로세스 생존)만 확인하고 readiness(의존 리소스 연결 완료)는 확인하지 않는다. 이 둘을 같은 것으로 착각하면, 이번 시나리오처럼 부팅은 됐지만 실제로는 요청을 처리할 준비가 안 된 인스턴스로 트래픽이 흘러가는 장애를 반복하게 된다.

오해 4 "장애가 완화되면 RCA는 끝난 것이다"

완화(mitigation)는 사용자 영향을 멈추는 것이고, RCA는 왜 그 일이 일어났는지와 재발을 어떻게 막을지까지 포함한다. 완화 후 근본 원인 분석과 postmortem을 생략하면, 같은 패턴(워밍업 부실)이 다른 서비스나 다음 스케일링 이벤트에서 그대로 반복된다.

오해 5 "최근 변경이 있었으니 그게 무조건 원인이다"

변경 상관관계는 강력한 단서이지만 증명이 아니다. 오토스케일링이 30분 전에 있었다고 바로 "오토스케일링이 원인"이라 단정하지 말고, 그 변경이 관측된 증상을 만들어낼 수 있는 구체적 메커니즘(워밍업 미완료 → DNS 부하 → 타임아웃)까지 재구성해야 진짜 RCA가 된다.

9. Senior-level Follow-up 질문

Follow-up 질문면접관의 의도답변 포인트
"신규 인스턴스만 문제인지 기존 인스턴스도 문제인지 어떻게 빠르게 구분하시겠습니까?" 대시보드/관측 도구를 실제로 다뤄봤는지, 데이터 기반 가설 검증 능력 확인 인스턴스 ID/가용영역 단위로 에러율과 지연을 쪼갠 대시보드(또는 로그 쿼리)를 즉시 확인한다는 절차를 구체적으로 설명하고, 신규 인스턴스에 에러가 집중된다면 오토스케일링을 강한 용의자로 확정한다는 점을 짚는다.
"이 장애를 막기 위해 배포 파이프라인에 어떤 게이트를 추가하시겠습니까?" 재발 방지를 시스템적으로(사람이 아니라 프로세스로) 설계할 수 있는지 확인 readiness probe를 CI/CD와 오토스케일링 그룹의 헬스체크 기준으로 강제하고, 신규 인스턴스가 일정 비율의 헬스체크를 안정적으로 통과할 때까지 트래픽을 점진적으로 늘리는 slow start를 표준 배포 정책으로 만든다는 점을 설명한다.
"만약 도쿄 리전이 아니라 전체 리전에서 동시에 발생했다면 조사 방향이 어떻게 달라집니까?" 영향 범위에 따라 가설 트리를 다르게 세울 수 있는 유연성 확인 전체 리전 동시 발생이라면 리전 국소 원인(오토스케일링, 로컬 DNS 리졸버)보다 글로벌 공유 컴포넌트(중앙 인증 서버, 글로벌 DB, 공통 배포된 코드/설정, 인증서 만료)를 우선 의심한다는 방향 전환을 설명한다.
"DNS 지연이 원인이 아니라 결과라는 것을 어떻게 데이터로 증명하시겠습니까?" 가설을 주장으로 끝내지 않고 반증 가능한 근거로 뒷받침하는 능력 확인 내부 DNS 리졸버의 QPS/지연 그래프를 오토스케일링 이벤트 타임스탬프와 겹쳐서, 신규 인스턴스 부팅 시각에 정확히 QPS가 튀는지 확인한다는 절차를 제시하고, 만약 오토스케일링 이전부터 이미 DNS 지연이 있었다면 별개 원인일 가능성을 열어둔다는 점도 짚는다.
"이 장애의 postmortem을 개발팀, 비즈니스 담당자에게 각각 어떻게 다르게 설명하시겠습니까?" 기술적 원인 규명을 넘어 이해관계자별 커뮤니케이션 역량 확인 개발팀에는 타임라인, 근본 메커니즘(readiness vs liveness), 코드/설정 수준의 재발 방지책을 상세히 공유하고, 비즈니스 담당자에게는 영향 범위(어느 리전, 몇 분, 매출 영향 추정치)와 재발 방지를 위한 일정, 향후 유사 위험을 얼마나 줄였는지를 비기술 용어로 요약한다는 점을 설명한다.

10. 자기 점검 Rubric

약한 답변 (Junior 수준)

  • 주어진 단서 중 하나(DNS 알림)만 보고 "DNS가 원인입니다"라고 바로 단정함
  • 영향 범위(도쿄 리전만)를 무시하고 전체 시스템을 뒤지듯 조사 계획을 나열함
  • 오토스케일링과 장애 사이의 30분 시간차를 무시하거나, 반대로 시간차만 보고 무관하다고 성급히 배제함
  • 즉시 완화와 근본 수정을 구분하지 못하고 "재시작하면 됩니다"로 답을 끝냄
  • RCA 결과를 문서화하거나 팀에 공유하는 과정을 언급하지 않음

강한 답변 (Senior 수준)

  • 영향 범위 → 타이밍 상관관계 → 최근 변경 우선 검증 → 계층별 메커니즘 확인 순으로 체계적으로 좁혀나감
  • DNS 지연을 "원인 후보"와 "파생 증상 후보" 둘 다로 열어두고 데이터(QPS 그래프, 타임스탬프)로 구분함
  • liveness와 readiness 헬스체크의 차이처럼 근본 메커니즘을 구체적인 인프라 개념으로 설명함
  • 즉시 완화(connection draining, slow start)와 근본 수정(readiness probe, pre-warm)을 명확히 구분해 제시함
  • postmortem 작성과 이해관계자별 커뮤니케이션까지 답변에 포함함
  • 단정적 주장 대신 "이 가설이 맞다면 어떤 데이터가 보일 것이다"라는 반증 가능한 검증 방식을 사용함

자기 점검 체크리스트

  • ☐ 영향 범위(리전/서비스/엔드포인트)를 먼저 확정하고 조사 방향을 좁혔는가?
  • ☐ 여러 단서의 발생 시각을 상관관계로 비교했는가?
  • ☐ 하나의 단서(DNS 알림)만으로 성급하게 원인을 단정하지 않았는가?
  • ☐ 원인과 파생 증상을 구분하려는 시도(데이터 기반 반증)를 보였는가?
  • ☐ liveness/readiness처럼 근본 메커니즘을 구체적인 개념으로 설명했는가?
  • ☐ 즉시 완화와 근본 수정을 구분해서 제시했는가?

Interview Day 1~10을 통해 네트워크 장애 분석 프레임워크, OSI/TCP-IP 계층 트러블슈팅, TCP/UDP 연결 관리, DNS, TLS 핸드셰이크, 로드밸런싱과 프록시, VPC 클라우드 네트워킹, 방화벽/NAT/VPN/Zero Trust, 네트워크 성능 최적화, 그리고 이 모든 것을 통합한 장애 RCA까지 다뤘다. 실제 면접에서는 이 열 가지 주제가 뒤섞인 형태로 나올 수 있으므로, 어떤 질문을 받든 "범위 → 계층 → 타이밍 → 근본 메커니즘 → 완화/수정 → 재발 방지"라는 동일한 사고 절차를 적용하는 것이 10년차 이상 개발자에게 기대되는 답변 방식이다.
