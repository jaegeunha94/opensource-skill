1. 핵심 면접 질문

Primary Question

"디자이너가 네이티브 <select> 대신 커스텀 드롭다운 UI를 요구합니다. 이 컴포넌트를 네이티브 select만큼 접근 가능하게 만들려면 ARIA를 어떻게 적용하시겠어요?"

Variant 1

"Accessibility Tree가 DOM Tree와 어떻게 다른가요? 브라우저는 어떤 요소를 Accessibility Tree에서 제외하나요?"

Variant 2

aria-hidden="true"를 실수로 잘못 사용해서 실제 서비스에서 문제가 생긴 경험이 있나요? 어떤 상황에서 aria-hidden이 위험한가요?

2. 왜 면접에서 묻는가

이 질문은 "ARIA 속성 이름을 외웠는가"가 아니라 "브라우저가 접근성 정보를 어떤 모델로 계산하고, 그 모델을 스크린 리더가 어떻게 소비하는지 시스템적으로 이해하는가"를 테스트합니다. 프론트엔드가 점점 더 많은 UI를 네이티브 HTML 요소 대신 div/span 조합으로 재구현하는 지금(커스텀 드롭다운, 커스텀 탭, 커스텀 모달, 커스텀 토글), 시니어 개발자는 "네이티브 요소가 공짜로 주던 접근성을 직접 재구현할 수 있는가"를 반드시 증명해야 합니다. 이 질문은 그 능력을 가장 직접적으로 드러내는 리트머스 시험지입니다.

면접관이 이 질문으로 평가하는 것:

  • Accessibility Tree의 존재와 구조를 아는가 — DOM Tree와 별개로 브라우저가 이 트리를 만들고, 스크린 리더는 오직 이 트리만 읽는다는 사실을 아는가.
  • role/name/state 세 축으로 컴포넌트를 분해할 수 있는가 — "이 요소는 무엇인가(role), 뭐라고 불리는가(name), 지금 어떤 상태인가(state)"로 나눠 생각하는가.
  • First Rule of ARIA를 실무에서 지키는가 — 네이티브 요소로 대체 가능하면 ARIA보다 네이티브를 우선하는가.
  • 키보드 상호작용까지 설계에 포함하는가 — ARIA role만 붙이고 키보드 동작(화살표 키, Esc, Home/End)을 빠뜨리는 "반쪽짜리 접근성"을 만들지 않는가.

10년차 이하에서 자주 나오는 답변 패턴: "role="listbox"랑 aria-expanded를 붙이면 됩니다." 이 답변은 속성 이름은 맞지만 왜 그 속성이 필요한지, 그것만으로 충분한지에 대한 설명이 빠져 있습니다. 시니어는 "ARIA 속성은 브라우저에게 '이 요소를 접근성 트리에서 이렇게 표현해달라'는 선언일 뿐이고, 실제 키보드 동작·포커스 관리는 여전히 JavaScript가 구현해야 한다"는 것, 즉 ARIA는 의미를 선언할 뿐 행동을 만들어주지 않는다는 것까지 설명합니다.

3. 전제 개념 — 쉬운 한국어로

Accessibility Tree란 무엇인가 — DOM과 무엇이 다른가

브라우저는 HTML을 파싱해서 DOM Tree를 만들지만, 스크린 리더는 이 DOM Tree를 직접 읽지 않습니다. 브라우저는 DOM Tree와 CSSOM을 조합해 Accessibility Tree(접근성 트리)라는 별도의 트리를 하나 더 계산하고, 스크린 리더나 스위치 컨트롤 같은 보조 기술(Assistive Technology)은 오직 이 트리만 읽습니다. Accessibility Tree는 DOM Tree의 "부분집합이자 변형"입니다. 예를 들어 display: none이거나 aria-hidden="true"인 요소, 순수하게 스타일을 위한 div wrapper 등은 접근성 트리에서 제외되거나 압축됩니다. 반대로 DOM에는 존재하지 않는 정보(예: 버튼의 계산된 "접근 가능한 이름")가 접근성 트리 노드에는 채워집니다. 즉 "화면에 보이는 것"과 "DOM에 있는 것"과 "스크린 리더가 읽는 것"은 세 가지 서로 다른 표현이고, 시니어 프론트엔드는 이 세 표현이 갈라지는 지점을 정확히 짚어낼 수 있어야 합니다.

ARIA의 세 축 — Role, Name, State

모든 접근성 트리의 노드는 결국 세 가지 정보로 요약됩니다. 이 세 가지를 "role/name/state 모델"이라고 부릅니다.

  • Role(역할) — "이 요소는 무엇인가"에 대한 답입니다. 버튼인지, 체크박스인지, 탭인지, 다이얼로그인지. 네이티브 <button>은 자동으로 role="button"을 가지며, 커스텀 div에 버튼처럼 동작하게 하려면 role="button"을 명시적으로 선언해야 합니다.
  • Name(접근 가능한 이름) — "이 요소를 뭐라고 부르는가"에 대한 답입니다. 브라우저는 정해진 우선순위 알고리즘(Accessible Name Computation)에 따라 aria-labelledby → aria-label → 연결된 label → 텍스트 콘텐츠 → title 순서로 이름을 계산합니다. 이 계산 결과가 스크린 리더가 실제로 읽는 텍스트입니다.
  • State/Property(상태와 속성) — "지금 어떤 상태인가"에 대한 답입니다. 체크박스가 체크되어 있는지(aria-checked), 드롭다운이 펼쳐져 있는지(aria-expanded), 탭이 선택되어 있는지(aria-selected) 등입니다.

커스텀 컴포넌트를 리뷰할 때 시니어는 항상 이 세 축으로 분해해서 점검합니다: "이 요소의 role이 스크린 리더에게 올바르게 전달되는가? name이 존재하는가? 상태 변화가 실시간으로 반영되는가?"

First Rule of ARIA — "쓰지 않을 수 있으면 쓰지 마라"

WAI-ARIA 명세 자체가 명시하는 첫 번째 원칙은 "네이티브 HTML 요소나 속성으로 필요한 semantics와 동작을 이미 표현할 수 있다면, ARIA role/state/property를 추가하지 말고 네이티브 요소를 사용하라"입니다. 이유는 명확합니다. 네이티브 <button>은 role, 키보드 동작(Enter/Space로 활성화), 포커스 가능 여부를 브라우저가 자동으로, 모든 브라우저·스크린 리더 조합에서 일관되게 제공합니다. 반면 <div role="button">은 role만 선언될 뿐, 키보드로 Tab 이동이 가능하게 하려면 tabindex="0"을 직접 추가해야 하고, Enter/Space 키 입력에 반응하게 하려면 keydown 이벤트 핸들러를 직접 작성해야 합니다. 하나라도 빠뜨리면 마우스 사용자에게는 완벽해 보이는 버튼이 키보드 사용자에게는 아예 작동하지 않는 컴포넌트가 됩니다.

aria-hidden이 위험한 이유 — "숨김"이 전파되는 방식

aria-hidden="true"는 해당 요소와 그 하위 트리 전체를 접근성 트리에서 완전히 제거합니다. 이때 주의할 점은 시각적으로는 여전히 보이는 요소에 이 속성을 잘못 적용하면, 마우스 사용자는 클릭할 수 있는데 스크린 리더 사용자는 그 요소가 아예 "존재하지 않는 것"으로 인식한다는 점입니다. 실무에서 흔한 사고는 모달을 열 때 배경(backdrop) 콘텐츠 전체에 aria-hidden="true"를 걸다가, 실수로 모달 자기 자신이나 포커스가 이동해야 할 요소까지 그 하위 트리에 포함시켜 버리는 경우입니다. 또한 aria-hidden="true"가 걸린 요소 내부에 포커스 가능한 요소(버튼, 링크)가 남아 있으면, 키보드로는 여전히 Tab이 이동해서 포커스가 가지만 스크린 리더는 그 요소를 "없는 것"처럼 다뤄서 포커스 위치와 스크린 리더가 읽는 내용이 완전히 어긋나는 매우 혼란스러운 버그가 발생합니다.

ARIA는 "의미"를 선언할 뿐, "행동"을 만들어주지 않는다

가장 흔히 오해하는 지점입니다. role="tablist", role="tab", aria-selected="true"를 완벽하게 붙여도, 스크린 리더는 "이것은 탭 목록이고, 이 탭이 선택되어 있다"고 안내할 뿐입니다. 실제로 화살표 키로 탭 사이를 이동하고, 선택된 탭의 내용을 보여주는 동작은 여전히 JavaScript가 구현해야 합니다. ARIA Authoring Practices Guide(APG)가 각 컴포넌트 패턴마다 "필요한 키보드 상호작용"까지 함께 명세하는 이유가 바로 이것입니다 — role/state 선언과 키보드 동작 구현은 항상 쌍으로 다뤄야 하는 두 가지 별개의 작업입니다.

4. 실무 연결

디자인 시스템에 커스텀 인터랙션 컴포넌트(드롭다운, 탭, 모달, 콤보박스, 토스트)가 하나씩 늘어날 때마다, 그 컴포넌트가 "접근성 부채"의 근원이 될지 아닐지가 이 시점에 결정됩니다.

커스텀 드롭다운 컴포넌트 — 실제로 무엇이 빠지기 쉬운가

디자이너가 브라우저 기본 select의 화살표 아이콘, 옵션 스타일을 커스터마이징할 수 없다는 이유로 div 기반 커스텀 드롭다운을 요구하는 것은 매우 흔한 요청입니다. 이때 실무에서 반복적으로 빠뜨리는 항목은 다음과 같습니다: 트리거 버튼에 aria-haspopup="listbox"와 aria-expanded 상태 동기화, 옵션 목록에 role="listbox"와 각 옵션에 role="option", 현재 선택된 옵션에 aria-selected="true", 그리고 화살표 키로 옵션 간 이동, Home/End로 처음/끝 이동, Esc로 닫기, 타이핑으로 옵션 검색(typeahead)까지의 키보드 상호작용 전체 구현입니다. ARIA APG의 Listbox 패턴 문서를 그대로 구현 명세로 삼는 것이 실무에서 가장 안전한 접근입니다.

모달(Dialog)에서 포커스 트랩과 aria-hidden의 조합

모달을 열 때는 세 가지가 동시에 맞아야 접근성이 완성됩니다: 모달에 role="dialog"와 aria-modal="true", 모달 바깥 콘텐츠에 inert 속성(또는 구형 브라우저 대응 시 aria-hidden="true" + tabindex="-1" 조합)으로 포커스와 스크린 리더 탐색을 동시에 차단, 그리고 모달이 열릴 때 포커스를 모달 내부로 이동시키고 Tab 키가 모달 밖으로 나가지 않도록 순환시키는 포커스 트랩입니다. 실무에서는 inert 속성이 최신 브라우저에서 aria-hidden + 포커스 차단을 한 번에 처리해주기 때문에, 직접 재귀적으로 형제 요소들에 aria-hidden을 다는 구식 구현보다 훨씬 버그가 적습니다.

디자인 시스템 팀에서 ARIA를 컴포넌트 레벨로 캡슐화하는 이유

매 페이지마다 개발자가 role과 aria-* 속성을 손으로 붙이게 하면 반드시 누락이 생깁니다. 그래서 성숙한 디자인 시스템은 Tabs, Dropdown, Dialog 같은 컴포넌트 내부에 필요한 role/state 계산과 키보드 이벤트 핸들러를 모두 캡슐화하고, 사용하는 개발자는 옵션 배열이나 children만 넘기면 되도록 만듭니다. 이렇게 하면 "이 프로젝트의 모든 드롭다운이 접근성 있는가"라는 질문이 "컴포넌트 하나가 접근성 있는가"라는 질문으로 축소되어 관리 가능해집니다.

5. 판단 프레임워크 — 커스텀 컴포넌트에 ARIA 적용하기

새로운 인터랙션 컴포넌트를 만들 때 아래 순서로 점검하면 됩니다.

  • 1
    네이티브 요소로 대체 가능한지부터 확인

    First Rule of ARIA — button, select, details/summary 등 네이티브 요소로 동일한 동작을 구현할 수 있는가? CSS로 원하는 시각적 스타일을 낼 수 없는지 먼저 검토했는가?

  • 2
    ARIA APG에서 대응하는 디자인 패턴 확인

    이 컴포넌트와 동일한 패턴(Listbox, Combobox, Tabs, Dialog, Menu 등)이 ARIA Authoring Practices Guide에 이미 정의되어 있는가? role 구조와 필요한 키보드 상호작용을 그대로 따르는가?

  • 3
    Role 선언

    컨테이너와 각 하위 요소에 올바른 role이 부여되어 있는가(예: listbox의 자식은 반드시 option)?

  • 4
    Name 계산 확인

    각 인터랙티브 요소가 접근 가능한 이름을 가지는가? 아이콘만 있는 버튼에 aria-label이 빠지지 않았는가?

  • 5
    State 동기화

    aria-expanded, aria-selected, aria-checked 같은 상태 속성이 실제 UI 상태 변화와 매 순간 동기화되는가?

  • 6
    키보드 상호작용 구현

    마우스 없이 Tab, 화살표 키, Enter/Space, Esc, Home/End만으로 컴포넌트의 모든 기능을 사용할 수 있는가?

  • 7
    포커스 관리

    컴포넌트가 열리거나 닫힐 때 포커스가 논리적인 위치로 이동하는가(모달 열림 → 내부로, 닫힘 → 트리거 버튼으로 복귀)?

  • 8
    실제 스크린 리더로 검증

    브라우저 개발자 도구의 접근성 트리 검사만이 아니라, 실제 스크린 리더(NVDA/VoiceOver 등)로 키보드만 사용해 완주해봤는가?

6. 항목별 판단 체크리스트

패턴 올바른 방법 흔한 오용 오용 시 실제 영향
클릭 가능 요소 <button> 사용 <div role="button">만 붙이고 끝 키보드 포커스/Enter·Space 활성화가 동작하지 않음
아이콘 전용 버튼 aria-label="닫기" 명시 아이콘 폰트/SVG만 넣고 텍스트 없음 접근 가능한 이름이 계산되지 않아 "버튼"으로만 읽힘
배경 콘텐츠 숨김(모달) inert 또는 aria-hidden+포커스 차단 동시 적용 aria-hidden만 걸고 포커스 이동은 그대로 둠 Tab 포커스는 이동하는데 스크린 리더는 "없는 요소"로 인식
상태 변화 aria-expanded를 실제 열림/닫힘과 매번 동기화 초기값만 설정하고 토글 시 갱신 누락 스크린 리더가 실제 상태와 다른 정보를 안내
커스텀 탭 role="tablist"/tab/tabpanel + 화살표 키 이동 role만 붙이고 Tab 키로만 각 탭 진입 가능 ARIA APG 표준 키보드 패턴과 어긋나 사용자 혼란
장식용 아이콘 aria-hidden="true"로 스크린 리더에서 제외 모든 svg에 무조건 role/label 부여 의미 없는 장식 요소까지 스크린 리더가 읽어 소음 발생

7. 답변 사고 순서

  1. 먼저 네이티브 대체 가능성을 짚습니다. First Rule of ARIA를 언급하며 "정말 커스텀이 필요한 이유"를 확인하는 태도를 보여줍니다.
  2. Accessibility Tree 개념을 도입합니다. 스크린 리더는 DOM이 아니라 이 트리를 읽는다는 전제를 세웁니다.
  3. role/name/state 세 축으로 컴포넌트를 분해합니다. 트리거 버튼과 옵션 목록 각각에 필요한 role, name, state를 구체적으로 나열합니다.
  4. ARIA는 의미 선언일 뿐이라는 점을 짚고 키보드 상호작용을 별도로 설명합니다. 화살표 키, Esc, Home/End 등 구체적인 키 목록을 제시합니다.
  5. ARIA APG를 근거로 언급합니다. 표준 패턴 문서를 실무에서 어떻게 활용하는지로 마무리합니다.

8. 핵심 답변 예시

면접관: "디자이너가 네이티브 select 대신 커스텀 드롭다운 UI를 요구합니다. 이 컴포넌트를 네이티브 select만큼 접근 가능하게 만들려면 어떻게 하시겠어요?"

STRONG ANSWER

먼저 정말 커스텀 드롭다운이 필요한지부터 확인합니다. 옵션에 아이콘이나 복잡한 레이아웃이 들어가는 등 네이티브 select로 표현할 수 없는 시각적 요구가 있을 때만 커스텀으로 진행하고, 단순히 화살표 아이콘 스타일 때문이라면 appearance: none과 배경 이미지로 네이티브 select를 커스터마이징하는 방법을 먼저 제안합니다.


커스텀이 꼭 필요하다면 ARIA Authoring Practices Guide의 Listbox 패턴을 그대로 구현 명세로 삼습니다. role/name/state 세 축으로 나누면, 트리거 버튼에는 aria-haspopup="listbox"와 현재 열림 여부를 나타내는 aria-expanded를 두고, 선택된 값 텍스트가 버튼의 접근 가능한 이름에 포함되게 합니다. 옵션 목록 컨테이너에는 role="listbox"를, 각 옵션에는 role="option"과 현재 선택 여부를 나타내는 aria-selected를 둡니다.


여기까지는 브라우저에게 "이것이 무엇인지" 선언하는 것일 뿐이고, 실제 동작은 별도로 구현해야 합니다. 드롭다운이 열려 있을 때 위/아래 화살표로 옵션 간 포커스가 이동하고, Home/End로 첫/마지막 옵션으로 이동하고, Enter나 Space로 옵션을 선택하며, Esc로 닫히면서 포커스가 트리거 버튼으로 돌아오도록 키보드 이벤트를 직접 구현합니다.


마지막으로 브라우저 개발자 도구의 Accessibility 패널로 role/name/state가 의도대로 계산되는지 확인하고, 실제로 마우스를 떼고 키보드와 스크린 리더만으로 옵션을 선택하는 것까지 완주 테스트합니다.

이 답변이 좋은 이유:

  • 커스텀 구현에 바로 들어가지 않고 First Rule of ARIA로 필요성부터 검증함
  • role/name/state 모델로 컴포넌트를 체계적으로 분해함
  • ARIA 선언과 키보드 동작 구현을 명확히 구분해서 설명함
  • ARIA APG라는 표준 근거를 제시하고, 실제 검증 절차(개발자 도구 + 스크린 리더)까지 언급함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"div에 role="listbox"랑 role="option"을 붙이고 aria-expanded로 열림 상태를 표시하면 됩니다."

이 답변이 약한 이유

  • role 이름은 정확하지만 키보드 상호작용(화살표 키 이동, Esc, Home/End)이 전혀 언급되지 않았습니다.
  • ARIA 속성이 "의미 선언"일 뿐이고 "동작 구현"은 별개라는 인식이 드러나지 않습니다.
  • 네이티브 select로 대체 가능한지 먼저 검토하는 태도가 없습니다.
WEAK ANSWER — 또 다른 패턴

"접근성을 높이기 위해 모든 요소에 aria-label과 role을 추가로 붙입니다."

이 답변이 약한 이유

  • "많이 붙일수록 좋다"는 인식은 First Rule of ARIA와 정반대입니다. 불필요한 ARIA는 오히려 스크린 리더의 안내를 장황하게 만들거나 네이티브 시맨틱과 충돌할 수 있습니다.
  • 구체적으로 어떤 요소에 어떤 role/state가 필요한지에 대한 판단 기준이 없습니다.
STRONG ANSWER 요소 체크리스트
  • 커스텀 구현 전에 네이티브 요소 대체 가능성을 먼저 검토한다(First Rule of ARIA)
  • role/name/state 세 축으로 컴포넌트를 분해해서 설명한다
  • ARIA 선언과 키보드 상호작용 구현을 별개의 두 작업으로 구분한다
  • ARIA APG 같은 표준 패턴 문서를 근거로 제시한다
  • 개발자 도구의 접근성 트리 검사와 실제 스크린 리더 테스트를 모두 언급한다

10. Senior-level Follow-up Q&A

Q: Accessible Name Computation의 우선순위는 어떻게 되나요? aria-label과 aria-labelledby가 동시에 있으면 어느 것이 이기나요?

모범 답변: 브라우저는 정해진 알고리즘에 따라 aria-labelledby를 가장 먼저 확인하고, 없으면 aria-label, 그다음 네이티브 레이블링(연결된 label 등), 그다음 텍스트 콘텐츠, 마지막으로 title 속성 순으로 접근 가능한 이름을 계산합니다. 따라서 aria-labelledby와 aria-label이 동시에 있으면 aria-labelledby가 우선합니다. 실무에서는 두 속성을 동시에 쓸 이유가 거의 없고, 오히려 어느 것이 실제로 적용되는지 헷갈리게 만드는 원인이 되므로 하나만 명확히 선택해서 쓰는 것이 유지보수에 유리합니다.

Q: aria-hidden="true"가 걸린 요소 안에 포커스 가능한 버튼이 있으면 어떻게 되나요?

모범 답변: aria-hidden은 접근성 트리에서만 요소를 제거할 뿐, DOM의 포커스 가능 여부(tabindex, 네이티브 포커스 가능 요소)에는 영향을 주지 않습니다. 그래서 Tab 키로는 여전히 그 버튼에 포커스가 이동하지만, 스크린 리더는 그 요소가 "존재하지 않는 것"처럼 다뤄서 아무것도 읽어주지 않는 모순적인 상태가 됩니다. 이 문제를 막으려면 aria-hidden="true"를 적용할 때 그 하위의 포커스 가능한 요소에도 함께 tabindex="-1"을 걸어 포커스 자체를 차단하거나, 최신 브라우저라면 inert 속성을 사용해 접근성 트리 제외와 포커스 차단을 한 번에 처리하는 것이 안전합니다.

Q: role 속성을 잘못 선언하면(예: 실제로는 버튼처럼 동작하는데 role="link"을 붙이면) 어떤 문제가 생기나요?

모범 답변: ARIA는 브라우저의 실제 동작을 바꾸지 않고 "이렇게 안내하라"는 선언만 바꿉니다. role="link"이 붙으면 스크린 리더 사용자는 "이것은 페이지를 이동시키는 링크"라고 기대하게 되는데, 실제로는 클릭 시 어떤 동작(예: 모달 열기)이 실행되면 기대와 실제 동작이 어긋나 사용자가 혼란을 겪습니다. 또한 스크린 리더에는 "이 페이지의 링크만 모아 보기", "버튼만 모아 보기" 같은 요소 유형별 탐색 기능이 있는데, role을 잘못 선언하면 사용자가 컴포넌트를 아예 찾지 못하는 경우도 생깁니다. role은 반드시 실제 동작 의미론과 일치해야 합니다.

Q: 커스텀 토스트(toast) 알림처럼 사용자 상호작용 없이 나타나는 UI는 어떤 role을 써야 하나요?

모범 답변: 단순 정보성 토스트(예: "저장되었습니다")는 role="status"(내부적으로 aria-live="polite"와 동일한 효과)를 사용해, 스크린 리더가 현재 읽던 것을 방해하지 않고 자연스러운 시점에 안내하도록 합니다. 반면 사용자의 행동을 즉시 중단시켜야 하는 긴급한 알림(예: "세션이 곧 만료됩니다")은 Day 3에서 다룬 role="alert"를 사용합니다. 토스트가 몇 초 후 자동으로 사라지는 UI라면, 스크린 리더 사용자는 화면을 보는 사용자보다 내용을 인지하는 데 시간이 더 걸릴 수 있으므로 자동 사라짐 시간을 충분히 길게 설정하거나, 사용자가 직접 닫기 전까지 유지되는 옵션을 함께 고려해야 합니다.

Q: 접근성 트리는 브라우저 개발자 도구에서 어떻게 직접 확인하나요?

모범 답변: Chrome DevTools의 Elements 패널에서 요소를 선택하면 우측에 "Accessibility" 탭이 있어, 해당 요소의 계산된 role, name, 그리고 관련 속성들을 실시간으로 확인할 수 있습니다. 또한 "Full accessibility tree" 뷰를 열면 페이지 전체의 접근성 트리 구조를 DOM 트리와 나란히 비교할 수 있습니다. 실무에서는 이 도구로 1차 검증을 하고, 이름/역할/상태가 의도대로 계산되는지 빠르게 확인한 뒤, 최종적으로는 실제 스크린 리더(NVDA, VoiceOver, JAWS 등)로 키보드만 사용해 시나리오를 완주하는 2차 검증을 거치는 것이 표준적인 접근성 QA 절차입니다.

11. 흔한 오해

오해 1

"ARIA 속성을 붙이면 그 요소가 자동으로 키보드로 조작 가능해진다"

ARIA는 접근성 트리에 "이 요소는 이런 의미와 상태를 가진다"고 선언할 뿐입니다. 실제 키보드 포커스 가능 여부(tabindex)와 키 입력에 대한 반응(이벤트 핸들러)은 반드시 별도로 구현해야 합니다.

오해 2

"ARIA 속성은 많이 붙일수록 접근성이 좋아진다"

WAI-ARIA의 First Rule은 정반대를 말합니다. 네이티브 요소로 대체 가능하면 ARIA를 쓰지 않는 것이 원칙이며, 불필요하거나 중복된 ARIA는 스크린 리더의 안내를 장황하게 만들거나 네이티브 시맨틱과 충돌해 오히려 사용성을 해칩니다.

오해 3

"Accessibility Tree는 DOM Tree와 사실상 같은 것이다"

Accessibility Tree는 DOM Tree를 기반으로 계산되지만, display: none, aria-hidden, 순수 레이아웃용 wrapper 등은 제외되거나 압축됩니다. 반대로 접근 가능한 이름처럼 DOM에는 명시적으로 존재하지 않는 계산된 값도 이 트리에는 채워집니다. 스크린 리더는 DOM이 아니라 이 트리만 읽습니다.

오해 4

"aria-hidden으로 숨긴 요소는 클릭도 안 될 것이다"

aria-hidden은 시각적 렌더링이나 마우스 클릭 가능 여부, 키보드 포커스 가능 여부에 전혀 영향을 주지 않습니다. 오직 접근성 트리에서만 제외됩니다. 그래서 포커스 가능한 요소를 aria-hidden 하위에 남겨두면 포커스는 이동하는데 스크린 리더는 아무것도 읽지 않는 모순이 발생합니다.

오해 5

"role만 정확하면 ARIA APG 문서의 키보드 패턴까지 따를 필요는 없다"

스크린 리더 사용자는 특정 role(예: tablist, listbox, menu)을 만나면 그 role에 표준적으로 기대되는 키보드 동작(화살표 키 이동 등)을 그대로 시도합니다. role만 선언하고 그 표준 키보드 동작을 구현하지 않으면, 사용자는 role이 잘못 선언된 것보다 더 혼란스러운 "작동하지 않는 컴포넌트"를 경험하게 됩니다.
