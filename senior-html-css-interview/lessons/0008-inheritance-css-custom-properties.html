1. 핵심 면접 질문

Primary Question

color는 자식 요소에 상속되는데 border는 왜 상속되지 않을까요? 이 차이를 design token 설계에 어떻게 활용하시겠어요?

Variant 1

CSS 커스텀 프로퍼티(--color-primary 같은 변수)도 상속되나요? SASS 변수와 동작 방식이 어떻게 다른가요?

Variant 2

inherit, initial, unset, revert 키워드의 차이는 무엇이고, 각각 실무에서 어떤 상황에 쓰시겠어요?

2. 왜 면접에서 묻는가

이 질문은 "상속되는 속성 목록을 외웠는가"가 아니라 "값이 정해지는 마지막 단계(Inheritance)를 이해하고, 이를 design token/테마 시스템의 전파 메커니즘으로 의도적으로 설계할 수 있는가"를 테스트합니다. Day 6~7에서 다룬 Cascade와 Specificity는 "어떤 선언이 이기는가"를 결정하는 규칙이었지만, Inheritance는 애초에 그 요소를 겨냥한 선언이 하나도 없을 때 값을 어디서 가져올지를 결정하는 별개의 메커니즘입니다. 이 구분이 흐릿하면 "왜 부모에 지정한 border가 자식에 안 먹히지?" 같은 잘못된 디버깅 방향으로 시간을 낭비하게 됩니다.

면접관이 이 질문으로 평가하는 것:

  • 상속 여부가 무작위가 아니라 속성의 "성격"에 따른 설계 원칙이라는 것을 아는가 — 텍스트/타이포그래피 관련 속성은 문서 전체 일관성을 위해 상속되고, 박스/레이아웃 관련 속성은 요소별 독립성을 위해 상속되지 않는다는 설계 의도를 설명할 수 있는가.
  • CSS 커스텀 프로퍼티가 SASS/LESS 변수와 근본적으로 다른 메커니즘이라는 것을 아는가 — 전처리기 변수는 빌드 타임에 텍스트로 치환되어 사라지지만, 커스텀 프로퍼티는 런타임에 Cascade를 거쳐 값이 계산되고 기본적으로 상속된다는 차이.
  • 이 상속 메커니즘을 실제로 design token, 다크모드, 컴포넌트 스코프 오버라이드에 응용해본 경험이 있는가.
  • inherit/initial/unset/revert를 정확히 구분해 의도적으로 캐스케이드를 제어할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "그냥 상속되는 속성이랑 안 되는 속성이 정해져 있어요, color는 되고 border는 안 돼요." 이 답변은 사실은 맞지만 "왜" 그렇게 설계되었는지에 대한 원칙이 없어서, CSS 커스텀 프로퍼티가 나왔을 때 "그럼 변수는 상속되나요?"라는 후속 질문에 막힙니다. 시니어는 "상속은 문서 전체에 일관되게 적용되어야 하는 값과, 각 요소가 독립적으로 결정해야 하는 값을 구분하는 설계 원칙"이라는 수준까지 설명하고, 이를 커스텀 프로퍼티 기반 토큰 시스템 설계로 확장합니다.

3. 전제 개념 — 쉬운 한국어로

Inheritance란 무엇인가 — "선언이 하나도 없을 때 부모에게 물어보는 마지막 단계"

Day 6에서 Cascade는 Origin/Importance → Cascade Layer → Specificity → Source Order 순으로 "경쟁하는 선언들 중 승자"를 가린다고 배웠습니다. 그런데 만약 어떤 요소의 어떤 속성에 대해 애초에 매칭되는 선언이 하나도 없다면? 이때 브라우저는 두 가지 선택지 중 하나를 취합니다. 그 속성이 "상속 속성"이면 부모 요소가 최종적으로 가진 값을 그대로 물려받고, "비상속 속성"이면 그 속성의 초기값(initial value)을 사용합니다. 즉 Inheritance는 Cascade가 끝난 뒤, 값이 여전히 비어있을 때 작동하는 별도의 후속 단계입니다.

왜 어떤 속성은 상속되고 어떤 속성은 상속되지 않는가 — 설계 원칙

이건 임의로 정해진 목록이 아니라 속성의 성격에 따른 의도적 설계입니다.

상속되는 속성 (주로 텍스트/타이포그래피 관련)

상속됨 — color, font-family, font-size, font-weight, line-height, letter-spacing, text-align, visibility, cursor, list-style

문서 전체에 걸쳐 일관되게 적용되는 것이 자연스러운 값입니다. 본문 텍스트 색이나 글꼴을 모든 <p>, <span>, <li>마다 반복해서 지정해야 한다면 CSS는 실무적으로 쓸 수 없을 것입니다.

상속되지 않는 속성 (주로 박스 모델/레이아웃 관련)

상속 안 됨 — border, margin, padding, width, height, background, display, position, flex, grid-*

각 요소가 독립적으로 자신만의 시각적 경계와 배치를 가져야 자연스러운 값입니다. 부모 <div>에 border: 1px solid를 줬는데 그 안의 모든 자식 <p>, <span>까지 전부 테두리가 생긴다면 레이아웃이 통제 불가능해집니다.

기준을 한 문장으로 요약하면: "이 값이 문서 전체의 톤을 결정하는가(상속) vs 이 요소만의 박스를 결정하는가(비상속)"입니다. 확실하지 않을 때는 MDN에서 속성별 Inherited 항목을 확인하는 것이 정확하지만, 이 원칙을 알면 처음 보는 속성도 어느 쪽에 가까운지 추론할 수 있습니다.

4가지 전역 키워드 — inherit, initial, unset, revert

기본 상속 규칙을 명시적으로 뒤집고 싶을 때 쓰는 키워드입니다.

키워드동작실무 사용 예
inherit 본래 비상속 속성이라도 강제로 부모의 계산된 값을 그대로 가져옴 border: inherit으로 중첩된 카드 컴포넌트가 부모 테두리 스타일을 그대로 물려받게 함
initial CSS 명세가 정의한 그 속성의 기본 초기값으로 리셋 (상속 여부와 무관) color: initial로 어딘가에서 상속받은 색을 명세 기본값(대개 검정)으로 되돌림
unset 상속 속성이면 inherit처럼, 비상속 속성이면 initial처럼 동작 — "이 속성의 기본 동작을 따르라" 어떤 속성이 상속되는지 매번 외우지 않고 "리셋"하고 싶을 때
revert author 스타일을 건너뛰고 브라우저 기본 스타일(user-agent stylesheet)로 되돌림 all: revert로 서드파티 위젯 안에서 프로젝트 CSS 리셋의 영향을 걷어내고 싶을 때

CSS 커스텀 프로퍼티(--foo)는 "특별한 상속 속성"이다

CSS 커스텀 프로퍼티는 --color-primary: #6c8eff;처럼 --로 시작하는 사용자 정의 속성입니다. 중요한 점은 커스텀 프로퍼티는 기본적으로 상속되는 속성으로 취급된다는 것입니다. :root에 정의한 --color-primary는 문서 전체 모든 요소에 상속되어 전파되고, 어떤 하위 요소에서 var(--color-primary)로 참조하든 가장 가까운 조상에서 재정의된 값을 그대로 받습니다. 이건 우연이 아니라, 커스텀 프로퍼티가 애초에 "값을 트리 전체에 전파하는 토큰"으로 쓰이도록 설계되었기 때문입니다.

CSS 커스텀 프로퍼티 vs SASS/LESS 변수 — 근본적으로 다른 메커니즘

겉보기에 둘 다 "변수"처럼 보이지만 동작 시점이 완전히 다릅니다.

SASS 변수 ($color-primary): 빌드 타임(컴파일 시점)에 텍스트로 그대로 치환되고 사라집니다. 컴파일된 CSS 파일에는 변수라는 개념 자체가 존재하지 않고 #6c8eff 같은 값만 남습니다. 브라우저가 로드하는 시점에는 이미 정적인 값입니다.

CSS 커스텀 프로퍼티 (--color-primary): 런타임에 Cascade를 거쳐 실제로 계산됩니다. 브라우저 DOM에 살아있는 값이라서, 미디어 쿼리·클래스 토글·JavaScript(element.style.setProperty())로 페이지가 로드된 이후에도 값을 바꿀 수 있고, 바뀐 순간 이를 참조하는 모든 곳에 즉시 반영됩니다.

커스텀 프로퍼티가 정의되지 않았을 때 — Guaranteed-Invalid Value

var(--없는-변수)처럼 정의되지 않은 커스텀 프로퍼티를 참조하면, 일반 CSS 속성처럼 "무시하고 이전 선언 유지"가 아니라 "guaranteed-invalid value"라는 특수한 값이 됩니다. 이 값을 최종적으로 사용하는 속성은 상속 속성이면 부모 값으로, 비상속 속성이면 unset과 동일하게 동작합니다. 그래서 실무에서는 var(--gap, 16px)처럼 두 번째 인자로 fallback 값을 지정해 정의 누락 시의 동작을 명시적으로 통제합니다.

4. 실무 연결

"다크모드를 토글했더니 배경색만 바뀌고 카드 컴포넌트 100개는 각각 색을 다시 지정해야 했다"는 문제의 근본 원인은 색상 값이 각 컴포넌트에 하드코딩되어 있었기 때문입니다. 커스텀 프로퍼티의 상속을 이용하면 이 문제가 구조적으로 사라집니다.

Design Token 시스템 — :root에서 상속으로 전체 트리에 전파

:root {
  --color-primary: #6c8eff;
  --color-surface: #1a1d2e;
  --space-sm: 8px;
  --space-md: 16px;
}

.button {
  background: var(--color-primary);
  padding: var(--space-sm) var(--space-md);
}

:root는 문서 트리의 최상위 조상이므로, 여기서 선언한 토큰은 Inheritance 메커니즘을 통해 별도 설정 없이 모든 자손 요소에 전파됩니다. 이것이 SASS 변수 방식(빌드 타임 텍스트 치환)과 달리, 런타임에 값 하나를 바꾸면 이를 참조하는 모든 곳이 실시간으로 갱신되는 이유입니다.

다크모드 테마 전환 — JavaScript로 스타일 재계산 없이

:root {
  --color-bg: #ffffff;
  --color-text: #111111;
}
[data-theme="dark"] {
  --color-bg: #0f1117;
  --color-text: #e2e8f0;
}
body {
  background: var(--color-bg);
  color: var(--color-text);
}

document.documentElement.dataset.theme = 'dark' 한 줄로 [data-theme="dark"] 스코프 안의 커스텀 프로퍼티 값이 재정의되고, 이를 참조하는 모든 하위 요소가 자동으로 새 값을 상속받아 리페인트됩니다. 각 컴포넌트가 개별적으로 다크모드 조건문을 가질 필요가 없습니다 — Inheritance가 전파를 대신 해줍니다.

컴포넌트 스코프 오버라이드 — 지역적으로 토큰 값을 바꿔치기

.card {
  --card-padding: 24px;
  padding: var(--card-padding);
}
.card.card--compact {
  --card-padding: 12px;
}

커스텀 프로퍼티가 상속되기 때문에, 특정 컴포넌트 스코프 안에서만 값을 재정의하면 그 서브트리에만 영향을 주고 나머지 문서는 그대로 유지됩니다. 이는 전역 SASS 변수를 오버라이드하려면 새 클래스를 만들고 다시 전체 속성을 반복 선언해야 했던 방식보다 훨씬 국소적이고 안전합니다.

왜 이 방식이 대규모 팀에서 특히 중요한가

디자인 시스템 팀과 기능 팀이 분리된 조직에서는, 디자인 토큰 값이 한 곳(:root 또는 테마 스코프)에서만 관리되고 나머지 코드베이스는 그 값을 참조만 하는 구조가 유지보수의 핵심입니다. 커스텀 프로퍼티의 런타임 상속은 이 "단일 진실 공급원(single source of truth) + 자동 전파" 구조를 브라우저 네이티브 기능만으로 구현하게 해줍니다. Day 6~7에서 다룬 Cascade Layers, low-specificity selector 전략과 결합하면, 값의 출처(토큰)와 우선순위(레이어/specificity)를 모두 예측 가능하게 통제할 수 있습니다.

5. 판단 프레임워크 — 커스텀 프로퍼티 기반 토큰 설계 절차

"이 값을 커스텀 프로퍼티로 뽑을지, 어느 스코프에 둘지"를 판단할 때 아래 순서로 접근합니다.

  • 1
    이 값이 문서/서비스 전체에서 일관되어야 하는가, 컴포넌트 로컬인가 판단

    브랜드 컬러, 타이포그래피 스케일, 기본 spacing 단위는 전역(:root) 토큰 후보. 특정 컴포넌트 내부 여백처럼 다른 곳에 영향을 주면 안 되는 값은 컴포넌트 스코프 커스텀 프로퍼티 후보.

  • 2
    테마/조건에 따라 값이 바뀌어야 하는가 확인

    다크모드, 브랜드별 화이트라벨링, 사용자 설정(글자 크기 등)처럼 런타임에 값이 바뀌어야 한다면 SASS 변수가 아니라 반드시 CSS 커스텀 프로퍼티로 설계합니다 — 빌드 타임 치환은 런타임 전환에 대응할 수 없습니다.

  • 3
    정의 누락 시 동작을 fallback으로 명시

    var(--gap, 16px)처럼 항상 두 번째 인자를 채워, 토큰이 아직 정의되지 않은 초기 렌더링이나 오타 상황에서도 예측 가능한 값을 보장합니다.

  • 4
    타입/애니메이션 가능 여부가 중요하면 @property 등록 고려

    기본 커스텀 프로퍼티는 타입이 없는 문자열 취급이라 transition이 되지 않습니다. 색상/각도/숫자를 부드럽게 전환해야 하면 @property --angle { syntax: '<angle>'; inherits: true; initial-value: 0deg; }처럼 타입을 등록해 애니메이션 가능하게 만듭니다.

  • 5
    상속을 의도적으로 차단해야 하면 @property의 inherits: false 활용

    기본적으로 모든 커스텀 프로퍼티는 상속되지만, 컴포넌트가 부모의 값을 절대 물려받지 않고 항상 자기 초기값에서 시작해야 하는 특수한 경우(예: 재귀적으로 중첩되는 컴포넌트의 depth 카운터) inherits: false로 명시적으로 차단합니다.

  • 6
    Cascade Layers/naming convention과 결합해 토큰 계층을 문서화

    primitive 토큰(--blue-500) → semantic 토큰(--color-primary: var(--blue-500)) → 컴포넌트 토큰(--button-bg: var(--color-primary)) 계층을 두어, 브랜드 색만 바꿔도 전체 UI가 일관되게 갱신되게 합니다.

6. 항목별 판단 체크리스트

항목 올바른 방법 흔한 오용 오용 시 실제 영향
전역 토큰 정의 :root에 primitive/semantic 토큰을 계층적으로 선언 컴포넌트마다 색상 값을 직접 하드코딩 브랜드 색 변경 시 수십 개 파일을 수동으로 수정해야 함
테마 전환 스코프 셀렉터([data-theme])로 커스텀 프로퍼티 재정의, 상속으로 자동 전파 JS로 각 요소의 style을 직접 순회하며 변경 테마 전환 시 리플로우 폭증, 새 컴포넌트마다 전환 로직 추가 필요
fallback 값 var(--token, 기본값)으로 정의 누락 시 동작 명시 fallback 없이 사용해 미정의 시 조용히 깨짐 오타난 토큰 이름이 빌드 타임에 걸러지지 않고 프로덕션에서만 발견됨
애니메이션 대상 값 @property로 타입 등록 후 transition 일반 커스텀 프로퍼티에 그냥 transition 시도 값이 즉시 전환되지 않고 스텝처럼 튐(타입 없는 문자열은 보간 불가)
SASS→CSS 변수 이관 런타임 변경이 필요한 값만 우선 이관, 순수 정적 값은 유지 가능 모든 SASS 변수를 무조건 커스텀 프로퍼티로 일괄 치환 불필요한 런타임 계산 비용 증가, 빌드 타임 최적화(트리 셰이킹 등) 손실

7. 답변 사고 순서

  1. Inheritance가 Cascade 이후 값이 비어있을 때 작동하는 별도 단계라는 것을 먼저 짚습니다. "선언 경쟁의 승자를 가리는 것"(Cascade)과 "선언이 아예 없을 때 값을 채우는 것"(Inheritance)이 다른 문제라는 걸 분명히 합니다.
  2. color/border가 각각 어느 그룹에 속하는지, 그리고 그 그룹 분류의 설계 원칙(텍스트 일관성 vs 박스 독립성)을 설명합니다.
  3. CSS 커스텀 프로퍼티가 기본적으로 상속 속성으로 설계되었다는 것으로 연결합니다. 이것이 SASS 변수(빌드 타임 치환)와 근본적으로 다른 런타임 메커니즘이라는 점을 대비시킵니다.
  4. 이 상속 특성을 design token, 다크모드 전환, 컴포넌트 스코프 오버라이드에 실제로 응용한 사례를 제시합니다.
  5. fallback 값, @property 같은 안정성/확장 도구까지 언급해 실무 완성도를 보입니다.

8. 핵심 답변 예시

면접관: "color는 자식 요소에 상속되는데 border는 왜 상속되지 않을까요? 이 차이를 design token 설계에 어떻게 활용하시겠어요?"

STRONG ANSWER

Inheritance는 Cascade가 끝난 뒤, 그 요소를 겨냥한 선언이 하나도 없을 때 값을 채우는 후속 단계입니다. 이때 속성마다 "매칭되는 선언이 없으면 부모 값을 물려받을지, 아니면 초기값을 쓸지"가 미리 정해져 있는데, 이건 임의가 아니라 속성의 성격에 따른 설계입니다. color, font-family 같은 타이포그래피 속성은 문서 전체가 일관된 톤을 가지는 게 자연스러워서 상속되고, border, margin, width 같은 박스 모델 속성은 각 요소가 독립적인 시각적 경계를 가져야 해서 상속되지 않습니다. 만약 border가 상속됐다면 부모에 테두리 하나 줬을 때 자식 전체가 테두리로 뒤덮이는 통제 불가능한 상황이 됐을 겁니다.


이 원칙을 design token에 그대로 활용합니다. CSS 커스텀 프로퍼티는 기본적으로 상속 속성으로 설계돼 있어서, :root에 --color-primary 같은 토큰을 선언하면 문서 트리 전체에 자동으로 전파됩니다. 이게 SASS 변수와 결정적으로 다른 점인데, SASS 변수는 빌드 타임에 텍스트로 치환되고 사라지는 반면 커스텀 프로퍼티는 런타임에 Cascade를 거쳐 실제로 계산되는 값입니다. 그래서 [data-theme="dark"] 스코프에서 토큰 값만 재정의하면, 이를 참조하는 모든 하위 컴포넌트가 JS 개입 없이 상속을 통해 자동으로 새 값을 받아갑니다. 반대로 특정 컴포넌트 내부에서만 값을 바꾸고 싶으면 그 컴포넌트 스코프에서 로컬로 재정의하면 되고, 이건 상속이 트리 구조를 따라 지역적으로 전파되기 때문에 가능한 겁니다.


실무에서는 여기에 var(--token, fallback)으로 정의 누락 시 동작을 명시하고, 값을 애니메이션해야 하면 @property로 타입을 등록해 transition이 되도록 만듭니다.

이 답변이 좋은 이유:

  • Inheritance를 Cascade와 명확히 구분된 별도 단계로 정확히 위치시킴
  • 상속 여부가 임의가 아니라 속성 성격에 따른 설계라는 원칙을 제시함
  • 커스텀 프로퍼티와 SASS 변수의 근본적 메커니즘 차이(런타임 vs 빌드타임)를 정확히 앎
  • 다크모드/컴포넌트 스코프 오버라이드라는 구체적 실무 응용까지 연결함
  • fallback, @property 같은 실무 완성도 도구까지 언급함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"color는 상속되는 속성이고 border는 상속 안 되는 속성이에요. CSS에 그렇게 정해져 있어요."

이 답변이 약한 이유

  • 사실은 맞지만 "왜" 그렇게 나뉘는지 설계 원칙이 없어 암기처럼 들립니다.
  • CSS 커스텀 프로퍼티와 연결하지 못해 후속 질문에서 막힙니다.
  • 실무 응용(design token)으로 확장되지 않아 시니어 레벨의 판단력을 보여주지 못합니다.
WEAK ANSWER — 또 다른 패턴

"CSS 변수도 결국 SASS 변수랑 비슷한 거 아닌가요? 그냥 값을 재사용하는 용도로 쓰면 되는 거죠."

이 답변이 약한 이유

  • 런타임 계산 값(커스텀 프로퍼티)과 빌드 타임 텍스트 치환(SASS 변수)이라는 근본적 차이를 모르고 있습니다.
  • 이 오해를 가지면 다크모드처럼 런타임 전환이 필요한 기능을 설계할 때 왜 SASS 변수로는 안 되는지 설명하지 못합니다.
STRONG ANSWER 요소 체크리스트
  • Inheritance를 Cascade 이후 값이 비어있을 때 작동하는 별도 단계로 정확히 설명한다
  • 상속 여부가 텍스트 일관성 vs 박스 독립성이라는 설계 원칙에 따른 것임을 설명한다
  • CSS 커스텀 프로퍼티가 기본적으로 상속 속성이라는 것을 안다
  • 커스텀 프로퍼티(런타임 계산)와 SASS 변수(빌드 타임 치환)의 메커니즘 차이를 정확히 구분한다
  • 다크모드, 컴포넌트 스코프 오버라이드 같은 구체적 실무 응용 사례를 제시한다
  • fallback 값, @property 같은 안정성 도구까지 언급한다

10. Senior-level Follow-up Q&A

Q: CSS 커스텀 프로퍼티가 정의되지 않은 채로 var(--foo)를 참조하면 어떻게 되나요?

모범 답변: "guaranteed-invalid value"라는 특수한 값이 됩니다. 일반 CSS 선언이 잘못됐을 때처럼 파서가 그 선언 전체를 무시하는 게 아니라, 이 값을 최종적으로 사용하는 속성이 상속 속성이면 부모의 값을, 비상속 속성이면 unset과 같은 동작(즉 initial 또는 inherited)을 하게 됩니다. 실무에서는 이 동작에 의존하지 않고 var(--foo, fallback)으로 두 번째 인자를 항상 채워서 정의 누락 시의 결과를 명시적으로 통제합니다.

Q: inherit, initial, unset, revert 중 리셋 CSS를 작성할 때는 무엇을 쓰는 게 맞나요?

모범 답변: 상황에 따라 다릅니다. 서드파티 위젯이나 CMS에서 삽입된 콘텐츠처럼 프로젝트 CSS의 영향을 완전히 걷어내고 브라우저 기본 스타일로 되돌리고 싶다면 all: revert가 적합합니다. 반면 속성별로 "이 속성의 기본 상속 동작을 따르라"는 의도라면 unset이 더 명확합니다 — 상속 속성이면 부모 값을, 비상속 속성이면 명세 초기값을 매 속성마다 알아서 선택해주기 때문에, 어떤 속성이 상속되는지 하나하나 외울 필요가 없습니다. initial은 상속 여부와 무관하게 항상 명세 초기값으로 못박고 싶을 때만 씁니다.

Q: 커스텀 프로퍼티도 !important나 specificity의 영향을 받나요?

모범 답변: 네, 커스텀 프로퍼티도 결국 하나의 선언이기 때문에 Day 6~7에서 다룬 Cascade 규칙(Origin/Importance → Layer → Specificity → Source Order)을 동일하게 따릅니다. .card { --gap: 8px !important; }처럼 !important를 붙일 수도 있고, 더 높은 specificity의 selector가 재정의하면 그 값이 이깁니다. 다만 실무에서는 값 자체(--gap의 재정의)와 그 값을 소비하는 속성(gap: var(--gap))이 서로 다른 selector에 있을 수 있다는 점 때문에 디버깅이 한 단계 더 복잡해집니다 — DevTools의 Computed 패널에서 커스텀 프로퍼티가 어느 selector에서 최종적으로 결정됐는지 추적하는 습관이 필요합니다.

Q: 재귀적으로 중첩되는 컴포넌트(예: 트리 뷰, 중첩 코멘트)에서 커스텀 프로퍼티가 상속되는 게 오히려 문제가 될 수 있나요?

모범 답변: 네, 될 수 있습니다. 예를 들어 --indent-level을 자식이 calc(var(--indent-level) + 1)로 누적하려는 의도였는데, 자식이 자신만의 값을 재정의하지 않으면 조상의 값을 그대로 상속받아 원치 않는 값을 참조하게 될 수 있습니다. 이런 "매 레벨마다 반드시 새로 계산되어야 하고 조상 값을 그대로 물려받으면 안 되는" 케이스는 @property로 inherits: false를 명시해 상속 자체를 차단하고, 항상 initial-value에서 시작하도록 강제하는 것이 안전합니다.

Q: 기존 SASS 기반 프로젝트를 커스텀 프로퍼티로 마이그레이션한다면 어떤 기준으로 우선순위를 정하시겠어요?

모범 답변: 모든 SASS 변수를 한 번에 바꾸기보다, "런타임에 값이 바뀌어야 하는가"를 기준으로 우선순위를 정합니다. 다크모드/화이트라벨 테마 색상, 사용자 설정 폰트 크기처럼 런타임 전환이 필요한 값은 커스텀 프로퍼티로 전환해야 SASS로는 애초에 불가능한 기능이 가능해집니다. 반면 빌드 시점에 고정되어도 무방한 순수 계산값(예: breakpoint 리스트를 mixin에서 쓰는 경우)은 SASS 변수로 남겨둬도 실무적으로 문제가 없고, 오히려 불필요한 런타임 비용을 줄일 수 있습니다. 즉 마이그레이션은 "전부 아니면 전무"가 아니라 두 도구를 용도에 맞게 공존시키는 것이 현실적인 접근입니다.

11. 흔한 오해

오해 1

"상속되는 속성 목록은 무작위이니 그냥 외워야 한다"

실제로는 "문서 전체 일관성이 필요한 텍스트 관련 속성은 상속, 요소별 독립성이 필요한 박스 모델 속성은 비상속"이라는 원칙에 따른 설계입니다. 원칙을 이해하면 처음 보는 속성도 어느 쪽에 가까운지 추론할 수 있습니다.

오해 2

"CSS 커스텀 프로퍼티는 SASS 변수의 CSS 네이티브 버전일 뿐, 동작이 거의 같다"

SASS 변수는 빌드 타임에 텍스트로 치환되어 사라지는 정적 값이고, 커스텀 프로퍼티는 런타임에 Cascade를 거쳐 계산되는 살아있는 값입니다. 이 차이 때문에 커스텀 프로퍼티만이 페이지 로드 이후의 동적 테마 전환을 가능하게 합니다.

오해 3

"커스텀 프로퍼티는 항상 상속되므로 상속을 막을 방법이 없다"

@property로 등록하면서 inherits: false를 지정하면 상속을 명시적으로 차단할 수 있습니다. 재귀적으로 중첩되는 컴포넌트처럼 매번 초기값에서 다시 시작해야 하는 경우 유용합니다.

오해 4

"unset과 initial은 사실상 같은 키워드다"

initial은 상속 여부와 무관하게 항상 명세 초기값으로 되돌리지만, unset은 속성이 상속 속성이면 inherit처럼, 비상속 속성이면 initial처럼 속성마다 다르게 동작합니다. 리셋 목적이라면 대개 unset이 더 예측 가능하고 안전합니다.

오해 5

"커스텀 프로퍼티에 fallback을 안 써도 큰 문제 없다"

fallback이 없으면 오타나 로딩 순서 문제로 토큰이 아직 정의되지 않았을 때 guaranteed-invalid value가 되어, 상속 속성이면 부모 값, 비상속 속성이면 initial로 조용히 대체됩니다. 이 실패가 빌드 타임에 걸러지지 않고 프로덕션에서 시각적 버그로만 드러나는 경우가 많습니다.
