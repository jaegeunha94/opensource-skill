1. 핵심 면접 질문

Primary Question

"코드 리뷰 중 #app .container .card .title처럼 selector가 계속 길어지고 있는 것을 발견했습니다. 이런 패턴이 왜 문제이고, 어떤 컨벤션으로 예방하시겠어요?"

Variant 1

"class를 11개 조합하면 ID selector 1개를 specificity로 이길 수 있나요? 왜 그런가요, 혹은 왜 아닌가요?"

Variant 2

:is()와 :where()는 둘 다 여러 selector를 그룹으로 묶어주는데, specificity 관점에서 어떤 차이가 있고 실무에서 언제 :where()를 선택하시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 "specificity 계산 공식을 외웠는가"가 아니라 "명시도가 코드베이스 전체의 유지보수 비용을 어떻게 결정하는지, 그리고 그 비용이 커지기 전에 구조적으로 막을 수 있는가"를 테스트합니다. Specificity 자체는 브라우저가 규칙을 하나 고르기 위한 단순한 비교 규칙일 뿐이지만, 팀이 이 규칙을 정확히 이해하지 못하면 "안 먹히면 selector를 더 구체적으로" 만드는 습관이 쌓여 코드베이스 전체의 selector가 나선형으로 길어지는 유지보수 재앙으로 이어집니다. Day 6에서 다룬 !important 지옥의 뿌리도 대부분 이 specificity 오해에서 출발합니다.

면접관이 이 질문으로 평가하는 것:

  • Specificity를 10진수 하나의 숫자가 아니라 자릿수가 섞이지 않는 3열(ID / class류 / 태그류) 비교로 정확히 이해하는가 — class를 아무리 많이 조합해도 ID 1개를 절대 이길 수 없다는 것을 근거와 함께 설명할 수 있는가.
  • :not(), :is(), :where(), 결합자(>, +, ~, 공백)의 specificity 기여도를 정확히 아는가 — 특히 :where()가 항상 specificity 0을 반환한다는 최신 스펙 지식이 있는가.
  • selector가 길어지는 것이 "증상"이지 "원인"이 아니라는 것을 아는가 — 근본 원인이 selector 자체가 아니라 우선순위를 통제할 다른 수단(레이어, 컴포넌트 격리, 컨벤션)이 없었다는 것을 진단할 수 있는가.
  • 실무에서 검증된 low-specificity 전략(BEM, 단일 class 원칙, :where() 활용)을 구체적으로 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "ID selector를 쓰지 말라고 하면 됩니다." 이 답변은 왜 ID selector가 문제인지(다시 ID로만 override 가능해지는 경직성), 그리고 class 기반 selector라도 nesting이 깊어지면 같은 문제가 재발한다는 것을 놓칩니다. 시니어는 "specificity를 낮고 평평하게 유지하는 것 자체가 목표"라는 원칙 수준의 답변을 하고, 그 원칙을 구현하는 구체적 컨벤션(BEM의 단일 class, :where()로 감싼 리셋/유틸리티)까지 제시합니다.

3. 전제 개념 — 쉬운 한국어로

Specificity란 무엇인가 — "같은 레이어, 같은 출처 안에서 selector끼리 싸울 때의 심판 규칙"

Day 6에서 봤듯이 Cascade는 출처/중요도 → Cascade Layer → Specificity → 작성 순서 순으로 승부를 가립니다. Specificity(명시도)는 이 중 세 번째 기준으로, 앞의 두 기준(출처/레이어)이 같을 때만 비교됩니다. 즉 selector를 아무리 구체적으로 써도, 더 낮은 Cascade Layer에 있으면 절대 이길 수 없습니다 — specificity는 "레이어 안에서의 싸움"이라는 점을 먼저 기억해야 합니다.

Specificity는 "숫자 하나"가 아니라 "3열(자릿수가 섞이지 않는) 튜플"이다

많은 개발자가 specificity를 "ID=100점, class=10점, 태그=1점"처럼 점수를 더해서 비교하는 하나의 숫자로 착각합니다. 이 착각이 바로 "class 11개를 조합하면 110점이 되어 ID(100점)를 이긴다"는 가장 흔하고 위험한 오해로 이어집니다. 실제로는 아래처럼 세 개의 독립된 열(column)을 순서대로, 자릿수 올림 없이(carry 없이) 비교합니다.

A-B-C 3열 비교 구조 (CSS Selectors Level 4 명세 기준):

A = ID selector 개수 B = class/속성 selector/가상 클래스 개수 C = 태그/가상 요소 개수

비교는 A열을 먼저 비교하고, A가 같을 때만 B열을, B도 같을 때만 C열을 비교합니다. B열이나 C열의 값이 아무리 커도 A열을 절대 뒤집을 수 없습니다. 마치 국가대표 선발전에서 "국가대표 경력자 1명"이 "동네 대회 우승 100번"보다 항상 우선하는 것과 비슷한 구조입니다 — 급이 다른 항목은 아무리 쌓아도 상위 항목을 이길 수 없습니다.

구체적인 계산 예시

#header .nav .item.active a:hover
→ A=1(#header), B=3(.nav, .item, .active, :hover 중 class류 4개... 실제로는 4개)
→ 정확히: ID 1개, class류(.nav .item .active :hover) 4개, 태그류(a) 1개
→ (1, 4, 1)

.button.button--primary
→ ID 0개, class류 2개, 태그류 0개
→ (0, 2, 0)

div#app > section.card p
→ ID 1개(#app), class류 1개(.card), 태그류 3개(div, section, p)
→ (1, 1, 3)

(1, 4, 1)과 (0, 2, 0)을 비교하면 A열에서 1 > 0으로 즉시 승부가 납니다. class류 개수(B열)는 전혀 비교되지 않습니다. 이것이 "class를 아무리 많이 써도 ID 1개를 이길 수 없다"는 규칙의 정확한 근거입니다.

specificity에 기여하지 않는 것들 — 결합자와 전체 선택자

자손 결합자(공백), 자식 결합자(>), 형제 결합자(+, ~)는 selector를 구조적으로 연결할 뿐 specificity에 전혀 기여하지 않습니다. 전체 선택자 *도 마찬가지로 기여도가 0입니다. 그래서 .card .title과 .card > .title은 구조는 다르지만 specificity는 완전히 동일하게 (0, 2, 0)입니다.

:not(), :is()는 "괄호 안에서 가장 명시적인 것"의 specificity를 가져온다

:not()과 :is() 자체는 가상 클래스이지만, 예외적으로 자기 자신이 아니라 괄호 안 selector 목록 중 가장 specificity가 높은 항목의 값을 그대로 가져옵니다. 예를 들어 :not(#foo, .bar)는 #foo(A=1)와 .bar(B=1) 중 더 큰 쪽인 (1, 0, 0)을 가져옵니다. 이 규칙을 모르면 "가상 클래스니까 specificity가 낮을 것"이라고 잘못 예측하게 됩니다.

:where()는 항상 specificity 0 — 실무에서 가장 유용한 최신 기능

:where()는 :is()와 문법·동작은 거의 같지만, 괄호 안에 무엇이 들어있든 항상 specificity (0, 0, 0)으로 계산됩니다. 즉 :where(#header .nav .item)는 ID가 들어있어도 specificity 기여도가 0입니다. 이 특성 덕분에 :where()는 "selector 구조는 복잡하게 지정하고 싶지만, 나중에 한 개의 일반 class로도 쉽게 override할 수 있게 만들고 싶을 때"(CSS 리셋, 컴포넌트 라이브러리, 유틸리티 코드)에 표준적으로 쓰입니다.

4. 실무 연결

"이 버튼만 다른 색으로 바꾸고 싶은데 class를 아무리 추가해도 안 먹혀요"라는 제보의 상당수는, 어딘가에 ID selector나 깊게 중첩된 descendant selector가 이미 박혀 있어서 발생합니다. 이 문제의 재발을 막는 것이 바로 low-specificity 전략의 실무 목표입니다.

Selector가 나선형으로 길어지는 실제 발생 과정

새 기능을 급하게 추가하던 개발자가 기존 스타일을 override하지 못하자, 임시로 .card .header .title.active처럼 descendant selector를 더 깊게 쌓아 specificity를 높여 "일단 이기는" 방식으로 문제를 해결합니다. 다음 개발자는 이 코드를 다시 override해야 할 때, 더 깊은 selector를 또 쌓습니다. 이 과정이 반복되면 팀이 마지막으로 만든 지 6개월 안 된 컴포넌트의 selector 깊이가 5단계, 6단계로 늘어나 있는 것을 발견하게 됩니다 — Day 6의 !important 지옥과 정확히 같은 메커니즘이 selector 길이에서 벌어지는 것입니다.

BEM(Block Element Modifier)이 specificity를 "평평하게" 유지하는 방법

BEM 컨벤션은 .card, .card__title, .card--highlighted처럼 계층 관계를 selector의 중첩이 아니라 클래스 이름 자체의 명명 규칙으로 표현합니다. 그 결과 실무 스타일 규칙의 대부분이 단일 class selector, 즉 항상 (0, 1, 0)이 됩니다. 모든 규칙의 specificity가 같은 층에 있으므로, 승부는 대부분 작성 순서(Source Order)로 결정되고, 이는 selector 자체보다 훨씬 예측 가능하고 통제하기 쉽습니다.

CSS 리셋/유틸리티 라이브러리가 :where()를 쓰는 이유

Tailwind의 preflight 리셋이나 여러 디자인 시스템의 기본 스타일은 :where(ul, ol) { margin: 0; }처럼 :where()로 selector를 감쌉니다. 리셋 스타일은 "가장 먼저 적용되고 가장 쉽게 덮어써져야 하는" 역할이기 때문에, specificity를 의도적으로 0으로 만들어 이후에 오는 어떤 프로젝트 코드(단일 class 하나)도 별도의 노력 없이 리셋을 이길 수 있도록 설계된 것입니다.

디자인 시스템 컴포넌트 라이브러리가 override 가능성을 보장하는 방법

사내 공용 컴포넌트 라이브러리를 만들 때, 컴포넌트 내부 구조가 복잡해 selector가 깊어지더라도 :where()로 내부 구조 selector를 감싸면, 라이브러리를 소비하는 팀이 단일 class 하나만으로도 항상 override할 수 있다는 계약(contract)을 제공할 수 있습니다. 이는 Day 6의 Cascade Layers와 함께 "라이브러리 코드를 안전하게 override 가능한 상태로 배포"하는 두 가지 핵심 도구입니다.

5. 판단 프레임워크 — Specificity 안정화 절차

"selector가 계속 길어지고 있다"는 문제를 받았을 때 아래 순서로 접근합니다.

  • 1
    현재 코드베이스의 specificity 분포를 진단

    stylelint의 no-descending-specificity나 specificity 그래프 도구(예: specificity-graph)로 ID selector 사용 빈도, 평균 nesting 깊이를 측정합니다.

  • 2
    ID selector를 스타일링 용도에서 제거

    ID는 JS 훅이나 프래그먼트 앵커(#section-2) 용도로만 남기고, 스타일은 항상 class로 작성하도록 컨벤션을 정합니다.

  • 3
    BEM 등으로 nesting을 클래스 명명 규칙으로 대체

    계층 관계를 descendant selector 중첩이 아니라 .block__element--modifier 이름으로 표현해, 대부분의 규칙을 (0, 1, 0) 수준으로 평평하게 만듭니다.

  • 4
    리셋/유틸리티/공용 컴포넌트는 :where()로 격리

    override되어야 하는 기반 스타일은 specificity 0으로 감싸, 이후 어떤 팀의 단일 class도 추가 노력 없이 이길 수 있게 만듭니다.

  • 5
    Cascade Layers와 조합해 이중 안전망 구성

    Day 6의 레이어 구조와 결합하면, specificity가 평평해도 레이어 순서로 우선순위를 한 번 더 명시적으로 통제할 수 있습니다.

  • 6
    린트 규칙으로 재발 방지

    stylelint의 selector-max-specificity, selector-max-id, selector-max-compound-selectors 규칙을 CI에 추가해 새 selector가 기준을 넘으면 리뷰에서 걸리도록 만듭니다.

6. 항목별 판단 체크리스트

항목 올바른 방법 흔한 오용 오용 시 실제 영향
ID selector JS 훅/앵커 전용으로만 사용, 스타일링에는 미사용 "확실히 이기려고" ID로 스타일 지정 이후 override에 또 다른 ID나 !important 필요
계층 표현 BEM 등 클래스 명명 규칙으로 계층 표현 descendant selector를 계속 중첩 selector 깊이와 specificity가 나선형으로 증가
리셋/유틸리티 :where()로 specificity 0으로 격리 일반 selector로 작성해 specificity가 발생 프로젝트 코드가 리셋을 override하기 위해 불필요하게 구체적인 selector 필요
selector 그룹핑 :is()/:where()로 중복 제거하며 의도에 맞게 specificity 선택 :not()/:is()가 항상 specificity 0이라고 가정 예상보다 높은 specificity로 의도치 않은 override 실패
재발 방지 stylelint로 selector 깊이/ID 사용 제한을 CI에 강제 코드 리뷰 시 육안으로만 확인 팀 규모가 커지면 규칙이 지켜지지 않고 재누적

7. 답변 사고 순서

  1. Specificity가 Cascade의 세 번째 기준이며, 앞선 출처/레이어가 같을 때만 비교된다는 것을 먼저 짚습니다. selector 문제가 항상 specificity만의 문제는 아니라는 것을 분명히 합니다.
  2. Specificity는 하나의 숫자가 아니라 자릿수 올림이 없는 3열(A-ID, B-class류, C-태그류) 비교라는 것을 정확히 설명합니다. class를 아무리 쌓아도 ID를 이길 수 없는 이유를 여기서 근거로 제시합니다.
  3. selector가 길어지는 근본 원인이 "우선순위를 통제할 다른 수단이 없었다"는 것임을 진단합니다. Day 6의 !important 지옥과 같은 뿌리라는 것을 연결합니다.
  4. BEM 같은 클래스 명명 규칙으로 specificity를 평평하게 만들고, :where()로 리셋/유틸리티를 격리하는 구체적 전략을 제시합니다.
  5. 린트/CI로 재발을 방지하는 프로세스까지 언급해 마무리합니다.

8. 핵심 답변 예시

면접관: "코드 리뷰 중 #app .container .card .title처럼 selector가 계속 길어지고 있는 것을 발견했습니다. 이런 패턴이 왜 문제이고, 어떤 컨벤션으로 예방하시겠어요?"

STRONG ANSWER

이 selector가 문제인 이유는 길이 자체가 아니라, ID(#app)를 포함하고 있어서 A열 specificity가 1로 고정된다는 점입니다. Specificity는 하나의 점수로 합산되는 게 아니라 ID 개수(A) → class류 개수(B) → 태그류 개수(C) 순으로 자릿수 올림 없이 비교되는 3열 구조라서, 이후 어떤 개발자가 이 스타일을 override하려 해도 class를 아무리 조합해도 이 ID를 절대 이길 수 없고, 결국 또 다른 ID나 !important를 쓰게 되는 악순환이 시작됩니다.


근본 원인은 이 팀에 "계층을 selector 중첩 없이 표현하는 방법"이 없었다는 것입니다. 그래서 BEM 같은 명명 규칙을 도입해 .card, .card__title, .card--highlighted처럼 계층 관계를 클래스 이름 자체로 표현하도록 리팩터링합니다. 이렇게 하면 대부분의 규칙이 단일 class, 즉 동일한 specificity (0, 1, 0)를 가지게 되어, 승부가 예측 불가능한 selector 경쟁이 아니라 예측 가능한 작성 순서로 결정됩니다.


리셋 스타일이나 공용 유틸리티처럼 "항상 쉽게 override되어야 하는" 코드는 :where()로 감싸 specificity를 의도적으로 0으로 만듭니다. 마지막으로 stylelint의 selector-max-id, selector-max-specificity 규칙을 CI에 추가해, 새로운 ID 기반 스타일이나 깊은 nesting이 리뷰 없이 병합되지 않도록 프로세스로 막습니다.

이 답변이 좋은 이유:

  • Specificity를 3열 비교 구조로 정확히 설명함
  • 문제의 근본 원인을 selector 길이가 아니라 ID 포함 여부로 정확히 짚음
  • BEM으로 specificity를 평평하게 만드는 구체적 전략을 제시함
  • :where()의 실무 활용처를 정확히 앎
  • 린트/CI를 통한 재발 방지까지 프로세스 레벨로 마무리함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"selector가 너무 기니까 짧게 줄이면 됩니다. .title 하나로 바꾸죠."

이 답변이 약한 이유

  • 왜 길어졌는지(ID 포함, nesting 습관) 원인 분석이 없습니다.
  • 단순히 짧게 줄이면 다른 곳의 .title과 충돌해 새로운 문제가 생길 수 있다는 것을 고려하지 않았습니다.
  • 재발 방지 컨벤션이나 도구가 전혀 언급되지 않았습니다.
WEAK ANSWER — 또 다른 패턴

"class를 충분히 많이 추가하면 ID보다 specificity가 높아져서 문제없이 override할 수 있습니다."

이 답변이 약한 이유

  • Specificity가 3열 독립 비교라는 것을 모르는 상태에서, class는 절대 ID를 이길 수 없다는 사실 자체를 틀리게 알고 있습니다.
  • 이 오해대로 실무에 적용하면 실제로는 override가 계속 실패해 원인을 찾지 못하는 상황에 빠집니다.
STRONG ANSWER 요소 체크리스트
  • Specificity를 A-B-C 3열 독립 비교로 정확히 설명한다
  • class가 아무리 많아도 ID를 이길 수 없는 이유를 근거와 함께 설명한다
  • 문제의 근본 원인을 selector 길이가 아니라 ID 포함/중첩 습관으로 짚는다
  • BEM 등으로 specificity를 평평하게 유지하는 구체적 전략을 제시한다
  • :where()를 override 가능성 보장 도구로 정확히 활용할 줄 안다
  • 린트/CI 재발 방지책까지 언급한다

10. Senior-level Follow-up Q&A

Q: class를 11개 조합하면 ID selector 1개를 specificity로 이길 수 있나요?

모범 답변: 아닙니다. Specificity는 10진수처럼 자릿수가 섞이는 하나의 숫자가 아니라, ID 개수(A) → class/속성/가상클래스 개수(B) → 태그/가상요소 개수(C) 순서로 독립적으로, 자릿수 올림 없이 비교되는 3열 구조입니다. class가 11개든 100개든 B열의 값일 뿐이고, A열에서 ID 1개(1)와 ID 0개(0)를 비교하는 순간 이미 승부가 나기 때문에 B열은 비교조차 되지 않습니다. "class 10개 = 100점, ID 1개 = 100점"처럼 점수를 합산해서 비교한다는 것은 CSS 명세와 다른 잘못된 멘탈 모델입니다.

Q: :not()과 :is()의 specificity는 어떻게 계산되나요? 가상 클래스니까 항상 낮은 값인가요?

모범 답변: 아닙니다. :not()과 :is()는 자기 자신의 specificity를 따로 갖지 않고, 괄호 안 selector 목록 중 가장 specificity가 높은 항목의 값을 그대로 가져옵니다. 예를 들어 :is(#foo, .bar)는 #foo(A=1)가 .bar(B=1)보다 높으므로 (1, 0, 0)이 됩니다. "가상 클래스니까 무조건 낮은 specificity"라고 가정하면 실무에서 예상치 못한 override 실패를 겪게 됩니다.

Q: 그럼 :where()는 :is()와 뭐가 다른가요? 실무에서 언제 :where()를 선택해야 하나요?

모범 답변: 문법과 매칭 동작은 거의 동일하지만, :where()는 괄호 안에 무엇이 들어있든 항상 specificity (0, 0, 0)으로 계산된다는 점이 유일하고 결정적인 차이입니다. 실무에서는 "이후에 오는 코드가 항상 쉽게 override할 수 있어야 하는" 코드 — CSS 리셋, 공용 컴포넌트 라이브러리의 내부 구조 스타일, 유틸리티 기본값 — 를 작성할 때 :where()를 선택합니다. 반대로 "이 그룹 중 어느 하나라도 매칭되면 이 specificity를 유지하고 싶다"는 의도라면 :is()가 맞습니다.

Q: 결합자(>, +, ~, 공백)나 전체 선택자 *는 specificity에 영향을 주나요?

모범 답변: 전혀 영향을 주지 않습니다. 이들은 selector를 구조적으로 연결하거나 "아무 요소나"를 의미할 뿐, A/B/C 어느 열에도 기여하지 않습니다. 그래서 .card .title과 .card > .title은 DOM 구조를 매칭하는 방식은 다르지만 specificity는 동일하게 (0, 2, 0)입니다. 다만 인라인 스타일(style="...")은 이 3열 비교 체계 밖에 있는 별개의 상위 기준으로, Origin/Importance 단계에서 이미 대부분의 author selector보다 우선한다는 점(Day 6)과 혼동하지 않아야 합니다.

Q: BEM을 도입하면 specificity 문제가 완전히 사라지나요?

모범 답변: 완전히 사라지지는 않습니다. BEM은 "계층을 표현하기 위해 selector를 중첩할 필요가 없게" 만들어 대부분의 규칙을 (0, 1, 0) 수준으로 평평하게 유지하는 컨벤션일 뿐, 강제 장치는 아닙니다. 팀원이 BEM 클래스를 쓰면서도 습관적으로 .page .card--highlighted .card__title처럼 다시 중첩하면 동일한 문제가 재발합니다. 그래서 컨벤션과 함께 stylelint 같은 도구로 selector 깊이나 specificity 상한을 강제하는 것이 실무에서 함께 필요합니다.

11. 흔한 오해

오해 1

"Specificity는 ID=100, class=10, 태그=1처럼 점수를 더한 하나의 숫자다"

실제로는 자릿수 올림이 없는 3열(A-B-C) 독립 비교입니다. class를 아무리 많이 조합해도(B열) ID 1개(A열)를 절대 이길 수 없습니다. "10진수처럼 자릿수가 넘어간다"는 가정 자체가 틀렸습니다.

오해 2

":not()과 :is()는 가상 클래스이므로 specificity가 낮다"

이 둘은 예외적으로 괄호 안 selector 목록 중 가장 specificity가 높은 항목의 값을 그대로 가져옵니다. ID가 들어있으면 ID의 specificity를 그대로 가집니다.

오해 3

":is()와 :where()는 사실상 같은 기능이다"

매칭 동작은 같지만 specificity 계산이 다릅니다. :is()는 괄호 안 가장 높은 값을 가져오고, :where()는 무엇이 들어있든 항상 0입니다. 이 차이가 override 가능 여부를 결정하는 실무적으로 매우 중요한 차이입니다.

오해 4

"ID selector는 무조건 나쁘고 절대 쓰면 안 된다"

ID 자체가 나쁜 것이 아니라, ID를 스타일링 용도로 쓰면 이후 override 경직성이 생기는 것이 문제입니다. JS 훅(getElementById)이나 페이지 내 앵커(#section-2) 용도의 ID는 문제가 없습니다. "스타일은 class로, 식별은 ID로"라는 역할 분리가 핵심입니다.

오해 5

"BEM을 쓰면 specificity 문제가 자동으로 완전히 해결된다"

BEM은 규칙(컨벤션)일 뿐 강제 장치가 아닙니다. 팀원이 컨벤션을 지키지 않고 다시 selector를 중첩하면 동일한 문제가 재발하므로, 린트 도구로 강제하는 프로세스가 함께 필요합니다.
