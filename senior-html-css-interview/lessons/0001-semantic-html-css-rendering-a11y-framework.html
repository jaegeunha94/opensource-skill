1. 핵심 면접 질문

Primary Question

"이 랜딩 페이지의 마크업/스타일을 코드 리뷰 해달라는 요청을 받았습니다. 무엇을 기준으로, 어떤 순서로 리뷰하시겠습니까?"

Variant 1

"<div onclick="...">로 만든 버튼을 봤습니다. 무엇이 문제인가요?"

Variant 2

"브라우저가 HTML과 CSS를 받아서 실제로 화면에 그리기까지 어떤 과정을 거치나요? 이 과정을 아는 것이 실무에서 왜 중요한가요?"

2. 왜 면접에서 묻는가

이 질문은 "태그와 속성을 외웠는가"가 아니라 "HTML/CSS를 브라우저가 실제로 해석하고 사용자가 실제로 사용하는 엔지니어링 산출물로 이해하고 있는가"를 테스트합니다. 컴포넌트 라이브러리와 CSS 프레임워크 위에서만 작업해본 사람과, 그 아래에서 실제로 무슨 일이 벌어지는지(DOM, Accessibility Tree, 렌더링 파이프라인)를 그릴 수 있는 사람을 구분하는 질문입니다.

면접관이 이 질문으로 평가하는 것:

  • 구조적 사고 — 시각적 결과물이 아니라 콘텐츠의 의미 구조(semantic)를 먼저 보는가.
  • 접근성 기본기 — 마우스/시각 중심 사용자만이 아니라 키보드·스크린 리더 사용자까지 고려한 판단을 하는가.
  • 렌더링 모델 이해 — CSS 변경이 "그냥 스타일이 바뀌는 것"이 아니라 실제 연산 비용(reflow/repaint/composite)을 유발한다는 것을 아는가.
  • 유지보수 관점 — Cascade와 specificity가 팀 규모가 커질수록 어떻게 기술 부채가 되는지 아는가.
  • Trade-off 판단력 — "정답"이 하나가 아닌 상황(Flexbox vs Grid, 반응형 전략, 크로스브라우저 대응)에서 근거를 들어 판단하는가.

10년차 이하에서 자주 나오는 답변 패턴: "브라우저에서 열어보고 예쁘게 나오면 됩니다" 또는 "class 이름만 잘 짓고 컴포넌트로 쪼개면 됩니다." 이 답변들은 시각적 결과에만 집중할 뿐, 그 결과물이 어떻게 만들어지는지(구조, 접근성, 렌더링 비용)에 대한 근거가 없습니다. 시니어는 시각적 결과 이전에 구조와 비용을 먼저 봅니다.

3. 전제 개념 — 쉬운 한국어로

Semantic HTML이란 무엇인가

Semantic(시맨틱) HTML이란 태그 자체가 그 안에 담긴 콘텐츠의 의미와 구조를 표현하는 것을 말합니다. 책에 비유하면, 목차가 있고 장(chapter)과 절(section)이 명확히 구분된 책과, 페이지 구분 없이 글자만 쭉 나열된 종이 뭉치의 차이와 같습니다. <div>와 <span>은 "여기 뭔가 있다"는 것만 알려줄 뿐 그것이 버튼인지, 제목인지, 내비게이션인지 전혀 알려주지 않습니다. 반면 <button>, <nav>, <h1>은 그 자체로 "이것은 버튼이다", "이것은 내비게이션이다", "이것은 최상위 제목이다"라는 의미를 전달합니다.

이 의미 정보는 사람 개발자만을 위한 것이 아닙니다. 브라우저, 스크린 리더, 검색 엔진 크롤러 같은 기계도 이 의미 정보를 읽고 동작을 결정합니다. 이 사실이 이번 레슨 전체의 핵심 전제입니다.

브라우저가 HTML/CSS를 화면으로 바꾸는 과정 (개요)

브라우저는 HTML과 CSS 텍스트를 받아서 다음 단계를 거쳐 실제 화면 픽셀로 바꿉니다. 건물을 짓는 과정에 비유하면 이해하기 쉽습니다.

  1. HTML 파싱 → DOM 생성: HTML 텍스트를 읽어 트리 구조(DOM)를 만듭니다. 건축 설계도를 그리는 단계입니다.
  2. CSS 파싱 → CSSOM 생성: CSS 텍스트를 읽어 스타일 규칙 트리(CSSOM)를 만듭니다. 인테리어 스타일 가이드를 만드는 단계입니다.
  3. Render Tree 결합: DOM과 CSSOM을 합쳐, 실제로 화면에 그려질 요소만 남긴 트리를 만듭니다 (display: none인 요소는 여기서 빠집니다).
  4. Layout(Reflow): 각 요소가 화면의 어느 위치에, 얼마나 큰 크기로 배치될지 계산합니다. 건물의 각 방의 정확한 좌표와 크기를 확정하는 단계입니다.
  5. Paint: 배치가 끝난 요소에 실제 색, 그림자, 테두리를 칠합니다.
  6. Composite: 여러 레이어로 나뉘어 칠해진 결과를 최종적으로 합성해 화면에 표시합니다.

지금 단계에서는 "이 6단계가 순서대로 존재한다"는 지도만 기억하면 됩니다. 각 단계에서 어떤 CSS 속성이 어떤 단계를 다시 유발하는지는 Day 16~18에서 심화합니다. 지금 중요한 것은, CSS 값 하나를 바꾸는 것이 "공짜"가 아니라 이 파이프라인의 일부(또는 전부)를 다시 실행시키는 연산이라는 사실입니다.

Accessibility Tree란 무엇인가

브라우저는 DOM 트리로부터 Accessibility Tree(접근성 트리)라는 또 다른 트리를 파생시킵니다. 스크린 리더 같은 보조 기술(assistive technology)은 화면의 픽셀을 보는 것이 아니라 이 Accessibility Tree를 읽습니다. 이 트리의 각 노드는 role(역할), name(접근 가능한 이름), state(상태) 세 가지 정보를 가집니다.

<button>저장</button>은 브라우저가 자동으로 role="button", name="저장"을 Accessibility Tree에 채워 넣습니다. 반면 <div onclick="save()">저장</div>은 시각적으로는 똑같아 보여도 Accessibility Tree에는 role이 없는 일반 텍스트 노드로 들어갑니다. 스크린 리더 사용자에게는 "저장"이라는 글자만 읽히고, 그것이 누를 수 있는 버튼이라는 정보는 전달되지 않습니다. 이것이 이번 레슨과 Day 3~4에서 반복해서 다룰 핵심 문제입니다.

Cascade와 Box Model (아주 기초만)

CSS의 Cascade(캐스케이드)는 같은 요소에 여러 스타일 규칙이 충돌할 때 "어느 규칙이 이길지" 결정하는 알고리즘입니다. Box Model은 모든 HTML 요소가 content, padding, border, margin으로 이루어진 상자로 취급된다는 모델입니다. 두 개념 모두 이번 레슨에서는 존재만 짚고, Cascade는 Day 6~8, Box Model은 Day 9에서 구체적으로 다룹니다.

4. 실무 연결

이 프레임워크는 추상적인 이론이 아니라, 실제 프로덕션에서 반복적으로 발생하는 세 종류의 사고를 예방하기 위한 것입니다.

시맨틱/접근성 부족이 만드는 실무 비용

div로 도배된 마크업은 스크린 리더 사용자를 서비스에서 사실상 배제합니다. 미국에서는 웹 접근성 미준수로 인한 ADA(Americans with Disabilities Act) 소송이 매년 수천 건 발생하며, 대상은 대부분 "기능은 있지만 시맨틱/ARIA가 없는" 사이트입니다. 접근성 문제는 나중에 "추가"하기 어렵습니다 — 처음부터 올바른 태그로 짜지 않으면 나중에 전체 컴포넌트를 다시 짜야 하는 경우가 많습니다.

렌더링 이해 부족이 만드는 실무 비용

애니메이션이나 스크롤 이벤트 핸들러 안에서 offsetHeight를 읽고 바로 스타일을 바꾸는 코드는 Layout을 강제로 반복 실행시켜(layout thrashing) 프레임 드롭(jank)을 유발합니다. 대규모 리스트를 렌더링할 때 각 아이템마다 reflow를 유발하는 속성을 건드리면 수백 ms의 렌더링 지연이 발생합니다. 이런 문제는 "느리다"는 막연한 버그 리포트로 올라오고, 원인을 렌더링 파이프라인 단위로 이해하지 못하면 원인을 못 찾고 헤매게 됩니다.

Cascade 설계 부실이 만드는 실무 비용

대규모 팀에서 !important가 누적되고 selector 우선순위가 예측 불가능해지면, "스타일 하나를 고쳤는데 다른 페이지가 깨지는" 사고가 반복됩니다. 이 문제는 Day 6~8(Cascade/Specificity/Inheritance)과 Day 14~15(Design Token/CSS Architecture)에서 구조적으로 해결하는 방법을 다룹니다.

5. 6단계 HTML/CSS 리뷰 프레임워크

시니어가 마크업/스타일 리뷰나 설계 질문을 받았을 때 실제로 따르는 순서입니다. 이후 커리큘럼(Day 2~20)의 모든 주제는 이 프레임워크의 한 단계를 깊게 파는 것과 같습니다.

  • 1
    Semantic Structure 검증 (Day 2~5)

    CSS를 완전히 제거해도 콘텐츠의 구조와 의미가 통하는가? 제목, 목록, 내비게이션, 버튼이 실제로 그 의미를 가진 태그로 표현되어 있는가?

  • 2
    Accessibility Tree 검증 (Day 3~4)

    Tab 키만으로 모든 인터랙션 요소에 접근·조작할 수 있는가? role/name/state가 스크린 리더에 올바르게 노출되는가?

  • 3
    Cascade/Specificity 설계 검토 (Day 6~8)

    스타일 우선순위가 예측 가능한가? !important나 과도하게 구체적인 selector에 의존하고 있지 않은가?

  • 4
    Layout 모델 근거 검토 (Day 9~12)

    Box model, Flexbox, Grid, stacking context 중 이 콘텐츠에 맞는 모델을 선택했는가? 매직 넘버나 임의의 z-index 값에 의존하고 있지 않은가?

  • 5
    렌더링 성능 영향 검토 (Day 16~18)

    애니메이션/인터랙션이 reflow를 반복 유발하는 속성을 건드리고 있지 않은가? 60fps를 유지할 수 있는 속성(transform, opacity)을 쓰고 있는가?

  • 6
    반응형/Cross-browser 검증 (Day 13, 19)

    여러 뷰포트 크기와 주요 브라우저(Chrome, Safari, Firefox)에서 동일하게 동작하는가? 최신 CSS 기능에 fallback이 있는가?

7. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 됩니다.

  1. Semantic 구조부터 확인합니다. CSS를 걷어냈을 때도 콘텐츠 구조가 의미를 가지는지, 올바른 태그가 쓰였는지 봅니다.
  2. Accessibility Tree 관점에서 검증합니다. 키보드만으로 조작 가능한지, role/name/state가 올바른지 확인합니다.
  3. Cascade/Specificity 설계를 검토합니다. 스타일 우선순위가 예측 가능하고 유지보수 가능한 구조인지 봅니다.
  4. Layout 모델 선택 근거를 확인합니다. Flexbox/Grid/Box model 선택이 콘텐츠 특성에 맞는지, 매직 넘버가 있는지 봅니다.
  5. 렌더링 성능 영향을 검토합니다. reflow/repaint를 유발하는 패턴이 있는지, 애니메이션이 GPU 가속 가능한 속성을 쓰는지 확인합니다.
  6. 반응형/크로스브라우저를 검증합니다. 여러 뷰포트와 브라우저에서 동일하게 동작하는지, fallback이 있는지 확인합니다.

8. 핵심 답변 예시

면접관: "이 랜딩 페이지의 마크업/스타일을 코드 리뷰 해달라는 요청을 받았습니다. 무엇을 기준으로, 어떤 순서로 리뷰하시겠습니까?"

STRONG ANSWER

먼저 브라우저 개발자 도구에서 CSS를 잠시 꺼보고, 순수 마크업만으로 콘텐츠 구조가 말이 되는지 확인합니다. 제목이 h1~h6로 계층을 이루는지, 클릭 가능한 요소가 div가 아니라 button이나 a로 되어 있는지를 봅니다.


다음으로 Accessibility 탭에서 Accessibility Tree를 열어, Tab 키만으로 모든 인터랙션 요소에 순서대로 접근되는지, 각 요소의 role과 name이 실제 기능과 일치하는지 확인합니다. 예를 들어 커스텀 드롭다운이 div로만 구현되어 있다면 스크린 리더 사용자는 그것이 드롭다운인지조차 알 수 없으므로 최우선으로 지적합니다.


그다음 스타일시트를 보면서 !important가 많이 쓰였는지, selector가 불필요하게 구체적이지는 않은지 확인합니다. 이런 패턴은 지금 당장은 동작해도 팀이 커지면 스타일 충돌의 원인이 됩니다.


레이아웃은 콘텐츠 특성에 맞는 모델을 썼는지 봅니다. 카드가 가변 개수로 줄바꿈되어야 하는 영역에 float나 고정 width 계산이 쓰여 있다면 Flexbox나 Grid로 바꿀 것을 제안합니다.


인터랙션이 있다면 Rendering 탭의 Paint flashing으로 어떤 속성이 반복적으로 다시 그려지는지 확인하고, 애니메이션이 top/left 같은 Layout 유발 속성 대신 transform/opacity를 쓰고 있는지 봅니다.


마지막으로 Chrome, Safari, 그리고 실제 모바일 뷰포트에서 열어보고, 최신 CSS 기능을 쓴 부분에 fallback이 있는지 확인합니다.

이 답변이 좋은 이유:

  • 6단계 프레임워크를 자연스럽게 녹여서 말함
  • "CSS를 꺼보고 구조를 먼저 본다"는 구체적이고 실행 가능한 검증 방법을 제시
  • Accessibility Tree, Rendering 탭 등 실제 도구를 구체적으로 언급
  • 매 단계마다 "왜 문제인지"를 실무 영향(스크린 리더 사용자 배제, 팀 규모 확대 시 충돌, jank)으로 연결
  • 레이아웃/렌더링/크로스브라우저까지 순서대로 빠짐없이 다룸

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"일단 브라우저에서 열어서 디자인 시안이랑 비교해보고, 다르면 CSS를 수정합니다."

이 답변이 약한 이유

  • 시각적 결과만 기준으로 삼고 있습니다 — 구조, 접근성, 성능은 전혀 언급되지 않았습니다.
  • "디자인과 다르면 고친다"는 마크업의 의미 구조는 아예 검토 대상이 아니라는 뜻입니다.
  • 구체적인 도구나 절차가 없어, 실제로 무엇을 어떻게 확인할지 알 수 없습니다.
WEAK ANSWER — 또 다른 패턴

"class 이름을 BEM 규칙에 맞게 잘 짓고, 컴포넌트 단위로 잘 쪼개져 있으면 좋은 코드라고 생각합니다."

이 답변이 약한 이유

  • 네이밍 컨벤션과 컴포넌트 분리는 유지보수성의 일부일 뿐, 접근성이나 렌더링 성능과는 무관합니다.
  • class 이름이 아무리 잘 지어져도 div로 버튼을 만들면 접근성 문제는 그대로 남습니다.
  • "좋은 코드"의 기준을 코드 스타일 하나로 좁혀서, 시니어가 실제로 확인해야 할 다차원적 기준(구조/접근성/성능/호환성)을 놓치고 있습니다.
STRONG ANSWER 요소 체크리스트
  • 시각적 결과 이전에 semantic 구조를 먼저 확인한다
  • Accessibility Tree/키보드 조작 관점에서 별도로 검증한다
  • Cascade/specificity가 유지보수 관점에서 예측 가능한지 본다
  • 레이아웃 모델 선택에 근거가 있는지 확인한다 (매직 넘버 지양)
  • 렌더링 성능(reflow/repaint 유발 여부)을 실제 도구로 확인한다
  • 여러 브라우저/뷰포트에서 동일하게 동작하는지 검증한다

10. Senior-level Follow-up Q&A

Q: <div onclick="save()">로 만든 버튼과 <button>의 차이를 설명해보세요.

모범 답변: <button>은 브라우저가 자동으로 Accessibility Tree에 role="button"을 채워 넣고, 기본적으로 Tab 키로 focus가 가능하며, Enter/Space 키로 활성화됩니다. <div onclick>은 시각적으로는 같아 보여도 이 세 가지(role, focus 가능 여부, 키보드 활성화)가 전혀 자동으로 제공되지 않습니다. tabindex="0", role="button", 키보드 이벤트 핸들러를 모두 직접 구현해야 겨우 비슷해지는데, 그마저도 :focus-visible 같은 브라우저 기본 스타일이나 폼 제출 시 기본 동작 같은 세부 사항까지는 완전히 재현하기 어렵습니다. "네이티브 시맨틱 요소를 쓰면 공짜로 얻는 것"을 직접 구현하는 것은 항상 더 비용이 크고 놓치는 부분이 생깁니다.

Q: <script> 태그를 만나면 브라우저 파싱이 어떻게 되나요? defer/async가 왜 중요한가요?

모범 답변: 기본적으로 브라우저는 HTML을 위에서부터 순서대로 파싱해 DOM을 만드는데, <script>를 만나면 파싱을 멈추고 스크립트를 다운로드·실행한 뒤에야 이어서 파싱합니다. 이는 스크립트가 document.write처럼 DOM 구조 자체를 바꿀 수 있기 때문에 안전을 위해 파싱을 막는 것입니다. defer는 스크립트를 백그라운드에서 다운로드하되 실행은 DOM 파싱이 끝난 뒤로 미루고, async는 다운로드가 끝나는 즉시(파싱 중이라도) 실행합니다. 이 차이를 모르면 <head>에 무심코 넣은 스크립트 태그 하나가 초기 렌더링을 몇 초씩 지연시키는 원인이 될 수 있습니다.

Q: Accessibility Tree와 DOM Tree는 어떻게 다른가요? ARIA는 이 관계에서 어떤 역할을 하나요?

모범 답변: DOM Tree는 HTML 요소와 속성을 그대로 반영한 트리이고, Accessibility Tree는 그중 보조 기술에 의미 있는 정보(role, name, state)만 추려서 파생된 트리입니다. display: none이거나 aria-hidden="true"인 요소는 DOM에는 존재해도 Accessibility Tree에서는 제외됩니다. ARIA 속성(role, aria-label 등)은 이 Accessibility Tree에 채워질 정보를 개발자가 직접 지정하는 수단입니다. 다만 ARIA는 브라우저의 기본 동작(키보드 조작, 포커스 관리)까지 대신 만들어주지는 않기 때문에, "ARIA를 붙였다"와 "접근성이 확보됐다"는 다른 이야기입니다. 이 내용은 Day 4에서 ARIA 사용 원칙과 함께 깊게 다룹니다.

Q: SEO 관점에서 시맨틱 마크업이 왜 중요한가요? 검색 엔진 크롤러가 실제로 무엇을 보나요?

모범 답변: 크롤러는 시각적 레이아웃이 아니라 HTML 구조와 텍스트 콘텐츠, 그리고 시맨틱 태그가 주는 힌트(제목 계층, nav, article 등)를 근거로 페이지의 주제와 중요도를 판단합니다. div로만 된 페이지도 텍스트는 읽히지만, 어떤 텍스트가 제목이고 어떤 것이 본문인지 구분할 근거가 약해집니다. 또한 클라이언트 사이드 렌더링에 크게 의존하는 페이지는 크롤러가 JavaScript 실행 이후의 최종 DOM까지 파싱해야 콘텐츠를 인식하는데, 이 과정이 항상 완전하게 보장되지는 않습니다. 이 내용은 Day 5에서 실무 판단 기준과 함께 구체적으로 다룹니다.

Q: Reflow와 Repaint의 차이는 무엇이고, 어떤 CSS 속성 변경이 Reflow를 유발하나요?

모범 답변: Reflow(재배치)는 요소의 크기나 위치가 바뀌어 Layout 단계를 다시 계산해야 하는 것이고, Repaint(재도색)는 위치/크기는 그대로인데 색이나 그림자처럼 시각적 속성만 다시 칠하는 것입니다. Reflow는 대상 요소뿐 아니라 그 요소와 관계된 다른 요소들의 레이아웃까지 연쇄적으로 다시 계산시킬 수 있어 비용이 더 큽니다. width, height, top, left, font-size 같은 속성을 바꾸면 Reflow가 발생하고, color, background-color 같은 속성은 Repaint만 발생시킵니다. transform과 opacity는 Layout이나 Paint 단계를 건너뛰고 Composite 단계에서만 처리될 수 있어 애니메이션에 권장됩니다. 구체적인 속성별 비용 표는 Day 17에서 다룹니다.

11. 흔한 오해

오해 1

"div에 class만 잘 지으면 semantic 마크업과 다를 게 없다"

class 이름은 사람이 읽기 위한 것일 뿐, 브라우저나 스크린 리더는 class 이름의 의미를 해석하지 않습니다. class="button"이 붙은 div는 여전히 Accessibility Tree에서 일반 텍스트로 취급됩니다. 의미를 기계에 전달하는 것은 태그와 ARIA 속성이지 class 이름이 아닙니다.

오해 2

"ARIA를 많이 붙일수록 접근성이 좋아진다"

불필요하거나 잘못된 ARIA는 오히려 스크린 리더 사용자를 혼란스럽게 만듭니다. 이미 시맨틱 태그가 제공하는 정보를 ARIA로 덮어써서 충돌을 일으키는 경우도 흔합니다. WAI-ARIA 공식 가이드의 첫 번째 규칙은 "네이티브 시맨틱 요소로 충분하다면 ARIA를 쓰지 마라"입니다. 이 내용은 Day 4에서 구체적으로 다룹니다.

오해 3

"CSS 성능은 selector를 최적화하는 것이 가장 중요하다"

현대 브라우저에서 selector 매칭 자체의 비용은 대부분의 실무 상황에서 미미합니다. 실제 성능에 훨씬 큰 영향을 주는 것은 어떤 속성을 바꾸느냐(Reflow/Repaint/Composite 중 무엇을 유발하는가)입니다. selector 최적화에 시간을 쓰기 전에 애니메이션/인터랙션이 어떤 렌더링 단계를 반복 실행시키는지부터 확인해야 합니다.

오해 4

"반응형은 media query만 있으면 충분하다"

media query는 뷰포트 크기만 알 수 있을 뿐, 그 컴포넌트가 실제로 배치되는 컨테이너의 크기는 알지 못합니다. 같은 컴포넌트가 사이드바에도, 전체 폭에도 들어가는 디자인 시스템에서는 media query만으로 한계가 있고 container query, fluid typography 같은 추가 기법이 필요합니다. Day 13에서 이 차이를 구체적으로 다룹니다.

오해 5

"크로스브라우저 이슈는 이제 다 옛날 얘기다"

주요 브라우저 간 렌더링 엔진 차이(특히 Safari의 WebKit)는 여전히 존재합니다. 최신 CSS 기능(container query, :has(), subgrid 등)은 브라우저별 지원 시점이 다르고, 모바일 Safari는 여러 CSS 동작에서 여전히 독자적인 특이 사항을 갖고 있습니다. "요즘은 다 똑같이 동작한다"는 가정으로 실제 브라우저 테스트를 생략하면 프로덕션에서 특정 브라우저 사용자에게만 UI가 깨지는 사고로 이어집니다.
