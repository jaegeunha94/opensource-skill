1. 핵심 면접 질문

Primary Question

"대규모 코드베이스에서 !important가 여기저기 남발되어 있는 CSS를 유지보수 가능한 구조로 리팩터링해야 합니다. 어떤 접근을 취하시겠어요?"

Variant 1

"브라우저가 여러 스타일 규칙 중 최종적으로 적용할 하나를 결정하는 과정을 단계별로 설명해주세요. Specificity는 그 과정에서 몇 번째 기준인가요?"

Variant 2

"Cascade Layers(@layer)를 도입하면 기존 CSS의 우선순위 문제가 실제로 해결되나요? 도입할 때 주의해야 할 점은 무엇인가요?"

2. 왜 면접에서 묻는가

이 질문은 "selector 우선순위 계산을 외웠는가"가 아니라 "여러 개발자가 몇 년에 걸쳐 쌓은 CSS가 왜 통제 불능이 되는지, 그리고 그 근본 원인을 프로세스/아키텍처 수준에서 되돌릴 수 있는가"를 테스트합니다. !important 남용은 코드가 나빠서 생기는 문제가 아니라, Cascade가 어떻게 동작하는지 팀 전체가 정확히 공유하지 못한 상태에서 "일단 이기고 보자"는 근시안적 선택이 누적된 결과입니다. 시니어는 이 누적 과정을 진단하고, 코드 리뷰나 린트 규칙 같은 프로세스 수준의 해법까지 제시할 수 있어야 합니다.

면접관이 이 질문으로 평가하는 것:

  • Cascade 알고리즘을 정확한 순서로 아는가 — origin(출처)과 importance(중요도)가 specificity보다 먼저 오는 상위 기준이라는 것, Cascade Layers가 이 순서 어디에 끼어드는지를 정확히 아는가.
  • !important가 "문제를 해결"하는 것이 아니라 "문제를 다음 사람에게 미루는" 도구라는 것을 이해하는가 — !important끼리 충돌하면 결국 다시 specificity와 source order 경쟁이 시작되어 같은 문제가 한 단계 위에서 반복된다는 것을 아는가.
  • 근본 원인을 selector 레벨이 아니라 아키텍처 레벨에서 찾는가 — 왜 그 !important가 필요했는지(예: 서드파티 위젯 스타일 덮어쓰기, 우선순위가 불명확한 전역 CSS 로딩 순서) 역추적할 수 있는가.
  • 리팩터링을 안전하게 수행하는 실무 절차를 아는가 — 수백 개의 !important를 한 번에 제거하는 것이 아니라, 회귀를 막으면서 점진적으로 걷어내는 전략을 설계할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "!important를 다 지우고 selector 우선순위를 다시 계산해서 고칩니다." 이 답변은 왜 그 !important들이 애초에 쌓였는지에 대한 원인 분석 없이 증상만 제거하려 합니다. 시니어는 "이 코드베이스에 CSS 로딩 순서 규칙이 없었거나, 서드파티 라이브러리의 스타일을 이길 방법이 !important 말고 없다고 생각했거나, 팀에 selector 작성 컨벤션이 없었을 가능성"까지 짚고, 재발 방지책(Cascade Layers 도입, 린트 규칙, CSS 아키텍처 컨벤션)을 함께 제안합니다.

3. 전제 개념 — 쉬운 한국어로

Cascade란 무엇인가 — "여러 개가 같은 것을 요구할 때 하나를 고르는 규칙"

같은 요소에 색을 지정하는 CSS 규칙이 동시에 여러 개 있을 수 있습니다. 브라우저 기본 스타일, 사용자가 브라우저에 설정한 스타일(접근성을 위해 글자 크기를 키우는 등), 그리고 개발자가 작성한 스타일이 한 요소를 두고 서로 다른 값을 요구할 수 있습니다. Cascade(캐스케이드)는 이렇게 충돌하는 후보 값들 중 "최종적으로 어떤 값을 적용할 것인가"를 결정하는 브라우저의 판정 규칙입니다. "CSS(Cascading Style Sheets)"라는 이름 자체가 이 캐스케이드 개념에서 나왔다는 것을 기억하면, 이 개념이 CSS의 지엽적인 세부사항이 아니라 CSS 전체를 관통하는 근본 원리라는 것을 알 수 있습니다.

Cascade가 후보를 정렬하는 4가지 기준 — 이 순서를 정확히 아는 것이 핵심

브라우저는 같은 요소·같은 속성에 대해 경쟁하는 선언들을 아래 기준을 순서대로 적용해 하나로 좁혀갑니다. 먼저 오는 기준에서 승부가 나면 그 아래 기준은 아예 보지도 않습니다. 이 순서를 뒤바꿔 이해하는 것이 바로 !important 남용과 selector 전쟁의 근본 원인입니다.

  1. Origin과 Importance(출처와 중요도) — 스타일이 어디서 왔는가(브라우저 기본값 / 사용자 설정 / 개발자가 작성한 스타일)와 !important가 붙었는가를 함께 고려하는 최상위 기준입니다. 낮은 우선순위부터 나열하면: User Agent(브라우저 기본) 일반 선언 → 사용자(User) 일반 선언 → Author(개발자) 일반 선언, 이 안에서 Cascade Layer 순서 적용 → Author !important → 사용자 !important → User Agent !important 순으로 뒤집힙니다. 개발자가 작성하는 CSS는 대부분 "Author 일반"과 "Author !important" 두 구간 안에서만 경쟁합니다.
  2. Cascade Layer 순서 — 같은 Author 출처 안에서, @layer로 선언된 레이어들은 나중에 선언된 레이어가 이깁니다. 그리고 레이어를 아예 지정하지 않은 "unlayered" 스타일은 어떤 레이어보다도 항상 우선합니다. 이 규칙 덕분에 "리셋 CSS는 낮은 레이어, 컴포넌트 스타일은 중간 레이어, 유틸리티 클래스는 높은 레이어"처럼 selector의 구체성과 무관하게 우선순위를 명시적으로 통제할 수 있습니다.
  3. Specificity(명시도) — 위 두 기준으로 승부가 나지 않으면(같은 출처, 같은 레이어), selector가 얼마나 "구체적으로" 요소를 지목했는지를 비교합니다. 자세한 계산법은 Day 7에서 다루지만, 핵심은 ID > class/속성/가상 클래스 > 태그 순으로 구체성이 높다는 것입니다.
  4. Source Order(작성 순서) — 출처, 레이어, 명시도까지 모두 같으면, 스타일시트에 나중에 나온 규칙이 이깁니다. 같은 파일 안의 순서뿐 아니라, <link>로 여러 CSS 파일을 로딩하는 순서도 여기에 포함됩니다.

실무에서 가장 자주 놓치는 지점: Cascade Layer는 Specificity보다 먼저 비교되는 상위 기준입니다. 즉 @layer utilities { .btn { color: red; } }가 #header .nav .btn { color: blue; }(ID를 포함한 매우 구체적인 selector)보다도 이깁니다. 레이어가 다르면 selector의 구체성은 승부에 아예 반영되지 않기 때문입니다. 이 특성이 Cascade Layers를 "specificity 전쟁을 구조적으로 끝내는 도구"로 만드는 이유입니다.

!important는 "무기를 한 단계 업그레이드"하는 것일 뿐 — 근본 해결이 아니다

!important는 위 기준 1(Origin/Importance)에서 개발자의 선언을 "Author !important" 구간으로 끌어올립니다. 문제는 다른 개발자도 똑같이 이 무기를 쓸 수 있다는 것입니다. 두 규칙이 모두 !important가 붙으면, 승부는 다시 Cascade Layer → Specificity → Source Order 순서로 내려가 재경쟁하게 됩니다. 즉 !important는 문제를 해결하는 것이 아니라, 같은 종류의 경쟁을 "일반 선언 구간"에서 "!important 선언 구간"으로 옮겨서 반복시키는 것입니다. 팀 전체가 이 사실을 모르면, !important가 통하지 않을 때 !important를 하나 더 강하게(더 구체적인 selector와 함께) 붙이는 악순환이 시작되고, 이것이 실무에서 "!important 지옥"이라 불리는 상태입니다.

Cascade Layers(@layer)가 실제로 해결하는 문제

@layer reset, base, components, utilities;처럼 레이어 순서를 먼저 선언해두면, 각 레이어 안에 작성하는 selector의 구체성과 무관하게 레이어 자체의 순서로 우선순위가 결정됩니다. 예를 들어 서드파티 컴포넌트 라이브러리의 CSS를 낮은 레이어로 감싸두면, 우리 팀의 아무리 낮은 specificity의 override 스타일도 항상 그 라이브러리 스타일을 이깁니다. 이전에는 이런 override를 위해 !important나 매우 구체적인 selector(body .app .widget .button)를 강제로 써야 했지만, Cascade Layers는 "내가 이 스타일을 나중에, 더 낮은 노력으로 이기고 싶다"는 의도를 selector 복잡도가 아니라 레이어 선언으로 표현할 수 있게 해줍니다.

4. 실무 연결

"이 버튼 색을 빨간색으로 바꿨는데 반영이 안 돼요"라는 제보가 들어왔을 때, 원인의 상당수는 selector 오타가 아니라 Cascade의 상위 기준(출처, 레이어, 중요도)에서 이미 승부가 끝나 있었던 것입니다. 이 지점을 이해하지 못하면 개발자는 습관적으로 !important를 덧붙이는 방식으로만 문제를 "해결"하게 됩니다.

서드파티 위젯/디자인 시스템 라이브러리의 스타일 override

실무에서 !important가 쌓이는 가장 흔한 시작점은 외부 컴포넌트 라이브러리(날짜 선택기, 리치 텍스트 에디터, 결제 위젯 등)의 스타일을 우리 서비스 디자인에 맞게 덮어써야 할 때입니다. 라이브러리 내부 CSS의 specificity를 예측하기 어렵기 때문에, 개발자는 안전하게 이기기 위해 !important를 습관적으로 붙입니다. Cascade Layers를 도입한 팀은 대신 @import url("vendor.css") layer(vendor);처럼 서드파티 스타일을 명시적으로 가장 낮은 레이어에 배치하고, 우리 팀의 override는 어떤 레이어에도 넣지 않거나 더 높은 레이어에 두어 !important 없이도 항상 이기도록 만듭니다.

전역 CSS 로딩 순서가 팀 컨벤션이 아니라 우연에 의존하는 문제

여러 팀이 각자 CSS 파일을 만들어 빌드 시스템이 임의로(또는 import 순서에 따라 우연히) 번들링하는 경우, 오늘은 되던 스타일이 다른 팀의 CSS 파일 순서 변경만으로 깨지는 사고가 발생합니다. 성숙한 팀은 CSS 로딩 순서를 @layer 선언으로 코드에 명시해, 번들러의 파일 병합 순서와 무관하게 우선순위가 고정되도록 만듭니다.

디자인 시스템 유틸리티 클래스(.mt-4 같은)와 컴포넌트 스타일의 충돌

utility-first 스타일(Day 15에서 CSS 아키텍처로 더 다룸)을 컴포넌트 스타일과 함께 쓰는 팀에서는, 유틸리티 클래스가 항상 컴포넌트 스타일을 이겨야 하는데 selector 구체성이 우연히 비슷해서 순서에 따라 승부가 뒤바뀌는 문제가 흔합니다. @layer components, utilities;로 유틸리티를 항상 나중 레이어에 두면, 클래스 작성 순서나 selector 구체성과 무관하게 유틸리티가 항상 이기도록 구조적으로 보장할 수 있습니다.

레거시 !important 코드베이스를 점진적으로 걷어내는 실제 절차

수백 개의 !important가 쌓인 코드베이스를 한 번에 리팩터링하는 것은 회귀 위험이 매우 큽니다. 실무에서는 먼저 새 Cascade Layer 구조를 도입해 기존 레거시 CSS 전체를 가장 낮은 레이어 하나에 몰아넣고, 이후 새로 작성하거나 수정하는 컴포넌트만 점진적으로 상위 레이어로 이관합니다. 이렇게 하면 레거시 코드는 그대로 두고도 새 코드가 항상 레거시를 이기도록 만들 수 있어, 기존 !important를 당장 지우지 않고도 실질적인 통제권을 되찾을 수 있습니다.

5. 판단 프레임워크 — !important 리팩터링 절차

"!important가 남발된 CSS를 리팩터링해달라"는 요청을 받았을 때 아래 순서로 접근합니다.

  • 1
    제거가 아니라 진단부터 시작

    각 !important가 "왜" 필요했는지를 먼저 역추적합니다. 서드파티 스타일 override, 로딩 순서 불명확, selector 구체성 예측 실패 중 어떤 패턴인지 분류합니다.

  • 2
    브라우저 지원 범위 확인

    @layer는 비교적 최신 기능이므로, 서비스가 지원해야 하는 브라우저 범위에서 Cascade Layers를 안전하게 쓸 수 있는지 Can I Use 등으로 확인합니다.

  • 3
    레이어 구조를 먼저 설계하고 선언

    @layer reset, vendor, base, components, utilities, overrides;처럼 우선순위 의도를 코드로 명시합니다. 이 선언 자체는 스타일을 하나도 담고 있지 않아도 되며, 순서만 고정합니다.

  • 4
    기존 레거시 CSS를 가장 낮은 레이어로 감싸 회귀 없이 이관

    기존 코드를 selector 단위로 건드리지 않고, 전체를 하나의 낮은 레이어로 감싸는 것만으로 "새 코드가 항상 이긴다"는 규칙을 먼저 확보합니다.

  • 5
    서드파티 스타일을 별도의 최하위 레이어로 격리

    외부 라이브러리 CSS를 명시적인 낮은 레이어에 배치해, 그 자체의 specificity와 무관하게 우리 코드가 override할 수 있게 만듭니다.

  • 6
    새 스타일부터 !important 없이 상위 레이어에 작성

    이후 수정/신규 컴포넌트는 레이어 구조상 이미 우선순위가 확보되어 있으므로 !important가 필요 없다는 것을 팀에 공유합니다.

  • 7
    린트 규칙과 코드 리뷰 체크리스트로 재발 방지

    stylelint의 declaration-no-important 같은 규칙을 CI에 추가해, 새로운 !important가 리뷰 없이 병합되지 않도록 프로세스로 막습니다.

6. 항목별 판단 체크리스트

항목 올바른 방법 흔한 오용 오용 시 실제 영향
서드파티 스타일 override vendor CSS를 낮은 @layer로 격리 override마다 !important 반복 추가 라이브러리 업데이트 시 selector 구체성이 바뀌며 override가 깨짐
CSS 로딩 순서 @layer 선언으로 순서를 코드에 고정 번들러의 import 순서에 우선순위를 암묵적으로 의존 무관한 파일 추가/순서 변경만으로 스타일이 우연히 깨짐
유틸리티 vs 컴포넌트 스타일 유틸리티를 항상 더 높은 레이어에 배치 클래스 작성 순서로 우선순위를 통제하려 시도 HTML의 class 순서를 바꾸면 스타일이 뒤바뀌는 불안정한 동작
레거시 코드 이관 레거시 전체를 낮은 레이어로 감싸고 점진적으로 이관 한 번에 !important를 전부 제거 시도 대규모 회귀, 예측 불가능한 스타일 붕괴
재발 방지 stylelint 등으로 신규 !important를 리뷰 대상으로 강제 문서에만 "쓰지 말자"고 명시 시간이 지나면 동일한 패턴으로 재누적

7. 답변 사고 순서

  1. 먼저 Cascade의 4단계 판정 순서(출처/중요도 → 레이어 → 명시도 → 작성 순서)를 언급해 문제를 구조적으로 정의합니다. !important 남용이 왜 문제인지를 이 프레임 안에서 설명합니다.
  2. !important가 "해결"이 아니라 "경쟁 구간의 이동"이라는 점을 짚습니다. 근본 원인이 selector가 아니라 우선순위 설계의 부재라는 것을 강조합니다.
  3. 각 !important가 왜 필요했는지 원인을 역추적하는 진단 단계를 먼저 제안합니다. 무작정 제거하지 않는다는 것을 분명히 합니다.
  4. Cascade Layers로 우선순위를 selector 구체성과 분리해 명시적으로 설계하는 해법을 제시합니다. 레거시 코드는 낮은 레이어로 감싸 회귀 없이 이관하는 점진적 전략을 설명합니다.
  5. 재발 방지를 위한 프로세스(린트, 코드 리뷰)까지 언급해 마무리합니다. 일회성 리팩터링이 아니라 지속 가능한 구조로 답변을 끝맺습니다.

8. 핵심 답변 예시

면접관: "대규모 코드베이스에서 !important가 여기저기 남발되어 있는 CSS를 유지보수 가능한 구조로 리팩터링해야 합니다. 어떤 접근을 취하시겠어요?"

STRONG ANSWER

먼저 !important를 무작정 지우지 않습니다. 브라우저는 스타일 경쟁을 출처/중요도 → 레이어 → 명시도 → 작성 순서 순으로 판정하는데, !important는 이 중 가장 상위 기준인 "중요도"를 끌어올리는 것뿐이라서, 그 !important가 애초에 왜 필요했는지(서드파티 스타일 override인지, CSS 로딩 순서가 불명확했는지, selector 구체성을 예측하지 못했는지) 원인을 먼저 분류합니다.


대부분의 경우 근본 원인은 "우선순위를 설계할 수단이 selector 구체성밖에 없었다"는 것입니다. 이를 해결하기 위해 Cascade Layers를 도입합니다. 레이어는 specificity보다 먼저 비교되는 상위 기준이라서, @layer reset, vendor, base, components, utilities;처럼 우선순위 의도를 코드로 명시하면 selector 구체성과 무관하게 원하는 우선순위를 보장할 수 있습니다.


기존 레거시 코드는 한 번에 !important를 걷어내지 않고, 전체를 가장 낮은 레이어 하나로 감싸는 것부터 시작합니다. 이렇게 하면 앞으로 작성하는 새 코드는 레이어 구조상 이미 레거시보다 우선순위가 높아, !important 없이도 항상 이기게 됩니다. 서드파티 라이브러리 스타일도 별도의 최하위 레이어로 격리해 안전하게 override할 수 있게 만듭니다.


마지막으로 stylelint에 declaration-no-important 규칙을 추가해 CI에서 새로운 !important가 리뷰 없이 병합되지 않도록 프로세스로 막아, 같은 문제가 재발하지 않게 합니다.

이 답변이 좋은 이유:

  • Cascade의 정확한 판정 순서를 근거로 문제를 구조적으로 정의함
  • 제거가 아니라 원인 진단부터 시작함
  • Cascade Layers가 specificity보다 상위 기준이라는 정확한 이해를 보여줌
  • 레거시 코드를 한 번에 건드리지 않는 회귀 안전한 점진적 전략을 제시함
  • 린트/CI를 통한 재발 방지까지 프로세스 레벨로 마무리함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"!important를 하나씩 지우고, 안 되는 부분은 selector를 더 구체적으로 바꾸면 됩니다."

이 답변이 약한 이유

  • 왜 !important가 필요했는지 원인 분석 없이 증상만 제거하려 합니다.
  • "selector를 더 구체적으로" 만드는 것은 결국 specificity 경쟁을 심화시켜 유지보수성을 더 악화시킵니다.
  • 회귀 위험을 관리하는 절차(점진적 이관)가 전혀 없습니다.
WEAK ANSWER — 또 다른 패턴

"CSS-in-JS나 CSS Modules로 전면 전환하면 !important 문제가 사라집니다."

이 답변이 약한 이유

  • 도구 전환이 근본 원인(우선순위 설계 부재)을 진단 없이 우회하려는 시도입니다.
  • 전면 전환의 비용과 리스크를 전혀 고려하지 않았습니다.
  • Cascade 자체는 CSS-in-JS를 쓰더라도 여전히 동일하게 적용된다는 것을 놓치고 있습니다.
STRONG ANSWER 요소 체크리스트
  • Cascade의 판정 순서(출처/중요도 → 레이어 → 명시도 → 작성 순서)를 정확히 언급한다
  • !important를 제거하기 전에 왜 필요했는지 원인을 분류한다
  • Cascade Layers가 specificity보다 상위 기준이라는 것을 정확히 이해하고 있다
  • 레거시 코드를 한 번에 건드리지 않는 회귀 안전한 이관 전략을 제시한다
  • 린트/CI 같은 프로세스 수준의 재발 방지책까지 언급한다

10. Senior-level Follow-up Q&A

Q: Cascade Layer와 Specificity 중 어느 쪽이 더 우선하나요? 예를 들어 낮은 레이어의 ID selector와 높은 레이어의 클래스 selector가 충돌하면 무엇이 이기나요?

모범 답변: 레이어가 specificity보다 먼저 비교되는 상위 기준이므로, 높은 레이어의 클래스 selector가 이깁니다. 낮은 레이어에 ID selector로 작성된 스타일은 아무리 명시도가 높아도, 더 높은 레이어에 있는 명시도가 낮은 클래스 selector를 이길 수 없습니다. 이 특성이 바로 Cascade Layers를 "specificity 전쟁을 구조적으로 끝내는 도구"로 만드는 이유이고, 레이어를 도입하기 전에는 이 관계를 팀 전체가 정확히 이해하고 있어야 예상치 못한 override 실패를 피할 수 있습니다.

Q: 레이어를 지정하지 않은(unlayered) 스타일은 어떤 우선순위를 가지나요?

모범 답변: 일반(!important가 아닌) 선언 기준으로, unlayered 스타일은 어떤 이름 있는 레이어보다도 항상 우선합니다. 즉 @layer 블록 밖에 그냥 작성한 CSS는 레이어 구조 전체보다 위에 있는 "최상위 안전망"처럼 동작합니다. 그래서 팀에서 레이어 체계를 도입할 때는 "예외적으로 반드시 이겨야 하는 긴급 override"를 위한 공간으로 unlayered 영역을 의도적으로 남겨두는 경우도 있지만, 이를 남용하면 레이어 체계 자체가 무력화되므로 신중하게 관리해야 합니다.

Q: !important가 붙은 스타일끼리 충돌하면 레이어 우선순위가 어떻게 되나요? 일반 선언과 같은 순서인가요?

모범 답변: 아닙니다. 이 부분이 실무에서 가장 자주 놓치는 함정입니다. !important 선언끼리 경쟁할 때는 레이어 우선순위가 반대로 뒤집힙니다. 일반 선언에서는 나중에 선언된 레이어가 이기지만, !important 선언에서는 먼저 선언된 레이어가 이깁니다. 그리고 unlayered !important는 어떤 레이어의 !important보다도 낮은 우선순위를 가집니다(일반 선언에서 unlayered가 항상 최상위였던 것과 정반대). 그래서 팀이 Cascade Layers를 도입했는데도 레이어 안에서 여전히 !important를 쓰고 있다면, 의도한 것과 정반대의 우선순위가 적용될 위험이 있습니다. 이것이 Cascade Layers를 도입하는 팀에게 "레이어 안에서는 !important를 아예 쓰지 않는다"는 규칙을 함께 정착시켜야 하는 이유입니다.

Q: 사용자가 브라우저에 직접 설정한 스타일(User 스타일시트)이 개발자의 !important를 이길 수 있나요?

모범 답변: 네, 가능합니다. Origin/Importance 기준에서 "User !important"는 "Author !important"보다 우선순위가 높습니다. 이는 의도적인 설계로, 시각 장애나 저시력 사용자가 접근성을 위해 브라우저 확장 프로그램이나 사용자 스타일시트로 강제한 글자 크기·색상 대비 설정이 웹사이트 개발자의 스타일에 항상 이길 수 있어야 한다는 원칙을 반영한 것입니다. 즉 개발자의 !important는 "무조건 이기는 무기"가 아니라 "Author 출처 안에서만 최상위"라는 것을 정확히 이해해야 합니다.

Q: @layer를 지원하지 않는 구형 브라우저를 지원해야 한다면 어떻게 하나요?

모범 답변: @layer를 인식하지 못하는 브라우저는 해당 at-rule 전체를 무시하므로, 레이어로 감싼 스타일이 통째로 적용되지 않는 것이 아니라 레이어 선언 문법 자체를 파싱하지 못해 무시하는 방식으로 우아하게 성능 저하(graceful degradation)됩니다. 다만 이 경우 레이어가 주던 우선순위 통제가 사라지고 순수 specificity/source order로 되돌아가므로, 구형 브라우저 지원이 필수인 서비스라면 레이어 도입 전 실제 지원 범위를 확인하고, 레이어에 의존하지 않아도 되는 폴백 우선순위 전략(예: BEM 같은 selector 컨벤션)을 함께 준비해야 합니다.

11. 흔한 오해

오해 1

"!important는 무조건 이기는 최종 무기다"

!important는 Origin/Importance 기준에서 우선순위를 한 단계 끌어올릴 뿐이며, 다른 !important와 경쟁하면 다시 레이어/명시도/작성 순서로 승부가 갈립니다. 게다가 User !important가 Author !important보다 우선하므로 절대적인 무기가 아닙니다.

오해 2

"Cascade는 결국 specificity 계산이다"

Specificity는 Cascade의 네 기준 중 세 번째입니다. Origin/Importance와 Cascade Layer가 먼저 비교되고, 그 단계에서 승부가 나면 specificity는 아예 비교 대상에 오르지도 않습니다.

오해 3

"Cascade Layers 안에서도 !important는 평소처럼 나중 레이어가 이긴다"

!important 선언 사이에서는 레이어 우선순위가 반대로 뒤집혀 먼저 선언된 레이어가 이깁니다. 레이어 도입 후에도 !important를 쓰면 의도와 정반대의 결과가 나올 수 있습니다.

오해 4

"!important가 많다는 것은 곧 코드 품질이 나쁘다는 뜻이다"

!important 자체가 원인이 아니라 증상입니다. 근본 원인은 대개 우선순위를 설계할 수단이 selector 구체성밖에 없었던 아키텍처 부재이며, 이를 고치지 않고 !important만 제거하면 같은 문제가 다른 형태로 재발합니다.

오해 5

"Cascade Layers를 도입하면 즉시 모든 우선순위 문제가 사라진다"

레이어는 우선순위를 설계할 수 있는 도구를 제공할 뿐, 레이어 순서 자체를 잘못 설계하거나(예: 유틸리티를 낮은 레이어에 둠) 레이어 안에서 !important를 계속 쓰면 문제는 형태만 바뀐 채 그대로 남습니다.
