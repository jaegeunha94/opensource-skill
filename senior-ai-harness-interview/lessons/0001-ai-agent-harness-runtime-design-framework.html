Day 1 · AI Harness / Agent Harness Interview

AI Harness와 Agent Runtime 설계/운영 프레임워크

"Agent = Model + Harness" — LLM을 감싸는 런타임을 왜, 어떻게 설계하는지 구조적으로 설명하는 능력
버전/시점 기준: 이 레슨은 2026-07 기준 Claude Agent SDK(공식 agent loop 문서화)와 OpenAI Agents SDK(Responses API 기반, Assistants API는 2026-08-26 완전 종료)를 대표 사례로 다룬다. "Harness Engineering"이라는 용어는 2026년 초 Martin Fowler와 Thoughtworks가 공식적으로 정립했다. 이 트랙은 LangChain/LangGraph 문법을 다루지 않는다(별도 트랙 참고) — provider를 넘나드는 원리에 집중한다.

전제 개념

Agent = Model + Harness — 왜 "LLM 호출"만으로는 부족한가

LLM 호출과 Agent의 차이

사용자가 "이번 분기 매출 리포트를 만들어서 팀 채널에 공유해줘"라고 요청했다고 하자. 이 작업을 끝내려면 모델은 데이터베이스에서 데이터를 조회하고, 결과를 분석하고, 리포트를 작성하고, Slack API를 호출해야 한다. 그리고 각 단계마다 이전 단계의 결과를 보고 다음에 뭘 할지 스스로 판단해야 한다. 데이터 조회가 실패하면 재시도할지, 다른 쿼리를 시도할지, 사용자에게 물어볼지도 모델이 결정해야 한다.

이건 API를 한 번 호출해서 텍스트 응답을 받는 것과는 완전히 다른 문제다. "모델이 여러 단계를 반복하며 스스로 판단해서 목표를 달성하는 구조"가 필요하고, 이 반복 구조 자체를 누군가는 코드로 만들어야 한다. 이 코드를 harness(하네스)라고 부른다.

Harness란 무엇인가

Harness는 모델에게 어떤 tool을 줄지 정의하고, 모델의 응답(tool 호출 요청)을 파싱해서 실제로 실행하고, 실행 결과를 다시 모델에게 넣어주고, 언제 루프를 멈출지 결정하는 "배관 공사(plumbing)" 역할을 한다. 업계에서는 이를 "Agent = Model + Harness"라고 표현한다. 모델(두뇌)만으로는 아무것도 실행할 수 없고, harness(몸통 + 신경계)가 있어야 실제로 도구를 쓰고, 상태를 유지하고, 실패를 복구할 수 있다.

2026년에 정립된 용어 — "Harness Engineering": Martin Fowler와 Thoughtworks의 Birgitta Böckeler는 2026년 초 이 개념을 "harness engineering"이라는 이름으로 정리했다. 핵심 주장은 "LLM은 agent 시스템에서 가장 작은 부분이다"라는 것이다. tool, 권한 관리, 상태 저장, 테스트, 로그/관측성, guardrail, 실패 복구 메커니즘 — 이 모든 "harness"가 실제로 시스템의 신뢰성과 비용, 보안을 결정한다. 면접관이 이 개념을 직접 언급하지 않아도, 이 사고방식 자체가 시니어 후보를 가려내는 기준이다.

Agent Loop의 표준 구조 — gather context → take action → verify → repeat

Anthropic의 Claude Agent SDK 공식 문서는 agent loop를 다음 세 단계의 반복으로 정의한다.

  1. Gather context (맥락 수집): 모델은 사용자 프롬프트, 시스템 프롬프트, tool 정의 목록, 지금까지의 대화 히스토리를 입력으로 받는다. "지금 상황이 무엇인가"를 파악하는 단계다.
  2. Take action (행동 실행): 모델은 주어진 tool 중 하나를 호출하기로 결정한다. Harness는 이 요청을 실제 함수 호출, API 요청, 셸 명령 등으로 실행한다.
  3. Verify & repeat (검증 후 반복): 실행 결과를 모델에게 다시 보여주고, 모델이 목표를 달성했는지 스스로(또는 harness가) 판단한다. 달성하지 못했다면 1단계로 돌아간다.

이 3단계 자체는 단순해 보이지만, 실무에서 harness가 진짜로 책임져야 하는 것은 "이 루프를 얼마나 안전하고, 관측 가능하고, 비용 효율적으로 반복시키는가"다. 

# Agent loop의 최소 골격 (개념 설명용 의사코드)
def run_agent(user_input, tools, max_steps=10):
    messages = [{"role": "user", "content": user_input}]
    for step in range(max_steps):  # ① 무한루프 방지: hard step cap
        response = model.call(messages, tools=tools)  # gather context

        if response.stop_reason == "end_turn":
            return response.text  # verify: 목표 달성 판단

        for tool_call in response.tool_calls:
            result = execute_tool(tool_call)  # take action: 실제 실행 + 권한 검사
            messages.append(tool_result_message(tool_call, result))

    raise MaxStepsExceeded()  # ② 강제 종료: 예산 초과 시 반드시 멈춘다

실제로는 여기에 token_budget, dollar_budget, 동일한 tool 호출 반복 감지, 권한 검사, 로깅/트레이싱까지 붙어야 "production-grade harness"가 된다. 이 세부 사항은 Day 4(제어 흐름), Day 5(guardrail), Day 6(샌드박스)에서 각각 깊게 다룬다.

Model / Tool / Runtime — 세 층으로 경계를 나눠서 사고하기

계층 책임 예시
Model 주어진 맥락에서 다음 행동을 추론·결정. harness가 무엇을 실행할지는 모른다. Claude, GPT 계열 등 — "두뇌"
Harness / Runtime 루프 실행, tool 호출 파싱·실행, 상태/메모리 관리, 권한 검사, 종료 조건, 관측성 Claude Agent SDK, OpenAI Agents SDK, 자체 구축 루프
Tool / Environment 실제 부수효과(side effect)가 일어나는 곳 — DB, 파일시스템, 외부 API, 셸 MCP 서버, 함수 호출, 샌드박스 내 코드 실행

이 세 계층을 명확히 구분하지 못하면, "모델이 실수했다"와 "harness가 잘못 설계됐다"를 구분하지 못한다. 예를 들어 모델이 존재하지 않는 함수를 호출하려고 시도하는 것은 모델의 추론 문제일 수 있지만, 그 호출이 검증 없이 그대로 실행되어 프로덕션 DB를 삭제했다면, 이는 명백히 harness(권한 검사 부재)의 책임이다. Senior 개발자는 장애가 났을 때 이 경계를 즉시 구분해서 원인을 좁힐 수 있어야 한다.

대표 프레임워크 두 개의 아키텍처 요약

"harness를 직접 만들 것인가, 기존 프레임워크를 쓸 것인가"를 판단하려면 대표적인 두 프레임워크가 각각 harness의 책임을 어떻게 나눠서 구현했는지 알아야 한다.

Claude Agent SDK: Claude Code CLI를 라이브러리처럼 감싼 형태다. SDK 자체는 별도 텔레메트리를 내보내지 않고, Claude Code CLI를 서브프로세스로 실행해서 그 안의 agent loop, 권한 모드(permission mode), hooks, 서브에이전트(Task tool을 통한 병렬 위임), OS 레벨 샌드박스(Linux bubblewrap / macOS Seatbelt)를 그대로 재사용한다. "코딩 에이전트/장시간 자율 작업"에 강점이 있는 구조다.
OpenAI Agents SDK: Python/TypeScript로 된 경량 오케스트레이션 프레임워크로, Agent와 Runner가 핵심 primitive다. Handoff(다른 agent에게 대화 제어권을 완전히 이전하는 기능)를 모델에게는 transfer_to_<agent_name>이라는 tool처럼 노출한다. Guardrail은 tripwire 방식으로 위반 시 예외를 던져 즉시 중단시키고, Session이 대화 메모리 계층을 표준화한다. Assistants API를 대체하는 표준 진입점이며, Swarm(2024년 실험적 버전)의 정식 후속작이다.
비교 항목 Claude Agent SDK OpenAI Agents SDK
기반 구조 Claude Code CLI를 서브프로세스로 감쌈 순수 Python/TS 라이브러리, provider 다중 지원
제어권 이전 서브에이전트(Task tool) — 위임 후 결과 회수 Handoff(완전 이전) / Agents-as-tools(결과 회수)를 모두 지원
안전장치 권한 모드, hooks, OS 레벨 샌드박스(bubblewrap/Seatbelt) Guardrail(tripwire), max_turns 기본값(5)
관측성 내장 OpenTelemetry 계측(옵트인) 기본 활성화된 자체 Tracing 대시보드 + OTel 브릿지
강점 영역 코딩/장시간 자율 작업, 파일시스템 조작 멀티 에이전트 라우팅, 고객지원형 워크플로우
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "SDK 사용법을 아느냐"가 아니라 "모델을 감싸는 런타임을 하나의 시스템으로 설계할 수 있느냐"를 본다. 2026년 기준 이 분야는 용어와 표준 프레임워크가 막 자리잡는 중이라, "튜토리얼을 따라해봤다" 수준과 "왜 이렇게 설계해야 하는지 원리로 설명할 수 있다" 수준의 차이가 특히 크게 드러난다.

  • Agent와 단순 tool calling을 구분하는가: 반복 루프, 상태 관리, 종료 조건이라는 본질을 이해하고 있는가
  • 계층을 구분해서 사고하는가: 모델의 책임과 harness의 책임을 명확히 나눠서 장애 원인을 추론할 수 있는가
  • Build vs Framework 판단 기준이 있는가: "무조건 프레임워크" 또는 "무조건 직접 구현"이 아니라, 상황에 따라 근거 있는 선택을 하는가
  • 운영 관점을 갖고 있는가: 루프가 무한히 돌거나 비용이 폭주하는 실패 모드를 설계 단계에서부터 고려하는가
Senior 개발자는 "Claude Agent SDK를 써봤다"가 아니라, "이 harness가 왜 이렇게 설계되어 있고, 우리 시스템에 그대로 맞을지, 어디를 커스터마이징해야 할지"를 판단할 수 있어야 한다. 실제 프로덕션 인시던트(11일간 도는 $47K짜리 무한루프 등)가 반복적으로 보고되는 만큼, 면접관은 이 위험을 실감하고 대비할 수 있는 사람을 찾는다.
실무 연결

실제 상황과 연결하기

상황 1: 사내 코딩 어시스턴트 / 리포지토리 자동화

파일을 읽고, 수정하고, 테스트를 돌리고, 결과를 검증하는 장시간 자율 작업이 핵심이라면 바퀴를 새로 발명할 이유가 없다. → Claude Agent SDK처럼 이미 서브에이전트, 권한 모드, 샌드박스가 검증된 harness를 채택하는 것이 빠르고 안전하다.

상황 2: 여러 LLM provider를 넘나드는 멀티테넌트 SaaS

고객사별로 Anthropic/OpenAI/오픈소스 모델을 선택할 수 있어야 한다면, 특정 벤더의 SDK에 강하게 결합된 harness는 이식성이 떨어진다. → tool 스키마와 메시지 포맷을 자체적으로 추상화하는 provider abstraction layer를 직접 설계해야 한다(Day 2에서 상세히 다룬다).

상황 3: 자체 제작 tool-loop가 장애를 일으키는 경우

실제 사례로, 4개의 LangChain agent가 "서로에게 확인을 요청"하며 11일간 264시간 동안 루프를 돌아 약 $47,000의 비용이 발생한 프로덕션 인시던트가 보고된 바 있다. 원인은 명확한 step cap과 progress detection(반복 상태 감지)이 없었기 때문이다. → 이런 사고는 "모델이 똑똑하지 못해서"가 아니라 harness의 안전장치 부재 때문에 일어난다. Vercel의 사례처럼 harness를 오히려 단순화(tool 개수를 80% 줄이고 파일시스템 기반 접근으로 전환)해서 성공률을 80%에서 100%로, 응답 시간을 3.5배 개선한 반례도 있다 — "harness가 복잡할수록 안전한 것이 아니라, 모델이 판단하기 쉬운 harness가 더 안전하다."

상황 4: 금융/의료 등 규제 산업의 승인 워크플로우

환불 처리, 계좌 이체처럼 되돌리기 어려운 작업은 harness 차원에서 tool 실행 전 승인 게이트가 필수다. → OpenAI Agents SDK의 guardrail이나 Claude Agent SDK의 권한 모드(permission mode)로 "이 작업은 사람 승인 없이는 절대 실행되지 않는다"는 것을 harness 레벨에서 강제한다. 프롬프트로 "신중하게 확인하세요"라고 적어두는 것은 강제력이 없다 — 승인 게이트는 인프라 레벨에서 강제해야 의미가 있다는 것이 2026년 기준 업계 공통 원칙이다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"Agent는 모델 혼자가 아니라 model과 harness의 조합입니다. 모델은 주어진 맥락에서 다음 행동을 추론할 뿐이고, 그 행동을 실제로 실행하고, 결과를 다시 모델에게 보여주고, 언제 멈출지 판단하는 것은 harness의 책임입니다. Claude Agent SDK 공식 문서가 정의하는 것처럼 이 과정은 'gather context → take action → verify → repeat'라는 반복 루프이고, 저는 이 3단계 각각에서 무엇을 검증하고 언제 강제로 종료시킬지를 harness 설계의 핵심으로 봅니다.

실무에서 프레임워크를 쓸지 직접 만들지는 두 가지 기준으로 판단합니다. 첫째, 우리가 필요한 워크플로우가 표준적인 패턴(코딩 자동화, 문서 기반 QA)인지, 아니면 멀티 provider 지원이나 도메인 특화 승인 흐름처럼 프레임워크가 가정하지 않은 요구사항인지를 봅니다. 표준적이면 Claude Agent SDK나 OpenAI Agents SDK처럼 이미 권한 모드, 샌드박스, guardrail이 검증된 프레임워크를 채택하는 게 더 빠르고 안전합니다. 둘째, 운영 안정성 관점에서 무한루프나 비용 폭주를 막을 안전장치 — step cap, 예산 cap, 반복 상태 감지 — 가 기본 제공되는지를 봅니다. 실제로 명확한 종료 조건 없이 만들어진 agent가 11일간 돌면서 5만 달러 가까운 비용을 발생시킨 프로덕션 사고 사례도 있어서, 이건 선택이 아니라 필수라고 생각합니다.

그리고 harness가 복잡하다고 무조건 안전한 게 아니라는 점도 중요합니다. Vercel의 사례처럼 오히려 tool 개수를 줄이고 모델이 판단하기 쉬운 구조로 harness를 단순화했더니 성공률과 응답 속도가 모두 개선된 경우도 있습니다. 그래서 저는 harness를 설계할 때 '얼마나 많은 기능을 넣을까'가 아니라 '모델이 안전하고 예측 가능하게 행동할 수 있는 최소한의 구조가 무엇인가'를 먼저 묻습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Claude Agent SDK와 OpenAI Agents SDK의 아키텍처 차이를 더 자세히 설명해주세요."

약한 답변: "둘 다 agent 만드는 SDK라서 비슷합니다."

강한 답변: "가장 큰 차이는 '무엇을 감싸고 있는가'입니다. Claude Agent SDK는 Claude Code CLI를 서브프로세스로 실행하는 구조라서, CLI 안에 이미 구현된 agent loop, 서브에이전트(Task tool), 권한 모드, OS 레벨 샌드박스를 그대로 물려받습니다. 코딩·파일 조작 같은 장시간 자율 작업에 강합니다. 반면 OpenAI Agents SDK는 순수 라이브러리로, Handoff라는 개념으로 '다른 agent에게 대화를 완전히 넘기는 것'과 '결과만 받아오는 agents-as-tools'를 구분해서 제공합니다. 고객 지원처럼 역할별로 라우팅되는 워크플로우에 강점이 있습니다. 관측성도 다른데, Claude Agent SDK는 OTel 계측이 옵트인이고, OpenAI Agents SDK는 자체 Tracing 대시보드가 기본 활성화되어 있습니다."

F2. "무한 루프나 비용 폭주를 어떻게 방지하시겠어요?"

약한 답변: "타임아웃을 걸어두면 됩니다."

강한 답변: "타임아웃 하나로는 부족합니다. 최소 5겹의 방어선을 둡니다. 첫째, hard step cap — 최대 스텝 수를 강제로 제한합니다. 둘째, 토큰/달러 예산 cap — 스텝 수가 아니라 실제 비용 기준으로도 제한합니다. 셋째, 반복 상태 감지 — 같은 tool을 같은 인자로 두 번 호출하면 진행이 안 되고 있다고 보고 중단시킵니다. 넷째, 에러가 반복되는 tool에 circuit breaker를 겁니다. 다섯째, task당 비용의 p95를 모니터링해서 평소보다 튀는 패턴을 알림으로 잡습니다. OpenAI Agents SDK의 max_turns 기본값이 5인 것도 이런 사고를 방지하려는 설계인데, 값을 None으로 풀어버리는 게 흔한 실수 포인트라는 것도 알고 있습니다."

F3. "이미 자체 제작한 tool-loop가 있는 팀에게 프레임워크로 마이그레이션을 제안한다면, 어떻게 설득하시겠어요?"

강한 답변: "무작정 새 프레임워크로 갈아타자고 하지 않습니다. 먼저 현재 harness에서 실제로 겪고 있는 문제 — 권한 검사 누락, 관측성 부재, 무한루프 방지 로직 없음 — 를 구체적으로 나열하고, 그 문제들이 프레임워크가 이미 검증된 방식으로 해결한 문제인지 확인합니다. 겹치는 부분이 많으면 트래픽이 적은 기능부터 점진적으로 옮기며 리스크를 통제하고, 프레임워크가 가정하지 않는 도메인 특화 요구(예: 특정 규제 승인 흐름)가 있다면 그 부분만 자체 로직으로 남겨둡니다. '전부 새로 만든다'가 아니라 '검증된 부분은 재사용하고, 우리만의 요구사항에만 엔지니어링 리소스를 쓴다'는 게 목표입니다."

F4. "Multi-agent 구조가 항상 더 좋은 선택인가요?"

강한 답변: "아닙니다. Anthropic의 멀티 에이전트 리서치 시스템 사례를 보면, lead agent와 여러 subagent로 구성된 구조가 단일 에이전트 대비 성능은 90.2% 향상됐지만, 토큰 비용은 약 15배 더 들었습니다. 이건 독립적으로 병렬화할 수 있는 'breadth-first' 작업(여러 방향의 리서치를 동시에 탐색하는 것 같은)에는 정당화되지만, 순차적으로 강하게 의존적인 작업이나 지연시간·비용에 민감한 작업에는 오히려 손해입니다. 그래서 저는 '작업을 독립적인 하위 작업으로 병렬 분해할 수 있는가'와 '그 병렬화로 얻는 가치가 15배에 가까운 비용 증가를 정당화하는가'를 먼저 따져보고 multi-agent 여부를 결정합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Agent를 Model + Harness의 조합으로 정의하고 각각의 책임을 나눈다
  • Agent loop를 gather-act-verify 구조로 설명하고, 각 단계의 안전장치를 언급한다
  • 프레임워크 선택에 명확한 판단 기준(표준 워크플로우 여부, 운영 안정성 요구)이 있다
  • 실제 프로덕션 장애 사례(비용 폭주, 무한루프)를 근거로 설계 원칙을 설명한다
  • "복잡한 harness = 안전한 harness"라는 통념을 반박할 수 있다

약한 답변의 특징

  • "Agent는 tool을 쓸 수 있는 LLM이다"처럼 루프/종료조건 개념이 빠진 정의
  • 특정 SDK의 함수 이름만 나열하고 왜 그렇게 설계됐는지 설명하지 못함
  • "프레임워크가 다 알아서 해준다"는 식으로 운영 리스크를 인지하지 못함
  • Multi-agent나 복잡한 harness를 무조건 더 나은 선택으로 전제
  • 모델 실패와 harness 설계 결함을 구분하지 못하고 뭉뚱그려 설명
핵심 차이: 강한 답변은 "이 SDK를 써봤다"가 아니라 "harness가 왜 필요하고, 무엇을 책임져야 하며, 그 책임을 다하지 못하면 실제로 어떤 사고가 나는지"를 구체적인 사례와 함께 설명한다. 면접관은 API 사용 경험보다 시스템을 신뢰성 있게 운영할 수 있는 사고방식을 확인한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Agent는 그냥 tool을 호출할 수 있는 LLM이다"

사실: tool을 한 번 호출하는 것은 agent라기보다 단순 function calling에 가깝다. Agent는 모델이 스스로 여러 단계를 반복하며 판단하는 loop 구조와, 그 loop를 안전하게 관리하는 harness가 있어야 성립한다.

오해 2: "검증된 프레임워크를 쓰면 harness를 신경 쓰지 않아도 된다"

사실: 프레임워크는 harness의 기본 골격을 제공할 뿐이다. 권한 스코핑, 도메인 특화 승인 흐름, 비용 모니터링 임계치처럼 우리 서비스에 특화된 안전장치는 프레임워크 위에 추가로 설계해야 한다. "프레임워크 = harness 전체"라는 생각은 위험하다.

오해 3: "Harness는 부수적인 배관 코드일 뿐이라 중요하지 않다"

사실: "Harness Engineering"이라는 용어가 정립된 이유가 바로 이 통념을 반박하기 위해서다. 실제 프로덕션 사고(무한루프로 인한 수만 달러 비용 폭주, 프로덕션 DB 삭제 등)의 원인은 대부분 모델의 추론 실패가 아니라 harness의 안전장치 부재였다. Fowler의 표현대로 "LLM은 agent 시스템에서 가장 작은 부분"이다.

오해 4: "복잡하고 기능이 많은 harness일수록 더 안전하고 강력하다"

사실: Vercel의 사례처럼, tool 수를 줄이고 harness를 단순화했더니 오히려 성공률과 응답 속도가 크게 개선된 경우가 있다. 모델이 선택해야 하는 옵션이 많을수록 잘못된 선택의 여지도 커진다. "얼마나 많은 기능을 제공하는가"가 아니라 "모델이 얼마나 예측 가능하게 행동할 수 있는가"가 좋은 harness의 기준이다.

자기 점검 Rubric

내 답변 수준 확인하기

미흡 "Agent는 tool을 쓸 수 있는 AI"라고만 설명하고, 반복 루프나 harness 개념을 언급하지 못함
주니어 특정 SDK(Claude Agent SDK 또는 OpenAI Agents SDK)로 tool을 붙인 agent를 만들어본 경험이 있고, 기본 사용법을 설명할 수 있음
중급 "Agent = Model + Harness" 개념과 agent loop(gather-act-verify) 구조를 설명할 수 있음. 두 SDK의 기본적인 차이를 말할 수 있음
시니어 Model/Tool/Runtime 계층 구분, harness engineering의 의미, build vs framework 판단 기준, 실제 프로덕션 장애 사례(비용 폭주, 도구 과다로 인한 성능 저하)를 근거로 설계 원칙까지 하나의 흐름으로 자연스럽게 연결해서 설명함. 특정 SDK 문법이 아니라 원리를 기준으로 답변함
자가 점검 질문:
1. "Agent = Model + Harness"가 무슨 뜻인지, harness가 구체적으로 어떤 책임을 지는지 예시를 들어 설명할 수 있는가?
2. Agent loop의 3단계(gather-act-verify)를 즉시 말하고, 각 단계에서 무엇이 잘못될 수 있는지 설명할 수 있는가?
3. Model/Tool/Runtime 중 어느 계층의 문제인지 장애 상황을 듣고 즉시 구분할 수 있는가?
4. "harness가 복잡할수록 안전하다"는 통념이 왜 틀렸는지, 반례를 들어 설명할 수 있는가?
5. Build vs Framework를 판단할 때 나만의 명확한 기준을 3초 안에 말할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • Agent = Model + Harness. 모델 혼자서는 agent가 아니다. 반복 루프를 실행·검증·종료하는 harness가 필요하다.
  • Agent loop는 gather context → take action → verify → repeat의 반복이다. 각 단계에 안전장치가 있어야 production-grade harness가 된다.
  • Model / Tool / Runtime 세 계층을 구분해서 사고하면 장애 원인을 빠르게 좁힐 수 있다.
  • Harness Engineering은 2026년에 정립된 용어로, "LLM은 agent 시스템에서 가장 작은 부분"이라는 원칙을 담고 있다.
  • Build vs Framework는 표준 워크플로우 여부, provider 요구, 운영 안정성 요구를 기준으로 판단하며, 두 선택은 배타적이지 않다.
  • 복잡한 harness가 항상 안전한 것은 아니다. 모델이 예측 가능하게 행동할 수 있는 최소한의 구조를 우선한다.
