Day 2 · AI Harness / Agent Harness Interview

Model/Tool/Runtime 경계와 Provider 추상화

Tool calling 스키마의 "공통 핵심"과 "새는 추상화(leaky abstraction)"를 구분해서 설명하는 능력
버전/시점 기준: 이 레슨은 2026-07 기준 Anthropic Messages API의 tool_use/input_schema, OpenAI Responses API의 tools(Assistants API는 2026-08-26 완전 종료), 그리고 MCP 2026-07-28 Release Candidate(스키마가 전체 JSON Schema 2020-12로 확장되고, Sampling/Roots/Logging이 SEP-2577에 따라 12개월 유예 후 폐지 예정이며, Tasks가 독립 extension으로 분리되는 버전)를 기준으로 작성됐다. Vercel AI SDK, LiteLLM, OpenRouter 같은 provider 추상화 레이어 사례도 함께 다룬다. 수치와 스펙 세부사항은 공식 문서 접근 제약으로 검색 결과와 2차 출처로 교차검증한 것이 섞여 있으므로, 실제 면접 전 공식 changelog로 재확인해야 한다.
면접 질문
Q1 — 주력 질문 "Tool calling 스키마가 Anthropic, OpenAI, MCP에서 어떻게 다른지, 그리고 이 위에 provider-agnostic한 harness를 만들려면 무엇을 추상화해야 하는지 설명해주세요."
Q2 — 변형 "여러 LLM provider를 넘나드는 tool-calling 추상화 레이어를 설계할 때, 완벽하게 숨길 수 없는 부분(leaky abstraction)은 무엇인가요?"
Q3 — 변형 "Tool을 설계할 때 어떤 원칙을 따르시나요? Tool은 많을수록 agent가 더 똑똑해지나요?"
Q4 — 변형 "Tool 실행이 실패했을 때, harness는 이 실패를 모델에게 어떻게 전달해야 하나요?"
전제 개념

Tool 정의의 "공통 핵심"과 그 위에 얹힌 provider별 차이

세 진영이 tool을 정의하는 방식

Day 1에서 harness가 "모델의 tool 호출 요청을 파싱해서 실제로 실행한다"고 했다. 그런데 이 "tool 호출 요청"의 형태(schema) 자체가 provider마다 다르다. 세 진영을 비교해보면 공통점과 차이점이 명확히 드러난다.

Provider Tool 정의 필드 선택 제어(tool_choice)
Anthropic (Messages API) name, description, input_schema(JSON Schema, type: object) auto / any / tool / none + disable_parallel_tool_use 플래그
OpenAI (Responses API) {"type": "function", "function": {name, description, parameters}} 봉투 구조 tool_choice + 별도의 parallel_tool_calls 불리언, strict: true 모드
MCP (2026-07-28 RC) name, title(옵션), description, inputSchema, outputSchema(옵션) — 전체 JSON Schema 2020-12 없음 — MCP는 "매 턴 모델 호출"을 다루는 프로토콜이 아니라 리소스/능력 노출 프로토콜이다

공통 핵심(portable core): name + description + "type: object/properties/required로 이뤄진 JSON Schema"라는 뼈대는 세 진영 모두 동일하다. Provider 추상화 레이어가 input_schema ↔ parameters ↔ inputSchema처럼 필드명을 바꾸고, OpenAI의 {"type":"function","function":{...}} 봉투를 씌우거나 벗기는 정도는 기계적으로 처리할 수 있다.

MCP는 "tool calling 표준"이 아니다: MCP는 tool 외에도 resources(구조화된 컨텍스트 데이터 노출), prompts(재사용 가능한 프롬프트 템플릿)를 서버 쪽 primitive로 정의하고, 클라이언트 쪽에는 roots(서버가 접근 가능한 범위 정의), sampling(서버가 거꾸로 클라이언트에게 "LLM 추론을 대신 해달라"고 요청하는 역방향 기능)까지 포함한다. 그래서 "MCP tool을 Anthropic tool_use로 바꾼다"는 말은 "리소스/프롬프트/샘플링까지 포함한 프로토콜 중 tool 부분 하나만 뽑아서 변환한다"는 뜻이지, "표준 스키마를 다른 표준 스키마로 바꾼다"는 말과는 다르다.

2026-07-28 RC — MCP 스키마와 primitive의 변화

2026-07-28에 발행 예정인 MCP RC는 세 가지 큰 변화를 포함한다. 이 트랙 다른 레슨(Day 7)에서 프로토콜 전체를 깊게 다루지만, tool 설계와 직결되는 부분만 먼저 짚는다.

  • Stateless 전환: initialize/initialized 핸드셰이크와 Mcp-Session-Id 헤더가 사라진다. 요청이 자체 완결적(self-contained)이 되어, 세션 고정 (sticky session)이나 공유 세션 스토어 없이도 단순 라운드로빈 로드밸런싱이 가능해진다.
  • 스키마 확장(폐지 아님): inputSchema/outputSchema가 전체 JSON Schema 2020-12로 확장된다. Input schema는 여전히 type: object 루트를 요구하지만 oneOf/anyOf/allOf 조합과 $ref/$defs를 허용하고, output schema는 완전히 자유로워진다. 단, 구현체는 외부 $ref URI를 자동으로 역참조(dereference)하면 안 되고, 스키마 깊이/검증 시간을 제한해야 한다는 보안 가이드가 함께 붙는다 — 이건 externally-controlled schema를 통한 DoS/SSRF 벡터를 막기 위한 조치다.
  • SEP-2577에 따른 폐지(12개월 유예): Sampling은 "provider LLM API 직접 통합"으로, Roots는 "tool 파라미터, 리소스 URI, 서버 설정"으로, Logging은 "stdio 전송에는 stderr, 구조화된 관측성에는 OpenTelemetry"로 대체된다.
  • Tasks — 장시간 실행되는 tool 호출을 위한 독립 extension: 2025-11-25 스펙에서 실험적 core 기능으로 들어왔던 Tasks가, 프로덕션 경험상 재설계가 필요할 만큼 문제가 많아 독립 extension으로 분리된다. 새 방식은 완전히 stateless다 — 서버는 tools/call에 결과 대신 task handle을 돌려줄 수 있고, 클라이언트는 tasks/get(폴링), tasks/update(진행 중 입력 전달), tasks/cancel로 진행 상황을 다룬다. 세션이 없어져서 "누구의 task인지" 안전하게 구분할 방법이 없어졌기 때문에 tasks/list(전체 task 목록 조회)는 아예 제거된다. "3분짜리 검색 인덱스 빌드를 MCP tool 호출 하나에 욱여넣었다가 클라이언트가 타임아웃 나는 문제"를 해결하기 위한 설계다.

Provider abstraction layer가 실제로 해야 하는 일

멀티 provider를 지원하는 harness를 만든다면, 추상화 레이어는 최소한 다섯 가지를 책임져야 한다.

  1. 메시지 포맷: Anthropic의 타입이 있는 content-block 배열(text/tool_use/ tool_result) vs OpenAI의 더 평평한 메시지 + tool_calls 배열 vs Gemini의 parts를 정규화
  2. Tool 스키마 변환: 하나의 canonical(보통 OpenAI 형태) tool 정의를 각 provider의 네이티브 봉투로 변환
  3. 스트리밍 이벤트 포맷: 텍스트 델타, tool 호출 인자 델타, 종료 사유(finish reason)를 하나의 이벤트 스트림 형태로 정규화
  4. 에러/재시도 시맨틱: HTTP/에러 코드를 정규화하고 실패 유형별 재시도 정책을 구성
  5. 토큰 사용량 집계: 입력/출력/캐시 read/write/reasoning 토큰처럼 필드명도 다르고 과금 방식도 다른 사용량 객체를 정규화

실제 사례로 Vercel AI SDK는 "provider 간 차이를 추상화해서 하나의 일관된 API로 스트리밍 텍스트, 구조화된 출력, tool 호출, 멀티스텝 agent를 다룬다"고 설명하며, provider 전환을 "코드 두 줄만 바꾸면 된다"고 표현한다. AI SDK 6는 ToolLoopAgent라는, "LLM 호출 → 요청된 tool 실행 → 결과 추가 → 반복"(기본 20스텝 상한)을 구현한 canonical agent loop까지 SDK 차원에서 제공한다. LiteLLM은 100개 이상의 provider 네이티브 포맷으로 요청을 "번역"하고 응답을 다시 하나의 일관된 형태로 되돌리는 프록시/SDK로, tool_choice의 auto/any/tool/none 전략을 각 provider의 대응 개념으로 매핑하는 것까지 명시적으로 처리한다. OpenRouter는 provider마다 다른 finish_reason 값을 고정된 enum(tool_calls/stop/length/ content_filter/error)으로 정규화하면서도, 추상화가 새는 경우를 위해 native_finish_reason이라는 탈출구를 별도로 노출한다 — 완벽한 추상화는 없다는 것을 제품 설계에 그대로 반영한 사례다.

완벽히 숨길 수 없는 것 — Leaky Abstraction 4가지

추상화 레이어를 아무리 잘 만들어도, 다음 네 가지는 provider 간 차이가 근본적으로 달라서 완전히 숨기기 어렵다. Senior 개발자는 "추상화했다"가 아니라 "무엇이 새는지 알고 그 지점을 명시적으로 설계했다"를 보여줘야 한다.

① Prompt Caching

Anthropic

  • 명시적 opt-in: 특정 content block에 cache_control 마커를 직접 붙여야 함
  • 캐시 쓰기는 기본 입력 단가의 약 +25%, 캐시 읽기는 약 -90% 할인

OpenAI

  • 완전 자동: 1,024 토큰 이상 프롬프트는 코드 변경 없이 자동 캐싱
  • 캐시 쓰기는 무료

Gemini는 Anthropic처럼 수동 마커가 필요하지만 캐시가 활성화되는 최소 토큰 기준이 훨씬 높고(32,768 토큰), read/write 할인이 아니라 저장 공간 기준 과금이다. OpenRouter 문서는 이 새는 지점을 직접 인정한다 — cache_control breakpoint를 여러 개 넣는 게 Anthropic류 provider에서는 그대로 동작하지만, Gemini에서는 마지막 breakpoint 하나만 실제로 적용되고 나머지는 조용히 무시된다. "겉보기엔 통일된 API인데 provider별로 조용히 성능이 저하되는" 전형적인 예시다.

② Reasoning / Thinking 토큰

Anthropic의 extended thinking은 thinking content block으로 개발자에게 그대로 보이고, 표준 출력 토큰 단가로 과금된다. 과거의 budget_tokens 파라미터는 최신 모델에서 effort 파라미터(low/high/xhigh/max)로 옮겨가는 중이고, 최신 Opus 계열은 작업 복잡도에 따라 예산을 자동 조절하는 "adaptive thinking"까지 제공한다. 반면 OpenAI의 reasoning 토큰은 응답에서 완전히 숨겨지고, usage.output_tokens_details.reasoning_tokens로만 노출되며, reasoning.effort(none~xhigh)로 제어한다. "reasoning 노력도"라는 개념 자체는 비슷해 보여도, 하나는 보이고 하나는 숨겨지며, 하나는 토큰 예산이고 하나는 이산적인 등급이라 하나의 범용 노브(knob)로 통합하면 정보 손실이 생긴다.

③ tool_choice / 병렬 호출 시맨틱

Anthropic은 tool_choice 객체 안에 disable_parallel_tool_use를 넣는 구조인 반면, OpenAI는 tool_choice와 별도로 최상위 parallel_tool_calls 불리언을 둔다. 게다가 OpenAI에서는 strict: true 모드와 병렬 tool 호출이 상호 배타적이다 — strict 모드를 쓰려면 parallel_tool_calls: false를 강제해야 하고, 이 조건(모든 객체에 additionalProperties: false, 모든 프로퍼티가 required에 포함)을 만족하지 못하면 요청 자체가 거부된다. Anthropic의 대응 개념인 "tool use를 위한 structured output"에는 이런 상호 배타 제약이 없다. "그냥 tool_choice를 매핑하면 된다"는 생각으로 만든 추상화는 이 경계에서 미묘하게 틀린 동작을 하게 된다.

④ 스트리밍 세밀도(Granularity)

Anthropic의 fine-grained tool streaming(2026-07 기준 전 모델/플랫폼에서 GA, 베타 헤더 불필요)은 tool의 입력 파라미터를 서버 측 버퍼링이나 JSON 검증 없이 그대로 스트리밍한다 — 큰 코드 블록이나 문서를 인자로 넘길 때 첫 조각이 도착하는 시간을 줄여준다. 반면 항상 JSON을 완성한 뒤 검증하고 나서야 이벤트를 내보내는 provider도 있다. 통합 스트리밍 추상화는 이 둘 중 하나를 골라야 한다 — "버퍼링 후 검증"을 기본으로 하면 fine-grained 모드의 이점을 잃고, "그대로 흘려보내기"를 기본으로 하면 호출자가 불완전한 JSON을 받을 수 있다는 걸 알아야 한다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • "공통 핵심"과 "새는 부분"을 구분할 수 있는가: "다 똑같다" 또는 "다 완전히 다르다"라는 극단이 아니라, 어디까지 통합할 수 있고 어디부터 provider별 특이사항을 명시적으로 다뤄야 하는지 아는가
  • Tool 설계를 아키텍처 문제로 다루는가: "tool을 많이 만들면 agent가 더 강력해진다"는 직관을 실제 사례(Vercel d0)로 반박할 수 있는가
  • 에러 처리를 harness의 핵심 책임으로 보는가: tool 실행 실패를 예외로 죽이지 않고 모델이 판단할 수 있는 정보로 되돌려주는 설계를 아는가
  • 최신 변화를 스스로 추적하는가: MCP RC, structured output GA, fine-grained streaming GA처럼 빠르게 바뀌는 스펙을 근거로 판단을 업데이트할 수 있는가
2026년 기준 이 영역은 "표준"이 이제 막 자리 잡는 중이라, 면접관 스스로도 최신 스펙을 다 따라잡지 못했을 수 있다. 그래서 면접관은 후보가 "정답을 외웠는가"보다 "변화가 생겨도 스스로 원리로 재구성할 수 있는가"를 더 중요하게 본다.
실무 연결

실제 상황과 연결하기

상황 1: 멀티 provider SaaS의 tool 스키마 관리

고객사별로 Anthropic/OpenAI/오픈소스 모델을 선택할 수 있어야 한다면, tool 정의를 provider별로 따로 유지하는 건 유지보수 재앙이다. → canonical 스키마(보통 OpenAI 형태의 JSON Schema)를 하나 두고, LiteLLM/Vercel AI SDK 같은 기존 추상화 레이어를 채택하거나 직접 얇은 변환 계층을 만든다. 단, prompt caching과 reasoning 토큰처럼 새는 지점은 provider별 설정으로 명시적으로 노출해야지, 추상화 뒤에 숨기려 하면 안 된다.

상황 2: MCP 서버가 늘어나며 context가 폭주하는 문제

MCP 서버 5개, tool 58개짜리 구성이 대화 시작 전에만 이미 약 5만 5천 토큰을 차지하고, Jira 연동 하나를 더 붙이면 총 오버헤드가 10만 토큰을 넘어설 수 있다는 사례가 보고된 바 있다. Anthropic은 이 문제에 두 가지 대응책을 내놓았다 — Tool Search Tool(개별 tool을 defer_loading: true로 표시해서 세션 시작 시 전체 로드 대신 필요할 때 검색으로 찾게 함)과 Programmatic Tool Calling(모델이 tool을 하나씩 호출하며 매 결과를 컨텍스트에 흘려보내는 대신, 여러 tool을 호출하고 출력을 처리하는 코드를 직접 작성해서 무엇을 컨텍스트에 넣을지 스스로 통제하게 함 — 수천 행짜리 스프레드시트를 컨텍스트 폭주 없이 다루는 "Claude for Excel"이 이 방식을 쓴다). → MCP tool이 늘어날수록 harness가 tool 노출 방식 자체를 설계해야 한다.

상황 3: "tool을 더 추가하면 성능이 좋아지겠지"라는 직관이 틀리는 경우

Vercel의 텍스트-to-SQL agent인 d0는 원래 스키마 조회, 쿼리 검증, 에러 복구, 의도 명확화, 조인 경로 탐색기, 구문 검증기, 분석 계획기, 결과 포맷터 등 16개의 전문화된 tool로 구성돼 있었다. 팀은 agent 자체를 개선하는 것보다 tool 스캐폴딩을 유지보수하는 데 더 많은 시간을 쓰고 있다는 걸 깨닫고, tool의 80%를 제거해서 사실상 샌드박스 안에서 임의의 셸 명령을 실행하는 범용 tool 하나(+파일시스템 직접 접근)로 재구성했다. 모델은 grep, cat, find, ls 같은 표준 유닉스 도구로 사람 분석가처럼 시맨틱 레이어 파일을 탐색하고, 스스로 멘탈 모델을 구축한 뒤 SQL을 작성한다. 결과: 성공률 80%→100%, 응답 시간 274초→77초(3.5배), 토큰 40% 감소, 스텝 수 40% 감소. Vercel의 결론: "모델은 점점 똑똑해지고 컨텍스트 윈도우는 점점 커지므로, 최선의 agent 아키텍처는 아키텍처가 거의 없는 것일 수 있다 — 우리가 선택을 대신 멈추면 모델이 더 나은 선택을 한다." → tool 설계의 목표는 "많은 기능"이 아니라 "모델이 예측 가능하게 판단할 수 있는 최소 구조"다(Day 1의 harness 원칙과 정확히 같은 결론).

상황 4: Tool 실행이 실패했을 때 harness가 예외를 던져 루프가 죽는 경우

Anthropic API의 tool_result content block은 is_error 불리언 필드를 지원한다. 즉, tool 실행이 실패해도 harness가 예외를 던지고 agent loop를 중단시키는 게 아니라, 실패 자체를 정상적인 대화 턴으로 모델에게 돌려줘서 모델이 재시도하거나, 다른 방법을 시도하거나, 사용자에게 보고할 수 있게 해야 한다. 그런데 실제로 이 필드를 제대로 채워 넣지 않고 에러를 그냥 평문 텍스트로 직렬화해서 보내는 프레임워크 구현 사례가 보고된 바 있다 — "표준 필드가 있어도 추상화 레이어가 제대로 연결하지 않으면 새어나간다"는 걸 보여주는 사례다. → harness를 만들 때 tool 에러 경로를 "예외 처리"가 아니라 "tool 실행의 정상적인 한 결과"로 설계해야 한다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

5
운영/추적 관점 마무리하기 (Senior 포인트) "이 영역은 스펙이 계속 바뀌기 때문에, 특정 시점의 세부 규칙을 외우기보다 원리(공통 핵심 vs 새는 지점을 구분하는 사고방식)를 유지하고 최신 changelog로 재확인하는 습관이 더 중요하다고 생각합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Tool 정의는 Anthropic, OpenAI, MCP 모두 name, description, 그리고 type: object 기반 JSON Schema라는 공통 뼈대를 갖고 있습니다. 그래서 추상화 레이어는 필드명을 바꾸고 OpenAI의 function 봉투 구조를 씌우거나 벗기는 정도는 기계적으로 처리할 수 있습니다. 하지만 그 위에 얹힌 provider별 동작은 완전히 숨길 수 없습니다. 예를 들어 prompt caching은 Anthropic이 명시적 cache_control 마커가 필요한 반면 OpenAI는 완전 자동이고, OpenRouter 문서를 보면 여러 개의 cache breakpoint를 넣어도 Gemini에서는 마지막 하나만 적용되고 나머지는 조용히 무시됩니다. tool_choice도 Anthropic은 disable_parallel_tool_use 플래그 하나로 처리하지만 OpenAI는 별도의 parallel_tool_calls 불리언이 있고, strict 모드와는 아예 상호 배타적이라 조건을 안 맞추면 요청 자체가 거부됩니다. 그래서 저는 추상화 레이어를 만들 때 '모든 걸 하나의 API로 완전히 숨긴다'는 목표 대신, '기계적으로 변환 가능한 공통 핵심'과 'provider별 설정으로 명시적으로 노출해야 하는 새는 지점'을 구분해서 설계합니다.

Tool 개수 자체도 아키텍처 결정이라고 봅니다. Vercel의 텍스트-to-SQL agent 사례가 좋은 반례인데, 16개의 전문화된 tool을 유지보수하는 데 시간을 쏟다가, 오히려 tool의 80%를 제거하고 샌드박스 셸 실행 하나로 재구성했더니 성공률이 80%에서 100%로, 응답 시간이 3.5배 개선됐습니다. 그래서 저는 tool을 설계할 때 '얼마나 많은 기능을 주는가'가 아니라 '모델이 예측 가능하게 판단할 수 있는 최소 구조가 무엇인가'를 먼저 묻습니다.

마지막으로 tool 실행이 실패했을 때는, harness가 예외를 던져서 루프를 죽이는 게 아니라 Anthropic의 is_error 필드처럼 실패 자체를 정상적인 tool_result로 모델에게 돌려줘야 한다고 생각합니다. 실제로 이 필드를 제대로 채우지 않고 에러를 평문으로 직렬화해버린 프레임워크 사례도 있었는데, 이건 '표준이 있어도 추상화 레이어가 제대로 연결하지 않으면 새어나간다'는 걸 보여줍니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "OpenAI strict 모드와 병렬 tool 호출이 왜 상호 배타적인지, harness는 이걸 어떻게 다뤄야 하나요?"

약한 답변: "그냥 둘 중 하나만 쓰면 됩니다."

강한 답변: "strict 모드는 모델이 스키마와 토큰 단위로 정확히 일치하는 인자를 생성하도록 제약 디코딩을 거는 방식이라, 여러 tool 호출을 동시에 생성하면서 그 제약을 모두 만족시키는 게 구조적으로 어렵습니다. 그래서 OpenAI는 strict:true를 쓰려면 parallel_tool_calls:false를 강제합니다. harness 입장에서는 이걸 provider별 capability 플래그로 다뤄야 합니다 — '이 provider·모드 조합에서는 병렬 호출을 기대하지 말라'는 걸 상위 로직에 명시적으로 알려주고, Anthropic처럼 이 제약이 없는 provider로 전환됐을 때는 병렬 호출을 다시 활성화하는 조건 분기를 둡니다. 이걸 추상화 뒤에 숨기면, provider를 바꿨을 때 원인을 알 수 없는 성능 저하로만 보이게 됩니다."

F2. "MCP tool이 수십, 수백 개로 늘어나면 harness는 어떻게 대응해야 하나요?"

강한 답변: "tool 정의 자체가 컨텍스트 예산을 먹는다는 걸 먼저 인식해야 합니다. 5개 MCP 서버, 58개 tool 구성이 대화 시작 전에 이미 5만 토큰 넘게 차지한다는 사례처럼, tool이 늘어날수록 '이 tool을 세션 시작 시 전부 로드할 것인가, 필요할 때 검색으로 찾게 할 것인가'를 설계해야 합니다. Anthropic의 Tool Search Tool처럼 defer_loading 플래그로 지연 로딩하거나, Programmatic Tool Calling처럼 모델이 여러 tool 호출을 코드로 묶어서 실행하고 중간 결과를 직접 필터링해 컨텍스트에 넣을 것만 고르게 하는 방식이 있습니다. 그리고 Anthropic이 강조하는 tool 설계 원칙 — 관련 있는 여러 API 호출을 하나의 스마트한 tool로 통합하고, 이름에 도메인 네임스페이스를 붙이고, 응답 형식을 concise/detailed로 선택 가능하게 만드는 것도 tool 수 자체를 줄이는 방향으로 작동합니다."

F3. "Provider 추상화 레이어를 만들 때 가장 위험한 leaky 지점은 어디라고 생각하나요?"

강한 답변: "저는 prompt caching이 가장 위험하다고 봅니다. 다른 새는 지점(tool_choice, 스트리밍)은 최소한 기능이 안 되거나 요청이 거부되는 식으로 명확하게 실패하지만, prompt caching은 조용히 성능만 저하됩니다. OpenRouter 문서가 인정하듯 여러 cache breakpoint를 넣어도 Gemini에서는 마지막 하나만 적용되고 나머지는 그냥 무시되는데, 에러도 나지 않고 API 응답도 정상이라서 모니터링에서 바로 드러나지 않습니다. 이런 지점은 추상화 레이어가 provider별로 실제 캐시 히트율을 관측성 지표로 노출해야만 발견할 수 있다고 생각합니다."

F4. "Tool 에러를 모델에게 그대로 돌려주면 정보 노출이나 프롬프트 인젝션 위험은 없나요?"

강한 답변: "있습니다. is_error로 구조화해서 돌려주는 것과, 그 안에 무엇을 넣을지는 별개의 문제입니다. 저는 원본 스택 트레이스, 환경 변수, 내부 파일 경로, 자격 증명을 그대로 문자열로 넣지 않고, '무엇이 잘못됐고 어떻게 하면 되는지'를 담은 짧고 실행 가능한 메시지로 가공합니다. 예를 들어 단순히 403이라고 돌려주는 대신 '이 대화방을 수정할 권한이 없습니다. 소유자에게 권한을 요청하거나 다른 thread_id를 쓰세요'처럼요. 또한 tool이 외부 데이터(예: 공격자가 영향을 줄 수 있는 HTTP 응답)를 다루는 경우, 그 tool의 에러 메시지도 성공 응답과 똑같이 신뢰할 수 없는 입력으로 취급해서 모델에게 그대로 흘려보내지 않고 sanitize해야 한다고 봅니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 공통 핵심과 새는 지점을 구체적인 필드/사례로 구분해서 설명한다
  • MCP를 "tool calling 표준"이 아니라 resources/prompts/sampling까지 포함한 프로토콜로 정확히 이해한다
  • Tool 개수를 아키텍처 트레이드오프로 다루고, 실제 사례(Vercel d0)로 "많을수록 좋다"는 통념을 반박한다
  • Tool 에러 처리를 harness의 핵심 책임으로 보고, 정보 노출/인젝션 위험까지 함께 고려한다
  • 스펙이 계속 바뀐다는 것을 인지하고, 원리 중심 사고를 강조한다

약한 답변의 특징

  • "JSON 스키마니까 다 똑같다"처럼 새는 지점을 인지하지 못함
  • MCP를 단순히 "또 다른 tool calling 방식"으로만 이해
  • "tool이 많을수록 agent가 강력해진다"는 검증되지 않은 직관을 그대로 전제
  • Tool 에러를 예외 처리 문제로만 다루고 모델에게 어떻게 돌려줄지는 생각하지 않음
  • 특정 시점의 파라미터 이름을 암기하지만 왜 그렇게 설계됐는지는 설명 못함
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "OpenAI 함수 호출 스키마를 그대로 Anthropic에 넘기면 된다"

사실: 파라미터 스키마 자체(JSON Schema)는 호환되지만, 봉투 구조({"type":"function", "function":{...}} vs {name, description, input_schema})와 tool_choice의 의미가 다르다. 특히 OpenAI의 strict 모드 제약(모든 프로퍼티가 required, additionalProperties:false)을 만족하지 못하는 스키마를 그대로 넘기면 요청이 거부된다.

오해 2: "Provider 추상화 레이어를 만들면 100% 이식 가능하다"

사실: prompt caching, reasoning 토큰, tool_choice/병렬 호출 상호작용, 스트리밍 세밀도는 provider의 근본적인 구현 모델 자체가 달라서 완전히 숨길 수 없다. OpenRouter조차 native_finish_reason 같은 탈출구를 남겨둔다 — 추상화는 "완전히 숨긴다"가 아니라 "무엇이 새는지 명시적으로 드러낸다"가 목표여야 한다.

오해 3: "Tool은 많을수록 agent가 더 똑똑해진다"

사실: Vercel의 d0 사례처럼, tool의 80%를 제거하고 범용 셸 실행 + 파일시스템 접근으로 단순화했더니 성공률과 속도가 모두 개선됐다. 모델이 선택해야 하는 tool이 많을수록 잘못된 선택의 여지도 커지고, tool 정의 자체가 컨텍스트 예산을 소모한다.

오해 4: "Tool 실행이 실패하면 예외를 던지면 된다"

사실: 예외를 던져서 agent loop 자체를 중단시키면 모델은 무엇이 잘못됐는지 알 방법이 없다. Anthropic의 is_error 필드처럼, 실패를 정상적인 tool_result 턴으로 되돌려줘서 모델이 재시도·대안 시도·보고를 스스로 판단하게 해야 한다. 단, 그 안에 원본 스택 트레이스나 민감 정보를 그대로 넣지 않는 것도 함께 지켜야 한다.

핵심 요약

오늘 반드시 기억할 것

  • Tool 정의의 공통 핵심은 name + description + JSON Schema 기반 파라미터다. 추상화 레이어는 이 부분을 기계적으로 변환할 수 있다.
  • 완전히 숨길 수 없는 4가지 — prompt caching, reasoning 토큰, tool_choice/병렬 호출 시맨틱, 스트리밍 세밀도 — 는 provider별로 명시적으로 설계해야 한다.
  • MCP는 tool calling 표준이 아니라 resources/prompts/sampling/roots까지 포함한 프로토콜이며, 2026-07-28 RC에서 stateless 전환, 스키마 확장(JSON Schema 2020-12), Sampling/Roots/Logging 폐지, Tasks의 독립 extension화가 진행 중이다.
  • Tool은 많을수록 좋은 게 아니다. Vercel d0 사례처럼 통합·단순화가 오히려 성공률과 속도를 개선할 수 있다.
  • Tool 에러는 예외가 아니라 정상적인 tool_result로 모델에게 돌려주되, 원본 스택 트레이스나 민감 정보는 제거하고 실행 가능한 메시지로 가공해야 한다.
