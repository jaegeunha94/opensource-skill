Day 8 · AI Harness / Agent Harness Interview

Handoff와 Multi-Agent Coordination

언제 하나의 agent를 여러 개로 쪼개는 게 정말 이득인가 — orchestrator-worker, handoff vs agents-as-tools(manager), 15x 토큰 비용 구조, 그리고 "쓰지 말아야 할 때"에 대한 업계 입장 변화
버전/시점 기준: 이 레슨은 2026-07-10 기준으로 작성됐다. Anthropic의 multi-agent-research-system 엔지니어링 블로그(90.2% 성능 향상, 15x 토큰 비용 수치의 원출처)와 Cognition의 두 블로그 포스트("Don't Build Multi-Agents"와 후속 "Multi-Agents: What's Actually Working")는 자동화 접근이 막혀 검색 스니펫과 다수의 독립 2차 출처로 교차검증했다 — 핵심 수치와 주장은 2개 이상 출처에서 일치했지만, 정확한 원문 인용이 필요하면 실제 브라우저로 재확인해야 한다. OpenAI Agents SDK의 handoff/agents-as-tools API와 Claude Agent SDK의 subagent 격리 원칙은 공식 문서 스니펫과 SDK changelog로 교차검증됐다.
면접 질문
Q1 — 주력 질문 "Multi-agent 시스템을 왜, 그리고 언제 쓰나요? 그냥 하나의 agent에게 도구를 많이 주면 안 되나요?"
Q2 — 변형 "Handoff와 'agent를 도구로 쓰는(agents-as-tools)' 패턴의 차이가 뭔가요? 각각 언제 선택하나요?"
Q3 — 변형 "Multi-agent 시스템의 토큰 비용은 어느 정도이고, 그 비용을 정당화하려면 어떤 조건이 필요한가요?"
Q4 — 변형 "Multi-agent 아키텍처를 쓰면 안 되는 상황을 구체적인 사례로 설명해주시겠어요?"
전제 개념

왜 agent를 여러 개로 쪼개는가 — Orchestrator-Worker 패턴

단일 agent의 한계부터 짚기

Day 2~4에서 다룬 단일 agent loop(gather context → take action → verify → repeat)는 도구를 아무리 늘려도 결국 하나의 context window, 하나의 실행 흐름 안에서 순차적으로 움직인다. 리서치처럼 "서로 독립적인 하위 질문 여러 개를 동시에 파고들어야 하는" 작업에서는, 순차 실행이 곧 지연시간이 되고 하나의 context window에 모든 조사 결과를 다 우겨넣으면 Day 3의 context rot 문제가 재현된다. Multi-agent 아키텍처는 이 지점 — "독립적으로 병렬화 가능한 하위 작업이 있고, 그 결과를 압축해서 상위로 올려도 정보 손실이 감당 가능한 경우"에 의미가 생긴다.

Orchestrator-worker(lead-subagent) 구조

Anthropic이 자사 리서치 agent를 만들며 공개한 패턴이 대표적이다. lead agent가 사용자 질문을 분석해 조사 전략을 세우고 3~5개의 subagent를 병렬로 띄운다. 각 subagent는 자기 몫의 하위 질문에 대해 다시 자체적으로 3개 이상의 도구를 병렬 호출하며 조사한다 — 즉 병렬화가 두 단계(lead→subagent, subagent→tool)로 겹친다. 이때 subagent는 전체 대화 기록을 통째로 올리는 게 아니라, 파일시스템에 조사 결과(artifact)를 기록하고 lead agent에게는 요약된 참조(reference)만 반환한다 — Day 3의 context 예산 원칙을 "누구의 context window에 무엇을 남길 것인가"라는 조직 설계 문제로 확장한 것이다. 마지막으로 별도의 synthesis(종합) 단계가 여러 subagent의 결과를 인용과 함께 하나의 답으로 재구성한다.

Anthropic은 이 구조로 단일 agent(Opus) 대비 90.2% 성능 향상을 내부 리서치 평가에서 확인했다고 밝혔다. 다만 이건 "breadth-first"(폭넓게 퍼져서 조사해야 하는) 과제에 대한 내부 평가 수치이지, 모든 태스크에 일반화되는 숫자가 아니라는 점을 면접에서 함께 짚어주는 게 좋다.

공짜가 아니다 — 15x 토큰 비용

Anthropic은 같은 블로그에서 multi-agent 시스템이 단일 턴 채팅 대비 약 15배의 토큰을 쓴다고 밝혔다(단일 agent만 써도 이미 채팅 대비 약 4배). 그리고 토큰 사용량 자체가 성능 편차의 약 80%를 설명하고, 도구 호출 횟수와 모델 선택이 나머지 대부분을 설명한다고 명시했다 — 즉 "agent가 똑똑해서" 성능이 오르는 게 아니라 "많은 토큰과 도구 호출을 병렬로 쏟아부을 수 있어서" 오르는 측면이 크다는 뜻이다. 이건 Day 13에서 다룰 비용/성능 trade-off의 핵심 전제이기도 하다 — multi-agent는 "성능을 돈으로 사는" 아키텍처이므로, 그 돈을 정당화할 만큼 과제의 가치가 높은지 항상 먼저 물어야 한다.

Anthropic이 명시한 적합/부적합 조건

적합한 조건

  • 과제의 가치가 높아 토큰 비용을 정당화할 수 있다
  • 하위 작업들이 서로 독립적으로 병렬화 가능하다
  • 필요한 정보량이 단일 context window를 초과한다
  • 다수의 복잡한 도구와 동시에 인터페이스해야 한다

부적합한 조건

  • agent들 사이에 공유 context가 계속 필요한 작업
  • agent 간 상호 의존성이 많은 작업(한쪽 결과가 다른 쪽 판단을 계속 바꿈)
  • 대부분의 코딩 작업 — 리서치보다 진짜 병렬화 가능한 하위 작업이 적고, 긴밀한 결합·순차 의존성·공유 mutable state가 많아 병렬화 이득보다 조율 비용이 더 커지기 쉽다
전제 개념

Handoff vs Agents-as-Tools — 제어권을 넘기는가, 결과만 받는가

공통점: 둘 다 모델에게는 "도구 호출"로 보인다

OpenAI Agents SDK는 multi-agent 조율을 두 가지 표준 패턴으로 정리한다. 흥미로운 점은 둘 다 구현상 모델에게는 tool call로 노출된다는 것이다 — 차이는 "그 도구를 부른 다음 누가 대화의 주도권을 갖는가"에 있다.

Handoff — 대화 소유권 자체를 이전

handoff(agent=...)로 만든 handoff는 모델에게 transfer_to_<agent_name> 같은 이름의 도구로 보인다(예: "Refund Agent" → transfer_to_refund_agent). 모델이 이 도구를 호출하면, 새 agent가 이후 대화 전체를 완전히 넘겨받아 다음 응답을 스스로 생성한다 — 원래 agent는 더 이상 개입하지 않는다. 고객 지원 triage(문의를 받아 결제/기술지원/환불 전문 agent로 라우팅)처럼 "누가 이 대화를 계속 이어받을지" 자체가 문제인 상황에 맞는 패턴이다. on_handoff 콜백으로 이전 시점의 로깅·사전 데이터 로딩을 걸 수 있고, input_filter로 새 agent에게 넘길 이전 대화 기록의 범위를 통제한다 — 너무 많이 넘기면 새 agent가 관련 없는 맥락에 끌려가고, 너무 적게 넘기면 사용자가 이미 준 정보를 다시 물어보게 된다.

Agents-as-Tools(Manager 패턴) — 제어권은 유지, 결과만 받아 종합

반대로 agent.as_tool()로 감싸면, 매니저(orchestrator) agent가 다른 agent를 평범한 함수처럼 호출하고 그 결과를 돌려받아 스스로 다음 판단을 이어간다 — 대화의 주도권은 절대 넘어가지 않는다. 문서 처리 파이프라인(분류 → 추출 → 검증)처럼 애초에 "이 대화를 누가 소유하는가"라는 개념 자체가 없고 하위 작업을 위임했다가 다시 돌려받는 구조가 자연스러운 상황에 맞는다.

Handoff가 맞는 경우

  • 라우팅/triage — "누가 이 대화를 이어받을지"가 핵심 문제
  • 전문 영역별로 시스템 프롬프트·도구 세트가 크게 다른 경우
  • 단점: 왜 그 전문가에게 넘어갔는지, 잘못 라우팅된 이유를 추적하려면 handoff 결정 지점을 따로 파봐야 해서 디버깅이 상대적으로 어렵다

Agents-as-Tools가 맞는 경우

  • 하위 작업 위임 — 대화 소유자가 하나로 고정된 파이프라인/워크플로우
  • 매니저가 여러 하위 결과를 조합·검증해야 하는 경우
  • 장점: 매니저의 판단 과정이 한 곳에 남아 트레이싱·디버깅이 상대적으로 쉽다

Claude Agent SDK의 subagent(Task tool) — 격리된 context와 "중첩 금지"

Claude Agent SDK/Claude Code는 Task tool로 subagent를 스폰한다. subagent는 자기만의 격리된 context window에서 실행되고, 부모 agent에게는 요약된 최종 결과만 반환된다(작업 도중의 세부 trace는 넘어가지 않는다) — 개념적으로는 앞서 본 orchestrator-worker artifact 패턴과 같은 방향이다. 공식 가이드는 "자기만의 context window·system prompt·도구 제한이 필요할 때만 subagent를 쓰고, 단순 함수 호출이면 tool을, 단순 지시문 묶음이면 skill을 쓰라"고 명시한다. 그리고 실무에서 자주 나오는 실패 패턴으로 "subagent sprawl"(subagent를 계속 늘리다가 메모리·컨텍스트·비용이 함께 불어나는 현상)을 지적하며, subagent는 자신의 하위 subagent를 다시 스폰할 수 없다(nesting 금지)는 제약도 명시적으로 둔다 — 무한히 깊어지는 조직도를 막기 위한 구조적 안전장치다.

전제 개념

Multi-Agent를 쓰지 말아야 할 때 — Cognition의 반론과 그 이후의 입장 수정

"Don't Build Multi-Agents" — context sharing과 conflicting decision 문제

Cognition의 Walden Yan은 multi-agent 아키텍처가 두 가지 이유로 잘 깨진다고 지적했다. ① agent 간 context sharing이 부족하다 — 서브agent끼리 전체 trace가 아니라 요약된 메시지만 주고받으면, 정보가 릴레이될 때마다 조금씩 깎여나가는 "전화 게임(game of telephone)" 문제가 생긴다. ② 의사결정이 여러 agent에 분산돼 서로 충돌한다 — 모든 행동에는 암묵적인 판단이 들어있는데, 그 판단을 공유하지 않은 채 각자 진행하면 결과물이 맞지 않게 된다. 예시로 든 것이 Flappy Bird 클론을 만들 때 한 agent에게는 배경/파이프/충돌판정을, 다른 agent에게는 새 캐릭터 스프라이트를 맡기는 경우다 — 서로 상대방의 암묵적 스타일·충돌 가정을 보지 못해 아트 스타일과 히트박스가 어긋난 결과물이 나온다.

Anthropic도 코딩에서는 같은 결론에 도달했다

흥미로운 건 리서치 시스템으로 orchestrator-worker의 성공을 보고한 Anthropic조차, 코딩 작업에 대해서는 같은 문제를 구체적으로 인정한다는 점이다 — 테스트를 작성하는 subagent가 구현 쪽 subagent의 설계 이유(왜 이렇게 짰는지)를 모르고, 리뷰하는 subagent가 탐색 과정에서 나온 맥락을 못 보는 실패 사례를 명시적으로 언급했다. 즉 "multi-agent가 리서치에서 통했다"는 결과와 "코딩에서는 신중해야 한다"는 결론이 같은 팀의 같은 블로그 안에서 공존한다 — 이 구분을 정확히 아는 게 면접에서 "orchestrator-worker면 다 좋다"는 식의 단순화를 피하는 핵심이다.

후속 입장: "쓰지 마라"에서 "이렇게 쓰면 된다"로

Cognition은 이후 "Multi-Agents: What's Actually Working"이라는 후속 글에서 입장을 다듬었다 — multi-agent가 통하는 경우는 쓰기(write)/실행이 항상 단일 지점에 머물고, 추가되는 agent는 "행동"이 아니라 "지능(intelligence)"만 보태는 구조일 때다. 구체적으로는 map-reduce-and-manage 형태 — 매니저가 작업을 쪼개고, 일시적인(ephemeral) child agent들이 격리된 context에서 조사/실행하고, 매니저가 그 결과를 종합해서 최종적으로 보고하는 구조다. 이건 앞서 본 Anthropic의 orchestrator-worker 패턴과 사실상 같은 모양으로 수렴한다. 반대로 여전히 안 통한다고 밝힌 건 agent들이 서로 직접 협상하는 "swarm"/GroupChat류의 peer-to-peer 구조다 — 중앙의 단일 판단 지점 없이 서로 대화하게 두면 "산만함(distraction)"이 된다는 것이다.

정리하면 handoff/multi-agent 논의에서 "쓰지 말라"와 "이렇게 쓰면 된다"는 서로 모순된 두 입장이 아니라, 같은 원칙의 앞뒤다 — 문제는 "여러 agent를 쓰는 것" 자체가 아니라 쓰기/의사결정을 여러 곳에 분산시키는 것이다. 단일 쓰기 지점(매니저 또는 단일 최종 통합자)을 유지한 채 조사·탐색만 병렬로 나누면 이득을 보고, 여러 agent가 각자 독립적으로 최종 산출물에 손을 대게 하면 깨진다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

  • "multi-agent = 무조건 더 좋다"는 유행어 사고에 빠지지 않는가: 병렬화 이득과 토큰 비용을 함께 저울질하는 비용-편익 사고를 하는지 확인한다.
  • handoff와 agents-as-tools를 실제로 구분해서 쓸 수 있는가: 둘 다 "여러 agent를 쓴다"는 점은 같지만 제어권 이전 여부가 완전히 다른 설계 결정이라는 걸 아는지 본다.
  • 업계의 입장 변화(Cognition의 반론 → 수정)를 알고, 그 이유를 설명할 수 있는가: "예전 블로그 글 하나"를 그대로 외운 게 아니라, 그 이후 논의가 어떻게 정교해졌는지 따라오고 있는지 확인한다.
  • 실패 모드를 구체적인 사례로 설명할 수 있는가: "context sharing 문제"라는 추상적 표현이 아니라 Flappy Bird 예시나 테스트/리뷰 subagent 사례처럼 구체적으로 짚을 수 있는지 본다.
  • 단일 agent + 좋은 도구 설계로 대체 가능한지 먼저 검토하는 습관이 있는가: Day 2의 tool 설계 원칙(Vercel d0 사례)과 연결해서, multi-agent가 첫 번째 선택지가 아니라 마지막 수단이라는 감각이 있는지 확인한다.
이 주제는 "우리 팀도 multi-agent를 도입해볼까요?"라는 실제 아키텍처 회의에서 자주 나오는 질문의 축소판이다. 면접관은 지원자가 이 결정을 유행이 아니라 비용/편익 계산으로 할 수 있는 사람인지를 이 질문 하나로 가늠한다.
실무 연결

실제 상황과 연결하기

상황 1: "리서치/경쟁사 분석/법률 검토처럼 폭넓게 흩어져서 찾아야 하는 작업"을 하나의 agent가 순차적으로 처리하고 있어 너무 느리다

하위 질문들이 서로 독립적이고 병렬화 가능하다면 orchestrator-worker 후보다. → lead agent가 하위 질문으로 쪼개고 3~5개 subagent를 병렬로 띄우되, 각 subagent는 전체 trace가 아니라 요약된 결과만 반환하게 해서 lead agent의 context 예산을 지킨다.

상황 2: 고객 지원 챗봇이 결제/기술지원/환불 문의를 전부 하나의 프롬프트로 처리하려다 프롬프트가 비대해지고 있다

"누가 이 대화를 이어받을지"가 핵심 문제이므로 handoff가 적합하다. → triage agent가 의도를 분류해 transfer_to_billing_agent 같은 handoff를 호출하고, input_filter로 환불 전문 agent에게 필요한 맥락만 넘긴다.

상황 3: 문서 처리 파이프라인(분류 → 추출 → 검증)에서 각 단계를 별도 agent로 만들고 싶다

대화 소유권 개념 자체가 없는 파이프라인이므로 agents-as-tools(manager)가 적합하다. → 매니저 agent가 분류/추출/검증 agent를 함수처럼 순서대로 호출하고, 각 결과를 검증해 다음 단계로 넘기며, 전체 결정 과정을 매니저 하나의 trace에 남긴다.

상황 4: 코드 리뷰를 자동화하면서 "구현 agent"와 "리뷰 agent"를 분리했는데, 리뷰 agent가 엉뚱한 코멘트를 계속 단다

리뷰 agent가 구현 agent의 탐색·설계 결정 맥락을 못 보고 있을 가능성이 높다 — Anthropic이 명시한 코딩 multi-agent 실패 패턴과 정확히 같다. → 리뷰를 완전히 분리된 agent로 두기보다, 같은 세션 안에서 구현 trace를 공유하거나, 최소한 설계 결정의 근거를 요약해 리뷰 agent에게 명시적으로 전달하는 구조로 바꾼다.

상황 5: 여러 agent가 자유롭게 서로 대화하며 작업을 나눠 갖는 "swarm" 구조를 설계하고 싶다는 제안이 나왔다

peer-to-peer로 agent끼리 직접 협상하게 하는 구조는 Cognition의 후속 글에서도 여전히 "산만함"으로 지적된 패턴이다. → 대신 단일 매니저가 작업을 쪼개고 종합하는 map-reduce-and-manage 구조로 재설계하고, agent 수가 늘어날수록 조율 비용(4개 agent=6개 연결, 10개=45개 연결)이 조합적으로 커진다는 점을 근거로 제시한다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

2
비용을 먼저 인정하고 조건을 제시 "다만 multi-agent는 채팅 대비 15배 정도 토큰을 쓰는 구조라서, 하위 작업이 정말 독립적이고 병렬화 가능한지, 그리고 그 비용을 정당화할 만큼 과제 가치가 높은지부터 판단합니다."
3

4
쓰지 말아야 할 조건을 구체적으로 짚기 "agent 간 공유 context가 계속 필요하거나 상호 의존성이 강한 작업, 특히 코딩처럼 결합도가 높은 작업은 오히려 단일 agent가 낫다는 게 Anthropic·Cognition 양쪽에서 공통적으로 나온 결론입니다."
5
핵심 원칙으로 마무리 "결국 원칙은 '여러 agent를 쓰느냐'가 아니라 '쓰기(write)와 의사결정을 단일 지점에 유지하느냐' 입니다. 조사·탐색만 병렬로 나누고 최종 통합은 한 곳에서 하면 이득을 보고, 각자 독립적으로 최종 산출물에 손을 대게 하면 깨집니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Multi-agent 아키텍처는 기본값이 아니라, 하위 작업이 서로 독립적으로 병렬화 가능하고 그 정보량이 단일 context window를 넘어설 때 고려하는 선택지라고 생각합니다. Anthropic이 자사 리서치 agent에서 공개한 orchestrator-worker 구조가 좋은 예인데, lead agent가 하위 질문을 나눠 3~5개 subagent를 병렬로 띄우고, 각 subagent는 전체 대화를 다 올리는 대신 파일시스템에 조사 결과를 남기고 요약만 반환해서 lead agent의 context 예산을 지킵니다. 다만 이 구조가 단일 agent 대비 90% 넘는 성능 향상을 냈다고 밝힌 반면, 같은 회사가 채팅 대비 15배 정도의 토큰을 쓴다는 것도 함께 밝혔습니다 — 성능은 공짜가 아니라 토큰으로 사는 것이고, 실제로 토큰 사용량 자체가 성능 편차의 80% 정도를 설명한다고 하니, 이 비용을 정당화할 만큼 과제 가치가 높은지를 먼저 따져야 한다고 봅니다.

패턴을 고를 때는 handoff와 agents-as-tools를 구분합니다. 둘 다 모델에게는 도구 호출로 보이지만, handoff는 새 agent에게 대화 소유권 자체를 완전히 넘기는 구조라서 고객 지원 triage처럼 '누가 이 대화를 이어받을지'가 핵심 문제일 때 씁니다. 반대로 agents-as-tools는 매니저가 대화 주도권을 계속 갖고 다른 agent를 함수처럼 호출해 결과만 받는 구조라서, 분류-추출-검증 같은 파이프라인처럼 애초에 대화 소유권 개념이 없는 하위 작업 위임에 맞습니다. 실무적으로는 handoff 쪽이 라우팅 결정 추적이 더 어렵고, agents-as-tools는 매니저의 판단 과정이 한 곳에 남아 디버깅이 상대적으로 쉽다는 차이도 있습니다.

쓰지 말아야 할 조건도 명확하다고 봅니다. Cognition은 multi-agent가 agent 간 context sharing 부족과 서로 충돌하는 의사결정 때문에 깨진다고 지적했는데, 배경/파이프를 만드는 agent와 캐릭터를 만드는 agent가 서로의 암묵적 스타일 결정을 못 봐서 결과물이 어긋나는 예시가 인상적이었습니다. 흥미로운 건 리서치에서 orchestrator-worker의 성공을 보고한 Anthropic조차 코딩 작업에 대해서는 같은 문제를 인정했다는 겁니다 — 테스트 작성 agent가 구현 agent의 설계 이유를 모르는 식으로요. 그래서 저는 이걸 '여러 agent를 쓰느냐 마느냐'의 문제가 아니라 '쓰기와 의사결정을 단일 지점에 유지하느냐'의 문제로 봅니다. Cognition도 이후 후속 글에서, 매니저가 작업을 쪼개고 일시적인 child agent가 조사만 하고 매니저가 종합하는 구조는 통하지만, agent들이 서로 직접 협상하는 swarm 구조는 여전히 산만하다고 입장을 다듬었는데, 이게 결국 Anthropic의 orchestrator-worker와 같은 그림으로 수렴한다고 생각합니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "그럼 프로덕션에서 multi-agent를 도입하기 전에 뭘 먼저 시도해봐야 하나요?"

약한 답변: "일단 만들어보고 성능이 안 나오면 그때 고민합니다."

강한 답변: "Day 2에서 다룬 것처럼 먼저 단일 agent에 좋은 도구 설계(통합, 네임스페이싱, poka-yoke)와 충분한 context 예산 편성으로 병목을 얼마나 해소할 수 있는지 확인합니다. Vercel d0 사례처럼 도구를 줄이는 것만으로 성공률과 속도가 크게 개선된 경우도 있으니, multi-agent는 '단일 agent를 최적화했는데도 병렬화가 필요한 게 명확할 때' 도입하는 두 번째 단계로 보는 게 맞다고 생각합니다."

F2. "Handoff에서 input_filter로 이전 대화를 얼마나 넘겨야 하는지, 그 기준을 어떻게 잡나요?"

강한 답변: "새 agent가 답변하는 데 실제로 필요한 최소한의 정보가 기준입니다. 너무 많이 넘기면 새 agent가 이전 전문가의 시행착오나 관련 없는 도구 호출 이력에 끌려가 판단이 흐려지고, 너무 적게 넘기면 사용자가 이미 준 정보를 다시 물어봐서 경험이 나빠집니다. 실무적으로는 사용자의 원래 요청과 직전 결론 정도로 좁히고, 세부 도구 호출 trace는 필요할 때만 요약해서 넘기는 쪽으로 시작해서 실제 실패 사례를 보며 조정하는 편이 안전합니다."

F3. "Subagent가 자기 subagent를 못 스폰하게 막아둔 이유가 뭐라고 생각하나요?"

강한 답변: "제한 없이 중첩을 허용하면 조직도가 무한히 깊어질 수 있고, 그만큼 컨텍스트·메모리·비용이 기하급수적으로 불어나는 'subagent sprawl'이 심해집니다. Day 4에서 다룬 step cap/budget cap과 같은 맥락의 안전장치라고 봅니다 — 제어 흐름이 끝나지 않을 가능성을 아키텍처 레벨에서 원천 차단하는 것이 개별 subagent가 알아서 잘 종료하길 기대하는 것보다 안전합니다."

F4. "Anthropic은 리서치에서 90% 넘는 성능 향상을 봤다는데, 왜 같은 회사가 코딩에서는 multi-agent를 신중하게 접근하라고 하나요? 모순 아닌가요?"

강한 답변: "모순이 아니라 조건의 차이라고 봅니다. 리서치는 하위 질문들이 서로 거의 독립적이라 병렬로 나눠도 결과를 나중에 합치기 쉽지만, 코딩은 구현·테스트·리뷰가 서로 강하게 결합돼 있어서 한쪽의 설계 결정을 모르면 다른 쪽이 잘못된 판단을 내리기 쉽습니다. 즉 같은 회사의 같은 아키텍처 원칙이 과제의 결합도(coupling)에 따라 정반대 결론으로 이어지는 것이고, 저는 이걸 '병렬화 가능한 하위 작업이 진짜로 독립적인가'를 항상 먼저 확인해야 한다는 근거로 씁니다."

F5. "Swarm처럼 agent들이 서로 직접 소통하는 구조가 유혹적으로 보이는데, 왜 계속 비권장되나요?"

강한 답변: "중앙의 단일 판단 지점이 없으면 agent 수가 늘어날수록 서로 조율해야 하는 관계의 수가 조합적으로 늘어납니다 — 4개면 6개, 10개면 45개 식으로요. 이러면 어느 한 agent의 판단 오류가 다른 agent들에게 순차적으로 전파되거나 서로 충돌하는 결정을 내려도 누구도 그걸 검증할 위치에 있지 않게 됩니다. 결국 Cognition의 후속 결론처럼, 여러 agent를 쓰더라도 최종 쓰기·의사결정은 한 곳(매니저)에 모아두는 map-reduce-and-manage 구조가 swarm보다 안전하다고 생각합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • multi-agent를 기본값이 아니라 조건부 선택으로 다룬다
  • 토큰 비용(15x)과 성능 향상(90.2%)을 함께 인용해 비용-편익으로 말한다
  • handoff와 agents-as-tools를 제어권 이전 여부로 명확히 구분한다
  • Anthropic·Cognition 양쪽의 구체적 실패 사례(코딩, Flappy Bird)로 부적합 조건을 설명한다
  • "쓰지 마라 → 이렇게 쓰면 된다"는 업계 논의의 변화 자체를 알고, 그 변화가 수렴하는 원칙(단일 쓰기 지점)을 짚는다

약한 답변의 특징

  • "agent를 여러 개 쓰면 더 똑똑해진다"는 막연한 낙관에 머문다
  • handoff와 agents-as-tools를 같은 것으로 혼동한다
  • 토큰 비용을 전혀 언급하지 않는다
  • "multi-agent는 무조건 좋다" 또는 "무조건 쓰면 안 된다"는 이분법에 머문다
  • 실패 사례 없이 추상적으로만 "coordination overhead가 있다"고 말한다
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "multi-agent 시스템은 언제나 단일 agent보다 성능이 좋다"

사실: Anthropic이 밝힌 90.2% 성능 향상은 병렬화 가능한 breadth-first 리서치 과제에 대한 내부 평가 수치다. 같은 회사가 코딩처럼 결합도 높은 작업에서는 오히려 신중하라고 명시했다 — "multi-agent면 무조건 낫다"는 일반화는 성립하지 않는다.

오해 2: "handoff와 agents-as-tools는 결국 같은 것이다(둘 다 agent를 여러 개 쓰는 것)"

사실: 둘 다 모델에게는 도구 호출로 보이지만, 대화 소유권이 완전히 이전되는지 (handoff) 매니저가 계속 소유한 채 결과만 받는지(agents-as-tools)가 근본적으로 다른 설계 결정이다. 라우팅 상황에 agents-as-tools를 쓰거나 파이프라인에 handoff를 쓰면 불필요하게 복잡해진다.

오해 3: "Cognition의 'Don't Build Multi-Agents'는 여전히 multi-agent를 쓰지 말라는 뜻이다"

사실: Cognition은 이후 후속 글에서 입장을 다듬었다 — 쓰기/의사결정이 단일 지점에 머무는 map-reduce-and-manage 구조는 통한다고 밝혔고, 여전히 안 통한다고 본 건 agent들이 서로 직접 협상하는 peer-to-peer swarm 구조다. "예전 글 하나"만 알고 있으면 최신 입장을 놓치게 된다.

오해 4: "subagent를 계속 늘리면 그만큼 병렬 처리 능력이 계속 좋아진다"

사실: agent 수가 늘어날수록 조율해야 하는 관계의 수는 조합적으로 늘어나고, "subagent sprawl"로 메모리·컨텍스트·비용이 함께 불어난다. Claude Agent SDK가 subagent의 subagent 스폰을 아예 막아둔 것도 이 문제를 구조적으로 차단하기 위해서다.

오해 5: "multi-agent 비용은 단순히 agent 수에 비례한다"

사실: Anthropic은 토큰 사용량 자체가 성능 편차의 약 80%를 설명하고, 도구 호출 횟수와 모델 선택이 나머지 대부분을 설명한다고 밝혔다 — 비용은 agent 수만이 아니라 각 agent가 얼마나 많은 도구를 얼마나 반복적으로 호출하는지에 더 크게 좌우된다.

내 답변 수준 확인하기

미흡 "agent를 여러 개 쓰면 더 똑똑해진다"는 막연한 답변에 머물거나, handoff와 agents-as-tools를 구분하지 못함
주니어 orchestrator-worker 패턴은 설명할 수 있지만, 토큰 비용이나 부적합 조건은 언급하지 못함
중급 비용(15x)과 성능(90.2%)을 함께 인용하고 handoff/agents-as-tools를 구분하지만, 업계 논의가 어떻게 변화해왔는지(Cognition의 입장 수정)까지는 확장하지 못함
시니어 비용-편익, 패턴 구분, 부적합 조건(코딩·swarm), 업계 입장 변화와 그 수렴 원칙(단일 쓰기 지점)까지 하나의 일관된 그림으로 설명하며, 구체적 실패 사례로 뒷받침함
자가 점검 질문:
1. Multi-agent 시스템을 도입하기 전에 먼저 검토해야 할 단일 agent 최적화가 무엇인지 설명할 수 있는가?
2. Orchestrator-worker 패턴의 성능 향상과 토큰 비용 수치를 함께 인용하며 그 의미를 설명할 수 있는가?
3. Handoff와 agents-as-tools를 "제어권 이전 여부"로 명확히 구분하고 각각의 적합한 상황을 예시로 들 수 있는가?
4. Multi-agent가 부적합한 조건(공유 context 필요, 코딩처럼 결합도 높은 작업)을 구체적 사례로 설명할 수 있는가?
5. "multi-agent를 쓰지 말라"는 주장과 "이렇게 쓰면 된다"는 후속 입장이 어떤 공통 원칙(단일 쓰기 지점)으로 수렴하는지 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • Multi-agent는 기본값이 아니라 조건부 선택이다. 하위 작업이 독립적으로 병렬화 가능하고, 정보량이 단일 context window를 넘고, 비용을 정당화할 만큼 과제 가치가 높을 때 고려한다.
  • 성능은 공짜가 아니다. orchestrator-worker는 채팅 대비 ~15x 토큰을 쓰고, 토큰 사용량 자체가 성능 편차의 ~80%를 설명한다.
  • Handoff와 agents-as-tools는 제어권 이전 여부로 구분한다. 라우팅/triage엔 handoff, 파이프라인식 하위 작업 위임엔 agents-as-tools(manager).
  • 부적합 조건이 명확히 있다. 공유 context가 계속 필요하거나 상호 의존성이 강한 작업, 특히 결합도 높은 코딩 작업은 multi-agent가 오히려 불리하다.
  • "쓰지 마라"와 "이렇게 쓰면 된다"는 하나의 원칙으로 수렴한다. 문제는 agent 수가 아니라 쓰기/의사결정을 여러 곳에 분산시키는 것이다 — 단일 쓰기 지점을 지키는 map-reduce-and-manage 구조는 통하고, peer-to-peer swarm은 여전히 위험하다.
