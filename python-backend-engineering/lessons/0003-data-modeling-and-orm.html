Day 3 실무자 기준 약 55분

데이터 모델링과 ORM
SQLAlchemy 2.x · Alembic 실전

스키마는 한 번 잘못 설계하면 되돌리기 가장 어려운 결정이다. 이 레슨은 SQLAlchemy 2.x의 Data Mapper 모델 정의, 관계 로딩 전략(N+1 문제), 그리고 무중단 배포를 견디는 Alembic 마이그레이션 전략까지 다룬다.

1. ORM은 편의 도구가 아니라 설계 결정이다

ORM을 쓰는 이유는 "SQL을 안 써도 되어서"가 아니다. 객체 그래프와 관계형 스키마 사이의 임피던스 불일치(impedance mismatch)를 일관된 방식으로 다루기 위해서다. SQLAlchemy는 두 가지 큰 설계 축을 제공한다.

패턴대표 구현특징
Active Record Django ORM 모델 인스턴스가 자기 자신을 저장/조회한다 (user.save()). 단순하지만 도메인 로직과 영속성이 강하게 결합된다.
Data Mapper SQLAlchemy 모델은 순수한 데이터 구조이고, Session이 영속성을 담당한다. 도메인 객체가 DB를 몰라도 된다 — 서비스 레이어와 궁합이 좋다.
Day 4와의 연결
Data Mapper 패턴은 Day 4의 Repository 패턴이 성립하는 전제 조건이다. 모델이 스스로 저장 방법을 알면(Active Record) Repository로 추상화할 이유가 없어진다.

2. SQLAlchemy 2.x 선언적 모델 — Mapped / mapped_column

2.x부터는 타입 힌트 기반의 Mapped[T] 문법이 표준이다. 1.x의 Column(Integer, ...) 방식은 여전히 동작하지만, 새 프로젝트에서는 타입 체커(mypy/pyright)와 잘 맞는 2.x 문법을 쓴다. 

# app/models/base.py
from sqlalchemy.orm import DeclarativeBase

class Base(DeclarativeBase):
    """모든 ORM 모델의 공통 베이스"""
    pass

# app/models/user.py
from datetime import datetime
from sqlalchemy import String, func
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base

class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    email: Mapped[str] = mapped_column(String(255), unique=True, index=True)
    name: Mapped[str] = mapped_column(String(100))
    password_hash: Mapped[str] = mapped_column(String(255))
    is_active: Mapped[bool] = mapped_column(default=True)
    created_at: Mapped[datetime] = mapped_column(server_default=func.now())

    orders: Mapped[list["Order"]] = relationship(back_populates="user")
server_default vs default
server_default=func.now()는 DB가 값을 채운다 (SQL DEFAULT NOW()). default=True는 Python 레벨에서 INSERT 전에 채운다. 시간·시퀀스처럼 DB가 원천이어야 하는 값은 server_default를 쓴다 — 애플리케이션 서버 시계가 어긋나도 DB 기준으로 일관된다.

3. 관계 매핑 — 1:N, N:N, cascade

# app/models/order.py
from sqlalchemy import ForeignKey
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base

class Order(Base):
    __tablename__ = "orders"

    id: Mapped[int] = mapped_column(primary_key=True)
    user_id: Mapped[int] = mapped_column(ForeignKey("users.id"))
    total_amount: Mapped[int]

    user: Mapped["User"] = relationship(back_populates="orders")
    items: Mapped[list["OrderItem"]] = relationship(
        back_populates="order",
        cascade="all, delete-orphan",   # 주문 삭제 시 아이템도 함께 삭제
    )

# N:N — association table
from sqlalchemy import Table, Column

product_tag = Table(
    "product_tag",
    Base.metadata,
    Column("product_id", ForeignKey("products.id"), primary_key=True),
    Column("tag_id", ForeignKey("tags.id"), primary_key=True),
)

class Product(Base):
    __tablename__ = "products"
    id: Mapped[int] = mapped_column(primary_key=True)
    tags: Mapped[list["Tag"]] = relationship(secondary=product_tag, back_populates="products")
cascade="all, delete-orphan"은 신중하게
이 설정은 부모 삭제 시 자식을 자동 삭제한다. 감사 로그(audit log)나 소프트 삭제 정책이 있는 도메인에서는 의도치 않은 데이터 손실로 이어질 수 있다. "삭제"가 실제로 물리적 삭제인지, 상태 변경(soft delete)인지 먼저 정책을 정한다.

4. N+1 문제 — 실무에서 가장 흔한 성능 버그

관계를 정의하면 기본적으로 lazy loading(접근하는 순간 쿼리 발생)이 적용된다. 목록을 순회하며 관계 속성에 접근하면, 목록 조회 1번 + 각 항목마다 관계 조회 1번 = N+1번의 쿼리가 발생한다. 

# 나쁜 예 — N+1 발생
orders = await session.scalars(select(Order))
for order in orders:
    print(order.user.email)   # 각 order마다 별도 SELECT 발생!

# 좋은 예 1 — selectinload: 관계를 별도의 IN 쿼리로 한 번에 가져옴
from sqlalchemy.orm import selectinload

stmt = select(Order).options(selectinload(Order.user))
orders = await session.scalars(stmt)
for order in orders:
    print(order.user.email)   # 추가 쿼리 없음

# 좋은 예 2 — joinedload: LEFT JOIN으로 한 번에 가져옴 (1:1, N:1에 적합)
from sqlalchemy.orm import joinedload

stmt = select(Order).options(joinedload(Order.user))
전략SQL 방식적합한 관계
lazy="select" (기본) 접근 시점마다 별도 SELECT 거의 접근하지 않는 관계
selectinload() 부모 ID 목록으로 WHERE id IN (...) 1회 추가 1:N — 자식이 많을 때 (JOIN 시 행 폭증 방지)
joinedload() LEFT OUTER JOIN 한 번 N:1, 1:1 — 자식 쪽에서 부모를 가져올 때
joinedload를 1:N에 잘못 쓰면?
부모 1건에 자식이 100건이면 JOIN 결과가 100행으로 늘어나 부모 데이터가 100번 중복 전송된다. 1:N 관계에는 selectinload를 우선 고려한다.

5. Async Engine과 Session 설정

# app/database.py
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@localhost/mydb",
    pool_size=10,        # 상시 유지할 커넥션 수
    max_overflow=5,      # 순간 초과 허용치
    pool_pre_ping=True,  # 끊긴 커넥션을 자동 감지 후 재연결
    echo=False,          # 개발 중에만 True로 SQL 로그 확인
)

async_session_maker = async_sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False,
)

async def get_db_session() -> AsyncSession:
    async with async_session_maker() as session:
        yield session
expire_on_commit=False가 필요한 이유
기본값(True)은 커밋 후 객체 속성 접근 시 자동으로 재조회한다. FastAPI 응답 직렬화 시점(커밋 이후)에 속성에 접근하면 이미 세션이 닫혀 DetachedInstanceError가 발생할 수 있다. False로 두면 커밋 후에도 메모리에 남은 값을 그대로 쓴다.

6. Alembic — 스키마 변경을 코드로 관리한다

6-1. 기본 흐름

# 최초 설정
alembic init migrations

# 모델 변경 후 마이그레이션 스크립트 자동 생성
alembic revision --autogenerate -m "add is_active to users"

# 적용
alembic upgrade head

# 롤백 (직전 리비전으로)
alembic downgrade -1
autogenerate는 초안일 뿐이다
Alembic의 자동 생성은 컬럼 이름 변경을 삭제 + 추가로 오인하는 등 완벽하지 않다. 생성된 스크립트를 항상 사람이 검토하고, 데이터 손실 가능성이 있는 drop_column은 특히 주의 깊게 확인한다.

6-2. 무중단 배포를 견디는 마이그레이션 순서

운영 중인 서비스는 스키마 변경과 애플리케이션 배포가 동시에 일어나지 않는다. 구버전 코드가 떠 있는 동안에도 DB가 깨지지 않아야 한다. "NOT NULL 컬럼 추가"를 예로 들면, 아래 순서로 나눠야 한다.

1단계: nullable 컬럼 추가 (구버전 코드도 정상 동작) ALTER TABLE users ADD COLUMN phone VARCHAR(20); 2단계: 애플리케이션 배포 (신버전이 phone을 쓰기 시작) 신버전 코드는 phone에 값을 채워 넣는다 3단계: 기존 행 백필(backfill) (배치 작업, 트랜잭션 잘게 쪼개서) UPDATE users SET phone = '' WHERE phone IS NULL; 4단계: NOT NULL 제약 추가 (모든 행이 값을 가진 뒤에만 안전) ALTER TABLE users ALTER COLUMN phone SET NOT NULL;
한 번에 하면 안 되는 이유
ADD COLUMN phone VARCHAR(20) NOT NULL을 배포 직전에 한 번에 실행하면, 아직 떠 있는 구버전 인스턴스의 INSERT가 phone 값 없이 실패한다. 롤링 배포 중에는 항상 신버전과 구버전이 동시에 트래픽을 받는다는 것을 전제해야 한다.

6-3. 컬럼 삭제도 같은 원칙

1단계: 애플리케이션이 해당 컬럼을 더 이상 읽지도, 쓰지도 않도록 배포 2단계: 일정 기간 모니터링 (롤백 가능성 대비) 3단계: 컬럼 삭제 마이그레이션 배포

7. Pydantic 스키마와 ORM 모델은 왜 항상 분리하는가

Day 2에서 Request/Response 스키마 분리를 다뤘다. ORM 모델 관점에서 보면 이유가 더 명확하다. ORM 모델은 DB 제약조건(스키마)을 표현하고, Pydantic 스키마는 API 계약을 표현한다. 둘의 변경 주기가 다르다. 

# ORM 모델을 그대로 반환하면 안 되는 이유
class User(Base):
    ...
    password_hash: Mapped[str]       # 절대 API로 노출되면 안 됨
    internal_risk_score: Mapped[int] # 내부 전용 필드

# Pydantic 응답 스키마 — 명시적으로 노출할 필드만 선언
class UserResponse(BaseModel):
    id: int
    email: str
    name: str
    is_active: bool
    model_config = {"from_attributes": True}   # ORM 객체 → Pydantic 변환 허용

"화이트리스트" 방식이 핵심이다. 새 컬럼을 ORM 모델에 추가해도 Response 스키마에 명시적으로 넣지 않는 한 API로 노출되지 않는다. 반대로 ORM 모델을 그대로 반환하면, 나중에 추가한 민감한 컬럼이 조용히 노출된다.

8. 데이터 모델링 셀프 리뷰 체크리스트

스키마 작성 시 확인할 것
  • 모든 외래키에 인덱스가 있다 (SQLAlchemy는 FK에 자동으로 인덱스를 만들지 않는다)
  • 1:N 관계 목록 조회에서 selectinload를 쓰고 있다
  • N:1 단건 조회에서 필요 시 joinedload를 쓰고 있다
  • 비밀번호, 토큰 등 민감 컬럼이 Response 스키마에 없다
  • NOT NULL 컬럼 추가는 nullable → 배포 → 백필 → NOT NULL 순서를 따른다
  • autogenerate로 만든 마이그레이션을 사람이 검토했다
  • cascade="all, delete-orphan"이 실제 삭제 정책과 일치한다
  • server_default와 default를 구분해서 썼다
