Python 백엔드 아키텍처
전체 구조 개요

실제 서비스를 지탱하는 백엔드가 어떤 계층으로 구성되고, 요청 하나가 어떻게 흘러가는지 전체 그림을 그린다. Python 문법이 아니라 구조적 사고가 이 레슨의 목표다.

1. 왜 구조를 먼저 이해해야 하는가

코드를 작성하기 전에 전체 구조를 파악하지 않으면 나중에 잘못 붙인 기능들이 서로 얽혀 수정이 불가능해진다. 실무 백엔드 엔지니어의 첫 번째 질문은 항상 이것이다:

핵심 질문
"이 요청은 어디서 들어와서, 어디를 거쳐, 어디에 저장되고, 어떻게 응답되는가?"

이 흐름을 Request Lifecycle이라 부른다. 이것을 알면 N+1 쿼리가 어디서 터지는지, 인증 토큰이 어느 계층에서 검증되는지, 에러가 어느 지점에서 잡혀야 하는지 즉시 판단할 수 있다.

각 계층은 독립적으로 교체 가능해야 한다. 서비스 레이어는 PostgreSQL인지 MySQL인지 알면 안 되고, API 레이어는 비즈니스 로직을 직접 가지면 안 된다. 이 관심사의 분리(Separation of Concerns)가 유지보수 가능한 코드의 핵심이다.

3. 프레임워크 선택 기준

Python 백엔드에서 가장 많이 묻는 질문: "FastAPI vs Django vs Flask, 무엇을 써야 하나요?"

기준 FastAPI Django Flask
타입 힌트 / 자동 문서화 ✅ 1급 지원 (OpenAPI 자동 생성) ⚠️ 별도 라이브러리 필요 ⚠️ 별도 라이브러리 필요
비동기 (async/await) ✅ 네이티브 ⚠️ ASGI 모드 필요 (Django 4.1+) ⚠️ 제한적
배터리 포함 ❌ 직접 구성 ✅ ORM·관리자·인증 내장 ❌ 직접 구성
학습 곡선 낮음 (Python 3.10+ 경험 있으면) 중간 (Django ORM, MTV 패턴) 낮음 (단순하지만 설계는 직접)
적합한 상황 신규 API 서버, 마이크로서비스 어드민·CMS 포함된 모놀리스 초소형 서비스, 프로토타입
실무 판단 기준

신규 프로젝트이고 팀이 타입 힌트에 익숙하다면 FastAPI가 기본 선택지다. 어드민 패널, 복잡한 인증, 레거시 연동이 많다면 Django를 고려한다. Flask는 특수 목적(ML 모델 서빙, 내부 툴)에 제한적으로 쓴다.

이 학습 과정은 FastAPI + SQLAlchemy + Celery + PostgreSQL 스택을 기준으로 한다.

4. 요청 수명 주기 — 코드로 보는 흐름

예시: POST /api/v1/orders 요청이 처리되는 과정. 각 계층이 무엇을 하는지 코드 조각과 함께 확인한다.

① 게이트웨이 → API 레이어 (FastAPI)

# app/api/v1/orders.py
from fastapi import APIRouter, Depends, status
from app.schemas.order import OrderCreate, OrderResponse
from app.services.order_service import OrderService
from app.api.deps import get_current_user, get_order_service

router = APIRouter()

@router.post("/orders", response_model=OrderResponse, status_code=status.HTTP_201_CREATED)
async def create_order(
    body: OrderCreate,                  # Pydantic이 JSON 파싱 + 검증
    current_user = Depends(get_current_user),  # 미들웨어가 JWT 확인
    svc: OrderService = Depends(get_order_service),
):
    return await svc.create_order(user_id=current_user.id, data=body)

라우터는 입력 검증과 의존성 주입만 담당한다. 비즈니스 로직은 전혀 없다.

② 서비스 레이어 (비즈니스 로직)

# app/services/order_service.py
class OrderService:
    def __init__(self, repo: OrderRepository, event_bus: EventBus):
        self.repo = repo
        self.event_bus = event_bus

    async def create_order(self, user_id: int, data: OrderCreate) -> Order:
        # 비즈니스 규칙: 재고 확인
        if not await self.repo.check_stock(data.product_id, data.quantity):
            raise InsufficientStockError()

        order = await self.repo.create(user_id=user_id, **data.model_dump())

        # 사이드이펙트: 이메일 발송은 백그라운드로
        await self.event_bus.publish("order.created", {"order_id": order.id})
        return order

서비스 레이어는 DB가 PostgreSQL인지, 캐시가 Redis인지 모른다. 오직 도메인 규칙에만 집중한다.

③ 데이터 레이어 (Repository + SQLAlchemy)

# app/repositories/order_repo.py
class OrderRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def create(self, **kwargs) -> Order:
        order = Order(**kwargs)
        self.session.add(order)
        await self.session.flush()   # DB에 INSERT, 트랜잭션은 아직 열려있음
        return order

④ 백그라운드 태스크 (Celery)

# app/tasks/notifications.py
from app.celery_app import celery

@celery.task(bind=True, max_retries=3, default_retry_delay=60)
def send_order_confirmation_email(self, order_id: int):
    try:
        send_email(order_id)
    except EmailServiceDown as exc:
        raise self.retry(exc=exc)  # 최대 3번 재시도

6. 관측성(Observability) 계층은 왜 수평으로 배치하는가

로깅·메트릭·트레이싱은 모든 계층에 걸쳐 동작해야 한다. 요청이 어느 계층에서 느려졌는지 알려면 각 계층을 관통하는 상관 ID(Correlation ID)가 있어야 한다. 

# 미들웨어에서 요청 ID를 발급하고 모든 로그에 붙인다
import uuid, structlog

async def correlation_id_middleware(request, call_next):
    request_id = str(uuid.uuid4())
    structlog.contextvars.bind_contextvars(request_id=request_id)
    response = await call_next(request)
    response.headers["X-Request-ID"] = request_id
    return response

이 패턴 덕분에 에러 로그를 보면 어느 사용자의 어느 요청에서 문제가 생겼는지 즉시 추적할 수 있다.
