Day 7 실무자 기준 약 65분

백그라운드 작업
Celery + Redis 아키텍처 · 재시도 전략 · 태스크 멱등성

Day 6까지는 요청-응답 안에서 끝나는 작업만 다뤘다. 하지만 이메일 발송, 이미지 리사이징, 리포트 생성처럼 시간이 걸리는 작업을 API 핸들러 안에서 그대로 처리하면 응답이 느려지고, 워커가 죽으면 작업이 중복 실행되거나 통째로 사라진다. 이 레슨은 그 작업을 API 밖으로 떼어내 안전하게 재시도하고, 두 번 실행돼도 결과가 망가지지 않게 만드는 방법을 다룬다.

1. "일단 동작은 하는" 동기 처리 코드

아래 코드는 회원가입 API 안에서 환영 이메일을 바로 보낸다. 로컬 테스트에서는 문제없이 통과하지만, 실제 트래픽에서는 세 가지 사고를 그대로 안고 있다. 

# 절대 이렇게 하지 말 것 — 문제가 있는 예시
@router.post("/users")
async def create_user(payload: UserCreate, user_service: UserService = Depends(...)):
    user = await user_service.create(payload)
    send_welcome_email(user.email)      # ← SMTP 서버가 3초 걸리면 API 응답도 3초 걸림
    return user                          # ← 이메일 발송이 실패하면 예외가 API 응답까지 전파됨
이 코드가 실제로 뚫리는 방식
  • SMTP 서버가 느려지거나 응답이 없으면 회원가입 API 전체가 타임아웃난다 — 회원가입은 이메일 발송과 무관한 기능인데도 함께 죽는다
  • 이메일 발송 중 예외가 나면 이미 DB에 저장된 사용자 생성까지 실패 응답으로 보이게 되어, 클라이언트가 재시도하면 중복 가입을 시도한다
  • 배포 중 서버가 재시작되면 처리 중이던 이메일 발송은 그냥 사라진다 — 재시도할 방법이 없다

해결책은 "시간이 걸리거나 실패할 수 있는 작업"을 API 요청 밖의 별도 프로세스(워커)로 넘기고, 그 작업이 실패해도 API 응답과 무관하게 재시도되도록 큐에 넣는 것이다.

2. Celery + Redis 아키텍처 — 누가 무엇을 책임지는가

[FastAPI 프로세스] [Redis: Broker] [Celery Worker 프로세스] | | | | task.delay(user_id) | | |------------------------------>| 큐에 메시지 적재 | | (즉시 반환, 응답 안 기다림) | | | |<-----------------------------| 워커가 메시지를 꺼내감 | | | 작업 실행 (이메일 발송) | | | | |<-----------------------------| 결과를 Result Backend에 저장 | task.get() (필요할 때만) | (Redis를 Result Backend로도 사용 가능) |------------------------------>|
구성 요소역할실무에서 흔한 선택
Broker API 프로세스와 워커 사이에서 "할 일" 메시지를 전달하는 큐 Redis (단순함), RabbitMQ (전달 보장이 더 강함)
Worker 큐에서 메시지를 꺼내 실제 함수를 실행하는 별도 프로세스 API 서버와 별도 컨테이너/프로세스로 배포 — 독립적으로 스케일
Result Backend 작업의 성공/실패/반환값을 저장 — API가 나중에 조회 가능 Redis (fire-and-forget이면 생략 가능)
왜 API 프로세스와 워커를 분리하는가
API 서버는 "빠르게 응답하는 것"이 목표고, 워커는 "느린 작업을 안전하게 끝까지 완료하는 것"이 목표다. 두 목표는 자원 요구량(CPU, 메모리, 동시성 모델)이 다르다. 같은 프로세스에서 처리하면 무거운 백그라운드 작업이 API 응답 지연 시간(latency)에 그대로 영향을 준다. 분리하면 워커만 독립적으로 스케일 아웃할 수 있다 — 이메일 발송이 몰리는 시간대에 API 서버는 그대로 두고 워커 인스턴스만 늘리면 된다.

3. 태스크 정의 — 무엇을 Celery 태스크로 만들 것인가

# infrastructure/tasks/celery_app.py
from celery import Celery

celery_app = Celery(
    "backend",
    broker=settings.redis_url,          # 예: redis://redis:6379/0
    backend=settings.redis_url,         # Result Backend도 Redis 재사용
)
celery_app.conf.update(
    task_serializer="json",             # pickle 대신 json — Section 7 참고
    accept_content=["json"],
    result_expires=3600,                # 결과를 1시간만 보관 (Redis 메모리 관리)
    task_acks_late=True,                # Section 5에서 설명
    worker_prefetch_multiplier=1,       # 워커가 한 번에 너무 많은 태스크를 가져가지 않도록
)

# infrastructure/tasks/email_tasks.py
from infrastructure.tasks.celery_app import celery_app

@celery_app.task(
    bind=True,
    autoretry_for=(SMTPServerDisconnected, ConnectionError),  # Section 4
    retry_backoff=True,
    retry_backoff_max=600,
    retry_jitter=True,
    max_retries=5,
)
def send_welcome_email_task(self, user_id: int) -> None:
    user = user_repo.get_sync(user_id)     # 워커 프로세스는 별도 동기 세션을 쓴다
    smtp_client.send(user.email, template="welcome")

# presentation/routers/user_router.py — API 핸들러에서는 큐에 넣기만 한다
@router.post("/users", status_code=201)
async def create_user(payload: UserCreate, user_service: UserService = Depends(...)):
    user = await user_service.create(payload)       # DB 트랜잭션은 여기서 완료됨
    send_welcome_email_task.delay(user.id)            # 큐에 적재 — 즉시 반환, 실패해도 API는 성공
    return user
흔한 실수 — 트랜잭션 커밋 전에 태스크를 큐에 넣음
user_service.create()의 DB 커밋이 아직 끝나지 않았는데 태스크를 먼저 큐에 넣으면, 워커가 그 태스크를 실행할 때 DB에서 아직 커밋되지 않은(또는 롤백된) 사용자를 조회하려다 실패할 수 있다. Celery 태스크는 같은 프로세스 내 다른 스레드/워커에서 비동기로 즉시 실행될 수 있다는 점을 항상 가정해야 한다. DB 트랜잭션 커밋이 끝난 뒤에만 태스크를 큐에 넣는다.

4. 태스크는 언젠가 두 번 실행된다 — 멱등성(Idempotency) 설계

브로커는 "적어도 한 번(at-least-once)" 전달을 기본으로 한다. 워커가 작업을 처리하는 도중 크래시하면, 브로커는 그 작업이 완료됐는지 알 수 없어 다시 전달한다. 즉, 같은 태스크가 두 번 실행되는 상황은 예외가 아니라 정상 동작이다. 태스크 코드는 두 번 실행돼도 결과가 같도록(멱등하도록) 짜야 한다. 

# 위험한 예시 — 실행될 때마다 중복 발생
@celery_app.task
def charge_order_task(order_id: int) -> None:
    order = order_repo.get_sync(order_id)
    payment_gateway.charge(order.amount)     # ← 두 번 실행되면 이중 결제!
    order.mark_paid()

# 멱등하게 만든 예시 — 고유 키로 중복 실행을 방어
@celery_app.task
def charge_order_task(order_id: int) -> None:
    order = order_repo.get_sync(order_id)
    if order.status == OrderStatus.PAID:
        return                                # 이미 처리됨 — 조용히 종료
    idempotency_key = f"order:{order_id}:charge"
    payment_gateway.charge(order.amount, idempotency_key=idempotency_key)
    # PG사가 idempotency_key로 중복 요청을 서버 쪽에서도 걸러줌 (이중 방어)
    order.mark_paid()
패턴적용 예핵심
상태 확인 후 종료 이미 결제됨/이미 발송됨 플래그 체크 가장 단순하지만 체크와 실행 사이 race condition 가능성 있음 (Day 5의 잠금과 연결)
고유 키(Idempotency Key) 결제 API, 외부 API 호출 외부 시스템이 같은 키의 두 번째 요청을 무시하도록 위임 — 가장 안전
UPSERT 알림 발송 이력 기록 INSERT ... ON CONFLICT DO NOTHING으로 DB 레벨에서 중복 방지

5. 재시도 전략 — 언제, 얼마나, 어떻게 다시 시도할 것인가

태스크 3번 정의에서 쓴 autoretry_for, retry_backoff, retry_jitter가 실제로 무엇을 하는지 살펴보자.

옵션동작왜 필요한가
autoretry_for 지정한 예외가 발생하면 자동으로 재시도 일시적 오류(네트워크 단절)와 영구적 오류(잘못된 이메일 주소)를 구분해서 재시도 대상만 지정
retry_backoff 재시도 간격을 지수적으로 늘림 (1초 → 2초 → 4초 → 8초...) 장애 중인 외부 서비스에 재시도 폭탄을 던지지 않음 (thundering herd 방지)
retry_jitter 재시도 간격에 무작위성을 추가 동시에 실패한 수천 개 태스크가 정확히 같은 타이밍에 몰려 재시도하는 것을 방지
max_retries 최대 재시도 횟수 제한 영구적으로 실패하는 태스크가 무한 재시도로 큐를 계속 점유하는 것을 방지
task_acks_late=True 의 의미: acks_late=False (기본값, 위험): 1. 워커가 큐에서 메시지를 꺼냄 2. 즉시 "처리 완료(ack)"로 표시 ← 실행 전에 이미 완료 처리! 3. 태스크 실행 중 워커가 크래시 4. 메시지는 이미 ack됐으므로 큐에서 사라짐 → 작업이 통째로 유실됨 acks_late=True (권장): 1. 워커가 큐에서 메시지를 꺼냄 2. 태스크 실행 3. 실행이 "성공적으로 끝난 후"에만 ack 4. 실행 중 워커가 크래시 → 메시지는 ack되지 않아 다른 워커에게 재전달됨 (단, 이 재전달 때문에 Section 4의 멱등성이 반드시 필요하다)
acks_late=True를 켜면 반드시 멱등성이 필요한 이유
acks_late=True는 "작업 유실"을 막아주는 대신 "중복 실행" 가능성을 높인다. 태스크가 실제로는 끝났는데 ack를 보내기 직전 워커가 죽으면, 브로커는 그 작업이 처리됐는지 알 방법이 없어 다시 전달한다. 즉 acks_late와 멱등성 설계는 항상 한 세트로 다뤄야 한다. 유실은 절대 안 되고 중복은 상관없는 작업(예: 결제, 재고 차감)에 이 옵션을 쓰고, 멱등성 없이 쓰면 실무 사고로 이어진다.

6. 최대 재시도를 넘긴 태스크는 어디로 가는가

max_retries를 다 소진하면 태스크는 실패(FAILURE) 상태로 종료된다. 이 상태를 그냥 로그에만 남기면 아무도 모르게 조용히 사라진다. 실무에서는 실패한 태스크를 별도로 확인하고 재처리할 수 있는 경로를 만든다. 

# infrastructure/tasks/email_tasks.py
@celery_app.task(bind=True, max_retries=5, ...)
def send_welcome_email_task(self, user_id: int) -> None:
    try:
        ...
    except (SMTPServerDisconnected, ConnectionError) as exc:
        raise self.retry(exc=exc)

@task_failure.connect(sender=send_welcome_email_task)
def on_final_failure(sender, task_id, exception, **kwargs):
    # 최종 실패 시: 알림 채널로 통보 + Dead-letter 테이블에 기록
    dead_letter_repo.record(
        task_name=sender.name, task_id=task_id, error=str(exception),
    )
    alert_slack(f"태스크 최종 실패: {sender.name} ({task_id}) — {exception}")

이렇게 기록해두면 운영자가 dead_letter 테이블을 조회해 원인을 파악하고 (예: 이메일 주소 형식 오류) 수동으로 재처리하거나, 근본 원인을 고쳐서 재실행할 수 있다. 이것이 메시지 큐 시스템에서 흔히 말하는 Dead Letter Queue(DLQ) 패턴의 Celery식 구현이다.

7. 태스크 직렬화 — pickle을 쓰면 안 되는 이유

Celery는 태스크 인자를 브로커에 전달할 때 직렬화한다. 과거 기본값이던 pickle은 역직렬화 시 임의 코드 실행이 가능한 보안 취약점이 있다 — Redis에 누군가 조작된 메시지를 넣을 수 있다면 워커 프로세스에서 임의 코드가 실행될 수 있다. 2번 섹션의 설정처럼 task_serializer="json"을 명시적으로 지정해 이 위험을 제거한다.

태스크 인자에 ORM 객체를 직접 넘기지 않는다
send_email_task.delay(user)처럼 SQLAlchemy 모델 객체를 직접 넘기면 JSON으로 직렬화가 안 될 뿐 아니라, 설령 된다 해도 워커가 실행되는 시점에는 그 객체가 원본 세션과 분리되어(detached) 오래된 데이터를 담고 있을 수 있다. 항상 user.id 같은 원시 식별자만 넘기고, 워커 안에서 별도 세션으로 최신 데이터를 다시 조회한다.

8. 백그라운드 작업 셀프 리뷰 체크리스트

코드 리뷰 시 확인할 것
  • API 응답 시간에 영향을 주는 느린 작업(이메일, 리포트, 외부 API 호출)은 태스크로 분리했다
  • 태스크는 DB 트랜잭션 커밋 이후에만 큐에 적재한다
  • 태스크 인자로 ORM 객체 대신 원시 식별자(id)만 넘긴다
  • task_serializer="json"을 명시했고 pickle을 쓰지 않는다
  • 일시적 오류만 autoretry_for 대상으로 지정하고, backoff + jitter를 적용했다
  • acks_late=True를 쓴다면 해당 태스크가 멱등하게 설계되어 있다
  • 최대 재시도를 넘긴 실패는 조용히 사라지지 않고 Dead Letter 기록 + 알림으로 이어진다
  • 결제/재고처럼 중복이 치명적인 작업은 idempotency key 또는 상태 체크로 이중 방어했다

Q4. Celery 태스크 직렬화에 pickle 대신 json을 명시적으로 지정해야 하는 주된 이유는?

Apickle은 역직렬화 시 임의 코드 실행이 가능한 보안 취약점이 있기 때문
Bjson이 pickle보다 항상 전송 속도가 빠르기 때문
CCelery가 pickle 형식을 더 이상 지원하지 않기 때문
Djson만 재시도(retry) 기능을 지원하기 때문

권장 1차 자료

  • Celery 공식 문서 — Tasks — 태스크 정의, 재시도, 직렬화 옵션의 표준 레퍼런스.
  • Celery 공식 문서 — task_acks_late — ack 타이밍과 유실/중복 트레이드오프 상세 설명.
  • Celery FAQ — Retry vs acks_late — 실무에서 자주 헷갈리는 재시도 전략 선택 가이드.
  • Stripe — Idempotent Requests — 실무에서 검증된 Idempotency Key 패턴의 대표 구현 사례.
  • Redis 공식 문서 — Distributed Locks — 멱등성 체크와 함께 자주 쓰이는 분산 락 패턴.
질문이 있으신가요?
Celery 대신 FastAPI의 BackgroundTasks를 써도 되는 경우가 언제인지, 또는 태스크 체이닝(chain, group, chord)으로 여러 단계를 어떻게 조립하는지 궁금하면 질문하세요. 예시: "결제 승인 → 재고 차감 → 알림 발송을 순서대로 실행하려면 어떻게 설계하나요?"
