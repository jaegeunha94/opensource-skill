캐싱과 성능 최적화
Redis 캐싱 전략 · 캐시 스탬피드 방지 · 응답 압축 · 커넥션 풀 튜닝 · 쿼리 프로파일링

Day 9까지 서비스는 설계되고, 배포되고, 관측 가능해졌다. 하지만 트래픽이 늘면 같은 코드가 갑자기 느려진다. 대부분의 경우 원인은 알고리즘이 아니라 같은 데이터를 반복해서 DB에 물어보는 것과 연결/리소스를 비효율적으로 쓰는 것이다. 이 레슨은 Day 3에서 다룬 N+1 문제 이후 단계 — 캐싱 전략, 캐시가 만드는 새로운 장애 (스탬피드, 오염된 무효화), 응답 압축, 커넥션 풀 튜닝, 그리고 "느리다"를 추측이 아니라 측정으로 진단하는 프로파일링을 다룬다.

1. 캐싱을 넣기 "전에" 확인할 것 — 캐싱은 공짜가 아니다

캐싱은 읽기 성능 문제를 해결하는 가장 강력한 도구지만, 동시에 가장 흔한 셀프 장애 원인이기도 하다. Phil Karlton의 유명한 말처럼 "캐시 무효화와 이름 짓기, 그리고 off-by-one 에러가 컴퓨터 과학의 두 가지 어려운 문제다" — 즉 캐시를 넣는 순간부터 "이 데이터가 언제 stale해지는가"라는 새로운 책임이 생긴다. 캐싱을 넣기 전에 먼저 물어야 한다.

질문왜 중요한가
이 데이터는 얼마나 자주 바뀌는가? 초 단위로 바뀌는 데이터를 캐싱하면 stale 데이터를 서빙할 위험이 이득보다 커진다
읽기:쓰기 비율이 어느 정도인가? 읽기가 압도적으로 많을 때만 캐싱의 이득이 무효화 비용을 넘어선다
stale 데이터를 잠깐 보여줘도 되는가? 잔액·재고처럼 정합성이 중요한 데이터는 캐싱 대상에서 제외하거나 매우 짧은 TTL만 허용
실제로 DB가 병목인가, 측정했는가? 프로파일링 없이 "느릴 것 같아서" 캐싱을 넣으면 무효화 버그만 늘어나고 체감 개선은 없다
캐싱보다 먼저 확인해야 할 것
N+1 쿼리(Day 3), 누락된 인덱스, 불필요하게 큰 응답 페이로드가 원인이라면 캐싱은 근본 원인을 덮어 증상만 완화한다. 캐싱은 "쿼리가 이미 최적화됐는데도 반복 조회 자체가 부하인 경우"에 쓰는 마지막 카드다.

2. Redis 캐싱 전략 — Cache-Aside, Write-Through, Write-Behind

셋 다 "DB 앞에 빠른 저장소를 둔다"는 목표는 같지만, 쓰기가 언제 캐시에 반영되는가가 다르고 이는 곧 정합성과 장애 시 데이터 유실 위험의 trade-off로 이어진다. 

# infrastructure/cache/product_cache.py — Cache-Aside (Lazy Loading), 실무에서 가장 흔한 패턴
async def get_product(product_id: int, redis: Redis, db: AsyncSession) -> Product:
    cache_key = f"product:v2:{product_id}"

    cached = await redis.get(cache_key)
    if cached is not None:
        return Product.model_validate_json(cached)          # 캐시 히트 — DB 조회 없음

    product = await product_repository.get_by_id(db, product_id)  # 캐시 미스 — DB 직접 조회
    if product is None:
        # Day 2-3에서 다룬 문제: 존재하지 않는 ID를 반복 조회하면 캐시를 계속 우회한다 (아래 4절 참고)
        await redis.set(cache_key, "", ex=30)
        return None

    await redis.set(cache_key, product.model_dump_json(), ex=300)  # TTL 5분
    return product

async def update_product(product_id: int, data: ProductUpdate, redis: Redis, db: AsyncSession) -> Product:
    product = await product_repository.update(db, product_id, data)
    await redis.delete(f"product:v2:{product_id}")           # 쓰기 시 캐시 삭제 — 다음 읽기가 재적재
    return product
전략동작장점단점 / 적용 시 주의
Cache-Aside
(Lazy Loading)		 읽기: 캐시 확인 → 미스 시 DB 조회 후 채움
쓰기: DB 갱신 후 캐시 삭제		 구현이 단순, 실제 조회된 데이터만 캐싱(불필요한 데이터 안 채움) 첫 요청은 항상 캐시 미스(cold start), 삭제와 DB 갱신 사이 짧은 race window 존재
Write-Through 쓰기 시 캐시와 DB를 같은 트랜잭션처럼 동시에 갱신 캐시가 항상 최신 — 미스 발생이 적음 모든 쓰기가 캐시 쓰기 지연을 추가로 부담, 자주 안 읽히는 데이터까지 캐싱되어 메모리 낭비 가능
Write-Behind
(Write-Back)		 캐시에 먼저 쓰고, DB 반영은 비동기로 지연 처리 (Day 7 백그라운드 작업과 결합) 쓰기 지연(latency)이 매우 낮음 캐시 장애 시 아직 DB에 반영 안 된 데이터 유실 위험 — 정합성이 중요한 데이터에는 부적합
쓰기 시 캐시를 "갱신"이 아니라 "삭제"하는 이유 (Cache-Aside)
쓰기 시 캐시를 새 값으로 직접 갱신하면, 동시에 실행 중인 다른 쓰기 트랜잭션과 순서가 엇갈릴 때 캐시에 더 오래된 값이 남을 위험이 있다. 캐시를 삭제만 하면 다음 읽기가 항상 DB의 최신 값을 다시 채우므로, 두 쓰기의 순서가 뒤바뀌어도 최종적으로는 DB와 캐시가 일치한다. 이 패턴을 실무에서 "invalidate, don't update"라고 부른다.

3. 캐시 키 설계와 무효화 — TTL만으로는 부족한 이유

TTL(Time-To-Live)은 "언젠가는 만료된다"는 안전망일 뿐, 데이터가 바뀐 즉시 반영되어야 하는 경우의 해답은 아니다. 실무에서는 TTL과 명시적 무효화를 함께 쓴다. 

# 캐시 키에 버전을 넣어 스키마 변경 시 구버전 캐시를 무력화 — 배포 시 캐시 오염 방지
cache_key = f"product:v2:{product_id}"          # v1 → v2로 응답 스키마가 바뀌면 키 prefix만 올린다
                                                  # (기존 v1 키는 그냥 TTL로 자연 소멸)

# 목록/집계 캐시는 개별 키가 아니라 "무효화 대상 그룹" 단위로 태깅
async def invalidate_category(category_id: int, redis: Redis) -> None:
    # SCAN으로 패턴 매칭 삭제 (KEYS는 O(N) 블로킹이라 프로덕션에서 절대 사용 금지)
    async for key in redis.scan_iter(match=f"category:{category_id}:*"):
        await redis.delete(key)
KEYS *를 프로덕션에서 쓰면 안 되는 이유
Redis는 싱글 스레드로 명령을 처리한다. KEYS는 전체 키스페이스를 한 번에 스캔하는 O(N) 블로킹 명령이라, 키가 수십만 개만 넘어가도 그 순간 다른 모든 클라이언트의 요청이 밀린다. 항상 SCAN(또는 클라이언트 라이브러리의 scan_iter)으로 커서 기반 non-blocking 순회를 써야 한다.
무효화 트리거적용 예
쓰기 API 내부에서 직접 삭제 단일 서비스, 단일 DB 안에서 가장 예측 가능하고 흔한 방식 (위 2절 예시)
이벤트 기반 무효화 다른 서비스가 소유한 데이터가 바뀌었을 때 — pub/sub 또는 메시지 큐(Day 7)로 무효화 이벤트 전파
TTL만 사용 (명시적 무효화 없음) 실시간성이 크게 중요하지 않은 대시보드 집계, 추천 목록 등 — 구현 단순성과 정합성을 맞바꿈

4. 캐시 스탬피드(Cache Stampede) — 캐시가 만드는 새로운 장애

인기 있는 키 하나의 TTL이 만료되는 순간, 그 키를 찾던 수백 개의 동시 요청이 전부 캐시 미스로 판정되어 동시에 DB를 때린다. 평소엔 캐시가 막아주던 트래픽이 한 번에 몰려서 DB가 순간적으로 과부하 상태에 빠진다 — 이를 Thundering Herd 또는 캐시 스탬피드라 부른다. Day 7에서 다룬 멱등성 문제와 본질적으로 같은 "동시성 사고" 이지만, 여기서는 원인이 캐시 만료라는 점이 다르다.

캐시 스탬피드가 발생하는 타임라인: t=0s "인기 상품 A" 캐시 TTL 만료 t=0.01s 동시 요청 500개가 캐시를 조회 → 전부 MISS t=0.02s 500개 요청이 동시에 DB에 "상품 A 조회" 쿼리 실행 t=0.05s DB 커넥션 풀 고갈 (아래 6절) → 다른 상품 조회 요청까지 타임아웃 시작 → 인기 상품 하나의 만료가 전체 서비스 장애로 번짐 
# infrastructure/cache/stampede_lock.py — Redis 분산 락으로 "한 프로세스만 DB를 조회"하게 제한
async def get_product_with_lock(product_id: int, redis: Redis, db: AsyncSession) -> Product:
    cache_key = f"product:v2:{product_id}"
    lock_key  = f"lock:product:{product_id}"

    cached = await redis.get(cache_key)
    if cached is not None:
        return Product.model_validate_json(cached)

    # SET NX EX — 락 획득을 원자적으로 시도 (락 획득 실패 = 다른 요청이 이미 채우는 중)
    acquired = await redis.set(lock_key, "1", nx=True, ex=10)
    if not acquired:
        await asyncio.sleep(0.05)                      # 짧게 대기 후 캐시 재확인 (다른 요청이 채웠을 것)
        cached = await redis.get(cache_key)
        if cached is not None:
            return Product.model_validate_json(cached)
        return await get_product_with_lock(product_id, redis, db)  # 재시도 (실무에서는 최대 재시도 횟수 제한)

    try:
        product = await product_repository.get_by_id(db, product_id)
        await redis.set(cache_key, product.model_dump_json(), ex=300)
        return product
    finally:
        await redis.delete(lock_key)                   # 락 해제 — 다음 만료 시 다시 한 요청만 통과
더 단순한 대안 — Probabilistic Early Expiration
분산 락 없이도 스탬피드를 크게 줄일 수 있다. TTL이 임박할수록 일부 요청만 확률적으로 미리 캐시를 갱신하게 하면(예: 남은 TTL이 짧을수록 갱신 확률을 높임), 만료 시점이 한 순간에 몰리지 않고 분산된다. 구현이 더 단순해 트래픽이 아주 크지 않다면 락보다 이 방식을 먼저 고려한다.

5. 응답 압축 — 대역폭과 CPU의 trade-off

JSON 응답이 크면(목록 API, 중첩된 객체) 압축으로 전송 바이트를 크게 줄일 수 있다. 단, 압축은 공짜가 아니라 CPU를 대역폭과 교환하는 것이므로 무조건 켜는 것이 능사는 아니다. 

# main.py
from fastapi.middleware.gzip import GZipMiddleware

app.add_middleware(GZipMiddleware, minimum_size=1000)   # 1KB 미만 응답은 압축 오버헤드가 이득보다 큼
고려 사항실무 판단 기준
최소 크기 임계값 수백 바이트짜리 작은 응답까지 압축하면 압축 자체의 CPU 비용이 절감된 전송량보다 클 수 있음
이미 압축된 데이터 이미지, 동영상 등은 다시 압축해도 거의 줄지 않음 — 텍스트 기반(JSON, HTML) 응답에만 적용
gzip vs brotli brotli가 보통 더 높은 압축률이지만 CPU 비용도 더 큼 — CDN/리버스 프록시(Day 9) 레벨에서 처리하면 앱 서버의 CPU 부담을 줄일 수 있음
애플리케이션 vs 프록시 레벨 Nginx/CDN이 앞단에 있다면 그쪽에서 압축을 처리하는 것이 앱 서버 CPU를 아끼는 일반적인 선택

6. 커넥션 풀 튜닝 — DB/Redis 연결을 무한정 늘릴 수 없는 이유

캐시 스탬피드 사례에서 봤듯, 요청이 몰릴 때 진짜 병목은 종종 DB 연산 자체가 아니라 DB 커넥션을 확보하는 대기 시간이다. PostgreSQL 같은 DB는 커넥션 하나당 메모리를 소비하므로 무한정 늘릴 수 없고, 애플리케이션 쪽 풀 설정이 그 한도를 넘지 않게 조절해야 한다. 

# infrastructure/db/engine.py
engine = create_async_engine(
    settings.database_url,
    pool_size=10,          # 상시 유지할 커넥션 수 — 앱 인스턴스 수 × pool_size가 DB 최대 커넥션 이내여야 함
    max_overflow=5,         # 순간 트래픽 급증 시 추가로 허용할 임시 커넥션 수 (pool_size + max_overflow가 상한)
    pool_timeout=10,        # 풀이 가득 찼을 때 커넥션을 기다리는 최대 시간(초) — 초과 시 예외 발생
    pool_recycle=1800,      # 30분마다 커넥션 재생성 — DB/방화벽의 idle timeout보다 짧게 설정해 끊긴 연결 재사용 방지
    pool_pre_ping=True,     # 커넥션 사용 전 가벼운 ping으로 죽은 연결 감지 (재시작/네트워크 단절 대응)
)
"커넥션 풀을 늘리면 빨라진다"는 흔한 오해
앱 인스턴스 10대가 각각 pool_size=50을 쓰면 최대 500개 커넥션을 시도할 수 있는데, PostgreSQL의 기본 max_connections은 보통 100 근처다. 풀을 무작정 키우면 DB가 거부하는 연결이 늘거나, DB 자체가 연결 수 증가로 메모리 부족에 빠진다. 풀 크기는 (DB의 max_connections − 관리용 예약분) ÷ 앱 인스턴스 수를 상한으로 계산하고, 그래도 대기가 잦다면 커넥션 자체를 늘리기보다 PgBouncer 같은 외부 커넥션 풀러로 다수 앱 인스턴스의 연결을 집약하는 것이 먼저다.

Redis 연결도 마찬가지로 재사용 가능한 풀을 명시적으로 구성해야 매 요청마다 TCP 핸드셰이크가 발생하지 않는다.

# infrastructure/cache/redis_client.py
redis_pool = redis.asyncio.ConnectionPool.from_url(
    settings.redis_url,
    max_connections=20,
    decode_responses=True,
)
redis_client = redis.asyncio.Redis(connection_pool=redis_pool)

7. "느리다"를 추측이 아니라 측정으로 — 쿼리 프로파일링

캐싱, 압축, 풀 튜닝 모두 실제 병목을 먼저 특정한 다음에 적용해야 효과가 있다. 추측으로 최적화하면 코드만 복잡해지고 체감 개선은 없는 경우가 많다. 

-- Postgres EXPLAIN ANALYZE — 계획과 실제 실행 시간을 함께 확인
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.id, o.total, c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.created_at > now() - interval '7 days';

-- 확인할 것:
-- 1) Seq Scan이 나오면서 실행 시간이 크면 인덱스 누락 가능성
-- 2) 예상 row 수(estimated)와 실제 row 수(actual)가 크게 다르면 통계 정보 갱신 필요 (ANALYZE table;)
-- 3) Buffers: shared hit/read 비율 — read가 많으면 디스크 I/O가 병목

# infrastructure/db/slow_query_logger.py — SQLAlchemy 이벤트로 슬로우 쿼리 자동 로깅
from sqlalchemy import event
from time import perf_counter

@event.listens_for(engine.sync_engine, "before_cursor_execute")
def _before(conn, cursor, statement, parameters, context, executemany):
    context._query_start = perf_counter()

@event.listens_for(engine.sync_engine, "after_cursor_execute")
def _after(conn, cursor, statement, parameters, context, executemany):
    duration_ms = (perf_counter() - context._query_start) * 1000
    if duration_ms > 200:                                    # 임계값 이상만 로깅 — 모든 쿼리를 남기면 로그가 압도됨
        logger.warning("slow_query", duration_ms=round(duration_ms, 1), statement=statement[:300])
도구용도
EXPLAIN ANALYZE 단일 쿼리의 실행 계획과 실제 소요 시간 확인 — 인덱스 누락, 잘못된 조인 순서 발견
SQLAlchemy 슬로우 쿼리 로거 애플리케이션 전체에서 임계값을 넘는 쿼리를 상시 감시 (Day 8 구조화 로깅과 연결)
py-spy 실행 중인 프로세스에 코드 수정 없이 attach해 CPU 프로파일링 — 프로덕션에서도 안전하게 사용 가능
cProfile / line_profiler 로컬 개발 환경에서 함수/라인 단위 CPU 시간 측정 — Python 코드 자체가 병목인지 확인
최적화 순서 — 항상 이 순서를 지킨다
(1) Day 8의 메트릭/트레이싱으로 어느 엔드포인트가 느린지 특정 → (2) 그 안에서 DB 쿼리인지, 캐시 미스인지, CPU 연산인지 EXPLAIN/프로파일러로 좁힘 → (3) 원인에 맞는 처방 (인덱스, 캐싱, 알고리즘 개선) 적용 → (4) 같은 메트릭으로 개선 효과를 다시 측정. "느낌"으로 캐싱부터 넣는 순서를 거꾸로 밟으면 원인 없는 복잡도만 늘어난다.

8. 캐싱/성능 작업 전 셀프 리뷰 체크리스트

캐싱을 추가하거나 성능 튜닝을 하기 전에 확인할 것
  • 메트릭/프로파일링으로 실제 병목을 특정했다 (추측으로 캐싱부터 넣지 않았다)
  • 캐싱 대상 데이터의 정합성 요구 수준(즉시 반영 필요 여부)을 확인했다
  • 쓰기 경로에서 캐시를 "삭제"하는지 "갱신"하는지 의도적으로 선택했다
  • 캐시 키에 버전을 포함해 스키마 변경 시 구버전 캐시가 자연 소멸하게 했다
  • KEYS * 대신 SCAN으로 패턴 삭제를 구현했다
  • 인기 키의 동시 만료로 인한 캐시 스탬피드 대응(락 또는 확률적 조기 갱신)을 고려했다
  • 응답 압축의 최소 크기 임계값을 설정해 작은 응답까지 압축하지 않는다
  • 커넥션 풀 크기를 (DB 최대 커넥션 ÷ 앱 인스턴스 수) 기준으로 계산했다
  • 슬로우 쿼리 로깅 또는 APM으로 회귀(regression)를 지속적으로 감시한다

🧠 이해도 점검

Q1. Cache-Aside 전략에서 쓰기 시 캐시를 새 값으로 "갱신"하지 않고 "삭제"만 하는 이유는?

ARedis는 값을 직접 갱신하는 명령을 지원하지 않기 때문
B삭제가 갱신보다 항상 더 빠른 명령이기 때문
C동시 쓰기 순서가 엇갈려도, 삭제 후 다음 읽기가 DB 최신 값으로 다시 채우므로 최종적으로 캐시와 DB가 일치하게 되기 때문
DTTL이 설정된 키는 값을 변경할 수 없기 때문

Q2. 캐시 스탬피드(Cache Stampede)를 가장 정확히 설명한 것은?

A캐시 서버 자체의 메모리가 부족해 값이 삭제되는 현상
B인기 키의 TTL 만료 순간 다수의 동시 요청이 전부 캐시 미스로 판정되어 동시에 DB를 때리며 순간 과부하를 일으키는 현상
C캐시 키 이름이 중복되어 서로 다른 데이터를 덮어쓰는 현상
DRedis 클러스터의 슬롯 재분배 중 발생하는 일시적 연결 끊김

Q3. 프로덕션 Redis에서 패턴에 맞는 키를 찾을 때 KEYS *pattern* 대신 SCAN을 써야 하는 이유는?

AKEYS 명령은 최신 Redis 버전에서 완전히 제거되었기 때문
BKEYS는 대소문자를 구분하지 못해 잘못된 키까지 매칭하기 때문
CSCAN만 TTL이 설정된 키를 조회할 수 있기 때문
DKEYS는 싱글 스레드에서 전체 키스페이스를 한 번에 순회하는 O(N) 블로킹 명령이라 키가 많으면 다른 모든 요청을 지연시키기 때문

Q4. "DB 응답이 느리니 커넥션 풀(pool_size)을 무작정 늘리면 해결된다"는 접근이 위험한 이유는?

A앱 인스턴스 수 × pool_size가 DB의 max_connections를 넘으면 DB가 연결을 거부하거나 메모리 부족에 빠질 수 있기 때문
BSQLAlchemy가 pool_size 값을 무시하고 항상 기본값만 사용하기 때문
C커넥션 풀은 읽기 쿼리에만 적용되고 쓰기에는 영향을 주지 않기 때문
D풀 크기를 늘리면 자동으로 pool_recycle 값이 0으로 초기화되기 때문
