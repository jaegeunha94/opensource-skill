Day 8 실무자 기준 약 65분

관측성
구조화 로깅 · Prometheus 메트릭 · OpenTelemetry 트레이싱

Day 1~7까지 API, 데이터, 트랜잭션, 인증, 백그라운드 작업을 다뤘다. 이 시스템이 실제 프로덕션에서 돌아가기 시작하면 새로운 질문이 생긴다 — "지금 느린 요청은 어디서 막히고 있는가", "에러율이 갑자기 올랐는데 원인이 API인가 DB인가 워커인가", "이 사용자의 요청이 서비스 세 개를 거치는 동안 정확히 어디서 실패했는가". 코드를 눈으로 다시 읽는 것으로는 답할 수 없다. 이 레슨은 로그·메트릭·트레이스 세 가지 신호로 운영 중인 시스템의 내부 상태를 "질문에 답할 수 있는 형태"로 만드는 방법을 다룬다.

1. "로그는 있는데 쓸모가 없는" 흔한 상황

아래 코드는 실제로 배포되는 코드에서 흔히 보이는 로깅이다. 로그가 "있다"는 사실만으로는 운영 중 장애를 진단하기에 충분하지 않다. 

# 절대 이렇게 하지 말 것 — 문제가 있는 예시
@router.post("/orders")
async def create_order(payload: OrderCreate, order_service: OrderService = Depends(...)):
    print(f"주문 생성 시작: {payload}")          # ← 프로덕션에서 stdout에 흩뿌려지고 검색 불가
    try:
        order = await order_service.create(payload)
        print("주문 생성 성공")                   # ← 어떤 주문인지, 어떤 사용자인지 알 수 없음
        return order
    except Exception as e:
        print(f"에러 발생: {e}")                  # ← 요청 ID도, 스택 트레이스도, 심각도도 없음
        raise
이 로그가 실전에서 쓸모없어지는 이유
  • 수백 대 서버에서 초당 수천 건씩 쌓이는 로그를 grep으로 뒤지려면 "이 요청" 하나를 특정할 수 있는 식별자가 있어야 하는데, 위 로그에는 없다
  • 같은 요청이 API → Service → DB → 외부 결제 API를 거치는데, 어느 로그가 같은 요청에 속하는지 연결할 방법이 없다 (분산 추적 불가)
  • "에러율이 5분 전보다 3배 늘었다"는 걸 알려면 로그를 세는 것이 아니라 시계열 수치(메트릭)가 따로 필요한데, 로그만으로는 알림을 걸기 어렵다
  • payload를 그대로 찍으면 비밀번호, 카드번호 같은 민감 정보가 로그 저장소에 평문으로 남는다

관측성(Observability)은 "로그를 많이 남긴다"가 아니라, 로그·메트릭·트레이스 세 신호를 서로 연결 가능한 구조로 만들어 "무슨 일이 일어났는가"라는 질문에 사전에 예상하지 못한 형태로도 답할 수 있게 하는 것이다.

신호답하는 질문대표 도구
구조화 로그 이 요청/사용자/주문에 정확히 무슨 일이 있었는가? structlog, JSON 로거 + Loki/ELK
메트릭 전체 시스템의 추세는 어떤가? (지금 이상 상태인가?) Prometheus + Grafana
트레이스 이 요청이 여러 서비스를 거치는 동안 어디서 느려지거나 실패했는가? OpenTelemetry + Jaeger/Tempo

2. 구조화 로깅 — 사람이 아니라 검색 엔진이 읽는 로그

구조화 로깅은 로그를 자연어 문장이 아니라 키-값(JSON)으로 남기는 것이다. 사람은 문장을 더 편하게 읽지만, 로그 수집 시스템(Loki, Elasticsearch)은 필드로 검색하고 필터링해야 하므로 구조화된 형태가 훨씬 유용하다. 

# infrastructure/logging/setup.py
import logging
import structlog

def configure_logging() -> None:
    structlog.configure(
        processors=[
            structlog.contextvars.merge_contextvars,   # 요청 컨텍스트 자동 병합 (Section 3)
            structlog.processors.TimeStamper(fmt="iso"),
            structlog.processors.add_log_level,
            structlog.processors.StackInfoRenderer(),
            structlog.processors.format_exc_info,       # 예외를 구조화된 필드로
            structlog.processors.JSONRenderer(),         # 최종 출력을 JSON으로
        ],
        wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
        logger_factory=structlog.PrintLoggerFactory(),
        cache_logger_on_first_use=True,
    )

logger = structlog.get_logger()

# presentation/routers/order_router.py
@router.post("/orders", status_code=201)
async def create_order(payload: OrderCreate, order_service: OrderService = Depends(...)):
    log = logger.bind(user_id=payload.user_id, item_count=len(payload.items))
    log.info("order_create_started")
    try:
        order = await order_service.create(payload)
    except InsufficientStockError as e:
        log.warning("order_create_failed", reason="insufficient_stock", item_id=e.item_id)
        raise
    log.info("order_create_succeeded", order_id=order.id, total_amount=order.total_amount)
    return order

# 실제 출력 (한 줄 JSON):
# {"event": "order_create_succeeded", "user_id": 42, "item_count": 3,
#  "order_id": 981, "total_amount": 45000, "level": "info", "timestamp": "2026-07-06T02:11:03Z"}
왜 메시지를 문장 대신 이벤트 이름으로 쓰는가
"order_create_succeeded"처럼 고정된 이벤트 이름을 쓰면 로그 시스템에서 event:"order_create_succeeded"로 정확히 필터링할 수 있다. 반면 f"주문 {order.id}번이 생성되었습니다"처럼 값이 메시지 문자열 안에 섞이면 검색 시 매번 정규식을 써야 하고, 값별 집계(예: "실패 사유별 카운트")가 불가능하다. 변하는 값은 필드로 분리하고, 메시지는 고정 문자열로 유지한다.

3. 요청 컨텍스트 — 흩어진 로그를 하나의 요청으로 묶기

한 API 요청 안에서 로그가 여러 줄 찍힐 때, 그 로그들이 "같은 요청"에 속한다는 것을 알아야 장애를 추적할 수 있다. 이를 위해 요청마다 고유한 request_id(또는 correlation ID)를 발급하고, 그 요청 안의 모든 로그에 자동으로 붙인다. 

# infrastructure/middleware/request_context.py
import uuid
import structlog
from starlette.middleware.base import BaseHTTPMiddleware

class RequestContextMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        request_id = request.headers.get("x-request-id", str(uuid.uuid4()))
        structlog.contextvars.clear_contextvars()
        structlog.contextvars.bind_contextvars(
            request_id=request_id,
            path=request.url.path,
            method=request.method,
        )
        response = await call_next(request)
        response.headers["x-request-id"] = request_id   # 클라이언트/게이트웨이도 같은 ID 사용
        return response

# main.py
app.add_middleware(RequestContextMiddleware)

이렇게 하면 logger.bind()를 매번 호출하지 않아도, 미들웨어가 설정한 contextvars가 Section 2의 merge_contextvars 프로세서를 통해 그 요청 안의 모든 로그에 자동으로 request_id가 붙는다. 장애 발생 시 하나의 request_id로 그 요청의 전체 로그를 시간 순서대로 모을 수 있다.

마이크로서비스 환경에서는 request_id를 전파해야 한다
API 서버가 다른 내부 서비스를 호출할 때 x-request-id 헤더를 그대로 전달하지 않으면, 서비스마다 새로운 ID가 생겨 로그를 연결할 수 없다. 서비스 간 호출 시 반드시 들어온 request_id를 다음 호출의 헤더로 그대로 전파해야 한다. (완전한 해결은 Section 5의 분산 트레이싱 — trace_id가 이 역할을 표준화한다.)

4. Prometheus 메트릭 — "지금 시스템이 평소와 다른가"에 답하기

로그는 개별 사건을 기록하지만, "에러율이 올랐다", "P99 응답 시간이 늘었다" 같은 추세는 매 요청마다 로그를 세는 것으로 실시간 판단하기 어렵다. 메트릭은 숫자를 시계열로 누적하고, Prometheus가 주기적으로 그 값을 수집(scrape)해 추세와 임계값 기반 알림을 가능하게 한다.

메트릭 타입용도예시
Counter 계속 증가만 하는 누적값 (재시작 전까지 감소하지 않음) 총 요청 수, 총 에러 수
Gauge 오르내릴 수 있는 현재 값 현재 활성 연결 수, 큐에 쌓인 태스크 수
Histogram 값의 분포 — 구간(bucket)별 개수를 누적, 퍼센타일 계산 가능 응답 시간 분포 (P50/P95/P99) 
# infrastructure/metrics/setup.py
from prometheus_client import Counter, Histogram, make_asgi_app

REQUEST_COUNT = Counter(
    "http_requests_total", "총 HTTP 요청 수",
    ["method", "path", "status_code"],          # 라벨 — 차원별로 나눠 조회 가능
)
REQUEST_LATENCY = Histogram(
    "http_request_duration_seconds", "HTTP 요청 처리 시간(초)",
    ["method", "path"],
    buckets=(0.01, 0.05, 0.1, 0.3, 0.5, 1.0, 2.0, 5.0),   # 서비스 SLO에 맞춘 구간
)

# main.py
app.mount("/metrics", make_asgi_app())   # Prometheus가 이 엔드포인트를 주기적으로 scrape

# infrastructure/middleware/metrics_middleware.py
import time

class MetricsMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.perf_counter()
        response = await call_next(request)
        duration = time.perf_counter() - start

        path = request.scope.get("route").path if request.scope.get("route") else "unmatched"
        REQUEST_COUNT.labels(request.method, path, response.status_code).inc()
        REQUEST_LATENCY.labels(request.method, path).observe(duration)
        return response
라벨에 고유값(cardinality 폭발)을 넣으면 안 되는 이유
path 라벨에 실제 URL을 그대로 넣으면(/orders/981, /orders/982...) 주문마다 새로운 시계열이 생겨 Prometheus 메모리가 폭발적으로 증가한다. 반드시 라우트 패턴(/orders/{order_id})을 라벨로 써야 한다. user_id, request_id, order_id처럼 값의 종류가 무한한 필드는 메트릭 라벨이 아니라 로그의 필드로 남긴다 — 이것이 메트릭과 로그의 역할이 갈리는 지점이다.
Prometheus의 Pull 모델: [FastAPI 앱] [Prometheus 서버] | /metrics 엔드포인트에 | | 현재까지 누적된 수치를 노출 | | | |<----------------------------------| 15초마다 스크레이핑(GET /metrics) | 텍스트 형식 메트릭 반환 --------->| | | 시계열 DB에 저장 | | Grafana가 쿼리해 대시보드/알림 생성

5. OpenTelemetry 트레이싱 — 요청이 여러 서비스를 거치는 경로 추적

API가 DB 쿼리, 외부 결제 API, Celery 태스크 큐를 순서대로 호출한다고 하자. 응답이 느려졌을 때 로그와 메트릭만으로는 "그 중 어느 단계가 느렸는지"를 정확히 알기 어렵다. 분산 트레이싱은 하나의 요청을 trace로 묶고, 그 안의 각 처리 단계를 span으로 기록해 시간 순서와 부모-자식 관계를 시각화한다. 

# infrastructure/tracing/setup.py
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.sqlalchemy import SQLAlchemyInstrumentor

def configure_tracing(app, engine) -> None:
    provider = TracerProvider()
    provider.add_span_processor(
        BatchSpanProcessor(OTLPSpanExporter(endpoint="otel-collector:4317", insecure=True))
    )
    trace.set_tracer_provider(provider)

    FastAPIInstrumentor.instrument_app(app)          # HTTP 요청마다 자동으로 span 생성
    SQLAlchemyInstrumentor().instrument(engine=engine)  # DB 쿼리마다 자동으로 span 생성

tracer = trace.get_tracer(__name__)

# application/services/order_service.py — 자동 계측으로 부족한 부분은 수동 span 추가
class OrderService:
    async def create(self, payload: OrderCreate) -> Order:
        with tracer.start_as_current_span("order_service.create") as span:
            span.set_attribute("user_id", payload.user_id)
            span.set_attribute("item_count", len(payload.items))

            order = await self._order_repo.create(payload)

            with tracer.start_as_current_span("payment_gateway.charge"):
                await self._payment_gateway.charge(order.id, order.total_amount)

            span.set_attribute("order_id", order.id)
            return order
하나의 trace 안에 있는 span들의 부모-자식 관계 (Jaeger/Tempo에서 보는 형태): trace_id: 7a3f... 총 소요 시간 420ms └─ HTTP POST /orders [420ms] ← FastAPIInstrumentor가 자동 생성 └─ order_service.create [410ms] ← 수동으로 추가한 span ├─ SELECT ... FROM items WHERE ... [15ms] ← SQLAlchemyInstrumentor가 자동 생성 ├─ INSERT INTO orders ... [20ms] └─ payment_gateway.charge [360ms] ← 병목! 전체 시간의 86%가 여기 └─ HTTP POST https://payment.../charge [355ms]
trace_id가 request_id를 대체하는가
완전히 대체한다. OpenTelemetry의 trace_id는 서비스 경계를 넘어 W3C traceparent 헤더로 자동 전파되므로, Section 3에서 직접 구현한 request_id 전파를 표준화된 방식으로 대신한다. 실무에서는 둘을 병행하는 경우도 많다 — 로그에 trace_id를 필드로 남기면(logger.bind(trace_id=...)) 로그와 트레이스를 서로 클릭해서 오갈 수 있는 관측성 스택(Grafana의 로그↔트레이스 연동 등)을 구성할 수 있다.

6. 메트릭을 알림으로 연결하기 — SLO 기반 사고

메트릭을 대시보드에 그리는 것만으로는 사람이 계속 화면을 보고 있어야 한다. 실무에서는 SLO(Service Level Objective)를 정의하고, 그 기준을 벗어나면 자동으로 알림이 오도록 구성한다. 

# Prometheus 알림 규칙 예시 (alerting_rules.yml)
groups:
  - name: order-service-slo
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_requests_total{status_code=~"5.."}[5m]))
          / sum(rate(http_requests_total[5m])) > 0.01
        for: 5m
        annotations:
          summary: "5xx 에러율이 5분간 1%를 초과함"

      - alert: HighLatencyP99
        expr: |
          histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 1.0
        for: 10m
        annotations:
          summary: "P99 응답 시간이 10분간 1초를 초과함"
개념의미실무 예
SLI (Indicator) 실제로 측정하는 지표 5xx 응답 비율, P99 지연 시간
SLO (Objective) SLI가 지켜야 할 목표치 "에러율 < 0.1%", "P99 < 300ms"
Error Budget SLO를 위반해도 되는 허용량 월 99.9% 가용성 → 월 약 43분까지 장애 허용
알림 피로(Alert Fatigue)를 피하는 것도 설계의 일부다
임계값을 넘는 즉시 알리면 순간적인 스파이크에도 알림이 울려 팀이 알림을 무시하게 된다. for: 5m처럼 "일정 시간 이상 지속될 때만" 알리는 조건을 반드시 넣는다. 또한 사용자가 직접 체감하는 지표(에러율, 지연 시간)에는 강하게 알리고, 원인 진단용 보조 지표(CPU 사용률 등)는 대시보드에만 남기고 알림을 걸지 않는 것이 실무에서 선호되는 방식이다.

7. 관측성 셀프 리뷰 체크리스트

코드 리뷰 시 확인할 것
  • 로그를 JSON 구조화 형태로 남기고, 값은 메시지 문자열이 아니라 필드로 분리했다
  • 비밀번호, 카드번호, 토큰 같은 민감 정보는 로그에 절대 남기지 않는다 (마스킹 처리)
  • 요청마다 request_id(또는 trace_id)를 발급하고, 서비스 간 호출 시 전파한다
  • 메트릭 라벨에 무한히 늘어나는 값(user_id, order_id 등)을 넣지 않았다 (cardinality 폭발 방지)
  • 응답 시간은 평균이 아니라 Histogram으로 P50/P95/P99를 볼 수 있게 수집한다
  • 서비스 경계를 넘는 호출(DB, 외부 API, 큐)에 트레이싱 span이 걸려 있다
  • SLO를 정의하고, 순간 스파이크가 아니라 일정 시간 지속될 때만 알림이 울리도록 설정했다
  • 알림에는 담당 서비스, 심각도, 다음 행동(runbook 링크)이 포함되어 있다

Q1. 구조화 로깅에서 "이벤트 이름은 고정하고 변하는 값은 필드로 분리한다"는 원칙이 중요한 이유는?

A파일 용량을 줄이기 위해서
BPython 로깅 모듈이 문자열 포매팅을 지원하지 않기 때문
C로그 수집 시스템에서 필드 기준 검색과 값별 집계가 가능해지기 때문
D구조화된 로그만 콘솔에 출력할 수 있기 때문

Q2. Prometheus 메트릭의 라벨에 user_id나 order_id 같은 고유 식별자를 넣으면 안 되는 이유는?

APrometheus가 문자열 라벨을 지원하지 않기 때문
B값 조합마다 새로운 시계열이 생성되어 메모리 사용량이 폭발적으로 증가하기 때문(cardinality 폭발)
C고유 식별자는 항상 보안상 노출되면 안 되기 때문
DHistogram 타입에서만 라벨 개수 제한이 있기 때문

Q3. 로그와 메트릭이 이미 있는데도 OpenTelemetry 분산 트레이싱이 추가로 필요한 이유로 가장 정확한 것은?

A트레이싱이 로그보다 저장 비용이 저렴하기 때문
B메트릭은 텍스트 검색이 불가능하기 때문
C구조화 로깅을 사용하면 트레이싱은 필요 없기 때문
D하나의 요청이 여러 서비스를 거치는 동안 정확히 어느 단계가 병목인지, span 간 부모-자식 관계와 소요 시간으로 특정할 수 있기 때문

Q4. Prometheus 알림 규칙에서 임계값 초과 즉시가 아니라 for: 5m처럼 지속 시간 조건을 두는 이유는?

A순간적인 스파이크에도 알림이 울려 팀이 알림을 무시하게 되는 alert fatigue를 방지하기 위해서
BPrometheus가 즉시 알림을 기술적으로 지원하지 않기 때문
C지속 시간 조건이 없으면 메트릭 수집 자체가 중단되기 때문
DError Budget을 자동으로 늘리기 위해서
