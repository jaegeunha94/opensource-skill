Day 9 실무자 기준 약 70분

배포와 운영
Docker 최적화 · Compose · 헬스체크 · 환경 변수 · CI/CD

Day 1~8에서 API, 데이터, 서비스 경계, 트랜잭션, 인증, 백그라운드 작업, 관측성을 다뤘다. 이 모든 코드가 "정확히 동작한다"는 것과 "프로덕션에 안전하게 올라간다"는 것은 다른 문제다. 코드는 완벽해도 이미지 빌드가 느리고, 배포 중 요청이 끊기고, 시크릿이 이미지에 새어 나가고, 배포와 마이그레이션 순서가 어긋나면 장애가 난다. 이 레슨은 "동작하는 코드"를 "안전하게 반복 배포 가능한 서비스"로 만드는 마지막 단계 — 컨테이너화, 로컬 재현 환경, 헬스체크, 설정 관리, CI/CD 파이프라인을 다룬다.

1. "제 컴퓨터에서는 됩니다"가 배포에서 무너지는 지점

아래는 실무에서 흔히 보이는, 겉보기엔 문제없어 보이는 배포 구성이다. 

# 절대 이렇게 하지 말 것 — 문제가 있는 예시
FROM python:3.12
COPY . /app
WORKDIR /app
RUN pip install -r requirements.txt
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]
이 Dockerfile이 프로덕션에서 문제가 되는 이유
  • python:3.12 풀 이미지는 900MB 이상 — 빌드/배포마다 네트워크와 시간 낭비
  • COPY . /app을 먼저 하면 소스 코드 한 줄만 바뀌어도 의존성 설치 레이어부터 다시 캐시가 깨져 매번 전체 재설치
  • root 유저로 프로세스가 실행되어, 컨테이너가 뚫리면 호스트 권한 상승 위험이 커짐
  • --host 0.0.0.0만 있고 워커 수, 헬스체크, graceful shutdown 처리가 없어 배포 중 요청이 강제로 끊길 수 있음
  • 환경 변수, DB 접속 정보, 시크릿이 이미지 어디에도 명시되지 않아 팀마다 다르게 주입 — 재현 불가능한 배포

이번 레슨은 이 다섯 가지 문제를 하나씩 실무 패턴으로 교체한다.

2. Dockerfile 최적화 — Multi-stage Build와 레이어 캐싱

이미지 크기와 빌드 속도는 배포 빈도에 직접 영향을 준다. 핵심 원칙은 두 가지다 — 변하지 않는 것을 먼저 COPY해서 캐시를 최대한 재사용하고, 빌드 도구와 런타임을 분리해 최종 이미지에는 실행에 필요한 것만 남긴다. 

# Dockerfile
# ── Stage 1: 의존성 빌드 ─────────────────────────
FROM python:3.12-slim AS builder

WORKDIR /app
RUN pip install --no-cache-dir poetry==1.8.3

# 소스 코드보다 먼저 의존성 정의만 복사 — requirements가 안 바뀌면 이 레이어는 캐시 재사용
COPY pyproject.toml poetry.lock ./
RUN poetry export -f requirements.txt --without-hashes -o requirements.txt \
    && pip install --no-cache-dir --user -r requirements.txt

# ── Stage 2: 런타임 이미지 ───────────────────────
FROM python:3.12-slim AS runtime

# 빌드 도구(poetry, gcc 등)는 최종 이미지에 포함되지 않음 — 이미지 크기와 공격 표면 축소
COPY --from=builder /root/.local /root/.local
ENV PATH=/root/.local/bin:$PATH

# 비root 유저 생성 — 컨테이너 탈취 시 권한 상승 방지
RUN useradd --create-home --uid 1000 appuser
WORKDIR /app
COPY --chown=appuser:appuser . .
USER appuser

EXPOSE 8000
HEALTHCHECK --interval=10s --timeout=3s --start-period=5s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health/live')" || exit 1

CMD ["gunicorn", "main:app", \
     "--workers", "4", \
     "--worker-class", "uvicorn.workers.UvicornWorker", \
     "--bind", "0.0.0.0:8000", \
     "--graceful-timeout", "30"]

# .dockerignore — 빌드 컨텍스트에서 불필요한 파일 제외 (캐시 효율 + 이미지 크기)
.git
.venv
__pycache__
*.pyc
tests/
.env
*.md
기법효과
Multi-stage build 빌드 도구(컴파일러, poetry 등)를 최종 이미지에서 제거 — 이미지 크기 및 CVE 표면 축소
의존성 파일을 소스보다 먼저 COPY 소스 코드만 바뀐 배포에서는 pip install 레이어가 캐시되어 빌드 시간 급감
python:3.12-slim 풀 이미지 대비 크기 대폭 축소 (필요 시 -alpine은 glibc 비호환 이슈로 신중히 선택)
비root 유저(USER appuser) 컨테이너 탈취 시에도 호스트 권한 상승 경로 차단
.dockerignore .git, 테스트, .env 등이 빌드 컨텍스트/이미지에 포함되지 않음
gunicorn + uvicorn worker를 함께 쓰는 이유
uvicorn 단독 실행은 단일 프로세스라 CPU 코어를 하나만 쓴다. gunicorn이 여러 UvicornWorker 프로세스를 관리하게 하면 멀티코어를 활용하면서, 워커가 죽었을 때 gunicorn이 자동으로 재기동해준다. 워커 개수는 보통 (2 × CPU 코어 수) + 1을 시작점으로 삼고 부하 테스트로 조정한다.

3. Docker Compose — 로컬에서 프로덕션과 같은 구성 재현하기

API 서버 하나만 띄워서는 실제 서비스를 재현할 수 없다. Postgres, Redis, Celery 워커까지 포함한 전체 스택을 로컬에서 동일하게 띄워야 "로컬에서는 됨" 문제를 줄일 수 있다. 

# docker-compose.yml
services:
  api:
    build: .
    ports: ["8000:8000"]
    environment:
      DATABASE_URL: postgresql+asyncpg://app:app@db:5432/appdb
      REDIS_URL: redis://redis:6379/0
    env_file: .env
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "python", "-c",
             "import urllib.request; urllib.request.urlopen('http://localhost:8000/health/ready')"]
      interval: 10s
      timeout: 3s
      retries: 3

  worker:
    build: .
    command: celery -A app.worker worker --loglevel=info
    environment:
      DATABASE_URL: postgresql+asyncpg://app:app@db:5432/appdb
      REDIS_URL: redis://redis:6379/0
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app"]
      interval: 5s
      timeout: 3s
      retries: 5

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  pgdata:
왜 depends_on에 condition: service_healthy가 필요한가
depends_on만 쓰면 Compose는 "컨테이너가 시작되었다"는 것만 보장하고, DB가 "연결을 받을 준비가 됐다"는 것은 보장하지 않는다. Postgres 컨테이너는 뜨자마자 초기화 스크립트를 실행하느라 몇 초간 연결을 거부할 수 있는데, 이때 API가 먼저 연결을 시도하면 첫 배포/재기동마다 연결 에러가 재현된다. condition: service_healthy는 각 서비스의 healthcheck가 통과할 때까지 의존 서비스의 기동을 미룬다.

4. 헬스체크와 Graceful Shutdown — 배포 중 요청을 끊기지 않게

오케스트레이터(Kubernetes, ECS, Compose)는 "이 컨테이너가 트래픽을 받을 준비가 됐는지", "이 컨테이너가 살아있는지"를 자체적으로 알 수 없다. 애플리케이션이 직접 답해야 한다. 이때 liveness와 readiness를 구분하지 않으면 흔한 장애가 생긴다.

체크 종류질문실패 시 동작
Liveness 이 프로세스가 살아있는가? (데드락, 무한 루프 등으로 멈추지 않았는가) 컨테이너를 강제 재시작
Readiness 지금 트래픽을 받아도 되는가? (DB/Redis 연결 등 의존성이 준비됐는가) 재시작하지 않고, 로드밸런서에서 트래픽만 제외 
# presentation/routers/health_router.py
router = APIRouter(prefix="/health", tags=["health"])

@router.get("/live")
async def liveness() -> dict:
    # 프로세스 자체가 응답 가능한지만 확인 — 외부 의존성 체크 금지
    return {"status": "alive"}

@router.get("/ready")
async def readiness(db: AsyncSession = Depends(get_db), redis: Redis = Depends(get_redis)) -> dict:
    # 실제 트래픽을 처리할 수 있는 상태인지 확인
    try:
        await db.execute(text("SELECT 1"))
        await redis.ping()
    except Exception:
        raise HTTPException(status_code=503, detail="dependency not ready")
    return {"status": "ready"}
liveness에 DB 체크를 넣으면 안 되는 이유
DB가 일시적으로 느려졌다고 해서 애플리케이션 프로세스 자체는 죽지 않았다. liveness에 DB 체크를 넣으면 DB 장애 시 오케스트레이터가 멀쩡한 컨테이너를 계속 재시작시켜 커넥션 풀을 새로 맺고 끊기를 반복하며 DB 부하를 오히려 가중시킨다("재시작 폭풍"). 의존성 상태는 readiness에서만 확인하고, liveness는 프로세스 생존 여부만 확인한다.

배포 시 컨테이너가 종료될 때도 처리 중인 요청을 끊으면 안 된다. 오케스트레이터는 종료 시 SIGTERM을 보내고 일정 유예 시간 후 SIGKILL로 강제 종료한다. 

# main.py — FastAPI lifespan으로 종료 시퀀스 제어
@asynccontextmanager
async def lifespan(app: FastAPI):
    yield
    # SIGTERM 수신 후 여기 도달 — 새 요청은 이미 로드밸런서에서 제외된 상태여야 함
    logger.info("shutdown_started")
    await engine.dispose()          # DB 커넥션 풀 정리
    await redis_client.aclose()     # Redis 연결 정리
    logger.info("shutdown_completed")

app = FastAPI(lifespan=lifespan)
안전한 배포 종료 순서: 1. 오케스트레이터가 이 인스턴스를 로드밸런서 대상에서 제외 (신규 트래픽 차단) 2. SIGTERM 전송 → 애플리케이션은 "새 요청은 안 받지만 처리 중인 요청은 끝까지" 처리 3. gunicorn --graceful-timeout 만큼 대기 (Section 2 예시: 30초) 4. 시간 내 처리 완료 → 정상 종료 / 초과 → SIGKILL 강제 종료 이 순서가 깨지면: 배포 중 요청이 중간에 끊기거나(5xx 급증), DB 커넥션이 정리되지 않고 좀비로 남는 문제가 생긴다.

5. 환경 변수와 시크릿 관리 — 12-Factor 원칙

같은 이미지가 로컬/스테이징/프로덕션에서 동일하게 동작해야 한다. 환경별 차이(DB 주소, 시크릿 키 등)는 이미지 안에 굽지 않고 실행 시점에 환경 변수로 주입한다 (12-Factor App의 Config 원칙). 

# infrastructure/config/settings.py
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", extra="ignore")

    database_url: str
    redis_url: str
    jwt_secret_key: str
    environment: str = "development"
    log_level: str = "INFO"

    @property
    def is_production(self) -> bool:
        return self.environment == "production"

settings = Settings()  # 누락된 필수 값은 앱 기동 시점에 즉시 에러로 실패 (fail fast)
환경주입 방식이유
로컬 개발 .env 파일 + env_file 빠른 반복, 단 .env는 반드시 .gitignore에 포함
CI CI 플랫폼의 encrypted secrets 레포에 시크릿을 두지 않고 실행 시점에만 주입
프로덕션 Secret Manager(AWS Secrets Manager, Vault 등) → 컨테이너 환경 변수 접근 감사(audit log), 로테이션, 최소 권한 원칙 적용 가능
이미지에 시크릿을 굽는 것이 왜 치명적인가
ENV JWT_SECRET_KEY=abc123처럼 Dockerfile에 값을 직접 쓰거나, 빌드 시 --build-arg로 시크릿을 넘기면 그 값이 이미지 레이어에 영구히 남는다. 이미지를 누구나 pull할 수 있는 레지스트리에 올리면 docker history만으로 과거 레이어의 값까지 노출될 수 있다. 시크릿은 반드시 런타임 환경 변수나 마운트된 시크릿 파일로만 주입한다.

6. CI/CD 파이프라인 — 테스트부터 배포까지 자동화

수동 배포는 사람이 실수하기 쉬운 단계(테스트 스킵, 잘못된 이미지 태그, 마이그레이션 누락)를 항상 포함한다. CI/CD는 이 단계를 코드로 고정해 매번 동일하게 반복한다. 

# .github/workflows/deploy.yml
name: build-and-deploy

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16-alpine
        env: { POSTGRES_PASSWORD: test, POSTGRES_DB: testdb }
        ports: ["5432:5432"]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt
      - run: pytest --cov=app --cov-fail-under=80    # 커버리지 미달 시 파이프라인 중단
      - run: ruff check .                              # 린트 실패도 배포 차단

  build-and-push:
    needs: test                                       # 테스트 통과해야만 이미지 빌드
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/build-push-action@v6
        with:
          push: true
          tags: registry.example.com/app:${{ github.sha }}   # latest가 아니라 커밋 SHA로 태깅
          cache-from: type=registry,ref=registry.example.com/app:buildcache
          cache-to: type=registry,ref=registry.example.com/app:buildcache,mode=max

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - name: Run DB migration                        # 마이그레이션은 앱 배포 "전에" 별도 실행
        run: |
          docker run --rm registry.example.com/app:${{ github.sha }} \
            alembic upgrade head
      - name: Rolling update
        run: ./deploy.sh registry.example.com/app:${{ github.sha }}
이미지 태그를 latest가 아니라 커밋 SHA로 고정하는 이유
latest는 시간이 지나면 가리키는 이미지가 계속 바뀌는 가변 태그다. 롤백할 때 "어제의 latest"를 다시 가져올 방법이 없다. 커밋 SHA로 태깅하면 특정 배포가 정확히 어떤 코드였는지 항상 추적 가능하고, 문제 발생 시 이전 SHA 태그로 즉시 롤백할 수 있다.

Day 3에서 다룬 무중단 마이그레이션 원칙(하위 호환 스키마 변경 → 배포 → 정리)이 배포 파이프라인 순서에도 그대로 적용된다. 마이그레이션은 새 애플리케이션 코드가 트래픽을 받기 전에 끝나 있어야 하며, 롤링 배포 중에는 신버전과 구버전 코드가 동시에 같은 DB 스키마를 봐야 하므로 파괴적 변경(컬럼 삭제, NOT NULL 즉시 추가 등)을 배포와 같은 단계에서 수행하면 안 된다.

안전한 배포 순서 (Day 3 무중단 마이그레이션 원칙과 연결): 1. CI: 테스트 + 린트 통과 2. 이미지 빌드 → 커밋 SHA로 태깅 → 레지스트리 푸시 3. 마이그레이션 실행 (하위 호환 변경만 — 구버전 코드도 여전히 정상 동작해야 함) 4. 롤링 배포 시작: [구버전 3대] → [신버전 1대, 구버전 2대] → [신버전 2대, 구버전 1대] → [신버전 3대] 각 단계마다 새 인스턴스의 readiness 통과를 확인한 뒤 다음 단계 진행 5. 배포 완료 후 문제 없음을 확인한 다음 릴리스에서 이전 스키마 정리(컬럼 삭제 등) 수행

7. 배포 전 셀프 리뷰 체크리스트

배포하기 전에 확인할 것
  • Dockerfile이 multi-stage build를 쓰고, 비root 유저로 실행된다
  • 의존성 설치 레이어가 소스 코드 COPY보다 먼저 와서 캐시를 재사용한다
  • .dockerignore에 .git, .env, 테스트 코드가 포함되어 있다
  • liveness와 readiness 엔드포인트가 분리되어 있고, liveness는 외부 의존성을 체크하지 않는다
  • SIGTERM 수신 시 처리 중인 요청을 끝까지 마치고 커넥션을 정리하는 graceful shutdown이 있다
  • 시크릿이 이미지나 레포에 하드코딩되지 않고 런타임에 주입된다
  • 이미지 태그가 latest가 아니라 커밋 SHA 등 불변 식별자로 고정된다
  • 마이그레이션이 하위 호환 방식으로 배포 전에 별도 실행되고, 파괴적 변경은 다음 릴리스로 분리된다
  • CI에서 테스트/린트가 실패하면 배포 파이프라인이 자동으로 중단된다
  • 롤백 절차(이전 SHA 태그로 재배포)가 문서화되어 있다

🧠 이해도 점검

Q1. Dockerfile에서 의존성 정의 파일(requirements.txt 등)을 소스 코드보다 먼저 COPY하는 이유는?

ADocker가 파일을 알파벳 순서로만 복사할 수 있기 때문
B소스 코드만 바뀐 배포에서 의존성 설치 레이어의 캐시를 재사용해 빌드 시간을 줄이기 위해서
Crequirements.txt가 없으면 이미지가 빌드되지 않기 때문
D보안 스캐너가 의존성 파일을 먼저 읽어야 하기 때문

Q2. liveness 체크에 DB 연결 확인을 포함시키면 안 되는 이유로 가장 정확한 것은?

Aliveness 엔드포인트는 인증이 필요 없어야 하기 때문
BDB 체크는 readiness보다 응답 속도가 느리기 때문
CDB 장애 시 오케스트레이터가 멀쩡한 프로세스까지 계속 강제 재시작시켜 DB 부하를 가중시키는 재시작 폭풍이 발생하기 때문
Dliveness는 HTTP가 아니라 TCP 체크만 지원하기 때문

Q3. 배포 이미지 태그를 latest 대신 커밋 SHA로 고정해야 하는 이유는?

Alatest는 가리키는 이미지가 계속 바뀌는 가변 태그라서, 특정 배포를 추적하거나 이전 상태로 정확히 롤백할 수 없기 때문
B레지스트리가 latest라는 이름을 예약어로 금지하기 때문
C커밋 SHA 태그만 압축 저장이 가능해서 용량이 줄어들기 때문
Dlatest 태그는 CI에서 자동 빌드를 지원하지 않기 때문

Q4. 롤링 배포 도중에 컬럼 삭제 같은 파괴적 마이그레이션을 함께 실행하면 안 되는 이유는?

AAlembic이 롤링 배포 환경을 기술적으로 인식하지 못하기 때문
B파괴적 마이그레이션은 항상 다운타임 없이 실행할 수 없기 때문
CCI/CD 파이프라인이 마이그레이션 단계를 병렬로 실행하지 못하기 때문
D배포 도중에는 신버전과 구버전 코드가 동시에 같은 스키마를 참조하는데, 구버전 코드가 삭제된 컬럼이나 위반된 제약조건 때문에 실패할 수 있기 때문
