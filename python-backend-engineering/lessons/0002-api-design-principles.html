Day 2 실무자 기준 약 50분

API 설계 원칙
REST · OpenAPI · FastAPI 실전

"API를 잘 만든다"는 것은 HTTP를 올바르게 쓰고, 클라이언트와 명확한 계약(contract)을 맺으며, 변경이 생겨도 소비자가 깨지지 않도록 진화시키는 능력이다. 이 레슨은 REST 원칙, 실무 FastAPI 패턴, OpenAPI 계약 관리까지 다룬다.

1. REST — 규칙이 아니라 설계 철학

REST(Representational State Transfer)는 Roy Fielding이 2000년 박사 논문에서 제안한 웹 아키텍처 스타일이다. "REST API"라 불리는 대부분의 서비스는 사실 REST-like HTTP API에 가깝다. 핵심은 자원(Resource) 중심의 URL 설계와 HTTP 동사의 올바른 의미론적 사용이다.

REST의 6가지 제약
  1. Stateless — 서버는 요청 간 클라이언트 상태를 저장하지 않는다.
  2. Client-Server — UI와 데이터 저장 관심사를 분리한다.
  3. Cacheable — 응답은 캐시 가능 여부를 명시해야 한다.
  4. Uniform Interface — URI로 자원을 식별, 표현으로 조작한다.
  5. Layered System — 클라이언트는 중간 레이어를 인식하지 않는다.
  6. Code on Demand (optional) — 실행 코드를 전달할 수 있다.

실무에서 가장 자주 위반되는 제약은 Stateless와 Uniform Interface다. 서버 세션에 클라이언트 상태를 두거나, URL에 동사를 넣으면(/getUserById) REST 계약이 무너진다.

2. HTTP 메서드 의미론 — 올바른 선택 기준

HTTP 메서드는 Safe(부수 효과 없음)와 Idempotent(같은 요청을 여러 번 보내도 결과가 동일) 라는 두 속성으로 분류된다. 이 속성을 지키지 않으면 클라이언트의 재시도 로직, 캐시, 프록시가 오동작한다.

메서드 Safe Idempotent 실무 사용 기준
GET ✅ ✅ 자원 조회. 본문 없음. 응답 캐시 가능.
POST ❌ ❌ 새 자원 생성 또는 복잡한 액션. 중복 생성 위험 → 멱등성 키 필요.
PUT ❌ ✅ 자원 전체 교체. 누락 필드는 기본값/null로 처리됨.
PATCH ❌ ⚠️ 구현 의존 자원 부분 수정. 변경할 필드만 전달. RFC 7396 Merge Patch 또는 JSON Patch.
DELETE ❌ ✅ 자원 삭제. 이미 없는 자원은 204 또는 404 반환(정책 통일 필요).
흔한 실수: POST로 모든 것을 처리하기
POST /getUser, POST /deleteOrder 패턴은 HTTP 캐시, 브라우저 히스토리, CDN, API Gateway의 라우팅 규칙 모두를 무력화한다. 액션이 자원 CRUD에 맞지 않는다면 POST /orders/{id}/cancel처럼 서브 리소스 + POST로 모델링한다.

3. URL 설계 원칙

3-1. 자원은 명사, 복수형으로

# 나쁜 예
GET  /getUsers
POST /createUser
DELETE /deleteUser/42

# 좋은 예
GET    /users          # 목록
POST   /users          # 생성
GET    /users/{id}     # 단건 조회
PUT    /users/{id}     # 전체 교체
PATCH  /users/{id}     # 부분 수정
DELETE /users/{id}     # 삭제

3-2. 계층 관계는 URL로 표현

GET  /users/{user_id}/orders          # 특정 사용자의 주문 목록
GET  /users/{user_id}/orders/{order_id}  # 특정 주문
POST /orders/{order_id}/items         # 주문에 아이템 추가

# 깊이가 3단계를 초과하면 독립 자원으로 분리
# 나쁜 예: /users/{uid}/orders/{oid}/items/{iid}/reviews
# 좋은 예: /reviews?order_item_id={iid}

3-3. API 버전 관리 전략

전략예시장단점
URL 경로 /v1/users 명확하고 캐시하기 쉬움. URL이 길어짐. 가장 흔히 쓰임.
쿼리 파라미터 /users?version=1 URL 변경 없음. 캐시 키 오염.
헤더 Accept: application/vnd.api.v1+json URL 깔끔. 테스트·디버깅 불편.

실무에서는 URL 경로 버전(/v1/, /v2/)이 가장 많이 쓰인다. FastAPI에서는 APIRouter를 prefix로 묶어 버전 분리를 자연스럽게 표현한다.

4. HTTP 상태 코드 — 실무 선택 기준

클라이언트가 상태 코드 하나로 다음에 뭘 해야 하는지 알 수 있어야 한다. 팀 내 정책을 통일하지 않으면 같은 에러에 200, 400, 422가 혼재한다.

코드의미실무 사용 예
200 OK성공GET 조회, PUT/PATCH 수정 성공
201 Created생성 성공POST로 자원 생성. Location 헤더에 새 URL 포함 권장
204 No Content성공, 본문 없음DELETE, 상태 변경 후 응답 불필요 시
400 Bad Request잘못된 요청비즈니스 규칙 위반, 도메인 에러
401 Unauthorized인증 필요토큰 없음 또는 만료
403 Forbidden인가 거부토큰 있지만 권한 없음
404 Not Found자원 없음ID로 찾았지만 없는 경우
409 Conflict상태 충돌이미 존재하는 이메일, 중복 예약
422 Unprocessable Entity유효성 검사 실패FastAPI 기본 Pydantic 검증 실패
429 Too Many RequestsRate limit 초과Retry-After 헤더 포함
500 Internal Server Error서버 오류예상치 못한 예외. 절대 상세 내부 정보 노출 금지
200 vs 422 — FastAPI의 기본 전략
FastAPI는 Pydantic 검증 실패 시 자동으로 422 Unprocessable Entity를 반환한다. 비즈니스 규칙 위반(잔액 부족, 중복 주문 등)은 개발자가 직접 HTTPException(status_code=400)을 발생시켜야 한다. 두 경우를 혼동하지 말 것.

5. FastAPI 라우터 실전 구조

5-1. 프로젝트 라우터 분리

# app/api/v1/routers/users.py
from fastapi import APIRouter, Depends, HTTPException, status
from app.schemas.user import UserCreate, UserResponse, UserUpdate
from app.services.user_service import UserService
from app.dependencies import get_user_service

router = APIRouter(prefix="/users", tags=["users"])

@router.post(
    "",
    response_model=UserResponse,
    status_code=status.HTTP_201_CREATED,
    summary="새 사용자 등록",
)
async def create_user(
    body: UserCreate,
    service: UserService = Depends(get_user_service),
) -> UserResponse:
    return await service.create(body)

@router.get("/{user_id}", response_model=UserResponse)
async def get_user(
    user_id: int,
    service: UserService = Depends(get_user_service),
) -> UserResponse:
    user = await service.get_by_id(user_id)
    if user is None:
        raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail="User not found")
    return user

@router.patch("/{user_id}", response_model=UserResponse)
async def update_user(
    user_id: int,
    body: UserUpdate,
    service: UserService = Depends(get_user_service),
) -> UserResponse:
    return await service.update(user_id, body)

@router.delete("/{user_id}", status_code=status.HTTP_204_NO_CONTENT)
async def delete_user(
    user_id: int,
    service: UserService = Depends(get_user_service),
) -> None:
    await service.delete(user_id)

6. Pydantic v2 스키마 — Request / Response 분리

가장 흔한 실수는 DB 모델을 API 응답 스키마로 직접 사용하는 것이다. DB 모델에 password_hash나 내부 플래그가 있다면 그대로 노출된다. Request 스키마, Response 스키마, DB 모델은 반드시 분리해야 한다. 

# app/schemas/user.py
from pydantic import BaseModel, EmailStr, Field, model_validator
from typing import Annotated

class UserBase(BaseModel):
    email: EmailStr
    name: Annotated[str, Field(min_length=2, max_length=100)]

class UserCreate(UserBase):
    """POST /users 요청 본문"""
    password: Annotated[str, Field(min_length=8)]

class UserUpdate(BaseModel):
    """PATCH /users/{id} — 변경할 필드만 포함, 모두 Optional"""
    name: Annotated[str | None, Field(min_length=2, max_length=100)] = None
    email: EmailStr | None = None

    @model_validator(mode="after")
    def at_least_one_field(self) -> "UserUpdate":
        if self.name is None and self.email is None:
            raise ValueError("변경할 필드를 하나 이상 입력하세요")
        return self

class UserResponse(UserBase):
    """GET, POST, PATCH 응답 — password는 절대 포함하지 않음"""
    id: int
    is_active: bool

    model_config = {"from_attributes": True}  # ORM 모델 호환
model_config = {"from_attributes": True}
Pydantic v2에서 SQLAlchemy ORM 객체를 Response 스키마로 변환하려면 from_attributes = True를 설정해야 한다. v1의 orm_mode = True와 동일한 역할이다.

6-1. 중첩 스키마와 선택적 관계

7. 쿼리 파라미터 설계 — 페이지네이션, 필터링, 정렬

# app/schemas/pagination.py
from pydantic import BaseModel, Field
from typing import Generic, TypeVar

T = TypeVar("T")

class PaginatedResponse(BaseModel, Generic[T]):
    items: list[T]
    total: int
    page: int
    page_size: int
    has_next: bool

# app/api/v1/routers/users.py
from fastapi import Query

@router.get("", response_model=PaginatedResponse[UserResponse])
async def list_users(
    page: int = Query(default=1, ge=1, description="페이지 번호"),
    page_size: int = Query(default=20, ge=1, le=100, description="페이지 크기"),
    search: str | None = Query(default=None, description="이름 또는 이메일 검색"),
    is_active: bool | None = Query(default=None, description="활성 사용자 필터"),
    service: UserService = Depends(get_user_service),
) -> PaginatedResponse[UserResponse]:
    return await service.list_users(page, page_size, search, is_active)
커서 기반 vs 오프셋 기반 페이지네이션

page=N&page_size=20 방식(오프셋 기반)은 구현이 쉽지만 데이터가 많아질수록 OFFSET이 느려지고, 데이터가 삽입/삭제되면 페이지가 어긋난다.

대량 데이터나 실시간 피드에는 커서 기반(after=eyJpZCI6MTAwfQ)이 적합하다. Base64로 인코딩된 마지막 항목 ID를 커서로 사용하면 안정적이다.

8. 에러 응답 표준화

API의 에러 형식이 엔드포인트마다 다르면 클라이언트가 파싱 로직을 여러 벌 만들어야 한다. RFC 9457(Problem Details for HTTP APIs)이나 자체 표준을 팀 전체에 통일해야 한다. 

# app/exceptions.py
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from pydantic import ValidationError

class AppError(Exception):
    def __init__(self, code: str, message: str, status_code: int = 400):
        self.code = code
        self.message = message
        self.status_code = status_code

def register_exception_handlers(app: FastAPI) -> None:

    @app.exception_handler(AppError)
    async def app_error_handler(request: Request, exc: AppError) -> JSONResponse:
        return JSONResponse(
            status_code=exc.status_code,
            content={"error": {"code": exc.code, "message": exc.message}},
        )

    @app.exception_handler(Exception)
    async def unhandled_error_handler(request: Request, exc: Exception) -> JSONResponse:
        # 내부 스택 트레이스는 절대 노출하지 않는다
        return JSONResponse(
            status_code=500,
            content={"error": {"code": "INTERNAL_ERROR", "message": "서버 오류가 발생했습니다"}},
        )

# 사용 예
class UserService:
    async def create(self, data: UserCreate) -> UserResponse:
        existing = await self.repo.find_by_email(data.email)
        if existing:
            raise AppError(
                code="EMAIL_ALREADY_EXISTS",
                message=f"{data.email}은 이미 등록된 이메일입니다",
                status_code=409,
            )
        ...

9-2. API 계약 버전 관리 — 하위 호환성

이미 배포된 API는 클라이언트가 있다. 하위 호환성을 깰 때는 반드시 버전을 올려야 한다.

변경 종류하위 호환?처리 방법
선택적 필드 추가 ✅ 호환 그냥 추가 (기본값 제공)
새 엔드포인트 추가 ✅ 호환 그냥 추가
필수 필드 추가 ❌ 파괴 /v2 버전 분리 후 /v1 Deprecate 알림
필드명 변경 ❌ 파괴 v2 분리. 과도기 동안 두 필드 모두 제공 가능
응답 구조 변경 ❌ 파괴 v2 분리 필수
에러 코드 변경 ⚠️ 상황에 따라 Deprecation 기간 후 v2 전환

10. FastAPI Dependency Injection — 서비스 연결 패턴

FastAPI의 Depends()는 단순한 파라미터 주입이 아니라 요청 수명 주기에 묶인 컨텍스트 관리를 제공한다. DB 세션, 인증된 사용자, 설정값 모두 이 메커니즘으로 주입된다. 

# app/dependencies.py
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession
from app.database import get_db_session
from app.repositories.user_repo import UserRepository
from app.services.user_service import UserService

async def get_user_repository(
    db: AsyncSession = Depends(get_db_session),
) -> UserRepository:
    return UserRepository(db)

async def get_user_service(
    repo: UserRepository = Depends(get_user_repository),
) -> UserService:
    return UserService(repo)

# 인증된 사용자 의존성 (Day 6에서 상세 다룸)
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    service: UserService = Depends(get_user_service),
) -> UserResponse:
    payload = verify_jwt(token)
    user = await service.get_by_id(payload["sub"])
    if not user:
        raise HTTPException(status_code=401, detail="Invalid credentials")
    return user

# 엔드포인트에서 사용
@router.get("/me", response_model=UserResponse)
async def get_my_profile(
    current_user: UserResponse = Depends(get_current_user),
) -> UserResponse:
    return current_user

11. 실무 API 설계 체크리스트

API 설계 셀프 리뷰 기준
  • URL에 동사가 없다 (/getUser → GET /users/{id})
  • HTTP 메서드가 의도를 정확히 표현한다
  • 상태 코드가 의미론적으로 올바르다 (200 vs 201 vs 204 vs 409)
  • Request 스키마 / Response 스키마 / DB 모델이 분리되어 있다
  • password, token, 내부 플래그가 응답에 포함되지 않는다
  • 에러 응답 형식이 서비스 전체에서 동일하다
  • 페이지네이션 파라미터에 상한(max page_size)이 있다
  • 하위 호환을 깨는 변경은 버전 분리가 계획되어 있다
  • 운영 환경에서 /docs가 비공개다
  • 500 응답에 스택 트레이스가 노출되지 않는다

12. 이해도 체크

Q1. PATCH /users/{id}와 PUT /users/{id}의 차이는?

PUT은 자원 전체를 교체한다. 요청 본문에 없는 필드는 기본값 또는 null로 덮어쓴다. 클라이언트가 전체 표현을 알고 있을 때 적합하다.

PATCH는 자원의 일부만 수정한다. 변경할 필드만 보내면 된다. Pydantic에서는 모든 필드를 Optional로 선언하고, model_validator로 최소 하나 이상 있음을 검사한다.

PUT의 멱등성: 같은 요청을 여러 번 보내면 결과가 동일하다.
PATCH의 멱등성: 구현에 따라 다르다 (증분 연산이면 비멱등).

Q2. 클라이언트가 존재하지 않는 사용자 ID로 DELETE를 보냈다. 어떤 상태 코드를 반환해야 하는가?

두 가지 입장이 공존한다:

  • 404 Not Found: 자원이 없음을 명확히 알린다. 클라이언트가 오류를 인식하고 처리할 수 있다.
  • 204 No Content: "삭제 요청의 결과로 그 자원은 없다"는 멱등적 관점. 이미 없으면 성공과 같다.

어느 쪽이든 팀 전체에서 일관성이 중요하다. REST 원칙에서는 404도 정당하고, 멱등성을 중시하면 204도 정당하다. 정책을 문서화하고 통일하는 것이 정답이다.

Q3. 다음 중 하위 호환을 깨는 변경은? (복수 가능)
a) 응답에 선택적 필드 추가   b) 필수 요청 필드 추가   c) 새 엔드포인트 추가   d) 응답 필드명 변경

b와 d가 파괴적 변경(Breaking Change)이다.

  • b) 필수 요청 필드 추가: 기존 클라이언트는 그 필드를 보내지 않으므로 모든 기존 요청이 422로 실패한다. → 버전 분리 필요.
  • d) 필드명 변경: 클라이언트가 이전 필드명으로 파싱하므로 응답 처리가 깨진다. → 버전 분리 필요. 과도기에 두 이름을 모두 내보낼 수도 있다.
  • a) 선택적 필드 추가: 기존 클라이언트는 모르는 필드를 무시하므로 호환된다.
  • c) 새 엔드포인트 추가: 기존 클라이언트에 영향 없음.
