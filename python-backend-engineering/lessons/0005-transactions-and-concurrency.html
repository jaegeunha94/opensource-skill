Day 5 실무자 기준 약 60분

트랜잭션과 동시성
ACID · 격리 수준 · 낙관적/비관적 잠금 · 데드락 방지

Day 4에서 응용 서비스가 commit()을 호출해 트랜잭션 경계를 결정한다고 배웠다. 이 레슨은 그 "commit 한 번"이 동시 요청 환경에서 실제로 무엇을 보장하고, 무엇을 보장하지 않는지 — 그리고 재고 초과 판매, 잔액 불일치 같은 사고를 코드로 어떻게 막는지를 다룬다.

1. 동시성 버그는 로컬 테스트에서 재현되지 않는다

아래 코드는 재고 차감 로직이다. 단위 테스트에서는 항상 통과한다. 문제는 같은 상품에 대한 두 요청이 거의 동시에 들어올 때다. 

async def decrease_stock(session: AsyncSession, product_id: int, qty: int):
    product = await session.get(Product, product_id)
    if product.stock < qty:
        raise OutOfStockError()
    product.stock -= qty          # ← 요청 A, B가 동시에 여기서 같은 값을 읽으면?
    await session.commit()
재고 10개, 두 요청이 각각 8개씩 구매 시도 시각 요청 A 요청 B t1 stock 조회 → 10 t2 stock 조회 → 10 (A의 커밋 전이라 아직 10) t3 10 >= 8 → 통과 t4 10 >= 8 → 통과 t5 stock = 10-8 = 2, commit t6 stock = 10-8 = 2, commit (덮어씀) 결과: 16개가 팔렸는데 재고는 2 → 실제로는 -6이어야 정상
이것이 "Lost Update" 문제다
두 트랜잭션이 같은 행을 읽고, 각자 계산한 값으로 서로를 덮어쓴다. 애플리케이션 레벨의 if 검증은 동시 접근 앞에서 아무 의미가 없다 — 두 요청 모두 "검증 시점"에는 조건을 통과했기 때문이다. 이 문제를 막는 것이 이 레슨 전체의 목표다.

2. ACID — 암기가 아니라 "무엇이 깨지면 무슨 사고가 나는가"로 이해한다

속성보장 내용깨지면 생기는 실무 사고
Atomicity
원자성		 트랜잭션 안의 여러 쓰기는 전부 성공하거나 전부 실패한다 결제는 성공했는데 주문 레코드는 생성 안 됨 (부분 실패)
Consistency
일관성		 트랜잭션 전후로 DB 제약조건(FK, CHECK, UNIQUE)이 항상 유지된다 존재하지 않는 user_id를 참조하는 주문이 저장됨
Isolation
격리성		 동시 실행되는 트랜잭션이 서로의 중간 상태를 보지 못한다 (정도는 격리 수준에 따라 다름) 위 재고 예시처럼 Lost Update, Dirty Read 발생
Durability
지속성		 commit된 데이터는 서버가 죽어도 사라지지 않는다 결제 완료 응답을 보냈는데 서버 재시작 후 기록이 사라짐

이 중 실무에서 매일 판단을 요구하는 것은 Isolation이다. Atomicity와 Durability는 DB 엔진이 알아서 보장하지만, Isolation은 "어느 수준으로 격리할지"를 개발자가 선택해야 하고, 선택에 따라 성능과 정합성이 트레이드오프 관계에 놓인다.

3. 트랜잭션 격리 수준 — 무엇을 허용하고 무엇을 막는가

격리 수준이 낮을수록 동시성(처리량)은 높아지지만 아래 세 가지 "이상 현상"이 허용된다. 격리 수준을 올릴수록 이상 현상은 막히지만 잠금 경합이 늘어난다.

이상 현상설명
Dirty Read 다른 트랜잭션이 아직 commit하지 않은 변경을 읽어버림
Non-repeatable Read 같은 트랜잭션 안에서 같은 행을 두 번 읽었는데 값이 다름 (그 사이 다른 트랜잭션이 commit)
Phantom Read 같은 조건으로 두 번 조회했는데 행의 개수가 다름 (그 사이 다른 트랜잭션이 INSERT)
격리 수준Dirty ReadNon-repeatable ReadPhantom Read비고
Read Uncommitted 발생발생발생 PostgreSQL은 이 수준을 Read Committed로 취급 — 사실상 사용 안 함
Read Committed 차단발생발생 PostgreSQL/MySQL InnoDB의 기본값. 대부분의 API 서버에 충분
Repeatable Read 차단차단PostgreSQL은 차단 (표준상은 발생 가능) 같은 트랜잭션 내 일관된 스냅샷이 필요할 때
Serializable 차단차단차단 가장 엄격 — 마치 트랜잭션이 순차 실행된 것과 같은 결과 보장. 성능 비용 큼 
# SQLAlchemy 2.x — 특정 트랜잭션에만 격리 수준 지정
from sqlalchemy import text

async with session.begin():
    await session.execute(
        text("SET TRANSACTION ISOLATION LEVEL SERIALIZABLE")
    )
    # ... 이 트랜잭션 안의 쿼리는 SERIALIZABLE로 실행됨
Read Committed가 재고 문제를 막아주지 않는 이유
Section 1의 Lost Update는 Read Committed에서도 발생한다. Dirty Read를 막는 것과 "두 트랜잭션이 같은 값을 읽고 서로 덮어쓰는 것"을 막는 것은 별개의 문제다. 후자는 격리 수준을 올리거나(Serializable), 아래에서 다루는 명시적 잠금으로 해결해야 한다.

4. 비관적 잠금 (Pessimistic Locking) — "충돌이 자주 난다"고 가정

행을 읽는 시점에 DB 잠금을 걸어, 다른 트랜잭션이 같은 행을 수정하지 못하게 막는다. PostgreSQL/MySQL 모두 SELECT ... FOR UPDATE로 구현한다. 

async def decrease_stock(session: AsyncSession, product_id: int, qty: int):
    stmt = (
        select(Product)
        .where(Product.id == product_id)
        .with_for_update()          # ← SELECT ... FOR UPDATE
    )
    product = await session.scalar(stmt)

    # 이 행의 잠금을 우리 트랜잭션이 쥐고 있는 동안
    # 다른 트랜잭션의 같은 SELECT FOR UPDATE는 우리가 commit할 때까지 대기한다.
    if product.stock < qty:
        raise OutOfStockError()
    product.stock -= qty
    await session.commit()          # commit 시점에 잠금 해제
요청 A: SELECT ... FOR UPDATE → 잠금 획득, stock=10 읽음 요청 B: SELECT ... FOR UPDATE → 잠금 대기 (A가 끝날 때까지 블록) 요청 A: stock=2로 갱신, COMMIT → 잠금 해제 요청 B: 잠금 획득, stock=2 읽음 (A가 갱신한 최신값) → 정확하게 순차 처리됨
장점단점
충돌을 애초에 차단 — 재시도 로직이 불필요 잠금을 쥔 트랜잭션이 길어지면 뒤 요청들이 줄줄이 대기 (처리량 저하)
구현이 직관적 여러 행에 잠금을 잘못된 순서로 걸면 데드락 위험 (Section 6 참고)

with_for_update(nowait=True)로 즉시 실패시키거나, with_for_update(skip_locked=True)로 이미 잠긴 행을 건너뛸 수도 있다. 후자는 Celery 워커 여러 개가 대기열 테이블에서 "아직 아무도 안 가져간 작업"을 경합 없이 나눠 가질 때 자주 쓰인다 (Day 7에서 다시 등장한다).

5. 낙관적 잠금 (Optimistic Locking) — "충돌은 드물다"고 가정

잠금을 걸지 않고 읽는다. 대신 행에 버전 번호(또는 updated_at)를 두고, 수정할 때 "내가 읽었던 버전이 아직 유효한가"를 UPDATE ... WHERE version = ?로 확인한다. 

# domain/models/product.py
class Product(Base):
    __tablename__ = "products"
    id: Mapped[int] = mapped_column(primary_key=True)
    stock: Mapped[int]
    version: Mapped[int] = mapped_column(default=0)

    __mapper_args__ = {"version_id_col": version}   # SQLAlchemy 내장 낙관적 잠금

async def decrease_stock(session: AsyncSession, product_id: int, qty: int):
    product = await session.get(Product, product_id)   # 잠금 없이 읽음, version=5라 하자
    if product.stock < qty:
        raise OutOfStockError()
    product.stock -= qty

    try:
        await session.commit()
        # SQLAlchemy가 내부적으로 실행:
        # UPDATE products SET stock=2, version=6 WHERE id=1 AND version=5
        # 영향받은 행이 0개면 → StaleDataError 발생
    except StaleDataError:
        # 다른 트랜잭션이 먼저 커밋해서 version이 바뀜 → 재시도
        raise ConcurrentUpdateError("다시 시도하세요")
실무 재시도 패턴
낙관적 잠금은 충돌 시 예외를 던지는 것으로 끝나면 안 된다. 호출부에서 지수 백오프로 2~3회 재시도하거나, 사용자에게 "다시 시도해 주세요"를 노출해야 한다. 재시도 없는 낙관적 잠금은 충돌이 잦은 자원(인기 상품 재고)에서는 사용자 경험을 해친다.

6. 언제 무엇을 쓰는가

기준비관적 잠금낙관적 잠금
충돌 빈도 높음 (인기 상품 재고, 좌석 예약) 낮음 (사용자 프로필 수정, 게시글 편집)
트랜잭션 길이 짧을수록 유리 (잠금 보유 시간 최소화) 무관 — 잠금 자체가 없음
실패 시 UX 대기 (사용자는 응답이 느려지는 것으로 체감) 재시도 또는 명시적 실패 노출 필요
구현 복잡도 낮음 재시도 로직까지 포함하면 비슷하거나 더 높음
실무 기본값
확신이 없으면 비관적 잠금 + 짧은 트랜잭션에서 시작하라. 구현이 예측 가능하고 디버깅하기 쉽다. 부하 테스트에서 잠금 경합이 병목으로 확인되면 그때 낙관적 잠금으로 전환을 검토한다 — 미리 최적화하지 않는다.

7. 데드락 — 원인과 예방

비관적 잠금을 여러 행에 걸 때, 두 트랜잭션이 서로 다른 순서로 잠금을 요청하면 서로를 영원히 기다리는 상태에 빠질 수 있다.

계좌 이체: A → B 그리고 동시에 B → A 트랜잭션 1 (A→B 이체) 트랜잭션 2 (B→A 이체) LOCK account A LOCK account B ... ... LOCK account B (대기: T2가 쥐고 있음) LOCK account A (대기: T1이 쥐고 있음) → 서로가 서로의 잠금 해제를 기다림 = 데드락 DB가 감지해서 한쪽 트랜잭션을 강제 롤백시킴 (deadlock victim)
예방 전략구현 방법
일관된 잠금 순서 항상 id 오름차순으로 잠금을 건다 — 위 예시라면 A와 B 중 id가 작은 쪽을 항상 먼저 잠금
짧은 트랜잭션 잠금을 쥔 상태에서 외부 API 호출, 이메일 발송 등 느린 I/O를 하지 않는다
잠금 타임아웃 SET lock_timeout = '3s'로 무한 대기를 방지하고 애플리케이션에서 재시도
데드락 재시도 DB가 롤백시킨 트랜잭션(DeadlockDetected)을 애플리케이션에서 자동 재시도 
# 일관된 잠금 순서 적용 — 이체 서비스
async def transfer(session: AsyncSession, from_id: int, to_id: int, amount: int):
    first_id, second_id = sorted([from_id, to_id])   # id 오름차순 고정

    first = await session.scalar(
        select(Account).where(Account.id == first_id).with_for_update()
    )
    second = await session.scalar(
        select(Account).where(Account.id == second_id).with_for_update()
    )

    accounts = {first.id: first, second.id: second}
    accounts[from_id].balance -= amount
    accounts[to_id].balance += amount
    await session.commit()
흔한 실수 — "요청받은 순서대로" 잠금
from_id, to_id를 받은 순서 그대로 잠그면, A→B 이체와 B→A 이체가 동시에 들어올 때 정확히 위 다이어그램 상황이 재현된다. 반드시 잠금 대상을 정렬된 고정 기준으로 접근해야 한다.

8. Day 4 레이어 구조에서 잠금은 어디에 있어야 하는가

with_for_update()나 버전 컬럼 재시도 로직은 DB 접근 세부사항이다. Day 4의 원칙을 그대로 적용하면, 이 로직은 Repository 또는 응용 서비스에 있어야 하고 도메인 서비스는 여전히 잠금의 존재를 몰라야 한다. 

# application/services/order_service.py
class DecreaseStockService:
    def __init__(self, product_repo: ProductRepository, session: AsyncSession):
        self._product_repo = product_repo
        self._session = session

    async def execute(self, product_id: int, qty: int) -> None:
        # 잠금 정책은 Repository 구현체 안에 있다 (아래 참고)
        product = await self._product_repo.get_for_update(product_id)
        if product.stock < qty:
            raise InsufficientStockError(product_id)
        product.decrease(qty)              # 도메인 모델의 순수 메서드
        await self._session.commit()       # 트랜잭션 경계는 여전히 응용 서비스가 결정

get_for_update()라는 이름 자체가 Repository 인터페이스에 노출되는 것은 허용된다 — "잠금이 필요한 조회"라는 의도는 도메인 개념이고, FOR UPDATE라는 SQL 구현만 Infrastructure 레이어에 숨으면 된다.

9. 트랜잭션/동시성 셀프 리뷰 체크리스트

코드 리뷰 시 확인할 것
  • "읽고 → 검증 → 쓰기" 패턴이 있는 곳마다 동시 요청 시나리오를 그려봤다
  • 재고/잔액처럼 충돌이 잦은 자원은 비관적 잠금 또는 버전 컬럼을 명시적으로 사용한다
  • 여러 행을 잠글 때 항상 동일한(정렬된) 순서로 잠근다
  • 잠금을 쥔 상태에서 외부 API 호출, 이메일 발송 같은 느린 작업을 하지 않는다
  • 낙관적 잠금 충돌(StaleDataError) 시 재시도 또는 사용자 안내가 있다
  • 기본 격리 수준(Read Committed)이 이 트랜잭션에 충분한지 명시적으로 판단했다

  • PostgreSQL 공식 문서 — Explicit Locking — SELECT FOR UPDATE, NOWAIT, SKIP LOCKED와 데드락 감지 동작.
  • Designing Data-Intensive Applications, Martin Kleppmann — 7장 "Transactions" — 격리 수준과 이상 현상을 실무 사고 사례와 함께 가장 정확하게 설명하는 자료.
