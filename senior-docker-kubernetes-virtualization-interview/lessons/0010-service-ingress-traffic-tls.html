1. Day 9 복기 — 오늘 다루는 것

Day 9에서는 Deployment가 여러 Pod를 새 버전으로 교체하는 과정 — maxSurge/maxUnavailable, 롤백의 실체, Deployment 기반 카나리의 한계를 다뤘다. 그런데 그 논의 내내 "Service"는 이미 트래픽을 Pod로 흘려보내는 배경으로 등장했다. 오늘은 그 배경을 전면에 꺼내서, 사용자의 요청 하나가 인터넷에서 출발해 실제로 어떤 컴포넌트를 순서대로 거쳐 Pod에 도달하는지를 다룬다. 시니어 인터뷰에서 이 주제는 "ClusterIP가 뭔지 아세요"가 아니라, "Service 세 타입의 포함 관계, Ingress가 Service와 근본적으로 다른 계층에서 동작하는 이유, 그리고 TLS를 어느 지점에서 끊을지 설계 판단을 할 수 있는가"를 확인하는 방식으로 나온다.

Service/Ingress 질문의 진짜 평가 포인트는 YAML 필드 이름을 외웠는지가 아니라, "Ingress를 배포했는데 왜 안 되나요"라는 실무에서 매우 흔한 문제를 마주쳤을 때, 문제를 "Kubernetes가 이상하다"가 아니라 "요청이 거치는 여러 컴포넌트 중 어느 지점에서 멈췄는가"라는 구조적 질문으로 바꿔서 좁혀나갈 수 있는가이다. 이 구조를 모르는 엔지니어는 Ingress 리소스 YAML만 계속 들여다보며 시간을 낭비하다가, 정작 원인은 Ingress Controller 자체가 클러스터에 설치되어 있지 않았다는 것을 뒤늦게 발견하는 경우가 실무에서 매우 흔하다.

2. 면접 질문 1 — ClusterIP/NodePort/LoadBalancer는 왜 세 가지나 있고, 왜 그래도 Ingress가 필요한가

핵심 면접 질문
"Service 타입에는 ClusterIP, NodePort, LoadBalancer가 있는데 이 셋은 서로 어떤 관계인가요? 그리고 LoadBalancer 타입으로 외부 트래픽을 이미 받을 수 있는데, 왜 프로덕션에서는 보통 Ingress를 추가로 씁니까?"

왜 면접에서 묻는가

이 질문은 세 Service 타입의 이름을 외웠는지가 아니라, 이 셋이 서로 대체재가 아니라 포함 관계에 있다는 것, 그리고 Service는 근본적으로 L4(TCP/UDP, IP+포트) 레벨에서만 트래픽을 분배할 뿐 L7(HTTP host/path)을 이해하지 못한다는 것을 아는지 확인한다. 이 한계를 모르면 "서비스가 10개면 로드밸런서도 10개 만들면 되지 않나요"처럼 클라우드 비용과 운영 복잡도를 무시하는 설계를 제안하게 된다.

전제 개념 — 쉬운 한국어로

Service란 — "Pod들 앞에 붙는 고정된 가상 주소"

Pod는 재시작되거나 재배치될 때마다 IP가 바뀐다. Service는 label selector에 맞는 Pod들을 묶어 하나의 고정된 가상 IP(ClusterIP)와 DNS 이름을 부여하고, 실제로 어떤 Pod IP들이 뒤에 있는지는 Endpoints(또는 EndpointSlice)라는 객체로 자동 추적한다. 각 노드의 kube-proxy는 이 Endpoints 목록을 보고 iptables 또는 IPVS 규칙을 만들어, ClusterIP로 온 트래픽을 실제 Pod IP 중 하나로 전달(대략 균등 분산)한다. 즉 Service는 "라우팅 규칙을 자동으로 관리해주는 로드밸런서"이지만, 어디까지나 IP와 포트 수준에서만 판단한다.

세 타입의 포함 관계 — 상위 타입은 하위 타입을 감싼다

ClusterIP는 클러스터 내부에서만 접근 가능한 가상 IP다. NodePort는 ClusterIP를 그대로 가지면서, 추가로 모든 노드의 동일한 포트(기본 30000~32767)를 열어 "노드IP:NodePort"로 외부에서 접근할 수 있게 한다. LoadBalancer는 NodePort를 그대로 가지면서, 클라우드 프로바이더에게 요청해 외부 로드밸런서(예: AWS NLB)를 자동으로 하나 프로비저닝하고, 그 로드밸런서가 각 노드의 NodePort로 트래픽을 전달하게 만든다. 세 타입은 경쟁 관계가 아니라 ClusterIP ⊂ NodePort ⊂ LoadBalancer 순서로 기능이 누적되는 구조다.

타입접근 범위대표 용도한계
ClusterIP클러스터 내부만내부 마이크로서비스 간 통신외부에서 직접 접근 불가
NodePort클러스터 내부 + 노드IP:포트온프레미스 임시 노출, 디버깅포트 범위 제한, 노드 IP를 직접 알아야 함
LoadBalancer퍼블릭 IP(클라우드 LB)외부 트래픽을 직접 받아야 하는 서비스서비스마다 로드밸런서 1개 = 비용·IP 낭비, L7 라우팅 불가

왜 위험한가 — Service만으로 여러 서비스를 노출할 때

HTTP 기반 서비스가 10개 있다고 해서 각각을 LoadBalancer 타입으로 만들면, 클라우드에 로드밸런서 10개가 생성된다. 이는 비용이 서비스 개수만큼 선형으로 증가할 뿐 아니라, Service의 로드밸런싱은 HTTP 요청의 host나 path를 전혀 읽지 못하기 때문에 "api.example.com은 A 서비스로, admin.example.com은 B 서비스로" 같은 도메인/경로 기반 라우팅을 아예 표현할 방법이 없다. 즉 Service 타입만으로는 "IP와 포트가 같은데 나눠서 보내고 싶다"는 요구를 근본적으로 만족시킬 수 없다.

답변 사고 순서

  1. 세 타입이 경쟁이 아니라 포함 관계라는 것을 먼저 명시.
  2. Service의 근본 한계를 계층으로 짚기: L4(IP+포트)만 이해하고 L7(host/path)은 모른다는 것.
  3. 이 한계가 실제로 문제 되는 상황 제시: 서비스 개수만큼 로드밸런서가 늘어나는 비용 문제, 도메인 기반 라우팅 불가.
  4. Ingress가 그 한계를 어떻게 메우는지 연결: 로드밸런서 하나로 여러 서비스를 host/path 기준으로 라우팅.

핵심 답변 예시

"세 타입은 서로 대체하는 관계가 아니라 ClusterIP를 NodePort가, NodePort를 LoadBalancer가 감싸는 포함 관계입니다. ClusterIP는 클러스터 내부용 가상 IP고, NodePort는 여기에 모든 노드의 특정 포트를 열어 외부 접근을 추가한 것이고, LoadBalancer는 거기에 클라우드 로드밸런서 프로비저닝까지 자동화한 것입니다.

그런데 이 세 타입은 모두 L4, 즉 IP와 포트 수준에서만 트래픽을 분배합니다. HTTP 요청의 host나 path를 보고 라우팅을 나누는 기능이 없습니다. 그래서 HTTP 서비스가 여러 개일 때 각각을 LoadBalancer로 만들면 서비스 개수만큼 로드밸런서가 생겨서 비용이 늘고, 애초에 'api.example.com은 이 서비스로, admin.example.com은 저 서비스로' 같은 요구 자체를 표현할 방법이 없습니다. Ingress는 이 L7 라우팅 문제를 해결하기 위한 계층으로, 로드밸런서 하나(보통 Ingress Controller의 Service)만 외부에 노출하고, 그 뒤에서 host/path 규칙에 따라 여러 개의 내부 Service로 트래픽을 나눠줍니다."

✓ 강한 답변
  • 세 타입이 포함 관계라는 것을 정확히 설명
  • Service가 L4까지만 다룬다는 계층적 한계를 명시
  • Ingress 도입 이유를 비용과 라우팅 표현력 두 측면에서 연결
✗ 약한 답변
  • 세 타입을 그냥 "외부 노출 방법 세 가지"로만 나열
  • Service와 Ingress의 계층 차이(L4 vs L7)를 언급 안 함
  • "Ingress가 더 최신 기능이라서 쓴다"처럼 근거 없이 답함

Senior-level Follow-up

면접관: "그러면 NodePort는 실무에서 아예 안 쓰이나요?"

시니어 답변 방향: "완전히 안 쓰이지는 않습니다. 온프레미스 환경처럼 클라우드 로드밸런서 자동 프로비저닝 자체가 불가능한 곳에서는, 자체 구축한 외부 로드밸런서(예: MetalLB 없이 HAProxy를 직접 구성)가 각 노드의 NodePort로 트래픽을 전달하는 방식으로 여전히 쓰입니다. 또한 Ingress Controller 자신을 외부에 노출할 때도 결국 그 Controller의 Service가 LoadBalancer 또는 NodePort 타입이어야 하므로, NodePort는 없어진 게 아니라 최종 사용자에게 직접 보이지 않는 하위 계층으로 이동한 것에 가깝습니다."

3. 면접 질문 2 — Ingress를 만들었는데 트래픽이 안 들어온다면 어디부터 봐야 하는가

핵심 면접 질문
"Ingress 리소스를 kubectl apply로 배포했는데 해당 도메인으로 접속이 안 됩니다. Ingress 리소스와 Ingress Controller는 어떤 관계이고, 이 상황을 어떤 순서로 진단하시겠습니까?"

왜 면접에서 묻는가

이 질문은 kubectl apply -f ingress.yaml 명령을 알고 있는지가 아니라, "Ingress 리소스는 선언(spec)일 뿐이고, 그 선언을 실제로 실행하는 별도의 프로그램(Ingress Controller)이 클러스터에 미리 떠 있어야 한다"는 관계를 이해하는지를 확인한다. 이 관계를 모르면 문제가 생겼을 때 Ingress YAML의 오타만 계속 의심하다가, 애초에 Ingress Controller 자체가 설치되지 않았거나 ingressClassName이 어긋나 있었다는 훨씬 근본적인 원인을 놓치게 된다.

전제 개념 — 쉬운 한국어로

Ingress 리소스 vs Ingress Controller — "규칙서"와 "규칙을 실행하는 사람"

Ingress는 "이 host로 오면 이 Service로, 저 path로 오면 저 Service로 보내라"는 라우팅 규칙을 선언하는 Kubernetes 오브젝트일 뿐이다. Deployment가 없으면 Pod가 안 뜨듯이, Ingress Controller가 클러스터에 없으면 Ingress 오브젝트는 etcd에 저장된 텍스트일 뿐 아무 트래픽도 처리하지 않는다. 실제로 트래픽을 받아 host/path를 검사하고 백엔드 Service로 전달하는 것은 nginx-ingress, Traefik, AWS Load Balancer Controller 같은 별도의 프로그램이며, 이 프로그램 자체도 Pod로 떠서 자신만의 Service(대개 LoadBalancer 타입)를 통해 외부 트래픽을 받는다. 즉 Ingress는 Ingress Controller가 "watch"하고 있는 설정 파일에 가깝다.

ingressClassName — "이 규칙은 어느 Controller가 실행하는가"

클러스터에 Ingress Controller가 여러 개 설치되어 있을 수도 있다(예: 내부용 nginx, 외부용 별도 Controller). 이때 Ingress 리소스가 spec.ingressClassName으로 대상 Controller를 명시하지 않으면, 어느 Controller도 그 규칙을 처리하지 않거나, 여러 Controller가 동시에 처리하려 해서 예상치 못한 동작이 생길 수 있다. 이 필드가 실제 클러스터에 설치된 IngressClass 이름과 정확히 일치하지 않는 것은 "Ingress를 만들었는데 반응이 없는" 흔한 원인 중 하나다.

확인 순서확인 대상흔한 원인
1Ingress Controller Pod가 떠 있는가애초에 클러스터에 Controller가 설치되지 않음
2Controller의 Service에 External IP가 할당됐는가LoadBalancer가 클라우드에서 아직 프로비저닝 중이거나 실패
3Ingress의 ingressClassName이 설치된 IngressClass와 일치하는가이름 오타, 기본 IngressClass 미설정
4DNS가 Controller의 External IP를 가리키는가도메인이 아직 예전 IP를 가리킴, DNS 전파 지연
5백엔드 Service의 Endpoints가 채워져 있는가label selector 불일치, 백엔드 Pod가 Ready 아님

왜 위험한가 — Ingress YAML만 계속 들여다볼 때

많은 팀에서 "트래픽이 안 온다"는 문제를 Ingress 리소스의 host/path 문법 오류로 지레짐작하고 YAML만 반복해서 고친다. 하지만 실제로는 1) Ingress Controller 자체가 설치 안 됐거나, 2) Controller의 LoadBalancer Service가 아직 External IP를 받지 못했거나, 3) DNS가 그 IP를 가리키지 않는 훨씬 앞단의 문제인 경우가 더 흔하다. 이 경우 Ingress YAML을 아무리 정교하게 고쳐도 트래픽은 애초에 Controller에조차 도달하지 못하므로 문제가 절대 해결되지 않는다. 즉 진단은 "트래픽이 어느 컴포넌트까지 도달했는가"를 바깥(DNS)에서 안쪽(백엔드 Pod)으로 좁혀가는 순서로 해야 한다.

답변 사고 순서

  1. Ingress와 Ingress Controller의 관계를 먼저 명시: 선언(규칙) vs 실행(별도 Pod + Service).
  2. 진단 순서를 바깥에서 안쪽으로 설계: DNS → Controller의 External IP → ingressClassName 매칭 → 백엔드 Endpoints.
  3. 각 단계에서 확인할 kubectl 명령을 구체적으로 언급.
  4. "Ingress YAML 문법 문제"는 마지막에 확인하는 이유를 설명.

핵심 답변 예시

"Ingress 리소스는 '어떻게 라우팅할지'를 선언하는 스펙일 뿐이고, 실제로 트래픽을 받아서 그 규칙대로 처리하는 건 nginx-ingress 같은 Ingress Controller라는 별도의 Pod입니다. 그래서 Ingress를 만들었는데 트래픽이 안 온다면, 저는 바깥에서부터 안쪽으로 좁혀갑니다. 먼저 도메인의 DNS가 실제로 Ingress Controller Service의 External IP를 가리키는지 확인하고, 그 Service가 애초에 External IP를 받았는지, Controller Pod 자체가 정상적으로 떠 있는지 kubectl get pods -n ingress-nginx로 확인합니다.

이게 다 정상이면 그다음 Ingress 리소스의 ingressClassName이 실제 설치된 IngressClass와 일치하는지 봅니다. 여기까지 문제가 없는데도 안 되면 마지막으로 백엔드 Service의 Endpoints가 채워져 있는지, 즉 label selector가 실제 Pod와 일치하고 그 Pod들이 Ready 상태인지 확인합니다. Ingress YAML의 host/path 문법 자체를 의심하는 건 이 모든 단계를 거친 다음입니다. 왜냐하면 실무에서 겪은 경험상 'Ingress가 안 된다'의 대부분은 문법 오류가 아니라 Controller 미설치나 ingressClassName 불일치였기 때문입니다."

✓ 강한 답변
  • Ingress(선언)와 Ingress Controller(실행)를 명확히 분리해서 설명
  • 바깥(DNS/LB)에서 안쪽(백엔드 Pod)으로 좁혀가는 체계적 순서 제시
  • 각 단계에 대응하는 구체적 확인 방법을 언급
✗ 약한 답변
  • Ingress 리소스 YAML을 고치는 것부터 바로 언급
  • Ingress Controller가 별도로 설치되어야 한다는 사실을 모름
  • ingressClassName의 존재나 역할을 언급 안 함

Senior-level Follow-up

면접관: "Ingress Controller Pod가 여러 개(replica 3개)로 떠 있다면, 이들 사이의 라우팅 규칙 동기화는 어떻게 이루어지나요?"

시니어 답변 방향: "각 Controller Pod는 독립적으로 Kubernetes API 서버를 watch하면서 Ingress/Service/Endpoints 변경을 실시간으로 감지하고, 자신의 내부 라우팅 설정(예: nginx라면 nginx.conf)을 각자 재생성합니다. 즉 Pod끼리 서로 직접 통신하며 동기화하는 게 아니라, 모두 같은 소스(API 서버의 상태)를 각자 구독해서 같은 결론에 수렴하는 구조입니다. 다만 이 재생성과 반영에는 약간의 지연이 있을 수 있어서, Ingress를 급하게 바꾼 직후에는 Controller replica마다 순간적으로 다른 라우팅 상태를 가질 수 있다는 점을 배포 타이밍에 고려합니다."

4. 면접 질문 3 — TLS는 어디서 끊어야 하는가

핵심 면접 질문
"HTTPS 트래픽의 TLS를 Ingress에서 종료하는 것과, 각 애플리케이션 Pod까지 암호화를 유지하는 것의 차이는 무엇인가요? 어떤 경우에 후자를 선택하나요?"

왜 면접에서 묻는가

이 질문은 cert-manager로 인증서를 자동 발급하는 방법을 아는지가 아니라, "TLS 종료(termination)"라는 개념 자체 — 어느 지점에서 암호화된 트래픽을 평문으로 복호화하는가 — 를 이해하고, 그 지점을 선택하는 것이 보안과 운영 편의성 사이의 트레이드오프라는 것을 아는지 확인한다. 이 트레이드오프를 모르면 "Ingress에서 HTTPS로 받으니까 다 안전하다"처럼 클러스터 내부 트래픽까지 암묵적으로 안전하다고 착각하게 된다.

전제 개념 — 쉬운 한국어로

TLS 종료란 — "암호를 푸는 지점"

클라이언트가 보낸 HTTPS 요청은 암호화되어 있어서, 그 내용(host, path, body)을 읽고 라우팅 결정을 내리려면 누군가는 반드시 그 암호를 풀어야 한다. 이 복호화가 일어나는 지점을 TLS 종료 지점이라 부른다. Ingress에서 TLS를 종료하는 것이 가장 흔한 방식으로, 인증서(Secret으로 저장)를 Ingress Controller 한 곳에만 설치하면 되고, cert-manager 같은 도구로 인증서 발급과 갱신을 한 곳에서 자동화할 수 있어 운영이 단순하다.

종료 이후 구간 — "클러스터 내부는 기본적으로 평문"

Ingress에서 TLS를 종료하면, 그 이후 Ingress Controller → Service → Pod 구간은 기본적으로 평문 HTTP로 흐른다. 클러스터 내부 네트워크는 대개 외부보다 신뢰도가 높다고 가정하기 때문에 실무에서 널리 쓰이는 방식이지만, 클러스터 내부에 있는 다른 Pod(악의적이거나 침해당한 Pod)가 이 평문 트래픽을 스니핑할 가능성은 여전히 남는다. 금융, 의료처럼 규제가 엄격하거나 제로 트러스트(zero-trust) 원칙을 요구하는 환경에서는 이 잔여 위험이 허용되지 않는다.

종단간 암호화 — TLS Passthrough와 서비스 메시 mTLS

이런 요구가 있을 때는 두 가지 방식을 쓴다. 하나는 TLS Passthrough로, Ingress가 암호를 풀지 않고 SNI(클라이언트가 접속하려는 도메인 이름, TLS 핸드셰이크 초반에 암호화되지 않은 채로 노출됨)만 보고 어느 백엔드로 연결을 그대로 전달할지 결정한 뒤, 실제 복호화는 백엔드 Pod 자신이 담당하게 하는 방식이다. 다른 하나는 서비스 메시(Istio, Linkerd)의 mTLS로, 각 Pod 옆에 붙는 사이드카가 Pod 간 통신 전체를 자동으로 상호 인증 및 암호화해서, 애플리케이션 코드는 평문으로 통신한다고 생각하지만 실제로는 사이드카끼리 암호화된 채널로 전송하는 방식이다.

방식복호화 지점인증서 관리클러스터 내부 구간 보안운영 복잡도
Ingress에서 종료Ingress Controller한 곳에서 집중 관리(cert-manager)평문(내부 신뢰 가정)낮음
TLS Passthrough각 백엔드 PodPod마다 개별 관리 필요암호화 유지높음(인증서 배포/갱신 분산)
서비스 메시 mTLS사이드카(자동)메시가 자동 발급/순환암호화 + 상호 인증중간(메시 도입/운영 필요)

흔한 오해 — "Ingress가 HTTPS면 트래픽 전체가 안전하다"

많은 사람이 브라우저 주소창의 자물쇠 아이콘을 보고 "이 요청은 끝까지 암호화되어 있다"고 생각한다. 하지만 Ingress에서 TLS를 종료하는 일반적인 구성에서는 클라이언트-Ingress 구간만 암호화되어 있고, Ingress-Pod 구간은 평문이다. "HTTPS를 쓰고 있다"는 사실만으로 클러스터 내부 트래픽까지 안전하다고 단정하는 것은, 어느 구간이 암호화되어 있는지를 구분하지 못하는 흔한 실수다.

답변 사고 순서

  1. TLS 종료의 정의를 명확히: 암호화된 트래픽을 평문으로 바꾸는 지점.
  2. 가장 흔한 기본값(Ingress 종료)의 장점과 잔여 위험을 함께 제시.
  3. 잔여 위험이 문제가 되는 구체적 상황 제시: 규제 산업, 제로 트러스트 요구.
  4. 대안(Passthrough, mTLS)의 동작 방식과 그에 따른 운영 비용 증가를 연결.

핵심 답변 예시

"TLS 종료는 암호화된 트래픽을 평문으로 복호화하는 지점을 말합니다. 가장 흔하게는 Ingress Controller에서 종료하는데, 이러면 인증서를 한 곳에만 두고 cert-manager로 발급과 갱신을 자동화할 수 있어서 운영이 단순합니다. 다만 이 방식은 Ingress 이후, 즉 Ingress에서 백엔드 Pod까지의 구간이 기본적으로 평문 HTTP로 흐른다는 걸 의미합니다. 클러스터 내부는 외부보다 신뢰도가 높다고 가정하기 때문에 대부분의 서비스에서는 이 정도로 충분합니다.

하지만 금융이나 의료처럼 규제가 엄격하거나 제로 트러스트를 요구하는 환경에서는 클러스터 내부의 평문 구간도 위험으로 간주됩니다. 이때는 TLS Passthrough로 Ingress가 SNI만 보고 연결을 그대로 백엔드로 넘겨 백엔드가 직접 복호화하게 하거나, Istio 같은 서비스 메시를 도입해서 사이드카가 Pod 간 통신을 자동으로 mTLS로 암호화하게 만듭니다. 두 방식 모두 종단간 암호화를 얻는 대신, 인증서 관리 지점이 늘어나거나 메시 자체를 운영해야 하는 복잡도가 추가되는 트레이드오프가 있습니다."

✓ 강한 답변
  • TLS 종료를 "복호화가 일어나는 지점"으로 명확히 정의
  • Ingress 종료 방식의 잔여 위험(내부 평문 구간)을 스스로 짚음
  • Passthrough/mTLS 도입을 요구사항(규제, 제로 트러스트)과 연결하고 비용까지 언급
✗ 약한 답변
  • "HTTPS 쓰면 다 안전하다"고 단정
  • Ingress 이후 구간이 평문일 수 있다는 사실을 모름
  • mTLS와 일반 TLS 종료의 차이를 구분 못 함

Senior-level Follow-up

면접관: "서비스 메시 도입 없이, 최소한의 비용으로 클러스터 내부 트래픽 위험을 줄일 방법이 있을까요?"

시니어 답변 방향: "메시 전체 도입 전에 할 수 있는 단계적 조치가 있습니다. 첫째, NetworkPolicy로 '이 Pod는 정해진 다른 Pod에서만 트래픽을 받을 수 있다'는 식으로 애초에 내부 트래픽을 스니핑할 수 있는 위치 자체를 제한합니다. 둘째, 정말 민감한 소수의 서비스 간 통신만 골라서 애플리케이션 레벨에서 자체적으로 TLS를 구현하거나 사이드카를 부분 도입하는 방법도 있습니다. 전체 메시 도입은 운영 부담이 크기 때문에, 저는 먼저 '어떤 통신 구간이 실제로 민감한 데이터를 다루는가'를 식별해서 위험이 큰 곳부터 좁혀서 대응하는 것을 우선합니다."

5. 시니어의 정신 모델 — 요청 하나의 전체 경로 추적

"사이트가 안 열려요"라는 알림을 받았을 때, 시니어는 요청 하나가 거치는 모든 구간을 순서대로 나열해두고, 어느 구간까지는 정상이고 어느 구간부터 끊겼는지를 이분 탐색하듯 좁혀나간다.

요청의 전체 경로와 구간별 확인 방법

구간역할장애 시 확인 방법
클라이언트 → DNS도메인을 IP로 변환dig/nslookup으로 실제 반환 IP 확인
DNS → 클라우드 로드밸런서Ingress Controller Service의 External IPkubectl get svc -n ingress-nginx로 IP 일치 확인
로드밸런서 → Ingress Controller PodTCP/TLS 수신, host/path 매칭kubectl get pods, kubectl logs로 Controller 상태 확인
Ingress Controller → 백엔드 ServiceingressClassName 일치 시 라우팅 규칙 적용kubectl describe ingress로 백엔드 매핑 확인
Service → Podlabel selector로 Endpoints 결정, kube-proxy가 실제 전달kubectl get endpoints로 IP 목록이 비어있지 않은지 확인

핵심 원칙: "사이트가 안 열린다"는 증상 하나에 원인이 될 수 있는 지점은 최소 다섯 군데 이상이다. 이 순서를 바깥(DNS)에서 안쪽(Pod)으로 하나씩 확인하면, 어느 구간에서 문제가 생겼는지 추측이 아니라 관찰로 좁힐 수 있다. 반대로 순서 없이 아무 곳이나 먼저 의심하면(대개는 애플리케이션 코드부터 의심하게 된다) 실제로는 DNS나 로드밸런서 단계에서 이미 막혀 있던 문제를 붙잡고 애플리케이션 로그만 몇 시간째 들여다보는 실수를 하게 된다.

6. 자기 점검 Rubric

□ ClusterIP/NodePort/LoadBalancer가 경쟁 관계가 아니라 포함 관계라는 것을 설명할 수 있다
□ Service가 L4(IP+포트)까지만 다루고 L7(host/path)을 이해하지 못한다는 한계를 설명할 수 있다
□ Ingress 리소스(선언)와 Ingress Controller(실행하는 별도 Pod)가 서로 다른 것이라는 관계를 설명할 수 있다
□ "Ingress가 안 된다"는 문제를 DNS → 로드밸런서 → Controller → ingressClassName → 백엔드 Endpoints 순서로 진단하는 절차를 설명할 수 있다
□ TLS 종료가 "암호화된 트래픽을 평문으로 바꾸는 지점"이라는 정의를 정확히 설명할 수 있다
□ Ingress에서 TLS를 종료하면 그 이후 클러스터 내부 구간은 기본적으로 평문이라는 잔여 위험을 설명할 수 있다
□ TLS Passthrough와 서비스 메시 mTLS가 각각 어떻게 종단간 암호화를 달성하는지, 그 대가로 어떤 운영 비용이 늘어나는지 설명할 수 있다
□ "HTTPS를 쓰면 트래픽 전체가 안전하다"는 오해를 구간별로 바로잡아 설명할 수 있다
