1. 왜 이 주제가 시니어 면접에서 어려운가

Docker / Kubernetes 면접에서 주니어와 시니어를 구분하는 기준은 단 하나다.

주니어: "컨테이너는 경량 VM입니다. docker run으로 실행하고, kubectl apply로 배포합니다."

시니어: "컨테이너는 VM이 아닙니다. 커널을 공유하고 네임스페이스로 격리합니다. 그래서 VM보다 시작이 빠르지만, 커널 취약점이 공유된다는 보안 trade-off가 있습니다. 우리 팀은 그래서 이미지에 non-root 유저를 강제하고 seccomp 프로파일을 적용했습니다."

면접관이 평가하는 것은 다음 세 가지 능력이다:

  1. 원리 이해: "왜 이렇게 동작하는가" — 단순 명령어 지식이 아니라 내부 메커니즘
  2. Trade-off 판단: "어떤 상황에서 이 선택이 맞는가, 반대 선택은 언제 맞는가"
  3. 운영 경험: "실제 프로덕션에서 어떤 문제를 겪었고 어떻게 해결했는가"

이 커리큘럼은 그 세 가지를 균형 있게 준비시키는 것이 목표다.

2. 면접 질문 1 — 컨테이너와 VM의 차이

핵심 면접 질문
"컨테이너와 VM의 차이점을 설명해주세요. 그리고 언제 VM이 더 나은 선택인지도 말씀해주세요."

왜 면접에서 묻는가

이 질문으로 면접관은 두 가지를 확인한다:

  • 컨테이너를 "마법의 도구"로 아는지, 아니면 Linux 커널의 기능을 활용하는 격리 메커니즘으로 이해하는지
  • 컨테이너가 만능이 아님을 알고, VM이 필요한 상황을 설명할 수 있는지 (시니어는 "항상 컨테이너"를 말하지 않는다)

전제 개념 — 쉬운 한국어로

VM이란?

VM(Virtual Machine)은 하드웨어를 소프트웨어로 흉내 낸 완전한 컴퓨터다. 물리 서버 위에 하이퍼바이저(Hypervisor)가 올라가고, 그 위에 각각의 VM이 자신만의 운영체제 커널을 갖는다.

  • VMware ESXi, KVM, Hyper-V가 대표적인 하이퍼바이저
  • VM은 하드웨어를 에뮬레이트하므로 부팅에 수십 초~수 분이 걸린다
  • 커널이 분리되므로 격리가 완전하다 — 한 VM의 커널이 크래시해도 다른 VM에 영향 없음

컨테이너란?

컨테이너는 Linux 커널의 두 가지 기능을 조합해 만든 격리된 프로세스 그룹이다:

  • 네임스페이스(Namespace): 프로세스, 네트워크, 파일시스템, 사용자 ID를 격리. 컨테이너 안의 PID 1이 호스트에서는 다른 PID로 보인다.
  • cgroup(Control Group): CPU, 메모리, I/O 자원 사용량을 제한. 컨테이너가 host의 자원을 독점하지 못하게 한다.

컨테이너는 커널을 공유한다. Host OS의 커널 위에서 격리된 프로세스로 실행될 뿐이다. 별도의 OS 부팅이 없으므로 밀리초 단위로 시작할 수 있다.

구조 비교

항목 VM 컨테이너
커널 VM마다 독립 커널 (GuestOS) Host 커널 공유
격리 수준 하드웨어 수준 (강함) 프로세스 수준 (더 약함)
시작 시간 수십 초 ~ 수 분 밀리초 ~ 수 초
이미지 크기 수 GB 수십 MB ~ 수백 MB
오버헤드 높음 (하이퍼바이저 + GuestOS) 낮음 (네임스페이스 + cgroup)
커널 취약점 공유 없음 있음 (커널 공유)
OS 선택 Host와 무관 (Windows VM on Linux) Host 커널과 같아야 함
밀도 (같은 서버에 몇 개) 낮음 (수십 개) 높음 (수백~수천 개)

답변 사고 순서

  1. 격리 메커니즘부터: VM은 하이퍼바이저 + GuestOS로 커널이 분리되고, 컨테이너는 Linux 네임스페이스 + cgroup으로 프로세스를 격리한다는 사실부터 시작한다.
  2. 결과적 차이 나열: 시작 시간, 자원 오버헤드, 밀도, 격리 강도
  3. Trade-off로 마무리: 컨테이너가 항상 좋은 게 아니라, 커널 공유라는 보안 제약이 있고, Windows 컨테이너를 Linux에서 네이티브로 실행할 수 없다는 한계가 있다.
  4. VM이 더 나은 케이스로 보강: 강한 격리가 필요한 멀티테넌트, OS가 다른 경우, 레거시 소프트웨어.

핵심 답변 예시

"VM은 하이퍼바이저가 하드웨어를 가상화하고 각 VM이 자신만의 GuestOS 커널을 가집니다. 완전한 격리지만 GuestOS 오버헤드로 부팅에 분이 걸리고, 동일 서버에서 수십 개가 한계입니다.

컨테이너는 Linux 커널의 네임스페이스로 파일시스템·네트워크·PID를 격리하고, cgroup으로 CPU/메모리 사용량을 제한합니다. 커널은 Host와 공유하기 때문에 수백 밀리초 안에 시작하고, 동일 서버에서 수백 개를 실행할 수 있습니다.

그러나 컨테이너는 커널을 공유하므로 커널 취약점은 모든 컨테이너에 영향을 미칩니다. 강한 멀티테넌트 격리가 필요하거나 Windows 서비스를 Linux 인프라에서 실행해야 한다면 VM이 더 나은 선택입니다. 우리 팀에서는 그런 이유로 멀티테넌트 SaaS 고객의 Heavy isolation이 필요한 작업은 VM 기반 워커를 별도로 유지했습니다."

✓ 강한 답변
  • 격리 메커니즘(네임스페이스, cgroup) 언급
  • 컨테이너의 보안 제약 (커널 공유) 명시
  • VM이 더 나은 케이스를 구체적으로 제시
  • 실무 경험으로 연결 ("우리 팀에서는...")
✗ 약한 답변
  • "컨테이너는 경량 VM입니다" — 기술적으로 틀림
  • 시작 시간만 나열하고 trade-off 없음
  • VM이 필요한 상황을 설명 못함
  • 실무 연결이 전혀 없음

Senior-level Follow-up

면접관: "gVisor나 Kata Containers 같은 기술을 들어보셨나요? 왜 생겼을까요?"

시니어 답변 방향: "컨테이너의 커널 공유 보안 문제를 해결하기 위한 시도입니다. gVisor는 컨테이너의 시스템 콜을 사용자 공간에서 인터셉트해 커널 공격면을 줄이고, Kata Containers는 경량 VM을 컨테이너처럼 사용해 커널을 분리합니다. VM의 격리 강도와 컨테이너의 빠른 시작을 동시에 얻으려는 접근입니다. 실제 AWS Fargate가 이 방식을 씁니다."

3. 면접 질문 2 — Kubernetes를 쓰는 이유

핵심 면접 질문
"Docker만으로 운영하는 것과 Kubernetes를 도입하는 것의 차이는 무엇인가요? 팀에 Kubernetes 도입을 제안한다면 어떤 기준으로 판단하시겠습니까?"

왜 면접에서 묻는가

이 질문은 기술 도입의 판단력을 테스트한다. "Kubernetes는 당연히 써야 한다"고 대답하면 감점이다. 면접관은 아래를 확인한다:

  • Docker 단독 운영의 한계를 실제로 경험했는가
  • Kubernetes가 해결하는 문제를 정확히 알고 있는가
  • Kubernetes가 오히려 과잉(over-engineering)인 케이스를 알고 있는가

전제 개념 — Docker 단독 운영의 한계

Docker만으로 여러 서비스를 운영하면 다음 문제들이 수동으로 남는다:

  • 컨테이너 다운 시 자동 재시작: --restart=always로 단일 호스트에서는 되지만, 호스트가 죽으면 다른 호스트로 컨테이너를 옮겨줄 오케스트레이터가 없다.
  • 트래픽 배분: 같은 서비스 컨테이너가 3개 뜨면 앞에 직접 로드밸런서를 구성해야 한다.
  • 무중단 배포: 신버전을 구버전 종료 없이 교체하는 롤링 업데이트를 직접 스크립트로 구현해야 한다.
  • 상태 확인 후 트래픽 전환: 새 컨테이너가 실제로 준비됐을 때만 트래픽을 받아야 하는데, 이를 healthcheck와 연동하는 로직이 필요하다.
  • 자원 배치: 어떤 서버에 어떤 컨테이너를 배치할지 결정하는 스케줄러가 없다.
  • Secret 관리: 환경변수로 직접 주입하거나, 각 서버에 .env 파일을 배포해야 한다.

Kubernetes는 이 모든 것을 선언적으로 관리해주는 플랫폼이다.

Kubernetes가 제공하는 것

Docker 단독의 문제 Kubernetes의 해결
컨테이너 크래시 후 수동 재시작 ReplicaSet이 원하는 수를 유지
새 서버에 컨테이너 수동 배치 Scheduler가 자원 여유 있는 Node에 자동 배치
로드밸런싱 수동 구성 Service 리소스가 kube-proxy로 자동 처리
롤링 업데이트 스크립트 작성 Deployment의 RollingUpdate 전략
서비스 디스커버리 수동 구성 CoreDNS + Service IP로 자동 DNS 등록
Secret 수동 배포 Secret 리소스 + RBAC
노드 장애 시 수동 대응 컨트롤러가 다른 노드에 Pod 재스케줄

답변 사고 순서 — 도입 판단 기준

  1. 서비스 수와 트래픽 규모: 단일 서비스, 트래픽 적음 → Docker Compose면 충분. 10개 이상 서비스, 트래픽 변동 큼 → K8s 도입 검토.
  2. 배포 빈도: 하루에 수십 번 배포하고 무중단이 필요하면 K8s의 RollingUpdate가 큰 가치를 준다.
  3. 팀 역량: K8s 자체가 복잡하다. etcd, 컨트롤 플레인, RBAC, 네트워크 정책까지 관리해야 한다. 팀에 K8s를 운영할 인력이 없으면 오히려 부담이다.
  4. 관리형 서비스 여부: EKS, GKE, AKS 같은 관리형 K8s를 쓰면 컨트롤 플레인 관리 부담이 줄어든다. 이 경우 도입 임계점이 낮아진다.
  5. 비용: K8s는 최소 3개 마스터 노드 + N개 워커 노드가 필요하다. 소규모 팀에서 이 비용을 감당할 수 있는지 확인해야 한다.

핵심 답변 예시:

"Docker만으로 운영할 때의 핵심 문제는 오케스트레이션이 없다는 겁니다. 컨테이너가 죽으면 자동으로 다시 뜨게 하고, 배포 중 트래픽을 끊지 않고, 여러 서버에 걸쳐 컨테이너를 분산시키는 로직을 전부 직접 구현해야 합니다.

Kubernetes는 이 오케스트레이션을 선언적으로 처리합니다. 'Deployment 3개를 항상 유지하고, 배포 시 maxSurge 1, maxUnavailable 0으로 교체하라'고 선언하면 나머지는 K8s가 합니다.

도입 기준은 상황에 따라 달라집니다. 서비스가 5개 이하, 배포 빈도가 낮다면 Docker Compose + 간단한 배포 스크립트로 충분합니다. K8s를 도입하면 YAML 관리, RBAC, 네트워크 정책, 업그레이드 등 운영 복잡도가 상당히 올라갑니다. 팀에 K8s를 운영할 인력과 시간이 있는지가 핵심 판단 기준입니다."

✓ 강한 답변
  • Docker 단독의 구체적 한계 나열
  • K8s가 해결하는 것을 선언적 관리로 연결
  • "무조건 K8s" 대신 도입 기준을 제시
  • K8s의 복잡도 비용을 언급
✗ 약한 답변
  • "K8s는 컨테이너 오케스트레이터입니다" — 설명 없는 정의
  • "규모가 크면 K8s 써야죠" — 구체적 기준 없음
  • K8s 도입 비용(복잡도, 인력) 무시
  • Docker Compose가 적합한 경우를 모름

Senior-level Follow-up

면접관: "Kubernetes 없이 컨테이너 오케스트레이션을 할 수 있는 다른 방법은 무엇이 있나요?"

시니어 답변 방향: "Docker Swarm은 K8s보다 훨씬 단순한 오케스트레이션을 제공합니다. 소규모 팀에서 빠르게 시작하기 좋지만, 생태계와 기능이 K8s에 비해 부족합니다. AWS ECS는 AWS 인프라에 묶이지만 운영 부담이 낮고 비용이 예측하기 쉽습니다. Nomad는 HashiCorp의 오케스트레이터로 컨테이너 외 VM, 바이너리도 스케줄링합니다. 각각 trade-off가 있고, '무조건 K8s'가 아니라 팀 규모와 인프라 의존성에 따라 선택이 달라집니다."

4. 면접 질문 3 — 프로덕션 클러스터 설계 시 가장 먼저 고려하는 것

핵심 면접 질문
"프로덕션 Kubernetes 클러스터를 새로 구성한다면 가장 먼저 고려해야 할 것들은 무엇인가요?"

왜 면접에서 묻는가

이 질문은 클러스터 전체를 시스템적으로 볼 수 있는지를 테스트한다. "Pod에 liveness probe 넣어야죠"처럼 파편화된 답변이 아니라, 가용성, 보안, 관측성, 운영성을 균형 있게 고려하는지 본다.

답변 사고 순서 — 7가지 설계 축

① 컨트롤 플레인 가용성

etcd와 API 서버를 3개 이상의 마스터 노드에 분산해야 한다. etcd는 홀수 노드로 쿼럼을 유지한다. 마스터가 하나면 컨트롤 플레인 장애 시 배포 불가능 (기존 Pod는 살아있지만 관리 불가).

흔한 오해: "Pod가 살아있으면 장애가 아니다"라고 생각하지만, 컨트롤 플레인이 죽으면 신규 배포, 스케일아웃, 롤백이 모두 막힌다.

② 노드 설계

워커 노드는 여러 가용 영역(AZ)에 분산한다. 단일 AZ 장애 시 서비스가 살아있어야 하므로. Pod Anti-Affinity Rule로 같은 서비스의 Pod가 동일 노드나 AZ에 집중되지 않게 해야 한다.

③ 네임스페이스 분리와 RBAC

팀/서비스/환경(dev/staging/prod)별로 네임스페이스를 분리하고, RBAC으로 최소 권한을 부여한다. "개발자가 prod 네임스페이스에서 kubectl delete pod를 실행할 수 없어야 한다".

④ Resource Requests/Limits

모든 Pod에 CPU/메모리 requests와 limits를 설정해야 한다. limits 없는 Pod는 같은 노드의 다른 Pod를 OOM으로 죽일 수 있다. requests는 스케줄러가 노드를 선택하는 기준이 된다.

⑤ 이미지 보안

신뢰할 수 있는 레지스트리(ECR, GCR, private registry)에서만 이미지를 Pull하도록 제한한다. 이미지 취약점 스캐닝(Trivy, Snyk)을 CI에 넣어 배포 전 검사한다. latest 태그 금지 — 재현 불가능한 배포의 원인이 된다.

⑥ 관측성 기반 마련

클러스터부터 애플리케이션까지 모든 레이어의 메트릭, 로그, 트레이싱을 처음부터 구성한다. Prometheus + Grafana, Loki, Jaeger / OpenTelemetry. 나중에 추가하려면 비용이 크다.

⑦ 배포 전략 정의

서비스별로 RollingUpdate 전략과 롤백 기준을 정의한다. maxSurge/maxUnavailable, readinessProbe 연동, 롤백 발동 조건. 프로덕션에서 처음 배포할 때 이게 없으면 장애가 나고서야 생각하게 된다.

핵심 답변 예시 (간결 버전):

"가장 먼저는 컨트롤 플레인 가용성입니다. etcd를 3개 이상으로 구성하고 마스터를 다중화합니다. 그다음 노드를 여러 AZ에 분산하고 Pod Anti-Affinity로 같은 서비스가 한 AZ에 몰리지 않게 합니다.

기술적으로는 모든 Pod에 resource requests/limits을 강제하는 것과 RBAC으로 최소 권한을 부여하는 것을 초기부터 해야 합니다. 나중에 수백 개의 Pod에 retroactively 추가하는 건 매우 어렵습니다. 관측성도 마찬가지로 처음부터 Prometheus + 중앙 로깅을 구성하지 않으면 첫 장애 때 눈이 멉니다."

Senior-level Follow-up

면접관: "etcd가 장애나면 어떤 일이 일어납니까? 서비스 중단이 발생하나요?"

시니어 답변 방향: "etcd는 클러스터의 상태 저장소입니다. etcd가 죽으면 API 서버가 상태를 읽고 쓸 수 없어서 새 Pod 생성, 스케일링, 배포가 불가능합니다. 하지만 이미 실행 중인 Pod는 계속 작동합니다. kubelet이 로컬에서 Pod를 유지하기 때문입니다. 즉, 유저는 서비스를 이용할 수 있지만 운영팀은 아무것도 변경할 수 없는 상태가 됩니다. 이것이 가장 위험한 이유입니다 — 장애는 안 난 것 같지만 사실 손발이 묶인 상태이기 때문입니다."

5. 시니어의 정신 모델 — 컨트롤 플레인과 데이터 플레인

Kubernetes를 다루는 시니어 개발자의 가장 중요한 멘탈 모델은 컨트롤 플레인과 데이터 플레인의 분리다. 이 구분 없이는 장애 상황에서 올바른 진단을 할 수 없다.

컨트롤 플레인 (Control Plane)

클러스터의 두뇌. "원하는 상태(desired state)"를 정의하고 현재 상태를 그것에 맞추는 역할.

  • kube-apiserver: 모든 클러스터 조작의 진입점. kubectl, 컨트롤러, 스케줄러가 모두 API 서버를 통해 통신.
  • etcd: 클러스터 전체 상태를 저장하는 분산 key-value 스토어. 이게 없으면 클러스터가 현재 상태를 모른다.
  • kube-scheduler: 새 Pod가 어느 노드에서 실행될지 결정. CPU/메모리 요청량, taint/toleration, affinity 규칙 등을 고려.
  • kube-controller-manager: ReplicaSet Controller, Deployment Controller 등 다양한 컨트롤러의 집합. 현재 상태와 원하는 상태의 차이를 지속적으로 보정(reconciliation loop).

데이터 플레인 (Data Plane)

실제 워크로드가 실행되는 워커 노드들. 컨트롤 플레인의 지시를 받아 실행.

  • kubelet: 각 노드에서 실행. API 서버로부터 Pod spec을 받아 컨테이너 런타임(containerd, Docker)을 통해 컨테이너를 실행하고 상태를 API 서버에 보고.
  • kube-proxy: 각 노드에서 Service의 네트워크 규칙을 관리. iptables 또는 IPVS로 Service IP → Pod IP 라우팅을 처리.
  • 컨테이너 런타임: containerd, CRI-O. 실제 컨테이너를 실행하는 엔진.

왜 이 구분이 중요한가 — 장애 진단 예시

상황: "새 배포를 했는데 Pod가 Pending 상태로 계속 있다."

  • 컨트롤 플레인 문제라면: 스케줄러가 Pod를 어느 노드에 배치할지 결정 못하는 것. kubectl describe pod에서 "no nodes available" 또는 "insufficient cpu" 같은 메시지.
  • 데이터 플레인 문제라면: 노드에 스케줄됐지만 kubelet이 컨테이너를 시작하지 못하는 것. ImagePullBackOff (이미지를 못 받아옴), 볼륨 마운트 실패 등.

이 구분을 모르면 "Pod가 안 뜬다"는 현상만 보고 랜덤하게 디버깅한다. 컨트롤 플레인/데이터 플레인을 아는 시니어는 첫 번째 명령어부터 정확한 곳을 본다.

6. 자기 점검 Rubric

□ 컨테이너가 VM이 아닌 이유를 "Linux 네임스페이스 + cgroup"으로 설명할 수 있다
□ 컨테이너의 보안 제약(커널 공유)을 명시하고, VM이 더 나은 케이스를 2가지 이상 말할 수 있다
□ Docker 단독 운영의 한계를 5가지 이상 나열하고, K8s가 각각 어떻게 해결하는지 연결할 수 있다
□ K8s 도입이 오히려 과잉인 케이스를 구체적으로 설명할 수 있다
□ 컨트롤 플레인의 4가지 컴포넌트(api-server, etcd, scheduler, controller-manager)의 역할을 설명할 수 있다
□ etcd 장애 시 서비스는 살아있지만 운영이 불가능한 이유를 설명할 수 있다
□ 프로덕션 클러스터 설계 시 7가지 축(가용성, 노드, RBAC, 리소스, 보안, 관측성, 배포전략)을 언급할 수 있다
□ Pod가 Pending인 경우 컨트롤 플레인 문제와 데이터 플레인 문제를 구분해 진단 방향을 말할 수 있다
□ 실무 경험과 연결해 "우리 팀에서는..." 또는 "실제로 겪었던 상황은..."으로 답변을 마무리할 수 있다
