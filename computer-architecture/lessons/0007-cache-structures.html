1. 캐시 주소 분해 — Tag · Index · Offset

CPU가 메모리 주소를 발행하면 캐시 하드웨어는 이 주소를 세 필드로 분해합니다. 각 필드의 비트 수는 캐시 설계 파라미터(슬롯 수, 캐시 라인 크기)에 의해 결정됩니다.

32비트 주소 예시: ┌────────────────────┬──────────────┬──────────────┐ │ Tag │ Index │ Offset │ │ (t 비트) │ (s 비트) │ (b 비트) │ └────────────────────┴──────────────┴──────────────┘ b = log₂(캐시 라인 바이트 수) → 캐시 라인 내 바이트 위치 선택 s = log₂(셋(Set) 수) → 어느 셋(슬롯 그룹)에 접근할지 결정 t = 32 - s - b → 남은 비트는 모두 태그 (유효성 비교) 구체 예시: 캐시 라인 = 64 B → b = 6 (2⁶ = 64) 셋 수 = 256 → s = 8 (2⁸ = 256) 태그 = 32 - 8 - 6 = 18 비트
왜 주소를 세 부분으로 쪼개는가?
Offset은 "캐시 라인 안에서 몇 번째 바이트인가?"를 결정합니다. Index는 "어느 슬롯(또는 셋)을 먼저 검사할까?"를 하드웨어가 빠르게 계산하도록 합니다. Tag는 "그 슬롯에 있는 데이터가 정말 내가 찾는 주소에서 왔는가?"를 확인하는 식별자입니다. 세 필드 덕분에 캐시는 O(1) 시간에 히트/미스를 판정할 수 있습니다.

2. 직접 매핑 캐시 (Direct-Mapped Cache)

2-1. 구조

캐시 = S개의 슬롯 (각 슬롯: Valid 비트 + Tag + Data) 주소 → Index 필드 → 단 하나의 슬롯 결정 (1:1 매핑) Tag 필드 → 슬롯의 Tag와 비교 → 일치 + Valid = 히트 예: S = 8 슬롯, 캐시 라인 = 4 B → b=2, s=3, t=나머지 주소 A = 0b 00000 000 00 → Index=0, Tag=0 주소 B = 0b 00000 001 00 → Index=1, Tag=0 주소 C = 0b 00001 000 00 → Index=0, Tag=1 ← A와 같은 슬롯! C에 접근하면 슬롯 0의 A가 교체됨 → A를 다시 접근하면 미스 A,C,A,C,A,C ... 교대 접근 시 모든 접근이 미스 → Conflict Miss

2-2. 하드웨어 동작 순서

① CPU 주소 → Index 추출 → 해당 슬롯 읽기 ② Valid 비트가 1인가? No → Miss (Compulsory/Capacity) ③ Tag 일치하는가? No → Miss (Conflict 포함) ④ 위 두 조건 모두 Yes → Hit, Offset으로 바이트 선택하여 CPU에 반환 Miss 처리: 하위 메모리에서 캐시 라인 전체(64 B) 로드 해당 슬롯에 덮어씀 (기존 내용 버림) Valid=1, Tag=새 태그 기록

2-3. 장단점

항목내용
장점하드웨어 단순. 주소 비교기 1개만 필요. 접근 속도 빠름. 면적·전력 작음
단점같은 Index를 공유하는 주소끼리 Conflict Miss 발생. 히트율 낮을 수 있음
사용 예일부 L1 캐시 (Instruction Cache), 마이크로컨트롤러

3. 완전 연관 캐시 (Fully Associative Cache)

3-1. 구조

모든 슬롯이 하나의 거대한 셋을 이룸 (Index 필드 없음) 주소 = [ Tag (32-b 비트) | Offset (b 비트) ] 주소가 들어오면 모든 슬롯의 Tag와 동시에(병렬로) 비교 → 일치하는 슬롯 발견 + Valid=1 = Hit 예: S = 8 슬롯 슬롯 0: Tag=A, Data=... 슬롯 1: Tag=B, Data=... ... 슬롯 7: Tag=H, Data=... 동일 주소 비트를 모든 슬롯의 Tag와 한 번에 비교

3-2. 장단점

항목내용
장점Conflict Miss 없음. 어떤 주소든 빈 슬롯 어디에나 들어갈 수 있어 히트율 최대
단점슬롯 수만큼 병렬 비교기(CAM: Content Addressable Memory) 필요. 면적·전력 매우 큼. 슬롯 수 커질수록 비용 폭증
사용 예TLB (Translation Lookaside Buffer — 가상 메모리 Day 15에서 다룸), 소용량 특수 캐시
CAM(Content Addressable Memory)이란?
일반 메모리는 주소(위치)로 데이터를 찾지만, CAM은 데이터의 내용(값)으로 위치를 찾습니다. "이 Tag 값이 어느 슬롯에 저장되어 있는가?"를 하드웨어가 한 사이클에 모든 슬롯을 병렬 비교하여 답합니다. 슬롯이 수천 개인 캐시에서 CAM을 쓰면 면적과 전력이 현실적이지 않기 때문에 완전 연관 캐시는 소용량(TLB, 수십~수백 항목 수준)에만 사용합니다.

4. 집합 연관 캐시 (Set-Associative Cache)

4-1. 구조

캐시를 S개의 "셋(Set)"으로 나누고, 각 셋은 E개의 슬롯(Way)을 가짐 총 슬롯 수 = S × E 주소 분해: [ Tag (t비트) | Index (s비트, s=log₂S) | Offset (b비트) ] 동작: ① Index → 하나의 셋 선택 ② 그 셋 안의 E개 슬롯 Tag를 병렬로 비교 ③ 일치 + Valid → Hit 예: 4-way Set-Associative, S=4 셋, E=4 Way → 16 슬롯 총 셋 0: [ Way0: Tag=A ] [ Way1: Tag=E ] [ Way2: Tag=I ] [ Way3: Tag=M ] 셋 1: [ Way0: Tag=B ] [ Way1: Tag=F ] [ Way2: Tag=J ] [ Way3: Tag= ] 셋 2: [ Way0: Tag=C ] [ Way1: Tag=G ] [ Way2: Tag=K ] [ Way3: Tag= ] 셋 3: [ Way0: Tag=D ] [ Way1: Tag=H ] [ Way2: Tag=L ] [ Way3: Tag= ] 같은 Index 0을 공유하는 주소 A, E, I, M이 있어도 4개의 Way에 각각 들어가 Conflict Miss 없음 5번째 주소가 셋 0에 오면 비로소 교체 필요

4-2. Way 수에 따른 스펙트럼

직접 매핑 ←────────────────────────────────→ 완전 연관 1-way 2-way 4-way 8-way N-way (N=슬롯수) S=전체슬롯 S/2 S/4 S/8 S=1 셋 Way 수 증가 → Conflict Miss 감소 → 히트율 상승 Way 수 증가 → 병렬 비교기 증가 → 면적·전력·지연 증가 현대 CPU 캐시: L1 D-cache: 4-way 또는 8-way L2 cache: 8-way 또는 16-way L3 cache: 16-way 또는 32-way

4-3. 세 가지 캐시 구조 비교

5. 교체 정책 (Replacement Policy)

집합 연관 캐시에서 셋이 꽉 찼을 때 새 데이터를 위해 어느 Way(슬롯)를 교체할지 결정하는 정책입니다. 직접 매핑은 슬롯이 하나뿐이므로 선택의 여지가 없습니다.

LRU (Least Recently Used — 최근 최소 사용): 가장 오랫동안 접근되지 않은 Way를 교체 시간적 지역성을 가장 잘 활용 구현: 각 Way에 최근 접근 순위를 기록하는 카운터 필요 단점: Way 수가 클수록 카운터 관리 비용 증가 실제로는 의사-LRU(Pseudo-LRU, PLRU)로 근사화 Random (무작위 교체): 임의의 Way를 교체 구현 단순 (하드웨어 PRNG) LRU와 히트율이 비슷하거나 경우에 따라 더 높은 경우도 있음 예측 불가능성이 보안(Spectre 완화) 측면에서 유리 FIFO (First-In, First-Out — 선입 선출): 가장 오래 전에 캐시에 들어온 Way를 교체 구현: 순환 포인터(포인터 하나)면 충분 LRU보다 단순하지만 히트율 다소 낮음
현대 CPU는 주로 Pseudo-LRU를 사용
정확한 LRU는 N-way 캐시에서 N! 상태를 추적해야 하지만, 트리 기반 PLRU는 N-1 비트만으로 LRU에 가까운 동작을 달성합니다. Intel 코어 아키텍처는 L3에서 Adaptive Replacement Cache(ARC)나 PLRU 변형을 씁니다.

6. 주소 분해 계산 예제

문제: 32비트 주소 체계 캐시 라인 크기: 64 B 2-way Set-Associative 캐시 총 캐시 용량: 32 KB 풀이: ① b (Offset 비트) = log₂(64) = 6 비트 ② 총 슬롯 수 = 캐시 용량 / 라인 크기 = 32 KB / 64 B = 512 슬롯 ③ 셋 수 S = 총 슬롯 수 / Way 수 = 512 / 2 = 256 ④ s (Index 비트) = log₂(256) = 8 비트 ⑤ t (Tag 비트) = 32 - 8 - 6 = 18 비트 주소 [31:0] = [ Tag(18비트) | Index(8비트) | Offset(6비트) ] 검증: 캐시 슬롯당 저장 크기 = 1(Valid) + 18(Tag) + 64×8(Data) = 531 비트 총 512 슬롯 × 531 = 약 272 KB (실제 하드웨어 면적) 데이터만 = 512 × 64 B = 32 KB (= 지정된 용량 ✓)

오늘의 핵심 요약

  • 주소 = Tag + Index + Offset: Index로 슬롯(셋) 위치 결정, Tag로 내용 일치 확인, Offset으로 바이트 선택
  • 직접 매핑: 주소 → 단 하나의 슬롯. 하드웨어 단순. Conflict Miss 취약
  • 완전 연관: 어디든 들어갈 수 있음. Conflict Miss 없음. CAM 때문에 하드웨어 비용 폭증 → TLB 등 소용량에만 사용
  • 집합 연관(N-way): 절충안. 셋 선택 후 그 셋 안의 N개 슬롯 비교. 현대 CPU 표준
  • Way 수 증가 → Conflict Miss ↓, 하드웨어 비용 ↑ (절충 필요)
  • LRU: 시간적 지역성 최대 활용. 현대 CPU는 의사-LRU(PLRU)로 근사
  • 캐시 용량 계산: 셋 수 S = (총 용량) / (라인 크기 × Way 수), Index 비트 = log₂S
