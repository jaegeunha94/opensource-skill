1. 읽기 히트는 쉽다 — 쓰기는 복잡하다

Day 7에서 캐시 히트·미스와 교체 정책을 배웠습니다. 읽기(Load) 히트는 단순합니다: 캐시에 데이터가 있으면 바로 CPU에 반환합니다. 그런데 쓰기(Store) 히트가 발생하면 어떻게 해야 할까요?

캐시에 수정한 데이터를 언제 아래 메모리 계층에 반영하느냐가 쓰기 정책(Write Policy)입니다. 이 선택은 메모리 트래픽, 일관성, 전력 소비에 직접 영향을 줍니다.

핵심 질문
캐시에 쓴 데이터가 아래 메모리(L2, RAM)와 다른 상태로 유지되는 것을 허용할 것인가?
허용하면 성능은 올라가지만 일관성 관리가 복잡해집니다.

2. Write-Through vs Write-Back

2-1. Write-Through (즉시 반영)

CPU → 캐시 슬롯 수정 → 동시에 아래 메모리(L2 또는 RAM)에도 즉시 기록 예) CPU가 주소 0x1000에 값 42를 저장: ┌────────────────────────────────────────────────────┐ │ 1. L1 캐시 슬롯 업데이트 (값 = 42) │ │ 2. L2/RAM 주소 0x1000에도 즉시 42 기록 │ └────────────────────────────────────────────────────┘ 캐시 슬롯에 "Dirty 비트" 불필요 — 캐시와 메모리 항상 동기화
항목내용
장점캐시와 메모리 항상 일치 → 일관성 관리 단순, 캐시 교체 시 그냥 버려도 됨
단점모든 쓰기가 하위 메모리로 전달 → 버스 트래픽 증가, 느린 쓰기 레이턴시
완화 기법Write Buffer: 쓰기 요청을 버퍼에 모아 메모리 버스를 비동기로 처리
사용 예일부 임베디드 시스템, 캐시가 작고 단순해야 하는 환경

2-2. Write-Back (지연 반영)

CPU → 캐시 슬롯 수정 → 슬롯에 Dirty 비트만 설정 (메모리 업데이트 보류) → 해당 슬롯이 교체될 때만 수정 내용을 아래 메모리에 기록 (Write-Back) 예) CPU가 주소 0x1000에 42 저장: ┌────────────────────────────────────────────────────────────────┐ │ 1. L1 캐시 슬롯 업데이트 (값 = 42, Dirty 비트 = 1) │ │ 2. 메모리 접근 없음 ← 핵심 이점! │ │ 3. 나중에 이 슬롯이 교체될 때: 메모리에 42 기록 후 슬롯 해제 │ └────────────────────────────────────────────────────────────────┘ Dirty 비트 = 0: 캐시와 메모리 동일 → 교체 시 그냥 버려도 됨 Dirty 비트 = 1: 캐시가 메모리보다 최신 → 교체 전 반드시 Write-Back 필요
항목내용
장점반복 쓰기 시 메모리 트래픽 최소화 (같은 슬롯을 100번 수정해도 메모리 접근은 교체 시 1번)
단점Dirty 비트 필요, 교체 시 추가 지연(Dirty 슬롯은 메모리 쓰기 완료 후 교체). 멀티코어 캐시 일관성 복잡
사용 예현대 CPU L1/L2/L3 대부분 (Intel Core, AMD Zen 시리즈 등)

2-3. 두 정책 비교

시나리오: 동일 캐시 슬롯(주소 A)에 10번 연속 쓰기 Write-Through: 쓰기 1 → 메모리 접근 1 쓰기 2 → 메모리 접근 2 ... 쓰기 10 → 메모리 접근 10 총 메모리 쓰기: 10회 (느린 DRAM 접근 10번) Write-Back: 쓰기 1~10 → L1 캐시 슬롯만 수정, Dirty=1 슬롯 교체 시 → 메모리 접근 1회 (최종 값만 저장) 총 메모리 쓰기: 1회 (DRAM 접근 1번) 결론: 쓰기가 빈번할수록 Write-Back이 유리

3. 쓰기 미스 처리 — Write-Allocate vs No-Write-Allocate

지금까지는 쓰기 히트를 다뤘습니다. 그렇다면 쓰기 미스(캐시에 없는 주소에 쓰기)는 어떻게 처리할까요?

3-1. Write-Allocate (쓰기 미스 → 캐시에 올린 뒤 쓰기)

CPU가 캐시에 없는 주소 0x2000에 쓰기 요청 (Write Miss) Write-Allocate 처리 순서: ① 메모리에서 0x2000이 포함된 캐시 라인 전체 읽기 (64 B Fetch) ② 캐시 슬롯에 로드 ③ 원하는 위치(0x2000)에 새 값 기록 ④ Dirty 비트 = 1 (Write-Back 정책과 쌍으로 사용) 이점: 이후 같은 주소 주변을 읽거나 쓸 때 캐시 히트 단점: 쓰기 미스마다 캐시 라인 Fetch 비용 발생

3-2. No-Write-Allocate (쓰기 미스 → 캐시 로드 없이 바로 메모리에 쓰기)

CPU가 캐시에 없는 주소 0x2000에 쓰기 요청 (Write Miss) No-Write-Allocate 처리 순서: ① 메모리의 0x2000에 직접 값 기록 (캐시 미사용) ② 캐시 슬롯 변경 없음 이점: 쓰기 미스 시 캐시 슬롯 오염 없음 (읽기 데이터가 밀려나지 않음) 단점: 이후 0x2000 주변을 다시 쓸 때도 캐시 이점 없음 조합: Write-Through와 주로 함께 사용

3-3. 실용적 조합

쓰기 정책 조합 일반적 사용 이유
Write-Back + Write-Allocate 현대 CPU (L1/L2/L3) 반복 접근 지역성 최대 활용, 메모리 트래픽 최소화
Write-Through + No-Write-Allocate 임베디드, 단순 캐시 구현 단순, 캐시-메모리 일관성 유지 용이
직관적 이해
Write-Back + Write-Allocate는 "쓰다 보면 나중에 또 쓸 테니 일단 캐시에 올려두자"는 낙관적 전략입니다.
Write-Through + No-Write-Allocate는 "한 번 쓰고 안 건드릴 데이터는 굳이 캐시를 오염시키지 말자"는 보수적 전략입니다.
스트리밍(순차적 대량 쓰기) 워크로드에서는 No-Write-Allocate가 유리할 수 있습니다.

4. 다단 캐시 (Multi-Level Cache)

4-1. L1 / L2 / L3 구조와 역할

CPU Core │ ▼ L1 Cache ── 가장 작고 빠름: 32–64 KB, ~4 사이클 │ Core별 전용 (Private). 명령어(I-Cache) + 데이터(D-Cache) 분리 │ ▼ L2 Cache ── 중간 크기·속도: 256 KB – 1 MB, ~12 사이클 │ Core별 전용 (Private) 또는 그룹 공유 │ ▼ L3 Cache ── 크고 느림: 8 – 64 MB, ~40 사이클 │ 모든 Core 공유 (Shared, Last-Level Cache = LLC) │ ▼ Main Memory (DRAM) ── 매우 큼: GB 단위, ~100–200 사이클 │ ▼ SSD / Disk (가상 메모리 영역) 각 단계 미스 시 다음 단계에서 캐시 라인 Fetch

계층마다 크기와 레이턴시의 균형을 맞춰 전체 평균 접근 시간(AMAT)을 낮춥니다.

AMAT (Average Memory Access Time) 공식: AMAT = Hit Time_L1 + Miss Rate_L1 × (Hit Time_L2 + Miss Rate_L2 × (Hit Time_L3 + Miss Rate_L3 × Hit Time_Mem)) 예시 (단순화): L1 Hit Time = 4사이클, Miss Rate = 5% L2 Hit Time = 12사이클, Miss Rate = 10% L3 Hit Time = 40사이클, Miss Rate = 1% Mem Hit Time = 150사이클 AMAT = 4 + 0.05 × (12 + 0.10 × (40 + 0.01 × 150)) = 4 + 0.05 × (12 + 0.10 × 41.5) = 4 + 0.05 × (12 + 4.15) = 4 + 0.05 × 16.15 = 4 + 0.81 ≈ 4.81 사이클 ← L1이 99% 이상 결정
왜 L1 히트율이 압도적으로 중요한가?
AMAT 공식에서 L2·L3 레이턴시는 L1 미스율에 의해 가중됩니다. L1 미스율이 5%에서 2%로 줄면 AMAT이 크게 개선됩니다. 그래서 컴파일러 최적화(루프 순서 변경, 캐시 친화적 자료구조)는 L1 히트율을 높이는 데 집중합니다.

5. 다단 캐시 포함 정책

L1과 L2, L2와 L3 사이에서 같은 캐시 라인이 여러 계층에 중복 존재할 수 있을까요? 이를 결정하는 것이 포함 정책(Inclusion Policy)입니다.

5-1. Inclusive (포함 정책)

규칙: L1에 있는 데이터는 반드시 L2에도 존재한다 (L1 ⊆ L2) L1 캐시: [A] [B] [C] L2 캐시: [A] [B] [C] [D] [E] [F] ... ← L1 내용이 L2에 중복 포함 장점: · L1 미스 시 L2에서 즉시 찾을 수 있음 (백킹 스토어 보장) · 캐시 스누핑(Snoop) 시 L2만 확인해도 됨 → 멀티코어 일관성 프로토콜 단순화 · L2 교체 시 L1에 알림(Invalidation) 필요 단점: · L1 용량이 L2 유효 용량에 "이중으로 카운트"됨 → 실제 고유 데이터 저장 용량 = L2 - L1 (L1 중복분 제외) · L1 크기가 증가하면 L2 유효 용량 손실 증가 사용 예: 일부 Intel 아키텍처 (Skylake 이전 L3)

5-2. Exclusive (배타 정책)

규칙: 동일 캐시 라인은 단 하나의 계층에만 존재한다 (L1 ∩ L2 = ∅) L1 캐시: [A] [B] [C] L2 캐시: [D] [E] [F] [G] ... ← L1과 겹치는 항목 없음 L1 미스 발생 시: L2에서 찾은 라인을 L1으로 이동 (Swap — L1의 교체 라인이 L2로) 장점: · 전체 L1+L2 용량을 완전히 활용 (중복 없음) · 소용량 L1도 실제 효과적 용량 극대화 단점: · L1 미스 시 Swap 연산 복잡도 · 멀티코어 스누핑 시 L2도 확인해야 함 → 일관성 프로토콜 복잡 사용 예: AMD Zen 시리즈 (L1↔L2 Exclusive), 일부 임베디드 CPU

5-3. NINE (Non-Inclusive Non-Exclusive / 비포함-비배타)

규칙: L1과 L2가 겹쳐도 되고 안 겹쳐도 된다 — 명시적 포함 관계 없음 L1 캐시: [A] [B] [C] L2 캐시: [B] [D] [E] ... ← B는 중복, A·C는 L1만, D·E는 L2만 동작: · L1 미스 → L2 확인, 있으면 L1으로 복사 (L2에서 제거하지 않음) · L1 교체 → L2에 알림 없이 그냥 제거 (L2에 복사본 있을 수도, 없을 수도) 장점: · Inclusive/Exclusive 간 절충 → 구현 유연성 · L3(LLC)로 많이 사용: 각 Core L2와 겹치는 부분이 자연스럽게 공유 데이터 역할 단점: · 일관성 프로토콜이 L1·L2 양쪽을 추적해야 할 수 있음 사용 예: Intel Skylake 이후 L3 (NINE LLC)

5-4. 세 정책 비교 요약

정책 중복 허용 유효 용량 스누핑 복잡도 대표 사용처
Inclusive L1 ⊆ L2 L2 − L1 (손실 있음) 낮음 (L2만 확인) Intel Skylake 이전 L3
Exclusive 완전 분리 L1 + L2 (최대) 중간 (스왑 필요) AMD Zen L1↔L2
NINE 가변 가변 (L2 이상) 높음 (양쪽 추적) Intel Skylake 이후 L3

6. 캐시 쓰기 정책과 실제 성능

6-1. 벤치마크 관점에서의 영향

[Write-Heavy 워크로드 — 동일 위치 반복 수정] Write-Through: 매 쓰기마다 메모리 버스 사용 → 버스 포화 시 성능 급락 Write-Back: 교체 시에만 메모리 접근 → 버스 트래픽 대폭 감소 [Sequential 대량 쓰기 — 스트리밍] Write-Allocate: 캐시 라인 Fetch 후 쓰기 → 순차 쓰기엔 불필요한 읽기 No-Write-Allocate: 직접 메모리 기록 → 스트리밍에 더 효율적 → 컴파일러 NT(Non-Temporal) 스토어 명령어(예: movntps)는 이를 직접 활용 [캐시 용량 초과 워크로드] Inclusive L3: L1 크기가 L3 용량에서 중복으로 "차감" → 대형 작업에서 유효 용량 감소 NINE L3: 중복 제한 없음 → 더 많은 고유 데이터 보관 가능

6-2. 프로그래머 관점: 캐시 친화적 코드

나쁜 예 (Column-Major 접근 — C 언어 2D 배열): for (j = 0; j < N; j++) // 외부 루프: 열(column) for (i = 0; i < N; i++) // 내부 루프: 행(row) sum += A[i][j]; // 메모리상 N칸씩 점프 → 캐시 라인 낭비 좋은 예 (Row-Major 접근): for (i = 0; i < N; i++) // 외부 루프: 행(row) for (j = 0; j < N; j++) // 내부 루프: 열(column) sum += A[i][j]; // 메모리상 연속 접근 → 캐시 라인 완전 활용 왜 차이가 나는가? C의 2D 배열은 행 우선(Row-Major) 저장 A[0][0] A[0][1] ... A[0][N-1] A[1][0] ... (연속 메모리) 내부 루프가 j(열)를 순회하면 연속 주소 → 같은 캐시 라인 재사용 내부 루프가 i(행)를 순회하면 N칸씩 점프 → 매번 다른 캐시 라인
Write 정책과 코드 최적화의 연결
Write-Back + Write-Allocate를 쓰는 현대 CPU에서 같은 메모리 위치를 반복 수정하는 코드는 첫 접근(미스 + Fetch) 이후 캐시 내에서 빠르게 수정됩니다. 반면 대용량 배열을 순차 1회 기록하는 스트리밍 코드는 Non-Temporal 스토어를 쓰거나 컴파일러 힌트(prefetch, restrict)를 활용하면 캐시 오염을 줄일 수 있습니다.

  • Write-Back: 캐시에만 기록(Dirty 비트), 교체 시 메모리 반영 → 트래픽 감소, 현대 CPU 표준
  • Write-Allocate: 쓰기 미스 시 캐시 라인 Fetch 후 쓰기 → Write-Back과 짝
  • No-Write-Allocate: 쓰기 미스 시 메모리 직접 기록 → Write-Through와 짝, 스트리밍에 유리
  • 다단 캐시 AMAT: L1 히트율이 압도적 영향 → L1 최적화(코드 지역성)가 핵심
  • Inclusive: L1 ⊆ L2, 스누핑 단순 but 유효 용량 손실
  • Exclusive: L1∩L2=∅, 용량 최대화 but 스왑 복잡
  • NINE: 가변 중복, 현대 Intel LLC 방식 — 유연하지만 일관성 관리 복잡
  • 코드 최적화: 행 우선 접근, 루프 순서 변경으로 캐시 라인 재사용률 극대화
