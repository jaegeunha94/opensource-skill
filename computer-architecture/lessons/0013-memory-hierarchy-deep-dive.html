1. 핵심 면접 질문

Primary Question

"메모리 접근이 성능 병목이 되는 이유는 무엇입니까? register, cache, RAM, disk 사이의 속도 차이를 근거로 설명해보세요."

Variant 1

"메모리의 latency와 bandwidth는 어떻게 다른가요? 왜 이 둘을 구분해서 얘기해야 하나요?"

Variant 2

"코어를 4개에서 8개로 늘렸는데 메모리를 많이 쓰는 작업의 성능이 기대만큼 오르지 않았습니다. 왜 이런 일이 생길 수 있을까요?"

2. 왜 면접에서 묻는가

이 질문은 "메모리가 느리다"는 말을 숫자와 원리로 뒷받침할 수 있는가를 확인합니다. 면접관이 듣고 싶은 것은 "메모리는 원래 느려요"라는 뭉뚱그린 답이 아니라, 어느 계층이 얼마나 느린지, 왜 그 차이가 CPU 설계상 필연적인지, 그리고 latency 문제와 bandwidth 문제를 구분해서 진단할 수 있는지입니다.

면접관이 이 질문으로 평가하는 것:

  • 정량적 감각(order-of-magnitude sense) — L1, L2, L3, RAM, SSD, HDD 사이의 속도 차이가 대략 몇 자릿수인지 감으로 알고 있는가. Day 1에서 다룬 표를 실제로 체화했는가.
  • latency vs bandwidth를 구분하는 능력 — "느리다"는 말이 "요청 하나에 시간이 오래 걸린다(latency)"는 뜻인지 "초당 옮길 수 있는 데이터양이 적다(bandwidth)"는 뜻인지 구분해서 말하는가. 이 둘은 원인도 다르고 해결책도 다릅니다.
  • "메모리 벽(memory wall)"에 대한 이해 — CPU 클럭 속도는 수십 년간 빠르게 개선되어 온 반면 DRAM의 latency는 상대적으로 덜 개선되어, CPU와 메모리 사이의 속도 격차가 구조적으로 벌어져 왔다는 배경을 아는가.
  • 멀티코어 환경에서의 확장 한계를 추론하는 능력 — 코어를 늘려도 메모리 대역폭은 보통 공유 자원이라, 메모리를 많이 쓰는 워크로드는 코어 수만큼 선형으로 빨라지지 않는다는 것을 설명할 수 있는가.

주니어 답변의 전형적인 패턴: "캐시는 빠르고 메모리는 느리다"까지만 말하고 멈춥니다. 10년차 이상은 정확한 지연 시간 자릿수, latency와 bandwidth의 차이, 그리고 이 차이가 멀티코어·대용량 데이터 처리 설계에 주는 실제 영향까지 연결해서 말합니다.

3. 전제 개념 — 쉬운 한국어로

복습 — 메모리 계층은 "속도와 용량의 트레이드오프 사다리"다

Day 1에서 다뤘듯 CPU에 가까울수록 빠르지만 용량이 작고 비싸며(register, L1), 멀어질수록 느리지만 용량이 크고 저렴합니다(RAM, SSD, disk). 이 사다리가 존재하는 이유는 "모든 데이터를 register만큼 빠른 저장 장치에 담기엔 비용이 감당이 안 되기 때문"입니다. 그래서 하드웨어는 자주 쓰는 데이터를 빠른 계층에 자동으로 복사해두는 캐시 구조로 이 문제를 절충합니다.

Latency(지연 시간)란 무엇인가

Latency는 "요청 하나를 보내고 응답을 받기까지 걸리는 시간"입니다. "이 데이터 하나를 읽어와" 라고 요청했을 때 결과가 돌아오기까지 걸리는 시간이 바로 latency입니다. 택배로 비유하면, 주문 하나를 넣고 물건이 도착하기까지 걸리는 시간입니다.

Bandwidth(대역폭)란 무엇인가

Bandwidth는 "단위 시간당 옮길 수 있는 데이터의 총량"입니다(보통 GB/s로 표기). 택배 비유를 이어가면, 하루에 트럭이 몇 대나 물건을 실어 나를 수 있는지에 해당합니다. 트럭 한 대의 배송 시간(latency)이 길어도, 트럭이 동시에 여러 대 다니면 하루 총 배송량(bandwidth)은 늘어날 수 있습니다. 즉 latency와 bandwidth는 서로 다른 축이며, 하나가 좋다고 다른 하나가 저절로 좋아지지 않습니다.

메모리 벽(Memory Wall)

1980~1990년대 이후 CPU의 처리 속도는 수십 년간 빠르게 향상되어 온 반면, DRAM의 접근 지연 시간은 상대적으로 훨씬 느리게 개선되었습니다. 그 결과 "CPU는 계산할 준비가 됐는데 메모리에서 데이터가 아직 안 왔다"며 CPU가 기다리는 상황이 구조적으로 심해졌습니다. 이 현상을 메모리 벽(memory wall)이라고 부릅니다. 캐시 계층, 프리페칭(prefetching), out-of-order 실행은 모두 이 메모리 벽이 성능에 주는 충격을 완화하기 위한 하드웨어적 장치입니다.

4. 메모리 계층과 bandwidth를 깊이 이해하기

실제 수치로 감을 잡아봅시다. 아래 표는 대표적인 현대 서버/데스크톱 CPU 기준의 대략적인 자릿수입니다(실제 값은 세대와 제품마다 다르지만, 자릿수 감각이 면접에서 중요합니다).

계층대략적인 Latency대략적인 Bandwidth비유
Register< 1 cycle (약 0.3ns)사실상 무제한(코어 내부)손에 쥔 도구
L1 Cache약 4 cycle (~1ns)매우 높음(코어 전용)바로 옆 서랍
L2 Cache약 10~20 cycle (~3~7ns)높음같은 방 안 선반
L3 Cache약 30~70 cycle (~10~20ns)코어끼리 공유, 중간 수준같은 건물 창고(공유)
DRAM (RAM)약 200~300 cycle (~70~100ns)수십 GB/s (채널 공유, 상한 존재)도시 외곽 물류창고
NVMe SSD약 10,000~100,000ns (수십 μs)수 GB/s고속도로로 타 도시 배송
HDD약 수 ms(탐색 포함)수백 MB/s선박으로 해외 배송

여기서 핵심은 register와 L1 사이의 격차보다 L3와 DRAM 사이의 격차가 훨씬 더 크다는 것입니다. 캐시 미스가 L3까지만 나면 수십 cycle이면 끝나지만, L3에서도 못 찾아 DRAM까지 가야 하면 수백 cycle이 걸립니다. CPU 입장에서는 "요청한 값이 왔는지"를 기준으로 계획을 세워야 하는데, 이 값이 정확히 어느 계층에서 나오느냐에 따라 대기 시간이 수십 배씩 달라집니다.

Memory-Level Parallelism(MLP) — latency를 "숨기는" 방법

한 번의 메모리 요청이 200 cycle 걸린다고 해서 CPU가 매번 200 cycle을 통으로 멈추는 것은 아닙니다. 현대 CPU는 out-of-order 실행과 하드웨어 프리페처(prefetcher)를 이용해, 서로 의존하지 않는 여러 메모리 요청을 동시에(겹쳐서) 날려놓고 기다립니다. 이렇게 동시에 처리 중인 미해결 메모리 요청의 수를 memory-level parallelism(MLP)이라고 합니다. MLP가 높으면 개별 요청의 latency는 그대로여도, 여러 요청이 동시에 진행되어 실질적인 처리량(throughput)은 크게 늘어날 수 있습니다.

하지만 MLP로 latency를 숨기는 데는 한계가 있습니다. 다음 요청이 이전 요청의 결과에 의존한다면(예: 포인터를 따라가는 연결 리스트 순회) 미리 요청을 날릴 수 없습니다 — 이전 값을 받아야 다음 주소를 알 수 있기 때문입니다. 이런 순차 의존적 접근(pointer chasing)은 MLP를 활용할 수 없어서 latency가 그대로 노출됩니다. 이 내용은 Day 4(cache line, pointer chasing)에서 더 깊게 다룹니다.

Bandwidth는 왜 "공유 자원"이고 왜 멀티코어에서 병목이 되는가

메모리 대역폭은 CPU 소켓당 몇 개의 메모리 채널로 정해진 상한이 있는 공유 자원입니다. 코어가 1개일 때는 그 코어 혼자 전체 대역폭을 거의 다 쓸 수 있지만, 코어가 8개, 16개로 늘어나면 모든 코어가 같은 메모리 버스와 채널을 나눠 써야 합니다. 각 코어가 메모리를 많이 읽고 쓰는 workload(memory-bound workload)를 동시에 돌리면, 코어 수를 늘려도 전체 처리량은 대역폭 상한에 막혀 선형으로 늘지 않고 어느 지점부터 정체됩니다. 이것이 "코어를 늘렸는데 메모리 집약적 작업의 성능이 기대만큼 안 오른다"는 현상의 하드웨어적 근거입니다.

Roofline 사고방식 — Compute-bound vs Memory-bound

어떤 코드가 느릴 때 "CPU 연산 능력이 부족해서 느린가(compute-bound)" 아니면 "메모리에서 데이터를 가져오는 속도가 부족해서 느린가(memory-bound)"를 구분하는 사고방식을 roofline model이라고 부릅니다. 핵심 지표는 arithmetic intensity(메모리에서 가져온 데이터 1바이트당 몇 번의 연산을 하는가)입니다. 가져온 데이터로 연산을 많이 하면 compute-bound에 가깝고, 가져오기만 하고 연산은 조금만 하면(예: 단순 합계, 복사) memory-bound에 가깝습니다. 이 개념은 Day 8(프로파일링)에서 실제 도구로 측정하는 방법과 함께 다시 다룹니다.

5. 구체 예시 — latency-bound vs bandwidth-bound 코드

다음 두 코드를 비교해봅시다. 둘 다 "메모리에서 값을 읽는다"는 점은 같지만, 성능을 제약하는 요인이 다릅니다.

// (A) 배열 순차 순회 — bandwidth-bound에 가까움
long sum = 0;
for (int i = 0; i < n; i++) {
    sum += arr[i];   // 다음 arr[i+1]은 arr[i]와 무관하게 미리 요청 가능
}

배열을 순차적으로 읽을 때는 각 원소를 읽는 요청이 서로 의존하지 않습니다. 하드웨어 프리페처는 다음에 읽을 주소를 쉽게 예측하고, 실제로 필요해지기 전에 미리 캐시로 가져다 놓습니다(MLP를 최대한 활용). 이 경우 개별 load의 latency는 거의 문제가 되지 않고, 전체 실행 시간은 "메모리에서 얼마나 빨리 대량의 데이터를 퍼올 수 있는가", 즉 bandwidth에 의해 좌우됩니다.

// (B) 연결 리스트(포인터) 순회 — latency-bound
struct Node { int value; Node* next; };
long sum = 0;
Node* cur = head;
while (cur != nullptr) {
    sum += cur->value;
    cur = cur->next;   // 다음 노드 주소는 cur->next를 "읽어야만" 알 수 있음
}

연결 리스트를 순회할 때는 다음 노드의 주소를 알기 위해 현재 노드를 먼저 읽어야 합니다. 즉 요청 N+1은 요청 N의 결과에 의존하는 순차 의존 체인입니다. 프리페처가 다음 주소를 미리 알 방법이 없고, out-of-order 실행도 이 의존성 자체를 없앨 수는 없습니다. 그 결과 각 노드 접근마다 캐시 미스가 나면 그 latency(예: DRAM 기준 약 200~300 cycle)가 그대로 누적되어 나타납니다. 원소 개수가 같아도 (A)보다 (B)가 훨씬 느린 이유는 알고리즘의 시간 복잡도가 아니라 메모리 접근 패턴이 MLP를 활용할 수 있는지 여부 때문입니다.

이건 Day 1에서 다룬 "배열이 연결 리스트보다 빠를 수 있다"는 결론에 왜라는 이유를 하나 더 추가한 것입니다: 캐시 라인 재사용(공간적 지역성)뿐 아니라, 순차 접근은 여러 요청을 동시에 진행시켜 latency를 숨길 수 있다는 이유도 있습니다. 이 지역성 관점은 Day 4에서 cache line 단위로 더 깊이 다룹니다.

6. 답변 사고 순서

  1. "느리다"를 latency 문제인지 bandwidth 문제인지부터 구분합니다. "요청 하나하나가 오래 걸리는가, 아니면 총량을 옮기는 속도가 부족한가?"
  2. 접근 패턴을 확인합니다. "요청들이 서로 독립적인가(순차 접근, 배열)? 아니면 앞 요청 결과에 의존하는가(포인터 체인)?"
  3. 계층별 자릿수로 근거를 댑니다. "L1은 몇 cycle, DRAM은 몇백 cycle 수준이라는 걸 알고 있고, 이번 병목이 어느 계층에서 발생하는지 추정합니다."
  4. 동시성(멀티코어) 관점을 더합니다. "이 워크로드가 여러 코어에서 동시에 메모리를 많이 쓴다면 대역폭 공유로 인해 코어를 늘려도 선형으로 빨라지지 않을 수 있습니다."
  5. 측정으로 확인하고 개선안을 제시합니다. "profiler로 memory-bound 비율, cache miss율을 확인한 뒤, 접근 패턴 변경(배치 처리, 자료구조 변경, prefetch 힌트)을 제안합니다."

7. 핵심 답변 예시

면접관: "메모리 접근이 성능 병목이 되는 이유는 무엇입니까?"

짧은 버전 (엘리베이터 피치)

SHORT ANSWER

CPU 속도는 수십 년간 메모리 속도보다 훨씬 빠르게 개선되어 왔고(메모리 벽), 그 결과 register/L1은 1 cycle 안팎이지만 DRAM은 수백 cycle이 걸리는 큰 격차가 생겼습니다. 이 격차는 latency(요청 하나의 대기 시간)와 bandwidth(초당 처리량)라는 두 축으로 나타나는데, 데이터를 순차적으로 접근하면 프리페처가 다음 데이터를 미리 가져와 latency를 어느 정도 숨길 수 있지만, 접근이 서로 의존적(예: 포인터 체인)이면 그 latency가 그대로 누적되어 병목이 됩니다.

깊이 있는 버전

DEEP ANSWER

배열을 순차 순회하는 코드와 연결 리스트를 순회하는 코드를 비교하면 이 차이가 잘 드러납니다. 배열 순회는 각 원소 접근이 서로 독립적이라 하드웨어 프리페처가 다음 캐시 라인을 미리 읽어오고, out-of-order 실행이 여러 load를 동시에 진행시켜(memory-level parallelism) 개별 latency를 상당 부분 숨깁니다. 이 경우 실제 실행 시간을 좌우하는 건 CPU와 메모리 사이의 bandwidth입니다.


반면 연결 리스트 순회는 다음 노드 주소를 알기 위해 현재 노드를 먼저 읽어야 하는 순차 의존 체인이라, 프리페처가 다음 주소를 예측할 수 없고 각 접근의 latency(캐시 미스 시 DRAM 기준 약 200~300 cycle)가 고스란히 누적됩니다. 그래서 같은 개수의 데이터를 순회해도 자료구조에 따라 체감 성능 차이가 매우 큽니다.


여기에 멀티코어 관점을 더하면, 메모리 대역폭은 소켓당 채널 수로 정해진 공유 자원이기 때문에, 메모리를 많이 쓰는 워크로드는 코어 수를 늘려도 대역폭 상한에 막혀 선형으로 확장되지 않습니다. 저는 실제로 배치 작업의 처리량이 코어를 늘려도 정체되는 걸 perf stat으로 memory bandwidth 사용률과 cache miss율을 확인해서, 계산 로직보다 메모리 접근 패턴(자료구조를 array-of-structs에서 struct-of-arrays로 재배치)이 원인임을 확인하고 개선한 경험이 있습니다.

이 답변이 좋은 이유:

  • "메모리가 느리다"를 latency와 bandwidth라는 서로 다른 축으로 분해함
  • 배열 vs 연결 리스트라는 구체 예시로 MLP와 순차 의존성의 차이를 보여줌
  • 단일 스레드를 넘어 멀티코어 대역폭 공유까지 확장해서 설명함
  • 실제 프로파일링 도구와 개선 경험으로 마무리함

8. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"메모리는 CPU보다 느리기 때문에 메모리에 접근할 때마다 병목이 생깁니다."

이 답변이 약한 이유

  • "느리다"는 말만 반복할 뿐, 어느 정도(cycle 수) 느린지, 어느 계층(L1/L2/L3/DRAM) 이야기인지 전혀 근거가 없습니다.
  • latency와 bandwidth를 구분하지 못해서, 병목의 종류에 따라 해결책이 달라진다는 점을 놓칩니다.
  • "메모리 접근마다 항상 병목"이라는 말은 틀렸습니다 — 순차 접근은 프리페칭으로 상당 부분 숨겨질 수 있습니다.
WEAK ANSWER — 또 다른 패턴

"멀티코어를 쓰면 메모리 작업도 코어 수만큼 빨라집니다."

이 답변이 약한 이유

  • 메모리 대역폭이 소켓당 채널 수로 정해진 공유 자원이라는 사실을 놓치고 있습니다.
  • CPU-bound 작업과 memory-bound 작업을 구분하지 않고 "코어가 많으면 무조건 좋다"는 단순화된 결론을 내립니다.
STRONG ANSWER 요소 체크리스트
  • register/L1/L2/L3/DRAM의 대략적인 cycle 수를 근거로 제시한다
  • latency와 bandwidth를 명확히 구분해서 설명한다
  • 순차 접근(MLP 활용 가능)과 의존적 접근(pointer chasing, MLP 활용 불가)을 구분한다
  • 멀티코어에서 메모리 대역폭이 공유 자원이라는 점을 언급한다
  • 실제 프로파일링 도구나 경험으로 근거를 뒷받침한다

9. Senior-level Follow-up Q&A

Q: Latency를 줄이는 것과 bandwidth를 늘리는 것 중, 어느 쪽이 더 쉬운 최적화인가요?

모범 답변: 일반적으로 bandwidth는 더 많은 메모리 채널, 더 넓은 버스, 여러 요청을 동시에 처리하는 방식으로 어느 정도 늘릴 수 있지만, latency는 물리적 거리(신호가 이동하는 시간)와 DRAM 자체의 접근 방식에 의해 정해지는 하드 리밋에 가깝습니다. 그래서 하드웨어/시스템 설계는 "latency 자체를 줄이기"보다 "latency를 여러 요청을 동시에 진행시켜 숨기기(MLP)"나 "애초에 계층을 통해 접근을 피하기(캐싱)" 전략을 주로 씁니다.

Q: Little's Law가 이 주제와 무슨 관련이 있나요?

모범 답변: Little's Law를 메모리 시스템에 적용하면 대략 "동시에 진행 중인 요청 수(concurrency) = bandwidth × latency"로 이해할 수 있습니다. 즉 latency가 긴 메모리 시스템에서 높은 bandwidth를 실제로 다 활용하려면, 그만큼 많은 요청을 동시에 미해결 상태로 띄워둘 수 있어야 합니다(충분한 MLP). CPU가 out-of-order 윈도우나 프리페처 큐 크기를 키우는 이유, 그리고 소프트웨어에서 배치 처리나 비동기 I/O로 여러 요청을 동시에 날리는 이유가 모두 이 원리와 연결됩니다.

Q: 왜 코어를 8개에서 16개로 늘렸는데 메모리 집약적 작업의 처리량이 2배가 안 되나요?

모범 답변: 메모리 대역폭은 소켓/노드당 정해진 채널 수로 상한이 있는 공유 자원입니다. 코어가 늘어날수록 각 코어가 나눠 쓸 수 있는 실질 대역폭은 줄어들고, 어느 시점부터는 CPU 연산 능력이 남아돌아도 메모리 버스가 포화되어 전체 처리량이 정체됩니다. 이럴 때는 코어를 더 늘리기보다, 메모리 접근량 자체를 줄이는 것(더 많은 것을 캐시에 남기기, 압축, 데이터 지역성 개선)이 더 효과적인 경우가 많습니다.

Q: 프리페칭은 항상 성능에 도움이 되나요?

모범 답변: 아닙니다. 프리페처는 앞으로 쓰일 것이라고 "예측"해서 미리 데이터를 가져오는 것이므로, 예측이 틀리면 실제로 쓰이지도 않을 데이터를 위해 귀중한 bandwidth를 소모하고, 캐시에 있던 다른 유용한 데이터를 밀어낼(cache pollution) 수 있습니다. 접근 패턴이 불규칙(랜덤 접근)할수록 하드웨어 프리페처의 예측 정확도는 떨어지고, 소프트웨어 프리페치 힌트를 남발해도 같은 문제가 생깁니다.

Q: NUMA(Non-Uniform Memory Access)는 이 주제와 어떻게 연결되나요?

모범 답변: 멀티소켓 서버에서는 각 CPU 소켓이 자신에게 물리적으로 가까운 메모리(local)와, 다른 소켓에 연결된 메모리(remote)를 모두 접근할 수 있지만 그 latency와 bandwidth가 다릅니다(remote 접근이 더 느림). 이것이 NUMA입니다. 메모리 대역폭이 공유 자원이라는 개념이 소켓을 넘어서면, "어느 코어가 어느 메모리 노드에서 데이터를 읽는가"까지 신경 써야 최적 성능을 낼 수 있습니다. 이 내용은 Day 6(멀티코어, cache coherence)와 함께 실무에서 자주 마주치는 확장 주제입니다.

10. 흔한 오해

오해 1

"메모리 접근은 항상 병목이다"

순차 접근이면 프리페칭과 MLP로 latency 상당 부분이 숨겨져 bandwidth가 허용하는 만큼은 빠르게 처리될 수 있습니다. 병목이 되는 전형적인 경우는 접근 패턴이 불규칙하거나(랜덤), 요청들이 서로 의존적일 때(포인터 체인)입니다.

오해 2

"Latency가 짧아지면 bandwidth도 자연히 좋아진다"

둘은 독립적인 축입니다. Latency가 짧아도 채널 수나 버스 폭이 좁으면 총 처리량(bandwidth)은 낮을 수 있고, 반대로 latency가 길어도 요청을 충분히 동시에 진행시키면(MLP) 높은 bandwidth를 달성할 수 있습니다.

오해 3

"코어를 늘리면 모든 작업이 그만큼 빨라진다"

CPU-bound 작업은 코어 확장의 이득을 잘 받지만, memory-bound 작업은 공유된 메모리 대역폭이라는 상한에 먼저 부딪힙니다. 어떤 작업이 memory-bound인지 compute-bound인지 먼저 구분해야 확장 전략을 제대로 세울 수 있습니다.

오해 4

"캐시를 늘리면 무조건 빨라진다"

캐시 용량을 키우면 더 많은 데이터를 담을 수 있지만, 일반적으로 캐시가 커질수록 access latency 자체는 조금씩 늘어나는 경향이 있습니다(L3가 L1보다 느린 이유 중 하나). 캐시 크기, latency, 히트율은 서로 얽힌 설계 trade-off이지, "크면 무조건 좋다"는 단순한 관계가 아닙니다.

오해 5

"프리페칭은 소프트웨어가 신경 쓸 필요 없는, 하드웨어가 알아서 다 해주는 기능이다"

하드웨어 프리페처는 규칙적인 접근 패턴에서는 잘 동작하지만, 접근 패턴이 불규칙하거나 예측하기 어려우면 효과가 크게 떨어집니다. 이럴 때는 자료구조를 접근 패턴에 맞게 재배치하거나(예: struct-of-arrays), 명시적 프리페치 힌트를 쓰는 것이 실제로 의미 있는 최적화가 될 수 있습니다.
