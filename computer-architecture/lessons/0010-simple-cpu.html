1. CPU의 본질 — 명령어를 반복하는 상태 기계

CPU는 "메모리에서 명령어를 읽어, 그 명령어가 지시하는 연산을 수행하고, 결과를 저장한다"는 동작을 클록마다 반복하는 상태 기계(state machine)입니다. 이 반복 사이클을 Fetch → Decode → Execute → Memory → Write-Back이라 부르며, 이 다섯 단계를 하나의 클록 사이클 안에 모두 수행하는 구조가 단일 사이클 데이터패스입니다.

명령어 사이클 (Instruction Cycle): ┌──────────────────────────────────────────────────────────┐ │ PC(Program Counter)가 가리키는 주소에서 명령어를 읽음 │ ← Fetch (IF) ├──────────────────────────────────────────────────────────┤ │ opcode 해석, 레지스터 번호 추출, 즉시값 부호 확장 │ ← Decode (ID) ├──────────────────────────────────────────────────────────┤ │ ALU가 연산 수행 (덧셈, 비교, 주소 계산 등) │ ← Execute (EX) ├──────────────────────────────────────────────────────────┤ │ 데이터 메모리에서 Load 또는 Store │ ← Memory (MEM) ├──────────────────────────────────────────────────────────┤ │ ALU 결과 또는 Load 데이터를 레지스터에 기록 │ ← Write-Back (WB) └──────────────────────────────────────────────────────────┘ 단일 사이클: 위 5단계가 하나의 클록 안에 조합 논리로 완료됨 파이프라인: 5단계를 겹쳐서 각 단계를 별도 클록 사이클에 수행 (Day 11 주제)

2. 핵심 하드웨어 블록

단일 사이클 CPU를 구성하는 주요 블록 다섯 가지입니다.

2-1. 프로그램 카운터 (PC)

PC (Program Counter) · 현재 실행 중인 명령어의 주소를 저장하는 32비트(또는 64비트) 레지스터 · 매 클록 사이클마다 PC + 4 로 자동 증가 (RISC-V: 명령어 크기 4 바이트) · Branch / Jump 명령어 실행 시 PC ← 분기 목표 주소로 덮어씌움 [PC] ──→ 명령어 메모리(IM) ──→ 명령어(32비트) 출력 ↑ PC+4 또는 PC+offset (Branch 목표)

2-2. 명령어 메모리 (Instruction Memory)

명령어 메모리 (IM, Instruction Memory / I-Cache) · PC 주소를 입력받아 32비트 명령어를 출력하는 읽기 전용 메모리 · 단일 사이클에서는 조합 논리로 모델링 (클록 없이 즉시 읽힘) · 실제 구현에서는 L1 I-Cache 또는 ROM 입력: 주소(32비트) 출력: 명령어(32비트)

2-3. 레지스터 파일 (Register File)

레지스터 파일 (Register File) · RISC-V: 32개 × 32비트 레지스터 (x0–x31) · x0는 항상 0 (쓰기 시 무시) · 단일 포트 쓰기, 이중 포트 읽기 (2개 레지스터를 동시에 읽을 수 있음) 입력: Read Register 1 (rs1) ──→ [레지스터 파일] ──→ Read Data 1 (RD1) Read Register 2 (rs2) ──→ ──→ Read Data 2 (RD2) Write Register (rd) ──→ Write Data ──→ (WB 단계에서 결과 기록) RegWrite (제어 신호) ──→ 1이면 쓰기 활성화 읽기는 조합 논리 (비동기), 쓰기는 클록 엣지에 동기화

2-4. ALU (Arithmetic Logic Unit)

ALU (산술 논리 장치) · 덧셈·뺄셈·AND·OR·XOR·비교(SLT) 등 연산 수행 · 두 32비트 입력 → 32비트 결과 출력 · ALU 제어 신호(ALUControl, 4비트)로 연산 종류 선택 입력: A (rs1 값 또는 PC) B (rs2 값 또는 즉시값 imm) ← MUX로 선택 ALUControl (연산 종류) 출력: ALU Result (32비트) Zero (결과 == 0이면 1, Branch 판정에 사용) ALUControl 코드 예: 0000 → AND 0001 → OR 0010 → ADD 0110 → SUB 0111 → SLT (set-less-than)

2-5. 데이터 메모리 (Data Memory)

데이터 메모리 (DM, Data Memory / D-Cache) · lw (Load Word) 와 sw (Store Word) 명령어가 접근하는 메모리 · ALU가 계산한 주소(effective address)를 사용하여 읽기/쓰기 입력: Address (ALU 결과, 유효 주소) Write Data (rs2 값, Store 시 사용) MemRead (1 → 읽기 활성화) MemWrite (1 → 쓰기 활성화) 출력: Read Data (lw 결과)

3. 단일 사이클 데이터패스 전체 구조

위 블록들을 연결하면 아래와 같은 데이터패스가 됩니다. MUX(멀티플렉서)는 제어 신호에 따라 입력 중 하나를 선택해 출력합니다.

┌─────────┐ 명령어(32비트) ┌──────────────────────────────────────────────┐ │ PC │ ────────────────→ │ 명령어 디코더 │ └────┬────┘ │ [31:25]=funct7 [24:20]=rs2 [19:15]=rs1 │ │ │ [14:12]=funct3 [11:7]=rd [6:0]=opcode │ │ +4 (또는 branch 주소) └──────┬────────┬──────────────┬────────────────┘ └──────────────────────────┐ │ │ │ │ ↓ rs1,rs2│rd │ 즉시값(imm) 부호 확장 ┌──────┴──────────────────┐ │ │ 레지스터 파일 │ ↓ │ RD1 (rs1 값) ─────────→├──→ [ALU입력A] │ RD2 (rs2 값) ──→[MUX]──┤ [MUX]──→ [ALU입력B] │ ↑imm │ (rs2 or imm) └──────────────────────────┘ ↓ ┌──────────┐ │ ALU │ │ (연산) │ └────┬─────┘ │ ALU 결과 ┌─────────────────────────┤ ↓ ↓ ┌───────────┐ [데이터메모리] │Write-Back │←─ [MUX] ←── Read Data (lw) │레지스터에 │ ↑─── ALU 결과 (R-type/ALU-I) │ 결과 저장 │ └───────────┘
MUX(Multiplexer)의 역할
MUX는 "여러 입력 중 하나를 선택해 출력하는 스위치"입니다. 제어 신호 값(0 또는 1)에 따라 어떤 경로를 선택할지 결정합니다.

데이터패스에서 핵심 MUX:
  • ALUSrc MUX: ALU 입력 B를 rs2 값으로 할지, 즉시값(imm)으로 할지 선택
  • MemtoReg MUX: 레지스터에 쓸 값을 ALU 결과로 할지, 메모리 Load 데이터로 할지 선택
  • PC 소스 MUX: 다음 PC를 PC+4로 할지, Branch 목표 주소(PC+offset)로 할지 선택

4. 명령어 타입별 데이터 흐름

4-1. R타입 — 레지스터 간 산술·논리 연산

예시: add rd, rs1, rs2 (rd ← rs1 + rs2) IF : PC → 명령어 메모리 → 32비트 명령어 fetch ID : 명령어[19:15]=rs1, [24:20]=rs2, [11:7]=rd 레지스터 파일에서 RD1, RD2 읽음 EX : ALUSrc=0 → ALU 입력 B = RD2 (rs2 값) funct7·funct3 → ALUControl=ADD ALU: RD1 + RD2 → ALU Result MEM: MemRead=0, MemWrite=0 → 메모리 접근 없음 WB : MemtoReg=0 → Write Data = ALU Result RegWrite=1 → rd 레지스터에 ALU Result 기록 활성 제어 신호: RegWrite=1, ALUSrc=0, ALUOp=10, MemtoReg=0 비활성: MemRead=0, MemWrite=0, Branch=0

4-2. I타입 Load — lw (Load Word)

예시: lw rd, imm(rs1) (rd ← Memory[rs1 + imm]) IF : 명령어 fetch ID : rs1, rd 레지스터 읽기. 즉시값(imm[11:0]) 부호 확장 → 32비트 EX : ALUSrc=1 → ALU 입력 B = 부호 확장된 imm (rs2 값이 아님) ALUControl=ADD ALU: rs1 + imm → 유효 주소(effective address) MEM: MemRead=1 → 데이터 메모리[유효 주소] → Read Data MemWrite=0 WB : MemtoReg=1 → Write Data = Read Data (메모리에서 읽은 값) RegWrite=1 → rd 레지스터에 메모리 값 기록 활성 제어 신호: RegWrite=1, ALUSrc=1, ALUOp=00(ADD), MemRead=1, MemtoReg=1 비활성: MemWrite=0, Branch=0

4-3. S타입 — sw (Store Word)

예시: sw rs2, imm(rs1) (Memory[rs1 + imm] ← rs2) IF : 명령어 fetch ID : rs1, rs2 레지스터 읽기 즉시값: imm[11:5]=[31:25], imm[4:0]=[11:7] → 부호 확장 EX : ALUSrc=1 → ALU 입력 B = 부호 확장된 imm ALU: rs1 + imm → 유효 주소 MEM: MemWrite=1 → 데이터 메모리[유효 주소] ← RD2 (rs2 값) MemRead=0 WB : RegWrite=0 → 레지스터 기록 없음 (저장 명령어는 레지스터를 바꾸지 않음) 활성 제어 신호: ALUSrc=1, ALUOp=00(ADD), MemWrite=1 비활성: RegWrite=0, MemRead=0, Branch=0, MemtoReg=X(don't care)

4-4. B타입 — beq (Branch if Equal)

예시: beq rs1, rs2, offset (if rs1 == rs2: PC ← PC + (offset * 2)) IF : 명령어 fetch ID : rs1, rs2 레지스터 읽기 즉시값: B타입 인코딩 → 13비트 부호 확장 (비트 0은 항상 0, 워드 정렬) EX : ALUSrc=0 → ALU 입력 B = RD2 (rs2 값) ALUControl=SUB (rs1 - rs2 계산) Zero = (rs1 - rs2 == 0) ? 1 : 0 분기 목표 주소: PC + (부호 확장된 offset) ← 별도 가산기로 계산 MEM: 메모리 접근 없음 WB : RegWrite=0 → 레지스터 기록 없음 PC 갱신: PCSrc = Branch AND Zero PCSrc=1 → PC ← PC + offset (분기 성립) PCSrc=0 → PC ← PC + 4 (분기 미성립) 활성 제어 신호: Branch=1, ALUSrc=0, ALUOp=01(SUB) 비활성: RegWrite=0, MemRead=0, MemWrite=0

4-5. I타입 ALU — addi (Add Immediate)

예시: addi rd, rs1, imm (rd ← rs1 + imm) IF : 명령어 fetch ID : rs1 읽기, 즉시값 부호 확장 EX : ALUSrc=1 → ALU 입력 B = 부호 확장된 imm funct3=000, opcode=0010011 → ALUControl=ADD ALU: rs1 + imm → 결과 MEM: 메모리 접근 없음 WB : MemtoReg=0 → Write Data = ALU Result RegWrite=1 → rd 레지스터에 결과 기록 lw와 제어 신호 비교: addi: MemRead=0, MemtoReg=0 (ALU 결과를 레지스터에 직접 기록) lw : MemRead=1, MemtoReg=1 (메모리 값을 레지스터에 기록)

5. 제어 유닛 (Control Unit)

제어 유닛은 명령어의 opcode(비트 6:0)를 입력받아 데이터패스 MUX와 메모리를 제어하는 신호들을 출력하는 조합 논리 회로입니다. funct3와 funct7는 ALU 제어 유닛(ALU Control)이 별도로 처리합니다.

5-1. 주요 제어 신호 정의

신호비트 폭역할

5-3. ALU 제어 유닛 (ALU Control)

ALU 제어 유닛은 제어 유닛의 ALUOp(2비트)과 명령어의 funct3·funct7을 받아 실제 ALU 연산 코드(ALUControl, 4비트)를 결정합니다. ALUOp=00 (Load/Store): ALUControl=ADD 고정 (주소 계산) ALUOp=01 (Branch) : ALUControl=SUB 고정 (rs1-rs2로 같은지 비교) ALUOp=10 (R/I-ALU) : funct3·funct7 조합으로 결정 ALUOp=10 결정 예 (RISC-V): funct7=0000000, funct3=000 → ADD (add, addi) funct7=0100000, funct3=000 → SUB (sub) funct7=0000000, funct3=111 → AND (and, andi) funct7=0000000, funct3=110 → OR (or, ori) funct7=0000000, funct3=010 → SLT (slt, slti) 2단계 제어 분해 이유: · opcode 7비트만으로 이미 명령어 대분류 결정 가능 · funct3·funct7은 R타입과 일부 I타입 내에서만 의미 있음 → 제어 유닛을 두 레벨로 나눠 복잡도를 줄임

6. 단일 사이클의 성능 — 가장 느린 명령어가 클록 주기를 결정한다

단일 사이클 설계에서 클록 주기(Tclk)는 데이터패스를 통과하는 최장 지연(critical path)에 의해 결정됩니다.

각 단계별 지연 시간 가정 (예시 값): 명령어 메모리 읽기: 200 ps 레지스터 파일 읽기: 30 ps ALU 연산: 200 ps 데이터 메모리 읽기: 250 ps 레지스터 파일 쓰기: 30 ps 명령어 타입별 총 지연: R타입: 200 + 30 + 200 + 0 + 30 = 460 ps (MEM 단계 없음) addi: 200 + 30 + 200 + 0 + 30 = 460 ps lw: 200 + 30 + 200 + 250 + 30 = 710 ps ← 가장 느림 (critical path) sw: 200 + 30 + 200 + 250 + 0 = 680 ps beq: 200 + 30 + 200 + 0 + 0 = 430 ps 결론: 클록 주기 T_clk = 710 ps (lw 기준) → R타입도 460 ps면 충분한데 710 ps를 기다려야 함 (250 ps 낭비) CPI(Clocks Per Instruction) = 1 이지만 T_clk이 길어서 실제 성능이 나쁠 수 있음 파이프라인: T_clk ≈ 200 ps (가장 느린 단일 단계 기준) CPI ≈ 1 (이상적) → 처리량 약 3.5배 향상

7. 하드와이어드 제어 vs 마이크로코드 제어

하드와이어드 제어 (Hardwired Control): · 제어 유닛을 고정 조합 논리 회로로 구현 · opcode → 제어 신호 매핑을 AND/OR 게이트로 직접 연결 · 장점: 빠름 (조합 논리 지연만 있음), 전력 효율 · 단점: 새로운 명령어 추가가 어려움 (회로 재설계 필요) · 사용: RISC (RISC-V, ARM), 단순 CPU 마이크로코드 제어 (Microcode / Microprogrammed Control): · 제어 신호 패턴을 ROM(마이크로 프로그램 메모리)에 저장 · opcode → ROM 주소 → 제어 신호 시퀀스 읽기 · 장점: 명령어 추가·수정이 ROM 업데이트로 가능 (펌웨어) · 단점: ROM 접근 지연, 복잡도 · 사용: CISC (x86의 복잡한 명령어를 마이크로-op으로 분해) 현대 x86: · 간단한 명령어 → 하드와이어드 고속 경로 (fast path) · 복잡한 명령어 → 마이크로코드 시퀀서 · 두 방식 혼합으로 CISC 호환성 + RISC 성능 달성

8. 전체 동작 추적 — add t0, t1, t2

명령어 add t0, t1, t2가 단일 사이클 데이터패스를 통과하는 과정을 클록 하나 동안 단계별로 추적합니다.

명령어: add t0, t1, t2 (x5 ← x6 + x7) 인코딩: 0000000_00111_00110_000_00101_0110011 funct7 rs2 rs1 f3 rd opcode ① Fetch (IF): PC = 0x0000_1000 명령어 메모리[0x0000_1000] → 32비트 명령어 출력 PC+4 계산기: 0x0000_1000 + 4 = 0x0000_1004 ② Decode (ID): opcode[6:0] = 0110011 → 제어 유닛: R타입 rs1[19:15] = 00110 → x6 (t1) rs2[24:20] = 00111 → x7 (t2) rd [11:7] = 00101 → x5 (t0) (아직 쓰지 않음) 레지스터 파일: RD1 = x6 값, RD2 = x7 값 제어 신호 출력: RegWrite=1, ALUSrc=0, ALUOp=10, MemRead=0, MemWrite=0, MemtoReg=0, Branch=0 ③ Execute (EX): ALUSrc=0 → ALU B = RD2 (x7 값) ALUOp=10, funct7=0000000, funct3=000 → ALUControl=ADD ALU: RD1 + RD2 = x6 + x7 ALU Result = 계산 결과 Zero = (결과 == 0) ? 1 : 0 ④ Memory (MEM): MemRead=0, MemWrite=0 → 데이터 메모리 접근 없음 (이 단계는 add에서 아무 일도 하지 않음) ⑤ Write-Back (WB): MemtoReg=0 → Write Data = ALU Result RegWrite=1 → 레지스터 파일: x5 ← ALU Result (클록 상승 엣지에 동기화하여 기록) ⑥ PC 갱신: Branch=0 → PCSrc=0 → PC ← PC+4 = 0x0000_1004

오늘의 핵심 요약

  • 단일 사이클 CPU: IF → ID → EX → MEM → WB 다섯 단계를 클록 하나에 수행하는 설계
  • PC: 다음 명령어 주소를 저장; 매 사이클 +4, 분기 시 목표 주소로 교체
  • 레지스터 파일: 2포트 읽기(비동기) + 1포트 쓰기(클록 동기); x0는 항상 0
  • ALU: ALUControl 신호로 연산 선택; Zero 신호는 Branch 판정에 사용
  • MUX: ALUSrc(rs2 vs imm), MemtoReg(ALU vs Mem), PCSrc(PC+4 vs 분기) 선택
  • 제어 유닛: opcode → 제어 신호 7가지 출력; ALU 제어 유닛이 funct3/funct7 처리
  • R타입: ALUSrc=0, RegWrite=1, MemtoReg=0 — 레지스터 간 연산
  • lw: ALUSrc=1, MemRead=1, MemtoReg=1, RegWrite=1 — 주소 계산 후 메모리 Load
  • sw: ALUSrc=1, MemWrite=1, RegWrite=0 — 주소 계산 후 메모리 Store
  • beq: ALUSrc=0, ALUOp=SUB, Branch=1, RegWrite=0 — Zero 신호와 AND로 분기 결정
  • 단일 사이클 한계: Critical Path(lw)가 클록 주기를 결정 → 파이프라인이 해결책
