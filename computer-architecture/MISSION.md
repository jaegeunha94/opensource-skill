# 컴퓨터 구조 10년차 이상 개발자 Interview 학습 미션

## 목표

10년차 이상 개발자 기술 면접에서 컴퓨터 구조 지식을 **성능 병목, 메모리 계층, CPU 실행 모델, 캐시 효율, 동시성 비용**과 연결해 설명할 수 있도록 훈련한다.

경력자에게 컴퓨터 구조를 묻는 이유는 회로를 그리게 하려는 것이 아니라, "왜 이 코드는 느린가", "왜 캐시 친화적인 구조가 중요한가", "CPU-bound와 memory-bound를 어떻게 구분하는가" 같은 실무 판단력을 확인하기 위해서다.

## 평가 대상 역량

| 역량 | 면접에서 확인하려는 것 |
|------|------------------------|
| 실행 모델 | CPU, 레지스터, 메모리, 명령어 실행 흐름을 코드 성능과 연결하는가 |
| 메모리 계층 | register, cache, RAM, disk의 지연 시간 차이를 설계 판단에 반영하는가 |
| 캐시 | locality, cache line, false sharing, cache miss를 설명하고 개선 방향을 제시하는가 |
| 파이프라인 | throughput, latency, branch prediction, hazard를 직관적으로 설명하는가 |
| 병렬성 | instruction-level parallelism, SIMD, multicore, memory bandwidth 한계를 이해하는가 |
| 성능 분석 | CPU profile, cache miss, branch miss, memory allocation 지표를 해석하는가 |
| 실무 연결 | 자료구조, DB 인덱스, serialization, batch 처리, high-performance API와 연결하는가 |

## Interview 레슨 작성 규칙

각 Day 레슨은 반드시 **면접 질문 중심**으로 작성한다.

1. **면접 질문**
   - 코드 성능과 하드웨어 동작을 연결하는 질문으로 시작한다.
   - 예: "같은 알고리즘인데 배열 기반 구현이 연결 리스트보다 빠를 수 있는 이유를 설명해보세요."
2. **전제 개념**
   - 어려운 개념은 먼저 쉬운 한국어로 설명한다.
   - CPU, 레지스터, 캐시 라인, locality, pipeline 같은 용어를 필요한 배경부터 풀어쓴다.
3. **왜 면접에서 묻는가**
   - 성능 병목을 원리 기반으로 추론하는지, 단순 Big-O 너머를 보는지 확인하는 이유를 적는다.
4. **실무 연결**
   - 대량 데이터 처리, API latency, DB 인덱스, GC, serialization, lock contention과 연결한다.
5. **답변 사고 순서**
   - 코드 접근 패턴 확인 -> 메모리 계층 영향 -> CPU 실행 특성 -> 측정 방법 -> 개선안 비교 순서로 답변하도록 안내한다.
6. **핵심 답변 예시**
   - 면접에서 바로 말할 수 있는 답변을 짧은 버전과 깊이 있는 버전으로 제공한다.
7. **Trade-off**
   - 성능, 메모리 사용량, 구현 복잡도, 이식성, 유지보수성을 비교한다.
8. **흔한 오해**
   - "Big-O가 같으면 성능도 비슷하다", "캐시는 개발자가 신경 쓰지 않아도 된다", "멀티코어면 자동으로 빨라진다" 같은 오해를 바로잡는다.
9. **Follow-up 질문**
   - 면접관이 더 깊게 물을 질문과 답변 방향을 포함한다.
10. **10년차 이상 답변 기준**
   - 교과서 정의가 아니라 측정, 병목 추론, 실제 개선 경험이 드러나는지 평가한다.

## 완료 기준

- 컴퓨터 구조 개념을 실제 코드 성능 문제와 연결해 설명할 수 있다.
- cache locality, branch prediction, memory bandwidth 같은 주제를 쉬운 한국어로 풀어낼 수 있다.
- "측정 없이 단정하지 않는다"는 성능 분석 태도를 답변에 반영한다.
- 약한 답변과 강한 답변의 차이를 스스로 설명할 수 있다.
