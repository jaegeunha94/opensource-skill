# 메시지 큐 / 비동기 처리 — 시니어 인터뷰 대비 학습 미션

## 학습 목표

10년차 이상 백엔드 개발자 인터뷰에서 출제되는 **비동기 아키텍처 설계, 장애 격리, 대량 처리 판단** 질문에 체계적으로 답변할 수 있는 능력을 키운다.

단순 큐 제품 비교나 API 암기가 아니라, **실무 제약 조건 안에서 왜 이 선택을 했는가**를 논리적으로 설명하는 것이 목표다.

## 전제 지식

- 기본 HTTP API 설계 경험이 있는 5년차 이상 백엔드 개발자 기준으로 진행
- Python, Java, Go 중 하나 이상의 백엔드 언어 경험 전제
- RDB 트랜잭션 기초 개념(ACID) 이해 전제
- Docker/컨테이너 기초 이해 전제

## 커리큘럼 범위

| 영역 | 핵심 주제 |
|------|-----------|
| 비동기 설계 기초 | 동기 vs 비동기 판단, 메시지 큐 도입 기준, 설계 프레임워크 |
| Kafka | 파티셔닝, 오프셋, 컨슈머 그룹, 순서 보장, 재처리 |
| RabbitMQ | Exchange 유형, 라우팅, ACK/NACK, DLQ 연동 |
| AWS SQS/SNS | Standard vs FIFO, 가시성 타임아웃, fan-out 패턴 |
| Celery | 작업 큐 설계, 재시도, priority, chord/chain |
| 신뢰성 설계 | Idempotency, 재시도 전략, Exponential Backoff |
| 장애 격리 | Dead-Letter Queue, 독성 메시지 처리, Circuit Breaker |
| 순서/정합성 | Ordering 보장 조건, 파티션 키 설계 |
| 대량 처리 | Backpressure, Consumer Rate Control, 처리량 산정 |
| 종합 설계 | 실무 시나리오 기반 전체 비동기 아키텍처 설계 |

## 인터뷰에서 평가하는 것

면접관은 다음을 보고 있다:

1. **왜 비동기인가** — 동기로 충분한 상황과 비동기가 필요한 상황을 구분하는가
2. **장애 시나리오 예측** — Consumer가 죽으면? 메시지가 중복 처리되면? 어떻게 설계로 방어하는가
3. **Trade-off 언어** — "이것이 최선"이 아니라 "이 제약 조건에서 이것이 낫다"고 말하는가
4. **운영 경험** — 운영 중 겪은 장애 패턴을 알고 있는가
5. **규모 감각** — 메시지 처리량, 지연 허용치, 재시도 비용을 숫자로 말할 수 있는가
