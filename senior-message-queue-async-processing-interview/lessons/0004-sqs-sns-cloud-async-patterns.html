오늘의 면접 질문
"주문 서비스가 결제 완료 후 재고, 배송, 알림 서비스에 비동기로 이벤트를 전달해야 합니다. 직접 브로커를 운영하지 않고 AWS 관리형 서비스만으로 이 구조를 설계한다면 SNS와 SQS를 어떻게 조합하시겠습니까? 그리고 만약 그중 정산(settlement) 처리 큐 하나는 반드시 주문이 들어온 순서대로 처리되어야 한다면, 설계에서 무엇이 달라져야 합니까?"
전제 개념

SQS는 "우편함", SNS는 "방송국" — 풀(Pull)과 푸시(Push)의 차이

Day 3에서 RabbitMQ는 Exchange(라우팅)와 Queue(저장)가 분리되어 있다고 배웠다. AWS도 같은 역할 분리를 두 개의 별도 관리형 서비스로 나눠놨다. SQS(Simple Queue Service)는 메시지를 저장하는 큐이고, Consumer가 직접 ReceiveMessage API를 호출해서 메시지를 "당겨가는(pull)" 방식이다. SNS(Simple Notification Service)는 저장소가 없는 Pub/Sub 토픽이며, 발행되는 즉시 구독자에게 "밀어주는(push)" 방식이다.

중요한 차이는 SNS 자체는 메시지를 오래 보관하지 않는다는 점이다. SNS는 발행 시점에 구독자(SQS 큐, Lambda, HTTP 엔드포인트, 이메일 등)에게 즉시 전달을 시도할 뿐이며, 구독자가 SQS 큐가 아니라면 안정적인 재시도·영속 저장이 보장되지 않는다. 그래서 실무에서는 SNS를 "라우팅/방송 계층", SQS를 "저장/재시도 계층"으로 조합해서 쓴다 — 이것이 SNS Fan-out 패턴이다. 

주문 서비스 ──publish──▶ SNS Topic: "order.completed"
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                      ▼
  SQS: inventory-queue   SQS: shipping-queue    SQS: notification-queue
        │                     │                      │
        ▼                     ▼                      ▼
  재고 Consumer(Lambda)  배송 Consumer(Lambda)   알림 Consumer(Lambda)
핵심 직관: SNS Topic은 RabbitMQ의 Fanout Exchange와 역할이 거의 같다 — "메시지를 저장하지 않고 구독된 여러 대상에 복제해서 전달"한다. 다만 RabbitMQ Exchange 뒤에는 항상 Queue가 붙어 메시지를 저장하는 반면, SNS 뒤의 구독자가 SQS가 아니라 Lambda 직접 호출이라면 Lambda 실행이 실패했을 때 재시도·영속성 보장 수준이 달라진다. "SNS만으로 신뢰성 있는 비동기 처리가 끝났다"고 생각하는 것이 이 개념에서 가장 흔한 함정이다.
왜 면접에서 묻는가

이 질문이 평가하는 것

"SQS와 SNS의 차이가 뭔가요?"는 문서를 읽었는지 확인하는 질문이지만, "이 구조를 어떻게 조합하겠는가"는 관리형 서비스를 조합해서 신뢰성 있는 아키텍처를 설계하는 능력을 본다.

  • 브로커를 직접 운영하지 않는 환경에서의 설계 판단: Kafka/RabbitMQ처럼 브로커 내부 동작을 세밀하게 튜닝할 수 없는 관리형 서비스에서, 주어진 기능(Fan-out, Visibility Timeout, Redrive Policy)만으로 같은 신뢰성 목표를 달성하는지 본다.
  • Standard와 FIFO의 트레이드오프를 요구사항에 매핑하는가: "정산은 순서가 중요하다"는 제약이 주어졌을 때, 순서 보장에는 처리량 제한과 블로킹 리스크라는 대가가 따른다는 것을 알고, 이 트레이드오프를 명시적으로 말하는지가 핵심이다.
  • 서버리스(Lambda)와 큐의 상호작용을 이해하는가: Consumer가 고정된 프로세스가 아니라 자동 스케일링되는 Lambda일 때 생기는 새로운 문제 (동시성 폭주, 부분 배치 실패, 재시도 중복)를 다뤄본 적이 있는지 본다.
실무 연결

왜 "SNS + SQS" 조합이 필요하고, Standard와 FIFO는 언제 갈라지는가

만약 SNS만 쓰고 Lambda를 직접 구독시키면, Lambda 실행이 실패할 때 SNS는 제한된 횟수만 재시도하고 (기본 재시도 정책은 짧고 구성이 제한적이다) 결국 메시지를 잃어버릴 수 있다. 반면 SNS와 SQS를 함께 쓰면, SNS가 각 SQS 큐로 메시지를 복제해 넣고, 각 큐는 Consumer가 복구될 때까지 메시지를 안전하게 보관한다. 즉 SNS는 "누구에게 보낼지"를 결정하고, SQS는 "안전하게 도착할 때까지 들고 있는" 역할을 분담한다. 재고, 배송, 알림처럼 서로 독립적으로 처리되고 순서가 중요하지 않은 이벤트는 이 구조로 충분하다.

그런데 정산 큐처럼 "주문이 들어온 순서대로 처리되어야 한다"는 요구가 생기면 이야기가 달라진다. 기본 SQS(Standard)는 높은 처리량을 위해 순서를 엄격히 보장하지 않고, 드물게 메시지가 중복 전달될 수 있다. 순서와 중복 방지가 필요하면 FIFO 큐로 전환해야 한다.

구분 Standard Queue FIFO Queue
순서 보장 보장 안 함 (best-effort) MessageGroupId 내에서 엄격히 보장 (Kafka 파티션 키와 유사한 개념)
중복 처리 at-least-once, 중복 가능성 있음 (Consumer가 idempotency로 방어해야 함) MessageDeduplicationId로 5분 창(window) 내 중복 발행 자동 제거
처리량 사실상 무제한에 가까움 표준 최대 초당 300건, High Throughput 모드에서 배치 시 초당 최대 3,000건 — 그룹 수로 병렬화 필요
적합한 상황 재고/알림처럼 순서 무관, 대량 처리 우선 정산, 계좌 잔액 반영처럼 순서·중복이 비즈니스 정합성에 직결되는 처리
흔한 오해: "순서가 중요하면 무조건 FIFO 큐를 쓰면 된다." FIFO는 MessageGroupId가 같은 메시지끼리만 순서를 보장하고, 같은 그룹 내에서는 한 번에 하나의 메시지만 처리되도록 직렬화된다. 즉 그룹을 세분화하지 않고 전체를 하나의 그룹으로 두면 사실상 큐 전체가 순차 처리되어 처리량이 크게 떨어진다. Kafka의 파티션 키 설계(Day 2)와 마찬가지로, "무엇을 같은 순서로 묶어야 하는가"를 먼저 정의하고 그 단위를 MessageGroupId로 매핑하는 설계가 먼저다.
핵심 개념

Visibility Timeout — SQS의 ACK 없는 신뢰성 전달 메커니즘

RabbitMQ는 Consumer가 명시적으로 ACK를 보내야 메시지가 삭제된다. SQS는 조금 다르게 동작한다. Consumer가 ReceiveMessage로 메시지를 가져가면, 그 메시지는 큐에서 삭제되지 않고 일정 시간(Visibility Timeout) 동안 다른 Consumer에게 보이지 않게(invisible) 숨겨진다. Consumer가 처리를 마치고 DeleteMessage를 호출해야 비로소 메시지가 완전히 삭제된다.

  1. 정상 흐름
    Consumer가 메시지를 받음 → Visibility Timeout 동안 숨김 → 처리 완료 → DeleteMessage 호출 → 큐에서 완전히 제거. 이 흐름이 RabbitMQ의 "수신 → 처리 → Manual ACK"와 대응된다.
  2. Consumer가 죽거나 처리 시간이 초과되면
    DeleteMessage가 호출되지 못한 채 Visibility Timeout이 만료되면, 메시지는 다시 큐에 "보이는" 상태로 돌아가 다른 Consumer가 가져갈 수 있다 — RabbitMQ의 NACK+requeue와 유사한 자동 재시도다. 다만 SQS는 이 과정이 명시적 NACK 없이 타임아웃으로만 트리거된다는 점이 다르다.
  3. 타임아웃 값 설계
    Visibility Timeout이 실제 처리 시간보다 짧으면, 아직 처리 중인 메시지가 "실패한 것"으로 오인되어 다른 Consumer에게 재전달되고 동시에 두 곳에서 처리되는 중복이 생긴다. 반대로 너무 길면, 진짜로 실패한 메시지가 재시도되기까지 불필요하게 오래 기다려야 한다. 일반적으로 예상 처리 시간의 3~6배 정도로 설정하고, 처리가 길어질 것 같으면 ChangeMessageVisibility로 동적으로 연장하는 방식을 함께 고려한다.
면접 팁: "Visibility Timeout을 씁니다"에서 멈추지 말고, "이 메커니즘은 명시적 ACK가 아니라 타임아웃 기반이기 때문에 처리 시간이 타임아웃을 넘기면 중복 수신이 발생할 수 있고, 그래서 Consumer 로직은 항상 멱등하게 작성해야 한다"까지 연결하면 at-least-once 전달의 본질을 이해하고 있다는 신호를 준다 (Idempotency는 Day 5에서 더 깊게 다룬다).

FIFO Fan-out의 함정 — 독성 메시지가 "그룹 전체"를 막는다

Day 3에서 RabbitMQ의 독성 메시지는 DLX로 격리하면 다른 메시지 처리에 영향을 주지 않는다고 배웠다. SQS Standard 큐도 마찬가지로, 메시지 하나가 계속 실패해도 다른 메시지는 병렬로 계속 처리된다. 그런데 FIFO 큐는 다르다. 같은 MessageGroupId를 가진 메시지는 순서를 지키기 위해 앞 메시지의 삭제(또는 Visibility Timeout 만료)를 기다렸다가 다음 메시지를 전달하는 구조이므로, 앞 메시지가 계속 실패하면 그 그룹의 나머지 메시지 전체가 멈춘다. 

FIFO Queue, MessageGroupId = "merchant-42"

[정산 msg#1: 처리 실패 반복] ──▶ 재시도 대기 중
[정산 msg#2] ──▶ 대기 (msg#1이 삭제되기 전까지 전달 안 됨)
[정산 msg#3] ──▶ 대기 ...

→ merchant-42 그룹 전체가 정체(head-of-line blocking)

이를 방지하려면 (1) maxReceiveCount를 넘긴 메시지는 Redrive Policy로 자동으로 DLQ(별도 FIFO 큐)로 이동시켜 그룹의 정체를 풀고, (2) MessageGroupId를 너무 크게 묶지 않아 한 그룹의 장애가 미치는 반경을 최소화해야 한다. 가맹점 단위, 사용자 단위처럼 실패 영향 범위를 좁게 설계하는 것이 핵심이다.

흔한 오해: "FIFO 큐도 DLQ만 걸어두면 RabbitMQ DLX와 동일하게 안전하다." DLQ로의 이동은 maxReceiveCount에 도달한 "이후"에 일어난다. 그 전까지는 재시도가 반복되는 동안 같은 그룹의 후속 메시지가 계속 막혀 있다는 점에서 RabbitMQ보다 격리 지연이 크다. 순서 보장이 꼭 필요한 범위를 최소 단위로 좁게 설계하는 것이 이 문제의 근본적인 완화책이다.
핵심 개념

Lambda Event Source Mapping — 서버리스 Consumer의 특수한 문제

전통적인 Consumer(고정 프로세스)와 달리 Lambda는 SQS 큐를 폴링하는 관리형 트리거(Event Source Mapping)를 통해 자동으로 스케일 아웃된다. 편리하지만 새로운 판단이 필요하다.

  • 배치(batch) 단위 실행: Lambda는 SQS 메시지를 여러 개씩 묶어(batch size) 한 번의 실행으로 처리한다. 배치 안의 메시지 중 일부만 실패하면, ReportBatchItemFailures를 사용해 실패한 메시지만 정확히 재시도 대상으로 지정해야 한다. 이걸 설정하지 않으면 배치 전체가 실패로 처리되어 이미 성공한 메시지까지 중복 재처리된다.
  • 동시성 폭주: 큐에 메시지가 급증하면 Lambda가 동시 실행 수를 자동으로 늘려 다운스트림(DB, 외부 API)에 순간적으로 과부하를 줄 수 있다. ReservedConcurrency 또는 FIFO 큐라면 MessageGroupId 수 자체가 자연스러운 동시성 상한이 된다 — 이는 Day 8의 Backpressure 설계와 직결된다.
  • Lambda 자체의 재시도 vs SQS 재시도: 비동기 호출(SNS→Lambda 직접 구독)에는 Lambda 자체의 재시도 정책이 적용되지만, SQS 트리거는 Visibility Timeout 기반 재시도가 우선한다. 두 메커니즘을 혼동하면 재시도 횟수·타이밍을 잘못 설계하게 된다.
답변 사고 순서

면접에서 이 질문을 받았을 때 생각하는 순서

  1. 배포 패턴을 먼저 정의
    "재고/배송/알림은 서로 독립적으로 동일 이벤트를 각자 받아야 하므로 SNS Fan-out 패턴이 필요합니다."
  2. SNS+SQS 조합 근거 제시
    "SNS Topic 하나에 서비스별 SQS 큐 세 개를 구독시켜, SNS는 라우팅만 하고 SQS가 각 소비자의 장애·지연을 흡수하며 안전하게 보관하도록 합니다."
  3. Standard vs FIFO를 요구사항별로 분리
    "재고/배송/알림은 순서가 중요하지 않으므로 Standard로 처리량을 확보하고, 정산 큐만 FIFO로 분리해 MessageGroupId를 가맹점 단위로 설계해 순서를 보장하되 병렬성을 최대한 유지합니다."
  4. 신뢰성 메커니즘 명시
    "Visibility Timeout을 처리 시간에 맞게 설정하고, Consumer 로직은 멱등하게 작성해 중복 수신에도 안전하도록 합니다."
  5. 장애 격리로 연결
    "각 큐에 Redrive Policy를 걸어 반복 실패 메시지를 DLQ로 격리하고, 특히 FIFO 정산 큐는 그룹 단위를 좁게 설계해 한 가맹점의 장애가 다른 가맹점 정산 처리를 막지 않도록 합니다."
핵심 답변 예시

실제 면접 답변 예시

"재고, 배송, 알림 서비스가 결제 완료 이벤트를 각자 온전히 받아야 하므로, SNS Topic 하나를 두고 서비스별 SQS 큐 세 개를 구독시키는 Fan-out 구조로 설계합니다. SNS는 라우팅만 담당하고, 실제 메시지 보관과 재시도는 각 SQS 큐가 맡아서 한 소비자의 장애가 다른 소비자에게 영향을 주지 않게 합니다.

이 세 큐는 순서가 중요하지 않으므로 Standard 큐로 두고 높은 처리량을 확보합니다. 다만 정산 큐는 가맹점별로 반드시 순서대로 처리되어야 하므로 별도 FIFO 큐로 분리하고, MessageGroupId를 가맹점 ID 단위로 설계합니다. 이렇게 하면 같은 가맹점 내에서는 순서가 보장되면서도, 서로 다른 가맹점 간에는 병렬로 처리되어 처리량 손실을 최소화할 수 있습니다.

신뢰성을 위해 모든 큐에 예상 처리 시간의 3~6배 정도로 Visibility Timeout을 설정하고, Consumer는 언제든 중복 수신될 수 있다는 전제로 멱등하게 작성합니다. 각 큐에는 Redrive Policy로 maxReceiveCount를 넘긴 메시지를 DLQ로 보내도록 설정해서, 특히 FIFO 정산 큐에서 특정 가맹점의 메시지가 반복 실패하더라도 그 가맹점 그룹만 격리되고 다른 가맹점 정산 처리는 막히지 않도록 합니다."

면접관이 이어서 물어볼 수 있는 질문들

Q. "SNS 대신 EventBridge를 쓰는 게 더 나은 경우는 언제입니까?"
SNS는 토픽 하나에 구독자를 붙이는 단순 방송 모델이라 라우팅 조건이 "받는다/안 받는다" 수준입니다. EventBridge는 이벤트 내용(payload) 기반의 세밀한 규칙(Rule)으로 라우팅하고, 스키마 레지스트리, 여러 SaaS·계정 간 이벤트 버스 연동, 아카이브·리플레이 기능까지 제공합니다. 단순히 "이 이벤트를 구독자 전체에게 방송"하면 되는 경우엔 SNS가 더 단순하고 비용 효율적이고, "이벤트 속성에 따라 다른 대상으로 세분화 라우팅"하거나 여러 시스템/계정을 느슨하게 연동해야 한다면 EventBridge가 낫습니다. RabbitMQ에 비유하면 SNS는 Fanout Exchange, EventBridge는 Topic Exchange + 스키마 관리가 결합된 형태에 가깝습니다.
Q. "FIFO 큐의 처리량 제한이 병목이 되면 어떻게 하시겠습니까?"
먼저 MessageGroupId가 지나치게 굵게 묶여 있어 병렬성이 죽어있는 건 아닌지 확인합니다 — 가맹점 단위처럼 더 세분화할 수 있다면 그룹 수를 늘려 병렬 처리량을 높입니다. High Throughput 모드로 배치 처리 시 초당 최대 3,000건까지 늘릴 수 있는지도 검토합니다. 그래도 부족하다면, 정말 "큐 레벨의 순서 보장"이 필요한지 다시 따져봅니다. 많은 경우 순서 보장은 큐가 아니라 애플리케이션에서 시퀀스 번호나 버전 컬럼으로 처리해도 충분한데, 이 경우 Standard 큐로 옮기고 Consumer에서 "더 최신 시퀀스가 아니면 무시"하는 방식으로 순서 문제를 이관해 처리량 제약에서 벗어날 수 있습니다.
Q. "Lambda Consumer가 배치 중 일부 메시지만 실패하면 어떻게 처리됩니까?"
Event Source Mapping에서 ReportBatchItemFailures를 활성화하지 않으면, Lambda 함수가 예외를 던지는 순간 배치 전체가 실패로 간주되어 배치 안의 모든 메시지가 Visibility Timeout 만료 후 재시도됩니다 — 이미 성공적으로 처리된 메시지까지 중복 실행됩니다. 이를 막으려면 함수가 각 메시지의 성공/실패를 개별적으로 반환해서, 실패한 메시지의 ID만 재시도 대상으로 지정해야 합니다. 이 설정 여부가 배치 처리에서 멱등성 부담을 얼마나 지게 될지를 좌우하는 핵심 포인트입니다.
Q. "Visibility Timeout을 매우 짧게(예: 5초) 설정하면 어떤 문제가 생깁니까?"
실제 처리에 5초 이상 걸리는 메시지는 아직 처리 중인데도 "실패한 것"으로 간주되어 다른 Consumer에게 재전달됩니다. 그 결과 같은 메시지가 두 곳 이상에서 동시에 처리되는 중복 실행이 생기고, 만약 Consumer 로직이 멱등하지 않다면(예: 결제 승인 API를 매번 새로 호출) 실제 비즈니스 사고로 이어질 수 있습니다. 반대로 너무 길게 설정하면 진짜로 죽은 Consumer가 붙잡고 있던 메시지가 재시도되기까지 오래 걸려 지연이 커집니다. 처리 시간의 분포(P99 등)를 실측한 뒤 타임아웃을 정하고, 처리 시간이 가변적이라면 ChangeMessageVisibility로 동적 연장을 함께 쓰는 것이 안전합니다.
답변 품질 비교

좋은 답변과 약한 답변의 차이

강한 답변 — 이렇게 말하면 시니어로 보인다

"SNS Fan-out으로 서비스별 SQS 큐를 분리하고, 순서가 필요한 정산만 FIFO로 두되 MessageGroupId를 가맹점 단위로 세분화해 병렬성을 지키며, Visibility Timeout·Redrive Policy·멱등 처리까지 함께 설계합니다."

→ 배포 패턴 정의 + Standard/FIFO 선택 근거 + 신뢰성 메커니즘 + 장애 격리 반경 최소화까지 하나의 논리로 연결

약한 답변 — 이렇게 말하면 주니어로 보인다

"SNS로 보내고 SQS로 받으면 됩니다."

→ SNS만으로는 신뢰성 있는 재시도가 보장되지 않는다는 것과, 왜 SQS가 반드시 필요한지에 대한 이해가 없음.

약한 답변 — 순서 요구를 잘못 처리

"순서가 중요하다니까 큐 전체를 FIFO로 바꾸겠습니다."

→ MessageGroupId 설계 없이 FIFO를 적용하면 처리량이 크게 떨어지고, 독성 메시지가 그룹 전체를 막는 위험을 인지하지 못함.

자기 점검 Rubric

내 답변을 이 기준으로 점검해보자

SNS/SQS 역할 분리: SNS는 라우팅(방송), SQS는 저장·재시도를 담당한다는 역할 차이를 설명했는가?
Standard vs FIFO 선택 근거: 요구사항(순서 필요 여부)에 따라 큐 종류를 다르게 선택하고, 그 대가(처리량 제한)를 언급했는가?
MessageGroupId 설계: FIFO를 쓸 때 그룹 단위를 세분화해야 병렬성과 격리 반경을 함께 확보할 수 있다는 것을 설명했는가?
Visibility Timeout 신뢰성: 타임아웃 기반 재전달 메커니즘과, 그로 인해 Consumer가 멱등해야 하는 이유를 연결했는가?
장애 격리: Redrive Policy로 DLQ 격리를 설계하고, 특히 FIFO에서 독성 메시지가 그룹 전체를 막을 수 있다는 위험까지 언급했는가?
서버리스 Consumer 고려: Lambda를 Consumer로 쓸 때의 배치 부분 실패(ReportBatchItemFailures)나 동시성 폭주 이슈를 알고 있는가?
오늘 배운 것 요약

Day 4 핵심 정리

  • SNS는 저장소 없는 Pub/Sub 방송 계층, SQS는 저장·재시도를 담당하는 큐이며, 신뢰성 있는 Fan-out은 SNS + SQS 조합으로 만든다.
  • Standard 큐는 높은 처리량과 best-effort 순서를, FIFO 큐는 MessageGroupId 단위의 엄격한 순서와 중복 제거를 제공하며 대신 처리량 제약이 있다.
  • Visibility Timeout은 명시적 ACK 없이 타임아웃으로 재전달을 트리거하는 SQS 고유의 신뢰성 메커니즘이며, 이로 인해 Consumer는 항상 멱등해야 한다.
  • FIFO 큐에서는 독성 메시지가 같은 MessageGroupId의 후속 메시지 전체를 막을 수 있으므로, 그룹을 좁게 설계하고 Redrive Policy로 신속히 격리해야 한다.
  • Lambda를 Consumer로 쓸 때는 배치 부분 실패(ReportBatchItemFailures)와 동시성 폭주를 별도로 설계해야 한다.
