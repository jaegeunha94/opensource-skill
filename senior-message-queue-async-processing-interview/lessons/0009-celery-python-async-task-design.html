Celery와 Python 비동기 작업 설계

2026-07-08 · 예상 학습 시간 85분
Celery acks_late Priority Queue chain/chord Worker 격리
오늘의 면접 질문
"회원가입이 완료되면 Celery로 후속 작업 4개(환영 이메일 발송, 프로필 이미지 리사이징, 추천 시스템 초기화, 가입 통계 집계)를 비동기로 처리하는 시스템이 있습니다. 최근 두 가지 사고가 있었습니다. 첫째, 배포 중 Worker 프로세스가 재시작되면서 실행 중이던 이미지 리사이징 작업 일부가 사라지거나, 반대로 같은 이미지가 중복으로 리사이징되는 문제가 발생했습니다. 둘째, 결제 확인처럼 지연이 치명적인 다른 팀의 긴급 작업이 이 무거운 이미지 리사이징 작업들 뒤에 큐잉되어 지연되는 문제가 있었습니다. 이 시스템을 Celery로 어떻게 재설계하시겠습니까? 회원가입 후속 작업들 사이의 순서와 실패 처리도 함께 설명해주세요."
전제 개념

Celery 구조와 "작업이 사라지거나 중복되는" 이유

Celery는 Client(작업을 호출하는 애플리케이션 코드) → Broker(Redis 또는 RabbitMQ, Day 3에서 다룬 메시지 브로커) → Worker(작업을 실제로 실행하는 프로세스) 구조로 동작한다. 애플리케이션이 task.delay()를 호출하면 실제 작업이 그 자리에서 실행되는 게 아니라, 브로커에 "이 작업을 실행해달라"는 메시지가 발행되고, Worker가 그 메시지를 가져와 실행한다. 이 구조 자체는 Kafka/RabbitMQ의 Producer-Broker-Consumer 흐름(Day 1)과 동일하다 — Celery는 그 위에 Python 함수 호출을 메시지로 직렬화해주는 계층을 얹은 것뿐이다.

  1. Celery의 기본 ACK 시점은 "실행 전"이다
    Celery의 기본 설정(task_acks_late=False)은 Worker가 메시지를 받자마자 브로커에 ACK를 보낸다. 즉 실제로 함수가 실행되기도 전에 브로커 입장에서는 "이 작업은 끝났다"고 처리된다. 이 상태에서 Worker 프로세스가 재시작되면, 브로커는 이미 ACK를 받았기 때문에 그 메시지를 다시 보내주지 않는다 — 실행되던 이미지 리사이징 작업이 그대로 유실되는 이유다.
  2. acks_late=True로 바꾸면 유실은 막지만 중복이 생긴다
    task_acks_late=True로 설정하면 ACK 시점이 "작업 실행 완료 후"로 바뀐다. Worker가 죽으면 브로커는 ACK를 받지 못했으므로 그 메시지를 다른 Worker에게 다시 전달한다(RabbitMQ의 redelivery, Day 3). 문제는 원래 Worker가 이미 리사이징을 절반쯤 끝낸 상태에서 죽었을 수도 있다는 점이다 — 이 경우 재전달된 작업이 다시 실행되며 같은 이미지가 두 번 리사이징된다.
  3. 이것은 At-most-once vs At-least-once의 재현이다(Day 1)
    acks_late=False는 사실상 At-most-once(최대 한 번 실행, 유실 가능)이고, acks_late=True는 At-least-once(최소 한 번 실행, 중복 가능)다. Celery가 마법처럼 이 문제를 없애주는 게 아니라, 어느 실패 모드를 선택할지 옵션으로 제공할 뿐이며 나머지는 태스크 설계(Idempotency, Day 5)의 몫이다.
핵심 직관: "Celery를 쓰면 재시도와 신뢰성이 알아서 해결된다"는 생각은 틀렸다. Celery는 브로커 위에 얹힌 태스크 실행 프레임워크일 뿐이고, ACK 시점을 언제로 할지는 결국 개발자가 acks_late로 명시적으로 결정해야 하는 신뢰성 트레이드오프다.
왜 면접에서 묻는가

이 질문이 평가하는 것

"Celery로 비동기 처리하면 됩니다"는 답이 아니다. 면접관은 Celery라는 도구 뒤에 숨어 있는 메시지 큐의 근본 원리를 여전히 적용할 줄 아는지를 본다.

  • ACK 시점을 설계 선택으로 인식하는가: 기본값(acks_late=False)을 그대로 쓰면 유실이 생길 수 있다는 것을 알고, 이를 켰을 때 생기는 중복이라는 새로운 문제까지 함께 설명할 수 있는지 본다. "이렇게 하면 해결됩니다"가 아니라 "이렇게 하면 다른 문제가 생기니 이렇게 방어합니다"까지 가야 시니어다.
  • Idempotency와의 연결을 놓치지 않는가: acks_late=True를 켜는 순간 태스크는 반드시 멱등해야 한다(Day 5). 이 연결을 스스로 짚지 못하면, "재시도는 설정했지만 중복 실행에 대한 방어가 없는" 절반짜리 설계가 된다.
  • 큐/Worker 분리로 격리를 설계하는가: 결제 확인처럼 긴급한 작업이 이미지 리사이징 뒤에 밀리는 문제는 "Celery의 priority 기능을 쓰면 된다"는 단편적 답으로는 부족하다. 브로커별로 priority 지원 수준이 다르다는 것과, 실무에서는 큐 자체를 분리하고 전용 Worker를 배정하는 방식이 더 신뢰성 있다는 것을 아는지 본다.
  • 다단계 워크플로우 실패 처리를 설계할 수 있는가: 여러 후속 작업을 chain/group/chord 같은 Canvas로 구성할 때, 중간 단계가 실패하면 전체 워크플로우가 어떻게 되는지 — 특히 chord의 콜백이 무한정 대기하는 함정을 아는지가 실전 운영 경험을 가른다.
실무 연결

Priority는 옵션이 아니라 큐 분리로 해결한다

"Celery에 priority 파라미터가 있으니 그걸 쓰면 되지 않나?"는 실무에서 흔히 걸려 넘어지는 지점이다. Priority 지원 여부와 신뢰도는 브로커에 따라 완전히 다르다.

브로커 Priority 지원 실무 권장 방식
RabbitMQ 큐 생성 시 x-max-priority로 우선순위 큐 지정 가능(0~10 권장) 지원은 되지만, 우선순위가 촘촘히 섞인 큐 하나에서 처리 순서를 보장하는 것보다, 결제처럼 중요한 작업은 아예 별도 큐(payments)로 분리해 다른 작업과 자원 경쟁 자체가 없도록 하는 편이 운영상 예측 가능하다.
Redis List 자료구조 기반이라 사실상 네이티브 priority 없음(kombu가 priority별로 별도 키를 만들어 우회) Redis를 브로커로 쓴다면 in-queue priority에 의존하지 말고, 큐 이름 자체를 역할별로 나누는 것이 사실상 유일한 신뢰 가능한 방법이다.

그래서 실무 설계는 다음과 같은 형태가 된다: 작업 종류별로 큐를 나누고 (payments, notifications, image_processing, stats), Worker를 큐 단위로 기동한다. 

# 결제 확인 전용 Worker — 무거운 작업과 완전히 분리된 자원
celery -A myapp worker -Q payments --concurrency=4 -n payments@%h

# 이미지 리사이징 전용 Worker — CPU 바운드라 prefork, concurrency는 CPU 코어 수 기준
celery -A myapp worker -Q image_processing --concurrency=2 -n image@%h --pool=prefork

# 알림/통계 전용 Worker — I/O 바운드라 gevent로 동시성 확보
celery -A myapp worker -Q notifications,stats --concurrency=100 -n io@%h --pool=gevent
흔한 오해: "Worker 하나로 여러 큐를 다 처리하게 하고 -Q로만 구분하면 충분하다." 큐를 구분해도 같은 Worker 프로세스가 여러 큐를 순회하며 가져온다면, 무거운 작업 하나가 오래 걸리는 동안 그 Worker의 슬롯은 여전히 묶여 있다. 진짜 격리는 큐 분리 + 전용 Worker(별도 프로세스/파드) 배정까지 가야 완성된다.
핵심 개념

acks_late를 켰다면 반드시 따라와야 하는 것들

  1. Idempotency 설계(Day 5)를 태스크 레벨에 적용한다
    이미지 리사이징 태스크는 "이미 이 원본 이미지의 이 사이즈 리사이징 결과가 존재하는지"를 먼저 확인하거나, 결과 파일명을 원본 이미지 ID 기반으로 결정적(deterministic)으로 만들어서 같은 작업이 두 번 실행돼도 같은 파일을 덮어쓰는 것으로 끝나게 만든다. 통계 집계처럼 "더하기" 연산은 특히 위험하다 — 중복 실행되면 카운트가 두 배로 늘어나므로, 집계 대상 이벤트 ID를 기록해 두고 이미 처리한 이벤트는 건너뛰는 방식으로 멱등하게 만들어야 한다.
  2. 재시도는 Exponential Backoff + Jitter로 설정한다(Day 7과 연결)
    Celery는 autoretry_for, retry_backoff=True, retry_backoff_max, retry_jitter=True 옵션으로 Day 7에서 다룬 지수 백오프와 지터를 그대로 구현할 수 있다. 일시적 실패(외부 이메일 API 타임아웃)와 영구적 실패(잘못된 이미지 포맷)를 구분해서, 후자는 max_retries에 도달하기 전에 즉시 실패 처리하고 DLQ 개념에 해당하는 실패 로그/알림으로 보내야 한다.
  3. prefetch를 처리 시간 특성에 맞게 조정한다(Day 8과 연결)
    acks_late=True인 상태에서 worker_prefetch_multiplier 기본값(4)을 그대로 두면, 오래 걸리는 이미지 리사이징 작업들을 Worker 하나가 여러 개 미리 가져가 쥐고 있으면서 뒤에 도착한 짧은 작업들이 대기하게 된다. 처리 시간이 긴 작업 큐는 worker_prefetch_multiplier=1로 낮춰서 "한 번에 하나씩만 가져가고 끝나면 다음 것"으로 동작시키는 것이 일반적이다.
면접 팁: "acks_late를 켜겠습니다"에서 멈추지 말고 "acks_late를 켜면 Worker 재시작 시 중복 실행 가능성이 생기므로, 태스크를 멱등하게 만들고 재시도는 지수 백오프로, 무거운 작업 큐는 prefetch를 낮춰 공정하게 분배되도록 하겠습니다"까지 하나의 묶음으로 설명하면 설계가 완결됐다는 인상을 준다.
핵심 개념

회원가입 후속 작업 4개를 어떻게 엮을 것인가 — Canvas(chain/group/chord)

네 가지 작업(이메일, 리사이징, 추천 초기화, 통계 집계)은 서로 순서 의존성이 다르다. Celery의 Canvas 기능으로 이 관계를 명시적으로 표현할 수 있다.

  • group — 서로 독립적인 작업을 병렬 실행: 이메일 발송, 추천 초기화, 통계 집계는 서로 순서와 무관하다. 이 셋은 group()으로 묶어 동시에 실행되도록 한다.
  • chain — 순서가 있는 작업을 직렬 연결: 이미지 리사이징은 원본 업로드가 끝난 뒤에만 실행 가능하므로, 업로드 완료 → 리사이징을 chain()으로 연결한다. 이전 단계의 결과가 다음 단계의 입력이 될 때 쓴다.
  • chord — 병렬 작업을 모두 끝낸 뒤 콜백 실행: 만약 "네 작업이 모두 끝난 뒤 가입 완료 알림 하나를 최종 발송"하는 요구사항이 있다면, chord(group(...))(callback) 구조로 네 작업을 그룹으로 병렬 실행하고 모두 끝난 뒤 콜백을 실행한다. 
from celery import chain, group, chord

workflow = chord(
    group(
        send_welcome_email.s(user_id),
        resize_profile_image.s(user_id),
        init_recommendation.s(user_id),
        aggregate_signup_stats.s(user_id),
    ),
    notify_signup_complete.s(user_id),  # 넷 다 끝난 뒤 실행되는 콜백
)
workflow.apply_async()
실무에서 자주 걸리는 함정 — chord의 "부분 실패" 처리: group 안의 작업 하나(예: 리사이징)가 최종적으로 실패하면, 기본 설정에서는 콜백(notify_signup_complete)이 실행되지 않고 chord 자체가 실패 상태로 멈춘다. 개발자가 link_error로 에러 콜백을 명시적으로 연결해두지 않으면, 이 실패가 조용히 묻혀서 "가입 완료 알림이 왜 안 갔지?"를 디버깅하는 데 오랜 시간이 걸린다. 특히 리사이징 실패가 이메일 발송 성공 여부와 무관해야 한다면(각 작업이 서로의 성공에 의존하지 않아야 한다면), 애초에 chord로 묶기보다 완전히 독립적인 group으로 실행하고 각 태스크의 실패를 개별적으로 모니터링하는 편이 나을 수 있다.
답변 사고 순서

면접에서 이 질문을 받았을 때 생각하는 순서

  1. 사고 원인을 ACK 시점 문제로 짚는다
    "작업 유실과 중복 둘 다 발생했다는 건, ACK 시점 설정이 명시적으로 고려되지 않았다는 신호입니다. 기본값(acks_late=False)이면 Worker 재시작 시 유실이, acks_late=True로 단순히 바꾸기만 하면 중복이 발생할 수 있습니다."
  2. 신뢰성 방향을 정하고 그에 따르는 대가를 함께 설계한다
    "저는 유실보다 중복이 다루기 쉬운 문제라고 판단해 acks_late=True를 선택하고, 대신 각 태스크를 멱등하게 설계해서(원본 ID 기반 결정적 파일명, 처리 완료 이벤트 ID 기록) 중복 실행이 부작용 없이 흡수되도록 하겠습니다."
  3. 격리 문제는 큐/Worker 분리로 해결한다
    "결제 확인 지연 문제는 Celery의 priority 파라미터에 의존하지 않고, 결제 작업을 별도 큐(payments)와 전용 Worker로 완전히 분리해서 이미지 리사이징과 자원 경쟁 자체가 없도록 하겠습니다."
  4. 회원가입 후속 작업의 관계를 Canvas로 명시한다
    "이메일 발송·추천 초기화·통계 집계는 서로 독립적이니 group으로 병렬 실행하고, 리사이징은 업로드 완료 이후 순서가 필요하니 chain으로 연결합니다. 하나의 실패가 다른 작업의 성공 여부와 무관해야 하므로, 이 넷을 chord로 묶어 하나의 콜백에 의존시키기보다는 독립적으로 실행하고 각각 실패를 모니터링하겠습니다."
  5. 재시도와 prefetch까지 마무리한다
    "일시적 실패는 지수 백오프로 재시도하고, 영구적 실패는 즉시 실패 로그로 남깁니다. 리사이징처럼 오래 걸리는 큐는 prefetch를 1로 낮춰 짧은 작업들이 뒤에서 굶지 않도록 합니다."
핵심 답변 예시

실제 면접 답변 예시

"두 사고 모두 ACK 시점과 큐 설계가 명시적으로 고려되지 않아서 생긴 문제로 보입니다. 기본 설정(acks_late=False)에서는 Worker가 메시지를 받자마자 ACK하기 때문에 재시작 시점에 실행 중이던 작업이 그대로 유실됩니다. 저는 유실보다 중복이 다루기 쉬운 실패 모드라고 보고 acks_late=True로 전환하되, 이미지 리사이징은 원본 이미지 ID 기반의 결정적 파일명을 쓰고, 통계 집계는 처리한 이벤트 ID를 기록해서 중복 실행이 들어와도 결과가 달라지지 않게 멱등하게 만들겠습니다.

결제 확인 지연 문제는 Celery의 priority 옵션에 의존하지 않고, 결제 작업을 별도 큐와 전용 Worker로 완전히 분리해서 이미지 리사이징 같은 무거운 작업과 자원 경쟁이 아예 발생하지 않도록 하겠습니다. 회원가입 후속 작업은 이메일 발송·추천 초기화·통계 집계처럼 서로 독립적인 것은 group으로 병렬 실행하고, 리사이징처럼 업로드 완료 이후에만 실행 가능한 것은 chain으로 연결합니다. 이 넷을 chord로 묶어 하나의 콜백에 의존시키면 리사이징 실패가 알림 발송까지 막아버릴 수 있으니, 서로 독립적으로 실행하고 각각의 실패를 별도로 모니터링하겠습니다. 마지막으로 일시적 실패는 지수 백오프로 재시도하고, 리사이징처럼 오래 걸리는 큐는 prefetch를 1로 낮춰 짧은 작업이 뒤에서 밀리지 않게 하겠습니다."

Senior-Level Follow-up

면접관이 이어서 물어볼 수 있는 질문들

Q. "통계 집계 태스크를 멱등하게 만들기 위해 '처리한 이벤트 ID를 기록'한다고 하셨는데, 그 기록 자체와 카운트 증가를 별도 연산으로 하면 또 경쟁 조건이 생기지 않습니까?"
맞습니다. "이미 처리했는지 확인 후 카운트를 올리는" 두 단계로 나누면 Day 5에서 다룬 check-then-act 결함이 그대로 재현됩니다. 실제로는 DB의 유니크 제약을 건 "처리 이벤트 기록" 테이블에 이벤트 ID를 유니크 키로 삽입하는 시도를 먼저 하고, 삽입이 성공했을 때만 카운트를 증가시키는 원자적 흐름으로 만들어야 합니다. 삽입이 유니크 제약 위반으로 실패하면 "이미 처리된 이벤트"로 간주하고 카운트 증가를 건너뜁니다.
Q. "Redis를 브로커로 쓸 때 acks_late의 신뢰성은 RabbitMQ와 동일합니까?"
아닙니다. RabbitMQ는 Consumer 연결이 끊기면 브로커가 그 사실을 즉시 알고 미확인 메시지를 다른 Consumer에게 재전달합니다. Redis를 브로커로 쓰는 Celery(kombu의 redis transport)는 이런 네이티브 연결 추적이 약해서, Worker가 죽었는지 판단하는 데 visibility_timeout이라는 시간 기반 방식에 의존합니다. 즉 Worker가 죽은 시점부터 visibility_timeout이 지나야 메시지가 재전달되므로, 실제 죽음과 재전달 사이에 지연이 생기고, 그 값을 작업의 최대 실행 시간보다 짧게 설정하면 아직 실행 중인 작업의 메시지가 시간 초과로 오판되어 중복 실행될 위험도 커집니다. Redis는 간단하지만 이 타임아웃 튜닝에 더 신경 써야 합니다.
Q. "이미지 리사이징 Worker에 prefork 풀을 쓰겠다고 하셨는데, 왜 gevent가 아니라 prefork입니까?"
이미지 리사이징은 CPU 바운드 작업입니다. gevent/eventlet 같은 그린렛 기반 동시성은 I/O 대기 시간을 다른 태스크 실행에 쓸 수 있게 해주는 것이지, CPU 연산 자체를 병렬화해 주지 않습니다. Python의 GIL 때문에 그린렛 방식으로는 CPU 바운드 작업을 동시에 여러 개 돌려도 실제로는 한 번에 하나씩만 CPU를 쓰게 되어 이점이 없습니다. prefork(멀티프로세스)는 프로세스마다 별도의 GIL을 가지므로 CPU 코어 수만큼 실제 병렬 연산이 가능합니다. 반대로 알림 발송처럼 외부 API 호출 대기가 대부분인 I/O 바운드 작업은 gevent로 훨씬 적은 메모리로 훨씬 높은 동시성을 얻을 수 있습니다.
Q. "결제 작업을 별도 큐로 분리했는데, 그 전용 Worker마저 다운되면 어떻게 됩니까? DLQ와는 어떻게 연결하시겠습니까?"
큐 분리는 다른 작업과의 자원 경쟁을 없애는 것이지 그 자체로 가용성을 보장하지는 않으므로, payments 큐도 최소 2대 이상의 Worker로 이중화해야 합니다. 그리고 Day 6에서 다룬 DLQ 개념을 그대로 적용해서, max_retries를 넘긴 결제 태스크는 실패로 끝내는 게 아니라 별도의 실패 큐나 실패 테이블로 옮기고 즉시 알림을 보내야 합니다. 결제처럼 금전이 걸린 작업은 "재시도 다 하고 그냥 실패 로그만 남기는" 수준이 아니라, 사람이 확인하고 수동으로 재처리하거나 보상 트랜잭션을 트리거할 수 있는 경로까지 마련해야 합니다.
답변 품질 비교

좋은 답변과 약한 답변의 차이

강한 답변 — 이렇게 말하면 시니어로 보인다

"두 사고의 근본 원인을 ACK 시점과 큐 설계 부재로 짚고, acks_late 전환이 만드는 새로운 문제(중복)까지 함께 설계에 포함시킵니다. Priority는 브로커 기능에 의존하지 않고 큐/Worker 분리로 해결하고, 회원가입 후속 작업들 사이의 의존 관계를 chain/group으로 정확히 표현하며, chord의 부분 실패 함정까지 미리 짚어 설계를 완결시킵니다."

→ 원인 진단 → 신뢰성 트레이드오프 선택과 그 대가 방어 → 격리 설계 → 워크플로우 의존성 모델링 → 실패 처리까지 하나의 흐름

약한 답변 — 이렇게 말하면 주니어로 보인다

"acks_late=True로 설정하고, 태스크에 priority를 높게 줘서 결제 작업을 먼저 처리되게 하겠습니다."

→ acks_late가 만드는 중복 문제에 대한 인식이 없고, Redis 브로커에서는 priority가 사실상 신뢰할 수 없다는 것도 모른 채 옵션 이름만 나열함.

약한 답변 — 도구 이름만 나열

"Celery의 chain, group, chord를 다 써서 워크플로우를 구성하겠습니다."

→ 어떤 작업이 왜 chain이고 왜 group인지 설계 근거가 없고, chord를 썼을 때 부분 실패가 콜백을 막아버릴 수 있다는 실패 모드에 대한 고려가 전혀 없음.

오늘 배운 것 요약

Day 9 핵심 정리

  • Celery는 메시지 큐 위에 얹힌 태스크 실행 프레임워크일 뿐이고, ACK 시점 (acks_late)은 여전히 At-most-once(유실 위험)와 At-least-once(중복 위험) 사이의 선택이다.
  • acks_late=True를 선택하면 태스크 멱등성 설계가 반드시 함께 따라와야 한다 — 결정적 결과 생성, 처리 이벤트 ID 기록 등으로 중복 실행을 흡수한다.
  • Priority는 브로커 기능(RabbitMQ x-max-priority)에 의존하기보다, 전용 큐 + 전용 Worker 분리가 더 예측 가능한 격리 방법이다. Redis 브로커는 특히 네이티브 priority가 약하다.
  • chain은 순서 의존 작업, group은 독립적인 병렬 작업, chord는 병렬 작업 완료 후 콜백에 쓰지만, 콜백을 부분 실패로부터 보호하는 link_error 설계 없이는 실패가 조용히 묻힐 수 있다.
  • Worker 풀 선택(prefork vs gevent)은 작업이 CPU 바운드인지 I/O 바운드인지에 따라 결정하고, 오래 걸리는 작업 큐는 prefetch를 낮춰 짧은 작업이 뒤에서 굶지 않게 한다.
