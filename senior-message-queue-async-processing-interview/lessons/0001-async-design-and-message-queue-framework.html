비동기 처리와 메시지 큐 설계 프레임워크

2026-06-30 · 예상 학습 시간 60분
비동기 설계 메시지 큐 At-least-once 설계 프레임워크 장애 격리
오늘의 면접 질문
"결제 완료 후 이메일 발송, 재고 차감, 포인트 적립을 처리해야 합니다. 동기 처리와 비동기 처리 중 어떻게 결정하시겠습니까? 메시지 큐를 도입한다면 설계를 어떻게 하시겠습니까?"
전제 개념

동기 처리와 비동기 처리 — 근본 차이

동기 처리(Synchronous)란 요청한 쪽이 결과가 나올 때까지 기다리는 방식이다. 고객이 결제 버튼을 누르면, 서버가 이메일 발송 → 재고 차감 → 포인트 적립을 순서대로 모두 완료한 후에야 "결제 완료" 응답을 보낸다. 모든 작업이 같은 HTTP 요청-응답 사이클 안에 들어 있다.

비동기 처리(Asynchronous)란 요청한 쪽이 결과를 기다리지 않고, 작업을 다른 대리자에게 위임하는 방식이다. 결제 서버는 "결제 완료" 이벤트를 메시지 큐에 넣고 즉시 고객에게 응답을 보낸다. 이메일 발송, 재고 차감, 포인트 적립은 별도 Consumer가 나중에 처리한다.

구분 동기 처리 비동기 처리 (메시지 큐)
응답 시간 모든 작업 완료까지 대기 → 느릴 수 있음 큐에 넣는 시간만 소요 → 빠름
장애 전파 이메일 서버 장애 → 결제 전체 실패 이메일 서버 장애가 결제에 영향 없음
일관성 즉시 일관성 보장 가능 최종 일관성(Eventual Consistency)
복잡도 단순 — 코드 흐름 직선 높음 — 중복 처리, 순서, 장애 복구 고려 필요
확장성 결제 처리량이 이메일 발송 속도에 종속됨 결제와 이메일 발송을 독립적으로 스케일
핵심 직관: 동기는 "모두 같이 가야 한다"는 계약이고, 비동기는 "일단 나는 갈게, 너는 나중에 따라와"라는 위임이다. 위임을 선택하면 속도와 격리를 얻지만, 위임된 작업이 실패했을 때 어떻게 알고 어떻게 복구하는지를 별도로 설계해야 한다.
왜 면접에서 묻는가

이 질문이 평가하는 것

이 질문은 단순히 "Kafka 써봤어요?"를 확인하는 것이 아니다. 면접관이 보고 싶은 것은 다음 세 가지다.

  • 판단 기준이 있는가: "다 비동기로 처리해야 한다" 또는 "동기면 충분하다"처럼 맥락 없이 한 방향을 고집하면 약하다. 어떤 조건에서 비동기가 필요하고, 어떤 조건에서 동기가 더 낫는지를 구분하는가.
  • 장애 시나리오를 먼저 떠올리는가: 메시지 큐를 도입하면 "Consumer가 죽으면 어떻게 되나?", "메시지가 두 번 처리되면?"같은 새로운 실패 모드가 생긴다. 이를 먼저 제기하는 사람이 시니어다.
  • 트레이드오프를 말할 수 있는가: "비동기가 더 좋습니다"가 아니라 "이 시나리오에서는 최종 일관성을 허용할 수 있고, 이메일 서버 장애가 결제에 전파되는 것이 더 큰 리스크이므로 비동기를 선택합니다"라고 말하는가.
실무 연결

실제 서비스에서 이 선택이 나타나는 상황

결제 후처리는 이 판단이 가장 선명하게 드러나는 시나리오다. 결제 자체(카드 승인, 금액 차감)는 반드시 동기로 처리해야 한다 — "결제는 됐는데 실제로 승인이 됐는지 모르는" 상태를 허용할 수 없기 때문이다.

반면 결제 후 수행하는 작업들은 성격이 다르다:

후처리 작업 동기 허용? 판단 이유
이메일/SMS 발송 비동기 권장 발송 지연이 결제 경험에 영향 주면 안 됨. 발송 실패가 결제를 무효화하면 안 됨.
재고 차감 상황에 따라 재고가 즉시 일관성이 필요한 경우 동기. 약간의 지연을 허용하면 비동기 가능.
포인트 적립 비동기 권장 포인트는 수 초 내 반영이면 충분. 포인트 서버 장애가 결제를 막으면 안 됨.
정산/회계 기록 비동기 권장 배치 집계 가능. 실시간 필요 없음.
사기 탐지 알림 비동기 권장 탐지가 결제를 막는 구조는 별도 flow로 분리.
흔한 오해: "재고는 정확해야 하니까 동기로 처리해야 한다." 그러나 재고를 낙관적 잠금(Optimistic Locking)이나 예약 방식으로 설계하면 비동기로도 정확도를 확보할 수 있다. '비동기 = 부정확'이 아니다. 비동기는 '결과를 나중에 확인하는' 것이고, 최종 일관성은 설계로 보장한다.
핵심 개념

메시지 큐 도입 결정 — 4가지 판단 축

면접에서 "비동기로 가야 한다"고 주장할 때 근거로 사용하는 4가지 판단 기준이다.

  1. 응답 시간 요구사항 (Latency Requirement)
    사용자가 이 작업의 완료를 기다려야 하는가? 이메일 발송은 사용자가 응답 안에서 결과를 볼 필요가 없다 → 비동기 가능. 결제 승인은 사용자가 결과를 바로 봐야 한다 → 동기 필요.
  2. 장애 격리 요구사항 (Failure Isolation)
    이 작업의 실패가 핵심 흐름을 막으면 안 되는가? 이메일 서버 장애가 결제를 실패시키면 안 된다 → 비동기로 분리해야 한다. 분리하면 이메일 서버가 죽어도 결제는 성공하고, 나중에 이메일만 재처리하면 된다.
  3. 처리량 불균형 (Throughput Mismatch)
    Producer(이벤트 발생 속도)와 Consumer(처리 속도)의 속도가 다른가? 결제는 초당 1,000건인데 포인트 서버 처리 속도가 200건/초라면, 동기 처리 시 결제가 병목에 걸린다. 큐를 버퍼로 두면 속도 차이를 흡수할 수 있다.
  4. 최종 일관성 허용 여부 (Eventual Consistency Tolerance)
    이 데이터는 "결국에는 맞으면 되는가", 아니면 "지금 당장 정확해야 하는가"? 포인트 잔액은 수 초 후에 반영돼도 큰 문제가 없다 → 최종 일관성 허용 → 비동기 가능. 재고는 초과 판매를 막아야 한다면 즉시 일관성 필요 → 신중한 설계 필요.
면접 팁: 이 4가지를 스스로 언급하면서 각 항목에 시나리오를 대입하면 "이 분은 판단 기준이 있다"는 인상을 준다. "비동기가 더 좋습니다"보다 "응답 시간과 장애 격리 관점에서 이 3개는 비동기, 재고는 검토가 필요합니다"가 훨씬 강하다.
핵심 개념

메시지 큐 기본 구조 — Producer → Broker → Consumer

메시지 큐 설계를 이야기할 때 사용하는 기본 용어다. 면접에서 이 용어를 정확히 사용해야 한다.

┌─────────────┐        ┌────────────────────┐        ┌──────────────────────┐
│  Producer   │──────▶ │  Message Broker    │──────▶ │  Consumer            │
│  (결제 서버) │  메시지  │  (Kafka / RabbitMQ │  메시지  │  (이메일 서비스)       │
│             │  발행   │   / SQS)           │  소비   │  (재고 서비스)         │
└─────────────┘        └────────────────────┘        │  (포인트 서비스)        │
                                                      └──────────────────────┘
  • Producer: 이벤트를 생성하여 큐에 넣는 쪽. 결제 서버가 "결제 완료" 이벤트를 발행.
  • Broker: 메시지를 받아 저장하고 Consumer에게 전달하는 중간 시스템. Kafka, RabbitMQ, AWS SQS 등.
  • Consumer: 큐에서 메시지를 꺼내 처리하는 쪽. 이메일 서비스, 재고 서비스 등이 각자 소비.
  • Topic / Queue: 메시지가 쌓이는 채널. Kafka는 Topic, RabbitMQ는 Queue.
메시지 전달 보장 수준 3가지 — 반드시 구분해야 함:

At-most-once (최대 한 번): 메시지가 유실될 수 있지만 중복은 없다. 로그 수집 등 유실을 허용하는 경우.
At-least-once (최소 한 번): 메시지가 반드시 전달되지만 중복 가능. 가장 흔한 설정. Consumer가 처리를 확인(ACK)하기 전에 죽으면 재전달됨 → Consumer가 멱등하게 처리해야 함.
Exactly-once (정확히 한 번): 중복도 유실도 없음. 구현이 복잡하고 성능 비용이 크다. Kafka 트랜잭션으로 가능하지만 운영 부담이 큼.
답변 사고 순서

면접에서 이 질문을 받았을 때 생각하는 순서

  1. 요구사항 확인부터 시작
    "우선 각 작업의 요구사항을 정리하겠습니다. 이메일 발송과 포인트 적립은 실시간 응답에 포함될 필요가 없고, 서비스 장애가 결제에 전파되면 안 됩니다. 재고 차감은 초과 판매 가능성이 있어 별도로 검토하겠습니다."
  2. 비동기 처리 범위 결정 + 이유 명시
    "이메일과 포인트는 장애 격리와 응답 시간 단축을 위해 비동기로 분리합니다. 재고는 즉시 일관성이 필요한지 추가 확인이 필요합니다."
  3. 메시지 큐 구조 설계
    "결제 완료 이벤트를 Topic에 발행하고, 이메일 Consumer, 포인트 Consumer가 각자 독립적으로 구독합니다. Consumer Group을 분리하면 이메일 처리가 느려도 포인트 처리에 영향이 없습니다."
  4. 실패 시나리오 먼저 언급
    "Consumer가 메시지를 처리 도중 죽으면 At-least-once 보장으로 재전달됩니다. 이때 이메일이 두 번 발송될 수 있으므로, Consumer는 멱등하게 처리해야 합니다. 예를 들어 '이미 이 결제 ID로 이메일을 보냈으면 스킵'하는 로직이 필요합니다."
  5. 운영 관점 마무리
    "재처리 실패 시 Dead-Letter Queue로 격리하고 알림을 받습니다. 큐 길이와 Consumer 지연(lag)을 모니터링해서 처리량 이슈를 조기에 감지합니다."
핵심 답변 예시

실제 면접 답변 예시

"우선 각 작업을 장애 격리 관점에서 분류하겠습니다. 이메일 발송과 포인트 적립은 결제 응답에 포함되지 않아도 되고, 이 서비스들이 느리거나 장애가 났을 때 결제가 실패해서는 안 됩니다. 따라서 이 두 작업은 메시지 큐를 통한 비동기 처리로 분리합니다.

설계는 결제 서버가 '결제 완료' 이벤트를 Kafka Topic에 발행하고, 이메일 Consumer와 포인트 Consumer가 각자 독립된 Consumer Group으로 구독하는 구조입니다. 그러면 이메일 처리가 병목이 되어도 포인트 적립에는 영향을 주지 않습니다.

Consumer가 처리 중 죽으면 At-least-once 보장에 의해 메시지가 재전달됩니다. 중복 이메일 발송을 막기 위해 Consumer는 결제 ID를 기준으로 멱등하게 처리해야 합니다. 처리 실패가 반복되면 Dead-Letter Queue로 격리하고 운영팀 알림을 보냅니다.

재고 차감은 초과 판매 허용 여부에 따라 다릅니다. 재고 예약(Reserve) 방식을 사용하면 비동기로도 정확도를 유지할 수 있지만, 즉시 일관성이 필요하다면 결제 트랜잭션 안에서 동기로 처리하는 게 안전합니다."

Senior-Level Follow-up

면접관이 이어서 물어볼 수 있는 질문들

Q. "Consumer가 메시지를 처리했는데 ACK를 보내기 전에 죽으면 어떻게 됩니까?"
At-least-once 보장이 적용됩니다. 브로커는 ACK를 받지 못하면 메시지를 재전달합니다. 따라서 Consumer는 같은 메시지를 두 번 받을 수 있다고 가정하고 설계해야 합니다. 결제 ID를 처리 완료 테이블에 기록하고, 이미 처리된 ID가 들어오면 스킵하는 방식으로 멱등성을 보장합니다. "처리 완료 레코드 삽입"이 트랜잭션 안에 있어야 정합성이 보장됩니다.
Q. "Consumer가 너무 느려서 큐가 쌓이면 어떻게 됩니까?"
Consumer Lag이 증가합니다. 대응책은 두 가지입니다. 첫째, Consumer를 수평 확장합니다 — Kafka는 파티션 수만큼 Consumer를 늘릴 수 있습니다. 둘째, Consumer의 처리 로직을 최적화합니다 — DB 쿼리 최적화, 배치 처리 도입 등. Backpressure 메커니즘으로 Producer 속도를 조절하는 방법도 있지만, 결제처럼 Producer를 늦출 수 없는 경우에는 Consumer 스케일아웃이 우선입니다.
Q. "Kafka와 RabbitMQ 중 이 시나리오에서는 어떤 걸 선택하시겠습니까?"
이 시나리오에서는 두 가지 모두 가능하지만 판단 기준을 먼저 말하겠습니다. 메시지를 영구 보관하고 Consumer가 과거 메시지를 재처리해야 한다면 Kafka가 낫습니다. 유연한 라우팅과 Priority Queue가 필요하다면 RabbitMQ가 낫습니다. 결제 후처리에서 "6개월 전 메시지를 재처리해야 할 경우가 있는가"가 핵심 질문입니다. 이미 AWS를 사용한다면 SQS가 운영 부담이 적어 실용적일 수 있습니다.
Q. "메시지가 순서대로 처리되어야 한다는 요구사항이 생기면 어떻게 하시겠습니까?"
순서 보장은 비동기 처리에서 가장 어려운 문제 중 하나입니다. Kafka는 파티션 내에서만 순서를 보장합니다 — 같은 사용자의 이벤트를 같은 파티션으로 보내는 파티션 키 설계가 필요합니다. SQS는 Standard Queue는 순서 미보장, FIFO Queue는 순서 보장하지만 처리량이 제한됩니다. "어느 단위에서 순서가 보장되어야 하는가"를 먼저 확인해야 합니다 — 전체 글로벌 순서가 필요한지, 특정 사용자/엔티티 단위 순서인지에 따라 설계가 달라집니다.
답변 품질 비교

좋은 답변과 약한 답변의 차이

강한 답변 — 이렇게 말하면 시니어로 보인다

"이메일과 포인트는 장애 격리가 필요해서 비동기로 분리합니다. Consumer가 죽으면 메시지가 재전달되므로, 결제 ID를 멱등 키로 사용해서 중복 처리를 막습니다. 실패가 반복되면 DLQ로 격리하고 모니터링합니다. 재고는 초과 판매 허용 여부를 먼저 확인해야 합니다."

→ 왜 비동기인지 이유 + 장애 시나리오 선제적 언급 + 구체적 대응 방법 + 재고의 다른 처리 방식 인지

약한 답변 — 이렇게 말하면 주니어로 보인다

"메시지 큐를 사용해서 비동기로 처리하면 됩니다. Kafka나 RabbitMQ를 쓰면 빠르게 처리할 수 있습니다."

→ 왜 비동기인지 판단 기준 없음. 장애 시나리오 언급 없음. "빠르다"만 말하고 트레이드오프 없음.

약한 답변 — 한 방향으로만 고집

"모두 다 비동기로 처리하면 됩니다. 동기 처리는 느리니까요."

→ 재고 차감처럼 즉시 일관성이 필요할 수 있는 작업을 고려하지 않음. 비동기에서 발생하는 복잡도를 모름.

오늘 배운 것 요약

Day 1 핵심 정리

  • 비동기 처리는 "빠르다"가 핵심이 아니라 장애 격리, 처리량 분리, 최종 일관성이 핵심이다.
  • 비동기 도입 판단은 4가지 축으로: 응답 시간 / 장애 격리 / 처리량 불균형 / 최종 일관성 허용 여부.
  • At-least-once가 가장 흔한 설정이며, Consumer는 멱등하게 설계해야 한다.
  • Producer → Broker → Consumer 구조에서 각 역할과 책임이 분리된다.
  • 실패는 DLQ로 격리하고, Consumer Lag을 모니터링하는 것이 운영의 기본이다.
  • 재고처럼 즉시 일관성이 필요한 작업은 비동기 도입 전에 초과 판매 허용 여부를 먼저 확인해야 한다.
