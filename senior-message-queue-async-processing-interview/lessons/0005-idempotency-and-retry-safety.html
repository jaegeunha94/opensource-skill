Idempotency (멱등성) 설계와 재시도 안전성

2026-07-04 · 예상 학습 시간 80분
Idempotency Key DB Unique Constraint 상태 기반 중복 방지 At-least-once Race Condition
오늘의 면접 질문
"결제 승인 이벤트를 처리하는 Consumer가 있습니다. 이 Consumer가 결제 대행사 승인 API를 호출한 직후 네트워크 타임아웃이 발생해서, 메시지 큐가 같은 메시지를 다시 전달했습니다. 실제로는 결제가 성공했을 수도, 실패했을 수도 있는 상황입니다. 이 Consumer를 멱등(idempotent)하게 설계한다면 어떻게 하시겠습니까? 그리고 두 개의 Consumer 인스턴스가 같은 메시지를 동시에 받는 경우까지 고려하면 설계가 어떻게 달라져야 합니까?"
전제 개념

멱등성이란 "같은 요청을 몇 번 보내도 결과가 한 번 보낸 것과 같다"는 성질

멱등(idempotent)이라는 단어 자체가 어렵게 느껴지지만, 뜻은 단순하다. 같은 작업을 한 번 실행하든 열 번 실행하든, 시스템이 최종적으로 도달하는 상태와 호출자가 받는 결과가 동일해야 한다는 것이다. HTTP로 비유하면 GET /users/1은 몇 번을 호출해도 결과가 같으므로 자연히 멱등하고, POST /payments로 "결제를 승인해줘"라고 매번 새로 요청하면 호출할 때마다 새 결제가 생기므로 멱등하지 않다.

Day 4에서 배운 SQS의 Visibility Timeout을 떠올려보자. Consumer가 처리를 마쳤는데 DeleteMessage 호출 직전에 타임아웃이 지나버리면, 큐는 "이 메시지가 처리됐는지 실패했는지 확신할 수 없으므로" 안전하게 다시 전달한다. 이것이 바로 at-least-once(최소 한 번) 전달의 본질이다. Kafka의 커밋되지 않은 오프셋, RabbitMQ의 미확인 상태에서의 재연결도 전부 같은 이유로 메시지를 다시 준다.

여기서 중요한 인식 전환이 필요하다. "메시지 큐가 정확히 한 번만 전달해주면 좋겠다"는 바람은 대부분의 실무 브로커 조합에서 완전히 충족되지 않는다. 브로커·네트워크 계층의 exactly-once는 매우 제한된 조건(예: Kafka의 트랜잭션 프로듀서 + 같은 Kafka 클러스터 내부 read-process-write)에서만 가능하고, 외부 API 호출이나 DB 쓰기가 섞이는 순간 진짜 보장은 사라진다. 그래서 실무의 표준 전략은 "전달은 at-least-once로 두고, 중복 수신을 Consumer가 멱등하게 흡수한다"는 것이다. 멱등성은 큐가 대신해줄 수 없고, 애플리케이션이 직접 설계해야 하는 책임이다.

핵심 직관: "정확히 한 번 전달"을 브로커에게 기대하지 말고, "여러 번 전달되어도 한 번 처리한 것과 같은 결과가 되도록" Consumer를 설계한다 — 이것이 멱등성 설계의 출발점이다.
왜 면접에서 묻는가

이 질문이 평가하는 것

"멱등성이 뭔가요?"는 용어를 아는지 확인하는 질문이지만, "결제 Consumer를 멱등하게 설계해보라"는 중복 실행이 실제 돈이나 재고에 영향을 미치는 상황에서, 정확한 지점에 정확한 방어를 배치할 수 있는가를 본다.

  • at-least-once가 시스템 전반에 미치는 영향을 이해하는가: 메시지 큐만이 아니라 API 재시도, 배치 재실행, 사용자의 새로고침 더블클릭까지 모두 같은 "중복 요청" 문제의 인스턴스라는 걸 아는지 본다.
  • 동시성(race condition)까지 고려하는가: 같은 키를 가진 요청이 "순차적으로 두 번" 오는 경우와 "동시에 두 인스턴스에서" 오는 경우는 방어 방법이 다르다. 후자를 놓치면 멱등 설계가 이론에만 머문다.
  • 어디에 멱등성을 둘지 계층을 구분하는가: DB 유니크 제약(가장 강력, 가장 좁음), 애플리케이션 상태 머신(비즈니스 규칙 기반), 캐시/락 기반 체크(빠르지만 만료 리스크) 중 상황에 맞는 조합을 고를 수 있는지 본다.
  • 멱등키 자체의 신뢰성 문제를 아는가: 멱등키를 언제 발급하고, 어디에 저장하고, TTL을 얼마나 두는지에 따라 "멱등 처리인 줄 알았는데 사실은 구멍이 있는" 설계가 나올 수 있다는 걸 인지하는지가 시니어와 주니어를 가른다.
실무 연결

멱등성을 어디에 둘 것인가 — 세 가지 방어선과 트레이드오프

실무에서 멱등성은 보통 세 가지 방식 중 하나 또는 조합으로 구현된다. 셋 다 "같은 작업을 두 번 하지 않는다"는 목표는 같지만, 얼마나 강하게 보장되는지, 어떤 대가를 치르는지가 다르다.

방식 동작 원리 강점 약점
DB Unique Constraint 멱등키(예: 주문 ID + 이벤트 타입)를 유니크 컬럼으로 만든 처리 이력 테이블에 INSERT. 두 번째 INSERT는 제약 위반으로 실패 DB 트랜잭션이 원자성을 보장 — 두 요청이 동시에 와도 하나만 성공. 가장 신뢰도가 높음 DB 쓰기가 반드시 필요. 처리 결과가 DB 밖(외부 API 호출 등)에 있으면 별도 조율 필요
Redis SETNX / 분산 락 멱등키로 SET key value NX EX ttl 실행. 이미 존재하면 처리 스킵 빠르고 DB 트랜잭션 없이도 원자적 체크-후-설정 가능 TTL이 만료되면 "처리 완료" 기록이 사라져 재처리 위험. Redis 장애 시 방어선 자체가 사라짐
상태 머신(State Machine) 전이 가드 "주문 상태가 PENDING일 때만 PAID로 전이 허용"처럼 비즈니스 상태 전이 자체를 가드로 사용 멱등키를 별도로 관리할 필요 없이 기존 도메인 상태로 자연스럽게 방어됨 상태가 세분화되지 않은 도메인에는 적용하기 어렵고, 상태 전이 조건 설계가 곧 멱등성 설계가 되므로 도메인 모델링 부담이 큼

결제 승인처럼 돈이 걸린 처리는 DB Unique Constraint를 1차 방어선으로 쓰는 것이 정석이다. 멱등키(보통 "주문 ID" 또는 "메시지 ID + 이벤트 타입"의 조합)를 유니크 컬럼으로 둔 processed_events 같은 이력 테이블을 만들고, 실제 비즈니스 로직(결제 승인 호출, 잔액 차감)과 이 INSERT를 같은 트랜잭션으로 묶는다. INSERT가 유니크 제약으로 실패하면 "이미 처리된 이벤트"로 판단하고 아무 것도 하지 않은 채 정상 응답(ACK)만 돌려준다. 

BEGIN;
  -- 1. 멱등키를 먼저 선점 (이미 있으면 여기서 실패)
  INSERT INTO processed_events (event_id, event_type, processed_at)
  VALUES ('order-8842', 'payment.approved', now());

  -- 2. 실제 비즈니스 로직 (같은 트랜잭션 안에서)
  UPDATE orders SET status = 'PAID' WHERE id = 'order-8842' AND status = 'PENDING';
COMMIT;
-- INSERT가 유니크 제약 위반이면 전체 트랜잭션이 롤백되고,
-- Consumer는 "이미 처리됨"으로 판단해 메시지를 정상 ACK 처리한다.
흔한 오해: "먼저 SELECT로 이미 처리됐는지 확인하고, 없으면 처리한다." 이 순서(check-then-act)는 두 Consumer가 동시에 같은 메시지를 처리하는 순간 무너진다. 두 스레드가 모두 SELECT에서 "없음"을 확인한 뒤 동시에 처리를 진행해버리기 때문이다. 반드시 DB의 유니크 제약이나 원자적 연산(SETNX)처럼 "동시에 하나만 성공하는" 메커니즘을 써야 하고, 애플리케이션 레벨의 if-then 체크만으로는 동시성 상황에서 멱등성이 깨진다.
핵심 개념

멱등키(Idempotency Key)는 무엇을 기준으로 만들어야 하는가

멱등키 설계에서 가장 많이 실수하는 지점은 "무엇이 같으면 같은 요청인가"를 잘못 정의하는 것이다. 메시지 큐의 메시지 ID(예: SQS의 MessageId, Kafka의 offset)를 그대로 멱등키로 쓰면 위험할 수 있다.

  1. 큐가 발급한 메시지 ID를 그대로 쓰면 안 되는 이유
    Producer가 네트워크 오류로 같은 결제 승인 이벤트를 새 메시지로 두 번 발행할 수도 있다. 이 경우 큐 입장에서는 완전히 별개의 메시지(서로 다른 MessageId)이므로, 큐가 주는 메시지 ID 기준 중복 제거로는 이 케이스를 못 잡는다. 반드시 비즈니스 의미가 담긴 키(주문 ID, 결제 요청 ID처럼 Producer가 최초 요청 시점에 발급하는 식별자)를 멱등키로 써야 한다.
  2. 이벤트 타입까지 키에 포함해야 하는 경우
    같은 주문 ID로 "결제 승인", "결제 취소" 이벤트가 각각 한 번씩 발생하는 정상 흐름이 있을 수 있다. 이 경우 order_id만 키로 쓰면 서로 다른 정상 이벤트를 중복으로 오판하므로, event_id 또는 (order_id, event_type) 조합을 키로 잡아야 한다.
  3. 멱등키의 유효 기간을 어떻게 정할 것인가
    이력을 영구 보관할지, 일정 기간 후 삭제할지는 트레이드오프다. 결제처럼 감사(audit) 요구가 있는 도메인은 보통 영구 보관하거나 최소 법적 보관 기간을 지킨다. 반면 단순 알림 발송 중복 방지 같은 경우는 TTL을 짧게 둬도 무방하다. 짧은 TTL은 저장 비용을 줄이지만, "TTL이 지난 뒤 같은 메시지가 다시 오면 새 요청으로 처리된다"는 리스크를 감수하는 것이다.
면접 팁: "멱등키로 메시지 ID를 씁니다"에서 멈추지 말고, "메시지 ID는 브로커가 재전송할 때만 동일하고, Producer가 애초에 이벤트를 중복 생성하는 경우는 못 잡기 때문에 비즈니스 식별자를 키로 씁니다"까지 말하면 멱등성이 "브로커 문제"가 아니라 "end-to-end 설계 문제"라는 걸 이해하고 있다는 신호를 준다.
핵심 개념

내가 멱등해도 외부 시스템이 멱등하지 않다면?

질문 상황을 다시 보자. 결제 대행사 API를 호출한 직후 네트워크 타임아웃이 났다. 이때 가장 어려운 부분은 우리 시스템이 아니라 외부 결제 대행사가 이미 승인 처리를 해버렸는지 알 수 없다는 것이다. 우리 쪽 DB에 멱등키를 걸어도, 외부 API 자체를 다시 호출하면 대행사 쪽에서 중복 승인이 날 수 있다.

이 문제의 표준 해법은 외부 API 호출에도 멱등키를 전달하는 것이다. 대부분의 결제 대행사 API(Stripe, 국내 PG사 등)는 Idempotency-Key 헤더를 지원한다. 같은 키로 같은 API를 다시 호출하면, 대행사 서버가 "이미 처리한 요청"으로 인식하고 새로 승인하지 않은 채 최초 호출과 동일한 응답을 그대로 돌려준다. 우리 시스템의 멱등키(주문 ID 등)를 그대로 이 헤더에 실어 보내면, "우리 쪽 중복 방지"와 "대행사 쪽 중복 방지"가 같은 키로 연결된다.

흔한 오해: "우리 DB에 유니크 제약을 걸어뒀으니 안전하다." DB 유니크 제약은 "같은 요청을 다시 처리하려는 시도"를 막아줄 뿐, 이미 외부 호출이 나가버린 뒤 응답을 못 받은 상황(타임아웃)에서 "그 호출이 실제로 성공했는지"는 알려주지 않는다. 이런 경우엔 (1) 외부 API가 멱등키를 지원하면 그대로 재호출해서 안전하게 결과를 확인하고, (2) 지원하지 않는다면 대행사의 조회 API로 "이 주문의 승인 상태"를 먼저 질의(reconciliation)한 뒤에만 재승인 여부를 결정해야 한다. "내부 멱등성"과 "외부 시스템과의 멱등성"은 별개의 문제다.
답변 사고 순서

면접에서 이 질문을 받았을 때 생각하는 순서

  1. 왜 중복이 생기는지 원인을 먼저 짚는다
    "메시지 큐는 at-least-once 전달이 기본이라, Consumer가 처리를 마쳤어도 ACK 직전에 장애가 나면 브로커는 안전을 위해 재전달합니다. 그래서 Consumer는 처음부터 여러 번 실행되는 것을 전제로 설계해야 합니다."
  2. 멱등키를 정의한다
    "결제 이벤트의 비즈니스 식별자(주문 ID + 이벤트 타입)를 멱등키로 사용합니다. 큐가 주는 메시지 ID는 Producer 중복 발행을 못 잡기 때문에 쓰지 않습니다."
  3. 동시성까지 견디는 원자적 방어를 배치한다
    "멱등키를 유니크 제약이 걸린 처리 이력 테이블에 INSERT하고, 실제 상태 변경과 같은 트랜잭션으로 묶습니다. 두 Consumer가 동시에 들어와도 하나만 성공하도록 DB가 원자성을 보장하게 합니다."
  4. 외부 API 경계까지 고려한다
    "결제 대행사 호출 자체도 Idempotency-Key 헤더로 같은 키를 전달해서, 타임아웃 이후 재시도가 대행사 쪽에서 중복 승인을 만들지 않도록 합니다."
  5. 키를 알 수 없거나 대행사가 멱등키를 지원하지 않는 경우의 대안까지 언급한다
    "만약 외부 API가 멱등키를 지원하지 않는다면, 재시도 전에 먼저 조회 API로 실제 승인 상태를 확인하는 reconciliation 절차를 거친 뒤에만 재승인을 시도합니다."
핵심 답변 예시

실제 면접 답변 예시

"메시지 큐는 at-least-once 전달이 기본이기 때문에, 이 Consumer는 같은 메시지를 여러 번 받을 수 있다는 전제로 설계합니다. 먼저 주문 ID와 이벤트 타입을 조합한 멱등키를 정의하고, 이 키를 유니크 제약이 걸린 처리 이력 테이블에 INSERT하는 작업을 실제 결제 상태 변경과 같은 DB 트랜잭션으로 묶습니다. 이렇게 하면 같은 메시지가 순차적으로 재전달되거나, 두 Consumer 인스턴스가 동시에 같은 메시지를 처리하려는 경우에도 DB의 원자성 덕분에 실제 비즈니스 로직은 정확히 한 번만 실행됩니다. 두 번째 시도는 유니크 제약 위반으로 실패하고, Consumer는 이를 '이미 처리됨'으로 해석해 그냥 정상 ACK만 보냅니다.

결제 대행사 API 호출 자체도 신경 써야 합니다. 우리 쪽에서 멱등하게 막아도, 타임아웃 이후 같은 요청을 대행사에 다시 보내면 대행사 쪽에서 중복 승인이 날 수 있기 때문입니다. 그래서 같은 멱등키를 결제 대행사 API의 Idempotency-Key 헤더로도 전달해서, 대행사가 같은 키의 재요청을 최초 응답 그대로 돌려주도록 합니다. 만약 대행사가 이 기능을 지원하지 않는다면, 재시도 전에 먼저 대행사의 조회 API로 실제 승인 여부를 확인한 뒤에만 재승인을 시도하는 reconciliation 절차를 추가합니다."

Senior-Level Follow-up

면접관이 이어서 물어볼 수 있는 질문들

Q. "DB 트랜잭션 없이, 즉 외부 API 호출만 있고 자체 DB 쓰기가 없는 Consumer는 어떻게 멱등하게 만듭니까?"
이 경우 Redis 같은 외부 저장소에 원자적 연산(SET key value NX EX ttl)으로 "처리 중/처리 완료" 마커를 거는 방식을 씁니다. 다만 이 방식은 Redis 장애나 TTL 만료 시 방어선이 사라지는 리스크가 있으므로, 가능하면 그 API 호출 결과를 최소한의 자체 저장소 (경량 로그 테이블이라도)에 기록해 두 번째 방어선을 만드는 것을 권장합니다. 완전히 상태를 가지지 않는 Consumer라면, 최소한 외부 API 자체가 멱등키를 지원하는지 먼저 확인하고 이를 1차 방어선으로 삼습니다.
Q. "멱등키 이력 테이블이 무한정 커지는 문제는 어떻게 관리합니까?"
도메인 요구사항에 따라 다릅니다. 결제처럼 감사·분쟁 대응이 필요한 도메인은 별도 아카이빙 정책(예: N개월 후 콜드 스토리지로 이관)으로 대응하고, 단순 중복 방지 목적이라면 처리 후 일정 기간이 지나면 삭제하는 배치를 둡니다. 삭제 기준을 정할 때 핵심은 "브로커가 같은 메시지를 재전달할 수 있는 최대 지연 시간"보다 보관 기간이 길어야 한다는 것입니다 — 그보다 짧으면 뒤늦게 재전달된 메시지가 이미 이력이 삭제된 상태라 중복으로 처리될 수 있습니다.
Q. "Kafka 컨슈머 그룹에서 리밸런싱이 일어나는 동안에도 멱등성이 깨질 수 있습니까?"
네, 그리고 이건 실무에서 자주 놓치는 부분입니다. Day 2에서 다룬 것처럼 리밸런싱 시 오프셋 커밋 시점과 파티션 재할당 시점이 어긋나면, 커밋되지 않은 오프셋 구간의 메시지를 새로 할당받은 컨슈머가 다시 처리하게 됩니다. 이는 정상적인 at-least-once 동작이고, 결국 DB 유니크 제약 같은 멱등 방어가 여기서도 동일하게 작동해야 안전합니다. 즉 멱등성 설계는 "정상 재시도"뿐 아니라 "리밸런싱, 배포, 오토스케일링으로 인한 재시도"까지 모두 포괄해야 합니다.
Q. "멱등성 설계가 성능에 주는 비용은 무엇이고, 언제 생략해도 됩니까?"
멱등키 이력 테이블에 매번 INSERT하고 트랜잭션을 거는 것 자체가 추가 쓰기 부하이자 지연입니다. 따라서 결제, 재고 차감, 포인트 적립처럼 중복 실행이 실제 손실로 이어지는 연산에는 반드시 적용하지만, 단순 알림 발송이나 캐시 갱신처럼 중복돼도 사용자 경험만 약간 나빠질 뿐 비즈니스 정합성에 영향이 없는 연산은 멱등 처리를 생략하거나 훨씬 가벼운 방식(TTL 짧은 캐시 체크)만 적용해도 됩니다. "모든 Consumer를 무조건 멱등하게 만들어야 한다"가 아니라, 중복의 대가를 계산해서 방어 강도를 조절하는 것이 시니어의 판단입니다.
답변 품질 비교

좋은 답변과 약한 답변의 차이

강한 답변 — 이렇게 말하면 시니어로 보인다

"at-least-once 전제 아래, 비즈니스 식별자 기반 멱등키를 DB 유니크 제약과 같은 트랜잭션으로 묶어 동시성까지 원자적으로 방어하고, 외부 API 호출에도 같은 키를 전달해 대행사 쪽 중복까지 막습니다."

→ 왜 중복이 생기는지 원인 → 멱등키 정의 기준 → 동시성 방어 → 외부 시스템 경계까지 하나의 논리로 연결

약한 답변 — 이렇게 말하면 주니어로 보인다

"이미 처리된 건지 확인하고, 처리된 거면 스킵합니다."

→ "확인 후 처리"가 check-then-act 패턴이라 동시 요청 상황에서 원자성이 깨진다는 걸 인지하지 못함. 무엇을 확인 키로 쓸지, 어디서 원자성을 보장할지가 빠져 있음.

약한 답변 — 외부 시스템 경계를 놓침

"저희 DB에 중복 방지 로직을 넣어뒀으니 문제 없습니다."

→ 내부 멱등성만으로는 이미 나가버린 외부 API 호출(결제 대행사 승인 등)의 중복 실행 위험을 막지 못한다는 것을 놓침.

자기 점검 Rubric

내 답변을 이 기준으로 점검해보자

중복의 원인 설명: 메시지 큐의 at-least-once 전달 특성 때문에 Consumer가 여러 번 실행될 수 있다는 것을 구체적으로(타임아웃, 리밸런싱 등) 설명했는가?
멱등키 정의 기준: 브로커의 메시지 ID가 아니라 비즈니스 식별자를 멱등키로 써야 하는 이유를 설명했는가?
동시성(race condition) 방어: check-then-act가 아니라 DB 유니크 제약·원자적 연산 같은 방식으로 동시 요청까지 방어했는가?
외부 시스템 경계 인식: 내부 DB 멱등성과 외부 API(결제 대행사 등) 호출의 멱등성이 별개 문제라는 것을 구분해서 설명했는가?
비용/적용 범위 판단: 모든 Consumer에 무조건 멱등성을 적용하는 대신, 중복의 실제 피해 규모에 따라 방어 강도를 조절해야 한다는 것을 언급했는가?
오늘 배운 것 요약

Day 5 핵심 정리

  • 메시지 큐는 기본적으로 at-least-once 전달이며, exactly-once는 매우 제한된 조건에서만 성립하므로 Consumer가 스스로 멱등하게 설계되어야 한다.
  • 멱등키는 브로커의 메시지 ID가 아니라 비즈니스 식별자(주문 ID + 이벤트 타입 등)로 정의해야 Producer의 중복 발행까지 방어할 수 있다.
  • check-then-act 방식은 동시 요청 상황에서 원자성이 깨지므로, DB 유니크 제약이나 원자적 연산(SETNX)처럼 "동시에 하나만 성공하는" 메커니즘을 써야 한다.
  • 내부 시스템의 멱등성과 외부 API 호출의 멱등성은 별개 문제이며, 외부 API가 멱등키를 지원한다면 같은 키를 전달하고, 지원하지 않는다면 재시도 전 상태 조회(reconciliation)로 확인해야 한다.
  • 멱등성 적용은 공짜가 아니므로, 중복 실행의 실제 피해 규모에 따라 방어 강도를 조절하는 것이 실무적인 판단이다.
