오늘의 면접 질문
"주문 이벤트를 처리하는 Kafka Consumer가 있습니다. 평소에는 초당 200건 정도지만, 프로모션이 시작되면 초당 5,000건까지 트래픽이 튑니다. Consumer는 이벤트 하나당 재고 확인, DB 갱신, 알림 발송까지 처리하는데 이 처리 속도가 유입 속도를 따라가지 못해 컨슈머 랙(lag)이 계속 쌓이다가, 결국 컨슈머 프로세스가 메모리 부족으로 죽는 사고가 발생했습니다. 왜 이런 일이 일어났고, Backpressure를 어떻게 설계해서 이 상황을 방지하시겠습니까? Consumer 개수를 늘리는 것 외에 어떤 제어 수단이 있는지도 설명해주세요."
전제 개념

Backpressure란 무엇이고, 왜 컨슈머가 죽었는가

Backpressure(배압)는 원래 유체역학에서 온 용어로, 파이프에 유량을 흘려보내는 속도보다 배출하는 속도가 느릴 때 압력이 역류해서 쌓이는 현상을 말한다. 소프트웨어 시스템에서는 생산 속도(producer)가 소비 속도(consumer)보다 빠를 때, 그 차이를 시스템이 어떻게 처리하느냐를 가리키는 개념이다. 질문 속 사고를 단계별로 분해해보자.

  1. Kafka는 pull 모델이라 브로커가 강제로 밀어넣지 않는다
    Kafka Consumer는 브로커에 poll()을 호출해서 자신이 처리할 수 있는 만큼만 가져오는 구조다. 그런데 문제는 poll로 가져온 이후다. 애플리케이션이 한 번에 많은 레코드를 poll해서 내부 큐나 리스트에 쌓아두고 순차 처리한다면, poll 자체는 브로커 부하를 주지 않아도 애플리케이션 메모리 안에 처리되지 않은 메시지가 계속 쌓인다.
  2. 처리 로직이 느린데 유입은 계속된다
    이벤트 하나당 DB 갱신과 알림 발송까지 포함된 처리는 수십~수백 ms가 걸릴 수 있다. 초당 5,000건이 들어오는데 처리량이 초당 500건이라면, 초당 4,500건씩 처리 대기 큐에 쌓인다. 이 상태가 몇 분만 지속돼도 수십만 건이 메모리에 쌓여 OOM으로 이어진다.
  3. "컨슈머 랙"은 이 격차를 보여주는 지표다
    Consumer Lag은 "브로커에 쌓인 마지막 오프셋 - Consumer가 커밋한 오프셋"으로, 유입과 처리 사이의 격차를 숫자로 보여준다. 랙이 계속 늘어난다는 것은 시스템이 구조적으로 트래픽을 따라가지 못하고 있다는 신호이지, 일시적 지연이 아니다.
핵심 직관: Backpressure 설계의 본질은 "느린 소비자를 무시하고 계속 밀어넣을 것인가, 아니면 생산 속도 자체를 늦추거나 처리 능력을 늘릴 것인가"를 명시적으로 결정하는 것이다. 아무 조치도 하지 않으면 시스템은 항상 가장 약한 고리(느린 소비자 쪽 메모리)에서 무너진다.
왜 면접에서 묻는가

이 질문이 평가하는 것

"Consumer를 더 늘리면 됩니다"는 절반의 답이다. 면접관은 왜 컨슈머 수를 늘리는 것만으로는 부족할 수 있는지, 그리고 처리 속도와 유입 속도 사이의 격차를 시스템 레벨에서 어떻게 통제하는지를 본다.

  • 스케일 아웃의 한계를 아는가: Kafka Consumer 수는 파티션 수를 넘을 수 없다(Day 2). 파티션이 10개인데 Consumer를 50개 띄워도 실제로 일하는 건 10개뿐이다. 무한정 늘릴 수 없는 구조적 상한이 있다는 것을 이해하는지 본다.
  • "빠르게 처리"와 "느리게 받기"를 구분하는가: 문제를 해결하는 방향이 처리 속도를 올리는 것(동시성, 배치 처리)뿐 아니라, 애초에 한 번에 너무 많이 가져오지 않도록 유입을 조절하는 것(prefetch, max.poll.records)도 있다는 것을 아는지 본다.
  • 브로커별 Backpressure 메커니즘 차이를 아는가: Kafka의 pull 기반 poll 제어, RabbitMQ의 prefetch(QoS), SQS의 폴링 배치 크기와 Lambda 동시성 제한은 겉보기엔 비슷해 보여도 작동 원리가 다르다. 이 차이를 구체적으로 설명할 수 있는지가 시니어와 주니어를 가른다.
  • 모니터링과 자동 대응을 연결하는가: 랙이 쌓이는 것을 사후에 발견하는 게 아니라, 랙 증가율을 지표로 삼아 오토스케일링이나 알림, 심하면 load shedding(선택적 유입 차단)까지 설계에 포함시키는지 본다.
실무 연결

브로커별 Backpressure 제어 지점 비교

Backpressure를 어디서 거는지는 브로커의 전달 모델(push vs pull)에 따라 다르다. 실무에서 가장 자주 쓰이는 세 가지 브로커의 제어 지점을 비교하면 다음과 같다.

브로커 전달 모델 주요 제어 파라미터 작동 방식
Kafka Pull (Consumer가 poll) max.poll.records, fetch.max.bytes, max.poll.interval.ms Consumer가 한 번의 poll에서 가져올 최대 레코드 수를 제한해, 애플리케이션 메모리에 쌓이는 미처리 메시지 양을 스스로 조절한다. 처리 시간이 max.poll.interval.ms를 넘기면 Consumer가 죽은 것으로 간주되어 그룹에서 제외되고 리밸런싱이 발생한다.
RabbitMQ Push (브로커가 밀어넣음) Prefetch Count (QoS) 브로커는 Consumer가 ack하지 않은 메시지 수가 prefetch 값에 도달하면 그 이상 밀어넣지 않고 멈춘다. 즉 Consumer가 처리를 못 따라가면 브로커 쪽에서 자동으로 유입을 멈추는 내장 Backpressure가 있다는 점이 Kafka와 큰 차이다.
AWS SQS Pull (폴링) + 이벤트 소스 매핑 폴링 MaxNumberOfMessages, Lambda ReservedConcurrency, Batch Size 직접 폴링이면 한 번에 가져오는 메시지 수(최대 10개)로 조절하고, Lambda 트리거라면 동시 실행 개수 상한(concurrency)으로 다운스트림(DB 등)에 가는 동시 요청 수 자체를 제한한다. 큐 자체는 사실상 무한 버퍼 역할을 하므로 압력이 소비자 쪽으로 전파되지 않고 큐 안에 쌓인다.
흔한 오해: "SQS는 관리형이니 Backpressure를 신경 쓸 필요가 없다." SQS 큐 자체는 대량의 메시지를 버퍼링해줄 수 있지만, 그 뒤에 있는 다운스트림 리소스(DB 커넥션 풀, 외부 API rate limit)는 무한하지 않다. 폴링 주기나 Lambda 동시성을 제한하지 않으면 큐가 대신 막아주지 않고, 그대로 다운스트림이 압도된다.

Consumer 수를 늘리는 것 외의 제어 수단

"Consumer를 늘린다"는 스케일 아웃 한 가지 방향일 뿐이다. 실제로는 여러 층위에서 처리량과 유입량의 균형을 맞추는 수단들이 있고, 시니어 답변은 이들을 조합해서 설명한다.

  1. Prefetch / poll 배치 크기 축소
    Kafka는 max.poll.records를 낮추고, RabbitMQ는 prefetch count를 낮춰서 Consumer가 한 번에 떠안는 미처리 메시지 양 자체를 줄인다. 처리 속도를 못 올려도, 최소한 메모리에 쌓이는 양을 통제해서 OOM은 막을 수 있다. 대신 처리량(throughput)은 다소 낮아질 수 있어 순수 성능과의 트레이드오프다.
  2. 병렬 처리(동시성)로 처리량 자체를 높인다
    하나의 Consumer 안에서도 스레드 풀이나 비동기 처리로 여러 메시지를 동시에 처리하면, Consumer 프로세스 수를 늘리지 않고도 처리량을 올릴 수 있다. 단, 이 경우 메시지 순서 보장이 깨질 수 있다는 점(Day 2에서 다룬 파티션 내 순서 보장)을 함께 고려해야 한다.
  3. 파티션/Consumer 수를 함께 늘린다(수평 확장)
    Kafka라면 파티션 수를 늘려야 Consumer도 그만큼 늘릴 수 있다(Day 2). 파티션을 늘리지 않고 Consumer만 늘리면 초과분은 유휴 상태로 남는다. 파티션 재설계는 사전에 계획해야 하는 일이라, 프로모션처럼 예측 가능한 트래픽 급증은 사전에 파티션 수와 Consumer 수를 함께 스케일링해야 한다.
  4. 처리 자체를 가볍게 만든다(비동기화·배치화)
    이벤트마다 동기로 DB 갱신과 알림 발송을 순차 처리하지 말고, 알림 발송처럼 지연을 허용할 수 있는 부분은 별도 큐로 다시 분리해서 핵심 처리(재고 확인, DB 갱신)만 빠르게 끝내는 것도 처리량을 늘리는 방법이다. 또한 DB 갱신을 건별로 하지 않고 여러 건을 모아 배치로 처리하면 처리 단가가 줄어든다.
  5. 유입 자체를 제한한다(Rate Limiting, Load Shedding)
    위 방법으로도 감당이 안 되는 극단적 트래픽이라면, 처리량 자체를 늘리는 대신 유입을 의도적으로 제한한다. API Gateway 단에서 요청을 rate limit하거나, 우선순위가 낮은 이벤트(예: 마케팅성 알림)는 임계 상황에서 잠시 버리거나 저장만 해두고 나중에 재처리하는 load shedding 전략도 대량 트래픽 설계에서는 정당한 선택이다.
면접 팁: "Consumer를 늘리겠습니다"에서 멈추지 말고, "먼저 prefetch/poll 배치를 낮춰 즉시 OOM을 막고, 파티션과 Consumer를 함께 늘려 처리량을 확보하고, 알림처럼 지연 허용 가능한 부분은 별도 큐로 분리하고, 그래도 감당 못 할 트래픽 스파이크는 API 단 rate limiting으로 유입 자체를 제어하겠습니다"처럼 여러 층위를 조합해서 답하면 훨씬 설득력 있다.
핵심 개념

랙을 사후에 발견하지 않으려면 무엇을 모니터링하는가

Backpressure 설계가 완성되려면, 격차가 벌어지는 것을 사고 나기 전에 감지할 수 있어야 한다. 핵심 지표와 대응 방식은 다음과 같다.

  • Consumer Lag(절대값과 증가율): 랙의 절대값보다 증가율(초당 얼마씩 늘고 있는가)이 더 중요한 신호다. 랙이 크더라도 줄어들고 있으면 회복 중이지만, 작더라도 계속 늘고 있으면 구조적으로 처리 능력이 부족하다는 뜻이다.
  • 처리 지연시간(end-to-end latency): 메시지가 발행된 시점부터 처리 완료까지 걸린 시간을 추적하면, "얼마나 최신 데이터가 아닌 상태로 처리되고 있는가(staleness)"를 비즈니스 관점에서 파악할 수 있다.
  • Consumer 자원 사용률(CPU, 메모리, 처리 중 큐 크기): 애플리케이션 내부에서 poll한 뒤 처리 대기 중인 메시지 수를 직접 노출시켜, 브로커 지표만으로 보이지 않는 애플리케이션 내부 병목까지 관측한다.
  • 자동 대응 — Lag 기반 오토스케일링: KEDA(Kubernetes Event-Driven Autoscaling) 같은 도구는 Consumer Lag을 지표로 삼아 Consumer(Pod) 수를 자동으로 늘리고 줄인다. 프로모션처럼 예측 가능한 트래픽은 사전 스케일링을, 예측 불가능한 스파이크는 랙 기반 오토스케일링을 함께 쓰는 것이 일반적이다.
실무에서 자주 나오는 실패 패턴: 랙이 눈에 띄게 쌓인 뒤에야 대시보드를 보고 대응을 시작하는 경우다. 랙 증가율에 대한 알림 임계값을 미리 설정해두지 않으면, "이미 몇십만 건이 밀린 뒤" 대응하게 되어 회복 자체에도 오랜 시간이 걸린다.
답변 사고 순서

면접에서 이 질문을 받았을 때 생각하는 순서

  1. 사고 원인을 정확히 짚는다
    "Consumer가 poll로 가져온 메시지를 처리하는 속도가 유입 속도를 따라가지 못하면서, 미처리 메시지가 애플리케이션 메모리에 계속 쌓여 결국 OOM으로 이어진 것입니다. 이는 Kafka가 pull 모델이라 브로커가 자동으로 유입을 멈춰주지 않기 때문에, poll 배치 크기를 제한하지 않으면 흔히 발생하는 패턴입니다."
  2. 즉시 조치(단기)와 구조 개선(중장기)을 구분한다
    "가장 먼저 max.poll.records를 낮춰서 한 번에 떠안는 메시지 양을 줄여 OOM 재발을 막습니다. 이는 처리량을 늘리는 게 아니라 안전장치입니다."
  3. 처리량을 실제로 늘리는 방법을 제시한다
    "그다음 파티션 수와 Consumer 수를 함께 늘려 수평 확장하고, 알림 발송처럼 지연을 허용할 수 있는 부분은 별도 큐로 분리해 핵심 처리 경로를 가볍게 만듭니다."
  4. 극단적 트래픽에 대한 방어선을 언급한다
    "프로모션처럼 예측 가능한 트래픽 급증은 사전에 파티션과 Consumer를 스케일링하고, 예측 불가능한 스파이크는 Consumer Lag 기반 오토스케일링과, 필요하면 API 단 rate limiting으로 유입 자체를 제어합니다."
  5. 모니터링으로 재발을 막는 체계를 설명한다
    "Consumer Lag의 절대값이 아니라 증가율에 알림 임계값을 설정해서, 랙이 회복 불가능한 수준으로 쌓이기 전에 미리 감지하고 대응할 수 있도록 합니다."
핵심 답변 예시

실제 면접 답변 예시

"이 사고는 Kafka Consumer가 poll로 가져온 메시지를 처리하는 속도가 유입 속도를 따라가지 못하면서, 미처리 메시지가 애플리케이션 메모리에 무제한으로 쌓여 OOM에 이른 것입니다. Kafka는 pull 모델이라 RabbitMQ의 prefetch처럼 브로커가 자동으로 유입을 멈춰주지 않기 때문에, poll 배치 크기를 애플리케이션이 직접 제한해야 합니다.

먼저 max.poll.records를 낮춰 한 번에 떠안는 메시지 양을 줄여 즉시 재발을 막고, 파티션 수와 Consumer 수를 함께 늘려 처리량 자체를 확보하겠습니다. 알림 발송처럼 지연을 허용할 수 있는 부분은 재고 확인·DB 갱신과 분리해 별도 큐로 넘겨 핵심 처리 경로를 가볍게 만듭니다. 프로모션처럼 예측 가능한 트래픽은 사전에 파티션과 Consumer를 함께 스케일링하고, 예측 불가능한 스파이크에는 Consumer Lag 기반 오토스케일링과 API 단 rate limiting을 방어선으로 둡니다. 마지막으로 랙의 절대값이 아니라 증가율에 알림 임계값을 설정해서, 회복 불가능한 수준으로 쌓이기 전에 미리 감지하도록 모니터링을 구성하겠습니다."

Senior-Level Follow-up

면접관이 이어서 물어볼 수 있는 질문들

Q. "병렬 처리로 처리량을 늘리면 순서 보장이 깨진다고 하셨는데, 주문 이벤트처럼 순서가 중요한 경우엔 어떻게 하시겠습니까?"
순서가 중요한 단위(예: 같은 주문 ID)는 같은 파티션·같은 처리 스레드로 고정되도록 파티션 키를 설계하고(Day 2), 그 파티션 안에서는 순차 처리를 유지합니다. 병렬화는 파티션 간에만 적용해서, 서로 다른 주문 간에는 동시 처리를 허용하되 같은 주문 안의 이벤트 순서는 깨지지 않도록 합니다. 즉 처리량 확장은 "순서가 필요 없는 단위" 사이에서만 일어나야 합니다.
Q. "Load Shedding으로 메시지를 버린다고 하셨는데, 어떤 메시지를 버려도 되는지는 어떻게 판단합니까?"
비즈니스 임팩트를 기준으로 메시지에 우선순위를 부여해야 합니다. 재고 차감이나 결제 확정처럼 정합성이 깨지면 돈 문제로 이어지는 이벤트는 절대 버리지 않고, 오히려 우선 처리 큐로 분리합니다. 반면 마케팅 알림, 추천 로그 수집처럼 유실돼도 비즈니스에 치명적이지 않은 이벤트는 임계 상황에서 샘플링하거나 잠시 버퍼링만 하고 나중에 배치로 재처리하는 방식을 씁니다. "버린다"보다는 "지금 당장 처리하지 않고 나중으로 미룬다"는 관점이 더 정확합니다.
Q. "RabbitMQ의 prefetch는 브로커가 자동으로 유입을 멈춰준다고 하셨는데, 그러면 Kafka보다 더 안전한 겁니까?"
미처리 메시지 폭주를 막는 측면에서는 RabbitMQ의 내장 Backpressure가 더 즉각적입니다. 하지만 대가가 있습니다. RabbitMQ는 큐 안의 메시지가 ack되기 전까지 브로커 메모리/디스크에 머무르며, prefetch로 유입을 멈추면 그만큼 큐 적체가 브로커 쪽에서 발생합니다. Kafka는 토픽이 커밋 로그 구조라 대량의 미처리 메시지가 쌓여도 브로커 자체는 여유롭게 견디고, 압력이 오로지 Consumer 쪽에만 걸립니다. 어느 쪽이 "더 안전한가"가 아니라, 압력이 브로커에 걸리느냐 Consumer 애플리케이션에 걸리느냐의 차이이고, 어느 쪽 자원이 더 확장하기 쉬운지에 따라 설계가 달라집니다.
Q. "Consumer Lag 기반 오토스케일링을 쓴다고 하셨는데, Consumer를 너무 급격히 늘리면 어떤 문제가 생길 수 있습니까?"
두 가지 리스크가 있습니다. 첫째, Kafka는 Consumer 수를 늘리면 그룹 리밸런싱이 발생하고, 리밸런싱 중에는 해당 그룹의 처리가 잠시 멈추므로 오히려 랙이 순간적으로 더 늘어날 수 있습니다. 둘째, Consumer가 늘어난 만큼 다운스트림 DB 커넥션이나 외부 API 호출량도 비례해서 늘어나므로, Consumer 확장이 다운스트림 병목을 새로운 장애 지점으로 옮기는 결과가 될 수 있습니다. 그래서 오토스케일링 상한을 다운스트림 용량에 맞춰 정하고, 스케일링 속도(step)도 완만하게 설정하는 것이 중요합니다.
답변 품질 비교

좋은 답변과 약한 답변의 차이

강한 답변 — 이렇게 말하면 시니어로 보인다

"pull 모델의 특성상 poll 배치 크기 제한이 없어 미처리 메시지가 메모리에 쌓인 것이 근본 원인임을 짚고, 즉시 조치(poll 배치 축소)와 구조 개선(파티션/Consumer 확장, 처리 분리)을 구분해서 제시합니다. 예측 가능한 트래픽과 예측 불가능한 스파이크에 대한 대응을 나누고, 랙 증가율 기반 모니터링으로 재발 방지까지 연결합니다."

→ 근본 원인 → 단기/중장기 대응 구분 → 확장의 구조적 한계(파티션 수) → 모니터링·자동화까지 하나의 흐름으로 연결

약한 답변 — 이렇게 말하면 주니어로 보인다

"Consumer 수를 늘려서 처리 속도를 높이겠습니다."

→ 파티션 수라는 구조적 상한을 언급하지 않고, poll 배치 제어·모니터링·순서 보장과의 트레이드오프가 전혀 없어 스케일 아웃 하나로 문제를 단순화함.

약한 답변 — 원인 진단 없이 대응만 나열

"오토스케일링을 걸어두면 트래픽이 늘어도 자동으로 대응됩니다."

→ 왜 애초에 OOM이 발생했는지(poll 배치 크기 미제어) 진단이 빠져 있고, 오토스케일링의 부작용(리밸런싱, 다운스트림 병목 이동)에 대한 고려도 없음.

근본 원인 진단: pull 모델에서 poll 배치 크기가 제한되지 않으면 미처리 메시지가 애플리케이션 메모리에 쌓인다는 인과관계를 설명했는가?

모니터링·재발 방지: 랙의 절대값이 아니라 증가율을 지표로 삼아 사고 전에 감지하는 체계를 설명했는가?
오늘 배운 것 요약

Day 8 핵심 정리

  • Backpressure는 생산 속도와 소비 속도의 격차를 시스템이 어떻게 처리할지 명시적으로 결정하는 것이다 — 방치하면 가장 약한 고리(느린 소비자의 메모리)에서 무너진다.
  • Kafka(pull)는 max.poll.records 같은 클라이언트 설정으로 유입을 직접 제어해야 하는 반면, RabbitMQ(push)는 prefetch(QoS)로 브로커가 자동으로 유입을 멈춰준다는 구조적 차이가 있다.
  • Consumer 수 확장은 파티션 수라는 상한이 있고, Consumer가 늘어난 만큼 다운스트림 부하도 늘어난다는 트레이드오프가 있다.
  • 처리량 확보 수단은 스케일 아웃 외에도 poll 배치 축소, 병렬 처리, 처리 분리(비동기화), rate limiting/load shedding까지 여러 층위가 있다.
  • 재발 방지는 Consumer Lag 증가율을 지표로 삼아 사고 전에 감지하고, Lag 기반 오토스케일링과 결합해서 대응해야 한다.
