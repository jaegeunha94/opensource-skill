1. 핵심 면접 질문

Primary Question

"스크롤할 때마다 요소의 위치를 읽고 다시 쓰는 코드를 작성했더니 스크롤이 심하게 버벅입니다. 무슨 일이 일어난 것이고, 어떻게 고치시겠어요?"

Variant 1

"width를 바꾸는 애니메이션과 transform: scale()을 쓰는 애니메이션 중 왜 후자가 훨씬 부드러운가요?"

Variant 2

"성능을 개선하겠다고 모든 요소에 will-change: transform을 걸었더니 오히려 스크롤이 느려지고 모바일에서 탭이 죽었습니다. 무슨 일이 일어난 걸까요?"

2. 왜 면접에서 묻는가

이 질문은 "requestAnimationFrame이 뭔지 아는가"가 아니라, 브라우저가 화면 한 프레임을 그리기 위해 실제로 어떤 계산을 반복하는지, 그리고 그 계산 비용이 왜 코드 작성 순서에 따라 몇 배씩 차이나는지를 설명할 수 있는가를 테스트합니다. 대부분의 주니어 개발자는 "느리면 requestAnimationFrame을 쓰면 된다"는 처방을 외우고 있지만, 왜 그게 도움이 되는지는 설명하지 못합니다. 시니어는 처방이 아니라 처방이 통하는 기전(mechanism)을 설명해야 합니다.

면접관이 이 질문으로 평가하는 것:

  • 렌더링 파이프라인을 블랙박스가 아닌 단계별 비용 모델로 이해 — Style → Layout → Paint → Composite 각 단계가 왜 앞 단계 결과에 의존하고, 왜 특정 단계를 건너뛸 수 있는지 아는가.
  • "읽기"와 "쓰기"를 뒤섞는 것이 왜 위험한지 설명 — DOM 속성을 읽는 것 자체가 브라우저에게 강제로 계산을 시키는 부수효과(side effect)가 있다는 것을 아는가.
  • GPU 가속을 만능 해결책이 아니라 트레이드오프로 이해 — 레이어를 늘리면 빨라지는 경우와, 오히려 메모리·합성 비용 때문에 느려지는 경우를 구분할 수 있는가.
  • DevTools Performance 패널로 실제 병목을 증명하는 습관 — "이게 원인일 것 같다"가 아니라 플레임 차트에서 Layout/Paint가 반복되는 구간을 실제로 짚어낼 수 있는가.

주니어~중니어에서 자주 나오는 답변 패턴: "requestAnimationFrame을 쓰면 됩니다" 또는 "transform을 쓰면 빨라집니다"처럼 결론만 외운 답변입니다. 이는 무엇을 쓰라고만 말할 뿐, 왜 그것이 렌더링 파이프라인의 특정 단계를 건너뛰게 만드는지는 설명하지 못합니다. 시니어는 "이 코드가 각 프레임마다 강제로 Layout을 재계산하게 만드는 지점이 어디인지"부터 짚습니다.

3. 전제 개념 — 쉬운 한국어로

렌더링 파이프라인이란 무엇인가

브라우저가 HTML/CSS/JS를 받아서 실제로 화면에 픽셀을 그리기까지 거치는 일련의 계산 단계입니다. 요리에 비유하면 재료 손질(Style 계산) → 레시피대로 배치 정하기(Layout) → 실제로 조리해서 접시에 담기(Paint) → 여러 접시를 하나의 상에 올리기(Composite) 순서와 같습니다. 뒷 단계는 앞 단계의 결과물이 있어야 시작할 수 있고, 앞 단계를 다시 하면 뒷 단계도 전부 다시 해야 합니다.

Layout(레이아웃, 또는 Reflow)이란 무엇인가

모든 요소가 화면에서 정확히 몇 픽셀 크기이고 어디에 위치하는지 계산하는 단계입니다. 한 요소의 크기가 바뀌면 그 부모, 형제, 자식 요소의 위치도 도미노처럼 영향을 받을 수 있기 때문에, Layout은 보통 "이 요소 하나만" 계산하지 못하고 관련된 영역 전체를 다시 계산해야 합니다.

Paint(페인트)란 무엇인가

이미 위치와 크기가 정해진 요소를 실제 픽셀(색상, 그림자, 테두리, 텍스트 모양)로 채우는 단계입니다. 위치는 그대로인데 배경색만 바뀌는 경우처럼, 다시 계산할 필요 없이 다시 "칠하기만" 하면 되는 경우에 발생합니다.

Composite(컴포지트)란 무엇인가

브라우저는 페이지 전체를 하나의 그림으로 그리지 않고, 특정 조건을 만족하는 요소를 투명 필름(레이어)에 따로 그린 뒤 마지막에 GPU가 이 필름들을 겹쳐서 화면에 합성합니다. 애니메이션 슬라이드나 팝업처럼 자주 움직이는 요소를 별도 레이어에 미리 그려두면, 움직일 때마다 Layout/Paint를 다시 할 필요 없이 "필름의 위치만 옮기면" 되므로 훨씬 빠릅니다.

왜 "속성을 읽는 것"이 위험한 행동이 될 수 있는가

JavaScript가 offsetTop, getBoundingClientRect()처럼 요소의 실제 화면 위치를 물어보면, 브라우저는 최신 값을 보장하기 위해 대기 중인 스타일 변경 사항을 즉시 반영해서 Layout을 다시 계산한 뒤 답을 돌려줘야 합니다. 이걸 강제 동기 레이아웃(Forced Synchronous Layout)이라고 부릅니다. 즉, 값을 "읽기만" 했을 뿐인데 브라우저 입장에서는 무거운 계산을 강제로 수행한 것입니다. 은행에 비유하면, 잔고를 "조회만" 했는데 시스템이 조회 시점에 밀린 모든 거래를 강제로 먼저 정산해야 하는 상황과 같습니다.

4. Critical Rendering Path 단계별 비용 모델

한 프레임(보통 60fps 기준 16.6ms)이 그려지는 과정은 다음 순서를 따릅니다. 각 단계는 이전 단계의 결과에 의존하므로, 파이프라인 중간부터 다시 시작할 수는 있어도 뒤에서 앞으로 거슬러 갈 수는 없습니다.

단계 하는 일 다시 계산이 필요해지는 트리거
JavaScript DOM 조작, 상태 변경, 이벤트 핸들러 실행 -
Style CSS 규칙을 계산해 각 요소의 최종 스타일(Computed Style) 결정 클래스 추가/제거, 인라인 스타일 변경 등 매칭되는 CSS 규칙이 바뀔 때
Layout(Reflow) 각 요소의 크기와 화면 좌표를 계산 — 이 단계가 가장 비쌉니다 width, height, top, left, font-size 등 기하학적(geometric) 속성 변경
Paint 계산된 위치에 실제 픽셀(색상, 그림자, 텍스트)을 그림 Layout이 다시 일어난 경우, 또는 background-color, box-shadow 등 위치와 무관한 시각 속성 변경
Composite 여러 레이어를 GPU가 겹쳐서 최종 화면을 합성 Paint가 다시 일어난 경우, 또는 transform, opacity처럼 레이어 자체만 변형하면 되는 속성 변경

왜 이 순서가 "폭포수"인가

위쪽 단계에서 다시 계산이 발생하면 그 아래 모든 단계도 함께 다시 실행되어야 합니다. width를 바꾸면 Layout → Paint → Composite가 전부 다시 일어나지만, transform만 바꾸면 Composite만 다시 일어나면 됩니다. 이것이 transform/opacity 애니메이션이 width/top 애니메이션보다 훨씬 부드러운 이유의 전부입니다 — 신비한 최적화가 아니라 파이프라인에서 건너뛰는 단계의 개수 차이입니다.

5. Reflow와 Repaint — 무엇이 다르고 무엇이 트리거하는가

둘 다 "다시 그린다"는 점에서 비슷해 보이지만 비용 차이가 큽니다. Reflow(Layout)는 "어디에, 얼마나 크게"를 다시 정하는 것이고, Repaint(Paint)는 "이미 정해진 자리에 무슨 색으로"를 다시 칠하는 것입니다.

속성 카테고리 예시 결과
기하학적 속성 width, height, top/left(position 기반), margin, padding, border-width, font-size Reflow + Repaint + Composite 전부 발생
시각적 속성(위치 무관) background-color, color, box-shadow, border-color, visibility Repaint + Composite만 발생 (Layout 생략)
Compositor 전용 속성 transform, opacity, filter(일부) Composite만 발생 (Layout, Paint 모두 생략)

실무에서 이게 왜 중요한가

"카드가 호버되면 살짝 커진다"는 흔한 UI 요구사항을 width/height로 구현하면 매 프레임 Layout이 재계산되어 요소 수가 많은 리스트에서는 눈에 띄게 버벅입니다. 같은 효과를 transform: scale()로 구현하면 Layout·Paint를 완전히 건너뛰고 GPU가 이미 그려둔 레이어의 크기만 조정하므로 요소 수와 거의 무관하게 부드럽습니다. 이 차이를 아는 것이 "왜 시니어는 애니메이션에 transform을 기본으로 선택하는가"의 답입니다.

6. Layout Thrashing이 스크롤을 버벅이게 만드는 과정

Layout Thrashing(레이아웃 스래싱)은 읽기와 쓰기가 반복적으로 교차할 때 발생하는, 실무에서 가장 흔한 렌더링 성능 버그입니다. 발생 과정을 단계별로 보면 다음과 같습니다.

  • 1
    스타일을 쓴다(write)

    스크롤 이벤트 핸들러 안에서 어떤 요소의 style.top을 변경합니다. 브라우저는 이 변경을 즉시 반영하지 않고 "대기 중(dirty)" 상태로 표시만 해둔 채, 다음 프레임을 그릴 때 한꺼번에 처리하려고 미룹니다(batch).

  • 2
    같은 루프 안에서 위치를 읽는다(read)

    바로 다음 줄에서 다른 요소의 offsetTop이나 getBoundingClientRect()를 호출합니다. 브라우저는 "지금 대기 중인 변경 사항을 반영하지 않으면 최신 값을 줄 수 없다"고 판단하고, 미뤄뒀던 Layout을 프레임 주기와 무관하게 그 자리에서 즉시 실행합니다. 이것이 강제 동기 레이아웃입니다.

  • 3
    읽은 값으로 다시 쓴다

    방금 읽은 값을 바탕으로 또 다른 스타일을 변경합니다. 다시 "대기 중" 상태가 됩니다.

  • 4
    루프의 다음 반복에서 다시 읽는다

    리스트의 다음 요소를 처리하며 또 위치를 읽습니다. 앞서 쓴 변경 사항 때문에 브라우저는 또 강제로 Layout을 재계산합니다.

  • 5
    요소 수만큼 Layout이 반복된다

    원래는 프레임당 한 번이면 충분한 Layout 계산이, 쓰기·읽기가 교차하는 루프 구조 때문에 요소 개수(N)만큼 강제로 반복됩니다. 100개 요소를 순회하면 Layout이 100번 강제 실행되는 셈이고, 이것이 "스크롤할 때마다 버벅인다"는 증상의 정체입니다.

Layout Thrashing을 유발하는 대표적인 속성/메서드

  • 읽기(강제 Layout 트리거): offsetTop/Left/Width/Height, clientTop/Left/Width/Height, scrollTop/Left/Width/Height, getBoundingClientRect(), getComputedStyle()
  • 쓰기(다음 읽기를 강제 계산으로 만듦): style.* 직접 대입, classList.add/remove, DOM 삽입/삭제

해결 패턴 — 읽기와 쓰기를 분리(batch)

// 나쁜 예: 읽기-쓰기가 매 반복마다 교차됨 (N번 강제 Layout)
items.forEach(item => {
  const top = item.offsetTop;       // 읽기 → 강제 Layout
  item.style.top = (top + delta) + 'px'; // 쓰기 → 다음 읽기가 다시 강제 Layout
});

// 좋은 예: 모든 읽기를 먼저 끝내고, 쓰기는 나중에 한 번에 (Layout 1번)
const tops = items.map(item => item.offsetTop); // 읽기만 모아서 수행
items.forEach((item, i) => {
  item.style.top = (tops[i] + delta) + 'px';    // 쓰기는 이후 한 번에 처리
});

여기에 requestAnimationFrame으로 쓰기 작업을 다음 프레임 시작 시점에 몰아주면, 브라우저가 원래 하려던 배치(batch) 방식과 코드의 실행 방식이 일치하게 되어 프레임당 Layout이 정확히 한 번만 발생합니다. 스크롤·리사이즈처럼 짧은 시간에 여러 번 발생하는 이벤트는 requestAnimationFrame으로 실행 횟수를 프레임 수만큼으로 묶거나(throttle), ResizeObserver/IntersectionObserver처럼 브라우저가 이미 계산해둔 값을 비동기로 전달받는 API로 대체하는 것이 근본적인 해결책입니다.

7. Compositor와 GPU 레이어 승격

transform/opacity 애니메이션이 빠른 이유는 해당 요소가 별도의 Compositor 레이어로 승격되어, 메인 스레드의 Layout/Paint를 거치지 않고 GPU가 직접 레이어의 위치·투명도만 바꾸기 때문입니다. will-change: transform이나 transform: translateZ(0) 같은 힌트는 브라우저에게 "이 요소를 미리 별도 레이어로 만들어 두라"고 요청하는 것입니다.

레이어 승격이 공짜가 아닌 이유

  • 메모리 비용 — 레이어마다 별도의 GPU 텍스처(비트맵)를 할당해야 합니다. 요소 수백 개에 무분별하게 will-change를 걸면 메모리 사용량이 급증하고, 특히 메모리가 제한적인 모바일 기기에서는 탭이 강제 종료(OOM kill)될 수 있습니다.
  • 합성(Composite) 비용 증가 — 레이어 수가 늘어날수록 GPU가 매 프레임 겹쳐야 할 레이어 수도 늘어나 합성 자체의 비용이 커집니다. 레이어를 너무 잘게 쪼개면 오히려 전체 프레임 시간이 늘어날 수 있습니다.
  • 레이어 폭발(layer explosion) — 승격된 레이어 위에 있는 다른 요소들도 겹침 순서(stacking context)를 지키기 위해 의도치 않게 함께 별도 레이어로 승격되는 경우가 있어, 예상보다 훨씬 많은 레이어가 생성될 수 있습니다.

실무 판단 기준

레이어 승격은 "자주, 짧게, 예측 가능하게 움직이는 요소"(모달, 캐러셀, 드래그 중인 카드)에는 명확한 이득이 있지만, "화면에 있는 모든 요소"에 습관적으로 적용하는 것은 안티패턴입니다. Chrome DevTools의 Layers 패널로 실제 생성된 레이어 수와 각 레이어의 메모리 사용량을 확인한 뒤, 애니메이션이 끝나면 will-change를 제거해서 불필요한 레이어를 해제하는 것이 시니어가 실제로 하는 작업입니다.

8. 답변 사고 순서

  1. 증상을 파이프라인 단계로 재정의합니다. "버벅인다"는 결과이므로, 어느 단계(Layout? Paint? Composite?)가 매 프레임 반복되는지부터 특정합니다.
  2. 코드에서 읽기/쓰기 교차 지점을 찾습니다. 스크롤 핸들러 안에서 offsetTop 같은 읽기 연산과 style.* 쓰기 연산이 섞여 있는지 확인합니다.
  3. DevTools Performance 패널로 증명합니다. "Layout" 항목이 프레임마다 보라색 막대로 반복되는지, "Forced reflow" 경고가 있는지 실제로 확인해서 추측이 아닌 근거로 설명합니다.
  4. 구조적 해결책을 제시합니다. 읽기와 쓰기를 분리(batch)하고, 실행 빈도를 requestAnimationFrame으로 프레임에 맞추며, 가능하면 transform처럼 Compositor 전용 속성으로 대체합니다.
  5. 트레이드오프를 언급합니다. 레이어 승격이 항상 이득은 아니며, 메모리·합성 비용이라는 반대급부가 있다는 것까지 짚어 "무조건 GPU를 쓰면 빠르다"는 인상을 주지 않습니다.

9. 핵심 답변 예시

면접관: "스크롤할 때마다 요소의 위치를 읽고 다시 쓰는 코드를 작성했더니 스크롤이 심하게 버벅입니다. 무슨 일이 일어난 것이고, 어떻게 고치시겠어요?"

STRONG ANSWER

가장 먼저 의심하는 것은 Layout Thrashing입니다. 브라우저는 스타일 변경을 즉시 반영하지 않고 다음 프레임 시작 시점에 한꺼번에 처리하려고 미루는데, 그 사이에 offsetTop이나 getBoundingClientRect()처럼 요소의 실제 위치를 읽는 코드가 실행되면, 브라우저는 정확한 값을 주기 위해 미뤄뒀던 Layout을 그 자리에서 즉시 강제로 계산합니다. 스크롤 핸들러 안에서 "읽고 → 쓰고 → 다음 요소에서 다시 읽는" 패턴이 반복되면, 요소 개수만큼 Layout이 강제로 반복 실행되어 프레임 시간이 크게 늘어납니다.


확인은 추측으로 끝내지 않고 Chrome DevTools Performance 패널을 켜서 스크롤을 재현합니다. 타임라인에 Layout(보라색) 구간이 프레임마다 여러 번 나타나는지, "Forced reflow" 경고가 뜨는지를 실제로 확인해서 병목 지점을 특정합니다.


해결은 두 방향으로 접근합니다. 첫째, 읽기 연산을 전부 먼저 수행해서 배열에 모아둔 뒤, 쓰기 연산을 그 다음에 한 번에 몰아서 실행합니다. 이렇게 하면 Layout이 프레임당 한 번만 발생합니다. 둘째, 가능하면 애초에 위치를 직접 읽고 쓰는 방식 대신 transform으로 이동을 구현해서 Layout/Paint 자체를 건너뛰고 Compositor 단계만 거치도록 바꿉니다. 스크롤 핸들러의 실행 빈도 자체도 requestAnimationFrame으로 프레임 주기에 맞춰 스로틀링해서, 브라우저가 원래 처리하려던 배치 방식과 코드 실행 방식을 일치시킵니다.


다만 transform이나 will-change로 레이어를 승격하는 것도 공짜가 아니라는 점을 함께 고려합니다. 레이어가 늘어나면 GPU 메모리 사용량과 합성 비용이 함께 늘어나므로, 실제로 자주 움직이는 요소에만 선택적으로 적용하고, 애니메이션이 끝나면 will-change를 해제해서 레이어를 반납합니다.

이 답변이 좋은 이유:

  • 증상을 "버벅인다"에서 그치지 않고 강제 동기 레이아웃이라는 구체적 메커니즘으로 설명함
  • 추측이 아니라 DevTools로 실제 증명하는 절차를 포함함
  • 해결책이 코드 패턴(batch) + 대체 속성(transform) + 실행 빈도 제어(rAF) 세 층위로 구성됨
  • GPU 가속의 반대급부(메모리, 합성 비용)까지 언급해 트레이드오프를 안다는 신호를 줌

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"requestAnimationFrame으로 감싸면 해결됩니다."

이 답변이 약한 이유

  • 왜 requestAnimationFrame이 도움이 되는지(프레임 주기와 실행 주기를 일치시켜 배치 처리를 가능하게 함)를 설명하지 못합니다.
  • 코드 안에 읽기/쓰기가 여전히 교차되어 있으면 requestAnimationFrame으로 감싸도 Layout Thrashing이 완전히 사라지지 않는다는 것을 놓칩니다.
  • 근본 원인(강제 동기 레이아웃)에 대한 언급이 전혀 없습니다.
WEAK ANSWER — 또 다른 패턴

"모든 요소에 will-change: transform을 걸어서 GPU 가속을 시키겠습니다."

이 답변이 약한 이유

  • GPU 가속을 만능 해결책처럼 제시하며, 레이어 폭발과 메모리 비용이라는 반대급부를 전혀 언급하지 않습니다.
  • 애초에 왜 스크롤이 버벅였는지(읽기/쓰기 교차)는 그대로 둔 채 증상만 다른 방식으로 가리는 접근입니다.
STRONG ANSWER 요소 체크리스트
  • 강제 동기 레이아웃이 왜 발생하는지 메커니즘으로 설명한다
  • DevTools로 실제 병목을 증명하는 절차를 포함한다
  • 읽기/쓰기 분리(batch)와 Compositor 전용 속성 대체를 함께 제시한다
  • GPU 레이어 승격의 비용(메모리, 합성 오버헤드)까지 언급한다

11. Senior-level Follow-up Q&A

Q: ResizeObserver나 IntersectionObserver가 왜 scroll 이벤트 안에서 직접 위치를 읽는 것보다 나은가요?

모범 답변: scroll 핸들러 안에서 getBoundingClientRect()를 호출하면 메인 스레드가 강제로 동기적으로 Layout을 계산해야 합니다. 반면 IntersectionObserver는 브라우저 내부적으로 이미 계산해둔 교차 여부를 비동기 콜백으로 전달해주기 때문에, 개발자 코드가 Layout을 강제로 트리거하지 않습니다. 즉, "직접 계산해서 물어보는 것"과 "브라우저가 계산이 끝나면 알려주는 것"의 차이이며, 후자가 메인 스레드 부하를 훨씬 줄입니다.

Q: 애니메이션에 transform만 쓰면 항상 안전한가요?

모범 답변: Layout/Paint를 건너뛴다는 점에서 대부분의 경우 유리하지만 무조건 안전하지는 않습니다. transform으로 요소를 옮기면 그 요소가 다른 요소와 겹치게 되는 시점에 페인트 순서(stacking context)가 바뀌어 예상치 못한 레이어가 추가로 생성될 수 있고, filter와 결합되면 일부 브라우저에서는 여전히 비싼 Paint 연산이 발생하기도 합니다. "이 속성을 쓰면 무조건 빠르다"가 아니라 "이 속성이 왜 파이프라인의 앞 단계를 건너뛰게 하는지"를 근거로 판단해야 합니다.

Q: 리스트에 아이템이 수천 개인데 스크롤 성능이 여전히 나쁩니다. Layout Thrashing을 다 고쳤는데도 느리다면 다음엔 뭘 의심하시겠어요?

모범 답변: DOM 노드 수 자체가 문제일 가능성을 봅니다. 화면에 보이지 않는 수천 개의 요소도 브라우저는 여전히 스타일 계산과 페인트 대상에 포함시킬 수 있습니다. 이 경우 근본 해결책은 코드 최적화가 아니라 아키텍처 변경, 즉 가상 스크롤(windowing/virtualization)을 도입해서 실제로 화면에 보이는 요소만 DOM에 존재하도록 만드는 것입니다.

Q (비기술적): 디자이너가 "카드 호버 애니메이션이 밋밋하니 그림자를 애니메이션으로 크게 키워달라"고 요청합니다. 어떻게 대응하시겠어요?

모범 답변: box-shadow를 직접 애니메이션하면 매 프레임 Paint가 다시 발생해 리스트 규모에 따라 버벅일 수 있다는 점을 먼저 설명합니다. 다만 요청 자체(더 입체적인 호버 효과)는 거부하지 않고, 실제 그림자 대신 미리 그려둔 그림자를 가진 의사(pseudo) 레이어의 opacity를 0에서 1로 전환하는 방식으로 같은 시각 효과를 Compositor 단계에서만 처리하도록 제안합니다. 즉, 디자인 의도는 유지하면서 구현 방식만 성능이 안전한 속성으로 바꾸는 협업 방식입니다.

12. 흔한 오해

오해 1

"GPU 레이어를 많이 쓰면 무조건 빠르다"

레이어마다 별도 메모리 할당과 합성 비용이 들기 때문에, 레이어 수가 지나치게 많아지면 오히려 프레임 시간이 늘어날 수 있습니다. GPU 가속은 "자주 움직이는 소수의 요소"에 선택적으로 적용해야 하는 도구이지, 모든 요소에 걸어두는 기본값이 아닙니다.

오해 2

"Reflow와 Repaint는 같은 것이다"

Reflow(Layout)는 위치·크기를 다시 계산하는 것이고, Repaint(Paint)는 이미 정해진 위치에 픽셀을 다시 칠하는 것입니다. Reflow가 일어나면 Repaint도 함께 일어나지만, 반대로 Repaint만 필요한 경우(배경색 변경 등)에는 Reflow 없이 끝날 수 있습니다. 이 차이를 구분하지 못하면 왜 어떤 속성 변경이 다른 것보다 훨씬 비싼지 설명할 수 없습니다.

오해 3

"offsetTop을 읽는 것은 부수효과가 없는 순수한 조회다"

값을 "읽기만" 하는 것처럼 보이지만, 대기 중인 스타일 변경이 있다면 브라우저는 최신 값을 보장하기 위해 그 순간 강제로 Layout을 재계산합니다. 읽기 연산이 계산 비용을 유발하는 부수효과를 가진다는 것이 Layout Thrashing의 핵심 원인입니다.

오해 4

"requestAnimationFrame으로 감싸기만 하면 Layout Thrashing이 무조건 사라진다"

requestAnimationFrame은 실행 시점을 프레임 시작에 맞춰줄 뿐, 콜백 내부에서 여전히 읽기와 쓰기가 교차되어 있다면 그 프레임 안에서 강제 Layout이 반복될 수 있습니다. 실행 시점 조정과 읽기/쓰기 분리(batch)는 별개의 최적화이며 둘 다 필요합니다.

오해 5

"성능 문제는 항상 JavaScript 실행 시간이 원인이다"

DevTools에서 프레임 시간이 길게 나와도 실제로는 JavaScript 실행 자체가 아니라 그 JavaScript가 유발한 강제 Layout/Paint 시간이 대부분을 차지하는 경우가 많습니다. 플레임 차트에서 "Scripting"과 "Rendering(Layout/Paint)" 항목을 구분해서 봐야 정확한 병목을 찾을 수 있습니다.
