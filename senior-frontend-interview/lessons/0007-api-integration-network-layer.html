1. 핵심 면접 질문

Primary Question

"백엔드 API가 간헐적으로 5xx를 반환하는 상황에서, 프론트엔드가 사용자 경험을 지키기 위해 어떤 재시도·폴백 전략을 설계하시겠어요?"

Variant 1

"결제 API 호출이 타임아웃되었는데, 실제로는 서버에서 결제가 성공적으로 처리됐을 수 있습니다. 클라이언트가 재시도를 안전하게 하려면 무엇을 보장해야 하나요?"

Variant 2

"우리 서비스가 REST에서 GraphQL로 마이그레이션을 검토 중입니다. 어떤 기준으로 전환 여부를 판단하고, 전환한다면 어떤 리스크를 관리하시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 "fetch 문법을 아는가"가 아니라, 네트워크는 항상 실패할 수 있다는 전제 위에서 사용자 경험을 방어적으로 설계할 수 있는가를 테스트합니다. 주니어는 "API를 호출해서 데이터를 받아온다"를 단순한 성공 경로로만 생각하지만, 시니어는 타임아웃, 부분 실패, 중복 요청, 순서 역전(race condition), 오프라인 전환 같은 실패가 기본값인 세계를 전제로 설계합니다.

면접관이 이 질문으로 평가하는 것:

  • 실패 모드를 구체적으로 나열할 수 있는가 — "네트워크 에러"를 뭉뚱그리지 않고, 타임아웃/4xx/5xx/네트워크 단절/응답 파싱 실패를 각각 다르게 다뤄야 한다는 것을 아는가.
  • 멱등성(Idempotency)을 이해하는가 — 어떤 요청은 안전하게 재시도할 수 있고 어떤 요청은 재시도 시 중복 부작용(이중 결제 등)을 낳는지 구분할 수 있는가.
  • 캐싱 계층을 여러 층위로 사고하는가 — 브라우저 HTTP 캐시, CDN 캐시, 클라이언트 상태 캐시(React Query 등)가 서로 다른 계층이며 각각 다른 무효화 전략이 필요하다는 것을 아는가.
  • 사용자에게 실패를 어떻게 노출할지 설계할 수 있는가 — 무한 스피너, 백지 화면, 이해 불가능한 에러 코드 대신 재시도 버튼, 낙관적 UI 롤백, 스켈레톤 폴백 같은 구체적 UX 패턴으로 답할 수 있는가.
  • API 계약 설계가 아키텍처 결정이라는 것을 아는가 — REST/GraphQL 선택이 유행이 아니라 오버페칭/언더페칭, 캐싱 용이성, 팀 규모, 타입 안정성 같은 구체적 트레이드오프의 결과라는 것을 설명할 수 있는가.

주니어~중니어에서 자주 나오는 답변 패턴: "실패하면 알림창을 띄우고 다시 시도 버튼을 누르게 하겠습니다"처럼 재시도를 단순 UI 문제로 축소하거나, "GraphQL이 더 최신이니까 GraphQL을 쓰겠습니다"처럼 근거 없는 유행을 따르는 경우입니다. 시니어는 어떤 요청이 안전하게 재시도 가능한지, 실패가 사용자에게 어떤 손실을 주는지부터 구분합니다.

3. 전제 개념 — 쉬운 한국어로

멱등성(Idempotency)이란 무엇인가

같은 요청을 한 번 보내든 여러 번 보내든 서버 상태에 미치는 최종 결과가 동일한 성질을 말합니다. 비유하면, 엘리베이터 버튼을 한 번 누르든 세 번 연타하든 엘리베이터는 한 번만 오는 것이 멱등한 동작이고, 자판기에 동전을 넣고 버튼을 누를 때마다 매번 음료가 나오는 것은 멱등하지 않은 동작입니다. HTTP에서는 GET, PUT, DELETE가 정의상 멱등하고(같은 리소스를 몇 번 조회하거나 몇 번 같은 값으로 덮어써도 결과는 같음), POST는 정의상 멱등하지 않습니다(주문 생성 요청을 두 번 보내면 주문이 두 개 생길 수 있음). 이 구분이 중요한 이유는 재시도가 안전한지 아닌지를 이 성질 하나로 판단하기 때문입니다.

왜 타임아웃이 "실패"를 의미하지 않는가

클라이언트가 타임아웃을 겪었다는 것은 "응답을 못 받았다"는 뜻이지 "서버가 처리를 못 했다"는 뜻이 아닙니다. 서버는 요청을 정상적으로 처리하고 응답을 보내는 중이었는데, 그 응답이 네트워크 어딘가에서 유실되었거나 클라이언트의 대기 시간이 서버의 처리 시간보다 짧아서 먼저 포기했을 수 있습니다. 결제 API처럼 부작용이 있는 요청에서 이 구분을 무시하고 "타임아웃 = 실패 = 재시도"로 단순화하면, 실제로는 결제가 이미 성공했는데 재시도로 인해 이중 결제가 발생하는 사고로 이어집니다.

오버페칭(Overfetching)과 언더페칭(Underfetching)이란 무엇인가

오버페칭은 화면에 필요하지 않은 데이터까지 서버가 응답에 포함시켜 보내는 것입니다. 예를 들어 사용자 이름만 필요한 화면인데 REST API의 /users/1 엔드포인트가 주소·주문내역·설정까지 통째로 응답하는 경우입니다. 언더페칭은 반대로 한 번의 요청으로 필요한 데이터를 다 못 받아, 여러 번 순차적으로 추가 요청을 보내야 하는 것입니다. 예를 들어 게시글 목록을 받은 뒤 각 게시글의 작성자 정보를 얻기 위해 게시글 개수만큼 추가 요청을 보내는 N+1 패턴입니다. REST는 엔드포인트가 고정된 응답 구조를 가지므로 이 두 문제가 함께 발생하기 쉽고, GraphQL은 클라이언트가 필요한 필드만 선언해서 요청할 수 있어 이 문제를 구조적으로 줄입니다.

클라이언트 캐시와 서버/CDN 캐시는 왜 다른 계층인가

브라우저의 HTTP 캐시나 CDN 캐시는 "이 URL의 응답을 얼마나 오래 재사용할 수 있는가"를 다루는 전송 계층의 캐시입니다. 반면 React Query나 SWR 같은 클라이언트 상태 캐시는 "이미 받아온 데이터를 화면 간에 어떻게 공유하고, 언제 다시 요청할 것인가"를 다루는 애플리케이션 계층의 캐시입니다. 비유하면 HTTP 캐시는 "택배 창고에 물건을 얼마나 오래 보관할지"이고, 클라이언트 상태 캐시는 "이미 집에 들어온 물건을 여러 방에서 어떻게 공유해서 쓸지"입니다. 두 계층은 독립적으로 존재하며, 하나만 관리해서는 데이터 신선도와 성능을 동시에 잡을 수 없습니다.

낙관적 업데이트(Optimistic Update)란 무엇인가

서버 응답을 기다리지 않고 요청이 성공할 것이라 가정하고 화면을 먼저 갱신하는 패턴입니다. 예를 들어 "좋아요" 버튼을 누르면 서버 응답을 기다리지 않고 즉시 하트를 채우고 카운트를 올립니다. 이렇게 하면 사용자는 네트워크 지연을 전혀 체감하지 못합니다. 다만 실제로 서버 요청이 실패하면 화면을 원래 상태로 되돌리는(rollback) 로직이 반드시 함께 있어야 하며, 이 롤백을 빠뜨리면 화면과 실제 서버 상태가 어긋난 채로 사용자가 잘못된 정보를 계속 보게 되는 사고가 발생합니다.

4. REST vs GraphQL — 계약 설계 트레이드오프

REST와 GraphQL의 근본적 차이

축RESTGraphQL
데이터 형태 결정 주체서버(엔드포인트가 고정 응답 구조 정의)클라이언트(쿼리로 필요한 필드 선언)
오버/언더페칭구조적으로 발생하기 쉬움클라이언트가 필요한 만큼만 받아 구조적으로 완화
HTTP 캐싱 친화성URL 단위 캐싱이 자연스러움(CDN, 브라우저 캐시)대부분 POST 단일 엔드포인트라 URL 기반 캐싱이 어려움
타입 안정성OpenAPI 등 별도 스키마 도구 필요스키마가 프로토콜에 내장, 코드젠으로 타입 자동 생성 용이
서버 복잡도엔드포인트별로 단순, 개수가 늘어남리졸버 계층, N+1 방지(DataLoader) 등 서버 구현 복잡도 증가
팀 규모 적합성소규모 팀, 단순 CRUD 도메인에 적합다수 클라이언트(웹/앱/파트너)가 서로 다른 데이터 형태를 요구할 때 유리

GraphQL이 "REST의 상위 호환"이 아닙니다. GraphQL은 캐싱을 URL 기반에서 정규화된 클라이언트 캐시(Apollo/Relay의 정규화 캐시)로 옮기는 대가를 치르고, 서버 쪽에서는 임의의 쿼리 조합이 만들어내는 성능 리스크(과도하게 깊은 쿼리, N+1)를 직접 관리해야 하는 부담을 집니다.

전환 여부를 판단하는 기준

  • 1
    실제로 오버/언더페칭이 측정 가능한 문제인가

    "이론적으로 GraphQL이 낫다"가 아니라, 실제 프로덕션에서 특정 화면이 N+1 요청을 하고 있는지, 응답 페이로드 중 사용하지 않는 필드 비율이 얼마나 되는지 실측합니다.

  • 2
    클라이언트 다양성이 있는가

    웹/모바일/서드파티 파트너처럼 같은 도메인 데이터를 서로 다른 형태로 필요로 하는 소비자가 여러 개일수록 GraphQL의 이점이 커집니다. 클라이언트가 하나뿐이라면 REST로도 충분한 경우가 많습니다.

  • 3
    팀이 새로운 캐싱·성능 리스크를 감당할 준비가 되어 있는가

    GraphQL 도입은 CDN 캐싱 전략 재설계, 쿼리 복잡도 제한(depth limiting, query cost analysis), N+1 방지용 DataLoader 패턴 도입 등 새로운 운영 부담을 동반합니다.

  • 4
    점진적 전환이 가능한가

    전면 재작성보다는 신규 화면이나 오버페칭이 가장 심한 화면부터 GraphQL 게이트웨이(BFF)를 앞에 두고, 기존 REST 백엔드를 그 뒤에서 점진적으로 통합하는 방식으로 리스크를 낮춥니다.

5. HTTP 캐싱과 클라이언트 캐싱 계층

HTTP 캐시 계층 — Cache-Control과 검증 캐시

헤더/전략동작적합한 리소스
Cache-Control: max-age=31536000, immutable만료 전까지 서버에 재검증 없이 그대로 재사용파일명에 해시가 포함된 정적 자산(JS/CSS 번들, 이미지)
Cache-Control: no-cache매번 서버에 재검증(ETag/Last-Modified) 요청, 304면 재다운로드 생략자주 바뀌지만 대역폭은 아끼고 싶은 API 응답
Cache-Control: no-store캐시 자체를 금지, 매번 완전히 새로 요청민감한 개인정보, 결제 정보가 담긴 응답
ETag / If-None-Match리소스 내용 해시로 변경 여부 검증, 미변경 시 304로 바디 생략내용이 자주 바뀌지 않는 API 응답(사용자 프로필 등)
stale-while-revalidate만료된 캐시라도 즉시 응답하고, 백그라운드로 최신 데이터를 갱신완전한 최신성보다 즉시 응답이 더 중요한 목록/피드 API

정적 자산에는 파일명 해시(app.a1b2c3.js)와 immutable을 함께 써서 "영구 캐시 + 배포 시 URL 자체가 바뀌어 무효화"라는 조합을 만드는 것이 실무 표준입니다. 반대로 API 응답에 이런 영구 캐시를 잘못 적용하면 사용자가 오래된 데이터를 계속 보는 사고로 이어집니다.

클라이언트 상태 캐시(React Query/SWR류)의 역할

HTTP 캐시가 "네트워크 요청을 생략할 수 있는가"를 다룬다면, 클라이언트 상태 캐시는 "이미 메모리에 있는 데이터를 여러 컴포넌트가 어떻게 재사용하고, 언제 다시 fetch할 것인가"를 다룹니다. 핵심 개념은 다음과 같습니다.

  • Stale-While-Revalidate 패턴: 캐시된 데이터를 즉시 화면에 보여주면서 동시에 백그라운드로 최신 데이터를 가져와 조용히 교체합니다. 사용자는 로딩 스피너 없이 즉시 화면을 보고, 데이터는 곧 최신화됩니다.
  • 쿼리 키 기반 캐시 정규화: ['user', userId]처럼 요청 파라미터를 키로 캐시를 식별해, 같은 데이터를 요청하는 여러 컴포넌트가 중복 요청 없이 캐시를 공유합니다.
  • 캐시 무효화(Invalidation) 시점 설계: 데이터를 변경하는 mutation이 성공한 직후 관련된 쿼리 키를 무효화(invalidate)해서 다음 조회 시 최신 데이터를 강제로 다시 받아오게 합니다. 이 시점 설계를 놓치면 사용자가 방금 수정한 데이터가 반영되지 않은 화면을 계속 보게 됩니다.

캐시 계층이 어긋나서 발생하는 전형적 버그

"방금 프로필 사진을 바꿨는데 새로고침해도 예전 사진이 보인다"는 버그는 보통 다음 계층 중 하나가 원인입니다.

  • CDN이 이미지 URL을 오래 캐싱하고 있는데, 업로드 후 URL을 바꾸지 않고 같은 경로에 덮어써서 CDN이 이전 버전을 계속 서빙(→ URL에 버전/해시를 포함시켜야 함).
  • 클라이언트 상태 캐시(React Query)에서 프로필 업데이트 mutation 후 관련 쿼리 키를 invalidate하지 않아, 화면은 여전히 메모리에 있는 이전 데이터를 보여줌.
  • 브라우저 HTTP 캐시가 API 응답에 과도하게 긴 max-age를 갖고 있어, 재검증 없이 이전 응답을 그대로 재사용.

이 사슬을 이해하지 못하면 "캐시를 지우면 되겠지"라는 임시방편만 반복하게 됩니다.

6. 에러 처리, 재시도, 멱등성

실패 유형별로 다른 대응이 필요한 이유

실패 유형의미적절한 대응
4xx (Client Error)요청 자체가 잘못됨(잘못된 입력, 인증 만료 등)재시도해도 결과가 같음 — 재시도 금지, 사용자에게 입력 수정 요구 또는 재로그인 유도
5xx (Server Error)서버 측 일시적 문제일 가능성제한된 횟수 내 지수 백오프 재시도가 유효할 수 있음
타임아웃서버 처리 결과를 알 수 없음(성공했을 수도 있음)멱등한 요청만 안전하게 재시도, 비멱등 요청은 상태 조회로 실제 결과 확인 우선
네트워크 단절요청 자체가 전송되지 않았거나 응답을 받을 수 없음연결 복구를 감지해 큐에 쌓인 요청을 재전송(오프라인 큐잉)
응답 파싱 실패서버 계약이 예상과 다름(스키마 변경, 손상된 응답)재시도로 해결 안 됨 — 즉시 에러 리포팅, 폴백 UI로 전환

"에러가 나면 재시도한다"는 하나의 규칙으로 뭉뚱그리면, 4xx를 무한 재시도해 서버에 불필요한 부하를 주거나, 비멱등 요청을 재시도해 중복 부작용을 만드는 사고로 이어집니다.

안전한 재시도의 두 가지 축 — 지수 백오프와 멱등성 키

지수 백오프(Exponential Backoff)는 재시도 간격을 1초, 2초, 4초, 8초처럼 점점 늘려서, 일시적 장애 상황에서 클라이언트들이 동시에 재시도를 몰아붙여 서버를 더 무너뜨리는(retry storm) 것을 방지합니다. 여기에 무작위 지연(jitter)을 더해 여러 클라이언트의 재시도 타이밍이 겹치지 않게 분산시키는 것이 실무 표준입니다.

async function fetchWithRetry(url, options, maxRetries = 3) {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const res = await fetch(url, options);
      if (res.status >= 500 && attempt < maxRetries) {
        const delay = 2 ** attempt * 1000 + Math.random() * 300;
        await new Promise(r => setTimeout(r, delay));
        continue;
      }
      return res;
    } catch (networkErr) {
      if (attempt === maxRetries) throw networkErr;
      const delay = 2 ** attempt * 1000 + Math.random() * 300;
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

멱등성 키(Idempotency Key)는 비멱등 요청(주로 POST)을 안전하게 재시도하기 위한 서버-클라이언트 계약입니다. 클라이언트가 요청 생성 시 고유 키(UUID)를 헤더에 담아 보내고, 서버는 같은 키로 들어온 요청을 이미 처리했다면 새로 처리하지 않고 이전 처리 결과를 그대로 반환합니다. 결제 API처럼 중복 실행이 치명적인 경우, 클라이언트 단독으로는 멱등성을 보장할 수 없고 반드시 이 서버 계약이 함께 있어야 합니다.

사용자에게 실패를 어떻게 보여줄 것인가

  • 스켈레톤/캐시된 이전 데이터 폴백: 완전한 백지 에러 화면 대신 마지막으로 성공한 데이터를 "새로고침이 지연되고 있습니다" 안내와 함께 유지합니다.
  • 낙관적 업데이트 롤백: 낙관적으로 반영했던 화면을 실패 시 원래 상태로 되돌리고, 무엇이 실패했는지 구체적으로 안내합니다(예: "좋아요 반영에 실패했습니다. 다시 시도해주세요").
  • 재시도 가능 여부를 사용자에게 구분해서 제시: 4xx(입력 문제)는 "다시 시도" 버튼보다 무엇을 고쳐야 하는지 안내하고, 5xx/네트워크 문제는 "다시 시도" 버튼과 자동 재시도를 함께 제공합니다.

7. 오프라인/불안정 네트워크 대응

오프라인 대응이 필요한 이유

모바일 사용자는 지하철, 엘리베이터, 해외 로밍 등으로 네트워크가 자주 순간적으로 끊깁니다. "오프라인이면 그냥 에러를 보여준다"는 접근은 이런 일시적 단절을 실제 실패로 오인시켜 사용자 경험을 불필요하게 나쁘게 만듭니다. 시니어 수준에서는 연결 상태를 감지하고, 가능하면 요청을 큐에 쌓아 연결이 복구되면 자동으로 재전송하는 설계를 고려합니다.

연결 상태 감지와 큐잉 패턴

  • 1
    navigator.onLine과 online/offline 이벤트로 상태 감지

    단, navigator.onLine은 "네트워크 인터페이스가 연결되어 있는가"만 알려줄 뿐 실제로 서버에 도달 가능한지는 보장하지 않으므로, 중요한 판단에는 실제 요청 성공/실패 결과를 함께 신뢰해야 합니다.

  • 2
    변경 요청(mutation)을 로컬 큐에 저장

    오프라인 중 발생한 사용자의 변경 액션(글 작성, 좋아요 등)을 즉시 실패시키지 않고 IndexedDB 등에 큐로 저장하며, UI는 낙관적으로 먼저 반영합니다.

  • 3
    연결 복구 시 큐를 순서대로 재전송

    online 이벤트나 Service Worker의 Background Sync API를 활용해 연결이 복구되면 큐에 쌓인 요청을 순서를 지켜 재전송합니다. 순서가 중요한 요청(댓글 순서 등)은 서버에서도 처리 순서를 보장해야 합니다.

  • 4
    충돌(Conflict) 처리 전략을 사전에 정의

    오프라인 중 로컬에서 수정한 데이터와 그 사이 서버에서 다른 클라이언트가 변경한 데이터가 충돌할 수 있습니다. "마지막 쓰기가 이긴다(Last-Write-Wins)"처럼 단순한 규칙부터, 버전 번호 비교로 충돌을 감지해 사용자에게 병합을 요청하는 방식까지 도메인 중요도에 따라 선택합니다.

8. 답변 사고 순서

  1. 요청의 성질을 먼저 분류합니다. 문제가 된 요청이 멱등한지(GET/PUT/DELETE) 비멱등한지(POST), 부작용의 심각도(단순 조회 vs 결제)를 확인합니다.
  2. 실패 유형을 세분화합니다. 5xx인지, 타임아웃인지, 네트워크 단절인지에 따라 재시도 가능 여부와 전략이 달라진다는 것을 전제로 둡니다.
  3. 서버 계약이 재시도를 안전하게 만들 수 있는지 확인합니다. 멱등성 키 같은 서버 측 지원이 있는지, 없다면 도입을 제안할 것인지 판단합니다.
  4. 재시도 정책을 설계합니다. 지수 백오프 + jitter, 최대 재시도 횟수, 재시도 대상 상태 코드를 구체적으로 정합니다.
  5. 사용자에게 노출할 UX를 설계합니다. 재시도 중임을 알릴지, 실패 시 어떤 폴백(캐시된 데이터, 재시도 버튼, 안내 문구)을 보여줄지 결정합니다.
  6. 재발 방지와 관측을 붙입니다. 실패율과 재시도 성공률을 모니터링에 남겨, 특정 엔드포인트의 실패가 급증하면 알림이 오도록 합니다.

9. 핵심 답변 예시

면접관: "결제 API 호출이 타임아웃되었는데, 실제로는 서버에서 결제가 성공적으로 처리됐을 수 있습니다. 클라이언트가 재시도를 안전하게 하려면 무엇을 보장해야 하나요?"

STRONG ANSWER

먼저 결제 요청은 POST이므로 정의상 멱등하지 않다는 점을 짚습니다. 타임아웃은 "서버가 실패했다"는 뜻이 아니라 "응답을 못 받았을 뿐 서버는 처리를 완료했을 수 있다"는 뜻이므로, 클라이언트가 임의로 같은 결제 요청을 그대로 재전송하면 이중 결제가 발생할 위험이 있습니다.


이 문제를 클라이언트 단독으로는 안전하게 풀 수 없기 때문에, 서버와 함께 멱등성 키(Idempotency Key) 계약을 도입하는 것을 제안합니다. 결제 요청을 생성하는 시점에 클라이언트가 UUID를 생성해 Idempotency-Key 헤더에 담아 보내고, 이 키는 재시도 시에도 동일하게 재사용합니다. 서버는 같은 키로 들어온 요청을 이미 처리했다면 새로 결제를 실행하지 않고 이전 처리 결과(성공/실패)를 그대로 반환하도록 구현합니다.


이 계약이 없는 상황이라면, 무작정 재시도하는 대신 결제 상태를 조회하는 별도 엔드포인트(GET /payments/{orderId}/status)로 먼저 실제 처리 결과를 확인하고, 그 결과가 없거나 명시적으로 실패인 경우에만 재시도하는 방식으로 폴백합니다.


UX 측면에서는 타임아웃이 발생한 순간 사용자에게 "결제 처리 중입니다"라는 중간 상태를 보여주고, 상태 조회가 끝날 때까지 중복 클릭을 막기 위해 결제 버튼을 비활성화합니다. 실제로 실패로 확인된 경우에만 재시도 버튼을 노출합니다.

이 답변이 좋은 이유:

  • 결제 요청이 비멱등하다는 것을 근거로 "무작정 재시도가 위험하다"는 결론을 도출함
  • 클라이언트 단독 해결이 불가능함을 인정하고 서버와의 계약(멱등성 키)을 제안함
  • 계약이 없는 현실적 상황에 대한 폴백(상태 조회 우선)까지 제시함
  • UX 레벨에서 중복 클릭 방지와 상태 안내까지 구체적으로 다룸

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"타임아웃이 나면 3번까지 자동으로 재시도하는 로직을 붙이겠습니다."

이 답변이 약한 이유

  • 결제처럼 부작용이 있는 비멱등 요청에 무조건적 재시도를 적용해 이중 결제 위험을 만듭니다.
  • 타임아웃이 "서버 처리 결과를 알 수 없는 상태"라는 것을 고려하지 않고 곧바로 재시도로 넘어갑니다.
  • 서버와의 계약(멱등성 키, 상태 조회 API) 없이 클라이언트 단독으로 문제를 풀려고 합니다.
WEAK ANSWER — 또 다른 패턴

"에러가 나면 사용자에게 알림창으로 '오류가 발생했습니다. 다시 시도해주세요'라고 보여주겠습니다."

이 답변이 약한 이유

  • 4xx(사용자가 고쳐야 할 문제)와 5xx(서버 문제)를 구분하지 않아 사용자가 무엇을 해야 할지 알 수 없습니다.
  • 이미 처리됐을 수도 있는 요청을 사용자가 그대로 재시도하게 만들어 근본 문제(멱등성)를 해결하지 못합니다.
STRONG ANSWER 요소 체크리스트
  • 요청이 멱등한지 아닌지부터 구분하고 재시도 가능 여부를 판단한다
  • 타임아웃을 실패가 아니라 "결과를 알 수 없는 상태"로 취급한다
  • 클라이언트 단독 해결의 한계를 인정하고 서버 계약(멱등성 키, 상태 조회)을 함께 설계한다
  • 실패 유형별로 다른 UX(재시도 버튼 vs 입력 수정 안내)를 제시한다
  • 재발 방지를 위한 관측(실패율 모니터링)까지 언급한다

11. Senior-level Follow-up Q&A

Q: 낙관적 업데이트(Optimistic Update)를 적용할 때, 어떤 기준으로 적용 여부를 결정하시나요?

모범 답변: 실패 확률이 낮고, 실패했을 때 되돌리기가 쉬우며, 사용자에게 즉각적인 반응이 중요한 액션(좋아요, 북마크, 읽음 표시)에는 낙관적 업데이트가 적합합니다. 반대로 결제, 주문 확정처럼 실패 시 되돌리기 어렵거나 사용자가 실패 여부를 정확히 알아야 하는 액션에는 낙관적 업데이트를 쓰지 않고 서버 응답을 기다린 후 반영합니다. 적용하더라도 실패 시 롤백 로직과 사용자에게 실패를 알리는 안내를 반드시 함께 구현해야 하며, 롤백 없는 낙관적 업데이트는 화면과 실서버 상태의 불일치를 방치하는 것과 같습니다.

Q: React Query 같은 클라이언트 캐시 라이브러리 없이 직접 fetch로 데이터 계층을 구성한다면, 어떤 것들을 직접 구현해야 하나요?

모범 답변: 최소한 다음을 직접 구현해야 합니다 — 동일 데이터를 요청하는 여러 컴포넌트 간 요청 중복 제거(deduplication), 캐시된 데이터의 신선도 판단과 재검증(stale time), 포커스 복귀나 재연결 시 자동 재검증, 에러 발생 시 재시도 정책, 그리고 mutation 후 관련 캐시 무효화. 이 목록을 나열해보면 React Query 같은 라이브러리가 "편의 기능"이 아니라 데이터 신선도·성능·일관성이라는 세 가지 문제를 구조적으로 함께 해결해주는 인프라라는 것이 드러나며, 이것이 왜 직접 구현보다 검증된 라이브러리를 쓰는 것이 실무적으로 합리적인지의 근거가 됩니다.

Q: GraphQL을 도입하면 HTTP 캐싱(CDN, 브라우저 캐시)을 포기해야 하나요?

모범 답변: 완전히 포기하는 것은 아니지만 REST만큼 자연스럽지는 않습니다. GraphQL은 보통 단일 엔드포인트에 POST로 쿼리를 보내기 때문에 URL 기반의 CDN/브라우저 캐싱이 기본적으로 작동하지 않습니다. 이를 보완하기 위해 Persisted Queries(자주 쓰는 쿼리를 서버에 미리 등록하고 짧은 해시로 GET 요청을 보내 URL 캐싱을 되살리는 기법)를 쓰거나, Apollo Client/Relay의 정규화된 클라이언트 캐시로 애플리케이션 계층에서 캐싱 이점을 확보하는 방향으로 전환합니다. 즉 캐싱의 책임이 전송 계층(CDN)에서 애플리케이션 계층(클라이언트 캐시)으로 옮겨간다고 설명하는 것이 정확합니다.

Q (비기술적): 백엔드 팀이 API 응답 구조를 변경하겠다고 통보 없이 배포해서 프론트엔드가 장애를 겪었습니다. 재발 방지를 위해 어떤 프로세스를 제안하시겠어요?

모범 답변: 먼저 이번 장애의 근본 원인이 "실수"가 아니라 "계약이 코드로 강제되지 않았다"는 프로세스 문제임을 짚습니다. 구체적으로는 OpenAPI/GraphQL 스키마를 단일 진실 공급원(Source of Truth)으로 두고, 스키마 변경이 있는 PR에서는 자동으로 프론트엔드가 실제 사용 중인 필드와의 호환성을 검사하는 계약 테스트(Contract Test, 예: Pact)를 CI에 추가하자고 제안합니다. 또한 breaking change와 non-breaking change를 구분하는 버전 정책(예: 필드 제거는 반드시 deprecation 기간을 거침)을 백엔드 팀과 합의하고, 이를 리뷰 체크리스트에 명시해 같은 사고가 개인의 실수가 아니라 프로세스로 방지되도록 만듭니다.

12. 흔한 오해

오해 1

"타임아웃은 곧 실패이므로 무조건 재시도해도 안전하다"

타임아웃은 응답을 못 받았다는 뜻일 뿐, 서버가 실제로 처리를 실패했다는 뜻이 아닙니다. 비멱등 요청(POST)을 무조건 재시도하면 서버가 이미 처리를 완료한 작업을 중복 실행시킬 수 있습니다.

오해 2

"GraphQL은 REST보다 항상 더 나은 최신 기술이다"

GraphQL은 오버/언더페칭을 줄이는 대신 URL 기반 HTTP 캐싱을 어렵게 만들고, 서버 쪽 쿼리 복잡도 관리(N+1, 과도하게 깊은 쿼리) 부담을 늘립니다. 클라이언트가 하나뿐이고 도메인이 단순한 서비스에서는 REST가 오히려 더 적합할 수 있습니다.

오해 3

"캐시를 쓰면 항상 최신 데이터보다 성능이 우선이다"

캐시 전략은 리소스의 성격에 따라 달라야 합니다. 정적 자산은 영구 캐시가 맞지만, 결제 상태나 재고 수량처럼 신선도가 중요한 데이터에 긴 캐시를 적용하면 사용자가 잘못된 정보를 근거로 행동하게 만드는 사고로 이어집니다.

오해 4

"navigator.onLine이 true면 서버에 정상적으로 요청을 보낼 수 있다"

navigator.onLine은 네트워크 인터페이스의 연결 여부만 알려줄 뿐, 실제로 목적지 서버까지 도달 가능한지는 보장하지 않습니다. 와이파이는 연결되어 있지만 인터넷이 끊긴 상황(캡티브 포털 등)에서도 이 값은 true일 수 있습니다.

오해 5

"낙관적 업데이트는 UX를 위해 항상 적용하는 것이 좋다"

실패 시 되돌리기 어려운 액션(결제, 삭제 확정)이나 실패율이 유의미하게 높은 요청에 낙관적 업데이트를 적용하면, 사용자가 잘못된 성공 화면을 신뢰하게 만들고 이후 롤백 시 혼란을 가중시킵니다. 적용 여부는 액션의 되돌림 가능성과 실패 시 손실 크기로 판단해야 합니다.
