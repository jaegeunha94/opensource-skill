1. 핵심 면접 질문

Primary Question

"우리 CSS가 점점 !important로 뒤덮이고 있습니다. 새 기능을 추가할 때마다 스타일이 깨져서 다들 !important로 땜질하고 있어요. 왜 이런 일이 생기고, 시니어라면 이 흐름을 어떻게 되돌리시겠어요?"

Variant 1

"<div onclick>로 버튼을 만드는 것과 <button>을 쓰는 것이 기능적으로 똑같아 보이는데, 왜 시맨틱 태그를 써야 한다고 생각하세요?"

Variant 2

"디자이너가 준 시안대로 만들었는데 특정 브라우저(예: Safari)에서만 레이아웃이 깨집니다. 어떻게 접근하시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 "CSS 속성을 아는가"가 아니라 "수백 개 컴포넌트와 수십 명의 기여자가 얽힌 코드베이스에서 스타일이 예측 가능하게 유지되도록 설계할 수 있는가"를 테스트합니다. 개인 프로젝트에서는 !important 한두 번 써도 문제가 안 보이지만, 조직 규모에서는 이것이 누적되어 "아무도 스타일을 건드리기 무서워하는" 상태로 이어집니다. 면접관은 이 누적 실패 패턴을 미리 겪어봤는지, 그리고 구조적으로 되돌릴 수 있는지를 봅니다.

면접관이 이 질문으로 평가하는 것:

  • 원인을 시스템으로 보는 시각 — "누가 !important를 썼는가"를 탓하지 않고, 어떤 구조적 조건이 !important를 필연적으로 만드는지 설명하는가.
  • Cascade 메커니즘의 실제 이해 — Specificity, 선언 순서, !important, 레이어(@layer) 간의 우선순위 계산을 정확히 알고 있는가.
  • 접근성·SEO를 부가 기능이 아니라 아키텍처 결정으로 보는 시각 — 시맨틱 마크업이 "나중에 접근성 감사할 때 고치면 되는 것"이 아니라 처음부터 구조에 내장돼야 하는 이유를 아는가.
  • 점진적 마이그레이션 능력 — 이미 망가진 대규모 CSS를 한 번에 다시 쓰지 않고, 리스크를 관리하며 단계적으로 개선하는 현실적 계획을 세울 수 있는가.
  • 크로스 브라우저 이슈에 대한 디버깅 습관 — "이 브라우저는 원래 이상해요"로 끝내지 않고, 표준과 벤더별 구현 차이를 구조적으로 확인하는 절차가 있는가.

주니어~중니어에서 자주 나오는 답변 패턴: "저는 !important를 안 쓰려고 노력합니다" 또는 "CSS Modules/Tailwind로 바꾸면 해결됩니다." 이는 도구 교체를 은탄환처럼 제시하는 답변입니다. 시니어는 도구 자체보다 "이 코드베이스에서 왜 Specificity 전쟁이 반복되는가"라는 근본 조건(전역 선택자 남용, 명확한 소유권 부재, 스타일 적용 범위의 불명확성)을 먼저 짚습니다.

3. 전제 개념 — 쉬운 한국어로

Cascade(캐스케이드)란 무엇인가

같은 요소에 여러 CSS 규칙이 동시에 적용될 수 있을 때, 브라우저가 "최종적으로 어느 규칙을 쓸지" 결정하는 절차 전체를 Cascade라고 부릅니다. 여러 사람이 동시에 같은 문서를 수정 제안하는 회의를 떠올리면 됩니다 — 누구 말을 따를지 정하는 규칙(직급 → 발언 순서 → 목소리 크기)이 필요한데, CSS에서 그 규칙이 출처(Origin) → 중요도(!important) → 레이어(@layer) → Specificity(구체성) → 선언 순서입니다.

Specificity(구체성)란 무엇인가

선택자가 "얼마나 구체적으로 대상을 지목했는가"를 점수로 매긴 것입니다. 인라인 스타일 > ID 선택자 > 클래스/속성/가상클래스 선택자 > 태그 선택자 순으로 점수가 높습니다. 이름표에 비유하면 "사람"(태그 선택자, 매우 넓음)보다 "3반 학생"(클래스 선택자)이 구체적이고, "3반 27번 김민준"(ID 선택자)이 가장 구체적인 것과 같습니다. 구체적인 이름표가 붙은 규칙이 우선 적용됩니다.

!important가 왜 "치트키"이자 "빚"인가

!important는 Specificity 계산을 건너뛰고 강제로 우선순위를 최상위로 올리는 장치입니다. 급한 불을 끄는 소화기와 같습니다 — 한 번은 화재를 끄지만, 계속 소화기로만 대응하면 애초에 화재가 왜 나는지(설계 결함)는 절대 고쳐지지 않고, 나중에는 소화기 위에 소화기를 덮어써야 하는 상황(더 강한 !important가 필요한 상황은 없지만, 실제로는 인라인 스타일 + !important까지 동원)이 됩니다. 한 팀에서 !important가 늘어나는 것은 "이 지점부터는 정상적인 Cascade로 예측이 불가능하다"는 경보 신호입니다.

시맨틱 HTML이란 무엇인가

<div>는 아무 의미가 없는 "빈 상자"이고, <button>, <nav>, <header> 같은 시맨틱 태그는 "이 상자 안의 내용이 무엇을 의미하는지"에 대한 정보(의미론, semantics)를 담고 있습니다. 책의 목차에 비유하면, 시맨틱 태그는 "이 부분은 제1장이다, 이 부분은 부록이다"라고 미리 표시해두는 것과 같습니다. 이 표시가 있어야 스크린리더, 검색엔진 크롤러, 브라우저의 접근성 트리(accessibility tree)가 문서를 올바르게 해석할 수 있습니다.

왜 "CSS 문법을 아는 것"과 "CSS 아키텍처를 설계하는 것"은 다른가

Flexbox, Grid 문법을 아는 것은 "벽돌을 쌓는 법"을 아는 것이고, CSS 아키텍처는 "이 건물에 100명이 동시에 벽을 추가해도 무너지지 않도록 설계 규칙을 정하는 것"입니다. 개인 프로젝트에서 안 보이던 문제(선택자 충돌, 스타일 소유권 불명확, 죽은 CSS 누적)는 팀 규모와 코드베이스 크기가 커질수록 기하급수적으로 커집니다. 시니어 인터뷰가 이 규모의 문제를 묻는 이유가 여기에 있습니다.

4. 시맨틱 HTML — 접근성·SEO와의 연결

<div onclick="...">로 만든 "버튼처럼 보이는 것"과 <button>의 차이는 시각적으로는 없지만, 브라우저 내부적으로는 큽니다.

기능 <div onclick> <button>
키보드로 포커스 이동(Tab) 기본 미지원 — tabindex 직접 추가 필요 기본 지원
Enter/Space로 클릭 기본 미지원 — 키보드 이벤트 핸들러 직접 구현 필요 기본 지원
스크린리더에 "버튼"으로 안내 기본 미지원 — role="button" 직접 추가 필요 기본 지원 (accessibility tree에 자동 노출)
폼 제출과의 연동 미지원 type="submit"으로 기본 연동
검색엔진의 문서 구조 이해 의미 없는 상자로 인식 <nav>/<main>/<article> 등과 결합 시 문서 구조 신호 제공

실무에서 이게 왜 "숨은 버그"인가

<div onclick>은 마우스 사용자에게는 완벽하게 동작하는 것처럼 보입니다. QA 체크리스트에 "키보드만으로 사용", "스크린리더로 사용" 항목이 없으면 이 결함은 프로덕션까지 그대로 배포됩니다. 이후 접근성 감사(외부 감사, 법적 요구사항, 대형 고객사의 접근성 요구)에서 한꺼번에 발견되어, 이미 수백 곳에 퍼진 패턴을 뒤늦게 고쳐야 하는 대규모 작업이 됩니다. 시니어는 이걸 "감사에서 걸리면 고치는 것"이 아니라 "애초에 컴포넌트 라이브러리 단에서 강제되는 것"으로 설계합니다(Day 11 디자인 시스템에서 심화).

시맨틱 마크업은 SEO에도 직접 연결됩니다. 검색엔진 크롤러는 <h1>~<h6>의 위계, <nav>/<main>/<article>의 구조를 통해 "이 페이지의 핵심 내용이 무엇인가"를 파악합니다. <div>만으로 구성된 페이지는 시각적으로 동일해도 크롤러에게는 구조 없는 텍스트 덩어리로 보입니다. (렌더링 전략과 SEO의 관계는 Day 16에서 심화합니다.)

5. Cascade/Specificity가 무너지는 실제 과정

대규모 CSS가 !important로 뒤덮이는 것은 우연이 아니라 예측 가능한 패턴을 따릅니다.

  • 1
    전역 선택자로 시작한다

    초기에는 .card, .button처럼 넓은 범위의 클래스를 전역으로 정의합니다. 프로젝트 초반에는 컴포넌트가 몇 개 없어 충돌이 안 보입니다.

  • 2
    예외 케이스가 생긴다

    "이 페이지에서만 카드 여백이 달라야 한다"는 요구가 생기면, 새 개발자는 기존 .card 정의를 건드리는 대신(다른 곳에서 깨질까 봐 두려워서) 더 구체적인 선택자(.page-x .card)를 추가합니다. Specificity가 조금 더 높은 규칙이 하나 늘어납니다.

  • 3
    Specificity 경쟁이 시작된다

    비슷한 패턴이 반복되면서 .a .b .c 같은 더 깊은 선택자, ID 선택자, 인라인 스타일이 등장합니다. 이 시점부터 새로운 스타일을 추가할 때 "어떤 규칙이 이길지" 예측하기 어려워집니다.

  • 4
    예측이 어려워지자 !important로 도피한다

    Specificity 경쟁에서 이기는 방법을 찾기보다, "무조건 이기는" !important를 쓰는 것이 개인에게는 더 빠른 해결책입니다. 하지만 이는 팀 전체의 Cascade 예측 가능성을 한 단계 더 파괴합니다.

  • 5
    !important끼리도 경쟁한다

    다른 규칙도 !important를 이겨야 하니 !important를 씁니다. 이제 Cascade의 마지막 우선순위 도구까지 무력화되어, "선언 순서"만 남은 상태(가장 나중에 로드된 CSS가 이기는 상태)가 됩니다. 이 시점에서는 CSS 파일 로드 순서를 바꾸는 것만으로 레이아웃이 깨질 수 있습니다.

구조적 해결책

  • Specificity를 낮고 평평하게 유지 — 클래스 선택자 한 겹만 쓰는 것을 원칙(BEM 같은 네이밍 컨벤션)으로 강제합니다.
  • 스타일의 범위(scope)를 물리적으로 격리 — CSS Modules, CSS-in-JS, Shadow DOM처럼 선택자가 애초에 전역으로 충돌할 수 없는 도구를 씁니다.
  • @layer로 우선순위 레이어를 명시 — reset, 기본 컴포넌트, 유틸리티, 오버라이드 레이어를 명시적으로 나누면 Specificity 값과 무관하게 레이어 순서로 우선순위가 정해져 예측 가능성이 회복됩니다.
  • 죽은 CSS 제거 파이프라인 — 사용되지 않는 규칙이 계속 쌓이는 것 자체가 Specificity 경쟁의 원인이므로 정기적으로 정리합니다.

6. CSS 아키텍처 방법론 비교

어떤 방법론이 "정답"은 아니며, 팀 규모·조직 구조·빌드 환경에 따라 트레이드오프가 다릅니다. 인터뷰에서는 "어떤 게 최고인가"가 아니라 "이 상황에서 왜 이걸 골랐는가"를 설명하는 것이 중요합니다.

방법론 핵심 아이디어 장점 비용/한계
BEM Block-Element-Modifier 네이밍으로 Specificity를 클래스 1단계로 고정 런타임 비용 없음, 어떤 빌드 환경에서도 동작, 학습 장벽 낮음 네이밍이 길어짐, 컨벤션 위반을 도구가 강제하지 못함(사람이 리뷰해야 함)
CSS Modules 빌드 타임에 클래스명을 파일 단위로 해시화해 충돌을 원천 차단 전역 충돌이 구조적으로 불가능, 죽은 CSS 추적이 쉬움(파일 단위 소유권) 동적 클래스 조합이 다소 번거로움, 빌드 도구 의존
CSS-in-JS 컴포넌트와 스타일을 같은 파일에서 관리, 종종 런타임에 스타일 생성 컴포넌트 단위 캡슐화가 매우 강력, JS 로직과 스타일을 함께 추론 가능 런타임 방식은 초기 렌더링/번들 비용 추가, SSR 스트리밍과 결합 시 추가 설정 필요
유틸리티 우선(Tailwind 계열) 미리 정의된 단일 목적 클래스를 조합해 스타일링 새 CSS 작성량 최소화, 디자인 토큰과 강하게 결합 가능, Specificity 전쟁 자체가 거의 발생하지 않음 마크업이 클래스로 뒤덮여 가독성 저하 가능, 팀의 유틸리티 클래스 학습 필요
디자인 토큰 + @layer 색상/간격/타이포그래피를 변수(토큰)로 추상화하고 레이어로 우선순위 명시 브랜드/테마 변경이 토큰 값만 바꾸면 전파됨, 크로스팀 거버넌스에 유리 토큰 설계 자체에 초기 투자 필요, 팀 전체의 합의 필요(Day 11 심화)

실무 판단 기준

스타트업 초기처럼 빠른 반복이 중요하면 유틸리티 우선이 생산성이 높습니다. 여러 팀이 하나의 디자인 시스템을 공유해야 하면 디자인 토큰 + CSS Modules/CSS-in-JS 조합이 장기적으로 유지보수에 유리합니다. 레거시 코드베이스를 점진적으로 개선해야 한다면 BEM처럼 빌드 도구 의존 없이 바로 적용 가능한 컨벤션부터 시작하는 것이 현실적입니다. "이 방법론이 최고다"라고 답하면 트레이드오프를 모른다는 신호로 읽힙니다.

7. 크로스 브라우저 레이아웃 사고

"Safari에서만 깨진다"는 리포트를 받았을 때, "Safari는 원래 이상해요"로 끝내면 시니어답지 않은 답변입니다. 실제로는 구조적으로 확인할 수 있는 몇 가지 흔한 원인이 있습니다.

흔한 크로스 브라우저 원인 설명
구현 시점 차이(spec lag) 같은 CSS 표준이라도 브라우저 엔진마다 구현 완료 시점이 다릅니다. gap의 Flexbox 지원, :has() 선택자 등은 브라우저별로 지원 시점이 크게 갈렸습니다.
기본 스타일(User Agent Stylesheet) 차이 브라우저마다 <button>, <input> 등의 기본 여백·폰트·테두리가 다릅니다. Reset/Normalize CSS 없이 배포하면 브라우저별로 다르게 보입니다.
폼 컨트롤과 스크롤바의 네이티브 렌더링 select, checkbox, 스크롤바는 OS/브라우저의 네이티브 UI를 그대로 쓰는 경우가 많아 CSS로 완전히 통일하기 어렵습니다.
서브픽셀 반올림 차이 퍼센트/뷰포트 단위 계산 시 브라우저마다 반올림 방식이 미세하게 달라 1px 수준의 레이아웃 차이가 생길 수 있습니다.
WebKit(Safari)의 독자적 제약 iOS에서는 정책상 모든 브라우저가 내부적으로 WebKit 엔진을 써야 해서, "Chrome에서 됐으니 다 될 것"이라는 가정이 iOS 전체에서 깨질 수 있습니다.

구조적 디버깅 절차

  1. 재현 확인 — 실제 해당 브라우저(또는 BrowserStack 등 실제 엔진 환경)에서 재현되는지 먼저 확인합니다. 개발자 도구의 "기기 흉내내기"는 실제 엔진이 아니므로 완전히 신뢰할 수 없습니다.
  2. 표준 문서 대조 — MDN의 브라우저 호환성 표(Can I Use)로 문제 속성/선택자의 지원 여부와 알려진 버그를 확인합니다.
  3. 기본 스타일 격리 — Reset/Normalize를 적용한 최소 재현 케이스를 만들어, 문제가 우리 CSS 때문인지 브라우저 기본 스타일 때문인지 분리합니다.
  4. 대안 속성 검토 — 지원이 갈리는 최신 속성 대신 더 널리 지원되는 대안(또는 @supports를 통한 점진적 향상)을 적용할 수 있는지 검토합니다.

8. 답변 사고 순서

  1. 증상을 구조적 원인 후보로 재정의합니다. "!important가 많다"는 결과이지, 원인이 아닙니다. Specificity 경쟁이 왜 시작됐는지(전역 선택자, 소유권 불명확) 먼저 짚습니다.
  2. 현재 코드베이스의 실제 데이터를 확인합니다. !important 사용 빈도, 평균 선택자 깊이, 죽은 CSS 비율 등을 정량적으로 파악합니다.
  3. 즉시 롤백이 불가능함을 인정합니다. "전부 다시 쓰겠다"는 비현실적이므로, 점진적 마이그레이션 전략(신규 코드부터 새 컨벤션 적용, 레이어 도입으로 우선순위 재정렬)을 제시합니다.
  4. 재발 방지 장치를 함께 제시합니다. Lint 규칙(!important 금지, 선택자 깊이 제한), 코드 리뷰 체크리스트, 디자인 토큰화로 향후 같은 문제가 반복되지 않도록 합니다.
  5. 트레이드오프를 언급합니다. 마이그레이션에 드는 시간·리스크와, 방치했을 때 커지는 유지보수 비용을 함께 저울질해서 우선순위를 제안합니다.

9. 핵심 답변 예시

면접관: "우리 CSS가 점점 !important로 뒤덮이고 있습니다. 왜 이런 일이 생기고, 시니어라면 어떻게 이 흐름을 되돌리시겠어요?"

STRONG ANSWER

!important가 늘어나는 것 자체는 원인이 아니라 결과라고 봅니다. 보통 그 이전 단계에서 전역 선택자(.card, .button 같은 넓은 클래스)를 여러 곳에서 공유하면서, 새로운 예외가 생길 때마다 원본 스타일을 고치는 대신 더 구체적인 선택자를 덧붙이는 패턴이 반복됩니다. Specificity 점수가 계속 높아지다 보면 어느 시점부터는 어떤 규칙이 이길지 예측하기 어려워지고, 팀원들은 "확실히 이기는" !important로 도피하게 됩니다.


이걸 되돌리려면 먼저 현재 상태를 정량화합니다. !important 사용 횟수, 평균 선택자 깊이, 특정 컴포넌트에 집중된 충돌 빈도를 확인해서 가장 심각한 영역을 파악합니다. 전체를 한 번에 다시 쓰는 것은 리스크가 크고 현실적이지 않으므로, 신규로 작성되는 컴포넌트부터 스타일 범위가 물리적으로 격리되는 방식(CSS Modules 또는 CSS-in-JS)이나 최소한 BEM 같은 낮은 Specificity 컨벤션을 강제합니다.


기존 코드는 한 번에 걷어내지 않고, @layer로 우선순위 레이어(reset → 기본 컴포넌트 → 유틸리티 → 예외 오버라이드)를 명시적으로 도입해서, Specificity 값에 의존하지 않고도 레이어 순서로 우선순위를 예측 가능하게 만듭니다. 그리고 이 흐름이 재발하지 않도록 Lint 규칙으로 !important 신규 추가와 과도한 선택자 깊이를 코드 리뷰 단계에서 걸러냅니다.


마지막으로 이 작업에는 시간과 리스크가 따른다는 점을 팀에 투명하게 공유합니다. 완전히 정리하는 데는 몇 스프린트가 걸릴 수 있지만, 방치할수록 신규 기능 개발 속도 자체가 계속 느려진다는 것을 데이터(예: 스타일 관련 회귀 버그 빈도 추이)로 보여주며 우선순위를 설득합니다.

이 답변이 좋은 이유:

  • 결과(!important)와 원인(전역 선택자 + Specificity 경쟁)을 구분해서 설명함
  • "전부 다시 쓰겠다" 같은 비현실적 해법 대신 점진적 마이그레이션을 제시함
  • 구체적 기술(CSS Modules, @layer, Lint 규칙)을 근거로 언급함
  • 재발 방지 장치까지 포함함
  • 비즈니스 언어(개발 속도 저하, 회귀 버그 빈도)로 우선순위를 설득하는 커뮤니케이션을 포함함

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"팀원들에게 !important를 쓰지 말라고 공지하겠습니다."

이 답변이 약한 이유

  • 왜 팀원들이 !important를 쓸 수밖에 없었는지(구조적 원인)를 짚지 않고 개인의 행동만 탓합니다.
  • 강제할 도구(Lint, 코드 리뷰 게이트) 없이 "공지"에만 의존하면 다시 반복됩니다.
  • 이미 쌓인 기존 !important는 어떻게 할지 언급이 없습니다.
WEAK ANSWER — 또 다른 패턴

"Tailwind로 전체를 마이그레이션하겠습니다."

이 답변이 약한 이유

  • 특정 도구를 은탄환처럼 제시할 뿐, 왜 이 도구가 이 팀의 근본 문제(Specificity 경쟁)를 해결하는지 설명이 없습니다.
  • 대규모 코드베이스의 전체 마이그레이션 비용과 리스크(회귀 발생 가능성, 팀 학습 곡선)를 전혀 언급하지 않습니다.
  • 점진적 전환 계획 없이 "전부 바꾸겠다"는 방식은 실제 조직에서 실행되기 어렵습니다.
STRONG ANSWER 요소 체크리스트
  • 결과(!important 남발)와 근본 원인(전역 선택자, Specificity 경쟁)을 구분한다
  • 현재 상태를 정량적으로 파악한 뒤 우선순위를 정한다
  • 전면 재작성이 아니라 점진적·리스크 관리된 마이그레이션을 제시한다
  • Lint/코드 리뷰 같은 재발 방지 장치를 포함한다
  • 기술적 결정을 비즈니스 언어(개발 속도, 회귀 빈도)로 설득한다

11. Senior-level Follow-up Q&A

Q: CSS Modules와 CSS-in-JS 중 어떤 걸 새 프로젝트에 추천하시겠어요?

모범 답변: 두 방식 모두 스타일 범위 격리라는 핵심 문제는 해결하지만, 결정 기준은 런타임 비용과 팀의 작업 방식입니다. 대량의 정적 콘텐츠와 SEO가 중요한 서비스라면 빌드 타임에 처리되는 CSS Modules가 초기 렌더링 비용 면에서 유리합니다. 컴포넌트 상태에 따라 스타일이 매우 동적으로 바뀌는(예: 사용자 설정 테마가 실시간으로 반영되는) 애플리케이션이라면 CSS-in-JS의 유연성이 더 큰 이득을 줄 수 있습니다. 다만 런타임 CSS-in-JS는 SSR 스트리밍과 결합할 때 스타일 주입 타이밍을 신경 써야 하는 비용이 있습니다.

Q: 디자인팀이 컴포넌트마다 미세하게 다른 간격 값을 요청합니다. 이게 왜 CSS 아키텍처 문제와 연결되나요?

모범 답변: 임의의 값(예: margin: 13px)이 계속 늘어나면 일관성이 깨지고, 나중에 디자인이 바뀔 때 수십 곳을 하나씩 찾아 고쳐야 합니다. 이건 디자인 토큰(간격 스케일: 4px/8px/16px/24px 등)을 도입해서 해결할 문제입니다. 디자이너에게는 "임의 값을 막는다"가 아니라 "토큰 스케일 안에서 선택하면 나중에 전체 리브랜딩도 값 하나만 바꾸면 전파된다"는 이득으로 설득합니다(Day 11에서 심화).

Q: 접근성 감사에서 시맨틱 마크업 위반이 수백 건 나왔습니다. 우선순위를 어떻게 정하시겠어요?

모범 답변: 모든 위반을 동일하게 취급하지 않습니다. 키보드로 아예 접근 불가능한 핵심 플로우(로그인, 결제, 폼 제출)를 최우선으로 두고, 그다음으로 스크린리더 사용자가 콘텐츠를 이해할 수 없게 만드는 문제(누락된 대체 텍스트, 잘못된 헤딩 위계), 마지막으로 개선하면 좋지만 치명적이지 않은 항목 순으로 우선순위를 정합니다. 또한 개별 위반을 하나씩 고치는 대신, 공통 컴포넌트(버튼, 폼 필드) 단에서 고치면 수십~수백 건이 한 번에 해결되는 지점을 먼저 찾습니다.

Q (비기술적): 기획팀이 "일단 !important로 빨리 고쳐달라"고 요청합니다. 마감이 촉박합니다. 어떻게 하시겠어요?

모범 답변: 마감이 촉박한 현실을 무시하지 않습니다. 이번 건은 !important로 처리하되, 이것이 "임시 처방"이라는 것을 팀 트래킹 도구에 기술 부채로 명시적으로 기록하고, 다음 스프린트에 근본 원인(전역 선택자 충돌)을 고치는 작업을 별도 티켓으로 등록합니다. 즉, 단기 압박에 굴복해 조용히 넘어가는 것이 아니라, "지금은 이렇게 처리하지만 이 빚은 반드시 갚는다"는 것을 가시화하는 것이 시니어의 역할입니다.

12. 흔한 오해

오해 1

"ID 선택자를 쓰면 성능이 더 좋다"

과거에는 브라우저의 선택자 매칭이 오른쪽에서 왼쪽으로 진행된다는 이유로 이런 통념이 있었지만, 현대 브라우저 엔진에서 선택자 종류에 따른 성능 차이는 실무에서 체감할 수준이 아닙니다. ID 선택자를 피해야 하는 진짜 이유는 성능이 아니라 Specificity가 지나치게 높아 재사용성과 오버라이드 가능성을 해친다는 유지보수성 문제입니다.

오해 2

"시맨틱 태그는 스크린리더 사용자를 위한 것일 뿐, 대부분의 사용자에게는 무관하다"

시맨틱 마크업은 SEO 크롤러, 브라우저의 자동 완성/번역 기능, 음성 비서, 그리고 키보드만으로 탐색하는 사용자(운동 장애가 아니어도 마우스가 고장 났거나 노트북 트랙패드가 불편한 상황 포함) 등 훨씬 넓은 범위에 영향을 줍니다. "일부 사용자를 위한 부가 기능"이 아니라 "다양한 입력·출력 방식에서 동일하게 작동하는 견고한 마크업"이라는 관점이 정확합니다.

오해 3

"CSS 방법론(BEM, CSS-in-JS 등)을 도입하면 Specificity 문제가 자동으로 해결된다"

도구는 문제를 만들기 어렵게 만들 뿐, 팀이 컨벤션을 지키지 않으면 어떤 도구를 써도 다시 얽힐 수 있습니다. 예를 들어 CSS-in-JS를 쓰면서도 전역 선택자를 계속 추가하면 같은 문제가 재발합니다. 도구 도입과 함께 코드 리뷰·Lint 같은 강제 장치가 있어야 실제로 효과가 있습니다.

오해 4

"크로스 브라우저 이슈는 마지막에 QA 단계에서 발견해서 고치면 된다"

레이아웃 구조 자체가 특정 브라우저의 제약을 무시하고 설계되면(예: 최신 CSS 기능에만 의존), QA 단계에서 발견했을 때는 이미 컴포넌트 구조를 다시 짜야 하는 큰 리스크가 됩니다. Can I Use 같은 호환성 확인은 설계 단계에서 먼저 하는 것이 비용이 훨씬 적습니다.

오해 5

"레거시 CSS가 엉망이면 처음부터 다시 짜는 게 가장 빠르다"

전면 재작성은 매력적으로 보이지만, 기존 CSS에 암묵적으로 의존하는 수백 개의 엣지 케이스(특정 브라우저 대응, 특정 고객사 요청으로 추가된 예외)를 놓치기 쉬워 재작성 이후 회귀 버그가 대량 발생하는 경우가 많습니다. 대부분의 시니어는 점진적 교체(strangler pattern)를 선호합니다.
