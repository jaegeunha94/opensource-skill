1. 핵심 면접 질문

Primary Question

"전역 상태 관리 라이브러리(Redux 등)에 서버에서 받아온 데이터까지 모두 넣고 있는 코드베이스를 물려받았습니다. 어떤 기준으로 리팩터링하시겠어요?"

Variant 1

"여러 화면에서 같은 서버 데이터(예: 사용자 정보)를 각자 fetch해서 따로 들고 있다가, 한쪽에서 수정한 내용이 다른 화면에 반영되지 않는 정합성 버그가 반복됩니다. 어떻게 접근하시겠어요?"

Variant 2

"장바구니 개수처럼 여러 컴포넌트에서 공유하는 값이 있습니다. 이걸 전역 상태로 올릴지, URL에 둘지, 서버 상태 캐시에 파생시킬지 어떻게 판단하시나요?"

2. 왜 면접에서 묻는가

이 질문은 "Redux나 Zustand 문법을 아는가"가 아니라, 서로 성질이 다른 두 종류의 상태를 구분하지 못해 발생하는 아키텍처 부채를 진단하고 되돌릴 수 있는가를 테스트합니다. 주니어는 "상태는 다 전역 store에 넣으면 편하다"는 식으로 접근하지만, 시니어는 상태마다 누가 소유하고, 얼마나 자주 바뀌고, 진실의 출처(source of truth)가 어디인지가 다르다는 것을 전제로 아키텍처를 설계합니다.

면접관이 이 질문으로 평가하는 것:

  • 클라이언트 상태와 서버 상태를 구조적으로 구분하는가 — 모달이 열려있는지 같은 값과, 서버에 있는 주문 목록 같은 값이 "다루는 방식 자체가 달라야 한다"는 것을 아는가.
  • 정합성 버그의 근본 원인을 진단할 수 있는가 — "여러 화면이 서버 데이터의 서로 다른 사본(copy)을 들고 있어서" 발생한다는 것을 구조적으로 설명할 수 있는가.
  • 파생 상태(Derived State)를 알아보고 제거할 수 있는가 — 이미 있는 데이터로 계산 가능한 값을 별도 상태로 중복 저장해서 동기화 버그를 만드는 패턴을 식별할 수 있는가.
  • 상태의 "위치"를 설계 결정으로 다루는가 — 모든 상태를 최상위로 올리는 것이 아니라, 사용하는 컴포넌트에 가깝게 두는 것(colocation)이 기본값이어야 한다는 것을 아는가.
  • 리팩터링을 점진적으로 안전하게 수행할 수 있는가 — "다 갈아엎겠습니다"가 아니라, 리스크를 낮추며 단계적으로 마이그레이션하는 현실적 계획을 제시할 수 있는가.

주니어~중니어에서 자주 나오는 답변 패턴: "상태 관리 라이브러리를 하나 더 도입하겠습니다"처럼 도구 교체로 문제를 해결하려 하거나, "그냥 다 useContext로 감싸겠습니다"처럼 근본 원인(상태의 성질 구분 부재)을 짚지 않고 구현 수단만 바꾸는 경우입니다. 시니어는 "이 값이 서버가 소유한 데이터의 사본인가, 아니면 이 화면에서만 의미 있는 UI 상태인가"부터 구분합니다.

3. 전제 개념 — 쉬운 한국어로

클라이언트 상태와 서버 상태는 왜 근본적으로 다른가

클라이언트 상태(Client State)는 이 화면, 이 브라우저 탭 안에서만 의미가 있는 값입니다. 모달이 열려있는지, 폼에 무엇을 입력 중인지, 어떤 탭이 선택되어 있는지가 그 예입니다. 이 값의 "진실"은 오직 클라이언트 자신에게만 있고, 서버는 이 값의 존재를 알지도 못합니다. 반면 서버 상태(Server State)는 진짜 소유자가 서버(그리고 그 뒤의 데이터베이스)이고, 클라이언트는 그것의 사본(copy)을 잠깐 빌려와 보여주는 값입니다. 사용자 프로필, 주문 목록, 재고 수량이 그 예입니다. 비유하면 클라이언트 상태는 "내 방에 있는 내 물건"이고, 서버 상태는 "도서관에서 빌려온 책"입니다. 빌려온 책은 언제든 도서관(서버) 쪽에서 다른 사람이 먼저 대출해가거나 내용이 바뀔 수 있으므로, 내가 들고 있는 사본이 "오래된 것(stale)"일 수 있다는 것을 항상 전제해야 합니다. 이 차이를 무시하고 두 종류를 같은 방식(하나의 전역 store)으로 다루면, 서버 상태 특유의 문제(비동기 로딩, 실패, 재검증, 여러 화면 간 공유)를 클라이언트 상태 도구로 억지로 풀게 되어 로딩/에러/캐시 로직을 직접 다 구현하는 부담이 생깁니다.

진실의 출처(Single Source of Truth)란 무엇인가

같은 데이터를 여러 곳에 복사해서 저장하지 않고, 단 하나의 장소만을 "진짜 값"으로 취급하는 원칙입니다. 예를 들어 사용자의 이메일 주소를 A 컴포넌트의 로컬 상태에도 저장하고, 전역 store에도 저장하고, 폼 상태에도 저장하면, 세 곳 중 하나가 갱신될 때 나머지 두 곳은 갱신되지 않을 수 있습니다. 이런 상황에서 "어느 값이 맞는 값이지?"라는 질문에 답할 수 없게 되는 것이 정합성 버그의 근본 원인입니다. 서버 상태에서는 보통 클라이언트 캐시(React Query 등)의 캐시 항목 하나를 그 데이터의 유일한 진실로 삼고, 화면들은 그 캐시를 구독(subscribe)해서 읽기만 하는 구조로 설계합니다.

파생 상태(Derived State)란 무엇이고 왜 위험한가

이미 가지고 있는 다른 상태로부터 계산해서 얻을 수 있는 값을 별도의 상태로 다시 저장하는 것을 말합니다. 예를 들어 장바구니 아이템 배열이 있는데, "장바구니 개수"를 별도의 useState로 만들어 아이템이 추가/삭제될 때마다 수동으로 개수를 갱신하는 코드가 파생 상태의 전형입니다. 이 개수는 items.length로 언제든 계산 가능한 값인데 별도 상태로 만들면, 아이템 배열은 갱신했지만 개수 갱신 코드를 빠뜨리는 순간 두 값이 어긋나는 버그가 생깁니다. 원칙은 "계산으로 얻을 수 있는 값은 상태로 만들지 않고, 렌더링 시점에 계산한다"입니다.

상태 배치(State Colocation)란 무엇인가

상태를 그 상태를 실제로 사용하는 컴포넌트에 최대한 가깝게 둔다는 원칙입니다. "나중에 다른 곳에서도 쓸 수도 있으니 일단 최상위나 전역 store에 올려두자"는 습관이 반대되는 패턴입니다. 상태를 필요 이상으로 상위(또는 전역)로 올리면, 그 상태가 바뀔 때마다 원래는 영향받지 않아도 될 하위 트리까지 리렌더링을 유발하거나, 상태를 사용하지 않는 코드가 그 상태의 존재를 알아야 하는 결합(coupling)이 생깁니다. 실제로 필요해지는 시점에 상위로 끌어올리는 것(lift up)이 처음부터 전역화하는 것보다 대체로 더 안전한 기본 전략입니다.

캐시 무효화(Cache Invalidation)와 정합성(Consistency)은 왜 어려운가

캐시 무효화는 "들고 있는 사본이 이제 오래되었으니 버리고 다시 가져와야 한다"는 신호를 정확한 시점에 보내는 것입니다. 이것이 어려운 이유는, 데이터를 변경한 주체(예: 이 화면에서의 mutation)는 알지만, 그 데이터를 참조하고 있는 다른 화면들은 자신이 들고 있는 사본이 오래되었다는 사실을 스스로 알 방법이 없기 때문입니다. 전산학자 Phil Karlton의 유명한 말처럼 "캐시 무효화는 컴퓨터 과학에서 가장 어려운 두 가지 문제 중 하나"인데, 프론트엔드에서는 이것이 "어떤 화면들이 이 데이터를 구독하고 있는지 추적하고, 변경 시점에 정확히 그들에게만 갱신 신호를 보내는 문제"로 구체화됩니다.

4. 클라이언트 상태 vs 서버 상태 분리 기준

세 갈래 상태 분류표

축클라이언트 상태(UI)서버 상태URL 상태
소유자현재 브라우저 탭서버/DB브라우저 주소창(공유 가능)
예시모달 열림, 폼 입력값, 아코디언 펼침주문 목록, 사용자 프로필, 재고검색 필터, 페이지 번호, 정렬 기준
비동기성동기적, 즉시 확정비동기, 로딩/에러 상태 필요동기적(네비게이션은 별도)
신선도(staleness)개념 자체가 없음(항상 최신)항상 "오래됐을 수 있음"을 전제개념 자체가 없음
공유 범위보통 단일 컴포넌트 트리여러 화면/여러 세션이 같은 데이터를 구독새로고침·뒤로가기·공유 링크로 재현되어야 함
적합한 도구useState, useReducer, 지역 ContextReact Query, SWR, RTK Query 등 서버 상태 전용 캐시URLSearchParams, 라우터의 쿼리 파라미터

실무에서 가장 흔한 실수는 서버 상태와 URL 상태를 전통적인 클라이언트 상태 도구(Redux의 단일 store 등)에 밀어 넣는 것입니다. 그 결과 로딩/에러/재검증/캐시 무효화를 전부 직접 구현하게 되고, 새로고침하면 필터가 초기화되는(URL에 반영되지 않아서) 문제가 생깁니다.

진단 질문 — 이 값은 어디에 둬야 하는가

  • 1
    "서버를 새로고침해도(다른 세션에서 봐도) 이 값이 존재해야 하는가?"

    그렇다면 서버 상태입니다. 클라이언트 상태 도구에 넣지 않고 서버 상태 캐시(React Query 등)로 관리합니다.

  • 2
    "URL을 복사해서 다른 사람에게 보냈을 때 같은 화면이 재현되어야 하는가?"

    그렇다면 URL 상태입니다. 검색어, 필터, 페이지네이션, 정렬 기준이 대표적입니다.

  • 3
    "이미 있는 다른 상태로부터 계산해서 얻을 수 있는 값인가?"

    그렇다면 상태로 만들지 않고 렌더링 중에 파생시킵니다(useMemo는 성능 최적화일 뿐 정합성 문제를 만들지 않기 위한 것이 아닙니다 — 계산 자체를 렌더 중에 하는 것이 핵심입니다).

  • 4
    "이 값을 쓰는 컴포넌트가 하나뿐이거나 인접한 부모-자식뿐인가?"

    그렇다면 지역 상태로 colocate합니다. 여러 개의 서로 관련 없는 트리가 동시에 필요로 할 때만 전역화를 고려합니다.

5. 캐시 무효화 전략

무효화 시점을 결정하는 네 가지 전략

전략동작적합한 상황
Mutation 기반 무효화데이터를 변경하는 요청이 성공하면 관련 쿼리 키를 즉시 무효화사용자 자신의 행동으로 발생하는 대부분의 변경(글 작성, 좋아요)
시간 기반 재검증(stale time)일정 시간이 지나면 캐시를 오래된 것으로 간주하고 다음 접근 시 재요청실시간성이 중요하지 않은 목록/설정 데이터
포커스/재연결 시 재검증탭이 다시 활성화되거나 네트워크가 재연결될 때 자동 재요청사용자가 자리를 비운 사이 바뀌었을 수 있는 대시보드류 데이터
서버 푸시 기반 무효화(WebSocket/SSE)서버가 변경 이벤트를 실시간으로 클라이언트에 알려 즉시 무효화다른 사용자의 변경을 즉시 반영해야 하는 협업 도구, 채팅

네 전략은 배타적이지 않고 계층적으로 조합됩니다. 예를 들어 기본적으로 stale time을 짧게 두고, mutation 시 즉시 무효화하며, 협업 기능이 있는 화면에서만 웹소켓 기반 무효화를 추가하는 식입니다.

쿼리 키 설계가 무효화 범위를 결정한다

React Query류의 라이브러리에서 쿼리 키(query key)는 캐시의 "주소"이자 무효화의 단위입니다. ['orders', userId]처럼 계층적으로 설계하면, ['orders'] 전체를 무효화할 수도 있고 특정 userId의 주문만 무효화할 수도 있습니다. 쿼리 키를 너무 뭉뚱그리면(예: 모든 데이터를 ['data'] 하나로) 작은 변경 하나에도 관련 없는 데이터까지 전부 재요청하게 되고, 반대로 너무 잘게 쪼개면 무효화해야 할 키를 빠짐없이 나열하는 부담이 커집니다.

// 주문 상태 변경 mutation 성공 후
const mutation = useMutation({
  mutationFn: updateOrderStatus,
  onSuccess: (_, variables) => {
    // 이 주문 하나만 정밀 무효화
    queryClient.invalidateQueries({ queryKey: ['orders', variables.orderId] });
    // 목록 화면도 함께 갱신되어야 하므로 목록 키도 무효화
    queryClient.invalidateQueries({ queryKey: ['orders', 'list'] });
  },
});

과소 무효화와 과다 무효화의 트레이드오프

무효화를 빠뜨리면(과소 무효화) 사용자가 오래된 데이터를 계속 보는 버그가 생기고, 반대로 관련 없는 것까지 다 무효화하면(과다 무효화) 불필요한 네트워크 요청이 늘어나 성능이 저하됩니다. 실무에서는 정합성이 사용자에게 직접 보이는 화면(방금 수정한 내용)은 정밀하게, 백그라운드 데이터(잘 안 바뀌는 참조 데이터)는 느슨하게 무효화 전략을 차등 적용합니다.

6. 데이터 정합성 유지 패턴

정규화된 캐시(Normalized Cache)

같은 엔티티(예: 사용자 ID 42)가 여러 API 응답에 중복으로 등장할 때, 엔티티를 ID 기준으로 한 곳에만 저장하고 여러 화면은 그 참조를 공유하게 만드는 패턴입니다. 예를 들어 게시글 목록 API와 게시글 상세 API가 둘 다 작성자 정보를 포함한다면, 정규화된 캐시는 작성자 데이터를 users[42] 하나로 저장하고 목록/상세 양쪽이 이 하나의 참조를 가리키게 합니다. 이렇게 하면 작성자 이름이 바뀌었을 때 한 곳만 갱신해도 그 데이터를 참조하는 모든 화면이 동시에 최신 상태가 됩니다. Apollo Client, Relay가 이 방식을 기본으로 채택하고, React Query는 쿼리 단위 캐시가 기본이지만 필요 시 수동으로 정규화 패턴을 흉내낼 수 있습니다.

낙관적 동시성 제어(Optimistic Concurrency) — 버전/ETag 기반 충돌 감지

두 사용자(또는 두 탭)가 같은 데이터를 동시에 수정하려 할 때, 먼저 읽었던 시점의 버전 정보를 수정 요청에 함께 실어 보내고, 서버는 그 사이 데이터가 바뀌었다면 요청을 거부하는 패턴입니다. 예를 들어 문서를 불러올 때 version: 5를 받았는데, 수정 요청 시점에 서버의 실제 버전이 이미 6으로 바뀌어 있다면 서버는 409 Conflict를 반환하고, 클라이언트는 "다른 곳에서 먼저 수정되었습니다. 최신 내용을 확인하세요"라고 사용자에게 안내합니다. 이 패턴이 없으면 "마지막에 저장을 누른 사람이 이긴다(Last-Write-Wins)"가 암묵적으로 적용되어, 먼저 작업한 사람의 변경 내용이 조용히 사라지는 사고가 발생합니다.

UI 상태와 서버 상태를 같은 store에 섞지 않기

전역 상태 관리 라이브러리(Redux, Zustand)에 서버에서 받아온 데이터를 그대로 저장하고 fetchStart/fetchSuccess/fetchFailure 액션을 손수 구현하는 것은, 서버 상태 전용 라이브러리가 이미 해결한 문제(중복 요청 제거, 캐시 신선도 판단, 자동 재검증, 재시도)를 처음부터 다시 만드는 것과 같습니다. 리팩터링의 방향은 전역 store에는 순수한 클라이언트 UI 상태만 남기고, 서버에서 온 데이터는 서버 상태 캐시 계층으로 옮기는 것입니다. 전역 store가 필요한 클라이언트 상태(예: 로그인한 사용자의 테마 설정, 사이드바 접힘 여부)만 남으면 store 자체의 크기와 복잡도가 크게 줄어드는 경우가 많습니다.

7. 답변 사고 순서

  1. 현재 코드베이스에서 상태를 종류별로 분류합니다. 전역 store 안의 값들을 하나씩 "클라이언트 상태인가, 서버 상태인가, URL 상태인가, 파생 가능한 값인가"로 라벨링합니다.
  2. 정합성 버그가 실제로 발생하는 지점을 구체적으로 찾습니다. "정합성이 깨진다"는 추상적 진단이 아니라, 어떤 두 화면이 같은 데이터의 서로 다른 사본을 들고 있는지 구체적으로 지목합니다.
  3. 서버 상태를 단일 캐시 계층으로 옮기는 목표 아키텍처를 정의합니다. React Query 같은 서버 상태 전용 도구를 도입해, 여러 화면이 같은 쿼리 키를 구독하는 구조로 재설계합니다.
  4. 파생 상태와 불필요한 전역화를 제거합니다. 계산 가능한 값은 상태에서 빼고, 단일 컴포넌트만 쓰는 상태는 지역으로 되돌립니다(colocate).
  5. 점진적 마이그레이션 계획을 제시합니다. 전체 재작성이 아니라, 정합성 버그가 가장 잦은 화면부터 순차적으로 옮기고, 기존 전역 store와 새 서버 상태 캐시가 과도기적으로 공존할 수 있는 경계를 설계합니다.
  6. 재발 방지 기준을 팀에 공유합니다. "서버에서 온 데이터는 전역 store에 넣지 않는다"는 규칙을 코드 리뷰 체크리스트나 린트 규칙으로 명문화합니다.

8. 핵심 답변 예시

면접관: "전역 상태 관리 라이브러리에 서버에서 받아온 데이터까지 모두 넣고 있는 코드베이스를 물려받았습니다. 어떤 기준으로 리팩터링하시겠어요?"

STRONG ANSWER

먼저 전역 store 안의 상태들을 성질별로 분류하는 작업부터 시작합니다. "이 화면을 새로고침해도 서버에 그대로 존재해야 하는 값인가"를 기준으로, 서버 상태(주문 목록, 사용자 프로필 등)와 순수 클라이언트 상태(사이드바 접힘, 선택된 탭 등)를 구분합니다. 실제로 대부분의 레거시 store를 열어보면 fetchStart/fetchSuccess/fetchFailure 액션과 로딩 플래그가 반복되는데, 이는 서버 상태를 클라이언트 상태 도구로 억지로 다루고 있다는 신호입니다.


이렇게 분류한 뒤, 서버 상태는 React Query 같은 서버 상태 전용 캐시 계층으로 옮기는 것을 목표로 삼습니다. 이 도구가 대신 해주는 것 — 중복 요청 제거, 캐시 신선도 판단, mutation 후 무효화, 재시도 — 을 손수 구현하고 있던 코드를 걷어낼 수 있기 때문입니다. 반면 전역 store에는 정말로 여러 화면이 공유해야 하는 순수 클라이언트 상태만 남깁니다.


전체를 한 번에 갈아엎으면 리스크가 크므로, 정합성 버그가 가장 자주 보고되는 화면(예: 사용자 프로필을 여러 곳에서 따로 fetch하는 화면)부터 먼저 서버 상태 캐시로 옮기고, 이 기간 동안 레거시 store와 새 캐시 계층이 공존하는 것을 허용합니다. 마이그레이션이 끝난 화면부터 관련 Redux 액션/리듀서를 제거해 코드베이스를 점진적으로 축소시킵니다.


마지막으로 재발 방지를 위해 "서버 응답을 그대로 전역 store 액션으로 dispatch하지 않는다"는 규칙을 코드 리뷰 체크리스트에 추가하고, 신규 기능은 처음부터 서버 상태 캐시 계층을 쓰도록 팀 컨벤션을 갱신합니다.

이 답변이 좋은 이유:

  • "라이브러리를 바꾸자"가 아니라 상태의 성질을 구분하는 진단 기준부터 제시함
  • 레거시 코드에서 서버 상태 오용을 알아보는 구체적 신호(loading 플래그 반복 패턴)를 제시함
  • 전체 재작성이 아니라 리스크를 낮추는 점진적 마이그레이션 계획을 제시함
  • 재발 방지를 위한 팀 프로세스(리뷰 체크리스트, 컨벤션)까지 언급함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"Redux가 오래된 기술이니 Zustand나 Jotai 같은 최신 상태 관리 라이브러리로 교체하겠습니다."

이 답변이 약한 이유

  • 문제의 근본 원인(클라이언트 상태와 서버 상태를 구분하지 않은 것)을 짚지 않고 도구 교체로 넘어갑니다.
  • 새 도구로 옮겨도 서버 상태를 여전히 같은 방식으로 다루면 로딩/에러/재검증 로직을 또 손수 구현하는 문제가 반복됩니다.
  • "최신이라서"라는 근거는 트레이드오프 분석이 아닙니다.
WEAK ANSWER — 또 다른 패턴

"정합성 버그가 난 화면에 useEffect로 다시 fetch하는 코드를 추가해서 최신화하겠습니다."

이 답변이 약한 이유

  • 근본 원인(여러 화면이 같은 데이터의 서로 다른 사본을 들고 있는 구조)을 해결하지 않고 증상만 화면별로 땜질합니다.
  • 다른 화면에서 같은 문제가 또 발생할 때마다 같은 패치를 반복해야 하는 구조적 부채가 남습니다.
STRONG ANSWER 요소 체크리스트
  • 상태를 클라이언트/서버/URL/파생 상태로 분류하는 진단 기준을 먼저 제시한다
  • 레거시 코드에서 서버 상태 오용을 식별하는 구체적 신호를 든다
  • 서버 상태를 단일 캐시 계층으로 통합하는 목표 아키텍처를 제시한다
  • 전체 재작성이 아니라 리스크를 낮추는 점진적 마이그레이션 순서를 제시한다
  • 재발 방지를 위한 팀 차원의 규칙/컨벤션까지 언급한다

10. Senior-level Follow-up Q&A

Q: 장바구니 아이템 개수처럼 자주 표시되는 값을 매번 items.length로 계산하면 성능에 문제가 생기지 않나요?

모범 답변: items.length 같은 계산은 배열 길이를 읽는 것으로 O(1)에 가까워 성능 문제가 되지 않습니다. 계산 비용이 실제로 큰 경우(대량 데이터 필터링/정렬 등)에는 useMemo로 캐싱하면 되는데, 이때도 useMemo는 "값을 저장하는 새로운 상태"가 아니라 "같은 계산을 반복하지 않기 위한 메모이제이션"이라는 점이 중요합니다. 즉 원본 데이터가 바뀌면 자동으로 재계산되므로 파생 상태를 별도 useState로 만들 때 생기는 동기화 누락 문제가 애초에 발생하지 않습니다. 성능과 정합성은 별개의 축이며, 정합성을 지키면서 성능을 최적화하는 도구가 useMemo입니다.

Q: 마이크로프론트엔드나 여러 팀이 각자 배포하는 아키텍처에서는 서버 상태 캐시를 어떻게 공유하나요?

모범 답변: 완전히 하나의 캐시 인스턴스를 여러 독립 배포 단위가 공유하는 것은 결합도를 높여 마이크로프론트엔드의 독립 배포 이점을 해칩니다. 대신 각 마이크로프론트엔드가 자신의 서버 상태 캐시를 독립적으로 소유하되, 같은 데이터를 여러 팀이 다루는 경우에는 (1) 데이터 소유 팀이 이벤트(커스텀 이벤트, BroadcastChannel API 등)로 변경을 브로드캐스트해 다른 팀의 캐시가 무효화되도록 하거나, (2) 공통 데이터 접근을 캡슐화한 공유 패키지(SDK)를 배포해 쿼리 키 규칙과 무효화 로직을 팀 간에 일관되게 유지하는 방식을 씁니다. 핵심은 캐시 인스턴스를 공유하는 것이 아니라 무효화 신호와 규칙을 공유하는 것입니다.

Q: 낙관적 동시성 제어(버전/ETag 충돌 감지)와 낙관적 업데이트(Optimistic Update, Day 7)는 이름이 비슷한데 어떻게 다른가요?

모범 답변: 둘 다 "낙관적"이라는 단어를 쓰지만 해결하는 문제가 다릅니다. 낙관적 업데이트는 내 요청이 성공할 것이라 가정하고 UI를 먼저 갱신하는 것으로, 대상은 "내 요청과 서버 응답 사이의 지연"입니다. 낙관적 동시성 제어는 다른 사용자와 내가 같은 데이터를 동시에 수정하지 않을 것이라 가정하고 버전 검증 없이 진행하되, 실제로 충돌이 발생하면 그때 감지해서 처리하는 것으로, 대상은 "나와 다른 클라이언트 사이의 동시 수정"입니다. 두 패턴은 함께 쓰일 수 있습니다 — UI는 낙관적으로 먼저 갱신하되, 실제 서버 요청에는 버전 정보를 실어 보내 충돌 시 롤백하는 식입니다.

Q (비기술적): 신규 팀원이 "전역 상태에 다 넣으면 어디서든 접근할 수 있어서 편한데 왜 굳이 나누나요?"라고 묻습니다. 어떻게 설명하시겠어요?

모범 답변: "편함"이 단기적인 개발 속도에서는 맞지만, 상태의 성질을 구분하지 않고 한 곳에 몰아넣으면 장기적으로 세 가지 비용이 늘어난다고 설명합니다. 첫째, 서버 상태 특유의 문제(로딩/에러/재검증/캐시 무효화)를 매번 손으로 구현해야 합니다. 둘째, 전역 상태가 바뀔 때마다 그 값을 구독하는 모든 컴포넌트가 리렌더링 대상이 되어 불필요한 성능 비용이 쌓입니다. 셋째, "이 값이 지금 최신인지"를 코드 전체를 뒤져야 알 수 있게 되어 디버깅 시간이 늘어납니다. 실제 사례(예: 지난 분기 정합성 버그로 인한 장애나 디버깅 시간)를 들어 "편함"과 "장기 유지보수 비용"이 트레이드오프 관계에 있다는 것을 구체적 숫자나 사례로 설득하는 것이 효과적입니다.

11. 흔한 오해

오해 1

"전역 상태 관리 라이브러리는 서버 데이터든 UI 상태든 다 담을 수 있는 범용 저장소다"

Redux/Zustand 같은 도구는 "값을 저장하고 구독한다"는 기능만 제공할 뿐, 서버 상태 특유의 문제(비동기 로딩, 캐시 신선도, 재검증, 재시도)는 스스로 해결해주지 않습니다. 이를 무시하고 서버 데이터를 그대로 넣으면 그 문제들을 전부 직접 구현해야 합니다.

오해 2

"상태는 최대한 위(상위 컴포넌트나 전역)로 올려두는 것이 안전하다"

불필요한 전역화는 상태 변경 시 리렌더링 범위를 넓히고, 상태의 위치와 실제 사용처 사이의 결합을 늘립니다. 원칙은 실제로 여러 트리가 공유해야 하는 시점에만 끌어올리는 것(lift up on demand)입니다.

오해 3

"파생 상태를 별도 state로 만들어 두면 매번 재계산하지 않아 성능에 유리하다"

실제로는 원본 데이터와 파생 값 사이의 동기화를 수동으로 관리해야 하는 부담과 버그 위험이 늘어납니다. 계산 비용이 실제로 문제될 때만 useMemo로 메모이제이션하고, 기본적으로는 렌더링 중 계산이 더 안전합니다.

오해 4

"React Query 같은 서버 상태 라이브러리를 도입하면 전역 상태 관리 라이브러리는 완전히 필요 없어진다"

서버 상태를 담당하는 도구와 순수 클라이언트 상태를 담당하는 도구는 서로 다른 문제를 풉니다. 로그인 사용자의 테마 설정, 여러 화면이 공유하는 위저드 진행 단계 같은 순수 클라이언트 상태는 여전히 전역 상태 도구(또는 Context)가 적합할 수 있습니다. 핵심은 "무엇을 어디에 둘지 구분"이지 "한쪽 도구를 완전히 없애는 것"이 아닙니다.

오해 5

"캐시 무효화는 mutation 성공 시 관련 쿼리를 무효화하기만 하면 끝이다"

사용자 자신의 행동으로 인한 변경은 mutation 기반 무효화로 충분하지만, 다른 사용자의 변경이나 백그라운드에서 바뀌는 데이터는 시간 기반 재검증이나 서버 푸시 없이는 감지할 수 없습니다. 요구되는 실시간성 수준에 따라 전략을 조합해야 합니다.

12. 자기 점검 Rubric

다음 항목을 말로 설명할 수 있으면 Day 8을 이해한 것입니다.

☐
클라이언트 상태, 서버 상태, URL 상태의 차이를 소유자·비동기성·신선도 기준으로 구분해서 설명할 수 있다.
☐
Single Source of Truth 원칙과, 같은 데이터를 여러 곳에 복사해서 저장했을 때 생기는 정합성 버그를 예로 들 수 있다.
☐
파생 상태(Derived State)가 무엇이고 왜 별도 상태로 만들지 않는 것이 원칙인지 설명할 수 있다.
☐
상태 배치(Colocation) 원칙과, 상태를 불필요하게 전역화했을 때의 리렌더링/결합도 비용을 설명할 수 있다.
☐
캐시 무효화의 네 가지 전략(mutation 기반, 시간 기반, 포커스/재연결, 서버 푸시)을 상황별로 구분해서 제시할 수 있다.
☐
쿼리 키 설계가 무효화 범위를 어떻게 결정하는지, 과소/과다 무효화의 트레이드오프를 설명할 수 있다.
☐
정규화된 캐시가 무엇이고, 왜 여러 화면 간 데이터 동기화 문제를 구조적으로 해결하는지 설명할 수 있다.
☐
낙관적 동시성 제어(버전/ETag 충돌 감지)와 Last-Write-Wins의 차이, 그리고 각각이 적합한 상황을 설명할 수 있다.
☐
레거시 전역 store에서 서버 상태 오용을 식별하는 구체적 신호(loading 플래그 반복 패턴 등)를 말할 수 있다.
☐
전체 재작성이 아닌 점진적 마이그레이션 계획을 리스크 관리 관점에서 제시할 수 있다.
