1. 핵심 면접 질문

Primary Question

"프로덕션 대시보드 페이지가 특정 고객사에서만 가끔 느리다는 리포트를 받았습니다. 리액트로 만들어진 페이지입니다. 프론트엔드 관점에서 어떻게 접근하시겠습니까?"

Variant 1

"사용자가 버튼을 클릭한 순간부터 화면이 실제로 바뀌기까지, 브라우저 안에서 어떤 일이 순서대로 일어나는지 설명해주세요."

Variant 2

"React를 쓰면 이런 브라우저 내부 동작을 몰라도 되는 것 아닌가요? 왜 알아야 한다고 생각하세요?"

2. 왜 면접에서 묻는가

이 질문은 "컴포넌트를 만들 줄 아는가"가 아니라 "브라우저라는 플랫폼 위에서 무슨 일이 벌어지는지 이해하고 있는가"를 테스트합니다. React 문서에 나온 패턴만 따라 해본 사람과, 그 패턴이 실제로 브라우저에게 어떤 작업을 시키는지(DOM 변경, 스타일 재계산, 레이아웃, 페인트)까지 그릴 수 있는 사람을 구분하는 질문입니다.

면접관이 이 질문으로 평가하는 것:

  • 플랫폼 기반 사고 — 프레임워크가 감춘 아래층(브라우저 렌더링, 네트워크, 메인 스레드)을 인지하고 있는가.
  • 모호한 문제를 좁혀나가는 절차 — "느려요", "가끔 그래요" 같은 애매한 리포트를 받았을 때, 바로 코드를 고치러 뛰어들지 않고 원인을 계층 단위로 좁혀가는 습관이 있는가.
  • 도구를 직접 다루는 능력 — DevTools Performance/Network 패널로 실제로 증상을 재현하고 수치를 확인하는가, 아니면 감으로 추측하는가.
  • 트레이드오프 언어 — "빠르게 만들자"가 아니라 "이 최적화는 어떤 비용(복잡도, 개발 속도, 접근성)을 치르는가"를 함께 말하는가.
  • 10년차다운 책임 범위 — 자신이 짠 컴포넌트 코드뿐 아니라, 그 코드가 배포되는 전체 프론트엔드 시스템(번들, 서드파티 스크립트, 캐시, 배포 파이프라인)까지 책임지는 시야가 있는가.

주니어~중니어에서 자주 나오는 답변 패턴: "메모이제이션을 추가하겠습니다" 또는 "이미지를 최적화하겠습니다." 이런 답변은 원인을 특정하지 않은 채 흔한 처방을 던지는 것입니다. 특정 고객사에서만 느리다는 단서를 무시하면 틀린 처방이 될 수 있습니다(예: 그 고객사만 데이터가 커서 느린 것이라면 메모이제이션은 효과가 없습니다). 시니어는 원인을 계층 단위로 좁힌 뒤 조치합니다.

3. 전제 개념 — 쉬운 한국어로

브라우저는 무엇으로 이루어져 있는가

브라우저는 하나의 덩어리가 아니라 여러 엔진이 협업하는 프로그램입니다. 주방에 비유하면, 네트워크 스택은 재료를 배달받는 배송 담당, 렌더링 엔진(예: Blink, WebKit)은 재료로 요리(화면)를 만드는 조리사, JavaScript 엔진(예: V8)은 레시피대로 계산과 판단을 수행하는 부주방장입니다. 이 셋이 각자 다른 속도와 다른 실패 모드를 가지고 있다는 것이 이번 레슨의 핵심 전제입니다.

React 같은 프레임워크는 이 주방 위에 놓인 "레시피 자동화 도구"입니다. 도구가 편리해도 재료 배송이 늦거나 조리사가 바쁘면 요리는 늦게 나옵니다. 프레임워크를 안다고 주방 자체를 아는 것은 아닙니다.

DOM과 CSSOM — 브라우저가 이해하는 두 개의 설계도

브라우저는 HTML을 읽어 DOM(Document Object Model)이라는 요소들의 트리 구조로 바꾸고, CSS를 읽어 CSSOM(CSS Object Model)이라는 스타일 규칙의 트리로 바꿉니다. 편지에 비유하면 DOM은 "편지에 무슨 내용이 있는가(문단, 제목, 목록)"이고 CSSOM은 "그 내용을 어떤 서체·색·간격으로 보여줄 것인가"입니다. 이 둘이 합쳐져야 비로소 "무엇을 어떻게 그릴지"가 결정됩니다.

화면에 그려지기까지의 다섯 단계 (개요)

DOM과 CSSOM이 준비된 후, 브라우저는 대략 다음 순서로 실제 픽셀을 만듭니다:

  1. Style(스타일 계산) — DOM의 각 요소에 어떤 CSS 규칙이 최종 적용되는지 계산 (Cascade, Specificity 반영)
  2. Layout(레이아웃, 흔히 Reflow) — 각 요소가 화면의 어디에, 얼마나 큰 크기로 위치할지 계산
  3. Paint(페인트) — 텍스트, 색, 그림자, 테두리 등을 실제 픽셀로 그리는 단계
  4. Composite(컴포지트) — 여러 레이어로 나뉘어 그려진 결과를 GPU가 하나의 화면으로 합성

이 단계들 각각이 시간을 소모하고, 각각 다른 이유로 느려질 수 있습니다. "화면이 버벅인다"는 리포트를 받았을 때, 이 단계 중 어디서 시간이 소모되는지 구분하는 것이 프론트엔드 성능 진단의 출발점입니다. (각 단계의 구체적인 비용과 최적화는 Day 3에서 심화합니다.)

메인 스레드 — 왜 하나의 길에서 모든 일이 줄을 서는가

브라우저의 JavaScript 실행, 스타일 계산, 레이아웃, 대부분의 페인트 작업은 기본적으로 같은 스레드(메인 스레드)에서 순서대로 처리됩니다. 계산대가 하나뿐인 마트를 떠올리면 됩니다 — 계산원이 한 명이면, 한 손님의 계산이 오래 걸리면 뒤에 줄 선 손님(사용자 클릭, 스크롤, 화면 갱신)은 모두 기다려야 합니다. 무거운 JavaScript 연산이 실행되는 동안 사용자의 클릭에 화면이 반응하지 않는 "멈춤" 현상이 바로 이 때문에 발생합니다. (메인 스레드와 이벤트 루프의 세부 동작은 Day 4에서 다룹니다.)

왜 "전체 경로"를 알아야 하는가

초보적인 사고 모델은 "컴포넌트 코드 → 화면"이라는 2단계 그림입니다. 하지만 실제로 사용자의 클릭이 화면 변화로 이어지기까지는 이벤트 처리, 프레임워크의 상태 갱신, 브라우저의 렌더링 파이프라인, 경우에 따라 네트워크 재요청까지 여러 구간을 거칩니다. 택배에 비유하면, 주문(클릭) 후 물건(픽셀)이 도착하기까지 물류센터·배송기사 같은 여러 중개 단계가 있는 것과 같습니다. "느려요"라는 신고를 받았을 때 어느 구간이 병목인지 알아야 정확한 처방을 내릴 수 있습니다.

4. 사용자 상호작용 → 화면 갱신 전체 지도

실무에서 "화면이 바뀌는 사건" 하나는 보통 다음 계층을 순서대로(또는 일부를 생략하며) 통과합니다. 이 표는 이후 Day 2~6에서 각 계층을 깊게 다룰 때의 지도 역할을 합니다.

계층 대표 역할 대표 기술/개념 흔한 실패·병목 신호
Network 초기 HTML/CSS/JS/이미지 로드, API 응답 수신 HTTP 캐시, CDN, 번들 크기 느린 TTFB, 큰 번들, render-blocking 리소스
Parse / Construct HTML → DOM, CSS → CSSOM 변환 HTML Parser, CSS Parser 인라인 <script>로 인한 파싱 중단, 거대한 CSS
JS Runtime / Event Loop 이벤트 핸들링, 상태 갱신 로직 실행 Call Stack, Task/Microtask Queue 메인 스레드 블로킹, 긴 Task로 인한 입력 지연(INP 저하)
Framework Layer 가상 DOM diff, 리렌더링 범위 결정, 배치 업데이트 React reconciliation, 훅 의존성 불필요한 리렌더링 전파, 과도한 컴포넌트 트리 재계산
Style Recalc 변경된 DOM에 CSS 규칙 재적용 Cascade, Specificity, Selector 비용 복잡한 선택자, 광범위한 클래스 토글
Layout (Reflow) 요소의 위치·크기 재계산 Box Model, Flow Layout Layout Thrashing(읽기·쓰기 반복), 큰 서브트리 영향
Paint / Composite 픽셀 생성 및 GPU 레이어 합성 Layer 승격, transform/opacity 과도한 페인트 영역, 레이어 폭증으로 인한 GPU 부담

이 지도가 실무에서 중요한 이유

"화면이 느리다/버벅인다"는 보고를 받았을 때, 위 7개 계층 중 어디인지 특정하지 않으면 엉뚱한 곳을 고치게 됩니다. 예를 들어 실제 병목이 Style Recalc/Layout 구간(복잡한 CSS 선택자와 잦은 클래스 토글)인데 API 응답 속도만 개선하면 사용자 체감은 거의 바뀌지 않습니다. 이 경로 지도를 항상 먼저 그리는 것이 시니어와 주니어를 가르는 습관입니다.

5. 6단계 프론트엔드 문제 진단 프레임워크

시니어가 "느려요/이상해요" 류의 모호한 프론트엔드 리포트를 받았을 때 실제로 따르는 순서입니다. 이후 커리큘럼(Day 2~17)의 모든 주제는 이 프레임워크의 한 단계를 깊게 파는 것과 같습니다.

  • 1
    증상을 측정 가능한 사실로 재정의 (Symptom → Metric)

    "가끔 느리다"는 주관적입니다. 어떤 지표(LCP, INP, 특정 인터랙션의 응답 시간)가, 어떤 조건(특정 브라우저·기기·네트워크·데이터 규모)에서, 얼마나 자주 나타나는지 확인합니다.

    질문 예시: "느리다는 게 첫 로딩인가요, 클릭 후 반응인가요? 특정 고객사에서만 그런가요? 그 고객사의 데이터 규모가 다른가요?"

  • 2
    경로 재현 (Path Reproduction)

    Chrome DevTools Performance 패널로 실제 인터랙션을 녹화해 플레임 차트를 확인합니다. Network 패널로 요청 크기·순서·타이밍을 확인하고, 네트워크/CPU Throttling으로 저사양 환경을 재현합니다.

    // Performance 탭 사용 흐름 (개념)
1. Record 시작
2. 문제 인터랙션 수행 (예: 특정 버튼 클릭)
3. Record 중지 → Main 트랙에서 Long Task(빨간 삼각형) 확인
4. Bottom-Up / Call Tree로 어떤 함수가 시간을 소모했는지 추적
  • 3
    병목 계층 특정 (Which Layer)

    플레임 차트에서 시간이 어디 몰려 있는지로 계층을 특정합니다. Scripting(노란색)이 크면 JS/프레임워크 로직, Rendering(보라색)이 크면 Style/Layout, Painting(초록색)이 크면 Paint/Composite, 막대 시작 전 여백이 크면 Network입니다.

  • 4
    계층 간 상관 확인 (Cross-layer Correlation)

    RUM(실사용자 모니터링) 데이터로 이 증상이 특정 브라우저/지역/디바이스/고객사 세그먼트에 몰려 있는지 확인합니다. 특정 고객사에서만 발생한다면 데이터 규모, 특정 기능 사용 여부 등 그 세그먼트만의 조건을 의심합니다.

  • 5
    격리 재현 (Layer Isolation)

    의심되는 원인을 하나씩 꺼서 재현되는지 확인합니다. 예: 특정 서드파티 스크립트를 차단하고 재현 — 사라지면 그 스크립트가 원인, 여전하면 자체 코드 문제입니다. React DevTools Profiler로 어떤 컴포넌트가 왜 리렌더링됐는지 확인하는 것도 이 단계입니다.

  • 6
    근본 원인 확정, 수정, 회귀 방지 (Root Cause, Fix & Guardrail)

    원인을 확정한 뒤 수정하고, 동일한 재현 시나리오로 다시 확인합니다. 더 나아가 성능 예산(Day 6)과 CI 게이트로 같은 문제가 재발하지 않도록 자동 감시를 추가합니다. "메모이제이션을 추가했다"처럼 증상만 가리는 조치가 아니라 원인 자체를 제거했는지, 그리고 재발 방지 장치를 뒀는지가 핵심입니다.

6. 증상별 원인 계층 분류표

증상만 보고도 어느 계층 문제일 확률이 높은지 좁힐 수 있는 표입니다. 각 항목은 이후 Day에서 더 깊게 다룹니다.

증상 의미 1차 의심 계층 흔한 원인
느린 LCP 가장 큰 콘텐츠가 화면에 그려지기까지 오래 걸림 Network / Parse 느린 서버 응답, render-blocking CSS/JS, 최적화 안 된 이미지 (Day 6 심화)
높은 CLS 레이아웃이 예고 없이 이동 Layout 크기 미지정 이미지/광고, 폰트 로딩 후 레이아웃 재배치(FOIT/FOUT) (Day 6 심화)
긴 INP / 클릭 후 멈춤 사용자 입력에 화면이 늦게 반응 JS Runtime / Framework 메인 스레드를 오래 점유하는 Long Task, 과도한 리렌더링 (Day 4, 9 심화)
스크롤 버벅임 스크롤 중 프레임 드랍 Layout / Paint / Composite Layout Thrashing, 레이어 승격 없이 무거운 페인트 반복 (Day 3 심화)
시간이 갈수록 느려짐, 탭 다운 메모리 사용량이 계속 증가 JS Runtime 이벤트 리스너 미정리, 클로저가 큰 객체 참조 유지 (Day 4 심화)
특정 고객사/데이터 규모에서만 느림 데이터량에 비례해 악화 Framework / Layout 가상화(virtualization) 없는 대량 리스트 렌더링, O(n²) 렌더링 로직 (Day 9 심화)

7. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 됩니다.

  1. 증상을 구체적인 사실로 바꿉니다. 어떤 지표/인터랙션이, 어떤 조건(브라우저·기기·데이터 규모·고객사)에서 느린지 확인합니다.
  2. DevTools로 재현합니다. Performance/Network 패널로 플레임 차트와 요청 타이밍을 직접 확인합니다.
  3. 병목 계층을 특정합니다. Scripting/Rendering/Painting/Network 중 시간이 몰린 구간을 찾습니다.
  4. 세그먼트 상관을 확인합니다. RUM 데이터로 특정 사용자군에서만 나타나는지 확인해 원인 후보를 좁힙니다.
  5. 의심 원인을 격리해 재현합니다. 서드파티 스크립트 차단, Profiler로 리렌더링 원인 추적 등으로 원인을 확정합니다.
  6. 근본 원인을 수정하고, 성능 예산 같은 회귀 방지 장치로 재발을 막습니다.

8. 핵심 답변 예시

면접관: "프로덕션 대시보드 페이지가 특정 고객사에서만 가끔 느리다는 리포트를 받았습니다. 리액트로 만들어진 페이지입니다. 어떻게 접근하시겠습니까?"

STRONG ANSWER

먼저 증상을 구체화합니다. "느리다"가 첫 로딩(LCP)인지, 특정 버튼 클릭 후 반응(INP)인지, 스크롤 버벅임인지 확인하고, 그 고객사와 다른 고객사의 차이(데이터 규모, 사용하는 기능, 브라우저·기기)를 먼저 파악합니다. 특정 고객사에서만 발생한다는 단서는 "그 고객사만의 조건"이 원인일 가능성이 높다는 신호이기 때문에, 무작정 전체 코드를 최적화하기 전에 이 단서부터 좁힙니다.


그다음 해당 고객사와 비슷한 데이터 규모로 로컬에서 재현하면서 Chrome DevTools Performance 패널로 녹화합니다. 플레임 차트에서 Long Task가 어디서 발생하는지, Scripting인지 Rendering(Layout/Style)인지 구분합니다. 만약 리스트나 테이블 컴포넌트에서 Scripting 시간이 데이터 개수에 비례해 늘어난다면, 리렌더링 범위나 알고리즘 복잡도를 의심합니다.


React DevTools Profiler로 어떤 컴포넌트가 왜 리렌더링되는지 확인합니다. 예를 들어 대시보드 상단의 필터 상태가 바뀔 때마다 전체 테이블이 리렌더링되고 있고, 그 테이블 행 개수가 이 고객사에서만 수천 건이라면, "리렌더링 범위 문제"와 "가상화(virtualization) 부재"가 결합된 것으로 원인이 좁혀집니다.


원인 후보가 좁혀지면, 실제로 가상화를 임시로 적용하거나 메모이제이션 경계를 좁혀서 같은 데이터로 다시 측정해 개선 여부를 확인합니다. 재현되지 않으면 다른 계층(네트워크나 서드파티 스크립트)을 의심합니다.


원인을 확정한 뒤에는 리스트 가상화나 메모이제이션 경계 조정 같은 근본 수정을 적용하고, 이후 유사한 회귀를 막기 위해 대시보드 페이지에 성능 예산(예: 특정 인터랙션의 P75 응답 시간 임계값)을 설정해 CI나 RUM 알림으로 감시합니다.

이 답변이 좋은 이유:

  • 6단계 진단 프레임워크를 자연스럽게 녹여서 말함
  • "특정 고객사에서만"이라는 단서를 놓치지 않고 데이터 규모라는 변수로 연결함
  • DevTools Performance/React Profiler 같은 실제 도구를 구체적으로 언급
  • 메모이제이션을 무작정 추천하지 않고, 원인을 확정한 뒤에 처방을 결정함
  • 수정 후 검증과 성능 예산을 통한 회귀 방지까지 포함

9. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"리액트 컴포넌트에 React.memo와 useMemo를 추가해서 리렌더링을 줄이겠습니다."

이 답변이 약한 이유

  • 원인을 확인하지 않고 흔한 처방부터 던집니다 — 병목이 리렌더링이 아니라 네트워크나 서드파티 스크립트라면 아무 효과가 없습니다.
  • "특정 고객사에서만"이라는 핵심 단서를 활용하지 않았습니다. 이 단서가 원인 범위를 크게 좁혀주는데도 무시하고 있습니다.
  • 측정 없이 처방하고 있습니다 — 수정 전후 비교할 지표(LCP, INP, 렌더링 시간)가 없으면 실제로 개선됐는지 알 수 없습니다.
  • useMemo를 남발하면 오히려 메모리 사용량과 코드 복잡도가 늘어나는 트레이드오프도 언급하지 않았습니다.
WEAK ANSWER — 또 다른 패턴

"이미지를 WebP로 바꾸고 CDN을 적용하겠습니다."

이 답변이 약한 이유

  • 문제가 초기 로딩(LCP)이라는 가정을 먼저 했습니다. 하지만 리포트는 "클릭 후 반응"일 수도 있고, 이 경우 이미지 최적화는 무관합니다.
  • 일반적인 성능 체크리스트를 그대로 적용하는 패턴입니다 — 이 특정 상황(리액트 대시보드, 특정 고객사)에 맞춘 진단이 아닙니다.
  • "가끔"이라는 단서(항상이 아니라 간헐적)를 설명하지 못합니다. 이미지 최적화는 간헐성을 설명하지 못하는 처방입니다.
STRONG ANSWER 요소 체크리스트
  • "느리다"를 구체적 지표(LCP/INP/스크롤)와 조건으로 재정의한다
  • "특정 고객사에서만"같은 단서를 원인 후보를 좁히는 데 적극 활용한다
  • DevTools/RUM 등 실제 도구로 측정한 뒤 처방을 결정한다
  • 처방의 트레이드오프(복잡도, 메모리, 유지보수 비용)를 함께 언급한다
  • 수정 후 동일한 조건으로 재현·검증하고, 회귀 방지 장치를 추가한다

10. Senior-level Follow-up Q&A

Q: React를 쓰면 이런 브라우저 내부 동작을 몰라도 되는 것 아닌가요?

모범 답변: React는 "무엇을 그릴지"에 대한 선언과 "언제 다시 그릴지"에 대한 스케줄링을 도와줄 뿐, 실제로 DOM을 변경하고 스타일을 재계산하고 레이아웃을 다시 잡는 작업은 여전히 브라우저가 수행합니다. React가 100번 리렌더링을 계산해도, 그 결과가 DOM에 반영되는 순간 브라우저의 Style/Layout/Paint 비용은 그대로 발생합니다. 따라서 React를 잘 쓰는 것과 브라우저 렌더링 비용을 이해하는 것은 서로 다른 층의 지식이며, 시니어는 두 층을 모두 연결해서 사고해야 성능 문제를 정확히 진단할 수 있습니다.

Q: 성능 문제인지 아닌지, 먼저 나서서 개선을 제안한 경험이 있나요? 어떻게 우선순위를 정했나요?

모범 답변: RUM 데이터에서 P75/P95 같은 상위 백분위 지표를 정기적으로 확인하는 습관을 갖고, 사용자 불만이 없어도 특정 페이지의 지표가 목표 성능 예산을 넘으면 먼저 이슈를 제기합니다. 우선순위는 "얼마나 많은 사용자에게 영향을 주는가(트래픽 비중) × 얼마나 나쁜가(지표 악화 정도) × 비즈니스 임팩트(전환율에 영향을 주는 핵심 플로우인가)"를 기준으로 정합니다. 모든 성능 이슈를 동시에 고칠 수 없기 때문에, 이 우선순위 기준을 명시적으로 세우는 것 자체가 시니어의 역할입니다.

Q: 모바일에서만 느리다는 리포트를 받았습니다. 어떻게 접근할까요?

모범 답변: 데스크톱과 모바일은 CPU 성능, 네트워크 환경, 메모리가 크게 다르기 때문에 먼저 DevTools의 CPU/Network Throttling으로 저사양 환경을 시뮬레이션해 재현을 시도합니다. 데스크톱에서 문제없던 코드가 모바일에서만 느리다면, 메인 스레드 연산량(무거운 JS 로직)이나 리소스 크기(이미지, 번들) 문제일 가능성이 높습니다. 또한 실제 저사양 기기(RUM에서 디바이스별 지표를 분리해서 봄)에서의 데이터를 함께 확인해, 시뮬레이션과 실제 사용자 데이터가 일치하는지 교차 검증합니다.

Q: 서드파티 스크립트(광고, 분석 툴) 때문에 성능이 나쁜 것 같은데 비즈니스팀이 제거를 거부합니다. 어떻게 하겠어요?

모범 답변: 먼저 Performance 패널이나 PerformanceObserver로 각 서드파티 스크립트가 실제로 얼마나 메인 스레드 시간과 네트워크 대역폭을 소모하는지 수치로 측정합니다. "느낌"이 아니라 "이 스크립트가 LCP를 300ms 지연시키고 있다"처럼 구체적 수치로 비즈니스팀과 대화합니다. 제거가 어렵다면 대안(지연 로딩, async/defer, 사용자 인터랙션 이후 로드, 서드파티 스크립트를 자체 서버로 프록시해 캐싱)을 제시해 "제거 vs 유지"의 이분법이 아니라 "성능 비용을 줄이면서 유지하는 방법"으로 대화를 전환합니다.

Q (비기술적): 디자이너가 제안한 애니메이션이 성능에 안 좋아 보입니다. 어떻게 대화하겠어요?

모범 답변: 먼저 왜 안 좋아 보이는지 구체적으로 확인합니다(예: width/top 같은 Layout을 유발하는 속성을 애니메이션하고 있는가). 디자이너에게 "안 된다"고 하기보다 "이 효과를 유지하면서 Layout 대신 transform/opacity로 구현하면 GPU 레이어에서 처리되어 동일한 시각 효과를 더 부드럽게 낼 수 있다"처럼 대안을 함께 제시합니다. 즉, 성능 제약을 이유로 디자인 의도를 꺾는 것이 아니라, 같은 사용자 경험 목표를 다른 구현 방식으로 달성하는 협업으로 프레이밍합니다.

11. 흔한 오해

오해 1

"React 같은 프레임워크를 쓰면 브라우저 렌더링을 몰라도 된다"

프레임워크는 DOM 조작을 자동화할 뿐, 그 결과가 Style/Layout/Paint 비용으로 이어지는 것은 그대로입니다. 프레임워크 위에서 잘못된 패턴(과도한 리렌더링, 잦은 인라인 스타일 변경)을 쓰면 브라우저 비용은 그대로, 혹은 더 커질 수 있습니다.

오해 2

"느리다는 건 항상 JavaScript 코드 문제다"

실제로는 네트워크 지연, render-blocking 리소스, 복잡한 CSS 선택자로 인한 Style Recalc, 서드파티 스크립트 등 코드 로직 외의 계층에서 발생하는 경우도 매우 흔합니다. 코드로 뛰어들기 전에 어느 계층인지부터 확인해야 합니다.

오해 3

"Lighthouse 점수가 100점이면 실제 사용자도 빠르다"

Lighthouse는 특정 조건(고정된 네트워크/CPU 시뮬레이션, 특정 기기)에서의 랩(Lab) 데이터입니다. 실제 사용자는 다양한 기기·네트워크·데이터 규모를 가지므로, RUM(실사용자 모니터링) 데이터와 함께 봐야 진짜 사용자 경험을 알 수 있습니다. (Day 6, 14에서 심화)

오해 4

"성능 최적화는 항상 나중에, 필요할 때만 하면 된다"

성능 예산 없이 개발하면 여러 작은 회귀가 누적되어 어느 순간 "전체가 느려진" 상태로 발견됩니다. 이 시점에는 원인이 하나가 아니라 여러 개가 얽혀 있어 디버깅 비용이 훨씬 커집니다. 시니어는 처음부터 측정 가능한 예산을 세우고 CI에서 감시합니다.

오해 5

"내 로컬 개발 환경에서 빠르면 사용자에게도 빠르다"

개발자의 로컬 환경은 대개 고성능 기기, 빠른 네트워크, 캐시가 이미 채워진 상태입니다. 실제 사용자는 저사양 기기, 느린 네트워크, 첫 방문(캐시 없음)인 경우가 많습니다. Throttling 시뮬레이션과 실사용자 데이터로 이 간극을 항상 검증해야 합니다.
