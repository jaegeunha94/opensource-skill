Day 5 · LangChain / LangGraph Interview

Retrieval과 Memory

RAG를 agent에 통합하는 방식(retrieval = tool)과, agent가 "기억"을 갖게 만드는 두 축 — thread 안에서의 short-term memory(checkpointer)와 thread를 넘나드는 long-term memory(Store) — 그리고 이 설계가 실제로 비용·정확도에서 무너지는 지점
버전 기준: 이 레슨은 2026-07 기준 LangChain 1.x / LangGraph 1.x(GA) API를 기준으로 작성됐다. 구버전 langchain.memory의 ConversationBufferMemory/ConversationSummaryMemory 등은 LangChain 0.3.1부터 deprecated 되었고 1.x에서는 동작하지 않는다 — 공식 마이그레이션 가이드는 이를 LangGraph의 checkpointer(단기)와 Store(장기)로 대체하라고 안내한다. docs.langchain.com은 자동화 접근을 종종 차단(403)하므로, 이 레슨의 근거는 공식 마이그레이션 가이드, GitHub 소스(raw.githubusercontent.com), GitHub 이슈 트래커를 우선 사용했고 문서 사이트 인용은 검색 스니펫으로 교차 확인한 내용임을 명시한다.
면접 질문
Q1 — 주력 질문 "LangChain agent에 RAG(retrieval)를 어떻게 통합하나요? create_agent에 retrieval 전용 파라미터가 따로 있나요?"
Q2 — 변형 "Agent의 short-term memory와 long-term memory는 어떻게 다르고, 각각 LangGraph의 어떤 컴포넌트로 구현하나요?"
Q3 — 변형 "대화가 길어질수록 memory/context 관리에서 어떤 문제가 생기나요? 실제로 비용이나 정확도에 어떤 영향을 주나요?"
전제 개념

Retrieval은 왜 "그냥 tool"인가, Memory는 왜 두 축으로 나뉘는가

Retrieval — create_agent에 별도 파라미터가 없다

많은 사람이 "LangChain에는 RAG 기능이 내장돼 있을 것"이라고 기대하지만, create_agent에는 retrieval 전용 파라미터가 존재하지 않는다. 공식 패턴은 벡터 스토어에서 리트리버를 만들고 (vectorstore.as_retriever()), 이걸 tool 하나로 감싸서 다른 tool과 똑같이 tools=[...]에 넣는 것이다. langchain.tools.retriever.create_retriever_tool 헬퍼나 직접 @tool 데코레이터로 감싸는 방식 모두 가능하다. 

from langchain.tools.retriever import create_retriever_tool
from langchain.agents import create_agent

retriever = vectorstore.as_retriever(search_kwargs={"k": 4})
search_docs = create_retriever_tool(
    retriever,
    name="search_product_docs",
    description="제품 매뉴얼에서 관련 문서를 검색한다. 사용자 질문이 제품 사용법·스펙에 관한 것일 때 사용한다.",
)

agent = create_agent(model="claude-sonnet-5", tools=[search_docs])
핵심 전제: 이건 게으른 설계가 아니라 의도된 아키텍처다. Retrieval을 agent 루프에 특별한 1급 개념으로 넣지 않고 "모델이 스스로 판단해서 호출하는 tool"로 유지하면, 모델이 "지금 검색이 필요한가", "몇 번 검색할 것인가", "검색어를 어떻게 재구성할 것인가"까지 스스로 판단하게 된다. 이게 바로 agentic RAG다 — 검색을 파이프라인의 고정 단계가 아니라 agent의 의사결정 대상으로 만드는 것.

더 세밀한 제어가 필요하면(예: 검색 결과를 반드시 특정 노드에서 후처리해야 함) create_agent 대신 LangGraph로 직접 그래프를 짜고 retrieval을 하나의 node로 넣는 패턴도 공식적으로 문서화되어 있다. 이건 Day 1에서 다룬 "create_agent vs StateGraph" 판단 기준이 retrieval에도 그대로 적용되는 사례다 — 단순 RAG는 tool 하나로 충분하지만, 검색→재랭킹→필터링처럼 결정적으로 항상 거쳐야 하는 다단계 파이프라인이면 그래프로 명시적으로 그리는 게 낫다.

구버전 Memory 클래스는 죽었다 — 무엇으로 대체됐는가

LangChain 0.x 시절의 ConversationBufferMemory, ConversationBufferWindowMemory, ConversationSummaryMemory, ConversationEntityMemory는 0.3.1부터 deprecated 되었고, 1.x 라인에서는 실질적으로 동작하지 않는다. 공식 마이그레이션 가이드는 이 클래스들을 LangGraph의 영속성(persistence) 메커니즘으로 대체하라고 명시한다. 이 영속성은 목적에 따라 완전히 다른 두 컴포넌트로 나뉜다.

축Short-term memoryLong-term memory
범위하나의 thread(대화 세션) 안thread를 넘나드는 사용자/조직 단위
구현체Checkpointer (MemorySaver/InMemorySaver: dev, SqliteSaver/PostgresSaver: prod)Store (BaseStore 인터페이스, InMemoryStore: dev, PostgresStore: prod)
저장 단위그래프의 전체 state 스냅샷(메시지 이력 포함)을 매 스텝마다개발자가 명시적으로 put()한 임의의 key-value(+ namespace)
조회 방식thread_id로 이전 상태 복원get/list_keys 또는 semantic search(임베딩 유사도)
대표 용도"방금 뭐라고 했더라" — 대화 맥락 유지, time travel/replay(Day 8)"이 사용자는 예전에 이런 걸 좋아했다" — 세션이 끝나도 남는 사용자 프로필/선호
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.store.memory import InMemoryStore
from langchain.agents import create_agent

checkpointer = InMemorySaver()   # short-term: thread 안의 대화 이력
store = InMemoryStore()          # long-term: thread를 넘나드는 사용자 메모리

agent = create_agent(
    model="claude-sonnet-5",
    tools=[search_docs],
    checkpointer=checkpointer,
    store=store,
)
# thread_id가 같으면 같은 대화, 다르면 새 대화(그러나 store는 공유됨)
agent.invoke({"messages": [...]}, config={"configurable": {"thread_id": "user-42-session-9"}})

Store의 semantic search — "기억"을 임베딩 유사도로 찾기

BaseStore는 단순 key-value 저장소가 아니다. 임베딩 함수를 설정해두면 search/ asearch가 자연어 쿼리를 받아 저장된 메모리를 의미적 유사도 순으로 반환한다. "사용자가 예전에 채식주의자라고 말한 적 있는지" 같은 걸 정확한 키 없이 자연어로 찾을 수 있다는 뜻이다. 이건 Day 4에서 다룬 구조화된 출력의 스키마 검증과는 다른 종류의 신뢰성 문제를 만든다 — 정확한 매칭이 아니라 확률적 유사도 매칭이므로 recall(관련 기억을 놓치지 않는가)과 precision(관련 없는 기억을 끼워오지 않는가) 사이의 트레이드오프가 항상 존재한다. 

from langgraph.store.memory import InMemoryStore

store = InMemoryStore(index={"embed": embeddings_model, "dims": 1536})
store.put(("user-42", "preferences"), "dietary", {"note": "채식주의자, 견과류 알레르기"})

# 나중에 정확한 키 없이 자연어로 검색
results = store.search(("user-42", "preferences"), query="이 사용자가 못 먹는 음식이 있나?")

"Memory를 자동으로 써주는 middleware"는 없다 — LangMem이 그 공백을 메운다

langchain.agents.middleware 소스를 직접 확인하면, PII 리댁션·호출 제한·HITL 같은 built-in middleware는 있어도 "대화에서 장기기억 후보를 추출해서 Store에 자동으로 써주는" middleware는 없다. 이유는 단순하다 — "무엇을 장기기억으로 남길지"는 도메인마다 판단 기준이 완전히 다르기 때문에, 프레임워크가 일반화해서 자동화하기 어렵다. 그래서 실무에서는 두 가지 중 하나를 택한다.

  • 직접 구현: after_model hook에서 "이 턴에 장기기억으로 남길 만한 정보가 있는가"를 판단하는 별도 LLM 호출을 걸고, 조건에 맞으면 store.put()을 호출한다.
  • LangMem 사용: langchain-ai/langmem은 BaseStore 위에 얹는 별도 패키지로, 메모리 추출·메모리 관리용 tool·백그라운드 메모리 매니저를 제공한다. 1.x 코어에 흡수된 게 아니라 여전히 독립 패키지로 유지되고 있다는 점을 정확히 알아야 한다 — "공식 장기기억 API"가 아니라 "가장 흔히 곁들이는 동반 라이브러리"다.

Context 압박 관리 — SummarizationMiddleware

Short-term memory(checkpointer)에 대화가 계속 쌓이면 매 모델 호출마다 전체 이력이 context에 들어가면서 토큰 비용이 선형으로 증가한다. 이걸 완화하는 built-in middleware가 SummarizationMiddleware다 — 토큰 수가 임계치(trigger)에 가까워지면 오래된 메시지를 요약으로 압축하고, 최근 메시지(기본 마지막 20개, keep으로 조절)와 AI/Tool 메시지 쌍은 그대로 보존한다. 

from langchain.agents.middleware import SummarizationMiddleware

agent = create_agent(
    model="claude-sonnet-5",
    tools=[search_docs],
    checkpointer=checkpointer,
    middleware=[
        SummarizationMiddleware(
            model="claude-haiku-4-5",  # 요약 전용으로는 저렴한 모델을 쓰는 게 실무 관행
            trigger={"tokens": 4000},
            keep={"messages": 20},
        ),
    ],
)
주의: 이건 short-term memory(대화 내 context)를 압축하는 것이지, long-term memory(Store)에 뭔가를 "저장"하는 게 아니다. 요약된 옛 대화는 압축된 채로 이번 thread에만 남고, 다른 thread에서는 보이지 않는다. 두 축을 섞어서 "요약하니까 장기기억도 해결됐다"고 착각하면 안 된다.

실제로 터진 문제 — checkpoint 직렬화 오버헤드

2026년에 보고된 실제 이슈(langgraph GitHub #7714)는 16턴짜리 ReAct agent의 기본 체크포인트 직렬화가 최적화된 형태 대비 저장 용량 85.3%, 토큰 수 37.8%가 불필요하게 부풀려진다는 걸 측정으로 보여준다. 즉 short-term memory를 "그냥 켜두기만" 하면 매 후속 호출의 입력 토큰 비용이 눈덩이처럼 불어나는 게 기본 동작이라는 뜻이다. 같은 시기 문서 이슈(docs #2356)는 "체크포인트에 쌓인 이력이 전부 자동으로 모델 context에 들어가는지, 개발자가 부분집합을 선택해야 하는지"가 문서상 불명확하다는 점도 지적한다 — 이건 실무에서 그대로 함정이 된다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "RAG 튜토리얼을 따라해봤는가"가 아니라, "검색과 기억이라는 두 개념을 혼동하지 않고 각각을 올바른 컴포넌트에 배치할 수 있는가", 그리고 "memory가 켜져 있으면 공짜가 아니라 토큰 비용과 정확도 리스크를 계속 지불하고 있다는 걸 아는가"를 함께 본다.

  • Retrieval의 위치: retrieval이 agent 루프의 특수 기능이 아니라 tool이라는 걸 알고, 언제 tool로 충분하고 언제 그래프로 명시적 파이프라인을 짜야 하는지 판단하는가
  • 두 축의 분리: short-term(checkpointer, thread-scoped)과 long-term(Store, cross-thread)을 정확히 구분하고, 각각의 조회 방식(복원 vs semantic search) 차이를 아는가
  • 비용을 설계에 포함하는가: memory를 켜두는 것 자체가 토큰 비용·직렬화 오버헤드를 만든다는 것과, 이를 압축(summarization)으로 완화하는 것을 아는가
  • 확률적 검색의 한계: semantic search가 정확한 매칭이 아니라 유사도 기반이라 recall/precision trade-off가 항상 존재한다는 걸 이해하는가
Senior 개발자는 "memory를 추가했다"를 하나의 스위치로 말하지 않고, "이 정보는 이번 대화에서만 필요한가, 다음 세션에도 남아야 하는가"라는 질문에 먼저 답한 뒤 컴포넌트를 고르고, 그 선택이 만드는 토큰 비용과 검색 정확도 트레이드오프까지 하나의 설계 결정으로 설명한다.
실무 연결

실제 상황과 연결하기

상황 1: 사용자가 세션을 다시 열 때마다 예전에 말한 선호를 처음부터 다시 설명해야 함

Checkpointer만 쓰고 있어서 thread_id가 바뀌는 순간(앱 재시작, 새 대화 시작) 사용자의 선호(알레르기, 선호 브랜드 등)가 전부 사라졌다. → Store를 추가해서 "이 정보는 세션이 끝나도 남아야 하는가"를 기준으로 일부 정보를 after_model hook에서 store.put()으로 명시적으로 승격시켰다. 모든 대화를 장기기억으로 남기지 않고, 재사용 가치가 있다고 판단된 정보만 선별해서 저장하는 게 핵심이었다.

상황 2: 긴 대화가 이어질수록 응답 속도가 느려지고 비용이 급증

고객지원 챗봇이 한 세션 안에서 40~50턴씩 이어지는 경우가 흔했는데, 매 턴마다 전체 이력이 context에 들어가면서 모델 호출 비용이 선형으로 증가했다. → SummarizationMiddleware를 추가해 토큰 임계치를 넘으면 오래된 대화를 요약하고 최근 20개 메시지만 원문으로 유지하도록 바꿔서 비용 증가를 완화했다. 다만 요약 과정에서 세부 수치(계약 금액 등)가 뭉개진 사고가 있어, 수치·ID처럼 정확해야 하는 정보는 요약 대상에서 제외하고 별도로 Store에 저장하는 정책을 추가했다.

상황 3: Semantic search가 관련 없는 옛 기억을 끌어와 답변이 엉뚱해짐

"이 사용자가 선호하는 색상"을 물었는데, 몇 달 전 다른 맥락에서 언급된 "빨간색 경고"라는 문구가 임베딩 유사도로 걸려서 잘못된 답변에 섞여 들어갔다. → 저장 시 namespace를 세분화(예: (user_id, "preferences", "color")처럼)하고, 검색 결과에 유사도 점수 하한선을 둬서 애매한 매칭은 아예 모델에 넘기지 않도록 필터링했다. Semantic search는 "관련 있어 보이는 것"이지 "정확히 맞는 것"이 아니라는 전제를 코드에 반영한 사례다.

상황 4: 리트리버 tool이 항상 문서를 4개 가져와서 관련 없는 문서까지 context에 섞임

k=4로 고정된 리트리버가 질문 난이도와 무관하게 항상 4개를 가져와서, 단순 질문에도 관련성 낮은 문서가 섞여 답변 품질이 떨어졌다. → 모델이 검색어와 필요 문서 수를 스스로 판단하게 tool description을 구체화하고, 리트리버 tool 안에서 유사도 점수로 2차 필터링을 추가했다. "retrieval = tool"이라는 설계를 활용해 모델이 검색 전략 자체를 조정할 수 있게 만든 게 핵심 교훈이었다.

Memory를 두 축으로 나눠서 설명하기 "Short-term memory는 checkpointer로 thread 안의 대화 이력을 유지하고, long-term memory는 Store로 thread를 넘나드는 정보를 key-value 또는 semantic search로 관리합니다. 이 둘은 목적과 조회 방식이 완전히 다릅니다."
3
Memory가 공짜가 아니라는 것을 비용 관점에서 짚기 "체크포인트에 대화가 쌓일수록 매 호출의 토큰 비용이 늘어나고, 직렬화 오버헤드도 실제로 보고된 문제입니다. 그래서 SummarizationMiddleware로 압축하거나 필요한 정보만 선별해서 Store로 승격시킵니다."
4
Semantic search의 한계를 인정하기 "Store의 semantic search는 임베딩 유사도 기반이라 정확한 매칭이 아닙니다. Recall과 precision 사이 트레이드오프가 있고, namespace 설계와 유사도 임계값으로 관련 없는 기억이 섞이는 걸 방지해야 합니다."
5
자동화의 공백을 인지하고 있음을 보여주기 (Senior 포인트) "프레임워크는 무엇을 장기기억으로 남길지 자동으로 판단해주지 않습니다. 직접 after_model hook에서 판단 로직을 짜거나, LangMem 같은 별도 패키지를 조합해야 합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"LangChain agent에서 retrieval은 별도 기능이 아니라 tool입니다. 벡터 스토어를 retriever로 만들고 create_retriever_tool로 감싸서 다른 tool과 동일하게 등록하면, 모델이 스스로 언제 검색할지, 검색어를 어떻게 재구성할지 판단합니다. 이걸 agentic RAG라고 부르고, 검색 파이프라인을 항상 고정된 순서로 강제해야 한다면 create_agent 대신 LangGraph로 직접 노드를 구성합니다.

Memory는 두 축으로 나눠서 생각합니다. Short-term memory는 checkpointer가 담당하고, thread_id로 구분되는 하나의 대화 세션 안에서 전체 state를 스냅샷으로 저장·복원합니다. Long-term memory는 Store가 담당하고, thread를 넘나드는 사용자 프로필이나 선호 같은 정보를 key-value로 저장하거나 임베딩 기반 semantic search로 찾습니다. 이 둘은 목적이 다르기 때문에 구버전 ConversationBufferMemory 처럼 하나로 뭉뚱그려 쓰지 않습니다.

다만 memory를 켜두는 것 자체가 공짜가 아닙니다. 체크포인트에 대화가 쌓일수록 매 호출마다 전체 이력이 context에 들어가서 토큰 비용이 늘어나고, 실제로 기본 직렬화 방식의 오버헤드가 문제로 보고된 사례도 있습니다. 그래서 SummarizationMiddleware로 오래된 메시지를 압축하되 수치처럼 정확해야 하는 정보는 요약 대상에서 제외합니다. Long-term memory 쪽은 무엇을 저장할지 프레임워크가 자동으로 판단해주지 않기 때문에, after_model hook에서 직접 판단 로직을 짜거나 LangMem 같은 별도 패키지를 조합합니다. 그리고 semantic search는 정확한 매칭이 아니라 유사도 기반이라는 걸 항상 전제하고, namespace 설계와 유사도 임계값으로 관련 없는 기억이 섞여드는 걸 막습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "왜 LangChain은 retrieval을 create_agent의 1급 파라미터로 만들지 않았을까요?"

약한 답변: "아직 그 기능을 안 만들어서 그런 것 같습니다."

강한 답변: "의도된 설계라고 봅니다. Retrieval을 특수한 1급 개념으로 넣으면 '언제 검색할지'가 프레임워크 레벨에서 고정되어 버립니다. Tool로 유지하면 모델이 질문의 성격에 따라 검색 여부, 검색 횟수, 검색어 재구성까지 스스로 판단할 수 있고, 이게 바로 agentic RAG의 핵심입니다. 결정적으로 항상 거쳐야 하는 다단계 검색 파이프라인이 필요하면 그때는 tool 대신 LangGraph 노드로 명시적으로 그리면 됩니다."

F2. "체크포인터에 쌓인 대화 이력이 전부 자동으로 모델 context에 들어가나요?"

강한 답변: "이 부분은 실제로 커뮤니티에서도 문서가 불명확하다고 지적된 지점입니다. 체크포인터는 state 전체를 저장·복원하는 역할이고, 그 state의 messages 필드가 보통 모델 호출의 context로 들어가는데, 개발자가 별도로 trim이나 summarization을 걸지 않으면 대화가 길어질수록 context가 계속 자랍니다. 그래서 저는 memory를 켤 때 반드시 SummarizationMiddleware나 메시지 트리밍 정책을 함께 설계해서 '무제한으로 쌓이지 않는다'는 걸 보장합니다."

F3. "Store의 semantic search는 항상 정확한가요?"

약한 답변: "임베딩 기반이니까 관련 있는 걸 잘 찾아줍니다."

강한 답변: "아닙니다. 임베딩 유사도는 확률적 매칭이라 recall과 precision 사이 트레이드오프가 항상 존재합니다. 벤치마크에 따라 장기기억 라이브러리마다 검색 지연시간과 정확도 특성도 크게 달라질 수 있어서, '정확한 조회'가 아니라 '관련 있어 보이는 후보를 가져오는 것'으로 이해하고 설계합니다. 그래서 namespace를 세분화하고 유사도 점수 하한선을 둬서 애매한 매칭이 그대로 모델에 들어가지 않게 막습니다."

F4. "왜 LangChain은 memory-write를 자동화하는 built-in middleware를 안 만들었을까요?"

강한 답변: "Context 압박 관리(SummarizationMiddleware)는 '토큰 임계치를 넘었는가'라는 일반화 가능한 기준이 있어서 built-in으로 만들기 쉽습니다. 반면 '이 정보를 장기기억으로 남길 가치가 있는가'는 도메인마다 판단 기준이 완전히 다릅니다 — 고객지원 봇과 코딩 어시스턴트가 남겨야 할 정보가 다르니까요. 그래서 프레임워크는 Store라는 저장소만 제공하고, 무엇을 저장할지는 after_model hook에서 직접 판단 로직을 짜거나 LangMem 같은 메모리 관리 전용 패키지를 조합하도록 열어뒀다고 봅니다."

F5. "체크포인트 직렬화 오버헤드는 실무에서 어떻게 완화하나요?"

강한 답변: "가장 먼저 체크포인터가 매 호출마다 전체 state를 그대로 직렬화한다는 걸 인지해야 합니다. 완화 방법은 세 가지로 조합합니다 — 첫째 SummarizationMiddleware로 오래된 메시지를 압축해서 state 자체를 줄이고, 둘째 state에 불필요한 대용량 필드(예: 원본 검색 결과 전체)를 그대로 들고 있지 않도록 노드에서 필요한 부분만 추려 저장하고, 셋째 프로덕션 checkpointer(Postgres 등)의 직렬화 방식과 스토리지 비용을 모니터링해서 대화가 길어지는 서비스에서 비용이 선형 이상으로 늘지 않는지 주기적으로 확인합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • Retrieval이 agent 루프의 특수 기능이 아니라 tool이라는 것을 정확히 안다
  • Short-term(checkpointer)과 long-term(Store)을 범위와 조회 방식으로 명확히 구분한다
  • Memory가 토큰 비용·직렬화 오버헤드를 만든다는 것을 실제 수치/사례로 안다
  • Semantic search의 recall/precision trade-off를 인지하고 필터링 전략을 설명한다
  • Memory-write 자동화의 공백을 인지하고 직접 설계 또는 보조 라이브러리 조합을 언급한다

약한 답변의 특징

  • "RAG는 벡터 DB에 넣고 검색하면 된다"는 단순 파이프라인 설명에 그침
  • Memory를 "대화 기억해주는 기능" 하나로 뭉뚱그리고 short/long-term을 구분 못 함
  • Memory를 켜두면 비용이 늘어난다는 것을 인지하지 못함
  • Semantic search가 항상 정확하다고 가정
  • 구버전 ConversationBufferMemory를 여전히 표준으로 알고 있음
핵심 차이: 강한 답변은 retrieval과 memory를 "기능 나열"이 아니라 "이 정보가 어느 범위에서 필요한가, 그리고 그 선택이 만드는 비용·정확도 트레이드오프가 무엇인가"라는 하나의 설계 축으로 설명한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "ConversationBufferMemory 같은 memory 클래스를 계속 써도 된다"

사실: 0.3.1부터 deprecated되었고 1.x에서는 동작하지 않는다. 공식 마이그레이션 가이드는 checkpointer(short-term)와 Store(long-term)로 대체하라고 명시한다.

오해 2: "memory를 켜두면 대화 맥락을 더 잘 이해하니 무조건 좋다"

사실: Checkpointer에 대화가 쌓일수록 매 호출의 토큰 비용이 늘고, 직렬화 오버헤드도 실제로 보고된 문제다. Memory는 정확도를 사기 위해 비용을 지불하는 트레이드오프지, 무조건적인 이득이 아니다.

오해 3: "Semantic search는 임베딩 기반이니 정확한 매칭이다"

사실: 유사도 기반 확률적 매칭이라 recall/precision trade-off가 항상 있다. 관련 없어 보이는 옛 기억이 유사도로 걸려 잘못된 답변에 섞여 들어갈 수 있다.

오해 4: "LangMem이나 Store를 쓰면 장기기억 저장이 자동으로 처리된다"

사실: Store는 저장소(인프라)를 제공할 뿐, "무엇을 저장할지" 판단은 자동화되어 있지 않다. LangMem도 그 판단을 돕는 도구를 제공하는 별도 패키지이지 완전 자동화가 아니며, 대부분 도메인 특화 판단 로직을 직접 설계해야 한다.

핵심 요약

오늘 반드시 기억할 것

  • Retrieval은 create_agent의 1급 기능이 아니라 tool이다. 리트리버를 tool로 감싸 모델이 검색 여부·전략을 스스로 판단하게 하는 게 agentic RAG이고, 고정된 다단계 파이프라인이 필요하면 LangGraph 노드로 명시적으로 구성한다.
  • 구버전 memory 클래스(ConversationBufferMemory 등)는 죽었다 — short-term memory는 checkpointer(thread-scoped), long-term memory는 Store(cross-thread)로 완전히 나뉘어 대체됐다.
  • Store는 semantic search를 지원하지만 이건 확률적 유사도 매칭이라 recall/precision trade-off가 항상 존재한다 — namespace 설계와 유사도 임계값으로 관리해야 한다.
  • Memory는 공짜가 아니다. Checkpointer에 대화가 쌓일수록 토큰 비용과 직렬화 오버헤드가 늘어나며, 이는 실제로 보고된 프로덕션 문제다. SummarizationMiddleware로 압축하되 정확해야 하는 정보는 요약 대상에서 제외한다.
  • "무엇을 장기기억으로 남길지"는 자동화되어 있지 않다. after_model hook에서 직접 판단 로직을 짜거나 LangMem 같은 별도 패키지를 조합해야 한다 — LangMem은 코어에 흡수된 게 아니라 독립 패키지다.
