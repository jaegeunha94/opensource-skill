Day 2 · LangChain / LangGraph Interview

Model / Tool / Message abstraction

provider마다 다른 LLM API를 어떻게 하나의 계약(contract)으로 통일하고, tool 실행 실패를 어떻게 설계로 흡수하는가
버전 기준: 이 레슨은 2026-07 기준 LangChain 1.x / LangGraph 1.x (GA) API를 기준으로 작성됐다. 특히 tool 실행 컨텍스트 주입은 과거의 InjectedState/InjectedStore/get_runtime()/InjectedToolCallId 방식에서 ToolRuntime 단일 인터페이스로 통합되는 중이다(이전 방식도 하위 호환되지만, 신규 구현은 ToolRuntime 기준으로 학습한다). 면접 준비 시점에 실제 시그니처를 공식 문서로 다시 확인하라.
면접 질문
Q1 — 주력 질문 "LangChain의 ChatModel 인터페이스가 제공하는 표준화란 구체적으로 무엇이고, 이게 없다면 실무에서 어떤 문제가 생기나요?"
Q2 — 변형 "Tool 함수를 정의할 때 입력 스키마는 어떻게 결정되나요? 그리고 tool 실행 중 예외가 발생하면 agent 전체는 어떻게 되나요?"
Q3 — 변형 "Tool 안에서 LangGraph의 현재 state나 tool_call_id 같은 실행 컨텍스트에 접근해야 한다면 어떻게 설계하시겠어요?"
전제 개념

모델·도구·메시지를 왜 "추상화"해야 하는가

문제의 출발점: provider마다 API가 다 다르다

OpenAI, Anthropic, Google 같은 provider는 각자 REST API의 요청/응답 형식이 다르다. "tool을 호출하라"는 지시 하나만 봐도 필드 이름, 중첩 구조, 스트리밍 이벤트 형식이 제각각이다. 만약 애플리케이션 코드가 특정 provider의 원시 응답 형식에 직접 의존하면, 모델을 바꾸거나 비교 실험을 할 때마다 애플리케이션 로직을 전부 다시 써야 한다.

LangChain의 ChatModel(정확히는 BaseChatModel) 인터페이스는 이 문제를 해결하기 위한 공통 계약(contract)이다. 어떤 provider를 쓰든 invoke(), stream(), batch()와 그 비동기 버전(ainvoke 등)을 똑같은 방식으로 호출할 수 있고, bind_tools()로 tool을 등록하는 방식도 동일하다.

핵심 전제: 표준화는 "더 적은 기능"이 아니라 "공통 부분을 통일하고, provider별 고유 기능은 표준 확장 지점으로 흡수하는 것"이다. 이 원칙을 이해하면 이후 middleware, structured output, streaming 레슨에서도 같은 사고방식이 반복된다.

메시지(Message) — 대화를 표현하는 공통 자료구조

LLM과의 대화는 결국 "역할(role) + 내용(content)"의 연속이다. LangChain은 이를 SystemMessage, HumanMessage, AIMessage, ToolMessage 네 가지 메시지 타입으로 표현한다.

타입역할대표 용도
SystemMessagesystem모델의 역할/지침 설정
HumanMessageuser사용자 입력(텍스트, 이미지 등 멀티모달 포함 가능)
AIMessageassistant모델 응답. tool_calls 속성에 요청한 tool 호출 목록이 담김
ToolMessagetooltool 실행 결과를 다시 모델에게 돌려줄 때 사용. tool_call_id로 어떤 호출에 대한 응답인지 연결

과거에는 각 provider가 응답 안에 텍스트, 이미지, reasoning(추론 과정), citation(인용) 같은 것을 제각각 다른 필드로 표현했다. LangChain 1.0부터는 AIMessage.content_blocks를 통해 표준 content block(text, tool_call, reasoning, image, audio, citation 등)로 통일해서 노출한다. 즉 "이 모델이 reasoning을 지원하는지, citation을 반환하는지"를 provider별 원시 응답 구조를 파싱하지 않고도 동일한 방식으로 접근할 수 있다. 

from langchain.chat_models import init_chat_model

model = init_chat_model("claude-sonnet-5", model_provider="anthropic")
response = model.invoke([{"role": "user", "content": "3 곱하기 4는?"}])

# response는 항상 AIMessage. provider가 바뀌어도 아래 접근 방식은 동일하다.
print(response.content)         # 텍스트 content
print(response.content_blocks)  # 표준 content block 리스트(text/tool_call/reasoning/...)
print(response.tool_calls)      # 모델이 요청한 tool 호출 목록

Tool — 모델이 호출할 수 있는 "함수 계약"

Tool은 "모델이 이름과 파라미터로 호출을 요청하고, 애플리케이션이 실제로 실행한 뒤 결과를 돌려주는 함수"다. LangChain에서 가장 흔한 정의 방법은 @tool 데코레이터다. 함수의 타입 힌트와 docstring에서 입력 스키마(args_schema, 내부적으로 Pydantic 모델)를 자동으로 추론한다. 

from langchain_core.tools import tool

@tool
def get_order_status(order_id: str) -> str:
    """주문 ID로 배송 상태를 조회한다.

    Args:
        order_id: 조회할 주문의 고유 ID.
    """
    # 실제로는 주문 시스템 API 호출
    return f"주문 {order_id}는 배송 중입니다."

docstring은 장식이 아니라 모델에게 전달되는 tool 설명 그 자체다. 모델은 이 설명만 보고 "지금 이 tool을 호출해야 하는지, 어떤 인자를 넣어야 하는지"를 판단한다. 설명이 모호하면 모델이 엉뚱한 인자를 채우거나 호출하지 말아야 할 때 호출하는 문제가 생긴다 — 이는 프롬프트 엔지니어링과 동일한 비중으로 다뤄야 할 설계 작업이다.

ToolRuntime — tool이 "실행 컨텍스트"에 접근하는 표준 방법

실무에서 tool은 단순 계산기가 아니다. "현재 대화 상태를 읽어야", "이 tool 호출의 ID를 알아야 ToolMessage를 정확히 매칭해야", "장기 메모리 store에 접근해야" 하는 경우가 흔하다. 과거에는 이런 요구마다 InjectedState, InjectedStore, get_runtime(), InjectedToolCallId 같은 서로 다른 메커니즘을 각각 배워야 했다.

최신 방식은 이를 ToolRuntime 하나로 통합한다. 함수 파라미터에 runtime: ToolRuntime 타입 힌트를 추가하면, 프레임워크가 자동으로 현재 state, tool_call_id, config, context, store, stream_writer, tools 목록이 담긴 객체를 주입해준다. 이 파라미터는 모델에게 노출되는 입력 스키마에는 포함되지 않는다 — 모델은 이 값을 몰라도 되고, 몰라야 한다. 

from langchain_core.tools import tool
from langchain.tools import ToolRuntime

@tool
def escalate_to_human(reason: str, runtime: ToolRuntime) -> str:
    """처리할 수 없는 요청을 사람 상담원에게 넘긴다."""
    # runtime.state: 현재 그래프 상태 (예: 대화 이력, 사용자 정보)
    # runtime.tool_call_id: 이 tool 호출의 고유 ID (ToolMessage 매칭용)
    # runtime.store: 장기 메모리 저장소
    ticket_id = create_escalation_ticket(runtime.state["user_id"], reason)
    return f"상담원 티켓 {ticket_id}가 생성되었습니다."
흔한 오해: "tool 함수는 순수 함수(pure function)여야 한다"고 생각해서 상태 접근이 필요한 로직을 억지로 tool 밖의 노드 로직으로 빼내는 경우가 있다. 하지만 ToolRuntime은 이런 상태 의존적 tool을 정식으로 지원하는 설계다. 문제는 "상태에 접근하는가"가 아니라 "그 접근이 명시적이고 타입-안전한가"이다.

Tool 실행 실패는 예외가 아니라 "관측 결과"다

Tool은 언제든 실패할 수 있다 — 외부 API 타임아웃, 잘못된 인자, 권한 부족 등. 문제는 "tool이 실패했을 때 agent 전체가 죽어도 되는가"이다. 정답은 대부분 "아니오"다. 사람도 도구가 안 되면 "이 방법은 안 되네, 다른 방법을 시도해보자"라고 생각하지, 그 자리에서 기절하지 않는다. Agent도 마찬가지로 tool 실패를 모델에게 다시 보여주고 스스로 복구를 시도하게 하는 것이 기본 원칙이다.

이를 위해 ToolException과 handle_tool_error(개별 tool 단위) / ToolNode(handle_tool_errors=...)(여러 tool을 실행하는 노드 단위) 설정이 있다. 활성화되어 있으면, tool에서 발생한 예외는 그래프를 중단시키는 대신 에러 내용을 담은 ToolMessage로 변환되어 모델에게 다시 전달된다. 모델은 이 메시지를 보고 다른 인자로 재시도하거나, 다른 tool을 쓰거나, 사용자에게 상황을 설명할 수 있다.

실제로 있었던 프로덕션 사고 사례: langgraph-prebuilt 1.0.1(langgraph 1.0.2 경유) 패치에서 ToolNode의 에러 핸들링이 기본값으로 비활성화되도록 동작이 바뀐 사례가 보고됐다 (GitHub Issue langchain-ai/langgraph#6486). 그 전까지는 tool 예외가 자동으로 ToolMessage로 감싸져 모델에게 전달됐는데, 이 패치 이후에는 handle_tool_errors=True를 명시하지 않으면 예외가 그대로 전파되어 그래프 실행 자체가 중단된다. 마이너/패치 버전만 올렸을 뿐인데 "잘 작동하던 에러 복구 로직이 프로덕션에서 갑자기 죽는" 사고가 발생할 수 있다는 뜻이다. ※ 이 이슈는 커뮤니티에서 제기된 보고이며 향후 패치로 되돌려질 수 있다. 면접 준비 시점과 실제 운영 시점 모두, 사용 중인 langgraph/langgraph-prebuilt 버전의 changelog에서 에러 핸들링 기본값을 반드시 재확인해야 한다. 
from langgraph.prebuilt import ToolNode

# 명시적으로 에러 핸들링을 켜서, 예외 발생 시에도 그래프가 죽지 않고
# 에러 내용이 ToolMessage로 모델에게 전달되도록 만든다.
tool_node = ToolNode([get_order_status, escalate_to_human], handle_tool_errors=True)
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 주제는 "API를 외웠는가"가 아니라 "프레임워크의 표준화 계층을 이해하고, 그 경계에서 발생하는 실패를 설계로 흡수할 수 있는가"를 본다. 표준화 계층은 편의를 위한 것이지만, 동시에 "어디까지 표준화되고 어디부터 provider별 차이가 새어 나오는지"를 모르면 프로덕션에서 예측 못 한 버그를 만든다.

  • 추상화의 목적을 이해하는가: ChatModel/Message 표준화가 "왜" 필요한지 (provider 종속성 제거, 비교 실험 용이성)를 원리로 설명할 수 있는가
  • tool 설계를 프롬프트 설계로 보는가: docstring과 타입 힌트가 모델의 판단에 직접 영향을 준다는 것을 인지하고 있는가
  • 실행 컨텍스트 주입을 안전하게 설계하는가: tool이 상태/저장소에 접근해야 할 때 무분별한 전역 변수 대신 ToolRuntime 같은 명시적 메커니즘을 쓸 줄 아는가
  • 실패를 시스템의 일부로 다루는가: tool 실패 시 전체가 죽는 것이 기본값이 아니라 선택이라는 것, 그리고 그 선택이 라이브러리 패치 하나로 바뀔 수 있다는 운영 리스크를 이해하는가
Senior 개발자는 "tool을 붙여봤다"가 아니라, "tool 실패가 프로덕션에서 어떤 결과를 낳는지, 그리고 그 동작이 라이브러리 업그레이드로 조용히 바뀔 수 있다는 것까지 대비하는지"를 보여줘야 한다. 이런 디테일이 실제 장애를 막는다.
실무 연결

실제 상황과 연결하기

상황 1: 모델 provider를 바꾸는 A/B 테스트

비용/품질 비교를 위해 같은 agent를 Claude와 다른 provider 모델로 번갈아 돌려야 한다. → 애플리케이션 코드가 AIMessage/content_blocks 같은 표준 인터페이스에만 의존하도록 짜두면, init_chat_model()의 모델 이름만 바꿔서 실험할 수 있다. 원시 응답 파싱 코드가 섞여 있으면 이 실험 자체가 불가능해진다.

상황 2: 외부 결제 API를 호출하는 tool

결제 tool이 타임아웃되면 agent가 통째로 죽어서 사용자가 빈 화면을 보게 되는 사고가 났다. → ToolNode(handle_tool_errors=True)로 예외를 ToolMessage로 흡수하고, 모델이 "잠시 후 다시 시도해달라"는 안내를 사용자에게 전달하도록 시스템 프롬프트에 지침을 추가한다. 동시에 재시도해도 안전한(idempotent) tool만 재시도하도록 tool 설계 단계에서 구분해둔다.

상황 3: 여러 tool이 사용자별 세션 데이터를 공유해야 함

"이전에 조회한 주문 ID를 다음 tool 호출에서도 참조해야 한다"는 요구가 생겼다. → tool 파라미터로 주문 ID를 모델이 다시 입력하게 하는 대신, ToolRuntime.state로 그래프 상태에서 직접 읽어오게 하면 모델이 잘못된 값을 "환각"으로 채워 넣을 여지가 줄어든다.

상황 4: 라이브러리 정기 업그레이드 프로세스

"1.0 GA 이후로는 breaking change가 없다"는 공지만 믿고 langgraph 패치 버전을 자동 업데이트했더니 tool 에러 핸들링 기본값이 바뀌어 야간 배치에서 예외가 그대로 전파됐다. → 메이저 안정성 약속과 별개로, 패치 업그레이드도 변경 이력(changelog)을 검토하고 스테이징에서 tool 실패 경로(failure path)를 회귀 테스트하는 절차를 CI에 포함시킨다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"LangChain의 ChatModel 인터페이스는 provider마다 다른 API 응답 형식을 감추는 표준 계약입니다. 어떤 모델을 쓰든 invoke/stream/batch 호출 방식이 같고, 응답은 항상 AIMessage로 오며, content_blocks를 통해 텍스트·tool 호출·reasoning·citation 같은 것들을 표준화된 형태로 받을 수 있습니다. 이 덕분에 모델을 교체하거나 비교 실험을 할 때 애플리케이션 로직을 건드릴 필요가 없습니다.

Tool을 정의할 때는 함수의 타입 힌트와 docstring이 곧 모델에게 전달되는 스펙이기 때문에, 저는 이걸 프롬프트 설계의 연장선으로 다룹니다. 그리고 tool이 그래프 상태나 저장소 같은 실행 컨텍스트에 접근해야 하면, 파라미터에 값을 몰래 흘려보내는 대신 ToolRuntime을 통해 명시적으로 주입받습니다. 이렇게 하면 모델이 몰라도 되는 값을 인자로 요구하지 않아서 환각으로 잘못된 값을 채워 넣을 위험도 줄어듭니다.

마지막으로 tool 실패는 예외로 전체 그래프를 죽이는 게 아니라, ToolMessage로 모델에게 돌려줘서 스스로 재시도하거나 다른 방법을 찾게 하는 것이 기본 원칙이라고 생각합니다. 다만 이 기본 동작 자체도 라이브러리 패치 버전에서 조용히 바뀐 사례가 있었기 때문에, 메이저 버전의 안정성 약속만 믿지 않고 패치 업그레이드 때도 tool 실패 경로를 스테이징에서 반드시 회귀 테스트하는 절차를 운영 프로세스에 포함시킵니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "tool docstring을 잘못 쓰면 실제로 어떤 문제가 생기나요? 예를 들어주세요."

약한 답변: "모델이 tool을 잘 못 씁니다."

강한 답변: "두 가지 실패 패턴이 흔합니다. 하나는 과소 호출 — 설명이 모호해서 모델이 이 tool을 언제 써야 할지 몰라 아예 호출하지 않고 잘못된 답을 지어내는 경우입니다. 다른 하나는 과다/오용 호출 — 파라미터 설명이 불충분해서 형식이 안 맞는 값을 채워 넣거나, 호출하지 말아야 할 상황에서도 호출하는 경우입니다. 예를 들어 order_id: str라고만 써두면 모델이 '최근 주문'이라는 표현만 보고 존재하지 않는 ID를 지어낼 수 있습니다. docstring에 '모르는 경우 사용자에게 먼저 물어봐라' 같은 명시적 지침을 넣는 것도 tool 설계의 일부입니다."

F2. "ToolRuntime으로 상태에 접근하는 tool과, 상태를 아예 모르는 순수 tool 중 언제 어떤 걸 쓰나요?"

강한 답변: "재사용성과 결합도 사이의 트레이드오프입니다. 외부 API 호출처럼 독립적인 기능은 순수 함수로 만들어야 다른 그래프나 프로젝트에서도 그대로 재사용할 수 있습니다. 반면 '현재 대화의 사용자 정보', '이전 단계에서 조회한 값' 같은 세션 컨텍스트가 필요한 tool은 모델에게 그 값을 다시 입력하라고 요구하면 환각 위험이 커지므로, ToolRuntime으로 상태에서 직접 읽는 것이 더 안전합니다. 저는 tool을 만들 때 '이 정보를 모델이 대화 맥락에서 정확히 알고 있는가'를 기준으로 파라미터로 받을지, runtime으로 주입받을지를 결정합니다."

F3. "tool 에러를 무조건 ToolMessage로 감싸서 모델에게 넘기면 안 되는 경우도 있을까요?"

약한 답변: "없습니다, 항상 감싸는 게 안전합니다."

강한 답변: "있습니다. 인증 실패나 권한 오류처럼 재시도로 해결되지 않는 구조적 에러는 모델에게 넘겨서 재시도를 유도하기보다 즉시 상위 로직(또는 사람)에게 에스컬레이션하는 게 낫습니다. 모델이 '다시 시도해볼게요'라며 같은 실패를 반복 호출하면 불필요한 토큰 비용과 지연시간만 늘어납니다. 그래서 저는 에러를 일시적 실패(재시도 가치 있음)와 구조적 실패(재시도 무의미)로 구분해서, 전자만 ToolMessage로 모델에 돌려주고 후자는 별도 예외 처리 경로로 분리합니다."

F4. "메이저 버전은 안정적이라는데, 왜 패치 버전 업그레이드도 검증해야 하나요?"

강한 답변: "'breaking API change가 없다'는 약속과 '내부 기본 동작(default behavior)이 절대 안 바뀐다'는 것은 다른 얘기이기 때문입니다. 실제로 langgraph-prebuilt의 패치 버전에서 ToolNode의 에러 핸들링 기본값이 바뀐 사례가 보고된 적이 있습니다. 함수 시그니처는 그대로였지만 '기본으로 무엇을 하는가'가 바뀌면서, 명시적으로 옵션을 지정하지 않은 기존 코드가 런타임에 다르게 동작한 것입니다. 그래서 저는 시맨틱 버저닝의 메이저 안정성 약속을 신뢰하되, 패치 업그레이드도 changelog를 읽고 스테이징에서 핵심 실패 경로를 회귀 테스트하는 절차는 생략하지 않습니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 표준화 계층의 목적(provider 독립성, 실험 용이성)을 원리로 설명한다
  • tool docstring/타입 힌트를 모델에게 전달되는 스펙으로 취급한다
  • 상태 접근이 필요한 tool을 ToolRuntime으로 명시적으로 설계한다
  • tool 실패를 일시적 vs 구조적으로 구분해서 처리 전략을 다르게 가져간다
  • 라이브러리 업그레이드의 운영 리스크까지 고려한다

약한 답변의 특징

  • "ChatModel은 그냥 LLM 호출 함수다" — 표준화의 이유를 설명 못 함
  • docstring을 사람이 읽는 주석 정도로만 취급
  • tool에서 상태가 필요하면 전역 변수나 클로저로 임시방편 처리
  • "tool 에러는 무조건 try/except로 감싸면 끝" — 재시도 무의미한 에러까지 동일하게 처리
  • "메이저 버전이 안정적이라니 패치 업그레이드는 신경 안 써도 된다"고 단정
핵심 차이: 강한 답변은 "이 함수를 어떻게 쓰는가"가 아니라 "이 추상화가 왜 이렇게 설계됐고, 그 경계에서 실패가 발생하면 어떤 원칙으로 처리하는가"를 보여준다. 표준화된 인터페이스 뒤에 숨은 실제 실행 세부사항(에러 핸들링 기본값, provider 차이)을 알고 있어야 프로덕션 사고를 예방할 수 있다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Message 표준화는 단순히 텍스트를 담는 그릇일 뿐이다"

사실: AIMessage는 텍스트뿐 아니라 tool 호출 요청(tool_calls), 표준 content block(reasoning, citation, image 등)까지 포함하는 구조화된 객체다. "content만 보면 된다"고 생각하면 tool_calls나 reasoning 정보를 놓치는 코드를 짜게 된다.

오해 2: "@tool 데코레이터는 그냥 함수를 등록하는 문법 설탕이다"

사실: 데코레이터는 함수의 타입 힌트와 docstring을 분석해 모델이 실제로 받아보는 tool 스펙(이름, 설명, 파라미터 스키마)을 자동 생성한다. 이 스펙의 품질이 곧 모델의 tool 사용 정확도를 결정하므로, 단순 등록 이상의 설계 작업이다.

오해 3: "tool 함수는 항상 순수 함수여야 하고, 상태 접근은 안티패턴이다"

사실: ToolRuntime은 상태·저장소·설정에 접근하는 tool을 정식으로 지원하도록 설계된 메커니즘이다. 문제는 상태 접근 자체가 아니라, 그 접근 방식이 암묵적(전역 변수, 클로저)인지 명시적(타입 힌트로 선언된 런타임 주입)인지다.

오해 4: "tool에서 예외가 나면 당연히 agent가 멈춰야 정상이다"

사실: 기본 원칙은 정반대다. tool 실패를 ToolMessage로 모델에게 되돌려줘서 스스로 복구를 시도하게 하는 것이 설계 의도다. 다만 이 동작은 명시적 설정 (handle_tool_errors)에 의존하며, 그 기본값조차 라이브러리 버전에 따라 달라질 수 있으므로 "당연히 이렇게 될 것"이라고 가정하지 말고 항상 확인해야 한다.

2. tool docstring 하나를 잘못 써서 모델이 오작동하는 시나리오를 즉석에서 만들 수 있는가?

4. tool 에러를 "일시적"과 "구조적"으로 나누고, 각각 어떻게 다르게 처리할지 설명할 수 있는가?
5. 메이저 버전 안정성 약속과 패치 버전 기본값 변경 리스크의 차이를 구분해서 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • ChatModel/Message 표준화는 provider 차이를 감춰서 모델 교체·비교 실험을 가능하게 하는 계약이다. content_blocks로 text/tool_call/reasoning/citation을 통일해서 다룬다.
  • Tool 정의는 프롬프트 설계다. docstring과 타입 힌트가 곧 모델이 받는 스펙이며, 품질이 낮으면 과소/오용 호출로 이어진다.
  • ToolRuntime은 state/tool_call_id/store 같은 실행 컨텍스트를 명시적이고 타입-안전하게 주입하는 통합 인터페이스다. 상태 접근 tool을 안티패턴으로 여기지 않는다.
  • Tool 실패는 예외가 아니라 관측 결과다. handle_tool_errors로 실패를 ToolMessage로 흡수해서 모델이 스스로 복구를 시도하게 하는 것이 기본 원칙이다.
  • 기본값도 버전에 따라 바뀔 수 있다. 메이저 버전 안정성 약속과 별개로, 패치 업그레이드 시 tool 실패 경로를 회귀 테스트하는 습관이 실무 사고를 막는다.
