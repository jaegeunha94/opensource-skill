Day 4 · LangChain / LangGraph Interview

Structured Output과 Streaming

Agent가 신뢰할 수 있는 형식으로 답을 내놓게 만드는 방법(ToolStrategy/ProviderStrategy)과, 실시간 UX를 위해 무엇을 얼마나 흘려보낼지 결정하는 stream_mode 설계, 그리고 이 둘이 실제로 충돌하는 지점
버전 기준: 이 레슨은 2026-07 기준 LangChain 1.x (GA, 1.2 계열 이후)의 create_agent(response_format=...)와 agent.stream()/astream()의 stream_mode를 기준으로 작성됐다. 구조화된 출력 전략(ToolStrategy/ProviderStrategy/AutoStrategy)과 stream_mode는 1.0에서 안정화된 API지만, provider별 strict 지원 범위, 특정 모델의 조용한 전략 폴백, 그리고 astream_events(version="v3") 기반의 신규 event streaming(typed projection) 계층은 이 레슨 작성 시점에도 계속 움직이고 있다. 실제 면접·구현 전에는 reference.langchain.com과 GitHub 이슈 트래커로 현재 동작을 재확인하라.
면접 질문
Q1 — 주력 질문 "create_agent에서 response_format으로 구조화된 출력을 받을 때 ToolStrategy와 ProviderStrategy는 뭐가 다르고, 어떤 기준으로 선택하나요? 모델이 스키마에 맞지 않는 출력을 내면 어떻게 처리되나요?"
Q2 — 변형 "Agent 응답을 스트리밍할 때 stream_mode의 values, updates, messages, custom은 각각 언제 쓰나요? UX와 payload 비용 측면에서 어떤 trade-off가 있나요?"
Q3 — 변형 "구조화된 출력을 요구하는 agent에서 스트리밍은 어떻게 동작하나요? 토큰 단위 스트리밍과 강제된 구조화 출력은 근본적으로 충돌하는 지점이 있지 않나요?"
전제 개념

구조화된 출력이 필요한 이유, 그리고 3가지 전략

왜 "JSON으로 답해줘"라는 prompt로는 부족한가

LLM의 기본 출력은 자유 텍스트다. 그런데 애플리케이션(라우팅 로직, DB 저장, downstream API 호출)은 파싱 가능한 데이터가 필요하다. 가장 흔한 초심자 접근은 "JSON 형식으로만 답해줘"라고 prompt에 적고 json.loads()로 직접 파싱하는 것인데, 이 방법은 실무에서 조용히 무너진다. 모델이 설명 문장을 JSON 앞뒤에 덧붙이거나, 마크다운 코드블럭(```json)으로 감싸거나, 필드명을 오타내거나, 따옴표를 깨뜨리는 경우가 실제로 자주 발생한다. 이런 실패는 결정적(deterministic)이지 않고 모델·prompt·입력에 따라 산발적으로 발생하기 때문에, 매번 다른 방어 코드를 덧붙이게 되고 결국 유지보수 불가능한 파싱 레이어가 쌓인다.

LangChain은 이 문제를 스키마를 코드로 선언하고, 모델이 그 스키마를 지키도록 강제하거나 검증 후 실패 시 되돌려서 재시도시키는 계층으로 해결한다. 이 계층은 두 군데에 존재한다 — 개별 모델 호출에 붙이는 with_structured_output(model-level)과, agent 루프 전체의 최종 응답 형식을 지정하는 response_format(agent-level, create_agent의 파라미터).

핵심 전제: 구조화된 출력은 "모델에게 예쁘게 부탁하는 기술"이 아니라, "모델의 자유 출력을 코드가 검증 가능한 계약(schema)으로 바꾸고, 계약 위반 시 되돌리는 제어 흐름을 agent 루프 안에 내장하는 것"이다. 이 관점이 있어야 아래에서 다루는 세 가지 전략의 차이와 실패 처리 방식이 자연스럽게 이해된다.

ToolStrategy — 스키마를 강제된 tool 호출로 감싸기

ToolStrategy는 지정한 스키마(Pydantic 모델/TypedDict/dataclass/JSON schema)를 하나의 tool 정의로 변환하고, 모델이 반드시 그 tool을 호출하도록(tool_choice를 강제) 만든다. 모델은 tool 호출 인자로 스키마에 맞는 JSON을 생성하고, LangChain이 그 인자를 코드로 파싱·검증한다. 이 방식은 tool calling을 지원하는 모든 provider에서 동작하므로 범용성이 가장 높다. 

from langchain.agents import create_agent
from langchain.agents.structured_output import ToolStrategy
from pydantic import BaseModel

class TicketTriage(BaseModel):
    intent: str
    priority: str
    needs_human: bool

agent = create_agent(
    model="claude-sonnet-5",
    tools=[search_kb_tool],
    response_format=ToolStrategy(
        schema=TicketTriage,
        handle_errors=True,  # 검증 실패 시 에러를 모델에 되돌려 재시도
    ),
)
result = agent.invoke({"messages": [...]})
result["structured_response"]  # -> TicketTriage 인스턴스

ProviderStrategy — provider의 네이티브 구조화 출력 API 사용

OpenAI, Anthropic(Claude), Gemini, xAI(Grok) 같은 일부 provider는 API 레벨에서 네이티브 구조화된 출력을 지원한다(예: OpenAI의 JSON schema response format). ProviderStrategy는 이 네이티브 기능을 직접 사용하므로, 지원되는 provider/모델 조합에서는 가장 신뢰할 수 있는 방법이다 — tool 호출을 흉내 낼 필요 없이 API 자체가 스키마 준수를 보장하기 때문이다. 

from langchain.agents.structured_output import ProviderStrategy

agent = create_agent(
    model="gpt-5",
    response_format=ProviderStrategy(schema=TicketTriage),
)
실무 주의: "ProviderStrategy가 항상 더 안전하다"고 단정하면 안 된다. 2026-07 시점 실제 이슈 트래커에는 ProviderStrategy를 써도 OpenAI의 strict: true JSON schema 플래그가 LangChain을 통해 완전히 노출되지 않는 경우, 그리고 일부 모델(예: Gemini 계열 + tools 동시 사용)에서 ProviderStrategy를 요청해도 조용히 ToolStrategy로 폴백하는 사례가 보고돼 있다. 즉 "어떤 전략을 요청했는가"와 "실제로 어떤 전략이 적용됐는가"가 다를 수 있으므로, LangSmith 트레이싱(Day 12)으로 실제 요청 payload를 확인하는 습관이 필요하다.

AutoStrategy — 스키마만 넘기면 자동으로 고른다

response_format에 ToolStrategy/ProviderStrategy로 감싸지 않고 스키마 타입을 직접 넘기면, LangChain이 현재 모델이 provider-native 구조화 출력을 지원하면 ProviderStrategy를, 지원하지 않으면 ToolStrategy를 자동으로 선택한다. 빠르게 시작할 때는 편리하지만, 어떤 전략이 실제로 선택됐는지 모른 채로 프로덕션에 배포하면 위에서 언급한 폴백/제약 문제를 인지하지 못한 채 넘어갈 위험이 있다. 

# 스키마를 직접 넘기면 AutoStrategy가 provider 지원 여부를 보고 전략을 고른다
agent = create_agent(model="claude-sonnet-5", response_format=TicketTriage)

검증 실패는 어떻게 처리되는가 — self-correction 루프

ToolStrategy는 handle_errors 파라미터로 실패 처리 방식을 제어한다 (bool/문자열 커스텀 메시지/특정 예외 타입/callable). 기본적으로 검증에 실패하면 LangChain은 에러 내용을 담은 ToolMessage를 대화에 추가하고 모델을 다시 호출해서 스스로 출력을 고치게 한다. 이건 Day 2에서 다룬 "tool 에러를 예외로 던지지 않고 모델에게 되돌려 스스로 고치게 한다"는 원칙과 정확히 같은 패턴이 구조화된 출력에도 적용된 것이다.

런어웨이 위험: 재시도 상한을 두지 않으면, 모델이 계속 스키마를 못 맞출 때 모델 호출이 무한히 반복될 수 있다. Day 3에서 다룬 ModelCallLimitMiddleware를 구조화된 출력 재시도 루프에도 함께 걸어두는 것이 실무 원칙이다.

알려진 한계 — 실무에서 맞닥뜨리는 진짜 제약

  • 정적 스키마: response_format은 agent 생성 시점에 고정되는 정적 스키마다. "대화 상태에 따라 허용 가능한 필드 값을 런타임에 좁히고 싶다" 같은 동적 스키마는 2026-07 시점 공식적으로 지원되지 않는다(관련 GitHub 이슈로 요청은 올라가 있으나 미해결).
  • Claude extended thinking과의 충돌: ToolStrategy는 구조화된 출력을 위해 tool_choice를 강제로 설정하는데, Anthropic API는 tool 사용을 강제할 때 extended thinking을 함께 켜는 것을 허용하지 않는다. 그래서 "reasoning 모델의 사고 과정도 보고 싶고, 최종 답은 구조화된 형식으로도 받고 싶다"는 요구를 create_agent 하나로는 바로 만족시키지 못하는 조합이 실제로 존재한다. 이런 경우 수동으로 LangGraph를 구성하거나, 마지막 턴에서만 with_structured_output을 별도로 호출하는 식으로 우회해야 한다.

with_structured_output vs response_format — 언제 어느 것을 쓰는가

with_structured_output은 ChatModel 하나에 스키마를 바인딩하는 model-level API다. Tool을 호출하며 여러 턴을 도는 agent 루프가 필요 없고, "이 입력을 한 번에 구조화된 형태로 추출/분류"하는 단발성 호출이면 이걸로 충분하다. 반면 agent가 tool을 여러 번 호출하며 작업을 수행한 뒤, 최종 결과만 구조화된 형태로 받고 싶다면 create_agent(response_format=...)가 맞는 선택이다 — agent 루프 중간의 tool 호출과 최종 구조화 응답은 서로 다른 관심사이기 때문이다.

스트리밍 — stream_mode 기본기

Agent 응답은 수 초에서 수십 초가 걸릴 수 있다. 사용자가 아무 반응 없이 기다리게 하지 않으려면 결과가 완성되기 전에 부분 결과를 흘려보내는 스트리밍이 필요하다. LangGraph 기반 agent(create_agent도 내부적으로 컴파일된 그래프)는 .stream()/.astream()의 stream_mode 파라미터로 무엇을 흘릴지 고른다.

stream_mode흘려보내는 것대표 용도
values매 단계 후의 전체 state 스냅샷클라이언트가 단순히 "현재 전체 상태"만 렌더링하면 될 때
updates매 단계에서 변경된 부분(diff)만대화 이력이 길어질수록 payload가 커지는 걸 피해야 할 때
messages토큰 단위의 AIMessageChunkChatGPT 스타일의 타이핑 효과 UI
custom개발자가 get_stream_writer()로 직접 흘려보내는 임의 이벤트tool 실행 중 "검색 중 40%..." 같은 진행 상태 보고
debug내부 실행 이벤트 전부(가장 verbose)로컬 디버깅 전용, 프로덕션 UI에는 부적합

stream_mode=["updates", "custom"]처럼 리스트로 여러 모드를 동시에 요청할 수 있고, 이때 각 이벤트는 어떤 모드에서 온 것인지 태그가 붙어서 나온다. 

for mode, chunk in agent.stream(
    {"messages": [...]},
    stream_mode=["updates", "messages"],
):
    if mode == "messages":
        token, metadata = chunk
        # token.content를 그대로 프론트로 흘려보내 타이핑 효과 구현
    elif mode == "updates":
        # 어떤 노드가 state의 어떤 부분을 바꿨는지 diff 형태로 확인
        ...
values vs updates 선택 기준: values는 클라이언트 구현이 단순하지만 대화 이력이 길어질수록 매 이벤트마다 전체 상태를 다시 보내는 비용이 누적된다. updates는 payload가 작지만 클라이언트가 diff를 스스로 누적해서 재구성해야 하므로 클라이언트 로직이 더 복잡해진다. "payload 비용이 문제인가, 클라이언트 복잡도가 문제인가"로 트레이드오프를 판단한다.

새로 등장한 계층 — event streaming(typed projection)

2026년 들어 LangChain/LangGraph는 stream_mode 위에 더 상위의 이벤트 프로토콜 계층을 추가했다(astream_events(..., version="v3")와 이를 감싸는 stream.messages/stream.tools/stream.lifecycle 같은 typed projection API). 이 계층은 메시지를 content block(텍스트, reasoning, tool-call 등) 단위로 모델링해서 "블록이 시작 → 여러 delta → 블록 종료"라는 명시적 경계를 갖게 하고, tool 실행 이벤트를 tool call ID로 원래 메시지의 tool-call 블록에 다시 연결할 수 있게 해준다. 또한 서브그래프/서브에이전트 상태를 추적하는 lifecycle 채널, 장시간 실행 중 연결이 끊겼다가 재연결(reconnect)하는 시나리오까지 다룬다.

이 계층은 stream_mode를 대체하는 것이 아니라, 복잡한 멀티 에이전트/서브그래프 UI(예: 여러 서브에이전트가 동시에 작업하는 화면을 각각 독립적으로 렌더링해야 하는 경우)를 위한 상위 추상화다. 대부분의 실무 코드는 여전히 stream_mode로 충분하지만, "최근 LangChain 스트리밍 아키텍처에 뭐가 추가됐는가"를 아는 것은 이 분야가 얼마나 빠르게 움직이는지 보여주는 senior-level 신호다.

구조화된 출력과 스트리밍의 충돌

ToolStrategy 기반 구조화된 출력은 하나의 강제된 tool 호출이다. 이 tool 호출의 인자(JSON)는 토큰 단위로 생성되긴 하지만, 완성되지 않은 중간 JSON 조각은 대부분 사용자에게 보여줄 수 없는 형태다(예: {"intent": "refu처럼 잘린 문자열). 자유 텍스트 응답처럼 "글자가 타이핑되는 느낌"의 스트리밍 UX를 그대로 구조화된 필드에 적용하면 깨진 조각이 화면에 노출되는 사고가 난다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "with_structured_output 한 줄 써봤는가"가 아니라, "모델의 확률적 출력을 애플리케이션이 신뢰할 수 있는 계약으로 바꾸는 전체 파이프라인(전략 선택 → 검증 → 실패 처리 → 재시도 상한)을 설계할 수 있는가", 그리고 "지연시간이 존재하는 시스템에서 사용자 체감 성능을 어떻게 관리하는가"를 함께 본다.

  • 전략 선택의 근거: ToolStrategy/ProviderStrategy/AutoStrategy의 차이와, "무엇을 요청했는가"와 "실제로 무엇이 적용됐는가"가 다를 수 있다는 것을 아는가
  • 실패를 설계에 포함하는가: 검증 실패 시 재시도가 자동으로 일어난다는 것과, 상한을 두지 않으면 비용이 폭주할 수 있다는 것을 동시에 이해하는가
  • 스트리밍 trade-off 판단력: values/updates/messages/custom을 상황에 맞게 고르고, payload 비용과 클라이언트 복잡도 사이의 균형을 설명할 수 있는가
  • 두 주제의 교차점을 보는가: 구조화된 출력과 토큰 스트리밍이 근본적으로 다른 성격의 요구라는 것, 그리고 실무에서 이를 혼합 전략으로 푸는 방법을 아는가
Senior 개발자는 "구조화된 출력을 쓴다"와 "스트리밍을 쓴다"를 각각의 기능으로 나열하지 않고, "사용자 체감 지연시간, 데이터 신뢰성, 비용이라는 세 축 사이에서 어떤 조합을 선택했는지"를 하나의 설계 결정으로 설명한다.
실무 연결

실제 상황과 연결하기

상황 1: 고객 문의 분류 결과를 downstream 라우팅 시스템에 넘겨야 함

고객 문의 agent의 최종 응답이 자유 텍스트라서, 이걸 다시 정규식/프롬프트로 파싱해 라우팅 시스템에 넘기다가 파싱 실패로 티켓이 잘못된 큐에 들어가는 사고가 반복됐다. → response_format=ToolStrategy(schema=TicketTriage, handle_errors=True)로 바꿔서 intent/priority/needs_human 필드를 코드가 검증한 뒤에만 downstream으로 넘기게 했다. 검증 실패는 모델이 스스로 재시도하고, 그래도 실패하면 명시적으로 사람에게 에스컬레이션한다.

상황 2: Reasoning 모델의 사고 과정과 구조화된 최종 답을 동시에 받고 싶음

Claude의 extended thinking을 켜서 판단 과정을 로깅하려던 팀이 동시에 response_format을 추가했더니 API가 "thinking이 켜진 상태에서 tool 사용을 강제할 수 없다"는 에러를 반환했다. → 원인은 ToolStrategy가 tool_choice를 강제하기 때문이었고, 팀은 마지막 턴에서 with_structured_output을 별도 호출로 분리하는 식으로 우회했다. "편의 API 하나로 모든 조합이 가능하지는 않다"는 걸 실제 장애로 배운 사례다.

상황 3: 챗봇 응답이 늦게 느껴진다는 사용자 불만

Agent가 tool을 여러 번 호출한 뒤에야 최종 텍스트를 반환해서, 사용자는 그 사이 아무 반응도 못 보고 수 초를 기다려야 했다. → stream_mode="messages"로 최종 답변의 토큰 단위 스트리밍을 적용했지만, tool 실행 구간에는 여전히 "멈춘 것처럼 보이는" 공백이 남았다. → custom 모드로 tool 내부에서 get_stream_writer()를 호출해 "검색 중...", "문서 3건 확인 중..." 같은 진행 상태를 함께 흘려보내 체감 지연시간을 줄였다.

상황 4: 구조화된 출력 필드를 그대로 스트리밍했다가 깨진 JSON이 화면에 노출

구조화된 응답도 "타이핑되는" 느낌을 주고 싶어서 stream_mode="messages"로 tool 호출 인자의 토큰을 그대로 화면에 흘렸다가, {"prior처럼 잘린 조각이 사용자에게 노출되는 사고가 났다. → 정책을 바꿔서 구조화된 필드는 검증이 끝난 뒤 한 번에 렌더링하고, 자유 텍스트로 된 설명/요약 필드만 토큰 단위로 스트리밍하도록 스키마를 "스트리밍 가능 필드"와 "완료 후 렌더링 필드"로 나눠 설계했다.

답변 사고 순서

면접에서 이렇게 답변 구조를 잡아라

1
구조화된 출력이 해결하는 문제부터 짚기 "자유 텍스트를 prompt로 JSON처럼 유도하는 방식은 산발적으로 실패합니다. 구조화된 출력은 스키마를 코드로 선언하고, 모델 출력을 검증한 뒤 실패하면 되돌려 재시도시키는 계약을 agent 루프에 내장하는 방식입니다."
2
세 전략을 구분해서 설명하기 "ToolStrategy는 스키마를 강제 tool 호출로 감싸 범용적으로 동작하고, ProviderStrategy는 provider의 네이티브 구조화 출력 API를 써서 지원 범위 내에서 가장 신뢰할 수 있습니다. AutoStrategy는 이 둘 중 자동으로 고르지만, 실제로 뭐가 선택됐는지는 트레이싱으로 확인해야 합니다."
3
실패 처리와 비용 상한을 함께 언급하기 "검증 실패는 handle_errors로 자동 재시도되는데, 상한이 없으면 무한 재시도로 비용이 폭주할 수 있어 ModelCallLimitMiddleware 같은 가드레일을 함께 겁니다."
4
스트리밍 trade-off를 구조적으로 설명하기 "values는 구현이 단순하지만 payload가 누적되고, updates는 payload가 작지만 클라이언트가 diff를 재구성해야 합니다. messages는 토큰 단위 타이핑 UX, custom은 tool 실행 중 진행 상태처럼 임의 이벤트를 위해 씁니다."
5
두 주제의 충돌 지점까지 연결하기 (Senior 포인트) "구조화된 출력의 tool 호출 인자는 완성 전까지 사용자에게 보여줄 수 없는 조각이라서, 구조화된 필드는 완료 후 렌더링하고 자유 텍스트 필드만 토큰 스트리밍하는 혼합 전략을 씁니다. 최근에는 이런 요구를 위해 content block 단위의 event streaming 계층도 추가됐습니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"구조화된 출력은 모델의 자유 텍스트 출력을 코드가 검증 가능한 계약으로 바꾸는 계층입니다. create_agent의 response_format에 ToolStrategy를 쓰면 스키마가 강제된 tool 호출로 변환되어 모든 tool calling provider에서 동작하고, ProviderStrategy를 쓰면 provider의 네이티브 구조화 출력 API를 직접 사용해 지원 범위 내에서 가장 신뢰할 수 있습니다. 스키마를 직접 넘기면 AutoStrategy가 둘 중 하나를 자동으로 고르지만, 실제로 어떤 전략이 적용됐는지는 LangSmith 트레이싱으로 확인하는 게 안전합니다 — 일부 모델 조합에서는 요청한 전략이 조용히 폴백되는 경우도 있기 때문입니다.

검증에 실패하면 handle_errors 설정에 따라 에러 내용이 모델에 다시 전달되어 스스로 고치는 재시도가 일어나는데, 재시도 상한을 걸어두지 않으면 모델이 계속 스키마를 못 맞출 때 호출이 폭주할 수 있어 호출 제한 middleware를 함께 씁니다. 또한 ToolStrategy는 tool_choice를 강제하기 때문에, Claude의 extended thinking처럼 강제 tool 사용과 충돌하는 기능이 있다는 것도 실제 제약으로 인지하고 있어야 합니다.

스트리밍은 stream_mode로 무엇을 흘릴지 고르는데, values는 전체 상태 스냅샷이라 구현이 단순하지만 대화가 길어질수록 payload가 커지고, updates는 diff만 보내 가볍지만 클라이언트가 상태를 스스로 재구성해야 합니다. messages는 토큰 단위 타이핑 UX에, custom은 tool 실행 중 진행 상태 같은 임의 이벤트에 씁니다. 다만 구조화된 출력의 tool 호출 인자는 완성되기 전까지 사용자에게 보여줄 수 없는 조각이기 때문에, 저는 구조화된 필드는 검증이 끝난 뒤 한 번에 렌더링하고 자유 텍스트 필드만 토큰 스트리밍하는 방식으로 두 요구를 함께 만족시킵니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "ProviderStrategy가 항상 ToolStrategy보다 더 신뢰할 수 있나요?"

약한 답변: "네, provider가 직접 지원하니까 항상 더 안전합니다."

강한 답변: "일반적으로는 그렇지만 절대적이지는 않습니다. Provider별로 strict 검증 지원 수준이 다르고, 일부 provider/모델 조합에서는 프레임워크가 요청한 대로 완전히 노출되지 않는 옵션(예: JSON schema의 strict 플래그)이 있을 수 있습니다. 또한 tools와 response_format을 동시에 쓰는 특정 모델 조합에서는 ProviderStrategy를 요청해도 내부적으로 ToolStrategy로 폴백하는 경우도 보고돼 있습니다. 그래서 '요청한 전략'과 '실제 적용된 전략'을 구분해서 검증하는 습관이 필요합니다."

F2. "response_format을 쓸 때 tool_choice가 강제되는 것과 extended thinking이 왜 충돌하나요?"

강한 답변: "ToolStrategy는 모델이 반드시 구조화된 출력용 tool을 호출하도록 tool_choice를 강제합니다. 그런데 Anthropic API 자체가 tool 사용을 강제하는 상태에서는 extended thinking(모델의 내부 사고 블록)을 함께 켤 수 없다는 제약을 두고 있습니다. 이건 LangChain의 버그가 아니라 provider API의 제약이 그대로 전파된 것이라, 이 조합이 필요하면 create_agent의 편의 기능 대신 수동으로 LangGraph를 구성하거나, 사고 과정은 별도 턴에서 확인하고 마지막에만 구조화된 출력을 받는 식으로 흐름을 나눠야 합니다."

F3. "stream_mode를 리스트로 여러 개 넘기면 이벤트를 어떻게 구분하나요?"

강한 답변: "각 이벤트가 (mode, payload) 튜플 형태로 나와서, mode 값을 보고 어떤 스트림에서 온 이벤트인지 분기 처리합니다. 예를 들어 updates와 messages를 함께 요청하면, 노드 실행 결과 diff와 토큰 단위 텍스트가 같은 반복문 안에서 태그가 붙은 채로 함께 흘러나옵니다. 이걸 활용하면 하나의 스트림 소비 루프에서 '진행 상태'와 '실제 답변 텍스트'를 동시에 다른 UI 요소에 매핑할 수 있습니다."

F4. "구조화된 출력 검증에 계속 실패하면 무한 루프에 빠질 위험은 없나요?"

약한 답변: "handle_errors를 켜두면 자동으로 해결됩니다."

강한 답변: "handle_errors는 실패를 모델에게 되돌려 재시도를 유도할 뿐, 재시도 횟수 자체를 제한하지는 않습니다. 모델이 계속 스키마를 못 맞추면 이론상 호출이 계속 반복될 수 있으므로, Day 3에서 다룬 ModelCallLimitMiddleware나 재시도 횟수를 직접 세는 커스텀 middleware로 상한을 명시적으로 걸어야 합니다. '자동 재시도'와 '재시도 상한'은 반드시 함께 설계해야 하는 짝입니다."

F5. "새로 나온 event streaming(typed projection)은 언제 stream_mode 대신 써야 하나요?"

강한 답변: "대부분의 단일 agent, 단순 챗봇 UI는 여전히 stream_mode로 충분합니다. 하지만 여러 서브에이전트가 동시에 작업하는 복잡한 멀티 에이전트 UI에서, 각 서브에이전트의 진행 상태를 독립적으로 렌더링하거나, tool 실행 이벤트를 원래 메시지의 tool-call 블록과 정밀하게 매칭하거나, 장시간 실행 중 연결이 끊겼다가 재연결해야 하는 시나리오라면 content block과 lifecycle 채널을 갖춘 event streaming 계층이 더 적합합니다. 이건 대체재가 아니라 더 복잡한 프로덕션 UI를 위한 상위 추상화로 이해하고 있습니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • ToolStrategy/ProviderStrategy/AutoStrategy를 동작 방식과 신뢰 범위로 구분한다
  • "요청한 전략"과 "실제 적용된 전략"이 다를 수 있다는 것을 알고 트레이싱으로 검증하는 습관을 언급한다
  • 검증 실패의 자동 재시도와 재시도 상한을 짝으로 설계한다
  • values/updates/messages/custom을 payload 비용·클라이언트 복잡도·UX 관점에서 고른다
  • 구조화된 출력과 토큰 스트리밍의 근본적인 충돌 지점과 혼합 전략을 설명한다

약한 답변의 특징

  • "with_structured_output 쓰면 된다"고만 말하고 agent-level response_format과의 차이를 구분 못 함
  • ProviderStrategy가 항상 무조건 더 안전하다고 근거 없이 단정
  • 재시도가 자동으로 된다는 것만 알고 비용 폭주 위험을 언급하지 않음
  • stream_mode는 "그냥 messages 쓰면 된다"로 단순화해버림
  • 구조화된 출력도 자유 텍스트처럼 자연스럽게 스트리밍된다고 잘못 가정
핵심 차이: 강한 답변은 각 API를 "기능 설명"으로 나열하지 않고, 신뢰성·비용·UX라는 세 축 사이의 trade-off를 실제 실패 사례와 함께 설명한다. 이는 실무에서 이 결정을 이미 내려본 경험이 있다는 신호다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "prompt에 'JSON으로만 답해줘'라고 쓰면 충분하다"

사실: 이 방법은 마크다운 코드블럭, 설명 문장 덧붙이기, 필드 오타 등으로 산발적으로 실패한다. 구조화된 출력 API는 여기에 스키마 검증과 실패 시 재시도까지 포함된 계약을 제공한다는 점이 본질적으로 다르다.

오해 2: "ProviderStrategy가 항상 ToolStrategy보다 안전하다"

사실: Provider/모델 조합에 따라 strict 지원 범위가 다르고, 특정 조합에서는 요청한 전략이 조용히 폴백되는 경우도 있다. "요청 = 실제 적용"이 아니라는 것을 트레이싱으로 확인해야 한다.

오해 3: "stream_mode=values가 기본값이자 더 나은 선택이다"

사실: values는 구현이 단순할 뿐, 항상 더 나은 것은 아니다. 대화 이력이 길어지는 서비스에서는 payload 비용이 누적되는 updates보다 오히려 더 비효율적일 수 있다.

오해 4: "구조화된 출력도 자유 텍스트처럼 토큰 단위로 자연스럽게 스트리밍된다"

사실: 강제된 tool 호출의 인자는 완성 전까지 사용자에게 보여주기 부적절한 중간 조각이다. 자유 텍스트 필드와 구조화된 필드를 구분해서 각기 다른 렌더링 전략(토큰 스트리밍 vs 완료 후 일괄 렌더링)을 적용해야 한다.

실무 체크리스트

구조화된 출력·스트리밍 설계 전 점검표

항목확인할 것리스크
전략 선택 ToolStrategy/ProviderStrategy/AutoStrategy 중 어떤 게 실제로 적용됐는지 트레이싱으로 확인했는가 기대한 신뢰성 미달
재시도 상한 handle_errors의 자동 재시도에 호출 횟수 상한(middleware)을 함께 걸었는가 비용 폭주

3. Claude extended thinking과 강제된 구조화 출력이 왜 함께 쓰기 어려운지 원인까지 설명할 수 있는가?
4. stream_mode=values와 updates 중 어떤 상황에서 어떤 것을 선택할지 payload 비용과 클라이언트 복잡도로 설명할 수 있는가?
5. 구조화된 출력 필드와 자유 텍스트 필드를 스트리밍 UI에서 왜 다르게 렌더링해야 하는지 구체적 사고 예시로 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • 구조화된 출력은 모델의 자유 출력을 코드가 검증 가능한 계약으로 바꾸는 계층이다. ToolStrategy(범용, 강제 tool 호출)와 ProviderStrategy(provider 네이티브, 지원 범위 내 최고 신뢰성)를 구분하고, AutoStrategy가 자동으로 고른 결과를 트레이싱으로 검증해야 한다.
  • 검증 실패는 자동 재시도되지만 상한이 없으면 비용이 폭주할 수 있다 — 재시도와 호출 제한 middleware는 항상 짝으로 설계한다.
  • ToolStrategy의 강제 tool_choice는 Claude extended thinking 등 일부 모델 기능과 충돌한다. "편의 API 하나로 모든 조합이 가능하지는 않다"는 provider 제약을 인지해야 한다.
  • stream_mode(values/updates/messages/custom/debug)는 payload 비용·클라이언트 복잡도·UX 목적에 따라 선택한다. 리스트로 여러 모드를 동시에 요청할 수 있다.
  • 더 복잡한 멀티 에이전트 UI를 위한 content block 기반 event streaming(typed projection) 계층이 stream_mode 위에 추가되고 있다 — 대체가 아니라 상위 추상화다.
  • 구조화된 출력과 토큰 스트리밍은 근본적으로 다른 성격이다. 구조화된 필드는 완료 후 렌더링하고, 자유 텍스트 필드만 토큰 스트리밍하는 혼합 전략이 실무 표준이다.
