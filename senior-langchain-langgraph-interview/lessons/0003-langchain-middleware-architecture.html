Day 3 · LangChain / LangGraph Interview

Middleware 아키텍처

서브클래싱 대신 middleware로 agent를 확장하는 이유, hook 종류와 실행 순서, 그리고 프로덕션에서 middleware 조합을 잘못 설계했을 때 생기는 문제
버전 기준: 이 레슨은 2026-07 기준 LangChain 1.x (GA)의 langchain.agents.middleware 모듈을 기준으로 작성됐다. Middleware는 1.0에서 가장 크게 바뀐 확장 지점이며, 과거의 AgentExecutor 서브클래싱이나 create_react_agent의 개별 파라미터(예: pre_model_hook)로 하던 커스터마이징을 대체한다. 실제 클래스/함수 시그니처는 면접 준비 시점에 공식 문서(reference.langchain.com)로 다시 확인하라 — hook 이름과 동작 방식은 안정화됐지만 built-in middleware 목록은 계속 추가되고 있다.
면접 질문
Q1 — 주력 질문 "LangChain 에이전트를 커스터마이징할 때 왜 서브클래싱 대신 middleware를 쓰나요? before_model/after_model 같은 hook과 wrap_model_call/wrap_tool_call 같은 hook은 뭐가 다르고, 언제 각각을 선택하나요?"
Q2 — 변형 "여러 middleware를 동시에 등록하면 실행 순서는 어떻게 결정되나요? 순서를 잘못 배치하면 실무에서 어떤 문제가 생길 수 있나요?"
Q3 — 변형 "SummarizationMiddleware나 HumanInTheLoopMiddleware 같은 built-in middleware를 프로덕션에 적용할 때 뭘 검증해야 하나요?"
전제 개념

Middleware란 무엇이고, 왜 서브클래싱을 대체했는가

문제의 출발점: agent 루프를 어디서 가로챌 것인가

Agent의 기본 동작은 단순하다 — "모델을 호출한다 → 모델이 tool 호출을 요청하면 tool을 실행한다 → 결과를 다시 모델에 보여준다 → 반복한다." 그런데 실무에서는 이 루프의 특정 지점에 공통 로직을 끼워 넣어야 하는 경우가 계속 생긴다. 예를 들어 "모델 호출 전에 대화가 너무 길면 요약한다", "tool 호출 결과에서 개인정보를 지운다", "위험한 tool은 사람 승인을 받는다", "모델 호출 비용이 임계치를 넘으면 강제로 멈춘다" 같은 것들이다.

과거 AgentExecutor 시절에는 이런 요구가 생길 때마다 클래스를 서브클래싱하거나 개별 파라미터(예: pre_model_hook, post_model_hook)를 하나씩 늘리는 방식으로 대응했다. 문제는 요구사항이 늘어날수록 서브클래스 계층이 꼬이고, "이 로직들이 정확히 어떤 순서로 실행되는지"를 코드를 직접 추적해야만 알 수 있었다는 점이다. Supervisor 패턴, swarm 패턴, deep agent(계획 목록을 관리하는 에이전트) 패턴은 사실상 각자 비슷한 확장 로직을 매번 처음부터 다시 구현한 것에 가까웠다.

Middleware는 이 문제를 "agent 루프의 정해진 지점(hook)에 함수를 꽂아 넣는 방식"으로 재구성한다. 웹 서버의 middleware(요청이 들어올 때 인증을 검사하고, 응답이 나갈 때 로깅하는 계층)와 정확히 같은 사고방식이다 — 핵심 로직(모델 호출, tool 실행)은 그대로 두고, 그 앞뒤에 독립적으로 조립 가능한 조각을 끼워 넣는다.

핵심 전제: Middleware는 "부가 기능을 위한 편의 도구"가 아니라, agent 확장의 표준 단위다. Supervisor나 deep agent 같은 이전의 "개별 아키텍처"들도 이제는 middleware 여러 개를 조합해서 재현할 수 있다. 즉 "어떤 아키텍처 패턴을 쓸 것인가"라는 질문이 상당 부분 "어떤 middleware를 어떤 순서로 조합할 것인가"라는 질문으로 바뀌었다.

Hook은 두 종류다 — "생명주기형(node-style)"과 "감싸기형(wrap-style)"

Middleware가 끼어들 수 있는 지점은 크게 두 가지 방식으로 나뉜다. 이 구분을 정확히 이해하는 것이 이 주제 전체의 핵심이다.

구분Hook동작 방식반환 값
생명주기형
(node-style)		before_agentagent 실행 시작 전 1회dict (state 업데이트, reducer로 병합) 또는 None
before_model매 모델 호출 직전
after_model매 모델 응답 직후
after_agentagent 실행 종료 후 1회
감싸기형
(wrap-style)		wrap_model_call모델 호출 자체를 handler 콜백으로 감쌈ModelResponse / AIMessage
wrap_tool_calltool 실행 자체를 handler 콜백으로 감쌈ToolMessage / Command

생명주기형 hook은 "관찰하고 state를 업데이트"하는 데 적합하다. dict를 반환하면 LangGraph의 reducer 메커니즘으로 state에 병합된다. 반면 감싸기형 hook은 실행 자체를 제어할 수 있다 — handler를 호출하면 실제 모델/tool 호출이 일어나고, 호출하지 않으면 아예 실행을 건너뛸 수도(short-circuit), 여러 번 호출하면 재시도를 구현할 수도 있다.

선택 기준: "모델/tool 호출이 실제로 일어나는지 자체를 통제해야 하는가?" — 그렇다면 wrap_model_call/wrap_tool_call. "호출 전후에 관찰하거나 부가 정보를 state에 기록하기만 하면 되는가?" — 그렇다면 before_model/after_model이 더 단순하고 안전하다. Retry, fallback, 캐싱처럼 호출 자체를 여러 번 하거나 건너뛰는 로직은 반드시 wrap 계열이어야 한다.

여러 middleware를 등록하면? — 실행 순서는 "양파(onion)" 구조다

create_agent(model, tools, middleware=[m1, m2, m3])처럼 여러 middleware를 리스트로 등록하면, 실행 순서는 등록한 순서 그대로 "들어가고", 반대 순서로 "나온다." 

# before_* / wrap_*: 등록 순서대로 (m1 → m2 → m3 → 실제 모델 호출)
# after_*: 반대 순서로 (실제 모델 호출 → m3 → m2 → m1)

#            m1 진입 → m2 진입 → m3 진입 → [ 모델 호출 ] → m3 탈출 → m2 탈출 → m1 탈출

이건 우연이 아니라 웹 서버 middleware 체인과 동일한 설계다. wrap_model_call 관점에서 보면 m1이 가장 바깥 껍질이고, m3이 실제 모델 호출을 가장 가까이서 감싼다. 즉 먼저 등록한 middleware일수록 더 넓은 범위(다른 모든 middleware를 포함해서)를 감싼다는 뜻이다. 이 순서를 착각하면 "PII를 지운 다음에 요약해야 하는데, 순서가 반대로 등록돼서 요약본에 개인정보가 남는" 것 같은 실제 버그가 생긴다.

제어 흐름(jump_to) — middleware가 그래프의 다음 목적지를 바꾼다

생명주기형 hook은 단순 관찰을 넘어 그래프의 다음 노드를 강제로 바꾸는 것도 할 수 있다. 반환하는 dict에 "jump_to" 키를 포함시키면 되는데, 이때 middleware는 can_jump_to로 자신이 어떤 목적지로 점프할 수 있는지 미리 선언해야 한다 (허용된 값은 "tools", "model", "end"). 

from langchain.agents.middleware import AgentMiddleware, hook_config

class MessageLimitMiddleware(AgentMiddleware):
    def __init__(self, max_messages: int = 20):
        self.max_messages = max_messages

    @hook_config(can_jump_to=["end"])
    def before_model(self, state, runtime):
        if len(state["messages"]) >= self.max_messages:
            # 모델을 더 호출하지 않고 즉시 그래프를 종료시킨다
            return {
                "messages": [AIMessage(content="대화가 너무 길어져 상담원에게 연결합니다.")],
                "jump_to": "end",
            }
        return None

이 메커니즘 덕분에 "무한 루프에 빠진 agent를 강제 종료", "위험 신호가 감지되면 즉시 사람에게 넘기기" 같은 가드레일을 middleware 하나로 캡슐화할 수 있다. 이건 Day 6(Guardrails/Context Engineering)에서 더 깊게 다룬다.

작성 방식 두 가지 — 데코레이터 vs 클래스

간단한 로직은 데코레이터(@before_model, @wrap_model_call 등)로 함수 하나만 작성하면 된다. 상태(설정값, 카운터 등)를 유지하거나 여러 hook을 함께 구현해야 하면 AgentMiddleware 서브클래스로 작성한다. 

from langchain.agents.middleware import wrap_model_call
from langchain.agents.middleware.types import ModelRequest, ModelResponse
from typing import Callable

@wrap_model_call
def retry_on_rate_limit(
    request: ModelRequest,
    handler: Callable[[ModelRequest], ModelResponse],
) -> ModelResponse:
    # handler(request)를 여러 번 호출할 수 있다 = retry를 middleware 레벨에서 구현
    for attempt in range(3):
        try:
            return handler(request)
        except RateLimitError:
            if attempt == 2:
                raise
            time.sleep(2 ** attempt)

Built-in Middleware — 자주 쓰는 문제는 이미 구현되어 있다

LangChain은 실무에서 반복적으로 필요한 패턴을 built-in middleware로 제공한다. 직접 구현하기 전에 이미 있는지부터 확인하는 것이 실무 원칙이다.

Middleware구현 hook용도
SummarizationMiddlewarebefore_model토큰/메시지 수 임계치 초과 시 오래된 메시지를 요약해서 context 예산을 관리. AI/Tool 메시지 쌍이 끊기지 않게 안전한 절단 지점을 탐색
HumanInTheLoopMiddlewarewrap_tool_call(interrupt)지정된 tool 호출 전에 실행을 멈추고 사람의 승인/거부/수정을 기다림
PIIMiddlewarebefore_model / after_model모델 입력·출력·tool 결과에서 개인정보를 마스킹/삭제/해시 처리
ModelCallLimitMiddlewarebefore_modelthread/run 단위로 모델 호출 횟수를 제한해 폭주 비용을 방지
ToolCallLimitMiddlewarewrap_tool_calltool 전체 또는 tool별로 호출 횟수 상한을 걸어 무한 루프/과다 호출 방지

LLMToolSelectorMiddleware / LLMToolEmulatorwrap_model_call / wrap_tool_calltool이 너무 많을 때 관련 tool만 추리거나, 테스트 단계에서 실제 tool 대신 LLM으로 응답을 흉내냄
from langchain.agents import create_agent
from langchain.agents.middleware import (
    SummarizationMiddleware,
    HumanInTheLoopMiddleware,
    PIIMiddleware,
    ToolCallLimitMiddleware,

agent = create_agent(
    model="claude-sonnet-5",
    tools=[search_tool, send_refund_tool],
    middleware=[
        # 순서 중요: PII 마스킹을 요약보다 먼저 둬서, 요약본에 원본 개인정보가 남지 않게 한다
        PIIMiddleware(pii_types=["email", "phone"], strategy="redact"),

        ToolCallLimitMiddleware(tool_name="send_refund_tool", run_limit=1),

)
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

Middleware 질문은 "create_agent에 tools 목록을 붙일 줄 아는가"가 아니라, "agent 내부 루프의 확장 지점을 이해하고, 여러 확장 지점을 조합했을 때의 상호작용까지 예측할 수 있는가"를 본다. 이건 실무에서 "간단한 튜토리얼 agent"와 "여러 팀의 요구사항이 누적된 프로덕션 agent"의 차이를 가르는 지점이다.

  • 확장 메커니즘의 본질을 이해하는가: middleware가 왜 서브클래싱을 대체했는지, "재사용 가능한 조합 단위"라는 설계 의도를 설명할 수 있는가
  • hook 종류를 구분하고 올바르게 선택하는가: 관찰/state 업데이트에는 생명주기형, 호출 자체를 제어(재시도/폴백/차단)해야 할 때는 감싸기형을 쓰는 판단 기준이 있는가
  • 합성(composition)의 함정을 아는가: 여러 middleware를 등록했을 때 순서가 결과를 바꾼다는 것, 그리고 순서를 잘못 설계했을 때 생기는 실제 실패 사례를 떠올릴 수 있는가
  • 바퀴를 다시 만들지 않는가: built-in middleware의 존재를 알고, 직접 구현하기 전에 "이미 있는 것을 재사용할 수 있는지"부터 검토하는 습관이 있는가
Senior 개발자는 middleware를 "훅 몇 개 붙이는 기능"이 아니라, "agent 아키텍처를 조립 가능한 단위로 분해하는 설계 원칙"으로 설명해야 한다. 이 관점이 있어야 Day 9(Human-in-the-loop), Day 10(Multi-agent), Day 11(Fault Tolerance)에서 다룰 더 복잡한 패턴도 "특수 기술"이 아니라 "middleware 조합의 응용"으로 자연스럽게 연결된다.
실무 연결

실제 상황과 연결하기

상황 1: 고객센터 agent의 대화가 길어져 컨텍스트가 넘침

장시간 상담이 이어지면 메시지 이력이 모델의 context window를 초과해 에러가 나거나 비용이 급증한다. → SummarizationMiddleware를 before_model에 등록해서 임계치를 넘으면 자동으로 오래된 대화를 요약한다. 이때 AI/Tool 메시지 쌍이 끊기지 않게 안전한 절단 지점을 찾는 로직이 이미 구현돼 있으므로, 직접 절단 로직을 짤 필요가 없다.

상황 2: 환불 처리 tool을 agent가 스스로 호출하게 뒀다가 사고 발생

모델이 애매한 요청을 "환불해달라는 뜻"으로 해석해 실제 환불 tool을 호출해버린 사고가 났다. → HumanInTheLoopMiddleware로 금액이 큰 tool 호출은 실행 직전에 멈추고 사람이 승인하게 하며, 동시에 ToolCallLimitMiddleware(tool_name="send_refund_tool", run_limit=1)로 한 세션에서 같은 tool이 반복 호출되는 것 자체를 막는다.

상황 3: PII 마스킹과 요약을 둘 다 등록했는데 순서를 잘못 배치

middleware=[SummarizationMiddleware(...), PIIMiddleware(...)] 순서로 등록했더니, 먼저 등록된 SummarizationMiddleware가 원본 메시지를 요약하면서 요약문 안에 이메일 주소가 그대로 남아버렸다. PII 마스킹은 그 뒤에 실행됐지만 이미 요약본에 개인정보가 녹아든 뒤였다. → 순서를 [PIIMiddleware(...), SummarizationMiddleware(...)]로 바꿔서, 마스킹이 먼저 실행되고 이미 마스킹된 텍스트를 요약하도록 수정했다. "여러 middleware의 등록 순서 = 실행 순서"라는 원칙을 실무에서 검증하지 않으면 이런 사고가 생긴다.

상황 4: 모델 provider 장애로 agent 전체가 에러 응답

주력 모델 provider가 순단(짧은 장애)이 발생하면 agent가 그대로 에러를 반환해 사용자에게 노출됐다. → wrap_model_call 기반 ModelFallbackMiddleware로 1차 실패 시 보조 모델로 자동 전환하고, ModelRetryMiddleware로 일시적 오류는 지수 백오프로 재시도한다. 이때 fallback/retry는 반드시 감싸기형 hook이어야 한다 — 생명주기형 hook은 "호출을 다시 시도"하는 제어 자체를 할 수 없기 때문이다.

middleware가 해결하는 문제부터 짚기 "서브클래싱은 요구사항이 늘수록 계층이 꼬이고 실행 순서를 코드로 추적해야 했습니다. Middleware는 agent 루프의 정해진 지점(hook)에 조합 가능한 단위를 꽂는 방식으로 이를 대체합니다."
2
두 종류의 hook을 구분해서 설명하기 "before/after 같은 생명주기형은 관찰과 state 업데이트에, wrap_model_call/wrap_tool_call 같은 감싸기형은 호출 자체의 실행 여부·횟수를 제어(재시도, 폴백, 차단)하는 데 씁니다."
3
실행 순서(onion 구조)를 명확히 하기 "여러 middleware를 등록하면 등록 순서대로 들어가고 반대 순서로 나옵니다. 이 순서 자체가 결과에 영향을 주기 때문에, 예를 들어 PII 마스킹은 요약보다 먼저 실행되도록 배치해야 합니다."
4
Built-in부터 검토하는 습관 언급하기 "Summarization, HITL, PII, 호출 제한, fallback 같은 자주 나오는 요구는 이미 built-in middleware로 제공되므로, 직접 구현하기 전에 재사용 가능한지부터 확인합니다."
5
가드레일/제어 흐름까지 연결하기 (Senior 포인트) "before_model 같은 hook에서 jump_to로 그래프 실행을 즉시 종료시키는 가드레일도 middleware 하나로 캡슐화할 수 있습니다. 이 덕분에 supervisor나 deep agent 같은 패턴도 middleware 조합의 특수 사례로 이해합니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"Middleware는 agent 루프에서 반복적으로 필요한 확장 로직을, 서브클래싱 없이 조합 가능한 단위로 끼워 넣는 메커니즘입니다. Hook은 크게 두 종류로 나뉘는데, before_model/after_model 같은 생명주기형은 호출 전후에 관찰하고 state를 업데이트하는 용도이고, wrap_model_call/wrap_tool_call 같은 감싸기형은 handler 콜백을 통해 실제 호출 자체를 제어합니다 — 재시도, 폴백, 캐싱처럼 호출을 여러 번 하거나 아예 건너뛰어야 하는 로직은 반드시 감싸기형으로 구현해야 합니다.

여러 middleware를 함께 등록할 때 실행 순서는 등록한 순서 그대로 들어가고 반대 순서로 나오는 양파 구조입니다. 이 순서는 결과에 실제로 영향을 주기 때문에, 예를 들어 PII 마스킹과 요약을 함께 쓴다면 마스킹을 먼저 실행해서 요약본에 원본 개인정보가 남지 않도록 순서를 설계해야 합니다.

실무에서는 요약, human-in-the-loop 승인, PII 마스킹, 호출 횟수 제한, 모델 fallback처럼 자주 나오는 요구는 이미 built-in middleware로 제공되기 때문에, 직접 구현하기 전에 재사용 가능한지부터 검토합니다. 그리고 무한 루프 방지 같은 가드레일은 before_model에서 jump_to로 그래프를 즉시 종료시키는 방식으로 middleware 하나에 캡슐화합니다. 이런 관점으로 보면 supervisor나 deep agent 같은 이전의 개별 아키텍처들도 결국 middleware를 어떤 조합과 순서로 쌓느냐의 문제로 환원됩니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "wrap_model_call 안에서 handler를 두 번 이상 호출하면 무슨 일이 일어나나요?"

약한 답변: "에러가 나거나 안 될 것 같습니다."

강한 답변: "정상적으로 동작합니다. handler는 '실제로 모델을 한 번 호출하는' 콜백일 뿐이라서, 여러 번 호출하면 그만큼 모델이 여러 번 불립니다. 이게 바로 재시도(retry) 로직을 middleware 레벨에서 구현하는 방법입니다. 예외를 잡아서 조건에 따라 handler를 다시 호출하거나, 아예 다른 request(다른 모델)로 handler를 호출하면 fallback이 됩니다. 다만 이때 토큰 비용과 지연시간이 호출 횟수만큼 늘어난다는 점을 감안해서 재시도 상한과 백오프 전략을 반드시 함께 설계해야 합니다."

F2. "같은 tool에 ToolCallLimitMiddleware를 tool별로도, 전체 기준으로도 걸어놨다면 어떤 게 먼저 적용되나요?"

강한 답변: "여러 인스턴스를 등록한 순서대로 wrap_tool_call 체인이 쌓이기 때문에, 등록 순서상 먼저 온 인스턴스가 더 바깥에서 검사합니다. 예를 들어 특정 tool 전용 제한을 먼저 등록하고 전체 제한을 나중에 등록했다면, tool 전용 제한이 먼저 걸립니다. 실무에서는 이걸 우연에 맡기지 않고, '가장 좁은 범위(특정 tool)의 제한을 먼저, 가장 넓은 범위(전체)의 제한을 나중에'라는 규칙을 팀 컨벤션으로 정해둡니다."

F3. "middleware 안에서 예외가 발생하면 agent 전체가 죽나요, 아니면 격리되나요?"

약한 답변: "middleware는 독립적이니까 격리될 것 같습니다."

강한 답변: "격리되지 않습니다. Middleware는 agent 실행 그래프 안에 컴파일되어 들어가는 코드이기 때문에, 여기서 처리하지 않은 예외는 그래프 실행 자체를 중단시킵니다. 그래서 재시도나 fallback을 구현하는 wrap 계열 middleware는 반드시 자기 책임 범위의 예외를 명시적으로 잡아야 하고, '이 예외는 복구 불가능하니 의도적으로 전파시킨다'는 판단도 코드에 드러나야 합니다. Day 2에서 다룬 tool 에러의 '일시적 vs 구조적' 구분이 middleware 설계에도 그대로 적용됩니다."

F4. "커스텀 middleware를 클래스로 작성할 때 state_schema는 왜 필요한가요?"

강한 답변: "Middleware가 agent의 기본 state(메시지 목록 등)에 없는 자기만의 필드(예: 호출 횟수 카운터, 마지막 요약 시각)를 state에 추가해서 그래프 전체에서 공유하고 싶을 때 state_schema로 선언합니다. 이걸 선언하지 않고 임의의 키를 반환하면 reducer가 그 키를 어떻게 병합해야 할지 모르거나, 다른 middleware/노드와 필드명이 충돌할 위험이 있습니다. state_schema는 '이 middleware가 그래프 상태에 어떤 계약을 추가하는지'를 명시적으로 선언하는 장치입니다."

F5. "Deep agent(할 일 목록을 관리하는 agent)나 supervisor 패턴이 왜 middleware 조합으로 재현 가능하다고 하나요?"

강한 답변: "이 패턴들의 실체는 결국 '모델 호출 전에 계획을 확인시키고(before_model 또는 TodoListMiddleware), tool 호출 전에 위험도를 검사하고(wrap_tool_call), 특정 조건에서 다른 흐름으로 점프시키는(jump_to)' 것의 조합입니다. 과거에는 이런 걸 하나의 커스텀 그래프/클래스로 통째로 짰지만, middleware로 쪼개면 '계획 관리'와 '위험 tool 승인'을 독립적으로 켜고 끄거나 다른 프로젝트에 재사용할 수 있습니다. 이게 middleware 아키텍처의 핵심 이점이라고 생각합니다 — 아키텍처 패턴이 코드 재사용 단위가 됐다는 것입니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 생명주기형(관찰/state 업데이트)과 감싸기형(호출 제어)을 구분하고 선택 기준을 제시한다
  • 여러 middleware의 실행 순서(onion)가 결과를 바꾼다는 것을 실제 사례로 설명한다
  • 재시도/fallback을 구현하려면 반드시 wrap 계열이어야 하는 이유를 안다
  • built-in middleware를 먼저 검토하는 습관을 언급한다
  • middleware 예외가 격리되지 않고 그래프 전체에 영향을 준다는 것을 안다

약한 답변의 특징

  • "middleware는 그냥 before/after 콜백 몇 개다" — hook 종류 구분 못 함
  • 여러 middleware를 등록할 때 순서는 상관없다고 단정
  • 재시도/fallback도 before_model 같은 생명주기형 hook으로 구현하려 함
  • PII, 요약, 승인 게이트 같은 기능을 매번 처음부터 직접 구현하려 함
  • "middleware는 독립적이라 에러가 나도 격리될 것"이라고 근거 없이 추측
핵심 차이: 강한 답변은 middleware를 "기능 목록"으로 나열하지 않고, "어떤 hook 종류가 어떤 제어 능력을 갖는지, 여러 개를 합성했을 때 순서가 왜 중요한지"를 구조적으로 설명한다. 이는 곧 프로덕션에서 실제로 middleware 조합을 설계할 능력이 있다는 신호다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "Middleware는 단순 로깅이나 프롬프트 전처리용이다"

사실: wrap_model_call/wrap_tool_call은 handler 호출 여부와 횟수를 제어할 수 있어 재시도, fallback, 캐싱, 심지어 "실제 tool을 호출하지 않고 흉내만 내기"(LLMToolEmulator)까지 구현할 수 있다. 로깅은 middleware가 할 수 있는 일 중 극히 일부다.

오해 2: "여러 middleware를 등록할 때 순서는 상관없다"

사실: 등록 순서는 곧 실행 순서(onion 구조)다. PII 마스킹과 요약처럼 한쪽 결과가 다른 쪽 입력에 영향을 주는 조합에서는 순서를 잘못 두면 조용히 잘못된 결과(예: 요약본에 개인정보 잔존)를 만든다.

오해 3: "복잡한 agent 패턴(supervisor, deep agent)은 middleware와 별개의 고급 기술이다"

사실: 이런 패턴들은 이제 middleware(계획 관리, 위험 tool 승인 게이트, 조건부 jump_to)의 조합으로 재현 가능하다. "완전히 새로운 아키텍처"가 아니라 "middleware를 어떻게 조립하는가"의 문제로 재구성됐다.

오해 4: "before_model에서 return None만 하면 아무 일도 안 일어난다"

사실: 맞는 말이지만, 이 사실 자체가 중요한 설계 지점이다. state를 굳이 바꿀 필요가 없는 middleware(순수 관찰/검증 목적)는 None을 반환하면 되고, 반대로 jump_to를 포함한 dict를 반환하면 그래프의 다음 목적지를 강제로 바꿀 수 있다는 양쪽 극단을 모두 이해해야 middleware를 가드레일 용도로 정확히 설계할 수 있다.

핵심 요약

오늘 반드시 기억할 것

  • Middleware는 서브클래싱을 대체하는 조합 가능한 확장 단위다. Agent 루프의 정해진 지점(hook)에 독립적인 로직을 꽂아 넣는다.
  • Hook은 생명주기형(before/after)과 감싸기형(wrap_model_call/wrap_tool_call) 두 종류다. 관찰·state 업데이트는 생명주기형, 재시도·폴백·차단처럼 호출 자체를 제어해야 하면 반드시 감싸기형이다.
  • 여러 middleware의 실행 순서는 양파(onion) 구조다 — 등록 순서대로 들어가고 반대 순서로 나온다. 순서가 실제 결과를 바꾼다.
  • jump_to로 생명주기형 hook이 그래프의 다음 목적지를 강제로 바꿀 수 있다 — 가드레일과 조기 종료 로직의 핵심 메커니즘이다.
  • Built-in middleware(Summarization, HumanInTheLoop, PII, 호출 제한, Fallback/Retry 등)가 자주 필요한 패턴을 이미 구현해 두었으므로, 직접 구현 전에 재사용 가능성부터 검토한다.
  • Supervisor·swarm·deep agent 같은 이전의 "개별 아키텍처"도 이제는 middleware 조합의 응용으로 이해할 수 있다.
