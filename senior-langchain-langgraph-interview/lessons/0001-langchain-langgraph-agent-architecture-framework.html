Day 1 · LangChain / LangGraph Interview

LangChain·LangGraph 기반 Agent 설계/운영 프레임워크

"Agent = Model + Harness" — create_agent와 StateGraph 중 무엇을, 언제, 왜 선택하는지 구조적으로 설명하는 능력
버전 기준: 이 레슨은 2026-07 기준 LangChain 1.x / LangGraph 1.x (GA, 1.0 이후 no breaking change 정책) API를 기준으로 작성됐다. 레거시 AgentExecutor(체인 기반 에이전트)와 langgraph.prebuilt.create_react_agent는 여전히 동작하지만, 신규 표준은 langchain.agents.create_agent다. 면접 준비 시점에 실제 버전을 changelog로 다시 확인하라.
면접 질문
Q1 — 주력 질문 "LangChain과 LangGraph의 관계를 설명해주세요. 왜 두 프레임워크가 함께 언급되나요?"
Q2 — 변형 "새 agent 기능을 구현할 때 create_agent를 쓸지, StateGraph로 직접 그래프를 짤지 어떻게 판단하시나요?"
Q3 — 변형 "레거시 AgentExecutor 기반 서비스를 유지보수하고 있다면, 마이그레이션이 필요한 이유와 방법을 어떻게 설명하시겠어요?"
전제 개념

Agent란 무엇이고, 왜 "그냥 LLM 호출"로는 부족한가

Agent = Model + Harness

LLM 하나만 호출하는 코드는 agent가 아니다. 사용자가 "서울 날씨 알려주고, 우산 필요하면 메일도 보내줘"라고 하면, 모델은 어떤 tool을 호출할지 스스로 결정하고, tool 실행 결과를 다시 읽어서 다음 행동을 또 결정해야 한다. 즉 "모델이 스스로 판단해서 여러 단계를 거쳐 목표를 달성하는 반복 구조(loop)"가 필요하다.

이 반복 구조를 감싸는 코드를 harness(하네스)라고 부른다. harness는 모델에게 어떤 tool을 줄지, 모델 응답을 어떻게 파싱할지, tool 실행 결과를 어떻게 다시 모델에게 넣어줄지, 언제 루프를 멈출지를 결정하는 "배관 공사(plumbing)" 역할을 한다. 그래서 업계에서는 "Agent = Model + Harness"라는 표현을 쓴다. 모델(두뇌)만으로는 아무것도 못 하고, harness(몸통)가 있어야 실제로 도구를 쓰고 행동할 수 있다.

핵심 전제: LangChain의 create_agent는 바로 이 harness를 만들어주는 함수다. "model + tools + prompt + middleware"를 조합해서 reasoning-act 루프를 자동으로 구성해준다.

LangGraph는 무엇이고 LangChain과 어떤 관계인가

단순한 반복 루프만으로는 실무 요구사항을 감당하기 어렵다. 실제 서비스에서는 "결제 승인은 사람이 확인한 뒤 진행", "네트워크 장애 시 재시도", "서버 재시작 후에도 이전 대화 상태 복구", "여러 agent가 병렬로 작업 후 결과를 합치기" 같은 제어 흐름(control flow)과 상태 관리(state management)가 필요하다.

LangGraph는 이런 요구사항을 위해 만들어진 그래프 기반 실행 엔진(runtime)이다. 작업 단계를 노드(node)로, 단계 간 흐름을 엣지(edge)로 표현하고, 전체 실행 상태를 명시적인 state(상태) 객체로 관리한다. 그래프이기 때문에 분기, 병렬 실행, 되돌아가기, 중간에 멈추고 사람 승인을 기다리기(interrupt), 실패 시 특정 단계부터 재시작하기(checkpoint) 같은 것들이 "그래프를 어떻게 그리느냐"의 문제로 자연스럽게 풀린다.

가장 흔한 오해: "LangGraph가 LangChain의 상위 개념이다" 또는 "LangChain 안에 LangGraph 기능이 포함되어 있다"고 막연히 이해하는 경우가 많다. 실제로는 반대에 가깝다. LangChain 1.0의 create_agent가 만들어내는 결과물은 내부적으로 컴파일된 LangGraph 그래프(StateGraph)다. 즉 LangGraph가 실행 엔진(하부 레이어)이고, LangChain의 agent API는 그 위에 얹힌 "자주 쓰는 패턴을 쉽게 만들어주는 상위 레이어(harness/DX 레이어)"다.

왜 이 구조로 재편되었는가 — AgentExecutor의 한계

과거 LangChain의 AgentExecutor는 체인(Chain) 추상화 위에 만들어진 초기 agent 구현이었다. 내부 동작이 블랙박스에 가까웠고, 중간 상태를 세밀하게 들여다보거나(step-level introspection), 특정 단계에서 멈춰서 사람이 승인하게 하거나, 실패한 지점부터 재개하는 것이 구조적으로 어려웠다. 커스터마이징하려면 클래스를 상속하고 내부 메서드를 오버라이드해야 하는 경우가 많아 유지보수 비용도 컸다.

LangGraph 기반 구조는 상태와 흐름을 명시적인 그래프로 드러내기 때문에, 위 문제들을 "그래프의 checkpointer/interrupt/conditional edge 기능"으로 자연스럽게 해결한다. 그 결과 LangChain 팀은 AgentExecutor를 legacy로 분류하고, langchain.agents.create_agent를 새 표준 진입점으로 제시했다. 이 함수는 사용법은 예전만큼 간단하지만, 내부는 LangGraph 위에서 동작한다.

create_agent 최소 예시

from langchain.agents import create_agent

def get_weather(city: str) -> str:
    """도시 이름으로 현재 날씨를 조회한다."""
    # 실제로는 외부 날씨 API 호출
    return f"{city}의 현재 날씨는 맑음, 22도입니다."

agent = create_agent(
    model="claude-sonnet-5",
    tools=[get_weather],
    system_prompt="너는 날씨 정보를 제공하는 어시스턴트다.",
)

result = agent.invoke({"messages": [{"role": "user", "content": "서울 날씨 알려줘"}]})

이 몇 줄 뒤에서 실제로는 StateGraph가 컴파일되어, "모델 호출 노드"와 "tool 실행 노드"가 조건부 엣지로 연결된 그래프가 만들어진다. create_agent는 이 반복적인 배선 작업을 대신 해주는 것뿐이다.

StateGraph로 직접 그래프를 그린다면

from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class AgentState(TypedDict):
    messages: list
    approved: bool

def call_model(state: AgentState) -> dict:
    # 모델 호출 로직
    return {"messages": [...]}

def call_tool(state: AgentState) -> dict:
    # tool 실행 로직
    return {"messages": [...]}

builder = StateGraph(AgentState)
builder.add_node("model", call_model)
builder.add_node("tool", call_tool)
builder.add_edge(START, "model")
builder.add_conditional_edges("model", lambda s: "tool" if ... else END)
builder.add_edge("tool", "model")

graph = builder.compile()

create_agent가 만들어주는 결과물과 원리는 같지만, 여기서는 노드 이름, 상태 스키마, 엣지 조건을 전부 직접 설계한다. 대신 표준 ReAct 루프를 벗어나는 흐름 — 예를 들어 "승인 노드를 하나 더 끼워 넣기", "두 tool을 병렬로 실행하고 결과를 합치기" — 를 자유롭게 만들 수 있다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

이 질문은 "LangChain 문법을 아느냐"가 아니라 "빠르게 변하는 프레임워크의 구조를 스스로 파악해서 아키텍처를 설계할 수 있느냐"를 본다. LangChain/LangGraph는 1~2년 주기로 표준 API가 바뀌었기 때문에, "예전에 배운 방식"만 알고 있으면 실무에서 바로 티가 난다.

  • 최신 상태를 스스로 추적하는가: AgentExecutor가 왜 legacy가 됐는지, 지금 표준이 무엇인지 스스로 파악하고 있는가 (아니면 몇 년 전 튜토리얼 수준에 머물러 있는가)
  • 레이어를 구분해서 설명하는가: "LangChain은 편의 계층, LangGraph는 실행 엔진"이라는 관계를 정확히 이해하고 있는가
  • 도구 선택에 판단 기준이 있는가: "무조건 create_agent" 또는 "무조건 StateGraph"가 아니라, 상황에 따라 다른 선택을 하고 그 이유를 설명할 수 있는가
  • 운영 관점을 갖고 있는가: 커스터마이징이 필요할 때 상속이 아니라 middleware/그래프 설계로 접근하는 최신 사고방식을 갖고 있는가
Senior 개발자는 "LangChain을 써봤다"가 아니라, "왜 지금 이 구조로 재편되었는지, 그리고 다음에 또 바뀌어도 원리를 기준으로 빠르게 따라갈 수 있는지"를 보여줘야 한다. 이 분야는 앞으로도 계속 바뀐다 — 면접관은 특정 API 암기가 아니라 변화에 적응하는 사고방식을 확인한다.
실무 연결

실제 상황과 연결하기

상황 1: 사내 FAQ 챗봇 (표준 ReAct 패턴)

사용자 질문 → 문서 검색 tool 호출 → 답변 생성이라는 단순한 반복 흐름이면, 커스텀 그래프를 짤 이유가 없다. → create_agent + retrieval tool이 가장 빠르고 유지보수 비용이 낮다.

상황 2: 결제 승인이 필요한 자동화 워크플로우

"환불 처리" 같은 민감한 작업은 tool 실행 전에 사람이 승인해야 한다. 표준 ReAct 루프에는 없는 "승인 대기 → 재개" 단계가 필요하다. → middleware의 human-in-the-loop 훅으로 우선 시도하고, 승인 로직이 복잡해지면 StateGraph로 승인 노드를 명시적으로 그린다.

상황 3: 여러 전문 agent가 협업하는 파이프라인

"리서치 agent가 자료를 모으고, 작성 agent가 보고서를 쓰고, 검수 agent가 확인한다"처럼 역할이 분리된 다단계 워크플로우는 각 단계의 상태를 격리하고 병렬 처리도 고려해야 한다. → subgraph 합성 또는 supervisor 패턴으로 StateGraph를 직접 설계한다.

상황 4: 레거시 AgentExecutor 서비스 유지보수

기존 서비스가 AgentExecutor로 되어 있고 장애가 잦다면, 무작정 새로 짜기보다 "지금 구조의 한계(상태 미노출, 재시작 시 복구 불가, 커스터마이징 어려움)를 먼저 진단하고, 트래픽이 몰리지 않는 기능부터 create_agent로 점진적으로 마이그레이션"하는 전략이 현실적이다.

면접에서 이렇게 답변 구조를 잡아라

Agent의 정의부터 명확히 하기 "Agent는 모델 혼자가 아니라 model과 harness의 조합입니다. harness가 tool 호출, 결과 반영, 루프 종료를 관리합니다."
2

3
선택 기준을 먼저 제시하기 "저는 표준 ReAct 루프로 충분한지, 아니면 승인 게이트·병렬 실행·다중 agent 같은 커스텀 제어 흐름이 필요한지를 먼저 봅니다."
4
현재 상황에 대입하기 "지금 질문하신 시나리오는 [표준 흐름 / 커스텀 흐름]에 해당하므로 [create_agent / StateGraph]를 선택하겠습니다."
5
마이그레이션/레거시 관점 언급 (Senior 포인트) "기존에 AgentExecutor로 만들어진 부분이 있다면, 상태 노출과 재시작 복구가 안 되는 게 구조적 한계이므로 트래픽 영향이 적은 기능부터 단계적으로 옮기겠습니다."
6
운영 관점으로 마무리하기 "어떤 방식을 택하든 결국 LangGraph 위에서 동작하므로, checkpointer로 상태를 영속화하고 LangSmith로 각 단계를 추적할 수 있다는 점은 동일하게 활용하겠습니다."
핵심 답변 예시

면접 현장 답변 예시 (전체)

"LangChain과 LangGraph는 계층이 다릅니다. LangGraph는 노드, 엣지, 상태로 실행 흐름을 명시적으로 표현하는 그래프 실행 엔진이고, LangChain 1.0의 create_agent는 그 위에서 '모델 호출 → tool 실행 → 다시 모델 호출'이라는 표준 ReAct 루프를 몇 줄로 만들어주는 상위 레이어입니다. 실제로 create_agent가 반환하는 것도 내부적으로 컴파일된 LangGraph 그래프입니다.

선택 기준은 '표준 흐름을 벗어나는 제어가 필요한가'입니다. 단순히 질문에 답하려고 tool 몇 개를 호출하는 정도면 create_agent와 middleware 조합으로 충분합니다. 하지만 결제 승인처럼 특정 단계에서 사람 승인을 기다려야 하거나, 여러 전문 agent가 병렬로 작업 후 결과를 합쳐야 하는 경우에는 StateGraph로 노드와 엣지를 직접 설계합니다.

예전에 AgentExecutor로 만들어진 서비스를 마이그레이션해야 한다면, 문제는 API 문법이 아니라 구조적 한계입니다. AgentExecutor는 중간 상태를 세밀하게 들여다보거나 실패 지점부터 재개하기가 어려운데, 이는 LangGraph의 checkpointer와 명시적 state 덕분에 자연스럽게 해결됩니다. 그래서 저는 트래픽이 적은 기능부터 create_agent 기반으로 옮기면서 checkpointer와 LangSmith 트레이싱을 함께 붙여 운영 가시성을 확보하는 방식으로 접근하겠습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "langgraph.prebuilt.create_react_agent와 langchain.agents.create_agent는 뭐가 다른가요?"

약한 답변: "둘 다 agent 만드는 함수라 비슷합니다."

강한 답변: "create_react_agent는 langgraph.prebuilt에 있던 '표준 ReAct agent를 빠르게 만드는 헬퍼'였습니다. langchain.agents.create_agent는 그 기능을 흡수하면서 middleware 배열을 통한 세밀한 커스터마이징(모델 호출 전후 가로채기, tool 호출 가로채기 등)까지 지원하도록 확장된 버전입니다. 기존 함수는 하위 호환을 위해 남아있지만, 신규 구현에서는 create_agent를 표준으로 씁니다. 다만 두 API 모두 결국 LangGraph 위에서 동작한다는 본질은 같습니다."

F2. "AgentExecutor로 만든 기존 서비스를 어떻게 마이그레이션하시겠어요?"

약한 답변: "새 API로 코드를 다시 작성합니다."

강한 답변: "세 단계로 접근합니다. 첫째, 현재 서비스에서 실제로 겪고 있는 구체적 문제(재시작 시 대화 유실, 특정 단계 디버깅 불가 등)를 먼저 정리해서 마이그레이션의 우선순위를 정합니다. 둘째, 트래픽이 적거나 실패 영향이 작은 기능부터 create_agent + checkpointer로 옮겨서 리스크를 통제합니다. 셋째, 마이그레이션한 기능에는 LangSmith 트레이싱을 붙여서 기존 대비 실제로 개선됐는지 (에러율, 응답 시간, 상태 복구 성공률) 측정합니다. '문법을 새 버전으로 바꾼다'가 아니라 '관측 가능하고 복구 가능한 구조로 바꾼다'는 목표를 분명히 합니다."

F3. "Middleware 아키텍처가 예전의 서브클래싱 방식보다 나은 이유는?"

강한 답변: "서브클래싱은 특정 클래스의 내부 구현에 강하게 결합됩니다. 프레임워크 내부가 바뀌면 오버라이드한 메서드가 깨지기 쉽고, 여러 커스터마이징을 조합하기도 어렵습니다. Middleware는 before_model, wrap_model_call, wrap_tool_call 같은 명확한 훅 지점에 함수를 꽂는 방식이라 여러 middleware를 리스트로 조합할 수 있고, 실행 순서도 '들어갈 때는 순서대로, 나올 때는 역순'으로 예측 가능합니다. 실제로 이전에는 supervisor, swarm, deep agent처럼 서로 다른 커스텀 아키텍처마다 별도 구현이 필요했는데, 지금은 이런 패턴들도 middleware 조합으로 재현할 수 있을 만큼 표현력이 넓어졌습니다."

F4. "StateGraph를 반드시 써야 하는 예시를 들어주세요."

강한 답변: "세 가지 대표 케이스가 있습니다. 첫째, 두 개 이상의 독립적인 작업을 병렬로 실행하고 결과를 합쳐야 하는 경우 — 표준 ReAct 루프는 순차 실행이 기본이라 병렬 분기(fan-out/fan-in)를 표현하려면 그래프를 직접 그려야 합니다. 둘째, 여러 전문 agent가 역할을 나눠 협업하는 supervisor/멀티 에이전트 구조 — 각 agent의 상태를 subgraph로 격리해야 합니다. 셋째, 도메인 특화된 승인/검증 단계가 여러 개 끼어드는 워크플로우 — 표준 tool-call 루프에 없는 노드를 명시적으로 추가해야 합니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • LangChain과 LangGraph를 레이어(상위/하부)로 구분해서 설명한다
  • create_agent vs StateGraph 선택에 명확한 기준이 있다
  • AgentExecutor가 왜 legacy가 됐는지 구조적 이유로 설명한다 (상태 미노출, 재개 불가 등)
  • 마이그레이션을 단계적 리스크 관리 관점에서 설명한다
  • 버전이 계속 바뀐다는 것을 인지하고, 원리 기준으로 사고한다

약한 답변의 특징

  • "LangGraph가 LangChain 안에 포함된 기능이다" — 레이어 관계를 반대로 이해
  • "둘 다 비슷해서 아무거나 쓰면 된다" — 선택 기준 없음
  • AgentExecutor deprecation을 "이름이 바뀐 것뿐"이라고 단순화
  • 마이그레이션을 "전체를 한 번에 새로 짠다"로 접근
  • 특정 버전의 함수명만 암기하고 원리를 설명하지 못함
핵심 차이: 강한 답변은 "이 API를 안다"가 아니라 "이 프레임워크가 왜 이렇게 설계되어 있고, 다음에 또 바뀌어도 원리를 기준으로 판단할 수 있는가"를 보여준다. 면접관은 특정 시점의 API 스냅샷을 외운 사람이 아니라, 변화하는 생태계에서 아키텍처를 판단할 수 있는 사람을 찾는다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "LangGraph는 LangChain의 상위 확장 기능이다"

사실: 반대다. LangGraph가 하부의 그래프 실행 엔진이고, LangChain의 agent API(create_agent)는 그 위에 얹힌 편의 계층이다. LangGraph는 LangChain 없이도 단독으로 쓸 수 있다.

오해 2: "Agent는 그냥 LLM이 tool을 호출하는 것이다"

사실: tool을 한 번 호출하는 것은 agent라기보다 단순 tool calling이다. Agent는 모델이 스스로 여러 단계를 반복하며 판단하는 loop 구조를 갖춰야 한다. 이 loop를 관리하는 코드(harness)가 agent의 핵심이다.

오해 3: "StateGraph를 쓰면 무조건 코드가 복잡해진다"

사실: create_agent 자체도 내부적으로 StateGraph를 컴파일해서 만든다. 즉 StateGraph는 "더 복잡한 것"이 아니라 "더 낮은 레벨에서 직접 제어하는 것"이다. 표준 흐름이면 오히려 create_agent가 더 짧고, 커스텀 제어 흐름이 필요할 때만 StateGraph의 복잡도가 정당화된다.

오해 4: "한 번 배운 LangChain 문법은 오래 쓸 수 있다"

사실: 1.0 GA 이후 2.0 전까지는 breaking change 없는 안정성을 공식적으로 약속했지만, 그 전까지는 AgentExecutor → LangGraph 기반 agent로의 전환처럼 표준 진입점 자체가 바뀐 적이 있다. 실무자는 특정 API 문법보다 "모델+harness", "그래프 기반 상태 관리" 같은 원리를 이해하고, 공식 changelog를 주기적으로 확인하는 습관을 가져야 한다.

실무 체크리스트

자가 점검 질문:
1. "Agent = Model + Harness"가 무슨 뜻인지 예시를 들어 설명할 수 있는가?
2. LangChain과 LangGraph 중 어느 쪽이 실행 엔진이고, 어느 쪽이 편의 계층인지 즉시 말할 수 있는가?
3. create_agent와 커스텀 StateGraph 중 무엇을 선택할지 3초 안에 판단 기준을 말할 수 있는가?
4. AgentExecutor가 legacy로 분류된 구조적 이유를 API 문법이 아니라 아키텍처 관점에서 설명할 수 있는가?
핵심 요약

오늘 반드시 기억할 것

  • Agent = Model + Harness. 모델 혼자서는 agent가 아니다. 반복 루프를 관리하는 harness가 필요하다.
  • LangGraph는 하부 실행 엔진(노드/엣지/상태 기반 그래프), LangChain의 create_agent는 그 위의 편의 계층이다. 관계를 반대로 이해하지 않는다.
  • create_agent: 표준 ReAct 루프에 middleware로 확장. 빠르고 유지보수 쉬움.
  • StateGraph: 병렬 분기, 다중 agent, 도메인 특화 승인 흐름처럼 표준 루프를 벗어날 때 직접 설계.
  • AgentExecutor는 legacy. 상태 미노출·재개 불가라는 구조적 한계 때문에 LangGraph 기반 구조로 대체됐다.
  • 버전은 계속 바뀐다. API 문법 암기보다 원리 이해와 공식 changelog 확인 습관이 실무·면접 모두에서 더 중요하다.
