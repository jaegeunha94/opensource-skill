LangGraph StateGraph 설계

Node/edge/reducer로 상태와 제어 흐름을 함께 설계하는 법, 그리고 병렬 실행(superstep) 모델을 모르면 왜 InvalidUpdateError에서 막히는지
버전/근거 기준: 이 레슨은 2026-07 기준 LangGraph 1.2.x(GA, main 브랜치 pyproject.toml 기준 1.2.8) API를 기준으로 작성됐다. docs.langchain.com/reference.langchain.com은 자동화 접근을 403으로 차단해서, StateGraph·채널·reducer의 정확한 동작은 raw.githubusercontent.com으로 직접 확인한 소스 원본(graph/state.py, graph/message.py, types.py, errors.py, channels/binop.py, channels/last_value.py)을 1차 근거로 쓰고, GitHub 이슈·블로그는 검색 결과로 교차 확인한 2차 근거임을 명시한다. 이슈의 제목/존재 여부는 검색으로 확인했지만, "회귀다/근본 원인이 X다" 같은 서술은 이슈 전문을 읽지 못했으므로 2차 근거로만 취급한다.
면접 질문
Q1 — 주력 질문 "StateGraph에서 state schema를 설계할 때 TypedDict, Pydantic, dataclass 중 무엇을 선택하나요? 그리고 여러 노드가 동시에 같은 state key에 쓰면 어떻게 되나요?"
Q2 — 변형 "conditional edge와 Command 객체 기반 라우팅은 뭐가 다른가요? 실무에서는 어떤 기준으로 선택하나요?"
Q3 — 변형 "Send API는 무엇이고, 왜 필요한가요? 그냥 conditional edge로 병렬 처리를 하면 안 되나요?"
전제 개념

StateGraph는 "합의된 상태를 두고 여러 함수가 턴을 주고받는" 실행 모델이다

Node = 상태를 읽고 부분 업데이트를 반환하는 함수, Edge = 다음 node를 정하는 연결

StateGraph는 Google의 분산 그래프 처리 프레임워크 Pregel에서 아이디어를 가져온 실행 모델이다. 전체 상태(state)는 하나의 공유 객체이고, node는 이 상태의 전체 또는 일부를 읽어서 "이 부분을 이렇게 바꿔라"라는 부분 업데이트(partial update)만 반환하는 함수다. edge는 한 node가 끝난 뒤 다음에 어떤 node(들)를 실행할지 정하는 연결이다. 이게 왜 중요하냐면, 이 모델 덕분에 "각 node는 자기 역할만 알면 되고, 전체 흐름은 그래프 구조가 결정한다"는 관심사 분리가 생기기 때문이다 — 이건 Day 1에서 배운 "왜 create_agent(고정 ReAct 루프)로는 부족해서 StateGraph로 내려가야 하는가"의 구체적인 답이기도 하다. 

from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class State(TypedDict):
    question: str
    draft: str
    approved: bool

def write_draft(state: State) -> dict:
    # 전체 state가 아니라 "바뀐 부분"만 반환한다
    return {"draft": generate(state["question"])}

def review(state: State) -> dict:
    return {"approved": is_good(state["draft"])}

graph = StateGraph(State)
graph.add_node("write_draft", write_draft)
graph.add_node("review", review)
graph.add_edge(START, "write_draft")
graph.add_edge("write_draft", "review")
graph.add_edge("review", END)
app = graph.compile()

State schema 세 가지: TypedDict / dataclass / Pydantic BaseModel

StateGraph(state_schema)의 state_schema는 세 가지 타입을 모두 허용한다. 내부적으로 is_typeddict/is_dataclass/issubclass(BaseModel)로 분기해서 각기 다른 방식으로 채널을 만든다. 셋의 실무 차이는 다음과 같다.

선택지런타임 검증노드 반환값실무 권고
TypedDict 없음(타입 힌트일 뿐) 부분 dict 그대로 반환 기본 선택. 검증 비용이 없고, 부분 업데이트를 그대로 dict로 반환하는 패턴과 자연스럽게 맞는다.
dataclass 없음(필드 타입 명세만) 여전히 부분 dict 반환(dataclass 인스턴스가 아님) 타입 힌트가 더 엄격하길 원할 때. 단, dataclass를 쓴다고 노드가 dataclass 인스턴스를 반환해야 하는 건 아니다 — 스키마 "모양"만 정의할 뿐, 반환 타입 계약은 여전히 부분 dict다.
Pydantic BaseModel 그래프 입력에서만 확실히 검증됨 부분 dict 반환(Pydantic이 자동 병합) 외부 입력을 신뢰할 수 없어 런타임 검증이 꼭 필요할 때만. 별도 비용(속도 저하)과 함정(다음 항목)이 있다.
Pydantic을 쓸 때 실제로 보고된 함정: 필드에 alias를 걸어도 state 입력 검증에서 populate_by_name 설정이 무시되는 사례(langgraph#2555, #6401), generic 타입을 쓴 모델에서 검증이 꼬이는 사례(#4060), 검증 실패 시 어느 node에서 실패했는지 알기 어려운 추적하기 힘든 traceback(#1978, #1977)이 보고됐다. 즉 "Pydantic을 쓰면 항상 더 안전하다"는 생각과 달리, Pydantic의 런타임 검증이 확실히 보장되는 지점은 그래프 "입력"뿐이고, 중간 node의 매 반환값마다 재검증된다고 기대하면 안 된다 — 이 경계를 정확히 아는 게 senior 신호다.

state_schema / input_schema / output_schema / context_schema — 4개의 서로 다른 스키마

StateGraph(state_schema, context_schema=None, *, input_schema=None, output_schema=None). 이 넷을 혼동하지 않는 게 중요하다.

  • state_schema: 그래프 내부에서 흐르는 전체 상태. node 간 공유되는 "진짜 상태"다.
  • input_schema/output_schema: 기본값은 state_schema와 같지만, 명시적으로 좁힐 수 있다. 예를 들어 내부적으로는 draft, approved, retry_count 같은 "비공개 스크래치패드" 키를 잔뜩 들고 다니면서도, 외부에는 question만 받고(input_schema) draft만 돌려주는(output_schema) 좁은 인터페이스를 만들 수 있다. 이건 API 설계에서 "내부 도메인 모델과 외부 DTO를 분리"하는 것과 동일한 원칙이다.
  • context_schema(옛 이름 config_schema, v0.6.0부터 deprecated, 2.0.0에서 제거 예정): state가 아니다. 실행 중 바뀌지 않는 불변 설정(예: user_id, db_connection, 권한 레벨)을 위한 별도 채널이다. 노드 시그니처에 runtime: Runtime[Context]를 추가해서 runtime.context로 접근한다 — 예전에 config["configurable"]에서 값을 꺼내던 패턴을 대체한다.
from dataclasses import dataclass
from langgraph.runtime import Runtime

@dataclass
class AppContext:
    user_id: str
    tenant_tier: str

def write_draft(state: State, runtime: Runtime[AppContext]) -> dict:
    if runtime.context.tenant_tier == "enterprise":
        ...
    return {"draft": generate(state["question"])}

graph = StateGraph(State, context_schema=AppContext)

Reducer — "이 키에 여러 개의 부분 업데이트가 동시에 들어오면 어떻게 합칠 것인가"

기본적으로 state의 각 키는 LastValue 채널이다 — 이름 그대로 마지막에 들어온 값으로 그냥 덮어쓴다. 문제는 "동시에 들어오면"이라는 조건이다. LangGraph는 병렬로 실행된 여러 node의 쓰기를 같은 superstep(아래 참고)에서 동시에 받을 수 있는데, LastValue는 한 스텝에 정확히 하나의 값만 받을 수 있게 설계돼 있다. 한 스텝에 2개 이상의 값이 들어오면 즉시 예외가 난다. 

# langgraph/channels/last_value.py 원본 동작 그대로
# "At key '{key}': Can receive only one value per step.
#  Use an Annotated key to handle multiple values."
raise InvalidUpdateError(...)  # ErrorCode.INVALID_CONCURRENT_GRAPH_UPDATE

이걸 막으려면 Annotated[Type, reducer_fn]으로 그 키의 채널을 LastValue 대신 BinaryOperatorAggregate로 바꿔야 한다. 이 채널은 한 스텝에 들어온 여러 값을 쓰기 순서대로 왼쪽에서 오른쪽으로 reducer(current, value)를 계속 접어(fold) 나가며 하나로 합친다. 

import operator
from typing import Annotated
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]      # id 기준 병합: 같은 id면 덮어쓰기, 다르면 append
    votes: Annotated[list[int], operator.add]    # 리스트 이어붙이기
    total: Annotated[int, operator.add]          # 정수 합산

add_messages(langgraph.graph.message)는 메시지 리스트 전용 reducer로, 같은 id를 가진 메시지가 다시 들어오면 append가 아니라 in-place 교체를 한다(예: 스트리밍 중인 메시지를 여러 번 업데이트할 때 중복 없이 최신 상태로 유지하는 용도). format="langchain-openai" 인자로 출력 메시지 형태를 정규화할 수도 있다.

1.x에서 추가된 Overwrite: reducer가 걸린 키라도, 특정 시점에는 "합치지 말고 그냥 통째로 교체"하고 싶을 때가 있다(예: 요약 후 메시지 히스토리를 통째로 갈아끼우는 경우). 이럴 때 Overwrite(value)(또는 JSON 직렬화 환경에서는 {"__overwrite__": value} 형태)로 감싸서 반환하면 reducer를 건너뛰고 강제로 교체한다. 단, 같은 스텝에 Overwrite가 2개 이상 들어오면 "Can receive only one Overwrite value per super-step"이라는 별도 InvalidUpdateError가 난다 — reducer가 있어도 "덮어쓰기 자체의 동시성 충돌"까지 막아주지는 않는다는 뜻이다.
Reducer가 associative/commutative하지 않으면 생기는 문제: 같은 스텝에서 여러 Send 분기(아래 참고)가 병렬로 완료되는 순서는 보장되지 않는다. reducer가 operator.add처럼 교환법칙이 성립하면 순서가 바뀌어도 결과가 같지만, 순서에 의존하는 커스텀 reducer(예: "마지막 세 개만 남기고 나머지는 버려라" 같은 로직을 순서에 의존해서 구현한 경우)는 실행할 때마다 결과가 달라지는 비결정성이 생긴다. Senior 인터뷰에서 "reducer를 왜 순수 함수이자 결합·교환 법칙을 만족하게 짜야 하는가"를 설명할 수 있어야 하는 이유다.

Conditional Edge vs Command — "라우팅만 하는 edge" vs "업데이트+라우팅을 한 번에 하는 node"

add_conditional_edges(source, path_fn, path_map=None)는 라우팅만 한다 — state를 바꿀 수 없고, 다음 node 이름(또는 이름들)만 결정한다. 반면 1.x에서 자리잡은 패턴은 node 함수 안에서 Command(update=..., goto=...)를 반환해서 "이번 턴에 state를 이렇게 바꾸고, 다음엔 이 node로 가라"를 한 번에 표현하는 것이다. 

from langgraph.types import Command
from typing import Literal

def review(state: State) -> Command[Literal["revise", "publish"]]:
    ok = is_good(state["draft"])
    return Command(
        update={"approved": ok},
        goto="publish" if ok else "revise",
    )

# 별도의 add_conditional_edges 선언이 필요 없다 — "edgeless" 그래프.
# add_node(..., destinations=(...)) 는 실행에 영향 없이 그래프 시각화용 힌트만 준다.

즉 정적으로 고정된 소수의 갈래를 미리 선언하고 싶으면 add_conditional_edges가 더 명시적이고 그래프 구조를 한눈에 보여주기 좋고, 업데이트와 분기 결정이 같은 계산(예: LLM 판단 결과를 바로 다음 행선지로 씀)에서 나온다면 Command로 node 하나에 합치는 게 코드 중복을 줄인다. 실무에서는 이 둘을 섞어서 쓰는 게 일반적이다 — "이건 명시적으로 그래프에 보이면 좋겠다"는 부분만 conditional edge로 남기고, 나머지는 Command로 축약한다.

실제로 보고된 문제: subgraph가 3단계 이상 중첩된 구조에서 Command(goto=...)가 ParentCommand 에러를 내는 사례(langgraph#6409), add_node(..., defer=True) (모든 상위 브랜치가 끝날 때까지 실행을 미루는 옵션)와 Command + conditional edge를 함께 쓰면 그래프가 깨지는 사례(#5182)가 보고됐다. → 깊은 subgraph 중첩이나 defer=True와 함께 Command를 쓸 때는 별도로 검증해야 한다(Day 10에서 subgraph를 깊게 다룰 때 다시 짚는다).

Send — 정적 분기가 아니라 "런타임에 몇 개가 생길지 모르는" fan-out

add_conditional_edges와 Command.goto는 모두 미리 정해진 node 이름들 중에서 고르는 구조다. 하지만 "리스트 안의 항목 수만큼 병렬로 처리하고 싶다"(예: 검색된 문서 N개를 각각 요약)처럼 런타임에야 몇 개의 병렬 작업이 생길지 알 수 있는 경우는 다른 문제다. 이때 쓰는 게 Send(node, arg)다. 

from langgraph.types import Send

def fan_out_to_summarizers(state: State) -> list[Send]:
    return [
        Send("summarize_doc", {"doc": d})
        for d in state["documents"]   # N은 런타임에만 정해짐
    ]

graph.add_conditional_edges("retrieve", fan_out_to_summarizers)

Send는 node 이름과, 그 인스턴스에만 줄 독립된 arg를 갖는다 — 그래프의 공통 state 모양과 다른 페이로드를 줄 수 있다는 뜻이다. 이렇게 생성된 N개의 병렬 실행은 모두 같은 superstep에 속하고, 각자의 반환값은 반드시 Annotated reducer가 걸린 키로 다시 합쳐져야 한다(안 그러면 위에서 본 InvalidUpdateError가 그대로 재현된다). 이게 map-reduce의 map 단계이고, reducer가 reduce 단계다.

Superstep — 병렬 실행과 reducer가 왜 세트로 다녀야 하는가

LangGraph 실행은 "슈퍼스텝의 연속"이다. 한 슈퍼스텝에서 실행 대상으로 잡힌 모든 node(같은 source에서 나온 복수의 정적 edge, 또는 Send로 스폰된 임의 개수의 task)는 동시에 실행되고, 각자의 반환값(부분 업데이트)은 그 스텝이 끝날 때까지 버퍼에 쌓였다가 한 번에 채널에 적용된다. 이 설계 덕분에 "node A가 쓴 값을 node B가 같은 스텝 안에서 읽는" 경쟁 조건(race condition) 자체가 구조적으로 없다 — 어차피 같은 스텝 안에서는 서로의 결과를 볼 수 없기 때문이다. 대신 "같은 스텝, 같은 키에 2개 이상의 쓰기가 몰리는" 상황은 항상 생길 수 있고, 이걸 프로그래머가 락(lock)으로 직접 막는 게 아니라 reducer라는 선언적 병합 규칙으로 처리하게 하는 게 LangGraph의 동시성 모델이다.

왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

"StateGraph를 써봤다"는 튜토리얼 경험이 아니라, 상태를 동시성 문제로 인식하고 설계할 수 있는가와 제어 흐름을 정적 구조와 동적 로직 중 어디에 둘지 판단할 수 있는가를 함께 본다.

  • 동시성에 대한 감각: "여러 node가 같은 키에 쓰면 무슨 일이 일어나는가"라는 질문에 "reducer가 없으면 예외가 난다"까지 구체적으로 답할 수 있는가 — state 설계를 데이터 모델링이 아니라 동시성 설계로 볼 수 있는가
  • 스키마 분리 감각: state_schema/input_schema/output_schema/ context_schema를 구분해서 "내부 상태와 외부 인터페이스를 분리하는 API 설계 원칙"을 그래프에도 적용할 수 있는가
  • 제어 흐름 설계 판단력: conditional edge와 Command, 정적 분기와 Send의 차이를 "언제 어떤 게 더 적절한가"로 설명할 수 있는가 — 단순히 "둘 다 쓸 수 있다"가 아니라
  • 실전 디버깅 경험: InvalidUpdateError 같은 실제 런타임 에러를 마주쳤을 때 원인(reducer 누락, 동시 쓰기)을 바로 짚어낼 수 있는가
Senior 개발자는 "state에 뭘 넣을지"를 데이터 필드 나열로 답하지 않고, "이 필드에 누가 쓰고, 동시에 쓰는 경우가 있는지, 있다면 어떻게 합칠지"를 먼저 묻는다. 이게 mid-level과 senior를 가르는 핵심 차이다.
실무 연결

실제 상황과 연결하기

상황 1: 병렬 subgraph 3개가 공유 카운터 키에 동시에 쓰다가 배포 직후 장애

리서치 에이전트가 세 개의 subgraph(웹 검색/사내 문서 검색/코드 검색)를 병렬로 돌리고, 각자 끝나면 "완료된 subgraph 수"를 세는 completed_count 키에 +1을 쓰도록 만들었다. 로컬 테스트에서는 순차 실행이라 문제가 없었는데, 프로덕션에서 세 subgraph가 진짜 병렬로 실행되자 InvalidUpdateError가 바로 터졌다(langgraph#7099 계열 사례와 동일 패턴). → Annotated[int, operator.add]로 바꾸고 나서야 안정화됐다. 로컬 개발 환경의 "우연히 순차적으로 실행되는" 상황이 병렬 동시성 버그를 가려버린 전형적인 사례다.

상황 2: Pydantic state에서 API alias가 무시돼 검증이 조용히 실패

외부 REST API의 필드명(camelCase)과 내부 state 필드명(snake_case)을 맞추려고 Pydantic Field(alias=...)와 populate_by_name=True를 걸었는데, 그래프 입력 검증에서 이 설정이 무시돼(langgraph#2555 계열 이슈) 필드가 None으로 채워진 채 그래프가 돌아갔다. 에러 없이 조용히 잘못된 값으로 진행됐기 때문에 원인 파악에 시간이 걸렸다. → alias 대신 그래프 진입 전 별도 어댑터 함수로 명시적 변환을 두는 방식으로 우회했다.

상황 3: 3단계로 중첩된 subgraph에서 Command 라우팅이 깨짐

supervisor → team subgraph → worker subgraph로 3단계 중첩 구조를 짜고, 가장 안쪽 worker에서 Command(graph=Command.PARENT, goto=...)로 팀 레벨까지 올라가려 했는데 ParentCommand 에러가 났다(langgraph#6409 계열). → 중첩을 2단계로 줄이고, 그 이상의 위임이 필요한 경우는 메시지 기반 handoff(Day 10에서 다루는 supervisor/swarm 패턴)로 우회했다. "Command로 부모까지 무한정 올라갈 수 있다"고 가정하지 말고, 중첩 깊이 제한을 먼저 확인해야 한다는 교훈.

상황 4: recursion_limit을 명시적으로 올렸는데도 기본값 25에서 계속 끊김

긴 멀티스텝 워크플로우에 recursion_limit=100을 config로 넘겼는데도 일부 실행이 25스텝 근처에서 GraphRecursionError로 끊기는 현상이 보고됐다(langgraph#7313 계열 — config 병합 과정에서 값이 조용히 리셋되는 사례). → 실행 직전에 실제 적용된 recursion_limit 값을 로그로 남겨 확인하는 방어 코드를 추가했다. "파라미터를 넘겼으니 적용됐을 것"이라고 가정하지 않고, 실제 적용값을 관측 가능하게 만드는 습관이 이런 조용한 설정 유실을 잡아낸다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"State schema는 기본적으로 TypedDict를 씁니다. 검증 비용이 없고, node가 전체 상태가 아니라 바뀐 부분만 dict로 반환하는 LangGraph의 패턴과 자연스럽게 맞기 때문입니다. Pydantic BaseModel은 외부 입력을 신뢰할 수 없어 런타임 검증이 꼭 필요할 때만 쓰는데, 검증이 확실히 보장되는 지점은 그래프 '입력'이지 매 node의 반환값마다 재검증되는 게 아니라는 걸 전제로 둡니다. 실제로 alias 설정이 입력 검증에서 무시되거나 generic 타입에서 검증이 꼬이는 이슈가 보고된 적이 있어서, Pydantic을 쓸 때는 그 경계를 명확히 인지하고 갑니다.

여러 node가 동시에 같은 state key에 쓰는 문제는 LangGraph의 실행 모델과 바로 연결됩니다. StateGraph는 Pregel 기반 superstep 모델이라, 같은 스텝에서 병렬로 실행된 node들의 부분 업데이트는 스텝이 끝날 때 한 번에 채널에 반영됩니다. 기본 채널(LastValue)은 한 스텝에 정확히 하나의 값만 받을 수 있게 설계돼 있어서, reducer 없이 두 node가 같은 키에 동시에 쓰면 InvalidUpdateError가 납니다. 이걸 막으려면 Annotated[Type, reducer_fn]으로 그 키를 BinaryOperatorAggregate 채널로 바꿔서 여러 값을 순서대로 fold해 합치게 해야 하고, 이 reducer는 결합·교환 법칙을 만족해야 병렬 분기의 완료 순서가 바뀌어도 결과가 결정적입니다.

제어 흐름은 두 축으로 나눠 생각합니다. 정적으로 고정된 소수의 분기는 add_conditional_edges로 그래프 구조에 명시적으로 드러내고, 업데이트와 분기 결정이 같은 계산에서 나오는 경우는 node 안에서 Command(update=, goto=)를 반환해 하나로 합칩니다. 런타임에야 몇 개의 병렬 작업이 생길지 정해지는 map-reduce 패턴, 예를 들어 검색된 문서 개수만큼 요약 작업을 병렬로 스폰하는 경우는 Send를 씁니다. Send로 생성된 작업들은 모두 같은 superstep에 속하고, 결과는 반드시 reducer가 걸린 키로 다시 합쳐져야 하는데, 이걸 놓치면 정확히 같은 InvalidUpdateError가 재현됩니다.

실무에서는 reducer 누락이 로컬 순차 실행 환경에서는 드러나지 않다가 프로덕션 병렬 환경에서 처음 터지는 경우가 많았기 때문에, state 설계 단계에서 '이 키에 누가 쓰는가, 병렬로 쓰는 경로가 있는가'를 먼저 체크리스트로 확인하는 걸 원칙으로 삼습니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "Pydantic으로 state schema를 만들면 항상 더 안전하지 않나요?"

약한 답변: "네, Pydantic이 검증도 해주니까 더 안전합니다."

강한 답변: "검증이 확실히 보장되는 지점은 그래프 '입력'뿐입니다. 중간 node의 매 반환값마다 재검증된다고 기대하면 안 되고, 실제로 alias 설정이 무시되거나 generic 타입에서 검증이 꼬이는 이슈, 검증 실패 시 어느 node가 원인인지 알기 어려운 traceback 문제가 보고된 적이 있습니다. 그래서 저는 외부 입력을 신뢰할 수 없어 런타임 검증이 꼭 필요한 경계에만 선택적으로 Pydantic을 쓰고, 나머지는 검증 비용이 없는 TypedDict를 기본값으로 씁니다."

F2. "reducer 없이 병렬 node를 실행하면 항상 에러가 나나요?"

강한 답변: "같은 슈퍼스텝 안에서 그 키에 실제로 2개 이상의 값이 동시에 쓰일 때만 에러가 납니다. 병렬 분기가 있어도 각 분기가 서로 다른 키에 쓴다면 문제가 없고, 조건부로 한쪽 분기만 그 키를 건드린다면 우연히 안 터질 수도 있습니다. 그래서 이게 더 위험합니다 — 테스트 시나리오에서는 조건이 안 겹쳐서 통과하다가, 프로덕션의 다른 입력 분포에서 처음 겹쳐서 터지는 식으로 나타나기 쉽습니다."

F3. "Command와 conditional edge를 같이 쓸 수 있나요?"

강한 답변: "네, 섞어 쓰는 게 일반적입니다. 다만 조합에 따라 알려진 문제가 있습니다 — defer=True로 설정된 node가 Command와 conditional edge를 함께 쓰는 조합에서 그래프가 깨진다는 보고가 있어서, 이런 조합을 쓸 때는 실제 컴파일된 그래프 구조를 시각화해서 의도한 대로 나오는지 반드시 확인합니다."

F4. "Send는 왜 필요한가요? 그냥 conditional edge로 리스트를 순회하면서 라우팅하면 안 되나요?"

약한 답변: "Send나 conditional edge나 결국 비슷한 거 아닌가요."

강한 답변: "conditional edge와 Command.goto는 그래프에 미리 선언된 고정된 node 이름들 중에서 고르는 구조입니다. 리스트 항목 수만큼 병렬 작업을 만들고 싶은데 그 개수가 런타임 입력에 따라 달라진다면, 정적 분기로는 표현할 수 없습니다. Send(node, arg)는 이런 동적 개수의 병렬 task를 스폰하면서, 각 task에 독립된 payload를 줄 수 있는 map 단계 전용 프리미티브입니다. 대신 그 결과는 항상 reducer가 걸린 키로 reduce해서 다시 모아야 한다는 책임이 따라옵니다."

F5. "context_schema와 state_schema를 굳이 분리해둔 이유가 뭔가요?"

강한 답변: "성격이 다른 데이터이기 때문입니다. state는 실행 중 계속 바뀌고 node 간에 공유되는 값이고, context는 실행 내내 변하지 않는 설정값(사용자 ID, 권한 레벨, DB 커넥션 같은 것)입니다. 이걸 하나의 state에 섞어 넣으면 reducer 설계, 체크포인트 직렬화, 스키마 검증 모두에서 '이 값은 왜 매번 바뀌지 않는데 state에 있냐'는 혼란이 생깁니다. context_schema로 분리하고 Runtime[Context]로 주입받으면, 이 값은 애초에 채널 동시성 문제의 대상이 아니라는 게 타입 레벨에서 명확해집니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • State 설계를 동시성 설계 문제로 인식하고, 병렬 쓰기 시나리오를 먼저 떠올린다
  • Reducer가 결합·교환 법칙을 만족해야 하는 이유를 실행 순서 비결정성과 연결해서 설명한다
  • Pydantic의 검증이 "그래프 입력"에만 확실히 보장된다는 경계를 정확히 안다
  • Conditional edge/Command/Send를 "정적 vs 동적, 라우팅만 vs 업데이트+라우팅" 기준으로 구분해서 선택한다
  • 실제 실패 사례(InvalidUpdateError, alias 무시, ParentCommand)를 근거로 든다

약한 답변의 특징

  • State schema를 "그냥 데이터 필드 나열"로만 설명한다
  • Reducer를 "여러 값을 합치는 기능" 정도로만 뭉뚱그려 설명하고, 왜 순서 안전성이 중요한지는 모른다
  • Pydantic이 모든 지점에서 항상 검증해준다고 착각한다
  • Command와 conditional edge가 완전히 같은 것이라고 답한다
  • 병렬 실행에서 왜 문제가 생기는지 superstep 개념 없이 설명하지 못한다
핵심 차이: 강한 답변은 StateGraph의 API 목록을 나열하지 않고, "이 상태가 언제, 누구에게, 동시에 쓰일 수 있는가"라는 하나의 동시성 질문을 축으로 전체 설계(schema/reducer/routing)를 일관되게 설명한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "State schema는 TypedDict든 Pydantic이든 어차피 결과는 같으니 아무거나 골라도 된다"

사실: 런타임 검증 여부, 검증이 보장되는 범위(그래프 입력 vs 매 node 반환값), alias·generic 관련 알려진 함정이 다르다. 검증 필요성이 없다면 검증 비용이 없는 TypedDict가 기본 선택이다.

오해 2: "여러 node가 같은 키에 쓰면 LangGraph가 알아서 병합해준다"

사실: 기본 채널(LastValue)은 한 스텝에 값 하나만 허용한다. 병합이 필요하면 Annotated[Type, reducer_fn]을 명시적으로 걸어야 하고, 안 걸면 InvalidUpdateError다.

오해 3: "conditional edge와 Command는 결국 같은 걸 하는 문법 차이일 뿐"

사실: add_conditional_edges는 라우팅만 하고 state를 바꿀 수 없다. Command는 node 안에서 업데이트와 라우팅을 동시에 반환한다. 그래프 구조를 명시적으로 드러낼지, 계산과 분기를 한 node에 합칠지의 설계 판단이 다르다.

오해 4: "set_entry_point/set_finish_point는 1.x에서 제거된 legacy API다"

사실: deprecated되지 않았다. StateGraph 소스 원본 기준으로 여전히 add_edge(START, key)/add_edge(key, END)/add_conditional_edges(START, ...)를 감싸는 얇은 wrapper로 존재한다. deprecated된 건 이것들이 아니라 MessageGraph(1.0.0에서 deprecated, 2.0.0 제거 예정, StateGraph + messages 키로 대체), NodeInterrupt (interrupt() 함수로 대체), config_schema(→ context_schema), input/output 생성자 인자(→ input_schema/output_schema)다. 오래된 블로그 글의 "제거됐다"는 서술을 그대로 믿지 않고 실제 소스로 확인하는 습관이 여기서 드러난다.

오해 5: "Send로 스폰한 병렬 작업들은 실행 순서가 보장된다"

사실: 같은 superstep에 속한 Send 작업들의 완료 순서는 보장되지 않는다. 그래서 그 결과를 합치는 reducer가 순서에 의존하면 실행할 때마다 결과가 달라지는 비결정성이 생긴다.

핵심 요약

오늘 반드시 기억할 것

  • StateGraph는 Pregel 기반 superstep 모델이다 — 같은 스텝에서 병렬 실행된 node들의 쓰기는 스텝이 끝날 때 한 번에 채널에 적용되고, 같은 키에 reducer 없이 2개 이상의 쓰기가 몰리면 InvalidUpdateError가 난다.
  • State schema는 TypedDict/dataclass/Pydantic BaseModel 모두 허용된다. 기본은 검증 비용이 없는 TypedDict이고, Pydantic은 그래프 "입력"에서만 검증이 확실히 보장된다는 경계를 안다.
  • Annotated[Type, reducer_fn]으로 LastValue 채널을 BinaryOperatorAggregate로 바꿔 동시 쓰기를 병합한다. reducer는 결합·교환 법칙을 만족해야 실행 순서와 무관하게 결과가 결정적이다. Overwrite로 reducer를 우회한 강제 교체도 가능하지만 스텝당 하나만 허용된다.
  • Conditional edge는 라우팅만, Command(update=, goto=)는 업데이트+라우팅을 한 node에서 함께 표현한다. Send(node, arg)는 런타임에만 개수가 정해지는 동적 병렬 fan-out (map-reduce의 map)이고, 결과는 반드시 reducer로 다시 모아야 한다.
  • state_schema/input_schema/output_schema/context_schema는 성격이 다른 4개의 스키마다 — 내부 상태, 공개 입출력 인터페이스, 실행 중 불변인 설정을 명확히 분리한다.
  • set_entry_point/set_finish_point는 deprecated가 아니다. 실제로 deprecated된 건 MessageGraph, NodeInterrupt, config_schema, 생성자의 input/output 인자다.
