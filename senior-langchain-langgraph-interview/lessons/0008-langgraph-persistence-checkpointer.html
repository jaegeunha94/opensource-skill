Persistence와 Checkpointer

"체크포인터를 왜 쓰는가"가 아니라 "어떤 백엔드를, 왜, 어떤 실패를 감수하고 쓰는가" — thread_id/checkpoint_ns/checkpoint_id 구성과 time travel, 그리고 풀 스냅샷 저장 모델이 만드는 운영 비용
버전/근거 기준: 이 레슨은 2026-07 기준 LangGraph 1.2.x(main 브랜치 libs/langgraph/pyproject.toml 확인 결과 1.2.8)과 체크포인트 관련 3개 서브패키지 — langgraph-checkpoint 4.1.1, langgraph-checkpoint-sqlite 3.1.0, langgraph-checkpoint-postgres 3.1.0(모두 main 브랜치 pyproject.toml 원본 확인) API를 기준으로 작성됐다. docs.langchain.com/oss/python/langgraph/durable-execution은 이번에도 자동화 접근을 403으로 차단해서, raw.githubusercontent.com으로 직접 읽은 소스 원본(checkpoint/base/__init__.py, checkpoint/memory/__init__.py, checkpoint-sqlite/__init__.py, checkpoint-sqlite/aio.py, checkpoint-postgres/__init__.py, checkpoint-postgres/shallow.py, langgraph/types.py, pregel/main.py)를 1차 근거로 썼다. pregel/main.py는 파일이 커서 stream 메서드의 durability 파라미터 docstring 전문을 한 번에 확인하지 못했고(요청이 중간에 잘림, 재요청 시 429 발생), types.py의 Durability = Literal["sync", "async", "exit"] 타입 정의와 ShallowPostgresSaver deprecation 안내(아래 참고)로 교차 확인했다. GitHub 이슈(#7714, #4987, #7259, #6125)는 이슈 본문을 직접 읽어 신고된 증상과 재현 수치는 확인했지만, "근본 원인이 X다"라는 서술 중 메인테이너 확인이 없는 부분은 제보자 본인의 진단임을 명시하고 2차 근거로만 취급한다. 커뮤니티 블로그(azguards.com)는 개념 설명 참고용이며 신뢰도가 낮음에 유의한다.

전제 개념

Checkpointer = "매 superstep이 끝날 때의 상태를 통째로 저장/복원하는 계층"

Day 7의 superstep 모델 위에 올라가는 것

Day 7에서 StateGraph가 Pregel 기반 superstep의 연속이고, 한 스텝이 끝날 때 부분 업데이트들이 채널에 한 번에 반영된다는 걸 봤다. Checkpointer는 바로 이 "한 스텝이 끝난 시점"마다 전체 상태의 스냅샷을 저장소에 기록하는 컴포넌트다. 이게 있어야 (1) 프로세스가 죽었다가 다시 뜬 뒤에도 어디까지 진행했는지 복원할 수 있고, (2) 같은 thread_id로 다시 들어온 사용자에게 이전 대화 맥락을 이어줄 수 있고(Day 5의 short-term memory가 바로 이 메커니즘 위에서 구현된다), (3) 실행 중간에 interrupt()로 멈췄다가 나중에 재개할 수 있다(Day 9에서 본격적으로 다룬다). 즉 checkpointer는 "메모리 기능"이 아니라 그래프 실행 자체의 durability(내구성) 계층이다.

BaseCheckpointSaver — 모든 백엔드가 구현해야 하는 인터페이스

langgraph-checkpoint 패키지(4.1.1)의 checkpoint/base/__init__.py 원본 기준으로, BaseCheckpointSaver는 다음 메서드들을 동기/비동기 쌍(get/aget 등)으로 정의한다. 

class BaseCheckpointSaver:
    # 조회
    def get_tuple(self, config) -> CheckpointTuple | None: ...
    def list(self, config, *, filter=None, before=None, limit=None): ...

    # 쓰기
    def put(self, config, checkpoint, metadata, new_versions) -> dict: ...
    def put_writes(self, config, writes, task_id, task_path="") -> None: ...

    # 정리(4.x에서 자리잡은 API — 아래 "체크포인트 폭증" 참고)
    def delete_thread(self, thread_id) -> None: ...
    def delete_for_runs(self, run_ids) -> None: ...
    def copy_thread(self, source_thread_id, dest_thread_id) -> None: ...
    def prune(self, ...) -> None: ...

put이 실제로 저장하는 Checkpoint는 다음 필드를 갖는 구조체다 — v(포맷 버전), id(단조 증가하는 고유 체크포인트 id), ts(ISO 8601 타임스탬프), channel_values (채널 이름 → 역직렬화된 값의 매핑), channel_versions(채널별 버전), versions_seen(어느 노드가 어느 버전까지 봤는지), updated_channels. 핵심은 channel_values가 그 시점 상태 "전체"의 매핑이라는 점이다 — 이전 체크포인트 대비 "바뀐 부분만"의 diff가 아니라, 매 스텝마다 전체 채널 값을 다시 담아 저장한다(직렬화 비용에 대한 실무적 함의는 아래 "체크포인트 폭증"에서 이어간다).

thread_id / checkpoint_ns / checkpoint_id — 체크포인트 키를 구성하는 3요소

  • thread_id: "어떤 대화/세션인가"를 식별한다. Day 5에서 다룬 thread-scoped short-term memory의 스코프가 바로 이 값이다. 같은 thread_id로 다시 invoke하면 마지막 체크포인트부터 이어서 실행된다.
  • checkpoint_ns(checkpoint namespace): "어느 그래프(또는 subgraph)에서 나온 체크포인트인가"를 식별한다. 루트 그래프는 빈 문자열("")이고, 노드 안에 subgraph가 있으면 그 subgraph의 체크포인트는 "|"로 구분된 경로(예: 부모 노드 이름이 child라면 "child", 그 안에 또 subgraph가 있으면 "child|grandchild")로 네임스페이스가 나뉜다. 이게 있어야 여러 subgraph가 같은 thread_id를 공유하면서도 서로의 체크포인트를 덮어쓰지 않는다 — Day 10에서 다룰 subgraph 합성의 전제 조건이다.
  • checkpoint_id: 같은 thread_id + checkpoint_ns 안에서 "어느 시점의 스냅샷인가"를 식별하는 단조 증가 id다. 이 값을 config["configurable"]에 명시하면 그 시점의 상태로 정확히 조회/재개할 수 있다 — time travel의 핵심 좌표다.
config = {
    "configurable": {
        "thread_id": "user-42-session-7",
        "checkpoint_ns": "",          # 루트 그래프. subgraph면 "node_name" 형태
        "checkpoint_id": "1f04...36",  # 특정 시점을 지목할 때만 명시
    }
}

백엔드 3종 — InMemorySaver / SqliteSaver / PostgresSaver

백엔드동시성/스케일영속성실무 권고
InMemorySaver
(구 MemorySaver)		 프로세스 메모리, 단일 프로세스 내에서만 프로세스 재시작 시 소실 디버깅/테스트 전용. 소스 docstring이 직접 "production use에는 langgraph-checkpoint-postgres 설치 후 PostgresSaver/AsyncPostgresSaver를 쓰라"고 명시한다.
SqliteSaver/AsyncSqliteSaver 동기 버전은 threading.Lock으로 단일 프로세스 내 스레드 안전성만 확보 파일 기반, WAL 모드로 영속 로컬 개발/데모/소규모 단일 노드 전용. 동기 버전 docstring이 "lightweight, synchronous use cases(데모와 소규모 프로젝트)를 위한 것이며 다중 스레드로 스케일되지 않는다"고 명시하고, 비동기 버전도 "SQLite의 쓰기 성능 한계 때문에 프로덕션 워크로드에는 권장하지 않는다"고 명시한다.
PostgresSaver/AsyncPostgresSaver 커넥션 풀(ConnectionPool/AsyncConnectionPool) 기반 다중 프로세스/다중 인스턴스 지원 DB 영속, 마이그레이션 지원 프로덕션 기본 선택. 단, setup()을 최초 1회 명시적으로 호출해야 스키마가 생성된다 (자동 실행되지 않는다) — docstring이 "반드시 사용자가 직접 호출해야 한다"고 강조한다.
이름이 헷갈리는 이유: MemorySaver는 여전히 존재하지만, 소스 원본을 보면 MemorySaver = InMemorySaver로 하위 호환용 별칭(alias)일 뿐이다 — deprecation 경고 데코레이터 없이 인라인 주석만 달려 있다. 즉 "이름이 바뀌었다"보다는 "정식 이름이 InMemorySaver로 정리됐고 옛 이름도 계속 동작한다"가 정확한 서술이다. 참고로 LangSmith 관리형 배포 환경에서는 checkpointer를 직접 지정하지 않아도 관리형 checkpointer가 자동으로 쓰인다는 안내도 같은 docstring에 있다.
Async 버전을 쓴다고 완전히 논블로킹은 아니다: 실제로 보고된 사례(langgraph#7259)에 따르면 AsyncPostgresSaver가 인스턴스 레벨 threading.Lock()을 두고 있어서, AsyncConnectionPool을 아무리 크게 잡아도(사례에서는 max_size=100) 같은 프로세스 안의 asyncio task들이 이 락에서 직렬화된다. 제보자가 GKE + Cloud SQL 환경에서 500명 동시 사용자로 60초 부하 테스트를 했을 때, 락이 걸린 구현은 199.90 req/s(평균 지연 1923ms), 락을 우회하고 풀에 직접 접근한 baseline은 1295.10 req/s(평균 지연 88ms)로 약 6.5배 처리량 차이가 났다. 이건 DB 용량 문제가 아니라 클라이언트 라이브러리의 락 설계 때문이라는 게 제보자의 진단이다(메인테이너 확인 여부는 fetch 시점 기준 미확인 — 2차 근거로 취급).

get_state / get_state_history / update_state — time travel의 실제 API

컴파일된 그래프(Pregel 인스턴스)는 checkpointer 위에서 세 가지 상태 조회/조작 메서드를 제공한다.

def get_state_history(
    self, config, *, filter: dict | None = None,
    before: RunnableConfig | None = None, limit: int | None = None,

# 같은 thread_id의 전체 체크포인트 이력을 최신순으로 순회한다 — 이게 "time travel"의 조회 절반이다.

def update_state(
    self, config, values, as_node: str | None = None, task_id: str | None = None,
) -> RunnableConfig: ...
# "as_node가 이 값을 반환한 것처럼" 새 체크포인트를 만든다. as_node를 생략하면
# "마지막으로 state를 업데이트한 노드"로 모호하지 않은 한 자동 추론한다.

StateSnapshot은 values(채널 값), next(다음 실행될 노드들), config, metadata, created_at, parent_config(이전 체크포인트를 가리키는 링크), tasks, interrupts를 담는 named tuple이다. parent_config가 체크포인트들을 연결 리스트처럼 잇는다 — 이게 있어서 "이전 체크포인트가 뭐였는지"를 계속 거슬러 올라갈 수 있다.

"과거 체크포인트에서 재실행하면 히스토리가 덮어써지는가, 새 분기가 생기는가"의 답은 설계 의도상 후자(fork)다. 과거의 checkpoint_id를 config에 넣어 invoke하거나 update_state를 호출하면, 그 지점을 parent_config로 삼는 새로운 체크포인트 체인이 만들어진다 — 원본 체크포인트들은 그대로 남고, 그 시점 이후의 "다른 미래"가 별도로 쌓인다. 다만 이게 항상 매끄럽게 동작한다고 단정할 수는 없다 — langgraph#4987에서는 특정 시나리오에서 과거 체크포인트로 재실행했을 때 새 체크포인트에 원본과 동일한 checkpoint_id가 재사용되면서 히스토리 조회가 꼬이는 문제가 보고됐고, 연결된 PR(#5591)이 있다는 것까지는 확인했지만 최종 반영 여부를 소스에서 재확인하지는 못했다. 즉 "포크가 설계 의도"라는 것과 "모든 버전에서 완벽히 그렇게 동작한다"는 것은 별개라는 점을 인지하고, time travel을 프로덕션 워크플로우(예: 승인 후 롤백)에 쓸 때는 실제 버전에서 get_state_history로 분기가 의도대로 생기는지 직접 검증해야 한다.

durability 파라미터 — "체크포인트를 언제 실제로 디스크에 반영할 것인가"

invoke/stream(및 비동기 버전)은 durability: Literal["sync", "async", "exit"] 파라미터를 받는다(langgraph/types.py 원본에 Durability 타입으로 정의돼 있고, 기본값은 "async"다). 의미는 대략 다음과 같다 — "sync"는 다음 스텝으로 넘어가기 전에 체크포인트 쓰기가 동기적으로 완료됨을 보장하고(가장 안전하지만 가장 느림), "async"(기본값)는 쓰기를 다음 스텝 실행과 겹쳐서 비동기로 흘려보내고, "exit"는 그래프 실행이 끝날 때(또는 interrupt로 멈출 때) 딱 한 번만 체크포인트를 기록한다. 이 셋 중 어느 것을 고르느냐가 바로 "매 스텝 풀 스냅샷을 저장"하는 기본 모델의 비용을 실무에서 조절하는 손잡이다 — 중간 스텝 복구가 필요 없는 워크플로우라면 durability="exit"로 쓰기 횟수 자체를 줄일 수 있다.

이걸 뒷받침하는 결정적 증거: langgraph-checkpoint-postgres의 ShallowPostgresSaver (매 스레드마다 최신 체크포인트 1개만 남기고 히스토리를 보존하지 않는 경량 버전, time travel 미지원)는 소스 docstring에서 버전 2.0.20부터 deprecated 처리됐고, 대안으로 명시된 게 바로 "PostgresSaver를 쓰고 graph.invoke(..., durability="exit")로 호출하라"는 것이다. 즉 "히스토리를 안 쌓고 최신 상태만 유지"하고 싶다는 요구는 이제 별도 Saver 클래스가 아니라 durability 파라미터 하나로 표현하는 쪽으로 API가 정리됐다는 뜻이다.

체크포인트 폭증(bloat) — 풀 스냅샷 저장 모델의 실제 비용

Day 5/7에서 다룬 "메시지 히스토리가 길어지면 checkpoint 직렬화 비용이 커진다"는 우려가 실측 수치로 보고된 사례가 있다(langgraph#7714). 제보자에 따르면 16턴짜리 ReAct 에이전트(메시지 65개) 상태를 기본 MemorySaver(현 InMemorySaver)로 직렬화하면 체크포인트가 21,850바이트가 되는데, 같은 상태를 바이너리 풀링 직렬화기로 압축하면 3,217바이트로 줄어든다 — 약 85.3% 스토리지 오버헤드(6.79배)다. 원인으로 지목된 건 매 체크포인트 쓰기마다 채널 값 하나하나를 dumpd()로 Pydantic 풀 메타데이터째 직렬화하는 방식이다. 같은 제보에서 이 직렬화가 컨텍스트 주입용 메시지에도 그대로 반영돼, 의미상 3,587토큰(10,703자)이면 되는 내용이 5,764토큰(20,516자)으로 약 37.8% 토큰 오버헤드가 매 LLM 호출마다 붙는다고 보고됐다. 제보자는 BaseCheckpointSaver에 pluggable SerializerProtocol을 노출해달라고 요청했다.

"풀 스냅샷 = diff 아님"의 함의: 상태에 큰 페이로드(예: 50MB PDF 원문, 대량 검색 결과)를 그대로 들고 있으면, 그래프가 10스텝을 도는 것만으로 체크포인터가 순수하게 500MB를 써야 한다는 계산이 나온다 — 스텝마다 "바뀐 부분"이 아니라 "그 시점의 전체"를 다시 담기 때문이다. 완화 방법은 세 갈래다. (1) Day 6에서 다룬 SummarizationMiddleware·context 압축으로 애초에 state에 들어가는 크기를 줄인다. (2) 큰 원문은 state에 직접 담지 않고 외부 스토리지에 두고 참조(id/URL)만 state에 넣는다. (3) durability="exit"이나 prune/delete_thread API로 저장 빈도·보존 기간 자체를 관리한다. "체크포인터가 알아서 최적화해줄 것"이라는 가정은 근거가 없다.
왜 면접에서 묻는가

면접관이 이 질문으로 평가하는 것

"checkpointer를 써봤다"가 아니라, durability를 운영 요구사항으로 번역할 수 있는가와 "체크포인트는 결국 스토리지 I/O"라는 비용 감각이 있는가를 함께 본다.

  • 백엔드 선택의 근거: "Postgres를 씁니다"가 아니라, 왜 InMemorySaver/SqliteSaver가 프로덕션에서 안 되는지(스케일·동시성 한계)를 소스/docstring 수준에서 설명할 수 있는가
  • 키 구성에 대한 이해: thread_id/checkpoint_ns/checkpoint_id가 "왜 3개나 필요한지"를 subgraph 격리라는 구체적 이유로 설명할 수 있는가 — 이건 Day 10 multi-agent 설계의 전제 지식이기도 하다
  • time travel의 정확한 모델: "덮어쓰기가 아니라 포크"라고 정확히 답하되, 이게 "완벽히 보장된 불변의 사실"이 아니라 알려진 edge case가 있는 영역이라는 걸 아는가 — 과신하지 않는 태도
  • 비용 감각: 체크포인트가 diff가 아니라 풀 스냅샷이라는 걸 알고, 이게 긴 대화·큰 페이로드에서 스토리지/토큰 비용으로 어떻게 이어지는지, durability로 어떻게 조절하는지 설명할 수 있는가
Senior 개발자는 "checkpointer 종류"를 암기해서 답하지 않고, "이 워크플로우가 중간 실패에서 얼마나 정밀하게 복구돼야 하는가"라는 요구사항에서 거꾸로 백엔드와 durability 모드를 도출한다. 이게 mid-level과 senior를 가르는 차이다.
실무 연결

실제 상황과 연결하기

상황 1: 로컬에서 잘 되던 대화 이어가기 기능이 배포 후 컨테이너 재시작마다 초기화됨

로컬 개발 중에는 InMemorySaver로 세션 전환·재접속 테스트가 모두 잘 됐다. 프로덕션에 그대로 배포했더니, 오토스케일러가 파드를 재시작하거나 새 요청이 다른 파드로 라우팅될 때마다 이전 대화가 통째로 사라지는 문제가 드러났다. InMemorySaver의 저장소는 프로세스 메모리이므로 프로세스 경계를 넘으면 아무 것도 남지 않는다는, docstring에 이미 명시된 제약을 실제로 겪은 사례다. → PostgresSaver + setup() 명시 호출로 전환하고서야 안정화됐다.

상황 2: "SQLite면 파일 하나로 되니까 간단하다"고 그대로 프로덕션에 올림

멀티 워커(uvicorn --workers 4) 구성으로 배포한 FastAPI 서비스에서 SqliteSaver를 그대로 썼다가, 동시 요청이 몰릴 때 쓰기 경합과 지연이 늘어나는 증상이 나타났다. SqliteSaver docstring이 명시적으로 "다중 스레드로 스케일되지 않는다"고 밝히고 있고, check_same_thread=False도 내부 threading.Lock으로 안전성을 확보하는 것이지 병렬 처리량을 늘려주는 게 아니었다. → 단일 워커 데모/CLI 도구에는 그대로 두고, 멀티 워커로 서빙하는 서비스는 PostgresSaver로 전환했다.

상황 3: AsyncPostgresSaver로 바꿨는데 동시 사용자 부하에서 처리량이 기대만큼 안 나옴

"비동기 드라이버 + 커넥션 풀이니 동시성이 잘 나올 것"이라는 가정으로 AsyncPostgresSaver(AsyncConnectionPool, max_size=100)로 구성했는데, 500명 동시 사용자 부하 테스트에서 처리량이 기대치를 크게 밑돌았다 (langgraph#7259 계열 사례 — 인스턴스 레벨 threading.Lock()이 asyncio task들을 직렬화한다는 보고, 락 있는 구현 199.90 req/s vs 우회 baseline 1295.10 req/s). → 체크포인터를 요청 경로의 병목 후보로 놓고 별도로 벤치마크하는 습관, 그리고 필요하면 체크포인트 쓰기 빈도를 durability로 낮추는 시도를 우선순위에 넣었다. "async 클래스니까 당연히 논블로킹일 것"이라는 가정을 그대로 믿지 않는 것이 여기서 senior 신호다.

상황 4: 16턴 정도의 대화인데 Postgres 체크포인트 테이블이 예상보다 훨씬 빨리 커짐

메시지 65개짜리 대화 하나의 체크포인트가 21KB를 넘게 잡아먹는다는 걸 뒤늦게 발견했다(langgraph#7714 계열 수치 — 풀 Pydantic 메타데이터 직렬화로 인한 약 85% 오버헤드). 사용자 수 × 평균 대화 길이로 계산해보니 스토리지 증가 속도가 예상 모델과 크게 어긋나 있었다. → 요약 미들웨어(Day 5/6)로 state에 실제로 들어가는 메시지 양 자체를 줄이고, 중간 스텝 복구가 필수가 아닌 워크플로우 구간은 durability="exit"로 전환해 쓰기 횟수를 줄였다. "체크포인터는 알아서 최적화될 것"이라는 가정 없이 실제 바이트 수를 찍어보는 습관이 이런 예산 초과를 조기에 잡아낸다.

핵심 답변 예시

면접 현장 답변 예시 (전체)

"프로덕션에서는 PostgresSaver 또는 AsyncPostgresSaver를 기본으로 씁니다. InMemorySaver는 프로세스 메모리에만 저장되기 때문에 재시작이나 다중 인스턴스 환경에서 상태가 유지되지 않고, SqliteSaver는 소스 docstring에 이미 '다중 스레드로 스케일되지 않는다'고 명시돼 있어서 멀티 워커로 서빙하는 서비스에는 부적합합니다. 다만 PostgresSaver를 쓸 때도 setup()을 최초 1회 명시적으로 호출해야 스키마가 생성된다는 걸 놓치기 쉽고, async 버전이라고 해서 완전히 논블로킹이라고 단정하지 않습니다 — 실제로 AsyncPostgresSaver의 인스턴스 레벨 락이 커넥션 풀의 동시성을 제한한다는 보고가 있었기 때문에, 체크포인터도 요청 경로의 병목 후보로 놓고 별도로 벤치마크합니다.

thread_id/checkpoint_ns/checkpoint_id는 서로 다른 걸 식별합니다. thread_id는 대화·세션 단위이고, checkpoint_ns는 어느 (sub)그래프에서 나온 체크포인트인지를 구분하는 네임스페이스로, 루트는 빈 문자열이고 중첩된 subgraph는 파이프로 구분된 경로를 갖습니다. 이게 없으면 여러 subgraph가 같은 thread_id를 공유할 때 서로의 상태를 덮어쓰게 되므로, subgraph를 쓰는 순간부터 checkpoint_ns는 선택이 아니라 필수 격리 메커니즘이 됩니다. checkpoint_id는 그 안에서 특정 시점을 지목하는 좌표입니다.

get_state_history로 thread의 체크포인트 이력을 최신순으로 조회할 수 있고, 이 중 과거의 checkpoint_id를 config에 넣어 다시 invoke하거나 update_state를 호출하면, 그 지점을 parent_config로 삼는 새로운 체크포인트 체인이 만들어집니다 — 설계 의도는 기존 히스토리를 덮어쓰는 게 아니라 분기(fork)하는 겁니다. 다만 이게 모든 버전에서 완벽하게 보장된다고 과신하지는 않습니다. 특정 시나리오에서 재실행 시 체크포인트 id가 재사용되며 히스토리 조회가 꼬였다는 보고를 본 적이 있어서, 승인 후 롤백처럼 time travel을 프로덕션 로직에 직접 쓸 때는 실제 버전에서 get_state_history로 분기가 의도대로 생기는지 검증하는 절차를 거칩니다.

마지막으로 체크포인트 저장 모델 자체의 비용을 감안합니다. 체크포인트는 이전 대비 diff가 아니라 그 시점 channel_values 전체의 스냅샷이라서, 메시지 히스토리가 길어지거나 state에 큰 페이로드가 들어 있으면 스텝마다 전체를 다시 저장하는 비용이 누적됩니다. 실제로 16턴 정도의 ReAct 에이전트에서 풀 메타데이터 직렬화 때문에 스토리지 오버헤드가 크게 나타난 사례가 보고된 적이 있어서, 저는 두 가지로 대응합니다 — 하나는 요약 미들웨어로 state에 들어가는 메시지 양 자체를 줄이는 것이고, 다른 하나는 중간 스텝 복구가 필수가 아닌 구간에서 durability='exit'로 체크포인트 쓰기 빈도 자체를 낮추는 것입니다. durability는 sync/async/exit 세 모드가 있고, 기본값은 async입니다."

Senior-level Follow-up

면접관이 이어서 묻는 질문들

F1. "InMemorySaver는 그냥 dict 아닌가요? 왜 굳이 checkpointer 인터페이스를 씌우나요?"

약한 답변: "그냥 메모리에 저장하는 거니까 dict나 마찬가지입니다."

강한 답변: "저장 위치는 dict에 가깝지만, InMemorySaver도 BaseCheckpointSaver 인터페이스(get_tuple/list/put/put_writes 등)를 그대로 구현합니다. 이게 중요한 이유는, 개발 중엔 InMemorySaver로 빠르게 반복하다가 프로덕션에서는 코드 변경 없이 config만 바꿔서 PostgresSaver를 주입할 수 있게 하기 위해서입니다. 체크포인터를 그래프 코드에서 분리해서 런타임에 주입 가능한 인터페이스로 만든 설계라는 걸 이해하는 게 핵심입니다."

F2. "SqliteSaver가 로컬 파일 하나로 충분한데 왜 프로덕션에 못 쓰나요?"

강한 답변: "파일 기반이라는 것 자체가 문제는 아니고, SQLite의 쓰기 모델이 다중 프로세스·다중 워커 동시 쓰기에 최적화돼 있지 않다는 게 핵심입니다. 동기 SqliteSaver는 threading.Lock으로 같은 프로세스 안의 스레드 안전성은 확보하지만, 이건 병렬 처리량을 보장하는 게 아니라 충돌을 막는 것뿐입니다. 비동기 버전도 docstring에서 SQLite 쓰기 성능 한계를 이유로 프로덕션에 권장하지 않는다고 명시돼 있습니다. 그래서 단일 프로세스 CLI 도구나 로컬 데모에는 적합하지만, 여러 워커가 동시에 쓰는 서비스에는 Postgres 같은 진짜 다중 접속 DB가 필요합니다."

F3. "과거 체크포인트에서 update_state를 호출하면 무슨 일이 일어나나요? 원본 히스토리가 사라지나요?"

강한 답변: "사라지지 않는 게 설계 의도입니다. update_state는 지정한 config(과거 checkpoint_id 포함)를 parent로 삼아 새 체크포인트를 만들기 때문에, 원본 체인은 그대로 남고 그 지점에서 갈라진 새로운 분기가 생깁니다. as_node를 지정하면 '그 노드가 이 값을 반환한 것처럼' 취급되고, 생략하면 마지막으로 state를 바꾼 노드로 추론합니다. 다만 이 fork 동작에 대해 일부 시나리오에서 checkpoint id가 예상과 다르게 재사용된다는 이슈 보고가 있었기 때문에, 이걸 신뢰해서 프로덕션 롤백 로직을 짤 때는 실제로 get_state_history를 찍어서 분기가 의도대로 생기는지 확인하는 방어적 습관을 둡니다."

F4. "checkpoint가 매 스텝 풀 스냅샷이면 긴 대화에서 비용이 선형이 아니라 더 커지는 거 아닌가요?"

약한 답변: "그런가요? 딱히 문제된 적은 없었습니다."

강한 답변: "맞습니다. 스텝마다 diff가 아니라 그 시점 channel_values 전체를 다시 저장하기 때문에, 메시지 리스트가 append 방식으로 계속 자라는 대화라면 매 턴 저장되는 바이트 수 자체가 계속 커지고, 턴 수가 늘수록 누적 저장량은 대화 길이의 제곱에 가까운 형태로 늘어날 수 있습니다. 실제로 16턴 정도의 에이전트에서 풀 Pydantic 메타데이터 직렬화로 인한 오버헤드가 실측된 사례도 있었습니다. 그래서 요약 미들웨어로 state에 들어가는 메시지 양 자체를 관리하거나, durability 모드로 쓰기 빈도를 조절하는 게 긴 세션을 다루는 프로덕션 설계에서 필수적입니다."

F5. "AsyncPostgresSaver를 쓰면 완전히 비동기 논블로킹인가요?"

강한 답변: "클래스 이름과 API 시그니처는 async이지만, 그게 곧 병목이 없다는 뜻은 아닙니다. 실제로 인스턴스 레벨 threading.Lock()이 커넥션 풀의 동시 활용을 제한한다는 부하 테스트 보고가 있었고, 그 사례에서는 락이 있는 구현과 없는 baseline 사이에 처리량이 6배 이상 차이 났습니다. 그래서 'async면 당연히 잘 스케일될 것'이라고 가정하지 않고, 실제 배포 환경에서 체크포인터를 요청 경로의 잠재적 병목으로 놓고 부하 테스트를 해봅니다."

좋은 답변 vs 약한 답변

차이를 명확히 보기

강한 답변의 특징

  • 백엔드 선택을 스케일·동시성 제약(단일 프로세스 vs 다중 워커)으로 설명한다
  • thread_id/checkpoint_ns/checkpoint_id를 subgraph 격리라는 구체적 필요로 연결한다
  • Time travel을 "fork"로 정확히 설명하되, 알려진 edge case를 알고 과신하지 않는다
  • 체크포인트가 diff가 아니라 풀 스냅샷이라는 저장 모델을 인지하고 비용으로 연결한다
  • "async 버전이니 논블로킹일 것"처럼 이름만 보고 성능을 단정하지 않고 실측을 우선한다

약한 답변의 특징

  • "Postgres가 제일 좋다"는 결론만 암기하고 이유(스케일 제약)를 설명하지 못한다
  • checkpoint_ns를 몰라서 생략하거나 thread_id와 혼동한다
  • Time travel을 "과거로 되돌아간다" 정도로만 뭉뚱그리고, 덮어쓰기인지 분기인지 구분하지 못한다
  • 체크포인트 저장을 "그냥 상태 저장"으로만 이해하고 비용 구조를 생각해본 적이 없다
  • 클래스 이름(Async...)만 보고 동시성 문제가 없다고 가정한다
핵심 차이: 강한 답변은 checkpointer API 목록을 나열하지 않고, "이 상태를 언제, 얼마나 자주, 얼마나 크게 저장하고, 실패 시 어디까지 되돌릴 수 있어야 하는가"라는 운영 요구사항을 축으로 백엔드·키 구성·durability를 일관되게 설명한다.
흔한 오해 정정

실무에서 자주 나오는 잘못된 믿음

오해 1: "MemorySaver가 이름만 InMemorySaver로 바뀐 거니 어차피 똑같다"

사실: 기능은 동일한 별칭(MemorySaver = InMemorySaver)이 맞지만, 이름이 바뀌었다는 사실 자체보다 중요한 건 이름과 무관하게 이 클래스는 프로덕션에 쓰면 안 된다는 것이다. 이름 변경 이력을 아는 것과, 그 클래스의 근본적인 스케일 제약을 아는 것은 다른 질문이다.

오해 2: "SQLite는 파일 기반이라 Postgres보다 무조건 더 가볍고 안전하다"

사실: 단일 프로세스·단일 워커 범위에서는 맞지만, 멀티 워커·멀티 인스턴스로 확장하는 순간 SQLite의 쓰기 모델이 병목이 된다. "가볍다"는 장점과 "동시 쓰기 확장성"은 서로 다른 축이다.

오해 3: "체크포인트는 이전 상태 대비 diff만 저장한다"

사실: Checkpoint.channel_values는 그 시점 채널 값 전체의 매핑이다. 매 superstep마다 전체 스냅샷을 다시 담아 저장하며, 이게 긴 대화·큰 페이로드에서 스토리지/토큰 비용으로 직결된다.

오해 4: "과거 체크포인트로 되돌아가 재실행하면 원본 히스토리가 그대로 덮어써진다"

사실: 설계 의도는 덮어쓰기가 아니라 포크(분기)다 — 지정한 과거 지점을 parent_config로 삼는 새 체크포인트 체인이 만들어지고 원본은 남는다. 다만 일부 시나리오에서 체크포인트 id 재사용 관련 이슈가 보고된 적이 있으므로, "항상 완벽하게 분기된다"고 무조건 신뢰하기보다는 실제 버전에서 검증하는 태도가 필요하다.

오해 5: "checkpoint_ns는 thread_id처럼 아무 값이나 넣어도 되는 부가 필드다"

사실: checkpoint_ns는 subgraph 경로를 식별하는 전용 네임스페이스다. 루트 그래프는 빈 문자열이고, 중첩된 subgraph는 "|"로 구분된 경로를 갖는다. 이 값이 없으면 같은 thread_id를 공유하는 여러 subgraph의 체크포인트가 서로 충돌한다.

핵심 요약

오늘 반드시 기억할 것

  • Checkpointer는 매 superstep 종료 시점의 전체 상태 스냅샷을 저장/복원하는 durability 계층이다 — 장애 복구, thread-scoped memory(Day 5), interrupt 재개(Day 9)의 공통 기반이다.
  • InMemorySaver(구 MemorySaver)/SqliteSaver는 개발·데모 전용, PostgresSaver/ AsyncPostgresSaver가 프로덕션 기본이다. setup()은 최초 1회 명시 호출이 필요하고, async 버전도 인스턴스 락으로 인한 동시성 제약이 보고된 적이 있어 성능을 실측해야 한다.
  • thread_id(세션)/checkpoint_ns(subgraph 네임스페이스, "|" 구분)/checkpoint_id (시점)가 체크포인트 키를 구성한다. checkpoint_ns는 subgraph 격리에 필수다.
  • get_state_history + update_state로 하는 time travel은 덮어쓰기가 아니라 포크다 — 과거 지점을 parent_config로 삼는 새 체크포인트 체인이 생긴다. 다만 알려진 edge case가 있어 프로덕션 적용 전 버전별 검증이 필요하다.
  • 체크포인트는 diff가 아니라 channel_values 전체의 풀 스냅샷이다. 긴 메시지 히스토리·큰 페이로드에서 스토리지/토큰 비용으로 직결되며, 요약/오프로딩(Day 5/6)과 durability(sync/async/exit, 기본 async)로 쓰기 빈도를 조절해 비용을 관리한다.
  • ShallowPostgresSaver(최신 체크포인트 1개만 유지, time travel 미지원)는 버전 2.0.20부터 deprecated됐고, 같은 요구는 PostgresSaver + durability="exit"로 대체됐다.
