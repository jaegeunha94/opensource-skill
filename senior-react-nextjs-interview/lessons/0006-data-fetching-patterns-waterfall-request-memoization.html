1. 핵심 면접 질문

Primary Question

"상세 페이지에서 사용자 정보를 가져온 뒤 그 사용자의 주문 내역을 가져오는데, 두 요청이 순차적으로 실행되어 페이지 로딩이 느립니다. 원인과 해결책을 설명해 주세요."

Variant 1

"같은 페이지 안에서 헤더 컴포넌트와 사이드바 컴포넌트가 각각 fetch(\"/api/user\")를 호출하는데, 네트워크 탭을 보면 같은 요청이 한 번만 나갑니다. 왜 그런 겁니까? Redux/React Query 없이 이게 어떻게 가능한가요?"

Variant 2

"어떤 데이터는 Server Component에서 fetch하고, 어떤 데이터는 클라이언트 컴포넌트에서 useEffect나 React Query로 fetch합니다. 이 둘을 나누는 기준이 무엇입니까?"

2. 왜 면접에서 묻는가

이 질문은 "fetch를 어디에 쓸 줄 아는가"가 아니라, "여러 개의 비동기 작업이 렌더링 트리 안에서 어떤 순서로 실행되는지를 머릿속에 그릴 수 있는가"를 테스트합니다. 워터폴 문제는 코드 문법 오류가 아니라 실행 순서에 대한 착각에서 생기고, 프로덕션에서 "이 페이지 왜 이렇게 느려요"라는 리포트의 상당수가 여기서 시작됩니다.

면접관이 이 질문으로 평가하는 것:

  • "독립적인 요청과 의존적인 요청을 구분할 수 있는가" — 두 fetch가 서로의 결과를 필요로 하는지, 아니면 그냥 코드 순서상 나란히 적혀 있을 뿐인지를 구별하는가.
  • await의 위치가 실행 시점을 결정한다는 것을 아는가 — await를 어디에 두느냐에 따라 같은 코드가 순차 실행이 될 수도, 병렬 실행이 될 수도 있다는 것을 아는가.
  • Next.js/React의 request memoization을 정확히 이해하는가 — "왜 같은 요청인데 한 번만 나가지?"라는 현상을 마법이 아니라 구체적인 캐시 키(입력값) 기반 메커니즘으로 설명할 수 있는가.
  • Server fetch와 client fetch의 역할 차이를 아는가 — 최초 페이지 렌더링에 필요한 데이터와, 사용자 상호작용 이후 발생하는 재요청을 다른 계층의 문제로 분리해서 설계하는가.
  • 병렬화의 한계를 아는가 — 무조건 Promise.all로 몰아넣는 게 아니라, 컴포넌트 트리 구조상 병렬화가 어려운 지점(깊이 중첩된 fetch)까지 처리하는 방법(preload 패턴)을 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "React Query를 쓰면 캐싱되니까 빨라져요." 이 답변은 워터폴의 원인이 캐싱 부재가 아니라 실행 순서 자체라는 점을 놓치고 있습니다. 첫 요청과 두 번째 요청이 서로 무관한데도 순차적으로 실행되도록 코드를 작성했다면, 아무리 좋은 캐시 라이브러리를 붙여도 최초 로딩에서는 여전히 두 배의 대기 시간이 발생합니다. 시니어는 캐시보다 먼저 "이 두 요청이 정말 서로를 기다려야 하는가"를 묻습니다.

3. 전제 개념 — 쉬운 한국어로

"워터폴(waterfall)"이란 무엇인가

워터폴이란 여러 개의 네트워크 요청이 폭포처럼 하나씩 순서대로 떨어지며 실행되는 상황을 말합니다. 요청 A가 끝나야 요청 B가 시작되고, B가 끝나야 C가 시작되는 식입니다. 문제는 이 세 요청이 서로의 결과를 실제로 필요로 하지 않는데도 코드가 우연히 순서대로 적혀 있어서 순차 실행되는 경우입니다. 이 경우 전체 로딩 시간은 세 요청 시간의 합이 되지만, 동시에 실행했다면 가장 오래 걸리는 한 요청의 시간만큼만 기다리면 됩니다.

"동기(synchronous)"와 "비동기(asynchronous)"의 차이를 다시 짚기

동기 코드는 한 줄이 끝나야 다음 줄이 실행됩니다. 비동기 코드(네트워크 요청, 타이머 등)는 "결과를 기다리는 동안 다른 일을 할 수 있는" 코드입니다. await는 "이 지점에서 결과가 올 때까지 이 함수의 다음 줄 실행을 멈춘다"는 뜻이지, "다른 요청도 다 멈춘다"는 뜻이 아닙니다. 이 차이를 정확히 알아야, 왜 await를 어디에 쓰느냐에 따라 같은 두 개의 fetch가 순차 실행이 되기도 하고 병렬 실행이 되기도 하는지 이해할 수 있습니다.

"메모이제이션(memoization)"이란 무엇인가

메모이제이션이란 같은 입력값으로 같은 함수를 다시 호출했을 때, 다시 계산하지 않고 이전에 계산해 둔 결과를 그대로 돌려주는 기법입니다. React/Next.js의 request memoization은 이 개념을 fetch 요청에 적용한 것입니다 — 같은 렌더링 한 번 안에서, 같은 URL과 같은 옵션으로 fetch를 여러 번 호출해도 실제 네트워크 요청은 한 번만 나가고, 나머지 호출은 첫 번째 요청의 결과를 재사용합니다.

"컴포넌트 트리에서의 fetch"라는 그림

Next.js App Router에서는 하나의 페이지가 여러 겹의 컴포넌트(레이아웃, 페이지, 하위 컴포넌트)로 이루어지고, 이들 각각이 서버에서 독립적으로 자기 데이터를 fetch할 수 있습니다. 이 구조의 장점은 "각 컴포넌트가 자기 데이터를 스스로 책임진다"는 관심사 분리이지만, 단점은 부모가 먼저 렌더링을 시작해야 자식이 렌더링을 시작할 수 있는 트리 구조 자체가, 부모의 fetch와 자식의 fetch를 의도치 않게 순차적으로 만들 수 있다는 것입니다. 이게 워터폴이 컴포넌트 트리 깊은 곳에서 "숨어서" 생기는 이유입니다.

4. 서버 fetch vs 클라이언트 fetch 판단 기준

fetch를 어디서 할지는 라이브러리 취향이 아니라, "이 데이터가 언제 필요한가"와 "누가 이 데이터를 다시 요청하게 만드는가"로 결정됩니다.

Server Component에서 fetch해야 하는 경우

페이지가 처음 열릴 때부터 필요한 데이터, 즉 최초 렌더링에 반드시 있어야 화면이 의미를 갖는 데이터입니다. 게시글 상세 내용, 상품 정보, 사용자 프로필처럼 "이게 없으면 애초에 보여줄 게 없는" 데이터가 여기에 해당합니다. 서버에서 fetch하면 브라우저로 API 키나 인증 토큰을 노출하지 않아도 되고, 클라이언트 자바스크립트 번들에 데이터 페칭 로직을 포함시키지 않아도 되고, 네트워크 왕복이 서버-DB/서버-API 사이의 빠른 내부망에서 일어난다는 이점이 있습니다.

클라이언트 컴포넌트에서 fetch해야 하는 경우

사용자의 상호작용 이후에 발생하는 요청입니다. 검색창에 타이핑할 때마다 자동완성을 요청하거나, 무한 스크롤로 다음 페이지를 가져오거나, "좋아요" 버튼을 눌러 서버에 반영하거나, 특정 데이터를 주기적으로 폴링하는 경우입니다. 이런 요청은 페이지가 이미 렌더링된 이후, 사용자의 행동을 트리거로 삼아 발생하므로 애초에 서버 렌더링 시점에 존재할 수 없는 요청입니다.

구분 기준을 한 문장으로: "이 요청이 페이지 렌더링 이전에 결정되는가, 아니면 사용자의 행동 이후에 결정되는가?" 전자는 서버(Server Component fetch), 후자는 클라이언트(React Query/SWR 등)입니다. 실무 페이지 대부분은 두 계층이 함께 존재하는 하이브리드 구조입니다 — 최초 목록은 서버에서 가져오고, "더 보기" 버튼을 누른 이후의 페이지들은 클라이언트에서 가져오는 식입니다.

5. 워터폴이 생기는 구조적 이유

// 안티패턴: 순차적 워터폴
async function OrderDetailPage({ userId }) {
  const user = await getUser(userId);           // 1) 사용자 정보 요청 완료까지 대기
  const orders = await getOrders(user.id);       // 2) 그 다음에야 주문 내역 요청 시작
  return ;
}

이 코드가 반드시 순차 실행되어야 하는 이유가 있는지부터 살펴봐야 합니다. getOrders가 user.id를 인자로 받는다는 사실만 보면 "당연히 순서대로 해야지"라고 생각하기 쉽지만, 실제로 필요한 것은 userId(URL 파라미터로 이미 갖고 있는 값)이지 user.id(API 응답에서 뽑아낸 값)가 아닐 수도 있습니다. 만약 주문 API가 userId 파라미터만으로 조회 가능하다면, 이 두 요청은 서로 의존 관계가 없는데도 코드가 순서대로 적혀 있어서 순차 실행되는, 진짜 워터폴입니다.

진짜 의존 관계 vs 가짜(우연한) 의존 관계

진짜 의존 관계는 "B 요청의 파라미터 자체가 A 요청의 응답에서만 얻을 수 있는 값"인 경우입니다(예: A가 반환한 orderId 목록으로 각 주문의 상세를 조회). 가짜 의존 관계는 "코드를 그렇게 적었을 뿐, B 요청에 필요한 값은 이미 다른 곳(props, URL, 세션)에 있는" 경우입니다. 워터폴 문제를 진단할 때 가장 먼저 할 일은 이 둘을 구분하는 것입니다 — 실무에서 나오는 워터폴 버그의 상당수는 후자(가짜 의존 관계)입니다.

컴포넌트 트리 깊이가 만드는 워터폴

더 알아채기 어려운 워터폴은 컴포넌트 트리의 서로 다른 계층에서 생깁니다.

6. 해결책 — 병렬 fetch 패턴

같은 컴포넌트 안의 워터폴 — Promise.all

// 개선: 두 요청을 동시에 시작
async function OrderDetailPage({ userId }) {
  const [user, orders] = await Promise.all([
    getUser(userId),
    getOrders(userId), // user.id가 아니라 userId를 그대로 사용 — 진짜 의존 관계가 없음을 확인했기 때문에 가능
  ]);
  return ;
}

핵심은 await를 각 fetch 호출 직후가 아니라, 두 Promise를 먼저 "시작"시켜 놓고 Promise.all로 한꺼번에 기다리는 것입니다. 이렇게 하면 네트워크 요청 두 개가 거의 동시에 나가고, 전체 대기 시간은 두 요청 시간의 합이 아니라 더 오래 걸리는 하나의 요청 시간으로 줄어듭니다.

진짜 의존 관계가 있을 때 — 부분 병렬화

// user는 orders 조회에 필요 없지만, 화면에 표시할 다른 값(프로필 등)이 필요한 경우
async function OrderDetailPage({ userId }) {
  const userPromise = getUser(userId);      // 시작만 해두고 아직 await 안 함
  const ordersPromise = getOrders(userId);  // 이것도 시작만 해둠

  const [user, orders] = await Promise.all([userPromise, ordersPromise]);
  return ;
}

여기서 배워야 할 원칙은 "Promise를 변수에 담아두는 시점"과 "await로 결과를 기다리는 시점"을 분리해서 생각하는 것입니다. const p = fetchSomething()은 이미 네트워크 요청을 시작시킵니다. await는 나중에 해도 되고, 그 사이에 다른 Promise를 추가로 시작시킬 수 있습니다.

7. Preload 패턴 — 컴포넌트 트리 깊은 곳의 워터폴

앞서 본 Page → OrderHistory처럼 서로 다른 컴포넌트 계층에 걸친 워터폴은 Promise.all 하나로 해결할 수 없습니다(애초에 서로 다른 함수/파일이기 때문입니다). 이때 쓰는 것이 preload 패턴입니다.

// lib/data.js
import { cache } from "react";

export const getUser = cache(async (userId) => {
  const res = await fetch(`https://api.example.com/users/${userId}`);
  return res.json();
});

export function preloadUser(userId) {
  void getUser(userId); // 결과를 기다리지 않고 요청만 미리 시작
}

// page.js
import { getUser, preloadUser } from "./lib/data";

export default async function Page({ userId }) {
  preloadUser(userId); // 부모가 자기 fetch를 하기 "전에" 자식이 필요할 fetch를 미리 트리거
  const user = await getUser(userId); // 나중에 실제로 필요할 때 호출하면, 이미 진행 중인 요청 결과를 재사용(request memoization)

  );
}

;
}

여기서 preloadUser는 결과를 await하지 않고 "미리 시작만 시켜두는" 트리거 함수입니다. 이렇게 하면 getUser와 getOrders(자식 컴포넌트가 실행하는)의 네트워크 요청이 거의 동시에 시작될 수 있습니다. 이 패턴이 성립하려면 반드시 다음 절에서 다룰 request memoization이 전제되어야 합니다 — 그렇지 않으면 preload로 미리 부른 요청과, 나중에 실제로 데이터를 쓸 때 다시 부르는 요청이 중복으로 두 번 나가게 됩니다.

8. Request Memoization — 같은 요청이 왜 한 번만 나가는가

Next.js App Router에서 fetch는 같은 렌더링 한 번(single render pass) 동안, 같은 URL과 같은 옵션(메서드, 헤더 등)으로 여러 번 호출되어도 실제 네트워크 요청은 한 번만 나갑니다. 두 번째, 세 번째 호출은 첫 번째 호출이 반환할 Promise를 그대로 재사용합니다.

// Header.js와 Sidebar.js가 각각 독립적으로 작성했지만
async function Header() {
  const user = await fetch("https://api.example.com/user").then((r) => r.json());
  return 
{user.name}
;
}

async function Sidebar() {
  const user = await fetch("https://api.example.com/user").then((r) => r.json());
  return 
{user.email}
;
}

// Page.js
export default function Page() {
  return (
    <>
      

      
    
  );
}

Header와 Sidebar는 서로의 존재를 모르고 각자 fetch를 호출했지만, 이 페이지를 한 번 렌더링하는 동안 실제 네트워크 요청은 한 번만 나갑니다. 이것이 "왜 같은 요청인데 중복 호출이 안 되지?"라는 현상의 정체이며, 마법이 아니라 URL과 옵션을 키로 삼아, 같은 렌더링 생명주기 안에서 Promise를 캐싱하는 구체적인 메커니즘입니다.

일반 함수를 memoize하려면 — React의 cache()

위 예시는 fetch 자체에 내장된 동작이지만, ORM 쿼리나 외부 SDK 호출처럼 fetch를 직접 쓰지 않는 데이터 소스는 자동으로 memoize되지 않습니다. 이럴 때는 React의 cache() 함수로 직접 감싸야 합니다.

import { cache } from "react";
import { db } from "./db";

// cache로 감싸지 않으면 Header, Sidebar가 각자 호출할 때마다 실제 DB 쿼리가 두 번 나감
export const getUser = cache(async (userId) => {
  return db.user.findUnique({ where: { id: userId } });
});

이 메커니즘이 프로덕션에서 갖는 실질적 의미: 컴포넌트를 작성하는 개발자는 "이 데이터를 다른 컴포넌트가 이미 가져왔는지"를 신경 쓸 필요 없이, 각자 필요한 데이터를 필요한 곳에서 자유롭게 fetch하도록 작성할 수 있습니다. 중복 제거는 프레임워크가 렌더링 트리 전체를 보고 자동으로 처리합니다 — 이것이 "컴포넌트 단위로 데이터를 요청하는" App Router의 설계가 성능 문제 없이 동작하는 핵심 전제입니다.

9. Request Memoization과 Data Cache의 경계

Request memoization을 Next.js의 fetch 응답 캐싱(Data Cache, Day 16에서 다룸)과 혼동하기 쉽습니다. 지금 단계에서는 이 둘의 경계만 정확히 짚어 둡니다.

구분 Request Memoization Data Cache
생존 범위 단 한 번의 렌더링(하나의 요청 처리) 동안만 렌더링을 넘어, 여러 사용자의 여러 요청에 걸쳐 서버 재시작 전까지
목적 같은 렌더링 안에서 컴포넌트끼리 중복 요청을 막음 같은 데이터를 반복해서 원본 소스(DB, API)에 다시 묻지 않도록 함
끄는 방법 끌 수 없음(React 렌더링 모델 자체의 동작) fetch(url, { cache: "no-store" }) 등으로 제어 가능
비유 같은 회의 안에서 같은 질문을 두 번 하면 "방금 답했잖아요" 다음 회의에서도 지난 회의에서 정리한 답변 문서를 그대로 재사용

지금 시점에서 기억해야 할 것은 하나입니다: request memoization은 "같은 페이지 렌더링 한 번 안에서 중복 요청을 없애는 것"이고, 그 이상(요청 간 캐싱)은 별도의 계층(Data Cache)이 담당한다는 것입니다. 이 경계를 헷갈리면 "왜 새로고침했는데도 옛날 데이터가 나오지?"(Data Cache 이슈)와 "왜 같은 페이지에서 요청이 중복되지?"(Request Memoization 이슈)를 서로 다른 원인인데 같은 원인으로 착각해 잘못된 곳을 고치게 됩니다.

10. 판단 기준표

상황 판단 근거
페이지가 열리자마자 필요한 데이터 Server Component fetch 최초 렌더링 전에 이미 필요가 확정되어 있음
사용자 클릭/타이핑 이후에만 필요한 데이터 클라이언트 fetch (React Query/SWR) 렌더링 시점에는 아직 필요 여부가 결정되지 않음
두 fetch가 서로 다른 데이터 소스를 조회하고, 인자가 서로 독립적 Promise.all로 병렬 시작 진짜 의존 관계 없음 — 순차 실행은 불필요한 대기
두 fetch가 서로 다른 컴포넌트 계층에 있지만 독립적 preload 패턴 + cache() 트리 구조상 병렬화가 안 보이지만, 미리 트리거해 실질적으로 동시 시작
B의 파라미터가 A의 응답 값 자체에서만 나옴 순차 실행 유지(진짜 워터폴 아님) 강제로 병렬화하면 B가 필요한 값이 아직 없어 실행 불가
fetch가 아닌 ORM/SDK 호출을 여러 컴포넌트가 공유 cache()로 명시적 memoize fetch 내장 memoization의 대상이 아님

11. 증상 기반 진단표

증상 의심할 원인 확인 방법
Network 탭에서 요청이 계단식으로 하나씩 순서대로 끝나야 다음이 시작됨 진짜 의존 관계가 없는 fetch를 순차 await로 작성함 각 fetch의 파라미터가 실제로 이전 fetch의 응답 값에 의존하는지 확인 → 독립적이면 Promise.all로 전환
코드에는 병렬처럼 안 보이는데도, 부모 렌더링이 끝나야 자식의 fetch가 시작됨 컴포넌트 트리 계층 간 암묵적 워터폴 서버 컴포넌트 렌더링 타이밍을 로그로 확인 → preload 패턴 적용
ORM/DB 쿼리가 여러 컴포넌트에서 각자 호출되어 같은 쿼리가 중복 실행됨 fetch가 아닌 데이터 소스라 request memoization 대상이 아님 DB 쿼리 로그에서 동일 쿼리 중복 확인 → 해당 함수를 cache()로 감싸기
preload를 추가했는데도 요청이 여전히 두 번 나감 preload에서 부른 함수와 실제 사용하는 함수가 cache()로 감싸지지 않았거나, URL/옵션이 미묘하게 다름 두 호출의 함수 참조와 인자가 완전히 동일한지 확인
새로고침해도 오래된 데이터가 계속 보임(중복 요청 문제와는 다른 증상) Request memoization이 아니라 Data Cache 설정 문제 이건 Day 16의 캐시 계층 이슈 — cache: "no-store" 또는 revalidate 설정 확인

12. 답변 사고 순서

  1. 먼저 의존 관계를 확인합니다. "주문 내역 조회가 정말 사용자 정보 조회의 결과를 필요로 하는지, 아니면 이미 갖고 있는 userId만으로 충분한지부터 확인하겠습니다."
  2. 진짜 의존 관계가 없다면 병렬화를 제안합니다. "의존 관계가 없다면 두 요청을 각각 await하는 대신, Promise.all로 동시에 시작해 전체 대기 시간을 두 요청의 합이 아니라 더 긴 하나의 요청 시간으로 줄이겠습니다."
  3. 컴포넌트 트리 계층 문제인지 확인합니다. "만약 이 두 fetch가 서로 다른 컴포넌트에 있어서 Promise.all로 묶기 어렵다면, 상위 컴포넌트에서 preload 함수로 자식이 필요할 요청을 미리 트리거하겠습니다."
  4. 중복 실행 문제와 구분합니다. "이때 preload와 실제 사용 지점에서 같은 요청이 중복 실행되지 않으려면, 그 함수가 fetch이거나 React의 cache()로 감싸져 있어야 한다는 전제를 함께 확인하겠습니다."
  5. 실무 연결로 마무리합니다. "이런 워터폴은 코드 리뷰에서 놓치기 쉬우므로, 새로운 페이지를 배포하기 전에 Network 탭에서 요청이 계단식으로 순차 실행되는지 확인하는 것을 체크리스트에 넣는 것이 실무적으로 도움이 됩니다."

13. 핵심 답변 예시

면접관: "상세 페이지에서 사용자 정보를 가져온 뒤 그 사용자의 주문 내역을 가져오는데, 두 요청이 순차적으로 실행되어 페이지 로딩이 느립니다. 원인과 해결책을 설명해 주세요."

STRONG ANSWER

가장 먼저 확인할 것은 주문 내역 조회가 정말로 사용자 정보 조회의 응답값을 필요로 하는지입니다. 만약 주문 API가 userId 파라미터만으로 조회 가능하고, 사용자 정보 API의 응답에서 나온 다른 값(예: 내부 계정 ID)이 실제로 필요한 게 아니라면, 이 두 요청은 서로 독립적인데도 코드 순서 때문에 우연히 순차 실행되는 워터폴입니다.


이 경우 두 await를 각각 순서대로 두는 대신, Promise.all([getUser(userId), getOrders(userId)])로 두 요청을 동시에 시작시키겠습니다. 이렇게 하면 전체 대기 시간이 두 요청 시간의 합이 아니라, 둘 중 더 오래 걸리는 요청 하나의 시간으로 줄어듭니다.


만약 이 두 fetch가 서로 다른 컴포넌트 계층에 있어서(예: 부모 페이지가 사용자 정보를, 자식 컴포넌트가 주문 내역을 각자 fetch하는 구조) 하나의 함수 안에서 Promise.all로 묶기 어렵다면, preload 패턴을 쓰겠습니다. 부모 컴포넌트에서 자식이 필요할 getOrders를 결과를 기다리지 않고 미리 트리거해두고, 실제로 그 값이 필요한 자식 컴포넌트에서 나중에 await하면 이미 진행 중인 요청의 결과를 그대로 받습니다. 이때 전제는 해당 함수가 fetch이거나 React의 cache()로 감싸져 있어서, 미리 트리거한 요청과 나중에 실제로 쓰는 호출이 같은 요청으로 memoize된다는 것입니다. 이 전제가 없으면 preload는 오히려 요청을 두 번 만듭니다.

이 답변이 좋은 이유:

  • 병렬화를 제안하기 전에 "이게 진짜 의존 관계인지"부터 확인하는 판단 순서를 지킴
  • Promise.all과 preload 패턴을 상황(같은 함수 안 vs 서로 다른 컴포넌트 계층)에 맞게 구분해서 제시함
  • preload가 성립하는 전제 조건(request memoization)까지 정확히 짚어, 어설프게 적용했을 때의 함정을 미리 방지함

14. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"React Query로 바꾸면 캐싱이 되니까 두 번째 방문부터는 빨라질 겁니다."

이 답변이 약한 이유

  • 문제는 "캐시가 없어서"가 아니라 "두 요청이 서로 기다릴 필요가 없는데도 순차 실행되도록 코드가 작성되어서"입니다. 캐싱 라이브러리를 붙여도 최초 방문(캐시가 비어 있는 상태)에서는 여전히 순차 실행되어 느립니다.
  • 실행 순서(워터폴)와 캐싱(반복 방문 시 재사용)은 서로 다른 문제인데, 이 둘을 혼동하고 있습니다.
WEAK ANSWER — 또 다른 패턴

"모든 fetch를 Promise.all로 감싸면 항상 빨라집니다."

이 답변이 약한 이유

  • B 요청이 A 요청의 응답값을 실제로 필요로 하는 진짜 의존 관계인 경우도 있다는 것을 놓치고 있습니다. 이런 경우 강제로 Promise.all에 넣으면 B가 필요한 값이 아직 존재하지 않아 실행 자체가 불가능하거나 잘못된 값으로 요청하게 됩니다.
  • "무조건 병렬화"는 판단이 아니라 규칙 암기이며, 면접관은 의존 관계를 먼저 구분하는 사고 과정을 보고 싶어합니다.
STRONG ANSWER 요소 체크리스트
  • 병렬화를 제안하기 전에 두 요청의 의존 관계(진짜 vs 우연)를 먼저 확인한다
  • 같은 함수 안의 워터폴과 컴포넌트 트리 계층 간 워터폴을 구분해서 각각 다른 해법(Promise.all vs preload)을 제시한다
  • preload 패턴이 request memoization을 전제로 한다는 것을 알고, 그 전제가 깨졌을 때의 부작용(중복 요청)까지 언급한다
  • 워터폴 문제와 캐싱(반복 방문 시 재사용) 문제를 서로 다른 계층의 문제로 구분한다

15. Senior-level Follow-up Q&A

Q: 클라이언트 컴포넌트에서도 여러 useEffect fetch가 워터폴을 만들 수 있나요? 서버 컴포넌트만의 문제인가요?

모범 답변: 아닙니다. 클라이언트에서도 부모 컴포넌트가 fetch를 끝내야 마운트되는 자식 컴포넌트가 자기 fetch를 시작하는 구조라면 동일한 워터폴이 생깁니다. 다만 React Query 같은 클라이언트 서버 상태 도구는 useQueries(여러 쿼리를 배열로 병렬 실행)나 부모-자식 관계 없이 형제 컴포넌트로 배치하는 방식으로 병렬화를 돕습니다. 원칙은 서버든 클라이언트든 동일합니다 — "이 fetch가 정말 다른 fetch의 결과를 필요로 하는가"를 먼저 확인하고, 아니라면 트리 구조가 순차 실행을 강제하지 않도록 배치를 조정합니다.

Q: preload 패턴에서 void getUser(userId)처럼 호출한 Promise가 만약 실패(reject)하면 어떻게 되나요? 에러가 조용히 사라지지 않나요?

모범 답변: 좋은 지적입니다. cache()로 감싼 함수가 반환한 Promise가 reject되면, 나중에 실제로 await하는 지점(자식 컴포넌트)에서 그 reject된 결과가 그대로 다시 던져집니다 — request memoization은 성공한 결과뿐 아니라 실패한 Promise 자체도 캐싱하기 때문입니다. 따라서 에러가 사라지는 것이 아니라, 실제로 값을 사용하려는 지점에서 정상적으로 다시 catch되거나 Error Boundary로 전파됩니다. 다만 preload 호출 시점에서 발생하는 unhandled rejection 경고를 피하려면, 대체로 프레임워크가 이 처리를 내부적으로 해주지만, 직접 preload 유틸을 구현할 때는 이 부분을 검증해야 합니다.

Q: 세 개 이상의 fetch 중 일부는 의존 관계가 있고 일부는 없다면 어떻게 구성해야 하나요?

모범 답변: 의존 관계 그래프를 단계(stage)로 나눠서 생각합니다. 1단계에서 서로 독립적인 요청들을 Promise.all로 동시에 시작하고, 그 결과 중 일부가 2단계 요청의 파라미터로 쓰인다면 2단계는 1단계 완료 후에 시작합니다. 예를 들어 사용자 정보와 설정값은 서로 독립적이라 1단계에서 병렬 조회하고, 사용자의 주문 목록에서 각 주문의 배송 상세를 조회하는 것은 주문 목록이 나온 뒤(2단계)에 그 주문 ID들로 다시 병렬 조회(Promise.all(orders.map(o => getShipment(o.id))))하는 식입니다. 중요한 것은 "전부 한꺼번에" 또는 "전부 순서대로"라는 이분법이 아니라, 의존 관계 그래프를 단계별로 쪼개 각 단계 안에서만 병렬화하는 것입니다.

Q: request memoization은 여러 사용자의 요청 사이에서도 캐시를 공유하나요? 보안 문제는 없나요?

모범 답변: 아닙니다. Request memoization은 하나의 서버 요청(하나의 렌더링 생명주기)에 한정됩니다. 사용자 A의 요청을 처리하는 동안 만들어진 캐시는 그 요청이 끝나면 사라지고, 사용자 B의 요청과는 절대 공유되지 않습니다. 이 때문에 인증된 사용자별 데이터(예: 로그인한 사용자의 개인정보)를 fetch해도 다른 사용자에게 유출될 위험이 없습니다. 이 점이 여러 요청/사용자에 걸쳐 공유될 수 있는 Data Cache와 근본적으로 다른 부분이며, Data Cache에 개인화된 데이터를 캐싱할 때는 반드시 사용자별로 캐시 키를 분리하거나 no-store로 캐싱을 꺼야 하는 이유이기도 합니다.

Q: 병렬 fetch가 항상 좋다면, 왜 실무에서 순차 fetch를 쓰는 경우가 여전히 많은가요?

모범 답변: 두 가지 이유가 있습니다. 첫째, 앞서 말한 진짜 의존 관계 — B의 파라미터가 A의 응답에서만 나오는 경우는 병렬화가 불가능합니다. 둘째, 병렬로 여러 요청을 한꺼번에 쏘면 백엔드나 서드파티 API의 순간 부하(동시 커넥션 수, rate limit)가 늘어날 수 있어, 상황에 따라 의도적으로 요청을 분산시키는 선택을 하기도 합니다. 다만 이런 의도적 선택과, 단순히 코드를 순서대로 적어서 우연히 생긴 워터폴은 구분해야 합니다 — 전자는 트레이드오프를 인지한 설계이고, 후자는 놓친 최적화 기회입니다.

16. 흔한 오해

오해 1

"await를 여러 번 쓰면 무조건 워터폴이다"

진짜 문제는 await 개수가 아니라, 그 요청들 사이에 실제 의존 관계가 없는데도 순차 실행되도록 배치했는가입니다. 진짜 의존 관계가 있는 순차 await는 최적화 대상이 아니라 올바른 코드입니다.

오해 2

"React Query/SWR을 쓰면 워터폴 문제가 자동으로 사라진다"

서버 상태 관리 라이브러리는 캐싱, 재검증, 중복 제거를 도와주지만, 컴포넌트를 어떤 순서로 배치해서 어떤 요청을 트리거하는지는 여전히 개발자의 설계입니다. 부모-자식 구조로 인한 워터폴은 라이브러리 도입만으로 해결되지 않습니다.

오해 3

"request memoization은 Next.js/React가 캐시 서버를 따로 두고 처리하는 것이다"

request memoization은 별도 저장소가 아니라, 하나의 렌더링이 진행되는 동안 자바스크립트 실행 컨텍스트 안에서 Promise 객체 자체를 재사용하는 메커니즘입니다. 렌더링이 끝나면 이 캐시도 함께 사라집니다.

오해 4

"preload 함수는 실제로 데이터를 반환해주는 함수다"

preload 함수는 결과를 반환하거나 기다리지 않고, 요청을 "미리 시작만" 시켜두는 부수효과용 함수입니다. 실제 데이터가 필요한 곳에서는 원래의 memoize된 함수를 다시 호출(await)해서 값을 받아야 합니다.

오해 5

"모든 데이터 페칭을 서버에서 하면 워터폴 문제가 아예 사라진다"

서버로 옮겨도 서로 다른 컴포넌트 계층에 걸친 fetch 순서 문제(preload가 필요한 상황)는 그대로 남습니다. 서버 fetch는 네트워크 왕복 비용과 보안 문제를 줄여주지만, 실행 순서 설계 자체를 대신 해주지는 않습니다.
