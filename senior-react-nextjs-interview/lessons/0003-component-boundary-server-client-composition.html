1. 핵심 면접 질문

Primary Question

"Client Component 안에서 Server Component를 직접 import해서 사용할 수 없는 이유와, 그 대안이 되는 composition pattern을 코드로 설명해 주세요."

Variant 1

"팀원이 페이지 최상단 레이아웃 파일에 "use client"를 걸어서 문제를 해결했다고 합니다. 이 방식의 문제점은 무엇이고, 어떻게 다시 설계하시겠어요?"

Variant 2

"DB 조회 로직이 들어있는 서버 전용 라이브러리를 import한 컴포넌트를 실수로 Client Component 트리 안에 두었더니, 번들에 DB 자격 증명 관련 코드가 포함됐습니다. 왜 이런 일이 일어났고, 어떻게 구조를 바꿔야 하나요?"

2. 왜 면접에서 묻는가

이 질문은 ""use client"를 파일 맨 위에 쓰면 된다는 것을 아는가"가 아니라 "그 한 줄이 모듈 그래프 전체에 어떤 컴파일 타임 결정을 강제하는지 이해하고 있는가"를 테스트합니다. App Router 프로젝트에서 실제로 가장 흔하게 발생하는 사고가 "빌드는 되고 화면도 잘 나오는데, 번들 분석기를 열어보면 서버 전용 코드나 무거운 라이브러리가 클라이언트 번들에 들어가 있다"는 유형입니다. 이 사고는 컴파일 에러 없이 조용히 발생하기 때문에, 경계를 어디에 그을지에 대한 정확한 모델이 없으면 프로덕션에서야 발견됩니다.

면접관이 이 질문으로 평가하는 것:

  • "use client"의 정확한 의미 — 이것이 "이 컴포넌트만 클라이언트에서 실행하라"는 지시가 아니라 "이 모듈과, 이 모듈이 import하는 모든 것을 클라이언트 번들 경계의 시작점으로 취급하라"는 모듈 그래프 분할 지시라는 것을 아는가.
  • 전이(propagation) 문제에 대한 인식 — 경계를 트리 상단(레이아웃, 페이지)에 잘못 그으면 그 하위 전체가 의도치 않게 클라이언트 번들로 편입된다는 것을 이해하는가.
  • composition pattern을 실제로 아는가 — "Client Component는 Server Component를 import할 수 없지만, children이나 다른 prop으로 이미 렌더링된 Server Component 결과를 받을 수는 있다"는 패턴을 코드로 설명할 수 있는가.
  • 경계 설계가 아키텍처 판단이라는 인식 — "이 인터랙션(클릭, 입력, 상태)이 정확히 어디서 필요한가"를 먼저 찾고, 그 인터랙티브한 리프(leaf) 노드에만 "use client"를 붙이는 최소 경계 설계 습관이 있는가.
  • 보안·비용 감각 — 서버 전용 자원(DB 클라이언트, API 시크릿, 무거운 서버 라이브러리)이 클라이언트 번들에 노출되는 것이 단순 성능 문제가 아니라 보안 사고로 이어질 수 있다는 것을 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "에러가 나서 "use client"를 위에 붙였더니 해결됐어요." 이 답변은 증상을 없앴을 뿐, 그 결정이 트리 전체의 번들 경계를 어떻게 바꿨는지 인식하지 못한 상태입니다. 시니어는 에러를 만나면 "왜 이 경계에서 막혔는가"를 먼저 진단하고, "이 파일 전체를 클라이언트로 보낼 것인가, 아니면 인터랙티브한 부분만 분리할 것인가"를 의식적으로 선택합니다.

3. 전제 개념 — 쉬운 한국어로

Server Component가 실제로 클라이언트로 보내는 것

Day 1에서 "Server Component는 코드가 아니라 결과를 클라이언트로 보낸다"고 배웠습니다. 조금 더 정확히 말하면, Next.js는 서버에서 컴포넌트 트리를 렌더링한 결과를 RSC(React Server Component) 페이로드라는 직렬화된 트리 구조로 만들어 브라우저에 전달합니다. 이 페이로드에는 HTML로 변환될 내용과, 그 사이사이에 "여기부터는 Client Component이니 이 JS 모듈을 불러와서 실행하라"는 참조(reference)가 섞여 있습니다. 즉 Server Component는 애초에 클라이언트에서 실행될 코드 자체가 존재하지 않습니다 — 결과만 있을 뿐입니다.

"use client"가 실제로 하는 일 — 코드 위치 지시가 아니라 모듈 그래프 경계 지시

초보적인 설명은 ""use client"를 붙이면 그 컴포넌트가 브라우저에서 실행된다"입니다. 틀린 말은 아니지만, 이 설명만으로는 "왜 Client Component가 Server Component를 import하면 안 되는가"를 설명할 수 없습니다. 정확한 모델은 이렇습니다.

  • "use client"는 파일(모듈) 맨 위에 선언하는 지시어이며, "이 모듈은 서버 모듈 그래프와 클라이언트 모듈 그래프의 경계선이다"라고 번들러(webpack/Turbopack)에게 알려주는 컴파일 타임 표시입니다.
  • 이 표시가 붙은 모듈은 클라이언트 번들에 실제 코드로 포함되고, 서버 렌더링 시에는 그 모듈이 "클라이언트에서 이 모듈을 불러오라"는 참조로 대체됩니다.
  • 중요한 점: 이 모듈이 import하는 다른 모듈들도 전부 같은 클라이언트 번들에 포함됩니다. "use client" 파일 안에서 별도 표시 없이 import한 헬퍼 함수, 유틸리티, 다른 컴포넌트는 모두 자동으로 클라이언트 번들의 일부가 됩니다.

왜 Client Component가 Server Component를 import할 수 없는가

이제 핵심 질문에 답할 수 있습니다. Client Component 파일 안에서 import ServerComp from './ServerComp'처럼 Server Component를 직접 import하면, 번들러는 "이 모듈(ServerComp)도 클라이언트 번들에 포함해야 한다"고 판단합니다. 그런데 ServerComp는 애초에 서버 전용 API(파일 시스템 접근, DB 쿼리, 서버 전용 환경 변수 등)를 쓰고 있을 수 있고, 무엇보다 서버에서 한 번만 렌더링되고 결과만 전달되는 것을 전제로 설계된 컴포넌트입니다. 이를 클라이언트에서 다시 "코드로" 실행하면 서버 전용 자원에 접근할 수 없어 에러가 나거나, 최악의 경우 서버 전용 코드(그리고 그 안에 하드코딩되었을 수 있는 민감한 로직)가 그대로 클라이언트 번들에 노출됩니다. React/Next.js는 이 위험을 아예 컴파일 타임에 막기 위해, Client Component 파일 안에서 Server Component를 import하면 실제로는 그 Server Component가 "use client" 규칙에 따라 강제로 Client Component로 취급되어 렌더링됩니다 — 즉 서버에서만 쓰려던 로직이 조용히 클라이언트에서 실행되거나, 서버 전용 API를 쓰는 코드라면 그 시점에 런타임 에러가 납니다.

4. "use client" 전이(propagation) 문제

"use client"의 가장 위험한 특성은 위쪽으로는 전이되지 않지만, 아래쪽(import하는 방향)으로는 전이된다는 것입니다. 이 방향성을 정확히 알아야 경계를 어디에 그어야 할지 판단할 수 있습니다.

상황 결과 이유
레이아웃 파일(layout.tsx)에 "use client"를 붙임 그 레이아웃이 import하는 모든 하위 컴포넌트가 클라이언트 번들로 편입됨(명시적으로 Server Component를 children으로 받는 부분 제외) 전이는 import 그래프를 따라 아래로 퍼짐 — 상위에서 건 경계가 하위 전체를 끌고 감
Client Component 안에서 순수 함수(날짜 포맷팅 등)를 import 그 유틸 함수도 클라이언트 번들에 포함됨 서버 전용 API를 안 쓰는 코드라도, "use client" 파일이 import하면 무조건 같은 번들로 편입됨
Server Component가 Client Component를 import해서 자식으로 렌더링 Server Component 자체는 여전히 서버에서 실행됨. 그 Client Component만 별도 번들 청크로 분리되어 클라이언트에 전달됨 전이는 아래로만 퍼지고 위로는 퍼지지 않음 — "use client" 컴포넌트를 사용하는 부모까지 클라이언트가 되지는 않음

실무에서 실제로 벌어지는 사고

가장 흔한 실수는 "이 페이지에 인터랙티브한 버튼 하나가 필요하다"는 이유로 페이지 컴포넌트 전체 또는 레이아웃 전체에 "use client"를 붙이는 것입니다. 이렇게 하면 그 페이지가 서버에서 fetch하던 데이터 조회 로직까지 전부 클라이언트로 옮겨가거나(서버 전용 API를 쓰면 빌드/런타임 에러), 무거운 날짜 처리 라이브러리, 마크다운 렌더러 같은 서버에서만 필요했던 무거운 의존성이 그대로 클라이언트 번들에 포함되어 초기 로딩 JS 크기가 급격히 커집니다. 이것이 시니어 인터뷰에서 "왜 번들이 갑자기 커졌는지 진단해보라"는 형태로 자주 등장하는 이유입니다.

올바른 원칙 — "리프(leaf)에 경계를 그어라"

정답은 인터랙티브한 부분(state, event handler, 브라우저 전용 API를 쓰는 부분)만 트리에서 가장 아래쪽의 작은 컴포넌트로 분리하고, 그 리프 컴포넌트에만 "use client"를 붙이는 것입니다. 페이지 전체나 레이아웃 전체가 아니라, "좋아요 버튼", "드롭다운 메뉴", "실시간 검색창" 같은 작은 단위에 경계를 그으면 나머지 트리(데이터 조회, 정적 콘텐츠 렌더링)는 계속 서버에 남아 있고 클라이언트 번들에는 포함되지 않습니다.

5. Composition Pattern — children으로 경계 넘기기

그런데 실무에서는 "Client Component 레이아웃(예: 토글 가능한 사이드바, 테마 provider) 안에, 서버에서 데이터를 조회해 렌더링한 콘텐츠를 넣고 싶다"는 요구가 자주 생깁니다. import로는 불가능하지만, 이미 렌더링된 Server Component의 결과를 children 같은 prop으로 전달하는 것은 가능합니다. 이것이 composition pattern입니다.

왜 이게 가능한가 — "import"와 "결과를 값으로 전달"의 차이

Client Component가 금지된 것은 "Server Component 모듈을 import해서 그 코드를 클라이언트 번들에 포함시키려는 시도"입니다. 반면 부모(Server Component)가 자식(Server Component)을 먼저 렌더링한 결과(React element)를 Client Component에게 children이나 일반 prop으로 넘겨주는 것은 전혀 다른 일입니다. 이 경우 Client Component는 그 Server Component를 import하지도, 그 코드를 알지도 못합니다. 단지 "이미 만들어진 결과물(트리 조각)을 어디에 배치할지"만 결정할 뿐입니다. Server Component는 여전히 서버에서만 렌더링되고, Client Component는 그 결과를 감싸는 "틀(shell)" 역할만 합니다.

// ClientShell.tsx
"use client";

import { useState } from "react";

// children으로 받은 것은 "이미 서버에서 렌더링된 결과"일 뿐,
// 이 파일이 그 내부 컴포넌트를 import하지 않는다.
export function ClientShell({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(true);

  return (
    <div>
      <button onClick={() => setOpen(!open)}>토글</button>
      {open && <div className="panel">{children}</div>}
    </div>
  );
}

// page.tsx (Server Component)
import { ClientShell } from "./ClientShell";
import { getDashboardData } from "./data"; // 서버 전용 DB 조회

export default async function Page() {
  const data = await getDashboardData(); // 서버에서만 실행

  return (
    <ClientShell>
      {/* 이 부분은 여전히 서버 컴포넌트다. ClientShell은 이 내용을
          "children"이라는 값으로 받아 배치할 뿐, import하지 않는다. */}
      <DashboardContent data={data} />
    </ClientShell>
  );
}

이 패턴이 실무에서 중요한 이유

이 패턴 덕분에 "토글 가능한 UI 틀은 클라이언트에서, 그 안에 들어갈 실제 데이터 조회와 콘텐츠 렌더링은 서버에서"라는 이상적인 경계 분리가 가능합니다. ClientShell은 여전히 인터랙티브하지만, getDashboardData 같은 서버 전용 조회 로직이나 DashboardContent가 쓰는 무거운 렌더링 로직은 클라이언트 번들에 전혀 포함되지 않습니다. Context Provider, 테마 전환, 모달/드로어처럼 "인터랙션은 필요하지만 내용물은 서버가 만들어야 하는" 거의 모든 레이아웃성 컴포넌트에 이 패턴이 적용됩니다.

6. Prop 직렬화 경계 — 무엇을 넘길 수 있고 무엇을 못 넘기는가

Server Component에서 Client Component로 prop을 넘길 때는 RSC 페이로드로 직렬화 가능한 값만 전달할 수 있습니다. 이 제약을 모르면 "왜 이 함수를 prop으로 넘기면 에러가 나는가"를 설명하지 못합니다.

전달 가능 전달 불가능(일반 함수)
문자열, 숫자, boolean, 배열, 일반 객체(JSON 직렬화 가능) 클로저를 포함한 일반 함수(예: 서버에서 정의한 콜백)
Date, Map, Set 등 React가 지원하는 확장 직렬화 타입 클래스 인스턴스(직렬화 방법이 정의되지 않은 경우), Symbol
다른 React element(children으로 전달되는 렌더링 결과) DB 커넥션, 파일 핸들 같은 서버 전용 객체
Server Action("use server"로 표시된 함수) — 특별히 직렬화 가능한 참조로 전달됨 —

여기서 Server Action은 예외라는 점이 중요합니다. 일반 함수는 직렬화할 수 없지만, "use server"로 표시된 함수는 React가 "클라이언트가 호출하면 서버로 요청을 보내는 특수한 참조"로 변환해 전달할 수 있습니다. 이 덕분에 Client Component의 폼에서 서버 로직을 직접 호출하는 패턴(예: <form action={serverAction}>)이 가능해집니다. 이 내용은 Data Fetching과 Server Actions를 본격적으로 다루는 이후 Day에서 더 깊게 다룹니다.

7. 경계 설계 판단표

이 컴포넌트가... 권장 경계 이유
DB/파일 시스템 접근, 서버 전용 시크릿 사용 Server Component 유지 클라이언트로 넘어가면 접근 자체가 불가능하거나 시크릿이 노출됨
useState, onClick 등 이벤트 핸들러, 브라우저 전용 API(window, localStorage)가 필요 가장 작은 리프 단위로 분리해 "use client" 인터랙션이 필요한 최소 범위에만 경계를 그어야 전이 범위를 줄일 수 있음
인터랙티브한 "틀"이 필요하지만 내부 콘텐츠는 서버가 조회해야 함(사이드바, 모달, 테마 provider) Client Component + children composition 틀만 클라이언트로, 내용물은 서버 렌더링 결과를 값으로 전달받아 유지
서드파티 UI 라이브러리가 내부적으로 useState나 Context를 씀(하지만 라이브러리에 "use client" 표시가 없음) 얇은 래퍼 컴포넌트를 만들어 그 래퍼에만 "use client" 부여 서드파티 코드 자체를 수정할 수 없으므로, 경계를 명시하는 얇은 wrapper로 감싸야 함
순수 계산/포맷팅 유틸 함수(부수효과 없음) 서버/클라이언트 구분 없이 공용 모듈로 두되, 호출하는 쪽의 경계를 따라감 이런 유틸은 어느 쪽에서 import되든 문제없지만, Client Component가 import하면 결국 번들에 포함됨을 인지해야 함

8. 답변 사고 순서

  1. "use client"의 정의를 정확히 말합니다. "이건 컴포넌트 하나를 클라이언트에서 실행하라는 지시가 아니라, 이 모듈을 시작점으로 하는 모듈 그래프 전체를 클라이언트 번들 경계로 표시하는 컴파일 타임 지시다."
  2. 왜 Client Component가 Server Component를 import할 수 없는지 설명합니다. "import하면 그 모듈도 클라이언트 번들에 포함되어야 하는데, Server Component는 서버 전용 자원에 의존하거나 클라이언트에서 재실행되도록 설계되지 않았기 때문이다."
  3. 대안(composition)을 제시합니다. "대신 부모인 Server Component가 자식을 먼저 렌더링한 결과를 children이나 prop으로 Client Component에 넘기면, Client Component는 그 컴포넌트를 import하지 않고도 결과물을 배치할 수 있다."
  4. 이 패턴이 실무에서 왜 중요한지 연결합니다. "이 패턴 덕분에 인터랙티브한 틀(클라이언트)과 데이터 조회/무거운 렌더링(서버)을 분리해서, 서버 전용 로직이나 무거운 의존성이 클라이언트 번들에 섞이지 않게 할 수 있다."
  5. 잘못된 경계 설정의 대가를 짚습니다. "경계를 트리 상단에 두면 전이 때문에 하위 전체가 클라이언트로 편입되어 번들이 커지고, 서버 전용 자원 접근이 막히거나 노출될 위험이 생긴다."

9. 핵심 답변 예시

면접관: "팀원이 페이지 최상단 레이아웃 파일에 "use client"를 걸어서 문제를 해결했다고 합니다. 이 방식의 문제점은 무엇이고, 어떻게 다시 설계하시겠어요?"

STRONG ANSWER

레이아웃 파일에 "use client"를 붙이면, 그 파일이 import하는 모든 하위 컴포넌트가 같은 클라이언트 번들 경계 안으로 편입됩니다. "use client"는 컴포넌트 하나를 클라이언트에서 실행하라는 지시가 아니라, 그 모듈을 시작점으로 하는 모듈 그래프 전체를 클라이언트 번들로 표시하는 컴파일 타임 지시이기 때문입니다. 레이아웃은 보통 페이지 콘텐츠 대부분을 감싸는 최상위 컴포넌트이므로, 여기에 경계를 걸면 원래 서버에서 데이터를 조회하고 렌더링하던 하위 트리 전체가 클라이언트 번들에 포함되어 초기 로딩 JS 크기가 크게 늘어납니다. 또한 서버 전용 API(DB 조회, 파일 시스템 접근 등)를 쓰는 하위 컴포넌트가 있다면 빌드 또는 런타임 에러로 이어질 수 있습니다.


먼저 팀원이 왜 레이아웃 전체에 "use client"를 걸었는지 원인을 확인하겠습니다. 대부분은 "레이아웃 안에 토글 가능한 사이드바나 테마 전환 버튼처럼 인터랙션이 필요한 작은 부분이 있는데, 그 부분이 상태를 쓰기 때문에 상위 파일 전체에 경계를 걸어버린 경우"입니다. 이 경우 인터랙션이 실제로 필요한 부분(사이드바 토글 버튼, 테마 스위처)만 별도의 작은 컴포넌트로 분리해 그 컴포넌트에만 "use client"를 붙이고, 레이아웃 자체는 다시 Server Component로 되돌립니다.


만약 "토글 가능한 틀 안에 서버에서 조회한 콘텐츠를 넣어야 한다"는 요구가 있다면, 그 틀 컴포넌트(ClientShell)를 Client Component로 두되 children prop을 받도록 만들고, 레이아웃(Server Component)에서 데이터를 조회해 렌더링한 콘텐츠를 children으로 전달합니다. 이렇게 하면 ClientShell은 내부 콘텐츠 컴포넌트를 import하지 않으므로, 데이터 조회 로직과 무거운 렌더링 로직은 여전히 서버에만 남고 클라이언트 번들에는 포함되지 않습니다.

이 답변이 좋은 이유:

  • "use client"를 컴포넌트 단위가 아니라 모듈 그래프 경계로 정확히 정의함
  • 레이아웃 전체에 경계를 거는 것의 구체적인 대가(번들 크기, 서버 API 접근 불가)를 명시함
  • 문제의 근본 원인(왜 그렇게 했을지)을 먼저 진단하는 접근을 보여줌
  • 단순히 "나누세요"에 그치지 않고 composition pattern으로 구체적인 재설계안을 제시함

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"레이아웃에 "use client"를 걸면 안 되는 건 알고 있어요. 그냥 필요한 컴포넌트에만 "use client"를 따로 붙이면 됩니다."

이 답변이 약한 이유

  • "안 되는 건 안다"는 규칙만 말할 뿐, 왜 안 되는지(전이, 모듈 그래프 경계)를 설명하지 못합니다.
  • "토글 틀 안에 서버 콘텐츠가 들어가야 하는 경우" 같은, 실무에서 실제로 발생하는 더 어려운 상황(composition이 필요한 경우)에 대한 대응이 없습니다.
  • 면접관이 "그럼 인터랙티브한 틀 안에 서버에서 조회한 콘텐츠를 넣어야 하면 어떻게 하나요?"라고 물으면 답이 막힐 가능성이 높습니다.
WEAK ANSWER — 또 다른 패턴

"Server Component는 서버에서 실행되고 Client Component는 클라이언트에서 실행되니까, 섞어 쓰려면 그냥 client 쪽에서 fetch를 다시 하면 됩니다."

이 답변이 약한 이유

  • 질문의 핵심(경계 설계, composition pattern)을 회피하고 "클라이언트에서 다시 조회하라"는 다른 해결책으로 우회합니다.
  • 이 접근은 서버에서 이미 조회 가능한 데이터를 클라이언트에서 중복 요청하게 만들어, 워터폴과 불필요한 네트워크 요청을 추가하는 더 나쁜 설계로 이어질 수 있습니다.
  • "왜 Client Component가 Server Component를 import할 수 없는가"라는 근본 메커니즘을 전혀 다루지 않았습니다.
STRONG ANSWER 요소 체크리스트
  • "use client"가 컴포넌트 단위가 아니라 모듈 그래프 경계 지시라는 것을 명시한다
  • 전이가 아래쪽(import 방향)으로만 퍼진다는 방향성을 정확히 설명한다
  • 잘못된 경계의 구체적 대가(번들 크기, 서버 API 접근 불가, 보안 노출 가능성)를 짚는다
  • children을 이용한 composition pattern을 코드 수준으로 설명한다
  • "인터랙션이 필요한 최소 리프 단위에만 경계를 그어야 한다"는 설계 원칙으로 마무리한다

11. Senior-level Follow-up Q&A

Q: Client Component가 Server Component를 children으로 받는 것과, 그냥 Server Component를 일반 prop(예: content)으로 받는 것에 차이가 있나요?

모범 답변: 메커니즘상으로는 동일합니다. children이든 다른 이름의 prop이든, "이미 렌더링된 React element를 값으로 전달한다"는 원리는 같습니다. children을 관용적으로 쓰는 이유는 JSX 문법상 <ClientShell>...</ClientShell> 형태로 자연스럽게 중첩해서 쓸 수 있고, "이 자리에 슬롯이 있다"는 의도가 코드만 봐도 명확하기 때문입니다. 슬롯이 여러 개 필요하면(예: 헤더 영역, 본문 영역을 따로 받아야 하면) header, body처럼 이름 있는 prop 여러 개로 나눠 받는 것도 흔한 패턴입니다.

Q: Server Component에서 Client Component로 함수를 이벤트 핸들러가 아니라 일반 콜백으로 넘기고 싶으면 어떻게 하나요?

모범 답변: 일반 함수(클로저를 포함한 임의의 서버 함수)는 직렬화할 수 없으므로 그대로 넘길 수 없습니다. 방법은 두 가지입니다. 첫째, 그 함수가 서버에서 어떤 부수효과(DB 저장, 재검증 등)를 수행해야 하는 로직이라면 "use server"로 표시한 Server Action으로 만들어 전달합니다 — React가 이를 "호출 가능한 참조"로 직렬화해주기 때문입니다. 둘째, 단순히 클라이언트 쪽에서 값을 조작하는 로직(포맷팅, 필터링 등)이라면 그 함수 자체를 Client Component 내부에 정의하거나 공용 유틸 모듈로 옮겨서 Client Component가 직접 import하게 만듭니다.

서드파티 라이브러리(예: 특정 UI 라이브러리)가 내부적으로 Context나 useState를 쓰는데, 그 라이브러리 파일 자체에는 "use client"가 없어서 에러가 납니다. 어떻게 해결하나요?

모범 답변: 라이브러리 소스 코드를 직접 수정할 수 없으므로, 그 라이브러리를 감싸는 아주 얇은 wrapper 파일을 직접 만들고 그 wrapper 파일 맨 위에만 "use client"를 붙입니다. 예를 들어 MyLibWrapper.tsx라는 파일에서 "use client"를 선언한 뒤 라이브러리 컴포넌트를 그대로 재export하거나 감싸서 사용하고, 애플리케이션 코드에서는 원본 라이브러리 대신 이 wrapper를 import합니다. 이렇게 하면 "이 라이브러리는 클라이언트 전용이다"라는 경계를 명시적으로 표시할 수 있고, 향후 라이브러리가 공식적으로 "use client"를 지원하면 wrapper를 제거하기도 쉽습니다.

Server Component와 Client Component 사이에 context(React Context)를 공유할 수 있나요?

모범 답변: Context Provider와 Consumer는 useContext라는 Hook에 의존하므로, Provider와 그 값을 읽는 Consumer 모두 Client Component여야 합니다. Server Component는 렌더링이 요청마다 서버에서 일회성으로 일어나고 Hook의 실행 모델 자체가 없기 때문에 Context에 직접 참여할 수 없습니다. 그래서 실무 패턴은 "Context Provider를 Client Component로 만들고, 그 Provider가 children으로 Server Component 트리를 감싸게" 하는 것입니다. Provider 자체는 클라이언트 상태(테마, 로케일 등)를 관리하지만, 그 안에 배치되는 콘텐츠는 여전히 서버에서 렌더링된 결과를 그대로 받을 수 있습니다.

경계를 "너무 잘게" 쪼개는 것도 문제가 될 수 있나요?

모범 답변: 네. 극단적으로 모든 작은 인터랙션마다 별도의 Client Component 파일을 만들면, 각 파일이 별도의 JS 청크로 분리되면서 청크 수가 과도하게 늘어나 네트워크 요청 오버헤드가 커질 수 있고, 코드 재사용성도 떨어질 수 있습니다. 실무에서는 "정말 인터랙티브한 최소 단위"를 기준으로 삼되, 같은 화면 영역에서 함께 동작하는 관련 인터랙션(예: 폼 안의 여러 입력 필드)은 하나의 Client Component로 묶는 것이 합리적입니다. 원칙은 "서버 전용 로직/데이터를 오염시키지 않는 선에서, 응집도 있는 인터랙션 단위로 나누는 것"입니다.

12. 흔한 오해

오해 1

"use client"는 그 컴포넌트 하나만 클라이언트에서 실행되게 만든다"

정확히는 그 모듈을 시작점으로 import 그래프 전체가 클라이언트 번들 경계 안으로 편입됩니다. "컴포넌트 하나"가 아니라 "모듈 그래프 전체"라는 점을 놓치면 전이 문제를 예측할 수 없습니다.

오해 2

"Server Component를 prop으로 넘기면 그 컴포넌트도 클라이언트에서 다시 렌더링된다"

children이나 prop으로 넘기는 것은 "이미 서버에서 렌더링이 끝난 결과(React element)"입니다. Client Component는 그 결과를 배치할 뿐, 해당 Server Component의 렌더링 함수를 다시 실행하지 않습니다. 이 차이를 모르면 "그럼 결국 서버 컴포넌트도 클라이언트에서 실행되는 거 아닌가요?"라는 혼동이 생깁니다.

오해 3

"안전하게 하려면 모든 컴포넌트에 use client를 붙이는 게 낫다"

정반대입니다. 기본값은 Server Component이고, 필요한 최소 범위에만 "use client"를 붙이는 것이 번들 크기와 서버 자원 접근성 양쪽에서 유리합니다. "안전"이라는 말과 달리, 무분별한 "use client"는 오히려 서버 전용 자원 노출과 번들 폭증이라는 실제 위험을 키웁니다.

오해 4

"use client 경계를 한 번 잘못 그으면 코드를 전부 다시 짜야 한다"

대부분의 경우 리팩터링은 "인터랙티브한 부분을 찾아 별도 파일로 추출하고 그 파일에만 "use client"를 옮기는" 국소적인 작업입니다. 전체 아키텍처를 다시 짜야 하는 경우는 드물며, 오히려 이 리팩터링 능력 자체(어디를 분리해야 하는지 빠르게 판단하는 능력)가 시니어에게 요구되는 실무 스킬입니다.

오해 5

"use client 파일 안에서도 서버 환경 변수(process.env의 시크릿)를 그냥 읽으면 안전하다"

Client Component 코드는 번들러에 의해 실제로 브라우저에 전달되는 JS 파일이 됩니다. 그 안에서 서버 전용 시크릿을 참조하면(설령 그 값이 빌드 시점에 인라인되더라도) 브라우저로 전송되는 코드에 그대로 노출됩니다. 시크릿은 반드시 Server Component나 Server Action, 서버 전용 모듈 안에서만 다뤄야 합니다.
