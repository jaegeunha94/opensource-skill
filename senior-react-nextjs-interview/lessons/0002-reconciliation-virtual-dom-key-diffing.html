1. 핵심 면접 질문

Primary Question

"리스트를 렌더링할 때 key로 배열 인덱스를 쓰면 안 되는 이유를 reconciliation 관점에서 설명해 주세요."

Variant 1

"React는 이전 트리와 새 트리를 비교할 때 정확히 무엇을 비교하나요? 왜 O(n³) 알고리즘 대신 O(n)에 가까운 방식을 쓸 수 있나요?"

Variant 2

"할 일 목록에서 항목을 맨 앞에 추가했더니, 입력 중이던 텍스트 필드의 값이 엉뚱한 항목으로 옮겨갔습니다. 왜 이런 일이 일어나고, 어떻게 고치시겠어요?"

2. 왜 면접에서 묻는가

이 질문은 "key를 안 쓰면 콘솔에 경고가 뜬다는 것을 아는가"가 아니라 "React가 이전 렌더 결과와 새 렌더 결과를 어떤 기준으로 매칭하는지 정확한 모델을 갖고 있는가"를 테스트합니다. key={item.id}를 습관처럼 붙이는 사람은 많지만, "왜 index를 쓰면 상태가 엉키는가"를 컴포넌트 인스턴스 재사용/재생성 관점에서 설명할 수 있는 사람은 훨씬 적습니다.

면접관이 이 질문으로 평가하는 것:

  • reconciliation의 정확한 동작 모델 — React가 "같은 위치의 같은 타입 엘리먼트는 같은 컴포넌트 인스턴스로 취급하고 state를 유지한다"는 기본 규칙을 알고 있는가.
  • key가 하는 일에 대한 정확한 이해 — key는 "화면에 보이는 값"이 아니라 "리스트 항목의 정체성(identity)을 React에게 알려주는 힌트"라는 것을 아는가.
  • 버그를 재현하고 설명하는 능력 — "state가 엉킨다"는 증상을 보고, 그 원인을 컴포넌트 인스턴스 재사용 규칙으로 역추적할 수 있는가.
  • 알고리즘 복잡도에 대한 감각 — 트리 diffing이 이론적으로 O(n³)이 될 수 있는 문제를, React가 어떤 휴리스틱(같은 레벨 비교, 타입 비교, key 비교)으로 O(n)에 가깝게 낮췄는지 설명할 수 있는가.
  • 실무 적용 판단력 — 언제 index를 key로 써도 안전한지(정적이고 재정렬·삽입·삭제가 없는 리스트), 언제 반드시 안정적 id가 필요한지 구분할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "key가 없으면 콘솔에 경고가 뜨니까 index라도 넣어줍니다." 이 답변은 "경고를 없애는 법"만 알 뿐, key가 실제로 무엇을 위한 것인지, index를 쓰면 왜 위험한지를 설명하지 못합니다. 시니어는 "이 리스트가 재정렬/삽입/삭제될 가능성이 있는가"를 먼저 판단하고, 그에 따라 key 전략을 다르게 선택합니다.

3. 전제 개념 — 쉬운 한국어로

Reconciliation이 정확히 무엇을 하는 작업인가

Day 1에서 "React는 새 트리를 계산한 뒤 이전 트리와 비교해서 달라진 부분만 실제 DOM에 반영한다"고 했습니다. 이 "비교"의 정식 이름이 reconciliation(재조정)입니다. 조금 더 정확히 말하면, reconciliation은 단순히 "무엇이 바뀌었는지 찾는" 것을 넘어 "이전 트리의 어떤 컴포넌트 인스턴스를 새 트리의 어떤 자리에 재사용할 것인가"를 결정하는 작업입니다. 이 결정에 따라 해당 컴포넌트의 state, DOM 노드(입력 중이던 값, 스크롤 위치, CSS transition 진행 상태 등)가 유지되거나 초기화됩니다.

"컴포넌트 인스턴스를 재사용한다"는 게 무슨 뜻인가

이사 비유를 들면 이렇습니다. 방(컴포넌트가 그리는 DOM 노드와 그 안의 state)이 있는데, 이사 갈 때 "이 방에 있던 가구(state)를 새 집의 같은 방에 그대로 옮길 것인가, 아니면 방을 통째로 부수고 새로 지을 것인가"를 결정해야 합니다. React가 "이전 위치의 컴포넌트와 새 위치의 컴포넌트가 같은 것이다"라고 판단하면 가구(state, DOM 노드, 생명주기)를 그대로 유지한 채 내용만 업데이트합니다. 반대로 "다른 것"이라고 판단하면 기존 컴포넌트를 언마운트(가구 폐기, useEffect cleanup 실행)하고 완전히 새로운 컴포넌트를 마운트합니다. 이 판단 기준이 이번 레슨의 핵심입니다.

왜 "전체 트리를 완전 탐색"하지 않는가

두 트리가 완전히 다른 구조일 수 있다는 가정 하에 최적의 변경 사항을 계산하는 일반적인 트리 비교 알고리즘은 이론적으로 O(n³)의 시간 복잡도를 가집니다. 노드가 1,000개만 되어도 이 방식은 실용적이지 않습니다. React는 UI 트리에 대해 실제로 관찰되는 두 가지 경험적 특성(heuristic)을 가정하고 이 문제를 O(n)에 가깝게 낮춥니다.

  • 가정 1 — 타입이 다른 엘리먼트는 다른 트리를 만든다. 예를 들어 <div>가 <span>으로 바뀌면, React는 내부를 비교하려 하지 않고 이전 트리 전체를 버리고 새로 만듭니다.
  • 가정 2 — 개발자는 key로 여러 렌더링에 걸쳐 안정적인 항목을 표시해줄 수 있다. 리스트의 각 항목에 안정적인 key를 붙이면, React는 "같은 key를 가진 항목은 순서가 바뀌어도 같은 항목이다"라고 신뢰하고 그 항목의 인스턴스를 재사용합니다.

즉 diffing은 "완벽한 최소 변경"을 계산하는 것이 아니라, 대부분의 UI 변경 패턴에서 충분히 빠르고 충분히 정확한 결과를 내는 실용적인 휴리스틱입니다. 이 사실을 알아야 "왜 완벽하지 않은 결과가 나올 수도 있는가"(예: 리스트 중간에 항목을 삽입했는데 key가 없으면 이후 모든 항목이 다시 렌더링되는 것)를 설명할 수 있습니다.

4. Diffing 알고리즘 — React가 실제로 비교하는 것

React는 같은 부모 아래, 같은 레벨(depth)에 있는 자식들끼리만 비교합니다. 다른 레벨로 이동한 노드는 아예 "이동"으로 인식하지 않고 "이전 것 삭제 + 새 것 생성"으로 처리합니다. 같은 레벨 안에서는 아래 두 경우로 나뉩니다.

비교 대상 React의 판단 결과
엘리먼트 타입이 다름
(예: <button> → <a>, 혹은 ComponentA → ComponentB)		 다른 트리로 간주 — 내부를 비교하지 않음 기존 DOM 노드/컴포넌트 인스턴스를 언마운트(state 소멸, cleanup 실행)하고 완전히 새로 마운트
엘리먼트 타입이 같음
(예: <div className="a"> → <div className="b">)		 같은 노드로 간주 — 속성만 비교 기존 DOM 노드를 유지한 채 변경된 속성(className 등)만 갱신. state와 하위 트리는 계속 재조정
같은 타입의 컴포넌트가 자식 리스트를 렌더링 기본적으로 자식들을 순서대로 비교. key가 있으면 순서 대신 key로 매칭 key가 없으면 "같은 위치(index)의 자식"을 같은 것으로 취급 → 리스트가 재정렬되면 잘못된 매칭 발생

이 규칙이 실무에서 중요한 이유

"엘리먼트 타입이 다르면 그 하위 트리 전체가 통째로 재생성된다"는 규칙 때문에, 조건부 렌더링을 할 때 {condition ? <ComponentA /> : <ComponentB />}처럼 타입 자체를 바꾸면 내부 state가 항상 초기화됩니다. 반대로 "state를 유지하고 싶다"면 같은 위치에 같은 타입의 컴포넌트를 유지한 채 props만 바꾸는 방식으로 설계해야 합니다. 이 판단을 못 하면 "왜 탭을 전환했더니 입력하던 값이 사라지나요?" 같은 버그를 만들게 됩니다.

5. key의 역할 — 왜 index를 쓰면 안 되는가

key는 화면에 보이는 값이 아니라, "여러 번의 렌더링에 걸쳐 이 항목이 동일한 항목임을 React에게 알려주는 정체성(identity) 힌트"입니다. key가 같으면 React는 순서가 바뀌어도 "같은 항목"으로 인식해 컴포넌트 인스턴스(그리고 그 state, DOM 노드)를 재사용합니다. key가 다르면(혹은 없어서 index로 대체되면) React는 다른 항목으로 인식할 수 있습니다.

배열 인덱스를 key로 쓰면 왜 위험한가

index를 key로 쓰면, React는 "각 위치(0번째, 1번째, ...)에 있는 항목"을 기준으로 정체성을 판단하게 됩니다. 문제는 리스트에 삽입, 삭제, 재정렬이 일어나면 "항목의 내용"과 "항목의 위치(index)"가 서로 어긋난다는 점입니다.

구체적인 버그 재현 — 할 일 목록에 항목을 맨 앞에 추가

// 각 TodoItem은 내부에 입력 중인 텍스트를 로컬 state로 갖는다고 가정
{todos.map((todo, index) => (
  <TodoItem key={index} todo={todo} />
))}

// 최초 상태: ["빨래하기", "장보기"]
//   index 0 → "빨래하기" (사용자가 이 항목의 메모 칸에 "세제 사기"라고 입력 중)
//   index 1 → "장보기"

// "회의 준비"를 맨 앞에 추가한 뒤: ["회의 준비", "빨래하기", "장보기"]
//   index 0 → "회의 준비"   (React: "index 0은 이전에도 index 0이었으니 같은 컴포넌트다" → 인스턴스 재사용!)
//   index 1 → "빨래하기"
//   index 2 → "장보기"

React는 index 0 위치의 컴포넌트 인스턴스를 그대로 재사용하기 때문에, 그 인스턴스가 들고 있던 로컬 state(입력 중이던 "세제 사기" 메모)는 사라지지 않고 그대로 유지됩니다. 문제는 이 state가 이제 내용이 바뀐 "회의 준비" 항목에 붙어버린다는 것입니다. 사용자 입장에서는 "내가 빨래하기 항목에 적던 메모가 회의 준비 항목으로 옮겨갔다"는 이해할 수 없는 버그로 보입니다.

안정적인 id를 key로 쓰면 왜 해결되는가

{todos.map((todo) => (
  <TodoItem key={todo.id} todo={todo} />
))}

// "회의 준비"를 맨 앞에 추가한 뒤에도:
//   key="todo-99" (회의 준비) → 새로운 key → 새 컴포넌트 인스턴스 마운트
//   key="todo-1"  (빨래하기)  → 이전에도 key="todo-1"이었음 → 인스턴스 재사용, state 유지
//   key="todo-2"  (장보기)   → 인스턴스 재사용, state 유지

안정적인 id를 key로 쓰면, React는 "내용이 아니라 정체성"으로 항목을 추적할 수 있습니다. "빨래하기" 항목은 위치가 바뀌어도 여전히 key="todo-1"이므로 그 인스턴스와 로컬 state가 정확히 유지되고, 진짜 새 항목("회의 준비")만 새 인스턴스로 마운트됩니다. 이것이 key가 존재하는 근본적인 이유입니다 — "내용이 아니라 정체성으로 매칭하기 위해서".

index를 key로 써도 괜찮은 경우

모든 경우에 index가 위험한 것은 아닙니다. 다음 조건을 모두 만족하면 index를 key로 써도 실질적인 문제가 없습니다.

  • 리스트가 재정렬되지 않는다 (항상 같은 순서로 렌더링됨)
  • 리스트 중간에 항목이 삽입되거나 삭제되지 않는다 (끝에만 추가되거나, 아예 정적임)
  • 리스트 항목이 로컬 state, uncontrolled input, 애니메이션 등 "위치에 따라 유지되면 안 되는 상태"를 갖고 있지 않다

예를 들어 정적인 네비게이션 메뉴 항목이나, 절대 재정렬되지 않는 것으로 보장된 상수 배열을 렌더링할 때는 index도 실무적으로 문제없습니다. 하지만 이 조건을 실수로 어길 가능성(나중에 다른 개발자가 정렬 기능을 추가하는 등)이 있다면, 처음부터 안정적인 id를 쓰는 것이 안전합니다.

6. Fiber 아키텍처 개요 — 왜 중단 가능한 렌더링이 필요한가

Day 1에서 "render 단계는 여러 번 호출되거나 도중에 버려질 수 있다"고 했습니다. 이것이 가능한 이유가 React 16부터 도입된 Fiber 아키텍처입니다. 이전(React 15 이전)에는 트리 전체를 재귀 함수 호출 스택으로 한 번에 끝까지 처리했기 때문에, 트리가 크면 중간에 멈출 수 없어 메인 스레드를 오래 점유했고, 그동안 사용자 입력이나 애니메이션이 멈추는 문제가 있었습니다.

Fiber는 트리의 각 컴포넌트를 "작업 단위(unit of work)"로 나눈 자료구조입니다. React는 이 작업 단위를 하나씩 처리하다가, 더 급한 작업(사용자 입력 등)이 들어오면 현재 작업을 중단하고 나중에 이어서 하거나 아예 버릴 수 있습니다. 이것이 Day 7에서 다룰 concurrent rendering, transition의 기반입니다.

이것이 reconciliation과 어떤 관계인가

Fiber는 "이전 트리(current tree)"와 "작업 중인 새 트리(work-in-progress tree)"를 이중 버퍼처럼 유지합니다. reconciliation은 이 두 Fiber 트리를 같은 레벨끼리 비교하며 진행되고, 앞서 설명한 타입 비교·key 비교는 바로 이 Fiber 단위 비교에서 일어나는 일입니다. commit 단계에서는 완성된 work-in-progress 트리가 current 트리로 교체되며 실제 DOM에 반영됩니다.

7. key 선택 판단표

상황 권장 key 이유
서버/DB에서 온 데이터 리스트 (게시글, 상품, 유저 등) 고유 id (item.id) 정렬, 필터링, 실시간 갱신으로 순서·개수가 바뀔 가능성이 항상 있음
사용자가 추가/삭제/재정렬 가능한 폼 필드 리스트 필드 생성 시 발급한 고유 id (crypto.randomUUID() 등, index 아님) 입력 중인 값(uncontrolled state)이 항목과 함께 이동해야 하므로 정체성 유지가 필수
완전히 정적이고 하드코딩된 목록 (예: 상수로 정의한 메뉴 5개) index 허용 순서·개수가 코드 수정 없이는 절대 바뀌지 않음
애니메이션/트랜지션이 걸린 리스트 (드래그 정렬, 삭제 애니메이션) 반드시 고유 id DOM 노드 identity가 유지되어야 CSS transition/애니메이션 라이브러리가 올바르게 동작함
배열을 index가 유일하게 있는 원시값 리스트로 렌더링할 때 가능하면 값 자체 + index 조합(`${value}-${index}`), 값이 유니크하면 값 자체 원시값만 있고 재정렬 가능성이 있다면 값 기반 key가 index보다 안전

8. 답변 사고 순서

  1. key가 하는 일을 한 문장으로 정의합니다. "key는 여러 렌더링에 걸쳐 리스트 항목의 정체성을 React에게 알려주는 힌트다."
  2. key가 없을 때 React가 무엇을 대신 쓰는지 밝힙니다. "key가 없으면 React는 자식의 위치(index)를 정체성으로 사용한다."
  3. 위치와 내용이 어긋나는 조건을 짚습니다. "리스트에 삽입/삭제/재정렬이 일어나면 특정 위치의 내용이 바뀌지만, index는 그대로이므로 React가 잘못된 인스턴스를 재사용한다."
  4. 구체적인 증상으로 연결합니다. "그 결과 컴포넌트가 들고 있던 로컬 state, uncontrolled input 값, 애니메이션 상태가 엉뚱한 항목에 붙는 버그가 생긴다."
  5. 해결책과 예외 조건을 함께 제시합니다. "안정적인 고유 id를 key로 쓰면 해결되며, 리스트가 절대 재정렬·삽입·삭제되지 않는 정적인 경우에만 index도 예외적으로 허용된다."

9. 핵심 답변 예시

면접관: "할 일 목록에서 항목을 맨 앞에 추가했더니, 입력 중이던 텍스트 필드의 값이 엉뚱한 항목으로 옮겨갔습니다. 왜 이런 일이 일어나고, 어떻게 고치시겠어요?"

STRONG ANSWER

이 증상은 리스트를 렌더링할 때 key로 배열 index를 쓰고 있을 가능성이 매우 높습니다. React는 reconciliation 과정에서 이전 트리와 새 트리를 비교할 때, key가 있으면 key로 "같은 항목"을 판단하지만, key가 없으면 자식의 위치(index)를 정체성으로 사용합니다.


"빨래하기"가 index 0에 있는 상태에서 그 항목의 메모 입력칸에 텍스트를 입력하면, 그 텍스트는 index 0 위치의 TodoItem 컴포넌트 인스턴스가 들고 있는 로컬 state가 됩니다. 이후 "회의 준비"를 배열 맨 앞에 추가하면, "빨래하기"는 이제 index 1로 밀려나지만, React는 index만 보고 "index 0 위치는 이전에도 index 0이었다"고 판단해 그 컴포넌트 인스턴스(그리고 그 안의 로컬 state)를 그대로 재사용합니다. 결과적으로 원래 "빨래하기"에 있던 메모가 이제 index 0에 새로 렌더링된 "회의 준비" 항목에 그대로 붙어버립니다. 내용은 바뀌었는데 인스턴스는 재사용됐기 때문에, 사용자 입장에서는 메모가 다른 항목으로 "이동"한 것처럼 보입니다.


해결책은 배열 index 대신 각 항목이 가진 안정적인 고유 id(예: todo.id)를 key로 쓰는 것입니다. id를 key로 쓰면 React는 위치가 아니라 정체성으로 항목을 추적하므로, "빨래하기" 항목은 배열에서 몇 번째에 있든 항상 같은 컴포넌트 인스턴스와 매칭되어 그 항목의 메모가 그대로 유지되고, 새로 추가된 "회의 준비"만 새로운 컴포넌트 인스턴스로 마운트되어 빈 메모 상태로 시작합니다.


추가로, 만약 배열에 고유 id 필드가 아예 없다면(예: 외부 API가 id 없이 값만 준다면) 항목 생성 시점에 crypto.randomUUID() 등으로 id를 직접 부여하는 것을 권장합니다. 배열의 값 자체를 key로 쓰는 것은 값이 중복될 수 있는 경우 여전히 위험합니다.

이 답변이 좋은 이유:

  • 증상(메모가 옮겨감)을 "인스턴스가 재사용됐다"는 reconciliation 메커니즘으로 정확히 역추적함
  • key가 없을 때 React가 index를 대신 쓴다는 구체적인 동작을 명시함
  • "위치는 유지되지만 내용은 바뀐다"는 어긋남을 정확히 짚음
  • 해결책(id를 key로)뿐 아니라 그 해결책이 왜 작동하는지(정체성 기반 매칭)까지 설명함
  • id가 없는 실무 상황에 대한 대안까지 언급해 판단의 폭을 보여줌

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"key를 index로 쓰면 안 된다고 React 문서에 나와 있어서, todo.id로 바꾸면 해결됩니다."

이 답변이 약한 이유

  • "문서에 그렇게 나와 있다"는 규칙을 암기했을 뿐, 왜 index가 문제인지 메커니즘을 설명하지 못합니다.
  • reconciliation, 컴포넌트 인스턴스 재사용이라는 핵심 개념이 답변에 전혀 등장하지 않습니다.
  • "해결되는 이유"를 설명하지 못하므로, 면접관이 조건을 살짝 바꿔 물으면(예: "그럼 index를 써도 되는 경우는 없나요?") 답이 막힐 가능성이 높습니다.
WEAK ANSWER — 또 다른 패턴

"Virtual DOM이 실제 DOM보다 빨라서 그런 것 같습니다. key를 다르게 주면 Virtual DOM이 다시 계산되면서 고쳐집니다."

이 답변이 약한 이유

  • "Virtual DOM이 빠르다"는 것과 이 버그는 사실 관련이 없는데, 관련 없는 개념을 억지로 끌어다 쓰고 있습니다.
  • "key를 다르게 주면 고쳐진다"는 현상만 말할 뿐, key가 무엇을 위한 것인지(정체성 매칭) 설명이 없습니다.
  • 개념 간의 인과관계 없이 키워드만 나열하는 답변은 실제 이해가 아니라 암기로 비칩니다.
STRONG ANSWER 요소 체크리스트
  • key가 없으면 index가 정체성 역할을 대신한다는 것을 명시한다
  • "위치는 유지, 내용은 변경"이라는 어긋남을 구체적인 데이터 흐름으로 설명한다
  • 컴포넌트 인스턴스 재사용/언마운트라는 reconciliation의 핵심 동작으로 원인을 설명한다
  • 해결책과 함께 "왜 해결되는지"까지 설명한다
  • id가 없는 경우의 대안, index를 써도 되는 예외 조건까지 다룬다

11. Senior-level Follow-up Q&A

Q: key를 중복해서 쓰면 어떤 일이 일어나나요?

모범 답변: 같은 key를 가진 형제 엘리먼트가 여러 개 있으면 React는 콘솔에 경고를 띄우고, 실제로는 정의되지 않은(undefined) 동작을 보일 수 있습니다. 보통은 같은 key를 가진 항목들 중 나중 항목이 이전 항목의 위치에 있던 컴포넌트 인스턴스를 덮어써서, 일부 항목이 사라지거나 잘못된 state를 물려받는 것처럼 보입니다. key의 전제 조건은 "형제 사이에서 유일해야 한다"는 것이며, 트리 전체에서 유일할 필요는 없습니다(다른 부모 아래 있는 같은 key는 문제없습니다).

Q: key는 props로 컴포넌트 내부에서 접근할 수 있나요?

모범 답변: 아니요. key는 React가 reconciliation에 쓰기 위한 예약된 특수 prop이라서, 컴포넌트 내부에서 props.key로 접근할 수 없습니다. 컴포넌트 내부에서도 그 값이 필요하다면, key와는 별도로 동일한 값을 다른 이름의 prop(예: id)으로 명시적으로 넘겨줘야 합니다.

Q: key를 강제로 바꿔서 컴포넌트를 "리셋"시키는 패턴을 써본 적 있나요? 왜 그렇게 하나요?

모범 답변: 네, "key를 바꾸면 React가 다른 컴포넌트로 인식해 기존 인스턴스를 언마운트하고 완전히 새로 마운트한다"는 점을 역이용하는 패턴입니다. 예를 들어 폼 컴포넌트가 복잡한 내부 state(여러 필드, 유효성 검사 상태 등)를 갖고 있는데, 사용자가 다른 대상(예: 다른 유저 프로필)을 편집하기 시작할 때 이전 state가 남아있으면 안 되는 경우, 폼 컴포넌트에 key={targetUserId}를 주면 대상이 바뀔 때마다 React가 컴포넌트를 통째로 새로 마운트해 state를 확실히 초기화합니다. 여러 개의 useState를 수동으로 리셋하는 코드보다 훨씬 간단하고 실수가 적은 방법입니다.

Q: 리스트 diffing이 O(n)에 가깝다고 했는데, 정확히 무엇에 대해 선형인가요? 여전히 느려지는 경우가 있나요?

모범 답변: "같은 레벨의 자식 개수"에 대해 대략 선형입니다 — React는 이전 자식들과 새 자식들을 key(또는 index)로 한 번씩 순회하며 매칭하기 때문입니다. 다만 이는 "각 노드 자체의 diffing 비용"과는 별개입니다. 리스트 항목 하나하나가 무겁게 렌더링되는 컴포넌트라면, 항목 수가 많아질수록 diffing 자체는 빨라도 재사용되지 않은 항목들의 render 비용(컴포넌트 함수 재실행)은 여전히 커질 수 있습니다. 이런 경우 React.memo로 props가 바뀌지 않은 항목의 재렌더링 자체를 건너뛰는 최적화를 함께 고려합니다(Day 4에서 다루는 주제입니다).

Q: 애니메이션이 걸린 리스트에서 key를 잘못 쓰면 diffing 문제와 별개로 왜 문제가 생기나요?

모범 답변: CSS transition이나 애니메이션 라이브러리(Framer Motion 등)는 대부분 "같은 DOM 노드가 유지된 채 속성이 바뀔 때"를 전제로 부드러운 전환을 계산합니다. key가 잘못되어 React가 실제로는 같은 항목인데 다른 인스턴스로 취급하면, 기존 DOM 노드가 삭제되고 새 DOM 노드가 생성되므로 "전환"이 아니라 "즉시 나타남/사라짐"으로 보입니다. 반대로 실제로는 다른 항목인데 같은 key(예: index)를 써서 같은 인스턴스로 취급되면, 서로 다른 항목 사이에서 부자연스러운 슬라이드/모핑 애니메이션이 발생합니다. 두 경우 모두 근본 원인은 "정체성 판단이 실제 데이터의 정체성과 어긋났다"는 것입니다.

12. 흔한 오해

오해 1

"key는 성능을 위해 React가 각 항목을 빠르게 찾도록 도와주는 인덱스 같은 것이다"

key의 본질은 "빠른 검색"이 아니라 "정체성 판단"입니다. key가 없어도 React는 동작하지만(위치 기반으로), key가 있으면 위치가 바뀌어도 정체성을 유지할 수 있다는 것이 핵심입니다. 성능은 부수적인 효과일 뿐, key의 존재 이유는 상태와 DOM 노드를 올바른 항목에 계속 붙여두기 위함입니다.

오해 2

"key만 잘 주면 렌더링 성능이 항상 좋아진다"

key를 올바르게 쓰면 "불필요한 언마운트/재마운트"는 막을 수 있지만, 재사용된 컴포넌트도 props가 바뀌면 여전히 render 함수가 다시 호출됩니다(즉 재렌더링 자체는 일어남). "재렌더링을 아예 건너뛰게" 하려면 React.memo 같은 별도의 메모이제이션이 필요합니다. key는 "인스턴스 재사용 여부"를 결정하는 것이지 "재렌더링 여부"를 결정하는 것이 아닙니다.

오해 3

"Virtual DOM diffing은 실제 DOM 트리 두 개를 비교하는 것이다"

정확히는 실제 DOM이 아니라, React가 만든 가벼운 자바스크립트 객체 트리(엘리먼트/Fiber 트리) 두 개를 비교합니다. 실제 DOM은 이 비교가 끝난 뒤 commit 단계에서 "계산된 최소 변경 사항"만 반영받는 대상이지, 비교의 대상 자체가 아닙니다.

오해 4

"key는 리스트를 렌더링할 때만 필요하다"

가장 흔한 경우가 리스트이지만, key의 본질(정체성 힌트)은 리스트가 아닌 상황에서도 유용합니다. 앞서 다룬 "key를 강제로 바꿔 컴포넌트를 리셋시키는" 패턴처럼, 단일 컴포넌트에 조건에 따라 다른 key를 줘서 의도적으로 재마운트를 유도하는 것도 흔한 실무 패턴입니다.

오해 5

"index를 key로 쓰는 것은 항상 안티패턴이라 절대 쓰면 안 된다"

앞서 다뤘듯 리스트가 정적이고 재정렬·삽입·삭제가 없다면 index도 실무적으로 문제없습니다. "절대 금지"라는 이분법적 규칙보다, "이 리스트가 앞으로도 순서와 구성이 바뀌지 않을 것이 보장되는가"를 판단 기준으로 삼는 것이 시니어다운 태도입니다.
