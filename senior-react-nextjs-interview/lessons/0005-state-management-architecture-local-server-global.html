1. 핵심 면접 질문

Primary Question

"우리 팀은 사용자 프로필 데이터를 useEffect로 fetch해서 Redux store에 dispatch하고, 다른 컴포넌트들은 그 store를 구독해서 화면을 그립니다. 이 방식의 문제점은 무엇이고, 어떻게 개선하시겠습니까?"

Variant 1

"Context API로 로그인한 사용자 정보와 테마, 장바구니 개수를 하나의 전역 상태로 묶어서 관리했더니, 장바구니 개수만 바뀌어도 테마와 무관한 컴포넌트까지 리렌더링됩니다. 원인과 해결책을 설명해 주세요."

Variant 2

"검색 페이지에서 필터 조건(카테고리, 정렬, 페이지 번호)을 useState로 관리했는데, 사용자가 새로고침하거나 링크를 공유하면 필터가 초기화됩니다. 어떻게 설계를 바꿔야 합니까?"

2. 왜 면접에서 묻는가

이 질문은 "Redux/Zustand/React Query 중 어떤 라이브러리를 써봤는가"가 아니라, "지금 다루려는 데이터가 어떤 종류의 상태인지 먼저 정확히 분류할 수 있는가"를 테스트합니다. 상태 관리 버그와 아키텍처 부채의 대부분은 라이브러리 선택이 아니라 분류 실패에서 시작됩니다 — 서버에서 받아온 데이터를 클라이언트 전역 상태처럼 다루거나, URL에 있어야 할 정보를 컴포넌트 로컬 상태로 가두거나, 계산으로 얻을 수 있는 값을 별도 state로 저장하는 식입니다.

면접관이 이 질문으로 평가하는 것:

  • "이 데이터를 누가 소유하는가"라는 기준으로 상태를 분류할 수 있는가 — 서버가 소유한 데이터(캐시일 뿐 진짜 소스는 서버)와 클라이언트가 소유한 데이터(UI 상호작용의 결과물)를 구분하는가.
  • 서버 상태의 고유한 문제(staleness, 동시 요청 중복, 백그라운드 재검증, 로딩/에러 상태 조합 폭발)를 아는가 — 이것이 왜 useState + useEffect로 직접 구현하면 안 되는 문제인지 설명할 수 있는가.
  • 전역 상태 도구를 "일단 크게 만들고 보는" 습관 대신, 필요한 최소 범위로 좁히는 판단을 하는가 — Context를 하나로 뭉치면 생기는 리렌더링 오염을 이해하는가.
  • URL도 상태 저장소의 하나라는 것을 아는가 — 공유 가능성, 새로고침 내구성, 브라우저 히스토리(뒤로 가기)가 필요한 데이터는 URL에 둬야 한다는 판단 기준이 있는가.
  • 파생 상태(derived state)를 별도 state로 중복 저장하지 않는 원칙을 아는가 — "동기화해야 할 state가 늘어날수록 버그 표면이 넓어진다"는 인과 관계를 이해하는가.

10년차 이하에서 자주 나오는 답변 패턴: "Redux 대신 Zustand를 쓰면 보일러플레이트가 줄어서 좋아요." 이 답변은 도구 교체에만 머물러 있고, 애초에 "이 데이터가 왜 전역 상태여야 하는가", "서버 데이터라면 왜 범용 상태 관리 도구가 아니라 서버 상태 전용 도구(React Query 등)를 써야 하는가"라는 분류 질문에 답하지 못합니다. 시니어는 도구 이름보다 먼저 "이 상태의 소유자와 수명(lifetime)이 무엇인가"를 묻습니다.

3. 전제 개념 — 쉬운 한국어로

"상태(state)"란 결국 무엇인가

React에서 상태란 "시간이 지나면서 바뀔 수 있고, 그 값이 바뀌면 화면이 다시 그려져야 하는 데이터"입니다. 이 정의 자체는 단순하지만, 실무에서 어려운 부분은 "이 데이터가 바뀌는 원인이 무엇이고, 누가 그 변경의 권한을 갖는가"를 구분하는 일입니다. 예를 들어 "모달이 열려 있는가"는 순전히 이 화면의 클릭으로만 바뀌는 데이터이고, "이 게시글의 좋아요 수"는 서버의 데이터베이스가 진짜 값을 갖고 있고 내 화면은 그 값을 잠시 빌려와 보여줄 뿐입니다. 이 둘을 같은 방식으로 다루면 문제가 생깁니다.

"소유권(ownership)"이라는 기준

상태 관리 설계에서 가장 중요한 질문은 "이 값의 진짜 원본(source of truth)이 어디에 있는가"입니다. 진짜 원본이 브라우저 메모리 안(이 컴포넌트 트리)에만 있다면 클라이언트 상태이고, 진짜 원본이 서버의 데이터베이스에 있고 내 화면은 그것을 복사해 온 것이라면 서버 상태입니다. 이 구분이 왜 중요한가 하면, 서버 상태는 내가 마지막으로 가져온 이후에 다른 사용자나 다른 탭이 이미 바꿔놓았을 수 있다는, 클라이언트 상태에는 없는 특성을 갖기 때문입니다.

Prop drilling과 전역 상태의 관계

Prop drilling이란 최상위 컴포넌트가 가진 값을 여러 단계 아래의 자식에게 전달하기 위해, 그 값을 사용하지도 않는 중간 컴포넌트들이 단지 "전달 통로" 역할만 하며 props를 릴레이하는 상황을 말합니다. 이 불편함 때문에 많은 사람이 "그럼 전역 상태에 넣어서 아무 데서나 꺼내 쓰자"고 성급하게 판단합니다. 하지만 전역 상태로 옮기는 것은 prop drilling의 불편함을 없애는 대신, "이 값이 어디서 바뀌는지 추적하기 어려워지는" 새로운 비용을 만듭니다. 이 트레이드오프를 이해하지 못하면 "일단 다 전역 상태로 두자"는 설계로 흘러가기 쉽습니다.

4. State 분류 프레임워크 — 4가지 상태 유형

시니어 프론트엔드 엔지니어는 새로운 데이터를 다루기 전에, 먼저 이 데이터가 다음 4가지 중 어디에 속하는지 판단합니다.

① 로컬 UI 상태 (Local UI State)

이 컴포넌트(또는 가까운 부모)만 알면 되는, 순수하게 화면 상호작용에서 생기는 상태입니다. "이 드롭다운이 열려 있는가", "이 입력창에 지금 무엇이 타이핑되고 있는가", "이 아코디언이 펼쳐졌는가" 같은 것들입니다. 진짜 원본이 이 컴포넌트 트리 안에만 존재하고, 페이지를 벗어나면 사라져도 문제없는 경우가 대부분입니다. useState/useReducer로 충분합니다.

② 서버 상태 (Server State)

진짜 원본이 서버(데이터베이스, 외부 API)에 있고, 클라이언트는 그것의 "스냅샷을 캐싱"하고 있는 상태입니다. 사용자 프로필, 게시글 목록, 주문 내역 같은 것들입니다. 이 상태는 비동기로 가져와야 하고(로딩 상태 필요), 실패할 수 있고(에러 상태 필요), 시간이 지나면 낡을 수 있고(staleness), 여러 컴포넌트가 같은 데이터를 요청할 수 있고(중복 요청 방지 필요), 클라이언트 상태와는 본질적으로 다른 문제를 갖습니다.

③ 전역 클라이언트 상태 (Global Client State)

서버에서 온 것은 아니지만, 여러 개의 서로 관련 없는 컴포넌트 트리에서 공유되어야 하는 클라이언트 전용 상태입니다. 로그인 여부에 따른 UI 분기, 다크모드 테마, 현재 열려 있는 전역 모달의 종류 같은 것들입니다. 진짜 원본이 여전히 브라우저 메모리 안에 있다는 점에서 로컬 UI 상태와 같지만, 공유 범위가 넓다는 점이 다릅니다.

④ URL 상태 (URL State)

주소창의 쿼리 파라미터나 경로 세그먼트에 저장되어야 하는 상태입니다. 검색어, 필터, 정렬 기준, 페이지 번호, 선택된 탭 같은 것들입니다. 이 상태의 특징은 "사용자가 이 화면을 북마크하거나 링크로 공유했을 때, 같은 화면이 재현되어야 하는가"입니다. 재현되어야 한다면 URL에 있어야 하고, 컴포넌트 state나 전역 상태에 두면 새로고침·공유 시 정보가 유실됩니다.

다섯 번째로, 이 네 가지 중 어디에도 속하지 않아야 하는 것이 있습니다 — 파생 상태(derived state)입니다. 다른 state나 props로부터 계산할 수 있는 값(예: 목록의 개수, 필터링된 결과, 합계)은 그 자체를 별도 state로 저장하면 안 됩니다. 렌더링 중에 계산하거나 useMemo로 감싸면 충분하며, 별도 state로 저장하는 순간 "원본 state가 바뀔 때마다 파생 state도 동기화해야 하는" 새로운 책임과 버그 표면이 생깁니다.

5. 서버 상태가 근본적으로 다른 이유

많은 개발자가 "서버에서 데이터를 받아와서 state에 넣는 것"과 "순수하게 클라이언트에서 만든 값을 state에 넣는 것"을 같은 작업으로 여깁니다. 하지만 서버 상태는 다음과 같은, 클라이언트 상태에는 없는 문제들을 동반합니다.

서버 상태의 고유 문제 왜 클라이언트 상태에는 없는가
비동기 로딩/에러/성공 상태 조합 클라이언트 상태는 동기적으로 즉시 결정되지만, 서버 상태는 네트워크 왕복이 필요하므로 "아직 없음/불러오는 중/실패함/있음"이라는 상태 기계가 항상 따라온다.
Staleness(신선도) 문제 마지막으로 가져온 이후 서버의 실제 데이터가 다른 사용자, 다른 탭, 백그라운드 작업에 의해 이미 바뀌었을 수 있다. 클라이언트가 만든 값은 이런 "외부에서 변경될 가능성"이 없다.
중복 요청(deduplication) 같은 데이터를 여러 컴포넌트가 동시에 요청하면, 네트워크 요청을 하나로 합쳐야 한다. 클라이언트 상태는 애초에 네트워크 요청이라는 개념이 없다.
백그라운드 재검증(revalidation) 사용자가 탭을 다시 포커스했을 때, 또는 일정 시간이 지났을 때 데이터를 조용히 다시 가져와 최신 상태를 유지해야 할 때가 많다. 이런 정책 자체가 클라이언트 상태에는 해당하지 않는 개념이다.
캐시 무효화(invalidation) 사용자가 글을 수정하면, 그 글을 보여주던 다른 화면의 캐시도 무효화되어 다시 가져와야 한다. "이 캐시가 다른 어떤 화면에 영향을 주는가"를 추적하는 것은 범용 상태 관리 도구가 해결하도록 설계된 문제가 아니다.
경쟁 상태(race condition) 사용자가 검색어를 빠르게 바꾸면 이전 요청의 응답이 최신 요청의 응답보다 늦게 도착할 수 있다. 이 순서를 직접 관리하지 않으면 화면에 오래된 결과가 잘못 표시된다.

이 여섯 가지 문제를 useState + useEffect로 직접 구현하면, 결국 React Query나 SWR 같은 라이브러리가 이미 만들어 놓은 것을 조악하게 재구현하게 됩니다. 이것이 "서버 상태는 서버 상태 전용 도구로 관리해야 한다"는 원칙의 근거입니다.

6. 안티패턴 — 서버 상태를 Redux/Context에 넣기

// 안티패턴: 서버 데이터를 범용 전역 상태 도구로 관리
function useUserProfile(userId) {
  const dispatch = useDispatch();
  const profile = useSelector((state) => state.user.profile);

  useEffect(() => {
    dispatch(setLoading(true));
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then((data) => dispatch(setProfile(data)))
      .catch((err) => dispatch(setError(err)))
      .finally(() => dispatch(setLoading(false)));
  }, [userId, dispatch]);

  return profile;
}

이 코드가 프로덕션에서 만드는 문제들:

  1. 1
    중복 요청 방지가 없다. 같은 userId를 사용하는 컴포넌트가 두 개 마운트되면, 두 개의 useEffect가 각각 fetch를 실행해 동일한 요청이 두 번 나간다.
  2. 2
    재검증 정책이 없다. 사용자가 다른 탭에서 프로필을 수정하고 돌아와도, 이 화면은 마운트 시점에 가져온 옛 데이터를 계속 보여준다.
  3. 3
    경쟁 상태를 처리하지 못한다. userId가 빠르게 바뀌면(예: 리스트에서 다른 사용자를 빠르게 클릭) 먼저 보낸 요청의 응답이 나중에 도착해 최신 userId의 데이터를 덮어쓸 수 있다.
  4. 4
    다른 화면과 캐시를 공유하지 못한다. 이미 프로필 목록 화면에서 같은 사용자 데이터를 가져온 적이 있어도, 상세 화면은 이를 재사용하지 못하고 또 네트워크 요청을 보낸다.
  5. 5
    action type, reducer, selector라는 boilerplate가 "그냥 데이터를 가져와 보여주는" 단순한 작업에 비해 과도하다. 이 boilerplate는 서버 상태가 필요로 하는 문제(재검증, 중복 제거, 캐시 무효화)를 해결해주지 않으면서 코드량만 늘린다.

개선 — 서버 상태 전용 도구로 이전

// React Query(TanStack Query)를 사용한 개선
function useUserProfile(userId) {
  return useQuery({
    queryKey: ["user", userId],
    queryFn: () => fetch(`/api/users/${userId}`).then((res) => res.json()),
  });
}

// 사용하는 곳
function ProfileCard({ userId }) {
  const { data, isLoading, isError } = useUserProfile(userId);
  if (isLoading) return ;
  if (isError) return ;
  return ;
}

이 코드는 queryKey가 같으면 자동으로 요청을 중복 제거하고, 캐시를 여러 컴포넌트가 공유하고, 탭 재포커스 시 자동 재검증하고, 오래된 요청의 응답을 무시하는 경쟁 상태 방지까지 기본으로 제공합니다. Redux/Context 코드에서 직접 구현해야 했던 문제들을 라이브러리가 대신 해결하므로, 애플리케이션 코드는 "이 데이터가 무엇이고 어떻게 가져오는가"만 선언하면 됩니다.

7. Context API의 리렌더링 함정

Context API 자체는 상태 관리 라이브러리가 아니라 "값을 트리 아래로 전달하는 방법"입니다. 이 사실을 놓치면, prop drilling을 피하려고 여러 개의 서로 무관한 값을 하나의 Context로 묶는 실수를 하게 됩니다.

// 안티패턴: 무관한 값들을 하나의 Context로 묶음
const AppContext = createContext();

function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState("light");
  const [cartCount, setCartCount] = useState(0);

  // 매 렌더링마다 새 객체 — Provider를 구독하는 모든 컴포넌트가 리렌더링 후보가 됨
  const value = { user, setUser, theme, setTheme, cartCount, setCartCount };

  return {children};
}

이 구조의 문제는 두 가지입니다. 첫째, cartCount만 바뀌어도 value 객체 전체가 새로 생성되므로, useContext(AppContext)를 호출하는 모든 컴포넌트가 리렌더링됩니다 — 실제로는 theme만 쓰는 컴포넌트까지 포함해서요. 둘째, 이 Context를 사용하는 관점에서 "user만 필요한 컴포넌트"도 어쩔 수 없이 theme와 cartCount의 변경에 함께 반응하는 구조가 됩니다.

해결 — 관심사별로 Context를 분리

const UserContext = createContext();
const ThemeContext = createContext();
const CartContext = createContext();

// 각 Provider를 필요한 만큼만 중첩
function AppProviders({ children }) {
  return (
    
      
        {children}
      
    
  );
}

이렇게 나누면 cartCount가 바뀌어도 CartContext를 구독하는 컴포넌트만 리렌더링되고, ThemeContext만 구독하는 컴포넌트는 영향받지 않습니다. 원칙은 "함께 바뀌는 빈도가 비슷하고, 실제로 관련 있는 값끼리만 하나의 Context로 묶는다"는 것입니다. 변경 빈도가 크게 다른 값을 한 Context에 묶으면, 자주 바뀌는 값이 드물게 바뀌는 값을 쓰는 컴포넌트까지 리렌더링시키는 비용을 만듭니다.

더 근본적인 대안 — 외부 상태 관리 라이브러리

전역 상태의 변경 빈도가 높고 구독 범위를 세밀하게 제어해야 한다면, Context보다 Zustand, Jotai 같은 선택적 구독(selective subscription)을 지원하는 라이브러리가 더 적합합니다. 이런 라이브러리는 "이 컴포넌트가 store의 어떤 슬라이스를 구독하는지"를 세밀하게 지정할 수 있어서, Context처럼 "Provider 전체 값이 바뀌면 구독자 전체가 리렌더링"되는 구조적 한계가 없습니다.

8. URL 상태와 파생 상태

URL 상태를 판단하는 기준

어떤 상태를 URL에 둘지 판단하는 가장 실용적인 질문은 다음과 같습니다: "이 화면을 북마크하거나, 새로고침하거나, 다른 사람에게 링크로 공유했을 때, 지금과 같은 화면이 다시 나타나야 하는가?" 그렇다면 URL 쿼리 파라미터나 경로에 저장해야 합니다.

// 안티패턴: 공유되어야 할 필터 상태를 컴포넌트 state로 가둠
function SearchPage() {
  const [category, setCategory] = useState("all");
  const [sortBy, setSortBy] = useState("recent");
  const [page, setPage] = useState(1);
  // 새로고침하거나 링크를 공유하면 항상 초기값으로 리셋됨
}

// 개선: URL 쿼리 파라미터를 단일 원천으로 사용
function SearchPage() {
  const searchParams = useSearchParams(); // Next.js App Router
  const category = searchParams.get("category") ?? "all";
  const sortBy = searchParams.get("sortBy") ?? "recent";
  const page = Number(searchParams.get("page") ?? "1");
  // 필터 변경 시 router.push로 URL을 갱신 → 새로고침/공유해도 동일한 화면 재현
}

URL 상태의 부가적인 이점은 브라우저의 뒤로 가기/앞으로 가기가 자연스럽게 동작한다는 것입니다. 컴포넌트 state로 필터를 관리하면 뒤로 가기를 눌러도 필터가 되돌아가지 않지만, URL 상태로 관리하면 브라우저 히스토리 자체가 상태 변경 이력을 담당합니다.

파생 상태를 별도로 저장하지 않는 원칙

// 안티패턴: 파생 가능한 값을 별도 state로 중복 저장
function ProductList({ products }) {
  const [filteredProducts, setFilteredProducts] = useState(products);
  const [category, setCategory] = useState("all");

  useEffect(() => {
    setFilteredProducts(
      category === "all" ? products : products.filter((p) => p.category === category)
    );
  }, [category, products]); // products나 category가 바뀔 때마다 동기화해야 함 — 누락 시 버그
}

// 개선: 렌더링 중 계산 (필요하면 useMemo)
function ProductList({ products }) {
  const [category, setCategory] = useState("all");

  const filteredProducts = useMemo(
    () => (category === "all" ? products : products.filter((p) => p.category === category)),
    [category, products]
  );
}

첫 번째 코드의 문제는 filteredProducts라는 "진짜 원본이 아닌 복사본"을 state로 만들어서, products나 category가 바뀔 때마다 useEffect로 동기화해줘야 하는 새로운 책임이 생긴다는 점입니다. 이 동기화 로직이 하나라도 누락되면(예: products를 의존성 배열에서 빠뜨리면) 화면에 오래된 필터 결과가 남는 버그가 생깁니다. 파생 가능한 값은 별도 state로 만들지 않고, 렌더링 중 계산하는 것이 원칙이며, 계산 비용이 클 때만 useMemo로 캐싱합니다.

9. 상태 유형별 도구 선택표

상태 유형 대표적인 예 권장 도구 선택 이유
로컬 UI 상태 모달 열림 여부, 입력 필드 값, 아코디언 펼침 useState / useReducer 이 컴포넌트(또는 가까운 부모)만 알면 되고, 공유 범위가 좁아 추가 도구가 오버엔지니어링이 됨
서버 상태 사용자 프로필, 게시글 목록, 주문 내역 React Query(TanStack Query), SWR, (Next.js App Router에서는) Server Component fetch + fetch 캐싱 캐싱, 중복 제거, 재검증, 로딩/에러 상태 관리를 라이브러리가 대신 처리
전역 클라이언트 상태 — 변경 빈도 낮음, 범위 넓음 로그인 여부, 테마 Context API (관심사별로 분리) 추가 라이브러리 없이 React 기본 기능으로 충분하고, 변경 빈도가 낮아 리렌더링 비용이 크지 않음
전역 클라이언트 상태 — 변경 빈도 높음, 세밀한 구독 필요 실시간 알림 개수, 복잡한 UI 상태 머신 Zustand, Jotai 선택적 구독으로 리렌더링 범위를 세밀하게 제어 가능, Context 대비 보일러플레이트가 적음
URL 상태 검색어, 필터, 정렬, 페이지 번호, 선택된 탭 useSearchParams/router.push (Next.js), 쿼리 파라미터 기반 상태 새로고침 내구성, 공유 가능성, 브라우저 히스토리(뒤로 가기) 지원이 필요
파생 상태 필터링된 목록, 합계, 개수 별도 state 없이 렌더링 중 계산, 필요 시 useMemo 원본 state와의 동기화 책임 자체를 없애 버그 표면을 제거

10. 증상 기반 진단표

증상 의심할 원인 확인 방법
같은 데이터를 여러 화면이 각자 다시 fetch해서 네트워크 요청이 중복됨 서버 상태를 컴포넌트별 로컬 state로 관리 중 — 캐시 공유가 없음 Network 탭에서 동일 엔드포인트 중복 호출 확인 → 서버 상태 전용 도구(React Query 등)로 이전 검토
전역 상태 하나만 바꿨는데 관련 없어 보이는 컴포넌트들까지 리렌더링됨 여러 관심사를 하나의 Context/store 슬라이스로 묶음 React DevTools Profiler로 리렌더링 원인 컴포넌트 확인 → Context를 관심사별로 분리하거나 선택적 구독 라이브러리로 전환
새로고침하거나 링크를 공유하면 필터/검색/페이지 상태가 초기화됨 공유 가능해야 할 상태가 URL이 아닌 컴포넌트 state에 있음 해당 상태가 "북마크 가능해야 하는가"를 질문 → URL 쿼리 파라미터로 이전
목록/필터 관련 state 두 개가 미묘하게 어긋나는 버그(예: 필터를 바꿨는데 개수 표시가 이전 값) 파생 가능한 값을 별도 state로 저장하고 동기화 effect를 누락 해당 state가 다른 state/props로부터 계산 가능한지 확인 → 별도 state 제거하고 렌더링 중 계산으로 대체
빠르게 조건을 바꿨을 때(검색어 연속 타이핑 등) 화면에 이전 결과가 잘못 표시됨 서버 상태를 직접 구현하면서 요청 순서(race condition) 처리를 누락 이전 요청 취소/최신 요청만 반영하는 로직이 있는지 확인 → 서버 상태 전용 도구는 기본으로 처리

11. 답변 사고 순서

  1. 먼저 데이터를 분류합니다. "이 프로필 데이터는 진짜 원본이 서버 데이터베이스에 있으므로 서버 상태이지, 클라이언트가 소유하는 전역 상태가 아니다."
  2. 현재 구조가 놓치고 있는 서버 상태 고유 문제를 짚습니다. "지금 구조는 중복 요청 방지, 백그라운드 재검증, 경쟁 상태 처리, 캐시 공유가 전혀 없다 — 이건 Redux 자체의 결함이 아니라, 서버 상태를 범용 상태 관리 도구로 다루기 때문에 생기는 구조적 누락이다."
  3. 도구 교체를 제안합니다. "React Query 같은 서버 상태 전용 도구로 이전하면 이 문제들을 라이브러리가 기본으로 해결해준다."
  4. 남는 전역 상태의 범위를 재확인합니다. "이전 후 Redux/Context에는 서버에서 온 데이터가 아니라, 진짜로 클라이언트가 소유하는 값(로그인 UI 분기, 테마 등)만 남기는 것이 맞다."
  5. 실무 연결로 마무리합니다. "이런 전환은 한 번에 전체를 바꾸기보다, 새로 추가되는 서버 데이터부터 React Query로 옮기고, 기존 Redux 슬라이스는 트래픽이 몰리거나 버그가 반복되는 부분부터 점진적으로 이전하는 것이 실무적으로 안전하다."

12. 핵심 답변 예시

면접관: "우리 팀은 사용자 프로필 데이터를 useEffect로 fetch해서 Redux store에 dispatch하고, 다른 컴포넌트들은 그 store를 구독해서 화면을 그립니다. 이 방식의 문제점은 무엇이고, 어떻게 개선하시겠습니까?"

STRONG ANSWER

가장 먼저 짚고 싶은 것은, 사용자 프로필은 서버 상태라는 점입니다. 진짜 원본이 서버 데이터베이스에 있고, 클라이언트는 그것의 스냅샷을 잠시 캐싱하고 있는 것뿐입니다. 이 특성 때문에 서버 상태는 클라이언트 전용 상태에는 없는 문제들 — 중복 요청 방지, staleness에 대한 재검증 정책, 여러 컴포넌트 간 캐시 공유, 요청 순서가 뒤바뀌었을 때의 경쟁 상태 처리 — 를 항상 동반합니다.


지금 구조(useEffect + dispatch)는 이 문제들을 전혀 다루지 않습니다. 예를 들어 같은 사용자 프로필을 보여주는 컴포넌트가 두 개 있으면 각각 fetch를 실행해 요청이 중복되고, 다른 탭에서 프로필을 수정하고 돌아와도 이 화면은 마운트 시점의 옛 데이터를 계속 보여줍니다. 이건 Redux 자체의 문제가 아니라, 범용 상태 관리 도구로 서버 상태 고유의 문제를 직접 재구현하려 했기 때문에 생기는 누락입니다.


제 제안은 이 데이터를 React Query 같은 서버 상태 전용 도구로 이전하는 것입니다. queryKey로 캐시를 관리하면 중복 요청이 자동으로 제거되고, 탭 재포커스 시 백그라운드 재검증이 기본 제공되며, 오래된 요청의 응답이 최신 상태를 덮어쓰는 경쟁 상태도 라이브러리가 처리해 줍니다. 그리고 이전 이후 Redux store에는 서버에서 온 데이터가 아니라, 로그인 여부에 따른 UI 분기나 테마처럼 진짜로 클라이언트가 소유하는 상태만 남기는 것이 맞다고 봅니다. 실제 전환은 한 번에 전체를 옮기기보다, 새로 추가되는 API부터 React Query로 작성하고, 기존 슬라이스는 버그가 자주 나거나 트래픽이 많은 화면부터 우선순위를 두고 점진적으로 이전하겠습니다.

이 답변이 좋은 이유:

  • 도구 이름을 말하기 전에 "이 데이터가 어떤 종류의 상태인가"부터 분류함
  • 현재 구조의 문제를 "Redux가 나쁘다"가 아니라 "서버 상태를 범용 도구로 다뤄서 생기는 구조적 누락"으로 정확히 짚음
  • 제안한 도구가 구체적으로 어떤 문제를 해결하는지(중복 요청, 재검증, 경쟁 상태) 하나씩 연결함
  • 전환 후 남는 전역 상태의 범위를 재정의하고, 실무적인 점진적 마이그레이션 전략까지 제시함

13. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"Redux는 보일러플레이트가 많으니까 Zustand나 Recoil처럼 더 가벼운 라이브러리로 바꾸면 좋을 것 같습니다."

이 답변이 약한 이유

  • 문제의 원인을 "Redux라는 도구의 문법이 장황하다"는 표면적인 이유로만 진단해서, 서버 상태의 근본 문제(중복 요청, 재검증, 경쟁 상태)를 전혀 짚지 못했습니다.
  • 제안한 대안(Zustand, Recoil)도 여전히 범용 클라이언트 상태 관리 도구이므로, 그대로 옮겨도 중복 요청/재검증 문제는 해결되지 않습니다 — 근본적으로 잘못된 카테고리의 도구를 다른 도구로 바꾼 것뿐입니다.
WEAK ANSWER — 또 다른 패턴

"전역 상태에 loading, error 필드를 추가하고, useEffect에 캐시 체크 로직과 재시도 로직을 직접 짜면 해결될 것 같습니다."

이 답변이 약한 이유

  • 이미 검증되고 최적화된 라이브러리가 존재하는 문제를 직접 재구현하려는 접근입니다 — 경쟁 상태 처리, 탭 포커스 감지, 캐시 무효화 전파 같은 세부 사항은 직접 구현 시 누락되거나 버그가 생기기 쉽습니다.
  • "서버 상태는 서버 상태 전용 도구가 존재하는 별도 카테고리"라는 인식 자체가 없어서, 문제를 어떻게든 기존 도구 안에서 땜질하려는 방향으로 흘러갑니다.
STRONG ANSWER 요소 체크리스트
  • 데이터의 소유권(서버 vs 클라이언트)을 먼저 판단하는 것으로 답을 시작한다
  • 현재 구조가 놓치는 서버 상태 고유 문제(중복 요청, staleness, 경쟁 상태, 캐시 공유)를 구체적으로 나열한다
  • Redux/Context 자체를 비난하지 않고, "잘못된 카테고리의 도구로 사용했다"고 정확히 진단한다
  • 전환 후 남는 전역 상태의 범위를 재정의한다
  • 점진적 마이그레이션이라는 실무적 실행 전략까지 제시한다

14. Senior-level Follow-up Q&A

Q: Next.js App Router의 Server Component에서 직접 데이터를 fetch하면, React Query 같은 클라이언트 서버-상태 라이브러리가 아예 필요 없어지는 것 아닌가요?

모범 답변: 완전히는 아닙니다. Server Component fetch는 "최초 페이지 렌더링에 필요한 데이터를 서버에서 한 번 가져오는" 문제를 잘 해결하지만, 그 이후 클라이언트에서 발생하는 상호작용(사용자가 필터를 바꿔서 재요청, 무한 스크롤, 폼 제출 후 목록 갱신, 특정 데이터만 주기적으로 폴링)은 여전히 클라이언트에서 처리해야 합니다. 실무에서는 최초 데이터는 Server Component에서 가져오고, 그 이후의 상호작용성이 필요한 부분만 React Query 같은 클라이언트 도구와 결합하는 하이브리드 구조를 씁니다. 즉 "정적인 최초 로드"와 "동적인 클라이언트 상호작용"은 여전히 다른 계층의 문제입니다.

Q: 서버에서 받아온 데이터를 사용자가 화면에서 수정(낙관적 업데이트)하는 경우는 어떻게 분류해야 하나요?

모범 답변: 이 경우도 여전히 서버 상태입니다 — 다만 "낙관적 업데이트(optimistic update)"라는 특수한 처리가 추가되는 것입니다. 사용자의 액션(예: 좋아요 클릭) 직후, 서버 응답을 기다리지 않고 캐시를 먼저 낙관적으로 갱신해 화면을 즉시 반영하고, 서버 요청이 실패하면 캐시를 이전 값으로 롤백합니다. React Query의 onMutate/onError 콜백이 이 패턴을 지원합니다. 중요한 것은 이 값의 진짜 원본은 여전히 서버에 있고, 클라이언트가 "먼저 맞다고 가정하고 보여주다가 틀리면 되돌리는" 전략을 쓴다는 점이며, 이것이 클라이언트 전역 상태로 분류를 바꾸는 이유는 되지 않는다는 것입니다.

Q: 폼(form) 입력 상태는 로컬 UI 상태인가요, 아니면 다른 범주인가요?

모범 답변: 원칙적으로는 로컬 UI 상태입니다 — 진짜 원본이 사용자가 지금 타이핑 중인 이 컴포넌트에만 있기 때문입니다. 다만 폼이 커지면(여러 단계 마법사, 필드 간 유효성 검증 의존, 배열 필드) 상태 전이 로직이 복잡해지므로, 이런 경우 useReducer나 React Hook Form 같은 폼 전용 라이브러리를 씁니다. 폼 전용 라이브러리는 "전역 상태 도구"가 아니라, 로컬 UI 상태 중에서도 "리렌더링 최소화와 유효성 검증"이라는 폼 특유의 문제를 해결하는 특화된 도구라는 점에서, 앞서 다룬 4가지 분류와는 별도의 축(문제의 복잡도에 따른 도구 선택)으로 이해해야 합니다.

Q: 여러 페이지에 걸쳐 유지되어야 하는 장바구니 상태는 전역 클라이언트 상태인가요, 서버 상태인가요?

모범 답변: 서비스의 요구사항에 따라 달라집니다. 비로그인 사용자도 쓸 수 있는 임시 장바구니라면, 진짜 원본이 브라우저(로컬스토리지나 클라이언트 메모리)에 있으므로 전역 클라이언트 상태로 다루는 것이 맞습니다. 하지만 로그인한 사용자의 장바구니가 여러 기기에서 동기화되어야 한다면, 진짜 원본은 서버에 있어야 하므로 서버 상태로 다루고 React Query 같은 도구로 관리해야 합니다. 이 질문에 답하려면 다시 "진짜 원본이 어디에 있어야 하는가"라는 소유권 기준으로 돌아가야 하며, 두 경우를 혼동하면 여러 기기 간 장바구니 불일치 같은 버그가 생깁니다.

Q: 상태 관리 라이브러리를 새로 도입하기 전에, 어떤 질문을 먼저 해야 하나요?

모범 답변: "이 문제가 정말 새 라이브러리가 필요한 문제인가, 아니면 지금 있는 상태를 잘못된 카테고리로 다루고 있어서 생긴 문제인가"를 먼저 묻습니다. 실무에서 상태 관리 문제의 상당수는 도구 부족이 아니라 분류 실패(서버 데이터를 전역 상태에 넣거나, URL에 있어야 할 값을 컴포넌트 state에 가두거나, 파생 가능한 값을 중복 저장하는 것)에서 옵니다. 분류를 정확히 하면 대부분의 경우 이미 있는 도구(useState, URL, 서버 상태 도구)의 조합만으로 해결되고, 새 전역 상태 라이브러리가 정말 필요한 경우는 "여러 무관한 트리에서 공유되는 변경 빈도 높은 클라이언트 전용 상태"로 범위가 좁혀집니다.

15. 흔한 오해

오해 1

"모든 공유 상태는 전역 상태 관리 라이브러리에 넣어야 한다"

서버에서 온 데이터는 전역 클라이언트 상태 라이브러리가 아니라 서버 상태 전용 도구가 다뤄야 할 별도 카테고리입니다. "공유되는가"만 보고 도구를 정하면, 서버 상태 고유의 문제(재검증, 캐시 무효화, 경쟁 상태)를 놓치게 됩니다.

오해 2

"Context API는 항상 Redux보다 가벼운 대안이다"

Context는 상태 관리 라이브러리가 아니라 값 전달 메커니즘일 뿐이며, 선택적 구독을 지원하지 않아 값이 바뀔 때마다 Provider 전체를 구독하는 컴포넌트가 리렌더링됩니다. 변경 빈도가 높은 상태에 Context를 쓰면 오히려 Redux나 Zustand(선택적 구독을 지원하는)보다 성능이 나쁠 수 있습니다.

오해 3

"필터/검색 상태는 사용자 화면 안의 일이니 컴포넌트 state로 충분하다"

공유·북마크·새로고침 내구성이 필요한 상태라면 컴포넌트 state는 진짜 원본이 될 수 없습니다. 이런 상태는 URL이라는 별도의 "영속적이고 공유 가능한 저장소"에 있어야 합니다.

오해 4

"파생 값을 state로 저장해두면 렌더링마다 다시 계산하지 않아도 되니 성능에 유리하다"

파생 값을 별도 state로 저장하면 원본이 바뀔 때마다 수동으로 동기화(주로 useEffect)해야 하는 책임이 생기고, 이 동기화가 누락되면 값이 어긋나는 버그가 생깁니다. 계산 비용이 실제로 크다면 useMemo로 캐싱하는 것이 옳은 해법이며, 별도 state로 만드는 것은 정확성을 해치는 방식의 최적화입니다.

오해 5

"React Query/SWR은 그냥 fetch를 감싼 유틸리티일 뿐이다"

이 도구들의 핵심 가치는 fetch 호출 자체가 아니라, 캐시 키 기반의 중복 제거, staleness 정책, 백그라운드 재검증, 경쟁 상태 방지 같은 "서버 상태 고유의 문제"를 해결하는 상태 머신입니다. 이것을 "fetch 래퍼"로만 이해하면, 왜 굳이 라이브러리를 써야 하는지 설득력 있게 설명하지 못합니다.
