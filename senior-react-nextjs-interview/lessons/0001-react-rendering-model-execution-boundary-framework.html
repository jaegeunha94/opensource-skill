1. 핵심 면접 질문

Primary Question

"이 Next.js 컴포넌트 트리를 보여드리겠습니다. 각 컴포넌트가 언제, 어디서 실행되는지 — 빌드 타임인지, 요청 시점에 서버에서인지, 브라우저에서인지 — 설명해 주시겠어요?"

Variant 1

"React가 상태를 업데이트할 때 화면이 실제로 갱신되기까지 내부적으로 어떤 단계를 거치나요?"

Variant 2

"같은 코드인데 Server Component로 두는 것과 Client Component로 두는 것의 실행 시점과 결과물이 어떻게 다른가요?"

2. 왜 면접에서 묻는가

이 질문은 "Hook 사용법을 외웠는가"가 아니라 "React와 Next.js가 코드를 실제로 어디서, 언제 실행시키는지 정확한 모델을 갖고 있는가"를 테스트합니다. useState, useEffect를 능숙하게 쓰는 사람은 많지만, "이 줄은 서버에서 한 번 실행되고 이 줄은 사용자마다 브라우저에서 실행된다"는 것을 즉시 구분하는 사람은 훨씬 적습니다. 이 구분이 안 되면 아키텍처 설계, 성능 최적화, 장애 진단 전부가 흔들립니다.

면접관이 이 질문으로 평가하는 것:

  • 실행 경계에 대한 정확한 모델 — Server Component(서버, 요청당 1회), Client Component 초기 렌더(서버 SSR + 클라이언트 hydration), Client Component 이후 상호작용(클라이언트만)을 구분할 수 있는가.
  • 렌더링 단계 분리 이해 — React의 render 단계(순수 계산)와 commit 단계(실제 DOM 변경)가 분리되어 있다는 것을 알고, 그것이 왜 필요한지 설명할 수 있는가.
  • 비용 감각 — 어떤 코드를 서버로 옮기면 무엇이 줄고(클라이언트 번들, 클라이언트 연산), 무엇이 늘어나는지(서버 부하, 요청당 렌더링 비용)를 trade-off로 설명할 수 있는가.
  • 디버깅 가능한 사고 모델 — "이 값이 서버에서만 유효한 값인데 왜 클라이언트에서 에러가 나지?" 같은 문제를 실행 경계 개념으로 설명할 수 있는가.
  • 새 개념을 기존 지식에 연결하는 능력 — Server Component라는 비교적 최근 개념을 SSR/CSR이라는 기존 개념과 혼동하지 않고 정확히 구분할 수 있는가.

10년차 이하에서 자주 나오는 답변 패턴: "Server Component는 서버에서 렌더링되는 컴포넌트이고 Client Component는 "use client"를 붙인 컴포넌트입니다." 이 답변은 문법적 정의만 반복할 뿐, 언제 코드가 실행되는지 · 결과물이 무엇인지 · 왜 이 구분이 성능에 영향을 주는지를 설명하지 못합니다. 시니어는 "무엇을 붙이는가"가 아니라 "언제, 어디서, 몇 번 실행되고 무엇이 클라이언트로 전달되는가"로 답합니다.

3. 전제 개념 — 쉬운 한국어로

렌더링이란 무엇인가

"렌더링(rendering)"은 데이터(state, props)를 화면에 보여줄 형태(UI 트리)로 변환하는 작업입니다. 요리에 비유하면, 재료(state/props)를 가지고 실제로 먹을 수 있는 요리(화면에 그려진 UI)를 만드는 과정입니다. React에서 렌더링은 "컴포넌트 함수를 실행해서 무엇을 그릴지 계산하는 것"을 의미하며, 이것이 곧바로 화면에 그려지는 것은 아닙니다 — 이 부분은 4절에서 자세히 다룹니다.

DOM과 Virtual DOM

DOM(Document Object Model)은 브라우저가 화면에 그리기 위해 실제로 들고 있는 트리 구조입니다. DOM을 직접 조작하는 것은 비용이 큽니다 — 스타일 재계산(reflow/repaint)이 뒤따르기 때문입니다. React는 실제 DOM을 매번 통째로 새로 그리는 대신, 자바스크립트 객체로 만든 가벼운 트리(Virtual DOM)를 먼저 계산하고, 이전 트리와 비교해서 달라진 부분만 실제 DOM에 반영합니다. 이 "비교해서 달라진 부분만 반영하는" 과정을 reconciliation(재조정)이라 부르며, Day 2에서 diffing 알고리즘과 함께 깊게 다룹니다. 오늘은 "React는 매번 전체를 다시 그리지 않는다"는 사실만 기억하면 충분합니다.

서버와 클라이언트, 그리고 "실행"이 의미하는 것

웹 애플리케이션에는 코드가 실행될 수 있는 곳이 여러 군데 있습니다. 이 워크스페이스에서는 아래 세 가지를 명확히 구분합니다.

  • 빌드 타임(Build Time) — 배포하기 전, 개발자의 CI 서버에서 미리 코드를 실행해 결과(HTML, JSON)를 만들어 두는 시점. 모든 사용자가 같은 결과를 공유합니다.
  • 요청 시점 서버(Request-time Server) — 사용자가 요청을 보낼 때마다 서버에서 코드를 실행하는 시점. 사용자마다 다른 결과(로그인 여부, 개인화 데이터)를 만들 수 있지만, 요청마다 서버 자원을 소모합니다.
  • 브라우저(Client) — 사용자의 기기에서 실행되는 시점. 서버 자원은 안 쓰지만, 사용자의 CPU/네트워크(다운로드해야 할 JS 번들)를 씁니다.

이 세 곳 중 "이 코드가 어디서 실행되는가"를 헷갈리면, 브라우저 API(window, localStorage)를 서버 코드에서 쓰려다 에러를 내거나, 민감한 서버 로직(DB 접근, API 키)을 실수로 클라이언트 번들에 포함시키는 사고로 이어집니다. Day 13(Server Component 심화), Day 14(Client Component 경계)에서 이 실수를 구체적인 사례로 다룹니다.

왜 "실행 경계"를 항상 먼저 그려야 하는가

택배 비유를 다시 쓰면, "이 물건은 공장에서 미리 포장됐는지(빌드 타임), 주문할 때마다 그 자리에서 포장하는지(요청 시점 서버), 아니면 내가 집에서 직접 조립해야 하는지(클라이언트)"를 모르면 배송 지연의 원인도, 불량의 원인도 특정할 수 없습니다. React/Next.js 코드도 마찬가지로, "이 줄이 언제 실행되는가"라는 지도를 먼저 그려야 성능 문제, hydration 에러, 데이터 노출 사고를 진단할 수 있습니다.

4. React 렌더링 파이프라인 — render와 commit

React가 상태 변화를 화면에 반영하는 과정은 명확히 두 단계로 나뉩니다. 이 구분을 모르면 "왜 useEffect 안에서 상태를 바꾸면 한 번 더 렌더링이 도는지", "왜 렌더링 중에 DOM을 직접 건드리면 안 되는지" 같은 질문에 답할 수 없습니다.

단계 무엇을 하는가 특징
Render (렌더 단계) 컴포넌트 함수를 호출해서 "무엇을 화면에 그릴지"를 나타내는 트리(React Element 트리)를 계산 순수 계산에 가까워야 함. 실제 DOM은 아직 건드리지 않음. 이 단계는 중단되거나 버려질 수 있음(동시성 렌더링, Day 7에서 심화)
Commit (커밋 단계) Render 단계에서 계산한 결과를 실제 DOM에 반영하고, DOM 변경 직후 필요한 부수효과(useEffect 등)를 실행 실제 화면이 바뀌는 시점. 한 번 시작하면 중간에 멈추지 않고 끝까지 실행됨

이 구분이 실무에서 중요한 이유

Render 단계는 여러 번 호출되거나 도중에 버려질 수 있기 때문에, 컴포넌트 함수 본문에서 외부 변수를 직접 변경하거나 네트워크 요청을 날리면 예측 불가능한 부작용(중복 요청, 렌더링 중 상태 변경 경고)이 생깁니다. 반대로 "DOM에 실제로 반영된 후"에만 할 수 있는 작업(포커스 설정, 외부 라이브러리 초기화, 구독 등록)은 commit 이후에 실행되는 useEffect에서 해야 합니다. "왜 useEffect가 필요한가"라는 질문의 근본적인 답은 이 render/commit 분리에서 나옵니다.

이 파이프라인이 서버에서는 어떻게 다른가

서버에는 실제 DOM이 없습니다. 그래서 서버에서의 "렌더링"은 render 단계까지만 수행하고, 그 결과를 실제 DOM에 commit하는 대신 HTML 문자열로 직렬화합니다. 이 HTML은 브라우저로 전송되어 먼저 화면에 그려지고(비어있지 않은 첫 화면), 이후 브라우저에서 React가 다시 한번 render를 수행해 서버가 만든 HTML과 매칭시키는 것이 hydration입니다(Day 9에서 심화). 즉 hydration은 "commit을 다시 하는 것"이 아니라 "이미 그려진 DOM에 이벤트 핸들러와 내부 상태를 연결하는 것"에 가깝습니다.

5. Next.js 실행 경계 전체 지도

App Router 기준으로, 하나의 페이지를 구성하는 컴포넌트들은 실제로 다음 시점 중 하나(또는 여러 시점에 걸쳐)에서 실행됩니다. 이 지도는 이후 Day 10~15에서 각 항목을 깊게 다룰 때의 기준점 역할을 합니다.

실행 시점 대표 상황 실행 위치 클라이언트로 전달되는 것
빌드 타임 SSG 페이지, generateStaticParams로 미리 생성한 경로 CI/빌드 서버 (Node.js) 완성된 HTML/JSON (모든 사용자 공통)
요청 시점 서버 — Server Component DB 조회, 인증 확인이 들어간 컴포넌트 render 요청을 처리하는 서버(Node.js/Edge 런타임) 렌더링 결과(HTML 조각 + RSC 페이로드). 컴포넌트 소스 코드나 사용한 라이브러리 자체는 전달되지 않음
요청 시점 서버 — SSR(Client Component 최초 렌더) Client Component도 최초 진입 시에는 서버에서 한 번 HTML로 렌더링됨 요청을 처리하는 서버 정적인 초기 HTML (아직 이벤트 핸들러는 동작하지 않음)
클라이언트 — Hydration 브라우저가 서버 HTML을 받은 뒤 React를 실행해 이벤트 핸들러/상태를 연결 브라우저 (해당 없음 — 이미 받은 HTML에 "생명"을 불어넣는 단계)
클라이언트 — 상호작용 이후 렌더링 버튼 클릭, 입력, 클라이언트 상태 변화로 인한 재렌더링 브라우저 (해당 없음 — 이미 브라우저에 있는 JS가 로컬에서 실행)

이 지도가 실무에서 중요한 이유

"이 페이지가 느리다"는 리포트를 받았을 때, 위 다섯 시점 중 어디가 느린지 특정하지 않으면 엉뚱한 곳을 고치게 됩니다. 예를 들어 실제 병목이 "요청 시점 서버에서 DB 쿼리가 느린 것"인데 클라이언트 번들 최적화를 하면 효과가 없습니다. 반대로 병목이 "클라이언트 번들이 너무 커서 hydration이 늦게 끝나는 것"인데 서버 쿼리를 최적화해도 체감 속도는 그대로입니다. 이 지도를 먼저 그리는 습관이 시니어와 주니어를 가르는 지점입니다.

6. 4단계 실행 경계 판단 프레임워크

면접에서 "이 컴포넌트는 어디서 실행되나요?"라는 질문을 받았을 때, 혹은 실제 코드 리뷰에서 "이 로직을 서버/클라이언트 중 어디에 둘지" 판단할 때 시니어가 실제로 쓰는 사고 순서입니다.

  • 1
    이 코드가 서버 전용 자원을 쓰는가? (Server-only Dependency Check)

    DB 클라이언트, 파일 시스템, 비밀 API 키, 서버 환경변수를 직접 참조하는가? 하나라도 해당되면 이 코드는 반드시 서버(Server Component 또는 서버 전용 함수)에 있어야 합니다. 클라이언트로 내려가는 순간 비밀 정보가 브라우저에 노출됩니다.

  • 2
    이 코드가 브라우저 전용 API나 상호작용을 쓰는가? (Client-only Dependency Check)

    window, document, localStorage, 이벤트 핸들러(onClick), useState/useEffect 같은 상태·생명주기 Hook을 쓰는가? 하나라도 해당되면 이 부분은 Client Component여야 합니다. 서버에는 window가 없으므로 서버에서 실행하면 에러가 납니다.

  • 3
    몇 번 실행되어야 하는가? (Execution Frequency)

    이 결과가 모든 사용자에게 동일해도 되는가(→ 빌드 타임에 한 번, SSG), 요청마다 달라져야 하는가(→ 요청 시점 서버, SSR/RSC), 아니면 사용자 상호작용마다 다시 계산돼야 하는가(→ 클라이언트)? 실행 빈도를 먼저 정하면 SSR/SSG/ISR 중 무엇을 쓸지가 거의 자동으로 결정됩니다(Day 10~12에서 심화).

  • 4
    클라이언트로 무엇이, 얼마나 전달되어야 하는가? (Payload Cost)

    Server Component는 결과(HTML/RSC 페이로드)만 클라이언트로 가고 컴포넌트 코드 자체나 그 안에서 쓴 라이브러리는 번들에 포함되지 않습니다. Client Component는 코드 자체가 JS 번들로 다운로드되어야 상호작용이 가능합니다. "이 컴포넌트가 상호작용이 필요 없다면, 굳이 클라이언트 번들 비용을 낼 이유가 있는가?"를 항상 마지막에 점검합니다.

7. 실행 위치별 특성 분류표

실행 위치별로 "무엇을 할 수 있고 무엇을 할 수 없는지"를 정리한 표입니다. 이후 Day에서 각 항목을 더 깊게 다룹니다.

실행 위치 상태(state) 사용 서버 자원 접근 브라우저 API 접근 클라이언트 번들 포함 여부
Server Component 불가 (useState 사용 불가) 가능 (DB, 파일시스템, 비밀 키) 불가 포함 안 됨 — 결과만 전달
Client Component (초기 SSR) 가능하지만 서버 렌더 시점엔 초기값만 반영 원칙적으로 지양 (있어도 서버 렌더 시 한정 사용) 불가 (서버 렌더 단계이므로) 포함됨
Client Component (Hydration 이후) 가능 불가 (직접 접근 시 보안 위험) 가능 포함됨

실무 판단 예시

"상품 목록을 DB에서 읽어와 보여주기만 하는 컴포넌트"는 상태도, 브라우저 API도 필요 없으므로 Server Component가 기본 선택입니다. "좋아요 버튼처럼 클릭에 즉시 반응해야 하는 컴포넌트"는 이벤트 핸들러와 로컬 상태가 필요하므로 Client Component여야 합니다. 실무에서 흔한 실수는 페이지 전체를 "use client"로 감싸서, 상태가 필요 없는 정적인 부분까지 전부 클라이언트 번들로 내려버리는 것입니다(Day 3, Day 14에서 이 경계 설계를 집중적으로 다룹니다).

8. 답변 사고 순서

면접에서 이 순서대로 30초~1분 정도로 요약해 말하면 됩니다.

  1. 질문받은 코드/컴포넌트를 앞의 4단계 프레임워크에 통과시킵니다. 서버 전용 자원 사용 여부 → 브라우저 전용 API/상태 사용 여부 → 실행 빈도 → 클라이언트 전달 비용 순으로 점검합니다.
  2. 실행 시점을 명확한 용어로 특정합니다. "빌드 타임", "요청 시점 서버", "hydration 시점", "hydration 이후 클라이언트" 중 어디인지 명시적으로 말합니다.
  3. render/commit 분리를 근거로 왜 그렇게 동작하는지 설명합니다. 필요하다면 "React가 render 단계에서 계산하고 commit 단계에서 실제로 반영하기 때문에"라는 근거를 덧붙입니다.
  4. trade-off를 언급합니다. 서버로 옮기면 무엇이 줄고 무엇이 늘어나는지(번들 크기 ↓, 서버 부하 ↑ 등)를 함께 말합니다.
  5. 실무 사고 사례로 연결합니다. "이 경계를 잘못 그으면 hydration mismatch가 나거나, 비밀 키가 클라이언트에 노출될 수 있다"처럼 실제 장애와 연결해서 마무리합니다.

9. 핵심 답변 예시

면접관: "이 Next.js 컴포넌트 트리를 보여드리겠습니다. 각 컴포넌트가 언제, 어디서 실행되는지 설명해 주시겠어요?"

// app/products/[id]/page.tsx (Server Component, 기본값)
export default async function ProductPage({ params }) {
  const product = await db.products.findById(params.id); // ① DB 직접 접근
  return (
    <div>
      <ProductInfo product={product} />      {/* Server Component */}
      <AddToCartButton productId={product.id} /> {/* "use client" */}
    </div>
  );
}
STRONG ANSWER

먼저 ProductPage와 ProductInfo는 "use client" 지시어가 없는 기본 Server Component입니다. 이 코드는 사용자가 요청을 보낼 때마다 서버에서 실행되고, 안에서 db.products.findById로 DB에 직접 접근합니다 — 이것이 4단계 프레임워크의 1번, 서버 전용 자원 사용에 해당하므로 서버에 있어야 맞습니다.


이 두 컴포넌트는 실행 결과(HTML 조각과 RSC 페이로드)만 클라이언트로 전달되고, 컴포넌트 소스 코드나 db 클라이언트 라이브러리 자체는 브라우저 번들에 포함되지 않습니다. 즉 이 부분은 "서버에서 렌더링해서 결과만 보내는" 요청 시점 서버 실행입니다.


반면 AddToCartButton은 "use client"가 붙어 있으므로, 클릭 이벤트 핸들러와 로컬 상태(장바구니 담기 중 로딩 상태 등)가 필요한 컴포넌트일 가능성이 높습니다. 이 컴포넌트는 최초에는 서버에서 한 번 정적인 HTML로 렌더링되어 내려오지만(SSR), 브라우저가 이 HTML을 받은 뒤 React가 다시 실행되어 이벤트 핸들러를 연결하는 hydration을 거쳐야 실제로 클릭이 동작합니다. 이후 사용자가 버튼을 클릭하면 그 이후의 재렌더링은 전부 클라이언트에서만 일어납니다.


이 구조의 장점은, ProductInfo처럼 상호작용이 필요 없는 부분은 클라이언트 JS 번들에 전혀 포함되지 않아 초기 로딩 비용이 줄어든다는 것입니다. 대신 ProductPage가 매 요청마다 서버에서 실행되므로, 트래픽이 몰리면 서버 렌더링 비용이 늘어난다는 trade-off가 있습니다. 만약 상품 정보가 자주 바뀌지 않는다면, 이 부분을 SSG나 ISR로 바꿔서 요청 시점 서버 렌더링 비용 자체를 없애는 것도 고려할 수 있습니다(Day 11~12에서 다루는 주제입니다).

이 답변이 좋은 이유:

  • 각 컴포넌트를 개별적으로 짚으며 실행 시점을 명시적 용어(빌드 타임/요청 시점 서버/hydration/클라이언트)로 구분함
  • "왜 서버에 있어야 하는가"를 4단계 프레임워크(서버 전용 자원)로 근거를 댐
  • Server Component의 결과만 전달된다는 것과 Client Component는 코드 자체가 번들에 포함된다는 차이를 정확히 설명함
  • SSR과 hydration을 혼동하지 않고 "정적 HTML 전달 → 이벤트 핸들러 연결"로 순서를 구분함
  • trade-off(번들 감소 vs 서버 부하 증가)를 언급하고, 대안(SSG/ISR)까지 자연스럽게 연결함

10. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"ProductPage랑 ProductInfo는 Server Component라서 서버에서 실행되고, AddToCartButton은 "use client"가 붙어서 클라이언트에서 실행됩니다."

이 답변이 약한 이유

  • "Server Component라서 서버에서 실행된다"는 정의를 그대로 반복한 것일 뿐, 왜 서버에 있어야 하는지(DB 접근 같은 서버 전용 자원 사용) 근거가 없습니다.
  • Client Component가 "클라이언트에서만 실행된다"고 단정했는데, 실제로는 최초 진입 시 서버에서 SSR로 한 번 렌더링된 뒤 클라이언트에서 hydration을 거칩니다. 이 구분이 빠졌습니다.
  • 클라이언트로 무엇이 전달되는지(결과만 vs 코드 자체)에 대한 언급이 없어, 번들 크기에 미치는 영향을 설명하지 못합니다.
  • trade-off나 대안(SSG/ISR)에 대한 언급이 전혀 없어 "정의는 아는데 판단은 못 하는" 인상을 줍니다.
WEAK ANSWER — 또 다른 패턴

"어차피 다 렌더링되는 거니까 "use client"를 최상단에 붙이면 편하고 문제없을 것 같습니다."

이 답변이 약한 이유

  • "렌더링되니까 다 같다"는 실행 위치(서버/클라이언트)에 따른 비용 차이를 전혀 고려하지 않은 발언입니다.
  • 최상단에 "use client"를 붙이면 그 하위 트리 전체가 클라이언트 번들에 포함되는 전이(propagation) 문제를 이해하지 못하고 있습니다(Day 14에서 심화).
  • 편의성만을 기준으로 아키텍처를 결정하는 것은 시니어 레벨에서 지양해야 할 태도로 비칩니다.
STRONG ANSWER 요소 체크리스트
  • 실행 시점을 "빌드 타임 / 요청 시점 서버 / hydration / hydration 이후 클라이언트"로 명확히 구분해서 말한다
  • 서버에 있어야 하는 이유(서버 전용 자원)와 클라이언트에 있어야 하는 이유(상태/이벤트/브라우저 API)를 근거로 댄다
  • Server Component는 결과만, Client Component는 코드 자체가 번들에 포함된다는 차이를 설명한다
  • SSR과 hydration을 분리해서 순서대로 설명한다
  • trade-off와 대안(SSG/ISR 등)까지 자연스럽게 연결한다

11. Senior-level Follow-up Q&A

Q: Server Component는 왜 useState를 쓸 수 없나요?

모범 답변: useState는 리렌더링 사이에 값을 기억하고, 값이 바뀌면 해당 컴포넌트를 다시 렌더링하도록 React에 알리는 메커니즘입니다. 이 메커니즘이 동작하려면 "이 컴포넌트 인스턴스가 지금 화면에 떠 있고, 언제든 다시 실행될 수 있다"는 지속적인 런타임이 필요합니다. Server Component는 요청 하나당 한 번 실행되어 결과(HTML/RSC 페이로드)만 만들고 끝나는 실행 모델이기 때문에, "다시 렌더링을 트리거해서 화면을 갱신한다"는 개념 자체가 성립하지 않습니다. 상태가 필요한 상호작용은 반드시 지속적으로 실행되는 클라이언트 런타임, 즉 Client Component에서 처리해야 합니다.

Q: Client Component 안에서 Server Component를 직접 import해서 쓸 수 있나요?

모범 답변: 원칙적으로 Client Component 파일 안에서 Server Component를 직접 import해서 사용할 수는 없습니다. Client Component 트리 전체는 클라이언트에서 다시 실행될 수 있어야 하는데, Server Component는 서버 전용 자원(DB 등)에 의존할 수 있으므로 클라이언트에서 재실행이 불가능하기 때문입니다. 대신 Server Component를 Client Component의 children이나 다른 prop으로 "이미 렌더링된 결과물"로 넘겨주는 패턴(composition pattern)을 사용합니다. 이렇게 하면 Server Component는 서버에서 미리 렌더링되고, Client Component는 그 결과를 그대로 배치만 하는 역할을 하게 됩니다. 이 패턴은 Day 13에서 구체적인 코드와 함께 다룹니다.

Q: "이 페이지 첫 로딩이 느려요"라는 리포트를 받으면 이 실행 경계 지도를 어떻게 활용해서 원인을 좁히시겠어요?

모범 답변: 먼저 느린 구간이 "서버가 응답을 만들기까지(TTFB)"인지 "응답은 빨리 왔는데 화면이 상호작용 가능해지기까지(hydration 완료 시점)"인지를 구분합니다. TTFB가 느리면 요청 시점 서버 실행 단계(Server Component의 DB 쿼리, 외부 API 호출)를 의심하고, TTFB는 빠른데 클릭이 한동안 안 먹히면 클라이언트 번들 크기나 hydration 비용(Client Component가 과도하게 많은 경우)을 의심합니다. 이 두 구간을 먼저 나누지 않고 무작정 캐싱을 걸거나 코드를 최적화하면 엉뚱한 곳을 고치게 됩니다. Network 탭에서 문서 응답 시간과 Performance 탭에서 hydration/Long Task 구간을 각각 확인하는 것이 실제 절차입니다.

Q: render 단계가 "여러 번 호출되거나 도중에 버려질 수 있다"는 게 무슨 의미이고, 왜 중요한가요?

모범 답변: React의 동시성 렌더링 모델에서는, 더 급한 업데이트(예: 사용자 입력)가 들어오면 진행 중이던 낮은 우선순위의 render 계산을 중단하고 버린 뒤 다시 계산할 수 있습니다(Day 7의 transition에서 심화). 만약 컴포넌트 함수 본문에서 외부 변수를 직접 변경하거나(mutation), 네트워크 요청 같은 부수효과를 직접 실행한다면, render가 여러 번 불리거나 버려질 때 그 부수효과도 여러 번 실행되거나 일관성 없이 실행될 수 있습니다. 그래서 React는 컴포넌트 본문을 "순수 함수처럼" 작성하라고 강제하고, 부수효과는 commit 이후에만 실행이 보장되는 useEffect로 분리하도록 설계했습니다.

Q: SSG로 만든 정적 HTML과 Server Component가 만드는 HTML은 근본적으로 뭐가 다른가요?

모범 답변: 겉보기 결과물(HTML)은 비슷해 보일 수 있지만 "언제 만들어지는가"가 다릅니다. SSG는 빌드 타임에 딱 한 번 실행되어 모든 사용자가 동일한 정적 파일을 받습니다. Server Component는 App Router에서 기본적으로 요청이 올 때마다(혹은 캐시 설정에 따라) 서버에서 실행될 수 있는 실행 모델이며, 사용자별로 다른 데이터(로그인 정보 등)를 반영한 결과를 만들 수 있습니다. 다만 Server Component도 캐싱 설정에 따라 사실상 SSG처럼 동작하게 만들 수 있습니다 — 이 부분은 Day 16(캐싱 계층)에서 fetch 캐시 옵션과 함께 자세히 다룹니다. 핵심은 "SSG/SSR/RSC는 서로 배타적인 카테고리가 아니라, 같은 실행 모델 위에서 캐싱 전략으로 조합되는 것"이라는 시각입니다.

12. 흔한 오해

오해 1

"Server Component는 그냥 예전 SSR을 이름만 바꾼 것이다"

SSR은 "HTML을 서버에서 미리 만들어 보낸다"는 렌더링 결과물의 전달 방식에 가깝고, 전통적인 SSR에서는 결국 클라이언트에서도 같은 컴포넌트 코드가 다시 실행(hydration)됩니다. Server Component는 애초에 클라이언트에서 실행될 필요가 없는 컴포넌트라는 개념이며, 코드 자체가 클라이언트 번들에 포함되지 않습니다. 즉 SSR은 "언제/어떻게 HTML을 만드는가"의 문제이고, Server/Client Component는 "이 코드가 애초에 클라이언트에 존재해야 하는가"의 문제로, 서로 다른 축입니다.

오해 2

"state 업데이트는 화면을 즉시 바꾼다"

state 업데이트 함수를 호출하는 순간 화면이 바뀌는 것이 아니라, React가 이를 예약해두고 render 단계에서 새 트리를 계산한 뒤 commit 단계에서 실제 DOM에 반영합니다. 같은 이벤트 핸들러 안에서 여러 번 state를 업데이트해도 대부분 batching(일괄 처리)되어 render/commit이 한 번만 일어납니다. "state를 바꾸면 즉시 그 값을 다시 읽을 수 있다"고 오해하면 이벤트 핸들러 안에서 업데이트 직후 같은 state 변수를 읽고 옛날 값이 나오는 것에 당황하게 됩니다.

오해 3

""use client"는 그 컴포넌트만 클라이언트에서 실행되게 만든다"

"use client"는 그 파일과 거기서 import되어 사용되는 모든 하위 컴포넌트를 클라이언트 번들 경계 안으로 포함시킵니다. 즉 최상위 근처에 "use client"를 붙이면 그 아래 트리 전체가 클라이언트 번들에 딸려 들어가는 전이(propagation) 효과가 있습니다. 이 오해 때문에 실무에서 클라이언트 번들이 불필요하게 커지는 경우가 매우 흔합니다(Day 14에서 구체적인 경계 설계를 다룹니다).

오해 4

"서버에서 렌더링하면 무조건 더 빠르다"

서버 렌더링은 첫 화면(첫 HTML)을 더 빨리 보여줄 수 있다는 장점이 있지만, 요청마다 서버 자원을 소모합니다. 트래픽이 많은데 캐싱 전략 없이 매 요청마다 무거운 서버 렌더링을 수행하면 오히려 서버가 병목이 되어 전체적으로 느려질 수 있습니다. "빠르다/느리다"는 항상 무엇을 기준으로(TTFB인지, 상호작용 가능 시점인지) 어떤 트래픽 패턴에서인지를 함께 말해야 정확한 답변이 됩니다.

오해 5

"Virtual DOM을 쓰기 때문에 React가 항상 순수 DOM 조작보다 빠르다"

Virtual DOM/reconciliation은 "무엇이 바뀌었는지 몰라도 되게 해주는" 개발 편의성과, "매번 전체를 다시 그리는 순진한 방식"보다 나은 성능을 제공하는 장치이지, 손으로 최적화한 직접적인 DOM 조작보다 항상 빠르다는 뜻은 아닙니다. React의 장점은 "적당히 신경 쓰지 않아도 대체로 충분히 빠른 결과를 자동으로 얻을 수 있다"는 데 있습니다. 이 세부 내용은 Day 2(Reconciliation)에서 diffing 알고리즘과 함께 다룹니다.
