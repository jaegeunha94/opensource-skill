1. 핵심 면접 질문

Primary Question

"검색창에 타이핑할 때마다 아래에 있는 무거운 결과 리스트를 다시 렌더링하는데, 그럴 때마다 입력창 자체가 버벅입니다(키를 눌러도 글자가 늦게 나타남). 원인과 해결책을 설명해 주세요."

Variant 1

"Suspense가 정확히 무엇을 '기다리는' 것인가요? 그냥 로딩 스피너를 보여주는 컴포넌트 아닌가요?"

Variant 2

"useTransition과 useDeferredValue는 둘 다 '급하지 않은 업데이트'를 다루는 것 같은데 차이가 뭔가요? 실무에서 어떤 기준으로 골라 쓰나요?"

2. 왜 면접에서 묻는가

이 질문은 "Suspense 문법을 쓸 줄 아는가"가 아니라, "렌더링에는 우선순위가 있고, 급한 업데이트와 급하지 않은 업데이트를 구분해서 설계할 수 있는가"를 테스트합니다. React 18 이전에는 렌더링이 한번 시작하면 끝까지 멈출 수 없는 하나의 동기 작업이었습니다. 동시성 렌더링은 이 전제 자체를 깨는 변화이고, 이걸 "그냥 새로운 Hook 두 개가 추가됐다"로 이해하면 왜 특정 상황에서만 성능이 개선되고 다른 상황에서는 아무 효과가 없는지 설명하지 못합니다.

면접관이 이 질문으로 평가하는 것:

  • "렌더링을 중단하고 버릴 수 있다"는 개념을 정확히 아는가 — React가 렌더링 중간에 더 급한 업데이트가 들어오면 진행 중이던 렌더링 작업을 버리고 급한 것부터 처리한 뒤 다시 시작할 수 있다는, 기존의 "한번 시작하면 끝까지 간다"는 사고방식을 뒤집는 모델을 이해하는가.
  • Suspense를 "로딩 스피너 컴포넌트"가 아니라 "비동기 경계를 선언하는 메커니즘"으로 이해하는가 — 자식 컴포넌트가 "아직 준비 안 됐다"고 선언하면 그 사실이 트리를 타고 올라가 가장 가까운 Suspense가 대신 fallback을 보여준다는 전파 구조를 아는가.
  • transition과 deferredValue의 역할 차이를 아는가 — 하나는 "내가 트리거하는 상태 업데이트"에, 다른 하나는 "외부에서 이미 들어온 값을 지연 반영하는 것"에 쓰인다는 주체의 차이를 구분하는가.
  • 실제 프로덕션 증상과 연결할 수 있는가 — "입력이 버벅인다", "탭을 전환할 때 화면이 잠깐 멈춘다" 같은 체감 성능 문제를 렌더링 우선순위 문제로 진단하고, 구체적인 API로 고칠 수 있는가.
  • 만능이 아니라는 것을 아는가 — transition을 아무 데나 붙인다고 무조건 빨라지는 게 아니라, 오히려 잘못 쓰면 사용자에게 필요한 즉각적 피드백(예: 클릭 후 버튼 상태 변화)까지 지연시킬 수 있다는 trade-off를 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "리스트 컴포넌트를 React.memo로 감싸면 될 것 같습니다." 이 답변은 문제의 원인이 "불필요한 재렌더링"이 아니라 "필요한 재렌더링이 너무 무거워서 메인 스레드를 오래 점유하는 것"이라는 점을 놓치고 있습니다. 리스트는 실제로 매번 새로운 검색어에 맞춰 다시 그려져야 하므로 memo로 막을 수 있는 성격의 문제가 아닙니다. 시니어는 "이 렌더링을 막을 수 있는가"가 아니라 "이 렌더링의 우선순위를 낮춰서 입력 반영을 먼저 끝낼 수 있는가"를 묻습니다.

3. 전제 개념 — 쉬운 한국어로

"메인 스레드를 점유한다"는 것이 무슨 뜻인가

브라우저는 자바스크립트 실행, 화면 그리기, 사용자 입력 처리를 기본적으로 하나의 스레드(메인 스레드)에서 순서대로 처리합니다. 만약 자바스크립트 코드 하나가 계산을 끝내는 데 200밀리초가 걸린다면, 그 200밀리초 동안 브라우저는 키보드 입력을 반영한 화면도 못 그리고 클릭 이벤트도 처리하지 못합니다. "입력창이 버벅인다"는 현상은 대부분 이것입니다 — 자바스크립트(여기서는 리스트를 다시 그리는 React 렌더링)가 메인 스레드를 길게 붙잡고 있어서, 사용자가 키를 눌러도 그 입력이 화면에 반영될 틈이 없는 것입니다.

"렌더링은 중단 가능하다"는 것의 의미

React 17까지의 렌더링은 한번 시작하면 트리 전체를 끝까지 계산할 때까지 멈추지 않는 동기적인 작업이었습니다. React 18의 Fiber 아키텍처는 이 계산을 작은 단위(Fiber 노드 단위)로 쪼개서, 각 단위를 처리한 뒤 "지금 더 급한 일이 생겼는가?"를 확인할 수 있는 지점을 마련했습니다. 급한 일(예: 사용자가 방금 누른 키)이 생기면 React는 진행 중이던(아직 끝나지 않은) 렌더링 작업을 버리고 급한 작업을 먼저 처리한 뒤, 필요하면 버려졌던 렌더링을 처음부터 다시 계산합니다. 이것이 "interruptible rendering(중단 가능한 렌더링)"입니다. 중요한 것은 이 "버림"이 render 단계에서만 가능하고, 화면에 실제로 반영하는 commit 단계는 한번 시작하면 중단할 수 없다는 것입니다(Day 1에서 다룬 render/commit 분리가 여기서 다시 등장합니다).

"우선순위(priority)"란 무엇인가

React 18은 모든 업데이트를 똑같이 취급하지 않습니다. 클릭이나 타이핑처럼 사용자가 직접 일으킨 입력은 "긴급(urgent)" 업데이트로 분류되어 최우선으로 처리됩니다. 반면 개발자가 명시적으로 startTransition으로 감싼 업데이트는 "긴급하지 않음(non-urgent, transition)"으로 분류되어, 긴급한 업데이트가 없을 때만 처리됩니다. 이것은 운영체제의 프로세스 스케줄링과 비슷합니다 — 급한 작업이 CPU를 먼저 쓰고, 급하지 않은 작업은 여유가 생겼을 때 처리되거나 중간에 밀려납니다.

"Promise를 throw한다"는 낯선 개념

Suspense의 내부 동작을 이해하려면 "컴포넌트가 렌더링 도중에 에러 대신 Promise를 throw할 수 있다"는 사실을 알아야 합니다. 일반적으로 throw는 에러를 던져서 프로그램 실행을 중단시키는 데 씁니다. Suspense가 지원하는 데이터소스(예: use() Hook, 프레임워크가 통합한 fetch)는 데이터가 아직 준비되지 않았을 때 "에러가 아니라 아직 끝나지 않은 Promise"를 던집니다. React는 이 Promise를 일반 에러와 다르게 취급해서, "이 컴포넌트는 아직 렌더링할 수 없으니 잠시 후 이 Promise가 끝나면 다시 시도하겠다"고 이해하고, 그 사이에는 가장 가까운 Suspense의 fallback을 대신 보여줍니다.

4. Suspense가 실제로 하는 일

<Suspense fallback={...}>는 "이 경계 안의 어떤 자식이든 아직 준비되지 않았다고 선언하면, 그 선언이 트리를 타고 이 경계까지 올라와서 자식 대신 fallback을 보여주겠다"는 선언적 계약입니다.

function ProfilePage() {
  return (
    <Suspense fallback={<Spinner />}>
      <ProfileDetails />   {/* 데이터가 없으면 이 지점에서 "suspend"됨 */}
    </Suspense>
  );
}

ProfileDetails가 렌더링 도중 아직 도착하지 않은 데이터에 접근하려고 하면(예: use(fetchProfile())), 그 컴포넌트는 "suspend"됩니다. React는 이 신호를 받아 ProfileDetails의 렌더링 결과를 버리고, 대신 가장 가까운 부모 Suspense의 fallback을 화면에 커밋합니다. 데이터가 준비되면(Promise가 resolve되면) React는 ProfileDetails를 처음부터 다시 렌더링합니다 — 이때는 데이터가 이미 있으므로 suspend되지 않고 끝까지 렌더링이 완료되어 fallback 대신 실제 내용이 커밋됩니다.

핵심은 이것입니다: Suspense는 "로딩 상태를 보여주는 컴포넌트"가 아니라, "아직 준비되지 않은 자식이 있을 때 그 사실을 처리할 책임을 가장 가까운 조상에게 위임하는 통신 메커니즘"입니다. 이 덕분에 ProfileDetails 컴포넌트 자체는 isLoading 같은 상태 변수나 if (loading) return <Spinner /> 분기를 직접 관리할 필요가 없습니다 — 로딩 상태 처리를 컴포넌트 트리 구조 자체로 표현하는 것이 Suspense의 설계 의도입니다.

5. 동시성 렌더링 — "중단 가능한 렌더링"이라는 모델

동시성 렌더링(concurrent rendering)은 병렬 처리(여러 스레드에서 동시에 실행)를 의미하는 것이 아닙니다. 자바스크립트는 여전히 싱글 스레드로 실행됩니다. 동시성이 의미하는 것은 React가 렌더링 작업을 여러 개의 작은 단위로 쪼개고, 그 사이사이에 "지금 이 작업을 계속할지, 중단하고 다른 작업을 먼저 처리할지" 판단할 수 있는 지점을 만들었다는 것입니다.

function SearchPage() {
  const [query, setQuery] = useState("");
  const [results, setResults] = useState([]);

  function handleChange(e) {
    const value = e.target.value;
    setQuery(value);          // 긴급 업데이트 — 입력창 글자는 즉시 반영되어야 함
    startTransition(() => {
      setResults(filterHugeList(value)); // 긴급하지 않은 업데이트 — 늦게 반영되어도 괜찮음
    });
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      <HeavyResultList results={results} />
    </>
  );
}

이 코드에서 setQuery는 감싸지 않았기 때문에 긴급 업데이트로 처리되어 즉시 커밋됩니다(입력창에 글자가 바로 나타남). setResults는 startTransition으로 감쌌기 때문에 긴급하지 않은 업데이트로 표시됩니다. 사용자가 다음 키를 계속 누르면, React는 아직 끝나지 않은(진행 중인) HeavyResultList 렌더링을 버리고 새로 들어온 입력(긴급 업데이트)을 먼저 처리합니다. 타이핑이 멈추고 여유가 생기면 그제서야 마지막 startTransition 업데이트가 끝까지 렌더링되어 화면에 반영됩니다.

이것이 없던 시절(React 17 이하)에는 어떻게 됐나

setQuery와 setResults를 같은 이벤트 핸들러 안에서 호출하면, 둘 다 구분 없이 하나의 동기적 렌더링으로 처리됩니다. HeavyResultList가 무거우면 이 렌더링 계산이 끝날 때까지 브라우저는 다음 키 입력을 화면에 반영할 수 없습니다. 즉 입력창의 글자 자체가 리스트 렌더링이 끝날 때까지 늦게 나타나는 현상이 발생합니다. 동시성 렌더링은 바로 이 문제 — "급하지 않은 계산이 급한 업데이트를 막는 것" — 을 해결하기 위한 것입니다.

6. useTransition / startTransition — 우선순위를 낮추는 도구

const [isPending, startTransition] = useTransition();

function handleChange(e) {
  const value = e.target.value;
  setQuery(value); // 긴급
  startTransition(() => {
    setResults(filterHugeList(value)); // 긴급하지 않음, isPending이 true가 되는 동안 진행
  });
}

useTransition은 "내가 직접 트리거하는 상태 업데이트"를 긴급하지 않은 것으로 표시할 때 씁니다. isPending은 그 낮은 우선순위 작업이 아직 끝나지 않았음을 나타내는 boolean이며, 이걸로 "업데이트 중" 같은 은은한 표시(예: 흐림 처리, 작은 스피너)를 보여줄 수 있습니다. 중요한 것은 isPending이 true인 동안에도 화면이 완전히 멈추지 않고, 이전 화면(오래된 results)이 계속 상호작용 가능한 상태로 보인다는 것입니다 — 이것이 "화면이 멈추는 것"과 "긴급하지 않은 업데이트가 뒤로 밀리는 것"의 차이입니다.

startTransition은 "느리게 만드는" 도구가 아니라 "새치기당해도 괜찮다고 표시하는" 도구입니다. 만약 아무 긴급 업데이트도 들어오지 않는다면, transition으로 감싼 업데이트는 지연 없이 곧바로 처리됩니다. 지연이 발생하는 것은 오직 그 사이에 더 긴급한 업데이트가 끼어들 때뿐입니다.

7. useDeferredValue — 값 자체를 지연시키는 도구

function SearchPage({ query }) { // query는 부모나 URL에서 내려온 값 — 내가 setState를 직접 호출하지 않음
  const deferredQuery = useDeferredValue(query);

  return (
    <>
      <input value={query} />
      {/* deferredQuery는 query보다 "뒤늦게" 따라감 — 그 사이엔 이전 값으로 렌더링 유지 */}
      <HeavyResultList query={deferredQuery} />
    </>
  );
}

useDeferredValue는 내가 상태 업데이트를 직접 트리거하는 게 아니라, 이미 어딘가에서 들어온 값을 "지연된 사본"으로 만들어 쓰는 상황에 씁니다. 예를 들어 query가 부모 컴포넌트의 props로 내려오거나 URL 쿼리 파라미터에서 읽어온 값이라면, 그 값을 만드는 setState 호출 자체를 내가 통제할 수 없습니다(startTransition으로 감쌀 지점이 없습니다). 이럴 때 useDeferredValue는 "이 값이 바뀌어도, 무거운 하위 트리는 즉시 따라가지 말고 여유가 생겼을 때 따라가라"고 지시합니다. 그동안 HeavyResultList는 이전 deferredQuery 값으로 렌더링된 상태를 유지합니다.

실무에서 자주 쓰는 패턴 — 오래된 값인지 표시하기

const deferredQuery = useDeferredValue(query);
const isStale = query !== deferredQuery;

return (
  <div style={{ opacity: isStale ? 0.6 : 1 }}>
    <HeavyResultList query={deferredQuery} />
  </div>
);

query(최신 값)와 deferredQuery(지연된 값)가 다르다는 것은 아직 리스트가 최신 상태를 따라가지 못했다는 뜻이므로, 이를 이용해 "갱신 중" 느낌의 UI를 별도의 isPending 상태 없이 만들 수 있습니다.

8. 언제 transition, 언제 deferredValue

기준 useTransition useDeferredValue
통제 대상 내가 호출하는 setState 자체 이미 존재하는(내가 만든 게 아닌) 값의 반영 시점
전형적 상황 버튼 클릭 → 탭 전환, 필터 변경 등 이벤트 핸들러 내부 props로 내려온 값, URL 파라미터, 다른 라이브러리가 준 값
진행 상태 표시 isPending boolean 제공 직접 값 비교(value !== deferredValue)로 판단해야 함
내부 동작 콜백 안의 업데이트를 낮은 우선순위로 스케줄 내부적으로 낮은 우선순위의 "따라가는 사본" 상태를 만듦(구현상 transition과 유사한 스케줄링 사용)

한 문장으로 구분하면: "내가 이 업데이트를 일으키는 setState 호출부를 갖고 있는가?" 그렇다면 useTransition, 아니라면(값만 받아서 쓰는 입장이라면) useDeferredValue입니다.

9. Suspense의 두 가지 용도 — code splitting과 data fetching

Suspense는 하나의 메커니즘이지만 실무에서는 두 가지 서로 다른 상황에 쓰입니다. 면접에서 이 둘을 구분하지 못하면 "Suspense는 데이터 로딩에만 쓰는 것"이라는 오해를 드러내게 됩니다.

용도 1 — 코드 분할 (React.lazy)

const HeavyChart = React.lazy(() => import("./HeavyChart"));

function Dashboard() {
  return (
    <Suspense fallback={<Spinner />}>
      <HeavyChart />
    </Suspense>
  );
}

여기서 기다리는 대상은 서버 데이터가 아니라 이 컴포넌트의 자바스크립트 코드 자체를 담은 청크 파일을 네트워크로 내려받는 것입니다. 파일 다운로드가 끝날 때까지 Suspense가 fallback을 보여줍니다. Day 18(번들 최적화)에서 다시 다룹니다.

용도 2 — 데이터 페칭 (Server Component, use())

async function ProfileDetails({ userId }) {
  const profile = await getProfile(userId); // 서버 컴포넌트에서 await하는 것 자체가 suspend 지점
  return <div>{profile.name}</div>;
}

function Page({ userId }) {
  return (
    <Suspense fallback={<Spinner />}>
      <ProfileDetails userId={userId} />
    </Suspense>
  );
}

Next.js App Router에서는 async 서버 컴포넌트가 await하는 지점 자체가 Suspense의 suspend 신호로 연결됩니다. 이것이 streaming SSR(Day 17에서 다룸)의 기반입니다 — 서버는 이 컴포넌트를 기다리는 동안 페이지의 나머지 부분(다른 Suspense 경계 밖의 내용)을 먼저 클라이언트로 보내고, 이 컴포넌트가 준비되면 스트림으로 이어서 보냅니다.

두 용도의 공통점: "이 하위 트리가 지금 당장 렌더링을 완성할 수 없다"는 신호를 던지고, 그 처리를 조상의 Suspense에게 위임한다는 점에서 동일한 메커니즘입니다. 다른 것은 그 신호가 "코드가 아직 없어서"인지 "데이터가 아직 없어서"인지뿐입니다.

10. Suspense boundary 배치 전략

Suspense를 어디에 두느냐는 "그냥 감싸면 되는" 문제가 아니라, 사용자가 로딩 상태를 어떤 단위로 체감하게 할 것인가를 결정하는 설계입니다.

안티패턴 — 페이지 전체를 하나의 Suspense로 감싸기

<Suspense fallback={<FullPageSpinner />}>
  <Header />      {/* 데이터 필요 없음 */}
  <ProfileInfo /> {/* 데이터 필요 */}
  <OrderHistory /> {/* 데이터 필요, 오래 걸림 */}
</Suspense>

이 구조는 Header처럼 데이터가 필요 없어 즉시 그릴 수 있는 부분까지도, 가장 느린 OrderHistory가 끝날 때까지 화면에 아무것도 보여주지 못하게 만듭니다. Suspense 경계가 하나뿐이면 그 안에서 가장 느린 컴포넌트가 전체 화면의 체감 속도를 결정합니다.

개선 — 세분화된 Suspense 경계

<Header /> {/* Suspense 밖 — 즉시 렌더링 */}
<Suspense fallback={<ProfileSkeleton />}>
  <ProfileInfo />
</Suspense>
<Suspense fallback={<OrderSkeleton />}>
  <OrderHistory /> {/* 느린 부분만 별도로 격리 */}
</Suspense>

이렇게 하면 Header는 즉시 보이고, ProfileInfo와 OrderHistory는 각자 준비되는 대로 독립적으로 화면에 나타납니다(streaming SSR과 결합하면 이 각각이 서버에서 스트림으로 순서대로 전달됩니다). 사용자는 "페이지가 통째로 멈춰 있다"가 아니라 "빈 부분이 하나씩 채워진다"는 체감을 얻습니다.

설계 원칙: Suspense 경계는 "데이터를 가져오는 단위"가 아니라 "로딩 상태를 사용자에게 어떤 덩어리로 보여주고 싶은가" 기준으로 나눕니다. 서로 관련 없는 정보(헤더와 주문 내역)를 같은 경계에 묶으면 느린 쪽이 빠른 쪽의 발목을 잡고, 반대로 지나치게 잘게 쪼개면 화면 곳곳에서 스켈레톤이 순서 없이 팝콘처럼 튀는 "layout shift 누적" 문제가 생깁니다.

11. 판단 기준표

상황 판단 근거
무거운 리스트/차트 렌더링이 입력창 반응을 막음 리스트를 업데이트하는 setState를 startTransition으로 감쌈 입력은 긴급, 리스트 재계산은 긴급하지 않음으로 분리
props/URL로 받은 값이 바뀔 때마다 무거운 하위 트리가 버벅임 useDeferredValue로 지연된 사본을 하위 트리에 전달 값의 원천을 내가 통제할 수 없어 startTransition 적용 지점이 없음
코드 청크가 커서 초기 로딩이 느림 React.lazy + Suspense로 코드 분할 필요한 시점에만 코드를 내려받도록 지연
서버 컴포넌트 중 일부만 응답이 느림 느린 부분만 별도 Suspense로 감싸 스트리밍 빠른 부분이 느린 부분 때문에 대기하지 않도록 격리
버튼 클릭 직후 즉각적인 시각 피드백(눌림 효과 등)이 필요 그 상태 업데이트는 transition으로 감싸지 않음 즉각 반응해야 하는 업데이트를 긴급하지 않음으로 낮추면 오히려 UX 저하

12. 증상 기반 진단표

증상 의심할 원인 확인 방법
키를 눌러도 입력창 글자가 늦게 나타남 같은 이벤트 핸들러 안의 무거운 setState가 긴급 업데이트와 구분되지 않고 동기 처리됨 React DevTools Profiler로 렌더링 소요 시간 확인 → 무거운 업데이트를 startTransition으로 격리
버튼을 눌러도 잠시 아무 반응이 없다가 갑자기 화면이 바뀜(눌림 피드백 자체가 없음) 즉시 피드백이 필요한 상태 업데이트까지 startTransition으로 감쌈 어떤 상태가 "즉각 보여야 하는 결과"인지 재검토 → 그 부분은 transition 밖으로 분리
페이지 전체가 로딩 스피너 하나로 오래 멈춰 있음 Suspense 경계가 하나로 뭉쳐 있어 가장 느린 컴포넌트가 전체를 막음 느린 부분만 별도 Suspense로 분리해 나머지는 먼저 보이게 함
스켈레톤 UI가 화면 곳곳에서 순서 없이 나타났다 사라지며 레이아웃이 계속 흔들림 Suspense 경계를 지나치게 잘게 쪼개 각자 다른 시점에 팝인(pop-in)됨 연관된 정보를 하나의 경계로 묶어 팝인 횟수를 줄임
isPending이 true인데도 화면이 완전히 멈춘 것처럼 보임 transition으로 감싼 업데이트가 실제로는 매우 무거워 낮은 우선순위로도 오래 걸림 우선순위 문제가 아니라 알고리즘 자체의 최적화(가상화, 메모이제이션)가 필요한지 재검토

13. 답변 사고 순서

  1. 증상의 원인을 렌더링 우선순위 문제로 규정합니다. "입력창이 버벅이는 것은 리스트를 다시 그리는 계산이 메인 스레드를 오래 점유해서, 그 사이에 들어온 키 입력이 화면에 반영될 틈이 없기 때문입니다."
  2. 어떤 업데이트가 긴급하고 어떤 게 아닌지 구분합니다. "입력창의 글자 자체(setQuery)는 사용자가 친 키를 즉시 보여줘야 하는 긴급 업데이트이고, 그 아래 리스트를 다시 계산하는 것(setResults)은 몇 프레임 늦게 반영돼도 사용자가 알아채기 어려운 긴급하지 않은 업데이트입니다."
  3. 구체적인 API로 연결합니다. "setResults 호출을 startTransition으로 감싸서, React가 더 급한 입력 업데이트를 우선 처리하고 리스트 업데이트는 여유가 있을 때 처리하도록 만들겠습니다."
  4. 제어 주체에 따라 도구를 재확인합니다. "만약 검색어가 제가 직접 관리하는 상태가 아니라 URL 파라미터나 부모로부터 내려온 props라면, startTransition을 걸 지점이 없으므로 대신 useDeferredValue로 리스트에 전달되는 값 자체를 지연시키겠습니다."
  5. 한계와 대안을 언급합니다. "다만 리스트 자체의 렌더링 비용이 지나치게 크다면 우선순위 조정만으로는 한계가 있으므로, 가상화(virtualization)나 페이지네이션으로 실제 렌더링해야 하는 항목 수 자체를 줄이는 것도 함께 검토하겠습니다."

14. 핵심 답변 예시

면접관: "검색창에 타이핑할 때마다 아래에 있는 무거운 결과 리스트를 다시 렌더링하는데, 그럴 때마다 입력창 자체가 버벅입니다. 원인과 해결책을 설명해 주세요."

STRONG ANSWER

이 증상의 원인은 자바스크립트가 싱글 스레드로 동작한다는 전제에서 나옵니다. 검색어가 바뀔 때마다 무거운 리스트를 다시 계산하는 렌더링이 메인 스레드를 길게 점유하면, 그 계산이 끝나기 전까지 브라우저는 사용자가 그다음 누른 키 입력을 화면에 반영할 기회를 얻지 못합니다. 결과적으로 입력창의 글자 자체가 늦게 나타나는 것처럼 느껴집니다.


해결책은 두 상태 업데이트의 긴급도를 구분하는 것입니다. 입력창에 표시할 query 상태는 사용자가 친 글자를 즉시 반영해야 하는 긴급 업데이트로 그대로 두고, 리스트를 갱신하는 results 상태 업데이트는 startTransition으로 감싸서 React에게 "이 업데이트는 긴급하지 않다"고 알리겠습니다. 이렇게 하면 사용자가 계속 타이핑하는 동안 React는 진행 중이던 리스트 렌더링을 중단하고 새로 들어온 입력을 먼저 처리한 뒤, 타이핑이 멈추고 여유가 생겼을 때 마지막 입력 기준으로 리스트를 완성해 커밋합니다.


만약 검색어가 제가 직접 setState로 관리하는 값이 아니라 URL 쿼리 파라미터처럼 외부에서 들어온 값이라면, startTransition을 걸 지점 자체가 없으므로 대신 useDeferredValue로 리스트에 넘기는 값을 지연시키는 방식을 쓰겠습니다. 그리고 이 방법은 렌더링 우선순위를 조정하는 것이지 렌더링 자체의 비용을 줄이는 것은 아니므로, 리스트 항목이 매우 많다면 가상화 같은 근본적인 렌더링 비용 절감도 함께 고려하겠습니다.

이 답변이 좋은 이유:

  • 증상을 "왜 발생하는가"(싱글 스레드, 우선순위 부재)부터 설명한 뒤 해결책으로 넘어가는 순서를 지킴
  • 두 가지 도구(startTransition, useDeferredValue)를 상황(내가 트리거하는가 여부)에 맞게 구분해서 제시함
  • 우선순위 조정과 렌더링 비용 절감을 서로 다른 문제로 구분해, 만능 해법이 아니라는 것을 명시함

15. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"리스트 컴포넌트를 React.memo로 감싸고 useCallback으로 핸들러를 고정하면 재렌더링을 막을 수 있습니다."

이 답변이 약한 이유

  • 이 문제의 본질은 "불필요한 재렌더링"이 아니라 "필요한 재렌더링이 너무 무거워 메인 스레드를 막는 것"입니다. 검색어가 바뀔 때마다 리스트는 실제로 새 결과를 보여줘야 하므로, memo로 렌더링을 막으면 오히려 리스트가 갱신되지 않는 버그가 됩니다.
  • 렌더링을 "막는 것"과 "우선순위를 낮추는 것"을 구분하지 못하고 있습니다.
WEAK ANSWER — 또 다른 패턴

"디바운스(debounce)를 걸어서 타이핑이 끝난 후 300ms 뒤에 리스트를 업데이트하면 됩니다."

이 답변이 약한 이유

  • 디바운스는 유효한 보완 기법이지만, "고정된 시간만큼 무조건 기다리는 것"이라 사용자가 잠깐 멈췄다가 다시 타이핑해도 불필요하게 지연이 발생하고, 반대로 리스트가 실제로는 매우 가벼운 경우에도 항상 지연을 강제합니다. transition은 실제로 더 긴급한 작업이 있을 때만 밀리고, 없으면 즉시 처리된다는 점에서 더 적응적(adaptive)입니다.
  • 면접관은 "임의의 시간을 정해서 미루는 기법"과 "React의 렌더링 스케줄러가 우선순위 기반으로 판단하게 하는 기법"의 차이를 아는지 보고 싶어합니다. 실무에서는 둘을 함께 쓰기도 하지만, 이 둘을 같은 것으로 착각하면 안 됩니다.
STRONG ANSWER 요소 체크리스트
  • 증상의 근본 원인을 "싱글 스레드가 긴 계산에 점유되는 것"으로 정확히 짚는다
  • 업데이트를 긴급/비긴급으로 나누는 사고를 보여주고, 그 판단에 맞는 구체적 API(startTransition vs useDeferredValue)를 상황별로 구분한다
  • 우선순위 조정이 렌더링 비용 자체를 줄이는 것은 아니라는 한계를 명시하고, 필요하면 가상화 같은 보완책을 함께 제시한다
  • 즉각적 피드백이 필요한 업데이트까지 transition으로 감싸면 안 된다는 trade-off를 안다

16. Senior-level Follow-up Q&A

Q: startTransition 안에서 상태 업데이트가 정말 "느리게" 실행되나요, 아니면 단지 우선순위만 낮아지는 건가요?

모범 답변: 후자입니다. startTransition은 지연 시간을 강제로 추가하는 것이 아니라, 해당 업데이트를 "새치기당해도 되는" 낮은 우선순위로 표시할 뿐입니다. 만약 그 사이에 아무런 긴급 업데이트도 들어오지 않는다면, transition 업데이트는 즉시(다음 프레임 안에) 처리됩니다. 지연이 눈에 띄는 것은 오직 사용자가 계속 타이핑하는 등 긴급 업데이트가 연속으로 들어와 transition 작업이 계속 뒤로 밀릴 때뿐입니다. 이 점에서 고정된 대기시간을 강제하는 디바운스와 근본적으로 다릅니다.

Q: Suspense boundary가 너무 상위(예: 페이지 최상단)에 있으면 어떤 문제가 생기나요?

모범 답변: 그 경계 안에 있는 모든 자식 중 가장 느린 하나가 전체 화면이 보이는 시점을 결정하게 됩니다. 헤더, 네비게이션처럼 데이터가 필요 없어 즉시 그릴 수 있는 부분까지도 가장 느린 컴포넌트(예: 외부 API를 호출하는 위젯)가 끝날 때까지 fallback 뒤에 가려집니다. 실무에서는 이것이 "왜 이 페이지는 로딩이 유독 느껴 보이지?"라는 리포트의 흔한 원인입니다. 해결책은 느린 컴포넌트만 별도의 하위 Suspense로 감싸 격리하고, 나머지는 상위 경계 밖에 두거나 더 빠르게 준비되는 별도 경계로 나누는 것입니다.

Q: useDeferredValue와 디바운스/쓰로틀의 차이는 정확히 무엇인가요?

모범 답변: 디바운스/쓰로틀은 "함수가 호출되는 시점 자체"를 인위적인 타이머로 제어하는 기법이며 React와 무관하게 동작합니다(입력이 멈추고 몇 ms 후에만 실제로 함수를 호출). useDeferredValue는 함수 호출 시점을 조절하는 것이 아니라, 렌더링 우선순위를 이용해 "새 값으로의 리렌더링"을 지연시킵니다. 값은 즉시 갱신되지만, 그 값을 반영하는 렌더링이 더 급한 작업 뒤로 밀릴 뿐입니다. 실무에서는 네트워크 요청 횟수 자체를 줄이고 싶다면 디바운스가 적합하고, 네트워크 요청 여부와 무관하게 화면에 무거운 컴포넌트가 있어 렌더링 자체가 버벅이는 문제라면 useDeferredValue가 적합합니다. 둘은 대체재가 아니라 서로 다른 계층의 문제를 다룹니다.

Q: Automatic batching과 concurrent rendering은 어떤 관계인가요?

모범 답변: Automatic batching(React 18에서 setTimeout, Promise 콜백, 네이티브 이벤트 핸들러 등 어디서든 여러 setState 호출을 하나의 렌더링으로 묶는 것)은 동시성 렌더링과 별개의 기능이지만, 같은 React 18 아키텍처(Fiber의 우선순위 기반 스케줄링) 위에서 함께 동작합니다. Automatic batching은 "여러 업데이트를 하나로 합쳐 렌더링 횟수를 줄이는 것"이고, transition/deferredValue는 "합쳐진(혹은 개별) 업데이트 중 어떤 것을 먼저 처리할지 우선순위를 매기는 것"입니다. 즉 batching이 렌더링 횟수를 줄인다면, concurrent 기능은 그 렌더링들 사이의 순서를 조정합니다.

Q: 서버 컴포넌트에서도 Suspense가 동작하나요? 클라이언트의 concurrent rendering과 같은 개념인가요?

모범 답변: 서버 컴포넌트에서도 Suspense는 동작하지만, 역할이 조금 다릅니다. 클라이언트에서는 Suspense가 "이미 화면에 있는 트리 중 일부를 fallback으로 바꿔치기"하는 데 쓰이지만, 서버 렌더링(streaming SSR)에서는 Suspense가 "이 경계 밖의 HTML을 먼저 스트림으로 보내고, 경계 안의 내용은 준비되는 대로 나중에 이어서 보낸다"는 신호로 쓰입니다. 우선순위 스케줄링이라는 개념 자체는 클라이언트의 concurrent rendering에 국한되지만, "아직 준비 안 된 하위 트리를 fallback으로 대체하고 나머지는 먼저 처리한다"는 Suspense의 근본 아이디어는 서버와 클라이언트 모두에서 동일하게 적용됩니다. 이 부분은 Day 17(Streaming)에서 더 깊이 다룹니다.

17. 흔한 오해

오해 1

"Suspense는 그냥 로딩 스피너를 보여주는 컴포넌트다"

Suspense는 스피너 컴포넌트가 아니라, 자식이 "아직 준비되지 않았다"고 선언하면 그 처리를 조상에게 위임하는 통신/전파 메커니즘입니다. fallback으로 무엇을 보여줄지는 자유이며 스피너는 그중 흔한 선택일 뿐입니다.

오해 2

"동시성 렌더링은 여러 스레드에서 병렬로 렌더링한다는 뜻이다"

자바스크립트는 여전히 싱글 스레드로 실행됩니다. "동시성(concurrent)"은 하나의 스레드 안에서 작업을 작은 단위로 쪼개 중단·재개·우선순위 조정을 할 수 있게 만든 스케줄링 모델을 의미하며, 병렬 실행(parallelism)과는 다른 개념입니다.

오해 3

"startTransition을 쓰면 업데이트가 항상 나중에, 느리게 실행된다"

실제로는 우선순위만 낮아질 뿐입니다. 더 긴급한 업데이트가 없다면 transition 업데이트도 즉시 처리됩니다. 지연은 "새치기당할 가능성"이지 "보장된 지연"이 아닙니다.

오해 4

"useTransition과 useDeferredValue는 같은 것을 다르게 부르는 이름일 뿐이다"

둘 다 낮은 우선순위 스케줄링을 이용하지만, 적용 대상이 다릅니다. useTransition은 내가 트리거하는 setState 호출에, useDeferredValue는 내가 통제할 수 없는 외부 값의 반영 시점에 씁니다. 코드 상에서 setState를 감쌀 지점이 있는지 없는지로 선택이 갈립니다.

오해 5

"모든 무거운 컴포넌트에 startTransition을 걸면 항상 성능이 좋아진다"

즉각적인 시각 피드백이 필요한 업데이트(버튼 눌림 효과, 폼 제출 직후 상태 변화 등)까지 transition으로 감싸면, 오히려 사용자가 "내가 누른 게 반영이 안 됐다"고 느끼는 UX 저하가 생깁니다. 또한 transition은 렌더링 비용 자체를 줄이지 않으므로, 근본적으로 너무 무거운 컴포넌트는 우선순위 조정만으로 해결되지 않습니다.
