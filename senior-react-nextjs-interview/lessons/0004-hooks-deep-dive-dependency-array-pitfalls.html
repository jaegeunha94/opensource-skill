1. 핵심 면접 질문

Primary Question

"useEffect의 의존성 배열에 함수나 객체를 넣었더니 무한 루프가 발생했습니다. 이 현상이 왜 일어나는지, 그리고 근본적으로 어떻게 해결해야 하는지 설명해 주세요."

Variant 1

"팀원이 이 문제를 useCallback으로 함수를 감싸서 해결했다고 합니다. 이 해결책이 항상 안전한가요? 언제 문제가 재발할 수 있나요?"

Variant 2

"실시간 채팅 컴포넌트에서 useEffect 안의 이벤트 핸들러가 오래된(stale) state 값을 참조해서, 사용자가 최근에 입력한 값이 반영되지 않는 버그가 있었습니다. 원인과 수정 방법을 설명해 주세요."

2. 왜 면접에서 묻는가

이 질문은 "의존성 배열 규칙(ESLint가 시키는 대로 채우면 된다)을 아는가"가 아니라, "왜 그 규칙이 존재하고, React가 실제로 무엇을 비교해서 effect를 재실행하는지 정확한 모델을 갖고 있는가"를 테스트합니다. useEffect는 React 생태계에서 가장 많은 프로덕션 버그(무한 루프, stale closure, race condition, 정리(cleanup) 누락으로 인한 메모리 누수)를 유발하는 API입니다. 이 버그들은 대부분 컴파일 에러 없이 조용히 발생하고, 재현이 간헐적이라 디버깅에 오래 걸립니다.

면접관이 이 질문으로 평가하는 것:

  • useEffect를 "생명주기 훅"이 아니라 "동기화 메커니즘"으로 이해하는가 — effect가 "마운트 시 한 번, 언마운트 시 한 번" 실행되는 것이 아니라 "렌더링 결과와 외부 시스템을 동기화하는 것"이며, 매 렌더링마다 "이 동기화가 여전히 유효한가"를 의존성 배열로 판단한다는 것을 아는가.
  • 의존성 비교 방식(Object.is 얕은 비교)을 정확히 아는가 — 매 렌더링마다 새로 생성되는 객체·배열·함수는 내용이 같아도 참조가 달라 "변경됨"으로 판정된다는 것을 아는가.
  • 무한 루프의 인과 관계를 추적할 수 있는가 — effect 내부에서 state를 변경 → 리렌더링 → 의존성으로 넣은 함수/객체가 새로 생성 → effect 재실행 → 다시 state 변경, 이 순환 구조를 코드를 보며 짚어낼 수 있는가.
  • stale closure를 이해하는가 — effect(또는 이벤트 핸들러) 내부의 함수가 "그 함수가 생성된 시점의" state/prop 값을 캡처한 클로저이며, 이후 state가 바뀌어도 그 함수 안에서는 여전히 옛날 값을 본다는 것을 아는가.
  • useCallback/useMemo가 "정확성 도구가 아니라 최적화 도구"라는 경계를 아는가 — 이 Hook들을 없애도 로직이 틀리게 동작해서는 안 되며, 있으면 재계산/재생성을 건너뛸 뿐이라는 것을 아는가.

10년차 이하에서 자주 나오는 답변 패턴: "ESLint가 경고하길래 useCallback으로 감쌌더니 해결됐어요." 이 답변은 증상을 없앴을 뿐, 왜 그 함수가 매 렌더링마다 새로 생성되었는지, useCallback이 실제로 무엇을 보장하는지(그 자체로 재생성을 완전히 막는 게 아니라 의존성이 바뀌지 않으면 이전 참조를 재사용한다는 것)를 설명하지 못합니다. 시니어는 "이 effect가 진짜로 동기화해야 하는 대상이 무엇인가"부터 질문하고, 필요하다면 애초에 effect의 구조 자체를 바꿉니다.

3. 전제 개념 — 쉬운 한국어로

렌더링과 effect의 타이밍 차이

Day 1에서 배운 render/commit 단계 구분을 다시 떠올려 봅시다. React는 먼저 컴포넌트 함수를 호출해서 "이번 렌더링에는 화면이 이렇게 생겨야 한다"는 결과(JSX 트리)를 계산합니다(render 단계). 그다음 그 결과를 실제 DOM에 반영합니다(commit 단계). useEffect로 등록한 콜백은 commit이 끝나고, 브라우저가 화면을 그린 뒤(paint 이후) 비동기적으로 실행됩니다. 즉 effect는 "렌더링의 일부"가 아니라 "렌더링이 끝난 뒤 부수효과를 처리하는 별도의 단계"입니다. 이 타이밍 차이 때문에 effect 안에서 DOM을 다시 읽거나, 외부 API를 구독하거나, 타이머를 설정하는 것이 안전합니다 — 화면은 이미 사용자에게 보여진 상태이기 때문입니다.

의존성 배열이 실제로 하는 일 — "얕은 비교(shallow comparison)"

많은 사람이 "의존성 배열에 넣은 값이 바뀌면 effect가 재실행된다"까지만 알고, "바뀜"을 어떻게 판단하는지는 정확히 모릅니다. React는 매 렌더링마다 이전 렌더링의 의존성 배열과 이번 렌더링의 의존성 배열을 배열의 각 항목을 Object.is로 하나씩 비교합니다. 숫자나 문자열 같은 원시값은 값이 같으면 Object.is도 같다고 판단하지만, 객체·배열·함수는 내용이 완전히 똑같아도 매번 새로 만들어진 것이면 다른 참조로 취급됩니다. 이것이 의존성 배열 함정의 근본 원인입니다.

왜 컴포넌트 함수 안에서 정의한 함수/객체는 매번 "새 것"인가

React 컴포넌트는 그냥 자바스크립트 함수입니다. 리렌더링이 일어난다는 것은 그 함수가 처음부터 다시 호출된다는 뜻입니다. 함수 본문 안에서 const options = { limit: 10 }나 const handleClick = () => {...}를 선언하면, 함수가 다시 호출될 때마다 자바스크립트 엔진은 매번 새로운 객체/함수 인스턴스를 메모리에 만듭니다. 내용(속성 값, 로직)이 이전 렌더링과 완전히 동일해도, 참조 주소는 다릅니다. 이것을 모르고 이런 객체나 함수를 의존성 배열에 넣으면, React 입장에서는 "매 렌더링마다 의존성이 바뀌었다"고 판단해 effect를 매번 재실행합니다.

4. useEffect의 정확한 정신 모델 — "동기화"이지 "생명주기 훅"이 아니다

클래스 컴포넌트 시절의 componentDidMount/componentDidUpdate/componentWillUnmount에 익숙한 사람들은 useEffect를 "마운트할 때 한 번, 업데이트할 때마다, 언마운트할 때 한 번 실행되는 훅"으로 이해하는 경우가 많습니다. 이 모델은 틀린 것은 아니지만, 왜 의존성 배열이 필요한지, 왜 그 배열을 임의로 조작하면 안 되는지를 설명하지 못합니다.

더 정확한 모델은 다음과 같습니다: useEffect는 "렌더링 결과(props, state)를 외부 시스템(DOM, 네트워크 구독, 타이머, 브라우저 API)과 동기화하는 함수"이고, 의존성 배열은 "이 동기화 로직이 사용하는 렌더링 결과값이 무엇인가"를 React에게 선언하는 것입니다. 매 렌더링 이후 React는 "지난번에 동기화했던 값들과 이번 렌더링의 값들이 같은가"를 확인하고, 다르면 지난 동기화를 정리(cleanup)하고 새 값으로 다시 동기화합니다.

이 모델이 왜 실무에 중요한가

"생명주기 훅"으로 이해하면 의존성 배열을 "이 effect를 언제 실행할지 제어하는 트리거 목록"으로 오해하게 됩니다. 그래서 "마운트할 때만 실행하고 싶으니 의존성 배열을 []로 비워두자"처럼, effect가 실제로 참조하는 값을 숨기고 배열을 임의로 조작하는 습관이 생깁니다. 반면 "동기화" 모델로 이해하면, 의존성 배열은 조작 대상이 아니라 effect 본문이 실제로 읽는 값을 정직하게 나열한 목록이어야 하고, "언제 실행할지"는 그 결과로 자연스럽게 따라오는 것임을 알게 됩니다.

5. 의존성 배열 함정 1 — 객체/함수 재생성으로 인한 무한 루프

function ChatRoom({ roomId }) {
  const [messages, setMessages] = useState([]);

  // 매 렌더링마다 새로 생성되는 객체
  const options = { roomId, serverUrl: "https://chat.example.com" };

  useEffect(() => {
    const connection = createConnection(options);
    connection.on("message", (msg) => {
      setMessages((prev) => [...prev, msg]); // state 변경 → 리렌더링 유발
    });
    connection.connect();
    return () => connection.disconnect();
  }, [options]); // options는 매 렌더링마다 새 참조 → 매번 "바뀜"으로 판정
}

이 코드의 순환 구조는 다음과 같습니다:

  1. 1
    컴포넌트가 렌더링되면서 options라는 새 객체가 생성된다.
  2. 2
    이전 렌더링의 options와 이번 options는 내용이 같아도 참조가 다르므로, React는 의존성이 "바뀌었다"고 판단해 effect를 재실행한다.
  3. 3
    effect 안에서 새 connection을 만들고 메시지를 받을 때마다 setMessages로 state를 바꾼다.
  4. 4
    state가 바뀌면 컴포넌트가 다시 렌더링되고, 그 과정에서 options가 다시 새로 생성된다 → 1번으로 돌아감.

메시지가 실제로 자주 도착하는 상황이라면 이 순환은 사실상 "연결을 계속 새로 맺고 끊는" 무한 루프에 가까운 동작으로 이어집니다. 근본 원인은 "매 렌더링마다 새로 생성되는 값을 의존성으로 사용하면서, effect 내부가 결과적으로 그 렌더링을 다시 유발한다"는 구조입니다.

근본적인 해결 — 의존성을 원시값으로 좁히기

가장 좋은 해결책은 useCallback/useMemo로 객체를 감싸는 것이 아니라, 애초에 effect가 원시값(문자열, 숫자)만 의존하도록 effect 내부에서 필요한 객체를 새로 만드는 것입니다.

function ChatRoom({ roomId }) {
  const [messages, setMessages] = useState([]);

  useEffect(() => {
    // options를 effect 내부에서 생성 — 컴포넌트 렌더링과 무관하게
    // 이 effect가 재실행될 때만 새로 만들어진다.
    const options = { roomId, serverUrl: "https://chat.example.com" };
    const connection = createConnection(options);
    connection.on("message", (msg) => {
      setMessages((prev) => [...prev, msg]);
    });
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]); // roomId는 원시값 — 실제로 값이 바뀔 때만 재실행
}

이렇게 하면 roomId라는 문자열/숫자만 비교하므로, roomId가 실제로 바뀔 때만 effect가 재실행됩니다. setMessages로 인한 리렌더링은 더 이상 effect 재실행을 유발하지 않습니다.

6. 의존성 배열 함정 2 — Stale Closure

클로저(closure)란 무엇인가 — 쉬운 설명

자바스크립트에서 함수는 자신이 정의될 때 주변에 있던 변수들을 "기억"합니다. 이것을 클로저라고 부릅니다. React 컴포넌트 함수는 렌더링될 때마다 처음부터 다시 실행되므로, 매 렌더링마다 그 안에서 정의되는 이벤트 핸들러나 effect 콜백은 "그 렌더링 시점의" props와 state 값을 클로저로 캡처한 별개의 함수 인스턴스입니다.

function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      // 이 콜백은 effect가 "처음 실행될 때"의 count(0)를 캡처했다.
      console.log(count); // 항상 0만 출력됨
      setCount(count + 1); // 항상 0 + 1 = 1로만 계산됨
    }, 1000);
    return () => clearInterval(id);
  }, []); // 빈 배열 — effect가 최초 한 번만 실행되고, 그때의 count(0)를 영원히 참조
}

여기서 버그는 setInterval의 콜백이 effect가 최초로 실행되었을 때의 count(즉 0)를 클로저로 영원히 붙잡고 있다는 것입니다. 의존성 배열이 []이므로 effect는 다시 실행되지 않고, 따라서 새 클로저(최신 count를 캡처한 새 콜백)도 만들어지지 않습니다. 결과적으로 1초마다 항상 "0 + 1 = 1"만 계산되어 화면에는 count가 1에서 멈춰 있게 됩니다.

해결책 1 — 함수형 업데이트로 최신 state 값 문제를 우회

useEffect(() => {
  const id = setInterval(() => {
    // 이전 state를 인자로 받는 함수형 업데이트는
    // React가 "커밋 직전의 최신 state"를 넣어 호출해준다.
    setCount((prev) => prev + 1);
  }, 1000);
  return () => clearInterval(id);
}, []);

이 방법은 "다음 state가 이전 state로부터 계산되는 경우"에 stale closure 문제를 우회하는 정석적인 해법입니다. 하지만 이것도 만능은 아닙니다 — count를 콘솔에 출력하거나, 다른 값을 계산하는 데 써야 한다면 여전히 stale 값 문제가 남습니다.

해결책 2 — 의존성 배열에 정직하게 값을 채우기

useEffect(() => {
  const id = setInterval(() => {
    console.log(count); // 이제 항상 최신 count를 출력
    setCount(count + 1);
  }, 1000);
  return () => clearInterval(id);
}, [count]); // count가 바뀔 때마다 effect가 정리되고 새로 실행됨

이 방법은 매초 count가 바뀔 때마다 이전 setInterval을 정리(clear)하고 새 setInterval을 등록합니다. 타이머 하나만 다루는 예제에서는 비용이 적지만, 무거운 구독/연결을 매번 재생성해야 하는 상황이라면 성능 문제가 될 수 있습니다. 이런 경우 useRef로 최신 값을 "참조"만 하고 effect 재실행을 유발하지 않는 패턴(ref를 이용한 최신 값 접근)을 함께 검토합니다.

7. useMemo/useCallback — 최적화 도구이지 정확성 도구가 아니다

많은 주니어 개발자가 "리렌더링이 많으니까 useMemo/useCallback을 여기저기 붙이면 빨라진다"고 생각합니다. 정확한 모델은 다릅니다.

useMemo는 "계산 결과를, useCallback은 "함수 참조"를, 의존성이 바뀌지 않는 한 이전 렌더링의 것을 그대로 재사용하게 해주는 캐싱 도구입니다. 이 Hook들을 걷어내도 애플리케이션이 계산하는 값이나 함수의 동작 자체는 똑같아야 합니다. 즉 이것은 "더 빠르게 만드는" 도구이지 "정확하게 만드는" 도구가 아닙니다. 만약 useMemo를 지웠더니 로직이 깨진다면, 그것은 애초에 설계가 그 최적화에 의존하고 있었다는 뜻이며 버그의 징후입니다.

상황 사용 여부 이유
비용이 큰 계산(대량 배열 필터링/정렬, 복잡한 파생 데이터)을 매 렌더링마다 반복하고 싶지 않을 때 useMemo 사용 권장 실제 측정 가능한 계산 비용 절감 효과가 있음
단순한 산술 연산, 문자열 결합처럼 계산 비용이 거의 없는 값 사용하지 않음 useMemo 자체의 비교/캐시 관리 비용이 계산 비용보다 클 수 있음
자식 컴포넌트가 React.memo로 감싸져 있고, 그 자식에게 함수를 prop으로 넘겨 불필요한 리렌더링을 막고 싶을 때 useCallback 사용 권장 함수 참조가 안정적이어야 React.memo의 얕은 비교가 의미를 가짐
함수를 자식에게 넘기지 않고 이 컴포넌트 안에서만 쓰는 경우 대부분 불필요 React.memo 비교 대상이 아니므로 참조 안정성의 이점이 없음
객체/함수를 useEffect의 의존성으로 넣어야 하는데 무한 루프를 막고 싶을 때 임시방편으로는 동작하지만, 우선 effect 구조 자체를 재검토 Section 5에서 본 것처럼, 근본 해법은 의존성을 원시값으로 좁히는 것 — useCallback은 "고쳤다는 착각"을 줄 뿐, 그 함수의 의존성 배열도 정확히 관리해야 하므로 문제를 한 단계 미룰 뿐인 경우가 많음

8. Hooks 사용 판단표

증상 의심할 원인 확인 방법
네트워크 탭에 같은 요청이 계속 반복되거나, 화면이 깜빡이며 무한 로딩됨 effect 의존성에 매 렌더링마다 새로 생성되는 객체/배열/함수가 있고, effect 내부가 리렌더링을 유발함 React DevTools Profiler로 어떤 컴포넌트가 계속 리렌더링되는지 확인 → 해당 effect의 의존성 배열 검사
이벤트 핸들러나 setInterval 콜백이 "예전 값"만 참조하는 것처럼 보임 stale closure — effect/콜백이 오래된 렌더링의 값을 캡처 의존성 배열이 비어 있거나 불완전한지 확인, 함수형 업데이트 적용 가능 여부 검토
리스트가 커질수록 타이핑/스크롤이 버벅임 비용이 큰 계산이 매 렌더링마다 반복 실행됨 Profiler로 렌더링당 소요 시간 측정 → 실제로 비용이 크면 useMemo 적용, 아니면 다른 병목(리렌더링 범위 자체)을 의심
구독/타이머를 정리하지 않아 컴포넌트가 언마운트된 후에도 콘솔 경고나 메모리 누수 발생 cleanup 함수 누락 또는 조건부 return으로 cleanup이 실행되지 않는 경로 존재 effect가 등록한 모든 리소스(구독, 타이머, 이벤트 리스너)에 대응하는 cleanup이 있는지 코드 리뷰

9. 답변 사고 순서

  1. useEffect의 정확한 모델을 먼저 말합니다. "useEffect는 생명주기 훅이 아니라, 렌더링 결과와 외부 시스템을 동기화하는 메커니즘이고, 의존성 배열은 이 동기화가 사용하는 값을 선언한 것이다."
  2. 무한 루프의 인과 관계를 짚습니다. "의존성에 넣은 객체/함수가 컴포넌트 함수 내부에서 매 렌더링마다 새로 생성되므로, React가 얕은 비교(Object.is)로 매번 '바뀌었다'고 판단해 effect를 재실행하고, effect 내부가 state를 바꿔 다시 렌더링을 유발하는 순환이 생긴다."
  3. 근본 해결책을 제시합니다. "임시방편으로 useCallback을 쓰기보다, effect가 실제로 필요한 값이 원시값(예: roomId)뿐이라면 객체를 effect 내부에서 생성하도록 구조를 바꿔 의존성을 원시값으로 좁힌다."
  4. useCallback을 쓸 때의 한계를 짚습니다. "useCallback으로 감싸도 그 자체의 의존성 배열을 정확히 관리하지 않으면 문제가 재발할 수 있고, 근본적으로는 최적화 도구이지 정확성을 보장하는 도구가 아니다."
  5. 실무 연결로 마무리합니다. "이런 종류의 버그는 컴파일 에러 없이 조용히 나타나기 때문에, 실무에서는 ESLint의 react-hooks/exhaustive-deps 규칙을 신뢰하고 임의로 의존성을 제거하지 않는 것이 1차 방어선이다."

10. 핵심 답변 예시

면접관: "useEffect의 의존성 배열에 함수나 객체를 넣었더니 무한 루프가 발생했습니다. 이 현상이 왜 일어나는지, 그리고 근본적으로 어떻게 해결해야 하는지 설명해 주세요."

STRONG ANSWER

이 문제는 React가 의존성 배열을 비교하는 방식(Object.is를 이용한 얕은 비교)과, 자바스크립트 함수가 리렌더링마다 처음부터 다시 실행된다는 두 가지 사실이 결합해서 생깁니다. 컴포넌트 함수 본문 안에서 const options = {`{ ... }`}처럼 객체를 선언하면, 컴포넌트가 리렌더링될 때마다 이 객체는 내용이 같아도 매번 새로운 참조로 생성됩니다. 이 객체를 useEffect의 의존성 배열에 넣으면, React는 이전 렌더링의 참조와 이번 참조가 다르므로 '의존성이 바뀌었다'고 판단해 effect를 재실행합니다.


여기서 그 effect 내부가 setState로 state를 변경하는 로직을 포함하고 있다면 순환이 생깁니다. effect 실행 → state 변경 → 리렌더링 → 객체가 새로 생성됨 → 의존성이 '바뀜'으로 판정 → effect 재실행, 이 흐름이 계속 반복되는 것입니다.


임시방편으로 그 객체를 useMemo로, 함수를 useCallback으로 감싸면 증상이 완화될 수 있지만, 이는 근본 해결이 아닙니다. 저라면 먼저 'effect가 실제로 필요로 하는 값이 무엇인가'를 다시 확인합니다. 대부분의 경우 필요한 것은 roomId같은 원시값 하나뿐이고, options 객체 자체는 effect 내부에서 그 원시값으로부터 새로 만들면 충분합니다. 이렇게 하면 의존성 배열에는 원시값만 남고, 그 원시값이 실제로 바뀔 때만 effect가 재실행되어 순환이 끊깁니다. 즉 이 문제의 진짜 해법은 'memoization으로 참조를 안정시키는 것'이 아니라 '애초에 effect의 의존성 표면을 원시값으로 좁히는 설계'입니다.

이 답변이 좋은 이유:

  • React의 의존성 비교 방식(얕은 비교)과 자바스크립트 함수 재실행이라는 두 가지 원인을 결합해서 인과 관계를 정확히 설명함
  • 무한 루프의 순환 구조를 단계별로 짚어냄
  • useCallback/useMemo가 "임시방편"이라는 것을 명시하고, 그 이유(그 자체의 의존성 관리 문제가 남음)까지 설명함
  • 근본 해결책(의존성 표면을 원시값으로 좁히는 설계 변경)을 구체적으로 제시함

11. 좋은 답변 vs 약한 답변

WEAK ANSWER — 이런 답변을 피하세요

"함수를 useCallback으로, 객체를 useMemo로 감싸면 참조가 유지돼서 무한 루프가 해결됩니다."

이 답변이 약한 이유

  • "왜" 무한 루프가 발생했는지(의존성 비교 방식, 리렌더링마다 새 참조 생성) 설명하지 않고 해결 도구 이름만 말합니다.
  • useCallback도 결국 그 함수 내부가 참조하는 값들을 의존성 배열에 정확히 넣어야 하는데, 이 관리가 소홀하면 문제가 재발할 수 있다는 한계를 짚지 못했습니다.
  • 면접관이 "그 useCallback의 의존성 배열은 어떻게 관리하나요?"라고 물으면 순환 논리에 빠질 가능성이 높습니다.
WEAK ANSWER — 또 다른 패턴

"의존성 배열을 비워두면([]) 무한 루프가 생기지 않으니, 웬만하면 빈 배열을 쓰는 게 안전합니다."

이 답변이 약한 이유

  • 의존성 배열을 임의로 비우는 것은 "effect가 실제로 참조하는 최신 값을 무시하겠다"는 뜻이며, 이는 무한 루프 대신 stale closure 버그를 만듭니다 — 문제를 해결한 것이 아니라 다른 문제로 바꾼 것입니다.
  • ESLint의 react-hooks/exhaustive-deps 규칙이 왜 존재하는지, 그리고 그 규칙을 임의로 무시하는 것이 왜 위험한지에 대한 인식이 없습니다.
STRONG ANSWER 요소 체크리스트
  • 의존성 배열 비교가 Object.is 기반 얕은 비교라는 것을 명시한다
  • 컴포넌트 함수가 리렌더링마다 처음부터 재실행되어 객체/함수가 매번 새로 생성된다는 것을 설명한다
  • state 변경 → 리렌더링 → 재생성 → effect 재실행이라는 순환 구조를 단계별로 짚는다
  • useCallback/useMemo를 "임시방편"으로 정확히 위치시키고, 근본 해결(의존성을 원시값으로 좁히는 설계)을 제시한다
  • exhaustive-deps 규칙을 신뢰해야 하는 이유를 언급한다

12. Senior-level Follow-up Q&A

Q: ESLint의 react-hooks/exhaustive-deps 규칙이 경고하는 의존성을 일부러 배열에서 빼도 되는 경우가 있나요?

모범 답변: 원칙적으로는 거의 없어야 합니다. 하지만 "그 값이 절대 바뀌지 않는다는 것을 구조적으로 보장할 수 있는 경우"(예: useState의 setter 함수 자체, useRef로 만든 ref 객체)는 React가 참조 안정성을 보장하는 값이므로 배열에서 빼도 안전하다고 알려져 있습니다. 이런 특수한 경우가 아니라면, 규칙을 끄는 대신 effect의 구조 자체(값을 ref로 옮기거나, effect를 원시값 의존성만 갖도록 재설계)를 먼저 검토해야 합니다. "경고가 귀찮아서 끈다"는 것은 항상 잘못된 이유입니다.

Q: useMemo/useCallback을 습관적으로 모든 곳에 붙이는 것이 왜 항상 좋은 것은 아닌가요?

모범 답변: useMemo/useCallback 자체도 매 렌더링마다 의존성 배열을 비교하고 이전 값을 저장/조회하는 비용이 있습니다. 계산 비용이 거의 없는 값(단순 문자열 결합, 간단한 산술)에 이 Hook을 붙이면, 캐싱에 드는 비용이 원래 계산 비용보다 커질 수 있습니다. 또한 이 Hook들을 남발하면 의존성 배열을 관리해야 할 지점이 늘어나 코드가 복잡해지고, 오히려 의존성 배열 실수(stale closure, 무한 루프)의 발생 지점이 늘어나는 역효과가 생길 수 있습니다. 실제로 측정했을 때 리렌더링 비용이 유의미한 지점에만 선택적으로 적용하는 것이 원칙입니다.

Q: effect 안에서 최신 state 값은 참조하고 싶지만, 그 값이 바뀔 때마다 effect가 재실행되는 것은 원하지 않는 경우 어떻게 하나요?

모범 답변: useRef를 이용해 최신 값을 "미러링"하는 패턴을 씁니다. 별도의 effect(또는 렌더링 중)에서 latestValueRef.current = value로 매 렌더링마다 ref를 최신 값으로 갱신하고, 원래 effect 안에서는 latestValueRef.current를 읽습니다. ref를 읽고 쓰는 것은 렌더링을 유발하지 않고 의존성 배열 비교 대상도 아니므로, effect는 재실행되지 않으면서도 항상 최신 값을 참조할 수 있습니다. 다만 이 패턴은 남용하면 '의존성 배열이 왜 이렇게 생겼는지' 코드만 봐서는 알기 어려워지므로, 꼭 필요한 곳(예: 이벤트 핸들러 등록/해제 비용이 매우 큰 경우)에만 신중하게 적용해야 합니다.

Q: React 19의 컴파일러(React Compiler)가 useMemo/useCallback을 자동으로 처리해준다고 들었습니다. 그러면 이 Hook들을 더 이상 몰라도 되나요?

모범 답변: React Compiler는 컴파일 타임에 코드를 분석해서 필요한 memoization을 자동으로 삽입해주는 도구이지만, 이것이 "의존성 배열 개념 자체를 몰라도 된다"는 뜻은 아닙니다. 첫째, 컴파일러가 적용되지 않는 레거시 코드베이스나 컴파일러 규칙을 위반하는 코드(예: 순수하지 않은 렌더링 로직)에서는 여전히 수동 관리가 필요합니다. 둘째, useEffect의 의존성 배열(외부 시스템과의 동기화)은 컴파일러가 대신 판단해줄 수 있는 영역이 아니라 개발자가 '이 effect가 무엇에 의존하는가'를 직접 선언해야 하는 영역입니다. 컴파일러는 useMemo/useCallback 같은 렌더링 최적화 영역을 자동화하는 것이지, useEffect의 동기화 로직 자체를 대신 설계해주지 않습니다.

Q: 두 개의 effect가 있는데, 하나는 데이터를 fetch하고 다른 하나는 그 데이터를 가공합니다. 이렇게 effect를 체이닝하는 것이 좋은 패턴인가요?

모범 답변: 좋은 패턴이 아닙니다. effect가 다른 effect의 결과(state)에 의존해서 또 다른 state를 계산하는 체이닝 구조는, 렌더링이 여러 번 연쇄적으로 일어나게 만들고 디버깅을 어렵게 만듭니다. 이런 경우 대부분은 "파생 상태(derived state)"이므로, 별도 effect를 만들 필요 없이 렌더링 중에 직접 계산하거나(필요하면 useMemo로 감싸서), 데이터 fetch 로직 안에서 가공까지 마쳐서 state를 한 번만 갱신하도록 구조를 바꾸는 것이 원칙입니다. 'state가 바뀌면 또 다른 state를 갱신해야 한다'는 요구가 보이면, 먼저 '이것이 정말 effect가 필요한 동기화인가, 아니면 그냥 렌더링 중 계산으로 대체 가능한가'를 질문해야 합니다.

13. 흔한 오해

오해 1

"useEffect는 componentDidMount/componentDidUpdate/componentWillUnmount를 합친 것이다"

동작 결과가 비슷해 보일 때가 많아 흔히 이렇게 설명하지만, 이 모델로는 "왜 의존성 배열이 필요한지"를 설명할 수 없습니다. 정확한 모델은 "렌더링 결과를 외부 시스템과 동기화하는 함수"이며, 의존성 배열은 그 동기화가 참조하는 값의 목록입니다.

오해 2

"의존성 배열은 그냥 effect를 언제 다시 실행할지 트리거하는 조건 목록이다"

의존성 배열을 "트리거 조건"으로 보면 "원하는 시점에만 실행되게 배열을 조작해도 된다"는 오해로 이어집니다. 실제로는 "effect 본문이 실제로 읽는 값을 정직하게 선언"하는 것이며, 실행 시점은 그 선언의 자연스러운 결과일 뿐입니다.

오해 3

"useCallback/useMemo를 쓰면 항상 성능이 좋아진다"

이 Hook들 자체도 비교와 캐시 관리 비용이 있습니다. 계산 비용이 낮은 값에 붙이면 오히려 손해일 수 있고, 실제 이득은 "비용이 큰 계산을 반복하지 않을 때" 또는 "React.memo로 감싼 자식에게 안정적인 참조를 넘겨야 할 때"에만 발생합니다.

오해 4

"stale closure는 드문 엣지 케이스이니 신경 쓰지 않아도 된다"

타이머, 이벤트 리스너, 웹소켓 구독, 디바운스/스로틀 처리된 콜백처럼 "한 번 등록되면 한동안 유지되는" 모든 비동기 패턴에서 stale closure는 매우 흔하게 발생합니다. 실무에서 "최근에 입력한 값이 반영되지 않는다", "몇 초 전 상태로 동작하는 것 같다" 같은 버그 리포트의 상당수가 이 원인입니다.

오해 5

"cleanup 함수는 언마운트될 때만 실행된다"

cleanup 함수는 언마운트 시뿐 아니라, 의존성이 바뀌어 effect가 재실행되기 직전에도 매번 실행됩니다. "이전 동기화를 정리하고 새로 동기화한다"는 모델을 생각하면 당연한 동작이지만, 이를 모르면 "effect가 여러 번 실행되는데 cleanup은 마지막에만 한 번 실행될 것"이라고 착각해 리소스 누수나 중복 구독 버그를 만들 수 있습니다.

다음 항목을 말로 설명할 수 있으면 Day 4를 이해한 것입니다.

☐
의존성 배열 비교가 Object.is 기반 얕은 비교이며, 객체/배열/함수는 내용이 같아도 참조가 다르면 "바뀜"으로 판정된다는 것을 설명할 수 있다.
☐
객체/함수 재생성으로 인한 무한 루프의 순환 구조(재생성 → 재비교 → effect 재실행 → state 변경 → 리렌더링 → 재생성)를 코드를 보며 짚어낼 수 있다.
☐
무한 루프의 근본 해결책이 "의존성을 원시값으로 좁히는 설계 변경"이며, useCallback/useMemo는 임시방편에 가깝다는 것을 설명할 수 있다.
☐
stale closure가 무엇이고, 왜 빈 의존성 배열이 이 문제를 만드는지 설명할 수 있다.
☐
함수형 업데이트(setState(prev => ...))와 ref 미러링 패턴이 각각 어떤 stale closure 상황에 적합한지 구분할 수 있다.
☐
useMemo/useCallback이 정확성이 아니라 최적화를 위한 도구이며, 언제 실제로 필요한지(비용이 큰 계산, React.memo 자식으로의 안정적 참조 전달) 구분할 수 있다.
☐
cleanup 함수가 언마운트 시뿐 아니라 의존성 변경으로 인한 재실행 직전에도 매번 호출된다는 것을 설명할 수 있다.
