Day 1 · 시스템 설계 Interview

시스템 설계 답변 프레임워크와 요구사항 정리

면접관이 가장 먼저 보는 것 — "이 사람은 스스로 문제를 정의할 수 있는가?"

이 레슨의 핵심 면접 질문

시스템 설계 면접의 시작은 거의 항상 다음과 같은 열린 질문이다.

면접 질문 유형 (공통 패턴)
  • "Instagram/Twitter 같은 소셜 미디어 서비스를 설계해주세요."
  • "하루 1억 명이 사용하는 URL 단축 서비스를 설계해주세요."
  • "실시간 채팅 서비스를 설계해주세요."
  • "대용량 파일 업로드/다운로드 서비스를 설계해주세요."

이 질문들의 공통점: 제약 조건이 일부러 빠져 있다. 면접관은 당신이 스스로 물어보고 정리하는 과정을 보고 싶다.

전제 개념 — 시스템 설계 면접이란?

이 개념을 먼저 이해하지 않으면, 왜 프레임워크를 따라야 하는지 납득하기 어렵다.

시스템 설계 면접은 "정답"이 없는 대화다

코딩 면접은 정답이 하나다. 하지만 시스템 설계 면접은 다르다. 같은 문제에 올바른 설계가 여러 가지 존재하고, 각각 다른 trade-off를 갖는다. 면접관은 당신이 어떻게 생각하는지를 본다.

  • 주니어를 거르는 기준: 스스로 전제 조건 없이 설계를 시작하는지 여부
  • 시니어를 판단하는 기준: trade-off를 인식하고 근거 있는 선택을 하는지 여부
  • 리드급을 구분하는 기준: 자신의 설계가 어디서 망가지는지, 어떻게 고칠지까지 먼저 말하는지 여부

45분 안에 무엇을 보여줘야 하는가?

일반적인 시스템 설계 면접은 45~60분이다. 이 시간 안에 면접관이 확인하려는 것은 크게 다섯 가지다.

  1. 문제 정의 능력: 주어진 문제를 스스로 구조화할 수 있는가?
  2. 규모 인식: 트래픽, 저장 용량, 지연 시간을 현실적으로 추정하는가?
  3. 컴포넌트 선택 근거: "왜 이걸 썼나요?"에 답할 수 있는가?
  4. 확장성 사고: "10배 트래픽이 들어오면?"에 즉각 답할 수 있는가?
  5. 장애 인식: 어디서 장애가 나고 어떻게 복구하는지 먼저 말하는가?

왜 면접에서 이것을 묻는가?

면접관의 관점

10년차 이상 개발자에게 기대하는 것은 독립적 문제 정의 능력이다. 주니어는 요구사항을 주면 구현한다. 시니어는 요구사항이 불명확할 때 스스로 명확하게 만든다. 면접관이 "URL 단축 서비스를 설계하세요"라고 모호하게 말하는 것은 일부러 정보를 빠뜨리는 것이다.

  • 바로 화이트보드에 그리기 시작하면? → "이 사람은 스스로 문제를 정의하지 않는다" 신호
  • 먼저 질문하면? → "이 사람은 실제 프로젝트에서 어떻게 일하는지 보인다" 신호
실무 연결

실제 프로젝트에서 기획자나 PM이 "이런 기능 만들어주세요"라고 올 때, 시니어 개발자는 곧바로 개발을 시작하지 않는다. "DAU가 얼마나 예상되나요?", "지연 시간이 몇 ms여야 하나요?", "데이터 보존 기간은요?"를 먼저 확인한다. 요구사항이 바뀌면 설계도 바뀐다. 이걸 미리 확인하지 않으면 나중에 재설계 비용이 폭발한다.

5단계 답변 프레임워크

시스템 설계 면접 답변에는 검증된 구조가 있다. 이 순서를 지키는 것이 중요하다. 순서를 건너뛰면 "정답처럼 보이는 설계"가 나올 수 있지만, 면접관은 과정을 더 중요하게 본다.

  1. 요구사항 정리 ⏱ 5~10분

    Functional Requirements(기능 요구사항)와 Non-functional Requirements(비기능 요구사항)로 나눠서 정리한다. 이 단계를 건너뛰면 이후 모든 설계가 근거 없이 흔들린다.

  2. Capacity Estimation (규모 산정) ⏱ 3~5분

    DAU(일일 활성 사용자), QPS(초당 요청 수), 저장 용량, 네트워크 대역폭을 추정한다. 숫자가 정확할 필요는 없다. 올바른 자릿수(order of magnitude)가 중요하다.

  3. High-Level Design ⏱ 10~15분

    주요 컴포넌트(클라이언트, API 서버, DB, 캐시, CDN 등)를 블록 다이어그램으로 그린다. 데이터가 어떻게 흐르는지 설명한다. 핵심 API와 DB 스키마를 간략히 정의한다.

  4. Deep Dive ⏱ 10~15분

    면접관과 합의한 1~2개 핵심 컴포넌트를 깊이 다룬다. 병목 지점, 확장 방법, 장애 시나리오, 대안과 그 trade-off를 설명한다.

  5. 마무리 & 개선점 ⏱ 3~5분

    현재 설계의 한계를 먼저 말한다. "이 부분은 아직 SPOF(Single Point of Failure)입니다. 개선한다면 이렇게 하겠습니다." 모니터링, 알림, 장애 복구 절차를 간략히 언급한다.

흔한 실수

많은 후보자가 Step 1, 2를 건너뛰고 Step 3(화이트보드 그림)부터 시작한다. 그 결과 면접관이 중간에 "사용자가 몇 명인가요?"라고 물으면 답변이 흔들린다. 전체 설계가 어떤 규모를 가정했는지 모르기 때문이다.

Step 1 상세 — 요구사항 정리하는 방법

Functional Requirements (기능 요구사항)

"이 시스템이 무엇을 해야 하는가?" — 사용자 관점에서 핵심 기능을 나열한다. 모든 기능을 다 다루려 하지 말고, 핵심 3~5개만 MVP로 정의하고 면접관과 합의한다.

URL 단축 서비스 예시:

  • 긴 URL을 입력하면 짧은 URL을 반환한다
  • 짧은 URL로 요청하면 원래 URL로 리다이렉트한다
  • (선택) URL 만료 시간을 설정할 수 있다
  • (선택) 클릭 통계를 제공한다

Non-functional Requirements (비기능 요구사항)

"이 시스템이 어떻게 동작해야 하는가?" — 면접관이 명시하지 않더라도 반드시 확인해야 할 것들이다. 이 부분을 확인하지 않으면 설계 방향이 완전히 달라진다.

반드시 물어봐야 할 질문 목록
  1. 규모: "DAU(일일 활성 사용자)가 얼마나 예상되나요?" → 설계 전반의 기준이 된다
  2. 읽기/쓰기 비율: "읽기 중심인가요, 쓰기 중심인가요?" → 캐시 전략과 DB 선택이 달라진다
  3. 지연 시간: "응답 시간이 얼마나 중요한가요? P99 기준으로요?" → 캐시 레이어 결정에 영향
  4. 가용성: "99.9% vs 99.99%? 다운타임 허용 시간은?" → 중복 구성 여부가 달라진다
  5. 일관성: "약간의 불일치(stale data)를 허용하나요?" → DB 복제 전략이 달라진다
  6. 데이터 보존: "데이터를 얼마나 오래 저장해야 하나요?" → 저장소 비용과 아키텍처가 달라진다
  7. 지역: "글로벌 서비스인가요, 단일 지역인가요?" → CDN/멀티 리전 필요 여부

요구사항 정리 답변 예시

면접에서 이렇게 말하면 된다:

답변 예시 — 요구사항 정리 단계

"먼저 요구사항을 정리하고 싶습니다. Functional Requirements 쪽에서는 긴 URL을 단축하는 기능, 단축 URL로 접근하면 원본으로 리다이렉트하는 기능을 핵심으로 보겠습니다. 만료 기능과 통계는 일단 범위 밖으로 두겠습니다 — 맞나요?

Non-functional 쪽으로는 몇 가지 여쭤봐도 될까요? 하루에 얼마나 많은 URL이 생성될 것으로 예상하시나요? 읽기 요청이 쓰기 요청보다 얼마나 많을까요? 리다이렉트 지연 시간은 얼마 이하여야 하나요? 가용성은 99.9%면 충분한가요, 더 높아야 하나요?"

강한 답변과 약한 답변의 차이

강한 답변 (Senior level)

"설계를 시작하기 전에 몇 가지 확인하고 싶습니다. DAU가 1억 명이면 쓰기 QPS는 대략 1,160 req/s, 읽기는 10배로 보면 약 11,600 req/s 정도 되겠네요. 이 정도면 단일 DB 서버로는 힘들 것 같고, 읽기 replica를 두거나 캐시를 앞에 놓는 방향으로 가야 할 것 같습니다. 일관성 요구사항은 어떻게 보세요? URL 리다이렉트는 stale data가 잠깐 있어도 괜찮을 것 같아서 eventual consistency로 가도 될 것 같은데, 맞나요?"

약한 답변 (Junior level)

"URL 단축 서비스는 MySQL에 URL을 저장하고, 해시값을 생성해서 단축 URL로 쓰면 됩니다. Base62 인코딩을 쓰면 6자리 URL이 나오고요. (화이트보드에 바로 그림 시작)"

왜 약한 답변이 약한가?

  • 규모를 모름: "MySQL 하나면 되나요?" — DAU가 1천만인지 10억인지에 따라 완전히 다른 설계가 필요하다
  • 요구사항 없이 구현 시작: 면접관이 원하는 기능이 무엇인지 확인하지 않음
  • Trade-off 없음: "Base62 해시"라고 말했지만, 왜 다른 방법(random UUID, Counter-based ID)을 선택하지 않았는지 설명 없음
  • 확장성 고려 없음: 현재 설계가 언제 한계에 도달하는지, 그때 어떻게 할지 없음

답변 사고 순서 — 실전 적용

머릿속 흐름 (45분 기준)

[0~1분] 문제를 들으면서 마음속으로 체크: "이 시스템은 읽기 heavy인가, 쓰기 heavy인가? 어떤 consistency가 필요한가? 어디서 병목이 생길까?"

[1~8분] 질문하면서 요구사항 정리. 칠판에 적거나 "제가 정리한 내용이 맞나요?"라고 확인. 면접관과 Scope를 합의한다.

[8~12분] Capacity estimation. QPS, 저장 용량을 계산하면서 "어디서 병목이 생길지" 예측한다.

[12~25분] High-level design. 블록 다이어그램 그리면서 설명. 중간중간 "이 방향 맞나요?"라고 체크인.

[25~40분] Deep dive. 면접관이 관심 있는 부분에 집중. "어느 쪽을 더 깊이 보고 싶으신가요?"라고 먼저 물어봐도 좋다.

[40~45분] 마무리. 현재 설계의 한계, 개선 방향, 모니터링 방법 언급.

Senior-Level Follow-up 질문과 답변 방향

요구사항 정리까지 잘 했으면, 면접관은 깊은 질문으로 들어온다. 이 질문들에 막히지 않으려면 준비해야 한다.

Follow-up 1

Q: "요구사항을 정리했는데, Availability와 Consistency 중 하나를 선택해야 한다면?"

답변 방향: 이 시스템이 "URL 단축"이라면 리다이렉트가 안 되는 것(Availability 손실)이 데이터가 잠깐 오래된 것(Consistency 손실)보다 더 큰 피해다. → Availability 우선, Eventual Consistency 선택.
반면 결제 시스템이라면 Consistency 손실이 금전 피해를 유발하므로 Consistency 우선.
포인트: CAP Theorem을 언급하되, 이 시스템의 특성에 맞게 근거를 댈 수 있어야 한다.

Follow-up 2

Q: "갑자기 트래픽이 10배로 뛰면 당신의 설계에서 어디가 먼저 죽나요?"

답변 방향: 지금 설계를 보면서 "Single Point of Failure"와 "가장 먼저 포화될 레이어"를 먼저 말한다. 예: "DB가 가장 먼저 죽을 것 같습니다. 읽기 쪽은 캐시 히트율이 높으면 버티겠지만, 쓰기 QPS가 10배면 Primary DB의 I/O가 포화됩니다. 이 경우 쓰기를 메시지 큐로 비동기 처리하거나, DB 샤딩을 먼저 고려하겠습니다."

Follow-up 3

Q: "이 설계에서 면접 30분 더 있다면 어떤 부분을 더 다루고 싶으신가요?"

답변 방향: 이 질문은 당신이 설계의 미완성 부분을 인식하고 있는지 체크한다. 좋은 답변: "Rate limiting을 아직 다루지 않았습니다. 단축 URL 생성이 DDoS 대상이 될 수 있어서요. 그리고 모니터링 — 어떤 메트릭을 수집하고, 어떤 알람을 설정할지도 빠졌습니다."

Trade-off와 흔한 오해

오해 1: "요구사항 질문을 많이 하면 시간이 낭비된다"

실제: 요구사항 정리는 시간 낭비가 아니라 방향 설정이다. 5분 투자해서 방향을 맞추는 게, 잘못된 방향으로 25분 설계하다 중간에 뒤집는 것보다 훨씬 낫다. 또한 면접관은 이 과정에서 "이 사람은 실무에서도 이렇게 일하겠구나"라고 좋은 신호를 받는다.

오해 2: "숫자를 정확하게 계산해야 한다"

실제: 면접관은 정확한 수치보다 올바른 사고 방식(reasoning)을 본다. "1억 DAU × 평균 10번 리다이렉트 / 86,400초 ≈ 11,574 QPS, 반올림해서 약 12,000 QPS로 잡겠습니다." 이 계산 과정이 맞다면 12,000이든 15,000이든 크게 중요하지 않다.

오해 3: "Non-functional Requirements는 면접관이 다 알려준다"

실제: 면접관은 일부러 빠뜨린다. 당신이 먼저 물어봐야 한다. 가용성, 일관성, 지연 시간 요구사항을 물어보지 않으면 "이 사람은 시니어 설계 경험이 없다"는 신호로 읽힌다.

Trade-off: Availability vs Consistency

요구사항 정리 단계에서 이미 이 trade-off가 결정된다. "잠깐 오래된 데이터를 보여줘도 되나요?"라는 질문 하나가 이후 DB 복제 전략, 캐시 전략, 분산 트랜잭션 필요 여부를 모두 결정한다.

  • Availability 우선(AP): 소셜 미디어 피드, URL 단축, 검색 결과 — 잠깐 stale해도 서비스 가용이 더 중요
  • Consistency 우선(CP): 결제, 재고 관리, 은행 잔고 — 데이터 불일치가 금전 피해를 유발

자기 점검 퀴즈

핵심 정리

  • 시스템 설계 면접에서 요구사항 정리가 가장 중요한 신호다 — 스스로 구조화할 수 있는지를 본다
  • Functional(무엇을 하는가) + Non-functional(어떻게 동작해야 하는가) 두 가지로 나눠서 정리한다
  • Non-functional Requirements 중 반드시 확인할 것: 규모(DAU), 읽기/쓰기 비율, 지연 시간, 가용성, 일관성
  • Capacity estimation은 정확한 수치보다 올바른 사고 과정이 중요하다
  • 5단계 프레임워크: 요구사항 정리 → 규모 산정 → High-level 설계 → Deep dive → 마무리
  • 좋은 답변은 설계의 한계를 먼저 말하고 개선 방향을 제시하는 것이다
