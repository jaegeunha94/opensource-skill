Day 4 · 시스템 설계 Interview

캐시 전략과 CDN — Cache-aside, Write-through, 캐시 무효화

Day 3에서 서버를 stateless로 만들고 여러 대로 늘렸다. 그런데 피크 QPS 7만이 그대로 DB로 몰리면 이번엔 DB가 무너진다 — DB 앞에 무엇을 세울 것인가

이 레슨의 핵심 면접 질문

캐시는 시스템 설계 인터뷰에서 가장 자주 등장하지만, "Redis 쓰면 됩니다"로 끝내는 순간 신호를 잃는 주제다.

면접 질문 유형
  • "DB 부하를 줄이기 위해 캐시를 어디에, 어떻게 두시겠습니까?"
  • "Cache-aside와 Write-through의 차이가 뭔가요? 이 시스템에는 어떤 걸 쓰시겠어요?"
  • "캐시에 있는 데이터가 DB의 실제 값과 달라지면(stale data) 어떻게 하시겠습니까?"
  • "인기 있는 캐시 키 하나가 만료되는 순간 무슨 일이 벌어질 수 있나요?"
  • "정적 파일이 아닌 동적 콘텐츠에도 CDN을 쓸 수 있나요?"

이 질문들의 공통점: "캐시를 쓴다"는 정답이 아니라 출발점이다. 면접관은 캐시가 만들어내는 새로운 일관성 문제와 실패 모드를 아는지 본다.

전제 개념 — 캐시란 무엇이고 왜 새로운 문제를 만드는가?

캐시는 "빠른 곳에 데이터 사본을 둔다"는 단순한 아이디어지만, 사본이 생기는 순간 원본과 사본이 달라질 수 있다는 근본 문제가 따라온다.

캐시(Cache)란 무엇인가?

느리고 비싼 저장소(DB, 원본 서버)에 매번 접근하는 대신, 자주 쓰이는 데이터의 사본을 빠르고 가까운 곳(메모리, 엣지 서버)에 미리 저장해두고 그곳에서 응답하는 기법이다. 비유하면 "매번 창고까지 가서 물건을 꺼내오는 대신, 자주 찾는 물건은 계산대 옆 서랍에 미리 꺼내둔다"는 것과 같다. 서랍(캐시)에 있으면 빠르지만(cache hit), 없으면 결국 창고(원본, DB)까지 가야 한다(cache miss). 문제는 창고의 물건이 바뀌었는데 서랍의 사본은 그대로일 수 있다는 것 — 이게 캐시의 근본 난제인 "일관성" 문제다.

Cache-aside (Lazy Loading) — 가장 흔한 패턴

애플리케이션이 캐시를 먼저 확인하고, 있으면(hit) 바로 반환, 없으면(miss) DB에서 읽어와 응답하면서 그 값을 캐시에도 채워 넣는 방식이다. 캐시 로직을 애플리케이션이 직접 관리하기 때문에 "aside(옆에서)"라는 이름이 붙었다.

장점: 실제로 조회된 데이터만 캐시에 쌓이므로(캐시 오염이 적음) 메모리 효율이 좋고, 캐시가 통째로 죽어도 DB로 폴백하면 되므로 구조가 단순하다. 단점: 첫 요청은 항상 cache miss라 느리고(cold start), 쓰기 시점에 캐시를 명시적으로 무효화하거나 갱신해줘야 한다 — 이걸 깜빡하면 stale data가 남는다.

Write-through와 Write-behind(Write-back) — 쓰기 시점 전략

Write-through는 쓰기가 발생하면 캐시와 DB에 동시에(순차적으로 둘 다 성공해야) 쓴다. 캐시가 항상 최신 상태를 유지하므로 stale data 위험이 낮지만, 매 쓰기마다 DB까지 왕복해야 해서 쓰기 지연(latency)이 늘어난다.

Write-behind(Write-back)는 쓰기를 캐시에만 먼저 반영하고, DB에는 나중에 비동기로 배치 반영한다. 쓰기 응답이 매우 빠르지만, 캐시가 DB에 반영되기 전에 캐시 서버가 죽으면 그 사이의 쓰기가 유실될 수 있다는 위험을 안고 간다.

TTL과 캐시 무효화(Invalidation)

TTL(Time To Live)은 캐시 항목이 자동으로 만료되는 시간이다. TTL만으로는 "만료 전까지는 stale data를 허용한다"는 뜻이므로, 실시간성이 중요한 데이터는 쓰기 시점에 캐시를 명시적으로 지우거나(invalidate) 갱신해야 한다. 컴퓨터 과학에서 "이름 짓기와 캐시 무효화, 오프바이원 에러가 가장 어려운 두 가지 문제다"라는 농담이 있을 만큼, 캐시 무효화 타이밍과 범위를 정확히 맞추는 건 실무에서도 어렵다.

Cache Stampede(Thundering Herd)란?

매우 인기 있는 캐시 키(예: 메인 페이지 데이터) 하나가 만료되는 순간, 동시에 들어온 수천 개의 요청이 전부 cache miss가 되어 한꺼번에 DB로 몰려가는 현상이다. DB가 그 순간의 폭증한 부하를 못 견디고 응답이 느려지거나 죽을 수 있다. "캐시가 있으니 안전하다"고 생각하기 쉽지만, 캐시가 만료되는 그 짧은 순간에는 오히려 캐시가 없을 때보다 더 위험할 수 있다.

CDN(Content Delivery Network)이란?

전 세계 여러 지역에 분산 배치된 엣지 서버(edge server)에 콘텐츠 사본을 캐싱해두고, 사용자의 요청을 지리적으로 가장 가까운 엣지 서버가 응답하게 하는 서비스다(Cloudflare, CloudFront 등). 전통적으로 이미지·CSS·JS 같은 정적 파일에 쓰였지만, 짧은 TTL과 적절한 캐시 키 설계를 통해 API 응답 같은 준-동적(semi-dynamic) 콘텐츠에도 확장해 쓸 수 있다. 핵심 효과는 두 가지다 — 지연시간 단축(물리적 거리가 줄어듦)과 원본 서버(origin) 부하 감소(엣지에서 대부분의 요청을 흡수).

왜 면접에서 이것을 묻는가?

면접관의 관점

"캐시를 추가하겠습니다"는 성능 문제를 아는 사람이면 누구나 말한다. 면접관이 진짜 확인하고 싶은 것은 다음이다.

  • 일관성과 성능의 trade-off를 아는가? — 캐시를 넣는 순간 "완전히 최신 데이터"라는 보장을 일부 포기한다는 걸 인지하는지
  • 쓰기 패턴에 맞는 전략을 고르는가? — 읽기 위주 데이터와 쓰기가 잦은 데이터에 같은 전략을 무비판적으로 적용하지 않는지
  • 캐시의 실패 모드를 아는가? — Cache stampede, 캐시 서버 장애, 캐시 오염(cache pollution) 같은 상황을 미리 설계에 반영하는지
  • 캐시가 만능이 아님을 아는가? — 쓰기가 압도적으로 많은 데이터, 일관성이 절대적으로 중요한 데이터(예: 잔액)에는 캐시를 신중하게 적용하거나 배제하는지
실무 연결

실무에서 캐시 장애는 흔히 "DB가 갑자기 느려졌다"는 알람으로 나타난다. 원인을 따라가 보면 캐시 서버(Redis)가 재시작되거나 클러스터 페일오버가 발생해서 순간적으로 모든 요청이 cache miss가 되고, 그 트래픽이 그대로 DB로 넘어가 DB 커넥션 풀이 고갈되는 경우가 많다. 이런 사고를 겪은 팀은 이후 캐시 워밍(warming, 배포/재시작 직후 인기 키를 미리 채워두는 것), 요청 병합(request coalescing, 같은 키에 대한 동시 cache miss를 하나의 DB 조회로 묶는 것), circuit breaker(캐시가 죽으면 DB로 가는 트래픽 자체를 제한)를 도입한다. CDN 쪽에서도 배포 직후 콘텐츠가 즉시 반영되지 않는 문제(캐시 무효화 지연)가 실무에서 자주 발생하는데, 이를 해결하기 위해 파일명에 버전 해시를 붙이는 cache busting이나 CDN의 즉시 purge API를 함께 쓴다.

캐시 설계 판단 프레임워크

DB 앞에 캐시를 두기로 결정했다면, 아래 순서로 사고를 전개하면 빠뜨리는 것 없이 설계를 전달할 수 있다.

  1. 무엇을 캐싱할지 판단한다

    읽기 비중이 압도적으로 높고(read-heavy), 같은 데이터가 반복 조회되는지 확인한다. 쓰기가 잦고 매번 최신값이 필요한 데이터는 캐싱 우선순위가 낮다.

  2. 캐시를 어느 레이어에 둘지 결정한다

    클라이언트 캐시(브라우저), CDN(엣지), 분산 캐시(Redis/Memcached), 애플리케이션 로컬 메모리 캐시 중 데이터 특성에 맞는 레이어를 고른다. 여러 레이어를 함께 쓰는 경우가 실무에서는 흔하다.

  3. 읽기 전략을 선택한다 (Cache-aside 기본값)

    대부분의 경우 Cache-aside가 기본 선택이다 — 구현이 단순하고, 캐시 장애 시 DB로 자연스럽게 폴백된다.

  4. 쓰기 전략을 선택한다 (Write-through vs Write-behind)

    일관성이 중요하면 Write-through, 쓰기 응답 속도가 중요하고 약간의 유실 위험을 감수할 수 있으면 Write-behind를 선택한다.

  5. TTL과 무효화 전략을 설계한다

    데이터의 실시간성 요구 수준에 맞게 TTL을 정하고, 쓰기 발생 시 캐시를 지울지(invalidate) 갱신할지(update)를 정한다.

  6. Cache stampede 방지책을 넣는다

    인기 키에는 mutex lock(하나의 요청만 DB를 조회하고 나머지는 대기), probabilistic early expiration(만료 전에 확률적으로 미리 갱신), 또는 TTL을 키마다 살짝 다르게 분산시키는 jitter를 적용한다.

  7. 캐시 장애 시나리오를 설계한다

    캐시가 죽었을 때 DB로 트래픽이 그대로 몰리지 않도록 circuit breaker나 rate limiting을 두고, 재시작 시 캐시 워밍으로 cold cache 상태를 최소화한다.

실전 예시 — Day 3의 뉴스 피드 서비스 이어서 설계하기

Day 3에서 stateless 서버와 로드밸런서까지 갖췄다. 이제 피크 읽기 QPS 7만이 DB로 직접 몰리지 않도록 캐시를 붙여보자.

1) 캐싱 대상 판단

  • 인기 게시물, 사용자 프로필처럼 읽기 비중이 압도적으로 높고 자주 반복 조회되는 데이터를 캐싱 우선 대상으로 선정
  • 실시간 좋아요 카운트처럼 쓰기가 매우 잦은 데이터는 캐싱하되 짧은 TTL이나 별도 전략(카운터는 캐시에 두고 주기적으로 DB에 flush)을 적용

2) 레이어 구성

  • 정적 자산(이미지, 프로필 사진)은 CDN에 캐싱해서 원본 서버 부하를 줄인다
  • 피드 데이터, 게시물 상세 조회는 Redis 같은 분산 캐시에 Cache-aside 방식으로 둔다

3) 읽기/쓰기 전략

  • 게시물 조회는 Cache-aside — 첫 조회 시 DB에서 읽어 캐시에 채우고, 이후 요청은 캐시에서 응답
  • 게시물 수정/삭제 시에는 Write-through 방식으로 캐시와 DB를 함께 갱신하거나, 간단히 해당 캐시 키를 invalidate해서 다음 조회 때 다시 채우게 한다

4) Stampede 방지

  • 인기 게시물(핫 키)의 TTL이 동시에 만료되지 않도록 TTL에 랜덤 jitter(예: 300초 ± 30초)를 부여
  • 같은 키에 대한 동시 cache miss가 발생하면 첫 요청만 DB를 조회하고 나머지는 짧게 대기 후 캐시 값을 재사용하는 mutex lock(또는 request coalescing) 적용
여기서 시니어가 반드시 짚는 포인트

캐시를 도입하는 순간 "이 데이터는 최대 TTL만큼 오래된 값일 수 있다"는 걸 명시적으로 인지하고 설계에 반영해야 한다. 좋아요 수 같은 데이터는 몇 초 stale해도 괜찮지만, 결제 잔액 같은 데이터는 stale이 사고로 이어진다. "어떤 데이터는 캐싱하지 않는다"는 결정도 캐싱 전략의 일부라는 걸 보여주는 것이 시니어 신호다.

답변 예시 — 이렇게 말하면 된다

답변 예시 — 캐시 전략 설계 단계

"피크 읽기 QPS 7만이 그대로 DB로 가면 DB가 병목이 되므로, 읽기 비중이 높은 게시물·프로필 데이터에 Redis 기반 Cache-aside 전략을 적용하겠습니다. 정적 자산은 CDN으로 빼서 원본 서버까지 도달하는 트래픽 자체를 줄이겠습니다. 쓰기 시에는 해당 캐시 키를 invalidate해서 다음 조회 때 최신 데이터로 다시 채우는 방식을 쓰고, 인기 게시물처럼 트래픽이 몰리는 키는 TTL에 jitter를 줘서 동시 만료를 피하고, 동시 cache miss 상황에는 mutex lock으로 DB에 한 번만 조회하도록 해서 cache stampede로 인한 DB 과부하를 막겠습니다."

강한 답변과 약한 답변의 차이

강한 답변 (Senior level)

"읽기 위주 데이터는 Cache-aside로 Redis에 캐싱하되, TTL과 함께 쓰기 시점 invalidation을 병행해서 stale window를 최소화하겠습니다. 인기 키의 cache stampede를 막기 위해 TTL jitter와 mutex lock을 적용하고, 캐시 서버 자체가 죽는 상황에 대비해 circuit breaker로 DB 트래픽을 제한하고 재시작 시 인기 키를 미리 채우는 캐시 워밍을 두겠습니다. 다만 결제 잔액처럼 절대적 일관성이 필요한 데이터는 캐싱 대상에서 제외하거나 매우 짧은 TTL만 허용하겠습니다."

약한 답변 (Junior level)

"Redis를 앞에 두고 캐싱하면 됩니다. DB 부하가 줄어들 거예요. (무효화 전략, stampede, 캐시 장애 시나리오, 어떤 데이터를 캐싱하면 안 되는지는 언급 없음)"

왜 약한 답변이 약한가?

  • 일관성 문제 인지 부족: 캐시를 넣으면 stale data가 생길 수 있다는 전제 자체를 언급하지 않는다 — 실제로 이 문제를 만나면 당황한다
  • 무효화 전략 없음: 쓰기가 발생했을 때 캐시를 어떻게 처리할지에 대한 계획이 없다
  • 실패 모드 고려 없음: 캐시 서버 장애, cache stampede 같은 상황에서 시스템이 어떻게 무너지는지 생각하지 못한다
  • 모든 데이터를 동일하게 취급: 캐싱하면 안 되거나 신중해야 하는 데이터(잔액, 재고 등)를 구분하지 못한다

답변 사고 순서 — 실전 적용

머릿속 흐름 (캐시 전략 설계 3~5분 기준)

[0~30초] DB 부하의 원인이 반복적인 읽기 요청임을 짚고, 캐시가 필요한 이유를 QPS 숫자로 뒷받침한다.

[30초~1.5분] 캐시 레이어(CDN, 분산 캐시, 로컬 캐시)를 데이터 종류별로 나눠 배치한다.

[1.5~2.5분] 읽기 전략(Cache-aside)과 쓰기 전략(Write-through/Write-behind/invalidation)을 선택하고 근거를 말한다.

[2.5~4분] Cache stampede 방지책과 캐시 장애 시 폴백 전략을 설계한다.

[4~5분] "이 데이터는 캐싱하지 않는다/짧게만 캐싱한다"는 예외를 짚어서 trade-off를 이해하고 있음을 보여준다.

Senior-Level Follow-up 질문과 답변 방향

캐시 레이어를 잘 그렸다고 끝이 아니다. 면접관은 일관성 실패와 장애 상황을 파고든다.

Follow-up 1

Q: "캐시 히트율(hit ratio)이 갑자기 떨어졌습니다. 어떻게 진단하시겠어요?"

답변 방향: "먼저 캐시 서버 자체가 재시작되거나 페일오버되어 캐시가 비워졌는지(cold cache) 확인합니다. 그다음 TTL이 너무 짧게 설정되어 있거나, 캐시 키 설계가 바뀌어서(예: 쿼리 파라미터가 추가되어 캐시 키가 세분화됨) 같은 데이터인데도 다른 키로 취급되고 있는지 봅니다. 또한 트래픽 패턴 자체가 바뀌어서(새로운 종류의 요청이 많아짐) 캐시에 없는 데이터에 대한 요청이 늘었을 수도 있습니다. 모니터링에서 hit ratio와 함께 캐시 서버 재시작 이력, TTL 설정, 키 분포를 같이 봐야 원인을 좁힐 수 있습니다."

Follow-up 2

Q: "Cache stampede를 구체적으로 어떻게 막으시겠어요?"

답변 방향: "세 가지를 조합합니다. 첫째, TTL에 랜덤 jitter를 줘서 같은 시각에 만료되는 키를 분산시킵니다. 둘째, 특정 키가 cache miss일 때 첫 번째 요청만 락을 획득해서 DB를 조회하고 캐시를 채우게 하고, 나머지 동시 요청은 짧게 대기했다가 갱신된 캐시 값을 읽게 하는 mutex 패턴(request coalescing)을 씁니다. 셋째, 매우 인기 있는 키는 TTL 만료 직전에 백그라운드에서 미리 갱신하는 probabilistic early expiration을 적용해서, 사용자 요청이 만료 순간과 맞닥뜨리는 확률 자체를 줄입니다."

Follow-up 3

Q: "Write-through와 Write-behind 중 어떤 걸 선택하시겠어요?"

답변 방향: "trade-off가 있습니다. Write-through는 캐시와 DB에 동시에 써야 하니 쓰기 지연이 늘지만, 캐시가 항상 최신 상태라 일관성이 높습니다. Write-behind는 캐시에만 먼저 쓰고 DB 반영을 비동기로 미루니 쓰기 응답은 빠르지만, DB에 반영되기 전에 캐시 서버가 죽으면 그 사이의 쓰기가 유실될 위험이 있습니다. 저는 결제나 재고처럼 유실이 곧 사고로 이어지는 데이터는 Write-through, 조회수·좋아요 카운트처럼 약간의 유실을 감수할 수 있는 데이터는 Write-behind로 처리해서 트래픽을 흡수하고 주기적으로 배치로 DB에 반영하겠습니다."

Follow-up 4

Q: "CDN에 캐싱된 콘텐츠를 즉시 갱신해야 하는데, 최대 몇 시간씩 안 바뀝니다. 어떻게 해결하시겠어요?"

답변 방향: "두 가지 접근이 있습니다. 하나는 파일명에 콘텐츠 해시나 버전을 붙이는 cache busting입니다 — 파일 내용이 바뀌면 URL 자체가 바뀌므로 TTL을 아주 길게 잡아도 무효화 문제가 원천적으로 없습니다. 정적 자산(JS, CSS, 이미지)에 적합합니다. 다른 하나는 CDN이 제공하는 즉시 purge API를 호출해서 특정 경로의 캐시를 강제로 지우는 방법인데, 이건 Push 방식이라 즉시 반영되지만 전체 엣지 서버에 전파되는 데 약간의 지연이 있고 API 호출 비용도 있어 자주 바뀌는 콘텐츠에는 남용하지 않는 게 좋습니다. 정적 자산은 cache busting, 자주 바뀌지만 드물게 강제 갱신이 필요한 콘텐츠는 purge API로 구분해서 쓰겠습니다."

Trade-off와 흔한 오해

오해 1: "캐시를 추가하면 무조건 빨라진다"

실제: Cache miss가 잦으면 캐시 조회 후 다시 DB를 조회하는 추가 홉이 생겨 오히려 느려질 수 있다. 캐시는 hit ratio가 충분히 높을 때만 효과가 있으며, 데이터 접근 패턴이 골고루 퍼져 있는(hit ratio가 낮은) 경우에는 캐시 도입 효과가 크지 않다.

오해 2: "TTL만 짧게 하면 일관성 문제가 해결된다"

실제: TTL을 짧게 하면 stale window는 줄어들지만 여전히 그 짧은 시간 동안은 stale data가 존재하고, 오히려 TTL이 짧을수록 cache miss가 잦아져 DB 부하가 늘고 cache stampede 위험도 커진다. 진짜 실시간성이 필요하면 TTL 조정이 아니라 쓰기 시점의 명시적 invalidation이 필요하다.

오해 3: "Write-through가 항상 더 안전하니 기본값으로 써야 한다"

실제: Write-through는 모든 쓰기가 DB 왕복을 기다려야 해서 쓰기가 잦은 시스템에서는 지연시간이 누적된다. 일관성이 절대적으로 필요하지 않은 데이터(조회수 등)까지 Write-through를 적용하면 불필요한 성능 저하를 감수하는 것이다.

오해 4: "CDN은 정적 파일에만 쓸 수 있다"

실제: 짧은 TTL(수 초~수 분)과 적절한 캐시 키(쿼리 파라미터, 인증 여부 구분) 설계를 통해 API 응답 같은 준-동적 콘텐츠도 CDN에서 캐싱할 수 있다. 예를 들어 로그인하지 않은 사용자에게 보여지는 공개 API 응답은 짧게라도 CDN에 캐싱해 원본 서버 부하를 크게 줄일 수 있다.

Trade-off: Cache-aside vs Write-through — 구현 단순성과 일관성의 교환

Cache-aside는 구현이 단순하고 캐시 장애에 강하지만(DB로 자연스럽게 폴백), 쓰기 시점에 invalidation을 깜빡하면 stale data가 남는 실수가 잦다. Write-through는 일관성이 높지만 매 쓰기마다 캐시와 DB 양쪽에 성공적으로 반영되어야 하므로 구현이 더 복잡하고 쓰기 지연이 늘어난다. 실무에서는 읽기 위주 데이터는 Cache-aside, 일관성이 중요한 일부 데이터만 Write-through로 섞어서 적용하는 경우가 많다.

핵심 정리

  • Cache-aside는 읽기 시점에 캐시를 채우는 기본 전략이고, Write-through/Write-behind는 쓰기 시점의 일관성-지연시간 trade-off를 조정하는 전략이다
  • 캐시를 넣는 순간 stale data 가능성을 설계에 명시적으로 반영해야 한다 — TTL과 invalidation을 함께 설계한다
  • Cache stampede는 인기 키의 동시 만료로 DB가 폭증 부하를 받는 현상이다 — TTL jitter, mutex lock, early expiration으로 방지한다
  • CDN은 정적 자산뿐 아니라 짧은 TTL로 준-동적 콘텐츠에도 적용할 수 있다 — cache busting과 purge API를 상황에 맞게 구분해서 쓴다
  • 모든 데이터를 캐싱해야 하는 건 아니다 — 일관성이 절대적인 데이터는 캐싱을 배제하거나 신중하게 적용하는 것도 설계의 일부다
