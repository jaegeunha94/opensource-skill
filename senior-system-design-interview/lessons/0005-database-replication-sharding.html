Day 5 · 시스템 설계 Interview

데이터베이스 확장 — 복제와 샤딩, Hot Shard 문제

Day 4에서 캐시로 읽기 트래픽 대부분을 흡수했다. 그런데 쓰기 트래픽과 데이터 자체의 크기는 캐시가 손대지 못한다 — DB 서버 한 대의 한계에 부딪히면 무엇을 어떻게 나눌 것인가

이 레슨의 핵심 면접 질문

DB 확장은 시스템 설계 인터뷰에서 가장 변별력이 큰 주제다. "복제하면 됩니다", "샤딩하면 됩니다"까지는 누구나 말하지만, 그 이후에 생기는 새로운 문제를 다루는 능력에서 시니어와 주니어가 갈린다.

면접 질문 유형
  • "DB 서버 한 대로 트래픽을 감당할 수 없습니다. 어떻게 확장하시겠습니까?"
  • "Read Replica를 두면 어떤 문제가 새로 생기나요?"
  • "샤딩 키(shard key)는 어떤 기준으로 고르시겠습니까?"
  • "특정 샤드에만 트래픽이 몰리는 Hot Shard 문제는 어떻게 대응하시겠어요?"
  • "샤딩된 DB에서 여러 샤드에 걸친 JOIN이나 트랜잭션은 어떻게 처리하나요?"

이 질문들의 공통점: "나눈다"는 답은 시작일 뿐이다. 면접관은 나누는 순간 사라지는 것(단일 트랜잭션, 단일 JOIN, 단일 진실 공급원)을 알고 그 손실을 어떻게 다루는지 본다.

전제 개념 — 복제와 샤딩은 서로 다른 문제를 푼다

둘 다 "DB를 여러 대로 늘린다"는 점은 같지만, 푸는 문제가 다르다. 이 차이를 먼저 명확히 하지 않으면 면접에서 두 개념을 섞어서 말하게 된다.

복제(Replication)란 무엇인가? — 같은 데이터를 여러 벌 두기

복제는 같은 데이터의 사본을 여러 서버에 두는 것이다. 하나를 Primary(또는 Master)로 두고 모든 쓰기를 여기로 보내며, 나머지 서버는 Replica(또는 Slave)로 두고 Primary의 변경 사항을 그대로 따라간다. 읽기 요청을 여러 Replica로 분산시키면 읽기 처리량(read throughput)을 늘릴 수 있고, Primary가 죽었을 때 Replica 중 하나를 승격시켜 가용성도 높일 수 있다. 비유하면 "같은 장부를 여러 벌 필사해서 여러 창구에 배치하는 것"과 같다 — 장부의 내용은 같지만 개수만 늘어난다.

복제 지연(Replication Lag)이란?

Primary에 쓰기가 반영된 시점과 그 변경이 Replica에 실제로 도착해 반영되는 시점 사이에는 항상 시간차가 있다. 이걸 복제 지연이라 한다. 네트워크 지연, Replica의 처리 속도, 쓰기량 폭증 등으로 이 지연이 커질 수 있다. 문제는 사용자가 글을 쓴 직후 바로 그 글을 조회했는데, 조회 요청이 아직 그 변경을 반영하지 못한 Replica로 갔다면 "방금 쓴 내 글이 안 보이는" 상황이 발생한다는 것이다 — 이게 복제가 만드는 새로운 일관성 문제다.

샤딩(Sharding)이란 무엇인가? — 같은 데이터를 여러 조각으로 나누기

샤딩은 복제와 반대로, 하나의 큰 데이터셋을 여러 개의 작은 조각(샤드)으로 나눠 서로 다른 서버에 저장하는 것이다. 예를 들어 사용자 1억 명의 데이터를 user_id 범위나 해시값에 따라 10개의 샤드로 나누면, 각 샤드는 전체의 10분의 1만 담당한다. 비유하면 "하나의 거대한 장부를 여러 권으로 쪼개서 각 창구가 서로 다른 페이지 범위만 담당하게 하는 것"이다 — 장부의 내용은 서로 다른 부분이고, 각 서버가 담당하는 데이터 양 자체가 줄어든다. 복제가 읽기 처리량과 가용성을 위한 것이라면, 샤딩은 쓰기 처리량과 저장 용량 자체의 한계를 넘기 위한 것이다.

샤딩 키(Shard Key) — 어떤 기준으로 나눌 것인가?

샤딩 키는 어떤 데이터가 어느 샤드에 저장될지를 결정하는 기준 컬럼이다. 크게 두 가지 전략이 있다.

Range-based Sharding(범위 기반): user_id 1~1000만은 샤드1, 1000만~2000만은 샤드2처럼 값의 범위로 나눈다. 범위 조회(예: "최근 가입자 조회")가 쉽지만, 신규 가입자가 몰리는 마지막 샤드에 트래픽이 집중되는 Hot Shard 문제가 생기기 쉽다.
Hash-based Sharding(해시 기반): user_id를 해시 함수에 넣어 나온 값으로 샤드를 결정한다. 데이터가 여러 샤드에 고르게 분산되어 Hot Shard 위험이 낮지만, 범위 조회가 어려워지고(1번부터 100번까지 순서대로 조회하려면 모든 샤드를 다 뒤져야 함) 샤드 개수를 바꿀 때(re-sharding) 대량의 데이터 재배치가 필요하다.

Hot Shard(핫 샤드) 문제란?

샤딩을 했다고 해서 트래픽이 항상 균등하게 나뉘는 건 아니다. 유명인의 계정처럼 특정 데이터 하나에 접근이 몰리거나, 샤딩 키를 잘못 골라(예: 가입일 기준 range sharding) 특정 샤드에만 활성 사용자가 몰리면, 그 샤드 하나만 과부하가 걸려 전체 시스템의 병목이 된다. 나머지 9개 샤드는 한가한데 1개 샤드만 죽어가는 상황 — 샤딩을 했음에도 확장성이 사라지는 역설적인 상황이다.

분산 트랜잭션과 Cross-shard Join의 어려움

하나의 DB 안에서는 당연했던 것들이 샤딩 이후에는 어려워진다. 사용자 A의 계좌(샤드1)에서 사용자 B의 계좌(샤드3)로 송금하는 트랜잭션은 더 이상 하나의 로컬 트랜잭션으로 처리할 수 없다 — 두 샤드에 걸친 분산 트랜잭션이 필요하다. 마찬가지로 "샤드1의 주문 테이블과 샤드3의 사용자 테이블을 JOIN"하는 쿼리도 DB가 자동으로 해주지 않는다. 애플리케이션이 직접 여러 샤드에 쿼리를 보내고 결과를 합쳐야 한다(scatter-gather). 이것이 "샤딩은 마지막 수단으로 고려해야 한다"고 말하는 이유다 — 얻는 확장성만큼 잃는 것(단순한 트랜잭션, 단순한 JOIN)이 크다.

왜 면접에서 이것을 묻는가?

면접관의 관점

DB 확장 질문은 "확장 방법을 아는가"가 아니라 "확장이 만드는 새로운 제약을 감당할 수 있는가"를 본다.

  • 복제와 샤딩의 목적 차이를 구분하는가? — 읽기 확장(복제)과 쓰기/용량 확장(샤딩)을 혼동하지 않는지
  • 복제 지연이 만드는 일관성 문제를 아는가? — "방금 쓴 데이터가 안 보이는" read-after-write 문제를 인지하고 대응책이 있는지
  • 샤딩 키 선택의 trade-off를 판단할 수 있는가? — 접근 패턴을 분석해서 Hot Shard를 최소화하는 키를 고르는지
  • 샤딩을 최후의 수단으로 여기는가? — 수직 확장(더 큰 서버), 복제, 캐시로 충분히 버틸 수 있는데도 성급하게 샤딩부터 꺼내지 않는지
  • 분산 시스템의 대가를 아는가? — cross-shard 트랜잭션, JOIN의 어려움을 실제로 겪어본 사람처럼 설명하는지
실무 연결

실무에서 "복제 지연으로 인한 read-after-write 문제"는 흔히 "방금 결제했는데 주문 내역에 안 보여요"라는 고객 문의로 나타난다. 원인을 따라가면 결제 API가 Primary에 쓰기를 완료하자마자, 주문 목록 조회 API는 Read Replica로 요청을 보냈고 그 Replica가 아직 그 쓰기를 따라잡지 못한 경우다. 이런 사고를 겪은 팀은 "쓰기 직후 일정 시간 동안은 Primary에서 읽기" 또는 "세션 내에서 자신이 쓴 데이터는 Primary에서 읽기"(read-your-writes) 정책을 도입한다. Hot Shard 문제는 실무에서 "특정 고객사(대형 테넌트)의 데이터가 몰린 샤드만 CPU 100%를 찍는다"는 형태로 나타나는데, 이 경우 해당 테넌트만 별도 샤드로 격리하거나(dedicated shard), 샤드 내부에서 다시 더 잘게 나누는 re-sharding으로 대응한다.

DB 확장 판단 프레임워크

DB가 한계에 부딪혔다고 바로 샤딩으로 가지 않는다. 아래 순서로 "가장 단순하고 되돌리기 쉬운 방법부터" 검토하는 것이 시니어의 사고 순서다.

  1. 병목이 읽기인지 쓰기인지 저장 용량인지 구분한다

    읽기가 병목이면 캐시나 Read Replica로 상당 부분 해결된다. 쓰기가 병목이거나 데이터 크기 자체가 한 대의 디스크/메모리 한계를 넘으면 샤딩을 고려한다.

  2. 수직 확장(Scale-up)으로 충분한지 먼저 확인한다

    더 큰 CPU/메모리/디스크의 서버로 옮기는 것이 가장 단순하고 애플리케이션 변경이 없다. 한계가 명확히 보일 때만 다음 단계로 간다.

  3. Read Replica로 읽기를 분산한다

    읽기 비중이 높다면 Primary-Replica 구조로 읽기 트래픽을 여러 대로 나눈다. 이때 복제 지연으로 인한 일관성 문제를 함께 설계한다.

  4. 그래도 부족하면 샤딩을 도입한다 (쓰기/용량 병목일 때)

    샤딩은 애플리케이션 복잡도를 크게 높이므로, 위 단계로 해결이 안 될 때의 마지막 수단으로 접근한다.

  5. 접근 패턴을 분석해 샤딩 키를 고른다

    가장 빈번한 쿼리가 어떤 컬럼으로 필터링/조회되는지 파악하고, 그 컬럼을 기준으로 데이터가 고르게 분산되도록 해시 기반 또는 범위 기반 전략을 선택한다.

  6. Hot Shard 가능성을 미리 점검한다

    특정 값(유명 계정, 대형 테넌트)에 트래픽이 몰릴 가능성을 시뮬레이션하고, 필요하면 해당 값만 별도로 격리하거나 더 잘게 나누는 전략을 마련한다.

  7. Cross-shard 연산이 필요한 지점을 미리 식별한다

    여러 샤드에 걸친 JOIN이나 트랜잭션이 필요한 기능을 미리 찾아, 애플리케이션 레벨에서 처리하거나 데이터 모델을 조정해 그런 연산 자체를 최소화한다.

실전 예시 — Day 4의 뉴스 피드 서비스 이어서 설계하기

Day 4까지 캐시로 읽기 부하를 크게 줄였다. 하지만 DAU 1000만, 하루 신규 게시물 5000만 건이 쌓이면서 게시물 테이블 자체가 단일 DB 서버의 저장 용량과 쓰기 처리량 한계에 다가가고 있다.

1) 병목 진단

  • 읽기는 Day 4의 캐시가 대부분 흡수 중이므로, 실제 병목은 게시물 테이블의 쓰기량과 누적 데이터 크기다
  • 단일 DB 서버로는 디스크 용량과 쓰기 처리량(IOPS) 모두 한계에 다다름을 모니터링 지표로 확인

2) 단계적 접근

  • 먼저 Read Replica 2~3대를 추가해 남은 읽기 트래픽(캐시 미스, 관리자 조회 등)을 분산
  • 쓰기량과 데이터 크기 자체가 한계이므로 Replica만으로는 해결 안 됨을 확인하고 샤딩으로 진행

3) 샤딩 키 선택

  • 가장 빈번한 쿼리 패턴은 "특정 사용자의 게시물 조회"이므로 user_id를 해시한 값을 샤딩 키로 선택
  • range 기반(가입일 순)은 신규 가입자가 몰리는 마지막 샤드에 쓰기가 집중되는 Hot Shard 위험이 크므로 배제

4) Hot Shard 대응

  • 팔로워 수백만 명의 인플루언서 계정은 해시로 나눠도 해당 계정의 게시물 조회 자체가 몰릴 수 있으므로, 이런 계정은 Day 4의 캐시 레이어에서 더 긴 TTL과 별도 캐시 용량을 할당해 DB까지 도달하는 트래픽을 최소화

5) Cross-shard 연산 최소화

  • "팔로우한 사용자들의 게시물을 시간순으로 합쳐 보여주는" 뉴스 피드 생성은 여러 샤드에 걸친 조회가 필요한 대표적 사례 — 이 문제는 Day 10에서 Fanout 전략으로 별도로 다룬다
  • 게시물 자체의 CRUD는 user_id 하나로 완결되므로 대부분 단일 샤드 내에서 처리 가능하도록 설계
여기서 시니어가 반드시 짚는 포인트

샤딩은 "확장성을 얻는 대신 단순성을 잃는" 거래다. 샤딩을 제안하기 전에 "이 문제가 정말 샤딩이 아니면 안 풀리는 문제인가"를 먼저 검증해야 한다. 실무에서 흔한 실수는 성급하게 샤딩을 도입한 뒤, 정작 자주 필요한 cross-shard JOIN 때문에 오히려 코드가 복잡해지고 성능도 나빠지는 경우다. "아직 샤딩할 때가 아니다"라고 판단하는 것도 훌륭한 답이라는 걸 보여주는 것이 시니어 신호다.

답변 예시 — DB 확장 설계 단계

"먼저 병목이 읽기인지 쓰기/용량인지 확인하겠습니다. 읽기 병목이라면 캐시와 Read Replica로 상당 부분 해결되므로, Replica 2~3대를 두고 읽기를 분산하되 복제 지연으로 인해 '방금 쓴 데이터가 안 보이는' 문제를 막기 위해 쓰기 직후 일정 시간은 Primary에서 읽도록 하겠습니다. 쓰기량과 데이터 크기 자체가 한계라면 그때 샤딩을 도입하겠습니다. 샤딩 키는 가장 빈번한 쿼리 패턴인 user_id 기준으로, Hot Shard를 피하기 위해 range가 아닌 해시 기반으로 고르겠습니다. 다만 인플루언서 계정처럼 특정 값에 트래픽이 몰릴 가능성은 해시 샤딩만으로 완전히 막을 수 없으므로, 이런 핫 키는 캐시 레이어에서 별도로 흡수하도록 하겠습니다."

강한 답변과 약한 답변의 차이

강한 답변 (Senior level)

"읽기와 쓰기/용량 병목을 구분해서 단계적으로 접근하겠습니다. 읽기는 Replica로, 그래도 부족한 쓰기/용량 문제만 샤딩으로 풉니다. 샤딩 키는 실제 쿼리 패턴을 분석해 해시 기반으로 골라 Hot Shard를 최소화하고, 그럼에도 특정 값에 트래픽이 몰릴 가능성은 별도 격리나 캐시로 보완하겠습니다. Cross-shard JOIN이 필요한 기능은 미리 식별해서 데이터 모델을 조정하거나 애플리케이션 레벨에서 scatter-gather로 처리하겠습니다."

약한 답변 (Junior level)

"DB를 여러 대로 샤딩하면 됩니다. 그러면 부하가 나뉘어서 처리량이 늘어납니다. (복제와 샤딩의 목적 구분, 복제 지연, 샤딩 키 선택 기준, Hot Shard, cross-shard 연산의 어려움은 언급 없음)"

왜 약한 답변이 약한가?

  • 복제와 샤딩을 구분하지 않음: 읽기 확장 문제인지 쓰기/용량 확장 문제인지 진단 없이 바로 샤딩을 꺼낸다
  • 샤딩 키 선택 근거 없음: 어떤 기준으로 나눌지, 왜 그 기준이 접근 패턴에 맞는지 설명하지 못한다
  • Hot Shard 인지 부족: 샤딩했다고 트래픽이 항상 고르게 나뉘는 게 아니라는 걸 모른다
  • 대가를 모름: 샤딩 이후 JOIN과 트랜잭션이 얼마나 복잡해지는지, 그걸 어떻게 다룰지에 대한 계획이 없다

답변 사고 순서 — 실전 적용

머릿속 흐름 (DB 확장 설계 3~5분 기준)

[0~30초] 병목이 읽기인지, 쓰기/용량인지 구분하고 그 근거(지표)를 짚는다.

[30초~1.5분] 수직 확장 → Read Replica 순으로 단순한 해법부터 검토했음을 보여준다.

[1.5~2.5분] Replica만으로 부족한 이유(쓰기/용량 병목)를 설명하고 샤딩 도입을 결정한다.

[2.5~4분] 접근 패턴 기반 샤딩 키 선택과 그 근거, Hot Shard 대응책을 설명한다.

[4~5분] 샤딩 이후 cross-shard 연산이 필요한 지점을 짚고 대응 방향을 제시한다.

Senior-Level Follow-up 질문과 답변 방향

샤딩 구조를 잘 그렸다고 끝이 아니다. 면접관은 복제 지연으로 인한 일관성 문제와 재샤딩(re-sharding)의 어려움을 파고든다.

Follow-up 1

Q: "Read Replica의 복제 지연 때문에 생기는 일관성 문제는 어떻게 해결하시겠어요?"

답변 방향: "세 가지 방법을 조합할 수 있습니다. 첫째, 쓰기 직후 일정 시간(예: 몇 초) 동안은 해당 사용자의 요청을 Primary로 보내는 'read-your-writes' 정책입니다. 둘째, 세션이나 요청 컨텍스트에 마지막 쓰기 시점을 기록해두고, 그 시점을 아직 따라잡지 못한 Replica는 회피하는 방식입니다. 셋째, 애초에 일관성이 중요한 요청(자신이 방금 작성한 데이터 조회)은 Primary로, 그 외의 일반 조회는 Replica로 라우팅하는 규칙을 애플리케이션 레벨에서 명확히 나누는 것입니다. 모든 읽기를 무조건 Replica로 보내지 않고, 데이터 특성에 따라 라우팅을 다르게 하는 게 핵심입니다."

Follow-up 2

Q: "샤드 개수를 나중에 늘려야 한다면(re-sharding) 어떻게 하시겠어요?"

답변 방향: "해시 기반 샤딩에서 샤드 개수를 단순히 늘리면 거의 모든 키의 해시값이 다른 샤드로 재배정되어 대량의 데이터 이동이 필요합니다. 이를 완화하기 위해 consistent hashing을 씁니다 — 샤드를 추가해도 이동해야 하는 데이터가 전체의 일부(이상적으로는 1/N)로 제한됩니다. 또한 처음부터 실제 서버 대수보다 훨씬 많은 논리적 샤드(예: 물리 서버 10대에 논리 샤드 1000개)를 만들어두면, 나중에 서버를 늘릴 때 논리 샤드를 물리 서버 간에 재배치하기만 하면 되므로 애플리케이션의 샤딩 키 로직 자체는 바꾸지 않아도 됩니다."

Follow-up 3

Q: "특정 대형 고객사(테넌트)의 데이터만 있는 샤드에 트래픽이 몰려서 그 샤드만 CPU 100%입니다. 어떻게 하시겠어요?"

답변 방향: "먼저 모니터링으로 어떤 테넌트 또는 어떤 키 범위가 해당 샤드에 몰려 있는지 확인합니다. 확인되면 두 가지 선택이 있습니다. 첫째, 그 대형 테넌트만 별도의 전용 샤드(dedicated shard)로 분리해서 다른 테넌트에 영향이 가지 않게 격리합니다. 둘째, 해당 샤드 내부를 다시 더 잘게 나누는 re-sharding을 적용합니다. 장기적으로는 샤딩 키 설계 시점에 '테넌트 크기'라는 변수를 고려하지 않은 것이 근본 원인이므로, 향후에는 테넌트 크기에 따라 다른 샤딩 전략(소형 테넌트는 여러 개를 한 샤드에 묶고, 대형 테넌트는 전용 샤드를 할당)을 쓰는 hybrid 접근을 제안하겠습니다."

Follow-up 4

Q: "샤드 A의 계좌에서 샤드 B의 계좌로 송금하는 트랜잭션은 어떻게 처리하시겠어요?"

답변 방향: "단일 DB의 로컬 트랜잭션으로는 처리할 수 없으므로, 애플리케이션 레벨에서 보상 트랜잭션(Saga) 패턴을 씁니다. 샤드 A에서 출금을 먼저 커밋하고, 샤드 B에서 입금을 커밋합니다. 만약 입금 단계가 실패하면 샤드 A의 출금을 취소하는 보상 트랜잭션을 실행합니다. 이 과정에서 '출금은 됐는데 입금이 안 된' 중간 상태가 잠시 노출될 수 있으므로, 각 단계는 반드시 멱등하게 설계하고 전체 송금 요청에 고유 ID를 부여해 재시도 시 중복 처리되지 않도록 합니다. 2단계 커밋(2PC) 같은 강한 분산 트랜잭션 프로토콜도 있지만, 가용성을 떨어뜨리고 구현이 복잡해 실무에서는 Saga + 멱등성 조합을 더 선호하는 편입니다."

Trade-off와 흔한 오해

오해 1: "샤딩하면 무조건 성능이 좋아진다"

실제: 샤딩 키를 잘못 고르면 특정 샤드에 트래픽이 몰리는 Hot Shard가 생겨 오히려 병목이 그대로 남는다. 또한 cross-shard 연산이 잦은 데이터 모델이라면 샤딩 이후 애플리케이션 복잡도와 지연시간이 오히려 늘어날 수 있다. 샤딩은 접근 패턴을 정확히 분석한 뒤에만 효과가 있다.

오해 2: "Read Replica를 늘리면 일관성 문제도 자동으로 해결된다"

실제: Replica는 읽기 처리량을 늘려주지만 복제 지연이라는 새로운 문제를 만든다. Replica 개수를 늘리는 것과 일관성 문제(read-after-write)를 해결하는 것은 별개의 문제이며, 후자는 라우팅 정책이나 애플리케이션 로직으로 별도로 다뤄야 한다.

오해 3: "샤딩 키는 골고루 분산되기만 하면 최선이다"

실제: 순수하게 균등 분산만 최적화하면(예: 완전 랜덤 해시) 범위 조회나 특정 그룹 단위 조회(예: 같은 테넌트의 모든 데이터 조회)가 여러 샤드에 걸쳐 흩어져 오히려 비효율적일 수 있다. 실제 쿼리 패턴(어떤 데이터를 함께 조회하는지)까지 고려해서 샤딩 키를 골라야 한다.

오해 4: "샤딩은 처음부터 해두는 게 안전하다"

실제: 이른 샤딩(premature sharding)은 불필요한 복잡도를 처음부터 떠안는 것이다. 단일 DB에 인덱스 최적화, 캐시, Read Replica로 상당 기간 버틸 수 있는데도 샤딩부터 설계하면, 개발 속도가 느려지고 팀 전체가 분산 시스템의 복잡도를 조기에 감당해야 한다. 실제로 그 규모에 도달했을 때 샤딩하는 것이 합리적인 순서다.

Trade-off: Range-based vs Hash-based Sharding

Range 기반은 범위 조회(정렬된 데이터 조회)가 쉽고 구현이 직관적이지만, 특정 범위(신규 데이터, 인기 범위)에 트래픽이 몰리는 Hot Shard 위험이 크다. Hash 기반은 데이터가 고르게 분산되어 Hot Shard 위험이 낮지만, 범위 조회가 사실상 불가능해지고 샤드 개수 변경 시 대량의 데이터 재배치가 필요하다(consistent hashing으로 완화 가능). 실무에서는 접근 패턴에 range 조회가 필수적인지(시계열 데이터 등) 여부로 둘 중 하나를 선택하거나, 두 방식을 계층적으로 조합하기도 한다.

핵심 정리

  • 복제는 같은 데이터의 사본을 늘려 읽기 처리량과 가용성을 높이고, 샤딩은 데이터를 조각내어 쓰기 처리량과 저장 용량의 한계를 넘는다 — 둘은 다른 문제를 푼다
  • 복제는 복제 지연이라는 새 일관성 문제를 만든다 — read-your-writes 같은 라우팅 정책으로 보완한다
  • 샤딩 키는 실제 쿼리 패턴을 근거로 골라야 하며, Range는 범위 조회에 유리하지만 Hot Shard에 취약하고 Hash는 분산에 유리하지만 범위 조회와 재샤딩이 어렵다
  • Hot Shard는 샤딩 이후에도 특정 키에 트래픽이 몰리는 문제로, 격리나 캐시로 별도 보완이 필요하다
  • 샤딩은 cross-shard 트랜잭션/JOIN의 복잡도라는 대가를 치르므로, 수직 확장과 복제로 해결이 안 될 때의 마지막 수단으로 접근한다
