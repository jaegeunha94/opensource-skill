Capacity Estimation 심화 — QPS, 저장 용량, 대역폭 계산

숫자를 못 내면 뒤의 모든 설계가 근거를 잃는다 — DAU 하나로 QPS, 저장 용량, 서버 대수까지 뽑아내는 법

이 레슨의 핵심 면접 질문

Day 1에서 요구사항을 정리했다면, 면접관은 곧바로 숫자로 검증하려 든다.

면접 질문 유형
  • "DAU가 1억 명이면 초당 요청 수(QPS)는 대략 얼마나 될까요?"
  • "이 서비스에 5년 동안 필요한 저장 용량은 어느 정도인가요?"
  • "피크 시간대 트래픽은 어떻게 반영하실 건가요?"
  • "이 QPS를 처리하려면 서버가 몇 대 필요할까요?"
  • "네트워크 대역폭은 충분한가요? 어디서 병목이 생길까요?"

이 질문들의 공통점: 정답이 하나의 숫자가 아니다. 면접관은 계산 과정을 본다.

전제 개념 — Capacity Estimation이란 무엇인가?

계산 자체는 사칙연산이다. 그런데도 많은 후보자가 여기서 막힌다. 왜 막히는지부터 이해하자.

Back-of-the-envelope calculation

"봉투 뒷면 계산"이라는 뜻으로, 정밀한 데이터 없이 합리적인 가정을 세우고 빠르게 어림잡아 계산하는 것을 말한다. 회계 감사가 아니다. 목적은 "이 설계가 감당해야 할 규모의 자릿수(order of magnitude)"를 파악하는 것이다.

  • QPS가 100인지 120인지는 중요하지 않다
  • QPS가 100인지 100,000인지는 설계 자체를 바꾼다 — 이 차이를 놓치면 안 된다

왜 정확한 숫자가 필요 없는가?

실제 서비스도 런칭 전에는 정확한 트래픽을 아무도 모른다. 스타트업이든 대기업 신규 서비스든 마찬가지다. 그런데도 설계는 미리 해야 한다. 그래서 실무에서 쓰는 방식이 자릿수 기반 추정이다. "10만 QPS를 버텨야 하는 설계"와 "1만 QPS면 되는 설계"는 완전히 다른 아키텍처가 필요하다. 반면 "1만 QPS"와 "1만 2천 QPS"는 같은 아키텍처로 커버된다. 면접관이 보고 싶은 것은 이 감각이다.

기본 계산 도구 — 암산으로 빠르게 하는 법

면접장에서는 계산기가 없거나 써도 흐름이 끊긴다. 아래 근사값을 외워두면 즉석에서 암산이 가능하다.

  • 하루 = 86,400초 ≈ 105초 (정확히는 8.64×104, 암산할 때는 10만으로 잡아도 자릿수는 안 틀린다)
  • 100만(1M) = 106, 10억(1B) = 109 — 큰 수는 지수로 다룬다
  • 1 byte = 8 bit, 저장 용량은 byte, 네트워크 대역폭은 보통 bit/s(bps)로 말한다 — 단위를 섞으면 안 된다
  • 1 KB ≈ 103B, 1 MB ≈ 106B, 1 GB ≈ 109B, 1 TB ≈ 1012B, 1 PB ≈ 1015B (2진수 기준 1024 아니라 1000으로 어림잡아도 면접에서는 충분하다)
Peak Factor (피크 계수)란?

평균 QPS만 계산하고 끝내면 위험하다. 실제 트래픽은 하루 24시간 균등하게 분산되지 않는다. 점심시간, 저녁 시간, 특정 이벤트에 트래픽이 몰린다. 이 평균 대비 순간 최대 배수를 Peak Factor라고 한다.

  • 일반적인 서비스: 평균의 2~3배
  • 실시간성이 강한 서비스(라이브 스트리밍, 티켓팅, 속보): 평균의 10배 이상도 가능

핵심: 설계는 평균이 아니라 피크를 버틸 수 있어야 한다. 평균만 보고 서버를 산정하면 트래픽이 몰리는 순간 장애가 난다.

왜 면접에서 이것을 묻는가?

면접관의 관점

Capacity Estimation은 단순 계산 테스트가 아니다. 면접관은 두 가지를 본다.

  • 숫자를 설계 결정에 실제로 쓰는가? — "QPS가 10만이니 단일 DB로는 안 됩니다"처럼 숫자가 결론으로 이어지는지
  • 가정을 명시적으로 말하는가? — "사용자당 하루 20번 조회한다고 가정하면"처럼 전제를 드러내는지. 시니어는 가정이 틀려도 무엇을 가정했는지 스스로 안다
실무 연결

실무에서 이 계산은 캐파시티 플래닝(capacity planning)이라는 이름으로 실제로 한다. 새 기능을 배포하기 전에 "이 기능이 하루에 몇 번 호출될 것 같고, DB에 몇 GB가 쌓일 것 같은지" 미리 추정해서 인프라팀과 서버 증설, DB 파티셔닝 여부를 논의한다. 이 추정을 안 하고 배포했다가 트래픽 폭증으로 장애가 나는 사례는 흔하다 — 블랙프라이데이 세일, 신규 기능 바이럴, 뉴스 노출 직후가 대표적이다. 면접관 중 상당수는 이런 장애를 실제로 겪어본 사람들이라, 이 계산을 대충 넘기는 후보자를 신뢰하지 않는다.

Capacity Estimation 7단계 프레임워크

매번 새로 생각하지 말고, 아래 순서를 그대로 따라가면 어떤 시스템 설계 문제에도 적용할 수 있다.

  1. 활성 사용자 규모 정의 (DAU/MAU)

    면접관이 주지 않으면 먼저 물어보거나, "일반적으로 이 정도 규모의 서비스라면"이라고 스스로 가정하고 명시한다.

  2. 사용자당 행동 빈도 정의

    하루에 몇 번 쓰기(포스트, 업로드)를 하는지, 몇 번 읽기(조회, 새로고침)를 하는지 가정한다. 읽기/쓰기를 분리해서 가정해야 한다.

  3. 일일 요청 수 → 평균 QPS 변환

    일일 요청 수를 86,400초로 나눈다. 읽기 QPS와 쓰기 QPS를 각각 따로 계산한다.

  4. Peak QPS 계산

    평균 QPS에 Peak Factor(보통 2~3배)를 곱한다. 설계는 이 피크 값을 기준으로 한다.

  5. 저장 용량(Storage) 산정

    레코드 1개 크기 × 일일 생성 건수 × 보존 기간. 여기에 복제(replication)와 인덱스 오버헤드를 반영한다.

  6. 네트워크 대역폭(Bandwidth) 산정

    Peak QPS × 평균 요청/응답 크기. Ingress(들어오는)와 Egress(나가는)를 분리해서 계산한다.

  7. 서버 대수 추정

    Peak QPS ÷ 서버 1대가 처리 가능한 QPS. 여기에 장애 대비 여유분(headroom)과 이중화(redundancy)를 더한다.

실전 예시 — 뉴스 피드형 소셜 서비스, DAU 1억

프레임워크를 실제 숫자에 적용해보자. 뒤에 Day 10(뉴스 피드 설계)에서 다시 다룰 예제이므로 여기서 계산 감각을 만들어두면 그때 훨씬 수월하다.

1) 가정 정의

  • DAU = 1억 명 (100,000,000)
  • 사용자당 평균 포스트 작성(쓰기) = 하루 2회
  • 사용자당 평균 피드 새로고침(읽기) = 하루 20회 → 읽기:쓰기 = 10:1
  • Peak Factor = 3배 (일반적인 소셜 서비스 트래픽 패턴)

2) QPS 계산

쓰기 QPS

  • 일일 쓰기 요청 = 1억 × 2 = 2억 건
  • 평균 쓰기 QPS = 200,000,000 ÷ 86,400 ≈ 2,315 QPS
  • 피크 쓰기 QPS = 2,315 × 3 ≈ 6,945 QPS

읽기 QPS

  • 일일 읽기 요청 = 1억 × 20 = 20억 건
  • 평균 읽기 QPS = 2,000,000,000 ÷ 86,400 ≈ 23,148 QPS
  • 피크 읽기 QPS = 23,148 × 3 ≈ 69,444 QPS

3) 저장 용량 계산

텍스트 포스트

  • 포스트 1건 평균 크기(텍스트+메타데이터: id, user_id, timestamp 등) ≈ 300 bytes
  • 일일 텍스트 저장량 = 2억 건 × 300B = 60GB/day
  • 5년(1,825일) 누적 = 60GB × 1,825 ≈ 109.5TB (원본)
  • DB 고가용성 표준 3x replication 적용 → 약 328.5TB

이미지 포함 포스트

  • 전체 포스트의 10%에 이미지 포함, 압축 후 평균 200KB로 가정
  • 일일 이미지 저장량 = (2억 × 10%) × 200KB = 2천만 × 200KB = 4TB/day
  • 5년 누적 = 4TB × 1,825 ≈ 7.3PB
여기서 시니어가 반드시 짚는 포인트

이미지는 S3 같은 Object Storage에 저장하는 게 일반적이다. Object Storage는 자체적으로 여러 AZ에 복제해서 11 9's(99.999999999%) 내구성을 보장하므로, DB처럼 애플리케이션 레벨에서 3배로 다시 곱할 필요가 없다. "저장 용량 = 원본 데이터 × 복제 배수"라는 공식을 기계적으로 모든 저장소에 적용하면 틀린다. 저장소의 종류에 따라 복제 전략이 다르다는 걸 아는지가 실무 경험의 신호다.

4) 네트워크 대역폭 계산

  • 쓰기 Ingress: 텍스트 요청 평균 페이로드 ≈ 1KB(헤더 포함) → 피크 쓰기 QPS(6,945) × 1KB ≈ 6.8MB/s. 이미지 업로드는 별도 경로: 피크 쓰기 QPS의 10% ≈ 695 QPS × 200KB ≈ 139MB/s(약 1.1Gbps)
  • 읽기 Egress: 피드 응답 1건에 포스트 20개(텍스트만, 6KB) 가정 → 피크 읽기 QPS(69,444) × 6KB ≈ 416MB/s(약 3.3Gbps). 이미지는 CDN이 직접 서빙하므로 오리진 서버 대역폭 계산에서 제외한다

5) 서버 대수 추정

  • 가정: 애플리케이션 서버 1대가 안정적으로 처리 가능한 QPS = 1,000 (부하 테스트 실측치를 가정한다고 명시)
  • 읽기 서버 = 69,444 ÷ 1,000 ≈ 70대 → 여유분(headroom) 30% + 이중화(N+2) 반영 → 약 95~100대
  • 쓰기 서버 = 6,945 ÷ 1,000 ≈ 7대 → 여유분 포함 → 약 10~12대
중요한 단서

이 70대라는 숫자는 "캐시가 전혀 없을 때의 최악의 경우"다. 실제로는 읽기 트래픽의 상당 부분을 캐시가 흡수하기 때문에 DB나 애플리케이션 서버가 받는 실제 부하는 훨씬 낮아진다. 이 사실을 스스로 언급하면 "이 사람은 캐시가 왜 필요한지 숫자로 설명할 수 있구나"라는 신호를 준다. (캐시 전략은 Day 4에서 다룬다.)

답변 예시 — 이렇게 말하면 된다

답변 예시 — Capacity Estimation 단계

"DAU를 1억으로 가정하겠습니다. 사용자당 하루 2번 포스트를 작성하고, 피드는 20번 정도 새로고침한다고 가정하면 쓰기는 하루 2억 건, 읽기는 20억 건입니다. 86,400초로 나누면 평균 쓰기 QPS는 약 2,300, 읽기 QPS는 약 23,000 정도 나옵니다. 트래픽이 특정 시간대에 몰리는 걸 감안해서 피크 팩터 3배를 적용하면, 피크 시점 쓰기는 약 7,000 QPS, 읽기는 약 70,000 QPS까지 나올 수 있다고 보고 설계하겠습니다. 이 정도 읽기 QPS면 DB가 직접 받기는 어렵고, 캐시 레이어가 필수적이라고 판단됩니다."

강한 답변과 약한 답변의 차이

강한 답변 (Senior level)

"가정을 먼저 말씀드리면, DAU 1억에 사용자당 읽기:쓰기를 10:1로 잡겠습니다. 계산하면 피크 읽기 QPS가 약 7만인데, 이 정도면 단일 DB는 물론이고 캐시 없는 애플리케이션 서버 계층도 감당하기 어렵습니다. 그래서 캐시 히트율을 80%로 가정하면 DB가 실제로 받는 QPS는 약 1만 4천으로 줄어듭니다. 이 숫자를 기준으로 읽기 replica를 몇 대 둘지 다음 단계에서 정하겠습니다."

약한 답변 (Junior level)

"음... DAU가 1억이니까 QPS도 꽤 많을 것 같아요. 서버는 여러 대 두면 될 것 같고, 캐시도 붙이면 좋을 것 같습니다. (구체적인 숫자 없이 바로 다음 단계로 넘어감)"

왜 약한 답변이 약한가?

  • 가정을 말하지 않음: "많을 것 같다"는 근거가 없다. 어떤 가정에서 그런 결론이 나왔는지 알 수 없다
  • 계산 과정 생략: 결론(서버 여러 대, 캐시)은 맞을 수도 있지만, 그 결론에 도달한 사고 과정을 보여주지 못한다
  • 숫자가 이후 결정에 연결되지 않음: "캐시도 붙이면 좋을 것 같다"고만 하고, 캐시가 왜 필요한지(QPS가 임계치를 넘어서) 근거를 대지 못한다
  • Peak factor 개념 없음: 평균과 피크를 구분하지 않아서, 실제 트래픽 몰림 상황을 설계에 반영하지 못한다

답변 사고 순서 — 실전 적용

머릿속 흐름 (Capacity Estimation 3~5분 기준)

[0~30초] 어떤 가정이 필요한지 목록을 먼저 떠올린다: DAU, 읽기/쓰기 빈도, 평균 데이터 크기, 보존 기간, Peak factor.

[30초~1.5분] 읽기/쓰기 QPS를 각각 계산한다. 말하면서 계산해도 괜찮다 — 오히려 사고 과정이 보여서 좋다.

[1.5~2.5분] Peak factor를 적용한다. "일반적으로 소셜 서비스는 피크가 평균의 2~3배입니다"라고 근거를 댄다.

[2.5~4분] 저장 용량을 계산한다. 데이터 종류별(텍스트/미디어)로 나눠서 계산하고 보존 기간, 복제 배수를 반영한다.

[4~5분] 계산한 숫자로 결론을 낸다. "이 QPS면 캐시가 필요하다" "이 저장 용량이면 샤딩을 고려해야 한다"처럼 다음 설계 단계로 연결한다.

Follow-up 1

Q: "Peak Factor를 3배로 잡으셨는데, 그 근거는 무엇인가요?"

답변 방향: "정확한 수치라기보다 일반적인 소셜 서비스의 트래픽 패턴에서 관찰되는 범위입니다"라고 솔직하게 말하되, 서비스 특성에 따라 달라진다는 것을 덧붙인다. "이 서비스가 특정 이벤트(스포츠 경기 생중계, 속보)에 민감하다면 피크가 10배 이상 날 수도 있어서, 그 경우엔 오토스케일링과 별도로 서킷 브레이커, 큐잉 같은 방어 장치가 더 중요해집니다." 포인트: 숫자를 방어하되, 숫자가 틀릴 가능성과 그에 대한 대응까지 언급하면 더 높은 신호를 준다.

Follow-up 2

Q: "미디어 저장 용량 계산에서 왜 3x replication을 안 곱하셨나요?"

답변 방향: "미디어는 관계형 DB가 아니라 S3 같은 Object Storage에 저장하는 걸 가정했습니다. Object Storage는 여러 데이터센터에 자동으로 복제해서 자체적으로 높은 내구성을 제공하기 때문에, 애플리케이션이 별도로 3x replication을 관리할 필요가 없습니다. 다만 여러 해상도(썸네일, 원본)를 저장한다면 그만큼 배수가 늘어난다는 점은 반영해야 합니다."

Follow-up 3

Q: "캐시 히트율을 80%로 가정했는데, 만약 60%로 떨어지면 어떻게 되나요?"

답변 방향: 실제 숫자로 다시 계산해서 보여주는 게 중요하다. "피크 읽기 QPS 7만에서 캐시 히트율이 60%면 DB가 받는 QPS는 2만 8천으로, 80% 가정 때(1만 4천)보다 2배 높아집니다. 이 정도면 읽기 replica를 추가하거나, 캐시 워밍(warming) 전략을 강화해서 히트율을 다시 끌어올리는 걸 고려하겠습니다. 즉 캐시 히트율은 고정 상수가 아니라 모니터링하면서 계속 튜닝해야 할 지표라는 점을 설계에 반영하겠습니다."

Follow-up 4

Q: "서버 1대당 1,000 QPS를 처리한다고 가정하셨는데, 이 숫자는 어디서 나온 건가요?"

답변 방향: "실무에서는 실제 부하 테스트(load test)로 이 수치를 얻습니다. 면접에서는 이 숫자를 정확히 알 방법이 없으니, CPU-bound 로직이 적고 I/O가 대부분인 일반적인 REST API 서버 기준으로 자주 인용되는 범위(수백~수천 QPS)에서 1,000을 가정했습니다. 실제로는 이 값이 로직의 복잡도, DB 커넥션 풀 크기, 네트워크 지연에 따라 크게 달라지기 때문에, 실제 설계에서는 반드시 부하 테스트로 검증해야 한다고 덧붙이겠습니다."

Trade-off와 흔한 오해

오해 1: "숫자를 정확히 못 맞추면 감점이다"

실제: 감점 요인은 숫자의 정확도가 아니라 계산 과정의 논리성이다. 가정이 다르면 최종 숫자는 얼마든지 달라질 수 있다. 면접관이 "그 가정은 좀 다를 것 같은데요"라고 하면 "네, 알겠습니다. 그럼 이 가정으로 다시 계산하면..."이라고 유연하게 대응하면 된다. 방어적으로 굴 필요 없다.

오해 2: "평균 QPS만 계산하면 충분하다"

실제: 평균만으로 설계하면 실제 서비스에서 피크 시간대에 장애가 난다. 실무에서 발생하는 장애 상당수가 "평균은 버텼는데 피크에서 터진" 케이스다. Peak Factor를 반드시 반영해야 한다.

오해 3: "저장소는 다 똑같이 3배 복제하면 된다"

실제: 저장소마다 복제/내구성 전략이 다르다. 관계형 DB의 replica, Object Storage의 자체 내구성, 캐시의 휘발성 등은 서로 다른 성격을 가진다. 기계적으로 "무조건 3배"를 곱하면 저장소 특성을 모른다는 신호를 준다.

오해 4: "서버 대수 = Peak QPS ÷ 서버당 QPS면 끝"

실제: 이 값은 최소 요구치일 뿐이다. 실무에서는 여기에 headroom(여유분, 보통 20~30%)과 이중화(redundancy, 서버 1~2대가 죽어도 버티는 여유)를 추가로 반영해야 한다. 이걸 빼먹으면 "서버가 딱 필요한 만큼만 있어서 1대라도 죽으면 전체 장애"라는 취약한 설계가 된다.

Trade-off: 계산 시간 vs 설계 시간

Capacity Estimation에 너무 오래 머무르면 정작 중요한 아키텍처 설계, 장애 대응 논의 시간이 줄어든다. 반대로 너무 대충 넘기면 이후 설계의 근거가 빈약해진다. 실전에서는 3~5분 안에 핵심 숫자(QPS, 저장 용량)만 뽑고, 세부 계산(대역폭, 서버 대수)은 필요할 때 Deep dive에서 다시 다룬다는 시간 배분 감각이 중요하다.

각 문제에서 가장 적합한 답을 고르고 이유를 생각해보자.

1. DAU가 5천만 명이고 사용자당 하루 평균 4번 요청한다고 가정할 때, 평균 QPS에 가장 가까운 값은?

2. Peak Factor를 반영하지 않고 평균 QPS만으로 서버 대수를 산정했을 때 가장 가능성이 높은 문제는?

3. 이미지 파일을 S3 같은 Object Storage에 저장할 때, DB 저장 용량 계산과 똑같이 3x replication을 곱하지 않는 이유로 가장 적절한 것은?

핵심 정리

  • Capacity Estimation은 정확한 숫자가 아니라 자릿수(order of magnitude)와 계산 과정을 보여주는 단계다
  • 7단계 프레임워크: DAU 정의 → 행동 빈도 → 평균 QPS → Peak QPS → 저장 용량 → 대역폭 → 서버 대수
  • 읽기/쓰기 QPS는 반드시 분리해서 계산한다 — 캐시, 복제 전략이 서로 다르게 적용되기 때문
  • Peak Factor(보통 2~3배)를 빼먹으면 평균은 버텨도 트래픽 몰림 순간 장애가 난다
  • 저장소마다 복제 전략이 다르다 — DB는 3x replication, Object Storage는 자체 내구성 보유
  • 서버 대수 = Peak QPS ÷ 서버당 처리량에 headroom과 이중화(redundancy)를 반드시 더한다
  • 계산한 숫자는 반드시 다음 설계 결정(캐시 필요 여부, 샤딩 여부 등)으로 연결해야 의미가 있다
