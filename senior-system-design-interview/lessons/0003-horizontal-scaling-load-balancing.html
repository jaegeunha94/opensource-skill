Day 3 · 시스템 설계 Interview

수평 확장과 로드밸런싱 — Stateless 서버, L4/L7 LB, 세션 관리

Day 2에서 계산한 피크 QPS 7만은 서버 한 대로 못 버틴다 — 서버를 여러 대로 늘렸을 때 진짜 문제가 시작된다

이 레슨의 핵심 면접 질문

숫자로 규모를 확인했다면, 다음은 "그래서 서버를 어떻게 늘릴 것인가"다. 여기서부터 후보자의 실무 경험이 드러난다.

면접 질문 유형
  • "트래픽이 늘어나면 어떻게 대응하시겠습니까? 서버 사양을 올리는 것과 대수를 늘리는 것 중 어떤 선택을 하시겠어요?"
  • "서버를 여러 대로 늘렸는데, 사용자의 로그인 상태는 어떻게 유지하시겠습니까?"
  • "로드밸런서는 L4와 L7 중 어떤 걸 쓰시겠습니까? 차이가 뭔가요?"
  • "Sticky session이 뭔지 아세요? 왜 문제가 될 수 있나요?"
  • "로드밸런서 자체가 죽으면 전체 서비스가 다운되는 것 아닌가요?"

이 질문들의 공통점: "서버를 늘리면 됩니다"는 답이 아니라 시작점이다. 면접관은 늘렸을 때 생기는 새 문제를 아는지 본다.

전제 개념 — 확장이란 무엇이고 왜 문제가 새로 생기는가?

단어 자체는 쉽다. 하지만 "서버를 늘린다"는 한 문장 뒤에 숨은 구조를 이해해야 다음 단계로 넘어갈 수 있다.

Vertical Scaling (Scale-up) vs Horizontal Scaling (Scale-out)

Vertical Scaling(수직 확장)은 서버 한 대의 사양(CPU, 메모리)을 올리는 것이다. 비유하면 "일꾼 한 명을 더 힘세게 만드는 것"이다. 코드 변경이 거의 없고 구조가 단순하다는 장점이 있다. 하지만 물리적 한계가 있고(한 대가 감당할 수 있는 사양에는 상한이 있다), 그 한 대가 죽으면 서비스 전체가 죽는다(SPOF, Single Point of Failure).

Horizontal Scaling(수평 확장)은 서버 대수를 늘리는 것이다. "일꾼을 여러 명 더 뽑는 것"이다. 이론적으로 한계가 거의 없고, 한 대가 죽어도 나머지가 버틴다. 하지만 여러 대에 요청을 어떻게 분배할지, 여러 대가 같은 상태(로그인 정보 등)를 어떻게 공유할지라는 새로운 문제가 생긴다. 이 레슨은 이 새로운 문제를 다룬다.

Stateless 서버 vs Stateful 서버

Stateful 서버는 서버 자신이 사용자별 상태(로그인 세션, 장바구니 등)를 메모리에 들고 있는 서버다. 문제는 사용자 A가 로그인할 때 서버 1번이 그 정보를 메모리에 저장했는데, 다음 요청이 로드밸런서에 의해 서버 2번으로 가면 서버 2번은 A가 로그인했다는 사실을 모른다 — 즉시 로그아웃된 것처럼 보인다.

Stateless 서버는 서버 자신은 아무 상태도 기억하지 않고, 상태는 외부(Redis, DB, 또는 클라이언트가 들고 다니는 토큰)에 둔다. 그러면 사용자 A의 다음 요청이 어느 서버로 가든 그 서버가 외부에서 상태를 조회해서 똑같이 응답할 수 있다. 수평 확장이 가능하려면 애플리케이션 서버가 stateless여야 한다는 것이 이 레슨의 핵심 전제다.

Load Balancer(로드밸런서)란 무엇인가?

여러 대의 서버 앞에 두고, 들어오는 요청을 어느 서버로 보낼지 결정하는 중개자다. 두 가지 계층에서 동작할 수 있다.

  • L4 (Transport Layer, 4계층): IP 주소와 포트 정보만 보고 라우팅한다. TCP/UDP 패킷을 그대로 전달(포워딩)한다. 요청 내용(HTTP 헤더, URL, 쿠키)은 보지 않는다. 그래서 속도가 빠르고 처리량이 높다.
  • L7 (Application Layer, 7계층): HTTP 요청의 내용(URL 경로, 헤더, 쿠키, 바디)까지 읽고 라우팅한다. 예: "/api/video/*"는 비디오 서버 그룹으로, "/api/user/*"는 유저 서버 그룹으로. 쿠키를 보고 같은 사용자를 같은 서버로 보낼 수도 있다(sticky session). 내용을 파싱하는 만큼 L4보다 느리지만, 훨씬 똑똑한 라우팅이 가능하다.
Sticky Session(고정 세션)이란?

로드밸런서가 특정 사용자의 요청을 항상 같은 서버로 보내도록 고정하는 방식이다. 보통 로드밸런서가 응답에 쿠키를 심어서(예: 어떤 서버로 라우팅했는지) 이후 같은 쿠키를 가진 요청을 같은 서버로 보낸다. Stateful 서버의 문제(로그아웃되는 문제)를 임시로 회피할 수 있지만, 근본 해결책은 아니다 — 왜 아닌지는 아래 "왜 면접에서 묻는가"와 "흔한 오해"에서 자세히 다룬다.

왜 면접에서 이것을 묻는가?

면접관의 관점

"서버를 늘리겠습니다"는 주니어도 말할 수 있다. 면접관이 진짜 확인하고 싶은 것은 다음이다.

  • 확장의 대가를 아는가? — 서버를 여러 대로 늘리면 상태 공유, 라우팅, 배포 복잡도가 함께 늘어난다는 걸 아는지
  • Stateless 설계를 설계 초반부터 고려하는가? — 뒤늦게 "아, 세션은 어디 두죠?"라고 묻는 후보자와, 처음부터 "서버는 stateless로 설계하고 세션은 Redis에 둡니다"라고 선언하는 후보자는 신호가 다르다
  • Sticky session의 함정을 아는가? — 겉보기엔 간단한 해법이지만 왜 대규모 시스템에서 기피되는지 설명할 수 있는지가 시니어와 주니어를 가른다
  • 로드밸런서 자체의 장애도 고려하는가? — 로드밸런서가 SPOF가 되지 않도록 이중화하는 것까지 생각이 미치는지
실무 연결

실무에서 이 판단은 배포 전략과 직결된다. 배포할 때 서버를 한 대씩 순차적으로 재시작하는 rolling deployment를 쓰는데, 서버가 stateful이면 재시작되는 순간 그 서버에 붙어 있던 사용자들이 모두 로그아웃되거나 세션을 잃는다. 오토스케일링도 마찬가지다 — 트래픽이 늘어 서버가 자동으로 추가될 때, 새로 추가된 서버는 기존 세션 정보를 전혀 모른다. 그래서 실무에서는 거의 예외 없이 세션을 Redis 같은 외부 저장소로 빼거나 JWT 같은 stateless 토큰 방식을 쓴다. 로드밸런서 이중화도 마찬가지로, AWS ALB나 GCP Load Balancer는 애초에 관리형으로 다중 AZ에 걸쳐 이중화되어 제공되며, 직접 Nginx/HAProxy로 LB를 구성할 때는 LB 앞에 또 다른 LB(DNS 기반 또는 keepalived+VIP)를 두는 걸 고려해야 한다.

수평 확장 설계 판단 프레임워크

Day 2에서 낸 Peak QPS를 입력값으로 삼아, 아래 순서로 사고를 전개하면 된다.

  1. 병목이 무엇인지 먼저 판단한다

    CPU-bound인지 I/O-bound인지, 서버 한 대로는 물리적으로 부족한 규모인지 확인한다. Peak QPS가 서버 1대 처리량을 크게 넘으면 수평 확장이 필요하다는 결론이 자연스럽게 나온다.

  2. Vertical vs Horizontal을 선택한다

    대규모 트래픽, 고가용성이 필요하면 Horizontal이 기본값이다. Vertical은 단기적 응급 대응이나 소규모 서비스에서만 유효하다는 걸 짚는다.

  3. 애플리케이션 서버를 Stateless로 만든다

    세션/로그인 상태를 서버 메모리가 아니라 Redis 같은 외부 저장소나 JWT 토큰으로 이동시킨다. 이게 되어야 다음 단계(로드밸런싱)가 의미가 있다.

  4. 로드밸런서 계층을 선택한다 (L4 vs L7)

    단순 트래픽 분산과 최고 성능이 필요하면 L4, URL 기반 라우팅/쿠키 기반 라우팅/세밀한 헬스체크가 필요하면 L7을 선택한다. 실무에서는 L4(NLB) 뒤에 L7(ALB)을 두는 계층 구조도 흔하다.

  5. 로드밸런싱 알고리즘을 선택한다

    Round Robin(순차 분배), Least Connections(가장 연결이 적은 서버 우선), Weighted(서버 사양에 따라 가중치), IP Hash(같은 클라이언트 IP를 같은 서버로) 중 트래픽 패턴에 맞는 것을 고른다.

  6. 헬스체크와 오토스케일링을 설계한다

    로드밸런서가 주기적으로 서버 상태를 확인해서 죽은 서버로는 트래픽을 보내지 않게 한다. 트래픽 증가/감소에 따라 서버 대수를 자동 조절하는 오토스케일링 정책(CPU 사용률 기준 등)을 붙인다.

  7. 로드밸런서 자체의 이중화를 고려한다

    로드밸런서가 SPOF가 되지 않도록 한다. 클라우드 관리형 LB는 기본 제공되고, 직접 구성 시 다중 AZ + DNS 기반 failover 또는 VIP(keepalived)를 고려한다.

실전 예시 — Day 2의 뉴스 피드 서비스 이어서 설계하기

Day 2에서 계산한 "피크 읽기 QPS 약 7만"을 그대로 가져와서 이번 단계에 적용해보자.

1) 병목 판단

  • 서버 1대가 처리 가능한 QPS를 1,000으로 가정(Day 2와 동일 가정)했을 때, 피크 읽기 QPS 7만은 단일 서버 대비 70배 규모다
  • 결론: Vertical scaling으로는 물리적으로 불가능한 규모 — Horizontal scaling이 필수

2) Stateless 전환

  • 기존에 서버 메모리에 로그인 세션을 저장하던 구조라면, 이를 Redis 세션 스토어로 이전한다
  • 또는 로그인 성공 시 서버가 아무 상태도 저장하지 않고, 사용자에게 서명된 JWT 토큰을 발급해서 매 요청마다 그 토큰으로 인증하는 방식으로 전환한다
  • 두 방식 모두 "어느 서버가 요청을 받아도 동일하게 응답 가능"하다는 조건을 만족시킨다

3) 로드밸런서 계층 선택

  • API 서버는 URL 경로별로 다른 서비스(피드 서비스, 업로드 서비스)로 라우팅할 필요가 있으므로 L7 로드밸런서(예: AWS ALB)를 선택
  • 이미지/동영상 업로드처럼 대용량 바이너리 전송이 많은 경로는 L4로 별도 처리해서 오버헤드를 줄이는 것도 고려할 수 있다고 언급

4) 알고리즘과 오토스케일링

  • 일반 API 요청은 Least Connections를 사용 — 서버별 요청 처리 시간 편차가 있을 수 있으므로 단순 Round Robin보다 부하가 고르게 분산된다
  • CPU 사용률 70% 이상이 5분 지속되면 서버를 자동 증설하고, 30% 이하가 10분 지속되면 축소하는 오토스케일링 정책을 둔다
  • 새 서버가 추가되면 헬스체크를 통과한 뒤에만 트래픽을 받도록 warm-up 기간을 둔다
여기서 시니어가 반드시 짚는 포인트

95~100대(Day 2에서 산정)라는 서버 대수 자체보다 "이 서버들이 죽었다 살았다를 반복해도 사용자는 아무것도 느끼지 못해야 한다"는 원칙이 더 중요하다. 이 원칙을 지키려면 서버가 stateless여야 하고, 로드밸런서가 헬스체크로 죽은 서버를 자동으로 빼야 하며, 세션이 서버 밖에 있어야 한다. 이 세 가지가 갖춰지지 않은 상태에서 "서버 100대"라고만 말하면, 실제로는 100개의 SPOF가 있는 것과 다름없다.

답변 예시 — 이렇게 말하면 된다

답변 예시 — 수평 확장 설계 단계

"Day 2에서 계산한 피크 읽기 QPS 7만은 서버 한 대로 감당할 수 없는 규모라, Vertical scaling이 아니라 Horizontal scaling으로 가겠습니다. 다만 서버를 여러 대로 늘리기 전에 먼저 서버를 stateless로 만들어야 합니다. 로그인 세션을 서버 메모리가 아니라 Redis에 두거나 JWT 토큰 방식으로 전환해서, 어느 서버가 요청을 받아도 동일하게 응답할 수 있게 하겠습니다. 그 위에 L7 로드밸런서를 둬서 URL 경로 기반으로 서비스를 분리하고, Least Connections 알고리즘으로 부하를 분산하겠습니다. 로드밸런서는 헬스체크로 죽은 서버를 자동 제외시키고, CPU 사용률 기준으로 오토스케일링을 걸어서 트래픽 변화에 대응하겠습니다."

강한 답변과 약한 답변의 차이

강한 답변 (Senior level)

"서버를 늘리기 전에 stateless 전환이 선행되어야 합니다. 그렇지 않으면 로드밸런서가 요청을 아무 서버로 분산하는 순간 세션이 깨지는 문제가 생깁니다. 세션은 Redis로 빼고, 로드밸런서는 헬스체크 실패 시 자동으로 해당 서버를 제외하도록 하겠습니다. 그리고 로드밸런서 자체도 SPOF가 될 수 있으니 다중 AZ에 걸쳐 이중화된 관리형 LB를 쓰겠습니다."

약한 답변 (Junior level)

"트래픽이 많으면 서버를 더 만들면 됩니다. 로드밸런서로 요청을 나눠주면 되고요. (세션 문제, LB 이중화, 헬스체크는 언급 없음)"

왜 약한 답변이 약한가?

  • Stateless 전제 누락: 서버를 그냥 늘리면 로그인 상태가 깨진다는 걸 인지하지 못했다 — 실제로 구현하면 바로 장애가 난다
  • 로드밸런서를 블랙박스로 취급: "요청을 나눠준다"까지만 말하고, 어떤 계층(L4/L7)인지, 어떤 알고리즘인지 설명하지 못한다
  • 장애 상황 고려 없음: 서버가 죽었을 때 로드밸런서가 어떻게 대응하는지(헬스체크), 로드밸런서 자체가 죽으면 어떻게 되는지에 대한 고려가 없다
  • 확장의 "대가"를 모름: 서버를 늘리는 게 공짜가 아니라 복잡도(상태 공유, 배포, 모니터링 대상 증가)를 늘린다는 인식이 없다

답변 사고 순서 — 실전 적용

머릿속 흐름 (수평 확장 설계 3~5분 기준)

[0~30초] 앞서 낸 Peak QPS와 서버 1대 처리량을 비교해서 Vertical로 안 되는 규모임을 확인하고 Horizontal을 선언한다.

[30초~1.5분] "서버를 늘리기 전에" stateless 전환이 선행되어야 함을 짚는다. 세션을 어디로 옮길지(Redis vs JWT) 선택하고 근거를 말한다.

[1.5~2.5분] 로드밸런서 계층(L4/L7)을 선택하고 이유를 말한다. 필요하면 알고리즘까지 언급한다.

[2.5~4분] 헬스체크, 오토스케일링으로 장애 대응 및 트래픽 변화 대응을 설계한다.

[4~5분] 로드밸런서 자체의 이중화까지 언급해서 "SPOF를 없앴다"는 결론으로 마무리한다.

Senior-Level Follow-up 질문과 답변 방향

구조를 잘 그렸다고 끝이 아니다. 면접관은 각 선택의 대안과 실패 시나리오를 파고든다.

Follow-up 1

Q: "로드밸런서 자체가 죽으면 어떻게 되나요?"

답변 방향: "로드밸런서가 단일 인스턴스면 SPOF가 됩니다. 클라우드 환경이라면 ALB, NLB 같은 관리형 로드밸런서를 쓰는 게 첫 번째 선택인데, 이들은 애초에 여러 가용영역(AZ)에 걸쳐 이중화되어 제공됩니다. 직접 Nginx나 HAProxy로 구성해야 한다면, 로드밸런서를 최소 2대 이상 두고 그 앞에 DNS 라운드로빈이나 Keepalived + VIP(가상 IP) 방식으로 하나가 죽으면 다른 하나가 그 IP를 넘겨받도록(failover) 구성해야 합니다." 포인트: "관리형 서비스가 이미 이중화를 해결해준다"는 것과 "직접 구성 시 어떻게 이중화하는지"를 둘 다 아는 것이 신호다.

Follow-up 2

Q: "Sticky session을 쓰면 안 되나요? 구현이 더 간단해 보이는데요."

답변 방향: "Sticky session은 소규모 트래픽이나 마이그레이션 과도기에는 임시방편으로 쓸 수 있지만, 대규모 시스템에서는 몇 가지 문제가 있습니다. 첫째, 특정 서버에 트래픽이 몰리면(핫스팟) 그 서버만 과부하가 걸리는데, sticky session이라 다른 서버로 옮길 수 없어 부하 분산이 깨집니다. 둘째, 그 서버가 재시작되거나 죽으면 거기 붙어있던 사용자 전원이 세션을 잃습니다 — SPOF가 사용자 그룹 단위로 여러 개 생기는 셈입니다. 셋째, 오토스케일링으로 서버를 줄일 때(scale-in) 그 서버에 붙은 세션을 안전하게 정리하기 어렵습니다. 그래서 근본 해결은 세션을 외부화하는 것이고, sticky session은 그 전환 전까지의 임시 완화책 정도로 봅니다."

Follow-up 3

Q: "세션을 Redis로 빼는 것과 JWT를 쓰는 것, 둘 중 어떤 걸 선택하시겠어요?"

답변 방향: "trade-off가 있습니다. Redis 세션 스토어는 서버가 매 요청마다 Redis를 조회해야 해서 네트워크 홉이 하나 늘고 Redis가 새로운 의존성이자 장애 지점이 되지만, 즉시 로그아웃(세션 무효화)이 쉽습니다 — 관리자가 특정 사용자를 강제 로그아웃시키려면 Redis에서 그 키만 지우면 됩니다. JWT는 서버가 별도 조회 없이 토큰 자체의 서명을 검증만 하면 되니 빠르고 외부 의존성이 없지만, 한 번 발급된 토큰은 만료 전까지 강제 무효화가 어렵다는 단점이 있습니다(블랙리스트를 또 어딘가에 둬야 하는 모순이 생깁니다). 저는 강제 로그아웃, 권한 변경이 실시간으로 반영되어야 하는 서비스라면 Redis, 단순 인증 위주면 JWT를 선택하겠습니다."

Follow-up 4

Q: "오토스케일링으로 새 서버가 추가되는 순간에도 트래픽이 안정적으로 들어가나요?"

답변 방향: "새 서버는 부팅 직후 캐시가 비어 있고(cold cache) 커넥션 풀도 아직 준비 안 된 상태라, 바로 풀 트래픽을 받으면 오히려 느려지거나 에러가 날 수 있습니다. 그래서 헬스체크를 통과한 직후 바로 100% 트래픽을 주는 게 아니라, 짧은 warm-up 구간을 두고 점진적으로 트래픽 비중을 늘리는 방식(gradual traffic shifting)을 씁니다. 또한 헬스체크 자체도 '프로세스가 떠 있는지'가 아니라 'DB, 캐시 연결까지 정상인지'를 확인하는 readiness probe 수준으로 설계해야 아직 준비 안 된 서버로 트래픽이 새는 걸 막을 수 있습니다."

Trade-off와 흔한 오해

오해 1: "서버만 늘리면 트래픽 문제는 끝난다"

실제: 서버를 stateless로 만들지 않은 채 대수만 늘리면 로그인이 계속 끊기거나 데이터 불일치가 생긴다. 수평 확장은 "서버 대수"의 문제가 아니라 "상태를 어디에 둘 것인가"의 문제다.

오해 2: "Vertical scaling은 무조건 나쁘다"

실제: 초기 단계 서비스나 트래픽이 크지 않은 내부 시스템, 혹은 DB처럼 수평 확장이 구조적으로 어려운 컴포넌트에서는 Vertical scaling이 훨씬 단순하고 합리적인 선택일 수 있다. "무조건 수평 확장"이 아니라 규모와 복잡도 비용을 함께 고려해야 한다는 걸 아는 게 시니어 신호다.

오해 3: "Round Robin이면 항상 충분하다"

실제: Round Robin은 서버 사양이 동일하고 요청 처리 시간이 균등할 때는 잘 동작한다. 하지만 요청마다 처리 시간 편차가 크거나(예: 어떤 요청은 짧은 조회, 어떤 요청은 무거운 집계) 서버 사양이 다르면 Least Connections나 Weighted 방식이 더 적절하다.

오해 4: "세션을 Redis에 넣으면 자동으로 완전한 stateless다"

실제: 애플리케이션 서버는 stateless가 되지만, 이제 Redis라는 새로운 상태 저장소가 시스템에 생긴 것이다. 이 Redis 자체가 SPOF가 되지 않도록 Redis Sentinel이나 Cluster로 이중화해야 진짜 의미의 고가용성이 완성된다. "상태를 서버 밖으로 옮겼다"와 "상태 관리 문제가 사라졌다"는 다른 이야기다.

Trade-off: L4 vs L7 — 성능과 유연성의 교환

L4는 패킷을 그대로 전달하기만 하므로 매우 빠르고 처리량이 높다. 하지만 URL 기반 라우팅, 쿠키 기반 세션 어피니티, 세밀한 헬스체크 같은 똑똑한 기능은 못 한다. L7은 이런 기능을 제공하지만 요청 내용을 파싱해야 해서 상대적으로 느리고 자원을 더 쓴다. 실무에서는 정적 파일이나 대용량 스트림처럼 단순 전달이면 충분한 경로는 L4, API 라우팅이 복잡한 경로는 L7로 계층을 섞어서 쓰는 경우가 많다.

핵심 정리

  • 수평 확장의 전제는 애플리케이션 서버를 stateless로 만드는 것이다 — 세션은 Redis나 JWT로 외부화한다
  • L4는 빠르지만 단순한 IP/포트 기반 전달, L7은 느리지만 URL/쿠키 기반의 똑똑한 라우팅이 가능하다 — 실무는 둘을 섞어 쓴다
  • Sticky session은 임시방편일 뿐 근본 해결책이 아니다 — 핫스팟과 서버 장애 시 세션 유실 문제가 남는다
  • 로드밸런서는 헬스체크로 죽은 서버를 자동 제외하고, 오토스케일링으로 트래픽 변화에 대응해야 한다
  • 로드밸런서 자체도 SPOF가 될 수 있다 — 관리형 LB 사용 또는 직접 구성 시 이중화(다중 AZ, VIP failover)가 필요하다
  • Vertical scaling이 항상 나쁜 것은 아니다 — 규모와 복잡도 비용을 함께 고려해서 선택해야 한다
